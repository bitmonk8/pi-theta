# Tool Calls

Loom code calls a Pi tool or a `.loom` callable directly via the bare-identifier form `<name>(args)`, where `<name>` is an entry in the loom's *callable set* (the resolved `tools:` field after any `as` rename and the default hyphen→underscore loom-basename rewrite; see [Parameters and Frontmatter](./frontmatter.md) and [Glossary — `callable set`](./glossary.md)). The same callable set is what the model sees during a `@`...`` query — the declaration is shared between the model-driven and code-driven call paths.

```loom
let contents = read({ path: "src/main.ts" })?
let matches  = grep({ pattern: "TODO", path: "src" })?
let summary  = summarise(contents)?
let label    = triage(summary)?
```

**No conversation turn.** A tool call is a direct call against Pi's tool runtime (or, for a `.loom` callable, a fresh subagent invocation; see below). It does **not** add a turn to the loom's conversation, does **not** consume model tokens, and does **not** appear in the conversation transcript. This is the deliberate distinction from `@`...`` queries: queries cross code → model in the current conversation; tool calls cross code → side-effect (or code → child conversation, for a `.loom` callable) without disturbing the current one.

**Argument shape.** Pi tools take a single object argument matching the tool's input schema (TypeBox / JSON Schema, exposed by Pi at registration). The argument is written as a bare object literal (`read({ path: "src/main.ts" })`) — the second of the two documented exceptions to the bare-object-literal prohibition in [Expression Sublanguage — Object construction](./expressions.md). The literal is parsed by the [Loom literal sublanguage](./grammar.md#loom-literal-sublanguage), the same notation used for `params:` defaults: primitive literals, `null`, array literals, bare-key object literals, `Enum.Variant` access, and variant-schema construction (`Cat { ... }`) are admitted. Operators, `let`-bound identifier references, function calls inside the argument, and `${...}` interpolation are not — violations are `loom/parse/tool-arg-not-literal`. To pass a computed value, bind it via `let` first and reference it inside a typed `params:` callee or use `invoke(...)`. A multi-argument form (`read({...}, {...})`) is `loom/parse/tool-arg-arity` regardless of the argument shapes. `.loom` callables take their callee `params:` as already-typed values, positionally in declaration order — the same argument-binding rules `invoke(...)` uses; bare object literals do not apply (the callee's typed `params:` supply the names, not an external schema). Type mismatches surface as `loom/parse/tool-arg-type-mismatch` when the callee is statically resolvable per [Invocation — Static resolution](./invocation.md#static-resolution); otherwise the runtime AJV check is the safety net. Argument arity is checked under the same rules as `invoke(...)` — see [Invocation — Argument arity](./invocation.md#argument-binding) for the parse-time vs runtime split (`loom/parse/invoke-arity-too-few` / `loom/parse/invoke-arity-too-many` apply equally to a `.loom` callable call). The path-restriction rule from [Invocation — Resolution](./invocation.md) also applies to `.loom` paths used as `tools:` entries: a path that escapes the active discovery roots is rejected with `loom/load/invoke-path-escape` and the callable is not created. Slash-command argument binding (LLM-driven; see [Slash-Command Argument Binding](./binder.md)) does not apply here — code-side callers pass typed values directly.

**Return type.** The result type depends on the callee kind:

| Callee kind | Return type | Notes |
|---|---|---|
| Pi tool | `Result<string, QueryError>` | V1 returns the tool's final output as a single `string` (mirroring untyped queries). Pi tool definitions ship an input schema but no output schema; provider tool-use conventions treat outputs as freeform text the model interprets. |
| Registered loom (subagent-mode) | `Result<T, QueryError>` where `T` is the callee's inferred return type | Same inference rule as `invoke<T>(...)`: when the callee `.loom` is statically resolvable per [Invocation — Static resolution](./invocation.md#static-resolution), its tail-expression type flows into the call site. Otherwise the runtime AJV check enforces it. |

As with queries and `invoke`, the call returns a `Result`; use `?` to propagate failure or `match` to handle.

**Failures.** Tool-call failures surface as the `CodeToolError` variant of `QueryError`. The schema lives in [Errors and Results — QueryError variants](./errors-and-results.md#queryerror-variants); the cause enum is `validation` (arguments failed input-schema validation), `execution` (tool's `execute()` threw or returned `isError: true`), `cancelled` (AbortSignal fired), or `unknown_tool` (callable was lost across a file-watcher reload — cannot occur during the lifetime of a single invocation, since the load-time `tools:` table holds strong references to resolved callables; see [Parameters and Frontmatter](./frontmatter.md)).

*Outcome enumeration (normative).* The `CodeToolError.cause` enum is the contract surface for loom authors and is closed at the four members above; it is **not** widened to cover every observable disposition of `execute()`. Three further tool-side outcomes are routed off the `CodeToolError` surface and pinned here so the lowering rule in [Pi Integration Contract — Tool execution from loom code](./pi-integration-contract.md#tool-execution-from-loom-code) is total over what `execute()` can do:

- *Non-settling Promise.* If `execute()` returns a Promise that never settles, the runtime makes no V1 attempt at internal timeout enforcement (consistent with [`spec.md` — Hard ceilings, NOCEIL-1](../spec.md#hard-runtime-ceilings)). The tool-call expression blocks at its `await` until `loomAbort.signal` fires, at which point cancellation surfaces through the existing `cause: "cancelled"` path. The deferred per-call timeout extension is reserved on the *V1 seam — per-call timeout* below; see [Future Considerations](./future-considerations.md).
- *Pre-evaluation setup throw inside the `.loom`-callable adapter.* When the model emits a parallel tool call (per **Concurrency** below) targeting a `.loom` callable and the adapter's pre-evaluation dispatch-setup sequence throws — i.e. one of the steps pinned in [Pi Integration Contract — `ActiveInvocationRegistry` Dispatch-site setup wrap](./pi-integration-contract.md#active-invocation-registry) (`new AbortController()`, `ActiveInvocationRegistry.add`, the inbound `signal.addEventListener('abort', …)` attach that registers the forwarding listener on the tool-exposed adapter's `signal` parameter, `createAgentSession(...)` rejection in subagent mode, or the post-`createAgentSession` `loomAbort.signal.addEventListener('abort', …)` attach that wires the one-shot `AgentSession.abort()` listener) raises before the loom body runs — the adapter MUST translate the captured error into a clean `{ isError: true, content: [{ type: "text", text: "loom <name> aborted with internal error: <error.message>" }] }` value and return it to Pi normally; in parallel, the runtime emits one `loom/runtime/internal-error` diagnostic (carrying `error.message` in `message` and `error.stack` — or `"<no stack available>"` when falsy — in `hint`) and one `loom-system-note` so an operator observes the failure even though the model's surface is the tool result. The adapter MUST NOT let the throw propagate as an unhandled rejection out of `tool.execute(...)`: silent loss of the call into Pi's parallel-tool-call dispatch is forbidden by the same "silent success on denial is forbidden" stance pinned in [Pi Integration Contract — No additional access channels](./pi-integration-contract.md#no-extra-mediation). The runtime-defect surface is the routing class — see [Errors and Results — Runtime panics](./errors-and-results.md#runtime-panics) — and the `.loom`-adapter `{ isError: true, ... }` translation is the per-surface form.
- *Non-conforming return shape.* If `execute()` resolves with a value whose shape violates the `{ content, isError }` envelope (the resolved value is not an object, `content` is not iterable, an entry is missing `type` / `text`, etc.), the lowering wrapper in [Pi Integration Contract — Tool execution from loom code](./pi-integration-contract.md#tool-execution-from-loom-code) catches the resulting throw and routes it through the `loom/runtime/internal-error` runtime-defect surface with `details.kind = "tool-return-shape"`, **not** through `CodeToolError`. A non-conforming Pi tool is a defect in that tool, not a value loom authors can usefully `match` on; the routing class is the same one [Errors and Results — Runtime panics](./errors-and-results.md) uses for unanticipated SDK throws.
- *Post-cancel resolution.* If `loomAbort` fires and the tool-call checkpoint surfaces `cause: "cancelled"`, any later settlement of the underlying Promise (resolve or reject, `Ok` or `isError: true` or throw) is discarded — no rebinding, no second `Err`, no second `RuntimeEvent`. The discard is mechanically realised by a swallowing handler the runtime MUST attach to the `execute()` Promise at construction time so that a late rejection arriving after the checkpoint surfaced `cause: "cancelled"` does **not** surface as a Node `unhandledRejection` process event; the handler MUST be installed before the first microtask boundary (a lazily-attached `.catch(() => {})` will not catch a rejection already queued for `unhandledRejection`). The discard is total and applies regardless of the late rejection's `.message` (an `OOMError`-style late reject is still discarded; promotion to `loom/runtime/internal-error` would re-introduce the second-event surface this rule forbids), and the discarded value is not traversed (cleanup of any handles it carries is the tool implementation's responsibility, not the runtime's). See [Cancellation — Race semantics](./cancellation.md) for the normative discard rule and its symmetric application to `@`-query provider Promises and `invoke` callee top-level Promises.

`CodeToolError` is distinct from `ModelToolError` (which covers tool failures *inside* the model's tool-call loop during a `@`...`` query). The variants carry different fields because the contexts differ — a code-side tool call has no `tool_call_id` issued by Pi's tool-loop machinery and no `raw_response` from the model; a model-loop failure has both. Authors who want to handle every tool failure uniformly write two `match` arms or a final `_ => ...` catch-all.

For a `.loom` callable, failures the callee returned cascade through the standard `InvokeCalleeError` variant (the call is, semantically, an `invoke`); failures from the loom infrastructure itself (callee unloadable, validation mismatch on the return value) cascade through `InvokeInfraError`. The only situation where `CodeToolError` arises for a `.loom` callable is V1's `"unknown_tool"` safety net.

**Concurrency.** V1 tool calls from loom code are sequential and synchronous-looking: the runtime awaits each call's underlying Promise before evaluating the next expression, so the loom interpreter yields to Pi's event loop during the wait (the TUI render loop, keypress handlers, signals, and other Pi machinery continue to run — the call is non-blocking at the runtime level even though it appears synchronous to the author). Streaming partial results (Pi's `onUpdate` callback) are not surfaced to loom code. Concurrent tool execution exists in Pi's model-driven "parallel tool mode" inside `@`...`` queries — when the model issues multiple tool-use blocks in one assistant message, Pi runs them concurrently. V1 permits this for both Pi tools and `.loom` callables: when the model emits parallel tool calls during a query whose callable set includes a `.loom` callable, each invocation spawns its own `AgentSession` with its own captured callable set, its own derived `AbortSignal`, and its own private transcript (see [Pi Integration Contract](./pi-integration-contract.md)). The wrapping `defineTool` adapter for a `.loom` callable is therefore re-entrant: two concurrent calls into the same callee execute as two independent invocations on the event loop. The schema cache (see [Implementation Notes — Runtime](./implementation-notes.md)) is read-mostly and safe to share across concurrent invocations; compiled validators are immutable post-compile and cache writes only occur on initial compile or file-watcher invalidation. No loom-level concurrency primitive (e.g. a `parallel { ... }` block) is exposed in V1; see [Future Considerations](./future-considerations.md).

> **V1 seam — per-call timeout.** The runtime-internal options record passed into the tool-call primitive (carrying per-call configuration not surfaced to authors in V1 — cancellation hookup, the captured `ToolDefinition` reference, the synthesised `loom-direct:` `toolCallId`) is an **open struct**, not a closed positional record. V1 reserves the right to add fields in a minor revision (e.g. a per-call timeout) without breaking call sites or test fixtures; consumers MUST tolerate unknown fields. The seam is what allows the deferred per-call timeout extension in [Future Considerations](./future-considerations.md) to land additively.

**Relationship with `invoke`.** `invoke("./path.loom", ...)` and a `.loom` callable call (`my_summariser(...)` after listing `./summariser.loom` in `tools:`) are operationally equivalent for subagent-mode callees — both spawn a fresh isolated conversation, both validate the return value against the callee's inferred or annotated schema, and both surface failures through the same `QueryError` variants, and both apply the arity, return-type-compatibility, and path-restriction rules from [Invocation](./invocation.md). Subagent-mode callees see only their own callable set inside the spawned `AgentSession`, never the caller's; prompt-mode callees (reachable only via `invoke(...)`) follow the prompt → prompt tools-lifetime swap defined in [Invocation — Tools and model](./invocation.md). Prompt-mode callees are rejected by `tools:` for the reason given in [Parameters and Frontmatter](./frontmatter.md). The two surfaces share a single error model and a single schema-lowering pipeline.
