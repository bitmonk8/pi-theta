# Pi Integration Contract

The runtime depends on a small, named surface from `@mariozechner/pi-coding-agent`. Each item below is the V1 contract; behaviour outside this surface is non-load-bearing and may be revised without spec changes. The V1 contract is anchored to `@mariozechner/pi-coding-agent ~0.72.1`; the lock-step pin requiring `@mariozechner/pi-agent-core`, `@mariozechner/pi-ai`, and `@mariozechner/pi-tui` to resolve to the same minor-version line as the installed `@mariozechner/pi-coding-agent`, the provenance for that lock-step, and the re-validation obligation on a Pi minor bump are owned by *Host prerequisites — Pi SDK pin* below. The contributor checklist that gates any such bump — the artefacts produced and the build-time tests that must be green before the `peerDependencies` range moves — is owned by [Pi version bump procedure](#pi-version-bump-procedure) at the end of this page.

**Host prerequisites.** Four host-side prerequisites are load-bearing for the contract below:

1. *Pi SDK pin.* The `@mariozechner/pi-coding-agent ~0.72.1` range above is the single source of truth; the spec orientation block (see [`spec.md` — Orientation — Prerequisites](../spec.md#orientation)) references this pin by location rather than restating the literal value. The lock-step minor-version line is inherited from `@mariozechner/pi-coding-agent`'s own `dependencies` block, which pins `@mariozechner/pi-agent-core`, `@mariozechner/pi-ai`, and `@mariozechner/pi-tui` at the same `~X.Y.Z` minor-version line. All four packages are released together from the `pi-mono` monorepo; no skew across the four is supported, and loom does not attempt to detect or accommodate skew. A Pi minor bump moves all four together; before the loom `peerDependencies` range is widened, the contributor MUST follow [Pi version bump procedure](#pi-version-bump-procedure) below — the build-time `peerDependencies` literal-read assertion and the build-time SDK surface-inventory assertion are the mechanical gates that fail the build until the procedure has been completed. Loom does not at runtime read `pi-coding-agent`'s `package.json` to verify the upstream pin — doing so would re-introduce the very skew-tolerance disclaimed above; the build-time literal-read assertion is the gate that keeps the four `peerDependencies` minors aligned. Loom's `peerDependencies` block declares all four packages as belt-and-braces; the redundancy is intentional and surfaces a clear install-time error under package managers (pnpm/yarn) that do not auto-deduplicate transitive peer-dep ranges.

   *Deliberate deviation from Pi's bundled-package convention.* Pi's `@mariozechner/pi-coding-agent` `docs/packages.md` — *Dependencies* prescribes `"*"` as the `peerDependencies` range for all five bundled packages it names (`@mariozechner/pi-ai`, `@mariozechner/pi-agent-core`, `@mariozechner/pi-coding-agent`, `@mariozechner/pi-tui`, `typebox`). Loom deliberately deviates from that convention for the four `@mariozechner/*` entries by pinning them to `~0.72.1` rather than `"*"`. The deviation gives up Pi-prescribed alignment (a future reader auditing loom's packaging against `packages.md` will find four entries that do not match the convention) and buys install-time skew detection on the package managers most production hosts use: `pnpm` (in its default `node-linker=isolated` mode) and `yarn` do not auto-deduplicate transitive peer-dep ranges, so a `~0.72.1` pin in loom's `peerDependencies` causes the install to fail loudly when the host has resolved any of the four `@mariozechner/*` packages outside the `~0.72.1` minor line, whereas a `"*"` declaration would silently accept the skew and defer the failure to a runtime `TypeError` at the first SDK call site. The four-anchor invariant is what the H1 `peerDependencies` literal-read test (per [`h1-scaffold.md`](../plan_topics/h1-scaffold.md)), the [Step 0 (d) Peer-dep version](#entry-capability-probe) probe, and the literal-anchor list in [Pi version bump procedure](#pi-version-bump-procedure) below are built around. The fifth package, `typebox`, follows the convention (`"typebox": "*"`) — see the sub-paragraph immediately below — because the runtime's only `typebox` dependency is `Type.Unsafe`, whose call shape is stable across the TypeBox `0.x` → `1.x` line and therefore does not need install-time minor-skew detection. The `typebox` `"*"` declaration MUST NOT be collapsed into the four-entry `~0.72.1` group: the two ranges are deliberately distinct and reflect different stability arguments (lock-step minor for the `@mariozechner/*` four; convention-following floor for `typebox`).

   *`typebox` (the fifth Pi-bundled package).* The four `@mariozechner/*` packages above are pinned to `~0.72.1` per the lock-step rule. `typebox` is the fifth Pi-bundled package the loom runtime imports (only `Type.Unsafe`, per **Tool definition shape** below); per Pi's bundled-package convention (`@mariozechner/pi-coding-agent` `docs/packages.md` — *Dependencies*) it MUST be declared as `"typebox": "*"` so the host's bundled version wins. The `"*"` literal MUST NOT be folded into the four-entry `~0.72.1` group asserted by the build-time lock-step assertion; it is asserted by its own one-line build-time literal-read assertion. The runtime's single load-bearing dependency on the bundled `typebox` namespace — the presence of `Type.Unsafe` as a callable function — is checked at extension-factory entry by **Step 0 (e)** below, which routes a missing or renamed member to `loom/load/host-incompatible` with `details.kind = "typebox-shape"` rather than to a runtime-time `TypeError` at the first tool-registration call (factory time for prompt mode, spawn time for subagent mode). This mirrors the framing already used for `AgentSession.prototype.abort` in **Step 0 (c)**: both are single-symbol Pi-bundled imports whose absence would otherwise crash the runtime at a registration call site rather than refuse cleanly at factory entry.
2. *Binder model.* Non-bypass looms require a structured-output-capable LLM resolved via `ctx.modelRegistry` for the binder pass described in [Slash-Command Argument Binding](./binder.md). When the registry cannot resolve a binder model, the loom fails to load with `loom/load/binder-model-unresolved`. Under `~0.72.1` the strict-capability probe is degraded and surfaces as `loom/load/binder-model-strict-capability-unknown` (W). Bypass cases (no-params, single-string with no default) skip the binder call entirely. V1 has no global binder opt-out; bypass is structural and per-loom.
3. *Binder credentials.* The binder reaches its model through the same `ctx.modelRegistry` / `Model<Api>` path Pi uses for its own queries; credentials are not stored, fetched, or cached by loom. Loom relies on whatever credential resolution Pi already performs for the resolved model.
4. *Pi-supplied `AbortSignal`.* Pi delivers an `AbortSignal` to every extension entry point loom uses — `ctx.signal` on the slash-command `ExtensionCommandContext` and the `signal` parameter of `tool.execute(...)`. The Web-standard `AbortSignal` / `AbortController` shape (the Node-bundled WHATWG implementation) is a fixed-shape SDK precondition; the canonical member-with-kind enumeration the runtime depends on is pinned by [Step 0 (b)](#entry-capability-probe) below. Subagent mode does not consume a `createAgentSession({ signal })` option — `CreateAgentSessionOptions` has no `signal` field in the V1 Pi SDK pin, so passing one would be silently ignored — and instead consumes the `AgentSession.abort()` capability on the handle returned by `createAgentSession(...)`, invoked from a one-shot `loomAbort.signal` listener registered at spawn time; see **Subagent session lifecycle** and **Cancellation source** below. The loom runtime's own `AbortController` wrapping rule is in **Cancellation source** below; this prerequisite is what guarantees the inputs that rule consumes.

**Extension entry point.** A single Pi extension module (`extensions/index.ts`, the file Pi auto-discovers from `package.json#pi.extensions = ["./extensions"]`) exporting the standard `default function (pi: ExtensionAPI)` factory.

<a id="entry-capability-probe"></a>
**Step 0 — Capability probe (precedes step 1 below).** Before any `pi.registerFlag`, `pi.registerCommand`, `pi.registerTool`, `pi.registerMessageRenderer`, or `pi.on` call — collectively, the **factory-time host-binding calls** — the factory runs a synchronous probe of three host preconditions in the **fixed order (a) → (b) → (c)** below and stops at the first failure (short-circuit). This paragraph is the **canonical source of truth** for every probe rule, the floor literal, the member-with-kind enumeration, and the factory-probable capability list; [`spec.md` — Host runtime obligations 1–3](../spec.md#orientation) is informative orientation that forward-links here under [Governance — GOV-12](./governance.md). The JavaScript engine value model is a non-checked invariant by design and is **not** probed; it is owned by [Runtime Value Model — JavaScript engine assumptions](./runtime-value-model.md#javascript-engine-assumptions).

*The four pinned constants the probe consumes — Node floor, `AbortSignal` / `AbortController` member-with-kind list, factory-probable capability list, peer-dep range — live in **one source of truth** inside the extension module that the build-time literal-read assertions also consume.*

**(a) Node floor.** Compare `process.versions.node` against the pinned floor `>=20.6.0` under SemVer ordering, treating prerelease tags as eligible — a Node nightly or release-candidate build (e.g. `"21.0.0-nightly20250101abcdef"`) whose stable equivalent satisfies the floor MUST satisfy the check. The prerelease-eligibility clause is load-bearing and is part of the normative contract; an implementation that rejects qualifying prerelease builds violates this rule. `process.versions.node` is the bare version string (no leading `"v"`); do not pass `process.version`. A `process.versions.node` value that is not a syntactically valid SemVer string routes to `kind: "probe-failed"` (see *Self-failure* below), not `kind: "node-floor"`.

*Recommended recipe (non-normative).* `semver.satisfies(process.versions.node, ">=20.6.0", { includePrerelease: true })` from the [`semver`](https://www.npmjs.com/package/semver) npm package, pinned by H1 as a direct production dependency of the loom package, satisfies this contract; `semver.valid(process.versions.node)` discriminates the `"probe-failed"` route. A future swap to a different SemVer implementation (or a hand-rolled comparator) is permitted as long as the contract above — range membership with prerelease tags eligible, and a syntactic-validity check that routes failures to `"probe-failed"` — is preserved end-to-end.

**(b) `AbortSignal` / `AbortController` shape.** This sub-step is the **canonical source of truth** for the WHATWG `AbortSignal` / `AbortController` member surface the loom runtime depends on; spec.md Host runtime obligation 2 is informative orientation that forward-links here. The probe applies a per-member rule keyed on the member's `kind` (the table below is the source of truth, co-located with the member name in the extension module's pinned-constants block that the build-time surface-inventory assertion also consumes):

| Member                                       | Kind                  | Check                                                          |
|----------------------------------------------|-----------------------|----------------------------------------------------------------|
| `AbortController` (global)                   | `constructor`         | `typeof AbortController === "function"`                        |
| `AbortSignal` (global)                       | `constructor`         | `typeof AbortSignal === "function"`                            |
| `AbortController.prototype.abort`            | `prototype-method`    | `typeof AbortController.prototype.abort === "function"`        |
| `AbortSignal.any`                            | `static-method`       | `typeof AbortSignal.any === "function"`                        |
| `AbortSignal.timeout`                        | `static-method`       | `typeof AbortSignal.timeout === "function"`                    |
| `AbortSignal.prototype.throwIfAborted`       | `prototype-method`    | `typeof AbortSignal.prototype.throwIfAborted === "function"`   |
| `AbortSignal.prototype.addEventListener`     | `prototype-method`    | `typeof AbortSignal.prototype.addEventListener === "function"` |
| `AbortSignal.prototype.aborted`              | `prototype-property`  | `"aborted" in AbortSignal.prototype`                           |
| `AbortSignal.prototype.reason`               | `prototype-property`  | `"reason" in AbortSignal.prototype`                            |

The `prototype-property` rule is `in`-based, not `typeof`-based: `aborted` and `reason` are accessor properties whose getters throw `TypeError: Illegal invocation` when read off the prototype, so a `typeof === "function"` rule would refuse every conformant host. The probe MUST NOT instantiate `new AbortController()` to read instance members — the `in` operator against the prototype answers the question without depending on the constructors the probe is itself checking.

**(c) Factory-probable SDK capabilities.** Check the eight named function members listed below by `typeof <path> === "function"`. These are the runtime-detectable members of the **factory-probable subset** of the seven SDK capabilities owned by [SDK capability inventory](#sdk-capability-inventory): items 1, 2, 3, 4, and 6 (five capabilities, eight function members — capabilities 3, 4, and 6 each contribute two function members). Item 5 (cancellation propagation) is covered jointly by step (b) above (the WHATWG `AbortSignal` / `AbortController` shape probe) and step (c) below (the `AgentSession.prototype.abort` member listed alongside `createAgentSession`); see [SDK capability inventory](#sdk-capability-inventory) item 5 for the inventory-side framing. Item 7 (binder LLM model) is **not** factory-probable because `ctx.modelRegistry` does not exist at extension-factory entry; it is detected at per-loom load time and surfaces as `loom/load/binder-model-unresolved` per [Slash-Command Argument Binding — Strict-capability requirement](./binder.md#strict-capability-requirement), with no `details.kind` discriminator on this diagnostic. The two diagnostic surfaces (`loom/load/host-incompatible` and `loom/load/binder-model-unresolved`) MUST stay disjoint.

  - `pi.registerCommand` (capability 1)
  - `pi.sendUserMessage` (capability 2)
  - `ExtensionCommandContext.waitForIdle` (capability 2; checked at slash-handler invocation, not at factory time — the probe verifies only the static export shape on the `ExtensionAPI`-imported namespace)
  - `createAgentSession` (capability 3)
  - `AgentSession.prototype.abort` (capability 3 / capability 5; the cancellation primitive the runtime invokes from the one-shot `loomAbort.signal` listener registered at spawn time — see **Subagent session lifecycle** and **Cancellation source** below. Probed at factory time as `typeof AgentSession.prototype.abort === "function"` against the imported `AgentSession` class on the `@mariozechner/pi-coding-agent` namespace; the import path is load-bearing — `AgentSession` is re-exported by `dist/core/index.d.ts` of `@mariozechner/pi-coding-agent ~0.72.1` from `./agent-session.js`, and the loom runtime depends on the re-export specifically, not on direct consumption from `@mariozechner/pi-agent-core` (where the underlying class lives), so a Pi minor that drops the re-export surfaces at this probe as `loom/load/host-incompatible` rather than as a silent fallback to a different import path. A missing or renamed `abort` member surfaces through the same diagnostic rather than as a runtime-time `TypeError` at the spawn site)
  - `pi.registerTool` (capability 4)
  - `pi.setActiveTools` and `pi.getActiveTools` (capability 4)
  - `pi.registerMessageRenderer` (capability 6)
  - `pi.sendMessage` (capability 6)

**(d) Peer-dep version (lock-step, four packages).** This check is folded into step (c)'s same source-of-truth pinned-constants block but runs after the factory-probable capability checks complete (i.e. it is the last of the four pinned-constants checks). The probe iterates over the fixed array of four lock-step `@mariozechner/*` peer-dep packages in this normative order (the same source-of-truth list order shared with the H1 `pinned-surface.test.ts` per-package coverage assertion):

  1. `@mariozechner/pi-coding-agent`
  2. `@mariozechner/pi-agent-core`
  3. `@mariozechner/pi-ai`
  4. `@mariozechner/pi-tui`

  For each package in turn, read the installed version from its `package.json` via `createRequire(import.meta.url).resolve("<pkg>/package.json")` (resolution context is the loom factory's module, since the extension may be installed globally) and compare against the pinned range `~0.72.1` under tilde-range SemVer membership — the conventional npm tilde semantic, which pins the minor uniformly across all major versions (so `~0.72.1` admits `0.72.1` through `<0.73.0`). The check terminates on the first out-of-range or malformed package in this fixed iteration order — the four-package iteration shares step (c)'s short-circuit slot and does not aggregate failures across packages. A `version` field that is not a syntactically valid SemVer string, a missing `version` field, a `MODULE_NOT_FOUND` on the resolution call, or any other foreseeable shape failure of the read routes to `kind: "peer-dep-malformed-version"` with `details.observed = <raw-string-or-"<unresolvable>">`, `details.required = "~0.72.1"`, and `details.package = "<scoped-name>"` naming the offending package. An installed `version` that parses but falls outside the pinned range routes to `kind: "peer-dep-out-of-range"` with the same `details.package` field set. The `kind` discriminator enumeration in **On failure: refusal and diagnostic** below does not grow — `details.package` is a sibling field that refines the existing two `kind`s. Foreseeable shape failures are *not* `kind: "probe-failed"`; only unforeseen throws inside step (d) (permission errors, hostile `Proxy` on the resolved object, `JSON.parse` against a binary file) escalate to `kind: "probe-failed"` (see *Self-failure* below; `details.package` carries the offending package name on that route too).

  The fifth Pi-bundled package `typebox` is **not** included in this iteration — it is pinned to the literal `"*"` per the bundled-package convention (see **Host prerequisites — Pi SDK pin**, the `typebox` sub-paragraph) and therefore has no lock-step minor obligation. Its single load-bearing host-shape check (the presence of `Type.Unsafe` as a callable function) is owned by **Step 0 (e)** below, not by this iteration; the lock-step iteration MUST NOT be extended to cover `typebox`.

*Recommended recipe (non-normative).* `semver.satisfies(installedVersion, "~0.72.1")` and `semver.valid(installedVersion)` from the [`semver`](https://www.npmjs.com/package/semver) npm package satisfy this contract for each iteration step. A future swap to a different SemVer implementation is permitted as long as the tilde-range semantic above and the syntactic-validity discriminator that routes failures to `"peer-dep-malformed-version"` are preserved end-to-end.

**(e) `typebox` host-shape.** Check `typeof Type.Unsafe === "function"` against the `Type` namespace imported from the `typebox` peer dependency declared in `package.json`. Position in the short-circuit order: after `(d)`, since `(e)` depends on the `typebox` peer-dep import resolving and is therefore meaningful only once the lock-step `@mariozechner/*` peer-dep iteration in `(d)` has succeeded. Failure routes to `kind: "typebox-shape"` with `details.observed = <typeof Type.Unsafe>` (e.g. the literal string `"undefined"` when the namespace lacks the member) and `details.required = "function"`. The probe MUST NOT itself construct a TypeBox schema (no `Type.Unsafe<unknown>({})` call); the `typeof` check alone is sufficient and preserves the probe-wide invariant that the probe never exercises the member it is checking. `details.observed` carries the `typeof` string rather than the namespace contents to keep the diagnostic payload small. The asymmetry with `AgentSession.prototype.abort` in `(c)` is removed by adding this check: both single-symbol Pi-bundled imports now surface absence as `loom/load/host-incompatible` rather than as a runtime-time `TypeError` at the first tool-registration call.

For brevity the probe is described above as five sub-steps `(a)`–`(e)`; `(d)` is part of step `(c)`'s pinned-constants execution group and shares its short-circuit slot, and `(e)` is the trailing `typebox` host-shape check. The fixed normative order for short-circuit is: `(a)` Node floor, then `(b)` `AbortSignal` shape, then `(c)+(d)` capability list and peer-dep version, then `(e)` `typebox` host-shape. A simultaneous-failure host (e.g. Node 18 *and* a polyfilled `AbortSignal` missing `AbortSignal.any`) emits the `(a)` failure; the implementation MUST NOT race or aggregate.

**Probe-wide invariants.**

- The probe MUST NOT use any member it is itself checking. The kind table above is constructed so this rule holds without instance construction.
- The probe is `typeof`- and `in`-only — no arity sniffing, no return-shape sniffing, no construction.
- Probes MUST NOT be added beyond the five enumerated checks.
- The factory MUST NOT throw. Each check is wrapped in a try/catch (see *Self-failure* below).

**On failure: refusal and diagnostic.**

- (i) The factory skips **every** subsequent factory-time host-binding call — the four `pi.register*` calls **and** every `pi.on` call (and every `pi.subscribe` call, if the SDK exposes one). Steps 1–5 below MUST NOT execute. The refusal is enforced once at the `PiExtensionAPI` adapter layer, not by per-call defensive checks.
- (ii) The factory emits exactly one `loom/load/host-incompatible` diagnostic (per [Diagnostics — `loom/load/*`](./diagnostics.md)) with `details: { kind, observed, required, ...optional }` where `kind ∈ { "node-floor", "abortsignal-shape", "sdk-capability-missing", "peer-dep-out-of-range", "peer-dep-malformed-version", "typebox-shape", "probe-failed" }`. Routed through the **System notes** fallback chain below (`sendSystemNote` → `ctx.ui.notify` → `console.error`), since the renderer itself may be the missing capability.
- (iii) The factory returns normally.

**Self-failure.** If any check throws (including `process.versions` evaluation, `typeof` evaluation against a hostile getter or `Proxy`, `"in"`-operator evaluation against a `null` prototype, `package.json` resolution / read / parse outside the foreseeable shape failures step (d) routes to `"peer-dep-malformed-version"`, or a hostile getter / `Proxy` on the imported `typebox` namespace in step (e)), the probe traps the throw, terminates immediately, and emits `loom/load/host-incompatible` with `details.kind = "probe-failed"`, `details.step ∈ { "node-floor", "abortsignal-shape", "sdk-capability-missing", "peer-dep-out-of-range", "typebox-shape" }` naming which check threw, and `details.cause = <error.message>` carrying the underlying error text. When `details.step = "peer-dep-out-of-range"`, `details.package = "<scoped-name>"` additionally names which of the four lock-step packages was being checked at the moment of the throw; the deterministic iteration order pinned in (d) above guarantees this assignment is unambiguous. `details.cause` is the JS `Error.message` string, not the `Error` object — diagnostics are serialisable per the existing diagnostics contract. A `"probe-failed"` event supersedes any in-order `kind` for that check; the `(a)→(b)→(c)+(d)→(e)` short-circuit ordering applies only to comparison verdicts.

**Idempotency.** A `/reload` re-runs the probe with no carried state. Subagent mode is unaffected because it runs only after a session has been created, by which point step 0 has already gated the factory.

<a id="post-probe-sdk-shape-drift"></a>
**Post-probe SDK-shape drift.** The capability probe above runs **exactly once** per factory invocation (a `/reload` is a fresh factory invocation per *Idempotency* above) and is **not** re-run on subsequent SDK calls. Any post-probe interaction with a previously-probed Pi member — including the snapshot-side `pi.setActiveTools` of the swap, the per-invocation `pi.sendUserMessage` call, the per-call `ctx.modelRegistry.find` re-resolution of the binder model handle (which [Slash-Command Argument Binding](./binder.md) constructs afresh on every binder call), `createAgentSession` and `AgentSession`-handle members past creation, `pi.sendMessage` for system notes, and any `pi.register*` or `pi.on` member touched after factory entry — that **throws synchronously, returns a Promise that rejects, or returns a value of the wrong shape** routes to `loom/runtime/internal-error` per the catch-all row in [Diagnostics](./diagnostics.md), *except* where one of the four dedicated codes already applies: `loom/runtime/active-set-restore-failed` (the **restore-side** `pi.setActiveTools` of the swap, after a successful snapshot-side call; the snapshot-side throw remains on the `internal-error` route and aborts the swap before the wrapped `fn` runs), `loom/runtime/system-note-delivery-failed` (synchronous throw from `pi.sendMessage` for a `loom-system-note`, after the `ctx.ui.notify` fallback has been attempted), `loom/runtime/subagent-dispose-failure` (`AgentSession.dispose()` throw inside the subagent-mode teardown `finally`), and `loom/runtime/registry-swap-failed` (the watcher's mid-rebuild throw on a changed `.loom`/`.warp` file). `pi.sendUserMessage` returns `void` per the `ExtensionAPI` surface (the `Promise<void>` shape is exclusive to `AgentSession.sendUserMessage` and `ReplacedSessionContext.sendUserMessage`); the runtime MUST NOT `await` it, so its observable post-probe failure mode is the synchronous-throw arm above. **Silent shape drift** — a previously-probed member that no longer matches its probed shape but does not throw, reject, or return a wrong-shape sentinel the catch-all wrap can observe (e.g. `pi.setActiveTools` becoming a `Promise<void>`-returning function whose rejection the runtime never sees because it does not `await`) — is **undefined behaviour**: the runtime is not required to detect it, and no diagnostic is owed for it. Operator triage of `loom/runtime/internal-error` therefore cannot, in V1, distinguish a host-drift failure from a loom-runtime defect on the `error.message` / `error.stack` payload alone; both surface through the same `"loom /<name> aborted with internal error: <message>"` system-note template (and the same `Err(InvokeInfraError { kind: "invoke_failure", cause: "internal_error", ... })` envelope on the `invoke` parent surface). A `details.kind` discriminator on `loom/runtime/internal-error` distinguishing host-drift from loom-defect is an additive registry-minor change deferred past V1 per [Diagnostics](./diagnostics.md)'s registry-change rules; nothing in the V1 routing contract precludes adding it later without breaking emitted shapes.

If step 0 succeeds, the factory then:

1. Registers the `--loom` CLI flag via `pi.registerFlag('loom', { type: 'string', description: 'Loom file or directory paths, joined with the OS path-list separator' })` **before** subscribing to `resources_discover`, so the value is readable from inside the discovery handler via `pi.getFlag('loom')`. Walks the five discovery sources defined in [Directory Convention](./discovery.md) directly — the global directory `~/.pi/agent/looms/`, the project directory `.pi/looms/`, every installed package's `pi.looms` entry or conventional `looms/` directory (per [Package discovery](./discovery.md#package-discovery)), the `looms` settings array (per [Settings file reads](./discovery.md#settings-file-reads)), and the `--loom` CLI flag (whose `pi.getFlag('loom')` value is split on `path.delimiter` per [Directory Convention](./discovery.md)) — at factory time and again on every `resources_discover` event. The event payload (`{ cwd, reason }`) supplies the project root used for the project-local source (`event.cwd` is preferred over any factory-captured `cwd`, so a future per-session cwd change is honoured on reload) and distinguishes initial scan (`reason: "startup"`) from reload (`reason: "reload"`); Pi delivers no paths to the handler — `ResourcesDiscoverResult` has no `loomPaths` slot — and the `pi` manifest namespace does not enumerate `pi.looms`. The loom extension owns the walk for all five sources; the event is purely a re-trigger, and the handler returns the empty result `{}` to satisfy the typed return.
2. Parses each discovered `.loom` file and builds a *pending registration list* keyed by slash name. The factory does **not** call `pi.registerCommand` synchronously — Pi's prompt templates and other extensions' commands are not yet enumerable at factory time, so cross-format collision detection cannot run here. Concretely, on the pre-bind runtime returned to the extension factory by `createExtensionRuntime()`, every action method that reads or mutates session-bound state — including `getCommands`, `getActiveTools`, `getAllTools`, `setActiveTools`, `sendMessage`, `sendUserMessage`, `setModel`, `getThinkingLevel`, `setThinkingLevel`, and the rest of the `notInitialized`-tagged slots in `@mariozechner/pi-coding-agent`'s `dist/core/extensions/loader.js` — is bound to a throwing closure (`Error("Extension runtime not initialized. Action methods cannot be called during extension loading.")`) that `Runner.bindCore()` later overwrites with the real implementation alongside `session_start`. The token `notInitialized` is the internal variable name of that throwing closure inside Pi's loader source; it is **not** an observable sentinel value an extension can compare against or switch on. Calling any of these methods from inside the factory raises an unhandled exception that aborts factory load. The methods Pi documents as safe before bind are `pi.registerCommand` / `pi.registerTool` / `pi.registerFlag` / `pi.registerMessageRenderer` (registration sinks) and `refreshTools` (a no-op on the pre-bind runtime). Every other reach for session state is deferred to a `session_start` (or later) handler. The signature pinned by V1 is `pi.getCommands(): SlashCommandInfo[]` (synchronous, returns the current registered-commands snapshot, where `SlashCommandInfo` is the interface exported from `@mariozechner/pi-coding-agent`'s `core/slash-commands.d.ts` with fields `{ name, description?, source: SlashCommandSource, sourceInfo }`); the returned array is mutable in the type system but MUST be treated as read-only by convention (the V1 collision check is a single forward pass and never mutates it); the snapshot is only meaningful on or after `session_start`. The build-time SDK surface inventory (per *Host prerequisites*) MUST list `SlashCommandInfo` and the three-arm `SlashCommandSource = "extension" | "prompt" | "skill"` explicitly so a Pi minor that renames either fails the build.
3. Subscribes a `session_start` handler that consults `pi.getCommands()`, drops any pending loom whose slash name collides with an existing entry whose `source` is in the [V1 cross-format collision source set](#v1-seam-pi-owned-subagents-collision-source-set) defined below — the three arms of Pi's `SlashCommandSource` union (`"prompt"`, `"extension"`, `"skill"`), per `@mariozechner/pi-coding-agent`'s `core/slash-commands.d.ts` and per [Discovery — Slash-name collisions across formats](./discovery.md) — registers each survivor via `pi.registerCommand(name, { description, getArgumentCompletions, handler })` — one slash command per loom — and emits a `loom/load/cross-format-collision` diagnostic for each drop through the channel defined in [Diagnostics](./diagnostics.md). The `description` field passed to `pi.registerCommand` carries `frontmatter.description` verbatim and only that — `argument-hint` is **not** concatenated, since Pi's extension-command autocomplete UI has no `argumentHint` slot and the binder is the only V1 consumer of `argument-hint` (see [Slash-Command Argument Binding](./binder.md)). The same handler also de-registers any previously-registered loom whose name now collides with a newly-appeared `.md` template or extension command (e.g. after a settings reload or another extension's activation), emitting the same diagnostic.
4. Subscribes a `session_shutdown` handler (`pi.on("session_shutdown", handler)`) that owns disposing every resource the extension instance holds. The Pi SDK documents `session_shutdown` as fired before an extension runtime is torn down due to quit, reload, or session replacement, with `event.reason: "quit" | "reload" | "new" | "resume" | "fork"`; loom uses it as the single teardown contract for the outgoing extension instance so that Pi's subsequent `ExtensionRuntime.invalidate(...)` call does not strand subagent provider connections, chokidar file handles, debounce timer handles, or `AbortController` listeners across `/reload`, `/new`, fork, or quit.

   **Unknown-reason rule.** Before sub-step 1 runs, the handler validates `event.reason` against the closed set `{"quit", "reload", "new", "resume", "fork"}`. The validation read of `event.reason` MUST be wrapped in a `try`/`catch` so a property-access throw (e.g., a getter that throws) is treated as an unknown reason rather than escaping the handler. The set comparison is the only pre-sequence branch the handler MAY take. On an unknown reason — or on a throwing access — the handler emits exactly one `loom/host/session-shutdown-reason-unknown` (W, runtime) diagnostic via `console.error` with `details: { observed: <String(event.reason)> }` *before* sub-step 1 runs (so the diagnostic survives even if the handler later throws), then runs the full five-sub-step sequence as if the reason were `"quit"` or `"reload"`. The no-active-invocations fast-path described in the *Edge cases* paragraph below MUST NOT be taken on the unknown-reason path: the registry drain, the bounded `Promise.allSettled` await, the watcher closure, and the listener detachment all run unconditionally. The `String(event.reason)` coercion in the `details.observed` payload tolerates symbols, `undefined`, numbers, booleans, and objects without itself throwing; on a throwing property access (where the value never materialised) `details.observed` is the literal string `"<unreadable>"`. The handler captures this single coerced string into a handler-scoped variable at handler entry — the same site this rule's `try`/`catch` runs, before sub-step 1 — and the same string is reused as `details.event.reason` by every `loom/runtime/cancelled-by-session-shutdown` note emitted under the **Per-invocation operator visibility (clean-cancel path)** rule below; the per-invocation `finally` does not re-read `event.reason`. The captured value is a closed-set member when Pi delivered one of the five literals (`"quit" | "reload" | "new" | "resume" | "fork"`), `String(event.reason)` on the unknown-reason path, or `"<unreadable>"` on the throwing-access path. Pin the coercion to `String(event.reason)` explicitly — a template-literal coercion (`` `${event.reason}` ``) re-introduces the symbol-throw bug `String()` is chosen to avoid.

   The handler runs in fixed order regardless of `event.reason`:

   1. **Stop accepting new work.** Mark the extension-scoped `LoomRegistry` as drained; on entry into a drained registry the slash-command `handler` registered in step 3 returns the same cancelled-binder-style `loom-system-note` (`content: "loom /<name>: extension shutting down"`, `display: true`) used elsewhere for binder cancellation, without dispatching the loom.
   2. **Cancel in-flight invocations.** For every entry in the **`ActiveInvocationRegistry`** (introduced below), call `loomAbort.abort(reason)` where `reason` is the synthesised teardown value pinned by [Cancellation — Abort-reason propagation](./cancellation.md). This propagates through the [`Checkpoint` seam](#checkpoint-seam) and, in subagent mode, through the one-shot `loomAbort.signal` listener that the runtime registered at spawn time and that calls `AgentSession.abort()` on the spawned session — per [Cancellation — Forwarding into `loomAbort`](./cancellation.md). The listener is detached in the same per-invocation `finally` that calls `AgentSession.dispose()` (see **Subagent session lifecycle** below).
   3. **Await subagent disposal.** `await Promise.allSettled(activeInvocations.map(inv => inv.disposeBarrier))`, where `disposeBarrier` is the promise that the **Subagent session lifecycle** `finally` block below settles after `AgentSession.dispose()` returns (subagent mode) or settles immediately (prompt mode). The await is bounded by the constant `SHUTDOWN_AWAIT_CAP_MS = 2000` measured against the runtime's injected [`Clock`](#clock--fakeclock-interface) seam: at handler entry the runtime captures `deadline = Clock.now() + SHUTDOWN_AWAIT_CAP_MS`, and the await resolves on the earlier of `Promise.allSettled(...)` settling or `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, ...)` firing. The deadline is absolute, not a refreshing slide — a slow `loomAbort.abort()` propagation in sub-step 2 does not extend the await. The pending `Clock.setTimeout` handle is cleared on the success path so a reload that completes early does not leak a timer onto the about-to-be-invalidated runtime. On timeout the runtime emits one `loom/runtime/reload-teardown-timeout` diagnostic naming the still-in-flight invocations and proceeds to step 4. The `SHUTDOWN_AWAIT_CAP_MS` constant lives in the same source-of-truth pinned-constants block as the Step 0 capability-probe constants so the build-time literal-read assertions can assert it.
   4. **Close watchers.** Call `discoveryWatcher.close()` (the chokidar watcher established in step 5 below) and `settingsWatcher.close()` (the settings-file watcher per [Discovery — Settings file reads](./discovery.md#settings-file-reads)); cancel any pending debounce timer handles via `Clock.clearTimeout`.
   5. **Detach forwarding listeners.** Remove every listener the runtime attached to Pi-side `ctx.signal`, tool `execute(signal)`, and parent-`invoke` signals — the same listeners [Cancellation](./cancellation.md) requires the per-invocation `finally` to remove, duplicated here for the reload-during-invocation path where Pi tears the runtime down before that `finally` ever reaches its forwarding-listener cleanup.

   **Per-step isolation.** Sub-steps 1, 4, and 5 each run inside their own `try`/`catch`; a throw from any of them MUST be caught, MUST emit exactly one `loom/host/session-shutdown-teardown-step-failed` (W, runtime) diagnostic via `console.error` with `details: { step: 1 | 4 | 5, call: <static call-site label>, error: <error.message> }`, and MUST NOT prevent the remaining sub-steps from running. Within sub-step 4 the rule applies independently to each of `discoveryWatcher.close()`, `settingsWatcher.close()`, and the debounce-handle `Clock.clearTimeout(...)` call (three potential emissions, one per failing call). Within sub-step 5 the rule applies independently to each enumerated listener-detach call. Sub-steps 2 and 3 retain their existing per-entry isolation rules (per-entry `try`/`catch` around `loomAbort.abort()`; `Promise.allSettled` over `disposeBarrier`s) and do not emit `teardown-step-failed`. The static call-site label in `details.call` is a fixed string per call site (e.g. `"discoveryWatcher.close"`, `"settingsWatcher.close"`, `"Clock.clearTimeout(debounce)"`, `"ctx.signal.removeEventListener"`) so deduplication on `(code, details.step, details.call)` is meaningful across runs. When the captured throw is not an `Error` instance, `details.error` is `String(error)`, falling back to the literal string `"<unreadable>"` if that coercion itself throws — symmetric with the `details.observed` convention pinned by the **Unknown-reason rule** above. The `console.error` call this paragraph mandates is itself wrapped per the **Diagnostic-emission isolation** rule defined under *Edge cases* below; a throw from `console.error` MUST be swallowed and MUST NOT prevent the remaining sub-steps from running. The new code is **not** added to the always-log set defined under **Runtime event channel** below, matching the existing `reload-teardown-timeout` and `session-shutdown-reason-unknown` exclusions.

   The handler is **idempotent**: a second `session_shutdown` fired before the first returns is a no-op (the second invocation observes the registry already drained and returns immediately). It does **not** call `ctx.reload()` (Pi is already executing it) and does **not** call `pi.unregisterTool` (Pi exposes none; the registration-cache leak is the V1 cosmetic acceptance documented under **Tool-registration lifetime and visibility** below).

   <a id="active-invocation-registry"></a>

   The **`ActiveInvocationRegistry`** is an extension-instance-scoped `Set<{ loomAbort: AbortController; disposeBarrier: Promise<void> }>` that the runtime maintains in lock-step with each loom invocation: the slash-command `handler`, the `tool.execute(...)` adapter for tool-exposed looms, and the parent-`invoke` spawn site each insert their own entry on entry and remove it from the same `finally` block that already disposes any subagent session (per **Subagent session lifecycle** below). The `disposeBarrier` is the promise that the per-invocation `finally` settles after `AgentSession.dispose()` returns (subagent mode) or immediately (prompt mode). Concurrent loom invocations therefore each carry one entry; the registry is the structure the teardown handler iterates.

   *Registry contract.*

   - **Insertion** happens at slash-command handler entry, `tool.execute(...)` adapter entry, and `invoke` spawn-site entry, before any awaitable work.
   - **Removal** happens in the same `finally` block that disposes the subagent `AgentSession`, after `disposeBarrier` settles.
   - **Dispatch-site setup wrap.** Each of the three insertion sites MUST wrap the pre-evaluation setup sequence — `new AbortController()` for `loomAbort`, the registry-`Set.add` insertion, the per-entry-point `source.addEventListener('abort', …)` attach that registers the forwarding listener on the inbound Pi-side `AbortSignal` (the slash-command handler's `ctx.signal`, the tool-exposed adapter's `signal` parameter, or the `invoke`-parent's parent-signal — see [Cancellation — Forwarding into `loomAbort`](./cancellation.md)), and (subagent mode only) the awaited `createAgentSession(...)` call followed by the post-resolution `loomAbort.signal.addEventListener('abort', …)` attach that wires the one-shot `AgentSession.abort()` listener — in a `try`/`catch` whose `catch` arm routes the captured throw or rejection through the **runtime-defect surface** defined in [Errors and Results — Runtime panics](./errors-and-results.md#runtime-panics) (single rule, three call sites; the `createAgentSession`-rejects paragraph in **Subagent session lifecycle** below is a worked example of this rule rather than a one-off). No new `details.kind` discriminator is introduced for the listener-attach throws — the existing `loom/runtime/internal-error` surface carries `error.message` and `error.stack` and that is sufficient for triage. On a throw before the `Set.add` insertion completes no registry entry leaks (nothing to remove); on a throw after insertion the same per-invocation `finally` removes the entry and settles `disposeBarrier`. The `cause: "internal_error"` arm of [`InvokeInfraError`](./errors-and-results.md#invoke-variants) and the slash-command system-note template `"loom /<name> aborted with internal error: <error.message>"` defined in [Errors and Results — Runtime panics](./errors-and-results.md#runtime-panics) apply unchanged. A throw from `loomAbort.abort()` invoked by the `catch` arm during cleanup of a half-constructed entry is itself swallowed by the per-entry `try`/`catch` of the **`session_shutdown`** sub-step 2 rule above and MUST NOT mask the original setup throw.
   - **Iteration order** in the `session_shutdown` handler's sub-step 2 (`loomAbort.abort()` over every entry) and sub-step 3 (`Promise.allSettled` over every entry's `disposeBarrier`) is insertion order, matching the V8 `Set` insertion-order invariant the rest of the spec already relies on. The order is observable to tests asserting on the order of `loom/runtime/reload-teardown-timeout`'s `<list>` rendering.
   - **`loomAbort.abort()` throwing inside the sub-step 2 iteration** is swallowed by a per-entry `try`/`catch`; the handler continues to the next entry. No diagnostic is emitted (the abort is best-effort and the entry's own `finally` will still settle `disposeBarrier`, which sub-step 3 already awaits with `Promise.allSettled`).
   - **The registry name is internal.** It is not part of the loom-facing or extension-facing contract; tests assert on observable side effects (entry counts via probe seams, ordered `loomAbort.abort()` calls, `disposeBarrier` settlement) rather than on the symbol itself.
   - **Per-mode concurrency invariants.** The registry tracks both prompt-mode and subagent-mode entries (so the `session_shutdown` teardown above reaches every in-flight invocation); the *insertion shape* differs by mode, but the entry shape and the iteration rule do not. *Subagent-mode* entries may be genuine concurrent siblings: parallel tool calls into the same `.loom` callable, sibling `invoke(...)` sites whose children resolve to subagent mode, and parent→child stacks of subagent-mode invocations all coexist as distinct registry entries that progress on the event loop independently. *Prompt-mode* entries are constrained to **linear nesting** within a single user session: at most one prompt-mode body holds an open `pi.setActiveTools` snapshot/restore window at a time, but a prompt-mode parent suspended on an `invoke(...)` to a prompt-mode child legitimately keeps its own registry entry while the child's entry is also live (the parent is suspended at the call site, not dispatched, and only the child's snapshot/restore window is open). The teardown `loomAbort.abort()` iteration in sub-step 2 above must reach the suspended parent's entry as well as the child's. The supporting mechanisms that pin prompt-mode linear nesting — Pi's per-session slash-handler serialisation pinned under **Tool-registration lifetime and visibility** below, the load-time rejection of prompt-mode `.loom` callees in another loom's `tools:` per [Parameters and Frontmatter — `tools`](./frontmatter.md#tools), and the suspend-on-invoke rule from [Invocation — Cross-mode semantics](./invocation.md) — are owned by those pages; the registry contract above relies on them but does not restate them. Subagent-mode children spawned from a prompt-mode parent are genuinely concurrent and add their own registry entries on the same insertion-site rules above.

   *Edge cases the handler must observe:*

   - `ExtensionRuntime.invalidate()` may already have been called by Pi when the handler runs; any `pi.*` call that reaches the stale runtime throws. The teardown sequence above touches only loom-internal state and the SDK objects (`AbortController`, chokidar `FSWatcher`, `AgentSession`) directly — it does not call back into `pi.*`.
   - The `sendSystemNote` fallback chain (the best-effort chain defined under **System notes** below) MUST NOT be invoked from the teardown handler — `pi.sendMessage` against a stale runtime throws, and the fallback to `ctx.ui.notify` would re-enter the same stale surface. Diagnostics emitted from the teardown handler use `console.error` directly, bypassing the persistent-diagnostic channel defined in [Diagnostics](./diagnostics.md). Three diagnostics may be emitted from this handler, all via `console.error`: `loom/runtime/reload-teardown-timeout` (referenced in sub-step 3), `loom/host/session-shutdown-reason-unknown` (referenced in the **Unknown-reason rule** above), and `loom/host/session-shutdown-teardown-step-failed` (referenced in the **Per-step isolation** paragraph above). No other diagnostic code may be emitted from this handler.
   - <a id="diagnostic-emission-isolation"></a> **Diagnostic-emission isolation.** Each `console.error` call inside the `session_shutdown` handler — the three teardown-handler emissions enumerated in the bullet immediately above (`loom/runtime/reload-teardown-timeout`, `loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-teardown-step-failed`) — MUST be wrapped in `try`/`catch`. A throw out of `console.error` (closed stdio, file-descriptor exhaustion, a host that proxies `console` and rejects writes during teardown) MUST be swallowed; the handler MUST continue to the next sub-step (or, for the **Unknown-reason rule** emission that runs *before* sub-step 1, MUST continue into sub-step 1). No further diagnostic is emitted on the inner throw — the only sink available to the teardown handler has already failed and the handler is the bottom of the stack with no further fallback. The wrap is around the `console.error` call only; the `String(event.reason)` coercion that builds the **Unknown-reason** diagnostic's `details.observed` payload, and the `String(error)` / `"<unreadable>"` coercion that builds the **Per-step isolation** diagnostic's `details.error` payload, already tolerate non-string values per the rules above and are outside the wrap. Emission ordering is preserved: a swallowed `console.error` throw leaves the handler in the same position it would be after a successful emit (in particular, the **Unknown-reason rule**'s "emits *before* sub-step 1 runs" obligation is satisfied even when the emission inner-throws). The rule is symmetric for all three teardown-handler diagnostic codes; no per-code carve-outs. The same isolation MUST be applied to the terminal `console.error` of the `sendSystemNote` fallback chain when that chain is reached from the per-invocation `finally` of the **Per-invocation operator visibility (clean-cancel path)** paragraph below: a throw from that terminal `console.error` MUST be swallowed and MUST NOT propagate out of the per-invocation `finally`.
   - `reason: "new" | "resume" | "fork"` does not always tear down the *extension* runtime (only the user session), but Pi's `session_shutdown` type doc states the event "is fired before an extension runtime is torn down" — the handler runs the full fixed five-sub-step sequence — including watcher closure in sub-step 4 — for every reason in the closed set, and the V1 position accepts that watchers may be closed on a reason (e.g. `"fork"`) that did not in fact tear the extension down; recovering watcher state in that case is one `/reload` away and is not load-bearing for V1. The fast-path framing applies to sub-steps 2 and 3 only (the abort-and-await over `ActiveInvocationRegistry`): a no-teardown reason makes those two sub-steps trivially empty because no active invocations exist at session boundaries (Pi serialises turns), but it does not skip sub-step 4 or any other sub-step. This emergent fast-path is available only for the five enumerated reasons; an unknown `event.reason` is routed by the **Unknown-reason rule** above and MUST run the full five-sub-step sequence even when the registry is empty.

   <a id="session-shutdown-semantics"></a>

   *Session-swap behaviour for in-flight invocations.* The five-sub-step sequence above defines the abort-and-await sequence; this paragraph closes the three obligations the sequence does not by itself state — per-invocation operator visibility on the cancellation path, partial-append fate during teardown, and the `invoke`-parent observation rule.

   - **Per-invocation operator visibility (clean-cancel path).** For each `ActiveInvocationRegistry` entry whose `disposeBarrier` settles inside the `SHUTDOWN_AWAIT_CAP_MS` window of sub-step 3 — i.e., the invocation cancelled cleanly within the bounded await — the cancellation path inside that invocation's own `finally` block emits exactly one `loom/runtime/cancelled-by-session-shutdown` (E, runtime) note on `loom-system-note` with `display: false`, `details.event.reason` carrying the handler-captured `event.reason` string typed as `"quit" | "reload" | "new" | "resume" | "fork" | string`: a closed-set member when Pi delivered one of the five literals, otherwise the `String(event.reason)` coercion captured by the **Unknown-reason rule** above (or the literal string `"<unreadable>"` when the property-access read threw). The wider `string` arm is the unknown-reason path; the value is always a string. The handler-captured string is the same single coerced value the **Unknown-reason rule** populates `details.observed` with — both diagnostics surface byte-identical reason strings for the same shutdown event, and `details.event.loom` carrying the invocation's registered slash name (without the leading `/`). The note is emitted from the invocation's own teardown — not from the `session_shutdown` handler itself — so the "`sendSystemNote` MUST NOT be invoked from the teardown handler" edge-case bullet above does not apply to it; the note routes through the standard `sendSystemNote` fallback chain (`pi.sendMessage` → `ctx.ui.notify` → `loom/runtime/system-note-delivery-failed` → `console.error`) defined under **System notes** below, which degrades through `ctx.ui.notify` and ultimately `console.error` if `ExtensionRuntime.invalidate(...)` has already invalidated `pi.sendMessage` by emission time. The chain's terminal `console.error` is wrapped per the [**Diagnostic-emission isolation**](#diagnostic-emission-isolation) rule above: a throw from that terminal `console.error` MUST be swallowed and MUST NOT propagate out of the per-invocation `finally`. An invocation either emits `cancelled-by-session-shutdown` (cleanly cancelled within the window) or contributes to the single `loom/runtime/reload-teardown-timeout` `<list>` of sub-step 3 (still in flight at the deadline), never both — the per-invocation note is the operator-visibility counterpart to the timeout diagnostic. The `cancelled-by-session-shutdown` code is **not** in the always-log set defined under **Runtime event channel** below; the exclusion mirrors the existing `cancelled` exclusion (operator-initiated, self-explanatory in context) and so the per-invocation note does not produce a `details: { event: RuntimeEvent }` group-A emission in addition to itself.

   - **Partial-append fate during teardown.** Conversation turns the runtime had already appended into the user session (prompt mode) or the spawned subagent session (subagent mode) before sub-step 2's `loomAbort.abort()` fired remain in whatever conversation Pi assigns them to under the swap's `event.reason`; the runtime makes no rollback, migration, or compensation attempt. This is the `event.reason`-scoped restatement of the [Errors and Results — Partial-append contract](./errors-and-results.md#partial-append-contract): `?`-propagation, panic, and steady-state cancellation already preserve appended turns, and a `session_shutdown`-driven cancellation preserves them on the same rule. The outgoing user session (when Pi tears it down on `reason: "new" | "resume" | "fork"`) carries the partial appends Pi assigns to it under the swap; the loom runtime makes no attempt to discard them, migrate them to the incoming session, or surface a manifest of what was appended. Authors who need teardown-safe append semantics must use [Errors and Results — No rollback](./errors-and-results.md#no-rollback)'s steady-state author obligations (idempotency, compensation in `match` arms before re-`Err`-ing).

   - **`invoke`-parent observation rule.** Parent and child invocations are aborted in the same iteration of sub-step 2: the registry is iterated once and `loomAbort.abort()` is called on every entry, parents and children alike. The parent's `?` / `match` arms covering the child's `Err(QueryError { kind: "cancelled" })` (or `kind: "invoke_callee_error"` wrapping it, per [Cancellation — Surfacing](./cancellation.md)) do **not** run inside the bounded-await window of sub-step 3: the parent's own `loomAbort` has already fired in the same iteration, its next [`Checkpoint` seam](#checkpoint-seam) await observes the abort, and the parent unwinds without re-entering interpreter dispatch. The parent's `disposeBarrier` settles independently of whether its child surfaced an `Err` to it — the registry treats every entry symmetrically. The downward-only propagation rule of [Cancellation — Propagation](./cancellation.md) is not contradicted: parent and child are not propagating into each other, they are both aborted by the same external `session_shutdown` trigger, and the parent's `Err`-handling arms remain inert during teardown for the same checkpoint reason that any cancelled invocation's post-abort code is inert.

5. Registers a chokidar file watcher over the discovered roots and maintains an internal mutable `LoomRegistry` (a `Map<slashName, parsedLoom>`) plus a `.warp` import graph. The slash-command `handler` registered in step 3 closes over the registry and dispatches by slash name on every invocation, so a registry swap takes effect on the next call without a `pi.registerCommand` re-registration. On a chokidar event for an **existing** loom or `.warp` file the watcher debounces (250 ms window, to coalesce editor `change` + `rename` bursts; the 250 ms window is measured against the runtime's injected `Clock` seam (per the runtime's DI skeleton — see [`Clock` / `FakeClock` interface](#clock--fakeclock-interface) below), the seam exists to make burst-coalescing assertions deterministic and is not a tunable runtime knob), re-parses just the changed file plus every transitive `.warp` importer reached through the import graph, and atomically swaps the affected entries in `LoomRegistry` via build-aside-then-publish: the runtime re-parses the changed file plus every transitive `.warp` importer into a staging set, recompiles their AJV validators into a staging cache, and only after every step succeeds installs both into `LoomRegistry` and the validator cache in a single synchronous step. If any rebuild step throws, the staging set is discarded and the prior `LoomRegistry` snapshot and AJV validator cache remain live; the failure surfaces as a single `loom/runtime/registry-swap-failed` diagnostic (see [Diagnostics](./diagnostics.md)). Compiled AJV validators for swapped entries are dropped from the per-loom cache as part of the publish step (not before). Tools synthesised from prompt-mode `.loom` callables are re-registered through `pi.registerTool` when the swap produces a new schema slug (`pi.registerTool` is documented as supported after startup); existing cached registrations under unchanged slugs are reused. The runtime does **not** call `ctx.reload()` for content edits — doing so would tear down every Pi extension, skill, prompt, and theme in the session for a single keystroke and would also race the post-`await` runtime-deadness rule that `ctx.reload()` documents.

   **In-flight invocation rule.** `LoomRegistry.dispatch` reads the entry exactly once at handler entry; a swap mid-execution does not affect a running invocation. A *drained* registry (set by the `session_shutdown` handler in step 4 above) short-circuits dispatch with the shutdown system note instead of consulting the entry table. The next invocation sees the post-swap version. In-flight invocations therefore complete against the pre-swap snapshot and any tool calls or queries they have already committed remain final, consistent with [Errors and Results — No rollback](./errors-and-results.md).

   **Structural changes.** Adding a brand-new `.loom` file or removing an existing one cannot register / unregister a slash command after extension load (Pi exposes no symmetric `pi.unregisterCommand` for the removal case, and adding a new command after the extension factory has run is not a documented post-startup mutation). When the watcher observes such an event, it emits a single `loom-system-note` with `content` formatted as `loom watcher: <N> file(s) added or removed; run /reload to refresh the slash command list`, `display: true`, and the `details` payload defined under System notes. The substitution rule is fixed: only `<N>` is substituted; every other character of the template — including the literal token `file(s)` (always rendered with the parenthesised `s`, never pluralised), the literal slash-command name `/reload` (not substituted from any registered command), and the trailing `; run /reload to refresh the slash command list` clause — ships verbatim regardless of `N`. `<N>` is rendered in base 10 with no thousands separator, no leading zero, and no sign, and equals `details.structural.added.length + details.structural.removed.length` (a path that appears in both arrays — e.g., a rename observed as removal-then-addition within the same debounce window — counts twice, since the two arrays are disjoint by role and not deduplicated against each other). Worked examples: with `added.length = 1, removed.length = 0`, the rendered content is `loom watcher: 1 file(s) added or removed; run /reload to refresh the slash command list`; with `added.length = 3, removed.length = 2`, it is `loom watcher: 5 file(s) added or removed; run /reload to refresh the slash command list`. The note MUST NOT be emitted when `added.length + removed.length === 0`. A debounce window that closes with no resolved add or remove paths produces no `loom-system-note` — including a settings-file edit whose post-merge `looms` array is byte-identical to the previous resolved set, an edit that touches only `looms.binderModel` or unrelated keys, and a chokidar burst over discovery roots that does not net any added or removed `.loom` / `.warp` files. The watcher's other obligations for the window (validator-cache invalidation per the **In-flight invocation rule** above, settings-file cache invalidation per the settings-file watcher path) still run; only the note is suppressed. A rename observed within the same debounce window as `removed` of path P followed by `added` of path P still has `added.length + removed.length === 2` and MUST emit the note (the two arrays are disjoint by role and not deduplicated against each other, per the rule above); the suppression applies only to the strictly-empty case. The note prompts the user to run `/reload` (which invokes Pi's `ctx.reload()` and re-executes the entire extension factory). Multiple structural events observed within a single 250 ms debounce window coalesce into one note; events in distinct windows fire separately. The note is informational, not a diagnostic; no `loom/load/*` code is registered for it. Settings-file edits affecting the `looms` array are treated as structural changes and routed through the same prompt; per-`.loom`-file content edits are not.

**Extension-bootstrap SDK failures.** A throw or rejection from any of the registration calls in steps 1–5 above is fatal at the granularity of the failing surface; the granularity rule is per call type, not per call site:

- **`pi.registerMessageRenderer` failure (factory-time, step 0 having succeeded so the capability is present but the call itself rejects).** The renderer registration is dropped; the extension factory still completes the remaining steps. System notes for this extension instance permanently degrade to the `ctx.ui.notify` arm of the **System notes** fallback chain below (the persistent-transcript surface is unavailable; the transient toast and the `console.error` last-resort arms remain). The runtime emits `loom/load/extension-bootstrap-failed` (E, load) with `details: { capability: "pi.registerMessageRenderer", error: <error.message> }` and proceeds.
- **`pi.registerCommand` failure (factory-time *or* `session_start`-time, per-loom).** Only the loom whose registration call failed is dropped; sibling looms in the pending registration list still register through their own `pi.registerCommand` calls. The runtime emits one `loom/load/extension-bootstrap-failed` (E, load) per failing loom with `details: { capability: "pi.registerCommand", loom: <slash-name>, error: <error.message> }`. The factory does not re-attempt the failing call within the same factory invocation; a `/reload` is the recovery path.
- **`pi.registerFlag` failure (factory-time, step 1).** Fatal to the whole extension because step 1's `--loom` flag value is what step 1's discovery walk and every subsequent `resources_discover` re-walk consume via `pi.getFlag('loom')`; a flag-less factory cannot honour the `--loom` source. The factory skips every subsequent `pi.register*` and `pi.on` call (steps 2–5 do not execute, mirroring the step-0 failure semantics) and emits a single `loom/load/extension-bootstrap-failed` (E, load) with `details: { capability: "pi.registerFlag", error: <error.message> }`.

The `loom/load/extension-bootstrap-failed` diagnostic routes through the same channel that step 0's `loom/load/host-incompatible` uses (the **System notes** fallback chain — `sendSystemNote` → `ctx.ui.notify` → `console.error`), because the renderer may itself have failed. The factory MUST NOT throw out of `default function (pi: ExtensionAPI)`; per-call `try`/`catch` around each step keeps the failure local to its granularity rule above. None of these three rules apply to the step-0 capability probe — a missing capability is `loom/load/host-incompatible` and stops the factory before it ever attempts a `pi.register*` call. `pi.registerTool` failures at factory time are governed by the prompt-mode registration cache rule under **Tool-registration lifetime and visibility** below; this paragraph does not duplicate that rule.

**Per-loom registration.** For each `.loom` file:

- The slash-command `handler` runs the binder (when applicable) and then the loom interpreter against the appropriate conversation.
- When a loom is referenced by another loom's `tools:` (i.e., exposed as a tool to a model), the runtime materialises a Pi `ToolDefinition` for it. *How* that definition reaches the model depends on the calling loom's mode and is specified in **Tool-registration lifetime and visibility** below; the *shape* of the definition is mode-independent and is specified here. The runtime constructs the definition through `defineTool(...)` (exported from `@mariozechner/pi-coding-agent`) rather than a bare object literal, so TypeBox's generic inference on `parameters` is preserved when the resulting value is spread into a `customTools: ToolDefinition[]` array (consumed by the subagent path) or passed to `pi.registerTool` (consumed by the prompt-mode path). All five fields (`name`, `label`, `description`, `parameters`, `execute`) are required by Pi's `ToolDefinition` interface. Field derivations:
    - `name` — the loom's registered tool name from its frontmatter / discovery (per [Directory Convention](./discovery.md)).
    - `label` — the loom file's basename with hyphens preserved and the leading character capitalised: `summarise.loom` → `"Summarise"`, `code-review.loom` → `"Code-Review"`. For the synthesised typed-query one-shot tool described in [Implementation Notes — Runtime](./implementation-notes.md#runtime) the label is the literal string `"Loom typed-query response"`.
    - `description` — the loom's frontmatter `description`.
    - `parameters` — a TypeBox `TSchema`, **not** a raw JSON Schema document. The lowered JSON Schema artefact produced by [Schema Subset — Lowering Algorithm](./schema-subset.md#lowering-algorithm) is wrapped via `Type.Unsafe<unknown>(loweredJsonSchema)` (imported from the `typebox` peer dependency declared in `package.json`; the presence of `Type.Unsafe` as a callable function is the load-bearing host-shape precondition checked at extension-factory entry by **Step 0 (e)** above, so a missing or renamed member surfaces as `loom/load/host-incompatible` with `details.kind = "typebox-shape"` rather than as a runtime-time `TypeError` at this call site) before being handed to `pi.registerTool`. The `Type.Unsafe` wrapper exists solely to satisfy Pi's structural type and to let TypeBox carry the JSON Schema through to the provider lowering layer; `Static<…>` is unused at runtime. Because the wrap is `Type.Unsafe<unknown>`, the `params` argument observed inside `execute` is effectively `unknown`, so the adapter MUST AJV-validate `params` against the original lowered JSON Schema before forwarding — Pi's structural typing offers no guarantees here. The optional `prepareArguments?: (args: unknown) => Static<TParams>` hook on `ToolDefinition` is the documented site for argument shaping in Pi's `ToolDefinition` contract; loom's respond-repair flow (specified in the plan corpus) instead operates at the response-validation boundary (see [Query — Schema-validation respond-repair](./query.md)) rather than at this per-call argument boundary.
    - `execute` — the adapter that spawns the loom as a subagent invocation (equivalent to `invoke<T>(path, ...)`) and returns its result envelope.

    The adapter is **re-entrant**: when the model emits parallel tool calls (Pi's parallel-tool-mode behaviour, see [Tool Calls — Concurrency](./tool-calls.md)) targeting the same `.loom` callable, two concurrent calls into the same `execute` adapter spawn two independent subagent invocations on the event loop. Adapter implementations MUST NOT serialise calls on shared closure state.

**Tool-registration lifetime and visibility.** Pi's tool registry has no `unregisterTool` API — `pi.registerTool` is the only mutation, and entries persist for the process lifetime. The runtime therefore wires loom-synthesised tools (`.loom` callables from another loom's callable set, and the typed-query `__loom_respond_<slug>` one-shot tool described in [Implementation Notes — Runtime](./implementation-notes.md#runtime)) into the model's reach by two distinct mechanisms, one per **calling-loom** mode, designed so that tool *visibility* tracks the loom's invocation rather than the process lifetime, and so the [Parameters and Frontmatter](./frontmatter.md#tools) "ambient tools are not inherited" invariant holds.

*Subagent mode (caller is `mode: subagent`).* The runtime never calls `pi.registerTool` for the synthesised tool. The `ToolDefinition` is passed through `createAgentSession({ customTools, tools, ... })` (see **Conversation drive — subagent mode** below). The tool exists only inside the spawned `AgentSession`, dies with `dispose()`, and is invisible to the user's bare Pi session and to every other extension or concurrent loom.

*Prompt mode (caller is `mode: prompt`).* The runtime maintains an extension-scoped registration cache: `Map<schemaSlug, registeredToolName>`, where the slug is the schema slug of the lowered tool `parameters` (produced by the recipe in [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash)). On first encounter of a unique slug the runtime calls `pi.registerTool` once with a content-addressed name (`__loom_callee_<slug>__<post-rename-name>` for loom callees; `__loom_respond_<slug>` for the typed-query one-shot tool). Subsequent uses of the same lowered shape reuse the existing registration.

On a subsequent cache hit (a new lowered schema hashing to a slug already present in the cache), the runtime MUST verify byte-equality of the cached canonical-form schema bytes against the new entry's canonical-form bytes before reusing the registration. The slug is a 64-bit truncation of SHA-256 (per [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash)), so silent collision is statistically negligible but not impossible; treating slug equality as schema equality without verification would silently alias two distinct tool schemas to one registered name. On byte-mismatch the runtime emits `loom/runtime/registration-cache-collision`, refuses to dedup, and registers the new schema under a disambiguated name with a monotonically increasing per-slug counter (`__loom_callee_<slug>_<n>__<post-rename-name>` for loom callees; `__loom_respond_<slug>_<n>` for typed-query one-shot tools, starting at `n = 2`). The cache stores the canonical-form bytes alongside the registered name so the equality check is a byte comparison, not a re-serialisation. Around each query — both untyped and typed — the runtime gates visibility via the active-set:

1. Snapshot `pi.getActiveTools()` into a local list.
2. Call `pi.setActiveTools([...snapshot, ...loomCallableSetNames, respondToolName?])` — the set the model sees for this turn is exactly the loom's declared callable set, plus the respond tool when the turn is a typed-query response turn.
3. Issue the query.
4. In a `finally` block (so cancellation, panic, and provider exceptions all preserve the invariant), call `pi.setActiveTools(snapshot)` to restore the user session's prior active set.

**Restore-failure protocol.** If the restoring `pi.setActiveTools(snapshot)` call inside the `finally` block of step 4 throws or rejects, the runtime MUST: (a) re-attempt the restore exactly once with the same snapshot (covering transient failures); (b) on a second failure, emit `loom/runtime/active-set-restore-failed` (E, runtime) with `message` carrying the underlying error's message and `hint` listing the snapshot tool names so an operator can manually restore via Pi's `/tools` interface; (c) emit a `display: true` `loom-system-note` whose `content` is the verbatim template `loom: failed to restore tool active-set after /<name>; the user session may have unexpected tools active. Run /reload to reset.` (only `<name>` is substituted — every other character ships verbatim, including the literal `/reload` and the trailing clause); and (d) propagate the original exception (or terminal `Err`) that the `finally` was protecting — restore failure does not mask the inner error, and the cascade-twin dedup rule under **Runtime event channel** below treats the active-set-restore note as independent of any inner-failure note. The retry MUST NOT chain back into `pi.setActiveTools` beyond the single re-attempt above (no exponential backoff, no third try). Subagent-mode invocations are unaffected — the subagent's tools are scoped to the spawned `AgentSession` and released by `dispose()`, so this rule applies only to the prompt-mode and prompt → prompt cross-mode `invoke` snapshot/restore paths in this section.

The snapshot/restore is the only control surface for visibility; the cached `pi.registerTool` entries themselves are intentionally permanent. Consequences:

- `pi.getAllTools()` and `/tools` listings accumulate one entry per distinct lowered schema seen in the extension's lifetime; this is bounded (looms author a small finite number of typed-query and `.loom`-callable shapes) and is documented as cosmetic V1 behaviour.
- The user's bare Pi session — and every other extension — never sees a loom-synthesised tool in its active set, because no code path other than the loom runtime's own swap ever activates them.
- Hot reload (the `_loom-reload` path described in **Extension entry point** step 5) drops the registration cache so a fresh extension instance starts empty; the orphaned registry entries persist in Pi's global registry but are unreachable, matching the V1 acceptance of cosmetic accumulation.
- `pi.setActiveTools` snapshots are name lists, not snapshots of the global registry. The `pi.setActiveTools(names: string[]): void` and `pi.getActiveTools(): string[]` signatures are pinned by `ExtensionAPI` in `dist/core/extensions/types.d.ts` of `@mariozechner/pi-coding-agent ~0.72.1` (the V1 Pi-SDK pin from **Host prerequisites** above) and are covered by the [Step 0 (c)](#entry-capability-probe) factory-probable capability inventory; a Pi minor that renames either member, widens the return type of `pi.setActiveTools` past `void`, or splits the snapshot/restore pair surfaces at the H1 SDK surface-inventory test before the spec contract drifts. The snapshot/restore protocol additionally relies on two Pi *behavioural* preconditions that the type signature does not encode: (a) the loom runtime treats `pi.setActiveTools(string[])` as synchronous-and-atomic on the JS event loop (a single swap or restore call cannot be interleaved with anything else); and (b) the loom runtime assumes Pi serialises slash-command dispatch per session (two loom-invocation slash dispatches against the same session cannot overlap their snapshot/restore windows). Precondition (b) covers only the slash entry point; the symmetric in-process guarantee that nested prompt → prompt `invoke(...)` calls cannot overlap their snapshot/restore windows is owned by the loom runtime itself — the `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns per [Invocation — Cross-mode semantics](./invocation.md), V1 exposes no parallel-`invoke` surface, and prompt-mode `.loom` callees in another loom's `tools:` are rejected at load time per [Parameters and Frontmatter — `tools`](./frontmatter.md#tools), so two prompt-mode snapshot/restore windows against the same user session can only ever be **strictly LIFO-nested**, never concurrently overlapping. If either Pi behavioural precondition above weakens in a future Pi minor, the snapshot/restore protocol above MUST be re-validated as part of [Pi version bump procedure](#pi-version-bump-procedure) below; the H1 SDK surface-inventory test cannot detect either weakening from the type surface alone.
- If another extension calls `pi.setActiveTools` between a loom's snapshot and its `finally` restore, the loom's restore overwrites that change. Extensions wishing to coexist with loom should mutate the active set outside of any in-flight loom invocation, or accept the overwrite. The loom runtime emits no diagnostic for this case. This overwrite rule covers prompt-mode untyped queries, prompt-mode typed-query free and forced phases, and prompt → prompt invocation bodies — same protocol, same rule. It does NOT apply to subagent mode, which never touches `pi.setActiveTools` on the user session.
- The `unknown_tool` cause on `CodeToolError` (see [Tool Calls](./tool-calls.md)) and the captured-reference dispatch model from [Parameters and Frontmatter — Resolution snapshot](./frontmatter.md#resolution-snapshot) are unaffected: code-side `<name>(...)` calls never touch the active-set machinery — they dispatch directly through the loom's load-time callable table.

No Pi-side change is required for V1; if Pi later adds `pi.unregisterTool`, the registration cache becomes the natural site to call it from on extension teardown without altering the per-mode wiring rule above.

<a id="no-extra-mediation"></a>

**No additional access channels.** The runtime interposes no privilege layer between loom code and the Pi extension host. Loom-side filesystem, network, and Pi-API access are bounded only by Pi's extension-host grants and by the per-loom `tools:` allowlist whose enforcement is specified above. The runtime MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy) in any V1.x release; widening or narrowing this rule is a major-version concern. The `tools:` allowlist constrains the *model's* reachable callable set, not the host process: a loom that declares a high-privilege callable (e.g. `bash`) exposes the full underlying capability of that callable to its model. Host-side denials of filesystem, network, or Pi-API access reach loom code through the tool that issued the request: a thrown or `isError: true` return is mapped to `Err(QueryError { kind: "code_tool", cause: "execution", ... })` per [Tool Calls — Failures](./tool-calls.md) and [Tool execution from loom code](#tool-execution-from-loom-code) below; silent success on denial is forbidden.

**Conversation drive — prompt mode.** The loom interpreter drives the user's session by:

- Issuing untyped queries via `pi.sendUserMessage(text)` when the agent is idle, or `pi.sendUserMessage(text, { deliverAs: "steer" })` when the agent is mid-stream. `pi.sendUserMessage` returns `void` — the `ExtensionAPI` surface declares both `sendUserMessage(...)` and `sendMessage(...)` as `void`-returning; the `Promise<void>` shape exists only on `AgentSession.sendUserMessage` (subagent mode, see **Conversation drive — subagent mode** below) and on `ReplacedSessionContext.sendUserMessage` (post-replacement contexts, out of scope here per the *captured `pi` reference* clause below). Only synchronous throws — e.g. argument-validation rejections inside Pi — are observable directly from the call. The runtime MUST NOT `await pi.sendUserMessage(...)` and MUST NOT attach a `.catch` handler — matching the rule pinned for `pi.sendMessage` under **System notes** below. The **primary** prompt-mode transport-failure surface is therefore the post-`waitForIdle()` `AgentSession.errorMessage` probe pinned under *Error detection* below, not the `pi.sendUserMessage` call site itself. As a narrow secondary mapping, a synchronous throw from `pi.sendUserMessage` (the only failure mode the call surface itself can signal) is mapped to `Err(QueryError { kind: "transport", message: <error.message>, http_status: null, provider: <resolved-model-provider>, retryable: false })` — not to `loom/runtime/internal-error`. The `provider` field is populated from the loom's resolved `model:` regardless of which surface (synchronous throw at the call site vs. the `errorMessage` probe) detected the failure, since the `errorMessage` string carries no structured provider field and the **Runtime event channel** always-log emission for `kind: "transport"` requires the field. The same dual-surface mapping applies when `pi.sendUserMessage` is the transport call inside a typed query's free phase or its forced respond turn. The `pi` reference here is the `ExtensionAPI` captured by the extension factory (the `pi` parameter of `default function (pi: ExtensionAPI)` from the **Extension entry point** above), which the runtime holds for the lifetime of each loom invocation; `sendUserMessage` is **not** a method on `ExtensionCommandContext`, so the per-handler `ctx` is consulted only for `ctx.isIdle()` to choose between the two forms (calling the no-options form mid-stream throws — see Pi's `examples/extensions/send-user-message.ts`). V1 prompt-mode looms always use `deliverAs: "steer"` when not idle (the loom is the active driver and wants its message delivered to *this* turn); `deliverAs: "followUp"` is reserved for future considerations and is not emitted by V1. The captured `pi` reference assumes the user session is **not** replaced mid-loom (`ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` invalidate it); V1 looms never trigger replacement (see [Future Considerations — Mid-loom user-session replacement](./future-considerations.md#mid-loom-user-session-replacement) for the post-V1 re-acquisition seam). Completion is awaited by `await ctx.waitForIdle()` on the `ExtensionCommandContext` passed to the slash-command handler. `waitForIdle` is session-scoped — it observes only the user session bound to this command — and is the prompt-mode driver's authoritative completion signal. *Resolution.* `waitForIdle()` returns `Promise<void>` and resolves once Pi emits `agent_end` for the user session and every awaited `agent_end` listener settles; it never rejects, so the runtime MUST NOT wrap the `await` in a `try`/`catch` expecting rejection as a failure signal. *Error detection.* Because `pi.sendUserMessage` returns `void` (no rejection path) and `waitForIdle()` never rejects, every transport- or provider-level failure of the user turn that is not a synchronous throw at the call site lands on the user `AgentSession`'s error state. This post-`waitForIdle()` probe is the **primary** transport-failure surface for prompt mode; the synchronous-throw branch above is the narrow secondary. Immediately after `waitForIdle()` resolves the runtime probes that state (the `errorMessage` field on the `AgentSession`, named per the pinned Pi version from **Host prerequisites** above), and a non-empty value maps to `Err(QueryError { kind: "transport", message: <errorMessage>, http_status: null, provider: <resolved-model-provider>, retryable: false })` per [Query](./query.md). The probe MUST read `errorMessage` synchronously on `waitForIdle()` resolution, before the runtime issues any subsequent `pi.sendUserMessage` on the same context — Pi may clear or overwrite the field on the next turn. The runtime MUST NOT install a global `pi.on("agent_error", …)` listener for this purpose, by the same per-session cross-fire reasoning given for the `agent_end` ban below. *Hang handling.* `waitForIdle()` has no internal deadline; hangs are bounded only by the cancellation path. `loomAbort.abort()` invokes `ctx.abort()` on the captured slash-command handler context (the `ExtensionCommandContext.abort` member already in the forwarded set per **`ExtensionContext` (member surface loom touches)** below), which cancels the active user run and lets `waitForIdle()` resolve; the runtime guards the propagation with a one-shot flag so a re-entrant `loomAbort.abort()` (e.g. from an `agent_end` observer that itself observes the abort) does not double-cancel. The post-`waitForIdle` error-state probe runs unconditionally on resolution, but if `loomAbort.signal.aborted` is true the runtime synthesises `Err(QueryError { kind: "cancelled" })` per [Cancellation](./cancellation.md) instead of reading session error state — even when Pi tore down cleanly and `waitForIdle()` resolved without any error written to the session. The symmetric statement on the abort side lives at [Cancellation — Forwarding into `loomAbort`](./cancellation.md). The runtime MUST NOT subscribe to the global `pi.on("agent_end", …)` event for query completion: that event fires for every `AgentSession` in the process (the user shell session and any concurrent subagent-mode sessions) with no per-session origin marker, so a global handler cross-fires across concurrent looms or sibling subagents and resolves the wait on the wrong turn. The accumulated assistant text from the final turn — read from the command context after `waitForIdle()` resolves — is the `Ok(string)` value. `pi-loom` performs no buffering, suppression, or re-styling of Pi's stream; user-visible behaviour is specified in [Slash-Command Invocation — User-visible streaming](./slash-invocation.md).
- Issuing typed queries that enforce the inferred or ascribed schema against the model's response. A typed query is a **two-phase tool-loop conversation**, exactly as specified in [Query — Typed queries are tool-loop-shaped](./query.md): (1) a *free phase* in which the model may call any frontmatter `tools:` entry the same way it does in an untyped query, with `tool_choice` left unconstrained; followed by (2) one *forced respond turn* once the model emits a plain text turn, in which the runtime issues a follow-up user message inlining the lowered response schema and forces the provider's tool choice to the synthesised one-shot respond tool. The synthesised tool's `execute` AJV-validates its input against the lowered schema and resolves the query's promise with the validated value. Only the forced respond turn uses provider tool-choice forcing; every preceding free-phase turn leaves the model free to think aloud and to use the frontmatter callable set, so the `frontmatter.md` "available to the model during query-time tool loops" guarantee, the `ModelToolError` surface, and the "same callable set serves both code and model" invariant (verified by the tool-call plan leaves in the plan corpus) all stay live for typed queries. The forced respond turn counts against the same `tool_loop.max_rounds` cap as free-phase turns (see [Query — Tool-call loop bound](./query.md)); respond-repair follow-ups (see [Query — Schema-validation respond-repair](./query.md)) restart the whole two-phase loop with a fresh `tool_loop` budget, so the model can re-tool before answering a respond-repair turn. The runtime MUST surface the validated payload exactly once and MUST NOT add additional user-visible turns to the conversation beyond the natural free-phase turns and the single forced respond turn (the forced respond turn is rendered as a tool-call card by Pi's standard transcript renderer, not as a redundant assistant turn). The synthesised one-shot tool follows the same `name` / `label` / `description` / `parameters` / `execute` shape and the same `Type.Unsafe<unknown>(loweredJsonSchema)` `parameters` wrap defined in **Per-loom registration** above, and follows the registration cache + active-set snapshot/restore protocol defined in **Tool-registration lifetime and visibility** above, so the synthesised typed-query tool, the `.loom`-callable Pi tool, and the subagent-mode `customTools` entries all use a single bridge from loom's lowered JSON Schema to Pi's TypeBox-typed `ToolDefinition` and a single visibility-control mechanism. The mechanism used to force tool choice on the respond turn (`options.toolChoice` on the per-provider pi-ai adapter) is described in [Implementation Notes — Runtime](./implementation-notes.md#runtime).

> **V1 seam — mid-loom user-session replacement.** <a id="v1-seam-mid-loom-user-session-replacement"></a> The prompt-mode driver MUST read `pi` from a single factory-captured reference (the `pi: ExtensionAPI` parameter of `default function (pi: ExtensionAPI)` from the **Extension entry point** above) held for the lifetime of each loom invocation; consumers MUST NOT re-acquire `pi` from any other surface (e.g. by reaching into `ExtensionCommandContext` or by re-entering the factory) during a loom invocation. The deferred *Mid-loom user-session replacement* extension in [Future Considerations](./future-considerations.md#mid-loom-user-session-replacement) introduces a `withSession`-fronted re-acquisition of the captured reference before the next `sendUserMessage` (so a loom-driven `ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` call no longer invalidates the captured `pi`) at exactly that single capture site; duplicating the capture across call sites in V1 is breaking, because the deferred re-acquisition hook would then have to be threaded into every duplicate.

**Provider compatibility for typed queries.** The forced respond turn requires the provider to honour named-tool forcing. V1 supports the providers for which pi-ai exposes a named-tool `toolChoice` mapping: `anthropic-messages`, `openai-completions`, `mistral`, and `amazon-bedrock`. At loom-load time, when a loom contains any typed-query expression and its resolved `model:` routes through a provider outside this set, the runtime emits `loom/load/typed-query-unsupported-provider` (warning) naming the model and listing the typed-query expression positions; the loom still loads, and any typed query at runtime returns `Err(QueryError { kind: "transport", message: "<provider> does not support forced tool-use; typed queries unavailable", http_status: null, provider: "<provider>", retryable: false })`. Untyped queries against the same loom continue to work. Future widening (Gemini, OpenAI Responses, weak local models via JSON-mode fallback) is listed in [Future Considerations](./future-considerations.md).

> **V1 seam — typed-query supported provider set.** <a id="v1-seam-typed-query-supported-provider-set"></a> The V1 typed-query-supported provider set (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`) MUST be exposed as a single named runtime constant referenced by every consumer — the load-time `loom/load/typed-query-unsupported-provider` warning emitter, the runtime guard that synthesises the unsupported-provider `TransportError`, and the typed-query-validation and subagent-typed-query plan leaves — so that there is one source of truth to widen. Inlining the four-element provider list at any consumer is a breaking simplification: the deferred *Typed-query support for providers without named-tool forcing* extension in [Future Considerations](./future-considerations.md) widens this constant (and adds a JSON-mode / system-prompt fallback for the new entries) without touching the typed-query behavioural contract above; widening the constant is the only behavioural surface the extension perturbs.

**V1 diagnostic limitation.** When a typed query routes through a provider in the V1 supported set (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`) — most commonly an OpenAI-compatible local backend — but the specific model ignores forced-tool selection (emitting plain text instead of calling `__loom_respond_<slug>` on the forced respond turn), the runtime cannot distinguish this from any other case where the model fails to call the respond tool. The runtime classifies the case as a schema-validation failure (NOT as `tool_loop_exhausted` — the `tool_loop_exhausted` `last_tool_name: null` clause in [Errors and Results — `ToolLoopExhaustedError`](./errors-and-results.md#queryerror-variants) is reserved for the case where `tool_loop.max_rounds` was already saturated by free-phase rounds before the forced respond turn could be issued). For each forced respond turn that produces plain text, the runtime synthesises a single `ValidationIssue` (per [Errors and Results — `ValidationError`](./errors-and-results.md#queryerror-variants)) of the form:

```
{ path: "",
  schema_keyword: "required",
  message: "model returned plain text instead of calling the forced respond tool" }
```

The synthesised issue feeds the same respond-repair pipeline as an AJV failure: it consumes one `respond_repair.attempts` slot, and the follow-up template is the methodology-appropriate one (the `validator_error` template's `<validator-errors>` placeholder is rendered from this synthesised issue exactly as if AJV had produced it). Terminal exhaustion returns:

```
Err(QueryError {
  kind: "validation",
  cause: "schema_validation",
  attempts: respond_repair.attempts,                  // the configured budget
  validation_errors: [<synthesised issue from final attempt>],
  raw_response: <plain-text body of the final forced respond turn>,
  message: "model did not call the forced respond tool"
})
```

With `respond_repair.methodology: none`, the *first* forced respond turn that produces plain text returns the same `Err` shape with `attempts: 0` (no follow-ups issued, consistent with the existing `none` contract). When a stub provider alternates plain-text and malformed tool-call payloads across respond-repair attempts, only the *last attempt's* issues appear in `validation_errors` (consistent with the existing "final malformed assistant text" wording in `errors-and-results.md`); `raw_response` is the plain-text body of the *last* forced respond turn, not a concatenation. The synthesised `ValidationIssue.path` is `""` (the JSON Pointer to the document root), matching AJV's empty-path convention for root-level failures. Separating provider-level from model-level non-compliance is out of scope for V1.

**Conversation drive — subagent mode.** The loom interpreter spawns a fresh in-process `AgentSession` via `createAgentSession`. The `createAgentSession(options: CreateAgentSessionOptions)` function and the `CreateAgentSessionOptions` interface are declared at `dist/core/sdk.d.ts` in `@mariozechner/pi-coding-agent ~0.72.1` (the V1 Pi-SDK pin from **Host prerequisites** above); the inline call shape and the no-`signal`-field claim below are pinned to that file and MUST be re-validated against it on each Pi minor bump per [Pi version bump procedure](#pi-version-bump-procedure) below. The two tool-related fields on `CreateAgentSessionOptions` carry distinct payloads and must both be populated from the same lowered set:

```ts
const customTools: ToolDefinition[] = lowerLoomToolsToPi(loom.tools); // built-in Pi tools resolved by name → their ToolDefinition; loom callees wrapped via defineTool
const loomSystemPrompt: string = renderLoomSystemPrompt(loom.frontmatter.system, params); // resolved-and-interpolated `system:` per Parameters and Frontmatter
const resourceLoader: ResourceLoader = {
  // Loom-owned adapter: getSystemPrompt is the only loom-load-bearing member; the rest return empty/defaults.
  getSystemPrompt: () => loomSystemPrompt,
  getAppendSystemPrompt: () => [],
  getExtensions: () => ({ extensions: [], errors: [] }),
  getSkills: () => ({ skills: [], diagnostics: [] }),
  getPrompts: () => ({ prompts: [], diagnostics: [] }),
  getThemes: () => ({ themes: [], diagnostics: [] }),
  getAgentsFiles: () => ({ agentsFiles: [] }),
  extendResources: () => {},
  reload: async () => {},
};
const { session } = await createAgentSession({
  customTools,
  tools: customTools.map((t) => t.name), // explicit allowlist suppresses Pi's default built-ins
  model,
  sessionManager: SessionManager.inMemory(cwd),
  resourceLoader,
  // ...
});
```

Four rules govern the spawn call:

1. `customTools` carries every `ToolDefinition` the subagent may use — both Pi built-ins (resolved by name from the model registry / extension API to their `ToolDefinition`) and `defineTool`-wrapped `.loom` callables. Each entry uses the same `name` / `label` / `description` / `parameters` / `execute` shape and the same `Type.Unsafe<unknown>(loweredJsonSchema)` `parameters` wrap defined in the **Per-loom registration** section above. `ToolDefinition.label` is required; the lowering step must supply it (basename-derived per the **Per-loom registration** rule).
2. `tools` is **always** passed as an explicit allowlist of those same names, even when the loom's callable set is empty (in which case `tools: []` is passed, matching `tools: []` ≡ absent `tools:` from [Parameters and Frontmatter](./frontmatter.md)). The explicit allowlist is what enforces the "ambient Pi tools NOT inherited" invariant; omitting `tools` would re-enable Pi's default built-in `read` / `bash` / `edit` / `write` tools regardless of what the loom declared.
3. The `tools` allowlist and the `customTools` array are derived from the same lowered set in a single step; the runtime does not let them drift. A `.loom` callable wrapped via `defineTool` has a `name` chosen by the loom runtime (post-`as` rename per the basename rules in [Parameters and Frontmatter](./frontmatter.md)); that same name must appear in the `tools` allowlist.
4. The `resourceLoader` passed to `createAgentSession` is a loom-constructed `ResourceLoader` adapter whose `getSystemPrompt()` returns the resolved-and-interpolated frontmatter `system:` string verbatim and whose `getAppendSystemPrompt()` returns `[]` (the loom's `system:` is the *complete* system prompt, not an append on top of a host base). The remaining `ResourceLoader` members return empty values / defaults (`getExtensions`, `getSkills`, `getPrompts`, `getThemes`, `getAgentsFiles`) or are no-ops (`extendResources`, `reload`); the loom MUST NOT pass the parent session's `DefaultResourceLoader` through unchanged. `CreateAgentSessionOptions` exposes no `systemPrompt` / `system` field in the V1 Pi SDK pin, so the loader's `getSystemPrompt()` is the only available delivery channel for the loom's `system:` value into the spawned `AgentSession.systemPrompt`. The adapter MAY be expressed as the object literal above or as a `class implements ResourceLoader` declaration; both are conformant. `DefaultResourceLoader`'s `systemPromptOverride: (base) => string | undefined` constructor option is an alternative channel but is **not** the recommended construction: it still runs all of `DefaultResourceLoader`'s discovery side-effects against the parent `cwd` to compute `base`, only to ignore the result, so the custom-adapter form is preferred.

Immediately after the awaited `createAgentSession(...)` resolves, the runtime registers a one-shot listener on `loomAbort.signal` that calls `AgentSession.abort()` on the returned handle. The listener is the sole cancellation-forwarding mechanism in subagent mode — `CreateAgentSessionOptions` (declared at `dist/core/sdk.d.ts` in the V1 Pi-SDK pin per the introductory paragraph above) has no `signal` field, so passing one would be silently ignored; the spawn call MUST NOT include a `signal` field. If `loomAbort.signal.aborted` is already true at the moment `createAgentSession(...)` resolves, the runtime invokes `session.abort()` synchronously *before* the listener is registered and *before* issuing the first user turn, so the spawn-then-immediate-cancel path remains correct without depending on microtask ordering. The listener is detached in the same per-invocation `finally` that calls `AgentSession.dispose()` (see **Subagent session lifecycle** below). The spawned session inherits the loom's `system:` prompt (from frontmatter, with `${param}` interpolation resolved at conversation-creation time), the lowered `customTools` set described above, and the loom's `model`. Queries against the spawned session use `session.sendUserMessage(text)` on the returned `AgentSession` and await completion by `session.subscribe(event => { if (event.type === "agent_end") … })`, which is scoped to that `AgentSession`'s own `ExtensionRunner` and cannot fire for any other session. The runtime MUST use the session-local `subscribe` API rather than the global `pi.on("agent_end", …)` event for the same per-session-scoping reason given for prompt mode. The runtime unsubscribes (via the function returned by `subscribe`) before resolving each query's promise so that the next query on the same long-lived subagent session does not observe the previous query's listener; a fresh subscription is attached per query. Typed queries against the spawned session honour the same behavioural contract as prompt-mode typed queries above and use the same mechanism described in [Implementation Notes — Runtime](./implementation-notes.md#runtime). The session is in-memory only — the spec mandates the transcript stays private to the loom and is discarded when the loom returns. The in-memory `SessionManager` is the mechanism that prevents subagent output from reaching Pi's user-facing UI; user-visible behaviour is specified in [Slash-Command Invocation — User-visible streaming](./slash-invocation.md). Each subagent-mode invocation gets its own `AgentSession`, its own captured `tools:` table (the load-time resolution snapshot from [Parameters and Frontmatter](./frontmatter.md)), and its own derived `AbortSignal`; sibling invocations triggered by parallel tool calls do not share mutable session state.

<a id="subagent-state-isolation-matrix"></a>

**Subagent state-isolation matrix.** A subagent-mode invocation runs in an isolated session whose state composition is fixed by the table below. The matrix is the canonical enumeration of what the spawned session inherits from the loom's frontmatter, what is forwarded from the caller's `ExtensionCommandContext`, and what is *not* inherited from the caller. The three columns are populated by distinct mechanisms (frontmatter resolution, the override-table forwarding rule under **`ExtensionContext` (member surface loom touches)** below, and the explicit suppression rules in **Conversation drive — subagent mode** above) and a future widening of any cell is a spec-versioned change.

| Inherited from the loom's frontmatter | Forwarded from the caller's `ExtensionCommandContext` | Not inherited from the caller |
|---|---|---|
| `system:` prompt (with `${param}` interpolation resolved at conversation-creation time) | `ctx.cwd` | caller's transcript (the spawned session uses `SessionManager.inMemory(cwd)`; transcript is private and discarded when the loom returns) |
| `model` (resolved from frontmatter) | `ctx.modelRegistry` | caller's system prompt (suppressed by the loom-constructed `ResourceLoader` adapter passed to `createAgentSession` whose `getSystemPrompt()` returns the resolved-and-interpolated loom frontmatter `system:` and whose `getAppendSystemPrompt()` returns `[]`; see rule 4 under **Conversation drive — subagent mode** above. `CreateAgentSessionOptions` exposes no `systemPrompt` field — delivery via the loader is the only available channel, and passing the parent session's `DefaultResourceLoader` unchanged would silently inherit the host project's `AGENTS.md`-derived prompt) |
| lowered `customTools` set (the loom's *callable set*; see [Glossary — `callable set`](./glossary.md)) | `ctx.getContextUsage` | ambient Pi tool set (suppressed by the explicit `tools` allowlist passed to `createAgentSession`; default rule when `tools:` is absent or empty is owned by [Parameters and Frontmatter — `tools`](./frontmatter.md#tools)) |
| captured `params` (load-time per-invocation values) | other non-overridden `ExtensionContext` members per **`ExtensionContext` (member surface loom touches)** below | caller's `loomAbort` controller (each invocation gets a fresh `AbortController`; lineage and forwarding are owned by **Cancellation source** below and [Cancellation](./cancellation.md)) |
| | | caller's `params` and variable bindings (the subagent runs a different loom file with its own `params`) |
| | | caller's `withActiveTools` snapshot scope (subagent-mode tool registration uses `customTools`, not `pi.setActiveTools`; see **Tool-registration lifetime and visibility** above) |

The `loomAbort` row is the structural reason a subagent-mode child cannot observe its parent's cancellation directly: cancellation forwards via the parent's `loomAbort.abort()` call propagating through a one-shot `loomAbort.signal` listener that calls `AgentSession.abort()` on the spawned `AgentSession`, not via shared controller identity and not via any `createAgentSession` option (no `signal` field exists on `CreateAgentSessionOptions` in the V1 Pi SDK pin). Nested-subagent cancellation behaviour is owned by [Cancellation — Signal source](./cancellation.md).

The matrix is consumed by `spec.md`'s preamble forward-link, by [Overview — Scope of a Loom File](./overview.md#scope-of-a-loom-file)'s isolation prose, and by the subagent-mode plan leaves (specified in the plan corpus) — each row is mechanically asserted by one or more of those leaves.

**Subagent session lifecycle.** The runtime owns the spawned `AgentSession` for exactly the duration of one subagent-mode loom invocation. Disposal is mandatory and runs in a `finally` block that wraps the entire interpreter execution against that session, so it fires on:

- normal return (the loom's tail expression resolves);
- any `Err` returned to the parent (query failure, tool failure, child invoke failure, validation failure, cancellation);
- any panic raised inside the subagent (`MatchError`, index out of bounds, null access — see [Errors and Results](./errors-and-results.md));
- any unexpected exception thrown by the interpreter or the Pi SDK.

Disposal calls `AgentSession.dispose()`, which removes all event listeners (including any in-flight `agent_end` subscription the runtime attached via `session.subscribe(...)` for the current query) and disconnects from the underlying agent. The same `finally` also detaches the one-shot `loomAbort.signal` listener registered at spawn (per **Conversation drive — subagent mode** above) — this is the listener-cleanup obligation [Cancellation — Forwarding into `loomAbort`](./cancellation.md) cites for the subagent-mode entry, and it runs alongside the existing detach of every other forwarding listener the runtime attached during the invocation. `dispose()` is invoked at most once per session and treated as idempotent at the call site: a second call (e.g. defensive cleanup in an outer scope) is a no-op. If `dispose()` itself throws, the runtime logs the disposal error via the `loom/runtime/subagent-dispose-failure` diagnostic (see [Diagnostics](./diagnostics.md)) but does not mask the original error that triggered teardown — the parent still observes the original `Err` or panic.

`AgentSession.abort()` (invoked by the cancellation listener above) returns `Promise<void>` and resolves once the agent has become idle. The runtime does **not** `await` the returned promise from inside the listener — the listener fires synchronously, calls `session.abort()`, and discards the returned promise; teardown progress is observed instead through the `disposeBarrier` and the subsequent `dispose()` call in the per-invocation `finally`. If the synchronous `session.abort()` call throws, or the discarded promise rejects, the runtime traps the error and routes it through `loom/runtime/internal-error` (see [Diagnostics — `loom/runtime/*`](./diagnostics.md)) rather than masking any original error the surrounding `finally` was protecting; this mirrors the `dispose()`-throws rule above. The `session_shutdown` handler's `SHUTDOWN_AWAIT_CAP_MS = 2000` budget covers the entire disposal phase, including any time `AgentSession.abort()`'s underlying idle-wait contributes via the subsequent `dispose()` call — there is no separate budget for `abort()`. The runtime is re-entrancy-safe by construction: the one-shot `loomAbort.signal` listener and the existing one-shot guard on `loomAbort.abort()` together make a double-abort (e.g. operator presses Esc twice) a no-op.

A thrown or rejected `createAgentSession`, or a returned handle whose `dispose` member is not a function, is treated as an unanticipated SDK reject and routed through `loom/runtime/internal-error` (see [Diagnostics — `loom/runtime/*`](./diagnostics.md), whose trigger description explicitly enumerates `createAgentSession` among the unanticipated SDK rejects). The parent invocation observes the routing already specified for `internal-error`: a slash-command surface emits `"loom /<name> aborted with internal error: <message>"` as a system note; an `invoke` parent observes `Err(InvokeInfraError { kind: "invoke_failure", cause: "internal_error", ... })`. No `AgentSession` was constructed, so the **Subagent session lifecycle** `finally` below has nothing to dispose; the registry entry from **`ActiveInvocationRegistry`** is removed by the same `finally` in the normal way.

Subagent-mode tool registration has no separate teardown step: the `.loom` callables and the typed-query one-shot tool reach the model via `customTools` on `createAgentSession`, so they exist only for the lifetime of the spawned `AgentSession` and are released by `dispose()` itself. No `pi.unregisterTool` call is required (or possible — Pi exposes none); see **Tool-registration lifetime and visibility** above for the full rule and the prompt-mode counterpart.

The per-invocation `finally` also settles the `disposeBarrier` promise the runtime registered with the **`ActiveInvocationRegistry`** (see **Extension entry point** step 4 above) and removes the registry entry. This is what lets the `session_shutdown` handler's `Promise.allSettled(...)` await observe completion of an in-flight subagent invocation that was cancelled by step 4's `loomAbort.abort()` call. If Pi tears the runtime down before this `finally` runs (a `/reload` issued mid-turn while the subagent is still talking to its provider), the teardown handler's bounded await is what guarantees `dispose()` is called and provider connections are released; the per-invocation `finally` and the teardown handler are two paths to the same disposal, and `dispose()`'s call-site idempotence (above) is what makes the overlap safe.

Edge cases:

- The `finally` block must run even when the parent's `AbortSignal` (see [Cancellation](./cancellation.md)) fires before the subagent's first turn — i.e. spawn-then-immediate-cancel still requires disposal.
- For nested subagent invocations (a subagent that itself invokes a subagent-mode child), each level owns its own session and runs its own `finally`. Disposal order is deepest-first, naturally produced by the call stack's `finally` unwinding.
- Disposal must precede the parent observing the result — the parent's `match`/`?` arm runs after the child's session is gone, so the child's listeners cannot leak into work the parent does next.

**System notes.** Echoes (binder result), `Err`-in-prompt-mode notes, binder failure messages, watcher structural-change notes, and operator-facing runtime events all emit through a single call shape:

```ts
pi.sendMessage(
  { customType: "loom-system-note", content, display, details },
  { triggerTurn: false },
);
```

The `display` and `content` arguments vary per the `details` variant documented below — they are not fixed to `display: true` or to a non-empty string. A `pi.registerMessageRenderer("loom-system-note", renderer)` registered by the loom extension factory formats these as one-line, dim-styled notes in the transcript. The custom-type approach (rather than `ctx.ui.notify(...)`) is chosen because notes need to persist in the session transcript for replay and `/tree` navigation, not just appear transiently.

**Renderer registration (`pi.registerMessageRenderer`).** The pinned signature, lifecycle, and ownership rules below are normative for the loom extension. The canonical Pi-side declarations live at `dist/core/extensions/types.d.ts` in `@mariozechner/pi-coding-agent ~0.72.1` (the V1 Pi-SDK pin from **Host prerequisites** above) and the `Component` interface lives at `@mariozechner/pi-tui`'s `dist/tui.d.ts`; the inline shapes below are the loom-load-bearing subset and MUST be re-validated against those files on each Pi minor bump (per the re-validating obligation in **Host prerequisites** above).

*Signature.* The Pi-side surface, reproduced verbatim from `dist/core/extensions/types.d.ts`:

```ts
export interface MessageRenderOptions {
  expanded: boolean;
}

export type MessageRenderer<T = unknown> = (
  message: CustomMessage<T>,
  options: MessageRenderOptions,
  theme: Theme,
) => Component | undefined;

// On `pi: ExtensionAPI`:
registerMessageRenderer<T = unknown>(
  customType: string,
  renderer: MessageRenderer<T>,
): void;
```

The call returns `void` (synchronous); the loom extension factory MUST NOT `await` it and MUST NOT attach a `.then`/`.catch`. Failures propagate as synchronous throws and fault the extension factory body — there is no separate Promise rejection path.

The `Component` return type is the `pi-tui` widget interface (`{ render(width: number): string[]; handleInput?(...): void; invalidate(): void; ... }`), **not** a string and **not** a React node. A renderer body of the form `(message) => message.content` is therefore a type error that silently leaves the channel unrendered — the loom renderer MUST construct a `pi-tui` `Component` instance (e.g. a text component composed from the `theme` parameter) and return it. Returning `undefined` is legal and tells Pi to skip rendering this message; the V1 loom renderer uses `undefined` only when `display === false` (see **Empty `content` is legal** below).

The `theme: Theme` parameter is Pi's active TUI theme; the loom renderer MUST source any styling (dim text colour, etc.) from `theme` rather than hard-coding ANSI escapes, so re-themes propagate without renderer changes.

The `options: MessageRenderOptions` carries `{ expanded: boolean }` from the Pi host; `expanded` is the Pi-host hint for whether the user has requested an expanded view of this message in the transcript. The V1 loom renderer MUST be defined for both `expanded === false` and `expanded === true`; V1 MAY render identically in both modes. Widening the renderer to use `expanded` to disclose the structured `details` payload (e.g. expanding the diagnostic batch or runtime-event fields) is reserved for a future revision and is not part of V1 conformance.

*Lifecycle.*

- **Registration timing.** The loom extension factory MUST call `pi.registerMessageRenderer("loom-system-note", renderer)` synchronously inside the factory body, before the factory returns. Registration after the factory returns (e.g. inside an event subscriber that fires asynchronously) is undefined behaviour at the Pi-SDK level and the loom runtime MUST NOT rely on it.
- **Re-registration within an extension.** Pi's loader stores renderers in a `Map<string, MessageRenderer>` keyed by `customType` (`@mariozechner/pi-coding-agent`'s `dist/core/extensions/loader.js`), so a second call from the same factory with the same `customType` silently overwrites the first; Pi emits no warning. The loom factory MUST register exactly once per factory invocation.
- **Teardown.** Pi exposes no `unregisterMessageRenderer` API. The renderer entry lives for the lifetime of the extension instance; on `ctx.reload()` or any other path that replaces the extension instance, Pi discards the prior `Map` along with the old extension, and the freshly loaded factory registers afresh against a Pi state that has no prior `loom-system-note` registration. The loom runtime MUST NOT attempt to clean up the registration in its `session_shutdown` handler.

*`customType` ownership and collision rule.*

- The literal `"loom-system-note"` is owned by the pi-loom extension. No other extension SHOULD register a renderer for this `customType`.
- Pi does not enforce ownership: collision is a coordination failure between extensions, not a Pi-level error. Pi's runner resolves a `customType` by iterating `this.extensions` in load order and returning the first hit (`@mariozechner/pi-coding-agent`'s `dist/core/extensions/runner.js` `getMessageRenderer`), so when two installed extensions both register `"loom-system-note"` the **first-loaded** extension's renderer wins and the later registration is unreachable. Whether pi-loom or the other extension wins is non-deterministic from loom's point of view (Pi controls extension load order).
- Loom emits no diagnostic for this case in V1 — ownership is by convention.
- The `customType` naming convention for loom-internal channels is `loom-<purpose>` (kebab-case, `loom-` prefix). Future loom channels MUST follow this convention; other extensions SHOULD NOT use the `loom-` prefix. V1 ships exactly one channel under this prefix (`loom-system-note`).

**Delivery surface.** All three `details` payload variants below emit through the single `pi.sendMessage` call above. The runtime has no second channel for `display: false` notes; they land in the same session transcript as `display: true` notes and are filtered out of visible rendering by the renderer (or by Pi's own `display` handling), but remain available to transcript-replay and `/tree` consumers. *Subagent-mode `display: false` cascades* (the subagent top-level `Err` case noted under **Runtime event channel** below) use the same `pi.sendMessage` call against the spawned `AgentSession` — i.e., they reach `pi.sendMessage` after the per-loom `sessionManager` swap from **Tool execution from loom code** above has routed messaging at the spawned session — so the note lands in the subagent's private transcript, not the parent user session's.

*Custom-message channel persistence and LLM-context entry.* Per `@mariozechner/pi-coding-agent`'s `dist/core/messages.js` `convertToLlm` transformer, every `CustomMessage` (including `loom-system-note`) is converted to `{ role: "user", content }` on every subsequent provider call. `triggerTurn: false` suppresses the immediate turn fire only; it does not exclude the message from the LLM context window. The `display` flag controls renderer behaviour, not serialization. Loom diagnostics therefore enter the user-session model context durably and contribute to `ctx.getContextUsage()` and compaction decisions. Operators authoring looms should expect parse errors, binder failures, panic notes, and always-log runtime events emitted in a session to be visible to subsequent model turns. The V1 Pi-SDK pin (`@mariozechner/pi-coding-agent ~0.72.1`) exposes no `CustomMessage.excludeFromContext` opt-out analogous to `BashExecutionMessage.excludeFromContext`; a future Pi minor that adds such a field would be picked up by the `pi-version-bump-procedure` re-validation gate, at which point the runtime SHOULD set the opt-out on the canonical `sendSystemNote` call.

**Empty `content` is legal.** When the variant prescribes `content: ""` (the `display: false` runtime-event case below), the runtime passes the empty string verbatim. `pi.sendMessage`'s `content` parameter accepts any string per Pi's `CustomMessage<T>` interface; no placeholder substitution is required or permitted. The renderer registered for `loom-system-note` MUST tolerate `content === ""` — in V1 this only ever co-occurs with `display: false` and `details: { event: RuntimeEvent }`, so the renderer MAY short-circuit on `display === false` without inspecting `content`.

The `details` field carries one of three normative payload shapes, distinguished by which key is present:

- `details: { diagnostics: Diagnostic[] }` — a parse / load / type / runtime-panic diagnostic batch, exactly the shape defined in [Diagnostics](./diagnostics.md). The companion `content` is the serialised `<file>:<line>:<col>: <code>: <message>` line(s) for parse / load / type batches; for the runtime-panic case (a single-element batch carrying a `loom/runtime/*` diagnostic) the companion `content` is the user-facing framing `"loom /<name> aborted: <message>"` (or `"loom /<name> aborted with internal error: <message>"` for `loom/runtime/internal-error`) per [Errors and Results — Runtime panics](./errors-and-results.md). See **Runtime event channel** below for the partition rule that routes panics through this shape rather than `details: { event }`.
- `details: { event: RuntimeEvent }` — a runtime failure event from one of the always-log kinds (see **Runtime event channel** below). The companion `content` is the normative user-facing template from [Slash-Command Invocation — Top-level `Err` in prompt mode](./slash-invocation.md) when `display: true`, or omitted (empty string) when `display: false`.
- `details: { structural: { added: string[]; removed: string[] } }` — informational note for watcher-observed structural changes (added or removed `.loom` files, settings-array changes that add or remove sources); `added` and `removed` carry absolute file paths from the debounce-window batch. Settings-array edits surface as the resolved `.loom` file paths the source change brought in or removed, not the settings-file path itself, so a single settings edit that adds N sources contributes N entries to `added`. The shape is disjoint from the `diagnostics` and `event` shapes by key, per the additive-only convention below. The companion `content` is the verbatim template defined under **Structural changes** above; `display: true` always.

Per-variant `display` / `content` pairings (normative):

| Variant | `display` | `content` |
|---|---|---|
| `details: { diagnostics: Diagnostic[] }`, parse / load / type batch | `true` | serialised `<file>:<line>:<col>: <code>: <message>` line(s) (multi-line for batches) — non-empty |
| `details: { diagnostics: [Diagnostic] }`, runtime panic (single-element batch, `loom/runtime/*` code) | `true` | `"loom /<name> aborted: <message>"` (or `"loom /<name> aborted with internal error: <message>"` for `loom/runtime/internal-error`) per [Errors and Results — Runtime panics](./errors-and-results.md) — non-empty |
| `details: { event: RuntimeEvent }`, top-level cascade in prompt mode | `true` | normative user-facing template per [Slash-Command Invocation — Top-level `Err` in prompt mode](./slash-invocation.md) — non-empty |
| `details: { event: RuntimeEvent }`, author-handled or subagent-mode cascade | `false` | `""` (empty string, verbatim) |
| `details: { structural: { added; removed } }` | `true` | verbatim template from **Structural changes** above — non-empty |

The three `details` shapes are disjoint by key; renderers MUST NOT assume more than one is present. Future payload shapes added to this channel are additive and disjoint by the same convention. Future variants MAY widen the `display: false` + `content: ""` combination to other `details` shapes additively; the renderer's `display === false` short-circuit above remains safe under that widening because it does not inspect `content`.

**Runtime event channel.** A subset of `QueryError` failures — the **always-log set** — emit a structured note through the `loom-system-note` channel exactly once per occurrence, regardless of whether the author matched the `Err`, propagated it via `?`, or discarded it via `let _ =`. The always-log set partitions by routing channel: members in **group A** emit `details: { event: RuntimeEvent }`; members in **group B** emit `details: { diagnostics: Diagnostic[] }`. A given failure emits through exactly one shape — there is no fan-out across the two `details` variants for a single failure.

`display: false` gates only renderer visibility; the underlying `CustomMessage` still enters subsequent provider calls per **Delivery surface** above (Pi's `convertToLlm` transform converts every `CustomMessage` to a `user`-role transcript entry on every subsequent provider call, with no opt-out under the V1 SDK pin). Operators MUST treat all `loom-system-note` content (regardless of `display`) as durable session-context input.

*Engine-assumption carve-out.* The exactly-once emission guarantee assumes the [Runtime Value Model — JavaScript engine assumptions](./runtime-value-model.md#javascript-engine-assumptions) hold (IEEE-754 numbers, native `Map`/`Set`, native `JSON.stringify`, `Object.is` equality — a non-checked invariant). Under a violation of those assumptions, emission may be silently dropped (e.g. a corrupted `Map` loses the dedup key), duplicated, or skipped entirely (e.g. a `pi.sendMessage` host throw bypasses the helper); some manifestations land on `loom/runtime/internal-error` via the runtime-defect surface in [Errors and Results — Runtime panics](./errors-and-results.md), but silent value corruption has no observable signal. Operators MUST treat a missing terminal event as one of: (a) the loom did not fail; (b) the loom failed with a kind not in the always-log set; (c) an engine-assumption violation. There is no in-band signal that distinguishes (a) from (c).

Group A — `details: { event: RuntimeEvent }`:

- `transport`
- `model_tool`
- `code_tool`
- `tool_loop_exhausted`
- `invoke_failure`
- Binder failures (every row of [Binder — Failure modes](./binder.md), including `needs_info`, `ambiguous`, malformed envelope, AJV validation, transport, cancellation)

Group B — `details: { diagnostics: Diagnostic[] }`:

- Runtime panics — every row of [Diagnostics — `loom/runtime/*`](./diagnostics.md), routed through the `details: { diagnostics: [...] }` shape per the panic-emission rule under **Deduplication and lifetime rules** below. The `RuntimeEvent` payload shape and the `(kind, query_site, message, occurred_at)` dedup key defined below apply to group A only and have no analogue for group B.

`validation`, `context_overflow`, `cancelled`, and `invoke_callee_error` are deliberately **not** in the always-log set: `validation` is a query-internal repair signal whose final outcome is what matters (the `attempts` field on the eventual terminal event captures the count); `context_overflow` and `cancelled` are user/operator-initiated and self-explanatory in context; `invoke_callee_error` wraps an inner `Err` whose origin already emitted its own runtime event at the originating site.

*Discard-site disposition.* The four excluded kinds emit nothing on this channel even at a discard site (`let _ = @"…"`, `void`-tail-expression discard, or any other disposition that drops the `Err`). The exclusion rationale survives the discard case for each kind: for `validation`, a discarded validation failure is by construction a programming error — the author wrote `let _ = @<Schema>\`…\`` and ignored the schema mismatch; the diagnostic at the originating site (when one fires — depth-5 violations and respond-repair failures both have `loom/runtime/*` codes available) plus the absence of any user-visible outcome is the intended signal. For `cancelled`, the author's choice to discard a user/operator-initiated outcome is itself the disposition. For `context_overflow`, the author's choice to discard a model-state outcome is itself the disposition. For `invoke_callee_error`, the inner `Err`'s origin-site emission is unchanged by the wrapper's disposition. The symmetric statement on the consumer side is owned by [Query — Observability of discarded results](./query.md).

The `RuntimeEvent` payload shape is normative and additive-only:

```ts
type RuntimeEvent = {
  kind: string;                       // QueryError.kind, the binder failure cause, or the loom/runtime/* code for panics
  code?: string;                      // diagnostic code when one applies (loom/runtime/*, loom/load/*, etc.)
  loom: string;                       // slash name of the loom that owned the failure ("/code-review")
  query_site?: { file: string; line: number; column: number }; // source location of the @-template, tool call, invoke, or discarding `let _ =`; absent for binder failures (the binder runs before any loom code)
  message: string;                    // the same message string surfaced through the user-facing template (or the panic message template, for panics that take the diagnostic shape)
  attempts?: number;                  // populated for `validation` events on respond-repair exhaustion; absent otherwise
  tokens_used?: number;               // populated for `context_overflow` events when the provider supplies the count; absent otherwise
  masked?: string[];                  // hard-ceiling co-fire enumeration; see "Hard-ceiling co-fire" below
  occurred_at: number;                // Unix epoch ms, stamped at the originating emission site via Clock.now()
};
```

The channel is operator-facing: events emit with `display: false` when the author handled the `Err` (matched, discarded with `let _ =`, or propagated via `?` to a non-top-level frame that itself handled it), and with `display: true` when the failure cascades out of the slash-command boundary as a top-level `Err` in prompt mode (in which case the `content` carries the normative user-facing template per [Slash-Command Invocation](./slash-invocation.md)). Subagent-mode top-level `Err` cascades likewise emit with `display: false` (the subagent transcript is private; see [Slash-Command Invocation — User-visible streaming](./slash-invocation.md)).

Deduplication and lifetime rules:

- A `?`-propagation chain emits exactly once at the originating site, not at each rethrow. A failure that is created at site A, propagated through frames B and C via `?`, and finally cascades out as a top-level `Err` at frame D produces one runtime event (origin: A) plus the user-facing top-level note at D when `display: true` applies; both share the same `RuntimeEvent` payload. The runtime emits the same `RuntimeEvent` payload at most twice per origin: once at the originating site (always), and once again at the boundary as the `details: { event }` payload of the user-facing top-level note when a cascade applies. Both emissions are byte-identical including `occurred_at`. Consumers that aggregate the event stream may therefore deduplicate on `(kind, query_site, message, occurred_at)` to collapse the cascade twin; this is a non-normative consumer concern and not part of V1 conformance. Re-emissions for symmetry MUST copy the originating `RuntimeEvent` instance verbatim — including `occurred_at` — rather than re-stamping. Two emissions from the same `query_site` with the same `kind` and `message` but distinct `occurred_at` values represent two distinct occurrences.
- Respond-repair attempts on a typed query (see [Query — Schema-validation respond-repair](./query.md)) do not emit per-failed-attempt events. Only the terminal outcome of the query is observable as a runtime event — either a `validation` event with `attempts` set to the exhausted count (when respond-repair runs out), or a non-validation event from the always-log set (when a respond-repair follow-up itself fails for a transport / tool / cancellation / overflow reason and propagates per [Query — Non-validation failures during a respond-repair follow-up](./query.md)).
- A panic emits exactly one `loom-system-note` per top-level panic with `details: { diagnostics: [Diagnostic] }` carrying a single `loom/runtime/*` diagnostic (message per the registered template in [Diagnostics](./diagnostics.md)). The companion `content` field on that same note carries the user-facing framing `"loom /<name> aborted: <message>"` (or `"loom /<name> aborted with internal error: <message>"` for `loom/runtime/internal-error`) per [Errors and Results — Runtime panics](./errors-and-results.md), where `<message>` is the registered diagnostic message template. The diagnostic-shape `details` and the `aborted:` framing in `content` are the **same** emission, not two notes; there is no separate `details: { event }` emission for panics, and the cascade-twin dedup key above does not apply to group B. The fallback chain to `ctx.ui.notify` and `loom/runtime/system-note-delivery-failed` (defined below) still applies if `pi.sendMessage` throws on the panic note; the fallback's diagnostic emission is independent of the panic note and not subject to any dedup key.
- Tool-host emission ownership: when a Pi tool that the loom invoked (either code-side via `<name>(args)` or model-side inside a query's tool-call loop) emits its own diagnostics, those are independent of the loom-runtime event. The loom runtime owns `code_tool` and `model_tool` events; the tool host owns whatever it emits internally. Renderers MUST NOT cross-correlate the two.

Aggregation, latency histograms, per-loom token reports, retention policies, and a consumer-facing read API for the event stream are deferred to [Future Considerations](./future-considerations.md). V1 ships the always-log set and the payload shape only; the channel is intentionally write-only at the spec level, with downstream consumers free to read from Pi's session transcript via existing surfaces.

<a id="pic-1"></a>

**PIC-1. Hard-ceiling co-fire (`masked`).** The optional `masked` field on `RuntimeEvent` (and on the `Diagnostic` shape per [Diagnostics — `masked` field](./diagnostics.md#masked-field-hard-ceiling-co-fire)) carries the hard-ceiling co-fire enumeration. This rule is the normative owner of the per-site reachable mask domain, the V1 reachable predicate, the pure-read MUST, the verbatim-copy obligation, and the dedup-key non-inclusion rule; the four-ceiling list, the ordering rules CIO-1…CIO-6, and the wire-location split are owned by [Hard Runtime Ceilings — `masked` field](./hard-ceilings.md#masked-field) and [Hard Runtime Ceilings — Interaction between ceilings](./hard-ceilings.md#ceiling-interaction-order).

*(a) Closed identifier set.* Each `masked` entry is one of the four string identifiers `"ceiling#1"`, `"ceiling#2"`, `"ceiling#3"`, `"ceiling#4"`, naming a sibling hard ceiling whose precondition was *also* satisfied at the same check site at the moment the surfaced ceiling fired.

*(b) Canonical-absence rule.* The field is **omitted** when no co-fire occurred; implementations MUST NOT emit `masked: []`. A consumer reading `RuntimeEvent` (or a `Diagnostic`) and finding the property absent MUST treat it as identical to absence — there is no tri-state.

*(c) Co-fire-eligible surfaces and per-site reachable mask domain.* `masked` is populated only on runtime events whose `kind` is one of the runtime-class hard-ceiling surfaces — in V1 this is (i) the `validation` event when its `cause` is `"schema_validation"` (ceiling #4 surface at the typed-query response / model-driven `tool_use` args / `params` merge / `invoke<T>` return boundary, per CIO-3 in [Hard Runtime Ceilings](./hard-ceilings.md#ceiling-interaction-order)), (ii) the `code_tool` event when its underlying `CodeToolError.cause` is `"validation"` and the `validation_errors[].schema_keyword` is `"maxDepth"` (ceiling #4 surface at the code-driven `<name>(args)` args boundary per CIO-3 — the fifth row of ceiling #4's per-boundary table), and (iii) the `tool_loop_exhausted` event (ceiling #2 surface at the tool-call-round boundary). The diagnostic-shape co-fire-eligible surface is the `loom/runtime/invoke-depth-exceeded` panic per [Diagnostics](./diagnostics.md). The per-site reachable mask domain in V1 is:

| Surfacing site | Surfaced ceiling | Reachable mask domain |
|---|---|---|
| Typed-query response / model-driven `tool_use` args / `params` merge / `invoke<T>` return AJV boundary (CIO-3) | ceiling #4 | `{ceiling#2}` (only on the typed-query response branch and only when the *V1 reachable predicate* below holds; empty on the other three sub-rows of this AJV boundary because no `tool_loop` slot has been incremented yet for the surfacing event's frame) |
| Code-driven `<name>(args)` AJV boundary (CIO-3, fifth row of ceiling #4's per-boundary table) | ceiling #4 | empty (the call site is not inside a tool-call-round counter context, so ceiling #2's slot accounting is not reachable; ceilings #1 and #3 are unreachable per CIO-1 / CIO-2 / CIO-5) |
| `invoke` entry (CIO-2) | ceiling #1 | empty (the callee body has not yet run, so neither #4's depth-walk nor #2's tool-call round slot accounting has been reached for this frame) |
| Tool-call-round boundary (CIO-4) | ceiling #2 | empty (ceiling #4's depth-walk runs at every AJV boundary earlier in the round per CIO-3, so any depth-6 payload would have already surfaced #4 and short-circuited the round before #2's slot accounting) |
| Slash-load `params` AJV (ceiling #3, load-time) | ceiling #3 | empty (ceiling #3's load-time surfaces always omit `masked` by construction. The slash-load `params` cross-ceiling sub-case — where ceiling #4's depth-walk *did* fire at the [post-default-merge AJV validation](./binder.md#post-default-merge-ajv-validation) hook before being routed to ceiling #3's AJV-on-`args` arm per CIO-1 and the per-boundary table at [Hard Runtime Ceilings — Per-boundary destination/surface table](./hard-ceilings.md#ceiling-4-table) — still omits `masked`: the originating ceiling is recoverable from the rendered note's `<ajv-summary>` placeholder (which names `maxDepth` and the canonical depth-violation message) per [Slash-Command Argument Binding — Failure-mode templates](./binder.md#failure-mode-templates-normative)) |

In V1 the only non-empty reachable mask is therefore `["ceiling#2"]`, reachable only on the `validation` event at the typed-query response sub-row of CIO-3's AJV boundary. Every other co-fire-eligible surface MUST omit `masked`.

*(d) V1 reachable predicate (the only co-fire that fires in V1).* On a `validation` event with `cause: "schema_validation"` raised at the typed-query response boundary, `masked` carries `["ceiling#2"]` *if and only if* the surfaced event was raised on a forced respond turn whose origin round, after CIO-4's slot increment for the just-completed free-phase round, leaves the `tool_loop` slot count equal to `max_rounds` (i.e. the forced respond turn is the final permitted slot). In every other event — including a respond-repair follow-up's depth-6 forced respond turn (each follow-up gets a fresh `tool_loop` budget per [Query — Tool-call loop bound](./query.md#tool-call-loop-bound), so the predicate is evaluated against the follow-up's own slot count, not the parent query's), a typed query with `max_rounds: 0` (the model gets an empty tool-set during the free phase per V6k, so the forced respond turn is the *only* turn and the predicate is unreachable; `masked` is always omitted), and any non-typed-query-response site — `masked` is omitted. The predicate collapses to two scalar reads: the post-increment `tool_loop` slot count and the surfacing turn kind (forced-respond vs. free-phase). Both scalars are state the two-phase tool-loop driver already maintains.

*(e) Pure-read MUST.* Detection at the surfaced site MUST be a pure read of the sibling preconditions: evaluating the predicate above MUST NOT advance the `tool_loop` counter, mutate round bookkeeping, re-invoke ceiling #4's depth-walk, or trip a second emission. Implementations MAY cache the two scalars at the moment the surfacing event is constructed; they MUST NOT re-derive them by re-running ceiling-checking code paths.

*(f) Verbatim-copy obligation on cascade-twin re-emission.* The cascade-twin re-emission rule under **Deduplication and lifetime rules** above (a `?`-propagation chain that cascades to a top-level boundary emits the originating `RuntimeEvent` payload twice — once at the origin, once at the boundary) MUST copy the originating instance verbatim including `masked`. Re-deriving `masked` at the boundary site is forbidden — the originating site is the only authoritative source, and the boundary site has no access to the per-frame scalars the predicate above reads.

*(g) Dedup-key non-inclusion rule.* The dedup tuple under **Deduplication and lifetime rules** above is `(kind, query_site, message, occurred_at)`; consumers MUST NOT include `masked` in the dedup tuple. Two emissions that differ only in `masked` represent the same origin and a malformed re-emission, not two distinct occurrences.

The wire location of `masked` (`details.event.masked` for runtime-event surfaces, `details.masked` for diagnostic-shape surfaces) is owned by [Hard Runtime Ceilings — `masked` field](./hard-ceilings.md#masked-field). A worked example exhibiting a non-empty `masked` payload is at [Query — Tool-call loop bound, *Worked example: depth-6 forced respond at `max_rounds`*](./query.md#tool-call-loop-bound).

The `pi.sendMessage` call for `loom-system-note` is treated as best-effort. `pi.sendMessage` returns `void` (synchronous); the runtime MUST NOT `await` it and MUST NOT attach a `.catch` handler. The best-effort fallback below covers synchronous throws only. If it throws, the runtime falls back in this order:

1. `ctx.ui.notify(content, "error")` — a transient surface so the user still sees the message in the current session, even if it does not persist in the transcript. **Skipped when `display: false`**: notifying the user transiently about an event whose author handled the underlying `Err` (or a subagent-private cascade) defeats the purpose of `display: false`; the fallback proceeds straight to step 2. This step is also skipped when the original note's `content` is `""`, since `ctx.ui.notify("", "error")` carries no signal — but in V1 this case is fully covered by the `display: false` skip rule above (the only `content: ""` variant is the `display: false` runtime-event case).
2. A `loom/runtime/system-note-delivery-failed` diagnostic emitted through the standard diagnostics channel (see [Diagnostics](./diagnostics.md)), with `message` set to the original note's `content` (which MAY be `""`) and `hint` set to the underlying `sendMessage` error's message. This diagnostic is itself best-effort: if both channels fail, the failure is logged to `console.error` and execution continues.

The fallback path is taken on any thrown value from `sendMessage`; it does not retry the original call. The fallback never aborts the slash-command handler or the spawned subagent session. Implementers must guard against re-entry: if a future `loom/runtime/*` handler ever routes diagnostics back through `loom-system-note`, the diagnostic step in this fallback MUST NOT re-invoke `pi.sendMessage`. `ctx.ui.notify` itself can throw (e.g. in print mode where Pi's UI is not attached); wrap it in the same try/catch and proceed to the diagnostic step. For panic-routed notes, the original panic message MUST be included in the final-resort `console.error` log so post-mortem debugging retains the stack.

**`estimateTokens` (named export).** Imported from `@mariozechner/pi-coding-agent`. Signature: `estimateTokens(message: Message): number`. V1 behaviour: a conservative `Math.ceil(chars / 4)`, where `chars` is the sum of `String.prototype.length` (UTF-16 code units, as defined by ECMA-262) taken over each contributing string in the message: text-block content, assistant `thinking`-block content, `toolCall.name`, `JSON.stringify(toolCall.arguments)` (with no `replacer` and no `space`, so two engines agree on the input string before `.length` is taken), and tool-result text content. Astral code points (U+10000 and above) therefore count as 2; lone surrogate halves count as 1; no normalisation, escaping, or special handling is applied. Non-text content blocks (e.g. images) follow Pi's own accounting in `@mariozechner/pi-coding-agent` rather than being re-specified here, since the `Message` shape is owned by Pi. *Reference vector (normative).* A user message whose only content is the text `"😀😀😀"` (three U+1F600, six UTF-16 code units) MUST estimate to `Math.ceil(6 / 4) = 2` tokens. *Pi parity (advisory).* The unit and formula match Pi's own compaction-decision estimator in `@mariozechner/pi-coding-agent`'s `core/compaction/compaction.ts`, which is why this unit was chosen — binder truncation behaviour stays consistent with Pi as model tokenizers evolve — but the normative requirement is the unit and formula above, not parity with any particular Pi revision. Used by [Slash-Command Argument Binding — Session-context truncation](./binder.md#session-context-truncation-bind_context-session) for per-message and per-turn token counts.

**`buildSessionContext` (named export).** Imported from `@mariozechner/pi-coding-agent`; this is a free function, **not** a method on `ctx.sessionManager` — `ExtensionContext.sessionManager` is typed as `ReadonlySessionManager` (a `Pick<>` of the underlying `SessionManager`; see the inline shape below) and deliberately excludes `buildSessionContext` and every mutating member. Signature: `buildSessionContext(entries: SessionEntry[], leafId?: string | null, byId?: Map<string, SessionEntry>): SessionContext`. The runtime calls it as `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId())` — both `getEntries()` and `getLeafId()` are inside the `ReadonlySessionManager` `Pick<>`, so the call composes against the read-only surface in both prompt mode and subagent mode. The returned `SessionContext`'s `.messages` field is the ordered message list the runtime consumes for session-aware features. V1 callsite: [Slash-Command Argument Binding — Session-context truncation](./binder.md#session-context-truncation-bind_context-session) sources the binder's caller-session message list from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`. The runtime MUST NOT substitute `ctx.getContextUsage()` for per-turn token accounting; that API reports aggregate session usage, not per-turn counts, and is reserved for compaction triggers and footer rendering.

**`ExtensionContext` (member surface loom touches).** The runtime touches exactly the members enumerated in the signature block below; every other member of Pi's `ExtensionContext` is non-load-bearing for V1 and may be added or revised by Pi without spec changes. The canonical Pi-side declaration lives at `dist/core/extensions/types.d.ts` in `@mariozechner/pi-coding-agent ~0.72.1` (the V1 Pi-SDK pin from **Host prerequisites** above); the inline shape below is the loom-load-bearing subset and MUST be re-validated against that file on each Pi minor bump:

```ts
interface ExtensionContext {
  cwd: string;                                          // working directory of the host Pi session
  signal: AbortSignal | undefined;                       // Pi-supplied turn signal; `undefined` in idle / non-turn contexts (see **Cancellation source** below)
  sessionManager: ReadonlySessionManager;                // host's session manager exposed as a read-only `Pick<SessionManager, …>` (members `getCwd | getSessionDir | getSessionId | getSessionFile | getLeafId | getLeafEntry | getEntry | getLabel | getBranch | getHeader | getEntries | getTree | getSessionName`); kept in lock-step with `dist/core/session-manager.d.ts`'s `ReadonlySessionManager` `Pick<>` and re-validated on each Pi minor bump. Subagent mode swaps the underlying object to the loom's spawned in-memory `SessionManager` (see per-mode override table below) but the static type narrows to the same `ReadonlySessionManager` surface in both modes — loom code MUST NOT capture the wider `SessionManager` instance in a parallel closure to bypass this contract. The free `buildSessionContext` named export (see above) is the supported way to obtain a `SessionContext` off the read-only surface.
  modelRegistry: ModelRegistry;                          // resolves provider + modelId to a `Model<Api>`; consumed by the binder pass and by binder-model resolution
  model: Model<any> | undefined;                         // host's currently-selected model; `undefined` arm reachable when no model is yet selected — loom never relies on the value being defined
  ui: ExtensionUIContext;                                // transient toast surface; the only member loom calls is `ui.notify(message: string, kind: "info" | "warning" | "error"): void` (synchronous, may throw — wrap in try/catch when called from the **System notes** fallback chain)
  isIdle(): boolean;                                     // true iff the host session has no in-flight turn; loom consults this only to choose between `pi.sendUserMessage` shapes (see **Conversation drive — prompt mode** above)
  abort(): void;                                         // documented by Pi as "abort the current agent operation"; **the loom runtime wraps this** — see override table below
  getContextUsage(): ContextUsage | undefined;           // aggregate session-level usage; `undefined` arm is the "not yet computable" sentinel (no throw); loom MUST NOT substitute this for per-turn token accounting (see `buildSessionContext` above)
  hasPendingMessages(): boolean;                          // observed by the slash-command handler's pre-turn idle probe; not consumed elsewhere in the loom runtime
  compact(options?: CompactOptions): void;               // host-driven compaction trigger; not invoked by loom in V1 (`await ctx.compact()` is a type-safe no-op — `await void` resolves immediately and the completion signal is silently lost); completion / failure observed via `options.onComplete(result: CompactionResult)` / `options.onError(err: Error)`; forwarded unchanged in both modes, listed here so the touched-surface inventory is exhaustive
  shutdown(): void;                                      // documented by Pi for in-process shutdown; not invoked by loom in V1
  getSystemPrompt(): string;                             // host's effective system prompt; not consumed by loom in V1
}
```

`CompactOptions` and `CompactionResult` (referenced by the `compact` row above) are declared at `dist/core/extensions/types.d.ts` lines 199–203 and 233 respectively in `@mariozechner/pi-coding-agent ~0.72.1` (the V1 Pi-SDK pin from **Host prerequisites** above) as `CompactOptions = { customInstructions?: string; onComplete?: (result: CompactionResult) => void; onError?: (error: Error) => void }`; `CompactionResult` is the host-defined result payload threaded through `onComplete`. Loom does not construct either value in V1 — both are listed only so the touched-surface inventory is closed against the cited declarations and re-validated on each Pi minor bump per [Pi version bump procedure](#pi-version-bump-procedure) below.

Members loom does not touch (e.g. `reload()`, `newSession()`, `fork()`, `switchSession()`) are deliberately omitted; their absence from this table is normative — the loom runtime MUST NOT call them in V1, and a future widening is a spec-versioned change. `waitForIdle()` is intentionally absent: it is a member of `ExtensionCommandContext` (the per-handler subtype that extends `ExtensionContext`; see **`ExtensionCommandContext` (per-handler subtype loom touches)** below), not of `ExtensionContext` itself, and is consulted only on the captured slash-command handler context (see **Conversation drive — prompt mode** above), never on the synthesised tool-execution `ctx`.

<a id="extensioncommandcontext-interface"></a>

**`ExtensionCommandContext` (per-handler subtype loom touches).** `ExtensionCommandContext` is the per-handler subtype Pi passes to a slash-command `handler`; it is declared at `dist/core/extensions/types.d.ts` in `@mariozechner/pi-coding-agent ~0.72.1` (the V1 Pi-SDK pin from **Host prerequisites** above) as `interface ExtensionCommandContext extends ExtensionContext { … }`, so it inherits every member of the `ExtensionContext` signature block above. The runtime touches exactly the additional members enumerated below; every other `ExtensionCommandContext`-only member is non-load-bearing for V1 and the loom runtime MUST NOT call it. The inline shape below is the loom-load-bearing subset and MUST be re-validated against the cited file on each Pi minor bump per [Pi version bump procedure](#pi-version-bump-procedure) below; a Pi minor that renames or moves the type, or removes any member loom touches, surfaces at the H1 SDK surface-inventory test before the spec contract drifts.

```ts
interface ExtensionCommandContext extends ExtensionContext {
  waitForIdle(): Promise<void>;                          // resolves once Pi emits `agent_end` for the user session and every awaited `agent_end` listener settles; never rejects (see **Conversation drive — prompt mode** above)
  // Members loom MUST NOT touch in V1: newSession, fork, navigateTree, switchSession, ReplacedSessionContext, …
}
```

Members loom MUST NOT touch in V1 (e.g. `newSession()`, `fork()`, `navigateTree()`, `switchSession()`, the `ReplacedSessionContext` surface) are deliberately omitted from the inline shape; their absence from this table is normative — the loom runtime MUST NOT call them in V1, and a future widening (the deferred *Mid-loom user-session replacement* extension in [Future Considerations](./future-considerations.md#mid-loom-user-session-replacement) is the only V1 surface that anticipates such use) is a spec-versioned change. The `waitForIdle()` member is the only `ExtensionCommandContext`-only member loom touches in V1; it is consulted on the captured slash-command handler context only (the `ExtensionCommandContext` Pi passes to the loom's slash-command `handler`), never on the synthesised tool-execution `ctx` from the **`ExtensionContext`** section above (which is typed as the wider `ExtensionContext` and does not expose `waitForIdle`).

*Per-mode override semantics.* The runtime synthesises the `ctx` it hands to `tool.execute(...)` by starting from the live host's `ExtensionContext` and overriding exactly three members. The overrides apply uniformly in both modes — there is no member that is overridden in one mode and forwarded unchanged in the other:

| Member | Prompt mode | Subagent mode |
|---|---|---|
| `signal` | replaced with `loomAbort.signal` (always defined) | replaced with `loomAbort.signal` (always defined) |
| `sessionManager` | replaced with Pi's user-session manager (the host's, observed via the captured `ExtensionCommandContext`) | replaced with the loom's spawned in-memory `SessionManager` (the one passed to `createAgentSession({ sessionManager })` per **Conversation drive — subagent mode** above) |
| `abort` | wrapped to call `loomAbort.abort()` rather than the host's `abort()` — so `ctx.abort()` tears down the loom's invocation, not the user's whole Pi turn; a throw from the underlying `loomAbort.abort()` invoked by the wrapper is trapped per the [Cancellation — Forwarding-listener throw](./cancellation.md) clause and routed through `loom/runtime/internal-error` | wrapped identically — `ctx.abort()` tears down the subagent invocation, not the parent's turn; the same forwarding-listener-throw trap applies |

Every other member from the signature block above forwards unchanged to the live host:

- *Prompt mode.* The live host is the `ExtensionCommandContext` passed to the slash-command handler, retained for the loom's lifetime. `cwd`, `ui`, `modelRegistry`, `model`, `isIdle`, `getContextUsage`, `hasPendingMessages`, `compact`, `shutdown`, and `getSystemPrompt` all reflect the user-session view at the moment of the tool call.
- *Subagent mode.* The spawned `AgentSession` does not carry an `ExtensionCommandContext`; the runtime forwards the parent slash-command handler's `ExtensionCommandContext` for the non-overridden members (so `model`, `modelRegistry`, `cwd`, `getContextUsage`, etc. reflect the parent's view, not a per-subagent one). The `cwd` value is the parent's `ctx.cwd` verbatim — V1 has no per-subagent frontmatter `cwd` override; widening is reserved for [Future Considerations](./future-considerations.md).

The `abort()` override applies in **both** modes because no spec-supported caller of `ctx.abort()` from inside loom code wants to tear down the user's turn — the in-loom code paths that call `abort()` are tools acting on behalf of the loom, whose authoritative cancellation signal is `loomAbort`. The contract pinned by this paragraph ("`ctx.abort()` aborts the loom's invocation and not the parent's turn") under the spec's "Tool execution from loom code" surface holds in both prompt and subagent modes.

<a id="tool-execution-from-loom-code"></a>

**Tool execution from loom code.** Code-side `<name>(args)` calls invoke the Pi tool's `execute(toolCallId, params, signal, onUpdate, ctx)` directly:

- `toolCallId` is a synthesised UUID prefixed `loom-direct:`.
- `params` is the loom value lowered to JSON (wire names applied).
- `signal` is the loom's current `loomAbort.signal` (always defined; see **Cancellation source** below).
- `onUpdate` is a no-op (V1 does not surface streaming partial results to loom code).
- `ctx` is the synthesised `ExtensionContext` defined under [`ExtensionContext` (member surface loom touches)](#extensioncontext-interface) above: the live host's `ExtensionContext` with `signal`, `sessionManager`, and `abort()` overridden per the per-mode-override table, and every other member forwarded to the live host.

The tool's returned `{ content, isError }` envelope shape — `content: ({ type: "text"; text: string } | …)[]` and `isError: boolean` — is declared on the tool-result type family in `dist/core/extensions/types.d.ts` of `@mariozechner/pi-coding-agent ~0.72.1` (the V1 Pi-SDK pin from **Host prerequisites** above; the `isError` field appears on the relevant tool-result type declarations in that file). The envelope shape is the contract every code-driven tool call's outcome routing depends on; the H1 SDK surface-inventory test asserts the field set on the cited type so a Pi minor that renames `isError`, drops the `text`-block discriminator, or widens `content` to a non-array shape surfaces at the build before the spec contract drifts. The envelope is lowered to the V1 string return value as follows. The lowering procedure inspects the resolved envelope and any throw raised during that inspection (the resolved value is not an object, `content` is not iterable, an entry is missing `type` / `text`, or any other shape violation that causes a property access or `.filter` / `.map` call to throw) is caught by the runtime and routed to `loom/runtime/internal-error` with `details.kind = "tool-return-shape"` (carrying `tool_name` and a discriminator describing the offending shape — e.g. `typeof resolved`, `Array.isArray(content)`); the call site observes the `internal-error` routing per [Errors and Results — Runtime panics](./errors-and-results.md), **not** an `Err(QueryError { kind: "code_tool", ... })`. The same rule applies if the lowering throws while assembling the `cause: "execution"` message described below (e.g. an `isError: true` return whose `content` array contains a non-iterable entry). The runtime filters `content` to entries with `type === "text"`, joins their `.text` values with `"\n"` (single line-feed; no separator before the first or after the last block), and returns the joined string. Non-text content blocks (images, resource references) are discarded during lowering; the loom-level observable is exhausted by the resulting `Ok("")` value. The runtime MUST NOT emit a `RuntimeEvent`, `loom-system-note`, or any other normative diagnostic on the discard path — non-text discard is not a `QueryError` and is not a member of the always-log set. Implementations MAY emit non-normative debug-channel logs for operator visibility. Widening to a richer return shape (preserving non-text blocks) is reserved for [Future Considerations](./future-considerations.md). An empty result string — either `content: []` or a content array with no surviving text blocks — is a legal `Ok("")` value when `!isError`. When `isError`, the runtime returns `Err(QueryError { kind: "code_tool", cause: "execution", message: <m>, tool_name, ... })` where `<m>` is the same filtered/joined text encoded as UTF-8 and truncated so that the resulting byte length is at most 4096 bytes. Truncation MUST cut on a Unicode code-point boundary: every code point in the output is represented by all of its UTF-8 bytes, and no bytes of a partial code point appear. When a code point would straddle the 4096-byte limit, that code point is dropped entirely; the resulting message MAY therefore be shorter than 4096 bytes by up to three bytes. Worked example: a filtered/joined text whose first 4095 bytes are ASCII followed by a 3-byte UTF-8 code point (e.g. U+2026 `…`) MUST truncate to 4095 bytes, because including the 3-byte code point would yield 4098 bytes. When no text survives the filter, `<m>` is the fixed string `"tool reported an error with no text content"`. When `cause: "execution"` originates from an `execute()` throw rather than `isError: true`, `<m>` is the thrown error's `.message` truncated under the same rule. Both forms share the `"execution"` cause; `execute()`-throw vs `isError: true` are distinguished by the `message` source, not by a new `cause` (a future revision MAY widen `cause` additively if the distinction needs to surface).

*Outcome routing summary.* The complete V1 routing of an `execute()` outcome at this boundary is fixed: a clean `{ content, isError: false }` lowers to `Ok(<filtered/joined text>)` (possibly `Ok("")`); a clean `{ content, isError: true }` lowers to `Err(CodeToolError { cause: "execution", ... })`; an `execute()` throw lowers to `Err(CodeToolError { cause: "execution", message: <error.message truncated>, ... })`; a `loomAbort.signal`-aborted call lowers to `Err(CodeToolError { cause: "cancelled", ... })` per the cancellation surfacing rule in [Cancellation — Surfacing](./cancellation.md); a non-conforming envelope (anything that throws during the lowering above) routes to `loom/runtime/internal-error` with `details.kind = "tool-return-shape"` and is **not** observable as a `CodeToolError`; a non-settling Promise blocks the call's `await` indefinitely until cancellation fires, at which point the cancelled path applies (V1 makes no internal timeout attempt, consistent with [`spec.md` — Hard ceilings, NOCEIL-1](../spec.md#hard-runtime-ceilings); the deferred per-call timeout extension lands additively on the open-struct seam in [Tool Calls](./tool-calls.md)); a settlement that arrives after the tool-call checkpoint has already surfaced `cause: "cancelled"` is discarded per [Cancellation — Race semantics](./cancellation.md) — the runtime MUST keep a swallowing handler attached to the underlying `execute()` Promise (and, symmetrically, to the underlying `@`-query provider Promise and the `invoke` callee's top-level execution Promise) for the lifetime of that Promise, installed at construction time before the first microtask boundary, so that a late rejection on the discarded path does **not** surface as a Node `unhandledRejection` process event; the discard is total (no `Err` rebinds the call site, no `RuntimeEvent` reaches the always-log channel, no diagnostic of any severity is emitted, and the discarded value is not traversed). The `CodeToolError.cause` enum (closed at `validation` / `execution` / `cancelled` / `unknown_tool` per [Errors and Results — `CodeToolError`](./errors-and-results.md#queryerror-variants)) is the contract surface for loom authors; the non-conforming-shape and non-settling-Promise dispositions are deliberately routed off it.

<a id="cancellation-source"></a>

**Cancellation source.** Pi's delivery of an `AbortSignal` at the two extension entry points loom uses (`ctx.signal` and `tool.execute`'s `signal` parameter) is the SDK prerequisite this section consumes; see **Host prerequisites** above. The two entry-point declarations live in `dist/core/extensions/types.d.ts` of `@mariozechner/pi-coding-agent ~0.72.1`: `ExtensionContext.signal: AbortSignal | undefined` (the slash-command handler's turn signal, with the JSDoc verbatim *“current abort signal, or undefined when the agent is not streaming”*) and the `signal: AbortSignal | undefined` parameter of the tool `execute(...)` member of `ToolDefinition`. Subagent mode has no `AbortSignal` entry point — `CreateAgentSessionOptions` (declared at `dist/core/sdk.d.ts` in the V1 Pi-SDK pin per **Conversation drive — subagent mode** above) has no `signal` field — and instead consumes the `AgentSession.abort()` capability on the spawned handle, per **Subagent session lifecycle** above. As described in [Cancellation](./cancellation.md), the loom's `AbortSignal` is `loomAbort.signal` from a runtime-owned `AbortController` constructed fresh per invocation — **not** `ctx.signal` directly. Per the JSDoc above, `ctx.signal` is `undefined` in idle / non-turn contexts, and a slash-command handler fires from idle, so the runtime MUST NOT use `ctx.signal` as the loom's authoritative signal: doing so would silently pass `undefined` into every adapter (the synthesised tool-execution `ExtensionContext.signal`, the child-invoke derived signal) and degrade every cancellation path to a no-op. Instead, the runtime forwards Pi's turn-side `ctx.signal` (when defined and observed inside one of the runtime's `tool_call` / `tool_result` / `message_update` / `turn_end` / `agent_end` handlers) into `loomAbort.abort()`, and likewise forwards the `signal` parameter of a tool-exposed loom's `execute(...)` into `loomAbort.abort()` via a one-shot listener registered at entry. All downstream queries, tool calls, and child invokes derive from `loomAbort.signal`. Because `loomAbort.signal` is always defined, the optional-chaining defence (`signal?.aborted`) used by tool adapters is no longer load-bearing inside the runtime; the runtime guarantees a defined signal to every adapter it calls. Subagent mode propagates cancellation through the one-shot `loomAbort.signal` listener defined in **Subagent session lifecycle** above: when `loomAbort` fires, the listener calls `AgentSession.abort()` on the spawned handle, which tears down the in-flight provider call. Forwarding listeners — including the subagent `abort()` listener — are torn down in the same `finally` block that disposes the subagent session (see **Subagent session lifecycle** above). The `reason` value forwarded into `loomAbort.abort(reason)` for each forwarding path — including the runtime-synthesised values for the `agent_end`-driven and `session_shutdown`-driven triggers, which have no source `AbortSignal` — is pinned by [Cancellation — Abort-reason propagation](./cancellation.md); the rule applies uniformly across this paragraph's three forwarding shapes.

<a id="checkpoint-seam"></a>

**`Checkpoint` seam.** The runtime exposes a single internal hook the interpreter `await`s immediately before each cancellation checkpoint defined by the **Granularity** rule in [Cancellation](./cancellation.md). The hook exists so that the two race rules in [Cancellation — Race semantics](./cancellation.md) — no retroactive `Ok(v)` → `Err({kind:"cancelled"})` rewrite, and no top-level synthesis when no further checkpoint executes after the abort — are deterministically testable without depending on JS microtask scheduling. Production wiring is a no-op (an already-resolved promise); the seam has no observable production effect beyond the cost of one already-resolved promise per checkpoint.

The interface shape is:

```ts
type CheckpointKind =
  | "loop-iter"
  | "query"
  | "tool-call"
  | "invoke"
  | "binder-call";

interface CheckpointSite {
  file: string;
  line: number;
  column: number;
}

interface Checkpoint {
  before(kind: CheckpointKind, site: CheckpointSite): Promise<void>;
}
```

Rules:

- The interpreter `await`s `checkpoint.before(kind, site)` immediately *before* reading `loomAbort.signal.aborted` at every checkpoint enumerated in [Cancellation — Granularity](./cancellation.md): each `for`/`while` iteration boundary, each `@`-query, each tool call, each `invoke`, and the binder LLM call (the binder checkpoint fires even though the binder runs outside the loom body, so the cancelled-binder failure-mode in [Slash-Command Argument Binding](./binder.md) is observable through the same seam). The await happens unconditionally; production no-op resolves on the next microtask, and the always-`await` shape is what lets a test inject async work between an operation's resolution and the signal-check.
- The hook does **not** observe non-checkpoint synchronous work (AJV validation, schema lowering, default-merging) — the **Granularity** rule already excludes those, and the seam mirrors that exclusion exactly. Implementations MUST NOT add `before(...)` calls at any site not enumerated above.
- Production wiring constructs one no-op `Checkpoint` per `loomAbort` (i.e., per loom invocation). For `invoke`, parent and child each own their own `Checkpoint` instance constructed from the same factory, mirroring the per-invocation `loomAbort` rule from [Cancellation](./cancellation.md); a parent's `before(...)` is not observed by the child and vice versa.
- Test wiring is the only consumer that observes calls. A test that fires `loomAbort.abort()` from inside `before(...)` exercises the *abort observed at this checkpoint* path; a test that fires `loomAbort.abort()` from inside the *previous* checkpoint's `before(...)` (or from any code awaited inside that previous `before(...)`) exercises the *abort landed between checkpoints* path — i.e., the no-retroactive-rewrite rule. A test that fires `loomAbort.abort()` after the loom's final checkpoint and before its top-level resolution exercises the no-tail-synthesis rule. These three patterns are the canonical use of the seam.
- Test fakes MUST treat the seam as call-once-per-checkpoint and MUST NOT skip the await (the no-skip rule is what gives the test deterministic control over the post-resolution / pre-signal-check window).
- The seam is internal: it is not exposed through `ExtensionAPI` and not visible to loom authors, Pi extensions, or tools. The runtime wires it through the same constructor-injection skeleton that owns `FileSystem`, `DiagnosticsSink`, `SchemaValidator`, etc.

<a id="clock--fakeclock-interface"></a>

**`Clock` / `FakeClock` interface.** The runtime reads wall-clock time and schedules deferred work exclusively through a `Clock` seam injected at construction time, modelled on the `FileSystem` seam. The interface members called out as load-bearing by other spec sections are:

- `now(): number` — monotonic milliseconds. Used by the **Runtime event channel** above to stamp `RuntimeEvent.occurred_at` at the originating emission site, and by [Discovery — Package discovery](./discovery.md#package-discovery) to read elapsed time for the `looms.scanPackagesTimeoutMs` cap.
- `setTimeout(fn: () => void, ms: number): TimerHandle` and `clearTimeout(handle: TimerHandle): void` — the only timer surface the runtime uses. The watcher's 250 ms debounce (step 5 above) and the settings-watcher debounce (per [Discovery — Settings file reads](./discovery.md#settings-file-reads)) both schedule through this seam. The debouncer holds the most recent timer handle and clears it on each new event.

Production wiring uses a `WallClock` adapter that delegates `now()` to `performance.now()` and the timer methods to the global `setTimeout` / `clearTimeout`. Tests use a `FakeClock` whose `advance(ms: number)` synchronously fires every timer whose deadline has elapsed in deadline order; equal-deadline timers fire in registration order, `clearTimeout` is a no-op for already-fired handles, and `now()` returns the fake's accumulated time and is *not* implicitly advanced. `Clock.now()` is monotonic (forbids `Date.now()`-style NTP drift). One `Clock` instance is constructed per runtime instance — never as a module-level global — and lives for the lifetime of that runtime; parallel runtimes get independent clocks. The runtime MUST NOT call `Date.now`, `performance.now`, `Date.prototype.getTime`, or the global `setTimeout` / `clearTimeout` outside the `WallClock` adapter; a build-time grep-test enforces this (parallel to the `process.env.HOME` ban for `homedir()`).

<a id="fakefilesystem--filesystem-interface"></a>

**`FakeFileSystem` / `FileSystem` interface.** The runtime reads the filesystem exclusively through a `FileSystem` seam injected at construction time; production wiring uses a `PiFileSystem` adapter that delegates to Node, and tests use an in-memory `FakeFileSystem` whose constructor takes the values it should report. The interface shape is normative:

```ts
interface FileStat {
  isDirectory(): boolean;
  isFile(): boolean;
  isSymbolicLink(): boolean;
}

interface FileSystem {
  readText(path: string): Promise<string>;          // rejects with a Node-style error whose `.code` is `"ENOENT"` for missing paths and `"EACCES"` / `"EPERM"` for permission failures; other I/O failures surface their underlying `.code` unchanged
  writeText(path: string, contents: string): Promise<void>;
  exists(path: string): Promise<boolean>;           // resolves `false` on `ENOENT`; rejects on any other error
  homedir(): string;                                // production never reads `process.env` directly
  readdir(path: string): Promise<readonly string[]>;// entry names only (no full paths); rejects with the same Node-style `.code` shape as `readText`
  lstat(path: string): Promise<FileStat>;           // does NOT follow symlinks; rejects with the same Node-style `.code` shape as `readText`
}
```

- `homedir()` is the single source of truth for the [Home-directory expansion](./discovery.md#home-directory-expansion) rule; the production `PiFileSystem` implementation calls Node's `os.homedir()` (resolving to `$HOME` on POSIX and `%USERPROFILE%` on Windows), and the `FakeFileSystem` implementation returns a constructor-injected string. The runtime MUST NOT read `process.env.HOME` or `process.env.USERPROFILE` directly and MUST NOT use any platform-conditional branch.
- `readText`, `writeText`, and `exists` are the load-bearing surface for [Directory Convention — Settings file reads](./discovery.md#settings-file-reads) and for every other normative file read or write the runtime performs.
- `readdir` and `lstat` are the load-bearing surface for the *clean leaf-`ENOENT`* ancestor walk defined in [Directory Convention](./discovery.md) (the **Failure modes** paragraph and the bullet that defines *clean leaf-`ENOENT`*) and for the package-discovery walk in [Directory Convention — Package discovery](./discovery.md#package-discovery). Rejected `.code` values map onto the discovery rules verbatim: an `ENOENT` chain that bottoms out cleanly is *missing*; an `EACCES` / `EPERM` / `ENOTDIR` (or any other code) anywhere on the chain is an unreadable-source failure.

<a id="filewatcher-interface"></a>

**`FakeFileWatcher` / `FileWatcher` interface.** Watcher attachment is a *separate* seam from `FileSystem` — production wires a chokidar adapter, and tests use an in-memory `FakeFileWatcher` whose `emit(event)` synchronously invokes every attached handler. The interface shape is normative:

```ts
type FileWatchEventKind = "add" | "change" | "unlink";

interface FileWatchEvent {
  kind: FileWatchEventKind;                         // matches chokidar's three load-bearing event names
  path: string;                                     // absolute path of the affected file
}

type Unsubscribe = () => void;

interface FileWatcher {
  watch(roots: readonly string[], handler: (event: FileWatchEvent) => void): Unsubscribe;
}
```

- `watch` attaches one handler over the supplied root paths. The returned `Unsubscribe` tears down the underlying watcher (chokidar's `close()` in production); calling it twice is a no-op. The handler observes only the three event kinds above — chokidar's `addDir`, `unlinkDir`, `ready`, `error`, etc. are filtered out by the adapter and never reach the runtime; the chokidar-adapter's own `error` events route through the `ctx.ui.notify` toast surface defined in [Diagnostics](./diagnostics.md), not through this seam.
- The watcher's 250 ms debounce, the build-aside-then-publish swap, and the structural-vs-content event split (per **Extension entry point** step 5 above) are runtime responsibilities layered on top of this seam — they are *not* properties of the seam itself; the seam only delivers raw events.
- One `FileWatcher` instance is constructed per runtime instance — never as a module-level global — and lives for the lifetime of that runtime; parallel runtimes get independent watchers.

**Discovery API.** The extension owns enumeration; it subscribes to `resources_discover` solely as a re-discovery trigger and returns the empty result `{}` (`ResourcesDiscoverResult` has no `loomPaths` field — looms reach Pi through `pi.registerCommand`, not as a Pi-managed resource type). The five sources, their priority order, the package-root walk, and the failure-mode table are all in [Directory Convention](./discovery.md). Cross-format collision detection consults `pi.getCommands()` on `session_start` (per the **Extension entry point** numbered list above); the call would throw if invoked during the extension factory itself (per the pre-bind action-method rule in **Extension entry point** step 2), which is why discovery is split across the factory (parse + pending list) and the `session_start` handler (collision check + `pi.registerCommand`).

> **V1 seam — Pi-owned subagents collision source set.** <a id="v1-seam-pi-owned-subagents-collision-source-set"></a> The set of `SlashCommandSource` values the `session_start` handler treats as collision candidates (`"prompt"`, `"extension"`, `"skill"` under `@mariozechner/pi-coding-agent ~0.72.1`) MUST be defined as a single named set inside the runtime, consulted by the collision check via membership test rather than open-coded as three separate string comparisons or a hard-coded `switch`. The deferred *Pi-owned subagents exposed as enumerable slash commands* extension in [Future Considerations](./future-considerations.md) lands by widening that set to four arms (adding a `"subagent"` entry) when a future Pi minor extends `SlashCommandSource` accordingly; widening the set is the only V1 surface that perturbs, and the asymmetric loser-drops collision rule defined in [Discovery](./discovery.md) is preserved by construction.

**No Pi-owned discovery path enumerates `.loom` or `.warp`.** Under the pinned `~0.72.1` peer-dep range, Pi exposes exactly three slash-command sources (`source: "prompt" | "extension" | "skill"`, per `@mariozechner/pi-coding-agent`'s `core/slash-commands.d.ts`); `prompt` and `skill` enumerate `*.md` files only, and `extension` requires programmatic `pi.registerCommand` calls. The `resources_discover` event carries `skillPaths` / `promptPaths` / `themePaths` only — there is no `loomPaths` slot — and the `pi` package-manifest namespace recognises only `extensions` / `skills` / `prompts` / `themes` / `video` / `image`. Therefore the only path by which a `.warp` file could become a slash command is through this extension's own discovery walk; that walk matches `*.loom` only (per [Directory Convention](./discovery.md) and [Discovery — File-extension namespace](./discovery.md#file-extension-namespace)), so the `.warp`-cannot-be-slash-invoked guarantee in [`spec.md`](../spec.md) Introduction holds by construction. A Pi minor that adds a fourth `SlashCommandSource` arm, a `loomPaths` field on `ResourcesDiscoverResult`, or a generic file-extension registry MUST trigger a re-validation of this paragraph as part of the [Pi version bump procedure](#pi-version-bump-procedure) below, in the same edit that widens `peerDependencies`.

## SDK capability inventory

The runtime depends on the seven named SDK capabilities enumerated below. Items 1, 2, 3, 4, and 6 are factory-probable and verified by [Step 0 (Capability probe)](#entry-capability-probe) above (which on any failure skips every factory-time host-binding call — the four `pi.register*` calls and `pi.on` — and emits `loom/load/host-incompatible`). Item 5 (cancellation propagation) is verified jointly by Step 0 item (b)'s `AbortSignal` / `AbortController` shape probe (the two Pi-side entry-point signals) and Step 0 item (c)'s `AgentSession.prototype.abort` member check (the subagent-mode primitive). Item 7 (binder LLM model) is verified at per-loom load time and surfaces as `loom/load/binder-model-unresolved` per [Slash-Command Argument Binding — Strict-capability requirement](./binder.md#strict-capability-requirement); it is **not** factory-probable and the two diagnostic surfaces (`loom/load/host-incompatible` and `loom/load/binder-model-unresolved`) are disjoint. [`spec.md` — Orientation — Prerequisites](../spec.md#orientation) name-links each item back to the obligation here so that plan leaves and tests cite a single anchored source.

1. <a id="sdk-cap-slash-command-registration"></a> **Slash-command registration.** Pi MUST expose `pi.registerCommand(name, { description, getArgumentCompletions, handler })` per **Extension entry point** above. The runtime calls this once per surviving discovered loom on `session_start`.

2. <a id="sdk-cap-prompt-conversation-drive"></a> **Prompt-mode conversation drive.** Pi MUST expose `pi.sendUserMessage(text [, { deliverAs }])` for issuing turns and `ExtensionCommandContext.waitForIdle()` for awaiting turn completion. Both are consumed by the prompt-mode driver per **Conversation drive — prompt mode** above.

3. <a id="sdk-cap-subagent-isolated-session"></a> **Subagent-mode isolated session.** Pi MUST expose `createAgentSession(...)` returning a disposable `AgentSession` whose transcript is private when `SessionManager.inMemory(cwd)` is supplied. Consumed by **Conversation drive — subagent mode** and **Subagent session lifecycle** above.

4. <a id="sdk-cap-tool-registration-gating"></a> **Tool registration and gating.** Pi MUST expose `pi.registerTool` (registration sink) and the `pi.getActiveTools` / `pi.setActiveTools` snapshot/restore pair for prompt-mode visibility control. Consumed by **Tool-registration lifetime and visibility** above. The lack of `pi.unregisterTool` and the resulting cosmetic registration-cache accumulation are documented in the same section.

5. <a id="sdk-cap-cancellation-propagation"></a> **Cancellation propagation.** Pi MUST supply an `AbortSignal` at the two extension entry points loom uses — `ctx.signal` on the slash-command `ExtensionCommandContext` and the `signal` parameter of `tool.execute(...)`. For subagent mode (which has no `AbortSignal` entry point — `CreateAgentSessionOptions` has no `signal` field in the V1 Pi SDK pin), Pi MUST expose `AgentSession.abort(): Promise<void>` on the handle returned by `createAgentSession(...)` so the runtime's one-shot `loomAbort.signal` listener can tear down the spawned session's in-flight provider call. The fixed-shape WHATWG `AbortSignal` / `AbortController` member surface is enumerated in [Step 0 (b)](#entry-capability-probe) above; `AgentSession.prototype.abort` is enumerated alongside `createAgentSession` under [Step 0 (c)](#entry-capability-probe)'s factory-probable members so a Pi version that drops or renames it surfaces as `loom/load/host-incompatible` rather than as a runtime-time `TypeError` at the spawn site. The loom-side `AbortController` wrapping rule and the listener-and-`abort()` mechanism are in **Cancellation source** and **Subagent session lifecycle** above and in [Cancellation](./cancellation.md). This rule and **Host prerequisites #4 — Pi-supplied `AbortSignal`** above are the same obligation viewed from the inventory and the host-prerequisite axes; the inventory entry is the cross-reference target for `spec.md` and consumers, the **Host prerequisites #4** paragraph is the in-page neighbour of the other host preconditions.

6. <a id="sdk-cap-custom-message-renderer"></a> **Custom-message channel and renderer.** Pi MUST expose `pi.sendMessage({ customType, content, display, details }, { triggerTurn })` for routing system notes through the custom-message channel and `pi.registerMessageRenderer(customType, renderer)` for rendering them in the transcript. The signature, lifecycle, and ownership rules are in **System notes** and **Renderer registration (`pi.registerMessageRenderer`)** above.

7. <a id="sdk-cap-binder-llm-model"></a> **Binder LLM model.** Non-bypass looms MUST be able to resolve a structured-output-capable LLM via `ctx.modelRegistry`. The runtime applies the **bypass criterion** at load time (no `params:`, or single string-typed parameter with no default) per [Binder — Bypass cases](./binder.md#bypass-cases); bypassable looms skip the binder call entirely, and non-bypass looms fail to load with `loom/load/binder-model-unresolved` if no structured-output-capable model is available. Strict-capability degradation, the same `loom/load/binder-model-unresolved` code on V1 strict-capability mismatch, and the `loom/load/binder-model-strict-capability-unknown` warning under the `~0.72.1` pin are owned by [Binder — Strict-capability requirement](./binder.md#strict-capability-requirement); this rule defers the diagnostic-code semantics to that page.

<a id="sdk-cap-peer-dep-revalidation"></a>
**Re-validation on `peerDependencies` widening.** Widening `peerDependencies` requires re-validating the seven obligations above against the new Pi minor before the range moves; the re-validation artefact is the build-time SDK surface-inventory assertion passing against the new package, and the procedure that gates the bump is [Pi version bump procedure](#pi-version-bump-procedure) below.

## Pi version bump procedure

*This subsection is a contributor checklist; the `plan_topics/` links below identify the plan leaves that own the cited build-time assertions.*

This subsection enumerates the contributor checklist that any change widening, narrowing, or otherwise moving the `@mariozechner/pi-coding-agent ~0.72.1` peer-dep pin (and the lock-step `pi-agent-core` / `pi-ai` / `pi-tui` minors) MUST follow. The three “requires re-validating” sentences elsewhere in the spec ([SDK capability inventory — Re-validation on `peerDependencies` widening](#sdk-cap-peer-dep-revalidation), the **Host prerequisites — Pi SDK pin** paragraph above on this page, and the strict-capability sentence in [Slash-Command Argument Binding — Strict-capability requirement](./binder.md#strict-capability-requirement)) all resolve to the steps below; the H1 tests cited at each step (per [`h1-scaffold.md`](../plan_topics/h1-scaffold.md)) are the mechanical gates that fail loudly until the corresponding manual step has been completed.

1. **Re-typecheck against the new package.** Install the candidate `@mariozechner/pi-coding-agent` minor (and its lock-step siblings), run `npm run typecheck`, and confirm no Pi-imported types under the loom-load-bearing surface have changed shape in a way that breaks the runtime. Type errors here mean the candidate minor is not a drop-in replacement and the surface descriptions on this page MUST be updated before the bump can proceed.
2. **Re-run the H1 SDK surface-inventory test.** The literal-read assertion (per [`h1-scaffold.md`](../plan_topics/h1-scaffold.md)) imports `@mariozechner/pi-coding-agent` at the candidate version and confirms each of the seven named SDK capabilities enumerated in [SDK capability inventory](#sdk-capability-inventory) above is present on the imported namespace. A red here means a capability has been removed or renamed; resolve by updating both the inventory constant and the corresponding capability obligation in [SDK capability inventory](#sdk-capability-inventory) (and the matching probe-side description and the `spec.md` cross-reference list) in the same edit before the bump.
3. **Re-confirm the `engines.node` floor.** Compare `@mariozechner/pi-coding-agent`'s `engines.node` floor at the candidate version against the loom `package.json#engines.node` literal pinned by the H1 `engines.node` literal-read test (per [`h1-scaffold.md`](../plan_topics/h1-scaffold.md)). If the upstream floor has moved, update the loom literal, [Step 0 (a)](#entry-capability-probe) above (the comparator and floor reference), the informative-orientation sentence in [`spec.md` — Host runtime obligation 1](../spec.md#orientation), and the H1 assertion in the same edit; if it has not moved, the H1 test stays green and no edit is needed.
4. **Update the version pin in `peerDependencies` and the equivalent literal here.** Move all four `peerDependencies` entries together (the H1 `peerDependencies` literal-read test enforces the joint move) and update the `~0.72.1` literal at the top of this page in the same commit. Splitting these into separate commits leaves a window in which the H1 test is red against `main`; do not split. The deliberate deviation from Pi's bundled-package convention recorded in **Host prerequisites — Pi SDK pin** above (the *Deliberate deviation from Pi's bundled-package convention* sub-paragraph) MUST be re-justified at this step if the candidate Pi minor changes the lock-step expectation `packages.md` prescribes — for example, if `packages.md` at the candidate minor stops naming the `@mariozechner/*` four as `"*"`-range packages, or if Pi changes the `pi-mono` release model so that the four packages are no longer guaranteed to ship together at a single minor. In either case, update the *Deliberate deviation* sub-paragraph (and the H1 `peerDependencies` literal-read test, the [Step 0 (d) Peer-dep version](#entry-capability-probe) probe, and the literal-anchor list above as required) in the same commit; if `packages.md` and the `pi-mono` release model are unchanged at the candidate minor, the deviation rationale stands as written and no edit to that sub-paragraph is required.
5. **Update the capability-probe pinned constants.** The probe described under **Extension entry point — Step 0 (Capability probe)** above consumes four pinned constants (Node floor, `AbortSignal` member list, capability list, peer-dep range) from a single source of truth in the extension module. If steps 1–4 have changed any of those four, update the constant in the same commit; the H1 SDK surface-inventory test imports that constant and will fail until the change is consistent with the rest of the bump.
6. **Re-validate the provider seed-field table.** Run the V16h provider seed-field fixtures against the candidate `@mariozechner/pi-ai` minor. If any supported `Api` value's seed field has been renamed, retyped, or moved between the supporting and non-supporting sets, update the table in [Provider seed-field mapping](#provider-seed-field-mapping) and the V16h fixture inputs in the same commit. If the candidate minor introduces a new `Api` value not yet listed, add a row (with `omitted` as the conservative default unless pi-ai documents a seed field for that provider) and a corresponding fixture in the same commit. The H1 `Api`-coverage assertion (per [`h1-scaffold.md`](../plan_topics/h1-scaffold.md)) is the mechanical gate that fails until this step has been completed. When adding a row, the contributor MUST consult pi-ai release notes and the new provider's request-adapter source to decide between a named field and `omitted` — do not silently regress a provider for which pi-ai has added seed support.
7. **Update the binder strict-capability probe.** The binder strict-capability check (per [Slash-Command Argument Binding — Strict-capability requirement](./binder.md#strict-capability-requirement)) reads `Model<Api>.strictCapable` by name. If the candidate `@mariozechner/pi-coding-agent` minor introduces the indicator under a different name, rename the probe constant in the same edit; the H1 SDK surface-inventory assertion that catches "field present on `Model<Api>` namespace but not under the probed name" is the mechanical gate that fails until this step has been completed. If pi-ai exposes the field under the probed name `strictCapable`, this step is a no-op.

The procedure produces no separate artefact file — its outputs are (a) a green H1 test run that proves the SDK surface, the `peerDependencies` literals, and the `engines.node` floor are consistent at the new pin, and (b) a single bump commit that touches the loom `package.json`, the constant block in the extension module, the four spec sentences cited above, and any inventory bullet whose underlying SDK shape changed. A bump commit that updates `peerDependencies` without a matching update to the constant block, the spec sentences, or the inventory bullets fails the H1 tests and MUST be amended before merge.
