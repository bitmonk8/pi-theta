# Glossary

This page collects the spec's coined vocabulary in one alphabetised list. Each entry is descriptive — a one-paragraph reminder of what a term means and where it is used. The canonical, normative definition lives on the feature page named in the entry's `See:` reference; if a glossary entry and its canonical page disagree, the canonical page wins. Add new entries here when the spec coins a new term that is reused on more than one page.

---

- **binder** — The LLM-driven argument-binding mechanism that maps a slash-command's free-form argument string onto a loom's typed `params:` envelope. Suppressed entirely in either of the two static *binder bypass* conditions (see *no-params bypass* / *single-string bypass*, below). See: [Slash-Command Argument Binding](./binder.md).

- **callable set** — The unified set of Pi tools and registered `.loom` paths declared by a loom's frontmatter `tools:` field. Both kinds of entry are callable from the model (during a query's tool-call loop) and from loom code (via the bare `<name>(...)` form). The Pi session's ambient tools are deliberately *not* inherited; if `tools:` is omitted the loom runs with an empty callable set. See: [Parameters and Frontmatter](./frontmatter.md), [Tool Calls](./tool-calls.md).

- **coercion (schema-validation)** — The typed-query repair loop that appends a follow-up *user* turn to the same conversation when AJV rejects the model's final response, asking the model to fix its output. Configured by the `coercion:` frontmatter block; bounded by `coercion.attempts` (default 3); exhaustion surfaces as `Err(QueryError { kind: "validation", ... })`. Distinct from the next entry — this term is about response repair, not value conversion. See: [Query](./query.md).

- **coercion (type, expression-level)** — Implicit conversion between value types in expressions (e.g. `number → string` for concatenation, truthy/falsy `value → boolean`). Loom does **not** perform any in V1; mismatches are parse or runtime errors. Listed here only to make the absence explicit, since the term is also used in the schema-validation sense above.

- **`InvokeInfraError`** / **`kind: "invoke_failure"`** — The infra-side invoke failure variant of `QueryError`. The schema name carries an `Infra` qualifier to partition it from `InvokeCalleeError` (the callee's own `Err` propagated through), but the wire `kind` discriminant remains `"invoke_failure"`: snake_case discriminants are stable wire contract and are not renamed when their loom-side schema name changes. This is the only V1 `QueryError` variant whose schema and wire names diverge other than by an explicit `field as "..."` rename. See: [Errors and Results — Invoke variants](./errors-and-results.md), [Invocation — Failures](./invocation.md).

- **loom (file unit)** — A single `.loom` file viewed as a named, invocable unit: the artefact a slash command resolves to, the thing `invoke(...)` targets, the thing whose frontmatter declares `mode:`. Used with an article ("a loom", "the loom"). Lowercase. See: [Overview — Scope of a loom file](./overview.md), [Invocation](./invocation.md).

- **Loom (language)** — The scripting language whose grammar and semantics this spec defines, shared by `.loom` and `.warp` files. Capitalised when used as a proper noun ("Loom expressions", "Loom code", "Loom follows Rust's…"); the lowercase form is reserved for sense 1 above. See: [Lexical Structure](./lexical.md), [Expression Sublanguage](./expressions.md), [Influences](./influences.md).

- **loom-side name** vs. **wire name** — A pair. The *loom-side name* is the identifier a schema field is declared and referenced by in loom code; it is the only name visible from the language. The *wire name* is the JSON property name used in lowered JSON Schema, in tool input, and in AJV validation, optionally renamed via `field as "WireName": T`. The runtime translates between the two at the validation boundary in both directions; loom code never references the wire name directly. See: [Schema Declarations](./schemas.md); operational detail in [Runtime Value Model](./runtime-value-model.md) and [Schema Subset](./schema-subset.md).

- **no-params bypass** vs. **single-string bypass** — A pair. Both are static binder-bypass conditions decided at loom-load time from the declared `params:` shape; both suppress the binder call entirely and construct no envelope schema. The *no-params bypass* triggers when `params:` is absent or `{}` — the loom takes no parameters and the binder does not run; `bind_echo: true` on such a loom is the load warning `loom/load/bind-echo-without-params` and produces no echo. The *single-string bypass* triggers when `params:` declares exactly one field, that field's type is `string`, and the field has no default — the runtime sets the param's value to the entire slash-argument string (whitespace-trimmed) and skips the binder; AJV validation still runs as a safety net. The no-params check runs before the single-string check, so `params: {}` does not accidentally match the single-string branch. Declaring `bind_echo: true` on a bypass-eligible loom is the parse warning `loom/parse/bind-echo-on-bypass`. See: [Slash-Command Argument Binding — Binder bypass](./binder.md#binder-bypass).

- **pi-loom (extension)** — The Pi extension that registers `.loom` files as slash commands, owns the discovery walk, and runs loom code. Referred to in prose as `pi-loom` or "the loom extension"; never capitalised "Loom Extension". See: [Pi Extension Integration](./pi-integration.md), [Pi Integration Contract](./pi-integration-contract.md).

- **prompt mode** / **subagent mode** — The two execution modes a `.loom` runs under, selected by frontmatter `mode:`. In *prompt mode* the loom drives the user's existing Pi session — its queries become user turns in that session and its emissions are visible to the user. In *subagent mode* the loom spawns a fresh, isolated `AgentSession` whose transcript is private to the loom and discarded when the loom returns. The two modes differ in driver mechanics, in what frontmatter fields are accepted (e.g. `system:` is subagent-only), and in which Pi APIs the runtime calls. See: [Overview and Conceptual Model](./overview.md), [Parameters and Frontmatter](./frontmatter.md), [Pi Integration Contract](./pi-integration-contract.md).

- **Pi tool** vs. **`.loom` callable** — The two kinds of entry that make up a callable set. A *Pi tool* is a tool registered against the Pi runtime (built-in or extension-supplied), referenced in `tools:` by name. A *`.loom` callable* is a path entry in `tools:` pointing at a subagent-mode loom file, which the runtime wraps as a tool. The two share the bare-name call form `<name>(args)` and differ only in resolution path and argument shape (Pi tools take a single object matching their input schema; loom callees take their `params:` positionally). See: [Parameters and Frontmatter](./frontmatter.md), [Tool Calls](./tool-calls.md).

- **top-level loom** — The outermost block of a `.loom` file. Acts as the implicit function body for the file: its tail expression is the loom's return value, `return expr` exits it, and `?` early-returns from it. Distinct from any nested function bodies declared inside the file. See: [Functions](./functions.md), [Return Statement](./return.md).

- **type sink** — An enclosing AST position whose declared type can supply the schema for a query expression during the outward type-inference walk. Examples include the RHS of a `let x: T = ...`, an argument matched to a typed parameter, and the operand of a `return` from a function with a declared return type. The walk crosses transparent constructs (parenthesisation, ternary branches when the ternary itself has a sink, etc.) and stops at any expression that consumes its operand without preserving its type; if no sink is found, the query is untyped and returns `string`. See: [Query](./query.md).
