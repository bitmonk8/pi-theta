# Lexical Structure

**Source files.** `.loom` and `.warp` files are decoded and normalised before lexing; every other rule on this page (and every other spec rule mentioning "newline") operates on the post-normalisation stream.

- **Encoding.** Source files are UTF-8. A leading UTF-8 BOM (`EF BB BF`) is consumed and ignored. Any other BOM, or any byte sequence that is not valid UTF-8 (including lone surrogates), is a load-time error `loom/load/invalid-encoding` reporting the file path and the byte offset of the first invalid byte. The runtime performs no transcoding from other encodings (Latin-1, UTF-16, etc.); a Notepad-style UTF-16 save fails fast rather than producing mojibake.
- **Newline normalisation.** Before lexing, line endings are normalised: `\r\n` → `\n` and bare `\r` → `\n`. Every spec rule that mentions "newline" — statement separation below, the "literal newline in regular string is a parse error" rule below, `@`...`` query newline-trim and dedent (see [Query](./query.md)), and `///` doc-comment joining (see [Descriptions](./descriptions.md)) — operates on the normalised stream. A `.loom` checked in with CRLF line endings therefore tokenises and dedents byte-identically to the same file with LF endings; this is observable behaviour, not an implementation detail.
- **Path literals.** Every path literal in source — `import "..."`, `invoke("...", ...)`, and `.loom` entries inside `tools:` (see [Imports](./imports.md), [Invocation](./invocation.md), [Parameters and Frontmatter](./frontmatter.md)) — uses forward-slash separators only. A backslash inside a path literal is a parse error `loom/parse/invalid-path-separator`. (Path literals are otherwise ordinary string literals; `\\` already lowers to a single backslash via the standard escape table below, so the parse error is unambiguous — there is no "did the author mean an escape" interpretation.) Discovery roots and the `--loom <path>` CLI flag accept OS-native paths with no syntactic restriction; their resolved targets are compared in normalised forward-slash form for the discovery collision rule (see [Directory Convention](./discovery.md)).
- **Diagnostic spans.** BOM consumption and newline normalisation both happen *before* span recording, so diagnostic line/column numbers refer to logical lines on the normalised stream — column 1 of line 1 starts after the BOM, and a CRLF source line counts as one newline, not two.

**Identifiers.** `[A-Za-z_][A-Za-z0-9_]*`, case-sensitive. The **first letter's case is enforced** by the parser — it is what makes case-based pattern disambiguation in `match` work without additional grammar:

- **PascalCase** (uppercase first letter) is required for: `schema` names, `enum` names, `enum` variant names, and any user identifier introduced as a type-like binding. The built-in `Ok`, `Err`, and `Result` follow the same rule.
- **lowercase-first** (a lowercase letter, or `_`) is required for: `let` and `let mut` bindings, function parameters, function names, and schema field names. Both `snake_case` (`experience_years`) and `lowerCamelCase` (`experienceYears`) are accepted; the parser only cares about the first letter. The lowercase-first rule applies to the **loom-side** field identifier; the field's *wire* name (what appears in JSON sent to and received from the model) may be any string via the `as "WireName"` rename clause described in [Schema Declarations](./schemas.md).

Violating either rule is a parse error ("schema name must start with an uppercase letter"; "binding name must start with a lowercase letter or `_`"). Inside `match` patterns the same first-letter rule then disambiguates without ambiguity: a lowercase identifier introduces a fresh binding, an uppercase identifier refers to an existing schema, enum, or constructor in scope (see [Errors and Results](./errors-and-results.md)). The casing rule is the *only* enforced naming constraint; identifier length, internal casing, and underscore use are otherwise free.

**Reserved keywords.** Cannot be used as identifiers: `let`, `mut`, `fn`, `if`, `else`, `for`, `in`, `while`, `break`, `continue`, `return`, `match`, `schema`, `enum`, `import`, `export`, `from`, `as`, `by`, `invoke`, `true`, `false`, `null`, `Ok`, `Err`, `Result`, `string`, `number`, `integer`, `boolean`, `array`, `void`. The discard binding `_` is also reserved (it is not a regular identifier and cannot be referenced after binding).

**Statement terminators.** Statements are separated by newlines; semicolons are not part of the grammar. A statement implicitly continues across newlines only when the parser cannot otherwise close it — open `(` / `{` / `[`, a trailing binary or unary operator, or a trailing comma. The same rule means single-line `if (x) stmt` does not exist; bodies of `if` / `for` / `while` / `fn` are always braced blocks.

**Comments.** Line comments only. `//` is a regular comment; `///` is a doc comment that lowers to a JSON Schema `description:` (see [Descriptions](./descriptions.md)). Block comments (`/* ... */`) are not supported. Comments inside the *text* of a `@`...`` query template are not comments — they are part of the rendered prompt. Comments inside a `${...}` interpolation behave exactly as in any other expression position.

**String literals.** Single- (`'...'`) and double-quoted (`"..."`) forms are equivalent. Escape sequences: `\"`, `\'`, `\\`, `\n`, `\t`, `\r`, `\u{XXXX}` (Unicode code-point, 1–6 hex digits). A backslash followed by any other character is a parse error. **Single-line only** — a literal newline inside a regular string is a parse error. **No interpolation** — the sequence `${` inside a regular string is plain text. Multi-line text and interpolation belong inside `@`...`` query templates; for non-query multi-line text, build via `+` concatenation with `\n` escapes, or factor the text into a query.

**Number literals.** Decimal only: `42`, `3.14`, `1e10`, `1.5e-3`, `0`, `0.5`. The literal carries no sign — negation is the unary `-` operator applied at parse time. No hex / octal / binary forms and no underscore separators in V1. See [Future Considerations](./future-considerations.md). A literal with no fractional or exponent part has type `integer`; otherwise `number`. `integer` widens implicitly to `number` in arithmetic and assignment positions; the reverse is a parse error.
