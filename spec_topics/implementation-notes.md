# Implementation Notes

This page collects the parser, runtime, and orientation notes. Two more focused implementation pages are split out:

- [Runtime Value Model](./runtime-value-model.md) — how Loom values are represented at runtime, equality semantics, wire-name translation.
- [Pi Integration Contract](./pi-integration-contract.md) — the named SDK surface from `@mariozechner/pi-coding-agent` the runtime depends on.

## Parser

- **Contract.** The parser MUST emit diagnostics matching [Diagnostics](./diagnostics.md), including multi-error reporting across the whole `.loom` or `.warp` file and its transitively imported `.warp` modules before failing. Parse errors surface as structured diagnostics compatible with Pi's error reporting.
- **Implementation hint (non-normative).** The reference implementation uses Chevrotain (TypeScript-native, no separate lexer generator required), with the grammar derived from the Chevrotain TinyC example and extended with `@`...`` query templates, `schema`, and the type-context schema-inference pass. The choice of toolkit and grammar lineage have no observable effect on a loom author and may be replaced by any implementation that satisfies the contract above.

## Runtime

- Implemented in TypeScript as a Pi extension module
- **Static-resolution load pass.** Each top-level loom load runs a single per-pass parse cache shared by every parse-time check that needs a callee's source: `invoke(...)` argument-type checking, cross-loom return-type inference at `invoke<T>(...)` and `.loom` callable call sites, and the parse-time invocation-cycle walker (per [Invocation — Static resolution](./invocation.md#static-resolution)). The pass walks transitively from the entry loom across literal `invoke` paths and `.loom` entries in `tools:`, parses and lowers each visited file once, and aggregates each visited file's diagnostics into the entry loom's drain (sorted by `(file, line, col)` per [Diagnostics — Multi-error reporting](./diagnostics.md)). When a transitively-reached callee fails to parse or lower, the parent emits `loom/load/callee-has-errors` at the referencing site — severity `error` for `tools:` entries (callable cannot be created; parent does not register), severity `warning` for `invoke(...)` literals (parent registers; static checks against the callee are skipped; runtime AJV is the safety net). The in-process re-parse path (per [Pi Integration Contract — Extension entry point](./pi-integration-contract.md)) drops the per-pass cache entry for the changed file and every transitive `.warp` importer; the validator-cache key (see the **Schema validation** bullet below) remains the schema slug, so a callee re-parse that produces an identical lowered schema reuses cached validators across reload. The drop happens as part of the `LoomRegistry` swap, not as a separate watcher-driven path
- Holds a reference to the target conversation (the caller's session for prompt mode; a freshly spawned isolated session for subagent mode) and drives it turn-by-turn
- Each `@`...`` query issues a user turn against that conversation, services any tool-call loop the model returns, and resolves to the final assistant response
- Typed queries (schema inferred from binding type, function parameter, return type, or explicit `@<Schema>`...``) lower the schema to a provider-shaped strict tool-input schema; the returned tool-call payload is validated with **AJV** before being handed back to loom code. The behavioural contract for typed queries — the two-phase loop, the round cap, the provider compatibility surface — is normative and lives in [Pi Integration Contract — Conversation drive (prompt mode)](./pi-integration-contract.md) and [Query — Typed queries are tool-loop-shaped](./query.md). The V1 reference implementation:
    - Synthesises a one-shot tool named `__loom_respond_<slug>` (where `<slug>` is the schema slug of the lowered query response schema, produced by the recipe in [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash)). The tool's `parameters` is the lowered query schema wrapped via `Type.Unsafe<unknown>(loweredJsonSchema)` (per the bridging convention in [Pi Integration Contract — Per-loom registration](./pi-integration-contract.md)); its `label` is the literal string `"Loom typed-query response"`; its `execute` AJV-checks the `unknown`-typed `params` against the original lowered JSON Schema and resolves the query's promise with the validated value.
    - Registers the respond tool alongside the loom's callable set for the duration of the typed query. Registration lifetime and visibility are governed by [Pi Integration Contract — Tool-registration lifetime and visibility](./pi-integration-contract.md): in **subagent mode** the respond tool is passed alongside frontmatter tools to `createAgentSession({ customTools, tools, ... })` and lives only inside the spawned `AgentSession`; in **prompt mode** the respond tool is registered once per unique `<slug>` via the registration cache (subsequent typed queries with the same lowered schema reuse the same registration) and made visible for the typed query via the `pi.setActiveTools` snapshot/restore protocol, with `[...frontmatter callable set, respondToolName]` as the active set for the duration of the query. Pi has no `unregisterTool` API; the spec deliberately does not call one.
    - Runs the two-phase loop: the runtime drives the conversation through `pi.sendUserMessage` (prompt mode) or `session.sendUserMessage` (subagent mode) with **no** `tool_choice` constraint; on each turn, if the model emits `tool_use` blocks the runtime services them (frontmatter tool or respond tool, dispatched by name) and loops; if the model emits a plain text turn, the runtime issues one follow-up user turn inlining the lowered response schema and forces tool choice to the respond tool *for that turn only*. Tool-choice forcing for the respond turn is set via pi-ai's `options.toolChoice` abstraction (`{ type: "tool", name }`), which the per-provider pi-ai adapter maps to the provider-native payload (Anthropic `tool_choice: { type: "tool", name }`; OpenAI Completions `tool_choice: { type: "function", function: { name } }`; Mistral and Bedrock map analogously). The runtime does **not** install a `before_provider_request` payload-rewrite hook in V1 — a global hook that rewrites every provider call in every session is the mechanism Option B of the resolved typed-query design rejected; the per-call `options.toolChoice` route is scoped to exactly the forced respond turn and avoids the cross-session-fingerprinting machinery a global hook would require.
    - Enforces the `tool_loop.max_rounds` cap (per [Parameters and Frontmatter — `tool_loop`](./frontmatter.md)) by counting tool-call rounds in the free phase plus the one forced respond turn; on exhaustion, returns `Err(QueryError { kind: "tool_loop_exhausted", rounds: <cap>, last_tool_name: <name | null>, raw_response: <text | null>, message: "tool-call loop exhausted" })`. Respond-repair follow-ups (per [Query — Schema-validation respond-repair](./query.md)) restart the whole two-phase loop with a fresh `tool_loop` budget.
    - Is observably equivalent to a native structured-output turn at the user-facing layer: the user sees free-phase tool-call cards (when the model uses frontmatter tools) followed by a single tool-call card for the forced respond turn carrying the validated payload, matching the behavioural contract's "no extra user-visible turns beyond the natural free-phase turns and the single forced respond turn" clause
- Tool calls from loom code (`<name>(args)`) dispatch through the per-loom callable table that was materialised at load time; resolution is by held reference, not by re-querying Pi's tool registry on every call (see [Parameters and Frontmatter — Resolution snapshot](./frontmatter.md)). For Pi tools the runtime invokes the captured `ToolDefinition`'s `execute(toolCallId, params, signal, onUpdate, ctx)` directly with a synthesized `toolCallId` and a no-op `onUpdate`, and awaits the resulting Promise before returning to loom code (non-blocking at the runtime level, sequential at the language level); for `.loom` callables the runtime spawns a subagent invocation equivalent to `invoke<T>(path, ...)`. Pi tools registered in `tools:` are also wired into every `@`...`` query as model-callable tools; `.loom` callables are lowered to a tool spec (params → input schema, inferred return → output schema, frontmatter `description` → tool description) and exposed alongside Pi tools to the model
- For subagent-mode looms, the spawned conversation's system prompt is taken from frontmatter `system:` (with `${param}` interpolation resolved at conversation-creation time) and applied to every query against that conversation
- Parameter schemas (frontmatter `params`) are likewise validated with AJV at invocation time
- For slash-command invocation, the runtime first runs the binder (per [Slash-Command Argument Binding](./binder.md)) unless the bypass condition holds. The binder is a one-shot ephemeral call to `bind_model` (resolved at loom-load time from a two-step chain — frontmatter `bind_model:`, then the loom-extension setting `looms.binderModel` per [Settings file reads](./discovery.md#settings-file-reads); no further fallback). The resolved model must support strict structured-output / strict tool-input; both the resolution and the capability check are load-time gates that surface `loom/load/binder-model-unresolved` and `loom/load/binder-model-not-strict-capable` respectively. The binder call itself runs with `temperature: 0` and a fixed seed where supported. The runtime constructs the binder's response schema dynamically from the loom's `params:` schema (the three-arm envelope `ok | needs_info | ambiguous`), passes it as a strict structured-output contract, and validates the returned `args` (on `kind: "ok"`) with AJV before merging in defaults and starting the loom. Binder failures and non-`ok` envelopes are surfaced as one-line system notes in the user's session; the loom is not started
- **Schema validation.** Schema validation is provided by an injected `SchemaValidator` service (see [Pi Integration Contract](./pi-integration-contract.md) and the runtime's DI seams). The validator's behavioural contract — what any conforming implementation must satisfy:
    - Validation MUST report every validation error in a single pass (no fast-fail) so [Diagnostics](./diagnostics.md) can present them all at once.
    - Validation MUST NOT convert values and MUST NOT fill defaults — model output is taken as-is, and loom does no implicit type conversion in V1. The loom-level **respond-repair** mechanism that repairs schema-validation failures via follow-up turns — configured by `respond_repair:` frontmatter, see [Query — Schema-validation respond-repair](./query.md) — is a separate concept and is unaffected.
    - `$ref` MUST resolve only within the lowered per-query schema document (the document produced by step 4 of the lowering algorithm in [Schema Subset — Lowering Algorithm](./schema-subset.md#lowering-algorithm)); external `$ref`s are not supported because lowered schemas never produce them.
    - Validation MUST silently accept any JSON-Schema `format` keyword that may appear in a value's schema. Loom-emitted schemas never use `format`, but model output can carry one; an implementation that raises on unknown formats violates this contract.
    - Validation results MUST be deterministic for a given (schema, value) pair.
  Architectural constraints carried by the DI seam itself: one `SchemaValidator` instance is constructed per runtime instance — never as a module-level global — and lives for the lifetime of that runtime. Two concurrent runtime instances (e.g. parallel tests) each get their own; sharing one across runtimes is disallowed. Cache invalidation on file change is a property of the validator service — the file watcher calls into the service's invalidate path; it does not reach into any module-scoped state.

  <a id="schemavalidator-interface"></a>

  The interface shape is normative; `LoweredSchema` is the lowered JSON-Schema document produced by step 4 of [Schema Subset — Lowering Algorithm](./schema-subset.md#lowering-algorithm), `schemaSlug` is the schema slug produced by the recipe in [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash), and `ValidationError` mirrors AJV's `ErrorObject` shape (the reference implementation hands AJV's array through unchanged):

  ```ts
  interface ValidationError {
    instancePath: string;                           // RFC 6901 JSON Pointer to the failing value
    schemaPath: string;                             // pointer into the schema that triggered the failure
    keyword: string;                                // the JSON-Schema keyword that failed ("type", "required", "enum", …)
    message: string;                                // human-readable failure description
    params: Record<string, unknown>;                // keyword-specific failure context (AJV's `params`)
  }

  interface CompiledValidator {
    validate(value: unknown):
      | { ok: true }
      | { ok: false; errors: readonly ValidationError[] };
  }

  interface SchemaValidator {
    compile(schema: LoweredSchema): CompiledValidator;
    invalidate(schemaSlug: string): void;           // file-watcher entry point per the cache-invalidation rule above
  }
  ```
- **Clock.** Wall-clock time and deferred-work scheduling are provided by an injected `Clock` service (see [Pi Integration Contract — `Clock` / `FakeClock` interface](./pi-integration-contract.md#clock--fakeclock-interface) and the runtime's DI seams). The seam carries `now()` (monotonic milliseconds), `setTimeout`, and `clearTimeout`; production wires a `WallClock` adapter and tests use a `FakeClock` whose `advance(ms)` deterministically fires due timers in deadline + registration order. One `Clock` instance is constructed per runtime instance — never as a module-level global — and lives for the lifetime of that runtime; parallel runtimes get independent clocks. The runtime MUST NOT call `Date.now`, `performance.now`, `Date.prototype.getTime`, or the global `setTimeout` / `clearTimeout` outside the `WallClock` adapter; a build-time grep-test enforces this. Load-bearing call sites: `RuntimeEvent.occurred_at` stamping at the originating emission site (see [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md)), the watcher's 250 ms debounce window (see [Pi Integration Contract — Extension entry point](./pi-integration-contract.md)), the settings-watcher debounce (see [Discovery — Settings file reads](./discovery.md#settings-file-reads)), and the `looms.scanPackagesTimeoutMs` cap on the package-discovery walk (see [Discovery — Package discovery](./discovery.md#package-discovery)).
- **Implementation hint (non-normative).** The reference implementation uses AJV v8 (`ajv ^8`, `ajv-formats ^3`), declared in loom's own `dependencies` rather than provided by Pi, with `strict: false`, `allErrors: true`, and `ajv-formats` registered as an instance-level side effect performed by each `SchemaValidator` on its own `Ajv`. These flag values are how AJV happens to satisfy the contract above: `allErrors: true` for one-pass multi-error reporting, the absence of AJV's `coerceTypes` / `useDefaults` options for no-type-conversion / no-defaults, AJV's default in-document `$ref` resolver for the ref-scope rule, and `ajv-formats` for silent format-keyword acceptance. The reference implementation also caches compiled validators per lowered-schema document, keyed by the schema slug of that document plus the AJV major version; the in-process re-parse path defined in [Pi Integration Contract — Extension entry point](./pi-integration-contract.md) drops cache entries belonging to changed files as part of the `LoomRegistry` swap; cache hit/miss policy beyond that drop is an optimisation and does not affect the deterministic-validation contract above
- **Per-invocation single-threaded execution.** The loom interpreter runs as plain async TypeScript on Pi's event loop. There are no worker threads and no loom-level concurrency primitive in V1. Within a single invocation the interpreter is strictly sequential: every `await` (query, tool call, invoke) yields the event loop and the next loom expression cannot run until the awaited Promise resolves, so an invocation can never re-enter itself. Concurrent invocations are independent: two slash-command invocations of the same loom run sequentially against the user's session in prompt mode, and each gets its own private `AgentSession` in subagent mode — transcripts, captured `tools:` tables, and derived `AbortSignal`s are per-invocation and do not leak across siblings. The schema-validator cache and the file-watcher are shared across invocations but are read-mostly: compiled validators are immutable post-compile and cache writes only occur on initial compile or watcher-driven invalidation, so concurrent reads from sibling invocations are safe without locking. Sibling invocations can interleave on the event loop — this happens whenever the model emits parallel tool calls into the same `.loom` callable (see [Tool Calls — Concurrency](./tool-calls.md)). <a id="no-invocation-cap"></a> V1 imposes no maximum on the number of in-flight loom invocations within an extension instance and no fairness or scheduling rule above Pi's event-loop ordering: the runtime MUST NOT introduce an admission cap, a priority queue, or any scheduler interposed between sibling invocations and the event loop. The rule constrains the runtime only — Pi's own session-level serialisation (e.g. prompt-mode slash-command turns within one user session, owned by [Pi Integration Contract](./pi-integration-contract.md)) is upstream and remains in force, and the rule does not promise resource unboundedness (host-OOM and analogous below-runtime exhaustion still route through `loom/runtime/internal-error` per [Errors and Results](./errors-and-results.md)). Widening or narrowing this rule is a major-version concern
- The loom's overall return value is the value of the last expression of its top-level block
