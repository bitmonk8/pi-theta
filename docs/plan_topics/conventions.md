# Plan structure and conventions

Three kinds of phase:

1. **Horizontal phases.** Project infrastructure — scaffold, dependency-injection skeleton, diagnostics primitive, Pi-extension shell, end-to-end harness, REQ-ID anchor / coverage-matrix automation. Horizontal leaves operationalise project-level conventions (this page) rather than implement a normative spec rule, and cite a **Convention.** field instead of a **Spec.** field.
2. **MVP phase.** The smallest end-to-end `.loom` that runs as a Pi slash command — a single narrow vertical proving the pipeline before slice work begins.
3. **Vertical slices.** Each slice is a coherent feature area (lexer, expressions, schemas, queries, …). Each leaf inside a slice is the smallest feature that can ship and be tested independently. Slices are roughly ordered by their dependency DAG; non-linear deps are stated in each leaf's **Deps** field. Reorder freely as long as the deps DAG is respected.

Horizontal leaves may be presented under an editorial sub-heading (such as a release-gate grouping) without forming a new category: such a sub-grouping creates no new leaf-ID prefix and no new **Convention.**/**Spec.** rule — the grouped leaves remain horizontal leaves in every other respect.

Leaves are the unit of work; slice grouping is editorial. Leaf IDs follow `<group><letter>` (e.g. `V4b`); MVP and vertical features additionally carry a paired **tests-task** leaf whose ID is the feature ID with a `-T` suffix (e.g. `V4b-T`, `M-T`). The IDs `H1`–`Hn`, `M`, and `V1`–`Vn` (with their leaf and `-T` forms) are reserved for plan phases — when plan prose needs to refer to the initial release of the loom language, write "loom 1.0" or "the initial release"; never reuse a plan ID for that meaning.

## Per-phase TDD ritual (mandatory)

TDD is expressed as **two separate tasks** for every MVP and vertical feature; horizontal infrastructure leaves run a single-task ritual.

**"Fail red for the intended reason" (defined).** A tests-task suite *fails red for the intended reason* when both hold:

1. **It compiles and runs.** The test files type-check against the stubbed or declared seams the tests task is permitted to add (the ritual permits "the minimum production code needed for the tests to compile and fail"). A suite that cannot type-check — a TypeScript/compile error, e.g. a reference to a not-yet-declared production symbol — is a **compile-time red** and does **not** satisfy the gate; declare or stub the seam so the suite compiles.
2. **Each test reds out on its own primary assertion because the implementation under test is absent or incomplete** — not on a missing or unconfigured fixture, a harness/setup throw, or an unrelated exception.

A suite red for any non-behavioural reason (compile error, missing/unconfigured fixture, harness/setup throw, unrelated exception) does **not** satisfy the gate and must be fixed before tagging `<id>-T-complete`. The companion impl-side phrasing "fail for the intended reason" carries the same meaning.

### MVP and vertical features — tests-task → implementation-task pairing

Each feature is an ordered pair of leaves, each landing as its own commit:

1. **Tests task** — leaf ID `<id>-T` (e.g. `V4b-T`). Write the failing tests for *every* spec REQ-ID the feature introduces. One assertion per rule where practical. A test that would pass when prerequisites are missing is a defect — fix it before the implementation task. The tests task adds no production code beyond the minimum needed for the tests to compile and *fail red for the intended reason* (defined above). **Ships when** the tests exist, compile, and *fail red for the intended reason* as defined above; tag the commit `<id>-T-complete`.
2. **Implementation task** — leaf ID `<id>` (e.g. `V4b`). Lists `<id>-T` in its **Deps.** Then run the loop:
   1. **Implement.** Write the smallest correct increment that turns the red tests green. Correctness is the goal; tests are the evidence. Do not under-implement to game a thin test, and do not add speculative APIs (unused exports, public hooks no test in this leaf or its declared downstream consumers exercises).
   2. **Run.** All `<id>-T` tests green; type-check clean; lint clean.
   3. **Self-review.** Re-read the spec section, the diff, the test list. Check: any rule unverified? any silent skip? any broad `catch` that should be a specific type? any global / static / singleton — including the closure-captured, lazy module-cache, and DI-container forms the `src/**` architectural test does not detect — creeping in? any indirect ambient-primitive access the `H3a` identifier-keyed scan does not witness — an aliased read (`const env = process.env`), a destructured read (`const { cwd } = process`), computed access (`process["env"]`), a helper wrapper, or re-export indirection — reaching a banned primitive outside its seam adapter? any *Doc updates* artifact missing for this leaf — the `README.md` status row, the dated `CHANGELOG.md` line, and (for non-plan discoveries) the `notes.md` entry?
   4. **Fix review issues.** Iterate from step 2 until the review is clean.
   5. **Phase exit gate.** "Ships when" criterion observable; tag commit `<id>-complete`.

A feature is **not** complete until the implementation task's exit gate is met. No "we'll fix it next slice" carry-overs.

### Horizontal leaves — single task

Infrastructure leaves are **not** test-paired (no `-T` leaf): they carry inline tests where they hold logic (architectural tests, lint-rule wiring, harness self-checks). A horizontal leaf runs the **Implement → Run → Self-review → Fix → exit gate** loop above in one task and ships when its "Ships when" condition is observable; tag the commit `<id>-complete`.

## Leaf format

Each leaf has the same fields, in the same order. The blank skeleton lives in [`leaf-template.md`](./leaf-template.md).

- **Spec.** Page(s) under [`../spec_topics/`](../spec_topics/) the leaf implements (an implementer MAY restrict their reading to these pages), *or* **Convention.** Section(s) of this page the leaf operationalises (used by horizontal phases, which derive from project-level conventions rather than spec rules). The **Spec** field MUST be closed under normative cross-link: closure runs from each listed topic outbound, applies transitively to a fixed point, and excludes narrative cross-links per [`governance.md` GOV-3](../spec_topics/governance.md).
- **Adds.** One sentence — what the leaf introduces. `Adds.` is **descriptive** by default: the binding obligations of a leaf are its **Tests.** REQ-ID bullets (together with the `conventions.md` cross-cutting rules they inherit) and its **Ships when** gate — `Adds.` prose does not, on its own, bind the implementer. A mechanism named in `Adds.` binds only when it is either (i) a **Class-1** observable behaviour traced to a normative spec obligation — a cited `PREFIX-N` REQ-ID, a cited `loom/...` diagnostic-registry code, or a named normative step on a code-keyed obligation page per [`coverage-matrix.md`](./coverage-matrix.md) "Code-keyed obligation areas (no numbered REQ-IDs)" (e.g. the concurrency construct the *Sequential by default* rule requires `Adds.` to name alongside its mandating REQ-ID or code-keyed obligation area), or (ii) a **Class-2** named cross-leaf **seam** that some other leaf binds against — i.e. a leaf listing this leaf in its `Deps.` names the seam in its own `Adds.` or `Tests.`. A seam named in `Adds.` with no such consumer remains illustrative (non-binding). Any other mechanism named in `Adds.` is a **Class-3** architecture or shape mandate — neither a Class-1 observable behaviour (REQ-ID- or code-keyed) nor a Class-2 consumer-bound seam — and is illustrative: it describes the intended shape, but `Adds.` prose alone does not authorise it.
- **Tests.** Bullet list — one bullet per **REQ-ID** the leaf claims to implement; cite the ID inline (e.g. `BNDR-7: ...`). Where a leaf implements only part of a rule, each Tests bullet still cites its REQ-ID; the coverage matrix's REQ-ID-to-leaf mapping is many-to-many. The REQ-ID prefix table lives in [`../spec_topics/governance.md`](../spec_topics/governance.md). Pure-narrative pages — those whose row in the prefix table carries the `(no IDs — narrative)` cell per [`governance.md` GOV-3](../spec_topics/governance.md) — carry no IDs and need no leaf citation.
- **Deps.** Other leaf IDs that must be complete first. Listed `-` if none. Cite specific leaf IDs (`V4b`, `V9a–V9e`); never a bare group token (`V4`) — group IDs are editorial only and a bare `VN` in `Deps.` is ambiguous between "every leaf" and "some subset". Use ranges where contiguous and comma-separated lists where not. An implementation task MUST list its paired `<id>-T` tests task here. A new leaf that can introduce an executable REQ-ID, a citing test that *closes* a coverage-matrix-mapped numbered REQ-ID, or an un-anchored normative MUST is additionally added to [`H5b`](./H5b-warn-only-canary.md)'s `Deps.` per the *Transitive-completeness plan-maintenance* clause in §REQ-ID discipline below, which owns the exact triggers.

  In the **coverage matrix**, each spec REQ-ID maps to the **implementation** leaf (`<id>`) that closes it (its green tests are the closure evidence); the paired `<id>-T` tests task is not listed separately.
- **Ships when.** A concrete, externally observable change.

## Cross-cutting rules (every leaf)

- **No globals, statics, singletons.** All collaborators passed by constructor. An architectural test in [`H2a`](./H2a-cross-cutting-gates.md) enforces this against `src/**` for the module-level binding forms it can detect. The same discipline bans direct ambient-primitive access: no `src/**` module may directly reference `process.env`, `process.cwd`, `crypto.randomUUID`, `Date.now`, or `setTimeout`; the only legitimate references live inside the seam-adapter modules that own each primitive. A direct reference is an **exempt ambient site** only if the same line carries a `// allow-ambient: <primitive> — <seam>` comment and the site is enumerated in the ambient-access allow-list the scan consults — the same same-line-comment-plus-allow-list mechanism the *Specific exception types only* and *Sequential by default* rules establish. The identifier-keyed architectural test declared in [`H3a`](./H3a-di-seam-skeleton.md) flags any direct reference to one of these primitives that is **not** at an allow-listed site. The allow-list is append-only and starts empty: the seam adapters are authored by the `V8*` leaves, so each adapter site is registered as its leaf lands and no entry exists at `H3a` time — registering a new exempt site requires no `H3a` edit. The manual residue these two scans concede — the closure-captured, lazy module-cache, and DI-container singletons the `src/**` architectural test does not detect, and the aliased, destructured, computed (`process["env"]`), helper-wrapper, and re-export-indirection ambient reads the identifier-keyed scan does not witness — is owned by the *Per-phase TDD ritual* self-review step above (and the loom 1.0 release-time reviewer inspection where one runs), mirroring the way the [`governance.md` GOV-22](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-22) residue routes to the [GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) reviewer-inspection step; the mechanical scans and that named review step partition the space with no third gap between them.
- **Specific exception types only.** No `catch (e)`, `catch (e: unknown)`, `catch (e: any)`, or `catch (e: Error)` — bind to a specific subtype or let the exception propagate. The rethrow-on-mismatch pattern (`catch (e) { if (!(e instanceof X)) throw e; … }`) is also forbidden. An ESLint rule (`no-broad-catch`) wired in [`H2a`](./H2a-cross-cutting-gates.md) enforces this. A `catch (e: unknown)` clause is permitted only at an **exempt broad-catch site**, and only if the same line carries a `// allow-broad-catch: <REQ-ID-or-code-keyed-area-or-registry-code-or-pi-sdk-boundary> — <spec-page>` comment and the call site is enumerated in the lint allow-list; the loom 1.0 closing gate asserts every allow-list entry's cited token resolves to one of: a REQ-ID present in [`coverage-matrix.md`](./coverage-matrix.md), an enumerated *Code-keyed obligation areas (no numbered REQ-IDs)* entry there, a `loom/...` diagnostics-registry code present in the [diagnostics registry](../spec_topics/diagnostics.md), or — for the structural **Pi SDK boundary site** (1), which traces to no spec obligation — the literal `pi-sdk-boundary` exempt-site token (whose trailing `<spec-page>` field names this *Specific exception types only* rule rather than a spec page); an entry resolving to none of these is a CI failure. This predicate parallels *Sequential by default*'s REQ-ID-or-code-keyed-area predicate, extended with the `loom/...` registry-code arm that site (3) needs and the structural `pi-sdk-boundary` token that site (1) needs, so every mandated exempt site below has a satisfiable token. An **exempt broad-catch site** is one of: (1) a **Pi SDK boundary site** — a call into, or a `typeof` / `"in"` / property access against, a value owned by `@earendil-works/pi-coding-agent` whose runtime shape loom cannot statically guarantee, so a hostile getter, `Proxy`, version-skewed payload, or `null`-prototype object may throw; (2) the [*Self-failure*](../spec_topics/pi-integration-contract.md) probe traps and their hostile-import defensive reads; (3) the [*Unknown-reason rule*](../spec_topics/pi-integration-contract/unknown-reason-rule.md#unknown-reason-rule) `event.reason` property-access read; (4) the [*Per-step isolation*](../spec_topics/pi-integration-contract.md) sub-step `try`/`catch` wraps of the `session_shutdown` handler, including its loom-internal `LoomRegistry` call sites; and (5) the [*Diagnostic-emission isolation*](../spec_topics/pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation) `console.error` wraps. Sites (2)–(5) qualify even though the wrapped operation is not always a direct Pi SDK member call: the spec mandates the broad catch there to keep a probe trap, a property-access throw, a teardown sub-step, or a diagnostic emission from escaping its handler. Each site cites a token the predicate above admits: site (1) cites the structural `pi-sdk-boundary` token (no spec obligation exists for it); site (3) cites its `loom/host/session-shutdown-*` diagnostics-registry code; sites (2), (4), and (5) cite the PIC REQ-IDs their owning leaves close (`V9a` PIC-3…6, `V9g` PIC-7, `V7a` PIC-21).
- **Sequential by default.** `Promise.all`, `Promise.race`, `Promise.allSettled`, and `Promise.any` are forbidden in production code (`src/**`, excluding `**/*.test.ts`) unless the calling leaf's `Spec.` field cites an obligation whose normative text mandates concurrency at this site — either (i) a spec REQ-ID, or (ii) a **code-keyed obligation area**: a named normative step on a page enumerated under [`coverage-matrix.md`](./coverage-matrix.md) *Code-keyed obligation areas (no numbered REQ-IDs)* whose normative text mandates concurrency at the site — and the leaf's `Adds.` field names the construct together with that REQ-ID or code-keyed obligation area. An ESLint `no-restricted-syntax` rule enumerates a per-site allow-list; each entry MUST carry a same-line comment `// allow: <REQ-ID-or-code-keyed-area> — <spec-page>`. The loom 1.0 closing gate asserts every allow-list entry's cited token resolves either to a REQ-ID present in [`coverage-matrix.md`](./coverage-matrix.md) or to an enumerated *Code-keyed obligation areas (no numbered REQ-IDs)* entry there; an entry resolving to neither is a CI failure. Test code (`**/*.test.ts`) is unrestricted but MUST NOT carry the exemption comment (so `grep`-based audits stay clean).
- **No silent test skipping.** Vitest's `assert.fail` / `expect.unreachable` when prerequisites are missing — never silent `return` early. A skipped test is a lie: it reports success when nothing was verified.
- **Spec drift.** If implementation reveals the spec is wrong, ambiguous, or under-specified, **stop**, fix the spec first in a dedicated commit, then resume.
- **Doc updates.** After each leaf, update `README.md`'s status table and append a one-line dated entry to `CHANGELOG.md`. The plan itself is updated only when the **plan** changes; non-plan discoveries go to `notes.md`. Each of these three artifacts is **created on first use if absent** — the first leaf that needs one creates it in the shape below, then writes its entry; do not pre-create them ahead of real content. Shapes: the `README.md` status table is a markdown table with columns `Leaf | Status | Date | Notes`, one row per completed leaf (replacing the current prose `## Status` section on first write); each `CHANGELOG.md` entry is a single line of the form `YYYY-MM-DD <leaf-id> — <one-line summary>`, newest at the bottom; `notes.md` is a free-form running log of non-plan discoveries with no fixed schema. These three artifacts live at the repository root, outside `docs/`; the orchestrator's `commitAddPaths` (`.pi/project-config.md`) auto-stages only `docs/`, so the leaf stages `README.md`, `CHANGELOG.md`, and `notes.md` explicitly (`git add`) as part of its commit. **Enforcement posture.** Unlike the cross-cutting rules backed by a mechanical surface — *Specific exception types only* (`no-broad-catch` lint), *Sequential by default* (`no-restricted-syntax` allow-list), *REQ-ID discipline* and *Diagnostic message anchors* (loom 1.0 closing-gate checks) — this rule has **no mechanical gate**: no lint rule, architectural test, or closing-gate check inspects `README.md`, `CHANGELOG.md`, or `notes.md`. It is contributor-discipline, verified at the *Per-phase TDD ritual* self-review step above (which checks the per-leaf doc artifacts were written) and, where one runs, the loom 1.0 release-time reviewer inspection — the same named manual-review surfaces the *No globals, statics, singletons* rule routes its scan residue to, not a new parallel gate.
- **Diagnostic message anchors.** Tests asserting a diagnostic's rendered message MUST cite the diagnostic code and source the expected string from the *Message* column of the [diagnostics registry](../spec_topics/diagnostics.md). Phrasings like "matches spec verbatim" without a code anchor are insufficient — the registry is the single source of truth for every author-visible message string.
- **REQ-ID discipline.** Per-prefix and per-ID governance is owned by [`../spec_topics/governance.md`](../spec_topics/governance.md): rows are immutable and append-only, prefixes are word-boundary-disjoint identifier tokens, the union of live and retired prefixes covers every prefix that appears in `spec_topics/*.md`, and mutations follow the procedures pinned there. An *executable spec REQ-ID* is a numbered `PREFIX-N` REQ-ID in the [`governance.md`](../spec_topics/governance.md) prefix table that denotes a runtime/parse/load obligation the loom implements — every numbered REQ-ID except the corpus-governance `GOV-*` family, which governs the spec corpus itself rather than runtime behaviour and maps to no runtime leaf. The set is the numbered `PREFIX-N` surface only: a registered prefix that anchors no live REQ-ID (e.g. `CEIL`) contributes nothing to it, and the `loom/{parse,load,runtime}/*` diagnostic-registry codes (including warning-severity `W` codes) and the code-keyed obligation areas are separate closing-gate surfaces — described below — not members of the executable-REQ-ID set. New diagnostic sites added by future spec work MUST land their REQ-IDs in the same edit. The loom 1.0 closing gate treats an **executable** spec REQ-ID without a coverage-matrix mapping, a retired-ID clashing with a live-ID, or a per-prefix numbering hole as a CI failure. A *per-prefix numbering hole* on a page is a positive integer `n ≤ max(live ∪ retired)` for some prefix the page owns — either a REQ-ID prefix per GOV-1 or an inline-label prefix per [GOV-16](../spec_topics/governance/stable-inline-labels.md#gov-16) — whose `n` appears in neither the page's live table for that prefix nor its retirement table for that prefix (the `## Retired REQ-IDs` section for REQ-ID prefixes per GOV-8, or the `## Retired inline labels` section for inline-label prefixes per GOV-16 *Retirement section*). Gaps that the retirement table accounts for are not failures; the gate only fires on numbers that are neither live nor retired. A test asserting a numbered REQ-ID MUST cite that REQ-ID inline in the test source — the same code-citation discipline the *Diagnostic message anchors* rule imposes on diagnostic codes — and the closing gate treats a [`coverage-matrix.md`](./coverage-matrix.md)-mapped numbered REQ-ID with no citing test in the test corpus as a CI failure, bringing the numbered-REQ-ID surface to the registry-code↔asserting-test parity stated below. This citing-test reconciliation is a best-effort scan for the inline `PREFIX-N` citation across the test sources: it certifies that a citing test exists for each mapped REQ-ID, not that the cited test's assertion is semantically faithful to the obligation — faithfulness stays a TDD and self-review obligation under the per-phase ritual. Like the unmapped-REQ-ID surface it is exercised against the seeded fixtures at [`H5a`](./H5a-closing-gate-automation.md) and binds against the live test corpus only at the loom 1.0 release gate activated by [`H6a`](./H6a-live-corpus-activation.md). Diagnostic codes carry the same closing-gate discipline: a new code added to the registry table of [`../spec_topics/diagnostics.md`](../spec_topics/diagnostics.md) MUST land its asserting test in the same edit; a registry code without an asserting test (and any asserted code not in the registry) is a CI failure. A third closing-gate surface covers the residue these two leave behind: a normative MUST/MUST-NOT on a non-narrative `spec_topics/**` page that carries neither a numbered `PREFIX-N` REQ-ID nor a `loom/...` diagnostics-registry code is the un-anchored-obligation class [`governance.md` GOV-22](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-22) names. Each such obligation MUST be enumerated in [`coverage-matrix.md`](./coverage-matrix.md) under *Code-keyed obligation areas (no numbered REQ-IDs)* with a named closing leaf, and the closing gate treats an un-anchored normative MUST/MUST-NOT with no enumerated closing leaf as a CI failure on the same live-corpus footing as an unmapped executable REQ-ID — exercised against the seeded fixtures at [`H5a`](./H5a-closing-gate-automation.md) and binding against the live `spec_topics/**` corpus only at the loom 1.0 release gate. The gate's recogniser is a best-effort `MUST`/`MUST NOT`-token text scan over non-narrative `spec_topics/**` pages — a `spec_topics/**` page is *non-narrative*, and thus in scope, iff its row in the [REQ-ID prefix table](../spec_topics/governance/req-id-prefix-table-active-a.md) (surfaced through [`governance.md`](../spec_topics/governance.md)) carries a cell that is **not** the byte-exact `(no IDs — narrative)` literal per [`governance.md` GOV-3](../spec_topics/governance/req-id-prefix-table-active-a.md#gov-3); this is the same byte-exact comparison GOV-3 mandates for REQ-ID extraction (and the same exclusion the *Leaf format* *Tests* bullet cites for leaf-citation scope), so the recogniser's page-scope and the REQ-ID extractor's page-scope share one definition — that excludes obligations already carrying a `PREFIX-N` REQ-ID or a `loom/...` registry code. A `loom/typecheck/*` build-time `tsc` brand string is **not** a diagnostics-registry code and does not anchor a normative obligation for this recogniser — the same carve-out [`H5a`](./H5a-closing-gate-automation.md) applies to registry reconciliation, restated here so the open `loom/...` form does not silently re-admit the `loom/typecheck/*` prefix as an anchor. Like the H3a identifier-keyed ambient-access scan, it catches only what the token scan witnesses: it cannot mechanically catch obligations phrased without the literal `MUST`/`MUST NOT` token (e.g. "shall", "is required to", lowercase, or split-across-table-cell phrasings), cannot distinguish a normative MUST from one inside narrative, an illustrative example, or a quoted passage. The gate does not mechanically exclude named cross-leaf seam obligations: it has no enumerated seam-name registry to read, so a MUST that is genuinely a Class-2 cross-leaf seam contract surfaces as a gate candidate and must be either enumerated in [`coverage-matrix.md`](./coverage-matrix.md) with a closing leaf or dispositioned as seam-name-dependent residue in the GOV-15 reviewer inspection. That residue — token-less obligations, narrative-ambiguous obligations, and seam-name-dependent obligations — is caught by the release-time editorial corpus review ([`governance.md` GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) reviewer-inspection step), not by the mechanical gate. Re-anchoring any individual MUST as a REQ-ID is a spec-side GOV-22 decision owned by the relevant spec-coverage finding, not by this plan rule, which only forces each un-anchored obligation onto a closing leaf. **Transitive-completeness plan-maintenance.** Whenever a new leaf is added that can introduce an executable REQ-ID, the citing test that *closes* a [`coverage-matrix.md`](./coverage-matrix.md)-mapped numbered REQ-ID (i.e. the test on the leaf the coverage matrix lists as that REQ-ID's closing leaf — the trigger turns on the coverage-matrix closing-leaf mapping, not on whether the citation is the REQ-ID's first textual occurrence, so a leaf that merely re-cites an already-closed REQ-ID does not qualify), or an un-anchored normative MUST, that leaf MUST be added to [`H5b`](./H5b-warn-only-canary.md)'s `Deps.` — the leaf that owns the complete coverage-producing dependency set — so the warn-only canary and the [`H6a`](./H6a-live-corpus-activation.md) hard-fail release gate (which inherits the set transitively through `H5b` rather than restating it) both stay sequenced after every coverage-producing leaf, otherwise the canary pre-flights — and the gate activates — against incomplete coverage. This obligation is mechanically backed: [`H5a`](./H5a-closing-gate-automation.md)'s standing transitive-completeness gate arm reconciles every closing-leaf cell in [`coverage-matrix.md`](./coverage-matrix.md) (excluding the `<new>` placeholder rows and the *IMPL* row's pure back-references) against the expanded membership of [`H5b`](./H5b-warn-only-canary.md)'s `Deps.` and fails CI on any closing-leaf cell none of whose listed leaves is in it — a multi-leaf (primary + co-witness) cell such as `CIO-5 | V16a, H7a` stays green as long as one listed leaf is present, so a co-witness leaf intentionally excluded from `H5b`'s `Deps.` does not redden the arm — so this rule and the gate stay in lockstep.
