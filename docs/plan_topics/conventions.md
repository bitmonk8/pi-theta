# Plan structure and conventions

Three kinds of phase:

1. **Horizontal phases.** Project infrastructure — scaffold, dependency-injection skeleton, diagnostics primitive, Pi-extension shell, end-to-end harness, REQ-ID anchor / coverage-matrix automation. Horizontal leaves operationalise project-level conventions (this page) rather than implement a normative spec rule, and cite a **Convention.** field instead of a **Spec.** field.
2. **MVP phase.** The smallest end-to-end `.loom` that runs as a Pi slash command — a single narrow vertical proving the pipeline before slice work begins.
3. **Vertical slices.** Each slice is a coherent feature area (lexer, expressions, schemas, queries, …). Each leaf inside a slice is the smallest feature that can ship and be tested independently. Slices are roughly ordered by their dependency DAG; non-linear deps are stated in each leaf's **Deps** field. Reorder freely as long as the deps DAG is respected.

Leaves are the unit of work; slice grouping is editorial. Leaf IDs follow `<group><letter>` (e.g. `V4b`); MVP and vertical features additionally carry a paired **tests-task** leaf whose ID is the feature ID with a `-T` suffix (e.g. `V4b-T`, `M-T`). The IDs `H1`–`Hn`, `M`, and `V1`–`Vn` (with their leaf and `-T` forms) are reserved for plan phases — when plan prose needs to refer to the initial release of the loom language, write "loom 1.0" or "the initial release"; never reuse a plan ID for that meaning.

## Per-phase TDD ritual (mandatory)

TDD is expressed as **two separate tasks** for every MVP and vertical feature; horizontal infrastructure leaves run a single-task ritual.

### MVP and vertical features — tests-task → implementation-task pairing

Each feature is an ordered pair of leaves, each landing as its own commit:

1. **Tests task** — leaf ID `<id>-T` (e.g. `V4b-T`). Write the failing tests for *every* spec REQ-ID the feature introduces. One assertion per rule where practical. A test that would pass when prerequisites are missing is a defect — fix it before the implementation task. The tests task adds no production code beyond the minimum needed for the tests to compile and fail for the intended reason. **Ships when** the tests exist, compile, and fail red for the intended reason; tag the commit `<id>-T-complete`.
2. **Implementation task** — leaf ID `<id>` (e.g. `V4b`). Lists `<id>-T` in its **Deps.** Then run the loop:
   1. **Implement.** Write the smallest correct increment that turns the red tests green. Correctness is the goal; tests are the evidence. Do not under-implement to game a thin test, and do not add speculative APIs (unused exports, public hooks no test in this leaf or its declared downstream consumers exercises).
   2. **Run.** All `<id>-T` tests green; type-check clean; lint clean.
   3. **Self-review.** Re-read the spec section, the diff, the test list. Check: any rule unverified? any silent skip? any broad `catch` that should be a specific type? any global / static / singleton creeping in?
   4. **Fix review issues.** Iterate from step 2 until the review is clean.
   5. **Phase exit gate.** "Ships when" criterion observable; tag commit `<id>-complete`.

A feature is **not** complete until the implementation task's exit gate is met. No "we'll fix it next slice" carry-overs.

### Horizontal leaves — single task

Infrastructure leaves are **not** test-paired (no `-T` leaf): they carry inline tests where they hold logic (architectural tests, lint-rule wiring, harness self-checks). A horizontal leaf runs the **Implement → Run → Self-review → Fix → exit gate** loop above in one task and ships when its "Ships when" condition is observable; tag the commit `<id>-complete`.

## Leaf format

Each leaf has the same fields, in the same order. The blank skeleton lives in [`leaf-template.md`](./leaf-template.md).

- **Spec.** Page(s) under [`../spec_topics/`](../spec_topics/) the leaf implements (an implementer MAY restrict their reading to these pages), *or* **Convention.** Section(s) of this page the leaf operationalises (used by horizontal phases, which derive from project-level conventions rather than spec rules). The **Spec** field MUST be closed under normative cross-link: closure runs from each listed topic outbound, applies transitively to a fixed point, and excludes narrative cross-links per [`governance.md` GOV-3](../spec_topics/governance.md).
- **Adds.** One sentence — what the leaf introduces. `Adds.` is **descriptive** by default: the binding obligations of a leaf are its **Tests.** REQ-ID bullets (together with the `conventions.md` cross-cutting rules they inherit) and its **Ships when** gate — `Adds.` prose does not, on its own, bind the implementer. A mechanism named in `Adds.` binds only when it is either (i) an observable behaviour traced to a normative spec obligation — a cited `PREFIX-N` REQ-ID, a cited `loom/...` diagnostic-registry code, or a named normative step on a code-keyed obligation page per [`coverage-matrix.md`](./coverage-matrix.md) "Code-keyed obligation areas (no numbered REQ-IDs)" (e.g. the concurrency construct the *Sequential by default* rule requires `Adds.` to name alongside its mandating REQ-ID or code-keyed obligation area), or (ii) a named cross-leaf **seam** that some other leaf binds against — i.e. a leaf listing this leaf in its `Deps.` names the seam in its own `Adds.` or `Tests.`. A seam named in `Adds.` with no such consumer remains illustrative (non-binding). Any other mechanism named in `Adds.` is illustrative — it describes the intended shape but authorises no Class-3 architecture mandate when it is neither a Class-1 observable behaviour (REQ-ID- or code-keyed) nor a Class-2 consumer-bound seam.
- **Tests.** Bullet list — one bullet per **REQ-ID** the leaf claims to implement; cite the ID inline (e.g. `BNDR-7: ...`). Where a leaf implements only part of a rule, each Tests bullet still cites its REQ-ID; the coverage matrix's REQ-ID-to-leaf mapping is many-to-many. The REQ-ID prefix table lives in [`../spec_topics/governance.md`](../spec_topics/governance.md). Pure-narrative pages — those whose row in the prefix table carries the `(no IDs — narrative)` cell per [`governance.md` GOV-3](../spec_topics/governance.md) — carry no IDs and need no leaf citation.
- **Deps.** Other leaf IDs that must be complete first. Listed `-` if none. Cite specific leaf IDs (`V4b`, `V9a–V9e`); never a bare group token (`V4`) — group IDs are editorial only and a bare `VN` in `Deps.` is ambiguous between "every leaf" and "some subset". Use ranges where contiguous and comma-separated lists where not. An implementation task MUST list its paired `<id>-T` tests task here.

  In the **coverage matrix**, each spec REQ-ID maps to the **implementation** leaf (`<id>`) that closes it (its green tests are the closure evidence); the paired `<id>-T` tests task is not listed separately.
- **Ships when.** A concrete, externally observable change.

## Cross-cutting rules (every leaf)

- **No globals, statics, singletons.** All collaborators passed by constructor. An architectural test in the scaffold leaf enforces this against `src/**`.
- **Specific exception types only.** No `catch (e)`, `catch (e: unknown)`, `catch (e: any)`, or `catch (e: Error)` — bind to a specific subtype or let the exception propagate. The rethrow-on-mismatch pattern (`catch (e) { if (!(e instanceof X)) throw e; … }`) is also forbidden. An ESLint rule (`no-broad-catch`) wired in the scaffold leaf enforces this. A `catch (e: unknown)` clause is permitted only at an **exempt broad-catch site**, and only if the same line carries a `// allow-broad-catch: <REQ-ID> — <spec-page>` comment and the call site is enumerated in the lint allow-list; the loom 1.0 closing gate asserts every allow-list entry has a matching REQ-ID present in [`coverage-matrix.md`](./coverage-matrix.md). An **exempt broad-catch site** is one of: (1) a **Pi SDK boundary site** — a call into, or a `typeof` / `"in"` / property access against, a value owned by `@earendil-works/pi-coding-agent` whose runtime shape loom cannot statically guarantee, so a hostile getter, `Proxy`, version-skewed payload, or `null`-prototype object may throw; (2) the [*Self-failure*](../spec_topics/pi-integration-contract.md) probe traps and their hostile-import defensive reads; (3) the [*Unknown-reason rule*](../spec_topics/pi-integration-contract/unknown-reason-rule.md#unknown-reason-rule) `event.reason` property-access read; (4) the [*Per-step isolation*](../spec_topics/pi-integration-contract.md) sub-step `try`/`catch` wraps of the `session_shutdown` handler, including its loom-internal `LoomRegistry` call sites; and (5) the [*Diagnostic-emission isolation*](../spec_topics/pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation) `console.error` wraps. Sites (2)–(5) qualify even though the wrapped operation is not always a direct Pi SDK member call: the spec mandates the broad catch there to keep a probe trap, a property-access throw, a teardown sub-step, or a diagnostic emission from escaping its handler.
- **Sequential by default.** `Promise.all`, `Promise.race`, `Promise.allSettled`, and `Promise.any` are forbidden in production code (`src/**`, excluding `**/*.test.ts`) unless the calling leaf's `Spec.` field cites an obligation whose normative text mandates concurrency at this site — either (i) a spec REQ-ID, or (ii) a **code-keyed obligation area**: a named normative step on a page enumerated under [`coverage-matrix.md`](./coverage-matrix.md) *Code-keyed obligation areas (no numbered REQ-IDs)* whose normative text mandates concurrency at the site — and the leaf's `Adds.` field names the construct together with that REQ-ID or code-keyed obligation area. An ESLint `no-restricted-syntax` rule enumerates a per-site allow-list; each entry MUST carry a same-line comment `// allow: <REQ-ID-or-code-keyed-area> — <spec-page>`. The loom 1.0 closing gate asserts every allow-list entry's cited token resolves either to a REQ-ID present in [`coverage-matrix.md`](./coverage-matrix.md) or to an enumerated *Code-keyed obligation areas (no numbered REQ-IDs)* entry there; an entry resolving to neither is a CI failure. Test code (`**/*.test.ts`) is unrestricted but MUST NOT carry the exemption comment (so `grep`-based audits stay clean).
- **No silent test skipping.** `assert.fail` / `panic` when prerequisites are missing — never silent `return` early. A skipped test is a lie: it reports success when nothing was verified.
- **Spec drift.** If implementation reveals the spec is wrong, ambiguous, or under-specified, **stop**, fix the spec first in a dedicated commit, then resume.
- **Doc updates.** After each leaf, update `README.md`'s status table and append a one-line dated entry to `CHANGELOG.md`. The plan itself is updated only when the **plan** changes; non-plan discoveries go to `notes.md`. Each of these three artifacts is **created on first use if absent** — the first leaf that needs one creates it in the shape below, then writes its entry; do not pre-create them ahead of real content. Shapes: the `README.md` status table is a markdown table with columns `Leaf | Status | Date | Notes`, one row per completed leaf (replacing the current prose `## Status` section on first write); each `CHANGELOG.md` entry is a single line of the form `YYYY-MM-DD <leaf-id> — <one-line summary>`, newest at the bottom; `notes.md` is a free-form running log of non-plan discoveries with no fixed schema. These three artifacts live at the repository root, outside `docs/`; the orchestrator's `commitAddPaths` (`.pi/project-config.md`) auto-stages only `docs/`, so the leaf stages `README.md`, `CHANGELOG.md`, and `notes.md` explicitly (`git add`) as part of its commit.
- **Diagnostic message anchors.** Tests asserting a diagnostic's rendered message MUST cite the diagnostic code and source the expected string from the *Message* column of the [diagnostics registry](../spec_topics/diagnostics.md). Phrasings like "matches spec verbatim" without a code anchor are insufficient — the registry is the single source of truth for every author-visible message string.
- **REQ-ID discipline.** Per-prefix and per-ID governance is owned by [`../spec_topics/governance.md`](../spec_topics/governance.md): rows are immutable and append-only, prefixes are word-boundary-disjoint identifier tokens, the union of live and retired prefixes covers every prefix that appears in `spec_topics/*.md`, and mutations follow the procedures pinned there. New diagnostic sites added by future spec work MUST land their REQ-IDs in the same edit. The loom 1.0 closing gate treats a spec REQ-ID without a coverage-matrix mapping, a retired-ID clashing with a live-ID, or a per-prefix numbering hole as a CI failure. A *per-prefix numbering hole* on a page is a positive integer `n ≤ max(live ∪ retired)` for some prefix the page owns — either a REQ-ID prefix per GOV-1 or an inline-label prefix per [GOV-16](../spec_topics/governance/stable-inline-labels.md#gov-16) — whose `n` appears in neither the page's live table for that prefix nor its retirement table for that prefix (the `## Retired REQ-IDs` section for REQ-ID prefixes per GOV-8, or the `## Retired inline labels` section for inline-label prefixes per GOV-16 *Retirement section*). Gaps that the retirement table accounts for are not failures; the gate only fires on numbers that are neither live nor retired. A test asserting a numbered REQ-ID MUST cite that REQ-ID inline in the test source — the same code-citation discipline the *Diagnostic message anchors* rule imposes on diagnostic codes — and the closing gate treats a [`coverage-matrix.md`](./coverage-matrix.md)-mapped numbered REQ-ID with no citing test in the test corpus as a CI failure, bringing the numbered-REQ-ID surface to the registry-code↔asserting-test parity stated below. This citing-test reconciliation is a best-effort scan for the inline `PREFIX-N` citation across the test sources: it certifies that a citing test exists for each mapped REQ-ID, not that the cited test's assertion is semantically faithful to the obligation — faithfulness stays a TDD and self-review obligation under the per-phase ritual. Like the unmapped-REQ-ID surface it is exercised against the seeded fixtures at [`H5a`](./H5a-closing-gate-automation.md) and binds against the live test corpus only at the loom 1.0 release gate activated by [`H6a`](./H6a-live-corpus-activation.md). Diagnostic codes carry the same closing-gate discipline: a new code added to the registry table of [`../spec_topics/diagnostics.md`](../spec_topics/diagnostics.md) MUST land its asserting test in the same edit; a registry code without an asserting test (and any asserted code not in the registry) is a CI failure. A third closing-gate surface covers the residue these two leave behind: a normative MUST/MUST-NOT on a non-narrative `spec_topics/**` page that carries neither a numbered `PREFIX-N` REQ-ID nor a `loom/{parse,load,runtime}/*` diagnostics-registry code — and is not a named cross-leaf seam — is the un-anchored-obligation class [`governance.md` GOV-22](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-22) names. Each such obligation MUST be enumerated in [`coverage-matrix.md`](./coverage-matrix.md) under *Code-keyed obligation areas (no numbered REQ-IDs)* with a named closing leaf, and the closing gate treats an un-anchored normative MUST/MUST-NOT with no enumerated closing leaf as a CI failure on the same live-corpus footing as an unmapped executable REQ-ID — exercised against the seeded fixtures at [`H5a`](./H5a-closing-gate-automation.md) and binding against the live `spec_topics/**` corpus only at the loom 1.0 release gate. The gate's recogniser is a best-effort `MUST`/`MUST NOT`-token text scan over non-narrative `spec_topics/**` pages that excludes obligations already carrying a `PREFIX-N` REQ-ID, a `loom/{parse,load,runtime}/*` registry code, or a named cross-leaf seam. Like the H3a identifier-keyed ambient-access scan, it catches only what the token scan witnesses: it cannot mechanically catch obligations phrased without the literal `MUST`/`MUST NOT` token (e.g. "shall", "is required to", lowercase, or split-across-table-cell phrasings), cannot distinguish a normative MUST from one inside narrative, an illustrative example, or a quoted passage, and only approximates the evolving cross-leaf seam-name set its non-seam arm depends on. That residue — token-less, narrative-ambiguous, or seam-name-dependent obligations — is caught by the release-time editorial corpus review ([`governance.md` GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) reviewer-inspection step), not by the mechanical gate. Re-anchoring any individual MUST as a REQ-ID is a spec-side GOV-22 decision owned by the relevant spec-coverage finding, not by this plan rule, which only forces each un-anchored obligation onto a closing leaf.
