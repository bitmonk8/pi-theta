# Plan structure and conventions

Three kinds of phase:

1. **Horizontal phases.** Project infrastructure — scaffold, dependency-injection skeleton, diagnostics primitive, Pi-extension shell, end-to-end harness, REQ-ID anchor / coverage-matrix automation. Horizontal leaves operationalise project-level conventions (this page) rather than implement a normative spec rule, and cite a **Convention.** field instead of a **Spec.** field.
2. **MVP phase.** The smallest end-to-end `.loom` that runs as a Pi slash command — a single narrow vertical proving the pipeline before slice work begins.
3. **Vertical slices.** Each slice is a coherent feature area (lexer, expressions, schemas, queries, …). Each leaf inside a slice is the smallest feature that can ship and be tested independently. Slices are roughly ordered by their dependency DAG; non-linear deps are stated in each leaf's **Deps** field. Reorder freely as long as the deps DAG is respected.

Horizontal leaves may be presented under an editorial sub-heading (such as a release-gate grouping) without forming a new category: such a sub-grouping creates no new leaf-ID prefix and no new **Convention.**/**Spec.** rule — the grouped leaves remain horizontal leaves in every other respect.

Leaves are the unit of work; slice grouping is editorial. Leaf IDs follow `<group><letter>` (e.g. `V4b`); MVP and vertical features additionally carry a paired **tests-task** leaf whose ID is the feature ID with a `-T` suffix (e.g. `V4b-T`, `M-T`). The IDs `H1`–`Hn`, `M`, and `V1`–`Vn` (with their leaf and `-T` forms) are reserved for plan phases — when plan prose needs to refer to the initial release of the loom language, write "loom 1.0" or "the initial release"; never reuse a plan ID for that meaning.

## Per-phase TDD ritual (mandatory)

TDD is expressed as **two separate tasks** for every MVP and vertical feature; horizontal infrastructure leaves run a single-task ritual.

**"Fail red for the intended reason" (defined).** A tests-task suite *fails red for the intended reason* when both hold:

1. **It compiles and runs.** The test files type-check (via the `tsc` / `npm run typecheck` toolchain [`H1a`](./H1a-scaffold-and-toolchain.md) provisions) against the stubbed or declared seams the tests task is permitted to add (the ritual permits "the minimum production code needed for the tests to compile and fail"). A suite that cannot type-check — a TypeScript/compile error, e.g. a reference to a not-yet-declared production symbol — is a **compile-time red** and does **not** satisfy the gate; declare or stub the seam so the suite compiles.
2. **Each test reds out on its own primary assertion because the implementation under test is absent or incomplete** — not on a missing or unconfigured fixture, a harness/setup throw, or an unrelated exception.

A suite red for any non-behavioural reason (compile error, missing/unconfigured fixture, harness/setup throw, unrelated exception) does **not** satisfy the gate and must be fixed before tagging `<id>-T-complete`. The companion impl-side phrasing "fail for the intended reason" carries the same meaning.

### MVP and vertical features — tests-task → implementation-task pairing

Each feature is an ordered pair of leaves, each landing as its own commit:

1. **Tests task** — leaf ID `<id>-T` (e.g. `V4b-T`). Write the failing tests for *every* spec REQ-ID the feature introduces. Default to one assertion per REQ-ID so each rule reds out in isolation; fold multiple rules into a single assertion only when they describe one indivisible behaviour that cannot fail rule-by-rule, and even then cite each rule's `PREFIX-N` inline so the failure still names the offending rule. A test that would pass when prerequisites are missing is a defect — fix it before the implementation task. The tests task adds no production code beyond the minimum needed for the tests to compile and *fail red for the intended reason* (defined above). **Ships when** the tests exist, compile, and *fail red for the intended reason* as defined above; tag the commit `<id>-T-complete` and push the tag to the shared remote.
2. **Implementation task** — leaf ID `<id>` (e.g. `V4b`). Lists `<id>-T` in its **Deps.** Then run the loop:
   1. **Implement.** Write the smallest correct increment that turns the red tests green. Correctness is the goal; tests are the evidence. Do not under-implement to game a thin test, and do not add speculative APIs (unused exports, public hooks no test in this leaf or in any leaf that lists this leaf in its **Deps.** exercises).
   2. **Run.** All `<id>-T` tests green; type-check clean (the `tsc` / `npm run typecheck` step [`H1a`](./H1a-scaffold-and-toolchain.md) provisions); lint clean.
   3. **Self-review.** Re-read the spec section, the diff, the test list. Check: any rule unverified? any silent skip? any broad `catch` that should be a specific type? any global / static / singleton — including the closure-captured, lazy module-cache, and DI-container forms the `src/**` architectural test does not detect — creeping in? any indirect ambient-primitive access the `H3a` identifier-keyed scan does not witness — an aliased read (`const env = process.env`), a destructured read (`const { cwd } = process`), computed access (`process["env"]`), a helper wrapper, or re-export indirection — reaching a banned primitive outside its seam adapter? any synchronous blocking call that stalls Pi's shared event loop that the `H2a` blocking-runtime check does not detect (busy-wait loops, dynamically-constructed sync calls, or other synchronous stalls whose callee name does not end in the literal suffix `Sync`) in `src/**`, per the *Sequential by default* blocking-runtime ban — that check mechanically catches every call whose callee name ends in the literal suffix `Sync` (a member access such as `fs.readFileSync` or a bare identifier such as `execSync`), leaving only this residue to self-review? any *Doc updates* artifact missing for this leaf — the `README.md` status row, the dated `CHANGELOG.md` line, and (for non-plan discoveries) the `notes.md` entry?
   4. **Fix review issues.** Iterate from step 2 until the review is clean.
   5. **Phase exit gate.** "Ships when" criterion observable; tag commit `<id>-complete` and push the tag to the shared remote.

A feature is **not** complete until the implementation task's exit gate is met. No "we'll fix it next slice" carry-overs.

### Horizontal leaves — single task

Infrastructure leaves are **not** test-paired (no `-T` leaf): they carry inline tests where they hold logic (architectural tests, lint-rule wiring, harness self-checks). A horizontal leaf runs the **Implement → Run → Self-review → Fix → exit gate** loop above in one task and ships when its "Ships when" condition is observable; tag the commit `<id>-complete` and push the tag to the shared remote.

## Leaf format

Each leaf has the same fields, in the same order. The blank skeleton lives in [`leaf-template.md`](./leaf-template.md).

- **Spec.** Page(s) under [`../spec_topics/`](../spec_topics/) the leaf implements (an implementer MAY restrict their reading to these pages), *or* **Convention.** Section(s) of this page the leaf operationalises (used by horizontal phases, which derive from project-level conventions rather than spec rules). The **Spec** field MUST be closed under normative cross-link: closure runs from each listed topic outbound, applies transitively to a fixed point, and excludes narrative cross-links per [`governance.md` GOV-3](../spec_topics/governance.md). A *normative cross-link* is a cite from a listed page to a rule on a **non-narrative** target page; a *narrative cross-link* is one whose target page is pure-narrative — a page whose row in the prefix table carries the byte-exact `(no IDs — narrative)` cell per [`governance.md` GOV-3](../spec_topics/governance.md), the same byte-exact narrative-page test the §Tests bullet below applies. Closure follows the normative cross-links and stops at the narrative ones.
- **Adds.** One sentence — what the leaf introduces. `Adds.` is **descriptive** by default: the binding obligations of a leaf are its **Tests.** REQ-ID bullets (together with the `conventions.md` cross-cutting rules they inherit) and its **Ships when** gate — `Adds.` prose does not, on its own, bind the implementer. A mechanism named in `Adds.` binds only when it is either (i) a **Class-1** observable behaviour traced to a normative spec obligation — a cited `PREFIX-N` REQ-ID, a cited `loom/...` diagnostic-registry code, or a named normative step on a code-keyed obligation page per [`coverage-matrix.md`](./coverage-matrix.md) "Code-keyed obligation areas (no numbered REQ-IDs)" (e.g. the concurrency construct the *Sequential by default* rule requires `Adds.` to name alongside its mandating REQ-ID or code-keyed obligation area), or (ii) a **Class-2** named cross-leaf **seam** that some other leaf binds against — i.e. a leaf listing this leaf in its `Deps.` names the seam in its own `Adds.` or `Tests.`. *Names the seam* is satisfied by an **unambiguous prose reference**: the consumer's `Adds.` or `Tests.` prose refers to the seam in a way that resolves to exactly one mechanism and cites this leaf's owning leaf-ID (e.g. `V5e`'s `Adds.` "consults `V16a`'s cross-ceiling arbitration seam"). It does **not** require the consumer to repeat an exact identifier token drawn from this leaf's `Adds.`; a prose description that pins both the seam and its owning leaf-ID binds it. A seam named in `Adds.` with no such consumer remains illustrative (non-binding). Any other mechanism named in `Adds.` is a **Class-3** architecture or shape mandate — neither a Class-1 observable behaviour (REQ-ID- or code-keyed) nor a Class-2 consumer-bound seam — and is illustrative: it describes the intended shape, but `Adds.` prose alone does not authorise it.
- **Tests.** Bullet list — one bullet per **REQ-ID** the leaf claims to implement; cite the ID inline (e.g. `BNDR-7: ...`). Where a leaf implements only part of a rule, each Tests bullet still cites its REQ-ID; the coverage matrix's REQ-ID-to-leaf mapping is many-to-many. A leaf that closes a GOV-16 stable-inline-label family by per-label single-leaf closure (e.g. `HC3`) cites each inline label the same way — one *Tests* bullet per inline label, the label cited inline (e.g. `HC3-a: ...`); a no-single-leaf family (e.g. `NOCEIL`) instead witnesses each inline label by a distributed diagnostic-code *Tests* bullet carrying the label as a parenthetical seam annotation. Both shapes follow the *stable-inline-label closure* semantics defined in §REQ-ID discipline below, which owns the two shapes and defers to [`coverage-matrix.md`](./coverage-matrix.md) for which family takes which; inline labels are citable *Tests*-bullet subjects exactly as REQ-IDs are, and inline-label bullets carry no coverage-matrix row. The REQ-ID prefix table lives in [`../spec_topics/governance.md`](../spec_topics/governance.md). Pure-narrative pages — those whose row in the prefix table carries the `(no IDs — narrative)` cell per [`governance.md` GOV-3](../spec_topics/governance.md) — carry no IDs and need no leaf citation.
- **Deps.** Other leaf IDs that must be complete first. Listed `-` if none. **Pickability predicate.** A `Deps.` entry is *satisfied* — and the leaf listing it becomes *pickable* on the dep-driven build order — exactly when the §Per-phase TDD ritual completion tag for that entry is present: an entry naming an implementation or horizontal leaf `<id>` is satisfied iff its `<id>-complete` git tag exists, and an entry naming a tests task `<id>-T` is satisfied iff its `<id>-T-complete` git tag exists. Those git tags are the authoritative pickability signal — sound only where the tag is visible to the evaluating clone: completion tags are pushed to the shared remote at the §Per-phase TDD ritual phase exit gate (which tags and pushes `<id>-complete` / `<id>-T-complete`), and pickability is evaluated against a clone that has fetched tags. An unpushed local tag, or a shallow / no-tags clone that has not fetched tags, makes the predicate read the tag as absent and silently misjudges pickability. The `README.md` status row and the dated `CHANGELOG.md` line (per the *Doc updates* rule above) are doc-update artifacts, not the pickability source. A leaf is pickable when every entry in its `Deps.` is satisfied. This predicate composes with, and does not override, the [§Blocked obligations](../plan.md#blocked-obligations) gating: a leaf flagged blocked (e.g. `V9l`) stays gated through its declared `Deps.` edge on the resolution-owner leaf, whose `<id>-complete` tag is absent until the deciding open question is resolved. Cite specific leaf IDs (`V4b`, `V9a–V9e`); never a bare group token (`V4`) — group IDs are editorial only and a bare `VN` in `Deps.` is ambiguous between "every leaf" and "some subset". Use ranges where contiguous and comma-separated lists where not. An implementation task MUST list its paired `<id>-T` tests task here. A new leaf whose `Tests.` bullets carry an executable REQ-ID, a citing test that *closes* a coverage-matrix-mapped numbered REQ-ID, or an un-anchored normative MUST is additionally added to [`H5b`](./H5b-warn-only-canary.md)'s `Deps.` per the *Transitive-completeness plan-maintenance* clause in §REQ-ID discipline below, which owns the exact triggers.

  In the **coverage matrix**, each spec REQ-ID maps to the **implementation** leaf (`<id>`) that closes it (its green tests are the closure evidence); the paired `<id>-T` tests task is not listed separately.
- **Ships when.** A concrete, externally observable change.

## Cross-cutting rules (every leaf)

The mechanical gates below are scoped to **`src/**`, the single production-code root**: the [`H2a`](./H2a-cross-cutting-gates.md) no-globals architectural test, the `no-broad-catch` and `no-restricted-syntax` lint rules, the [`H3a`](./H3a-di-seam-skeleton.md) ambient-primitive scan, and the [`H5a`](./H5a-closing-gate-automation.md) / [`H5c`](./H5c-broad-catch-allow-list-gate.md) closing-gate allow-list reconciliation all read `src/**` and no other tree. The Pi extension module the host loads — `extensions/index.ts`, auto-discovered from `package.json#pi.extensions = ["./extensions"]` per [`host-prerequisites.md` §"Extension entry point"](../spec_topics/pi-integration-contract/host-prerequisites.md) — sits outside `src/**`; it is a **thin delegating entry shim** that carries no rule-subject logic and exists only to re-export the `src/**` factory ([`H4a`](./H4a-factory-shell-and-harness.md)). The entry file's purity — that it holds no broad `catch`, no global / static / singleton, no ambient-primitive read, and no `Promise` combinator of its own — is the property that keeps rule-subject behaviour from falling outside the gated tree; because the file sits outside `src/**`, none of those mechanical gates inspects it, so that purity is verified at release time by the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) (checklist item 8), the same named manual-review surface the cross-cutting rules above route their conceded scan residue to, rather than by an unconditional mechanical guarantee the `src/**`-scoped gates do not provide.

- **No globals, statics, singletons.** All collaborators passed by constructor. An architectural test in [`H2a`](./H2a-cross-cutting-gates.md) enforces this against `src/**` for the module-level binding forms it can detect. The same discipline bans direct ambient-primitive access: no `src/**` module may directly reference `process.env`, `process.cwd`, `crypto.randomUUID`, or any of the timing primitives the spec's WallClock ban enumerates — `Date.now`, `performance.now`, `Date.prototype.getTime`, `setTimeout`, `clearTimeout` (this timing set is a manually-maintained derivation of the [`host-interfaces-services.md`](../spec_topics/pi-integration-contract/host-interfaces-services.md) `Clock`/`WallClock` ban (PIC-12), required to cover at least every primitive the spec ban enumerates (scan list ⊇ spec ban) and currently equal to the PIC-12 enumeration `Date.now`/`performance.now`/`Date.prototype.getTime`/`setTimeout`/`clearTimeout`, and is re-reconciled against the current spec ban by the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) (checklist item 9); the timing primitives' intended exempt site is the `WallClock` adapter, exempted by the same `// allow-ambient:` comment that exempts every other primitive at its owning adapter); the only legitimate references live inside the seam-adapter modules that own each primitive. A direct reference is an **exempt ambient site** only if its line carries a `// allow-ambient: <primitive> — <seam>` comment; the phrase *the ambient-access allow-list* denotes exactly the set of those same-line comments the scan discovers across `src/**`, not a separate registry file or artefact. Like the *Specific exception types only* and *Sequential by default* rules, exemption is comment-keyed and the "allow-list" is just the set of in-source comments — but, unlike those two ESLint-backed rules, the ambient scan applies no closing-gate token-resolution check to its comments (none is added by this rule). The identifier-keyed architectural test declared in [`H3a`](./H3a-di-seam-skeleton.md) flags any direct reference to one of these primitives that is **not** at an allow-listed site. The allow-list is append-only and starts empty: the seam adapters are authored by the `V8*` leaves, so each adapter site is registered as its leaf lands and no entry exists at `H3a` time — registering a new exempt site requires no `H3a` edit. The manual residue these two scans concede — the closure-captured, lazy module-cache, and DI-container singletons the `src/**` architectural test does not detect, and the aliased, destructured, computed (`process["env"]`), helper-wrapper, and re-export-indirection ambient reads the identifier-keyed scan does not witness — is owned by the *Per-phase TDD ritual* self-review step above and, at release time, by the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) below — its checklist items 1 (closure-captured / lazy-cache / DI-container singletons) and 2 (indirect ambient reads) enumerate exactly this rule's conceded residue, mirroring the way the [`governance.md` GOV-22](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-22) residue routes to that inspection's checklist item 5. The [`H2a`](./H2a-cross-cutting-gates.md) architectural test and the [`H3a`](./H3a-di-seam-skeleton.md) identifier-keyed scan catch the violation forms they can statically detect; release-time residue-inspection checklist items 1 (closure-captured / lazy-cache / DI-container singletons) and 2 (indirect ambient reads) are the best-effort human review of the residue those scans concede. Each item is governed by its **Clean when** condition, not by its illustrative example list — so a residue form that lies outside both the scans' reach and an item's named examples is still caught when it breaches that item's **Clean when** condition.
- **Specific exception types only.** No `catch (e)`, `catch (e: unknown)`, `catch (e: any)`, or `catch (e: Error)` — bind to a specific subtype or let the exception propagate. The rethrow-on-mismatch pattern (`catch (e) { if (!(e instanceof X)) throw e; … }`) is also forbidden. An ESLint rule (`no-broad-catch`) wired in [`H2a`](./H2a-cross-cutting-gates.md) enforces this. A `catch (e: unknown)` clause is permitted only at an **exempt broad-catch site** whose line carries a `// allow-broad-catch: <REQ-ID-or-code-keyed-area-token-or-registry-code-or-pi-sdk-boundary> — <spec-page>` comment; the phrase *the lint allow-list* denotes exactly the set of those same-line comments the scan discovers across `src/**`, not a separate registry file or artefact. The loom 1.0 closing gate asserts every allow-list entry's cited token resolves to one of: a REQ-ID present in [`coverage-matrix.md`](./coverage-matrix.md), the canonical *code-keyed-area token* (`cka-<n>`) of one row in [`coverage-matrix.md`](./coverage-matrix.md)'s *Code-keyed obligation areas (no numbered REQ-IDs)* table — resolving iff the cited token matches exactly one such row's *Token* cell, a concrete `loom/...` diagnostics-registry code present in the [diagnostics registry](../spec_topics/diagnostics.md) — the cited token MUST be a literal code that resolves to exactly one registry entry, never a glob or wildcard family (the gate's "code present in the registry" resolver matches no wildcard) — or — for the structural **Pi SDK boundary site** (1), which traces to no spec obligation — the literal `pi-sdk-boundary` exempt-site token (whose trailing `<spec-page>` field names this *Specific exception types only* rule rather than a spec page); an entry resolving to none of these is a CI failure. This predicate parallels *Sequential by default*'s REQ-ID-or-code-keyed-area predicate, extended with the `loom/...` registry-code arm that site (3) needs and the structural `pi-sdk-boundary` token that site (1) needs, so every mandated exempt site below has a satisfiable token. An **exempt broad-catch site** is one of: (1) a **Pi SDK boundary site** — a call into, or a `typeof` / `"in"` / property access against, a value owned by `@earendil-works/pi-coding-agent` whose runtime shape loom cannot statically guarantee, so a hostile getter, `Proxy`, version-skewed payload, or `null`-prototype object may throw; (2) the [*Self-failure*](../spec_topics/pi-integration-contract.md) probe traps and their hostile-import defensive reads; (3) the [*Unknown-reason rule*](../spec_topics/pi-integration-contract/unknown-reason-rule.md#unknown-reason-rule) `event.reason` property-access read; (4) the [*Per-step isolation*](../spec_topics/pi-integration-contract.md) sub-step `try`/`catch` wraps of the `session_shutdown` handler, including its loom-internal `LoomRegistry` call sites; and (5) the [*Diagnostic-emission isolation*](../spec_topics/pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation) `console.error` wraps. Sites (2)–(5) qualify even though the wrapped operation is not always a direct Pi SDK member call: the spec mandates the broad catch there to keep a probe trap, a property-access throw, a teardown sub-step, or a diagnostic emission from escaping its handler. Each site cites a token the predicate above admits: site (1) cites the structural `pi-sdk-boundary` token (no spec obligation exists for it); site (3) cites the concrete diagnostics-registry code its `event.reason` read emits — `loom/host/session-shutdown-reason-unknown` — plus, for the spec-permitted single widened handler-entry `try`/`catch`, the concrete `loom/host/session-shutdown-pinned-constant-unreadable` code (each a literal registry code, not a glob); sites (2), (4), and (5) cite the PIC REQ-IDs their owning leaves close (`V9a` PIC-3…6, `V9g` PIC-7, `V7d` PIC-21). **Enforcement posture.** The two mechanical surfaces — the [`H2a`](./H2a-cross-cutting-gates.md) `no-broad-catch` lint and the loom 1.0 closing-gate token-resolution arm above — verify only requirement (a), that each exempt site carries a `// allow-broad-catch:` comment whose cited token resolves; neither checks requirement (b), that the site is one of the five enumerated exempt-broad-catch-site categories. That category-membership residue carries **no mechanical gate** and is verified by the *Per-phase TDD ritual* self-review step's broad-`catch` check above and, at release time, by the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) (checklist item 6) — the plan-side review that runs alongside the spec's [`governance.md` GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) source-language-equivalence reviewer-inspection step, the same named manual-review surface the *No globals, statics, singletons* rule routes its scan residue to, not a new parallel gate.
- **Sequential by default.** `Promise.all`, `Promise.race`, `Promise.allSettled`, and `Promise.any` are forbidden in production code (`src/**`, excluding `**/*.test.ts`) unless the calling leaf's `Spec.` field cites an obligation whose normative text mandates concurrency at the specific operation the combinator call implements — either (i) a spec REQ-ID, or (ii) a **code-keyed obligation area**: a named normative step on a page enumerated under [`coverage-matrix.md`](./coverage-matrix.md) *Code-keyed obligation areas (no numbered REQ-IDs)* whose normative text mandates concurrency at the specific operation the combinator call implements — and the leaf's `Adds.` field names the construct together with that REQ-ID or code-keyed obligation area. An ESLint `no-restricted-syntax` rule enumerates a per-site allow-list; each entry MUST carry a same-line comment `// allow: <REQ-ID-or-code-keyed-area-token> — <spec-page>`. The phrase *the per-site allow-list* denotes exactly the set of those same-line `// allow:` comments the scan discovers across `src/**`, not a separate registry file or artefact. The loom 1.0 closing gate asserts every allow-list entry's cited token resolves either to a REQ-ID present in [`coverage-matrix.md`](./coverage-matrix.md) or to the canonical *code-keyed-area token* (`cka-<n>`) of one row in [`coverage-matrix.md`](./coverage-matrix.md)'s *Code-keyed obligation areas (no numbered REQ-IDs)* table, resolving iff the cited token matches exactly one such row's *Token* cell; an entry matching neither a REQ-ID nor exactly one code-keyed-area token is a CI failure. Test code (`**/*.test.ts`) is unrestricted but MUST NOT carry the exemption comment (so `grep`-based audits stay clean). This rule subsumes the CLAUDE.md *"Never block the async runtime"* directive: synchronous blocking of the event loop in `src/**` — `fs.readFileSync`, `execSync`, busy-wait loops, and any other synchronous call that stalls the host event loop — is forbidden, because the loom interpreter runs as plain async TypeScript on Pi's shared event loop (see [`../spec_topics/implementation-notes.md`](../spec_topics/implementation-notes.md)) and a synchronous stall blocks every concurrent invocation. **Enforcement posture.** The statically named blocking forms are mechanically caught: the `no-restricted-syntax` blocking-runtime check wired by [`H2a`](./H2a-cross-cutting-gates.md) flags **every call whose callee name ends in the literal suffix `Sync`** — whether a member access such as `fs.readFileSync` or a bare identifier such as `execSync` (those two are examples of the recognised form, not a finite set standing alongside an undefined residual family) — over `src/**` (excluding `**/*.test.ts`), permitting a call only where its line carries a `// allow-sync: <reason>` comment, alongside the `no-restricted-syntax` allow-list for the four Promise-combinator forms above. Only the residue this static scan cannot witness — busy-wait loops and dynamically-constructed sync calls — rests on the seam-adapter discipline (file/process I/O routed through the `V8*` FileSystem seam) and the *Per-phase TDD ritual* self-review step's blocking-call check together with the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) (checklist item 3), the same named manual-review surface the *No globals, statics, singletons* rule routes its scan residue to.
- **No silent test skipping.** Vitest's `assert.fail` / `expect.unreachable` (the test runner and assertion API [`H1a`](./H1a-scaffold-and-toolchain.md) provisions) when prerequisites are missing — never silent `return` early. A skipped test is a lie: it reports success when nothing was verified.
- **Spec drift.** If implementation reveals the spec is wrong, ambiguous, or under-specified, **stop**, fix the spec first in a dedicated commit, then resume.
- **Doc updates.** After each leaf, update `README.md`'s status table and append a one-line dated entry to `CHANGELOG.md`. The plan itself is updated only when the **plan** changes; non-plan discoveries go to `notes.md`. Each of these three artifacts is **created on first use if absent** — the first leaf that needs one creates it in the shape below, then writes its entry; do not pre-create them ahead of real content. Shapes: the `README.md` status table is a markdown table with columns `Leaf | Status | Date | Notes`, one row per completed leaf (replacing the current prose `## Status` section on first write); each `CHANGELOG.md` entry is a single line of the form `YYYY-MM-DD <leaf-id> — <one-line summary>`, newest at the bottom; `notes.md` is a free-form running log of non-plan discoveries with no fixed schema. These three artifacts live at the repository root, outside `docs/`; the orchestrator's `commitAddPaths` (`.pi/project-config.md`) auto-stages only `docs/`, so the leaf stages `README.md`, `CHANGELOG.md`, and `notes.md` explicitly (`git add`) as part of its commit. **Enforcement posture.** Unlike the cross-cutting rules backed by a mechanical surface — *Specific exception types only* (`no-broad-catch` lint), *Sequential by default* (`no-restricted-syntax` allow-list), *REQ-ID discipline* and *Diagnostic message anchors* (loom 1.0 closing-gate checks) — this rule has **no mechanical gate**: no lint rule, architectural test, or closing-gate check inspects `README.md`, `CHANGELOG.md`, or `notes.md`. It is contributor-discipline, verified at the *Per-phase TDD ritual* self-review step above (which checks the per-leaf doc artifacts were written) and, where one runs, the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) (checklist item 4) — the same named manual-review surface the *No globals, statics, singletons* rule routes its scan residue to, not a new parallel gate.
- **Diagnostic message anchors.** Tests asserting a diagnostic's rendered message MUST cite the diagnostic code and source the expected string from the *Message* column of the [diagnostics registry](../spec_topics/diagnostics.md). Phrasings like "matches spec verbatim" without a code anchor are insufficient — the registry is the single source of truth for every author-visible message string.
- **REQ-ID discipline.** Per-prefix and per-ID governance is owned by [`../spec_topics/governance.md`](../spec_topics/governance.md): rows are immutable and append-only, prefixes are word-boundary-disjoint identifier tokens, the union of live and retired prefixes covers every prefix that appears in `spec_topics/*.md`, and mutations follow the procedures pinned there. An *executable spec REQ-ID* is a numbered `PREFIX-N` REQ-ID in the [`governance.md`](../spec_topics/governance.md) prefix table that denotes a runtime/parse/load obligation the loom implements — every numbered REQ-ID except the corpus-governance `GOV-*` family, which governs the spec corpus itself rather than runtime behaviour and maps to no runtime leaf. The set is the numbered `PREFIX-N` surface only: a registered prefix that anchors no live REQ-ID (e.g. `CEIL`) contributes nothing to it, and the `loom/{parse,load,runtime}/*` diagnostic-registry codes (including warning-severity `W` codes) and the code-keyed obligation areas are separate closing-gate surfaces — described below — not members of the executable-REQ-ID set. A **GOV-16 stable-inline-label family** (e.g. `HC3`, `NOCEIL`, registered in the inline-label prefix table on [`governance.md`](../spec_topics/governance.md)) is likewise **not** an executable spec REQ-ID and contributes nothing to the executable-REQ-ID set: it carries no coverage-matrix mapping obligation and is exercised by none of the three executable-/code-keyed closing-gate surfaces below — it is neither a numbered `PREFIX-N` REQ-ID requiring a coverage-matrix mapping and citing test, a `loom/...` registry code, nor a code-keyed obligation area. The one closing-gate surface that does reach an inline-label prefix is the per-prefix numbering-hole check defined below, which already covers inline-label prefixes per [GOV-16](../spec_topics/governance/stable-inline-labels.md#gov-16). A stable-inline-label family closes in one of two shapes, as the [`coverage-matrix.md`](./coverage-matrix.md) `CIO` / `HC3` / `NOCEIL` / `CEIL` closure rationale names for each family — that rationale owns which family takes which shape (do not restate its per-label leaf mappings here): **(a)** *per-label single-leaf closure* — a single authoring leaf closes the family via that leaf's *Tests* bullets, one bullet per inline label, each citing its inline label inline (e.g. `HC3-a: ...`) under the same code-citation discipline this rule imposes on numbered REQ-IDs; or **(b)** *distributed-seam closure for no-single-leaf families* — a family with no single closing leaf is witnessed by diagnostic-code *Tests* bullets distributed across several leaves, each carrying the inline label as a parenthetical seam annotation (e.g. `loom/parse/timeout-field-rejected … (NOCEIL-1 seam)`), with the family's cross-cutting closure carried by the [`governance.md` GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) release-time corpus-review obligation rather than a runtime leaf. Either shape closes under the same code-citation discipline and the family adds no coverage-matrix row of its own. New diagnostic sites added by future spec work MUST land their REQ-IDs in the same edit. The loom 1.0 closing gate treats an **executable** spec REQ-ID without a coverage-matrix mapping, a retired-ID clashing with a live-ID, or a per-prefix numbering hole as a CI failure. A *per-prefix numbering hole* on a page is a positive integer `n ≤ max(live ∪ retired)` for some prefix the page owns — either a REQ-ID prefix per GOV-1 or an inline-label prefix per [GOV-16](../spec_topics/governance/stable-inline-labels.md#gov-16) — whose `n` appears in neither the page's live table for that prefix nor its retirement table for that prefix (the `## Retired REQ-IDs` section for REQ-ID prefixes per GOV-8, or the `## Retired inline labels` section for inline-label prefixes per GOV-16 *Retirement section*). Gaps that the retirement table accounts for are not failures; the gate only fires on numbers that are neither live nor retired. A test asserting a numbered REQ-ID MUST cite that REQ-ID inline in the test source — the same code-citation discipline the *Diagnostic message anchors* rule imposes on diagnostic codes — and the closing gate treats a [`coverage-matrix.md`](./coverage-matrix.md)-mapped numbered REQ-ID with no citing test in the test corpus as a CI failure, bringing the numbered-REQ-ID surface to the registry-code↔asserting-test parity stated below. The *coverage-matrix-mapped numbered REQ-ID* set this arm iterates is the matrix mappings — including each member expanded from a [`coverage-matrix.md`](./coverage-matrix.md) `X-n … X-m` range — intersected with the **live** executable REQ-ID set, mirroring the unmapped-REQ-ID direction that enumerates the live executable set: a retired interior ID accounted for by the prefix's `## Retired REQ-IDs` table is excluded, carries no citing-test obligation, and never reddens the mapped-REQ-ID-with-no-citing-test arm. The exclusion is confined to this citing-test direction; a number neither live nor retired remains a per-prefix numbering hole as defined above. This citing-test reconciliation is a best-effort scan for the inline `PREFIX-N` citation across the test sources: it certifies that a citing test exists for each mapped REQ-ID, not that the cited test's assertion is semantically faithful to the obligation — faithfulness stays a TDD and self-review obligation under the per-phase ritual. Per-REQ-ID citing-test existence likewise does not certify per-facet completeness for a multi-leaf [`coverage-matrix.md`](./coverage-matrix.md) row whose closing leaves each close a distinct facet of the obligation: one citing test anywhere in the corpus satisfies this scan for the whole REQ-ID even when one facet's closing leaf carries no facet-naming citing test, so that split-row joint-completeness residue is routed to the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) (checklist item 7). Like the unmapped-REQ-ID surface it is exercised against the seeded fixtures at [`H5a`](./H5a-closing-gate-automation.md) and binds against the live test corpus only at the loom 1.0 release gate activated by [`H6a`](./H6a-live-corpus-activation.md). The *loom 1.0 release gate* is the loom 1.0 closing gate on its live-corpus footing — one gate in two footings, not two gates — activated by [`H6a`](./H6a-live-corpus-activation.md), which owns the equivalence. Diagnostic codes carry the same closing-gate discipline: a new code added to the registry table of [`../spec_topics/diagnostics.md`](../spec_topics/diagnostics.md) MUST land its asserting test in the same edit; a registry code without an asserting test (and any asserted code not in the registry) is a CI failure. A third closing-gate surface covers the residue these two leave behind: a normative MUST/MUST-NOT on a non-narrative `spec_topics/**` page that carries neither a numbered `PREFIX-N` REQ-ID nor a `loom/...` diagnostics-registry code is the un-anchored-obligation class [`governance.md` GOV-22](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-22) names. Each such obligation MUST be enumerated in [`coverage-matrix.md`](./coverage-matrix.md) under *Code-keyed obligation areas (no numbered REQ-IDs)* with a named closing leaf, and the closing gate treats an un-anchored normative MUST/MUST-NOT with no enumerated closing leaf as a CI failure on the same live-corpus footing as an unmapped executable REQ-ID — the un-anchored normative-MUST text-scan arm [`H5e`](./H5e-un-anchored-must-gate.md) owns, exercised against the seeded fixtures at [`H5e`](./H5e-un-anchored-must-gate.md) and binding against the live `spec_topics/**` corpus only at the loom 1.0 release gate. A [`coverage-matrix.md`](./coverage-matrix.md) `<new>` placeholder row — one whose closing-leaf column holds exactly the literal bare `<new>` token and so names no real leaf yet (the placeholder convention defined in that file's *Code-keyed obligation areas* section, which pins the recogniser: the placeholder is exactly that one literal token, and any other closing-leaf token not resolving to an existing plan leaf ID reddens the gate as a defect rather than reading as a benign placeholder) — does **not** satisfy the "enumerated with a named closing leaf" condition: an un-anchored MUST whose only coverage-matrix entry is a `<new>` placeholder row is treated identically to one absent from the table altogether, because the placeholder names no real closing leaf and no asserting test. On this live-corpus footing the gate therefore reddens for any un-anchored MUST mapped only to a `<new>` row at the loom 1.0 release gate, while a seeded mid-authoring `<new>` placeholder stays green until the [`H6a`](./H6a-live-corpus-activation.md) activation flips this footing to hard-fail. The gate's recogniser is a best-effort `MUST`/`MUST NOT`-token text scan over non-narrative `spec_topics/**` pages — a `spec_topics/**` page is *non-narrative*, and thus in scope, iff its row in the [REQ-ID prefix table](../spec_topics/governance/req-id-prefix-table-active-a.md) (surfaced through [`governance.md`](../spec_topics/governance.md)) carries a cell that is **not** the byte-exact `(no IDs — narrative)` literal per [`governance.md` GOV-3](../spec_topics/governance/req-id-prefix-table-active-a.md#gov-3); this is the same byte-exact comparison GOV-3 mandates for REQ-ID extraction (and the same exclusion the *Leaf format* *Tests* bullet cites for leaf-citation scope), so the recogniser's page-scope and the REQ-ID extractor's page-scope share one definition — that excludes obligations already carrying a `PREFIX-N` REQ-ID or a `loom/...` registry code. This iff is two-valued only over pages that *have* a row; a third branch governs a `spec_topics/**` page absent from the table altogether (no row at all — neither a non-narrative prefix cell nor the `(no IDs — narrative)` literal). Because no governance rule makes the table exhaustive over `spec_topics/**` ([`governance.md` GOV-6](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-6) requires only that every prefix *appearing in a REQ-ID* be in the table), such un-rowed pages exist; an un-rowed page is **in scope** and reddens the gate as an un-rowed-page residue defect, unless it is a [`governance.md` GOV-24](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-24) table-of-contents hub stub — a subtree TOC file that deliberately carries no row and no REQ-ID anchors — which is excluded. The spec-side root-cause fix (a page-completeness invariant alongside GOV-6) is owned by the relevant spec-coverage finding, not this plan rule, which only reddens the gate so an un-rowed page is surfaced rather than silently skipped. A `loom/typecheck/*` build-time `tsc` brand string is **not** a diagnostics-registry code and does not anchor a normative obligation for this recogniser — the same carve-out [`H5a`](./H5a-closing-gate-automation.md) applies to registry reconciliation, restated here so the open `loom/...` form does not silently re-admit the `loom/typecheck/*` prefix as an anchor. Like the H3a identifier-keyed ambient-access scan, it catches only what the token scan witnesses: it cannot mechanically catch obligations phrased without the literal `MUST`/`MUST NOT` token (e.g. "shall", "is required to", lowercase, or split-across-table-cell phrasings), cannot distinguish a normative MUST from one inside narrative, an illustrative example, or a quoted passage. The gate does not mechanically exclude named cross-leaf seam obligations: it has no enumerated seam-name registry to read, so a MUST that is genuinely a Class-2 cross-leaf seam contract surfaces as a gate candidate and must be either enumerated in [`coverage-matrix.md`](./coverage-matrix.md) with a closing leaf or dispositioned as seam-name-dependent residue in the GOV-15 reviewer inspection. That residue — token-less obligations, narrative-ambiguous obligations, and seam-name-dependent obligations — is caught by the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) (checklist item 5), the plan-side release-time review that runs alongside the spec's [`governance.md` GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) source-language-equivalence reviewer-inspection step, not by the mechanical gate. Re-anchoring any individual MUST as a REQ-ID is a spec-side GOV-22 decision owned by the relevant spec-coverage finding, not by this plan rule, which only forces each un-anchored obligation onto a closing leaf. **Transitive-completeness plan-maintenance.** Whenever a new leaf is added whose `Tests.` bullets carry an executable REQ-ID, the citing test that *closes* a [`coverage-matrix.md`](./coverage-matrix.md)-mapped numbered REQ-ID (i.e. the test on the leaf the coverage matrix lists as that REQ-ID's closing leaf — the trigger turns on the coverage-matrix closing-leaf mapping, not on whether the citation is the REQ-ID's first textual occurrence, so a leaf that merely re-cites an already-closed REQ-ID does not qualify), or an un-anchored normative MUST, that leaf MUST be added to [`H5b`](./H5b-warn-only-canary.md)'s `Deps.` — the leaf that owns the complete coverage-producing dependency set — so the warn-only canary and the [`H6a`](./H6a-live-corpus-activation.md) hard-fail release gate (which inherits the set transitively through `H5b` rather than restating it) both stay sequenced after every coverage-producing leaf, otherwise the canary pre-flights — and the gate activates — against incomplete coverage. This obligation is mechanically backed: [`H5d`](./H5d-transitive-completeness-gate.md)'s standing transitive-completeness gate arm reconciles every [`coverage-matrix.md`](./coverage-matrix.md) row's right-hand closing-leaf column (reading no leaf-ID tokens from the left-cell obligation-area prose, and excluding the `<new>` placeholder rows whose closing-leaf column lists no real leaf yet) against the expanded membership of [`H5b`](./H5b-warn-only-canary.md)'s `Deps.` and fails CI on any closing-leaf cell none of whose listed leaves is in it — a multi-leaf (primary + co-witness) cell such as `CIO-5 | V16a, H7a` stays green as long as one listed leaf is present, so a co-witness leaf intentionally excluded from `H5b`'s `Deps.` does not redden the arm. The arm is therefore a **per-cell at-least-one** test, not an every-genuine-closer test: it reddens only for a closing-leaf cell *none* of whose listed leaves reaches `H5b`'s `Deps.`, but it stays green for a genuine multi-facet split row — where each listed leaf closes a distinct facet (e.g. `ERR-17 | V4d, V13d`, `DISC-4 | V10a, V9m`) and this rule demands *all* of them in `H5b`'s `Deps.` — as soon as any one listed leaf is present, so it cannot catch a secondary facet-closer omitted while a sibling facet-closer for the same cell remains. That every-facet-closer residue is surfaced not by this arm but by [`H5b`](./H5b-warn-only-canary.md)'s warn-only canary and the [loom 1.0 release-time residue inspection](#release-time-residue-inspection) (checklist item 7, split-row joint-completeness).

## Loom 1.0 release-time residue inspection

The cross-cutting rules above each concede a residue their mechanical scans cannot witness and route it to a single named manual-review surface, the **loom 1.0 release-time residue inspection** enumerated here. This is a **plan-side** review surface, distinct from the spec's [`governance.md` GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) reviewer-inspection step: that spec step verifies cross-release source-language equivalence (return values, ordered diagnostic-code sequences, `loom-system-note` content) and the ceiling-set / diagnostic-registry carve-out invariants against the inter-release diff, and enumerates none of the implementation-residue classes below. loom's residue inspection runs at the loom 1.0 release gate as the plan-side superset that additionally sweeps the conceded mechanical-scan residue this page routes to it; the two reviews share the release-time occasion but not their checklists. Mechanising or re-anchoring any individual class stays owned by the cross-cutting rule that concedes it, not by this section.

The inspection is a reviewer reading `src/**` — and, for item 4, the repository-root doc artifacts; for item 5, the `spec_topics/**` corpus; for item 7, [`coverage-matrix.md`](./coverage-matrix.md) and the test corpus; for item 8, `extensions/index.ts`; for item 9, the [`H3a`](./H3a-di-seam-skeleton.md) ambient-primitive scan's timing-primitive list and the [`host-interfaces-services.md`](../spec_topics/pi-integration-contract/host-interfaces-services.md) `Clock`/`WallClock` ban — against the checklist below. Each item names what the reviewer looks for and the condition under which the item is clean; a residue defect found at any item blocks the loom 1.0 release until fixed.

1. **Closure-captured / lazy-cache / DI-container singletons** (*No globals, statics, singletons*). Read `src/**` for cross-invocation mutable state held by closure capture, by a lazily-populated module cache, or by a DI-container singleton registration — the singleton forms the [`H2a`](./H2a-cross-cutting-gates.md) architectural test does not detect. **Clean when** every stateful collaborator is constructor-injected and no module-level binding retains cross-invocation mutable state outside an injected seam.
2. **Indirect ambient-primitive reads** (*No globals, statics, singletons*). Read `src/**` for a banned ambient primitive reached through an alias (`const env = process.env`), a destructure (`const { cwd } = process`), computed access (`process["env"]`), a helper wrapper, or re-export indirection — the indirect reads the [`H3a`](./H3a-di-seam-skeleton.md) identifier-keyed scan does not witness. **Clean when** every banned-primitive read outside its seam adapter is either absent or carries the `// allow-ambient: <primitive> — <seam>` comment the scan already admits.
3. **Blocking-runtime calls** (*Sequential by default*). Read `src/**` (excluding `**/*.test.ts`) for synchronous calls that stall Pi's shared event loop that the [`H2a`](./H2a-cross-cutting-gates.md) blocking-runtime check cannot witness — busy-wait loops, dynamically-constructed sync calls, or any other synchronous stall whose callee name does not end in the literal suffix `Sync`; that check mechanically catches every call whose callee name ends in the literal suffix `Sync` (a member access such as `fs.readFileSync` or a bare identifier such as `execSync`) (and the `no-restricted-syntax` rule the four Promise-combinator forms), permitting a flagged call only where its line carries a `// allow-sync: <reason>` comment, leaving only this residue to the reviewer. **Clean when** all file/process I/O is routed through the `V8*` seam — each such seam call carrying its `// allow-sync: <reason>` comment — and no synchronous event-loop stall remains in `src/**`.
4. **Doc-update artifacts** (*Doc updates*). For each completed leaf, confirm its `README.md` status-table row, its dated `CHANGELOG.md` line, and (for a non-plan discovery) its `notes.md` entry exist in the shapes the *Doc updates* rule defines; no lint, architectural test, or closing-gate check inspects these files. **Clean when** every completed leaf has its required doc artifacts present and well-formed.
5. **Un-anchored-MUST residue** (*REQ-ID discipline*; [`governance.md` GOV-22](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-22)). Sweep the non-narrative `spec_topics/**` pages — together with any non-hub-stub `spec_topics/**` page absent from the [REQ-ID prefix table](../spec_topics/governance/req-id-prefix-table-active-a.md) altogether, which the row-keyed recogniser does not classify (a [`governance.md` GOV-24](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-24) hub stub is excluded from this sweep) — for normative obligations the best-effort `MUST`/`MUST NOT` token scan cannot witness — token-less phrasings (`shall`, `is required to`, lowercase, split-across-table-cell), obligations the scan cannot distinguish from narrative / illustrative / quoted passages, and seam-name-dependent Class-2 obligations the gate has no registry to exclude. **Clean when** every such obligation is accounted for: either enumerated in [`coverage-matrix.md`](./coverage-matrix.md) under *Code-keyed obligation areas (no numbered REQ-IDs)* with a named closing leaf — and, for each such row whose obligation carries neither a numbered `PREFIX-N` REQ-ID nor a `loom/...` diagnostics-registry code (the pure-prose subset; a code-keyed row anchored by a `loom/...` registry code is already covered by the diagnostic-code↔asserting-test parity gate and is not double-gated here), the reviewer confirms that named closing leaf carries a test asserting the obligation's MUST/MUST-NOT, not merely that the row names a leaf — or recorded as dispositioned residue here; and every swept un-rowed page is either given a prefix-table row by the spec owner or recorded as dispositioned residue here. Re-anchoring an individual MUST as a REQ-ID is a spec-side GOV-22 decision owned by the relevant spec-coverage finding, not a step in this inspection.
6. **Broad-catch category membership** (*Specific exception types only*). Read `src/**` for each `// allow-broad-catch:` exempt site and confirm it falls within one of the five enumerated exempt-broad-catch-site categories — (1) Pi SDK boundary, (2) *Self-failure* probe traps and hostile-import reads, (3) *Unknown-reason rule* `event.reason` property-access read, (4) *Per-step isolation* `session_shutdown` sub-step wraps, and (5) *Diagnostic-emission isolation* `console.error` wraps; the [`H2a`](./H2a-cross-cutting-gates.md) `no-broad-catch` lint and the loom 1.0 closing-gate token-resolution arm verify only that the comment's cited token resolves, not that the catch sits at one of these categories. **Clean when** every exempt broad-catch site sits at one of the five enumerated categories.
7. **Split-row joint-completeness** (*REQ-ID discipline*). Sweep every closing-leaf cell in [`coverage-matrix.md`](./coverage-matrix.md) — across **both** the *Numbered REQ-IDs* table and the *Code-keyed obligation areas (no numbered REQ-IDs)* table — whose closing-leaf cell lists two or more leaves, deriving the row's **facet partition** from the cell alone — with no recourse to the [`coverage-matrix.md`](./coverage-matrix.md) *`CIO` / `HC3` / `NOCEIL` / `CEIL` closure rationale* prose — by this rule: a listed leaf whose **immediately-following parenthetical annotation contains the literal token `co-witness`** (e.g. `CIO-5 | V16a, H7a (co-witness — cross-site never-interleaves integration witness)`, where `H7a` is the co-witness, leaving the single facet `V16a`) is a **co-witness** and is dropped from the partition — the primary + co-witness shape [`H5d`](./H5d-transitive-completeness-gate.md) reads. Every other listed leaf is a **facet**, whether it is bare or carries a non-`co-witness` parenthetical *facet annotation* naming the facet it closes (e.g. `DISC-4 | V10a (collision-detection closure), V9m (superseded-entry-dispatch closure)`, the `PIC-17, PIC-18` row's `V15d` annotation, or the `cka-33` swallowing-handler row's per-site annotations); the two parenthetical forms read as opposite classifications — a co-witness marker drops its leaf from the facet set, a facet annotation keeps it in. The co-witness marker binds to the single backtick-delimited leaf it immediately follows, so a cell with two co-witnesses marks each one (e.g. `CIO-1 | V16a, V4e (co-witness — …), V11f (co-witness — …)`, both `V4e` and `V11f` dropped, leaving the single facet `V16a`). Because [`H5d`](./H5d-transitive-completeness-gate.md) tokenises a cell by its backtick-delimited spans only and drops the parenthetical prose, neither the co-witness marker nor the facet annotation alters the leaf-ID set [`H5d`](./H5d-transitive-completeness-gate.md) and the H5a per-REQ-ID citing-test scan read. For a *Numbered REQ-IDs* row the H5a per-REQ-ID citing-test scan certifies only that one citing test exists somewhere for the mapped REQ-ID, and the [`H5d`](./H5d-transitive-completeness-gate.md) transitive-completeness arm requires only one listed leaf to be reachable, so neither arm witnesses a facet whose closing leaf carries no citing test of its own; a split REQ-ID can ship with one facet fully tested and another carrying no citing test at all with both gates green. A *Code-keyed obligation areas* row is reached by no stronger gate: the [`H5d`](./H5d-transitive-completeness-gate.md) transitive-completeness arm is the same per-cell at-least-one check, the [`H5e`](./H5e-un-anchored-must-gate.md) un-anchored-MUST gate verifies only that the obligation is enumerated with a named closing leaf, and the diagnostic-code↔asserting-test parity gate either does not apply (a swallowing-handler row such as `cka-33` carries no diagnostic code) or accepts one asserting test per shared code, so a code-keyed facet — e.g. `cka-33`'s subagent-`AgentSession.abort()` suppression closing at `V9o` — can likewise ship with no citing test while every gate stays green, the *No silent test skipping* failure mode the plan forbids. **Clean when** every facet of every multi-leaf row in either table carries its own distinct facet-naming citing test on that facet's closing leaf — a row where one facet is tested and another facet's closing leaf carries no citing test is a residue defect.

8. **Entry-shim purity** (*Cross-cutting rules scope*). Read `extensions/index.ts` — the Pi extension entry shim ([`H4a`](./H4a-factory-shell-and-harness.md) authors it), which sits outside `src/**` and is therefore read by none of the `src/**`-scoped mechanical gates (the [`H2a`](./H2a-cross-cutting-gates.md) no-globals test, the `no-broad-catch` and `no-restricted-syntax` lint rules, the [`H3a`](./H3a-di-seam-skeleton.md) ambient-primitive scan, and the [`H5a`](./H5a-closing-gate-automation.md) / [`H5c`](./H5c-broad-catch-allow-list-gate.md) allow-list reconciliation) — and confirm it remains a pure re-export of the `src/**` factory carrying no broad `catch`, no global / static / singleton, no ambient-primitive read, and no `Promise` combinator of its own. **Clean when** the entry shim re-exports the `src/**` factory and holds none of these rule-subject constructs.

9. **Ambient timing-primitive list reconciliation** (*No globals, statics, singletons*; [`host-interfaces-services.md` PIC-12](../spec_topics/pi-integration-contract/host-interfaces-services.md#pic-12)). Re-read the [`H3a`](./H3a-di-seam-skeleton.md) ambient-primitive scan's enumerated timing-primitive list — `Date.now`, `performance.now`, `Date.prototype.getTime`, `setTimeout`, `clearTimeout` — against the current [`host-interfaces-services.md`](../spec_topics/pi-integration-contract/host-interfaces-services.md) `Clock`/`WallClock` ban (PIC-12); the list is a manually-maintained derivation with no mechanical reconciliation, so a spec edit that extends the `Clock`/`WallClock` surface does not move it and nothing else flags the divergence. The list is required to be a superset of the spec ban (scan list ⊇ spec ban) and currently equals the PIC-12 enumeration `Date.now`/`performance.now`/`Date.prototype.getTime`/`setTimeout`/`clearTimeout`, so this item does not require strict equality with the spec ban. **Clean when** every primitive the current spec ban enumerates is covered by the scan's list (the scan list ⊇ the spec ban) and any scan-list entry beyond the spec ban is a deliberate superset addition recorded here. Re-anchoring or widening the spec's `Clock`/`WallClock` ban is a spec-side PIC-12 decision owned by the relevant spec-coverage finding, not a step in this inspection.
