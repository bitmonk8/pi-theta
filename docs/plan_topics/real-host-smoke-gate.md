# Manual real-host smoke gate (release-gate / version-bump governance)

A cross-cutting governance note, not a leaf of work — it carries no `H<n><letter>`/`V<N><letter>` leaf ID and closes no spec REQ-ID. It is the home of the **manual real-host smoke governance protocol** that the [`H4a`](./H4a-factory-shell-and-harness.md) end-to-end harness, the [`H7a`](./H7a-integration-acceptance.md) integration-acceptance fixture, the [`H6a`](./H6a-live-corpus-activation.md) release-gate auditor, and the [`V18d`](./V18d-version-bump-acceptance.md) version-bump revert path all reference.

This note defines the **manual real-host smoke governance protocol** — a non-automated pre-merge gate that never runs under `npm test`. Its audience is the bump/merge contributor and the [`H6a`](./H6a-live-corpus-activation.md) release-gate auditor, not the [`H4a`](./H4a-factory-shell-and-harness.md) implementer who builds the factory shell, end-to-end harness, and session-double fidelity contract.

The concrete pre-merge gate for a double-vs-real divergence is a **manual real-host smoke run** that drives [`H7a`](./H7a-integration-acceptance.md)'s committed multi-feature fixture `.loom` against a live Pi host instead of the in-process double — the same fixture `.loom` `H7a` drives against the double, reused here so the smoke and the integration-acceptance gate never diverge into separate fixtures. **Activation window.** Both triggers are dormant until [`H7a`](./H7a-integration-acceptance.md) has landed: the smoke run is defined entirely in terms of two `H7a`-owned inputs — its committed multi-feature fixture `.loom` and the committed permitted-code list criterion (e) scores against — and neither exists in the `H4a`→`H7a` window. A trigger event (a Pi version bump under trigger (1), or a fidelity-axis-touching merge under trigger (2)) that occurs before `H7a` lands is therefore **deferred**, not scored against the absent inputs: its owner records the deferral in the triggering commit message (the same evidence record the recorded run uses) and re-runs the smoke once `H7a` has landed, rather than treating the missing fixture or permitted-code list as a criterion failure or a behavioural-divergence finding. A window-deferral is an **un-runnable-gate condition** of the same family as the unsatisfied-precondition case the Live-host precondition below defines — it does NOT score as a fail and so does not spuriously block the triggering change from merging. **Live-host precondition.** The smoke can only run against a live Pi host that satisfies the host-side prerequisites [`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) already enumerates: (1) a Pi install at the pinned SDK range ([`host-prerequisites.md#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)); (2) a structured-output-capable binder model resolvable through `ctx.modelRegistry` ([`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) item 2), since criterion (b) below — a structurally-valid binder pass — is unsatisfiable without one; and (3) credentials Pi already resolves for that model ([`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) item 3 — loom stores none). The smoke's two named owners (the bump contributor for trigger (1) and the merging contributor for trigger (2), below) each run the gate against their own Pi install configured to satisfy this precondition. An unsatisfied binder-model or credential precondition — most concretely, no structured-output-capable binder model registered, which load-fails with `loom/load/binder-model-unresolved` — is an **un-runnable-gate condition, not a behavioural-divergence finding**: the owner provisions the missing prerequisite and re-runs, and a missing or uncredentialed binder model does NOT score as the criterion (a) failure below (and so does not spuriously block the merge). **Pass/fail criterion:** a live, non-deterministic LLM — not `H7a`'s deterministic in-process double — drives the fixture here, so the run is scored on model-output-invariant observables, not on exact reproduction of `H7a`'s double-recorded goldens (which a real model on this multi-feature pipeline is not assumed to reproduce in exact turn order/count or exact emitted-code set). The live-host run **passes** iff all of the following hold: (a) the run completes without error — no thrown exception, no aborted pipeline; (b) the binder pass produces output that validates against the binder envelope schema ([`V11c`](./V11c-bypass-envelope.md); [`binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params`](../spec_topics/binder/binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params)) — the run parses and conforms to the three-arm `ok | needs_info | ambiguous` envelope schema, a structural-validity check, not an exact-value match against `H7a`'s deterministic goldens; (c) cancellation propagation is observed at the injected cancel point; (d) the four session-double fidelity-contract axes — streamed-token order relative to `ctx.waitForIdle()` resolution, each streamed assistant response is appended as exactly one prompt-mode turn (a per-response invariant, independent of total turn count), the `pi.on` cancel-forward subscription, and cancellation propagation — hold; and (e) the emitted `loom-system-note` codes are a **subset** of the **committed permitted-code list** checked in alongside [`H7a`](./H7a-integration-acceptance.md)'s fixture `.loom` — the single committed, reviewed reference set: a manually-maintained, best-effort union of the `loom/...` codes the slices in `H7a`'s **Deps** can emit, intended as a superset of `H7a`'s committed golden diagnostics list (only that `golden ⊆ permitted` containment is mechanically gated, by `H7a`; the union's completeness is not mechanically verified, so a stale union can omit a benign code a slice emits) — so the runner scores (e) against one fixed committed reference rather than reconstructing the union by hand. A failure of any one of (a)–(e) — a thrown or aborted run, binder output that fails to validate against the binder envelope schema ([`V11c`](./V11c-bypass-envelope.md)), unobserved cancellation, a violated fidelity-contract axis, or an emitted code outside the permitted set — is a **fail** (a confirmed behavioural-divergence finding), so a genuine divergence still produces a falsifiable fail that drives the revert / merge-block path below; benign live-model variance in turn order, turn count, or which permitted codes fire is **not** a fail. `H7a`'s in-process gate keeps its exact-match goldens unchanged — against the deterministic double, exact-transcript and exact-code-set equality is the correct criterion. The smoke is a required pre-merge gate on two independent triggers, each with a named owner who MUST run it and record the result — in the triggering change's commit message (the bump commit for trigger (1), the merge commit for trigger (2)), with no separate artefact file, the same manual-run-evidence convention the spec fixes for the version-bump path ([`version-bump-triggers.md`](../spec_topics/pi-integration-contract/version-bump-triggers.md)'s "produces no separate artefact file"; [`version-bump-step2.md`](../spec_topics/pi-integration-contract/version-bump-step2.md)'s outcomes "in the bump commit message"), so the [`H6a`](./H6a-live-corpus-activation.md) release-gate evidence record locates the run the same way wherever the smoke fires — before the change merges: (1) each Pi version bump — owned by the contributor performing the bump, who runs the smoke against the candidate Pi-SDK pin before the bump merges, under the [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure); and (2) any merge whose diff **touches the four fidelity-contract axes** — owned by the contributor of that merge, who runs the smoke before the merge lands — — observably, a commit that edits either the in-process double's modelling of, or the production seam realising, any of (i) streamed-token order relative to `ctx.waitForIdle()` resolution, (ii) single-turn prompt-mode append semantics, (iii) the `pi.on` cancel-forward subscription, or (iv) cancellation propagation — so a divergence introduced by a loom change rather than a Pi bump is caught before it reaches `main`. This pre-merge step is a **contributor obligation, not a CI-enforced gate**: loom 1.0 has no mechanical hook that observes whether the smoke ran, and the trigger-(2) condition (a diff touching the four fidelity-contract axes) has no automatic diff detector — recognising it is itself part of the contributor's obligation. Consequently a skipped run, or a divergence whose author does not recognise their diff as touching the four axes, is **not blocked from merging**; such an omission is detectable only **after the fact**, by reading the triggering commit message for the recorded run (the same evidence record [`H6a`](./H6a-live-corpus-activation.md)'s release gate audits), not at merge time. A confirmed behavioural-divergence finding from that smoke is a **merge blocker**: a Pi-bump-triggered finding blocks the bump from merging — the prior Pi-SDK pin is restored before merge (see [`V18d`](./V18d-version-bump-acceptance.md)'s revert path, which owns the pre-merge pin-restore) — and a four-axes-change-triggered finding blocks the merge until the divergence is resolved; in neither case is it merely an editorial-review note.
