# H1 — Repository scaffold and test framework

**Spec.** None — infrastructure leaf; no normative spec page. (Cross-cutting rules enforced here — no globals/statics/singletons, no broad catch — are defined in [`conventions.md`](./conventions.md), not in `spec_topics/`.)

**Convention.** [Cross-cutting rules](conventions.md#cross-cutting-rules-every-phase) — "No globals, statics, singletons" (architectural test), "Specific exception types only" (ESLint rule), "Sequential by default", "No silent test skipping". Project hygiene (TypeScript strict mode, Vitest, Prettier, GitHub Actions, npm scripts, source layout) trace to project-level conventions, not to any spec rule.

**Adds.** TypeScript project (strict mode, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`); Vitest + coverage; ESLint with `@typescript-eslint`, no-floating-promises, no-globals, no-broad-catch (with a per-site allow-list of `<file>:<line-range>` Pi SDK boundary sites where each entry requires a same-line `// allow-broad-catch: <REQ-ID> — <spec-page>` comment, per the Pi SDK boundary exception in the Specific-exception-types rule of [`conventions.md`](./conventions.md); the canonical seed entries — `sendSystemNote` wraps of `pi.sendMessage` and `ctx.ui.notify` (H4, [`pi-integration-contract.md`](../spec_topics/pi-integration-contract.md) — System notes); `PiSubagentSpawner` wrap of `AgentSession.dispose()` (V12a, same page — Subagent session lifecycle); top-level interpreter wrap (V18m); and `invoke` boundary wrap (V18n) both routed to `loom/runtime/internal-error` — are distinct entries; V18m and V18n MUST NOT collapse into one), no-restricted-syntax (banning `Promise.all`, `Promise.race`, `Promise.allSettled`, `Promise.any` in `src/**`, excluding `**/*.test.ts`, except at allow-listed call sites carrying a `// allow: <REQ-ID> — <spec-page>` same-line comment, per the Sequential-by-default rule in [`conventions.md`](./conventions.md)); Prettier; `depcheck` as a devDependency with a committed `.depcheckrc` (or `package.json` `depcheck` block) declaring Vitest, ESLint plugins, Prettier, and `@types/*` as known-used so the gate's real-world behaviour is what is being validated; npm scripts (`build`, `test`, `test:watch`, `typecheck`, `lint`, `lint:actions`, `format`, `depcheck`); GitHub Actions workflow file at `.github/workflows/ci.yml` with jobs `typecheck`, `lint`, `test`, `depcheck`; `actionlint` as a dev-dep + npm script (`npm run lint:actions`); project bootstrap files `CHANGELOG.md` (Keep-a-Changelog header: `# Changelog`, the standard "format based on Keep a Changelog" / "adheres to Semantic Versioning" links, and an empty `## [Unreleased]` section) and `notes.md` (`# Notes` header plus a one-line description of its purpose), created once at the project root so the per-leaf doc-update convention has stable targets.

**Source layout:**

test/{unit (mirrors src/), integration, fixtures/{loom, warp, schemas}, fakes/}

**Tests.**
- Sentinel test (`1 + 1 === 2`) proving toolchain runs.
- Per-directory presence test (`__present` re-exported by every `src/*/index.ts`).
- `no-static-state.test.ts` walks `src/**/*.ts` and inspects every module-top-level binding (any `VariableDeclaration` whose parent is `Program` or `ExportNamedDeclaration` directly under `Program`). Uses the TypeScript compiler API rather than line-anchored regex so that `export const`, `export let`, `export var`, and indented continuations are caught. Fails on:
    - any `let` or `var` at module top-level (regardless of `export`);
    - any `const` (including `export const`) whose initialiser is not one of the permitted forms below.

  Permitted initialiser forms for module-top-level `const`:
  1. A literal of primitive type — string, number (incl. `bigint`), boolean, `null`, `undefined`.
  2. A `RegExp` literal (`/.../flags`) without the `g` or `y` flag (stateful regex bindings are forbidden because `RegExp.lastIndex` is shared mutable state).
  3. A `TemplateLiteral` with no substitutions, or a string built from `+` of items each themselves on this list.
  4. An `as const` expression (`<expr> as const`) where `<expr>` is itself an array literal, object literal, or tuple of items each on this list — TypeScript's `as const` produces a deeply-readonly type, so downstream mutation is a type error.
  5. A direct `Object.freeze(<expr>)` call where `<expr>` is an object literal whose every property value is on this list, or an array literal whose every element is on this list. Shallow `Object.freeze` is acceptable because the contained values are themselves immutable by rules 1–4.
  6. A reference to an `enum`-like exported `const` from the same package whose declaration itself satisfies this list (computed transitively, single-pass — no need for full graph closure).

  Forbidden initialiser forms (non-exhaustive, listed for clarity): bare `[]` / `{}` / `new Map()` / `new Set()` / `new WeakMap()`; function or arrow-function expressions; `class` expressions; any call expression other than `Object.freeze(...)`; any identifier reference not covered by rule 6.

- Tests-for-the-test: a fixture directory under `test/fixtures/no-static-state/` containing `ok-*.ts` files that must pass and `bad-*.ts` files that must fail, one fixture per allow-list rule and one per forbidden form, asserted by running the checker against each fixture and comparing exit code and reported binding name.
- Workflow shape test: parses `.github/workflows/ci.yml` as YAML; asserts top-level `on.push` and `on.pull_request` triggers exist; asserts `jobs.typecheck`, `jobs.lint`, `jobs.test`, `jobs.depcheck` each exist and each contains a step whose `run` command invokes the matching npm script (`npm run typecheck`, `npm run lint`, `npm test`, `npm run depcheck` respectively).
- `depcheck` self-test: a fixture package under `test/fixtures/depcheck-unused/` declares an unused dependency; the test invokes `npx depcheck` against the fixture (using the same `.depcheckrc` shape committed at repo root) and asserts a non-zero exit and that the unused dep name appears in stdout. A second fixture under `test/fixtures/depcheck-clean/` with no unused deps asserts exit 0. This negative-fixture pair is load-bearing — without it, future leaves can drift `depcheck`'s configuration (e.g. a broad `ignorePatterns`) into a state where the gate passes vacuously and no one notices.
- Workflow lint test: `npm run lint:actions` (`actionlint`) exits 0 on the committed workflow; exits non-zero on a fixture workflow under `test/fixtures/ci-bad/` that contains an unknown job key. `actionlint` is a Go binary, not an npm package; install via `actionlint-installer` or pin a release archive in a `postinstall` script. The `lint:actions` script must fail loudly (non-zero exit, no silent skip) if the binary is absent — per the cross-cutting "no silent test skipping" rule in [`conventions.md`](./conventions.md).
- `no-broad-catch` ESLint rule fires on fixtures containing each of `catch (e)`, `catch (e: unknown)`, `catch (e: any)`, `catch (e: Error)`, and the rethrow-on-mismatch pattern (`catch (e) { if (!(e instanceof X)) throw e; … }`); passes on `catch (e: FooError)` for any user-defined subtype. The rule is restricted to `src/`; `test/` and explicit boundary modules listed in the lint config are exempt so that catching standard-library `Error` subtypes (`AbortError`, `TypeError` from JSON parsing) at well-defined boundaries remains possible. The exempt list lives in the committed lint config so it is reviewable. The rule additionally honours the per-site Pi SDK boundary allow-list (per the Pi SDK boundary exception in the Specific-exception-types rule of [`conventions.md`](./conventions.md)). Fixture tests assert: (i) `catch (e: unknown) // allow-broad-catch: PI-N — pi-integration-contract.md` passes when the call site is enumerated in the committed `<file>:<line-range>` allow-list; (ii) the same `catch` without the same-line comment fails the rule; (iii) the same `catch` with the comment but at a `<file>:<line-range>` not enumerated in the allow-list fails the rule; (iv) the rule still fires on an unmarked `catch (e: unknown)` placed inside a lint-config-exempt boundary module (per-site comment requirement applies inside exempt files too, so audit-by-grep stays exhaustive); (v) the transitional spec-page-anchor form (`// allow-broad-catch: per pi-integration-contract.md — System notes`) passes the rule and emits a deprecation diagnostic rather than a hard fail, mirroring the `Promise.all` rule's H6 transition posture. A separate gate-level fixture (deferred to V18s and stubbed here) asserts that an allow-list entry whose comment cites a REQ-ID absent from [`coverage-matrix.md`](./coverage-matrix.md) fails the V18s coverage gate. The committed allow-list is empty in H1 (no production call sites yet exist); the file shape and parser are wired so H4, V12a, V18m, and V18n can append the canonical seed entries (enumerated in `Adds.` above) without re-bootstrapping the rule.
- `no-restricted-syntax` ESLint rule fires on fixtures under `src/` containing each of `Promise.all(…)`, `Promise.race(…)`, `Promise.allSettled(…)`, and `Promise.any(…)`; passes on the same fixtures when the call site is listed in the committed allow-list (`<file>:<line-range>`) and carries a same-line comment of the form `// allow: <REQ-ID> — <spec-page>`; passes unconditionally under `**/*.test.ts`. A separate fixture asserts that an allow-listed call site lacking the comment fails the rule, and that a comment present on a non-allow-listed line is ignored. The committed allow-list is empty in H1 (no production call sites yet exist); the file shape and parser are wired so V14/V15/V18 leaves can append entries without re-bootstrapping the rule.
- `package.json` `engines.node` literal-read test: parses `package.json` as JSON; asserts `engines.node === ">=20.6.0"` literally (not a semver-range comparison) so a future leaf cannot silently widen, narrow, or delete the field without updating this assertion. The literal value is anchored in [`pi-integration-contract.md`](../spec_topics/pi-integration-contract.md) under **Host prerequisites** and in [`spec.md`](../spec.md) under **Orientation — Prerequisites — Host runtime**; a Pi minor bump that moves the floor must update all three sites in one edit. Additionally, the test asserts cross-package equality between the loom `package.json#engines.node` literal and `@mariozechner/pi-coding-agent`'s `engines.node` field (at the pinned `~0.72.1` version), sourcing the upstream literal from the `pi-engines-node` row of `SDK_SURFACE_INVENTORY` (see the SDK surface-inventory literal-read test below) rather than from a parallel literal in this bullet — so a Pi minor bump that moves the upstream Node floor fails this gate at the bump commit, not at user runtime.
- SDK surface-inventory literal-read test (`test/extension/pinned-surface.test.ts`): a static `SDK_SURFACE_INVENTORY` constant under `src/extension/` is the **single source of truth** that the capability probe described in [`pi-integration-contract.md` — Step 0 (Capability probe)](../spec_topics/pi-integration-contract.md#entry-capability-probe) also consumes; the four pinned constants the probe enumerates (Node floor, `AbortSignal` member-with-kind list, factory-probable capability list, peer-dep range) MUST be derived from one source so a single drift fails one test, not four. The constant carries one entry per probe-relevant member, each tagged with a kind discriminator that the probe and this test both branch on:
  - `{ kind: "node-floor", literal: ">=20.6.0" }` — the Node floor literal; the test asserts equality against `package.json#engines.node`.
  - `{ kind: "abortsignal-member", path: "<global>.<member>", memberKind: "constructor" | "prototype-method" | "static-method" | "prototype-property" }` — one entry per member of the kind table in [Step 0 (Capability probe)](../spec_topics/pi-integration-contract.md#entry-capability-probe) item (b). The test applies the kind-appropriate runtime check (`typeof === "function"` for the first three, `"<name>" in AbortSignal.prototype` for `prototype-property`) so an inventory drift on the kind tag fails this test, not the runtime probe.
  - `{ kind: "namespace-function", path: "pi.<name>" }` — one entry per function member of the **factory-probable** subset of [SDK capability inventory](../spec_topics/pi-integration-contract.md#sdk-capability-inventory): `pi.registerCommand` (capability 1), `pi.sendUserMessage` (capability 2), `ExtensionCommandContext.waitForIdle` (capability 2), `createAgentSession` (capability 3), `AgentSession.prototype.abort` (capability 3 / capability 5 — the cancellation primitive the runtime invokes from the one-shot `loomAbort.signal` listener registered at spawn time, per [`pi-integration-contract.md` — Subagent session lifecycle and Cancellation source](../spec_topics/pi-integration-contract.md)), `pi.registerTool` (capability 4), `pi.setActiveTools` and `pi.getActiveTools` (capability 4), `pi.registerMessageRenderer` (capability 6), `pi.sendMessage` (capability 6). Five capabilities, eight function members — capabilities 3, 4, and 6 each contribute two. Capability 5 (cancellation propagation) is covered jointly by the `abortsignal-member` entries above (the two Pi-side entry-point signals) and by the `AgentSession.prototype.abort` entry here (the subagent-mode primitive); capability 7 (binder LLM model) is **not** factory-probable and carries the entry kind below.
  - `{ kind: "type-union-snapshot", path: "SessionShutdownEvent.reason", literals: ["quit", "reload", "new", "resume", "fork"] as const }` — the closed `event.reason` set the `session_shutdown` handler validates against per [`pi-integration-contract.md` — Unknown-reason rule](../spec_topics/pi-integration-contract.md). The `literals` array MUST be declared `as const` so its element type is the narrow string-literal union rather than `string[]`. The test asserts the array equals the union members of `SessionShutdownEvent['reason']` (imported from `@mariozechner/pi-coding-agent`) at typecheck time via a **bidirectional** TypeScript type-equality probe of the form `type Reason = SessionShutdownEvent["reason"]; type Expected = (typeof SDK_SURFACE_INVENTORY)[<index>]["literals"][number]; const _exhaustive: Reason extends Expected ? Expected extends Reason ? true : never : never = true;` so a Pi patch that widens the union (right-to-left red) and a patch that narrows it (left-to-right red) both fail `npm run typecheck` rather than reaching user runtime. The probe MUST live in a `.ts` file that participates in `npm run typecheck` — a runtime-only assertion in this test file alone cannot observe type-level drift. A one-sided `extends` check (the common `Assert<X extends Y>` pattern) catches widening but not narrowing and is forbidden here. The same constant entry is the source-of-truth array the runtime's set-membership check inside the `session_shutdown` handler imports (single-site edit; no separate copy in the handler). The runtime-side fallback (the `loom/host/session-shutdown-reason-unknown` diagnostic and the full five-sub-step teardown) is the gap-window contract between a Pi patch landing on a user's machine and the matching loom patch shipping; this build-time pin is the upstream gate, not a replacement for that fallback.
  - `{ kind: "load-time-resolution", path: "ctx.modelRegistry.find", surface: "loom/load/binder-model-unresolved" }` — capability 7. The test does NOT assert this name on the imported namespace (it is not on the namespace at extension-factory entry); it asserts that the named diagnostic code is registered in [`diagnostics.md`](../spec_topics/diagnostics.md) and is owned by V16e per [`coverage-matrix.md`](./coverage-matrix.md).
  - `{ kind: "strict-capability-probe", probedName: "strictCapable" }` — the binder strict-capability probe field name (per [Slash-Command Argument Binding — Strict-capability requirement](../spec_topics/binder.md#strict-capability-requirement)). The test asserts that no member of the `Model<Api>` namespace under the pinned `~0.72.1` version matches the strict-capability concept under any name OTHER than `strictCapable` — if a sibling name resembling "strict" / "capable" / "structured" / "toolInput" appears, the test fails with a message naming the suspect member so the bump-procedure step 7 (rename probe constant) can be triggered consciously rather than silently. This catches the silent-rename failure mode where a Pi minor exposes the indicator under a different name and the runtime would otherwise see `undefined` forever and emit `loom/load/binder-model-strict-capability-unknown` (W) for every loom.
  - `{ kind: "api-coverage", apiNamespace: "@mariozechner/pi-ai" }` — the `Api` literal-union surface enumerated by pi-ai's model registry (per [Pi Integration Contract — Provider seed-field mapping](../spec_topics/pi-integration-contract.md#provider-seed-field-mapping)). The test enumerates the canonical `Api` value list (consumed from the same `SEED_FIELD_TABLE` constant the runtime classifier uses) and asserts every value appears as a row key in the seed-field table constant. A type-level assertion `type Check = Api extends keyof typeof SEED_FIELD_TABLE ? true : never; const _: Check = true;` catches drift at typecheck time; the runtime test asserts table-key membership against the same constant the type check guards. A new pi-ai `Api` value lights up this assertion red at the bump commit and triggers step 6 of [Pi version bump procedure](../spec_topics/pi-integration-contract.md#pi-version-bump-procedure).
  - `{ kind: "peer-dep-range", literal: "~0.72.1", packages: ["@mariozechner/pi-coding-agent", "@mariozechner/pi-agent-core", "@mariozechner/pi-ai", "@mariozechner/pi-tui"] }` — the peer-dep range and the fixed iteration array of the four lock-step `@mariozechner/*` peer-dep packages the Step 0 (d) probe checks. The test asserts that (i) `literal` equals the same string asserted by the `peerDependencies` literal-read test below; (ii) `packages` is exactly the four-entry array above in that fixed order (no additions, no removals, no reordering, no `typebox`); (iii) every entry in `packages` appears as a key in `package.json#peerDependencies`. This entry exists so the probe loop, the `peerDependencies` literal-read test, and the per-package coverage assertion all consume the same source-of-truth list — a future leaf that drops a package from the probe loop without updating this constant fails this test loudly rather than silently weakening the lock-step gate. The entry MUST NOT be split into one `peer-dep-range` row per package; the single-row-with-`packages`-array shape keeps the literal-and-coverage assertion atomic.

  The test imports `@mariozechner/pi-coding-agent` (one of the few production-package imports H1 permits, since the goal is to observe the real installed shape rather than a fake) and applies the kind-appropriate assertion against each entry at the pinned `~0.72.1` version. Failure modes the test must distinguish in its assertion messages: capability missing entirely (Pi removed it), capability renamed (a sibling name appears on the namespace), inventory drift (a name in the constant that is no longer cited by [`pi-integration-contract.md`](../spec_topics/pi-integration-contract.md) or vice-versa), and kind drift (an entry whose kind tag disagrees with what the probe currently branches on). This test is the mechanical gate cited by [Pi Integration Contract — Pi version bump procedure](../spec_topics/pi-integration-contract.md#pi-version-bump-procedure) step 2; a Pi minor bump that widens `peerDependencies` without re-confirming the surface fails this test loudly at the bump commit rather than at user runtime. H1 also adds `semver` (and `@types/semver`) as direct production dependencies of the loom package per the `**Loom-package implementation dependencies (V1).**` sub-paragraph of [`pi-integration-contract.md`](../spec_topics/pi-integration-contract.md) under **Host prerequisites**; the probe consumes `semver.satisfies` for both the Node-floor and peer-dep comparisons per the recommended-recipe paragraphs under [Step 0 (Capability probe)](../spec_topics/pi-integration-contract.md#entry-capability-probe) (the library identity is a non-normative recipe; the normative contract is the SemVer comparison itself), and the H1 manifest test asserts both the `dependencies` entry and the resolved-version literal so a future minor bump cannot silently widen comparator behaviour.
- `package.json` `peerDependencies` literal-read test: parses `package.json` as JSON; asserts that `peerDependencies["@mariozechner/pi-coding-agent"]`, `peerDependencies["@mariozechner/pi-agent-core"]`, `peerDependencies["@mariozechner/pi-ai"]`, and `peerDependencies["@mariozechner/pi-tui"]` all equal the same literal string `"~0.72.1"` (not a semver-range comparison) and that all four entries are present. Separately, the test asserts that `peerDependencies["typebox"] === "*"` and that the `typebox` entry is present — this is the bundled-package convention from `@mariozechner/pi-coding-agent` `docs/packages.md` (*Dependencies*) and is anchored in [`pi-integration-contract.md`](../spec_topics/pi-integration-contract.md) under **Host prerequisites — Pi SDK pin** (the `typebox` sub-paragraph). The `typebox` assertion MUST NOT be folded into the four-entry `~0.72.1` lock-step assertion; a future leaf that pins `typebox` to a versioned range fails this dedicated assertion. The literal `~0.72.1` value is anchored in four sites — `package.json` (this test's read target), [`pi-integration-contract.md`](../spec_topics/pi-integration-contract.md) under the preamble, [`pi-integration-contract.md`](../spec_topics/pi-integration-contract.md) under **Host prerequisites — Pi SDK pin**, and this test's own constant — and a Pi minor bump that moves the four-package lock-step must update all four sites in one edit. `spec.md` deliberately references this pin by location (per **Host prerequisites — Pi SDK pin**) rather than restating the literal, and so is NOT in the literal-anchor list. The four-entry redundancy in `peerDependencies` is intentional (per the **Host prerequisites — Pi SDK pin** rule that loom does not attempt to detect or accommodate skew across the four) and the test fails loudly if any of the four entries is removed, renamed, or pinned to a different range than the other three.
- File-presence test: `CHANGELOG.md` and `notes.md` exist at the project root. `CHANGELOG.md` starts with `# Changelog`, contains the Keep-a-Changelog format/versioning links, and has an `## [Unreleased]` section header; `notes.md` starts with `# Notes`. The test asserts these header strings literally so future leaves cannot silently retarget the convention to a different file shape.
