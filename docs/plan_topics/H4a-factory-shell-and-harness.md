# `H4a` — Extension factory shell and end-to-end harness

**Convention.** [`conventions.md`](./conventions.md) (phase categories — Pi-extension shell, end-to-end harness).

**Adds.** The Pi extension factory entry point — the standard `default function (pi: ExtensionAPI)` factory returning `void | Promise<void>`, which establishes the extension by side-effect registration calls on the injected `pi` handle (`pi.registerMessageRenderer`, `pi.registerCommand`, `pi.registerFlag`, `pi.registerTool`, `pi.on`) under per-call `try`/`catch` and MUST NOT throw out of the factory body — and a reusable end-to-end test harness that loads the extension against an in-process Pi session double and drives a slash dispatch, together with the **session-double fidelity contract** — the enumerated set of pinned-`@earendil-works/pi-coding-agent` behaviours the in-process double must reproduce: streamed-token order relative to `ctx.waitForIdle()` resolution, single-turn prompt-mode append semantics, the `pi.on` cancel-forward subscription, and cancellation propagation. Alongside those four observable-output axes the contract defines one shared **response-programming surface** — the single input-side API a test uses to script what the double does — that gives a test an observable way to: (a) script the sequence of assistant turns the double emits and the streamed fragments per turn; (b) script the `tool_use` result the double returns for a given tool call, including an `isError: true` tool-result and a mixed-success parallel `tool_use` batch (the `V13c` tool-calls vector); (c) script the binder/provider-call response or failure fed to the binder path, including a malformed-envelope failure and a transport-class failure that drive the `V11f` per-class retry budget; (d) force a `tool_loop.max_rounds` round-exhaustion (the `V13c` ceiling #2 vector); and (e) inject an abort/cancellation at a chosen point in the drive — pre-call, during an in-flight provider call, and during a budgeted retry (the `V11f` / `V17a` vectors). This response-programming surface is the single shared scripting contract every harness-driven leaf consumes — `M`, `M-T`, `H7a`, `V9c`, `V11f`, `V12a`, `V13c`, `V13d`, and `V17a` — so each scripts against one API rather than inventing a per-leaf surface; the named injection points above are the content the contract must define, while the API's exact shape is the implementer's to finalise. The capability-probe refusal logic is added by `V9a`; this leaf only establishes the never-throw factory boundary, the harness, and the fidelity contract its self-check enforces.

**Tests.**
- `Convention:` (phase categories) the factory completes its side-effect registrations on the injected `pi` handle and never throws even when a host seam is absent (each host-binding call is `try`/`catch`-wrapped per the exempt-broad-catch sites in `conventions.md`).
- `Convention:` (end-to-end harness) the harness loads the extension and asserts a registered command can be dispatched end-to-end against the session double.
- `Convention:` (end-to-end harness) the **session-double fidelity contract** is asserted by a harness self-check: the in-process double reproduces, relative to the pinned `@earendil-works/pi-coding-agent` SDK, (i) streamed assistant tokens observable in the transcript before `ctx.waitForIdle()` resolves, (ii) one streamed assistant response appended as a single prompt-mode turn, (iii) the `pi.on` cancel-forward subscription, and (iv) cancellation propagation. The self-check asserts these behaviours *as modelled by the in-process double* — the harness deliberately avoids a live Pi session, so it cannot detect the double diverging from real Pi, and loom 1.0 has no mechanical real-host fidelity gate (`V18c`'s version-bump runtime-evidence gate also runs against this same double and is explicitly not real-host coverage — see [`V18c`](./V18c-version-bump-checklist.md)). The double's fidelity to real Pi is therefore a host-behaviour **presupposition**, on the same footing as the unpinned presuppositions enumerated in [`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md): the Pi behaviours the four contract axes model are audited by editorial review under the [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) — axis (i)'s streamed-token-before-`ctx.waitForIdle()` ordering via [bump-checklist item (ak)](../spec_topics/pi-integration-contract/version-bump-step2.md#bump-checklist-user-visible-streaming-ordering), and the `ctx.waitForIdle()`-settling / turn-liveness and `AgentMessage[]` ordering axes via that checklist's other presupposition items — not mechanically verified by this self-check. The concrete post-merge detection mechanism for a double-vs-real divergence is a **manual real-host smoke run** — driving a representative `.loom` (typed query + tool loop + invoke + schema validation + binder + cancellation) against a live Pi host — checked at each Pi version bump under the [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure); it is a best-effort representative run, not exhaustive real-host coverage. A confirmed behavioural-divergence finding from that smoke is a **revert trigger** that forces restoration of the prior Pi-SDK pin (see [`V18c`](./V18c-version-bump-checklist.md)'s revert path), not merely an editorial-review note. The self-check additionally asserts that the shared response-programming surface drives the double **deterministically** — replaying the same scripted assistant turns, `tool_use` results, binder responses/failures, `tool_loop.max_rounds` round-exhaustion, and cancellation-injection points yields the same observable transcript on every run.

**Deps.** `H3a`

**Ships when.** `npm test` loads the extension through the harness, dispatches a no-op command end-to-end, and the session-double fidelity-contract self-check passes.
