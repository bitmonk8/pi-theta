# `H4a` — Extension factory shell and end-to-end harness

**Convention.** [`conventions.md`](./conventions.md) (phase categories — Pi-extension shell, end-to-end harness).

**Adds.** The Pi extension factory entry point — the standard `default function (pi: ExtensionAPI)` factory returning `void | Promise<void>`, which establishes the extension by side-effect registration calls on the injected `pi` handle (`pi.registerMessageRenderer`, `pi.registerCommand`, `pi.registerFlag`, `pi.registerTool`, `pi.on`) under per-call `try`/`catch` and MUST NOT throw out of the factory body — and a reusable end-to-end test harness that loads the extension against an in-process Pi session double and drives a slash dispatch. The harness also defines an **in-memory fixture-supply mechanism**: the harness supplies the fixture `.loom` content to the extension in memory so a harness-driven leaf's discovery reads from the harness-provided source rather than the real filesystem — no ambient `src/**` filesystem read, no `FileSystem` seam (`V8b`) dependency, keeping the `H3a` ambient-access scan clean. `M` and `M-T` consume this mechanism for their single-source happy-path discovery. The harness is established together with the **session-double fidelity contract** — the enumerated set of pinned-`@earendil-works/pi-coding-agent` behaviours the in-process double must reproduce: streamed-token order relative to `ctx.waitForIdle()` resolution, single-turn prompt-mode append semantics, the `pi.on` cancel-forward subscription, and cancellation propagation. Alongside those four observable-output axes the contract defines one shared **response-programming surface** — the single input-side API a test uses to script what the double does — that gives a test an observable way to: (a) script the sequence of assistant turns the double emits and the streamed fragments per turn; (b) script the `tool_use` result the double returns for a given tool call, including an `isError: true` tool-result and a mixed-success parallel `tool_use` batch (the `V13c` tool-calls vector); (c) script the binder/provider-call response or failure fed to the binder path, including a malformed-envelope failure and a transport-class failure that drive the `V11f` per-class retry budget; (d) force a `tool_loop.max_rounds` round-exhaustion (the `V13c` ceiling #2 vector); and (e) inject an abort/cancellation at a chosen point in the drive — pre-call, during an in-flight provider call, and during a budgeted retry (the `V11f` / `V17a` vectors); and (f) drive a nested `invoke(...)` child to completion — a produced final value from the child invocation — so a test can observe a completed invoke-child outcome through the double rather than the live `V15a` surface (the `V4c` ERR-13 completed-invoke-child vector); and (g) script/observe a subagent-mode callee — a private subagent `AgentSession` dispatched per the `V9i` subagent-mode session contract ([`subagent.md`](../spec_topics/pi-integration-contract/subagent.md)), modelled so a test can observe a subagent-loom outcome through the double rather than the live `V9i` surface (the `V4c` ERR-12 subagent-loom vector). This response-programming surface is the single shared scripting contract every harness-driven leaf consumes — `M`, `M-T`, `H7a`, `V4c`, `V9c`, `V11f`, `V12a`, `V13c`, `V13d`, and `V17a` — so each scripts against one API rather than inventing a per-leaf surface; the named injection points above are the content the contract must define, while the API's exact shape is the implementer's to finalise. The capability-probe refusal logic is added by `V9a`; this leaf only establishes the never-throw factory boundary, the harness, and the fidelity contract its self-check enforces.

**Tests.**
- `Convention:` (phase categories) the factory completes its side-effect registrations on the injected `pi` handle and never throws even when a host seam is absent (each host-binding call is `try`/`catch`-wrapped per the exempt-broad-catch sites in `conventions.md`).
- `Convention:` (end-to-end harness) the harness loads the extension and asserts a registered command can be dispatched end-to-end against the session double.
- `Convention:` (end-to-end harness) the **session-double fidelity contract** is asserted by a harness self-check: the in-process double reproduces, relative to the pinned `@earendil-works/pi-coding-agent` SDK, (i) streamed assistant tokens observable in the transcript before `ctx.waitForIdle()` resolves, (ii) one streamed assistant response appended as a single prompt-mode turn, (iii) the `pi.on` cancel-forward subscription, and (iv) cancellation propagation. The self-check asserts these behaviours *as modelled by the in-process double* — the harness deliberately avoids a live Pi session, so it cannot detect the double diverging from real Pi, and loom 1.0 has no mechanical real-host fidelity gate (`V18d`'s version-bump runtime-evidence gate also runs against this same double and is explicitly not real-host coverage — see [`V18d`](./V18d-version-bump-acceptance.md)). The double's fidelity to real Pi is therefore a host-behaviour **presupposition**, on the same footing as the unpinned presuppositions enumerated in [`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md): the Pi behaviours the four contract axes model are audited by editorial review under the [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) — axis (i)'s streamed-token-before-`ctx.waitForIdle()` ordering via [bump-checklist item (ak)](../spec_topics/pi-integration-contract/version-bump-step2.md#bump-checklist-user-visible-streaming-ordering), and the `ctx.waitForIdle()`-settling / turn-liveness and `AgentMessage[]` ordering axes via that checklist's other presupposition items — not mechanically verified by this self-check. The concrete pre-merge gate for a double-vs-real divergence is a **manual real-host smoke run** that drives [`H7a`](./H7a-integration-acceptance.md)'s committed multi-feature fixture `.loom` against a live Pi host instead of the in-process double — the same fixture `.loom` `H7a` drives against the double, reused here so the smoke and the integration-acceptance gate never diverge into separate fixtures. **Pass/fail criterion:** a live, non-deterministic LLM — not `H7a`'s deterministic in-process double — drives the fixture here, so the run is scored on model-output-invariant observables, not on exact reproduction of `H7a`'s double-recorded goldens (which a real model on this multi-feature pipeline is not assumed to reproduce in exact turn order/count or exact emitted-code set). The live-host run **passes** iff all of the following hold: (a) the run completes without error — no thrown exception, no aborted pipeline; (b) the binder pass produces output that validates against the binder envelope schema ([`V11c`](./V11c-bypass-envelope.md); [`binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params`](../spec_topics/binder/binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params)) — the run parses and conforms to the three-arm `ok | needs_info | ambiguous` envelope schema, a structural-validity check, not an exact-value match against `H7a`'s deterministic goldens; (c) cancellation propagation is observed at the injected cancel point; (d) the four session-double fidelity-contract axes — streamed-token order relative to `ctx.waitForIdle()` resolution, single-turn prompt-mode append semantics, the `pi.on` cancel-forward subscription, and cancellation propagation — hold; and (e) the emitted `loom-system-note` codes are a **subset** of the **committed permitted-code list** checked in alongside [`H7a`](./H7a-integration-acceptance.md)'s fixture `.loom` — the single committed, reviewed reference set that enumerates the union of `loom/...` codes the slices in `H7a`'s **Deps** can emit (a superset of `H7a`'s committed golden diagnostics list) — so the runner scores (e) against one fixed committed reference rather than reconstructing the union by hand. A failure of any one of (a)–(e) — a thrown or aborted run, binder output that fails to validate against the binder envelope schema ([`V11c`](./V11c-bypass-envelope.md)), unobserved cancellation, a violated fidelity-contract axis, or an emitted code outside the permitted set — is a **fail** (a confirmed behavioural-divergence finding), so a genuine divergence still produces a falsifiable fail that drives the revert / merge-block path below; benign live-model variance in turn order, turn count, or which permitted codes fire is **not** a fail. `H7a`'s in-process gate keeps its exact-match goldens unchanged — against the deterministic double, exact-transcript and exact-code-set equality is the correct criterion. It is a best-effort representative run against that one fixture, not exhaustive real-host coverage. The smoke is a **required pre-merge gate** on two independent triggers, each with a named owner who MUST run it and record the result — in the triggering change's commit message (the bump commit for trigger (1), the merge commit for trigger (2)), with no separate artefact file, the same manual-run-evidence convention the spec fixes for the version-bump path ([`version-bump-triggers.md`](../spec_topics/pi-integration-contract/version-bump-triggers.md)'s "produces no separate artefact file"; [`version-bump-step2.md`](../spec_topics/pi-integration-contract/version-bump-step2.md)'s outcomes "in the bump commit message"), so the [`H6a`](./H6a-live-corpus-activation.md) release-gate evidence record locates the run the same way wherever the smoke fires — before the change merges: (1) each Pi version bump — owned by the contributor performing the bump, who runs the smoke against the candidate Pi-SDK pin before the bump merges, under the [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure); and (2) any merge whose diff **touches the four fidelity-contract axes** — owned by the contributor of that merge, who runs the smoke before the merge lands — — observably, a commit that edits either the in-process double's modelling of, or the production seam realising, any of (i) streamed-token order relative to `ctx.waitForIdle()` resolution, (ii) single-turn prompt-mode append semantics, (iii) the `pi.on` cancel-forward subscription, or (iv) cancellation propagation — so a divergence introduced by a loom change rather than a Pi bump is caught before it reaches `main`. This pre-merge step is a **contributor obligation, not a CI-enforced gate**: loom 1.0 has no mechanical hook that observes whether the smoke ran, and the trigger-(2) condition (a diff touching the four fidelity-contract axes) has no automatic diff detector — recognising it is itself part of the contributor's obligation. Consequently a skipped run, or a divergence whose author does not recognise their diff as touching the four axes, is **not blocked from merging**; such an omission is detectable only **after the fact**, by reading the triggering commit message for the recorded run (the same evidence record [`H6a`](./H6a-live-corpus-activation.md)'s release gate audits), not at merge time. A confirmed behavioural-divergence finding from that smoke is a **merge blocker**: a Pi-bump-triggered finding blocks the bump from merging — the prior Pi-SDK pin is restored before merge (see [`V18d`](./V18d-version-bump-acceptance.md)'s revert path, which owns the pre-merge pin-restore) — and a four-axes-change-triggered finding blocks the merge until the divergence is resolved; in neither case is it merely an editorial-review note. The self-check additionally asserts that the shared response-programming surface drives the double **deterministically** — replaying the same scripted assistant turns, `tool_use` results, binder responses/failures, `tool_loop.max_rounds` round-exhaustion, invoke-child completions, subagent-mode callee dispatch, and cancellation-injection points yields the same observable transcript on every run.
- `Convention:` (end-to-end harness) the harness self-check additionally asserts the **functional effect** of each response-programming-surface injection-point category — that each scripted input yields its expected observable, not merely a stable one — so the seam contract is verified at this owning leaf rather than only transitively by downstream consumers. The self-check asserts, for the categories `Adds` enumerates: (a) scripted assistant turns and the per-turn streamed fragments appear in the transcript in the scripted order; (b) a scripted `isError: true` tool-result surfaces as an error tool-result in the transcript, and a mixed-success parallel `tool_use` batch lowers each sibling call's outcome independently; (c) a scripted malformed-envelope failure and a scripted transport-class failure each drive the observable binder failure / retry path; (d) a scripted `tool_loop.max_rounds` round-exhaustion produces the round-exhaustion observable; and (e) an abort injected at each chosen point — pre-call, during an in-flight provider call, and during a budgeted retry — surfaces the cancellation observable; and (f) a scripted invoke-child completion surfaces its produced final value as an observable completed-invoke-child outcome in the transcript; and (g) a scripted subagent-mode callee surfaces its outcome as an observable subagent-loom outcome in the transcript, modelled on the `V9i` subagent-mode session contract rather than a plain scripted turn. The assertions are phrased in terms of the observable each scripted input must yield; the surface's API shape stays the implementer's to finalise.

**Deps.** `H3a`

**Ships when.** `npm test` loads the extension through the harness, dispatches a no-op command end-to-end, and the session-double fidelity-contract self-check passes — including its determinism gate and the per-category functional-effect assertions (a)–(g) for the response-programming surface.
