# `H4a` — Extension factory shell and end-to-end harness

**Convention.** [`conventions.md`](./conventions.md) (phase categories — Pi-extension shell, end-to-end harness).

**Adds.** The Pi extension factory entry point — the factory implementation lives under `src/**` (the single production-code root the [`conventions.md`](./conventions.md) cross-cutting gates scope to), and the on-disk module `extensions/index.ts` (the file Pi resolves under the `package.json#pi.extensions = ["./extensions"]` package-discovery directory [`H1a`](./H1a-scaffold-and-toolchain.md) declares, per [`host-prerequisites.md` §"Extension entry point"](../spec_topics/pi-integration-contract/host-prerequisites.md); the manifest entry is a directory and `extensions/index.ts` is the file resolved under it) is a **thin delegating entry shim** that carries no rule-subject logic and re-exports that `src/**` factory as the standard `default function (pi: ExtensionAPI)` export. The factory's declared return type stays `void | Promise<void>` for host-interface conformance, but loom pins it to the **synchronous arm**: the factory body runs synchronously and returns `void` — it is not declared `async` and awaits no work, so there is no returned `Promise` arm that could later reject. It establishes the extension by side-effect registration calls on the injected `pi` handle (`pi.registerMessageRenderer`, `pi.registerCommand`, `pi.registerFlag`, `pi.registerTool`, `pi.on`) under per-call `try`/`catch`, so that — because the body returns synchronously with no awaited work — "MUST NOT throw out of the factory body" is the complete never-fault prohibition for this boundary — and a reusable end-to-end test harness that loads the extension against an in-process Pi session double and drives a slash dispatch. The harness also defines an **in-memory fixture-supply mechanism**: the harness supplies the fixture `.loom` content to the extension in memory so a harness-driven leaf's discovery reads from the harness-provided source rather than the real filesystem — no ambient `src/**` filesystem read, no `FileSystem` seam (`V8b`) dependency, keeping the `H3a` ambient-access scan clean. `M` and `M-T` consume this mechanism for their single-source happy-path discovery. The harness is established together with the **session-double fidelity contract** — the enumerated set of pinned-`@earendil-works/pi-coding-agent` behaviours the in-process double must reproduce: streamed-token order relative to `ctx.waitForIdle()` resolution, single-turn prompt-mode append semantics, the `pi.on` cancel-forward subscription, and cancellation propagation. The full scripted-injection **response-programming surface** the double's input side exposes is owned by [`H4b`](./H4b-response-programming-surface.md), which depends on this leaf. The capability-probe refusal logic is added by `V9a`; this leaf only establishes the never-throw factory boundary, the harness, and the fidelity contract its self-check enforces.

The leaf further establishes the manual real-host smoke governance protocol — a non-automated pre-merge gate that never runs under `npm test`; its audience is the bump/merge contributor and the [`H6a`](./H6a-live-corpus-activation.md) release-gate auditor, not the `H4a` implementer. The concrete pre-merge gate for a double-vs-real divergence is a **manual real-host smoke run** that drives [`H7a`](./H7a-integration-acceptance.md)'s committed multi-feature fixture `.loom` against a live Pi host instead of the in-process double — the same fixture `.loom` `H7a` drives against the double, reused here so the smoke and the integration-acceptance gate never diverge into separate fixtures. **Activation window.** Both triggers are dormant until [`H7a`](./H7a-integration-acceptance.md) has landed: the smoke run is defined entirely in terms of two `H7a`-owned inputs — its committed multi-feature fixture `.loom` and the committed permitted-code list criterion (e) scores against — and neither exists in the `H4a`→`H7a` window. A trigger event (a Pi version bump under trigger (1), or a fidelity-axis-touching merge under trigger (2)) that occurs before `H7a` lands is therefore **deferred**, not scored against the absent inputs: its owner records the deferral in the triggering commit message (the same evidence record the recorded run uses) and re-runs the smoke once `H7a` has landed, rather than treating the missing fixture or permitted-code list as a criterion failure or a behavioural-divergence finding. A window-deferral is an **un-runnable-gate condition** of the same family as the unsatisfied-precondition case the Live-host precondition below defines — it does NOT score as a fail and so does not spuriously block the triggering change from merging. **Live-host precondition.** The smoke can only run against a live Pi host that satisfies the host-side prerequisites [`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) already enumerates: (1) a Pi install at the pinned SDK range ([`host-prerequisites.md#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)); (2) a structured-output-capable binder model resolvable through `ctx.modelRegistry` ([`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) item 2), since criterion (b) below — a structurally-valid binder pass — is unsatisfiable without one; and (3) credentials Pi already resolves for that model ([`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) item 3 — loom stores none). The smoke's two named owners (the bump contributor for trigger (1) and the merging contributor for trigger (2), below) each run the gate against their own Pi install configured to satisfy this precondition. An unsatisfied binder-model or credential precondition — most concretely, no structured-output-capable binder model registered, which load-fails with `loom/load/binder-model-unresolved` — is an **un-runnable-gate condition, not a behavioural-divergence finding**: the owner provisions the missing prerequisite and re-runs, and a missing or uncredentialed binder model does NOT score as the criterion (a) failure below (and so does not spuriously block the merge). **Pass/fail criterion:** a live, non-deterministic LLM — not `H7a`'s deterministic in-process double — drives the fixture here, so the run is scored on model-output-invariant observables, not on exact reproduction of `H7a`'s double-recorded goldens (which a real model on this multi-feature pipeline is not assumed to reproduce in exact turn order/count or exact emitted-code set). The live-host run **passes** iff all of the following hold: (a) the run completes without error — no thrown exception, no aborted pipeline; (b) the binder pass produces output that validates against the binder envelope schema ([`V11c`](./V11c-bypass-envelope.md); [`binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params`](../spec_topics/binder/binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params)) — the run parses and conforms to the three-arm `ok | needs_info | ambiguous` envelope schema, a structural-validity check, not an exact-value match against `H7a`'s deterministic goldens; (c) cancellation propagation is observed at the injected cancel point; (d) the four session-double fidelity-contract axes — streamed-token order relative to `ctx.waitForIdle()` resolution, each streamed assistant response is appended as exactly one prompt-mode turn (a per-response invariant, independent of total turn count), the `pi.on` cancel-forward subscription, and cancellation propagation — hold; and (e) the emitted `loom-system-note` codes are a **subset** of the **committed permitted-code list** checked in alongside [`H7a`](./H7a-integration-acceptance.md)'s fixture `.loom` — the single committed, reviewed reference set: a manually-maintained, best-effort union of the `loom/...` codes the slices in `H7a`'s **Deps** can emit, intended as a superset of `H7a`'s committed golden diagnostics list (only that `golden ⊆ permitted` containment is mechanically gated, by `H7a`; the union's completeness is not mechanically verified, so a stale union can omit a benign code a slice emits) — so the runner scores (e) against one fixed committed reference rather than reconstructing the union by hand. A failure of any one of (a)–(e) — a thrown or aborted run, binder output that fails to validate against the binder envelope schema ([`V11c`](./V11c-bypass-envelope.md)), unobserved cancellation, a violated fidelity-contract axis, or an emitted code outside the permitted set — is a **fail** (a confirmed behavioural-divergence finding), so a genuine divergence still produces a falsifiable fail that drives the revert / merge-block path below; benign live-model variance in turn order, turn count, or which permitted codes fire is **not** a fail. `H7a`'s in-process gate keeps its exact-match goldens unchanged — against the deterministic double, exact-transcript and exact-code-set equality is the correct criterion. The smoke is a required pre-merge gate on two independent triggers, each with a named owner who MUST run it and record the result — in the triggering change's commit message (the bump commit for trigger (1), the merge commit for trigger (2)), with no separate artefact file, the same manual-run-evidence convention the spec fixes for the version-bump path ([`version-bump-triggers.md`](../spec_topics/pi-integration-contract/version-bump-triggers.md)'s "produces no separate artefact file"; [`version-bump-step2.md`](../spec_topics/pi-integration-contract/version-bump-step2.md)'s outcomes "in the bump commit message"), so the [`H6a`](./H6a-live-corpus-activation.md) release-gate evidence record locates the run the same way wherever the smoke fires — before the change merges: (1) each Pi version bump — owned by the contributor performing the bump, who runs the smoke against the candidate Pi-SDK pin before the bump merges, under the [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure); and (2) any merge whose diff **touches the four fidelity-contract axes** — owned by the contributor of that merge, who runs the smoke before the merge lands — — observably, a commit that edits either the in-process double's modelling of, or the production seam realising, any of (i) streamed-token order relative to `ctx.waitForIdle()` resolution, (ii) single-turn prompt-mode append semantics, (iii) the `pi.on` cancel-forward subscription, or (iv) cancellation propagation — so a divergence introduced by a loom change rather than a Pi bump is caught before it reaches `main`. This pre-merge step is a **contributor obligation, not a CI-enforced gate**: loom 1.0 has no mechanical hook that observes whether the smoke ran, and the trigger-(2) condition (a diff touching the four fidelity-contract axes) has no automatic diff detector — recognising it is itself part of the contributor's obligation. Consequently a skipped run, or a divergence whose author does not recognise their diff as touching the four axes, is **not blocked from merging**; such an omission is detectable only **after the fact**, by reading the triggering commit message for the recorded run (the same evidence record [`H6a`](./H6a-live-corpus-activation.md)'s release gate audits), not at merge time. A confirmed behavioural-divergence finding from that smoke is a **merge blocker**: a Pi-bump-triggered finding blocks the bump from merging — the prior Pi-SDK pin is restored before merge (see [`V18d`](./V18d-version-bump-acceptance.md)'s revert path, which owns the pre-merge pin-restore) — and a four-axes-change-triggered finding blocks the merge until the divergence is resolved; in neither case is it merely an editorial-review note.

**Tests.**
- `Convention:` (phase categories) the factory completes its side-effect registrations on the injected `pi` handle and never throws even when a host seam is absent (each host-binding call is `try`/`catch`-wrapped per the exempt-broad-catch sites in `conventions.md`), and the factory returns **synchronously** — its return value is `undefined`, not a thenable, witnessing the synchronous-arm pin (a non-`async`, non-`Promise` return) so the never-fault property covers the returned-value path and not merely the synchronous-throw path.
- `Convention:` (end-to-end harness) the harness loads the extension and asserts a registered command can be dispatched end-to-end against the session double.
- `Convention:` (end-to-end harness) the **session-double fidelity contract** is asserted by a harness self-check: the in-process double reproduces, relative to the pinned `@earendil-works/pi-coding-agent` SDK, (i) streamed assistant tokens observable in the transcript before `ctx.waitForIdle()` resolves, (ii) one streamed assistant response appended as a single prompt-mode turn, (iii) the `pi.on` cancel-forward subscription, and (iv) cancellation propagation. The self-check asserts these behaviours *as modelled by the in-process double* — the harness deliberately avoids a live Pi session, so it cannot detect the double diverging from real Pi, and loom 1.0 has no mechanical real-host fidelity gate (`V18d`'s version-bump runtime-evidence gate also runs against this same double and is explicitly not real-host coverage — see [`V18d`](./V18d-version-bump-acceptance.md)). The manual real-host smoke run that gates a double-vs-real divergence is **not** part of this self-check and never runs under `npm test`; it is established as a governance protocol in **Adds** above.

**Deps.** `H3a`

**Ships when.** `npm test` loads the extension through the harness, dispatches a no-op command end-to-end, and the session-double fidelity-contract self-check passes.
