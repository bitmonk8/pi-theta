# `H6a` — Live-corpus closing-gate activation (loom 1.0 release gate)

**Convention.** [`conventions.md`](./conventions.md) — *REQ-ID discipline* (the live-corpus footing of the closing gate) and the [`coverage-matrix.md`](./coverage-matrix.md) closure obligation.

**Adds.** The terminal release-gate step that, alongside flipping the closing gate to its live-corpus footing, carries the loom 1.0 release-gate confirmation that the [**manual real-host smoke run**](./real-host-smoke-gate.md) was executed and passed against the shipping Pi-SDK pin — the recorded real-host fidelity evidence loom 1.0 ships on. It flips [`H5a`](./H5a-closing-gate-automation.md)'s closing gate from its seeded-fixture footing to its live-corpus footing: the gate is reconfigured to reconcile the live spec REQ-ID set, the live `spec_topics/**` normative-MUST set, and the live test corpus against the live [`coverage-matrix.md`](./coverage-matrix.md), so that from this leaf onward an unmapped executable REQ-ID, a coverage-matrix-mapped numbered REQ-ID with no citing test, or an un-enumerated un-anchored MUST reddens `npm test`. This is the "loom 1.0 release gate" named in [`plan.md`](../plan.md) item 5 and the [`coverage-matrix.md`](./coverage-matrix.md) release-gate clause; until this leaf lands those three failure modes are exercised only against the seeded fixtures at `H5a`. Because the live-corpus footing reddens CI the moment any executable REQ-ID lacks a mapping, this leaf MUST be sequenced after every leaf that can introduce an executable REQ-ID or an un-anchored normative MUST (see **Deps.**); it sequences immediately after the warn-only [`H5b`](./H5b-warn-only-canary.md) canary, which carries that complete coverage-producing dependency set and reconciles the same live surfaces this leaf hard-fails on without reddening CI, and activating it earlier would redden `main` on coverage that later leaves are still landing. **Rollback.** Recovery from a bad flip — whether from an incomplete dependency set or live coverage not yet closed — is a single revert: reverting this leaf's activation commit returns [`H5a`](./H5a-closing-gate-automation.md)'s closing gate to its seeded-fixture footing, so `main` stops reddening on live coverage that later work is still landing. There is no data, schema, or registry migration to unwind, so the revert is the complete recovery step. The live-corpus footing reconciles the live spec/test corpus exclusive of the dedicated test-fixtures root [`H5a`](./H5a-closing-gate-automation.md) places the seeded gate fixtures under, so the flip cannot permanently red `main` on a permanent-by-design seeded violation.

**Tests.**
- `Convention:` (*REQ-ID discipline* — live-corpus footing) `npm test` runs the closing gate against the live spec corpus, the live test corpus, and the live [`coverage-matrix.md`](./coverage-matrix.md), green: every executable spec REQ-ID maps to a closing leaf, every coverage-matrix-mapped numbered REQ-ID has a citing test in the live test corpus, and every un-anchored normative MUST/MUST-NOT that the best-effort `MUST`/`MUST NOT`-token scan recognises is enumerated with a closing leaf; token-less, narrative-ambiguous, or seam-name-dependent residue is covered by the [`governance.md` GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) reviewer-inspection step, not this gate.
- `Convention:` (*REQ-ID discipline* — live-corpus footing) the gate reddens when a live executable spec REQ-ID has no `coverage-matrix.md` row (asserted by removing one mapping within the test and restoring it), confirming the non-gating→gating flip is active rather than still fixture-only.
- `Convention:` (*REQ-ID discipline* — live-corpus footing) the gate reddens when a live coverage-matrix-mapped numbered REQ-ID has no citing test in the live test corpus (asserted by removing one REQ-ID citation within the test and restoring it), confirming the citing-test mode flips to the live corpus alongside the unmapped-REQ-ID and un-anchored-MUST modes.

**Deps.** `H5b, H7a`

_(This leaf depends on the warn-only canary [`H5b`](./H5b-warn-only-canary.md) and flips the hard-fail gate immediately after it, and on the terminal integration-acceptance leaf [`H7a`](./H7a-integration-acceptance.md) — the producer of the committed multi-feature fixture `.loom` that this leaf's **Ships when** manual real-host smoke release-gate criterion drives against the live host — so that fixture exists before the release gate references it. That `H7a` edge is pinned directly here rather than threaded through `H5b` because `H7a` closes no new spec REQ-ID and so is not a member of `H5b`'s coverage-producing set. `H5b` owns the complete coverage-producing dependency set (`H1a`, `H5a`, `M`, `V1a`–`V18d`), kept transitively complete under the standing plan-maintenance obligation in *REQ-ID discipline* on [`conventions.md`](./conventions.md); this leaf inherits that completeness transitively through `H5b` rather than restating the set, so the warn-only and hard-fail footings cover an identical set of leaves by construction. The live surfaces this leaf hard-fails on are exactly the surfaces `H5b` warn-only-reconciles immediately prior — both run `H5a`'s gate machinery over the live corpus — so the two footings cannot drift.)_

**Ships when.** `npm test` runs the closing gate on its live-corpus footing — the live spec REQ-ID set, the **token-recognised** live `spec_topics/**` normative-MUST set (the residue the `MUST`/`MUST NOT`-token scan cannot witness is covered by the [`governance.md` GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) reviewer-inspection step, not this gate), and the live test corpus reconciled against the live [`coverage-matrix.md`](./coverage-matrix.md) — and passes green, and the removed-mapping and removed-citation checks above confirm the gate reddens on an unmapped executable REQ-ID and on a coverage-matrix-mapped numbered REQ-ID with no citing test, **and** the loom 1.0 release gate's **manual real-host smoke** acceptance is satisfied: the [**manual real-host smoke run**](./real-host-smoke-gate.md) — driving [`H7a`](./H7a-integration-acceptance.md)'s committed multi-feature fixture `.loom` against a live Pi host — was executed and **passed** against the shipping Pi-SDK pin (the single-source-of-truth pin literal at [`host-prerequisites.md#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)), where pass is the model-output-invariant criterion [`real-host-smoke-gate.md`](./real-host-smoke-gate.md) defines for the live-host run — no-error completion, binder output that validates against the binder envelope schema ([`V11c`](./V11c-bypass-envelope.md); [`binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params`](../spec_topics/binder/binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params)), observed cancellation propagation, the four fidelity-contract axes, and emitted `loom-system-note` codes within the **committed permitted-code list** checked in alongside [`H7a`](./H7a-integration-acceptance.md)'s fixture `.loom` (the manually-maintained, best-effort union [`H7a`](./H7a-integration-acceptance.md) defines, whose completeness is not mechanically verified) — not exact reproduction of `H7a`'s double-recorded goldens. This is a **manual** gate — loom 1.0 has no mechanical real-host fidelity gate (see [`real-host-smoke-gate.md`](./real-host-smoke-gate.md)) — gated at the release gate so loom 1.0 never ships without recorded real-host fidelity evidence. "Recorded" obligates a falsifiable evidence record committed in this leaf's **gate-activation commit message** — no separate artefact file, following the same manual-run-evidence convention the spec already fixes for the version-bump path ([`version-bump-triggers.md`](../spec_topics/pi-integration-contract/version-bump-triggers.md)'s "produces no separate artefact file"; [`version-bump-step2.md`](../spec_topics/pi-integration-contract/version-bump-step2.md)'s per-item outcomes recorded "in the bump commit message") — capturing the falsifiable record of the run: the Pi-SDK pin literal the run executed against (the single-source-of-truth pin at [`host-prerequisites.md#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)), the named owner who ran the [pre-merge real-host smoke gate](./real-host-smoke-gate.md) (the contributor performing the bump), the date of the run, and the observed result against the [`real-host-smoke-gate.md`](./real-host-smoke-gate.md) narrowed model-output-invariant criterion. The release does not pass until that gate-activation commit message records the run — not merely when the box is ticked — so a skipped, mis-recorded, or stale-pin run is detectable after the fact by reading the commit message. From this leaf onward the live-corpus failure modes gate `npm test`.
