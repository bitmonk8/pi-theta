# `H6a` — Live-corpus closing-gate activation (loom 1.0 release gate)

**Convention.** [`conventions.md`](./conventions.md) — *REQ-ID discipline* (the live-corpus footing of the closing gate) and the [`coverage-matrix.md`](./coverage-matrix.md) closure obligation.

**Adds.** The terminal release-gate step that, alongside flipping the closing gate to its live-corpus footing, carries the loom 1.0 release-gate confirmation that the [`H4a`](./H4a-factory-shell-and-harness.md) **manual real-host smoke run** was executed and passed against the shipping Pi-SDK pin — the recorded real-host fidelity evidence loom 1.0 ships on. It flips [`H5a`](./H5a-closing-gate-automation.md)'s closing gate from its seeded-fixture footing to its live-corpus footing: the gate is reconfigured to reconcile the live spec REQ-ID set, the live `spec_topics/**` normative-MUST set, and the live test corpus against the live [`coverage-matrix.md`](./coverage-matrix.md), so that from this leaf onward an unmapped executable REQ-ID, a coverage-matrix-mapped numbered REQ-ID with no citing test, or an un-enumerated un-anchored MUST reddens `npm test`. This is the "loom 1.0 release gate" named in [`plan.md`](../plan.md) item 5 and the [`coverage-matrix.md`](./coverage-matrix.md) release-gate clause; until this leaf lands those three failure modes are exercised only against the seeded fixtures at `H5a`. Because the live-corpus footing reddens CI the moment any executable REQ-ID lacks a mapping, this leaf MUST be sequenced after every leaf that can introduce an executable REQ-ID or an un-anchored normative MUST (see **Deps.**); activating it earlier would redden `main` on coverage that later leaves are still landing.

**Tests.**
- `Convention:` (*REQ-ID discipline* — live-corpus footing) `npm test` runs the closing gate against the live spec corpus, the live test corpus, and the live [`coverage-matrix.md`](./coverage-matrix.md), green: every executable spec REQ-ID maps to a closing leaf, every coverage-matrix-mapped numbered REQ-ID has a citing test in the live test corpus, and every un-anchored normative MUST/MUST-NOT is enumerated with a closing leaf.
- `Convention:` (*REQ-ID discipline* — live-corpus footing) the gate reddens when a live executable spec REQ-ID has no `coverage-matrix.md` row (asserted by removing one mapping within the test and restoring it), confirming the non-gating→gating flip is active rather than still fixture-only.
- `Convention:` (*REQ-ID discipline* — live-corpus footing) the gate reddens when a live coverage-matrix-mapped numbered REQ-ID has no citing test in the live test corpus (asserted by removing one REQ-ID citation within the test and restoring it), confirming the citing-test mode flips to the live corpus alongside the unmapped-REQ-ID and un-anchored-MUST modes.
- **Release-gate acceptance (manual real-host smoke).** The loom 1.0 release gate carries a manual checklist item confirming the [`H4a`](./H4a-factory-shell-and-harness.md) **manual real-host smoke run** — driving [`H7a`](./H7a-integration-acceptance.md)'s committed multi-feature fixture `.loom` against a live Pi host — was executed and **passed** against the shipping Pi-SDK pin (the single-source-of-truth pin literal at [`host-prerequisites.md#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)), where pass is the criterion `H4a` defines: appended turns match `H7a`'s committed golden transcript in order and count and the run emits exactly `H7a`'s committed golden `loom-system-note` diagnostics list. This is a **manual** gate — loom 1.0 has no mechanical real-host fidelity gate (see [`H4a`](./H4a-factory-shell-and-harness.md)) — recorded at the release gate so loom 1.0 never ships without recorded real-host fidelity evidence; the release does not pass until this item is checked.

**Deps.** `H5a`, `M`, `V1a`–`V18c`

_(The dependency set is the complete coverage-producing set: `H5a` supplies the gate machinery, and every MVP and vertical implementation leaf (`V1a`–`V18c`) can introduce an executable REQ-ID, a numbered-REQ-ID citing test, or an un-anchored normative MUST. The set MUST stay transitively complete — any future leaf that can introduce an executable REQ-ID, a numbered-REQ-ID citing test, or an un-anchored MUST is a new dependency of this leaf, otherwise the gate can activate against incomplete coverage.)_

**Ships when.** `npm test` runs the closing gate on its live-corpus footing — the live spec REQ-ID set, the live `spec_topics/**` normative-MUST set, and the live test corpus reconciled against the live [`coverage-matrix.md`](./coverage-matrix.md) — and passes green, and the removed-mapping and removed-citation checks above confirm the gate reddens on an unmapped executable REQ-ID and on a coverage-matrix-mapped numbered REQ-ID with no citing test, **and** the manual real-host smoke checklist item above is recorded as executed-and-passed against the shipping Pi-SDK pin. From this leaf onward the live-corpus failure modes gate `npm test`.
