# `H1a` — Project scaffold and toolchain

**Convention.** [`conventions.md`](./conventions.md) (phase categories; cross-cutting rules) and the *Doc updates* cross-cutting rule.

**Adds.** The TypeScript project skeleton — `package.json` (with the four `@earendil-works/pi-*` packages plus `typebox` declared as `peerDependencies`, the same four `@earendil-works/pi-*` packages **additionally** declared in loom's own `devDependencies` at that same loom 1.0 Pi-SDK pin range so a fresh `npm install` provisions them under loom's own `node_modules`, `ajv`/`ajv-formats`/`semver`/`chokidar`/`yaml`/`minimatch` declared as loom's own runtime `dependencies`, `vitest` declared as loom's own `devDependencies` test runner and assertion library, the lint toolchain — `eslint` (the ESLint engine), `@typescript-eslint/parser` (the TypeScript-aware ESLint parser), and `eslint-plugin-loom-local` (the local custom-rule plugin entry for the bespoke `no-broad-catch` rule) — declared as loom's own `devDependencies`, an `engines.node` field, a `"type": "module"` ES-module declaration sourced from [`extension-bootstrap-and-per-loom.md` §"Extension ES-module declaration"](../spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md#extension-esm-type-module) (the loom runtime is ES-modules-only at loom 1.0), and a `pi.extensions` manifest entry declared `["./extensions"]` — the package-discovery directory value (a directory, **not** a file path) Pi auto-discovers loom's extension factory module from, per [`extension-bootstrap-and-per-loom.md` §"Extension entry point"](../spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md#extension-entry-point); the file that resolves under that directory is `extensions/index.ts`, pinned by [`H4a`](./H4a-factory-shell-and-harness.md)'s factory entry point, and the directory value here MUST stay consistent with that file path), `tsconfig.json`, the Vitest test runner and its assertion API, and `npm test` / `npm run build` / `npm run typecheck` scripts — on which every later leaf builds. loom's own build, test, and CI all run under **npm** with a flat/hoisted `node_modules` tree, and that is the package manager and layout the leaf's two install-time reliances below assume: the `eslint-plugin-loom-local` `file:./tools/eslint-plugin-loom-local` specifier's npm-style `file:` resolution (witnessed by the green `npm install` in *Ships when*) and the fresh-`npm install` provisioning of the four `@earendil-works/pi-*` `devDependencies` under loom's own `node_modules` for the downstream `V18a`/`V18b`/`V18c` reads (including `V18c`'s live read of the installed `package.json`). Both reliances are scoped to that npm + flat/hoisted-`node_modules` combination. The `pnpm`/`yarn` language in [`host-prerequisites.md` §`#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin) is the **consumer-facing** install-time skew-detection contract for hosts installing loom — it does not govern loom's own checkout, whose `file:` plugin resolution and layout-dependent provisioning reads would differ under pnpm's isolated linker. The `npm run typecheck` script is the build-time `tsc` type-check runner the version-bump procedure names ([`version-bump-intro.md` step 1](../spec_topics/pi-integration-contract/version-bump-intro.md)); it is the runner that compiles the `loom/typecheck/*` brand-string type-equality assertions (consumed at [`V18c`](./V18c-version-bump-checklist.md)'s Ships-when for the `loom/typecheck/session-shutdown-reason-snapshot` gate). The `tsc` provider behind that `npm run typecheck` script is the standalone `typescript` package, declared in loom's own `devDependencies` so a fresh `npm install` provisions `tsc`, declared at any range that satisfies the TypeScript floor of `>=5.4.0` owned by [`host-prerequisites.md` §`#tsc-toolchain-floor`](../spec_topics/pi-integration-contract/host-prerequisites.md#tsc-toolchain-floor) and sourced from that anchor rather than restated as a fresh plan-side literal, on the same footing as the spec-anchored `engines.node` floor this leaf also sources. That floor is load-bearing: [`V18c`](./V18c-version-bump-checklist.md)'s `loom/typecheck/session-shutdown-reason-snapshot` brand-string gate relies on `tsc` surfacing that literal verbatim and grep-ably in its type-mismatch diagnostic, a property `host-prerequisites.md` §`#tsc-toolchain-floor` confirmed against that floor. `H1a` is the initial-population owner of the `package.json#engines.node` field; its value is the pinned Node floor owned by [`capability-probe.md` §(a)](../spec_topics/pi-integration-contract/capability-probe.md#entry-capability-probe), sourced from that anchor rather than restated here. This is the literal consumed by the build-time `engines.node` literal-read test ([`overview-and-orientation.md` §"Node version floor"](../spec_topics/overview-and-orientation.md#orientation)) and by [`V18c`](./V18c-version-bump-checklist.md)'s three-way `engines.node` equality gate (operand (i), [`version-bump-step2b.md` step 3](../spec_topics/pi-integration-contract/version-bump-step2b.md)). The four `@earendil-works/pi-*` peers pinned at the loom 1.0 Pi-SDK pin range ([`host-prerequisites.md` §`#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)) and `typebox` pinned `"*"` separately — the two ranges MUST NOT be collapsed — all in `peerDependencies`, are host prerequisites per [`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) item 1. The same four `@earendil-works/pi-*` packages are **additionally** declared in loom's own `devDependencies`, pinned to that same loom 1.0 Pi-SDK pin range ([`host-prerequisites.md` §`#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)), so a fresh `npm install` provisions the SDK under loom's own `node_modules` for the downstream build-test gates that read and import it: [`V18a`](./V18a-capability-inventory.md)/[`V18b`](./V18b-inventory-audit.md)'s `SDK_SURFACE_INVENTORY` surface-inventory presence and closure audits, [`V18c`](./V18c-version-bump-checklist.md)'s `engines.node` three-way equality operand (iii) live read of the installed `@earendil-works/pi-coding-agent` `package.json` and its `loom/typecheck/session-shutdown-reason-snapshot` brand-string `tsc` gate, and the [`H4a`](./H4a-factory-shell-and-harness.md)/[`V18d`](./V18d-version-bump-acceptance.md) harness that compiles and runs the extension. This `devDependencies` declaration is additive to — not a replacement for — the `peerDependencies` entries: `peerDependencies` stays the consumer-facing tilde-pinned skew-detection contract (a consumer installing loom never pulls loom's `devDependencies`), while the four `devDependencies` entries provision loom's own checkout. The four `devDependencies` entries MUST stay lock-stepped to the same pin literal as the four `peerDependencies` entries — the version-bump procedure's step-4 joint move ([`version-bump-step2b.md`](../spec_topics/pi-integration-contract/version-bump-step2b.md)) moves both sets together — otherwise `V18c`'s operand (iii) live read resolves the stale `devDependency`-installed version rather than the candidate pin. `typebox` (pinned `"*"` per its own carve-out, not folded into the four-entry tilde-pinned group) takes the same `devDependencies` provisioning only if a build-test gate imports it. `ajv`/`semver`/`chokidar`/`yaml` belong in loom's own `dependencies` per [`../spec_topics/implementation-notes.md`](../spec_topics/implementation-notes.md) §"Loom-package implementation dependencies (loom 1.0)"; `minimatch` belongs in loom's own `dependencies` as the glob matcher engine for `pi.looms` resolution — the `!`/`+`/`-` override ordering and exact-path operand treatment — grounded in [`discovery/package-and-settings.md` §DISC-5](../spec_topics/discovery/package-and-settings.md#disc-5) (consumed in [`V10b`](./V10b-package-discovery.md)); `yaml` is the YAML-parsing mechanism behind the frontmatter parser (consumed in [`V6a`](./V6a-frontmatter-contract.md)); `ajv` (with `ajv-formats`) is the non-normative reference validator behind the validator-neutral `SchemaValidator` seam — see that page's §"Implementation hint (non-normative)" — not a host-prerequisite contract, so it is not cited to `host-prerequisites.md`. The lint toolchain belongs in loom's own `devDependencies`: `eslint` (the ESLint engine) and `@typescript-eslint/parser` (the parser must be TS-aware so the rules can lint `src/**` `.ts` sources), plus `eslint-plugin-loom-local` (the local custom-rule plugin entry for the bespoke `no-broad-catch` rule). The custom-rule plugin is a distinct provisioning concern from the engine — a local rule module / plugin package must be installable and loadable — so it is enumerated separately rather than folded into the engine entry; its `devDependencies` value may be a relative / workspace specifier (e.g. `file:./tools/eslint-plugin-loom-local`) rather than a published registry version, since the rule lives in loom's own tree. Because the `file:` specifier only resolves during `npm install` when a loadable package already exists at that path, `H1a` also creates the minimal loadable plugin package at `./tools/eslint-plugin-loom-local` — a directory carrying its own resolvable `package.json` plus an entry module that ESLint can load, exporting a placeholder/empty rule set — so the `eslint-plugin-loom-local` devDependency resolves on a fresh checkout. `H1a` owns only the loadable package skeleton at this path; [`H2a`](./H2a-cross-cutting-gates.md) populates the `no-broad-catch` rule body into that package. [`H2a`](./H2a-cross-cutting-gates.md) (`Deps: H1a`) consumes exactly this toolchain — the same three `eslint` / `@typescript-eslint/parser` / `eslint-plugin-loom-local` keys — to build the `no-broad-catch` rule and the `no-restricted-syntax` sequential-by-default allow-list and wire them into `npm test`.

**Tests.**
- `Convention:` (*phase categories*) `npm run build`, `npm test`, and `npm run typecheck` all run green on an empty `src/**` tree.
- `Convention:` (*phase categories*) an architectural test reads `package.json#devDependencies` and asserts `vitest` is declared, so every later leaf's `npm test` and inline-assertion references resolve to one concrete runner + assertion library.
- `Convention:` (*phase categories*) an architectural test reads `package.json#devDependencies` and asserts the three concrete lint-toolchain keys are declared — `eslint`, `@typescript-eslint/parser`, and `eslint-plugin-loom-local` — so `H2a`'s `no-broad-catch` / `no-restricted-syntax` gates resolve to an installed engine, parser, and custom-rule package rather than a toolchain the consuming leaf must invent. The `eslint-plugin-loom-local` assertion checks the key is present and accepts a relative / workspace specifier (e.g. a `file:` value) so it does not demand a published registry version.
- [`host-prerequisites.md` §`pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pic-33) (`PIC-33`): the dependency manifest pins the Pi SDK packages at one tilde-pinned minor line; an architectural test reads `package.json` and asserts the four `@earendil-works/*` peer deps (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`) share that line, so the four move together at a single Pi minor line and no skew across the four is carried.
- `Convention:` (*phase categories*) an architectural test reads `package.json#devDependencies` and asserts the four `@earendil-works/pi-*` packages are declared there at the same tilde-pinned Pi-SDK line as the four `peerDependencies` entries (on the same footing as the `vitest` and lint-toolchain `devDependencies` assertions above), so a fresh `npm install` provisions the SDK under loom's own `node_modules` for the downstream `V18a`/`V18b`/`V18c`/`V18d` gates that read and import it. The provisioning is gated at `H1a` rather than discovered when a downstream `V18` test fails to resolve the module. The assertion fails when any of the four packages is absent from `devDependencies` or carries a line other than the shared four-peer Pi-SDK pin.
- [`host-prerequisites.md` §`pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pic-35) (`typebox` sub-paragraph; `PIC-35`): `typebox`'s own one-line build-time literal-read assertion reads `package.json#peerDependencies` and asserts `typebox` is declared `"*"` as its own entry, not folded into the four-entry `@earendil-works/pi-*` tilde-pinned line. The assertion fails both when `typebox` is given the tilde range and when it is dropped from `peerDependencies` entirely; it is independent of the four-peer shared-line assertion above (both run against `package.json#peerDependencies`).
- [`extension-bootstrap-and-per-loom.md` §"Extension entry point"](../spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md#extension-entry-point): a build-time literal-read assertion reads `package.json#pi.extensions` and asserts it equals `["./extensions"]` — the package-discovery directory Pi resolves loom's extension factory module from. The assertion fails when `pi.extensions` is absent or carries any other value, so a manifest a real Pi host cannot discover fails `H1a`'s own `npm test` rather than shipping green while the end-to-end harness imports the factory directly through a build-internal path. The directory value (`["./extensions"]`) MUST stay consistent with the `extensions/index.ts` factory file path [`H4a`](./H4a-factory-shell-and-harness.md) pins — the manifest entry is a directory, not the file path.
- [`capability-probe.md` §(a)](../spec_topics/pi-integration-contract/capability-probe.md#entry-capability-probe) (and [`overview-and-orientation.md` §"Node version floor"](../spec_topics/overview-and-orientation.md#orientation)): a build-time literal-read assertion reads `package.json#engines.node` and asserts it equals the pinned Node floor owned by `capability-probe.md` §(a) — the same floor `H1a`'s **Adds** populates. The comparison operand is the Node-floor literal transcribed inline into this `H1a` test directly from `capability-probe.md` §(a) — an independently-sourced copy resident in the test itself. At `H1a` sequencing (`Deps: -`) no in-tree single-source constant exists for the test to read: neither `V9a`'s module-resident pinned-constants block nor `V18a`'s `SDK_SURFACE_INVENTORY` `pi-engines-node` row has been authored yet. The test therefore asserts `package.json#engines.node` equals this transcribed literal rather than reading `package.json` against itself — the latter would be a vacuous tautology that never catches the manifest typo this bullet exists to catch. This transcribed literal is an `H1a`-resident copy of the spec floor; a Node-floor move during the version-bump procedure ([`version-bump-step2b.md` step 3](../spec_topics/pi-integration-contract/version-bump-step2b.md)) keeps it aligned with `capability-probe.md` §(a). This is the population-time check `H1a` owns: a comparison of the manifest literal against the transcribed spec-anchored floor, so a typo, stale value, or value disagreeing with `capability-probe.md` §(a) fails `H1a`'s own `npm test` rather than going undetected until `V18c`. The assertion fails when `engines.node` is absent or does not equal the transcribed floor literal. It MUST NOT read `V9a`'s pinned-constants block or `V18a`'s `pi-engines-node` inventory row, nor duplicate `V18c`'s three-way `engines.node` equality gate — the live read of the installed `@earendil-works/pi-coding-agent` floor (operand (iii)) and the `SDK_SURFACE_INVENTORY` operand remain owned by `V18c`.
- [`host-prerequisites.md` §`tsc-toolchain-floor`](../spec_topics/pi-integration-contract/host-prerequisites.md#tsc-toolchain-floor): a build-time literal-read assertion reads `package.json#devDependencies` and asserts the `typescript` key is present and its declared range satisfies the TypeScript floor of `>=5.4.0` owned by `host-prerequisites.md` §`tsc-toolchain-floor` — the `tsc` provider behind the `npm run typecheck` script this leaf defines, sourced from that anchor rather than restated as a fresh plan-side literal (on the same footing as the `engines.node` floor assertion above). The assertion fails when `typescript` is absent from `devDependencies` or carries a range that does not satisfy the spec floor, so a fresh checkout that cannot provision `tsc` at a floor where the brand string surfaces grep-ably fails `H1a`'s own `npm test` rather than surfacing only when `V18c`'s brand-string gate runs.
- [`extension-bootstrap-and-per-loom.md` §"Extension ES-module declaration"](../spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md#extension-esm-type-module): a build-time literal-read assertion reads `package.json#type` and asserts it equals `"module"` — the ES-module declaration the loom runtime requires at loom 1.0, sourced from that anchor rather than restated as a fresh plan-side literal (on the same footing as the `engines.node` and `tsc` floor assertions above) and the obligation `H1a`'s **Adds** populates initial. The assertion fails when `package.json#type` is absent or not `"module"`, so a fresh checkout that cannot load loom's ESM extension fails `H1a`'s own `npm test` rather than surfacing later. The companion family-(4) CJS-reach half of `extension-bootstrap-and-per-loom.md` §"Extension ES-module declaration"'s build-time gate (absence of CommonJS-reach shapes in `src/`) is owned by the inventory-closure audit at [`V18b`](./V18b-inventory-audit.md), not duplicated here.
