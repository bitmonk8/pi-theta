# `H1a` — Project scaffold and toolchain

**Convention.** [`conventions.md`](./conventions.md) (phase categories; cross-cutting rules) and the *Doc updates* cross-cutting rule.

**Adds.** The TypeScript project skeleton — `package.json` (with the four `@earendil-works/pi-*` packages plus `typebox` declared as `peerDependencies`, `ajv`/`semver`/`chokidar`/`yaml`/`minimatch` declared as loom's own runtime `dependencies`, `vitest` declared as loom's own `devDependencies` test runner and assertion library, the lint toolchain — `eslint` (the ESLint engine), `@typescript-eslint/parser` (the TypeScript-aware ESLint parser), and `eslint-plugin-loom-local` (the local custom-rule plugin entry for the bespoke `no-broad-catch` rule) — declared as loom's own `devDependencies`, and an `engines.node` field), `tsconfig.json`, the Vitest test runner and its assertion API, and `npm test` / `npm run build` scripts — on which every later leaf builds. `H1a` is the initial-population owner of the `package.json#engines.node` field; its value is the pinned Node floor owned by [`capability-probe.md` §(a)](../spec_topics/pi-integration-contract/capability-probe.md#entry-capability-probe), sourced from that anchor rather than restated here. This is the literal consumed by the build-time `engines.node` literal-read test ([`overview-and-orientation.md` §"Node version floor"](../spec/overview-and-orientation.md#orientation)) and by [`V18c`](./V18c-version-bump-checklist.md)'s three-way `engines.node` equality gate (operand (i), [`version-bump-step2b.md` step 3](../spec_topics/pi-integration-contract/version-bump-step2b.md)). The four `@earendil-works/pi-*` peers pinned at the loom 1.0 Pi-SDK pin range ([`host-prerequisites.md` §`#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)) and `typebox` pinned `"*"` separately — the two ranges MUST NOT be collapsed — all in `peerDependencies`, are host prerequisites per [`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) item 1. `ajv`/`semver`/`chokidar`/`yaml` belong in loom's own `dependencies` per [`../spec_topics/implementation-notes.md`](../spec_topics/implementation-notes.md) §"Loom-package implementation dependencies (loom 1.0)"; `minimatch` belongs in loom's own `dependencies` as the glob matcher engine for `pi.looms` resolution — the `!`/`+`/`-` override ordering and exact-path operand treatment — grounded in [`discovery/package-and-settings.md` §DISC-5](../spec_topics/discovery/package-and-settings.md#disc-5) (consumed in [`V10b`](./V10b-package-discovery.md)); `yaml` is the YAML-parsing mechanism behind the frontmatter parser (consumed in [`V6a`](./V6a-frontmatter-contract.md)); `ajv` (with `ajv-formats`) is the non-normative reference validator behind the validator-neutral `SchemaValidator` seam — see that page's §"Implementation hint (non-normative)" — not a host-prerequisite contract, so it is not cited to `host-prerequisites.md`. The lint toolchain belongs in loom's own `devDependencies`: `eslint` (the ESLint engine) and `@typescript-eslint/parser` (the parser must be TS-aware so the rules can lint `src/**` `.ts` sources), plus `eslint-plugin-loom-local` (the local custom-rule plugin entry for the bespoke `no-broad-catch` rule). The custom-rule plugin is a distinct provisioning concern from the engine — a local rule module / plugin package must be installable and loadable — so it is enumerated separately rather than folded into the engine entry; its `devDependencies` value may be a relative / workspace specifier (e.g. `file:./tools/eslint-plugin-loom-local`) rather than a published registry version, since the rule lives in loom's own tree. [`H2a`](./H2a-cross-cutting-gates.md) (`Deps: H1a`) consumes exactly this toolchain — the same three `eslint` / `@typescript-eslint/parser` / `eslint-plugin-loom-local` keys — to build the `no-broad-catch` rule and the `no-restricted-syntax` sequential-by-default allow-list and wire them into `npm test`.

**Tests.**
- `Convention:` (*phase categories*) `npm run build` and `npm test` both run green on an empty `src/**` tree.
- `Convention:` (*phase categories*) an architectural test reads `package.json#devDependencies` and asserts `vitest` is declared, so every later leaf's `npm test` and inline-assertion references resolve to one concrete runner + assertion library.
- `Convention:` (*phase categories*) an architectural test reads `package.json#devDependencies` and asserts the three concrete lint-toolchain keys are declared — `eslint`, `@typescript-eslint/parser`, and `eslint-plugin-loom-local` — so `H2a`'s `no-broad-catch` / `no-restricted-syntax` gates resolve to an installed engine, parser, and custom-rule package rather than a toolchain the consuming leaf must invent. The `eslint-plugin-loom-local` assertion checks the key is present and accepts a relative / workspace specifier (e.g. a `file:` value) so it does not demand a published registry version.
- `Convention:` (*phase categories*) the dependency manifest pins the Pi SDK packages at one tilde-pinned minor line; an architectural test reads `package.json` and asserts the four peer deps share that line.
- [`host-prerequisites.md` §`pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin) (`typebox` sub-paragraph): `typebox`'s own one-line build-time literal-read assertion reads `package.json#peerDependencies` and asserts `typebox` is declared `"*"` as its own entry, not folded into the four-entry `@earendil-works/pi-*` tilde-pinned line. The assertion fails both when `typebox` is given the tilde range and when it is dropped from `peerDependencies` entirely; it is independent of the four-peer shared-line assertion above (both run against `package.json#peerDependencies`).
- [`capability-probe.md` §(a)](../spec_topics/pi-integration-contract/capability-probe.md#entry-capability-probe) (and [`overview-and-orientation.md` §"Node version floor"](../spec/overview-and-orientation.md#orientation)): a build-time literal-read assertion reads `package.json#engines.node` and asserts it equals the pinned Node floor owned by `capability-probe.md` §(a) — the same floor `H1a`'s **Adds** populates. This is the population-time check `H1a` owns: a comparison of the manifest literal against the spec-anchored floor, so a typo, stale value, or value disagreeing with `capability-probe.md` §(a) fails `H1a`'s own `npm test` rather than going undetected until `V18c`. The assertion fails when `engines.node` is absent or does not equal the spec floor. It MUST NOT duplicate `V18c`'s three-way `engines.node` equality gate — the live read of the installed `@earendil-works/pi-coding-agent` floor (operand (iii)) and the `SDK_SURFACE_INVENTORY` operand remain owned by `V18c`.
