# `V18d` — Pi version-bump runtime-evidence acceptance gate and revert path

**Spec.** [`../spec_topics/pi-integration-contract/version-bump-intro.md`](../spec_topics/pi-integration-contract/version-bump-intro.md), [`../spec_topics/pi-integration-contract/version-bump-triggers.md`](../spec_topics/pi-integration-contract/version-bump-triggers.md), [`../spec_topics/pi-integration-contract/host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md).

**Adds.** The runtime-evidence acceptance gate (output (c) of the version-bump procedure) — an [`H4a`](./H4a-factory-shell-and-harness.md)-backed end-to-end run of a representative integrated `.loom` (typed query + tool loop + invoke + schema validation + binder + cancellation) against the bumped Pi-SDK pin — and the revert path that restores the prior pin **before merge**: when the double-backed runtime-evidence run is red the restore is pre-merge (the bump MUST NOT merge at the candidate pin), and when the bump's [`H4a`](./H4a-factory-shell-and-harness.md) **manual real-host smoke** — the required pre-merge gate per [`H4a`](./H4a-factory-shell-and-harness.md) — surfaces a confirmed behavioural-divergence finding the restore is likewise pre-merge (the smoke runs before merge, so the divergent pin never lands on `main`). **Blast radius.** A divergent pin's blast radius is every runtime leaf — it is real-host-only and is **not** witnessed by the double-backed acceptance gate above, which is why the pre-merge real-host smoke is the gate that closes the window. The static surface-inventory / `engines.node` / peer-dep / reason-snapshot gates are owned by [`V18c`](./V18c-version-bump-checklist.md).

**Tests.**
- Runtime-evidence acceptance gate (output (c) of [`version-bump-triggers.md`](../spec_topics/pi-integration-contract/version-bump-triggers.md)): the [`H4a`](./H4a-factory-shell-and-harness.md) end-to-end harness runs a representative integrated `.loom` (typed query + tool loop + invoke + schema validation + binder + cancellation) against the bumped Pi-SDK pin and passes; a green surface-inventory run alone does not satisfy this gate. The gate's coverage is bounded by the `H4a` session double's fidelity to the bumped pin (see [`H4a`](./H4a-factory-shell-and-harness.md)) — a green double-backed run is not real-host coverage.

**Deps.** `V18d-T`, `H4a`, `V5d`, `V11f`, `V13c`, `V14a`, `V15a`, `V17a`, `V18c`

**Ships when.** The [`H4a`](./H4a-factory-shell-and-harness.md) end-to-end harness passes against the bumped pin (the runtime-evidence acceptance gate) on `main`. Both triggers restore the prior pin **before merge**. If the bump's acceptance evidence is red — the build-time runtime-evidence / [`H4a`](./H4a-factory-shell-and-harness.md)-harness gate — the prior pin is restored before merge and the bump MUST NOT merge at the candidate pin (matching [`version-bump-triggers.md`](../spec_topics/pi-integration-contract/version-bump-triggers.md)'s "a bump whose runtime-evidence run is red MUST NOT be merged at the candidate pin"). If instead the bump's **manual real-host smoke** (the required pre-merge gate named in [`H4a`](./H4a-factory-shell-and-harness.md), driving a representative `.loom` against a live Pi host before the bump merges) surfaces a confirmed behavioural-divergence finding, the prior pin is restored before merge and the bump MUST NOT merge at the candidate pin — per [`H4a`](./H4a-factory-shell-and-harness.md)'s merge-blocker semantics, which delegate the pre-merge pin-restore for a Pi-bump-triggered smoke finding to this leaf. Because the smoke runs before merge, the divergence signal exists at merge-decision time, so this "restored before merge" precondition references a signal that is present when the restore must fire. In either case the restore reverts step 4's edit in one commit: the single-source-of-truth Pi-SDK pin literal at [`host-prerequisites.md#pi-sdk-pin`](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin) and the four `@earendil-works/*` `peerDependencies` entries (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`). After the prior pin is restored, the contributor MUST re-run [`V18c`](./V18c-version-bump-checklist.md)'s static build-time gates against the restored prior pin — the step-2(a)/2(b) surface-inventory tests, the `engines.node` three-way equality gate, the `peerDependencies` literal-read, and the `loom/typecheck/session-shutdown-reason-snapshot` brand-string gate — and confirm they pass green before the revert is merged. That green re-run is what establishes the revert is complete: if any gate is red, the revert did not restore consistency (a co-edited capability-probe constant, the `engines.node` literal, or the snapshot entry was left at its candidate value) and the revert MUST be widened to restore those operands in the same commit.
