## Finding

The **Resolution** paragraph in `docs/spec_topics/invocation.md` closes with a long `*Non-normative implementation note.*` trailer covering three implementation-facing topics: (a) that on case-insensitive hosts Node's `fs.promises.realpath` returns the on-disk casing of the resolved tail so filesystem-equivalent paths canonicalise to the same byte sequence; (b) why the byte-exact comparison is correct on each host class and why symlink-farm callees crossing case-variant directory entries stay well-defined; and (c) that the test `FakeFileSystem` mimics per-host `realpath` casing. None of this is part of the resolution rule. The rule is fully specified before the trailer — the comparison is decided on the byte-exact output of `FileSystem.realpath` (whose normative full-transitive-symlink-resolution and byte-stable canonical-output guarantees PIC-13 owns), composed with the segment-boundary within-root predicate. The trailer is rationale for why that rule is well-defined on the current Pi host plus a test-fake correspondence note, both of which belong with implementation guidance.

The trailer's placement also widens the Resolution paragraph past readability: a reader scanning Resolution for the normative containment contract must wade past Node-specific behaviour, host-class case-analysis, and test-fake correspondence to confirm they have reached the end of the paragraph. The peer INV-1 boxed note that immediately follows is correctly placed — it grounds the single-named-function seam-factoring decision that the deferred `openat2(..., RESOLVE_NO_SYMLINKS | RESOLVE_BENEATH)` extension hangs off, so it is load-bearing context for the seam itself, not background rationale for a separately-stated rule.

`docs/spec_topics/implementation-notes.md` already has a Runtime section that hosts exactly this kind of host-specific runtime guidance and a Parser section for parser-side equivalents; it is the canonical destination.

The two later edits to the same paragraph (`bd635ce` adding the segment-boundary containment predicate; `6ab9f7b` linking `FileSystem.realpath` to PIC-13 and pinning the transitive-resolution / byte-stability guarantees) carried the trailer through unchanged. The defect entered as the case-sensitivity resolution shipped its rationale inline rather than routing it to `implementation-notes.md`; subsequent rule-tightening commits preserved the misplacement.

## Solution Space

**Shape:** single

### Recommendation

Delete the `*Non-normative implementation note.*` trailer from the Resolution paragraph in `docs/spec_topics/invocation.md` (the three sentences starting "On case-insensitive hosts Node's `fs.promises.realpath`…" through "…the test `FakeFileSystem` mimics per-host `realpath` casing accordingly."), and append an equivalent bullet to the Runtime section of `docs/spec_topics/implementation-notes.md` under a new sub-bullet anchored from the Resolution paragraph (a parenthetical "(see [Implementation Notes — Runtime](./implementation-notes.md))" at the point where the trailer used to sit). Preserve the three substantive points verbatim:

- On case-insensitive hosts Node's `fs.promises.realpath` returns the on-disk casing of the resolved tail, so filesystem-equivalent paths canonicalise to the same byte sequence and the byte-exact comparison is correct; on case-sensitive hosts byte-exactness already coincides with filesystem equivalence.
- A symlink-farm callee whose `realpath` crosses a case-variant directory entry stays well-defined because both sides are compared on `realpath`'s output.
- Stating the rule as byte-exact on `realpath` output rather than as any platform's case behaviour is the guard a future port to a host whose `realpath` does not preserve on-disk casing must respect — the test `FakeFileSystem` mimics per-host `realpath` casing accordingly.

Leave the INV-1 boxed implementation note in place: it documents the single-named-function seam-factoring decision the deferred `openat2(..., RESOLVE_NO_SYMLINKS | RESOLVE_BENEATH)` extension hangs off, which is load-bearing context for the seam blockquote and would lose its anchor if relocated.

Edge cases for the implementer: (i) the new `implementation-notes.md` bullet must sit in the **Runtime** section (the casing behaviour is observed at invocation/load time, not parse time), and should be co-located with — or cross-referenced from — the existing static-resolution load-pass bullet that already cites `invocation.md`, so a reader navigating from either direction lands at the right adjacency; (ii) keep the back-reference from `invocation.md` to `implementation-notes.md` Markdown-anchor-stable (add an explicit `<a id="...">` if the new bullet's auto-generated anchor is fragile); (iii) the related finding "`realpath` leaf/byte canonicalisation guaranteed but no mechanism given" against PIC-13 may later force a concrete `fs` call sequence into PIC-13's normative body — when that lands, fold any redundancy between this Runtime bullet and the PIC-13 mechanism into a single owner, with the Runtime bullet keeping only the Node-host-specific rationale.

## Consequence

**Severity:** cosmetic

The Resolution paragraph's normative content is unchanged by removing the trailer; the rule (`byte-exact` `FileSystem.realpath` output, no independent case-folding, segment-boundary containment) already stands without it. Cost is reader load on the longest paragraph in `invocation.md` and a placement-lens defect: implementation-facing host/test rationale is mixed into the normative resolution rule.
