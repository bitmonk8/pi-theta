## Finding

The Per-loom registration paragraph in `extension-bootstrap-and-per-loom.md` defines the `ToolDefinition.label` field as "the loom file's basename with hyphens preserved and the leading character capitalised", then gives `summarise.loom → "Summarise"` and `code-review.loom → "Code-review"` as the two worked examples. Read against `path.basename` (the only natural-language sense of "basename" in a Node-based runtime, and the one implementers will reach for), `path.basename("summarise.loom")` is `"summarise.loom"` — the worked examples and the prose rule are inconsistent, because the examples plainly strip the `.loom` extension before capitalising.

An implementer following the prose materialises labels like `"Summarise.loom"` / `"Code-review.loom"`; an implementer following the examples materialises `"Summarise"` / `"Code-review"`. The leaf that asserts this contract (`V9f`) restates the prose ("basename with interior hyphens preserved") while pinning the example output `"Code-review"`, so its `npm test` assertion catches the divergence at one site but does not remove the ambiguity at the spec source — and a typed-query one-shot tool's literal label `"Loom typed-query response"` (also pinned in this same bullet) gives no precedent either way for whether the extension is part of the input.

The right reading is the one the examples already encode: the input to the capitalisation step is the filename with the `.loom` extension removed (the "stem"). The fix is a one-word substitution in the rule's input phrase.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md`, in the Per-loom registration bullet defining `label`, replace "the loom file's basename with hyphens preserved" with "the loom filename stem (the filename with the trailing `.loom` extension removed), with hyphens preserved". Keep the rest of the sentence — the capitalisation rule, both worked examples (`summarise.loom → "Summarise"`, `code-review.loom → "Code-review"`), and the synthesised typed-query one-shot tool's literal `"Loom typed-query response"` — unchanged; the examples already encode the intended algorithm and stay correct.

Edge cases for the implementer to watch:

- The capitalisation rule is "leading character capitalised" (i.e. only the first character of the stem, not each hyphen-separated segment) — `code-review.loom → "Code-review"`, not `"Code-Review"`. This is the property `6d8b472` already corrected and that `V9f` asserts; keeping the example unchanged preserves it.
- The stem is computed before capitalisation; an input filename with no `.loom` extension is out-of-scope at this surface (the registration is keyed on `.loom` discovery per the Directory Convention), so no separate "if the extension is missing" branch is needed.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: a "basename" reader produces operator-visible slash-command tool labels carrying the `.loom` suffix (`"Summarise.loom"`, `"Code-review.loom"`), while an "examples" reader produces stripped labels (`"Summarise"`, `"Code-review"`). The divergence surfaces in the Pi slash-command UI and in any model-visible tool metadata at the moment the loom is exposed as a tool to another loom, with no diagnostic to flag the mismatch.
