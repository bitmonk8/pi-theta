## Finding

`V6c`'s `Adds.` explicitly claims ownership of "the default for absent/empty `tools:`". The spec pins this default normatively: in `frontmatter-fields-a.md`, the `tools` field-contract row and the `tools` prose state that an absent `tools:` field resolves to an **empty callable set** (the model cannot make tool calls and loom code has no `<name>(...)` callables), that the Pi session's ambient tools are *deliberately not inherited*, and that `tools: []` and an absent `tools:` field are equivalent.

Neither `V6c-T`'s nor `V6c`'s `Tests.` block asserts the resolved callable set for that default path. The three Tests bullets cover only: (i) `loom/load/prompt-mode-callable` rejection of a prompt-mode `.loom` callee, (ii) `loom/load/tool-name-collision` plus `as` rename resolution, and (iii) "the resolved callable set is frozen (no ambient inheritance); both YAML spellings parse." Bullet (iii)'s "no ambient inheritance" clause is an assertion about freezing an *explicitly-listed* callable set, and "both YAML spellings" names the flow/comma vs. block list forms — neither exercises the absent-or-empty-resolves-to-empty default. The spec-owned default therefore ships with no closing test, so a regression in which the absent/empty case silently inherits the Pi session's ambient tools (the exact "why did my loom touch the filesystem?" surprise the spec calls out) passes the TDD gate.

## Solution Space

**Shape:** single

### Recommendation

Add a Tests bullet to `V6c-T`'s `Tests.` block asserting that an absent `tools:` field and an explicit `tools: []` both resolve to the empty callable set with no ambient-tool inheritance (the model has no callable tools and loom code resolves no `<name>(...)` callable). Mirror the same bullet into `V6c`'s `Tests.` block so the paired implementation leaf and its `-T` partner stay aligned 1:1, per the per-phase TDD ritual.

A concrete bullet body, citing the spec's equivalence: `` Absent `tools:` and explicit `tools: []` both resolve to the empty callable set — the model has no callable tools and loom code resolves no `<name>(...)` callable; the Pi session's ambient tools are not inherited (`frontmatter-fields-a.md`, `tools` row). `` Keep the assertion on the resolved callable-set value; do not restate the spec's diagnostic taxonomy here, since the absent/empty case is a successful load with no diagnostic.

No spec edit is required: `frontmatter-fields-a.md` already pins the default and the `[]`≡absent equivalence. The coverage matrix needs no new row either — `V6c` is already a closing leaf for `frontmatter/frontmatter-fields-a.md` (FRNT) in `coverage-matrix.md`.

## Consequence

**Severity:** advisory

The spec-owned absent/empty `tools:` default (empty callable set, no ambient-tool inheritance) has no asserting test, so a leaf that silently inherits the Pi session's ambient tools — or otherwise resolves the absent/empty case wrong — turns its exit gate green. The gap is a coverage hole, not an ambiguity: the requirement is stated in both `V6c`'s `Adds.` and the spec, so implementers know the target, but no automated witness guards it against regression.
