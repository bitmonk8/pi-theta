## Finding

`V5e` inlines the loom-owned JSON-depth counting algorithm in its **Adds** (`ceiling = 5; scalar/empty = 1; non-empty = 1 + max child; \`anyOf\` arms are not levels`) and restates the last clause in a **Tests** bullet ("`anyOf` arms are not counted as levels"); `V5e-T` carries the same Tests bullet. The clause `` `anyOf` arms are not levels `` is copied from `schema-subset.md` §Depth Enforcement **without** the spec's load-bearing disambiguator, which reads in full: "`anyOf` arms are not levels: depth is measured against the **materialised** value at runtime (one arm exists; the others do not)."

Stripped of that parenthetical the clause admits two readings: (a) the `anyOf` node is transparent and an arm's depth is measured from the enclosing container, versus (b) the depth counter resets to 1 inside each arm. The spec means (a) — more precisely, the walk runs over the post-decode runtime JSON value, which never contains an `anyOf` node at all, so the node simply never appears and reading (b) is structurally impossible. The leaf's inline algorithm presents itself as a self-contained restatement of the spec rule, yet the one clause where the spec needed an explicit disambiguation is the one clause it abbreviates. No `V5e`/`V5e-T` Tests vector exercises an `anyOf`-shaped value, so the ambiguity is not closed by the test surface either.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review") introduced the abbreviated `` `anyOf` arms are not levels `` clause in `V5e`'s **Adds** at file creation; `9d880b8` (2026-06-17, "resolve \"V5e before-AJV Tests bullet asserts an ordering its isolated harness cannot witness\"") later added the "`anyOf` arms are not counted as levels" depth-computation Tests bullet to `V5e`/`V5e-T`, propagating the same abbreviation into the test surface.
**History:** `git log --follow` on both leaf files and `git log -S` on the two clause variants show the Adds clause has carried the imprecise phrasing since the leaf was first authored; the Tests-side restatement inherited it from the Adds when the depth-computation bullet was added. The defect is original to the leaf, not a regression introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

Restore the spec's disambiguating clause everywhere the abbreviated form appears. In `docs/plan_topics/V5e-depth-enforcement.md` **Adds**, replace `` `anyOf` arms are not levels `` with the spec's framing — e.g. `` `anyOf` arms are not levels (depth is measured against the materialised runtime value: exactly one arm exists, the others do not) ``. In the depth-computation **Tests** bullet of both `docs/plan_topics/V5e-depth-enforcement.md` and `docs/plan_topics/V5e-T-depth-enforcement.md`, replace "`anyOf` arms are not counted as levels" with the same disambiguated wording, keeping it aligned with `schema-subset.md` §Depth Enforcement. The spec is the authoritative source for the exact text; mirror it rather than paraphrasing. `schema-subset.md` is read-only for this fix — the spec already states the rule correctly.

## Consequence

**Severity:** advisory

An implementer working from the leaf's inline algorithm alone could read "`anyOf` arms are not levels" as resetting the depth counter within an arm and waste effort handling a case the runtime value can never present. The walk runs on the post-decode runtime JSON value (which contains no `anyOf` node), so a correct implementation that consults the cited `schema-subset.md` will not diverge; the gap is in the leaf's self-described summary, not in the spec.
