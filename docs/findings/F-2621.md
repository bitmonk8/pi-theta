## Finding

The **Depth Enforcement** section of `docs/spec_topics/schema-subset.md` runs two distinct integer-numbering schemes side by side within the same span of prose and tables:

- A **section-local** enumeration of the five depth-walk enforcement points — the numbered list (`1.`–`5.`) and the per-boundary table whose first column keys rows `#1`–`#5`, referenced downstream in prose as `(row #1)` … `(row #5)` and bare `#1`–`#5`.
- The **corpus-wide hard-ceiling** enumeration `ceiling #1` … `ceiling #4`, whose canonical owner is `overview-and-orientation.md#hard-runtime-ceilings` and `hard-ceilings/ceilings-3-and-4.md`. The Depth Enforcement section cites these as `ceiling #2`, `ceiling #3`, and `ceiling #4`.

The two schemes share the digits `2`, `3`, and `4`, and the collisions land in adjacent table cells: row `#2` (model-driven tool-call args) says "unless the loop later exhausts under **ceiling #2**"; row `#4` (`params` validation) says "routes through **ceiling #3**'s no-retry classification"; the closing paragraph references the per-boundary table "for **ceiling #4**". Disambiguation rests entirely on the adjacent token (`ceiling` vs `row`/bare `#`), which is easy to lose when scanning a dense table — a reader can momentarily map `ceiling #3` onto enforcement point `#3` (code-driven tool args) or `ceiling #4` onto enforcement point `#4` (`params` validation), which are unrelated referents.

The references all resolve correctly on careful reading, so this is a clarity/readability defect, not a correctness one: the section is internally consistent and the canonical `ceiling #N` form matches the rest of the corpus.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `1750848` ("pi-loom spec: resolve \"Ceiling #4 routing class collapses four boundaries whose destinations differ\"", 2026-05-07)
**History:** The collision did not exist before this commit. `git show 1750848^:spec_topics/schema-subset.md` contains neither the `| Enforcement point | Destination |` table nor the `ceiling #2` / `ceiling #3` tokens. The single edit `1750848` introduced the `#1`–`#5` per-boundary enforcement-point table and, in the very same cells and surrounding prose, the `ceiling #2` / `ceiling #3` / `ceiling #4` references — so the two overlapping `#N` schemes were born together in that commit (`git log -S'Enforcement point | Destination'` and `git log -S'ceiling #3' -- docs/spec_topics/schema-subset.md` both first attribute the tokens to `1750848`; later commits only refine the table). The corpus-wide `ceiling #N` form predates this and is unchanged.

## Solution Space

**Shape:** single

### Recommendation

Leave the `ceiling #1`…`ceiling #4` references untouched — that numbering is the canonical corpus-wide form (owned by `overview-and-orientation.md#hard-runtime-ceilings` and reused throughout `hard-ceilings/ceilings-3-and-4.md`) and must not be renumbered for one section's convenience. Remove the collision on the **section-local** side: re-label the five depth-walk enforcement points with a non-numeric identifier scheme so the two enumerations never share a token. Concretely, key the per-boundary table's first column by name or letter (e.g. `EP-a` … `EP-e`, or the existing site names "Typed-query response", "Tool-call args, model-driven", …) and update the in-section prose that currently says `(row #1)` … `(row #5)` / bare `#1`–`#5` to match. After the edit, every `ceiling #N` reference reads against a section-local scheme that uses no bare digits.

Edge cases for the implementer:
- Do **not** rename the enforcement points to `CIO-N`. `CIO-1`…`CIO-6` (in `hard-ceilings/ceilings-3-and-4.md`) is the *ceiling-interaction-order* scheme, distinct from the ceiling identities — `CIO-4` actually concerns `ceiling #2`. The original "hard ceiling CIO-4" phrasing conflates the two and would introduce a fresh error.
- The five enforcement-point sites also appear (in the same order) in `hard-ceilings/ceilings-3-and-4.md` (`#ceiling-4-table`, `CIO-3`). If the local labels are changed, keep that sibling page's references readable — though it uses prose site-names rather than `#N`, so no cross-page renumbering is forced.

## Consequence

**Severity:** cosmetic

No observer's behaviour diverges and no implementer would build a different system: every reference resolves unambiguously once the `ceiling` / `row` qualifier is read. The cost is avoidable cognitive friction when cross-reading two `#N` schemes that reuse the same digits in adjacent table cells.
