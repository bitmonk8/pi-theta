## Problem

Three normative sites — FRNT-1 (`frontmatter/frontmatter-fields-b-and-templates.md`),
QRY-16 (`query/query-tool-loop.md`), and the "No parallel fan-out across
prompt-mode `.loom` callees" non-goal
(`future-considerations/model-changes-and-non-goals.md`) — phrase
parallel-tool dispatch as conditioned on a per-provider predicate: "the
runtime executing them all (in parallel where the provider supports
parallel tool calls)". The phrasing reads as a loom-side branch, but the
spec names no runtime-evaluable signal the runtime would consult to
evaluate that predicate: no capability field on `Model<Api>`, no pi-ai
capability flag, no provider allowlist, and no host-prerequisite
presupposition. Two conforming implementers can each pick a different (or
no) predicate and produce divergent observable batch shapes for the same
loom against the same model, with no way to choose between them from the
spec text.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 82be125 — spec: adopt P1 typed-query two-phase loop (Option A) (2026-05-04, Thomas Andersen); 0e12140 — pi-loom spec: resolve "Non-goals are not consolidated into a single section" (2026-05-08, Thomas Andersen)
**History:** The "in parallel where the provider supports parallel tool calls" phrase entered the corpus in 82be125 when the typed-query two-phase loop was adopted; that commit authored the FRNT-1 `tool_loop` paragraph (then on `frontmatter.md`) and the QRY-16 `Tool-call loop bound` paragraph (then on `query.md`) with the unsourced predicate already in place. 0e12140 propagated the same phrase verbatim into the consolidated `future-considerations` non-goals when the "No parallel fan-out across prompt-mode `.loom` callees" disposition was lifted into its own normative bullet, extending the predicate's reach to a third normative site. Subsequent edits (the 100KB split in f5e89f4, the FRNT-1 anchor coining in 4af6fb3, the CIO-4 cross-references in 0e2eb05) carried the phrasing through unchanged. The defect is therefore the interaction of "predicate-shaped phrasing without a named signal" (82be125) with "non-goal scope grounded on that same predicate" (0e12140) — fixing only one site leaves the other contradicting it.

## Solution approach

Rewrite the dispatch parenthetical in FRNT-1 (`frnt-1`) and QRY-16
(`qry-16`) to describe the parallel/serial choice as Pi-side behaviour the
runtime observes rather than a predicate it gates on, forward-linking to
`tool-calls.md`'s `#parallel-tool-batch-delivery-presupposition`
consumption posture. Re-anchor the "No parallel fan-out across prompt-mode
`.loom` callees" non-goal's cross-reference from FRNT-1's predicate
phrasing to that same `#parallel-tool-batch-delivery-presupposition`
anchor.

## Solution constraints

- Preserve QRY-16's round-boundary slot-counting rule (one assistant
  message emitting any number of `tool_use` blocks consumes one round);
  the rephrasing MUST NOT reintroduce a per-`tool_use`-block round reading.
- Out of scope: `tool-calls.md`'s Concurrency section and the
  `#parallel-tool-batch-delivery-presupposition` posture
  (settle-all-before-next-turn, per-sibling independent lowering, V14b) —
  forward-link to it, do not restate or weaken it.
