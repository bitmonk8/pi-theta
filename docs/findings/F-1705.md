## Finding

Three normative sites in the spec corpus phrase parallel-tool dispatch
behaviour as conditioned on a per-provider predicate — "the runtime
executing them all (in parallel where the provider supports parallel
tool calls)":

- `frontmatter/frontmatter-fields-b-and-templates.md` § FRNT-1
  (`tool_loop`),
- `query/query-tool-loop.md` § QRY-16 (every query's tool-call loop),
- `future-considerations/model-changes-and-non-goals.md` § "No parallel
  fan-out across prompt-mode `.loom` callees" (which reuses the FRNT-1
  phrase verbatim to ground its scope statement).

The phrasing reads as a runtime-side branch — *if* the provider
supports parallel tool calls, the runtime dispatches them concurrently;
otherwise, sequentially. But the spec never names the signal the
runtime would consult to evaluate that predicate. There is no capability
field on `Model<Api>`, no `pi-ai` capability flag (the corpus is in fact
explicit that the related seed-field mapping "is not derived from any
pi-ai capability flag", per `pi-integration-contract/provider-error-
mapping.md`), no provider allowlist of the sort the typed-query
forced-respond mechanism uses (`pi-integration-contract/conversation-
drive.md` § "Provider compatibility for typed queries" enumerates
`{anthropic-messages, openai-completions, mistral, amazon-bedrock}` via
pi-ai's named-tool `toolChoice` mapping), and no host-prerequisite
presupposition naming such a signal.

`tool-calls.md` § "Concurrency" *does* describe concurrent execution as
upstream Pi-side behaviour, and the parallel-tool-mode batch shape is
recorded as a consumption posture re-audited on each Pi minor bump
(`pi-integration-contract/version-bump-step2.md` checklist item (an)
`#bump-checklist-parallel-tool-batch-delivery`). But none of those
locations specify a runtime-evaluable predicate either; they describe
Pi-side behaviour the runtime *observes* rather than a signal the
runtime *reads*. The three normative sites are therefore underspecified
in the only direction the phrase invites the reader to look: two
conforming implementers can each pick a different (or no) predicate
and produce divergent dispatch behaviour, with no way to choose
between them from the spec text.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 82be125 — spec: adopt P1 typed-query two-phase loop (Option A) (2026-05-04, Thomas Andersen); 0e12140 — pi-loom spec: resolve "Non-goals are not consolidated into a single section" (2026-05-08, Thomas Andersen)
**History:** The "in parallel where the provider supports parallel tool calls" phrase entered the corpus in 82be125 when the typed-query two-phase loop was adopted; that commit authored the FRNT-1 `tool_loop` paragraph (then on `frontmatter.md`) and the QRY-16 `Tool-call loop bound` paragraph (then on `query.md`) with the unsourced predicate already in place. 0e12140 propagated the same phrase verbatim into the consolidated `future-considerations` non-goals when the "No parallel fan-out across prompt-mode `.loom` callees" disposition was lifted into its own normative bullet, extending the predicate's reach to a third normative site. Subsequent edits (the 100KB split in f5e89f4, the FRNT-1 anchor coining in 4af6fb3, the CIO-4 cross-references in 0e2eb05) carried the phrasing through unchanged. The defect is therefore the interaction of "predicate-shaped phrasing without a named signal" (82be125) with "non-goal scope grounded on that same predicate" (0e12140) — fixing only one site leaves the other contradicting it.

## Solution Space

**Shape:** single

### Recommendation

Reframe the three sites to drop the loom-side "where the provider
supports parallel tool calls" predicate and instead describe the
parallel/serial choice as Pi-side / pi-ai-side behaviour the runtime
observes rather than gates on. Concretely:

- In FRNT-1 (`frontmatter-fields-b-and-templates.md`), replace "the
  runtime executing them all (in parallel where the provider supports
  parallel tool calls)" with "the runtime dispatching the model's
  `tool_use` blocks to Pi / pi-ai's provider adapter and awaiting the
  whole batch to settle; whether the adapter executes the siblings
  concurrently or serially is a Pi-side property loom does not gate on,
  recorded as the parallel-tool-mode batch-delivery consumption
  posture at [Tool Calls — Concurrency](../tool-calls.md#parallel-tool-batch-delivery-presupposition)".
- In QRY-16 (`query/query-tool-loop.md`), apply the same rephrasing to
  the round-definition parenthetical. Keep the slot-counting rule ("a
  model that emits three parallel tool calls in one round consumes one
  slot") — that rule is independent of dispatch shape and only requires
  that the runtime treats one assistant message as one round regardless
  of how many `tool_use` blocks it contained.
- In the "No parallel fan-out across prompt-mode `.loom` callees"
  non-goal (`future-considerations/model-changes-and-non-goals.md`),
  re-anchor the cross-reference from FRNT-1's old phrasing to the
  `#parallel-tool-batch-delivery-presupposition` anchor so the
  non-goal's scope statement rests on the same Pi-side consumption
  posture rather than on a predicate the spec never defines.

Edge cases the implementer must watch:

- The slot-counting rule must remain a round-boundary rule, not a
  per-`tool_use`-block rule; the dispatch-shape rephrasing must not
  accidentally re-introduce the loose reading that each parallel call
  is a separate round.
- The non-goal's exclusion mechanism (prompt-mode `.loom` paths
  rejected from `tools:` at load time) is independent of the dispatch
  predicate and stays intact; only its grounding cross-reference
  changes.
- The reworded FRNT-1 / QRY-16 phrasing should not weaken `V14b`'s
  observable contract (settle-all-before-next-turn, per-sibling
  independent lowering). Those properties stay on `tool-calls.md`
  unchanged; the rephrasing only removes the implication of a
  loom-side predicate.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on dispatch behaviour: one reads
"where the provider supports" as a loom-side branch and either invents
a predicate or hard-codes an allowlist; the other treats the
parallel/serial choice as fully delegated to Pi / pi-ai and always
forwards the model's batch unconditionally. The two implementations
produce different observable batch shapes for the same loom against the
same model, and the round-counting rule ("a parallel batch counts as
one slot", `cka-13` / `V14b`) is then anchored on whichever branch each
implementer took. The non-goal's scope statement inherits the same
ambiguity, leaving the boundary of "fan-out across prompt-mode `.loom`
callees" undefined for any provider the spec does not classify.
