## Finding

`V14c` (Code-side tool-call off-surface outcome routing) names its first off-surface outcome, in all three of its **Adds**, **Tests**, and **Ships when** fields, as "a pre-eval setup throw inside the `.loom`-callable **parallel-batch** adapter." The same wording is carried in the paired `V14c-T` Tests bullet. The spec section the leaf cites inline — `tool-calls.md` §Outcome enumeration — titles this case "Pre-evaluation setup throw inside the `.loom`-callable adapter," with **no** "parallel-batch" qualifier. The spec body does describe the adapter as one "entered from the model's parallel-tool batch," so the qualifier is not invented, but it is not part of the adapter's canonical name.

The `.loom`-callable adapter and its `execute()` setup-throw machinery are owned by `V14a` (Tool calls, code-side), which `V14c.Deps` (`V14c-T`, `V14a`, `V8a`, `V4d`) declares and which `V14c`'s **Adds** explicitly builds on. `V14b` owns a distinct deliverable: the model-driven parallel tool-call batch handling — the `Promise.allSettled` settle-all-before-next-turn orchestration and per-sibling independent lowering. `V14c`'s per-call off-surface routing consumes none of that batch-orchestration machinery, so `V14b` is correctly absent from `V14c.Deps`.

The defect is that the word "parallel-batch" in the adapter's name reads as a reference to `V14b`'s deliverable, contradicting the cited spec heading and creating the appearance of a missing `V14b → V14c` dependency edge. The disposing evidence (the spec heading the leaf itself cites, plus `V14a`'s ownership of the adapter) shows the adapter is `V14a`'s and the Deps are already correct; it is the prose that is loose.

## Consequence

**Severity:** advisory

An implementer reading "the `.loom`-callable parallel-batch adapter" can reasonably conclude `V14c` consumes `V14b`'s settle-all batch machinery and add a spurious `V14b` edge to `V14c.Deps`, over-sequencing the build (V14c would needlessly wait on V14b). The leaf still produces a working result either way, but two implementers would diverge on the dependency set and on which adapter the setup-throw routing targets.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V14c-tool-calls-off-surface-routing.md` (the **Adds**, **Tests**, and **Ships when** fields) and `docs/plan_topics/V14c-T-tool-calls-off-surface-routing.md` (the Tests bullet), name the outcome's site as the cited spec heading names it — "the `.loom`-callable adapter" — dropping the "parallel-batch" qualifier so the leaf prose no longer reads as a reference to `V14b`'s batch-orchestration deliverable. Leave `V14c.Deps` (`V14c-T`, `V14a`, `V8a`, `V4d`) unchanged: `V14a` owns the adapter and its `execute()` setup-throw machinery, and `V14c`'s per-call off-surface routing consumes none of `V14b`'s settle-all / independent-lowering batch handling, so no `V14b` edge is warranted.
