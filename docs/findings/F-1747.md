## Finding

The phrase *session-only reasons* — defined operationally as the closed set `{"new", "resume", "fork"}` — is used as a coined technical partition label, italicised at every site, across `docs/spec_topics/session-model-and-appendix.md` (SM-4 names the term in its heading; SM-5 and SM-6 cross-reference SM-4 via the rendered text "session-only reasons"), `docs/spec_topics/pi-integration-contract/session-only-degraded-state.md` (the parenthetical "the three *session-only reasons*: Pi swaps the user session without tearing the extension runtime down" introduces it inline), and `docs/spec.md`'s Session model paragraph. It also appears unmarked in `docs/spec_topics/pi-integration-contract/host-prerequisites.md`, `…/version-bump-step2.md`, and `…/active-invocation-registry.md`, and is referenced indirectly via the per-branch acceptance criteria on the PIC page (branches (a)/(b) keyed on "clean session-only reason" / "session-only reason with one sub-step throw").

The corpus has a Glossary page at `docs/spec_topics/glossary.md` whose stated purpose is exactly "to collect the spec's coined vocabulary in one alphabetised list" with a `See:` pointer to the canonical defining page. The glossary already carries entries for analogous multi-page coined partition labels (`session-only` does not appear; `callable set`, `schema slug`, `respond-repair`, `occurrence` vs. `origin`, `prompt mode` / `subagent mode`, etc., do). The session-only-reasons partition is on the same footing as those — it is a multi-page coined name for a closed-set subset that gates a normative branch (the degraded-state branch's tag-transition predicate, diagnostic-emission predicate, and SM-4/SM-5/SM-6 implementation gate) — and its absence from the glossary is the asymmetry the finding flags. A reader landing on SM-4 (or on any of the indirect callsites in PIC) currently must navigate to the deep PIC `session-only-degraded-state.md` page to recover the `{"new", "resume", "fork"}` enumeration and the partition rationale (Pi's `session_shutdown` teardown-vs-session-swap distinction).

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `52b9101` ("pi-loom spec: resolve T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state") for the PIC-page coinage; `40ffdc5` ("pi-loom spec: resolve T19a — Replace session-model paragraph with eight SM-N sub-units") for the session-model SM-4/SM-5/SM-6 callsites.
**History:** `git log -G '\*session-only reasons\*' --reverse --all` shows the italicised coinage first landing in `52b9101` on `docs/spec_topics/pi-integration-contract.md` (the pre-shard PIC page) as part of the degraded-state branch design. The SM-restructure commit `40ffdc5` then propagated the term into the new SM-4 heading and the SM-5/SM-6 cross-references on `docs/spec.md`'s Session model paragraph, and the subsequent page-collapse commit `81bf52a` moved the bodies into the per-page topic files where they now live. `git log -- docs/spec_topics/glossary.md` shows the glossary has been edited 26 times across the corpus's lifetime — including for analogous coined partition labels — but no commit ever added a `session-only reasons` entry. The defect is therefore an omission contemporaneous with the term's coinage in `52b9101`: the glossary-entry obligation was never satisfied, not regressed.

## Solution Space

**Shape:** single

### Recommendation

Add a single alphabetised entry to `docs/spec_topics/glossary.md` between `respond-repair` and `schema slug` (or whatever position the alphabetisation actually dictates — the page is alphabetised by term head-word) with the following shape, modelled on the existing partition-label entries (`prompt mode` / `subagent mode`, `occurrence` vs. `origin`):

> **session-only reasons** — The subset `{"new", "resume", "fork"}` of `SessionShutdownEvent['reason']`'s closed set on which Pi swaps the user session without tearing the extension runtime down, partitioned against the complement `{"quit", "reload"}` (the *always-tear-down reasons*) by Pi's documented `session_shutdown` teardown-vs-session-swap distinction. The partition is the gating predicate for the degraded-state branch: the post-sub-step-4 `LoomRegistry` drain-state transition to `"degraded-needs-reload"`, the `loom /<name>: extension degraded; /reload to recover` slash note, and the `loom/host/session-shutdown-runtime-degraded` diagnostic emission. The triplet is inlined as a normative runtime literal at the `loom/host/session-shutdown-runtime-degraded` emission site rather than read off the `SessionShutdownEvent['reason']` snapshot's `literals` field; the inline triplet is kept in lock-step with the snapshot under the Pi version bump procedure's inbound-reference sweep. See: [Pi Integration Contract — `reason: "new" | "resume" | "fork"` edge-case bullet](./pi-integration-contract/session-only-degraded-state.md#session-only-reason-degraded-state); cross-referenced by [Session Model — SM-4](./session-model-and-appendix.md#sm-4-session-only-degraded-state).

Edge cases the implementer must watch:

- The companion complement label *always-tear-down reasons* is used in the corpus (e.g. branch (c) of the PIC acceptance criteria, "Negative — always-tear-down reasons") and is at the same coinage tier. The recommended entry above mentions it parenthetically; the corpus is small enough that one entry covers both halves, but a separate alphabetised entry for *always-tear-down reasons* with a back-reference is equally acceptable if a future grep gate prefers head-word lookup.
- Do not duplicate normative content on the glossary page — defer to the canonical PIC bullet for the partition rationale, the predicate definitions, and the inline-triplet pin, on the same footing as every other glossary entry.
- The italicisation at all existing callsites (`*session-only reasons*` on SM-4 and the inline PIC introduction) is the coinage convention used elsewhere on the glossary — leave the callsites untouched; only the glossary page is edited.
- The glossary's preamble already pins the precedence rule ("if a glossary entry and its canonical page disagree, the canonical page wins"), so the new entry inherits that precedence without restating it.

## Consequence

**Severity:** advisory

A first-time reader hitting SM-4, SM-5, or SM-6 (or any indirect PIC callsite) cannot resolve the coined term locally and must traverse to the deep PIC page to recover the `{"new", "resume", "fork"}` membership and the Pi-side rationale for the partition. Implementers can still produce a correct implementation because every normative obligation that consumes the partition is anchored to the canonical PIC bullet, but the glossary's stated role as the single discovery surface for coined vocabulary is undermined — and the corpus-wide consistency invariant that every multi-page coined name has a glossary entry is silently relaxed.
