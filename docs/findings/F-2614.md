## Finding

On the forced respond turn of a typed query, QRY-14 step 2 states that the synthesised `__loom_respond_<slug>` tool's `execute` "AJV-validates the call payload … and resolves the query's promise with the validated value." A promise resolves exactly once, but the spec never bounds the turn to exactly one `tool_use` block. Two reachable multiplicity cases have no defined disposition:

- The provider emits **two or more `__loom_respond_<slug>` calls** in the same turn (parallel tool calls of the only offered tool). Each would invoke `execute` with a potentially different validated payload; the spec does not say which call resolves the promise, nor what happens to the extra calls.
- The provider emits **one `__loom_respond_<slug>` call alongside one or more sibling `tool_use` blocks**. The spec does not say whether the siblings are executed (firing side effects after the answer is already in hand) or discarded.

That multiplicity is reachable on this turn is established by the spec itself: ERR-17's wrong-tool branch explicitly handles a turn carrying multiple `tool_use` blocks and pins "when multiple wrong-tool calls appear in the same turn, the first block's `name` is used." ERR-17, however, scopes that rule to turns that "carried **only** `tool_use` blocks whose `name` is not the synthesised respond tool" — i.e. the all-wrong-tool arm. The arm where at least one respond-tool block is present (alone-but-duplicated, or accompanied by siblings) falls outside both ERR-17 branches and outside the QRY-14 success path, so it is unspecified. The analogous binder path (`binder-inference.md`), dispatched through the same off-session `complete()` channel, already resolves its equivalent question — "the **first** `ToolCall` whose `name` matches the binder's structured-output tool supplies the envelope" — leaving the typed-query forced respond turn the only `complete()`-driven extraction without a stated multiplicity rule.

(The original heading also alleged that a free-phase `tool_use` for a name not in the callable set has no stated disposition. That half does not hold: `ModelToolError` is defined to fire when "the named tool is absent from the resolved callable set," so the free-phase unknown-tool-name case surfaces as `ModelToolError` rather than a loopable `isError` result. Only the forced-respond-turn multiplicity gap is a real defect.)

The respond-tool-present multiplicity arm (the gap in this finding) was left open at inception and not closed by that near-miss. ## Solution Space

**Shape:** single

### Recommendation

State, at QRY-14 step 2 (the forced respond turn) in `query-tool-loop.md`, a first-match resolution rule mirroring the binder's `complete()`-channel extraction: the runtime resolves the typed query's promise from the **first `tool_use` block in provider emission order whose `name` equals `__loom_respond_<slug>`**; any additional respond-tool blocks and any sibling `tool_use` blocks on that same turn are discarded without execution (no second resolution; no side-effecting sibling dispatch). Validation of the selected call's payload follows the existing path — on AJV/depth failure the schema-validation respond-repair pipeline runs as normal; there is no fall-through to a later sibling block.

Tie this to ERR-17's existing boundary by noting that the wrong-tool branch ("carried **only** `tool_use` blocks whose `name` is not the synthesised respond tool") fires precisely when no respond-tool block is present; the presence of at least one respond-tool block takes the success path under the first-match rule. This keeps the respond-tool-present and all-wrong-tool arms mutually exclusive and exhaustive.

Edge cases the implementer must watch: (a) a turn carrying a respond-tool call plus plain text resolves from the respond-tool call (plain text is not the wrong-tool/plain-text non-compliance arm once a respond-tool block exists); (b) discarding sibling `tool_use` blocks means their `execute` is never invoked, so no tool-result is fed back and the turn terminates the typed query regardless of the siblings.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on which validated payload resolves a typed query whose forced respond turn carries more than one respond-tool call (first-wins vs last-wins vs error), and on whether a sibling `tool_use` block on that turn is executed (firing a side effect after the answer is produced) or discarded. The return value of a typed query — and the presence or absence of a post-answer side effect — is observable and would differ across conforming runtimes.
