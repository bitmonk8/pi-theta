## Problem

`extension-bootstrap-and-per-loom.md`'s `pi.registerMessageRenderer` failure bullet promises that, after a factory-time renderer-registration throw, system notes for that extension instance permanently degrade to the `ctx.ui.notify` arm of the **System notes** fallback chain. But the fallback chain in `runtime-event-channel.md` is entered only on a thrown value from `pi.sendMessage` (and, for group-A notes, from `Clock.wallNow()`). A successful `pi.sendMessage` against a `customType` with no registered renderer does not throw — Pi persists the `CustomMessage` and silently drops it at render time — so the chain is never entered and the "degrade to `ctx.ui.notify`" promise is undeliverable. No per-extension-instance renderer-availability state is named and no rule amends the chain's trigger, so two implementers diverge: one routes post-failure notes to `ctx.ui.notify`, the other lets them sink unrendered.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `11b9199` (2026-05-04, "pi-loom spec: resolve \"pi.sendMessage failure leaves system notes silently dropped\"") and `2d2fc8e` (2026-05-06, "pi-loom spec: resolve \"Pi SDK capability calls have no failure contract\"")
**History:** `11b9199` authored the System-notes fallback chain with a `pi.sendMessage`-throws trigger only; at that commit no rule yet claimed an unrendered-channel degradation mode, so the trigger was complete for the cases the chain was designed to cover. Two days later, `2d2fc8e` added the **Extension-bootstrap SDK failures** rule, including the `pi.registerMessageRenderer` failure bullet's promise that "System notes for this extension instance permanently degrade to the `ctx.ui.notify` arm of the **System notes** fallback chain below" — but did not amend the chain in `pi-integration-contract.md` to add a renderer-unavailable trigger or a per-instance bypass; the chain's trigger has remained synchronous-throw-only across every subsequent commit, including the corpus split in `f5e89f4` (2026-06-04) that moved the two sentences into separate files (`extension-bootstrap-and-per-loom.md` and `runtime-event-channel.md`), increasing the distance between them. Each commit was internally consistent against the corpus it landed in; the defect is the unstated obligation between them.

## Solution approach

Clarify the `pi.registerMessageRenderer` failure bullet in `extension-bootstrap-and-per-loom.md` so the "degrade to `ctx.ui.notify`" promise is operational: name the per-extension-instance renderer-availability state set at the failure site, and state that subsequent always-log emissions for that instance bypass the `pi.sendMessage` arm of the **System notes** fallback chain. Amend the best-effort fallback-chain preamble and trigger in `runtime-event-channel.md` to add the renderer-unavailable entry alongside the existing `pi.sendMessage`-throws and `Clock.wallNow()`-throws triggers.

## Solution constraints

- The `display: false`-skips-step-1 behaviour of the fallback chain MUST continue to hold in the renderer-unavailable mode.
- Reuse `loom/runtime/system-note-delivery-failed` for the renderer-unavailable path; do not mint a new diagnostic code.
