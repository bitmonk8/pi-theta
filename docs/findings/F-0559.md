## Finding

V14c's off-surface routing bullet (repeated verbatim in **Adds**, **Tests**, and **Ships when**, and in V14c-T's **Tests**) lists "a pre-eval setup throw → `{isError:true}` + one `loom/runtime/internal-error` diagnostic" with no scope qualifier. The spec gives this outcome a deliberately narrow scope: `tool-calls.md` §Outcome enumeration names it "*Pre-evaluation setup throw inside the `.loom`-callable adapter*" and qualifies it as occurring only "when the model emits a parallel tool call (per **Concurrency**) targeting a `.loom` callable and the adapter's pre-evaluation dispatch-setup sequence throws" — i.e. one of the enumerated dispatch-site setup steps (`new AbortController()`, `ActiveInvocationRegistry.add`, the inbound `signal.addEventListener('abort', …)` attach, `createAgentSession(...)` rejection in subagent mode, or the post-`createAgentSession` `loomAbort.signal` attach).

That scope is what distinguishes this outcome from a code-side Pi-tool `execute()` throw, which lowers to `Err(CodeToolError { cause: "execution" })`, not `{isError:true}`. By stating only the bare disposition, the leaf bullet reads as though *any* code-side tool pre-evaluation throw routes to `{isError:true}` + `internal-error`. A red-first author writing V14c-T could author tests for Pi-tool pre-eval throws — a disposition the spec does not enumerate — gating the implementation on behaviour the spec never requires.

The other three off-surface outcomes in the same bullet (non-conforming return shape, non-settling Promise, post-cancel late settlement) apply to code-side `execute()` generally and need no such qualifier; only the pre-eval setup throw carries the narrow `.loom`-callable-adapter / model-driven-parallel-batch scope.

## Solution Space

**Shape:** single

### Recommendation

In `V14c-tool-calls-off-surface-routing.md` (the **Adds**, **Tests**, and **Ships when** occurrences of the pre-eval setup throw clause) and `V14c-T-tool-calls-off-surface-routing.md` (the **Tests** occurrence), qualify the clause to name the `.loom`-callable parallel-batch adapter scope. Suggested literal form for the clause: "a pre-eval setup throw in the `.loom`-callable parallel-batch adapter (when the model emits a parallel tool call targeting a `.loom` callable) → `{isError:true}` + one `loom/runtime/internal-error` diagnostic". Anchor the scope to `tool-calls.md` §Outcome enumeration, "Pre-evaluation setup throw inside the `.loom`-callable adapter".

Edge cases for the implementer: the spec's scoped form carries two further obligations the bare bullet also elides — the runtime emits one `loom-system-note` alongside the `loom/runtime/internal-error` diagnostic, and the `{isError:true}` message uses the bare callable-set name (`loom <name> aborted with internal error: …`), not the slash-prefixed `/<name>` form. Leave the other three off-surface outcomes in the bullet unqualified; only the pre-eval setup throw needs the scope.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one scopes the assertion to the `.loom`-callable parallel-batch adapter (matching the spec), the other writes a test asserting `{isError:true}` + `internal-error` routing for a code-side Pi-tool pre-eval throw — a disposition the spec does not mandate (such a throw lowers to `Err(CodeToolError { cause: "execution" })`). The second path gates V14c on out-of-spec behaviour and can red-first against a spec-faithful implementation.
