## Finding

The *Baseline acquisition* paragraph in `version-bump-step2.md` establishes the canonical "from" side for every `dist/*.js` diff on the page by mandating, as a `MUST`, that the contributor run `npm pack @earendil-works/pi-coding-agent@<pinned-version>` and extract the tarball. Four editorial-review audits read against that extracted `package/dist/` tree ā€” item (e) (`AgentSession.prototype.prompt` cooperative-await), item (r) (`convertToLlm` context-entry), item (aj) (`SessionManager.inMemory` transcript non-retention), and item (am) (`createAgentSession` `tools`-allowlist suppression). (The preamble names only (e), (r), and (aj); item (am) also reads against the same baseline tree, so the dependency surface is four items, not three.)

Two problems follow. First, the step assumes the pinned version is still retrievable from the registry at bump time. A yanked version, a registry outage, or an air-gapped CI environment makes `npm pack` fail, and the spec states no precondition and no fallback baseline source ā€” so all four diff/JS-body audits silently lose their "from" side. The obvious local fallback (the pinned tree already present in `node_modules/@earendil-works/pi-coding-agent/dist/` before step 1 overwrites it) is acknowledged only as a reason to order acquisition early, never offered as an alternate baseline source. Second, the requirement is over-prescribed: it pins the exact `npm pack` command and explicitly forecloses `npm install --prefix`, where the only observable obligation is "materialise the loom-1.0-pinned `dist/` tree without pulling its transitive dependencies." Any mechanism that produces that tree satisfies the audit; the spec should pin the outcome and treat `npm pack` as a worked example.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the *Baseline acquisition* paragraph so it pins the observable outcome and its preconditions rather than a single command:

- State the precondition explicitly: the loom-1.0-pinned `dist/*.js` tree must be obtainable. Name a fallback baseline source for when registry retrieval is unavailable (yanked / outage / air-gapped) ā€” the natural one is the pinned tree already installed under `node_modules/@earendil-works/pi-coding-agent/dist/`, captured into the scratch directory **before** step 1 overwrites it. The "run acquisition before step 1" ordering already cited becomes the primary fallback path rather than merely an ordering note.
- Restate the requirement behaviourally: the contributor MUST materialise the loom-1.0-pinned `dist/` tree (without pulling its transitive dependency tree) into a scratch directory; the extracted tree is the "from" side. Demote `npm pack @earendil-works/pi-coding-agent@<pinned-version>` (and the `npm pack`-not-`npm install --prefix` rationale) to a non-normative worked example of one conforming mechanism.
- Correct the consumer enumeration in the preamble to list all four dependent items ā€” (e), (r), (aj), **and (am)** ā€” since item (am) reads against the same extracted tree but is currently omitted.

Keep the ephemerality and MUST-NOT-commit rules unchanged. Edge case the implementer must watch: the fallback "copy from `node_modules`" path is valid only when run before step 1's install ā€” after step 1 the local tree holds the candidate minor, so the registry-retrieval path (with the explicit `@<pinned-version>` specifier) remains the only correct source post-install.

## Consequence

**Severity:** advisory

In a yanked-version, registry-outage, or air-gapped-CI environment the mandated `npm pack` fails and the four diff-based audits (e/r/aj/am) cannot obtain their baseline; the conservative-posture rule forces item (e) to `fail` while (r)/(aj)/(am) get recorded `not-performed`, degrading audit coverage with no specified remedy. Two contributors hitting the failure would improvise different baseline sources, and the foreclosure of `npm install --prefix` blocks an equally valid materialisation mechanism.
