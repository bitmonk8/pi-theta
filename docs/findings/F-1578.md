## Finding

The **Tool execution from loom code** section of `host-interfaces-core.md` routes every code-side tool-call outcome on a returned `{ content, isError }` envelope: a clean `{ content, isError: false }` lowers to `Ok(<filtered/joined text>)`, a clean `{ content, isError: true }` lowers to `Err(CodeToolError { cause: "execution", ... })`, and an `execute()` throw lowers to a separate `Err(CodeToolError { cause: "execution", ... })`. The prose further claims the `isError: boolean` field "is declared on the tool-result type family in `dist/core/extensions/types.d.ts`" and that the SDK surface-inventory test asserts the field set on that type.

At the loom 1.0 Pi-SDK pin (`@earendil-works/pi-coding-agent@0.75.5`), `execute()` returns `Promise<AgentToolResult<TDetails>>`, and `AgentToolResult<T>` (pi-agent-core `dist/types.d.ts:305`) declares exactly `content: (TextContent | ImageContent)[]`, `details: T`, and `terminate?: boolean` â€” there is **no `isError` field** on the `execute()` return. The `isError: boolean` field does exist at the pin, but only on types that are not the `execute()` return value: the persisted `ToolResultMessage` (pi-ai `dist/types.d.ts:203`, produced by the dispatcher), the `afterToolCall` hook context, `AfterToolCallResult`, and the `tool_execution_end` event. The `AgentTool.execute` JSDoc is explicit about the actual error convention at the pin: *"Throw on failure instead of encoding errors in `content`."*

Because loom invokes `execute()` directly and out-of-band (bypassing Pi's dispatcher â€” see the sibling implementation-notes finding), no `ToolResultMessage` is ever produced for a code-side call, so the dispatcher-derived `isError` flag is never observable to the loom runtime either. The consequence is that the spec's `clean { content, isError: true } â†’ Err` branch is unreachable against the pin: a returned `AgentToolResult` has no error flag to read, so every cleanly-resolving call lowers to `Ok`, and a tool that signals an error by the SDK's documented mechanism (throwing) is the only error path that actually exists. The promised surface-inventory assertion ("asserts the field set on the cited type") cannot be written as described, since the cited return type carries no `isError`.

## Solution Space

**Shape:** single

### Recommendation

Re-anchor the code-side outcome routing on the SDK's actual `execute()` contract at the pin: `execute()` returns `AgentToolResult { content, details, terminate? }` and signals failure by **throwing**, not by an `isError` flag.

- Treat a cleanly-resolving `AgentToolResult` as the success path: filter `content` to `type === "text"` entries, join their `.text` with `"\n"`, and return `Ok(<joined text>)` (still `Ok("")` when no text survives â€” the existing non-text-discard and empty-result rules are unaffected; `content`'s `(TextContent | ImageContent)[]` shape matches the pin).
- Treat an `execute()` throw as the sole `cause: "execution"` error path: `Err(CodeToolError { cause: "execution", message: <thrown value coerced to the underlying-error string and truncated to 4096 bytes>, tool_name, ... })`.
- Remove the `clean { content, isError: true } â†’ Err(CodeToolError { cause: "execution" })` branch and the now-dead `"tool reported an error with no text content"` fixed message; with no `isError` on the return, that state cannot arise. Collapse the Outcome-routing-summary three-way split (`isError:false` / `isError:true` / throw) into a two-way split (clean resolve â†’ `Ok`; throw â†’ `Err`). The `cancelled` and non-conforming-shape dispositions are unchanged.
- Fix the declaration citation in the same edit: `AgentToolResult` is declared in pi-agent-core `dist/types.d.ts` (re-exported via `dist/core/extensions/types.d.ts`), and `isError` is **not** part of that return type â€” so reframe the SDK surface-inventory assertion to assert the `AgentToolResult` field set (`content` block shape, array discriminator) rather than an `isError` field.

Edge case the implementer must watch: a tool that resolves cleanly while semantically reporting an error in its `content` text now lowers to `Ok` at the pin â€” this is the SDK's documented behaviour ("throw on failure"), and loom does not second-guess it. The `CodeToolError.cause` enum (`validation` / `execution` / `cancelled` / `unknown_tool`) stays closed; no new cause is needed.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one follows the spec literally â€” finds no `isError` on the `execute()` return, and lowers every cleanly-resolving call to `Ok`, silently treating tool-reported errors as success; another reads the SDK and routes the throw as the error path. The literal reading produces wrong behaviour (silent success on tool failure), and the promised SDK surface-inventory assertion cannot be authored against the cited type, so the bump-time drift guard the section claims does not exist.
