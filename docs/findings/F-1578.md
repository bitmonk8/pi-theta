## Problem

The **Tool execution from loom code** section of `host-interfaces-core.md` (anchor `tool-execution-from-loom-code`) routes every code-side `execute()` outcome on a returned `{ content, isError }` envelope and claims `isError: boolean` is declared on the tool-result return type. At the loom 1.0 Pi-SDK pin, `execute()` returns `AgentToolResult { content, details, terminate? }`, which carries no `isError` field, and the SDK's documented failure convention is to throw, not to set a flag. The `isError` field exists only on dispatcher-side types (`ToolResultMessage`, hook contexts, the `tool_execution_end` event) that loom never observes for an out-of-band code-side call. The *Outcome routing summary*'s `clean { content, isError: true } â†’ Err` branch is therefore unreachable: cleanly-resolving calls all lower to `Ok`, tool-reported errors are silently treated as success, and the promised surface-inventory assertion cannot be written against the cited return type.

## Solution approach

Rewrite the envelope-shape paragraph and the *Outcome routing summary* under anchor `tool-execution-from-loom-code` to re-anchor outcome routing on the SDK's actual `execute()` contract at the pin: `execute()` returns `AgentToolResult { content, details, terminate? }` and signals failure by throwing, with no `isError` flag. Collapse the three-way routing into a two-way split â€” a cleanly-resolving `AgentToolResult` lowers to `Ok(<filtered/joined text>)`, and an `execute()` throw lowers to `Err(CodeToolError { cause: "execution", ... })` â€” removing the dead `clean { content, isError: true } â†’ Err` branch. Reframe the SDK surface-inventory assertion to assert the `AgentToolResult` field set rather than an `isError` field, and fix the declaration citation to the type that actually carries the return shape.
