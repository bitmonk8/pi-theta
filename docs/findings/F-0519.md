## Finding

The `V8a` leaf is titled `` `Checkpoint` seam `` and its body covers only the `Checkpoint` seam — its `Adds` explicitly delegates the `SchemaValidator` seam to `V8c` ("The `SchemaValidator` seam is owned by [`V8c`](./V8c-schema-validator-seam.md)"). Yet the file is named `V8a-checkpoint-validator-seams.md` (with its tests partner `V8a-T-checkpoint-validator-seams.md`), embedding `validator` in the slug. The `SchemaValidator` seam now has its own dedicated leaf file, `V8c-schema-validator-seam.md`.

A developer searching the `plan_topics/` file tree for the `SchemaValidator` seam — e.g. grepping or eyeballing filenames for `validator` — is drawn to `V8a`'s file, which does not implement that seam. The slug is a residue of an earlier design where `V8a` bundled both seams; it was not updated when the `SchemaValidator` seam was split out.

The plan-index link text and every cross-leaf reference already use the correct "`Checkpoint` seam" naming, so the defect is confined to the on-disk filename and the markdown link *targets* that point at it.

**Verdict:** single-commit-introduction
**Introducing commits:** cb8737a ("pi-loom plan: resolve \"Systemic leaf over-bundling across the plan corpus\"")
**History:** `git log --follow -- docs/plan_topics/V8a-checkpoint-validator-seams.md` shows the file was first added in c6a664e ("build/update plan for spec.md + review") with no subsequent rename; `git log --diff-filter=A -- docs/plan_topics/V8c-schema-validator-seam.md` shows `V8c` first appeared in cb8737a. `git show c6a664e:docs/plan_topics/V8a-checkpoint-validator-seams.md` confirms that at inception `V8a` was titled "`Checkpoint` and `SchemaValidator` seams" and covered both seams (PIC-10 + PIC-11), so the slug `checkpoint-validator-seams` was accurate then. `git show cb8737a:docs/plan_topics/V8a-checkpoint-validator-seams.md` shows that the over-bundling-split commit narrowed `V8a`'s title and body to the `Checkpoint` seam alone and delegated `SchemaValidator` to the newly-created `V8c`, but left the `V8a` (and `V8a-T`) file slug — and the link targets pointing at it — unchanged. The stale slug therefore entered the corpus with that single split commit (2026-06-12).

## Solution Space

**Shape:** single

### Recommendation

Rename the two files to drop `validator` from the slug, leaving the `V8a` leaf ID unchanged (the ID is `V8a`; only the `<short-name>` portion of the `plan_topics/<id>-<short-name>.md` filename changes):

- `docs/plan_topics/V8a-checkpoint-validator-seams.md` → `docs/plan_topics/V8a-checkpoint-seam.md`
- `docs/plan_topics/V8a-T-checkpoint-validator-seams.md` → `docs/plan_topics/V8a-T-checkpoint-seam.md`

Then retarget the three markdown links that reference the old path so they resolve after the rename:

- `docs/plan.md` line 112 — `[`V8a` — `Checkpoint` seam](./plan_topics/V8a-checkpoint-seam.md)`
- `docs/plan_topics/V15h-invoke-ceiling-swallowing.md` — the `Adds` link `[`V8a`](./V8a-checkpoint-seam.md)`
- `docs/plan_topics/V17c-checkpoint-granularity.md` — the `Adds` link and the `Ships when` link, both `[`V8a`](./V8a-checkpoint-seam.md)`

Bare `Deps`/prose references to the `V8a` leaf ID (e.g. in V13f, V14f, V9c, H7a) do not change — they cite the ID, not the file path. Leave `V8c-schema-validator-seam.md` untouched; it already owns the `SchemaValidator` seam and carries the correct slug.

## Consequence

**Severity:** cosmetic

A developer locating seams by filename is misdirected to `V8a` when looking for the `SchemaValidator` seam, which `V8a` delegates to `V8c`. No leaf body, test, dependency edge, or coverage cell is wrong; the harm is navigation-only and does not cause two implementers to diverge.
