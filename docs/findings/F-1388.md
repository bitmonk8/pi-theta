## Problem

PIC-21's normative carve-out in `extension-bootstrap-and-per-loom.md` ("no `loom/runtime/*` diagnostic, operator signal, or telemetry is emitted for a caught render-time failure (Pi exposes no observation callback for it)") asserts a negative claim about the Pi SDK as fact without grounding. Every other Pi-side surface in the same Renderer-registration section is pinned to a `dist/...` location at the loom 1.0 Pi-SDK pin and threaded through `version-bump-step2.md` checklist item (q) so a future Pi minor trips a re-audit; PIC-21's absence claim does neither. The claim is therefore unfalsifiable from the spec, and item (q) — which today covers only the loader-`Map` overwrite and `getMessageRenderer` first-hit behaviours — does not gate render-time throw observability, so a Pi minor adding a render-error callback, event, or hook passes the audit while PIC-21's "no telemetry" contract silently understates what the runtime could observe. The same defect-shape recurs on the immediately-following Teardown bullet.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `841a085` (2026-06-06, pi-loom spec: resolve "renderer-throw during render has no defined failure mode").
**History:** PIC-21 entered the corpus in `841a085` as a single self-contained paragraph; the negative claim "Pi exposes no callback by which the loom runtime can observe a render-time renderer throw" and its closing parenthetical restatement ("Pi exposes no observation callback for it") were both present in the introducing diff. `git log -S` against both phrasings on `extension-bootstrap-and-per-loom.md` returns only `841a085` for content (the later spec-review consolidation commit `fc51232` is the report itself, not a content edit). No subsequent commit has tightened the claim, cited a `dist/...` location, or added it to `version-bump-step2.md`'s checklist; the related re-audit checklist item (q) (`b077c07`, 2026-06-05) was authored a day earlier on adjacent surfaces and never extended to cover render-throw observability when PIC-21 landed.

## Solution approach

Rewrite PIC-21's no-telemetry carve-out so the absence claim reads as a pinned presupposition re-validated via checklist item (q), mirroring how the loader-`Map` overwrite and `getMessageRenderer` first-hit behaviours on the same page are framed, with a forward-link to `#bump-checklist-pre-bind-and-renderer-resolution`. Extend item (q) so its audit covers render-time throw observability — that the candidate Pi minor still exposes no surface by which the loom runtime can observe a `MessageRenderer` invocation throwing.

## Solution constraints

- Out of scope: PIC-21's renderer normative obligations (MUST NOT throw out of `MessageRenderer`; catch-and-return a minimal `Component`/`undefined` on internal failure) — only the no-telemetry carve-out framing changes.
- Co-edit `extension-bootstrap-and-per-loom.md` and `version-bump-step2.md` in the same commit per the GOV-3 co-edit obligation in force on item (q).
