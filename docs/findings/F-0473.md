## Finding

`H2a`'s `src/**` no-globals architectural test flags a module-level `const` "whose initializer is a mutable object or array literal" (same phrase in the **Adds** paragraph, the *No globals, statics, singletons* **Tests** bullet, and **Ships when**). The phrase has two incompatible readings for the AST scan to implement:

1. **Literal syntax only** — the scan flags object-literal (`{…}`) and array-literal (`[…]`) initializer forms, and nothing else.
2. **Any mutable-producing initializer** — the scan additionally flags expressions such as `new Map()`, `new Set()`, `new Array()`, `Object.create(null)`, which produce shared mutable state at module scope just as a literal does.

The leaf's own rationale narrows the escape hatch to three categories — "closure-captured, lazy module-cache, and DI-container singletons are not mechanically detected; that residue is owned by the *Per-phase TDD ritual* self-review step." A bare module-level `const cache = new Map()` is **none** of those three: it is a directly-observable initializer that is not a literal. It therefore falls in a gap — not provably flagged by the mechanical scan under reading 1, and not provably claimed by the residue sweep either (`conventions.md` release-time residue-inspection item 1 names only the closure / lazy-cache / DI-container forms). Two implementers building the H2a scan would diverge on whether such a binding reddens CI.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** d56d65f — pi-loom plan: resolve "H2a architectural-test predicate hinges on reused across calls" (2026-06-11, Thomas Andersen)
**History:** d56d65f rewrote the predicate from "a top-level `const` bound to a mutable object/array reused across calls" to "a top-level `const` whose initializer is a mutable object or array literal" to remove the prior runtime-reuse ambiguity. That rewrite introduced the present literal-vs-mutable-producing-expression ambiguity: the new wording is grounded in syntactic literal forms but never states whether non-literal mutable initializers (`new Map()`, `Object.create(null)`) are in or out of the scan's flag set.

## Solution Space

**Shape:** multiple

### Option A — Literal-only scan; residue owns non-literal mutable initializers

**Approach.** The H2a AST scan flags only object-literal (`{…}`) and array-literal (`[…]`) initializer syntax. Module-level `const` bindings to other mutable-producing expressions (`new Map()`, `new Set()`, `new Array()`, `Object.create(null)`, and similar) are not mechanically flagged and are swept by `conventions.md` release-time residue-inspection item 1.

**Plan edits.** In `H2a-cross-cutting-gates.md`, make the Adds paragraph and the *No globals* Tests bullet state that the scan flags object/array **literal initializer syntax** only (the existing "decides the ban on the observable initializer shape" clause already points this way). In `conventions.md` residue-inspection item 1, extend the Clean-when to explicitly name module-level `const` bindings to non-literal mutable initializers (e.g. `const m = new Map()`) so the residue sweep demonstrably owns them.

**Spec edits.** None.

**Pros.** Matches the existing division of labour (cheap syntactic mechanical gate + manual residue sweep); minimal AST scan complexity; no open-ended enumeration to keep current.

**Cons.** A module-level `const cache = new Map()` passes the mechanical gate and is caught only at release-time manual review.

**Risks.** Item 1 currently names only closure-captured / lazy-cache / DI-container singletons; unless it is extended, a bare `new Map()` module const falls in the gap between the literal-only scan and the three named residue categories.

### Option B — Enumerate the additional mutable-producing initializer kinds the scan flags

**Approach.** Extend the AST scan to flag, in addition to literals, a fixed enumerated set of mutable-producing initializer expressions (e.g. `new Map`/`new Set`/`new Array`/`new WeakMap`/`new WeakSet`, `Object.create(...)`).

**Plan edits.** In `H2a-cross-cutting-gates.md`, name the additional flagged expression kinds in the Adds paragraph and the *No globals* Tests bullet, alongside the literal forms.

**Spec edits.** None.

**Pros.** Stronger automated coverage of the common module-level-singleton forms; fewer cases deferred to manual review.

**Cons.** Open-ended — a custom class constructor producing mutable state still escapes, so the residue sweep is still required; larger AST scan surface to author and maintain.

**Risks.** An incomplete enumeration invites a false sense of completeness; newly-introduced mutable-producing forms are not auto-flagged and silently escape.

### Recommendation

Take Option A. Restrict the mechanical scan to object/array literal initializer syntax and extend `conventions.md` residue-inspection item 1's Clean-when so module-level non-literal mutable initializers (`new Map()`, `Object.create(null)`, etc.) are demonstrably within the residue sweep's scope. This makes the scan's flag set unambiguous without an open-ended AST enumeration and aligns with the existing mechanical-gate-plus-residue division. Edge case the implementer must watch: a bare `const m = new Map()` at module scope is none of item 1's three currently-named singleton categories, so item 1 must be widened or the form falls in the gap between the literal-only scan and the residue sweep.

## Consequence

**Severity:** correctness

Two reasonable implementers build different no-globals scans: one flags only `{…}`/`[…]` literal initializers, the other also flags `new Map()` / `Object.create(null)`. A module-level `const cache = new Map()` then either reddens CI or ships silently depending on which scan was built, and the H2a **Ships when** ("a module-level mutable binding of a flagged form") cannot be evaluated consistently. The gap is only partly backstopped by manual residue-inspection item 1, and only if that item is read to own non-literal mutable initializers — which it does not currently name.
