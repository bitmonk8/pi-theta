## Finding

The `V9k` leaf file (`V9k-extension-bootstrap-failures.md`) names itself two different ways. The H1 heading reads `Extension-bootstrap SDK-failure abort surfaces` (short form), while the `Adds.`/`Tests.`/`Ships when.` body consistently uses the qualified phrase **whole-extension abort** surfaces — and the qualifier is load-bearing. It is exactly the "whole-extension" scope that distinguishes the two surfaces V9k owns (`registerFlag` and `pi.on(...)` failures, each of which aborts the entire extension) from the three **non-abort** surfaces (renderer degrade, per-loom `registerCommand` drop, `getCommands()` read-failure drop) that the sibling leaf `V9p` owns. A reader who maps "abort surfaces" (heading) onto the `V9k` ↔ `V9p` abort/non-abort partition has to re-derive that the heading and body denote the same thing.

The `plan.md` index entry (line 134) repeats the heading's short form, so the divergence appears in both the leaf file and the plan index. The mismatch is purely terminological — the body text fully and correctly defines the leaf's scope — but it muddies the abort vs non-abort split that V9k/V9p exist to express.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `5a94e2e` (2026-06-17) — "pi-loom plan: resolve \"V9k bundles five independent bootstrap-failure surfaces in one leaf\""
**History:** `V9k` was created in `6df646a` (2026-06-12) as a single leaf headed `Extension-bootstrap SDK-failure granularity`, before the abort/non-abort split existed. Commit `5a94e2e` split the leaf — narrowing `V9k` to the abort surfaces and spinning the non-abort surfaces out into `V9p` — and in that same change renamed the `V9k` heading to the short `…abort surfaces` while authoring the body with the qualified "whole-extension abort" phrasing. The same commit set the `plan.md:134` index link text to the short form. The heading-vs-body mismatch and the divergent index link both entered the corpus in this one commit.

## Solution Space

**Shape:** single

### Recommendation

Align the leaf's title with the qualifier its body already uses, in both places that carry the title:

- In `docs/plan_topics/V9k-extension-bootstrap-failures.md`, change the H1 heading to ``# `V9k` — Extension-bootstrap SDK-failure whole-extension abort surfaces``.
- In `docs/plan.md` line 134, change the index link text to match: ``- [`V9k` — Extension-bootstrap SDK-failure whole-extension abort surfaces](./plan_topics/V9k-extension-bootstrap-failures.md)``.

Both edits insert the single word `whole-extension` before `abort surfaces`; the leaf filename and leaf-ID are unchanged. Edge case for the implementer: `V9p`'s counterpart heading is `…non-abort surfaces` (unqualified), so after this change the pair reads "whole-extension abort" / "non-abort"; that asymmetry is acceptable because "whole-extension" is the meaningful scope marker in V9k's body, and no `V9p` edit is required.

## Consequence

**Severity:** cosmetic

If shipped unfixed, the V9k leaf's title and body name its scope differently and the plan index repeats the less-precise title; the implementer can still produce the correct leaf because the body is unambiguous, so the only cost is a reader briefly reconciling the heading against the abort/non-abort partition V9k and V9p encode.
