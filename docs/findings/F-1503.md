## Problem

The §5 `<key>` bullet in `placeholder-rendering-b.md` joins two non-equivalent predicates with "i.e.": it cross-references [Lexical — Identifiers](../lexical.md) and then glosses that as the character class `[A-Za-z_][A-Za-z0-9_]*`. Lexical's identifier definition layers a position-keyed first-letter case constraint on top of that character class (PascalCase-required vs lowercase-first-required positions, with case-mismatch parse errors), so "identifier-shaped per Lexical" admits both a case-blind and a case-sensitive reading. The operative predicate is genuinely the case-blind runtime regex `^[A-Za-z_][A-Za-z0-9_]*$`, as the Category 5 identifier-predicate bullet in the Edge cases section pins. A reader who stops at §5 may apply Lexical's case rule to runtime keys and quote uppercase-start keys (`"Kind"`, `"FOO"`), contradicting the same page's Edge cases treatment.

## Issue introduction

**Verdict:** present-since-inception (of the rule)
**Introducing commit:** `61ea1a5` (2026-05-06, *pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"*)
**History:** The `<key>` bullet was authored verbatim in `61ea1a5` as part of the initial introduction of §5 ("Source-derived placeholders") into `spec_topics/diagnostics.md`, with the misleading "(i.e. would not match `[A-Za-z_][A-Za-z0-9_]*`)" gloss in place from the first version. The same commit added the disambiguating Category 5 Edge Cases bullet, so the internal tension has been present since the rule existed. The 100KB split (`f5e89f4`, 2026-06-04, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*) relocated the §5 block from `diagnostics.md` into `diagnostics/placeholder-rendering-b.md` byte-for-byte without amending it. Subsequent commits to `placeholder-rendering-b.md` (`72e2c31`, `bf85092`, `d3ef0cf`, `89eabf1`, `62d49d6`, …) extended §7 and §6 but never revisited the §5 `<key>` wording.

## Solution approach

Rewrite the §5 `<key>` bullet so the operative predicate is the runtime regex `^[A-Za-z_][A-Za-z0-9_]*$` named inline, and delete the load-bearing "identifier-shaped per Lexical" framing that makes the cross-reference appear to carry the case constraint. State that the predicate is case-blind. Add a forward-link to the Category 5 identifier-predicate bullet in the [Edge cases](#edge-cases) section for the full edge-case treatment.

## Solution constraints

- Out of scope: the §7 `<value>` rule (and its YAML-scalar carve-out) is owned by a separate finding; do not edit §7.
- The rewrite must not contradict the Category 5 identifier-predicate bullet in the Edge cases section (reserved-keyword-valued keys render bare).
