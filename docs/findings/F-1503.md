## Finding

The `<key>` rule in §5 of [Placeholder rendering b](../../docs/spec_topics/diagnostics/placeholder-rendering-b.md) reads:

> **`<key>`** — quoted with double quotes only when the key string is *not* identifier-shaped per [Lexical — Identifiers](../lexical.md) (i.e. would not match `[A-Za-z_][A-Za-z0-9_]*`); otherwise rendered bare.

The parenthetical "i.e." treats two predicates as equivalent that are not equivalent. The Lexical page defines the identifier *character class* as `[A-Za-z_][A-Za-z0-9_]*` but then layers a normative first-letter case constraint on top: a PascalCase first letter is **required** for `schema` / `enum` / `enum` variant names, and a lowercase-first letter (or `_`) is **required** for `let` bindings, parameters, function names, and schema field names. Violations are parse errors (`loom/parse/schema-case-mismatch`, `loom/parse/binding-case-mismatch`). "Identifier-shaped per Lexical" therefore admits the case-blind character-class reading *or* the position-keyed case reading depending on which sentence the reader anchors on; the gloss equates them by fiat.

This matters because the same file resolves the ambiguity in the opposite direction one screen later, in its Edge Cases section:

> **Category 5 identifier predicate.** The identifier-shape check for `<key>` is a *runtime* string predicate against the regular expression `^[A-Za-z_][A-Za-z0-9_]*$`; a key like `kind` renders bare, `my-key` renders quoted. Keys whose string value happens to match a reserved keyword … are still identifier-shaped for this rule (the reserved-keyword rule fires at parse time on identifier *positions* in source, not at runtime on string values).

So the operative predicate is genuinely the bare regex (no case constraint, reserved keywords included), and the §5 cross-reference to Lexical is at best decorative — but a reader who stops at §5 sees the cross-reference, treats it as load-bearing, and may then apply Lexical's case rule to runtime keys (rendering `"Kind"` quoted, since an uppercase-start string would fail Lexical's lowercase-first rule for binding positions). The §7 `<value>` rule compounds the exposure by delegating to "category 5's `<key>`" without restating the predicate.

## Solution Space

**Shape:** single

### Recommendation

Drop the misleading Lexical cross-reference from the §5 `<key>` bullet and pin the runtime regex inline as the sole operative predicate. Concretely, replace the existing bullet:

> **`<key>`** — quoted with double quotes only when the key string is *not* identifier-shaped per [Lexical — Identifiers](../lexical.md) (i.e. would not match `[A-Za-z_][A-Za-z0-9_]*`); otherwise rendered bare. The identifier-shape predicate is a runtime check on the key string, not a parse-time grammar production.

with wording that names the regex directly and notes the case-blindness explicitly, e.g.:

> **`<key>`** — rendered bare when the key string matches `^[A-Za-z_][A-Za-z0-9_]*$`; otherwise double-quoted. The predicate is a runtime string check against the regex above (no case constraint and no reserved-keyword carve-out — Lexical's first-letter case rule and reserved-keyword rule fire at parse time on identifier *positions* in source, not at runtime on string values, so `"Kind"` and `"let"` both render bare here). See [Category 5 identifier predicate](#edge-cases) for the full edge-case treatment.

Edge cases the implementer should preserve:

- The runtime predicate is case-blind: `"kind"`, `"Kind"`, and `"FOO"` all render bare; only characters outside `[A-Za-z0-9_]` (or a leading digit) force quoting.
- Keys whose string value coincides with a reserved keyword (`"let"`, `"if"`, …) still render bare — already pinned by the Category 5 Edge Cases bullet; the new wording should not contradict it.
- The §7 `<value>` delegation to "category 5's `<key>`" inherits the clarified predicate unchanged; no edit to §7 is required for this finding (the YAML-scalar carve-out is a separate concern — see the related `§7 <value>` finding below).
- The two existing §5 test vectors (`obj["kind"]` → bare, `obj["my-key"]` → quoted) and the §7 vector with `"25"` continue to hold under the clarified rule; no test-vector updates are needed.

## Consequence

**Severity:** advisory

A literal reading of the §5 cross-reference alone could lead an implementer to apply Lexical's case constraint and quote uppercase-start keys (`"Kind"`, `"FOO"`), even though the Edge Cases bullet on the same page pins the case-blind regex. The two readings disagree only on byte-identical rendering of uppercase-start keys, and the spec is internally consistent for a reader who follows the whole page; the defect is a clarity gap, not a hidden behaviour.
