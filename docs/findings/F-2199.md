## Problem

QRY-18's *Stringification of interpolated values* Notes bullet specifies that a `${expr}` interpolation whose operand type is statically unresolvable falls back to a runtime panic carrying the `loom/parse/interpolated-result` diagnostic code, and the `code-registry-parse.md` row records the same design. This collides with the closed runtime-panic surface at `error-model.md` `#runtime-panics`, which enumerates exactly six panic sources, each carrying a registered `loom/runtime/*` code with a row in the normative *Panic message string* template table. A `loom/parse/*` panic code is admitted by none of these — it is absent from the closed list and has no message template — and QRY-21's panic enumeration omits it. An implementer cannot satisfy both contracts, so reasonable implementers diverge on the emitted code, the panic message string, and the operator-facing surface.

## Issue introduction
**Verdict:** single-commit

**Introducing commit:** `0c9ce54` — "pi-loom spec: resolve \"Stringification rule for ${expr} interpolations is unspecified\"" (2026-05-04)

**History:** The spec corpus is git-tracked. The QRY-18 stringification rule — including both the "runtime renderer falls back to a panic carrying the same `loom/parse/interpolated-result` diagnostic code" wording and the registry's "raises the same code as a panic" note — was authored together in `0c9ce54` (`git log -S "falls back to a panic carrying the same"` and `git log -S "raises the same code as a panic"` each return `0c9ce54` as the earliest commit; later hits are the file-split commit `f5e89f4`). At that commit the *Runtime panics* section already read "V1 panic sources — each carrying its registered `loom/runtime/*` code" and "This list is closed" (verified via `git show 0c9ce54:spec_topics/errors-and-results.md`), and the panic-message-template table already enumerated only `loom/runtime/*` codes. The defect therefore entered fully formed in `0c9ce54`: a runtime-panic fallback carrying a `loom/parse/*` code was introduced into a corpus whose panic surface was already closed and already required per-source `loom/runtime/*` codes. The subsequent `f8f4813` (which added the `loom/runtime/internal-error` runtime-defect surface) and `f5e89f4` (the 100KB size-cap file split) neither caused nor reconciled the conflict.

## Solution approach

Repoint QRY-18's runtime-renderer Notes bullet and the `code-registry-parse.md` `loom/parse/interpolated-result` row so the statically-unresolvable runtime case names a dedicated `loom/runtime/*` panic code rather than the parse code, treating it as a genuine author-caused panic rather than routing it through `loom/runtime/internal-error`. Register that code in `code-registry-runtime.md` and add it as a source row to the closed *Runtime panics* list and to the *Panic message string* template table in `error-model.md`, updating the panic-source count assertions across `error-model.md` and `code-registry-runtime.md` in lock-step. Add the new source to QRY-21's panic enumeration.

- Out of scope: the duplicate panic-message-template-table dedup decision owned by F-2147 — add the new code's row consistently with the sibling rows, but do not resolve the table duplication here.
