## Finding

The `H3a` ambient-access architectural test scans `src/**` for direct references to a hard-coded set of timing primitives — `Date.now`, `performance.now`, `Date.prototype.getTime`, `setTimeout`, `clearTimeout`. `conventions.md` (*No globals, statics, singletons*) and `H3a` both state this enumeration is a **manually-maintained derivation** of the `host-interfaces-services.md` `Clock`/`WallClock` ban (PIC-12), required to be a superset of the spec ban (`scan list ⊇ spec ban`) and currently equal to it.

Nothing mechanical enforces that superset invariant. The only reconciliation is release-time residue-inspection item 9 in `conventions.md`, which is a reviewer manually re-reading the two lists. PIC-12 itself only requires that *a* build-time check rejecting direct lexical references exist — it explicitly leaves the primitive list and the check's realization to the implementation and does not expose a machine-readable enumeration. So if the spec's PIC-12 ban is widened mid-build to enumerate an additional timing primitive (or PIC-20 / an adjacent rule adds one), the `H3a` scan list silently falls out of the `scan list ⊇ spec ban` superset, the newly-banned primitive is not scanned, and the divergence is invisible until a human runs item 9 at the loom 1.0 release gate.

## Consequence

**Severity:** advisory

If PIC-12's banned-primitive set widens after `H3a` lands, the `H3a` scan keeps passing while production code can use the newly-banned primitive undetected; the divergence surfaces only when a reviewer manually runs release-time residue item 9. No defect escapes the release gate (item 9 blocks release), but the no-globals discipline can be silently violated for the entire mid-build window, where remediation is most expensive.

Commit `1c9ede1` resolved a subsequent "lockstep over-promise" finding by downgrading that claim to the present framing — a "manually-maintained derivation … with no mechanical reconciliation," re-reconciled only by release-time residue inspection item 9 (added in the same commit). The `scan list ⊇ spec ban` / release-time-only condition this finding targets therefore entered the corpus deliberately at `1c9ede1`, as the honest replacement for the earlier false lockstep promise. The present finding asks the plan to go one step further and mechanise that reconciliation.

## Solution Space

**Shape:** multiple

### Option A — Mechanical superset assertion in `H3a`

- **Approach.** Add a `src/**`-scoped test/CI assertion that the `H3a` scan's timing-primitive list covers every primitive the current PIC-12 ban enumerates (`scan list ⊇ spec ban`), deriving the spec set from `host-interfaces-services.md` so a spec widening reddens CI rather than waiting for release-time item 9.
- **Plan edits.** In `H3a-di-seam-skeleton.md` **Tests**, add a check that fails when a PIC-12-banned timing primitive is absent from the scan list. In `conventions.md`, note that the superset invariant is mechanically backed and narrow item 9 to the residual it no longer covers (e.g. deliberate superset entries beyond the spec ban).
- **Spec edits.** PIC-12's ban is prose; a robust mechanical read needs a stable, parseable enumeration. If the adjacent backtick-delimited primitive tokens at the `#pic-12` anchor are not a sufficiently stable parse target, add an anchored machine-readable enumeration of the banned primitives to PIC-12 for the test to read.
- **Pros.** Closes the drift gap at CI time; a spec widening fails fast instead of surviving to the release gate.
- **Cons.** Couples a plan-side test to spec prose/anchor format; may require a spec edit to expose a stable enumeration.
- **Risks.** A brittle parse of PIC-12 prose could false-fail on benign spec reformatting; the stable-enumeration spec edit is itself a PIC-12 change owned spec-side.

### Option B — Spec-edit-time plan-maintenance trigger

- **Approach.** Keep the list manual but route its reconciliation to a known maintenance trigger that fires whenever PIC-12 changes, mirroring the existing *Transitive-completeness plan-maintenance* rule, instead of relying solely on the release-time sweep.
- **Plan edits.** Add a cross-cutting plan-maintenance rule in `conventions.md` (alongside the *No globals* rule) obliging any edit that widens the PIC-12 `Clock`/`WallClock` ban to update the `H3a` scan list in the same change; cross-reference it from `H3a` and from residue item 9.
- **Spec edits.** None.
- **Pros.** Matches the plan's existing plan-maintenance idiom; no spec format change; lower blast radius.
- **Cons.** Still contributor-discipline, not mechanical — a missed trigger reproduces the same silent drift, just earlier than release.
- **Risks.** Relies on the spec-editing contributor noticing the cross-cutting obligation.

### Recommendation

Option A. The finding's defect is precisely that reconciliation is non-mechanical; only a CI assertion that the scan list covers the PIC-12 enumeration removes the drift window rather than relocating it. Where PIC-12's prose is not a stable parse target, add an anchored machine-readable enumeration to PIC-12 and have the `H3a` test read that. Edge case the implementer must watch: the scan list is permitted to be a strict superset of the spec ban, so the assertion checks coverage (`⊇`), not equality, and must not red on a deliberate extra scan-list entry.
