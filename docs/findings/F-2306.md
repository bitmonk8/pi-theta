## Finding

`H1a` is the single owner of loom's own toolchain provisioning. Its **Adds** declares the four `@earendil-works/pi-*` SDK packages, `vitest`, the `eslint` / `@typescript-eslint/parser` / `eslint-plugin-loom-local` lint toolchain, and `typescript` (the `tsc` provider behind `npm run typecheck`) as loom's own `devDependencies`, "so a fresh `npm install` provisions them under loom's own `node_modules`." Several downstream gates read or import directly from those devDependency-provisioned modules:

- `V18a` / `V18b` — the `SDK_SURFACE_INVENTORY` presence and closure audits, which import the installed SDK surface.
- `V18c` — the `engines.node` three-way equality gate's operand (iii) live read of the installed `@earendil-works/pi-coding-agent` `package.json`, and the `loom/typecheck/session-shutdown-reason-snapshot` brand-string gate compiled by the `npm run typecheck` `tsc`.
- `V18d` / `H4a` — the harness that compiles and runs the extension against the installed SDK.
- `H2a` — the `eslint-plugin-loom-local` `file:./tools/eslint-plugin-loom-local` `file:` resolution (witnessed by the green `npm install` in `H1a`'s **Ships when**) plus the `no-broad-catch` / `no-restricted-syntax` gates that load the installed `eslint` engine and parser.

Every one of these reads from a `node_modules` tree that `dependencies` alone does not populate. `H1a` pins the package manager and layout ("loom's own build, test, and CI all run under **npm** with a flat/hoisted `node_modules` tree") and names `npm install` as the Ships-when witness, but it never states that the install must be **dev-inclusive**. A build or CI step configured with `npm ci --omit=dev` (or a `--production` install) would provision only `dependencies`, leaving the SDK packages, `vitest`, the lint toolchain, and `tsc` absent — and the gates above would fail with module-not-found / unresolved-plugin errors. The `pnpm`/`yarn` skew language in `host-prerequisites.md` governs the **consumer-facing** install; it does not cover this orthogonal `dev`-vs-`production` dimension of loom's own checkout.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** c8564bb (2026-06-12), 8b14129 (2026-06-16)
**History:** The plan corpus is git-tracked. The dev-inclusive reliance entered `docs/plan_topics/H1a-scaffold-and-toolchain.md` in two pieces, neither of which stated the constraint. `c8564bb` ("resolve 'Pi SDK never provisioned into build-test environment'") added the four `@earendil-works/pi-*` packages to loom's own `devDependencies` "so a fresh `npm install` provisions the SDK … for the downstream `V18a`/`V18b`/`V18c`/`V18d` gates", establishing the SDK-read half of the reliance. `8b14129` ("resolve 'H1a never pins the package manager its file: resolution and flat node_modules layout assume'") pinned npm + flat/hoisted `node_modules` and the `eslint-plugin-loom-local` `file:` resolution, establishing the lint-plugin half. Both commits assumed the default dev-inclusive `npm install`; neither added a statement excluding a `--omit=dev` / `--production` install. The gap is the interaction of these two provisioning reliances with an install-mode dimension that no commit addressed.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H1a-scaffold-and-toolchain.md`, extend the existing **Adds** sentence that already pins the package manager and layout ("loom's own build, test, and CI all run under **npm** with a flat/hoisted `node_modules` tree, and that is the package manager and layout the leaf's two install-time reliances below assume") so it additionally pins that loom's own build/test/CI install is **dev-inclusive** — it resolves `devDependencies` (a plain `npm install`, not `npm ci --omit=dev` or a `--production` install) — because the `V18a`–`V18d` reads, the `loom/typecheck/*` brand-string `tsc` gate, and the `eslint-plugin-loom-local` `file:` resolution all read from devDependency-provisioned `node_modules`. The `npm install` witness already in **Ships when** stays as-is. This is a plan-internal statement only; the consumer-facing skew contract in `host-prerequisites.md` is read-only for this fix and the SDK pin/range fields are untouched.

## Consequence

**Severity:** advisory

A build/test/CI step run with a production install (`npm ci --omit=dev` / `--production`) provisions only `dependencies`, so the `V18a`–`V18d` SDK reads, the `loom/typecheck/*` brand-string `tsc` gate, and `H2a`'s `eslint-plugin-loom-local` `file:` resolution fail with module-not-found / unresolved-plugin errors. The `H1a` Ships-when witness already names `npm install`, so an implementer following the leaf produces a working checkout; the gap is the missing explicit constraint that a downstream CI author would otherwise be free to violate.
