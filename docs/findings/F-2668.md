## Finding

`type-system.md` refers to the anonymous object-type form (`{ field: T, ... }`) under several different labels with no declared equivalence between them:

- the type-expression intro bullet names the form **"Inline anonymous objects"** (line 10);
- TYPE-1's reflexivity note and the **Operational definition** paragraph call it **"inline objects"** / **"an inline object"** (lines 29, 35);
- the load-bearing compatibility rules TYPE-8 and TYPE-10 call it **"inline object type"**, and TYPE-10 additionally introduces **"inline-object value"** for a value of that type (lines 42, 52).

Nothing on the page states that "Inline anonymous objects", "inline objects", and "inline object type" denote the same construct. A reader encountering the intro bullet ("Inline anonymous objects") and then TYPE-8 ("inline object type" — explicitly contrasted with `NamedType`) must infer the equivalence to follow the nominal-vs-structural distinction that TYPE-8/TYPE-10 turn on. The page's normative rules consistently use **"inline object type"** (and reserve **"inline-object value"** for the value, correctly distinguishing type from value); the intro bullet and the TYPE-1/Operational-definition prose are the outliers.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** present-since-inception (intro bullet "Inline anonymous objects" / TYPE-1 "inline objects"); `9a4e078` "pi-loom spec: resolve \"TYPE-8 field-wise compatibility scope ambiguous\"" (2026-06-07) introduced the diverging "inline object type" / "inline-object value" labels.
**History:** The corpus is git-tracked. `git log -S` shows the intro bullet "Inline anonymous objects" and TYPE-1's "identical inline objects" already present in `type-system.md` (and in the pre-split `spec.md`) before commit `9a4e078`; at that commit's parent, TYPE-8 read "Field-wise on object schemas: an object type ..." (no "inline object type" label) and TYPE-10 did not exist. Commit `9a4e078` rewrote TYPE-8 to "Field-wise on inline object types: an inline object type ...", added TYPE-10 introducing "inline object type" and "inline-object value", and amended the Operational definition to mention "an inline object" — establishing the third label set alongside the inception-era ones without reconciling the intro bullet. The inconsistency is thus the interaction of the inception-era labels with the labels `9a4e078` added; neither commit is independently wrong.

## Solution Space

**Shape:** single

### Recommendation

Standardise `type-system.md` on **"inline object type"** as the name of the form, matching the normative usage already in TYPE-8 and TYPE-10:

- Rename the intro bullet from **"Inline anonymous objects"** to **"Inline object types"** (the `{ field: T, ... }` example and the "named schemas are preferred" gloss stay).
- In the **Operational definition** paragraph, change "a named schema and an inline object with identical fields" to "... an inline object type with identical fields".
- In TYPE-1's note, change "identical inline objects" to "identical inline object types".

Keep TYPE-10's **"inline-object value"** as-is: it denotes a *value* of an inline object type (contrasted with "named-schema value"), so the value/type distinction it draws is intentional and correct. Do not rename it to a type label.

Edge case for the implementer: the grammar production site (`grammar.md`, `ObjectType ::= ...` and the "Inline object types." note) uses "inline anonymous object type", and `influences.md` / `placeholder-rendering-a.md` use "inline anonymous object types". This finding is scoped to `type-system.md`; aligning those other pages on the same canonical term is out of scope here but worth a follow-up if a corpus-wide term is desired.

## Consequence

**Severity:** cosmetic

A reader must infer that three labels name one construct; the inference is easy enough that no two implementers would diverge on behaviour. Left unfixed it is a wording inconsistency on a page whose nominal/structural distinction (TYPE-8 vs TYPE-10) leans on the term being read precisely.
