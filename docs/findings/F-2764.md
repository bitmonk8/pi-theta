## Finding

The `loom/load/invalid-tool-rename` Tests bullet in `V6c` (and its mirror in `V6c-T`) reads: "a `tools:` `as` rename target that is not loom-identifier-shaped (e.g. `as MyTool`) is rejected at load time." The parenthetical example `as MyTool` is meant to demonstrate a non-identifier-shaped target, but `MyTool` is a perfectly well-formed identifier under the general `[A-Za-z_][A-Za-z0-9_]*` grammar (`lexical.md` §Identifiers). What makes it invalid is loom's **lowercase-first** rule: binding-position identifiers (which a `tools:` rename target is) must start with a lowercase letter or `_`, so an uppercase first letter trips `loom/parse/binding-case-mismatch` / the `invalid-tool-rename` load-time check (`code-registry-load.md`: "`as <name>` target is not loom-identifier-shaped (e.g. `as MyTool` instead of `as my_tool`)").

A reader who does not already know the lowercase-first rule cannot infer *why* `MyTool` is rejected — the example looks like a valid name. The bullet therefore fails to communicate which property the test must assert, leaving a test author free to reject `MyTool` for an unrelated reason or to mis-state the violated property.

The cited spec topic (`frontmatter-fields-a.md` §`tools`) does state the rule plainly ("the override target must obey loom's lowercase-first identifier rule"), so an implementer who opens the Spec page recovers the intent; the gap is confined to the plan bullet reading as self-contained when it is not.

## Consequence

**Severity:** advisory

A test author reading only the plan bullet cannot tell why `as MyTool` is rejected and may assert the wrong property (e.g. test a different malformed-name class) or transcribe a confusing example into the test. The cited spec topic carries the rule, so a faithful implementer who reads the Spec field still lands a correct test; the cost is local confusion and a risk of a mis-targeted assertion, not a build break.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V6c-tools-set.md` and `docs/plan_topics/V6c-T-tools-set.md`, revise the `loom/load/invalid-tool-rename` Tests bullet so the example names the rule it violates rather than leaving the reader to infer it. Keep the spec's canonical `as MyTool` example (it matches `frontmatter-fields-a.md` and `code-registry-load.md`) and annotate it with the violated property — for instance: "a `tools:` `as` rename target that is not loom-identifier-shaped (e.g. `as MyTool` — an uppercase first letter violates loom's lowercase-first identifier rule) is rejected at load time." Apply the identical edit to both the implementation leaf and its `-T` partner so the two stay in lockstep. No spec edit is needed; the spec already states the rule and is read-only for this fix.
