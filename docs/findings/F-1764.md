## Finding

Step 6 of the version-bump procedure (`version-bump-triggers.md`) relies on a "build-time `Api`-coverage assertion … the mechanical gate that fails until this step has been completed" to detect when "the candidate minor introduces a new `Api` value not yet listed". The canonical owner — `provider-error-mapping.md` §Provider seed-field mapping — describes the assertion as one that "enumerates pi-ai's exposed `Api` literal-union values and asserts every value appears as a row key in the seed-field table constant". That names the *source* of the candidate set (pi-ai's `Api` literal-union) but never pins the *mechanism* that makes the assertion mechanically checkable.

The gap is concrete because `Api` is a TypeScript literal-union *type* declared in `@earendil-works/pi-ai` (per `host-interfaces-core.md` §model-registry-pin), with no runtime value that lists its members — the seed-field mapping is "keyed on the resolved binder model's `api` field", which yields a single runtime value per model, not the full union. Enumerating a literal-union type's members at build time is therefore necessarily a type-level operation; it cannot be done by a runtime fixture iterating host-available models. The spec never states (a) which `@earendil-works/pi-ai` surface the assertion imports `Api` from, (b) whether the assertion is gated by `npm run typecheck` or `npm test`, or (c) how the type is reduced to a checkable form.

This stands in sharp contrast to the sibling `SessionShutdownEvent['reason']` assertion in step 5, which is pinned in full: the import is from the bare-package specifier, the gate is `npm run typecheck` (distinct from the step-2(a) `npm test` literal-read), the failure surfaces a verbatim grep-able brand string (`loom/typecheck/session-shutdown-reason-snapshot`), and the type-equality construct is described down to a non-normative example. The `Api`-coverage assertion is named but its parallel detail is absent.

## Solution Space

**Shape:** single

### Recommendation

On `provider-error-mapping.md` §Provider seed-field mapping (the canonical owner of the gate), pin the `Api`-coverage assertion as a type-level coverage check gated by `npm run typecheck`, parallel to the `SessionShutdownEvent['reason']` type-equality assertion:

- **Import surface.** State that the assertion imports the `Api` type from the same `@earendil-works/pi-ai` surface `host-interfaces-core.md` §model-registry-pin pins for `Model`/`Api` (re-exported through the model-registry surface), so a rename of the type fails import-resolution and routes through the surface-inventory update exactly as the reason-assertion's import does.
- **Mechanism.** The assertion is a compile-time check that every member of the `Api` literal-union appears as a key of the seed-field table constant — e.g. a conditional type that resolves to a diagnostic-bearing `never` for any `Api` member not assignable to `keyof typeof seedFieldTable`. A new pi-ai `Api` value then fails `tsc` and names the missing member. This is type-level because the union is erased at runtime; a runtime fixture reading the model registry cannot enumerate the full union and MUST NOT be used as the coverage mechanism.
- **Gate and surfacing.** State that the assertion runs under `npm run typecheck` (not the step-2(a) `npm test` literal-read), and that its failing `tsc` diagnostic MUST carry a verbatim grep-able brand string (an `Api`-coverage analogue of `loom/typecheck/session-shutdown-reason-snapshot`), so the step-6 trigger is mechanically identifiable, on the same footing as step 5.
- **Step 6.** Replace step 6's bare "build-time `Api`-coverage assertion" reference with a cross-reference to the pinned mechanism above, keeping the fixture-rerun obligation (which is the genuine `npm test` half of step 6) distinct from the coverage assertion.

Edge case the implementer must watch: the seed-field fixtures rerun under `npm test` (the provider seed-field *values*) are a separate gate from the coverage assertion (the `Api`-key *presence* check under `npm run typecheck`); keep the two gates and their arms distinct so a new-`Api` red and a seed-field-rename red are individually routable.

This also aligns the plan: `V18c`'s Ships-when must move the `Api`-coverage gate from the `npm test` arm to the `npm run typecheck` arm, and `V18a`'s `api-coverage` row must back a type-level check rather than a runtime read.

## Consequence

**Severity:** correctness

Two implementers diverge on how to build the gate: a type-level coverage assertion under `npm run typecheck` versus a runtime fixture under `npm test`. The runtime approach cannot enumerate the erased literal-union type and would only ever see host-available models, producing a gate that silently fails to fire on a new pi-ai `Api` value — so a newly-added provider silently defaults to `omitted` seed handling with no deliberate decision. The plan already guessed `npm test` for this gate, so the divergence is not hypothetical.
