## Problem

§Provider seed-field mapping (`id="provider-seed-field-mapping"`) describes the `Api`-coverage assertion only by its source — pi-ai's `Api` literal-union values — without pinning the build-time mechanism: which `@earendil-works/pi-ai` surface the assertion imports `Api` from, whether it is gated by `npm run typecheck` or `npm test`, and how the literal-union type is reduced to a checkable form. Because `Api` is a TypeScript literal-union type with no runtime value listing its members, enumerating it is necessarily a type-level operation; a runtime fixture iterating host-available models cannot see the full union. This stands in contrast to the sibling step-5 `SessionShutdownEvent['reason']` assertion, which is pinned in full (import specifier, `npm run typecheck` gate, verbatim grep-able brand string). Left unpinned, an implementer could build a runtime fixture that silently fails to fire on a new pi-ai `Api` value, so a new provider defaults to `omitted` seed handling with no deliberate decision.

## Issue introduction

**History:** The `Api`-coverage assertion entered the spec (then `spec_topics/pi-integration-contract.md`) in 52219c4 with the phrasing "enumerates pi-ai's exposed `Api` literal-union values and asserts every value appears as a row key" — naming the source but not pinning the build-time mechanism (import surface, `npm test` vs `npm run typecheck`, how the literal-union type is reduced to a check). Later edits preserved that phrasing verbatim through the size-cap reflow (f5e89f4, 2026-06-04) and the page split into `provider-error-mapping.md` / `version-bump-triggers.md` (81bf52a, 2026-06-11), while the sibling `SessionShutdownEvent['reason']` assertion was progressively pinned in detail across later June commits, widening the asymmetry the finding observes.

## Solution approach

Pin the `Api`-coverage assertion on §Provider seed-field mapping (anchor `id="provider-seed-field-mapping"`) as a type-level, compile-time coverage check gated by `npm run typecheck`, parallel to step 5's `SessionShutdownEvent['reason']` type-equality assertion. State which `@earendil-works/pi-ai` surface the assertion imports `Api` from — the surface `host-interfaces-core.md` §model-registry-pin already pins for `Model`/`Api` — that the check asserts every member of the erased `Api` literal-union appears as a key of the seed-field table constant (a runtime fixture cannot enumerate the union and is not the coverage mechanism), and that its failing `tsc` diagnostic surfaces a verbatim grep-able brand string. Rewrite step 6 of `version-bump-triggers.md` to cross-reference the pinned mechanism while keeping its `npm test` fixture-rerun obligation distinct from the coverage assertion.

## Solution constraints

- Pin the mechanism only on `provider-error-mapping.md` §Provider seed-field mapping (the canonical owner); step 6 on `version-bump-triggers.md` must cross-reference it rather than restate the mechanism, to avoid a two-source-of-truth split.
