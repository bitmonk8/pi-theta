## Finding

The ceiling-#4 per-boundary table (`hard-ceilings/ceilings-3-and-4.md#ceiling-4-table`) enumerates five enforcement points, and `V5e`'s per-boundary routing-*decision* text delegates the live-carrier wrapping of each runtime breach to a named site owner: typed-query response → `V13c`, code-driven tool args → `V14e`, `params` / `invoke<T>` return → `V15j`, slash-load `params` cross-route → `V4e`. Of these delegated live carriers, every one has a coverage-matrix presence except `V15j`: `V13c` and `V4e` appear as `CIO-1` co-witnesses, and `V14e` is listed in the `cka-13` (`tool-calls.md`) row. `V15j` appears in no row of either the numbered table or the code-keyed-obligation-areas table.

`V15j` exists, is wired into `H5b`'s Deps, and is named in `V5e`'s decision text as the `InvokeInfraError` carrier for the `params` (runtime `invoke(...)`) and `invoke<T>`-return ceiling-#4 rows — but nothing in `coverage-matrix.md` records that mapping. The matrix's own contract is that it maps each executable obligation to its closing leaf(s); the delegated live carrier for the invoke-boundary ceiling-#4 routing is the sole ceiling-#4 carrier missing from that record, leaving the invoke-boundary arm of `CIO-3` with no traceable closing-leaf entry and producing the asymmetry with `V14e`/`cka-13`.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `323d361` — *pi-loom plan: resolve "V15h bundles two orthogonal failure modes"* (2026-06-18)
**History:** `git log -S 'V15j' -- plan_topics/coverage-matrix.md` returns no commits — `V15j` has never appeared in the coverage matrix. The `V15j` / `V15j-T` leaves were split out as the dedicated live carrier for the ceiling-#4 `params` / `invoke<T>`-return rows in `323d361`, which also added `V15j` to `H5b`'s Deps and named it as the `InvokeInfraError` carrier in `V5e`'s per-boundary decision text — but omitted the corresponding matrix entry. The sibling tool-call carrier `V14e` was added to the `cka-13` row in its own split commit `5a1915d` (*"V14a bundles eight independently-testable concerns"*), so the asymmetry dates precisely to the `323d361` split skipping the matrix update that the `V14e` split performed.

## Solution Space

**Shape:** multiple

### Option A — Code-keyed-area row mirroring `cka-13`

**Approach.** Add a new append-only `cka-<n>` row (next free token) keyed to the invocation-boundary ceiling-#4 routing, with closing leaf `V15j`, mirroring how `cka-13` (`tool-calls.md`) houses `V14e`.
**Plan edits.** In `coverage-matrix.md`'s Code-keyed obligation areas table, append a row whose *Token* cell is the next free `cka-<n>`, *Spec area* names the `ceilings-3-and-4.md` ceiling-#4 `params` / `invoke<T>`-return invoke-boundary rows (cross-referencing `#ceiling-4-table` / `#cio-3`), and *Closing leaf(s)* cell is the backtick-delimited token `` `V15j` ``.
**Spec edits.** None.
**Pros.** Restores exact structural symmetry with `V14e`/`cka-13` — both ceiling-#4 live carriers tracked the same way, on the page-keyed cka mechanism. Closes the asymmetry the finding names directly.
**Cons.** The cka table is framed for obligations carrying no numbered `PREFIX-N` REQ-ID; the ceiling-#4 routing is anchored by `CIO-3`, so a cka row slightly stretches that framing (though `V14e` in `cka-13` is the same precedent).
**Risks.** Must use the next free `cka-<n>` (append-only; tokens never reused) and backtick-delimit `` `V15j` `` exactly, per *Closing-leaf column authoring*, or the `H5d` tokeniser silently drops it.

### Option B — `CIO-3` co-witness in the numbered table

**Approach.** Record `V15j` as a delegated live-carrier co-witness on the `CIO-2, CIO-3, CIO-4` row, mirroring the existing co-witness annotations on `ERR-13` and `CIO-1`.
**Plan edits.** In the Numbered REQ-IDs table, extend the `CIO-2, CIO-3, CIO-4 | V16a` row's closing-leaf cell to add `` `V15j` `` with a parenthetical annotation scoping it to the delegated ceiling-#4 invoke-boundary (`params` / `invoke<T>`-return) live-carrier role.
**Spec edits.** None.
**Pros.** Keeps the `CIO-3`-anchored obligation in the numbered table where its REQ-ID lives; matches the established delegated-live-carrier co-witness annotation precedent (`ERR-13`, `CIO-1`).
**Cons.** Introduces a *different* asymmetry — the tool-call carrier `V14e` stays in `cka-13` while the invoke carrier becomes a numbered-table co-witness, so the two ceiling-#4 carriers are tracked by different mechanisms.
**Risks.** The annotation must scope `V15j` to `CIO-3` only (it does not co-witness `CIO-2`/`CIO-4`); backtick-delimit the leaf ID, leave annotation prose unbackticked.

### Recommendation

Take **Option A**. The finding's defect is the asymmetry with `V14e`/`cka-13`; placing `V15j` in a page-keyed `cka-<n>` row resolves it on the same mechanism already used for the sibling tool-call live carrier, whereas Option B would replace one asymmetry with another. Append the next free `cka-<n>` token, set the closing-leaf cell to `` `V15j` `` (backtick-delimited, no other backticked span in the cell), and cross-reference `ceilings-3-and-4.md#ceiling-4-table` / `#cio-3` in the spec-area cell. Edge case: do not fold `V15j` into the existing `cka-14` (`invoke` parse/load diagnostics → `V15f`) or `cka-15` (cross-mode → `V15d`) rows — neither's scope is ceiling-#4 routing.

## Consequence

**Severity:** correctness

`coverage-matrix.md` claims to map every executable obligation to its closing leaf(s), but the invoke-boundary ceiling-#4 live carrier `V15j` has no entry, so an auditor (or the residue-inspection item-7 split-row sweep) tracing ceiling-#4 coverage finds no closing-leaf record for the `params` / `invoke<T>`-return rows. A future regression or removal of `V15j` would not be flagged by any matrix-keyed gate, and two maintainers would diverge on whether the invoke-boundary carrier is tracked.
