## Problem

Every mechanical correctness gate in the plan is scoped to `src/**` — the `H2a` no-globals architectural test, the `no-broad-catch` and `no-restricted-syntax` lint rules in `conventions.md`, the `H3a` ambient-primitive scan, and the `H5a`/`H5c` closing-gate allow-list reconciliation — yet no plan file states `src/**` as the production-code root. The spec roots the actual entry point elsewhere: `host-prerequisites.md` §"Extension entry point" declares the loaded module is `extensions/index.ts` (auto-discovered from `package.json#pi.extensions = ["./extensions"]`). Production code under `extensions/` — including the entry file Pi loads on every invocation — sits outside `src/**` and is invisible to every gate above. A broad `catch`, a module-level singleton, a banned ambient-primitive read, or a `Promise.all` placed in the entry module passes CI unflagged.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `9a53d9c` (2026-05-01, "Initial scaffold: package.json, README, spec.md, extensions/" — set `package.json#pi.extensions = ["./extensions"]` and created `extensions/.gitkeep`); `288f191` ("Add implementation plan with horizontal/MVP/vertical-slice phases" — first plan content scoping gates to `src/`). The current gate corpus (`H2a`/`H3a`/`H5a`/`H5c` plus the `conventions.md` cross-cutting rules) was re-authored in `c6a664e` (2026-06-10) and carries the same unreconciled `src/**` scope.
**History:** The `src/**`-scoped gate convention and the `package.json#pi.extensions = ["./extensions"]` entry-point declaration have coexisted across every plan revision. `git log -G 'src/'` over `docs/plan_topics/conventions.md` and the pre-split `plan.md`, and `git log -S '"extensions": ["./extensions"]'` over `package.json`, show no commit that ever stated the single production-code root or reconciled it against the entry path. The gap is a standing omission, not introduced by any single later edit.

## Solution approach

In `conventions.md` §"Cross-cutting rules (every leaf)", declare the single production-code root the gates assume and reconcile it with the Pi entry point: production logic lives under `src/**`, and `extensions/index.ts` is a thin delegator re-exporting the `src/**` factory, so no rule-subject behaviour falls outside the gated tree. Cite `host-prerequisites.md` §"Extension entry point" for the entry-path fact rather than restating it. Update `H4a`'s Adds to pin the factory implementation under `src/**` and describe `extensions/index.ts` as the delegating entry shim. Close the escape hatch for the entry file specifically by stating it carries no rule-subject logic and exists only to re-export the `src/**` factory.

## Solution constraints

- Adopt the `extensions/index.ts` factory placement owned by the H1a/H4a entry-point finding — this reconciliation states where the gated `src/**` code meets the entry path; it does not re-decide where the entry file lives.
- The spec is read-only for this fix: cite `host-prerequisites.md` §"Extension entry point" rather than editing it.
