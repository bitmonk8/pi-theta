## Finding

The Depth Enforcement section in `schema-subset.md` pins two implementation choices that have no observable correlate. First, the enforcement-point paragraph states that "the depth check lives in the `SchemaValidator` service (the same component that owns the AJV cache)" — a component-layout and ownership decision. Second, the walk paragraph states the implementation "is a recursive descent over the parsed JSON value with a depth counter" — one specific traversal strategy. Neither constrains conforming behaviour: any traversal (recursive, iterative, stack-based, BFS) that computes the same per-node depth metric and reports the same first-too-deep node is conformant, and which runtime component physically hosts the check is invisible to every observer.

What is genuinely observable, and must stay normative, is already stated alongside this prose: the counting algorithm (scalar/empty = 1, non-empty = `1 + max(child)`, `anyOf` arms not counted, cap `≤ 5`) and its worked examples; that the walk runs **before** AJV at each of the five enumerated enforcement sites; and the failure carrier, whose `path` is the JSON Pointer to the first node exceeding depth 5 (`schema_keyword: "maxDepth"`, message `"JSON document depth exceeds 5"`, `cause: "schema_validation"`). The over-pinned sentences should be narrowed to those observables, with the descent-mechanism wording dropped or demoted to a non-normative hint, and the component-ownership statement confined to Implementation Notes (which already owns the `SchemaValidator`/AJV-cache description). The neutral behavioural phrasing "policed by the validator service alongside AJV" already present at the head of the section is acceptable as-is.

The defect is therefore an overcorrection that entered with the section's creation, not a pre-existing condition.

## Solution Space

**Shape:** single

### Recommendation

Narrow the two over-pinned sentences to their observable content:

- In the enforcement-point paragraph, drop "lives in the `SchemaValidator` service (the same component that owns the AJV cache)". State only the behavioural requirement: the depth check runs before AJV at each of the five enumerated sites. The neutral "policed by the validator service alongside AJV" at the section head already conveys the necessary behavioural framing and can stay.
- In the walk paragraph, replace "Implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits and produces the failure" with an observable statement: the walk reports a violation whose `path` is the JSON Pointer to the first node whose depth exceeds 5. If a concrete traversal is worth illustrating, demote "recursive descent with a depth counter" to a parenthetical non-normative implementation hint.
- Add the component-ownership detail (which runtime service hosts the check and its relationship to the AJV cache) to `implementation-notes.md` — Runtime, which already describes the injected `SchemaValidator` and its compiled-validator cache, and cross-reference it from the Depth Enforcement section rather than asserting the layout normatively.

Keep normative and unchanged: the counting-algorithm bullets, the worked examples, the five enforcement-point rows and their destination/surface table, and the JSON-Pointer-of-first-too-deep-node in the error carrier — these are the observable obligations.

## Consequence

**Severity:** advisory

An implementer can still build a conformant validator from this text, so nothing breaks. The cost is over-constraint: the prescriptive sentences read as normative requirements, so an implementer may believe a recursive-descent traversal or a specific `SchemaValidator`-owns-the-AJV-cache layout is mandated when only the depth metric, the before-AJV ordering, and the first-too-deep-node JSON Pointer are actually load-bearing.
