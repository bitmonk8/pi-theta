## Finding

The pickability predicate in `conventions.md` §Leaf format (**Deps.**) is defined purely on tag *presence*: a `Deps.` entry is satisfied "exactly when the §Per-phase TDD ritual completion tag for that entry is present" — `<id>-complete` for an implementation/horizontal leaf, `<id>-T-complete` for a tests task — evaluated against a clone that has fetched tags. The predicate's only stated failure modes are an unpushed local tag and a shallow/no-tags clone (both make the tag read as *absent*). It never addresses the inverse: a tag that is present but no longer reflects completed work.

Git tags are append-only by nature — reverting a leaf's work in a later commit does not remove the tag that was pushed at the leaf's exit gate. The plan itself relies on revert as a first-class workflow step: the §Release gate prose states "Reverting the `H6a` activation commit returns the gate to its `H5a` seeded-fixture footing", and `V18d` owns a pre-merge "revert path that restores the prior pin". After such a revert the `H6a-complete` / `V18d-complete` tag still points at the now-reverted commit, so the pickability predicate continues to read the dependency as satisfied even though the work no longer holds. The plan never states whether a reverted leaf retracts its completion tag, nor any other mechanism by which a reverted leaf's pickability signal is invalidated.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 4ee2700 — pi-loom plan: resolve "Deps satisfied build-order step names no machine-checkable pickability predicate" (2026-06-17, Thomas Andersen)
**History:** The formal tag-presence pickability predicate ("a `Deps.` entry is *satisfied* … exactly when the … completion tag for that entry is present") was introduced into `conventions.md` §Leaf format by commit 4ee2700 on 2026-06-17. That commit defined satisfaction purely on tag presence and enumerated only the tag-absent failure modes (unpushed tag, shallow clone); the tag-lifecycle-on-revert gap has been present since the predicate's introduction.

## Solution Space

**Shape:** single

### Recommendation

In `conventions.md` §Leaf format, within the **Deps.** pickability-predicate paragraph (alongside the existing unpushed-tag / shallow-clone caveats), state the completion-tag lifecycle under revert. The added text must establish that the predicate's tag-presence reading is only sound while the tag reflects work that still holds, and pin the obligation that resolves it — either that a leaf whose completion is reverted has its `<id>-complete` / `<id>-T-complete` tag removed from the shared remote (so the predicate reads it as absent), or an equivalent stated mechanism by which a reverted leaf's pickability signal is invalidated. Name the plan's own revert scenarios — the `H6a` activation revert and the `V18d` pre-merge pin-restore — as the concrete cases the statement must cover. The implementer should choose between the tag-retraction wording and an alternative invalidation mechanism based on what the team's git-tag workflow can actually enforce.

## Consequence

**Severity:** advisory

If a completed leaf's work is later reverted (the H6a activation revert or the V18d pre-merge pin-restore are the plan's own examples), its `-complete` / `-T-complete` git tag persists at the reverted commit, so the pickability predicate still reports the dependency as satisfied. A contributor evaluating pickability against a tag-fetched clone could therefore pick a dependent leaf whose dependency is no longer actually complete, with no signal that the satisfied reading is stale.
