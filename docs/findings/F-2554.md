## Finding

`grammar.md` (Loom literal sublanguage) defines `NamedValueLit ::= Ident "." Ident` with the gloss *"Enum.Variant access; head is an enum name in scope, tail a declared variant"*, and separately asserts *"The is-literal check runs at parse time against the Loom AST; it does not require a separate parser."* These two statements are in tension. Deciding that the head `Ident` is an in-scope enum and the tail `Ident` is a declared variant of that enum is name/scope resolution — it cannot be performed by a pure AST-shape check. The spec never says whether the enum-in-scope / declared-variant validation is folded into the parse-time is-literal gate or deferred to a downstream resolution pass, so two implementers diverge on *when* the check runs and *which* diagnostic it produces.

The diagnostic assignment is correspondingly under-specified for the failure cases in a literal position (a `params:` default RHS or a Pi-tool call argument):

- A literal-position `Enum.Variant` whose **tail** is not a declared variant: `loom/parse/unknown-variant` exists (owned by `schemas.md` — Variant access, and registered in the parse code-registry), but the spec never states that it fires inside the literal sublanguage, nor in what order relative to the is-literal shape gate. The is-literal gate's failure codes (`loom/parse/default-not-literal` / `loom/parse/tool-arg-not-literal`) are the competing candidates, and nothing disambiguates.
- A literal-position dotted pair whose **head** does not resolve to an enum (a `let` binding, a schema or `fn` name, or an unresolved identifier): no diagnostic is named at all. The forbidden-forms list says "member access on anything other than `Enum.Variant`" is rejected — but classifying `obj.field` as *not* `Enum.Variant` itself requires resolving that `obj` is not an enum, which is exactly the resolution step whose phase is unstated. The candidate outcomes (`default-not-literal`/`tool-arg-not-literal`, `loom/parse/unknown-identifier`, or an unnamed not-an-enum code) are never reconciled.

Because diagnostic codes are an observable, version-pinned surface, leaving the phase model and the head-not-an-enum disposition unspecified lets conforming implementations emit different codes for the same source.

## Solution Space

**Shape:** multiple

### Option A — is-literal stays a pure AST-shape gate; enum-scope resolution rides the whole-file resolution pass

- **Approach:** The is-literal check admits any `Ident "." Ident` node as a syntactically well-formed `NamedValueLit` without consulting scope. Enum-in-scope and declared-variant validation are performed by the existing whole-file resolution pass — the same pass that resolves `params:` `NamedType` references (`frontmatter-fields-a.md`) and already owns `loom/parse/unknown-variant`. A literal-position dotted pair whose tail is not a declared variant fires `loom/parse/unknown-variant`; one whose head resolves to a non-enum symbol or to nothing fires a head-not-an-enum diagnostic.
- **Spec edits:** In `grammar.md`, replace the production comment's "head is an enum name in scope, tail a declared variant" with a shape-only gloss plus a note that enum-in-scope/variant validation is a downstream resolution-pass check; scope the "runs at parse time against the Loom AST; does not require a separate parser" sentence to AST shape and forward-link the resolution-pass check. In `code-registry-parse.md`, state `unknown-variant` fires in literal positions and register a head-not-an-enum code (or designate `unknown-identifier` for the unresolved-head case).
- **Pros:** Matches the whole-file forward-reference resolution model used everywhere else; reuses `unknown-variant` and preserves its specific "unknown variant X on enum Y" message; keeps the is-literal gate a cheap pure-AST check, consistent with the "no separate parser" claim.
- **Cons:** Requires naming/registering the head-not-an-enum disposition and confirming `unknown-variant`'s cross-position applicability.
- **Risks:** The ordering between the AST-shape gate and the resolution pass must be pinned, so a malformed shape (e.g. `a.b.c`, member chains) fires the position's not-literal code before resolution runs.

### Option B — wire enum scope into the is-literal check

- **Approach:** The is-literal check itself consults enum scope; a literal-position `Ident "." Ident` whose head is not an in-scope enum or whose tail is not a declared variant is rejected directly as `loom/parse/default-not-literal` / `loom/parse/tool-arg-not-literal`.
- **Spec edits:** Amend the is-literal-check prose to state it requires enum-scope resolution (removing the "does not require a separate parser" claim) and that bad head/variant in literal positions surface as the position's not-literal code.
- **Pros:** One diagnostic per position; no new code to register.
- **Cons:** Contradicts the standing "runs at parse time against the Loom AST; does not require a separate parser" claim; collapses the `unknown-variant` signal (authors lose the specific variant message in literal positions); diverges from the whole-file resolution model used everywhere else.
- **Risks:** A scope-dependent literal gate interacts badly with whole-file forward references — a literal referencing an enum declared later in the same file must still resolve.

### Recommendation

Adopt **Option A**. State in `grammar.md` that the is-literal check is a pure AST-shape gate that admits any `Ident "." Ident` and that enum-in-scope / variant validation rides the whole-file resolution pass: `loom/parse/unknown-variant` for an unknown tail, and a registered head-not-an-enum code (or `loom/parse/unknown-identifier` when the head resolves to nothing) for a head that is not an enum. Pin that the AST-shape gate runs first — non-`Ident "." Ident` member-access shapes fire the position's not-literal code — and the resolution check second. Edge case the implementer must watch: a literal referencing an enum declared later in the same file must resolve under whole-file resolution rather than fail as not-a-literal.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on which diagnostic fires for a literal-position `Enum.Variant` whose head is not an enum or whose variant is unknown (`loom/parse/default-not-literal`/`tool-arg-not-literal` vs `loom/parse/unknown-variant` vs `loom/parse/unknown-identifier`), and on whether the check happens in the parse-time literal gate or a later resolution pass. Diagnostic codes are an observable, version-pinned surface, so the same source produces different conformance-visible output across implementations.
