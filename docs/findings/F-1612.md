## Finding

The H1 of `docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` is `# Unknown reason rule` (no hyphen), but every other reference to the rule — including its own bolded definition site one paragraph down (`**Unknown-reason rule.**`), the four cross-paragraph back-references on the same page (e.g. "per the **Unknown-reason rule** above"), the eight inbound prose citations from sibling topic files (`session-only-degraded-state.md`, `session-shutdown-semantics.md`, `drain-state-contract.md`, `patch-skew-degradation.md`, `version-bump-intro.md`, `version-bump-triggers.md`, `active-invocation-registry.md`, `diagnostic-emission-isolation.md`), the `docs/spec.md` SM-2 cross-link ("[Pi Integration Contract — Unknown-reason rule]"), and the plan leaf titles (`V9h` / `V9h-T` — *Unknown-reason rule*) — spell the term `Unknown-reason rule` with the hyphen. The hyphenated spelling is also the substring downstream tooling reads: §SM-2 links to the rule using the visible text `Unknown-reason rule` as the human-readable anchor.

The pi-integration-contract subdirectory layout was introduced by the size-cap split commit (`f5e89f4`, 2026-06-04) that broke the 576 KB single-file `pi-integration-contract.md` into one index plus thirty children. In the pre-split file the rule existed only as the bolded paragraph opener `**Unknown-reason rule.**`; there was no H1 to disagree with. The split created the child file and authored a fresh H1, and the spacing variant `# Unknown reason rule` was chosen for that one site while every other surface continued to spell the term hyphenated. The defect is therefore not editorial drift across many sites but a single inconsistent site that has not yet been reconciled with the corpus norm it sits inside.

The H2 anchor (`<a id="unknown-reason-rule"></a>`) is unaffected by the H1 spelling — it is an explicit anchor, not auto-derived from the heading text — so all 32 inbound `#unknown-reason-rule` links continue to resolve regardless of how the H1 is written.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `f5e89f4` — *docs: enforce 100KB hard / 30KB recommended size cap on spec set* (2026-06-04, Thomas Andersen)
**History:** The defect entered the corpus in a single commit when `docs/spec_topics/pi-integration-contract.md` (576 KB) was split into a subdirectory of thirty child pages plus an index. `git log --diff-filter=A --follow` on the child file returns only `f5e89f4`, and `git log -L 1,1:…/unknown-reason-rule.md` confirms line 1 (`# Unknown reason rule`) has not been edited since the file was created. In the pre-split file the rule lived only as the bolded paragraph opener `**Unknown-reason rule.**` (with the hyphen); the split authored a fresh H1 for the new child file and chose the unhyphenated spacing variant for that one site, while every other reference on every other page — including the same paragraph's own bolded opener carried over from pre-split — continued to spell the term hyphenated. The defect is therefore an authoring artefact of the split, not corpus-wide drift.

## Solution Space

**Shape:** single

### Recommendation

Rename the H1 in `docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` from `# Unknown reason rule` to `# Unknown-reason rule`, and update the matching child-index bullet in `docs/spec_topics/pi-integration-contract.md` line 11 from `[Unknown reason rule]` to `[Unknown-reason rule]` so the index's visible link text agrees with the renamed page. Leave the `<a id="unknown-reason-rule"></a>` anchor and the filename unchanged — the anchor is explicit (not heading-derived) and every inbound `#unknown-reason-rule` link continues to resolve, and the filename slug is the document's stable identifier across both pre- and post-rename worlds. No prose elsewhere needs to change: every other site already spells the term hyphenated.

## Consequence

**Severity:** cosmetic

Search and human triage are mildly degraded: a reader (or grep) keying on the hyphenated term — the form every other site uses — misses the H1; a reader keying on the unhyphenated heading text misses every prose reference. No runtime behaviour, no anchor resolution, no fixture, and no test contract is affected.
