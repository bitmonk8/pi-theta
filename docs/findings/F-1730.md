## Finding

`InvokeInfraError.cause` partitions input-side validation failures (`"validation"`) from typed-`invoke<T>` return-value AJV failures (`"return_validation"`). The schema declaration in `errors-and-results/queryerror-variants.md` (Invoke variants) carries both members, and every prose site that names the return-side surface uses `"return_validation"`:

- `schema-subset.md` row #5 (`invoke<T>` return value) → `cause: "return_validation"`.
- `hard-ceilings/ceilings-3-and-4.md` row 5 (`invoke<T>` return value) → `cause: "return_validation"`.
- `tool-calls.md` Failures paragraph explicitly contrasts the input-side `.loom`-callable arm (`cause: "validation"`) with the return-side surface (`cause: "return_validation"`).

The only site that disagrees is the *Empty-tail callee compatibility* paragraph in `invocation.md` (Typed return). It worked the same return-side AJV path — "the runtime AJV check rejects the literal `null` against a non-nullable `Schema`" — and surfaces it as `Err(InvokeInfraError { cause: "validation", ... })`. That is the wrong arm: by every other normative site this is `cause: "return_validation"`.

The failure mode is concrete. An author who reads the empty-tail paragraph and writes `match err { InvokeInfraError { cause: "validation", ... } => ... }` will not catch the empty-tail-against-non-nullable rejection at runtime — it surfaces under `"return_validation"` per the schema and the sibling prose. The same partition mistake also blurs the input/return split that `tool-calls.md` deliberately calls out as needing two distinct `match` arms.

## Issue introduction

**Verdict:** later-commit-introduced (omission)
**Introducing commits:** `f7a505e` (2026-06-06, *"pi-loom spec: resolve 'cause validation opposite directions on two error types'"*).
**History:** The *Empty-tail callee compatibility* paragraph entered `spec_topics/invocation.md` in commit `52219c4` (2026-05-06, *"Address all 41 spec-review findings"*), originally with `reason: "validation"`. Commit `9bd29ab` (*"cause vs reason sub-discriminator fields inconsistent across error variants"*) renamed `reason` → `cause` in the same paragraph without touching the value. Commit `f7a505e` then introduced the dedicated `"return_validation"` arm to the `InvokeInfraError.cause` enum in `queryerror-variants.md`, repartitioned `"validation"` to mean input/params-side only, and updated the two sibling tables in `schema-subset.md` row #5 and `hard-ceilings/ceilings-3-and-4.md` row 5 to read `cause: "return_validation"` — but did **not** edit the *Empty-tail callee compatibility* paragraph in `invocation.md`, which describes the same return-side AJV path and continued to read `cause: "validation"`. The defect is the omission in `f7a505e`: the partition-introducing commit touched every other call site on the return-side arm and missed exactly this one.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/invocation.md`, in the *Empty-tail callee compatibility* paragraph under **Typed return**, change the runtime-rejection surface from `Err(InvokeInfraError { cause: "validation", ... })` to `Err(InvokeInfraError { cause: "return_validation", ... })`. No other change is needed: the schema declaration in `queryerror-variants.md` and the sibling rows in `schema-subset.md` / `ceilings-3-and-4.md` / `tool-calls.md` already use the correct arm.

Edge cases for the implementer:

- The neighbouring *Argument arity* paragraph's *Too few arguments* bullet keeps `cause: "validation"` — that is the input/params arm and is correct under the post-`f7a505e` partition. The edit is scoped to the return-side sentence only.
- The `loom/parse/invoke-return-type-mismatch` parse error mentioned in the same paragraph is unrelated and unchanged; it fires before runtime and does not surface through `InvokeInfraError.cause` at all.
- The `V15j` plan leaf already asserts the runtime return-side surface as `cause: "return_validation"`; the edit removes the only spec site that would have caused that assertion to disagree with `invocation.md`.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one mirrors `invocation.md` and surfaces empty-tail-callee-against-non-nullable-`Schema` rejections as `cause: "validation"`; the other mirrors the schema declaration and the sibling tables and surfaces them as `cause: "return_validation"`. Author code that pattern-matches `InvokeInfraError.cause` to separate input from return validation breaks on whichever runtime picked the other arm, and the `V15j` runtime test would red against the `invocation.md`-mirroring implementer.
