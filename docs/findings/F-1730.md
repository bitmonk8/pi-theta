## Problem

`InvokeInfraError.cause` partitions input-side validation failures (`"validation"`) from typed-`invoke<T>` return-value AJV failures (`"return_validation"`). The *Empty-tail callee compatibility* paragraph under **Typed return** in `invocation.md` describes the return-side AJV path (the runtime rejecting literal `null` against a non-nullable `Schema`) but surfaces it as `cause: "validation"`. Every other normative site for this surface — the schema declaration in `errors-and-results/queryerror-variants.md`, `schema-subset.md` row #5, `hard-ceilings/ceilings-3-and-4.md` row 5, and the `tool-calls.md` Failures paragraph — uses `cause: "return_validation"`. An author pattern-matching `InvokeInfraError.cause` to separate input from return validation will mis-handle the empty-tail rejection on whichever arm the runtime did not pick.

## Issue introduction

**Verdict:** later-commit-introduced (omission)
**Introducing commits:** `f7a505e` (2026-06-06, *"pi-loom spec: resolve 'cause validation opposite directions on two error types'"*).
**History:** The *Empty-tail callee compatibility* paragraph entered `spec_topics/invocation.md` in commit `52219c4` (2026-05-06, *"Address all 41 spec-review findings"*), originally with `reason: "validation"`. Commit `9bd29ab` (*"cause vs reason sub-discriminator fields inconsistent across error variants"*) renamed `reason` → `cause` in the same paragraph without touching the value. Commit `f7a505e` then introduced the dedicated `"return_validation"` arm to the `InvokeInfraError.cause` enum in `queryerror-variants.md`, repartitioned `"validation"` to mean input/params-side only, and updated the two sibling tables in `schema-subset.md` row #5 and `hard-ceilings/ceilings-3-and-4.md` row 5 to read `cause: "return_validation"` — but did **not** edit the *Empty-tail callee compatibility* paragraph in `invocation.md`, which describes the same return-side AJV path and continued to read `cause: "validation"`. The defect is the omission in `f7a505e`: the partition-introducing commit touched every other call site on the return-side arm and missed exactly this one.

## Solution approach

In the *Empty-tail callee compatibility* paragraph under **Typed return** in `docs/spec_topics/invocation.md`, change the runtime-rejection surface from `cause: "validation"` to `cause: "return_validation"` so the paragraph agrees with the schema declaration and the sibling sites.

## Solution constraints

- Out of scope: the neighbouring *Argument arity* paragraph's *Too few arguments* bullet, which keeps `cause: "validation"` as the input/params arm.
