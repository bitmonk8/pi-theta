## Finding

`InvokeInfraError.cause` carries two AJV-validation arms that name the same kind of failure with different conventions. The input-side arm — args/params failing the callee's input schema — is `"validation"` (unqualified). The return-side arm — a typed `invoke<T>`'s return value failing AJV — is `"return_validation"` (qualified with the side). Within a single enum partition the two related sub-causes therefore disagree on whether the side is part of the token: one is bare, one is prefixed.

The unqualified `"validation"` token is not arbitrary: it deliberately matches `CodeToolError.cause: "validation"` so that an author handling input-validation failures across both callee kinds (`Pi tool` and `.loom` callable) matches one token on both arms. `tool-calls.md` relies on this explicitly: an author handling every input-validation failure in one place must `match` both `CodeToolError { cause: "validation", ... }` and `InvokeInfraError { cause: "validation", ... }`. So the asymmetry is the price of a cross-variant alignment the spec intends.

The residual cost is intra-variant: a reader scanning the `InvokeInfraError` enum sees `"validation"` next to `"return_validation"` and has no in-place signal that the bare token is specifically the *input* side rather than a catch-all. The inline schema comment notes "(input side, like `CodeToolError.cause "validation"`)", but the asymmetry of qualification is never stated as intentional, so it reads as an oversight. The wire strings are fully pinned, so no implementation diverges; the harm is interpretive friction when authoring exhaustive `match` arms.

## Issue introduction

**Verdict:** later-commit-introduced (omission)
**Introducing commits:** `f7a505e` (2026-06-06, *"pi-loom spec: resolve 'cause validation opposite directions on two error types'"*).
**History:** The *Empty-tail callee compatibility* paragraph entered `spec_topics/invocation.md` in commit `52219c4` (2026-05-06, *"Address all 41 spec-review findings"*), originally with `reason: "validation"`. Commit `9bd29ab` (*"cause vs reason sub-discriminator fields inconsistent across error variants"*) renamed `reason` → `cause` in the same paragraph without touching the value. Commit `f7a505e` then introduced the dedicated `"return_validation"` arm to the `InvokeInfraError.cause` enum in `queryerror-variants.md`, repartitioned `"validation"` to mean input/params-side only, and updated the two sibling tables in `schema-subset.md` row #5 and `hard-ceilings/ceilings-3-and-4.md` row 5 to read `cause: "return_validation"` — but did **not** edit the *Empty-tail callee compatibility* paragraph in `invocation.md`, which describes the same return-side AJV path and continued to read `cause: "validation"`. The defect is the omission in `f7a505e`: the partition-introducing commit touched every other call site on the return-side arm and missed exactly this one.

## Solution Space

### Recommendation

Leave the `InvokeInfraError.cause` wire tokens unchanged: `cause: "validation"` for the input/params-side AJV failure and `cause: "return_validation"` for the return-side AJV failure. In `queryerror-variants.md` *Invoke variants* (the `InvokeInfraError` schema, `#invoke-variants`), add one explicit sentence adjacent to the schema stating that the qualification asymmetry is intentional and covering both arms: the bare `"validation"` token is the input side, named without a side qualifier so it matches `CodeToolError.cause "validation"` and lets an author handle input-validation failures across both callee kinds (`Pi tool` and `.loom` callable) with a single `match` token; the return side carries the `"return_validation"` qualifier to disambiguate it within the same enum. Word the sentence so it stands on its own rather than merely repeating the existing inline schema comment. Make no token changes and no edits in any other file.

## Consequence

**Severity:** advisory

The wire discriminator strings are fixed by the schema declaration, so no implementer diverges on behaviour. The asymmetric qualification imposes a small interpretive burden: an author writing exhaustive `match` arms must learn that the bare `"validation"` token is the input side while only the return side is qualified, and the spec never states that the asymmetry is intentional.
