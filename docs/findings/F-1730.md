## Problem

`InvokeInfraError.cause` carries two AJV-validation arms named with different conventions: the input-side arm (args/params failing the callee's input schema) is the unqualified `"validation"`, while the return-side arm (a typed `invoke<T>`'s return value failing AJV) is the qualified `"return_validation"`. The unqualified `"validation"` is intentional — it matches `CodeToolError.cause "validation"` so an author can handle input-validation failures across both callee kinds with a single `match` token. But the spec never states that the qualification asymmetry is deliberate, so a reader scanning the enum reads the bare token as an oversight rather than as the price of a cross-variant alignment.

## Issue introduction

**Verdict:** later-commit-introduced (omission)
**Introducing commits:** `f7a505e` (2026-06-06, *"pi-loom spec: resolve 'cause validation opposite directions on two error types'"*).
**History:** The *Empty-tail callee compatibility* paragraph entered `spec_topics/invocation.md` in commit `52219c4` (2026-05-06, *"Address all 41 spec-review findings"*), originally with `reason: "validation"`. Commit `9bd29ab` (*"cause vs reason sub-discriminator fields inconsistent across error variants"*) renamed `reason` → `cause` in the same paragraph without touching the value. Commit `f7a505e` then introduced the dedicated `"return_validation"` arm to the `InvokeInfraError.cause` enum in `queryerror-variants.md`, repartitioned `"validation"` to mean input/params-side only, and updated the two sibling tables in `schema-subset.md` row #5 and `hard-ceilings/ceilings-3-and-4.md` row 5 to read `cause: "return_validation"` — but did **not** edit the *Empty-tail callee compatibility* paragraph in `invocation.md`, which describes the same return-side AJV path and continued to read `cause: "validation"`. The defect is the omission in `f7a505e`: the partition-introducing commit touched every other call site on the return-side arm and missed exactly this one.

## Solution approach

Clarify, adjacent to the `InvokeInfraError` schema at `#invoke-variants` in `queryerror-variants.md`, that the `cause` qualification asymmetry is intentional: the bare `"validation"` token is the input side, left unqualified so it matches `CodeToolError.cause "validation"` for cross-callee-kind input-validation handling, while `"return_validation"` qualifies the return side to disambiguate it within the same enum. Make the statement stand on its own rather than relying on the existing inline schema comment.

## Solution constraints

- Do not change the `InvokeInfraError.cause` wire tokens (`"validation"`, `"return_validation"`) — they are fixed wire contract and `"validation"` deliberately aligns with `CodeToolError.cause "validation"`.
- Out of scope: the corpus-wide validation-cause token decision owned by F-2205; edit only the `#invoke-variants` schema region.
