## Finding

QRY-20 (*Observability of discarded results*, `query-escapes-stringification.md`) specifies that a discarded query's failure event preserves the failing query's `kind` / `code` / `message` plus its source location "carried in the `RuntimeEvent` `discard_site` field", and defines the per-form population rule it owns: the location of the discarding `let _ =` binding for the expression-statement form, and the start of the tail `@`-expression for the void-tail-function form. The field itself — its wire type `{ file: string; line: number; column: number }`, its presence rule (present only on discarded-query events; omitted, not `null`, elsewhere), and its relationship to `query_site` and the `(kind, query_site, message, occurred_at)` dedup tuple — is declared and owned by the Pi Integration Contract's Runtime event channel sub-page (`pi-integration-contract/runtime-event-channel.md`).

The gap is navigational: no `RuntimeEvent` shape is exhibited anywhere in the query cluster, and QRY-20's only pointer to the owning declaration is the link text "Pi Integration Contract — Runtime event channel" whose href targets the PIC **index aggregator** (`../pi-integration-contract.md`), not the `runtime-event-channel.md` sub-page that actually declares the field. The single `RuntimeEvent` payload rendered in the shard — the depth-6 worked example in `query-tool-loop.md` — carries `query_site`, never `discard_site` (it is a `validation` event, not a discard). An implementer reading QRY-20 must therefore land on the index page and traverse a further hop to recover `discard_site`'s type and its distinction from `query_site`.

The coarse link is an inconsistency rather than a deliberate convention: the same shard already cites the sub-page with a precise anchor elsewhere (`query-tool-loop.md` references `../pi-integration-contract/runtime-event-channel.md#pic-1`). The parallel "Runtime event channel" link in `query-tool-loop.md`'s worked-example payload paragraph (`../pi-integration-contract.md`) carries the same imprecision.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `6afca020c5f8f35e25d2cf69c4a4cf989ba62de4` (2026-06-03, "pi-loom spec: resolve \"discard-site field double-specified on RuntimeEvent\"") and `f5e89f4fb8f7e3816d3f5aacb2750c27d34dbc1d` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** `git log -S 'discard_site'` shows the field and the inbound "Runtime event channel" cross-reference were authored together in `6afca02`, which split the discard-site location out of the mode-overloaded `query_site` slot into a distinct `discard_site` field on the then-monolithic `pi-integration-contract.md` and pointed `query.md`'s link at that same page. The link was accurate at that point — the page contained the `RuntimeEvent` declaration. The size-cap split `f5e89f4` extracted the Runtime event channel into the `pi-integration-contract/runtime-event-channel.md` sub-page and repointed most inbound links to the new sub-page anchors (e.g. `unknown-reason-rule`, `diagnostic-emission-isolation`, `capability-probe`, `binder-inference#success-side-null-policy`) but left the QRY-20 (and the `query-tool-loop.md` payload) "Runtime event channel" link aimed at the now index-only aggregator, stranding the `discard_site` declaration one hop away. The "no in-shard `RuntimeEvent` shape exhibits `discard_site`" aspect predates the split (present since `6afca02`) but was benign while the link still resolved to the shape. The durable fix therefore addresses the cross-reference precision left stale by the split, not a revert of either commit.

## Solution Space

**Shape:** single

### Recommendation

Repoint QRY-20's "Pi Integration Contract — Runtime event channel" link from the index aggregator (`../pi-integration-contract.md`) to the sub-page that declares the `RuntimeEvent` shape, `../pi-integration-contract/runtime-event-channel.md` (anchored on its `#runtime-event-channel` heading), matching the precise sub-page-anchor citation style already used at `query-tool-loop.md` (`../pi-integration-contract/runtime-event-channel.md#pic-1`). Apply the same repoint to the parallel "Runtime event channel" link in `query-tool-loop.md`'s worked-example payload paragraph. Optionally append a one-clause note at the `discard_site` mention that it shares `query_site`'s `{ file, line, column }` shape and is present only on discarded-query events, so the type is recoverable without the hop.

QRY-20 retains only the per-form population rule it already owns (which form stamps which location); do not restate the field's presence or dedup-tuple rules in the query shard — those stay owned by the PIC sub-page.

Edge case: the `RuntimeEvent` declaration block in `runtime-event-channel.md` currently has no dedicated `<a id>` at the code block. Point at the existing `#runtime-event-channel` section heading anchor; for a more durable target, add an explicit anchor at the `RuntimeEvent` type declaration and cite that.

## Consequence

**Severity:** advisory

An implementer can fully recover `discard_site`'s type, presence rule, and dedup relationship from the PIC sub-page, so no behavioural divergence results. The cost is navigation friction: QRY-20's link lands on the PIC index rather than the field declaration, so the field's shape and its distinction from `query_site` require an extra hop to confirm.
