## Finding

The *Custom-message channel persistence and LLM-context entry* paragraph in `runtime-event-channel.md` closes with: "a future Pi minor that adds such a field would be picked up by the `pi-version-bump-procedure` re-validation gate, at which point the runtime SHOULD set the opt-out on the canonical `sendSystemNote` call." This embeds a forward-phase behavioural obligation — keyed on a `CustomMessage.excludeFromContext` field that does not exist at the loom 1.0 Pi-SDK pin — inside an otherwise descriptive/normative paragraph, with no future-phase label demarcating it from the loom 1.0 contract around it.

Two problems compound. First, the modal is a bare `SHOULD` attached to an action the runtime cannot take at loom 1.0 (there is no field to set), so a reader cannot tell whether it is a loom 1.0 obligation, a recommendation, or a deferred future-phase action. Second, the durable home for the re-validation already exists: the paragraph itself forward-links [version-bump editorial-review checklist item (r)](version-bump-step2.md#bump-checklist-convert-to-llm-context-entry), which owns the per-bump confirmation that `CustomMessage` still exposes no context-exclusion opt-out. Item (r) covers detecting the field's appearance but does **not** carry the consequent "set the opt-out" action; that action lives only here, inlined as a `SHOULD`. The result is a forward-phase obligation orphaned from the procedure that should own it, stated with a modal that misrepresents its phase.

The loom 1.0 fact the paragraph needs to assert is narrow and unambiguous: at the pinned SDK there is no `CustomMessage.excludeFromContext` opt-out, so every `loom-system-note` enters the provider context durably. That fact is descriptive (or a non-normative operator note), not a `SHOULD`; the conditional future action belongs with the version-bump re-audit it references.

## Solution Space

**Shape:** single

### Recommendation

In `runtime-event-channel.md`, replace the trailing future-phase clause with a phase-neutral statement of the loom 1.0 fact only: at the pinned Pi-SDK there is no `CustomMessage.excludeFromContext` opt-out (analogous to `BashExecutionMessage.excludeFromContext`), so every `loom-system-note` — including `display: false` notes — enters the provider context durably. Express this as plain descriptive prose (or fold it into the existing non-normative operator note), not a `SHOULD`. Keep the cross-reference to version-bump checklist item (r).

Move the conditional future action to checklist item (r) in `version-bump-step2.md`, which already owns the per-bump detection that no such field exists: extend item (r) so that, on a bump where the candidate Pi minor *does* introduce a `CustomMessage` context-exclusion opt-out, the consequent action — adding the opt-out to the canonical `sendSystemNote` call — is recorded there as the follow-on obligation. That keeps detection and consequent action co-located in the procedure that fires on the triggering event, and removes the dangling future-phase modal from the descriptive paragraph.

Edge cases for the fixer: do not delete the no-opt-out fact itself (it backs the durable-context observability claim and item (r)'s "no-opt-out claim is keyed on that analogy"); preserve the `BashExecutionMessage.excludeFromContext` analogy referent and its `dist/core/messages.d.ts` declaration pin; and keep the existing item (r) re-audit forward-link intact in the paragraph.

## Consequence

**Severity:** advisory

An implementer or auditor reading the paragraph cannot tell that the `SHOULD set the opt-out` clause is a deferred future-phase action rather than a loom 1.0 obligation, and the action is divorced from the version-bump checklist that should own it. No working loom 1.0 system is precluded (there is no field to set), but the misplaced modal degrades the page's normative/descriptive discipline and risks the forward action being lost when the field eventually appears.
