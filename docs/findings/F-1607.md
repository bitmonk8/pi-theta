## Finding

`docs/spec_topics/binder/defaulting-system-note-echo.md` rule 2 ("Length cap.")
describes the optional grapheme-cluster back-off in two consecutive sentences
that contradict each other on the rule's normative status. The first sentence
says: *"…an implementation **MAY** back the truncation point off to the nearest
preceding extended grapheme cluster boundary, yielding a resulting note of ≤120
code points with the trailing `…` still appended."* The next sentence says:
*"This back-off is **non-normative** and tests MUST NOT assert cluster-aware
behaviour."*

`MAY` (RFC 2119 / BCP 14) is itself a normative keyword — it grants a normative
permission. A clause cannot simultaneously use `MAY` to confer that permission
and label the same behaviour "non-normative". The contradiction has a concrete
downstream cost: a conformance-test author cannot decide whether the back-off
is part of the spec surface (and the `MUST NOT assert cluster-aware behaviour`
side-constraint is therefore an enforceable obligation) or merely an editorial
illustration (and the `MUST NOT` is also editorial, not enforceable). The
authoring sub-clause and the conformance sub-clause cannot both be
non-normative if one of them contains a `MUST NOT`.

## Solution Space

**Shape:** multiple

### Option A — Drop "non-normative"; keep the `MAY` permission and the test-side `MUST NOT`

- **Approach:** Treat the back-off as what it grammatically is — a normative
  optional permission — and reframe the conformance-test sub-clause as a
  parallel normative obligation, not as a footnote to a "non-normative"
  qualifier.
- **Spec edits:** In rule 2, replace the sentence *"This back-off is
  non-normative and tests MUST NOT assert cluster-aware behaviour."* with
  *"Conformance tests MUST NOT assert cluster-aware behaviour: the back-off
  is implementation-optional, and an implementation that always truncates at
  the scalar boundary remains conformant."* Leave the `MAY` sentence
  unchanged. No edits elsewhere.
- **Pros:**
  - Keeps the `MAY` consistent with how the rest of the spec uses RFC 2119
    keywords (e.g. rule 3's *"Renderers MAY style the prefix distinctly"*).
  - Preserves the `MUST NOT` against testing cluster-aware behaviour as an
    enforceable obligation on conformance suites — which is the load-bearing
    half of the original intent.
  - One-sentence edit; no ripple to other shards or plan leaves.
- **Cons:** None of substance; the change is editorial.
- **Risks:** None.

### Option B — Demote the whole back-off sub-clause to a non-normative implementation note; drop the `MAY` and the `MUST NOT`

- **Approach:** Take the original "non-normative" label at face value and
  pull the entire back-off paragraph out of the normative rule body into a
  clearly-marked non-normative aside. Conformance-test authors are then
  governed only by the surrounding exactly-120 requirement.
- **Spec edits:** In rule 2, delete both sentences (the `MAY` sentence and
  the `non-normative / MUST NOT` sentence). Immediately after rule 2, add an
  italicised aside: *"Implementation note (non-normative). An implementation
  may, as a rendering courtesy, back the truncation point off to the nearest
  preceding extended grapheme cluster boundary so long as the resulting note
  is ≤120 code points and the trailing `…` is still appended. The rule above
  permits but does not require this; an implementation that always
  truncates at the scalar boundary is conformant."*
- **Pros:**
  - Eliminates the `MAY`/`non-normative` collision by removing both
    keywords from the same paragraph.
  - Cleanly separates the courtesy behaviour from the normative cap rule.
- **Cons:**
  - Drops the `MUST NOT assert cluster-aware behaviour` obligation, which
    was the original clause's load-bearing constraint on conformance suites.
    A test suite that happened to assert cluster-aware behaviour would now
    be merely under-specified rather than non-conformant.
  - Adds a new paragraph to the section, mildly worsening the rule body's
    density.
- **Risks:** A future spec editor could re-introduce a cluster-aware
  conformance test on the strength of the now-silent spec.

### Recommendation

Option A. The original wording's only durable obligation is the conformance
side-constraint — *do not write tests that observe cluster-aware behaviour* —
and that obligation is unambiguously normative. Option B silently weakens it,
so the cost of Option B exceeds the cost of Option A even though both fix the
surface contradiction. Implementer edge case to watch when applying Option A:
the rewritten conformance sentence must remain inside rule 2's body (it
qualifies the cap rule); do not promote it to its own numbered rule, which
would re-open the precedence question `c15da5c` just closed.

## Consequence

**Severity:** advisory

Two careful readers can still arrive at the same user-visible behaviour — the
back-off is optional in either reading, and the test prohibition is honoured
out of caution — so no behavioural divergence is likely. The cost is
implementer/conformance-author friction: time spent re-resolving the
contradiction on every encounter, and a real risk that a future spec editor
"resolves" it the wrong way (e.g. drops the `MUST NOT` because the surrounding
clause is "non-normative").
