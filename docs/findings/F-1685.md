## Finding

The `session_shutdown` handler's *all-three-throw corner case* — sub-step 1's `LoomRegistry.drain`, sub-step 1's `LoomRegistry.initDrainStateTag`, and the post-sub-step-4 `LoomRegistry.markRuntimeDegraded` all throwing and being swallowed on a single session-only `session_shutdown`, leaving the registry at `(drained: false, tag: undefined)` — is fully described in prose across two pages. `drain-state-contract.md`'s *Slash-dispatch interaction* clause states that a slash invocation arriving before the next `session_shutdown` observes the steady-state tuple at the `readDrainState` site and **routes through arm (a) — dispatching the loom into the half-torn-down runtime, emitting neither the shutting-down nor the degraded system note**, and PIC-30 forbids adding a fourth arm or alternative gating field that would suppress that dispatch. `session-only-degraded-state.md`'s *Idempotent re-entry and the all-three-throw corner case* clause states that a follow-up session-only `session_shutdown` re-runs the full handler and emits an **additional `loom/host/session-shutdown-runtime-degraded` row**, breaching the at-most-once-per-instance bound for this case only.

Both of these are observable, loom-1.0-accepted outcomes, yet neither acceptance-criteria set pins them. The `drain-state-contract.md` acceptance block covers only the `(drained: true, tag: undefined)` → arm (b) path. The `session-only-degraded-state.md` acceptance criteria (a)–(m) cover every other branch — clean session-only, single-site throws, always-tear-down reasons, unknown/snapshot-failure paths, recovery, idempotent short-circuited re-entry, the two read-failure call sites — but no fixture exercises the all-three-throw tuple. Worse, fixture (b) explicitly defers multi-site failure ("Multiple simultaneously-failing sites are out of scope for (b) (covered only by the all-three-throw corner case)"), naming the corner case as the coverage vehicle that is never actually instantiated.

The leaf owns fixture-shape and the acceptance criteria are the normative fixture set, so a test author working from these criteria writes no fixture for either the arm-(a) silent-dispatch outcome or the re-entry duplicate-emission outcome. Both are rare (three independent swallowed throws on one event) and silent (a dispatch into a runtime whose watchers, listeners, and in-flight aborts are already torn down), so the gap is unlikely to be caught incidentally.

## Solution Space

**Shape:** single

### Recommendation

Add one acceptance fixture for the all-three-throw corner case covering both observable outcomes, in the same `session-only-degraded-state.md` acceptance-criteria block as (a)–(m) (the section that already owns the corner case's prose and is the natural home for the re-entry assertion):

- **Reachability + arm-(a) dispatch.** Drive a session-only `session_shutdown` (`reason ∈ {"new","resume","fork"}`) with `LoomRegistry.drain`, `LoomRegistry.initDrainStateTag`, and the post-sub-step-4 `LoomRegistry.markRuntimeDegraded` all stubbed to throw (each caught by per-step / interstitial-seam isolation). Assert the post-handler snapshot is `(drained: false, tag: undefined)` and that one `teardown-step-failed` row is emitted per throwing sub-step-1 site (`details.call` = `loomRegistry.drain`, `loomRegistry.initDrainStateTag`; the `markRuntimeDegraded` seam swallow emits none). Then assert a slash invocation arriving before the next `session_shutdown` routes through arm (a) — the loom **is dispatched**, and **neither** the `"…extension shutting down"` nor the `"…extension degraded; /reload to recover"` system note is returned — witnessing PIC-30's no-suppression prohibition.
- **Re-entry duplicate emission.** Assert that a follow-up session-only `session_shutdown` (with a clean follow-up snapshot read) is **not** short-circuited (the predicate `snapshot.drained === true || snapshot.tag !== undefined` is false on `(false, undefined)`), re-runs the full handler, and emits an **additional** `loom/host/session-shutdown-runtime-degraded` row (plus any `teardown-step-failed` row from sub-step 4's double-close on the already-closed watchers), confirming the at-most-once bound is intentionally breached for this case and no suppression seam is added.

Reference PIC-30 from the fixture so the prohibition has a witnessing criterion. Keep PIC-30 as the structural prohibition it is; the gap is the absent fixture, not the rule.

## Consequence

**Severity:** advisory

The behaviour is unambiguously specified in prose, so an implementer can build it correctly; the gap is a missing conformance fixture, not a correctness ambiguity. Without it, a regression that adds a suppression gate on the post-handler tuple (the very thing PIC-30 forbids), or one that fails to re-emit on re-entry, ships unverified — the corner case is silent and rare enough that nothing else exercises it.
