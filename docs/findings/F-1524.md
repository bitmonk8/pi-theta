## Finding

The `context.tools` entry the binder builds is specified twice with two incompatible shapes. The opening paragraph of `binder-inference.md` pins `Context.tools` to `Tool[]` from `@earendil-works/pi-ai`'s `dist/types.d.ts`, where `Tool` declares exactly three required fields (`name`, `description`, `parameters`) and no `label` or `execute`. The third bullet of the `complete(model, context, options)` enumeration then describes "Its three remaining required `ToolDefinition` fields" — `ToolDefinition` is the **pi-coding-agent** type with five required fields (`name`, `label`, `description`, `parameters`, `execute`) — and pins `label` to the literal `"Loom binder envelope"`. Three problems follow:

1. The bullet uses `ToolDefinition` to count fields ("three remaining"), but `ToolDefinition` has four remaining required fields after `parameters` (`name`, `label`, `description`, `execute`); the count silently drops `execute`.
2. `label` is asserted as a fixed literal on a value the spec has already typed as `Tool`, which has no `label` member at all.
3. The relationship between pi-ai's `Tool` and pi-coding-agent's `ToolDefinition` — same shape, subset, or convert via a bridge — is never stated, even though the synthesised typed-query respond tool the bullet cross-references is a `ToolDefinition` (it ships through `pi.registerTool` / `customTools`) while the binder entry is a `Tool` (it ships through pi-ai's `complete()` `Context.tools` parameter).

An implementer following the bullet literally either constructs a `ToolDefinition` (extra `label` field rejected at the `Context.tools` boundary's structural type, plus a never-invoked `execute` they had to invent) or constructs a `Tool` (in which case the pinned `label` literal is unrepresentable). Either way the spec contradicts itself.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `61592d7` (2026-06-05, "pi-loom spec: resolve 'Binder structured-output tool name and label undefined'")

**History:** Before `61592d7` the `context.tools` bullet pinned only `parameters` (wrapped via `Type.Unsafe<unknown>(envelopeSchema)`) and named no further fields. The commit, resolving a prior review finding that the binder tool's `name`/`label` were undefined, added the "three remaining required `ToolDefinition` fields" clause and the literal pins for `name` (`__loom_bind_<slug>`), `label` (`"Loom binder envelope"`), and `description`. The fix reached for the wrong type: the synthesised typed-query respond tool whose pin it mirrors is a `ToolDefinition` (registered via `pi.registerTool`), but `Context.tools` is `Tool[]` — the opening paragraph of the same file had already pinned that typing two commits earlier under `f5e89f4` (2026-06-04, the spec-split commit that created the file). A follow-up commit `e63ba6c` (2026-06-05) edited the same bullet to add the `complete()` forced-tool presupposition cross-reference but did not touch the type framing. The defect has lived in its current form since `61592d7` and has not been re-edited since.

## Solution Space

**Shape:** single

### Recommendation

Re-anchor the bullet to pi-ai's `Tool` (the type `Context.tools` already requires) and stop borrowing the pi-coding-agent `ToolDefinition` shape. The fix:

1. Replace "Its three remaining required `ToolDefinition` fields are fixed literals" with wording that explicitly identifies the type as `@earendil-works/pi-ai`'s `Tool` (declared at `dist/types.d.ts`, already pinned by the opening paragraph of this file), and enumerates the two remaining required `Tool` fields after `parameters` — `name` and `description` — as fixed literals. The `name` derivation (`__loom_bind_<slug>`) and the `description` literal (`"Return the binder result envelope for the slash-command argument binding."`) keep their current values.
2. Drop the `label` literal `"Loom binder envelope"`. `Tool` has no `label` member; the typed-query respond tool's `label` precedent does not transfer because that tool is a `ToolDefinition` reaching the model through `pi.registerTool`, whereas the binder entry reaches the provider through `complete()`'s `Context.tools` parameter and never traverses Pi's tool registry or UI layer.
3. Add a one-sentence reconciliation note stating that pi-ai's `Tool` and pi-coding-agent's `ToolDefinition` are distinct types with distinct field sets — `ToolDefinition` extends the shape with `label` and `execute` (among others) because it is registered with Pi's UI/dispatch surface, while `Tool` is the minimal contract pi-ai's provider adapters consume — so the binder entry carries neither `label` nor `execute`, and no conversion bridge is required at the binder site. The cross-reference to the **Per-loom registration** field enumeration above should be retained for the `parameters` wrap precedent only, not for the field-set precedent.

Implementer-relevant edge cases: the synthesised typed-query forced respond turn (`conversation-drive.md`) also routes through `complete()` but is documented as following the full `name`/`label`/`description`/`parameters`/`execute` shape — confirm whether that turn's tool reaches `complete()` via `Context.tools` (in which case the same `Tool`-shape correction applies there) or whether it is constructed as a `ToolDefinition` and the runtime extracts a `Tool`-shaped projection for `Context.tools`. Either way is consistent with the binder's `Tool`-shape pin; the reconciliation note should be written to cover both possibilities or the typed-query path should be tightened in the same edit.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge. One materialises the binder entry as a pi-coding-agent `ToolDefinition` with a no-op `execute` and the `"Loom binder envelope"` `label`; another, honouring the opening paragraph's `Tool[]` typing, materialises a pi-ai `Tool` and silently drops `label`. The forced-tool envelope reaches the provider in both shapes, so the divergence is invisible at run time but breaks the `V9j-T` "the call is issued with the forced-tool envelope" assertion the moment it asserts the entry's field set.
