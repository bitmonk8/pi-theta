## Problem

The `context.tools` entry the binder builds is specified with two incompatible type shapes. The opening paragraph of `binder-inference.md` pins `Context.tools` to `Tool[]` from `@earendil-works/pi-ai`'s `dist/types.d.ts` (three required fields: `name`, `description`, `parameters`; no `label` or `execute`), but the `context.tools` bullet describes "three remaining required `ToolDefinition` fields" — `ToolDefinition` being the pi-coding-agent type with five required fields — and pins `label` to the literal `"Loom binder envelope"`. The field count is wrong (`ToolDefinition` has four remaining required fields after `parameters`, silently dropping `execute`), and `label` is asserted on a value already typed as `Tool`, which has no `label` member. An implementer cannot satisfy both the opening-paragraph `Tool[]` typing and the bullet's `ToolDefinition`-shaped pins.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `61592d7` (2026-06-05, "pi-loom spec: resolve 'Binder structured-output tool name and label undefined'")

**History:** Before `61592d7` the `context.tools` bullet pinned only `parameters` (wrapped via `Type.Unsafe<unknown>(envelopeSchema)`) and named no further fields. The commit, resolving a prior review finding that the binder tool's `name`/`label` were undefined, added the "three remaining required `ToolDefinition` fields" clause and the literal pins for `name` (`__loom_bind_<slug>`), `label` (`"Loom binder envelope"`), and `description`. The fix reached for the wrong type: the synthesised typed-query respond tool whose pin it mirrors is a `ToolDefinition` (registered via `pi.registerTool`), but `Context.tools` is `Tool[]` — the opening paragraph of the same file had already pinned that typing two commits earlier under `f5e89f4` (2026-06-04, the spec-split commit that created the file). A follow-up commit `e63ba6c` (2026-06-05) edited the same bullet to add the `complete()` forced-tool presupposition cross-reference but did not touch the type framing. The defect has lived in its current form since `61592d7` and has not been re-edited since.

## Solution approach

Re-anchor the `context.tools` bullet to pi-ai's `Tool` — the type the opening paragraph already requires — and stop borrowing the `ToolDefinition` shape. Rewrite the field clause to enumerate the two remaining required `Tool` fields after `parameters` (`name` and `description`) as fixed literals, keeping the existing `name` derivation (`__loom_bind_<slug>`) and `description` literal; drop the `label` literal `"Loom binder envelope"`, since `Tool` has no `label` member. Add a reconciliation statement distinguishing pi-ai's `Tool` from pi-coding-agent's `ToolDefinition` so a reader understands the binder entry carries neither `label` nor `execute` and needs no conversion bridge.
