## Finding

`V15c` introduces the `Resolver` seam for relative `.warp` resolution. Its `IMP-1` behaviour defines an unresolvable path as "non-relative, **no byte-exact final-segment entry**, or **unreadable**" — directory enumeration plus file reads, i.e. the `FileSystem.readdir` and `FileSystem.readText`/`readBytes` members. That `FileSystem` seam is owned exclusively by `V8b`, but `V15c`'s `Deps` are `V15c-T`, `V1a`, `V15a`, which do not name `V8b`. `V15c-T` likewise lists only `V1a`, `V15a`.

`V8b` is reachable transitively — `V1a`'s `Deps` name `V8b` directly, and `V15a → V10a → V8b` — so the leaf is still pickable and `V8b` is guaranteed complete before `V15c` runs. The gap is one of declaration completeness: a leaf that directly consumes a Class-2 seam should declare a direct `Deps` edge to the owning leaf, so the "reorder freely as long as the deps DAG is respected" guarantee in `conventions.md` does not silently rest on an unrelated leaf's edge surviving. If `V1a` or `V10a` were ever re-pointed off `V8b`, `V15c`'s seam dependency would vanish from the DAG with no local signal.

**Verdict:** present-since-inception

**History:** The plan corpus is git-tracked. `docs/plan_topics/V15c-imports.md` was created in `c6a664e` already carrying the `Resolver` seam (byte-exact final-segment + unreadable resolution, i.e. the `FileSystem` consumption) and the `Deps. V15c-T, V1a, V15a` line without `V8b`. The missing `V8b` edge has therefore been present since the leaf first entered the corpus, not introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

Add `V8b` to both leaves' `Deps` fields.

- In `docs/plan_topics/V15c-imports.md`, the `Deps.` field currently reads `` `V15c-T`, `V1a`, `V15a` ``; add `` `V8b` `` so the consumer of `FileSystem.readdir`/`readText` declares the seam owner directly.
- In `docs/plan_topics/V15c-T-imports.md`, the `Deps.` field currently reads `` `V1a`, `V15a` ``; add `` `V8b` `` to match.

This mirrors how `V1a` and `V10a` already declare `V8b` directly as `FileSystem` consumers. No coverage-matrix row changes (IMP-1 → V15c is unaffected) and no spec edit is required; the fix is internal to the two leaf files' `Deps` fields.

## Consequence

**Severity:** advisory

The leaf is buildable today because `V8b` is transitively guaranteed via `V1a`. The undeclared direct seam edge means `V15c`'s reliance on the `FileSystem` seam is invisible to anyone reading `V15c`'s `Deps` in isolation, and the DAG's reorder-freely guarantee for this consumer depends on an unrelated leaf's edge rather than a declared one.
