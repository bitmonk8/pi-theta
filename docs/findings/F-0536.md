## Finding

`V9j` owns the binder `complete()` inference call. Per [`binder-inference.md`](../../../docs/spec_topics/pi-integration-contract/binder-inference.md), that single call populates `context.tools` with the **binder envelope schema** and `context.systemPrompt` with the **rendered binder system prompt** ‚Äî so the call site is the point where the binder context is assembled. The envelope schema is produced by `V11c` (`Adds`: "the dynamic envelope schema ‚Ä¶ consumed by the inference call"); the system-prompt builder is produced by `V11d`. `V9j`'s `Deps` are `V9j-T, V4d, V11a` ‚Äî neither `V11c` nor `V11d` appears, and no separate "assemble the binder context" leaf is identified.

Because build order is dep-driven (How-to-use step 3), nothing forces `V11c`/`V11d` to land before `V9j`: `V9j` becomes pickable once `V11a`/`V4d` complete, which can happen before either producer exists. Its `Tests` then require `complete()` to be "issued with the forced-tool envelope" ‚Äî an assertion that cannot exercise `V11c`'s real schema or `V11d`'s real prompt, only an invented stub. The only leaf whose `Deps` lists `V11c` is the terminal `H7a` integration gate, so in the unit build order the envelope-schema producer and its consuming call site are first connected at the cross-slice gate, not at any unit leaf.

The result is a producer‚Üíconsumer seam with no dependency edge: the implementer of `V9j` must fabricate an envelope/prompt shape to satisfy the `complete()` test, and a divergence between that stub and `V11c`/`V11d`'s eventual real shapes surfaces only at `H7a`.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** `git log -L` over `V9j`'s `Deps` line shows it was authored as `V9j-T, V4d, V11a` at file creation in `c6a664e` and never amended; `V11c` and `V11d` were created in the same commit. The missing producer‚Üíconsumer edge has existed since the leaf-decomposed plan was first authored ‚Äî no subsequent fix commit (the plan corpus has had many) touched `V9j`'s `Deps`.

## Solution Space

**Shape:** multiple

### Option A ‚Äî Wire the producers into V9j's Deps

**Approach:** Treat `V9j` as the assembly point it already is (it owns the `complete()` call). Add `V11c` (envelope schema) and `V11d` (system-prompt builder) to `V9j`'s `Deps`, and in `V9j`'s `Adds` name the two consumed seams ‚Äî the envelope schema from `V11c` attached to `context.tools`, and the system prompt from `V11d` placed on `context.systemPrompt`. Mirror the added edges on the paired `V9j-T` tests task.

**Plan edits:** `V9j` `Deps` gains `V11c`, `V11d`; `V9j-T` `Deps` gains `V11c`, `V11d`; `V9j` `Adds` cites the two consumed seams. Add a V9/V11 interleave note (or extend the existing one) recording that `V11c`/`V11d` land before `V9j`.

**Spec edits:** None.

**Pros:** Smallest diff; keeps a single call-site owner per `binder-inference.md`; no new leaf ID to allocate.

**Cons:** Grows `V9j`'s dependency fan-in; the interleave story for the V9/V11 seam gets one more entry.

**Risks:** Resequencing `V11c`/`V11d` before `V9j` ‚Äî confirm no cycle: `V11c` `Deps` are `V5d`/`V5f`/`V8c`, `V11d` `Deps` are `V11a`/`V2a`/`V2d`; neither reaches `V9j`, so the edge is acyclic.

### Option B ‚Äî Introduce a dedicated context-assembly leaf

**Approach:** Add a new leaf (`<new>`) that depends on `V11c` and `V11d` (and `V11a`), assembles `context` (tools + systemPrompt + model), and exposes it as the seam `V9j` consumes; add `<new>` to `V9j`'s `Deps`.

**Plan edits:** Create `<new>` with `Deps` `V11a`/`V11c`/`V11d`; `V9j` `Deps` gains `<new>`; coverage-matrix authoring for any REQ-ID the new leaf would close.

**Spec edits:** None.

**Pros:** Isolates assembly from the provider-error-mapping concern `V9j` also carries.

**Cons:** Fragments a single spec-defined call site (`binder-inference.md` assigns assembly to the `complete()` call) across two leaves; requires allocating a new leaf ID and re-deriving its coverage role.

**Risks:** A new leaf that closes no REQ-ID adds a node the transitive-completeness and coverage machinery must account for.

### Recommendation

Take Option A: add `V11c` and `V11d` to both `V9j` and `V9j-T` `Deps`, and name the two consumed seams in `V9j`'s `Adds`. `binder-inference.md` assigns `context.systemPrompt` and `context.tools` population to the `complete()` call itself, so `V9j` already owns assembly; a separate leaf would split one spec-defined call site. Watch the resequencing: `V11c`/`V11d` must land before `V9j`, and the V9/V11 interleave note should record that seam so the build order is read off `Deps`, not slice number.

## Consequence

**Severity:** correctness

`V9j` is pickable as soon as `V11a`/`V4d` complete, yet its `complete()` test consumes `V11c`'s envelope schema and `V11d`'s system prompt, which may not exist at that point. Two reasonable implementers diverge ‚Äî one stubs the envelope/prompt shape, another stalls ‚Äî and a stub that does not match `V11c`/`V11d`'s eventual real shapes is only caught at the `H7a` integration gate, forcing rework across the binder consumers (`V11e`, `V11f`, `V13d`) that build on `V9j`.
