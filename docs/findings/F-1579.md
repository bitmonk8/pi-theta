## Finding

The *Pi SDK pin* prerequisite (item 1) explains where loom's four-package lock-step comes from: "The lock-step minor-version line is inherited from `@earendil-works/pi-coding-agent`'s own `dependencies` block, which pins `@earendil-works/pi-agent-core`, `@earendil-works/pi-ai`, and `@earendil-works/pi-tui` at the same `~X.Y.Z` minor-version line." The tilde (`~X.Y.Z`) characterisation of the upstream manifest is factually wrong. At the pinned version, `@earendil-works/pi-coding-agent@0.75.5` pins all three siblings with a caret: `"@earendil-works/pi-agent-core": "^0.75.5"`, `"@earendil-works/pi-ai": "^0.75.5"`, `"@earendil-works/pi-tui": "^0.75.5"`.

While Pi remains on a `0.x` line the distinction is invisible — `^0.75.5` and `~0.75.5` both resolve to `>=0.75.5 <0.76.0`, so the "same minor-version line" reasoning holds today. The error is in the literal operator named for a third-party manifest, and it diverges once Pi reaches `1.x`: `^1.Y.Z` admits the whole major line (`<2.0.0`) whereas `~1.Y.Z` admits only the minor. A contributor performing a Pi version bump who trusts this sentence would mis-predict upstream's resolution behaviour at that point.

This is a description-only error. Loom's own `peerDependencies` pin — the load-bearing constraint — is correctly and deliberately stated as tilde (`~0.75.5`) elsewhere in the same item and is the canonical single source of truth; loom does not read `pi-coding-agent`'s `package.json` at runtime. The fix is confined to the one inherited-from sentence (and the parallel "at the same `~X.Y.Z`" phrasing it anchors).

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commit:** `c307a62` — "pi-loom spec: resolve "^X.Y.Z prose label minor-version line only coincides with npm caret semantics while Pi is in 0.x"" (2026-05-07)
**History:** The corpus is git-tracked. `git log -S`/`-G` on the upstream-`dependencies` sentence shows that before `c307a62` the prose read "pins … at the same `^X.Y.Z` minor-version line", which correctly matched upstream's caret operator. `c307a62` was a deliberate caret→tilde sweep whose intent was to change loom's *own* pin to tilde (so loom's minor-lock holds even past `0.x`); that change to loom's own `peerDependencies` is correct. The sweep over-applied the substitution to the sentence describing upstream's manifest, flipping its correct `^X.Y.Z` to an incorrect `~X.Y.Z`. The defect later carried through the `81bf52a` `spec/`→`spec_topics/` page split unchanged, arriving at the current path.

## Solution Space

**Shape:** single

### Recommendation

In item 1's *Pi SDK pin* paragraph, restore the caret characterisation for the upstream manifest: describe `@earendil-works/pi-coding-agent`'s own `dependencies` block as pinning the three siblings at `^X.Y.Z` (concretely `^0.75.5`) on the same minor line, and note that the minor-line equivalence with loom's own tilde pin holds only while Pi is on `0.x`. Leave loom's own `peerDependencies` tilde pin (`~0.75.5`) and the canonical single-source-of-truth statement untouched — the two pins are deliberately different operators and the contrast (upstream caret vs. loom tilde) is the point. Do not change the caret→tilde edits `c307a62` correctly made to loom's own pin literal; only the upstream-`dependencies` description is wrong.

## Consequence

**Severity:** advisory

Correcting the operator changes no implementation behaviour: loom pins its own `peerDependencies` at tilde regardless of what upstream does. The defect is a grounding-accuracy gap in load-bearing rationale ("inherited from") that misinforms a version-bump contributor about upstream's actual resolution semantics, and becomes materially misleading the moment Pi crosses to `1.x`.
