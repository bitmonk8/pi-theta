## Problem

H1a's last Tests bullet (the `capability-probe.md` §(a) literal-read bullet) specifies a build-time assertion that reads `package.json#engines.node` and asserts it equals the pinned Node floor owned by `capability-probe.md` §(a), but never names where the comparison operand comes from. H1a's `Deps` is `-`, so neither V9a's module-resident pinned-constants block nor V18a's `SDK_SURFACE_INVENTORY` `pi-engines-node` row — the two in-tree single-source candidates — exists at H1a sequencing. An implementer is left with three defective readings: hardcode the floor (a third uncontrolled copy that drifts silently), read `package.json` against itself (a vacuous tautology that never catches the typo the bullet promises to catch), or read a not-yet-existing module constant (impossible at H1a).

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** fd2aa64 — pi-loom plan: resolve "engines.node populated but value not asserted in H1a" (2026-06-11, Thomas Andersen)
**History:** The `engines.node` field, populated and sourced from the `capability-probe.md` §(a) anchor "rather than restated here", entered H1a in f9cf76f (2026-06-10) with no asserting test. The build-time literal-read Tests bullet — "asserts it equals the pinned Node floor owned by `capability-probe.md` §(a)" — was added in fd2aa64 (2026-06-11) to close the "value not asserted" finding; that bullet introduced the comparison-operand requirement without naming the operand's in-code source, and at H1a's `Deps: -` sequencing no module-resident constant (V9a) or inventory row (V18a) yet exists to supply it.

## Solution approach

Amend the `capability-probe.md` §(a) Tests bullet in H1a to name the concrete in-code comparison operand and to state that no in-tree single-source constant exists at H1a sequencing. Specify the operand as the Node-floor literal transcribed inline into the H1a test from `capability-probe.md` §(a) (currently `>=22.19.0`), with the test asserting `package.json#engines.node` equals that independently-sourced transcribed literal rather than itself, so a `package.json` typo is caught. State that this transcribed literal is an H1a-resident copy the version-bump floor-move keeps aligned with `capability-probe.md` §(a).

## Solution constraints

- Cross-floor reconciliation across V9a's pinned-constants block and V18a's `pi-engines-node` inventory row remains owned by V18c's three-way `engines.node` gate; H1a's test must not read those later in-tree copies or duplicate that gate.
- The spec (`capability-probe.md` §(a)) is read-only for this fix.
