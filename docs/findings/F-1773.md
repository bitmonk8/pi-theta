## Finding

`defaulting-system-note-echo.md` opens its `## System-note rendering` section with: *"All binder-emitted system notes — the success echo, the `needs_info` and `ambiguous` failure messages, and the three runtime-emitted failure rows in the table below — share one line-discipline."* The failure-mode templates table is not below in the same file — it lives in `determinism-cancellation-failure.md` (the `### Failure-mode templates (normative)` table reached via the `#failure-mode-templates-normative` anchor). A reader following the in-prose "in the table below" pointer scrolls to the end of `defaulting-system-note-echo.md` and finds the `## Echo policy` section instead, with the BNDR-6 reference-renderings table being the only table on the page.

The row count is also stale. "Three runtime-emitted failure rows" matched the table as it stood when the rendering block was first written, but the table today carries five rows that are runtime- or classifier-emitted rather than echoing model text — *Binder model transport failure*, *Binder returned malformed envelope*, *AJV validation of the binder's `args` failed*, the `loomAbort.signal`-aborted cancellation row, and the `CustomMessage.customType` not-transcript-safe row added by BNDR-9. Whether the "three" was ever accurate (the cancellation row appears to have been omitted from the count even at introduction) is a side issue; the count no longer matches the table either way, and a reader cross-referencing rendering rules against actual rows cannot tell which rows the discipline applies to.

Both halves point at the same underlying placement issue: the shared rendering discipline is homed in a file whose name (`defaulting-system-note-echo`) advertises only two of its five audiences (success echo + the two `bind_echo` outputs), while the other three audiences — the runtime/classifier-emitted failure rows — live in a sibling file. The intra-file pointer and the miscount are both downstream of that.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `989b93e` (2026-05-04, *pi-loom spec: resolve "needs\_info / ambiguous message strings flow into system notes without bounds"*) and `f5e89f4` (2026-06-04, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*).
**History:** `git log -S "three runtime-emitted"` returns exactly the two commits above for the spec corpus. `989b93e` added the `**System-note rendering.**` block to the then-monolithic `spec_topics/binder.md` carrying the exact phrasing "the three runtime-emitted failure rows in the table below"; at that commit the rendering block and the six-row failure-modes table both lived in `binder.md`, so the intra-file pointer was accurate. The "three" count already excluded the cancellation row at that point — the table had needs_info, ambiguous, transport, malformed, AJV, cancellation, of which arguably four (transport, malformed, AJV, cancellation) are runtime-emitted, not three. `f5e89f4` (the 100KB-cap split) moved the rendering block to `docs/spec_topics/binder/defaulting-system-note-echo.md` and the failure-modes table to `docs/spec_topics/binder/determinism-cancellation-failure.md` via a map-driven rewrite that repointed 1571 cross-references but did not rewrite the in-prose pointer "in the table below"; that is the commit at which the pointer became stale and cross-file. A later edit (`f41af5d`, 2026-06-24, BNDR-9) appended the *custom-message type is not transcript-safe* row, widening the gap between the prose count and the table's actual runtime-emitted row count further still without touching the "three" wording.

## Solution Space

**Shape:** multiple

### Option A — Repoint in place

**Approach.** Keep `## System-note rendering` in `defaulting-system-note-echo.md` and fix the cross-file pointer plus the miscount. Replace "the three runtime-emitted failure rows in the table below" with an anchored cross-file reference along the lines of "the runtime-emitted failure rows of the [failure-mode templates](./determinism-cancellation-failure.md#failure-mode-templates-normative)" — drop the numeric count rather than committing to a number that future BNDR additions will desync again. Apply the same anchored-link replacement to rule 4's "the failure-modes table" and rule 5's "the failure-modes table" so all three references in the section share one citation form. Audit the sibling cancellation paragraph in `determinism-cancellation-failure.md` line 21 ("the failure-modes table below") for consistency — that one is intra-file and remains accurate.

**Spec edits.** Three edits in `defaulting-system-note-echo.md` (lead paragraph, rule 4, rule 5) plus one consistency check on the cancellation paragraph in `determinism-cancellation-failure.md`.

**Pros.** Single-file, low-risk edit. Preserves all existing inbound anchors (`#system-note-rendering`, the BNDR-6 sub-letter anchors, `#post-default-merge-ajv-validation`). Resolves both halves of the finding (stale pointer + miscount) without touching the failure-mode-templates table or its anchor.

**Cons.** Leaves the structural mismatch in place — the shared rendering discipline is still homed in a file whose name advertises only two of its five audiences. A future reader's first instinct will still be that the discipline is `bind_echo`-scoped.

**Risks.** None material; the inbound cross-references that today resolve into `defaulting-system-note-echo.md#system-note-rendering` continue to resolve unchanged.

### Option B — Relocate to a shared file

**Approach.** Lift `## System-note rendering` (including the reference-rendering paragraph) out of `defaulting-system-note-echo.md` into either `determinism-cancellation-failure.md` (immediately before `### Failure-class taxonomy`) or its own new sibling `docs/spec_topics/binder/system-note-rendering.md`. The receiving site's name then matches the discipline's actual scope. Update the lead-paragraph wording to match the new location (drop the stale "in the table below", drop the "three" count, or rewrite to "the runtime-emitted failure rows below" if the receiving site is `determinism-cancellation-failure.md` and the section lands above the templates table).

**Spec edits.** Move the `## System-note rendering` block and its `<a id="system-note-rendering"></a>` anchor; preserve the anchor at the new location so the eight existing inbound cross-references (in `queryerror-variants.md`, `code-registry-parse.md`, `query-forms.md` ×3, `binder-model-and-context.md`, `binder-bypass-and-envelope.md` ×2, plus the `V11e`/`V11e-T` plan leaves' five-rule citations and `findings/F-1200.md`) continue to resolve. Decide between relative-link rewrites or anchor-preservation via a redirect stub; the latter is cheaper but leaves a stub.

**Pros.** Eliminates the cross-file pointer entirely (if relocated to `determinism-cancellation-failure.md`) and homes the shared discipline at a file whose name advertises its scope, removing the underlying placement issue.

**Cons.** Moves an anchor that ~10 other sites cross-reference; every relative path must be re-resolved. Either grows `determinism-cancellation-failure.md` (current size ≈10KB, well under cap) or adds a fourth file to the binder/ subdir. Touches more files than Option A for the same observable improvement.

**Risks.** Anchor-preservation slip: a missed inbound link silently breaks navigation. `docs/_tools/check_links.py` would catch it but the surface area is wider than Option A.

### Recommendation

**Option A.** The finding is fundamentally a navigation/clarity defect with no behavioural consequence, and Option A resolves both halves (stale pointer + miscount) with a three-line edit and zero anchor churn. Option B's structural improvement is real but its cost — moving an anchor that ~10 sites depend on — is disproportionate to the navigational gain. Implementer notes: (a) drop the numeric count rather than committing to "four" or "five" so future BNDR additions do not re-stale the prose; (b) leave the sibling pointer in `determinism-cancellation-failure.md` line 21 ("the failure-modes table below") as-is — that one is intra-file and correctly oriented downward.

## Consequence

**Severity:** advisory

A reader who follows "the table below" verbatim from the rendering section lands in `## Echo policy` rather than at the failure-mode templates, and the stated row count does not match the table either way. The line-discipline still binds (the failure-mode-templates intro restates that every row renders through the shared discipline), so no implementer is at risk of producing the wrong observable bytes; the cost is purely navigational and a reviewer second-guessing whether the discipline applies to rows the count appears to exclude.
