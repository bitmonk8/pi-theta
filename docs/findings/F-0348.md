## Finding

`lexical.md` §Encoding makes the offset reporting normative: a non-UTF-8 byte sequence (or a non-UTF-8 BOM) is a load-time `loom/load/invalid-encoding` error that MUST report **the zero-based byte offset of the first invalid byte in the original (pre-normalisation) content**, and **`0` for a non-UTF-8 BOM** (the BOM's leading byte, explicitly *not* the byte at which the UTF-8 state machine first faults). The diagnostics registry reinforces this with the fixed message template `invalid UTF-8 encoding at byte offset <offset>`. `V1a` owns `loom/load/invalid-encoding` and is the closing leaf for the `lexical.md` (LEX) coverage row.

`V1a`'s **Adds** only frames the offset as a *capability* — invalid bytes and their offsets are "recoverable pre-normalisation" via the `V8b` `readBytes` raw-`Uint8Array` seam — and the `loom/load/invalid-encoding` **Tests** bullet (identical in `V1a` and `V1a-T`) reads "a non-UTF-8 byte sequence fails load with this code **at the byte offset**." Neither the test bullet nor **Ships when** pins *which* offset must be reported: the spec's "zero-based offset of the first invalid byte" and the non-UTF-8-BOM `offset = 0` special case are absent from the acceptance criterion. Because the standard `TextDecoder({fatal:true})` path throws without exposing any offset, a correct implementation requires a byte-walking validator (or equivalent), which the leaf leaves entirely implicit.

The practical consequence is in the TDD ritual: the `V1a-T` author writes the failing test from a bullet that asserts only "fires at the byte offset," so a test that checks the code fires with an arbitrary or unverified offset satisfies the gate. An implementation reporting the fault-machine position for a non-UTF-8 BOM (rather than `0`), or any other wrong offset, ships green.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** The `loom/load/invalid-encoding` test bullet "a non-UTF-8 byte sequence fails load with this code at the byte offset" is present verbatim in both `V1a-lexer-core.md` and `V1a-T-lexer-core.md` as of the initial plan-build commit `c6a664e`; the loose acceptance property has never pinned "zero-based offset of the first invalid byte" or the non-UTF-8-BOM `offset = 0` case. A later commit `fa18160` ("resolve 'V1a presupposes a pre-decode byte source that no seam supplies'", 2026-06-10) added the `V8b` raw-`Uint8Array` byte-source framing and the "byte offsets are recoverable pre-normalisation" wording to **Adds**, but did not elevate offset reporting into the Tests/Ships-when acceptance criterion. The defect therefore dates to plan inception, not to a later edit; `git log -G` over the bullet text confirms no intervening rewrite of the acceptance criterion.

## Solution Space

**Shape:** single

### Recommendation

Tighten the `loom/load/invalid-encoding` acceptance criterion in both leaves so the reported offset is a witnessed property rather than an unspecified "the byte offset."

- In `docs/plan_topics/V1a-T-lexer-core.md`, replace the `loom/load/invalid-encoding` Tests bullet with one that asserts the reported offset equals the **zero-based byte offset of the first invalid byte in the original pre-normalisation content**, and add a vector asserting a **non-UTF-8 BOM reports offset `0`** (per `lexical.md` §Encoding). Suggested literal text: ``` `loom/load/invalid-encoding`: a non-UTF-8 byte sequence fails load with this code reporting the zero-based byte offset of the first invalid byte; a non-UTF-8 BOM fails load reporting offset `0`. ```
- Make the same replacement to the corresponding bullet in `docs/plan_topics/V1a-lexer-core.md` so the implementation leaf's acceptance criterion matches its `-T` partner.
- Leave the decode mechanism (byte-walking validator vs. any equivalent that surfaces the offset, given `TextDecoder({fatal:true})` does not expose one) to the implementer; the **Adds** "recoverable … via the `V8b` byte-read seam" framing already supplies the byte source and needs no change.

The `coverage-matrix.md` LEX/GRAM row already maps `V1a`; no matrix edit is required.

## Consequence

**Severity:** advisory

If shipped unfixed, the `V1a-T` failing test is authored from a bullet that pins only that `loom/load/invalid-encoding` fires "at the byte offset," so an implementation that reports a wrong offset — or omits the spec's non-UTF-8-BOM `offset = 0` special case — passes the gate, and the `invalid UTF-8 encoding at byte offset <offset>` message ships with an incorrect offset undetected by CI. The cited `lexical.md` §Encoding fully specifies the correct behaviour, so a careful implementer who reads the spec still produces a conforming leaf; the gap is that the acceptance criterion does not independently witness the offset value.
