## Finding

`discovery/package-and-settings.md` DISC-5 places normative weight on the `minimatch` engine: it pins the glob matcher loom uses to resolve `pi.looms` patterns, and grounds the dialect by asserting parity with "the same package Pi applies to its own resource arrays". That obligation is load-bearing — V10b's `DISC-5` test asserts the `!`/`+`/`-` precedence and the per-entry contribution rule by exercising the matcher's behaviour against minimatch's dialect.

The owning inventory for loom-package implementation dependencies is the `**Loom-package implementation dependencies (loom 1.0).**` bullet in `implementation-notes.md`, which enumerates `semver`, `chokidar`, and `yaml` as entries in loom's own `dependencies` block (each named, located in `dependencies`, version range deferred to the manifest). `minimatch` is missing from that enumeration. The plan side already commits to provisioning it (`H1a` declares `minimatch` in loom's runtime `dependencies` and `V10b` consumes it), so the operational fact — loom carries its own `minimatch`, distinct from Pi's — is settled; the spec's inventory page just doesn't reflect it.

This matters because the dialect-parity claim in DISC-5 is only meaningful if the reader can tell which `minimatch` instance loom resolves against. Today the spec leaves three possibilities open: loom reuses Pi's instance, loom carries its own copy, or loom hot-imports Pi's bundled copy. They have different version-skew exposure: if loom carries its own `minimatch`, two independent semver ranges can drift between loom and Pi, and the dialect-parity claim becomes a property the inventory must keep, not a property the package boundary guarantees. An implementer re-deriving the dependency set from the spec alone would miss `minimatch` entirely.

## Issue introduction

**Verdict:** single-commit

**Introducing commits:**

- `1cb54c7` (2026-06-06) — *pi-loom spec: resolve "Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths"*

**History:** Commit `1cb54c7` introduced the normative `minimatch` reliance into DISC-5, pinning both the engine and the dialect-parity claim against Pi's `applyPatterns`. The `Loom-package implementation dependencies (loom 1.0)` bullet on `implementation-notes.md` already existed at that point — added by `1ea0455` (2026-06-04) for `semver` and `chokidar`, later extended by `1064946` (2026-06-10) for `yaml` in response to a parallel "YAML frontmatter parsing mechanism is never declared as a dependency" finding — so the inventory site that this finding asks to be backfilled was a known landing surface; the defect is that `1cb54c7` added the DISC-5 reliance without the symmetric inventory entry the earlier and later sibling additions modelled.

## Solution Space

**Shape:** single

### Recommendation

Add a `minimatch` entry to the `**Loom-package implementation dependencies (loom 1.0).**` bullet on `docs/spec_topics/implementation-notes.md`, in the same style as the existing `semver` / `chokidar` / `yaml` entries: name the package (link to `https://www.npmjs.com/package/minimatch`), state that it is "declared in the loom package's `dependencies` block on the same footing as `semver`, `chokidar`, and `yaml`", note that this clause names the dependency and not its version, and add the parallel inventory-closure-audit carve-out ("`minimatch` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it is out of scope for the inventory-closure audit's *Target surface categories* and requires no `// allow-pi-surface:` exemption marker at any `minimatch` import site"). State explicitly that loom carries its own `minimatch` instance and does not reuse Pi's; that pins the version-skew question to loom's own manifest rather than to Pi's bundled copy.

In DISC-5, replace the bare normative reliance with a cross-reference to the new inventory entry on the model of how the H1a plan leaf already grounds it, so the dialect-parity claim ("the same package Pi applies") is anchored to the inventory rather than floating on the bundler-output symbol citation. The version-skew posture this creates — loom's own minimatch range needs to stay dialect-compatible with the range Pi ships — should be stated once at the inventory entry; the editorial-review re-audit obligation on Pi version bump can attach there.

Implementer notes:
- Do **not** add a `@types/minimatch` entry: `minimatch` ships its own TypeScript bindings (parallel to `yaml`'s carve-out in the same inventory bullet).
- Leave the dependencies-block grounding in `H1a` consistent with the new inventory entry — the existing H1a prose ("`minimatch` belongs in loom's own `dependencies` as the glob matcher engine for `pi.looms` resolution … grounded in `discovery/package-and-settings.md` §DISC-5") may optionally be re-anchored at the new `implementation-notes.md` inventory site (matching how `H1a` cites `semver`/`chokidar`/`yaml`), but this is editorial alignment, not a correctness requirement.
- The hardening of DISC-5's `dist/core/package-manager.js` / `applyPatterns` citation — adding the `known-fragile evidence` hedge or dropping the bundler-output symbol path — is owned by the sibling finding noted under *Related Findings* below and is not part of this fix.

## Consequence

**Severity:** correctness

The dependency inventory that the spec asks readers to consult is incomplete relative to the normative engine DISC-5 names, leaving the loom-vs-Pi-instance ownership question unresolved on the spec side. An implementer wiring `pi.looms` resolution from the spec alone could omit the `minimatch` dependency declaration or assume Pi's instance is re-used, and the dialect-parity claim DISC-5 leans on has no inventory entry to attach version-skew discipline to.
