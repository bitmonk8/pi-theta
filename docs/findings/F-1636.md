## Finding

DISC-5 requires `minimatch` for `pi.looms` glob resolution, but the canonical `**Loom-package implementation dependencies (loom 1.0).**` bullet in `implementation-notes.md` lists `semver`/`chokidar`/`yaml` and omits `minimatch`; its version-skew posture against Pi's own `minimatch` is unstated.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `1cb54c7` — pi-loom spec: resolve "Glob !/+/- precedence and matcher engine unspecified for pi.looms / loomPaths" (2026-06-06, Thomas Andersen); `1ea0455` — pi-loom spec: resolve "DI-seam vs reference-implementation placement boundary" (2026-06-04, Thomas Andersen); `1064946` — pi-loom plan: resolve "YAML frontmatter parsing mechanism is never declared as a dependency" (2026-06-10, Thomas Andersen)
**History:** The normative `minimatch` reliance in DISC-5, together with the `dist/core/package-manager.js` / `applyPatterns` private-symbol pin, entered the corpus in commit `1cb54c7` (2026-06-06) — `git log -S "applyPatterns"` and `git log -S "minimatch"` against `package-and-settings.md` each return exactly that one commit. The dependency inventory bullet in `implementation-notes.md` predates that reliance (created by `1ea0455`, 2026-06-04, for `semver`) and was subsequently extended for sibling packages (`chokidar`, then `yaml` via `1064946`, 2026-06-10) in response to parallel "X is never declared as a dependency" findings, but was never extended for `minimatch` — `git log -S "minimatch" -- docs/spec_topics/implementation-notes.md` returns nothing. The "undeclared dependency" gap thus arose from the interaction between the reliance-introducing commit and the separately-evolving inventory that closed the sibling cases but never the `minimatch` one.

## Solution Space

**Shape:** single

### Recommendation

Add a `minimatch` entry to the dependency-inventory bullet (anchor `loom-package-implementation-dependencies-loom-1-0`) on the same footing as the existing `semver`/`chokidar`/`yaml` entries: state that loom 1.0 depends on the `minimatch` npm package as the glob matcher behind DISC-5's `pi.looms` resolution, declared in the loom package's `dependencies` block, naming the dependency rather than a version; note that loom carries its own `minimatch` instance distinct from Pi's (the DISC-5 parity claim is about dialect identity, not a shared runtime instance), so version skew is a deliberate, tolerated posture; mirror the `chokidar`/`yaml` audit-scope clause (`minimatch` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it is out of scope for the inventory-closure audit's *Target surface categories* and needs no `// allow-pi-surface:` exemption marker at any `minimatch` import site).

## Consequence

**Severity:** advisory

An implementer provisioning loom's `package.json` from the canonical dependency inventory would omit `minimatch`, leaving DISC-5's `pi.looms` glob resolution unbacked; the gap is recoverable because DISC-5 names the engine and the `H1a` leaf already declares it, but the authoritative inventory is incomplete and the version-skew posture is unstated. Separately, the normative `dist/core/package-manager.js` / `applyPatterns` pin goes stale silently if Pi changes its backing matcher, entangling loom's correct obligation (track Pi's glob dialect) with a now-wrong library identity.
