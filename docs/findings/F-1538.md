## Finding

PIC-22 (*Parallel subagent spawn initiation*, `subagent.md`) mandates that when the model emits N ≥ 2 subagent-mode `.loom` callables as parallel tool calls, the runtime MUST initiate `createAgentSession` for all N invocations before any one of them returns — i.e. N spawn calls are concurrently in flight, interleaving at their `await` points. The correctness of the resulting N sessions silently depends on a Pi-side property the spec never states: that concurrent `createAgentSession` calls yield mutually independent sessions, with no shared Pi-side mutable state mutated non-atomically across the spawn's `await` boundaries (e.g. a module-level registry, a default-tool table, or a shared session counter that two interleaved spawns could cross-contaminate).

The isolation guarantees the page does make are all *loom-side*: "Each subagent-mode invocation gets its own `AgentSession`, its own captured `tools:` table, and its own derived `AbortSignal`; sibling invocations triggered by parallel tool calls do not share mutable session state", and the state-isolation matrix. These describe the loom-owned per-invocation structures the runtime constructs around each spawn — they say nothing about whether Pi's own `createAgentSession` machinery is safe to drive concurrently. The PIC-22 conformance test witnesses only the *initiation ordering* (all N sessions created and entered before any blocked `sendUserMessage` is released) against a **fake** `AgentSession`, so it never exercises a real Pi `createAgentSession` under concurrency and cannot detect a shared-state spawn hazard.

This is the same class of unpinned host-behaviour presupposition the spec already handles systematically elsewhere: sibling behaviours such as the parallel-tool *batch delivery* shape (checklist item (an)), the `tools`-allowlist suppression of Pi default built-ins (item (am)), and the spawn satellite types (items (o)/(aj)) are each recorded as a named presupposition with an *Authoritative source* pointer and an on-bump editorial-review checklist item. Concurrent-spawn independence — a strictly distinct property from any of those — has no such record and no bump-time re-check. The spec's own rule that "any future presupposition added … that routes its detection to editorial review … MUST be added to this checklist in the same edit" underscores that an uncaptured presupposition of this kind is a gap in the established discipline, not a deliberate omission.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commit:** `7afe9bb` (2026-05-31, "pi-loom spec: resolve \"no-invocation-cap MUST NOT is unobservable\"")
**History:** The spec corpus is git-tracked. `git log -S` on the parallel-spawn rule text ("before any one of the N returns") and `git show 7afe9bb -- docs/spec_topics/pi-integration-contract.md` show that commit `7afe9bb` first introduced the parallel-spawn MUST ("the runtime MUST initiate `createAgentSession` for all N invocations before any one of the N returns") together with its fake-`AgentSession` conformance-test framing into the then-monolithic `pi-integration-contract.md`; the pre-image carried no parallel-spawn obligation. The concurrent-spawn-independence presupposition was absent at that introduction and was never added afterward: the rule was carried forward through the file split into `subagent.md` (`f5e89f4`, 2026-06-04), the N≥3→N≥2 bound narrowing (`f566349`, 2026-06-23), and the PIC-40..43 numbering pass (`3253d16`, 2026-06-25) unchanged, while the editorial-review checklist independently grew sibling presupposition items (e.g. (am), (an)) without ever minting one for concurrent-spawn independence. The defect is therefore an omission present since the parallel-spawn rule entered the corpus at `7afe9bb`.

## Solution Space

**Shape:** single

### Recommendation

Record concurrent `createAgentSession` independence as a named host-behaviour presupposition, following the established presupposition + bump-checklist pattern this corpus already uses for items (am)/(an)/(o):

1. In `subagent.md`, attach a short presupposition note to PIC-22 stating that the parallel-spawn obligation presupposes that N concurrent `createAgentSession` calls yield mutually independent sessions — no Pi-side mutable spawn-time state is shared or mutated non-atomically across the concurrently in-flight spawns — and that this property is neither factory-probed nor enumerated by step 2(a)'s seven-capability literal-read assertion, so a regression would surface only as cross-contaminated sibling sessions at runtime. Distinguish it explicitly from the loom-side "do not share mutable session state" claim (which covers only loom's own per-invocation structures) and from item (an)'s parallel-tool *batch delivery* property. Add an *Authoritative source* pointer in the same known-fragile-evidence style used by the page's other presuppositions.
2. In `version-bump-step2.md`, add a new SHOULD-level editorial-review checklist item (the next free letter after (ap)) cross-referencing the PIC-22 note, instructing the contributor to confirm against the candidate Pi minor that concurrent `createAgentSession` spawns remain mutually independent, with the standard "escalates to a build-time pin once Pi exposes a typed contract" clause. The spec's own "any future presupposition … MUST be added to this checklist in the same edit" rule requires this item to land in the same change.
3. Update `V18c` / `V18c-T`'s acceptance enumeration (currently "items (a)–(al)") to include the new item.

Edge case the implementer must watch: keep the new presupposition scoped to *concurrent-spawn* independence specifically — do not let it be folded into item (an) (parallel-tool batch delivery) or item (am) (tools-allowlist), since neither audits whether `createAgentSession` itself is concurrency-safe.

## Consequence

**Severity:** advisory

At the loom 1.0 Pi-SDK pin the assumption almost certainly holds (Pi's `createAgentSession` is presumably re-entrant), so implementers can build a working system today. The gap is that a future Pi minor that introduced shared spawn-time mutable state would silently corrupt concurrently-spawned sibling sessions with no SDK surface-inventory signal and no bump-time editorial re-check to catch it — the precise failure mode the editorial-review checklist exists to prevent.
