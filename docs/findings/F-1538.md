## Problem

PIC-22 (`subagent.md`, `#pic-22`) mandates that the runtime initiate `createAgentSession` for all N parallel subagent-mode spawns before any one returns, leaving N spawn calls concurrently in flight. The resulting sessions silently depend on a Pi-side property the spec never states: that concurrent `createAgentSession` calls yield mutually independent sessions, with no shared Pi-side spawn-time mutable state mutated non-atomically across the spawns' `await` boundaries. The isolation guarantees the page does make — the "do not share mutable session state" claim and the state-isolation matrix — are all loom-side and cover only the per-invocation structures the runtime builds around each spawn. Unlike sibling host presuppositions (items (am)/(an)/(o)), this concurrent-spawn-independence property has no named presupposition record, no *Authoritative source* pointer, and no bump-time editorial re-check; the PIC-22 conformance test uses a fake `AgentSession` and never exercises real Pi `createAgentSession` under concurrency.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commit:** `7afe9bb` (2026-05-31, "pi-loom spec: resolve \"no-invocation-cap MUST NOT is unobservable\"")
**History:** The spec corpus is git-tracked. `git log -S` on the parallel-spawn rule text ("before any one of the N returns") and `git show 7afe9bb -- docs/spec_topics/pi-integration-contract.md` show that commit `7afe9bb` first introduced the parallel-spawn MUST ("the runtime MUST initiate `createAgentSession` for all N invocations before any one of the N returns") together with its fake-`AgentSession` conformance-test framing into the then-monolithic `pi-integration-contract.md`; the pre-image carried no parallel-spawn obligation. The concurrent-spawn-independence presupposition was absent at that introduction and was never added afterward: the rule was carried forward through the file split into `subagent.md` (`f5e89f4`, 2026-06-04), the N≥3→N≥2 bound narrowing (`f566349`, 2026-06-23), and the PIC-40..43 numbering pass (`3253d16`, 2026-06-25) unchanged, while the editorial-review checklist independently grew sibling presupposition items (e.g. (am), (an)) without ever minting one for concurrent-spawn independence. The defect is therefore an omission present since the parallel-spawn rule entered the corpus at `7afe9bb`.

## Solution approach

Record concurrent `createAgentSession` independence as a named host-behaviour presupposition, following the presupposition + bump-checklist pattern the corpus already uses for items (am)/(an). Add a presupposition note to PIC-22 in `subagent.md` stating that the parallel-spawn obligation presupposes N concurrent `createAgentSession` calls yield mutually independent sessions, distinguished from the loom-side "do not share mutable session state" claim, with an *Authoritative source* pointer in the page's known-fragile-evidence style. Add the next-free-letter editorial-review checklist item after (ap) in `version-bump-step2.md`, cross-referencing the PIC-22 note. Update `V18c` / `V18c-T`'s acceptance enumeration to include the new item.

## Solution constraints

- The PIC-22 presupposition note and the new editorial-review checklist item MUST land in the same edit, per `version-bump-step2.md`'s own rule that any future presupposition routing detection to editorial review be added to this checklist in the same change.
- Keep the new presupposition scoped specifically to concurrent-spawn independence; do not fold it into item (an) (parallel-tool batch delivery) or item (am) (tools-allowlist), neither of which audits whether `createAgentSession` itself is concurrency-safe.
