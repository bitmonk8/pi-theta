## Finding

The System-notes best-effort fallback chain is canonically defined in `runtime-event-channel.md` as an ordered step sequence ‚Äî `pi.sendMessage` ‚Üí `ctx.ui.notify` ‚Üí `loom/runtime/system-note-delivery-failed` ‚Üí terminal `console.error`. That definition site describes the chain entirely in terms of the underlying SDK calls; it never coins or introduces the identifier `sendSystemNote`.

Yet `sendSystemNote` is used as the *name* of that chain at numerous reference sites ‚Äî PIC-54 ("a throw ‚Ä¶ MUST NOT propagate out of the `sendSystemNote` fallback chain") and the `diagnostic-emission-isolation.md` paragraph that imports PIC-54, plus `active-invocation-registry.md` ("The `sendSystemNote` fallback chain"), `extension-bootstrap-and-per-loom.md`, `capability-probe.md`, `session-shutdown-semantics.md`, and the `code-registry-load.md` / `code-registry-runtime.md` rows (all rendering the chain as `sendSystemNote` ‚Üí `ctx.ui.notify` ‚Üí `console.error`). None of these sites ‚Äî and no glossary entry ‚Äî say what `sendSystemNote` *is*.

A reader cannot tell from the spec alone whether `sendSystemNote` denotes a loom-internal helper wrapping `pi.sendMessage`, a Pi SDK export, or an informal shorthand for the whole chain. The chain's behaviour is fully pinned, so this is a terminology gap rather than a behavioural one, but it leaves the most-cited name in the System-notes machinery as an unbound term that every reference site assumes is defined elsewhere.

## Issue introduction

**Verdict:** single-commit (with later propagation)
**Introducing commit:** `5e8b499` (2026-05-06, "pi-loom spec: resolve \"Hot-reload teardown contract for the loom extension instance is unspecified\"")
**History:** `5e8b499` was the first commit to use `sendSystemNote` as a name in the spec corpus, in the teardown-handler call-prohibition bullet ("The `sendSystemNote` fallback chain (the best-effort chain defined under **System notes** below)"). The term was imported from the implementation vocabulary (the plan's `sendSystemNote` helper, which wraps `pi.sendMessage`/`ctx.ui.notify`) and cross-referenced the **System notes** section ‚Äî but that section defines the chain only in terms of `pi.sendMessage`, so the name was unbound from its very first spec use. Later commits added more reference sites inheriting the same undefined label: `2d2fc8e` (2026-05-06) routed the capability-probe and extension-bootstrap failures through "the **System notes** fallback chain (`sendSystemNote` ‚Üí `ctx.ui.notify` ‚Üí `console.error`)", and `cfb3943` coined PIC-54 ("MUST NOT propagate out of the `sendSystemNote` fallback chain") at the definition site itself without introducing the name. The defect was present at the term's introduction and was never closed as reference sites accumulated.

## Solution Space

**Shape:** single

### Recommendation

At the System-notes fallback-chain definition in `runtime-event-channel.md` (the paragraph that pins the `pi.sendMessage` ‚Üí `ctx.ui.notify` ‚Üí diagnostic ‚Üí terminal `console.error` order), introduce `sendSystemNote` by name as the label for that chain ‚Äî stating that `sendSystemNote` is the loom-internal best-effort emission helper that drives this ordered fallback (its first step being the `pi.sendMessage` call), not a Pi SDK export. Every existing reference site ("the `sendSystemNote` fallback chain", PIC-54, the capability-probe / extension-bootstrap / code-registry rows) then binds to a defined term with no edit to those sites.

Edge cases for the implementer:
- The definition must make clear `sendSystemNote` names the *whole ordered chain* (the helper), not solely the `pi.sendMessage` first step, since `code-registry-runtime.md` says a note "routes through the standard `sendSystemNote` fallback chain (degrading to `ctx.ui.notify` and ultimately `console.error`)".
- Keep the term consistent with the plan's `V7d` usage (`sendSystemNote` ‚Üí `ctx.ui.notify` ‚Üí `console.error`); do not coin a competing name.

## Consequence

**Severity:** advisory

Two implementers reading the spec arrive at the same chain behaviour (the steps are fully pinned) but disagree on what `sendSystemNote` is ‚Äî a loom helper, a Pi SDK symbol, or pure shorthand ‚Äî so cross-references to "the `sendSystemNote` fallback chain" carry an unresolved referent. No working system is blocked, but the most-cited name in the System-notes machinery is undefined.
