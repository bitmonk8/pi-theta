## Finding

`V4b`'s final Tests bullet (and the mirror bullet in `V4b-T`) asserts: "A host-fatal uncatchable condition (NOCEIL-3 carve-out) emits no diagnostic at all — in particular, no `loom/runtime/internal-error` is delivered." The catchable-vs-uncatchable boundary that makes this bullet testable is carried entirely in the NOCEIL-3 text, which lives on `spec_topics/hard-ceilings/ceiling-invariants-and-audit.md`. Both leaves' **Spec** fields link only `errors-and-results.md` and `errors-and-results/error-model.md`, and neither of those pages states the carve-out — `errors-and-results.md` has no mention of "uncatchable", "NOCEIL", or "heap out of memory".

NOCEIL-3 draws a sharp line that the leaf does not reproduce: *catchable* allocation failures (`RangeError: Invalid string length`, `RangeError: Invalid array length`, `RangeError: Maximum call stack size exceeded`, and any JavaScript-exception-shaped allocation failure) are part of the runtime-defect surface and **do** route through `loom/runtime/internal-error`; only *uncatchable* engine fatals (V8's `FATAL ERROR: Reached heap limit … JavaScript heap out of memory`, the `OOMErrorCallback` / `abort()` path that bypasses the JS exception machinery) produce no diagnostic. An implementer working from V4b's linked pages alone cannot see this distinction, so the "uncatchable condition emits no diagnostic" bullet is under-specified at the point it is tested.

Sibling leaves that cite ceiling inline-labels carry the seam definition inline to compensate — `V15b` spells out the NOCEIL-4 frame-depth seam, `V6a` spells out the NOCEIL-1 `timeout:` seam. `V4b` is the outlier: it names NOCEIL-3 but neither links its definition nor restates the stimulus.

## Solution Space

**Shape:** single

### Recommendation

Give both `V4b` and `V4b-T` access to the NOCEIL-3 catchable-vs-uncatchable boundary. Two equally acceptable content moves (one or both):

- Add the NOCEIL-3 definition to each leaf's **Spec** field. Because `#noceil-3` does not currently resolve (no HTML anchor at the NOCEIL-3 site), link the existing section anchor `../spec_topics/hard-ceilings/ceiling-invariants-and-audit.md#no-additional-ceilings`, or — if a precise deep link is wanted — add a GOV-1-conformant `<a id="noceil-3"></a>` anchor at the NOCEIL-3 site (a spec edit, hence the option-dependent tag on the spec file) and link `#noceil-3`.
- Restate the stimulus inline in the carve-out bullet so the boundary is unambiguous without leaving the leaf: catchable JavaScript-exception-shaped allocation failures (e.g. `RangeError: Invalid string length` / `Invalid array length` / `Maximum call stack size exceeded`) route through `loom/runtime/internal-error`; only uncatchable engine fatals (V8 heap-OOM / `OOMErrorCallback` / `abort()` path) emit no diagnostic.

Linking the spec page is the lower-maintenance move (the inline-stimulus restatement duplicates spec text that can drift). If the spec edit to add the `noceil-3` anchor is taken, keep it within the inline-label anchor conventions in `governance/stable-inline-labels.md`; otherwise the section anchor is sufficient. The same edit applied to `V4b-T` keeps the tests leaf and its implementation leaf in lockstep.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on which conditions the carve-out covers: one writes a V4b-T test asserting that a catchable `RangeError` allocation failure emits no diagnostic — directly contradicting NOCEIL-3, which requires it to route through `loom/runtime/internal-error` — while another correctly scopes the test to process-fatal heap-OOM. The first produces a leaf that ships behaviour at odds with the spec while its own test stays green.
