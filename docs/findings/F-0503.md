## Problem

The final Tests bullet of `V4b` (and the mirror bullet in `V4b-T`) asserts that a host-fatal uncatchable condition (NOCEIL-3 carve-out) emits no diagnostic, in particular no `loom/runtime/internal-error`. The catchable-vs-uncatchable boundary that makes this bullet testable lives only in the NOCEIL-3 text on `spec_topics/hard-ceilings/ceiling-invariants-and-audit.md`, which neither leaf links. Both leaves' **Spec** fields link only `errors-and-results.md` and `errors-and-results/error-model.md`, neither of which mentions the carve-out. NOCEIL-3 requires *catchable* allocation failures (e.g. `RangeError: Invalid string length`) to route through `loom/runtime/internal-error` while only *uncatchable* engine fatals emit nothing; an implementer working from V4b's linked pages alone cannot see this distinction and may write a test that contradicts NOCEIL-3.

## Solution approach

Give both `V4b` and `V4b-T` access to the NOCEIL-3 catchable-vs-uncatchable boundary, applying the same edit to both leaves to keep the tests leaf and its implementation leaf in lockstep. Either add a NOCEIL-3 spec link to each leaf's **Spec** field — the existing section anchor `../spec_topics/hard-ceilings/ceiling-invariants-and-audit.md#no-additional-ceilings` resolves today, whereas `#noceil-3` does not — or restate the catchable-vs-uncatchable stimulus inline in the carve-out bullet so the boundary is unambiguous without leaving the leaf.
