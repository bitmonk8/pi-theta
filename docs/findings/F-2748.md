## Finding

`V18d` carries two independently-verifiable version-bump obligations in one leaf:

1. **The runtime-evidence acceptance gate** — an `H4a`-backed end-to-end run of a representative integrated `.loom` (typed query + tool loop + invoke + schema validation + binder + cancellation) against the *bumped* Pi-SDK pin, asserting green. Verified by the first Tests bullet, against the `H4a` harness double at the bumped pin.

2. **The revert/rollback path** — on a red runtime-evidence run (or a confirmed real-host-smoke behavioural-divergence finding), restore step 4's edit (the single-source-of-truth Pi-SDK pin literal plus the four `@earendil-works/*` `peerDependencies` entries) *before merge*, then re-run `V18c`'s static build-time gates against the *restored prior* pin and widen the revert if any gate stays red. Verified by the second Tests bullet, against the `H4a` and `V18c`-gate doubles at the prior pin.

The two are distinct code paths with distinct setups: the acceptance gate drives a green H4a run at the bumped pin; the revert path drives a *simulated-red* run and then exercises the `V18c` static-gate doubles against the prior pin to observe the complete-revert and incomplete-revert (widen) signals. `V18d`'s **Ships when** has two arms (one per obligation), and the paired `V18d-T` carries two corresponding red scenarios. An agent cannot author, drive, and verify both obligations in a single coherent TDD pass — they share no fixture beyond the H4a harness and observe orthogonal outcomes.

This is the same step-atomicity concern flagged across the V18 slice (`V18c` too large, `V18b` gate concerns): the plan's convention treats a leaf as the smallest feature that can ship and be tested independently, and a leaf with two Ships-when arms over two unrelated code paths exceeds that unit.

## Consequence

**Severity:** advisory

The leaf is implementable as written, but the bundle forces an agent to hold two unrelated code paths — a green H4a acceptance run at the bumped pin and a simulated-red revert sequence re-running the `V18c` static-gate doubles at the prior pin — in one TDD pass, each with its own Ships-when arm and `-T` scenario. The widened critique surface and the dual done-signal make the leaf harder to land and verify coherently than either obligation alone.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `8af3204` (2026-06-11), `dc98f78` (2026-06-17)
**History:** The plan corpus is git-tracked. `git log --follow -- docs/plan_topics/V18d-version-bump-acceptance.md` shows the leaf was created at `8af3204` (2026-06-11, "resolve 'V18c bundles static gates with runtime-evidence acceptance gate'") — itself a split that carved the acceptance gate out of `V18c`. `git show 8af3204:` confirms the leaf was born titled "runtime-evidence acceptance gate and revert path" and bundled both concerns from inception, but the revert path was then prose-only in Adds and a single Ships-when sentence, with no Tests bullet — i.e. not yet an independently-verifiable obligation. The bundling tipped into a step-atomicity defect at `dc98f78` (2026-06-17, "resolve 'revert/rollback path has no verification gate'"), whose diff added the dedicated revert-verification Tests bullet to both `V18d` and `V18d-T`, plus the second Ships-when arm gating the revert path's completion. That fix legitimately closed the revert path's verification gap; its side effect was a leaf with two fully-verified, orthogonal obligations. The durable fix is therefore to extract the now-verifiable revert path into its own leaf, not to revert `dc98f78`.

## Solution Space

**Shape:** single

### Recommendation

Extract the revert/rollback path from `V18d` into a new paired leaf (suggested ID `V18e` / `V18e-T`; the implementer picks the final ID per the plan's leaf-ID scheme), leaving `V18d` scoped to the runtime-evidence acceptance gate alone.

- In `docs/plan_topics/V18d-version-bump-acceptance.md`: remove the revert-path content from **Adds** (the "and the revert path that restores the prior pin…" clause through the re-run-`V18c`-gates / widen-the-revert text), the second **Tests** bullet ("Revert / rollback verification …"), and the second **Ships when** arm (the "and the revert-verification assertion above observes…" clause). Retain the acceptance-gate Adds, its Tests bullet, and the acceptance-gate Ships-when arm. The `H4a`-fidelity bounding note stays with the acceptance gate.
- In `docs/plan_topics/V18d-T-version-bump-acceptance.md`: remove the second Tests bullet ("Revert / rollback verification …"), leaving the acceptance-gate test.
- Create `docs/plan_topics/<new>-version-bump-revert.md` and its `-T` partner holding the revert-path obligation: the merge-blocking MUST of `version-bump-triggers.md` ("a bump whose runtime-evidence run is red MUST NOT be merged at the candidate pin"), the restore of step 4's edit (the `host-prerequisites.md#pi-sdk-pin` literal and the four `@earendil-works/*` `peerDependencies` entries), the re-run of `V18c`'s static gates against the restored prior pin, and the widen-on-red behaviour with its complete-revert / incomplete-revert signals. Its **Deps** name `V18d` and `V18c` (carrying their `-T` partners per the pickability predicate).
- In `docs/plan.md` §Vertical slices, add the new leaf to the V18 — Build-time SDK gates listing after `V18d`.
- In `docs/plan_topics/coverage-matrix.md`, the `cka-19` row currently maps both the acceptance-gate MUST (output (c)) and the "a red runtime-evidence run MUST NOT be merged at the candidate pin" revert-blocking MUST to `V18d`. Re-point the merge-blocking facet to the new revert-path leaf (the acceptance-gate facet stays on `V18d`), reconciling the closing-leaf cell with the split so the merge-blocking MUST has its closing leaf.
- In `docs/plan_topics/H5b-warn-only-canary.md`, if the new revert-path leaf closes the merge-blocking MUST (a coverage-producing leaf), add it to the **Deps** coverage-producing set per `conventions.md` §REQ-ID discipline (*Transitive-completeness plan-maintenance*).

The `V18c` static-gate definitions the revert path re-runs stay owned by `V18c` / `V18c-T` and are consumed as doubles by the new leaf, exactly as `V18d` consumes them today; do not redefine them in the new leaf.
