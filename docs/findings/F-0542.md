## Finding

`V10a`'s **Adds** names two distinct discovery-collision regimes: cross-format-collision (same-priority candidates derive one slash name; none register; `loom/load/cross-format-collision` fires — `DISC-4`) and cross-source-shadow (the same slash name resolves from two sources of *different* priority; the higher-priority source's loom registers, the lower-priority one does not, and `loom/load/cross-source-shadow` is emitted naming both paths). The five-source priority order (CLI → Settings → Project → Packages → Global) is part of the same Adds clause.

`V10a`'s **Tests** (and the paired `V10a-T` tests task) assert only the collision arm — `DISC-4`. No test bullet exercises cross-source-shadow resolution or the five-source priority order. They are exercised only incidentally, if at all, by the `DISC-1`/`DISC-2`/settings-integration bullets.

This obligation is doubly unguarded. The spec's **Source priority (high to low)** block (in `discovery-sources.md`, between `DISC-1` and `DISC-2`) carries no numbered REQ-ID, so it has no row in `coverage-matrix.md` (which maps `DISC-1…DISC-3` and `DISC-4` to `V10a` but nothing for source priority / `cross-source-shadow`). The `H5a` per-REQ-ID citing-test scan therefore cannot detect a missing citing test for it. With neither a leaf-local asserting test nor a coverage-matrix gate, a wrong priority table — or a shadow path that registers the wrong loom or emits the wrong (or no) diagnostic — ships green.

The defect is original to the leaf, not introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

Add a cross-source-shadow / priority-ordering asserting test. Land it first on `docs/plan_topics/V10a-T-discovery-walk.md`'s **Tests** field (the tests task), then mirror the same bullet into `docs/plan_topics/V10a-discovery-walk.md`'s **Tests** field, per the per-phase TDD ritual.

The new test bullet must assert two things from `discovery-sources.md`'s **Source priority (high to low)** block:

- A fixture placing the same slash name in two sources at different priority levels (e.g. Settings and Global) asserts the higher-priority source's loom registers, the lower-priority one does **not** register, and `loom/load/cross-source-shadow` is emitted naming both paths (the `'<higher>' wins over '<lower>'` message template; no structured `details` payload). Note: the lower loom is shadowed *with* a diagnostic — it is not silent. (The original finding's "no diagnostic" wording is incorrect; the spec mandates the `cross-source-shadow` code.)
- A fixture exercising the full five-source priority order (CLI → Settings → Project → Packages → Global) asserts the winning loom for a name present in all five is the CLI entry, and the documented ordering is observable end-to-end (a wrong priority table reds the test).

Update `V10a-T`'s and `V10a`'s **Ships when** so the asserting run covers the shadow/priority case alongside the existing collision/validity cases.

Edge cases the implementer must watch: settings/`--loom` entries that resolve to the same absolute path are deduplicated *before* shadow detection (not a shadow); the shadow rule is cross-source only, distinct from the intra-source `DISC-3` case-collision rule and the same-priority `DISC-4` cross-format rule.

## Consequence

**Severity:** correctness

The source-priority resolution and the `loom/load/cross-source-shadow` behaviour have no asserting test and no coverage-matrix REQ-ID, so no gate fires when the priority order is implemented wrong. Two reasonable implementers could order the five sources differently — or shadow the wrong loom, or emit no diagnostic — and every gate stays green; the wrong loom silently registers for a given slash name.
