## Problem

`V10a`'s **Adds** names the five-source discovery union with its priority order (CLI → Settings → Project → Packages → Global) and cross-source-shadow resolution, but the **Tests** fields of both `V10a` and `V10a-T` assert only the `DISC-4` cross-format-collision arm. No test bullet exercises cross-source-shadow (per `discovery-sources.md`'s **Source priority (high to low)** block: the higher-priority source's loom registers, the lower-priority one does not, and `loom/load/cross-source-shadow` is emitted naming both paths) or the five-source priority order. The spec's Source priority block carries no REQ-ID, so it has no `coverage-matrix.md` row and the `H5a` per-REQ-ID citing-test scan cannot detect the gap either. With neither a leaf-local asserting test nor a coverage-matrix gate, a wrong priority table — or a shadow that registers the wrong loom or emits the wrong diagnostic — ships green.

## Solution approach

Add a cross-source-shadow / priority-ordering asserting test bullet to `V10a-T`'s **Tests** field first, then mirror it into `V10a`'s **Tests** field, per the paired-leaf TDD ritual. The bullet must assert both the shadow behaviour from `discovery-sources.md`'s **Source priority (high to low)** block (higher-priority loom registers, lower-priority one does not, and `loom/load/cross-source-shadow` is emitted naming both paths) and that the five-source priority order CLI → Settings → Project → Packages → Global is observable end-to-end. Extend both leaves' **Ships when** so the asserting run covers the shadow/priority case alongside the existing collision/validity cases.
