## Problem

The `loom/runtime/internal-error` row's *Trigger* column in `code-registry-runtime.md` populates the `hint` field with the underlying `error.stack`, rendering `"<no stack available>"` only when `error.stack` is falsy. That recipe presumes an `Error`-shaped caught value, but this surface catches at a boundary where a JavaScript producer may throw any value: the caught value may be a non-object, or an object whose own `.stack` is a truthy non-string (object, array, number) or a throwing getter. A truthy non-string `.stack` has no defined rendering, and a throwing `.stack` getter escapes the catch site entirely — breaching NOCEIL-3 on the very surface whose purpose is to absorb unanticipated throws. The §6 *Underlying-error coercion* rule in `placeholder-rendering-b.md#underlying-error-coercion` does not cover this: it is scoped to message strings (`v.message`) and never touches the stack extraction.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `f8f4813` (2026-05-05) — *"pi-loom plan: resolve 'Panic routing has no defined path for unexpected interpreter exceptions'"* — first added the `loom/runtime/internal-error` row to the diagnostics registry, with the literal stack-extraction phrasing "`hint` carries the underlying `error.stack` for operator triage (rendered as `\"<no stack available>\"` when `error.stack` is falsy)" present from the row's first appearance. The Error-shape assumption on `.stack` is therefore present-since-inception at the row level; no coercion rule yet existed in §6.
- `bf85092` (2026-06-06) — *"pi-loom spec: resolve 'non-Error-throw coercion + session_start pi.getCommands() failure contract'"* — introduced the §6 *Underlying-error coercion* sub-rule in `placeholder-rendering-b.md` and the matching `code-registry-runtime.md` preamble. The coercion repaired the **message-carrying** paths (the `<error.message>` placeholder, the row's `message` field, and any `hint` carrying a throw's *message*) but is scoped exclusively to `v.message` extraction; it does not reach the row's `.stack` extraction and the row's literal `error.stack` falsy-only rendering was left in place.

**History:** Present-since-inception at the row level (f8f4813); a later, narrower coercion pass (bf85092) covered the message half of the same surface but left the stack half outside any coercion or defensive-read rule. The remaining defect is the residue of that incomplete pass, not a regression.

## Solution approach

Rewrite the `hint` stack-extraction clause in the `loom/runtime/internal-error` row's *Trigger* column so the read is shape-tolerant and throw-tolerant: `hint` carries the underlying `.stack` only when the caught value is an object whose own `.stack` is a string, and renders the literal `"<no stack available>"` in every other case — non-object throws, absent or non-string `.stack`, and a `.stack` read that itself throws. State that the read is defensive so a throwing `.stack` getter does not escape the catch site. The existing falsy-case wording is subsumed by the non-string branch.

## Solution constraints

- Do not widen §6's *Underlying-error coercion* (`placeholder-rendering-b.md#underlying-error-coercion`) to cover stacks; it remains scoped to message strings, and the stack-tolerant rule is authored on the row.
