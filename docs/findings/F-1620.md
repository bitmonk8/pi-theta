## Finding

The `loom/runtime/internal-error` row in `code-registry-runtime.md` populates the diagnostic's `hint` field as the underlying `error.stack` "for operator triage (rendered as `\"<no stack available>\"` when `error.stack` is falsy)." That recipe presumes the caught value is `Error`-shaped, but the runtime catches at a boundary where the JavaScript producer is free to `throw <any value>`: the caught `v` may be a string, `null`, a Proxy, or an object whose own `.stack` is non-string (an object, an array, a number) or a throwing getter. The falsy fallback only handles `undefined` / `null` / `""` / `0`; a truthy non-string `.stack` has no defined rendering, and a throwing `.stack` getter escapes the catch site entirely — defeating the very surface whose contract (per `error-model.md` and NOCEIL-3) is to absorb every unanticipated throw.

The §6 *Underlying-error coercion* sub-rule in `placeholder-rendering-b.md` does not patch this hole. That rule is scoped to *message* strings — it binds `<error.message>` / `<dispose error first line>` placeholders and runs against `v.message`, not `v.stack` — so the only existing coercion path on the page never touches the stack extraction. The runtime-table preamble's "Wherever a row below carries a caught throw's error message … in a `message` / `hint` field" qualifier likewise routes only message-carrying values through §6; a stack-carrying `hint` is not in scope.

The defect is narrower than the umbrella claim that `error.message` / `error.stack` extraction lacks throw-domain bounding (the message half has the §6 coercion, the row's literal `error.message` phrasing notwithstanding); it specifically isolates the stack-extraction half, where no coercion or defensive-read rule exists on the page or on the row.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `f8f4813` (2026-05-05) — *"pi-loom plan: resolve 'Panic routing has no defined path for unexpected interpreter exceptions'"* — first added the `loom/runtime/internal-error` row to the diagnostics registry, with the literal stack-extraction phrasing "`hint` carries the underlying `error.stack` for operator triage (rendered as `\"<no stack available>\"` when `error.stack` is falsy)" present from the row's first appearance. The Error-shape assumption on `.stack` is therefore present-since-inception at the row level; no coercion rule yet existed in §6.
- `bf85092` (2026-06-06) — *"pi-loom spec: resolve 'non-Error-throw coercion + session_start pi.getCommands() failure contract'"* — introduced the §6 *Underlying-error coercion* sub-rule in `placeholder-rendering-b.md` and the matching `code-registry-runtime.md` preamble. The coercion repaired the **message-carrying** paths (the `<error.message>` placeholder, the row's `message` field, and any `hint` carrying a throw's *message*) but is scoped exclusively to `v.message` extraction; it does not reach the row's `.stack` extraction and the row's literal `error.stack` falsy-only rendering was left in place.

**History:** Present-since-inception at the row level (f8f4813); a later, narrower coercion pass (bf85092) covered the message half of the same surface but left the stack half outside any coercion or defensive-read rule. The remaining defect is the residue of that incomplete pass, not a regression.

## Solution Space

**Shape:** single

### Recommendation

Tighten the `loom/runtime/internal-error` row's *Trigger* column so the `hint` field's stack extraction is shape-tolerant and throw-tolerant on its own terms (the §6 coercion stays scoped to message strings; do not widen it to cover stacks).

Replace the existing "rendered as `\"<no stack available>\"` when `error.stack` is falsy" clause with a rule of the form: the `hint` is the underlying `.stack` only when the caught value `v` is an object whose own `.stack` is a string; in every other case (`v` is not an object, `.stack` is absent or non-string, or reading `.stack` throws), `hint` renders the literal `"<no stack available>"`. State explicitly that the read is defensive — a throwing `.stack` getter MUST NOT escape the catch site and falls to the `"<no stack available>"` rendering. The row's existing falsy-case wording is subsumed by the non-string branch (a falsy non-string `.stack` is still non-string).

Edge cases the implementer must handle:

- Non-object throws (string, `null`, number, symbol): no `.stack` access is attempted; `hint` is `"<no stack available>"`.
- Proxy throws with throwing `.stack` traps: caught and rendered `"<no stack available>"`; do not re-enter the catch site recursively.
- A truthy non-string `.stack` (object, array, number): treated identically to absent — `"<no stack available>"`. The existing `JSON.stringify` / `String(...)` temptation is rejected to keep the field bounded and free of operator-confusing renderings.

Keep §6's *Underlying-error coercion* scope unchanged. The stack-tolerant rule is authored on the row, not as a §6 widening, because §6's contract is "produce the underlying string used by message-family placeholders," which is a different shape than "extract an optional stack-trace field."

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads `error.stack` literally and emits `[object Object]` or a stringified array in `hint`; another guards the read and renders `<no stack available>`. A third hits the throwing-getter case and lets the throw escape the `internal-error` catch site, breaching NOCEIL-3 on the very surface whose purpose is to absorb unanticipated throws. The blast radius is narrow (one field on one row) but the surface is the runtime's last-resort defence, so the failure mode is structurally significant.
