## Finding

`V14b` models real Pi's model-driven parallel-tool mode: when the model issues several `tool_use` blocks in one assistant message, the runtime awaits every sibling to settle (`Promise.allSettled`) before constructing the next user turn and lowers each sibling's outcome independently. The leaf's own Adds is explicit that loom performs no batching or settle-barrier of its own — the concurrent-execution, settle-all-before-next-turn, and independent-per-sibling-lowering properties are entirely Pi-side, an unpinned consumption posture with no typed contract and no build-time SDK-surface signal (`tool-calls.md#parallel-tool-batch-delivery-presupposition`).

`V14b`'s green test drives this batch through the in-process session double, so it proves only the double's *modelled* behaviour. The spec routes real-Pi detection to a SHOULD-level editorial audit — version-bump checklist item `(an)` (`version-bump-step2.md#bump-checklist-parallel-tool-batch-delivery`), surfaced in `V18c`. That audit is a manual per-bump confirmation, not a running step; it never drives a parallel batch.

The only place a parallel batch could be observed against a live host is the manual real-host smoke (`real-host-smoke-gate.md`), which reuses `H7a`'s committed multi-feature fixture `.loom`. But that fixture's pipeline is "typed query → tool loop → code-tool invoke → schema lowering/validation → binder → cancellation" — a sequential tool *loop* (`V13c`), not a model-driven parallel `tool_use` batch (`V14b`). `V14b` is in `H7a`'s Deps for permitted-code provenance, but its distinctive batch shape is not driven by the fixture. The net result: the runtime's one fully Pi-side correctness dependency on this path is exercised by neither the offline double-backed test nor the real-host smoke — only by a manual editorial confirmation whose method is unspecified.

Commit `2123dc9` ("fix F-0560 — surface V14b parallel-tool-mode presupposition via new bump checklist item (an)") added the unpinned-consumption-posture framing in `tool-calls.md` and `V14b`, and routed detection to the SHOULD-level editorial item `(an)` in `version-bump-step2.md`; it documented the gap but did not add a runtime real-Pi observation step, so the verification gap was consciously accepted rather than closed.

## Solution Space

**Shape:** single

### Recommendation

Extend `H7a`'s committed multi-feature fixture `.loom` so the integrated pipeline includes a model-driven parallel `tool_use` batch mixing one succeeding and one failing sibling, and add the resulting turns to the committed golden transcript and the failing sibling's `isError: true` tool-result codes to the committed golden diagnostics list and the permitted-code list (per `H7a`'s per-Deps-slice provenance obligation). Because the manual real-host smoke (`real-host-smoke-gate.md`) reuses `H7a`'s committed fixture against a live Pi host, driving the parallel batch through that fixture makes the batch-delivery posture observed against real Pi on every smoke run — at minimum each Pi version bump under trigger (1) (the MUST-level trigger).

`V14b` is already listed in `H7a`'s Deps, and `H4b` category (b) already supplies mixed-success parallel `tool_use`-batch scripting (the `V14b` vector), so the addition is constructible against the in-process double for `H7a`'s own `npm test` gate as well. In `V14b`, add a one-line cross-reference stating that real-Pi observation of the parallel-batch posture occurs through `H7a`'s smoke fixture (alongside the existing `(an)` editorial route), so the leaf no longer reads as if editorial review is the sole detection path.

The spec is read-only for this fix: item `(an)` stays SHOULD-level per `version-bump-step2.md`, and the consumption-posture framing in `tool-calls.md` is unchanged. Escalating `(an)` to a mandatory runtime gate would contradict the spec's deliberate "no typed contract → editorial audit" footing and is out of scope absent a separate spec-drift escalation.

## Consequence

**Severity:** advisory

The parallel-batch concurrent/settle-all/independent-lowering posture is the runtime's only purely Pi-side correctness dependency on this path, yet no running step (offline or real-host) ever drives a parallel batch against real Pi. If a Pi minor constructed the next user turn before every sibling settled, or collapsed a failing sibling instead of lowering it independently, the lowering rules would silently break with no test signal — only the SHOULD-level manual editorial confirmation `(an)` covers it, and the real-host smoke fixture does not drive a parallel batch, so the divergence could reach `main` undetected.
