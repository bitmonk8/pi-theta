## Finding

PIC-53 (Untyped-query `Ok(string)` trailing-turn extraction) defines the success value as "the `text` content of every `assistant` message in the final turn, concatenated in chronological order with a single `\n` separator between successive assistant messages." The *final turn* is defined immediately above as the trailing `user` message plus **every subsequent `assistant` / `toolResult` / `custom` message** through the end of the list. Whenever the model uses a tool mid-turn, two or more `assistant` messages are separated by an interposed `toolResult` (or `custom`) message, so "successive assistant messages" admits two divergent readings:

- **(a) filter-then-join** — drop all non-`assistant` messages, then place one `\n` between each adjacent pair in the resulting assistant-only list (so two `assistant` messages separated only by a `toolResult` are joined by exactly one `\n`).
- **(b) directly-adjacent only** — "successive" requires adjacency in the *unfiltered* message list, so a single `\n` is emitted only between two `assistant` messages with nothing between them; the separator for assistant messages split by a `toolResult` is then left undefined (no separator, or unspecified).

The two readings produce different `Ok(string)` bytes for any untyped query involving mid-turn tool use. The clause claims to be "identical to the `[assistant]` body selection in [Compact-transcript format] item 4," but that reference does not disambiguate the separator: item 4 governs the body *within a single `assistant` message*, while the inter-message separator in the compact transcript is driven by per-message role-prefixed lines and the turn delimiter (item 2) — neither of which is the omit-tool-messages-then-join behaviour PIC-53 needs. The intended reading is almost certainly (a), but the spec never states it.

The same defect reaches subagent mode: PIC-43 (`agent_end` query-result extraction) assembles its string "by the same assistant-text rule the prompt-mode driver pins above … concatenated in chronological order with a single `\n` separator," so it inherits the ambiguity by reference. Pinning the prompt-mode wording resolves both sites.

## Solution Space

**Shape:** single

### Recommendation

In PIC-53, replace "with a single `\n` separator between successive assistant messages" with wording that fixes the filter-then-join reading explicitly: the runtime first selects every `assistant` message in the final turn (discarding interposed `toolResult` / `custom` messages), then joins their concatenated `text` bodies with a single `\n` between each adjacent pair in that assistant-only sequence. State that interposed tool/custom messages contribute no bytes and do not affect the separator count, so two assistant messages split only by a `toolResult` are joined by exactly one `\n`. Keep the existing carve-outs (the `thinking` array and all `toolCalls` entries omitted; a final turn with no assistant text yields the empty string).

Because PIC-43 assembles its subagent-mode string "by the same assistant-text rule the prompt-mode driver pins above," tightening the PIC-53 sentence is sufficient — PIC-43 needs no separate edit, but the implementer should confirm the cross-reference still reads coherently after the rewrite. Drop or soften the "identical to the `[assistant]` body selection in Compact-transcript format item 4" claim if the new wording makes the reference misleading (item 4 governs intra-message body, not the inter-message separator).

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the `Ok(string)` value returned to loom code (and surfaced to the user) for any untyped query whose final turn contains mid-turn tool use: one collapses tool-separated assistant fragments into a single `\n`-joined string, another emits a different separator or drops the join entirely. The divergence is silent — both pass any conformance vector that omits interleaved tool messages — and is duplicated across prompt and subagent modes.
