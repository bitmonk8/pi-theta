## Problem

PIC-53 (`#pic-53`) defines the untyped-query `Ok(string)` value as the `text` of every `assistant` message in the final turn "concatenated in chronological order with a single `\n` separator between successive assistant messages," but the final turn includes interposed `toolResult` / `custom` messages. When the model uses a tool mid-turn, "successive assistant messages" admits two divergent readings — filter-then-join (one `\n` between each adjacent pair in the assistant-only sub-list) versus directly-adjacent-only (separator left undefined for assistant messages split by a `toolResult`) — which produce different output bytes for any query involving mid-turn tool use. The cited equivalence to Compact-transcript item 4 does not disambiguate it: item 4 governs the body within a single `assistant` message, not the inter-message separator. PIC-43 (`#pic-43`) inherits the same ambiguity by reference for subagent mode.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `024215d` — "pi-loom spec: resolve \"untyped-query Ok(string) extraction mechanism (prompt + subagent)\"" (2026-06-01)
**History:** `git log -S'successive assistant messages'` and `git log -S'separator between successive'` both bottom out at `024215d`, the commit that first specified the `<a id="untyped-query-ok-extraction">` extraction mechanism (then in `pi-integration-contract.md`, later relocated to `conversation-drive.md` by the size-cap split `f5e89f4`). `git show 024215d` confirms the prior text said only "The accumulated assistant text from the final turn … is the `Ok(string)` value" with no separator detail; the ambiguous "single `\n` separator between successive assistant messages" phrasing entered verbatim with that commit and has been carried unchanged through subsequent relocations. The defect was present from the moment the separator was first specified.

## Solution approach

Rewrite PIC-53's separator clause (`#pic-53`) to state the filter-then-join reading explicitly: interposed `toolResult` / `custom` messages are discarded before the join, contribute no bytes, and do not affect the separator count, so two `assistant` messages split only by a `toolResult` are joined by exactly one `\n`. If the rewrite makes the "identical to the `[assistant]` body selection in Compact-transcript format item 4" reference (`#compact-transcript-format-normative`) misleading — item 4 governs intra-message body, not the inter-message separator — clarify or soften that claim.

## Solution constraints

- PIC-43 (`#pic-43`) derives its separator behaviour from PIC-53 by reference and the "same final-assistant-text rule on both sides" invariant must hold — do not author a separate separator rule in PIC-43.
- The rewrite must not drop PIC-53's existing carve-outs: the `thinking` array and all `toolCalls` entries omitted, and a final turn with no assistant text yielding the empty string.
