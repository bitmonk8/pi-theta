## Finding

The spec pins a four-row provider seed-field table at [`provider-error-mapping.md` §Provider seed-field mapping](../../docs/spec_topics/pi-integration-contract/provider-error-mapping.md): `openai-completions` places the seed under `seed`, `mistral` under `random_seed`, and `anthropic-messages` / `amazon-bedrock` receive **no** seed key. [`binder-inference.md` §Binder inference call](../../docs/spec_topics/pi-integration-contract/binder-inference.md) restates the runtime behaviour as two distinct branches: "the fixed seed … is placed under that field name … providers whose row omits the seed field receive no seed key." `V9j` lists "the per-provider seed-field table" in its **Adds**, but the closing assertion in both paired files reads only `complete()` is issued with the forced-tool envelope, `temperature:0`, and **the mapped seed field** (singular).

A single generic assertion exercises at most one provider row. It covers neither the per-provider field-*name* mapping (e.g. a runtime that hardcodes `seed` and ships `mistral` under the wrong key) nor the omit branch (a runtime that incorrectly attaches a seed key to `anthropic-messages` / `amazon-bedrock`). Both defects pass green if the exemplar happens to be `openai-completions`.

This gap is not closed elsewhere. `V18c`'s provider seed-field fixture asserts the **table constant** (it reddens when a provider's seed field is renamed, retyped, or moved between the supporting and non-supporting sets) and the `Api`-coverage assertion guards that every pi-ai `Api` value has a row — but neither exercises the runtime `complete()` payload that `V9j` builds from that table. The runtime placement is `V9j`'s own obligation, and only `V9j` asserts it.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V9j-binder-inference-provider-mapping.md` and `docs/plan_topics/V9j-T-binder-inference-provider-mapping.md`, replace the singular seed-field clause in the `complete()` Tests bullet — currently `complete() is issued with the forced-tool envelope, temperature:0, and the mapped seed field` — so the assertion covers the seed-field handling for each row of the spec's provider seed-field table rather than one provider. The asserted behaviour must distinguish both branches the spec defines:

- The seed is placed under the row's field name for each seed-supporting provider — `openai-completions` → `seed`, `mistral` → `random_seed`.
- No seed key is attached for each non-supporting provider — `anthropic-messages` and `amazon-bedrock`.

Keep the two paired files' Tests bullets in lockstep (they currently carry identical text). The seed-field source of truth stays the spec table at [`provider-error-mapping.md` §Provider seed-field mapping](../../docs/spec_topics/pi-integration-contract/provider-error-mapping.md); do not duplicate or restate the mapping in the spec — this fix is internal to the two plan leaves. Leave `V18c`/`V18c-T` untouched: they cover the table constant and `Api`-coverage, a separate obligation from the runtime payload `V9j` asserts.

## Consequence

**Severity:** correctness

The seed pins binder determinism per provider; a single-provider assertion lets a wrong field name for a non-exemplar provider, or a stray seed key on a provider whose row omits it, ship green. Two reasonable implementers diverge — one tests all four rows, one tests only the exemplar — and the second produces a binder that is non-deterministic (or rejected) for the untested providers while `npm test` stays green.
