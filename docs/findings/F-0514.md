## Finding

`V5e` enumerates the five ceiling-#4 enforcement sites and asserts the per-boundary routing *decision* in isolation against the harness, then delegates the live wrapping of a depth-6 breach to a named site-owner for each loom-code-surfacing row: typed-query response → `ValidationError` at `V13c`, code-driven tool args → `CodeToolError` at `V14e`, `params`/`invoke<T>` return → `InvokeInfraError` at `V15j`, and the slash-load `params` cross-route at `V4e`. Four of the five site-classes therefore have a downstream live-integration witness.

The fifth — **model-driven tool-call args → model feedback** — is the lone exception. `V5e`'s Tests bullet lumps it with the slash-load `params` row ("The model-driven and slash-load `params` rows produce no loom-code `Err` … and are decision-only here"), but the two are not alike. The slash-load row genuinely cross-routes to ceiling #3 and is correctly witnessed at `V4e`/`V11f`; the model-driven row has observable runtime behaviour the spec mandates: per `ceilings-3-and-4.md` (model-driven row) and `schema-subset.md` row #2, a depth-6 model-issued `tool_use` argument trips the loom-owned depth walk before AJV and is lowered to an `isError` tool-result fed back as the next user turn, with the round counting against `tool_loop.max_rounds`. No leaf in the corpus asserts that path: `grep` for `model-driven` across `plan_topics/` returns only `V5e` and `V14b` (parallel batch, not depth), and no ceiling-#4 / `maxDepth` witness leaf exercises a model-issued `tool_use` depth breach.

The result is a coverage asymmetry: the model-driven row is asserted only as a routing *decision* in `V5e`'s isolated harness, with no leaf proving the live model-driven depth-breach → model-feedback path that the spec's audience-coverage invariant treats as observable to the model.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commit:** `3fa39a9` — "pi-loom plan: resolve \"V5e per-boundary routing test asserts destination error surfaces its Deps cannot reach\"" (2026-06-11)
**History:** The plan corpus is git-tracked. `V5e-depth-enforcement.md` was created in `c6a664e` (2026-06-10) with a Tests bullet that claimed `V5e` itself fires `maxDepth` "at each of the five sites" with correct routing — an assertion `V5e` could not reach. `git log -S "decision-only"` / `git log -S "are decision-only here"` on the leaf both return exactly one commit, `3fa39a9`. That commit restructured `V5e` into a decision-only isolated-harness test and delegated the live carriers, naming witnesses for the four loom-code-surfacing rows (`V13c`, then-`V14a`/`V15a`, later split into the dedicated `V14e`/`V15j` carriers, and `V4e`) while relegating the model-driven row to "decision-only here" with no live witness named. The coverage gap entered with that delegation: before `3fa39a9` the model-driven row was nominally (if impossibly) covered by `V5e`'s own five-site claim; after it, the row alone among the five was left without a downstream home.

## Solution Space

**Shape:** multiple

### Option A — Add a live-carrier witness for the model-driven row

**Approach.** Give `V5e`'s model-driven tool-args ceiling-#4 routing row a named live carrier, analogous to `V14e` (code-driven) and `V15j` (`params`/`invoke<T>`-return). The natural home is `V13c` (the model tool-call loop) or a new `V14`-series sibling leaf.

**Plan edits.** Add a Tests bullet to the chosen leaf asserting that a depth-6 model-issued `tool_use` argument trips the loom-owned depth walk (`V5e`) before AJV and is lowered to an `isError` tool-result fed back as the next user turn, with no `QueryError` surfacing and the round counting against `tool_loop.max_rounds`. In `V5e`'s per-boundary Tests bullet, name that leaf as the model-driven row's live carrier and drop the model-driven row from the "decision-only here" clause (leaving only the slash-load `params` row there). If a `<new>` leaf is created, add its backtick-delimited leaf-ID to the relevant closing-leaf cell in `coverage-matrix.md`.

**Spec edits.** None — `ceilings-3-and-4.md` (model-driven row) and `schema-subset.md` (boundary #2) already define the behaviour.

**Pros.** Closes the path end-to-end; restores symmetry with the other four rows; satisfies the spec's audience-coverage invariant (observable to the model).

**Cons.** Adds a leaf/Tests bullet; the witness must drive a real model-issued `tool_use` through the loop harness.

**Risks.** The witness must exercise a live model-issued `tool_use` (not the isolated walk), which needs the harness's response-programming surface (`H4b`).

### Option B — State the decision-only rationale explicitly

**Approach.** Keep `V5e` decision-only for the model-driven row but document why no live carrier exists, instead of the bare "decision-only here".

**Plan edits.** Expand `V5e`'s model-driven clause to cite the spec reconciliation — the model-driven row is silent at the loom-code level, observable to the model via the fed-back tool-result and to the operator via ceiling #2 only if the loop later exhausts — and point to wherever the in-loop model feedback is exercised, if anywhere.

**Spec edits.** None.

**Pros.** Cheapest; no new leaf.

**Cons.** Leaves an observable, spec-mandated runtime behaviour with no asserting leaf; weaker than the four sibling rows and conflates the model-driven row with the slash-load row, which is decision-only for a different (cross-route) reason.

**Risks.** A future reader may treat the model-driven feedback as genuinely unobservable like the slash-load arm and never wire it.

### Recommendation

Take **Option A**. The model-driven row's feedback is an observable runtime behaviour the spec mandates — unlike the slash-load `params` row, which legitimately stays decision-only because it cross-routes to ceiling #3 (witnessed at `V4e`). Give the model-driven row a live witness in `V13c` or a new `V14`-series sibling. Edge cases the implementer must watch: do not move the slash-load `params` row out of "decision-only" (it is correctly decision-only); assert that no `QueryError` surfaces on the model-driven row and that the round still counts against `tool_loop.max_rounds`; and if the witness drives the model loop through the harness, it inherits the same harness-Deps concern flagged for `V5e`'s other "against the harness" tests.

## Consequence

**Severity:** correctness

The model-driven `tool_use` depth-breach → model-feedback path is mandated by the spec but has no closing leaf: `V5e` is decision-only for that row and no downstream leaf witnesses it. Two reasonable implementers would diverge — one builds the live model-driven depth-feedback test, another never wires the model-side arg-depth check at all — and the path can silently ship unimplemented, since the four sibling rows each have a live carrier that this one lacks.
