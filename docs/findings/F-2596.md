## Finding

`package-and-settings.md` § "Package discovery" lists five package roots and then specifies a single, flat enumeration rule that applies "within each root": every immediate child directory whose name does not begin with `@` is a candidate package, and every `@`-prefixed immediate child is a scope directory whose own immediate children are candidate packages. That rule is correct for the three npm-shaped roots (`.pi/npm/`, `node_modules/`, `~/.pi/agent/npm/`), where packages sit at depth 1 and scoped packages at depth 2 (covered by the one-level `@scope` unwrap).

It does not work for the two git roots. The roots are listed as `.pi/git/<host>/<path>/` (project) and `~/.pi/agent/git/<host>/<path>/` (global), where `<host>/<path>` is a multi-segment layout — Pi clones git packages to `<root>/<host>/<path>` per `@earendil-works/pi-coding-agent/docs/packages.md` § git ("Cloned to `~/.pi/agent/git/<host>/<path>` (global) or `.pi/git/<host>/<path>` (project)"). The `<path>` component mirrors the repository path (`user/repo` on GitHub, `group/subgroup/repo` on hosts with subgroups), so the actual package root — the directory holding `package.json` — sits two or more levels below `.pi/git/`. Under the flat rule, the immediate children of the git root are host directories (`github.com`, …), which are not packages and do not begin with `@`, so each is mis-classified as a candidate package whose `package.json` parse fails, and the walk never descends to the real clone directory. Git-installed loom packages are therefore silently undiscoverable.

The contradiction is internal: the same page commits to the multi-segment git layout in its roots list and to a depth-1 enumeration that cannot reach it, and the spec also commits to "the extension walks installed package roots itself; it does not delegate to Pi", so there is no fallback path that would rescue the git case.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** d6cbb37 — pi-loom spec: resolve "pi.looms is an extension-owned convention" (2026-05-04, Thomas Andersen)
**History:** Commit d6cbb37 authored the package-discovery section, introducing the multi-segment git roots (`.pi/git/<host>/<path>/`) and the flat "every immediate child" enumeration rule in the same change; the two were never reconciled, so git roots were unreachable from the section's first commit. Commit 475155c (2026-05-05) later extended the flat rule with `@scope` unwrapping for scoped npm packages — touching the enumeration rule without closing the git-depth gap — and f7a0dae (2026-06-08) deduplicated two contradictory statements of the rule, again leaving the git case unaddressed.

## Solution Space

**Shape:** single

### Recommendation

Split the candidate-enumeration rule by root kind in `package-and-settings.md`:

- **npm roots** (`.pi/npm/`, `node_modules/`, `~/.pi/agent/npm/`): keep the existing flat immediate-child rule plus the one-level `@scope` unwrap unchanged.
- **git roots** (`.pi/git/`, `~/.pi/agent/git/`): state a presence-driven descent. The extension traverses the `<host>/<path>` directory tree and treats the first directory along each path that contains a `package.json` as the candidate package (the clone directory); intervening host- and path-prefix directories that hold no `package.json` are traversed, not classified as packages. The `@`-scope-unwrap step does not apply under git roots — the git clone layout is not npm's scope layout. Termination: descent stops at the clone directory once its `package.json` is found; the walk does not recurse into a package's own subdirectories looking for nested packages.

Cross-reference `packages.md` § git as the authority for the `<host>/<path>` clone layout. State that the DISC-6 bounds apply unchanged to git descent — every `package.json` probe along the descent counts against `looms.scanPackagesMaxFiles` and the walk time counts against `looms.scanPackagesTimeoutMs`.

Edge cases the implementer must watch: hosts with subgroup hierarchies (e.g. GitLab `group/subgroup/repo`) yield variable-depth paths, so termination must key on `package.json` presence rather than a fixed segment count; and a git clone directory that ships no `package.json` (bare or non-package repo) contributes nothing, consistent with the existing parse-success gate.

## Consequence

**Severity:** correctness

Git-installed loom packages are unreachable: the walk classifies host directories as packages, their `package.json` probes fail, and descent into the clone directory never happens, so no looms from git roots are discovered. Two implementers also diverge — one may add an ad-hoc descent, another may follow the literal flat rule — producing different discovered loom sets for the same install.
