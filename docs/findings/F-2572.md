## Finding

PIC-12 (host-interfaces-services.md) makes the timing-primitive ban enforceable by promoting a static check to a normative conformance criterion: "A conforming implementation MUST carry a build-time check that rejects any **direct lexical reference** to those timing primitives outside the `WallClock` adapter … the presence of such a check and the references it rejects are the conformance criterion." The Clock bullet in implementation-notes.md restates the same "MUST carry a build-time check" wording and defers to PIC-12 for the criterion.

The verb "carry" does not state the operational obligation. An implementation can literally *contain* a check script (e.g. a grep test committed to the repo) that is never wired into the build or CI, never runs, and never fails anything — and still satisfy "MUST carry a build-time check" while providing zero enforcement. The criterion names *what the check must reject* (direct lexical references) and frees *how* it is realised (grep / lint / equivalent), but it never pins the one property that makes the gate meaningful: that the check is **executed on every build (or CI run) and fails the build when it finds a violation**. Two conformance reviewers can reasonably disagree on whether a present-but-unexecuted check conforms.

This is a clarity gap in the conformance criterion, not a reason to weaken the criterion. The check-existence requirement is the deliberate, mechanism-agnostic gate that makes the otherwise-untestable negative prohibition ("MUST NOT call `Date.now`, `performance.now`, … outside the adapter") observable; the underlying behavioural prohibition and the explicit freedom of mechanism are both sound and should be preserved. The defect is solely that "carry" leaves the enforcement cadence (must-run, build-failing) unspecified.

The "carry"/enforcement-cadence ambiguity entered the corpus with that promotion; `git log -S` for both "carry a build-time check" and "direct lexical reference" returns only this commit across both files.

## Solution Space

**Shape:** single

### Recommendation

In PIC-12 (the canonical criterion statement), replace "MUST carry a build-time check that rejects …" with wording that pins the enforcement cadence and effect: a conforming implementation MUST include a build-time check that **runs as part of every build / CI run and fails the build** when it finds a direct lexical reference to the banned timing primitives outside the `WallClock` adapter. Keep unchanged: (1) the behavioural prohibition (`MUST NOT call Date.now / performance.now / Date.prototype.getTime / global setTimeout / clearTimeout outside the adapter`) as the underlying requirement; (2) the free choice of mechanism (grep / lint / equivalent); (3) the "presence of such a check and the references it rejects are the conformance criterion" framing; (4) the direct-lexical-reference-only scope and the concession of indirect forms to plan-side review.

Mirror the same cadence wording in the implementation-notes.md Clock bullet so the two statements do not drift; the `homedir()` / `cwd()` env-var ban inherits the corrected criterion shape automatically because it already cites PIC-12 as canonical.

Do **not** demote the check to SHOULD / non-normative or restate the requirement as the bare "no ambient time primitive" property: that gate is the deliberate, mechanism-agnostic way to make the negative prohibition testable, and removing it re-opens the untestable-negative gap. Edge case the editor must respect: the `WallClock` adapter's own `wallNow()` → `Date.now()` and timer delegations are the single permitted exemption and must remain carved out of the check.

## Consequence

**Severity:** advisory

A present-but-unexecuted check satisfies the literal wording while enforcing nothing, so conformance reviewers can diverge on whether such an implementation passes the PIC-12 criterion. Loom's observable runtime behaviour is unaffected; the gap is confined to the conformance/CI-tooling obligation, so implementers can still produce a working, compliant runtime.
