## Finding

The *Wire serialisation* clause in `audit-wire-and-canary.md` pins a line-routing wire contract: each violation, infrastructure-failure, or canary record is one `\n`-terminated line on stdout whose first whitespace-separated token is the `audit/<class>/<family>/<symptom>` discriminator that a CI parser uses to route by leading-token prefix without parsing the rest of the line. The rule presupposes — but never states — that (i) each record's bytes reach stdout as a single atomic write so the four tab-delimited fields cannot be torn across two writes, and (ii) no other writer interleaves output between the discriminator token and the trailing newline of the same record.

The trailing harness-wiring epilogue then explicitly permits the audit to **run in-process to the harness** (e.g. as a Vitest test case whose `expect(...).toBe(0)` over the audit's exit code becomes the test's pass/fail). On that wiring the audit's stdout is the *same* file descriptor as the test runner's own stdout: a parallel Vitest reporter, a parallel worker test case's `console.log`, a concurrent diagnostic on the same process's stderr-mirrored-to-stdout sink, or a single record long enough to exceed Node's internal `process.stdout.write` chunking threshold can interleave between or split the bytes of one record, leaving lines on the wire that no longer satisfy the leading-token-then-three-tab-fields shape the routability obligation pins. The same gap exists on the child-process variant if the harness merges child stdout with its own reporter stream rather than capturing it into a dedicated buffer.

Without pinning the precondition (either as a stdout-integrity obligation on the implementation — a single atomic write per record plus an isolation guarantee from the harness wiring — or as a hard constraint requiring the audit to run in a child process with separately-captured stdout), the spec's leading-token routing contract is satisfiable only by accident on the in-process variant the same page admits.

## Solution Space

**Shape:** multiple

### Option A — Pin a stdout-integrity precondition that both wirings must satisfy

**Approach.** Extend the *Wire serialisation* clause with an explicit precondition: each emitted record MUST reach stdout via a single atomic write (a single `process.stdout.write(line)` call producing one `write(2)` syscall, no partial flushes), AND the harness wiring MUST guarantee that no other writer interleaves output between the start of one record's discriminator and the trailing `\n` of the same record on the byte stream a CI parser consumes. The in-process Vitest variant satisfies clause two by suppressing the test runner's reporter output on the audit's stdout (e.g. routing reporter output to a separate sink) or by isolating the audit's stdout writes behind a captured buffer the harness later replays in one piece; the child-process variant satisfies it by writing to its own captured stdout.

**Spec edits.**
- Insert the precondition sentence at the end of the *Wire serialisation* clause in `audit-wire-and-canary.md`, before the "Stderr MAY carry…" sentence.
- Extend the harness-wiring epilogue paragraph to state that both wiring choices MUST demonstrate the precondition is met by their concrete adoption (the implementation pins which mechanism it uses, on the same footing as the existing "single concrete adoption" rule already there).
- Add a `V18b` / `V18b-T` acceptance bullet: a fixture seeds a record whose total line length exceeds Node's typical stdout chunk threshold (e.g. ≥16 KiB by padding `proposed-resolution` with ASCII filler) and asserts the captured stdout still contains the line intact and routable by leading-token prefix; and (for the in-process variant) a fixture that emits one audit record while a parallel `console.log` runs and asserts no interleave occurs in the captured stream.

**Pros.** Preserves both wiring choices the existing epilogue already admits; minimal disruption to V18b's acceptance posture; the precondition is testable.

**Cons.** Shifts a non-trivial obligation onto the harness configuration (Vitest reporter routing or stdout capture wrapping) the spec previously treated as an implementation detail; the in-process fixture is mildly fragile (Node's stdout chunking threshold is not pinned by ECMA/Node and is environment-dependent).

**Risks.** An implementer satisfies the precondition by inspection rather than by fixture and ships a wiring that holds in CI but breaks on a developer's machine with a noisier reporter.

### Option B — Require a child process with separately-captured stdout

**Approach.** Narrow the harness-wiring epilogue to require the audit to run as a child process the harness spawns, with the child's stdout captured into a dedicated buffer (not merged with the harness's own reporter stream); withdraw the in-process Vitest variant for the audit's record-emission path. The fail-closed exit-code semantics already pinned in the epilogue carry over unchanged.

**Spec edits.**
- Rewrite the harness-wiring epilogue paragraph: replace "irrespective of whether the audit runs in-process to the harness … or as a child process …; either wiring is permitted" with "the audit MUST run as a child process the harness spawns, whose stdout is captured into a dedicated buffer disjoint from the harness's own reporter stream and from any other concurrent process's stdout".
- Delete the "For the in-process variant, the wrapping `expect(...)` MUST be unconditionally registered before the audit invocation" sentence, since the in-process variant no longer exists.
- Add a `V18b` / `V18b-T` acceptance bullet asserting the harness wiring spawns a child process and captures its stdout into a dedicated buffer.

**Pros.** Removes the in-process surface entirely, so the precondition holds by construction (a separately-captured child stdout cannot interleave with the harness's writes); spec stays short and the wire contract is satisfied without additional fixtures.

**Cons.** Removes a wiring option the spec previously admitted; child-process startup is slower than an in-process Vitest case (matters only for repeated test runs); some harness configurations (e.g. coverage instrumentation that prefers in-process) become incompatible.

**Risks.** A future contributor reintroduces an in-process wrapper for convenience without noticing the obligation moved.

### Recommendation

Adopt **Option A**. The existing epilogue already commits both wiring choices into the spec and downstream guidance (the related-finding `npm test` harness-wiring epilogue ... places it cross-shard); narrowing to child-process-only is a larger surface change for the same protective effect Option A obtains by pinning the precondition explicitly. The implementer must watch one edge case: the in-process fixture's line-length threshold should be at least 64 KiB to comfortably exceed Node's `process.stdout.write` high-watermark on every supported platform, and the harness MUST use a stdout-capture mechanism that observes raw byte writes (not a reporter that line-buffers or re-formats), otherwise the fixture will mask the very interleave it is intended to detect.

## Consequence

**Severity:** correctness

Two reasonable implementers would diverge: one reads the harness-wiring epilogue's "either wiring is permitted" sentence and ships the audit as an in-process Vitest case sharing stdout with the test runner; the other reads the *Wire serialisation* leading-token-routability obligation and ships a child-process wiring with captured stdout. Under sufficient log volume, large `proposed-resolution` payloads, or parallel Vitest worker activity, the in-process wiring intermittently emits records whose first token is not the discriminator, silently breaking the CI parser the wire contract was designed to support — observably wrong behaviour, surfaced as flake rather than a clean fail-closed red.
