## Finding

`H5b`'s `Tests.` bullet validates the warn-only canary by "seeding one gap of each kind, confirming the canary reports it, and confirming the run's exit status stays green, then restoring the seed." The trailing clause "then restoring the seed" is directionally ambiguous. It can be read two opposite ways: (a) revert the seeded gaps so the live corpus returns to its clean, gap-free pre-seed state — the intended cleanup — or (b) re-apply / leave the seed in place, which would leave the live corpus permanently carrying an unmapped REQ-ID, an uncited numbered REQ-ID, and an un-enumerated un-anchored MUST.

The seeded gaps in this test are introduced into the **live** surfaces (live spec REQ-ID set, live `spec_topics/**` MUST set, live test corpus) precisely so the canary's live-corpus reconciliation can be exercised — they are not placed under the dedicated test-fixtures root that `H5a`/`H5b` exclude from the scan. An implementer who follows reading (b) therefore leaves a real, permanent coverage gap in the live corpus.

The defect is original to the leaf, not introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

In `H5b`'s `Tests.` bullet, replace the trailing clause "then restoring the seed" with text that names the direction explicitly — reverting the seeded gaps so the corpus returns to its clean pre-seed state. Concretely, change the final phrase from:

> `…confirming the run's exit status stays green, then restoring the seed).`

to:

> `…confirming the run's exit status stays green, then reverting the seeded gaps so the corpus is restored to its clean pre-seed state).`

This makes the cleanup direction unambiguous and rules out leaving the live corpus permanently seeded. No other leaf carries the phrasing (`grep` over `docs/plan_topics/` and `docs/plan.md` matches only `H5b`), so the edit is fully local to this leaf.

## Consequence

**Severity:** advisory

An implementer who reads "restoring the seed" as re-applying the seed leaves the live corpus with one unmapped/uncited/un-anchored gap of each kind. The warn-only canary itself never reddens CI, so the gap is invisible there; it surfaces only later as a spurious red at `H6a`'s hard-fail live-corpus flip, costing a debugging cycle to trace back to the canary's own test fixture.
