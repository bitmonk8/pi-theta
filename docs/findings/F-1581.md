## Finding

The Appendix › Governance bullet in `session-model-and-appendix.md` reproduces an inline REQ-ID inventory verbatim from `governance.md`'s opening paragraph:

> *"the rules (GOV-1, GOV-3 through GOV-9, GOV-15, GOV-16, GOV-30, GOV-31; GOV-2, GOV-10, GOV-11, GOV-12, GOV-13, GOV-14 retired) that govern REQ-ID coining, anchoring, and retirement."*

That inventory contradicts its own authoritative source. The retired registry on `governance/anchor-scheme-and-retired.md` (the page the bullet itself names as canonical) lists the retired set as `{GOV-2, GOV-4, GOV-10, GOV-11, GOV-12, GOV-13, GOV-14, GOV-21}`, and the prefix tables register live rules through GOV-31. Against that authority the appendix bullet is wrong three ways:

- It lists **retired GOV-4 as live**, since "GOV-3 through GOV-9" spans GOV-4 (retired at commit `1d773d6`, replaced by GOV-24).
- It **omits GOV-4 and GOV-21 from the retired sub-list** (GOV-21 was split into GOV-25–GOV-29).
- It **omits the live rules GOV-19, GOV-20, GOV-22, GOV-24, GOV-25–GOV-29** entirely.

The bullet sits in the Appendix index — a primary navigation entry point — so a maintainer orienting from it receives an inventory that disagrees with the governance sub-pages it links to. The root problem is that the enumeration is a hand-maintained *duplicate* of the governance registry; it drifts every time a GOV ID is retired or added.

## Solution Space

**Shape:** single

### Recommendation

Delete the inline parenthetical enumeration from the Appendix › Governance bullet and replace it with a bare forward-link to `governance.md`, whose retirement registry (`governance/anchor-scheme-and-retired.md`) and prefix tables own the authoritative inventory. The Appendix is an index; it should describe the page's role ("spec-corpus governance: REQ-ID prefix table, retirement registry, corpus-direction and binding-scope rules") and link out, not maintain a third copy of the rule list. Dropping the duplicate is the durable fix — correcting the enumeration in place would re-create the same drift hazard the next time a GOV ID is retired or added.

Edge case for the fixer: the forward-link target (`governance.md`'s own opening paragraph) carries the identical stale enumeration — that is a separate finding; this edit should point at `governance.md` / its retirement-registry sub-page as the authority, not reproduce any count inline. The existing separate `GOV-17 / GOV-18` callout in the same bullet may stay or fold into the same link.

## Consequence

**Severity:** advisory

A spec maintainer or REQ-ID-mapping tool orienting from the Appendix index believes GOV-4 is live and that GOV-19/20/22/24–29 do not exist, contradicting the authoritative governance sub-pages. No runtime implementation diverges — the governance corpus targets maintainers, not implementers — but the navigation entry point is an unreliable inventory.
