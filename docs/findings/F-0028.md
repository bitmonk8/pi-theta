## Finding

The in-process test double that `H4a` establishes is canonically named the **Pi session double** ("in-process Pi session double"). `H4a` introduces it under that name in its Adds, and `H7a` and `V12a`/`V12a-T` all consume it as the "in-process Pi session double".

`V4c-terminal-outcomes.md` and `V4c-T-terminal-outcomes.md` instead call the same artefact the "`H4a` session double" in their `ERR-13` Tests bullets (two occurrences in each file), and the second `ERR-13` bullet additionally calls the harness the "`H4a` invocation harness" where the canonical harness reference is "`H4a` harness" / "`H4a` end-to-end harness". The `H4a` leaf-ID prefix does keep the reference unambiguous, but the artefact's canonical term ("Pi session double") never appears in either V4c file, so a corpus search for "Pi session double" does not surface V4c and a reader cannot confirm at a glance that the "`H4a` session double" is the same object H4a, H7a, and V12a describe.

This is a one-directional naming drift introduced into V4c/V4c-T after the canonical term was already established elsewhere; no equivalence statement links the two surface forms.

## Issue introduction

**Verdict:** introduced by a later plan-fix edit (single commit)
**Introducing commits:** `e8f0236` — "pi-loom plan: resolve \"V4c-T/V4c assert no-rollback over later-slice surfaces\"" (2026-06-11)
**History:** `git log -S'session double' --follow` over `docs/plan_topics/V4c-terminal-outcomes.md` and `docs/plan_topics/V4c-T-terminal-outcomes.md` shows the term "session double" first entered both files at `e8f0236`, and it entered already spelled "`H4a` session double" in the rewritten `ERR-13` bullet. At that commit `H4a`'s Adds already used the canonical "in-process Pi session double" (`git show e8f0236:docs/plan_topics/H4a-factory-shell-and-harness.md`), so the non-canonical form was introduced — not inherited from inception — when a prior no-rollback-scope fix re-pointed the `ERR-13` vectors at the double. The defect is confined to that single re-authoring; it was not present at plan inception (`c6a664e`) because the vectors did not reference the double at all then.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V4c-terminal-outcomes.md` and `docs/plan_topics/V4c-T-terminal-outcomes.md`, replace each occurrence of "`H4a` session double" (two per file, both in the `ERR-13` bullets) with the canonical "Pi session double". On the first occurrence in each file, the form "Pi session double (`H4a`)" preserves the leaf-ID traceability the current text carries. While editing the second `ERR-13` bullet, align "`H4a` invocation harness" to the canonical harness phrasing "`H4a` harness" so the bullet's two references to H4a artefacts read consistently with H4a/H7a.

The `ERR-12` "`H4a` harness" reference is already canonical (it names the harness, not the double) and needs no change.

There is no `terms.md` / occurrences-index file in the plan corpus, so the original suggestion to add an index occurrence entry has no target; the canonical term is owned by `H4a`'s Adds and that is the sole authority to align against.

## Consequence

**Severity:** advisory

A corpus search for the canonical "Pi session double" misses V4c/V4c-T, weakening cross-leaf discoverability of every site that exercises the H4a double. The `H4a` leaf-ID prefix keeps the reference unambiguous, so an implementer can still produce a working leaf; the cost is terminology drift and a search-traceability gap, not divergent behaviour.
