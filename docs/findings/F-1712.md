## Finding

The spec uses three phrases — **query template**, **query expression**, and bare **query** — for the same `@`...`` primitive without ever coining a canonical form. `overview.md` bolds "**query template**" as the coined term in the Query-and-Await section; `query.md` opens "The `@`...`` query template is the only construct that crosses code → model. It is an **expression**…" and `query/query-forms.md` then runs with "query expression" for the schema-inference walk ("A query expression searches *outward* through its enclosing AST…"). The feature page itself is titled "Query". `lexical.md`, `influences.md`, `grammar.md`, `implementation-notes.md`, `descriptions.md`, `frontmatter/frontmatter-fields-*.md`, and `diagnostics/code-registry-parse.md` each pick one of the three with no governing rule.

The glossary, which already hosts entries that read "Authors and implementers MUST use the term `callable set` … the bare phrases `tools: set`, `tool set` … are spec-prose synonyms to be avoided" and equivalent MUSTs for `schema slug` and `.loom callable`, has no entry for the most cross-cut primitive in the language. The `type sink` entry already leans on the term ("a query expression takes its schema from the sink"), but the term it leans on is undefined.

The consequence is procedural rather than semantic: the spec's own glossary convention — coin once, list the avoid-synonyms, anchor a future grep gate — is silently violated at exactly the construct that crosses code → model. A reader cannot tell whether "query expression" in `query-forms.md` is a deliberate sub-distinction (the `@`...`` form qua *expression* vs. qua *template body*) or unmanaged drift; the schema-inference walk happens to read more naturally with "query expression" precisely because it is reasoning about an AST node, which is suggestive but never made normative.

## Issue introduction

**Verdict:** present-since-inception (with synonym-three added at split)
**Introducing commits:**
- `53f5831` (2026-05-03) — *spec: rewrite query primitive, system prompts, and schema syntax*. Coined the `@`...`` primitive and introduced both **`query template`** (bold, in the Conceptual Model section) and the bare lowercase **`query expression`** (in "The response schema flows into the query expression from any of the following type contexts") in the same commit. The two synonyms were therefore in free variation from the moment the primitive was named.
- `b4fbfac` (2026-05-04) — *pi-loom spec: resolve "no central glossary"*. Created `spec_topics/glossary.md` with eight initial entries (including `type sink`, which already leans on "query expression"); the primitive itself was not given an entry, so the canonical-term convention the glossary later enshrined (`callable set`, `schema slug`, `.loom callable`) skipped over it.
- `fecb504` (2026-05-04) — *Split spec.md and plan.md into per-topic / per-phase files*. Split the query material into `spec_topics/query.md` (page title "Query") and added "the `@`...`` query template is the only construct…" as its opener. The split is what turned the bare page-title **Query** into a third reference form: the page title now reads as a candidate canonical term distinct from both `query template` and `query expression`.

**History:** The defect was present at the construct's coining (`53f5831`) — `query template` and `query expression` shipped together in the same commit, neither flagged as canonical and neither cross-referenced. The glossary-introduction commit (`b4fbfac`) had the opportunity to lock in a canonical term while writing the convention for analogous coinages but did not. The topic split (`fecb504`) introduced the third synonym (the page title `Query`). No later commit has consolidated the three; the original `spec-review` cycle's resolution `bb841d4` ("Glossary type sink defined query-only") refined the surrounding `type sink` entry but did not anchor the primitive it depends on.

## Solution Space

**Shape:** multiple

### Option A — Canonicalise on `query template`

**Approach.** Coin `query template` as the canonical term in the glossary, following the prose pattern of `callable set` / `schema slug` / `.loom callable` ("Authors and implementers MUST use this term; the synonyms `query expression`, bare `query` are spec-prose drift to be avoided"). Update `overview.md`'s bolded coining to point at the glossary entry. Sweep `query.md`, `query/query-forms.md`, and every other site to use `query template` (or, when the AST-node sense is being discussed, `query-template expression` — the longer form that preserves the sense distinction without re-introducing the bare synonym).

**Spec edits.**
- Add a glossary entry (alphabetised at `q`), modelled on the existing `schema slug` entry, naming the synonyms to avoid and citing `query.md` as the canonical page.
- Keep the page title `Query` (it is a feature/section name, not a term-of-art reference; analogous to the page `Tool Calls` whose canonical term is `callable set`).
- Replace bare `query expression` in `query/query-forms.md` with `query-template expression` in the two algorithm-walk sentences; replace bare `query` referring to the primitive with `query template` everywhere else it appears as a noun-phrase (not as a verb / not as a wire field name like `typed-query`).
- The wire surfaces and code identifiers — `loom/parse/typed-query-*` diagnostic codes, `__loom_respond_<slug>`, `tool_loop`, `respond_repair`, `QueryError`, `ContextOverflowError` — are not renamed.

**Pros.** Matches the form already bolded in `overview.md`; preserves the "template" reading that makes newline-trim/dedent/interpolation rules feel native; the `query-template expression` long form lets `query-forms.md` keep its AST-walk register without dropping back to a bare synonym.

**Cons.** Long; "query template" reads awkwardly in a few sites where the construct is being discussed *as an expression* rather than *as a template* (the schema-inference walk, the `?`-unwrap discussion, the discard rules); the `query-template expression` compound is a fourth phrase to add (mitigated by treating it as a sub-form of the canonical, declared in the glossary entry).

**Risks.** A reader of the schema-inference walk who reads "query-template expression" may infer that some other expression kind also serves as a sink — none does. The glossary entry must say so explicitly.

### Option B — Canonicalise on `query`

**Approach.** Coin bare `query` as the canonical term (matching the page title `Query` and the glossary's preference for short canonical handles, cf. `binder`, `loom`, `callable set`). Declare `query template` an avoid-synonym (with a carve-out for the few places where the *template-body* sense — the source text between backticks before interpolation — is what's actually being discussed); declare `query expression` an avoid-synonym across the board.

**Spec edits.**
- Add a glossary entry for `query`, defining it as "the `@`...`` construct that crosses code → model" with the synonyms-to-avoid list.
- Rewrite `overview.md`'s "The primitive that crosses code → model is the **query template**" to "The primitive that crosses code → model is the **query** — written `@`...``".
- Sweep `query/query-forms.md` to use bare `query` for the AST-node sense, freeing the awkward `query-template expression` compound.
- Keep `query template` only in passages whose subject is the template-body string itself (newline-trim, dedent, interpolation, escapes) — Vector commentary, the `loom/parse/empty-template` and `loom/parse/illegal-template-escape` diagnostic descriptions, and the dedent vector table; these are talking about the template, not the expression.

**Pros.** Short canonical handle, consistent with the rest of the glossary's coining style; the page title becomes the term-of-art reference; AST-walk prose reads naturally ("a query searches outward through its enclosing AST"); the residual `query template` retains a sharp, narrow meaning (the source string between backticks) instead of being free-variation with `query`.

**Cons.** Bare `query` collides with the common-English verb "to query" the model; existing `query-time tool loops`, `query template body`, `the rendered query template`, and `query template's static body` phrases are reworded, which is a wider mechanical sweep than Option A; reviewers familiar with the bolded `query template` coining will need to re-orient.

**Risks.** The "template body / template / expression" three-way sense split must be made explicit in the glossary, or the sweep will re-introduce drift the next time a contributor edits one of the dedent / escape / interpolation passages.

### Recommendation

**Adopt Option A — canonicalise on `query template`, with `query-template expression` as the declared sub-form for AST-node-sense passages.** The bolded coining already exists in `overview.md`, two of the three current synonyms collapse into it directly, and the wider compound (`query-template expression`) is a one-time addition documented in the glossary rather than a sweeping rewrite of every "rendered query template" / "query template body" phrase that already appears in the dedent and interpolation passages. Implementer edge cases to watch: keep wire identifiers (`__loom_respond_<slug>`, `typed-query` / `untyped-query` adjectives in `loom/parse/typed-query-*` codes and prose like "typed queries return `Err(QueryError…)`") unchanged — the rename governs the noun-phrase form of the primitive, not adjective compounds or wire identifiers; and ensure the glossary entry explicitly says `query-template expression` is the AST-node sub-form so the schema-inference walk in `query-forms.md` does not appear to coin a fourth synonym.

## Consequence

**Severity:** advisory

Implementers can still build — context disambiguates the three synonyms in every passage. The cost is procedural: the glossary convention that grounds the `callable set` / `schema slug` / `.loom callable` grep gates does not extend to the most-cited primitive in the spec, which leaves future drift uncaught and forces every contributor encountering the term to re-decide which form to write.
