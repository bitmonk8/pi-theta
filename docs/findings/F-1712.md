## Finding

The spec uses three phrases — **query template**, **query expression**, and bare **query** — for the same `@`...`` primitive without ever coining a canonical form. `overview.md` bolds "**query template**" as the coined term in the Query-and-Await section; `query.md` opens "The `@`...`` query template is the only construct that crosses code → model. It is an **expression**…" and `query/query-forms.md` then runs with "query expression" for the schema-inference walk ("A query expression searches *outward* through its enclosing AST…"). The feature page itself is titled "Query". `lexical.md`, `influences.md`, `grammar.md`, `implementation-notes.md`, `descriptions.md`, `frontmatter/frontmatter-fields-*.md`, and `diagnostics/code-registry-parse.md` each pick one of the three with no governing rule.

The glossary, which already hosts entries that read "Authors and implementers MUST use the term `callable set` … the bare phrases `tools: set`, `tool set` … are spec-prose synonyms to be avoided" and equivalent MUSTs for `schema slug` and `.loom callable`, has no entry for the most cross-cut primitive in the language. The `type sink` entry already leans on the term ("a query expression takes its schema from the sink"), but the term it leans on is undefined.

The consequence is procedural rather than semantic: the spec's own glossary convention — coin once, list the avoid-synonyms, anchor a future grep gate — is silently violated at exactly the construct that crosses code → model. A reader cannot tell whether "query expression" in `query-forms.md` is a deliberate sub-distinction (the `@`...`` form qua *expression* vs. qua *template body*) or unmanaged drift; the schema-inference walk happens to read more naturally with "query expression" precisely because it is reasoning about an AST node, which is suggestive but never made normative.

## Issue introduction

**Verdict:** present-since-inception (with synonym-three added at split)
**Introducing commits:**
- `53f5831` (2026-05-03) — *spec: rewrite query primitive, system prompts, and schema syntax*. Coined the `@`...`` primitive and introduced both **`query template`** (bold, in the Conceptual Model section) and the bare lowercase **`query expression`** (in "The response schema flows into the query expression from any of the following type contexts") in the same commit. The two synonyms were therefore in free variation from the moment the primitive was named.
- `b4fbfac` (2026-05-04) — *pi-loom spec: resolve "no central glossary"*. Created `spec_topics/glossary.md` with eight initial entries (including `type sink`, which already leans on "query expression"); the primitive itself was not given an entry, so the canonical-term convention the glossary later enshrined (`callable set`, `schema slug`, `.loom callable`) skipped over it.
- `fecb504` (2026-05-04) — *Split spec.md and plan.md into per-topic / per-phase files*. Split the query material into `spec_topics/query.md` (page title "Query") and added "the `@`...`` query template is the only construct…" as its opener. The split is what turned the bare page-title **Query** into a third reference form: the page title now reads as a candidate canonical term distinct from both `query template` and `query expression`.

**History:** The defect was present at the construct's coining (`53f5831`) — `query template` and `query expression` shipped together in the same commit, neither flagged as canonical and neither cross-referenced. The glossary-introduction commit (`b4fbfac`) had the opportunity to lock in a canonical term while writing the convention for analogous coinages but did not. The topic split (`fecb504`) introduced the third synonym (the page title `Query`). No later commit has consolidated the three; the original `spec-review` cycle's resolution `bb841d4` ("Glossary type sink defined query-only") refined the surrounding `type sink` entry but did not anchor the primitive it depends on.

## Solution Space

### Recommendation

Coin `query template` as the canonical term for the `@`...`` primitive and govern it through the glossary the same way `callable set`, `schema slug`, and `.loom callable` are governed.

Add a glossary entry alphabetised at `q`, modelled on the existing `schema slug` entry: state that authors and implementers MUST use `query template`, list `query expression` and bare `query` (as a noun-phrase for the primitive) as spec-prose synonyms to be avoided, and cite `query.md` as the canonical page. In that same entry, declare `query-template expression` as the AST-node sub-form of the canonical term, used when a passage is reasoning about the construct *as an expression* (the schema-inference walk, the `?`-unwrap discussion, the discard rules); state explicitly that no other expression kind serves as a schema sink, so the sub-form does not read as a fourth synonym.

Update `overview.md`'s bolded coining in the Query-and-Await section to point at the glossary entry. Keep the page title `Query` unchanged — it is a feature/section name, not a term-of-art reference (analogous to the `Tool Calls` page whose canonical term is `callable set`).

Sweep the remaining sites:
- In `query/query-forms.md`, replace bare `query expression` with `query-template expression` in the two algorithm-walk sentences.
- Everywhere else the primitive appears as a noun-phrase, replace bare `query` with `query template`.
- Do not touch verb uses ("to query the model") or wire/adjective forms.

Leave all wire surfaces and code identifiers unchanged: the `loom/parse/typed-query-*` diagnostic codes, the `typed-query` / `untyped-query` adjectives and prose like "typed queries return `Err(QueryError…)`", `__loom_respond_<slug>`, `tool_loop`, `respond_repair`, `QueryError`, and `ContextOverflowError`. The rename governs only the noun-phrase form of the primitive, not adjective compounds or wire identifiers.

## Consequence

**Severity:** advisory

Implementers can still build — context disambiguates the three synonyms in every passage. The cost is procedural: the glossary convention that grounds the `callable set` / `schema slug` / `.loom callable` grep gates does not extend to the most-cited primitive in the spec, which leaves future drift uncaught and forces every contributor encountering the term to re-decide which form to write.
