## Finding

`V7a` (and its paired `V7a-T`) carries a Multi-error Tests bullet asserting that the batched `Diagnostic[]` "is ordered by `(file, line, col)` across an entry `.loom` and ≥2 transitively-imported `.warp` modules (per implementation-notes.md — Static-resolution load pass IMPL area, **which aggregates each visited file's diagnostics into the entry loom's drain in this order**)." This bullet is internally inconsistent about the cross-file (`file`) ordering basis. "ordered by `(file, line, col)`" reads as a global sort keyed on the `file` field, but the trailing "aggregates each visited file's diagnostics into the entry loom's drain in this order" reads as per-file grouping in load-pass *visitation order* (entry loom first, then imports in walk order). Only the within-file `(line, col)` order is unambiguous.

The two readings diverge. The spec actually resolves the question, but the leaf paraphrase obscures it: `diagnostic-shape.md` pins `file?: string  // absolute path`, and `implementation-notes.md` §Runtime *Static-resolution load pass* states the pass "aggregates each visited file's diagnostics into the entry loom's drain **(sorted by `(file, line, col)`)**." Read together these mean a global lexical sort on the absolute `Diagnostic.file` path, then line, then column — under which a `/a/lib.warp` import sorts before a `/a/main.loom` entry, i.e. *not* entry-first visitation order. The V7a/V7a-T bullet dropped the spec's "(sorted by …)" parenthetical and substituted "in this order," introducing the visitation-order misreading.

This matters because the bullet drives a byte-exact cross-file batch assertion. A test author taking the visitation-order reading and an implementer taking the lexical-path reading (or vice versa) produce different batch content, so `V7a-T`'s golden and `V7a`'s output can disagree on cross-file diagnostic ordering.

The ambiguity entered with that clause when the IMPL aggregation order was mapped onto V7a; the paraphrase substituted "in this order" for the spec's "(sorted by `(file, line, col)`)" parenthetical, dropping the lexical-sort signal.

## Solution Space

**Shape:** single

The spec already fixes the answer — `Diagnostic.file` is the absolute path (`diagnostic-shape.md`) and the drain is "sorted by `(file, line, col)`" (`implementation-notes.md`) — so the cross-file order is a global ascending lexical sort on the absolute `file` path, then `(line, col)`. The visitation-order reading contradicts the spec; the fix is to remove it from the plan, not to add a new rule.

**Recommendation.** In both `V7a-diagnostics-primitive.md` and `V7a-T-diagnostics-primitive.md`, edit the Multi-error Tests bullet so the cross-file ordering basis is explicit and matches the spec. Strike the trailing visitation-order phrasing "which aggregates each visited file's diagnostics into the entry loom's drain in this order" and replace it with a statement that the batch is ordered by ascending `Diagnostic.file` (the absolute path, per `diagnostic-shape.md`), then by `(line, col)` within a file — keeping the existing parenthetical citation to `implementation-notes.md` §Static-resolution load pass, whose "(sorted by `(file, line, col)`)" wording this aligns with. Keep both leaves byte-identical in this bullet (they already are). No spec edit is required; `implementation-notes.md`, `diagnostic-shape.md`, and the `coverage-matrix.md` IMPL row remain read-only for this fix. The implementer should note that, because `file` is the absolute path, the `V7a-T` fixture must control the entry-`.loom`/imported-`.warp` absolute paths so the expected lexical order is reproducible.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one authors `V7a-T`'s byte-exact batch golden in load-pass visitation order (entry first), the other in lexical absolute-path order, so the tests task and the implementation can disagree on cross-file diagnostic ordering and the gate reddens for the wrong reason — or, worse, a green test asserts an ordering that contradicts the spec's "sorted by `(file, line, col)`" rule.
