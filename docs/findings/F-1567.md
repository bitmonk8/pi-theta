## Finding

The binder *Failure-mode templates (normative)* section pins a user-facing system note — `loom /<name>: argument binding produced invalid args — <ajv-summary>` — whose interpolated placeholder is named `<ajv-summary>`. The token spelling embeds the name of the `ajv` package directly into a normative spec artefact. The same section, however, defines the placeholder's content as the **validator-neutral** in-order concatenation of `ValidationIssue` entries (`<path> <message>` joined by `; ` in `validation_errors` order per ERR-14), states the summary is "independent of which validator produced the issues", and relegates AJV to a single explicitly non-normative parenthetical ("the loom 1.0 reference produces this string via the AJV helper `errorsText(...)`"). Everywhere else, validation is owned by the injected `SchemaValidator` seam (PIC-11, `docs/spec_topics/pi-integration-contract/host-interfaces-services.md#pic-11`), with AJV named only as the non-normative reference implementation behind that seam.

The result is an internal inconsistency: a normative identifier is named after a dependency the spec treats as swappable, while the identifier's own definition disclaims any dependence on that validator. The token has also propagated to `query/query-failure-and-repair.md` (the `validator_error` follow-up template), `errors-and-results/queryerror-variants.md` (ERR-14 ordering, ERR-17 synthesised-issue rendering), so the AJV-named token now spans several pages.

This is a naming-hygiene defect, not a behavioural one: the placeholder token is a spec-internal label that is interpolated away, so no rendered byte of any user-visible note contains the string `ajv-summary`. Renaming the token to a validator-neutral form leaves the verbatim user-facing output unchanged. It is the same class of AJV-coupling the corpus already corrects elsewhere (the `SchemaValidator`/PIC-11 routing convention and the sibling finding on AJV-named normative sentences).

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `b3bc4ce` (2026-05-04, "spec: pin failure-observability surface (Cluster A, Option C)") → `9d10d82` (2026-06-01, "resolve \"Failure-mode templates — `<ajv-summary>` rendering pinned to a concrete AJV call\"")
**History:** `b3bc4ce` introduced the `<ajv-summary>` token into the spec body (then `spec_topics/binder.md`) and deliberately AJV-coupled it — the placeholder was defined as rendered by AJV's `errorsText(errors, { separator: '; ' })`, "regardless of how AJV evolves its internal formatting". At that point the token name was consistent with its AJV-coupled content. `9d10d82` resolved a prior finding that the rendering was pinned to a concrete AJV call by reworking the **content** description to be validator-neutral (in-order `ValidationIssue` concatenation, "independent of which validator produced the issues") and demoting AJV to a non-normative parenthetical — but left the token name `<ajv-summary>` unchanged and did not rename it at the propagated sites. The inconsistency this finding flags (AJV-named token alongside explicitly validator-neutral content) is the residue of that incomplete genericisation.

## Solution Space

**Shape:** single

### Recommendation

Rename the placeholder token from `<ajv-summary>` to a validator-neutral form (e.g. `<validation-summary>`) at every site that defines or references it, leaving the surrounding verbatim template text and all rendered bytes unchanged:

- `binder/determinism-cancellation-failure.md` — the *Failure-mode templates (normative)* prose (placeholder enumeration, the pinned-derivation paragraph, the depth-walk fast-fail clause) and the *AJV validation … failed* table row.
- `query/query-failure-and-repair.md` — the `<ajv-summary>` bullet and the `validator_error` follow-up template body.
- `errors-and-results/queryerror-variants.md` — the ERR-14 ordering clause and the ERR-17 synthesised-issue rendering sentence that name the token.
- The four plan leaves that quote the token in acceptance criteria (`V11f`/`V11f-T`, `V13h`/`V13h-T`).

Keep AJV named only in the existing non-normative parenthetical / implementation-hint bullet and behind the `SchemaValidator`/PIC-11 seam. Because the token is interpolated, the rename is a pure identifier substitution with no observable output change; perform it atomically across all sites in one commit so no page references a stale token name.

## Consequence

**Severity:** cosmetic

The token is a spec-internal placeholder identifier interpolated out of every rendered note, so no user-visible output or implementer behaviour changes regardless of its spelling. The only cost is a documentation inconsistency: a normative identifier named after a swappable dependency that its own definition declares validator-neutral, which can mislead a maintainer into believing the placeholder is AJV-specific.
