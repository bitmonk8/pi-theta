## Problem

The *Sequential by default* cross-cutting rule in `docs/plan_topics/conventions.md` permits the four banned `Promise` combinators in `src/**` only when the calling leaf's `Spec.` field cites an obligation whose normative text "mandates concurrency at this site" (and "at the site" for the code-keyed-area arm). That anchoring phrase admits two materially different scopes: a **call-site** reading (the cited obligation mandates concurrency at the exact operation the combinator call implements) and a **feature-area** reading (any obligation anywhere in the leaf's spec area suffices). The two readings admit different exemption sets — the feature-area reading would launder a combinator past review at a call site that has no concurrency mandate, the unguarded-concurrency case the rule exists to prevent. The loom 1.0 closing gate only checks that the cited token resolves to a REQ-ID or `cka-<n>` row, not the concurrency semantic, so the ambiguity is resolved entirely at authoring and self-review time.

That edit resolved the earlier "documented spec reason" anchoring gap (choosing the spec-page-obligation reading over a plan-prose reading) but introduced the residual call-site-vs-feature-area scope ambiguity in the new phrase, which has persisted verbatim through every subsequent edit (later renames moved the file under `docs/plan_topics/` and added the code-keyed-area arm, which re-used the same phrase).

## Solution approach

Rewrite both occurrences of the ambiguous anchor in the *Sequential by default* bullet of `docs/plan_topics/conventions.md` to pin the scope explicitly to the operation the combinator call implements (the call-site reading), so a cited obligation qualifies only when its normative text mandates concurrency at that specific operation. Apply identical call-site-scoped phrasing to the REQ-ID arm ("at this site") and the code-keyed-area arm ("at the site") so both arms carry the same scope.
