## Finding

Step 7 of the version-bump procedure (`docs/spec_topics/pi-integration-contract/version-bump-triggers.md`) closes with: *"If pi-ai exposes the field under the probed name `strictCapable`, this step is a no-op."* That sentence enumerates two arms of the candidate-minor inspection — field-absent under the probed name (no-op) and field-present under the probed name (also no-op) — and identifies the rename-detection arm (field present on `Model<Api>` but **not** under the probed name) as the only one that fails the SDK surface-inventory assertion red.

That enumeration is wrong for the field-present-under-the-probed-name arm. The *Strict-capability absence-under-the-probed-name arm* in `docs/spec_topics/pi-integration-contract/inventory-audit-intro.md` extends the `strict-capability-probe` literal-read assertion (per step 2(a)) so that it **MUST fail** whenever any reachable `Model<Api>` declaration at the model-registry surface exposes a `strictCapable` member under the probed name. The plan leaves V18c (build-time static gates) consume this arm explicitly — `V18c-version-bump-checklist.md` and its tests-leaf both spell out "arm (2) — the absence-under-the-probed-name arm — a reachable `Model<Api>` declaration exposes a `strictCapable` member under the probed name". The runtime claim it guards (the binder degrades to the universal-`W` branch because `strictCapable` is absent on every Pi-supplied `Model<Api>` at the loom 1.0 pin, pinned at `audit-target-categories.md#strict-capability-absence-pin`) only holds while that absence is true; the arm exists precisely so a Pi minor that starts shipping the field causes a build-time red that forces a deliberate contributor response.

Two concrete defects follow. (1) Step 7 tells a contributor the field-present case is a no-op, while the build is in fact red and cannot be merged until that red is cleared — the procedure's reader is set up to look in the wrong place for the failure that step 2(a) just produced. (2) Even after the contributor finds the red, step 7 names no resolution branch for it: there is no documented sequence for re-validating the binder's strict-capability handling against the now-present value, retracting the `strict-capability-absence-pin` claim, updating the degraded-to-universal-`W` narrative in `binder-model-and-context.md`, and editing whatever the absence-arm gate keys on so the build goes green. Both arms of the candidate-minor inspection need positive procedural treatment; only the rename arm currently has it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `b0cdf1b` (May 6 2026, "pi-loom spec: resolve \"Re-validating obligation has no defined output and no enforcement gate\"") authored the original "If no such indicator has been added, this step is a no-op" sentence; `6829a79` (June 6 2026, "pi-loom spec: resolve \"Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)\"") later introduced the absence-under-the-probed-name arm that the sentence now contradicts.
**History:** `git log -S "this step is a no-op"` shows `b0cdf1b` first lands the step (then numbered 6) with the "field absent → no-op" arm only; at that point the procedure had no symmetric red gate for the field-present case, so the sentence's enumeration was exhaustive. `git log -G "absence-under-the-probed-name"` shows the *absence-under-the-probed-name arm* (the build-time literal-read red triggered when a reachable `Model<Api>` declaration exposes `strictCapable` under the probed name) is introduced in `6829a79`, which adds it adjacent to the `strict-capability-absence-pin` anchor and explicitly names "the rename-detection arm owned by bump-procedure step 7 (which catches the field present under a *different* name)" — but does not edit step 7 to name the new arm or retract the no-op claim. The subsequent moves (`bfe08cf` fixing the strictCapable package attribution, `bf92fe6` relocating the absence-arm MUST from `audit-target-categories.md` to `inventory-audit-intro.md`) preserved the contradictory wording on both sides. The defect is therefore the **interaction**: the no-op sentence was correct at authoring and became incorrect when the symmetric red arm was added.

## Solution Space

**Shape:** single

### Recommendation

Replace step 7's trailing single-arm no-op sentence with an explicit two-arm enumeration that mirrors the two builds the candidate-minor inspection can produce, names the resolution branch for each, and cross-references the absence-arm gate by anchor. Concretely, rewrite the last sentence of step 7 so it reads (substance, not wording):

- *Arm (1) — field absent under the probed name (the loom 1.0 baseline).* No edit is required: step 2(a)'s `strict-capability-probe` literal-read assertion stays green, the [`strict-capability-absence-pin`](./audit-target-categories.md#strict-capability-absence-pin) production claim is unchanged, and the binder's degraded universal-`W` branch continues to apply.
- *Arm (2) — field present under the probed name.* Step 2(a) fails red via the [Strict-capability absence-under-the-probed-name arm](./inventory-audit-intro.md#strict-capability-absence-under-probed-name) and cannot be cleared by editing only the pinned-constants block (the gate keys on the SDK declaration itself). Resolve in the same commit by (a) retracting the `strict-capability-absence-pin` claim at [audit-target-categories.md](./audit-target-categories.md#strict-capability-absence-pin) so the production-time absence statement no longer holds; (b) re-validating the binder strict-capability requirement at [binder-model-and-context.md](../binder/binder-model-and-context.md#strict-capability-requirement) against the now-present value — the three-valued read still applies, but the universal-`W` "production behaviour" paragraph that depends on absence MUST be replaced with the SDK-determined value's three-arm distribution (some `Model<Api>` instances admit, some fail with `loom/load/binder-model-not-strict-capable`, some still emit the `…-strict-capability-unknown` W); and (c) editing whatever the absence-arm gate keys on (the spec edit pins the behavioural contract; the absence-arm's implementation-owned discriminator and the fixture that asserts arm (2) reddens stay in lock-step with the spec edit) so step 2(a) goes green at the candidate pin. Steps 5–6 of this procedure are unaffected.
- *Rename arm (already named in this step's body — preserve unchanged).* If the indicator is present on `Model<Api>` under a name **other** than `strictCapable`, rename the probe constant per the existing sentence; the surface-inventory assertion's rename-detection arm is the mechanical gate.

Implementer edge cases: (i) a single candidate-minor bump can hit arm (2) **and** the rename arm simultaneously if the SDK both adds the field under the probed name on some `Model<Api>` instances and exposes a sibling under a different name on others — both resolution branches MUST be performed in the same commit. (ii) The arm (2) resolution is a contract-level edit that touches three pages atomically; it is **not** the same shape as the snapshot-edit-only sweep step 5 trigger (ii) authors, and MUST NOT be misrouted to step 5. (iii) The resolution does not change V18c's two-fixture obligation: both fixtures (rename arm + absence-under-probed-name arm) continue to exist after arm (2) reddens once on a real bump and is cleared, because the gate still has to red on the next regression.

## Consequence

**Severity:** correctness

A contributor running the bump procedure against a candidate Pi minor that adds `strictCapable` under the probed name will see step 2(a) go red, then read step 7 as guaranteeing "this step is a no-op" and have no documented path from the red to a green merge. Two implementers handed this would diverge — one keeping the absence-pin and editing the gate, one retracting the absence-pin and reworking the binder universal-`W` narrative — without the spec choosing.
