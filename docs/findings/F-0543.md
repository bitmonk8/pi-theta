## Finding

`V10b` (Package discovery — bounded walk) closes `DISC-6`, whose normative behaviour on a walk-bound trip is fully pinned in the spec: on a global-cap trip (`looms.scanPackagesMaxFiles` / `looms.scanPackagesTimeoutMs`) the extension "stops opening additional `package.json` files … aborting further package inspection" while the looms already inspected continue to contribute; on a per-read deadline trip the timed-out package is "treated as unreadable **for this scan only**" and "the walk continues with the next candidate" (see `docs/spec_topics/discovery/package-and-settings.md` DISC-6). In both cases the spec mandates **partial-set-proceeds** — the discovered looms register, they are not discarded.

`V10b`'s `DISC-6` test bullet asserts only that `loom/load/discovery-slow` and `package-read-timeout` fire at the respective bounds (with the file-count-before-time tie-break). It never asserts *which looms register* once a bound trips. An implementation that discards the partial set on a trip — or that aborts the entire walk on a per-read timeout instead of skipping only the timed-out package — would satisfy every `V10b` / `V10b-T` test green while silently dropping looms from registration on a slow filesystem, emitting only a warning.

This is the same untested-normative-behaviour class the plan forbids under *No silent test skipping*: a spec-clear outcome with no citing test in the closing leaf. (The original heading names `V10a` as well, but the bounded walk and its trip outcomes live entirely in `V10b`; `V10a` carries no walk bound and is not affected.)

## Solution Space

**Shape:** single

### Recommendation

Add a registration-outcome assertion to `V10b`'s `Tests.` field (mirrored into the `V10b-T` tests-task page, which carries the same bullets) covering both trip kinds against the spec's partial-set-proceeds contract:

- A global-cap trip fixture (`scanPackagesMaxFiles` or `scanPackagesTimeoutMs` driven through the merged-settings + `FakeClock` seams) where packages inspected before the trip ship looms: assert those looms register, the walk stops opening further `package.json` files, and the pre-trip looms are **not** discarded.
- A per-read deadline trip fixture: assert the timed-out package contributes nothing for this scan while looms from the other (non-timed-out) candidate packages still register — i.e. the walk continues past the timed-out candidate rather than aborting.

Optionally surface the partial-set-proceeds outcome in `V10b`'s `Adds.` field as a clause alongside the existing bounded-walk description (e.g. that a bound trip keeps the looms discovered so far and a per-read timeout skips only the offending package), pointing at the DISC-6 spec text. The spec is the authoritative source and stays read-only.

Edge case the implementer must cover: the per-read-timeout fixture must keep the timed-out read's eventual settlement silenced (`.catch(() => {})`, per DISC-6) and assert it is not re-routed into the discovery pass, so the "skip only this package" assertion is not contaminated by a late-resolving read.

## Consequence

**Severity:** advisory

The spec is unambiguous, so an implementer who reads it will keep the partial set; but no `V10b` test enforces it, so a misimplementation that discards discovered looms on a bound trip (or aborts the whole walk on a per-read timeout) ships green. On a slow or large-package install this silently drops looms from registration with only a `discovery-slow` / `package-read-timeout` warning to signal it.
