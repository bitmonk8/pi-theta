## Finding

`host-prerequisites.md` is the corpus's canonical inventory of unpinned Pi host-behaviour presuppositions. Every such presupposition lives there as a member of one explicit family — degraded-state branch prerequisites (a)–(d), binder-model / `model:` load-time resolution, settings write-back key preservation, interactive-session slash dispatch, flag-namespace overlap, `getCommands()` completeness, `registerTool` post-startup acceptance and persistence, `AgentMessage[]` ordering and leading-`user` guarantee, production `TokenEstimator` per-message determinism, and triggering-slash-turn commit posture. Each is introduced with the same "loom 1.0 treats it as a **presupposition** — in the same sense as … the *No concurrent user sessions* framing in Future Considerations" phrasing, each carries an *Authoritative source* pointer, and each is wired to a lettered editorial-review item in the Pi version-bump checklist.

The *Factory-time `cwd == project root`* presupposition is the sole member of that family that is not in `host-prerequisites.md`. It is authored inline inside the `cwd()` bullet of the `FileSystem` seam contract (PIC-13) in `host-interfaces-services.md`, under the anchor `#cwd-project-root-presupposition`. It uses the identical "treats it as a **presupposition** … in the same sense as the degraded-state prerequisites and the *No concurrent user sessions* framing" phrasing, carries its own *Authoritative source* pointer, and is gated by version-bump checklist item (w) — i.e. it is structurally a member of the family but is shelved away from it.

The established precedent for the correct placement is the production `TokenEstimator` per-message-determinism presupposition: its prose lives in `host-prerequisites.md` with the rest of the family, while the seam mechanics it concerns stay in `host-interfaces-services.md` PIC-16, joined by a cross-reference. The `cwd` presupposition should follow the same split — presupposition prose with its family in `host-prerequisites.md`, seam mechanics (the `cwd()` member contract and the no-`process.cwd()` ban) staying in PIC-13 with a cross-reference.

(The original heading also alleged that the DI-seam normative contracts (PIC-10/11/12/13/14/16/20) are mis-filed under PIC and that the TypeScript `>=5.4.0` build-toolchain floor is mis-homed among host prerequisites. Neither is a defect: the corpus deliberately assigns DI-seam normative ownership to PIC and carries cross-referencing summaries in `implementation-notes.md`, and the TypeScript floor is load-bearing for the PIC version-bump brand-string gate and is deliberately contrasted in-text with the version-unpinned `semver`/`chokidar` choices. This finding is scoped to the one genuine placement defect.)

## Issue introduction

**Verdict:** single-commit introduction (present since the presupposition's inception)
**Introducing commit:** `4ea0544` — "pi-loom spec: resolve 'Factory-time cwd == project root premise is unpinned'" (2026-06-05)
**History:** The corpus is git-tracked. `git log -S 'cwd-project-root-presupposition'` and `git log -S 'cwd == project root'` show the presupposition block — and its version-bump checklist item (w) — were first authored in commit `4ea0544`, added (2 lines) directly into `host-interfaces-services.md` (PIC-13); `git log -S 'cwd == project root' -- host-prerequisites.md` returns nothing, confirming the block was never in `host-prerequisites.md`. The presupposition family it belongs with already existed in `host-prerequisites.md` at that point: `git log -S 'as a **presupposition**' -- host-prerequisites.md` shows family members landing earlier (e.g. `79fa3b9`, 2026-06-04, the binder-model registry-population presupposition). The mis-homing was therefore a placement choice made at authoring time against an already-established convention. Subsequent commits widened the divergence without correcting it: `803106d` (2026-06-07) added *Authoritative source* pointers to both the in-place `cwd` block and the `host-prerequisites.md` family, and later commits (e.g. `341bcd9`, 2026-06-27, the `TokenEstimator` per-message-determinism presupposition) kept adding new members to the family in `host-prerequisites.md`, none relocating the `cwd` one.

## Solution Space

**Shape:** single

### Recommendation

Relocate the *Factory-time `cwd == project root` presupposition* prose (currently the `#cwd-project-root-presupposition` block under PIC-13's `cwd()` bullet in `host-interfaces-services.md`) into `host-prerequisites.md`, placing it alongside the other unpinned host-behaviour presuppositions and joining the explicit "in the same sense as … the *No concurrent user sessions* framing" family. Carry the `cwd-project-root-presupposition` anchor id onto the relocated block so the existing cross-references — version-bump checklist item (w), and the `cwd()` bullet's own reference — continue to resolve. Mirror the `TokenEstimator` per-message-determinism precedent precisely: presupposition prose (including its *Authoritative source* pointer) in `host-prerequisites.md`; the `cwd()` member contract, the production `PiFileSystem` capture-once behaviour, and the no-direct-`process.cwd()` ban stay in PIC-13, with the bullet keeping a one-line cross-reference to the relocated presupposition. Retarget version-bump-step2.md item (w)'s link to the new home.

Edge cases for the implementer: do not duplicate the block — remove it from PIC-13 once relocated; preserve the anchor id verbatim and confirm no dangling references to the old location remain.

## Consequence

**Severity:** cosmetic

The presupposition is correctly stated, anchored, and gated by version-bump checklist item (w), so no implementer is misled and no runtime behaviour diverges. The cost is consistency and discoverability: a contributor auditing the host-behaviour-presupposition family in `host-prerequisites.md` (e.g. when extending it or running the bump editorial review) will not find the `cwd` assumption among its siblings, and future family-wide edits can silently skip it.
