## Finding

`V10b`'s **Adds** field describes the bounded package walk via the bare constants `maxFiles` 2000 / `timeoutMs` 2000 and references `looms.scanPackagesTimeoutMs` only inside the per-read deadline formula `max(200, floor(looms.scanPackagesTimeoutMs / 10))`. Two further normative behaviours that the spec mandates for the same walk are absent from Adds and surface only inside the DISC-6b **Tests** bullet:

1. **`looms.scanPackages: false` → walk skipped wholesale.** Per `package-and-settings.md` DISC-6 and the `looms.scanPackages` settings-key definition, when the key is `false` no package root is scanned at all (only Global, Project, Settings, and CLI sources contribute looms).
2. **`looms.scanPackagesMaxFiles` is the operative trip bound, not a hardcoded `2000`.** Per DISC-6 the walk stops once it has inspected `looms.scanPackagesMaxFiles` files (default `2000`), tripping `loom/load/discovery-slow` at the operator-supplied value sourced from `V10c`'s merged settings — not at the constant `2000`.

The Adds field is the leaf's contract surface: it states what the leaf adds independently of how its tests verify it. With these two behaviours present only in a test bullet, the settings-driven integration contract (the merged `looms.*` bounds flow into the walk; `scanPackages: false` disables it) is discoverable to an implementer only by reading the Tests section, and the Adds reads as though the bounds are fixed constants.

## Solution Space

**Shape:** single

### Recommendation

Extend `V10b`'s **Adds** field so the two settings-driven walk behaviours are stated alongside the existing timeout formula:

- The walk is skipped wholesale when merged settings carry `looms.scanPackages: false` (no package root is scanned; only Global, Project, Settings, and CLI sources contribute looms), per `package-and-settings.md` DISC-6.
- The `discovery-slow` trip bound is the merged `looms.scanPackagesMaxFiles` value (default `2000`) sourced from `V10c`'s merged settings — not a hardcoded `2000` constant — matching the existing treatment of `looms.scanPackagesTimeoutMs` already present in the Adds.

Replace the Adds clause `the bounded walk (\`maxFiles\` 2000, \`timeoutMs\` 2000)` with text that names the operative settings keys (`looms.scanPackagesMaxFiles` / `looms.scanPackagesTimeoutMs`, defaults `2000`/`2000 ms`, sourced from V10c's merged settings) and add a clause stating the `looms.scanPackages: false` skip. The DISC-6b/DISC-6c Tests bullets stay as the verification of this contract.

## Consequence

**Severity:** advisory

An implementer reading V10b's Adds in isolation would model the walk bounds as the fixed constants `2000`/`2000 ms` and could miss both the `scanPackages: false` wholesale-skip and the obligation to source `scanPackagesMaxFiles` from V10c's merged settings. The paired DISC-6b/DISC-6c tests still red on a hardcoded implementation, so the TDD ritual recovers the behaviour; the gap is a contract-surface omission, not a behaviour that ships unverified.
