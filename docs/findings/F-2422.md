## Finding

In `binder/binder-bypass-and-envelope.md` the section `### Binder system prompt` (and its `#### System-prompt structure (normative)`) sits as a `###` child of `## Binder envelope`. But the page's own prose scopes "Binder envelope" to the binder's **output**: "The binder is asked to return one of three structured outputs … `{ kind: "ok" | "needs_info" | "ambiguous" }`", and the correctly-nested `### Binder envelope schema (constructed dynamically from params:)` formalises that output schema. The system prompt is the opposite direction of data flow — the **input** the runtime constructs and sends *to* the binder model (loom identity, parameter table, user arguments, session context, return-shape instruction).

Filing an input contract under a heading whose subject is the output envelope misrepresents the document's structure: a reader scanning the outline reads "Binder system prompt" as a facet of the envelope, and a reader who has formed the (correct) mental model that "envelope = what the binder returns" is led to believe the system prompt is part of that returned shape. The system-prompt material is a peer concern to bypass and to the envelope, and should be a sibling top-level `## Binder system prompt` section, not a subordinate of the envelope. Promoting it leaves the normative content (the eight structure items, *Type display*, *Default-literal rendering*, and *Parameter-line reference renderings*) byte-identical; only the heading level changes.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `c1e5d64` — "spec: REQ-ID infrastructure + Phase 12a heading promotion on five named pages (Decision #12, partial)" (2026-05-05)
**History:** Before `c1e5d64` the binder material was a flat list of run-in bold labels under `# Slash-Command Argument Binding` — `**Binder envelope.**`, `**Binder envelope schema …**`, and `**Binder system prompt**` all at the same prose level (confirmed in `spec_topics/binder.md` at `c1e5d64^`, originating from the spec split at `fecb504`). The Phase-12a heading-promotion pass in `c1e5d64` converted those run-in labels into a heading hierarchy and assigned `Binder system prompt` a `###` subordinate to the new `## Binder envelope` (rather than a sibling `##`), introducing the parent/child relationship that is the defect. The later size-cap split `f5e89f4` (which created `binder/binder-bypass-and-envelope.md`) carried the nesting forward verbatim. The conceptual adjacency of the two topics predates `c1e5d64`; the misleading heading subordination does not.

## Solution Space

**Shape:** single

### Recommendation

Promote `### Binder system prompt` to a top-level `## Binder system prompt`, making it a sibling of `## Binder bypass` and `## Binder envelope`, and demote its child `#### System-prompt structure (normative)` to `### System-prompt structure (normative)`. The *Type display*, *Default-literal rendering*, and *Parameter-line reference renderings* sub-blocks stay where they are relative to the structure section. Do not alter any prose, table, or fenced example.

Edge cases for the implementer:
- The plan and coverage matrix reference this material via the anchor `#system-prompt-structure-normative`, which Markdown derives from heading *text*, not level; re-levelling preserves it, so no cross-reference needs updating. Verify no link targets `#binder-system-prompt` specifically (none found in the corpus) before relying on this.
- The file title is "Binder bypass and envelope"; the system prompt now becomes a third visible top-level topic. This is acceptable (bypass is already a third), but if a tighter title is desired that is a separate editorial decision, not part of this fix.

## Consequence

**Severity:** cosmetic

A reader navigating the outline can mistake the input-side system-prompt contract for a facet of the output envelope; the normative obligations themselves are intact and discoverable, so no implementer produces divergent behaviour. Pure document organisation.
