## Finding

`V6a` (`docs/plan_topics/V6a-frontmatter-contract.md`) carries four binding **Tests** bullets: `missing-mode`, `unknown-frontmatter-field` (warning-tolerated), `loom/parse/timeout-field-rejected` (NOCEIL-1), and `loom/load/model-unresolved` (load failure). Under the per-phase TDD ritual in `conventions.md` (Tests bullets are the leaf's binding obligations; **Ships when** is "a concrete, externally observable change"), the exit gate should reflect those obligations.

The leaf's **Ships when** line reads: "`npm test` parses frontmatter, requires `mode:`, and tolerates unknown keys as warnings." It names only the first two Tests bullets and omits both failure paths — the per-call `timeout` rejection and the unresolvable-`model:` load failure. The externally observable exit condition therefore under-states what the leaf must demonstrate to be tagged `V6a-complete`.

**History:** `git log --follow` on `V6a-frontmatter-contract.md` shows the leaf's first commit `c6a664e` introduced both the `loom/parse/timeout-field-rejected` Tests bullet (`git log -S 'timeout-field-rejected'`) and the current narrow Ships-when line (`git log -S 'parses frontmatter, requires'`) — so the Ships-when-narrower-than-Tests gap existed from inception. `git log -S 'model-unresolved'` shows the `model-unresolved` Tests bullet was added later by `4088e2e` without extending Ships-when, widening the same gap.

## Solution Space

**Shape:** single

### Recommendation

Extend `V6a`'s **Ships when** line so the externally observable exit condition names the two failure-path outcomes already asserted by its Tests bullets. Replace the current line:

> **Ships when.** `npm test` parses frontmatter, requires `mode:`, and tolerates unknown keys as warnings.

with one that additionally names the per-call timeout rejection (`loom/parse/timeout-field-rejected`) and the unresolvable-`model:` load failure (`loom/load/model-unresolved`), e.g.:

> **Ships when.** `npm test` parses frontmatter, requires `mode:`, tolerates unknown keys as warnings, rejects a per-call `timeout` field (`loom/parse/timeout-field-rejected`), and fails the load on an unresolvable `model:` (`loom/load/model-unresolved`).

If the related model-unresolved overclaim finding (below) rescopes the `model:` outcome wording in the Tests bullet, mirror that rescoped wording here rather than asserting the downstream non-registration consequence.

## Consequence

**Severity:** advisory

An implementer reading only the Ships-when line could tag `V6a-complete` on the strength of the parse/require/tolerate behaviours while the two failure-path assertions are weak or absent; the Tests bullets still bind, so a faithful implementer is unaffected, but the exit gate gives a misleadingly narrow completion signal.
