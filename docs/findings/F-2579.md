## Finding

INV-4 caps the nesting depth of an `invoke` chain at 32, and the trailing paragraph asserts the count is per-chain, crosses subagent-mode boundaries unchanged, and that for subagent-mode children "the counter passes through the `customTools` / `AgentSession` boundary as a runtime-side invariant carried by the invoke trampoline; it is not part of the wire-level data passed to the spawned session." This fixes the *semantics* (a `subagent ‚Üí subagent` or `prompt ‚Üí subagent` crossing neither resets nor shares budget) but names no concrete *carrier*. The phrase "invoke trampoline" appears nowhere else in the corpus, and the only per-invocation runtime structure the spec does define ‚Äî the `ActiveInvocationRegistry` entry (`Set<{ loomAbort, disposeBarrier, shutdownReason, loom, invocationId }>` in pi-integration-contract/active-invocation-registry.md) ‚Äî carries no depth field.

A subagent-mode child runs in a freshly spawned, detached `AgentSession` whose nested `invoke`s execute asynchronously, and two concurrent subagent siblings spawned from the same parent must each see the parent's depth as their starting count while remaining independent of one another (the "sibling invokes do not share budget" rule). A module-level or extension-instance-level depth variable cannot satisfy both constraints simultaneously: concurrent siblings interleaving on Pi's event loop would corrupt each other's count, producing either spurious `loom/runtime/invoke-depth-exceeded` panics or a silently defeated cap. Because the spec states only the invariant and not the mechanism that propagates the parent's depth into each spawned session, two reasonable implementers can choose incompatible carriers (ambient counter vs. per-spawn-threaded value) and diverge on whether the cap holds under concurrency.

## Solution Space

**Shape:** single

### Recommendation

Specify a concrete per-invocation depth carrier and the propagation rule across the spawn boundary, replacing the under-specified "invoke trampoline" prose:

- Carry the current chain depth as an explicit field threaded into each `invoke` spawn ‚Äî either an added `depth` field on the per-invocation runtime invoke-options handed to the spawn site, or an added `depth: number` member on the `ActiveInvocationRegistry` entry ‚Äî so each spawned `AgentSession` (and each `.loom`-via-`tools:` / cross-file `.warp fn` frame) receives the parent's depth by value at spawn time rather than reading ambient state.
- State that the child's starting count is the parent's depth captured at spawn, incremented before the child frame begins executing (consistent with the existing "incremented before the child frame begins" rule), and that two concurrent subagent siblings each capture the same parent depth independently.
- Explicitly forbid a module-global / extension-instance-scoped counter as the carrier, naming the concurrent-sibling corruption as the reason.

Drop or define the "invoke trampoline" term; once the field-threaded carrier is named, the "not part of the wire-level data passed to the spawned session" observation remains accurate (the carrier is a runtime-side value, not Pi SDK wire data) and can be retained as a clarifying note. Watch the interaction with the `tools:`-dispatched `.loom` callable path, which crosses the same `customTools` boundary and must thread depth identically to the direct-`invoke` path.

## Consequence

**Severity:** correctness

Without a named carrier, an implementer may back the depth counter with ambient (module/extension-instance) state that is corrupted by concurrent subagent siblings interleaving on the event loop, yielding spurious depth-exceeded panics or a cap that is silently defeated for subagent recursion ‚Äî the exact runaway-recursion case INV-4 exists to bound. The semantic guarantee ("crosses the boundary unchanged") is untestable until the carrier is pinned.
