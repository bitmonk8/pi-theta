## Finding

`V13f` is the `@`-query entry in the four-site swallowing-handler routing set that `V17a` delegates to its owning leaves (`V14f`, `V13f`, `V15h`, `V9o`). Each sibling pins, via `Deps`, the leaf that constructs its abandonable Promise: `V14f` → `V14a` (code-side `execute()`), `V15h` → `V15a` (`invoke` child execution), `V9o` → `V9i` (subagent `AgentSession.abort()`). `V13f` pins `V9n`, which is correct: the spec (`cancellation.md`, the *Race semantics — swallowing-handler attachment on every abandonable Promise* paragraph) names the site "the underlying `@`-query **provider** Promise", and `V9n` is the prompt-mode leaf that drives the provider call and observes its abort path. The dependency is sound.

The defect is a single inconsistent token in `V13f`'s `Adds`. Its first clause correctly says "The `@`-query **provider** abandonable-Promise site"; its second clause then says "the per-site witness that the `@`-query **dispatch** Promise attaches its swallowing handler at the construction site." The leaf's own `Tests` and `Ships when` bullets both say "provider Promise", and the spec says "provider Promise"; only this one clause says "dispatch Promise".

"`@`-query dispatch" is the exact phrase `V13c` uses for a different construct — its pre-call checkpoint site ("Checkpoint before `@`-query dispatch", the `cka-47` `V13c` facet, fired "immediately before each `@`-query dispatch on the live `V13c` query-tool-loop surface"). The `Adds`' "dispatch Promise" therefore reads as a `V13c`-owned construction site in the query-tool loop, contradicting the `V9n` dependency and the leaf's own provider-Promise wording elsewhere.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `eaa5000` (2026-06-16, "pi-loom plan: resolve \"V13c bundles four independent spec obligations into one leaf\"")
**History:** `V13f-query-cancellation-routing.md` was created in `eaa5000` when the `@`-query swallowing-handler routing was split out of the `V13c` bundle. The `Adds` carried both the correct "provider abandonable-Promise site" phrasing and the inconsistent "the `@`-query dispatch Promise" clause from that first commit. `git log -L` over the `Adds` block shows the line touched only by later four-site-roster edits (`V9i`→`V9o`, `V15a`→`V15h`, `V14a`→`V14f`); the "dispatch Promise" token itself was never revised. The defect is a creation-time wording inconsistency, not a regression introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V13f-query-cancellation-routing.md`, in the `Adds` field, change the second clause's "the `@`-query dispatch Promise attaches its swallowing handler at the construction site" to "the underlying `@`-query provider Promise attaches its swallowing handler at the construction site". This aligns the `Adds` with the spec's "underlying `@`-query provider Promise" wording (`cancellation.md`, *Race semantics — swallowing-handler attachment on every abandonable Promise*), with the leaf's own `Tests` and `Ships when` bullets (both already say "provider Promise"), and with the `V9n` dependency. Leave `Deps` (`V13f-T`, `V9n`, `V8a`, `H4b`) unchanged — the dependency is correct. The spec is read-only for this fix; this is a plan-leaf wording change only, and `V13c`'s "Checkpoint before `@`-query dispatch" text must not be touched (it correctly names a distinct pre-call checkpoint site).

## Consequence

**Severity:** advisory

A reader of `V13f`'s `Adds` may believe the abandonable Promise is constructed and handler-attached inside `V13c`'s query-tool loop (the named `@`-query-dispatch checkpoint surface) rather than at the `V9n` provider layer the `Deps` pin, since "`@`-query dispatch" is `V13c`'s own term. The `Deps`, `Tests`, and `Ships when` bullets all point to the provider Promise, so a working leaf is still reachable, but the contradictory clause invites an implementer to attach the swallowing handler at the wrong site.
