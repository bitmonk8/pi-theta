## Finding

Step 5 of the version-bump procedure (`version-bump-triggers.md`) names the placement options for the `SessionShutdownEvent['reason']` brand-string type-equality assertion, then constrains the `src/extension/` option with two prescriptions: the assertion "MUST be a dedicated file matching the suffix convention `src/extension/**/*.assert.ts`", and "inlining the assertion into a non-suffix-matching `src/extension/**/*.ts` file is **not** an *equivalent form* under this step's permission." The stated justification is a separation-of-concerns invariant — that an inlined assertion would "couple the typecheck-only assertion to runtime code paths that share the file, complicating both `tsconfig` configuration and the … separation-of-concerns invariant."

The observable obligations for this assertion are only two: (1) it MUST be reached by the `npm run typecheck` invocation (the existing "loom `tsconfig` MUST include it" rule, plus the split-tsconfig-exclusion prohibition), and (2) its Pi-side surface references MUST be accounted for by the inventory-closure audit. Neither obligation requires a dedicated module. An inlined assertion is fully accounted for under the audit's own rules: `audit-resolution.md` (*Audit scope*) already states that an embedded assertion in a non-suffix-matching `src/extension/**/*.ts` file "is **not** excluded by this item; the audit MUST resolve the … `Type.Unsafe` (and any other in-scope) references … through the normal inventory rules." For this specific assertion those references are the `SessionShutdownEvent` category-(2) named import (resolves via the inventory join key) and `Type.Unsafe` (resolves via the typebox member-access allow-list `{ Unsafe }`) — both already authorised. An inlined assertion is therefore observably conformant, and step 5's prohibition forecloses it on organisational grounds (tsconfig tidiness, separation of concerns) that are not part of the contract.

The prescription is compounded by a dual-authority hazard: step 5 restates the literal `src/extension/**/*.assert.ts` glob that *Audit scope* already owns and pins as the audit's sole syntactic exclusion discriminator. The behavioural requirement step 5 should carry is "reachable by typecheck, and either excluded by the *Audit scope* suffix convention or audited normally" — not a re-pinned glob plus a layout mandate.

## Solution Space

**Shape:** single

### Recommendation

In `version-bump-triggers.md` step 5, replace the dedicated-file mandate and the restated glob with the two observable obligations and a delegating reference:

- Keep both placement options (`test/` and `src/extension/`) and keep the existing observable obligations verbatim: the loom `tsconfig` MUST include the assertion, and a split-tsconfig layout that excludes it is a violation (not a silent demotion to the runtime `loom/host/session-shutdown-reason-unknown` fallback).
- For the `src/extension/` case, state the audit relationship by reference rather than restating the glob: the assertion module is excluded from the inventory-closure audit only when it matches the `src/extension/**/*.assert.ts` suffix convention **owned by *Audit scope***; an assertion placed in a non-matching `src/extension/**/*.ts` file is permitted but is then audited under the normal inventory rules (its `SessionShutdownEvent` import and `Type.Unsafe` access already resolve), per the path *Audit scope* already describes.
- Drop the prohibition on inlining (or demote the separation-of-concerns rationale to a non-normative note). It is an organisational preference, not an observable contract, and re-asserting it as a MUST creates a needless foreclosure plus a dual-authority restatement of the `*.assert.ts` glob.

Edge cases the implementer must watch: do **not** weaken *Audit scope* itself — the `*.assert.ts` suffix is the audit's only syntactic exclusion discriminator and is load-bearing there; this fix only stops step 5 from re-pinning it and from forbidding the inline alternative. The typecheck-reachability and split-tsconfig-violation obligations remain MUSTs and must survive the edit.

## Consequence

**Severity:** advisory

An implementer following step 5 must place the brand-string assertion in a dedicated `src/extension/**/*.assert.ts` file (or under `test/`) and is forbidden from inlining it, even though an inlined assertion is observably conformant — typecheck-reachable, with every Pi-side reference already resolving through the inventory and the typebox allow-list. The rule over-constrains layout without an observable basis; it produces no divergence or wrong behaviour, but needlessly forecloses a valid implementation.
