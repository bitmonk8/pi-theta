## Finding

The binder system-prompt structure (`binder-bypass-and-envelope.md`, *System-prompt structure (normative)* item 4) makes the per-field `— <description>` segment conditional on a normalisation result it does not own: the segment "is appended iff the field carries a non-empty `description:` (per [Descriptions](../descriptions.md), **after that section's normalisation**)" and "is omitted … When the field's `description:` is absent or empty after normalisation". Both the trigger predicate and its negation hinge on two notions delegated to `descriptions.md`: a named operation called *normalisation*, and an emptiness test applied to its output.

`descriptions.md` defines neither. It specifies a multi-line join ("Consecutive `///` lines are joined with newlines"), a common-leading-whitespace strip ("same algorithm as query-template dedent"), and a byte-for-byte "No transformation" emission rule — but it never uses the word "normalisation", so the cross-reference points at a phase that does not exist by that name, and it never states what makes a description "empty". The gap is load-bearing precisely because this is a byte-exact contract: item 4 ends each per-field line "immediately after `<requirement>` with no trailing whitespace" when the description is empty, and the *Parameter-line reference renderings* table pins a fourth row for the description-omitted form (`  language (string) required`). A `///` block that joins/strips to a whitespace-only or newline-only string (e.g. blank `///` lines, which `descriptions.md` says "become blank lines") is "non-empty" under a byte reading but "empty" under a trim-then-test reading. The two readings emit different bytes for the same loom source — one appends ` — \n` -ish content verbatim, the other omits the segment — diverging on a MUST-conform, reference-pinned surface.

The fix is one coherent edit at the owning page: `descriptions.md` defines the normalisation operation as a named, anchorable step (join + common-leading-whitespace strip) and states the emptiness predicate it feeds (e.g. "empty after normalisation = the normalised string contains no non-whitespace character", or an explicit byte-empty test), and item 4 cites that anchor instead of the bare phrase "after that section's normalisation".

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/descriptions.md`, name and anchor the normalisation step explicitly under the *Multi-line* rule: define *description normalisation* as the multi-line join followed by the common-leading-whitespace strip already described, and add the emptiness predicate that consumers test against — a normalised description is **empty** when it contains no non-whitespace code point (so an all-blank `///` block normalises to empty), otherwise **non-empty**. Give the definition a stable anchor (e.g. `#description-normalisation`).

In `docs/spec_topics/binder/binder-bypass-and-envelope.md` item 4, replace the bare "(per [Descriptions](../descriptions.md), after that section's normalisation)" with a link to that anchor, and confirm the appended `<description>` is the normalised result (so the byte appended after the U+0020 U+2014 U+0020 separator is the post-normalisation string, consistent with the existing reference renderings).

Edge cases the implementer must watch: the all-blank `///` block (`descriptions.md` says blank `///` lines "become blank lines") must resolve to **empty** so the segment is omitted and the line ends after `<requirement>` with no trailing whitespace, matching the pinned `  language (string) required` row; and a description containing non-ASCII whitespace (e.g. U+00A0) must follow whichever whitespace set the emptiness predicate names — pick the same ASCII whitespace set already pinned for slash-argument and system-note handling, or state explicitly that the predicate is "no non-whitespace under Unicode `\s`", to avoid re-introducing the U+00A0 ambiguity resolved elsewhere in the binder pages.

## Consequence

**Severity:** correctness

Two conforming implementers diverge on the per-field line emitted for a field whose `///` description normalises to whitespace/newlines only: one appends the ` — <description>` segment (treating it as non-empty), the other omits it (treating it as empty). Because the binder system-prompt per-field line is a byte-exact MUST contract with pinned reference renderings, the divergence produces different binder prompts and conflicting conformance-test outcomes for the same loom source.
