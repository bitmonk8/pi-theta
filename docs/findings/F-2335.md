## Problem

`V2c`'s **Adds** names the JS representation of loom object-schema values as a "native map", but `runtime-value-model.md`'s representation table specifies an object schema (named or anonymous) is a **JS plain object keyed by loom-side names**. "native map" reads naturally as a JS `Map` — the same spec page lists native `Map`/`Set` among the engine primitives — and a `Map`-backed object value diverges on every observable boundary: `JSON.stringify` yields `{}` (dropping all fields outbound), loom-side key-set structural equality cannot use plain key enumeration, and the inbound wire-name rebuild assumes plain-object field access. The leaf's own Tests operationalise the correct behaviours, so the term contradicts the leaf's behavioural assertions.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** `git log -S'native map' -- docs/plan_topics/V2c-value-model.md` returns exactly one commit, c6a664e, which is also the file's first commit (`git log --reverse`). The two later commits that touched `V2c` (2864421 reject-gate dep change, d4f6e5c the V2c/wire-name split) did not alter the "native map" wording. The defect entered with the plan's initial authoring of this leaf and has carried through unchanged.

## Solution approach

In `V2c`'s **Adds**, rename the object-schema representation noun from `native map` to the spec's term, `JS plain object keyed by loom-side names`. The authoritative wording is the "Object schema (named or anonymous)" row of the representation table in `docs/spec_topics/runtime-value-model.md`.

## Solution constraints

- Out of scope: the enum-tag and internal-`Result`-tag wording on the same Adds line (a separate concern), and `V2c-T`, which does not carry the token.
- The spec (`docs/spec_topics/runtime-value-model.md`) is read-only for this fix.
