## Finding

`H6a`'s **Ships when** spells out the full commit-message evidence-record *format* for the manual real-host smoke run: it obligates "a falsifiable evidence record committed in this leaf's **gate-activation commit message** — no separate artefact file", and enumerates the four fields the record must capture — the Pi-SDK pin literal the run executed against, the named owner who ran the smoke, the date of the run, and the observed result against the `real-host-smoke-gate.md` criterion. That is governance *procedure* — how a smoke run is recorded — not the acceptance condition `H6a`'s gate turns on.

The dedicated governance note `real-host-smoke-gate.md` is the documented home of the manual real-host smoke governance protocol (`plan.md` §Release gate names it as the home for "triggers, named owners, activation window, live-host precondition, pass/fail criterion, and merge-blocker / revert path"). That note already states the recording convention — "record the result — in the triggering change's commit message …, with no separate artefact file, the same manual-run-evidence convention the spec fixes for the version-bump path" — but it does **not** enumerate the four record fields. The result is one convention stated in two places: the no-separate-artefact rule appears in both documents, and the four-field enumeration appears only in `H6a`. The evidence-record format should be consolidated into `real-host-smoke-gate.md`, leaving `H6a`'s Ships-when to state the acceptance condition (the smoke ran and passed against the pin, and the run is recorded per the governance note) plus a cross-reference.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** eca63cf, fd22a3d
**History:** eca63cf (2026-06-11) authored the evidence-record format (Pi-SDK pin, date, observed result) directly in `H6a`'s release-gate acceptance text at a time when no dedicated governance document existed — the protocol was then referenced through `H4a`, so housing the format in `H6a` was the only available home. fd22a3d (2026-06-17) extracted the manual real-host smoke governance protocol into the new `real-host-smoke-gate.md`, carrying the recording convention (commit message, no separate artefact) into it but leaving the four-field evidence-record enumeration behind in `H6a`; the placement split crystallised at this commit. Intervening commits 4099d0a and 0bd1978 had moved/refined the format text within `H6a` (defining the "gate-activation commit message" home and relocating it into Ships-when) but predate the dedicated governance note and so did not introduce the misplacement.

## Solution Space

**Shape:** single

### Recommendation

Relocate the evidence-record format from `H6a`'s **Ships when** into `real-host-smoke-gate.md`, consolidating it with the recording convention already there.

- In `real-host-smoke-gate.md`, extend the existing recording-convention sentence (the one ending "… the same manual-run-evidence convention the spec fixes for the version-bump path …") so it also states the fields the record captures: the Pi-SDK pin literal the run executed against (`host-prerequisites.md#pi-sdk-pin`), the named owner who ran the smoke, the date of the run, and the observed result against the pass/fail criterion this note defines. This is where the no-separate-artefact / gate-activation-commit-message convention already lives, so the full format is then stated once.
- In `H6a`'s **Ships when**, strike the `"Recorded" obligates …` sentence that enumerates the four fields and the no-separate-artefact rule, and replace it with the acceptance condition plus a pointer: the manual real-host smoke run was executed and passed against the shipping Pi-SDK pin, and the run is recorded in this leaf's gate-activation commit message per the evidence-record convention in `real-host-smoke-gate.md`. Keep `H6a`'s existing statement that the release does not pass until the commit message records the run (the acceptance condition), but let the field list and the no-separate-artefact rule live only in the governance note.

The cross-reference from `H6a` to `real-host-smoke-gate.md` already exists, so no new link wiring is required; the auditor following it lands on the single authoritative format.

## Consequence

**Severity:** advisory

The evidence-record format is split across two documents: the no-separate-artefact convention is duplicated in `H6a` and `real-host-smoke-gate.md`, and the four-field enumeration sits only in `H6a`. A later edit to the recording format in one document silently desyncs the other, and a contributor or the `H6a` auditor must read both to learn the complete record format the release gate audits. No implementer is blocked, but the duplication is a maintenance divergence surface on a load-bearing release-gate convention.
