## Problem

Spec `BNDR-9` defines a `customType` as transcript-safe only when it is non-empty and contains none of U+000A, U+000D, `]`, or the two-byte sequence `: `, so the empty string is non-transcript-safe and must fire `loom/runtime/custom-type-unsafe` (the runtime diagnostic registry agrees, scoping the trigger as "empty, or containing …"). The `BNDR-9` Tests bullet in both `V11b` and `V11b-T` enumerates only the forbidden-character half ("containing any of `\n`, `\r`, `]`, or the two-byte sequence `: ` …"), reading as the complete definition. An implementer can therefore satisfy the leaf's tests while accepting an empty `customType` — a state the spec and registry both require to be rejected.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** The BNDR-9 Tests bullet entered the corpus at the initial plan build (c6a664e) reading "a non-transcript-safe `customType` (containing `\n`/`\r`/`]`/`: `) fires …" — the parenthetical enumerated forbidden characters only and never carried the empty-string requirement. Two later edits touched the same line (fe694dd, 2026-06-10, disambiguating the colon-space token; 31c8cf2, 2026-06-17, fixing the code-span/U+003A U+0020 annotation) but both refined the character enumeration and neither added the empty-string case. The omission has been continuously present since inception in both `V11b` and `V11b-T`.

## Solution approach

Extend the `BNDR-9` Tests bullet so the test obligation covers the empty-string trigger alongside the forbidden characters — add `empty` to the non-transcript-safe parenthetical (e.g. "empty, or containing any of `\n`, `\r`, `]`, or the two-byte sequence `: ` …"). Apply the identical edit to the bullet in both `docs/plan_topics/V11b-bind-context-transcript.md` and `docs/plan_topics/V11b-T-bind-context-transcript.md` so the tests task and implementation task stay aligned.

## Solution constraints

- The spec is read-only for this fix — `BNDR-9` and the `loom/runtime/custom-type-unsafe` registry already state the empty-string case correctly.
