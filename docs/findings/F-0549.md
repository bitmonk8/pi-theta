## Finding

Spec [`BNDR-9`](../../docs/spec_topics/binder/binder-model-and-context.md) defines a `CustomMessage.customType` as **transcript-safe** when it "is **non-empty and** contains none of" U+000A, U+000D, `]`, or the two-byte sequence `: `. The empty string is therefore non-transcript-safe and must fire `loom/runtime/custom-type-unsafe`; the runtime diagnostic registry entry for that code agrees, scoping the trigger as "empty, or containing U+000A … or the two-byte sequence `: `".

The `BNDR-9` Tests bullet in both `V11b` and `V11b-T` enumerates only the forbidden-character half of the predicate: "a non-transcript-safe `customType` (containing any of `\n`, `\r`, `]`, or the two-byte sequence `: ` …) fires `loom/runtime/custom-type-unsafe`". The parenthetical reads as the complete definition of "non-transcript-safe", so an implementer can satisfy the leaf's tests while accepting an empty `customType` — a state the spec and the diagnostic registry both require to be rejected.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** The BNDR-9 Tests bullet entered the corpus at the initial plan build (c6a664e) reading "a non-transcript-safe `customType` (containing `\n`/`\r`/`]`/`: `) fires …" — the parenthetical enumerated forbidden characters only and never carried the empty-string requirement. Two later edits touched the same line (fe694dd, 2026-06-10, disambiguating the colon-space token; 31c8cf2, 2026-06-17, fixing the code-span/U+003A U+0020 annotation) but both refined the character enumeration and neither added the empty-string case. The omission has been continuously present since inception in both `V11b` and `V11b-T`.

## Solution Space

**Shape:** single

### Recommendation

Extend the `BNDR-9` Tests bullet in both `docs/plan_topics/V11b-bind-context-transcript.md` and `docs/plan_topics/V11b-T-bind-context-transcript.md` so the test obligation covers the empty-string trigger alongside the forbidden characters, matching spec BNDR-9's "non-empty and contains none of" and the `loom/runtime/custom-type-unsafe` registry wording. Concretely, change the parenthetical from:

`a non-transcript-safe `customType` (containing any of `\n`, `\r`, `]`, or the two-byte sequence `: ` (U+003A U+0020)) fires `loom/runtime/custom-type-unsafe`.`

to:

`a non-transcript-safe `customType` (empty, or containing any of `\n`, `\r`, `]`, or the two-byte sequence `: ` (U+003A U+0020)) fires `loom/runtime/custom-type-unsafe`.`

The two leaf files mirror each other on this bullet; apply the identical edit to both so the tests task and implementation task stay aligned. The spec and the diagnostic registry already state the empty-string case correctly, so no spec edit is required.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one rejects an empty `customType`, the other accepts it, and both pass the leaf's BNDR-9 test as written. The accepting implementation ships green while violating BNDR-9 and the `custom-type-unsafe` registry contract — an empty `customType` then renders a `[custom:]` role tag that the binder was supposed to reject before transcript construction.
