## Finding

The `string` stdlib table in [`docs/spec_topics/expressions.md`](../../docs/spec_topics/expressions.md) (under *Built-in methods and properties*) gives `trim()` the gloss "Strips Unicode whitespace from both ends." No standard is named, no whitespace set is enumerated, and no reference vectors are supplied. "Unicode whitespace" is not a single well-defined set: the JS `String.prototype.trim()` set (Unicode `White_Space` ∪ line-terminator code points), the regex `\s` set, and the `Zs` general-category set differ from each other on edge-case code points such as U+0085 (NEL), U+00A0 (NBSP), U+000B (VT), U+000C (FF), U+180E, and U+FEFF. Two conforming implementations can read this row, pick different sets, and disagree on `" \u00A0x\u00A0 ".trim()` or `"\u000Bx\u000B".trim()` — the kind of silent divergence the rest of the table was explicitly engineered to prevent.

The surrounding rows in the same table do not have this problem. `startsWith`/`endsWith`/`includes` say "JS semantics", `toLowerCase`/`toUpperCase` say "Locale-independent (`String.prototype.toLowerCase`)" with a named host method, `replace` carries five normative reference vectors that pin scan direction, `$`-handling, and empty-`from` behaviour, and `length` and `split` each pin a host-specific decision (UTF-16 code units; code-unit decomposition for empty separator). `trim()` is the only row that names neither a host method nor a vector.

The contrast with the binder shard is sharper still: `defaulting-system-note-echo.md` rule 1 and `binder-bypass-and-envelope.md` (`<a id="slash-argument-whitespace"></a>`) go out of their way to enumerate the whitespace set explicitly as `{U+0009, U+000A, U+000B, U+000C, U+000D, U+0020}` and call out that "Non-ASCII whitespace, including U+00A0 … is preserved verbatim", precisely *because* the spec authors recognise that an ambient "whitespace" word is ambiguous. The expressions table imports none of that discipline for `trim()`, so it is not clear whether `string.trim()` is meant to share the binder's ASCII-only set, match host `String.prototype.trim()` (which strips U+00A0, U+2000–U+200A, U+FEFF, etc.), or something else again.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the `trim()` row of the `string` table in `docs/spec_topics/expressions.md` to bind the semantics to the host method and add normative reference vectors, matching the pattern used by `toLowerCase()`/`toUpperCase()` and `replace`.

Concretely, in the `string` table, replace the current row

```
| `trim()` | `(): string` | Strips Unicode whitespace from both ends |
```

with

```
| `trim()` | `(): string` | JS semantics (`String.prototype.trim()`): strips leading and trailing code points whose Unicode `White_Space` property is `Yes`, plus the line-terminator code points U+000A, U+000D, U+2028, U+2029 — i.e. the set host `String.prototype.trim()` strips. No other normalisation; the interior of the string is preserved verbatim. |
```

and immediately after the `string` table (alongside the existing `replace(from, to)` reference-vectors block) add a small `trim()` reference-vectors block:

```
`trim()` reference vectors (normative; conforming implementations MUST reproduce these exactly):

| Expression | Result |
| --- | --- |
| `"  x  ".trim()` | `"x"` |
| `"\t\n\r x \t\n\r".trim()` | `"x"` |
| `"\u00A0x\u00A0".trim()` | `"x"` |
| `"\u000Bx\u000C".trim()` | `"x"` |
| `"\u2028x\u2029".trim()` | `"x"` |
| `"\uFEFFx\uFEFF".trim()` | `"x"` |
| `"a  b".trim()` | `"a  b"` |
| `"".trim()` | `""` |
```

The six "stripped" vectors pin the controversial code points that distinguish the host-`trim()` set from the ASCII-only binder set, from the regex `\s` set, and from `Zs`-only: U+00A0 (NBSP, in `White_Space`), U+000B/U+000C (in `White_Space`), U+2028/U+2029 (line-terminator code points host `trim()` strips), and U+FEFF (BOM/ZWNBSP, which host `trim()` strips). The `"a  b"` vector pins that interior whitespace is not collapsed; `""` pins the empty-string identity. Together they discriminate the host set from every plausible alternative.

Edge cases the implementer must watch:

- This is deliberately a **different** whitespace set from the binder's ASCII-only set in `defaulting-system-note-echo.md` rule 1 and the `slash-argument-whitespace` anchor. The binder set is narrow because it scrubs model output without molesting deliberate NBSPs the model emitted; `string.trim()` is a general-purpose user-facing call and should match the host method users intuit. Do not cross-import either definition into the other site.
- The host method strips `U+FEFF` (it is `White_Space` in JS's set despite *not* having the Unicode `White_Space` property); the vector above pins this so an implementation that follows the Unicode property table verbatim will fail the gate and be forced to use the host method.
- The lowering must call host `String.prototype.trim()` directly (no hand-rolled set), which is the simplest path to passing the vectors; the wording "JS semantics (`String.prototype.trim()`)" mirrors the `toLowerCase` row deliberately.

## Consequence

**Severity:** correctness

Two conforming implementations can choose different whitespace sets (host `String.prototype.trim`, ASCII-only à la the binder shards, or regex `\s`) and disagree on inputs containing U+00A0, U+000B, U+0085, U+FEFF, or U+2028. Loom programs that rely on `trim()` to normalise model output, slash-argument shrapnel, or pasted source will silently behave differently across implementations, and no `V3f-T` conformance assertion can catch it because the spec has no fixed answer to check against.
