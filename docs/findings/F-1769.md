## Problem

The `trim()` row of the `string` stdlib table in `docs/spec_topics/expressions.md` (under *Built-in methods and properties*) glosses the method as "Strips Unicode whitespace from both ends" without naming a host method, enumerating a whitespace set, or supplying reference vectors. "Unicode whitespace" is not a single well-defined set — the host `String.prototype.trim()` set, the regex `\s` set, and the `Zs` general-category set differ on edge-case code points such as U+0085, U+00A0, U+000B, U+000C, U+180E, and U+FEFF. Two conforming implementations can therefore pick different sets and disagree on inputs like `" \u00A0x\u00A0 ".trim()`. Every neighbouring row in the same table pins either a named host method or normative reference vectors; `trim()` is the only row that does neither.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** fae85d3 (`Tighten spec for implementation-plan readiness`, 2026-05-04)
**History:** The `string` stdlib table was authored in `fae85d3` as part of the May-2026 "stdlib precise semantics" pass on the then-monolithic `spec.md`. The `trim()` row was introduced in that commit with the wording "Strips Unicode whitespace from both ends" and has not been touched since — `git log -S "Strips Unicode whitespace"` returns only that commit and the two later structural moves (`fecb504` splitting `spec.md` into per-topic files; `31ff060` moving the corpus under `docs/`). The same commit pinned the neighbouring rows to "JS semantics" / "Locale-independent (`String.prototype.toLowerCase`)" and supplied the `replace` vectors, so the omission on `trim()` is a local authoring miss in the row that introduced the surface — not a regression from a later edit.

## Solution approach

Rewrite the `trim()` row's gloss in the `string` table to bind the method to host `String.prototype.trim()` semantics — mirroring the named-host-method pattern the `toLowerCase()` row already uses — so the stripped set is the host method's set rather than an ambient "Unicode whitespace". Add a normative `trim()` reference-vectors block alongside the existing `replace(from, to)` reference-vectors block after the `string` table, with vectors that discriminate the host set from the regex-`\s`, `Zs`-only, and ASCII-only alternatives on the controversial code points (U+00A0, U+000B/U+000C, U+2028/U+2029, U+FEFF) and that pin interior-whitespace-preserved and empty-string behaviour.

## Solution constraints

- Out of scope: do not cross-import or modify the binder whitespace set (`defaulting-system-note-echo.md` rule 1, `binder-bypass-and-envelope.md#slash-argument-whitespace`); `string.trim()`'s set is host-method-defined and intentionally distinct from the binder's ASCII-only set.
