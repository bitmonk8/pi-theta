## Finding

The QRY-5 "Newline trim" worked example in `query-forms.md` shows a multi-line template whose source lines are indented four spaces (raw markdown), then states the before-dedent intermediate render as `"  The author...\n  with...\n  Produce..."` — only **two** leading spaces. The two indentation widths disagree. Reading the raw source, the example reads as "four-space source indent dedents to a two-space intermediate," which contradicts dedent's actual behaviour (the full common prefix is stripped, not a fixed amount).

The disagreement is sharpened by QRY-7's normative vector table. Vector 1 — `` @`\n    The author...\n    with...\n    Produce...\n` `` → `"The author...\nwith...\nProduce..."` — is annotated "the worked example above, restated as a normative vector," and uses a **four**-space indent. So the same example is presented twice at two different indentation widths: the QRY-5 illustration (two-space intermediate) and the QRY-7 normative vector (four-space).

A complicating factor an editor must account for when fixing this: the QRY-5 example sits inside a `- **Newline trim.**` bullet, so markdown strips two spaces of list-continuation indent from the fenced block in the rendered view. In the *rendered* doc the template lines therefore display at two spaces, matching the two-space intermediate but disagreeing with the four-space QRY-7 vector; in the *raw* file the template lines are four spaces, disagreeing with the two-space intermediate. The defect is a three-way indentation inconsistency across (raw template source, before-dedent intermediate, QRY-7 vector 1), no two of which agree in both views.

Behaviour is not actually undecidable — QRY-7 declares itself the normative authority and its vectors are unambiguous — but the illustrative QRY-5 example is internally and cross-referentially inconsistent and can mislead a reader calibrating dedent off it.

## Solution Space

**Shape:** single

### Recommendation

Make the QRY-5 worked example agree with QRY-7 vector 1 at a four-space indent in the *rendered* view, so the illustration and its normative restatement match:

- Set the before-dedent intermediate to `"    The author...\n    with...\n    Produce..."` (four leading spaces), matching QRY-7 vector 1's four-space template.
- Ensure the template lines in the fenced `loom` block render at four spaces. Because the block is nested in the `- **Newline trim.**` bullet, markdown removes two spaces of list-continuation indent, so the raw source template lines must carry **six** spaces to display as four. (Equivalently, lift the example out of the bullet list so the raw four-space indent displays unchanged.)

After the edit, verify the rendered indentation of the template, the before-dedent intermediate string, and QRY-7 vector 1 are all four spaces. Edge case to watch: the list-continuation strip is the trap — editing the intermediate to four spaces while leaving the raw template at four spaces would fix the raw-file mismatch but reintroduce a two-space rendered template, so the raw indent must be adjusted in lockstep.

## Consequence

**Severity:** advisory

The shipped dedent behaviour is governed by the QRY-7 normative vectors, which are unaffected, so a careful implementer produces a correct system. The QRY-5 worked example, however, presents the same case at an indentation that disagrees with its own normative restatement and (in raw form) implies dedent strips a fixed two spaces rather than the full common prefix; a reader calibrating off the illustration can be momentarily misled.
