## Finding

`active-invocation-registry.md` closes the set of diagnostics the `session_shutdown` teardown handler may emit. The final sentence of a long Edge-cases bullet enumerates the five permitted codes — `loom/runtime/reload-teardown-timeout`, `loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, `loom/host/session-shutdown-runtime-degraded` — and ends with the closure MUST-NOT: *"No other diagnostic code may be emitted from this handler."* That closure is the conformance-testable obligation, yet it sits at the tail of an unanchored prose paragraph alongside the `sendSystemNote`-fallback prohibition and the `console.error` sink note, with no dedicated section anchor of its own.

The diagnostics corpus partly mirrors the *set*: `diagnostics/diagnostic-shape.md` enumerates the same five codes as "five carved-out exceptions," but it frames them around the delivery sink (`console.error`), not around the closure ("these are the only codes the teardown handler may emit"). The per-code registry pages that own these codes — `diagnostics/code-registry-host.md` (the four `loom/host/session-shutdown-*` rows) and `diagnostics/code-registry-runtime.md` (the `reload-teardown-timeout` row) — describe each code's trigger but carry no back-reference to a single authoritative allowlist/closure. The negative-closure conformance assertion therefore exists at exactly one buried, unanchored site.

The consequence is discoverability and drift risk: a future spec editor adding a sixth `loom/host/session-shutdown-*` code, or an implementer wiring an additional emission into the teardown path, has no anchored closure to consult or cross-reference from the registry pages where the codes are defined, and a conformance test asserting "the teardown handler emits only these five codes" has no normative anchor to cite.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commit:** `9559e19` ("pi-loom spec: resolve \"session_shutdown unknown reason: no closed-set validation, no fail-safe\"", 2026-05-07)
**History:** `git log -S "No other diagnostic code may be emitted from this handler"` first introduces the closure at `9559e19`, inside the then-monolithic `spec_topics/pi-integration-contract.md`, where it was authored as the tail sentence of the Edge-cases prose paragraph and listed two codes. It was authored buried — with no dedicated anchor and no mirror in the diagnostics pages — from inception. Subsequent commits grew the list (`14424b4` reached the present count) and the corpus-split commit `f5e89f4` (2026-06-04) relocated the paragraph verbatim into `active-invocation-registry.md`; neither changed the burial. The placement defect is therefore co-extensive with the closure's original introduction, not a product of the split.

## Solution Space

**Shape:** single

### Recommendation

In `active-invocation-registry.md`, lift the closed-set closure out of the Edge-cases bullet into a dedicated normative statement with its own `<a id>` anchor (e.g. `teardown-handler-diagnostic-allowlist`): list the five permitted codes and state the MUST-NOT ("the `session_shutdown` teardown handler MUST emit no diagnostic code outside this set"). Keep the `console.error`-sink and `sendSystemNote`-prohibition prose where it is — only the allowlist closure moves to the anchored rule.

Then back-reference that anchor from the registry pages that own the codes: each of the four `loom/host/session-shutdown-*` rows in `code-registry-host.md` and the `loom/runtime/reload-teardown-timeout` row in `code-registry-runtime.md` should link to the closure anchor, and `diagnostic-shape.md`'s "five carved-out exceptions" enumeration should cite it as the authoritative closed set rather than restating it independently.

Edge cases for the implementer: the closure governs only the teardown handler, not the per-invocation `finally` (`loom/runtime/cancelled-by-session-shutdown` is emitted from the invocation's own `finally`, not the handler, and must stay out of the allowlist); and the anchor name must be stable so the registry-page back-references and any conformance test can cite it.

## Consequence

**Severity:** advisory

The closure is stated, so a careful reader can still build a conforming handler; the defect is discoverability. Without an anchored, registry-cross-referenced closure, the "only these five codes" obligation is easy to violate silently when the teardown diagnostic set evolves, and a conformance test for the closure has no normative citation.
