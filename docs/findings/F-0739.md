## Finding

DISC-5 names `minimatch` as the matcher engine and pins three result-affecting knobs: the override order (plain → `!` → `+` → `-`), the three match targets (package-root-relative path, basename, POSIX-normalised absolute path), and `nocase: off`. The paragraph carries two coupled defects on the same matcher-contract surface, pulling in opposite directions:

- **Under-specification (correctness).** Every other minimatch option that materially changes which paths match is left at the engine default by silent omission — `dot`, `noglobstar` (consequential because DISC-5's starting set is "every path under the package root"), `nobrace` / brace expansion, and `matchBase` (DISC-5 already does basename matching as one of its three targets). The minimatch *major version* is also unpinned: defaults are not frozen across majors (v3→v9 changed `nonegate`/`noext` defaults and POSIX bracket handling), and H1a cites no version literal. Two implementers can each follow DISC-5 to the letter, install different majors with different default flags, and produce different match sets for the same `pi.looms` array.

- **Over-prescription (advisory).** DISC-5 grounds the contract by naming `minimatch` and citing `@earendil-works/pi-coding-agent`'s `dist/core/package-manager.js` symbol `applyPatterns` (and `matchesAnyExactPattern`) inside normative text. The genuine requirement is *behavioural parity with the Pi resource-array glob dialect and override ordering*. Naming the library locks a loom implementer to `minimatch`; the `dist/`-internal, unexported symbols read as importable when they are not in the SDK inventory — so DISC-5's own "delegate to `applyPatterns`" escape is closed off, leaving reimplementation against minimatch as the only path, which needs the option set DISC-5 does not enumerate.

The two pulls are a genuine fork on one paragraph. Independent of which is chosen, the unimportable internal-symbol citations should not remain inside normative text.

## Solution Space

**Shape:** multiple

### Option A — Enumerate the option set DISC-5 hands to minimatch and pin the major version

**Approach.** Extend DISC-5 with a closed enumeration of every result-affecting minimatch option loom configures, alongside `nocase: off`; fix each to a literal value with a one-line rationale. Pin the minimatch major in `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)", parallel to `semver`/`chokidar`/`yaml`. Regardless of this choice, demote the `dist/core/package-manager.js` / `applyPatterns` / `matchesAnyExactPattern` references to a non-normative note.

**Pros.** Closed enumeration: implementers and reviewers can mechanically check option-by-option; version-bump tests can assert the installed major and fail loudly on drift.

**Cons.** Couples the spec to a specific matcher library more tightly; a future engine swap must re-render the option set. Adds prose to a dense paragraph.

**Risks.** A non-exhaustive option list lets the divergence resurface; each minimatch major bump requires re-auditing the enumerated values.

### Option B — Reframe DISC-5 as behavioural parity with Pi's resource-array glob dialect

**Approach.** Demote the literal `minimatch` / `applyPatterns` references to a non-normative note and state the normative requirement as parity with Pi's resource-array dialect over the three match targets and the override order DISC-5 already fixes. Define parity observationally; leave the option set unenumerated (implied by parity). Resolves the over-prescription concern directly.

**Pros.** Survives a future Pi-side matcher swap without a spec change; sidesteps the `dist/`-internal reachability problem; keeps `pi.looms` indistinguishable from Pi's other resource arrays.

**Cons.** Replaces a finite enumeration with an unbounded conformance problem: "matches Pi" is not directly testable without a Pi instance, so V10b tests must ship a corpus of `(pattern, candidate-path, expected)` triples or import Pi at test time.

**Risks.** Pi's dialect is not specified in the public docs — `applyPatterns` lives at the same `dist/`-internal path — so the parity contract reduces to "matches whatever Pi's compiled `dist/` does today".

### Recommendation

The two findings merged here pulled opposite directions on the same paragraph and must be resolved together by picking one option. The testability balance favours **Option A**: DISC-5 already pins one option and three match targets, so extending it plus pinning the major closes the divergence with one edit V10b-T can mechanically test; Option B's parity contract has no testable referent today. The over-prescription concern that motivates Option B is partly valid and should be honoured under either option by moving the unimportable `dist/core/package-manager.js` / `applyPatterns` / `matchesAnyExactPattern` citations into a non-normative note.

Edge cases (Option A): author the option set against a *specific* minimatch major; minimatch's default `dot: false` means `*.loom` does not match `.foo.loom` — state whether intended and pin it; pin `noglobstar` consistent with the resolved recursion-depth rule; pick a `matchBase` value and assert it in V10b; name the pinned major in `implementation-notes.md` (e.g. "minimatch v9").

## Consequence

**Severity:** correctness

Two implementers each following DISC-5 to the letter can produce divergent match sets for the same `pi.looms` array — most visibly for `**`, leading-dot segments, or `{a,b}` braces. The divergence is silent: each passes its own unit tests, and only cross-implementation comparison surfaces it. Separately, a literal reading brittles loom to Pi's current internal matcher choice and mis-suggests an unexported Pi symbol is part of loom's contract surface.
