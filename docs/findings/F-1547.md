## Finding

The **Runtime event channel** section of `runtime-event-channel.md` closes the **Deduplication and lifetime rules** list with a two-sentence future-work block:

> "Aggregation, latency histograms, per-loom token reports, retention policies, and a consumer-facing read API for the event stream are deferred to [Future Considerations](../future-considerations.md). loom 1.0 ships the always-log set and the payload shape only; the channel is intentionally write-only at the spec level, with downstream consumers free to read from Pi's session transcript via existing surfaces."

This enumeration of deferred surfaces and the loom-1.0 scope restatement are already owned, in full, by `future-considerations/surface-extensions.md` → **Richer runtime-event telemetry** ("per-loom token aggregation, latency histograms, cost reporting, retention policies, and a consumer-facing read API for the `details: { event }` stream … loom 1.0 ships only the per-occurrence emission and the `RuntimeEvent` payload shape; downstream consumers in loom 1.0 read from Pi's session transcript via existing surfaces"). The inline copy adds no normative obligation that the surface-extensions entry does not already carry; it restates a deferral list and a scope boundary that live elsewhere, embedded directly among the page's normative dedup/lifetime rules.

The two copies have already drifted: the inline list says "per-loom token reports" with no cost-reporting item, while the surface-extensions entry says "per-loom token aggregation" plus "cost reporting". A reader cannot tell which enumeration is authoritative, and the duplication guarantees the two will continue to diverge as the deferral list evolves.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** b3bc4ce — spec: pin failure-observability surface (Cluster A, Option C) (2026-05-04, Thomas Andersen)
**History:** The Runtime event channel section was introduced wholesale by b3bc4ce, which simultaneously added the inline deferral sentences (the "intentionally write-only at the spec level" / "consumer-facing read API for the event stream" prose, then in `pi-integration-contract.md`) and the `future-considerations.md` deferral entry. The duplication was therefore born with the section in a single commit; the later size-cap split (f5e89f4) only carried the inline copy into `runtime-event-channel.md` and the deferral entry into `surface-extensions.md` without removing the overlap.

## Solution Space

**Shape:** single

### Recommendation

Replace the two inline sentences with a bare anchored cross-reference to the canonical owner — e.g. "Aggregation, retention, per-loom telemetry, and a consumer-facing read API for this channel are out of loom 1.0 scope; see [Future Considerations — Richer runtime-event telemetry](../future-considerations/surface-extensions.md)." Do not re-enumerate the deferred surfaces here; the surface-extensions entry is the single source for that list. Retain the one load-bearing scope fact that downstream consumers in loom 1.0 read from Pi's session transcript via existing surfaces only if it is not already adequately carried by the cross-reference target (it is — surface-extensions states it verbatim), so the cleaner form is the bare cross-reference with no restated enumeration.

## Consequence

**Severity:** cosmetic

Pure organisation/duplication; no implementer behaviour changes. The cost is drift risk and reader friction — the deferral list already disagrees across the two locations, so a reader at the runtime-event-channel page may believe a different set of surfaces is deferred than the canonical Future Considerations owner declares.
