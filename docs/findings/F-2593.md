## Finding

Package discovery (`docs/spec_topics/discovery/package-and-settings.md` §Package discovery) treats every non-`@` immediate child directory of a scanned root as a candidate package and states only the positive condition: "A candidate package contributes looms only when its `package.json` parses successfully." The spec never states what happens when a candidate's `package.json` is **absent**, **unreadable** (EACCES / transient I/O, distinct from the DISC-6 per-read *timeout*), or **present but malformed JSON**. The behavioural outcome ("no looms from that package") is implied, but the *diagnostic disposition* — silent skip vs. warning vs. error — is undefined for all three cases.

This is an asymmetry against the sibling settings-file reader in the same file, whose §"Settings file reads → Failure modes" block defines exactly these cases: missing/unreadable → `loom/load/settings-unreadable`; present-but-invalid JSON → `loom/load/settings-invalid-json`. The package manifest reader has no analogue. The diagnostic registry confirms the gap: `code-registry-load.md` carries `loom/load/manifest-invalid` (the `pi.looms` *value* is not a `string[]`, judged after a successful parse), `loom/load/manifest-escapes-package` (a `pi.looms` entry escapes the package root), and `loom/load/package-read-timeout` (the DISC-6 per-read deadline), but no code for an absent / unreadable / malformed `package.json` read itself.

The case is not hypothetical: candidate enumeration admits any immediate child whose name does not begin with `@`, so directories such as `node_modules/.bin`, `.cache`, or any non-package directory become candidates with no `package.json`. Because the diagnostic registry is closed, an implementer cannot emit a code that is not registered, so the malformed/unreadable cases are forced silent — diverging from the settings analogue's deliberate surfacing of the same condition.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** d6cbb37 — pi-loom spec: resolve "pi.looms is an extension-owned convention" (2026-05-04, Thomas Andersen)
**History:** Package discovery was first specified at d6cbb37, which introduced "every immediate child whose `package.json` parses successfully is treated as a candidate package" together with the `loom/load/manifest-invalid` and `loom/load/manifest-escapes-package` codes — establishing the positive parse condition and the post-parse `pi.looms` value checks but never a disposition for an absent / unreadable / malformed manifest read. Later edits refined the surrounding rules (f7a0dae rephrased the clause to "contributes looms only when its `package.json` parses successfully"; the DISC-6 work added `loom/load/package-read-timeout` for the per-read *timeout* case only), but none added the missing/unreadable/malformed disposition, so the gap has been present since the section's first commit.

## Solution Space

**Shape:** single

### Recommendation

In `package-and-settings.md` §Package discovery, add the manifest-read disposition for a candidate `package.json`, mirroring the conventional-location asymmetry DISC-2 already establishes (a scanned candidate is not author-named, so absence is the normal case):

- **Absent** (`ENOENT` on the candidate's `package.json`): silently skipped — the directory is simply not a package; no diagnostic. This matches the "conventional locations silently tolerate absence" rule in DISC-2 and the existing silent-skip for non-existent roots.
- **Unreadable** (EACCES / transient I/O on the read itself, distinct from the DISC-6 per-read timeout): a single non-fatal load-time **warning**; the package contributes no looms and the walk continues with the next candidate.
- **Present but malformed JSON** (parse failure): a single non-fatal load-time **warning**; the package contributes no looms and the walk continues.

Register one load-phase code (or one per case) in `code-registry-load.md` for the unreadable/malformed dispositions, symmetric to the existing `loom/load/settings-unreadable` / `loom/load/settings-invalid-json` pair, with the source descriptor naming the package (e.g. `` "package `foo` (package.json)" ``). Warning severity (not error) matches the conventional-location asymmetry, since the candidate was auto-scanned rather than explicitly named by an author.

Edge cases for the implementer: keep this rule distinct from `loom/load/package-read-timeout` (which already owns the per-read *deadline* trip and treats the package as unreadable "for this scan only"); the malformed/unreadable outcome here is likewise per-scan and re-attempted on a subsequent reload. A candidate directory beginning with `.` (e.g. `.bin`) does not begin with `@`, so it enters candidate enumeration and hits the absent-manifest arm — confirm the silent-skip arm covers it without noise.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the diagnostic stream for a candidate `package.json` that is unreadable or malformed: one stays silent (forced by the closed registry), another registers a new code and warns, mirroring the settings analogue. Conformance tests assert on diagnostics, so the divergence is observable; a corrupt or permission-denied manifest in a dependency tree disappears with no operator signal under one implementation and surfaces under another.
