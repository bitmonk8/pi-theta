## Finding

The spec's canonical label for PIC-49 is **"Cross-window rebuild serialization"** (`docs/spec_topics/pi-integration-contract/registration-steps.md`, anchor `#pic-49`). The paired tests leaf `V10d-T` and `V10d`'s own Tests bullet both name the obligation "cross-window rebuild serialization", matching that label. `V10d`'s **Adds** field, however, introduces the divergent noun phrase "the across-window rebuild-serialization guard" (different preposition `across-` vs `cross-`, an added hyphen between "rebuild" and "serialization", and an appended "guard").

The drift is internal to `V10d`: the same leaf describes the same obligation two ways — "across-window rebuild-serialization guard" in Adds, "cross-window rebuild serialization" in Tests and Ships-when. The PIC-49 token is cited inline in the Adds, so an implementer can still resolve to the spec; the cost is terminology inconsistency that defeats grep/term-based audits keyed on the canonical "cross-window rebuild serialization" phrase. (The spec body does use the adjective "across-window" once, but only descriptively — "this rule governs only the across-window case" — never as the obligation's name; the obligation's name is the `#pic-49` heading "Cross-window rebuild serialization".)

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `9a97573`
**History:** The string "across-window rebuild-serialization guard" was authored in `9a97573` (2026-06-27, "pi-loom spec-plan: fix F-1891 — close PIC-49 cross-window rebuild serialization on V10d; coverage-matrix row"); `git log -S 'across-window'` and `git log -G 'across-window rebuild-serialization guard'` over `docs/plan_topics/V10d-reload-debounce.md` both report that commit as the only one touching the token, and it is the commit that first introduced PIC-49 closure on `V10d`. The same commit's diff wrote the Tests bullet as "cross-window rebuild serialization" — so the Adds/Tests divergence was present at the moment PIC-49 was attached to the leaf and has not been edited since. The defect has been present since `V10d`'s PIC-49 closure was authored.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V10d-reload-debounce.md`, in the **Adds** field, rename the obligation to match the canonical PIC-49 label and the term already used by `V10d`'s Tests/Ships-when and by `V10d-T`. Concretely, change the clause `It also adds the across-window rebuild-serialization guard at the debounce→rebuild boundary (`PIC-49`):` so the phrase reads `cross-window rebuild serialization guard` — i.e. `across-window` → `cross-window` and drop the hyphen so "rebuild serialization" matches the spec heading "Cross-window rebuild serialization", with "guard" retained as the trailing descriptor. No other field changes; `V10d-T`, the Tests bullet, and Ships-when already carry the canonical term.

## Consequence

**Severity:** cosmetic

A reader of `V10d` sees the same PIC-49 obligation named two ways within one leaf and named differently from the spec heading; a grep-based term audit keyed on "cross-window rebuild serialization" misses the Adds occurrence. The PIC-49 token is cited inline, so no implementer diverges and no gate is affected — the defect is pure wording.
