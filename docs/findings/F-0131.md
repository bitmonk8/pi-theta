## Finding

The leaf pair `docs/plan_topics/V5d-subset-lowering.md` /
`docs/plan_topics/V5d-T-subset-lowering.md` carries the path slug
`subset-lowering`, but V5d's title and Adds scope it to the
**schema-subset reject gate** only. The lowering pass and canonical-hash
recipe were split out to `V5f`, whose own file
(`V5f-subset-lowering-hash.md`) correctly names lowering. V5d's Adds even
disclaims that scope explicitly: "The lowering pass and the canonical-hash
recipe … are owned by `V5f`."

The result is two files whose slugs both advertise "subset lowering"
(`V5d-subset-lowering` and `V5f-subset-lowering-hash`) while only V5f
actually owns lowering. A reader navigating the `plan_topics/` directory
or following a TOC link by filename cannot tell which leaf owns the
lowering pass; the slug points at the wrong leaf.

The leaf-ID `V5d` and all `Deps.` citations (which reference the bare ID,
not the filename) are correct and unaffected — the defect is confined to
the file path slug and the two hyperlinks that target it.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** cb8737a — pi-loom plan: resolve "Systemic leaf over-bundling across the plan corpus" (2026-06-12, plan author)
**History:** At inception (c6a664e) V5d was a single leaf titled "Schema-subset gate, lowering, and canonical hash" that genuinely owned the reject gate, the lowering pass, and the canonical hash — so the `subset-lowering` slug was accurate then. Commit cb8737a de-bundled the leaf: it created `V5f-subset-lowering-hash.md` to own lowering + canonical hash and renamed V5d's title to "Schema-subset reject gate", but left V5d's filename slug `subset-lowering` unchanged, stranding it on the narrowed scope.

## Solution Space

**Shape:** single

### Recommendation

Rename the V5d leaf pair so the slug reflects its narrowed scope:

- `docs/plan_topics/V5d-subset-lowering.md` → `docs/plan_topics/V5d-reject-gate.md`
- `docs/plan_topics/V5d-T-subset-lowering.md` → `docs/plan_topics/V5d-T-reject-gate.md`

Then retarget the two inbound hyperlinks that reference the old path:

- `docs/plan.md` line 84 — change the V5d TOC link target from
  `./plan_topics/V5d-subset-lowering.md` to `./plan_topics/V5d-reject-gate.md`.
- `docs/plan_topics/V5f-subset-lowering-hash.md` Adds — change the
  `[`V5d`](./V5d-subset-lowering.md)` link target to `./V5d-reject-gate.md`.

Leave the leaf ID `V5d` / `V5d-T` unchanged: every `Deps.` citation and
the `coverage-matrix.md` row (`schema-subset.md (SUBS) | V5d, V5e, V5f`)
reference the bare ID and require no edit. A repo-wide grep for
`V5d-subset-lowering` and `V5d-T-subset-lowering` after the rename must
return zero hits.

## Consequence

**Severity:** advisory

A reader using the filename to locate the lowering-pass owner is
misdirected to V5d, which no longer owns it; the two `subset-lowering`
slugs compete for the same concept. The leaf bodies disambiguate, so no
implementer produces a wrong leaf — this is a wayfinding/navigation gap,
not a correctness defect.
