## Problem

`diagnostic-shape.md` states the `loom-system-note` renderer MUST be registered synchronously inside the extension factory **before** the first discovery scan kicks off, so the first batch of scan diagnostics renders through the loom-specific renderer. Both halves are unverifiable: "the first discovery scan kicks off" is an internal runtime event with no host-API proxy a conformance test can timestamp against the `pi.registerMessageRenderer` call, and the "renders correctly" rationale is circular — observing that the renderer received the message requires the renderer to have already been registered. The same ordering rule already has an observable pin one layer down at `extension-bootstrap-and-per-loom.md` §Renderer registration → *Registration timing* ("synchronously inside the factory body, before the factory returns"), which a probe can test at factory-settle. The diagnostic-shape sentence shadows that testable pin with a softer, untestable restatement.

## Solution approach

Rewrite the `before the first discovery scan kicks off` sentence in `diagnostic-shape.md` to defer to the observable ordering pin at `extension-bootstrap-and-per-loom.md` §Renderer registration → *Registration timing* (anchor `#renderer-registration`), forward-linking to it rather than restating the rule. Drop the unobservable "first discovery scan kicks off" anchor and the circular "renders correctly" rationale.
