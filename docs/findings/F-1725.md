## Finding

The *Settings file reads* paragraph in `discovery/package-and-settings.md` explicitly pins its resolution base — the project `.pi/settings.json` resolves against the factory-time working directory supplied by the `FileSystem.cwd()` seam, with the `resources_discover` event substituting `event.cwd` — and cross-links the `FileSystem.cwd()` seam member that supplies it. Three sibling sites that resolve paths against the **same** base do not state the anchor at all:

- The *Roots scanned* list in §Package discovery enumerates `.pi/npm/`, `.pi/git/<host>/<path>/`, and `node_modules/` as bare relative paths.
- The *Discovery sources* bullet list and the *Discovery roots* paragraph in `discovery/discovery-sources.md` list `.pi/looms/` as a bare relative path.
- The walk-disabled clause in DISC-6 re-mentions `node_modules/`, `.pi/npm/`, and `.pi/git/` without an anchor.

The base is *recoverable* — the host-interfaces-services.md `cwd()` bullet names "the factory-time project-local discovery root (`.pi/looms/`) and project `.pi/settings.json` resolution" as the seam's load-bearing consumers, and the `#cwd-project-root-presupposition` paragraph repeats `.pi/looms/` — but the package-root paragraphs are nowhere claimed by that seam, and a reader of the discovery shards alone has to infer the anchor. An implementer following only the discovery-shard text could legitimately resolve `node_modules/` against `process.cwd()` directly, against the loom-package's own install directory, or via Node's module-resolution algorithm; the spec never says which.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** c6e464a (`pi-loom spec: resolve "Registration step 1 — factory-time working-directory source unstated"`, 2026-06-04)
**History:**

The package-root list (`.pi/npm/`, `.pi/git/<host>/<path>/`, `node_modules/`, `~/.pi/agent/npm/`, `~/.pi/agent/git/`) entered the corpus in d6cbb37 (`pi-loom spec: resolve "pi.looms is an extension-owned convention"`, 2026-05-04) as the **Roots scanned** enumeration on `spec_topics/discovery.md`, with no explicit base anchor. The project discovery root `.pi/looms/` predates the per-topic split (fecb504, 2026-05-04) and was likewise bare-relative throughout. At that point the *Settings file reads* paragraph was symmetric — it read "resolved relative to the working directory" with no seam citation either, so all three sites were uniformly under-specified rather than asymmetric.

c6e464a (2026-06-04) tightened only the settings-reads paragraph, rewriting `(resolved relative to the working directory)` as `(resolved relative to the factory-time working directory supplied by the FileSystem.cwd() seam member; on a resources_discover event the project root is event.cwd instead, per Registration steps step 1)` and cross-linking the new `cwd-project-root-presupposition` anchor on host-interfaces-services.md it introduced in the same commit. The sibling paragraphs at §Package discovery *Roots scanned* and the discovery-sources lists were not touched. The defect-as-currently-formulated (asymmetric grounding — one site cites the seam, the rest don't) arose at exactly that moment; the underlying latent under-specification is older but was not previously visible as an *inconsistency*.

## Solution Space

**Shape:** single

### Recommendation

State the shared base once and cite it from each site that lists project-scope paths.

In `docs/spec_topics/discovery/package-and-settings.md` §Package discovery, immediately before the **Roots scanned** list, insert a sentence: *"The project-scope roots below (items 1–3) and the project discovery root `.pi/looms/` resolve against the same factory-time working directory supplied by the [`FileSystem.cwd()` seam member](../pi-integration-contract/host-interfaces-services.md#cwd-project-root-presupposition); on a `resources_discover` event the project root is `event.cwd` instead, per [Registration steps](../pi-integration-contract/registration-steps.md) step 1. The global-scope roots (items 4–5) resolve via [Home-directory expansion](./discovery-sources.md#home-directory-expansion)."* Phrase it as a forward reference so the numbered list itself stays unchanged.

In `docs/spec_topics/discovery/discovery-sources.md`, at the **Discovery roots** paragraph, add the same one-line cross-link for `.pi/looms/` rather than restating the rule: *"The project root `.pi/looms/` resolves against the same `FileSystem.cwd()`/`event.cwd` base as `.pi/settings.json` — see [Package discovery — resolution base](./package-and-settings.md#package-discovery)."*

Edge cases the implementer must watch:

- The wording must say "project-scope roots (items 1–3)" / "global-scope roots (items 4–5)" so the `~`-prefixed roots stay under the home-directory-expansion rule and are not double-anchored.
- The DISC-6 walk-disabled clause (`no node_modules/, .pi/npm/, .pi/git/, ~/.pi/agent/npm/, or ~/.pi/agent/git/ root is scanned`) does not need its own anchor — it inherits the §Package discovery framing — but a reviewer should check the new sentence's cross-link still resolves under the §Package discovery header it sits inside.
- The `cwd-project-root-presupposition` paragraph on `host-interfaces-services.md` currently names only `.pi/looms/` and `.pi/settings.json` as its consumers; once the package roots cite the seam, the consumer list in that paragraph should pick them up too, so the bump-checklist item (w) re-audit surface stays accurate. (This sibling tightening is the same-cluster edit owned by the "`cwd()` project-root presupposition homed in a DI-seam bullet" finding.)

## Consequence

**Severity:** advisory

Two reasonable implementers reading the discovery shards in isolation could anchor `node_modules/`, `.pi/npm/`, and `.pi/git/` against different bases — `process.cwd()` ad-hoc, loom's own install directory, or Node's module-resolution algorithm — producing divergent package-walk behaviour in any host that launches the extension factory from a directory other than the project root. The asymmetry with the explicitly-grounded settings paragraph invites the inference "the others are deliberately different" rather than "the others were left implicit."
