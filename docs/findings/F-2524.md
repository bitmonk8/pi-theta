## Finding

The entire frontmatter field contract is built on a typed YAML scalar model: the runtime must distinguish a string scalar from a number, boolean, or null in order to decide observable behaviour. The `mode:` row splits a missing key (`loom/load/missing-mode`) from a present-but-unrecognised value (`loom/load/unknown-mode-value`); the `bind_context:` and `respond_repair.methodology:` rows treat "a non-string scalar" as the present-but-bad arm; the `model:` row routes "a non-string scalar" to `loom/load/model-unresolved`. FRNT-1 goes further and pins a number-resolution rule — `max_rounds` "integer-ness is judged on the parsed numeric value, not the lexical form (`25` and `25.0` are both accepted)", with `"25"` rejected as a non-number scalar. Every one of these decisions presupposes a specific scalar-resolution outcome for a given source token.

That outcome is not implementation-neutral. Whether a bare token resolves to a string, a boolean, a number, or null depends on the YAML schema the parser applies. The YAML 1.1 type library resolves `yes`/`no`/`on`/`off` and `y`/`n` to booleans and admits sexagesimal and octal-prefixed integers; the YAML 1.2 core schema resolves only `true`/`false`/`null`/`~` and a narrow numeric grammar, leaving the rest as strings. So `bind_context: none`, `model: no`, or a numeric-looking `max_rounds: 010` resolve differently — string vs boolean vs differently-valued integer — and therefore fire different diagnostics (or none) under the two schemas.

`implementation-notes.md` names the `yaml` npm package as the parsing mechanism but states explicitly that "this clause names the dependency, not its version", and no page names the scalar-resolution schema (1.1 vs 1.2 core) the contract assumes. The dependency's *version* is legitimately an implementation choice, but the scalar-resolution schema is observable: it determines which diagnostic fires for a given author keystroke. The contract pins the consequences (the per-field recognised-value splits, the `25`/`25.0` rule) without pinning the premise those consequences rest on.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 24a8079 (2026-05-07) — first reliance on a parsed-numeric scalar model (the `max_rounds` integer-ness rule, `25`/`25.0` both accepted); 5f5c487 (2026-06-03) — extended the typed-scalar reliance with the `mode:`/`bind_context:`/`model:` non-string-scalar recognised-value splits; 1064946 (2026-06-10) — named the `yaml` dependency but scoped itself to "names the dependency, not its version", a near-miss that left the scalar-resolution schema unpinned.
**History:** The typed-scalar premise has been load-bearing since the frontmatter contract first encoded value-shape-dependent diagnostics. `git log -S` over the cited tokens shows the parsed-numeric rule entering at 24a8079 and the non-string-scalar splits at 5f5c487, neither accompanied by a named YAML schema. The one commit that touched the adjacent gap (1064946, resolving "YAML frontmatter parsing mechanism is never declared as a dependency") added the `yaml` dependency clause but deliberately declined to pin a version, and pinned no scalar-resolution schema either — so the omission was visible at that point and left open. The defect is an inception-level omission rather than a regression introduced by a single later commit.

## Solution Space

**Shape:** single

### Recommendation

Pin the YAML scalar-resolution schema as normative behaviour at the frontmatter contract owner (`frontmatter-fields-a.md`, in the Field-contract section that introduces the table, or equivalently the `frontmatter.md` YAML-frontmatter intro): the runtime resolves frontmatter scalars per the **YAML 1.2 core schema** — `true`/`false` are booleans, `null`/`~` (and the empty value) are null, the 1.2 core numeric grammar yields numbers, and every other unquoted scalar is a string. State this as a behavioural conformance point independent of which library or version implements it, so the parsing-library version stays an implementation choice (preserving the existing `implementation-notes.md` "names the dependency, not its version" stance) while the observable scalar-resolution outcome is fixed.

Make two facts that the contract already depends on explicit under that schema: (a) `bind_context: none`, `model: <bare-word>`, and similar bare tokens resolve to strings (so the recognised-value / non-string-scalar splits behave as the per-field rows describe), and (b) FRNT-1's "integer-ness judged on the parsed numeric value" reads `25` and `25.0` as the same `number` under the 1.2 core numeric grammar. Have FRNT-1 (`frontmatter-fields-b-and-templates.md`) and the parsed-scalar rendering note in `placeholder-rendering-b.md` cross-reference the pinned schema rather than restating scalar-resolution behaviour. Edge case for the implementer: YAML 1.2 core leaves `yes`/`no`/`on`/`off` as strings — the pin must call this out so a `mode: yes` or `bind_context: no` is unambiguously a present-but-unrecognised *string* value (its existing recognised-value diagnostic), never a boolean.

## Consequence

**Severity:** correctness

Two reasonable implementers — one configuring the `yaml` parser to the 1.1 type library, one to the 1.2 core schema — diverge on whether tokens like `bind_context: none`, `model: no`, or `max_rounds: 010` are strings, booleans, or differently-valued numbers, and therefore on which load diagnostic fires (or whether the loom loads at all). The same author-authored `.loom` file is accepted by one conforming runtime and rejected by another.
