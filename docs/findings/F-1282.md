## Problem

The spec leaves unspecified what happens to already-loaded, in-scope, non-bypass, `bind_model:`-absent looms whose binder model was resolved from the prior `looms.binderModel` value when a settings edit changes only `looms.binderModel`. `docs/spec_topics/binder/binder-model-and-context.md` §`binder-model-hot-reload` says the change "re-resolves on the next loom load" and adds a recovery `loom-system-note` for the success branch, but does not pin whether already-loaded looms are re-resolved and re-probed at debounce-window close (and dropped on failure) or instead retain their previously-resolved handle until the next `/reload`. The same gap leaves the symmetric loss case open: nothing states whether a loom whose binder model previously resolved but would now fail produces a loss-side note. Two implementers can ship opposite user-observable behaviour after a `looms.binderModel` typo.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `0fa81fa` (spec: pin binder model resolution; document envelope two-arm rationale (Decisions #5, #6); 2026-05-05)
**History:** The hot-reload paragraph entered the corpus in `0fa81fa` as a single sentence — "Hot-reload of Pi settings (`looms.binderModel` changed at runtime) re-resolves on the next loom load; it does not retroactively fix already-failed loads." — which was already silent on what happens to already-loaded looms whose binder model was resolved from the prior value. Subsequent refinements narrowed the recovery branch without filling the gap: `319228c` (2026-05-05) added the `loom-system-note` recovery prompt; `f88698e` (2026-06-05) expanded the note into the full template with `<N>`/`<names>` substitution; `97430aa` (2026-06-06) tightened the membership predicate to "previously binder-model-unresolved" looms and added the disclaimer that membership makes no claim that `/reload` will succeed. In parallel, `6d1a949` (2026-06-05) added the structural-change suppression rule that exempts `looms.binderModel`-only edits from the structural-change note, sharpening the contrast with the recovery-side spec without normatively pinning the no-re-resolution semantics for already-loaded looms. Across all four edits the loss-side case (a previously-successful loom whose new resolved model fails) and the in-flight re-validation question were never normatively addressed.

## Solution approach

Add a normative clarification to §`binder-model-hot-reload`, after the existing "re-resolves on the next loom load" sentence, pinning that a `looms.binderModel`-only edit does not re-run binder-model resolution or the `strictCapable` probe for any already-loaded loom — each continues using its previously-resolved handle until its next load (`/reload`, or a content-edit reload per `#watcher-hot-reload-registration`), and the new value is consulted only at that next load. State that no symmetric loss-side `loom-system-note` is emitted: a binder model that would now fail surfaces only at the next load, through that load's own `loom/load/binder-model-unresolved` / `loom/load/binder-model-not-strict-capable` diagnostic. Mirror the existing suppression rule at `registration-steps.md` §`structural-changes-no-unregister`.

## Solution constraints

- Out of scope: the existing recovery-note membership predicate (looms whose original failure was `loom/load/binder-model-unresolved` and whose binder model now re-resolves) — do not widen it.
