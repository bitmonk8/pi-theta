## Finding

Step 5 of `docs/spec_topics/pi-integration-contract/version-bump-triggers.md`
mandates a build-time audit gate that re-runs the two-stage triplet /
complement grep over `src/`, `test/`, and `docs/` on each `npm test`
and "fails red on a result-delta against the documented enumeration";
the same paragraph requires the audit to "apply the same
documented-enumeration exclusion list the contributor-side recipe
applies (so the transiently-stale `docs/` restatements [step 1]'s
carve-out classifies — for instance the `docs/spec.md` SM-2 listing —
are not reported as deltas)". The "documented enumeration" and the
"exclusion list" exist only as English prose embedded in this step
(and in step 1's carve-out it points at). The spec never names a
machine-readable artefact that holds them, never says what key the
audit matches each enumerated/excluded entry against, and never says
how the prose listings reduce to the artefact the audit consumes.

This is load-bearing. The audit's red/green decision is, by
construction, *exclusion-list membership* (`hit ∈ excluded ⇒ no-op`,
`hit ∉ excluded ⇒ delta`). Two reasonable implementers will pick
different artefact shapes (line-keyed `path:line` lists; symbol-keyed
quadruples of `(path, anchor, triplet-member, second-stage anchor)`;
a pickle of regex matches; an in-test inline literal table) and each
shape has a different rot profile: a line-keyed list silently
mis-excludes on every unrelated edit to the cited file and produces a
perpetually-red gate; a coarse path-keyed list silently authorises
new inbound runtime references added to the same file and produces a
silently-non-firing gate. Neither failure mode surfaces at the audit
itself — it just emits the wrong verdict.

The same gap applies to the symmetric `(quit|reload)` complement pass
in the next sentence ("MUST also be run on each bump"), which inherits
whatever artefact answer step 5 picks but is never told it does.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** c0259ca — pi-loom spec: resolve "Version-bump procedure: four obligations have no verifiable acceptance criterion" (2026-06-07, Thomas Andersen)
**History:** Prior to `c0259ca` the build-time audit was opt-in
(`The implementation MAY additionally implement a build-time audit pass
over the same three grep targets so a result-delta against the
documented enumeration fails npm test rather than relying on
contributor-side discipline; PIC does not require the audit`) and the
exclusion list was not normatively named. `c0259ca` promoted the
audit from `MAY` to `MUST`, added the explicit `npm test` red, and
introduced the "audit MUST apply the same documented-enumeration
exclusion list the contributor-side recipe applies" sentence — but
without defining where the documented enumeration and exclusion list
live as an artefact, what they store, or how the audit matches a grep
hit against them. The gap is wholly produced by that single
`MAY → MUST` escalation: the previous MAY framing tolerated the
under-specification because PIC owned no artefact contract; once the
gate became mandatory the missing artefact contract became
load-bearing.

## Solution Space

**Shape:** single

### Recommendation

In the *Enumeration mechanism — grep target* paragraph, immediately
after the sentence introducing the build-time audit gate, pin the
exclusion-list artefact's three properties:

1. **Storage site.** The exclusion list is stored as a constant in the
   same source-of-truth pinned-constants block in the extension module
   that holds the four Step-0-probe constants, the
   `SessionShutdownEvent['reason']` snapshot entry, and the typebox
   allow-lists (groupings (i)–(iii) of this step). Add a fifth
   grouping `(v)` for the inbound-reference exclusion list, on the
   same footing — co-edited under the same step-5 trigger discipline,
   and updated in the same commit that introduces a new transiently-
   stale `docs/` restatement step 1's carve-out classifies.

2. **Match key.** Each entry is a `(path, second-stage-anchor)` pair:
   the relative path of the file the restatement lives in (matching
   the contributor-side recipe's normalised relative path), and the
   one second-stage-anchor token (`event.reason` / `SessionShutdownEvent`
   / `capturedEventReason` / `entry.shutdownReason`) that the hit
   line carries. The audit matches a grep hit against the list by the
   pair (the path the hit came from, the second-stage anchor whose
   regex it matched in stage 2); a line-keyed match is explicitly
   rejected (lines rot on unrelated edits) and a path-only match is
   explicitly rejected (silently authorises new inbound references in
   the same file).

3. **Reduction rule.** The list is the closed enumeration of every
   site step 1's transiently-stale `docs/` carve-out classifies as
   no-op (loom 1.0 contents: one entry, `(docs/spec.md, event.reason)`
   pointing at the SM-2 listing). The contributor's obligation in the
   *Enumeration mechanism* paragraph's third arm — "a hit at an
   illustrative restatement that step 1's carve-out classifies as
   transiently stale … is likewise a no-op" — reduces mechanically to
   "the `(path, anchor)` pair of the hit is in this constant"; the
   contributor MUST add a new pair to the constant in the same edit
   that lands a new transiently-stale restatement, on the same
   footing as the other constant-block co-edit obligations in
   groupings (i)–(iv). A grep hit whose `(path, anchor)` pair is not
   in the constant is, by definition, a result-delta and fails the
   gate red.

Apply the same artefact to the `(quit|reload)` complement sweep one
sentence later by adding a parallel constant (or a single combined
constant keyed on `(path, anchor, triplet-or-complement)`) covering
the complement's exclusion list, so the symmetric gate inherits an
explicit artefact rather than a load-bearing silence.

Implementer-relevant edge cases: (a) the constant's relative-path
field MUST be normalised to forward slashes and a single canonical
form so a Windows contributor and a Linux contributor produce the
same exclusion-list key for the same file (parallel to the *Cross-
platform glob normalisation* rule in `audit-resolution.md`'s *Audit
scope*); (b) the second-stage-anchor field MUST be one of the four
tokens the second-stage filter already pins, so the artefact cannot
drift from the filter on a future extension of the anchor set
(extending the anchor set already obliges a same-edit update under
the *Any future runtime-side reference shape* sentence; extend that
obligation to cover this constant on the same footing); (c) the audit
MUST report a result-delta for any constant entry whose `(path,
anchor)` pair no longer corresponds to a current grep hit (so a
restatement deleted from `docs/spec.md` without a matching constant-
edit fails the gate on the next bump, mirroring the *Stale-marker
discriminator* shape `audit-resolution.md` already pins).

## Consequence

**Severity:** correctness

An implementer with no artefact pin will produce one of: (a) a
perpetually-red gate the contributor must learn to ignore, (b) a
silently-non-firing gate that admits new inbound runtime references
without flagging them — the exact failure mode step 5's prose was
escalated from `MAY` to `MUST` to prevent. Both outcomes violate the
step's "fails red on a result-delta" obligation and degrade the
inbound-reference sweep to contributor-discipline-only — also the
exact outcome the escalation was meant to eliminate.
