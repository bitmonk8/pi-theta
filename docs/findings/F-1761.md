## Problem

Step 5 of `docs/spec_topics/pi-integration-contract/version-bump-triggers.md`
mandates a build-time audit gate that re-runs the triplet / complement grep on
each `npm test` and "fails red on a result-delta against the documented
enumeration", and requires the audit to apply "the same documented-enumeration
exclusion list the contributor-side recipe applies". The audit's red/green
decision is, by construction, exclusion-list membership — yet the documented
enumeration and the exclusion list exist only as English prose embedded in this
step and step 1's carve-out. The spec never names a machine-readable artefact
that holds them, never says what key the audit matches each grep hit against,
and never says how the prose listings reduce to that artefact. Different
implementers will pick different artefact shapes with different rot profiles (a
line-keyed list silently mis-excludes on unrelated edits and goes perpetually
red; a path-only list silently authorises new inbound references in the same
file and never fires) — and the symmetric `(quit|reload)` complement sweep one
sentence later inherits whatever shape Step 5 picks but is never told it does.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** c0259ca — pi-loom spec: resolve "Version-bump procedure: four obligations have no verifiable acceptance criterion" (2026-06-07, Thomas Andersen)
**History:** Prior to `c0259ca` the build-time audit was opt-in
(`The implementation MAY additionally implement a build-time audit pass
over the same three grep targets so a result-delta against the
documented enumeration fails npm test rather than relying on
contributor-side discipline; PIC does not require the audit`) and the
exclusion list was not normatively named. `c0259ca` promoted the
audit from `MAY` to `MUST`, added the explicit `npm test` red, and
introduced the "audit MUST apply the same documented-enumeration
exclusion list the contributor-side recipe applies" sentence — but
without defining where the documented enumeration and exclusion list
live as an artefact, what they store, or how the audit matches a grep
hit against them. The gap is wholly produced by that single
`MAY → MUST` escalation: the previous MAY framing tolerated the
under-specification because PIC owned no artefact contract; once the
gate became mandatory the missing artefact contract became
load-bearing.

## Solution approach

In the *Enumeration mechanism — grep target* paragraph, immediately after the
build-time audit gate sentence, specify the machine-readable artefact the audit
consumes: its storage site, the key the audit matches each grep hit against,
and how the prose enumeration and exclusion listings reduce to it. Specify the
match key so it neither rots on edits to the cited file unrelated to inbound
references nor authorises new inbound references added to a file already on the
list. Extend the same artefact to the symmetric `(quit|reload)` complement
sweep so that gate inherits an explicit artefact rather than a load-bearing
silence.
