## Finding

Step 0 (c) opens by enumerating the factory-probable subset of the seven SDK capabilities as "items 1, 2, 3, 4, and 6 (five capabilities, nine function members — capability 1 contributes one, capability 2 contributes one, capabilities 3 and 6 each contribute two, and capability 4 contributes three)". Capability 5 (cancellation propagation) is excluded from both the item list and the per-capability member tally.

Two later sentences in the same step contradict that exclusion. The narrative immediately states that "Item 5 (cancellation propagation) is covered jointly by step (b) above ... and step (c) below (the `AgentSession.prototype.abort` member listed alongside `createAgentSession`)", and the `AgentSession.prototype.abort` bullet itself is annotated "(capability 3 / capability 5)". So one member in step (c)'s loop is normatively *also* contributing to capability 5's coverage, yet the parenthetical's cardinality summary attributes that bullet entirely to capability 3 and re-lists capability 5 only under "covered jointly".

An implementer wiring the partition `FACTORY_PROBABLE_CAPABILITIES` (the constant `V9a` exports and `V18a` reconciles against `CAPABILITY_OBLIGATIONS`' factory-probed flags) cannot tell from this paragraph whether capability 5 belongs in that set. Reading the item list and per-capability tally gives a 5-element answer `{1, 2, 3, 4, 6}`; reading the dual-coverage sentence and the bullet annotation gives a 6-element answer `{1, 2, 3, 4, 5, 6}`. The same ambiguity infects diagnostics framing: an `sdk-capability-missing` failure on the `AgentSession.prototype.abort` member is simultaneously a capability-3 failure and a capability-5 failure on one reading, but a capability-3-only failure on the other.

Subsequent edits to the paragraph (`3e6ce2e` 2026-06-05 removed `ExtensionCommandContext.waitForIdle` and bumped the count to nine; `aa334c2` and `d8d5ad0` expanded surrounding text) preserved the contradiction unchanged.

## Solution Space

**Shape:** single

### Recommendation

Place capability 5 (cancellation propagation) inside the factory-probable subset. The invariant the partition must hold is "the factory-probable subset is exactly the set of capabilities the factory probe runs at least one check for", and step (c) does probe a load-bearing capability-5 member (`AgentSession.prototype.abort`). Capability 5 is discharged jointly by step (b)'s `AbortSignal` / `AbortController` shape probe and step (c)'s `AgentSession.prototype.abort` member — both checks belong to the factory-probable set, so capability 5 is a member of the set rather than covered from outside it.

Apply these spec edits:

1. In `capability-probe.md` step (c)'s opening, replace "items 1, 2, 3, 4, and 6 (five capabilities, nine function members — capability 1 contributes one, capability 2 contributes one, capabilities 3 and 6 each contribute two, and capability 4 contributes three)" with "items 1, 2, 3, 4, 5, and 6 (six capabilities, nine function members — capability 1 contributes one, capability 2 contributes one, capability 3 contributes two, capability 4 contributes three, capability 5 contributes one (`AgentSession.prototype.abort`, shared as the second function member of capability 3), and capability 6 contributes two; the shared member is counted once)". Reword the joint-coverage sentence to "Capability 5 (cancellation propagation) is itself factory-probable: it is discharged jointly by step (b) (the WHATWG `AbortSignal` / `AbortController` shape probe) and by the `AgentSession.prototype.abort` member in step (c) below, which the listing also attributes to capability 3." Leave the `AgentSession.prototype.abort` bullet's "(capability 3 / capability 5)" annotation as-is — it is now consistent with the partition. Keep the "counted once" clause explicit so the shared member is not double-counted into the per-capability tally.
2. In `capability-inventory-items.md`, edit item 5 to state capability 5 is factory-probable via dual coverage (step (b) plus the `AgentSession.prototype.abort` bullet of step (c)), replacing its current forward-links that do not claim factory-probable status.
3. Flip `V18a`'s `CAPABILITY_OBLIGATIONS` item-5 partition flag from "verified-otherwise" to "factory-probed" and the matching `V18a-T` assertion; update `V9a`'s `FACTORY_PROBABLE_CAPABILITIES` literal to `{1, 2, 3, 4, 5, 6}`. The 9-member tally is unchanged because the shared member is counted once, so step (c)'s loop and `V9a`'s pinned-constants member count need no adjustment.

Because capability 3 and capability 5 *share* the `AgentSession.prototype.abort` function member, a naïve `sum(per_capability_counts) === FUNCTION_MEMBER_COUNT` assertion becomes `6+1 ≠ 9`. Encode the partition as `{capability → members[]}` and assert `union(members).size === 9` (preferred — no special case), or keep a numeric tally and add an explicit `SHARED_MEMBERS = [{ member: "AgentSession.prototype.abort", capabilities: [3, 5] }]` carve-out that the count assertion subtracts.

## Consequence

**Severity:** correctness

Two reasonable implementers will partition `FACTORY_PROBABLE_CAPABILITIES` differently — one writes `{1, 2, 3, 4, 6}` from the item list and member tally, the other writes `{1, 2, 3, 4, 5, 6}` from the dual-coverage sentence and the bullet annotation. The `V18a` partition assertion that consumes `FACTORY_PROBABLE_CAPABILITIES` will accept whichever choice was made (it is a self-consistency check, not an external pin), so the divergence silently propagates from the spec into `CAPABILITY_OBLIGATIONS`' factory-probed/verified-otherwise flags and from there into the `loom/load/host-incompatible` diagnostic interpretation when an `AgentSession.prototype.abort`-missing host fails the probe.
