## Finding

The step-2 preamble of `version-bump-step2.md` fixes a single per-item recording shape for the editorial-review checklist — `pass` / `fail` / `N/A` with a one-line rationale (for items (a)–(e)), with an additional `not-performed` token for SHOULD-level items (f)–(ak) — and declares omission of the row non-conformant. Item (a) then layers a *second*, distinct obligation onto its row: it carries an "open question that MUST be resolved before SM-4/5/6 are implemented," and "the contributor MUST record an authoritative determination of whether loom's package-loaded `extensions/index.ts` is governed by [Pi's] documented teardown-and-rebind lifecycle … or whether the package-loaded path provably differs from it for the session-only reasons." That determination is binary (`governed-by` vs `provably-differs`), one-shot rather than recurring, and dispositive — it gates `V9l` unblocking and, on the `governed-by` arm, the rework of `SM-4`/`SM-5`/`SM-6`/`SM-3b`.

The spec never says how the determination is recorded. The pass / fail / N/A slot is the only recording surface defined for item (a), but it is shaped for the recurring per-bump audit outcome, and the spec gives no mapping from `{governed, differs}` onto `{pass, fail, N/A}`. An implementer is left to guess one of three incompatible disciplines: (i) collapse the determination into the audit outcome (e.g. `pass — package-loaded path provably differs …` / `fail — governed by teardown-and-rebind …`), losing the determination/audit distinction and conflating a one-shot resolution with a per-bump re-audit on subsequent bumps; (ii) record the determination only in the rationale slot while picking one of pass/fail/N/A arbitrarily, so the outcome token carries no truth-value; or (iii) emit a second row of unspecified shape alongside the audit row, which the preamble's "row present/omitted" conformance check has no schema for.

Because `V9l` is explicitly blocked until the version-bump editorial-review checklist item (a) "authoritatively resolves" the contradiction (`plan.md` §Blocked obligations cites `version-bump-step2.md#bump-checklist-instance-survival` as the deciding-resolution anchor), the unblocking gate has no recordable shape to test against, and the `V18c` release-gate auditor — checking only that "the bump commit message records a per-item outcome for every item `(a)`–`(ak)` … in the shape `version-bump-step2.md` mandates" — cannot detect whether the authoritative determination was recorded at all.

## Issue introduction

**Verdict:** defect-introducing commit identified
**Introducing commits:** `7447d1e` (2026-06-10) — *pi-loom plan: resolve "Session-only degraded-state presupposition (a) contradicts Pi's documented lifecycle"*
**History:** Commit `7447d1e` extended item (a) with the "Open at the loom 1.0 pin, not merely a future-minor re-audit" block and the `MUST record an authoritative determination …` obligation, in the same edit that anchored `bump-checklist-instance-survival` and added `V9l`'s blocked-obligation linkage to it. The step-2 preamble's `pass / fail / N/A with one-line rationale` recording schema predated this commit and was not extended in the same edit to carry a binary determination distinct from a per-bump audit outcome. The defect is therefore a single-commit introduction: a new obligation landed on item (a) without a corresponding extension of the recording schema that the preamble fixes. No later commit on the file rewords either side of the mismatch.

## Solution Space

**Shape:** single
**State:** shaped

### Recommendation

Extend the item (a) recording shape (only) with a dedicated one-shot `determination: governed | differs | unresolved` field that lives alongside — not inside — the per-bump `pass / fail / N/A` outcome. State in the preamble that item (a)'s row carries two fields rather than one, define `unresolved` as the value for any bump landed before the open question is resolved, and pin that `V9l` is unblocked iff the most recent bump commit records `determination: governed` or `determination: differs` (with `governed` additionally triggering the spec rework path the host-prerequisites clause already names). The determination is structurally a different artefact from the per-bump audit outcome — one-shot vs recurring, binary vs ternary, and gating a spec rework path the recording slot does not otherwise carry — so it must not be conflated into the single audit token, which would force every later reader (V18c's release-gate auditor, V9l's unblocking gate, subsequent bumps) to re-parse the rationale to recover what should be a typed value.

Spec edits:
- Step-2 preamble: add an item-(a)-specific clause to the recording-schema sentence stating that item (a)'s row carries an additional `determination` field with the three-value vocabulary above, and that the field is mandatory.
- Item (a) body: replace "MUST record an authoritative determination of whether …" with "MUST record the `determination` field defined for item (a) in the step-2 preamble; the value MUST be one of `governed` / `differs` / `unresolved`."
- "Outcome recording conventions" block (or its replacement, if the Recording-conventions placement finding is co-resolved): cross-reference the new field as the second item-specific carve-out alongside item (e)'s two-sub-outcome shape.
- `plan_topics/V18c-version-bump-checklist.md` **Ships when**: extend the release-gate prose to check the item (a) `determination` field is recorded.

The asymmetric row shape this introduces is bounded: item (a) is the only checklist item carrying a one-shot open-question resolution. Implementer edge cases to watch: (i) state the value of `determination` for bumps landed before the question is resolved (the recommendation pins `unresolved`, but the V18c gate must accept that value rather than reading the row as malformed); (ii) once `governed` or `differs` is recorded, subsequent bumps' rows still carry the field — pin that later bumps SHOULD re-record the prior value verbatim, and that a re-recording change (e.g. `differs` → `governed`) is itself the trigger for a spec re-evaluation; (iii) the V9l unblocking gate keys on the *latest* bump commit's `determination` value, not on any historical bump's, so a later flip to `unresolved` re-blocks V9l.

## Consequence

**Severity:** correctness

Two reasonable implementers will record item (a) differently on the same bump — one mapping `governed` → `fail` and another recording `pass` with a `differs` rationale — and the spec's release-gate check ("recorded in the shape `version-bump-step2.md` mandates") cannot tell either apart from a conformant audit-only record. Because `V9l` is gated on "authoritatively resolves the … open contradiction," the absence of a defined determination-recording shape leaves the unblocking criterion permanently ambiguous: nobody can falsify whether the recorded outcome carries the determination at all. On the recurring side, conflating the one-shot determination with each subsequent bump's per-bump audit risks a contributor re-deriving it from scratch on every bump rather than treating it as settled, or — worse — silently flipping it.
