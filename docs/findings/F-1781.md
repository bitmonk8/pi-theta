## Finding

The "Transient toasts (auxiliary)" paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` enumerates three failure modes — the chokidar watcher itself throwing, an unrecoverable settings I/O exception that no `loom/load/settings-*` code covers, and an internal extension invariant violation — and pins the delivery as `ctx.ui.notify(message, "error")`. It pins post-failure subsystem state, `LoomRegistry` drain-state tag behaviour, and inner-throw swallowing. It does **not** pin the `message` string for any of the three modes.

This is the only operator-facing diagnostic surface in loom 1.0 whose author-/operator-visible string is unspecified. Every other diagnostic site is governed by **DIAG-4** (the *Message* column is normative, "renderers MUST emit it character-for-character"), and even the *non*-rendered surfaces — the construction-site `"<unreadable>"` sentinel, the underlying-error coercion rule in `placeholder-rendering-b.md`, the per-row `details.*` token vocabularies — are explicitly enumerated. The transient-toast paragraph is the lone hole. Two conforming implementations may therefore display arbitrarily different `message` text for the same chokidar throw, the same settings-read `EIO`, or the same invariant violation, and the spec gives tests no string to assert against.

The hole is not papered over by routing-channel analogy: the persistent channel carries registry-coded diagnostics whose *Message* templates DIAG-4 governs; the transient-toast modes by construction carry **no** `loom/...` code (the paragraph explicitly forbids routing coded diagnostics through this surface) and so fall outside DIAG-4's reach. Without an explicit pin in this paragraph, there is no contract.

Three subsequent commits enriched the paragraph without addressing the omission: `32d2b76` (2026-05-?, "resolve 'No behaviour defined when ctx.ui.notify itself throws'") added the inner-throw swallow rule; `c2e278f` ("fix F-1398 — pin per-arm post-failure disposition for transient toasts") added the per-arm subsystem-state and drain-state-tag rules; `0b195eb` ("fix F-1299+F-1308 — coin PIC-55 watcher post-error posture + loom/runtime/watcher-terminated") cross-linked the watcher-terminated persistent diagnostic. The message-content rule was never pinned by any of these passes — the hole has been continuously present since the paragraph's introduction.

## Solution Space

**Shape:** single

### Recommendation

Pin a normative `message` template per mode in the *Transient toasts (auxiliary)* paragraph, on the same contract footing DIAG-4 establishes for registry-coded diagnostics — i.e. tests are entitled to assert on the rendered string, and wording changes are spec-versioned.

Add a per-mode enumeration with three rows immediately after the existing per-arm subsystem-state enumeration. For each row, the `message` argument passed to `ctx.ui.notify` is the byte-identical concatenation of a fixed leading literal and (where applicable) a single host-derived tail rendered by the *Underlying-error coercion* rule of `placeholder-rendering-b.md` §6 (first-line truncation; `<no message>` substitution on the empty case; `<unreadable>` on a coercion throw):

- **(1) chokidar watcher throw** — `` `loom watcher error: <error>` `` where `<error>` is the caught throw coerced per the §6 rule. (This is the informational chokidar-error case the paragraph already reserves for the transient toast, distinct from the terminal stopped-delivery case that emits the persistent `loom/runtime/watcher-terminated` note.)
- **(2) unrecoverable settings I/O exception not covered by `loom/load/settings-*`** — `` `loom settings read failed: <error>` `` where `<error>` is the caught throw coerced per the §6 rule. The `loom/load/settings-unreadable` and `loom/load/settings-invalid-json` paths remain on the persistent channel; this toast covers only the residual `EIO`/permission-class throws those codes do not enumerate.
- **(3) internal extension invariant violation** — `` `loom internal invariant violated: <label>` `` where `<label>` is the invariant's short identifier (the same kebab-case label form `loom/typecheck/<name>` brands use under the [`loom/typecheck/*` namespace](https://example/docs/spec_topics/diagnostics/diagnostic-shape.md#loom-typecheck-namespace) — lowercase ASCII letters, digits, and `-`, no `<...>` placeholders, no interpolated host-derived tail). Label-naming guidance is implementer-side; the contract is that two implementations name the same invariant identically when they share an invariant.

State explicitly that the *Underlying-error coercion* of `placeholder-rendering-b.md` §6 applies to the `<error>` tail on rows (1) and (2), so a non-Error throw produces `<no message>` rather than the literal `undefined` and a coercion throw produces `<unreadable>` rather than an escaping `TypeError` that would defeat the paragraph's own outer-throw-swallow rule. Note that the byte-identicality of the leading literal on each row is testable on the same partial-match footing the §8 host-derived-freeform-tail rule already establishes; tests anchor on the byte-identical prefix and treat the `<error>` / `<label>` interpolation as the implementation-defined tail.

Implementer edge cases: (a) the host-derived tail on rows (1) and (2) may be empty (renders `<no message>`); (b) the §6 coercion can itself throw — the existing inner-throw swallow rule still catches it and a `<unreadable>` substitution is what the `message` carries before the swallow fires; (c) row (3) intentionally carries no host-derived tail, so the `<label>` is the only variable component and is implementer-named rather than runtime-derived.

## Consequence

**Severity:** correctness

Two reasonable implementers will produce different operator-visible text for the same chokidar throw, the same `EIO` on a settings read, and the same internal-invariant trip, with no spec-side basis for treating either as wrong. Tests for the three toast sites have no `message` to assert (DIAG-4's normative-`Message` discipline does not apply here, by construction), so the surface is non-conformance-testable in loom 1.0 unless the spec pins it.
