## Finding

`H1a`'s **Ships when** field states: "the green `npm install` is the witness that the in-tree `eslint-plugin-loom-local` package at `./tools/eslint-plugin-loom-local` resolves **and loads** (the architectural devDependencies-key test above checks only the manifest key, not loadability)." The parenthetical explicitly positions the green `npm install` as loom's loadability witness for the plugin.

`npm install` does not exercise loadability of an ESLint plugin. For a `file:` specifier pointing at a local directory, install resolves the target's `package.json` and links the directory into `node_modules`; it never imports or executes the plugin's entry module. A broken entry module (syntax error, bad export shape) installs green and is only discovered when ESLint actually loads the plugin to run a rule. ESLint is wired into `npm test` by `H2a` ("ÔÇªwired into `npm test`"), not `H1a` ÔÇö and `H1a`'s own `npm test` runs against an empty `src/**` with no ESLint invocation in its Tests bullets. So nothing at `H1a` ever loads the plugin module; only install-time resolution/linkage is witnessed.

The "and loads" half of the Ships-when claim is therefore vacuous at `H1a`: it asserts a loadability witness the named `npm install` gate cannot provide. Loadability is in fact first witnessed at `H2a`, where the `no-broad-catch` fixture tests force ESLint to load the plugin.

## Consequence

**Severity:** advisory

The Ships-when reads as if `H1a` gates that the plugin module loads, when its named `npm install` witness only covers resolution/linkage. A reader could believe a broken plugin entry module is caught at `H1a`; in practice it surfaces no earlier than `H2a`. No leaf is blocked and the implementation does not diverge ÔÇö the wording overstates what the gate observes.

## Solution Space

**Shape:** multiple

### Option A ÔÇö Soften the witness to what `npm install` actually proves

**Approach:** Restate the Ships-when clause so the green `npm install` witnesses only that the `file:` plugin specifier *resolves* (the package is found and linked), and note that the plugin module's loadability is first exercised at `H2a` when ESLint runs against its fixtures.

**Plan edits:** In `docs/plan_topics/H1a-scaffold-and-toolchain.md` **Ships when**, replace "resolves and loads" with wording that asserts only resolution/linkage ÔÇö e.g. "ÔÇªis the witness that the in-tree `eslint-plugin-loom-local` package at `./tools/eslint-plugin-loom-local` resolves and links under a fresh `npm install`; the plugin module's loadability is first exercised when ESLint runs at `H2a`." Keep the existing parenthetical that the devDependencies-key test checks only the manifest key.

**Spec edits:** None.

**Pros:** Smallest, lowest-risk change; no new test surface at `H1a`; aligns the claim with the actual `H1a ÔåÆ H2a` sequencing where ESLint is wired in.

**Cons:** Plugin loadability remains first witnessed at `H2a` rather than at its owning leaf `H1a`.

**Risks:** None material.

### Option B ÔÇö Add a minimal ESLint invocation at `H1a` so loading is genuinely witnessed

**Approach:** Add an `H1a` test that runs ESLint with the `eslint-plugin-loom-local` plugin configured (against a trivial/empty source or a no-op fixture) so the placeholder plugin's entry module is actually imported under `H1a`'s `npm test`, making the "loads" claim true at the owning leaf.

**Plan edits:** In `docs/plan_topics/H1a-scaffold-and-toolchain.md` add a Tests bullet asserting a minimal ESLint run loads the placeholder `eslint-plugin-loom-local` plugin green; the Ships-when "loads" claim then holds at `H1a`.

**Spec edits:** None.

**Pros:** Plugin loadability is witnessed at the leaf that owns the plugin skeleton; a broken entry module fails `H1a` rather than `H2a`.

**Cons:** Introduces an ESLint invocation and a loadable-but-rule-empty configuration at `H1a` that overlaps `H2a`'s wiring; larger change for an advisory-severity wording defect.

**Risks:** The placeholder plugin exports an empty rule set, so the invocation must be configured to load the plugin without requiring a rule ÔÇö easy to mis-author into a vacuous pass.

### Recommendation

Take **Option A**: change "resolves and loads" to assert only resolution/linkage and point loadability to `H2a`. Loadability is already genuinely witnessed at `H2a` via the `no-broad-catch` fixture tests, so the cheaper fix is to make `H1a`'s claim match what its `npm install` gate observes rather than to add an overlapping ESLint invocation at `H1a`. Watch that the revised clause keeps the existing distinction from the architectural devDependencies-key test (which checks the manifest key only).
