## Finding

The typed-query reference-implementation bullet in `implementation-notes.md` describes the forced respond turn and, mid-sentence, enumerates concrete provider-native forced-tool payload shapes as bare facts: `options.toolChoice` (`{ type: "tool", name }`) "which the per-provider pi-ai adapter maps to the provider-native payload (Anthropic `tool_choice: { type: "tool", name }`; OpenAI Completions `tool_choice: { type: "function", function: { name } }`; Mistral and Bedrock map analogously)." These are externally-owned `@earendil-works/pi-ai` / provider wire shapes that loom does not define.

The authoritative treatment of this mapping lives at `conversation-drive.md` under the `complete-forced-tool-presupposition` anchor: it pins the `options.toolChoice = { type: "tool", name }` → provider-native-payload property as an unpinned host presupposition, names the four supported providers, marks the property as outside pi-ai's typed surface, and routes per-provider forced-tool re-validation to the editorial-review checklist (item (u) of the Pi version-bump procedure). The implementation-notes re-narration carries none of that: no pointer to the authoritative presupposition anchor, and no non-normative / illustrative tag marking the concrete per-provider shapes as a pi-ai illustration rather than a loom-owned contract. The general pointer at the head of the same bullet ("the provider compatibility surface … is normative and lives in [Conversation drive]") sits several sentences upstream and does not anchor the inline payload enumeration.

The result is a second, un-anchored copy of external provider-payload facts. It is correct today, but it duplicates externally-owned shapes outside the one site that pins them and routes their re-validation, with no link tying the two together.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 82be125 — spec: adopt P1 typed-query two-phase loop (Option A) (2026-05-04); e63ba6c — pi-loom spec: resolve "Per-provider complete() forced-tool re-validation gate" (2026-06-05)
**History:** The concrete per-provider `tool_choice` payload narration (Anthropic / OpenAI Completions shapes, "Mistral and Bedrock map analogously") entered with the two-phase typed-query loop at 82be125, where it was the only description of the forced-tool mapping. The authoritative `complete-forced-tool-presupposition` section in `conversation-drive.md` — which pins the mapping property and routes its per-provider re-validation to the editorial-review checklist — was added later at e63ba6c, but the implementation-notes copy was never back-linked to it. The defect is the interaction: the inline re-narration became an un-pointed duplicate of external-entity facts once the canonical pinning site was established elsewhere.

## Solution Space

**Shape:** single

### Recommendation

At the implementation-notes re-narration site, anchor the concrete per-provider payloads to the authoritative presupposition and mark them as illustrative. Concretely: keep the loom-side call shape (`complete()`'s `options.toolChoice = { type: "tool", name }`) as loom's own pi-ai usage, but for the per-provider native payloads (Anthropic `tool_choice: { type: "tool", name }`; OpenAI Completions `tool_choice: { type: "function", function: { name } }`; Mistral / Bedrock analogues) either (a) replace the inline enumeration with a cross-reference to `conversation-drive.md#complete-forced-tool-presupposition`, or (b) retain the enumeration but tag it as a non-normative pi-ai illustration and add the same cross-reference, so the authoritative mapping plus its version-bump editorial-review routing (item (u)) remains the single source of truth. Edge case for the implementer: do not duplicate the supported-provider set or the bump-audit routing into implementation-notes — the pointer must carry those, otherwise the same drift hazard reappears in a different form.

## Consequence

**Severity:** advisory

No implementer diverges today — the enumerated per-provider shapes are accurate. The defect is a maintenance/guidance gap: the duplicated external-entity facts can drift from the authoritative presupposition, and a maintainer updating the canonical mapping (or its bump-audit routing under editorial-review item (u)) gets no cue that this second copy exists, so the implementation-notes copy is not on the version-bump radar.
