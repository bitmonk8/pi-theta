## Problem

PIC-25 on `diagnostic-emission-isolation.md` pins three serialiser-throw fallback forms: the bare `console.error(code)` form for the flat-`details` teardown-handler codes, and the two-token `` `${code} ${detailsEventReason}` `` form (interpolating the pre-hoisted local) for the two nested-shape codes `loom/host/session-shutdown-runtime-degraded` and `loom/runtime/cancelled-by-session-shutdown`. The page's `## Acceptance criteria` block enumerates fixtures only for the PIC-26 construction-site path, leaving the PIC-25 serialiser-throw path — structurally distinct, occurring inside the serialisation-and-emission wrap with `detailsEventReason` already populated — with no fixture mandate. An implementation that omits the serialisation-and-emission `try`/`catch`, lets a serialiser throw propagate out of the handler, or collapses the nested-shape codes to a bare `code` emit therefore passes every required fixture. PIC-25's behavioural rules (no re-throw, handler continues to the next sub-step, byte-exact wire shape, the per-row dedup discriminator preserved via the hoisted local) are all testably unenforceable.

## Solution approach

Extend the `## Acceptance criteria` block in `diagnostic-emission-isolation.md` with fixtures for the PIC-25 serialiser-throw path, covering the bare-`code` fallback for the flat-`details` codes and the two-token `` `${code} ${detailsEventReason}` `` fallback (read from the pre-hoisted local) for the two nested-shape codes `loom/host/session-shutdown-runtime-degraded` and `loom/runtime/cancelled-by-session-shutdown`. Broaden the block's opening sentence so it scopes both PIC-25 and PIC-26 rather than only the construction-site obligation.
