## Finding

`docs/spec_topics/pi-integration-contract/diagnostic-emission-isolation.md` pins three distinct serialiser-throw fallback forms under PIC-25:

1. the bare `console.error(code)` form for the three flat-`details` teardown-handler codes (`loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`) plus `loom/runtime/reload-teardown-timeout`;
2. the two-token form `` `${code} ${detailsEventReason}` `` for `loom/host/session-shutdown-runtime-degraded` (the nested-shape teardown-handler code), interpolating the pre-hoisted `detailsEventReason` local;
3. the same two-token form, also via the hoisted local, for the per-invocation `loom/runtime/cancelled-by-session-shutdown` note, plus the analogous handling on the terminal `console.error` of the `sendSystemNote` fallback chain.

The Acceptance criteria block on that page only enumerates fixtures for the PIC-26 *construction-site* path — the two-token `` `${code} <unreadable>` `` form for the runtime-degraded row, the three-token `` `${code} ${entry.loom} <unreadable>` `` form for the per-invocation note, and the self-wrapped inner throw. The PIC-25 serialiser-throw path (which is what each form was coined for and is structurally distinct: it occurs *inside* the serialisation-and-emission wrap, with the `detailsEventReason` local already populated, rather than from a construction-site throw before payload assembly) has no fixture mandate at all.

An implementation that omits the serialisation-and-emission `try`/`catch`, lets a `JSON.stringify` throw propagate out of the handler, or emits the bare `code` instead of the two-token form on the nested-shape codes therefore passes every required PIC-26 fixture. The behavioural rules under PIC-25 (no re-throw; handler continues to the next sub-step; the bare-`code` / two-token wire shape is byte-exact; the per-row dedup discriminator survives via the hoisted local) all lack a normative test bound, leaving every PIC-25 obligation testably unenforceable on the implementation surface.

## Solution Space

**Shape:** single

### Recommendation

Extend the existing `## Acceptance criteria` block in `docs/spec_topics/pi-integration-contract/diagnostic-emission-isolation.md` with three PIC-25 fixtures (lettered (d)–(f), continuing the existing (a)/(b)/(c) PIC-26 series), and broaden the opening sentence so it scopes the block to "PIC-25 and PIC-26" rather than only "the construction-site catch-arm obligation ([PIC-26](#pic-26), building on [PIC-25](#pic-25))". The new fixtures are:

- **(d) Serialiser throw — flat-`details` code (bare-`code` form).** Force the leaf-owned serialisation primitive to throw (e.g. via a circular `cause` or a `BigInt` field in the payload) for one of the four bare-`code` rows — pick `loom/host/session-shutdown-teardown-step-failed` as the representative case the existing `V9g-T` Tests bullet already touches. Assert: the catch arm emits exactly `console.error(code)` (the bare diagnostic-code string, no second argument, no token suffix); the catch arm does not re-throw; the handler continues to the next sub-step; the serialiser-throw emission counts as the single emission against the per-event bound (PIC-28). A parallel fixture for `loom/runtime/reload-teardown-timeout` is optional but recommended as it sits at a different sub-step boundary.
- **(e) Serialiser throw — `loom/host/session-shutdown-runtime-degraded` (two-token form, via hoisted local).** Force the serialiser to throw with a successful `details.event` construction and a populated `detailsEventReason` local. Assert: the catch arm emits the byte-exact two-token form `` console.error(`${code} ${detailsEventReason}`) `` carrying the captured `event.reason` string (cover the closed `{"new","resume","fork"}` set and at least one `"<unreadable>"` sentinel case from the *Hoist obligation*'s sub-step-2 stamp-throw path so the dedup-preservation rationale is exercised); the catch arm does not re-throw; the handler continues to the next sub-step. The fixture MUST read the interpolated reason from the pre-hoisted local — verifying byte-equality with `detailsEventReason` rather than re-reading `entry.shutdownReason` — so an implementation that re-reads the partially-constructed payload fails red.
- **(f) Serialiser throw — `loom/runtime/cancelled-by-session-shutdown` (two-token form, per-invocation note).** Same shape as (e) but on the per-invocation `finally`'s emission. Cover at least one captured-value reason from the closed five-arm union and the `"<unreadable>"` sub-step-2 stamp-throw case; assert the byte-exact two-token form via the hoisted local and the per-invocation-bound count semantics.

Each fixture should additionally assert (or share a `beforeEach` that asserts) the PIC-27 inner-throw swallow — that a throw out of the bare-`code` / two-token `console.error` itself is swallowed and the handler still continues — so the PIC-25 catch arm's full obligation set is covered. Update `docs/plan_topics/V9g-T-session-shutdown.md`'s `DIAG-1` Tests bullet to reference the broader fallback-form coverage (it currently names only the bare-`code` form for `loom/host/session-shutdown-teardown-step-failed`); the existing `cka-29` row in `docs/plan_topics/coverage-matrix.md` already enumerates "the bare-`code` / two-token / three-token serialiser-throw fallback forms" under `V9g`, so no coverage-matrix edit is required.

## Consequence

**Severity:** correctness

Two compliant-by-the-letter implementations can diverge silently on the most operationally-load-bearing fallback in the teardown stack: one may re-throw out of `JSON.stringify`, unwinding the handler and skipping subsequent sub-steps; another may collapse the nested-shape codes to a bare `code` emit, destroying the `details.event.reason` and `details.event.loom` dedup buckets operator tooling keys on. Both pass `V9g-T`. The PIC-28 invocation-site count contract and the *Hoist obligation*'s dedup-preservation rationale rest on behaviour that no fixture asserts.
