## Problem

In the Numbered REQ-IDs table of `coverage-matrix.md`, the `ERR-8 … ERR-12` row maps the five-ID range to the cell `` `V4c`, `V9i` (live subagent-surface ERR-8 co-witness) ``. The annotation names only ERR-8, yet sits inside a five-ID range and the *Closing-leaf column authoring* rule makes the H5d tokeniser read `V9i` as a contributor to the whole cell. A reader cannot tell whether `V9i` closes ERR-9 … ERR-12 or only co-witnesses ERR-8, and the row diverges from the per-leaf facet scoping the matrix uses for analogous multi-leaf cells (e.g. the DISC-4 and PIC-17/PIC-18 rows).

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `7bd65a7` (2026-06-18) — "pi-loom plan: resolve \"V4c/V4f defer ERR-8 to ERR-13 live-surface confirmation to downstream owners that never re-assert it\""
**History:** Before `7bd65a7` the row read `` | ERR-8 … ERR-12 | `V4c` | `` — a single, unambiguous closer with no co-witness and no range/facet ambiguity. `git log -S"live subagent-surface ERR-8 co-witness" -- docs/plan_topics/coverage-matrix.md` returns that single commit; its diff adds `V9i` to the cell with the prose annotation "(live subagent-surface ERR-8 co-witness)". The defect was introduced by that prior review fix: adding the co-witness to a contiguous-range row without scoping the non-`V9i` facets created the whole-range-vs-ERR-8-only ambiguity from inception of the annotation.

## Solution approach

Rewrite the `V9i` annotation on the `ERR-8 … ERR-12` row in `coverage-matrix.md` so it scopes each leaf explicitly: `V9i` co-witnesses ERR-8 only (the live subagent-surface re-assertion exercised as the ERR-12 subagent-loom restatement of ERR-8), and `V4c` is the sole closer of ERR-9 through ERR-11. Bring the row into line with the per-leaf facet scoping the matrix already uses for its other multi-leaf cells.

## Solution constraints

- Honour the *Closing-leaf column authoring* backtick rule: `` `V4c` `` and `` `V9i` `` stay the only backtick-delimited spans; every REQ-ID referenced inside the annotation prose stays bare so the H5d tokeniser captures no spurious token.
- Keep the `ERR-8 … ERR-12` row contiguous; do not split it into per-ID rows (the H5a per-prefix-numbering-hole expectation reads the `…` range).
- Out of scope: the `V4c` and `V9i` leaves require no edit, and the spec is read-only for this fix.
