## Finding

The Numbered REQ-IDs table in `coverage-matrix.md` maps the contiguous range `ERR-8 … ERR-12` to the closing-leaf cell `` `V4c`, `V9i` (live subagent-surface ERR-8 co-witness) ``. The annotation names only ERR-8, but it sits inside a five-ID range row, and the *Closing-leaf column authoring* rule makes the H5d tokeniser read `V9i` as a backtick-delimited contributor to the whole cell — i.e. to all of ERR-8 … ERR-12. A reader of the matrix in isolation cannot tell whether `V9i` is a closing leaf across the whole range or only co-witnesses ERR-8, and the annotation says nothing about ERR-9, ERR-10, ERR-11, or ERR-12.

The underlying leaves are unambiguous and narrower than the cell suggests. `V4c` is the sole closer of ERR-9 (no compensating turn), ERR-10 (symmetry across cancellation and `?`-propagation), and ERR-11 (the non-mutation window). `V9i`'s only ERR contribution is a delegated live-carrier witness for ERR-8 — a mid-stream cancellation driven inside the real subagent `AgentSession` does not mutate its committed turns — which is exactly the live-surface re-assertion of the ERR-12 subagent-loom restatement of ERR-8 that `V4c` defers from the `H4a` double. `V9i` does not close ERR-9 … ERR-11. The matrix annotation under-scopes that: by naming "ERR-8" but offering no statement about the rest of the range, it leaves `V9i`'s role for ERR-9 … ERR-12 unspecified on the page.

The matrix already uses per-leaf facet scoping for analogous multi-leaf cells (e.g. DISC-4: `` `V10a` (collision-detection closure), `V9m` (superseded-entry-dispatch closure) ``; the PIC-17/PIC-18 facet split). The ERR-8 … ERR-12 row diverges from that convention by leaving the non-`V9i` facets unattributed.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/coverage-matrix.md`, Numbered REQ-IDs table, rewrite the `V9i` annotation on the `ERR-8 … ERR-12` row so it states `V9i`'s scope explicitly: `V9i` is the live subagent-surface co-witness for ERR-8 only — exercised as the ERR-12 subagent-loom restatement of ERR-8 — and `V4c` is the sole closer of ERR-9 through ERR-11. A literal cell that carries this is:

```
| ERR-8 … ERR-12 | `V4c`, `V9i` (co-witness — live subagent-surface re-assertion of ERR-8, exercised as the ERR-12 subagent-loom restatement; ERR-9 through ERR-11 close solely in V4c) |
```

Honour the *Closing-leaf column authoring* backtick rule: `` `V4c` `` and `` `V9i` `` remain the only backtick-delimited spans in the cell, and every REQ-ID reference inside the annotation prose (ERR-8, ERR-9, ERR-11, ERR-12) stays bare so the tokeniser captures no spurious token. The range row stays contiguous; do not split it into per-ID rows — both leaves already tokenise correctly at the cell level and the H5a per-prefix-numbering-hole expectation reads the `…` range, so splitting would fragment a deliberate contiguous range without changing what the gate verifies.

`coverage-matrix.md` owns this annotation; the spec is not involved, and the `V4c` / `V9i` leaves already carry the correct scoped test prose, so no leaf edit is required.

## Consequence

**Severity:** advisory

An auditor reading the coverage matrix in isolation cannot tell whether `V9i` closes ERR-9 … ERR-12 or only co-witnesses ERR-8's subagent surface; the H5d tokeniser treats `V9i` as a whole-cell contributor while its tests cover only ERR-8 (and the ERR-12 subagent restatement of ERR-8). This mis-scopes the manual split-row joint-completeness sweep (residue-inspection item 7), which could wrongly read ERR-9 … ERR-11 as having a second closer. The leaves themselves are unambiguous, so no implementer produces a divergent leaf; the gap is in the traceability prose.
