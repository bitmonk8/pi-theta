## Finding

BNDR-12's normative MUST is framed around a "second call" that never happens in loom 1.0 (exactly one binder call per invocation), making its load-bearing clause unfalsifiable; "re-entrant per loom turn" is itself ambiguous between concurrency and statelessness.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `0fa81fa` (jargon), `e64c7a3` (BNDR-12 testability)
**History:** The corpus is git-tracked. The "tier-2 model" descriptor entered with the binder at its inception (`83802db`, 2026-05-04) as a *positive* term — the third-tier "built-in default (cheap tier-2 model)" fallback. `0fa81fa` (2026-05-05, "pin binder model resolution") removed that built-in default ("**no further fallback**") but retained a now-antecedent-less *negated* reference ("no 'tier-2' default") and coined "cost premise" in the same edit, leaving both terms undefined. The "re-entrant per loom turn" wording first appeared as a *non-normative* escalation-seam note; `e64c7a3` (2026-06-28, "fix F-2094 — promote binder re-entrancy constraint to BNDR-12 MUST") promoted that note to the normative BNDR-12 MUST, which is the edit that introduced the untestable within-invocation second-call clause.

## Solution Space

**Shape:** single

### Recommendation

Restate the MUST as the observable, single-call property: on each binder call the binder's input record (parameter table, raw slash text, optional session-context block) and the resolved binder-model handle MUST be constructed afresh from current inputs with no state cached from any prior binder call or from loom load — wording that is testable on the one 1.0 call (e.g. a binder invoked after a prior invocation under a changed `bind_context`/session observes the new inputs, with no carried snapshot). Replace "re-entrant per loom turn" with this concrete statelessness phrasing. Move the "second call serves a non-stale snapshot" property and the automatic-context-escalation rationale entirely into the existing adjacent non-normative note as the forward-compat justification, so no normative MUST depends on an unreachable second call.

## Consequence

**Severity:** advisory

Undefined jargon makes the binder-model resolution paragraph harder to read than necessary but does not change behaviour. The untestable BNDR-12 clause is a guidance gap: an implementer still builds a working system (fresh construction per call), but a conformance author cannot write a passing/failing test for the stated MUST, so the requirement is effectively unverifiable in 1.0.
