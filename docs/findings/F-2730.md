## Finding

The *Transitive-completeness plan-maintenance* rule in `conventions.md` §REQ-ID discipline imposes a strict obligation: *every* coverage-producing leaf — including each secondary facet-closer of a multi-leaf coverage-matrix row — MUST be added to `H5b`'s `Deps.`, so the warn-only canary and the `H6a` hard-fail release gate stay sequenced after the complete coverage-producing set.

The same paragraph then asserts "This obligation is mechanically backed:" and cites `H5d`'s standing transitive-completeness gate arm. But `H5d` is, by its own description in the very same paragraph, a **per-cell at-least-one** test: it reddens only for a closing-leaf cell *none* of whose listed leaves reaches `H5b.Deps`, and stays green for a genuine multi-facet split row (e.g. `ERR-17 | V4d, V13d`, `DISC-4 | V10a, V9m`) as soon as *any one* listed leaf is present. The arm therefore cannot catch a secondary facet-closer omitted while a sibling facet-closer for the same cell remains listed — exactly the kind of omission the strict every-facet obligation exists to forbid.

The lead-in clause "This obligation is mechanically backed," read in isolation, claims more enforcement than `H5d` delivers. The paragraph does disclose the per-cell limit and route the every-facet-closer remainder to the `H5b` warn-only canary and release-time residue-inspection item 7 in the sentences that follow, so a reader of the full paragraph gets the accurate picture; the overclaim is confined to the bare lead-in framing.

## Consequence

**Severity:** advisory

A contributor who reads or quotes the bare "This obligation is mechanically backed" clause may trust the `H5d` gate to catch every coverage-producing-leaf omission, when it only enforces per-cell at-least-one; a secondary facet-closer omitted while a sibling facet-closer for the same multi-leaf row stays listed passes `H5d` silently. The omission is caught later only by the warn-only `H5b` canary and residue-inspection item 7. Because the same paragraph already discloses the limit, no leaf is blocked and no implementer is forced off-script — the degradation is a misleading enforcement claim, not a coverage hole.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, in the *REQ-ID discipline* → *Transitive-completeness plan-maintenance* paragraph, soften the lead-in clause that currently reads "This obligation is mechanically backed: [`H5d`](./H5d-transitive-completeness-gate.md)'s standing transitive-completeness gate arm reconciles…" so it states that the backing is partial and names where the remainder is caught. Suggested replacement content for the lead-in: "This obligation is **partially** mechanically backed — `H5d`'s standing transitive-completeness gate arm enforces only the per-cell at-least-one floor (it reconciles…); the every-facet-closer remainder is surfaced by the `H5b` warn-only canary and the loom 1.0 release-time residue inspection (checklist item 7, split-row joint-completeness)."

Keep the existing per-cell at-least-one explanation, the multi-leaf-cell example, and the existing every-facet-closer / item-7 routing sentence that already follow — the edit only re-frames the lead-in so it no longer asserts full mechanical backing in isolation. Do not change `H5d`'s behaviour, the `H5b.Deps` obligation, or the coverage-matrix examples.
