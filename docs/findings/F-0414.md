## Finding

`V18c`'s `Adds.` field closes with a description of the contributor's *editorial-review checklist* (items `(a)`–`(ak)`): "a manual per-bump audit of each host presupposition against the candidate Pi minor, MUST-level for `(a)`–`(e)` and SHOULD-level-audit-but-MUST-record for `(f)`–`(ak)`." Per [`conventions.md`](../../docs/plan_topics/conventions.md) §Leaf format, `Adds.` is "one sentence — what the leaf introduces," descriptive of the code/mechanism the leaf builds once. The editorial-review checklist is not something `V18c` builds: it is a recurring manual per-bump process whose items, normative levels, and pass/fail rule are authoritatively defined in the spec ([`version-bump-step2.md`](../../docs/spec_topics/pi-integration-contract/version-bump-step2.md)). `V18c`'s actual code contribution is the static build-time gates (surface-inventory, `engines.node` three-way equality, peer-dep pin, reason-snapshot, provider seed-field, strict-capability probe) — the editorial-review checklist sits at a different abstraction level.

The same MUST/SHOULD blocking distinction is already restated, more fully, in `V18c`'s `Ships when` ("a `fail` on any MUST-level item `(a)`–`(e)` blocks the bump, whereas a recorded `fail` / `not-performed` on a SHOULD-level item `(f)`–`(ak)` is a conformant non-blocking outcome, per [`version-bump-step2.md`](../../docs/spec_topics/pi-integration-contract/version-bump-step2.md)"). The `Adds.` clause is therefore a partial third copy of a rule the spec owns and `Ships when` already gates against — a stale-on-spec-change hazard: if the spec re-levels an item, three locations drift out of lockstep.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V18c-version-bump-checklist.md`, edit the `Adds.` field: strike the trailing editorial-review process restatement — the clause beginning "Separately — and *not* a build-time gate — the contributor's editorial-review checklist (items `(a)`–`(ak)`)…" through the MUST-level/SHOULD-level breakdown — and replace it with a non-restating pointer that names the checklist as out-of-scope for this leaf and cites its owner, e.g.: "The per-bump editorial-review checklist (items `(a)`–`(ak)`) is a manual process owned by [`version-bump-step2.md`](../spec_topics/pi-integration-contract/version-bump-step2.md); `V18c` adds no build-time gate for it." Do not relocate the `(a)`–`(e)` / `(f)`–`(ak)` MUST/SHOULD level text into `real-host-smoke-gate.md` — that note governs the fidelity-divergence smoke run, not the version-bump checklist, and the checklist's normative levels are already owned by the spec and gated by `V18c`'s `Ships when`. The `Spec.` field and `Ships when` are read-only for this fix; the spec is not edited.

## Consequence

**Severity:** advisory

`Adds.` is non-binding by convention, so no implementer is forced to diverge; the harm is maintenance hygiene. The MUST/SHOULD level breakdown now lives in three places (spec, `Adds.`, `Ships when`); a future spec re-leveling silently leaves the `Adds.` copy stale, and the process detail at the wrong abstraction level invites the reader to mistake the recurring manual checklist for part of `V18c`'s one-time code deliverable.
