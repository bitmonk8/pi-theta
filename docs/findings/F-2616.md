## Finding

RET-1 states that `return expr` "is type-checked against the enclosing scope's declared return type," and FN-3 states that a `fn` with an explicit return annotation "type-checks its tail and every `return` operand against that annotation." Both obligations mandate a static `⊑` check that can fail, but neither names the diagnostic code that fires on failure, and the `loom/parse/*` registry contains no code for this site.

Every comparable compatibility-check site names its own code: typed `let` RHS → `loom/parse/let-rhs-type-mismatch`, plain `fn`-argument slot → `loom/parse/fn-arg-type-mismatch`, `invoke` return → `loom/parse/invoke-return-type-mismatch`, array element → `loom/parse/array-element-type-mismatch`, `match` arm → `loom/parse/match-arm-type-mismatch`. TYPE-9 explicitly enumerates "Three of the compatibility check sites … not covered by the invoke / array / match codes" and names diagnostics for exactly the typed-`let`, `fn`-argument, and ternary sites — the annotated return-type check is absent from that list. The two registry codes adjacent to `return` cover different cases: `loom/parse/return-no-common-type` is the *annotation-less / top-level inference* case (no declared type to check against), and `loom/parse/bare-return-in-non-void` is the *no-operand* case. The case where an annotated `fn`'s tail expression or `return expr` operand has a type incompatible with the declared annotation is orphaned.

The consequence is no testable pass/fail criterion: the check is mandated but its observable surface (the emitted diagnostic code) is undefined. Under the spec's closed diagnostic registry, conformance tests key on codes, so an unnamed code leaves the rejection untestable and lets implementations invent divergent codes for the same failure.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** none single; the obligation has lacked a named code since the return-statement semantics were first written. Earliest tracked appearance: `fecb504` (2026-05-04, "Split spec.md and plan.md into per-topic / per-phase files"), where `return.md` already reads "`return expr` is type-checked against the enclosing scope's declared return type" with no diagnostic named. The semantics predate the split (present in the `53f5831`/`83802db` rewrites of spec.md).
**History:** The defect is an omission carried through every subsequent diagnostic-naming pass without being closed. `839db71` ("Diagnostic codes are declared as namespaces but never assigned") assigned the bare-`return` case its code (`loom/parse/bare-return-in-non-void`) but not the `return expr` mismatch case. `648365c` ("Three subtype check sites name no diagnostic codes") introduced TYPE-9 naming codes for the typed-`let`, `fn`-argument, and ternary sites and explicitly scoped itself to "three" sites, excluding the annotated return-type check. `08886d9` (2026-06-07) added FN-3's "type-checks its tail and every `return` operand against that annotation" prose and registered the *inference-only* `loom/parse/return-no-common-type`, but added no code for the annotated-mismatch case. `c0c6853` (2026-06-25) coined the RET-1..3 identifiers, formalising the obligation in RET-1 while still naming no code. The token `loom/parse/return-type-mismatch` has never appeared in the spec corpus (only in review/store artefacts). The durable fix addresses the standing omission, not any single commit.

## Solution Space

**Shape:** single

### Recommendation

Add one parse diagnostic, `loom/parse/return-type-mismatch`, for the case where, in a scope with an **explicitly declared** return annotation, the tail expression's type or any `return expr` operand's type is statically incompatible (`T ⋢ ReturnType`, both operands statically resolvable) with the declared annotation. Wire it into three places:

1. **Registry.** Add a row to the `loom/parse/*` table in `docs/spec_topics/diagnostics/code-registry-parse.md` (severity `E`, phase `type`, spec rule [Return Statement](../return.md) / [Function Definitions — Loom return type](../functions.md#loom-return-type)), with a message of the form `return value type mismatch: expected <expected>, got <actual>`.
2. **TYPE-9.** Extend TYPE-9's per-site enumeration to include the annotated return-type check (the current "Three … sites" becomes four), citing the new code, and update the closing registration sentence to list the new row alongside the `let`- and `fn`-site rows.
3. **RET-1 / FN-3.** Name the code inline at RET-1 and at FN-3's "type-checks its tail and every `return` operand against that annotation" sentence so the obligation points to its code.

Edge cases the implementer must watch:

- The code fires **only** when an explicit return annotation exists. The annotation-less / top-level-loom path is governed by the LUB inference rule and keeps `loom/parse/return-no-common-type`; RET-1 already draws this split. Do not let the new code displace `return-no-common-type`.
- Apply the TYPE-9 "Unresolvable operands" posture: when the operand type is past the parser's static view (e.g. depends on a Pi-tool schema not visible at parse time), skip the parse-time check and let the runtime AJV net decide; only emit on statically-resolvable operands.
- A single code covering both the tail expression and `return expr` operands is sufficient — both are "checked against that annotation" per FN-3 and share one failure surface; no separate tail-specific code is needed.
- `void`-annotated bodies discard the tail value silently (FN-4) and bare `return` is owned by `loom/parse/bare-return-in-non-void` (RET-2); the new code does not subsume either.
- The `?`-operator `Err`-arm desugars to `return Err(e)`; its operand-type precondition is owned by `loom/parse/question-on-non-result` (ERR-18), not this code.

## Consequence

**Severity:** correctness

Two conforming implementers diverge on the observable failure for an annotated `fn` whose tail or `return` operand violates the declared return type: one may emit an unregistered code, another may reuse `return-no-common-type` (semantically wrong — that is the inference case), another may defer to a runtime AJV failure. The closed registry treats codes as the conformance surface, so this site is currently untestable and the rejection behaviour is not pinned.
