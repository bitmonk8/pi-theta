## Finding

The *Infra-aborted-run carve-out* tells CI parsers to exclude infra-aborted runs identified "by the presence of one or more infrastructure-failure records", but the *Pre-emission termination carve-out* permits a non-zero-exit run that emits zero records (OOM-kill, host shutdown, mid-write panic, torn-pipe write failure). Such a run carries no infra record, so the record-presence predicate fails to classify it as infra-aborted and a CI parser then falsely flags the missing canary record as a canary-obligation violation.

**Verdict:** present-since-inception

**History:** The *Non-empty-scan canary*, its strictly-greater-than-zero floor, the *Infra-aborted-run carve-out*, and the *Pre-emission termination carve-out* all entered the corpus together in e5cd514, which introduced the negative-direction audit apparatus into `pi-integration-contract.md`. Both defects were present at that introduction: the `> 0` floor shipped without a stated precondition, and the infra-aborted-run carve-out's record-presence identification shipped in the same paragraph as the pre-emission zero-record path it fails to cover. `git log -S`/`-G` over the canary token, the carve-out token, and the floor wording confirms e5cd514 as the sole introducing commit; later commits relocated and reframed the section (f5e89f4 size-cap split, 81bf52a collapse into `spec_topics/`, 74633aa post-1.0 reframe) but none touched either gap. Both are omission/interaction defects present since the apparatus's inception, not introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

Extend the infra-aborted-run carve-out's identification rule so a zero-record run is also excluded from the canary obligation — e.g. a CI parser MUST treat a run with zero records, a non-zero `npm test` exit, and empty audit stdout as an infra-aborted (pre-emission-terminated) run on the same footing as a run carrying ≥1 `infra`-class record, and MUST NOT apply the "exactly one canary record per invocation" assertion to it. Cross-reference the *Pre-emission termination carve-out* as the path this branch covers.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on both points: one treats the canary's `> 0` floor as universal (red on any Pi-free tree), another assumes the runtime-always-has-Pi-surface precondition silently, so their canary behaviour differs on the same edge tree. And a conforming CI parser following the infra-aborted-run carve-out will raise a spurious canary-obligation violation on a host-killed zero-record run, defeating the carve-out's purpose of distinguishing infra aborts from real canary breakage.
