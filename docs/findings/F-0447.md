## Finding

The phrases *"is a CI failure"*, *"fails CI"*, and *"reddening CI"* are the operative enforcement semantic for nearly every mechanical gate the plan defines. `conventions.md` §REQ-ID discipline uses *"is a CI failure"* for an unmapped executable REQ-ID, an un-anchored normative MUST with no closing leaf, a per-prefix numbering hole, a registry code with no asserting test, and an unresolvable broad-catch / `// allow:` token; `coverage-matrix.md`'s intro states an unmapped REQ-ID *"fails CI"*; the release-gate prose speaks of gates *"reddening CI"* and divergence findings being a *"merge blocker"*. Yet no file in the plan corpus enumerates what that substrate is: which runner executes the gate, on what event it runs (push / merge candidate / release), and whether — and how — a red gate actually blocks a merge.

Every leaf grounds its own gate in `npm test` (each `Ships when` is an `npm test` invocation), so the *observability* of a gate is well-defined. What is undefined is the leap from "an `npm test` arm reds" to "CI fails and the change cannot merge". `real-host-smoke-gate.md` sharpens the gap: it explicitly distinguishes the manual smoke as *"a contributor obligation, not a CI-enforced gate"* with *"no mechanical hook"* — a contrast that presupposes a CI-enforced gate category exists elsewhere, while that category's substrate is never defined anywhere. The repository carries no CI configuration (`.github/`, workflow YAML, or equivalent), and no leaf provisions one. The result is that the verdict word the whole gate architecture leans on — and the *"blocks merge"* / *"MUST NOT be merged"* guarantees built on it — has no defined mechanism behind it.

## Solution Space

**Shape:** multiple

### Option A — Define the verdict word in `conventions.md`; treat merge-blocking as named project policy

**Approach.** Add a one-time definition near the top of §Cross-cutting rules stating that *"a CI failure"* / *"reddening CI"* / *"fails CI"* denotes the unified `npm test` gate `H1a` provisions exiting non-zero, and that the substrate which runs `npm test` on the merge path and blocks a merge on a non-zero exit is the project's required-check policy (named there as the substrate, e.g. the repo's branch-protection / required-check configuration), out of loom 1.0 leaf scope. Then replace the bare *"is a CI failure"* / *"fails CI"* usages with the defined term so each gate's verdict resolves to the one definition.

**Plan edits.** `conventions.md`: insert the definition; sweep its *"is a CI failure"* sites to the defined term. `coverage-matrix.md`: point its *"fails CI"* usage at the same defined term.

**Spec edits.** None.

**Pros.** Minimal surface; no new leaf; honest that loom 1.0 ships no pipeline; keeps `real-host-smoke-gate.md`'s "not a CI-enforced gate" contrast meaningful.

**Cons.** The *"blocks merge"* guarantees remain policy rather than mechanically provisioned by a leaf.

**Risks.** A reader may still assume an automated gate the plan does not provide; the definition must say explicitly that loom 1.0 provisions only the `npm test` gate, not the merge-blocking substrate.

### Option B — Provision the CI substrate via a horizontal leaf

**Approach.** Give `H1a` (or a new horizontal leaf) the obligation to provision a CI workflow that runs `npm install && npm run build && npm run typecheck && npm test` on the merge-candidate event, configured as a required check; have `conventions.md`'s verdict word cross-reference that leaf.

**Plan edits.** `H1a` Adds / Tests / Ships when gain the workflow-provisioning obligation and a check that the workflow file exists and is well-formed; `conventions.md` cross-references `H1a`; `coverage-matrix.md` term-sweep as in Option A.

**Spec edits.** None (GOV-18 bars the spec from specifying this).

**Pros.** Grounds the merge-blocking guarantee in a provisioned artifact.

**Cons.** Larger surface; introduces runner-specific config the plan otherwise avoids; the merge-blocking half (branch-protection / required-check admin setting) is repo configuration a leaf cannot fully assert.

**Risks.** A leaf can assert a workflow file exists but cannot assert the repo-admin required-check setting that actually blocks merge, so the guarantee stays partly out of a leaf's reach even under this option.

### Recommendation

Take Option A: define the verdict word once in `conventions.md` and name the merge-blocking substrate as cross-referenced project policy rather than a provisioned loom 1.0 leaf, because every leaf already scopes its gate to `npm test` observability and the plan provisions no host/runner config — so the honest grounding is "`npm test` is the gate; the merge-blocking substrate is project policy." The concrete edit: insert a definition at the top of §Cross-cutting rules whose text fixes that *"a CI failure"* / *"reddening CI"* / *"fails CI"* means the `H1a`-provisioned unified `npm test` gate exiting non-zero, and that the substrate running `npm test` on the merge path and blocking merge on red is the project's required-check policy (named there), explicitly outside loom 1.0 leaf scope; then retarget the bare usages in `conventions.md` and `coverage-matrix.md` to that term. If a provisioned pipeline is judged in scope, fold Option B's workflow obligation into `H1a` instead. Watch: `real-host-smoke-gate.md` already declares the manual smoke is *not* CI-enforced — the new definition must not read so broadly that it sweeps the manual smoke into the CI-enforced category.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads *"is a CI failure"* / *"blocks merge"* as a mandate to provision a merge-blocking automated pipeline (and invents a runner, trigger events, and required-check configuration the plan never pins); another reads it as shorthand for "a local `npm test` arm reds" and ships no automated enforcement at all, leaving every *"merge blocker"* / *"MUST NOT be merged"* guarantee in the gate and version-bump prose unbacked. The merge-blocking semantics are load-bearing across the closing gate, the release gate (`H6a`), and the version-bump revert path (`V18d`), so an under-enforced reading silently weakens the entire gate architecture without any gate reddening.
