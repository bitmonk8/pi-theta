## Finding

The file-extension-grammar bullet in `overview-and-orientation.md` describes the `.warp` top-level forms as "(currently five: `import`, `export`, `fn`, `schema`, `enum` — *orientation aggregator per [Governance — GOV-30](./governance.md)*)". The word "currently" is a time-relative qualifier in normative-adjacent text: read locally, it leaves open whether the count is fixed at five for loom 1.0 or whether the set is an open list that may expand without a seam.

The surrounding governance machinery actually answers this — but only if the reader chases two cross-references. The list's normative owner, [Imports — Permitted top-level forms](./imports.md#permitted-top-level-forms), closes it ("Top-level may contain only `import`, `export`, `schema`, `enum`, and `fn` declarations", with `loom/parse/warp-top-level-statement` firing on anything else), so the count is fixed at five for loom 1.0 and there is no 1.0 extension seam for additional `.warp` top-level forms. GOV-30 makes the aggregator informative and lock-steps it to its source (its worked example for a future change is literally "adds a sixth `.warp` permitted form" — a spec-version event, not a pre-built seam), and GOV-31 registers "the five `.warp` permitted top-level forms" as an integer-count aggregator whose parseable integer literal MUST equal the source-set count at every commit.

So the defect is local imprecision, not a corpus-level gap: "currently" should be replaced by phrasing that pins the count to loom 1.0 (resolving the fixed-vs-expandable ambiguity in favour of fixed), while still preserving a parseable integer literal `five` in the same bullet as GOV-31 requires and retaining the GOV-30 orientation-aggregator tag.

## Issue introduction

## Solution Space

**Shape:** single

### Recommendation

In the `#file-extension-grammar` bullet, replace "currently five" with a version-pinned form that names loom 1.0 explicitly and keeps the parseable integer literal `five`, e.g.:

> `.warp` files are library modules whose top level is restricted to the declaration forms enumerated in [Imports — Permitted top-level forms](./imports.md#permitted-top-level-forms) (exactly five in loom 1.0: `import`, `export`, `fn`, `schema`, `enum` — *orientation aggregator per [Governance — GOV-30](./governance.md)*).

Constraints the editor must honour:

- Preserve a parseable integer literal `five` somewhere in the same bullet — the bullet is a GOV-31 integer-count aggregator, so paraphrasing the count away is itself a spec defect.
- Retain the `*orientation aggregator per [Governance — GOV-30]*` tag (the GOV-30 lock-step convention is what governs adding a sixth form in a future version).
- Do **not** add a forward-compat seam reference: there is no loom 1.0 seam for additional `.warp` top-level forms; expansion is a future spec-version change under GOV-30 lock-step, not a pre-built extension point.

## Consequence

**Severity:** cosmetic

The "currently" qualifier is locally imprecise, but the normative owner (`imports.md`) closes the list and the aggregator is explicitly informative under GOV-30, so no implementer diverges. The cost is reader friction in an orientation paragraph; the fix tightens wording without changing any observable behaviour.
