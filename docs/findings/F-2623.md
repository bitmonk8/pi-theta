## Finding

`descriptions.md` mandates that `///` doc comments lower to JSON Schema `description:` fields, emitted byte-for-byte into the lowered schema and passed to the model for structured output. The lowering pass therefore emits the `description` keyword as part of every loom-declared schema that carries doc comments.

The authoritative enumeration in `schema-subset.md` — introduced by the sentence "The normative subset loom emits and enforces is:" — never lists `description`. It enumerates Types, Composition, Validation (`enum`, `const`), Objects, Arrays, Reuse, Nullability, Discriminated unions, Depth, and Draft, and a separate "Explicitly **not** supported … rejected at parse time" list. `description` appears in neither list. The Lowering Algorithm's per-type-form emission rules (step 3) likewise never emit `description`, and the Canonical-schema-hash recipe's HTML comment enumerating keywords deliberately absent from the lowered fragment (`default`, the structural keywords) does not mention `description` either.

This is a contradiction between two normative pages: `descriptions.md` requires `description:` in the lowered output, while `schema-subset.md` — the page that claims to be the source of truth for "the normative subset loom emits and enforces" — omits it entirely, neither admitting it as an emitted annotation keyword nor excluding it. The omission is load-bearing because `schema-subset.md`'s subset is consumed as an allowlist (reject-by-default): an implementer building the emit set or the accept gate strictly from this enumeration would not account for `description` at all, and the canonical-hash recipe's silence leaves undefined whether emitted `description` text participates in the slug-determining canonical-form bytes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 53f5831 (`spec: rewrite query primitive, system prompts, and schema syntax`, 2026-05-03)
**History:** The Schema Subset normative enumeration was first authored in `spec.md` at 53f5831, where the `### Schema Subset` section already opened with "the normative subset is:" and listed Types/Validation/Objects/Arrays/etc. without `description`, while the Descriptions section in the same revision already stated `///` "lower to JSON Schema `description:` fields." The omission has carried through unchanged ever since: the enumeration was reaffirmed at 8eea523, split into the topic file `descriptions.md`/`schema-subset.md` at fecb504, relocated at 81bf52a, and amended by later subset/hash findings (111e533 "Canonical hash recipe cites keywords the lowered subset never emits", 6892aa3 SUBS-1), none of which added `description`. `git log -S "description"` over `docs/spec_topics/schema-subset.md` reports no commit that ever added or removed the token, confirming `description` was never present in that page's enumeration. The defect is thus a co-introduced gap between the emit-set enumeration and the description-lowering rule that has existed since the subset was first written, not a regression from any later edit.

## Solution Space

**Shape:** single

### Recommendation

Add `description` to the normative emitted-subset enumeration in `schema-subset.md`, classified as an **emitted annotation keyword** rather than a validation keyword: it is generated by the lowering pass from `///` doc comments (never author-written as raw JSON Schema), carries no validation semantics, and is emitted byte-for-byte per `descriptions.md`. State explicitly that `description` is part of the lowered fragment so the reject-by-default allowlist (V5d) admits it and the Lowering Algorithm step 3 accounts for it.

Reconcile the Canonical-schema-hash recipe at the same time: state whether emitted `description` text participates in the canonical-form bytes that determine the schema slug. The straightforward disposition is that it **does** participate (it is part of the lowered fragment, subject to the same RFC 8259 string-escaping and key-sorting rules), with the implementer-visible consequence that two structurally identical inline schemas differing only in their doc-comment text receive different slugs and do not dedup. If the intended behaviour is the opposite — that `description` is stripped before hashing so doc-comment edits do not perturb cached/replayable artefacts — then the HTML-comment list of keywords absent from the hashed fragment must add `description`, and the recipe must say `description` is removed prior to canonicalisation.

Edge cases the implementer must watch: `///` on a `fn` does not lower into JSON Schema (functions have no schema), so no `description` is emitted there; and a schema with no doc comments emits no `description` key, so its presence is conditional and must not be added to `required` or any all-keys-present obligation.

## Consequence

**Severity:** correctness

An implementer building the emit set or the reject-by-default allowlist strictly from `schema-subset.md`'s enumeration would omit `description`, contradicting `descriptions.md`'s byte-for-byte emission mandate; one conforming implementation emits `description`, another drops or rejects it. Because the recipe never states whether emitted `description` text is part of the canonical-form bytes, two implementations can also produce different schema slugs for the same schema — and the slug is a declared byte-exact breaking-change contract governing inline-schema dedup and synthesised tool names.
