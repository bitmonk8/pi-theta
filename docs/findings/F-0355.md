## Finding

Both `V4e` and its paired tests leaf `V4e-T` terminate their **Deps.** field with a multi-sentence parenthetical that explains *why* each dependency exists and *how* the watcher-time reload failure-injection seam is wired — e.g. `V4e`'s Deps closes with "(`V5e` owns the loom-owned depth walk whose ceiling-#4 slash-load `params` routing decision and depth-6 breach `ERR-16` wraps; `V9b` declares the `ReloadFailureInjector` interface (`injectReloadFailure`) that `V4e` imports … and `ERR-7`'s synthetic failure is injected through that one interface)". This is design / seam-wiring narrative, which `conventions.md` §Leaf format and `leaf-template.md` both assign to **Adds.** (descriptive prose) — the **Deps.** field is specified as the enumeration of leaf IDs that must complete first, optionally with minimal inline labels, and the pickability predicate keys only on those IDs.

For `V4e` the rationale is largely already duplicated in its **Adds.** paragraph (which states the `V16a` arbitration-seam consultation and the `V5e` depth-walk binding), so the Deps parenthetical is redundant prose. For `V4e-T` the **Adds.** is the minimal "Failing tests for the paired `V4e` implementation leaf", so the same wiring rationale lives *only* in its Deps parenthetical, where it does not belong.

The leaf-ID list itself is correct in both leaves; only the placement of the explanatory prose is wrong.

## Solution Space

**Shape:** single

### Recommendation

Relocate the seam-wiring rationale from the **Deps.** parenthetical to **Adds.** in both leaves, and reduce each **Deps.** field to the leaf-ID list with at most short inline labels.

- In `docs/plan_topics/V4e-pre-evaluation-failures.md`, strike the trailing parenthetical from the **Deps.** line (everything from "(`V5e` owns the loom-owned depth walk…" through "…injected through that one interface)"), leaving `**Deps.** V4e-T, V9a, V6a, V11f, V10a, V16a, V9b, V10d, V5e`. The `V16a`/`V5e` binding rationale is already carried by the **Adds.** paragraph; fold the `ReloadFailureInjector` wiring sentence (that `V9b` declares the `injectReloadFailure` interface `V4e` imports, with the registry-swap / `.loom`-`.warp` re-parse arms in `V9b` and the settings-re-merge arm in `V10d`) into **Adds.** if it is not already stated there.
- In `docs/plan_topics/V4e-T-pre-evaluation-failures.md`, strike the trailing parenthetical from the **Deps.** line, leaving `**Deps.** V9a, V6a, V11f, V10a, V16a, V9b, V10d, V5e`, and move the `V5e` depth-walk / `ReloadFailureInjector` wiring rationale into its **Adds.** field (which currently reads only "Failing tests for the paired `V4e` implementation leaf").

Preserve every dependency ID and every cross-leaf seam reference; the edit changes only which field the prose lives in. Note that any seam named in Adds binds only per the `conventions.md` §Leaf format Adds-binding rule (a downstream leaf naming the seam in its own Deps), so moving the prose does not change binding semantics.

## Consequence

**Severity:** cosmetic

The dependency IDs are all present and correctly listed, so the build-order / pickability machinery is unaffected and no implementer is misled about *what* must complete first. The defect is organisational: load-bearing seam-wiring rationale sits in a field specified to hold only IDs, duplicating (for `V4e`) or hiding (for `V4e-T`) content that belongs in Adds, which degrades readability and field-discipline consistency across the corpus.
