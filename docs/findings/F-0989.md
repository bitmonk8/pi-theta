## Finding

`bindings.md` and `grammar.md` together define `let x = e` and the `LetStmt` grammar, and `loom/parse/immutable-rebinding` covers a *reassignment* (`x = …`) of a non-`mut` binding. Neither file states what a *second* `let x = e` (or `let x: U = e`) inside the same block does. The grammar admits the second `LetStmt` and the diagnostic registry has no code that fires on it; the prose closest to the question — expressions.md §"Identifier resolution", which says local bindings "shadow everything else lexically, the same as in Rust or TypeScript" — is about identifier resolution across the four namespaces (locals, top-level `fn`/import, callable set), not about whether two `let` statements may introduce the same name in the same scope. The simile is also self-cancelling: Rust accepts same-scope shadowing and rebinds; TypeScript rejects it with a duplicate-declaration error.

A conforming implementation can therefore pick either Rust-style shadowing (the second `let` introduces a fresh binding, possibly with a new type, that masks the first for the remainder of the block) or TypeScript-style rejection (a same-scope second `let` of an in-scope name is a parse error) and remain consistent with every cited rule. The two behaviours are observable: under shadowing, `let x = 0; let x: string = "a"; f(x)` typechecks against `f(s: string)`; under rejection, the second `let` fails to parse and no later sites see the new binding. A conformance test can pin neither outcome today.

## Solution Space

**Shape:** single

### Recommendation

Make a `let`/`let mut` whose pattern names an identifier already bound in the **same** lexical scope a parse error, `loom/parse/duplicate-binding`. Cross-scope shadowing (an inner block introducing a same-named binding) remains permitted and resolves per `expressions.md` §Identifier resolution.

Apply these spec edits:

- In `bindings.md`, add a `**Same-scope redeclaration.**` paragraph: a `let`/`let mut` whose pattern names an identifier *already bound by an earlier `let`, `let mut`, function parameter, `for` iteration variable, or `match` pattern in the same lexical scope* is `loom/parse/duplicate-binding`; cross-scope shadowing (inner block introducing the same name) is permitted and resolves per `expressions.md` §Identifier resolution.
- Register `loom/parse/duplicate-binding` in `docs/spec_topics/diagnostics/code-registry-parse.md` (E, parse): "second `let`/`let mut` introducing a name already bound in the same lexical scope", with rendered template `cannot redeclare binding '<name>' in the same scope; use a different name or wrap in a nested block to shadow`, owner link back to `bindings.md`.
- In `grammar.md` §`let form`, add a sentence pinning this disposition to the grammar: a second `LetStmt` introducing a name already bound in the same lexical scope is `loom/parse/duplicate-binding`.
- In `expressions.md` §"Identifier resolution", change "shadow everything else lexically, the same as in Rust or TypeScript" to "shadow non-local namespaces lexically; same-scope redeclaration is `loom/parse/duplicate-binding` (see Bindings)".

The fix paragraph in `bindings.md` must enumerate the source scopes the rule counts (top-level loom, function body, `if`/`else`/`while`/`for` body block, `match`-arm block expression, bare block expression) so the line between "same lexical scope" and "nested scope" is precise — the diagnostic must not catch `for x in xs { let x = … }`, where the `let` is in the body's nested scope.

Three edge cases govern when the diagnostic fires:

- The rule fires on the *same* lexical scope only — an inner block legitimately introduces a fresh binding.
- The rule applies symmetrically to function parameters and `for` / `match` bindings as the colliding *prior* binding, not only to a prior `let`.
- The per-scope check runs after `Pattern` is resolved, so `let _ = …` followed by `let _ = …` is permitted (the discard sigil is not a name).

## Consequence

**Severity:** correctness

Two reasonable implementers will produce divergent parsers: one accepts `let x = 0; let x: string = "a"` and rebinds, the other rejects it. The divergence is silent (no diagnostic differs in name or shape — under the shadowing reading there is no diagnostic at all, under the rejection reading a presently-unnamed code fires), so cross-implementation `.loom` portability and any test suite asserting either behaviour are at risk.
