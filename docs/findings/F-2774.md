## Finding

`V9a` and its paired test leaf `V9a-T` both carry the PIC-6 `Tests` bullet:

> `PIC-6`: the factory never throws — each check is `try`/`catch`-wrapped.

The phrase "the factory never throws" is a **factory-wide** property, but `V9a`'s mechanism and acceptance only cover the capability probe's five checks. `V9a` adds the probe (Node floor, `AbortSignal`/`AbortController` shape, SDK members, peer-dep lock-step, `typebox` shape), routing any check that throws to `loom/load/host-incompatible` with `details.kind = "probe-failed"` per the spec's *Self-failure* clause. Throws elsewhere in the factory — the constant exports (`FACTORY_PROBABLE_CAPABILITIES`, `SHUTDOWN_AWAIT_CAP_MS`), the `host-incompatible` emission path, and the registration / subscription calls — are not in `V9a`'s scope and are not observed by its tests.

The plan already partitions this responsibility explicitly: `V9k` and `V9p` each state "The factory never-throw boundary is owned by `H4a`," and `H4a`'s own text reads "this leaf only establishes the never-throw factory boundary, the harness, and the fidelity contract." `H4a`'s acceptance is the one that asserts "the factory … never throws even when a host seam is absent" across the whole factory body. So `V9a`'s bullet, by restating "the factory never throws," claims an ownership it does not hold and duplicates `H4a`'s boundary in prose. The spec's PIC-6 invariant (`capability-probe.md`, *Probe-wide invariants* + *Self-failure*) is itself probe-scoped — its "factory MUST NOT throw" is established by the per-check try/catch trapping each check to `probe-failed` — so the leaf wording overstates relative to both the spec's intent and the plan's own ownership split. The coverage matrix correctly maps PIC-6 to `V9a`; only the leaf's test-bullet prose is overbroad.

## Consequence

**Severity:** advisory

An implementer reading `V9a`'s PIC-6 bullet may write a test asserting the *whole* factory never throws (covering constant exports and the emission path) — work that belongs to `H4a` — or, conversely, leave the probe's self-failure trap under-tested because the bullet's wording does not name the `probe-failed` routing it actually verifies. The duplicate-ownership prose also makes the V9a/H4a boundary harder to reason about during the fix loop. No leaf is blocked and no gate misfires; the probe still ships correctly.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V9a-capability-probe.md` and `docs/plan_topics/V9a-T-capability-probe.md`, rewrite the PIC-6 `Tests` bullet so it describes the probe self-failure behaviour `V9a` actually verifies and cites the factory-wide boundary to `H4a`. Replace:

```
- `PIC-6`: the factory never throws — each check is `try`/`catch`-wrapped.
```

with text scoped to the probe, for example:

```
- `PIC-6`: each probe check is `try`/`catch`-wrapped so a throw is trapped and emitted as `host-incompatible` with `details.kind = "probe-failed"` rather than escaping; the factory-wide never-throw boundary is owned by `H4a`.
```

Apply the identical edit to both the implementation leaf and its `-T` pair so their `Tests` lists stay in lock-step. Leave the PIC-6 → `V9a` row in `coverage-matrix.md` unchanged — PIC-6 remains closed by `V9a`; only the leaf's prose is being scoped. Do not edit `H4a`, `V9k`, `V9p`, or the spec: the factory-wide boundary already lives in `H4a` and the spec's PIC-6 invariant is already probe-scoped via its *Self-failure* clause.
