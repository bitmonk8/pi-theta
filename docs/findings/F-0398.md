## Problem

The `[invocation.md — cross-mode matrix]` Tests bullet in both `V15a-invocation-core.md` and `V15a-T-invocation-core.md` closes with "a child uses its own model/tools/system." That phrase compresses the spec's **Tools and model** rule in `invocation.md` and drops its load-bearing "not inherited from the caller" half. Read in isolation, the clause leaves open whether the child's configuration is wholly independent of the parent's or seeded/merged from it — the discriminating case when parent and child declare different `model:` / `tools:` in their `.loom` frontmatter. The bullet links to the spec section that disambiguates it, so the defect is the leaf prose's own terseness, not the spec.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commit:** `c6a664e` — "pi-loom plan: build/update plan for spec.md + review" (2026-06-10)

**History:** The terse clause has been present in the cross-mode-matrix Tests bullet of both `V15a-invocation-core.md` and `V15a-T-invocation-core.md` since the initial plan build. `git grep -l "uses its own model/tools/system" c6a664e` returns both files, while the parent `c6a664e^` returns nothing, pinning `c6a664e` as the introducing commit. `git log -S "uses its own model/tools/system"` scoped to both files returns only `c6a664e`, so the clause was authored once and never reworded; the later over-bundling split (`cb8737a`) and the prompt→prompt snapshot/restore additions (`6a94146`) relocated and extended the surrounding bullets but left this clause unchanged. The spec's own **Tools and model** rule in `invocation.md` has always carried the unambiguous "not inherited from the caller" wording that the plan paraphrase compressed away.

## Solution approach

In the `[invocation.md — cross-mode matrix]` Tests bullet, rewrite the "a child uses its own model/tools/system" clause to carry the no-inheritance half of the spec's **Tools and model** rule — that the child uses the model, tools, and system prompt from its own `.loom` frontmatter and does not inherit the caller's. Apply the identical edit to both `V15a-invocation-core.md` and `V15a-T-invocation-core.md` so the implementation leaf and its paired `-T` test leaf stay in lockstep.

## Solution constraints

- The spec (`invocation.md` and the rest of `docs/spec_topics/`) is read-only for this fix; its **Tools and model** rule already states the contract correctly.
- Out of scope: the prompt→prompt parent-suspend facet of the matrix, owned by `V15d`.
