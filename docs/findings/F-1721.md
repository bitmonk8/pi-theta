## Finding

In `docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md`, the fourth bullet of the "Consequences" list under the snapshot/restore protocol carries a ~460-word block describing the defensive per-extension-instance runtime mutex: its construction (wrapping `snapshot → swap → body → restore`, keyed on the factory-captured `pi: ExtensionAPI` reference), its runtime contract (`<a id="recovery-mutex-runtime-contract"></a>` — strictly serialising FIFO `await`, no steady-state acquisition timeout, `session_shutdown` teardown bounded by `SHUTDOWN_AWAIT_CAP_MS` via the holder's `finally`), and its keying-granularity invariants. The block is written as if it were part of the always-shipping snapshot/restore protocol.

It is not. The block governs only the *recovery path* — the corrective edit a contributor performs in the same commit as a Pi minor bump when the editorial-review checklist's item-(e) audit (`version-bump-step2.md#bump-checklist-slash-dispatch-serialisation`) records a `fail` (or non-positive verdict under the conservative-posture rule) on sub-outcome (e.i) or (e.ii). Both the inline prose and item (e) make this explicit: *"loom 1.0 ships without it on a pass outcome, since precondition (b)'s cooperative-`await` chain establishes the non-overlap property in that case."* Loom 1.0 ships on the pass arm, so the mutex's construction, FIFO contract, keying invariants, and teardown semantics are all dead prose for any implementer of loom 1.0 itself.

The placement defect is that this conditional contingency sits among the unconditional `pi.setActiveTools` consequences and immediately above the unconditional PIC-8 / PIC-19 failure protocols, with no heading, anchor, or sub-section break separating "ships in loom 1.0" from "ships only if a future Pi minor bump records `fail` on item (e)." The inline `<a id="recovery-mutex-runtime-contract"></a>` is exactly the kind of pin that benefits from a labelled home rather than being buried mid-bullet. A linear reader of PIC-8/PIC-19 must parse the recovery contract every time they read the snapshot/restore protocol, and a reader trying to discover what loom 1.0 actually ships under this section cannot tell at-a-glance which sentences are in-scope without reading them.

## Solution Space

**Shape:** single

### Recommendation

Extract the recovery-mutex prose from the `pi.setActiveTools` consequences bullet into a clearly-conditional sub-section on the same page, immediately after the snapshot/restore consequences list and before any unconditional content that follows. Concretely:

- Add a sub-heading along the lines of `### Recovery path (item-(e)-fail arm only)` (or an equivalent labelled block) that opens with one normative sentence stating the gating predicate: "This sub-section is in scope only when [item (e) of the editorial-review checklist](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) records `fail` (or a non-positive verdict under the conservative-posture rule) on sub-outcome (e.i) or (e.ii); loom 1.0 ships on the pass arm and does not implement it."
- Move the entire recovery-mutex prose — construction, FIFO `await` runtime contract, `session_shutdown` teardown interaction, and keying-granularity invariants — into that sub-section verbatim.
- Preserve the `<a id="recovery-mutex-runtime-contract"></a>` anchor at the same prose position so the inbound cross-reference from `version-bump-step2.md` item (e) continues to resolve byte-for-byte.
- Leave the PIC-8 / PIC-19 bodies untouched; in the now-trimmed `pi.setActiveTools` consequences bullet, replace the relocated prose with a one-line pointer ("On the recovery path engaged by an item-(e) fail outcome, see [Recovery path](#recovery-mutex-runtime-contract) below for the defensive per-extension-instance serialisation contract loom adds in the same edit as the bump.") so a linear reader of the bullet still knows the contingency exists.

Edge cases an implementer must watch:

- The cross-reference from `version-bump-step2.md` item (e) uses the fragment `#recovery-mutex-runtime-contract` — verify the anchor still resolves after the move (it should, since the anchor travels with the prose) and that the link text remains accurate.
- The defect lives in `tool-registration-lifetime.md` only; do **not** move the prose to `version-bump-step2.md`. The finding's "or move to the version-bump item (e)" alternative is discarded because the runtime contract describes how the loom runtime serialises its own snapshot/restore window — content that belongs on the snapshot/restore page, not in the bump procedure that triggers its construction.
- Do not collapse the relocated prose into the version-bump page's item (e) body either; that page already cites this section by name and depends on `#recovery-mutex-runtime-contract` being authored once at the snapshot/restore site.

## Consequence

**Severity:** advisory

Loom 1.0 ships on the item-(e) pass arm, so no implementer is misled into building the mutex. The cost is reader load on every pass through PIC-8/PIC-19 and the obscured scope of what loom 1.0 actually implements — a structural defect, not a correctness one.
