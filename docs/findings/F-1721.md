## Problem

In `tool-registration-lifetime.md`, the `pi.setActiveTools` consequences bullet under the snapshot/restore protocol carries a large inline block (with the `<a id="recovery-mutex-runtime-contract"></a>` anchor) describing the defensive per-extension-instance runtime mutex — its construction, FIFO `await` contract, `session_shutdown` teardown interaction, and keying-granularity invariants. That block governs only the recovery path engaged when item (e) of the editorial-review checklist (`./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation`) records a `fail` on a future Pi minor bump; loom 1.0 ships on the pass arm and does not implement it. The conditional contingency sits inline among the unconditional `pi.setActiveTools` consequences and the unconditional PIC-8 / PIC-19 protocols with no heading or sub-section break, so a reader cannot tell at-a-glance which sentences are in scope for loom 1.0.

**Verdict:** introduced-by-named-commits
**Introducing commits:** 57d8235 (2026-05-11, "pi-loom spec: resolve \"T23 per-session slash-handler serialisation citation\""); 823ce54 (2026-06-06, "pi-loom spec: resolve \"Recovery-mutex acquisition semantics and teardown-budget interaction undefined\"")
**History:** Commit 57d8235 introduced the defensive per-extension-instance runtime mutex prose by appending it to the existing `pi.setActiveTools` consequences bullet in `docs/spec_topics/pi-integration-contract.md` (pre-split path); the inline placement among unconditional snapshot/restore consequences — without a conditional sub-section break — dates from that commit. Commit 823ce54 then enlarged the inline block by adding the `<a id="recovery-mutex-runtime-contract"></a>` block (FIFO `await` contract, `session_shutdown` teardown observation, `SHUTDOWN_AWAIT_CAP_MS` interaction), resolving the acquisition-semantics finding by adding content at the same in-bullet site rather than relocating it. Commit f5e89f4 (2026-06-04) moved the file to `docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md` as part of the 100KB-cap split; that commit preserved the inline placement, it did not introduce it. No later commit has reorganised this bullet.

## Solution approach

Move the recovery-mutex prose out of the `pi.setActiveTools` consequences bullet into a clearly-conditional sub-section on the same page, opening with the gating predicate (in scope only on an item-(e) `fail` arm; loom 1.0 ships on the pass arm without it). Carry the `<a id="recovery-mutex-runtime-contract"></a>` anchor with the prose, and leave a short pointer in the now-trimmed consequences bullet so a linear reader still discovers the contingency.

## Solution constraints

- Preserve the `recovery-mutex-runtime-contract` anchor so the inbound cross-reference from `version-bump-step2.md` item (e) (`#recovery-mutex-runtime-contract`) continues to resolve.
- Out of scope: do not relocate the prose into `version-bump-step2.md` item (e); the runtime contract is authored once on the snapshot/restore page that item (e) cites.
