## Finding

The `H5d` transitive-completeness gate arm reconciles each `coverage-matrix.md` closing-leaf cell against the membership of `H5b`'s `Deps.` field — it reads the literal text of `H5b`'s `Deps.` line as input data. `H5d`'s own `Deps.` is `H5a` and correctly does not name `H5b`. Separately, `H5b`'s warn-only canary runs `H5a`'s unified closing-gate machinery, of which the `H5d` arm is a part.

`H5d`'s `Adds.` prose describes the read ("requires AT LEAST ONE of the leaf IDs … to be a member of `H5b`'s `Deps.`") but never states that this is a page/data read — satisfied merely by `H5b` having been authored — rather than a completion-tag dependency. The pickability predicate in `conventions.md` (§Leaf format) makes a `Deps.` entry a completion-tag edge (`<id>-complete` git tag must exist). A maintainer who reads "`H5d` uses `H5b`'s `Deps.`" as "`H5d` depends on `H5b`-complete" would add `H5b` to `H5d`'s `Deps.`. Because `H5b` already runs machinery that includes the `H5d` arm, that mis-edit closes a circular `H5b`⇄`H5d` completion dependency that no leaf could then satisfy.

The current plan text is not broken — the directions are correct as written — but the relationship is left implicit, so the safe authoring is undocumented and the unsafe edit is one plausible misreading away.

From that first commit its `Adds.` prose has read `H5b`'s `Deps.` as data while its own `Deps.` was `H5a`, and the text carried no statement disambiguating a data/page read from a completion-tag dependency (no "page read", "completion", or "MUST NOT list" language). Later edits (2ad25ab, 8b69cd9) refined the tokenisation and at-least-one wording but never added the disambiguation, so the latent ambiguity is structural to how the leaf was authored.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H5d-transitive-completeness-gate.md`, in the `Adds.` field where the arm's use of `H5b`'s `Deps.` is described, state explicitly that the arm reads `H5b`'s `Deps.` field as input data — a read satisfied by `H5b` being authored — and that this is not a completion-tag dependency, so `H5d`'s `Deps.` stays `H5a` and does not list `H5b`. The `Deps.` line itself already reads `H5a` and needs no change.

`conventions.md`'s *Transitive-completeness plan-maintenance* clause owns the single statement of the obligation that backs this arm; do not duplicate the obligation there. The spec is read-only for this fix.

## Consequence

**Severity:** advisory

If `H5b` is later added to `H5d`'s `Deps.` under the misreading, the `H5b`⇄`H5d` completion-tag cycle leaves both leaves permanently unpickable on the dep-driven build order. The current `Deps.` values are correct, so a contributor who follows the plan verbatim still produces a working leaf; the exposure is a maintenance/clarity gap, not a live break.
