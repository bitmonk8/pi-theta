## Finding

The `### Rules` list in `docs/spec_topics/descriptions.md` contains two adjacent bullets that disagree about whether dedenting happens. The **Multi-line** bullet is normative and explicit: "Common leading whitespace inside the description is stripped (same algorithm as query-template dedent)." The very next operative bullet, **No transformation**, then states: "no escaping, dedenting, or wrapping is performed beyond the multi-line join and common-leading-whitespace strip defined above."

"Common-leading-whitespace strip" and "dedent" are the same operation — the Multi-line bullet itself names the strip "same algorithm as query-template dedent", and the query-template rendering rules elsewhere in the spec consistently call that step *dedent*. Listing "dedenting" inside the *exclusion* clause of the No-transformation bullet therefore contradicts both its own `beyond …` carve-out and the Multi-line bullet two lines above.

The `beyond …` qualifier technically resolves the contradiction for a reader who notices the two names refer to one transform, but the surface reading is the opposite of intent: an implementer scanning the bullet headlines (`No transformation.` → `no … dedenting … is performed`) can plausibly conclude that no dedenting happens at all, contradicting the byte-for-byte lowering tests called out by the V5c leaves. The fix is to stop using "dedenting" as a synonym for an *excluded* transform when the spec uses it as the canonical name for the *performed* transform.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** 7f217a8 ("pi-loom spec: resolve \"Language-core hidden assumptions: enum-tag sidecar, AJV-config silence, Markdown claim\"", 2026-06-07)
**History:** Before 7f217a8 the bullet read `**Markdown.** Description text is treated as Markdown by providers; no transformation is performed.` — a one-sentence Markdown-provenance claim with no enumeration of excluded transforms, and therefore no opportunity to contradict the Multi-line bullet. The fix-loop commit 7f217a8 rewrote the bullet to its current form (`**No transformation.** Loom emits description text byte-for-byte into the lowered schema; no escaping, dedenting, or wrapping is performed beyond the multi-line join and common-leading-whitespace strip defined above. *(Non-normative provenance.)* …`). The Multi-line bullet that performs the strip predates the rewrite (introduced in 53f5831, 2026-05-03, "spec: rewrite query primitive, system prompts, and schema syntax"), so the contradiction is wholly attributable to the 7f217a8 rewrite — that single commit chose the word "dedenting" for the exclusion list while the same paragraph already used the same operation in the carve-out.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/descriptions.md`, in the `**No transformation.**` bullet, drop `dedenting,` from the exclusion list so the sentence becomes:

> **No transformation.** Loom emits description text byte-for-byte into the lowered schema; no escaping or wrapping is performed beyond the multi-line join and common-leading-whitespace strip defined above. *(Non-normative provenance.)* …

The Multi-line bullet retains exclusive authority over the dedent/common-leading-whitespace-strip behaviour; the No-transformation bullet keeps its byte-for-byte-emission claim and its remaining exclusions (escaping, wrapping). No other site needs to change — the Multi-line bullet already names the strip as "same algorithm as query-template dedent", which preserves the link to QRY-7 for readers tracking the dedent algorithm. Leave the `beyond the multi-line join and common-leading-whitespace strip defined above` carve-out in place; it remains useful as a forward-pointer that pins exactly which transforms *are* performed.

## Consequence

**Severity:** advisory

A careful reader reconciles the two bullets via the `beyond …` carve-out and the Multi-line bullet's authority, so a conforming implementation is still derivable. But the contradiction wastes review attention every time the page is read, and a hasty reader who anchors on the No-transformation headline can talk themselves into "no whitespace stripping" — which would diverge from the multi-line join + common-leading-whitespace strip the V5c leaves assert.
