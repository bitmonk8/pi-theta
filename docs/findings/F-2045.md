## Problem

The hard-ceiling-#4 per-boundary destination/surface table exists as two near-identical normative copies: the canonical owner at `ceilings-3-and-4.md#ceiling-4-table` and a depth-walk-specific mirror in `schema-subset.md`'s `## Depth Enforcement` section. No normative lock-step binds the two copies, so a future edit to either table has no trigger requiring the sibling copy to be updated in the same commit, and the two can silently drift. The only acknowledgement of the relationship — a trailing sentence in `## Depth Enforcement` — points readers at the aggregator `overview-and-orientation.md#hard-runtime-ceilings`, which holds only a one-line forward-link, not the full table, rather than at the canonical owner. GOV-30's lock-step convention governs only `spec.md`-aggregator-vs-source parity, not source-page-to-source-page duplication.

**History:** `1750848` added the boundary-dependent per-boundary surfaces table to `schema-subset.md` (the duplicate) together with the "this section's table mirrors it … lives in `spec.md` — Hard ceilings" sentence, while the canonical table was still inline in the spec.md orientation block. Later the same day, `1c7f550` relocated the canonical table out of informative orientation into its dedicated anchored home (`ceilings-3-and-4.md#ceiling-4-table`), after which the `schema-subset.md` "mirrors it" pointer no longer resolved to the table's actual location and no reciprocal source-to-source lock-step pin was ever added on either page.

## Solution approach

In `schema-subset.md`'s `## Depth Enforcement`, repoint the trailing "mirrors it" cross-link from `overview-and-orientation.md#hard-runtime-ceilings` to the canonical owner `ceilings-3-and-4.md#ceiling-4-table`. At `ceilings-3-and-4.md#ceiling-4-table`, add a reciprocal forward-link naming the `## Depth Enforcement` per-boundary surfaces table as the depth-walk mirror and asserting the lock-step relationship. Add or extend a governance lock-step rule binding the source-to-source mirror so that editing either table obliges updating the sibling copy in the same commit.

## Solution constraints

- The governance lock-step rule MUST name the canonical owner `ceilings-3-and-4.md#ceiling-4-table` explicitly and MUST NOT inherit GOV-30's `spec.md`-aggregator-relative phrasing (which mis-points).
