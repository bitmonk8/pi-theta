## Finding

`coverage-matrix.md`'s range note defines `X-n … X-m` as the inclusive contiguous range — *every* REQ-ID from `X-n` through `X-m`, both endpoints included — and explicitly handles the one failure mode it anticipates: a numbering *hole* inside a range (a number that is neither live nor retired) surfaces as the `H5a` per-prefix-numbering-hole gate failure rather than a silent skip. It says nothing about a **retired interior** REQ-ID. A retired ID is accounted for by the prefix's `## Retired REQ-IDs` retirement table, so per `conventions.md` *REQ-ID discipline* ("Gaps that the retirement table accounts for are not failures") it passes the numbering-hole check silently while the range notation still textually spans it.

The two reconciliation directions of the `H5a` gate treat the matrix asymmetrically. The unmapped-executable-REQ-ID direction enumerates the *live* executable REQ-ID set from the governance prefix table (retired IDs are excluded by definition — "a registered prefix that anchors no live REQ-ID contributes nothing"). The citing-test direction, however, enumerates *from the matrix*: `conventions.md` treats "a coverage-matrix-mapped numbered REQ-ID with no citing test in the test corpus" as a CI failure. If the gate expands `X-n … X-m` textually and treats every member as a mapped REQ-ID, a retired interior ID becomes a "mapped" REQ-ID that — being retired — has no citing test by design, tripping a spurious "mapped-REQ-ID-with-no-citing-test" failure.

The defect is latent: it materialises the first time a REQ-ID interior to a live `…` range is retired (the matrix already uses ranges spanning multiple endpoints, e.g. `TYPE-1 … TYPE-10`, `ERR-1 … ERR-7`, `CIO-2 … CIO-4`). The gate's range-expansion behaviour is unstated on this point, so a reasonable `H5a` implementer can build a naive textual expander that produces the false positive, while another intersects the expansion with the live executable set and does not. The fail direction is safe (loud, not silent), but it would spuriously redden the warn-only canary and the loom 1.0 release gate until the matrix range is manually split.

## Solution Space

**Shape:** single

### Recommendation

Amend `coverage-matrix.md`'s range note (the `X-n … X-m` paragraph) so the retired-interior-ID case is handled alongside the existing holes clause, and the gate's "mapped REQ-ID" enumeration excludes retired IDs before the citing-test reconciliation runs. Append to that paragraph a sentence to the effect of:

> A REQ-ID interior to a `…` range that the prefix's `## Retired REQ-IDs` retirement table accounts for is **not** a coverage-matrix-mapped REQ-ID for the citing-test reconciliation: the `H5a` gate intersects the expanded range with the live executable `PREFIX-N` set (excluding retired IDs) before requiring a citing test, so a retired interior ID carries no citing-test obligation and never reddens the mapped-REQ-ID-with-no-citing-test arm.

Mirror the same exclusion in `conventions.md` *REQ-ID discipline* where the citing-test reconciliation is defined: state that the "coverage-matrix-mapped numbered REQ-ID" set the citing-test scan iterates is the matrix mappings intersected with the live executable REQ-ID set, so retired IDs (already outside the executable set) are likewise outside the citing-test obligation even when they fall interior to a `…` range.

Edge cases the implementer must watch: the exclusion must apply only to the citing-test direction's range expansion, not weaken the per-prefix numbering-hole arm (a genuine hole — neither live nor retired — must still redden); and a retired endpoint (not interior) of a range should not silently shrink the live coverage obligation for the still-live members.

## Consequence

**Severity:** correctness

The `H5a` citing-test gate arm is underspecified on retired-interior-ID handling in range expansion: one reasonable implementer builds a textual expander that treats a retired interior ID as a mapped REQ-ID and the arm fires a spurious "mapped-REQ-ID-with-no-citing-test" failure; another intersects expansion with the live executable set and it does not. The first variant reddens the warn-only canary and the loom 1.0 release gate the first time a REQ-ID interior to a live range is retired, blocking the release until the range is manually split — a false positive in the safe (loud) direction.
