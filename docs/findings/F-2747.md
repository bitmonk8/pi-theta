## Finding

`V18c`'s **Ships when** condition has two structurally different arms joined into one leaf-completion signal. The first arm is fully mechanical: `npm run typecheck` runs the `loom/typecheck/session-shutdown-reason-snapshot` brand-string assertion green, and `npm test` runs the step-2(a)/2(b) surface-inventory, `engines.node` three-way equality, `peerDependencies` literal-read, snapshot literal-array consistency, provider seed-field `Api`-coverage, and strict-capability probe gates green on `main`. The second arm is explicitly *not* mechanical — the leaf's own text says it is "confirmed by editorial review — not by `npm test`": a per-bump human audit of checklist items `(a)`–`(aq)`, MUST-level for `(a)`–`(e)`, recorded as per-item outcomes in the bump commit message.

These are qualitatively different done-signals. The mechanical arm is a build-time gate authored once and re-run forever by CI; the editorial arm is a recurring per-bump human procedure whose outcome lives in a commit message and is invisible to any test runner. The pickability predicate (`conventions.md` §Leaf format **Deps**) keys leaf completion on the presence of a `V18c-complete` git tag, which a contributor tags at the TDD-ritual phase-exit gate once `npm test` is green — but under the current Ships-when that tag also silently asserts a human editorial sign-off that no automated surface can witness. The leaf bundles "the gate tests are authored and green" with "a per-bump editorial audit was performed and recorded," so completing and verifying `V18c` cannot be done in one coherent mechanical pass.

The leaf already separates the two in its **Adds** prose ("Separately — and *not* a build-time gate — the contributor's editorial-review checklist"), which makes the residual co-location in **Ships when** the precise defect: the classification was separated but both done-signals still gate the same leaf.

## Consequence

**Severity:** advisory

The leaf cannot be completed and verified in a single mechanical pass: a green `npm test` / `npm run typecheck` run does not satisfy `V18c`'s stated Ships-when, yet the `V18c-complete` git tag (the pickability signal) is produced at the CI-green phase-exit gate. A contributor following the mechanical arm tags `V18c` complete while the human editorial-review acceptance — a per-bump MUST-level audit — has no observable completion surface, so its omission is not caught by any gate.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commit:** `f1cda4830f779b741a035bcd8f841bff6a699fc0` (2026-06-11) — "pi-loom plan: resolve 'V18c mislabels editorial-review checklist as a static build-time gate'"
**History:** At the leaf's inception (`c6a664e`), `V18c`'s Ships-when was purely mechanical (`npm test` runs the step-2(a)/2(b), `engines.node`, peer-dep, and reason-snapshot gates), and the editorial-review checklist was mis-listed among the build-time gates in **Adds**. Commit `f1cda48` correctly reclassified the editorial-review checklist as *not* a build-time gate in **Adds** and, in the same edit, added its acceptance as a second Ships-when arm ("confirmed by editorial review — not by `npm test`"), introducing the present co-location of two done-signals in one leaf's completion gate. Later commits refined the editorial arm without introducing the bundling: `828b7d3` (2026-06-16) added the MUST/SHOULD blocking distinction, and a subsequent edit extended the item range to `(aq)`. The de-bundling defect therefore dates to the `f1cda48` reclassification, which separated the *classification* of the editorial review but kept both signals gating the same leaf.

## Solution Space

**Shape:** multiple

### Option A — Extract the editorial-review acceptance into its own version-bump process-gate leaf

**Approach:** Move the editorial-review checklist acceptance out of `V18c`'s Ships-when into a new leaf `<new>` with `Deps. V18c`. `V18c`'s Ships-when keeps only the mechanical `npm run typecheck` + `npm test` arms. The new leaf carries the per-bump editorial-review acceptance (items `(a)`–`(aq)`, the MUST/SHOULD blocking distinction, and the commit-message per-item-outcome recording).

**Plan edits:**
- `V18c-version-bump-checklist.md` **Ships when** — strike the `; and the editorial-review checklist's acceptance is confirmed by editorial review — not by \`npm test\` — that …` clause, ending the gate at the strict-capability probe arm.
- Add a new leaf `docs/plan_topics/<new>-version-bump-editorial-review.md` whose Ships-when is the editorial-review acceptance text struck from `V18c`, with `Deps. V18c`.
- `plan.md` §V18 listing — link the new leaf under "V18 — Build-time SDK gates."

**Spec edits:** None.

**Pros:** Each leaf has one homogeneous done-signal (CI-green for `V18c`, human-recorded acceptance for `<new>`); parallels `V18d`, which already carries the runtime-evidence acceptance as its own leaf. Keeps the editorial acceptance trackable in the dep graph.

**Cons:** Adds a leaf whose done-signal is a commit-message record rather than a test, the same not-fully-mechanical shape the editorial arm already had — it relocates the human signal rather than removing it.

**Risks:** The new leaf's completion remains human-attested; if the leaf-ID overlaps a separately-proposed `V18e`, the implementer must allocate a distinct ID under the project's ID-allocation policy.

### Option B — Relocate the editorial-review acceptance to a cross-cutting governance note

**Approach:** Move the editorial-review checklist acceptance to a cross-cutting governance note (sibling to `real-host-smoke-gate.md`), referenced from the §Release gate / version-bump area, leaving `V18c` with only its mechanical gates.

**Plan edits:**
- `V18c-version-bump-checklist.md` **Ships when** — strike the editorial-review clause as in Option A.
- Add a governance note (e.g. `docs/plan_topics/version-bump-editorial-review-gate.md`) carrying the editorial-review checklist acceptance, parallel to how `real-host-smoke-gate.md` is "a cross-cutting governance note, not a leaf."
- `plan.md` — add a reference to the note alongside the existing real-host-smoke-gate pointer.

**Spec edits:** None.

**Pros:** Matches the established treatment of the manual real-host smoke governance protocol (a note, not a leaf); keeps `V18c` fully mechanically verifiable; no new leaf-ID is consumed.

**Cons:** The editorial-review acceptance leaves the dep graph entirely, so it is not surfaced by the pickability predicate at all; tracking that the per-bump audit happened rests solely on the governance note and the bump commit message.

**Risks:** Two version-bump acceptance surfaces (`V18d` runtime-evidence as a leaf, editorial-review as a note) diverge in form, which a future reader may find inconsistent.

### Recommendation

Take **Option A**: extract the editorial-review acceptance into a dedicated process-gate leaf depending on `V18c`. This leaves `V18c`'s Ships-when fully mechanical (the green CI run is the whole signal) and gives the per-bump editorial audit its own homogeneous human-recorded done-signal, parallel to `V18d`'s runtime-evidence acceptance leaf. Allocate the new leaf's ID under the project's ID-allocation policy (use the `<new>` placeholder until then; do not assume `V18e`, which a separate `V18d`-split finding also proposes). Watch that the relocated editorial-review acceptance text retains its MUST-level `(a)`–`(e)` blocking distinction and the per-item commit-message recording requirement verbatim, since downstream findings about checklist range and the item-`(e)` second-pass obligation attach to that text.
