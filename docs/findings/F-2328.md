## Problem

`M`'s and `M-T`'s SLSH-2 acceptance criteria require dispatching a prompt-mode loom through the `H4a` harness, issuing one untyped `` @`<literal>` `` query, and observing one streamed assistant response appended as a single prompt-mode turn. `H4a`'s **Adds** provides a "minimal response-emission capability" for the session double, but scopes it as "sufficient only to drive self-check items (i) and (ii)" of the fidelity contract; it never states that this emission fires on the harness-dispatched prompt-mode query path. The scripted-injection route that drives the double's emission is category (a) of the `H4b` response-programming surface, and `M`/`M-T` depend only on `H4a` — not `H4b`. A fresh implementer cannot tell whether `Deps: H4a` is sufficient for the SLSH-2 dispatch path or whether that path has no defined emission source until `H4b`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `c6a664e` (`pi-loom plan: build/update plan for spec.md + review`, 2026-06-10) — inception of `M`/`M-T`'s SLSH-2 "assistant response … as one appended turn" requirement on the harness-dispatched query path; `32ea081` (`pi-loom spec-plan: fix F-0477 — define H4a minimal emission capability + H4a/H4b boundary`, 2026-06-20) — added the "minimal response-emission capability … sufficient only to drive self-check items (i) and (ii)" scoping to `H4a`.
**History:** `git log -S "assistant response"` on the `M`/`M-T` files shows the SLSH-2 assistant-response requirement present since the inception commit `c6a664e`. `git log -S "sufficient only to drive self-check"` on `H4a` shows the narrowing scoping entered only at `32ea081` (F-0477); before F-0477, `H4a` defined no minimal emission at all and the full response-programming surface was entirely `H4b`'s. The defect is the interaction: F-0477 introduced an `H4a`-owned emission deliverable but bounded it to the internal fidelity self-check, without stating that the same emission is observable on the harness-dispatched prompt-mode query path that `M`/`M-T` had depended on since inception. Neither commit is wrong in isolation; the gap is in the seam between the inception-era `M`/`M-T` requirement and the F-0477 scoping.

## Solution approach

In `docs/plan_topics/H4a-factory-shell-and-harness.md`, clarify in the **Adds** that the minimal response-emission capability is observable through a harness-dispatched prompt-mode untyped `@`-query — the path `M`/`M-T` exercise — and not solely through the fidelity self-check. Add a corresponding `H4a` **Tests** bullet whose observable is a harness-dispatched prompt-mode untyped `@`-query yielding one streamed assistant response appended as a single prompt-mode turn. This makes `M`/`M-T`'s `Deps: H4a` sufficient without pulling in `H4b`'s scripted-injection surface.

## Solution constraints

- Out of scope: `H4b` category (a) (scripting the sequence of assistant turns) remains `H4b`'s; do not add `H4b` to `M`/`M-T`'s `Deps`.
