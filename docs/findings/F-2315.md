## Problem

`V18b` lands the build-time inventory-closure audit as a fail-closed gate inside `npm test` (Ships when: "`npm test` runs the closure audit green on `main`"), so every contributor's `npm test` and CI is gated by the audit's recogniser. The family-(4) "out-of-scope import/access shape" discriminator is deliberately non-exemptible: per the `V18b` Adds clause and `audit-failures.md` §Failure-surface contract, an `// allow-pi-surface:` marker on a family-(4) line cannot suppress the family-(4) record. Non-exemptibility is correct for genuine out-of-scope shapes, but it means a recogniser bug that misroutes a legitimate category-(1)/(2)/(3) reference to family-(4) reds the whole suite for every contributor with no in-band remedy until the walker is fixed. The `V18b` leaf states no operational recovery for that case, unlike the revert posture `H6a` documents for its activation commit.

## Solution approach

Add to the `V18b` leaf an operational-recovery statement for a recogniser false positive that reds the non-exemptible family-(4) arm. Name the revert of the single audit-introducing commit — which `inventory-audit-intro.md` already pins to land the audit in one commit — as the path that restores `main`'s pre-audit green footing, mirroring the revert posture `H6a` documents for its activation commit.
