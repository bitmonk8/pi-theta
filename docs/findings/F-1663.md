## Problem

The "Other arithmetic" section in `docs/spec_topics/expressions.md` carries two rules that disagree on the static type of `integer % 0`. The general `%` operator-typing rule pins the result type from operand static types, making `n % 0` (with `n: integer` and a literal `0`) statically `integer`; the immediately following modulo-by-zero clause reasons from the runtime `NaN` value back to the static type and widens the result to `number`. The two give contradictory outcomes for `let x: integer = n % 0` — the operator rule accepts it, the widening clause requires rejection via `loom/parse/integer-narrowing` — leaving conformance for the typed-binding case undecidable. The sibling safe-integer-overflow paragraph already establishes the opposite precedent for the same shape of case: an `integer`-typed result whose runtime value exceeds what the static type can represent retains the operator-assigned `integer` type rather than widening.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `a1f34f2` (2026-06-01, "% by zero is unspecified") and `bd51505` (2026-06-05, "% operand-typing rule contradicts integer subset number widening")
**History:** `git log -S "an \`integer % 0\` result widens to \`number\`" -- docs/spec_topics/expressions.md` returns `a1f34f2`. At that point the surrounding `%` rule was "requires same-typed operands and, for a non-zero divisor, preserves the type" — under which `integer % 0` was outside the "preserves the type" carve-out, so claiming the result "widens to `number`" was internally consistent (the divisor-is-zero case was the only `%` case the page typed at all, and it typed it as `number`). Four days later, `bd51505` resolved a sibling finding by rewriting the general `%` rule to "two `integer` operands produce `integer`, and either operand being `number` widens the result to `number`" — pinning the static type from operand types alone — but left the `% 0` clause from `a1f34f2` verbatim. The contradiction is the interaction: each commit was locally coherent, and the conflict only appears in the combined text. The sibling commit `537612e` (2026-06-05, integer-overflow paragraph) further established the precedent that runtime values exceeding what an `integer` can represent do **not** retroactively widen the static type, sharpening the present contradiction by giving the page two competing dispositions for the same kind of case.

## Solution approach

Rewrite the modulo-by-zero clause in the "Other arithmetic" section so `integer % 0`'s static type follows the operator-typing rule (statically `integer`, binding the runtime `NaN`), aligning it with the adjacent safe-integer-overflow paragraph's disposition rather than widening to `number`. Add a normative conformance vector to the `V3a` / `V3a-T` "Other arithmetic" *Tests* bullet pinning that `let x: integer = n % 0` compiles and binds to `NaN`.
