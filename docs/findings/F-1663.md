## Finding

The "Other arithmetic" paragraph in `docs/spec_topics/expressions.md` carries two rules that disagree on the static type of `integer % 0`.

The general `%` rule (the second sentence on `%`) pins the result type by operand static types: "two `integer` operands produce `integer`, and either operand being `number` widens the result to `number`." Under this rule `n % 0` with `n: integer` and a literal `0` (also `integer`) is statically `integer`.

The very next sentence then says: "because `NaN` is a `number`, an `integer % 0` result widens to `number` — the same `integer ⊑ number` widening defined in [TYPE-2]." This reasons from the runtime value (`NaN`) back to the static type, which the type system disallows — `n % 0` is statically `integer` whether or not `n` happens to be `0`, since the divisor's static type is `integer`, not "is-it-runtime-zero". The two sentences give contradictory expected outcomes for `let x: integer = n % 0`: the operator-typing rule accepts it; the widening clause requires `loom/parse/integer-narrowing` (or some equivalent rejection) because the RHS is now `number`. Conformance for the typed-binding case is undecidable.

The sibling overflow paragraph immediately after this one already states the correct precedent for the same family of cases: an `integer`-typed result whose runtime magnitude exceeds the safe-integer bound "retains the static `integer` type assigned by the operator's widening rule above rather than widening to `number`." `% 0` is the same shape (runtime value outside what the static type can faithfully represent), and should follow the same rule.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `a1f34f2` (2026-06-01, "% by zero is unspecified") and `bd51505` (2026-06-05, "% operand-typing rule contradicts integer subset number widening")
**History:** `git log -S "an \`integer % 0\` result widens to \`number\`" -- docs/spec_topics/expressions.md` returns `a1f34f2`. At that point the surrounding `%` rule was "requires same-typed operands and, for a non-zero divisor, preserves the type" — under which `integer % 0` was outside the "preserves the type" carve-out, so claiming the result "widens to `number`" was internally consistent (the divisor-is-zero case was the only `%` case the page typed at all, and it typed it as `number`). Four days later, `bd51505` resolved a sibling finding by rewriting the general `%` rule to "two `integer` operands produce `integer`, and either operand being `number` widens the result to `number`" — pinning the static type from operand types alone — but left the `% 0` clause from `a1f34f2` verbatim. The contradiction is the interaction: each commit was locally coherent, and the conflict only appears in the combined text. The sibling commit `537612e` (2026-06-05, integer-overflow paragraph) further established the precedent that runtime values exceeding what an `integer` can represent do **not** retroactively widen the static type, sharpening the present contradiction by giving the page two competing dispositions for the same kind of case.

## Solution Space

**Shape:** single

### Recommendation

Delete the "because `NaN` is a `number`, an `integer % 0` result widens to `number` — the same `integer ⊑ number` widening defined in [TYPE-2]" clause from the `Modulo by zero` sentence. Replace it with the same disposition the adjacent safe-integer-overflow paragraph already uses: `n % 0` produces `NaN` at runtime and does not panic; the static type of the expression is the type the operator-typing rule above assigns from the operand static types (so `integer % integer` is statically `integer`, and `let x: integer = n % 0` is well-typed and binds `x` to runtime `NaN`), and `/ 0` is unaffected because `/` already produces `number`. Add one normative vector to the `V3a` / `V3a-T` "Other arithmetic" *Tests* bullet pinning that vector (`let x: integer = n % 0` compiles and binds to `NaN`).

Edge cases the implementer must watch: `number % 0` still trivially yields `number`-typed `NaN` (no change); the runtime equality rule that `NaN == NaN` is `true` (runtime-value-model.md) applies to the bound value regardless of its static type, so `x == x` for the `integer`-typed `NaN` above is `true`; and downstream code reading `x: integer` as if it were a safe integer is on the author, the same way it is for an overflowed `integer`-typed product per the sibling paragraph.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on `let x: integer = n % 0`: one accepts it (per the operator-typing rule) and binds `x` to runtime `NaN`; another rejects it with `loom/parse/integer-narrowing` (per the runtime-widening clause). The defect is narrow — modulo by an `integer`-typed zero — but it sits on the typed-binding path, so divergence surfaces as either a spurious parse-time rejection or a silent `integer`-typed `NaN`, both of which are observable to author code via `match` arms and downstream arithmetic.
