## Finding

`V6d` (`docs/plan_topics/V6d-system-interpolation.md`) resolves a Path-only `${…}` in the frontmatter `system:` field and, per its Tests bullet, "routes the resolved value through the shared renderer" while "row-level correctness stays owned by `V13a`". `V6d-T` mirrors this: its vectors "witness only that the `system:` resolve-then-stringify path feeds each type into the shared renderer." But neither leaf names that renderer — no function identifier, no argument shape — and `V13a` (`docs/plan_topics/V13a-query-render.md`) does not declare its per-type stringification as a consumable surface. `V13a`'s Adds describes "the stringify-by-loom-type table with the degenerate-template short-circuit" purely as an internal part of the `@`-template `render` path; nothing states that an external caller (the `system:` path) may reach it.

The plan demonstrates the discharge pattern elsewhere: `V2d` declares "the canonical number-rendering pure function **shared by** the schema canonical-hash recipe (`V5f`) and the binder argument echo (`V11h`)", explicitly naming the owner-side seam and its consumers. The stringification seam shared between `V13a`'s `@`-template path and `V6d`'s `system:` path has no equivalent declaration, so the producer/consumer contract is carried only by the prose phrase "the shared renderer."

A second, sharper aspect: `V13a`'s render combines per-value stringification with the trim → dedent template pass, and per the spec stringification runs *before* dedent so multi-line interpolations participate in dedent. `V6d`'s `system:` field has no trim → dedent stage. The seam `V6d` actually needs is the per-value stringification step alone, not the whole-template renderer. The plan does not separate these, leaving the consumable surface's boundary unstated.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 3625ee0 — pi-loom plan: resolve "system: interpolation per-type stringification not witnessed" (2026-06-11, Thomas Andersen)
**History:** At the corpus's first commit (c6a664e, 2026-06-10) `V6d` carried its own "stringification table" in Adds and tested "stringifies per the table"; `V13a` independently owned the stringify-by-loom-type table from that same commit. Commit 3625ee0 rewrote `V6d`'s Tests to "route the resolved value through the shared renderer" owned by `V13a` and added "row-level correctness stays owned by `V13a`", but named no `V13a` entry point and added no `V13a`-side declaration that the renderer is a consumable seam — introducing the unnamed-seam coupling.

## Solution Space

**Shape:** single

### Recommendation

Establish the stringification as a named seam owned by `V13a`, following the `V2d` precedent, and have `V6d` consume it by that exact name:

- In `docs/plan_topics/V13a-query-render.md` Adds, restate "the stringify-by-loom-type table" as a reusable pure function — the per-value interpolation renderer mapping an evaluated value plus its Loom static type to rendered text per [`../spec_topics/query/query-escapes-stringification.md`](../spec_topics/query/query-escapes-stringification.md) — owned by `V13a` and reachable independently of the `@`-template trim → dedent pass. Name it with a single stable identifier and list its consumers (`V13a`'s own `@`-template rendering and `V6d`'s `system:` interpolation), mirroring `V2d`'s "the canonical … pure function shared by …" phrasing. The named surface is the per-value stringifier, distinct from the whole-template renderer that also performs trim → dedent.
- In `docs/plan_topics/V6d-system-interpolation.md`, replace the prose "the shared renderer" in the Adds and Tests bullets with `V13a`'s exact seam identifier, stating that the `system:` path feeds each resolved value into that `V13a`-owned per-value renderer and introduces no second renderer.
- In `docs/plan_topics/V6d-T-system-interpolation.md` Tests, replace "feeds each type into the shared renderer" with the same named seam.

Edge cases the implementer must watch: `V6d`'s `params:` types never include `Result`, so the `loom/parse/interpolated-result` row stays owned by `V13a` and does not arise on the `system:` surface — `V6d`'s consumption is a strict subset of `V13a`'s renderer surface. The seam `V6d` calls must be the per-value stringification, not the trim → dedent template render, since `system:` has no dedent stage. (The exact identifier string is the implementer's to fix in `V13a`; downstream leaves must then cite it verbatim — see the related Class-2 "names the seam" convention finding.)

## Consequence

**Severity:** advisory

The canonical spec table pins observable output, so two implementers converge on behaviour even if they diverge in structure; the concrete risk is that the `V6d` implementer, finding no named `V13a` entry point, re-implements per-type stringification locally, producing a second renderer that drifts from `V13a`'s over later edits. Because `V13a`'s render bundles stringification with trim → dedent, an implementer who reaches for "the renderer" wholesale may also drag the dedent pass into the `system:` path where it does not belong.
