## Finding

In `patch-skew-degradation.md`, the (P-a) / (P-b-1) / (P-c-1) fixture clauses pin the *mechanism* by which each fixture witnesses that the full five-sub-step teardown ran, not only the observable property. The current wording requires the fixture to assert each of the eight `details.call` sites and the sub-step-2 iteration and sub-step-3 bounded await "via spies on the injected `LoomRegistry`, `chokidar`-watcher, `Clock`, and signal-listener seams", "via a per-entry registry-iteration spy", and "via an injected `disposeBarrier` or `Clock.setTimeout` spy", with (P-c-1) additionally requiring "a fresh `LoomRegistry` per case". The observable property — that each of the eight `details.call` sites was invoked and that the sub-step-2 per-entry iteration and the sub-step-3 bounded `Promise.allSettled` await both ran — is already a complete witness; the seam list, the spy primitive, the per-entry iteration spy, the `disposeBarrier`-vs-`Clock.setTimeout` choice, and the fresh-registry-per-case scaffolding are all *how* the implementer might detect those observables, not *what* the contract requires.

Every other "decomposes into fixture obligations" block in this shard observes a stricter discipline. `unknown-reason-rule.md`, `drain-state-contract.md`, `tool-registration-lifetime.md`, `session-only-degraded-state.md`, and `diagnostic-emission-isolation.md` each open with the formula "all of which the implementation MUST satisfy (the leaf owns fixture-shape; branch coverage and \<some normative pin\> are normative)" — pinning the observable and explicitly delegating fixture realisation to the implementer. The patch-skew clause itself closes with the same delegation ("the implementation owns their `<file>` placement and exact assertion shape") and is then immediately undercut by the spy-mechanism prose preceding it.

The same paragraph also names `chokidar` directly. PIC's normative watcher seam is `FileWatcher` (`host-interfaces-services.md` PIC-14), and `host-prerequisites.md` already delegates `chokidar` to non-normative implementation-notes as the production adapter behind that seam. Naming `chokidar` in a normative fixture clause both leaks the implementation choice into the contract and contradicts the existing layering — a fixture wired against the `FakeFileWatcher` test double of PIC-14 should still satisfy this obligation.

## Solution Space

**Shape:** single

### Recommendation

In the *Patch-skew degradation contract* paragraph, rewrite the (P-a) / (P-b-1) / (P-c-1) per-sub-step observable enumeration to state only the observable obligation and delete the prescribed verification mechanism. Concretely, replace the sentence beginning "In each of (P-a), (P-b-1), and (P-c-1), the assertion that *the full five-sub-step teardown still runs* is not satisfied by witnessing a single teardown observable: …" through the end of the (P-c-1) fresh-`LoomRegistry` clause with text along the lines of:

> In each of (P-a), (P-b-1), and (P-c-1), the assertion that *the full five-sub-step teardown still runs* is not satisfied by witnessing a single teardown observable: the fixture MUST witness that each of the eight call sites enumerated in the closed `details.call` label set of the **Per-step isolation** paragraph below was invoked on this teardown, and MUST witness that sub-step 2's per-entry `ActiveInvocationRegistry` iteration ran and that sub-step 3's bounded `Promise.allSettled` await over the entries' `disposeBarrier`s ran (these two sub-steps carry no `details.call` label and are witnessed separately). The witness technique is implementation-owned — the leaf owns fixture-shape; the closed `details.call` set, the per-entry iteration, and the bounded await are the normative observables.

This deletes (a) the seam-list naming `LoomRegistry` / `chokidar`-watcher / `Clock` / signal-listener; (b) the per-entry registry-iteration spy prescription; (c) the `disposeBarrier`-or-`Clock.setTimeout` spy disjunction; and (d) the (P-c-1) fresh-`LoomRegistry`-per-case requirement. None of those are observables; each is one way an implementer might detect the observables the spec already pins. The "fresh registry per case" specifically is a test-isolation hygiene concern that the implementer will arrive at the moment they try to write the fixture — it does not need to be hoisted into the contract.

Two edge cases the implementer must still cover after the rewrite:

- The observable that sub-step 1's `LoomRegistry.drain` ran in (P-c-1) is preserved by the closed `details.call` set cross-reference, since `"loomRegistry.drain"` is one of the eight labels; a fixture harness that reuses a single drained registry across cases will fail this observable on the second case and the implementer will reach for fresh-per-case isolation without it being spec-mandated.
- The `details.call` set names the observable identity of each call site (e.g. `"discoveryWatcher.close"`); how the fixture demonstrates that the call happened — a spy on the seam, a counter on a `FakeFileWatcher`, a synchronous side-effect on a test double — is the implementer's choice. The contract remains falsifiable: a teardown that skipped `discoveryWatcher.close()` would emit no observable bound to that label.

Separately within the same edit, where the rewritten passage above (or any adjacent prose) references the file-watcher, use the `FileWatcher` seam name from PIC-14 rather than `chokidar`; `chokidar` is the production adapter and belongs only in the implementation-notes layer `host-prerequisites.md` already delegates it to.

## Consequence

**Severity:** advisory

Implementers can still produce a conformant fixture suite under the current wording, but the corpus convention ("the leaf owns fixture-shape") is weakened by the one paragraph that ignores it, and the `chokidar` mention either forces fixtures to wire the production adapter (defeating the `FakeFileWatcher` seam PIC-14 provides for exactly this purpose) or invites the implementer to ignore the literal prescription, training inattention to other prescriptions on the same page.
