## Finding

`functions.md` mandates three parse-reject diagnostics that V3d owns: `loom/parse/nested-fn` and `loom/parse/function-as-value` (FN-1 — nested `fn` declarations are rejected, and a function name used as a value surfaces `function-as-value`) and `loom/parse/return-no-common-type` (FN-3 — return/tail operands that share no common upper bound and that no sink narrows). The coverage matrix confirms ownership: `cka-7` maps `functions.md` (FN) and `return.md` (RET) to `V3d`.

Neither `V3d` nor its paired `V3d-T` carries a Tests bullet that fires any of these three codes. The only reject code asserted in the pair is `loom/parse/unreachable-code` (RET-3). The leaf's Adds enumerates the constraints these codes enforce ("no nested functions, closures, or functions-as-values", and return-type inference with the `return-no-common-type` failure mode), so the codes are in V3d's scope — they are simply unpinned by any acceptance criterion.

Because the matrix row is code-keyed (no numbered REQ-ID), the per-citing-test reconciliation does not bind these codes either, and the DIAG-2 reverse-gate that would catch the gap only binds at the H6a live-corpus release footing. An implementer following the leaf's Tests bullets can therefore mark V3d (and V3d-T) complete with all three reject codes untested — and plausibly unimplemented — until release.

## Solution Space

**Shape:** single

### Recommendation

Add Tests bullets to `V3d-T` (and mirror them into `V3d` per the paired-file convention) asserting that each of the three owned reject codes fires at its offending span:

- `loom/parse/nested-fn` fires on a nested `fn` declaration (FN-1, `functions.md`).
- `loom/parse/function-as-value` fires when a function name appears in value position — bound to `let` or passed as a call argument (FN-1, `functions.md`).
- `loom/parse/return-no-common-type` fires when the tail expression and `return` operands share no common upper bound and no sink narrows them (FN-3, `functions.md`); the `return expr` participating in the LUB regardless of static reachability per RET-1/RET-3.

Cite `functions.md` FN-1 / FN-3 and the Diagnostics code registry as the source anchors. These are code-keyed (`cka-7`) assertions with no numbered REQ-ID. The fix is internal to the two plan leaf files; the spec is read-only.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads the Adds and implements/tests all three FN/RET reject codes, another follows the Tests bullets verbatim and ships V3d without them. A V3d marked done can omit `nested-fn`, `function-as-value`, and `return-no-common-type` entirely — the gap goes undetected until the DIAG-2 reverse-gate binds at H6a, far downstream of where the codes are introduced.
