## Finding

`V17a` defines a four-site swallowing-handler routing set and delegates per-site routing to four owning leaves: `V14f`, `V13f`, `V15h`, and `V9o`. Three of those four leaves title themselves with the routing role they fill — `V13f` is "`@`-query provider swallowing-handler per-site routing", `V14f` is "Code-side `execute()` swallowing-handler per-site routing", and `V9o` is "Subagent-mode `AgentSession.abort()` swallowing-handler per-site routing". `V15h` alone titles itself by the mechanism outcome — "Invoke ceiling-#4 depth surfaces and swallowing-handler suppression" — rather than by its routing role.

The divergence makes `V15h` the one routing-site leaf that cannot be recognised as a member of the four-site routing set from its title alone; "suppression" names what the handler does, not the role `V15h` plays in `V17a`'s delegated set. The leaf's body already correctly frames itself as "the `invoke`-child entry in the four-site routing set `V17a` delegates to its owning leaves", so only the title is out of step. The same title text is mirrored in `docs/plan.md`'s `V15` table-of-contents entry, so the two must move together.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `5f4d2711c61f2e938df3d8fe40deceb169fa441d` — "pi-loom plan: resolve \"V15a bundles ~14 independent test obligations into one impl+T pair\"" (2026-06-17)

**History:** `docs/plan_topics/V15h-invoke-ceiling-swallowing.md` has exactly one commit in its history — its creating commit `5f4d271` — and its `# `V15h`` H1 carried "…swallowing-handler suppression" at inception (`git show 5f4d271` shows the file added with that title line). The sibling per-site-routing naming convention was already established when `V15h` was authored: `V13f`'s title token dates to `eaa5000` (2026-06-16) and `V14f`'s to `5a1915d` (2026-06-17 16:24, ~18 minutes before `V15h`'s 16:42 creation), with `V9o` adopting the same form shortly after in `6d7661d`. The divergence was therefore present from `V15h`'s first appearance — the leaf was created without picking up the already-live sibling pattern — rather than drifting in via a later edit; no subsequent commit reconciled the title.

## Solution Space

**Shape:** single

### Recommendation

Rename `V15h`'s H1 in `docs/plan_topics/V15h-invoke-ceiling-swallowing.md` so its routing role reads in the sibling "…per-site routing" form. Replace the current title line

```
# `V15h` — Invoke ceiling-#4 depth surfaces and swallowing-handler suppression
```

with

```
# `V15h` — Invoke ceiling-#4 depth surfaces and `invoke`-child swallowing-handler per-site routing
```

Apply the identical text change to the mirroring `V15h` table-of-contents entry under `### V15 — Invocation and imports` in `docs/plan.md` so the link text and the leaf H1 stay byte-identical. The leaf body keeps its existing "three-channel suppression" / "three-side-channel late-settlement suppression" wording — only the title's routing-role phrasing changes.

Edge case: if `V15h` is split into a depth half and a cancellation half (see the related splitting finding), the "…`invoke`-child swallowing-handler per-site routing" phrasing belongs on the cancellation half; the depth half keeps the ceiling-#4 depth-surface phrasing. Resolve the split first if it is taken, then apply this naming on the routing-site leaf that survives.

## Consequence

**Severity:** cosmetic

A reader scanning leaf titles for the four members of `V17a`'s routing set finds three named "…per-site routing" and one named "…suppression", so `V15h`'s membership is discoverable only by reading its body. No implementer produces a different leaf because of the title; the defect is pure naming inconsistency within an otherwise uniform set.
