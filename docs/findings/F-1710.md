## Finding

The QRY-6 parse-time `loom/parse/empty-template` bullet in `docs/spec_topics/query/query-forms.md` (line 101) describes the predicate it evaluates against the static body as: "*newline-trim and dedent **notionally** applied to those segments, the escape rewrites of [Query escapes — Escapes] **notionally not** applied*". The hedging adverb "notionally" obscures the algorithm: a conforming parser cannot tell, from this prose, whether it is required to (a) literally run newline-trim and dedent on each literal segment before testing for whitespace-only, (b) run them on a virtual concatenation of segments, or (c) merely reason about their effect without performing them. The same ambiguity applies in mirror image to "notionally not applied" for escape rewrites: must escapes be left as their source bytes (so the source `\n` reads as a two-character non-whitespace literal), or are they replaced and the predicate evaluated against the pre-escape source, or something else?

The surrounding text gives a partial answer — the suppression-hatch sentence (`\n` evaluated as the two-character sequence `` \n ``) commits the spec to **pre-escape source**. So the actual intent is: evaluate the predicate on the source segment bytes after newline-trim and dedent are applied, with escape rewrites not yet performed. The QRY-6 bullet should say this directly rather than via the indirection of "notionally".

## Solution Space

**Shape:** single

### Recommendation

Replace the two-clause "notionally" phrasing in the parse-time-warning bullet's predicate description with a direct statement that names the input form and the transforms explicitly. Concretely, in `docs/spec_topics/query/query-forms.md` line 101, change:

> … every literal segment between interpolations, taken as the parser's conceptual static-analysis approximation of the rendered static body — newline-trim and dedent notionally applied to those segments, the escape rewrites of [Query escapes — Escapes](query-escapes-stringification.md#escapes) notionally **not** applied — is empty or whitespace-only …

to:

> … the predicate is evaluated against the concatenation of the template's literal segments (the source bytes between interpolations, with escape rewrites of [Query escapes — Escapes](query-escapes-stringification.md#escapes) **not** yet applied), after newline-trim and dedent have been applied to that concatenation; if the result is empty or whitespace-only …

Edge cases the rewrite must preserve and the implementer should check:

- The suppression-hatch sentence that follows ("evaluating the predicate pre-escape, the body is the non-whitespace two-character sequence `` \n `` and the warning is suppressed") remains consistent — it reads more naturally once "notionally" is gone, because "pre-escape" now matches the bullet's predicate verbatim.
- The whitespace set pinned later in the same parenthetical (ASCII set per `defaulting-system-note-echo.md#system-note-rendering` rule 1, never the regex `\s` class) is unchanged.
- The wording must keep the predicate operating on **literal segments only**, not on interpolation placeholders or interpolated values — the static-analysis scope is what distinguishes the parse-time warning from the runtime short-circuit in the next bullet.
- The phrase "*static* body" (italic emphasis) should stay; it is the term-of-art the page uses to distinguish parse-time scope from the runtime "fully-rendered" text.

## Consequence

**Severity:** advisory

Two implementers reading "notionally" can land on the same observable behaviour by luck (the suppression-hatch sentence is the tiebreaker), but the predicate statement does not, on its own, pin which transforms run and on what input. Removing "notionally" in favour of a concrete pre-escape, post-trim/dedent statement closes the gap without changing what passes or fails the warning.
