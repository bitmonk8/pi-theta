## Finding

`V13d`'s **Tests** field carries four assertions, the last of which is the `ERR-17` failure path: "forced-respond non-compliance injects the synthesised `ValidationIssue` (path `""`, keyword `"required"`, branch-specific message) into the respond-repair loop." This is the closing test for spec REQ-ID `ERR-17` ([`queryerror-variants.md` — ERR-17](../spec_topics/errors-and-results/queryerror-variants.md#err-17)) at this leaf — `coverage-matrix.md` maps `ERR-17` to `V4d`, `V13d`.

`V13d`'s **Ships when** field, however, enumerates only three criteria: "`npm test` asserts attempt accounting, the context-overflow short-circuit, and the proximate-variant propagation." The `ERR-17` forced-respond non-compliance assertion is not named. The Ships-when gate is therefore narrower than the leaf's own Tests field: it omits one of the four asserted behaviours, and specifically the one failure-path criterion the leaf claims to close.

The asserting test does run under `npm test` (it is a Tests bullet), so the TDD ritual still exercises `ERR-17`. The defect is that the explicit gate criterion — the condition a contributor reads to decide whether `V13d` is shippable and may receive its `V13d-complete` tag — does not list the `ERR-17` assertion, so the gate prose understates the behaviours that must be green.

## Consequence

**Severity:** advisory

A contributor reading `V13d`'s Ships-when as the authoritative completion criterion sees three behaviours and may tag `V13d-complete` without treating the `ERR-17` forced-respond non-compliance assertion as gating, even though the leaf claims to close that failure path. Because the `ERR-17` test is a Tests bullet that runs under `npm test`, the TDD ritual still covers it, so this degrades the precision of the gate rather than blocking the leaf or leaving `ERR-17` unimplemented.

## Solution Space

**Shape:** single

### Recommendation

This brings the gate into agreement with the leaf's Tests field, which already asserts that behaviour.
