## Finding

The `invoke`-parent observation rule in `session-shutdown-semantics.md` states that "The parent's `?` / `match` arms covering the child's `Err(QueryError { kind: "cancelled" })` … do **not** run inside the bounded-await window of sub-step 3". The phrasing groups the `?` operator into a single "arms" category with `match`. The two are different constructs: `match` is a scrutinee with handler arms (the rule the surrounding paragraph genuinely targets), while `?` is short-circuit error propagation that does not have "arms" at all.

A careful reader is left to decide whether the rule covers `?`-propagation as well, or whether `?` was named only as a sibling of `match` for completeness (so the rule applies to all parent-side handling of the child `Err`, propagation included). The intent — established by the next clause, which grounds inertness in the parent's own `loomAbort` having fired and its next `Checkpoint` await observing the abort — is clearly that *all* parent code observing the child's `Err` is inert during teardown, including a bare `child?`. The text just bundles `?` into the wrong noun.

The downstream `cancellation.md` "Propagation" paragraph inherits the same `?` / `match` arms wording when it summarises the session-shutdown carve-out, so the ambiguity propagates to the cancellation audience as well.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the offending sentence in `session-shutdown-semantics.md`'s `invoke`-parent observation bullet to separate `?`-propagation from `match` arms, and apply the same separation to the inheriting sentence in `cancellation.md`'s Propagation paragraph.

Replace:

> The parent's `?` / `match` arms covering the child's `Err(QueryError { kind: "cancelled" })` (or `kind: "invoke_callee"` wrapping it, per [Cancellation — Surfacing](../cancellation.md)) do **not** run inside the bounded-await window of sub-step 3

with:

> The parent's `?`-propagation and explicit `match` branches covering the child's `Err(QueryError { kind: "cancelled" })` (or `kind: "invoke_callee"` wrapping it, per [Cancellation — Surfacing](../cancellation.md)) do **not** run inside the bounded-await window of sub-step 3

— i.e., adopt the suggested-fix wording, replacing the noun phrase "`?` / `match` arms" (which mis-bundles `?` under "arms") with two coordinated noun phrases that correctly name each construct. The clause that follows ("the parent's own `loomAbort` has already fired … its next `Checkpoint` seam await observes the abort, and the parent unwinds without re-entering interpreter dispatch") already provides the mechanical justification covering both forms — `?`-propagation expansion and a `match` arm are both interpreter-dispatched steps after the child returns — and needs no change.

Apply the same noun-phrase split to the corresponding clause in `cancellation.md`'s Propagation paragraph ("parent's `?` / `match` arms covering the child's `Err` do not run inside the bounded-await window …"), so the two surfaces stay in lockstep.

Edge case: the rule explicitly only covers `?` and `match` against the child's `Err`. Any other parent-side post-child code (a non-`Err`-matching statement following an `Ok`-bound `let`) is already covered by the general "parent unwinds at its next `Checkpoint`" rule and does not need to be enumerated; do not expand the rewrite to enumerate every parent-side construct.

## Consequence

**Severity:** advisory

Two implementers could read the rule differently: one treats both `let x = child?;` and `match child { … }` as inert during teardown (the intended reading); the other reads `?` as still propagating the child `Err` upward inside the bounded-await window and only suppresses the `match` form. The behavioural divergence is small — both implementations still unwind at the next `Checkpoint` — but the surface the second implementer builds (a `?`-propagated `Err` returned from the loom during teardown) would diverge in what its enclosing frames observe before the parent's own checkpoint fires.
