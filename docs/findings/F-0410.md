## Problem

`V17a-T` authors failing-red cancellation-core tests that script aborts and a late settlement through the end-to-end harness — the forwarding, downward-only propagation, and swallowing-handler suppression tests all drive the scripted-injection / late-settlement API that is the response-programming surface owned by `H4b` (which `H4a` deliberately does not provide). But `V17a-T`'s `Deps` field lists only `V8a`, `V4d`, `H4a`, omitting `H4b`. The paired impl leaf `V17a` correctly lists `H4b`, and `H4b`'s `Adds` field names `V17a` as a consumer; `V17a-T` is the sole harness-consuming test leaf that does not declare the `H4b` dependency. Without it, `V17a-T` becomes pickable before the response-programming surface exists, so an implementer would improvise an ad-hoc cancellation mechanism and produce a suppression test that can pass vacuously.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** fd6169b — pi-loom plan: resolve "H4a bundles three independent units into one leaf" (2026-06-12, Thomas Andersen)
**History:** Before fd6169b the response-programming surface lived in `H4a`, and both `V17a` and `V17a-T` depended on `H4a` alone, so no asymmetry existed. Commit fd6169b split the scripting surface out into the new `H4b` leaf and updated the `V17a` impl `Deps` from `…H4a` to `…H4a, H4b`, but left the paired `V17a-T` `Deps` at `V8a, V4d, H4a`. The pickaxe (`git log -S 'H4b' -- V17a-T-cancellation-core.md`) confirms the `H4b` token has never appeared in `V17a-T`; the dependency divergence was introduced entirely by that single incomplete-propagation edit.

## Solution approach

Add `H4b` to the `Deps` field of `docs/plan_topics/V17a-T-cancellation-core.md`, matching the paired `V17a` impl leaf and the consumer enumeration in `H4b`'s `Adds`. This pins `V17a-T` so it cannot be scheduled before the response-programming surface exists.
