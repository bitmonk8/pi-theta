## Finding

Version-bump step 6 (`docs/spec_topics/pi-integration-contract/version-bump-triggers.md`) tells the contributor to add a row for a newly-introduced `Api` value "with `omitted` as the conservative default unless pi-ai documents a seed field for that provider". The verb "documents" is an unbounded evidence threshold: it admits at least three distinct readings — (a) the field appears in pi-ai's published type declarations, (b) it is called out in pi-ai release notes, (c) it is observable in pi-ai's in-source request-adapter code. These yield different verdicts for the same provider (a field may be present in source/types but unmentioned in release notes, or vice versa), so two contributors can pick different defaults for the same candidate minor.

The very next sentence in the same step states the operative rule precisely: the contributor "MUST consult pi-ai release notes and the new provider's request-adapter source to decide between a named field and `omitted`". The conditional that gates the `omitted` default (`unless pi-ai documents a seed field`) is therefore looser than, and out of step with, the two concrete evidence sources the following MUST names. A reader who acts on the parenthetical's "documents" alone has no anchor to the release-notes-plus-adapter-source standard the step actually requires.

## Solution Space

**Shape:** single

### Recommendation

Align the gating conditional with the two evidence sources the step's MUST already names. Replace the parenthetical default clause so the `omitted` default is overridden only by the same evidence the contributor is required to consult — e.g. "(with `omitted` as the conservative default unless pi-ai's release notes or the new provider's request-adapter source explicitly identify a seed field for that provider)". This makes the conditional and the following MUST state one threshold. Leave the existing MUST sentence and the "do not silently regress a provider for which pi-ai has added seed support" obligation intact; the edit only tightens the parenthetical to match them.

## Consequence

**Severity:** advisory

Two contributors reconciling a new provider against the same candidate pi-ai minor could disagree on whether to seed a named field or the conservative `omitted` default, because the gating conditional and the operative MUST state different evidence bars. The build-time `Api`-coverage assertion forces a row to exist but does not adjudicate `omitted` vs a named field, so a provider could be silently entered as `omitted` (degraded) when adapter-source evidence would have justified a named field. The downstream MUST already constrains a careful reader, which bounds the divergence to the local wording.
