## Finding

`docs/spec_topics/future-considerations/surface-extensions.md` holds two of the four top-level future-considerations categories defined in the hub preamble, in sequence:

1. `## Tooling deferrals (no loom 1.0 impact)` — category 1 (items that ship as new tools/commands with no loom 1.0 runtime decision: LSP support, `loom test`, richer telemetry, automated Pi-patch re-import cadence).
2. `## Surface extensions (loom 1.0 leaves a seam)` — category 2 (items that extend a loom 1.0 type/struct/call shape).

The file name, its H1 (`# Surface extensions`), and the hub's Contents link (`docs/spec_topics/future-considerations.md` → `## Contents` → `[Surface extensions](./future-considerations/surface-extensions.md)`) all name only category 2. Nothing at the navigation surface signals that category-1 (Tooling deferrals) content lives inside this file. A reader who follows the hub's explicit four-category framing to locate category 1 has no Contents entry, no filename, and no H1 pointing here; the only thread is the `<a id="tooling-deferrals-no-loom-1-0-impact">` redirect stub (mirrored in the hub just above `## Contents`), which is an anchor, not a discoverable label.

The page is non-normative orientation/future-work material, so the defect is one of discoverability and organisation rather than runtime or contract correctness, but it is real: the canonical name of the file silently drops half its content.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, Thomas Andersen)
**History:** Before f5e89f4 all four future-considerations categories lived in a single `future-considerations.md` whose name correctly covered every section, so no misname existed. f5e89f4 split the over-limit corpus files into a subdirectory+index layout; it grouped category 1 (Tooling deferrals) and category 2 (Surface extensions) into one child page but named that page `surface-extensions.md` after only category 2, while categories 3–4 went to `model-changes-and-non-goals.md`. The category-1 content has been present under a category-2 name in every commit of this file since its creation; the dual-section content has not changed materially since.

## Solution Space

**Shape:** single

### Recommendation

Give the Tooling-deferrals category its own page so each future-considerations category lives in one file, matching the hub's four-category model.

- Create `docs/spec_topics/future-considerations/tooling-deferrals.md` and move the `## Tooling deferrals (no loom 1.0 impact)` section into it, leaving `surface-extensions.md` holding only `## Surface extensions (loom 1.0 leaves a seam)`.
- Move both anchor alias spellings — `tooling-deferrals-no-loom-1-0-impact` and `tooling-deferrals-no-v1-impact` — into the new page together; relocating only one breaks a redirect.
- Remove the now-misleading top-of-file redirect-anchor stub from `surface-extensions.md`.
- In `docs/spec_topics/future-considerations.md`, add a `## Contents` entry pointing to `./future-considerations/tooling-deferrals.md`, and repoint the hub's redirect-anchor stub to the new page.
- Repoint the two cross-references to the tooling-deferrals anchor — in `host-prerequisites.md` and `patch-skew-degradation.md` — to `tooling-deferrals.md`. The ≈50 category-2 path references to `surface-extensions.md` stay valid and are left unchanged.

## Consequence

**Severity:** cosmetic

A reader following the hub's four-category framing cannot locate the Tooling-deferrals category by filename, H1, or Contents link, and must discover it incidentally by opening the category-2 file. No normative content, runtime behaviour, or implementer decision is affected.
