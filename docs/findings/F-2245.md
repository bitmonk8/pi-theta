## Problem

`docs/spec_topics/future-considerations/surface-extensions.md` holds two of the hub's four future-considerations categories: `## Tooling deferrals (no loom 1.0 impact)` (category 1) and `## Surface extensions (loom 1.0 leaves a seam)` (category 2). The file name, its H1 (`# Surface extensions`), and the hub's `## Contents` link all name only category 2, so nothing at the navigation surface signals that category-1 content lives in this file. A reader following the hub's explicit four-category framing to locate Tooling deferrals has no filename, H1, or Contents entry pointing here — only the `tooling-deferrals-no-loom-1-0-impact` redirect anchor, which is not a discoverable label. The page is non-normative orientation material, so this is a discoverability/organisation defect rather than a runtime or contract correctness one.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, Thomas Andersen)
**History:** Before f5e89f4 all four future-considerations categories lived in a single `future-considerations.md` whose name correctly covered every section, so no misname existed. f5e89f4 split the over-limit corpus files into a subdirectory+index layout; it grouped category 1 (Tooling deferrals) and category 2 (Surface extensions) into one child page but named that page `surface-extensions.md` after only category 2, while categories 3–4 went to `model-changes-and-non-goals.md`. The category-1 content has been present under a category-2 name in every commit of this file since its creation; the dual-section content has not changed materially since.

## Solution approach

Split the `## Tooling deferrals (no loom 1.0 impact)` section out of `surface-extensions.md` into its own future-considerations child page named for the Tooling-deferrals category, leaving `surface-extensions.md` holding only `## Surface extensions (loom 1.0 leaves a seam)`, so each hub category lives in one file. Relocate the tooling-deferrals redirect anchors and the now-misleading top-of-file stub onto the new page. In `docs/spec_topics/future-considerations.md`, add a `## Contents` entry pointing to the new page and repoint the hub's redirect-anchor stub, then repoint the two cross-references that target the tooling-deferrals anchor (in `host-prerequisites.md` and `patch-skew-degradation.md`).

## Solution constraints

- The `tooling-deferrals-no-loom-1-0-impact` and `tooling-deferrals-no-v1-impact` anchor spellings must relocate together — moving only one breaks a redirect.
