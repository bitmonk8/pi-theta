## Finding

`unknown-reason-rule.md` pins the `"throw:<String(error)>"` value of the `details.failure` discriminator as a byte-exact contract: the *Acceptance criteria* section declares "the exact discriminator string [is] normative", and PIC-48 lists the `"throw:<String(error)>"` template as anchor-stable contract surface. The escape grammar inside that string is spelled out — the truncated `String(error)` payload has `\n`/`\r`/`\t` rendered as the two-character escapes `\\n`/`\\r`/`\\t`, and "every other Unicode code point in U+0000–U+001F … and U+007F–U+009F as the six-character escape `\uXXXX`".

The `\uXXXX` form never pins the case of the four hex digits. For a control code such as U+007F, both `\u007f` and `\u007F` satisfy the rule as written, yet they are different byte sequences. Two conforming implementations therefore produce different `details.failure` strings for the same thrown `error`, defeating the byte-exact claim: any exact-output conformance fixture asserting the escaped payload disagrees across implementations, and operator deduplication on `(code, details.failure)` splits a single logical failure across two buckets.

The same unspecified `\uXXXX` form is restated in two further sites that must stay aligned with the owning rule: acceptance criterion (b) ("every other code point in U+0000–U+001F and in U+007F–U+009F becomes the six-character `\uXXXX` escape"), which is the fixture obligation that would assert the exact escaped output, and the `loom/host/session-shutdown-pinned-constant-unreadable` row in `diagnostics/code-registry-host.md`, which the rule itself states "MUST stay aligned" with the owning page.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 75a9626 — pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type" (2026-05-09, Thomas Andersen)
**History:** The `"throw:<String(error)>"` discriminator and its control-character escape grammar — including the `\uXXXX` six-character escape with no hex-digit-case pin — were first authored in 75a9626 in the then-monolithic `pi-integration-contract.md`; the defect has been present since that grammar was introduced. The 100KB-cap split (f5e89f4) relocated the text verbatim into `unknown-reason-rule.md`, and 8008b91 (2026-06-25) propagated the same unspecified `\uXXXX` form into acceptance criterion (b) when it added the fixture blocks; neither commit introduced the ambiguity, they carried/duplicated it.

## Solution Space

**Shape:** single

### Recommendation

Pin the hex digits of the `\uXXXX` escape to lowercase (`0-9`, `a-f`) at all three sites. In the *Unknown-reason rule* prose, change "the six-character escape `\uXXXX`" to state that `XXXX` are exactly four lowercase-hex digits, zero-padded (e.g. U+007F → `\u007f`); this matches the default output of `codePoint.toString(16)`/`String.fromCharCode`-style construction and of `JSON.stringify`. Restate the same lowercase pin in acceptance criterion (b) so the fixture asserts the byte-exact lowercase output. Update the restated escape grammar in the `loom/host/session-shutdown-pinned-constant-unreadable` row of `diagnostics/code-registry-host.md` to match, per the rule's existing "the two pages MUST stay aligned on the predicate text" obligation.

Edge case for the implementer: U+007F–U+009F and the sub-U+0020 controls all fit in two hex digits, so the upper two of the four positions are always `00`; the rule must still mandate the full four-digit zero-padded form (`\u007f`, not `\u7f`) for the fixed six-character width the grammar already names.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on hex-digit case (`\u007f` vs `\u007F`), producing different `details.failure` bytes for the same control-character-bearing `error`. This breaks the declared byte-exact contract, makes the criterion-(b) conformance fixture non-portable across implementations, and splits operator deduplication buckets for what is one logical failure.
