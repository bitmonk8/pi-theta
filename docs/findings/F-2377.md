## Problem

`V9b` is the single construction site of the model-reference matcher, yet its Spec field (and `V9b-T`'s) cites only `registration-steps.md`, `implementation-notes.md`, and `host-interfaces-core.md` — omitting `binder-model-and-context.md` (`#binder-model-parse-rule`), the page that pins the full matching semantics, including the multi-provider ambiguity rule (a bare `modelId` matching models under more than one provider resolves to **no model**, not pick-first). The cited `host-interfaces-core.md#binder-model-matcher-inputs` names the `Model<Api>` fields the resolver compares but forwards to `#binder-model-parse-rule` for the ambiguity rule. `V9b`/`V9b-T` Tests assert `.id`/`.provider` exact matching but never the ambiguous-bare-`modelId` case. An implementer working only from `V9b`'s declared reading set can therefore ship a green matcher that does pick-first on an ambiguous reference, contradicting the resolution contract `V11a` later presumes already holds.

## Solution approach

Add `binder-model-and-context.md` (anchored at `#binder-model-parse-rule`) to the Spec field of both `V9b` and `V9b-T`, alongside the existing `host-interfaces-core.md#binder-model-matcher-inputs` reference. Add a `V9b`/`V9b-T` Tests vector asserting the constructed matcher resolves an ambiguous bare `modelId` — one matching available models under more than one provider — to no model, exercised at `V9b`'s construction site alongside the existing `.id`/`.provider` exact-match assertions.

## Solution constraints

- The ambiguity verdict must be the matcher object's own behaviour — the single instance both `V6a` and `V11a` bind, consistent with the instance-identity wiring already specified — not a check inlined into either consumer resolution path.
