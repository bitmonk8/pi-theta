## Finding

The **Resolution** paragraph in `docs/spec_topics/invocation.md` is a single long run of normative prose: it states the byte-exact `realpath`-output containment rule, the segment-boundary within-root predicate, the two-channel escape report, the load-time/runtime re-check semantics, and the residual-race acceptance. Appended to the end of that same paragraph — with no paragraph break, callout, or REQ-ID box separating it — is a block opened by `*Non-normative implementation note.*` covering Node `fs.promises.realpath` on-disk-casing behaviour on case-insensitive vs case-sensitive hosts, the symlink-farm/case-variant well-definedness argument, and the `FakeFileSystem` per-host casing mimicry.

The note is explicitly labelled non-normative, so it does not mislead a reader into treating it as a requirement. The defect is purely organisational: a non-normative OS-portability/test-fixture hint is welded to the end of the densest normative paragraph on the page, lengthening the normative run and forcing a reader to parse implementation/test commentary inline with the rules it annotates. Every other non-normative aside on this page (the INV-1, INV-2, INV-3 seam boxes) is set apart in a delimited blockquote rather than run on inside normative prose; this one is the outlier.

The fix is to extract the casing note from the Resolution paragraph into a clearly-delimited non-normative location. The most natural home is the INV-1 box, which already carries its own `*Non-normative implementation note.*` and pins the identical-containment-semantics-across-both-call-sites guarantee the casing note instantiates; `implementation-notes.md` is an acceptable alternative home. Either way the normative Resolution paragraph keeps the byte-exact rule and drops the trailing implementation commentary.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `9aa73b1` ("pi-loom spec: resolve \"Filesystem case-sensitivity for .warp basenames and invoke/tools containment\"")
**History:** `git log -S "on-disk casing of the resolved tail" -- docs/spec_topics/invocation.md` and `git show 9aa73b1 -- docs/spec_topics/invocation.md` show the Resolution paragraph rewritten in that single commit to add both the byte-exact-on-`realpath`-output containment rule and the trailing `*Non-normative implementation note.*` casing block. The defect (non-normative note welded to the end of the normative paragraph rather than placed in a delimited block) entered with that edit; it did not exist in the prior revision, whose Resolution paragraph contained neither the byte-exact comparison rule nor the casing note. The note has not been relocated since.

## Solution Space

**Shape:** single

### Recommendation

Remove the `*Non-normative implementation note.*` casing block (from "On case-insensitive hosts Node's `fs.promises.realpath`…" through "…the test `FakeFileSystem` mimics per-host `realpath` casing accordingly.") from the end of the Resolution paragraph, leaving that paragraph ending at "…not the calling loom's own root." Re-home the extracted text as a delimited non-normative note inside the INV-1 box (`#inv-1`), appended after INV-1's existing `*Non-normative implementation note.*`, since INV-1 already owns the identical-containment-semantics pin the casing note instantiates and already sets its non-normative material apart in a blockquote. Preserve the text verbatim and keep the `*Non-normative implementation note.*` marker. Watch one edge case: the casing note references the `FakeFileSystem` test fixture, so if it is moved to `implementation-notes.md` instead, ensure the relocated note still resolves the byte-exact-on-`realpath`-output rule it annotates (cross-reference `invocation.md` Resolution) rather than reading as free-floating fixture commentary.

## Consequence

**Severity:** cosmetic

The note is already marked non-normative, so no implementer is misled and no behaviour diverges. The only cost is readability: implementation/test-fixture commentary sits inline in the page's densest normative paragraph instead of in a delimited non-normative block like the page's other asides.
