## Finding

The strict-capability probe in `docs/spec_topics/binder/binder-model-and-context.md` (the paragraph anchored by `strict-capability-requirement`) defines the probe as:

> The probe is duck-typed: `(model as { strictCapable?: boolean }).strictCapable` is read as a three-valued check — `true` admits the model with no diagnostic; `false` emits `loom/load/binder-model-not-strict-capable` (E) and refuses to register the loom; `undefined` (the field is absent) emits `loom/load/binder-model-strict-capability-unknown` (W) and registers the loom.

The contract — the three-valued branch on an optional `strictCapable` boolean read off the resolved `Model<Api>` — is wedged into a single TypeScript expression that fuses (i) the behavioural rule with (ii) one particular TypeScript surface syntax (an inline type-assertion expression `as { strictCapable?: boolean }`). An implementer reading the line as normative must reproduce that exact cast form, even though equivalent realisations (a typed accessor helper, a declared local interface widened against `Model<Api>`, an `unknown`-narrowed branch) yield the same observable behaviour. The cast is one TypeScript dialect of "read the optional field"; the spec is silent on which aspects of the expression are normative and which are illustrative.

The other normative sentences in the same paragraph follow this pattern correctly — they name the field (`strictCapable`), name the three branches, and name the diagnostic codes — without pinning the access syntax. Only the cast expression conflates surface form with contract.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the probe sentence so the behavioural rule stands alone and the TypeScript cast — if retained at all — is demoted to a parenthetical reference rendering. Concretely:

- Replace the existing sentence "The probe is duck-typed: `(model as { strictCapable?: boolean }).strictCapable` is read as a three-valued check — …" with a behaviour-only formulation, e.g.: "The probe is duck-typed: the runtime reads the optional `strictCapable` boolean off the resolved `Model<Api>` as a three-valued check — `true` admits the model with no diagnostic; `false` emits `loom/load/binder-model-not-strict-capable` (E) and refuses to register the loom; the field's absence emits `loom/load/binder-model-strict-capability-unknown` (W) and registers the loom."
- If the cast is retained as an audit-greppability hook (the original rationale in `9b9e0cb`), demote it to a single parenthetical non-normative aside attached to the same sentence, marked as such: "(non-normative reference rendering: `(model as { strictCapable?: boolean }).strictCapable`)." Implementers MAY realise the read in any equivalent way.

Edge cases the implementer must preserve verbatim:

- The three-valued branch table (`true` / `false` / absent → admit / E / W) and the two diagnostic codes — already pinned and not affected by this edit.
- The MUST short-circuit clause for the unresolved-model case ("The runtime MUST short-circuit the `strictCapable` probe when binder-model resolution yields no model …") — unchanged.
- The probe-field-name pin ("The probe field name `strictCapable` is the loom-side constant …") — unchanged; this is the actual rename-detection hook.
- The bypass-eligibility carve-out ("Bypass-eligible looms … skip both checks") — unchanged.

The downstream callsites (`code-registry-load.md` rows for the two diagnostics, `version-bump-triggers.md` step 7, `audit-target-categories.md`'s strict-capability-absence-pin, plan leaves `V11a` / `V11a-T`) already reference the probe behaviourally and need no further edit.

## Consequence

**Severity:** advisory

A strict reading of the spec arguably requires the literal cast `(model as { strictCapable?: boolean }).strictCapable` in implementation code, even though a typed helper or a narrowed `unknown` branch is behaviourally identical and may be preferred by the project's TypeScript style. The fix is straightforward and the runtime is not at risk of divergence, but the over-prescription muddies the line between the normative behavioural contract and an illustrative surface rendering — exactly the boundary the surrounding paragraph otherwise keeps clean.
