## Problem

The reload-debounce Tests bullet in both `V10c` and `V10c-T` binds the `Clock` seam (the `FakeClock` test double driving `Clock.setTimeout` / `Clock.clearTimeout`), but that seam is owned by `V8d`, which neither leaf lists in `Deps`. `V10c`'s Deps are `V10c-T, V8b, V9b`; `V10c-T`'s are `V8b, V9b`; `V8b` owns only the `FileSystem` seam, not `Clock`. Under the plan's declared dep-driven build order ("pick the next leaf whose Deps are satisfied"), an orchestrator can schedule `V10c-T` before `V8d` exists, leaving the `Clock` / `FakeClock` references with no declared type to bind against and failing the suite's compile. Correct sequencing currently holds only by slice-numbering coincidence, which the plan explicitly disclaims as authoritative.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `cb8737a` (primary — "pi-loom plan: resolve \"Systemic leaf over-bundling across the plan corpus\"", 2026-06-12); contributing context `2dc65d0` ("pi-loom plan: resolve \"Reload-debounce test does not bind to the injected Clock seam\"", 2026-06-11).
**History:** `2dc65d0` rewrote the V10c/V10c-T reload-debounce Tests bullet to bind explicitly to the `Clock` seam via `FakeClock` (`Clock.setTimeout`/`Clock.clearTimeout`); at that revision the `Clock` seam was still owned by `V8b` (file slug `clock-fs-id-watch-token-seams`), which both leaves already listed in `Deps`, so the dependency was correctly declared. `cb8737a` later split the over-bundled V8 seam leaf, creating `V8d` (the new `Clock`/`IdSource` owner) and `V8e`, and moved `Clock` ownership out of `V8b` — but did not retarget the V10c/V10c-T `Deps` edges from `V8b` to `V8d` (`cb8737a` made no changes to either V10c file). The dangling dependency is the un-propagated residue of that ownership move.

## Solution approach

Add the `V8d` edge to the `Deps` field of both `docs/plan_topics/V10c-settings-merge.md` and `docs/plan_topics/V10c-T-settings-merge.md`, alongside their existing entries.

## Solution constraints

- Keep the existing `V8b` Deps entry on both leaves — `V10c` genuinely consumes the `FileSystem` seam `V8b` owns; this fix adds `V8d`, it does not retarget `V8b`.
- Out of scope: the sibling `V10b`/`V10b-T`, `V11b`/`V11b-T` leaves and the `V8b` file slug — each carries a related pattern resolved by its own finding.
