## Finding

The Indexed-access bullet in `expressions.md` defines the runtime out-of-bounds panic as: "At runtime an array index outside `0..arr.length` panics with `loom/runtime/index-out-of-bounds`." The `0..arr.length` notation is a Rust-style half-open range, but the spec never declares that `..` denotes a half-open `[start, end)` interval, and loom is positioned as a TypeScript subset, not a Rust dialect. A reader without the Rust convention cannot tell whether `arr.length` is itself a valid index (inclusive upper bound) or the first out-of-bounds index (exclusive upper bound).

This boundary is load-bearing: it decides whether `arr[arr.length]` succeeds or panics, i.e. it governs the off-by-one behaviour of every array-index access. The intended semantics — exclusive upper bound, so a valid index is in `[0, arr.length)` and any index that is negative or `≥ arr.length` panics — is the only behaviour consistent with the surrounding spec (`slice` already documents an exclusive `end`, and array length is the element count), but the prose leaves it to the reader to infer rather than stating it. The fix is to replace the bare range notation with an explicit predicate.

## Issue introduction

**Verdict:** single-commit

**History:** The Indexed-access bullet originally stated only the parse-time `loom/parse/non-indexable-receiver` rule and carried no runtime out-of-bounds prose. Commit d6f17ca extended the bullet with the runtime disposition sentence, introducing the bare `0..arr.length` range notation for the panic boundary without declaring the interval's exclusivity; the ambiguity entered the corpus with that single commit.

## Solution Space

**Shape:** single

### Recommendation

Replace the bare range notation in the panic-boundary clause with an explicit predicate that names both ends. Change "an array index outside `0..arr.length`" to "an array index that is negative or `≥ arr.length` (i.e. outside the half-open interval `[0, arr.length)`)". This keeps the intended exclusive-upper-bound semantics, makes `arr[arr.length]` unambiguously a panic, and aligns with the already-exclusive `end` of `slice`. Apply the same wording to any other site that reuses the `0..arr.length` form. Leave the parse-time non-integer-index disposition to its own finding.

## Consequence

**Severity:** correctness

Two reasonable implementers can diverge on whether `arr[arr.length]` is a valid access or a panic: one reading `0..arr.length` as inclusive admits an off-by-one out-of-bounds read; one reading it as exclusive panics. Divergent boundary behaviour at the most common array operation is a correctness hazard, and the `loom/runtime/index-out-of-bounds` conformance test cannot be written without the exact boundary.
