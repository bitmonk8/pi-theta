## Finding

A `while` or `for` body that contains no checkpointed operation (no `@`-query, tool call, or `invoke` — only arithmetic, comparison, field/index access, and `let mut` reassignment) can iterate forever when its exit condition never becomes false. loom 1.0 defines no terminal outcome for this case: it produces no `Err`, no panic, and no diagnostic. The loop simply does not terminate.

The hard-runtime-ceiling set does not bound it either. The ceilings (`invoke`-chain depth, `tool_loop.max_rounds`, load-time JSON depth, AJV-boundary depth) and the four `NOCEIL-1 … NOCEIL-4` non-existence claims are scoped to model rounds, tokens, runtime-value memory, and frame depth. `NOCEIL-1` ("no wall-clock timeout") is explicitly scoped to *query / tool-call / invoke* latency; the wall-clock consumed by a pure-compute loop between checkpoints is not a query/tool/invoke latency, so it is not cleanly carried by `NOCEIL-1`'s text, and no NOCEIL claim, ceiling, or non-goal names compute-iteration count or compute-loop wall-clock as an axis. The four-axis closure's own routing-obligation MUST (`hard-ceilings/ceiling-invariants-and-audit.md`) requires every runtime resource to surface either as a ceiling-list entry or as a documented fall-through to a `NOCEIL-N` claim; compute-loop liveness is the one time-axis trajectory the closure does not explicitly route.

The only escape today is cancellation. The loop-iteration checkpoint (`cancellation.md` — *Granularity*) yields one macrotask turn before reading the signal precisely so a Pi-dispatched abort can land between iterations of a compute-bound loop, so an interactive operator pressing Esc — or a `session_shutdown` teardown — *can* interrupt such a loop. But cancellation is an external, observer-driven event: in a non-interactive context (a subagent-mode `invoke`, a tool-exposed loom, or any autonomous run with no operator at the keyboard) nothing fires the abort, and the loop hangs indefinitely with no loom-defined outcome. The spec authors have conceptualized this trajectory — the ceiling-retirement clause names a "non-terminating absent cancellation" form whose verbal claim is "operator MUST rely on cancellation; no loom 1.0 termination guarantee" — but that form exists only as a retirement bookkeeping shape, never as a present, stated disposition for loop bodies.

## Solution Space

**Shape:** single

### Recommendation

State, as an explicit loom 1.0 disposition, that loom defines no compute-iteration count or compute-loop wall-clock termination guarantee: a loop body whose exit condition never becomes false does not terminate on its own and produces no `Err`, panic, or diagnostic; liveness for such a body is delegated to cancellation (operator Esc or `session_shutdown` teardown), which the loop-iteration checkpoint already makes reachable per `cancellation.md` — *Granularity*.

Land it in two places so the four-axis closure stays closed:

- In `hard-ceilings/ceiling-invariants-and-audit.md`, add a documented time-axis fall-through discharging the routing-obligation MUST: compute-loop wall-clock / iteration count is uncapped and routes to the cancellation-delegated liveness claim (extending the `NOCEIL` closure list with a new claim, or appending to `NOCEIL-1`'s time-axis coverage so that "no wall-clock bound" explicitly spans pure-compute time and not only query/tool/invoke latency). Whichever form is chosen, the entry must name the axis (time), the routing destination (cancellation-delegated liveness; no loom-layer bound), and the one-clause justification, per the per-axis-entry requirement on that page.
- Record the disposition as a loom 1.0 non-goal in `future-considerations/model-changes-and-non-goals.md` ("No compute-loop termination/liveness guarantee"), cross-linking the cancellation *Granularity* mechanism, and add a one-line forward-link from `control-flow.md`'s `while`/`for` prose so a reader at the loop site reaches the disposition.

This aligns with the spec's deliberate posture — `NOCEIL-1` already declares no wall-clock timeout and the four-axis closure intentionally leaves the time axis uncapped — and reuses the existing cancellation reachability rather than introducing a new bound. Edge case the editor must keep correct: the disposition holds only because the loop-iteration checkpoint yields a macrotask each iteration (so cancellation *can* land); a future change that made any loop body synchronous-without-yield would break the "cancellation is the escape" premise and would have to revisit this entry.

## Consequence

**Severity:** advisory

A non-interactive loom whose author writes a compute-only loop with an unreachable exit condition hangs forever with no loom-level outcome, and the spec never tells the implementer whether that is intended (delegate to cancellation) or a gap to be bounded. Two implementers will not diverge on *behaviour* — both will let it hang, since no bound is specified — but the absence leaves the four-axis closure's own routing-obligation MUST unsatisfied for the time axis, and an implementer cannot cite a stated disposition when the case is raised.
