## Finding

`runtime-event-channel.md` pins a dense set of best-effort delivery obligations for the `loom-system-note` channel — the three-step fallback ordering (`ctx.ui.notify` → `loom/runtime/system-note-delivery-failed` diagnostic → terminal `console.error`), the `display: false` / `content: ""` skip of step 1, the payload-sourcing rule (`message` = the original note's `content`, `hint` = the underlying throw's message), the `Clock.wallNow()`-throw rule (when stamping `occurred_at` for a group-A event throws, the `RuntimeEvent` is never constructed and the runtime MUST NOT re-invoke `Clock.wallNow()` — no retry, no sentinel — and the occurrence is accounted for by the step-2 diagnostic), and PIC-54's terminal-`console.error` swallow that MUST hold "regardless of the reach-path from which the chain was entered" (the per-invocation `finally` clean-cancel path, the extension-bootstrap SDK-failure path, and the capability-probe `host-incompatible` path).

Unlike its sibling pages in the same `pi-integration-contract/` family — `diagnostic-emission-isolation.md` and `unknown-reason-rule.md` both close with a normative `## Acceptance criteria` sub-section — `runtime-event-channel.md` has no acceptance-criteria block at all. The obligations are stated only as inline prose. The two hardest-to-verify obligations are exactly the ones a conformance author is most likely to leave untested: the `Clock.wallNow()`-throw branch (a narrow timing/exception path whose observable signature is "no `Clock.wallNow()` re-invocation, no `occurred_at` recovery, occurrence visible through the fallback diagnostic"), and PIC-54's reach-path independence (the swallow must hold from all three entry paths, not just one).

The paired plan leaves (`V7d` / `V7d-T`) already test the ordering, the `display: false` / `content: ""` skip, the throwing-`ctx.ui.notify` proceed, and the PIC-54 swallow "regardless of the reach-path", but neither leaf carries a `Clock.wallNow()`-throw scenario nor enumerates the three PIC-54 reach-paths as independent fixtures. A spec-side acceptance-criteria section would pin those scenarios as conformance obligations so the test leaves inherit them rather than relying on an implementer noticing the gap.

## Solution Space

**Shape:** single

### Recommendation

Add a `## Acceptance criteria` sub-section to `runtime-event-channel.md` (matching the form used by `diagnostic-emission-isolation.md` and `unknown-reason-rule.md`), enumerating one fixture per scenario:

1. **`display: true` happy-path fallback** — `pi.sendMessage` throws; `ctx.ui.notify(content, "error")` is called, then a `loom/runtime/system-note-delivery-failed` diagnostic is emitted with `message` = the original note's `content` and `hint` = the `pi.sendMessage` throw's message; the slash-command handler / spawned subagent session does not abort.
2. **`display: false` / `content: ""` skip** — `pi.sendMessage` throws on a `display: false` (and/or `content: ""`) note; step 1 (`ctx.ui.notify`) is skipped and the chain proceeds straight to the step-2 diagnostic.
3. **`Clock.wallNow()` throw during `occurred_at` stamping** — for a group-A `details: { event }` note, `Clock.wallNow()` throws; the `RuntimeEvent` is never constructed, the runtime does **not** re-invoke `Clock.wallNow()` (no retry, no sentinel substitution), the step-2 diagnostic's `hint` carries the `Clock.wallNow()` throw message, and the occurrence is accounted for by that diagnostic rather than silently dropped.
4. **PIC-54 terminal-`console.error` swallow across all three reach-paths** — a throwing terminal `console.error` is silently swallowed and does not propagate out of the `sendSystemNote` fallback chain, asserted independently for each entry path: the per-invocation `finally` clean-cancel path, the extension-bootstrap SDK-failure path, and the capability-probe `host-incompatible` path (so the factory body, the `session_start` handler, and the per-invocation `finally` never unwind).

Phrase each criterion as an observable property (emission count / payload / which step ran / whether the handler unwound), not as an internal mechanism, so it remains falsifiable by a fixture and consistent with the observable-only stance the channel's other rules take.

## Consequence

**Severity:** advisory

The fallback-chain behaviour is fully specified in prose, so a careful implementer can still produce a conforming system; the gap is in verifiability, not in the requirement itself. But the absence of pinned acceptance criteria for the `Clock.wallNow()`-throw branch and PIC-54's three reach-paths makes it easy to ship an implementation that silently mishandles those rare paths (e.g. re-invokes `Clock.wallNow()` on the fallback, or swallows the terminal `console.error` only on the path that happened to get a fixture) without any red test.
