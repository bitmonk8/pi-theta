## Finding

The manual real-host smoke gate exists to catch a divergence between the in-process session-double and a live Pi host. Its trigger (2) — "any merge whose diff touches the four fidelity-contract axes" — is the standing branch of the gate that guards exactly the change class loom itself can introduce (an edit to the double's modelling, or to the production seam realising any of the four axes). `real-host-smoke-gate.md` states plainly that this is "a **contributor obligation, not a CI-enforced gate**: loom 1.0 has no mechanical hook that observes whether the smoke ran, and the trigger-(2) condition (a diff touching the four fidelity-contract axes) has no automatic diff detector — recognising it is itself part of the contributor's obligation."

Both halves of the gate are therefore unobserved at merge time: nothing detects that a merge falls under trigger (2), and nothing detects that a recorded run is absent when it does. A merge whose author does not recognise their diff touches one of the four axes ships a real-host divergence to `main` with no signal. The document itself concedes the omission is "detectable only **after the fact**, by reading the triggering commit message for the recorded run" — the same record `H6a`'s release-gate audit reads. But the `H6a` audit only fires at the release gate and only reads for a *recorded* run; a merge that silently skipped trigger (2) leaves no record for `H6a` to find, so the divergence is invisible to that audit as well. The gate can be silently skipped for precisely the change class it exists to catch, between any contributor merge and the eventual release-gate read.

## Solution Space

**Shape:** multiple

### Option A — Mechanical merge-time recognition aid

**Approach:** Add a CI check that flags candidate trigger-(2) merges — a diff touching the production seams / double-modelling files realising any of the four fidelity axes — and reddens (or requires an explicit recorded-run / waiver line in the merge commit) when such a diff lands without one; bump commits at a changed Pi-SDK pin carry a recorded-run line for trigger (1).

**Plan edits:** Add the check's definition to `H2a` (cross-cutting gates) or a new horizontal leaf (`<new>`); cite the H7a-owned seam/Deps set the path matcher keys on; reference the check from `real-host-smoke-gate.md`'s trigger-(2) paragraph.

**Spec edits:** None.

**Pros:** Closes the gap mechanically; the gate can no longer be silently skipped for its own change class.

**Cons:** Depends on a CI substrate that no reviewed plan file currently enumerates (which runner, on what events, that a failing gate blocks merge — see the related CI-substrate finding); a path/seam matcher is heuristic and can mis-classify; materially larger surface.

**Risks:** A wrong or incomplete seam-path list re-introduces the silent-skip gap with a false sense of coverage; the merge-block semantic is meaningless until the CI substrate is pinned.

### Option B — Record the residue and add a recognition aid

**Approach:** State inline in `real-host-smoke-gate.md` that trigger-(2) merge-time detection and run-presence enforcement are infeasible at the loom-1.0 pin and are accepted residue (consistent with the manual nature of the gate and trigger (2) being plan-introduced defensive governance), and add a PR-template / contributor-checklist recognition line that prompts the merger to consider whether their diff touches the four fidelity axes.

**Plan edits:** Add the accepted-residue sentence to `real-host-smoke-gate.md`'s trigger-(2) / activation-window prose; add the recognition line to the PR template (or the contributor checklist the project uses).

**Spec edits:** None.

**Pros:** Minimal; honest about the limitation; introduces no dependency on an unenumerated CI substrate; keeps the gate manual by design.

**Cons:** The enforcement gap itself remains — the mitigation is a recognition prompt plus a recorded concession, not detection.

**Risks:** A checklist line is itself skippable; the residue note must name the after-the-fact `H6a` audit as the only backstop so the limitation is not understated.

### Recommendation

Adopt Option B at the loom-1.0 pin: record the trigger-(2) non-enforcement as explicit accepted residue in `real-host-smoke-gate.md` and add the contributor recognition line, because no CI substrate is currently enumerated for the plan and trigger (2) is plan-introduced defensive governance rather than spec-mandated scope. Treat Option A as the follow-on once the CI substrate is pinned. Edge cases the implementer must watch: the residue note must explicitly name the `H6a` after-the-fact commit-message audit as the sole backstop and acknowledge it cannot recover a merge that left no recorded-run line; keep the recognition line keyed to the same four-axes enumeration the gate already defines so the two never drift.

## Consequence

**Severity:** advisory

A loom-introduced double-vs-real divergence on any of the four fidelity-contract axes can reach `main` undetected when the merging contributor does not recognise their diff as trigger-(2) material, and the `H6a` release-gate audit cannot recover it because no recorded-run line was ever written for it to read. The gate's central purpose is defeasible by silent omission; the residue is currently neither mechanically backstopped nor explicitly recorded as accepted.
