## Finding

The **Deps.** bullet in `conventions.md` §Leaf format ends with a trailing sub-paragraph that states how the coverage matrix maps REQ-IDs:

> In the **coverage matrix**, each spec REQ-ID maps to the **implementation** leaf (`<id>`) that closes it (its green tests are the closure evidence); the paired `<id>-T` tests task is not listed separately.

This is a near-verbatim restatement of the rule already carried by `coverage-matrix.md`'s opening paragraph:

> Every executable spec REQ-ID is mapped here to the **implementation** leaf (`<id>`) that closes it (its green tests are the closure evidence; the paired `<id>-T` tests task is not listed separately).

The coverage-matrix authoring rule (a REQ-ID maps to the implementation `<id>`, not its paired `-T` tests task) is therefore stated normatively in two places. `coverage-matrix.md`'s opening is the natural authoritative home for how the matrix itself is authored; the `conventions.md` restatement is a secondary normative surface that can drift out of sync with it under future edits.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** f9c5354 (2026-06-09), c6a664e (2026-06-10)
**History:** The `conventions.md` **Deps.** sub-paragraph carrying the "implementation leaf (`<id>`) … the paired `<id>-T` tests task is not listed separately" rule was introduced on 2026-06-09 in `f9c5354` ("pi-loom plan: add scope section + two-task TDD pairing convention"). At that point `coverage-matrix.md`'s opening paragraph said only "mapped here to the leaf(s) that close it" and carried no such detail, so there was no duplication. The second surface appeared on 2026-06-10 in `c6a664e` ("pi-loom plan: build/update plan for spec.md + review"), which expanded `coverage-matrix.md`'s opening paragraph to restate the same "implementation leaf / `-T` not listed separately" rule. Neither commit alone is the defect; the duplication is the interaction of the conventions sub-paragraph and the later coverage-matrix expansion.

## Solution Space

**Shape:** single

### Recommendation

Strike the duplicated sub-paragraph from the **Deps.** bullet in `docs/plan_topics/conventions.md` §Leaf format. The literal text to remove is:

> In the **coverage matrix**, each spec REQ-ID maps to the **implementation** leaf (`<id>`) that closes it (its green tests are the closure evidence); the paired `<id>-T` tests task is not listed separately.

`coverage-matrix.md`'s opening paragraph remains the single authoritative statement of the rule. The **Deps.** bullet already links to `coverage-matrix.md` for the closing-leaf trigger; if a pointer to the matrix-authoring rule is wanted at this site, replace the struck text with a bare cross-reference to `coverage-matrix.md` rather than any normative restatement.

## Consequence

**Severity:** cosmetic

The same coverage-matrix authoring rule lives on two normative surfaces. If a future edit revises one (e.g. changes how `-T` tasks are listed) without the other, an author populating the matrix gets conflicting guidance. No implementer is currently blocked or would diverge; the two surfaces presently agree.
