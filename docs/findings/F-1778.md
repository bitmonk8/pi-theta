## Finding

The "Tool execution from loom code" surface in `pi-integration-contract/host-interfaces-core.md` (anchored at `#tool-execution-from-loom-code`) — and the runtime bullet in `implementation-notes.md` that points at it — invoke a captured Pi `ToolDefinition.execute(toolCallId, params, signal, onUpdate, ctx)` *directly* for every code-side `<name>(args)` call, bypassing Pi's tool dispatcher. The invocation passes (a) a runtime-minted `toolCallId` carrying the synthetic `loom-direct:` prefix that no Pi-side dispatcher ever registered, (b) a no-op `onUpdate`, and (c) a synthesised `ExtensionContext` (the live host's with `signal`, `sessionManager`, and `abort()` overridden). The spec narrates *what* the runtime does, never *that Pi supports it*: nothing pins, even informally, the behavioural precondition that an arbitrary Pi `ToolDefinition` is safe to drive out-of-band — i.e. that it requires no dispatcher-side per-call setup, tolerates a `toolCallId` Pi never minted (no internal correlation table lookup that would fault), and treats a no-op `onUpdate` as a legal contract for tools whose `execute` body emits progress.

The corpus already inventories a dozen analogous unpinned Pi behaviours as named **presuppositions** on `pi-integration-contract/host-prerequisites.md` (`#register-tool-post-startup-presupposition`, `#registry-persistence-presupposition`, `#messages-chronological-order-presupposition`, `#getcommands-completeness-presupposition`, `#pi-slash-handler-promise-lifecycle-presupposition`, …), each routed to a numbered re-audit item in the *Editorial-review checklist for unpinned host presuppositions* at `pi-integration-contract/version-bump-step2.md`. The out-of-band-`execute()` assumption is the only tool-dispatch-side precondition of comparable scope that has not been pulled into that inventory, despite being the foundation every code-side tool call rests on.

If a future Pi tool implementation correlates progress by `toolCallId` against a dispatcher-side per-call table, requires `onUpdate` to be a real sink, validates that `toolCallId` was minted by Pi, or expects any other dispatcher-side per-call setup, the call would misbehave with no loom-side diagnostic — neither `CodeToolError` (the failure mode would not be `execute()`-throw or `isError: true`) nor any `loom/...` code is owed for the case.

The named-presupposition inventory on `host-prerequisites.md` grew up around the surface (`register-tool-post-startup-presupposition`, `registry-persistence-presupposition`, etc., several added as late as `fa69d8f`), but no commit ever pulled the out-of-band `execute()` precondition into it. The defect is one of category coverage rather than a regression: the surface was authored before the presupposition pattern existed and was never retrofitted when the pattern was generalised.

## Solution Space

**Shape:** single

### Recommendation

Add a named host-behaviour presupposition entry on `pi-integration-contract/host-prerequisites.md` — sibling to `register-tool-post-startup-presupposition` and `registry-persistence-presupposition` — carrying its own anchor (e.g. `#tool-direct-invocation-presupposition`) and stating, in the same shape as those neighbours, that for the loom 1.0 Pi-SDK pin: a captured `ToolDefinition.execute` (a) accepts a `toolCallId` minted by the caller without requiring Pi-side dispatcher registration of that id, (b) treats a no-op `onUpdate` as a legal sink, and (c) requires no other dispatcher-side per-call setup. The entry MUST cite the consumption site (the `#tool-execution-from-loom-code` anchor in `host-interfaces-core.md`) and the loom 1.0 Pi-SDK pin paragraph, in line with the cited siblings.

Cross-reference the new entry from two places: (1) the `host-interfaces-core.md` *Tool execution from loom code* paragraph, immediately under the five-bullet `execute(...)` parameter list, in the same shape as the closing cross-references the paragraph already carries; (2) the `implementation-notes.md` runtime bullet that describes the same call, so the assumption is reachable from both shards. The cross-references replace the current bare-assertion framing.

Add a corresponding lettered item to the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md`, alphabetised into the existing sequence, with a `SHOULD re-confirm` clause directing the bump contributor to re-read the candidate Pi minor's `ToolDefinition.execute` dispatch path (the relevant `dist/core/...` JS body) and verify that the three sub-conditions still hold. Use the same recording shape as the neighbouring items so the new item routes through the existing pass/fail/N/A bump-commit-message convention.

Implementer-relevant edge cases: the new entry must not silently weaken (a) and (b) by stating them only in the positive — phrase them as "Pi MUST tolerate / MUST treat as legal", matching the established prose style — so a Pi minor that introduces *optional* dispatcher hooks does not slip through as conformant by reading the prose as guidance rather than contract. Keep the `IdSource` seam reference (`PIC-20`) intact in the consumption-site paragraph: the `loom-direct:` prefix is what makes the new presupposition implementable (Pi-minted ids by convention do not carry it), and that wiring is already pinned.

## Consequence

**Severity:** correctness

A Pi minor (or any Pi tool an end user adds via `tools:` whose `execute` body depends on a dispatcher-side per-call hook the spec never disclaims) that broke any of the three sub-assumptions — unregistered-`toolCallId` tolerance, no-op-`onUpdate` legality, no dispatcher-side per-call setup — would silently corrupt every code-side tool call without surfacing any of loom's `CodeToolError.cause` values or a `loom/...` diagnostic. Two implementers reading the spec today can disagree on whether wrapping the invocation in a Pi-dispatcher path is required: there is no contract to point to either way.
