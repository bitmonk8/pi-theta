## Finding

The compact-transcript format in `binder-model-and-context.md` is an explicit byte-exact, MUST-reproduce-exactly contract: it pins loom's input-reproducibility guarantee, and the BNDR-7 reference renderings (BNDR-7a–BNDR-7j) exist precisely so a conformance test has a byte-level oracle for each input shape. The rendering function is also required to be **total** over the `AgentMessage` union.

For an `assistant` message, rule 4 says the body is the concatenation of every `TextContent.text` in `content` order and that `ThinkingContent` blocks MUST be omitted. Rule 4 then pins exactly one "thin" sub-case: an assistant message with **no `TextContent` but at least one `ToolCall`** MUST still emit the `[assistant]: ` prefix line (empty body, rule-3 trailing U+0020) followed by the `[tool-call …]` lines, with a matching oracle in BNDR-7f. The adjacent sub-case — an assistant message whose `content` is **only `ThinkingContent`** (no `TextContent`, no `ToolCall`) — is never pinned. Under rule 4 its text concatenation is empty, all thinking blocks are dropped, and there are no tool-call lines, so two reasonable implementers diverge: one follows rule 3 / the totality requirement / the BNDR-7f precedent and emits a bare `[assistant]: ` line (empty body + trailing space); the other observes that a thinking-only turn carries no content the binder grounds against and omits the message entirely. No BNDR-7 rendering covers this case, so there is no byte-exact oracle to adjudicate.

This is a real input shape, not a hypothetical one. `AssistantMessage.content` is typed `(TextContent | ThinkingContent | ToolCall)[]`, and a turn captured mid-thinking — for example an assistant turn aborted during extended thinking before any text or tool call was produced, then committed to the session snapshot the binder reads — can contain thinking blocks only. Because the divergence changes the prompt bytes, and the section itself states that wobble in the transcript can change the sampled output even at `temperature: 0`, the gap directly undermines the input-reproducibility contract these rules exist to guarantee.

The file split at `f5e89f4` only relocated the prose; it did not introduce the gap. The defect has therefore been present since the section's inception.

## Solution Space

**Shape:** single

### Recommendation

Pin the thinking-only assistant message to render the same way as the no-`TextContent`-with-`ToolCall` case minus the tool-call lines: emit a single `[assistant]: ` prefix line with an empty body and the rule-3 trailing U+0020, and nothing else. This is the reading already forced by rule 3 (every message's first line begins with a role tag), the totality requirement on the rendering function, and the BNDR-7f precedent ("role boundary stays detectable; MUST NOT collapse to … no owning role"); making it explicit removes the divergence.

Concretely:

- In rule 4's `assistant` bullet, add a sentence covering the case where `content` has no `TextContent` and no `ToolCall` (only `ThinkingContent`, or — degenerately — empty `content`): the message still emits the `[assistant]: ` prefix line with empty body and the rule-3 trailing space, and no further lines.
- Add a BNDR-7 reference rendering (next free sub-letter, BNDR-7k) with a thinking-only assistant turn and its byte-exact transcript body, so the conformance oracle covers it. Per the BNDR-7 append rule, take the next free sub-letter and do not renumber existing ones.

Edge case for the implementer: the empty-`content` assistant message (no blocks at all) falls under the same clause and renders identically to the thinking-only case — fold both into the one rule so neither is left implicit.

## Consequence

**Severity:** correctness

Two conforming implementations diverge on a real input (thinking-only assistant turn): one emits a bare `[assistant]: ` line, the other emits nothing. The divergent prompt bytes break the byte-identical-input contract the compact-transcript rules exist to guarantee and can shift the binder's sampled output, with no BNDR-7 oracle to detect the divergence in a conformance test.
