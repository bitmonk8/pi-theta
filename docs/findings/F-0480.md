## Finding

`H5a`'s **Adds** enumerates the closing-gate failure conditions, one of which is "a retired/live ID clash". The companion authoritative clause in `conventions.md` §REQ-ID discipline likewise lists "a retired-ID clashing with a live-ID" among the CI-failure conditions. Neither location defines what a *clash* is or names the source of the "retired" set. The adjacent failure condition in the same `conventions.md` sentence — the *per-prefix numbering hole* — is given a full inline definition (a positive integer `n ≤ max(live ∪ retired)` appearing in neither the live nor the retirement table), but the retired/live clash condition beside it gets none.

Two readings are available to an implementer building the gate arm. (1) An ID-level duplication: a single `PREFIX-N` REQ-ID that appears in both a page's live REQ-ID table and that page's `## Retired REQ-IDs` retirement table — a violation of the GOV-8 append-only/immutable/non-reuse invariant. (2) A prefix-level reuse: a prefix appearing in both the live and retired prefix sets, which `conventions.md` already states must be disjoint ("the union of live and retired prefixes covers every prefix … a retired inline-label prefix is never reused"). These produce different gate scans. The retired-ID source is also unpinned at the gate: the implementer must infer that the authoritative retirement registry lives in the spec governance pages (`spec_topics/governance.md` and the `## Retired REQ-IDs` / *Retired prefixes* sub-tables), since neither `H5a` nor the `conventions.md` clause cites it where the clash condition is stated.

The result is that the clash gate arm is not implementable from a single reading: two reasonable implementers would build different reconciliations, and `H5a` ships with no seeded fixture for this arm either, so the divergence is not caught downstream.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md` §REQ-ID discipline — at the sentence that reads "The loom 1.0 closing gate treats an **executable** spec REQ-ID without a coverage-matrix mapping, a retired-ID clashing with a live-ID, or a per-prefix numbering hole as a CI failure." — add an inline definition of the clash condition parallel to the per-prefix-numbering-hole definition that already follows that sentence. The definition must state (a) that a *retired/live ID clash* is a `PREFIX-N` REQ-ID (or GOV-16 inline label) that appears in both a page's live table for that prefix and that page's retirement table (`## Retired REQ-IDs` per GOV-8, or `## Retired inline labels` per GOV-16) — a violation of the GOV-8 append-only/immutable/non-reuse invariant — and (b) that the retired-ID set is sourced from the spec governance retirement registry on `../spec_topics/governance.md` (and the per-prefix `## Retired REQ-IDs` / *Retired prefixes* sub-tables governance owns).

`H5a`'s **Adds** may keep its terse "a retired/live ID clash" wording, since its Convention field already defers to `conventions.md` §REQ-ID discipline; no separate definition is needed in the leaf once the convention clause is defined.

Edge case for the implementer: the clash condition is per-ID duplication across the live and retirement tables, distinct from the per-prefix numbering hole (a number that is *neither* live *nor* retired) — the two arms must not be conflated, and `H5a` currently carries no seeded fixture exercising the clash arm.

## Consequence

**Severity:** correctness

The clash gate arm is one of `H5a`'s enumerated CI-failure conditions; left undefined, two implementers build divergent checks (ID-level duplication vs prefix-level reuse) against an unpinned retired-ID source, and one of them implements a gate arm that does not fire on the condition the plan intends. Because `H5a` carries no seeded fixture for this arm, the divergence is not mechanically caught.
