## Problem

The closing-gate sentence in `conventions.md` §REQ-ID discipline lists "a retired-ID clashing with a live-ID" as a CI-failure condition (mirrored by `H5a`'s **Adds** as "a retired/live ID clash"), but neither location defines what a *clash* is or names the source of the "retired" set. The adjacent failure condition in the same sentence — the per-prefix numbering hole — carries a full inline definition, while the clash condition beside it gets none. Two readings are available: an ID-level duplication (a single `PREFIX-N` appearing in both a page's live table and its retirement table, violating the GOV-8 append-only/immutable/non-reuse invariant) versus a prefix-level reuse (a prefix in both live and retired prefix sets). These produce different gate scans, and `H5a` ships no seeded fixture for this arm, so the divergence is not caught downstream.

The defect is present in each clause's first commit.

## Solution approach

In `conventions.md` §REQ-ID discipline, add an inline definition of the retired/live ID clash condition parallel to the per-prefix-numbering-hole definition that already follows the same closing-gate sentence. The definition must disambiguate the two readings in favour of per-ID duplication across a page's live and retirement tables (the GOV-8 invariant), distinct from the numbering-hole arm, and must pin the source of the retired-ID set to the spec governance retirement registry. `H5a`'s terse **Adds** wording can stand once the convention clause is defined, since its Convention field already defers there.
