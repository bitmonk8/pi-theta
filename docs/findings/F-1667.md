## Finding

The `loom/load/package-read-timeout` row in `docs/spec_topics/diagnostics/code-registry-load.md` pins the message template `package '<name>' package.json read exceeded <deadline>ms during package discovery` and the trigger paragraph says the candidate `package.json` read is abandoned when the per-read deadline fires. By construction the diagnostic emits *because* the read failed, so the package's authored `name` field is unobservable at the emission site.

Neither the row nor the DISC-6 paragraph in `docs/spec_topics/discovery/package-and-settings.md` (which is the spec rule the row points at) names what `<name>` renders in that case. The placeholder-rendering catalogue routes `<name>` through `placeholder-rendering-b.md` §5 (source-derived placeholders), whose rule is "render verbatim from the source as the offending text would appear there" with the identifier-shape predicate from `lexical.md` — neither the source string nor an unavailable-source fallback exists for this site, so the §5 rule leaves a hole rather than dictating an answer. Two conformant implementations can therefore produce non-byte-identical messages (one rendering the on-disk directory entry, another rendering an absolute path, a third rendering `<unknown>` or `<unreadable>`), in violation of the registry's byte-identicality contract for non-§8 placeholders.

The walker already knows a deterministic, source-of-truth identifier for every candidate before it opens `package.json` at all: the immediate-child directory entry name from the candidate-enumeration rule at the top of `docs/spec_topics/discovery/package-and-settings.md` (with `@scope/pkg` joined when the candidate was a scope-directory child). That entry name is the natural and only available identity at timeout time, and pinning it closes the rendering hole without changing the per-read mechanism or the message surround.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `25e8a60` (2026-05-07) "pi-loom spec: resolve 'T08 — Per-package.json read timeout: overrun is unbounded'" — added the per-read-deadline mechanism and the message `package '<name>' package.json read exceeded <deadline>ms during package discovery` on the `loom/load/unreadable-source` row, with `<name>` unsourced for the timeout case.
- `65989e2` (2026-06-01) "pi-loom spec: resolve 'unreadable-source row carries two message formats'" — spun the per-read-timeout sub-trigger off into its own `loom/load/package-read-timeout` row, carrying the `<name>` placeholder forward verbatim without specifying its source-at-timeout.

**History:** `git log -S "package '<name>' package.json read exceeded"` against `docs/spec_topics/diagnostics/code-registry-load.md` and the predecessor `spec_topics/diagnostics.md` shows the message text first appears at `25e8a60` riding the `unreadable-source` row, and `65989e2` lifts that text into the new `package-read-timeout` row. Neither commit's diff adds a sentence on what `<name>` renders when the read fails, and the candidate-enumeration paragraph in `discovery/package-and-settings.md` (which would be the natural site for the rule) was untouched by both. The defect is the interaction: `25e8a60` introduced an interpolation against a field that is by construction unobservable at the emission site, and `65989e2` preserved that interpolation when separating concerns rather than tightening it. The later split through `f5e89f4` (the 100KB-cap-driven file split that moved the row from `diagnostics.md` into `diagnostics/code-registry-load.md`) is mechanical relocation and not an introducing commit.

## Solution Space

**Shape:** single

### Recommendation

Pin `<name>` on the `loom/load/package-read-timeout` row to render the **candidate package's directory-entry name** — the immediate-child directory name the walker enumerated under one of the five `Roots scanned` directories (`.pi/npm/`, `.pi/git/<host>/<path>/`, `node_modules/`, `~/.pi/agent/npm/`, `~/.pi/agent/git/<host>/<path>/`) per `docs/spec_topics/discovery/package-and-settings.md` — joined as `<@scope>/<pkg>` when the candidate was a scope-directory child. This is the package identity the walker already holds at timeout time; no `package.json` parse is required, so the value is always available.

Concretely:

- In the `loom/load/package-read-timeout` row's Trigger cell, add a sentence: *"`<name>` renders the candidate package's directory-entry name from the package-discovery walk (the `node_modules/<pkg>` / `@scope/<pkg>` entry the walker enumerated under one of the five `Roots scanned` directories per `docs/spec_topics/discovery/package-and-settings.md`), not the `name` field of the package's `package.json` (which is by construction unreadable when this row fires)."*
- Mirror a one-sentence statement onto the DISC-6 paragraph in `docs/spec_topics/discovery/package-and-settings.md` at the existing "a single `loom/load/package-read-timeout` warning is emitted naming the package" clause: replace "naming the package" with "naming the candidate package's directory-entry name (per the candidate-enumeration rule above; not the unread `package.json` `name` field)".
- Rendering proceeds through the existing §5 source-derived-placeholder rule on the resulting string — identifier-shaped per `lexical.md` for an unscoped name, double-quoted only when the joined `@scope/pkg` form contains the `/` separator (which is not identifier-shaped, so §5's bare-when-identifier-shaped clause already covers it without a new sub-rule).

Edge cases for the implementer: a same-directory-name collision across roots is not an issue here (a single diagnostic instance is anchored to the one root being scanned, so the entry name is unambiguous in context); a candidate whose directory entry name contains characters outside the slash-name regex still renders verbatim per §5 (the entry name is a host filesystem string, not a loom slash name).

## Consequence

**Severity:** correctness

Two conformant implementations of `V10b` will produce non-byte-identical `loom/load/package-read-timeout` messages for the same timed-out package — one rendering the directory entry, another a full path or a sentinel — and operator-side log greps / fixtures that key on the message will diverge. The registry's byte-identicality contract for non-§8 placeholders is silently violated at this single emitting site.
