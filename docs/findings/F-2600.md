## Finding

The `<error.message>` / `<dispose error first line>` underlying-error coercion in `placeholder-rendering-b.md` ┬¦6 binds a *caught thrown value* `v` (any JS value, since `throw <any value>` is legal) and coerces it to a string by a two-arm rule:

1. when `v` is an object whose `.message` is a string, the underlying string is that `.message`;
2. otherwise the underlying string is `String(v)`, or the literal `<unreadable>` when the `String(v)` coercion itself throws.

The rule explicitly defends arm 2 against a throwing coercion (the `<unreadable>` fallback) and closes with the guarantee that "a non-Error throw never yields `undefined`, a non-string value that defeats the truncation rule, or a synchronous `TypeError`." But arm 1's own predicate ŌĆö "an object whose `.message` is a string" ŌĆö requires *reading* `v.message`, and that read can itself throw when `.message` is a throwing accessor (`Object.defineProperty(v, 'message', { get() { throw ŌĆ” } })`). An implementer evaluating arm 1 literally writes an unguarded `v.message` access; for such a value the predicate evaluation raises synchronously, the coercion never reaches arm 2, and the ┬¦6 guarantee ("never ŌĆ” a synchronous `TypeError`") is violated by the very mechanism meant to uphold it.

The defect is the symmetric twin of arm 2's guarded `String(v)` read: arm 2 anticipates a throwing coercion but arm 1 does not anticipate a throwing property read. Because the coercion is also reused by ┬¦8's `<error>` placeholders (`loom/load/extension-bootstrap-failed`, `loom/runtime/active-set-restore-failed`) and by every producer site that populates an error-message field from a caught throw, an unguarded `.message` read can crash during emission of the diagnostic intended to report the underlying failure.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** bf85092 ŌĆö pi-loom spec: resolve "non-Error-throw coercion + session_start pi.getCommands() failure contract" (2026-06-06, Thomas Andersen)
**History:** The "Underlying-error coercion" block did not exist before bf85092; that commit authored the two-arm rule in its current form, guarding arm 2's `String(v)` coercion with the `<unreadable>` fallback but never guarding arm 1's `.message` property read. The throwing-getter gap was therefore present from the moment the coercion rule was introduced.

## Solution Space

**Shape:** single

### Recommendation

Add a throwing-getter guard to arm 1 of the ┬¦6 coercion so that evaluating the "object whose `.message` is a string" predicate cannot escape synchronously. Restate arm 1 as: when `v` is an object and reading `v.message` *succeeds* and yields a string, the underlying string is that `.message`; otherwise (including when reading `v.message` throws) proceed to arm 2 (`String(v)`, falling back to `<unreadable>` when `String(v)` itself throws). This keeps the stated guarantee ŌĆö the coercion never raises a synchronous error ŌĆö intact across both the property read and the string coercion.

Add a ┬¦6 test vector pinning the behaviour: a caught value whose `.message` is a throwing accessor renders via `String(v)` (or `<unreadable>` when that also throws), not a synchronous `TypeError`. Edge case the implementer must watch: `String(v)` for such a value can independently invoke a throwing `Symbol.toPrimitive`/`toString`/`valueOf`, which the existing arm 2 `<unreadable>` fallback already covers ŌĆö the guard on the `.message` read must not short-circuit that fallback.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one wraps arm 1's `.message` read in a guard and falls through to `String(v)`/`<unreadable>`, the other reads `v.message` directly and throws a synchronous `TypeError` mid-emission ŌĆö exactly the outcome ┬¦6 promises cannot occur. The crash surfaces while rendering the diagnostic that reports the original throw, so the underlying failure is masked by a secondary failure.
