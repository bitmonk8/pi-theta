## Problem

`docs/spec_topics/pi-integration-contract.md` is a 37-line index page (heading, intro paragraph, 30-entry **Contents** list pointing at child files under `pi-integration-contract/`); it carries none of the normative rules its link text names. Throughout the spec corpus, roughly eighty inbound links written as `](../pi-integration-contract.md)`, `](./pi-integration-contract.md)`, or `](pi-integration-contract.md)` (no fragment) resolve to this bare index even though their link text names a specific sub-rule defined in a known child file (e.g. `[PIC's **idempotent** clause]` whose rule lives in `pi-integration-contract/drain-state-contract.md#idempotent-handler-clause`). A reader following such a link lands on the TOC and must guess which of the 30 child files holds the named rule, frequently next to an adjacent link in the same cell that already cites the precise child anchor. The misdirection is silent — the link resolves, the file exists, and `check_links.py` does not flag it — so two implementers can land on different files and build delivery channel, emission timing, or payload shape against the wrong contract.

## Issue introduction

**Verdict:** single-commit

**Introducing commits:** `f5e89f4` (2026-06-04, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*).

**History:** Before `f5e89f4` the spec corpus contained a single 576 KB `docs/spec_topics/pi-integration-contract.md` that held the whole contract in one file; all cross-references targeted that file (with or without a fragment) and resolved to the normative paragraph the link text named. `f5e89f4` split the file into a 37-line index page plus 30 children under `pi-integration-contract/`, preserving every `<a id>` anchor on the child that received the originating paragraph. The split tool (`docs/_tools/split_spec.py`) recomputed link paths via a global `(target_rel, frag) → owner` map, but the map keys on `(file, fragment)` pairs — a bare `(file, None)` link with no fragment falls through to the *"unchanged target file"* arm and is only path-recomputed, never repointed. Every `[…](pi-integration-contract.md)` without a fragment therefore survived the split rewriting the path but still pointing at the (now-emptied) index; the same commit message explicitly notes "2 dangling refs remain" but does not flag the much larger class of bare links that resolve syntactically yet land on a TOC. The defect is the predictable side-effect of one commit's rewrite policy; no later commit has touched these specific links, and the corpus before the split did not have the failure mode.

## Solution approach

Sweep every fragment-less `pi-integration-contract.md` link in `docs/spec_topics/` and repoint each to the precise `pi-integration-contract/<child>.md#<anchor>` target whose normative rule the surrounding link text already names; the corpus supplies the lookup, since an adjacent occurrence in the same shard almost always cites the precise child anchor for the same link text. Discriminate by link text: repoint when it names a specific sub-rule, and leave the link pointing at the index when it names the contract as a whole. Where the destination paragraph has no anchor, add an `<a id>` at that paragraph in the same edit and link to it.

## Solution constraints

- Out of scope: the same defect against the other ten `f5e89f4` split parents (`diagnostics.md`, `errors-and-results.md`, `binder.md`, `query.md`, `governance.md`, `hard-ceilings.md`, `frontmatter.md`, `future-considerations.md`, `discovery.md`, `spec.md`), the `binder.md` bypass-eligibility link owned by F-1529, and any new regression-preventing validator.
- Edit inbound-link targets (and destination anchors where missing) only; do not move rules, alter behavioural prose, or add or remove normative content.
