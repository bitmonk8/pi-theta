## Finding

DISC-6 in `docs/spec_topics/discovery/package-and-settings.md` mandates a specific JavaScript idiom for handling the late settlement of an abandoned per-read promise: *"the read's eventual settlement MUST be silenced with `.catch(() => {})` and its result MUST NOT be re-routed back into the discovery pass"*. The first clause pins one implementation; the second clause is the observable rule (no unhandled rejection, no late re-entry into the walk).

The same DISC-6 paragraph already adopts the looser shape for the deadline-race construct itself — *"the implementation construct used to await whichever resolves first is left to the implementer (e.g. an `AbortController` derived from `Clock.setTimeout`)"* — and the per-read deadline obligations are otherwise stated behaviourally (the timer MUST be scheduled through `Clock.setTimeout`, the package is unreadable for this scan only, the warning fires, the walk continues). The `.catch(() => {})` MUST is the lone implementation-level prescription in an otherwise behavioural rule and conflates two distinct concerns: *what the runtime must guarantee* (no unhandled rejection; result discarded) and *how a TypeScript implementer might achieve it* (one of: a trailing `.catch`, a `try`/`await` around a separately-awaited handle, an `AbortController` whose abort rejection is swallowed in the same race, etc.).

The observable conformance test is on the behaviour (no Node unhandled-rejection event; no later candidate observes the abandoned read's value); it is not on the source-text presence of `.catch(() => {})`.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the per-read-timeout sentence in DISC-6 so the MUST attaches to the observable contract, and demote the `.catch(() => {})` idiom to a non-normative example, matching the same paragraph's existing treatment of the deadline-race construct (*"left to the implementer (e.g. an `AbortController` …)"*).

**Spec edits.**

In `docs/spec_topics/discovery/package-and-settings.md`, DISC-6, the sentence beginning *"On per-read timeout the in-flight read is abandoned …"*, replace the parenthetical

> (the file handle is dropped and GC'd; the read's eventual settlement MUST be silenced with `.catch(() => {})` and its result MUST NOT be re-routed back into the discovery pass)

with

> (the in-flight read is left to settle on its own; its settlement MUST NOT produce a Node unhandled-rejection event and its result MUST NOT be re-routed back into the discovery pass — e.g. by attaching a `.catch(() => {})` to the abandoned read promise, or by consuming its rejection inside the same deadline race)

Implementer-relevant edge cases the fixer must preserve when rewording:

- The two MUSTs (no unhandled rejection; no re-routing) are independent and both observable — the no-re-route MUST is the one that defends against the "late successful read reappears as a registered loom" failure mode and must not be elided when the idiom is dropped.
- Keep the no-cancellation framing: the spec deliberately does not require `fs.readFile` to be cancelled (no portable seam exists), only that its eventual settlement be inert. The "file handle is dropped and GC'd" wording can be dropped (it's a GC/mechanism claim, same category as the §"strong reference"/GC finding) or kept as a non-normative aside; either is fine, but the rewritten sentence must not imply the runtime aborts the underlying read.
- The example list must include both shapes (`.catch(() => {})` on the dropped handle *and* consuming the rejection inside the deadline race) so an implementer using the `AbortController`-based deadline race already sanctioned earlier in the paragraph is not left guessing whether their shape is conformant.

## Consequence

**Severity:** advisory

A literal-minded implementer reads the MUST as a source-level requirement and either pattern-matches `.catch(() => {})` in code review or rejects equivalent shapes (e.g. swallowing the rejection inside the same `Promise.race` await, or using an `AbortController` whose abort-rejection is consumed in the deadline branch) as non-conforming. A reviewer using the existing `no-broad-catch` ESLint rule (per `H2a`) may also flag the prescribed `.catch(() => {})` as a broad catch and force a project-specific carve-out that the spec did not anticipate. The observable contract (no unhandled rejection; result not re-routed) is unaffected either way; this is an implementability/prescription gap, not a correctness one.
