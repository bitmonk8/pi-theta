## Finding

Two normative sentences fix the destination of binder-emitted system notes
to "the user's session" without qualifying the loom's `mode:`. In
`defaulting-system-note-echo.md` the Echo policy says "the runtime appends
a one-line system note to the user's session immediately before the loom
starts", and in `determinism-cancellation-failure.md` Failure modes says
"Binder failures are runtime-handled and surface as system notes in the
user's session". For a `mode: prompt` slash loom the user's session and
the loom's driven session are the same conversation, so the phrasing is
unambiguous. For a `mode: subagent` slash loom they are not: the slash
turn lands in the caller's Pi session while the loom drives a freshly
spawned, private `AgentSession` (per `overview.md` and `slash-invocation.md`).
The binder runs before the loom's queries, may halt with `needs_info` /
`ambiguous` / a failure-mode-table row in which case the spawned
conversation is never created, and emits the success echo "immediately
before the loom starts" — so its destination has to be pinned independently
of the loom's spawned conversation existing.

The cited prose does not pin it. A reasonable implementer can route the
binder note either to the caller's user session (the slash-dispatch
boundary) or, on success, into the spawned subagent's transcript at
spawn time. Both are user-observable: the second is invisible to the user
because subagent transcripts are private (`slash-invocation.md` SLSH-2),
which silently regresses the operator's ability to see *which* arguments
were bound on a subagent invocation and *why* a subagent invocation never
started. The analogous slash-dispatch question for a top-level `Err`
returning from a subagent-mode loom was explicitly settled in SLSH-3 —
"this note is emitted at the slash-dispatch boundary in the user's
session, not in the subagent's private transcript" — establishing the
precedent the binder notes' destination prose needs to match.

## Solution Space

**Shape:** single

### Recommendation

State, in both `defaulting-system-note-echo.md` Echo policy and
`determinism-cancellation-failure.md` Failure modes, that every
binder-emitted system note (the success echo on the Echo policy side; the
seven Failure-mode-templates rows — `needs_info`, `ambiguous`, the
transport/malformed-envelope/AJV-on-`args` rows, the cancelled-binder row,
and the custom-type-unsafe row — on the Failure modes side) is appended
to the **caller's session at the slash-dispatch boundary**, regardless of
the loom's `mode:`. For a slash invocation the caller is the user (per
glossary §caller sense (1)), so this is the user's Pi session in both
modes; the spawned subagent conversation is never the destination. Cite
this alignment as the same rule SLSH-3 already pins for the top-level
`Err` surface so the two slash-dispatch-boundary surfaces are stated as
one contract rather than two ad-hoc decisions.

Implementer notes:

- The `invoke(...)` / `.loom`-callable surfaces are out of scope: per
  `binder.md`'s opening paragraph the binder does not run for those
  callers, so there is no binder note to route.
- Pin that the routing rule fires at the moment the binder produces its
  outcome, not at loom start; on `needs_info` / `ambiguous` / a failure
  row the loom never starts and no subagent session has been created, so
  "the spawned subagent conversation" is not even a candidate
  destination on those arms.
- The `loom-system-note` delivery shape stays as today — `pi.sendMessage`
  with `display: true` against the caller's Pi session, per
  `runtime-event-channel.md` — only the destination-session anchor is
  newly named.

## Consequence

**Severity:** correctness

Two implementers can ship conforming binders that disagree on whether a
subagent-mode loom's bound-argument echo and binder failure notes are
visible to the operator. The likely worst case — routing the echo and
failure notes into the spawned subagent's private transcript on a
`mode: subagent` slash invocation — silently swallows the only
operator-visible signal of which arguments were bound and why the loom
declined to start.
