## Problem

The Echo policy section in `defaulting-system-note-echo.md` and the Failure
modes section in `determinism-cancellation-failure.md` both fix
binder-emitted system notes to "the user's session" without qualifying the
loom's `mode:`. For a `mode: subagent` slash loom the user's (caller's) Pi
session and the loom's spawned private `AgentSession` are distinct, and the
binder runs тАФ and may halt with `needs_info` / `ambiguous` / a failure-mode
row тАФ before that spawned conversation exists, so the destination is
unpinned. A conforming implementer can route the note either to the
caller's session or into the spawned subagent's private transcript; the
latter silently hides which arguments were bound and why a subagent
invocation never started.

## Solution approach

Clarify the destination clause in `defaulting-system-note-echo.md`'s
`#echo-policy` section and in `determinism-cancellation-failure.md`'s
Failure modes / `#failure-mode-templates-normative` rows so every
binder-emitted system note тАФ the success echo and all seven failure-mode
rows тАФ is pinned to the caller's session at the slash-dispatch boundary
regardless of the loom's `mode:`. Forward-link SLSH-3 (`#slsh-3`), which
already pins this destination for the top-level `Err` surface, so the two
slash-dispatch-boundary surfaces read as one contract rather than two
ad-hoc decisions.
