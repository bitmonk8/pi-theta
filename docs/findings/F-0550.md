## Finding

`V11c`'s **Tests** list ends with the bullet "The envelope is a relaxed copy (`maxLength:500` as a model budget, not a user cap); the bypass path skips the LLM call." Unlike the three bullets above it (`BNDR-1`, `BNDR-2`, `BNDR-3`), this bullet cites no REQ-ID and contains no test-obligation verb — it is design rationale, not a test. The leaf format (`conventions.md` §Leaf format) defines **Tests** as "one bullet per REQ-ID … cite the ID inline", with descriptive prose belonging in **Adds**. The stray bullet leaves an implementer following the per-phase TDD ritual ("default to one assertion per REQ-ID") unsure whether it mandates a separate test or is merely commentary.

Both facts the bullet states are already covered elsewhere: the relaxed copy with `maxLength:500` is named in **Adds** ("the relaxed copy with `maxLength:500`") and the cap is part of the envelope schema that `BNDR-1`/`BNDR-2`/`BNDR-3` exercise; the spec confirms the cap "is a budget for the binder model, not a user-visible cap" (`binder-bypass-and-envelope.md` §Binder envelope schema). The bypass-path behaviour is named in **Adds** ("The binder bypass path") and asserted in **Ships when** ("validates the three-arm envelope and the bypass path").

The identical bullet also appears in the paired tests leaf `V11c-T-bypass-envelope.md`; the two **Tests** lists must stay in lockstep, so any fix applies to both.

The defect was present from the leaf's first authoring and survived the later refactor.

## Solution Space

**Shape:** multiple

### Option A — Relocate the rationale to Adds (placement)

**Approach.** Remove the non-REQ-ID bullet from the **Tests** section of both `V11c-bypass-envelope.md` and `V11c-T-bypass-envelope.md`, leaving only the `BNDR-1`/`BNDR-2`/`BNDR-3` bullets. Carry the "model budget, not a user cap" nuance in **Adds** if it is not already implied; the **Adds** sentence already names "the relaxed copy with `maxLength:500`".
**Plan edits.** Strike the bullet "The envelope is a relaxed copy (`maxLength:500` as a model budget, not a user cap); the bypass path skips the LLM call." from both leaves' **Tests** lists.
**Spec edits.** None.
**Pros.** Restores the one-bullet-per-REQ-ID **Tests** invariant; the cap is already exercised by the envelope-schema tests and the bypass path is already in **Adds** + **Ships when**, so no coverage is lost.
**Cons.** The bypass-path behaviour then has no dedicated **Tests** bullet — it rests on **Ships when** alone (a pre-existing condition, not introduced by this fix).
**Risks.** Low; pure relocation of redundant prose.

### Option B — Recast as an explicit bypass-path test obligation (clarity)

**Approach.** Keep a bullet but rewrite it as a real test obligation for the bypass path (e.g. "the single-string / no-params bypass path constructs no envelope and issues no inference call"), anchored to a verifiable identifier.
**Plan edits.** Replace the rationale bullet in both leaves with a verb-bearing test bullet; mirror it in `V11c-T`.
**Spec edits.** None, but note the anchor constraint below.
**Pros.** Adds genuine, observable bypass-path coverage rather than leaning on **Ships when**.
**Cons.** The spec's bypass cases (`binder-bypass-and-envelope.md` §Binder bypass) are prose with no numbered `BNDR-N` REQ-ID; only the `loom/load/bind-echo-without-params` diagnostic is code-keyed. A recast bullet must anchor to that diagnostic code or a Convention, **not** invent a `BNDR` ID (`no-invented-ids`).
**Risks.** Larger surface; risks introducing an unanchored MUST that the H5e scan flags.

### Recommendation

Take Option A: delete the non-REQ-ID rationale bullet from both `V11c-bypass-envelope.md` and `V11c-T-bypass-envelope.md`. The bullet is faithful rationale already covered by **Adds** (relaxed copy / `maxLength:500`) and **Ships when** (bypass path), and the schema cap is exercised by the `BNDR-1`/`BNDR-2`/`BNDR-3` envelope-schema tests. Apply the identical edit to both leaves so the paired **Tests** lists stay in lockstep. Edge case for the implementer: confirm the bypass-path "skips the LLM call" behaviour is observed by some existing test or remains adequately gated by **Ships when**; if genuine bypass-path coverage is wanted, that is Option B and requires a diagnostic-code / Convention anchor, never a fabricated `BNDR` ID.

## Consequence

**Severity:** advisory

An implementer reading the **Tests** list encounters a bullet with no REQ-ID and no test verb; under the "one assertion per REQ-ID" ritual it is ambiguous whether an additional unanchored test is required. The likely outcome is harmless (the cap and bypass facts are already covered), but the format anomaly is a guidance gap that could prompt a spurious extra test or be silently ignored.
