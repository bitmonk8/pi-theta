## Problem

The `coverage-matrix.md` section headed `## Code-keyed obligation areas (no numbered REQ-IDs)` holds two structurally distinct obligation kinds: genuinely code-keyed areas that own their obligations through `loom/...` diagnostic codes, and un-anchored normative MUST/MUST-NOT obligations (the GOV-22 residue, rows `cka-14`…`cka-46`) that carry no `PREFIX-N` REQ-ID and no `loom/...` registry code. By the table's own intro definition an un-anchored MUST is precisely *not* code-keyed, so the leading noun "Code-keyed" mis-describes the larger half of the rows. The defect is a skim-level scope ambiguity at the canonical name every cross-reference uses for the section.

## Issue introduction

**Verdict:** present-since-inception heading + multi-commit accretion
**Introducing commits:** `c6a664e` (2026-06-10) introduced the heading when the table held only genuinely code-keyed rows and the heading was accurate. The mismatch was introduced incrementally by the series of plan-fix commits that appended un-anchored MUST rows under the unchanged heading (`cddd2b4` `cka-14` through `77f2e88` `cka-46`).
**History:** The heading string itself never changed after `c6a664e`. Each later commit added `(un-anchored; GOV-22 residue)` rows to satisfy the `H5e` un-anchored-MUST closing gate, but none revisited the heading, so the leading noun "Code-keyed" silently became inaccurate for the majority of rows.

## Solution approach

In `docs/plan_topics/coverage-matrix.md`, under the `## Code-keyed obligation areas (no numbered REQ-IDs)` heading, state the table's dual scope so it names both obligation kinds — genuinely code-keyed areas and the un-anchored MUST/MUST-NOT GOV-22 residue — before the rows. Keep the heading string itself unchanged.

## Solution constraints

- Do not change the heading string `## Code-keyed obligation areas (no numbered REQ-IDs)` — its exact text is the canonical cross-reference target in `conventions.md`, `H5e`, `H5d`, `H5f`, `H5c`, `H2a`, and within `coverage-matrix.md`.
