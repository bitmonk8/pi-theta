## Finding

The **Renderer registration** section pins every host type it consumes to an exact declaration site so the inline shapes can be re-validated on each Pi minor bump: the preamble states the canonical declarations live at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent`, and that "the `Component` interface lives at `@earendil-works/pi-tui`'s `dist/tui.d.ts`". The reproduced `MessageRenderer` signature, however, names a third type ‚Äî `theme: Theme` ‚Äî that is pinned nowhere. Neither the preamble, the signature block, nor the follow-up paragraph ("The `theme: Theme` parameter is Pi's active TUI theme‚Ä¶") states where `Theme` is declared, so it is silently excluded from the per-bump re-validation set that the rest of the surface relies on.

The omission is also a correctness trap for the obvious guess. `Theme` is **not** a pi-tui export and does not live alongside `Component` in `dist/tui.d.ts`; pi-tui only declares component-scoped theme interfaces (`EditorTheme`, `MarkdownTheme`, `SelectListTheme`, etc.). The `Theme` referenced by `MessageRenderer` is a class declared by pi-coding-agent itself at `dist/modes/interactive/theme/theme.d.ts` (`export declare class Theme`), re-exported from the package root `dist/index.d.ts`, and imported into `dist/core/extensions/types.d.ts` via `../../modes/interactive/theme/theme.ts`. The original framing's natural inference ‚Äî pin it next to `Component` in pi-tui ‚Äî points at the wrong package.

The result is a normative renderer contract (PIC-21 and the styling MUST that requires sourcing colours from `theme`) built against a named host type whose owning file the reader must reverse-engineer, with the only plausible guess being wrong.

The asymmetry has been present since that block's inception; the later page-split commit (78a2f94) carried it verbatim into `extension-bootstrap-and-per-loom.md`.

## Solution Space

**Shape:** single

### Recommendation

Pin `Theme` to its actual owning declaration and add it to the per-bump re-validation set. `Theme` is declared by `@earendil-works/pi-coding-agent` at `dist/modes/interactive/theme/theme.d.ts` (`export declare class Theme`) and re-exported from the package root `dist/index.d.ts`; it is imported into `dist/core/extensions/types.d.ts` from there. Extend the Renderer-registration preamble so the inline-shape re-validation obligation names `Theme`'s declaration site explicitly ‚Äî e.g. add to the sentence that already pins the `types.d.ts` declarations and the pi-tui `Component`: "and the `Theme` class lives at `dist/modes/interactive/theme/theme.d.ts` in `@earendil-works/pi-coding-agent` (re-exported from its package root)". Do not pin `Theme` to pi-tui; pi-tui exposes only component-scoped theme interfaces (`EditorTheme`, `MarkdownTheme`, ‚Ä¶), not the `Theme` class the renderer receives.

## Consequence

**Severity:** advisory

An implementer can still locate `Theme` (it is re-exported from the package root), but the missing pin drops it from the per-bump re-validation obligation and invites the wrong owning-file guess (pi-tui rather than pi-coding-agent). A future Pi rename or move of `Theme` would go undetected at bump time because no re-validation target names it.
