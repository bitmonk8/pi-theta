## Finding

Within the single *Target surface categories* bullet in `docs/spec_topics/pi-integration-contract/audit-target-categories.md` (anchor `audit-target-surface-categories`), the same construct — the closed enumeration of `kind` discriminator strings carried by `SDK_SURFACE_INVENTORY` entries — is named two ways. The bullet first speaks of "the implementation's entry-kind taxonomy" (twice, including the typebox carve-out sentence) and then, in the precondition obligation immediately after the "Satisfying the three in-scope categories" sentence, switches to "the implementation's entry-kind enumeration" without flagging that the two phrases refer to the same artefact.

Every cross-shard reference uses the "taxonomy" wording: `inventory-audit-intro.md` calls it the "entry-kind taxonomy", and the plan leaves `V18a-capability-inventory.md` / `V18a-T-capability-inventory.md` each use "entry-kind taxonomy" three times. The "enumeration" form is unique to this one sentence.

A reader walking the bullet linearly may reasonably suspect two distinct artefacts — a coarse classification (the "taxonomy") versus a closed list of literal kind strings (the "enumeration") — especially given that the bullet does pin distinct minimum-entry-shape obligations and per-kind payload field shapes. Nothing in the prose disambiguates, and no glossary entry anchors either term.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09, "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"")
**History:** The *Inventory-closure audit* and *Target surface categories* prose did not exist in `pi-integration-contract.md` before `e5cd514`; that single commit introduces both wordings together, with three occurrences of "entry-kind taxonomy" (then qualified as "[`h1-scaffold.md`](../plan_topics/h1-scaffold.md)'s entry-kind taxonomy") and one of "entry-kind enumeration" (then "[`h1-scaffold.md`](../plan_topics/h1-scaffold.md)'s entry-kind enumeration") in the same bullet. Subsequent edits preserved the drift: `f5e89f4` (2026-06-04) split the monolithic file into `audit-target-categories.md` / `inventory-audit-intro.md` without touching the wording; `23d28a1` (2026-06-22) realigned cluster basenames; `d5a714a` and `bf92fe6` (both 2026-06-25) edited adjacent text but not these phrases. The defect entered the corpus with the inventory-closure audit itself and has propagated through every cross-shard reference (each of which independently chose "taxonomy") without ever closing the original sentence-level inconsistency.

## Solution Space

**Shape:** single

### Recommendation

In `audit-target-categories.md`, change the single "the implementation's entry-kind enumeration" occurrence to "the implementation's entry-kind taxonomy". The sentence reads "The new kinds MUST be coined in the implementation's entry-kind enumeration as a precondition to the audit's first run on `main`"; rewrite as "The new kinds MUST be coined in the implementation's entry-kind taxonomy as a precondition to the audit's first run on `main`". No other token in the bullet, in `inventory-audit-intro.md`, or in either V18a plan leaf needs to move.

Standardise on "taxonomy" (not "enumeration") because every cross-shard site — the sibling spec file, both plan leaves — already uses it; the spec is realigning to the dominant term rather than to a minority spelling. Implementers consuming the spec for `SDK_SURFACE_INVENTORY` shape obligations then see one phrase everywhere the construct is named.

Edge cases the implementer must watch: do not introduce a `<a id>` anchor for the term (no current cross-reference uses one, and adding one freezes the wording against future renames); do not retitle the construct in code (the implementation's literal identifier is delegated and remains implementation-owned per the same bullet's "PIC delegates the literal kind names … to the implementation" sentence).

## Consequence

**Severity:** advisory

A static-AST implementer who treats normative wording strictly may waste a cycle deciding whether the bullet's "minimum entry shape every kind MUST satisfy" obligations attach to the same artefact as the "precondition" entry-kind list, or to a sibling artefact. Two implementers would still converge on the same enum-of-strings construct, since the surrounding context (`kind` discriminator field, `namespace-function` examples, branch-table arms) anchors both terms to the same surface — the drift is irritating rather than divergence-producing.
