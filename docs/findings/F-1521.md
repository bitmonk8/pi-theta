## Problem

The binder inference call forces the provider's tool choice to the single structured-output tool, so a successful response resolves with `stopReason: "tool_use"` and a matching `ToolCall`. The "structured-output result is extracted…" paragraph in [Binder inference call](#binder-inference-call) then states two overlapping rules with no precedence on the same `AssistantMessage`: a success-extraction rule (matching `ToolCall` supplies the envelope) and a failure-classification rule triggered by "a `stopReason` other than `\"stop\"`". A forced-tool success satisfies both at once, because `"tool_use" ≠ "stop"`. The classifier that the failure rule routes into explicitly excludes `tool_use` from its domain, so the value falls through to the catch-all `TransportError`, the retry budget exhausts after three identical successes, and the loom never starts.

## Issue introduction

**Verdict:** present-since-inception (with a downstream commit sharpening the contradiction)

**Introducing commits:**
- `fb3474f` (2026-06-04, *pi-loom spec: resolve "Binder inference call — no pi-ai entry point pinned"*) — first introduced both the forced-tool mechanism (`options.toolChoice` to the single structured-output tool) and the *"A `stopReason` other than `"stop"`"* trigger in the same paragraph, so the contradiction has been latent in this section since it was first written.
- `ef4cfb8` (2026-06-07, *pi-loom spec: resolve "Stop-reason → QueryError variant mapping is undefined"*) — added the `Stop-reason classification` block in `provider-error-mapping.md`, whose domain sentence ("other than the `end_turn` / `stop` / `tool_use` reasons the tool loop treats as a normal turn boundary") explicitly excludes `tool_use`. This commit did not introduce the defect; it sharpened it by making the binder-inference trigger refer into a classifier whose domain provably excludes the value the trigger feeds in.

**History:** `git log --follow` over the `binder-inference.md` paragraph and `git log -L` over the `Stop-reason classification` block in `provider-error-mapping.md`. The structured-output extraction paragraph has only been touched by `fb3474f`, the size-cap split `f5e89f4`, and `d0bcdb2` / `c0de1a7` (which adjusted retry timing and `onResponse` join wording but did not revise the success/failure trigger). No commit has revisited the `stopReason: "tool_use"` overlap.

## Solution approach

Rewrite the "structured-output result is extracted…" paragraph so that success extraction has explicit precedence over `stopReason`-based failure classification on the same `AssistantMessage`: a matching `ToolCall` extracts the envelope (handing off to the post-extraction envelope/AJV pipeline) regardless of `stopReason`, `errorMessage`, or HTTP status, and the failure routing applies only when no matching `ToolCall` is present. Narrow the failure trigger from "a `stopReason` other than `\"stop\"`" to also exclude `"tool_use"`, keeping the remaining triggers (non-empty `errorMessage`, rejected promise, 4xx/5xx/network failure) unchanged. Make explicit that a matching `ToolCall` whose `arguments` then fail parse or schema validation is the malformed-envelope class at [Failure-class taxonomy](../binder/determinism-cancellation-failure.md#failure-class-taxonomy), not a transport failure, and that a degenerate response carrying both a matching `ToolCall` and a non-empty `errorMessage` extracts as success.

## Solution constraints

- Out of scope: the `Stop-reason classification` block in `provider-error-mapping.md` — its domain sentence already correctly excludes `tool_use`; the fix is entirely on the binder-inference side.
