## Finding

`docs/spec_topics/pi-integration-contract/binder-inference.md` forces the provider's tool choice to the binder's structured-output tool (`options.toolChoice` pinned at the single `__loom_bind_<slug>` entry). A successful provider response under that forced-tool regime resolves with `stopReason: "tool_use"` and an `AssistantMessage.content` carrying a matching `ToolCall` — that is the well-formed success case.

The paragraph that extracts the result then states two overlapping rules on the same `AssistantMessage`, with no precedence: (a) "the first `ToolCall` whose `name` matches the binder's structured-output tool supplies the envelope JSON in its `arguments`" — which a successful response satisfies — and immediately after, (b) "A `stopReason` other than `\"stop\"`, a non-empty `errorMessage`, a rejected promise, or any 4xx/5xx/network failure is classified through the [Provider error mapping] table" — which the same successful response also satisfies, because `"tool_use" ≠ "stop"`. Read literally, every successful forced-tool binder response triggers both rules at once and the page is self-contradictory.

The classifier the second rule routes into reinforces, rather than resolves, the contradiction. The `Stop-reason classification` block in `provider-error-mapping.md` explicitly scopes itself to stop reasons "other than the `end_turn` / `stop` / `tool_use` reasons the tool loop treats as a normal turn boundary": `tool_use` is *not* in its domain. A `stopReason: "tool_use"` therefore matches no table row, no overflow signature, and no stop-reason arm, and falls through to the page's general catch-all (`TransportError`). That `TransportError` is then fed back through `Failure-class taxonomy`'s transport-class budget, the binder issues another call, the next success classifies identically, the per-invocation budget exhausts after 3 calls (HC3-d), and the `Binder model transport failure` note surfaces — for a sequence of three textbook-correct binder responses. The loom never starts.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the *"The structured-output result is extracted…"* paragraph so that success extraction has explicit precedence over `stopReason`-based failure classification, on the same `AssistantMessage`:

1. **First**, run the success-extraction rule: if `AssistantMessage.content` contains a `ToolCall` whose `name` matches the binder's structured-output tool, that call's `arguments` is the envelope JSON and the attempt's outcome is the post-extraction envelope/AJV pipeline (`Failure-class taxonomy`'s malformed-envelope / AJV-on-`args` / `kind` arms). This rule fires regardless of `stopReason`, regardless of `errorMessage` non-emptiness, and regardless of HTTP status — a matching `ToolCall` is by construction proof that the provider produced a well-formed forced-tool response, which is the success case the forced-tool mechanism is designed to produce.
2. **Otherwise** — and only when no matching `ToolCall` is present — apply the existing failure routing. Narrow the trigger from "a `stopReason` other than `\"stop\"`" to "a `stopReason` other than `\"stop\"` or `\"tool_use\"`" so it cannot fire on a forced-tool success even hypothetically, and keep the remaining triggers (non-empty `errorMessage`, rejected promise, 4xx/5xx/network failure) unchanged. An `AssistantMessage` that carries no matching `ToolCall` and matches none of the failure triggers (e.g. plain text with `stopReason: "stop"` and empty `errorMessage`) remains the malformed-envelope condition already pinned at `Failure-class taxonomy`.

Edge cases the rewrite must cover explicitly:

- A `ToolCall` with the matching `name` whose `arguments` then fail JSON parse or envelope-schema validation is **not** a transport failure: success extraction succeeded; the malformed-envelope class in `Failure-class taxonomy` owns it.
- A response with multiple `ToolCall`s where one matches: the existing "first" wording is fine, but pin that "first" is in `AssistantMessage.content` order, so two conforming implementations agree.
- A response carrying both a matching `ToolCall` *and* a non-empty `errorMessage` (degenerate but possible): the success-precedence rule extracts the envelope and the `errorMessage` is ignored — pi-ai surfaces `errorMessage` for error envelopes, and a forced-tool success is not one. Make this explicit so an implementer does not re-introduce the same overlap on a different field.

Do not also try to "fix" the `Stop-reason classification` block — its domain sentence already excludes `tool_use` and is correct as written; the defect is entirely on the binder-inference side, which sends the classifier values outside the classifier's domain.

## Consequence

**Severity:** correctness

Taken literally, a textbook-correct forced-tool binder response is both extracted as success *and* classified as `TransportError`, with no rule deciding which wins. A charitable implementer guesses success-wins; a literal implementer (or one prioritising the second sentence) routes every successful binding through the retry budget, exhausts it after three identical successes, and surfaces `argument binder unavailable` — the loom never runs. Two reasonable implementers diverge on whether the binder works at all under the forced-tool path the same paragraph mandates.
