## Problem

`SUBS-1` is a live, numbered, executable REQ-ID anchored in `schema-subset.md` §Lowering Algorithm step 3 as a normative union-lowering MUST, but it has no row in the **Numbered REQ-IDs (runtime obligations)** table of `coverage-matrix.md`. The only SUBS-related entry, `cka-10`, sits under **Code-keyed obligation areas (no numbered REQ-IDs)** and by that heading cannot satisfy the closure obligation for a numbered REQ-ID. No V5 leaf's Tests bullet cites `SUBS-1` inline. As a result the `H5a` unmapped-executable-REQ-ID gate arm reddens at the `H6a` release-gate activation, and the union-lowering behaviour itself has no citing-test obligation.

## Issue introduction

**Verdict:** single-commit introduction (spec mint not mirrored in the plan coverage matrix)
**Introducing commit:** `6892aa3` (2026-06-20) — "pi-loom spec-plan: fix F-0786 — pin primitive-union/nullability lowered form (SUBS-1) for canonical-hash byte-identity"
**History:** `SUBS-1` first appeared as a numbered REQ-ID in `6892aa3`, which edited only `docs/spec_topics/schema-subset.md` (and removed one line in `docs/spec-plan-review.md`); it touched neither `coverage-matrix.md` nor any V5 leaf. `git log -S "SUBS-1" -- docs/plan_topics/coverage-matrix.md` is empty (the Numbered REQ-IDs table has never carried a SUBS row), and `git log -S "SUBS-1"` over the V5d/V5f/V5f-T leaves is empty (no leaf has ever cited it). The obligation has therefore been an unmapped executable REQ-ID since the moment it was minted: the spec edit pinned the lowered form without the matching plan-side coverage-matrix row or citing test.

## Solution approach

Map `SUBS-1` to closing leaf `V5f` (the leaf owning the lowering/emit-per-type-form pass). Add a row to the **Numbered REQ-IDs (runtime obligations)** table in `coverage-matrix.md` mapping `SUBS-1` to `V5f`. Add a Tests bullet to the `**Tests.**` line of both `V5f-subset-lowering-hash.md` and its paired `V5f-T-subset-lowering-hash.md` that cites `SUBS-1` inline and asserts the lowering split (primitive-only union → `{ "type": [...] }`, any non-primitive arm → `{ "anyOf": [...] }`, source-order arms with `"null"` last).

## Solution constraints

- The spec is read-only for this fix; `schema-subset.md` already defines `SUBS-1` correctly — do not edit it.
- Out of scope: the existing `cka-10` row, which continues to close `schema-subset.md`'s code-keyed diagnostics — leave it unchanged; the new `SUBS-1` numbered row sits alongside it.
