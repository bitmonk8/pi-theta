## Finding

The binder's input-reproducibility contract (`binder-model-and-context.md` — *Why the transcript bytes are pinned*) asserts that "the same session always constructs the same prompt bytes … on every run and across processes." The compact-transcript renderer and the session-context truncation walk both source their material from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`, and the rendered bytes are the verbatim per-variant field content of those messages — `UserMessage` text, the `AssistantMessage.content` `TextContent.text` and each `ToolCall` `name`/`arguments`, the `ToolResultMessage` content blocks, and `CustomMessage` content. For the byte-identical-input claim to hold, an unchanged logical session must yield content-identical message bodies on every read.

Loom already pins, as explicit best-effort host presuppositions in `pi-integration-contract/host-prerequisites.md`, four of the facets that contract rests on: chronological ordering (`#messages-chronological-order-presupposition`), the leading-`user`-message guarantee (`#messages-leading-user-message-presupposition`), production `TokenEstimator` per-message determinism (`#token-estimator-determinism-presupposition`), and the triggering-slash-turn commit posture (`#triggering-slash-turn-commit-posture-presupposition`). It does **not** pin the content-stability facet: nothing states that, for an unchanged session, the message bodies the read returns are themselves stable across reads, runs, and processes. A Pi minor that rewrote or compacted historical session entries (history summarisation/compaction is a real host concern) would alter the rendered transcript bytes and silently break the input-reproducibility contract — with no SDK surface-inventory signal and, unlike its four siblings, no version-bump editorial-review item to catch the regression.

The original finding also raised the fixed 8000-token slice plus system prompt plus envelope schema potentially exceeding a cheap binder model's context window. That concern is already discharged: `determinism-cancellation-failure.md` — *Failure-class taxonomy* — *Context-overflow handling* classifies a `ContextOverflowError` as transport-class, so window insufficiency self-heals on retry or surfaces as the *Binder model transport failure* system note. No load-time context-window gate is owed, and no edit is required for that half; the actionable gap is solely the missing content-stability precondition.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `3cd40c5` (2026-06-03, "Compact-transcript byte-exact rendering — state the rationale") and `8f7266c` (2026-06-03, "Binder Determinism reproducibility scope overreaches provider contract") — the commits that scoped the byte-identical / input-reproducibility contract to loom's input bytes.
**History:** The input-reproducibility contract has rested on an unstated session-read content-stability premise since it was first scoped. The contract's other supporting host presuppositions were pinned incrementally by later commits — `.messages` ordering (`05e84dd`, 2026-06-06), the leading-`user`-message guarantee (`ce4e476`, 2026-06-06), production `TokenEstimator` per-message determinism (`341bcd9`, 2026-06-27), and the triggering-slash-turn commit posture (`d3a1687`, 2026-06-28) — but none enumerated the content-stability facet. `git log -S "content-stability"` and `-S "content-identical"` over `docs/` return no commit, confirming the precondition was never authored. The gap is therefore a latent omission dating to the contract's authoring rather than a regression introduced by any single later edit.

## Solution Space

**Shape:** single

### Recommendation

Add a content-stability host presupposition to `host-prerequisites.md`, as a sibling to the four `.messages` presuppositions already there, framed identically (best-effort against the loom 1.0 Pi-SDK pin, re-audited per minor bump): for an unchanged logical session, `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages` returns content-identical message bodies — specifically the per-variant fields the compact-transcript renderer reads (`UserMessage` text; each `AssistantMessage.content` `TextContent.text` and each `ToolCall` `name`/`arguments`; `ToolResultMessage` content blocks; `CustomMessage` content) — across reads, runs, and processes, so a re-read of the same session yields byte-identical transcript material. Note the read pair `getEntries()`/`getLeafId()` and the `buildSessionContext` derivation by name, and identify the failure mode a Pi minor would introduce (history compaction or entry rewrite that mutates already-committed message bodies).

In the same commit:
- Update `binder-model-and-context.md` — *Why the transcript bytes are pinned* — to cite the new precondition in its facet list, alongside the ordering, leading-`user`, per-message-determinism, and triggering-turn-commit references already there.
- Add a SHOULD-level editorial-review item to the version-bump checklist (`version-bump-intro.md` and the consuming step), mirroring the token-determinism `(aq)` and triggering-turn items, so a Pi minor that mutates committed session content is caught by contributor review.

Do not add a load-time context-window gate: window insufficiency is already an accepted self-healing transport-class failure per `determinism-cancellation-failure.md` — *Context-overflow handling*, and that clause needs no change.

## Consequence

**Severity:** advisory

The byte-identical-binder-input claim rests on an unstated premise that session reads are content-stable. Implementers can still ship a working binder (input-reproducibility is best-effort and the determinism guarantee is loom-side seed/temperature only), but the contract's audit surface is incomplete: a future Pi history-compaction or entry-rewrite change would silently invalidate the reproducibility/prompt-cache claim with no editorial-review hook, whereas its four sibling facets each carry one.
