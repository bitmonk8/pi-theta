## Finding

`V9p` closes the three non-abort extension-bootstrap surfaces. Its first surface — a factory-time `pi.registerMessageRenderer` rejection — is described as "this extension instance's system notes **permanently** degrade to the `ctx.ui.notify` arm of the System-notes fallback chain", with no statement that the degrade is escapable. The two sibling non-abort surfaces in the same leaf and in the backing spec page (`pi.registerCommand` failure and the `session_start`-time `pi.getCommands()` read failure) both explicitly carry "a `/reload` is the recovery path"; the renderer bullet does not.

The recovery semantics do exist in the spec corpus — the `loom/load/extension-bootstrap-failed` diagnostics-table entry states "A `/reload` is the recovery path for a transient failure", and the renderer **Teardown** lifecycle section states that on `ctx.reload()` Pi discards the prior renderer `Map` and the freshly loaded factory registers afresh. So the degrade is genuinely per-extension-instance and is cleared on reload, with no special re-attempt logic to implement. The gap is that `V9p` (and the renderer bootstrap-failure bullet it closes) never surfaces this, so the word "permanently" reads as unrecoverable. A reader or auditor of `V9p` can mistake the degrade for a terminal, unrecoverable state and either over-engineer an in-instance re-attempt mechanism or mis-scope the test for the degrade's lifetime.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commits:** `2d2fc8e` (2026-05-06, "pi-loom spec: resolve 'Pi SDK capability calls have no failure contract'") authored the Extension-bootstrap SDK-failure protocol. In that same commit the `pi.registerCommand` bullet and the `loom/load/extension-bootstrap-failed` diagnostics-table entry received the `/reload`-recovery wording, while the `pi.registerMessageRenderer` bullet was written as "permanently degrade … and proceeds" with no inline recovery note — the asymmetry was present from the first authoring of the contract. `5a94e2e` (2026-06-17, "resolve 'V9k bundles five independent bootstrap-failure surfaces in one leaf'") split `V9p` out of `V9k`, inheriting the renderer bullet's recovery-silence verbatim.

**History:** The `pi.getCommands()` read-failure bullet later (`bf85092`, 2026-06-06) also adopted the explicit `/reload`-recovery wording, widening the gap — by the time `V9p` was created, two of the three non-abort surfaces carried the inline recovery note and the renderer surface did not. The defect therefore traces to the contract's inception, not to a later edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9p-extension-bootstrap-nonabort.md`, surface the renderer degrade's recovery path on the leaf:

- In the **Adds** paragraph, where the renderer surface is described as "renderer-failure degrade-and-proceed", state that the degrade is scoped to the current extension instance and is cleared on `/reload` (the freshly loaded factory re-registers the renderer afresh against a Pi state with no prior `loom-system-note` registration), cross-referencing the renderer *Teardown* lifecycle semantics in `extension-bootstrap-and-per-loom.md`. There is no in-instance re-attempt to implement — recovery is the standard factory re-run.
- In the first **Tests** bullet (the `registerMessageRenderer` rejection case), keep the "permanently degrade … for this extension instance" assertion but make the per-instance scoping explicit so the test does not assert an unrecoverable / cross-reload-persistent degrade.

The spec already documents the recovery path (diagnostics-table entry plus the *Teardown* section), so no spec edit is required for correctness. Optionally, for inline parity with the sibling `pi.registerCommand` / `pi.getCommands()` bullets, the spec's `pi.registerMessageRenderer` bullet may gain a trailing "a `/reload` is the recovery path" clause — but this is a separate, optional spec-consistency improvement, not part of closing this finding.

## Consequence

**Severity:** advisory

The renderer degrade is recoverable (per-instance, cleared on `/reload`) and requires no special code, but `V9p`'s "permanently degrade" wording does not say so. An implementer or auditor reading `V9p` in isolation can read the degrade as terminal — either adding an unnecessary in-instance re-attempt mechanism or mis-scoping the degrade-lifetime assertion in the renderer-failure test.
