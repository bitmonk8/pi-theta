## Finding

`capability-probe.md` declares `PIC-5` as a normative `MUST NOT` — *"Probes MUST NOT be added beyond the five enumerated checks"* — but pins no observable through which a violation can be caught. The five "enumerated checks" are themselves only described in prose: the body counts step (a) Node floor, (b) `AbortSignal` shape, (c) factory-probable SDK capabilities, (d) peer-dep version, and (e) `typebox` host-shape, while a later sentence simultaneously says (d) "is part of step (c)'s pinned-constants execution group and shares its short-circuit slot", which already leaves the cardinality interpretively soft (is it five sub-steps or four execution groups?). There is no closed array of group identifiers in the pinned-constants block, no build-time count assertion, and no fixture that a sixth `loom/load/host-incompatible` route would redden.

The plan leaf `V9a-T` lists a test bullet *"`PIC-5`: there are exactly five checks; no probes beyond them"*, but this presupposes a mechanical operand the spec does not provide: the assertion has nothing to count `.length === 5` against, and the *no probes beyond them* clause has no surface to inspect. A contributor inserting a sixth check inside one of the existing step bodies — for example a second `typeof`-probed member under step (c) that routes to the already-enumerated `sdk-capability-missing` `kind`, or an additional sub-probe under step (b) that routes to `abortsignal-shape` — would pass every existing test: the closed-`kind` discriminator set is unchanged, the per-step short-circuit order is unchanged, and no new diagnostic surfaces. PIC-5 reduces to contributor discipline.

The asymmetry with the seven-capability count is sharp: `CAPABILITY_OBLIGATIONS.length === 7` (per `version-bump-step2.md`) is the explicit mechanical gate that locks the capability inventory's cardinality; PIC-5 has no analogue.

## Solution Space

**Shape:** single

### Recommendation

Add a closed pinned constant `STEP0_CHECK_GROUPS` to the same source-of-truth pinned-constants block that already holds the Node floor, `AbortSignal` member-with-kind list, factory-probable capability list, and peer-dep range. Make it a literal five-element array whose entries are the step-group identifiers in the spec's fixed normative short-circuit order — `["node-floor", "abortsignal-shape", "sdk-capabilities-and-peer-dep", "typebox-shape"]` collapses (c)+(d), so prefer the five-element form keyed on the five sub-steps: `["node-floor", "abortsignal-shape", "sdk-capability-missing", "peer-dep-version", "typebox-shape"]`. Pin two mechanical assertions against it:

1. `STEP0_CHECK_GROUPS.length === 5` (the count gate, mirroring `CAPABILITY_OBLIGATIONS.length === 7`).
2. A closed-membership equality against a literal whose contents and order match the spec's normative `(a)→(b)→(c)+(d)→(e)` short-circuit sequence (so a re-ordering is also caught, on the same footing as the `CAPABILITY_OBLIGATIONS` member-order obligation in `version-bump-step2.md`).

Make the probe's top-level dispatch iterate `STEP0_CHECK_GROUPS` directly rather than running five hand-written sequential blocks, so adding a sixth check is impossible without extending the array — at which point the count assertion fires. Independently, add a partition-style build-time assertion (homed in `V18a`) that each `host-incompatible` `details.kind` discriminator the runtime can emit (`node-floor`, `abortsignal-shape`, `sdk-capability-missing`, `peer-dep-out-of-range`, `peer-dep-malformed-version`, `typebox-shape`, `probe-failed`) maps back to exactly one `STEP0_CHECK_GROUPS` entry (with `peer-dep-*` and `sdk-capability-missing` mapping per the (c)+(d) execution-group framing already in the spec). This closes the "sixth check inserted under an existing step routing to an enumerated `kind`" loophole the finding names: any new probe that does not extend `STEP0_CHECK_GROUPS` either has no `kind` to route to (caught by the closed enumeration) or shares a `kind` with an existing group (caught by the one-to-one partition).

Update `version-bump-triggers.md` step 5 grouping (i) to enumerate `STEP0_CHECK_GROUPS` alongside the existing four Step-0-probe constants so Pi minor bumps that legitimately add or remove a probe group land the constant edit on the same footing. Edge cases the implementer must handle: the (c)+(d) execution-group collapse (the spec already says (d) "shares (c)'s short-circuit slot", so the closed-membership literal must match whichever framing the probe iterates — pick the five-sub-step framing for symmetry with the existing prose); the `probe-failed` route (it is not one of the five groups but a self-failure route per any group, so the partition assertion's domain is the six comparison-verdict `kind`s, not seven); and the `details.member` / `details.package` sibling fields (these refine existing `kind`s and do not extend the partition).

## Consequence

**Severity:** advisory

The rule is unenforced: nothing in the build catches a contributor who slips an additional probe into an existing step body, and nothing catches a sixth top-level group either. The probe still works in the field — the inserted check would simply run and route as today's existing `kind`s do — so the gap does not corrupt runtime behaviour. What erodes is the architectural invariant PIC-5 exists to protect (a small, fixed probe surface), the symmetry with `CAPABILITY_OBLIGATIONS.length === 7`, and contributor-side review confidence on Pi version bumps.
