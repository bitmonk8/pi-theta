## Finding

The `tools` field prose in `frontmatter-fields-a.md#tools` enumerates "Two kinds of entry". The `.loom`-path entry is fully self-contained: it names its resolution rules and every failure diagnostic inline (`loom/load/unresolvable-loom-path`, `loom/load/prompt-mode-callable`, `loom/load/invalid-tool-rename`, `loom/load/tool-name-collision`). The Pi-tool-name entry, by contrast, reads in full: *"**Pi tool names** (`read`, `bash`, `grep`, ...) resolve against Pi's tool registry at loom-load time, exactly as for Pi subagents."* It pins neither the resolution predicate nor the failure path at the site that introduces the rule.

Two things are left to an external system with no anchor. First, the resolution predicate itself — how a `tools:` entry string is matched against Pi's tool registry (exact byte-equal lookup, case-sensitivity, whether the registered name is taken verbatim) — is delegated wholesale to "exactly as for Pi subagents" with no cross-reference into the Pi Integration Contract or any loom-owned rule. An implementer cannot derive the match semantics from the loom spec alone. Second, the unrecognised-name failure is not surfaced here at all: the `loom/load/unknown-tool` disposition exists, but it lives only in the sibling page `frontmatter-fields-b-and-templates.md` (the consolidated "Unknown Pi tool names (`loom/load/unknown-tool`) … prevent the loom from being registered" paragraph) and the diagnostics registry — neither of which is linked from the Pi-tool-name entry. The asymmetry with the `.loom` branch, which names its own failure modes inline, means a reader of the Pi-tool-name entry sees an external-deference claim with no pointer to the predicate or the diagnostic.

The fix is to make the Pi-tool-name entry self-contained: state the resolution predicate (single exact-match lookup against Pi's tool registry, name taken verbatim) and cross-reference the unrecognised-name disposition (`loom/load/unknown-tool`) and/or the PIC tool-registration surface, rather than leaving "exactly as for Pi subagents" to stand alone.

**History:** `git log -S "exactly as for Pi subagents"` traces the phrase to `8eea523`, the commit that first authored the `tools:` field prose. Subsequent reorganisations only moved the failure-disposition prose without ever pinning resolution at the cited site: `7cd9cc6` (2026-05-04, "spec: unify tools and loom-as-tool callables under tools:") restructured the prose into the "Two kinds of entry" bullets, dropping the trailing clause from the Pi-tool-name bullet and relocating it to a consolidated paragraph; `839db71` (2026-05-04, "resolve 'Diagnostic codes … never assigned'") assigned `loom/load/unknown-tool` to that consolidated paragraph; and `f5e89f4` (2026-06-04, the 100KB size-cap split) moved the consolidated paragraph onto the sibling page `frontmatter-fields-b-and-templates.md`, where `git log -L 78` confirms the Pi-tool-name bullet has carried the bare "exactly as for Pi subagents" sentence — with no inline predicate and no cross-reference — ever since `frontmatter-fields-a.md` was created. The external-deference defect is therefore an inception-era property of the `tools:` text, not a regression introduced by any single later commit.

## Solution Space

**Shape:** single

### Recommendation

Make the Pi-tool-name entry self-contained at `frontmatter-fields-a.md#tools`. Extend the "Pi tool names" bullet so it (a) states the resolution predicate — a single exact-match lookup of the entry string against Pi's tool registry at loom-load time, with the registry name taken verbatim as the exposed callable name (already stated downstream: "the entry's name is the Pi tool name verbatim") — and (b) cross-references the unrecognised-name disposition `loom/load/unknown-tool` (and, where it is the natural anchor, the PIC tool-registration surface), so the entry no longer relies on "exactly as for Pi subagents" as its sole authority. The cross-reference target already exists (the consolidated disposition paragraph on `frontmatter-fields-b-and-templates.md` and the `loom/load/unknown-tool` registry row); this fix closes the loop by linking it from the site that introduces the rule, restoring symmetry with the inline-diagnostic treatment the `.loom`-path entry already has.

Edge case for the implementer: do not over-pin Pi's internal lookup mechanism — pin only the observable contract (exact-match against the registry; unrecognised → `loom/load/unknown-tool`; matched name exposed verbatim). The file-watcher reload path (a previously-resolved Pi tool later absent from the registry → `loom/load/unknown-tool` on the next invocation) is already covered on the sibling page and need not be restated.

## Consequence

**Severity:** advisory

A working implementation is still achievable — the `loom/load/unknown-tool` diagnostic exists and the disposition is documented on the sibling page — but a reader of the Pi-tool-name entry gets no pointer to either, and the match predicate (exact vs normalised/case-insensitive lookup) is nowhere pinned in loom's own text. The diagnostics registry's `loom/load/unknown-tool` row back-links to "Frontmatter — `tools`", but the owning prose at that site no longer documents the trigger, leaving a dangling back-reference.
