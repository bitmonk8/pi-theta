## Finding

In `docs/spec_topics/related-work.md`, the *Direct influences* section's `mech` bullet links `[backlot](../backlot)` and names the inline path `backlot/docs/MECH_SPEC.md`. Resolved relative to the document's location (`docs/spec_topics/`), `../backlot` points at `docs/backlot`, which does not exist; the actual sibling `backlot` checkout lives above the repository root (`../../../backlot` from this file). The Markdown link is therefore dead as written, and the inline `backlot/docs/MECH_SPEC.md` code-span names a real file at a path not resolvable from this document.

The defect is confined to a non-normative appendix — the file itself states "nothing in the rest of the spec depends on it" — so the only observable effect is broken reader navigation within the appendix. The original framing ("non-existent sibling directory") is imprecise: the `backlot` project does exist as a sibling of the repo root; the link is simply off by directory levels after the spec was relocated. The substance of the finding is unaffected.

## Issue introduction

**Verdict:** introduced-by-relocation (single introducing commit)
**Introducing commits:** `fecb504` ("Split spec.md and plan.md into per-topic / per-phase files")
**History:** The `[backlot](../backlot)` link was authored in `53f5831` (2026-05-03) inside the monolithic repo-root `spec.md`; from the repo root, `../backlot` correctly resolved to the sibling checkout, so the link was sound at inception. `fecb504` split `spec.md` into `spec_topics/related-work.md` — one directory deep — without deepening the relative path, breaking resolution. `31ff060` (2026-05-08) later moved the topic tree under `docs/`, widening the gap to two levels, but the link was already dead as of `fecb504`. The inline `backlot/docs/MECH_SPEC.md` code-span entered with the same `53f5831` authoring and was likewise never repointed.

## Solution Space

**Shape:** multiple

### Option A — Repoint to the sibling checkout

**Approach:** Change the link target from `../backlot` to `../../../backlot` and correct the inline path to `../../../backlot/docs/MECH_SPEC.md`, so both resolve to the sibling `backlot` checkout from `docs/spec_topics/related-work.md`.

**Spec edits:** In the `mech` bullet, replace `[backlot](../backlot)` with `[backlot](../../../backlot)` and `backlot/docs/MECH_SPEC.md` with `../../../backlot/docs/MECH_SPEC.md`.

**Pros:** Preserves click-through navigation for a contributor whose working tree mirrors the assumed sibling-checkout layout.

**Cons:** The target points outside the repository, so it breaks on any rendered/published view of the docs (GitHub blob view, a docs site) and silently re-breaks if the checkout layout differs. It also re-encodes a layout assumption that has already caused two regressions.

**Risks:** Reintroduces a fragile path that the next relocation will break again.

### Option B — Demote to plain prose

**Approach:** Drop the hyperlink and the inline path entirely; name the `mech` project and the `backlot` repository in plain prose (optionally as a stable public URL if one exists).

**Spec edits:** In the `mech` bullet, replace `the sibling [backlot](../backlot) project (`backlot/docs/MECH_SPEC.md`)` with plain text naming the `mech` engine in the `backlot` project — no Markdown link and no intra-tree path. If a canonical public URL for `backlot`/`MECH_SPEC` exists, use that absolute URL instead.

**Pros:** Robust against relocation and against rendered/published views; nothing to re-break. Matches the non-normative, orientation-only purpose of the appendix.

**Cons:** Loses one-click navigation to the sibling spec.

**Risks:** None material.

### Recommendation

Take Option B. A relative link into a sibling repository outside the published doc tree is inherently fragile — it has already silently broken twice on relocation and cannot render on any web view — and the reference lives in a non-normative orientation appendix where click-through is a convenience, not a requirement. If a stable public URL for `backlot`/`MECH_SPEC.md` is available, use it as an absolute link; otherwise name the project in plain prose. Watch that the edit removes both the hyperlink and the inline code-span path, not just one.

## Consequence

**Severity:** cosmetic

A reader following the `mech`/`backlot` reference in the Related Work appendix hits a broken link. The appendix is non-normative and nothing in the spec corpus or plan depends on it, so no implementer behaviour diverges.
