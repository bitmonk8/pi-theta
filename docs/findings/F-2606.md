## Finding

The **Provider error mapping** paragraph and the **`TransportError.retryable` population** rule both enumerate "a surfaced 3xx" as one of the captured statuses that map to `TransportError` with `retryable: false` ("…and every other captured status including a non-200 2xx (201/202/204) or a surfaced 3xx…"). The page never states under what condition a 3xx redirect status would actually reach the classifier. The classifier reads its HTTP-status signal exclusively from `ProviderResponse.status` delivered through `@earendil-works/pi-ai`'s `StreamOptions.onResponse` callback (per the *Classifier input surface* note on the same page); HTTP clients and provider SDKs normally auto-follow 3xx redirects, so a bare 3xx is consumed inside the transport layer and never surfaces as a resolved `ProviderResponse.status` to loom. loom neither configures nor can configure redirect-following through pi-ai, so the cited "surfaced 3xx" is, on the described input surface, an unreachable case.

The classification rule itself is total and unambiguous — the leading sentence already says every classifier-reaching response that does not match an overflow signature maps to `TransportError` "regardless of HTTP status", which subsumes any 3xx that could ever arrive. The "or a surfaced 3xx" parenthetical therefore adds no behavioural content; its only effect is to imply, misleadingly, that a 3xx is an expected status loom must arrange to observe (e.g. by disabling redirect-following), which it is not. The example reads as a contract loom owns when it is at most a defensive catch-all for a status the described surface does not produce.

The defect is thus a side-effect of the F-1853 totalisation: broadening the rule to cover the full status domain introduced illustrative status examples without checking each example against the page's own input surface (`ProviderResponse.status` via `onResponse`), against which a 3xx is not reachable.

## Solution Space

**Shape:** single

### Recommendation

Keep the totalised rule (the "regardless of HTTP status → `TransportError(retryable: false)`" catch-all is correct and should remain), and remove the misleading "surfaced 3xx" example from both sites — the **Provider error mapping** paragraph and the **`TransportError.retryable` population** rule. Drop "or a surfaced 3xx" from each parenthetical, leaving "and every other captured status" (and the retained "non-200 2xx (201/202/204)" example, which the same `ProviderResponse.status` surface can plausibly carry) to cover the residual status domain. No behavioural detail is lost: any 3xx that did somehow reach the classifier remains covered by the explicit "regardless of HTTP status" clause.

If the example is retained for completeness rather than deleted, annotate it inline as a defensive catch-all for a status loom does not expect to observe on the `onResponse`/`ProviderResponse.status` surface (because redirect-following is internal to the transport layer loom does not configure) — never as a status loom must arrange to surface. Do not add a redirect-following-disable requirement: loom has no control over pi-ai's HTTP client.

## Consequence

**Severity:** cosmetic

The classification is already fully determined for every status by the "regardless of HTTP status → `TransportError(retryable: false)`" rule, so no two implementers diverge on disposition. The only cost is reader confusion: the "surfaced 3xx" example can be misread as an obligation to arrange 3xx visibility (e.g. disable redirect-following) — work loom cannot perform through pi-ai and need not perform, since the catch-all already covers the case.
