## Problem

The `loom/load/invalid-tool-rename` Tests bullet in both `V6c` and `V6c-T` specifies rejection of a `tools:` `as` rename target "that is not loom-identifier-shaped (e.g. `as MyTool`)." The phrase "loom-identifier-shaped" is undefined in the leaf, and the single `as MyTool` example illustrates only one rejection class; an implementer reading the leaf alone cannot tell whether the predicate is "lowercase first letter," the full identifier grammar, or something that also excludes reserved keywords. The leaf's `Spec.` pointer (`frontmatter-fields-a.md` → `lexical.md` §Identifiers) does resolve the dominant case, but the leaf restates the spec's loose trigger phrasing rather than pointing at the definition. Reserved-word rename targets (`read as for` — grammar-shaped and lowercase-first, so it does not trip the lowercase-first rule) are genuinely unspecified spec-wide.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `7cd9cc6` (2026-05-04, "spec: unify tools and loom-as-tool callables under tools:; add ToolCallError") introduced the `tools:` `as` rename, the `loom/load/invalid-tool-rename` diagnostic, and the "not loom-identifier-shaped (e.g. `as MyTool`)" trigger text into the spec, already carrying the unspecified reserved-word handling. `60c9400` (2026-06-17, "pi-loom plan: resolve 'tools:-resolution load codes have no asserting plan leaf'") propagated the identical loose trigger text into the `V6c` and `V6c-T` Tests bullets.
**History:** `git log -S "as MyTool"` traces the example back to `7cd9cc6`; the reserved-word question has never been narrowed by any later commit. The plan leaves restated the spec's pre-existing loose phrasing verbatim rather than introducing a new defect — the ambiguity has been present since the `tools:` `as` rename feature was first written.

## Solution approach

In both `V6c` and `V6c-T`, rewrite the `loom/load/invalid-tool-rename` Tests bullet so the rejection predicate references the spec's identifier definition (`frontmatter-fields-a.md`'s lowercase-first identifier rule, and through it `lexical.md` §Identifiers) instead of the bare phrase "loom-identifier-shaped," keeping the `as MyTool` example. Surface the reserved-word residual (`read as for` — which diagnostic, if any) as a separate spec-clarity item against `frontmatter-fields-a.md` / `lexical.md` rather than encoding a guessed behaviour into the leaf.

## Solution constraints

- The spec is read-only for this fix: do not edit `frontmatter-fields-a.md`, `lexical.md`, or the diagnostics registry.
- Do not invent a plan-side answer for the unresolved reserved-word case; any reserved-word test vector added to the leaf must follow the spec's resolution of that question, not precede it.
