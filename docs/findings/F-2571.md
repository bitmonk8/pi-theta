## Finding

The Runtime section of `implementation-notes.md` states, as a normative property of the typed-query forced-respond mechanism, that "the runtime does **not** install a `before_provider_request` payload-rewrite hook in loom 1.0" — the global-hook design (Option B) was rejected in favour of the per-call `complete()` `options.toolChoice` route, and `conversation-drive.md` carries the same statement ("the `before_provider_request` payload-rewrite hook loom 1.0 rejects"). `before_provider_request` is a real Pi extension event subscribed via `pi.on("before_provider_request", …)`, so this is a falsifiable prohibition on a concrete host surface, not merely descriptive prose.

The prohibition carries no conformance hook. There is no build-time check, no test predicate, and no SDK signal named that would catch an implementation that installs such a hook. This is out of step with the sibling prohibition on the same page: the Clock ban ("the runtime MUST NOT call `Date.now`, `performance.now`, …") is explicitly backed by a build-time lexical-reference check cited to PIC-12. The surface-set-closure / inventory-closure audit does **not** close the gap either — that audit asserts every *used* Pi surface resolves to an inventory entry or exemption; it does not forbid any particular surface, so a `pi.on("before_provider_request", …)` subscription could simply be added to the inventory and pass. Nothing in the corpus makes the no-hook property observable.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** 82be125fb8cda00e058d57d3b90f07599a86a3e7 (2026-05-04) "spec: adopt P1 typed-query two-phase loop (Option A)"
**History:** The corpus is git-tracked. Before 82be125 the reference implementation *used* a `before_provider_request` hook with per-payload fingerprinting (the mechanism expanded in 6bd1844, 2026-05-04, "resolve 'before_provider_request fires on every request'"). Commit 82be125 flipped the design to the two-phase `complete()` route and replaced the hook mechanism with the prohibition "The runtime does **not** install a `before_provider_request` payload-rewrite hook in V1 … Option B … rejected" — introducing the no-hook obligation *without* any accompanying verification clause (`git log -S 'before_provider_request' --follow` / `git show 82be125 -- spec_topics/implementation-notes.md`). Later commits (7f75947 restating it in `conversation-drive.md`, the 31ff060 `docs/` relocation, and subsequent re-narrations) preserved the prohibition verbatim but never added a build-time check or test predicate. The defect has thus been present since the prohibition itself was authored; the sibling Clock prohibition received a PIC-12 build-time check, this one did not.

## Solution Space

**Shape:** multiple

### Option A — Build-time lexical-reference gate (PIC-12 analogue)

- **Approach:** Add a normative conformance criterion to the no-hook statement in `implementation-notes.md` requiring a build-time check that the loom runtime never references the `before_provider_request` event name (i.e. no `pi.on("before_provider_request", …)` subscription) anywhere in `src/`, mirroring the wording PIC-12 uses for the Clock primitive ban ("a conforming implementation MUST carry a build-time check rejecting direct lexical references … how the check is realised is an implementation choice").
- **Spec edits:** Append to the no-hook sentence a clause naming the conformance criterion and cross-referencing the PIC-12 pattern; state that the `"before_provider_request"` event-name string is banned in `src/` (no adapter carve-out is needed, since the runtime never legitimately uses it).
- **Pros:** Statically proves the property over the whole tree, not just exercised code paths; consistent with the established PIC-12 / H3a-scan precedent; cheap to implement as a `no-restricted-syntax` or identifier/string scan.
- **Cons:** A pure string-literal scan can be evaded by computing the event name dynamically (same residue class PIC-12's scan already concedes for indirect ambient reads).
- **Risks:** Adds one more manually-maintained banned-token entry to the cross-cutting gate; low.

### Option B — Runtime conformance test on the typed-query path

- **Approach:** Specify a test obligation that, while driving a typed query end-to-end, asserts the runtime never subscribes a `before_provider_request` handler (spy on `pi.on` and assert no call with that event name).
- **Spec edits:** Add a test-predicate sentence to the no-hook statement requiring the spy-based assertion during a typed query.
- **Pros:** Behavioural and host-realistic; catches dynamically-named subscriptions a string scan would miss.
- **Cons:** Only witnesses absence on the exercised path/turn, not a true never-installed property; weaker than a whole-tree static check.
- **Risks:** Could give false confidence that the hook is absent everywhere when it only proves absence for the tested scenario; low.

### Recommendation

Adopt Option A: attach a build-time lexical-reference conformance criterion to the no-hook statement, cross-referenced to the PIC-12 pattern, so the prohibition is verified the same way the sibling Clock ban is. The implementer should treat the `"before_provider_request"` event-name string as the banned token in `src/` with no exempt site, and accept the same dynamic-construction residue PIC-12 already concedes (owned by human self-review at release time).

## Consequence

**Severity:** advisory

An implementer can satisfy every behavioural typed-query test while still installing a `before_provider_request` hook (e.g. re-introducing the rejected Option-B fingerprinting machinery), because no check witnesses its absence. The functional output is unaffected, so this is a conformance/guidance gap rather than a divergence in observable runtime behaviour; the cost is an undetected drift back toward the cross-session-fingerprinting design the spec deliberately rejected.
