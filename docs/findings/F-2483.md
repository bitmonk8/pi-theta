## Finding

The **Serialised content format** paragraph in `diagnostic-shape.md` defines the `content` string of a `loom-system-note` as a byte-exact wire surface: it fixes the main line format, the `"\n  hint: <hint>"` continuation, and the two-space-indented `"  <file>:<line>:<col>: <message>"` related-site lines. This string is observable — tests, LSP integrations, and test harnesses consume it — and the diagnostics contract elsewhere (DIAG-4) treats rendered text as character-for-character normative. Two byte-exact details are left to prose rather than pinned, so two conforming implementations can emit different `content` bytes for the same diagnostic.

First, when a single diagnostic carries **both** a `hint` and one or more `related` entries, the relative order of the hint continuation line and the related-site lines is never stated. The hint is described as following the main line and related sites are "appended on its own line … matching the hint line", which suggests but does not pin `main → hint → related`. An implementer could equally emit `main → related → hint`. The two-space indentation shared by both line kinds means there is no other structural cue to disambiguate.

Second, the multi-error batch separator is specified as "successive blocks are separated by **a single blank line**". This is prose, not a pinned character sequence. A blank line between two blocks is most naturally `\n\n`, but the spec does not say so, leaving open whether the join is `\n\n`, `\n\n\n` (if blocks are treated as already newline-terminated), or a line containing whitespace. For a `content` string that downstream consumers parse and that tests assert on, the exact byte sequence must be fixed.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `fae85d3` (2026-05-04 "Tighten spec for implementation-plan readiness") — introduced the serialisation line `"<file>:<line>:<col>: <code>: <message>"`, optionally followed by `"\n  hint: <hint>"`, with "Related sites are appended as additional indented lines"; the hint continuation and related-site appending have coexisted from this point with no pinned relative ordering (defect half (a)).
- `ac8c3af` (2026-05-04 "resolve LoadExtensionsResult.errors is the wrong delivery channel for loom diagnostics") — added the multi-error batch clause "successive blocks are separated by a single blank line", introducing the unpinned-separator prose (defect half (b)).

**History:** Both halves entered during initial authoring of the serialised-content surface and were never tightened. `8837a93` (2026-06-05 "resolve Serialised content related-site line format is undefined") later expanded "Related sites are appended as additional indented lines" into the detailed `"  <file>:<line>:<col>: <message>"` format but did not pin the hint/related relative order; the single-blank-line separator prose has been carried verbatim through the `f5e89f4` diagnostics.md → diagnostic-shape.md split untouched. Neither half is a regression introduced by a later edit — each is present-since-inception of its own clause.

## Solution Space

**Shape:** single

### Recommendation

In the **Serialised content format** paragraph, pin both byte-exact details:

- **Line ordering within one diagnostic block.** State that the lines emit in the fixed order: main line, then the `"\n  hint: <hint>"` continuation (when a hint is present), then each `related` line in `Diagnostic.related` array order. Make explicit that the hint line precedes all related lines.
- **Multi-error block separator.** Replace "separated by a single blank line" with the literal sequence: each line block is joined to the next by `"\n\n"` (i.e. a `\n` terminating the preceding block's last line plus a `\n` producing the intervening blank line), with no trailing separator after the final block and no whitespace on the blank line.

Edge cases the implementer must cover: a diagnostic with a hint but no related entries (hint line present, no related lines); a diagnostic with related entries but no hint (main line straight to related lines, no blank line between them); and a single-element batch (no separator emitted at all).

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the `content` bytes — one ordering hint-before-related, the other related-before-hint, and disagreeing on the exact separator byte sequence between batched error blocks. Because the `content` string is an observable conformance surface that tests and LSP/test-harness consumers read, byte-exact assertions written against one implementation fail against the other despite both being "conformant" to the prose.
