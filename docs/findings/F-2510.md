## Finding

The glossary defines **type sink** as "an enclosing AST position whose declared type supplies a type to its operand during the outward type-inference walk", and that term is used consistently across the corpus — `grammar.md` (`array<T>` literal type-sink rule), `query/query-forms.md` (schema inference), and `control-flow.md` (element-type sink for empty-array literals) all say "type sink".

Inside `expressions.md` **Array construction** (line 216), the prose paragraph instead uses the bare word "context" three times for that same concept: "its element type is inferred from **context** (binding annotation, parameter type, or surrounding constructor field)", "narrowed by **context** if applicable", and "no **context** to narrow against". The numbered common-type rules immediately below (lines 220–227) then switch to the canonical term — "If a **type sink** is in scope … `T_element ⊑ T_sinkElement`". The single section thus names one concept two ways within a few lines.

"Context" is a broader English word than the precise, closed sink-set the glossary pins; using it in the lead-in prose while the normative numbered rules use "type sink" invites a reader to treat the prose notion as wider than the enumerated sink positions. The parenthetical enumeration in the first occurrence mitigates this, but the second and third occurrences carry no such qualifier.

## Solution Space

**Shape:** single

### Recommendation

Replace the three "context" occurrences in the Array-construction prose paragraph with "type sink", matching the glossary term and the numbered rules below. Retain the parenthetical sink enumeration on first use, e.g. "its element type is inferred from a type sink in scope (binding annotation, parameter type, or surrounding constructor field)"; "narrowed by a type sink if applicable"; "no type sink to narrow against". This is a pure wording edit confined to one paragraph in `expressions.md`; no rule, diagnostic, or cross-reference changes.

## Consequence

**Severity:** cosmetic

A terminology inconsistency within one section. The normative numbered rules already use "type sink" precisely, so an implementer is unlikely to diverge on behaviour; the cost is reduced glossary-term discoverability and a mild risk of reading "context" as broader than the closed sink set.
