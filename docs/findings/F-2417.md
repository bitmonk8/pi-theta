## Finding

The *Per-family record-shape table* in `audit-failures.md` (anchor `audit-per-family-record-shape-table`) pins a violation record as four fixed fields — `path`, `line`, `symbol`, `proposed-resolution` — and states the conformance MUST as: a record "that omits any of the four fields, or that carries a value **outside the per-family range** below for a given family, violates this rule." For `path`, `line`, and `symbol` the per-family cells give a mechanically decidable shape (a bare member identifier, a package-qualified name, a normalised single-line shape literal, or the exact `<n/a>` sentinel). For `proposed-resolution`, the cells give only a semantic-type description — e.g. family (1)'s "inventory-entry-suggestion: the `SDK_SURFACE_INVENTORY` entry the contributor would add … or the canonical `// allow-pi-surface:` marker", family (4)'s "recognised-shape suggestion", family (5)'s "deletion-or-rewrite suggestion … naming whichever of that branch's five sub-cases the marker falls under". None of these is a grammar or predicate, so "outside the per-family range" is not a decidable test: a conformance suite cannot mechanically determine whether a given `proposed-resolution` string is in-range, and two implementers can emit materially different free-text suggestions that both pass (or both fail) against the notional check the MUST promises.

The `proposed-resolution` field carries a second normative content obligation that is equally unverifiable. The *Category-(2) inventory join key* in `audit-resolution.md` closes with: "For legal multi-segment matches … the audit's `proposed-resolution` field reports the entry with the lexicographically-smallest `path` for determinism." To assert that the audit reported *that specific* entry, a test must locate the entry reference inside the `proposed-resolution` string — but the field's content format is never pinned (no leading token, no machine-parseable entry reference). The lex-min selection therefore imposes a content requirement with no test oracle: an implementation could select the correct entry and serialise it in a form no parser can confirm.

The defect is confined to `proposed-resolution`. The `audit/<class>/<family>/<symptom>` discriminator token — the field a CI parser actually routes on — is fully pinned and testable, so the audit's fail-closed routing behaviour is unaffected; only the record-shape MUST's claim that `proposed-resolution` is a checkable obligation, and the lex-min selection's verifiability, are at issue.

## Solution Space

**Shape:** multiple

### Option A — Demote `proposed-resolution` out of the mechanically-checkable record-shape MUST

**Approach:** Narrow the *Per-family record-shape table*'s "carries a value outside the per-family range … violates this rule" MUST to the three fields with a decidable shape (`path`, `line`, `symbol`). Reclassify `proposed-resolution` as advisory, contributor-facing remediation guidance whose free-text content is explicitly *not* a conformance-checkable obligation. Demote the *Category-(2) inventory join key*'s lex-min sentence to a non-normative recommendation, retaining only a pinned same-implementation reproducibility invariant (two runs over the same tree emit byte-identical output) as the testable property — matching the permissive formulation the *Wire serialisation* file-iteration ordering rule already uses.

**Spec edits:** Reword the record-shape MUST sentence in `audit-failures.md` so the "range"/"violates this rule" predicate scopes to `path`/`line`/`symbol`; add a clause marking `proposed-resolution` advisory. In `audit-resolution.md`, change "reports the entry with the lexicographically-smallest `path` for determinism" to a non-normative "reports a single entry chosen by a deterministic, run-stable tie-break the implementation pins (e.g. the lexicographically-smallest `path`)".

**Pros:** Removes the untestable MUST with the smallest wire-format footprint; keeps `proposed-resolution` human-useful as remediation text; aligns the tie-break with the already-permissive file-iteration ordering rule one section over; lets V18b-T author a deterministic assertion over only the fields that have a decidable shape.

**Cons:** Sheds any machine-readable confirmation that the audit suggested the correct fix; a CI parser can no longer verify the lex-min selection or the suggestion content.

**Risks:** If cross-implementation byte-identical `proposed-resolution` is in fact intended downstream, demoting silently drops a guarantee — but no clause in the audit cluster states such an intent today.

### Option B — Pin a machine-parseable leading entry-reference token in `proposed-resolution`

**Approach:** Define a parseable format for `proposed-resolution` so the per-family "range" and the lex-min selection become mechanically decidable. For families (1)/(2)/(3) the field begins with the resolved `SDK_SURFACE_INVENTORY` entry's `path` (or the canonical `// allow-pi-surface:` marker form) as a bare leading token, on the same footing as the canary's established `walked=<N> recognised=<M>` plain-ASCII convention; a parser can then confirm range membership and the lex-min-chosen `path` by exact match.

**Spec edits:** Add a sentence to the *Per-family record-shape table* (or *Wire serialisation*) pinning the leading-token form per family; keep the lex-min sentence normative in `audit-resolution.md` but tie it to the pinned token so "reports the entry" names a confirmable serialised slot.

**Pros:** Preserves a machine-checkable resolution suggestion and a verifiable lex-min selection; gives a test leaf a stable assertion target for the full four-field record.

**Cons:** Invents a parser-facing format the wire contract does not otherwise need (routing already rides the discriminator token); overloads a contributor-resolution field with a parser contract; adds a new byte-exact surface to keep coherent with the family-(4)/family-(5) suggestion forms, which are not simple entry references.

**Risks:** The pinned token must be plain printable tab/newline-free ASCII (consistent with the field's existing constraint); families (4) and (5) have no inventory-entry reference to lead with, so the format must define their leading token too, or scope the pin to families (1)–(3).

### Recommendation

Adopt **Option A**. `proposed-resolution` is contributor-facing remediation guidance, and the wire contract's machine-routing already rides entirely on the `audit/<class>/<family>/<symptom>` discriminator, so nothing downstream needs `proposed-resolution` to be parser-checkable; demoting it removes the untestable MUST without inventing a format the spec does not otherwise use, and aligns the tie-break with the permissive file-iteration ordering rule. Edge case the editor must watch: if a future requirement genuinely needs a *cross-implementation* byte-identical or machine-confirmable suggestion, do not demote — take Option B and pin the leading-token format, stating the cross-implementation reproducibility need explicitly as its justification.

## Consequence

**Severity:** correctness

The record-shape MUST asserts a conformance check ("violates this rule") over a `proposed-resolution` value that has no decidable in-range predicate, and the lex-min selection has no parseable target, so two reasonable implementers diverge both on what `proposed-resolution` content is conformant and on whether the field is enforced at all — and a conformance test (V18b-T) cannot be written to gate it. The audit still fails closed and routes by the testable discriminator token, so the gate's safety property holds; the divergence is confined to the emitted record's `proposed-resolution` content and to the missing test oracle for it.
