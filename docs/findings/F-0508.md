## Finding

`V4e`'s **Adds** and **Tests** both lean on `CIO-1` to fix the load-time cross-route for `ERR-16`: the Adds says the slash-load `params` arm of ceiling #4 "**consults** `V16a`'s cross-ceiling arbitration seam at slash-load per CIO-1", and the `ERR-16` test bullet says it is "cross-routed via CIO-1 / ceiling #3 no-retry". But `V4e`'s **Spec.** field links only `errors-and-results/error-model.md`. `CIO-1` is a full REQ-ID owned elsewhere — `spec_topics/hard-ceilings/ceilings-3-and-4.md#cio-1` — whose normative text is what actually fixes the routing: ceiling #3 is evaluated at slash-load time before any runtime-class ceiling fires, and the slash-load `params` arm of ceiling #4 is routed by ceiling #3's failure-mode templates per the per-boundary table.

Without a Spec link to that definition, "per CIO-1" is an unresolved token at the leaf. An implementer cannot tell from `V4e`'s linked spec page whether `CIO-1` is a sequencing constraint, a routing/destination template, or something else, and two implementers can produce incompatible `ERR-16` load-time routing logic. The token is partially reachable — `error-model.md`'s own `ERR-16` entry mentions `CIO-1` and links the ceiling-#4 per-boundary table — but it never links the `#cio-1` anchor that carries the operative load-vs-runtime ordering rule, so the leaf still does not pin the constraint it depends on.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V4e-pre-evaluation-failures.md`, add the `CIO-1` definition page to the **Spec.** field alongside the existing `error-model.md` link, pointing at the anchored target `../spec_topics/hard-ceilings/ceilings-3-and-4.md#cio-1`. Because the `V4e-T` tests task shares this leaf page, the single Spec-field edit covers both tasks; no separate `V4e-T` file exists to edit.

Inlining the operative constraint instead of linking is an acceptable alternative only if the link is also added — `CIO-1` is a governed REQ-ID and the leaf should resolve to its anchor. Confirm the `#cio-1` fragment resolves at `docs/spec_topics/hard-ceilings/ceilings-3-and-4.md` (it does as authored). Spec pages are read-only for this fix; do not edit `ceilings-3-and-4.md` or `error-model.md`.
