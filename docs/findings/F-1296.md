## Problem

The `options.onResponse` callback feeds the captured `ProviderResponse { status, headers }` to the `provider-error-mapping.md#classifier-input-surface` classifier, joined per `complete()` invocation. The spec gives no rule for which `ProviderResponse` is joined with the resolved `AssistantMessage` when `onResponse` fires more than once before `complete()` resolves — which it can if pi-ai's `complete()` retries internally. Because the classifier's `TransportError.retryable` decision and the context-overflow-vs-network precedence both read `ProviderResponse.status`, two conformant implementations can disagree on `retryable` and on precedence for the same provider trace.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `44782e9` (2026-06-06, "resolve 'TransportError catch-all / pi-ai provider-error surface'") — added the `options.onResponse` bullet and the matching *classifier-input-surface* paragraph in `provider-error-mapping.md` (same commit), pinning a join keyed "per `complete()` invocation" but silent on cardinality should the SDK retry internally.
**History:** `git log -S'onResponse' -- docs/spec_topics/pi-integration-contract/binder-inference.md` returns `44782e9` alone. The defect is one facet of a larger `complete()`-call-shape under-pinning across `binder-inference.md` and `provider-error-mapping.md`.

## Solution approach

Clarify the `options.onResponse` bullet in `binder-inference.md`'s `#binder-inference-call` section, and its mirror clause at `provider-error-mapping.md#classifier-input-surface`, to pin a deterministic rule for which captured `ProviderResponse` the classifier joins when `onResponse` fires more than once before `complete()` resolves, alongside the existing no-fire (network-level) case. The rule governs any `complete()`-routed loom call; multi-fire reachability from the binder call itself is bounded by F-1295's `maxRetries` disposition.

## Solution constraints

- Out of scope: F-1199's rejected/thrown `complete()` → `QueryError` mapping and the overflow-vs-network precedence rule — only the multi-`onResponse` join half lands here.
- Out of scope: the `maxRetries` / `maxRetryDelayMs` disposition owned by F-1295; reference its `maxRetries = 0` pin, do not restate or re-decide it.
