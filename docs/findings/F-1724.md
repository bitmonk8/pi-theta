## Finding

`loom/runtime/reload-teardown-timeout` (`docs/spec_topics/diagnostics/code-registry-runtime.md`, the timeout row) fires whenever the `session_shutdown` handler's sub-step 3 bounded await on every in-flight `disposeBarrier` exhausts the `SHUTDOWN_AWAIT_CAP_MS` cap, and that handler "runs in fixed order regardless of `event.reason`" (`docs/spec_topics/pi-integration-contract/patch-skew-degradation.md`, sub-step 3) — i.e. on `"quit"`, `"reload"`, `"new"`, `"resume"`, `"fork"`, and on the **Unknown-reason rule**'s `string` arm. The code identifier names only one of those five reasons, so an operator or test author reading the code, or seeing the message `reload teardown timed out after <ms>ms; …`, is led to believe the diagnostic is gated on `event.reason === "reload"` — which the trigger explicitly is not.

The identifier itself cannot be changed within loom 1.x: [DIAG-3](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-3) classifies a rename as a breaking change to the public diagnostics contract and defers it to loom 2.0, and the [GOV-15 Diagnostic-registry carve-out](../../docs/spec_topics/governance/source-language-stability.md#diagnostic-registry-carve-out) reinforces that renames are loom 2.0 concerns. The same is true of the *Message* template, deferred by [DIAG-4](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-4). The spec already carries a precedent for this exact situation on the sibling `loom/load/cross-format-collision` row (`docs/spec_topics/diagnostics/code-registry-load.md:38`): a name/scope mismatch that DIAG-3 prevents renaming is reconciled by adding a self-disclosure clause to the registry row's *Trigger* cell — "The code name retains the historical `cross-format` token as a stable diagnostics-contract identifier per [Diagnostic shape — DIAG-3](diagnostic-shape.md#diag-3); its scope is the full same-priority slash-name collision rule, not only the cross-format arm." The `reload-teardown-timeout` row carries no equivalent self-disclosure, so the in-1.x mitigation that DIAG-3 already permits is unused.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commits:** `5e8b499` (2026-05-06, *pi-loom spec: resolve "Hot-reload teardown contract for the loom extension instance is unspecified"*).

**History:** `5e8b499` introduced both the `loom/runtime/reload-teardown-timeout` registry row (with the `reload-` token and the `reload teardown timed out …` *Message* template) and, in the same commit on `spec_topics/pi-integration-contract.md`, the framing that the `session_shutdown` handler "runs in fixed order regardless of `event.reason`" and "treats every reason identically" — explicitly enumerating `/reload`, `/new`, fork, or quit as the reasons it must cover. The name/trigger mismatch was therefore present at the row's first appearance, not introduced by a later widening of the trigger. Subsequent commits (`9559e19` 2026-05-07 unknown-reason rule, `52b9101` 2026-05-11 fork-reason watcher closure) widened the carve-out enumeration around the code but did not edit the row name or message and did not introduce the mismatch. No prior version of the row used a more general name.

## Solution Space

**Shape:** single

### Recommendation

In the `loom/runtime/reload-teardown-timeout` row of `docs/spec_topics/diagnostics/code-registry-runtime.md`, append a self-disclosure sentence to the *Trigger* cell, parallel to the existing precedent on the `loom/load/cross-format-collision` row, with the substantive content:

> The code name retains the historical `reload-` token as a stable diagnostics-contract identifier per [Diagnostic shape — DIAG-3](diagnostic-shape.md#diag-3); its scope is the full `session_shutdown` handler's bounded-await timeout on `event.reason ∈ {"quit","reload","new","resume","fork"}` and on the **Unknown-reason rule**'s `string` arm, not only the `reload` reason. The *Message* template's `reload teardown timed out after <ms>ms; …` literal is preserved verbatim under [DIAG-4](diagnostic-shape.md#diag-4); operators MUST NOT infer the firing `event.reason` from the rendered message.

Edge cases the implementer must watch:

- Place the sentence at the end of the *Trigger* cell, after the existing residual-gap EXCEPT clause and before the `console.error`-routing sentence, so the row's narrative order (what fires it → how it is routed) is preserved.
- Do **not** edit the row's *Code* cell, *Message* template, or sub-step-3 cross-references — those are DIAG-3 / DIAG-4 changes deferred to loom 2.0. The fix is strictly an additive *Trigger*-cell self-disclosure.
- Do **not** introduce a new spec-side "rename queue" or 2.0-migration register; the loom 2.0 rename is already adequately governed by DIAG-3 itself, and the `cross-format-collision` precedent does not maintain such a queue either. A `details` field for the firing `event.reason` is independently provided by the sibling `cancelled-by-session-shutdown` row's `details.event.reason`, not by this row's `<list>`; that asymmetry is intentional and out of scope here.

## Consequence

**Severity:** advisory

Operators and test authors triaging a `session_shutdown` teardown timeout on `event.reason ∈ {"quit","new","resume","fork"}` see a `reload teardown timed out …` message and a `reload-teardown-timeout` code that name a reason the actual event was not, and are led to look for a `/reload` cause that does not exist. Implementation behaviour is not divergent — the code's emission site is well-pinned — but the diagnostic's operator-visible surface misdirects triage, and the in-1.x mitigation the spec already uses on a sibling row is missing.
