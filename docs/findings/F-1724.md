## Problem

`loom/runtime/reload-teardown-timeout` (`docs/spec_topics/diagnostics/code-registry-runtime.md`, the timeout row) fires whenever the `session_shutdown` handler's bounded await on every in-flight `disposeBarrier` exhausts the `SHUTDOWN_AWAIT_CAP_MS` cap, and that handler runs in fixed order regardless of `event.reason` — on `"quit"`, `"reload"`, `"new"`, `"resume"`, `"fork"`, and on the **Unknown-reason rule**'s `string` arm. The code identifier names only the `reload` reason, so an operator or test author reading the code, or the message `reload teardown timed out after <ms>ms; …`, is led to believe the diagnostic is gated on `event.reason === "reload"`, which it is not. The identifier and *Message* template cannot be changed within loom 1.x ([DIAG-3](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-3) defers a rename to loom 2.0; [DIAG-4](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-4) defers the message), so the row lacks the in-1.x self-disclosure mitigation the spec already applies on the sibling `loom/load/cross-format-collision` row.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commits:** `5e8b499` (2026-05-06, *pi-loom spec: resolve "Hot-reload teardown contract for the loom extension instance is unspecified"*).

**History:** `5e8b499` introduced both the `loom/runtime/reload-teardown-timeout` registry row (with the `reload-` token and the `reload teardown timed out …` *Message* template) and, in the same commit on `spec_topics/pi-integration-contract.md`, the framing that the `session_shutdown` handler "runs in fixed order regardless of `event.reason`" and "treats every reason identically" — explicitly enumerating `/reload`, `/new`, fork, or quit as the reasons it must cover. The name/trigger mismatch was therefore present at the row's first appearance, not introduced by a later widening of the trigger. Subsequent commits (`9559e19` 2026-05-07 unknown-reason rule, `52b9101` 2026-05-11 fork-reason watcher closure) widened the carve-out enumeration around the code but did not edit the row name or message and did not introduce the mismatch. No prior version of the row used a more general name.

## Solution approach

In the `loom/runtime/reload-teardown-timeout` row's *Trigger* cell, add a self-disclosure clause parallel to the existing `loom/load/cross-format-collision` precedent, stating that the code retains the historical `reload-` token as a stable diagnostics-contract identifier per [DIAG-3](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-3) and that its scope is the full `session_shutdown` bounded-await timeout across every `event.reason` arm, not only `reload`. Note that the *Message* literal is preserved verbatim under [DIAG-4](../../docs/spec_topics/diagnostics/diagnostic-shape.md#diag-4) and that operators cannot infer the firing `event.reason` from the rendered message.

## Solution constraints

- Out of scope: the *Code* cell, the *Message* template, and the sub-step-3 cross-references — those are DIAG-3 / DIAG-4 renames deferred to loom 2.0. The fix is an additive *Trigger*-cell change only.
