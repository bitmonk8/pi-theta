## Finding

The *Worked example: depth-6 forced respond at `max_rounds`* in `query-tool-loop.md` is declared normative — its own prose states "It is normative: the `RuntimeEvent`-shape conformance test and the typed-query test suite both cite this vector," and it is the canonical exhibit PIC-1 cross-references for the only non-empty `masked` payload reachable in V1 (`["ceiling#2"]`). It therefore functions as a fixed conformance vector: a test author is expected to assert the emitted `Err(QueryError {...})` and `RuntimeEvent {...}` byte-for-byte against it.

Two of the vector's fields are left as prose placeholders even though the cited spec makes them deterministic, so the example cannot serve as the conformance oracle it claims to be:

- `validation_errors[].path: "<JSON Pointer to first too-deep node>"` is deterministic for the fixed payload `{"deeply":{"nested":{"value":{"a":{"b":"x"}}}}}`. Under the counting algorithm in `schema-subset.md` (recursive descent, "the first node whose depth would exceed `5` short-circuits"), the descent reaches depth 6 at the scalar `"x"`, whose JSON Pointer is `/deeply/nested/value/a/b`. There is no degree of freedom here; leaving it as English forces every test author to re-derive (and risk diverging on) a value the spec already pins.
- `message: "<the validation message>"` (and the matching `RuntimeEvent.message`) has ambiguous normative status. `schema-subset.md` pins the canonical depth-violation message `"JSON document depth exceeds 5"` for `validation_errors[].message`, but the example neither inlines it nor marks the top-level `message` field as illustrative-only. A test author cannot tell whether the `message` string is part of the asserted vector or free prose.

The remaining placeholders in the block (`raw_response`, `query_site.{file,line,column}`, `occurred_at`) are genuinely run-dependent and are correctly left abstract; the defect is confined to the two fields above that the spec renders deterministic. The example is correctly located in `query-tool-loop.md` — it is intrinsically tool-loop-shaped (it exercises `max_rounds`, the free phase, and the forced respond turn), and PIC-1 deliberately cross-references it from `runtime-event-channel.md` as the canonical worked example; relocation is not required and would orphan the example from the tool-loop mechanics it demonstrates.

## Solution Space

**Shape:** single

### Recommendation

Make the normative vector self-contained in place (keep it in `query-tool-loop.md`):

- Replace `path: "<JSON Pointer to first too-deep node>"` with the concrete pointer `path: "/deeply/nested/value/a/b"` in both the `Err(QueryError {...})` block and (implicitly) anywhere the vector's `validation_errors` are cited. Add a half-sentence noting the pointer is the first node at depth 6 under the `schema-subset.md` counting algorithm, so the value's provenance is auditable.
- Inline the canonical message: set `validation_errors[].message: "JSON document depth exceeds 5"` (the value `schema-subset.md` pins for `maxDepth`).
- Resolve the top-level `message` field's status explicitly: either inline the asserted string or annotate it as illustrative-only ("`message` reproduces the user-facing template string and is not part of the asserted vector"). Pick one and state it, so the conformance author knows whether to assert on it.
- Leave `raw_response`, `query_site.{file,line,column}`, and `occurred_at` as abstract placeholders — they are genuinely run-dependent — but add a one-line note distinguishing the deterministic fields (asserted) from the run-dependent ones (not asserted), so the vector's testable surface is unambiguous.

Edge case for the implementer: confirm the descent semantics name the depth-6 scalar (`/deeply/nested/value/a/b`) rather than its depth-5 parent before pinning the literal; the pointer is deterministic either way, but the exact node must match the implemented short-circuit point.

## Consequence

**Severity:** correctness

The example is cited as the fixture for the `RuntimeEvent`-shape conformance test and the typed-query suite, yet two reasonable test authors will produce divergent oracles: one may hardcode `path: "/deeply/nested/value/a/b"` and `message: "JSON document depth exceeds 5"`, another may skip those fields as "illustrative." The conformance suite that is supposed to pin loom's only V1 hard-ceiling co-fire then fails to actually pin the payload it cites.
