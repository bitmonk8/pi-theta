## Finding

In `docs/spec_topics/pi-integration-contract/`, the two single-paragraph audit pages have basenames and H1 headings that contradict each other:

- `audit-recognised-shapes.md` carries H1 `# Audit target categories` and its single paragraph is the `<a id="audit-target-surface-categories"></a>*Target surface categories.*` block (plus the `strict-capability-absence-pin` sub-paragraph).
- `audit-target-categories.md` carries H1 `# Audit recognised shapes` and its single paragraph is the `<a id="audit-recognised-shapes"></a>*Recognised import/access shapes.*` block.

The mismatch compounds in two ways. First, the anchor `audit-recognised-shapes` lives inside `audit-target-categories.md`, so the link `audit-recognised-shapes.md#audit-recognised-shapes` (which a reader following naming intuition would type, and which exists in the corpus today as a sibling-file relative reference inside the same paragraph) does not resolve — the anchor is in the *other* file. Second, the index page `docs/spec_topics/pi-integration-contract.md` papers over the mismatch by listing the two pages with link-text that matches the H1s but URLs that match the filenames (`[Audit target categories](./pi-integration-contract/audit-recognised-shapes.md)` and `[Audit recognised shapes](./pi-integration-contract/audit-target-categories.md)`), which is internally consistent but means every filename-based reference (git history, IDE jump-to-definition, grep, the 21 cross-spec link sites that already cite these files by URL, plus `plan_topics/V18b-inventory-audit.md`, `V18b-T-inventory-audit.md`, `V18c-version-bump-checklist.md`, `V18c-T-version-bump-checklist.md`, and `coverage-matrix.md` row `cka-20`) lands on a page whose H1 advertises the opposite topic.

The defect is recorded a third time elsewhere in the source review (heading `audit-target-categories.md — heading "Audit recognised shapes" vs filename / sibling-file collision`) as a "three-way name collision" corroborating site; both surface the same underlying inversion and resolve together.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `f5e89f4` — *docs: enforce 100KB hard / 30KB recommended size cap on spec set* (Thomas Andersen, 2026-06-04). Split the pi-integration-contract monolith via `docs/_tools/split_spec.py`. The `PLANS` table allocates source-line 826 to slug `audit-recognised-shapes` and source-line 827 to slug `audit-target-categories`, but the content at those line positions in the pre-split monolith was the inverse: line 826 was the *Target surface categories* paragraph and line 827 was the *Recognised import/access shapes* paragraph. The script's `title_from_slug` synthesised each child's H1 from its filename, so on commit the H1s matched the filenames but neither matched the content the child page actually contained.
- `7e3e94e` — *pi-loom spec: resolve "audit-cluster files have H1 headings inverted"* (Thomas Andersen, 2026-06-05). A targeted fix that swapped the two H1 headings (and the link-text on the parent index) so each page's H1 now matched its body — but left the basenames, the surviving `<a id="audit-recognised-shapes">` anchor, and every URL-bearing cross-reference pointing at the names from the split. The fix resolved the heading↔body half of the inversion and re-pointed the heading↔filename half into the present state.

**History:** `git log --follow` shows both files first appear in `f5e89f4` (the split) and the only subsequent rewrite touching either H1 is `7e3e94e`. The defect is not isolable to either commit alone: `f5e89f4` introduced the slug↔content mismatch from the wrong line-range table; `7e3e94e` chose the swap-headings arm over the rename-files arm, locking the slug↔H1 inversion into the corpus. The durable fix addresses that arm choice rather than reverting either commit.

## Solution Space

**Shape:** single
**State:** shaped

### Recommendation

Realign each filename's H1 and body by swapping the bodies (and the owning anchor) so each child page's H1 matches its basename. This option's surface area is much smaller than a file rename: two body moves and one index-page rewrite, all confined to the `pi-integration-contract/` directory and its parent, with no inbound URL sweep, no plan-leaf Spec-field co-edits, and no tooling change.

Spec edits:
- Move the *Target surface categories* paragraph (and the `strict-capability-absence-pin` sub-paragraph) from `audit-recognised-shapes.md` into `audit-target-categories.md`, replacing its current body.
- Move the *Recognised import/access shapes* paragraph from `audit-target-categories.md` into `audit-recognised-shapes.md`, replacing its current body.
- Restore each page's H1 to match its filename (`# Audit recognised shapes` and `# Audit target categories`).
- Rewrite the `[Audit target categories]` and `[Audit recognised shapes]` links on the parent `pi-integration-contract.md` index page so link-text and URL agree.
- The anchors travel with the paragraphs that own them, so `audit-target-surface-categories`, `audit-recognised-shapes`, and `strict-capability-absence-pin` stay at the same `<file>#<anchor>` coordinates; no inbound URL or plan-leaf Spec-field edits are required.
- Re-check the forward-reference `[*Target surface categories*](./audit-recognised-shapes.md#audit-target-surface-categories)` inside the *Recognised import/access shapes* paragraph — after the swap that URL becomes `./audit-target-categories.md#audit-target-surface-categories`.
