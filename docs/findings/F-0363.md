## Finding

`V6c` performs two load-time rejections that both require knowing facts about a `.loom` callee that only the static-resolution pass produces: `loom/load/prompt-mode-callable` rejects a callee whose resolved frontmatter `mode` is prompt-mode, and `loom/load/unresolvable-loom-path` rejects a `tools:` `.loom` entry whose path does not exist or is not readable. Determining the resolved `mode` requires the callee's frontmatter to have been parsed; determining readability requires the callee's path to have been resolved and read. Neither value is something `V6c` produces itself.

`V15a` owns the single source of both: its static-resolution per-pass parse cache walks transitively across `.loom` `tools:` entries, parsing and lowering each visited file exactly once. The resolved `mode` is part of the lowered frontmatter held in that cache, and the path-resolution / readability outcome surfaces during the same walk (it is also the trigger for `V15f`'s `loom/load/callee-has-errors`, fired against the same cache). `V6c` correctly lists `V15a` in its `Deps`, but its `Adds`/`Tests` prose never states that `V6c` reads the resolved `mode` and the readability outcome *from* that cache. An implementer is left to invent a private frontmatter re-parse or an ad-hoc filesystem read to satisfy the two rejection tests.

A second parse/read at the `V6c` site violates the spec's single-pass invariant (`implementation-notes.md` §Static-resolution load pass / `invocation.md#static-resolution`: each visited file is parsed and lowered exactly once). The defect is a naming gap, not a missing capability: the seam exists in `V15a`; `V6c` just never points at it.

The readability dimension widened the same gap at 60c9400, which added the `loom/load/unresolvable-loom-path` test without naming any path-resolution / readability seam.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V6c-tools-set.md`, make the `Adds.` field state that `V6c`'s prompt-mode rejection (`loom/load/prompt-mode-callable`) and unresolvable-path rejection (`loom/load/unresolvable-loom-path`) read the resolved callee `mode` and the path-resolution / readability outcome from `V15a`'s static-resolution per-pass parse cache — the same single transitive parse/lower pass — rather than performing a private frontmatter re-parse or an ad-hoc filesystem read. Keep the existing `Deps.` on `V15a`.

In `docs/plan_topics/V15a-invocation-core.md`, confirm the static-resolution per-pass parse cache exposes, per visited `.loom` `tools:` entry, the resolved frontmatter `mode` and the path-resolution / readability outcome to load-time consumers; if that exposure is only implicit today, state it explicitly in the `Adds.` field so `V6c` has a named value to read.

Edge case for the implementer: readability failures already surface via `V15f`'s `loom/load/callee-has-errors` (error severity for a `tools:` `.loom` entry) against this same cache. `V6c`'s `unresolvable-loom-path` rejection must consume that outcome, not open a second independent filesystem read that could diverge from the `callee-has-errors` verdict.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one consults `V15a`'s static-resolution per-pass parse cache for the resolved `mode` and readability outcome, the other re-parses the callee frontmatter and re-reads the file at the `V6c` site. The latter double-parses, violating the spec's single-parse-and-lower-per-file invariant, and risks the `V6c` `unresolvable-loom-path` check disagreeing with `V15f`'s `callee-has-errors` verdict over the same callee.
