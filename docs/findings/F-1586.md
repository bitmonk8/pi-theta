## Finding

In `docs/spec_topics/pi-integration-contract/host-interfaces-core.md`, the `*Pi-owned snapshot.* …` marker paragraph appears twice in succession in the `ExtensionContext` section (lines 43 and 47), word-for-word identical. The two occurrences are separated only by the `CompactOptions` / `CompactionResult` paragraph; both attach to the same `ExtensionContext` shape block above.

The marker is the standard per-member-block normativity disclaimer used across the file: the enumerated member set is normative; each TypeScript signature is a non-normative Pi-owned snapshot re-validated on each Pi minor bump. One such marker is intended per member-pinning block — the `ExtensionCommandContext` section (line 75) and the `AgentSession` section (line 121) each carry exactly one — so the duplicate is a pure editorial defect, not a deliberate restatement. The duplication adds no constraint and slightly degrades readability of the `ExtensionContext` block.

## Issue introduction

**Verdict:** single-commit-introducing
**Introducing commits:** `15c0f17` — "pi-loom spec-plan: fix F-1284 — add uniform Pi-owned-snapshot markers to the four host-interfaces-core.md member-pinning surfaces (member set normative; TS signatures non-normative snapshot, re-validated at bump)"
**History:** `git log -L 43,50:docs/spec_topics/pi-integration-contract/host-interfaces-core.md` shows `15c0f17` as the only commit touching this range. The diff inserts the marker paragraph twice — once before the `CompactOptions`/`CompactionResult` paragraph and once after it — in a single hunk; both lines arrived together. The earlier creating commit `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap on spec set") introduced the surrounding `CompactOptions` paragraph with no marker present. The F-1284 fix was intended to add one uniform marker to each of four member-pinning surfaces in this file; the `ExtensionContext` surface received two instead of one, while the `ExtensionCommandContext` and `AgentSession` surfaces (also edited in the same commit) received exactly one each as intended. The defect is therefore a single-commit transcription error in an otherwise consistent change.

## Solution Space

**Shape:** single

### Recommendation

Delete the second `*Pi-owned snapshot.* …` paragraph (the one immediately following the `CompactOptions`/`CompactionResult` paragraph at line 47), leaving the first occurrence (line 43, directly under the `ExtensionContext` code fence) as the section's marker. This matches the one-marker-per-member-pinning-block pattern already used in the `ExtensionCommandContext` and `AgentSession` sections of the same file. No cross-references point at the duplicate, so no other edits are required.

## Consequence

**Severity:** cosmetic

No observer is affected: the duplicate paragraph is identical to its sibling and constrains nothing new. The only cost is reader friction at one of the corpus's most-consulted PIC surfaces.
