## Finding

Every other Pi-owned symbol the subagent spawn block and the **Per-loom registration** section consume by name carries a `dist/…d.ts` declaration-site pin against the [loom 1.0 Pi-SDK pin](docs/spec_topics/pi-integration-contract/host-prerequisites.md) plus a stated per-Pi-minor-bump re-validation obligation: `createAgentSession` / `CreateAgentSessionOptions` (`dist/core/sdk.d.ts`), the closed `create*ToolDefinition` factory family and `ToolName`/`allToolNames` (`dist/core/tools/*.d.ts`, `dist/core/tools/index.d.ts`), `ResourceLoader` (`dist/core/resource-loader.d.ts`) and `SessionManager.inMemory` (`dist/core/session-manager.d.ts`) under editorial-review item (o), and even `DefaultResourceLoader` / `DefaultResourceLoaderOptions` — which loom consumes neither — are folded into the same item (o) re-validation expressly "so a Pi-side reshape … is caught at the bump rather than going silently stale."

`defineTool` is the lone exception. It is named only by package — "constructed through `defineTool(...)` (exported from `@earendil-works/pi-coding-agent`)" in the `extension-bootstrap-and-per-loom.md` **Per-loom registration** note, and "loom callees wrapped via `defineTool`" in the `subagent.md` spawn-block comment and rules 1/3 — with no owning declaration file and no entry in the per-bump re-validation set. In the pinned SDK it is declared at `dist/core/extensions/types.d.ts` and re-exported through `dist/core/extensions/index.d.ts` and the package root `dist/index.d.ts`. The page already demoted `defineTool` to a non-normative implementation note ("a maintainability convention, not an observable conformance point") with a bare object literal as the explicit fallback, so the asymmetry is not a correctness hole — but against the page's own stated pin discipline (which pins even non-consumed types precisely to catch silent drift), `defineTool` is the one consumed-by-name symbol whose drift across a Pi bump would go unnoticed.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 8362824 — pi-loom spec: resolve "Demote over-prescriptive and non-testable MUSTs on unobservable implementation structure" (2026-06-04, pi-loom); 158ed09 — pi-loom spec: resolve "Missing owning-declaration pins for bare Pi SDK symbols" (2026-06-05, pi-loom); 5d55c12 — pi-loom spec: resolve "Subagent-spawn satellite types ResourceLoader and SessionManager.inMemory not pinned" (2026-06-05, pi-loom); e279e0c — pi-loom spec: resolve "built-in Pi tool name → ToolDefinition resolution names no concrete API" (2026-06-06, pi-loom)
**History:** An earlier draft pinned `defineTool` to its declaration site, but 8362824 demoted the `defineTool(...)` construction mandate to a non-normative implementation note ("exported from `@earendil-works/pi-coding-agent`", package-only), framing it as a maintainability convention. The asymmetry then opened as the subsequent declaration-site-pin sweeps — 158ed09 (bare-symbol owning-declaration pins), 5d55c12 (ResourceLoader / SessionManager satellite pins), and e279e0c (the create*ToolDefinition factory family) — added `dist/…d.ts` pins and per-bump re-validation obligations to every other consumed symbol on these pages while leaving the now-demoted `defineTool` outside the pin set. The defect is the interaction: the demotion and the later pinning discipline together single out `defineTool` as the only consumed-by-name symbol with no pin. (The two files were later split out of the predecessor PIC page by the size-cap commit f5e89f4, which moved but did not change the framing.)

## Solution Space

**Shape:** multiple

### Option A — Pin `defineTool` and fold it into the per-bump re-validation set

- **Approach:** Treat `defineTool` like the other consumed-by-name symbols. Pin it to its owning declaration (`dist/core/extensions/types.d.ts` on `@earendil-works/pi-coding-agent`, re-exported through the package root `dist/index.d.ts`) at the loom 1.0 Pi-SDK pin, and add it to the editorial-review checklist's SHOULD-level audit-and-record set — the same item (o) bucket the `ResourceLoader` / `SessionManager` / `DefaultResourceLoader` pins occupy.
- **Spec edits:** In the `extension-bootstrap-and-per-loom.md` **Per-loom registration** note, change "exported from `@earendil-works/pi-coding-agent`" to name the declaration file; add one re-validation clause (or extend item (o)) in `version-bump-step2.md`. Optionally cross-reference from the `subagent.md` spawn-block comment.
- **Pros:** Restores full symmetry with the page's stated pin discipline; drift is caught at the next bump like every sibling symbol; matches the precedent of pinning even non-consumed types (`DefaultResourceLoader`).
- **Cons:** Adds a checklist item for a symbol whose drift cannot break conformance (the object-literal fallback exists), arguably re-elevating a deliberately-demoted maintainability convention.
- **Risks:** Minimal; the SHOULD-level audit bucket is non-blocking, so a `defineTool` rename records a `fail`/`not-performed` outcome without blocking a bump.

### Option B — State that `defineTool`'s drift is immaterial and the pin omission is intentional

- **Approach:** Keep the package-only naming but add one clause making the exemption explicit: because the construction-helper choice is non-normative and a bare `ToolDefinition` object literal is an equivalent fallback, a Pi-side rename/removal of `defineTool` requires no spec change and is intentionally outside the pin / re-validation set.
- **Spec edits:** Append a single sentence to the `extension-bootstrap-and-per-loom.md` **Per-loom registration** non-normative note; no `version-bump-step2.md` change.
- **Pros:** Lowest-effort; faithful to the prior deliberate demotion; documents *why* `defineTool` is the one symbol without a pin, closing the "silent staleness" gap by declaring there is nothing to catch.
- **Cons:** Leaves a visible inconsistency with the page's otherwise-uniform pin discipline; relies on the object-literal equivalence remaining true.
- **Risks:** If a future change makes `defineTool` load-bearing (e.g. a behaviour no object literal reproduces), the exemption would need revisiting.

### Recommendation

Take Option B. The page deliberately demoted `defineTool` to a non-normative maintainability convention with an explicit object-literal equivalent, so the correct fix is to make that exemption explicit rather than re-elevate the symbol into a re-validation obligation whose `fail` outcome could never block a bump. Watch one edge case: the clause must state the object-literal fallback is genuinely equivalent at the pinned SDK, so the exemption is conditioned on that equivalence rather than asserted unconditionally.

## Consequence

**Severity:** advisory

If a future Pi minor renames, moves, or removes `defineTool`, the two spec mentions go silently stale: nothing in the per-bump re-validation checklist flags it, unlike every sibling consumed symbol on the same pages. No conformance break results — the note is non-normative and a bare `ToolDefinition` object literal is the stated equivalent — but the page's drift-detection discipline has a single unmanaged hole.
