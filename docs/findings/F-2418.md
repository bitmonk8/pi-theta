## Finding

The inventory-closure audit's wire contract pins a four-field violation record — `path`, `line`, `symbol`, `proposed-resolution` — in the *Per-family record-shape table* (`docs/spec_topics/pi-integration-contract/audit-failures.md`) and the *Wire serialisation* clause (`docs/spec_topics/pi-integration-contract/audit-wire-and-canary.md`). The wire clause goes to byte-level lengths elsewhere (tab separators, the `path:line` segment split on the rightmost `:`, the `<n/a>` sentinel, source-walk emission order, and an explicit "two runs against the same source tree emit byte-identical output" determinism claim), but it leaves four field-level encoding questions unanswered, each of which two conforming implementations would resolve differently:

- **Which physical line the `line` field carries for a multi-line reference.** The *Multi-line surface placement* paragraph defines an *originating line* per reference shape (e.g. the property-identifier line for a split `pi\n.registerCommand(...)`, the `ctx`-identifier line for a split `ctx`-rooted chain), but it defines it as *"the line a marker MUST trail to authorise that reference, and equivalently the offending surface line"* — i.e. for marker placement and clause-(e) malformed-marker detection. The *Failure-surface contract* says the record must identify *"the offending source … line"*, but it never states normatively that the emitted `line` field equals the originating line. The fixture obligation that exercises the originating-line rule is likewise framed against *"the placement rule"*, not against the emitted field value. An implementation that emits line 1 of any multi-line span into the `line` field satisfies every stated record obligation.
- **Whether `line` is 0-based or 1-based.** No clause states the numbering base for the `line` field. The gap is conspicuous because the same corpus pins the decode-error byte offset as *"the zero-based byte offset of the first invalid byte sequence"* (the *Source-file encoding* rule in `audit-resolution.md`), so the audit's other positional value is explicitly 0-based while the record's `line` value is silent.
- **The `path` field's normal form.** Neither relative-vs-absolute, the path separator, nor any canonicalisation is pinned for the emitted `path`. The wire clause's only path-form remark is an example (*"a deterministic ordering, e.g. lexicographic on POSIX-form path"*) for emission order, not a normative pin of the emitted field, and the same clause's rightmost-`:` split rule explicitly accommodates *"Windows hosts whose `path` field may itself contain a `:` in the drive prefix"* — i.e. it anticipates host-dependent absolute Windows paths in the field. That directly undercuts the byte-identical-output determinism claim across hosts and lets two implementations on the same tree emit different `path` strings.
- **The decode byte-offset has no field.** The *Source-file encoding* rule mandates that on a strict-UTF-8 decode failure the audit *"MUST report the zero-based byte offset of the first invalid byte sequence"*. A decode failure routes through *Infrastructure-failure handling* as a parse failure, whose record carries the same four fields (`path`, `line`, `symbol`, `proposed-resolution`). None of those four fields is a byte offset, and no clause says which field (or what encoding) carries the mandated offset. The reporting obligation therefore has no home in the wire format that is the sole machine-parseable surface.

These are four facets of one defect: the audit's emitted record fields are not defined precisely enough for the byte-identical, CI-parser-relied-upon wire contract the page claims. All four resolve at the same surface (the *Per-family record-shape table* plus the *Wire serialisation* and *Source-file encoding* clauses).

## Solution Space

**Shape:** single

### Recommendation

Pin the four field-encoding points in one coherent edit across the *Per-family record-shape table* / *Wire serialisation* / *Source-file encoding* clauses:

- **`line` value for multi-line references.** Add a normative sentence to the *Failure-surface contract* (or the *Per-family record-shape table* preamble) stating that the `line` field carries the *originating line* of the offending reference as defined by *Multi-line surface placement* (so the field value tracks the same per-shape rule already pinned for marker placement, and the existing per-shape fixtures double as the field-value oracle). For family-(4) shapes and family-(5) marker records, state the line each carries explicitly (the offending shape's first physical line and the marker's own line, respectively).
- **Numbering base.** Pin `line` as **1-based** (matching the conventional editor/contributor line numbering a `path:line` record is read against), and state it explicitly alongside the existing 0-based byte-offset rule so the two positional values are unambiguously distinct. (Either base is defensible; pick 1-based and pin it.)
- **`path` normal form.** Pin the emitted `path` to a host-independent canonical form — repo-relative, forward-slash (POSIX) separators — so the byte-identical-output determinism claim holds across hosts. If a drive-prefix `:` can no longer occur once paths are repo-relative POSIX-form, note that the rightmost-`:` split rule then guards only against a `:` in a filename rather than a Windows drive prefix (or simplify the split rule accordingly).
- **Decode byte-offset carrier.** Give the mandated zero-based decode byte-offset a defined home in the wire record for the parse-failure infrastructure record — e.g. carry it in the `proposed-resolution` field as a pinned plain-ASCII token (`byte-offset=<N>`), or extend the *Infrastructure-failure handling* record shape with the offset, and state the encoding explicitly. State which field, and keep it within the no-tab / printable-ASCII constraints the *Wire serialisation* clause already imposes.

Keep all four as MUSTs at the existing clauses; the edit is a small set of declarative pins co-located at the wire-record definition and does not introduce a new topic page.

## Consequence

**Severity:** correctness

Two reasonable implementations diverge on the multi-line `line` value, the 0-vs-1 numbering base, the `path` normal form (notably host-dependent on Windows), and where the mandated decode byte-offset is reported — each producing different wire records for the same source tree. This breaks the page's own byte-identical-output determinism claim across hosts and the CI-parser-may-rely-on contract, and leaves the mandated decode-offset reporting with no machine-parseable home. Implementation can proceed (the audit is a post-1.0 hardening), so this is divergence rather than a hard block.
