## Finding

`V3c` implements `if` / `for` / `while` / `break` / `continue`, but its sole Tests bullet (mirrored in `V3c-T`) is `CTRL-1` â€” the once-only iterand-snapshot semantics. The spec page `spec_topics/control-flow.md` carries five normative reject-path diagnostics for exactly the constructs `V3c` builds, every one of them registered in `spec_topics/diagnostics/code-registry-parse.md`:

- `loom/parse/non-array-iterand` â€” `for x in expr` where `expr` is not `array<T>`.
- `loom/parse/break-outside-loop` â€” `break` outside a `for` / `while` body.
- `loom/parse/continue-outside-loop` â€” `continue` outside a `for` / `while` body.
- `loom/parse/break-with-value` â€” `break expr` (loom 1.0 `break` carries no value).
- `loom/parse/non-boolean-condition` â€” a non-`boolean` `if` / `while` / ternary condition or `&&` / `||` operand (truthiness home: `expressions.md#truthiness`, owned by `V3a`, whose Adds names "boolean-only truthiness").

A repo-wide search (`grep` over `docs/plan_topics/` and `docs/plan.md`) finds none of these five codes asserted by any leaf's Tests. The gap also exists in the gate mapping: the `Code-keyed obligation areas` table in `coverage-matrix.md` has no `control-flow.md` row, so even at release the closing gate has no closing-leaf mapping for these codes to enumerate against. `V3c` therefore reaches its `Ships when` exit gate (empty-array iterand commits exactly once) with every loop/branch reject path unverified, and no leaf owns the assertions that would close them.

## Solution Space

**Shape:** multiple
