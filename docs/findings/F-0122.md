## Finding

`V3c` implements `if` / `for` / `while` / `break` / `continue`, but its sole Tests bullet (mirrored in `V3c-T`) is `CTRL-1` ã the once-only iterand-snapshot semantics. The spec page `spec_topics/control-flow.md` carries five normative reject-path diagnostics for exactly the constructs `V3c` builds, every one of them registered in `spec_topics/diagnostics/code-registry-parse.md`:

- `loom/parse/non-array-iterand` ã `for x in expr` where `expr` is not `array<T>`.
- `loom/parse/break-outside-loop` ã `break` outside a `for` / `while` body.
- `loom/parse/continue-outside-loop` ã `continue` outside a `for` / `while` body.
- `loom/parse/break-with-value` ã `break expr` (loom 1.0 `break` carries no value).
- `loom/parse/non-boolean-condition` ã a non-`boolean` `if` / `while` / ternary condition or `&&` / `||` operand (truthiness home: `expressions.md#truthiness`, owned by `V3a`, whose Adds names "boolean-only truthiness").

A repo-wide search (`grep` over `docs/plan_topics/` and `docs/plan.md`) finds none of these five codes asserted by any leaf's Tests. The gap also exists in the gate mapping: the `Code-keyed obligation areas` table in `coverage-matrix.md` has no `control-flow.md` row, so even at release the closing gate has no closing-leaf mapping for these codes to enumerate against. `V3c` therefore reaches its `Ships when` exit gate (empty-array iterand commits exactly once) with every loop/branch reject path unverified, and no leaf owns the assertions that would close them.

## Solution Space

### Recommendation

Close the five control-flow reject-path diagnostics at the leaves that build the constructs, and add the matching code-keyed coverage-matrix row so the release gate can enumerate them.

**Coverage matrix.** In `docs/plan_topics/coverage-matrix.md`, add one row to the `Code-keyed obligation areas (no numbered REQ-IDs)` table enumerating the four loop/branch parse codes ã `loom/parse/non-array-iterand`, `loom/parse/break-outside-loop`, `loom/parse/continue-outside-loop`, `loom/parse/break-with-value` ã and name `V3c` as their closing leaf. Add `loom/parse/non-boolean-condition` under the expressions/truthiness area (its registry anchor is `expressions.md`), naming `V3a` as its closing leaf.

**Leaf Tests.** In `docs/plan_topics/V3c-control-flow.md` and `docs/plan_topics/V3c-T-control-flow.md`, add one Tests bullet per loop/branch code (`loom/parse/non-array-iterand`, `loom/parse/break-outside-loop`, `loom/parse/continue-outside-loop`, `loom/parse/break-with-value`), and extend each leaf's `Ships when` to name the new firings. In `docs/plan_topics/V3a-expression-evaluator.md` and `docs/plan_topics/V3a-T-expression-evaluator.md`, add a `loom/parse/non-boolean-condition` Tests bullet ã V3a owns boolean-only truthiness, enforced uniformly across `if` / `while` / ternary / `&&` / `||` ã and extend its `Ships when` accordingly. Source every expected message string from the `code-registry-parse.md` Message column, per `conventions.md` ôÏDiagnostic message anchors.

**Edge case.** The `non-boolean-condition` assertion at `V3a` fires across all condition sites; include a non-boolean `while` condition in the test corpus so `V3c`'s loop-construct reject path is actually exercised.

## Consequence

**Severity:** correctness

Two reasonable implementers could ship a `V3c` that passes its iterand-snapshot gate yet silently parses `break` outside a loop, a non-array iterand, or a non-boolean `while` condition without firing the registered diagnostic ã the reject paths are never exercised. Because `control-flow.md` is also absent from the coverage matrix's code-keyed table, the `H5a`/`H6a` closing gate has no mapping to enumerate these codes against, so the only mechanical catch is deferred to release with no leaf owning the fix.
