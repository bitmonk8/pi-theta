## Finding

`V9b` is the single declaration site of the test-only `ReloadFailureInjector` interface and explicitly wires two of its arms itself: the registry-swap arm (`loom/runtime/registry-swap-failed`) and the `.loom`/`.warp` re-parse arm (the settings-re-merge arm is contributed by `V10d`). Its **Adds** states "`V9b` itself wires the registry-swap arm … and the `.loom`/`.warp` re-parse arm".

However, `V9b`'s **Tests** bullets (mirrored in `V9b-T`) and its **Ships when** condition only exercise the registry-swap arm: the swap atomicity, the `loom/runtime/registry-swap-failed` fire, the `getCommands()` read-only pass, the structural-change watcher-note suppression/emission, and the model-reference-matcher wiring. None of them injects a synthetic `.loom`/`.warp` re-parse failure or asserts it routes onto the `loom-system-note` surfacing path. The arm `V9b` owns and wires is therefore not confirmed by `V9b`'s own gate.

The behaviour is covered transitively: `V4g` (slice V4) imports the `ReloadFailureInjector` interface from `V9b` and exercises `ERR-7`, which drives both the registry-swap and the `.loom`/`.warp` re-parse arms together with `V10d`'s settings-re-merge arm. So no spec obligation ships untested — but the verification of an arm `V9b` wires lives in a downstream leaf that depends on `V9b`, not in `V9b`'s own `npm test`.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `66acde6` (2026-06-11) — first attaches the `.loom`/`.warp` re-parse failure to `V9b`'s reload failure-injection seam; `fdce2a2` (2026-06-12) — sharpens the wording to "`V9b` itself wires the registry-swap arm … and the `.loom`/`.warp` re-parse arm" while leaving the Tests block asserting only the registry-swap arm.
**History:** `V9b`'s Tests block has carried a registry-swap-arm assertion and no re-parse-arm assertion at every revision since the seam was introduced (verified via `git show`: zero occurrences of "re-parse" inside `V9b`'s `**Tests**` block at `66acde6` and at HEAD). The gap therefore coexisted with the arm from the moment it was assigned to `V9b`; it is not a regression introduced by a later edit.

## Solution Space

**Shape:** multiple

### Option A — In-leaf re-parse-arm assertion

**Approach.** Add a `.loom`/`.warp` re-parse failure-injection test to `V9b-T` (the failing-test owner), mirrored in `V9b`'s **Tests** and **Ships when**, paralleling the existing registry-swap arm.
**Plan edits.** In `V9b-T-registration-reload-wiring.md` and `V9b-registration-reload-wiring.md`, add a Tests bullet that injects a synthetic `.loom`/`.warp` re-parse failure through `ReloadFailureInjector.injectReloadFailure` and asserts the seam routes it onto the `loom-system-note` surfacing path without standing up a live watcher. Extend `V9b`'s **Ships when** so `npm test` proves the re-parse arm routes onto `loom-system-note` alongside the registry-swap arm.
**Spec edits.** None.
**Pros.** `V9b`'s own gate reddens if the re-parse wiring regresses; symmetric with the already-asserted registry-swap arm; satisfies the per-leaf TDD signal locally.
**Cons.** The re-parse routing is also exercised by `V4g`/ERR-7, so there is partial assertion overlap.
**Risks.** Minor — the in-leaf test must avoid standing up a live watcher (same constraint the registry-swap arm already honours).

### Option B — Cite V4g/ERR-7 in V9b's Ships-when

**Approach.** Make the existing transitive coverage explicit rather than adding a new in-leaf test.
**Plan edits.** In `V9b-registration-reload-wiring.md`'s **Ships when** (and optionally **Tests**), state that the `.loom`/`.warp` re-parse arm's routing onto `loom-system-note` is closed by `V4g` (`ERR-7`), referring to the seam by its declared name `ReloadFailureInjector`.
**Spec edits.** None.
**Pros.** No assertion duplication; records the existing coverage path.
**Cons.** `V9b`'s own `npm test` still does not redden on a re-parse-wiring regression; the verifying gate stays in a downstream leaf (`V4g`) that depends on `V9b`, so the failure surfaces later in implementation order.
**Risks.** A re-parse-wiring regression introduced at `V9b` is caught only when `V4g` is built/run.

### Recommendation

Take **Option A**: add the in-leaf re-parse-arm assertion to `V9b-T`, mirrored in `V9b`'s **Tests** and **Ships when**. `V9b` is the arm's wiring owner and its registry-swap sibling arm is already asserted in-leaf, so the re-parse arm should fail fast in the same gate rather than only at a downstream consumer. Edge cases the implementer must watch: the synthetic re-parse injection must not stand up a live watcher (mirror the registry-swap arm's no-live-watcher constraint), and the assertion must confirm routing onto the `loom-system-note` surfacing path — not an evaluation outcome.

## Consequence

**Severity:** advisory

A regression in `V9b`'s `.loom`/`.warp` re-parse wiring does not redden `V9b`'s own gate; it surfaces only later, at `V4g`'s build/run, which depends on `V9b`. The behaviour is not silently shipped (V4g/ERR-7 closes it), so this is a locality/fail-fast gap rather than a coverage hole — an implementer can still produce a working leaf, but `V9b`'s gate does not confirm an arm `V9b` itself wires, weakening the per-leaf TDD signal the plan otherwise enforces.
