## Problem

Every harness-driven test runs against the in-process Pi session double, and the double's fidelity to the real Pi host is a presupposition, not a mechanically-verified property: `H4a`'s fidelity-contract self-check asserts the four axes only *as modelled by the double*, and `H4a` states loom 1.0 "has no mechanical real-host fidelity gate." The plan's sole backstop is the manual smoke run in `real-host-smoke-gate.md`, which is "a contributor obligation, not a CI-enforced gate" — its trigger (2) ("any merge whose diff touches the four fidelity-contract axes") has no automatic diff detector, so a skipped run or an unrecognised divergence is not blocked from merging and surfaces only after the fact when the `H6a` release-gate auditor reads the commit message. The plan documents this limitation thoroughly but never states it as an *accepted* residual at the gate's decision point — that a loom-change-introduced divergence can reach `main` undetected until the `H6a` audit is described as a mechanical limitation, not as a recorded decision.

## Solution approach

In `real-host-smoke-gate.md`, record the no-mechanical-detection gap as an *accepted* loom-1.0 residual rather than leaving it as an implicit consequence: state that a loom-change-introduced double-vs-real divergence can reach `main` undetected until the `H6a` release-gate audit, with trigger-(2) contributor recognition as the only pre-merge defence. The decision point is the manual smoke-gate protocol's contributor-obligation / no-CI-enforcement clause, where the gap is currently described but never explicitly accepted.

## Solution constraints

- Out of scope: building the mechanical pre-merge backstop (the path-glob / commit-message evidence check) — this fix records the gap as an accepted residual; it does not add CI enforcement.
