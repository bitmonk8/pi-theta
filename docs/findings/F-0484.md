## Finding

`H6a` (the terminal loom 1.0 release-gate leaf) performs a single gate-machinery flip that reconfigures every closing-gate arm from its seeded-fixture footing to its live-corpus footing. `H5c`'s **Adds** explicitly enrolls itself in that flip: "its live-corpus binding flips to the live `src/**` allow-list at the loom 1.0 release gate alongside `H5a`'s live surfaces — both footings are reconfigured by the single `H6a` gate-machinery flip." So H6a is the leaf that activates H5c's broad-catch allow-list arm on the live corpus.

H6a's **Deps** field is `H5b, H7a`. It reaches the un-anchored-MUST arm (`H5e`) transitively because `H5b`'s coverage-producing **Deps** name `H5e`. But `H5c` is named in no leaf's **Deps** anywhere in the plan: it is absent from `H5b`'s coverage-producing set, absent from `H7a`'s **Deps**, and absent from H6a's own **Deps**. H5c closes no spec REQ-ID (it is a CI gate arm reconciling the `no-broad-catch` allow-list), so — exactly like `H7a`, which is pinned directly to H6a for the same reason — it is not a member of H5b's coverage-producing set and cannot be reached transitively through H5b. The result is that H6a reconfigures an arm it does not declare any dependency on.

Because H5c depends only on `H5a` (built very early), the build order holds incidentally today. But nothing forces H5c to complete before H6a: an implementer whose `H5b` and `H7a` deps are satisfied can pick and ship H6a with H5c still unbuilt, activating the release gate while the broad-catch allow-list arm is never flipped to its live-corpus footing.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** cb8737a
**History:** `H5c` was created in `cb8737a` (2026-06-12, "pi-loom plan: resolve 'Systemic leaf over-bundling across the plan corpus'"). At birth its **Adds** already declared "both footings are reconfigured by the single `H6a` gate-machinery flip", yet the same commit did not add `H5c` to `H6a`'s **Deps** (which was `H5b` at that point, becoming `H5b, H7a` one day earlier). On 2026-06-11 commit `1cab2ae` ("H6a consumes H7a's golden artifacts but no dependency edge") had added the `H7a` edge under precisely the governing principle here — a leaf that H6a reconfigures/consumes but which closes no REQ-ID, and so is not a member of `H5b`'s coverage-producing set, must be pinned directly in `H6a`'s **Deps**. `H5c`, split out the next day, matches that precedent but was overlooked. The omission has been present in `H5c` since its inception; `H6a` has never named `H5c`.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H6a-live-corpus-activation.md`:

- Add `H5c` to the **Deps.** field, so it reads `H5b, H5c, H7a` instead of `H5b, H7a`.
- Extend the Deps rationale parenthetical (the `_(This leaf depends on the warn-only canary …)_` block) to record the H5c edge alongside the existing H7a justification: `H5c` is the `no-broad-catch` allow-list closing-gate arm whose live-corpus binding the single H6a gate-machinery flip reconfigures (per H5c's **Adds**), and like `H7a` it closes no spec REQ-ID, so it is not a member of `H5b`'s coverage-producing set and is not reachable transitively through `H5b` — it is pinned directly here.

Edge cases for the implementer: `H5c` currently depends only on `H5a`, so adding this edge introduces no cycle. This edit is confined to plan files; the spec is read-only for this fix. The separate `H5d ↔ H5b` data-read relationship is unrelated and must not be altered by this change.

## Consequence

**Severity:** correctness

H6a's dependency DAG omits the very arm its single flip reconfigures, and no other leaf depends on `H5c` either, so H6a can be picked and shipped while H5c is unbuilt — activating the release gate with the broad-catch allow-list arm silently never flipped to its live-corpus footing (a release-gate arm passing vacuously). Even when the order holds incidentally (H5c deps only H5a), the missing edge is a latent reordering trap: two reasonable implementers diverge on whether H5c must precede H6a.
