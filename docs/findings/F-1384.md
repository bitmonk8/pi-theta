## Problem

BNDR-4 (`integer` canonical decimal form) and BNDR-5 (`number` shortest round-tripping fixed-point form) are defined only as bullets inside the binder Echo policy *Format rules* list in `docs/spec_topics/binder/defaulting-system-note-echo.md`, a section scoped to the bind-echo one-line system note. Two non-echo subsystems consume the same algorithm normatively: the canonical schema hash recipe in `schema-subset.md` and the `${expr}` / `system:` interpolation table in `query-escapes-stringification.md`. The defining site carries no signal that the rules are shared, so a future Echo-policy edit narrowing BNDR-4 / BNDR-5 to echo-specific precision would silently invalidate canonical schema slugs and prompt-rendering bytes. The defect is forward editorial-drift risk: today the three consumers agree because they all cite the same anchors.

## Issue introduction

**Verdict:** multi-commit-interaction (placement-defect-on-introduction; multi-consumer dependency layered on later)

**Introducing commits:**
- `e3ddb6f` (2026-05-08, "pi-loom spec: resolve \"Bind-echo formatter: no rendering rule for scalar non-string bound values\"") — introduced the `integer` and `number` rendering bullets directly into the binder Echo policy *Format rules* list. The rules were echo-only at this point and the placement was locally correct.
- `c5bb5e1` (2026-06-03, "pi-loom spec: resolve \"Canonical schema hash numeric serialization underspecified\"") — wired `docs/spec_topics/schema-subset.md` step 2 of the canonical-hash recipe to consume the binder-echo `integer` / `number` rendering algorithm, creating the first cross-subsystem dependency on the rules at their echo-only address.
- `a9c5d7b` (2026-06-05, "pi-loom spec: resolve \"number row omits scientific-notation and -0 pins\"") — repointed the `integer` and `number` rows of `docs/spec_topics/query/query-escapes-stringification.md`'s stringification table to BNDR-4 / BNDR-5, creating the second cross-subsystem dependency.
- `0bdd6ea` (2026-06-04, "pi-loom spec-review: park T06 …") — promoted the bullets to their BNDR-4 / BNDR-5 / BNDR-6 anchors, providing the stable IDs the later cross-references key against; it did not relocate the rules.

**History:** The rules were authored in May at their current echo-internal site to fix a then-local "no rule for scalar non-string bound values" gap (`e3ddb6f`). Anchors and IDs were added (`0bdd6ea`) without relocation. The placement defect — "single-site authority shared across three subsystems" — crystallised across two June commits that wired the canonical-hash recipe (`c5bb5e1`, 2026-06-03) and then query interpolation (`a9c5d7b`, 2026-06-05) into the binder-anchored bullets. No single commit introduced the defect; the durable fix is the *placement / shared-ownership* arrangement, not a revert.

## Solution approach

Restructure `defaulting-system-note-echo.md`'s Echo policy *Format rules* list so the `bndr-4` / `bndr-5` bullets and the BNDR-5 *Derivation recipe* are enclosed in a labelled shared-algorithm subsection that names the algorithm as shared and enumerates its three consumers by anchor — the bind-echo system note, `schema-subset.md#canonical-schema-hash`, and `query-escapes-stringification.md#stringification-of-interpolated-values`. Keep the `bndr-4` / `bndr-5` / `bndr-6` anchors at their current site so existing inbound cross-references continue to resolve. The *Format rules* slots vacated by the two numeric bullets become on-page forward references into the new subsection.

## Solution constraints

- Out of scope: `query-escapes-stringification.md` and `schema-subset.md` — their existing cross-references already resolve to the BNDR-4 / BNDR-5 / BNDR-6 anchors and must not be edited.
