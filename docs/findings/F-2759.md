## Finding

`V5f` (Schema lowering and canonical hash) requires the canonical hash to be "SHA-256 over the keys-sorted, whitespace-free, binder-number-rendered canonical form" (Tests bullet) but states nowhere where the SHA-256 implementation comes from, nor whether it must be routed through a DI seam. No `V8*` seam owns a hashing primitive, no Deps edge points at one, and `conventions.md` is silent on the digest.

The plan's *No globals, statics, singletons* discipline is identifier-keyed: the `H3a` scan bans direct ambient references only to `process.env`, `process.cwd`, `crypto.randomUUID`, and the enumerated timing primitives, and the sibling non-determinism source `crypto.randomUUID` is explicitly seamed (`V8d` `IdSource`, PIC-20). `crypto.createHash` is not in the banned enumeration, so a direct `node:crypto` SHA-256 call in the V5f lowering module would pass every mechanical gate untouched — yet the visible precedent of seaming `crypto.randomUUID` invites an implementer to build a `Hasher` seam instead.

Two good-faith implementers therefore diverge architecturally: one writes a direct `node:crypto` `createHash` call, the other constructs a DI-injected hashing seam (and looks, fruitlessly, for the `V8*` leaf and Deps edge that should own it). Both produce byte-identical, correct hashes — the divergence is in seam discipline, not in observable output — but the leaf gives no signal which posture is intended.

## Consequence

**Severity:** advisory

An implementer cannot tell whether to call `node:crypto` directly or to build and depend on a hashing seam; the two readings yield materially different module architectures even though both compute the same hash. Because `crypto.createHash` is outside the identifier-keyed ambient ban, a direct call ships unflagged, so the ambiguity surfaces as inconsistent DI posture rather than a build break.

## Solution Space

**Shape:** single

### Recommendation

State in `V5f` that the SHA-256 digest over the canonical form is a deterministic, stateless primitive that is **not** within the no-globals ambient-primitive ban and needs no DI seam and no `V8*` Deps edge. Add this disposition to `V5f`'s Adds. (or to the canonical-hash Tests bullet), naming the source concretely as a direct `node:crypto` hash computation and giving the reason it is exempt: unlike `crypto.randomUUID`, a hash introduces no non-determinism and no cross-invocation state, so the seam discipline that governs `IdSource` does not apply.

Anchor the exemption to the mechanism that already makes it safe: the `H3a` identifier-keyed scan bans `crypto.randomUUID` (and `process.env` / `process.cwd` / the timing set) but not `crypto.createHash`, so a direct hash call passes the gate without an `// allow-ambient:` comment and requires no new exempt site. Do not introduce a `Hasher` seam or a `V8*` owning leaf — there is no determinism or test-control benefit to justify the extra seam, and the precedent for seaming exists only for the genuinely non-deterministic `crypto.randomUUID`.

Optionally mirror the one-line disposition into the `conventions.md` *No globals, statics, singletons* ambient-primitive enumeration so the "deterministic stateless digest is not a banned ambient" carve-out is discoverable from the rule itself, not only from the leaf.
