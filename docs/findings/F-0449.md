## Finding

The *Doc updates* rule in `conventions.md` obligates each leaf to write a `README.md` status-table row and a dated `CHANGELOG.md` line (and, for non-plan discoveries, a `notes.md` entry). That rule carries **no mechanical gate** — it is contributor discipline verified at exactly two places: the §Per-phase TDD ritual self-review step (which "checks the per-leaf doc artifacts were written") and the loom 1.0 release-time residue inspection item 4.

Both verification surfaces test only **existence and well-formedness**. Item 4's "Clean when" reads "every completed leaf has its required doc artifacts present and well-formed"; the self-review check confirms only that the artifacts "were written." Neither asserts that the documented status, date, or notes describe the behaviour the leaf actually shipped. A leaf can therefore write a status row that misstates its state, a CHANGELOG line whose one-line summary diverges from what was implemented, or a row carried forward stale after later work changed the leaf — and pass both the per-leaf self-review and the terminal release-time backstop.

This is the same existence-vs-content shape the plan already recognises and closes for other residue classes: item 5's "Clean when" requires the reviewer to confirm a named closing leaf "carries a test asserting the obligation's MUST/MUST-NOT, not merely that the row names a leaf," and item 7 requires each facet to carry "its own distinct facet-naming citing test." Item 4 stops at presence, leaving documentation content unverified by any surface.

## Issue introduction

**Verdict:** present-since-inception (of the doc-update verification surfaces)
**Introducing commits:** `c06749d` ("resolve 'Doc-updates convention unenforced / mislabeled Tests bullets'", 2026-06-11) — introduced the self-review existence check and the "no mechanical gate / contributor discipline" enforcement posture; `e94ca10` ("resolve 'GOV-15 release-time reviewer inspection has no enumerated residue checklist'", 2026-06-16) — introduced residue-inspection item 4 with its existence-and-well-formedness "Clean when".
**History:** `git log -S`/`-G` over `conventions.md` shows the phrases "Doc-update artifacts", "present and well-formed", and "every completed leaf has its required doc artifacts" all entered in `e94ca10`, and "checks the per-leaf doc artifacts were written" entered in `c06749d`. A content-accuracy cross-check ("describe the behaviour actually shipped") never appears in any commit. Both verification surfaces were therefore authored existence-only from the moment each was introduced; the gap is not a regression from a previously-correct state but an obligation realized across the two commits that established doc-update verification.

## Solution Space

**Shape:** single

### Recommendation

Strengthen item 4's "Clean when" in `conventions.md` §Loom 1.0 release-time residue inspection so the reviewer additionally confirms, for each completed leaf, that the `README.md` status row and the `CHANGELOG.md` line describe the behaviour the leaf actually shipped — cross-checked against that leaf's *Ships when* condition — rather than merely confirming the artifacts are present and well-formed. This aligns item 4 with the content-checking discipline items 5 and 7 already carry.

The existing presence/well-formedness condition stays; the content-accuracy cross-check is added to it. The check remains a manual reviewer step (consistent with the *Doc updates* rule's "no mechanical gate" posture — do not introduce a lint, architectural test, or closing-gate check, since README/CHANGELOG/notes.md sit at the repository root outside the `src/**` and `docs/` scopes those gates read).

Edge case for the implementer: a `notes.md` entry has no fixed schema and no per-leaf "shipped behaviour" to cross-check against, so the content-accuracy obligation applies to the status row and CHANGELOG line; the `notes.md` entry stays existence-checked.

## Consequence

**Severity:** advisory

A completed leaf's `README.md` status row or `CHANGELOG.md` line can describe behaviour that diverges from what the leaf actually shipped — wrong status, mismatched summary, or a row left stale after later work — and both verification surfaces pass on existence alone, so the project documentation can silently misdescribe the shipped product through the loom 1.0 release. The artifacts are non-load-bearing (explicitly "not the pickability source"), so no gate or pickability decision is corrupted; the degradation is confined to documentation fidelity.
