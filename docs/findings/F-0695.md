## Finding

In `docs/spec_topics/expressions.md` the `string` stdlib row for `replace(from, to)` specifies "Replaces all occurrences via a single left-to-right, non-overlapping scan … (matching host `String.prototype.replaceAll`)." The spec's own clarifying parenthetical names the host method whose semantics actually match — `replaceAll`, not `replace`. JavaScript's `String.prototype.replace(str, str)` replaces only the **first** occurrence; this is the dominant prior art any author writing loom expressions will carry into the language.

The result is a silent semantic divergence on a familiar name. A loom author writing `s.replace("X", "Y")` expecting first-only substitution (the JS default for the literal-string overload) instead gets every occurrence replaced. The mismatch produces wrong values rather than a parse-time or type-time diagnostic, and the surrounding prose makes the divergence pointedly visible — the spec already concedes the behaviour is `String.prototype.replaceAll`, then attaches that behaviour to a different name.

The five normative reference vectors (`"aXbXc".replace("X", "[$&]")` → `"a[$&]b[$&]c"`, etc.) bake the all-occurrences semantics into conformance; an implementation that mapped `replace` to JS `String.prototype.replace` first-only behaviour would fail the vectors. The defect is therefore the surface name, not the behaviour: the method's behaviour is well-pinned, but its identifier carries the wrong host-language association.

## Solution Space

**Shape:** single
**State:** shaped

### Recommendation

Treat the divergence as a naming bug and align the loom method name with the host method whose semantics it already matches: rename to `replaceAll(from, to)`.

Spec edits:
- In `docs/spec_topics/expressions.md`, rename the row, the reference-vector table caption, the five vector expressions, and the paragraph that follows the vectors to use `replaceAll`. Drop the now-redundant "(matching host `String.prototype.replaceAll`)" parenthetical or shorten it to a JS-semantics note like the sibling rows.
- In `docs/spec_topics/hard-ceilings/ceiling-invariants-and-audit.md`, replace "`split` / `replace`" with "`split` / `replaceAll`".
- Update the V3f / V3f-T leaf prose ("`String.replace` reference vectors", "the five `replace` vectors") and the anchor target if the heading shifts.

This removes the JS-collision trap entirely: readers' JS instincts now align with loom behaviour and no permanent caveat is needed in the spec text. The rename is mechanical (one method on one row, five reference-vector expressions, two plan leaves, one cross-reference in `ceiling-invariants-and-audit.md`), the reference vectors already pin the behaviour so no semantic drift is possible, and reserving `replaceAll` for the operation that actually `replaceAll`s aligns loom with the JS prior art every author already carries. Watch two edge cases when applying: the section heading anchor (`#built-in-methods-and-properties`) is stable so internal cross-references survive, but the `V3f` / `V3f-T` link text and `Ships when` strings reference the old name verbatim and must be updated in lockstep with the spec edit.

## Consequence

**Severity:** correctness

Authors writing loom expressions silently produce wrong output when they reach for the JS-familiar single-occurrence `replace` and get all-occurrences instead. The error surfaces only at runtime (or in author-written tests), not at parse or type-check; nothing in the lexer, grammar, or type system constrains the receiver-string contents enough to catch the mistake. The conformance vectors and host cross-reference make the semantics unambiguous for implementers, so the cost falls on every loom author rather than diverging implementations.
