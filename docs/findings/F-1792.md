## Finding

`docs/spec_topics/imports.md` says, in the **Visibility** paragraph:

> Every top-level `schema`, `enum`, and `fn` in a `.warp` file is implicitly exported. **There is no `export` keyword on declarations and no privacy modifier**; `.warp` files have no internal-only symbols in loom 1.0.

Read in isolation that sentence asserts the language has no `export` keyword at all. But the same page uses `export` as a keyword in three other places:

1. The permitted-top-level-forms list one paragraph above: *"Top-level may contain only `import`, `export`, `schema`, `enum`, and `fn` declarations."*
2. The **Re-exports** section's two code samples: `export { Author } from "./personas.warp"` and `export { Author as Reviewer } from "./personas.warp"`.
3. The **Unknown imported symbol** paragraph's normative rule, which talks about `export { Foo } from` and `export { Foo as Bar } from` specifiers and uses the phrase "transitive re-export (`export ŌĆ” from`)".

A reader who hits **Visibility** before reaching **Re-exports** ŌĆö the natural reading order down the page ŌĆö has been told there is no such keyword and may reasonably reject the re-export examples that appear immediately after it as inconsistent, or assume the re-export syntax is a typo. The intended distinction (the keyword is forbidden on a *declaration* but required on a *re-export specifier*) is recoverable from context, but it is not stated.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 53f5831 ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03)
**History:** The Visibility paragraph ("There is no `export` keyword on declarations ŌĆ”"), the Re-exports section with its `export { Author } from "./personas.warp"` examples, and the permitted-top-level-forms bullet listing `export` as one of the allowed top-level forms were all introduced together in 53f5831 ŌĆö the commit that first added the `.warp` import surface to the then-monolithic `spec.md`. Subsequent commits to `docs/spec_topics/imports.md` (split-out in fecb504; later `enum` admission in b0b9a14; `loom 1.0` rename in a6d82db) only adjusted incidental tokens in the Visibility sentence and never touched the `export`-keyword wording. The contradictory framing has therefore been the only framing the spec has ever shipped for this surface.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the **Visibility** paragraph so its denial of `export` is scoped to declarations and explicitly forwards to **Re-exports** for the form that does use the keyword. Replace the current sentence:

> Every top-level `schema`, `enum`, and `fn` in a `.warp` file is implicitly exported. There is no `export` keyword on declarations and no privacy modifier; `.warp` files have no internal-only symbols in loom 1.0.

with prose along these lines (exact wording at the fixer's discretion, provided the three pieces below are all present):

> Every top-level `schema`, `enum`, and `fn` in a `.warp` file is implicitly exported, and there is no privacy modifier ŌĆö `.warp` files have no internal-only symbols in loom 1.0. The `export` keyword therefore never appears **on a declaration**; its only use is the `export { X } from "<path>"` re-export specifier described in [Re-exports](#re-exports) below.

Edge cases the fixer must preserve:

- The "implicitly exported" claim and the "no privacy modifier / no internal-only symbols" claim must both remain ŌĆö they are the normative content of the paragraph and other surfaces (e.g. `V15i`'s test list) already cite them.
- The forward reference must point at the **Re-exports** section on the same page; do not add a new anchor scheme. The existing section heading is already addressable.
- Do not touch the permitted-top-level-forms bullet (`import`, `export`, `schema`, `enum`, `fn`) ŌĆö once the Visibility paragraph is scoped to declarations, that bullet is consistent on its own.
- Do not introduce a privacy modifier or any new keyword; the fix is purely editorial.

## Consequence

**Severity:** advisory

A careful implementer reading the whole page will reconcile the two paragraphs and arrive at the right rule (no `export` on declarations; required on `export { X } from` re-export specifiers). A skim-reader, or one who lands on **Visibility** via an anchor, may treat the re-export examples as inconsistent and either omit `export ŌĆ” from` parsing or implement a privacy modifier they think the page bans wholesale. No observable runtime divergence between two compliant implementations follows, but the spec's first claim about `export` flatly contradicts its later examples.
