## Finding

The Number-literals paragraph in `docs/spec_topics/lexical.md` justifies rejection of `-12345678901234567890` with the parenthetical "the unary `-` does not bring the magnitude back into range". Read literally, that wording is incoherent: applying unary `-` never *changes* a token's magnitude, so it cannot "bring it back into range" any more than it could push it out. The framing implies a reader might expect the sign operator to reduce magnitude, which conflates two distinct concepts (sign vs. magnitude) the rest of the paragraph is careful to separate.

The point the sentence is actually trying to make is structural, not arithmetic: the range check is **per-token**, run by the lexer before the parser folds the unary `-`. So `12345678901234567890` exceeds `2^53 - 1` as a lexed token and is rejected, and there is no later "signed-value-fits" rescue path. That is the rule the surrounding text has just established ("The magnitude of a number literal is judged on each lexed token individually, before the parse-time unary-`-` fold"); the parenthetical should restate the per-token framing, not invoke a magnitude-shrinking action that does not exist.

The defect is purely editorial — the rule itself is sound and the diagnostic code (`loom/parse/integer-literal-out-of-range`) is unambiguous — but the phrasing invites a reader to look for a magnitude-reduction mechanism that the spec does not (and cannot) define.

## Solution Space

**Shape:** single

### Recommendation

Replace the parenthetical so it restates the per-token framing instead of invoking a non-existent magnitude-reduction effect of unary `-`. In `docs/spec_topics/lexical.md`, change

> Because the bound is per-token, `-12345678901234567890` is rejected — the unary `-` does not bring the magnitude back into range — whereas an out-of-range value reached only through arithmetic on in-range literals is governed by the runtime `number` rules in [Expressions](./expressions.md), not by this parse-time rule.

to

> Because the bound is per-token, `-12345678901234567890` is rejected — the lexed token `12345678901234567890` already exceeds the bound, and the parse-time unary `-` fold runs after the range check, so applying it cannot rescue the token — whereas an out-of-range value reached only through arithmetic on in-range literals is governed by the runtime `number` rules in [Expressions](./expressions.md), not by this parse-time rule.

Edge cases the implementer must watch:

- Keep "lexed token" / "parse-time unary-`-` fold" / "per-token" terminology aligned with the sentence two clauses earlier ("The magnitude of a number literal is judged on each lexed token individually, before the parse-time unary-`-` fold"); do not invent a new synonym here.
- Do not delete the worked example `-12345678901234567890` — it is the only callsite that pins the lexer-vs-parser ordering, and `V1b-T`'s `loom/parse/integer-literal-out-of-range` test exists precisely to exercise that ordering.
- Leave the arithmetic-vs-literal contrast (the "whereas …" tail) untouched; it is doing independent work pointing readers at the runtime `number` rules.

## Consequence

**Severity:** cosmetic

Implementers reading the surrounding sentences (the per-token rule and the diagnostic codes) will produce the right lexer behaviour regardless of the parenthetical. The cost is reader confusion at the example, not divergent implementations.
