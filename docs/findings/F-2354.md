## Finding

`V17c`'s loop-iteration macrotask-yield bullet (present in both the implementation leaf and its paired `-T` tests file) states the single-macrotask abort-flip behaviour as bare fact: the loop-iteration checkpoint "yields one macrotask turn before reading the signal, so a Pi-dispatched abort (a macrotask) lands without a genuine `await` in the loop body." The `V17c` Ships-when restates the same property as an unconditional acceptance criterion.

In the spec, the sufficiency of that single yielded macrotask is **not** a guarantee — it is presupposition (vi) of the *Pi-side slash-handler promise lifecycle (consumption posture)* block in `host-prerequisites.md`: the loom presupposes Pi flips `loomAbort.signal.aborted` within a single macrotask, and a future Pi that deferred the flip across multiple macrotasks could let an Esc during a compute-bound loop be missed at the next iteration. The spec routes the per-bump re-check of (vi) to editorial-review checklist item (ad) (`version-bump-step2.md#bump-checklist-single-macrotask-abort-flip`). The `V17c` deterministic `Checkpoint`-seam test exercises only the loom-side yield against `FakeClock`/`V8a`; it does not — and cannot — exercise real Pi abort-flip timing.

The leaf therefore presents an unpinned host-timing presupposition as a proven loom-side fact, with no pointer to (vi) or (ad). The corpus already has an established precedent for the opposite handling: `V14b`'s Adds explicitly flags its concurrent/settle-all parallel-tool-batch behaviour as "a real-Pi parallel-tool-mode consumption posture, not a loom-side guarantee … an unpinned host presupposition whose detection routes to editorial review per version-bump checklist item (an)." `V17c` is the parallel case for cancellation and omits the equivalent flag.

## Solution Space

**Shape:** single

### Recommendation

At the loop-iteration macrotask-yield bullet in `V17c-checkpoint-granularity.md` (Adds and Tests) and its verbatim duplicate in `V17c-T-checkpoint-granularity.md` (Tests), surface the single-macrotask sufficiency as an unpinned loom-side consumption posture rather than an unconditional fact, and cross-reference the existing spec presupposition and its bump-checklist audit route:

- presupposition (vi) of the *Pi-side slash-handler promise lifecycle (consumption posture)* block — `host-prerequisites.md#pi-slash-handler-promise-lifecycle-presupposition`;
- version-bump editorial-review checklist item (ad) — `version-bump-step2.md#bump-checklist-single-macrotask-abort-flip`.

State that the deterministic `Checkpoint`-seam test witnesses only the loom-side yield (one macrotask turn before the signal-check) and that the sufficiency of that single turn depends on Pi flipping `loomAbort.signal.aborted` within one macrotask — an unpinned host-timing presupposition re-checked per bump at item (ad), not a property this leaf's green test proves. This mirrors the handling already in `V14b`'s Adds for the parallel-tool-batch presupposition routed to item (an).

Do **not** author a new presupposition or a new checklist item — presupposition (vi) and item (ad) already exist in the spec; the fix is purely to reference them. The spec pages are read-only for this fix.

## Consequence

**Severity:** advisory

An implementer reading `V17c` sees the single-macrotask sufficiency as an unconditional fact, with no signal that it rests on an unpinned Pi abort-flip-timing presupposition audited only at version-bump item (ad). The deterministic test passes regardless, so a working leaf still ships; the gap is that a future Pi minor deferring the abort flip across multiple macrotasks would silently break Esc-during-compute-bound-loop cancellation without the leaf ever connecting its green test to the named presupposition (vi)/(ad) that should have caught it.
