## Finding

PIC-50/51 derive the synthesised `TransportError.provider` for a prompt-mode driven turn from `ctx.model` — the `Api`-shaped `Model<Api>.api` value of the *user session's currently-selected* model. The rule rests on an explicit behavioural claim about Pi: that the driven user turn "actually routes through" `ctx.model`, and that the loom's own resolved `model:` (its off-session / subagent model) "does not govern the user session's provider." This is a presupposition about Pi's request routing, not a property the SDK type surface encodes — `ctx.model` is typed only as `Model<any> | undefined`, which says nothing about whether the turn `pi.sendUserMessage` drives is dispatched against that model.

The claim is asserted inline in PIC-50 prose but has no corresponding consumption-posture paragraph and no item in the (a)–(ap) *Editorial-review checklist for unpinned host presuppositions* (`version-bump-step2.md`). This is conspicuous against the sibling presuppositions on the same page, each of which the spec's own discipline pins with a dedicated consumption-posture paragraph plus a checklist item: driven-turn session-commit ordering → item (ac), user-visible streaming ordering → item (ak), and `waitForIdle()` no-rejection → item (al). The routing claim is the one behavioural presupposition introduced for the `ctx.model`-based derivation that was left without that treatment.

The checklist item that is closest, (m) (*Prompt-mode and subagent-mode transport-failure `AssistantMessage` fields*), audits only the `stopReason` / `errorMessage` field surface — it does not cover which model the user turn routes through, and the `ctx.model` member-surface re-validation under `host-interfaces-core.md` only confirms the field's type shape, not its routing semantics. Consequently a Pi minor that changed user-session routing (for example so the loom's `model:` could influence the user turn's provider, or so `ctx.model` no longer tracked the model the turn was dispatched against) would make `provider` attribution wrong on the always-log `kind: "transport"` emission with no build-time SDK surface-inventory signal and no editorial-review prompt.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `6f38d17` — "pi-loom spec-plan: fix F-1145 — derive prompt-mode TransportError.provider from ctx.model + sentinel" (2026-06-25)
**History:** `git log --follow` / `git show` on `conversation-drive.md` shows that before `6f38d17` PIC-50/51 sourced `provider` from "the loom's resolved `model:`", which carried no routing presupposition (the loom's own config needs none). Commit `6f38d17` switched the derivation to the user-session `ctx.model` and added the routing claim ("the model the driven user turn actually routes through, **not** the loom's resolved `model:`"), introducing the behavioural presupposition. The same commit touched only `spec-plan-review.md`, `queryerror-variants.md`, and `conversation-drive.md` — it did not add a consumption-posture paragraph or a `version-bump-step2.md` checklist item, so the presupposition entered unpinned at that commit. The sibling consumption-posture paragraphs and their checklist items (e.g. item (al) at commit `1ca13ec`) predate or are independent of it; the defect is the omission in `6f38d17`, not a multi-commit interaction.

## Solution Space

**Shape:** single

### Recommendation

Confirm against the pinned `@earendil-works/pi-coding-agent` SDK that the prompt-mode driven turn (`pi.sendUserMessage` on the user session) is dispatched against `ctx.model`, then pin the presupposition with the two-part edit the sibling presuppositions on this page already use:

1. Add a *consumption-posture* paragraph to `conversation-drive.md`, anchored (e.g. `#driven-turn-provider-routing-presupposition`) and placed alongside the existing driven-turn / streaming / `waitForIdle` posture paragraphs. It states that PIC-50/51's `provider` attribution presupposes the driven user turn routes through `ctx.model`'s provider and that the loom's resolved `model:` does not govern the user session, that `@earendil-works/pi-coding-agent` encodes no typed contract for this routing, and that a Pi minor re-routing the user turn would mis-attribute `provider` with no SDK surface-inventory signal — routing detection to editorial review.
2. Add the matching SHOULD-level item to the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md` (the next free letter, `(aq)`, after the current `(ap)`), cross-linking the new posture paragraph and following the established `*title* — link above. The contributor SHOULD confirm …` wording and per-item outcome-recording shape. Update the checklist's introductory `(f) through (ap)` range references and the V18c/V18c-T acceptance enumeration to include the new item.

Edge case to keep intact while editing: the `ctx.model === undefined` → `provider: "unknown"` sentinel rule already in PIC-50 is orthogonal to the routing presupposition and must remain unchanged.

## Consequence

**Severity:** advisory

At the current Pi-SDK pin the derivation is well-defined — two implementers both read `ctx.model.api` and produce identical attribution — so nothing breaks today. The gap is bump-time detection robustness: an unpinned routing presupposition with no checklist item means a future Pi minor that re-routed the user turn could silently mis-attribute `TransportError.provider` on the always-log transport emission, with no build-time or editorial signal — exactly the failure mode the sibling presuppositions on this page are pinned to prevent.
