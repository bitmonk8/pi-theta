## Problem

PIC-50/51 derive the synthesised prompt-mode `TransportError.provider` from `ctx.model` ŌĆö the user session's currently-selected model ŌĆö resting on an unpinned behavioural claim about Pi: that the driven user turn routes through `ctx.model` and that the loom's resolved `model:` does not govern the user session's provider. The SDK types `ctx.model` only as `Model<any> | undefined`, encoding no contract for this routing. Unlike the sibling presuppositions on the same page ŌĆö driven-turn commit-ordering (checklist item (ac)), user-visible streaming (item (ak)), and `waitForIdle()` no-rejection (item (al)), each pinned with a consumption-posture paragraph plus a checklist item ŌĆö this routing claim has neither; the closest item (m) audits only the `stopReason`/`errorMessage` field surface, not routing. A Pi minor that re-routed the user turn would mis-attribute `provider` on the always-log `kind: "transport"` emission with no build-time SDK surface-inventory signal and no editorial-review prompt.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `6f38d17` ŌĆö "pi-loom spec-plan: fix F-1145 ŌĆö derive prompt-mode TransportError.provider from ctx.model + sentinel" (2026-06-25)
**History:** `git log --follow` / `git show` on `conversation-drive.md` shows that before `6f38d17` PIC-50/51 sourced `provider` from "the loom's resolved `model:`", which carried no routing presupposition (the loom's own config needs none). Commit `6f38d17` switched the derivation to the user-session `ctx.model` and added the routing claim ("the model the driven user turn actually routes through, **not** the loom's resolved `model:`"), introducing the behavioural presupposition. The same commit touched only `spec-plan-review.md`, `queryerror-variants.md`, and `conversation-drive.md` ŌĆö it did not add a consumption-posture paragraph or a `version-bump-step2.md` checklist item, so the presupposition entered unpinned at that commit. The sibling consumption-posture paragraphs and their checklist items (e.g. item (al) at commit `1ca13ec`) predate or are independent of it; the defect is the omission in `6f38d17`, not a multi-commit interaction.

## Solution approach

Pin the routing presupposition with the two-part treatment the sibling presuppositions on this page already use. Add a consumption-posture paragraph to `conversation-drive.md`, carrying a stable anchor and placed alongside the existing driven-turn / streaming / `waitForIdle` posture paragraphs, recording that PIC-50/51's `provider` attribution presupposes the driven user turn routes through `ctx.model`'s provider while the loom's resolved `model:` does not govern the user session, that the SDK encodes no typed contract for this routing, and routing detection to editorial review. Add the matching SHOULD-level item to the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md`, forward-linking the new posture paragraph. Update the checklist's `(f) through (ap)` range reference and the V18c/V18c-T acceptance enumeration to include the new item.
