## Problem

The first Tests bullet in both `V13a` and `V13a-T` reads "the trim → dedent order reproduces the eight normative vectors; illegal-escape and unterminated-template diagnostics fire." Both leaves are scoped to the `@`-template query construct, where the authoritative escape diagnostic is `loom/parse/illegal-template-escape`. The bare name `illegal-escape` is not a shortened form of that code — `loom/parse/illegal-escape` is a separate real registry code that fires for a backslash escape inside a regular string literal and is owned by `V1b`. The companion `unterminated-template` is correctly named but cited without its `loom/parse/` prefix, so neither code resolves unambiguously from the plan alone; an implementer following the bullet literally asserts the wrong diagnostic and the `V13a-T` test fails red for the wrong reason.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (`pi-loom plan: build/update plan for spec.md + review`, 2026-06-10)
**History:** `git log -S 'illegal-escape and unterminated-template'` over both `V13a-query-render.md` and `V13a-T-query-render.md` returns only c6a664e, the initial plan-build commit. The Tests bullet has carried the `illegal-escape` name and the prefix-less `unterminated-template` since the leaf pair was first authored; no later edit touched the diagnostic names.

## Solution approach

In the first Tests bullet of both `V13a-query-render.md` and `V13a-T-query-render.md`, rename the cited diagnostic codes to `loom/parse/illegal-template-escape` and `loom/parse/unterminated-template`. Do not substitute the string-literal code `loom/parse/illegal-escape`.

## Solution constraints

- Keep the first Tests bullet byte-identical between `V13a` and `V13a-T` (the `-T` suite asserts what `V13a` implements).
- Out of scope: `loom/parse/illegal-escape` and any `V1b` content; the spec registry is read-only for this fix.
