## Finding

The V1 diagnostic registry is sharded across four sibling pages (`code-registry-parse.md`, `code-registry-load.md`, `code-registry-runtime.md`, `code-registry-host.md`). The intro paragraph on `code-registry-parse.md` defines the normative column legend — *Sev*, *Phase* (closed taxonomy `lex` / `parse` / `type` / `load` / `runtime`), *Trigger*, *Spec rule*, *Hint*, *Message* — and asserts "the column legend below applies to all four tables." The legend never leaves this page. A reader landing on a continuation page (the common entry path, since cross-references link directly to specific rows) sees columns whose semantics are defined elsewhere, with no in-page pointer. `diagnostic-shape.md` §DIAG-4 is keyed to "the *Message* column" — but DIAG-4's home page hosts none of the tables, and the column it names is defined on a third page.

The governance gap has produced concrete header drift on the `loom/host/*` page: `code-registry-host.md` declares `Code | Severity | Phase | Trigger | Anchor | Message` — *Sev* spelled *Severity*, *Spec rule* renamed *Anchor*, *Hint* omitted. Two of those three contradict the legend directly (which pins *Sev* and *Spec rule* for every table), so a deep-link reader hits columns whose names do not match the schema the legend governs, and DIAG-4's "Message column" referent has no on-page anchor for the host namespace. The missing *Hint* column is defensible (the four host rows route via `console.error` to operators and carry no author-facing hint) but is an unauthorised editorial choice. The runtime page carries a parallel lower-severity drift (`Message template` vs `Message`); handled separately, it exists for the same reason — the legend is not collocated with the rules that govern it.

The legend, the *Phase* taxonomy, and the DIAG-4 *Message*-column reference are governance for the registry as a whole and belong with the four DIAG rules in `diagnostic-shape.md`; once relocated, the sibling tables (host first) must align to the moved legend's labels.

## Solution Space

**Shape:** single

### Recommendation

Move the column legend and *Phase* taxonomy from `code-registry-parse.md` into `diagnostic-shape.md`, then align the sibling tables to the relocated labels:

1. In `diagnostic-shape.md`, after the DIAG-1…DIAG-4 block, insert a `**Registry table columns.**` paragraph carrying the *Sev* / *Phase* / *Trigger* / *Spec rule* / *Hint* / *Message* definitions, the closed *Phase* taxonomy, and the *E/W* severity carve-out sentence. Use canonical spellings ("Sev", "Spec rule", "Message" — singular, no "template").
2. In `code-registry-parse.md`, replace the legend with a one-line back-reference to `diagnostic-shape.md#code-registry-rules-normative`; keep the "split across four pages" intro.
3. Prepend the same back-reference before the table heading on `code-registry-load.md`, `code-registry-runtime.md`, `code-registry-host.md`.
4. Update DIAG-4's "the *Message* column" to anchor at the collocated legend (e.g. "the *Message* column defined above").
5. Align the `loom/host/*` header to `| Code | Sev | Phase | Trigger | Spec rule | Hint | Message |`, widen the alignment row to seven columns, and add a `—` cell to the *Hint* column of all four host rows. The *Anchor* → *Spec rule* rename is a pure header edit — each host row already cites a Pi Integration Contract anchor in the expected form. Land this in the same change as the `code-registry-runtime.md` *Message template* → *Message* rename so the four tables become uniform in one sweep.

Edge cases: preserve the *E/W* carve-out's cross-references (a Discovery table and `loom/load/callee-has-errors`) verbatim when moving the paragraph; repoint any extractor/test keying on the host page's `Severity`/`Anchor` labels (the `V7b-T` harness is the likely consumer); `placeholder-rendering-b.md` cross-references cite the page, not the labels, and need no edit.

## Consequence

**Severity:** advisory

Readers entering at any continuation page hit undefined columns, and DIAG-4's "the *Message* column" dangles off its own page — acutely on the host page, where the column is also renamed. The legend's "applies to all four tables" claim is locally false on the host and runtime pages, and a header-keyed extractor would fail or silently mis-parse the host page. No diagnostic emission changes, but the gap is the soil in which further header drift accrues.

## Relationships

- `code-registry-runtime.md` "Message template" vs sibling "Message" — co-resolve (the runtime rename should land in the same sweep as the host-header alignment so the legend becomes true at once).
- `diagnostic-shape.md` `loom/typecheck/*` namespace governance misplaced on a runtime page — same-cluster (both move governance prose out of enumeration-only pages; resolve independently).
- `diagnostics.md` opening framing of what the registry rules apply to — same-cluster (resolve independently).
