## Finding

The `AgentSession` member-surface snapshot in `host-interfaces-core.md` declares the subagent-mode query driver as:

```ts
sendUserMessage(text: string): Promise<void>;
```

The Pi-owned declaration at the pinned SDK (`@earendil-works/pi-coding-agent`, `dist/core/agent-session.d.ts:377`) is wider:

```ts
sendUserMessage(content: string | (TextContent | ImageContent)[], options?: { deliverAs?: "steer" | "followUp" }): Promise<void>;
```

The return type matches, but the snapshot narrows the first parameter from `string | (TextContent | ImageContent)[]` to bare `string` and drops the optional `options?: { deliverAs?: "steer" | "followUp" }` argument entirely. The block's own preamble declares each member's TypeScript signature a "non-normative snapshot of the Pi-owned shape at the loom 1.0 Pi-SDK pin … re-validated on each Pi minor bump," and the version-bump editorial-review checklist item (n) is the procedure that re-validates this exact surface against `agent-session.d.ts`. An inaccurate baseline defeats that re-validation: a contributor diffing the snapshot against the candidate SDK records a spurious mismatch on every bump, and the wrong baseline can mask a genuine future drift in the same method. The normative member set (which members loom may touch) is unaffected — only the signature snapshot is wrong.

## Issue introduction

**Verdict:** present-since-inception (of the `AgentSession` member-surface snapshot block)
**Introducing commit:** `aa334c2` — "pi-loom spec: resolve \"AgentSession consumed member surface not pinned\"" (2026-06-05)
**History:** `git log -S 'sendUserMessage(text: string): Promise<void>' --follow` on `host-interfaces-core.md` returns a single commit, `aa334c2`, which added the `AgentSession` member-surface snapshot. The line was authored with the simplified single-argument signature and has not changed since. At that same commit the Pi SDK pin (`host-prerequisites.md#pi-sdk-pin`) already read `@earendil-works/pi-coding-agent ~0.75.5`, whose `agent-session.d.ts` declares the multi-argument `content` + `options` form (as the currently-pinned `dist/core/agent-session.d.ts:377` still does), so the snapshot was narrower than the Pi-owned shape at the moment it was written.

## Solution Space

**Shape:** single

### Recommendation

Replace the snapshot's parameter list with the full Pi-owned shape so the baseline matches `agent-session.d.ts`:

```ts
sendUserMessage(content: string | (TextContent | ImageContent)[], options?: { deliverAs?: "steer" | "followUp" }): Promise<void>;
```

Keep the trailing comment's substantive point (subagent-mode query driver; returns `Promise<void>` unlike `pi.sendUserMessage`'s `void` shape). `TextContent` and `ImageContent` are Pi-owned content types from the same SDK surface; reproduce them verbatim as written in the `.d.ts` rather than substituting loom-local names. The member-set normativity statement and the per-bump re-validation note already cover the snapshot's status and need no change. Optionally note that loom calls the method with the string content arm and does not use `options`.

## Consequence

**Severity:** advisory

The per-bump re-validation (checklist item (n)) compares the SDK against a baseline that is already wrong, producing a spurious mismatch each bump and potentially masking a real future drift in `sendUserMessage`. No runtime divergence results: loom only ever calls the method with a string content arm, so the normative member set keeps the implementation correct.
