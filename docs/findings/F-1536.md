## Finding

The `tool-calls.md` Concurrency paragraph and its mirror sentence in `implementation-notes.md` justify sharing the per-query compiled-validator cache across concurrently-interleaving sibling invocations with a single clause: "compiled validators are immutable post-compile and cache writes only occur on initial compile or file-watcher invalidation". A reader who knows AJV will notice that a compiled validator is **not** wholly immutable: each compiled function mutates its own `.errors` property (and an internal evaluation cursor) on every call, then leaves the result in place for the caller to read. Two interleaved invocations of the same `validate()` function on a multi-threaded runtime would race that mutable state and could observe each other's `.errors`.

The reason the loom design is actually safe is not validator immutability but JavaScript's single-threaded, run-to-completion semantics for synchronous calls: AJV's `validate(value)` is a fully-synchronous function that never `await`s and never yields the event loop between entry and return, so two sibling invocations on the same Node process serialise into back-to-back calls — each one reads its own `.errors` before the next one starts. The shared validator object is safe because the *call* is atomic, not because the object is. The current wording leaves an implementer two ways to read it: (a) "all per-validator state is read-only after compile — sharing is free under any concurrency model", which is wrong; (b) "the validator's mutable per-call state is fenced by run-to-completion of the synchronous call" — the correct reason, but not what the text says. The mismatch matters at every later point that wants to lean on this contract: a future change that introduces async validation, a swap to a validator that exposes `.errors` via a stateful iterator, or an attempt to share validators across worker threads would each silently break the invariant the prose says is already proved.

The same misstated justification appears verbatim in `implementation-notes.md` (per-invocation single-threaded execution bullet), so the fix needs to land in both places.

## Solution Space

**Shape:** single

### Recommendation

Replace the "immutable post-compile" justification, in both sites, with one that names the actual mechanism. In `tool-calls.md` Concurrency, change the final clause of the validator-sharing sentence to read along these lines: "The schema cache (see [Implementation Notes — Runtime](./implementation-notes.md)) is shared across concurrently-interleaving invocations because `validate(value)` is a fully-synchronous JavaScript call: it never `await`s, so two sibling invocations on the same event loop cannot interleave inside a single `validate()` and cannot observe each other's per-call mutable state (notably `.errors`). Cache writes occur only on initial compile or file-watcher invalidation; the cross-invocation isolation contract that makes those writes safe against in-flight reads is owned by [PIC-49 — Cross-window rebuild serialization](./pi-integration-contract/registration-steps.md#pic-49) (one rebuild at a time against the live validator cache)." Mirror the same correction in the `implementation-notes.md` "Per-invocation single-threaded execution" bullet, keeping its existing cross-link to `tool-calls.md#concurrency`.

Edge cases the implementer must watch:

- The fix is purely a rationale rewording — no observable behaviour, no diagnostic code, no API shape, and no test moves. The `Tests`/`Ships when` clauses of `V8c` (validator seam) and `V14b` (parallel batch) stay as written.
- Do not drop the cache-write conditions ("initial compile or file-watcher invalidation"): they are still load-bearing for the write/read interleaving question, which run-to-completion alone does not address — PIC-49's one-rebuild-at-a-time rule is what fences cache writes against in-flight reads, and the cross-reference makes that explicit.
- The validator-neutral framing already established for the binder failure surface (the `<ajv-summary>` rename finding) does not apply here: this paragraph is about the runtime's choice to share *whatever* `CompiledValidator` the seam returns, so naming the synchronous-call property of the contract — not the AJV implementation — is consistent with the validator-seam abstraction.

## Consequence

**Severity:** advisory

An implementer building loom 1.0 today still produces a correct system: the run-to-completion semantics hold whether or not the spec names them. The cost is on future readers and future changes — code review of a hypothetical validator swap, async-validation seam, or worker-thread experiment loses its load-bearing invariant, because the stated invariant (validator immutability) is false and the actual one (synchronous run-to-completion) is unstated.
