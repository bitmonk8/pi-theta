## Finding

The swallowing-handler paragraph in `cancellation.md` (*Race semantics — swallowing-handler attachment on every abandonable Promise*, and the symmetric CNCL-3 clause) asserts that side-channel suppression is **total**: "no Node `unhandledRejection` process event fires, no second `RuntimeEvent` reaches the always-log channel for the same invocation, and no diagnostic of any severity is emitted." Two of these three are negative observables that a conformance test must *witness the absence of* — specifically the absence of a second emission on the `loom-system-note` / always-log channel and the absence of any diagnostic.

The paragraph cites only one test substrate: the `Checkpoint` seam, which it credits with making "the rule … verifiable without relying on JS microtask scheduling." But the `Checkpoint` seam (PIC-10) only controls *when* a late settlement lands relative to a checkpoint — it lets a test deterministically place the post-cancel late settlement. It does nothing to make the always-log emission stream itself inspectable. To assert "no second `RuntimeEvent` reaches the always-log channel," a test must observe the `pi.sendMessage`-driven `loom-system-note` emission stream; that observability is established elsewhere (the `RuntimeEvent` stream and its emission-count/payload comparison in `runtime-event-channel.md`, e.g. the PIC-1(e) conformance-fixture pattern), not in `cancellation.md`.

The result is that the page states a negative-emission acceptance criterion while pinning only the *timing* half of its verification recipe and leaving the *channel-observation* half uncited. An implementer or test author reading `cancellation.md` alone can land the late settlement deterministically but has no pointer to how the always-log channel (or the diagnostics channel) is observed to confirm no second event fired.

## Solution Space

**Shape:** single

### Recommendation

In `cancellation.md`, where the swallowing-handler paragraph (and the CNCL-3 clause) state the total three-side-channel suppression, add a cross-reference for the two negative-emission halves of the acceptance criterion: the "no second `RuntimeEvent` reaches the always-log channel" assertion is witnessed by observing the emitted `RuntimeEvent` stream on the `loom-system-note` / always-log channel as defined in [Runtime event channel](./pi-integration-contract/runtime-event-channel.md) (the same emission-count/payload comparison the PIC-1(e) *Observable pure-read MUST* conformance fixture relies on), and the "no diagnostic of any severity" half is witnessed through the diagnostics channel.

Keep the existing `Checkpoint`-seam sentence but scope it precisely to what it provides — deterministic *landing* of the late settlement at a chosen checkpoint boundary — so the page no longer implies the seam alone makes the negative-emission assertions verifiable. The two substrates are complementary: the `Checkpoint` seam places the late settlement; the always-log `RuntimeEvent` stream observation confirms the absence of a second emission.

Edge case for the implementer: the assertion is that *no* second always-log emission occurs for the same invocation after the checkpoint already surfaced `cause: "cancelled"` — the originating cancellation surfacing is itself not an always-log member (`cancelled` is deliberately excluded from the always-log set per `runtime-event-channel.md`), so the test asserts zero always-log emissions attributable to the abandoned Promise's late settlement, not "exactly one."

## Consequence

**Severity:** advisory

A test author can still build the suppression test by inferring that the always-log channel is the `pi.sendMessage` / `loom-system-note` stream and capturing it, so behaviour is not at risk of divergence. The gap is a verification-guidance hole: the page asserts a negative-emission criterion but pins only the landing-timing substrate, leaving the channel-observation half of the recipe to be reconstructed from other pages.
