## Problem

H4a's third **Tests** bullet opens with the four session-double fidelity-contract self-check assertions (i)–(iv) — genuine automated assertions that run under `npm test` — but then continues, in the same bullet, with the entire manual real-host smoke governance protocol: the live-host precondition, the pass/fail criteria (a)–(e), the un-runnable-gate-vs-divergence distinction, the two named-owner triggers, the contributor-obligation posture, and the merge-blocker rules. None of that smoke-governance prose is an automated assertion — the protocol explicitly never runs under `npm test`; it is a manual pre-merge gate run against a live Pi host. Housing it inside the **Tests** field misrepresents H4a's automated coverage to an implementer and parks normative process obligations (owner duties, triggers, merge-blocker rules) in a field the version-bump and merge audiences do not read.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 328ba4d — pi-loom plan: resolve "real-host verification gap" (2026-06-10, Thomas Andersen)
**History:** The manual real-host smoke run first entered H4a in `328ba4d`, which appended it to the end of the third **Tests** bullet (then a one-sentence "post-merge detection mechanism … revert trigger" clause) rather than to a non-test field — the misplacement was present from the moment the smoke text was authored. Later commits amplified the already-misplaced block in place rather than introducing the defect: `15eb44c` (2026-06-11, "real-host divergence pre-merge gate (posture i)") reframed it as a required pre-merge gate with named-owner triggers and merge-blocker implications, and `9b934cb` (2026-06-12, "manual real-host smoke prerequisites") added the live-host precondition paragraph — each enlarging the governance prose inside the Tests bullet.

## Solution approach

Move the manual real-host smoke governance protocol out of H4a's third **Tests** bullet and into the **Adds** field — the field for what the leaf establishes — leaving the **Tests** bullet with only the four fidelity-contract self-check assertions (i)–(iv) that actually run under `npm test`. Strike the rationale-only sentences (the double-vs-real "presupposition" framing and the "best-effort representative run, not exhaustive real-host coverage" caveats) on the move; they explain motivation rather than asserting an observable.

## Solution constraints

- Preserve the pass/fail criteria labels (a)–(e) verbatim in their new home so the existing cross-references from H6a and V18d that cite "H4a's pass/fail criteria (a)–(e)" keep resolving.
- Out of scope: the H7a fixture `.loom` and permitted-code-list references — do not alter them.
