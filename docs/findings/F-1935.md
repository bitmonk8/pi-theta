## Finding

The four per-site swallowing-handler leaves â€” `V9o`, `V13f`, `V14f`, `V15h` (and their `-T` partners) â€” carry a byte-identical "Swallowing-handler attachment at this site" Tests bullet that asserts two things: (1) the abandonable Promise "attaches its swallowing handler at the Promise-construction site (before the first microtask boundary)", and (2) "a build that bypasses the substrate reddens this leaf's tests". The only test vector the bullet (and the impl-leaf Ships-when) actually describes lands **a late settlement** via the `Checkpoint` seam *after* the checkpoint has surfaced `cause: "cancelled"`, and asserts three-channel suppression (no `unhandledRejection`, no second `RuntimeEvent`, no diagnostic).

A late-settlement-after-cancellation vector only proves a handler is attached *by the time the late settlement lands* â€” it does not witness attachment at the construction site before the first microtask boundary. The spec is explicit that lazy attachment is the failure mode being guarded against: "a `.catch(() => {})` registered lazily after cancellation fires will not catch a rejection that has already been queued for `unhandledRejection`" ([cancellation.md â€” *Race semantics*](../../docs/spec_topics/cancellation.md)). A build that attaches the handler lazily â€” anytime after construction but before the late settlement arrives â€” passes all three channels green yet violates the spec MUST ("The handler MUST be attached at the same site that constructs the Promise, before the first microtask boundary"). The timing claim is therefore unwitnessed by the planned test.

Separately, the suppression assertions are purely outcome-based, so the clause "a build that bypasses the substrate reddens this leaf's tests" over-claims: a build that bypasses the `Checkpoint`-seam substrate but reproduces identical three-channel suppression by other means is not detected by an outcome-only test. Finally, in `V15h`/`V15h-T` the bare phrase "the substrate" is ambiguous between the swallowing-handler substrate and the `Checkpoint`-seam substrate (both are named in the leaf's Adds), and "bypasses" is left undefined.

## Solution Space

### Recommendation

In each of the eight leaves â€” `V9o`, `V13f`, `V14f`, `V15h` and their `-T` partners â€” add an early-rejection test vector to the "Swallowing-handler attachment at this site" Tests bullet (and reflect it in the impl-leaf Ships-when) alongside the existing late-settlement vector. The early-rejection vector constructs the abandonable Promise, rejects it within the first microtask boundary (before any lazy-attach point), and asserts that no `unhandledRejection` fires: a build that attaches the handler at the construction site catches the rejection and passes, while a build that attaches lazily leaves the rejection queued for `unhandledRejection` and reddens. This witnesses the spec MUST that the handler is attached at the same site that constructs the Promise, before the first microtask boundary, using the `Checkpoint` seam the spec designates as the test substrate.

In all eight leaves, drop the unfalsifiable clause "a build that bypasses the substrate reddens this leaf's tests" from the Tests bullet, the impl-leaf Ships-when, and the `V15h` Adds â€” an outcome-only assertion cannot witness substrate bypass. State instead the property the vectors deliver: the handler is attached before the abandoned Promise can settle, and a late settlement is suppressed on all three side channels (no `unhandledRejection`, no second `RuntimeEvent`, no diagnostic of any severity).

In `V15h`/`V15h-T`, replace the bare phrase "the substrate" with its explicit referent, the `Checkpoint`-seam swallowing-handler substrate, so the reference is unambiguous against the two substrates named in the leaf's Adds.

When writing the early-rejection vector, land the rejection before the first microtask boundary with no intervening `await` or other attach point â€” an intervening `await` would itself create an attach point and defeat the test. The spec's lazy-`.catch` failure description is the reference for getting this timing right.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one writes only the planned late-settlement vector and ships green, believing the construction-site / first-microtask timing is verified; the other notices the gap. The spec's construction-site attachment MUST ships effectively unverified â€” a lazily-attaching build (the exact defect the spec calls out) passes all three channels, and the "bypasses the substrate" claim cannot fire because the assertions are outcome-only. The leaf's gate passes vacuously for the construction-site property it claims to witness.
