## Problem

`V18c`'s **Adds.** field describes the editorial-review checklist's `(f)`–`(ak)` items with the spliced coinage `SHOULD-level-audit-but-MUST-record`, which fuses two distinct obligations — performing the audit and recording its outcome — into one token without naming which clause each level governs. Read in isolation it admits two readings: that recording a per-item outcome is mandatory for every item even when the audit is skipped (the authoritative reading), or that only a performed audit's outcome must be recorded. The authoritative semantics in `version-bump-step2.md` and in `V18c`'s own **Ships when.** field resolve to the first: the `(f)`–`(ak)` audits stay SHOULD-level but a per-item outcome (`not-performed` when skipped) MUST always be recorded. The Ships-when gate already carries the unambiguous obligation, so no shipped artefact diverges; the defect is confined to the Adds prose.

## Issue introduction

**Verdict:** introduced-by-prior-fix (single commit)
**Introducing commits:** `f1cda48` (2026-06-11)
**History:** The token did not exist before `f1cda48`. The parent revision's Adds field lumped "the editorial-review checklist items" in with the build-time gates (mislabelling the editorial checklist as a static gate, with no MUST/SHOULD distinction stated). `f1cda48` ("pi-loom plan: resolve \"V18c mislabels editorial-review checklist as a static build-time gate\"") corrected that mislabel by splitting the editorial checklist out as *not* a build-time gate, and in doing so introduced the normative-level distinction in the compressed form `MUST-level for (a)–(e) and SHOULD-level-audit-but-MUST-record for (f)–(ak)`. The ambiguity is therefore collateral of an otherwise-correct prior fix: the level distinction was added correctly but condensed into a single non-standard token rather than two clauses. `git log -S 'SHOULD-level-audit-but-MUST-record'` returns only `f1cda48`; later commits touching the file (`828b7d3`, `622d0d3`, `1e6ac07`) did not alter the token.

## Solution approach

In `V18c`'s **Adds.** field, rewrite the `(f)`–`(ak)` clause to separate the audit obligation from the recording obligation: state that `(a)`–`(e)` are MUST-level audits, that `(f)`–`(ak)` are SHOULD-level audits, and that a per-item outcome MUST be recorded for every `(f)`–`(ak)` item regardless of whether its audit is performed (`not-performed` when skipped). Match the authoritative wording already in `version-bump-step2.md` and `V18c`'s **Ships when.** field. Leave the **Ships when.** field unchanged — its gate is already unambiguous.
