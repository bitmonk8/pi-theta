## Finding

`provider-error-mapping.md` carries two static lookup tables — the **Provider error mapping** overflow-signature table and the **Provider seed-field mapping** table — both with a leftmost column headed `Provider` and both with `api`-shaped row keys (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`). The seed-field prose is explicit that the table "is keyed on the resolved binder model's `api` field"; the error-mapping table's row keys are likewise `api`-shaped, and the synthesised `TransportError.provider` field elsewhere carries the same `api`-shaped value (e.g. `mistral`, `anthropic-messages`). The column header "Provider" contradicts all of this: `host-interfaces-core.md` draws a sharp distinction between `Model<Api>.provider` (the short provider-id form `anthropic`/`openai`, typed `Provider`) and `Model<Api>.api` (the api-shaped value `anthropic-messages`), and states that the model resolver "compares the reference's `provider` component against `.provider` and **never** against `.api`". An implementer who reads "Provider" and keys these tables on `Model<Api>.provider` matches no row for `openai-completions` (whose `.provider` is `openai`) or `anthropic-messages` (whose `.provider` is `anthropic`), silently downgrading every context-overflow response to `TransportError` and emitting the wrong seed field name (or none).

Separately, the four overflow rows describe the same match surface in two different vocabularies. The `anthropic-messages` and `openai-completions` rows say "`errorMessage` matching `/…/i`"; the `mistral` row says "body matching `/context.*length/i`" and the `amazon-bedrock` row says "`ValidationException` with body matching `/…/i`". The page's *Classifier input surface* note establishes that pi-ai "surfaces no parsed JSON error body" and that "every body-wording match in the table is a match against [the `AssistantMessage.errorMessage`] string". The two "body matching" rows therefore name a field — a parsed HTTP body — that loom never receives; an implementer reading those rows literally builds a classifier branch against an unavailable JSON body, while the same predicate on the anthropic/openai rows is correctly written against `errorMessage`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 82be125 — spec: adopt P1 typed-query two-phase loop (Option A) (2026-05-04, Thomas Andersen); 513a5cc — pi-loom spec: resolve "overflow table predicates key on structured JSON fields errorMessage does not expose" (2026-06-09, Thomas Andersen)
**History:** Both tables — the `Provider`-headed column over `api`-shaped row keys, and the mistral/amazon-bedrock rows worded "body matching" — entered in 82be125 when the provider error and seed tables were first authored (then in the monolithic `pi-integration-contract.md`, later split into this page by f5e89f4 without content change). At that point all four overflow rows referenced structured fields uniformly. 513a5cc rewrote the `anthropic-messages` and `openai-completions` rows to match against `errorMessage` and added the *Classifier input surface* clause that "every body-wording match in the table is a match against that string", but left the `mistral` and `amazon-bedrock` rows saying "body matching" — converting uniform wording into the present inter-row inconsistency. The "Provider" column header was untouched by either commit.

## Solution Space

**Shape:** single

### Recommendation

In `provider-error-mapping.md`, rename the leftmost column header of **both** tables from `Provider` to `` `api` value `` (or `` `Model<Api>.api` value ``), and add one sentence to the **Provider error mapping** prose stating the table is keyed on the resolved model's `Model<Api>.api` field — mirroring the wording the seed-field table prose already carries ("keyed on the resolved binder model's `api` field"). This makes both tables consistent with the `.api` vs `.provider` distinction pinned in `host-interfaces-core.md` and with the api-shaped `TransportError.provider` value.

Reword the `mistral` and `amazon-bedrock` overflow rows so their match predicate names `errorMessage`, consistent with the `anthropic-messages`/`openai-completions` rows and the *Classifier input surface* note: e.g. `mistral` → "HTTP 400 with `errorMessage` matching `/context.*length/i`"; `amazon-bedrock` → "`errorMessage` matching `/(input is too long\|context window)/i` (the pi-ai-formatted `ValidationException` body)". Preserve the existing regexes and the "token counts not surfaced — both fields `null`" clauses verbatim.

Edge cases for the implementer: the seed-field table prose already states `.api` keying, so only its header needs the rename, not new prose. Keep `ValidationException` referenced in the amazon-bedrock row as descriptive provenance, but make the match target unambiguously the `errorMessage` string. Do not alter the regex literals while rewording (the `\|`/alternation question is a separate finding).

## Consequence

**Severity:** correctness

An implementer who keys either table on `Model<Api>.provider` — the natural reading of a column headed "Provider", and the field the spec elsewhere insists the resolver uses — matches nothing for the api-shaped row keys, downgrading every `ContextOverflowError` to `TransportError` (with null token counts) and omitting/mis-naming the request seed field. The two "body matching" overflow rows direct that same implementer to match against a parsed JSON body pi-ai never exposes, so the mistral and amazon-bedrock overflow branches never fire. Two reasonable implementers diverge on which field these tables key on and which field the overflow regexes run against.
