## Problem

`placeholder-rendering-b.md` §7's *Parse-time literal-value* sub-rule renders `<value>` per category 5's `<key>` shape for six parse-time rows, **except** for "YAML scalars with no enclosing source quoting", which render unquoted regardless of identifier shape. YAML 1.2 admits five scalar styles, but only *plain* cleanly satisfies "no enclosing source quoting"; the rule says nothing about how single-quoted, double-quoted, block-literal, and block-folded scalars render. Author inputs that are routinely quoted at the V1 surface (e.g. `mode: "wibble"`, `respond_repair.methodology: 'none'`) admit two equally defensible readings that produce non-byte-identical messages, breaking §7's closure premise that every enumerated placeholder renders byte-identically across conformant implementations.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `f930818` — "pi-loom spec: resolve 'placeholder rendering exemption open-ended'" (2026-05-07)
**Migration commit:** `f5e89f4` — "docs: enforce 100KB hard / 30KB recommended size cap on spec set" (2026-06-04), which split the monolithic `spec_topics/diagnostics.md` into `diagnostics/placeholder-rendering-{a,b}.md` without altering the carve-out's wording.
**History:** `git log -S "YAML scalars with no enclosing source quoting"` on the spec corpus returns exactly the two commits above. The §7 *Parse-time literal-value* sub-rule (carve-out clause included) was introduced *de novo* by `f930818` as part of closing the placeholder-rendering surface to eight categories. The carve-out has been the same single sentence ever since; subsequent commits to `placeholder-rendering-b.md` touched adjacent rules but did not revisit it. The defect is therefore inception-state in §7 itself, not the residue of a later edit.

## Solution approach

Rewrite §7's *Parse-time literal-value* carve-out clause to pivot its discriminator from "presence/absence of enclosing source quoting" to YAML 1.2's plain-scalar-style classification (§8.1.2.1): plain scalars render unquoted regardless of identifier shape, and the four non-plain styles (single-quoted, double-quoted, block-literal, block-folded) fall through to category 5's `<key>` rule operating on the parsed scalar value. Ground the carve-out against the YAML 1.2 scalar-style definition so the discriminator is well-defined for every scalar at parse time.

- Out of scope: category 5's `<key>` identifier-shape rule itself (F-1503's territory) — this finding routes non-plain scalars *to* that rule, it does not redefine it.
