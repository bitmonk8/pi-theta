## Finding

`placeholder-rendering-b.md` §7's *Parse-time literal-value* sub-rule pins `<value>` for six parse-time rows (`loom/load/unknown-mode-value`, `loom/load/unknown-methodology-value`, `loom/load/unknown-bind-context-value`, `loom/load/model-unresolved`, `loom/parse/duplicate-enum-value`, `loom/parse/duplicate-discriminator-value`) to category 5's `<key>` shape — bare when identifier-shaped per the runtime regex `^[A-Za-z_][A-Za-z0-9_]*$`, double-quoted otherwise — **except** for "YAML scalars with no enclosing source quoting (e.g. `mode:` values)", which render unquoted regardless of identifier shape. The carve-out's discriminator is "no enclosing source quoting", but YAML 1.2 admits five scalar styles and only one of them (*plain*) cleanly meets that description. The other four — single-quoted (`'…'`), double-quoted (`"…"`), block literal (`|`), and block folded (`>`) — all carry distinct enclosing source markers, and the rule says nothing about how to render `<value>` when the offending scalar uses any of them.

Two readings are equally defensible and produce non-byte-identical messages. An author who writes `mode: "wibble"` triggers `loom/load/unknown-mode-value`; one renderer reads the carve-out as "the source had enclosing quoting, so fall through to the §5 `<key>` rule" and emits `unknown 'mode:' value '"wibble"'`, another reads it as "the carve-out fires whenever YAML allows the value unquoted at this position" and emits `unknown 'mode:' value 'wibble'`. The §7 closure premise that every enumerated placeholder renders byte-identically across conformant implementations is broken at the V1 surface, not just at some hypothetical edge: `mode:`, `respond_repair.methodology:`, and `bind_context:` are routinely quoted by authors. Worse, non-identifier-shaped values amplify the divergence: a `mode: "my-mode"` source would render bare under the open reading and double-quoted under the strict reading.

The clean fix is to pivot the discriminator from "presence/absence of enclosing source quoting" to "YAML scalar style is *plain*" — a five-way classification fixed by YAML 1.2 §8.1.2.1, well-defined for every scalar at parse time, and reducing the four non-plain styles to the existing §5 `<key>` rule (which already operates on the parsed string value irrespective of source style).

## Solution Space

**Shape:** single

### Recommendation

In `placeholder-rendering-b.md` §7's *Parse-time literal-value* bullet, replace the carve-out clause

> — except that YAML scalars with no enclosing source quoting (e.g. `mode:` values) render unquoted regardless of identifier shape.

with

> — except that **YAML plain scalars** (the scalar is written with no surrounding `'`, `"`, `|`, or `>` — YAML 1.2 §8.1.2.1) render unquoted regardless of identifier shape; single-quoted, double-quoted, block-literal, and block-folded scalars fall through to category 5's `<key>` rule and are rendered against the *parsed* scalar value (bare when identifier-shaped, double-quoted otherwise).

Two implementer-relevant notes the `V7c` leaf and its `DIAG-4` test vectors must observe:

1. The discriminator pivots from a source-text predicate ("any enclosing quoting?") to YAML's own scalar-style classification (one of `plain` / `single-quoted` / `double-quoted` / `literal` / `folded`). Every YAML parser exposes this; for `js-yaml` it is the node `style` field on a scanned scalar event. The four non-plain styles fall through to the §5 `<key>` rule, which operates on the parsed string value and is unambiguous.
2. `V7c-T`'s `DIAG-4` vectors for the parse-time literal-value bullet must add boundary vectors for at least one row (the natural choice is `loom/load/unknown-mode-value`, since `mode:` is the carve-out's own example): `mode: wibble` (plain, identifier-shaped → bare), `mode: 'wibble'` (single-quoted, identifier-shaped → bare per §5), `mode: "wibble"` (double-quoted, identifier-shaped → bare per §5), and `mode: "my-mode"` (double-quoted, non-identifier-shaped → double-quoted per §5). Block-literal/folded styles are syntactically legal on these scalar fields but practically unidiomatic; one vector for completeness is sufficient and may be deferred.

## Consequence

**Severity:** correctness

Two conformant implementations targeting the same author input (`mode: "wibble"`, `respond_repair.methodology: 'none'`, `bind_context: "my-context"`, etc.) can produce non-byte-identical diagnostic messages, violating §7's closure premise that every enumerated placeholder renders byte-identically. The `DIAG-4` per-category vectors that `V7c` ships against cannot be written unambiguously for the parse-time literal-value bullet under the current wording.
