## Finding

The `V9c` / `V9c-T` `PIC-2` Tests bullet reads: "within a single user session, no two prompt-mode bodies hold an open active-set gating window (the `pi.setActiveTools` snapshot/restore bracket) simultaneously — a nested prompt→prompt `invoke(...)` opens its window only after the parent body's window is restored (cross-body non-overlap, distinct from a single query's snapshot/restore)." It presents cross-body non-overlap as a flat property to assert.

The spec (`conversation-drive.md` PIC-2) is explicit that this is **not** an independent runtime obligation: "This is a *derived* property rather than an independent runtime obligation: it follows from Pi's per-session slash-handler serialisation guarantee … composed with the loom-runtime-owned guarantees that (i) prompt-mode `.loom` callees in another loom's `tools:` are rejected at load time, (ii) `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns and loom 1.0 exposes no parallel-`invoke` surface, and (iii) the `subagent → prompt` fan-out into the user session is closed." The non-overlap of two *top-level* slash dispatches is a host guarantee the capability probe (`V9a`) does not verify; only the loom-owned suspend / no-parallel-`invoke` composition is loom code.

Because the leaf flattens this distinction, an implementer working only from the bullet may mis-read PIC-2 as a loom enforcement obligation and build an inter-body mutex, or author a test that tries to assert Pi's per-session serialisation directly — a host behaviour the `H4a` session double does not model — yielding a vacuous or wrong-by-construction test rather than one that exercises the loom-owned suspend semantics.

## Solution Space

**Shape:** single

### Recommendation

In the `PIC-2` Tests bullet of both `docs/plan_topics/V9c-conversation-drive.md` and `docs/plan_topics/V9c-T-conversation-drive.md`, note that PIC-2 is a derived property — it follows from Pi's per-session slash-handler serialisation guarantee composed with the loom-owned suspend / no-parallel-`invoke` guarantees — and is not enforced independently by an inter-body mutex. Make explicit that the loom-owned, harness-observable piece the test exercises is the nested prompt→prompt `invoke(...)` suspend semantics (the child's `pi.setActiveTools` window opens only after the parent body's window is restored), and that the top-level two-dispatch non-overlap rests on Pi's serialisation guarantee rather than on a loom mechanism.

Suggested bullet content (wording at the implementer's discretion): append a clause such as "— PIC-2 is a *derived* property: cross-body non-overlap of two top-level slash dispatches follows from Pi's per-session slash-handler serialisation guarantee (a host behaviour, not loom-enforced; see [conversation-drive.md#pic-2](../spec_topics/pi-integration-contract/conversation-drive.md#pic-2)); this leaf asserts only the loom-owned piece, that a nested prompt→prompt `invoke(...)` window opens after the parent's restore." Keep the existing snapshot/restore-bracket reference. No spec edit is required; the spec is read-only for this fix.

## Consequence

**Severity:** advisory

If shipped unfixed, an implementer reading the PIC-2 bullet in isolation may treat cross-body non-overlap as a loom-enforced invariant — implementing a redundant inter-body mutex, or writing a test that asserts Pi's per-session serialisation against the `H4a` double (which does not model host slash-handler serialisation), producing a vacuous green or a wrong-by-construction red. A working leaf is still reachable via the spec link, so this is a guidance gap rather than a blocker.
