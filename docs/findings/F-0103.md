## Finding

`V15d` owns the prompt→prompt parent-suspend with the `setActiveTools`
snapshot/restore window. The spec defines the restore as unconditional:
[`invocation.md`](../../docs/spec_topics/invocation.md) §prompt → prompt says the
snapshot is restored "on return (or any `Err` / panic / cancellation) … in a
`finally` block", and [`tool-registration-lifetime.md` PIC-17](../../docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md#pic-17)
step 4 restores in a `finally` "so cancellation, panic, and provider exceptions
all preserve the invariant". That enumerates four settle paths through the same
`finally`: normal success return, an ordinary terminal `Err`, cancellation, and
a thrown panic.

`V15d`'s own **Adds** field restates the obligation as restore "once the child
settles, including the fail/cancel/throw paths" — three failure modes. But the
**Tests** bullet (carried identically in `V15d` and `V15d-T`) narrows the
explicit coverage to "Cover both the cancel and the throw sub-case", and the
**Ships when** condition asserts restore only "across both the cancel and throw
sub-cases". The ordinary `Err`-return restore (the "fail" path the Adds field
itself names) and the success-path restore are therefore unasserted.

The success-path restore is the dominant runtime case — every successful
prompt→prompt child must hand the parent's ambient tool set back. Because the
gated sub-cases exercise only the two exception paths, an implementation that
restores via a `catch`/rethrow shape (rather than a true `finally`), or that
guards the restore on an error condition, would ship green while leaking the
child's callable set into the user session after a successful or
ordinary-`Err` child. The acceptance criteria do not pin the invariant they
claim to gate.

## Solution Space

**Shape:** single

### Recommendation

Extend the settle-path coverage in both leaves so the gate asserts restore on
every path the `finally` transits.

- In `docs/plan_topics/V15d-T-prompt-suspend-snapshot.md`, amend the PIC-17
  Tests bullet so its explicit sub-case enumeration covers the success-return
  and ordinary terminal-`Err` paths in addition to cancel and throw — i.e.
  replace "Cover both the cancel and the throw sub-case, since both transit the
  same `finally`" with coverage of all four settle outcomes (successful return,
  ordinary `Err` return, cancellation, throw), each asserting
  `pi.getActiveTools()` returns the pre-invoke snapshot once the child settles
  and (for the three failure paths) the inner failure is surfaced, not masked.
- In `docs/plan_topics/V15d-prompt-suspend-snapshot.md`, apply the identical
  amendment to its PIC-17 Tests bullet, and update the **Ships when** condition
  so it asserts restore "across the success-return, ordinary-`Err`, cancel, and
  throw sub-cases" rather than only "both the cancel and throw sub-cases".

Keep the existing distinction notes intact (the restore-on-inner-failure path
remains distinct from PIC-8/PIC-19 owned by `V9f` and PIC-2 owned by `V9c`); the
success-return sub-case asserts only restore and unchanged final-value
propagation, with no inner-failure surfacing clause.

## Consequence

**Severity:** correctness

The leaf's acceptance gate verifies restore on only two of the four settle paths
the spec mandates, omitting the most common one (successful return) and the
ordinary terminal-`Err` path. An implementation whose restore covers only the
exception paths — or that the test suite never drives down the success path —
ships green while leaving the child's tool set active in the user session after
a normal child invocation, defeating the snapshot/restore invariant the leaf
exists to enforce.
