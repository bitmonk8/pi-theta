## Problem

The compact-transcript renderer's Rule 4 `toolResult` branch in `docs/spec_topics/binder/binder-model-and-context.md` serialises any non-text block in `ToolResultMessage.content` with bare `JSON.stringify` and no whitespace, naming no key-canonicalisation step. BNDR-8 in the same file requires key-sorting canonicalisation (object keys in ascending Unicode code-point order at every nesting level) for assistant tool-call arguments, explicitly stating that `JSON.stringify` alone is not key-order-stable — directly contradicting the `toolResult` branch for the same JS primitive. Because BNDR-7g exercises only a single-key non-text block (`{"chartId":7}`), the reproduce-exactly oracle never catches the gap. A multi-key non-text block makes two conforming implementations emit different transcript bytes for the same `(session, prompt-bytes, temperature: 0, seed)` input, defeating BNDR-7's input-reproducibility contract.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 20569f8 — pi-loom spec: resolve "Compact-transcript format for the session-context block is unspecified" (2026-05-07, Thomas Andersen); 01c9db9 — pi-loom spec: resolve "compact-transcript renderer references undeclared SessionContext/AgentMessage fields" (2026-06-03, Thomas Andersen); 1a41db2 — pi-loom spec: resolve "Compact-transcript BNDR-7 oracle coverage and assistant byte-determinism (T034/T106)" (2026-06-06, Thomas Andersen)
**History:** Commit `20569f8` first introduced Rule 4's `toolResult` body as "structured (non-string) content is `JSON.stringify`'d with no whitespace and used as the body", with no key-canonicalisation. Commit `01c9db9` rewrote the clause for the typed-block array shape but kept the bare `JSON.stringify` recipe ("any non-text block `JSON.stringify`'d with no whitespace and concatenated in the same array order"). The contradiction only crystallised when commit `1a41db2` added BNDR-8 (and the new BNDR-7e rendering) pinning byte-determinism for assistant tool-call arguments and explicitly stating "`JSON.stringify` alone is not key-order-stable … so a key-sorting canonicalisation step is required" — that same commit also added BNDR-7g but exercised only a single-key non-text block (`{"chartId":7}`), so the new canonicalisation discipline never propagated to the `toolResult` non-text branch and no oracle caught the gap.

## Solution approach

Rewrite the Rule 4 `toolResult` bullet so the non-text block serialisation cross-references [BNDR-8](#bndr-8)'s canonical no-whitespace JSON serialisation, mirroring how the `assistant` bullet already cites BNDR-8 for `<args-json>`, and generalise BNDR-8 so it is the renderer's single normative source for canonical JSON serialisation across both sites. Extend [BNDR-7g](#bndr-7g), or add a sibling reference rendering under the next free sub-letter, to exercise a multi-key non-text block (e.g. serialised form `{"caption":"Q3","chartId":7}` for an input whose insertion order is `chartId` then `caption`) so the BNDR-7 reproduce-exactly oracle has bytes to fail on when an implementation skips the key-sort.

## Solution constraints

- Do not renumber existing BNDR-7 sub-letters; if BNDR-7g is already cited elsewhere, add the multi-key vector under the next free sub-letter rather than editing 7g in place (BNDR-7's "existing sub-letters are not renumbered" rule).
- The non-text block's internal shape is owned upstream by the Pi-SDK tool-result type family; loom's contract is over its serialised form only — do not pin the SDK block's internal structure.
