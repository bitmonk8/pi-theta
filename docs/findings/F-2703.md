## Finding

The four live-corpus coverage reconciliation surfaces — the unmapped-executable-REQ-ID arm and the numbered-REQ-ID citing-test arm (`H5a`), the un-anchored normative-MUST text-scan arm (`H5e`), and the per-facet citing-test arm for multi-leaf coverage-matrix rows (`H5f`), all reconciled against the live `coverage-matrix.md` — hard-fail `npm test` only after the terminal `H6a` activation commit flips the gate from its seeded-fixture footing to its live-corpus footing. `H5a` states this explicitly: each live-corpus failure mode "is exercised against the seeded fixtures here and first becomes binding at the same `H6a` release-gate activation." Until that flip, the live spec/test corpus is never reconciled in a CI-blocking mode.

The only pre-activation pass over the live corpus is the warn-only `H5b` canary, and it runs as a single pre-flight "immediately before `H6a`'s hard-fail flip" — i.e. also at release time. There is no recurring or per-merge warn-only reconciliation during the long mid-build window while `V1a`–`V18d` are still landing. A coverage regression introduced mid-build — an executable REQ-ID merged with no `coverage-matrix.md` row, a mapped numbered REQ-ID whose citing test was dropped, a multi-leaf-row facet that lost its facet-naming citing test, or a newly-introduced un-anchored normative MUST — is invisible to every gate from the moment it is merged until the single release event, concentrating detection (and therefore remediation) at the most expensive point in the schedule.

This is a deliberate, sound design choice for the *hard-fail* gate: activating the live-corpus footing earlier would redden `main` on coverage that later leaves are still legitimately landing. The gap the finding records is that no proportionate, non-blocking detection surface fills the mid-build window, and the plan does not acknowledge the resulting risk concentration as an accepted tradeoff.

## Consequence

**Severity:** advisory

If the plan ships unchanged, a mid-build coverage regression (dropped citing test, unmapped REQ-ID, un-anchored MUST, broken per-facet closure) survives undetected through every intervening merge and first surfaces at the `H6a` release gate, where the closing leaf and its context are long out of working memory and remediation is most expensive. No leaf is blocked and the release gate does eventually catch the gap, so this degrades detection timeliness rather than correctness.

## Solution Space

**Shape:** multiple

### Option A — Document the accepted tradeoff

**Approach.** State explicitly that the live-corpus coverage modes are deliberately not CI-blocking until the `H6a` release flip, that the `H5b` warn-only canary is the single pre-flight, and that mid-build coverage gaps are by design surfaced only at release.

**Plan edits.** In `docs/plan_topics/conventions.md` *REQ-ID discipline* (or the *Loom 1.0 release-time residue inspection* preamble), add prose recording that the live-corpus unmapped-REQ-ID, numbered-REQ-ID citing-test, un-anchored-MUST, and per-facet citing-test reconciliations are intentionally gated only at the `H6a` release flip — with the rationale that an earlier hard-fail footing would redden `main` on coverage that later leaves are still landing — and that the sole pre-activation live pass is the warn-only `H5b` canary run immediately before the flip. Optionally mirror a one-line pointer in `docs/plan.md`'s Release gate section.

**Spec edits.** None.

**Pros.** Proportionate to an advisory-severity gap; no new mechanism or maintenance surface; makes the deliberate design visible so a contributor understands a merge's coverage gap will not surface until release.

**Cons.** Does not actually shorten the detection window — a mid-build regression still surfaces only at release.

**Risks.** None beyond leaving the timeliness gap in place.

### Option B — Add a recurring warn-only live-corpus checkpoint

**Approach.** Reuse `H5a`'s existing warn-only live-corpus mode (already returns a structured per-gap findings collection) in a non-blocking CI step that runs on every merge, not only at the `H6a` pre-flight, surfacing coverage deltas continuously.

**Plan edits.** Add a `<new>` horizontal leaf (Deps reaching `H5a`/`H5e`/`H5f` for the gate machinery) that wires `H5a`'s warn-only live-corpus mode into a recurring CI surface emitting the returned findings without setting a non-zero exit status; reference it from `docs/plan.md`'s Release gate section alongside `H5b`. Leave the `H6a` hard-fail flip unchanged.

**Spec edits.** None.

**Pros.** Genuinely shortens the detection window — a regression shows up at the merge that introduced it.

**Cons.** Mid-build the live corpus is intentionally incomplete (many REQ-IDs have no closing leaf yet), so a raw warn-only run reports a large standing volume of not-yet-landed coverage as "gaps"; isolating an actual regression requires a per-merge baseline-delta mechanism, which is materially more work than the warn-only mode alone provides.

**Risks.** A noisy or un-baselined recurring report trains contributors to ignore it, leaving the gap effectively unaddressed while incurring maintenance cost.

### Recommendation

Adopt **Option A**. The release-time-only hard-fail footing is a deliberate, well-reasoned design (an earlier hard gate would redden `main` on legitimately-pending coverage), and the severity is advisory, so the proportionate fix is to make the tradeoff explicit in `conventions.md` *REQ-ID discipline* rather than to build a recurring CI surface. Option B's per-merge signal is real but is low-value without a baseline-delta mechanism, because mid-build the live corpus is intentionally incomplete and a raw warn-only run is dominated by not-yet-landed coverage. If the project later wants the earlier signal, Option B is the path — but it must carry a delta/baseline mechanism to be actionable, not just the existing warn-only enumeration.
