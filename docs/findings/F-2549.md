## Finding

The Grammar Appendix (`grammar.md`) declares its own scope up front: it is normative only for "the few surface-syntax forms that no single topic page owns end-to-end, and for the **literal sublanguage**." Its `## `array<T>` literal type-sink rule` section breaks that contract. That section is type-inference semantics for expression-context array literals — it enumerates the exhaustive *type-sink* set (binding annotation, parameter type, surrounding constructor field, recursive array-element descent), states the `for`-iterand carve-out ("the iterand of a `for x in expr` is **not** a sink"), and raises `loom/parse/array-no-common-type`. None of that is surface syntax or literal-sublanguage admission; it is the element-type-inference relation for array literals wherever they appear in expression context.

That subject is already owned by `expressions.md` under `## Array construction`, which states the same `[]`/`[a, b, c]` element-type inference, the same `loom/parse/array-no-common-type` diagnostic, and the numbered common-type rules (sink-in-scope check, least-upper-bound computation, no-implicit-schema-unification). The grammar appendix section therefore both sits in the wrong page and partially duplicates the expressions.md treatment — the appendix even forward-links to `[Expression Sublanguage — Array construction]` for the very concept it then re-specifies. Two normative homes for one inference rule is a drift hazard: a later edit to one (e.g. adding a sink kind, or refining the carve-out) silently diverges from the other.

The legitimate residue is small: array literals (`ArrayLit`) genuinely are part of the literal sublanguage the appendix owns, and in literal positions the element type is supplied by the external sink (the `params:` declared type or the Pi-tool input schema). The appendix may keep a one-line pointer to that effect; the exhaustive sink enumeration, the `for`-iterand carve-out, and the `array-no-common-type` diagnostic are expression-context inference and belong in `expressions.md`.

## Solution Space

**Shape:** single

### Recommendation

Move the type-sink enumeration, the `for`-iterand carve-out, and `loom/parse/array-no-common-type` from the `grammar.md` `## `array<T>` literal type-sink rule` section into `expressions.md` `## Array construction`, consolidating with the rules already there:

- Fold the four-item exhaustive sink set (binding annotation, function parameter type at a call site, surrounding constructor field type, recursive array-element descent) into the existing sink-in-scope rule (rule 1 of the numbered Array-construction rules), so the sink set is enumerated once.
- Add the `for`-iterand carve-out to `expressions.md`: `for x in expr` does not supply a sink, so `for x in []` with no other sink is `loom/parse/array-no-common-type`, the same diagnostic `let xs = []` raises unannotated.
- Keep `loom/parse/array-no-common-type` defined at this single site (it is already named in `expressions.md`); remove the duplicate definition from the appendix.

In `grammar.md`, retain only a one-line forward reference under the literal sublanguage / `ArrayLit` production noting that an array literal's element type is determined by the type-sink rule defined in `[Expression Sublanguage — Array construction](./expressions.md#array-construction)`, and that in literal positions the sink is supplied externally (the `params:` declared type or the Pi-tool registered input schema). Do not restate the enumeration or the diagnostic there.

Edge cases for the implementer: preserve the recursive-descent sink case (an array literal nested as an element of an array-typed sink) when folding into rule 1; ensure the relocated carve-out text reads as a normative rule (the original "this is the same hole; resist any `for`-specific carve-out" is reviewer-advisory and should be restated as the plain rule that `for` supplies no sink).

## Consequence

**Severity:** advisory

The rule is correct and present, so an implementer can build a working system. But the same type-sink/common-type inference is normatively stated in two pages with no single source of truth; a future edit to one risks diverging from the other, and a reader trusting the appendix's stated scope would not expect expression-context inference there.
