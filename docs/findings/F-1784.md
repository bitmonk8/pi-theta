## Finding

`docs/spec_topics/schema-subset.md`'s **Depth Enforcement → Enforcement point** table, row #1, describes the typed-query depth-walk boundary as running "**after the model's final assistant text is JSON-parsed and before the value is handed back to loom code**". This sentence pins both the *source* (JSON-parsed final assistant text) and the *timing* (after parsing, before hand-off) of the validated value at the typed-query boundary.

`docs/spec_topics/query/query-tool-loop.md` QRY-14 contradicts both. A typed query runs a two-phase loop ending in a **forced respond turn**: the model is forced to invoke the synthesised one-shot `__loom_respond_<slug>` tool, whose `execute` *"AJV-validates the call payload against the lowered response schema and resolves the query's promise with the validated value"*. The depth-walk fires **inside the respond tool's `execute` (per CIO-3) before AJV runs** (worked example, step 3). The validated value is therefore the JSON-decoded **respond-tool call payload** (the tool's `arguments`), never the JSON-parsed assistant text — typed queries terminate on a `tool_use` turn, not a plain-text turn, so there is no "final assistant text" to parse at row #1.

An implementer following row #1 builds a separate JSON-parse-then-validate pipeline that does not exist; an implementer following QRY-14 instals the depth walk in the respond tool's `execute`. The two paths diverge on which transcript surface the validated value, the depth-walk path/Pointer, and the `raw_response` field of any resulting `QueryError` are derived from.

## Solution Space

**Shape:** single

### Recommendation

Rewrite row #1 of the **Enforcement point** list in `docs/spec_topics/schema-subset.md` § Depth Enforcement to describe the boundary as it actually exists on the two-phase typed-query path: the depth walk runs **inside the synthesised `__loom_respond_<slug>` tool's `execute`, on the JSON-decoded respond-tool call payload, before AJV runs**, per QRY-14 in [`query/query-tool-loop.md`](./query/query-tool-loop.md). The text should:

- Drop "after the model's final assistant text is JSON-parsed" — typed queries terminate on a `tool_use` turn, so there is no assistant-text JSON-parse stage at this boundary.
- Name the respond-tool call payload (i.e. the tool's `arguments` after JSON decoding) as the value the depth walk and the validated hand-off operate on.
- Forward-link the relevant authority: QRY-14 in `query/query-tool-loop.md` (for the two-phase loop), and CIO-3 in `hard-ceilings/ceilings-3-and-4.md` (for the depth-walk's "inside `execute`, before AJV" ordering — the same ordering claim QRY-14's worked example already cites).
- Keep the existing destination/surface row in the per-boundary routing table (the `Err(QueryError { kind: "validation", cause: "schema_validation", ... })` row) unchanged — only the *provenance* clause of row #1 in the bulleted "every site where a Loom-declared schema is validated against runtime JSON" enumeration is wrong; the routing table below is consistent with QRY-14.

Edge cases the implementer must keep correct after the rewrite:

- The **Edge cases** bullet "The depth walk runs on the **post-decode** JSON value, not on the raw assistant text. A response that fails to parse as JSON surfaces as the existing JSON-parse validation failure, not a depth failure." applies in the new framing to a malformed respond-tool `arguments` payload (JSON-decode failure of the call payload), not to a malformed assistant text. Adjust the bullet's "raw assistant text" wording to refer to the respond-tool call payload bytes the model emitted, so the two-phase boundary's parse-failure surface is also correctly described.
- The `raw_response` field on the resulting `QueryError` (cited in the depth-6 worked example as "the depth-6 payload as JSON text") is the respond-tool call's `arguments` JSON, not assistant text — re-confirm this in passing so no reader is left to reconcile row #1 against the worked example unaided.
- QRY-9 routes failures-to-invoke-the-respond-tool through schema-validation respond-repair via a *synthesised* `ValidationIssue`; that path is separate from the depth-walk and is unaffected by this rewrite. Do not blur the two.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on where the typed-query validated value originates: one wires the depth walk into a post-assistant-text JSON-parse stage (which does not exist on the two-phase typed-query path), the other wires it into the respond tool's `execute`. Beyond the unreachable code path, the divergence propagates into `QueryError.raw_response` (assistant text vs. the respond-tool call's `arguments` JSON), the JSON Pointer reported in `validation_errors[*].path`, and the respond-repair re-walk surface — all of which are externally observable on failure.
