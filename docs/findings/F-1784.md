## Problem

Row #1 of the **Enforcement point** list in `docs/spec_topics/schema-subset.md` § Depth Enforcement describes the typed-query depth-walk boundary as running "after the model's final assistant text is JSON-parsed and before the value is handed back to loom code". QRY-14 in `query/query-tool-loop.md` contradicts this: a typed query terminates on a forced respond turn in which the model invokes the synthesised `__loom_respond_<slug>` tool, and the depth walk fires inside that tool's `execute` (per CIO-3) before AJV runs, operating on the JSON-decoded respond-tool call payload. Typed queries terminate on a `tool_use` turn, so there is no "final assistant text" to parse at this boundary; an implementer following row #1 builds a JSON-parse-then-validate pipeline that does not exist, while an implementer following QRY-14 instals the walk in the respond tool's `execute`. The divergence propagates into `QueryError.raw_response`, the reported JSON Pointer, and the respond-repair re-walk surface.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `216130d` (2026-05-04 18:13:15 +0200, "pi-loom spec: resolve \"Depth ≤5: counting algorithm and enforcement point unspecified\"") authored row #1 with the JSON-parse-assistant-text wording; `82be125` (2026-05-04 21:10:10 +0200, "spec: adopt P1 typed-query two-phase loop (Option A)") replaced the typed-query mechanism with the two-phase loop terminating on a forced respond tool call, invalidating row #1's wording without updating it.
**History:** `git log -L '/Typed-query response validation/,+1:docs/spec_topics/schema-subset.md'` shows row #1 added exactly once, in `216130d`, with the present "after the model's final assistant text is JSON-parsed" phrasing — natural at the time, because the pre-two-phase typed-query design did extract the value from the model's final assistant text. Roughly three hours later on the same day, `82be125` ("spec: adopt P1 typed-query two-phase loop") rewrote the typed-query mechanism in `query.md`, `pi-integration-contract.md` and `implementation-notes.md` to use a synthesised one-shot respond tool driven by `options.toolChoice`, making the validated value the respond-tool call payload (now pinned in QRY-14 and in the depth-6 worked example as "fires *inside* the respond tool's `execute` (per CIO-3) before AJV runs"). That rewrite touched the typed-query mechanism pages but did not revisit the Depth Enforcement table, so row #1's pre-two-phase wording survived as a localised stale description. Subsequent commits to `schema-subset.md` (`6892aa3`, `7321f01`, `e1f5c2a`, `2128e5f`, `111e533`, `c5bb5e1`, …) touched the lowering algorithm, canonical hash, and slug-collision posture but never re-touched row #1.

## Solution approach

Rewrite row #1's provenance clause in `docs/spec_topics/schema-subset.md` § Depth Enforcement to describe the boundary as it exists on the two-phase typed-query path: the depth walk runs inside the synthesised `__loom_respond_<slug>` tool's `execute`, on the JSON-decoded respond-tool call payload, before AJV runs. Forward-link QRY-14 in `query/query-tool-loop.md` (for the two-phase loop) and CIO-3 in `hard-ceilings/ceilings-3-and-4.md` (for the depth-before-AJV ordering). Reconcile the Edge cases bullet's "raw assistant text" wording so its parse-failure surface refers to the respond-tool call payload bytes rather than assistant text.

## Solution constraints

- Out of scope: the per-boundary routing table's `#1 Typed-query response` row (the `Err(QueryError { kind: "validation", cause: "schema_validation", … })` surface) — it is already consistent with QRY-14; only the provenance clause is wrong.
- Do not blur the depth-walk path with QRY-9's failure-to-invoke respond-repair path (the synthesised `ValidationIssue`), which is separate and unaffected by this rewrite.
