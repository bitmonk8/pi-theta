## Finding

The "Tool execution from loom code" section of `host-interfaces-core.md` states that the `execute()`-returned envelope shape "is declared on the tool-result type family in `dist/core/extensions/types.d.ts` of `@earendil-works/pi-coding-agent`", and promises an SDK surface-inventory test that "asserts the field set on the cited type". At the pinned SDK (`@earendil-works/pi-coding-agent 0.75.5`) the type a tool's `execute()` resolves to is `AgentToolResult<TDetails>`, which is **declared in `@earendil-works/pi-agent-core`'s `dist/types.d.ts` (line 305)** and only *imported and re-exported* through `dist/core/extensions/types.d.ts` (the latter file imports it at line 10 and re-exports it at line 32; line 354 types `execute(...): Promise<AgentToolResult<TDetails>>`).

A literal-read surface-inventory assertion targeting the cited `dist/core/extensions/types.d.ts` therefore finds only a re-export line for `AgentToolResult`, not its field-set declaration. The promised "assert the field set on the cited type" check cannot resolve the type's members against the cited file; it must follow the re-export back to pi-agent-core to find the declaration site.

The miscitation appears to have arisen from locating the file where the token `isError` literally appears: `isError` does occur in `dist/core/extensions/types.d.ts`, but on TUI render-state and `tool_result`-event types (lines 323, 538, 634, 729), none of which is the `execute()` return type. The envelope-declaration sentence should point at the package and file that actually declare the return type.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `68ff2d1` — *pi-loom spec: resolve "Pi API surfaces asserted without .d.ts citations"*
**History:** The corpus is git-tracked. `git log -G` / `git log -S` over the envelope-declaration sentence and the phrase "declared on the tool-result type family" show the citation was added by `68ff2d1`, the commit that resolved a finding about Pi API surfaces lacking `.d.ts` citations. The task note that commit resolved (visible in its diff) located the declaration at `dist/core/extensions/types.d.ts` by where the `isError` token appears (lines 323/538/634/729), and the commit carried that file forward into the envelope sentence — citing the file where the token surfaces rather than the file where the `execute()`-return type is declared. The wrong-file citation has been present continuously since that commit; subsequent edits to the same paragraph (`ec6ab5c`, `f5e89f4`, and the scope-rename and SDK-pin sweeps) preserved the citation unchanged. The defect was introduced when the citation was first added, not by a later interaction.

## Solution Space

**Shape:** single

### Recommendation

Cite `@earendil-works/pi-agent-core`'s `dist/types.d.ts` as the declaration site of the `execute()`-return envelope type `AgentToolResult<T>`, and state that `@earendil-works/pi-coding-agent`'s `dist/core/extensions/types.d.ts` re-exports it (so a literal-read assertion against the type must follow the re-export to pi-agent-core). Frame the SDK surface-inventory assertion against the declaring package's file. Keep both packages within the four-package lock-step minor line per the Pi SDK pin, so the cited pi-agent-core file tracks the same `~0.75.5` minor as the re-exporting `pi-coding-agent` file.

Edge case for the implementer: the field set named in the prose must agree with whatever envelope contract the sibling `execute()`-`isError` finding settles — `AgentToolResult<T>` at the pin declares `{ content, details, terminate? }` and carries no `isError` member, so the corrected citation and the field set described must be reconciled together rather than leaving the citation pointing at a member the declaring type does not have.

## Consequence

**Severity:** advisory

A surface-inventory assertion written against the cited `dist/core/extensions/types.d.ts` would find a bare re-export of `AgentToolResult`, not its declared field set, so the "assert the field set on the cited type" check described here cannot be implemented as written without first chasing the re-export to `@earendil-works/pi-agent-core`. The correct declaration is discoverable via the re-export, so the cost is implementer friction and a misleading audit target rather than divergent runtime behaviour.
