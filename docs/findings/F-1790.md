## Finding

The `ValidationError` wire schema declares `attempts: number` with the inline comment `// respond-repair follow-ups made before giving up`. The field name is the unqualified noun `attempts`, which in similar APIs commonly counts *total tries* (the initial turn plus retries), while the comment claims it counts *follow-ups only* (initial turn excluded). The two conventions differ by one for every populated value and an implementer reading the schema alone has no in-shard signal to pick the comment's convention over the field-name convention.

The corpus does carry the disambiguating rule elsewhere — `query-failure-and-repair.md` QRY-70 says "`attempts` counts only follow-ups that produced an assistant response which the runtime then re-validated; the slot debits when that re-validation *runs*", `frontmatter-fields-b-and-templates.md` pins `attempts: 0` on every `methodology: none` path, and the ERR-17 terminal-exhaustion shape on the same page assigns `attempts: respond_repair.attempts` (the configured *follow-up* budget, not budget+1). The wire-schema comment is consistent with all three. But two conforming implementers reading the schema box at face value can still diverge: one populates `attempts` with the initial-included total ("3 tries: initial + 2 follow-ups → `attempts: 3`"), one with the follow-up-only count ("2 follow-ups → `attempts: 2`"). That divergence is silently wrong on the wire — every `match` arm and every conformance test that probes the value reads a different integer from the two runtimes.

The same ambiguity is reinforced by the field's siblings: the frontmatter key is `respond_repair.attempts` (a *cap on follow-ups*, per the explicit `attempts: 0` ≡ no follow-ups equivalence), so a reader who notices the name re-use across surfaces may even assume the wire field mirrors the frontmatter semantics, but nothing in the schema box says so.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `53f5831` (2026-05-03, "spec: rewrite query primitive, system prompts, and schema syntax")
**History:** the `ValidationError` schema was introduced in `53f5831` with the field `attempts: number, // coercion follow-ups made before giving up`. The wording has tracked only one rename since: `708a42f` (2026-05-06, "coercion overloaded: schema-validation repair vs. type conversion") swapped `coercion` for `respond-repair` across the spec, producing today's `// respond-repair follow-ups made before giving up`. The name-vs-comment shape — short field name `attempts`, long comment "<X> follow-ups made before giving up" — was present in the originating commit and was never edited to add a self-contained schema-level convention statement. The defect is the original design choice, not a regression.

## Solution Space

**Shape:** single

### Recommendation

Tighten the inline comment on the `attempts` field of the `ValidationError` schema box so the population convention is self-contained at the field. Replace

```
attempts: number,                          // respond-repair follow-ups made before giving up
```

with

```
attempts: number,                          // number of respond-repair follow-up turns issued (excludes the initial query; 0 = no follow-ups)
```

Keep the field name `attempts` (renaming would churn the wire surface, the frontmatter key `respond_repair.attempts`, the `<dotted-key>` enumeration in `diagnostics/placeholder-rendering-b.md`, and the four plan leaves above for no observable gain — and the comment alone removes the ambiguity). Make the same one-line tightening on the ERR-17 terminal-exhaustion shape immediately below in the same file, where the field re-appears with comment `// the configured budget`:

```
attempts: respond_repair.attempts,         // the configured follow-up budget (excludes the initial query)
```

The two edits sit in the same file, are co-resolved by one pass, and produce no cross-file ripple — every other site (QRY-70, `methodology: none`, glossary `respond-repair`) already states the convention compatibly. Do not add a separate normative bullet outside the schema box; the convention belongs on the field itself where a reader of the schema sees it.

## Consequence

**Severity:** advisory

A schema reader who never consults `query-failure-and-repair.md` (a separate page, separate topic file) and who follows the dominant `attempts = total tries` convention populates the field one greater than the spec intends. The divergence is silent on the wire (the field is still a `number`), corrupts `match` arms that test `attempts == N`, and breaks conformance tests built against the other implementation; it does not block implementation, but it makes two compliant-looking implementations disagree on every populated value.
