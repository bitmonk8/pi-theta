## Problem

`H3a`'s **Adds** field contains a Checkpoint parenthetical whose trailing
clause asserts that `V8a` lists `Checkpoint` under `host-interfaces-core.md`.
That claim is stale: `V8a` (and `V8a-T`) cite `host-interfaces-services.md`,
which is the only page that defines the `Checkpoint` seam contract (PIC-10) at
its `#checkpoint-seam` anchor. The result is a direct internal contradiction —
`H3a` says `V8a` cites `core`; `V8a` cites `services` — forcing a reader
auditing corpus consistency to confirm which leaf is correct.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 86ab9af (`pi-loom plan: resolve "V8a / V8a-T Spec field points at pages that do not define PIC-10"`, 2026-06-16)
**History:** The H3a parenthetical was introduced by 592219d (`pi-loom plan: resolve "H3a seam interfaces cite no source for member shapes"`, 2026-06-11), where it was accurate — at that point `V8a`'s Spec field did cite `host-interfaces-core.md`. Commit 86ab9af later corrected `V8a`'s (and `V8a-T`'s) Spec field from `host-interfaces-core.md` to `host-interfaces-services.md` to resolve a separate finding, but did not propagate that change to `H3a`'s cross-reference, leaving the parenthetical stale. The contradiction is thus the residue of a fix that updated `V8a` without updating the sibling note in `H3a` that described it.

## Solution approach

In `docs/plan_topics/H3a-di-seam-skeleton.md`, **Adds** field, delete the stale
trailing clause `, even though `V8a` lists `Checkpoint` under
`host-interfaces-core.md`` from the Checkpoint parenthetical, leaving it
anchored only at `host-interfaces-services.md#checkpoint-seam`.

## Solution constraints

- Out of scope: `V8a` and `V8a-T`, which already cite
  `host-interfaces-services.md` correctly and must not be edited.
