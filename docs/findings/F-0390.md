## Finding

The binder argument-echo implementation leaf `V11h` selects between the BNDR-4 integer rendering and the BNDR-5 number rendering by passing the canonical number renderer (`V2d`) an explicit `integer`-vs-`number` kind discriminator. Both `V11h`'s `Adds.` and `V2d`'s `Adds.` pin the origin of that discriminator: it is sourced from the bound value's **static type via the value model (`V2c`)**, never from the value's runtime integrality. `V11h` correctly lists `V2c` in its `Deps.` (`V11h-T, V11g, V2d, V2c`).

The paired tests-task leaf `V11h-T` lists `Deps.` of only `V11g, V2d`. To author the BNDR-6 echo reference renderings (6a–6x) and have them fail red for the intended reason, the test must construct bound values that carry the `integer`-vs-`number` static-type distinction that drives the discriminator — and that static-typed value representation is exactly what `V2c` introduces. `V2c` is therefore a real prerequisite of `V11h-T`, yet it is neither listed directly nor reachable transitively: `V2d`'s `Deps.` is `V2d-T, V2a`, and `V11g`'s chain (`V11a, V5d, V5f, V8c`) reaches none of them to `V2c`. Under the plan's dep-driven pickability predicate, `V11h-T` therefore reports as pickable before `V2c-complete` exists, even though its tests cannot be written against the absent value-model seam.

## Issue introduction

**Verdict:** introduced-by-later-edit
**Introducing commits:** `480795847ae5bf768848169f16856d833d435002` (2026-06-17, "pi-loom plan: resolve \"renderer input contract: integer-vs-number discriminator origin\"")
**History:** Both leaves were created together in `cb8737a` (2026-06-12, the over-bundling split), at which point neither cited `V2c` and there was no asymmetry. Commit `4807958` later added the discriminator-origin contract: it rewrote `V11h`'s `Adds.` to source the kind discriminator from the value model and appended `V2c` to `V11h`'s `Deps.`, and likewise touched `V2d` and `V5f`. That commit did **not** touch `docs/plan_topics/V11h-T-argument-echo.md` (its `git log --follow` history shows only `cb8737a`), so the new `V2c` dependency was added to the implementation leaf but never propagated to the paired test leaf — leaving the asymmetry this finding reports.

## Solution Space

**Shape:** single

### Recommendation

Add `V2c` to the `Deps.` field of `docs/plan_topics/V11h-T-argument-echo.md` so it reads `**Deps.** ` followed by `` `V11g`, `V2d`, `V2c` ``, matching the `V2c` dependency already carried by the paired implementation leaf `V11h`. The "confirm `V2d → V2c` makes it transitive" alternative from the raw report is not available: `V2d` depends only on `V2d-T` and `V2a`, and `V11g`'s dependency closure never reaches `V2c`, so the test leaf must list `V2c` directly. Cite `V2c` exactly as `V11h` does; do not introduce a parenthetical transitive-reachability note, because the edge is not transitive.

## Consequence

**Severity:** correctness

The missing `Deps.` edge makes `V11h-T` falsely pickable before the `V2c` value model is built. An implementer who picks it early has no static-type-carrying value representation to drive the `integer`-vs-`number` discriminator, so they invent an ad-hoc value construction that diverges from `V2c` — producing BNDR-6 tests that do not actually exercise the static-type discrimination the spec mandates (e.g. they fall back to runtime integrality, which `V2d` explicitly forbids as the discriminator source).
