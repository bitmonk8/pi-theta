## Problem

`V7a`'s **Adds** names the `loom-system-note` emission fallback chain (`sendSystemNote` → `ctx.ui.notify` → `console.error`) as a deliverable, but neither `V7a` nor `V7a-T` carries a Tests bullet that asserts it. The leaves' only failure-path bullet is the `PIC-21` renderer-body-throw assertion, which the spec designates a *separate* failure mode from the emission fallback chain — leaving the emission-degradation path entirely ungated. The fallback chain is a normative multi-step obligation in `runtime-event-channel.md` §System notes (ordered `ctx.ui.notify` → `loom/runtime/system-note-delivery-failed` diagnostic → `console.error`, with `display: false` / `content: ""` skip rules and terminal continue-on-failure semantics), none of which any assertion observes.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** `c6a664e` created `V7a`/`V7a-T` with the fallback chain already named in `V7a`'s Adds but with Tests limited to `DIAG-1`, the multi-error batch, and re-scan — no fallback-chain assertion. A later commit `af2ec83` (2026-06-10, "resolve 'PIC-21 renderer exception safety missing asserting test in V7a'") added the `PIC-21` renderer-body-throw bullet to both leaves in response to a prior review, but that addresses the *separate* renderer-throw failure mode and did not close the emission-fallback-chain gap, which has been open since inception.

## Solution approach

Add a Tests bullet to `docs/plan_topics/V7a-T-diagnostics-primitive.md` asserting the `loom-system-note` emission fallback chain against the obligation source `runtime-event-channel.md` §System notes, and mirror it into `docs/plan_topics/V7a-diagnostics-primitive.md`'s Tests so the paired lists stay 1:1. The bullet should gate the fallback ordering (`ctx.ui.notify(content, "error")` → `loom/runtime/system-note-delivery-failed` diagnostic → terminal `console.error`), the `display: false` / `content: ""` skip of `ctx.ui.notify`, the catch of a throwing `ctx.ui.notify`, and the continue-without-aborting terminal behaviour. Cite the `loom/runtime/system-note-delivery-failed` diagnostic code as the bullet's traceability token rather than `PIC-21`.

## Solution constraints

- Out of scope: the existing `PIC-21` renderer-body-throw bullet, which gates the separate renderer-throw failure mode — do not modify it or fold the emission-chain assertion into it.
- The spec is read-only for this fix; assert against `runtime-event-channel.md` §System notes without editing it.
