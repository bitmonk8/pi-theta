## Finding

`docs/spec_topics/discovery/package-and-settings.md` §"Package discovery" derives the per-read deadline as `deadline = max(200, floor(looms.scanPackagesTimeoutMs / 10))` ms, with the gloss "raising the global cap automatically raises the per-read budget" — i.e. the per-read deadline is meant to be a *fraction* of the global walk cap. The same file, in the "Scalar-key validation" block, accepts `looms.scanPackagesTimeoutMs` as "an integer ≥ 1".

For any operator-supplied `looms.scanPackagesTimeoutMs` between `1` and `199`, the floor branch produces `0`, the `max` clamps to `200`, and the per-read deadline therefore exceeds the global walk cap — a single `package.json` read is allowed to consume up to two orders of magnitude more time than the entire walk is budgeted for. That contradicts the "fraction of the global cap" intent and the surrounding "the global cap dominates" tone (the cap-check site is "*before each new candidate-package read attempt*", which presupposes the per-read share is smaller than the global share).

The spec is silent on which of the two rules governs in this range: an implementer reading the acceptance set first will permit small values and arm a `200 ms` timer that races a much smaller wall-clock budget; an implementer reading the formula first will treat `200 ms` as a hard floor that effectively overrides any smaller cap. Two reasonable implementations diverge on observable behaviour (whether the per-read `loom/load/package-read-timeout` warning can fire at all, and how quickly the global `loom/load/discovery-slow` trips), so the gap is not purely editorial.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `25e8a60` (2026-05-07, "pi-loom spec: resolve \"T08 — Per-package.json read timeout: overrun is unbounded\""), `cb82bbf` (2026-06-01, "pi-loom spec: resolve \"Settings scalar keys lack a malformed / out-of-range rule\"")
**History:** The per-read formula `deadline = max(200, floor(looms.scanPackagesTimeoutMs / 10))` was introduced by `25e8a60` when the spec resolved an earlier "overrun is unbounded" finding by wrapping each candidate read in a `Promise.race` against a derived deadline. At that point no lower bound on `looms.scanPackagesTimeoutMs` was documented, so the `max(200, …)` floor was implicit defence against pathologically small caps and the contradiction was latent. The acceptance set "`looms.scanPackagesTimeoutMs` — an integer ≥ 1" was added later by `cb82bbf` when "Settings scalar keys lack a malformed / out-of-range rule" was resolved; that commit established a per-key validation table without re-auditing the per-read formula, and the gap between "≥ 1 is accepted" and "per-read floor is `200`" became observable. Neither commit on its own is wrong — the defect is the interaction between the derived per-read formula and the independently-authored acceptance bound.

## Solution Space

**Shape:** multiple

### Option A — Clamp the per-read deadline by the global cap

**Approach.** Keep the `≥ 1` acceptance bound and rewrite the per-read formula as `deadline = min(looms.scanPackagesTimeoutMs, max(200, floor(looms.scanPackagesTimeoutMs / 10)))` ms (or equivalent prose: "the per-read deadline never exceeds the global cap; subject to that ceiling it is `max(200, floor(looms.scanPackagesTimeoutMs / 10))`"). When the operator cap is `< 200`, the per-read deadline collapses onto the global cap.

**Spec edits.** In `docs/spec_topics/discovery/package-and-settings.md` §"Package discovery" DISC-6 paragraph, replace the formula and add one sentence: "The per-read deadline is additionally clamped to never exceed `looms.scanPackagesTimeoutMs`, so a global cap below `200 ms` makes the per-read share equal to the global share and the global `loom/load/discovery-slow` warning is the only one that can fire."

**Pros.**

- Preserves the `≥ 1` acceptance bound, so the validation rule introduced by `cb82bbf` stays consistent with `looms.scanPackagesMaxFiles` (also `≥ 1`) and operators retain the full range of overrides.
- Honours the stated intent that "per-read is a fraction of the global cap" across the entire accepted range, not just `≥ 200`.
- No diagnostic vocabulary change; `V10c`'s settings-merge tests do not need new error paths.

**Cons.**

- The `200 ms` floor becomes a *ceiling-relative* floor rather than an absolute one — slightly harder to explain in prose.

**Risks.**

- `V10b`'s `DISC-6` "per-read deadline scales with the operator override" test fixture must cover the `< 200` regime (assert that `package-read-timeout` does not fire and `discovery-slow` fires first when the operator cap is, say, `50 ms`).

### Option B — Raise the accepted minimum for `looms.scanPackagesTimeoutMs`

**Approach.** Change "`looms.scanPackagesTimeoutMs` — an integer ≥ 1" to "an integer ≥ 200" (or a similar value at or above the per-read floor). Any value below the new minimum is treated as absent per the existing scalar-key validation rule and fires `loom/load/settings-value-out-of-range`. The per-read formula is unchanged.

**Spec edits.** In `docs/spec_topics/discovery/package-and-settings.md` §"Settings file reads" → "Scalar-key validation" acceptance list, replace `≥ 1` with `≥ 200` for `scanPackagesTimeoutMs` (and keep `≥ 1` for `scanPackagesMaxFiles`). Add a one-sentence rationale in the DISC-6 paragraph: "The minimum accepted value is `200 ms` so the per-read deadline can never exceed the global cap."

**Pros.**

- Keeps the per-read formula and its derivation gloss verbatim.
- Makes the `200 ms` floor an absolute floor that an implementer can reason about without re-deriving it from a `min/max` composition.

**Cons.**

- Creates an asymmetry between the two cap keys: `scanPackagesMaxFiles ≥ 1` but `scanPackagesTimeoutMs ≥ 200`, which is harder to memorise and adds an out-of-range branch operators can hit accidentally.
- An operator who explicitly wants very tight discovery latency (e.g. `100 ms`) cannot express it; the cap silently reverts to the `2000 ms` default with a non-fatal diagnostic.

**Risks.**

- `V10c`'s settings-merge tests gain a new out-of-range case for `scanPackagesTimeoutMs < 200` and must assert that the default is restored.

### Recommendation

Adopt **Option A**. Clamping the per-read deadline by the global cap honours the stated "fraction of the global cap" intent across the full accepted range, keeps the two cap keys symmetric on the `≥ 1` floor (the property `cb82bbf` established uniformly), and avoids forcing operators to choose between a tight walk budget and a working configuration. The implementer should note that when `looms.scanPackagesTimeoutMs` is `< 200`, the per-read `loom/load/package-read-timeout` warning becomes unreachable in practice — the global `loom/load/discovery-slow` warning is the only one observable in that regime — and the `V10b` test for the operator-override path should assert this collapse rather than fixture a per-read timeout there.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on observable behaviour when an operator sets `looms.scanPackagesTimeoutMs` below `200`: one fires `loom/load/discovery-slow` long before any per-read `package-read-timeout` is reachable, the other lets a single hung read consume many multiples of the configured walk budget. Conformance tests that exercise the operator-override path (`V10b`'s `DISC-6` "per-read deadline scales with the operator override" test) have no fixed expectation for this range.
