## Problem

`discovery-sources.md` pins `--loom` as a single `type:'string'` flag (multiple paths joined with `path.delimiter`) in its page-top source bullet, yet DISC-2 rules (2) and (3) still describe it as a repeatable flag: rule (2)'s diagnostic descriptor example `"--loom flag #1"` is flag-indexed, and rule (3) uses the plural "other `--loom` flags". This is residual cruft from before Decision #9 flipped the source to single. Under the single-flag model a repeated `--loom` is coalesced last-wins by Pi before loom observes it, so the plural/numbered phrasing describes behaviour the host cannot deliver, and implementers diverge on whether a second `--loom` is silently dropped, emits a diagnostic, or whether a per-flag-index descriptor ever fires.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `5b772b6` (2026-05-04, "pi-loom spec: resolve \"Discovery source failure-mode policy is unspecified\"") + `b5ba88e` (2026-05-05, "spec: --loom CLI flag is single string with OS path-list separator (Decision #9)")
**History:** Commit `5b772b6` introduced the DISC-2 failure-modes table together with rules (2) and (3) — including the plural `"--loom flag #1"` descriptor example and the "other `--loom` flags … still process to completion" wording — at a time when the `--loom` source was still authored as "`--loom <path>` (repeatable, optional)" in the page's top bullet. The next day, `b5ba88e` (Decision #9) flipped the source bullet to "single flag … joined with the OS path-list separator", noting in its own commit body that the change "removes the multiple-flags test (no longer representable in Pi's SDK)" and rewriting plan leaf V14o accordingly. That second commit edited only the source bullet and the registration step in `pi-integration-contract.md`; it did **not** sweep DISC-2 rules (2)/(3), leaving the plural-flag descriptor and prose stranded against the new single-flag model. The defect is therefore the *interaction* between the two commits — neither in isolation is wrong — and the durable fix is to bring DISC-2 into alignment with the single-flag model `b5ba88e` chose.

## Solution approach

Rewrite DISC-2 rule (2)'s `"--loom flag #1"` descriptor example so it indexes a path component within the single delimited `--loom` value rather than a flag occurrence, and state the indexing base. Rewrite rule (3)'s plural `other --loom flags` clause to operate at the path-component level, matching the per-component reality after the `path.delimiter` split. Add a sentence near DISC-2 (or the source bullet above it) pinning that a repeated `--loom` flag is coalesced last-wins by Pi before loom observes it — so no per-occurrence diagnostic is owed — and forward-link `host-prerequisites.md#loom-flag-namespace-presupposition`.

## Solution constraints

- Out of scope: the single-flag `--loom` source model (the page-top source bullet and `host-prerequisites.md#loom-flag-namespace-presupposition`) — align DISC-2 to it, do not change it.
- Do not mint a new `loom/load/*` diagnostic code; `loom/load/missing-source` remains the sole code for an explicit-source missing path.
