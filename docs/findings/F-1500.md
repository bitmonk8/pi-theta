## Finding

`docs/spec_topics/discovery/discovery-sources.md` pins the `--loom` CLI source as a **single** flag (`type:'string'`, multiple paths delimited with `path.delimiter`), explicitly because `pi.registerFlag` exposes no array type — the bullet at the top of the page says "single flag; multiple paths joined with the OS path-list separator". But three artefacts on the same page still read as if `--loom` could appear more than once on the CLI:

- The `loom/load/missing-source` descriptor example in DISC-2 rule (2) — `"--loom flag #1"` — is indexed, implying a numbered enumeration of multiple `--loom` occurrences.
- DISC-2 rule (3) — "other `--loom` flags … still process to completion" — uses the plural noun phrase.
- `loom/load/missing-source` itself is the only error code DISC-2 names for a missing `--loom` path; nothing scopes it to the path *component* level vs. the flag level.

The spec never reconciles these. What `pi.getFlag('loom')` actually observes for a repeated `type:'string'` flag is fixed by Pi at the pinned SDK: `pi.getFlag(name)` is typed `boolean | string | undefined` (`dist/core/extensions/types.d.ts`), the CLI parser stores unknown flags in a `Map<string, boolean | string>` keyed by name (`dist/cli/args.js`), and `applyExtensionFlagValues` writes that value into a per-name slot (`dist/core/agent-session-services.js`) — so two `--loom A --loom B` invocations are silently coalesced to last-wins (`B`); there is no per-occurrence enumeration loom can observe. The leftover plural / numbered phrasing therefore describes behaviour that the host cannot deliver, and two implementers reading DISC-2 will disagree on whether (a) a second `--loom` is silently dropped (matches Pi's actual behaviour and the single-flag pin), (b) the runtime emits a `loom/load/<…>` diagnostic for it, or (c) per-component descriptors like `"--loom flag #1"` ever fire (they cannot, given the single-flag model — components are addressable as path indices into the delimited string, not as flag indices).

The page's top bullet and the `pi.registerFlag('loom', …)` namespace presupposition in `pi-integration-contract/host-prerequisites.md` (Authoritative source) leave no room for the plural model; the DISC-2 prose is residual cruft from the pre-Decision-#9 era when `--loom` was authored as a repeatable flag.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `5b772b6` (2026-05-04, "pi-loom spec: resolve \"Discovery source failure-mode policy is unspecified\"") + `b5ba88e` (2026-05-05, "spec: --loom CLI flag is single string with OS path-list separator (Decision #9)")
**History:** Commit `5b772b6` introduced the DISC-2 failure-modes table together with rules (2) and (3) — including the plural `"--loom flag #1"` descriptor example and the "other `--loom` flags … still process to completion" wording — at a time when the `--loom` source was still authored as "`--loom <path>` (repeatable, optional)" in the page's top bullet. The next day, `b5ba88e` (Decision #9) flipped the source bullet to "single flag … joined with the OS path-list separator", noting in its own commit body that the change "removes the multiple-flags test (no longer representable in Pi's SDK)" and rewriting plan leaf V14o accordingly. That second commit edited only the source bullet and the registration step in `pi-integration-contract.md`; it did **not** sweep DISC-2 rules (2)/(3), leaving the plural-flag descriptor and prose stranded against the new single-flag model. The defect is therefore the *interaction* between the two commits — neither in isolation is wrong — and the durable fix is to bring DISC-2 into alignment with the single-flag model `b5ba88e` chose.

## Solution Space

**Shape:** single

### Recommendation

Bring DISC-2 into alignment with the single-flag-with-delimiter model `pi.registerFlag('loom', { type: 'string' })` forces, by editing `docs/spec_topics/discovery/discovery-sources.md` DISC-2 rules (2) and (3) only:

- Rule (2) descriptor list: replace `"--loom flag #1"` with a component-indexed example such as `"--loom path component #1"` (or equivalently `"--loom component #1"`), and add a parenthetical clause anchoring the index to "the zero-or-one-based position of the path inside the `path.delimiter`-split list, per the single-flag source bullet above". Pick one indexing base and state it; do not leave it open.
- Rule (3): replace "other `--loom` flags and the other four sources still process to completion" with "other `--loom` path components and the other four sources still process to completion", so the per-entry fatality phrasing matches the per-component reality. The "explicit user intent → error on missing/unreadable/wrong-type" semantics in the table row already operate at the component level after the `path.delimiter` split (per the *Discovery roots* paragraph and the registration-steps step-1 split), so this is a wording fix, not a behavioural change.
- Add one sentence to DISC-2 (or to the source bullet immediately above the failure-modes table) pinning the disposition of a repeated `--loom` flag on the CLI: under the [`loom-flag-namespace-presupposition`](../pi-integration-contract/host-prerequisites.md#loom-flag-namespace-presupposition) `pi.getFlag('loom')` reads a single shared-slot value, so a repeated `--loom` is silently coalesced to a single string by Pi before loom observes it; the loom runtime emits no diagnostic for the second occurrence because no second occurrence is observable. Cross-link the host-prerequisite presupposition.

Edge cases the implementer must watch:
- The `loom/load/missing-source` descriptor string is reader-facing; pick the new wording so an author can map it back to a position inside their `--loom A:B:C` argument unambiguously (e.g. component #2 = `B`).
- No new diagnostic code is owed — `loom/load/missing-source` remains the single code for an explicit-source missing path, regardless of whether the path arrived through settings or a `--loom` component.
- The V10a / V10a-T leaves should gain a vector asserting Pi's last-wins coalescing is observable (only one path's worth of components reaches the walk) so an implementation that tried to read a per-occurrence enumeration through some other Pi surface would fail.

## Consequence

**Severity:** correctness

Two implementers diverge on (a) whether the loom runtime is obligated to surface a diagnostic per redundant `--loom` occurrence (it cannot, since the host only delivers one value), and (b) what `<n>` ever evaluates to in a `"--loom flag #<n>"` descriptor (only ever `#1` under the single-flag pin, making the index dead text). Either implementation passes the spec as written; only one matches Pi's actual SDK behaviour.
