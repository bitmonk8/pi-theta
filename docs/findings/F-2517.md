## Finding

When a `.loom` file is referenced by another loom's `tools:`, the runtime materialises a Pi `ToolDefinition` for it so the model can invoke it. The field-derivation list in [`extension-bootstrap-and-per-loom.md` — Per-loom registration](../../docs/spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md) fixes the five required fields (`name`, `label`, `description`, `parameters`, `execute`) but leaves three derivation preconditions unpinned, each of which two reasonable implementers would resolve differently:

1. **Failure-result envelope of the `.loom`-as-tool `execute` adapter.** The `execute` derivation says only that the adapter "spawns the loom as a subagent invocation (equivalent to `invoke<T>(path, ...)`) and **returns its result envelope**" — the success path. A model-invoked `.loom`-as-tool runs `execute` from Pi's tool-call batch and must return a Pi `AgentToolResult` (`{ content, isError }`) to the model, but the spec never states how an *in-loop* failure of the spawned invocation maps onto that envelope: a callee `Err` (`InvokeCalleeError` / `InvokeInfraError`), a panic inside the subagent, or a mid-run cancellation. [`tool-calls.md`](../../docs/spec_topics/tool-calls.md) pins only the *pre-evaluation setup throw* case (lowered to `{ isError: true, content: [{ type: "text", text: "loom <name> aborted with internal error: …" }] }`) and the *post-cancel late-settlement discard*; it does not pin what `execute` resolves with once the loom body has begun and then fails or is cancelled. The code-driven path (loom code calling a `.loom` callable) is fully specified — failures cascade as `InvokeCalleeError` / `InvokeInfraError` `Result`s — but that contract is for a value loom code `match`es on, not for the `{ content, isError }` value a model-invoked adapter must feed back. The model-driven failure-result envelope is the gap.

2. **Frontmatter `description` requiredness.** The list asserts all five `ToolDefinition` fields are "required by Pi's `ToolDefinition` interface" and derives `description` as "the loom's frontmatter `description`". But frontmatter `description` is **optional** — [`frontmatter-fields-a.md`](../../docs/spec_topics/frontmatter/frontmatter-fields-a.md) marks it `required: no`, default `null`, with the explicit note that "internal-only looms legitimately omit this." A `.loom` whose frontmatter omits `description` therefore derives a `null` (absent) value for a field Pi requires, and the spec defines no behaviour for that case — neither a synthesised default, nor a load-time diagnostic refusing to expose a description-less loom as a tool, nor a requiredness assertion scoped to tool-exposed looms.

3. **`label` capitalisation for non-cased leading characters.** The `label` derivation is "the loom file's basename with hyphens preserved and the leading character capitalised" (`summarise.loom` → `"Summarise"`, `code-review.loom` → `"Code-review"`). The transform is undefined when the basename's leading character has no uppercase form — a digit, an underscore, or a non-ASCII codepoint — so two implementers diverge on the label for, e.g., `2fa-check.loom` or `_internal.loom`.

The original framing of this finding also alleged that the synthesised typed-query respond tool's `name` and its `<slug>` token are undefined. That sub-claim does not hold: [`implementation-notes.md` — Runtime](../../docs/spec_topics/implementation-notes.md) names the tool `__loom_respond_<slug>` and defines `<slug>` as the schema slug of the lowered response schema produced by the canonical-schema-hash recipe ([`schema-subset.md` — Canonical schema hash / Synthesised names](../../docs/spec_topics/schema-subset.md)), and the non-compliance diagnostics (ERR-17, QRY-9) key on that fully-defined name. The `label` special-case in the field-derivation list is present precisely because the synthesised respond tool's full shape (name, label, parameters, execute) is owned at that authoritative site. That sub-claim is dropped here; the three obligations above are the real defects.

## Consequence

**Severity:** correctness

A model-invoked `.loom`-as-tool whose body fails or is cancelled has no specified tool-result, so two implementers feed the model different (or no) `{ content, isError }` envelopes, diverging on whether the model sees the failure at all and how it retries. A description-less loom exposed as a tool either crashes Pi's required-field check or registers an ill-formed `ToolDefinition`, depending on the implementer. The `label` edge case produces non-deterministic tool labels for digit/underscore/non-ASCII-leading basenames.
