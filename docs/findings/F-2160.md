## Finding

The Type-grammar section of `grammar.md` enumerates the positions where a bare `Type` may appear and ends the list with "`invoke<Type>` / type-ascription contexts" (and again in the lowered-schema `Result` carve-out). It also lists "a type ascription" as one of the `void`-rejecting `Type` positions. But no `TypeAscription` production exists in the grammar, the phrase is never defined, and it carries no cross-reference, so a reader cannot map "type-ascription context" to a concrete syntactic surface.

Loom 1.0 has exactly two inline angle-bracket type-annotation surfaces — the `invoke<Type>` return annotation (already named in the same sentence) and the query `@<Schema>` explicit-ascription form (`query/query-forms.md`, QRY-3/QRY-4). The keyword `as` is reserved, but `lexical.md` reserves it solely for the wire-name rename clause (`as "WireName"`); there is no `expr as Type` cast operator anywhere in the corpus. A reader therefore cannot tell whether "type-ascription context" denotes the existing `@<Schema>` query form or implies a distinct, unparsed ascription syntax they must build.

The same undefined term propagates into `diagnostics/code-registry-parse.md`: the `loom/parse/void-in-non-return-position` row lists "type ascription" among the rejecting positions. Because the term resolves to no concrete site, that diagnostic's position list lacks a testable syntactic vector for the ascription case.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `f841b12` ("spec: resolve Cluster 2 (literal sublanguage + grammar appendix)", 2026-05-04)
**History:** `git log -S 'type-ascription contexts' --reverse --all` identifies `f841b12` as the first commit to introduce the phrase, in the then-`spec_topics/grammar.md` Type-grammar section, as part of the bare-`Type`-positions enumeration ("…and `invoke<Type>` / type-ascription contexts"). It was never accompanied by a production or cross-reference. The phrase carried forward unchanged through the `docs/` relocation (`31ff060`, 2026-05-08). The void-grammar reconciliation (`4c76dfb`, 2026-06-05) then propagated the same undefined term ("a type ascription") into the new `void`-in-non-return-position enumeration and its diagnostics-registry row. The defect has been present since the term was coined and was amplified, never repaired, by the later edit.

## Solution Space

**Shape:** single

### Recommendation

In `grammar.md`, replace the undefined "type-ascription contexts" phrase — in both the bare-`Type`-positions sentence and the lowered-schema `Result` carve-out — with the two concrete inline angle-bracket annotation surfaces it denotes: the `invoke<Type>` return annotation and the query `@<Schema>` explicit-ascription form. Add forward-links to [Invocation — Typed return](../invocation.md) and [Query — Explicit form](../query/query-forms.md#explicit-ascription-override).

Rename the `void`-position "a type ascription" entry in the same paragraph, and the matching "type ascription" item in the `loom/parse/void-in-non-return-position` row of `diagnostics/code-registry-parse.md`, to the concrete `@<Schema>` query ascription, so both name a cross-referenceable, testable site (e.g. `@<void>\`...\``).

State that loom 1.0 defines no standalone `expr as Type` cast operator, bounding the term to these inline annotation forms. Watch one consistency point: the lowered-schema `Result` carve-out currently admits `Result` in "`invoke<Type>` / type-ascription contexts" — keep that admission accurate against the renamed phrase (`Result` is admitted in an `invoke<Result<…>>` annotation but a `@<Schema>` query ascription is a lowered-schema position, so `Result` there is `loom/parse/result-in-schema-position`).

## Consequence

**Severity:** advisory

An implementer cannot enumerate the "type-ascription" positions from the spec text; a literal reading of "type-ascription context" could prompt building an `expr as Type` cast operator that loom 1.0 does not define, or leave the `void`-in-ascription rejection untested. A careful reader can still ship a correct parser by treating the two existing angle-bracket surfaces as the bounded set, so the gap is guidance-level rather than behaviour-diverging.
