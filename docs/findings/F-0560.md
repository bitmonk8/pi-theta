## Finding

`V14b` (`docs/plan_topics/V14b-tool-calls-parallel-batch.md`) states the model-driven parallel `tool_use` batch behaviour as established fact — "the loom runtime awaits every sibling call in a model-issued batch to settle via `Promise.allSettled` … before it constructs the next user turn, and lowers each sibling's outcome independently." That behaviour rests on a Pi host property: that Pi's model-driven "parallel tool mode" runs the model-issued sibling `tool_use` blocks concurrently and that the runtime can observe every sibling settle before the next user turn is constructed (the property the spec describes at [`tool-calls.md` §Concurrency](../../docs/spec_topics/tool-calls.md#concurrency)). V14b verifies this only through the in-process Pi session double, and `H4a`'s session-double fidelity contract enumerates only four axes (streamed-token order, single-turn append semantics, the `pi.on` cancel-forward subscription, cancellation propagation) — parallel-batch delivery/settle is not among them.

The structurally identical `V12a` leaf already follows the project's established pattern for exactly this situation: its SLSH-2 bullet names the real-Pi streaming-ordering behaviour it models as "an unpinned host presupposition audited by editorial review per [bump-checklist item (ak)]". V14b carries no analogous surfacing, and no item (a)–(ak) of the version-bump editorial-review checklist ([`version-bump-step2.md`](../../docs/spec_topics/pi-integration-contract/version-bump-step2.md)) covers parallel-tool-mode batch delivery. Item (v) covers turn-lifecycle event delivery (`tool_call`/`tool_result`/…) but not the concurrent-execution + settle-all-before-next-turn semantic V14b depends on.

Because the only verification footing is the double, a future Pi minor that changed parallel-tool-mode delivery (e.g. surfacing siblings one at a time, or constructing the next turn before all siblings settle) would leave V14b green against the double while diverging from real Pi, with no detection hook anywhere in the plan or the bump procedure.

## Solution Space

**Shape:** single

### Recommendation

Mirror the established `V12a`/SLSH-2 → bump-checklist-item-(ak) pattern for the parallel-tool-mode presupposition, in two coordinated edits landed together (the spec edit first, so the plan citation points at a real anchor):

1. **Spec — `docs/spec_topics/pi-integration-contract/version-bump-step2.md`.** Add a new editorial-review checklist item (next free letter, e.g. `(al)`, with its own `<a id="bump-checklist-...">` anchor) routing the parallel-tool-mode batch-delivery presupposition to editorial review, on the same SHOULD-level footing and escalation-clause shape as items (f)–(ak). Its audit subject: on each Pi minor bump, confirm against the candidate `@earendil-works/pi-coding-agent` minor that Pi's model-driven parallel tool mode still (a) runs the model-issued sibling `tool_use` blocks of one assistant message concurrently and (b) lets the runtime await every sibling to settle before constructing the next user turn, with each sibling's outcome lowered independently into its own `tool_use` result block. The item's presupposition anchor should live in `docs/spec_topics/tool-calls.md` §Concurrency (where the behaviour is already described as fact); version-bump-step2.md's own closing maintenance rule — "Any future presupposition added to this page or to `future-considerations.md` that routes its detection to editorial review under this procedure MUST be added to this checklist in the same edit" — requires the checklist item and the routed presupposition note to land in the same edit.

2. **Plan — `docs/plan_topics/V14b-tool-calls-parallel-batch.md`.** Add a sentence (in the Adds paragraph or the `cka-13` Tests bullet) stating that the real-Pi parallel-tool-mode batch behaviour V14b models — concurrent sibling execution and settle-all-before-next-turn — is an unpinned host presupposition audited by editorial review per the new bump-checklist item, with a link to its anchor, mirroring V12a's SLSH-2 bullet pointer to item (ak).

Edge case for the implementer: the spec presupposition note and the new checklist item must be a single edit (per the maintenance rule above); do not land the checklist item without the §Concurrency anchor it routes, or vice versa.

## Consequence

**Severity:** advisory

V14b can still be implemented and shipped from its current text; the gap is a missing fidelity-drift detection hook. A Pi minor that alters parallel-tool-mode batch delivery would pass V14b's double-backed gate while real Pi diverges, and — unlike the streaming-ordering axis V12a surfaced — nothing routes that divergence to the version-bump audit, so it ships undetected.
