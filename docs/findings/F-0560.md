## Problem

`V14b` (`docs/plan_topics/V14b-tool-calls-parallel-batch.md`) states the model-driven parallel `tool_use` batch behaviour — concurrent sibling execution, settle-all-before-next-turn, and independent per-sibling lowering — as established fact, but verifies it only through the in-process Pi session double. That double's fidelity is fixed by `H4a`'s contract, which enumerates only four axes (streamed-token order, single-turn append semantics, the `pi.on` cancel-forward subscription, cancellation propagation); parallel-batch delivery/settle is not among them. Unlike the structurally identical `V12a` leaf — whose `SLSH-2` bullet surfaces its analogous real-Pi streaming-ordering behaviour as an unpinned host presupposition routed to editorial review per bump-checklist item (ak) — V14b carries no such surfacing, and no item (a)–(ak) of `version-bump-step2.md` covers parallel-tool-mode batch delivery. A future Pi minor that changed parallel-tool-mode delivery would leave V14b green against the double while diverging from real Pi, with no detection hook in the plan or the version-bump procedure.

## Solution approach

Mirror the established `V12a`/`SLSH-2` → bump-checklist pattern for the parallel-tool-mode presupposition. Add a new editorial-review checklist item (next free letter) to `docs/spec_topics/pi-integration-contract/version-bump-step2.md`, on the same SHOULD-level footing as items (f)–(ak), routing the parallel-tool-mode batch-delivery presupposition to editorial review, with its presupposition note anchored in `docs/spec_topics/tool-calls.md` §Concurrency where the behaviour is already described. Add a sentence to V14b's Adds paragraph or its `cka-13` Tests bullet stating that the real-Pi parallel-tool-mode batch behaviour V14b models is an unpinned host presupposition audited by editorial review per the new checklist item, linking to its anchor, mirroring V12a's SLSH-2 pointer to item (ak). Land the spec edit first so the plan citation resolves.
