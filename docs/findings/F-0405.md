## Finding

`V16a` declares a stateless Class-2 cross-leaf seam `arbitrate(candidate) → { surfaced, masked? }`. Its **Output** half is pinned to the field level: `surfaced` is the single string identifier drawn from the closed set `"ceiling#1"` … `"ceiling#4"`, and `masked` is the closed-set enumeration of co-fired siblings, omitted when empty (never `masked: []`). The **Input** half — the *ceiling-candidate* — is described only in prose: "the ceiling class(es) whose precondition is satisfied at a single check site, tagged with the check-site / ceiling-class the candidate carries," with the four check-site→ceiling-class mappings spelled out in parentheses. No field names and no struct shape are given for the candidate.

Five leaves consult this seam and bind it through their `Deps` on `V16a`: `V5e` (ceiling-#4 first-enforcement, populates `masked`), `V11f` (ceiling-#3 slash-load), `V13c` (ceiling-#2 round-boundary), `V15b` (ceiling-#1 `invoke`-entry), and the load-time `V4e` slash-load `params` cross-route. Three of these are authored outside `V16a`'s shard (`V5e`/`V11f`/`V4e`), so their authors reason about the candidate they must construct from the seam's leaf text rather than from shared in-shard context. With the input contract unpinned, each producer must guess the candidate's field shape — in particular how the check-site tag is represented and how *co-present* sibling ceiling classes are carried — while the output it consumes is exactly specified. The asymmetry between a field-pinned output and an unpinned input is the defect.

This is the producer-side face of the same gap the consumer-side finding "V5e/V4e assume V16a's cross-ceiling arbitration seam shape" records: the consult sites assume a seam shape that `V16a` never pins.

**Verdict:** single-commit

## Solution Space

**Shape:** multiple

### Option A — Pin the candidate fields in `V16a`'s leaf text

**Approach.** Give the Input bullet the same field-level treatment the Output bullet already has: name the candidate's fields — the check-site / ceiling-class tag and the set of co-present satisfied ceiling classes — with their domains drawn from the same `"ceiling#1"` … `"ceiling#4"` closed set.

**Plan edits.** Edit the `**Input**` bullet in `docs/plan_topics/V16a-ceiling-order-masked.md`'s Adds to enumerate the candidate's fields and each field's value domain.

**Spec edits.** None.

**Pros.** Keeps the full seam contract human-readable in the leaf, symmetric with the Output bullet; cross-shard producers resolve against leaf prose without reading code.

**Cons.** Risks drift between the prose field list and the actual exported TypeScript type once `V16a` is implemented.

**Risks.** A future type change to the candidate must be mirrored in prose or the leaf text goes stale.

### Option B — Declare the exported candidate type the binding contract

**Approach.** State in `V16a`'s Input bullet that `V16a`'s exported TypeScript candidate type is the binding Class-2 seam contract that every consult-site `Deps` on `V16a` resolves against, consistent with the Class-2 "named cross-leaf seam … imported and consulted directly" model already declared in the leaf and in `conventions.md` §Adds.

**Plan edits.** Add a sentence to the `**Input**` bullet in `docs/plan_topics/V16a-ceiling-order-masked.md` naming the exported candidate type as the binding contract; the five consult-site leaves' Deps language already binds the seam, so they reference that type.

**Spec edits.** None.

**Pros.** Single source of truth in code, no prose/type drift; matches how the five producers actually compile against the seam.

**Cons.** The candidate shape is not legible from the leaf text alone; a reader must consult the exported type.

**Risks.** Until `V16a` lands the type, downstream authors have no concrete shape to read; the binding-contract statement defers, rather than pins, the shape in the planning artefact.

### Recommendation

Adopt Option B: state in `V16a`'s Input bullet that the exported candidate type is the binding Class-2 seam contract all consult-site `Deps` resolve against. It removes prose/type drift and aligns with the Class-2 seam mechanism the leaf already invokes and the way the five cross-shard producers consume the seam. Edge case the implementer must watch: the candidate must represent the *set* of co-present satisfied ceiling classes at a single check site, not a single ceiling class — `masked` (CIO-6) enumerates co-fired siblings, so a single-ceiling-class input would make co-fire enumeration unrepresentable.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: `V16a`'s author picks a candidate input shape, while the five cross-shard consult-site authors construct candidates against the shape they each inferred from prose. Mismatched field names or representations cause the producers' construction code to fail to type-check against the seam, or — worse — to mis-populate the inputs that drive the `masked` co-fire enumeration, so a co-fire surfaces the wrong `surfaced`/`masked` pairing without a compile error.
