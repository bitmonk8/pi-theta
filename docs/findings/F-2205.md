## Problem

A single underlying failure — AJV (or the depth walk) rejecting a data payload — is keyed under two different `cause` tokens depending on the `QueryError` variant carrying it: `ValidationError` names the AJV-rejection arm `"schema_validation"`, while `CodeToolError` and `InvokeInfraError` name the equivalent input-rejection arm `"validation"`. An author pattern-matching on `.cause` across variants must remember the validation-failure token is spelled differently for `ValidationError` than for the tool-call and invoke variants. The divergence is deliberate at the `ValidationError` site (the longer token disambiguates the AJV arm from the `"empty_template"` arm within `kind: "validation"`), but the spec records no rationale at the point of definition, so the asymmetry reads as an accidental inconsistency.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 64f30b4 — pi-loom spec: add ValidationError.cause field (schema_validation | empty_template) (2026-05-06, Thomas Andersen)
**History:** The bare `cause: "validation"` token for the tool-call/invoke variants is the older of the two spellings, present since the tool-call error variant was first introduced. `ValidationError` originally had no `cause` field at all — it was a single undifferentiated `kind: "validation"` variant. Commit 64f30b4 added the two-arm `cause: "schema_validation" | "empty_template"` partition and chose `schema_validation` for the AJV-failure arm to contrast with `empty_template` within the same `kind`, even though the sibling variants already named the same AJV-rejection concept `validation` — that commit's own added prose flags the sibling pattern ("consistent with the established `CodeToolError.cause` … patterns"). The asymmetry was introduced at that point and carried verbatim through the f5e89f4 size-cap split that moved the union into `queryerror-variants.md`.

## Solution approach

Add a clarifying note co-located with the `ValidationError.cause` definition in the `cause` prose block under *Query-time variants*, stating that `"schema_validation"` is the qualified spelling chosen to contrast with the sibling `"empty_template"` arm within `kind: "validation"`, and that it denotes the same AJV-rejection concept that the tool-call and invoke variants key as `"validation"`.

## Solution constraints

- Make no wire-token changes — keep `"schema_validation"`, `"empty_template"`, and `"validation"` as-is across all three variants.
