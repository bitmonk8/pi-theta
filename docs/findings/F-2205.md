## Finding

A single underlying failure — AJV (or the depth walk) rejecting a data payload — is given two different `cause` token names depending on which `QueryError` variant carries it. In `ValidationError` the AJV-rejection arm is `cause: "schema_validation"` (contrasting, within the same `kind: "validation"`, with `cause: "empty_template"`). In `CodeToolError` the equivalent input-rejection arm is `cause: "validation"`, and in `InvokeInfraError` the input-side arm is likewise `cause: "validation"` (its comment even reads "input side, like CodeToolError.cause 'validation'").

The result is that the same observable event — schema validation rejected this payload — is keyed as `"schema_validation"` on one variant and `"validation"` on two others. An author writing handling that pattern-matches on `.cause` across `QueryError` variants must memorise that the validation-failure token is spelled differently for `ValidationError` than for the tool-call and invoke variants. The divergence is deliberate at the `ValidationError` site (the longer token disambiguates the AJV arm from the `empty_template` arm), but nothing in the spec records that rationale at the point of definition, so the asymmetry reads as an inconsistency rather than an intentional choice.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 64f30b4 — pi-loom spec: add ValidationError.cause field (schema_validation | empty_template) (2026-05-06, Thomas Andersen)
**History:** The bare `cause: "validation"` token for the tool-call/invoke variants is the older of the two spellings, present since the tool-call error variant was first introduced. `ValidationError` originally had no `cause` field at all — it was a single undifferentiated `kind: "validation"` variant. Commit 64f30b4 added the two-arm `cause: "schema_validation" | "empty_template"` partition and chose `schema_validation` for the AJV-failure arm to contrast with `empty_template` within the same `kind`, even though the sibling variants already named the same AJV-rejection concept `validation` — that commit's own added prose flags the sibling pattern ("consistent with the established `CodeToolError.cause` … patterns"). The asymmetry was introduced at that point and carried verbatim through the f5e89f4 size-cap split that moved the union into `queryerror-variants.md`.

## Solution Space

### Recommendation

In the `cause` prose block under *Query-time variants* in `docs/spec_topics/errors-and-results/queryerror-variants.md`, add a one-sentence note co-located with the `ValidationError.cause` definition (not in a *Notes* section) stating that `"schema_validation"` is the qualified spelling chosen to contrast with the sibling `"empty_template"` arm within `kind: "validation"`, and that it denotes the same AJV-rejection concept that the tool-call and invoke variants key as `"validation"`. Keep all three wire tokens as-is; make no wire-token changes and no edits outside this file.

## Consequence

**Severity:** advisory

Implementers transcribe the wire tokens verbatim, so no runtime behaviour diverges. The cost falls on loom authors writing cross-variant `cause` handling, who must learn that the AJV-rejection token is `"schema_validation"` on `ValidationError` but `"validation"` on `CodeToolError` / `InvokeInfraError`; the unrecorded rationale makes the split look accidental and invites a "fix" that breaks one site or the other.
