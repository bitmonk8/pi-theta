## Finding

The closed continuation-trigger table in `grammar.md` § Newline continuation lists `?` as part of the trailing **binary or ternary** operator set (`+ - * / % == != < <= > >= && || ? :`). Read literally, a line ending in `?` therefore continues across the newline.

But `?` is also loom's postfix error-propagation operator (`expressions.md` § Postfix error-propagation, precedence level 1, owned by `errors-and-results/error-model.md`), which is a *completed* expression — `result?` already evaluates to the unwrapped `Ok` value or `return Err(e)`s. A literal application of the trigger table to source like

```
let x = result?
next_stmt()
```

makes the two lines one statement (parse failure / unrelated diagnostic), even though every other page treats `result?` as a complete expression and `next_stmt()` as a new statement. The trigger set is declared "closed" and the table is normative, so no implementer is licensed to silently carve postfix `?` out; two conforming front-ends will diverge on whether a trailing `?` ends a statement.

The same ambiguity does not affect the leading-operator row: a line beginning with `?` is unambiguously the ternary `?` (postfix `?` never starts a line on a fresh operand). The defect is isolated to the trailing-operator row.

## Solution Space

**Shape:** single
**State:** shaped

### Recommendation

Drop `?` from the trailing-operator set in the continuation-trigger table; postfix `?` is then unambiguously a statement-completing token, and ternary continuation across a newline must use the leading-operator form (`cond\n  ? a : b`), which is already a documented trigger.

Spec edits:
- `grammar.md` § Newline continuation, table row 2: change the trailing-operator string to `+ - * / % == != < <= > >= && || :` (drop `?`). Add one sentence after the table: "A trailing `?` is the postfix error-propagation operator and is never a continuation trigger; ternary continuation across a newline uses the leading-operator form (`cond\n  ? a : b`)."
- `lexical.md` § Statement terminators: mirror the change in the inline prose ("a trailing binary or ternary operator" → "a trailing binary operator or `:`") so the two sites do not redrift.
- `V1a` / `V1a-T`: drop the trailing-`?` positive vector if one was planned; add a vector that `result?\nnext` tokenises as two statements; keep the leading-`?` ternary vector.

Rationale and edge cases: the postfix `?` operator is the dominant user-facing form (every `Result` consumer uses it); the trailing-form ternary is an editorial preference with a working substitute (leading-operator), and lexer simplicity is the section's stated design goal (the trigger set is closed). The lexer needs no parse-state context (ternary nesting is not tracked). Edge case to watch in V1a-T: assert that `cond ?\n  a : b` is **not** accepted as continued (`?` no longer trails), and that `cond\n  ? a : b` **is** accepted (leading-operator path unchanged).

## Consequence

**Severity:** correctness

Two reasonable implementations of the trigger table diverge on `result?\n<next>`: one continues (per the literal trailing-`?` rule), the other closes the statement (per postfix `?` being a completed expression). The disagreement is silent — both can claim conformance — and shows up only as a parse-error mismatch on real loom code that uses the dominant `?` form.
