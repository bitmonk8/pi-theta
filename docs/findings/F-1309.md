## Finding

The watcher's atomic `LoomRegistry` rebuild mechanism is named "build-aside-then-publish" throughout the spec — `registration-steps.md` (step 5), `code-registry-runtime.md`, `host-interfaces-services.md`, `session-shutdown-semantics.md`, `version-bump-step2.md`, and `discovery/package-and-settings.md` all use that exact compound. The coverage-matrix row `cka-23` matches it ("step 5 build-aside-then-publish"). The `V9b` **Adds** bullet, however, calls the same mechanism "`LoomRegistry` build-aside-publish swap" — dropping the "then" that the spec's canonical term carries. A reader who greps the corpus for the spec/`cka-23` term "build-aside-then-publish" does not find the `V9b` **Adds** mention of the very mechanism `V9b` owns.

The other surfaces the original finding flagged are already spec-consistent and need no change: the `V9b` / `V9b-T` **Tests** bullets use the link label "registry swap" (the spec's own generic noun for the operation, e.g. "a registry swap takes effect", "atomic registry swap" in `pi-integration.md`) plus the parenthetical gloss "the swap is atomic (build-aside, then publish)", and the `V9b` **Ships when** bullet uses "(build-aside then publish)". Those are faithful prose renderings of the spec mechanism. The single out-of-step token is the hyphenated compound in `V9b` **Adds**.

There is no separate plan-level "terms index" defining "build-aside-publish swap"; the divergent compound appears only in `V9b` **Adds**. The facet sub-label `cka-23.swap` mechanically ties the `V9b` / `V9b-T` **Tests** bullets to the `cka-23` coverage row regardless of the prose wording, so coverage gating is unaffected — this is a term-wayfinding inconsistency, not a coverage or behaviour gap.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** The divergent compound "build-aside-publish swap" (no "then") entered the corpus when the `V9b` leaf was first authored at `c6a664e`, in the file then named `V9b-registration-drain-state.md` (later renamed to `V9b-registration-reload-wiring.md`). The spec's canonical "build-aside-then-publish" already predated that authoring. Later commits that touched the same `V9b` **Adds** line (`324813d`, a `V4e`→`V4g` rename; `44a96f5`, a slug/scope resolution) preserved the divergent token unchanged. The defect is an authoring divergence present from the leaf's inception, not a regression introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9b-registration-reload-wiring.md`, **Adds** bullet, change the token `LoomRegistry build-aside-publish swap` to `LoomRegistry build-aside-then-publish swap`, matching the spec (`registration-steps.md` step 5) and the `cka-23` coverage row. The spec is read-only for this fix — it already carries the canonical term; align the plan to it. No edit is required to `V9b` **Tests** / **Ships when** (their "registry swap" link label and "(build-aside, then publish)" gloss already track the spec) or to `V9b-T` or `cka-23`.

## Consequence

**Severity:** cosmetic

A reader scanning for the spec's "build-aside-then-publish" term misses the `V9b` **Adds** description of the mechanism that leaf owns; the variant compound suggests two distinct operations where there is one. No implementer behaviour diverges — the mechanism is described identically and the `cka-23.swap` facet linkage keeps coverage gating correct.
