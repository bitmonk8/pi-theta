## Finding

`H6a`'s Deps-note parenthetical asserts: "`H5b` owns the complete coverage-producing dependency set (`H1a`, `H5a`, `M`, `V1a`–`V18d`), kept transitively complete …". That inline enumeration is stale. `H5b`'s actual `Deps.` are `H1a`, `H5a`, `H5e`, `H5f`, `M`, `V1a`–`V18d`, and `H5b`'s own Deps-note names `H5e` (the un-anchored normative-MUST recogniser arm) and `H5f` (the per-facet citing-test arm for multi-leaf rows) as members of the coverage-producing set. `H6a`'s restated copy drops both.

The same paragraph also claims `H6a` "inherits that completeness transitively through `H5b` rather than restating the set" — yet the parenthetical does restate it, and restates it incompletely. The result is internally self-contradictory: the prose says it does not restate the set while restating a subset of it. A reader reconstructing the closing-gate dependency closure from `H6a`'s text concludes `H5e`/`H5f` are outside the coverage-producing set, when they are squarely inside it.

The dependency graph itself is correct — `H6a` depends on `H5b`, which depends on `H5e`/`H5f` — so this is a documentation-fidelity defect in `H6a`'s explanatory prose, not a missing edge. It is also recurrent: an earlier pass already had to patch this same parenthetical when it omitted `H1a`.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H6a-live-corpus-activation.md`, in the Deps-note parenthetical, eliminate the second, independently-maintained copy of the coverage-producing set. Strike the inline enumeration `(\`H1a\`, \`H5a\`, \`M\`, \`V1a\`–\`V18d\`)` and refer to `H5b`'s `Deps.` as the single source of truth — for example, replace "`H5b` owns the complete coverage-producing dependency set (`H1a`, `H5a`, `M`, `V1a`–`V18d`), kept transitively complete …" with wording that points at "the complete coverage-producing dependency set named in [`H5b`](./H5b-warn-only-canary.md)'s `Deps.`". This keeps `H6a`'s claim that it "inherits that completeness transitively through `H5b` rather than restating the set" literally true and removes the surface that drifts every time a new arm is carved out of `H5a`.

If, instead, a literal enumeration is retained at this mention, its members must equal `H5b`'s actual `Deps.` — i.e. it must include `H5e` and `H5f` — and must stay in lockstep with `H5b`'s `Deps.` whenever a coverage-producing leaf is added. The spec is read-only for this fix; the edit is confined to `H6a`'s plan leaf.

## Consequence

**Severity:** advisory

The actual dependency closure is correct, so no leaf is blocked and no gate misfires. The defect is a stale, self-contradicting restatement: a reader trusting `H6a`'s parenthetical concludes `H5e`/`H5f` are not part of the coverage-producing set, and each future arm carved out of `H5a` drifts the copy further out of sync.
