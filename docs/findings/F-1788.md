## Finding

`audit-resolution.md` *Source-file encoding* makes the audit's strict UTF-8 decoder a normative MUST: on encountering any invalid byte sequence the decoder "MUST report the zero-based byte offset of the first invalid byte sequence in the original (pre-normalisation) file content", explicitly mirroring the runtime `loom/load/invalid-encoding` offset semantics.

A decode failure is routed through *Infrastructure-failure handling* (i) (implementation-agnostic parse-failure path), so the surfacing surface is the four-field stdout record pinned by *Wire serialisation* on `audit-wire-and-canary.md` — `path`, `line`, `symbol`, `proposed-resolution`. None of these four fields has been allocated to carry a decimal byte offset:

- `path` carries the audited file path.
- `line` carries an integer source line and per the infra-failure rule "MAY be the literal `<n/a>` sentinel when the failure precedes line-level resolution" — which a pre-decode UTF-8 failure does, so `line` is `<n/a>` here.
- `symbol` carries a named symbol or `<n/a>`; the infra-failure rule again makes it `<n/a>` when no symbol applies (e.g. "a parse failure on the file header"), which covers a decode failure with no parseable tokens.
- `proposed-resolution` is constrained to plain printable ASCII and, when the contributor-facing resolution arm cannot be quoted directly, MUST be "a printable-ASCII paraphrase that names the arm by reference (e.g. by the relevant `#bump-step-2b-*` anchor) rather than by quoted source text" — i.e. a resolution-arm reference, not a free slot for data.

Stderr is permitted for "non-machine-parseable human-readable summary or warning text", but is explicitly **not** a machine-parseable surface a CI parser may consume. The byte offset therefore has no place to go on the wire that a fixture (or a CI parser) can read. Two conformant audit implementations may legitimately drop the offset, fold it into the `<symptom>` token, append it into `proposed-resolution` against the printable-ASCII paraphrase rule, or write it to stderr — and the *Source-file encoding* MUST cannot be falsified by reading the audit's pinned output.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e5cd514` (2026-05-09) for the four-field wire format on the audit; `6829a79` (2026-06-06) for the strict-UTF-8 decoder MUST with the first-invalid-byte offset obligation.
**History:** `git log -S 'tab-delimited record segments' -- docs/` lands on `e5cd514` ("T35 SDK capability inventory closed-set negative-direction audit") as the commit that introduced the four-field `path` / `line` / `symbol` / `proposed-resolution` wire shape on what became `audit-wire-and-canary.md`. `git log -S 'strict UTF-8 decoder' -- docs/` and `git log -S 'first invalid byte sequence' -- docs/` both land on `6829a79` ("resolve 'Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)'") as the commit that added the *Source-file encoding* decoder MUST to `audit-resolution.md`. Diffing `6829a79~1` against `6829a79` on `audit-resolution.md` confirms the MUST is net-new there; the wire-shape contract was already in place from a month earlier and was not revisited in `6829a79` to allocate a slot for the new offset obligation. Neither commit alone produces the defect — `e5cd514` predates the offset MUST, and `6829a79` did not touch the wire format — so the two changes interact to create a normative obligation whose surfacing surface has no allocated slot.

## Solution Space

**Shape:** multiple

### Option A — Reuse `symbol` for the decimal offset on the UTF-8 decode infra-failure family

**Approach.** Specify, on the *Source-file encoding* paragraph of `audit-resolution.md`, that a UTF-8 decode failure surfaces as an `audit/infra/<family>/<symptom>` record whose `<family>` token is reserved for the decode case, `path` carries the offending file path, `line` is `<n/a>`, `symbol` carries the zero-based byte offset rendered as decimal ASCII digits, and `proposed-resolution` names the re-save-as-UTF-8 resolution arm. Cross-link the rule from `audit-wire-and-canary.md`'s *Infrastructure-failure handling* per-field `<n/a>` rule with one sentence carving out the decode-family exception.

**Spec edits.**
- `audit-resolution.md` *Source-file encoding*: replace the bare "MUST report the zero-based byte offset" with the wire-surface pin above, naming the field bindings and the decode `<family>` token.
- `audit-wire-and-canary.md` *Infrastructure-failure handling*: append one carve-out sentence to the per-field rule (immediately after "`symbol` is `<n/a>` when no named symbol applies") stating that the decode-family record `symbol` carries the decimal byte offset rather than `<n/a>`.

**Pros.** Reuses an existing field; no new field, no new column on the per-family record-shape table, no parser change beyond knowing that `symbol` is numeric for this one `<family>` token. Field is already declared tab-free and ASCII.

**Cons.** Mildly overloads `symbol` (everywhere else: a bare identifier or `<n/a>`); a CI parser that types `symbol` as "identifier or sentinel" needs to relax that for the decode `<family>`.

**Risks.** A future contributor adding a similar non-identifier offset to another infra `<family>` may follow the precedent and erode the `symbol`-is-identifier convention further; mitigated by keeping the carve-out scoped to the named decode `<family>` token rather than to infra records in general.

### Option B — Encode the offset in `proposed-resolution` as a key=value pair

**Approach.** Specify that the decode-family infra record's `proposed-resolution` is the plain-ASCII key=value pair `re-save-as-utf-8 offset=<N>` (or equivalent leaf-pinned key=value form whose `offset` key names the byte offset and whose other key names the resolution arm by `#bump-step-2b-*`-style anchor reference). Keep all four `path`/`line`/`symbol` fields on their existing rules (`symbol` = `<n/a>`).

**Spec edits.**
- `audit-resolution.md` *Source-file encoding*: replace the bare "MUST report" with the `proposed-resolution` key=value pin above.
- `audit-wire-and-canary.md`: no edit (the existing `proposed-resolution` printable-ASCII paraphrase rule already admits the `walked=<N> recognised=<M>` precedent the canary clause sets, so the offset key=value is the same shape).

**Pros.** No field-shape carve-out; mirrors the existing `walked=<N> recognised=<M>` pattern *Green-path emission contract* already pins on `proposed-resolution`; keeps `symbol` exclusively for identifiers/`<n/a>`.

**Cons.** Mixes structured data (an offset a fixture asserts on) into a field whose primary role is naming a contributor-facing resolution arm; CI parsers and fixtures must learn to split the field on whitespace and look up `offset=`.

**Risks.** Sets a precedent that any data not modelled by the four fixed fields can be packed into `proposed-resolution`; mitigated by keeping the convention scoped to two named contexts (this decode-family record plus the canary record).

### Recommendation

Adopt **Option A**. The byte offset is fixture-bearing data, and putting it in a dedicated field beats packing it inside a free-text-shaped slot — `symbol`'s no-tab plain-text invariant already accommodates a decimal-ASCII value, the per-field `<n/a>` rule is the natural place to scope the carve-out, and a fixture asserting on `symbol === "<N>"` reads at the same site as the existing `<n/a>` assertions. Edge cases the implementer must watch: (i) the carve-out is scoped to the named decode `<family>` token only, not to infra records in general — the `symbol = <n/a>` rule still holds for every other pre-decode-resolution infra failure; (ii) the decode record's `<family>` and `<symptom>` token strings remain implementation-owned per *Infrastructure-failure handling* but must stay disjoint within the `infra`-class `<family>` token set per that paragraph's existing rule; (iii) a non-UTF-8 BOM still reports `0`, matching `loom/load/invalid-encoding`'s mirrored runtime semantics.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one drops the offset (violating the decoder MUST), one packs it into `<symptom>` (producing an unbounded discriminator vocabulary the three-class partition forbids), one folds it into `proposed-resolution` (against that field's paraphrase rule), and one writes it to stderr (a non-machine-parseable surface a CI parser may not consume). A fixture that asserts "first-invalid-byte offset is reported as N" cannot be authored because the spec pins no surface for N, so the MUST is unenforceable end-to-end.
