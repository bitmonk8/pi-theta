## Finding

DISC-5 (`docs/spec_topics/discovery/package-and-settings.md`) normatively pins `pi.looms` glob resolution to the `minimatch` engine — "the same package Pi applies to its own resource arrays in `@earendil-works/pi-coding-agent` (`dist/core/package-manager.js`, symbol `applyPatterns`)" — and `loomPaths`' `loompaths-entry-schema` section inherits that engine choice. Realising DISC-5 therefore requires loom to consume `minimatch` at runtime: the cited `applyPatterns` symbol lives under `dist/core/` of a peer package, is not part of any documented `@earendil-works/*` export surface, and cannot be relied on by an implementer.

The spec's own runtime-dependency enumeration — `implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)" (anchor `#loom-package-implementation-dependencies-loom-1-0`) — names `semver`, `chokidar`, and `yaml` as the loom package's runtime `dependencies`, each with a parallel clause stating the seam it backs. It does not mention `minimatch`. The spec corpus therefore names a third-party package as a load-bearing matcher in a normative rule but never tells the implementer where it comes from. Two readings are left open: that loom declares its own `dependencies` entry on `minimatch` (the `semver` / `chokidar` / `yaml` model), or that loom reaches into the copy bundled with `@earendil-works/pi-coding-agent` (suggested by the `dist/core/package-manager.js` / `applyPatterns` parenthetical, even though `applyPatterns` is an internal symbol Pi makes no stability promise about). The plan side (`docs/plan_topics/H1a-scaffold-and-toolchain.md`) has already chosen the first reading — `minimatch` is declared in loom's runtime `dependencies` and the leaf cites DISC-5 — but the spec impl-deps paragraph the plan defers to doesn't carry the `minimatch` entry, so the plan's `H1a` citation points at a paragraph that doesn't list the dependency it claims to source.

## Solution Space

**Shape:** single
**State:** shaped

### Recommendation

In `docs/spec_topics/implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)", add a `minimatch` clause parallel to the existing `semver` / `chokidar` / `yaml` clauses. The clause should: name `minimatch` as the loom 1.0 runtime dependency consumed by DISC-5's glob matcher (link to `discovery/package-and-settings.md#disc-5`), realising the `pi.looms` resolution rule (override ordering `plain → ! → + → -` and exact-path operand treatment); state that `minimatch` is declared in **loom's own** `dependencies` block on the same footing as `semver`, `chokidar`, and `yaml` — loom does not reach into the copy bundled with `@earendil-works/pi-coding-agent`, and the `dist/core/package-manager.js` / `applyPatterns` reference in DISC-5 is an editorial pointer at Pi's matching shape, not an import target; follow the established "names the dependency, not its version" wording, leaving the pinned range to `package.json`; and add the standard inventory-closure carve-out sentence already used for `chokidar` and `yaml` — `minimatch` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it is out of scope for the inventory-closure audit's *Target surface categories* and requires no `// allow-pi-surface:` exemption marker at any `minimatch` import site.

`minimatch` ships its own TypeScript bindings in current major versions, so a `@types/minimatch` `devDependencies` entry is not required (parallel with `yaml`, not with `semver`). After the edit, `H1a`'s existing citation lands on a clause that actually exists, so no plan-side rewording is needed.

Edge case: the cross-reference at DISC-5 to `applyPatterns` is a non-normative behavioural-reference parenthetical and stays as-is for this edit; its load-bearing-ness is the subject of the sibling "DISC-5 over-prescribes" finding, which — if accepted — would demote the reference to a non-normative implementation note and shift this clause's citation from DISC-5 to that note (resolve the over-prescription question first to avoid re-doing the citation).

## Consequence

**Severity:** advisory

An implementer following the plan provisions `minimatch` correctly; nothing is blocked. The defect is a documentation-parity gap: the spec's runtime-dependency enumeration is incomplete relative to the dependency set DISC-5 actually induces (`semver` / `chokidar` / `yaml` enumerated, `minimatch` not), leaving the "own dependency vs. Pi-bundled copy" question implicit, and making the plan's `H1a` citation point at a paragraph that doesn't list the dependency it claims to source. A second implementer working from the spec alone has no anchored answer to whether loom ships its own `minimatch` runtime dependency, omits the entry assuming Pi supplies one, or imports an internal Pi symbol that carries no stability promise.
