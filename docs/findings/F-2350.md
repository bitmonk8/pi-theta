## Finding

`H6a`'s **Rollback.** section claims that reverting the activation commit is a complete recovery step: "reverting this leaf's activation commit returns `H5a`'s closing gate to its seeded-fixture footing … There is no data, schema, or registry migration to unwind, so the revert is the complete recovery step." This is a pure prose assertion. None of `H6a`'s Tests bullets exercise the post-revert state — they verify only that the *live-corpus* footing passes green and reddens on the three live failure modes (unmapped REQ-ID, missing citing test, missing per-facet citing test). There is no named observable that confirms the gate has actually returned to its seeded-fixture footing after the activation commit is reverted.

The recovery claim is, in fact, true and already observable: `H5a`'s seeded green/red tests (`H5a` **Ships when** — green against the seeded no-violation fixture, red against each seeded violation fixture) run against the dedicated test-fixtures root that lives outside the live corpus, so they execute identically whether or not `H6a`'s live-corpus footing is active. After a revert they continue to pass, which is exactly the evidence that the seeded-fixture footing was restored. The defect is that `H6a` never ties its rollback claim to this existing observable, leaving "the revert is the complete recovery step" resting on argument alone.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H6a-live-corpus-activation.md`'s **Rollback.** section, tie the recovery claim to its existing observable: state that the post-revert observable is `H5a`'s seeded green/red tests, which run against the dedicated test-fixtures root independently of the live-corpus footing and therefore continue to pass after `H6a`'s activation commit is reverted — that continuation is what confirms the gate returned to its seeded-fixture footing, with no migration to unwind.

`H5a` already provides this observable (its **Ships when** asserts the gate runs green against the seeded no-violation fixture and red against each seeded violation fixture under the test-fixtures root that the live-mode path selection excludes), so no new test machinery is required and `H5a` stays read-only. Cite `H5a` by ID where the rollback prose makes the "returns to its seeded-fixture footing" claim. Do not introduce a new footing-state flag or a dedicated post-revert assertion in `H6a` unless `H5a`'s seeded tests are found not to run on the post-revert footing — they do, because the seeded fixtures are permanent-by-design and reconciled exclusive of the live corpus.

## Consequence

**Severity:** advisory

If `H6a` ships with the rollback claim unverified, a contributor reverting a bad activation flip has no stated check that the gate actually fell back to its seeded footing; they rely on the prose "no migration to unwind." The recovery path is genuinely a single revert and `H5a`'s seeded tests already cover it, so no implementer is blocked — the gap is a traceability one between the claim and the observable that substantiates it.
