## Problem

In `runtime-event-channel.md` the **Deduplication and lifetime rules** give group A (`details: { event: RuntimeEvent }`) an observable, falsifiable handle on its exactly-once guarantee — the dedup tuple, a worked `?`-propagation derivation, the cascade-twin re-emission rule, and the clause-(e) conformance fixture at `#pic-1`. Group B (runtime panics emitting `details: { diagnostics: Diagnostic[] }`) carries the bare panic-emission rule with no analogous acceptance fixture, and the page states the dedup tuple "applies to group A only and ha[s] no analogue for group B." Three group-B behaviours are normative but unexercised by any pinned fixture: a directly-raised panic emits exactly one note with zero `details: { event }` emissions; a `?`-propagated panic emits once at its origin, not per frame; and a panic reaching the slash-dispatch boundary emits no cascade twin. The only emission-count assertion is the generic V9d/V9d-T "Each always-log kind emits exactly once" bullet, which neither enumerates the three scenarios nor pins the no-cascade-twin asymmetry.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 5c8d782 — pi-loom spec: resolve "Panic routing through `loom-system-note` is under-specified across two pages" (2026-05-06, Thomas Andersen)
**History:** The group-A/group-B partition and the "A panic emits exactly one `loom-system-note` per top-level panic" rule were introduced together at 5c8d782 while resolving the panic-routing ambiguity. The same edit furnished group A with a worked `?`-propagation + cascade-twin derivation, but gave group B only the bare one-line rule with no companion fixture, so the fixture asymmetry has existed since the rule entered the corpus. (The always-log exactly-once concept itself predates this at b3bc4ce; group A's clause-(e) `masked` conformance fixture was added later at d75f2a8, further widening the gap — but the unfixtured group-B panic rule originates at 5c8d782.)

## Solution approach

Add a fixture-obligation block under the panic-emission bullet in the **Deduplication and lifetime rules** section of `runtime-event-channel.md`, mirroring the clause-(e) falsifiable-fixture style at `#pic-1`, pinning acceptance fixtures for the three group-B behaviours: a directly-raised panic emitting exactly one note with zero `details: { event }` emissions; a `?`-propagated panic emitting once at its origin; and a boundary-cascading panic emitting no cascade twin. Sharpen the V9d/V9d-T "Each always-log kind emits exactly once" plan bullet to cite the three group-B scenarios explicitly.

## Solution constraints

- Out of scope: per-`loom/runtime/*`-code message templates (owned by plan leaf V4b) — pin the fixtures to one representative code, not per code.
- Out of scope: the `loom/runtime/internal-error` `content` variant — it is a `content`-string concern, not an emission-count one.
