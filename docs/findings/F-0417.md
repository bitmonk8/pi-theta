## Finding

The reload-failure injection DI seam carries two distinct names for one concept. The leaves bold the descriptive term **"watcher-time reload failure-injection seam"** (`V4e`, `V4e-T`, `V9b`, `V10c`, `V10d`), while the same seam's interface is named `ReloadFailureInjector` (with method `injectReloadFailure`), declared once by `V9b` and consumed by `V4e`/`V4e-T`/`V10d`. The seam *is* the injection point of the `ReloadFailureInjector` interface — there is no second concept hiding behind the second name.

This breaks the interface-name pattern the plan uses for every other DI seam. The `V8` host seams are each named by their interface: `Checkpoint` seam, `FileSystem` seam, `SchemaValidator` seam, `Clock` seam, `IdSource` seam, `FileWatcher` seam, `TokenEstimator` seam (see `docs/plan.md` §V8 — Pi host seams). The reload-failure seam is the lone exception that introduces a free-standing descriptive name alongside a differently-named interface.

The binding code-level symbol (`ReloadFailureInjector.injectReloadFailure`) is used consistently across the leaves, so no implementer would build the wrong interface; the cost is the dual defined-term presentation — a bolded prose term and an interface name for one thing — which a reader can mistake for two seams.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `66acde6` (2026-06-11, "resolve 'ERR-7 test injects at an undefined channel seam'") first introduced the descriptive "failure-injection seam" term; `d64dce5` (2026-06-11, "resolve 'Watcher-time reload failure-injection seam under-specified and ungated'") added the `ReloadFailureInjector` interface name alongside the already-present descriptive term, making the dual-naming coexistence live.
**History:** The plan corpus is git-tracked. `git log -G "failure-injection seam" --reverse -- docs/plan_topics/` first surfaces the descriptive seam term at `66acde6`; `git log -S "ReloadFailureInjector" -- docs/plan_topics/` first surfaces the interface name at `d64dce5` (same day). The two-name coexistence was then preserved and propagated through `fdce2a2` (2026-06-12, "ReloadFailureInjector seam has two owners") and `f14e929` (2026-06-16, the `V10c`→`V10d` split), neither of which collapsed the descriptive term into the interface name. No single commit introduced "two names for one concept" in isolation — it emerged from the interface name landing next to a pre-existing descriptive term and surviving subsequent reworks.

## Solution Space

**Shape:** single

### Recommendation

Replace the bolded descriptive term **"watcher-time reload failure-injection seam"** with **"`ReloadFailureInjector` seam"** at every occurrence, aligning it with the `V8` interface-name seam pattern. The five files carrying the descriptive term are:

- `docs/plan_topics/V4e-pre-evaluation-failures.md` — `ERR-7` Tests bullet.
- `docs/plan_topics/V4e-T-pre-evaluation-failures.md` — `ERR-7` Tests bullet.
- `docs/plan_topics/V9b-registration-reload-wiring.md` — Adds (the "single declaration site … for the whole watcher-time reload failure-injection seam" clause).
- `docs/plan_topics/V10c-settings-merge.md` — Adds (the "settings-re-merge sub-arm of the watcher-time reload failure-injection seam" clause).
- `docs/plan_topics/V10d-reload-debounce.md` — Adds and Deps (both "settings-re-merge sub-arm of the watcher-time reload failure-injection seam" mentions).

The interface name `ReloadFailureInjector` and method `injectReloadFailure` already match the target name, so no code-level symbol changes. The regenerated terms index picks up the single seam name once the prose term is unified; no separate hand-edited index entry is required. The descriptor "watcher-time" can be retained as ordinary prose where it usefully describes *when* the injection happens, but it should no longer stand as the seam's name.

This fix is purely internal to the plan leaves; the spec is read-only for it (the `ReloadFailureInjector` interface is test-only and lives in the plan, not the spec).

## Consequence

**Severity:** cosmetic

A reader encountering the bolded "watcher-time reload failure-injection seam" in one leaf and the `ReloadFailureInjector` interface in another can read them as two distinct seams rather than one. The interface symbol is unambiguous and consistently used, so the behaviour stays implementable; the defect is presentational consistency only.
