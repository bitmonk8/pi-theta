## Finding

QRY-7 rule 4 in `docs/spec_topics/query/query-forms.md` pins two distinct `\r`-handling paths: source `\r`/`\r\n`, which Lexical Structure normalises to `\n` *before* newline-trim and dedent see it, and a `\r` introduced into the rendered text **by an interpolated value** — which rule 4 calls out as "the one path by which a carriage return can reach dedent, since interpolated values are not subject to source newline normalisation" — and pins as ordinary content (does not split a line, does not satisfy the whitespace-only-line predicate of rule 1).

The 10-vector normative table exercises only the source path. Vector 9 (`` @`\n    a\r\n    b\n` ``) places the `\r\n` inside the *source* between the backticks, so by Lexical Structure's pre-lex normalisation it has been collapsed to `\n` before QRY-7's machinery runs at all; vectors 1–8 and 10 do not involve `\r` at any layer. There is no vector exercising the only path QRY-7 rule 4 actually singles out.

An implementation that normalises `\r\n`/`\r` → `\n` uniformly across both the source stream *and* the post-interpolation rendered stream (a common single-pass implementation choice — running newline normalisation once over the fully assembled buffer) passes all 10 vectors while violating rule 4: it would silently mis-render any user-supplied string whose value contains a CR or CRLF (e.g. data pasted from a Windows clipboard interpolated into a prompt), folding multi-line content the author never intended to be multi-line and making leading whitespace on what was an interpolated single line participate in dedent's common-prefix computation. The rule is normative but unmechanically checkable; passing the conformance suite is consistent with violating the rule on the one input shape it exists to govern.

## Solution Space

**Shape:** single

### Recommendation

Add one normative vector to the table in `docs/spec_topics/query/query-forms.md` whose template body contains a `${…}` interpolation, where the interpolated `string` value contains an embedded `\r` (and, to make the rule maximally discriminating, an embedded `\r\n`). The vector must demonstrate that the `\r` is preserved verbatim in the rendered text — neither stripped, nor translated to `\n`, nor used as a line boundary for dedent's common-prefix walk.

Concrete shape (vector 11, with matching commentary):

```
| 11 | `` @`\n    ${s}\n    end\n` `` where `s = "a\rb\r\nc"` | `"a\rb\r\nc\nend"` |
```

Commentary to add at the bottom of the "Vector commentary:" list:

> 11. The interpolated `string` value carries a bare `\r` and a `\r\n`; both are introduced **after** Lexical Structure's source-newline normalisation has run (interpolated values are not subject to it — see rule 4), so they reach dedent intact. Per rule 4 they are ordinary content: they do not split lines and they do not satisfy rule 1's whitespace-only-line predicate, so the slot occupied by `${s}` is a single line for dedent's purposes. Dedent's common-prefix walk sees two lines — the four-space-indented line containing `${s}`'s expansion, and the four-space-indented `end` — strips the shared four-space prefix, and emits `"a\rb\r\nc\nend"`. An implementation that ran `\r\n`/`\r` → `\n` normalisation over the post-interpolation stream would wrongly emit `"a\nb\nc\nend"` and would fail this vector; vector 9 alone cannot detect that bug because rule 4's source-stream normalisation reduces it to the lone-`\n` case before dedent runs.

Edge cases the implementer must watch:

- The `\r` in column "Template" denotes a literal carriage-return *byte inside the `${…}` value*, not a `\r` byte in the source between the backticks. The vector relies on the QRY-18 Note that stringification of an interpolated `string` value renders "the value itself, no quoting, no escaping" and runs *before* newline-trim and dedent, so byte-for-byte the value's `\r`s land in dedent's input.
- The rendered-text column likewise uses `\r` to denote a literal CR byte; the cell follows the same convention the preamble already establishes for `\n`, `\r`, `\t`, and `\u00A0`.
- Update the table preamble if needed to make clear that `\r` and `\r\n` in the Template column denote bytes inside the interpolated value rather than between the backticks (the existing preamble covers backtick-body bytes only).
- Update `docs/plan_topics/V13a-query-render.md` and `docs/plan_topics/V13a-T-query-render.md` to say "the eleven ordered vectors" / "the eleven normative vectors" / "the eleven dedent vectors" — and consider rephrasing to "the QRY-7 vector table" so future growth does not re-introduce the same drift.

Do **not** "fix" this by widening vector 9 or rewording rule 4: rule 4's two-path structure (source-normalised vs. interpolation-survivor) is load-bearing for the wire-text guarantee, and the gap is exactly that the table fails to exercise the second path. Adding the vector closes that gap without changing any normative rule.

## Consequence

**Severity:** correctness

Two reasonable implementers — one writing a single post-render `\r\n` → `\n` sweep, one preserving interpolated `\r` verbatim per rule 4 — both pass the published conformance vectors but produce diverging model-visible text for any interpolated string containing a CR byte. The divergence is silent (no diagnostic), author-invisible, and shape-changing (line counts and dedent results differ), so production prompts can be silently corrupted on Windows-origin data without any conformance-suite signal.
