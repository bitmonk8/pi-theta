## Finding

`conventions.md` is the per-leaf authoring conventions document: its body (Per-phase TDD ritual, Leaf format, Cross-cutting rules) tells the implementer building one leaf what to do. Its trailing `## Loom 1.0 release-time residue inspection` section is a different artefact — a nine-item manual sweep performed **once, at the loom 1.0 release gate, by the `H6a` release-gate auditor**, who reads `src/**`, the repository-root doc artifacts, `spec_topics/**`, the coverage matrix, `extensions/index.ts`, and the `H3a` timing-primitive list and records each item clean-or-dispositioned in the gate-activation commit message. Its audience is the release-gate reviewer, not the per-leaf implementer the rest of the file addresses.

The plan already houses a structurally identical release-gate reviewer surface — the `real-host-smoke-gate.md` manual smoke protocol — in its own governance note, which opens by stating "Its audience is the bump/merge contributor and the `H6a` release-gate auditor, not the `H4a` implementer." The residue inspection is the same kind of release-time, `H6a`-owned, reviewer-facing checklist, yet it lives inline in the authoring-conventions file rather than being given an equivalent audience boundary.

The section is well-demarcated (its own H2 heading, an intro paragraph establishing it is a plan-side release-gate review owned by `H6a`), so no implementer is led into producing a wrong leaf. The defect is one of placement/audience consistency: the release-gate reviewer checklist is co-located with the per-leaf authoring conventions, without the explicit audience boundary the parallel `real-host-smoke-gate.md` note carries.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** e94ca10 (2026-06-16)
**History:** The `## Loom 1.0 release-time residue inspection` heading, its `release-time-residue-inspection` anchor, and the checklist were first authored directly inside `conventions.md` in e94ca10 ("pi-loom plan: resolve 'GOV-15 release-time reviewer inspection has no enumerated residue checklist'"). `git log -S` on the heading string returns that single commit, confirming the section was never housed elsewhere and relocated — it was born in `conventions.md`. The checklist later grew (item 8 entry-shim purity in 4b3c2c0, item 9 timing-list reconciliation in 1c9ede1) and `H6a` was pinned as its owner in ee9acd0, but the decision to place the block inside the per-leaf authoring conventions dates to the section's inception. The audience/placement asymmetry has existed since the block was created.

## Solution Space

**Shape:** multiple

### Option A — Relocate to a dedicated release-gate governance note

- **Approach:** Move the whole `## Loom 1.0 release-time residue inspection` section (its intro paragraph, the nine checklist items, and the `<a id="release-time-residue-inspection"></a>` anchor) out of `conventions.md` into a new sibling governance note under `docs/plan_topics/` (e.g. `release-residue-inspection.md`), modeled on `real-host-smoke-gate.md`: a cross-cutting governance note carrying no leaf ID and closing no REQ-ID, opening with an audience statement that it is performed by the `H6a` release-gate auditor, not the per-leaf implementer.
- **Plan edits:** Create the new file holding the relocated section, keeping the anchor id `release-time-residue-inspection` so existing fragment links still resolve. Remove the section from `conventions.md`; the §"Cross-cutting rules" in-page `#release-time-residue-inspection` links become external links to the new file. Retarget the `conventions.md#release-time-residue-inspection` references in `coverage-matrix.md`, `H3a`, `H4a`, `H5d`, `H5f`, and `H6a` to the new file path (anchor unchanged).
- **Spec edits:** None.
- **Pros:** Consistent with the `real-host-smoke-gate.md` precedent; cleanly separates the release-gate auditor checklist from the per-leaf authoring conventions.
- **Cons:** Splits each residue item from the cross-cutting rule it backs (today both sit in one file, reached by in-page anchors); spreads the edit across 6+ files.
- **Risks:** A missed link retarget leaves a dangling cross-reference; the larger diff widens the review surface.

### Option B — Keep in `conventions.md`, sharpen the section's audience scope

- **Approach:** Leave the section in place (preserving every in-page anchor and cross-reference) and add an explicit audience-scope statement at the section head, paralleling `real-host-smoke-gate.md`'s audience sentence — stating the residue inspection is a release-gate reviewer surface performed once by `H6a` at the loom 1.0 release gate, distinct from the per-leaf authoring conventions above it.
- **Plan edits:** `conventions.md` only — at the head of the `## Loom 1.0 release-time residue inspection` section, state its audience is the `H6a` release-gate auditor (not the per-leaf implementer) and that it runs once at the release gate. The section already establishes the release-gate / `H6a`-owned framing; this makes the audience boundary explicit.
- **Spec edits:** None.
- **Pros:** Preserves the tight coupling between each cross-cutting rule and its residue item via in-page anchors; minimal diff confined to one file; no cross-reference churn.
- **Cons:** The release-gate reviewer checklist still lives in the authoring-conventions file, so the file-placement asymmetry vs `real-host-smoke-gate.md` remains.
- **Risks:** Minimal.

### Recommendation

Take Option B. Each of the nine items is bound to a specific cross-cutting rule in the same file and is reached through an in-page anchor, and the section already declares it runs at the release gate and is `H6a`-owned. Adding an explicit audience-scope statement at the section head resolves the audience-mismatch at minimal cost and risk, whereas relocation would split coupled content and churn cross-references across six files for a cosmetic gain. Keep the `release-time-residue-inspection` anchor id intact either way so the fragment links from `conventions.md`, `coverage-matrix.md`, `H3a`, `H4a`, `H5d`, `H5f`, and `H6a` keep resolving.

## Consequence

**Severity:** cosmetic

If shipped unfixed, a release-gate reviewer checklist remains embedded in the per-leaf authoring conventions document, inconsistent with the parallel `real-host-smoke-gate.md` governance note. The section is clearly headed and `H6a`-owned, so no implementer is misdirected and no gate is weakened; the cost is organisational only.
