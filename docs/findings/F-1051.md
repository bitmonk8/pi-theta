## Finding

Two separate paragraphs in `version-bump-step2.md` cross the "what vs. how" line and pin implementation details where the underlying obligation is purely behavioural.

**(1) `CAPABILITY_OBLIGATIONS` shape and placement (step 2(a) positive-direction paragraph, anchor `bump-step-2-positive`).** The behavioural gate is straightforward: the literal-read assertion must witness that there are exactly seven obligations and that they correspond 1:1, in order, to items 1–7 of `capability-inventory-items.md`. The spec, however, additionally mandates the concrete data structure (`"a closed array of seven anchor IDs whose contents and ordering are the <a id> values on items 1–7 … in that exact order"`, followed by an inline enumeration of all seven literal anchor IDs) **and** mandates colocation (`"pinned in the same source-of-truth pinned-constants block in the extension module that holds the four Step-0 probe constants"`). Whether obligations are an array of anchor strings, an array of enum-tagged records, a generated table, or a `Map` keyed by anchor — and whether they sit alongside the Step-0 probe constants or in a sibling module — is a code-organisation choice the gate's correctness does not depend on. The seven-anchor enumeration also duplicates what `capability-inventory-items.md` is the source of truth for, creating two places that must be kept in lockstep.

**(2) `npm pack` mandated as the baseline-acquisition command (anchor `bump-baseline-acquisition`).** The Baseline acquisition block says the contributor "MUST … by running `npm pack @earendil-works/pi-coding-agent@<pinned-version>` and extracting the resulting tarball", with a parenthetical justification ("rather than `npm install --prefix` so the lock-step siblings' transitive dependency tree is not pulled pointlessly"). The behavioural requirement is that the loom 1.0-pinned `dist/*.js` tree be materialised somewhere on disk to serve as the "from" side of the recipe's diffs, **without** disturbing or being polluted by the candidate-minor install that step 1 has already laid down in `node_modules/`. `npm pack` is one reasonable way to satisfy that; a pre-installed read-only scratch checkout, an `npm install --prefix` into a throwaway directory, a registry tarball fetch via `curl`, or a CI-cached snapshot all satisfy the same property. Mandating the specific command — including the `@<pinned-version>` specifier shape and the "MUST either run before that install, or … pass the explicit `@<pinned-version>` specifier" sequencing it forces — pins the *means*, not the *requirement*.

The two paragraphs were introduced by separate fixes (see Issue introduction) and over-shot independently; resolving them is best done as two sequential edits rather than one combined pass.

## Solution Space

### Recommendation

Make two spec-only edits to `version-bump-step2.md`, in this order: first the `#bump-step-2-positive` paragraph, then the `#bump-baseline-acquisition` block nested inside item (e). Editing the step-2(a) paragraph first puts it on a clean baseline (the seven-anchor enumeration no longer inlined) before rewriting the separate baseline-acquisition block. Neither edit touches plan leaves' Tests/Ships-when fields; re-read `V18a`/`V18c` after both land to remove any incidental phrasings that imply the data-structure or `npm pack` mandate, but no acceptance criterion changes.

**Edit 1 — state the `CAPABILITY_OBLIGATIONS` gate behaviourally (anchor `#bump-step-2-positive`).**
- Replace the sentence that introduces `CAPABILITY_OBLIGATIONS` (currently "a closed array of seven anchor IDs … in that exact order: [seven anchors] …") with a behavioural restatement: name the constant and the assertion (`CAPABILITY_OBLIGATIONS.length === 7`), reference `capability-inventory-items.md` items 1–7 as the source of truth for membership and ordering, and leave the data structure and the constant's module location an implementation choice.
- Strike the colocation mandate ("pinned in the same source-of-truth pinned-constants block in the extension module that holds the four Step-0 probe constants"), or demote it to a non-normative implementation note. The Step-0 probe-constants colocation is a separate concern owned by `capability-probe.md` / V18a.
- Remove the inline enumeration of seven anchor IDs; the anchors remain the source of truth in `capability-inventory-items.md`, and `inventory-audit-intro.md`'s `PIC-15` cross-reference already points there.
- Keep the trailing sentences pinning re-ordering and renaming under the step-2(b) co-edit obligation — they are behavioural co-edit rules, not representation pins.

**Edit 2 — state the baseline-acquisition requirement behaviourally (anchor `#bump-baseline-acquisition`).**
- Replace the MUST sentence "the contributor MUST materialise the loom 1.0 Pi-SDK pin's `dist/*.js` tree into a scratch directory by running `npm pack @earendil-works/pi-coding-agent@<pinned-version>` and extracting the resulting tarball" with a behavioural MUST that names the property — a scratch copy of the loom 1.0-pin's `dist/*.js` tree, distinct from `node_modules/@earendil-works/pi-coding-agent/dist/` after step 1's install — and presents `npm pack … && tar -xf …` as one means (`e.g.` / `for example`).
- Move the "rather than `npm install --prefix` so the lock-step siblings' transitive dependency tree is not pulled pointlessly" parenthetical to an illustrative rationale on the example, not a global prohibition.
- Keep the constraints that are behavioural: (i) `<pinned-version>` must be captured before step 4 rewrites the pin literal, (ii) the acquisition must run before step 1's install **or** explicitly target the pre-step-4 pin so it does not resolve against the rewritten `package.json`, (iii) the scratch tree is ephemeral and MUST NOT be committed.

## Consequence

**Severity:** advisory

The build-time gate works as intended either way: the literal-read assertion reddens on a missing or renamed capability regardless of whether `CAPABILITY_OBLIGATIONS` is an array of anchor strings or a richer record type, and the `dist/*.js` diffs land either way regardless of whether the baseline tree was acquired via `npm pack` or by an equivalent mechanism. The cost is implementer freedom and a duplicated source-of-truth surface (the seven anchor IDs are now enumerated in both `capability-inventory-items.md` and inside the step-2(a) paragraph), which a future addition/reorder of capabilities must keep in lockstep on top of the existing co-edit obligation.
