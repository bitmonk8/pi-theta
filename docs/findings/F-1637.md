## Finding

The *Lookup-failure-to-discriminator routing* clause in `docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` enumerates four arms that exhaust the snapshot-side failure space (arm (1): `SDK_SURFACE_INVENTORY === undefined`; arm (2): iteration primitive returns no matching entry; arm (3): per-entry / `.literals` read throws → `"throw:<String(error)>"`; arm (4): structurally-invalid `literals` → `"literals-shape-invalid"`), and pins arm (1) to discriminator `"missing-entry"` with an explicit pre-iteration `SDK_SURFACE_INVENTORY === undefined` guard MUST. The same paragraph adds a parallel per-sub-case fixture mandate for the four `"literals-shape-invalid"` sub-cases (each constructs a synthetic snapshot with one structural defect, dispatches a `session_shutdown` event, and asserts exactly one `loom/host/session-shutdown-pinned-constant-unreadable` with `details.failure === "literals-shape-invalid"` and no co-emitted `…reason-unknown`).

No equivalent obligation exists for the two `"missing-entry"` arms. The `V9h` / `V9h-T` *Tests* sections mirror that gap: they require `details.failure` "from the closed set" and call out the four `"literals-shape-invalid"` sub-cases each by their own fixture, but neither plan leaf nor the spec pins a fixture witnessing arm (1) (snapshot binding `undefined`) or arm (2) (no matching entry). A naive implementation that omits the explicit `SDK_SURFACE_INVENTORY === undefined` guard and writes the warned-against `SDK_SURFACE_INVENTORY.find(…)?.literals` shape mis-routes arm (1) to the throw arm (a `TypeError` from `.find` on `undefined`, surfacing as `"throw:TypeError: …"`), yet still passes every fixture the spec presently mandates — the closed-set fixture, the throwing-property-access fixture, and the four `"literals-shape-invalid"` sub-cases all hold, and the explicit "discriminator MUST be `\"missing-entry\"`, **not** any `\"throw:TypeError: …\"` value" pin loses its red gate.

This is a witness-coverage hole in the closed-discriminator-set guarantee: every other element of the discriminator union (`"literals-shape-invalid"` × 4 sub-cases, `"throw:<String(error)>"` via the throw-discriminator-grammar acceptance block, and the closed-set / property-access throw paths owned by the *Unknown-reason rule* tests) carries a per-arm fixture obligation; `"missing-entry"` alone does not.

## Solution Space

**Shape:** single

### Recommendation

In the `"missing-entry"` discriminator paragraph of `docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` (immediately after the existing "covers both the literal lookup-returned-nothing case and the defensive kind-mismatch case" sentence), append a fixture-mandate sentence parallel in structure to the `"literals-shape-invalid"` mandate, requiring a per-arm fixture for each of the two `"missing-entry"` arms the *Lookup-failure-to-discriminator routing* clause enumerates:

- one fixture for **arm (1)** — the `SDK_SURFACE_INVENTORY` binding resolves to `undefined` at handler entry (using the same static-import-preserving substitution mechanic the (P-b) / (P-c-1) fixtures in `patch-skew-degradation.md` establish, substituting the production export to `undefined` for the duration of the dispatched `session_shutdown` event), asserting the handler emits exactly one `loom/host/session-shutdown-pinned-constant-unreadable` with `details.failure === "missing-entry"`, no co-emitted `loom/host/session-shutdown-reason-unknown`, and (decisively) **no** `details.failure` value matching `startsWith("throw:")` — the latter is the assertion that fails on the warned-against `?.find` shape and red-gates it; and
- one fixture for **arm (2)** — the iteration primitive returns no matching entry (the substituted snapshot is a structurally-valid `SDK_SURFACE_INVENTORY` array containing no entry satisfying the composite predicate `(entry.kind === "type-union-snapshot") && (entry.path === "SessionShutdownEvent.reason")`, e.g. all entries renamed or removed), asserting the handler emits exactly one `loom/host/session-shutdown-pinned-constant-unreadable` with `details.failure === "missing-entry"` and no co-emitted `…reason-unknown`.

The defensive kind-mismatch sub-case the paragraph already collapses into arm (2) (an entry whose `path` matches but whose `kind` is not `"type-union-snapshot"`) need not carry its own fixture — the paragraph explicitly notes the handler cannot and need not distinguish it from the literal lookup-returned-nothing case — but a third fixture asserting the same `details.failure === "missing-entry"` outcome on the kind-mismatch path is permissible and the spec should leave that as leaf-owned (parallel to how leaf ownership of `<file>` placement and exact assertion shape is pinned for the `"literals-shape-invalid"` mandate). Cross-reference the new sentence into `V9h-T`'s and `V9h`'s **Tests** bullet so the plan-side inventory ratchets in the same way it already does for the four `"literals-shape-invalid"` sub-cases.

Edge cases the implementer must watch: the arm (1) fixture's substitution must replace the production export with `undefined` rather than mutating the `SDK_SURFACE_INVENTORY` import in place (the `as const` declaration must remain intact, mirroring the no-write-through invariant pinned for the (P-b) / (P-c-1) fixtures); and the `no startsWith("throw:")` assertion in the arm (1) fixture is load-bearing — an assertion that only checks `details.failure === "missing-entry"` without negating the throw template would still pass on a `?.find` mis-route whose discriminator happens to be `"missing-entry"` for some other reason (e.g. if a future arm shuffle made the throw arm produce a `"missing-entry"`-prefixed value).

## Consequence

**Severity:** advisory

A spec-conformant implementer who follows every pinned MUST will produce correct behaviour, but the test suite the spec mandates does not red-flag the dominant misroute the *Lookup-failure-to-discriminator routing* paragraph was explicitly written to prevent (the `?.find` anti-pattern on an `undefined` binding). Operator deduplication then silently splits across implementations whose iteration-primitive choice and explicit-guard discipline diverge — the precise outcome the four-arm routing was constructed to forbid — without any acceptance fixture catching it.
