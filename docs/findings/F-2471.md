## Finding

The bind-args echo (`bind_echo:`, default `true`) is specified in prompt-mode-centric terms. The [Echo policy](../spec_topics/binder/defaulting-system-note-echo.md#echo-policy) says the runtime "appends a one-line system note to the user's session immediately before the loom starts," and `slash-invocation.md` repeats "appends a one-line echo system note to the session before the loom starts." Neither clause says what happens when the dispatched loom runs in `mode: subagent`.

The omission is conspicuous because the spec draws a per-mode destination distinction everywhere else this surface appears. `binder.md` states the binder footprint splits by mode — it "never adds turns to the user's session (in prompt mode) or to the loom's spawned conversation (in subagent mode)." `slash-invocation.md` SLSH-3 explicitly pins the *failure*-side slash-dispatch note: the top-level `Err` note lands in the user's session "regardless of whether the loom ran in prompt or subagent mode." The success-side echo carries no equivalent "regardless of mode" clause, so a subagent-mode reader cannot tell whether the echo (a) lands in the caller's (user's) session, (b) lands in the spawned private conversation, or (c) is suppressed — the three readings are each defensible from a different sentence in the corpus.

The literal "the user's session" wording leans toward (a), but `slash-invocation.md`'s subagent-isolation rule ("no … system notes from a subagent's queries surface to any ancestor transcript") and `binder.md`'s subagent-footprint clause actively invite (b)/(c). Because the spec resolves the analogous error-note case explicitly and leaves the echo case to inference, two reasonable implementers can diverge on an observable, user-facing surface.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `83802db` (`spec: collapse to single typed argument style; add LLM-driven slash-command binder`)
**History:** The binder, `bind_echo`, and the Echo policy were all introduced together in `83802db`. The echo-destination clause was written as "appends a one-line system note to the user's session immediately before the loom starts" in that same commit, with no subagent-mode treatment. The per-mode binder-footprint distinction ("the user's session (in prompt mode) or … the loom's spawned conversation (in subagent mode)") was authored in the very same commit but never propagated to the echo-destination clause. `git log -S "immediately before the loom starts"` shows only later doc-reorganisation (`fecb504` topic split, `31ff060`/`f5e89f4` docs move and size-cap split) and BNDR rendering refinements — none added a subagent echo-destination rule. The gap has therefore existed continuously since the feature's inception.

## Solution Space

**Shape:** single

### Recommendation

In [Echo policy](../spec_topics/binder/defaulting-system-note-echo.md#echo-policy), state that the bind-args echo lands in the caller's (user's) session immediately before the loom starts **regardless of whether the loom runs in `mode: prompt` or `mode: subagent`** — mirroring the wording SLSH-3 already uses for the failure-side slash-dispatch note. Add that the spawned subagent conversation never carries the echo (it is a slash-dispatch artefact, not a turn inside the subagent conversation, so it does not fall under the subagent-isolation rule that suppresses a subagent's *query* system notes). The echo is still subject to the existing `bind_echo: false` and single-string-bypass auto-suppression rules in both modes.

Edge case the implementer must watch: a subagent-mode loom invoked via `invoke(...)` or as a registered tool has no binder run at all (per `binder.md`, the binder applies only to slash invocation), so there is no echo to place in those paths — the rule scopes to slash-dispatched subagent looms only.

## Consequence

**Severity:** correctness

If the spec ships unfixed, two conforming implementations can place a subagent-mode slash loom's bound-args echo differently — caller session, spawned conversation, or nowhere — producing an observable user-facing divergence in what the operator sees after issuing the slash command. The narrow blast radius (one note, one mode) keeps it from being blocking, but the behaviour is not pin-downable from the current text.
