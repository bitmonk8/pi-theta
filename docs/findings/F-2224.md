## Finding

`session-only-degraded-state.md` opens with a clause-scoped implementation gate that declares two specific clauses **non-gated (buildable now)** — the *Drain-state tag write (state-independent)* paragraph (defining the `LoomRegistry.markRuntimeDegraded` setter contract) and the *Isolation* clause under *Diagnostic emission* — while gating "every other clause on this page," explicitly including the *Predicate split* clause.

The non-gated *Drain-state tag write* paragraph reads: "Immediately **after sub-step 4 returns and before sub-step 5 runs**, the handler writes the registry's drain-state tag to `"degraded-needs-reload"` **unconditionally**." The word *unconditionally* there is intended in the *state-independent* sense — the setter overwrites whatever the tag previously held without reading the prior state. But the surrounding prose ("the handler writes … immediately after sub-step 4 returns") also asserts a firing site and timing, with no mention that the write is conditioned on the event reason. The actual firing condition — the broad tag-transition predicate `capturedEventReason ∉ {"quit","reload"}` — lives entirely in the **gated** *Predicate split* clause, where it is declared to govern "the unconditional `"degraded-needs-reload"` tag write."

An implementer working strictly from the non-gated carve-out (the API/seam contract the `V9m`/`V9g` slices and the sibling `drain-state-contract.md` page are told they may consume now) sees a write that fires after sub-step 4 on every handler invocation, with no reason gate. The gated *Predicate split* says the write must fire only when `capturedEventReason ∉ {"quit","reload"}`. The two clauses therefore disagree about whether the write is unconditional in the firing sense, and the firing condition needed to wire the call site correctly is on the wrong side of the gate boundary the same banner draws.

**History:** The corpus is git-tracked. `session-only-degraded-state.md` was created at `f5e89f4` carrying both the *Drain-state tag write … unconditionally* paragraph and the `capturedEventReason`-keyed predicate content, but with a **file-level** gate ("The obligations on this page are blocked pending resolution of the open contradiction…") — both clauses were gated together, so no cross-gate dependency existed. `8bcd52a` re-scoped the gate "by clause, not by page," carving out the *Drain-state tag write (state-independent)* paragraph as non-gated while leaving the *Predicate split* (which holds the `∉ {"quit","reload"}` firing predicate the write depends on) gated. That clause-level partition is the edit that placed a non-gated clause's firing condition behind the gate boundary.

## Solution Space

### Recommendation

Narrow the non-gated *Drain-state tag write (state-independent)* paragraph in `session-only-degraded-state.md` so it states only the `LoomRegistry.markRuntimeDegraded` setter contract: when called, the setter overwrites the drain-state tag to `"degraded-needs-reload"` state-independently — without reading the prior value. Rephrase the paragraph's opening so the word "unconditionally" unambiguously means this *state-independent overwrite* sense, and keep the "state-independent / without reading the prior state" wording intact, since that overwrite semantics (acceptance criteria (i)/(j)) is the substantive non-gated obligation.

Remove the firing/timing assertion from this non-gated paragraph: replace "the handler writes … immediately after sub-step 4 returns and before sub-step 5 runs" with a statement that the call site and its firing predicate (`capturedEventReason ∉ {"quit","reload"}`) are defined by the gated *Predicate split* clause. This keeps the gate boundary intact — branch behaviour (including the firing predicate's three-arm analysis) stays gated, while only the setter API the `V9m`/`V9g` slices and the sibling `drain-state-contract.md` page actually consume is non-gated.

Add a one-line cross-reference in the gate banner stating that the non-gated carve-out is the setter contract, not the call-site firing logic, so a reader of the carve-out is pointed to the gated *Predicate split* for when the setter is invoked.

## Consequence

**Severity:** correctness

An implementer building the explicitly-buildable non-gated carve-out wires the drain-state write to fire after sub-step 4 on every reason, because the reason gate (`∉ {"quit","reload"}`) is gated out of reach. Two implementers reading the same carve-out diverge on whether the write fires on `"quit"`/`"reload"`. The runtime effect of an over-firing write is benign on those reasons (the host tears down anyway, per the *Harmlessness on quit/reload sub-cases* note), but the spec-level contract is contradictory and the non-gated portion cannot in fact be implemented in isolation.
