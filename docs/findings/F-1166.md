## Finding

`host-prerequisites.md`'s clause (a) under **Host prerequisites for the degraded-state branch** records, as the spec's own text, that the session-only degraded-state branch relies on a premise that **directly contradicts** Pi's published extension lifecycle at the pinned SDK version (`@earendil-works/pi-coding-agent ~0.75.5`). The branch presupposes that on a session-only `session_shutdown` (`reason ∈ {"new","resume","fork"}`) the same extension instance — and its closed-over `LoomRegistry`, drained flag, and `drainStateTag` — survives the swap, so that a subsequent slash invocation on the swapped-in session observes the `"degraded-needs-reload"` tag the handler wrote.

The contradiction has now been investigated against the SDK on disk and resolved. The verdict is **`governed-by-rebind`**: loom's package-loaded extension instance does **not** survive a session-only swap. The premise is false, and the entire session-only degraded-state feature is dead-by-construction.

**Investigation result (evidence at the loom 1.0 Pi-SDK pin).**

1. *Documented behaviour.* `docs/extensions.md` states for `/new`,`/resume` (L388), `/fork` (L405), and the `withSession` footgun list (L1117–1121): on these reasons Pi "emits `session_shutdown` for the old extension instance, **reloads and rebinds extensions** for the new session, then emits `session_start`"; "the old runtime has been torn down… the **new extension instance** has already received `session_start`… any state invalidated by your `session_shutdown` handler is already gone."

2. *Swap path re-runs the full loader.* `agent-session-runtime.js` — `switchSession`/`newSession`/`fork` each call `teardownCurrent(reason)` (emits `session_shutdown`, then `session.dispose()`, which `invalidate()`s the old runtime) and then `this.createRuntime({…})`. `main.js` (the single `createRuntime` factory reused for startup *and* every `/new`,`/resume`,`/fork`,import) calls `createAgentSessionServices(...)`, which constructs a **new `DefaultResourceLoader`** and `await resourceLoader.reload()` (`agent-session-services.js`). `reload()` unconditionally calls `loadExtensions(...)` (`resource-loader.js`).

3. *Loading rebuilds the instance and its closures.* `extensions/loader.js` `loadExtensionModule` creates jiti with **`moduleCache: false`** and re-imports the module; `loadExtension`/`createExtension` build a **fresh** extension object (new handler/tool/command Maps) and **re-run the factory** — so any closed-over `LoomRegistry`/drained flag/`drainStateTag` is reconstructed empty.

4. *No package-vs-dev carve-out.* `package.json#pi.extensions` manifests (loom's distribution path) funnel into the same `loadExtensions → loadExtension → loadExtensionModule` path as `index.ts`/`-e`. The only `isBunBinary`-vs-Node difference is jiti module *resolution*, not instance lifetime. `pi install` only changes the discovery root.

5. *The lone hopeful hint is disposed of.* `previousSessionFile` is a field on the `session_start` *event* handed to the *new* instance — session-level data, consistent with rebind, not evidence of instance survival.

Because a fresh instance with a non-drained `LoomRegistry` and freshly-established watchers answers every post-swap slash invocation, the degraded tag is never read, the degraded slash note never fires, and the `"runtime-degraded"` diagnostic describes a condition that does not persist — the exact silent-hiding failure mode the `/reload`-only recovery framing was built to prevent.

## Issue introduction

**Verdict:** multi-commit-interaction

**Introducing commits:**
- `52b9101` (2026-05-11, "resolve T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state") — **the commit that introduced the session-only degraded-state feature.** It added the deterministic-degraded-state framing, the operator-visible degraded signal, and the `/reload`-recovery contract to answer T24.
- `bfef26e` (2026-06-05) and earlier T22-family commits hardened the per-`event.reason` lifecycle prose the feature sits on.
- `f5e89f4` (2026-06-04, corpus size-cap split) carried the feature onto its own `session-only-degraded-state.md` page.
- `7447d1e` (2026-06-10, "resolve 'Session-only degraded-state presupposition (a) contradicts Pi's documented lifecycle'") reclassified clause (a) from an unpinned presupposition into the **open contradiction** and added the `MUST be resolved before SM-4/5/6` gate — i.e. it correctly *recognised* the defect but recorded it as open rather than resolving it.

**History — and why this is the load-bearing fact for the fix.** The session-only degraded-state feature was authored as the resolution of **T24** (importance: high; kind: clarity/cruft/completeness/error-model/assumptions). T24's complaint was that on a session-only `session_shutdown` the spec closed both watchers (sub-step 4) but **left the extension runtime alive** with no description of the resulting intermediate state and zero operator visibility: "On a `"fork"` shutdown the extension therefore enters a permanent state in which every subsequent slash invocation returns 'extension shutting down' until the operator issues `/reload`." T24's premise — *a live instance with closed watchers, silently no-opping edits* — is the **same instance-survival premise** the `governed-by-rebind` verdict now falsifies. Under teardown-and-rebind there is no surviving instance to be silently degraded: Pi disposes the old instance and loads a fresh one whose factory re-establishes the `discoveryWatcher`/`settingsWatcher`. The defect is therefore not that the degraded-state feature was implemented carelessly; it is that **both T24 and its fix were reasoned entirely from the loom side without the host lifecycle that makes the failure mode impossible.** Removing the feature does not reintroduce T24, because T24's feared state cannot occur (see the non-regression argument in the Solution Space).

## Solution Space

**Shape:** single

The verdict is dispositive, so this finding resolves to one option: **record the determination, then remove the session-only degraded-state feature.** (The previously-mapped Option A, `provably-differs`, is unreachable: the published lifecycle and the on-disk loader agree, with no package-loaded exception.)

### Step 1 — Record the determination

In `version-bump-step2.md` item (a) (anchor `#bump-checklist-instance-survival`), record the authoritative determination as **`governed-by-rebind`** in item (a)'s recording shape: actor = bump contributor; cited sites = `docs/extensions.md` L388/L405/L1117–1121, `agent-session-runtime.js` `teardownCurrent`/`createRuntime`, `main.js` `createRuntime`→`createAgentSessionServices`, `agent-session-services.js` (new `DefaultResourceLoader` + `reload()`), `resource-loader.js` (`loadExtensions`), `extensions/loader.js` (`moduleCache:false`, fresh factory run); evidence = the chain above. This closes the open contradiction and clears the implementation gate by **removing the gated obligations**, not by satisfying them.

### Non-regression argument (does not reintroduce T24)

T24 must be explicitly addressed in the removal commit message and in the rewritten clause (a), because the surface fear ("watchers closed, runtime alive, edits silently no-op") looks like exactly what removing the degraded-state machinery would leave behind. It does not, for one reason: **T24's premise is the instance-survival premise this finding falsifies.** Under `governed-by-rebind`:

- On `/new`,`/resume`,`/fork`, Pi disposes the old instance after `session_shutdown` and loads a **fresh** instance whose factory re-runs and re-establishes the `discoveryWatcher`/`settingsWatcher`. There is no live-but-watcher-less instance, hence no "silently degraded state" and no window in which editor edits are silently ignored.
- Closing the watchers in sub-step 4 is now plain teardown cleanup on an instance about to be discarded (release file handles before disposal), exactly the "do cleanup work in `session_shutdown`" pattern `docs/extensions.md` prescribes — not the entry into a persistent degraded mode.
- `/reload` remains a real host capability (`reason: "reload"`, factory re-executed), but it is no longer "the only recovery path for a session-only degraded state," because no such state exists.

The rewritten clause (a) MUST record this reasoning (T24's failure mode is **prevented by the host lifecycle, not merely unmitigated**) so a future reviewer scanning the teardown sequence does not re-file T24 against the now-feature-less page.

### What must NOT change

To bound the blast radius: the five-sub-step `session_shutdown` teardown sequence, the `"shutting-down"`/`drained` short-circuit for genuine teardown reasons, and the four surviving teardown diagnostics are independent of the verdict and stay intact.

## Consequence

**Severity:** blocking

Until the determination is recorded and the feature removed, `V9l`/`V9l-T` cannot proceed and SM-3b/4/5/6 are not implementable as written — `plan.md` already classifies `V9l` as **blocked** on this resolution. Worse, `session-only-degraded-state.md` reads as if its dozens of normative MUSTs were buildable; an implementer who skips the *Implementation gate* blockquote would build against contradicted behaviour and lock the wrong contract into tests. The removal eliminates a substantial, internally-elaborate but unreachable surface (the degraded predicate split, seam-minimality rules, read-failure arm (c), idempotent re-entry/at-most-once machinery, residual-gap analyses, and acceptance criteria (a)–(m)), and resolves the downstream findings listed below.
