## Finding

`diagnostic-emission-isolation.md` pins several byte-exact emission forms for the teardown-time `console.error` sites, but its `## Acceptance criteria` section only enumerates fixtures for the **construction-site** catch arm (PIC-26, building on PIC-25): cases (a) two-token form on a `details.event`/hoist throw for `loom/host/session-shutdown-runtime-degraded`, (b) three-token form for `loom/runtime/cancelled-by-session-shutdown`, and (c) the self-wrapped inner throw. The section's own framing scopes it to "The construction-site catch-arm obligation … decomposes into the fixture obligations below."

Two other normatively-pinned behaviours on the **main** (non-construction-site) path carry no acceptance fixtures at all:

- **PIC-25 serialiser-throw fallbacks.** On a serialiser throw in the wrapped serialise-and-emit sequence, the catch arm MUST emit the bare diagnostic code `console.error(code)` for flat-`details` codes, and the two-token `` console.error(`${code} ${detailsEventReason}`) `` form for the two nested-shape codes — including the `"<unreadable>"` sentinel substituted on the sub-step-2 stamp-throw clean-cancel path — and MUST NOT re-throw. These forms are byte-exact and load-bearing for the per-reason operator-dedup buckets, yet nothing asserts them.
- **PIC-27 handler-isolation swallow.** A throw out of `console.error` on the main path (closed stdio, FD exhaustion, a `console`-proxying host) MUST be swallowed, with the handler continuing to the next sub-step (or into sub-step 1 for the pre-sub-step-1 Unknown-reason emission) and no further diagnostic emitted. No fixture exercises this swallow on the main path; only the construction-site self-wrap swallow (case (c), PIC-26) is covered.

The construction-site fixtures do not transitively cover these: the construction-site path and the wrapped-sequence path are explicitly mutually exclusive, so a fixture forcing a construction-site throw never enters the serialise-and-emit catch arm where the bare-code/two-token fallback and the PIC-27 swallow live.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 9559e19 (2026-05-07, `"<unreadable>"` sentinel), 75a9626 (2026-05-09, bare diagnostic-code serialiser-throw fallback), 52b9101 (2026-05-11, two-token form + `MUST be swallowed`), 8008b91 (2026-06-25, F-1217, added the `## Acceptance criteria` section)
**History:** The normative emission forms accreted across several finding-resolution commits, each adding an obligation without a paired fixture: the `"<unreadable>"` sentinel (9559e19), the bare-code serialiser-throw fallback (75a9626), and the two-token form plus the handler-isolation swallow (52b9101). The forms then moved verbatim into this file when the oversized `pi-integration-contract.md` was split (f5e89f4). The `## Acceptance criteria` section was introduced later by F-1217 (8008b91), but scoped deliberately to the construction-site catch-arm (PIC-26) only — entrenching the gap rather than closing it for the main-path PIC-25 fallbacks and the PIC-27 swallow. The defect is the interaction between the obligation-adding commits and the partial acceptance block, not any single revert.

## Solution Space

**Shape:** single

### Recommendation

Extend the existing `## Acceptance criteria` section with main-path fixtures that mirror the construction-site block's byte-exact / branch-coverage discipline, keying on the same codes already in scope and introducing no new diagnostic code or wire field:

- **Serialiser-throw — bare-code fallback (flat-`details` codes).** Force the leaf-owned serialisation primitive to throw inside the wrapped serialise-and-emit sequence for a flat-`details` teardown code (e.g. `loom/host/session-shutdown-teardown-step-failed`); assert the catch arm emits the byte-exact `console.error(code)` bare form, does not re-throw, and the emission counts as the single invocation against the per-event bound (PIC-28: the fallback is the same wrapped sequence, not a second invocation).
- **Serialiser-throw — two-token fallback (nested-shape codes).** Force a serialiser throw for `loom/host/session-shutdown-runtime-degraded` and for `loom/runtime/cancelled-by-session-shutdown`; assert the catch arm emits `` console.error(`${code} ${detailsEventReason}`) `` interpolating the pre-hoisted local, does not re-throw, and preserves the per-reason dedup bucket. Include the sub-step-2 stamp-throw clean-cancel variant where the hoisted local is the literal `"<unreadable>"` sentinel, asserting it is interpolated verbatim.
- **PIC-27 handler-isolation swallow (main path).** With the wrapped sequence reaching its `console.error` (structured or bare-code fallback), force `console.error` itself to throw; assert the throw is swallowed, no further diagnostic is emitted on the inner throw, and the handler continues to the next sub-step — and that emission ordering is preserved for the Unknown-reason emission that runs *before* sub-step 1 (it still continues into sub-step 1).

Edge cases the fixtures must hold: the bare-code/two-token fallback is the catch arm of the same wrapped `try`/`catch`, so it never counts as a second invocation against the per-event "exactly once"/"at most once" bound; retry-on-inner-throw is forbidden; and the sentinel substitution must keep the construction-site path's value space inside the captured-value enumeration the Unknown-reason rule pins. The terminal `console.error` of the `sendSystemNote` fallback chain is governed by PIC-54 at its own definition site (`runtime-event-channel.md`) — reference it rather than restating its swallow here.

## Consequence

**Severity:** advisory

The emission forms are fully pinned, so two implementers will not diverge on *what* to build; the risk is that the bare-code/two-token serialiser-throw fallbacks and the PIC-27 main-path swallow ship untested, leaving the operator-dedup-preserving fallback wire forms and the "never unwind the handler" guarantee unverified. A regression in those branches (e.g. a re-throw on serialiser failure, or a lost `"<unreadable>"` sentinel collapsing per-reason dedup buckets) would pass acceptance.
