## Finding

`V9b`'s Adds names `V9b` as the single declaration site of the test-only `ReloadFailureInjector` interface and states that `V9b` itself wires **two** arms onto that interface: the registry-swap arm (`loom/runtime/registry-swap-failed`) and the `.loom`/`.warp` re-parse arm. Both arms are supposed to route a synthetic failure onto the `loom-system-note` surfacing path through the one `injectReloadFailure` method.

`V9b-T` (and the mirrored `V9b` Tests section) assert only the registry-swap arm: `PIC-36` covers atomic swap and the `loom/runtime/registry-swap-failed` emission, and the remaining bullets cover `getCommands()` read-only-ness (`PIC-39`), empty-window suppression (`PIC-37`), same-window rename emission (`PIC-38`), and model-reference-matcher wiring. No bullet drives a synthetic `.loom`/`.warp` re-parse failure through `injectReloadFailure`, so the re-parse arm — wired by `V9b` — is never exercised at its owning leaf.

The only delegated coverage is `V4g`/`V4g-T`'s `ERR-7`, but its wording exercises "both the re-parse/re-merge diagnostic arm and the `loom/runtime/registry-swap-failed` registry-swap arm" — it treats `V9b`'s re-parse arm and `V10d`'s settings-re-merge arm as one combined "re-parse/re-merge diagnostic arm". A faithful `ERR-7` implementation that injects only a settings-re-merge failure (V10d's arm) satisfies that combined clause while never touching the `.loom`/`.warp` re-parse arm. Consequently the re-parse arm has no gate — neither leaf-local at `V9b` nor arm-specific at `V4g` — and can ship green while broken.

## Issue introduction

**Verdict:** present-since-inception (of the unified multi-arm seam)
**Introducing commits:** `44a96f5` ("pi-loom plan: resolve \"V9b filename slug contradicts retitled scope\"", 2026-06-16) introduced the `injectReloadFailure` multi-arm seam into `V9b` with the Adds wiring both the registry-swap arm and the `.loom`/`.warp` re-parse arm, while `V9b-T` (authored in the same commit) gated only the registry-swap arm. `324813d` ("split ERR-7 reload scenario out of V4e routing checks", 2026-06-18) relocated the delegated `ERR-7` coverage from `V4e` to the new `V4g`/`V4g-T` and introduced the combined "re-parse/re-merge diagnostic arm" phrasing that does not pin the re-parse arm specifically.
**History:** `git log -G "wires the registry-swap arm" -- V9b...` and `git log -S "injectReloadFailure" -- V9b-T...` confirm `V9b-T` has carried no re-parse-arm test bullet at any point since the seam was authored; the gap is co-extensive with the existence of the two-arm seam, not introduced by a later regression.

## Solution Space

**Shape:** multiple

### Option A — Add a re-parse-arm assertion to V9b's own test leaf

- **Approach:** Add a Tests bullet to `V9b-T` (mirrored into `V9b`'s Tests section per the per-phase TDD ritual) that supplies a synthetic `.loom`/`.warp` re-parse failure to `ReloadFailureInjector.injectReloadFailure` and asserts it routes onto the `loom-system-note` surfacing path, kept distinct from the registry-swap arm's `loom/runtime/registry-swap-failed` assertion so it cannot pass by exercising the swap arm.
- **Plan edits:** `V9b-T` Tests (new bullet); `V9b` Tests (mirror); extend `V9b` Ships when to name the re-parse-arm routing observable.
- **Spec edits:** None — cite the existing re-parse arm / `loom-system-note` routing anchor in `registration-steps.md` (or `error-model.md` ERR-7).
- **Pros:** The gate lives at the arm's owning leaf; the re-parse arm is verified the moment `V9b` ships, independent of `V4g`'s later integration test and its combined-arm phrasing.
- **Cons:** Small overlap with `V4g`'s `ERR-7` integration assertion.
- **Risks:** Minimal.

### Option B — Make V4g's ERR-7 exercise the re-parse arm specifically

- **Approach:** Re-author `V4g`/`V4g-T` `ERR-7` so it separately injects a `.loom`/`.warp` re-parse failure (distinct from `V10d`'s settings-re-merge arm), guaranteeing the re-parse arm is specifically driven rather than satisfied by the combined "re-parse/re-merge diagnostic arm" clause.
- **Plan edits:** `V4g` Tests and `V4g-T` Tests (split the combined arm clause into arm-specific assertions).
- **Spec edits:** None.
- **Pros:** Keeps all reload-arm coverage consolidated in the dedicated integration leaf.
- **Cons:** The re-parse arm wired in `V9b` stays without a gate at its own leaf, so the "leaf-local" gap the finding names persists; coverage of a `V9b`-wired arm depends on `V4g` (which depends on `V9b`) landing.
- **Risks:** The arm-agnostic "re-parse/re-merge" phrasing could re-creep on later edits.

### Recommendation

Option A. The re-parse arm is wired by `V9b`, so its gate belongs at `V9b-T`; placing it there closes the leaf-local gap the finding names and removes reliance on `V4g`'s combined-arm phrasing. The new bullet must drive the re-parse arm specifically — a synthetic `.loom`/`.warp` re-parse failure through `injectReloadFailure` asserting routing onto `loom-system-note` — and stay distinct from the registry-swap arm's `loom/runtime/registry-swap-failed` assertion so it cannot pass vacuously by exercising the swap arm.

## Consequence

**Severity:** advisory

The `.loom`/`.warp` re-parse failure arm wired by `V9b` can regress or be mis-implemented (e.g. fail to route onto `loom-system-note`, or route with the wrong `triggerTurn` posture) without any test going red, because `V9b-T` asserts only the registry-swap arm and `V4g`'s `ERR-7` is satisfied by exercising the settings-re-merge arm alone. The implementer still has clear wiring guidance, but the coverage gate over this arm is vacuous.
