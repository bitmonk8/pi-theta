## Problem

The grammar appendix in `docs/spec_topics/grammar.md` invokes a `Stmt` nonterminal at five block-production sites (`BlockExpr`, `FnBody`, `LoomBody`, `StmtBlock`, and `ArmBody`'s `BlockExpr` alternative) but never writes a `Stmt ::= …` production. The only `*Stmt`-typed RHS defined is `LetStmt`; the remaining statement forms (`IfStmt`/`WhileStmt`/`ForStmt`, reassignment, expression-statement, `return`/`break`/`continue`, `import`/`export`, and top-level declarations) are described only in prose scattered across other topic pages. A reader consulting the appendix has no enumeration of what `Stmt*` admits. The two reasonable readings of the undefined `Stmt` — "anything not an `Expr`" versus "exactly the enumerated forms" — produce materially different parsers, so the gap is a correctness defect, not an editorial one.

## Issue introduction

**Verdict:** present-since-inception (then accreted)
**Introducing commits:**
- `f841b12` (2026-05-04, *spec: resolve Cluster 2 (literal sublanguage + grammar appendix)*) — the appendix was created and immediately referenced `Stmt*` in the `match`-arm block-expression alternative (`ArmBody ::= … | "{" Stmt* Expr "}"`) without defining `Stmt ::=`. Of the `*Stmt`-suffixed nonterminals it introduced, only `LetStmt` had a RHS.
- `52219c4` (2026-05-06, *Address all 41 spec-review findings*) — added the `BlockExpr` / `FnBody` / `LoomBody` productions, each of which uses `Stmt*`. The bare `Stmt` was still undefined, so the new productions inherited the original omission and broadened its surface from one site to four.
- `816852b` (2026-06-05, *resolve "Block expressions — no grammar production for statement-form if/for/while bodies"*) — added `StmtBlock` (a fifth `Stmt*` site) and the `IfStmt` / `WhileStmt` / `ForStmt` productions (implicit `Stmt` members), again without defining `Stmt`.

**History:** the defect was present at the appendix's inception and grew with each subsequent enrichment of the block-production set; no commit ever revisited the bare `Stmt` to give it a RHS. Subsequent commits (`190f10e` adding `FnDecl`, `4e656ac` generalising the tail-`?` carve-out) touched the surrounding text without addressing it.

## Solution approach

Clarify the `Stmt` nonterminal by adding a `Stmt ::= …` production to `grammar.md`, positioned so the block productions that reference it sit below the definition. Enumerate the closed statement set, referencing the existing `LetStmt` production rather than duplicating its RHS, and add forward-links to each alternative's owning topic page rather than restating its semantics. Mark the top-level-only subset (`import`/`export`, schema/enum/`fn` declarations) as grammatical members of `Stmt` whose placement is restricted to `LoomBody` / `.warp` top level, so the closed set is not misread as permitting them anywhere a `Stmt*` is admitted.

## Solution constraints

- Out of scope: the statement-form topic pages — the new `Stmt` production cross-references them rather than restating their semantics; do not edit them.
- Out of scope: the `StmtBlock` tail-`?` carve-out clause (owned by F-1289).
