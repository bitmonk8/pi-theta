## Finding

`V12a`'s three `SLSH-2` test bullets exercise behaviours that only the end-to-end harness can produce: scripted streamed assistant fragments and an `Err`-after-partial-text path (H4b response-programming-surface category (a) — "script the sequence of assistant turns … and the streamed fragments per turn"), and a mid-stream cancellation injection (H4b category (e) — "inject an abort/cancellation at a chosen point in the drive"). The first `SLSH-2` bullet additionally names the **session-double fidelity contract** in `H4a` directly ("whose `ctx.waitForIdle()`-vs-streaming ordering is fixed by the session-double fidelity contract ([`H4a`](./H4a-factory-shell-and-harness.md))"), consuming fidelity-contract item (i) (streamed tokens observable before `ctx.waitForIdle()` resolves).

Despite this, `V12a`'s `Deps` field lists only `V12a-T`, `V9c`, `V11f`, `V4a`, `V13c` — neither `H4a` nor `H4b`. Symmetrically, `H4b`'s `Adds` paragraph enumerates "the single shared scripting contract **every** harness-driven leaf consumes — `H7a`, `V4c`, `V4f`, `V9i`, `V11f`, `V13c`, `V13d`, `V13f`, `V14b`, and `V17a`" and omits `V12a`. The contract owner's own enumeration therefore undercounts its consumers, contradicting its "every harness-driven leaf" claim.

The ordering is incidentally preserved today because two of V12a's existing deps — `V11f` and `V13c` — both list `H4b` in their `Deps`, and `H4b` deps `H4a`, so `H4a`/`H4b` are transitively reachable from `V12a`. The defect is that the consumption is undeclared on V12a's side and unrecorded on H4b's side: if a later refactor dropped `H4b` from `V11f`/`V13c`'s deps, `V12a`'s harness/fidelity ordering guarantee would silently vanish, and any audit of H4b's consumer set is already wrong.

## Issue introduction

**Verdict:** present-since-inception (with a compounding propagation miss at the H4a/H4b split)
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review") for the undeclared-dependency facet; `fd6169b` (2026-06-12, "pi-loom plan: resolve \"H4a bundles three independent units into one leaf\"") for the consumer-list omission.
**History:** The plan corpus is git-tracked. `V12a-slash-dispatch.md` was created in `c6a664e`; at creation its `Deps` were already `V12a-T, V9c, V11f, V4a, V13c` and its `SLSH-2` bullets already drove "the in-process Pi session double" without declaring any harness leaf (`git log -S 'H4b'` over the file returns no commit — `H4b` was never present in its `Deps`). The `H4b` leaf did not yet exist at that point: the response-programming surface and fidelity contract lived inside `H4a` until `fd6169b` split `H4a` into `H4a` + `H4b` and authored the consumer enumeration (`H7a, V4c, V4f, V9i, V11f, V13c, V13d, V13f, V14b, V17a`), which omitted `V12a`. The SLSH-2 prose that names the H4a fidelity contract directly was added around `85fc906` (2026-06-10). The core omission (V12a never declaring the harness/fidelity surface it consumes) thus predates the H4a/H4b split; the consumer-list asymmetry was introduced as a propagation miss at the split commit.

## Solution Space

**Shape:** single

### Recommendation

Make the V12a↔H4a/H4b consumer relationship explicit on both ends:

- In `docs/plan_topics/V12a-slash-dispatch.md`, add `H4b` to the `Deps` field — V12a scripts streamed assistant fragments (H4b category (a)) and injects a mid-stream cancellation (H4b category (e)) for its `SLSH-2` bullets.
- In the same `Deps` field, add `H4a` — the first `SLSH-2` bullet consumes session-double fidelity-contract item (i) (streamed tokens observable before `ctx.waitForIdle()` resolves) directly, not only transitively through H4b.
- In `docs/plan_topics/H4b-response-programming-surface.md`, add `V12a` to the consumer enumeration in the `Adds` paragraph (the "`H7a`, `V4c`, `V4f`, `V9i`, `V11f`, `V13c`, `V13d`, `V13f`, `V14b`, and `V17a`" list), so the "every harness-driven leaf consumes" claim is accurate.

Edge case for the implementer: `H4a`/`H4b` are already transitively reachable from `V12a` via `V11f` and `V13c` (both dep `H4b`, which deps `H4a`), so these edits add no new ordering constraint and cannot introduce a cycle — they record an existing dependency that was implicit.

## Consequence

**Severity:** advisory

`V12a` can still be implemented and ordered correctly today because `H4a`/`H4b` are reachable transitively through `V11f` and `V13c`, so nothing is currently blocked. The gap is a traceability/durability one: `H4b`'s contract owner has no record that `V12a` scripts categories (a) and (e), and V12a's direct consumption of fidelity-contract item (i) is undeclared — so a future change to V11f/V13c's deps could silently strip V12a's harness ordering guarantee, and H4b's "every harness-driven leaf" enumeration is already inaccurate.
