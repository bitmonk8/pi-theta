## Finding

`unknown-reason-rule.md` (and the *Hoist obligation* clause) pin the variable names `capturedEventReason`, `pinnedConstantReadOk`, and `detailsEventReason`, which never appear in observable output.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `52219c4` (2026-05-06, "Address all 41 spec-review findings", tied to the Step-0 capability-probe constant block established same-day in `126ff8a`); `75a9626` (2026-05-09, "pi-loom spec: resolve \"SessionShutdownEvent.reason closed set has no build-time pin against the SDK type\""); `52b9101` (2026-05-11, "pi-loom spec: resolve \"T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state\"")
**History:** `git log -S` over the corpus shows no single commit introduced "the over-prescription"; the cluster accreted as a sequence of well-intentioned fix commits, each adding mechanism prose to close a specific earlier finding. The `SHUTDOWN_AWAIT_CAP_MS` "same source-of-truth pinned-constants block" sentence entered first (`52219c4`, alongside the `126ff8a` capability-probe constant-block rewrite). The `SDK_SURFACE_INVENTORY === undefined` guard entered with `75a9626` while closing the build-time-pin gap on the reason closed set. The *Hoist obligation*, `capturedEventReason`, and `pinnedConstantReadOk` clauses all entered together in `52b9101` while closing the T24 degraded-state defect. Each fix tightened mechanism to nail a specific observable, and the cumulative drift past PIC-1(e)'s observable-only stance is the interaction this finding flags — not any one commit.

## Solution Space

**Shape:** single

### Recommendation

Replace the pinned names with their observable/behavioural descriptors — the handler-entry captured-reason value defaulting to `"<unreadable>"`, the snapshot-read-success flag defaulting to `false` and set true on the snapshot success arm, and the hoisted reason value feeding the two-token catch arm. Where `session-only-degraded-state.md`'s *Predicate split* clause consumes `pinnedConstantReadOk` as a "seam", refer to the snapshot-read-success flag conceptually rather than by runtime identifier (or mark the name explicitly non-normative-referent-only).

## Consequence

**Severity:** advisory

Implementers reading the clauses as MUSTs are needlessly forbidden from using equivalent, conforming alternatives (a different undefined-guard expression, a re-read from a stable source instead of a hoisted local, other variable names, a different constant layout), and the over-pinned mechanism grows the normative surface future maintainers and conformance fixtures must track. No two conforming implementations diverge in observable behaviour, so this is a guidance/quality gap rather than a correctness hazard.
