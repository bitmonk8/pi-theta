## Finding

The `loom-system-note` best-effort model in `runtime-event-channel.md` rests entirely on one behavioural property of the host: `pi.sendMessage` returns `void` synchronously and never returns a rejecting `Promise`. The page states it directly — "`pi.sendMessage` returns `void` (synchronous); the runtime MUST NOT `await` it and MUST NOT attach a `.catch` handler" — and the whole emission discipline (no `await`, no `.catch`, swallow only synchronous throws, exactly-once accounting, the `sendSystemNote → ctx.ui.notify → console.error` fallback that fires only on a synchronous throw) is built on it. If a future Pi minor made `pi.sendMessage` return a rejecting `Promise`, the rejection would escape unobserved as an `unhandledRejection`, because the runtime is explicitly forbidden from awaiting or catching it.

Unlike the comparable behavioural presuppositions the corpus consumes, this one is left un-pinned on both axes the spec otherwise uses to protect such assumptions:

- **No declaration pin.** Sibling Pi surfaces are pinned to a concrete `.d.ts` declaration site — `pi.on` and `pi.getFlag` to `dist/core/extensions/types.d.ts`, `CustomMessage` to `dist/core/messages.d.ts`, `AssistantMessage.stopReason`/`errorMessage` to `@earendil-works/pi-ai`'s `dist/types.d.ts`, `complete()` to `dist/stream.js`. `pi.sendMessage`'s `void`-returning declaration is asserted only as prose — `conversation-drive.md` says "the `ExtensionAPI` surface declares both `sendUserMessage(...)` and `sendMessage(...)` as `void`-returning" with no file/signature citation, and `capability-inventory-items.md` item 6 names the call shape but not the return type or its declaration site.
- **No bump-checklist enrollment.** The version-bump editorial-review checklist (`version-bump-step2.md`, items (a)–(as)) carries dedicated items for directly analogous never-rejecting / void behaviours — item (al) for `ctx.waitForIdle()` no-rejection, item (ab) for `complete()`'s `options.signal` rejection-on-abort, item (m) for the `AssistantMessage` transport-failure fields. There is no item for `pi.sendMessage` returning `void` / never a rejecting `Promise`. The build-time surface-inventory assertion (step 2(a)) is a presence-only literal-read, and the Step-0 capability probe is `typeof`-only by PIC-4, so neither mechanically witnesses the return type. Consequently a Pi minor that turned `pi.sendMessage` async-rejecting would pass every gate and silently break the best-effort model with no detection at bump time.

This also violates the spec's own closing discipline on that checklist: "Any future presupposition added to this page … that routes its detection to editorial review under this procedure MUST be added to this checklist in the same edit." The void / never-rejecting property is exactly such a presupposition, and it is absent.

The `capability-probe.md` "silent shape drift" paragraph does dispose of the *runtime* outcome — a previously-probed member that becomes a `Promise`-returning function whose rejection the runtime never observes is declared undefined behaviour with no diagnostic owed. That covers what happens if drift occurs in production; it does not pin the declaration and does not provide the *bump-time* detection hook that every sibling presupposition gets, which is what this finding is about.

## Issue introduction

**Verdict:** single-commit (introduced; remediation gap perpetuated)
**Introducing commit:** `29c29b2` (2026-05-06) — *pi-loom spec: resolve "pi.sendMessage is synchronous (void); spec treats it as returning a Promise"*
**History:** The corpus is git-tracked. `git log -S` over the defect tokens shows the prose "`pi.sendMessage` returns `void` (synchronous); the runtime MUST NOT `await` it and MUST NOT attach a `.catch` handler" was introduced verbatim by `29c29b2`, the commit that flipped the model from treating `pi.sendMessage` as a `Promise` to treating it as synchronous `void`. That commit added the behavioural assertion but no `.d.ts` declaration pin and no version-bump re-validation hook. Two later corpus-wide sweeps closed the analogous gap for sibling surfaces but passed over this one: `158ed09` (2026-06-05, *Missing owning-declaration pins for bare Pi SDK symbols*) added `.d.ts` declaration pins for bare Pi symbols, and `257c545` (2026-06-05, *Host presuppositions lack version-bump-procedure re-audit hooks*) added the editorial-review checklist hooks for host presuppositions — neither touched `pi.sendMessage`'s void / never-rejecting property (their diffs reference `pi.sendMessage` only incidentally). The defect therefore dates to the introduction of the synchronous-void model and survived the two passes that established the very pin-and-enroll discipline it is now missing against.

## Solution Space

**Shape:** single

### Recommendation

Bring `pi.sendMessage` onto the same footing as the other consumed Pi surfaces, mirroring the established pattern for behavioural never-rejecting presuppositions:

1. **Pin the declaration.** At the no-`await` rule in `runtime-event-channel.md` (and on capability 6 in `capability-inventory-items.md`), cite `pi.sendMessage`'s declaration site, signature, and `void` return — the `ExtensionAPI` declaration at `dist/core/extensions/types.d.ts` at the loom 1.0 Pi-SDK pin, the same file the spec already pins `pi.on` and `pi.getFlag` to — so the "returns `void`" claim names a concrete declaration like every sibling surface rather than living only as `conversation-drive.md` prose.

2. **Enroll in the version-bump checklist.** Add one editorial-review item to the (a)–(as) list in `version-bump-step2.md`, modelled on item (al) (`ctx.waitForIdle()` no-rejection): the contributor SHOULD confirm against the candidate Pi minor that `pi.sendMessage` still returns `void` and never a rejecting `Promise`, anchored to the `runtime-event-channel.md` best-effort model (no-`await`/no-`.catch`/swallow-synchronous-throw), and MUST record the per-item outcome — so a minor that makes the surface async-rejecting is caught at bump time and the spec satisfies its own "every editorial-review-routed presupposition MUST be on the checklist" rule.

Edge cases for the implementer: the new pin must be an **editorial-review** item, not a mechanical step-2(a) gate — the surface-inventory literal-read is presence-only and the Step-0 probe is `typeof`-only (PIC-4), so neither can witness a return type; do not attempt to encode the `void` return as a mechanically-asserted `SDK_SURFACE_INVENTORY` row. The existing `capability-probe.md` "silent shape drift = undefined behaviour" disposition is the runtime-time outcome and stays as-is; the new item closes the orthogonal bump-time **detection** gap, so the two are complementary, not redundant.

## Consequence

**Severity:** correctness

The entire best-effort emission model — no `await`, no `.catch`, swallow-on-synchronous-throw, and the exactly-once runtime-event accounting layered on it — silently breaks if Pi makes `pi.sendMessage` async-rejecting: rejections escape as `unhandledRejection`s the runtime never sees. Because the property is neither declaration-pinned nor enrolled in the bump checklist (while every comparable property is), such a regression ships with no bump-time signal, and the spec is internally inconsistent with its own rule that editorial-review-routed presuppositions must appear on the checklist.
