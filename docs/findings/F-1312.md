## Problem

`H5d` (the transitive-completeness closing-gate arm) sits in no dependency closure reachable from the terminal release-gate leaf `H6a`. `H6a`'s `Deps.` is `H5b, H5c, H7a`; `H5b`'s coverage-producing set does not name `H5d`, no other leaf names it, and `H5d`'s own `Deps.` is just `H5a`. Under the plan's dep-driven pickability model, `H6a` can be picked and flip the closing gate to its hard-fail live-corpus footing with `H5d` never having been built or run, so the transitive-completeness backstop that `conventions.md` §REQ-ID discipline advertises as "mechanically backed" can be absent exactly when the release gate fires. `H5d` is the only gate arm left out of the release-gate closure entirely — `H5e`/`H5f` reach it through `H5b`, and `H5c` is pinned directly on `H6a`.

## Issue introduction

**Verdict:** introduced by a single commit
**Introducing commit:** `617a3ad` — "pi-loom plan: resolve 'H5a bundles gate arms spanning four data sources'" (2026-06-16)
**History:** `617a3ad` split `H5d` out of `H5a` as a standalone arm and gave it `Deps. H5a`; that commit touched only `docs/plan.md`, `H5a`, and the new `H5d` file — it added `H5d` neither to `H5b`'s `Deps.` nor to `H6a`'s. `git log -S 'H5d'` over both `H6a-live-corpus-activation.md` and `H5b-warn-only-canary.md` returns no commits, confirming `H5d` has never appeared in either `Deps.` closure since its creation. The later `9d63d2a` established the direct-pin precedent for a non-coverage-producing arm (`H5c`) but did not extend it to `H5d`.

## Solution approach

Add `H5d` to `H6a`'s `Deps.` in `docs/plan_topics/H6a-live-corpus-activation.md`, forcing it into the release-gate dependency closure so its arm must be landed and running green before `H6a` activates the hard-fail footing. Pin it directly on `H6a` rather than threading it through `H5b`, since `H5d` closes no new spec REQ-ID — the same rationale already given for the direct `H5c`/`H7a` pins — and extend the `Deps.` rationale parenthetical to record `H5d` alongside `H5c` as a directly-pinned non-coverage-producing arm. Leave `H5d`'s own `Deps.` (`H5a`) unchanged.
