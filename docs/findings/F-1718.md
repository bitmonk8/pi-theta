## Finding

PIC-13's `realpath` bullet (`docs/spec_topics/pi-integration-contract/host-interfaces-services.md:114`) binds the **Case-canonicalisation guarantee** — "on a case-insensitive filesystem, two input paths that differ only in component or leaf case but resolve to one directory entry MUST yield byte-identical `realpath` output" — onto *both* the production `PiFileSystem` adapter and `FakeFileSystem`, and explicitly states that bare delegation to `fs.promises.realpath` is insufficient (Node's `fs.promises.realpath` and `fs.promises.realpath.native` differ on leaf-case output, so the adapter must canonicalise leaf case to satisfy it). This makes the guarantee a real conformance obligation on the production adapter, not just on the fake.

The witnessing fixture — two case-variant paths that resolve to one directory entry — cannot be constructed against the production adapter on a case-sensitive filesystem. On Linux/ext4, the default `npm test` CI substrate, `Plan.loom` and `plan.loom` are *distinct* directory entries, so the inputs the assertion needs do not exist; on macOS default APFS or Windows NTFS they are alternate spellings of one entry and the assertion is meaningful. PIC-13 carries no platform-host requirement, no skip protocol, and no delegation to the manual real-host smoke gate (`docs/plan_topics/real-host-smoke-gate.md`) that already houses comparable real-host-only behaviours. `V8b`'s **Tests** bullets stop at the Node `.code` mapping and the `readBytes` raw-bytes contract — they neither assert the case-canonicalisation guarantee nor record it as deferred — and the project-wide "no silent test skipping" rule (`CLAUDE.md` §Testing) forbids the obvious workaround of an early-return / skip-macro on Linux, since that would report success while having verified nothing.

The result is a normative MUST whose production half is not exercisable under `npm test` on the only CI substrate the project commits to, and is not routed to any other witness either, leaving the leaf-case canonicalisation behaviour in the production `PiFileSystem` adapter unverified.

The guarantee was therefore untestable against the production adapter under `npm test` from its first commit, and remains so on HEAD.

## Solution Space

**Shape:** multiple

### Option A — Require production-adapter case-canonicalisation test on a case-insensitive host with loud platform-skip

**Approach.** Amend PIC-13's `realpath` bullet to state that the case-canonicalisation guarantee's production-adapter conformance test MUST run on a case-insensitive filesystem and MUST loudly fail-skip on a case-sensitive host (per `CLAUDE.md` §Testing: no silent skips — use `assert!`/`panic!` to surface the skip as a recorded outcome, not an early return). Add the assertion as a `V8b` / `V8b-T` **Tests** bullet that constructs `Plan.loom` and `plan.loom` in a temp directory, calls `PiFileSystem.realpath` on both, and asserts byte-identical output; on a case-sensitive host the bullet executes a loud-skip recorder that flags the unverified-on-this-host outcome through a recognised channel (e.g. logs a `loom-system-note`-equivalent test artefact and continues, rather than silently passing).

**Spec edits.**
- In PIC-13's `realpath` bullet, append after the **Case-canonicalisation guarantee** sentence a clause: "*Production-adapter conformance test.* This guarantee's production-adapter half is verified by a conformance test that MUST run on a case-insensitive filesystem (Windows NTFS, macOS default APFS); on a case-sensitive host the test MUST loudly record an unverified-on-this-host outcome rather than silently pass."
- In `V8b` / `V8b-T`, add a **Tests** bullet: "`PIC-13` (case-canonicalisation, production adapter): with `Plan.loom` and `plan.loom` constructed in a temp directory, `PiFileSystem.realpath` on both inputs yields byte-identical output; the test loud-fail-skips on a case-sensitive host."

**Pros.**
- Keeps the production-adapter assertion inside the automated `npm test` surface on hosts where it is constructible.
- Detects the bare-`fs.promises.realpath` regression at first developer run on Windows/macOS, before merge.
- Requires no manual-gate discipline once the host has the right filesystem.

**Cons.**
- On a Linux-only CI (the project's current substrate) the loud-skip permanently records "unverified-on-this-host" with no positive evidence, which is closer to an admission of the gap than a closure of it.
- Adds a host-platform branch to the test runner (a one-off, but the only such branch in `V8b`'s leaf — every other PIC-13 assertion is host-agnostic).

**Risks.**
- Contributors run only on Linux locally; the production-adapter half is never positively witnessed until a CI matrix or a Windows/macOS developer happens to run the suite — which may never occur in the loom 1.0 window.

### Option B — Scope `npm test` to the FakeFileSystem; delegate production-adapter half to the real-host smoke gate

**Approach.** Add a non-normative platform note to PIC-13 stating that the case-canonicalisation guarantee's production-adapter half is witnessed at the manual real-host smoke gate (`docs/plan_topics/real-host-smoke-gate.md`) rather than under `npm test`, and that the FakeFileSystem half (whose in-memory symlink table can simulate case-insensitive collapse on any host) remains under `npm test`. Add a real-host smoke criterion that the smoke run, when executed on a case-insensitive Pi host, exercises a fixture pair such as `Plan.loom` / `plan.loom` and verifies `PiFileSystem.realpath` returns byte-identical output. Update `V8b` / `V8b-T` **Tests** to assert the FakeFileSystem half and to reference the smoke-gate delegation for the production half, mirroring how `V4c` / `V9i` delegate the ERR-8 mid-stream-cancellation live-surface confirmation to the same smoke gate.

**Spec edits.**
- In PIC-13's `realpath` bullet, append a *Production-adapter platform note* clause: "*Non-normative — production-adapter platform note.* The production-adapter half of the case-canonicalisation guarantee (and of the byte-stable canonical-output guarantee, of which it is the case-insensitive-FS instance) is witnessable only on case-insensitive hosts; `npm test` asserts the algorithmic shape against the `FakeFileSystem` (whose in-memory symlink table simulates the case-insensitive-collapse on any host), and the production `PiFileSystem` half is witnessed at the manual real-host smoke gate (`docs/plan_topics/real-host-smoke-gate.md`) per its activation triggers."
- In `real-host-smoke-gate.md`'s Pass/fail criteria list, add criterion (f): "when the live Pi host's filesystem is case-insensitive, a `Plan.loom` / `plan.loom` fixture pair under a discovery root yields one byte-identical `realpath` output from `PiFileSystem.realpath`, witnessing the PIC-13 case-canonicalisation guarantee on the production adapter."
- In `V8b` / `V8b-T`, add a **Tests** bullet asserting the case-canonicalisation guarantee on `FakeFileSystem` (case-variant inputs resolve to one canonical path) and note in the leaf that the production-adapter half is delegated to the smoke gate per the platform note above.

**Pros.**
- Works on Linux-only CI without a permanently-skipped automated assertion.
- Uses the project's established convention for real-host-only behaviours (`V4c` / `V9i` already delegate ERR-8's live-surface confirmation to the same gate), so the spec gains no new mechanism.
- The FakeFileSystem assertion under `npm test` continues to cover algorithmic shape — DISC-3 and INV-1 consume the guarantee through the fake.

**Cons.**
- The production-adapter conformance becomes a manual gate, and the smoke gate is a contributor obligation rather than a CI-enforced check; an absentminded contributor on a case-sensitive Pi host can skip the criterion silently (recoverable only post-hoc by the H6a evidence audit).
- Delays detection of a bare-`fs.promises.realpath` regression in the production adapter from `npm test` to whichever smoke run next fires.

**Risks.**
- The smoke gate's two triggers (Pi version bump; merge touching the four fidelity-contract axes) do not include a PIC-13-adapter-only change — a refactor of `PiFileSystem.realpath` that drops the leaf-case canonicalisation step would not, by itself, fire the smoke gate, so the gap closes only at the next bump or fidelity-axis merge.

### Recommendation

Take **Option B**. It mirrors the convention this project already uses for real-host-only behaviours (`V4c` / `V9i` delegate ERR-8 the same way) so no new spec mechanism is invented, and it lets `npm test` produce a positive automated witness on the FakeFileSystem half — which is what every downstream consumer (DISC-3, INV-1) actually exercises through the seam — rather than a permanently-skipped Linux-CI placeholder. The contributor-obligation gap Option B leaves on the production adapter is real but bounded: the H6a release-gate auditor reads the smoke-run evidence from commit messages post-hoc, which is the same exposure ERR-8 already accepts. To narrow that gap, extend the smoke gate's trigger (2) ("a merge whose diff touches the four fidelity-contract axes") so that an edit to `PiFileSystem.realpath`'s case-canonicalisation step also fires the smoke; the implementer should write that trigger extension into the same edit to `real-host-smoke-gate.md` that adds criterion (f).

Implementer watch-points: (i) the `FakeFileSystem` test must drive a case-variant input pair through the in-memory symlink table, not bypass it — otherwise the assertion regresses to a no-op; (ii) the smoke-gate criterion (f) is conditional on the live host being case-insensitive — on a Linux Pi host the criterion is an *un-runnable-gate condition* (same family as the binder-model-missing case the gate already defines), not a failure; (iii) the spec edit must touch both the case-canonicalisation guarantee *and* the byte-stable canonical-output guarantee since the former is named the case-insensitive-FS instance of the latter, so the same testability protocol covers both.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one ships a production adapter that bare-delegates to `fs.promises.realpath` and silently violates the case-canonicalisation guarantee — and the violation is not caught by `npm test` on Linux CI, because the witnessing fixture can't be built there — while another writes a custom adapter canonicalising leaf case. The DISC-3 case-collision dedup paragraph and the INV-1 invocation-time containment re-check both rely on the production adapter actually honouring the guarantee, so a silent-non-conforming adapter produces wrong behaviour at first contact with a case-insensitive host (false-negative case-collision dedup; sibling-case-variant invoke paths that should canonicalise to one root-contained path but do not).
