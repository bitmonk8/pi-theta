## Finding

`V2d`'s heading is "Canonical integer/number renderer," but the first sentence of its **Adds** field names the deliverable "The canonical number-rendering pure function." The label "number-rendering" reads as scoped to the `number` primitive, while the leaf in fact renders both kinds: an `integer`-kind value as canonical base-10 (BNDR-4) and a `number`-kind value as shortest round-tripping fixed-point (BNDR-5), selected from a caller-supplied `integer`-vs-`number` discriminator.

The body of the Adds and the Tests bullet already disambiguate fully — both BNDR-4 (integer) and BNDR-5 (number) renderings are described and asserted — so no implementer can omit integer rendering. The residual is purely the first-sentence label drifting from the heading's "integer/number" wording. Two downstream consumers (`V5f` schema canonical-hash, `V11h` binder argument echo) refer to it as "the `V2d` renderer" / "canonical number renderer," so the umbrella term is in use elsewhere, but the leaf's own naming sentence should match its heading.

## Consequence

**Severity:** cosmetic

A reader skimming only the Adds first sentence might momentarily read the function as `number`-primitive-only, but the remainder of the Adds and the BNDR-4/BNDR-5 Tests make both modes explicit, so no implementer diverges and no coverage is lost. The defect is wording-only.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V2d-number-rendering.md`, change the **Adds** field's first sentence so its noun phrase matches the heading's "integer/number" wording — e.g. "The canonical integer/number rendering pure function shared by the schema canonical-hash recipe (`V5f`) and the binder argument echo (`V11h`)." No other field changes; the kind-discriminator description, BNDR-4/BNDR-5 split, and Tests already cover both modes. The consumer leaves (`V5f`, `V11h`) need no edit.
