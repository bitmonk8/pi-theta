## Problem

The `loom/load/invocation-cycle` Tests bullet on both `V15b-T` and `V15b` ends "an unresolvable callee is a leaf (undetected until fixed)". That trailing clause is behaviourally ambiguous: it reads either as a required negative assertion (a cycle routed through an unresolvable callee must NOT fire `loom/load/invocation-cycle`) or as an informational aside carrying no test obligation. `invocation.md` §Cycle detection pins the behaviour as an observable conformance point — a cycle closed only through an unresolvable callee emits `loom/load/callee-has-errors` and does not emit `loom/load/invocation-cycle` until the underlying file is fixed. Because `V15b-T`'s entire product is failing tests, the two readings yield divergent deliverables and the closing parity gate is blind to a missing negative assertion.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (2026-06-10) — "pi-loom plan: build/update plan for spec.md + review"
**History:** Both `V15b` and `V15b-T` were created in c6a664e, the initial plan-build commit, with the `loom/load/invocation-cycle` Tests bullet and the "an unresolvable callee is a leaf (undetected until fixed)" phrase authored verbatim (`git log -S "undetected until fixed"` and `git log -S "invocation-cycle"` against both files return only c6a664e). Subsequent commits to these files (084f171, e8c1d65, ac99f79, c82371b) touched only the INV-4 / NOCEIL frame-depth bullet, never the invocation-cycle bullet. The ambiguity is original to the leaf, not introduced by a later edit.

## Solution approach

Rewrite the `loom/load/invocation-cycle` Tests bullet on `V15b-T` so it states the unresolvable-callee behaviour as a test obligation rather than a parenthetical, splitting it into the positive assertion (a static-resolution cycle over resolvable callees fires `loom/load/invocation-cycle` at parse time) and the negative assertion (a cycle whose closing edge runs through an unresolvable callee — one that produced `loom/load/callee-has-errors` — does not fire `loom/load/invocation-cycle`), grounding both in `invocation.md` §Cycle detection. The negative-assertion corpus must close the cycle only through the unresolvable node, since a resolvable closing edge would exercise the positive path instead. Apply the same disambiguation to the matching bullet on `V15b`, and extend `V15b`'s **Ships when** line so the gate observes the negative unresolvable-callee assertion alongside the positive cycle.

## Solution constraints

- The spec is read-only for this fix; `invocation.md` §Cycle detection already pins the behaviour and is the grounding source, not an edit target.
