## Problem

The *Out of scope: `loom/typecheck/*`* paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` (anchor `loom-typecheck-namespace`) closes with an *Audit-gate carve-out* sub-clause imposing a normative MUST on build-time audit tooling: an audit gate scanning `loom/...` literals against the Code registry MUST treat `loom/typecheck/` as a prefix-disjoint namespace and MUST NOT report a `loom/typecheck/*` literal as a missing registry entry. The host file owns the runtime-observable diagnostic surface (envelope shape, delivery channels, the closed Code registry); the audit-gate reconciliation behaviour is a build-time surface owned by the SDK surface-inventory audit family in `pi-integration-contract`. The MUST is therefore stranded: a reader looking for diagnostic-shape rules does not expect it, and a reader implementing or reviewing the audit gate cannot find it via the audit-family pages, even though the closing-gate automation and the inventory-closure audit both depend on the carve-out.

## Solution approach

Move the *Audit-gate carve-out* MUST out of `diagnostic-shape.md`'s `loom-typecheck-namespace` paragraph and into the *Inventory-closure audit* section of `docs/spec_topics/pi-integration-contract/inventory-audit-intro.md` (anchor `sdk-cap-inventory-closure-audit`), binding it on the build-time audit that reconciles `loom/...` literals against the diagnostics Code registry, and add a stable anchor at the relocated obligation so dependent leaves can link to it directly. In `diagnostic-shape.md` replace the relocated MUST with a forward-link to the new anchor, retaining the in-scope namespace claim and the reader-orientation sentence about how to read `loom/typecheck/*` matches when auditing the `loom/...` surface.
