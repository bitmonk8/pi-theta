## Finding

`V13c` ("Query tool loop and typed two-phase") carries, in one leaf, two distinct deliverable classes. The first is the query tool loop itself: the free-phase `CIO-4` round-advancement mechanics, the typed two-phase `complete()` forced-respond turn, the `max_rounds:0` boundary, and the untyped-exhaustion `tool_loop_exhausted` path — all sourced from `query-tool-loop.md`, all consuming the same loop machinery. The second is a depth-6 ceiling-#4 co-fire live-carrier vector: a depth-6 typed-query response trips the loom-owned depth walk (`V5e`) before AJV, surfaces in loom code as `Err(QueryError { kind: "validation" })` with `schema_keyword: "maxDepth"`, while the co-satisfied ceiling #2 is enumerated as `masked: ["ceiling#2"]` on the operator-facing `RuntimeEvent`. That vector pulls in `V5e` (the depth walk), `V16a` (the cross-ceiling `masked` arbitration), and the `RuntimeEvent` shape owned by `V9d` (PIC-1) — subsystems unrelated to the loop's own round-advancement behaviour. A fifth thread, the `ERR-13` delegated completed-callee-finality live-carrier witness for `V4f`, rides along as well.

The plan already establishes a clear pattern for ceiling-#4 depth-6 live carriers: `V14e` ("Ceiling-#4 depth-6 code-driven-tool-args routing (live carrier)") and `V15j` ("Invoke ceiling-#4 depth-6 `params`/`invoke<T>`-return routing (live carrier)") are each their own dedicated leaf, depending on their feature-core leaf (`V14a`, `V15a`) plus `V5e`. `V5e`'s own per-boundary delegation note names all five routing rows and delegates each to its "site owner": `ValidationError` at `V13c`, `CodeToolError` at `V14e`, `InvokeInfraError` at `V15j`. Two of those three site owners are dedicated single-purpose live-carrier leaves; the typed-query-response row alone is folded inline into the much larger loop leaf. The decomposition is therefore applied inconsistently across structurally identical routing rows.

The remaining bundle is not a case where coupling precludes a split. A `V14e`/`V15j`-shaped extraction depends on `V13c` for the loop machinery exactly as `V14e` depends on `V14a` and `V15j` on `V15a`, so the established pattern transfers cleanly. (Note also that `V13c`'s depth-6 bullet asserts the `masked` enumeration on a `RuntimeEvent` whose shape is owned by `V9d`, yet `V13c` lists no `Deps` edge on `V9d` — a grounding gap the extracted live-carrier leaf should close by carrying that edge.)

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `c6a664e` (2026-06-10) "pi-loom plan: build/update plan for spec.md + review"
**History:** The `V13c` leaf was authored already bundling the query-tool-loop mechanics with the depth-6 ceiling co-fire vector in the initial plan build (`c6a664e`). The depth-6 vector's wording was later refined — `0dd2436` (2026-06-11, surfaced ceiling), `9d880b8` (before-AJV ordering), `f1ae6bc`/`a2a0ba8` (masked attributed to `RuntimeEvent` not `QueryError`) — but the bundling structure was never changed. A subsequent commit, `eaa5000` (2026-06-16) "resolve 'V13c bundles four independent spec obligations into one leaf'", de-bundled two *other* obligations (the parallel-batch concurrency rule → `V14b`, and the swallowing-handler attachment → `V13f`) out of `V13c`, but left the depth-6 ceiling-#4 live-carrier vector co-resident with the loop mechanics. The current bundle is the residue of that partial de-bundling.

## Solution Space

### Recommendation

Extract the depth-6 typed-query-response ceiling-#4 co-fire vector out of `V13c` into a dedicated live-carrier leaf, mirroring the `V14e`/`V15j` precedent, then register that leaf on the coverage-gating surfaces. Sequence the extraction first so the coverage-matrix and `H5b` Deps edits land on a baseline where the new leaf ID already exists and resolves.

**Extract the live-carrier leaf.**
- Create a new live-carrier leaf `<new>` and its paired `<new>-T` that own the depth-6 typed-query-response ceiling-#4 vector. Author `<new>-...-live-carrier.md` and `<new>-T-...-live-carrier.md` carrying that depth-6 typed-query-response vector verbatim, with `Deps` `<new>-T`, `V13c`, `V5e`, `V16a`, `V9d`. The `V13c` edge is required because the leaf builds on the loop's forced-respond/`max_rounds` machinery; the `V9d` edge is required because it grounds the `RuntimeEvent` `masked`-enumeration assertion that is currently ungrounded in `V13c`. Both edges must be present, or the leaf loses the context it needs.
- In `V13c-query-tool-loop.md` and `V13c-T-query-tool-loop.md`, strike the "typed depth-6 co-fire vector" Tests bullet (the `query-tool-loop.md` "Worked example: depth-6 forced respond at `max_rounds`" bullet). Drop the now-unused `V5e` entry from `V13c`'s and `V13c-T`'s `Deps`; retain `V16a` (still consulted for the round-boundary ceiling-#2 evaluation) and the `ERR-13`/`V4f` delegated-witness thread. Keep the loop mechanics, `max_rounds:0`, untyped exhaustion, and the `ERR-13` delegated witness on `V13c`.
- In `V5e-depth-enforcement.md` and `V5e-T-depth-enforcement.md`, retarget the per-boundary delegation note's "`ValidationError` at `V13c`" site-owner reference to `<new>`.
- In `plan.md`, add the new leaf's link under the V13 slice.

**Wire the leaf into coverage gating.**
- In `coverage-matrix.md`, add `` `<new>` `` to the `cka-12` (`query/` QRY) closing-leaf cell (or the cell the implementer determines owns this routing row), backtick-delimited per the *Closing-leaf column authoring* rule. Leave the `ERR-13` and `ERR-19` rows' existing `V13c` references in place — the `ERR-13` delegated witness stays on `V13c`. An un-backticked token tokenises to empty and the `H5d` gate misreads the cell.
- Add `` `<new>` `` to `H5b-warn-only-canary.md`'s `Deps` per the *Transitive-completeness plan-maintenance* clause, since the new leaf carries a citing test that closes a coverage-matrix-mapped obligation. Omitting it lets the canary pre-flight against incomplete coverage.

No spec edits are required.

## Consequence

**Severity:** advisory

A reviewer or CI run can only record that `V13c` as a whole passed or failed; the depth-6 ceiling-#4 co-fire vector's correctness cannot be isolated from the loop mechanics, so an authored-but-wrong ceiling vector can hide behind green loop tests. The structural inconsistency (two of three ceiling-#4 site owners are dedicated live-carrier leaves, one is folded inline) also obscures the `V9d`/`RuntimeEvent` dependency the inline vector silently relies on. The leaf remains implementable as written; the cost is review granularity and structural uniformity, not implementer divergence.
