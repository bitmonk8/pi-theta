## Finding

Two normative statements describe how a contributor resolves a family-(4) violation (an out-of-scope import/access shape), and they do not agree.

The *Failure-surface contract* family→step-2(b) routing table (`audit-failures.md`, `#audit-failure-surface-contract`) gives the family-(4) row three permitted arms: **branch (1) [delete]**, **branch (2) [promote]**, and **branch (5) [rewrite-shape]**, with an explicit exclusion that "branch (3) exempt MUST NOT appear here per the *Exemption mechanism* family (4) prohibition."

The *Exemption mechanism* prose (`audit-resolution.md`, `#audit-exemption-mechanism`) instead says: "a contributor confronted with such a shape MUST rewrite the source line into a recognised shape … **rather than annotate it, per step 2(b) branch (5)**." Read literally, this MUST narrows family-(4) resolution to a single arm — the branch-(5) rewrite — and does not accommodate the routing table's branch (1) [delete] (which removes the offending line outright, producing no recognised shape) or branch (2) [promote] (which adds an inventory entry, impossible for a *shape* until after a rewrite). The phrase is ambiguous between (a) "branch (5) is the only permitted non-exempt source change" (which contradicts the table) and (b) "make a source change rather than annotate it" (which is consistent with the table but is not what the sentence states).

The inconsistency compounds with branch (5)'s own definition. Branch (5) (`version-bump-step2b.md`, `#bump-step-2b-rewrite-shape`) says to rewrite the offending line into a recognised shape "and resolve the rewritten reference via **branch 1, 2, or 3** above as that branch's criterion fits." So after a branch-(5) rewrite, the rewritten reference may legitimately be exempted via branch (3). If the family-(4) row's branches (1) and (2) are meant to be the *terminal* arms reached through branch (5), then branch (3) should appear on the row too (branch (5) permits it); if instead branches (1) and (2) *directly* resolve a family-(4) red without a rewrite, then they are direct arms and branch (5)'s follow-on resolution is a separate matter. The spec mixes both models: it lists branches (1)/(2) on the row but asymmetrically excludes (3), and the prose names only branch (5). A contributor or reviewer cannot determine from the text whether a family-(4) line may be deleted directly (branch 1) or must first be rewritten (branch 5), nor why branch (2) is admitted on the family-(4) row while branch (3) is forbidden when both are only reachable after a rewrite.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `e5cd514` ("pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"", 2026-05-09)
**History:** The five-family *Failure-surface contract* and the five-branch step-2(b) resolution set were authored together in a single commit, `e5cd514`, which introduced the closed-set negative-direction audit. `git log -S 'family (5)'`, `git log -S 'branch (5)'`, and `git log -S 'Resolve by exactly one of the following five branches'` (all `--reverse`, over `docs/**`) each first appear in `e5cd514`; no predecessor revision carried either ordinal space. The overload was therefore present from the feature's inception — both 1–5 spaces were coined with the same `(N)` notation in the same change and never disambiguated — rather than drifting in via a later edit. The later `81bf52a` page-collapse and `23d28a1` basename realignment relocated the prose but did not introduce or alter the notation collision.

## Solution Space

### Recommendation

Define each family→step-2(b) routing-table row as the set of arms that directly resolve *that family's* red, and make the family-(4) arm set exactly two arms: **branch (1) [delete]** and **branch (5) [rewrite-shape]**.

In `audit-failures.md` (*Failure-surface contract*, `#audit-failure-surface-contract`), change the family-(4) row to list `branch (1) [delete], branch (5) [rewrite-shape]`. Drop branch (2) [promote] from the row: a *shape* cannot be promoted to inventory directly — promotion applies only to the recognised-shape reference that a branch-(5) rewrite produces, which is a separate family-(1)/(2)/(3) red resolved on its own row. Keep the existing branch-(3) exclusion note (branch (3) exempt never appears on the family-(4) row).

In `audit-resolution.md` (*Exemption mechanism*, `#audit-exemption-mechanism`), reword the MUST so it admits branch (1) and frames branch (5) as the rewrite path: a contributor confronted with a family-(4) shape MUST resolve it by a source change — delete the line (branch 1) or rewrite it into a recognised shape (branch 5) — rather than annotate it; branch (3) exempt never applies to a family-(4) line.

In `version-bump-step2b.md` (branch (5), `#bump-step-2b-rewrite-shape`), add one sentence clarifying that branch (5)'s follow-on branch-1/2/3 resolution resolves the *rewritten* reference's own red, recorded against that reference's family row, not the family-(4) row.

Edge case to preserve: a family-(4) shape that the runtime depends on cannot be deleted; it must go through branch (5), and the rewritten reference's promote/exempt is then recorded against its own family row, co-committed per the existing dual-emission/co-commit discipline. Re-read the land-green sweep wording in `V18b` ("promoted … or rewritten") so it does not imply direct promotion of a shape.

## Consequence

**Severity:** correctness

Two reasonable contributors fixing the same family-(4) violation would diverge: one reads the prose's MUST and rewrites the line into a recognised shape (branch 5); another reads the routing table and deletes the line outright (branch 1) or promotes after rewriting (branch 2). Reviewers checking branch-fit on the bump-commit diff, and implementers pinning the leaf-side family→step-2(b) routing table as a permanent test invariant, have no single authoritative arm set for family (4) — and the asymmetric exclusion of branch (3) (while branches 1/2 are admitted) cannot be justified from the text.
