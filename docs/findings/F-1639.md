## Problem

Two normative statements describe how a contributor resolves a family-(4) violation (an out-of-scope import/access shape) and they disagree. The family→step-2(b) routing table at `#audit-failure-surface-contract` (audit-failures.md) gives the family-(4) row three arms — branch (1) [delete], branch (2) [promote], branch (5) [rewrite-shape] — while explicitly excluding branch (3) [exempt]. The *Exemption mechanism* prose at `#audit-exemption-mechanism` (audit-resolution.md) instead states a contributor MUST rewrite the source line into a recognised shape "rather than annotate it, per branch (5)", which read literally narrows family-(4) resolution to a single arm and contradicts the table's branch (1)/(2) admission. A contributor or reviewer cannot determine the authoritative arm set, and the asymmetric admission of branch (2) [promote] while excluding branch (3) [exempt] — when both are only reachable after a branch-(5) rewrite — cannot be justified from the text.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `e5cd514` ("pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"", 2026-05-09)
**History:** The five-family *Failure-surface contract* and the five-branch step-2(b) resolution set were authored together in a single commit, `e5cd514`, which introduced the closed-set negative-direction audit. `git log -S 'family (5)'`, `git log -S 'branch (5)'`, and `git log -S 'Resolve by exactly one of the following five branches'` (all `--reverse`, over `docs/**`) each first appear in `e5cd514`; no predecessor revision carried either ordinal space. The overload was therefore present from the feature's inception — both 1–5 spaces were coined with the same `(N)` notation in the same change and never disambiguated — rather than drifting in via a later edit. The later `81bf52a` page-collapse and `23d28a1` basename realignment relocated the prose but did not introduce or alter the notation collision.

## Solution approach

Reconcile the family-(4) resolution arms across the `#audit-failure-surface-contract` routing-table row and the `#audit-exemption-mechanism` MUST so both name the same arm set. Resolve the underlying model ambiguity — whether branches (1) [delete] and (2) [promote] directly resolve a family-(4) red, or are reachable only after a branch-(5) rewrite — noting that a *shape* cannot be promoted to inventory directly and that branch (2)'s admission is currently asymmetric with branch (3)'s exclusion. Clarify at `#bump-step-2b-rewrite-shape` whether branch (5)'s follow-on branch-1/2/3 resolution attaches to the rewritten reference's own family row rather than the family-(4) row.

## Solution constraints

- Family-(4) MUST remain non-exemptible: branch (3) [exempt] MUST NOT be added to the family-(4) arm set (the absolute family-(4) prohibition in *Recognised import/access shapes* / *Exemption mechanism* stands).
- Out of scope: the *Failure-surface contract* section's placement and formal five-family establishment, owned by F-2066 — limit edits to the family-(4) routing-table row content.
