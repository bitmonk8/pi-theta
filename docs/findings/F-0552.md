## Finding

The rule-5 Tests bullet in `V11e` (and its paired `V11e-T`) reads: "the `ambiguous` failure row renders only the model's `message` and **never surfaces `candidates` in loom 1.0**". The absolute quantifier "never" and the version-scoped qualifier "in loom 1.0" are mutually inconsistent — "never" asserts a permanent prohibition, while "in loom 1.0" scopes the non-surfacing to the current release.

The spec's own rule 5 (`spec_topics/binder/defaulting-system-note-echo.md` §System-note rendering, rule 5: "**`ambiguous.candidates` is not rendered in loom 1.0.**") phrases the obligation consistently: "the loom 1.0 runtime **does not surface** it". The version scope is intentional and correct — the spec explicitly defers the rendering question (and array-truncation rules) to a documented *Future Considerations — Binder refinement loop*. The plan leaf should match the spec's "does not surface" phrasing while retaining the "in loom 1.0" scope; the deferral is owned in the spec, so no follow-up section is needed.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V11e-system-note-determinism.md` and `docs/plan_topics/V11e-T-system-note-determinism.md`, in the rule-5 Tests bullet, replace `never surfaces \`candidates\` in loom 1.0` with `does not surface \`candidates\` in loom 1.0`, matching the spec rule-5 phrasing ("the loom 1.0 runtime does not surface it"). Keep the "in loom 1.0" scope — it is the spec's own version qualifier and the deferred work is already owned in the spec's *Future Considerations — Binder refinement loop*. No new follow-up section is required.

## Consequence

**Severity:** cosmetic

The contradiction is recoverable against the spec's clear rule-5 wording, so no implementer would diverge on observable behaviour. The stray "never" only adds a moment of reader doubt about whether the prohibition is permanent or release-scoped.
