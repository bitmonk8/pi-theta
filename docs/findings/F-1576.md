## Finding

The `AgentSession` interface snapshot in `host-interfaces-core.md` declares the subagent-mode query driver as `sendUserMessage(text: string): Promise<void>`. The actual Pi-owned shape at the pinned SDK (`@earendil-works/pi-coding-agent ~0.75.5`, `dist/core/agent-session.d.ts:377`) is `sendUserMessage(content: string | (TextContent | ImageContent)[], options?: { deliverAs?: "steer" | "followUp" }): Promise<void>`. The snapshot drops the content-array arm and the entire `options` parameter.

This does not change loom's runtime behaviour: loom always passes the `string` arm and never supplies `options`, so the bare-string call remains type-compatible with the real signature. The defect is in the snapshot's accuracy, not the call site. The page itself frames the per-member TypeScript signatures as "a non-normative snapshot of the Pi-owned shape at the loom 1.0 Pi-SDK pin, re-validated on each Pi minor bump per the Pi version bump procedure" — i.e. the snapshot is a stated diff target. A baseline that already disagrees with the pinned SDK defeats that purpose: a future minor-bump diff either flags a spurious change or masks a real one, and the build-time surface-inventory assertion described elsewhere cannot be written against an inaccurate baseline.

## Issue introduction

**Verdict:** present-since-inception (of the snapshot block)
**Introducing commit:** `aa334c2` — "pi-loom spec: resolve "AgentSession consumed member surface not pinned"" (2026-06-05)
**History:** `git log -S` / `git log -L` on the `sendUserMessage(text: string)` line in `host-interfaces-core.md` return a single commit, `aa334c2`, which added the `AgentSession` member-surface snapshot. The line was authored with the simplified single-arg signature. At that same commit the Pi SDK pin (`host-prerequisites.md#pi-sdk-pin`) already read `~0.75.5`, whose `agent-session.d.ts` already declared the multi-arg `content` + `options` form — so the snapshot was inaccurate at the moment it was written, not a victim of later upstream drift. No subsequent commit touched the line.

## Solution Space

**Shape:** single

### Recommendation

Update the snapshot line to the pinned shape:

```ts
sendUserMessage(content: string | (TextContent | ImageContent)[], options?: { deliverAs?: "steer" | "followUp" }): Promise<void>;
```

Keep the existing trailing comment's intent (subagent-mode query driver; `Promise<void>` return unlike `pi.sendUserMessage`'s `void` shape), and note that loom uses only the `string` arm and never supplies `options`. Confirm `TextContent` / `ImageContent` are referenced consistently with how the page cites other Pi-owned types (these names already appear in the neighbouring `pi.sendUserMessage` form discussion / SDK snapshots), so the snapshot stays self-consistent.

## Consequence

**Severity:** advisory

The snapshot is explicitly a re-validation diff target; an inaccurate baseline undermines the Pi-minor-bump diff check and the build-time surface-inventory assertion built on it (spurious or masked diffs). No implementer-observable runtime behaviour diverges, because the only call site passes the `string` arm.
