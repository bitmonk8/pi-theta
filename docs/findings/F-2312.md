## Problem

`V2a` owns `array<T>` type-sink resolution over the exhaustive four-member sink set declared in `grammar.md` §`array<T>` literal type-sink rule (binding annotation, function-parameter type at a call site, surrounding constructor field, and recursive descent into an array-typed sink). The only sink-related vector on `V2a` / `V2a-T` is `loom/parse/array-no-common-type`, the negative no-sink reject path. Positive resolution from each of the four sinks emits no diagnostic, so no reverse-gate vector exercises it and no other leaf picks it up (`V2b`'s `TYPE-9` covers `⊑`-engine common-type compatibility, not literal sink resolution; the `cka-1` matrix row maps `grammar.md` to `V2a` at file granularity). An implementer can therefore satisfy `V2a`'s **Ships when** while implementing only the no-sink reject and never the positive resolution the spec marks as the substance of the rule.

**Verdict:** present-since-inception
**Introducing commits:** c6a664e — pi-loom plan: build/update plan for spec.md + review (2026-06-10, Thomas Andersen)
**History:** `V2a-type-grammar.md` and `V2a-T-type-grammar.md` were both first authored in c6a664e (they do not exist in its parent). From inception their Tests sections enumerated only `loom/parse/array-no-common-type` (the no-sink reject); no positive sink-resolution vector was ever present. The gap is coeval with the leaf, not introduced by any later edit.

## Solution approach

Add positive sink-resolution vectors to the paired `V2a-T` (tests) and `V2a` (implementation) Tests sections — one per member of the exhaustive sink set in `grammar.md` §`array<T>` literal type-sink rule, plus the recursive nested-array case. Since positive resolution emits no diagnostic, make each resolved element type observable rather than asserting an absence. Retain the existing `loom/parse/array-no-common-type` no-sink vector.

## Solution constraints

- Out of scope: the `V2a` parser-atomicity split owned by F-2374 — resolve only the missing positive vectors here.
- `grammar.md` and `coverage-matrix.md` are read-only for this fix — the four sinks are already enumerated and the `cka-1` row already lists `V2a`.
