## Finding

`H4b`'s first Tests bullet asserts that the response-programming surface "drives the double **deterministically** — replaying the same scripted assistant turns, `tool_use` results, binder responses/failures, `tool_loop.max_rounds` round-exhaustion, invoke-child completions, subagent-mode callee dispatch, and cancellation-injection points yields the same observable transcript on every run." That guarantee holds only if every non-deterministic host seam whose output can reach the observable transcript is wired as a deterministic double. Two such seams exist: `IdSource` (`newInvocationId()` UUIDs — the `invocationId` source coverage-matrix `cka-26` keys to `IdSource.newInvocationId()`), and `Clock` (any time-derived transcript content). Both are declared in `H3a`'s seam graph and implemented downstream in `V8d`.

The (f) invoke-child-completion and (g) subagent-mode-callee categories `H4b` introduces are exactly the paths that produce `invocationId`-bearing transcript content through the harness's own scripting contract rather than the live `V15a`/`V9i` surfaces, so a transcript carrying a real `IdSource`-minted UUID (or a real `Clock` timestamp) would differ run-to-run and fail the determinism self-check.

`H4b` never states this precondition. An implementer who wires the harness against a real `IdSource`/`Clock` would see the determinism self-check fail for a reason the leaf gives no language to diagnose; the leaf neither requires deterministic seam doubles nor scopes the asserted transcript to fields free of seam-derived non-determinism.

**Introducing commit:** `27e12be` — "pi-loom plan: resolve \"Session double's model / tool / binder-response scripting surface is undefined\"" (2026-06-10)
**History:** `git log -S "yields the same observable transcript on every run" --all --reverse -- docs/` identifies `27e12be` as the first commit to introduce the determinism self-check into the plan corpus; `git show 27e12be` confirms it appended the clause "the shared response-programming surface drives the double **deterministically** … yields the same observable transcript on every run" to the harness self-check in `plan_topics/H4a-factory-shell-and-harness.md`. That commit was itself the resolution of an earlier plan-review finding ("scripting surface is undefined") — the fix that added the scripting surface and its determinism self-check did not name the deterministic-seam precondition. A later edit `fd6169b` (2026-06-12, the H4a→H4a/H4b split) relocated the determinism clause into the new `H4b-response-programming-surface.md` as its own Tests bullet and broadened it (adding invoke-child completions and subagent-mode dispatch — the very `IdSource`-bearing paths), but preserved the missing precondition. The defect is original to the `27e12be` clause, not to plan inception, and has carried unchanged through the split.

## Solution Space

**Shape:** multiple

### Option A — name the deterministic-seam precondition

**Approach.** State in `H4b`'s determinism self-check Tests bullet that the guarantee presupposes the harness wires deterministic `Clock` and `IdSource` doubles (so `invocationId`s and time-derived content are reproducible across replays). The deterministic-seam wiring belongs to the harness, so the supporting note may land in `H4a`'s Adds (the harness / session-double fidelity contract) with `H4b` referencing it.

**Plan edits.** In `docs/plan_topics/H4b-response-programming-surface.md`, amend the determinism Tests bullet to record that "the same observable transcript" holds *given the harness's deterministic `Clock`/`IdSource` seam doubles* (the `cka-26` `invocationId` source and time-derived content being the seam outputs that would otherwise vary). Optionally add a one-clause note to `H4a`'s harness/fidelity-contract paragraph that the harness supplies deterministic `Clock`/`IdSource` doubles.

**Spec edits.** None.

**Pros.** Matches how harness doubles are normally built; the determinism guarantee stays unqualified over the whole transcript; a single concept ("deterministic seam doubles") covers both `IdSource` and `Clock` and any future seam.

**Cons.** Couples `H4b`'s self-check phrasing to a harness-wiring fact; if the note lands in `H4a` it touches a second leaf.

**Risks.** At `H4b` time the seam *implementations* (`V8d`) do not yet exist, so the harness must supply its own deterministic stand-ins for `Clock`/`IdSource` — the edit must make clear the doubles are the harness's, not a `V8d` dependency.

### Option B — scope the asserted transcript to seam-non-deterministic-free fields

**Approach.** Leave seam wiring unconstrained and instead constrain what the determinism self-check compares: the asserted "same observable transcript" excludes (or normalises) fields sourced from non-deterministic seams — `invocationId`s and any `Clock`-derived timestamps.

**Plan edits.** In `docs/plan_topics/H4b-response-programming-surface.md`, amend the determinism Tests bullet so the replay-equality assertion is taken over the transcript with seam-derived fields (`invocationId` per `cka-26`, time-derived content) excluded or normalised before comparison.

**Spec edits.** None.

**Pros.** Keeps the self-check robust regardless of which `Clock`/`IdSource` the harness happens to wire; no second-leaf edit.

**Cons.** Requires the leaf to enumerate (or characterise) which transcript fields are seam-derived, which can drift as new seam-sourced fields enter the transcript; weakens the guarantee to a subset of the transcript.

**Risks.** An un-enumerated future seam-derived field silently re-introduces the flakiness the exclusion was meant to remove.

### Recommendation

Take Option A: state the deterministic-`Clock`/`IdSource`-injection precondition on `H4b`'s determinism self-check bullet (with the harness-wiring note in `H4a` if the implementer locates the doubles there). It is the smaller, more durable edit — one concept covers every seam — and it aligns with the harness already being the source of all scripted inputs. Edge case the implementer must watch: the (f) invoke-child and (g) subagent-mode categories are the transcript paths that mint `invocationId`s via `IdSource`, so the deterministic `IdSource` double must be in force for those categories' self-checks specifically, not only for the plain scripted-turn path.

## Consequence

**Severity:** advisory

If the plan ships unfixed, two reasonable implementers diverge on harness construction: one wires deterministic `Clock`/`IdSource` doubles and the self-check passes; one wires real seams and the determinism self-check fails with no leaf-provided explanation, forcing a detour to discover that `invocationId`/timestamp leakage is the cause. The leaf still yields a working harness once the implementer infers the precondition, so the gap is guidance, not a blocker.
