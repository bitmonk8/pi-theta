## Finding

`conventions.md` uses two different names for the same concept — the non-`-T` half of a paired feature, i.e. the leaf whose ID is the bare `<id>`. The §Per-phase TDD ritual section calls it the **"Implementation task"** ("2. **Implementation task** — leaf ID `<id>` (e.g. `V4b`)"), and the §Leaf format **Deps** bullet reinforces that term ("An implementation task MUST list its paired `<id>-T` tests task here."). The coverage-matrix sub-paragraph immediately below that bullet instead writes **"implementation leaf"**: "In the **coverage matrix**, each spec REQ-ID maps to the **implementation** leaf (`<id>`) that closes it …".

Both phrasings denote the identical `<id>` artifact, but no equivalence gloss ties them together. A reader consulting both sections cannot be certain "implementation leaf" and "implementation task" are the same thing rather than two distinct plan concepts.

## Issue introduction

**Verdict:** present-since-feature-introduction
**Introducing commits:** `f9c5354` ("pi-loom plan: add scope section + two-task TDD pairing convention", 2026-06-09), `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** The two-task pairing nomenclature — "tests task" / "implementation task" — entered `conventions.md` at `f9c5354`, the commit that introduced the TDD pairing convention; the same commit also added the "implementation leaf" phrasing to `conventions.md` §Leaf format. The matching "implementation leaf" wording in `coverage-matrix.md`'s intro was set the following day at `c6a664e`. Both surface terms therefore co-arrived with the pairing convention itself; the dual naming has been present since that feature entered the corpus, not introduced by a later divergent edit.


## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, §Leaf format, edit the coverage-matrix sub-paragraph that follows the **Deps** bullet so it uses the term the surrounding text already uses ("implementation task"). Change the phrase `the **implementation** leaf (`<id>`)` to `the **implementation task** (`<id>`)`, leaving the rest of the sentence ("that closes it (its green tests are the closure evidence); the paired `<id>-T` tests task is not listed separately") intact. This makes the section name the `<id>` half consistently with both the §Per-phase TDD ritual ("Implementation task") and the preceding Deps-bullet sentence ("An implementation task MUST list its paired `<id>-T` tests task here").

## Consequence

**Severity:** cosmetic

A reader reconciling the §Per-phase TDD ritual and §Leaf format sections sees two names for the same `<id>` half with no stated equivalence; the wording wobble adds momentary ambiguity but does not change any implementer's behaviour or leaf outcome.
