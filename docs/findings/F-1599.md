## Problem

The *Per-family record-shape table* on `audit-failures.md` defines the family-(4) `symbol` field as "the verbatim source text of the offending expression or import declaration", but the *Recognised import/access shapes* paragraph routes roughly eighteen distinct shapes to family (4), most of which are neither expressions nor import declarations — parameter declarations, destructured-parameter declarations, type-parameter declarations, `extends` / `implements` / `&` subtype-creation declarations, and non-parameter carrier-annotated bindings. For these shapes the field description names a string the offending site does not contain. Two conforming implementers diverge on what to emit, and the leaf-side per-`<symptom>` fixture floor bakes whichever choice the first implementer made into a permanent test invariant.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `e5cd514` — *pi-loom spec: resolve "T35 SDK capability inventory closed-set negative-direction audit"* (2026-05-09)
**History:** `git log -S "expression or import declaration" -- docs/` resolves the phrase to a single introducing commit, `e5cd514`, which is also the commit that first authored the *Per-family record-shape table* and the *Recognised import/access shapes* paragraph (then on the monolithic `pi-integration-contract.md`, later split into the current per-topic files by the sibling-basename realignment in `23d28a1`). The same commit's recognised-shapes paragraph already enumerates the non-expression, non-import shapes — `git show e5cd514` shows "any function/method parameter declaration", "any class field declaration", and "any TypeScript declaration anywhere … that creates a subtype" all present at introduction. The narrow description and the broader shape set therefore shipped together in the audit's introducing commit; the defect is not the residue of a later widening of the shape set against a description that was once correct.

## Solution approach

Rewrite the family-(4) `symbol` field description in the *Per-family record-shape table* so it names the full syntactic-category set the *Recognised import/access shapes* paragraph routes to family (4), not only expressions and import declarations. Keep the single-line normalisation (whitespace-collapse) rule on the broadened description.

- Leave the `audit/<class>/<family>/<symptom>` discriminator and the four-field tab-delimited record shape pinned by *Wire serialisation* unchanged.
