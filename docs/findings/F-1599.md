## Finding

The *Per-family record-shape table* on `audit-failures.md` defines the family-(4) `symbol` field as "the verbatim source text of the offending **expression or import declaration**, normalised to its single-line form…". The companion *Recognised import/access shapes* paragraph on `audit-recognised-shapes.md` enumerates roughly eighteen distinct prohibited shapes that all route to family (4), of which only a minority — `import *`, default imports, dynamic `import()`, side-effect-only imports, `pi[name]`, `ctx[name]`, value-binding aliases, captured rebindings, `Object.assign({}, pi)`, `keyof typeof pi`, barrel re-exports — actually are expressions or import declarations in the TypeScript grammar. The remainder are **declarations of other syntactic categories**: off-canonical-name / off-canonical-annotation **parameter declarations** (e.g. a `request: ExtensionContext` parameter, a `ctx: any` parameter); destructured-binding parameter declarations; type-parameter declarations whose constraint is a carrier literal (e.g. `<C extends ExtensionContext>`); **subtype-creation declarations** in `extends` / `implements` / `&` position (e.g. `interface MyCtx extends ExtensionContext`); and non-parameter **bindings** annotated with a carrier literal (class fields, top-level `const`/`let`/`var`, class-property initialisers).

For every one of these non-expression, non-import shapes the table tells the implementer to emit "the verbatim source text of the offending expression or import declaration" — a string the offending site does not contain. Two conforming implementers diverge on what to put in the `symbol` field (the whole parameter declaration? just the annotation? the whole `interface` header? the class-field declarator? something synthesised?), and the leaf-side fixture floor that asserts the record shape on stdout (per the *Failure-surface contract*'s per-`<symptom>` rule, exercised in `V18b` / `V18b-T`) bakes whichever choice the first implementer made into a permanent test invariant that may not match the next implementer's reading.

The fix is mechanical: broaden the field description to cover every syntactic category the recognised-shapes paragraph already routes to family (4), and pin the single-line normalisation rule on that broader description.

## Solution Space

**Shape:** single

### Recommendation

Edit the family-(4) row of the *Per-family record-shape table* on `docs/spec_topics/pi-integration-contract/audit-failures.md` so the `symbol` field description names the full syntactic-category set the *Recognised import/access shapes* paragraph routes to family (4), rather than only expressions and import declarations. Concretely, replace "the verbatim source text of the offending expression or import declaration, normalised to its single-line form…" with a description of the form: "the verbatim source text of the **offending syntactic construct** — for the shapes the *Recognised import/access shapes* paragraph routes to family (4), this is the smallest enclosing syntactic node that fully contains the offending shape (the import declaration for an import-shape violation; the expression for an access / rebinding violation; the function/method parameter declaration for a parameter-name / parameter-annotation / destructured-parameter violation; the type-parameter declaration for a generic-laundering violation; the `extends` / `implements` / `&` type-reference for a subtype-creation violation; the binding declaration for a non-parameter `ExtensionAPI` / `ExtensionContext` / `ExtensionCommandContext` annotation), normalised to its single-line form (every run of one or more whitespace characters, including newlines, collapsed to a single space) so the record fits the one-record-per-line wire format below."

Implementer-relevant edge cases:

- Pin the smallest-enclosing-syntactic-node rule rather than "the whole declaration", so an off-canonical-name parameter inside a multi-parameter function declaration surfaces the parameter declaration alone (not the surrounding function signature).
- Keep the single-line normalisation rule (whitespace collapse) on the broader description; the *Wire serialisation* paragraph already relies on it to guarantee tab-freedom for the family-(4) `symbol` value.
- For the subtype-creation shape, the field naturally surfaces the `extends` / `implements` clause or the `&` operand, not the entire `interface` / `class` / `type` declaration body — the *prohibition is on the subtype-creation declarations themselves* clause in *Recognised import/access shapes* already pins the declaration as the unique detection site, so the field needs to identify that site verbatim without including the whole body.
- The broadening is editorial-only on the audit's wire contract: the leading `audit/<class>/<family>/<symptom>` discriminator and the four-field tab-delimited record shape pinned by *Wire serialisation* are unchanged.

## Consequence

**Severity:** correctness

Two reasonable implementers — one reading the table row literally, one reading the recognised-shapes paragraph — produce divergent `symbol` field values for the same parameter-declaration, class-field, or subtype-creation violation, and the family-(4) discriminator's machine-parseable wire shape stops being byte-comparable across implementations. CI parsers that join the audit's stdout records against an external manifest by `(discriminator, symbol)` quietly misroute or fail to deduplicate, and the leaf's per-`<symptom>` fixture invariant bakes whichever interpretation the first implementer chose.
