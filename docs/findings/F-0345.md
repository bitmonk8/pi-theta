## Problem

`H7a`'s **Adds** prose says each committed golden turn and each `loom/...` code is "sourced from that leaf's already-established expected output." That phrase has two incompatible referents: the normative expected output each Deps leaf's **Tests** bullets assert, versus the observed output the leaf's passing implementation in fact emits. They diverge for any Deps leaf whose Tests assert behaviour abstractly rather than pinning exact output. If an author derives a golden from observed implementation output, a bug the leaf's coarse assertion did not catch is baked into the golden as the regression target, and `H7a` then mechanically certifies the buggy composition as correct.

A later commit, `0904502` ("golden-derivation methodology and human-review obligation parked in a Tests bullet"), relocated the clause from the Tests bullet into the **Adds** prose but preserved the "already-established expected output" wording unchanged, so the ambiguity has been present since the clause first entered the corpus and survived the relocation untouched.

## Solution approach

In `H7a`'s **Adds** paragraph, rewrite the golden-source phrase so it pins the normative referent: name the expected output each Deps leaf's **Tests** bullets assert as the per-leaf golden source, not whatever a passing implementation emits. For a Deps leaf whose Tests assert behaviour only abstractly, direct the golden value to the spec text that leaf cites (the source its Tests trace to) rather than to a pipeline run.
