## Finding

The statement-form `if` / `for` / `while` paragraph in `docs/spec_topics/grammar.md` (the prose immediately following the `BlockExpr` / `FnBody` / `LoomBody` / `StmtBlock` productions) carries a parenthetical that is meant to generalise the tail-`?` early-return carve-out from "the `@…?` form" to any operand admitted by ERR-18. The current wording reads:

> "…except that a tail `Expr` ending in the postfix `?` operator — applied to any operand `?` admits, per [Errors and Results — `?` operand-type precondition](./errors-and-results/error-model.md#err-18) — still triggers `?` early-return on failure…"

The fragment "applied to any operand `?` admits" inverts subject and verb. The intended noun phrase is "any operand that `?` admits" (i.e. any operand satisfying ERR-18's `Result<_, QueryError>` precondition), but as written, "any operand" appears to be the subject of the verb "admits", which produces no grammatical reading. An implementer reading the appendix in isolation has to chase the ERR-18 link to recover the meaning; the prose itself does not carry it.

The defect is purely surface; the underlying rule (the StmtBlock-tail-`?` carve-out applies to every `?`-tailed operand ERR-18 admits, not only `@…`-headed ones) is correct and unchanged.

## Issue introduction

**Verdict:** introduced by a specific commit
**Introducing commits:** `4e656ac` (2026-06-20) — `pi-loom spec-plan: fix F-0707 — StmtBlock tail ? carve-out generalised to ERR-18 operands`
**History:** The carve-out paragraph was first added by `816852b` (2026-06-05) using the narrower phrasing "a tail `Expr` of the `@…?` form still triggers `?` early-return on failure". Commit `4e656ac` rewrote that clause to generalise the carve-out beyond `@`-queries to every ERR-18-admitted operand; in doing so it replaced "the `@…?` form" with the parenthetical "ending in the postfix `?` operator — applied to any operand `?` admits, per [Errors and Results — `?` operand-type precondition] —". The garbled subject–verb order ("any operand `?` admits" instead of "any operand that `?` admits") is exactly the edit `4e656ac` introduced. No subsequent commit has touched this sentence (`190f10e` added the `FnDecl` block below; `8ff5f11` rewrote the unrelated newline-continuation row).

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/grammar.md`, in the statement-form `if` / `for` / `while` paragraph under `## Block expressions`, restore an unambiguous relative clause inside the parenthetical. Replace

> applied to any operand `?` admits, per [Errors and Results — `?` operand-type precondition](./errors-and-results/error-model.md#err-18)

with

> applied to an operand satisfying `?`'s operand-type precondition (any operand of static type `Result<_, QueryError>`, per [Errors and Results — `?` operand-type precondition](./errors-and-results/error-model.md#err-18))

Keep the surrounding sentence (the "still triggers `?` early-return on failure, because `?` desugars to `return Err(e)`…" clause) and the link target unchanged — the fix is confined to the parenthetical. The inlined "any operand of static type `Result<_, QueryError>`" gloss is preferred over the bare "satisfying `?`'s operand-type precondition" wording from the original suggestion because it lets the appendix carry the rule's substance without forcing the reader to follow the ERR-18 link to know what is admitted; the link remains for the full normative statement.

## Consequence

**Severity:** cosmetic

A reader who follows the ERR-18 link recovers the intended meaning, and the productions plus the cross-reference jointly pin the behaviour, so no implementer would diverge on what `?` does in a `StmtBlock` tail. The cost is purely readability: the appendix's single most subtle carve-out reads as broken English at first pass.
