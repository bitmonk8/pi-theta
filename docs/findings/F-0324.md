## Problem

The manual real-host smoke governance note (`real-host-smoke-gate.md`), `H6a`'s **Ships when** gate (`H6a-live-corpus-activation.md`), and the version-bump checklist (`V18c-version-bump-checklist.md`) all make the triggering change's commit message the sole durable evidence record, audited after the fact by reading that message on `main` — with no separate artefact file. None states the repository's merge strategy, so the convention silently assumes the commit carrying the evidence is the same commit an auditor later reads on `main`. Under a squash- or rebase-merge, individual commit messages are rewritten or discarded, so the evidence can fail to survive onto `main` and the after-the-fact auditor — and `H6a`'s release-blocking Ships-when gate — can find no record even though the run was performed and recorded. The spec already resolves this exact ambiguity for GOV-1 / GOV-22 by naming the first-parent diff as authoritative for merge or squash commits, but the commit-message evidence convention never adopts it.

The merge-persistence assumption entered with that home-choosing edit: the commit selected the commit message as the evidence store without addressing whether the message survives the merge workflow, inheriting the same unaddressed assumption from the spec convention it copied.

## Solution approach

Make the commit-message evidence requirement name which commit on `main` is authoritative, aligning it with the existing GOV-1 / GOV-22 first-parent-diff semantics so the "read the commit message" audit step is unambiguous under squash- or rebase-merges. Pin the requirement once at the spec version-bump convention that `real-host-smoke-gate.md`, `H6a`, and `V18c` mirror, keeping those leaves as inheritors. If the spec is read-only for this fix, instead pin the requirement in the plan and surface the resulting plan/spec divergence as a follow-up for spec review.
