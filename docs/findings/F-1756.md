## Finding

`docs/spec_topics/pi-integration-contract/capability-probe.md` introduces the
bundled-peer carve-out under the bolded sub-heading **"Deployment-shape
precondition."** The body immediately under that heading says step 0 (d)
"*presupposes* each of the four `@earendil-works/*` peers is installed as a
discrete, on-disk package" and that a bundled / vendored build "does not
satisfy this precondition and is **outside** the version-read contract" — i.e.
the runtime does not enforce it as a separate probe check; bundled builds
"observably present as condition (1) or (2) below rather than as a distinct
route." There is no `kind` value, no `details.*` payload, and no `MUST` /
short-circuit obligation attached to the heading itself.

That framing is the spec-wide definition of a **presupposition**, the
term-of-art `docs/spec_topics/pi-integration-contract/host-prerequisites.md`
establishes and re-uses for every other unpinned Pi behavioural assumption
(degraded-state clauses (b)–(d), binder-model / `model:` load-time resolution,
settings write-back key preservation, operator-always-present,
`pi.registerFlag('loom', …)`, `pi.getCommands()` completeness,
`pi.registerTool` post-startup acceptance and persistence, `AgentMessage[]`
chronological ordering and leading-`user`-message guarantee, slash-handler
promise lifecycle — every one bolded as `**… presupposition.**`, often with a
matching `#…-presupposition` anchor). Step 0 itself reserves the word
**precondition** for the three *enforced* host preconditions the probe runs in
fixed order (a) → (b) → (c) and short-circuits on, each routing to its own
`kind` discriminator under `loom/load/host-incompatible`. The Step 0 (d)
sub-heading is therefore the only place in the PIC corpus that uses
"precondition" for a clause whose body describes a presupposition, and it does
so directly inside the section that fixes the enforced sense of the word.

**History:** `git log -L '/Deployment-shape/,+1:docs/spec_topics/pi-integration-contract/capability-probe.md'` shows the bolded sub-heading **"Deployment-shape precondition."** was added in commit `d8d5ad0` as part of a wider rewrite that introduced the bundled-build carve-out paragraph and split the recommended `createRequire` recipe into its own callout. Prior to that commit the paragraph did not exist; no earlier revision used a different header for the same content. The mismatch entered the corpus together with the paragraph itself — author used "precondition" in the heading and "presupposes" / "outside the version-read contract" in the body in a single edit, while the parallel `**… presupposition.**` convention in `host-prerequisites.md` already predated this commit. No subsequent commit has touched the heading.

## Solution Space

**Shape:** single

### Recommendation

Rename the bolded sub-heading from **"Deployment-shape precondition."** to
**"Deployment-shape presupposition."** in
`docs/spec_topics/pi-integration-contract/capability-probe.md` (around
line 51, opening the bundled-peer carve-out paragraph). In the same
paragraph, change the in-prose phrase "does not satisfy this
**precondition**" to "does not satisfy this **presupposition**" so the
body matches the heading. Leave the rest of the paragraph — the
"presupposes …" verb, the "outside the version-read contract"
clause, and the "observably presents as condition (1) or (2) below"
routing — unchanged; that is already the correct presupposition framing
and the renaming is what aligns the label to it.

Edge cases the implementer must watch:

- No anchor (`<a id="…">`) currently targets this heading; `grep -rn
  'deployment-shape'` across `docs/` should return zero hits before and
  after the edit. If a later commit has added one, rename it to
  `#deployment-shape-presupposition` and sweep cross-references.
- Do **not** touch the top-of-Step-0 phrasing "synchronous probe of three
  host **preconditions**" — that use of "precondition" is correct (the
  three enforced checks (a) → (b) → (c)) and is the very contrast this
  finding turns on.
- Do **not** promote the carve-out into the `host-prerequisites.md`
  presupposition list or give it a `SHOULD re-confirm` checklist item;
  that is a scope expansion the body deliberately avoids ("outside the
  version-read contract"). The edit is purely a heading + one-word
  in-prose rename.

## Consequence

**Severity:** advisory

A reader scanning Step 0 by bolded sub-heading sees "precondition" and
reasonably infers an enforced check on equal footing with steps (a)–(c) —
i.e. a fourth `kind` (or a sibling diagnostic) for bundled / vendored
builds — when the body in fact treats the bundled-build case as silently
re-routed onto conditions (1) or (2). The same lexical slip on the
opposite side — calling a real precondition a presupposition — would
silently demote an enforced check. No diagnostic shape or runtime
behaviour changes, but the heading misleads against a vocabulary the
neighbouring PIC page has spent considerable surface fixing.
