## Finding

The closing-gate machinery has no arm that verifies per-facet citing-test coverage for a multi-leaf coverage-matrix row. The two arms that touch a multi-leaf row are both weaker than the obligation those rows carry: the `H5a` per-REQ-ID citing-test scan certifies only that *a* citing test exists somewhere in the corpus for the whole REQ-ID, and the `H5d` transitive-completeness arm is an explicit **per-cell at-least-one** check that reddens only when *none* of a cell's listed leaves reaches `H5b`'s `Deps.`. Neither arm inspects whether each individual facet-closer in a split row carries its own facet-naming citing test.

This is not a corner case the matrix avoids. Genuine multi-facet split rows are pervasive — `ERR-17 | V4d, V13d`; `DISC-4 | V10a (collision-detection closure), V9m (superseded-entry-dispatch closure)`; and the five-way code-keyed row `cka-33 | V17a, V14f, V13f, V15h, V9o`, where each leaf closes a distinct per-site swallowing-handler facet. Under the current gates, such a row can ship with one facet fully tested and a sibling facet — including failure-path / cancellation-suppression facets such as `cka-33`'s subagent-`AgentSession.abort()` suppression at `V9o` or the `@`-query provider routing at `V13f` — carrying no citing test at all, with `H5a`, `H5d`, and (for code-keyed rows) `H5e` all green. `conventions.md` already documents this gap and names it the *No silent test skipping* failure mode the plan forbids.

The only backstop is the manual *Loom 1.0 release-time residue inspection* item 7 (split-row joint-completeness). Item 7 distinguishes co-witness from facet annotations and is well-specified as a reviewer procedure, but it is a human read with no mechanical enforcement and no reproducible per-row record, so for split rows the *No silent test skipping* rule the plan declares as a hard invariant is in practice enforced only by reviewer diligence.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 953e3fa — pi-loom plan: resolve "closing gate checks numbered-REQ-ID matrix mapping, not test assertion" (2026-06-10, Thomas Andersen); cda161b — pi-loom plan: resolve "split REQ-IDs pass closing gate on one facet citing test" (2026-06-17, Thomas Andersen)
**History:** The citing-test reconciliation arm was added to `conventions.md` by 953e3fa (2026-06-10) as an existence-only, per-REQ-ID scan, so it never gated per-facet completeness of multi-leaf rows. The resulting split-row gap was explicitly recognised by cda161b (2026-06-17), which introduced the split-row joint-completeness wording and routed the residue to manual item 7 rather than mechanising it; later commits (2ad25ab, 685e45b, cf1bb3e) refined item 7 (per-cell at-least-one phrasing, co-witness vs facet distinction, code-keyed-table sweep) but kept the backstop manual. The mechanical gap therefore dates to the gate's existence-only design and has been documented-but-unmechanised since.

## Solution Space

**Shape:** multiple

### Option A — Mechanical per-facet citing-test gate arm

- **Approach:** Add a closing-gate arm that, for every multi-leaf row in both coverage-matrix tables, derives the facet partition exactly as item 7 already specifies (drop each leaf whose immediately-following parenthetical annotation contains the literal `co-witness` token; keep every other listed leaf as a facet) and asserts each remaining facet leaf carries a citing test naming that facet. Mirror the existing seeded-fixture / live-corpus footing posture so it binds at the `H6a` activation, like the other citing-test surfaces.
- **Plan edits:** Add the arm and its seeded violation/no-violation fixtures to the citing-test owner `H5a-closing-gate-automation.md` (the per-REQ-ID citing-test scan) or the facet-partition owner `H5d-transitive-completeness-gate.md`; record the new arm in `conventions.md` §REQ-ID discipline; establish in `coverage-matrix.md` a machine-parseable facet-annotation convention so each facet is nameable and associable to a citing test.
- **Spec edits:** None.
- **Pros:** Closes the gap in CI; brings split rows up to the same enforcement as the rest of the closing gate; removes reliance on reviewer diligence for the *No silent test skipping* invariant.
- **Cons:** Facet annotations are currently free prose, so a parseable convention must be authored first; the facet→citing-test association is heuristic (the test must name the facet); larger build surface than the manual route.
- **Risks:** Spurious reds if facet annotations are not normalised; relies on the citing-test scan corpus glob being pinned (see Related Findings).

### Option B — Reproducible per-row item-7 checklist

- **Approach:** Keep item 7 manual but make it reproducible: give it a concrete per-row procedure that enumerates each multi-leaf row, derives its facet partition by the item-7 rule, and for each non-co-witness facet leaf confirms a facet-naming citing test exists, recording the per-row result so a later reviewer can reproduce the verdict.
- **Plan edits:** Expand `conventions.md` §Loom 1.0 release-time residue inspection item 7 with the per-row procedure; optionally tighten the `coverage-matrix.md` facet-annotation prose so each facet is named.
- **Spec edits:** None.
- **Pros:** Low build cost; no new gate machinery; reproducible across reviewers.
- **Cons:** Stays manual — a reviewer can still skip it; not enforced in CI; can drift from the matrix.
- **Risks:** Checklist/matrix divergence over time.

### Recommendation

Adopt Option A — the mechanical per-facet arm — as the durable fix, because the manual route has already been taken once (cda161b) and the *No silent test skipping* rule the plan declares as a hard invariant should not rest on reviewer diligence for the pervasive split-row case. If Option A is deferred, Option B must land as the interim, and the facet-annotation convention it shares with Option A should be authored first so the later mechanical arm reads a stable, named facet partition. Watch the code-keyed rows (e.g. `cka-33`) specifically: they carry no diagnostic code, so the diagnostic-code↔asserting-test parity gate gives them no incidental coverage, and a facet there is the most likely to ship untested.

## Consequence

**Severity:** correctness

A secondary facet of a split coverage-matrix row can reach the loom 1.0 release with no test exercising it, while `H5a`, `H5d`, and `H5e` all report green — the closing gate certifies coverage it has not actually verified per facet. Because the exposed facets include cancellation-suppression and failure-path behaviours (e.g. `cka-33`'s per-site swallowing-handler closers), a silently-untested facet is a class of defect the gate exists to prevent, and the *No silent test skipping* invariant the plan declares holds only as far as a human performs item 7.
