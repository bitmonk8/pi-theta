## Problem

The prompt-mode `sessionManager` row in the *Per-mode override semantics* table of `host-interfaces-core.md` describes the replacement as "Pi's user-session manager (the host's, observed via the captured `ExtensionCommandContext`)". This admits two incompatible readings: an identity-for-symmetry reading where the runtime forwards whatever the captured handle currently resolves to, and a capture-time-pin reading where the runtime snapshots the reference at slash-handler entry and re-installs it for the invocation's lifetime. The two are observationally identical in loom 1.0 â€” the *Mid-loom user-session replacement* seam rules out any in-flight swap â€” but they prescribe materially different runtime structures and diverge the moment that seam is widened. The sibling `pi`-reference capture obligation in the same contract is already pinned as a present-tense MUST, making this row the local outlier.
