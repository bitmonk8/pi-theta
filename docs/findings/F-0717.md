## Finding

`host-prerequisites.md` §`pi-sdk-pin` (item 1) asserts that "the lock-step minor-version line is inherited from `@earendil-works/pi-coding-agent`'s own `dependencies` block, which pins `@earendil-works/pi-agent-core`, `@earendil-works/pi-ai`, and `@earendil-works/pi-tui` at the same `~X.Y.Z` minor-version line." The upstream manifest at the [loom 1.0 Pi-SDK pin](#pi-sdk-pin) does not do this: in `@earendil-works/pi-coding-agent@0.75.5`'s installed `package.json`, the three siblings are each pinned with **caret** (`^0.75.5`), not tilde. The two ranges coincide today only because Pi is in `0.x`, where npm caret degrades to "lock to the same minor"; the inherited "same minor-version line" claim becomes false the moment any of the four packages crosses `1.0.0`, at which point `^1.Y.Z` admits the entire `1.x` major and the alleged inheritance silently widens loom's effective skew window.

Loom's own `package.json#peerDependencies` correctly pin all four `@earendil-works/pi-*` packages at `~0.75.5` (witnessed by the H1a four-peer shared-line architectural test in `H1a-scaffold-and-toolchain.md`), so the operative minor-lock-step guarantee for loom consumers is loom's own tilde range, not inheritance from upstream. The spec needs to (a) correct the upstream description to caret and (b) re-anchor the minor-lock-step guarantee on loom's own tilde `peerDependencies` rather than on a non-existent upstream tilde pin.

## Issue introduction

**Verdict:** specific-commit
**Introducing commit:** `c307a62` ("pi-loom spec: resolve \"^X.Y.Z prose label minor-version line only coincides with npm caret semantics while Pi is in 0.x\"", 2026-05-07)
**History:** Before `c307a62`, the same paragraph correctly described upstream as pinning the three siblings at `^X.Y.Z` (matching `@earendil-works/pi-coding-agent`'s `dependencies` block, which has used caret throughout). `c307a62` flipped loom's own `peerDependencies` from caret to tilde — the correct edit, motivated by npm caret no longer coinciding with "lock to minor" once Pi crosses `1.0.0` — and in the same diff rewrote the upstream-inheritance description from `^X.Y.Z` to `~X.Y.Z`, projecting the loom-side change onto the upstream description rather than re-inspecting upstream. The subsequent pin moves (`f985a17` → `~0.74.1`, `ae529db` → `~0.75.5`, `5e73072` scope rename to `@earendil-works`) and the May-2026 spec split (`81bf52a`) carried the defective `~X.Y.Z` text forward verbatim; none of them touched the inheritance claim.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the affected sentence in `host-prerequisites.md` §`pi-sdk-pin` to:

1. State upstream's actual pin shape: `@earendil-works/pi-coding-agent`'s `dependencies` block pins `@earendil-works/pi-agent-core`, `@earendil-works/pi-ai`, and `@earendil-works/pi-tui` at `^X.Y.Z` (caret), not `~X.Y.Z`.
2. Drop the load-bearing word "inherited" from the lock-step claim and re-anchor that claim on loom's own `peerDependencies` block, which pins the four `@earendil-works/pi-*` entries at `~X.Y.Z` (the literal already asserted by the H1a four-peer shared-line architectural test). Phrase the upstream block as a *concordant* observation ("at the loom 1.0 Pi-SDK pin, where Pi is in `0.x`, the upstream caret and loom's tilde coincide on the same minor-version line; the guarantee loom enforces is the tilde literal in its own `peerDependencies`, not derivation from the upstream caret"), so the prose remains accurate after a future Pi `1.x` release widens upstream's caret semantics without loosening loom's tilde.
3. Keep the existing "released together from `pi-mono`; no skew supported" and "build-time `peerDependencies` literal-read assertion is the gate" sentences as-is — they survive the rewording unchanged. Edge case the implementer must watch: the `typebox` `"*"` carve-out paragraph below the affected sentence currently relies on the surrounding "four-entry tilde-pinned" framing to distinguish itself from the `@earendil-works/*` four; that framing must be preserved or restated when the inheritance sentence is rewritten.

## Consequence

**Severity:** correctness

A reader who takes the inheritance clause at face value will believe loom's minor-lock-step guarantee is structurally enforced by Pi's monorepo discipline; the actual upstream uses caret, so the guarantee survives a Pi `1.0` only because of loom's own `peerDependencies` tilde — a brittle premise to leave miscited. Implementers cross-checking the spec against `@earendil-works/pi-coding-agent`'s real manifest will find the two disagree and have to decide which to trust.
