## Finding

`docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md` closes the per-mode wiring / consequences section with a free-standing sentence (line 31):

> No Pi-side change is required for loom 1.0; if Pi later adds `pi.unregisterTool`, the registration cache becomes the natural site to call it from on extension teardown without altering the per-mode wiring rule above.

Both halves of the sentence are non-normative future-work: the first half restates a fact the same page already pins authoritatively at line 3 ("Pi's tool registry has no `unregisterTool` API — `pi.registerTool` is the only mutation, and entries persist for the process lifetime …" anchored as the [registry-persistence presupposition](docs/spec_topics/pi-integration-contract/host-prerequisites.md)); the second half speculates about how a hypothetical future Pi API would integrate. Neither half generates an implementer obligation for loom 1.0, and the sentence sits unflagged among the bullet list of normative consequences that follows PIC-19 — between the "another extension calls `pi.setActiveTools`" rule (a MUST-shaped behavioural commitment) and the `Acceptance criteria — PIC-17 active-set install vector` heading.

`docs/spec_topics/future-considerations/surface-extensions.md` already owns the project's deferral inventory and contains no entry for an `unregisterTool` cleanup pass — the aside is the only place in the spec where the hypothesised API is mentioned. Leaving it inline both inflates the page (the spec set is under a 100KB hard / 30KB recommended cap per the `f5e89f4` commit) and invites a future reader to mistake "natural site to call it from" for a soft commitment the runtime must keep the cache wired for.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 27ec53d (`Resolve Cluster-A: per-mode tool registration`, 2026-05-04)
**History:** `git log -S "if Pi later adds"` shows exactly three commits touching the string: `27ec53d` introduces the sentence on the original `docs/spec.md`; `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") shards the spec and carries the sentence verbatim into `docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md`; `7dd8b27` is a review pass that does not modify it. The defect therefore entered with the Cluster-A resolution and has been present in this page since the page itself was created.

## Solution Space

**Shape:** single

### Recommendation

Delete the sentence outright from `docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md`. Do **not** relocate it to `docs/spec_topics/future-considerations/surface-extensions.md` — the substantive "no `unregisterTool` API" claim is already pinned at line 3 of the same page (the registry-persistence presupposition), and a future Pi minor that introduces `pi.unregisterTool` would be picked up under the existing `Pi version bump procedure` page; the spec gains nothing by pre-committing to a particular wire-up site for an API that does not exist. The deletion is a single-paragraph drop with no anchor consumers (no `<a id>` is attached to the sentence and no other spec page links to it — verified by `grep -rn "if Pi later adds" docs/` matching only this one site).

Edge cases for the fixer:
- The blank lines bracketing the sentence (line 30 trailing blank, line 32 leading blank) should collapse so the `Acceptance criteria — PIC-17 active-set install vector` heading is preceded by exactly one blank line, matching the rest of the page's heading spacing.
- Do not edit the line 3 registry-persistence presupposition while doing this; its wording is load-bearing for [Host prerequisites](docs/spec_topics/pi-integration-contract/host-prerequisites.md) and for plan leaf `V9f`'s **Adds** clause.

## Consequence

**Severity:** cosmetic

The sentence carries no normative content (its substantive claim is already pinned at the page's opening registry-persistence presupposition) and gates no behaviour. The cost is editorial: a future-work aside in the normative consequences list dilutes the section's signal-to-noise ratio and risks being misread as a soft loom 1.0 obligation to keep the cache structured for an unregister pass.
