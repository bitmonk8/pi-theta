## Finding

`V11a`'s **Tests** list five red-first bullets that `V11a-T` authors: the `loom/load/binder-model-unresolved` firing, the three-valued `strictCapable` probe, the binder-model **two-step fallback to `looms.binderModel`**, the **hot-reload recovery note**, and the single-matcher cross-resolution reconciliation. `V11a`'s **Ships when** gate, however, names only three of these: "`npm test` asserts the resolution chain, the three strict-capability outcomes, and that `V6a`'s `loom/load/model-unresolved` resolution and this leaf's binder-model resolution bind the single matcher instance `V9b` constructs." The two-step `looms.binderModel` fallback and the hot-reload recovery-note emission are absent from the completion gate.

Both omitted behaviours are normative spec obligations, not incidental detail. `spec_topics/binder/binder-model-and-context.md#binder-model` mandates the two-step chain (`bind_model:` frontmatter, then the `looms.binderModel` extension setting, with **no further fallback**); the `V11a` Tests bullet specifically asserts the merged-settings value reaches the binder from `V10c`'s merged settings rather than a hardcoded model — a cross-leaf settings-propagation behaviour. `binder-model-and-context.md#binder-model-hot-reload` mandates a single consolidated `loom-system-note` when a settings change lets a previously-unresolved binder model re-resolve; the Tests bullet asserts that emission. Both are exactly the kind of non-trivial behaviour an exit gate should re-assert.

Per `conventions.md` §Leaf format, a leaf's binding obligations are its **Tests** REQ-ID bullets *and* its **Ships when** gate, and the §Per-phase TDD ritual exit gate is the "Ships when criterion observable" check at `<id>-complete` tagging. With these two bullets present in Tests but absent from Ships-when, the observable acceptance condition under-specifies the leaf relative to the behaviours it claims to implement: the gate can read green while re-asserting only three of the five tested behaviours.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V11a-binder-model-resolution.md`, extend the **Ships when** field so the observable gate also asserts the two behaviours its **Tests** already red out:

1. The binder-model two-step fallback to `looms.binderModel` — with merged settings supplying `looms.binderModel` and a non-bypass loom whose frontmatter `bind_model:` is omitted, the merged value resolves the binder model (no `loom/load/binder-model-unresolved`), and the value is sourced from `V10c`'s merged settings rather than a hardcoded model.
2. The hot-reload recovery-note emission — a hot reload that lets a previously binder-model-unresolved loom re-resolve emits the consolidated `loom-system-note` per `binder-model-and-context.md#binder-model-hot-reload`.

The existing three clauses (resolution chain, three strict-capability outcomes, single-matcher instance identity) stay. The edit is confined to `V11a`'s Ships-when field; the **Tests** list already carries both behaviours, and no `Deps` change is implied by this finding. The spec is read-only here — both obligations already exist in `binder-model-and-context.md`; do not amend the spec.

## Consequence

**Severity:** advisory

The two omitted behaviours remain bound by `V11a`'s red Tests bullets, so a conforming implementer following the TDD ritual still turns them green before tagging. But the exit gate — the externally observable acceptance condition a reviewer reads at `<id>-complete` — does not independently re-assert the cross-leaf `looms.binderModel` settings-propagation path or the consolidated hot-reload recovery note, so a thin or regressed implementation of either could pass the stated gate without the omission being visible at review time.
