## Finding

The `Binder envelope schema (constructed dynamically from `params:`)` section of `docs/spec_topics/binder/binder-bypass-and-envelope.md` presents the envelope as a single top-level `{"anyOf": [...]}` JSON object with three arms; the `ok` arm's `args` slot is the placeholder `<params-schema-with-defaulted-fields-relaxed>`. The prose immediately below the skeleton states two normative facts: (1) the `args`-arm fragment "may carry `$ref`s into the loom file's `$defs`", and (2) "the envelope schema document handed to the provider (and to AJV) carries the transitive `$defs` closure of the params schema, computed by the same per-query pruning rule as [Schema Subset step 4](../schema-subset.md#lowering-algorithm)."

The skeleton, however, shows no `$defs` member anywhere and never pins where the closure attaches. Schema-Subset step 3 emits `$ref`s as `{ "$ref": "#/$defs/<Name>" }` — root-anchored JSON Pointers that resolve against the document root — so the closure can only live at the envelope-document root. Nothing in the section says that. An implementer reading the skeleton in isolation can plausibly inline `$defs` inside the `args` arm (next to `type`, `properties`, etc.), at which point `#/$defs/Author` no longer resolves to the inlined block and every named-schema reference in the params shape becomes a dangling `$ref` at validation time.

The defect is structural, not editorial: the skeleton is the only place the envelope's document shape is shown, and it omits a load-bearing member the surrounding prose presupposes.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `816c398` (2026-05-04) — *pi-loom spec: resolve "Binder envelope schema is hand-crafted, off-subset, and fails its own AJV safety net"*
**History:** Commit `816c398` rewrote the envelope skeleton to conform to the Schema Subset (adding per-arm `type: "object"` and `additionalProperties: false`, retyping the `candidates` union, and adding the `<params-schema-...-relaxed>` invariants paragraph). The same commit appended the new paragraph stating that "the envelope schema document … carries the transitive `$defs` closure of the params schema, computed by the same per-query pruning rule as Schema Subset step 4" without adding a `$defs` member to the skeleton above it or pinning its attachment site. `git log -G '\$defs'` and `git log -S 'transitive \`$defs\` closure'` across `spec_topics/binder.md` and (post-split) `docs/spec_topics/binder/binder-bypass-and-envelope.md` show no other touch to the `$defs` wording; the later file-split commit `f5e89f4` (2026-06-04) carried the defect forward unchanged when the binder shard was moved into the subdirectory layout.

## Solution Space

**Shape:** single

### Recommendation

Update the skeleton in `### Binder envelope schema (constructed dynamically from params:)` so the envelope-document root, not just the three-arm `anyOf`, is what is shown:

```json
{
  "$defs": <transitive-$defs-closure-of-params-schema>,
  "anyOf": [
    { ... ok arm ... },
    { ... needs_info arm ... },
    { ... ambiguous arm ... }
  ]
}
```

Then immediately after the skeleton (folding into the existing closure paragraph), pin the attachment site and the resolution rule with one sentence: *"`$defs` is a member of the envelope-document root, sibling to `anyOf`; the `args`-arm fragment's `#/$defs/<Name>` references resolve against this root unchanged, exactly as in any Schema-Subset document."*

Edge cases the implementer must watch:

- When the params schema is a primitive (single-string bypass aside, this can still apply to e.g. `params: { count: integer }`), the transitive closure is empty. The skeleton should make clear that the `$defs` member is present only when the closure is non-empty; otherwise it is omitted entirely (matching the per-query pruning rule in Schema Subset step 4).
- The closure is computed against the **lowered** params schema, not the relaxed copy. The `<params-schema-with-defaulted-fields-relaxed>` transformation only edits `required`; it does not introduce or remove `$ref`s, so the closure computed against the lowered original is the correct envelope-level `$defs`.

## Consequence

**Severity:** correctness

Two implementers building the envelope document from this section will diverge on `$defs` placement. The implementation that nests `$defs` inside the `args` arm will ship envelope schemas whose `args`-arm `#/$defs/<Name>` `$ref`s do not resolve; depending on the provider and AJV configuration this surfaces as either schema-compilation errors at load time or silent acceptance of malformed binder outputs (because the unresolved `$ref` matches nothing). The defect is invisible to any loom whose `params:` references no named schemas, so it can ship and only fire when an author first uses a typed `params` field.
