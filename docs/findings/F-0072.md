## Finding

`conventions.md` §Per-phase TDD ritual instructs the tests task (`<id>-T`) to add "no production code beyond the minimum needed for the tests to compile and *fail red for the intended reason*", and the "Fail red for the intended reason" definition tells the implementer to "declare or stub the seam so the suite compiles." Both halves bracket the *outcome* — item 1 forbids a compile-time red, item 2 forbids a test going green on a premature working body — but neither states *how* the stub is to be constructed. The demarcation between "minimum stub needed to compile" and "implementation that belongs in `<id>`" is left to per-leaf judgement.

Two implementers authoring the same `-T` task can therefore diverge: one declares each seam's type signature with a throwing body so every test reds on its behavioural assertion; another writes a partial working body (turning some tests green prematurely, which only fails the gate after the fact via item 2) or under-declares the seam (hitting a compile-red and failing the gate via item 1). The fail-red gate catches both failure modes when they occur, but the absence of a construction rule means the gate is reached by trial and error rather than by following a stated convention. The same "fail red for the intended reason" passage was revised once to disambiguate compile-red from behavioural-red without ever pinning the stub body shape.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md` §Per-phase TDD ritual, add the stub-construction convention to the "It compiles and runs" clause (item 1 of the "Fail red for the intended reason" definition). State that the minimum production code the tests task may add is the seam's declared type signature plus a body that throws an explicit not-implemented error (e.g. `throw new Error('not implemented')`) — never a body that returns a working result. This makes the demarcation concrete: the type signature and throwing body are the "minimum stub" the tests task supplies; any body that satisfies a behavioural assertion is "implementation belonging in `<id>`" and must wait for the implementation task. A throwing body keeps each test reding on its own primary assertion (item 2) rather than passing prematurely, and keeps the suite compiling (item 1).

Edge case the implementer must watch: tests that assert a seam *throws* a specific error type cannot be satisfied by a bare `Error('not implemented')` stub — those tests still red because the thrown type or message does not match the asserted behaviour, which is the intended behavioural red, not a stub defect.

## Consequence

**Severity:** advisory

Without a stated stub convention, implementers reach the `<id>-T-complete` gate by trial and error: a partial working body silently turns some tests green (caught only after the fact by the fail-red definition), and under-stubbing causes a compile-red, both costing rework. Implementers can still converge on a working `-T` leaf, so this is a guidance gap rather than a blocker.
