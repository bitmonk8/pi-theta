## Problem

The second Tests bullet of both `V13a` and its paired tests leaf `V13a-T` cites `interpolated-result` and `discarded-query-result` as bare suffixes. In the spec these are fully-namespaced parse-phase diagnostic codes — `loom/parse/interpolated-result` and `loom/parse/discarded-query-result` — registered in `docs/spec_topics/diagnostics/code-registry-parse.md`. `conventions.md` §"Diagnostic message anchors" requires a test asserting a diagnostic to cite the full diagnostic code and treat the registry as the single source of truth. The bare suffixes force an implementer to reconstruct the namespace from outside the leaf, which is exactly the reconstruction the convention exists to remove.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10) — "pi-loom plan: build/update plan for spec.md + review"
**History:** The corpus is git-tracked. `git log --follow` on `V13a-query-render.md` shows three commits; `git log -S 'interpolated-result'` and `git log -S 'discarded-query-result'` over both `V13a-query-render.md` and `V13a-T-query-render.md` each return only `c6a664e`, the initial plan-build commit. The two leaf files were authored with the bare-suffix citations from inception and the Tests bullet has not been reworded since.

## Solution approach

In the second Tests bullet of both `V13a` and `V13a-T`, replace the bare token `interpolated-result` with `loom/parse/interpolated-result` and the bare token `discarded-query-result` with `loom/parse/discarded-query-result`, matching the registry rows in `docs/spec_topics/diagnostics/code-registry-parse.md`.
