## Finding

The paragraph beginning `***loom 1.0.0 rejected-candidate record (closed).***` on `docs/spec_topics/hard-ceilings/ceiling-invariants-and-audit.md` enumerates three candidate ceilings (regex backtracking, JSON Schema width pathologies, pattern-match recursion depth) that were considered and rejected during loom 1.0.0 authoring. It is explicitly self-labelled `(closed)` and "a closed record of loom 1.0.0 worked examples", and it explicitly instructs future spec-editors **not** to append to it ("future spec-editors performing the four-axis check supply their own per-edit rationale rather than appending to this loom 1.0.0 worked-examples sub-list").

The paragraph imposes no MUST, no SHOULD, and no obligation that constrains any current implementer, parser, runtime, or editor behaviour. Every fact it carries is already pinned at the page it forward-links to: the no-regex stance is normative at [Expressions — Built-in methods and properties](../expressions.md#built-in-methods-and-properties); the JSON-Schema-width keyword exclusions are normative under *Explicitly not supported* in the [Schema Subset](../schema-subset.md) intro; the `NOCEIL-3` / `NOCEIL-4` fall-through for unbounded recursion is normative in the same file three paragraphs above. The paragraph adds noise to a section already dense with normative routing-obligation prose (the surrounding *Audit methodology* / *Four-axis closure list* / *Routing obligation for future ceilings* / *Release gate* blocks), and a reader scanning the page for live constraints must parse it only to discover it is decision-log carry-over.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `2ff03c2` (2026-05-10) — *pi-loom spec: resolve "Source-language stability bullet leaves equivalence undefined at the aggregator and silently contradicts the open-set ceiling clause"*
**History:** `git log -S "candidate ceiling cases considered and rejected"` and `git log -S "V1.0 rejected-candidate"` both name `2ff03c2` as the first commit to introduce the paragraph string. That commit added the entire *Audit methodology* / *Four-axis closure list* / *Routing obligation* / *V1.0 rejected-candidate record* / *Release gate* block to `docs/spec_topics/hard-ceilings.md` in one edit (`git show 2ff03c2 -- docs/spec_topics/hard-ceilings.md` shows the paragraph added as one of the new `***...***` blocks in a `+23 / -2` diff). One later commit moved it without re-authoring its content: `f5e89f4` (2026-06-04, *docs: enforce 100KB hard / 30KB recommended size cap on spec set*) split `hard-ceilings.md` into the index page plus the two children `ceilings-3-and-4.md` and `ceiling-invariants-and-audit.md`, relocated the paragraph wholesale into `hard-ceilings/ceiling-invariants-and-audit.md`, and performed the `V1.0` → `loom 1.0.0` rename pass. No subsequent commit edits the paragraph body. The defect is therefore a property of the introducing block as authored at `2ff03c2`, not of any later evolution: the "closed worked examples" framing and the explicit "do not append" instruction were present in the original prose.

## Solution Space

**Shape:** single

### Recommendation

Delete the `***loom 1.0.0 rejected-candidate record (closed).***` paragraph from `docs/spec_topics/hard-ceilings/ceiling-invariants-and-audit.md` outright. The paragraph contains no normative content the spec corpus relies on; the no-regex, JSON-Schema-width, and pattern-match-recursion facts are already owned by the pages it forward-links to.

The corpus has no `decisions/` or `adr/` directory, so an "ADR move" is not on the table; the paragraph is not load-bearing enough to justify creating one. The deletion is safe to perform as a single-commit, single-file edit: no anchor, no `<a id>`, and no cross-reference targets the paragraph (`grep -rn "rejected-candidate\|worked-examples sub-list" docs/ --include="*.md"` returns only the paragraph itself), so no link-repointing is required. The surrounding *Audit methodology* paragraph, *Four-axis closure list*, *Routing obligation for future ceilings* paragraph, *Release gate* paragraph, *Five-site list co-edit obligation* paragraph, and all `NOCEIL-N` / `CIO-N` markers and anchors are untouched.

## Consequence

**Severity:** cosmetic

No conforming implementation diverges either way. The paragraph is dead weight on an already-dense normative page: a small reading-cost tax on every reviewer who scans the section, and a minor decision-log-on-spec-page hygiene defect of the kind the spec corpus polices elsewhere (e.g. the `(formerly GOV-14)` parenthetical, the `Rationale:` inline clauses, and the *V1 / loom 1.0* alias drift findings in the same review).
