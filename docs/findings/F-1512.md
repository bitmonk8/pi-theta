## Finding

The Indexed-access bullet in [`expressions.md` §Supported forms](./docs/spec_topics/expressions.md) pins the index-type rule only on the **object** branch — a non-`string` index on an object receiver (e.g. `obj[0]`) is rejected at parse time as `loom/parse/non-string-object-index`. It says nothing about the required static type of the index expression on the **array** branch. The receiver-side rule (`loom/parse/non-indexable-receiver` for non-array, non-object receivers) settles only what may appear on the left of `[...]`, not what may appear inside it.

For an array receiver, the bullet shows literal forms `a["b"]`, `a[0]`, `a[i]` — across the three the index types are `string`, `integer`, and the inferred type of `i` — without saying which the parser must accept. The downstream runtime rule (`loom/runtime/index-out-of-bounds` for `i < 0` or `i >= arr.length`) presupposes a numeric index but is not itself a static-type rule and does not foreclose static admission of `arr["x"]`, `arr[3.5]`, or `arr[true]`. Two implementers will diverge on the diagnostic (and on whether one fires at all) for each of those forms; nothing in `expressions.md`, the receiver bullet, the `array<T>` stdlib table, or `code-registry-parse.md` decides it.

The asymmetry is also a code-registry hole: the parse-rule table carries `non-string-object-index` for the object branch but no array-branch counterpart, so a fixture asserting that `arr["x"]` is rejected at parse time has no code to match against.

## Issue introduction

**Verdict:** single-commit introduction (asymmetric extension)
**Introducing commit:** `d6f17ca` — *"pi-loom spec: resolve 'indexed-access static semantics and runtime disposition'"* (2026-06-07).
**History:** The Indexed-access bullet first acquired any typed rule in `01ba226` (2026-06-04), which added only the receiver-side rule `loom/parse/non-indexable-receiver` — settling that strings are not indexable but pinning no index-type constraint on either branch. `d6f17ca` then extended the bullet with the object-receiver branch end-to-end (parse rule `loom/parse/non-string-object-index`, result-type rule, runtime-panic crossreference, read-order pin) and added the matching row to `code-registry-parse.md`, but did not author the symmetric array-receiver branch. The asymmetry is local to that commit's diff: prior to `d6f17ca` neither branch was pinned, so the hole is the omission of the array half of the bilateral fix `d6f17ca`'s commit title implies. `git log -S 'non-string-object-index'` confirms no later commit closes the array side.

## Solution Space

**Shape:** single

### Recommendation

Extend the Indexed-access bullet in [`expressions.md` §Supported forms](./docs/spec_topics/expressions.md) with an array-receiver counterpart, parallel in shape to the existing object-receiver clause, and add the matching row to the `loom/parse/*` table in [`code-registry-parse.md`](./docs/spec_topics/diagnostics/code-registry-parse.md).

Concrete edits:

1. **`expressions.md` Indexed-access bullet — add an array-branch clause** immediately after the existing object-branch clause, of the shape:

   > For an **array** receiver the index expression must be of type `integer`; a non-`integer` index (e.g. `arr["x"]`, `arr[3.5]`, `arr[true]`) is rejected at parse time as `loom/parse/non-integer-array-index`. The `integer ⊑ number` widening ([TYPE-2](./type-system.md#type-2)) is one-way, so an index whose static type is `number` is rejected as `loom/parse/integer-narrowing` on the same footing as any other `number → integer` narrowing site (see [Lexical — Number literals](./lexical.md)) rather than under this rule. The static result type of `arr[i]` is the array's element type `T`. At runtime an index outside `0..arr.length` panics with `loom/runtime/index-out-of-bounds`; the read order is the parse-time index-type check first, then the runtime out-of-bounds panic — symmetric with the object branch's parse-then-runtime ordering above.

   Pinning the required type as `integer` (rather than `number`) keeps the rule consistent with the array's discrete-index semantics, routes `arr[3.5]` to a parse-time reject via the `integer`/`number` distinction the type system already pins, and re-uses the existing `loom/parse/integer-narrowing` code for the `number`-as-index case rather than inventing a second one for it.

2. **`code-registry-parse.md` `loom/parse/*` table — add one row** immediately after the existing `loom/parse/non-string-object-index` row:

   ```
   | `loom/parse/non-integer-array-index` | E | type | Indexed access `arr[k]` on an `array<T>` receiver whose index expression `k` is not of type `integer` (e.g. `arr["x"]`, `arr[true]`; a `number`-typed index is `loom/parse/integer-narrowing`, not this code). | [Expressions — Supported forms](../expressions.md) | Array index must be an `integer`. | `array index must be integer; got <type>` |
   ```

Edge cases the implementer must watch:

- An `integer` literal that exceeds JS's safe-integer range is governed by the existing `lexical.md` literal rules, not by this code.
- A `number`-typed index (e.g. `let i: number = 0; arr[i]`) routes to `loom/parse/integer-narrowing`, **not** to `non-integer-array-index` — the rule explicitly defers to the existing narrowing code so the two codes do not double-fire on the same site.
- Negative `integer` literals (`arr[-1]`) are admitted at parse time and route to the runtime `loom/runtime/index-out-of-bounds` per the existing runtime rule; the array-branch parse rule pins index *type*, not index *sign*.
- `V3a-T` and `V3a` must be extended with fixtures for each of `arr["x"]`, `arr[3.5]` (rejected as `integer-narrowing`), `arr[true]`, and a `number`-typed-variable index (also `integer-narrowing`), so the new code's coverage is asserted alongside the object-branch's existing coverage.

## Consequence

**Severity:** correctness

Without the array-side rule, two implementers will disagree on whether `arr[3.5]`, `arr["x"]`, and `arr[true]` are parse-time rejections, runtime panics, or silently coerced; conformance tests cannot be written against an unnamed diagnostic. The hole is symmetric with the object-branch rule that already exists, so the divergence shows up immediately the first time a fixture exercises a non-conforming array index.
