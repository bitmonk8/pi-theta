## Finding

PIC-22's normative obligation — "when the model emits N (for any fixed N ≥ 2) subagent-mode `.loom` callables as parallel tool calls in a single assistant turn, the runtime MUST initiate `createAgentSession` for all N invocations before any one of the N returns" — is immediately followed, in the same sentence run, by a verification recipe: "A conformance test witnesses the obligation with a fake `AgentSession` whose `sendUserMessage` blocks until explicitly released: a conformant runtime has created all N sessions and entered each session's `sendUserMessage` before any blocked call is released, whereas a runtime that admitted invocations behind a cap, queue, or scheduler would leave at least one session uncreated."

The recipe is test-construction guidance (use a fake `AgentSession`, block `sendUserMessage`, observe entry before release), not a runtime obligation. Splicing it into the requirement body conflates *what the runtime must do* with *how a test author observes it*, lengthening the MUST a reader must parse to extract the actual constraint and blurring the line between the verification-layer artefact and the behavioural contract. The observable rule (all N `createAgentSession` initiated before any returns) is self-standing without the witnessing apparatus.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `7afe9bb` — "pi-loom spec: resolve \"no-invocation-cap MUST NOT is unobservable\"" (2026-05-31)
**History:** `git log -G "conformance test witnesses"` and `git log -S "blocks until explicitly released"` identify `7afe9bb` as the commit that first added the parallel-spawn rule together with its inline fake-`AgentSession` witnessing recipe — the recipe was introduced precisely *to* make the previously-unobservable no-invocation-cap MUST NOT testable, and was placed directly inside the requirement body at that point. Later edits preserved the embedding rather than introducing it: `7a3b907` ("subagent parallel-initiation MUST has no closing leaf", 2026-06-10) lifted the rule out of the **Concurrent-invocation isolation** paragraph into its own `<a id="pic-22">` anchor, carrying the recipe verbatim, and `f566349` (F-1224) narrowed the bound from N ≥ 3 to N ≥ 2 without touching the recipe's placement. `f5e89f4` only relocated the text during the monolith split.

## Solution Space

**Shape:** single

### Recommendation

In `subagent.md`, keep the PIC-22 body as the behavioural rule only: the runtime MUST initiate `createAgentSession` for all N invocations (N ≥ 2) before any one of the N returns, observable at the parallel-tool-call surface. Excise the "A conformance test witnesses the obligation with a fake `AgentSession`…" sentence from the MUST and relocate it to a clearly labelled verification-layer block immediately after the rule (e.g. a "Test approach" / "Conformance" note), preserving its content verbatim: a fake `AgentSession` whose `sendUserMessage` blocks until explicitly released, with a conformant runtime having created all N sessions and entered each session's `sendUserMessage` before any blocked call is released, and a capped/queued/scheduled runtime leaving at least one session uncreated. Tag the block non-normative (it describes how to witness the rule, not an additional obligation). Leave the existing `tool-calls.md#concurrency` cross-reference and the resource-ownership prose untouched here.

## Consequence

**Severity:** cosmetic

The behavioural requirement and the conformance instructions are both fully present; only their organisation is at issue. A reader extracting the runtime obligation must skip past the test-construction sentence, and the verification artefact is not visibly separated from the contract, but no implementer would build a different runtime as a result.
