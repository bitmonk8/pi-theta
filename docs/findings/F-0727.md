## Finding

`lexical.md` §Encoding pins three observables that can only be produced from raw bytes that have not yet been UTF-8-decoded: (i) detection of a non-UTF-8 BOM and of arbitrary invalid UTF-8 (including lone surrogates) as `loom/load/invalid-encoding`; (ii) the diagnostic's `<offset>` placeholder, which is the **zero-based byte offset of the first invalid byte in the original (pre-normalisation) file content**, populated by "the loom-side UTF-8 decode step" (the wording in `diagnostics/code-registry-load.md`); and (iii) the explicit fail-fast on a Notepad-style UTF-16 save (no transcoding). All three presuppose that the runtime obtains the source file as a raw byte sequence and runs its own UTF-8 validator over it.

The only spec-pinned seam through which the runtime reaches the filesystem is `FileSystem` in `pi-integration-contract/host-interfaces-services.md` (PIC-13). Its read members are **text-shaped**: `readText(path: string): Promise<string>`. There is no `readBytes` (or equivalent) member, and no normative paragraph anywhere in the spec corpus states that `.loom` / `.warp` sources are read as bytes before decoding. With only `readText` in scope, the runtime has no specified mechanism for observing invalid bytes at all — the Node `TextDecoder` underlying any conforming `readText` implementation has already either thrown, replaced, or passed through the bytes by the time loom code sees the string, and the byte offsets demanded by the diagnostic are no longer recoverable.

The defect is the missing connective tissue between `lexical.md` §Encoding (which says what loom MUST report) and PIC-13 (which says how loom MUST read the filesystem). Without it, two conforming implementations would diverge on every invalid-encoding case: one might delegate to Node's `fs.readFile(path, "utf8")` and never fire `loom/load/invalid-encoding` for a lone surrogate (Node silently substitutes U+FFFD by default); another might add an ad-hoc raw-bytes path purely to satisfy the diagnostic; a third might fire the code but populate `<offset>` from the decoded-string code-unit position rather than the original byte offset.

The defect is the *interaction* between two independently well-intentioned edits: the lexical rule assumes a byte-oriented read path; the seam pin closes the read path on a string. Neither commit, taken alone, would be a defect; together they leave the spec contradictory.

## Solution Space

**Shape:** single
**State:** shaped

### Recommendation

Add a `readBytes` member to the PIC-13 `FileSystem` seam. Extend the `FileSystem` interface block in `host-interfaces-services.md` with a `readBytes(path: string): Promise<Uint8Array>` member, with the same Node-style `.code` rejection mapping (`ENOENT` / `EACCES` / `EPERM`, other I/O failures pass through) already pinned for `readText`. Add a bullet stating that `readBytes` is the load-bearing surface for `.loom` / `.warp` source reads at the lexical layer, and add a one-sentence cross-reference under `lexical.md` §Encoding stating "source files are obtained via `FileSystem.readBytes` and UTF-8-validated by the runtime before decoding". The production `PiFileSystem` adapter delegates to `fs.promises.readFile(path)` (no encoding arg). The `FakeFileSystem` returns a constructor-injected `Uint8Array`.

Spec edits: PIC-13 interface block (add member + rejection annotation + load-bearing bullet); PIC-13 production-wiring bullets in `implementation-notes.md` (add one line); `lexical.md` §Encoding (add the cross-reference, retaining the existing observables); diagnostics row for `loom/load/invalid-encoding` (re-anchor "the loom-side UTF-8 decode step" to the new bullet).

This matches what the plan already builds (`V1a` consumes "the raw `Uint8Array` from the `V8b` `FileSystem` byte-read seam"; `V8b` already names `readBytes`); promoting `readBytes` to a first-class PIC-13 member removes the GOV-18 arm-(a) escape hatch both leaves rest on, matches the build-time surface, and keeps `readText`'s contract narrow (it stays the cross-cutting text reader for settings, manifests, etc.). The byte-offset observable in the diagnostic becomes directly testable through the seam: a `FakeFileSystem` returning a chosen byte sequence exercises every invalid-encoding case.

Edge cases the implementer must watch:
- The leading-UTF-8-BOM-is-consumed rule operates on the **decoded** stream; the byte-offset reported on a non-UTF-8 BOM is still `0` (the BOM's leading byte in the original byte content), per the existing diagnostic-row wording. The new `readBytes`-then-decode pipeline preserves this only if the decode-step prose explicitly says so — keep that sentence on `lexical.md` §Encoding's BOM line rather than displacing it to PIC-13.
- The production `PiFileSystem` `readBytes` adapter must call `fs.promises.readFile(path)` without an encoding argument and return the resulting `Buffer` (which is a `Uint8Array` subtype). The "no transcoding" rule in `lexical.md` §Encoding (Latin-1 / UTF-16 fail fast rather than produce mojibake) hinges on this — the adapter MUST NOT default to `"utf8"`, MUST NOT use `TextDecoder("utf-8", { fatal: false })`, and MUST NOT use chokidar's content-providing variants (chokidar is the watcher seam, not the read seam).
- `FakeFileSystem.readBytes` must accept a `Uint8Array` constructor argument; the fake's existing settings-content `string` fixtures must not be silently re-used through `TextEncoder` — that would mask the very invalid-byte-sequence test vectors the new member exists to enable.

## Consequence

**Severity:** correctness

Two conforming implementations of `lexical.md` §Encoding currently diverge on every invalid-encoding case: whether `loom/load/invalid-encoding` fires at all (decoder dependent), how `<offset>` is computed (byte offset vs code-unit offset of the decoded string), and whether a UTF-16-encoded file fails fast or is silently mangled. The implementation plan already routes around the gap with an unsanctioned `readBytes` seam member; until the spec admits the same surface, conformance tests targeting the diagnostic's byte-offset rendering cannot be written against the spec text alone.
