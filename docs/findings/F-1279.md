## Problem

The *Retired REQ-IDs* table on `docs/spec_topics/governance/anchor-scheme-and-retired.md` (anchor `id="retired-req-ids"`) is followed by a normative sentence stating that the `Retired in` column carries either a 7-character abbreviated commit SHA or a release tag. The GOV-12 and GOV-14 rows carry the literal `pending` in their `Retired in` cells, which is neither form, so those two rows violate the column rule restated immediately below the table. A maintainer cannot trace either retirement back to its demoting commit, and the table contradicts its own stated provenance rule.

## Issue introduction

**Verdict:** present-since-introduction (per-row)
**Introducing commits:**
- `fe8aa5c` — *"pi-loom spec: resolve \"GOV-12 bundles ≥3 independently-testable sub-obligations\""* (2026-06-05) added the GOV-12 row with `pending` in the `Retired in` cell.
- `5a772e3` — *"pi-loom spec: resolve \"GOV-14 two SHOULD-NOTs admit no mechanical witness\""* (2026-06-05) added the GOV-14 row with `pending` in the `Retired in` cell.

**History:** `git log -L '/GOV-12/,+1:docs/spec_topics/governance/anchor-scheme-and-retired.md'` and the symmetrical `GOV-14` query each return a single commit, both of which insert the row with `pending` already in place; the cell has never held any other value. The two commits self-reference the act of retirement, so the author had no SHA to record at write time and used the same placeholder idiom the page's earlier `<retirement commit>` cells used. Commit `3f00256` (one day earlier, 2026-06-05) had already established the resolve-the-placeholder convention for the page by filling in concrete SHAs for the GOV-2 / GOV-10 / GOV-11 / GOV-4 / GOV-21 cells and removing the parenthetical that previously legitimised the placeholder form (`"the <retirement commit> placeholder is replaced with a concrete SHA at the moment of the retiring commit"`); the follow-up resolution step that `3f00256` performed for the earlier cells was simply never run for the GOV-12 / GOV-14 cells. No interaction with other commits is involved — each row's defect is self-contained in its own introducing commit.

## Solution approach

Replace the `pending` literal in the GOV-12 row's `Retired in` cell with `` `fe8aa5c` `` and in the GOV-14 row's cell with `` `5a772e3` `` — the abbreviated SHAs of each row's introducing retirement commit, formatted as inline code to match the sibling rows.
