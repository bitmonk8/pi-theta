## Finding

`V3a`'s **Adds** names three distinct evaluator behaviours that the spec defines normatively but that no `Tests` bullet on either `V3a` or its paired `V3a-T` asserts:

- **Integer-overflow non-panic.** `expressions.md` §Other arithmetic mandates that an `integer`-typed result of `-`/`*`/`%`/unary-`-` whose magnitude exceeds `2^53 - 1` is computed in IEEE-754 double precision, silently loses precision, retains the static `integer` type, and **does not panic** (deliberately excluded from the closed panic list). The single `Other arithmetic` Tests bullet only asserts that **div/mod-by-zero** produce `Inf`/`NaN` without panic — a different operation. The safe-integer-overflow non-panic path is unasserted.
- **NaN-unordered ordering.** `expressions.md` §Ordering comparisons mandates that `<`, `<=`, `>`, `>=` against `NaN` all evaluate to `false` and never panic. The `Other arithmetic` Tests bullet names only cross-type `==` → `false`; the NaN ordering rule is unasserted.
- **Object/array construction.** `Schema { field: expr, ... }` object construction (§Object construction) and `[a, b, c]` array-literal construction with inferred element type (§Array construction) are named in Adds but exercised by no Tests bullet.

Because these behaviours are named in Adds but absent from the `Ships when` gate, a regression in any of them passes green. This is the *No silent test skipping* failure mode: the leaf reports success while three named behaviours go unverified. The `V3a` and `V3a-T` Tests sections currently carry identical bullets, so the gap exists on both the tests task and the implementation leaf.

## Solution Space

**Shape:** single

### Recommendation

Add assertions for the three behaviours to the **Tests** section of `docs/plan_topics/V3a-T-expression-evaluator.md` (the tests task that authors the failing tests) and mirror them into `docs/plan_topics/V3a-expression-evaluator.md`'s Tests section so the two stay in sync. The assertions may be added as new bullets or folded into the existing `Other arithmetic` bullet; the required content is:

- An `integer`-typed result of binary `-`/`*`/`%` or unary `-` whose magnitude exceeds `2^53 - 1` is computed in IEEE-754 double precision, loses precision silently, retains the static `integer` type, and does not panic — citing [`expressions.md` §Other arithmetic](../../docs/spec_topics/expressions.md#other-arithmetic).
- `<`, `<=`, `>`, `>=` against `NaN` (e.g. `NaN < 1`, `1 < NaN`, `NaN <= NaN`) each evaluate to `false` and do not panic — citing [`expressions.md` §Ordering comparisons](../../docs/spec_topics/expressions.md#ordering-comparisons).
- `Schema { ... }` constructs a schema-typed value (every declared field present) and `[a, b, c]` constructs an array literal whose element type is inferred per the array-literal LUB rule — citing [`expressions.md` §Object construction](../../docs/spec_topics/expressions.md#object-construction) and §Array construction.

Extend `V3a`'s **Ships when** line so the gate proves these behaviours (e.g. add "proves integer-overflow does not panic, NaN ordering comparisons evaluate to `false`, and object/array literals construct"). The spec already defines all three behaviours, so this fix is plan-side only; `expressions.md` is read-only here.

## Consequence

**Severity:** advisory

A regression in safe-integer-overflow non-panic, NaN ordering (`<`/`<=`/`>`/`>=` → `false`), or object/array literal construction ships green because the `V3a` gate never asserts these spec-named behaviours. The narrow edge cases (overflow non-panic, NaN ordering) are unlikely to be caught downstream; a faithful implementer following Adds still produces a working leaf, so the defect is a verification gap rather than an implementation ambiguity.
