## Finding

`docs/spec_topics/frontmatter/frontmatter-fields-a.md` line 49 is a standalone design-motivation paragraph that explains *why* `mode:` is the sole required frontmatter field, citing the blast-radius asymmetry between `prompt` (turns inject into the live Pi session) and `subagent` (private spawned conversation) and asserting that defaulting would route private looms into the user's session or vice versa. None of this is a behavioural rule.

The normative content the paragraph could be read as conveying is already covered upstream: the field-contract table's `mode` row (line 36) carries the "required: yes" obligation, the `loom/load/missing-mode` and `loom/load/unknown-mode-value` diagnostic split, and the explicit "missing" vs "present-but-bad" distinction. The paragraph adds no requirement, no diagnostic, no defaulting rule, and no enforcement-site contract that an implementer needs to read; it is pure rationale prose carried forward verbatim from the original frontmatter shard when the field-contract table was first introduced.

## Solution Space

**Shape:** single

### Recommendation

Delete the paragraph at `docs/spec_topics/frontmatter/frontmatter-fields-a.md` line 49 outright. The field-contract table row for `mode` (line 36) already pins the normative requirement, the `loom/load/missing-mode` diagnostic, and the recognised-key / unrecognised-value split (`loom/load/unknown-mode-value`), so removing the paragraph loses no obligation. The repository carries no ADR / decision-log surface that motivation paragraphs are routed to, so do not stage the prose elsewhere — delete it. After deletion, confirm the surrounding paragraph at line 51 (the "Frontmatter mirrors Pi's prompt-template frontmatter…" paragraph) still reads cleanly when it follows the table directly with a single blank-line separator.
