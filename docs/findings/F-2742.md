## Finding

The spec's checkpoint-granularity rule (`docs/spec_topics/cancellation.md` §Granularity) is a universal negative: the interpreter "checks the cancellation signal at exactly these points and no others" — the five fixed sites (each `for`/`while` iteration, each `@`-query dispatch, each tool call, each `invoke`, and the slash-command argument binder's LLM call), and at no other node kind. Closing this obligation requires evidence both that a checkpoint fires at each of the five sites (the positive arm, distributed across V17c, V13c, V14a, V15a) **and** that no checkpoint fires at any other interpreter node kind (the negative arm).

V17c's negative arm does not close the negative half. Its "Checkpoint granularity" Tests bullet (in both `V17c-checkpoint-granularity.md` and `V17c-T-checkpoint-granularity.md`) drives only the handful of non-checkpoint categories the test happens to exercise — inside a primitive operation (arithmetic, comparison, field/index access) and at a straight-line statement boundary — and asserts no checkpoint at those. The leaf then explicitly concedes the gap: "a checkpoint emitted at a node kind outside the test corpus is not caught by this assertion." A stray checkpoint at any interpreter node kind the test does not drive therefore ships unverified, and the spec's "no others" clause is never mechanically enforced.

This is compounded by a coverage attribution that already claims more than the leaf delivers: the `cka-47` row in `coverage-matrix.md` credits V17c with "the best-effort negative arm, **and the closed five-site set**." The matrix asserts the closed negative is covered at V17c, while the leaf only provides a best-effort negative scoped to a couple of driven categories.

## Consequence

**Severity:** correctness

The plan's `cka-47` coverage attribution credits V17c with the closed five-site set, but the leaf's negative arm is best-effort and cannot witness a checkpoint at an un-driven node kind — so an implementation that violates the spec's "no others" clause (a checkpoint at a sixth node kind) passes `npm test` green. The plan thus claims a verification it does not deliver, and the spec MUST goes unenforced at its closing leaf.

## Solution Space

**Shape:** single

### Recommendation

Strengthen V17c's negative arm into a closed negative over the interpreter's enumerated node-kind universe, in both `docs/plan_topics/V17c-T-checkpoint-granularity.md` (the failing test) and `docs/plan_topics/V17c-checkpoint-granularity.md` (the implementation leaf that proves it), in the "Checkpoint granularity" Tests bullet and V17c's **Ships when**.

Replace the best-effort negative-arm text — the clause "A best-effort negative arm also asserts the seam witnesses no checkpoint at the non-checkpoint categories the test drives … a checkpoint emitted at a node kind outside the test corpus is not caught by this assertion" — with a closed negative: drive the `V8a` `Checkpoint` seam across a generated corpus that exercises every interpreter node kind, and assert a checkpoint fires only at the five fixed sites named in `cancellation.md` §Granularity and at no other node kind. State the universe as "every interpreter node kind" (rather than an inline literal list) so a future node kind is not silently excluded from the negative.

Because the `@`-query-dispatch, tool-call, and `invoke` per-site presence arms remain distributed to V13c/V14a/V15a via the `V4f` chain, the V17c corpus need not construct those call sites to assert their *absence elsewhere*; the closed negative asserts that the only node kinds emitting a checkpoint are the five enumerated sites. This makes the leaf actually deliver the "closed five-site set" that the `cka-47` row in `coverage-matrix.md` already attributes to V17c, so no coverage-matrix edit is required. Update V17c's **Ships when** in lockstep so the shipped gate matches the strengthened Tests bullet.
