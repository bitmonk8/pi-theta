## Problem

The `H5d` transitive-completeness gate tokenises each `coverage-matrix.md` closing-leaf cell by taking exactly its backtick-delimited code spans as leaf-ID tokens, which makes the gate's correctness silently conditional on an authoring convention stated nowhere: that every closing-leaf-cell leaf-ID is backtick-delimited and that nothing else in that column is. A leaf-ID authored without backticks is dropped from the tokenised set with no signal — emptying a single-leaf cell into a spurious failure, or silently removing a facet-closer from a multi-leaf cell while the at-least-one rule keeps it green and the dropped leaf unverified. The converse half is already violated by the `PIC-17, PIC-18` row, whose parenthetical backtick-delimits the non-leaf spans `` `invoke` ``, `` `PIC-17` ``, and `` `finally` ``, so the cell tokenises to `{V9c, V15d, invoke, PIC-17, finally}` rather than the `{V9c, V15d}` that `H5d`'s Adds prose asserts.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 0315c11 — pi-loom plan-review: park F-0218 (fast-loop: partial — tokenisation rule self-contradictory) (2026-06-17, Thomas Andersen)
**History:** `H5d` was created in 617a3ad (2026-06-16) with an Adds paragraph that read the closing-leaf column and required at-least-one listed leaf in `H5b`'s Deps, without specifying how leaf-IDs are extracted from the cell. Commit 0315c11 (2026-06-17), resolving F-0218 ("tokenisation rule self-contradictory"), rewrote that paragraph to add the explicit tokenisation step — "taking exactly the backtick-delimited code spans the cell contains." That edit introduced the load-bearing dependency on `coverage-matrix.md`'s backtick authoring convention while leaving the convention itself unstated as a precondition.

## Solution approach

In `docs/plan_topics/H5d-transitive-completeness-gate.md` Adds, where the tokenisation rule is defined, state the coverage-matrix authoring precondition the tokeniser relies on and its observable consequence (leaf-IDs dropped when un-delimited; non-leaf spans wrongly captured when delimited). Record the same precondition as the closing-leaf-column authoring rule in `docs/plan_topics/coverage-matrix.md` (or in `conventions.md`'s coverage-matrix closure obligation) so row authors are bound, not only the gate implementer. Reconcile the live `PIC-17, PIC-18` row in `docs/plan_topics/coverage-matrix.md` by removing the backticks from its non-leaf parenthetical spans so the cell tokenises to `{V9c, V15d}`.

## Solution constraints

- Out of scope: the closing-leaf-cell facet-vs-co-witness annotation conventions owned by F-0319 (same-cluster).
- Preserve `H5d`'s existing `<new>` placeholder-row exclusion — a placeholder cell carries no leaf-ID and must continue to tokenise to the empty set, excluded as a placeholder rather than treated as a gate failure.
