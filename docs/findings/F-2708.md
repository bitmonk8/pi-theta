## Finding

`H1a`'s `capability-probe.md §(a)` Tests bullet defines its comparison operand as the Node-floor literal **transcribed inline into the test** — "an independently-sourced copy resident in the test itself" — because at `H1a` sequencing (`Deps: -`) no in-tree single-source constant exists to read. That framing is correct and deliberate. The defect is in the bullet's summary failure clause, which claims the assertion fails on "a typo, stale value, or value **disagreeing with `capability-probe.md` §(a)**." The test never reads live `capability-probe.md §(a)`; it compares the manifest only against the resident transcribed copy.

The two are equivalent only while the copy equals the current spec floor. If the spec floor moves and the transcribed copy is not updated in lock-step, a manifest value that disagrees with the *current* `§(a)` but still matches the *stale* copy passes green — exactly the case the clause asserts it catches. Keeping the copy aligned on a floor move is the version-bump procedure's obligation, which the same bullet already discloses ("a Node-floor move during the version-bump procedure … keeps it aligned with `capability-probe.md` §(a)"). The failure clause therefore claims gate reach the mechanism does not provide on its own.

## Consequence

**Severity:** cosmetic

The mechanism the test implements is correct; only the bullet's summary claim overstates it. A reviewer trusting the clause could believe `H1a`'s `npm test` independently catches a `capability-probe.md §(a)` floor move, when in fact a floor move not propagated to the transcribed copy passes silently until `V18c`'s live three-way gate. No implementer is blocked or led to a divergent implementation.

## Issue introduction

**Verdict:** introduced-by-specific-edit
**Introducing commits:** `f1a276a` (2026-06-16, pi-loom plan: resolve "engines.node literal-read test names no in-code comparison operand")
**History:** The `engines.node` value-assertion bullet was first added in `fd2aa64` (2026-06-11), where the comparand was described as "the spec-anchored floor" and the "value disagreeing with `capability-probe.md` §(a) fails" clause was consistent with that idealised framing. `f1a276a` (2026-06-16) narrowed the comparand to an explicit in-test **transcribed copy** ("an independently-sourced copy resident in the test itself") to fix a separate finding about the test naming no in-code operand, but carried the `§(a)`-disagreement failure clause forward verbatim — opening the gap between what the operand is (a copy) and what the clause claims it catches (disagreement with live `§(a)`). `1752a4e` (2026-06-20, F-0470) later dropped the inline `>=22.19.0` snapshot literal but left the overclaiming clause intact.

## Solution Space

**Shape:** single

### Recommendation

In the `capability-probe.md §(a)` Tests bullet of `docs/plan_topics/H1a-scaffold-and-toolchain.md`, tighten the summary failure clause so it claims only what the transcribed-copy comparand delivers: detection of a manifest value that disagrees with the resident transcribed floor literal. Replace the operative phrase "or value disagreeing with `capability-probe.md` §(a) fails `H1a`'s own `npm test`" with a phrasing scoped to the copy, e.g. "or value disagreeing with the transcribed `§(a)` floor literal fails `H1a`'s own `npm test`."

Retain the bullet's existing disclosure that a Node-floor move during the version-bump procedure keeps the transcribed copy aligned with `capability-probe.md §(a)` — that sentence is what makes catching a live-`§(a)` move a version-bump obligation rather than an `H1a`-test guarantee, and it should remain so the reader sees where the residual detection lives. No change to the comparand mechanism (transcribed copy), to the "MUST NOT read `V9a`/`V18a`" exclusions, or to `V18c`'s three-way gate is needed.
