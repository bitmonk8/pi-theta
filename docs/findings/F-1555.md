## Finding

`diagnostic-shape.md` documents the structural wire field as `file?: string  // absolute path; omitted for file-less codes` and pins the serialised content-line format as `"<file>:<line>:<col>: <code>: <message>"` — i.e. the rendered first column is a substitution of `Diagnostic.file`. The §5 *Source-derived placeholders* rule in `placeholder-rendering-b.md`, however, fixes a single rendering for `<file>` (jointly with `<path>`): "the literal text inside the path-literal quotes, with no realpath normalisation, no symlink resolution, no scheme prefixing." The two specifications collide on the same token `<file>` in the same shard: the structural field claims an absolute (and, by neighbouring spec, `realpath`-canonical) form, while §5 mandates verbatim source-text without normalisation.

The conflict is not just one of normalisation — §5's framing is "inside the path-literal quotes", which only fits message-template placeholders that interpolate a *source-text* path literal (`'<path>' does not end in .loom`, `cannot resolve .warp import '<path>'`, etc.). For the structural `Diagnostic.file` and for the leading column of the serialised line, there is no enclosing path-literal in the source: the value identifies the *file being scanned* (a `loom/parse/*` or `loom/load/*` row's input file), which has no "path-literal quotes" frame at all. Applying §5 verbatim makes the rule inapplicable to its own headline use site.

The behavioural consequence is two divergent implementations on the same input: implementer A renders `Diagnostic.file` via `FileSystem.realpath` output (matching the discovery-root containment check in `invocation.md` and the `non-canonical-extension` dedup in `discovery-sources.md`, both of which are byte-exact on `realpath` output); implementer B copies the source-text path literal verbatim. Editor jump-to-file, LSP `file://` URIs, multi-error batch ordering by `(file, line, col)` (asserted by the `V7a` leaf), and any equality test across diagnostics emitted from the same file via symlink-equivalent paths all diverge across the two.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") + `61ea1a5` (2026-05-06, "pi-loom spec: resolve \"Diagnostic message placeholders have no defined rendering\"")
**History:** The `Diagnostic.file: string  // absolute path` annotation was introduced in `fae85d3` as part of the implementation-plan tightening pass on the then-monolithic `spec.md`. Two days later, `61ea1a5` added §5 *Source-derived placeholders* to `spec_topics/diagnostics.md` to close the unresolved "Diagnostic message placeholders have no defined rendering" review item, pinning `<file>` (jointly with `<path>`) to the verbatim "literal text inside the path-literal quotes" rule. Neither commit reconciled the two derivations: the field comment kept its "absolute path" framing and the new placeholder rule introduced a conflicting one, both surviving the `f5e89f4` (2026-06-04) shard split intact. The most recent diagnostics-shape edit, `b2f780b` (2026-06-24, F-1152), reworked the same `file?` comment line to swap "location-less codes" for "file-less codes" while preserving "absolute path" verbatim — confirming that the conflict has been carried through every subsequent edit of either site without the two ever being read against each other. The defect is the interaction between the two commits rather than either in isolation: `fae85d3` was reasonable for an unfilled placeholder surface, `61ea1a5` was reasonable for message-template path interpolation, and the conflict surfaces only at their join — the structural field whose name §5 happens to also use as a placeholder token.

## Solution Space

**Shape:** single

### Recommendation

Pin two distinct derivations and scope §5 explicitly to the second.

1. **Structural `Diagnostic.file`** (the wire field, and the first-column substitution in the serialised content-line format): the byte-exact output of [`FileSystem.realpath`](../pi-integration-contract/host-interfaces-services.md#pic-13) applied to the source file the emission site reads, with forward-slash normalisation per *Path literals* in `lexical.md`. This is the same canonical-string form already pinned for discovery-root containment in `invocation.md` and for `non-canonical-extension` per-source dedup in `discovery-sources.md`, so the diagnostic surface inherits their `realpath` guarantee rather than defining a parallel one. The same derivation applies to each `Diagnostic.related[i].file`.

2. **§5 message-template `<path>` / `<file>` placeholder**: keep the verbatim "literal text inside the path-literal quotes" rule, but restrict its scope to interpolation inside *Message* templates whose surrounding template names a source-text path literal — i.e. the `<path>` rows in `code-registry-parse.md` and `code-registry-load.md` (`invoke path '<path>' …`, `cannot resolve .warp import '<path>'`, `discovery source path does not exist: <descriptor>`, etc.) and any analogous future `<file>` placeholder. State explicitly that this rule does **not** govern the structural `Diagnostic.file` field or the first-column substitution in the serialised content-line format; the *Serialised content format* paragraph in `diagnostic-shape.md` interpolates the canonical `Diagnostic.file` value from (1), not a §5 placeholder render.

Edits required:

- In `diagnostic-shape.md`, change the `file?:` field comment from `// absolute path; omitted for file-less codes …` to `// realpath-canonical absolute path of the source file (per PIC-13 FileSystem.realpath, forward-slash-normalised); omitted for file-less codes …`, and add a sentence to the *Serialised content format* paragraph stating that the `<file>` substitution in `"<file>:<line>:<col>: <code>: <message>"` is the `Diagnostic.file` value defined above, not a §5 placeholder.
- In `placeholder-rendering-b.md`, edit the §5 `<path>` / `<file>` bullet to read "the literal text inside the path-literal quotes in the *source text* the offending message names, with no realpath normalisation, no symlink resolution, no scheme prefixing", and add a follow-on sentence: "This rule governs `<path>` / `<file>` *placeholder interpolation* inside *Message* templates that name a source-text path literal; it does **not** govern the structural `Diagnostic.file` field, whose derivation is pinned in [diagnostic-shape.md](./diagnostic-shape.md)."
- Cross-link both sites (a back-reference from §5 to the structural field, a forward link from the field comment to §5) so a future reader hits whichever rule applies to the position they are reading.

Edge cases the implementer must watch: (i) `related?` array entries carry their own `file` keys, and the structural derivation in (1) applies elementwise; (ii) the existing `loom/parse/invoke-non-loom-extension` example §5 cites as its origin remains correct under the narrowed scope because it interpolates a source-text path literal into the message body, not into `Diagnostic.file`; (iii) on case-insensitive hosts the `realpath` canonical form preserves on-disk casing per PIC-13, so two diagnostics emitted from the same file under case-variant path inputs share one `Diagnostic.file` value, which is the intended `(file, line, col)`-ordering and dedup behaviour `V7a` asserts on.

## Consequence

**Severity:** correctness

Two conformant implementers produce different `Diagnostic.file` strings and different serialised first columns for the same input — one realpath-canonical, the other verbatim-source-text — so editor jump-to-file, LSP URI construction, the `(file, line, col)` ordering `V7a` asserts on the assembled `Diagnostic[]`, and cross-file equality of related-site `file` keys all diverge. Tests sourcing the rendered string from the registry per DIAG-4 will disagree on byte content. The contradiction also leaves `loom/parse/*` and whole-file `loom/load/*` rows with no rendering rule that fits, because §5's "path-literal quotes" framing presupposes a source-text host that those rows do not have.
