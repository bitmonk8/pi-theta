## Problem

`diagnostic-shape.md`'s `file?:` field comment specifies `Diagnostic.file` as an absolute path (and, by the neighbouring discovery-root containment and dedup spec, a `realpath`-canonical one), and the *Serialised content format* paragraph substitutes that value as the first column of `"<file>:<line>:<col>: <code>: <message>"`. The §5 *Source-derived placeholders* rule in `placeholder-rendering-b.md` instead fixes the `<file>` token (jointly with `<path>`) to the verbatim literal text inside the path-literal quotes, with no realpath normalisation. The two specifications collide on the token `<file>` in the same shard, so two conformant implementers produce divergent `Diagnostic.file` strings and serialised first columns — breaking editor jump-to-file, LSP URI construction, the `(file, line, col)` ordering `V7a` asserts on the assembled `Diagnostic[]`, and cross-file equality of related-site `file` keys. §5's "inside the path-literal quotes" framing also has no source-text frame for the structural `Diagnostic.file` value, which identifies the file being scanned rather than an interpolated path literal.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") + `61ea1a5` (2026-05-06, "pi-loom spec: resolve \"Diagnostic message placeholders have no defined rendering\"")
**History:** The `Diagnostic.file: string  // absolute path` annotation was introduced in `fae85d3` as part of the implementation-plan tightening pass on the then-monolithic `spec.md`. Two days later, `61ea1a5` added §5 *Source-derived placeholders* to `spec_topics/diagnostics.md` to close the unresolved "Diagnostic message placeholders have no defined rendering" review item, pinning `<file>` (jointly with `<path>`) to the verbatim "literal text inside the path-literal quotes" rule. Neither commit reconciled the two derivations: the field comment kept its "absolute path" framing and the new placeholder rule introduced a conflicting one, both surviving the `f5e89f4` (2026-06-04) shard split intact. The most recent diagnostics-shape edit, `b2f780b` (2026-06-24, F-1152), reworked the same `file?` comment line to swap "location-less codes" for "file-less codes" while preserving "absolute path" verbatim — confirming that the conflict has been carried through every subsequent edit of either site without the two ever being read against each other. The defect is the interaction between the two commits rather than either in isolation: `fae85d3` was reasonable for an unfilled placeholder surface, `61ea1a5` was reasonable for message-template path interpolation, and the conflict surfaces only at their join — the structural field whose name §5 happens to also use as a placeholder token.

## Solution approach

Pin two distinct derivations and scope §5 to the second. Rewrite the `file?:` field comment and the *Serialised content format* paragraph in `diagnostic-shape.md` to define `Diagnostic.file` (and each `related[i].file`) as the `realpath`-canonical absolute path per PIC-13 `FileSystem.realpath`, and to state that the serialised first-column `<file>` substitution is that value rather than a §5 placeholder render. Narrow the §5 `<path>` / `<file>` bullet in `placeholder-rendering-b.md` to govern only message-template placeholder interpolation that names a source-text path literal, stating that it does not govern the structural `Diagnostic.file` field. Add cross-links between the two sites so a reader reaches whichever rule applies to the position they are reading.
