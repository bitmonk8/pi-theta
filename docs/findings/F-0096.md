## Finding

`V14a` ("Tool calls (code-side) and `CodeToolError`") carries two separable concerns in one leaf. The **parse-time argument checks** (`loom/parse/tool-arg-not-literal`, `loom/parse/tool-arg-arity`, `loom/parse/tool-arg-type-mismatch`) reject a malformed `<name>(args)` call before any dispatch occurs â€” they are testable against a stub executor and need only the resolved callable set plus the type system. The remainder of the leaf is the **runtime execution path**: the closed `CodeToolError` enum, the three off-surface outcome handlings, both return-type rows, the host-denialâ†’`Err{code_tool,execution}` routing (the `conversation-drive.md` *No additional access channels* MUST), the depth-6 ceiling-#4 wrapping witness, and the cancellation swallowing-handler three-channel suppression at the `Checkpoint` seam. These two halves share no fixture: the parse checks short-circuit before the executor runs, while the runtime assertions all stand up the executor, the `Checkpoint` seam (`V8a`), and the depth walk (`V5e`).

A reviewer of the `V14a`/`V14a-T` pair must therefore verify a static call-site validator and a full runtime-execution + cancellation surface in a single pass, enlarging the per-commit review surface beyond what either half needs.

There is an additional wrinkle bearing on where the parse checks should land: `loom/parse/tool-arg-not-literal` is already asserted in `V2a`/`V2a-T` as a literal-sublanguage violation (`loom/parse/default-not-literal`, `loom/parse/tool-arg-not-literal`, `loom/parse/missing-object-field`). So the three "parse-time argument" codes in `V14a` are not homogeneous: `not-literal` is a literal-sublanguage concern (already co-owned by `V2a`), whereas `tool-arg-arity` and `tool-arg-type-mismatch` are call-site semantic checks against the tool's input schema. Any relocation should treat the arity/type-mismatch pair as the call-site unit and avoid double-relocating `not-literal`.

### Recommendation
