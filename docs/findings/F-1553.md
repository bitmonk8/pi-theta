## Finding

`loom/runtime/cancelled-by-session-shutdown` carries an SDK-event sibling `details.event.loom` whose value is the cancelled invocation's *canonical-key name* per the [`ActiveInvocationRegistry`](../spec_topics/pi-integration-contract/active-invocation-registry.md) (post-`pi.looms` mapping, stem taken verbatim, no leading `/`). Across the rest of the spec corpus, the bare token "loom" denotes either the source-language ("loom 1.0", "loom code") or a `.loom` *file unit* ("the `.loom` callable", "a loom invocation"). The field name therefore reads as "the loom" — i.e. the file or language — whereas the value is the invocation's registered identifier.

The same naming overload also propagates to the construction-site fallback wire shape `` `${code} ${entry.loom} <unreadable>` `` pinned by [Diagnostic-emission isolation](../spec_topics/pi-integration-contract/diagnostic-emission-isolation.md) and to the registry's own `loom: string` member spelled in active-invocation-registry.md. An operator-tooling consumer parsing `details.event.{...}` for the first time has to read the row's *Trigger* prose to discover that "loom" here means "the slash-name (or callee-only canonical key) of the invocation that was cancelled", not "the cancelled `.loom` file" — and worse, "the slash-name" itself is only true for slash-registered looms; callee-only `.loom` files reached via literal `invoke("./path.loom", …)` or via `tool.execute(...)` for subagent-mode tool-only callees carry the same field but have **no** slash registration. The honest name is something like `loom_name` / `canonical_key` / `name`; the current `loom` actively misdirects on both axes (file-unit vs identifier; slash-name vs canonical key).

DIAG-3 makes registered **code identifiers** loom-2.0-deferred; the wire-payload field `details.event.loom` is not itself a code, but renaming it would still be a breaking change to the operator-tooling consumer contract that the [`details` shape conventions](../spec_topics/diagnostics/diagnostic-shape.md#session-shutdown-details-conventions) explicitly entitle ("operator-tooling consumers MUST whitelist `details.event.loom` … when applying the strict-validation entitlement"). The substantive rename therefore belongs in the same loom-2.0 migration bucket; the loom-1.x action is to record the rename intent at the field's normative home so a future maintainer doesn't re-litigate the choice and current readers can map the field name to its true value space.

## Solution Space

**Shape:** single

### Recommendation

Add an inline rename-intent NOTE at the field's normative home — the [`session-shutdown-details-conventions`](../spec_topics/diagnostics/diagnostic-shape.md#session-shutdown-details-conventions) *Runtime-constructed sibling carve-out* clause in `docs/spec_topics/diagnostics/diagnostic-shape.md`, where `details.event.loom` is pinned — recording that the loom-1.x field name `loom` overloads the "loom file / language" reading and is scheduled to be renamed to **`details.event.loom_name`** in the loom 2.0 wire-shape migration, on the same footing as the DIAG-3 code-rename deferral. Pick `loom_name` rather than `slash_name` because the field's pinned value space includes the canonical-key form for callee-only `.loom` files reached via literal `invoke("./path.loom", …)` and via `tool.execute(...)` on subagent-mode tool-only callees — which carry no slash registration — and `loom_name` truthfully names that broader space; `slash_name` would re-misname the callee-only arm.

The NOTE must also (a) cite that the same rename applies to the `entry.loom` token in the *Construction-site wrap* three-token fallback `` `${code} ${entry.loom} <unreadable>` `` so the structured and fallback wire shapes stay coherent across the 2.0 migration, and (b) flag that the [`ActiveInvocationRegistry`](../spec_topics/pi-integration-contract/active-invocation-registry.md) entry's own `loom: string` member is the source-of-truth field; renaming the wire payload without the registry member would split the lowerCamelCase-on-registry / snake_case-on-wire mapping the spec already documents for `invocationId` ↔ `invocation_id`, so the 2.0 rename ships both spellings (`entry.loomName` on the registry side; `loom_name` on the wire) as one coupled change.

Edge cases the loom-1.x NOTE must call out:

- The NOTE is **non-normative** for loom 1.x — no wire shape changes today. The current `details.event.loom` field and `entry.loom` registry member stay exactly as pinned. The strict-validation entitlement and the construction-obligation member set remain unchanged.
- The NOTE is **co-located** with the existing `reload-teardown-timeout` and `loom/load/cross-format-collision` rename-intent notes (siblings under the same loom-2.0 naming-deferral pattern); the three should be discoverable together so the loom-2.0 maintainer finds them as one batch.
- Do not introduce a `loom/typecheck/*` build-time brand for the rename — brands are reserved for SDK-pin equality assertions per the [`loom/typecheck/*` carve-out](../spec_topics/diagnostics/diagnostic-shape.md#loom-typecheck-namespace); the wire-field rename is a spec-versioned wording change governed by DIAG-4 (and the analogous wire-shape-stability pin), not an SDK-pin gate.

## Consequence

**Severity:** advisory

An operator-tooling implementer wiring strict validation against the closed-shape entitlement reads the field name `loom` and must consult the row prose to learn the field actually carries the invocation's canonical-key identifier (slash-name when registered; canonical key otherwise). The misnomer does not break any wire contract — the value space and presence MUSTs are fully pinned — but it sets wrong expectations on every first read, the same way "reload"-prefixed timeout-codes set wrong expectations on `session_shutdown` reasons other than `"reload"`. Without a recorded rename intent, the choice will be re-litigated each time the field is encountered in review.
