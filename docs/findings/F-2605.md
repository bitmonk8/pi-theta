## Finding

The *Stop-reason classification* rule in `provider-error-mapping.md` ends with the
catch-all bullet: *"Any other **non-terminal** stop reason — including a
content-filter terminator, or any stop reason the runtime does not otherwise
recognise — maps to `TransportError` with `retryable: false` (the terminator is
a definite outcome of the request, not a transient condition …)."*

"non-terminal" is the wrong adjective and internally contradicts the bullet that
carries it. A stop reason is, by construction, the signal that a 200 response has
*terminated*; the same sentence then justifies the mapping precisely because "the
terminator is a definite outcome of the request." A reader who takes "non-terminal"
as the operative predicate is left with an empty set (no stop reason is
non-terminal), and the named example — a content-filter terminator — is itself a
terminal outcome, so the qualifier directly contradicts its own illustration.

The intended set is simply "every stop reason that is not already classified by
the rows above": neither the normal turn boundaries `end_turn` / `stop` /
`tool_use` (handled by the tool loop) nor the output-boundary `length` (handled by
the preceding bullet, which maps it to `ContextOverflowError`). The fix is to drop
the misleading adjective and state the residual set by exclusion, preserving the
existing example list ("content-filter terminator … or any stop reason the runtime
does not otherwise recognise") that already makes the catch-all exhaustive.

## Solution Space

**Shape:** single

### Recommendation

In the catch-all bullet of *Stop-reason classification*, replace "Any other
non-terminal stop reason" with a phrasing that defines the residual set by
exclusion and drops "non-terminal", e.g.:

> Any other stop reason — i.e. any resolved `AssistantMessage.stopReason` other
> than the normal turn-boundary reasons (`end_turn` / `stop` / `tool_use`) or the
> output-boundary `length` handled above — including a content-filter terminator,
> or any stop reason the runtime does not otherwise recognise — maps to
> `TransportError` with `retryable: false` …

Keep the existing parenthetical rationale and the example list unchanged. The edit
is purely terminological: the behaviour (everything not matched by an earlier row →
`TransportError(retryable:false)`) is unchanged, so no diagnostic, table, or
acceptance criterion needs to move.

## Consequence

**Severity:** advisory

The contradictory qualifier obscures which stop reasons fall to the
`TransportError(retryable:false)` catch-all. The explicit example list
("content-filter terminator … or any stop reason the runtime does not otherwise
recognise") keeps the rule recoverable, so a careful implementer still arrives at
the correct mapping; the defect is a clarity gap rather than a behavioural fork.
