## Finding

PIC-13's `realpath` bullet ([`docs/spec_topics/pi-integration-contract/host-interfaces-services.md` line 114](../../docs/spec_topics/pi-integration-contract/host-interfaces-services.md)) binds two normative MUSTs on the production `PiFileSystem` adapter: a **Case-canonicalisation guarantee** (case-variant inputs that resolve to one directory entry yield byte-identical output) and a **Byte-stable canonical-output guarantee** (two input paths naming the same on-disk target yield byte-identical output). The text then explicitly states that bare delegation to `fs.promises.realpath` is insufficient — `fs.promises.realpath` and `fs.promises.realpath.native` disagree on leaf-case output — and concludes that "the adapter must canonicalise leaf case to satisfy it."

What the adapter must *do* to canonicalise leaf case is then left blank. No canonicalisation target is named (on-disk `readdir` entry casing? the casing of the entry returned by the OS when the path was created? Unicode-normalised? input-byte-preserving?), no `fs` call sequence is prescribed, and no decomposition into the available primitives (`fs.promises.realpath` plus `readdir` on the parent and a case-insensitive equality test, vs. `realpath.native` with a known-failing case) is given. Node provides neither call that satisfies the MUST on its own, so the spec mandates a behaviour for which it gives the implementer no constructive recipe.

The load-time and INV-1 invocation-time discovery-root containment checks in [Invocation — Resolution](../../docs/spec_topics/invocation.md), the hot-reload eviction comparison in [`V15e`](../../docs/plan_topics/V15e-hot-reload-cache-eviction.md), and the DISC-3 dedup in [`discovery-sources.md`](../../docs/spec_topics/discovery/discovery-sources.md) all decide on byte-for-byte equality of `realpath` output, so two adapters that pick different canonical leaf casings (e.g. on-disk `readdir` casing on one and `path.basename(input).toLowerCase()` on another) would each individually satisfy "byte-identical for case-variant inputs" while disagreeing on which canonical bytes the containment / dedup comparison ever sees — and the spec admits both as conforming.

## Solution Space

**Shape:** single

### Recommendation

Pin the canonical leaf casing as **the entry name `readdir` on the parent directory returns for the case-insensitively-matching entry** and pin the adapter's call sequence inline on the PIC-13 `realpath` bullet:

1. Call `fs.promises.realpath(input)` to resolve the symlink chain and get an absolute path `p`.
2. For each component of `p` from the root downward, call `readdir` on the accumulated parent and replace the component with the unique entry whose name compares case-insensitively (using `String.prototype.localeCompare` with `{ sensitivity: "accent" }`, which folds case but preserves accented characters — matching PIC-13's existing "MUST NOT subject to Unicode normalisation" posture on `readdir`).
3. If no entry case-insensitively matches (the path was deleted between the `realpath` and the `readdir`), reject with the same `.code` Node would have surfaced (`ENOENT`).
4. Return the assembled canonical path.

Pinning the target as `readdir` entry bytes makes both the Case-canonicalisation guarantee and the Byte-stable canonical-output guarantee fall out of one rule — every input that resolves to the same on-disk entry walks back to the same `readdir`-byte chain — and reuses the entry-name encoding posture PIC-13 already binds (raw on-disk UTF-8 bytes, no Unicode normalisation, per the **Entry-name encoding guarantee** bullet on the `readdir`/`lstat` line). The `FakeFileSystem` already resolves case-variant inputs to one canonical path against its in-memory symlink table, so the additional pin lands cleanly on the production adapter only.

Edge cases the implementer must watch: (a) on a case-sensitive Linux filesystem, the `readdir`-walk step is a no-op (no two distinct entries case-insensitively match the same component), so the byte-stable rule degenerates to `fs.promises.realpath`'s own output and the call sequence stays correct; (b) the walk must surface `EACCES`/`EPERM` from `readdir` on any intermediate directory verbatim as the `.code` the PIC-13 bullet already pins; (c) two entries that differ only in case in the *same* directory (the DISC-3 case-collision case) are explicitly handled by DISC-3's lexicographically-first-wins rule one layer up, so the canonicalisation step can pick the first case-insensitive match deterministically (sorted byte-ascending) without violating DISC-3.

## Consequence

**Severity:** correctness

Two reasonable implementers will pick different canonical leaf casings — `readdir`-on-parent matching, `realpath.native`'s on-disk casing, or input-byte preservation are all defensible readings of the current text — and produce production builds that disagree byte-for-byte on the containment / dedup comparison they each individually satisfy as PIC-13-conforming. The mismatch is silent: each build's tests pass against its own fake, but a callee path admitted by one build can escape under the other, and the DISC-3 dedup rule fires on one build and stays silent on the other.
