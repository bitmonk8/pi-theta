## Finding

The spec couples two outcomes for an unresolvable `model:` value: `frontmatter/frontmatter-fields-a.md` §`model` states a present-but-unresolvable `model:` is the load-time error `loom/load/model-unresolved` **and** the loom is not registered. V6a's `loom/load/model-unresolved` Tests bullet (mirrored in V6a-T) imports that coupled phrasing verbatim — "fails the load and the loom is not registered" — into a leaf whose `Deps` are `{V6a-T, V1a, V5a}`.

The "loom is not registered" half is not observable anywhere in that closure. Registration is owned by V9b (steps 1–5: walk discovered looms, build the pending-registration list, `registerCommand` the survivors). V6a cannot depend on V9b — V9b already depends on V6a for the model-reference-matcher injection seam, so the edge would be circular. V6a's tests can therefore assert only the half its hook actually produces: the parser's model-resolution hook fires `loom/load/model-unresolved` and the load fails. The "not registered" consequence is asserted neither in V6a nor in V9b (V9b's tests assert atomic registry swap, structural-change-note suppression, and matcher instance-identity — none of them assert that a model-unresolved loom is dropped from the pending-registration list).

Two compounding gaps make this a real defect rather than a wording nit. First, V6a's `Ships when` ("`npm test` parses frontmatter, requires `mode:`, and tolerates unknown keys as warnings") omits the model-unresolved outcome entirely, so the leaf's acceptance gate does not fire on it. Second, `coverage-matrix.md` maps the **entire** §`model` rule — including "the loom is not registered" — to `V6a` alone, so the coverage gate is satisfied by a leaf that can only ever test the code-fires half; the "not registered" clause passes vacuously and ships unverified.

## Solution Space

**Shape:** multiple

This finding carries two obligations — (1) gate the model-unresolved outcome in V6a's acceptance, and (2) give the "loom is not registered" half a closing assertion site — and the source offers two materially different ways to discharge the second.

### Option A — Close the "not registered" half at V9b; split the coverage attribution

**Approach.** Restate V6a's (and V6a-T's) `loom/load/model-unresolved` Tests bullet so the in-leaf assertion covers only what V6a's closure observes — the parser's model-resolution hook fires `loom/load/model-unresolved` and the load fails — and convert the "and the loom is not registered" clause into a forward cross-reference to V9b rather than an in-leaf assertion. Add a model-unresolved clause to V6a's `Ships when` (e.g. append "and fails the load with `loom/load/model-unresolved` on a present-but-unresolvable `model:` value"). In V9b/V9b-T, add a test asserting a discovered loom whose `model:` resolves to no available model is excluded from the pending-registration list and never reaches `pi.registerCommand`, cross-referencing V6a's `loom/load/model-unresolved`. Amend the `coverage-matrix.md` §`model` row so the "loom is not registered" clause is attributed to V9b (or to both V6a and V9b).
**Plan edits.** V6a Tests bullet + Ships when; V6a-T Tests bullet; V9b Tests + Ships when; V9b-T Tests; `coverage-matrix.md` §`model` row.
**Spec edits.** None.
**Pros.** The "not registered" consequence becomes observable at a leaf (V9b) whose `Deps` already include V6a; the coverage row stops passing vacuously.
**Cons.** Larger diff spanning V9b and the coverage matrix.
**Risks.** Interacts with the proposed V9b split (see Related Findings) — the assertion must attach to the registration walk, not the matcher-wiring concern.

### Option B — Keep V6a as the sole closing site; restate it to "load fails" only

**Approach.** Restate V6a's (and V6a-T's) Tests bullet to assert only "the load fails" (the code fires), strike the "not registered" claim entirely, and add the model-unresolved clause to V6a's `Ships when`, delegating registration-suppression to V9b's existing pending-registration semantics without a dedicated V9b assertion. To avoid leaving the coverage row vacuous, also narrow the `coverage-matrix.md` §`model` row's V6a attribution to the code-fires half.
**Plan edits.** V6a Tests bullet + Ships when; V6a-T Tests bullet; `coverage-matrix.md` §`model` row.
**Spec edits.** None.
**Pros.** Smallest diff; bounded to V6a plus the coverage row.
**Cons.** The "loom is not registered" consequence remains asserted by no leaf; it ships unverified.
**Risks.** A spec outcome silently has no closing assertion.

### Recommendation

Take Option A — the "not registered" half is a spec-mandated outcome and deserves an observing leaf rather than silent delegation, and V9b is the natural host because it already depends on V6a and owns the registration walk. Resolve in order: first land the scope-bounding V6a edit (restate the model-unresolved Tests bullet to the assertable "load fails" half, add the `Ships when` clause, mirror in V6a-T) so V6a sits on a stable baseline; then add the "not registered" assertion at V9b/V9b-T and split the `coverage-matrix.md` §`model` row. Edge case to watch: attach the not-registered assertion to V9b's registration walk (steps 1–5 / pending-registration list), not to the model-reference-matcher wiring, so a later extraction of that wiring into a sibling leaf does not strand it.

## Consequence

**Severity:** correctness
