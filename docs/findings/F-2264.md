## Finding

Step 5 of the version-bump procedure (`version-bump-triggers.md`) carries a `MUST`-level engineering deliverable that is not a constants edit: *"The implementation MUST implement a build-time audit pass over the same three grep targets (`src/`, `test/`, and `docs/`) and second-stage anchor filter so that a result-delta against the documented enumeration fails `npm test` red rather than relying on contributor-side discipline."* This obligates the implementation to build a new CI gate — a standing enumeration audit over the source tree, wired to fail `npm test` — distinct from the per-bump constants edit the step is nominally about.

The deliverable is buried in the interior of step 5, whose heading scope is "Update the capability-probe pinned constants." It is reachable only by reading the long *Enumeration mechanism — grep target* sub-paragraph to its end. Nothing surfaces it as a deliverable at the level a reader scans for scope: the procedure's introduction (`version-bump-intro.md`) does not name it; the `Non-goals` index there does not place it in or out of scope; and the steps 5–7 output summary — which enumerates the procedure's concrete outputs as (a) a green build-time test run, (b) a single bump commit, and (c) green runtime evidence — does not list the audit gate among them. Output (a) names only the SDK-surface / `peerDependencies` / `engines.node` consistency assertions, not the inbound-reference audit.

The result is a now-mandatory implementation artifact that is invisible at every place a contributor or planner would look to scope the bump procedure's deliverables. A contributor reading step 5's heading as a constants-only edit, or sizing the work from the output summary, will not see that a CI gate must be authored — so the gate that was introduced precisely to remove reliance on contributor discipline can itself be silently skipped.

## Issue introduction
**Verdict:** single-commit

**Introducing commit:** `c0259ca` — *"pi-loom spec: resolve 'Version-bump procedure: four obligations have no verifiable acceptance criterion'"* (2026-06-07)

**History:** The build-time audit clause first entered the corpus at `52b9101` (2026-05-11, *"resolve T24 — Fork-reason watcher closure..."*), where it was an optional, appropriately-scoped adornment: *"The implementation MAY additionally implement a build-time audit pass over the same three grep targets ... PIC does not require the audit."* As a `MAY`, it carried no obligation to surface it as a deliverable. Commit `c0259ca` upgraded that `MAY` to a `MUST` (*"The implementation MUST implement a build-time audit pass ... fails `npm test` red"*) and inserted the forward-reference *"at loom 1.0 a build-time audit gate (pinned later in this step) re-runs that enumeration on each `npm test`"*, in order to give the inbound-reference sweep a verifiable acceptance criterion. That modality flip converted an optional nicety into a mandatory engineering deliverable, but `c0259ca` made no corresponding edit to the procedure's introduction, its `Non-goals` index, or the step-7 output summary — leaving the now-required gate un-surfaced. The scope defect therefore dates entirely to `c0259ca`; the prior `MAY` form was not defective.

## Solution Space

### Recommendation

Surface the build-time inbound-reference audit gate as a named, scannable deliverable at the procedure's scoping surfaces, while leaving step 5's detailed mechanics in place.

- Edit the steps 5–7 output-summary sentence in `version-bump-triggers.md` so the procedure's enumerated outputs explicitly include the inbound-reference audit gate — extend output (a) or add a sibling output reading "the inbound-reference audit gate that fails `npm test` red on an unreconciled enumeration delta." Word it as an *implementation deliverable* (a CI gate the implementation builds), not merely a per-bump action, so it maps cleanly onto plan leaf `V18c`.
- Add a one-line mention of the audit gate to the `Pi version bump procedure` intro in `version-bump-intro.md`, so the deliverable is visible before step 5.
- Keep the normative detail in step 5, but reduce its interior occurrence in the *Enumeration mechanism — grep target* sub-paragraph to a back-reference to the now-named deliverable.

This surfaces the gate with the smallest diff: no step renumbering, so the many `step 5` / `step 6` / `step 7` cross-references across the PIC cluster stay valid. The change is purely additive surfacing plus one back-reference.

## Consequence

**Severity:** advisory

The audit gate is normatively required, but because it is invisible in the procedure's deliverable surface (intro, non-goals, output summary) and absent from the plan leaf that owns the static build-time gates, it can be overlooked entirely. If it is, the inbound-reference sweep falls back to contributor discipline — the exact failure mode the `MUST` was added to eliminate — and a missed inbound `SessionShutdownEvent['reason']` reference silently no-ops on a future bump with no red.
