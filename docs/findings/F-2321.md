## Finding

`H5d`'s Adds defines how the transitive-completeness gate arm expands the two sides it compares: it expands contiguous `<group><letter>` ranges by letter suffix and adds "`H5b`'s named singletons (`H1a`, `H5a`, `M`)". The parenthetical reads as the exhaustive enumeration of H5b's non-range bare leaf IDs.

It is not. `H5b`'s `Deps.` lists five non-range singletons — `H1a`, `H5a`, `H5e`, `H5f`, `M` — alongside its `<group><letter>` ranges. `H5e` (un-anchored normative-MUST gate arm) and `H5f` (per-facet citing-test gate arm) are both coverage-producing leaves H5b depends on directly, and both are absent from H5d's enumeration.

An implementer building the H5d arm from the literal `(H1a, H5a, M)` list would omit `H5e`/`H5f` from the named-singleton membership set. Any coverage-matrix closing-leaf cell whose only listed leaf were `H5e` or `H5f` would then resolve to no member of the expanded `H5b`-`Deps.` set and the arm would spuriously redden. No current coverage-matrix row closes solely on `H5e`/`H5f` (both are gate-machinery leaves rather than spec-REQ-ID closers), so the misfire is latent today — but H5d's rule no longer matches its own stated grounding (H5b's `Deps.`), and the enumeration silently drifts further out of sync each time a new bare singleton is added to H5b.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `5f33832` ("resolve 'H5a bundles seven gate arms behind a single Ships-when'", 2026-06-18) and `26a8794` ("resolve 'split coverage-matrix rows can ship a facet untested'", 2026-06-18)
**History:** `H5d` was created in `617a3ad` (2026-06-16) with the clause "`H5b`'s named singletons (`H1a`, `H5a`, `M`)". At that commit `H5b`'s `Deps.` named exactly those three singletons (`H1a, H5a, M`), so the enumeration was correct as authored. `5f33832` later added `H5e` to `H5b`'s `Deps.` and `26a8794` added `H5f`; neither commit touched `docs/plan_topics/H5d-transitive-completeness-gate.md`, so H5d's hand-copied mirror of the singleton set fell out of sync with its source. The defect is the divergence between the two files, not a flaw present at either leaf's inception.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H5d-transitive-completeness-gate.md`, the Adds clause currently reads "… and `H5b`'s named singletons (`H1a`, `H5a`, `M`)". Rewrite it so the named-singleton membership set is defined as every non-range bare leaf ID in `H5b`'s `Deps.` — currently `H1a`, `H5a`, `H5e`, `H5f`, `M` — rather than a fixed list. Grounding the clause in `H5b.Deps` (the authoritative coverage-producing set) keeps the two in lockstep so a future singleton added to `H5b.Deps` does not silently re-introduce the same drift.

`H5b` is the source of truth for this set and is read-only for this fix; only H5d's Adds prose changes. H5d's Tests and Ships-when do not enumerate the singletons, so no further edit is required there.

## Consequence

**Severity:** correctness

If the H5d arm is built from the literal `(H1a, H5a, M)` enumeration, its named-singleton membership set excludes `H5e`/`H5f`, so any coverage-matrix cell whose sole closing leaf is `H5e` or `H5f` would resolve to no reachable leaf and the gate would spuriously fail. No current row closes solely on `H5e`/`H5f`, so the misfire is latent; but the rule contradicts its own grounding in `H5b`'s `Deps.`, so two reasonable implementers — one reading the parenthetical as exhaustive, one grounding the set in `H5b.Deps` — produce divergent gates.
