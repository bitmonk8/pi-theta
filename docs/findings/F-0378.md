## Finding

`V9b`'s `Ships when` field carries a model-reference-matcher clause that re-narrates the construction-and-injection *mechanism* already fully described in the `Adds` field: "constructs the model-reference matcher from `ctx.modelRegistry.getAvailable()` once at the load pass, injecting that single instance into the `V6a` parser seam so `V6a`'s `loom/load/model-unresolved` resolution binds that instance". The `Adds` field already owns that mechanism ("`V9b` constructs the concrete model-reference matcher … run against `ctx.modelRegistry.getAvailable()` … and injects it into the frontmatter parser through the injection seam `V6a` defines").

A `Ships when` clause should state the observable acceptance criterion that closes the leaf, not re-describe how the implementation is built. The load-bearing, testable property here is the one stated in the trailing parenthetical — *instance identity, not equivalence-of-outcome*: `V6a`'s load-time `loom/load/model-unresolved` resolution binds the same matcher instance `V9b` constructs, observed by single-source-of-construction rather than by comparing outputs against a shared fake. The `getAvailable()`-and-inject narrative around that parenthetical duplicates `Adds` and adds no gating signal.

**Verdict:** single-commit

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9b-registration-reload-wiring.md`, trim the model-matcher clause of the `Ships when` field so it asserts only the observable gate — that `V6a`'s load-time `loom/load/model-unresolved` resolution binds the same matcher instance `V9b` constructs at the load pass (instance identity, not equivalence-of-outcome against a shared fake) — and strike the `ctx.modelRegistry.getAvailable()` construct-and-inject mechanism wording, which is already stated in the `Adds` field. The registration / atomic-swap / `registry-swap-failed` / structural-change-note clauses of `Ships when` are unaffected. No spec edits and no `Adds` edits are required (the mechanism stays in `Adds`).

## Consequence

**Severity:** cosmetic

The `Ships when` already includes the observable instance-identity criterion, so the gate remains checkable; the redundant mechanism narrative is duplicated organisation between `Adds` and `Ships when`, not a divergence risk. Left unfixed it enlarges the field a future reviewer must reconcile against `Adds` and weakens the field's role as the leaf's acceptance contract.
