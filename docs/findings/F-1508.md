## Finding

The binder pages bind the `params`-validation step to the concrete dependency name *AJV* throughout normative prose. `binder-bypass-and-envelope.md` says the runtime "AJV-validates the merged result" on the `ok` envelope, the relaxed-params paragraph says defaults are filled "before AJV validates the merged result", and the system-prompt section says the binder's `args` are validated against the relaxed copy by AJV (item 1). `defaulting-system-note-echo.md` names the post-default-merge hook **"post-default-merge AJV validation"** (anchor `#post-default-merge-ajv-validation`), describing it as "the `SchemaValidator.validate()` call that AJV-validates the merged `args` object". `determinism-cancellation-failure.md` coins the **AJV-on-`args` class** in the failure-class taxonomy and uses "AJV validation of the binder's `args` failed" as the failure-mode-template row label. The same coinage cascades into `hard-ceilings/ceilings-3-and-4.md` (HC3-c, the ceiling-4 table) and `pi-integration-contract/runtime-event-channel.md` (the per-site reachable-mask table).

The behavioural requirement at every one of these sites is JSON-Schema validation of the merged `args` against the lowered params schema, dispatched through the injected `SchemaValidator` service. The spec already pins this elsewhere: `implementation-notes.md` demotes AJV to a "non-normative implementation hint" behind the `SchemaValidator` seam; `errors-and-results/queryerror-variants.md` states that `validation_errors` "isolates Loom's surface from the AJV API: a future validator swap is not a breaking change"; `determinism-cancellation-failure.md`'s own `<ajv-summary>` paragraph describes the placeholder content as **"validator-neutral"**, parenthesising AJV as a non-normative reference. The binder pages' AJV-named prose contradicts these guarantees — it elevates an internal implementation choice into the operator-visible vocabulary (`AJV-on-args`, the `#post-default-merge-ajv-validation` anchor, the failure-template row label) and into REQ-ID-citable inline labels (HC3-c).

The defect is sprawl, not divergence: behaviour is not in dispute, but normative wording and a load-bearing anchor name a swappable dependency. The same cleanup has already been applied successfully to two adjacent surfaces (runtime bullets in `implementation-notes.md` and the `<ajv-summary>` rendering paragraph), so the precedent and shape of the fix are established.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `9a53d9c` (initial scaffold seeded "JSON Schema validation … uses **AJV** or equivalent"); `fecb504` ("Split spec.md and plan.md into per-topic / per-phase files") propagated "AJV-validates" / "AJV validation" wording into the new per-topic `spec_topics/binder.md`; `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap") split that file into `binder/binder-bypass-and-envelope.md`, `binder/defaulting-system-note-echo.md`, and `binder/determinism-cancellation-failure.md`, carrying the AJV-named prose and creating the `#post-default-merge-ajv-validation` anchor and the AJV-on-`args` class label.
**History:** Two later commits cleaned the same pattern on adjacent surfaces but did not reach the binder pages: `9d10d82` ("Failure-mode templates — `<ajv-summary>` rendering pinned to a concrete AJV call") rewrote the `<ajv-summary>` rendering paragraph as "validator-neutral … independent of which validator produced the issues" while leaving the surrounding "AJV-on-`args`" classification language and the AJV-named row label in place; `6866193` ("Runtime bullets name AJV in normative prose") replaced "AJV-validates" with the `SchemaValidator` service in `implementation-notes.md`'s typed-query and `params`-validation bullets, but applied the change only to that file. The validator-neutrality guarantee then crystallised in `errors-and-results/queryerror-variants.md` ("a future validator swap is not a breaking change"). The current defect is the residue of those two partial cleanups: the validator-neutral promise is in place, the runtime bullets and `<ajv-summary>` rendering have been migrated, but the binder pages (and the cross-page citations into them, including HC3-c) still carry the original AJV-named coinage.

## Solution Space

**Shape:** single

### Recommendation

Apply the same `implementation-notes.md` / `<ajv-summary>` cleanup to the binder pages and their cross-references, in one coherent edit:

1. **`binder-bypass-and-envelope.md`** — replace "AJV-validates", "AJV validates the merged result", and "AJV validation still runs as a safety net" with "validates via `SchemaValidator.validate()`" (or equivalent, with a link to the seam contract). In the system-prompt item-1 paragraph, replace "validated against the relaxed copy by AJV" with the validator-neutral form.
2. **`defaulting-system-note-echo.md`** — rename the hook from **"post-default-merge AJV validation"** to **"post-default-merge schema validation"** (or "post-default-merge `SchemaValidator.validate()` hook"). Rename the anchor from `#post-default-merge-ajv-validation` to `#post-default-merge-validation` (keep the old anchor as a duplicate `<a id>` for one release per the spec's anchor-retirement convention if such a convention exists; otherwise repoint inbound links atomically in the same commit). Rewrite "the `SchemaValidator.validate()` call that AJV-validates the merged `args`" as "the `SchemaValidator.validate()` call that validates the merged `args`".
3. **`determinism-cancellation-failure.md`** — rename **AJV-on-`args` class** to **schema-validation-on-`args` class** (or **validation-on-`args` class**) in the failure-class taxonomy, the per-invocation-retry-budget paragraph, the abort-disposition paragraph, and the failure-mode-templates row label ("AJV validation of the binder's `args` failed" → "schema validation of the binder's `args` failed"). Where AJV is still referenced behaviourally (e.g. "AJV did not run at this site" in the depth-walk fast-fail clause), replace with "the validator did not run".
4. **`hard-ceilings/ceilings-3-and-4.md`** — update HC3-c's text from "AJV-on-`args` failures" to "schema-validation-on-`args` failures" (the inline label `HC3-c` itself is stable and does not change). Update the ceiling-4 table row's "binder's AJV-on-`args` arm" prose and the cross-reference target to the renamed anchor.
5. **`pi-integration-contract/runtime-event-channel.md`**, **`schema-subset.md`**, **`errors-and-results/error-model.md` (ERR-16)** — repoint the "AJV-on-`args` arm" / "Slash-load `params` AJV" wording onto the renamed class.
6. **Plan co-edits** — repoint `V11g`'s Adds/Ships-when/heading, `V11g-T`'s heading, `V11f`/`V11f-T`'s "AJV-on-args" citations, `H7a`'s leaf citation, and `coverage-matrix.md` row `cka-40` (the cited anchor and surrounding prose) onto the renamed anchor and class label. The leaf filenames need not change (the slug `defaulting-revalidation` is already validator-neutral).

In every site, leave AJV exactly one mention — confined to the existing non-normative parenthetical ("the loom 1.0 reference produces this string via the AJV helper `errorsText(...)`") and the existing `implementation-notes.md` implementation hint. Do not invent new "AJV" mentions while doing the cleanup.

Edge cases the implementer must watch:
- The `<ajv-summary>` placeholder token itself is out of scope for this finding (a separate finding, "`<ajv-summary>` placeholder named after a swappable dependency", owns that rename). Do not collide with it.
- HC3-a/HC3-b/HC3-c/HC3-d/HC3-e are stable inline labels; rename only the *text* of HC3-c, never the label.
- The anchor rename in `defaulting-system-note-echo.md` has eight inbound cross-references (plan_topics included); the rename must repoint all of them in the same commit to keep the corpus link-check green.
- The "AJV validates the merged result" claim in `binder-bypass-and-envelope.md` was last touched by commit `043d465` ("ok-envelope validate-vs-fill ordering contradiction") to fix the ordering relative to default-fill; preserve the *post-fill* ordering it established when rewording.

## Consequence

**Severity:** advisory

The behaviour is not at risk — every consumer ends up calling `SchemaValidator.validate()` either way, and a validator swap remains internally feasible. The cost is naming drift: an "AJV-on-`args`" failure class, an HC3-c inline label, a `#post-default-merge-ajv-validation` anchor, and an operator-visible failure-template row all bake the dependency name into surfaces the rest of the spec promises are validator-neutral, raising the cost (and contradicting the in-corpus claim) that a future validator swap is not a breaking change.
