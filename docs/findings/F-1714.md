## Problem

The cascade-twin re-emission rule under **Deduplication and lifetime rules** in `runtime-event-channel.md` describes the boundary emission two incompatible ways: one sentence says the same `RuntimeEvent` instance is re-emitted verbatim (reuse-the-reference), while a later sentence and PIC-1 clause (f) say re-emissions MUST copy the originating instance verbatim (make-a-copy). The spec never declares the payload immutable, so an implementer cannot resolve the reference-vs-copy tension from the text alone. Separately, "copy … verbatim" does not state copy depth: `RuntimeEvent` carries nested `query_site`/`discard_site` objects and the `masked` array, and shallow-vs-deep copy is left unpinned. Because the dedup tuple `(kind, query_site, message, occurred_at)` is keyed on the object-valued `query_site`, a consumer comparing it by reference would behave differently depending on the runtime's copy strategy.

## Issue introduction

**Verdict:** introduced by a specific commit (sharpened by a later one)
**Introducing commits:** `bc50e69` (2026-05-05, "pi-loom spec: resolve \"H2 Clock seam — … RuntimeEvent occurred_at\""); wording tension sharpened by `d75f2a8` (2026-06-06, "pi-loom spec: resolve \"Runtime-event channel occurrence/origin, observable PIC-1 (e), mask-domain placement\"").
**History:** `git log -S "rather than re-stamping"` and `git log -S "byte-identical \`RuntimeEvent\`"` on the corpus show the verbatim-copy obligation ("Re-emissions for symmetry MUST copy the originating `RuntimeEvent` instance verbatim — including `occurred_at` — rather than re-stamping") first entered the normative spec in `bc50e69`, in the pre-split `pi-integration-contract.md`. It pinned the byte-identical target but never specified reference-vs-copy or copy depth — the ambiguity has been present since that commit. `git show d75f2a8` shows the bullet later rewritten to lead with "the same `RuntimeEvent` instance is re-emitted verbatim", introducing the reuse-by-reference phrasing that now sits in tension with the unchanged "MUST copy … rather than re-stamping" clause. Both the depth gap and the wording tension are properties of the current text, not of any single isolated edit, but the originating defect is `bc50e69`.

## Solution approach

Rewrite the cascade-twin re-emission bullet and align PIC-1 clause (f) so the obligation is stated once: the boundary re-emission carries a payload byte-identical to the origin in every field — including the nested `query_site`/`discard_site` objects, `masked`, and `occurred_at` — with no field re-computed or re-stamped at the boundary, produced either by re-emitting the same instance or by a structural copy. Remove the "same instance" / "copy" wording tension so the two readings collapse to the single byte-identity obligation.
