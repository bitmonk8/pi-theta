## Finding

The cascade-twin re-emission rule (under **Deduplication and lifetime rules**, `runtime-event-channel.md`) describes the boundary emission two incompatible ways in the same paragraph. The first sentence says "the same `RuntimeEvent` instance is re-emitted verbatim once more at the boundary" — a reuse-the-reference reading. The last sentence says "Re-emissions MUST copy the originating `RuntimeEvent` instance verbatim — including `occurred_at` — rather than re-stamping" — a make-a-copy reading. PIC-1 clause (f) reinforces the copy reading ("MUST copy the originating instance verbatim including `masked`"). Reusing the same instance and copying it are different operations, and the spec never declares the payload immutable/frozen, so an implementer cannot resolve the tension from the text alone.

Compounding this, "copy … verbatim" does not state copy depth. `RuntimeEvent` carries nested objects — `query_site` / `discard_site` (`{file,line,column}`) and the `masked` string array. A shallow spread shares those nested references with the origin; a deep copy clones them. The paragraph does pin the observable target ("both carry byte-identical `RuntimeEvent` payloads"), which settles field values, but it does not say whether the copy is by-reference, single-level, or deep, nor whether any field may be re-computed. Because the dedup tuple `(kind, query_site, message, occurred_at)` is keyed on an object-valued field (`query_site`) and consumer dedup is explicitly left non-normative, a consumer comparing `query_site` by reference rather than by value would behave differently depending on which copy strategy the runtime picked.

The fix is to state the exact obligation in one place: the boundary re-emission carries a payload byte-identical to the origin in every field (including nested `query_site`/`discard_site` and `masked` and `occurred_at`), produced either by re-emitting the same instance or by a structural copy, with no field re-computed or re-stamped — and to remove the "same instance" / "copy" wording tension.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the cascade-twin re-emission bullet (and align PIC-1 clause (f)) so the obligation is stated once and unambiguously:

- Drop the "the same `RuntimeEvent` instance is re-emitted verbatim" phrasing in favour of a single statement: the boundary re-emission's payload MUST be byte-identical to the origin payload in **every** field — `kind`, `code`, `loom`, `invocation_id`, `query_site`, `discard_site`, `message`, `attempts`, `tokens_used`, `masked`, and `occurred_at`.
- State that no field is re-computed, re-stamped, or re-derived at the boundary (the origin is the sole authoritative source — this already holds for `occurred_at` and `masked`; extend it explicitly to all fields including the nested `query_site`/`discard_site` objects).
- Permit either implementation mechanism (re-emit the same instance, or a structural copy) since both satisfy byte-identity; make explicit that the choice is unobservable precisely because every field, including nested objects, must match.
- Optionally note that, given the above, consumer dedup on `query_site` is a structural (value) comparison, since reference identity is not guaranteed.

Edge cases for the implementer: `masked` (array) and `query_site`/`discard_site` (objects) are the only non-scalar fields and are the cases a shallow copy would share by reference — byte-identity covers them either way; ensure `occurred_at` is carried, never re-stamped via `Clock.wallNow()` at the boundary.

## Consequence

**Severity:** advisory

Implementers can still ship a working system because the byte-identical-payload target is stated, but the same-instance-vs-copy wording tension and the unstated copy depth leave a guidance gap. The only place divergence becomes visible is reference-based consumer deduplication on the object-valued `query_site` field, which the spec marks non-normative.
