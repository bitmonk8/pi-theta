## Finding

The `mode: subagent` bullet in overview.md ("Scope of a loom file") states that
when a subagent loom finishes, "the intermediate transcript is private to the
loom and is **not retained by the runtime** after the loom returns." The clause
bundles two assertions of different epistemic status. The privacy half — the
caller's conversation never receives the loom's intermediate query turns, and an
`invoke` parent gets back only the typed final value — is an externally
observable guarantee that conformance tests can witness. The "not retained by the
runtime" half is an internal-memory-lifetime claim: no loom-level API exposes the
spawned session's transcript, so no test can distinguish a runtime that actively
discards the transcript from one that merely lets it become unreachable and
garbage-collected.

As written, the sentence reads as a normative guarantee that an implementer must
satisfy, yet the only part of it that can be checked is the privacy/isolation
property. The unfalsifiable retention clause invites implementers to believe they
owe an active free/zeroise step that the spec cannot test for and that no observer
can detect, while the genuinely testable obligations sit alongside it without
being called out as the conformance surface.

The orientation page already defers any transcript-surfacing capability (testing,
replay, observability) to Future Considerations, so the retention clause is not
load-bearing for any 1.0 behaviour; it is an implementation aside dressed as a
guarantee. It should be narrowed to the verifiable scope, with the retention
detail demoted to a non-normative note or dropped.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the `mode: subagent` bullet's guarantee clause to assert only the
observable scope, and demote the retention detail. Concretely, replace "the
intermediate transcript is private to the loom and is not retained by the runtime
after the loom returns" with a statement of the two testable properties: the
loom's intermediate query turns never appear in the caller's conversation, and
the `invoke` return surfaces only the typed final value (not the transcript).

If the in-memory-discard behaviour is worth recording at all, move it to a
non-normative aside (e.g. an italic "the runtime holds no reference to the
spawned session's transcript after return" note) so it is clearly not a
conformance obligation, or drop it entirely — the existing forward-reference to
the deferred `loom test` surfacing capability already covers the "no transcript
exposed" intent. Keep the existing Future Considerations link intact.

## Consequence

**Severity:** advisory

An implementer reading the bullet may infer an untestable obligation to actively
free the transcript and waste effort on a step no observer can verify, while the
actually-checkable guarantees (no intermediate turns leak to the caller; the
invoke return carries only the typed value) are not flagged as the conformance
surface. No divergence in observable behaviour results, but the guidance is
unfalsifiable as stated.
