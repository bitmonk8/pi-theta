## Finding

`docs/spec_topics/query/query-forms.md` §"Degenerate rendered templates" defines `loom/parse/empty-template` to fire when a template's *static* body — "every literal segment between interpolations, after newline-trim and dedent" — is empty or whitespace-only. The same bullet then tells authors who genuinely intend a whitespace-only prompt that they "can suppress the warning by writing an explicit literal escape (`` \n ``)." Nothing in the section says whether the static-body emptiness/whitespace predicate is evaluated against the source-level literal segments (the bytes between the backticks, with `` \n ``, `` \t ``, etc. still in their escaped two-character form) or against the post-escape literal segments (the same segments after `query-escapes-stringification.md` rewrites `` \n `` → U+000A, `` \t `` → U+0009, and so on).

The two readings diverge on the exact example the spec cites as the suppression mechanism. Take `` @`\n` ``:

- **Pre-escape reading.** The static body is the two-character sequence `\n`; backslash is not in the pinned ASCII-whitespace set, so the body is non-whitespace, the warning is suppressed, and the author's intent stands. The spec's suppression hatch works.
- **Post-escape reading.** The static body is a single U+000A; that *is* in the pinned ASCII-whitespace set, so the body is whitespace-only, the warning fires, and the author has no way to suppress it short of inserting a non-whitespace character — contradicting the bullet that just told them `` \n `` would suffice.

Conforming implementers reading the same paragraph will therefore disagree about whether a vector as fundamental as `` @`\n` `` warns. The order relative to escape processing must be pinned, and (because the spec already commits to `` @`\n` `` as the suppression hatch) only the pre-escape order is internally consistent.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `82be125` ("spec: adopt P1 typed-query two-phase loop (Option A)", 2026-05-04)
**History:** Commit `82be125` is the first commit to add a "Degenerate rendered templates" section anywhere in the spec corpus (`git log -S 'explicit literal escape' --reverse`, `git log --diff-filter=A -- docs/spec_topics/query/query-forms.md`, and a `git log -L` walk of the section all bottom out at this commit). The introduced text already contained both halves of the contradiction in their current form: the "static body … after newline-trim and dedent" predicate and the "suppress … by writing an explicit literal escape (`` \n ``)" hatch, with no statement of order relative to escape processing. The later commit `f14a45e` pinned the *whitespace set* (ASCII vs `\s`) but did not touch the pre-/post-escape order. The defect has been present since the section was first authored.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/query/query-forms.md`, amend the "Parse-time warning" bullet to pin the predicate's input as the *source-level* literal segments — i.e. evaluated **before** the escape rewrites defined in `query-escapes-stringification.md` are applied (newline-trim and dedent still apply first to the surrounding template, but escape sequences inside the literal segments are not yet expanded when the emptiness/whitespace predicate runs).

A minimal edit at the bullet's parenthetical clause is sufficient — for example, recasting "every literal segment between interpolations, after newline-trim and dedent" as "every literal segment between interpolations, after newline-trim and dedent and **before** escape processing (so a segment whose only characters are `` \n ``, `` \t ``, `` \\ ``, `` \` ``, or `` \$ `` is non-whitespace for this predicate's purposes)". The clarification should explicitly call out that this is why `` @`\n` `` suppresses the warning even though its *rendered* text is a single newline that does trigger the runtime short-circuit; the two passes inspect different inputs by design (parse-time predicate is pre-escape, runtime short-circuit is post-render).

Edge cases the implementer must watch:

- The runtime short-circuit's input is unchanged — it remains the fully-rendered, post-escape, post-trim, post-dedent text. The author's `` @`\n` `` therefore loads without a parse warning and then short-circuits at query time to `ValidationError{cause: "empty_template", attempts: 0}`. That is the intended observable: the author asked for a whitespace-only prompt, the parser believed them, and the runtime still refused to spend a provider round-trip on it.
- Multi-segment templates such as `` @`${x}\n${y}` `` have two literal segments (`""` and `\n` and `""`); the predicate runs over each segment's source bytes, so the `\n` segment is non-whitespace and suppresses the warning, matching the single-segment case.
- The pinned ASCII-whitespace set (per the `defaulting-system-note-echo.md` rule 1 cross-reference already in the bullet) continues to govern *which characters count as whitespace* for both passes; only the *input* to the parse-time pass changes.

## Consequence

**Severity:** correctness

Two reasonable implementers reading the current text produce observably divergent parsers for the smallest non-empty template the spec itself names as the canonical suppression escape hatch. The defect is localised (one diagnostic, one example), but it sits at the boundary between two pinned passes — escape processing and the empty-template predicate — and any implementer who picks the post-escape reading silently breaks the documented author workflow.
