## Finding

`V9j`'s Adds names the full per-provider seed-field table as part of the `complete()` inference call, and the spec's `provider-error-mapping.md` §Provider seed-field mapping pins a four-row table with three materially distinct request behaviours: `openai-completions` carries the seed under the key `seed`, `mistral` carries it under `random_seed`, and both `anthropic-messages` and `amazon-bedrock` carry no seed key at all.

The classifier Tests bullet fully exercises the error-classification table ("maps **each** provider response to its `QueryError` variant"), but the request-envelope Tests bullet asserts only that `complete()` is issued with "the mapped seed field" — singular. As written, the acceptance gate (Ships when: "the deterministic `complete()` envelope") is satisfied by verifying a single provider row. The differing field name for `mistral` (`random_seed` vs `seed`) and the seed-omission behaviour for `anthropic-messages` / `amazon-bedrock` are left unverified.

A runtime regression in payload construction — emitting `seed` regardless of provider, or attaching a seed key for an omit-row provider — passes the singular test. The build-time seed-field gate owned by `V18c` (the `Api`-coverage assertion plus the rename/retype/move fixture) verifies the *table constant's* completeness and shape against the SDK; it does not witness that `V9j`'s runtime `complete()` call selects the correct field per provider on the request side. No leaf closes that observation.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (2026-06-10)
**History:** `V9j-binder-inference-provider-mapping.md` and `V9j-T-binder-inference-provider-mapping.md` were both created in `c6a664e` (the plan-build commit) and have no subsequent history (`git log --follow` returns one commit each). `git log -S "the mapped seed field"` confirms the singular request-envelope bullet was present at inception and never revised; the leaf pair has never carried a per-provider request-side assertion.

## Solution Space

**Shape:** single

### Recommendation

Replace the singular request-envelope Tests bullet in both `V9j-T-binder-inference-provider-mapping.md` and `V9j-binder-inference-provider-mapping.md` (the bullet currently reading "`complete()` is issued with the forced-tool envelope, `temperature:0`, and the mapped seed field.") with a parametrised assertion that ranges over every row of the `provider-error-mapping.md` §Provider seed-field mapping table: for a resolved `openai-completions` model the `complete()` `options` carries `seed`; for `mistral` it carries `random_seed`; for `anthropic-messages` and `amazon-bedrock` `options` carries no seed key. Keep the forced-tool-envelope and `temperature:0` assertions in the same bullet or alongside it. Author the failing form in `V9j-T` first (per the paired-leaf TDD ritual), then mirror it in `V9j`. Optionally update `V9j`'s Ships when to name the per-provider envelope ("the deterministic `complete()` envelope, per provider seed-field row") so the exit gate reflects the widened coverage. The seed-field table is spec-owned in `provider-error-mapping.md`; do not restate the table in the plan and do not edit the spec.

## Consequence

**Severity:** advisory

The implementation guidance is unambiguous (the spec table is explicit), so a working leaf can still be produced, but the leaf's acceptance gate verifies only one of the four seed-field rows. A per-provider request-side regression — wrong field name for `mistral`, or a spurious seed key for the two omit-row providers — ships undetected, since neither this bullet nor the `V18c` table-constant gate observes the runtime emission per provider.
