## Problem

The `GOV-N (governance rule)` entry in `docs/spec_topics/glossary.md` states the live range is "currently running through `GOV-23`, with some IDs retired". The authoritative range registered on `docs/spec_topics/governance.md` runs through `GOV-31`, so the glossary literal is wrong by eight IDs. The same entry names `governance.md` as authoritative and tells readers to "cite `governance.md` for the authoritative range rather than a fixed upper bound", yet contradicts it on a number, so a reader treating the glossary as orientation will believe no `GOV-N` above 23 exists and read in-corpus citations of `GOV-24`…`GOV-31` as dangling.

## Issue introduction

**Verdict:** stale-after-extension
**Introducing commit:** `96cfc08` (2026-06-04, "pi-loom spec: resolve \"Glossary GOV-N entry under-describes the live GOV range and scope\"")
**Staleness-introducing commits:** `1d773d6` (2026-06-04, added `GOV-24`), `8e3ccb4` (2026-06-05, added `GOV-25`–`GOV-29` from the `GOV-21` split), `fe8aa5c` (2026-06-05, added `GOV-30`/`GOV-31` from the `GOV-12` split)
**History:** `git log -S 'through ``GOV-23``' -- docs/spec_topics/glossary.md` shows the literal entered the glossary at `96cfc08`, replacing an earlier "(`GOV-1` through `GOV-8`)" wording. At that commit `GOV-23` was the live maximum, so the literal was correct on the day it was written. Three later commits — `1d773d6`, `8e3ccb4`, `fe8aa5c` — each extended `governance.md` with new live `GOV-N` IDs without co-updating the glossary literal, even though `96cfc08`'s own forward-pointing clause ("cite `governance.md` for the authoritative range rather than a fixed upper bound") was authored precisely to discourage that literal. The defect is therefore drift across a sequence of single-page edits, not a single mis-statement; the authoring page anticipated the failure mode and embedded a hedge instead of removing the literal.

## Solution approach

Rewrite the `GOV-N (governance rule)` entry in `docs/spec_topics/glossary.md` to remove the fixed numeric upper bound, deferring entirely to `governance.md` for the live set while retaining the existing forward-link to [Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids) for the retired sub-list. This aligns the entry with its own existing "cite `governance.md` for the authoritative range" clause.

## Solution constraints

- MUST NOT reintroduce a fixed numeric upper bound (e.g. restating the maximum as `GOV-31`); doing so recreates the same per-allocation lock-step drift vector this finding addresses.
