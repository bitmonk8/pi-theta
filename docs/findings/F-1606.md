## Finding

The `GOV-N (governance rule)` entry in `docs/spec_topics/glossary.md` states the live range is "currently running through `GOV-23`, with some IDs retired". The authoritative range, as actually registered on `docs/spec_topics/governance.md` and its `governance/` sub-pages, runs through `GOV-31`: `GOV-24` (per-row invariant), `GOV-25`–`GOV-29` (dual-anchor convention for legacy-token HTML anchors), and `GOV-30`/`GOV-31` (`spec.md` aggregator lock-step and integer-count literal preservation) are all live, with `<a id="gov-…">` anchors on their owning pages. The glossary literal is therefore wrong by eight IDs.

The entry partially insulates itself — its next clause tells readers to "cite `governance.md` for the authoritative range rather than a fixed upper bound" — but that self-deferral does not undo the false numeric claim. A reader who treats the glossary as orientation (which the glossary's own preamble invites them to) will believe no `GOV-N` above 23 exists, and will read any in-corpus cross-link to `GOV-24` … `GOV-31` as a dangling reference. The entry simultaneously names governance.md as authoritative and contradicts it on a number, which is the worst of both options: a normative-looking literal that the entry itself disclaims.

## Issue introduction

**Verdict:** stale-after-extension
**Introducing commit:** `96cfc08` (2026-06-04, "pi-loom spec: resolve \"Glossary GOV-N entry under-describes the live GOV range and scope\"")
**Staleness-introducing commits:** `1d773d6` (2026-06-04, added `GOV-24`), `8e3ccb4` (2026-06-05, added `GOV-25`–`GOV-29` from the `GOV-21` split), `fe8aa5c` (2026-06-05, added `GOV-30`/`GOV-31` from the `GOV-12` split)
**History:** `git log -S 'through ``GOV-23``' -- docs/spec_topics/glossary.md` shows the literal entered the glossary at `96cfc08`, replacing an earlier "(`GOV-1` through `GOV-8`)" wording. At that commit `GOV-23` was the live maximum, so the literal was correct on the day it was written. Three later commits — `1d773d6`, `8e3ccb4`, `fe8aa5c` — each extended `governance.md` with new live `GOV-N` IDs without co-updating the glossary literal, even though `96cfc08`'s own forward-pointing clause ("cite `governance.md` for the authoritative range rather than a fixed upper bound") was authored precisely to discourage that literal. The defect is therefore drift across a sequence of single-page edits, not a single mis-statement; the authoring page anticipated the failure mode and embedded a hedge instead of removing the literal.

## Solution Space

**Shape:** single

### Recommendation

Drop the fixed upper bound from the entry. Replace "currently running through `GOV-23`, with some IDs retired — see [Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids)" with prose that defers entirely to `governance.md` for the live set and to the retirement registry for the retired set — for example: "with some IDs retired; see `governance.md` for the live set and [Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids) for the retired sub-list." The entry's existing next clause ("cite `governance.md` for the authoritative range rather than a fixed upper bound") already commits the glossary to this posture, so removing the literal aligns the entry with its own guidance and pre-empts the same drift on the next `GOV-N` allocation.

Re-stating the current maximum as `GOV-31` is rejected: it reintroduces the same lock-step burden that produced this defect (every future `GOV-N` allocation would have to remember to co-edit the glossary), and the entry's authoring commit explicitly chose the deferral framing over a literal upper bound.

## Consequence

**Severity:** advisory

A reader trusting the glossary's literal upper bound concludes that `GOV-24`…`GOV-31` are unallocated and treats in-corpus citations of those IDs as dangling. Because the glossary entry already names `governance.md` authoritative, a careful reader who follows the forward-link recovers correctness, but the defect costs them a hop and contradicts a hub the spec mechanically depends on. No implementer is blocked, no runtime behaviour changes.
