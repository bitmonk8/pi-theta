## Problem

The *Per-family record-shape table* in `audit-failures.md` pins the `proposed-resolution` field for violation families (1) (`pi.<member>`), (2) (peer/typebox import), and (3) (`ctx.<member>`) as two arms joined by "or" — the `SDK_SURFACE_INVENTORY` promote entry per `#bump-step-2b-promote` *or* the `// allow-pi-surface:` exemption marker per `#bump-step-2b-exempt` (families (2)/(3) inherit this via "as for family (1)"). The cell gives no rule for which arm the audit emits and no permission to emit both, so the field value is under-determined. The other families in the same table are deterministic: family (4) names a single arm, and family (5) selects by the marker's observed state. The *Wire serialisation* clause in `audit-wire-and-canary.md` requires byte-identical stdout across runs against the same tree, which this either/or defeats — two reasonable implementers may emit promote-only, exempt-only, or both for the identical violation.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commit:** `e5cd514` (2026-05-09) — "pi-loom spec: resolve T35 SDK capability inventory closed-set negative-direction audit"
**History:** The *Per-family record-shape table* was created in `e5cd514`, the single commit that introduced the entire inventory-closure audit. The family-(1) `proposed-resolution` cell carried the "…per branch (2), **or** the canonical `// allow-pi-surface:` marker per branch (3)" either/or from that first revision, with families (2)/(3) deferring to it via "as for family (1)". `git log -S "inventory-entry-suggestion"` and `git log -S "Per-family record-shape table"` both bottom out at `e5cd514`; the later commit `8c5a126` only attached the `#audit-per-family-record-shape-table` anchor and never touched the field value. No commit ever narrowed the field to a single arm or pinned a selection rule, so the non-determinism has been present since the audit's first drop rather than introduced by a regression.

## Solution approach

Rewrite the family (1) `proposed-resolution` cell in the *Per-family record-shape table* so the field is one content-invariant value that names both the promote arm (`#bump-step-2b-promote`) and the exempt arm (`#bump-step-2b-exempt`) by anchor reference, rather than an either/or runtime selection. State that this is a single byte-identical string emitted for every family-(1)/(2)/(3) record. Leave the "as for family (1)" references in rows (2)/(3) pointing at the revised cell.
