## Finding

The *Per-family record-shape table* in `audit-failures.md` pins the `proposed-resolution` field value for violation families (1) (`pi.<member>`), (2) (peer/typebox import), and (3) (`ctx.<member>`) as: "the `SDK_SURFACE_INVENTORY` entry the contributor would add per step 2(b) branch (2), **or** the canonical `// allow-pi-surface:` marker per step 2(b) branch (3)" (families (2)/(3) inherit this via "as for family (1)"). A violation record carries exactly one `proposed-resolution` field — segment (3) of the four tab-delimited tokens packed onto each stdout line per the *Wire serialisation* clause in `audit-wire-and-canary.md`. The cell names two arms joined by "or" but gives no rule for which one the audit emits, and no permission to emit both. The selection criterion is simply absent.

This contrasts with the other families in the same table, which are deterministic: family (4) names a single arm (branch (5) rewrite-shape), and family (5) selects its value by the marker's observed state ("naming whichever of that branch's five sub-cases the marker falls under"). Only families (1)/(2)/(3) leave the field as a free choice between two arms with no resolving rule.

The `proposed-resolution` field is normatively pinned — the table states a record "that carries a value outside the per-family range below for a given family, violates this rule" — yet the "range" for these three families is ambiguous as to whether it admits the promote arm, the exempt arm, both, or a choice. The *Wire serialisation* rule additionally requires byte-identical output across runs against the same tree; an unspecified field value undermines that contract and leaves two reasonable implementers free to emit different strings (promote-only, exempt-only, or both) for the identical violation.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commit:** `e5cd514` (2026-05-09) — "pi-loom spec: resolve T35 SDK capability inventory closed-set negative-direction audit"
**History:** The *Per-family record-shape table* was created in `e5cd514`, the single commit that introduced the entire inventory-closure audit. The family-(1) `proposed-resolution` cell carried the "…per branch (2), **or** the canonical `// allow-pi-surface:` marker per branch (3)" either/or from that first revision, with families (2)/(3) deferring to it via "as for family (1)". `git log -S "inventory-entry-suggestion"` and `git log -S "Per-family record-shape table"` both bottom out at `e5cd514`; the later commit `8c5a126` only attached the `#audit-per-family-record-shape-table` anchor and never touched the field value. No commit ever narrowed the field to a single arm or pinned a selection rule, so the non-determinism has been present since the audit's first drop rather than introduced by a regression.

## Solution Space

### Recommendation

In the *Per-family record-shape table* in `audit-failures.md`, make the `proposed-resolution` field for family (1) a single fixed plain-ASCII string that names both branch (2) [promote] and branch (3) [exempt] as the available constructive arms, by reference (mirroring family (5)'s by-reference suggestion). The field is one byte-identical, content-invariant value for every family-(1)/(2)/(3) record, not a runtime selection.

In the family (1) cell, replace the "…or the canonical `// allow-pi-surface:` marker per branch (3)" choice with a fixed literal, e.g. `inventory-entry-or-exemption suggestion: promote per #bump-step-2b-promote or exempt per #bump-step-2b-exempt`, and state explicitly that this is one content-invariant string (not a selection). Leave the "as for family (1)" references in rows (2)/(3) pointing at the revised cell.

The fixed string must be byte-identical across all family-(1)/(2)/(3) records and contain no ASCII tab or newline so the four-field tab split is preserved (it references arms by anchor rather than quoting source, so this holds).

## Consequence

**Severity:** correctness

Two reasonable implementers will produce different `proposed-resolution` strings for the same family-(1)/(2)/(3) violation (promote-only, exempt-only, or both arms concatenated), defeating the *Wire serialisation* byte-identical-output contract and leaving the test invariant the audit pins under-determined. A contributor consuming a single record receives an arbitrary subset of the available constructive arms with no guarantee of which.
