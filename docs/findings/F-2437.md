## Finding

The success-echo formatter (`Echo policy`, `docs/spec_topics/binder/defaulting-system-note-echo.md`) is specified as a set of per-value rendering rules whose output is fixed by the **static** type of each value, not by the runtime JSON alone. Three of its rules cannot be evaluated from a plain bound value:

- **Object first field.** `{first-field-value, …}` renders "the first field listed in the declaring `schema` block's source order." The merged `args` object is keyed by wire name and carries no guarantee that its property order matches the declaring schema's source order (the binder returns the keys in whatever order it emits them, and `as`-renames decouple wire name from loom-side declaration). Picking "the first field" therefore requires the declaring schema's field order, which a runtime value does not carry.
- **Discriminated-union variant.** "For a value whose static type is a discriminated union, the variant's declared fields are used in the variant's own source order." The formatter must (a) know the static type is a union, (b) identify *which* variant the runtime value is, and (c) recover that variant's declared field order. BNDR-6h (`Cat { name: "Whiskers" }` → `{cat, …}`, with the implicit discriminator `kind: "cat"` selected as the first field) only resolves with the union type in hand; nothing in the value alone tells the formatter that `kind` precedes `name`.
- **Inline anonymous object.** The first field is "the leftmost field of the inline type expression as written in the loom source" — pure static-syntax information with no runtime representative at all.

These rules are governed by BNDR-6 reference renderings tagged **MUST-reproduce-exactly**, yet the spec never states that the formatter is handed the loom's static `params:` type descriptors, never says it walks the value tree and the type tree in lockstep, and never names a canonical param-type representation it consults. The only acknowledgement is the parenthetical that the rendered text "depends on the loom's `params:` and the bound values," which neither pins an input nor describes a traversal. An implementer who builds a value-only formatter (driven by JSON key order and JS `typeof`) reproduces the tabulated rows by luck but diverges on any input whose JSON key order differs from schema source order, or on any discriminated-union value whose variant must be identified by its discriminator — and there is no contract telling them they need the static type at all.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `83802db` (2026-05-04, "spec: collapse to single typed argument style; add LLM-driven slash-command binder")
**History:** The corpus is git-tracked. `83802db` introduced the echo formatter already type-dependent: it shipped "Top-level `params:` fields shown in declaration order" and "Object values shown as `{first-field-value, …}`" without any statement of how the formatter obtains the static type. `git log --follow` over `docs/spec_topics/binder/defaulting-system-note-echo.md` shows the type-dependence only deepened — `edc2e1b`/`ec92c87` (2026-05-06/07) pinned the "first field" notion and the single-field case, `800b3f9` (2026-06-05) added the recursive composite-first-field rule, and `71e8313` (2026-06-03) added the inline-anonymous-object leftmost-field rule. No commit on this path ever introduced a "formatter receives the static param type tree / walks value and type in lockstep" input contract, so the gap is an inception-era omission compounded by each later refinement rather than a regression from any single later commit.

## Solution Space

**Shape:** single

### Recommendation

In `Echo policy` (`defaulting-system-note-echo.md`), before the format-rule list, state the formatter's input contract: the formatter is handed the loom's static `params:` type descriptors alongside the merged `args` value, and it traverses the value tree and the type tree **in lockstep** — at each object level the field order is taken from the type (declaring `schema` source order, the selected discriminated-union variant's source order, or the inline-object type expression's leftmost-to-rightmost order), never from the runtime value's key order. Pin variant selection explicitly: for a discriminated-union-typed value the formatter identifies the variant by matching the value's discriminator field against the union's variants, then uses that variant's static field order. Reference the canonical param-type representation the formatter consults (the lowered `params:` type model in `type-system.md` / `schemas.md`) by anchor so the descriptor shape is not re-invented. State that scalar leaves (`integer`/`number`/`boolean`/`null`/enum-as-wire-string) remain a flat value switch — the deliberate enum design in the existing enum bullet stays unchanged; only the object/union/inline-object ordering decisions consume the type tree.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one threads the static param type tree and walks it in lockstep with the value (reproducing the BNDR-6 MUST-reproduce-exactly renderings), the other builds a value-only formatter and produces a different first field whenever the binder's JSON key order, an `as`-rename, or a discriminated-union discriminator decides the answer. The byte-stable echo contract is silently broken on inputs outside the reference table.
