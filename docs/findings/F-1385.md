## Problem

The `session-shutdown-details-conventions` paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` interleaves producer-side obligations (what the loom runtime MUST construct and emit) with consumer-directed MUSTs that bind external operator-tooling and log-pipeline code (what consumers MUST validate, whitelist, discriminate, and key on). The consumer-directed MUSTs constrain code loom does not ship and cannot exercise, so no loom conformance test can witness their violation — they inflate the normative surface without being falsifiable on loom's side. They also blur the actual wire contract, which is exclusively producer-side. The same consumer-directed MUST prose is restated in the per-row Trigger cells of `loom/runtime/cancelled-by-session-shutdown` and `loom/host/session-shutdown-runtime-degraded`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `52b9101` (2026-05-11) — *pi-loom spec: resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"* — introduced the `session-shutdown-details-conventions` paragraph wholesale, including the original consumer-directed MUSTs ("consumers MUST validate the field against the per-code value space", "operator-tooling consumers MUST whitelist `details.event.loom`", "Consumers MUST treat the construction-site fallback … as a separate value-space", "Consumers MUST treat fallback-path emissions as a separate wire shape … MUST NOT validate them against the structured presence-entitlement").
- `63bddcd` (2026-05-22) — *pi-loom spec: resolve cluster "T19a invocation_id correlation key" (+2 members)* — added `details.event.invocation_id` as a second runtime-constructed sibling under the same carve-out, with the matching "a test asserting its presence MUST first discriminate the structured-emission path on the same footing as the `details.event.loom` discrimination rule above" MUST, propagating the consumer-directed pattern that 52b9101 established.

**History:** `git log -S "operator-tooling consumers parsing" -- docs/spec_topics/diagnostics.md` confirms 52b9101 is the introducing commit for the operator-tooling-consumer audience and the closed-`{reason}` strict-validation entitlement; `git log -S "details.event.invocation_id" -- docs/` confirms 63bddcd is the introducing commit for the `invocation_id` sibling and its discrimination MUST. Neither commit was wrong on its own — each correctly pinned a new consumer-observable shape — but the cumulative effect is the untestable-MUST surface this finding flags. `f5e89f4` (2026-06-04) is the diagnostics-shard split that moved the paragraph from `diagnostics.md` into the `diagnostics/diagnostic-shape.md` child file without altering its content; it is not the introducing commit.

## Solution approach

Demote every consumer-directed MUST / MUST NOT in the `session-shutdown-details-conventions` paragraph to advisory language, and apply the same demotion to the matching consumer-directed MUSTs restated in the per-row Trigger cells of `loom/runtime/cancelled-by-session-shutdown` and `loom/host/session-shutdown-runtime-degraded`. Leave the producer-side MUSTs normative — the *Runtime construction obligation*, the per-row `details.event.loom` / `details.event.invocation_id` presence MUSTs, and the construction-shape pins. The closed-set entitlement prose is already framed as entitlements rather than MUSTs and needs no change.

## Solution constraints

- Producer-side MUSTs (the *Runtime construction obligation*, the per-row `details.event.*` presence MUSTs, the construction-shape pins) stay normative; do not demote them.
- Out of scope: the producer-side forbidden-JS-construct list in the *Runtime construction obligation*, owned by the same-cluster "`details.event` runtime-construction obligation over-prescribes JS constructs" finding — do not relitigate the construction wording.
