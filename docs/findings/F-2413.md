## Finding

For the wedged-in-process-budget case the *Wall-clock budget* clause itself names ("A hung audit that does not emit and does not exit"), the only backstop is an out-of-process `npm test`-side timeout, currently stated as a parenthetical ("with the implementation pinning a `npm test`-side timeout") rather than a MUST. An implementation may omit it and still hang `npm test` indefinitely without ever going red.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `e5cd514` ("pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"", 2026-05-09)
**History:** The corpus is git-tracked. `git log -S` over `Wall-clock budget`, `hung audit`, the `npm test`-side-timeout parenthetical, and the harness-existence-deferral sentence all bottom out at a single commit, `e5cd514`, which authored the entire inventory-closure audit apparatus (then in `docs/spec_topics/pi-integration-contract.md`, before the page split in `81bf52a` and the size-cap reflow in `f5e89f4`). The introducing diff shows both defects present at inception: the *Wall-clock budget* clause carried the hung-audit fallback as a non-normative parenthetical from the first line, and the harness-wiring paragraph deferred the gate's existence to `h1-scaffold.md` (later reworded to "implementation choices" in the post-1.0 reframing commit `74633aa`, preserving the defect). No earlier commit touches either clause; both entered the corpus when the audit feature was added.

## Solution Space

**Shape:** single

### Recommendation

In `audit-wire-and-canary.md`'s *Wall-clock budget* / *Pre-emission termination carve-out* clauses, state that the implementation MUST bound the overall run with an out-of-process timeout (e.g. a `npm test`-side / harness-level timeout) that drives a non-zero `npm test` exit when the audit's own in-process budget fails to fire, so a synchronously-wedged audit is fail-closed. Keep the concrete timeout value and mechanism implementation-pinned.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on whether the fail-closed guarantee actually holds. One that reads the parenthetical as advisory ships an audit with no out-of-process timeout; a synchronously-wedged audit then hangs `npm test` forever — never red, defeating the audit's sole purpose silently. One that reads "existence … is an implementation choice" literally ships no `npm test` gate, leaving every fail-closed and version-bump-acceptance obligation bound to a gate that does not exist.
