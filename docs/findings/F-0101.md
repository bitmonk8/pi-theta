## Problem

Spec INV-4 (`docs/spec_topics/invocation.md` §INV-4) defines two distinct normative surfacing outcomes for a `loom/runtime/invoke-depth-exceeded` panic: a top-level overflow surfaces as a Pi system note, and an overflow nested inside an invoke chain surfaces to the parent as `Err(InvokeInfraError { cause: "panic", ... })`. The INV-4 Tests bullet in both `V15b-invoke-depth-cycle.md` and `V15b-T-invoke-depth-cycle.md` joins these two outcomes with "vs", reading as a single either/or choice rather than two separately-required scenarios, and `V15b`'s **Ships when** requires only that the panic fire at the 32-frame boundary. An implementer can therefore satisfy the bullet by asserting whichever surfacing mode their one scenario exercises, leaving the other normative routing path unverified.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review")
**History:** `git log --follow` over both leaf files shows c6a664e as the root commit that created the V15b / V15b-T pair; `git show c6a664e:docs/plan_topics/V15b-invoke-depth-cycle.md` already carries the "a top-level system note vs a nested `InvokeInfraError{panic}`" phrasing in the INV-4 bullet. `git log -S'system note vs'` confirms the token entered at c6a664e and was not introduced by a later edit. Subsequent commits (084f171, e8c1d65) touched neighbouring INV-4 text (the cross-file `.warp fn` clause and NOCEIL-4 annotation) but left the "vs" construction intact.

## Solution approach

Rewrite the INV-4 Tests bullet in both `V15b-invoke-depth-cycle.md` and `V15b-T-invoke-depth-cycle.md` so the two surfacing modes — the top-level Pi-system-note path and the nested `InvokeInfraError{panic}`-to-parent path — read as separately-required scenarios joined with "and" rather than a "vs" either/or. Extend `V15b`'s **Ships when** so it requires the depth-exceeded panic to be observed firing in both surfacing modes, not only at the boundary.

## Solution constraints

- Out of scope: `docs/spec_topics/invocation.md` is read-only — INV-4 already states both modes; the fix only makes the plan's test obligation match it.
- Keep the `V15b-T` INV-4 bullet byte-aligned with the `V15b` INV-4 bullet (the `-T` bullet mirrors the impl bullet).
