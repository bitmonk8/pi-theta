## Finding

The diagnostics shard states several diagnostic-code *trigger* conditions ŌĆö the normative "when does this code fire" contract ŌĆö in terms of the production implementation library rather than the observable behaviour. `code-registry-runtime.md` describes `loom/runtime/watcher-terminated` as firing when "the chokidar file watcher's `error` route fires or it threw such that one or more watched roots stop delivering events", scopes `loom/runtime/validator-cache-collision` to "the per-query AJV compiled-validator cache", and names "an AJV-recompile step" inside the `loom/runtime/registry-swap-failed` trigger. `diagnostic-shape.md`'s transient-toast section keys its arm (1) on "the chokidar watcher itself throw[ing]".

Only the *behaviour* is observable at the surface a conformance test exercises: watcher events reach the runtime through the `FileWatcher` / `FakeFileWatcher` seam ([PIC-14](../pi-integration-contract/host-interfaces-services.md#pic-14)), whose member shape is explicitly illustrative (GOV-18 arm (a)) and which filters chokidar's native event names before they reach the runtime; AJV validation is reached through the `SchemaValidator` service. A test asserting `watcher-terminated` fires drives the `FakeFileWatcher`'s terminal-signal channel ŌĆö it cannot observe "chokidar's `error` route". The trigger cells already carry the behavioural condition alongside the library name ("one or more watched roots stop delivering events"), so the library mention is redundant with ŌĆö and slightly more prescriptive than ŌĆö the condition that actually governs the code.

The library is, separately, already pinned once: chokidar is declared an implementation-side, version-unpinned choice in [Host prerequisites](../pi-integration-contract/host-prerequisites.md) (behind the `FileWatcher` seam) and the watcher-lifecycle posture is owned behaviourally by [PIC-55](../pi-integration-contract/registration-steps.md#pic-55); AJV is the established validator throughout [Schema Subset](../schema-subset.md). The diagnostic-trigger prose should lean on those pins and the seam abstraction rather than re-naming the concrete library at each firing condition.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `c2e278f` (2026-06-25, F-1398 ŌĆö pin per-arm post-failure disposition for transient toasts), `0b195eb` (2026-06-25, F-1299+F-1308 ŌĆö coin PIC-55 watcher post-error posture + `loom/runtime/watcher-terminated`), `d656588` (2026-06-04 ŌĆö resolve "pi.registerTool failure during watcher-driven registry swap"), `2128e5f` (2026-06-07 ŌĆö resolve "Schema-slug collision posture pinned only for registerTool cache")
**History:** No single commit introduced the defect. Each diagnostic that names a library entered in its own unrelated fix commit, and each independently phrased its trigger with the concrete production library: `d656588` added the "AJV-recompile step" wording to `registry-swap-failed`; `2128e5f` scoped `validator-cache-collision` to "the per-query AJV compiled-validator cache"; `c2e278f` keyed the transient-toast arm on "the chokidar watcher itself throwing"; `0b195eb` coined `watcher-terminated` with a "chokidar file watcher's `error` route" trigger in both `code-registry-runtime.md` and `diagnostic-shape.md`. `git log -S` confirms the tokens (`chokidar`, `AJV`, `validator-cache-collision`) first appear in these files at those commits (with the transient-toast chokidar prose pre-dating the `f5e89f4` size-cap shard split, traced back to `c2e278f`). The durable cause is the absence of a behavioural-phrasing convention for diagnostic triggers, which led each code's author to reach for the concrete library name; the fix addresses that convention rather than any one commit.

## Solution Space

**Shape:** single

### Recommendation

Restate each affected trigger condition behaviourally and let the library name live only at its existing pinned-once anchor:

- `loom/runtime/watcher-terminated` (both `code-registry-runtime.md` and `diagnostic-shape.md` arm (1)): phrase the trigger as "the file watcher errored or threw such that one or more watched roots stop delivering events", cross-referencing the `FileWatcher` seam ([PIC-14](../pi-integration-contract/host-interfaces-services.md#pic-14)) and the watcher-lifecycle owner ([PIC-55](../pi-integration-contract/registration-steps.md#pic-55)) for the terminal-signal channel; drop "chokidar" from the firing condition.
- `loom/runtime/validator-cache-collision`: scope it to "the per-query compiled-validator cache" and reference the `SchemaValidator` / PIC-11 validator-cache rule; the AJV binding is already pinned in [Schema Subset](../schema-subset.md).
- `loom/runtime/registry-swap-failed`: replace "an AJV-recompile step" with "a validator-recompile step".

Keep a single library binding named where it already lives ŌĆö chokidar in [Host prerequisites](../pi-integration-contract/host-prerequisites.md) behind the `FileWatcher` seam, AJV in [Schema Subset](../schema-subset.md) ŌĆö so no firing condition repeats the implementation choice. Leave PIC-14/PIC-55's own descriptive "production wires a chokidar adapter" mentions intact: those are seam-binding documentation, not diagnostic-trigger contract. Edge case: do not weaken the observable byte-exact `message`/`content` templates while rephrasing the surrounding trigger prose ŌĆö only the trigger condition changes wording, not the rendered diagnostic strings.

## Consequence

**Severity:** advisory

Naming the production library in normative trigger prose ties the testable "when does this code fire" contract to an implementation detail that tests reach only through the `FileWatcher` / `SchemaValidator` seams; as worded, the trigger cannot be asserted from the seam surface, and a watcher- or validator-library swap would read as invalidating the diagnostic contract though only behaviour is observable. No divergence in shipped behaviour results ŌĆö the behavioural condition is already present alongside the library name ŌĆö so this is a guidance/testability-phrasing gap rather than a correctness defect.
