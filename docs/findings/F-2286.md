## Finding

The `V9d`/`V9d-T` Tests block carries the bullet *"Each always-log kind emits exactly once; the success side honours the null-policy."* It asserts the exactly-once emission guarantee across the always-log set but neither enumerates that set nor cross-references the spec section that closes it. The spec (`runtime-event-channel.md` §"Runtime event channel") defines the always-log set as a closed enumeration: **group A** (`transport`, `model_tool`, `code_tool`, `tool_loop_exhausted`, `invoke_infra`, and every row of [Binder — Failure modes]) routed through `details: { event: RuntimeEvent }`, and **group B** (every `loom/runtime/*` panic code *except* the five `console.error`-routed teardown codes) routed through `details: { diagnostics: Diagnostic[] }`.

Because the bullet names no members and points at no spec anchor, the natural test realisation is to iterate whatever set the implementation under test exposes. That is tautological: a member silently dropped from the implementation's set definition is also dropped from the test's iteration, so the missing member never reds. The per-member exactly-once guarantee is therefore unverified — the test confirms only that whatever the implementation thinks is in the set emits once, not that the spec's full membership does.

The adjacent group-A/B single-shape routing bullet in the same Tests block already anchors to "the group-A / group-B partition" with a spec link; the exactly-once bullet is the one member of the block that does not.

## Solution Space

**Shape:** single

### Recommendation

In both `V9d` and `V9d-T`, rewrite the Tests bullet *"Each always-log kind emits exactly once; the success side honours the null-policy."* so the exactly-once assertion is anchored to the spec's enumerated always-log set rather than to an implicit set. The bullet should cite `runtime-event-channel.md` §"Runtime event channel" and name the membership it covers — the group-A members (`transport`, `model_tool`, `code_tool`, `tool_loop_exhausted`, `invoke_infra`, and every Binder failure-modes row) and the group-B members (every `loom/runtime/*` panic code except the five `console.error`-routed teardown codes `loom/runtime/reload-teardown-timeout`, `loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, `loom/host/session-shutdown-runtime-degraded`) — such that each enumerated member is exercised by a fixture and a member dropped from the implementation reds the test. Mirror the anchoring style the sibling group-A/B routing bullet already uses (a spec-linked partition reference). Retain the success-side null-policy clause unchanged. Reflect the same anchoring in `V9d`'s **Ships when** sentence, which currently restates "the always-log exactly-once emission" without the enumeration tie.

## Consequence

**Severity:** advisory

A test that iterates the implementation's own always-log set definition passes even if a required member (e.g. `invoke_infra`, a specific Binder failure cause, or a `loom/runtime/*` panic code) is omitted from that definition, so a missing always-log emitter could ship without reddening `npm test`. The exactly-once guarantee is asserted in aggregate but not per enumerated member.
