## Finding

The two children under `docs/spec_topics/frontmatter.md` carry the H1 titles `# Frontmatter fields a` (`frontmatter-fields-a.md`, line 1) and `# Frontmatter fields b and templates` (`frontmatter-fields-b-and-templates.md`, line 1). These "a"/"b" labels are pure pagination markers — the split was byte-count driven (size-cap enforcement, single commit) and the slug-generator just appended sequential letters. They convey nothing about which fields each page owns, and the parent index page `frontmatter.md` propagates the same opaque labels into its `## Contents` link text (`- [Frontmatter fields a](...)` / `- [Frontmatter fields b and templates](...)`), so the only entry point a reader has to the field reference forces them to click both pages to find out which holds which field.

The opacity is compounded by an unclean split. Page (a) ends mid-`tools` (after the "Pi tool names" rule and `as`-rename rules) and page (b) begins with the `tools:` YAML-shape elaboration (comma-form vs list-form, resolution-snapshot, file-watcher reload semantics) before moving on to `system`, `respond_repair`, `tool_loop`, and a `## Template Interpolation` section. So `tools` straddles the page boundary — a reader landing on (b) sees normative `tools` rules without the field-contract row that defines them, and vice versa. Any rename that pretends each page covers a clean field group will mislead.

This is a navigation/discoverability defect, not a content defect: the normative material is correct and complete, just hard to find by title. Sibling page-title findings on `ceilings-3-and-4.md` / `ceiling-invariants-and-audit.md` (cluster: "Ceiling invariants and audit") and on `query/query-escapes-stringification.md` (cluster: "Page title understates scope") have the same shape and resolve the same way.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `f5e89f4` — "docs: enforce 100KB hard / 30KB recommended size cap on spec set" (2026-06-04).
**History:** Before `f5e89f4`, `docs/spec_topics/frontmatter.md` was a single ~30KB file with no children. That commit added `docs/_tools/split_spec.py`, whose `PLANS` dictionary mechanically split eleven over-cap files into `index + N children`. The slug list for `frontmatter.md` is literally `("frontmatter-fields-a", 5, 88), ("frontmatter-fields-b-and-templates", 89, 155)` — the `-a` / `-b` suffixes were assigned by the script's sequential-letter convention against a line-range split chosen to fit the 30KB cap, not against a topical seam. The H1 of each child was then generated by `title_from_slug` from the same slug, so the opaque title and the opaque filename both trace to one mechanical pass. `git blame` on lines 7–8 of `frontmatter.md` (the `## Contents` bullets) confirms both bullets entered in `f5e89f4` and have never been edited; `git log --follow` on either child file shows only the introducing commit (file-rename detection threshold not crossed). The defect is therefore single-commit-introduced and present-on-every-line-since-creation — no later edit narrowed or broadened it.

## Solution Space

**Shape:** multiple

### Option A — Retitle H1s and TOC entries in place (no filename change)

- **Approach.** Edit three lines: the two child H1s and the two bullet-link captions on `frontmatter.md`. Leave the filenames, anchors, REQ-ID prefixes, and every existing cross-reference (in spec pages, plan-topic `Spec.` rows, coverage-matrix rows, `split_spec.py`) untouched.
- **Spec edits.**
  - `frontmatter.md` line 7: `- [Frontmatter fields a](./frontmatter/frontmatter-fields-a.md)` → `- [Field contract and load-time fields (mode, description, argument-hint, model, bind_*, params, tools)](./frontmatter/frontmatter-fields-a.md)` (or a shorter equivalent: `Field contract and load-time fields`).
  - `frontmatter.md` line 8: `- [Frontmatter fields b and templates](...)` → `- [Runtime fields (tools shape, system, respond_repair, tool_loop) and template interpolation](...)` (or `Runtime fields and template interpolation`).
  - `frontmatter-fields-a.md` line 1: `# Frontmatter fields a` → `# Field contract and load-time fields` (or the long form chosen above).
  - `frontmatter-fields-b-and-templates.md` line 1: `# Frontmatter fields b and templates` → `# Runtime fields and template interpolation`.
- **Pros.** Three-line edit. Zero churn to cross-references. Anchor / REQ-ID stability invariants under GOV-20 are not even engaged. The unclean `tools` split is honestly disclosed by naming each page after the *field-group majority* and saying "and" instead of pretending exclusivity.
- **Cons.** Filenames remain opaque, so any cross-reference quoted by URL still reads `frontmatter-fields-b-and-templates.md`. Doesn't fix the underlying mid-`tools` split — the title still slightly misrepresents what each page contains.
- **Risks.** None of substance — pure prose change.

### Option B — Re-split at a clean field-group seam, then rename both filenames and H1s

- **Approach.** Move the `tools:` YAML-shape elaboration (the first paragraph and YAML examples of `frontmatter-fields-b-and-templates.md`, plus the **Resolution snapshot** sub-block) back to `frontmatter-fields-a.md` immediately after the existing `tools` prose, so `tools` is owned end-to-end by page (a). Then rename the files to descriptive stems (e.g. `frontmatter-fields-load-time.md` and `frontmatter-fields-runtime-and-templates.md`), update both H1s, and repoint every cross-reference (coverage-matrix row `cka-11`, the five `V6a..V6e` leaves and their `-T` siblings, the `M-` and `M-T-minimal-slash-command` leaves, `split_spec.py`'s `PLANS` entry, and any in-spec links).
- **Spec edits.** Same H1 / TOC edits as Option A plus: relocate ~10 lines of `tools:` YAML-shape prose; `git mv` both child files; sweep ≥18 file references (`grep -rn 'frontmatter-fields-[ab]'` lists every site).
- **Pros.** Eliminates the mid-`tools` split, so each page covers one clean field group. Filenames and titles agree. Future cross-references read naturally.
- **Cons.** Re-violates the 30KB cap risk (page (a) gains ~10 lines and the relocated YAML; verify post-move). Forces a churn pass across plan-topics and coverage-matrix. Requires extra audit that no anchor `<a id="...">` moved (the `resolution-snapshot` and `v1-seam-system-expression-sublanguage` anchors must stay on whichever page their referrers expect — GOV-20 anchor stability).
- **Risks.** Anchor-target drift breaks links silently if the relocation sweep misses a referrer. The wider edit invites a follow-up critique cycle that pure-rename would not.

### Recommendation

Take Option A. The defect is navigation friction caused by the H1 / TOC link text; once those four strings are descriptive, every reader entering through the index page sees what each child covers. Filenames are durable URLs that cost more to change than they buy back, and the mid-`tools` split — while real — does not interfere with reading the field-contract table linearly because `tools` rules on (b) explicitly back-link to `frontmatter-fields-a.md#tools` for the per-entry grammar. State the split honestly in the rewritten link text (e.g. "`tools` shape" appears on the (b) caption) so a reader chasing a `tools`-related citation knows to consult both pages. Edge cases for the implementer: keep the existing `<a id="...">` anchors (`tools`, `argument-hint-binder-only`, `binder-model-root-word-convention`, `resolution-snapshot`, `frnt-1`, `loom-1-0-seam-system-expression-sublanguage`, `v1-seam-system-expression-sublanguage`) byte-identical — the rename touches H1 prose only, never anchor IDs — and do not introduce new H1 anchors that collide with existing slugified-heading anchors elsewhere in the corpus.

## Consequence

**Severity:** cosmetic

A reader navigating from the `Parameters and Frontmatter` index has no way to pick the right child page without opening both — every cross-link from leaves, coverage-matrix rows, and other spec pages that says "see `frontmatter-fields-b-and-templates.md`" forces a blind click. No normative behaviour changes; the cost is wall-clock and cognitive friction for every author and implementer who follows a frontmatter cross-reference.
