## Finding

The normative overflow signatures in the `Provider error mapping` table live inside inline-code spans within GFM table cells â€” e.g. the `anthropic-messages` row's `` `/(prompt is too long\|exceeds .* context window\|maximum context length)/i` `` and the `openai-completions` row's `` `/maximum context length\|context_length_exceeded/i` ``. Each alternation pipe is written `\|`. Inside a GFM table cell a literal pipe must be backslash-escaped so it is not parsed as a column delimiter, and that escaping applies even inside code spans; the escape is stripped during rendering, so the *rendered* regex reads `/(prompt is too long|exceeds .* context window|maximum context length)/i` â€” correct JS-regex alternation.

The problem is that the two readings of the same source text diverge. An implementer who consumes the rendered spec sees alternation and builds a working classifier. An implementer who transcribes the regex from the raw Markdown source sees `\|` and reasonably reads it as a JS-regex escaped pipe â€” which matches a *literal* `|` character, not alternation. A classifier built from the literal reading never matches any real provider overflow message, so every context-overflow response silently falls through to the `TransportError` catch-all with `tokens_used` / `tokens_limit` null. The defect is that a normative regex is expressed in a syntax where Markdown's escaping convention and the target language's escaping convention collide on the same character, leaving the operative meaning reader-dependent.

The fix is structural: the overflow signatures should be expressed somewhere the alternation pipe appears literally, with no Markdown escaping in play, so there is exactly one defensible reading of the operator.

## Solution Space

**Shape:** single

### Recommendation

Move the per-provider overflow signatures out of the GFM table so no pipe is subject to Markdown cell-escaping. Replace the `Overflow signature â†’ ContextOverflowError` column with a per-provider bulleted list (or a definition list), and place each regex in a fenced code block, where `|` appears literally and unescaped and reads only as JS-regex alternation. Keep the surrounding condition prose with each provider entry, and preserve:

- the `openai-completions` HTTP-200-with-`stopReason: "error"` alternative branch that matches the same regex;
- the per-provider `tokens_used` / `tokens_limit` outcome and the cross-reference to *Overflow token-count extraction* (counts extracted for `anthropic-messages` / `openai-completions`, both `null` for `mistral` / `amazon-bedrock`).

After restructuring, verify no `\|` remains in any regex in the section.

## Consequence

**Severity:** correctness

The latter silently downgrades every `ContextOverflowError` to `TransportError` with null token counts, defeating overflow detection and its repair short-circuit on every overflow response.
