## Finding

The capability-probe leaf pair `V9a` and `V9a-T` each carry two consecutive Tests bullets that both lead with the label `PIC-5`, but the two bullets assert different obligations:

- Bullet 1 — "there are exactly five checks; no probes beyond them." This is the spec obligation `PIC-5` (`capability-probe.md`: *"Probes MUST NOT be added beyond the five enumerated checks"*).
- Bullet 2 — "`FACTORY_PROBABLE_CAPABILITIES` enumerates exactly the five factory-probable capability identifiers (inventory items 1/2/3/4/6) and nothing else, pinning the symbol `V18a`/`V18c` import." This is a plan-level constant-cardinality invariant on an exported symbol. Its authority is `capability-probe.md` step (c) (*"Factory-probable SDK capabilities … items 1, 2, 3, 4, and 6 (five capabilities, nine function members)"*) plus the SDK capability inventory — it is **not** `PIC-5`.

The two assertions are independent: `PIC-5` bounds the probe's check count (five sub-steps `(a)`–`(e)`), while the second bullet bounds the cardinality of the exported `FACTORY_PROBABLE_CAPABILITIES` list that `V18a`/`V18c` reconcile their partition against. Reusing the `PIC-5` label for the second bullet collides the two and reads as a redundant restatement of the first rather than a separate acceptance criterion.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V9a-T-capability-probe.md` and `docs/plan_topics/V9a-capability-probe.md`, change the leading label of the second (`FACTORY_PROBABLE_CAPABILITIES`) Tests bullet from `PIC-5` to a distinct label that identifies it as the constant-cardinality invariant rather than a spec REQ-ID — for example `FACTORY_PROBABLE_CAPABILITIES cardinality (spec capability-probe.md step (c))`. Leave the first `PIC-5` bullet ("there are exactly five checks; no probes beyond them") unchanged, since it correctly cites the spec `PIC-5` obligation. The bullet body text need not change; only the label does. Apply the identical relabel to both paired files so they stay in lock-step.

## Consequence

**Severity:** correctness

An implementer authoring `V9a-T` can read the two identically-labeled `PIC-5` bullets as one duplicated requirement and write a single test, silently dropping the `FACTORY_PROBABLE_CAPABILITIES` cardinality assertion that pins the exported symbol `V18a`/`V18c` reconcile against. The constant's five-element cardinality then ships unverified, and two reasonable implementers diverge on whether the second bullet is a distinct test at all.
