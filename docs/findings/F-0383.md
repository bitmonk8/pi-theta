## Finding

`V9a` is the single source-of-truth pinned-constants block: it exports `SHUTDOWN_AWAIT_CAP_MS = 2000` and physically holds the `SessionShutdownEvent['reason']` closed-set snapshot entry that the `session_shutdown` handler reads. Both test leaves write failing tests that exercise symbols resident in that block:

- `V9g-T` asserts `loom/runtime/reload-teardown-timeout` "fires at the cap" and that sub-step 3's `Promise.allSettled` await is "bounded by `SHUTDOWN_AWAIT_CAP_MS`" — the constant `V9a` exports.
- `V9h-T` asserts the `loom/host/session-shutdown-pinned-constant-unreadable` failure modes, which are produced by reading the `SessionShutdownEvent['reason']` snapshot entry out of the same `V9a`-established pinned-constants block.

Each paired implementation leaf lists `V9a` in its Deps (`V9g` Deps: `V9g-T, V9a, V9e, V9h, V17a`; `V9h` Deps: `V9h-T, V9a, V9m, V18c`), but the test leaves drop it: `V9g-T` Deps is `V9e, V9h, V17a` and `V9h-T` Deps is `V9m, V18c`. The dependency on `V9a` is therefore only incidentally satisfied through transitive closure — `V9g-T` reaches `V9a` via `V9h` and via `V17a`, and `V9h-T` reaches it via `V9m`→`V9b`→`V9a` and via `V18c`→`V18a`→`V9a`. The symbols the test leaves consume come from `V9a`, but `V9a` is not named as a direct prerequisite.

The Deps field's stated semantics are "other leaf IDs that must be complete first" with pickability keyed on the prerequisite's completion tag. A leaf whose tests will not compile until `V9a`'s exported symbol exists has a direct prerequisite on `V9a`; expressing it only through a multi-hop chain that runs through unrelated intermediates leaves the prerequisite implicit and fragile.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` ("pi-loom plan: build/update plan for spec.md + review")
**History:** Both test leaves were created in the corpus-building commit `c6a664e` with Deps that excluded `V9a` (`V9g-T`: `V9e, V9h, V17a`; `V9h-T`, then named `V9h-T-degraded-unknown-reason.md`: `V9b, V18c`). `git log -S "V9a"` over each test leaf's full history (including the `V9h-T` rename at `b28e62f`) returns no commit — the `V9a` token has never appeared in either leaf's Deps. The omission is original to the leaves' authoring; the only later Deps change was `V9h-T`'s `V9b`→`V9m` swap, which did not touch the `V9a` question. The paired implementation leaves carried `V9a` from the same inception commit, so the impl/test asymmetry is also present-since-inception.

## Solution Space

**Shape:** single

### Recommendation

Add `V9a` to the Deps field of both test leaves, so each names the owner of the pinned-constants-block symbol its tests consume — matching the paired implementation leaves.

- `docs/plan_topics/V9g-T-session-shutdown.md`: change the `Deps.` line from `` `V9e`, `V9h`, `V17a` `` to include `` `V9a` `` (e.g. `` `V9a`, `V9e`, `V9h`, `V17a` ``).
- `docs/plan_topics/V9h-T-unknown-reason-rule.md`: change the `Deps.` line from `` `V9m`, `V18c` `` to include `` `V9a` `` (e.g. `` `V9a`, `V9m`, `V18c` ``).

The two edits are independent one-line Deps additions; no acceptance criteria, Tests bullets, or Ships-when text change.

## Consequence

**Severity:** advisory

Nothing breaks today: `V9a` is transitively guaranteed complete before either test leaf is pickable, so the test sources still compile. The defect is latent fragility — the prerequisite on `V9a`'s exported symbol is carried only by intermediate Deps edges through unrelated leaves, so any future edit that drops `V9g-T`'s edge on `V9h`/`V17a` or `V9h-T`'s edge on `V9m`/`V18c` would silently remove `V9a` from the closure and break test compilation with no Deps change naming `V9a`.
