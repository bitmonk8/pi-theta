## Finding

The *No globals, statics, singletons* cross-cutting rule bans direct ambient access to a fixed set of timing primitives — `Date.now`, `performance.now`, `Date.prototype.getTime`, `setTimeout`, `clearTimeout` — enforced by the `H3a` identifier-keyed architectural scan. That list is a **manually-maintained derivation** of the spec's `Clock`/`WallClock` ban at [`host-interfaces-services.md` PIC-12](docs/spec_topics/pi-integration-contract/host-interfaces-services.md). The plan states the required invariant (scan list ⊇ spec ban) and routes its upkeep to the once-per-release manual *loom 1.0 release-time residue inspection* item 9.

The mechanical scan checks `src/**` against the hand-copied list only; nothing checks the list itself against PIC-12. If a future spec edit widens the `Clock`/`WallClock` surface (adds a new banned primitive), the scan list silently becomes a non-superset and the `H3a` gate keeps passing — the newly-banned primitive can be referenced ambiently in `src/**` with no CI signal. The single backstop is a human re-reading PIC-12 against the list at release time (item 9). Between a ban-widening spec edit and that release-time read, the divergence is unflagged.

The plan already documents this residue accurately and concedes it is manual-only; what it does not do is either mechanise the superset assertion or mark the manual-only disposition as a deliberately-accepted residue rather than an open gap.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** The `H3a` ambient-primitive scan was defined in the initial plan build (`c6a664e`, 2026-06-10) as a hardcoded primitive list (`Date.now`, `setTimeout`, … outside their seam adapters) with no mechanical reconciliation against the spec ban. The list was later expanded to the current five primitives (`c6824f2`, 2026-06-12, "ambient-primitive scan omits spec-banned clock primitives"), and the over-stated "sourced directly … so the scan and the spec stay in lockstep" wording was corrected to the accurate "manually-maintained derivation … scan list ⊇ spec ban … re-reconciled by item 9", with residue-inspection item 9 added in the same edit (`1c9ede1`, 2026-06-18, "ambient-primitive ban list lockstep over-promise"). The absence of a mechanical superset check — the defect this finding names — has been present since the scan was first defined; `1c9ede1` surfaced and documented it and added the manual item-9 reconciliation, but deliberately left the check manual.

## Solution Space

**Shape:** multiple

### Option A — Mechanise the superset assertion

**Approach:** Add a closing-gate arm to the `H5*` gate machinery that asserts the `H3a` scan's primitive list is a superset of a machine-readable enumeration of the PIC-12 ban, reddening CI on divergence. Because PIC-12 is currently prose, this requires the spec to expose the banned-primitive set as a parseable artefact (e.g. a fenced enumeration co-located with PIC-12 on `host-interfaces-services.md`) that both the gate and the prose stay aligned to.

**Plan edits:** Add the reconciliation arm to `H5a` (or a dedicated `<new>` gate leaf); update `conventions.md` §No-globals and item 9, and the `H3a` Tests bullet, to cite the mechanical arm instead of (or alongside) the manual reconciliation.

**Spec edits:** `host-interfaces-services.md` PIC-12 — add the machine-readable banned-primitive enumeration the gate reads.

**Pros:** Divergence is caught at CI on the ban-widening spec edit, not deferred to a once-per-release human read.

**Cons:** Adds a spec-side machine-readable artefact plus a parser; couples the gate to a spec format; the enumeration can itself drift from the surrounding prose.

**Risks:** Brittle parsing of the spec; a new gate surface to maintain for a slow-moving primitive set.

### Option B — Formally accept as recorded residue

**Approach:** Keep the manual reconciliation and make the manual-only disposition an explicit, deliberate acceptance rather than an unmarked gap. In `conventions.md` item 9 (and, by reference, the §No-globals rule and the `H3a` Tests bullet), state that the absence of a mechanical superset check is an accepted residue whose upkeep is the item-9 reviewer's obligation, on the same footing as the other release-time residue-inspection items.

**Plan edits:** `conventions.md` item 9 + §No-globals prose; optionally a one-line note on the `H3a` Tests bullet. No new gate, no spec edit.

**Spec edits:** None.

**Pros:** Zero new machinery; consistent with the plan's established residue-routing pattern for conceded mechanical-scan gaps; near-zero diff.

**Cons:** The window between a ban-widening spec edit and the release-time re-read remains unguarded; relies on the item-9 reviewer actually performing the reconciliation.

**Risks:** If PIC-12's surface is expected to grow during the 1.0 cycle, the manual gate fires too late.

### Recommendation

Take Option B: formally record the manual-only reconciliation as accepted residue in `conventions.md` item 9 and the §No-globals rule. The plan already documents the gap, states the ⊇ invariant, and assigns ban-widening to a spec-side PIC-12 decision; the remaining gap is that the manual-only disposition reads as an open hole rather than a deliberate choice. The PIC-12 ban is a small, slow-moving surface, and Option A's value (catching drift at CI) is bought with a new spec-side machine-readable artefact whose own drift would then need guarding. Watch edge case: if the team anticipates the `Clock`/`WallClock` ban expanding before the 1.0 release, prefer Option A so the superset check fires on the spec edit rather than at release time.

## Consequence

**Severity:** advisory

If PIC-12's `Clock`/`WallClock` surface is widened after the `H3a` list is fixed, the `H3a` ambient scan stays green while the newly-banned primitive is used ambiently in `src/**`, undetected until the once-per-release manual item-9 re-read. The exposure is future-conditional (the list is a correct superset today) and is already routed to a named manual gate, but the only thing standing between a ban-widening spec edit and a leaked ambient use is a human remembering to perform item 9.
