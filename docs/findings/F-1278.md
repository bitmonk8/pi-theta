## Problem

`expressions.md` is registered non-narrative under the `EXPR` prefix yet carries zero `EXPR-N` REQ-ID anchor sites, so none of its normative content — operator precedence, evaluation-order / short-circuit rules, identifier-resolution order, truthiness, equality and ordering semantics, stdlib method contracts (`string` / `array<T>` / `object`), construction rules, and the closed-set arithmetic dispositions — is addressable by a `#expr-n` URL fragment. Because GOV-9 makes a `#prefix-n` cross-link the canonical citation form for any page depending on these rules, every cross-page citation today either restates the rule locally (two normative sources that can drift) or hangs off an unstable heading slug. The plan corpus reflects the same gap: a single `cka-3` whole-page rollup row in `coverage-matrix.md` plus `(EXPR code-keyed area)` citations on the V3-family leaves stand in for the entire numbered-REQ-ID coverage the page would otherwise expose.

## Issue introduction

**Verdict:** present-since-prefix-registration
**Introducing commits:** `c1e5d64` (Tue May 5 2026, "spec: REQ-ID infrastructure + Phase 12a heading promotion on five named pages (Decision #12, partial)")
**History:** The expression sublanguage prose has existed in some form since the earliest spec scaffolding (`9a53d9c`, Initial scaffold). The `EXPR` prefix was registered against `expressions.md` in `c1e5d64`, whose own commit message explicitly defers the per-rule REQ-ID-assignment sweep to a follow-on "Phase 12b" pass: *"Phase 12b (per-rule REQ-ID assignment) and the per-leaf REQ-ID citation pass remain as the mechanical sweep across the rest of spec_topics/"*. That sweep never landed on `expressions.md`. The split into `docs/spec_topics/expressions.md` (`fecb504`, Mon May 4 2026) preserved the same anchor-free state, and the nineteen subsequent commits touching the page (`a6d82db` through `7686ba4`) all resolved individual content findings without coining IDs. The GOV-22 progressive-coinage rule (`4f588e0`, Wed Jun 3 2026 — "pi-loom spec: resolve Non-narrative pages with zero coined REQ-IDs non-referenceable") was authored precisely to drain residues like this one going forward, but it triggers only where a commit's diff edits a defining obligation site's normative-modal token, and `expressions.md` carries exactly one `MUST` token in its current body — so the rule does not drain the page through ordinary editing traffic. The residue is therefore not a regression of a previously-anchored state; it is the un-finished tail of the Phase 12b sweep, persistent because the page's prose almost entirely uses declarative ("evaluates", "produces", "is rejected") rather than `MUST`/`MUST NOT` modal forms.

## Solution approach

Coin one `EXPR-N` REQ-ID per independently-testable obligation on `expressions.md`, using the page's existing `##` / `###` headings as the partition boundary, in GOV-1 dual-form layout (`<a id="expr-n"></a> **EXPR-N.**`) with numeric tails allocated consecutively per GOV-3. Replace `coverage-matrix.md`'s `cka-3` whole-page rollup row with a numbered-REQ-ID block citing the new `EXPR-N` IDs against `V3a` / `V3f` / `V3g` / `V3h`. Rewrite the `*Tests*` bullets on the `V3a` / `V3f` / `V3g` / `V3h` leaves and their `-T` pairs to cite the new `EXPR-N` IDs inline instead of the `(EXPR code-keyed area)` rollup form.

## Solution constraints

- Out of scope: the normative rule prose itself — coin anchors and IDs only; do not reword the obligation text.
- Retain `cka-4` for `loom/parse/non-boolean-condition` in `coverage-matrix.md` — it is a reject-code closure, not an EXPR-numbered obligation.
