## Problem

Step 5's *Enumeration mechanism — grep target* paragraph defines a build-time audit pass that the session-only triplet sweep (`(new|resume|fork)`) is explicitly hard-gated against: a result-delta fails `npm test` red rather than relying on contributor-side discipline. The symmetric complement sweep (`(quit|reload)`), which audits the claim that loom 1.0 carries no inbound runtime reference to a `{"quit","reload"}` literal subset, is described only as "MUST also be run on each bump", with no statement of whether it is part of that mandatory build-time pass or a separate manual step. Because the gate is described as running "the same three grep targets and second-stage anchor filter" but the first-stage filter differs between the two sweeps, the prose does not settle whether the gate runs the complement filter at all. Two conforming implementations diverge on whether the complement check is mechanically enforced.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `52b9101` (2026-05-11, *resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"*); `c0259ca` (2026-06-07, *resolve "Version-bump procedure: four obligations have no verifiable acceptance criterion"*)
**History:** `git log -S` on `symmetric grep` / `complement-set claim` shows the complement-grep clause entered the corpus in `52b9101`, which at the same time introduced the build-time audit pass as an *optional* capability (*"The H1 leaf … MAY additionally implement a build-time audit pass … PIC does not require the audit"*). At that point the triplet sweep and the complement grep were on equal, soft footing — both "run on each bump", neither hard-gated — so no asymmetry existed. `git show c0259ca` confirms that commit replaced *"MAY additionally implement a build-time audit"* with *"MUST implement a build-time audit pass … fails `npm test` red"* for the triplet sweep, but left the complement-grep clause unchanged. The defect is the interaction: hardening one half of a deliberately-symmetric pair into a mandatory gate without updating the complement clause in lockstep opened the undefined-gate-status gap.

## Solution approach

Clarify the *Enumeration mechanism — grep target* paragraph to settle the complement sweep's gate status relative to the mandatory build-time audit pass, on the same footing the triplet sweep already carries. Place the complement's `(quit|reload)` first-stage filter under the same `npm test` gate so a result-delta fails red rather than relying on contributor-side discipline.
