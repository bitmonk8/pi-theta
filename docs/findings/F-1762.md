## Finding

Step 5's *Enumeration mechanism — grep target* paragraph defines two greps over the same three targets (`src/`, `test/`, `docs/`) and the same second-stage anchor filter. The first-stage filter for the session-only triplet is `"(new|resume|fork)"`; the symmetric complement uses `"(quit|reload)"` to audit the claim that loom 1.0 carries no inbound runtime reference to a `{"quit","reload"}` literal subset.

The triplet sweep is given an unambiguous enforcement status: *"The implementation MUST implement a build-time audit pass over the same three grep targets … and second-stage anchor filter so that a result-delta against the documented enumeration fails `npm test` red rather than relying on contributor-side discipline."* The complement grep, by contrast, is described only as *"MUST also be run on each bump to audit that complement-set claim"* — with no statement of whether it is part of that same mandatory build-time pass or a separate manual bump step. Because the build-time audit pass is described in terms of "the same three grep targets and second-stage anchor filter" but the first-stage filter differs between the two sweeps (`(new|resume|fork)` vs `(quit|reload)`), the prose does not settle whether the mandated `npm test` gate runs the complement first-stage filter at all.

The silence is load-bearing: the spec explicitly rejected reliance on contributor-side discipline for the triplet, yet leaves the complement-set claim — structurally the same kind of inbound-reference audit — without a stated gate. Two conforming implementations diverge on whether the complement check is mechanically enforced.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `52b9101` (2026-05-11, *resolve "T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state"*); `c0259ca` (2026-06-07, *resolve "Version-bump procedure: four obligations have no verifiable acceptance criterion"*)
**History:** `git log -S` on `symmetric grep` / `complement-set claim` shows the complement-grep clause entered the corpus in `52b9101`, which at the same time introduced the build-time audit pass as an *optional* capability (*"The H1 leaf … MAY additionally implement a build-time audit pass … PIC does not require the audit"*). At that point the triplet sweep and the complement grep were on equal, soft footing — both "run on each bump", neither hard-gated — so no asymmetry existed. `git show c0259ca` confirms that commit replaced *"MAY additionally implement a build-time audit"* with *"MUST implement a build-time audit pass … fails `npm test` red"* for the triplet sweep, but left the complement-grep clause unchanged. The defect is the interaction: hardening one half of a deliberately-symmetric pair into a mandatory gate without updating the complement clause in lockstep opened the undefined-gate-status gap.

## Solution Space

**Shape:** single

### Recommendation

Fold the complement grep into the same mandatory build-time audit pass. State in the *Enumeration mechanism — grep target* paragraph that the build-time audit pass runs both first-stage filters — `(new|resume|fork)` and `(quit|reload)` — over the three grep targets and the shared second-stage anchor filter, so a result-delta on either fails `npm test` red on the same footing.

Name the complement's baseline explicitly: the complement-set documented enumeration is currently empty (loom 1.0 has no inbound runtime reference to a `{"quit","reload"}` literal subset), so the gate's pass condition is "zero hits after the second-stage filter and the documented-enumeration exclusion list", and any hit fails red and must be enumerated in the same edit on the same footing as the future-edit obligation already stated.

This keeps the complement check under the same mechanical enforcement the spec already mandates for the triplet, rather than reintroducing the contributor-discipline reliance the triplet clause explicitly rejects. Edge case the implementer must watch: the complement and triplet share the second-stage anchor filter and exclusion list, so the audit should apply the same documented-`docs/`-restatement exclusions (e.g. the `docs/spec.md` SM-2 listing) to the complement first-stage filter as well, to avoid the complement half reporting transiently-stale illustrative restatements as deltas.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one folds the `(quit|reload)` first-stage filter into the mandatory build-time audit so a hit fails `npm test` red; another treats the complement grep as a manual bump-time step outside the gate, relying on the contributor-discipline the spec rejected for the triplet half. On the second implementation a future `{"quit","reload"}` inbound runtime reference can be added without the build-time gate catching it, silently breaking the complement-set claim the sweep exists to defend.
