## Finding

The `V9k` code-keyed row in `coverage-matrix.md` describes the
`loom/load/extension-bootstrap-failed` granularity contract as "the **five**
per-call-type ... granularity surfaces", but its parenthetical enumerates only
four items: renderer-failure degrade-and-proceed; per-loom `registerCommand`
drop; whole-extension `registerFlag`/`pi.on` abort; and `session_start`-time
`getCommands()` read-failure pending-list drop. The third item collapses two
distinct surfaces ŌĆö the factory-time `pi.registerFlag` abort and the `pi.on(...)`
subscription abort ŌĆö into one entry, so the enumerated count is four while the
stated count is five.

The authoritative sources confirm five distinct surfaces. The spec section
`extension-bootstrap-and-per-loom.md` ┬¦Extension-bootstrap SDK failures lists
five separate bullets ŌĆö `pi.registerMessageRenderer`, `pi.registerCommand`,
`pi.registerFlag`, `pi.on(...)`, and `pi.getCommands()` ŌĆö and closes with "None
of these **five** rules apply to the step-0 capability probe." The `V9k` leaf
has five Tests bullets (one per surface) and a Ships-when clause requiring
`npm test` to exercise "each of the five bootstrap-failure surfaces." The matrix
row's parenthetical is the sole place where the enumeration disagrees with the
count; separating `registerFlag` and `pi.on` restores five.

## Solution Space

**Shape:** single

### Recommendation

In `coverage-matrix.md`, the `V9k` row's parenthetical, split the collapsed
third item "whole-extension `registerFlag`/`pi.on` abort" into two separate
items so the enumeration lists five surfaces matching the leading count.
Replace the parenthetical

`(renderer-failure degrade-and-proceed; per-loom `registerCommand` drop; whole-extension `registerFlag`/`pi.on` abort; `session_start`-time `getCommands()` read-failure pending-list drop with no drain-state write)`

with

`(renderer-failure degrade-and-proceed; per-loom `registerCommand` drop; whole-extension `registerFlag` abort; whole-extension `pi.on` abort; `session_start`-time `getCommands()` read-failure pending-list drop with no drain-state write)`

This aligns the row with the five spec bullets and the leaf's five Tests
bullets. Do not instead reduce the count to four ŌĆö five surfaces genuinely
exist per the spec and the `V9k` leaf. The spec file is read-only for this fix;
the correction is confined to the coverage-matrix row.

## Consequence

**Severity:** advisory

The coverage-matrix row is a traceability surface; the count/enumeration
mismatch makes a reader cross-referencing the row against the `V9k` leaf doubt
which surface is missing. The leaf remains authoritative and internally
consistent, so an implementer working from it still produces all five surface
tests ŌĆö the defect is a documentation discrepancy in the matrix prose, not a
missing obligation or a gate that fails.
