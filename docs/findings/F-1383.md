## Problem

The *Context-overflow handling* bullet in `determinism-cancellation-failure.md` rests on the premise that "the binder system prompt is fixed across attempts" to distinguish flake-equivalent from structural overflow, but that premise is asserted only inside a rationale clause and is not pinned by any normative obligation in the binder pages. The nearest pin — the malformed-envelope retry paragraph — fixes only the *envelope schema* across attempts ("no schema mutation between attempts") and says nothing about the system-prompt bytes (the rendered Parameters block, the `User arguments:` line, or the *Recent session context* block). The re-entrancy seam in `binder-model-and-context.md` cuts the other way: "constructed afresh on every binder call" describes cross-invocation re-entrancy but can be misread as licensing per-attempt re-rendering. The term "binder call" compounds the ambiguity — the retry-budget paragraph uses "binder LLM call" for each provider attempt while the seam uses "binder call" for one logical invocation — so a retry implementer cannot tell whether to snapshot the input record once per slash invocation or re-render it on each attempt, the latter silently breaking the flake-vs-structural argument when the session mutates during the retry window.

## Solution approach

Add a normative obligation to the per-invocation retry-budget paragraph (`#per-invocation-retry-budget` in `determinism-cancellation-failure.md`), alongside the existing same-envelope-schema sentence, pinning the binder's input record — the rendered system-prompt bytes, the envelope schema, the resolved binder-model handle, and the FNV-1a seed — as constructed once per slash invocation and reused byte-identically across every budgeted retry, even when the caller's session mutates during the retry window. Disambiguate "binder LLM call" (a single provider attempt) from "slash invocation" (the retry-budget scope) within that section. Add a cross-reference reconciling the pin with the `#loom-1-0-seam-automatic-context-escalation` seam's "constructed afresh on every binder call", clarifying that afresh-construction is per slash invocation, not per attempt.
