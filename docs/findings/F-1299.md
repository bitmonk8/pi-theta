## Problem

PIC-14's chokidar-adapter bullet states that the adapter filters `error` out of the `FileWatcher` seam and routes its own `error` events through the `ctx.ui.notify` toast surface, but does not state what the watcher does afterward: whether chokidar keeps delivering `add`/`change`/`unlink` events, whether one or more roots have stopped delivering (leaving content silently stale until the next `/reload`), whether the extension instance is now degraded in the PIC drain-state sense, or whether `/reload` is the operator recovery path. Chokidar's `error` callback is non-fatal by default, but specific underlying syscall failures can stop event delivery on a watched root without surfacing a separate closed signal. The diagnostic-shape.md transient-toast paragraph names the same surface but explicitly does not state post-error watcher state, scan continuation, or message content, so neither owning page closes the loop. Implementers diverge: one re-arms each root and treats the toast as informational; another lets the watcher die silently and assumes `/reload` restarts it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `8a6d15f` (2026-05-06) introduced the PIC-14 `FileWatcher` bullet that routes chokidar `error` events through `ctx.ui.notify` without stating post-`error` watcher continuation / recovery. `11cabfe` (2026-06-22) relocated the seam block but introduced no new content.
**History:** The watcher-error gap has been present since `8a6d15f`; the recent relocation moved the block but did not close it.

## Solution approach

At PIC-14's chokidar-`error` bullet, clarify the watcher's post-`error` continuation and recovery posture: distinguish the case where event delivery continues from the case where one or more watched roots stop delivering, and name `/reload` as the operator recovery path for the latter. Tie the stopped-delivery case into the existing degraded-state drain-state so subsequent slash dispatches stop running against stale content. Register a `loom/runtime/*` diagnostic row in `code-registry-runtime.md` for the terminal-watcher event and add a cross-reference from diagnostic-shape.md's transient-toast paragraph.

## Solution constraints

- A new `loom/runtime/*` diagnostic code is governed by DIAG-2 (closed registry) and DIAG-1: land its `code-registry-runtime.md` row in the same edit and never emit an unregistered code.
- The operator-facing terminal-watcher note (a `loom/runtime/*` code) MUST use the persistent `loom-system-note` channel as its primary sink, not `ctx.ui.notify` (diagnostic-shape.md routing rule); the toast surface stays reserved for the chokidar-throw case it already names.
