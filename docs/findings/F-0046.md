## Finding

H5a's standing transitive-completeness arm reconciles every closing-leaf cell in `coverage-matrix.md` against `H5b`'s `Deps.` "after expanding both sides' contiguous within-group `<group><letter>` ranges by letter suffix." The arm prose pins two grammar facts — the two endpoints must share the same `<group>` number, and the range enumerates over its contiguous letter suffixes (`V2a–V2d` → `V2a, V2b, V2c, V2d`) — but leaves the rest of the token grammar to the implementer's inference, and `conventions.md` §Leaf format only offers authoring guidance ("Cite specific leaf IDs (`V4b`, `V9a–V9e`); … Use ranges where contiguous and comma-separated lists where not"), not a parse grammar the arm can be built against.

The unpinned aspects are concrete. (1) The separator glyph is never named: the real `H5b` `Deps.` line uses an en-dash (U+2013), and the `H5a`/`conventions.md` examples do too, but nothing states the en-dash is the canonical separator or that an ASCII hyphen / em-dash is rejected. (2) The example token formatting diverges from the real data: `H5a` and `conventions.md` write a range as a single code span (`V2a–V2d`, dash inside the backticks), whereas the live `H5b` `Deps.` writes two code spans joined by a bare dash (`` `V1a`–`V1b` ``). (3) The arm reads a *mixed* element list — `H5b`'s `Deps.` interleaves singletons (`H1a`, `H5a`, `M`) and range tokens — so the parser must first split comma elements, then classify each as singleton-or-range; that disambiguation order is unstated. (4) The suffix is assumed to be a single lowercase letter, with no statement of behaviour for a group that outgrows a single-letter suffix.

Because the arm expands tokens textually with no stated requirement to reject a token that fails to match the grammar, any of these drifts mis-expands silently rather than failing loudly. Today the corpus is internally consistent (single-letter suffixes, en-dash, the two-code-span form on the `H5b` side; the coverage-matrix closing-leaf column currently uses only comma-separated singletons), so an arm built directly against the live data works — but the grammar the arm depends on is not written down anywhere an implementer or a later editor can rely on.

## Issue introduction

**Verdict:** introduced-by-edit (multi-commit narrowing, gap not closed)
**Introducing commits:** `08bd641` (2026-06-11, "resolve 'H5b coverage Deps completeness has no mechanical backstop'"); `3a84938` (2026-06-12, "resolve 'H5a transitive-completeness arm range expansion'")
**History:** The transitive-completeness arm and its "expanding both sides' contiguous ranges" requirement first entered `H5a-closing-gate-automation.md` in `08bd641`, which added the range-expansion obligation but pinned no parse grammar (its example, `V1a–V18d`, even crossed group numbers). `3a84938` narrowed the grammar — it corrected the example to `V2a–V2d` and added the endpoint-group-equality and contiguous-letter-suffix-enumeration constraints — but left the separator glyph, the single-vs-two-code-span token formatting, the comma-vs-range disambiguation, and the malformed-token behaviour unspecified. The defect is the residue of an incompletely-closed prior fix, not a regression from a single edit.

## Solution Space

**Shape:** single

### Recommendation

Pin the leaf-ID range-token grammar in `conventions.md` §Leaf format's `Deps.` bullet (the existing home of the "Use ranges where contiguous" guidance), and have `H5a`'s transitive-completeness arm cite it rather than re-describe it. The grammar must fix, for a leaf-ID range token:

- The canonical separator is the en-dash `U+2013` between two backtick-delimited leaf-ID code spans (the live `` `V1a`–`V1b` `` form); an ASCII hyphen-minus or em-dash is not a range separator.
- A `Deps.`/closing-leaf cell is a comma-separated list whose elements are each either a singleton leaf ID or a range token; commas split elements first, then each element is classified.
- The suffix is a single lowercase letter; the two endpoints share the same `<group>` number (already stated in `H5a`) and the token enumerates the inclusive contiguous letter suffixes between them.
- A token that does not match this grammar is a CI failure of the arm, not a silently-dropped element — so a separator-glyph drift or an unparseable suffix reds the gate instead of mis-expanding.

`H5a`'s Adds keeps its current endpoint-group-equality and letter-suffix-enumeration sentence but points at the `conventions.md` grammar as the authoritative definition so the arm and the authoring rule cannot diverge. No spec edit is required — this is a plan-authoring convention, not a spec obligation. Honour the existing leaf-ID scheme (`<group><letter>`) per §Leaf format; do not introduce a new ID form.

## Consequence

**Severity:** advisory

A fresh implementer building the arm must infer the range-token grammar from one example and the live data, and two reasonable implementers can diverge on a malformed token — one rejecting it loudly, one skipping it silently. With no pinned grammar and no stated reject-on-mismatch behaviour, a later dash-glyph drift, a group that outgrows a single-letter suffix, or a token-formatting change in `H5b`'s `Deps.` would mis-expand without a guard, so a closing-leaf cell could pass vacuously (missing coverage ships) or redden spuriously.
