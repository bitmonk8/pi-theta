## Problem

The three Tests bullets in `V15i` and its paired tests leaf `V15i-T` open with plain-English labels — `Auto-export visibility:`, `Re-export with alias:`, and `Plain `import` is not re-exported:` — and cite the obligation they close only via a parenthetical rule reference. None carries any admitted anchor form (a `PREFIX-N` REQ-ID, a `cka-<n>` token, or `Convention:`). The `imports.md` §Visibility and §Re-exports obligations are un-anchored normative MUSTs whose only citable anchor is the `cka-<n>` token of the `coverage-matrix.md` row that enumerates them. As a result the tests `V15i-T` produces are untraceable by inline token to the obligations they assert, and an implementer reading the bullets reproduces the plain-prose form.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 6b7c7e8 (2026-06-18) — "pi-loom plan: resolve \"V15c bundles import resolution/diagnostics and export-visibility into one leaf\""
**History:** `git log --follow` on `docs/plan_topics/V15i-export-visibility.md` and `docs/plan_topics/V15i-T-export-visibility.md` shows a single commit each — 6b7c7e8 — which created both leaves with the three plain-prose Tests labels already in place. `git log -S 'Auto-export visibility'` shows the same prose first entered the corpus at 2316cac (2026-06-11) on `V15c`, and was carried verbatim into the new `V15i` / `V15i-T` leaves when 6b7c7e8 split export-visibility out of `V15c`. No version of the `V15i` leaves has ever cited a `cka` (or any) anchor token; the citation-format gap is present since the leaves' inception.

## Solution approach

Rewrite the three Tests bullets in both `V15i-export-visibility.md` and `V15i-T-export-visibility.md` so each cites the `imports.md` §Visibility + §Re-exports coverage-matrix `cka-<n>` token as its citation prefix, replacing the plain-English labels (`Auto-export visibility:`, `Re-export with alias:`, `Plain `import` is not re-exported:`). Keep each bullet's existing assertion text and its parenthetical rule reference. The `cka-<n>` number is allocated by the related coverage-matrix finding (which must land first); cite whatever token that row receives rather than inventing a number.

## Solution constraints

- Out of scope: the spec is read-only for this fix — do not edit `imports.md`.
- Out of scope: creation of the `coverage-matrix.md` §Visibility + §Re-exports row is owned by the related coverage-matrix finding; this fix consumes its `cka-<n>` token only.
