## Finding

"statically resolvable" / "static resolution" is a multi-page load-bearing qualifier (canonical at `invocation.md#static-resolution`, reused in `functions.md`, `tool-calls.md`, frontmatter, and ~six parse-diagnostic rows) that meets the glossary's admission criterion but has no entry.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `b4fbfac` ("pi-loom spec: resolve \"no central glossary\"", 2026-05-04) — the commit that created the glossary.
**History:** The spec corpus is git-tracked. `git log --diff-filter=A -- '*glossary.md'` shows the glossary was first created in `b4fbfac`; `git log -S 'final value' --` and `git log -S 'statically resolvable'` against the glossary file (both current and pre-move paths) return no commits, i.e. neither term has ever had a glossary entry. The terms themselves predate the glossary (`final value`/`return value` and `statically resolvable` were already in use across `invocation.md`/`functions.md`/`overview.md` when `b4fbfac` landed), so the omission has existed continuously since the glossary was introduced — it is not a regression from any later edit.

## Solution Space

**Shape:** single

### Recommendation

Add a `glossary.md` entry for **statically resolvable** (covering the noun form *static resolution*) summarising the predicate — the runtime can open, parse, and lower the callee during the calling loom's load pass — and pointing `See:` at `invocation.md#static-resolution`.

## Consequence

**Severity:** advisory

A reader cannot confirm from the glossary that "return value" and "final value" name one concept, and "statically resolvable" lacks the single-lookup definition the glossary promises for reused coinages. Both canonical sites (FN-5, `invocation.md#static-resolution`) still define the behaviour unambiguously, so implementers do not diverge — the gap is discoverability and terminology consistency, not behavioural correctness.
