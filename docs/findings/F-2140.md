## Finding

The `session_shutdown` teardown contract leans on a specific runtime behaviour of `chokidar`: that calling `discoveryWatcher.close()` (and `settingsWatcher.close()`) a second time on an already-closed watcher is tolerable. Three normative arms re-enter sub-step 4 ("Close watchers") against already-closed handles and accept whatever the second `.close()` does — the *idempotent*-clause **all-three-throw corner case** and the *Read-failure fallback* re-entry in `drain-state-contract.md`, and the drain-throw / read-failure fixtures (branches (j) and (m)) in `session-only-degraded-state.md`. Each pins the outcome on what the spec itself names "chokidar's acknowledged-unpinned double-close behaviour": the per-call `try`/`catch` of *Per-step isolation* routes any double-close throw into one additional `loom/host/session-shutdown-teardown-step-failed` row per failing call site, and no further compensation is specified.

The spec is explicit that this behaviour is unpinned. `implementation-notes.md` declares `chokidar` in the loom package's `dependencies` but states "this clause names the dependency, not its version," and `host-prerequisites.md` contrasts "the version-unpinned `semver` / `chokidar` choices" against loom's TypeScript build toolchain, which *does* carry a load-bearing version floor (`>=5.4.0`) precisely because a build-time gate relies on a specific `tsc` behaviour — and which `host-prerequisites.md` further protects with a re-confirmation obligation on every bump.

The asymmetry is the defect. `tsc`'s load-bearing behaviour is both pinned and gated against silent regression; `chokidar`'s double-close behaviour is load-bearing in the teardown acceptance yet carries no version floor, no behaviour-locking test, and no re-confirmation entry. A future `chokidar` upgrade that changed double-close semantics — for example, a `.close()` that on the second call rejects asynchronously (chokidar's `.close()` returns a Promise) rather than throwing synchronously, escaping the synchronous per-step `try`/`catch` — would silently invalidate the re-entry / all-three-throw / read-failure-fallback acceptance with no signal. Because `chokidar` is loom's own dependency rather than a Pi-supplied presupposition, loom controls when that upgrade lands, which makes a loom-side guard both feasible and the natural home.

**Verdict:** single introducing commit
**Introducing commit:** `52b9101` (2026-05-11, "pi-loom spec: resolve \"T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state\"")
**History:** The corpus is git-tracked. `git log -S 'double-close' -- docs/` and `git log -G 'unpinned double-close'` trace the earliest spec-body occurrence (excluding spec-review store commits) to `52b9101`, which introduced the *idempotent* clause and the *all-three-throw corner case* into the then-monolithic `pi-integration-contract.md`. `git show 52b9101` confirms that commit coined both "chokidar's unpinned double-close behaviour" and "chokidar's acknowledged-unpinned double-close behaviour," establishing the load-bearing reliance and simultaneously labelling it unpinned, with no accompanying pin, behaviour-locking test, or re-confirmation gate. Later commits (`2775382` adding throwing-`readDrainState` acceptance criteria; `f5e89f4` the size-cap split into `drain-state-contract.md` / `session-only-degraded-state.md`) propagated the same phrasing but did not introduce the gap. The reliance has been unguarded since `52b9101`.

## Solution Space

### Recommendation

Add a behaviour-locking fixture that double-closes a real `chokidar` watcher through the `session_shutdown` teardown path and asserts the spec's accepted outcome: the second `.close()` is absorbed and surfaces at most one `loom/host/session-shutdown-teardown-step-failed` row per failing call site, with nothing escaping the handler. The fixture MUST cover both the synchronous-throw path and the asynchronous-rejection path — `chokidar`'s `.close()` returns a Promise, and an escaped Promise rejection (rather than a synchronous throw) is the most plausible regression a future bump introduces. Run the fixture against the real library, not a `FakeFileWatcher`, so any `chokidar` upgrade that changes double-close semantics fails loom's own CI.

Pin this behaviour-locking obligation as a normative fixture in the teardown-contract test surface, referenced from the *all-three-throw corner case* and *Read-failure fallback* arms in `drain-state-contract.md`. State explicitly that the close calls must absorb both synchronous throws and asynchronous rejection. Add a one-line cross-reference from the `chokidar` dependency clause in `implementation-notes.md` to this fixture so the relied-on double-close behaviour is documented at the declaration site.

## Consequence

**Severity:** advisory

The spec is internally consistent today: the per-step `try`/`catch` absorbs a synchronous double-close throw, and the fixtures phrase row counts conditionally ("per double-close that throws"). The exposure is durability, not present-tense divergence — a `chokidar` upgrade that altered double-close semantics (notably an async rejection that escapes the synchronous catch) would silently break the teardown re-entry acceptance with no version floor, behaviour-locking test, or re-confirmation gate to catch it, unlike the parallel `tsc` floor the spec already pins and guards.
