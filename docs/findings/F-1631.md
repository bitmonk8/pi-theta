## Problem

The diagnostic code `loom/load/cross-format-collision` is emitted for every DISC-4 same-priority slash-name collision — both the same-format `.loom`-vs-`.loom` arm and the cross-format `.loom`-vs-Pi-owned-`.md` arm — yet the code name names only the cross-format arm. When an author triages a same-format collision in a project shipping no `.md` artefacts, the code reads as a category error and invites the wrong hypothesis (a stray `.md` template in the discovery union). DIAG-3 defers the rename to loom 2.0, so the code cannot change within loom 1.x. The spec already discloses the mismatch, but the disclosure is buried mid-cell in the Trigger paragraph, and no entry in the corpus records the queued rename target.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `839db71` (2026-05-04, "Diagnostic codes are declared as namespaces but never assigned to specific errors") and `d124b8c` (2026-05-05, "spec: same-priority slash-name collisions are uniform load-time errors (Decision #10)").
**History:** `839db71` first introduced the `loom/load/cross-format-collision` row with a Trigger column that read "A `.loom` and a Pi prompt template (`.md`) or subagent share the same slash name." — code name and trigger were aligned. The next day, `d124b8c` broadened the rule to also cover same-format (`.loom` vs `.loom`) collisions at the same priority without renaming the code (its commit body explicitly notes "loom/load/cross-format-collision row broadened to cover both shapes; no new code introduced"). At that point the code name became a misnomer for the new same-format arm. A later partial fix, `7e440c9` (2026-06-06, "resolve `cross-format-collision misnomer / DISC-4 heading scope`"), renamed the DISC-4 heading to "Slash-name collision rules" and appended the DIAG-3 self-disclosure sentence to the registry row, but did not lift the disclosure to a visible callout or pin a loom 2.0 rename target. The defect's residual surface — disclosure prominence and missing rename intent — has held in equivalent form since that commit.

## Solution approach

Promote the historical-token disclosure on the `loom/load/cross-format-collision` row in `code-registry-load.md` from Trigger-cell prose to a visible note adjacent to the row, and trim the duplicated sentence from the Trigger cell so the disclosure lives in one place (preserve the Trigger's scope-coverage sentence describing what the code fires for). Add a deferred-diagnostic-code-renames register section in `future-considerations/model-changes-and-non-goals.md` recording the loom 2.0 rename target (`loom/load/cross-format-collision` → `loom/load/slash-name-collision`), cross-linked back to DIAG-3 and to the disclosing row.

## Solution constraints

- The code identifier `loom/load/cross-format-collision`, its Message template, and its Hint string MUST remain unchanged — DIAG-3 defers the rename to loom 2.0, and V7b/V10a gate on the byte-exact code.
- The new deferred-renames section MUST NOT be counted among the `Forward-compatibility seams` inventory tallied on `overview-and-orientation.md` — it is a rename register, not a forward-compatibility seam.
