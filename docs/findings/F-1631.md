## Finding

`loom/load/cross-format-collision` is emitted for **every** DISC-4 same-priority slash-name collision: the loom-vs-loom case (two `.loom` files derived to the same slash name from the same priority — two packages, two settings entries, two `--loom` components) **and** the loom-vs-Pi-owned case (a `.loom` colliding with a Pi-owned `.md` prompt, `.md` skill, or another extension's command at the `session_start` cross-priority comparison). The code name names only the second arm, so when an author triages a `lint.loom` vs `lint.loom` collision in a project that ships no `.md` artefacts, the diagnostic code reads as a category error and invites the wrong hypothesis (a stray `.md` template somewhere in the discovery union).

The spec already self-discloses the mismatch — the `code-registry-load.md` Trigger cell ends with "The code name retains the historical `cross-format` token as a stable diagnostics-contract identifier per [DIAG-3](diagnostic-shape.md#diag-3); its scope is the full same-priority slash-name collision rule, not only the cross-format arm." — but the disclosure is buried mid-cell in a 600-word Trigger paragraph, and there is no entry anywhere in the corpus recording that a rename to a scope-accurate identifier (e.g. `loom/load/slash-name-collision`) is queued for the loom 2.0 cycle that DIAG-3 directs all renames to. DIAG-3 itself contains only the generic "renames are deferred to loom 2.0 migration" rule; nothing pins this specific candidate, so the disclosure is a free-floating note rather than a recorded migration intent.

The fix is bounded by DIAG-3: the code cannot be renamed within loom 1.x. The remediation is therefore to make the existing in-place disclosure prominent (lift it out of the Trigger-cell tail) and to record the rename target in the loom 2.0 / future-considerations corpus so the intent survives the next time the registry is re-examined.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `839db71` (2026-05-04, "Diagnostic codes are declared as namespaces but never assigned to specific errors") and `d124b8c` (2026-05-05, "spec: same-priority slash-name collisions are uniform load-time errors (Decision #10)").
**History:** `839db71` first introduced the `loom/load/cross-format-collision` row with a Trigger column that read "A `.loom` and a Pi prompt template (`.md`) or subagent share the same slash name." — code name and trigger were aligned. The next day, `d124b8c` broadened the rule to also cover same-format (`.loom` vs `.loom`) collisions at the same priority without renaming the code (its commit body explicitly notes "loom/load/cross-format-collision row broadened to cover both shapes; no new code introduced"). At that point the code name became a misnomer for the new same-format arm. A later partial fix, `7e440c9` (2026-06-06, "resolve `cross-format-collision misnomer / DISC-4 heading scope`"), renamed the DISC-4 heading to "Slash-name collision rules" and appended the DIAG-3 self-disclosure sentence to the registry row, but did not lift the disclosure to a visible callout or pin a loom 2.0 rename target. The defect's residual surface — disclosure prominence and missing rename intent — has held in equivalent form since that commit.

## Solution Space

**Shape:** single

### Recommendation

Promote the existing self-disclosure on the `loom/load/cross-format-collision` row from in-cell prose to a visible callout, and record the rename target in the future-considerations corpus.

Concretely:

1. In `docs/spec_topics/diagnostics/code-registry-load.md`, immediately after the `loom/load/cross-format-collision` table row, emit a `**NOTE:**` paragraph (sitting between the row and the next row, outside the table) with text along the lines of:

   > **NOTE.** The `cross-format` token in this code name is historical: the code fires for the full DISC-4 same-priority slash-name collision rule, including the same-format `.loom`-vs-`.loom` arm. Renaming is deferred to loom 2.0 per [DIAG-3](diagnostic-shape.md#diag-3); the rename target is recorded under [Future Considerations — Deferred diagnostic-code renames](../future-considerations/model-changes-and-non-goals.md#deferred-diagnostic-code-renames). Triage on the rendered `<paths>` list, not on the code's `cross-format` token.

   Trim the corresponding sentence currently buried in the Trigger column ("The code name retains the historical `cross-format` token …") so the disclosure lives in exactly one place. Preserve the Trigger column's scope-coverage sentence ("Covers both the same-format case … and the cross-format case …") since that defines what the code actually fires for.

2. In `docs/spec_topics/future-considerations/model-changes-and-non-goals.md`, add a new section anchor `<a id="deferred-diagnostic-code-renames"></a>` titled "Deferred diagnostic-code renames" listing the loom 2.0 rename intents. The initial list carries this finding's target — `loom/load/cross-format-collision` → `loom/load/slash-name-collision` — together with the two sibling DIAG-3-deferred renames already surfaced by the same review (`loom/host/session-shutdown-reload-teardown-timeout` → `…/shutdown-await-timeout` and `details.event.loom` → `details.event.slash_name`). Cross-link each entry back to DIAG-3 and to the row whose disclosure points at this section.

Edge cases for the implementer:

- The code identifier itself (`loom/load/cross-format-collision`), the rendered Message template (`slash name '<name>' collides at the same priority: <paths>`), and the Hint string are unchanged. `V7b`'s DIAG-2/DIAG-3 closing gate continues to find the row at the same byte-exact code; `V10a`'s DISC-4 test continues to assert the same code on both arms.
- Keep the new NOTE paragraph outside the table proper (a Markdown table row cannot host a separate block). A blank line plus the bolded NOTE label is the established pattern.
- The new future-considerations section is a register of deferred renames, not a deferred-extension seam, so it sits under "Tooling deferrals (no loom 1.0 impact)" or as a peer section — not under "Surface extensions (loom 1.0 leaves a seam)", which is the inventoried seam set the `Forward-compatibility seams` tally on `overview-and-orientation.md` counts. Place it so it is excluded from that tally.

## Consequence

**Severity:** advisory

An author whose `lint.loom` collides with another package's `lint.loom` sees `loom/load/cross-format-collision` and a message naming both `.loom` paths; the diagnostic-code identifier directly contradicts the file types it lists, slowing triage and inviting a wrong hypothesis (a hidden `.md` template). The rendered Message + Hint already give the operator enough to act, so no implementer diverges and no implementation breaks; the cost is author friction at the diagnostic site and a lost forward-compatibility hook (no recorded loom 2.0 rename target) for the next time the registry is revisited.
