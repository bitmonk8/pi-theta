## Finding

The degraded-state branch accepts that several corner cases breach the
"`loom/host/session-shutdown-runtime-degraded` fires at most once per
extension-instance lifetime" bound, and in each place it softens that breach
with the standing rationale that "operator deduplication on
`(code, details.event.reason)` collapses the resulting duplicate rows." This
appears four times across two pages: in `drain-state-contract.md` it backs the
*all-three-throw corner case* (the idempotent clause), the PIC-30 follow-up
session-only re-emission paragraph, and the PIC-31 *Read-failure fallback*; in
`session-only-degraded-state.md` it backs the *Idempotent re-entry and the
all-three-throw corner case* section.

The rationale fails on its own terms for the exact scenario the corner cases
describe. Each corner case re-emits on a **follow-up session-only
`session_shutdown`** against the same extension instance — and a session-only
swap can carry any of `"new"`, `"resume"`, `"fork"`. The diagnostic-emission
predicate populates `details.event.reason` with the *captured reason of that
invocation*, so an initial degraded emission on `"new"` followed by a re-entry
emission on `"fork"` produces two rows that differ in `details.event.reason`. A
dedup key of `(code, details.event.reason)` buckets them separately by
construction — it does **not** collapse them. The collapse claim holds only for
the narrower case where the follow-up reason equals the original; the
cross-reason re-entry that the same paragraphs explicitly admit is precisely the
case the key cannot collapse.

A secondary weakness compounds this: the `runtime-degraded` row is written to
`console.error` (it is a write-only stdio path; loom 1.0 ships no consumer/query
API), so the "operator deduplication" the rationale leans on is external
log-pipeline tooling the operator is assumed to run, not anything loom provides.
The spec does contemplate "operator-tooling consumers" elsewhere
(`diagnostics/diagnostic-shape.md`), so this premise is at least internally
declared — but combined with the reason-equality gap it means the stated
mitigation overstates how much of the breach is actually absorbed.

The runtime behaviour is not in dispute: the spec already mandates the runtime
MUST NOT add a suppression seam, so duplicates are emitted as designed. The
defect is in the *acceptance rationale* — it claims a mitigation (dedup collapse)
that does not apply to the multi-reason re-entry path it is invoked to excuse.

## Solution Space

**Shape:** single

### Recommendation

Correct the dedup rationale at all four occurrence sites (the three in
`drain-state-contract.md` — idempotent clause, PIC-30, PIC-31 — and the one in
`session-only-degraded-state.md`) with a single coherent edit:

- State the collapse precondition explicitly: the `(code, details.event.reason)`
  key collapses duplicate rows **only when both the code and the captured reason
  are identical**, and acknowledge that a cross-reason re-entry (e.g. an initial
  `"new"` followed by a `"fork"`) emits rows that differ in
  `details.event.reason` and are therefore **not** collapsed by this key.
- Demote the dedup mention from a load-bearing justification to a non-normative
  illustration: the operative loom 1.0 position is that these corner-case
  duplicates are accepted residual gaps on the existing footing (the runtime
  MUST NOT add a suppression seam), with operator deduplication offered only as a
  best-effort same-reason convenience for external log tooling, not as a
  guarantee the bound is restored.

Keep the wording aligned across the four sites (the spec already flags these
paragraphs as cross-referenced); a future edit to one must sweep all four in the
same commit. No change to runtime behaviour, diagnostic shape, or acceptance
fixtures is implied — only the rationale text changes.

## Consequence

**Severity:** advisory

The spec's stated acceptance argument for the at-most-once-bound breaches
overstates its mitigation: in the multi-reason session-swap corner cases the
duplicate `runtime-degraded` rows are **not** collapsed by the cited key, so an
operator (or log pipeline) sees distinct un-deduplicated rows the rationale
implies are absorbed. No two implementers diverge in runtime behaviour (the
"MUST NOT suppress" rule is unaffected); the cost is a residual-gap rationale
that is partly false where it is most relied upon.
