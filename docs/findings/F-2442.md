## Finding

The slash-command entry bullet under *Forwarding into `loomAbort`* in `cancellation.md` says: "The runtime MUST tolerate `ctx.signal` being `undefined` at slash-command entry … and MUST NOT depend on its truthiness for any pre-turn checkpoint." This pairs an input condition (`ctx.signal === undefined` at entry) with a prohibition (no truthiness dependence) but never states the positive observable behaviour that must result. The verb "tolerate" is not defined anywhere in the corpus, so a conformance author has no pass/fail oracle: there is no statement that dispatch proceeds, that no exception or diagnostic is raised, or that cancellation detection is deferred to a later observation.

The prohibition half ("MUST NOT depend on truthiness") rules out one wrong reading — treating undefined as already-aborted or as "cancellation impossible" — but it does not pin the behaviour. An implementer must infer the intended outcome (the loom dispatches normally and only forwards `loomAbort.abort()` once an event handler later observes an aborted `ctx.signal` or an `agent_end`) from surrounding prose rather than reading it from the rule itself. The defect is a missing Then-clause on an otherwise normative MUST, not a contradiction with other text.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 8395c03 — pi-loom spec: resolve "Cancellation source ctx.signal is undefined when a slash-handler starts" (2026-05-04, Thomas Andersen)
**History:** The slash-command entry bullet, including the "MUST tolerate `ctx.signal` being `undefined` … MUST NOT depend on its truthiness" clause, was authored in 8395c03 when the *Signal source* / *Forwarding into `loomAbort`* structure was introduced to resolve a prior finding about `ctx.signal` being undefined when a slash-handler starts. That fix established the input condition and the prohibition but never added the observable outcome, so the testability gap has been present since the clause first entered the corpus. The wording survived the later `docs/` relocation (31ff060) unchanged.

## Solution Space

**Shape:** single

### Recommendation

Append the observable outcome to the slash-command entry bullet so the rule is testable. State that when `ctx.signal` is `undefined` at slash-command entry the runtime proceeds with dispatch normally — it does not throw, emit a diagnostic, or treat the undefined value as either aborted or as "cancellation impossible" — and that cancellation detection for the slash-command path is deferred to the first event-handler observation of an aborted `ctx.signal` (or an `agent_end` reporting a user-cancelled turn), which then fires `loomAbort.abort(...)` per the existing forwarding rule.

Phrase it as a falsifiable conformance statement, e.g.: "A slash-command dispatch with `ctx.signal === undefined` at entry runs to its first real checkpoint without exception or diagnostic; the only path that can subsequently abort the run is an event-handler observation of an aborted signal or an `agent_end` trigger." Keep the existing "MUST NOT depend on truthiness" prohibition — it now reads as the rationale for the positive rule rather than standing alone.

## Consequence

**Severity:** advisory

A conformance author cannot write a falsifiable test for an undefined `ctx.signal` at slash-command entry, because the rule names no observable. Two implementers will likely still converge on "dispatch proceeds" given the surrounding source-of-truth prose, so behavioural divergence is low; the concrete loss is an untestable MUST and a small risk that one implementer adds a defensive guard that subtly diverges (e.g. emitting a diagnostic) without violating the letter of the rule.
