## Finding

The *Outcome routing summary* paragraph in `host-interfaces-core.md` restates the late-settlement swallowing-handler obligation at the tool-execution boundary: the runtime "MUST keep a swallowing handler attached to the underlying `execute()` Promise … installed at construction time before the first microtask boundary, so that a late rejection on the discarded path does **not** surface as a Node `unhandledRejection` process event." The surrounding three-clause discard rule is cross-referenced precisely — each clause carries a REQ-ID anchor (`CNCL-1` no-rebind, `CNCL-2` no-second-`Err`, `CNCL-3` no-second-`RuntimeEvent`) that a test or conformance fixture can pin. The swallowing-handler / `unhandledRejection` half is not: at this restatement site it has no anchor and no linked acceptance criterion, so a reader cannot tell *how* the "no `unhandledRejection`" guarantee is meant to be verified.

Two distinct gaps drive this. First, the obligation is asymmetric to its `CNCL-1/2/3` siblings: those are numbered, anchored REQ-IDs, whereas the swallowing-handler attachment / total-suppression rule is carried only as the un-anchored code-keyed obligation `cka-33` (per the plan's coverage matrix), so nothing at the host-interfaces-core.md restatement points at a citable acceptance criterion. Second, the "installed … before the first microtask boundary" timing constraint has no observable correlate distinct from the `unhandledRejection`-absence outcome — a conformance test can only observe that no `unhandledRejection` fires on the discarded path; it cannot directly witness the microtask-relative attachment moment. The owning paragraph in `cancellation.md` does name the test substrate (the `Checkpoint` seam, "verifiable without relying on JS microtask scheduling"), but the `host-interfaces-core.md` restatement neither cross-references that substrate nor flags the timing clause as a non-observable mechanism.

The behaviour itself is unambiguous (no `unhandledRejection` on the discarded path), so this is a spec-text testability/anchoring gap rather than a behavioural divergence: an implementer can still build and test the property — the plan already does, via `cka-33`'s closing leaves — but the restatement site reads as a free-standing MUST with no acceptance hook and a timing clause that cannot be tested as written.

## Solution Space

**Shape:** single

### Recommendation

At the `host-interfaces-core.md` *Outcome routing summary* restatement of the swallowing-handler MUST, replace the free-standing timing MUST with a cross-reference to the owning obligation and its test substrate: cite `cancellation.md`'s *Race semantics — swallowing-handler attachment on every abandonable Promise* paragraph (which already names the `Checkpoint` seam as the deterministic-test substrate that makes the no-`unhandledRejection` assertion verifiable without depending on JS microtask scheduling), so the restatement inherits a citable acceptance hook rather than asserting an anchorless obligation. Demote the "installed at construction time before the first microtask boundary" clause to a non-normative mechanism note at this site, since its only observable correlate is the `unhandledRejection`-absence outcome the cross-referenced acceptance criterion already pins. If the editors prefer symmetry with `CNCL-1/2/3`, promote the swallowing-handler obligation in `cancellation.md` to a numbered REQ-ID with an explicit `Checkpoint`-seam acceptance criterion (migrating `cka-33` from a code-keyed row to a numbered REQ-ID row and re-pointing the `cka-33` closing-leaf citations); the host-interfaces-core.md restatement then cites that anchor. Either way, the observable obligation (late settlement discarded along all three side channels; never escapes as `unhandledRejection`) is what carries the acceptance criterion; the construction-time/first-microtask wording is the mechanism, not the test target.

## Consequence

**Severity:** advisory

The "no `unhandledRejection`" guarantee at the tool-execution boundary is stated without a citable acceptance criterion and with a microtask-timing clause that has no observable correlate, so a conformance author working from `host-interfaces-core.md` alone has no test oracle to pin and may treat the timing MUST as testable-as-written. Behaviour is unambiguous and the plan already covers the property, so a working system is still buildable; the gap is spec-text verifiability and anchoring symmetry.
