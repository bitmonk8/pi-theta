## Finding

Version-bump step 5 states the routing contract for its two snapshot/probe sub-triggers as: *"each failure-output entry the implementation emits MUST be unambiguously routable by a reader to exactly one of trigger (i) or trigger (ii) without an out-of-band translation table — and the implementation MUST expose a stable test-output discriminator on each such failure that satisfies that routability obligation."* The load-bearing predicate — "unambiguously routable by a reader … without an out-of-band translation table" — is a human-judgment criterion. There is no deterministic test a conformance reviewer (or a gate) can run to decide whether a given set of discriminator strings satisfies it; two reviewers could disagree on whether a pair of discriminators is "unambiguously routable."

The step elsewhere does the right thing mechanically — it requires the two sub-triggers to surface as "orthogonal, individually-discriminable failures," it makes the implementation own and pin the discriminator literal strings into "permanent test invariants," and the trigger-(ii) type-equality sub-trigger already keys on the grep-able `tsc` brand string `loom/typecheck/session-shutdown-reason-snapshot`. But none of that pins the actual checkable property that "routable by a reader" is gesturing at: that the per-trigger discriminators are demonstrably distinct. The normative conformance bar is left as the soft predicate while the genuinely mechanical obligations sit beside it without being elevated to the predicate's role.

The fix is local: keep the implementation ownership of the literal strings, demote the readability phrasing to non-normative rationale, and pin a mechanical predicate the suite can assert.

## Solution Space

**Shape:** single

### Recommendation

Demote the "unambiguously routable by a reader … without an out-of-band translation table" phrasing to a non-normative rationale clause, and pin a mechanical predicate as the normative conformance criterion in its place:

- The discriminator token a failure carries for trigger (i) and the token it carries for trigger (ii) MUST NOT be byte-equal.
- Each trigger's failure-output discriminators carry a fixed, disjoint per-trigger prefix token (e.g. one reserved prefix for trigger (i) failures, a distinct one for trigger (ii)), and a permanent test invariant asserts the prefix is present and that the two triggers' prefixes are disjoint.

Keep the existing implementation ownership of the literal strings and the "pinned into permanent test invariants" obligation; only add the distinctness/disjoint-prefix assertion as the checkable bar.

Edge case the implementer must watch: the trigger-(ii) type-equality sub-trigger discriminator is the `tsc`-emitted brand string `loom/typecheck/session-shutdown-reason-snapshot`, not an implementation-owned string. The mechanical predicate must span the full discriminator set across both triggers — the implementation-owned tokens plus this brand string — so the disjointness assertion treats the brand string as trigger (ii)'s fixed prefix carrier rather than excluding it.

## Consequence

**Severity:** advisory

No deterministic gate or reviewer procedure can confirm that an implementation's per-trigger discriminators meet the routability bar; conformance rests on eyeball judgment. Two implementations could each pin discriminators they deem "routable" that a third party finds confusable, and the spec's own test-invariant requirement would not catch it. Implementers can still ship a working bump procedure, so the gap is in verifiability rather than behaviour.
