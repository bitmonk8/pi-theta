## Finding

`V9f`'s **Adds** enumerates six behaviours, two of which have no asserting **Tests** bullet in either `V9f` or its paired `V9f-T`: (i) the no-`unregisterTool` rule — the design invariant that loom-synthesised tools gain and lose model visibility through the `PIC-17` snapshot/restore active-set machinery and the cached `pi.registerTool` entries are intentionally permanent, so the runtime never relies on an unregister-style mutation (`tool-registration-lifetime.md` §Tool-registration lifetime and visibility: "Pi's tool registry has no `unregisterTool` API … entries persist for the process lifetime"); and (ii) the `Type.Unsafe` schema bridge that lowers a tool's canonical `parameters` schema into the registered `ToolDefinition`.

The current `V9f`/`V9f-T` Tests assert `PIC-8` (restore-failure protocol), `PIC-19` (snapshot/swap-install-failure protocol), callable-set visibility plus `registration-cache-collision` disambiguation, and the `ToolDefinition.label` derivation. None of these exercises whether the runtime avoids unregister-style teardown, nor whether the lowered schema is faithfully bridged into the registered `ToolDefinition.parameters`. Both behaviours can therefore regress while `V9f`'s exit gate stays green.

These are two independent coverage gaps that happen to live in the same leaf; each is closed by its own assertion and neither depends on the other.

The later commit cfd76f7 (2026-06-12) added the `ToolDefinition.label`-derivation Tests bullet but did not touch either gap. The defect is original to the leaf, not introduced by a subsequent edit.

## Solution Space

**Shape:** multiple

This finding carries two independent assertion obligations. Resolve both; the options below are one per obligation, not competing approaches. Per the per-phase TDD ritual, each new Tests bullet lands red in `V9f-T` first, then `V9f` turns it green and the `V9f` **Ships when** condition is updated to name the new observable.

### Option A — assert the no-`unregisterTool` rule

**Approach.** Add a Tests bullet to `V9f-T` that drives a register → visibility-gate → teardown cycle against the Pi tool-surface seam (the `ExtensionAPI` stand-in used elsewhere in `V9f`'s tests) and asserts no unregister-style mutation is invoked — visibility is surrendered only via `pi.setActiveTools(snapshot)` and the cached `pi.registerTool` entry persists.

**Plan edits.** New Tests bullet in `V9f-T`; mirror in `V9f` and extend `V9f`'s Ships when to name the assertion.

**Spec edits.** None — `tool-registration-lifetime.md` already states the rule.

**Pros.** Bounded; pins the spec's "entries persist for the process lifetime" invariant as a regression guard.

**Cons.** At the loom 1.0 Pi-SDK pin no `unregisterTool` member exists on the surface, so the assertion guards against a future code path rather than a present one.

**Risks.** Low — the spy must target the same seam the runtime calls, not a real Pi surface.

### Option B — assert the `Type.Unsafe` schema bridge

**Approach.** Add a Tests bullet to `V9f-T` asserting that a lowered tool `parameters` schema is bridged into the registered `ToolDefinition.parameters` via `Type.Unsafe` — i.e. the materialised `ToolDefinition` carries the lowered canonical schema (the same canonical form that feeds the `registration-cache` slug).

**Plan edits.** New Tests bullet in `V9f-T`; mirror in `V9f` and extend `V9f`'s Ships when.

**Spec edits.** None — the bridge is described in `tool-registration-lifetime.md` and `Type.Unsafe` callability is probed by `V9a`.

**Pros.** Closes the schema-fidelity gap so a malformed bridge is caught at `V9f`'s gate rather than downstream at a consumer (`V6c`, `V14a`, `V15d`).

**Cons.** Requires constructing a representative lowered schema fixture; larger than Option A.

**Risks.** Low — assert against the canonical-form bytes already used for the cache slug to avoid re-serialisation drift.

### Recommendation

Implement both. Land Option A first (the bounded never-invoked guard), then Option B (the schema-bridge fidelity assertion), so the larger bridge-materialisation test lands on a baseline where the teardown-cycle scaffolding already exists. Both bullets land red in `V9f-T` before `V9f` turns them green.

## Consequence

**Severity:** advisory

Two behaviours named in `V9f`'s Adds ship with no test witness: a code path that grows an unregister-style teardown (e.g. once a future Pi exposes `pi.unregisterTool`), or a `Type.Unsafe` bridge that materialises a wrong-shaped `ToolDefinition.parameters`, would pass `V9f`'s exit gate undetected. The leaf is still implementable; the gap is missing regression coverage, not a contradiction or an unpickable leaf.
