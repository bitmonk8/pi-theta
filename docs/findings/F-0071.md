## Finding

The cancellation spec ([`spec_topics/cancellation.md`](../../../docs/spec_topics/cancellation.md) — *Race semantics — swallowing-handler attachment on every abandonable Promise*) requires that a late settlement landing after a checkpoint has surfaced `cause: "cancelled"` is suppressed along all three side channels, including the assertion that **no Node `unhandledRejection` process event fires**. The plan turns that obligation into a test bullet at every abandonable-Promise site: `V13c-T`/`V13c` (`@`-query provider Promise), `V14a-T`/`V14a` (code-side `execute()` Promise), `V15a-T`/`V15a` (`invoke` child top-level Promise), `V9i-T`/`V9i` (subagent `AgentSession.abort()` Promise), and `V17a-T`/`V17a` (the `Checkpoint`-seam substrate). Each bullet asserts "no Node `unhandledRejection`".

Asserting the *absence* of an `unhandledRejection` is not a free observation. Vitest's default behaviour is to **fail the suite** when an unhandled rejection escapes, so a test that wants to observe and assert "none fired during the abandoned-Promise window" needs a deliberately set-up, per-test process-level observer: registered before the test, scoped to it, reset afterward, and configured to *record* an observed rejection for the assertion rather than auto-failing the run. No plan leaf enumerates where that observation/reset capability comes from. The harness leaves that would naturally own it — `H4a` (the end-to-end harness and its self-check) and `H4b` (the shared response-programming surface every harness-driven leaf consumes, including the abort-injection point (e)) — define scripting inputs and fidelity-contract self-checks but never an `unhandledRejection`-observation capability. Neither do the swallowing-handler bullets name a harness affordance they rely on; they cite only the `Checkpoint` seam (`V8a`) for landing the late settlement.

Because the capability is undefined, each of the five leaves must independently invent its own process-level observer wiring. A careless wiring — asserting against an observer that was never registered, or one that does not survive the microtask window in which the late rejection lands — yields a "no `unhandledRejection`" assertion that passes vacuously, so the swallowing-handler test goes green without ever exercising the suppression the spec mandates.

## Solution Space

**Shape:** single

### Recommendation

Pin the per-test `unhandledRejection`-observation/reset capability in one harness leaf and have the swallowing-handler bullets rely on it rather than each inventing its own observer.

- Add the capability to `H4a`'s **Adds.** field (the harness owner; it already owns the self-check and side-channel observation surface). State that the harness provides a per-test affordance to observe Node `unhandledRejection` process events: registered before the test body, scoped to that test, reset afterward, and configured so an observed rejection is captured for assertion rather than auto-failing the suite — so a test can affirmatively assert that none fired during the abandoned-Promise window. (If `H4a` is split — see the related too-large finding — this capability lands in whichever split half owns the harness substrate; it is harness infrastructure, not part of `H4b`'s scripting-input surface, though `H4b` is an acceptable home if the split places the side-channel observation contract there.)
- In the swallowing-handler Tests bullets of `V13c-T`/`V13c`, `V14a-T`/`V14a`, `V15a-T`/`V15a`, `V9i-T`/`V9i`, and `V17a-T`/`V17a`, reference the harness-provided observation capability as the source of the `unhandledRejection`-absence assertion (alongside the existing `Checkpoint` seam (`V8a`) reference for landing the late settlement), so each leaf's "no Node `unhandledRejection`" assertion is anchored to one shared affordance.

Edge case the implementer must watch: the observer must be installed before the Promise-construction site under test (and before the first microtask boundary the spec calls out), so a rejection already queued for `unhandledRejection` is still observable; an observer registered lazily inside the test body can miss it and make the assertion vacuous.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on how to observe the absence of an `unhandledRejection`, and a wiring that asserts against an unregistered or mis-scoped observer passes green without exercising the suppression — the swallowing-handler gate that is meant to redden a substrate-bypassing build can pass vacuously instead, leaving the spec's three-channel suppression obligation untested at all five sites.
