## Problem

The `V3a` implementation leaf and its paired `V3a-T` test leaf both title themselves "Expression evaluator and stdlib", but `V3a`'s Adds defers the standard-library member surface to `V3e` ("The `string`/`array<T>`/`object` stdlib members build on this interpreter in `V3e`"), and `V3e` is a separate leaf titled "Expression stdlib members" that owns those members. The "and stdlib" clause names scope the leaf does not own; it appears at three sites — the `V3a` heading, the `V3a-T` heading, and the `docs/plan.md` §V3 TOC link text (line 65). A reader scoping work from the heading rather than the Adds is told `V3a` covers stdlib, contradicting the leaf body and the `V3a`→`V3e` ownership split.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** cb8737a — "pi-loom plan: resolve \"Systemic leaf over-bundling across the plan corpus\"" (2026-06-12)
**History:** At the initial plan build (c6a664e) the "and stdlib" heading was accurate: `V3a`'s Adds then read "…object/array construction, and the string/array/object stdlib members", and no `V3e` leaf existed. Commit cb8737a split the stdlib member surface out into the new `V3e` leaf and rewrote `V3a`'s Adds to defer it ("…build on this interpreter in `V3e`"), but left the `V3a` heading, the `V3a-T` heading, and the `docs/plan.md` TOC link still carrying "and stdlib". The defect is the stale heading left behind by that split refactor.

## Solution approach

Drop the "and stdlib" clause from both leaf headings (`V3a-expression-evaluator.md` and `V3a-T-expression-evaluator.md`, line 1) and from the `docs/plan.md` §V3 TOC link text (line 65), so the titles read "Expression evaluator" / "Expression evaluator (tests)" and match the leaf's core-evaluator scope while `V3e` owns stdlib. Edit only the TOC link text, not its target; no file rename is needed since the slugs already omit "stdlib".
