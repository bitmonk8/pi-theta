## Finding

The session-context truncation walk calls `estimate()` per message but neither PIC-16 nor the walk specifies behaviour if `estimate()` throws; the walk's disposition (abort, surface an Err, or treat as over-budget) is undefined.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commits:**
- `8a6d15f` (2026-05-06) "pi-loom spec: resolve 'FileSystem seam interface signature lives outside the spec corpus'" — brought the `FileSystem` seam signature into the spec corpus with `writeText` as a bare `Promise<void>` and no `.code` note.
- `d826fae` (2026-05-04) "pi-loom spec: resolve 'Schema cache ownership is unspecified…'" — introduced the `SchemaValidator` seam without a `compile()`-throw disposition.
- `1f37b15` (2026-06-05) "pi-loom spec: resolve 'Session-context truncation — no TokenEstimator DI seam…'" — introduced the `TokenEstimator` seam without an `estimate()`-throw disposition.

**History:** Each gap is an omission that has been continuously present since the corresponding seam was authored — no commit ever introduced the missing contract. `git log -L`/`-S` on `host-interfaces-services.md` confirms the `writeText` line has carried a bare `Promise<void>` with no rejection comment since `8a6d15f`, while later commits pinned the read-sibling `.code` contracts around it (`f2a5d92` added `readBytes`, `2a3a1a1` the `readdir` encoding note, `6ab9f7b`/`a765f6c` the `realpath` guarantees), sharpening the `writeText` asymmetry without touching it. PIC-16 was subsequently refined by `341bcd9` (per-message determinism precondition) but the throw path remained unaddressed; PIC-11 likewise accreted cache/collision rules without ever naming a `compile()`-throw disposition.

## Solution Space

**Shape:** single

### Recommendation

State the truncation-walk behaviour on an `estimate()` throw at the §Session-context truncation site (and/or as a PIC-16 contract bullet) — e.g. the throw aborts binder-input construction and surfaces as the relevant loom-side binder failure, with no partial transcript emitted. Keep it consistent with PIC-16's deterministic-pure-function posture (a throw is a host/internal failure, not a normal walk outcome).

## Consequence

**Severity:** correctness

Each unstated disposition lets two reasonable implementations diverge on an error path: a `compile()` throw, a `writeText` rejection, or an `estimate()` throw could variously crash the runtime, surface a loom-observable Err, or be silently swallowed. The `writeText` gap is the mildest while no consumer exists, but it leaves a "load-bearing write surface" with no rejection mapping; the `estimate()`-throw gap sits on a path that runs on every `bind_context: session` binder call.
