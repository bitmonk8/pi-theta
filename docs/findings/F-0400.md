## Finding

`V15c` (Imports — `.warp` library files) carries two concern groups that ship and verify independently. The first is the import **resolution and diagnostics path**: the `Resolver` seam, the `IMP-1` resolver-failure contract (`loom/load/unresolvable-warp-path`), the parse diagnostics `loom/parse/warp-top-level-statement` and `loom/parse/import-non-warp-extension`, the load diagnostic `loom/load/import-cycle` (with `import-unknown-symbol` / `import-name-collision`), and the resolver success-path baseline. The second is **export-visibility semantics**: auto-export of every top-level `schema`/`enum`/`fn`, aliased re-export (`export { A as B } from`), and the negative re-export rule (a plain `import` is not re-exported downstream). Across the implementation and paired test leaf this is 8+ separately-assertable scenarios under a single `Ships when` gate.

The convention for vertical slices is that "each leaf inside a slice is the smallest feature that can ship and be tested independently" (`conventions.md` §phase categories). The visibility/re-export rules depend on resolution (they observe the resolved file's declarations) but resolution and its diagnostics ship and test with no dependency on visibility, so the two are genuinely separable. Bundled under one `Ships when`, partial completion — for example resolution and all diagnostics done, visibility not yet wired — produces no observable shippable state, and the single-pass review surface is correspondingly larger.

This is borderline rather than severe: the leaf maps to one coherent spec topic (`imports.md`) and is implementable as written. The defect is granularity, not correctness — an implementer can produce a working leaf from it today.

## Solution Space

**Shape:** single

### Recommendation

Split `V15c` so the resolution/diagnostics path keeps the `V15c` ID and the export-visibility semantics move to a new leaf depending on it:

- **Retain `V15c` (Imports — resolution and diagnostics).** Keep the `Resolver` seam, `IMP-1` (`loom/load/unresolvable-warp-path`), `loom/parse/warp-top-level-statement`, `loom/parse/import-non-warp-extension`, `loom/load/import-cycle` (with `import-unknown-symbol` / `import-name-collision`), and the resolver success-path baseline. Strike the "auto-export, re-exports" clause from **Adds**, and strike the "Auto-export visibility", "Re-export with alias", and "Plain `import` is not re-exported" bullets from **Tests** and their matching `Ships when` clauses.
- **Add a new leaf `<new>` — `.warp` export-visibility / re-export semantics**, with `Deps.` `V15c` plus its own `<new>-T`. It owns the `imports.md` Visibility rule (every top-level `schema`/`enum`/`fn` is implicitly exported) and the Re-exports rule (both halves: aliased `export { A as B } from` visibility, and the negative plain-`import`-not-re-exported rule). Move the three visibility/re-export **Tests** bullets here; its `Ships when` asserts an auto-exported symbol resolves in a downstream importer and an `export … from` re-export is visible while a plain `import` is not.
- **Split `V15c-T` correspondingly:** the resolution/diagnostics + resolver-success bullets stay in `V15c-T`; the three visibility/re-export bullets move to the new `<new>-T` leaf.
- **Link the new leaf** into `docs/plan.md` §V15 — Invocation and imports, in dependency order after `V15c`.
- **Add the new leaf to `H5b`'s Deps** so the transitive-completeness coverage-producing set stays complete (per `plan.md` How-to-use step 2): it becomes the closing leaf for the `imports.md` Visibility and Re-exports un-anchored MUSTs.
- **Leave `V15b`, `V15b-T`, `V15e`, `V15e-T` Deps on `V15c` unchanged**, and leave the `IMP-1 | V15c` coverage-matrix row unchanged — those consumers need the import-edge graph / cycle detection / `IMP-1` contract, all of which stay in the retained `V15c`.

The new leaf's ID is the implementer's to allocate under the project's leaf-ID scheme (`<new>` here); pick it so it falls within `H5b`'s `V15…` Deps range or extend that range when adding it.

Edge cases the implementer must watch: the new visibility leaf depends on `V15c` because visibility is only observable once resolution binds the resolved file's declarations; `import-unknown-symbol` and `import-name-collision` stay with `V15c` (they fire at parse/resolve time against the resolved file's declaration set, not at downstream-visibility time).

## Consequence

**Severity:** advisory

If shipped unsplit, the leaf works but its single `Ships when` gate makes partial progress unobservable — an implementer cannot land and tag the resolution/diagnostics behaviour separately from the visibility/re-export behaviour, and a reviewer absorbs both concern groups in one pass. No spec rule goes uncovered and no downstream leaf is blocked; the cost is review ergonomics and coarse completion granularity.
