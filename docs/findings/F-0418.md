## Finding

The paired leaves `V4e` and `V4e-T` are internally inconsistent in how they
name the concept they exist to cover. The leaf title (`Pre-evaluation
failures`) and the **Adds** bullet (`The closed pre-evaluation failure set
…`) use the full term `pre-evaluation`, which is also the only form the spec
uses — `docs/spec_topics/errors-and-results/error-model.md` writes
`pre-evaluation` in every occurrence and never the clipped form. The
**Tests** bullets (`ERR-1` … `ERR-7`, `ERR-16`) and the **Ships when** bullet
instead use the shorthand `pre-eval` (`… routes pre-eval`, `all eight
pre-eval causes route …`).

The two forms denote the same concept, so the inconsistency does not change
what the leaves require. It is a wording drift between the leaf's own header /
`Adds` and its acceptance criteria, and a drift away from the canonical spec
spelling that the leaf otherwise adopts. Aligning the body text on
`pre-evaluation` makes the leaf match both its own title and the spec term it
implements.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** Both `V4e` and `V4e-T` were added in `c6a664e`, and the
mismatch existed in that first revision: the `Adds` bullet read
`pre-evaluation` while every `Tests` bullet and the `Ships when` bullet read
`pre-eval`. No later edit introduced or narrowed the drift — `git log -S
"pre-eval"` on the leaf file returns only `c6a664e`. The inconsistency is an
authoring artefact of the initial plan build, not a regression from a
subsequent leaf edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V4e-pre-evaluation-failures.md` and
`docs/plan_topics/V4e-T-pre-evaluation-failures.md`, replace the shorthand
`pre-eval` with `pre-evaluation` everywhere it appears in the **Tests**
bullets and the **Ships when** bullet, matching the canonical spec term and
each leaf's own title and **Adds** bullet. Concretely, the occurrences to
change in each file are:

- `ERR-1`: `host-incompatible pre-eval failure routes …` → `host-incompatible pre-evaluation failure routes …`
- `ERR-2`: `… failure routes pre-eval.` → `… failure routes pre-evaluation.`
- `ERR-3`: `frontmatter failure routes pre-eval.` → `frontmatter failure routes pre-evaluation.`
- `ERR-4`: `binder-model resolution failure routes pre-eval.` → `… routes pre-evaluation.`
- `ERR-5`: `binder arg-binding failure (ceiling #3) routes pre-eval.` → `… routes pre-evaluation.`
- `ERR-6`: `tools:` resolution failure routes pre-eval.` → `… routes pre-evaluation.`
- `ERR-7`: `… routes pre-eval to loom-system-note …` → `… routes pre-evaluation to loom-system-note …`
- `ERR-16`: `… no-retry, routes pre-eval.` → `… no-retry, routes pre-evaluation.`
- **Ships when** (`V4e`): `all eight pre-eval causes route …` → `all eight pre-evaluation causes route …`

Scope is the body text of these two paired leaves only; other leaves that use
`pre-eval` (e.g. `V14c` / `V14c-T`) are outside this finding.

## Consequence

**Severity:** cosmetic

A reader sees one concept spelled two ways within a single leaf and against
the spec's canonical term. No implementer would diverge on meaning, and no
acceptance criterion changes; the cost is pure readability and term hygiene.
