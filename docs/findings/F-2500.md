## Finding

PIC-29 makes the tuple-to-arm mapping normative across all three arm-(b) tuples, but `(false, "shutting-down")` — reached when sub-step-1 `LoomRegistry.drain` throws and `LoomRegistry.initDrainStateTag` then succeeds on a `quit`/`reload` teardown (so `markRuntimeDegraded` never runs) — has no witnessed routing assertion. Item (j)'s quit/reload variant asserts only the post-handler tag value and explicitly forbids a slash-arm assertion, and no `readDrainState`-return-mapping fixture covers the tuple.

## Issue introduction

**Verdict:** later-commit-introduced (one introducing commit per vector)

**Introducing commits:**
- Drain-state arm-(b) vector: `8008b91` ("pi-loom spec-plan: fix F-1217 — add acceptance-criteria/fixture blocks for four Shard-08 MUSTs", 2026-06-25).
- Forced-respond no-transcript vector: `7f75947` ("pi-loom spec: resolve 'Typed-query forced-respond tool choice delivery channel'", 2026-06-05).

**History:** `git log -S` over `drain-state-contract.md` shows the *Acceptance criteria — `(drained: true, tag: undefined)` → arm (b)* section first appeared in `8008b91`; its item (b) enumerates all three arm-(b) tuples (`(false, "shutting-down")`, `(true, "shutting-down")`, `(true, undefined)`) but the commit authored a witnessed routing fixture only for `(true, undefined)`, leaving `(false, "shutting-down")` uncovered from the section's inception (the PIC-29 six-tuple decomposition itself predates it, having entered `drain-state-contract.md` at `f5e89f4`, 2026-06-04, when the contract was split into its own file). `git log -S 'attaches no turn to the driven session'` over `conversation-drive.md` and `git log -S 'not rendered as a transcript turn'` over `implementation-notes.md` both resolve to `7f75947`, which introduced the off-session delivery channel and its no-transcript property; no later commit added an acceptance criterion for it, so that vector is present-since-introduction.

## Solution Space

**Shape:** single

### Recommendation

Add a fixture obligation that asserts `LoomRegistry.readDrainState` returning `(false, "shutting-down")` routes through arm (b) (the `"loom /<name>: extension shutting down"` system note), witnessed *at the `readDrainState` return mapping* rather than via a live slash dispatch — using the same technique the `(drained: true, tag: undefined)` section already states for the `quit`/`reload` tear-down-before-dispatch case. Author it either as a new bullet in drain-state-contract.md's arm-(b) acceptance section or as a `readDrainState`-return-mapping arm assertion extending item (j)'s quit/reload variant in session-only-degraded-state.md; whichever site is chosen, cross-reference PIC-29's tuple-to-arm map so the three arm-(b) tuples are each independently witnessed.

## Consequence

**Severity:** advisory

The normative rules (PIC-29's tuple-to-arm map; the no-transcript MUST) are unambiguous, so implementers do not diverge on intent; the gap is purely in the conformance surface. But a runtime carrying a fall-through bug — mis-routing `(false, "shutting-down")` into a half-torn-down dispatch, or appending the forced respond turn to the driven transcript — passes the suite with the current fixtures, so the defect ships latent.
