## Finding

The frontmatter field-contract table in `frontmatter-fields-a.md` is normative: for each recognised field it pins down what observable behaviour the present-but-bad case produces, and every other "present-but-bad" row states its registration outcome explicitly. `mode`'s unknown-value arm says "the loom is not registered"; `bind_context` says "the loom is not registered"; `model` says "the loom is not registered"; `respond_repair.methodology` (in the load registry) says "The loom is not registered." The `system` row is the lone exception: it gives only the diagnostic code — "presence on a `mode: prompt` loom is `loom/parse/system-on-prompt-mode`" — and never states whether the offending loom registers, loads-and-ignores the field, or fails to load.

The `system:` prose in `frontmatter-fields-b-and-templates.md` repeats the same gap: it explains *why* the field is subagent-mode-only but likewise stops at naming the code without stating the disposition. An implementer working from the field-contract table alone cannot tell whether a `mode: prompt` loom carrying `system:` is dropped or whether it loads with the `system:` field silently ignored — and `system:` being an optional field invites the loads-and-ignores reading.

The diagnostics registry (`code-registry-parse.md`) classifies `loom/parse/system-on-prompt-mode` as severity `E`, phase `parse`, which does imply the file fails its parse/static checks and therefore does not register. The defect is that the field-contract row — the surface an implementer reads first — does not carry that outcome, breaking the parallelism its sibling rows establish.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** fb54586 — pi-loom spec: resolve "Frontmatter field required/optional/default contract is unspecified" (2026-05-04, Thomas Andersen)
**History:** The field-contract table was authored in fb54586, and the `system` row carried only `loom/parse/system-on-prompt-mode` with no registration outcome from its very first commit. When the diagnostic code was subsequently wired into the registry and the `system:` prose (839db71, 2026-05-04), the outcome was likewise omitted at both sites. No later commit ever added the disposition, so the gap has been present since the row was introduced.

## Solution Space

**Shape:** single

### Recommendation

Add the registration outcome to the `system` row so it matches its sibling "present-but-bad" rows. State that a `system:` field on a `mode: prompt` loom fires `loom/parse/system-on-prompt-mode` and **the loom is not registered** (the file fails its parse/static checks), consistent with the `E`/`parse` classification in `code-registry-parse.md`. Mirror the same disposition in the `system:` prose bullet in `frontmatter-fields-b-and-templates.md` so the field-contract table and the prose agree. Reuse the exact "the loom is not registered" wording the `mode`, `bind_context`, and `model` rows already use, so the parallelism is verbatim.

## Consequence

**Severity:** advisory

An implementer consulting only the field-contract table cannot determine whether a `mode: prompt` loom carrying `system:` is dropped or loads with the field ignored; the optional-field framing invites the loads-and-ignores reading. The diagnostics registry's `E`/`parse` classification resolves it for an implementer who chases the code through to the registry, so the gap is a guidance/parallelism defect rather than a hard contradiction.
