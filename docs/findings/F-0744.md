## Finding

The settings-file shape-failure surface carries two coordinated gaps.

**Non-object JSON root has no defined behaviour.** `package-and-settings.md` ¬ß"Failure modes" enumerates exactly two failure modes: missing/unreadable (`loom/load/settings-unreadable`) and "present but not valid UTF-8 JSON" (`loom/load/settings-invalid-json`). The companion ¬ß"Top-level shape validation" handles per-key malformations of `loomPaths` / `looms` but presupposes the root JSON value is an object. A settings file whose body parses as a JSON value other than an object ‚Äî `[1, 2, 3]`, `42`, `true`, `"hello"`, or bare `null` ‚Äî satisfies neither rule: it is valid UTF-8 JSON (so `settings-invalid-json` does not fire) and there is no top-level container-kind check before per-key inspection. The spec does not say whether such a file is treated as `{}`, crashes the loader reading `.loomPaths` off a non-indexable root, or which diagnostic surfaces.

**Non-scalar `<observed>` rendering is undefined.** `loom/load/settings-value-out-of-range`'s *Trigger* now also fires on top-level shape failures (a `loomPaths` that is not a JSON array, a `looms` that is not a JSON object). The `<observed>` token (`settings key <key> value is out of range; got <observed>`) is governed by the ¬ß8 parsed-scalar carve-out, which enumerates only `number` / `boolean` / `null` / `string`. It does not cover `array` or `object`, even though the trigger now routes such values through `<observed>`. A producer faced with `loomPaths: { "a": 1 }` or `looms: [1, 2]` has no normative answer for the byte string `<observed>` substitutes, and DIAG-4's byte-identicality guarantee fails. The ¬ß7 `<kind>` closure for `settings-invalid-entry` / `manifest-invalid` already enumerates `array` / `object`; the ¬ß8 carve-out finished its table only for scalars.

## Solution Space

**Shape:** single

### Recommendation

Resolve both gaps in one settings-shape pass.

1. **Non-object root ‚Äî add a dedicated `loom/load/settings-not-object` code.** Keep `settings-invalid-json` scoped to UTF-8 / JSON parse failures and introduce a third settings-load diagnostic for the valid-JSON / non-object-root case: add a ¬ß"Failure modes" bullet "File present and valid UTF-8 JSON but its parsed root is not a JSON object (array, number, boolean, string, or `null`) ‚Äî `loom/load/settings-not-object`"; prepend to ¬ß"Top-level shape validation" "If the parsed root is not a JSON object, the file is treated as absent in its entirety per the Failure modes rule above and no per-key `settings-value-out-of-range` cascades for that file"; register `loom/load/settings-not-object` in `code-registry-load.md` (severity `warning`, phase `load`) and confirm in `placeholder-rendering-b.md` it carries only the `<path>` placeholder (category 5). The two failure modes are semantically distinct and the registry already pays the per-code cost for the same distinction in the frontmatter family. Edge cases: bare `null` MUST trip the new code (not the missing-file arm); the cascade MUST be fully suppressed so a `[1,2,3]` file logs exactly one diagnostic; per-file isolation still holds ‚Äî a non-object root in one file MUST NOT suppress a well-formed value of the same key in the other.

2. **Non-scalar `<observed>` ‚Äî render the JSON kind.** Extend the ¬ß8 parsed-scalar carve-out so `array` and `object` render the JSON-kind token (`array`, `object`), matching the ¬ß7 `<kind>` closure; scalars continue rendering their value verbatim. Add to the carve-out: "an `array` renders the literal token `array`; an `object` renders the literal token `object`. This rendering applies only on the top-level `loomPaths`/`looms` shape-failure arm of `loom/load/settings-value-out-of-range`; the frontmatter row never observes `array`/`object` inputs." Add test vectors: `loomPaths: { "a": 1 }` renders `... got object`; `looms: [1, 2]` renders `... got array`. No *Message* template change. The kind-token choice closes the gap with full byte-identicality and no new decisions (the *Trigger* row already names the offending key via `<key>`); rendering modes split per *Trigger* arm, not per JSON kind globally, so the disjunction is well-defined.

## Consequence

**Severity:** correctness

A user who ships `.pi/settings.json` as a JSON array ‚Äî or overwrites it with `null` ‚Äî triggers implementation-defined behaviour: one loader treats it as `{}`, another raises `TypeError`, a third silently returns no settings and skips the diagnostic, breaking the per-file isolation and non-fatal-degradation contract the surrounding block promises. Separately, two conformant implementations can render the same `loomPaths: { "a": 1 }` input as visibly different `<observed>` substrings, defeating DIAG-4's byte-identicality contract; a conformance test for the non-scalar arm cannot be written from the current spec.
