## Finding

`pi-integration.md`'s Slash-command discovery bullet asserts as fact that "The `description` field from frontmatter is what populates the autocomplete entry that Pi's slash-command UI shows" and that each loom appears in autocomplete as `/<filename>` "exactly mirroring Pi's prompt-template behaviour". Both halves are behavioural claims about Pi's current slash-command UI — what Pi renders, where, and from which field — yet neither is tied to a named SDK capability, a host-behaviour presupposition, or a bump-time re-audit step.

The asymmetry with the sibling `argument-hint` sentence (in the same bullet) is what makes the gap conspicuous: `argument-hint` is explicitly scoped to binder grounding, its extension-API gap is documented at the owning [`frontmatter-fields-a.md` §`argument-hint` autocomplete contract](../spec_topics/frontmatter/frontmatter-fields-a.md#argument-hint-binder-only) (pinning that Pi's `RegisteredCommand` has no `argumentHint` slot), and the upstream-Pi follow-up is tracked under [`future-considerations/surface-extensions.md`](../spec_topics/future-considerations/surface-extensions.md). The `description` claim has no equivalent backing. What PIC actually pins for `description` is the registration *call shape* — [`capability-inventory-items.md` item 1](../spec_topics/pi-integration-contract/capability-inventory-items.md#sdk-cap-slash-command-registration) (`pi.registerCommand(name, { description, handler })`) — i.e. that the field is *accepted as an argument*, not that Pi's autocomplete dropdown renders it (and certainly not that it renders it as the "autocomplete entry").

The same gap covers the "mirrors Pi's prompt-template behaviour" claim: prompt templates are `*.md` files Pi enumerates internally via `source: "prompt"`, while looms reach Pi via `pi.registerCommand`; whether the autocomplete UI treats the two arms identically is a UI behavioural claim Pi's typed surface does not encode. The pattern matches several other unpinned UI/lifecycle behaviours already inventoried as host-behaviour presuppositions in [`host-prerequisites.md`](../spec_topics/pi-integration-contract/host-prerequisites.md) (settings write-back key preservation, `pi.registerTool` post-startup acceptance, `pi.getCommands()` completeness, etc.), each of which carries a named bump-checklist item. The `description`-rendering claim has none.

## Issue introduction

**Verdict:** present-since-introduction
**Introducing commits:** `53f5831` ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03) introduced the unsourced `description` → autocomplete claim in its current form when it first added the slash-command-discovery bullet (`"The description and argument-hint from frontmatter populate the autocomplete entry."`). `1e103d5` ("spec: rescope argument-hint to binder grounding only (Decision #7)", 2026-05-05) then asymmetrically split the bullet — giving `argument-hint` a fully-cited SDK-gap framing (forwarded to Future Considerations and the `frontmatter-fields-a.md` autocomplete-contract anchor) while *retaining* the `description` half as a bare behavioural assertion, which is the precise asymmetry this finding flags.
**History:** `git log -L '/Slash-command discovery/,+1:docs/spec_topics/pi-integration.md'` and `git log -G 'populate the autocomplete entry'` show a four-commit chain (`fecb504` → `d6cbb37` → `1e103d5` → `4aef0e2` → `4da6c6c`): the bullet originated in `53f5831` against the top-level `spec.md`, was carried into the split `spec_topics/pi-integration.md` by `fecb504`, and the description-vs-argument-hint split was performed in `1e103d5` (the only commit that touched the rendering language); subsequent commits only adjusted the discovery-sources clause and the `argument-hint` cross-reference. No commit ever added a host-behaviour anchor for `description`. The defect is therefore present-since-introduction in `53f5831` and made asymmetric (and more conspicuous) by `1e103d5`'s targeted rescope of the sibling clause.

## Solution Space

**Shape:** multiple

### Option A — Promote to a named host-behaviour presupposition

**Approach.** Add a new entry to the host-behaviour presupposition list in `host-prerequisites.md` (e.g. *`pi.registerCommand` `description` is rendered in the slash-command autocomplete entry*), following the established `presupposition` template used by `pi.registerTool` post-startup acceptance and `pi.getCommands()` completeness. Add a corresponding editorial-review checklist item under `version-bump-step2.md` so each Pi minor bump re-audits the claim. In `pi-integration.md`, replace the bare assertion with a forwarding sentence to the new presupposition (the same pattern the `argument-hint` clause uses to forward to `frontmatter-fields-a.md`).

**Spec edits.**
- `pi-integration.md`: rewrite the `description`/autocomplete-rendering sentence and the "mirroring Pi's prompt-template behaviour" claim as a single forwarding reference to the new presupposition.
- `host-prerequisites.md`: insert the new presupposition paragraph with anchor, *Authoritative source* note (observed at `dist/core/extensions/` autocomplete-render path at the pin, known-fragile-evidence boilerplate), and bump-checklist forward-link.
- `version-bump-step2.md`: add a new editorial-review checklist item (next free letter, e.g. `(aq)`) keyed to the new presupposition.

**Pros.** Treats the claim with the same discipline already applied to comparable unpinned Pi behaviours; gives the implementer a single anchor to cite; gives the bump contributor a concrete review obligation that catches a Pi UI redesign at bump time rather than at user-report time; restores symmetry with the `argument-hint` clause.

**Cons.** Adds one more entry to an already-long presupposition list and one more checklist item to the bump procedure. The bump-time editorial review for a UI-rendering claim is necessarily a manual visual check (no SDK surface-inventory signal can detect it), which is the same posture the other UI-adjacent presuppositions already accept.

**Risks.** Mis-classification: presuppositions are typically about behaviours loom-side code consumes; a UI-rendering claim is consumed by the operator, not by loom code. Mitigate by being explicit in the presupposition text that the consumer is the operator-facing autocomplete UI (the same framing the `operator-always-present` presupposition uses).

### Option B — Demote to a non-normative observation

**Approach.** Rewrite the `description`-rendering and "mirrors prompt-template behaviour" claims as informative orientation: "*Non-normative.* At the [loom 1.0 Pi-SDK pin](../pi-integration-contract/host-prerequisites.md#pi-sdk-pin), Pi's autocomplete UI is observed to render the `description` argument passed to `pi.registerCommand`; this is informative orientation only — the normative obligation loom asserts is the registration-call shape pinned at [SDK capability inventory item 1](../pi-integration-contract/capability-inventory-items.md#sdk-cap-slash-command-registration)." Add no new presupposition, no new bump-checklist item.

**Spec edits.**
- `pi-integration.md`: rewrite the sentence as a non-normative observation, with explicit forwarding to capability item 1 for the normative half.
- No edits to PIC or version-bump pages.

**Pros.** Minimal surface change; truthful to the fact that loom code never observes whether Pi rendered the description; avoids growing the presupposition list for a claim no loom code path depends on.

**Cons.** Loses the bump-time editorial-review safety net; a future silent Pi UI change still degrades discoverability with no named anchor. Diverges from the precedent set by the sibling `argument-hint` clause, which kept the gap visible by forwarding to a Future Considerations entry rather than demoting.

**Risks.** A `pi-integration.md` reader looking for the autocomplete contract may take "non-normative" as license to ignore the field entirely, since the page is the natural entry point for the integration story.

### Recommendation

Option A. The autocomplete-rendering behaviour is on the same footing as several other unpinned UI/lifecycle behaviours already promoted to host-behaviour presuppositions in `host-prerequisites.md`, and the discoverability impact (silent degradation with no named anchor) is exactly the failure mode that pattern exists to catch. The marginal cost — one presupposition paragraph plus one bump-checklist item — is the same loom 1.0 already pays for `pi.registerTool` post-startup acceptance, settings write-back key preservation, and `pi.getCommands()` completeness. Edge cases for the implementer to watch: (1) frame the new presupposition's consumer as the operator-facing autocomplete UI (matching the `operator-always-present` precedent), not loom-side code; (2) make the presupposition cover both halves of the original claim — that Pi *renders* the `description` argument and that extension-registered commands *appear in the same autocomplete surface* as prompt-template commands — since both are independent behavioural assertions; (3) the new bump-checklist item must be filed under the next free letter in the existing `(f)–(ap)` sequence and named consistently with the surrounding items.

## Consequence

**Severity:** advisory

If Pi's slash-command autocomplete UI ever stops rendering the `description` argument it accepts (a Pi-side UI redesign, a switch to a separate "summary" field on `RegisteredCommand`, or a divergence in how extension-registered commands render versus prompt templates), loom discoverability degrades silently with no named presupposition to fault, no bump-time editorial-review step to catch it, and nothing in the SDK surface-inventory assertion that would tell the contributor the assumption no longer holds. The runtime keeps registering commands successfully; users just see empty-looking dropdown entries — and the spec offers the implementer no anchor to point to as the broken contract.
