## Problem

`functions.md` carries five anchored numbered REQ-IDs (`FN-1` … `FN-5`) and `return.md` carries three (`RET-1` … `RET-3`) — eight live `PREFIX-N` obligations under the registered live `FN`/`RET` prefixes. `coverage-matrix.md` nevertheless lists both pages only in row `cka-7` of its *Code-keyed obligation areas (**no numbered REQ-IDs**)* table, whose heading asserts those pages own their obligations through `loom/...` diagnostic codes rather than numbered REQ-IDs — directly false for these eight. The eight live IDs have no row in the matrix's *Numbered REQ-IDs* table, so the `H5a` closing gate reddens on the unmapped-executable-REQ-ID arm even though `V3d` genuinely closes them. This is the same reclassification commit `47915a2` performed for the analogous PIC code-keyed rows, with `cka-7` overlooked.

## Issue introduction

**Verdict:** introduced (multi-commit-interaction)
**Introducing commits:** `b97fab3` (`pi-loom spec-plan: fix F-1272 — coin FN-1..5 on functions.md`, 2026-06-25), `c0c6853` (`pi-loom spec-plan: fix F-1271 — coin RET-1..3 on return.md`, 2026-06-25); perpetuated by `47915a2` (`pi-loom spec-plan: fix F-1890 — map ~26 omitted live PIC REQ-IDs in coverage-matrix; retire wholly-numbered cka rows; add inline citations`, 2026-06-27)
**History:** `functions.md`/`return.md` were classified as code-keyed obligation areas in `coverage-matrix.md` since the plan's inception (`c6a664e`, 2026-06-10), which was correct then — neither page carried numbered REQ-IDs. The defect crystallised when `FN-1`…`FN-5` (`b97fab3`) and `RET-1`…`RET-3` (`c0c6853`) were coined on the spec pages on 2026-06-25; both commits touched only the spec pages (`functions.md` / `return.md`) and did **not** update `coverage-matrix.md`, leaving the `cka-7` "no numbered REQ-IDs" classification stale. The matrix `cka-7` cell has been unchanged since `7259777` (2026-06-18, the `cka-<n>` token-scheme commit). The companion reclassification (retire the wholly-numbered `cka` row, add *Numbered REQ-IDs* rows, add inline citations) was later performed for the PIC `cka` rows by `47915a2` (F-1890) but that pass did not touch `cka-7`, so the FN/RET omission survived.

## Solution approach

In `coverage-matrix.md`, add rows to the *Numbered REQ-IDs (runtime obligations)* table mapping `FN-1` … `FN-5` and `RET-1` … `RET-3` to their closing leaf `V3d`, and retire `cka-7` in place following the established retirement form already used by the numbered-drained `cka` rows (`cka-12`, `cka-18`, `cka-22`, etc.). In `V3d-T-functions-and-return.md` (and its mirror `V3d-functions-and-return.md`), add inline `FN-N` / `RET-N` citations to the Tests bullets so each of the eight mapped IDs carries a citing test, replacing the current "(… code-keyed area)" annotations. Update the now-stale "(RET code-keyed area)" annotations on the `return.md` re-citations under `V9i*` and `V15a*`.

## Solution constraints

- The spec is read-only for this fix: do not re-coin or edit `functions.md`, `return.md`, or the REQ-ID prefix table.
- The matrix *Token* column is append-only and tokens are never reused: retire `cka-7` in place rather than deleting the row or reassigning its token.
