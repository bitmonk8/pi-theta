## Finding

`V18c` consumes two constants from `V9a`'s single source-of-truth pinned-constants block: `FACTORY_PROBABLE_CAPABILITIES` (the build-time partition-reconciliation symbol) and `SHUTDOWN_AWAIT_CAP_MS` (read by V18c's build-time literal-read assertion). Both are named explicitly in V18c's `Adds.` and exercised by its Tests. V18c's `Deps.` is `V18c-T, V18a, V18b` — `V9a` is not listed; it is reached only transitively through `V18a` (whose `Deps.` is `V18a-T, V9a`).

This is a declaration-completeness gap, not a sequencing defect: the dependency DAG is respected (V18a forces V9a complete before V18c), so the leaf remains pickable and executes in a valid order. The gap is that a direct cross-leaf seam consumption is not backed by a direct `Deps.` edge to the seam's owner.

The corpus convention for this seam is to declare V9a directly: `V18a` and `V18a-T` both list `V9a` in `Deps.` precisely because they consume `FACTORY_PROBABLE_CAPABILITIES`, and `V17a`/`V9i` likewise list `V9a` for the constants they read. V18c (and its tests partner V18c-T) consume the same V9a-owned constants but diverge from that pattern.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** b9de3ee — pi-loom plan: resolve "V18a partition assertion imports undeclared V9a probe constant" (2026-06-11, plan author); 622d0d3 — pi-loom plan: resolve "SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block placement" (2026-06-11, plan author)
**History:** V18c was created (c6a664e, 2026-06-10) with `Deps.` `V18c-T, V18a, V18b` and no reference to V9a's constants. Commit b9de3ee later wired V18c's gates to consume V9a's exported `FACTORY_PROBABLE_CAPABILITIES`, and 622d0d3 added the `SHUTDOWN_AWAIT_CAP_MS` literal-read consumption. Neither commit added a `V9a` edge to V18c's (or V18c-T's) `Deps.`, so the direct-consumption-without-direct-edge gap arose from those two consumption additions landing against the unchanged Deps line.

## Solution Space

**Shape:** single

### Recommendation

Add `V9a` to V18c's `Deps.` (`V18c-T, V18a, V18b` → `V18c-T, V18a, V18b, V9a`) and to V18c-T's `Deps.` (`V18a, V18b` → `V18a, V18b, V9a`), making the direct consumption of V9a's exported `FACTORY_PROBABLE_CAPABILITIES` and `SHUTDOWN_AWAIT_CAP_MS` pinned constants explicit. This mirrors the existing direct `V9a` edges in `V18a` and `V18a-T`, which consume the same V9a-owned symbol. The V9a edge is purely additive to the existing transitive guarantee, so it changes no leaf's pickability or ordering.

## Consequence

**Severity:** advisory

The leaf can still be picked up and built in a valid order — V9a is transitively guaranteed via V18a — so nothing fails today. The residual risk is robustness and consistency: if V18a's `V9a` edge were ever refactored away, V18c would silently lose the guarantee that the constants it reads exist, and the missing direct edge is inconsistent with how V18a/V18a-T/V17a/V9i declare V9a for the same constants.
