## Finding

All three of `V5e`'s Tests bullets (and the mirrored bullets in `V5e-T`) exercise the loom-owned JSON-depth walk "in isolation against the harness". In this plan's terminology "the harness" is a reserved term: it names the end-to-end test harness that `H4a` (`H4a-factory-shell-and-harness.md`) establishes — an in-process Pi session double — whose scripted-injection response-programming surface is owned by `H4b`. Every other leaf that genuinely drives that harness (e.g. `V4c`, `V4f`) declares `H4a` and `H4b` in its `Deps`. `V5e`'s `Deps` are only `V5e-T`, `V5d`, `V16a`, `V4d`; `V5e-T`'s are `V5d`, `V16a`, `V4d`. Neither names `H4a`/`H4b`.

The phrasing is internally contradictory with the bullets' own intent. The first bullet states the test is run "mirroring `V16a`'s seam pattern, where the decision is exercised directly and the live AJV-boundary sites are built downstream". `V16a` exercises its arbitration through synthesised candidates "in isolation" and depends on neither `H4a` nor `H4b`; its depth-walk analogue here is a pure recursive function over a materialised value that needs no in-process session double, no slash dispatch, and no scripted assistant turns. So the depth walk does not, in fact, require the `H4a`/`H4b` harness — yet it borrows the word "harness", which in this plan implies exactly that dependency.

The result is an ambiguity that two reasonable implementers resolve in opposite directions: one reads "the harness" literally, treats `H4a`/`H4b` as undeclared dependencies, and builds `V5e` only after the whole extension shell and session double exist; the other follows the "mirroring `V16a`" clause and builds the depth walk standalone. The git history confirms the collision was not original to the leaf (see Issue introduction).

**Verdict:** introduced-by-later-edit
**Introducing commit:** `3fa39a9` — "pi-loom plan: resolve \"V5e per-boundary routing test asserts destination error surfaces its Deps cannot reach\"" (2026-06-11)
**History:** The plan corpus is git-tracked. The original build of this leaf (`c6a664e`, "build/update plan for spec.md + review") had no "harness" phrasing: its Tests bullet read "A materialised value of depth 6 fires `schema_keyword:\"maxDepth\"` … at each of the five sites", i.e. it asserted at the live enforcement sites, and `Deps` were `V5e-T`, `V5d`, `V16a`. Commit `3fa39a9` refactored the Tests from live-site assertions to isolated assertions (resolving an unrelated "Deps cannot reach the destination surfaces" finding) and in doing so introduced the "in isolation against the harness" / "mirroring `V16a`'s seam pattern" phrasing — the point at which the reserved "harness" term collided with `Deps` that omit `H4a`/`H4b`. Commit `9d880b8` ("resolve \"V5e before-AJV Tests bullet asserts an ordering its isolated harness cannot witness\"", 2026-06-17) propagated the same "against the harness" phrasing to the remaining (depth-computation) bullet. The defect is single-commit in origin (`3fa39a9`) with later propagation (`9d880b8`); it was not present since inception.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V5e-depth-enforcement.md` and `docs/plan_topics/V5e-T-depth-enforcement.md`, strike the reserved term "the harness" from the depth-walk Tests bullets so the phrasing matches `V16a`'s `Deps`-free isolated-seam wording. Concretely, in each file replace the two occurrences of "in isolation against the harness" and the one occurrence of "asserted in isolation against the harness" with "in isolation" (the phrasing `V16a` already uses, and the phrasing `V5e`'s own Ships-when line already uses: "drives a depth-6 value through the loom-owned walk in isolation"). Leave `V5e`/`V5e-T` `Deps` (`V5e-T`, `V5d`, `V16a`, `V4d` / `V5d`, `V16a`, `V4d`) unchanged — the depth walk is a pure function over a materialised value and, per the leaf's own "mirroring `V16a`'s seam pattern" clause, does not consume the `H4a` session double or the `H4b` scripting surface.

Edge case for the implementer: this fix is correct only if the depth-walk test truly needs no in-process session double. If, when authoring the test, the implementer finds it must drive the walk through the `H4a`/`H4b` session double (e.g. to reach a live boundary site), then the opposite resolution applies — add `H4a` and `H4b` to both `Deps` and keep the "against the harness" wording — and `V5e`/`V5e-T` become blocked on `H4a`/`H4b`. The `V16a` precedent and the pure-function nature of the walk indicate the standalone (reword) path.

## Consequence

**Severity:** correctness

The two readings produce materially different plans: one adds `H4a`/`H4b` to `V5e`/`V5e-T` `Deps` and gates the pure depth walk behind the entire extension shell and session double; the other builds it standalone. A `Deps`-driven picker following the literal "harness" reading would also serialise `V5e` behind `H4a`, an ordering constraint the leaf's `V16a`-mirroring intent does not actually require.
