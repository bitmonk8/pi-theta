## Problem

The *surface-set closure* property — defined in `inventory-audit-intro.md` and the `glossary.md` entry, cross-referenced from `overview-and-orientation.md#orientation` — guarantees that every Pi-side surface reference in the *audited source tree* resolves to an `SDK_SURFACE_INVENTORY` entry, a typebox sibling allow-list, or a declared `// allow-pi-surface:` exemption. But the *audited source tree* is defined in `audit-resolution.md#audit-scope` as the closed set matching `src/**/*.ts` minus enumerated exclusions, while the spec leans on the stronger claim that the audit closes over all Pi-touching runtime source. That stronger claim rests on an unstated, unverified premise — that no Pi-touching hand-authored runtime source exists outside `src/**/*.ts` (e.g. a `.mts`/`.cts`/`.tsx` carrier, or any source rooted outside `src/`) — which is never asserted as a precondition and which nothing detects. A runtime module in such a carrier would carry live `pi.<member>` / peer-package / `ctx.<member>` references the audit never walks, silently escaping closure with no audit signal.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09, "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"")
**History:** The negative-direction audit machinery — the `src/**/*.ts` *Audit scope* definition, the *audited source tree* concept, the *inventory-closure audit*, and the *surface-set closure* property — was introduced as a single unit in `e5cd514` (then living in `spec.md`, before the corpus was split into `spec_topics/` at `81bf52a`). `git log -S` confirms the closed `src/**/*.ts` glob and the closure property first appear together in that commit, and that the spec equated "audited source tree" with "the runtime's hand-authored TypeScript runtime source" from the outset. No later commit (the `589d9a4` surface-set-closure glossary entry, the `f5e89f4` size-cap edit, the `81bf52a` topic split) ever added a precondition asserting that the entire Pi-touching footprint resides within that glob; the gap has existed unchanged since the audit feature was first specified.

## Solution approach

Add a precondition to the `#audit-scope` paragraph in `audit-resolution.md` asserting that at loom 1.0 the entire Pi-touching hand-authored runtime source footprint resides under `src/` with the bare `.ts` extension, covering the three escape vectors — non-bare-`.ts` TS extensions (`.mts`, `.cts`, `.tsx`), out-of-`src/` roots, and the generated-output-in-`src/` case already noted in that paragraph — and bind it to the existing future-change governance rule there so a later carrier cannot be adopted without extending the include glob in the same edit. Add forward-links from the surface-set-closure definitions in `inventory-audit-intro.md` and the `glossary.md` entry to that precondition so the bridge from *audited source tree* to all Pi-touching runtime source is explicit.

## Solution constraints

- Out of scope: widening the inventory-closure audit's five-family set to add a mechanical detector for the precondition — the closure rule in `#audit-scope` forbids a sixth family in this PIC edit; the precondition is a static adoption-time check only.
