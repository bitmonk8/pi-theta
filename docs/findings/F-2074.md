## Finding

The *surface-set closure* property — defined identically in `overview-and-orientation.md` (Orientation → Prerequisites) and `inventory-audit-intro.md` — is stated as: *every Pi-side surface reference in the **audited source tree** resolves to an `SDK_SURFACE_INVENTORY` entry, a typebox sibling allow-list, or a declared `// allow-pi-surface:` exemption.* The audited source tree is in turn defined in `audit-resolution.md` (*Audit scope*) as the closed file set matching the include glob `src/**/*.ts` minus enumerated exclusions, with files outside `src/` and files inside `src/` whose extension is not bare `.ts` (`.tsx`, `.cts`, `.mts`, `.d.cts`, `.d.mts`, `.js`, `.mjs`, `.cjs`, …) declared out of scope "by not matching the include glob and need no separate exclusion."

As scoped, the closure property is trivially complete over the audited source tree — but the value the spec actually leans on (e.g. the trust-boundary reasoning, and `spec.md`'s framing that "the runtime's hand-authored TypeScript runtime source under `src/`" *equals* the audited source tree) is the stronger claim that the audit closes over **all** Pi-touching runtime source. That stronger claim depends on an unstated, unverified premise: that no Pi-touching hand-authored runtime source exists outside `src/**/*.ts`. The premise is never asserted as a precondition and nothing detects its violation. A runtime module authored as `src/foo.mts` or `src/foo.cts` (both valid Node/TS module extensions), a `.tsx` runtime file, or any Pi-touching source rooted outside `src/` would carry live `pi.<member>` / peer-package / `ctx.<member>` references that the audit never walks — silently escaping closure with no audit signal.

The *Audit scope* paragraph does carry a forward-looking governance rule: a future change adopting an unenumerated convention (it names `*.tsx`, `*.gen.ts`, `**/fixtures/**`, etc.) "MUST extend either the include glob or the exclusion list … in the same edit." But that rule is contributor-discipline only — there is no mechanical detector — and it governs only *newly adopted* conventions; it neither asserts the loom 1.0 starting precondition (that the present Pi-touching footprint is wholly within `src/**/*.ts`) nor covers out-of-`src/`-root carriers. The closure guarantee is therefore sound only by assumption, and the assumption is invisible to a reader who trusts the property as stated.

## Issue introduction
**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09, "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\"")
**History:** The negative-direction audit machinery — the `src/**/*.ts` *Audit scope* definition, the *audited source tree* concept, the *inventory-closure audit*, and the *surface-set closure* property — was introduced as a single unit in `e5cd514` (then living in `spec.md`, before the corpus was split into `spec_topics/` at `81bf52a`). `git log -S` confirms the closed `src/**/*.ts` glob and the closure property first appear together in that commit, and that the spec equated "audited source tree" with "the runtime's hand-authored TypeScript runtime source" from the outset. No later commit (the `589d9a4` surface-set-closure glossary entry, the `f5e89f4` size-cap edit, the `81bf52a` topic split) ever added a precondition asserting that the entire Pi-touching footprint resides within that glob; the gap has existed unchanged since the audit feature was first specified.

## Solution Space

### Recommendation

In `audit-resolution.md` (*Audit scope*), add an explicit precondition, verified at adoption rather than merely asserted, that at loom 1.0 the entire Pi-touching hand-authored runtime source footprint resides under `src/` with the bare `.ts` extension — and that consequently no Pi-touching surface lives in a `.tsx`/`.cts`/`.mts`/other-extension file or outside the `src/` root. Add one new precondition sentence plus a one-line adoption-verification note to the *Audit scope* paragraph. Bind the precondition to the existing future-change governance rule so that any later adoption of such a carrier MUST extend the include glob in the same edit, and so a later carrier cannot be adopted silently.

The precondition text must bind all three escape vectors: non-bare-`.ts` TS extensions (`.mts`, `.cts`, `.tsx`), out-of-`src/` roots, and the generated-output-in-`src/` case already noted in the same paragraph.

Update the surface-set-closure framing in `spec.md` and `inventory-audit-intro.md` to cite this precondition as the bridge from "audited source tree" to "all Pi-touching runtime source," and add a cross-reference from the surface-set-closure definition in `overview-and-orientation.md` / `inventory-audit-intro.md` to the precondition.

This is a static, adoption-time check: the verification reduces to a one-time confirmation of the source-tree shape at adoption. A future `.mts`/`.cts`/`.tsx` runtime file remains covered only insofar as the contributor honours the bound future-change governance rule.

## Consequence

**Severity:** correctness

The spec asserts a closure guarantee that is stronger than what the audit actually enforces. An implementer can build the audit exactly as specified and still ship a system where Pi-touching surface in a `.mts`/`.cts`/`.tsx` file (or any out-of-`src/` carrier) is wholly unaudited, defeating the surface-set-closure property that the trust-boundary and version-bump reasoning rely on — without any test going red.
