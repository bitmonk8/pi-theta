## Problem

The `Loom-package implementation dependencies (loom 1.0)` bullet (anchors `#loom-package-implementation-dependencies-loom-1-0` / `#loom-package-implementation-dependencies-v1`) and the adjacent `Implementation hint (non-normative)` AJV bullet in `implementation-notes.md` record reference-implementation packaging decisions — npm dependency-block placement, the `@types/semver` devDependency, which libraries ship their own TypeScript bindings — flat in the spec body, interleaved among genuinely normative runtime bullets (Schema validation, Clock) with no structural boundary. The same bullet also carries a conformance-relevant statement — that `chokidar` and `yaml` fall outside the inventory-closure audit's *Target surface categories* and so need no `// allow-pi-surface:` exemption marker — buried among packaging trivia headed "implementation dependencies". This blurs which sentences bind a conforming implementation and which are incidental choices of the reference, and risks an audit implementer treating the binding out-of-scope rule as droppable trivia.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 1ea0455 — pi-loom spec: resolve "DI-seam vs reference-implementation placement boundary" (2026-06-04, Thomas Andersen); 1064946 — pi-loom plan: resolve "YAML frontmatter parsing mechanism is never declared as a dependency" (2026-06-10, Thomas Andersen)
**History:** The `Loom-package implementation dependencies (loom 1.0)` bullet was introduced by 1ea0455, the same commit that drew the DI-seam vs reference-implementation placement boundary; it landed the `semver`/`chokidar` packaging recipe (dependency-block placement, `@types/semver`, audit out-of-scope clause) flat in the spec body. Commit 1064946 later widened the same bullet with the `yaml` clause. The AJV `Implementation hint (non-normative)` bullet predates these and was only relocated, not created, by later moves. The packaging-recipe-in-spec-body shape was complete at 1ea0455; 1064946 extended the existing pattern rather than introducing it through interaction.

## Solution approach

Add a non-normative reference-implementation-packaging sub-section in `implementation-notes.md` and move the packaging-recipe prose under it — the `semver`/`chokidar`/`yaml` dependency-block placement and `@types/semver` devDependency detail from the `#loom-package-implementation-dependencies-loom-1-0` bullet, and the AJV packaging detail in the `Implementation hint (non-normative)` runtime bullet — fencing it from the normative Schema-validation and Clock runtime bullets. Carry both the `#loom-package-implementation-dependencies-loom-1-0` and `#loom-package-implementation-dependencies-v1` anchors onto the relocated content so the plan citations sourcing from them are not orphaned.

## Solution constraints

- The `chokidar`/`yaml` inventory-closure-audit out-of-scope statement (no `// allow-pi-surface:` marker required, per `audit-target-categories.md`) MUST remain normative and MUST NOT inherit the non-normative label of the relocated packaging prose.
