## Finding

The **Canonical schema hash** recipe in `schema-subset.md` declares itself a byte-exact, breaking-change contract: "The recipe is part of the on-disk and on-wire contract — changing it is a breaking change for any cached artefact, fixture snapshot, or replayable provider payload — so the loom 1.0 spec pins it exactly." The slug is the first 16 hex of SHA-256 over the canonical-form UTF-8 bytes, so the byte sequence must be reproducible across conforming implementations down to the last byte.

The canonical-form *string* serialisation clause (step 2) does not meet that bar. It says only: "strings escaped per RFC 8259 minimal-escape rules (only the characters JSON requires escaping; no gratuitous `\u` escapes for printable ASCII)." This pins the printable-ASCII case but leaves two degrees of freedom that RFC 8259 explicitly permits:

- **Control-character form.** For C0 control characters (U+0000–U+001F), RFC 8259 requires escaping but allows *either* the two-character short escapes (`\b`, `\f`, `\n`, `\r`, `\t`) *or* the six-character `\uXXXX` form. "Minimal-escape" does not disambiguate: both are valid minimal escapes of the same character. So a `const`/`enum` string containing a newline can serialise as `\n` in one implementation and `\u000a` in another.
- **Hex-digit case.** The `\uXXXX` form admits both uppercase and lowercase hex digits (`\u001F` vs `\u001f`); the clause fixes hex-digit case for the slug output (step 4, "lowercased") but not for `\uXXXX` escapes inside the canonical form.

(The forward-slash question — whether `/` is escaped as `\/` — is already resolved by "only the characters JSON requires escaping," since RFC 8259 does not require escaping `/`.)

Because the numeric-serialisation sibling clause was later pinned by delegating to the binder's `integer`/`number` rendering algorithm, the string clause is now the sole remaining unpinned facet of an otherwise byte-exact contract. Any `const` or `enum` string value carrying a control character will produce divergent canonical-form bytes — and therefore a divergent slug — across two conforming implementations.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `e21449b` — "pi-loom spec: resolve \"__inline_<hash> hash function is not pinned\"" (2026-05-04)
**History:** `e21449b` authored the entire **Canonical schema hash** recipe, including the step-2 clause verbatim: "strings escaped per RFC 8259 minimal-escape rules (only the characters JSON requires escaping; no gratuitous `\u` escapes for printable ASCII)." The string-escaping under-pinning has existed since that introduction. A later commit `c5bb5e1` ("Canonical schema hash numeric serialization underspecified") tightened the adjacent *numeric* clause in the same byte-sequence block — delegating it to the binder `integer`/`number` rendering algorithm — but left the string-escaping line untouched (`git show c5bb5e1` carries it as unchanged context). `git log -S 'minimal-escape' --all` confirms no commit has since narrowed the escaping rule. The defect is thus original to the recipe, not introduced by a later edit; the numeric-clause fix simply isolated it as the last unpinned facet.

## Solution Space

**Shape:** single

### Recommendation

In step 2 of the Canonical schema hash recipe, replace the "RFC 8259 minimal-escape" string clause with one that pins a single byte sequence per source string. Name `JSON.stringify`'s string-escaping behaviour as the normative authority (consistent with the numeric clause's delegation pattern), or, if an in-spec rule is preferred, state explicitly:

- C0 control characters that have a standard short escape (`\b` U+0008, `\t` U+0009, `\n` U+000A, `\f` U+000C, `\r` U+000D) MUST use the short form; all other characters in U+0000–U+001F MUST use the `\uXXXX` form;
- `\uXXXX` hex digits MUST be lowercase;
- `"` and `\` use their short escapes; `/` is not escaped; all other characters (including non-ASCII) are emitted as raw UTF-8.

Naming `JSON.stringify` as the authority resolves all three facets at once and matches the Node-only host assumption already in force. Edge case the implementer must watch: U+2028/U+2029 are *not* escaped by `JSON.stringify` and must therefore be emitted raw under the byte-exact rule.

## Consequence

**Severity:** correctness

Two conforming implementations produce different schema slugs for the same source whenever a `const`/`enum` string contains a C0 control character, since "minimal-escape" leaves short-form-vs-`\uXXXX` and hex-digit case free. Divergent slugs mean divergent `__inline_<slug>` `$defs` keys, divergent AJV cache keys, and divergent on-wire synthesised tool names (`__loom_respond_<slug>`, `__loom_callee_<slug>__…`, `__loom_bind_<slug>`), breaking the explicitly-declared byte-exact cross-implementation, cache-reuse, and replay contract.
