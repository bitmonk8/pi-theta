## Finding

`V9f` ("Tool-registration lifetime and visibility") carries six facets in one leaf: per-mode tool wiring (subagent `customTools` vs prompt-mode registration cache), the `Type.Unsafe` schema bridge, the active-set gating window's snapshot/restore failure paths (`PIC-8`/`PIC-19`), the no-`unregisterTool` rule, registration-cache-collision disambiguation, and the `ToolDefinition.label` derivation (the `cka-17` code-keyed obligation). Five of these are cohesive facets of one theme — how a loom's tool definition is installed into and removed from Pi's active-tool set, including the snapshot/restore failure handling.

The sixth facet, the `ToolDefinition.label` derivation, is orthogonal to that machinery. It is a pure string transform over a loom filename (`code-review.loom` → `"Code-review"`, plus the synthesised typed-query tool's literal label `"Loom typed-query response"`) with no dependency on the snapshot/restore lifetime, the cache, or the failure paths. It rides along in `V9f` only because both touch the same `ToolDefinition` struct. A reviewer must context-switch between the stateful restore-failure machinery and an isolated string-munging rule that shares nothing with it. The label derivation closes a distinct coverage obligation (`cka-17`) on a different spec page (`extension-bootstrap-and-per-loom.md`) from the lifetime obligations (`tool-registration-lifetime.md`, `PIC-8`/`PIC-19`).

## Issue introduction

**Verdict:** introduced by a later plan-fix (single introducing commit)
**Introducing commit:** `cfd76f7` — "pi-loom plan: resolve \"Per-loom registration ToolDefinition field-derivation MUSTs unnamed by any leaf\"" (2026-06-12)
**History:** At inception (`c6a664e`, 2026-06-10) `V9f` carried five cohesive facets, all within the tool-registration-lifetime theme; it had no `ToolDefinition.label` content and cited only `tool-registration-lifetime.md`. The sixth, orthogonal facet was grafted on by `cfd76f7`, which closed a prior review's coverage gap (the `ToolDefinition` field-derivation MUSTs had no owning leaf) by attaching the label derivation to `V9f`, adding the `extension-bootstrap-and-per-loom.md` Spec citation, and inserting the `cka-17` coverage-matrix row. The underlying coverage gap that `cfd76f7` closed was real; its side effect was to make `V9f` host a concern unrelated to the rest of the leaf.

## Solution Space

**Shape:** multiple

### Option A — Relocate the `cka-17` label derivation to its own leaf

**Approach.** Give the `ToolDefinition.label` derivation a cohesive home of its own, separate from the lifetime/restore machinery, while keeping the `cka-17` obligation owned (do not revert to the unowned-MUSTs state `cfd76f7` closed).

**Plan edits.**
- In `V9f-tool-registration-lifetime.md`: strike from `Adds` the clause "and the materialised `ToolDefinition.label` derivation — … `"Loom typed-query response"`."; remove the fourth `Tests` bullet (the `extension-bootstrap-and-per-loom.md` label-derivation bullet); strike from `Ships when` the "and the `ToolDefinition.label` derivation (…)" clause; drop the `extension-bootstrap-and-per-loom.md` reference from the `Spec` field (it is cited only for the label derivation). `PIC-8`/`PIC-19`, the cache-collision/visibility test, and the remaining Adds facets stay in `V9f`.
- In `V9f-T-tool-registration-lifetime.md`: remove the matching label-derivation `Tests` bullet.
- Add a new Vertical leaf `<new>` (and paired `<new>-T`) owning the `ToolDefinition.label` derivation, citing `extension-bootstrap-and-per-loom.md` §Per-loom registration and carrying the two literal-label assertions and the GOV-22 un-anchored-residue note.
- In `coverage-matrix.md`: move the `cka-17` row's closing leaf from `V9f` to `<new>`.
- In `plan.md` Vertical slices: add the `<new>` / `<new>-T` index entry if that section enumerates leaves.

**Spec edits.** None.

**Pros.** Each leaf reviews as one concern; the orthogonal string rule no longer rides with the restore-failure machinery; `cka-17` stays owned.

**Cons.** Re-fragments what `cfd76f7` deliberately consolidated; produces a small singleton leaf; touches 4–5 plan files.

**Risks.** The new singleton leaf could itself draw a "too small" atomicity note; the `cka-17` row move must keep the row's joint-completeness clean (conventions item 7) — here it is a single-leaf row, so the move is mechanical.

### Option B — Keep `V9f` as-is and accept the theme cohesion

**Approach.** Treat all six facets as facets of "materialise and register the loom's `ToolDefinition`", on the grounds that the label is a field of the same struct the leaf installs.

**Plan edits.** None.

**Spec edits.** None.

**Pros.** Zero churn; no new singleton leaf; preserves `cfd76f7`'s consolidation.

**Cons.** The reviewer context-switch the finding identifies persists; the label rule remains coupled to unrelated failure-path machinery.

**Risks.** None beyond the standing reviewability cost.

### Recommendation

Take Option A. The issue-introduction history shows the label derivation was grafted onto `V9f` as a convenient owner while closing an unrelated coverage gap, so the durable fix is a cohesive home for `cka-17` rather than leaving it coupled to the restore-failure machinery. Order the edits to bound scope: first strike the label clauses from `V9f`/`V9f-T`, then create `<new>`/`<new>-T`, then relocate the `cka-17` coverage row — so the row move lands on an already-stripped baseline. Watch that dropping the label clause also removes `V9f`'s only reason to cite `extension-bootstrap-and-per-loom.md`, and that the GOV-22 un-anchored-residue note travels with the obligation to `<new>`.

## Consequence

**Severity:** cosmetic

The leaf reviews as two unrelated concerns under one heading; a reviewer holding the snapshot/restore failure semantics must also hold an isolated filename-to-label string rule. No implementer is blocked, no spec obligation goes uncovered, and no two implementers would diverge — the leaf is fully specified. The cost is purely reviewability.
