## Finding

Five numbered REQ-IDs (`PIC-24`…`PIC-28`) live inside one unbroken Markdown paragraph in `diagnostic-emission-isolation.md`, and four (`PIC-45`…`PIC-48`) live inside one unbroken paragraph in `unknown-reason-rule.md`. Because each paragraph is a single physical source line, the line-resolved ID projection (`ids.md`) maps every member of each group to the same `file:line` location: `PIC-24`–`PIC-28` all resolve to `diagnostic-emission-isolation.md:3`, and `PIC-45`–`PIC-48` all resolve to `unknown-reason-rule.md:6`. A reviewer auditing conformance through the index cannot navigate to a specific ID's obligation — every sibling lands at the identical line, and the per-ID scope boundary (where one obligation's normative text ends and the next begins) is discoverable only by reading the whole wall of prose and scanning for the next inline `**PIC-N.**` label.

The obligations themselves are correctly identified and individually anchored — each ID carries its own `<a id="pic-N">` anchor, and the finer-grained sub-parts that warrant independent citation (`PIC-45`/`46`/`47`'s `(i)`/`(ii)`/`(iii)` and `PIC-48`'s four protected substrings) already carry their own sub-anchors (`#unknown-reason-rule-membership-check`, `#unknown-reason-rule-substring-a` … `-d`, etc.). The defect is purely the prose packaging: the in-document anchors exist, but the line-based projection collapses them, defeating the per-ID traceability the numbering was coined to provide. The fix is paragraph structure, not re-numbering — neither `PIC-25` (one coherent serialisation-and-emission wrapping contract) nor `PIC-48` (one anchor-stability rule enumerated over four already-sub-anchored substrings) is a bundle of independent obligations that needs decomposing into new IDs.

## Issue introduction

**Verdict:** introduced by specific commits
**Introducing commits:** `82856ab` (PIC-24…PIC-28), `a9fb344` (PIC-45…PIC-48), both 2026-06-25
**History:** The host paragraphs are older prose; the multi-ID-per-line collapse could not exist until the numbered IDs were coined into them. `git log -S 'pic-24'` on `diagnostic-emission-isolation.md` returns exactly `82856ab` ("coin PIC-24..28 anchors on diagnostic-emission-isolation.md"), and `git log -S 'pic-45'` on `unknown-reason-rule.md` returns exactly `a9fb344` ("coin PIC-45..48 on unknown-reason-rule.md"). `git show --stat 82856ab` shows a 2-line change to the page (`1 insertion, 2 deletions`) and `git show 82856ab^:…` confirms the line-3 paragraph already existed beforehand — the commit retrofitted the inline labels into the pre-existing unbroken paragraph rather than splitting it. The two commits are an hour apart and share the identical pattern (coin sibling IDs inline without breaking the host paragraph), so the defect entered at each of the two ID-coining commits independently rather than at a single point.

## Solution Space

**Shape:** single

### Recommendation

Break each host paragraph so that every `**PIC-N.**` inline label begins its own Markdown paragraph (hence its own source line), restoring per-ID line resolution in the `ids.md` projection. Preserve verbatim, with no content change: every inline `**PIC-N.**` label, every `<a id="pic-N">` anchor, and every sub-anchor (`#unknown-reason-rule-membership-check`, `#unknown-reason-rule-constant-source`, `#unknown-reason-rule-handler-trycatch`, `#unknown-reason-rule-substring-a` … `-d`, `#pic-emission-swallow`, `#pic-emission-visibility`).

Do not coin `PIC-48a`–`d` and do not decompose `PIC-25` into separate IDs: `PIC-48` is a single anchor-stability rule applied uniformly across four substrings that already carry their own sub-anchors, and `PIC-25` is one coherent serialisation-and-emission wrapping contract; both keep their internal sub-clauses inside their own (now-standalone) paragraph.

Edge cases the implementer must watch:

- In `diagnostic-emission-isolation.md`, the paragraph opens with the `**Diagnostic-emission isolation.**` umbrella (carrying the `#diagnostic-emission-isolation` anchor and the `(a)`/`(b)`/`(c)` site-class enumeration that precedes `PIC-24`). Keep that umbrella as its own leading paragraph, then give `PIC-24`–`PIC-28` each a following paragraph.
- In `unknown-reason-rule.md`, `PIC-45`/`46`/`47` currently share the `*Sub-anchors and pinned substrings.*` lead-in sentence. Keep that lead-in attached to `PIC-45` (or as its own short preamble paragraph) so the `(i)`/`(ii)`/`(iii)` framing is not orphaned; then `PIC-48` gets its own paragraph.
- The split shifts line numbers; the index regenerates on the next review run. All in-corpus citations to these IDs are by anchor (`#pic-24`, `#diagnostic-emission-isolation`) or by ID label, not by line, so no inbound citation breaks — but confirm none of the `## Acceptance criteria` cross-references or the coverage matrix keys on a raw `file:line`.

## Consequence

**Severity:** advisory

Conformance auditing via the ID index is degraded: a reviewer citing `PIC-26` (or any sibling) is sent to a `file:line` shared by four or five other IDs and must read a 1,000-plus-word paragraph to find the obligation's boundaries. No two implementers diverge on behaviour — the obligations remain readable and anchored — so a working system is still producible; the harm is to review/audit navigability, not to the implemented contract.
