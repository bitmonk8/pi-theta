## Finding

The harness-wiring epilogue at the end of `audit-wire-and-canary.md` pins a fail-closed MUST against the specific failure mode where the audit child process exits non-zero but emits zero records: "The implementation MUST wire the audit so that an audit child process whose exit code is non-zero AND that emits zero records (including process-killed and uncaught-exception terminations, per the *Pre-emission termination carve-out* above) causes `npm test` to exit non-zero. For the in-process variant, the wrapping `expect(...)` MUST be unconditionally registered before the audit invocation so a process kill aborts the harness and propagates non-zero. An implementation wiring that interprets empty stdout as 'no failures' violates this obligation." This is the load-bearing safety net for the *Pre-emission termination carve-out* (`SIGKILL`, walker panic mid-write, OOM-killer) — the one path on which no `audit/<class>/<family>/<symptom>` record can ever appear, so a CI parser asserting on the stdout discriminator cannot witness the failure on its own.

Every other normative section in the PIC corpus that pins a behaviour the implementation must satisfy on a specific code path follows the pattern of a dedicated "Acceptance criteria" block that decomposes the MUST into per-branch fixture obligations the leaf must cover — `drain-state-contract.md` does this for the three-arm `readDrainState` enumeration, `session-only-degraded-state.md` enumerates 13+ per-branch fixtures (a)–(m), `diagnostic-emission-isolation.md` enumerates the construction-site throw fixtures, and `audit-failures.md` itself pins "Each negative-test fixture MUST assert that …" against every violation-family discriminator. The fail-closed wiring MUST in this page has no such block: nothing requires a fixture that seeds an audit child which exits non-zero before any record is flushed and asserts the wrapping `npm test` invocation goes red.

The downstream V18b leaf (`plan_topics/V18b-inventory-audit.md`, paired with `V18b-T`) enumerates tests for off-inventory references, exemption passing, the non-empty canary, and family-(4) shape detection, but does not enumerate a SIGKILL / uncaught-exception / pre-emission-termination fixture against the harness wiring. An implementer reading the wiring epilogue without an acceptance-criteria block will ship a wiring that *appears* to satisfy the MUST (the `expect(...)` pattern is described as the in-process recipe) while the silent-pass failure mode — a wiring whose assertion sees empty stdout and concludes "no failures" rather than "process killed before emitting" — goes undetected on every `npm test` invocation. The MUST exists; the gate that proves it holds does not.

## Solution Space

**Shape:** single

### Recommendation

Add a fixture-obligation clause to the harness-wiring epilogue in `audit-wire-and-canary.md`, parallel to the "Each negative-test fixture MUST assert that …" clause the *Failure-surface contract* uses for violation records. The clause must require at least two fixtures the leaf MUST cover, both seeded against the wiring path the MUST pins:

1. **Child-process variant.** Seed an audit child process that exits non-zero before emitting any record on stdout (e.g. by having the fixture invoke a wrapper that issues `process.kill(audit.pid, 'SIGKILL')` before the first record flush, or by injecting an uncaught throw inside the audit walker at the same pre-emission point), wired through the chosen `npm test` script, and assert the outer `npm test` invocation exits non-zero with empty audit stdout. This fixture witnesses the *Pre-emission termination carve-out* path end-to-end and would catch any wiring that interprets empty stdout as success.
2. **In-process variant.** Seed an in-process audit invocation that throws (or is otherwise aborted) before any record is produced and assert that the unconditionally pre-registered wrapping `expect(...)` causes the harness to go red. This fixture witnesses the "wrapping `expect(...)` MUST be unconditionally registered before the audit invocation" clause and would catch a deferred-registration wiring.

Both fixtures live alongside the existing audit fixtures the V18b / V18b-T pair owns and follow the leaf-owns-shape / PIC-owns-the-required-coverage division the other PIC acceptance-criteria sections use. The clause should explicitly call out that these two fixtures are required regardless of which wiring (in-process or child-process) the leaf adopts as its primary — the leaf publishes both wirings or, if it elects only one, publishes a justification for skipping the other variant's fixture. Update `V18b` and `V18b-T`'s `Tests.` list to enumerate the two new fixtures so the plan leaf carries the obligation forward into implementation.

Edge case for the implementer: the SIGKILL fixture must be implementable on Windows hosts (`process.kill` with `SIGKILL` is treated as a forceful terminate on Win32), or the fixture must use a platform-portable equivalent (`process.exit(1)` from inside the audit before any record flush) that exercises the same wiring path; the spec clause should pin the *behavioural* requirement (non-zero exit, zero records on stdout) rather than a specific kill mechanism.

## Consequence

**Severity:** advisory

The fail-closed wiring is contributor-discipline rather than mechanically gated. Two implementers reading the section can both believe they satisfy the MUST while one of them has a wiring that silently treats empty stdout from a killed child as green — the exact failure mode the *Pre-emission termination carve-out* depends on the wiring to catch. The audit's other guarantees (the family-(N) violation records, the canary, the infrastructure-failure records) all have fixture-bearing acceptance criteria; only the pre-emission-termination path is left to honour-system implementation, and it is the one path that has no in-band record an external CI parser could assert on.
