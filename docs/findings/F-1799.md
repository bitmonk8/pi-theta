## Problem

The harness-wiring epilogue at the end of `audit-wire-and-canary.md` pins a fail-closed MUST: an audit child process that exits non-zero while emitting zero records MUST cause `npm test` to exit non-zero, and for the in-process variant the wrapping `expect(...)` MUST be registered before the audit invocation. Unlike every other normative behaviour in the PIC corpus that carries a fixture obligation — the *Failure-surface contract*'s "Each negative-test fixture MUST assert that …" clause being the sibling on this same page — this MUST has no acceptance-criteria block requiring a fixture that exercises it. The *Pre-emission termination carve-out* is the one path on which no `audit/<class>/<family>/<symptom>` record ever appears, so a CI parser cannot witness the failure in-band. Without a fixture obligation the fail-closed wiring is honour-system, and a silent-pass wiring that reads empty stdout as "no failures" goes undetected on every `npm test` run.

## Issue introduction

**Verdict:** defect-introducing-commit
**Introducing commits:** e5cd514 ("pi-loom spec: resolve 'T35 SDK capability inventory closed-set negative-direction audit'", 2026-05-09)
**History:** `git log -G "audit child process whose exit code is non-zero AND that emits zero records"` traces the harness-wiring epilogue — including the zero-records / non-zero-exit MUST and the "An implementation wiring that interprets empty stdout as 'no failures' violates this obligation" clause — to its introduction in e5cd514 on `docs/spec_topics/pi-integration-contract.md`. The commit added the entire audit-wire/canary surface (the *Failure-surface contract*, *Infrastructure-failure handling*, *Non-empty-scan canary*, and the harness-wiring epilogue) in one edit, and paired the violation-family discriminators with the explicit "Each negative-test fixture MUST assert that …" obligation in the *Failure-surface contract* section, but landed the harness-wiring epilogue without any equivalent fixture clause. The page was later mechanically extracted to `audit-wire-and-canary.md` by f5e89f4 (2026-06-04, "enforce 100KB hard / 30KB recommended size cap on spec set", a content-preserving split) and re-pointed by 81bf52a (2026-06-11, spec/ → spec_topics/ consolidation, also content-preserving); neither file move altered the epilogue prose. The defect is the omission, on the original authoring commit, of a fixture clause against the fail-closed wiring MUST while the sibling violation-record MUST received one.

## Solution approach

Add a fixture-obligation clause to the harness-wiring epilogue in `audit-wire-and-canary.md`, parallel to the *Failure-surface contract*'s "Each negative-test fixture MUST assert that …" clause, requiring fixtures that seed an audit terminated before emitting any record — covering both the child-process and in-process wirings the epilogue already names — and assert that the wrapping `npm test` invocation exits non-zero with empty audit stdout. Pin the behavioural requirement (non-zero exit, zero records on stdout) rather than a specific kill mechanism so the fixture is platform-portable.
