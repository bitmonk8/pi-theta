## Finding

H3a's **Adds** makes H3a the owner of each host-seam interface's full member signatures (`Checkpoint`, `SchemaValidator`, `Clock`, `FileSystem`, `FileWatcher`, `TokenEstimator`, `IdSource`), declaring them "the single source of each interface's shape" sourced from `host-interfaces-core.md` / `host-interfaces-services.md`, with the `V8a`–`V8e` leaves implementing against those declarations "rather than redeclaring members." This makes the H3a-declared signatures load-bearing for every downstream seam consumer.

H3a's **Tests**, however, assert only two things: that two constructed runtime roots yield isolated seam graphs sharing no mutable state, and the architectural ambient-access scan. Neither test — nor the **Ships when** condition — asserts that the declared member signatures (member names, arity, parameter/return types) actually match the cited seam-contract spec pages. A signature that is mis-named, has the wrong arity, or the wrong type type-checks cleanly at H3a precisely because H3a is the single source of shape and nothing in the leaf contradicts it.

Because the `V8a`–`V8e` leaves implement against H3a's declarations rather than re-deriving members from the spec, and their own Tests assert seam *behaviour* (e.g. V8a's `PIC-10` checkpoint-yield semantics) rather than signature fidelity against the spec page, a wrong signature is not reliably caught downstream either — at best it surfaces transitively if a consumer happens to exercise the exact member shape, otherwise it ships. The leaf claims a fidelity guarantee ("single source of shape," "sourced from the seam-contract pages") that no test enforces.

## Issue introduction

**Verdict:** introduced-by-specific-edit
**Introducing commits:** `592219d` — "pi-loom plan: resolve \"H3a seam interfaces cite no source for member shapes\"" (2026-06-11)
**History:** Before `592219d`, H3a's Adds read "Seam *interfaces* only; their normative behaviour is implemented by the `V8*` leaves" and made no claim to declare or own the member signatures — so there was no signature-fidelity obligation to verify. `592219d` rewrote the Adds to make H3a "the single source of each interface's shape," sourced from `host-interfaces-core.md` / `host-interfaces-services.md`, and have the V8* leaves implement against it. That edit added the ownership claim and its spec sourcing but added no companion Tests bullet asserting the declared signatures match the cited pages; the fidelity-check gap entered the corpus with that commit. The `git log --follow` history of `H3a-di-seam-skeleton.md` shows no later commit closing the gap (subsequent edits — `c6824f2`, `cb8737a` — touched the ambient-scan ban set and over-bundling, not the seam-signature verification arm).

## Solution Space

**Shape:** single

### Recommendation

Add a Tests bullet to `docs/plan_topics/H3a-di-seam-skeleton.md` that pins each declared seam interface against its owning spec anchor, so the "single source of shape" claim is mechanically backed rather than asserted. The new test must verify, for each of the seven seam interfaces, that the H3a-declared member set matches exactly the members the cited page defines (member names, arity, and parameter/return types) — a structural / `tsc`-level assertion against `host-interfaces-core.md` and `host-interfaces-services.md` (with `Checkpoint` pinned at `host-interfaces-services.md#checkpoint-seam`, matching the anchor the Adds already cites). The assertion's fidelity target is the spec pages, which remain read-only for this fix; do not move signature ownership out of H3a or have the V8* leaves redeclare members.

Edge cases the implementer must watch: the spec pages declare only loom-load-bearing *subsets* of several Pi types (e.g. the `ExtensionContext` / `AgentSession` member surfaces are explicitly "members loom touches," not Pi's full type), so the assertion must pin H3a's declared members against the subset the cited page enumerates, not against the upstream Pi `.d.ts`; and `Checkpoint` is cited under `host-interfaces-core.md` by V8a but anchored in `host-interfaces-services.md` — the test must follow the anchor the Adds names.

## Consequence

**Severity:** correctness

A member-signature defect in H3a's declared interfaces (wrong name, arity, or type) type-checks at H3a and is not asserted against the spec by any H3a test; because the V8* leaves implement against H3a's declarations and assert seam behaviour rather than signature fidelity, the defect can reach a consumer transitively or ship undetected, yielding a seam surface that diverges from the spec pages H3a claims to mirror.
