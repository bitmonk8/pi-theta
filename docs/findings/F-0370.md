## Finding

`V9a` and its test twin `V9a-T` both carry the PIC-5 test bullet *"there are exactly five checks; no probes beyond them."* The second clause is a negative universal over the probe's runtime behaviour: on a capable host every enumerated check passes and produces no observable output, so a behavioural test cannot distinguish a probe that runs exactly the five enumerated sub-steps `(a)`–`(e)` from one that also runs a sixth host-member probe that happens to pass. A behavioural suite can prove that each of the five checks *fires* (by removing a member and observing the deterministic `details.kind` / `details.member` payload, which the failure-kind bullet already covers), but it cannot prove the *absence* of an additional probe.

The spec invariant this bullet implements — [`capability-probe.md` PIC-5](../spec_topics/pi-integration-contract/capability-probe.md): *"Probes MUST NOT be added beyond the five enumerated checks"* — is sound; the defect is that the plan restates it as a behavioural test obligation that the test cannot discharge. The companion PIC-5 bullet (`FACTORY_PROBABLE_CAPABILITIES` enumerates exactly the five factory-probable capability identifiers) pins the *constant*, and PIC-3/PIC-4 constrain the probe to `typeof`/`in` over named members with no construction, but none of these closes "the probe body performs no further host-member sniffing." That residual universal is observable only from the probe's source, not its behaviour.

## Solution Space

**Shape:** multiple

### Option A — Scope the claim to the verifiable property

**Approach.** Reword the first PIC-5 bullet so it asserts only what is observable: that the probe runs the five enumerated sub-steps `(a)`–`(e)` in the fixed short-circuit order. Drop the "no probes beyond them" / "no further host-member sniffing" promise, leaning on the enumeration bullet plus PIC-3/PIC-4 for the rest.

**Plan edits.** In `V9a-capability-probe.md` and `V9a-T-capability-probe.md`, replace the bullet text `there are exactly five checks; no probes beyond them` with a bullet that asserts the five enumerated sub-steps fire in the spec's fixed order. Leave the `FACTORY_PROBABLE_CAPABILITIES` enumeration bullet unchanged.

**Spec edits.** None.

**Pros.** Minimal diff; no new test machinery; no source-inspection tooling.

**Cons.** PIC-5's normative "MUST NOT add probes beyond the five" invariant is left unverified at the leaf level; a future edit adding a sixth host-member probe would not be caught by `V9a`'s suite.

**Risks.** Silently narrows coverage of a normative invariant while `coverage-matrix.md` still maps `PIC-5 → V9a`, implying coverage that no longer fully exists.

### Option B — Add a source/static-inspection check

**Approach.** Re-express the first PIC-5 bullet as a source-level (static / AST) assertion over the probe function body: the set of host-member probe sub-steps in the body equals the enumerated `(a)`–`(e)` set, with no additional host-member probe. Both "exactly five checks" and "no probes beyond them" become observable from the probe's source rather than its runtime behaviour.

**Plan edits.** In `V9a-capability-probe.md` and `V9a-T-capability-probe.md`, reword the first PIC-5 bullet to specify a static inspection of the probe body confirming the host-member probe set equals the enumerated five sub-steps and contains no sixth probe.

**Spec edits.** None.

**Pros.** Actually verifies PIC-5's normative invariant; catches a future sixth probe; consistent with PIC-4, which already frames the probe as `typeof`/`in`-only and is itself a static-posture claim.

**Cons.** Requires source/AST inspection in the test; more sensitive to internal probe-body refactors.

**Risks.** The static check needs maintenance if the probe's internal structure changes.

### Recommendation

Adopt Option B. PIC-5 is a normative invariant the coverage matrix already attributes to `V9a`, and a static/source-level inspection is the only way to verify the "no probes beyond the five" universal; PIC-3/PIC-4 already establish a static-analysis posture for the probe, so an additional source-level check is consistent with how the suite already reasons about the probe. Edge case the implementer must watch: "five checks" maps to the five logical sub-steps the spec enumerates — `(a)` Node floor, `(b)` `AbortSignal` shape, `(c)`+`(d)` capability list and peer-dep iteration, `(e)` `typebox` shape — not to a literal count of `typeof` invocations (step `(c)` alone iterates nine SDK members and step `(d)` iterates four packages). The inspection must count logical probe sub-steps, not raw host-member accesses.

## Consequence

**Severity:** advisory

An implementer can satisfy the first PIC-5 bullet with a behavioural assertion that passes vacuously — it observes the five checks firing but never verifies that no sixth probe exists — so PIC-5's normative "MUST NOT add probes beyond the five" invariant goes unverified even though the coverage matrix attributes it to `V9a`. The probe still works and the leaf still ships; the gap is undetected coverage erosion, not a broken probe.
