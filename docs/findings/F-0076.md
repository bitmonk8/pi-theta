## Problem

`V10b` enforces the per-read deadline via `Clock.setTimeout` and `V10b-T` drives the `package-read-timeout` / `discovery-slow` bounds through the `FakeClock` test double, but the `Clock` runtime seam and its `FakeClock` fake are owned by `V8d`. Neither leaf declares `V8d` in **Deps** (`V10b`: `V10b-T, V10a, V10c, V8b`; `V10b-T`: `V10a, V10c, V8b`); `V8b` covers only the `FileSystem` seam after the V8 split moved `Clock` to `V8d`. Under the plan's dependency-driven build order, `V10b-T` / `V10b` can be scheduled before `V8d` introduces `Clock` / `FakeClock`, at which point the references fail to compile; the leaf builds today only by group-order accident, which the plan disclaims as a sequencing guarantee.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** c6a664e ŌĆö pi-loom plan: build/update plan for spec.md + review (2026-06-10, Thomas Andersen); cb8737a ŌĆö pi-loom plan: resolve "Systemic leaf over-bundling across the plan corpus" (2026-06-12, Thomas Andersen)
**History:** At c6a664e the `Clock` seam lived in the bundled V8b leaf (slug `clock-fs-id-watch-token-seams`), and V10b/V10b-T correctly cited `V8b` in Deps while already using `Clock.setTimeout`/`FakeClock`. cb8737a split that bundle, moving `Clock`/`IdSource` into the new `V8d` leaf, but did not retarget V10b/V10b-T's Deps from `V8b` to `V8d`. The Clock dependency was therefore correct at creation and was orphaned by the later seam split ŌĆö the defect arises from the interaction of the two commits.

## Solution approach

Add `V8d` to the **Deps** field of both `docs/plan_topics/V10b-package-discovery.md` and `docs/plan_topics/V10b-T-package-discovery.md`, alongside the existing entries, so the `Clock` / `FakeClock` dependency is declared on each leaf.

## Solution constraints

- Retain `V8b` on both leaves ŌĆö it remains the owner of the `FileSystem` seam the bounded walk reads; this is an addition, not a retarget.
- Out of scope: `V10c`'s parallel `V8d` omission, tracked as a separate finding.
