## Finding

Across the plan, *closing gate* is the canonical name for the single CI enforcement gate owned by `H5a` (`H5a` — "REQ-ID / diagnostic-code closing-gate automation"). Every other leaf that mentions the gate — `H5b`, `H5c`, `H6a`, `coverage-matrix.md`, `plan.md` — refers to it as "the closing gate" / "the `H5a` closing gate" and treats `H5a` as its sole owner.

`V7b`'s leaf title — "Diagnostic code registry and closing gate" — borrows that same phrase, but `V7b` does not own a gate. Its `Adds.` states it contributes "the machine-checkable diagnostic registry … and the closed-set + stable-id enforcement that the `H5a` gate *consumes*", and its `Tests.` (DIAG-2/3/4) and `Ships when.` describe the registry data and rules that the `H5a` gate reads. The title therefore implies `V7b` *is* a closing gate rather than the registry data and closed-set/stable-id rules a gate enforces, colliding with `H5a`'s ownership of the term. The same overloaded phrase is carried verbatim into the paired tests leaf title (`V7b-T`) and the `plan.md` table-of-contents entry.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e
**History:** The `V7b` leaf file was added in `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10) already carrying the title "Diagnostic code registry and closing gate"; the overloaded "closing gate" phrase has been present since the leaf's inception. The only later commit touching the file, `f284534` ("resolve 'DIAG-2 src/** emission scan vs closing gate'"), left the title unchanged. The `V7b-T` title and the `plan.md` TOC entry carry the same phrase from the same origin.

## Solution Space

**Shape:** single

### Recommendation

Strike "and closing gate" from `V7b`'s title so the leaf name describes the registry/enforcement data it owns rather than the gate `H5a` owns. Apply the rename consistently in all three places the overloaded title text appears:

- `docs/plan_topics/V7b-code-registry.md` line 1 — change `# `V7b` — Diagnostic code registry and closing gate` to `# `V7b` — Diagnostic code registry`.
- `docs/plan_topics/V7b-T-code-registry.md` line 1 — change `# `V7b-T` — Diagnostic code registry and closing gate (tests)` to `# `V7b-T` — Diagnostic code registry (tests)`.
- `docs/plan.md` line 106 — change the TOC link text `[`V7b` — Diagnostic code registry and closing gate]` to `[`V7b` — Diagnostic code registry]`.

If the enforcement aspect should remain visible in the title, a phrasing such as "Diagnostic code registry and closed-set enforcement" is acceptable — the only constraint is that the canonical *closing gate* term, owned by `H5a`, is not reused for `V7b`. Leave the `V7b` `Adds.`/`Tests.`/`Deps.`/`Ships when.` body unchanged; they already describe the registry as data the `H5a` gate consumes.

## Consequence

**Severity:** cosmetic

An implementer reading only the `V7b` title may expect to build a CI gate at `V7b`, duplicating or conflicting with the `H5a` gate; the `V7b` `Adds.`/`Tests.` body clarifies the true scope (registry + enforcement data), so the risk is confined to title-level confusion rather than divergent implementation. References to `V7b` elsewhere (`coverage-matrix.md`, `V4b`, `V7c`) cite the leaf by ID, not by title, so a rename does not break them.
