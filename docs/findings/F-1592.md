## Finding

`spec_topics/errors-and-results/queryerror-variants.md` ERR-17 ("Forced-respond non-compliance synthesised shapes") cites the synthesised tool name `__loom_respond_<slug>` four times — twice in prose, once inside the wrong-tool branch `message` literal (`"model invoked tool '<provider-emitted-tool-name>' instead of the forced respond tool '__loom_respond_<slug>'"`), and once in the `respond_repair.methodology: none` paragraph. The page never defines what `<slug>` denotes and never cross-links the definition. The only outbound pointer that fires near these citations is to `pi-integration-contract/conversation-drive.md#pic-typed-query-noncompliance`, which is the *behavioural-classification* limitation, not the slug-derivation rule.

This is a conformance hazard because the `message` literal is a normative wire string the runtime emits, with `<slug>` interpolated for the actual schema slug at runtime. A second implementer reading ERR-17 in isolation can plausibly read `<slug>` as one of: (a) a literal placeholder to be emitted as the four-character string `<slug>`; (b) the tool's sanitised post-rename name; (c) a per-invocation identifier; or (d) some hash variant of the schema. Only (e) — the 16-lowercase-hex schema slug produced by the canonical schema hash recipe in `spec_topics/schema-subset.md` — is correct, and that recipe is two cross-references away from ERR-17 (the prose never names it).

The fix is a single sentence in ERR-17, at first use of `__loom_respond_<slug>`, pinning `<slug>` to the canonical schema hash and the synthesised-names list: e.g. *"`<slug>` is the 16-lowercase-hex schema slug of the lowered response schema, produced by the canonical schema hash recipe in [Schema Subset — Canonical schema hash](../schema-subset.md#canonical-schema-hash); it is the same slug under which the respond tool was registered per [Schema Subset — Synthesised names](../schema-subset.md#synthesised-names)."* Once that sentence lands, both message literals and the prose references resolve unambiguously, and the same pin propagates to the wrong-tool branch's runtime-interpolation contract.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `82be125` (2026-05-04, "spec: adopt P1 typed-query two-phase loop (Option A)") coined `__loom_respond_<slug>` as the synthesised tool-name pattern alongside the typed-query two-phase loop, originally homed in `spec_topics/pi-integration-contract.md` next to the registration cache prose; `22619b0` (2026-05-25, "T11d / T11a — forced-respond exempt-routed terminator") added the wrong-tool branch and its `message` literal containing the bare `'__loom_respond_<slug>'` token to that same PIC page; `34517a0` (2026-06-05, "Normative ValidationIssue/Err shapes live on a protocol page") relocated the synthesised `ValidationIssue` and terminal `Err` shapes — including every `__loom_respond_<slug>` mention — out of `pi-integration-contract/conversation-drive.md` and into the freshly-coined ERR-17 block on `errors-and-results/queryerror-variants.md`, leaving behind only the behavioural-classification cross-link.
**History:** `git log --follow --diff-filter=AM -S '__loom_respond_<slug>'` on the spec corpus pins the three commits above as the only edits that introduced the defect tokens; `git diff 22619b0^..22619b0` and `git show 34517a0 -- spec_topics/errors-and-results/queryerror-variants.md` confirm that the wrong-tool `message` literal moved verbatim from the PIC page to the queryerror-variants page in the relocation. Before `34517a0`, the page hosting the literal also hosted the registration prose that named the slug source ("On first encounter of a unique slug the runtime calls `pi.registerTool` once with a content-addressed name …`__loom_respond_<slug>`"); after `34517a0`, ERR-17 references `__loom_respond_<slug>` four times with no co-located slug-derivation pointer. The defect is therefore introduced by the interaction of the original placeholder-style naming (`82be125`) and the relocation that severed it from its in-shard derivation context (`34517a0`); neither commit on its own is the proximate cause, but `34517a0` is where the in-shard discoverability gap opens.

## Solution Space

**Shape:** single

### Recommendation

In `spec_topics/errors-and-results/queryerror-variants.md`, at the first mention of `__loom_respond_<slug>` inside the ERR-17 paragraph (the opening sentence: "When a typed query's forced respond turn does not invoke the synthesised `__loom_respond_<slug>` tool …"), add a parenthetical or trailing sentence pinning the slug derivation and the runtime-interpolation contract, anchored to the existing source-of-truth recipe:

> `<slug>` is the 16-lowercase-hex schema slug of the lowered response schema, produced by the canonical schema hash recipe in [Schema Subset — Canonical schema hash](../schema-subset.md#canonical-schema-hash); it is the same slug under which the respond tool was registered per [Schema Subset — Synthesised names](../schema-subset.md#synthesised-names), and the wrong-tool branch's `message` literal MUST interpolate the actual slug rather than emitting the placeholder token `<slug>` verbatim.

Edge cases the implementer must watch:

- The `respond_repair.methodology: none` paragraph in the same block also says "the *first* forced respond turn that does not invoke `__loom_respond_<slug>`" — the same interpolation rule applies (it is the same slug for the same lowered schema across attempts), so no second pin is needed once the first-use sentence lands.
- The `validator_error` follow-up template in `query-failure-and-repair.md` already pins its `<slug>` placeholder to the same recipe; the ERR-17 sentence above intentionally restates the same pin rather than forward-cross-linking to the query page, because ERR-17 is the wire-shape owner and the query page is the template renderer — both pages independently cite the slug and both should ground out at `schema-subset.md` directly.
- Do not move the literal `<slug>` notation out of the `message` string. The string is the on-the-wire template; the runtime interpolates at emission time. The pin sentence clarifies *what* gets interpolated, not the literal shape of the template.

## Consequence

**Severity:** correctness

Two reasonable implementers reading ERR-17 in isolation will produce divergent runtime output. The wrong-tool branch `message` is a normative literal; an implementer who reads `<slug>` as a literal placeholder (rather than an interpolation point pointing at the canonical schema hash) will emit `…the forced respond tool '__loom_respond_<slug>'` verbatim, and the runtime's diagnostic output, the respond-repair follow-up `<ajv-summary>` rendering (which carries this `message` forward into the next user turn the model sees), and any conformance fixture asserting exact-string equality will all diverge from the intended behaviour.
