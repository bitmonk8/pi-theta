## Problem

The `parameters`-derivation in §Per-loom registration states a normative MUST that hard-names AJV ("the adapter MUST AJV-validate `params` against the original lowered JSON Schema") instead of citing the validator-neutral seam every other validation site in loom routes through ([PIC-11 `SchemaValidator`](./host-interfaces-services.md#pic-11)). The page never says where AJV is provisioned, whether the `Type.Unsafe<unknown>(loweredJsonSchema)` wrap yields an AJV-consumable document at the pinned JSON Schema 2020-12 draft, or whether the adapter may route through `SchemaValidator`. A second-source implementer cannot tell whether AJV is contractual — and must therefore appear in their dependency surface — or whether they may use the seam like every neighbouring validation site does.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `cb43deb` (2026-05-04, "pi-loom spec: resolve \"ToolDefinition is mis-shaped: missing label, and parameters must be a TypeBox TSchema\"")
**History:** Before `cb43deb` the `parameters` field was specified as `<params-schema-as-typebox>` and the `execute` adapter carried no AJV obligation. `cb43deb` reshaped the field derivation to wrap the lowered JSON Schema via `Type.Unsafe<unknown>(loweredJsonSchema)` and, in the same edit, added the sentence "the adapter MUST AJV-validate `params` against the original lowered JSON Schema before forwarding — Pi's structural typing offers no guarantees here." The `Type.Unsafe<unknown>` wrap was the load-bearing reason: it erases the typed surface inside `execute`, leaving `params: unknown`, and a runtime validation step had to be reintroduced. The sentence was added to the field-derivation bullet rather than via the established `SchemaValidator` seam (which already owned every other validation site), and it has carried AJV by brand ever since. The May 5 follow-up `ed86d46` (plan-side) confirmed the shape of the obligation but did not relocate or seam-route it; the later split commit `f5e89f4` (2026-06-04) relocated the paragraph from `pi-integration-contract.md` into the `extension-bootstrap-and-per-loom.md` child page verbatim. The defect is therefore the under-specified phrasing introduced in `cb43deb`, not a regression from a later edit.

## Solution approach

Rewrite the `parameters`-derivation clause in `pi-integration-contract/extension-bootstrap-and-per-loom.md` §Per-loom registration so the adapter-input-validation obligation is validator-neutral and routed through the injected [`SchemaValidator`](./host-interfaces-services.md#pic-11) seam, and state that the lowered JSON Schema artefact from [Schema Subset — Lowering Algorithm](../schema-subset.md#lowering-algorithm) is consumable by that seam without transformation at the pinned JSON Schema 2020-12 draft. Apply the same seam-routing rewrite to the symmetric `execute` clause for the synthesised `__loom_respond_<slug>` tool in [Implementation Notes — Runtime](../implementation-notes.md#runtime). Leave AJV named only at its existing non-normative site under [Implementation Notes](../implementation-notes.md#runtime).

## Solution constraints

- Out of scope: the binder's post-default-merge AJV validation ([Defaulting](../binder/defaulting-system-note-echo.md#post-default-merge-ajv-validation)), the binder envelope `args` validation ([Failure-class taxonomy](../binder/determinism-cancellation-failure.md#failure-class-taxonomy)), and the single-string-bypass safety net ([Binder — Single-string bypass](../binder/binder-bypass-and-envelope.md)) — these are separate validation sites and keep their current wording.
- Do not add dependency-enumeration prose to the bootstrap page; AJV provisioning stays enumerated only at [Loom-package implementation dependencies](../implementation-notes.md#loom-package-implementation-dependencies-loom-1-0).
