## Finding

`H5a`'s **Adds** describes the gate's warn-only live-corpus mode as one "that runs the same three live-corpus reconciliation surfaces — the live spec REQ-ID set and the live numbered-REQ-ID citing-test surface this leaf owns, and the live `spec_topics/**` normative-MUST surface the `H5e` arm contributes." `H5a` depends only on `H1a`, while `H5e` depends on `H5a` (`H5e` is strictly downstream). So at `H5a` build time the third surface — the un-anchored normative-MUST scan — does not yet exist, and the present-tense "runs the same three … surfaces" reads as if `H5a` itself exercises all three.

`H5a` already attributes the third surface to `H5e` ("the `H5e` arm contributes") and notes the `H5b` canary consumes the mode, so the ownership is stated. What is not pinned is the temporal fact: the complete three-surface warn-only run is not exercisable until `H5e` lands, and it is first exercised by `H5b` (whose **Deps** name both `H5a` and `H5e`). The prose therefore forward-references a capability that `H5a` only partially realises — it builds the warn-only mode shell plus its own two surfaces, not the third.

There is no executability break and no dependency edge is wrong: `H5a` must not depend on the downstream `H5e`, and `H5b` already gathers both. The defect is purely that the prose does not signal the deferred-contribution / deferred-exercise ordering, leaving a reader to reconcile a present-tense three-surface claim against `H5a`'s `H1a`-only **Deps**.

That extraction is what created the forward reference from `H5a` (then Deps `H4a`, later `H1a` at `352cd13`) to the strictly-downstream `H5e`. `git log -S "warn-only live-corpus mode"` and `git log -S "arm contributes"` on `H5a`, plus `git log --diff-filter=A` on `H5e`, corroborate this.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H5a-closing-gate-automation.md`'s **Adds** field, annotate the warn-only-mode sentence to record that the third surface is contributed by the downstream `H5e` arm and that the complete three-surface warn-only run is first exercisable once `H5e` lands — first exercised by `H5b`, whose **Deps** name both `H5a` and `H5e`. For example, extend the existing clause to read along the lines of: "… and the live `spec_topics/**` normative-MUST surface the `H5e` arm contributes (not present at `H5a` build time, since `H5a` precedes `H5e`; `H5a` establishes the mode shell and its own two surfaces, and the full three-surface warn-only run is first exercised by `H5b`, which depends on both `H5a` and `H5e`)."

Do not add a `Deps` edge from `H5a` to `H5e` — `H5a` is upstream of `H5e` and `H5b` already gathers both. This change is confined to `H5a` prose; no `Tests` or `Ships when` edit is required.

## Consequence

**Severity:** cosmetic

The wording is accurate but temporally flat: an implementer reading `H5a` against its `H1a`-only **Deps** may momentarily question how the warn-only mode "runs" a surface the not-yet-built `H5e` contributes. The `Tests`/`Ships when` for `H5a` already scope the work to the seeded fixtures and the two surfaces it owns, so no implementer produces a wrong leaf; the gap is clarity only.
