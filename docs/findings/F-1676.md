## Finding

The **Persistent diagnostics (default)** paragraph at the top of `docs/spec_topics/diagnostics/diagnostic-shape.md` names the universe of codes that flow through the `loom-system-note` `pi.sendMessage` channel as "All `loom/parse/*`, `loom/load/*`, and `loom/runtime/*` diagnostics" — and then immediately carves out **five** `console.error` exceptions, four of which sit in a namespace (`loom/host/*`) the superset never mentions: `loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, and `loom/host/session-shutdown-runtime-degraded`. As written, the only normative statement about how a `loom/host/*` code is delivered is "carved-out exception to a superset that does not contain it" — which is internally incoherent.

The reader who needs to route a `loom/host/*` row is left with two plausible readings. (a) `loom/host/*` is implicitly inside the persistent channel and the four named codes are the carve-outs (the obvious reading given the "carved-out exceptions" framing, but unsupported by the prose). (b) `loom/host/*` is wholly outside the persistent channel and the carve-outs paragraph is just naming the four `console.error` rows it happens to know about — in which case a future, currently-unnamed `loom/host/*` code (e.g. one observed mid-scan rather than mid-teardown) has no defined channel at all. The `loom/host/*` namespace bullet later in the same file says "Emitted via `console.error` since the typical observation site is mid-teardown when the persistent-diagnostic channel may already be invalidated" — "typical" leaves the non-typical case unresolved and does not retroactively repair the superset enumeration.

The same incoherence has now been carried forward through four separate edits that added `loom/host/*` rows to the carve-out list one at a time (the namespace itself was introduced in May 2026, the row count has since grown from one to five) without ever revisiting the superset enumeration. Each new row makes the mismatch worse: the carve-out list now contains more `loom/host/*` rows than the only `loom/runtime/*` row it formally carves out.

## Issue introduction

**Verdict:** present-since-namespace-inception, repeatedly compounded

**Introducing commits:**

- `9559e19` (2026-05-07) — "pi-loom spec: resolve 'session_shutdown unknown reason: no closed-set validation, no fail-safe'". Coined the `loom/host/*` namespace and added `loom/host/session-shutdown-reason-unknown` to the carve-out list (raising the count from one to two), but left the superset enumeration unchanged as `loom/parse/*`, `loom/load/*`, `loom/runtime/*`. The mismatch was born here.

**History:**

- `5e8b499` (2026-05-06) — established the `Persistent diagnostics (default)` paragraph with a single carved-out exception (`loom/runtime/reload-teardown-timeout`), still coherent because the carve-out namespace was a subset of the superset.
- `9559e19` (2026-05-07) — coined `loom/host/*` and added the first `loom/host/*` carve-out without amending the superset; mismatch introduced.
- `75a9626` (2026-05-09) — added two more `loom/host/*` carve-outs (`session-shutdown-pinned-constant-unreadable`, `session-shutdown-teardown-step-failed`), bumping the wording to "four carved-out exceptions"; superset still unchanged.
- A later edit (visible in the current prose) added `loom/host/session-shutdown-runtime-degraded` as the fifth carve-out; superset still unchanged.
- `b2f780b` (2026-06-24) — `git log --follow` shows the most recent edit to the paragraph (the located-site classification rewrite via F-1152) did not touch the superset enumeration.

The mismatch has been load-bearing for ~7 weeks across four edits, each of which added a `loom/host/*` row to the carve-out list without revisiting the superset; it has never been the subject of a targeted fix.

## Solution Space

**Shape:** single

### Recommendation

Add `loom/host/*` to the superset enumeration in the **Persistent diagnostics (default)** paragraph and reframe the carve-outs as the per-code exceptions they actually are. Concretely, rewrite the opening clause as: "All `loom/parse/*`, `loom/load/*`, `loom/runtime/*`, and `loom/host/*` diagnostics are delivered via the channel below, with five carved-out exceptions — all emitted via `console.error` from the `session_shutdown` handler …" and leave the existing five-row enumeration unchanged (the four `loom/host/session-shutdown-*` rows plus `loom/runtime/reload-teardown-timeout`).

Two consequential clarifications follow from the rewrite and MUST land in the same edit:

1. The `loom/host/*` namespace bullet under **Code namespaces** currently says "Emitted via `console.error` since the typical observation site is mid-teardown when the persistent-diagnostic channel may already be invalidated." The word "typical" becomes the only remaining wedge for the no-default-channel reading; replace it with prose that pins the same default as the rest of the superset (persistent channel) and notes that the four enumerated teardown-handler rows are the only `loom/host/*` carve-outs in loom 1.0 — additions are governed by the [GOV-15 diagnostic-registry carve-out](../governance/source-language-stability.md#diagnostic-registry-carve-out) which already requires per-row channel disclosure at registry-row-add time.
2. The `loom/host/*` namespace bullet's "anomalies that surface during loom's interaction with the Pi host" definition stays unchanged — the rewrite is about *channel*, not about *namespace scope*.

Implementer-relevant edge cases:

- The `code-registry-host.md` rows' per-*Trigger* `console.error` clauses remain the source of truth for per-code routing on the four enumerated rows; the superset paragraph asserts only the default. Do not delete those per-row clauses — they survive a future row addition that the GOV-15 carve-out admits without immediately re-touching the superset paragraph.
- The companion paragraph immediately after the rewrite ("Each of these five teardown-handler `console.error` calls is itself wrapped per Pi Integration Contract — Diagnostic-emission isolation …") already says "these five" and needs no edit beyond verifying the count word remains in lockstep with the carve-out enumeration.
- The hub `docs/spec_topics/diagnostics.md` intro currently uses the `loom/parse/*` / `loom/load/*` / `loom/runtime/*` triplet in its `LoadExtensionsResult.errors`-exclusion sentence; that sentence is about Pi's loader sink, not about the persistent channel, and does not need to be amended by this finding. (A separate related finding — "`LoadExtensionsResult.errors` exclusion stated in the hub intro, not the delivery-channel contract" — proposes relocating that sentence; if both fixes land, the relocated copy MUST also include `loom/host/*`.)

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on the delivery channel for any `loom/host/*` code, and a future `loom/host/*` code added outside the teardown handler will have no normatively-named channel at all. Operator-tooling consumers parsing the four enumerated rows from `console.error` will accept the current channel mix, but downstream conformance tests that key on "all `loom/host/*` rows route via `console.error`" cannot cite the superset paragraph as their normative source — the row-level *Trigger* prose is the only anchor — which weakens the audit-gate story.
