## Problem

The **Persistent diagnostics (default)** paragraph names the universe of codes delivered via the persistent `loom-system-note` channel as "All `loom/parse/*`, `loom/load/*`, and `loom/runtime/*` diagnostics", then carves out five `console.error` exceptions — four of which are `loom/host/*` codes from a namespace the superset never lists. The only normative statement about how a `loom/host/*` code is delivered is therefore "carved-out exception to a superset that does not contain it", which is internally incoherent. A reader routing a `loom/host/*` row cannot tell whether the namespace is implicitly inside the persistent channel or wholly outside it, and a future non-teardown `loom/host/*` code has no normatively-named channel at all. The `loom/host/*` namespace bullet's "typical observation site is mid-teardown" wording leaves the non-typical case unresolved and does not repair the superset enumeration.

## Issue introduction

**Verdict:** present-since-namespace-inception, repeatedly compounded

**Introducing commits:**

- `9559e19` (2026-05-07) — "pi-loom spec: resolve 'session_shutdown unknown reason: no closed-set validation, no fail-safe'". Coined the `loom/host/*` namespace and added `loom/host/session-shutdown-reason-unknown` to the carve-out list (raising the count from one to two), but left the superset enumeration unchanged as `loom/parse/*`, `loom/load/*`, `loom/runtime/*`. The mismatch was born here.

**History:**

- `5e8b499` (2026-05-06) — established the `Persistent diagnostics (default)` paragraph with a single carved-out exception (`loom/runtime/reload-teardown-timeout`), still coherent because the carve-out namespace was a subset of the superset.
- `9559e19` (2026-05-07) — coined `loom/host/*` and added the first `loom/host/*` carve-out without amending the superset; mismatch introduced.
- `75a9626` (2026-05-09) — added two more `loom/host/*` carve-outs (`session-shutdown-pinned-constant-unreadable`, `session-shutdown-teardown-step-failed`), bumping the wording to "four carved-out exceptions"; superset still unchanged.
- A later edit (visible in the current prose) added `loom/host/session-shutdown-runtime-degraded` as the fifth carve-out; superset still unchanged.
- `b2f780b` (2026-06-24) — `git log --follow` shows the most recent edit to the paragraph (the located-site classification rewrite via F-1152) did not touch the superset enumeration.

The mismatch has been load-bearing for ~7 weeks across four edits, each of which added a `loom/host/*` row to the carve-out list without revisiting the superset; it has never been the subject of a targeted fix.

## Solution approach

Add `loom/host/*` to the superset enumeration in the **Persistent diagnostics (default)** paragraph so the five `console.error` rows read as the per-code exceptions they are. Clarify the `loom/host/*` bullet under **Code namespaces** to pin the persistent channel as the default delivery sink for the namespace, replacing the "typical observation site" wording that currently leaves the non-teardown case unresolved. Reference the [GOV-15 diagnostic-registry carve-out](../governance/source-language-stability.md#diagnostic-registry-carve-out) as the mechanism governing future `loom/host/*` row additions and their channel disclosure.

## Solution constraints

- Out of scope: the per-row `console.error` *Trigger* clauses in `code-registry-host.md` — they remain the source of truth for per-code routing and MUST NOT be deleted when the carve-outs are reframed.
- Out of scope: the hub `docs/spec_topics/diagnostics.md` intro's `LoadExtensionsResult.errors`-exclusion sentence — it concerns Pi's loader sink, not the persistent channel, and is not amended by this finding.
