## Finding

`V15a` (Invocation core) owns the *fresh-vs-attach cross-mode matrix selection* and its **Ships when** asserts that `npm test` "resolves and spawns an invoke across the fresh-vs-attach cross-mode matrix". Spawning a real invoke across that matrix routes a subagent-mode callee onto a freshly spawned `AgentSession` and a prompt-mode callee onto the caller's conversation. Neither mechanism is owned by `V15a`: the subagent `AgentSession` spawn sequence (`createAgentSession`, `ResourceLoader` adapter, `SessionManager.inMemory`, abort-listener attach) is owned by `V9i`, and the prompt-mode conversation drive (`sendUserMessage`/`waitForIdle`, active-set gating, attach to the caller's conversation) is owned by `V9c`.

`V15a`'s **Deps** are `V15a-T`, `V10a`, `V2b`, `V3d`, `V8b`, `V4f` — none of which provides the spawn-and-attach machinery or the end-to-end harness (`H4a`/`H4b`) needed to drive a live session. The paired tests task `V15a-T` is thinner still (`V10a`, `V2b`, `V3d`, `V8b`), so the test that would witness the spawn cannot be authored against any declared collaborator. Sibling leaves that genuinely need session/harness machinery declare it — `V17a` lists `H4a`/`H4b`; `V15d` (the prompt→prompt facet peeled off `V15a`) lists `V9f`. `V15a`'s own **Adds** scopes the leaf to cross-mode matrix *selection*, so the **Ships when** overclaims relative to the leaf's stated scope.

The pickability predicate is satisfied by `V15a`'s declared Deps, so a contributor can pick `V15a` and then discover the **Ships when** is unsatisfiable: there is no session spawn or conversation-drive surface to exercise. The implementer either invents a stand-in spawn/attach mechanism (which may diverge from the real `V9i`/`V9c` surfaces) or stalls.

## Solution Space

**Shape:** multiple

### Option A — Wire the spawn-and-attach seams into V15a's Deps

**Approach:** Make the two cross-mode routing targets available to `V15a` by declaring the leaves that own them, plus the harness that drives a live session.
**Plan edits:** Append `V9c`, `V9i`, `H4a`, `H4b` to `V15a`'s **Deps**; append `V9c`, `V9i`, `H4a`, `H4b` to `V15a-T`'s **Deps**. Leave the **Ships when** unchanged (it then becomes literally satisfiable).
**Spec edits:** None.
**Pros:** The **Ships when** is met exactly as written; the cross-mode matrix is exercised end-to-end against the real subagent and prompt-mode surfaces.
**Cons:** Large dependency fan-in pushes `V15a` late in the build order (after most of V9), even though `V15a` is conceptually the invocation core that subagent/prompt-mode routing dispatch *into*.
**Risks:** `V9i` and `V9c` do not currently depend on `V15a`; before adding these edges confirm no latent cycle is created if a later edit gives `V9i`/`V9c` a `V15a` dependency.

### Option B — Narrow V15a's Ships-when to the matrix-selection decision

**Approach:** Restate the **Ships when** so `V15a` asserts only that the cross-mode matrix *selects* fresh-vs-attach by callee mode (the routing decision its **Adds** already scopes it to), not that it spawns a live invoke across both modes. The live spawn-and-attach is witnessed by the leaves that own the session machinery (`V9i` for the subagent path; `V15d`/`V9c`-driven for the prompt path).
**Plan edits:** Rewrite the `V15a` **Ships when** clause "resolves and spawns an invoke across the fresh-vs-attach cross-mode matrix" to assert the matrix selects fresh-vs-attach by callee mode; mirror the change in `V15a-T`. If the narrowed test drives session doubles, add `H4a`/`H4b` to `V15a` and `V15a-T` **Deps**; if it is a pure selection-logic test, no harness dep is needed.
**Spec edits:** None.
**Pros:** Keeps `V15a` as the pure invocation-core routing leaf with a small dependency surface; it stays pickable early. Aligns **Ships when** with the leaf's own **Adds** scope.
**Cons:** The live cross-mode spawn is no longer witnessed at `V15a`; coverage of the live prompt-mode attach path must be confirmed to live on a downstream leaf.
**Risks:** If no downstream leaf witnesses the live prompt-mode attach (caller's-conversation routing) end-to-end, narrowing leaves a coverage gap.

### Recommendation

Take Option B: narrow the **Ships when** to the matrix-selection decision, since `V15a`'s **Adds** already scopes the leaf to "cross-mode matrix selection" and the live session machinery is deliberately owned elsewhere (`V9i`, `V9c`, `V15d`). Edge case the implementer must confirm: that the live prompt-mode attach path is witnessed by an existing downstream leaf; if none witnesses it, record that as a separate coverage gap rather than re-absorbing the spawn obligation into `V15a`.

## Consequence

**Severity:** correctness

A contributor can pick `V15a` (its declared Deps are satisfiable without the session machinery) but cannot meet the **Ships when** as written — there is no spawn-and-attach surface to drive. Two reasonable implementers diverge: one invents a session/attach stand-in that may not match the real `V9i`/`V9c` surfaces, another defers the matrix witness; either way the cross-mode routing, a core invocation behaviour, is verified inconsistently or not at all.
