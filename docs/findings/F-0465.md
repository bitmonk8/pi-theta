## Finding

The manual real-host smoke run is scored against a five-part pass criterion (a)–(e). Four of the five parts state an observable: (a) "the run completes without error — no thrown exception, no aborted pipeline"; (b) "the binder pass produces output that validates against the binder envelope schema"; (c) "cancellation propagation is observed at the injected cancel point"; (e) "the emitted `loom-system-note` codes are a **subset** of the **committed permitted-code list**". Criterion (d) breaks the pattern: it lists the four session-double fidelity-contract axes — streamed-token order relative to `ctx.waitForIdle()` resolution, single-turn prompt-mode append, the `pi.on` cancel-forward subscription, and cancellation propagation — and requires only that they "hold", with no per-axis observable that a contributor running the smoke (or the `H6a` auditor later reading the recorded run) can check.

The observable form already exists adjacent to this criterion. `H4a`'s session-double fidelity-contract self-check (`docs/plan_topics/H4a-factory-shell-and-harness.md`) states the same four axes as observables: "(i) streamed assistant tokens observable in the transcript before `ctx.waitForIdle()` resolves, (ii) one streamed assistant response appended as a single prompt-mode turn, (iii) the `pi.on` cancel-forward subscription, and (iv) cancellation propagation." Criterion (d) can be expressed in those terms; the bare verb "hold" is the only gap.

(Note: axis (iv) of criterion (d) — cancellation propagation — restates criterion (c). That overlap is tracked separately under the criterion overclaim finding and is an edge case the fixer should de-duplicate while rewording, not a second defect this finding owns.)

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/real-host-smoke-gate.md`, replace criterion (d)'s trailing verb "— hold" with a per-axis observable pass condition for each of the four axes, sourced from the observable phrasing already in `H4a`'s session-double fidelity-contract self-check. Concretely, the four axes should each carry an observable a smoke runner and the `H6a` auditor can confirm:

- streamed assistant tokens are observable in the transcript before `ctx.waitForIdle()` resolves;
- each streamed assistant response appends as exactly one prompt-mode turn;
- the `pi.on` cancel-forward subscription fires;
- cancellation propagation is observed.

Edge cases the implementer must watch: the fourth axis duplicates criterion (c) (cancellation propagation observed at the injected cancel point) — collapse the two so the criterion does not require the same observable twice. Keep the observable wording aligned with `H4a` line-12 so the smoke criterion and the harness self-check do not drift into two different statements of the same four axes.

## Consequence

**Severity:** correctness

Two reasonable contributors running the smoke, and the `H6a` auditor reviewing the recorded run, have no shared test for whether criterion (d) passed — so the same run can be scored pass by one and fail by another. The smoke gate's falsifiability claim ("a genuine divergence still produces a falsifiable fail") is unenforceable for exactly the four fidelity axes the gate exists to protect.
