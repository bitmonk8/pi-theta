## Finding

`V9b`'s `Adds.` states the leaf "itself wires the registry-swap arm (`loom/runtime/registry-swap-failed`) and the `.loom`/`.warp` re-parse arm" of the watcher-time reload failure-injection seam. But `V9b`'s and `V9b-T`'s `Tests.` only exercise the registry-swap arm (the registry-swap bullet asserts "a failed swap fires `loom/runtime/registry-swap-failed`"); neither leaf has a bullet that injects a synthetic `.loom`/`.warp` re-parse failure and asserts its routing onto the `loom-system-note` surfacing path. `V9b`'s `Ships when` likewise names only `registry-swap-failed`, not the re-parse arm.

The re-parse arm's only gate lives in another slice: `V4e`'s `ERR-7` bullet, which the coverage matrix maps as the sole closer of `ERR-7`. `ERR-7` additionally bundles re-parse with the settings-re-merge arm — its wording exercises "both the re-parse/re-merge diagnostic arm and the `loom/runtime/registry-swap-failed` registry-swap arm" — so even at `V4e` the re-parse arm is not witnessed distinctly from re-merge.

The consequence is that `V9b` can be tagged `V9b-complete` against a green suite while its re-parse arm wiring is broken or absent: nothing in `V9b`/`V9b-T` observes it, and the breakage surfaces only later at `V4e` in the V4 slice, decoupled from the leaf that introduced the code. This violates the "smallest feature that can ship and be tested independently" principle for `V9b`.

**History:** At inception (c6a664e, 2026-06-10) `V9b` carried no failure-injection seam and no re-parse arm — only the registration steps and drain-state contract. Commit b2d5a1f (2026-06-11) added the watcher-time reload failure-injection seam and the `.loom`/`.warp` re-parse arm to `V9b`'s `Adds.` while placing the gating obligation in `V4e`'s `ERR-7` bullet rather than adding a paired re-parse test to `V9b`/`V9b-T`; the gap has existed since. Commit d64dce5 (same day) refined the seam to the named `ReloadFailureInjector` interface but preserved the gate-at-`V4e`-only structure.

## Solution Space

**Shape:** single

### Recommendation

Give the re-parse arm a distinct gate inside the leaf that wires it.

- In `docs/plan_topics/V9b-T-registration-reload-wiring.md` `Tests.`, add a failing-test bullet that supplies a synthetic `.loom`/`.warp` re-parse failure to `ReloadFailureInjector.injectReloadFailure` and asserts the seam routes it onto the `loom-system-note` surfacing path without standing up a live watcher — distinct from the registry-swap bullet and from the settings-re-merge arm.
- In `docs/plan_topics/V9b-registration-reload-wiring.md` `Tests.`, mirror the same re-parse-arm bullet (the paired impl leaf), and extend the `Ships when` sentence so its gate names the re-parse arm's routing to `loom-system-note` alongside the existing `registry-swap-failed` clause.
- In `docs/plan_topics/V4e-pre-evaluation-failures.md` (and its `-T`) `ERR-7` bullet, separate the re-parse assertion from the re-merge assertion so `ERR-7` witnesses the re-parse arm distinctly rather than as a bundled "re-parse/re-merge diagnostic arm".

Edge cases for the implementer: the re-parse test's arm discriminator depends on how `injectReloadFailure`'s parameter distinguishes arms — see the related signature finding, which must resolve first so the re-parse case can target the arm unambiguously. `V9b-T` adds no new `Deps` for this (the `ReloadFailureInjector` interface is declared in its own paired `V9b` scope).

## Consequence

**Severity:** advisory

`V9b` can reach its `V9b-complete` tag with a green suite while the re-parse arm it wires is broken, because no `V9b`/`V9b-T` test observes that arm; the defect would only surface at `V4e`'s `ERR-7` in a later slice, decoupled from its cause. Because `ERR-7` also bundles re-parse with re-merge, the re-parse arm is never witnessed in isolation anywhere.
