## Finding

The `ActiveInvocationRegistry` *Registry contract* closes with: "The registry name is internal. It is not part of the loom-facing or extension-facing contract; tests assert on observable side effects (entry counts via probe seams, ordered `loomAbort.abort()` calls, `disposeBarrier` settlement) rather than on the symbol itself." Two of the three named side effects are observable through surfaces the corpus already defines тАФ `loomAbort.abort()` calls through the test-injected `AbortController` instances, and `disposeBarrier` settlement through the per-invocation promise тАФ but the first, **entry counts via probe seams**, names a "probe seam" that is defined nowhere. The phrase "probe seam" appears only at this site; no PIC-numbered seam, no member, and no test-construction accessor exposes the registry's live entry count.

This is the only one of the three observables that is otherwise inaccessible. The registry is an internal `Set` that is explicitly "not exposed through `ExtensionAPI`" and is wired "through the same constructor-injection skeleton that owns `FileSystem`, `DiagnosticsSink`, `SchemaValidator`, etc." Every other test-only observation point on that skeleton is declared as a first-class seam with a behavioural contract in `host-interfaces-services.md` тАФ the `Checkpoint` seam (PIC-10), `TokenEstimator` (PIC-16), `IdSource` (PIC-20) тАФ each marked non-binding in shape per [GOV-18 arm (a)] but pinned in behaviour. The registry-count "probe seam" has no analogous declaration, so a conformance-test author cannot determine how to read the count: a steady-state assertion such as "two concurrent invocations produce two live entries" has no defined surface to read against, and different implementers would improvise different (or no) probe surfaces.

## Solution Space

**Shape:** single

### Recommendation

Declare the registry entry-count observation point as an internal test-only surface, on the same footing as the sibling internal seams it is wired alongside. In `host-interfaces-services.md`, add a short contract (modelled on PIC-10/PIC-20) stating that the runtime exposes the live `ActiveInvocationRegistry` entry count to test wiring through the same constructor-injection skeleton that owns the registry тАФ its member shape non-binding per GOV-18 arm (a), the behavioural guarantee being that the surface reports the current count of live entries (one per in-flight invocation, insertion-ordered consistent with the `loom/runtime/reload-teardown-timeout` `<list>`), and that the surface is internal: not reachable through `ExtensionAPI` or visible to loom authors, Pi extensions, or tools. Then change the registry-contract bullet's parenthetical from the undefined "entry counts via probe seams" to a cross-reference to that declared surface.

Implementer notes: keep the count observable a *read* surface only (it must not let tests mutate the set), and pin it as consistent with the insertion-order invariant the teardown sub-step-2/sub-step-3 iterations already rely on, so an entry-count assertion and a teardown-order assertion read the same ordering. If the project prefers not to add a new PIC entry, the equivalent acceptance is to state inline in the registry contract that the live count is observable through the runtime's construction-time handle to the registry (the DI skeleton the test harness already constructs) тАФ either anchor resolves the finding; what must not remain is an "entry counts via probe seams" reference with no defined surface anywhere in the corpus.

## Consequence

**Severity:** advisory

A working runtime can be built without the missing declaration, but the conformance-test contract for registry-teardown correctness is incomplete: the registry's entry count тАФ the observable that distinguishes "all in-flight invocations were tracked" from "an insertion site leaked or dropped an entry" тАФ has no defined read surface, so two reasonable implementers write divergent (or absent) probe surfaces and the entry-count assertions intended by this bullet are not portably testable.
