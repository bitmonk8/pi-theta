## Problem

`V15e` canonicalizes the watcher-delivered changed-file path through `realpath`-then-forward-slash at eviction time, and every non-canonical form it enumerates (symlink, case-variant, `.`/`..`, relative) is a transformation of a path that still exists on disk. But the watcher seam `V9b` delivers `unlink` (deletion) events, and `FileSystem.realpath` rejects with `ENOENT` when a path component is missing, so a deleted file cannot be `realpath`-resolved at eviction time. The leaf never defines what eviction does in this case: neither whether the `ENOENT` aborts (and fails the `LoomRegistry` swap), nor whether the deleted file's own parse-cache entry and the entries of every file that transitively imports it across the `.warp` import-edge graph are dropped at all. Those importers must be re-parsed after a delete so the next load pass surfaces the missing-import diagnostic rather than serving a stale parse.

## Issue introduction

**Verdict:** introduced-by-specific-edit
**Introducing commit:** `b63e915` (2026-06-17) — "pi-loom plan: resolve \"hot-reload eviction pins one canonical path identity\""
**History:** The plan corpus is git-tracked. The `V15e` leaf was created in `a3dd2f0` (2026-06-16) with a bare eviction step that did not canonicalize the watcher-delivered path, so no existence presupposition was present then. `b63e915` added the `realpath`-then-forward-slash canonicalization together with the enumerated non-canonical-form list (symlink, case-variant, `.`/`..`, relative); that enumeration covers only transformations of an extant path and never the `unlink`/deletion case, so the existence-presupposition defect entered with the canonicalization itself. Later edits `f541bc3` (2026-06-17, cache renaming) and `c0a8a04` (2026-06-17, stale-serve coverage) did not touch the deletion gap, which has persisted on the leaf since `b63e915`.

## Solution approach

Define `unlink`/deletion behaviour for the `V15e` eviction step. Because a deleted file cannot be `realpath`-resolved at eviction time, match the deleted file against its already-canonical stored parse-cache key (`V15a`) without re-canonicalizing, reconciling host case-sensitivity per [invocation.md — Resolution], then run the same transitive `.warp` importer walk (`V15c`) over the post-swap import-edge graph to drop the deleted file's entry and its transitive importers' entries. Add matching coverage to the `V15e`/`V15e-T` Tests and Ships-when so deleting an imported `.warp` file drops both the deleted file's entry and its transitive importers' entries.

## Solution constraints

- Out of scope: the spec is read-only — this is a `V15e` plan-leaf gap, not a spec gap (invocation.md already treats deletion as a legitimate `load_failure` cause).
- Reuse the canonical path form and host case-sensitivity rule from [invocation.md — Resolution] rather than restating them in the leaf, per the leaf's existing reuse convention.
