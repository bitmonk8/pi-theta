## Finding

The closing sentence of the **Deps.** bullet in `conventions.md` §Leaf format re-enumerates the three transitive-completeness triggers verbatim: "A new leaf whose `Tests.` bullets carry an executable REQ-ID, a citing test that *closes* a coverage-matrix-mapped numbered REQ-ID, or an un-anchored normative MUST is additionally added to `H5b`'s `Deps.` per the *Transitive-completeness plan-maintenance* clause in §REQ-ID discipline below, which owns the exact triggers."

That same three-condition trigger list is stated authoritatively further down the *same page* in the §REQ-ID discipline **Transitive-completeness plan-maintenance** clause, and the `Deps.` sentence explicitly defers to it ("which owns the exact triggers"). The enumeration is therefore stated twice on one page while declaring a single owner, so the `Deps.` copy is a redundant second source. If the authoritative trigger list is ever refined, the `Deps.` restatement must be hand-updated in lockstep with no mechanical guard, making it a divergence surface. The drift is not hypothetical: a sibling finding records that this enumeration uses the actuality phrasing ("`Tests.` bullets **carry**") while the plan.md §How-to-use step-2 copy of the same list uses the capability phrasing ("**can** introduce"), demonstrating the exact wording drift restated triggers invite.

Note the `Deps.`-field mention exists deliberately to keep the maintenance obligation visible to an author working at the `Deps.` field (the reason it was reintroduced — see Issue introduction). The fix must preserve that visibility: collapse the enumeration to a forward-pointer, not delete the reference outright.

## Issue introduction

**Verdict:** introduced-by-specific-change
**Introducing commit:** `ab8e297` (2026-06-11) — *pi-loom plan: resolve "Transitive-completeness obligation invisible at leaf authoring"*
**History:** The authoritative *Transitive-completeness plan-maintenance* clause was relocated into §REQ-ID discipline by `37733fd` (2026-06-11, *"H6a transitive-completeness rule parked in Deps, not conventions"*), which moved the rule out of the `Deps.` field and into the conventions section. `ab8e297`, later the same day, then re-added a mention at the `Deps.` field to resolve the obligation being invisible to an author reading only the `Deps.` field — but it did so by restating the full three-condition trigger list rather than emitting a bare pointer. `git log -S "which owns the exact triggers"` and `git log -S "is additionally added to"` over `docs/plan_topics/conventions.md` both resolve to `ab8e297` as the sole introducing commit; the duplication has existed continuously since. The fix therefore must retain the authoring-time visibility `ab8e297` added while removing the restatement it introduced.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md` §Leaf format, **Deps.** bullet, replace the enumerated trigger list in the final sentence with a bare forward-pointer that names neither the three conditions nor the `H5b` action in full, while still directing an author at the `Deps.` field to the owning clause. Strike the enumeration "A new leaf whose `Tests.` bullets carry an executable REQ-ID, a citing test that *closes* a coverage-matrix-mapped numbered REQ-ID, or an un-anchored normative MUST is additionally added to `H5b`'s `Deps.`" and leave only the pointer to §REQ-ID discipline (*Transitive-completeness plan-maintenance*) as the home that states the triggers and the `H5b` obligation. Keep the pointer present (do not delete the reference entirely): it preserves the leaf-authoring visibility that `ab8e297` added. After the edit, §REQ-ID discipline is the only place on the page that enumerates the triggers.

## Consequence

**Severity:** cosmetic

Two copies of the trigger list on one page must be kept in sync by hand; a future refinement of the authoritative §REQ-ID discipline triggers can leave the `Deps.` restatement stale and contradictory. No implementer produces a wrong leaf today — both copies currently agree — so the impact is a plan-maintenance divergence hazard, not a build-order or coverage defect.
