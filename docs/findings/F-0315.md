## Finding

`conventions.md` names the test toolchain as load-bearing fact in two places: the §Per-phase TDD ritual depends on the TypeScript type-check / compile step (the "compiles and runs" gate, the "compile-time red" definition, and "type-check clean" in the Run step), and the §Cross-cutting rules "No silent test skipping" bullet names `Vitest`'s `assert.fail` / `expect.unreachable`. The whole ritual further presumes a runnable test command (`npm test` / `npm run typecheck`) exists. None of these references points to the leaf that actually provisions and pins that toolchain — `H1a`, which declares `vitest` and `typescript` in `devDependencies`, defines the `npm test` / `npm run typecheck` scripts, and owns the `tsc` floor.

The page already follows the opposite convention for every mechanical gate it names: the no-globals architectural test and ambient-primitive scan cross-link `H2a` / `H3a`, the closing-gate checks cross-link `H5a` / `H5c`, and the transitive-completeness arm cross-links `H5d`. The toolchain references are the sole load-bearing infrastructure the page names without the parallel owning-leaf link. Confirmed against the file: `H1a` does not appear anywhere in `conventions.md`, while `H2a`/`H3a`/`H5a`/`H5c`/`H5d` each appear at their gate-naming sites.

A reader of `conventions.md` following the page's own cross-link discipline can navigate from each named gate to the leaf that establishes it, but cannot do so for the test runner, the assertion API, or the compile step the TDD ritual rests on.

The defect is therefore an asymmetry that accreted as the toolchain references were added post-inception, not a present-since-inception gap.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, add a cross-link to `H1a` ([`H1a`](./H1a-scaffold-and-toolchain.md)) at each point the page names the test toolchain, mirroring the existing owning-leaf cross-links for the mechanical gates:

- In the §Cross-cutting rules "No silent test skipping" bullet, where it names `Vitest`'s `assert.fail` / `expect.unreachable`, attribute the Vitest runner and assertion API to `H1a` (the leaf that declares `vitest` in `devDependencies` and defines `npm test`).
- In the §Per-phase TDD ritual, where it gates on the type-check / compile step ("compiles and runs", "compile-time red", and the "type-check clean" Run-step item), attribute the `tsc` type-check / `npm run typecheck` toolchain to `H1a` (the leaf that pins the `typescript` floor and defines the `npm run typecheck` / `npm test` scripts).

The added links target `H1a` only; do not edit `H1a` itself. The fix is confined to `conventions.md`.

## Consequence

**Severity:** advisory

A reader of `conventions.md` cannot navigate from the named test toolchain (Vitest assertion API, the type-check/compile step the TDD ritual gates on) to the leaf that guarantees it exists, breaking the owning-leaf cross-link convention the page applies to every mechanical gate. No implementer is blocked and no leaf content is wrong; the gap is one of navigational consistency in the foundational read-once conventions page.
