## Finding

`M-T`'s **Tests** section lists two bullets that both cite `SLSH-2` for the same happy-path run:

- bullet 1 — "a dispatched prompt-mode loom issues one untyped `@`-query and streams its **assistant response** into the user session as one appended turn";
- bullet 2 — "running the fixture loom through the harness produces exactly one appended turn and no diagnostic".

The two bullets overlap heavily: both assert the single-appended-turn outcome. Bullet 2's only non-redundant additions over bullet 1 are the explicit cardinality ("exactly one appended turn") and the cleanliness assertion ("no diagnostic").

`conventions.md` §Per-phase TDD ritual defaults to **one assertion per REQ-ID** so each rule reds out in isolation, and folds multiple rules into a single assertion only when they describe one indivisible behaviour. Multiple bullets for a single REQ-ID is permitted (e.g. `V12a` carries three distinct `SLSH-2` failure-path bullets), but in that precedent each bullet is a clearly distinct, independently-failing facet. `M-T`'s split carries no such annotation and the two bullets substantially restate one another, so a fresh implementer authoring the tests-task cannot tell whether the split encodes two independently-failing modes or is accidental duplication. The granularity is left ambiguous.

The paired `M` leaf carries a single `SLSH-2` Tests bullet, so the duplication is confined to `M-T`.

## Issue introduction

**Verdict:** introduced-by-specific-edit
**Introducing commits:** c06749d ("pi-loom plan: resolve \"Doc-updates convention unenforced / mislabeled Tests bullets\"", 2026-06-11)
**History:** At plan creation (`c6a664e`) `M-T`'s second Tests bullet was a `` `Convention:` (*Doc updates*) `` bullet — "running the fixture loom through the harness produces exactly one appended turn and no diagnostic" — distinct in citation from the first `SLSH-2` bullet, so no duplicate-`SLSH-2` situation existed. Commit `c06749d` relabeled that bullet's prefix from `` `Convention:` (*Doc updates*) `` to `` `SLSH-2` `` (its content was unchanged), which is what created the present pair of two bullets both citing `SLSH-2` with no stated justification for the split. The relabel was content-appropriate (the bullet asserts happy-path run behaviour, not a doc-updates obligation) but left the two SLSH-2 bullets unreconciled.

## Solution Space

**Shape:** multiple

### Option A — Fold the non-redundant content into the single SLSH-2 bullet

**Approach.** Merge bullet 2's distinct assertions into bullet 1 so `M-T` carries one `SLSH-2` Tests bullet covering the happy path, then delete bullet 2.

**Plan edits.** In `docs/plan_topics/M-T-minimal-slash-command.md` Tests, extend the surviving `SLSH-2` bullet to also assert "exactly one appended turn and no diagnostic" (the cardinality and no-diagnostic content from bullet 2), and remove the second `SLSH-2` bullet.

**Spec edits.** None.

**Pros.** Matches the §Per-phase TDD ritual one-assertion-per-REQ-ID default; removes the overlap outright; smallest change.

**Cons.** The "exactly one" cardinality and "no diagnostic" checks no longer red out in isolation from the streaming/append assertion.

**Risks.** Minimal.

### Option B — Keep the split but annotate the two bullets as independently-failing modes

**Approach.** Retain both `SLSH-2` bullets but make each one's distinct, independently-failing facet explicit (mirroring how `V12a` distinguishes its three SLSH-2 bullets), so the split reads as intentional.

**Plan edits.** In `docs/plan_topics/M-T-minimal-slash-command.md` Tests, reword the two `SLSH-2` bullets so bullet 1 owns the streaming-assistant-response-into-session facet and bullet 2 owns the distinct cardinality + no-diagnostic facet, removing the textual overlap on the single-appended-turn outcome.

**Spec edits.** None.

**Pros.** Preserves isolated red-out per facet; consistent with the `V12a` multi-bullet-per-REQ-ID precedent.

**Cons.** Requires sharpening the wording so the two facets do not restate each other; more edit surface than Option A.

**Risks.** If the rewording does not cleanly separate the facets, the ambiguity persists.

### Recommendation

Apply **Option A**. The two bullets substantially restate the same single-appended-turn outcome; the only genuinely new content in bullet 2 is "no diagnostic", which does not warrant its own isolated bullet at this minimal happy-path leaf. Fold "exactly one appended turn and no diagnostic" into the single `SLSH-2` bullet in `docs/plan_topics/M-T-minimal-slash-command.md` and delete the second bullet, leaving `M-T` with one `SLSH-2` Tests bullet consistent with the §Per-phase TDD ritual default. Leave `M`'s single `SLSH-2` bullet as-is.

## Consequence

**Severity:** advisory

Two implementers authoring `M-T` could diverge on test granularity — one writing a single combined assertion, another two — because the unannotated split gives no signal about whether the second bullet is an independently-failing mode. The leaf still produces working red tests either way, so this is a guidance gap rather than a correctness break.
