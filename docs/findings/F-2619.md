## Finding

`runtime-value-model.md` carries a bold-led paragraph, **Reference encoding (non-normative).**, sandwiched between the normative representation table and the **Equality (`==`)** paragraph. It states two concrete reference-implementation shapes: the enum tag realised as a non-enumerable `__loomEnum` string property, and `Result<T, E>` realised as `{ ok: true, value: T }` / `{ ok: false, error: E }`. These are implementation details — explicitly "neither reachable from loom code, neither appears in any wire schema, and either may change without a spec revision." That is precisely the shape of the non-normative **Implementation hint** bullets already collected on `implementation-notes.md` (e.g. the Chevrotain parser hint and the AJV-flags hint). Placing the concrete shapes inline on the value-model page, directly adjacent to genuinely normative material, invites a reader to treat them as part of the contract — the same over-prescription concern the page's normative rows were already rewritten to avoid (the rows now state only the observable contract: the tag MUST NOT appear in JSON output; the in-memory `Result` shape is not part of the language surface).

The paragraph also ends with an embedded future-author action item: "If a future host-interop surface ever exposes a live `Result` or enum value to JS (rather than a JSON-serialised projection), this section MUST be revisited before that surface ships." This is a conditional MUST addressed to a future spec editor, not an obligation an implementer of loom 1.0 can satisfy or test. Exposing a live `Result`/enum value to JS would change the runtime value model, which is exactly the class of deferral that `future-considerations/model-changes-and-non-goals.md` (category 3, model-level changes) exists to record — yet no entry there captures it. The reminder is therefore both an action item masquerading as a normative requirement and a deferral that is invisible from the Future Considerations buckets where a reader would look for it.

That commit resolved an over-prescription finding by demoting those shapes out of the normative rows — but planted them as a new standalone **Reference encoding (non-normative).** paragraph on the same page and added the conditional-MUST future-revision reminder inline in that paragraph, introducing both halves of this defect in one edit. `git log -S` on both the `Reference encoding (non-normative)` and `MUST be revisited before that surface ships` tokens shows `0a7884a` as the sole spec-corpus content commit; the only later commit to touch the paragraph (`31ff060`) relocated the entire corpus into `docs/` without altering the text. (Subsequent `git log -S` hits are spec-review tooling/store commits, not spec-content edits.)

## Solution Space

**Shape:** single

### Recommendation

Dismantle the **Reference encoding (non-normative).** paragraph on `runtime-value-model.md` and redistribute its two halves:

1. **Move the concrete reference shapes to `implementation-notes.md`.** Add a non-normative **Implementation hint** bullet under the **Runtime** list (alongside the existing Chevrotain and AJV hints) stating that the reference interpreter realises the enum tag as a non-enumerable `__loomEnum` string property and `Result<T, E>` as `{ ok: true, value: T }` / `{ ok: false, error: E }`, preserving the qualifier that these are implementation details, are not reachable from loom code, appear in no wire schema, and may change without a spec revision. Optionally leave a one-line non-normative pointer from `runtime-value-model.md` to that hint.

2. **Record the host-interop case as a deferral in `future-considerations/model-changes-and-non-goals.md`.** Add a bullet under **Model-level changes (no loom 1.0 seam expected)** noting that exposing a live `Result` or enum value to JS (rather than a JSON-serialised projection) — e.g. a future host-interop surface — would change the runtime value model and is not anticipated by loom 1.0. Drop the conditional-MUST framing entirely: it is a spec-author action item, not an implementer obligation.

Leave the normative rows on `runtime-value-model.md` untouched — they already state the observable contract (the tag MUST NOT appear in JSON output; the in-memory `Result` shape is not part of the language surface). Watch two edges: (a) the observable "tag absent from `JSON.stringify` output" MUST stays on the value-model page — only the concrete shape detail moves; (b) do not relocate the **Equality (`==`)** semantics that immediately follow the paragraph.

## Consequence

**Severity:** advisory

A reader can mistake the concrete `__loomEnum` / `{ ok, value }` shapes for part of the value-model contract and pin an implementation to them, re-introducing the over-prescription the normative rows were rewritten to remove. Separately, the conditional MUST is an open spec-author action item with no implementer correlate, and the host-interop deferral it implies is absent from the Future Considerations buckets, so it is discoverable only by reading the value-model page top to bottom.
