## Finding

The normative `RuntimeEvent` payload in `runtime-event-channel.md` declares two source-location fields — `query_site?: { file: string; line: number; column: number }` (the originating `@`-template / tool call / `invoke`) and `discard_site?: { file: string; line: number; column: number }` (the discarding `let _ =` / `void`-tail expression). Their inline comments describe *what site* each denotes but never state *where the `file`/`line`/`column` values come from*: nothing says the runtime reads a parse-time source span attached to the originating IR node, nor pins the indexing convention.

This is the only sourcing gap left on the `RuntimeEvent` shape. Every sibling field has a stated provenance: `occurred_at` is "stamped at the originating emission site via `Clock.wallNow()`", and `loom` / `invocation_id` are sourced from the `ActiveInvocationRegistry` entry (the **Binder-failure sourcing** paragraph pins this explicitly). `query_site` / `discard_site` alone are left to implementer invention.

The omission is sharper because a parallel producing contract already exists for the *diagnostic* surface — `lexical.md` (**Diagnostic spans**) fixes line/column numbering on the BOM-stripped, newline-normalised stream, and `diagnostic-shape.md` pins `Diagnostic.range` as **1-indexed**. The `RuntimeEvent` source fields neither restate nor cross-reference this model, so an implementer cannot tell whether `query_site.line`/`.column` are 1-indexed logical positions read from the same parse-time span the diagnostic pipeline records, or something else (e.g. positions reconstructed from a host JavaScript stack trace at emission time, which would yield transpiled-host coordinates, a different file, and a different index base).

Note also that within a single implementation the cascade-twin re-emission is *not* the exposure: clauses (d)/(f) and the **Deduplication and lifetime rules** require the boundary re-emission to copy the originating instance verbatim, so `query_site` is never re-derived at the boundary. The real exposure is cross-implementation divergence and instability of the field across distinct occurrences from the same source site.

## Issue introduction

**Verdict:** present-since-inception (of the field)
**Introducing commits:** `b3bc4ce` ("spec: pin failure-observability surface (Cluster A, Option C)", 2026-05-04)
**History:** `git log -S 'query_site'` traces the field to `b3bc4ce`, which first pinned the `RuntimeEvent` payload shape. From that commit `query_site` carried only a semantic comment ("source location of the @-template, tool call, invoke, or discarding `let _ =`") with no producing contract for the `file`/`line`/`column` values and no indexing convention. The `discard_site` field was later split out as a distinct field (`6afca02`, `13e58fe`) but inherited the same omission. The gap survived the 100KB file-split (`f5e89f4`) into `runtime-event-channel.md` and the subsequent sibling-sourcing fixes that pinned every *other* `RuntimeEvent` field's provenance — `loom` / `invocation_id` via the `ActiveInvocationRegistry` (`20f9645`, "Binder-failure RuntimeEvents have no pinned source for invocation_id / loom") and `occurred_at` via `Clock.wallNow()` — none of which extended to `query_site` / `discard_site`. The defect is an omission introduced with the feature and never closed.

## Solution Space

**Shape:** single

### Recommendation

On `runtime-event-channel.md`, beside the `RuntimeEvent` payload shape (or as a short adjacent paragraph mirroring **Binder-failure sourcing**), pin a producing contract for the two source-location fields:

- `query_site` is populated from the parse-time source span the parse pipeline already attaches to the originating IR node — the `@`-template, Pi-tool call, or `invoke` expression that raised the failure. `discard_site` is populated from the source span of the discarding `let _ =` binding (expression-statement form) or the tail `@`...`` expression's start (void-tail-function form), consistent with the discard-site location wording already in QRY-20.
- The runtime reads these captured spans; it MUST NOT reconstruct positions from a host stack trace or any runtime-only coordinate.
- `line` / `column` are **1-indexed** logical positions on the BOM-stripped, newline-normalised stream — the same span model the parse pipeline records for diagnostics (cross-reference `lexical.md` **Diagnostic spans** and `diagnostic-shape.md` `range` "1-indexed"). State this explicitly rather than leaving it to inference from the diagnostic surface.

This is one coherent producing contract covering both fields with the same span model; the edit is wide only in that it pins file/line/column origin and index base in one place.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the operator-visible `query_site` / `discard_site` values: one attaches a 1-indexed parse-time source span to the originating IR node (matching the diagnostic span model); another reconstructs positions from a runtime stack trace, producing a different index base, possibly a transpiled-host file, or unstable coordinates. Because `query_site` participates in the consumer dedup tuple `(kind, query_site, message, occurred_at)` that distinguishes occurrences from the same site, inconsistent or unstable positions break occurrence identity and cross-tool correlation for log scrapers, replay, and `/tree` consumers.
