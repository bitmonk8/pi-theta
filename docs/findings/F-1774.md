## Finding

Five normative use sites lean on specific rows of the canonical interpolation-stringification table — the `null` row, the `number` row's `NaN` / `±Infinity` cases, the `integer` row's shortest-decimal rule, the `Result<T, E>` row's parse-error contract — yet every one of them cites the table as `[Query — Stringification of interpolated values](../query.md)`. `docs/spec_topics/query.md` is the index page; it contains only a four-bullet contents list. The actual table lives on the child shard `docs/spec_topics/query/query-escapes-stringification.md` under the `## Stringification of interpolated values` heading (anchor `#stringification-of-interpolated-values`). The pre-split version of `query.md` did contain the table, so the references were correct when written; the spec-split repointed the URLs to the index page but did not append the anchor that survives the split.

Concretely, the sites that need fixing are:

- `frontmatter/frontmatter-fields-b-and-templates.md` line 43 — `*Stringification.*` paragraph of the `system:` field; the prose calls out specific rows (`null`, `Result<T, E>`, `number` non-finite cases) but the reader cannot find them on `../query.md` to cross-check rows the prose does not enumerate (`string`, `boolean`, enum variant, `array<T>`, schema-typed object, `integer`).
- `frontmatter/frontmatter-fields-b-and-templates.md` line 69 — Template Interpolation closing paragraph asserting the `@`-template surface uses the same table.
- `diagnostics/placeholder-rendering-a.md` line 32 — placeholder-rendering rule defers to the table for category 2.
- `diagnostics/placeholder-rendering-a.md` line 79 — the `integer`-row claim about shortest-decimal rendering.
- `diagnostics/code-registry-parse.md` line 69 — *Reference* column of the `loom/parse/interpolated-result` row.

A correct precedent already exists at `diagnostics/placeholder-rendering-b.md` line 89, which cites `../query/query-escapes-stringification.md#stringification-of-interpolated-values` directly; the five sites above need the same form.

The implementability cost is concrete: an implementer following any of these links lands on a four-bullet contents page with no table, has to guess which child shard owns the rule, and may produce divergent rendering for the rows the citing prose does not enumerate inline (notably `string`, `boolean`, enum variant, `array<T>`, schema-typed object) — exactly the rows whose model-visible text the canonical-table invariant exists to keep uniform.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `f5e89f4` (`docs: enforce 100KB hard / 30KB recommended size cap on spec set`, 2026-06-04)
**History:** The pre-split `docs/spec_topics/query.md` (visible at `f5e89f4^`) was a single 280-line file whose `## Stringification of interpolated values` heading sat at line 147; references to `../query.md` resolved to the table directly. `f5e89f4` split that file into an index page plus four child shards (including `query/query-escapes-stringification.md` which now owns the table) and "repointed 1571 cross-references … via a map-driven rewrite". The rewrite preserved the file path `../query.md` but did not append the per-section anchor the split shard now requires, so the five citations carried forward as bare index-page links. `f5e89f4` is also the commit that created the `frontmatter-fields-b-and-templates.md`, `placeholder-rendering-a.md`, and `code-registry-parse.md` shards in their current form, so the defect is present-since-split at every use site (no prior shard-local history to walk). The single correct anchored reference, at `placeholder-rendering-b.md:89`, was authored later and avoided the bulk rewrite. No subsequent commit has touched the bare-index citations.

## Solution Space

**Shape:** single

### Recommendation

At each of the five use sites, replace `[Query — Stringification of interpolated values](../query.md)` with `[Query — Stringification of interpolated values](../query/query-escapes-stringification.md#stringification-of-interpolated-values)` — the exact form already in use at `placeholder-rendering-b.md:89`. Edit sites:

1. `frontmatter/frontmatter-fields-b-and-templates.md` line 43 — `system:` Stringification paragraph.
2. `frontmatter/frontmatter-fields-b-and-templates.md` line 69 — Template Interpolation closing paragraph.
3. `diagnostics/placeholder-rendering-a.md` line 32 — category-2 *Rule*.
4. `diagnostics/placeholder-rendering-a.md` line 79 — category-4 *Rule*.
5. `diagnostics/code-registry-parse.md` line 69 — `loom/parse/interpolated-result` row *Reference* cell.

The diagnostics-registry row at edit site 5 is built-time-asserted on its *Message* column, not its *Reference* column, so the URL change is text-only and triggers no test-vector update; the other four sites are pure prose. Do not delete or relocate the row enumerations the surrounding prose performs (the `null` / `NaN` / `±Infinity` / `Result<T, E>` callouts at site 1, the integer-rule restatement at site 4) — keeping them inline at the use site continues to anchor the reader on the rows the cite-and-defer pattern most needs to surface; the anchor change only ensures the deferred rows are one click away too.

Optional follow-on (not required to close the finding): when the wider `query.md` filename convention is next revisited, consider repointing all `../query.md` citations that target post-split content to their owning shards, since the index page rarely is the intended landing — but treat that as a separate cleanup and do not block this fix on it.

## Consequence

**Severity:** advisory

An implementer reading any of the five citing paragraphs follows the link and hits a contents page with no table, then has to grep the corpus to discover the rule lives on `query/query-escapes-stringification.md`. The risk is silent rendering divergence on the rows the citing prose does not enumerate inline — `string`, `boolean`, enum variant, `array<T>`, schema-typed object, `integer` — because each implementer reconstructs the rule from memory or from JavaScript defaults.
