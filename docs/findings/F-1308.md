## Problem

When the chokidar watcher throws, the spec specifies surfacing the throw through `ctx.ui.notify` (the *Transient toasts (auxiliary)* paragraph in `diagnostic-shape.md` and the PIC-14 chokidar-adapter `error`-route prose) but is silent on the watcher's post-throw lifecycle, so whether hot-reload survives is undefined. A reasonable implementer could leave the watcher subscribed and continue, tear it down so the next reload requires `/reload`, or tear down and mark the extension `"degraded-needs-reload"` via the `LoomRegistry` drain-state tag — each consistent with the surfacing rule but yielding a different post-throw system. A silently dead watcher is indistinguishable from a live one until the operator notices nothing reloads.

## Solution approach

Clarify the chokidar watcher's post-throw lifecycle posture at `registration-steps.md` step 5 (anchor `watcher-hot-reload-registration`), which owns the watcher lifecycle: state whether hot-reload survives a watcher throw and, if the watcher is torn down, how the operator recovers it. The existing structural-change `/reload` system-note pattern in step 5 is an available landmark to mirror when recovery is operator-driven. Name the chosen posture in the PIC-14 chokidar-adapter `error`-route prose in `host-interfaces-services.md`, and add a back-reference from the *Transient toasts (auxiliary)* paragraph in `diagnostic-shape.md`.
