## Finding

`V17a`'s Adds describes the leaf as adding "the cancellation-runtime *semantics* of `SHUTDOWN_AWAIT_CAP_MS = 2000` … **as a conceptual consumer**." The phrase "conceptual consumer" is defined nowhere in `docs/plan_topics/conventions.md`, `docs/plan.md`, or `docs/plan_topics/coverage-matrix.md`. An implementer cannot tell from it whether `V17a`'s source must contain a runtime `import { SHUTDOWN_AWAIT_CAP_MS }` (and a use site) or merely a doc comment that names the constant's meaning.

The ambiguity is sharpened into an outright contradiction by the sibling leaf. `V9a`'s Adds states the pinned-constants block "`V9g`, `V9i`, and `V17a` import rather than redeclare" — asserting `V17a` is a *runtime importer*. But `V17a`'s own Adds positions it as a *semantics owner* and a *conceptual* consumer, and lists only `V9g` and `V9i` as the leaves that "consume that same block constant." The two files disagree on whether `V17a` imports the literal.

First-hand reading resolves the question against `V17a` being a runtime importer. `SHUTDOWN_AWAIT_CAP_MS` bounds two concrete runtime sites: `V9g`'s `session_shutdown` disposal await (`spec_topics/pi-integration-contract/patch-skew-degradation.md` §`session_shutdown` sub-step 3) and `V9i`'s subagent-disposal bound (`PIC-9`: "`SHUTDOWN_AWAIT_CAP_MS` covers disposal"). The coverage-matrix maps the sub-step-3 settle-all-bounded-by-`SHUTDOWN_AWAIT_CAP_MS` obligation to `V9g` alone. `V17a`'s own additions — the `loomAbort` controller, forwarding, abort-reason propagation, the fixed checkpoint set, and the late-settlement discard rules — contain no site that reads the 2000 ms value. `V17a` therefore owns the constant's *semantics* (its meaning within the cancellation contract) but has no runtime use for the literal; a physical `import` in `V17a` would be a dead import. The polished resolution must say so in `V17a` and must drop `V17a` from `V9a`'s importer list so the two leaves agree.

## Solution Space

**Shape:** single

### Recommendation

Resolve the ambiguity in favour of `V17a` being the *semantics owner* — not a runtime importer of the literal — because `V17a` has no site that reads the 2000 ms value (the runtime await lives in `V9g`'s `session_shutdown` disposal await and `V9i`'s subagent-disposal bound, and the coverage-matrix maps the `patch-skew-degradation` sub-step-3 cap to `V9g`). Apply the decision in both leaves so they agree:

- In `docs/plan_topics/V17a-cancellation-core.md`, Adds: replace "as a conceptual consumer" with wording that states `V17a` defines/owns the *semantics* (meaning) of `SHUTDOWN_AWAIT_CAP_MS` within the cancellation contract and does **not** import or read the runtime literal — the runtime consumers are `V9g` (session-shutdown disposal await) and `V9i` (subagent-disposal bound), each importing the value from `V9a`'s pinned-constants block. Keep the existing statement that the constant physically resides in `V9a`'s block and that `V18c` reads it at build time.
- In `docs/plan_topics/V9a-capability-probe.md`, Adds: change "`V9g`, `V9i`, and `V17a` import rather than redeclare" to name only the runtime importers — `V9g` and `V9i` import the constant rather than redeclare it — and state that `V17a` owns the constant's semantics without importing the literal. Leave the `V18c` build-time literal-read clause intact.

Edge case for the implementer: `V17a` keeping `V9a` in its `Deps` is independently fine (the swallowing-handler / checkpoint wiring shares other surfaces), but the dependency must not be justified by a `SHUTDOWN_AWAIT_CAP_MS` import that does not exist in `V17a`.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads `V9a`'s "`V17a` … import rather than redeclare" and adds a dead runtime import of `SHUTDOWN_AWAIT_CAP_MS` to `V17a` (which has no site that uses the value); another reads `V17a`'s "conceptual consumer" and writes only a comment. The two leaves contradict each other on whether `V17a` imports the literal, so neither implementer can be sure they matched the plan.
