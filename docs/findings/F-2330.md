## Finding

The ceiling-#4 spec pages name two of the five enforcement boundaries "**tool-call args, model-driven**" and "**tool-call args, code-driven**" — see the per-boundary destination/surface table in [`schema-subset.md`](../spec_topics/schema-subset.md) (rows #2/#3) and [`ceilings-3-and-4.md`](../spec_topics/hard-ceilings/ceilings-3-and-4.md) (the per-boundary table and CIO-3). The token of art is "tool-call args": the arguments of a `tool_use` / `<name>(args)` tool call, validated at the AJV boundary.

The V5e and V5e-T "Per-boundary routing decision" Tests bullet drops the load-bearing word "call", writing "model-driven tool args → model feedback" and "code-driven tool args → `CodeToolError`". Without "call", "tool args" reads as the arguments to a loom-code call expression generally, rather than the specific ceiling-#4 tool-call-argument boundary the spec names. The two other site-classes in the same bullet ("typed-query response", "`params` and `invoke<T>` return") keep their spec terms, so the truncation is localized to the two tool-call rows.

The defect is a terminology drift from the spec's canonical boundary name in a table that exists precisely to map each spec boundary to its destination surface. It also breaks grep-based audits that key on the spec token "tool-call args".

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** The plan corpus is git-tracked. `git log -S 'model-driven tool args'` on both `docs/plan_topics/V5e-depth-enforcement.md` and `docs/plan_topics/V5e-T-depth-enforcement.md` returns only `c6a664e`, the initial plan-build commit; `git show c6a664e:docs/plan_topics/V5e-depth-enforcement.md` shows the truncated phrase ("model-driven tool args, code-driven tool args") present in the leaf's first Tests bullet from inception. The site-enumeration prose later migrated from that first bullet into the "Per-boundary routing decision" bullet across subsequent plan-fix commits (`3fa39a9`, `6994be4`, `9d880b8`), but the "tool args" wording carried through every edit unchanged. The defect has therefore existed since the V5e leaf pair was first authored, in both paired files.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V5e-depth-enforcement.md` and `docs/plan_topics/V5e-T-depth-enforcement.md`, in the "Per-boundary routing decision" Tests bullet, restore the spec's boundary term: replace `model-driven tool args` with `model-driven tool-call args` and `code-driven tool args` with `code-driven tool-call args`, matching rows #2/#3 of the `schema-subset.md` per-boundary table and the `ceilings-3-and-4.md` table. The other three site-class labels in the bullet (typed-query response, `params` and `invoke<T>` return, slash-load `params`) already match the spec and need no change. Apply the identical edit to both paired files so the Tests bullets stay verbatim-aligned.

## Consequence

**Severity:** advisory

An implementer reading the routing bullet may briefly misread "tool args" as loom-code call arguments rather than the ceiling-#4 tool-call-argument AJV boundary; the adjacent destination column (`→ CodeToolError`, `→ model feedback`) disambiguates, so the leaf still produces a correct gate. The concrete cost is terminology drift from the spec's boundary name and a grep-audit miss for the spec token "tool-call args".
