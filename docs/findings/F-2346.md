## Problem

The fidelity of the in-process session double to a real Pi host is the load-bearing premise behind every automated runtime gate (H4a, H4b/H4c, H7a, V18d), and the only thing that verifies it is the manual real-host smoke run defined in `real-host-smoke-gate.md`. That gate is a contributor obligation with no CI hook: trigger (2) ("a diff touching the four fidelity-contract axes") has no automatic detector, so a skipped run or an unrecognised axis-touching diff is "detectable only after the fact, by reading the triggering commit message." The H6a gate-activation commit-message evidence record is the one structured audit point, but it records only the run that happened — the Pi-SDK pin literal, owner, date, and result — not which trigger-(2)-relevant diffs landed since the last recorded run. Consequently the after-the-fact audit cannot confirm that every axis-touching change between recorded runs was actually covered, leaving a real-Pi divergence in a core pipeline premise unauditable.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** e7f14dd — pi-loom plan: resolve "Real-host fidelity of the session double has no reproducible detection point" (2026-06-11, Thomas Andersen)
**History:** The manual real-host smoke design — the trigger-(2) "touches the four fidelity-contract axes" mechanism and its "detectable only after the fact" posture — was introduced by e7f14dd, which established the smoke run as the sole double-vs-real detection point. Subsequent commits hardened the evidence record (eca63cf added a falsifiable record; adba28b corrected a "closed at source" overclaim to after-the-fact-only; ee9acd0/F-0913 pinned H6a as evidence-record owner) but each recorded the run that happened, not which trigger-(2)-relevant diffs landed since the last run, so the coverage-auditability gap has persisted since e7f14dd.

## Solution approach

Retain the manual real-host smoke gate as designed and close the auditability gap. Extend the H6a gate-activation commit-message evidence record so it also captures the trigger-(2)-relevant diffs that landed since the last recorded smoke run (commits editing the double's modelling of, or the production seam realising, any of the four fidelity-contract axes), recording explicitly when none landed, so the auditor can reconcile axis-touching changes against the runs that covered them. Update the "contributor obligation, not a CI-enforced gate … detectable only after the fact" clause in `real-host-smoke-gate.md` so the named after-the-fact audit surface is this coverage record (the diffs-since-last-run set), not only the recorded run.

## Solution constraints

- Any optional CI advisory flagging fidelity-axis-touching commits must stay non-blocking; a blocking CI gate would contradict the deliberate manual-gate design and pull host/SDK coupling into CI.
