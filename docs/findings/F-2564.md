## Finding

The *Tool execution from loom code* section of `host-interfaces-core.md` specifies that a code-side `<name>(args)` call invokes a Pi tool's `execute(toolCallId, params, signal, onUpdate, ctx)` **directly**, passing a `toolCallId` that is a freshly-minted UUID prefixed `loom-direct:` (generated through the `IdSource` seam, PIC-20). This `toolCallId` is synthesised by the loom runtime and is **never registered through Pi's tool-call dispatcher** — the loom drives `execute()` out-of-band, bypassing the dispatcher path through which a model-issued `tool_use` block would normally reach the same `execute()` member.

For this design to be correct, the receiving tool's `execute()` implementation must treat the inbound `toolCallId` as an **opaque** value: it must not look the id up in any dispatcher-side registry, must not correlate it against a prior `tool_use` block, and must not validate its shape or provenance. The spec relies on this consumption posture for every code-side tool call, but never states it. The section pins how the id is *generated* (format, minting seam) and what loom *reads back* from the result envelope (the `isError`-free `AgentToolResult.content`, explicitly noting loom never observes dispatcher-side types for an out-of-band call), yet it is silent on what the callee is assumed to *do* with the id it receives.

This is precisely the class of unpinned Pi host-behaviour presupposition the corpus otherwise inventories rigorously: `host-prerequisites.md` carries a named presupposition paragraph for each such reliance (chronological `AgentMessage[]` ordering, `complete()` forced-tool behaviour, slash-handler promise lifecycle, `registerTool` post-startup acceptance, and others), and each is routed to the version-bump editorial-review checklist in `version-bump-step2.md` (items `(f)`–`(aq)`) so a Pi minor that falsifies it is caught at bump time. The toolCallId-opaqueness reliance has neither a presupposition paragraph nor a checklist row, leaving a hole in the bump-time safety net the spec elsewhere maintains.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `27ec53d` (2026-05-04, "Resolve Cluster-A: per-mode tool registration") is the earliest commit where the direct out-of-band `execute(toolCallId, …)` design with a synthesised `loom-direct:` id is textually locatable; that commit references the *Tool execution from loom code* prose as already-present, so the design — and the omission of the opaqueness presupposition — predates it.
**History:** The corpus is git-tracked. The out-of-band direct-`execute()` design has carried the unstated opaqueness/non-correlation consumption posture since the section was first authored. Two later edits refined the id's *generation* without ever surfacing the *consumption* posture: `0c6c347` (2026-06-25, F-1396) pinned the `toolCallId` UUID body format and routed minting through PIC-20, and `95d4dd9` (2026-06-27, F-1923) named `IdSource.newToolCallId()`. A history search for any opaqueness / non-dispatcher-correlation statement tied to `toolCallId` (`git log -S`/`-G` over `opaque`, `correlat`, `dispatcher`, `out-of-band` across the corpus) returns no commit that ever added one. The defect is therefore an omission present since inception, not one a specific later commit introduced.

## Solution Space

**Shape:** single

### Recommendation

Add a named presupposition to the `host-prerequisites.md` register — in the same shape as the existing entries (anchored paragraph stating the relied-on Pi behaviour and naming the consuming site) — asserting that, at the loom 1.0 Pi-SDK pin, a Pi tool's `execute(toolCallId, …)` treats the `toolCallId` as an **opaque** value: it does not look the id up in a dispatcher-side registry, does not correlate it against a prior model-issued `tool_use` block, and does not reject an id absent from any dispatcher state. Cite the *Tool execution from loom code* section of `host-interfaces-core.md` as the consumer and note the id is synthesised out-of-band (`loom-direct:` prefix, minted via PIC-20) and never registered through Pi's dispatcher.

Route it to the version-bump editorial-review checklist by adding one SHOULD-level item to the `(f)`–`(aq)` range in `version-bump-step2.md`, pointing at the new `host-prerequisites.md` anchor, so the per-bump audit re-confirms the opaqueness posture against each candidate Pi minor. Adding the item extends the recorded-outcome range `V18c` asserts on the bump commit; update that range accordingly.

If the contributor can instead locate an authoritative Pi SDK statement that `execute()` callees treat the `toolCallId` as opaque, cite it at the `host-interfaces-core.md` call site in place of the presupposition — but absent such a citation, the presupposition-plus-checklist form is required to match the discipline applied to every other unpinned host reliance in the corpus.

## Consequence

**Severity:** advisory

At the loom 1.0 Pi-SDK pin the assumption holds for Pi's built-in tools and for `defineTool`-wrapped `.loom` callables, so an implementer can ship a working runtime. The exposure is bump-time: because the reliance is neither stated as a presupposition nor enumerated on the version-bump editorial-review checklist, a future Pi minor whose tool contract began validating or dispatcher-correlating the `toolCallId` would break every code-side `execute()` call with no surface-inventory or editorial-review signal — the same gap the spec deliberately closes for its other host presuppositions.
