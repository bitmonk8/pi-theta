## Finding

PIC-17 step 4 places the `pi.setActiveTools(snapshot)` restore in a `finally` "so cancellation, panic, and provider exceptions all preserve the invariant," but the acceptance-criteria section asserts only the happy-path install vectors (a)/(b); no criterion verifies the restore actually runs after the protected body throws or is cancelled.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, Thomas Andersen); 19b193b — resolve "PIC-17 active-set swap/snapshot throws have no failure-mode rule" (2026-06-05, Thomas Andersen); 823ce54 — resolve "Recovery-mutex acquisition semantics and teardown-budget interaction undefined" (2026-06-06, Thomas Andersen); 8008b91 — fix F-1217: add acceptance-criteria/fixture blocks for four Shard-08 MUSTs (2026-06-25, Thomas Andersen)
**History:** The PIC-17 step-4 `finally` invariant and the PIC-8 restore-failure protocol were carried into this file at its first commit (f5e89f4, the size-cap split); 19b193b added the PIC-19 setup-failure protocol and 823ce54 authored the recovery-mutex contingent contract inline in the consequences bullet — each adding a failure-path MUST without a fixture. The gap crystallised at 8008b91, which added the `Acceptance criteria — PIC-17 active-set install vector` section but scoped its fixtures to the happy-path install vectors (a)/(b) only, never extending coverage to the exception path or the PIC-8/PIC-19 protocols. The defect is the interaction: acceptance-criteria coverage did not keep pace with the failure-path protocols, and the contingency block was never separated from must-build prose.

## Solution Space

**Shape:** single

### Recommendation

Add a criterion (c) to `## Acceptance criteria — PIC-17 active-set install vector` asserting that when the query body throws or the turn is cancelled, `pi.setActiveTools(snapshot)` is called exactly once with the step-1 snapshot and the user session's prior active set is observably restored, with the inner failure/cancellation surfaced and not masked. Cover both the throw and the cancel sub-cases (both transit the same `finally`).

## Consequence

**Severity:** advisory

The normative behaviour is fully specified in prose, but the spec's verification surface is incomplete: the load-bearing "restore preserves the invariant under throw/cancel" property and the PIC-8/PIC-19 failure protocols carry no acceptance criterion, so a regression that drops the restore from `finally` (or mishandles a restore-throw / setup-throw) would pass the section's stated criteria. The inline contingency contract additionally risks an implementer building the recovery mutex into loom 1.0 when it is contingency-only.
