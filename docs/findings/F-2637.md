## Finding

`conversation-drive.md` carries three full *consumption-posture* paragraphs (driven-turn session-commit ordering, user-visible streaming ordering, `waitForIdle()` no-rejection), each an unpinned host presupposition routed to a `version-bump-intro.md` editorial-review item. These are catalogued nowhere in `host-prerequisites.md`, the page that owns the unpinned-host-presupposition inventory, so the presupposition catalogue is incomplete and the postures are discoverable only inline.

## Solution Space

**Shape:** single

### Recommendation

Add each posture as an entry in the `host-prerequisites.md` presupposition catalogue (name, the SDK surface it rests on, and the editorial-review item it routes to), and reduce the inline `conversation-drive.md` text to a one-clause statement plus a cross-reference to the catalogue entry. Keep the behavioural prose that each posture qualifies (PIC-50/51/53 probes) in place.

## Consequence

**Severity:** advisory

Discoverability and over-build hazards rather than divergent behaviour: the re-entrancy `MUST NOT` buried in a field-derivations list can be missed by an implementer reading the Concurrency contract; deferred/future material (`expanded` widening) described inside the normative body risks being built for loom 1.0; and the test-infrastructure prescription over-constrains the test harness. Implementers can still produce a working, conformant system from the current text.
