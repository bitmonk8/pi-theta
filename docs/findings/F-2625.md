## Finding

`schema-subset.md#subs-2` reads: *"Authors and implementers MUST use `schema slug` (or the bare form `slug` when context is unambiguous) … the synonyms `schema hash`, `schema-hash`, `sha12`, `lowered-schema hash`, and `lowered-schema content hash` are spec-prose drift to be avoided. This obligation is the source of truth for a future grep gate over those synonyms."* Two parts of this framing conflict with the corpus's own binding-scope rule, [GOV-18](./governance/corpus-direction-and-scope.md#gov-18).

First, the obligation binds **"implementers."** GOV-18 fixes the only two parties a spec obligation may bind: arm (a), the implementation target (software whose observable behaviour on `.loom`/`.warp` inputs the spec describes), and arm (b), the spec corpus itself. An implementer's choice of prose vocabulary in their own code or docs is neither — it is exactly the kind of third party GOV-18's `#gov-18-binding-prohibition` clause forbids ("The spec corpus MUST NOT bind any third party"). Second, "the source of truth for a future grep gate" cites a verification venue, which the same clause explicitly strips: *"A spec rule whose pre-diff wording cited a verification venue … loses that citation under this rule; the bare invariant survives as arm (b) content, and auditing the invariant is third-party tooling's responsibility carried outside the spec corpus."* The "future grep gate" has no corpus boundary, no threshold, and no existing implementation, so the MUST cannot be discharged or falsified today — it is untestable as written.

The defect's framing makes SUBS-2 self-inconsistent with the governance corpus rather than wrong about lowering. The actual durable intent — that within the spec corpus the canonical-hash output is named `schema slug`/`slug` and the listed synonyms are not used as its name — is a legitimate arm-(b) invariant about the corpus's own content (on the same terms as GOV-1's anchor-form invariant or GOV-5's prefix-grammar invariant); it is only the "Authors and implementers MUST" subject and the grep-gate citation that overreach. The glossary `schema slug` entry ([glossary.md](./glossary.md)) carries a matching "source of truth for a future grep gate" clause that must move in lockstep.

## Solution Space

**Shape:** single

### Recommendation

Reframe SUBS-2 in place on `schema-subset.md` as a GOV-18 arm-(b) invariant about the spec corpus's own content, and update the glossary reference to match:

- Restate SUBS-2 as a property of the corpus's prose rather than a directive to people. For example: *"**SUBS-2.** *Schema-slug terminology.* Within the spec corpus, the 16-hex output of this recipe is named `schema slug` (or the bare form `slug` when context is unambiguous); the synonyms `schema hash`, `schema-hash`, `sha12`, `lowered-schema hash`, and `lowered-schema content hash` do not appear as the name of this value. The rule is purely terminology — the on-the-wire synthesised tool names ([step 5](#synthesised-names)) and the `<slug>` placeholder token in diagnostic registry rows and template bodies are unaffected."*
- Remove the "Authors and implementers MUST use" subject (binds a third party) and the sentence "This obligation is the source of truth for a future grep gate over those synonyms" (cites a verification venue). Per GOV-18 the bare invariant survives as arm (b); auditing it is third-party tooling's responsibility carried outside the corpus, and the spec takes no position on the gate's existence or shape.
- Update the `glossary.md` `schema slug` entry's matching clause ("the source of truth for a future grep gate") so it no longer cites the gate, mirroring the reframed SUBS-2.

Keep SUBS-2 on `schema-subset.md`: the `SUBS` REQ-ID prefix is bound to that page by the [REQ-ID prefix table](./governance/req-id-prefix-table-active-a.md#req-id-prefix-table), and the obligation sits beside the canonical-hash recipe whose output it names; physically relocating the REQ-ID to a governance page would break the prefix→page binding and orphan it from `SUBS-1`.

Edge cases for the implementer: confine the reframe to SUBS-2 and its glossary reference. The sibling REQ-IDs `FRNT-2`/`FRNT-3` (frontmatter-fields-a.md) carry the identical "Authors and implementers MUST" pattern from the same F-1640 commit but are separate obligations on a separate page — out of scope here. SUBS-2 is governed by GOV-1/GOV-3/GOV-8 as a REQ-ID, so this is a GOV-8 *Pure rewording* (modal restatement of the same obligation), not a split or deletion; the identifier and anchor stay.

## Consequence

**Severity:** advisory

SUBS-2 is a normative MUST the corpus's own GOV-18 forbids (it binds "implementers" and cites a verification venue), so it stands as an internal self-consistency defect that a GOV-18 audit flags. Because the cited "future grep gate" does not exist, the MUST is unfalsifiable and gives a false impression of an enforceable conformance check. No runtime behaviour diverges — the obligation is purely terminological.
