## Finding

`frontmatter-fields-b-and-templates.md` pins two normative behaviours for `respond_repair.methodology:` that V6e and V6e-T never assert:

1. **Load-time value rejection.** "A present `methodology:` value outside the recognised set — including a non-string scalar — is the load-time error `loom/load/unknown-methodology-value` and the loom is not registered, the same recognised-key / unrecognised-value split as `mode:`." This is a distinct diagnostic from the `loom/load/frontmatter-value-out-of-range` code V6e-T already asserts (which covers numeric `max_rounds` / `attempts`, not the methodology enum).

2. **`methodology: none` ≡ `attempts: 0`.** Under `none`, "a declared non-zero `respond_repair.attempts` is silently ignored — the loom loads with no diagnostic … and behaves exactly as `attempts: 0`", and `attempts` on the returned `ValidationError` is `0` on every surfacing path (initial AJV failure, depth-walk failure, ERR-17 forced-respond branch). V13d's Tests assert "`none`/`0` means no follow-up" but not the silent-ignore of a declared non-zero `attempts`, nor that the returned `ValidationError.attempts` is `0`.

V6e/V6e-T's only Tests bullets are FRNT-1 (`tool_loop` bounds) and `loom/load/frontmatter-value-out-of-range`. A loom-load implementation that omits the methodology-enum rejection, or that honours a non-zero `attempts` under `methodology: none`, ships green: no leaf-level test exercises either obligation. The diagnostic-code closing gate (DIAG-2 / H6a) only witnesses code existence at release time, and the `none`-equivalence is a behaviour with no code to register.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `c6a664e` (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** The V6e and V6e-T Tests blocks have carried exactly two bullets — FRNT-1 and the out-of-range frontmatter code — since the initial plan-build commit `c6a664e`; no `methodology`-value bullet was ever present. The only later commit to touch these files, `4c76a7c` ("resolve 'V6e/V6e-T assert non-existent loom/parse diagnostic code'"), edited the same Tests block but only renamed the out-of-range code prefix `loom/parse/` → `loom/load/`; it neither introduced nor addressed the methodology-assertion gap. The gap is original to the leaf, not a regression.

## Solution Space

**Shape:** multiple

The finding carries two independent obligations with different ownership: a load-time frontmatter-validation assertion (V6e's own domain) and a runtime respond-repair behaviour that overlaps V13d's existing `none`/`0` bullet. They touch different leaves and different Tests blocks, so each is its own option; both are required.

### Option A — load-time `unknown-methodology-value` rejection

**Approach** Assert that an out-of-set or non-string `methodology:` value fires `loom/load/unknown-methodology-value` and leaves the loom unregistered.

**Plan edits** Add a Tests bullet to `V6e-T-respond-repair-tool-loop.md` (mirrored in `V6e-respond-repair-tool-loop.md` Tests) asserting that a `methodology:` value outside the recognised set, and a non-string scalar `methodology:`, each fire `loom/load/unknown-methodology-value` and the loom is not registered. Extend `V6e`'s Ships-when to name this code among the asserted load diagnostics.

**Spec edits** None — `frontmatter-fields-b-and-templates.md` already pins the code.

**Pros** Self-contained to the V6e leaf pair; no cross-leaf reconciliation; the diagnostic is unambiguously V6e-owned (frontmatter field validation, parallel to `mode:`).

**Cons** None material.

**Risks** Confirm `unknown-methodology-value` is the exact registered code spelling in the diagnostic registry before citing it.

### Option B — `methodology: none` ≡ `attempts: 0` behaviour

**Approach** Assert the silent-ignore of a declared non-zero `attempts` under `none` (no diagnostic, behaves as `attempts: 0`) and that the returned `ValidationError.attempts` is `0` on every surfacing path.

**Plan edits** Extend `V13d`/`V13d-T`'s existing "`none`/`0` means no follow-up" Tests bullet to also assert that a declared non-zero `respond_repair.attempts` under `methodology: none` is silently ignored (loads with no diagnostic, behaves as `attempts: 0`) and that the returned `ValidationError.attempts` is `0` on the initial-AJV, depth-walk, and ERR-17 forced-respond paths.

**Spec edits** None.

**Pros** Lands the runtime half where the respond-repair loop is already tested, avoiding a duplicate ownership surface in V6e.

**Cons** Touches V13d, a leaf the finding is not primarily about; requires reconciling against V13d's existing none/0 bullet so the obligation is stated once.

**Risks** Do not also add a competing runtime-behaviour assertion to V6e-T; that would create a second normative home for the same behaviour.

### Recommendation

Resolve **Option A first** — it is scope-bounded to the V6e leaf pair and lands the load-time rejection on a stable baseline. Then resolve **Option B**, extending V13d's existing `none`/`0` Tests bullet so the runtime `none`-equivalence is asserted once where the respond-repair loop is already exercised. Keep the load-time concern (Option A) in V6e and the runtime concern (Option B) in V13d; do not duplicate either assertion across both leaves. The V6e–V13d Deps edge already exists (V6e Deps name `V13d`), so no new dependency is introduced.

## Consequence

**Severity:** correctness

A normative load-time diagnostic (`loom/load/unknown-methodology-value`) and a normative accept-and-silently-ignore behaviour (`methodology: none` ⇒ `attempts: 0`) can be omitted or mis-implemented while every leaf-level test stays green. Two implementers reading only V6e/V6e-T would diverge on whether a bad methodology value rejects loom registration and on whether a declared non-zero `attempts` is honoured under `none`.
