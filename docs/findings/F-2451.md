## Finding

The Step-0 opening paragraph of `capability-probe.md` declares that "the factory runs a synchronous probe of **three host preconditions** in the **fixed order (a) → (b) → (c)**" and asserts that this same paragraph "is the **canonical source of truth** for every probe rule". The body of the very same page, however, enumerates **five** lettered sub-steps — `(a)` Node floor, `(b)` `AbortSignal`/`AbortController` shape, `(c)` factory-probable SDK capabilities, `(d)` peer-dep lock-step version, `(e)` `typebox` host-shape — and the reconciliation paragraph near the end states the correct short-circuit order as `(a)` → `(b)` → `(c)+(d)` → `(e)`. PIC-5 reinforces this with "Probes MUST NOT be added beyond the **five** enumerated checks", and the `details.kind` enumeration carries the `peer-dep-*` and `typebox-shape` discriminators that only `(d)` and `(e)` produce.

The opening's parenthetical `(a) → (b) → (c)` therefore omits `(d)` and `(e)` outright. The "three" count is defensible as the conceptual precondition grouping inherited from the orientation aggregator (Node / `AbortSignal` / SDK-capability surface, with peer-dep and typebox folded under them), but pairing that count with an explicit sub-step letter range that stops at `(c)` directly contradicts the page's own normative enumeration. Because the paragraph claims canonical-source-of-truth status, a reader who anchors on it — rather than reading to the reconciliation paragraph — can conclude the gating probe ends at `(c)` and that `(d)` peer-dep version and `(e)` `Type.Unsafe` presence are not part of the refusal contract.

The fix is local to the opening sentence: restate it so the sub-step range matches the body's `(a)` → `(b)` → `(c)+(d)` → `(e)` order (and, if the conceptual "three preconditions" phrasing is retained, decouple it from the sub-step lettering so the two do not collide).

## Solution Space

**Shape:** single

### Recommendation

Reword the Step-0 opening sentence so its sub-step enumeration matches the body's reconciled order. Replace "a synchronous probe of **three host preconditions** in the **fixed order (a) → (b) → (c)** below" with a phrasing that names the full ordered set — `(a)` → `(b)` → `(c)+(d)` → `(e)` — consistent with the reconciliation paragraph and PIC-5's "five enumerated checks". If the "three host preconditions" conceptual grouping (inherited from the `overview-and-orientation.md` orientation aggregator) is worth keeping, decouple it from the sub-step lettering: state the three conceptual precondition groups separately from the five executable sub-steps, rather than equating "three preconditions" with the letter range `(a) → (b) → (c)`. Edge cases for the editor: keep the paragraph's "canonical source of truth" claim intact (it is correct and load-bearing); do not alter the `(d)`/`(e)` short-circuit semantics, only the opening's description of them; and leave the `overview-and-orientation.md` "preconditions 1–3" orientation framing as-is — it is a conceptual grouping, not a sub-step count, so no forward-link relabel is required by this fix.

## Consequence

**Severity:** correctness

A reader who trusts the opening paragraph's self-declared "canonical source of truth" status and its `(a) → (b) → (c)` range can treat the `(d)` peer-dep and `(e)` `typebox` checks as outside the gating probe. An implementation that omits them lets a mis-versioned peer or a host missing callable `Type.Unsafe` pass Step 0 and then fail later as an uncaught runtime `TypeError` at the first tool-registration call, instead of refusing cleanly with `loom/load/host-incompatible` (`details.kind = "peer-dep-*"` / `"typebox-shape"`). The body, PIC-5, and the reconciliation paragraph all state five, so a careful reader recovers — but the contradiction sits in the paragraph that claims to override the others.
