## Finding

The `loom/runtime/internal-error` row in `code-registry-runtime.md` is the runtime-defect surface: it fires precisely on an *unanticipated* interpreter or adapter throw outside the closed panic-source list. Its trigger prose states `message` is the underlying `error.message` and `hint` carries the underlying `error.stack` "for operator triage (rendered as `"<no stack available>"` when `error.stack` is falsy)". JavaScript permits `throw <any value>` — a non-Error, non-object, or hostile-object value — so the caught value's `.stack` is not guaranteed to be a safely-readable string.

The companion `<error.message>` read is hardened against exactly this hazard: the §6 underlying-error coercion in `placeholder-rendering-b.md#underlying-error-coercion` coerces a caught throw to a guaranteed string (use `.message` when it is a string; otherwise `String(v)`, or the literal `<unreadable>` when even that throws), explicitly so "a non-Error throw never yields `undefined`, a non-string value that defeats the truncation rule, or a synchronous `TypeError`." That discipline is scoped to the *message* read (and the enumerated message-bearing `hint`/`message`/`details` fields). It is never applied to the `.stack` read.

The `.stack` read therefore carries only a *falsy* guard. Two cases fall through it:

- **Throwing getter.** A caught value with a throwing `.stack` accessor (`{ get stack() { throw … } }`) throws *during emission of the very diagnostic meant to report the originating defect*. The falsy check never runs because the property access itself throws.
- **Truthy non-string.** A caught value whose `.stack` is a truthy non-string (an object, a number) passes the falsy guard and is carried into the `hint` field as-is — a non-string `hint` value, the exact "value that defeats the truncation rule" failure that §6 was written to prevent on the message side.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `f8f4813` (2026-05-05) "pi-loom plan: resolve \"Panic routing has no defined path for unexpected interpreter exceptions\"" — introduced the `loom/runtime/internal-error` row whose `hint` reads `error.stack` guarded only by `"<no stack available>"` when `error.stack` is falsy. No coercion existed anywhere at this point.
- `bf85092` (2026-06-06) "pi-loom spec: resolve \"non-Error-throw coercion + session_start pi.getCommands() failure contract\"" — introduced the §6 underlying-error coercion that rigorously handles non-Error throws (undefined / non-string / throwing) for the *message* read and the enumerated message-bearing fields, but did not extend that discipline to the `.stack` read.

**History:** `git log -G 'for operator triage'` and `git log -S 'error.stack'` over `code-registry-runtime.md` (and its pre-split `docs/spec` ancestry) place the falsy-only `error.stack` read at the row's inception in `f8f4813`. `git show bf85092 -- code-registry-runtime.md` shows that commit added the §6 cross-reference preamble (message coercion) without touching the `.stack` read. The defect was latent from `f8f4813` and became a visible asymmetry once `bf85092` hardened the message path while leaving the stack path on the original falsy-only guard.

## Solution Space

**Shape:** single

### Recommendation

Extend the `loom/runtime/internal-error` row's `.stack` handling to apply the §6 underlying-error coercion discipline to the `error.stack` read, replacing the falsy-only guard. Specify a guarded read that yields `"<no stack available>"` whenever the read does not produce a non-empty string: (a) the property access itself throws (throwing getter), (b) the value is a truthy non-string, or (c) the value is falsy (the case already covered). Phrase it so the read can never throw during diagnostic emission and the `hint` is always a string. Cite `placeholder-rendering-b.md#underlying-error-coercion` as the model so the stack read and the message read share one coercion posture.

Edge cases for the implementer: the access must be wrapped before any truthiness test (so a throwing getter cannot escape); a truthy non-string `.stack` resolves to the `"<no stack available>"` sentinel rather than being `String()`-coerced into an opaque blob, keeping the `hint` operator-meaningful; this is the only `.stack` read in the registry, so the change is confined to this one row.

## Consequence

**Severity:** correctness

A hostile or merely non-Error thrown value with a throwing or non-string `.stack` defeats the falsy-only guard: the throwing-getter case raises a synchronous exception while emitting the last-resort defect diagnostic (the throw escapes the emission path it was meant to report), and the non-string case carries a non-string `hint`. Two reasonable implementers diverge — one mirrors the §6 coercion onto the `.stack` read, the other implements only the literal falsy check the prose states.
