## Finding

`query-tool-loop.md` carries one normative worked-example transcript ("Worked example: depth-6 forced respond at `max_rounds`") and two normative dispatch rules whose boundary behaviour is pinned only in prose. Neither the structurally-different `max_rounds: 0` arm nor the forced-respond right-trim rule has a normative input/output vector to assert against.

**`max_rounds: 0` boundary.** The depth-6 vector uses `tool_loop.max_rounds: 2`, exercises two free-phase rounds, and reaches the forced respond turn via CIO-4's `max_rounds`-final branch firing *after* the second round's slot increment (`2 == 2`). The `max_rounds: 0` case is pinned only in prose, in three coupled places: step 2 of *Typed queries are tool-loop-shaped*, the *Tool-call loop bound* paragraph, and the closing sentence of the depth-6 example (with `max_rounds: 0`, `masked` is omitted on the surfaced event). There is no normative vector pinning what an implementer must emit on this path: exactly one provider call whose user-message body is `<rendered-prompt> + U+000A + <forced-respond-instruction>` with the lowered schema interpolated into the `<schema-json>` placeholder, the CIO-4 `slot_count == max_rounds` evaluation at initialisation, the absence of any free-phase provider call, and the `masked`-omission on a `validation` event raised at that turn. A conformant implementation that issued a spurious zero-tool free-phase call before the forced respond turn — semantically wrong but observationally subtle — would still satisfy every prose MUST and pass the depth-6 vector. `V13c-T` cites "[`max_rounds:0` boundary]" but has nothing normative behind it.

**Right-trim rule.** The "Forced respond turn" step introduces a normative right-trim rule: "The runtime right-trims any trailing U+000A bytes the rendered prompt text carries before inserting this separator … keeping the prompt-to-instruction gap exactly one line feed in every case." The rule applies on *every* typed-query provider call and is the load-bearing guarantee that an author who writes one or more trailing blank lines inside the `@<T>` template still produces a single-U+000A gap before the schema-bearing instruction sentence. It has no normative input/output vector. The companion file `query-forms.md` pins eight rendered-text vectors for the dedent/newline-trim normalisations, but the right-trim rule sits in a different phase (typed-query dispatch, after template render) and is covered by none of them, nor by the depth-6 example, nor by the `max_rounds: 0` paragraph. An implementation that omits the right-trim, trims a single U+000A instead of all of them, or trims any whitespace rather than only U+000A is not caught by any cited vector.

## Solution Space

**Shape:** single
**State:** shaped

### Recommendation

Add two normative input/output vectors to `query-tool-loop.md`, both observable at the provider boundary.

**`max_rounds: 0` worked example.** Add a second worked-example transcript, sibling to the depth-6 vector, titled *Worked example: `max_rounds: 0` boundary* (with an anchor). Pin: a minimal typed query with `tool_loop: { max_rounds: 0 }`, `respond_repair: { attempts: 0 }`, an empty `tools:` list, and a one-line `@<T>`...`` body whose rendered post-trim form is a single short ASCII string; the slot-accounting precondition that at typed-query start `slot_count == max_rounds` (`0 == 0`) so CIO-4 fires at initialisation and the forced respond turn is the first and only turn with no free-phase provider call; a mock provider transcript with exactly one provider call showing its user-message body byte-for-byte as `<rendered-prompt> + U+000A + <forced-respond-instruction>` and `tool_choice` forced to `__loom_respond_<slug>`; a success arm (`Ok(<validated value>)`, no `validation` event); and a failure arm mirroring the depth-6 `Err(QueryError { kind: "validation", cause: "schema_validation", attempts: 0, ... })` with the `RuntimeEvent` payload's `masked` field **omitted** (per PIC-1's V1 reachable predicate — no free-phase round ran). Add the new vector to the citations carried by the depth-6 example so `V13c-T`'s `max_rounds:0` boundary test has a normative anchor.

**Right-trim vector table.** Immediately after the right-trim sentence in step 2, add a short normative input/output vector table in the same shape as the eight `query-forms.md` vectors: input rendered prompt text → forced-respond user-message body, separated from the [Initial forced respond turn template](#initial-forced-respond-turn-template-normative) by a single U+000A. At minimum three rows — zero trailing U+000A (no-op), exactly one (trim one), two or more (trim all) — plus a fourth row covering a trailing `\t` or trailing ASCII space *after* a U+000A pinning that the rule trims only U+000A bytes, not whitespace generally. Cite the new vectors from the `Tests` bullets of `V13c` and `V13c-T`.

Implementer edge cases: both vectors must use the spec's post-trim / post-dedent / post-interpolation prompt form, not the source-text `@<T>`...`` form; vector wording must say "rendered prompt text" to avoid suggesting a parse-time edit. The interaction with `query-forms.md` vector 8 is observable — an author who writes `\n  ` before the closing backtick produces a rendered text ending in `\n`, which the dispatch-time right-trim then removes — pin this composition. The two vectors may share one worked-example block.

## Consequence

**Severity:** correctness

Two implementers can both pass the depth-6 vector and the prose MUSTs while diverging on the `max_rounds: 0` provider-call shape (one call vs. an empty-tool free-phase call followed by the forced respond turn), on whether CIO-4 is evaluated at initialisation, on whether `masked` is emitted on a `validation` event raised at that turn, and on the right-trim of the prompt↔instruction separator. The right-trim divergence is observable in the exact case the rule was added to close, so the provider sees a different user-turn body across conforming implementations. `V13c-T`'s `max_rounds:0` boundary test has no normative target to assert against.
