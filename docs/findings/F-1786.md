## Finding

Several spec sites cross-link into the §6 coercion paragraph of `docs/spec_topics/diagnostics/placeholder-rendering-b.md` using the noun-phrase link text "underlying-error string" — e.g. `code-registry-runtime.md`'s preamble (`"that string is the throw's [underlying-error string](placeholder-rendering-b.md#underlying-error-coercion)"`), the `loom/load/extension-bootstrap-failed` row in `code-registry-load.md`, the `loom/host/session-shutdown-teardown-step-failed` row in `code-registry-host.md`, `capability-probe.md`'s self-failure paragraph (`details.cause`), `extension-bootstrap-and-per-loom.md`'s wrap-up paragraph (`details.error`), and the outcome-routing summary in `host-interfaces-core.md` (`message: <underlying-error string truncated>`). The link target is the `#underlying-error-coercion` anchor, whose bold heading is **Underlying-error coercion** and whose prose names the result variously as "the underlying string" — the phrase "underlying-error string" never appears at the destination.

The link therefore works (it lands at the correct paragraph), but a reader following it finds a procedure named *coercion* whose own prose never uses the noun phrase the inbound link promised. There is no equivalence sentence at the destination establishing that "the underlying-error string" *is* the result of the coercion, so the inbound references read as introducing terminology the destination does not own.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `bf85092` — *pi-loom spec: resolve "non-Error-throw coercion + session_start pi.getCommands() failure contract"* (2026-06-06)
**History:** Commit `bf85092` simultaneously introduced the `#underlying-error-coercion` anchor and the §6 paragraph at the destination (whose prose uses "the underlying string") and the inbound noun-phrase link text "underlying-error string" at every consumer site that uses it today (`code-registry-runtime.md` preamble, `code-registry-load.md` row, `code-registry-host.md` row, `host-interfaces-core.md` outcome-routing summary, `extension-bootstrap-and-per-loom.md` wrap-up paragraph). The later commit `62d49d6` (2026-06-22) extended the producer-site enumeration in §6 to cover `details.cause` and added the same noun-phrase link text at `capability-probe.md`'s self-failure paragraph, preserving the mismatch rather than introducing it. The defect is therefore present-since-inception of the coercion paragraph itself: the name mismatch between the inbound link text and the destination heading/body has existed in every commit the §6 anchor has.

## Solution Space

**Shape:** single

### Recommendation

Add a single equivalence sentence to the §6 *Underlying-error coercion* paragraph in `docs/spec_topics/diagnostics/placeholder-rendering-b.md` immediately after the two-step coercion list, defining "the underlying-error string" as the string the coercion produces — so the inbound link text matches a defined term at the destination. Concrete wording (one sentence, slotted before the existing "The truncation rule below then operates on this guaranteed string …" sentence): *"The string this coercion yields is what the rest of the spec refers to as the **underlying-error string**."* Once that equivalence is in place, the inbound references read coherently: each `[underlying-error string](placeholder-rendering-b.md#underlying-error-coercion)` link lands at a paragraph that defines the noun phrase being clicked.

Do not rename the anchor (`#underlying-error-coercion` is a stable target with many inbound references, and the procedure–result distinction the two names express is real — *coercion* is the verb-noun the destination paragraph performs, *underlying-error string* is the result it produces). Do not rewrite the inbound references either — their noun-phrase form ("the throw's underlying-error string", `<underlying-error string truncated>`) reads naturally at every consumer site. The fix is solely the destination-side equivalence sentence.

## Consequence

**Severity:** cosmetic

The cross-references resolve to the correct anchor, so no implementer is misrouted; the harm is comprehension friction at the seam between the coercion procedure and the noun phrase its consumers use to name its output. An implementer scanning the §6 destination for the term they followed in from will not find it, and may double-check elsewhere before concluding that "underlying-error string" and "underlying string" refer to the same value.
