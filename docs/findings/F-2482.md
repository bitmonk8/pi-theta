## Finding

`diagnostic-shape.md` tells the `loom-system-note` renderer it MUST switch on which `details` key is present (`details.diagnostics` vs `details.event`) and MUST NOT assume both. To implement that switch the renderer author needs two things that do **not** live in the diagnostics shard: (a) the `RuntimeEvent` payload shape, and (b) the membership of the always-log **Group A** set whose members route through `details: { event: RuntimeEvent }`. Both are in fact pinned normatively in the Pi Integration Contract — the `type RuntimeEvent = { … }` block and the explicit `Group A — details: { event: RuntimeEvent }` enumeration both live in `pi-integration-contract/runtime-event-channel.md`, along with the four-shape key-switch enumeration anchored at `#system-note-details-shapes`. So the substance the renderer needs exists and is unambiguous.

The defect is the cross-link, not a missing definition. The diagnostics-shard sentence resolves the dependency only as *"specified under 'System notes' and 'Runtime event channel' in [Pi Integration Contract](../pi-integration-contract.md)"*. That target is the PIC **table-of-contents page**: `pi-integration-contract.md` is a 30-odd-entry index that contains none of the named content — "System notes", "Runtime event channel", the Group-A list, and the `RuntimeEvent` type all sit one hop further, in the `runtime-event-channel.md` shard, where "System notes" and "Runtime event channel" are bold inline labels with no heading anchors and the Group-A enumeration and `RuntimeEvent` type block carry no `<a id>` either. A renderer author following the link lands on an index, must guess the correct child page, then scan it for two bold labels. The key-switch obligation is asserted in the diagnostics shard with no precise pointer to the four-shape key enumeration or to the two artefacts (`RuntimeEvent` shape, Group-A set) that make the switch implementable.

The fix is to repoint the reference at the owning shard and its anchors, and to give the two currently-anchorless targets stable anchors so the cross-link can be precise.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04, sharding commit)
**History:** Before f5e89f4 the diagnostics content lived in `docs/spec_topics/diagnostics.md` and the sentence linked `[Pi Integration Contract](./pi-integration-contract.md)`, which at that time resolved to a single content page that actually contained the "System notes" and "Runtime event channel" sections. The f5e89f4 size-cap commit simultaneously split the diagnostics page into `diagnostics/diagnostic-shape.md` and split `pi-integration-contract.md` into 30 child shards — relocating those two sections (and the `RuntimeEvent` type / Group-A enumeration) into `pi-integration-contract/runtime-event-channel.md` while turning the parent into a table-of-contents aggregator. The link text was not updated to target the new shard or its anchors, so the cross-reference that was precise pre-split became a pointer to an index from that commit onward.

## Solution Space

**Shape:** single

### Recommendation

In `diagnostic-shape.md`, replace the bare `[Pi Integration Contract](../pi-integration-contract.md)` reference in the key-switch sentence with explicit links into the owning shard `pi-integration-contract/runtime-event-channel.md`:

- the four-shape `details`-key enumeration at its existing anchor `#system-note-details-shapes` (the authoritative key-switch table the renderer keys off);
- the "Runtime event channel" section's **Group A** enumeration (the membership that routes through `details: { event: RuntimeEvent }`);
- the `type RuntimeEvent = { … }` payload-shape block.

Add stable `<a id>` anchors in `runtime-event-channel.md` for the Group-A enumeration and the `RuntimeEvent` type block (the four-shape enumeration already has `#system-note-details-shapes`), and target them from the diagnostics-shard links so the pointer is anchor-precise rather than page-level. Keep the dependency by cross-reference — the diagnostics shard should not duplicate the `RuntimeEvent` shape or Group-A list, since PIC's `runtime-event-channel.md` is their normative owner. Edge case for the implementer: the diagnostics-shard sentence also names "System notes" (the call-shape block) — that label likewise lives in `runtime-event-channel.md`, so repoint it to the same shard rather than leaving it on the aggregator.

## Consequence

**Severity:** advisory

A renderer implementer can still reach the `RuntimeEvent` shape and Group-A set — they are fully pinned in PIC — but only by traversing a table-of-contents index and scanning a sibling page for two anchorless bold labels. The dependency is recoverable, so no two implementers diverge on behaviour; the cost is navigation friction and a cross-link that silently rotted when the monolithic PIC page was sharded.
