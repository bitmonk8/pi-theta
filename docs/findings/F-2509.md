## Finding

The indexed-access bullet in `expressions.md` (Supported forms) fully specifies the index-type contract for the **object** branch but leaves the **array** branch's index type unconstrained. For an object receiver the index expression "must be of type `string`"; a non-`string` index (e.g. `obj[0]`) is a parse-time `loom/parse/non-string-object-index`. For an `array<T>` receiver, the only specified dispositions are `loom/parse/non-indexable-receiver` (which fires on the *receiver* type, not the index) and the runtime `loom/runtime/index-out-of-bounds` panic (which only covers `i < 0` or `i >= arr.length`). Nothing states what type an array index expression must be.

Consequently, three forms have no defined disposition: `arr["x"]` and `arr[true]` reach a valid (array) receiver, so `non-indexable-receiver` does not fire and `non-string-object-index` is object-only — no parse diagnostic is named. `arr[1.5]` (a `number`-typed, non-`integer` index) is within bounds yet non-integral, and the out-of-bounds panic does not cover it. The array method signatures already treat indices as `integer` (`slice(start: integer, …)`, `indexOf(…): integer`), so the intended contract is an `integer` index, but the indexed-access rule never says so and names no diagnostic for the violation.

The defect is the missing branch of an otherwise symmetric rule: the object branch carries both a parse-time index-type check and an explicit read-order statement ("parse-time key-type check first, then the runtime missing-key panic"); the array branch carries neither.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** d6f17ca ("pi-loom spec: resolve \"indexed-access static semantics and runtime disposition\"", 2026-06-07)
**History:** Before d6f17ca the indexed-access bullet specified only the receiver-type rule (`loom/parse/non-indexable-receiver`) for both branches, with no index-type contract on either side — a symmetric, low-salience gap. d6f17ca elaborated the object branch (adding `loom/parse/non-string-object-index`, the static result type, the runtime `missing-object-key` panic, and the explicit parse-then-runtime read order) and added the array `index-out-of-bounds` runtime panic, but did not add a symmetric array index-type rule. That commit is therefore where the present asymmetry — object branch fully specified, array branch missing its parse-time index-type check and read-order statement — entered the corpus. No later commit touching `expressions.md` (through current HEAD) has closed it.

## Solution Space

**Shape:** single

### Recommendation

Add a parse-time array index-type rule symmetric to the object branch. Require the index expression of an `array<T>` receiver to be statically of type `integer`; a non-`integer` index (`arr["x"]`, `arr[true]`, `arr[1.5]`) is `loom/parse/non-integer-array-index`. Register the new code in `code-registry-parse.md` alongside `non-string-object-index` (severity E, phase `type`, owner `[Expressions — Supported forms]`, message of the shape `array index must be integer; got <type>`). In the `expressions.md` bullet, state the read order for the array branch mirroring the object branch: the parse-time index-type check fires first, then the runtime `loom/runtime/index-out-of-bounds` panic for an in-type-but-out-of-range index.

Requiring `integer` (not `number`) matches the existing `integer`-typed array-method signatures and closes the runtime `arr[1.5]` case entirely — a statically-`integer` index, enforced inbound by validation, cannot be `1.5` at runtime, so no new runtime diagnostic is needed. Edge case the implementer must keep: a valid `integer` index that is negative or `>= arr.length` (e.g. `arr[-1]`) remains type-valid and is still handled by the existing `index-out-of-bounds` panic — do not let the new parse rule reject negative integer literals.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one rejects `arr["x"]` / `arr[true]` / `arr[1.5]` at parse time, another admits them and produces host-defined runtime behaviour (JS string-keyed property access, `undefined`, or a coerced lookup). The asymmetry also means the only typed-index guarantee that closes the runtime non-integer case is unstated, so `arr[1.5]` has no specified outcome at all.
