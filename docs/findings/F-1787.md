## Finding

The "Untyped return type (loom 1.0)" subsection in `docs/spec_topics/query/query-tool-loop.md` opens its only body paragraph with authoring-time rationale — "loom 1.0 deliberately keeps it as `string` to minimise surface area; freezing a richer structure before real provider integration would lock in details that real-world use is likely to revise" — before reaching the normative content (the `Ok(string)` shape and the cross-link to the extraction mechanism in `pi-integration-contract/conversation-drive.md`). The rationale carries no obligation an implementer or test author can act on.

Beyond being non-actionable, the phrasing "real-world use is likely to revise" is stronger than the spec's own disposition. The future-widening disposition lives at `future-considerations/model-changes-and-non-goals.md` under the heading "Model-level changes (**no loom 1.0 seam expected**)": the change is anticipated *post-loom 1.0*, not within it. "Likely to revise" reads as in-lifetime instability — a `string` today, possibly something else in a later loom 1.x point release — which is in tension with the source-language stability promise at `governance/source-language-stability.md#gov-15` (GOV-15) that a `.loom` file loading cleanly under loom 1.0.0 produces identical return values under every loom 1.x release. The actual position is the opposite of what the prose implies: the shape is frozen for loom 1.x and only widens at a major-version boundary.

## Solution Space

**Shape:** single

### Recommendation

Delete the rationale clause and route the reader to the existing disposition. Replace the sentence

> loom 1.0 deliberately keeps it as `string` to minimise surface area; freezing a richer structure before real provider integration would lock in details that real-world use is likely to revise. See [Future Considerations](../future-considerations.md).

with the shorter, disposition-matching:

> The shape is frozen for loom 1.x; widening to a richer return type (e.g. `Result<string | AssistantMessage, QueryError>` exposing tool-use traces, multiple content blocks, citations) is a post-loom 1.0 model-level change with no loom 1.0 seam expected — see [Future Considerations — Model-level changes](../future-considerations/model-changes-and-non-goals.md).

Edge cases the implementer must watch:

- The cross-link must be deepened from `../future-considerations.md` (the hub) to `../future-considerations/model-changes-and-non-goals.md` (the shard owning the "Richer untyped-query return shape" bullet) — the bare hub link is the same kind of TOC-not-rule miss flagged elsewhere in this review.
- The "no loom 1.0 seam expected" phrasing must be quoted exactly from the section heading at `future-considerations/model-changes-and-non-goals.md#model-level-changes-no-loom-1-0-seam-expected` so future grep-driven audits stay aligned.
- The neighbouring normative sentence pinning the extraction mechanism (`pi-integration-contract/conversation-drive.md#untyped-query-ok-extraction`, symmetric across prompt mode and subagent mode) is untouched and remains the substantive content of the subsection.
- GOV-15 itself is not cited in the replacement — the disposition link is sufficient and avoids introducing a second cross-reference into a non-normative paragraph.

## Consequence

**Severity:** cosmetic

A close reader hits a contradiction between "real-world use is likely to revise" and GOV-15's loom 1.x stability promise and must reconcile them before trusting either. Implementers and tests are not affected — the normative `Ok(string)` shape and its extraction mechanism are unambiguous. The fix is a pure prose edit.
