## Finding

The closing sentence of the `array<T>` literal type-sink rule in `grammar.md` reads:

> The iterand of a `for x in expr` is **not** a sink — `for` cannot supply `T` to `[]`. `for x in []` with no other sink is `loom/parse/array-no-common-type`, the same diagnostic that `let xs = []` raises in unannotated position. This is the same hole; resist any `for`-specific carve-out.

Two wording defects sit in the trailing clause:

(a) **"the same hole" is undefined jargon.** "hole" is not a defined term in the spec — the glossary defines `type sink`, not any "hole" / "gap" concept — and the referent is left implicit. The intended meaning is "the same missing-sink condition that `let xs = []` hits in unannotated position", but a first-time reader has to reconstruct that.

(b) **"resist any `for`-specific carve-out" is reviewer-advisory, not normative.** It addresses a future spec author ("don't be tempted to special-case `for`") rather than stating an implementer obligation. The substantive rule it is gesturing at — that the `for` iterand is not a member of the exhaustive sink set and that an empty literal in that position fires `loom/parse/array-no-common-type` — is already stated normatively in the two preceding sentences, so the clause adds only editorial emphasis in advisory voice.

The underlying behaviour is correct and unambiguous; the issue is confined to the trailing clause's vocabulary and register. The fix is a single rewrite of that clause to drop the "hole" jargon and replace the advisory instruction with a normative statement (the iterand MUST NOT be treated as a type sink; no `for`-specific exception to the exhaustive sink set is admitted).

## Solution Space

**Shape:** single

### Recommendation

Rewrite the trailing clause of the rule so it (a) drops "hole" in favour of a plain description of the missing-sink condition and (b) states the carve-out as a normative prohibition rather than advice. For example:

> The iterand of a `for x in expr` is **not** a sink — `for` cannot supply `T` to `[]`. An empty array literal reaching a `for`-iterand position with no other sink in scope is therefore `loom/parse/array-no-common-type`, exactly as `let xs = []` is in unannotated position. The `for` iterand MUST NOT be treated as a type sink; the exhaustive sink set above admits no `for`-specific exception.

Keep the existing first two sentences and the exhaustive sink list intact — only the final "This is the same hole; resist any `for`-specific carve-out." sentence changes. Edge case for the implementer: the rewrite must not imply a new diagnostic; the only diagnostic remains `loom/parse/array-no-common-type`.

## Consequence

**Severity:** cosmetic

A reader must reconstruct what "hole" refers to, and the advisory register obscures that the carve-out prohibition is a real rule. No implementer diverges in behaviour: the preceding sentences already pin the diagnostic and the not-a-sink rule, so the defect is wording and register only.
