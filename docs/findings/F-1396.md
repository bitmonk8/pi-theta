## Problem

The `toolCallId` bullet at `host-interfaces-core.md` `#tool-execution-from-loom-code` pins only the `loom-direct:` prefix of the synthesised UUID handed to a Pi tool's `execute()`, leaving two observable conformance properties unspecified: the UUID body format (version, lettercase, canonical rendering) and the generation path. The §7 placeholder convention pins `<invocation-id>` / `<uuid>` to canonical lowercase 8-4-4-4-12 hex, but the `toolCallId` bullet never cites it, so conforming implementations may emit uppercase, braced, or non-v4 bodies and still observe the spec. PIC-20's MUST / MUST-NOT routing-through-`IdSource` ban is scoped by its own wording to `invocationId` only, leaving an implementer free to call `crypto.randomUUID()` directly at the `toolCallId` dispatch site and defeat PIC-20's deterministic-test rationale for an identifier that crosses the same Pi/loom boundary.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** fae85d3 (`2026-05-04` "Tighten spec for implementation-plan readiness"); e9b0597 (`2026-06-05` "pi-loom spec: resolve \"invocationId has no injectable id-source seam\"")
**History:** `git log --reverse -S'loom-direct'` on `spec.md` and the post-split `docs/spec_topics/pi-integration-contract/host-interfaces-core.md` traces the bullet *"`toolCallId` is a synthesised UUID prefixed `loom-direct:`."* back to `fae85d3`, which introduced the "Tool execution from loom code" surface verbatim in `spec.md` and has carried the same wording — UUID body and generation path unpinned — through every subsequent edit (split in `fecb504`, relocation in `31ff060`, size-cap reshape in `f5e89f4`). The conformance gap only became sharp when `e9b0597` later added PIC-20 (`docs/spec_topics/pi-integration-contract/host-interfaces-services.md` §*`IdSource` seam*) with a MUST/MUST-NOT framed narrowly around `invocationId`: that commit established a runtime-wide ban on ambient `crypto.randomUUID` *for invocationId* but did not co-edit the `toolCallId` line one file over, leaving the dispatch site outside PIC-20's stated scope. The defect is the interaction — a pre-existing under-specified line on one page, plus a later seam on a sibling page whose explicit scope leaves the line untouched — rather than either commit's own diff in isolation.

## Solution approach

Rewrite the `toolCallId` bullet at `host-interfaces-core.md` `#tool-execution-from-loom-code` to pin its body to the canonical lowercase 8-4-4-4-12 hex form (forward-linking the §7 placeholder convention at `placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders`) and to state that the UUID is minted through the PIC-20 `IdSource` seam, fresh per code-side tool call. Broaden PIC-20's normative sentence at `host-interfaces-services.md` `#pic-20` so its MUST / MUST-NOT covers the `loom-direct:` `toolCallId` minting site alongside `invocationId`.
