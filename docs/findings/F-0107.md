## Finding

`V16a-T`'s Adds scopes the leaf to "synthesised ceiling-candidates driven through the cross-ceiling arbitration seam, asserting its `{ surfaced, masked }` output." Four of its six `CIO-N` Tests bullets honour that scope: CIO-1, CIO-3, CIO-5, and CIO-6 each name a synthesised candidate driven through the seam and assert the seam's surfacing/masking decision. Two bullets do not:

- `CIO-2`: "ceiling #1 (`invoke` depth) is evaluated at `invoke` entry before the callee body."
- `CIO-4`: "ceiling #2 (`tool_loop.max_rounds`) is evaluated at the round boundary, post-slot-increment, pre-next-turn; `max_rounds:0` typed takes the final branch at start."

Both are spec-restating declarative sentences naming the live enforcement site (`invoke` entry, round boundary) with no reference to a synthesised candidate or the `arbitrate()` seam. The paired implementation leaf `V16a` already states the correct form for both — its CIO-2 reads "a synthesised candidate tagged as an `invoke`-entry event resolves ceiling #1…" and its CIO-4 reads "a synthesised candidate tagged as a round-boundary event resolves ceiling #2…" — so the `-T` leaf is the lagging copy.

Because the `-T` leaf is authored first under the TDD ritual, an implementer reading CIO-2/CIO-4 literally could write a test against the live `invoke` entry / round-boundary sites. Those sites are built by downstream leaves (`V15b`, `V13c`) that `V16a` does not depend on (`V16a` Deps are `V16a-T, V9d`), so they do not exist when `V16a` is picked up; such a test cannot compile at this leaf's position. The Adds and the implementation leaf both disambiguate toward the seam-only strategy, so the literal reading self-corrects on contact, but the bullet text itself points the wrong way.

**History:** `git log -L '/CIO-2/,+1:docs/plan_topics/V16a-T-ceiling-order-masked.md'` shows the CIO-2 line was introduced once, in the initial plan-build commit `c6a664e`, and never edited since. At that point all six CIO bullets shared a uniform declarative form, so no bullet stood out. `git show a6fba55` and `git show 8ef7055` on the same file show the synthesised-candidate conversion was applied piecemeal: `a6fba55` rewrote CIO-3, and `8ef7055` rewrote CIO-1 and CIO-5, each time leaving CIO-2 and CIO-4 untouched. The defect is therefore an interaction: the inconsistency that lets CIO-2/CIO-4 mislead toward live-site testing emerged when the sibling sweep converted four bullets but skipped these two, not from any single commit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V16a-T-ceiling-order-masked.md`, rewrite the CIO-2 and CIO-4 Tests bullets into the seam-driven synthesised-candidate form already used by their siblings and already present in the paired `V16a` implementation leaf, so the `-T` and implementation leaves state the same assertion:

- CIO-2 → `` `CIO-2`: a synthesised candidate tagged as an `invoke`-entry event resolves ceiling #1 (`invoke` depth) before the callee body. ``
- CIO-4 → `` `CIO-4`: a synthesised candidate tagged as a round-boundary event resolves ceiling #2 (`tool_loop.max_rounds`) post-slot-increment, pre-next-turn; `max_rounds:0` typed takes the final branch at start. ``

This keeps every CIO-N bullet expressed as a candidate driven through the `arbitrate()` seam, consistent with the leaf's Adds. No spec edit, no Deps change, and no coverage-matrix change is needed — the `CIO-2, CIO-3, CIO-4 | V16a` row already maps these obligations to `V16a`.

## Consequence

**Severity:** advisory

A `-T` author reading CIO-2/CIO-4 literally could attempt to assert against the live `invoke`-entry / round-boundary enforcement sites, which do not exist at `V16a`'s sequencing position; the test would fail to compile, forcing a reread of the Adds. No working leaf is blocked, but the inconsistent bullet form is a guidance gap that wastes an authoring cycle and undercuts the leaf's stated seam-only test strategy.
