## Problem

`V9q`'s Adds clause emits the persistent `loom/runtime/watcher-terminated` `loom-system-note` on the **stopped-delivering â€” terminal** case while declaring it "does not redefine the `FileWatcher.watch` seam shape" (`V8e`, PIC-14/16). But the only runtime channel `V9q` declares is the `V8e` `FileWatcher` seam, whose handler observes only `add`/`change`/`unlink` (PIC-14); chokidar `error` events are filtered by the adapter and routed to `ctx.ui.notify`, never reaching the runtime. There is therefore no enumerated path by which the adapter's observation of a terminal failure reaches the runtime's persistent-note emission point. Compounding this, neither PIC-55 nor PIC-14 states an operational criterion distinguishing **stopped-delivering â€” terminal** from **continues-delivering**, and the documented `FakeFileWatcher.emit(event)` injects only the three change kinds, so `V9q-T`'s PIC-55 assertion has no defined way to drive the terminal case deterministically.

## Solution approach
