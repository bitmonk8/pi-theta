## Finding

The leaf file `docs/plan_topics/V8a-checkpoint-validator-seams.md` (and its tests partner `V8a-T-checkpoint-validator-seams.md`) carries the slug component `checkpoint-validator-seams`, which advertises that V8a owns both the `Checkpoint` and `SchemaValidator` seams. The leaf body contradicts the slug: the heading reads `` `V8a` — `Checkpoint` seam ``, and the Adds explicitly disclaims the validator — "The `SchemaValidator` seam is owned by [`V8c`](./V8c-schema-validator-seam.md)." The `SchemaValidator` seam is in fact a separate leaf, `V8c` (`V8c-schema-validator-seam.md`), with its own `PIC-11` test.

The slug is therefore stale relative to the leaf's current scope. The body text is authoritative and correct, so the defect is navigational/naming hygiene: a reader locating leaves by filename, or scanning the V8 slice list, sees a path implying V8a still co-owns validation.

The plan.md link at line 104 (`[\`V8a\` — \`Checkpoint\` seam](./plan_topics/V8a-checkpoint-validator-seams.md)`) inherits the misleading path. Cross-leaf Deps citations are not affected: every other leaf references V8a by its leaf ID `V8a` (e.g. V13c, V14a, V15a, V17a, V9c, V9i), never by filename. There is no `index/`, `terms.md`, or occurrences artefact in this corpus, so the "index locator" referenced by the raw finding has no referent here.

**History:** The corpus is git-tracked. `V8a-checkpoint-validator-seams.md` was added at `c6a664e` with heading `` `V8a` — `Checkpoint` and `SchemaValidator` seams `` and an Adds covering both seams (`PIC-10` + `PIC-11`) — at that point the slug `checkpoint-validator-seams` was accurate. The over-bundling split at `cb8737a` rewrote V8a's heading to `` `Checkpoint` seam ``, narrowed its Adds to Checkpoint-only, disclaimed `SchemaValidator` to the newly-created `V8c-schema-validator-seam.md`, and moved `PIC-11` to V8c — but did not rename the V8a file pair. The slug was orphaned by that split rather than wrong at inception; the durable fix is to realign the filename with the post-split scope.

## Solution Space

**Shape:** single

### Recommendation

Rename the leaf pair to drop the orphaned `validator` component:

- `docs/plan_topics/V8a-checkpoint-validator-seams.md` → `docs/plan_topics/V8a-checkpoint-seam.md`
- `docs/plan_topics/V8a-T-checkpoint-validator-seams.md` → `docs/plan_topics/V8a-T-checkpoint-seam.md`

Update the single inbound path in `docs/plan.md` line 104, changing the link target from `./plan_topics/V8a-checkpoint-validator-seams.md` to `./plan_topics/V8a-checkpoint-seam.md` (the display text `` `V8a` — `Checkpoint` seam `` already matches and needs no change).

No Deps citations require edits: all cross-leaf references to V8a use the leaf ID `V8a`, not the filename. There is no index/occurrences artefact in this corpus to reconcile. This is a pure rename with no semantic change to the leaf body.

## Consequence

**Severity:** cosmetic

A reader navigating by filename or scanning the slice list is told V8a co-owns the `SchemaValidator` seam, which V8c actually owns. The leaf body (heading + Adds) is correct and authoritative, so no implementer would build the wrong seam; the cost is navigation friction and a misleading path propagated into the plan.md link.
