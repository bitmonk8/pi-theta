## Problem

The spec (`active-invocation-registry.md`) makes the canonical-name resolver a normative MUST: each of the three insertion sites populates the registry entry's `loom` field by reading the canonical key from the runtime's loom catalog, *not* by re-deriving it at the site from the authored identifier the call site carries. `V9e`'s Adds names the canonical-name resolver (three insertion sites) as a deliverable, but neither `V9e` nor `V9e-T` carries a Tests bullet asserting the entry's `loom` value or that it is sourced from the catalog lookup. The three present bullets cover only insertion-order iteration, `disposeBarrier` settlement, and `invocationId` allocation. An implementation that re-derives the name per site would pass every existing V9e/V9e-T assertion while violating the resolver's MUST and splitting operator dedup buckets on `details.event.loom`.

The test gap is original to the plan, not introduced by a later edit.

## Solution approach

Add a Tests bullet to `V9e-T` (mirrored as the corresponding `V9e` Tests bullet per the paired-leaf TDD ritual) asserting that the registry entry's `loom` field equals the loom-catalog canonical key for each of the three insertion arms — slash-command handler entry, `tool.execute(...)` adapter entry, and `invoke` spawn-site entry. Include at least one case where the authored identifier the call site carries differs from the canonical key, so the assertion fails against a per-site re-derivation and passes only when the value is read from the catalog. Cite `active-invocation-registry.md` (PIC area) as the bullet's source. Leave the assertion's API shape to the implementer.
