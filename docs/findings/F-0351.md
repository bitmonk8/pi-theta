## Finding

The `loom/runtime/match-error` Tests bullet in `V4a-T` first describes the runtime behaviour fully — "a value matching none of the six pattern forms raises the runtime `loom/runtime/match-error` panic … while a value matching one of the six pattern forms … binds and evaluates the selected arm" — and then summarises it with the back-reference "`V4a-T` asserts only this **raise-versus-bind exhaustion behaviour**". The phrase "raise-versus-bind exhaustion behaviour" is a compressed coinage defined nowhere; the reader has to map it back onto the preceding clause to recover its meaning.

The label is also mildly self-contradictory. The same bullet states "loom 1.0 does not statically check exhaustiveness", yet the recap names the behaviour an "exhaustion behaviour" — inviting a reader to expect an exhaustiveness check that the leaf explicitly disclaims. The actual behaviour being asserted is the runtime dispatch dichotomy (no-match → raise, match → bind-and-evaluate), which has nothing to do with static exhaustiveness.

The identical bullet appears in the paired implementation leaf `V4a` (`V4a-match-result.md`), so the coinage occurs in both files and should be fixed in lockstep.

## Issue introduction

**Verdict:** single introducing commit
**Introducing commit:** `02654f5` ("pi-loom plan: resolve \"V4a-T match-error assertion scope left undetermined by co-asserted note\"", 2026-06-17)
**History:** Before `02654f5`, the `loom/runtime/match-error` bullet ended with "`V4b` also closes `loom/runtime/match-error`; co-asserted here." — the phrase "raise-versus-bind exhaustion behaviour" did not exist. Commit `02654f5` (which resolved a prior review finding about the assertion scope being left undetermined by that "co-asserted here" note) rewrote the trailing clause in both `V4a-T-match-result.md` and `V4a-match-result.md` simultaneously, introducing the compound coinage to name the scope `V4a-T` asserts and the deferral to `V4b-T`. The coinage has been present and identical in both files since that single edit; no later commit has touched it.

## Solution Space

**Shape:** single

### Recommendation

In the `loom/runtime/match-error` Tests bullet, replace the coinage "this raise-versus-bind exhaustion behaviour" with a plain back-reference to the dispatch behaviour the bullet already describes, dropping the misleading "exhaustion" word while retaining the explicit deferral to `V4b-T`. Apply the same edit to the identical bullet in both `docs/plan_topics/V4a-T-match-result.md` (line 11) and `docs/plan_topics/V4a-match-result.md` (line 11).

Concretely, replace the clause:

> `V4a-T` asserts only this raise-versus-bind exhaustion behaviour; the panic's `?`/`match` bypass and its registered `loom/runtime/match-error` message template are deferred to and closed by `V4b-T`.

with text that names the asserted scope without coining an undefined term, e.g.:

> `V4a-T` asserts only this match-dispatch behaviour — a no-match scrutinee raises the panic, a matching arm binds and evaluates — while the panic's `?`/`match` bypass and its registered `loom/runtime/match-error` message template are deferred to and closed by `V4b-T`.

The deferral clause naming `V4b-T` as the closer of the bypass and message-template behaviour must be preserved (it is the scope boundary the prior fix established). No spec edit is needed and the leaves' acceptance criteria do not change.

## Consequence

**Severity:** cosmetic

The full behaviour is stated in the antecedent clause of the same bullet, so an implementer can still produce the correct test; the coinage costs only a re-parse. The "exhaustion" word is the sharper risk: it can lead a reader to expect a static exhaustiveness check the leaf explicitly disclaims one sentence earlier.
