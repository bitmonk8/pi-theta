## Finding

The **Unknown-reason rule** in `unknown-reason-rule.md` pins `details: { observed: <String(event.reason)> }` for `loom/host/session-shutdown-reason-unknown` and applies no length cap, no control-character escape, and no whitespace normalisation to the coerced value before it is written to `console.error`. The rule explicitly tolerates `String(event.reason)` on symbols, `undefined`, numbers, booleans, and ordinary objects "without itself throwing" — the success branch is the dominant one — but a Pi-side getter, proxy, or `Symbol.toPrimitive` whose coercion succeeds against, say, a multi-MB JSON blob or a `Buffer` (`util.inspect`-shaped output) yields an arbitrarily-large `details.observed` string. The sibling discriminator `"throw:<String(error)>"` on the **same paragraph**, defined for `loom/host/session-shutdown-pinned-constant-unreadable`'s `details.failure` field, was hardened in the same edit family with a 256-UTF-16-code-unit truncation, a `\n`/`\r`/`\t` two-character literal escape, a U+0000–U+001F / U+007F–U+009F six-character `\\uXXXX` escape, and a coercion-failure `"throw:<unreadable>"` fallback — explicitly to bound "the discriminator string [and] the teardown `console.error` write cost it carries". `details.observed`, written by the same `console.error` sink, on the same `session_shutdown` handler entry path, with the same operator-deduplication mandate (`(code, details.observed)`), carries none of those defences.

The asymmetry has two concrete consequences that the rule's text already implies. First, the **Per-invocation operator visibility (clean-cancel path)** rule pins that the handler-captured `event.reason` string "is the same single coerced value the **Unknown-reason rule** populates `details.observed` with" and is reused as `details.event.reason` by **every** `loom/runtime/cancelled-by-session-shutdown` note emitted on that shutdown — so a single pathological `event.reason` blob is teed into `console.error` once and then into the per-invocation note stream N times, multiplying the write cost across an unbounded fan-out the `"throw:"` cap was sized to prevent on a single emission. Second, the rule pins `(code, details.observed)` as the operator-side deduplication key for `session-shutdown-reason-unknown`; an unescaped CR/LF (or a control sequence that resets a terminal) in `details.observed` not only corrupts log-line parsing — it can also split the dedup bucket across normalisation-divergent consumers, defeating the at-most-one-per-`session_shutdown` invariant the **mutually exclusive** clause relies on.

The fix the rule's own structure suggests is the symmetric one: apply the same 256-UTF-16-code-unit truncation and control-character escape pinned for `"throw:<String(error)>"` to `details.observed`, and state that the bounded form is what reaches `details.event.reason` on every per-invocation note. The `"throw:"`-prefix-preservation half of the sibling defence has no analogue here (the rule already documents that `details.observed` carries no `"throw:"` prefix and falls back to the bare `"<unreadable>"` sentinel on a coercion throw), so the carry-over is the cap and the escape only, not the prefix-preserving fallback.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `9559e19` (2026-05-07, *pi-loom spec: resolve "session_shutdown unknown reason: no closed-set validation, no fail-safe"*) and `75a9626` (2026-05-09, *pi-loom spec: resolve "SessionShutdownEvent.reason closed set has no build-time pin against the SDK type"*).
**History:** `git log -S 'details: { observed' -- docs/` shows `9559e19` as the commit that introduced `details: { observed: <String(event.reason)> }` on the `loom/host/session-shutdown-reason-unknown` emission — at that point the only `details.<field>` payload in the surrounding text, so length/control-character bounds were not yet a question of asymmetry. Two days later, `75a9626` added the sibling `loom/host/session-shutdown-pinned-constant-unreadable` diagnostic with its `details.failure` `"throw:<String(error)>"` arm, and `git show 75a9626 -- docs/spec_topics/pi-integration-contract.md` shows the cap-and-escape grammar ("truncated to at most 256 UTF-16 code units … control characters in the truncated payload are escaped before concatenation") landing inline on the *same* **Unknown-reason rule** paragraph as `details.observed`, with no reciprocal edit to the earlier field. The defect is the *interaction* — `9559e19`'s `details.observed` shape, paired with `75a9626`'s sibling cap on `details.failure`, made the asymmetry visible — not either commit on its own. Subsequent edits in this region (`aa2f9d7` adding the `String(event.reason)` coercion-throw `"<unreadable>"` fallback for `details.observed`, `a9fb344` coining PIC-45..48, `8008b91` adding the acceptance-criteria block for the `"throw:"` grammar) all touched the surrounding prose and the four-arm fixture floor for `"throw:"` without revisiting the unbounded `details.observed` payload, confirming the gap is durable rather than a transient mid-edit state.

## Solution Space

**Shape:** single

### Recommendation

Edit the **Unknown-reason rule** paragraph in `docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` to pin a symmetric content bound on `details.observed`, written as a one- to two-sentence insertion immediately after the existing clause that establishes the bare `"<unreadable>"` fallback for `details.observed`:

> The `String(event.reason)` payload (on the success arm) is truncated to at most 256 UTF-16 code units (`String.prototype.length`) and control-character-escaped under the same grammar pinned later in this paragraph for the `"throw:<String(error)>"` discriminator (`\n`/`\r`/`\t` as the two-character literal escapes `\\n`/`\\r`/`\\t`; every other code point in U+0000–U+001F and U+007F–U+009F as the six-character `\\uXXXX` escape; all other characters passed through unchanged) before it is written to `details.observed`. The bounded form is the value the handler captures into the handler-scoped variable above and is therefore the same string reused as `details.event.reason` by every `loom/runtime/cancelled-by-session-shutdown` note. The `"throw:"`-prefix-preservation half of the sibling discriminator's defence has no analogue here — the bare `"<unreadable>"` fallback already pinned for the throwing-property-access and coercion-throw paths is unchanged.

Edge cases the implementer must watch:

- **Truncation boundary.** Inclusive at 256, matching the (a) *Truncation boundary* fixture obligation in the existing **Acceptance criteria — `"throw:<String(error)>"` discriminator grammar** block; the same fixture pattern can be lifted onto `details.observed`.
- **Capture-site ordering.** The bound MUST be applied *before* the handler captures the value into the handler-scoped `capturedEventReason` variable that sub-step 2 stamps onto each `ActiveInvocationRegistry` entry's `shutdownReason` field; otherwise the per-invocation `finally` would observe the bounded form for the emission but the unbounded form on the entry, breaking the "same single coerced value" invariant the rule pins.
- **`"<unreadable>"` sentinel passthrough.** The literal seven-character `"<unreadable>"` sentinel and the four pre-init / throwing-access / coercion-throw / snapshot-failure paths that emit it are already below the 256-code-unit cap and contain no control characters, so the bound is a no-op on those paths; the rule's *Sub-anchors and pinned substrings* paragraph (PIC-47 sub-anchor (iii)) and the *Lookup-failure-to-discriminator routing* paragraph need no edit beyond the source-rule insertion.
- **Acceptance-criteria coverage.** Either fold a parallel `details.observed` row into the existing acceptance-criteria block (renaming the block to cover both fields' grammar) or add a sibling acceptance-criteria block under the same H2; the first option keeps the rule's single source of truth for the cap-and-escape grammar and is the lighter edit.
- **Out-of-scope.** The build-time `loom/typecheck/session-shutdown-reason-snapshot` assertion (Pi version bump procedure step 5) operates on the snapshot's `literals` array, not on a runtime `event.reason` value, so the bound has no build-time pairing edit. Anchor-stable substrings (a)–(d) of PIC-48 are not touched (none of them mention `details.observed`'s content shape).

## Consequence

**Severity:** advisory

A Pi-side `event.reason` whose `String()` coercion succeeds against a multi-MB blob or whose value embeds CR/LF or terminal-escape control characters lets the teardown handler write a single arbitrarily-large `console.error` row *and* multiplies that write across every `loom/runtime/cancelled-by-session-shutdown` note via the captured-string reuse into `details.event.reason`; an embedded newline additionally corrupts the `(code, details.observed)` operator-dedup key and can split the at-most-one-per-`session_shutdown` invariant across normalisation-divergent log consumers. Two reasonable implementers will diverge on whether to apply the sibling `"throw:"`-arm cap-and-escape to `details.observed` by analogy or to take the rule's silence on length as license to emit the raw coerced string.
