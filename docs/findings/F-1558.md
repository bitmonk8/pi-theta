## Finding

DISC-5's minimatch-parity sentence locates "the same package Pi applies to its own resource arrays" by naming a bundler-output path and symbol inside `@earendil-works/pi-coding-agent` â€” `dist/core/package-manager.js`, symbol `applyPatterns`. That locator is a Pi build artefact, not part of Pi's public API: the file name, directory layout under `dist/`, and the exported symbol name can change between Pi minor versions (a bundler change, a refactor that renames the internal helper, or a code-splitting change) without any change to minimatch's glob dialect, which is what loom actually depends on.

The load-bearing claim of the paragraph is already carried by two adjacent statements: (i) "matched with the `minimatch` engine", which names the package by public identity, and (ii) "this paragraph pins loom's *use* of that matching, not minimatch's glob dialect, which the package owns", which delegates dialect semantics to minimatch itself. The internal-symbol citation adds no constraint a conforming implementer can act on, and creates a maintenance hazard: when Pi reorganises its bundle the citation goes stale, and a reader who chases it can't find what it points to even though the actual contract (use minimatch; dialect parity owed to Pi via the shared package) is unaffected.

## Solution Space

### Recommendation

Edit the DISC-5 minimatch sentence to drop the `(`dist/core/package-manager.js`, symbol `applyPatterns`)` parenthetical. Replace the run "the same package Pi applies to its own resource arrays in `@earendil-works/pi-coding-agent` (`dist/core/package-manager.js`, symbol `applyPatterns`)" with "the same package Pi applies to its own resource arrays in `@earendil-works/pi-coding-agent`". The adjacent dialect-delegation clause ("this paragraph pins loom's *use* of that matching, not minimatch's glob dialect, which the package owns") already pins the public dependency and is left intact.
