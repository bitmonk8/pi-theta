## Finding

`V10c` (Settings reads and merge) declares in its `Adds` that it implements "the settings-source reads (the five keys), the deep-merge precedence …, and validation." The spec page `discovery/package-and-settings.md` defines that validation concretely: the **Scalar-key validation** rule requires each recognised `looms.*` scalar key whose JSON value fails its declared type/range to be treated as absent and to log exactly one `loom/load/settings-value-out-of-range` diagnostic per offending key per file (acceptance sets: `looms.binderModel` non-empty string; `looms.scanPackages` boolean; `looms.scanPackagesMaxFiles` / `looms.scanPackagesTimeoutMs` integer ≥ 1, `null` out of range for all). The adjacent **Top-level shape validation** rule emits the same `settings-value-out-of-range` code for a malformed top-level `loomPaths` / `looms` key, and the `loomPaths` entry schema emits `loom/load/settings-invalid-entry` for a non-string array entry.

`V10c`'s `Tests` (and the paired `V10c-T` failing-test task) assert only DISC-7 (deep-merge precedence) and `loom/load/settings-invalid-json`. No test supplies a wrong-typed or out-of-range value for any of the five keys and asserts the documented absent-fallback plus the `settings-value-out-of-range` (or `settings-invalid-entry`) diagnostic. The `settings-value-out-of-range` code does not appear in any plan-topics file.

Because the per-phase TDD ritual makes a leaf shippable on its observable `Ships when` (deep-merge + a malformed-settings diagnostic), the validation behaviour named in `Adds` can ship absent or wrong while every gate stays green — the "No silent test skipping" failure mode the plan forbids.

## Solution Space

**Shape:** single

### Recommendation

Add validation test coverage to the paired `V10c-T` (authored first, must fail red) and `V10c` (`Ships when` / `Tests` must observe it) for the per-key settings validation the spec defines in `discovery/package-and-settings.md`. The added tests must, for each validated key, supply a wrong-typed or out-of-range value and assert both the documented absent-fallback (built-in default for `scanPackages` / `scanPackagesMaxFiles` / `scanPackagesTimeoutMs`; `bind_model:` resolution fallback for `binderModel`) and the emission of `loom/load/settings-value-out-of-range` (severity `error`, non-fatal). Cover the acceptance sets the spec pins: `looms.binderModel` non-empty string; `looms.scanPackages` JSON `true`/`false`; `looms.scanPackagesMaxFiles` and `looms.scanPackagesTimeoutMs` integer ≥ 1 (with `null` out of range for every key, and `2000.0` accepted while `25.5` is rejected). Also assert the top-level shape rule (a malformed `loomPaths`/`looms` value logs exactly one `settings-value-out-of-range` per malformed top-level key per file, and a malformed `looms` does not additionally log one per nested key) and the `loomPaths` non-string-entry rejection via `loom/load/settings-invalid-entry`. Extend `V10c`'s `Ships when` so the malformed-value validation, not only the malformed-JSON diagnostic, is part of the observable acceptance condition.

The validation is purely plan-internal — the spec already defines the codes and acceptance sets; no spec edit is required, and the spec page is read-only for this fix.

## Consequence

**Severity:** correctness

The settings-value validation the spec mandates (per-key type/range rejection with `loom/load/settings-value-out-of-range`, plus `settings-invalid-entry` for non-string `loomPaths` entries) is named in V10c's `Adds` but forced by no test, so two reasonable implementers could ship V10c with the validation missing, partial, or emitting the wrong code/severity, and the leaf's gates would still pass green.
