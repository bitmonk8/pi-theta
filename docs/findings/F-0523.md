## Finding

Several `Tests` bullets carry corpus-governance classification parentheticals that restate the GOV-22 un-anchored-obligation classification already owned by `coverage-matrix.md`:

- `V9b` / `V9b-T` — three bullets tagged `(PIC area; un-anchored GOV-22 residue; facet cka-23.*)` (`cka-23.getcommands`, `cka-23.note-empty`, `cka-23.note-rename`), plus the registry-swap bullet tagged `(PIC area; facet cka-23.swap)`.
- `V9f` — the `ToolDefinition.label` derivation bullet tagged `(GOV-22 un-anchored residue)`.
- `V9l` / `V9l-T` — all four `session-only-degraded-state.md` bullets tagged `(un-anchored; GOV-22 residue)`.

The GOV-22 classification of each underlying obligation is authoritative in `coverage-matrix.md` §Code-keyed obligation areas: `cka-17` → `V9f`, `cka-23` → `V9b`, `cka-27` → `V9l`, each already labelled `(un-anchored; GOV-22 residue)` against its named closing leaf. The inline copies in the leaf `Tests` bullets add no implementation guidance — the spec link in each bullet already names the normative source — and become a divergence surface if the matrix classification changes. The duplication is applied inconsistently: `V9f` carries the tag but its `V9f-T` partner does not, which can mislead a reader about whether the classification is normatively significant.

One element inside the `V9b`/`V9b-T` parentheticals is **not** cruft and must survive: the `facet cka-23.*` sub-labels. The `cka-23` row in `coverage-matrix.md` defines those facet sub-labels as "cited inline by the matching `V9b` / `V9b-T` *Tests* bullet" — the matrix relies on those tokens being present in the leaf to partition the three independently-verifiable obligation areas of `cka-23`. Only the `PIC area; un-anchored GOV-22 residue` classification prose is redundant; the facet sub-label is load-bearing.

## Solution Space

**Shape:** single

### Recommendation

Strike the redundant GOV-22 classification prose from the affected `Tests` bullets; the spec link in each bullet already names the normative source and `coverage-matrix.md` is the authoritative classification home. `coverage-matrix.md` is read-only for this fix — do not migrate the classification there or alter the cka rows. No spec edit; the change is internal to the plan leaf files.

Per file:

- `docs/plan_topics/V9f-tool-registration-lifetime.md` — in the `ToolDefinition.label` derivation `Tests` bullet, delete the ` (GOV-22 un-anchored residue)` parenthetical. This also makes the bullet consistent with its already-untagged `V9f-T` partner.
- `docs/plan_topics/V9l-session-only-degraded-branch.md` and `docs/plan_topics/V9l-T-session-only-degraded-branch.md` — in each of the four `session-only-degraded-state.md` `Tests` bullets, delete the ` (un-anchored; GOV-22 residue)` parenthetical.
- `docs/plan_topics/V9b-registration-reload-wiring.md` and `docs/plan_topics/V9b-T-registration-reload-wiring.md` — remove the GOV-22 classification text but **retain the `facet cka-23.*` sub-label token** in each bullet, because `coverage-matrix.md`'s `cka-23` row cites those sub-labels inline. Concretely: replace `(PIC area; un-anchored GOV-22 residue; facet cka-23.getcommands)` with `(facet cka-23.getcommands)`, and likewise for `cka-23.note-empty` and `cka-23.note-rename`; replace `(PIC area; facet cka-23.swap)` with `(facet cka-23.swap)`.

Edge case for the implementer: the `cka-23.*` facet sub-labels are the only part of the V9b/V9b-T parentheticals the matrix depends on — deleting the whole parenthetical (as a naive reading of "delete the GOV-22 parentheticals" suggests) would break the `cka-23` facet citation. Preserve those sub-labels.
