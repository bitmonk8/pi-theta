## Finding

Two closing-gate reconciliation arms read the test sources to certify that a test exists: the numbered-REQ-ID **citing-test** scan (a best-effort inline-`PREFIX-N`-citation scan) and the diagnostic-code **asserting-test** scan (a registry code without an asserting test, or an asserted code absent from the registry, is a CI failure). `conventions.md` §REQ-ID discipline describes the first as a scan "across the test sources" that "binds against the live test corpus", and `H5a`'s `Adds.`/`Tests.` repeat "in the test corpus" / "over the test sources". The actual file set those scans read is never pinned anywhere in the plan corpus.

The only glob the plan names — `**/*.test.ts` — is introduced exclusively as the *exclusion* scope for the `src/**` lint gates (the *Sequential by default* rule excludes `**/*.test.ts` from `no-restricted-syntax`, and "Test code (`**/*.test.ts`) is unrestricted but MUST NOT carry the exemption comment"). It is never stated to be the citing-test / asserting-test scan corpus. The plan does establish one negative-space carve-out for these scans — the live-mode path selection "reconciles the live spec/test corpus exclusive of that fixtures root" (the dedicated seeded-fixtures root) — but it never positively defines what the test corpus *is*.

An implementer building the `H5a` gate must therefore guess the scan glob. A glob that is too narrow (e.g. one that misses an integration-test directory, or resolves to zero files in some checkout) silently certifies fewer or zero citing/asserting tests while every gate stays green — the gate passes vacuously. Because the same unpinned phrasing carries through to the warn-only canary (`H5b`) and the live-corpus footing (`H6a`), a wrong corpus choice propagates to the loom 1.0 release gate.

No commit ever removed a previously-present pin; the defect entered with each arm at authoring time rather than via a later regression.

## Solution Space

**Shape:** single

### Recommendation

Pin the scan corpus once in `conventions.md` §REQ-ID discipline — the authoritative home of the "across the test sources" / "in the test corpus" phrasing that both arms inherit — by defining the citing-test and diagnostic-code asserting-test scan corpus as the repository's `**/*.test.ts` test sources, exclusive of the dedicated test-fixtures root `H5a` reserves for the permanent-by-design seeded gate fixtures. The plan already names `**/*.test.ts` as the test-code set and already establishes the seeded-fixtures-root exclusion for these scans' live mode, so the pin reconciles those two existing facts into a positive corpus definition rather than introducing a new concept.

In `H5a`'s `Adds.`, replace the bare "in the test corpus" / "over the test sources" references for the citing-test and registry-code↔asserting-test arms with a reference to that pinned corpus definition, so the operational leaf resolves to the same file set the convention names. `H5b` and `H6a` need no edit beyond inheriting the now-resolvable term; their "live test corpus" references resolve through the pinned definition.

The literal corpus to state: the `**/*.test.ts` files across the repository, minus the dedicated seeded-gate-fixtures root (the same root the live-mode path selection already excludes). Watch the edge case that this corpus must be the production test sources, not the seeded fixtures — the existing fixtures-root exclusion already covers that and must remain in force after the pin.

## Consequence

**Severity:** correctness

The citing-test and diagnostic-code asserting-test scans read an undefined file set, so two implementers can pin different corpora — or pick a glob that resolves to too few or zero files. A too-narrow corpus makes the gate certify "a citing/asserting test exists" while skipping the tests that would falsify it, so a load-bearing closing gate passes vacuously and a REQ-ID or diagnostic code ships effectively untested with CI green.
