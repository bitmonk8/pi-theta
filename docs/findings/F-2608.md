## Finding

`docs/spec_topics/query/query-escapes-stringification.md` is titled "Query escapes stringification" and its H1/filename advertise exactly two concerns: template escapes (QRY-17) and stringification of interpolated values (QRY-18). It then carries three further sections that are not about either concern:

- **QRY-19 — "Discarded query results are a parse error"**: the must-use discipline that makes a bare `@`...`` expression-statement a `loom/parse/discarded-query-result` error and forces the author to acknowledge the `Result` (`?`, `let _ =`, or bind). This is query-Result-handling discipline, adjacent to the rest of query failure semantics — not interpolation or escaping.
- **QRY-20 — "Observability of discarded results"**: the operator-facing delivery contract for a discarded `Err` (one runtime event on the always-log set, `display: false`, `discard_site` population, fire-once-per-discarded-`Err`). This is a runtime-event-channel delivery contract; its canonical channel is owned by `pi-integration-contract/runtime-event-channel.md`, which this section already cross-references.

The file's title therefore under-describes its contents: a reader looking for discard discipline or discard observability has no reason to open a page named "escapes stringification", and a reader of the page sees three sections that do not match its declared scope. (QRY-21, "Panics during interpolation are not caught by `let _ =`", straddles both topics — it is about interpolation-time panics escaping the discard form — so its placement is defensible either way and is an edge case for the fixer, not part of the core defect.)

The plan corpus already treats this as two distinct concerns: leaf `V13a` ("Query render and escapes") owns the stringification/render surface, while a separate leaf `V13g` ("Discarded-query result discipline and discard observability") owns QRY-19/QRY-20 and points additionally at `runtime-event-channel.md`. The spec file structure does not reflect that decomposition.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** f5e89f4 ("docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** The discard sections originated inside the monolithic `docs/spec_topics/query.md`, where they sat between "Stringification of interpolated values" and the tool-loop sections under the undifferentiated "Query" page — no file-title mismatch was possible there. Commit f5e89f4 split that page into topic-scoped files under `query/` to satisfy the spec-set size cap and, in doing so, bundled "Escapes", "Stringification of interpolated values", "Discarded query results are a parse error", "Observability of discarded results", and "Panics during interpolation" into a single file titled `# Query escapes stringification`. The title-vs-contents mismatch was introduced by that bundling choice; the later QRY-1..21 code coinage (716a13b) only labelled the already-misfiled sections and did not introduce or relocate them.

## Solution Space

**Shape:** multiple

### Option A — Extract the discard sections into a dedicated topic file

- **Approach:** Move "Discarded query results are a parse error" (QRY-19) and "Observability of discarded results" (QRY-20) out of `query-escapes-stringification.md` into a new single-topic file (e.g. `docs/spec_topics/query/query-discard.md`) titled for discard discipline and observability. Add it to the `query.md` child-page index. Repoint the `glossary.md` `#observability-of-discarded-results` cross-reference and the `code-registry-parse.md` "Discarded query results" link at the new file. Update the `V13g` / `V13g-T` Spec pointers.
- **Spec edits:** new file with the two sections verbatim; index entry in `query.md`; anchor-target updates in `glossary.md` and `code-registry-parse.md`; leaf Spec-pointer updates.
- **Pros:** Matches the established single-topic file convention in `query/` and mirrors the plan's `V13a`-vs-`V13g` decomposition; leaves `query-escapes-stringification.md` accurately scoped to escapes + stringification.
- **Cons:** Touches the most sites (new file, index, two cross-references, leaf pointers); largest diff.
- **Risks:** A missed anchor reference becomes a dead link; the QRY-21 placement decision (stays with stringification vs moves with discard) must be made explicitly.

### Option B — Retitle the file in place to cover all its contents

- **Approach:** Leave all sections where they are and widen the H1 (e.g. "Query stringification and discard semantics") so the title describes the discard sections too. Keep the filename `query-escapes-stringification.md` unchanged so no cross-reference or leaf pointer breaks; update only the `query.md` index link text to match the new H1.
- **Spec edits:** H1 line in `query-escapes-stringification.md`; index link text in `query.md`.
- **Pros:** Minimal churn; zero risk of dead anchors because no file or section moves.
- **Cons:** Leaves a multi-topic page whose filename still reads "escapes-stringification"; does not align the spec structure with the corpus's single-topic convention or the plan's leaf split; the page keeps accreting unrelated concerns.
- **Risks:** Filename remains misleading even after the H1 is fixed, so topic-by-filename navigation stays degraded.

### Recommendation

Take **Option A**. The `query/` subtree is already organised one-topic-per-file and the plan splits render/escapes (`V13a`) from discard discipline and observability (`V13g`), so a dedicated discard file is the structure both the corpus and the plan already imply. Keep QRY-21 ("Panics during interpolation are not caught by `let _ =`") with the stringification content, since its subject is interpolation-time panics, and leave a forward-reference to the new discard file. Watch that every anchored reference (`glossary.md`, `code-registry-parse.md`, the `query.md` index, and the `V13g` / `V13g-T` Spec pointers) is repointed so no link goes dead.

## Consequence

**Severity:** cosmetic

The discard rules are complete and correct and remain reachable via the anchored cross-references from the glossary, the diagnostic registry, and the plan leaves; nothing an implementer builds diverges. The defect is purely organisational: the page's title and filename misrepresent its scope, raising discovery cost for a reader navigating by topic.
