## Finding

The per-invocation-retry-budget paragraph in `docs/spec_topics/binder/determinism-cancellation-failure.md` (the sentence anchored at `#per-invocation-retry-budget`) pins inter-attempt timing as: "the runtime MUST issue each budgeted retry as soon as the preceding failure is classified, with no loom-side delay, backoff, or `Retry-After` wait inserted between a loom-level attempt and its retry." The phrase "as soon as … classified" carries no operational upper bound; "no loom-side delay" reads as a restriction but does not name the observable a conformance test would assert against. A test that fakes a transport failure and then asserts "the second `complete()` happened" cannot distinguish a synchronous re-issue from a one-microtask hop, a `queueMicrotask`, or a 5 ms `Clock.setTimeout(..., 5)`; any of those satisfies an English reading of "as soon as the preceding failure is classified".

The spec already carries the right testing primitive for binder-call timing: `PIC-12.` `Clock` / `FakeClock` is the **only** timer surface the runtime is permitted to use (`docs/spec_topics/pi-integration-contract/host-interfaces-services.md`, `#pic-12`), and the `loop-iter` checkpoint demonstrates the pattern by anchoring its macrotask-yield property to `clock.advance(0)`. The binder retry rule does not adopt the same anchor; converting "no loom-side delay" into an explicit `Clock`-mediated predicate removes the ambiguity without changing the intended behaviour.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** d0bcdb2 — pi-loom spec: resolve "binder per-invocation retry inter-attempt timing" (2026-06-05, Thomas Andersen)
**History:** Prior to d0bcdb2 the per-invocation-retry-budget paragraph said nothing about inter-attempt timing — the question was an open spec-review item (`docs/spec-review.md`). d0bcdb2 closed the open item by appending the "issue each budgeted retry as soon as the preceding failure is classified, with no loom-side delay, backoff, or `Retry-After` wait" sentence to the binder paragraph and updating `binder-inference.md`'s closing line to delegate per-attempt backoff and `Retry-After` to `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`. The defect is the wording chosen at that resolution: it pins the policy direction but not a black-box-observable predicate.

## Solution Space

**Shape:** single

### Recommendation

Restate the inter-attempt-timing clause as a `Clock`-anchored predicate, reusing PIC-12 as the conformance handle. Replace the sentence

> The inter-attempt timing of this budget is immediate re-issue: the runtime MUST issue each budgeted retry as soon as the preceding failure is classified, with no loom-side delay, backoff, or `Retry-After` wait inserted between a loom-level attempt and its retry.

with text along these lines:

> The inter-attempt timing of this budget is immediate re-issue: between classifying a failure and issuing its budgeted retry, the runtime MUST NOT schedule any deferred work through the [`Clock`](../pi-integration-contract/host-interfaces-services.md#clock--fakeclock-interface) seam (no `Clock.setTimeout` call — including a 0-ms one — and no use of any other delay primitive; per [PIC-12](../pi-integration-contract/host-interfaces-services.md#pic-12), `Clock.setTimeout` is the runtime's only permitted timer surface, so this exhausts the loom-side delay surface). Equivalently, a `FakeClock`-driven test that classifies a binder failure observes the next `complete(...)` invocation **without** an intervening `clock.advance(...)` tick. This excludes any loom-side backoff and any server-requested wait such as an HTTP `Retry-After`.

Keep the existing follow-up sentence about `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs` unchanged — per-attempt backoff inside a single `complete(...)` is owned by `@earendil-works/pi-ai` and is not what this clause governs. Keep the NOCEIL-1 cross-link sentence: the new predicate continues to support that argument (no `Clock.setTimeout` call between attempts ⇒ loom introduces no wall-clock-timeout delay layer ⇒ NOCEIL-1 stands without a carve-out).

After landing the spec edit, extend `V11f-T`'s `HC3-a` / `HC3-b` bullets so each retry case asserts the new predicate (the implementation spy / `FakeClock` already exists in the leaf's `Deps` via `V17a`'s `Checkpoint` seam discipline); `V11f`'s `npm test` `Ships when` line gains the same conformance-test obligation.

Edge cases the implementer must watch:
- Microtask hops (`await`, `.then`, `queueMicrotask`) are not "delay" under this predicate — they do not consult the `Clock` seam — and remain permitted; the predicate intentionally allows the natural awaited boundary between `complete(...)` resolution and re-issue.
- Pi-SDK-internal backoff inside one `complete(...)` call is also out of scope: it does not go through loom's `Clock` instance, and the existing `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs` delegation sentence remains the home for that.
- The predicate covers both the malformed-envelope retry path (where classification is the synchronous envelope-validation step on the assistant message) and the transport retry path (where classification is the provider-error-mapping table consultation on the rejected/error result); neither path may consult `Clock.setTimeout` before issuing its retry.

## Consequence

**Severity:** advisory

Two reasonable implementers will write the binder retry the same way (no one inserts a `setTimeout` between classification and re-issue when the spec says "no loom-side delay"), so behaviour does not diverge in practice. What does diverge is the **test**: `V11f-T` has no spec-anchored predicate that asserts the inter-attempt timing property, so the leaf's `npm test` line for HC3-a / HC3-b can ship a green budget-counting test that would still pass against an implementation that hopped one macrotask between attempts.
