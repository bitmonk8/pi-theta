## Problem

The `ERR-11` test bullet in `V4c` and its paired `V4c-T` reads "the non-mutation window binds between the cancelled turn and the next driver send," which mis-paraphrases spec `ERR-11` (`error-model.md#err-11`, "Mid-stream cancellation, respond-repair scope window"). The spec scopes the obligation to typed queries with respond-repair follow-ups, binds the `ERR-8` non-mutation and `ERR-9` no-compensating-injection obligations between the cancelled streaming turn and the next driver send, and carves out the respond-repair loop's own conversation appends (governed by schema-validation respond-repair). The plan bullet erases that scoping and the carve-out, and it coins "non-mutation window" — an undefined noun where the spec speaks of the non-mutation *obligation* binding over an interval. With the scope stripped, the bullet collapses into a generic non-mutation-interval restatement of `ERR-8`, so an implementer would transcribe a duplicate `ERR-8` assertion rather than the respond-repair-scope-window boundary test the spec mandates.

## Solution approach

Rewrite the `ERR-11` Tests bullet in both `V4c-terminal-outcomes.md` and `V4c-T-terminal-outcomes.md` so it mirrors spec `ERR-11` instead of paraphrasing it. Scope it to typed queries with respond-repair follow-ups, state that the `ERR-8` non-mutation and `ERR-9` no-compensating-injection obligations bind between the cancelled streaming turn and the next driver send, and preserve the carve-out that the respond-repair loop's own conversation appends remain governed by schema-validation respond-repair. Replace the coined "non-mutation window … binds" with the spec's obligation phrasing and link the spec anchor `../spec_topics/errors-and-results/error-model.md#err-11`.
