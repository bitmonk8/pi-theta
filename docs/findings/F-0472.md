## Finding

`H2a` and `H5c` validate the *same* `// allow-broad-catch:` comment token against the same four-arm predicate from `conventions.md` *Specific exception types only*: a coverage-matrix REQ-ID, a Code-keyed-obligation-areas token, a concrete `loom/...` registry code, or the structural `pi-sdk-boundary` token. The two leaves describe the Code-keyed-areas arm at different precision.

`H2a`'s first *Tests* bullet admits "an enumerated *Code-keyed obligation areas* entry" with no statement of what an "entry" is or how it resolves. `H5c`'s *Tests* bullet and *Ships when* pin the arm exactly: "a *Code-keyed obligation areas (no numbered REQ-IDs)* row's canonical `cka-<n>` token (matching exactly one row's *Token* cell)". `coverage-matrix.md` §Code-keyed obligation areas confirms the canonical form — a row-unique `cka-<n>` token in the leading *Token* column, resolving iff it matches exactly one row's *Token* cell.

The vague H2a phrasing and the precise H5c phrasing describe one predicate. An implementer reading `H2a` alone, building its lint fixture and the recognised passing-token set, can encode the Code-keyed-areas arm by a different rule than the `cka-<n>` exactly-one-Token-cell match (e.g. matching the spec-area prefix, the closing-leaf id, or any token appearing in the row), producing an H2a fixture whose admitted token `H5c`'s CI gate would reject — or vice versa. The two enforcement points of the same allow-list then disagree by read order, not by design.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `96563f9` (2026-06-10, "resolve broad-catch allow-comment predicate unsatisfiable for exempt sites"); `7259777` (2026-06-18, "resolve code-keyed obligation areas lack a canonical citable token")
**History:** `96563f9` authored H2a's widened-predicate *Tests* bullet with the arm "an enumerated *Code-keyed obligation areas* entry" — correct at the time, because no canonical citable token for those rows yet existed. `7259777` then introduced the canonical `cka-<n>` token and propagated the precise one-`Token`-cell resolution into `H5c`, `conventions.md`, `coverage-matrix.md`, `V14b`, and `V9g`, but did not touch `H2a` (confirmed by its file list). The divergence opened at `7259777`: the formalisation landed in the sibling gate while H2a's parallel arm was left at its pre-canonical-token phrasing. The durable fix back-propagates that formalisation into H2a, not a revert of either commit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H2a-cross-cutting-gates.md`, first *Tests* bullet (`Specific exception types only`), strike `an enumerated *Code-keyed obligation areas* entry` from the admitted-token list and insert in its place a Code-keyed-areas arm that names the canonical token and its resolution rule, mirroring `H5c`: `` a *Code-keyed obligation areas* row's canonical `cka-<n>` token (matching exactly one row's *Token* cell in [`coverage-matrix.md`](./coverage-matrix.md)) ``. Leave the other three arms (coverage-matrix REQ-ID, concrete `loom/...` registry code, structural `pi-sdk-boundary` token) unchanged. No edit to `H5c`, `coverage-matrix.md`, `conventions.md`, or the spec is required — those already carry the authoritative form. Watch that the inserted relative link target (`./coverage-matrix.md`) is correct for the `plan_topics/` directory, matching the existing `H5c` link.

## Consequence

**Severity:** correctness

Two reasonable implementers can build the H2a `no-broad-catch` fixture's admitted-token recognition for the Code-keyed-areas arm on a different rule than H5c's `cka-<n>` exactly-one-Token-cell resolution, so the lint rule and the CI reconciliation gate validate the same `// allow-broad-catch:` corpus inconsistently — a token the ESLint fixture treats as passing can be rejected by H5c (or never independently exercised), defeating the single-allow-list design.
