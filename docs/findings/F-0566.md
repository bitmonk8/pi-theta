## Finding

`V15b` (and its paired tests leaf `V15b-T`) assert that the `loom/runtime/invoke-depth-exceeded` runtime panic surfaces in two separately-required modes — a top-level overflow delivered as a Pi system note, **and** a nested overflow surfaced to the parent as `Err(InvokeInfraError { cause: "panic", ... })`. Both the `Tests` bullet and the `Ships when` condition turn on those two surfacing modes. Yet `V15b`'s Deps are `V15b-T, V15a, V15c, V16a` and `V15b-T`'s are `V15a, V15c, V16a`. Neither names a leaf that produces the surfaces under test.

The producing machinery lives elsewhere: the `loom-system-note` delivery channel (the top-level-overflow surface) is owned by `V7d`, and the closed runtime-panic set that registers `loom/runtime/invoke-depth-exceeded` with its message template is owned by `V4b`. The nested-overflow `InvokeInfraError { cause: "panic" }` parent-observation is at least transitively reachable — `V15b` → `V15a` → `V4f`, whose `ERR-13` case 4 drives "a panic in an `invoke` child (parent observes `InvokeInfraError { cause: "panic" }`)" — but the top-level system-note surface (`V7d`) and the panic source/template (`V4b`) are reachable from neither leaf's Deps closure.

The leaf already demonstrates the project's binding convention for a consulted cross-leaf surface: it lists `V16a` in Deps precisely because it consults that arbitration seam (and sibling leaves do the same — `V17a` lists `H4a`/`H4b`, `V15d` lists `V9f`). The two surfacing-mode producers are simply absent from that wiring.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** c6a664e — "pi-loom plan: build/update plan for spec.md + review" (2026-06-10)
**History:** `V15b` was created in the initial plan build already asserting the dual surfacing mode ("a top-level system note vs a nested `InvokeInfraError{panic}`") with Deps `V15b-T, V15a, V16a` — naming neither `V4b` nor `V7d`. Subsequent edits did not introduce or close the gap: `084f171` added `V15c` to Deps (for `.warp fn` frame counting), and `ac99f79` reworded the two surfacing modes into the current "both separately-required modes" phrasing, leaving the Deps set and the missing producer edges unchanged. The defect is original to the leaf, not introduced by a later edit.

## Solution Space

**Shape:** multiple

### Option A — Bind the producing leaves in Deps

- **Approach:** Add `V7d` (the `loom-system-note` delivery channel) and `V4b` (the runtime-panic set that sources `invoke-depth-exceeded`) to the Deps of both `V15b` and `V15b-T`, alongside the existing `V16a` binding, so the two surfacing-mode assertions trace to the leaves that produce those surfaces.
- **Plan edits:** `V15b-invoke-depth-cycle.md` Deps and `V15b-T-invoke-depth-cycle.md` Deps.
- **Spec edits:** None.
- **Pros:** Makes both surfacing-mode assertions traceable to real producing surfaces; the pickability predicate then prevents `V15b`/`V15b-T` being picked before those surfaces exist; mirrors the leaf's own `V16a` binding and the `V17a`/`V15d` precedent.
- **Cons:** Increases `V15b`'s Deps fan-in and pushes it later in build order.
- **Risks:** None observed — `V4b` and `V7d` do not depend on `V15b`, so no dependency cycle is created.

### Option B — Declare the surfaces are stood in by a `V15b-T`-owned double

- **Approach:** State in `V15b`/`V15b-T` that the two surfacing modes are exercised against a test double standing in for the panic-routing / system-note seam, naming the seam the double stands in for, rather than against the live `V4b`/`V7d` surfaces.
- **Plan edits:** `V15b-T-invoke-depth-cycle.md` Tests (and a corresponding note in `V15b-invoke-depth-cycle.md` Adds).
- **Spec edits:** None.
- **Pros:** Keeps `V15b`'s Deps minimal; appropriate if the genuine live surfacing is asserted by a downstream live-carrier witness or the `H7a` integration run.
- **Cons:** A double can diverge from `V7d`'s real `loom-system-note` channel, leaving the top-level-overflow surfacing unverified against the production surface.
- **Risks:** False-green at `V15b` if the double diverges from the real channel.

### Recommendation

Take Option A. `V15b-T` asserts the actual `loom/runtime/invoke-depth-exceeded` template and two real surfacing channels, so binding `V4b` and `V7d` in Deps keeps the leaf honest under the pickability predicate; the nested-overflow `InvokeInfraError { cause: "panic" }` mode is already transitively reachable via `V15a` → `V4f`, so the specific missing edges are `V7d` (system-note channel) and `V4b` (panic source/template). Edge case to watch: these are Deps-wiring edges, not coverage-producing additions, so confirm no `H5b` transitive-completeness / coverage-matrix obligation is triggered by adding them.

## Consequence

**Severity:** correctness

The tests-task `V15b-T` is pickable while `V4b` and `V7d` are still incomplete, because neither is in its Deps closure. Two reasonable test authors then diverge: one stands the panic source and the system-note channel in via a local double, the other assumes the live `V4b`/`V7d` surfaces exist — and a double can pass green while diverging from `V7d`'s actual `loom-system-note` channel behaviour, so the top-level-overflow surfacing mode the leaf exists to verify may never be exercised against the real surface.
