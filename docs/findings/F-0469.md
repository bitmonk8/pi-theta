## Finding

`real-host-smoke-gate.md` defines the manual real-host smoke as "a required pre-merge gate on two independent triggers": **(1)** each Pi version bump, and **(2)** any merge whose diff touches the four session-double fidelity-contract axes. Trigger (1) has a direct spec basis — `version-bump-triggers.md` output (c) mandates "green end-to-end runtime evidence at the new pin" before a bump is considered landed, and `version-bump-intro.md` §Non-goals (a) explicitly keeps that runtime-evidence acceptance gate *in scope* for the spec. Trigger (2) has no spec counterpart: it is a standing pre-merge obligation on every contributor, introduced by the plan to mitigate a divergence risk the plan itself creates by testing against an in-process session double rather than a live Pi host.

The issue is not that trigger (2) exists — it is defensible defensive governance that catches a divergence class (loom edits to the four fidelity axes between Pi bumps) no other gate covers. The issue is that the note presents both triggers with identical normative weight ("a required pre-merge gate on two independent triggers"), with no marker distinguishing the spec-mandated obligation from the plan-introduced one. A reader or release-gate auditor cannot tell which trigger is spec-anchored and which is plan-local defensive hygiene, and a future spec-fidelity sweep that prunes plan-beyond-spec surface has no signal that trigger (2) is *deliberately* beyond spec rather than unreconciled drift.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `e7f14dd` (2026-06-11, "pi-loom plan: resolve 'Real-host fidelity of the session double has no reproducible detection point'")
**History:** The two-trigger structure of the manual real-host smoke gate — including trigger (2), any merge whose diff touches the four session-double fidelity-contract axes — entered the corpus in `e7f14dd` as the resolution of the plan-review finding "Real-host fidelity of the session double has no reproducible detection point". `git log -S "touches the four fidelity"` over `docs/` returns `e7f14dd` as the earliest commit carrying the condition; no prior commit holds the trigger-(2) text. Trigger (2) was therefore plan-introduced from the moment it existed and has never had a spec basis — the no-direct-spec-basis property is intrinsic to the construct's design, not the result of a later edit or a spec/plan drift. Subsequent commits (e.g. `0bd1978`, `42dca68`, `fd22a3d`) refined the wording and relocated the protocol into its own `real-host-smoke-gate.md` file but did not change the plan-introduced character of trigger (2).

## Solution Space

**Shape:** multiple

### Option A — Keep trigger (2) as a required gate; annotate its plan-introduced status

- **Approach:** Retain trigger (2) as a required pre-merge gate, and add a marker in `real-host-smoke-gate.md` that trigger (2) is plan-introduced defensive governance with no direct spec basis, distinguishing it from trigger (1).
- **Plan edits:** In `real-host-smoke-gate.md`, at the two-trigger definition ("a required pre-merge gate on two independent triggers"), state that trigger (1) realises the spec's version-bump runtime-evidence acceptance gate (`version-bump-triggers.md` output (c); `version-bump-intro.md` §Non-goals (a)) while trigger (2) is a plan-introduced obligation with no spec counterpart, added to catch a double-vs-real divergence the in-process-double approach creates.
- **Spec edits:** None.
- **Pros:** Preserves the divergence-catching value of trigger (2); makes the fidelity status auditable so a future spec-fidelity sweep does not misread it as drift.
- **Cons:** Keeps plan surface beyond spec (the property the lens flags).
- **Risks:** An annotation phrased as a downgrade could be misread as making trigger (2) optional.

### Option B — Demote trigger (2) to contributor guidance

- **Approach:** Reframe trigger (2) from a required pre-merge gate to recommended contributor guidance, leaving trigger (1) as the only required smoke gate, minimising plan surface beyond spec.
- **Plan edits:** In `real-host-smoke-gate.md`, change the trigger-(2) framing from "required pre-merge gate" to contributor guidance, and reconcile the dependent merge-blocker prose ("a confirmed behavioural-divergence finding from that smoke is a merge blocker") so the blocker consequence applies only when the smoke was in fact run.
- **Spec edits:** None.
- **Pros:** Removes a normative obligation with no spec basis; smallest plan-beyond-spec surface.
- **Cons:** Weakens the only defence against a loom-change-introduced double-vs-real divergence reaching `main`; the targeted divergence class becomes detectable only at the next version bump (trigger 1).
- **Risks:** A real fidelity divergence ships to `main` between bumps.

### Recommendation

Option A. Trigger (2) targets a divergence class — loom edits to the four fidelity axes between Pi bumps — that no other gate covers, so demoting it trades away real protection; annotating its plan-introduced status resolves the fidelity-transparency defect without losing coverage. Edge case for the implementer: trigger (1)'s spec cross-references stay intact, and the plan-introduced marker is read together with the "required pre-merge gate" wording so a reader does not infer the gate is optional.

## Consequence

**Severity:** advisory

If the plan ships unfixed, contributors and the H6a release-gate auditor cannot distinguish the spec-anchored runtime-evidence obligation (trigger 1) from the plan-introduced fidelity-axis pre-merge obligation (trigger 2). A later spec-fidelity reduction pass risks either silently dropping trigger (2) as apparent drift or mistaking it for spec-required and over-constraining the spec. No implementer leaf is blocked.
