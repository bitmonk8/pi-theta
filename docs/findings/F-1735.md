## Finding

The Pi version-bump procedure (`version-bump-triggers.md`, outputs paragraph) makes output (c) a merge-blocking acceptance gate: "A bump is therefore not considered landed until the runtime evidence in (c) is green: a bump whose runtime-evidence run is red MUST NOT be merged at the candidate pin." Output (c) is defined as "a representative integrated `.loom` exercised through the loom's end-to-end harness against the bumped SDK — driving a typed query, the tool-call loop, an `invoke(...)`, schema validation, the binder, and cancellation."

What the spec never states is how that harness obtains provider responses. Driving a typed query, a tool-call loop, the binder, and cancellation all require model turns; the spec is silent on whether those turns come from a live provider (real credentials, network egress, per-run token cost, non-deterministic output) or from recorded/scripted responses against an in-process double. "The loom's end-to-end harness" is named only here in the entire spec corpus and is nowhere defined. The only place the corpus mentions the live-vs-recorded distinction at all is the deferred `loom test` entry in `future-considerations/surface-extensions.md` ("runs a loom against a recorded transcript or a stub model without hitting a live provider"), which is explicitly out of loom 1.0 scope and does not govern output (c).

The distinction is not cosmetic: it decides whether the merge-blocking gate can run in ordinary CI at all. A live-provider gate needs provisioned credentials and network access for every supported provider and incurs cost and flakiness on each bump; a recorded/double-backed gate is deterministic and credential-free but does not exercise the real provider wire. An implementer building the gate from the spec alone has no basis to choose, and the spec does not point at where the harness is defined.

## Solution Space

**Shape:** single

### Recommendation

In the output-(c) clause of `version-bump-triggers.md`, state the provider-execution posture explicitly: the runtime-evidence run executes the representative integrated `.loom` against a deterministic in-process session double / recorded provider responses at the bumped pin — it requires no live-provider credentials or network — so the gate is runnable under the ordinary `npm test`-class CI lane. Add a sentence (or forward-reference) noting that a green double-backed run is bounded by the double's fidelity to the bumped SDK and is **not** real-host coverage, and that live-host behavioural divergence is covered by the separate, explicitly manual real-host smoke gate rather than by output (c). This aligns the spec with the model the plan already implements (`H4a` session double; `V18d` acceptance gate; the manual real-host smoke protocol) without prescribing the harness's internal layout, which remains implementation-owned.

Edge cases for the implementer: keep the wording behavioural (what the run must exercise and that it needs no live credentials), not a binding to a named test file or runner; and ensure the clause does not promise real-provider coverage it cannot deliver, since the deferred `loom test` (recorded transcript / stub model) surface is out of loom 1.0 scope and must not be conflated with this gate.

## Consequence

**Severity:** correctness

Two reasonable implementers working from the spec alone diverge on a merge-blocking gate: one builds a live-provider acceptance run (needs per-provider credentials and network, costs tokens, is non-deterministic, and is un-runnable in a default CI lane), the other builds a recorded/double-backed run (deterministic and credential-free but not real-host coverage). The downstream plan happened to choose the double-backed model independently, so build work is not actually broken, but the spec — the authority the gate cites — remains ambiguous and contradicts neither nothing nor itself only because the plan supplied the missing answer.
