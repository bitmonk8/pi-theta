## Problem

Output (c) of the version-bump procedure (`version-bump-triggers.md`, outputs paragraph) is a merge-blocking acceptance gate, but the spec never states how "the loom's end-to-end harness" obtains its provider responses — from a live provider (real credentials, network egress, per-run cost, non-deterministic output) or from recorded/double-backed responses (deterministic, credential-free, but not real-host coverage). The harness is named only here in the corpus and is nowhere defined. The distinction decides whether the merge-blocking gate can run in an ordinary CI lane at all, and an implementer building it from the spec alone has no basis to choose.

## Solution approach

Clarify output (c)'s provider-execution posture in `version-bump-triggers.md`: state that the runtime-evidence run exercises the representative integrated `.loom` against deterministic in-process / recorded provider responses at the bumped pin, requiring no live-provider credentials or network, so the gate runs under the ordinary `npm test`-class CI lane. State that a green double-backed run is bounded by the double's fidelity to the bumped SDK and is not real-host coverage, with live-host behavioural divergence covered by the separate manual real-host smoke gate rather than by output (c).

## Solution constraints

- Out of scope: do not conflate output (c) with the deferred `loom test` recorded-transcript / stub-model surface (`future-considerations/surface-extensions.md`), which is out of loom 1.0 scope; the clause must not promise real-provider coverage it cannot deliver.
- Do not bind the harness's internal layout, test-file name, or runner — those remain implementation-owned.
