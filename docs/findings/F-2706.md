## Finding

`conventions.md` §REQ-ID discipline enumerates the diagnostics-registry code namespace inconsistently with itself and with the gate leaves that consume it. The *executable spec REQ-ID* definition characterises the registry as the closed triple — "the `loom/{parse,load,runtime}/*` diagnostic-registry codes (including warning-severity `W` codes) … are separate closing-gate surfaces." Every other plan surface that resolves a registry code uses the open `loom/...` form: the *Specific exception types only* broad-catch predicate admits "a concrete `loom/...` diagnostics-registry code present in the registry," `H5c` reconciles allow-list tokens against that open form, and `H5a` reconciles "the diagnostics code registry" as a whole (carving out only the `loom/typecheck/*` build-time brand prefix).

The closed triple is wrong, because the diagnostics registry has a fourth namespace shard. The spec's `diagnostic-shape.md` defines `loom/parse/*`, `loom/load/*`, `loom/runtime/*`, **and** `loom/host/*`, and `code-registry-host.md` registers concrete `loom/host/*` codes (`loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, `loom/host/session-shutdown-runtime-degraded`). The plan relies on these `loom/host/*` codes resolving as concrete registry codes in two places: the broad-catch exempt-site (3) tokens (`V9h` cites `loom/host/session-shutdown-reason-unknown` and `loom/host/session-shutdown-pinned-constant-unreadable`), and the `H5a` registry↔asserting-test parity reconciliation that covers the `loom/host/*` shard.

Read literally, the closed triple and the open form disagree: under the triple the `loom/host/*` allow-list tokens fall outside the registry namespace and the broad-catch gate (`H5c`) would treat them as unresolvable (CI failure), and the `V9h` `loom/host/*` codes sit outside the parity surface; under the open `loom/...` form used everywhere else they resolve. The plan asserts both. The registry namespace is spec-owned; the plan's restatement in §REQ-ID discipline is the surface that is wrong.

## Consequence

**Severity:** correctness

An implementer reading the §REQ-ID discipline restatement in isolation can conclude `loom/host/*` is not a registry code, then either wire the `H5a` parity scan to only the three-shard namespace (silently dropping the `loom/host/*` shard from registry↔test reconciliation) or build the `H5c` broad-catch resolver to reject the `V9h` `loom/host/*` allow-list tokens as unresolvable. Two good-faith implementers reading the contradictory clauses would diverge; the same document both admits and excludes the host shard.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md` §REQ-ID discipline, in the *executable spec REQ-ID* definition, replace the closed-triple enumeration `the `loom/{parse,load,runtime}/*` diagnostic-registry codes (including warning-severity `W` codes)` with the open form `the `loom/...` diagnostic-registry codes (including warning-severity `W` codes)`, matching the form `H5a`, `H5c`, and the *Specific exception types only* broad-catch predicate already use. This is the only remaining closed-triple occurrence in the bullet; the un-anchored-MUST recogniser arm and the broad-catch predicate already read open.

The registry namespace is spec-owned (`diagnostics/diagnostic-shape.md` defines `loom/{parse,load,runtime,host}/*`), so the open `loom/...` form is the faithful restatement; it admits the `loom/host/*` shard the plan relies on. Preserve the existing `loom/typecheck/*` build-time brand-string carve-out already stated later in the same section. Do not narrow the broad-catch resolver and do not rename the `V9h` `loom/host/*` codes — both are correct as written and are what this restatement must agree with.

(An equivalent alternative — widening the literal enumeration to the four-shard set `loom/{parse,load,runtime,host}/*` — would also resolve the contradiction, but the open `loom/...` form is preferred because it matches the existing gate-leaf wording verbatim and does not re-introduce a fixed shard list that a future spec-side shard addition would again desynchronise.)
