## Finding

The `loom/load/missing-source` row in the `loom/load/*` registry table (`docs/spec_topics/diagnostics/code-registry-load.md`) carries `E/W` in the *Sev* column. The normative column legend (`diagnostic-shape.md#column-legend`) defines `E/W` as a severity "not fixed at the code level but resolved per diagnostic by that row's own *Trigger* prose" — i.e. the token asserts that both an `E` and a `W` resolution exist, decided per case.

For `missing-source` the trigger prose resolves to only two outcomes, neither of which is `W`: "error for explicit references (`pi.looms` entries, settings `loomPaths` entries, `--loom` flags), warning never — conventional locations (global, project, package `looms/`) emit no diagnostic when missing." The discovery failure-modes table (`discovery/discovery-sources.md`, DISC-2) confirms this: every "Missing path" cell is either `error` (the three explicit references) or `silent` (the three conventional locations). There is no source for which a missing path yields a `missing-source` warning. The `W` arm of the `E/W` token is therefore unreachable for this code; the only severity it can ever resolve to is `E`.

This contrasts with the two sibling rows that legitimately use `E/W`: `loom/load/unreadable-source` (error for `--loom`, warning for every other source) and `loom/load/wrong-type-source` (error for the three explicit references, warning for the conventional roots) — both genuinely resolve to either letter per the failure-modes table. `missing-source` was lumped into the same shorthand even though its trigger admits no `W` resolution. The accurate fixed-level severity is `E`.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commit:** `5b772b6` — "pi-loom spec: resolve \"Discovery source failure-mode policy is unspecified\"" (Mon May 4 2026)
**History:** The corpus is git-tracked. `git log -S 'warning never'` and `git log -S '| \`loom/load/missing-source\` | E/W |'` both bottom out at `5b772b6`, the commit that first authored the discovery-source diagnostic rows together with the per-source `E/W` legend (then in `spec_topics/diagnostics.md`; relocated verbatim into `diagnostics/code-registry-load.md` by the `f5e89f4` size-cap split and the `a37d4ef` legend relocation, neither of which altered the row's `E/W` value). In that same commit the row simultaneously gained the `E/W` token and the "warning never" trigger clause: `missing-source` was given the shared `E/W` shorthand alongside its genuinely-bivalent siblings (`unreadable-source`, `wrong-type-source`) even though its own trigger admits no warning resolution. The contradiction has been present since the row's inception; no later commit introduced or widened it.

## Solution Space

**Shape:** single

### Recommendation

Change the *Sev* column of the `loom/load/missing-source` row from `E/W` to `E`. The existing trigger prose ("error for explicit references … warning never — conventional locations … emit no diagnostic when missing") already states the per-source disposition and needs no change; with a fixed `E` severity it now reads consistently — the code, when it fires, is always an error, and conventional-location misses produce no diagnostic at all rather than a warning.

Leave the sibling rows `loom/load/unreadable-source` and `loom/load/wrong-type-source` at `E/W`: their triggers genuinely resolve to either severity per the failure-modes table, so the carve-out token is correct for them. Do not weaken the legend's `E/W` definition — it is sound; only this one row mis-applied it.

## Consequence

**Severity:** advisory

A code's *Sev* value is part of the closed, normative registry contract (DIAG-2 makes a severity change a spec change). The `E/W` label tells a conformance/test author that `missing-source` can surface as a warning, so they may encode a warning-severity branch or fixture for a path that can never execute. Runtime behaviour itself is unambiguous from the trigger prose and the failure-modes table, so implementers do not diverge on observable behaviour — the defect is a self-contradictory severity label, not a behavioural gap.
