## Finding

`code-registry-parse.md` registers `loom/parse/interpolated-result` as a Phase `type` diagnostic in the `loom/parse/*` namespace: a `${expr}` interpolation whose Loom static type is `Result<T, E>` is rejected at parse time. The row's trigger prose, and the QRY-18 Notes bullet in `query/query-escapes-stringification.md`, add a runtime escape hatch: when the operand type is *statically unresolvable* (e.g. an inferred binding that widens past the parser's view), "the runtime renderer falls back to a panic carrying the same `loom/parse/interpolated-result` diagnostic code."

That runtime occurrence is unreconciled against three closed contracts that all govern runtime panics:

- **Closed panic-source enumeration.** `errors-and-results/error-model.md#runtime-panics` declares exactly six panic sources, all `loom/runtime/*`, and `code-registry-runtime.md` states its table "carries one row per code in that list." `loom/parse/interpolated-result` is not one of the six and has no `loom/runtime/*` row, yet it is described as firing "as a panic" at runtime.
- **Normative six-row message-template table.** The panic-message contract in `error-model.md` pins a six-row template table and requires a conformant runtime to emit the registered string for "every panic of that source." There is no template for `loom/parse/interpolated-result` as a panic, so the message a runtime must emit for this occurrence is undefined.
- **Closed located-site classification.** `diagnostic-shape.md#located-site-classification` is a closed three-category partition; it assigns every `loom/parse/*` row to **Located** (carries `file` + `range`, single token span). A runtime occurrence whose type was unresolvable has no static token span, so the `file`/`range` form for the runtime case is unspecified and conflicts with the parse-time classification of the same code.

Consequently the runtime occurrence has no pinned delivery channel (slash system note vs. `Err(QueryError)` to an `invoke` parent), no panic message template, and no located-site form. The spec already provides `loom/runtime/internal-error` as the runtime-defect surface for "any throw … not one of the six closed-list sources," with a defined delivery channel and a location-less form — which is exactly the surface this occurrence belongs on if it is not to be promoted to a seventh panic source.

The later file split (`31ff060`, `f5e89f4`) and the `f5e89f4` size-cap edit moved the prose but did not introduce or repair the tension.

## Solution Space

### Recommendation

Reword the QRY-18 Notes bullet in `query/query-escapes-stringification.md` and the `loom/parse/interpolated-result` row trigger prose in `diagnostics/code-registry-parse.md` so the unresolvable-type runtime occurrence is described as an unexpected interpreter exception routed through the existing `loom/runtime/internal-error` runtime-defect surface — the spec's defined surface for any failure outside the closed panic-source list, with a fixed delivery channel (slash system note + `InvokeInfraError { cause: "internal_error" }`) and a location-less form. Do not describe this occurrence as "raising the same code as a panic."

Keep `loom/parse/interpolated-result` a parse-time-only diagnostic (Phase `type`, Located); the closed six-source panic list, the six-row message-template table in `errors-and-results/error-model.md`, and the located-site classification in `diagnostic-shape.md` stay untouched. Add a cross-reference from the `error-model.md` runtime-defect-surface paragraph to this occurrence.

Ensure the message a runtime emits for this case is the `internal-error` template, not the parse-time message, and that no `loom/parse/*` code is described as a runtime panic. Verify QRY-21's "panics a `${…}` can trip" enumeration does not list or imply `interpolated-result` as a panic after the rewording.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the runtime occurrence: one routes it through `loom/runtime/internal-error` (location-less, generic message), one mints a seventh panic source with its own template and located-site form, and a third emits the bare `loom/parse/interpolated-result` code with no defined message or channel. The runtime diagnostic an author sees for an unresolvable-`Result` interpolation — its text, its `file`/`range`, and whether an `invoke` parent receives a panic vs. infra error — is therefore implementation-defined.
