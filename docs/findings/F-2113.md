## Problem

The QRY-18 Notes bullet in `query/query-escapes-stringification.md` and the `loom/parse/interpolated-result` row trigger prose in `diagnostics/code-registry-parse.md` describe a runtime occurrence: when a `${expr}` operand's `Result<T, E>` type is statically unresolvable, "the runtime renderer falls back to a panic carrying the same `loom/parse/interpolated-result` diagnostic code." That runtime occurrence is unreconciled against three closed contracts. The closed six-source panic enumeration at `error-model.md#runtime-panics` (all `loom/runtime/*`) admits no `loom/parse/*` code, and the normative six-row panic-message-template table has no entry for this code as a panic, so the runtime message is undefined. The closed `diagnostic-shape.md#located-site-classification` partition assigns every `loom/parse/*` row to **Located** (file + single-token range), which a type-unresolvable runtime occurrence cannot satisfy.

## Issue introduction
**Verdict:** single introducing commit
**Introducing commit:** `0c9ce54` — *pi-loom spec: resolve "Stringification of `${expr}` interpolations is unspecified"* (2026-05-04)
**History:** The corpus is git-tracked. `git log -S` over the defect token shows the `${expr}` stringification rule entered whole at `0c9ce54`: that commit's diff added both the `Result<T, E>` parse-error table row and the runtime-fallback clause ("when the type is unresolvable … the runtime renderer falls back to a panic carrying the same `loom/parse/interpolated-result` diagnostic code"). The closed panic-source enumeration and the panic-message-template framing it collides with already existed earlier the same day (`8eea523` "fill implementation-blocking gaps" at 08:04; `0c230a1` "Panic message content per source unspecified" at 17:47), so the clause was authored against an already-closed `loom/runtime/*` panic contract without reconciling the parse-namespace code against it. The `loom/runtime/internal-error` runtime-defect surface that the cleanest fix routes through was formalised one day later at `9d7b1b0` (2026-05-05), so at authoring time the natural escape hatch did not yet exist. The later file split (`31ff060`, `f5e89f4`) and the `f5e89f4` size-cap edit moved the prose but did not introduce or repair the tension.

## Solution approach

Rewrite the QRY-18 Notes runtime-fallback clause and the `loom/parse/interpolated-result` row trigger prose so the unresolvable-type runtime occurrence is routed through the existing `loom/runtime/internal-error` runtime-defect surface rather than described as raising the parse code as a panic. Keep `loom/parse/interpolated-result` a parse-time-only diagnostic (Phase `type`, Located). Add a forward-link from the `error-model.md` runtime-defect-surface paragraph to this occurrence. Clarify QRY-21's panic enumeration so it neither lists nor implies `interpolated-result` as a runtime panic.

## Solution constraints

- Out of scope: the closed six-source panic enumeration at `error-model.md#runtime-panics`, its six-row message-template table, and the `diagnostic-shape.md#located-site-classification` partition — none may gain a row or category for this code.
- Co-resolve with F-2199 — both close on the same internal-error-vs-new-code choice.
