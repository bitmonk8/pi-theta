## Finding

The **Field rules** block under *Loom literal sublanguage* in `grammar.md` enumerates the field-presence semantics for `BareObjectLit` and `NamedObjectLit` (the object forms admitted in `params:` defaults and Pi-tool call arguments). It states one rejection arm — *"Every declared field … MUST be present. Omissions are `loom/parse/missing-object-field`"* — but says nothing about the opposite arm: a field present in the literal that the LHS schema (or named-variant schema) does **not** declare (a surplus or typo'd key). The block presents itself as the complete field-semantics enumeration for the literal sublanguage, yet it surfaces only the missing-field half.

The corpus does define the missing arm's counterpart: `expressions.md` *Object construction* states *"extra fields are `loom/parse/extra-object-field`"*, and `loom/parse/extra-object-field` is registered in `diagnostics/code-registry-parse.md`. So a diagnostic exists; the defect is the **local asymmetry** in `grammar.md` — the literal-sublanguage owner restates `missing-object-field` but neither restates nor forward-links the extra-field arm. A parser author who implements the literal sublanguage from this block alone (it is the normative owner for these positions) gets missing-field rejection but silently accepts surplus/misspelled keys in `params:` defaults and Pi-tool arguments.

A second gap compounds this for the Pi-tool-argument position specifically: there the object shape comes from an **externally-owned** Pi-tool input schema, not a Loom schema, so the `extra-object-field` framing ("a field not declared by the schema", phrased for Loom schemas with `additionalProperties: false`) does not transparently transfer — whether surplus fields are rejected depends on the external schema's `additionalProperties`. The Field rules give no disposition for that case.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** f841b12 ("spec: resolve Cluster 2 (literal sublanguage + grammar appendix)", 2026-05-04)
**History:** The corpus is git-tracked. `git log --follow -S 'Every declared field of the LHS schema' -- docs/spec_topics/grammar.md` attributes the *Field rules* block to a single commit, f841b12, which authored the literal-sublanguage section with the `loom/parse/missing-object-field` bullet and no extra/undeclared-field bullet. `git log -S 'extra-object-field' -- docs/spec_topics/expressions.md` shows the `extra-object-field` diagnostic already existed in `expressions.md` as of an earlier commit the same day (839db71, 17:43; f841b12 landed at 22:52), so the asymmetry was baked in at authoring time — the block restated `missing-object-field` while omitting an arm that was already part of the corpus. The omission has persisted unchanged since that section's inception.

## Solution Space

**Shape:** single

### Recommendation

Add a Field-rules bullet to `grammar.md`'s *Loom literal sublanguage* section, parallel to the existing missing-field bullet, stating that a field present in the literal but not declared by the LHS schema (or the named-variant schema) is rejected as `loom/parse/extra-object-field`. This keeps the Field rules in lock-step with `expressions.md` *Object construction*, which already names both arms; a forward-link to that section is acceptable in lieu of restating the diagnostic name, but the extra-field arm must be visible from the literal-sublanguage owner rather than discoverable only by following the intro link.

Edge cases the implementer must watch:

- The Pi-tool-argument position takes its shape from an externally-owned Pi-tool input schema, not a Loom schema with `additionalProperties: false`. Do **not** assert blanket surplus-field rejection there; defer the surplus-field disposition for that position to the Tool Calls owner (see the related Pi-tool field-presence finding), and have the Field rules note that the extra-field arm applies as specified to Loom-schema-supplied shapes (`params:` defaults, named-variant constructors), with the Pi-tool external-schema case governed by Tool Calls.
- Keep the discriminator-field carve-out intact: implicit discriminator fields supplied by a variant schema are not "undeclared" keys.

## Consequence

**Severity:** advisory

An implementer reading `grammar.md`'s Field rules as the complete literal-sublanguage field contract implements missing-field rejection but not surplus-field rejection, silently accepting typo'd or extra keys in `params:` defaults and Pi-tool arguments; the authoritative `extra-object-field` rule exists in `expressions.md` and the registry, so a faithful implementer who follows the cross-link still reaches it, but the local enumeration is incomplete and the externally-owned Pi-tool-schema case has no stated disposition.
