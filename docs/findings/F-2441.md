## Finding

The **Surfacing** rules in `cancellation.md` give a child invoke whose signal aborts two distinct envelopes to the parent:

> A child invoke whose signal aborts surfaces to the parent as `Err(QueryError { kind: "invoke_callee", inner: { kind: "cancelled", ... } })` when the abort originated inside the child, or directly as `kind: "cancelled"` when the parent's own signal fired first.

The discriminator phrases — "when the abort originated inside the child" vs "when the parent's own signal fired first" — are not disjoint in the case that matters most: a parent-initiated cancellation (operator Esc) that fires *while the child is already in flight*. Under **Propagation**, that abort travels *down* the derived controller and is observed at one of the child's own checkpoints, so it both "originated [from] the parent's signal" and "was observed inside the child." A reader cannot tell which envelope applies, and two implementers will diverge: one surfaces bare `kind: "cancelled"` (because the parent's signal is the cause), the other surfaces `kind: "invoke_callee", inner: { kind: "cancelled" }` (because the child observed and propagated it).

The spec already contains the authoritative boundary, but the Surfacing arm does not reference it. The `invoke(...)`-entry forwarding rule states: *"If the parent's signal is already aborted at child-spawn time, the child surfaces `Err(QueryError { kind: "cancelled", ... })` synchronously without spawning a session."* That is the true bare-`cancelled` case — the parent's signal is already aborted **at the pre-invoke checkpoint**, so no session spawns and the parent's own checkpoint surfaces `cancelled` directly. Every other case — the parent's signal fires after dispatch, propagating down to an in-flight child, *or* the child cancels for its own internal reason — is observed inside the child and must surface as `invoke_callee` wrapping inner `cancelled`. The fix is to pin the bare-`cancelled` arm to the pre-invoke-checkpoint (child-not-spawned) condition rather than the temporally vague "fired first."

The same ambiguous two-arm phrasing is mirrored in `errors-and-results/error-model.md`'s per-cause caller-surfaces table (Cancellation row: "a child-internal abort wraps `invoke_callee`…; a parent-own-signal abort surfaces bare `cancelled`"), so the disambiguation must be applied there too to keep the two pages in lockstep.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** fae85d3 — Tighten spec for implementation-plan readiness (2026-05-04, Thomas Andersen)
**History:** The two-arm child-invoke Surfacing rule was authored in fae85d3, which first introduced the `## Surfacing` section into `spec.md`; the disambiguator "when the parent's own signal fired first" was never tied to a checkpoint boundary, so the in-flight-propagation case was ambiguous from the moment the arm appeared. A later same-day commit (8395c03, 2026-05-04) added the authoritative `invoke`-entry rule ("if the parent's signal is already aborted at child-spawn time, the child surfaces `cancelled` synchronously without spawning") but did not reconcile the Surfacing arm against it, leaving the gap. Subsequent edits (8455d5e) only renamed the variant (`invoke_callee_error` → `invoke_callee`) and never touched the temporal phrasing.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the child-invoke Surfacing arm to key the two cases on the pre-invoke checkpoint, not on temporal "fired first":

- **Bare `Err(QueryError { kind: "cancelled", ... })`** applies *only* when the parent's signal was already aborted at the pre-invoke checkpoint — the child is never spawned and the parent's own checkpoint surfaces the cancellation. This is the case the `invoke(...)`-entry forwarding rule already describes; cross-reference it by anchor.
- **`Err(QueryError { kind: "invoke_callee", inner: { kind: "cancelled", ... } })`** applies whenever the abort is observed *inside* the child after it has spawned — both the child's own internal cancellation and a parent-initiated abort that propagated *down* to the in-flight child via the derived controller.

Apply the identical disambiguation to the Cancellation row of the per-cause caller-surfaces table in `errors-and-results/error-model.md` so both pages state the boundary in the same terms.

Edge cases the implementer must watch: a parent-initiated in-flight cancellation surfaces `invoke_callee` from the invoke expression even though the parent's own `loomAbort` is also aborted — CNCL-5 forbids retroactively rewriting that returned `Err` when the parent's next checkpoint subsequently observes its own abort. Correct `V17a`'s propagation test bullet to assert `invoke_callee`/inner `cancelled` for the child-observed case and reserve bare `cancelled` for the parent-already-aborted-at-pre-invoke-checkpoint case.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on which `QueryError` discriminant a parent-initiated in-flight cancellation of a child invoke produces — bare `cancelled` vs `invoke_callee`/inner `cancelled`. This changes the wire form loom authors must handle in `match` arms (an arm written for one form silently ignores the other) and changes whether SLSH-5 chain attribution appends a ` from <callee> invoked at <parent>:<line>` suffix. `V17a`'s own propagation test bullet currently asserts a child cancelling internally surfaces bare `Err(cancelled)`, which contradicts the `invoke_callee` arm and would have to be corrected.
