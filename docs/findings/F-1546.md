## Problem

`invocation.md` §Argument arity admits an `invoke(...)` / `.loom`-callable call that omits trailing defaulted `params:` positionals (arity-legal when the callee is statically resolvable), but nothing in `invocation.md`, `tool-calls.md`, or `frontmatter.md` states what happens to an omitted defaulted position between arity-acceptance and the enforcement-point-#4 AJV validation. The lowered `params` schema carries `required: [every property]` with `additionalProperties: false`, while the only existing default-fill contract (`binder/defaulting-system-note-echo.md` §Defaulting, plus the relaxed-copy schema) is explicitly scoped to the slash-command binder path. Two conforming implementers diverge: one fills the declared default before AJV and the call succeeds; another runs AJV against the full lowered schema and the call fails with `Err(InvokeInfraError { cause: "validation", ... })`. The arity rule's permissiveness and the lowered schema's `required` check cannot both be obeyed without an explicit fill-before-validate rule on the code-side path.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commit:** `833d8d5` — *spec: resolve five invoke edge cases (Decision #4)* (2026-05-05)
**History:** Commit `833d8d5` introduced the *Argument arity* section on the `invoke(...)` path, including the rule that *too few arguments* means "fewer than the count of non-defaulted `params:`" — i.e. omitting trailing defaulted positionals is arity-legal — and the runtime safety-net text `Err(InvokeInfraError { reason: "validation", ... })` for the not-statically-resolvable case. The same commit's diff did not introduce any matching default-fill step on this path. The matching default-fill contract for the *binder* path (the fill-if-absent rule and the relaxed-copy `<params-schema-with-defaulted-fields-relaxed>`) was introduced later, on the binder shard, by `771a08f` (*Defaulting — fill semantics when binder supplies a defaulted field*, 2026-06-03) and `816c398` (binder envelope rewrite, 2026-05-04), both explicitly scoped to the slash-command boundary. The asymmetry — code-side arity admits defaulted omission, code-side runtime behaviour silent on default-fill — has been present since `833d8d5` and was not addressed by the later commits that hardened the binder-path defaulting (`963e07a`, `9bd29ab`, `f5e89f4`); none of those touched the `invoke(...)` default-fill question, only adjacent concerns (non-trailing ordering, `cause` field name, file relocation).

## Solution approach

Add a normative rule on `invocation.md` §Argument binding (or a new sub-paragraph after §Argument arity) for the code-side path: when an `invoke(...)` or `.loom`-callable call omits trailing defaulted `params:` positionals, the runtime fills each omitted position from the param's declared default (fill-if-absent — a supplied value is never overwritten) before the enforcement-point-#4 AJV validation runs, with AJV validating against the full lowered `params` schema rather than the binder-path relaxed copy. Add a scope note in `binder/defaulting-system-note-echo.md` §Defaulting forward-linking to the new `invocation.md` rule so the binder-path and code-side defaulting contracts read as a matched pair. Update the enforcement-point-#4 `invoke(...)` arm in `schema-subset.md` §Depth Enforcement to name the post-default-merge AJV step, mirroring the binder-path qualifier already present in that row.
