## Finding

[`invocation.md` — Argument arity](../../docs/spec_topics/invocation.md) permits a code-side caller to pass *fewer than the count of non-defaulted `params:`* and, when the callee is statically resolvable, reports only over- and under-non-defaulted-count arity, not omission of trailing defaulted params (i.e. an `invoke("./c.loom", a, b)` against a callee declaring `params: { a: T1, b: T2, c: T3 = lit }` is statically arity-legal). The same applies to `.loom`-callable calls through `tools:` entries via the "same arity rules apply" cross-reference. Nothing in `invocation.md`, `tool-calls.md`, or `frontmatter/frontmatter-fields-a.md` then states what happens to the omitted `c` between arity-acceptance and AJV validation: is the declared default literal filled in before AJV runs, or does AJV see an `args` object missing `c`?

The defaulting contract that *does* exist — [`binder/defaulting-system-note-echo.md` §Defaulting](../../docs/spec_topics/binder/defaulting-system-note-echo.md) and the binder envelope's `<params-schema-with-defaulted-fields-relaxed>` in `binder-bypass-and-envelope.md` — is explicitly scoped to the slash-command binder path: the runtime fill-if-absent happens *after the binder returns*, the schema AJV validates against is a *relaxed copy* with defaulted fields removed from `required`, and the post-default-merge AJV is enforcement point #4's binder-path arm in [`schema-subset.md` — Depth Enforcement](../../docs/spec_topics/schema-subset.md). Enforcement point #4's `invoke(...)` arm in the same table surfaces `Err(InvokeInfraError { cause: "validation", ... })` but says nothing about whether defaults are filled first or which schema AJV consumes. The lowered `params` schema that the rest of the pipeline emits carries `required: [every property]` with `additionalProperties: false`, so without an explicit fill-before-validate rule on the `invoke(...)` path, AJV will reject any arity-legal-but-defaulted-field-omitting call.

Two conforming implementers will diverge: one fills declared defaults before AJV (matching the binder-path mental model and the arity rule's permissiveness), one runs AJV directly against the full lowered schema and the call fails with `cause: "validation"`. A third may run AJV against the relaxed copy on this path too. The arity rule and the runtime outcome contradict each other under the literal reading.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commit:** `833d8d5` — *spec: resolve five invoke edge cases (Decision #4)* (2026-05-05)
**History:** Commit `833d8d5` introduced the *Argument arity* section on the `invoke(...)` path, including the rule that *too few arguments* means "fewer than the count of non-defaulted `params:`" — i.e. omitting trailing defaulted positionals is arity-legal — and the runtime safety-net text `Err(InvokeInfraError { reason: "validation", ... })` for the not-statically-resolvable case. The same commit's diff did not introduce any matching default-fill step on this path. The matching default-fill contract for the *binder* path (the fill-if-absent rule and the relaxed-copy `<params-schema-with-defaulted-fields-relaxed>`) was introduced later, on the binder shard, by `771a08f` (*Defaulting — fill semantics when binder supplies a defaulted field*, 2026-06-03) and `816c398` (binder envelope rewrite, 2026-05-04), both explicitly scoped to the slash-command boundary. The asymmetry — code-side arity admits defaulted omission, code-side runtime behaviour silent on default-fill — has been present since `833d8d5` and was not addressed by the later commits that hardened the binder-path defaulting (`963e07a`, `9bd29ab`, `f5e89f4`); none of those touched the `invoke(...)` default-fill question, only adjacent concerns (non-trailing ordering, `cause` field name, file relocation).

## Solution Space

**Shape:** single

### Recommendation

In `invocation.md` §Argument binding (or as a new sub-paragraph immediately after §Argument arity), add a normative rule for the code-side path: *On `invoke(...)` and `.loom`-callable calls through `tools:` entries, when the call omits one or more trailing defaulted `params:` positionals (arity-legal per §Argument arity), the runtime fills each omitted position from the param's declared literal default (per [Parameters and Frontmatter — Defaults](./frontmatter.md)) before the [enforcement-point-#4 AJV validation](./schema-subset.md#depth-enforcement) runs. AJV validates against the **full lowered `params` schema** (the same schema the rest of the pipeline emits, with `required:[every property]` and `additionalProperties: false`) — the relaxed copy `<params-schema-with-defaulted-fields-relaxed>` is binder-path-specific and is not used on the code-side path. A typed argument supplied at a defaulted position is preserved unchanged (fill-if-absent, matching the binder-path semantic).*

Cross-link from `binder/defaulting-system-note-echo.md` §Defaulting with a one-sentence scope note ("this section scopes the *binder* path; the code-side default-fill on the `invoke(...)` / `.loom`-callable path is owned by [`invocation.md` §Argument binding](../invocation.md#argument-binding)") so the two paths' defaulting contracts read as a matched pair rather than the binder-path rule being mistaken for the universal one. Update the enforcement-point-#4 row in `schema-subset.md`'s per-boundary table so the `invoke(...)` arm names the post-default-merge AJV step (mirroring the binder-path's already-present "post-default-merge" qualifier), with the same `cause: "validation"` surface and `InvokeInfraError` carrier already pinned there.

Edge cases the implementer must respect:

- The fill-if-absent rule applies only to omitted *trailing* defaulted positionals — a non-trailing default is already structurally rejected at `params:` parse time (`loom/parse/non-trailing-default`).
- A type-checked argument supplied at a defaulted position (e.g. `invoke("./c.loom", a, b, explicit_c)`) survives unchanged; defaults never overwrite a supplied value.
- The default literal is reconstructed under the [Runtime Value Model](./runtime-value-model.md) — `Enum.Variant` defaults retain their runtime enum brand, and discriminated-union variant defaults retain their `kind` discriminator — matching the binder-path obligation pinned in `frontmatter-fields-a.md` §Defaults.
- The fill step runs *before* AJV (and therefore before the depth-walk that `schema-subset.md` pins as the first sub-check at enforcement point #4); a depth-6 default literal — though unusual — would surface as the same `maxDepth` violation that any other depth-6 `args` payload would on this boundary.
- The not-statically-resolvable arity-too-few fallback to `Err(InvokeInfraError { cause: "validation", ... })` is unaffected: the safety-net path applies when the callee cannot be parse-cache-resolved, and the new fill step requires the callee's `params:` schema, which is exactly what static resolvability supplies.

## Consequence

**Severity:** correctness

Two conforming implementations will diverge on whether an arity-legal `invoke("./c.loom", a, b)` against `params: { a: T1, b: T2, c: T3 = lit }` succeeds (one fills `c` before AJV) or fails with `Err(InvokeInfraError { cause: "validation", ... })` (one runs the full lowered schema's `required` check directly). The same divergence reproduces on every `.loom`-callable code-side call through `tools:` entries. The arity rule's permissiveness and the lowered `params` schema's `required:[every property]` cannot both be obeyed without an explicit default-fill step the spec does not currently install.
