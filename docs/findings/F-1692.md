## Problem

PIC-52 (`id="pic-52"` in `docs/spec_topics/pi-integration-contract/trust-boundary.md`) closes its no-additional-access-channels prohibition with "widening or narrowing this rule is a major-version concern." Because the underlying rule is a prohibition, "widening or narrowing this rule" reads two opposite ways: narrowing could mean relaxing the prohibition (allowing a previously-forbidden channel) or strengthening it (forbidding more channels). A future spec editor cannot tell which class of follow-on edit forces a loom 2.0 bump and which is admissible in a minor release, so PIC-52's purpose of time-binding the no-extra-mediation guarantee is undermined.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `52219c4` ("Address all 41 spec-review findings", 2026-05-06) added the "widening or narrowing this rule is a major-version concern" clause as part of the original *No additional access channels* NFR text under the trust-boundary cluster fix.
**History:** The clause has carried verbatim through `e4017a8` (2026-06-25, F-1260 relocation to `trust-boundary.md`) and `1acdc55` (2026-06-25, F-1344 PIC-52 anchor coinage). Neither relocation nor anchor coinage changed the wording. The ambiguity has therefore existed for the entire lifetime of the rule itself; no later edit caused the defect.

## Solution approach

Rewrite the trailing clause of the PIC-52 sentence so the version-bump trigger names both edit directions in terms of the prohibition itself rather than via the meta-reference "this rule." State the relax direction (permitting any of the listed channel categories) and the strengthen direction (covering categories beyond the listed three) as distinct major-version concerns. Since the defect is the wording, align the verbs with those already used in the version-bump topics.

## Solution constraints

- Out of scope: adding any new normative obligation — the edit clarifies PIC-52's existing governance follow-on, not its MUST-NOT.
