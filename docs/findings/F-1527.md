## Finding

Step 2(b)'s branch tie-breaks make branch (3) (add a declared `// allow-pi-surface:` exemption) "reserved for the strict subset of cases where the offending reference resolves to exactly one callsite in the audited source tree (a single `<path>:<line>:<symbol>` triple, mechanically checkable on the audited source tree at bump time)". This single-callsite predicate is a normative branch-eligibility gate — a contributor MUST pick the branch whose criterion the violation satisfies, and a reviewer MUST confirm the fit on the bump-commit diff — yet `<symbol>`, the component that determines what counts as "one" callsite, is never defined at this gate.

The page emits the triple notation inline with no cross-reference. There *is* a `symbol` field defined in the *Per-family record-shape table* in `audit-failures.md` (e.g. for a family-(2) named import the value is the package-qualified imported name `<package-specifier>#<imported-name>`, which is the SDK-side member rather than the local binding), but the branch (3) gate does not bind its `<symbol>` token to that field. The gate is a counting predicate over the source tree, not over emitted audit records, so a contributor reading `version-bump-step2b.md` in isolation has no defined granularity for "one callsite."

The two plausible readings diverge. Under the SDK-member-path reading, an SDK surface aliased to two local names collapses to a single symbol; under a local-binding-name reading it counts as two. Because the gate decides exempt-vs-promote eligibility, two reasonable contributors can reach different branch selections for the same violation, and a reviewer cannot mechanically check a predicate whose counting unit is unstated.

## Solution Space

**Shape:** single

### Recommendation

At the branch (3) tie-break in `version-bump-step2b.md`, bind `<symbol>` explicitly to the `symbol` field defined in the *Per-family record-shape table* in `audit-failures.md`, and state that the callsite-counting unit is the resolved SDK surface (per family: the `pi.`/`ctx.` member identifier, or the package-qualified imported name `<package-specifier>#<imported-name>` for named imports), not the local binding name. Add the explicit consequence that an SDK surface reached through two distinct local aliases counts as the same `<symbol>`, so aliasing alone does not multiply callsites for the purpose of the single-callsite gate. A bare cross-reference to the *Per-family record-shape table* `symbol` row is sufficient; do not re-define `symbol` independently here (that would risk drift between the two sites).

Edge case the contributor must watch: two import lines of the same SDK surface are two `<path>:<line>` pairs and therefore two callsites regardless of the `<symbol>` reading — the gate disqualifies branch (3) in that case on the `path`/`line` components, independent of the `symbol` clarification.

## Consequence

**Severity:** correctness

Two reasonable contributors can select different resolution branches (exempt vs promote) for the same surface violation, depending on whether `<symbol>` is read as the resolved SDK member or the local binding name; the divergence is most acute for an SDK surface imported under two local aliases. The reviewer obligation to "check that fit on the bump-commit diff" is unenforceable while the counting unit is unstated.
