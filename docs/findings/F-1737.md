## Problem

The preamble opening the `loom/runtime/*` registry section in `docs/spec_topics/diagnostics/code-registry-runtime.md` restates, code by code, the identity, trigger category, routing/delivery channel, and spec-rule pointer of every non-panic code — content the table row immediately below already carries verbatim in its *Code*, *Trigger*, and *Spec rule* columns. The first paragraph runs to roughly a screen of prose mirroring the table, working against the "row is the source of truth" framing the section's other paragraphs adopt. Two preamble facts are *not* row-duplicated and must survive any trim: the closure statement that loom 1.0.0 has exactly six panic sources, and the second paragraph's cross-row rule that a caught throw's interpolated error message is rendered via the underlying-error coercion. The duplication invites silent desync when a future row-level edit forgets the matching preamble restatement.

## Issue introduction

**Verdict:** multi-commit-accretion
**Introducing commits:** `11b9199` (first "One additional `loom/runtime/*` code" sentence, May 4 2026), then accreted across `1e2a6c7`, `833d8d5`, `9e6bb1f`, `3e6e2c0`, `9d7b1b0`, `2128e5f`, `9568906`, `0b195eb`
**History:** `git log -G 'additional `loom/runtime' -- spec_topics/diagnostics.md docs/spec_topics/diagnostics/code-registry-runtime.md` shows the preamble started at `839db71` (May 4 2026) as a single sentence: *"V1 has exactly five panic sources. Each surfaces through the channels defined in [Errors and Results — Runtime panics] …"* — no per-code pre-summary. Commit `11b9199` appended *"One additional `loom/runtime/*` code — `system-note-delivery-failed` — is not a panic but a delivery-failure diagnostic …"*, establishing the additive pattern. Each subsequent commit that introduced a new non-panic runtime code (`subagent-dispose-failure`, `registry-swap-failed`, `registration-cache-collision`, `internal-error` in `9d7b1b0`, `reload-teardown-timeout`, `active-set-restore-failed`, `subagent-model-unresolved` in `8bbf887`, `validator-cache-collision` in `2128e5f`, `custom-type-unsafe` in `9568906`, `watcher-terminated` in `0b195eb`) extended the pre-summary by another "One further code — …" / "X further codes — …" sentence rather than letting the new table row speak for itself. No single commit introduced the duplication; the defect is the interaction of an additive editing habit with a growing row count, which only crossed the screen-of-prose threshold over the cumulative sequence.

## Solution approach

Delete the per-code "One further code — …" / "Six further codes — …" sentences from the first preamble paragraph in `code-registry-runtime.md`, including the standalone `reload-teardown-timeout` delivery-channel sentence whose `console.error`-routing content is already carried in that row's *Trigger* column. Preserve the panic-source-closure statement citing `errors-and-results/error-model.md#runtime-panics` and the second paragraph's underlying-error coercion rule citing `placeholder-rendering-b.md#underlying-error-coercion`. Any structural summary that replaces the deleted prose must not introduce new normative content — normative claims belong in a row, not the preamble.
