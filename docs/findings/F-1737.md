## Finding

The preamble paragraph that opens the `loom/runtime/*` registry section in `docs/spec_topics/diagnostics/code-registry-runtime.md` walks the table's non-panic codes one by one — "One further code — `loom/runtime/internal-error` — covers …", "Six further codes — `system-note-delivery-failed`, `subagent-dispose-failure`, … — are delivery-, rebuild-, registration-, teardown-, or active-set-restore-failure diagnostics …", "One further code — `loom/runtime/subagent-model-unresolved` — is …", and another five "One further code —" sentences for `validator-cache-collision`, `reload-teardown-timeout` (its differing delivery channel), `watcher-terminated`, and `custom-type-unsafe`. For every code mentioned, the preamble restates the code identity, its trigger category, its routing/delivery channel, and the spec-rule pointer — content that the row immediately below already carries verbatim in its *Code*, *Trigger*, and *Spec rule* columns.

The pre-summary survives only because it accreted: each spec change that introduced a new runtime code appended a fresh "One further code —" sentence on top of the existing prose instead of either reshaping the preamble or letting the new row speak for itself. The duplication is now substantial — the first paragraph alone runs to roughly a screen of prose mirroring information that is then re-emitted in the table — and it actively works against the "row is the source of truth" framing the section's other normative paragraphs adopt.

Two pieces of preamble content are *not* duplicated by any row and must survive any trim: (a) the closure statement that loom 1.0.0 has exactly six panic sources, identified by matching the table's *Code* cells against the panic-source list in `errors-and-results/error-model.md`, and (b) the cross-row rule in the preamble's second paragraph that a caught throw's error message interpolated into any row's *Message template* or `message`/`hint` field is rendered via the underlying-error coercion in `placeholder-rendering-b.md`. A third snippet — the DIAG-2 carve-out clarification that the closed-registry rule "constrains codes, not triggers" — is also unique to the preamble but is the subject of a sibling finding that argues for relocating it; the trim here neither preserves nor deletes it unilaterally.

## Issue introduction

**Verdict:** multi-commit-accretion
**Introducing commits:** `11b9199` (first "One additional `loom/runtime/*` code" sentence, May 4 2026), then accreted across `1e2a6c7`, `833d8d5`, `9e6bb1f`, `3e6e2c0`, `9d7b1b0`, `2128e5f`, `9568906`, `0b195eb`
**History:** `git log -G 'additional `loom/runtime' -- spec_topics/diagnostics.md docs/spec_topics/diagnostics/code-registry-runtime.md` shows the preamble started at `839db71` (May 4 2026) as a single sentence: *"V1 has exactly five panic sources. Each surfaces through the channels defined in [Errors and Results — Runtime panics] …"* — no per-code pre-summary. Commit `11b9199` appended *"One additional `loom/runtime/*` code — `system-note-delivery-failed` — is not a panic but a delivery-failure diagnostic …"*, establishing the additive pattern. Each subsequent commit that introduced a new non-panic runtime code (`subagent-dispose-failure`, `registry-swap-failed`, `registration-cache-collision`, `internal-error` in `9d7b1b0`, `reload-teardown-timeout`, `active-set-restore-failed`, `subagent-model-unresolved` in `8bbf887`, `validator-cache-collision` in `2128e5f`, `custom-type-unsafe` in `9568906`, `watcher-terminated` in `0b195eb`) extended the pre-summary by another "One further code — …" / "X further codes — …" sentence rather than letting the new table row speak for itself. No single commit introduced the duplication; the defect is the interaction of an additive editing habit with a growing row count, which only crossed the screen-of-prose threshold over the cumulative sequence.

## Solution Space

**Shape:** single

### Recommendation

Replace the first paragraph of the `loom/runtime/*` preamble with two short sentences:

1. A panic-source-closure sentence — "loom 1.0.0 has exactly six **panic sources** — the closed list enumerated in [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics); the table below carries one row per code in that list, identifiable by matching each row's *Code* cell against that closed list." (Keep the trailing channels parenthetical — slash-command system note vs `Err(QueryError { kind: "invoke_infra", cause: "panic", ... })` — because no row column states the panic *channel*.)
2. A row-as-source-of-truth sentence pointing readers at each row's *Trigger*, *Spec rule*, and the runtime-defect-surface notes carried in the affected rows themselves — e.g. "All other `loom/runtime/*` codes (the runtime-defect surface and the delivery-/rebuild-/registration-/teardown-/active-set-restore-/precondition-/cache-/watcher-lifecycle-/binder-input-precondition diagnostics) are defined in full by their table row below, including trigger, routing channel, and back-reference to the owning normative rule."

Delete every per-code "One further code — `loom/runtime/<name>` — …" / "Six further codes — …" sentence in the first paragraph, including the standalone "`reload-teardown-timeout` differs from the other five in delivery channel: …" sentence whose unique content (the `console.error` routing rationale) is already carried in the `reload-teardown-timeout` *Trigger* column.

Preserve the preamble's second paragraph verbatim — the cross-row rule pinning underlying-error coercion for caught-throw messages is unique to the preamble and has no row equivalent.

Do not unilaterally move or delete the DIAG-2 trigger carve-out sentence ("The code is registered and stable; only its trigger condition is intentionally open-ended …"); leave its fate to the sibling finding that owns its placement.

Edge cases the implementer must watch:

- After the trim, the preamble must still satisfy any cross-file links that target text it currently contains (search for incoming links to the registry section before deleting the per-code sentences; if any peer file deep-links to a phrase that only the pre-summary carries, repoint the link to the relevant row anchor or to the row's *Spec rule* destination).
- The panic-source-closure sentence must continue to cite `errors-and-results/error-model.md#runtime-panics`; that anchor is the source of truth for the closed panic-source list and the `loom/runtime/cancelled-by-session-shutdown` row's commentary depends on the count being stated exactly once in the preamble.
- The replacement sentences must not introduce a new normative claim — they only summarise structural facts already carried elsewhere. Any sentence that would qualify as new normative content belongs in a row, not in the preamble.

## Consequence

**Severity:** cosmetic

A reader of the registry section sees each non-panic code described twice — once in the preamble pre-summary and once in the table row — with no normative difference between the two descriptions. Implementers and audit tooling key off the row columns, so no observable behaviour shifts; the cost is reader effort and a higher risk that a future row-level edit silently desynchronises from a forgotten preamble restatement.
