## Finding

The MVP feature pair `M-T` (tests task) / `M` (implementation task) describes the SLSH-2 obligation asymmetrically. `M-T` carries **two** `SLSH-2` Tests bullets:

- "a dispatched prompt-mode loom issues one untyped `@`-query and streams its **assistant response** into the user session as one appended turn."
- "running the fixture loom through the harness produces exactly one appended turn and no diagnostic."

`M` carries a **single** `SLSH-2` bullet covering only the appended-turn streaming assertion — the "no diagnostic" clause has no counterpart in `M`'s Tests or Ships-when. An implementer reading `M`'s scope as the contract sees no "no diagnostic" obligation, even though the failing-test task `M-T` asserts it.

The asymmetry also violates the `conventions.md` Leaf-format rule that each Tests entry is "one bullet per **REQ-ID**": `M-T` lists `SLSH-2` twice. The two M-T bullets describe one happy-path observation (a single appended prompt-mode turn, emitting no diagnostic) split across two lines with no second REQ-ID to justify the split.

## Issue introduction

**Verdict:** introduced by a single later edit
**Introducing commits:** `c06749d` — "pi-loom plan: resolve 'Doc-updates convention unenforced / mislabeled Tests bullets'" (2026-06-11)
**History:** At plan inception (`c6a664e`) `M-T`'s second Tests bullet was a `Convention:` (*Doc updates*) bullet reading "running the fixture loom through the harness produces exactly one appended turn and no diagnostic", and `M` carried a single `SLSH-2` bullet. Commit `c06749d` relabeled that second `M-T` bullet from `Convention:` (*Doc updates*) to `SLSH-2` without reconciling `M`, simultaneously creating the two-vs-one `SLSH-2` asymmetry and the one-bullet-per-REQ-ID violation in `M-T`. `M`'s single `SLSH-2` bullet has carried only the appended-turn assertion since inception, unchanged by `c06749d`.

## Solution Space

**Shape:** single

### Recommendation

Consolidate `SLSH-2` to one bullet per leaf, carrying both the appended-turn and the no-diagnostic clauses, and mirror that bullet across `M-T` and `M` so the tests/impl pair aligns 1:1 and the one-bullet-per-REQ-ID convention holds.

- In `docs/plan_topics/M-T-minimal-slash-command.md`, replace the two `SLSH-2` Tests bullets with a single `SLSH-2` bullet that folds in the no-diagnostic clause, e.g.:
  `` `SLSH-2`: a dispatched prompt-mode loom issues one untyped `@`-query and streams its **assistant response** into the user session as exactly one appended prompt-mode turn, with no diagnostic emitted. ``
- In `docs/plan_topics/M-minimal-slash-command.md`, update the single `SLSH-2` Tests bullet to mirror it, adding the no-diagnostic clause, e.g.:
  `` `SLSH-2`: a real `pi /<name>` dispatch through the harness streams the untyped query's **assistant response** into the user session as exactly one appended prompt-mode turn, with no diagnostic emitted. ``

Do not re-introduce the "no diagnostic" observation as a separate `Convention:` (*Doc updates*) Tests bullet: the *Doc updates* convention has no mechanical gate and is verified at the self-review step, so it is not a Tests assertion — that mislabel is what `c06749d` was reconciling. Keep the observation attached to `SLSH-2`, its actual owning obligation.

## Consequence

**Severity:** advisory

The implementation leaf `M` turns `M-T`'s tests green, so the "no diagnostic" assertion is still binding through `M-T` and a correct happy-path leaf does get produced; the practical risk is a scope-description mismatch rather than a shipped defect. The mismatch nonetheless misleads an implementer reading `M`'s Tests as the authoritative scope and leaves `M-T` in violation of the one-bullet-per-REQ-ID convention, eroding the 1:1 tests/impl alignment the per-phase TDD ritual relies on.
