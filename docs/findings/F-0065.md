## Problem

The *Sequential by default* rule in `conventions.md` bans synchronous event-loop stalls in `src/**` production code (`fs.readFileSync`, `execSync`, the `*Sync` family, busy-wait loops), but its **Enforcement posture** paragraph concedes the entire ban to manual review: the `H2a` `no-restricted-syntax` rule matches only the four Promise-combinator forms, not blocking sync calls, so detection rests on seam-adapter discipline and the per-phase self-review step. The named high-value forms (`fs.readFileSync`, `execSync`, the `*Sync` family) are statically detectable as cheaply as the Promise combinators the existing allow-list already catches. Conceding the whole ban rather than mechanically closing the detectable subset lets a leaf be tagged `<id>-complete` with a blocking `readFileSync`/`execSync` present and nothing reddens. Only the genuine residue a static scan cannot witness (busy-wait loops, dynamically-constructed sync calls) is irreducibly manual.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commit:** `c70ebce` — "pi-loom plan: resolve \"Sequential by default omits Never block the async runtime\"" (2026-06-12)
**History:** The *Sequential by default* rule originally covered only the four Promise-combinator forms. Commit `c70ebce` added the blocking-runtime ban (subsuming the CLAUDE.md "Never block the async runtime" directive) and, in the same edit, authored the **Enforcement posture** paragraph that explicitly concedes "no mechanical gate" and routes detection to seam-adapter discipline plus the self-review step. The un-gated posture entered the corpus together with the ban it qualifies — it was not a later regression but a concession baked in at the moment the ban was added while resolving a prior review finding.

## Solution approach

Add an approximating static check at the `H2a` lint surface (the same surface wiring the `no-restricted-syntax` Promise-combinator allow-list) that flags the synchronous blocking `*Sync` forms the ban names — including `fs.readFileSync` and `execSync` — over `src/**`, with a comment-keyed line-level exemption mirroring the surrounding allow-list pattern and the `V8*` seam-adapter modules excluded. Add the corresponding `H2a` Tests and Ships-when coverage for the new check. Revise the `conventions.md` *Sequential by default* **Enforcement posture** paragraph so it no longer claims the whole ban is un-gated: state that the named `*Sync`/`execSync` forms are mechanically caught and that only the residue a static scan cannot witness routes to self-review. Sequence the `H2a` gate ahead of the `conventions.md` revision so the posture describes a gate that exists.

## Solution constraints

- The spec is read-only for this fix; do not introduce a spec obligation for the blocking-runtime ban — it is plan-owned cross-cutting discipline.
