## Finding

The `?`-operator section of `expressions.md` states: "A function or loom that uses `?` thus implicitly returns `Result<T, QueryError>` where `T` is the type of its last expression." This is narrower than — and contradicts — the authoritative return-type rule in `functions.md` FN-3 (`#loom-return-type`).

FN-3 defines the inferred return type as the *least upper bound* (under `⊑`) of the tail-expression type **and** the success-payload of every early `return expr` syntactically present in the body. The success arm `T` of the implicit `Result<T, QueryError>` is therefore not "the type of its last expression" alone; for a body that early-returns values of types other than the tail's, `T` is the common upper bound across the tail and all `return` operands (or `loom/parse/return-no-common-type` if none exists).

An implementer who reads only the `?` section computes `T` from the tail expression alone, omitting the early-`return` contributions. For multi-`return` functions this yields a different (too-narrow) `T` than FN-3 mandates, and skips the `loom/parse/return-no-common-type` diagnostic. The two normative sections must agree on a single rule.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 53f5831 (`spec: rewrite query primitive, system prompts, and schema syntax`, 2026-05-03) and 08886d9 (`pi-loom spec: resolve "loom return-type inference does not reconcile early-return operand types"`, 2026-06-07)
**History:** The phrase "`?` … implicitly returns `Result<T, QueryError>` where `T` is the type of its last expression" entered the corpus in 53f5831, in the then-monolithic `spec.md`. At that point return-type inference said only that "the same inference rule that applies to a tail expression applies to `return`'s argument" — no cross-`return` reconciliation — so the simplification was internally consistent. The contradiction emerged in 08886d9, which introduced the FN-3 least-upper-bound-of-tail-plus-every-`return`-operand rule but left the `?`-section phrase unchanged, making it stale. A later commit (87dfa1d, 2026-06-28) relocated the stale sentence verbatim from `error-model.md` into its current home in `expressions.md`, carrying the defect forward without correcting it. The defect is thus the product of the FN-3 refinement not being propagated to the `?`-section phrasing, not a single faulty edit.

## Solution Space

**Shape:** single

### Recommendation

In the `?` section of `expressions.md`, replace "where `T` is the type of its last expression" with a statement that defers to FN-3 rather than restating an inference rule. For example: "A function or loom that uses `?` thus implicitly returns `Result<T, QueryError>`, where `T` is the success-payload type determined by the Loom return-type rule (see [Functions — Loom return type](./functions.md#loom-return-type)) — the least upper bound of the tail expression and every early-`return` operand's success payload."

Keep the cross-reference pointing at the `#loom-return-type` anchor so the single source of truth for return-type inference stays in FN-3; the `?` section should describe only the early-return/unwrap semantics and link out for how `T` is computed. Do not re-derive the LUB discipline inline (that would re-create the same drift risk).

## Consequence

**Severity:** correctness

Two implementers reading the conflicting sections diverge on the inferred success type `T` for any `?`-using body that also early-returns a differently-typed operand: the `?` section yields the tail type alone while FN-3 yields the LUB across tail and all `return` operands. The divergence also changes whether `loom/parse/return-no-common-type` fires, so conforming implementations would accept/reject the same loom differently.
