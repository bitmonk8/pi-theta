## Finding

In version-bump step 5, the sentence governing where the build-time `SessionShutdownEvent['reason']` type-equality assertion lives is written as a `whether … or …` construction:

> "The assertion MUST be reached by the `npm run typecheck` invocation … — whether the implementation places its source file under `test/` (alongside the surface-inventory test) or under `src/extension/` (**a long parenthetical**), the loom `tsconfig` MUST include it, and a split-tsconfig layout that excludes it is a violation of this step …"

The second branch's parenthetical (opened at "or under `src/extension/`") has grown to hold three distinct statements — (1) the MUST that an `src/extension/`-placed file match the `src/extension/**/*.assert.ts` suffix convention, (2) the MUST-NOT prohibiting inlining the assertion into a non-suffix-matching `src/extension/**/*.ts` file, and (3) a self-contained disclaimer that "The audit's authorisation of `Type.Unsafe` (per typebox carve-out) is unrelated to this prohibition." Because the parenthetical spans multiple full sentences and its closing `)` falls immediately before "the loom `tsconfig` MUST include it", the reader cannot tell from the punctuation whether the `tsconfig`-inclusion MUST is (a) a top-level obligation that applies to **both** placements, or (b) a continuation scoped only to the `src/extension/` branch. The intended reading is (a) — the assertion must be type-checked regardless of placement, and excluding it from `tsconfig` is a violation in either layout — but the structure admits the reading that `tsconfig` inclusion is only required when the file sits under `src/extension/`.

The defect is purely structural punctuation/scoping: a normative MUST is left dangling off the end of an over-long parenthetical attached to only one of two enumerated alternatives, so its scope (and arguably its normative status, versus being read as part of the parenthetical aside) is ambiguous.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `75a9626` (2026-05-09, "resolve 'SessionShutdownEvent.reason closed set has no build-time pin against the SDK type'"), `e5cd514` (2026-05-09, "resolve 'T35 SDK capability inventory closed-set negative-direction audit'"), `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** The corpus is git-tracked; `git log -S`/`-G` over the constituent clauses shows no single commit authored the ambiguous structure — it accreted. `75a9626` introduced the `tsconfig`-inclusion / split-tsconfig MUST clause. `e5cd514` grafted the `*.assert.ts` suffix-convention rule and the standalone `Type.Unsafe`-is-unrelated disclaimer into the same parenthetical. `f5e89f4` (the page-collapse/size-cap reorganisation) folded these accreted clauses into the current `whether … under test/ … or under src/extension/ ( … )` sentence, at which point the closing `)` came to sit immediately before the dangling `tsconfig` MUST. A sibling finding "normative MUST buried inside non-normative parenthetical" was resolved on this same page (`05656bb`, 2026-06-05) without addressing this `)`-boundary scoping ambiguity.

## Solution Space

**Shape:** single

### Recommendation

Restructure the sentence so the three branch-specific statements and the placement-independent obligation are separated:

1. Close the `src/extension/` parenthetical at the inlining prohibition (after "… structurally separable from any runtime code under `src/extension/`. Inlining would couple …"), so the parenthetical contains only the two placement MUSTs.
2. Move the `Type.Unsafe`-is-unrelated disclaimer out of the parenthetical into its own sentence following it (it is a cross-reference disclaimer, not part of the placement rule).
3. Promote "The loom `tsconfig` MUST include the assertion file, and a split-tsconfig layout that excludes it is a violation of this step rather than a silent demotion to the runtime-only `loom/host/session-shutdown-reason-unknown` fallback." to a standalone sentence that follows the `whether … or …` placement sentence, making explicit that it applies to **both** placements.

Edge cases the editor must preserve: keep the existing cross-references (`*Audit scope*` for the suffix convention, the typebox carve-out disclaimer's referent) intact, and retain the existing wording of the two placement MUSTs — only the sentence boundaries and the scope of the `tsconfig` obligation change.

## Consequence

**Severity:** advisory

An implementer who reads the `tsconfig`-inclusion MUST as scoped to the `src/extension/` branch could place the assertion under `test/` and omit it from the loom `tsconfig`, defeating the gate's intent. The risk is contained because the same step's overarching "The assertion MUST be reached by the `npm run typecheck` invocation" obligation already guarantees the assertion is type-checked regardless of layout, so a careful reader still reaches a working configuration; the cost is reader effort and the standing hazard of a mis-scoped MUST.
