## Problem

In version-bump step 5, the sentence governing where the build-time `SessionShutdownEvent['reason']` type-equality assertion lives is a `whether … under test/ … or under src/extension/ ( … )` construction whose second branch's parenthetical has grown to span multiple full sentences (the `*.assert.ts` suffix MUST, the non-suffix-inlining prohibition, and a standalone `Type.Unsafe`-is-unrelated disclaimer). The parenthetical's closing `)` falls immediately before "the loom `tsconfig` MUST include it", so the reader cannot tell from the punctuation whether that `tsconfig`-inclusion MUST is a top-level obligation applying to **both** placements or a continuation scoped only to the `src/extension/` branch. The intended reading is placement-independent, but the structure admits the narrower reading. The defect is purely structural punctuation/scoping — a normative MUST is left dangling off the end of an over-long parenthetical attached to only one of two enumerated alternatives.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `75a9626` (2026-05-09, "resolve 'SessionShutdownEvent.reason closed set has no build-time pin against the SDK type'"), `e5cd514` (2026-05-09, "resolve 'T35 SDK capability inventory closed-set negative-direction audit'"), `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** The corpus is git-tracked; `git log -S`/`-G` over the constituent clauses shows no single commit authored the ambiguous structure — it accreted. `75a9626` introduced the `tsconfig`-inclusion / split-tsconfig MUST clause. `e5cd514` grafted the `*.assert.ts` suffix-convention rule and the standalone `Type.Unsafe`-is-unrelated disclaimer into the same parenthetical. `f5e89f4` (the page-collapse/size-cap reorganisation) folded these accreted clauses into the current `whether … under test/ … or under src/extension/ ( … )` sentence, at which point the closing `)` came to sit immediately before the dangling `tsconfig` MUST. A sibling finding "normative MUST buried inside non-normative parenthetical" was resolved on this same page (`05656bb`, 2026-06-05) without addressing this `)`-boundary scoping ambiguity.

## Solution approach

Restructure step 5's `whether … or …` placement sentence so the `tsconfig`-inclusion / split-tsconfig MUST reads unambiguously as a placement-independent obligation rather than a continuation of the `src/extension/` parenthetical. Narrow the parenthetical to the two `src/extension/`-branch placement MUSTs, and move the `Type.Unsafe`-is-unrelated disclaimer out of the parenthetical so it stands as its own cross-reference disclaimer.

## Solution constraints

- This is a clarity-only restructure: preserve the normative content of the two placement MUSTs and the existing cross-references (the `*Audit scope*` suffix-convention reference and the typebox carve-out disclaimer's referent); only sentence boundaries and the scope of the `tsconfig` obligation change.
