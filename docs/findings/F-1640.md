## Finding

The glossary preamble (`docs/spec_topics/glossary.md`) declares the page strictly descriptive and non-authoritative: *"Each entry is descriptive — a one-paragraph reminder of what a term means and where it is used. The canonical, normative definition lives on the feature page named in the entry's `See:` reference; if a glossary entry and its canonical page disagree, the canonical page wins."* This establishes a single, unambiguous authority rule — for any term, the feature page governs and the glossary merely mirrors it.

Three entries contradict that rule by carrying standalone normative directives that exist nowhere else in the corpus:

- **`callable set`** — *"Authors and implementers MUST use the term `callable set` … the bare phrases `tools: set`, `tool set`, `loom's tools`, and `available tools` are spec-prose synonyms to be avoided."*
- **`schema slug`** — *"Authors and implementers MUST use `schema slug` (or the bare form `slug` …). The synonyms `schema hash`, `schema-hash`, `sha12`, `lowered-schema hash`, and `lowered-schema content hash` are spec-prose drift to be avoided; a future grep gate uses this entry as its source of truth."*
- **`Pi tool` vs. `.loom callable`** — *"Authors and implementers MUST use the canonical term `.loom callable` … the synonyms `registered loom`, `registered subagent loom`, `registered-loom call`, and `loom callee` are spec-prose drift to be avoided."*

None of these MUSTs is presented as a ratified copy of a feature-page obligation, and no feature page is named as their authority. The canonical pages run the dependency the other way: `tool-calls.md` cites *"[Glossary — `callable set`]"* as the term's reference, and the `schema slug` entry explicitly declares itself *"the source of truth"* for a future grep gate. The glossary is therefore the sole bearer of these obligations while simultaneously declaring itself non-authoritative — a two-authority contradiction. A reader who applies the preamble's "canonical page wins" rule treats the MUSTs as non-binding reminders and looks to the feature page for the real constraint, finding none; a reader who takes the MUSTs at face value treats the glossary as binding. The two readings diverge on whether, e.g., `tool set` is a forbidden synonym.

The contradiction is reinforced by the corpus's own governance machinery: `glossary.md` is registered in the REQ-ID prefix table (`governance/req-id-prefix-table-active-b.md`) as `(no IDs — narrative)`, which structurally exempts it from the un-anchored-MUST closing gate (plan leaf `H5e`). These three normative MUSTs thus escape both REQ-ID anchoring (GOV-3) and the gate's coverage-matrix reconciliation — they are normative obligations sitting on a page classified as non-normative narrative, and consequently are unenforced.

**History:** `git log --follow -S "MUST use the term \`callable set\`"`, `-S "MUST use \`schema slug\`"`, and `-S "MUST use the canonical term"` on `docs/spec_topics/glossary.md` each return one commit (the two above plus `b4fbfac` for the page itself). The defect is the interaction: each later fix migrated a corpus-wide naming MUST into the entry it considered the canonical home (each subject was already cross-referenced from multiple feature pages), without revisiting whether the glossary's narrative-page framing still held after that MUST landed there. Each commit is locally well-motivated in isolation; the rule that breaks is one neither commit edited.

## Solution Space

### Recommendation

Relocate each terminology MUST and its forbidden-synonym list out of `glossary.md` and onto the canonical feature page that owns the term, so the preamble's existing "canonical page wins / descriptive" rule applies uniformly with no carve-out. Designate exactly one owning page per term:

- `callable set` MUST (and the forbidden synonyms `tools: set`, `tool set`, `loom's tools`, `available tools`) → `frontmatter.md`.
- `.loom callable` MUST (and the forbidden synonyms `registered loom`, `registered subagent loom`, `registered-loom call`, `loom callee`) → `frontmatter.md`.
- `schema slug` MUST (and the forbidden synonyms `schema hash`, `schema-hash`, `sha12`, `lowered-schema hash`, `lowered-schema content hash`) → `schema-subset.md`, Canonical schema hash subsection.

For each term:

- Add the normative paragraph to the owning feature page, anchored with a REQ-ID or a `coverage-matrix.md` row with a closing leaf, so the relocated MUST is subject to the un-anchored-MUST gate (`H5e`) and disposition it in the coverage matrix.
- Delete the MUST sentence and its synonym list from the glossary entry, leaving the entry as a descriptive one-paragraph reminder of what the term means with a `See:` reference pointing to the relocated obligation on the owning page.
- Repoint the `schema slug` grep-gate "source of truth" reference from the glossary entry to its new home in `schema-subset.md`.

Keep each term's MUST on a single designated owning page (do not split a MUST between `frontmatter.md` and `tool-calls.md`) to avoid re-introducing drift across candidate homes. Execute the three separate per-entry placement findings in this review (see Relationships) as the per-site work of this single authority decision, not as independent edits.

## Consequence

**Severity:** correctness

Two reasonable spec authors diverge on whether the glossary MUSTs bind: one applies the preamble's "canonical page wins / descriptive" rule and treats them as non-binding reminders (looking to a feature-page authority that does not exist), the other treats them as authoritative. Because `glossary.md` is classified as `(no IDs — narrative)`, the obligations also escape the un-anchored-MUST gate, so the terminology constraints they assert are structurally unenforced. The defect degrades the spec's terminology-governance discipline rather than runtime behaviour, so implementation can proceed.
