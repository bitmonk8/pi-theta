## Problem

The glossary preamble (`docs/spec_topics/glossary.md`) declares the page strictly descriptive and non-authoritative â€” "the canonical, normative definition lives on the feature page named in the entry's `See:` reference â€¦ the canonical page wins." Yet three entries â€” `callable set`, `schema slug`, and `Pi tool` vs. `.loom callable` â€” carry standalone `Authors and implementers MUST use â€¦` directives, each with a forbidden-synonym list that exists nowhere else in the corpus and names no feature page as its authority. The page is therefore simultaneously the sole bearer of these obligations and self-declared non-normative, a two-authority contradiction that leaves readers diverging on whether the listed synonyms are actually forbidden. Because `glossary.md` is registered as `(no IDs â€” narrative)` in the REQ-ID prefix table, the MUSTs also escape the un-anchored-MUST gate (plan leaf `H5e`) and are structurally unenforced.

## Solution approach

Move each terminology MUST and its forbidden-synonym list off `glossary.md` onto the feature page that owns the term â€” `callable set` and `.loom callable` to `frontmatter.md`, `schema slug` to `schema-subset.md`'s Canonical schema hash section (`#canonical-schema-hash`) â€” so the preamble's existing "canonical page wins / descriptive" rule applies uniformly with no carve-out. Add a REQ-ID or coverage-matrix anchor to each relocated obligation so it falls under the un-anchored-MUST gate. Delete the MUST sentence and synonym list from each glossary entry, leaving a descriptive one-paragraph reminder whose `See:` reference points at the relocated obligation. Rewrite the `schema slug` grep-gate "source of truth" reference to point at its new home in `schema-subset.md`.

## Solution constraints

- Each term's relocated MUST lives on exactly one owning page; do not split a term's MUST across multiple feature pages (e.g. `frontmatter.md` and `tool-calls.md`), which would re-introduce cross-page drift across candidate homes.
- Anchor each relocated MUST under the owning page's already-registered REQ-ID prefix (or a coverage-matrix row); do not silently introduce a new prefix â€” a new prefix requires a GOV-7 row surfaced for human review.
