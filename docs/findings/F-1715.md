## Finding

The "Re-scan deduplication" paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` §24 contains two adjacent claims with mismatched normative force. The first sentence — "A watcher-triggered reload re-emits the persistent diagnostic for any file whose contents are still broken" — is descriptive indicative prose, not a MUST. The only normative clause in the paragraph is the trailing "the renderer MUST NOT attempt to suppress duplicates."

A renderer-side non-suppression rule does not, on its own, oblige the runtime to *produce* the duplicate in the first place. A conforming implementation could legitimately read the paragraph as: (a) we describe the current behaviour as one in which a reload re-emits, (b) we forbid the renderer from filtering — but nothing forbids the runtime from short-circuiting re-emission when it observes that the file's contents (or the resulting `Diagnostic` set) are unchanged since the last load. That reading would silently drop the recurring author-facing diagnostic line the paragraph's rationale ("Authors will therefore see the same diagnostic line recur after each reload until the underlying file is fixed") relies on, and the V7d / V7d-T plan leaves' test "A re-scan re-emits without dedup/supersede" would have no normative anchor in the spec to point at.

The cross-reference from `docs/spec_topics/discovery/package-and-settings.md` §"Watcher-time reload failures" ("re-emitted on this path rather than at first load (per the **Re-scan deduplication** rule in [Diagnostic shape])") also assumes a normative re-emit obligation exists at the cited rule; today the citation lands on a descriptive sentence.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `ac8c3af` (2026-05-04, "pi-loom spec: resolve \"LoadExtensionsResult.errors is the wrong delivery channel for loom diagnostics\"")
**History:** The "Re-scan deduplication" paragraph was added in a single hunk by `ac8c3af` into `spec_topics/diagnostics.md` (later moved to `docs/spec_topics/diagnostics/diagnostic-shape.md` by the `docs/` reorganisation and the 100KB shard split `f5e89f4`). The wording at first commit already paired the descriptive "A watcher-triggered reload re-emits the persistent diagnostic for any file whose contents are still broken" sentence with the renderer-only "the renderer MUST NOT attempt to suppress duplicates" clause — the asymmetry has never been a MUST on the runtime, and no later commit (`a6d82db` `V1`→`loom 1.0` rename, `f5e89f4` size-cap split) altered the normative shape of the paragraph. `git log -G "watcher-triggered reload re-emits"` returns only the introducing commit and the two non-substantive follow-ups.

## Solution Space

**Shape:** single

### Recommendation

Promote the descriptive runtime sentence to a normative obligation in the same paragraph. Replace

> A watcher-triggered reload re-emits the persistent diagnostic for any file whose contents are still broken.

with

> A watcher-triggered reload MUST re-emit the persistent diagnostic for every still-broken file, with no dedup or suppression against prior emissions for the same `(code, file)` or any other key.

Keep the existing renderer-side "the renderer MUST NOT attempt to suppress duplicates" clause as-is; it now reinforces a producer-side MUST rather than carrying the entire obligation alone. The paragraph's existing rationale ("Pi's `pi.sendMessage` API has no documented 'remove' or 'replace by metadata' counterpart … Authors will therefore see the same diagnostic line recur after each reload until the underlying file is fixed; this is the loom 1.0 contract") survives unchanged and now follows from a stated MUST rather than from descriptive prose.

Implementer edge cases the wording must cover:
- "Still broken" means the re-parse / re-merge produces a `Diagnostic` of the same `code` for the same `file`. The MUST re-emit even when the produced `Diagnostic` is byte-identical to the previous one (no producer-side content-equality dedup).
- The MUST applies per watcher-triggered reload pass; it does not turn the bounded `loom/runtime/system-note-delivery-failed` fallback chain in V7d into a retry loop.

## Consequence

**Severity:** advisory

Two reasonable implementers reading today's wording could legitimately differ on whether the runtime is obliged to re-emit on every watcher reload or merely permitted to. The descriptive framing combined with a renderer-only MUST gives a runtime that suppresses unchanged-since-last-load re-emissions a defensible reading; the V7d tests would then catch the divergence late, but the spec itself does not force it.
