## Problem

`V3e` (and its paired tests leaf `V3e-T`) — Expression stdlib members — implements the standard-library member surface for three unrelated value types in one leaf: the `string` members (plus the `String.replace` reference vectors and `concat`), the `array<T>` members, and the `object` members — roughly fifteen members across three independent type namespaces, each carrying its own behaviour table, failure mode, and ordering guarantee. The `string`, `array<T>`, and `object` member surfaces each ship and test independently; none depends on another's behaviour. Packing all three into `V3e`/`V3e-T` is coarser than the plan's own leaf-sizing rule (`conventions.md` §Vertical slices: "each leaf inside a slice is the smallest feature that can ship and be tested independently").

## Issue introduction

**Verdict:** introduced (multi-commit interaction)
**Introducing commits:** `20b4689` ("pi-loom plan: resolve \"Stdlib members beyond replace/concat have no named assertion\"", 2026-06-11) introduced the three-namespace bundle; `cb8737a` ("pi-loom plan: resolve \"Systemic leaf over-bundling across the plan corpus\"", 2026-06-12) carried it forward into the present `V3e` leaf.
**History:** The plan corpus is git-tracked. The stdlib member surface first lived inside the expression-evaluator test leaf (`V3a-T`). Commit `20b4689` added the named `string`, `array<T>`, and `object` member assertions to that one leaf in a single diff, creating the three-namespace bundle. Commit `cb8737a`, resolving an earlier corpus-wide over-bundling finding, extracted the stdlib surface out of `V3a`/`V3a-T` into the new `V3e`/`V3e-T` pair (`git log --follow` records the file as a copy from `V3a-T-expression-evaluator.md`) — but extracted all three namespaces together, so the split that was meant to reduce bundling preserved the three-namespace bundle as `V3e`. The defect is durable from `20b4689` onward; `cb8737a` relocated rather than introduced or fixed it.

## Solution approach

Split the `V3e`/`V3e-T` paired leaves so each independently-shippable stdlib member surface (`string`, `array<T>`, `object`) lands in its own leaf, allocating new IDs under the V3-group `<group><letter>` scheme. Sequence the work so the new paired leaves are authored and the old `V3e`/`V3e-T` removed before retargeting references, so the retarget pass lands against stable new IDs. Then retarget the four reference sites that name `V3e`: the `coverage-matrix.md` `expressions.md` (EXPR) closing-leaf row, `V3a`'s Adds. forward-link, `H5b`'s Deps. range, and the `plan.md` V3 vertical-slice TOC.

## Solution constraints

- The spec (`expressions.md`) is read-only for this fix — it is a plan-packaging change; the member behaviour content is unchanged.
- New leaf IDs follow the Plan-rules `<group><letter>` scheme (next free letter, never reuse a retired ID) and each feature keeps its bare `<id>` impl / `<id>-T` tests pairing.
- Keep `coverage-matrix.md` and `H5b`'s `Deps.` consistent after the split: the EXPR coverage row must list the new closing leaves, and each must remain in `H5b`'s `Deps.` per the transitive-completeness plan-maintenance rule.
