## Problem

The `PIC-17, PIC-18` row in `coverage-matrix.md` is a split row (two leaves: `V9c`, `V15d`), so `conventions.md` item 7 (*Split-row joint-completeness*) applies. Item 7's first branch assigns a facet to each *annotated* leaf, but only `V15d` carries a parenthetical facet annotation; `V9c` is bare. The bare leaf falls into a gap between item 7's two branches — neither named as a facet nor marked a co-witness — so its facet membership is indeterminate and the row's facet partition cannot be derived mechanically by a reviewer running the item-7 sweep.

That single edit made the row multi-leaf (so item 7 now applies) and annotated `V15d` but left `V9c` un-annotated — introducing the asymmetric partition in one commit, as collateral of the prior fix. No later commit touched the row.

## Solution approach

On the `PIC-17, PIC-18` row in `coverage-matrix.md`, add a parenthetical facet annotation to `V9c`, mirroring the annotation `V15d` already carries on that row (and the pattern on the `DISC-4` row). The facets `V9c` closes are derivable from `V9c-T`'s Tests bullets: the per-query `PIC-17` active-set snapshot/set/restore-in-`finally` and the `PIC-18` prompt-mode `pi.on` subscription. With both leaves annotated, item 7's first branch resolves the full partition.

## Solution constraints

- `V9c`'s `PIC-17` facet must remain distinct from `V15d`'s prompt→prompt-`invoke`-path `PIC-17` facet — both touch the `PIC-17` step-4 `finally` restore, and the partition must keep them as two facets, not collapse into a re-closure of the same path.
