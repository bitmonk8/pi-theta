## Finding

V6a (`docs/plan_topics/V6a-frontmatter-contract.md`) and its paired tests leaf `V6a-T` carry a `loom/load/model-unresolved` Tests bullet that asserts the failing-resolution case "fails the load **and the loom is not registered**", while immediately naming the witnessing mechanism as "the parser's model-resolution hook fires the code against the matcher passed through the model-reference-matcher injection seam `V6a` defines." Those two halves are not co-observable at V6a's position. Registration is owned by `V9b` ("looms discovered are registered; the swap is atomic"), and V6a's Deps are `V6a-T, V1a, V5a` — no registry, registration step, or `LoomRegistry` swap exists in scope. The test mechanism the bullet itself names can only observe the parser hook firing `loom/load/model-unresolved` through the matcher seam; it cannot observe whether a loom was or was not registered, because there is no registration machinery to observe.

The "and the loom is not registered" clause is the spec-level consequence (`frontmatter-fields-a.md §model`, restated in `coverage-matrix.md`'s V6a row) of a load-time failure — true as an obligation, but downstream-verified at `V9b` (and end-to-end at `H7a`), not at V6a. Stated as part of a binding V6a-T Tests bullet under the per-phase TDD ritual, it asks the V6a test author to witness something their harness cannot produce.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 4088e2e — pi-loom plan: resolve "model/bind_* resolution hooks named in V6a Adds with no closing assertion" (2026-06-10, Thomas Andersen)
**History:** The `loom/load/model-unresolved` Tests bullet — including "fails the load and the loom is not registered" — was added to both `V6a-frontmatter-contract.md` and `V6a-T-frontmatter-contract.md` in commit 4088e2e, which closed the previously-unasserted `model:` resolution hook (the same commit added the corresponding `coverage-matrix.md` V6a row). That fix imported the spec's load-time consequence ("the loom is not registered", per `frontmatter-fields-a.md §model`) into a V6a-scoped test where the registration machinery (`V9b`) does not exist. `git log -S "the loom is not registered"` over the V6a leaf pair returns only this commit; no earlier revision carried the clause.

## Solution Space

**Shape:** single

### Recommendation

In the `loom/load/model-unresolved` Tests bullet of both `docs/plan_topics/V6a-frontmatter-contract.md` and `docs/plan_topics/V6a-T-frontmatter-contract.md`, scope the asserted observable to what the named mechanism witnesses at V6a: the parser's model-resolution hook fires `loom/load/model-unresolved` through the model-reference-matcher injection seam for each unresolvable-`model:` case (non-string scalar, malformed reference, no matching available model, bare `modelId` ambiguous across providers). Strike "and the loom is not registered" from the witnessed assertion; if the consequence is worth keeping for traceability, retain it only as an explicit downstream-verified note pointing at `V9b` (registration) — for example phrased as "non-registration is the downstream consequence witnessed at `V9b`" — so it is not read as a V6a-level assertion. Leave `coverage-matrix.md`'s V6a row unchanged: it states the spec obligation (code fires AND loom not registered) with V6a as closing leaf, which is correct at the obligation level. Keep the existing matcher-seam wording and the `V11a` shared-contract cross-reference intact.

## Consequence

**Severity:** advisory

The binding V6a-T Tests bullet asks for an assertion the V6a harness cannot make. An implementer either tries to witness non-registration — pulling in V9b registration machinery that is not a V6a dependency (scope creep / false sequencing) — or silently weakens the test to assert only the code firing, leaving the bullet's "not registered" claim vacuous while `V6a-T`'s "fails red for the intended reason" Ships-when still passes. Either way the bullet overstates what V6a verifies.
