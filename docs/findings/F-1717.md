## Finding

The opening sentence of **Provider error mapping** in
`docs/spec_topics/pi-integration-contract/provider-error-mapping.md` states that
"The matching rules are version-coupled to `@earendil-works/pi-ai` and **MUST**
be re-validated on each upgrade." The very next sentence (the *Re-validation
gate (loom 1.0.0)* paragraph) names the only acceptance mechanism that backs
that obligation: "a contributor performing the bump **SHOULD** run [the
fixtures] against the candidate `@earendil-works/pi-ai` minor per item (i) of
the *Editorial-review checklist for unpinned host presuppositions*". The
paragraph then notes outright that no CI gate is wired ("loom 1.0.0 ships
without a CI-wired bump-procedure step that re-runs the provider-error
fixtures").

The defect is the modal pairing: a MUST-level obligation whose **only** stated
acceptance check is SHOULD-level (and acknowledged to be un-gated). A reader
cannot derive a definitive pass/fail signal — under MUST a bump that skipped
the fixtures has shipped non-conforming, but under the SHOULD acceptance
mechanism the same bump is conforming. The two modals contradict.

The same pairing recurs in `version-bump-step2.md` item (i), which is internally
consistent ("The contributor **SHOULD** re-run … This obligation is SHOULD-level
only at loom 1.0"); the contradiction lives entirely in the
provider-error-mapping page's choice of modal for the headline obligation.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `82be125` (`spec: adopt P1 typed-query two-phase loop (Option A)`, 2026-05-04) — earliest commit reachable from `main` to introduce a "MUST be re-validated on each upgrade" sentence into the spec corpus.
- `aebec4e` (`pi-loom spec: resolve "Pi version-bump gate cannot detect three unstated host/provider presuppositions"`, 2026-06-04) — added the *Re-validation gate (loom 1.0.0)* sub-paragraph in `provider-error-mapping.md`, initially worded as "a contributor performing the bump **is expected to** run them" (non-modal, so no MUST/SHOULD contradiction yet).
- `c326bfa` (`pi-loom spec: resolve "Shard-13 clarity / cruft cluster on provider-error and session-shutdown pages"`, 2026-06-07) — rewrote that sub-paragraph's "is expected to run them" to "**SHOULD** run them … per item (i) of the *Editorial-review checklist*". This is the commit that introduced the literal MUST/SHOULD pairing on the same paragraph and made the modal contradiction explicit.

**History:** The MUST-level obligation on the matching rules predates the
*Re-validation gate* sub-paragraph by a month. When the sub-paragraph was first
inserted (`aebec4e`) it used the non-modal phrase "is expected to", which is
soft enough that it does not visibly contradict the MUST above it. Three days
later (`c326bfa`) the soft phrase was rewritten as SHOULD to align with the
editorial-review checklist's vocabulary, which is internally consistent — but
the rewrite did not revisit the still-MUST obligation in the immediately
preceding sentence on the same page, surfacing the defect. The durable defect
is therefore the interaction between a pre-existing MUST and a later SHOULD on
adjacent sentences, not a single regression.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/pi-integration-contract/provider-error-mapping.md`, change
the headline modal in the opening sentence of **Provider error mapping** from
MUST to SHOULD: "The matching rules are version-coupled to
`@earendil-works/pi-ai` and **SHOULD** be re-validated on each upgrade." This
brings the obligation into line with its only stated acceptance mechanism — the
*Editorial-review checklist for unpinned host presuppositions*, which is
uniformly SHOULD-level at loom 1.0 by design, with escalation to MUST documented
as the post-1.0.0 maintenance follow-up.

Make no other edits:

- Leave the *Re-validation gate (loom 1.0.0)* sub-paragraph as-is
  (SHOULD-run-fixtures, no CI gate). Do **not** delete it — it carries the
  acceptance-mechanism citation that the SHOULD now points at, and removing it
  would re-open the same modal gap from the opposite direction.
- Leave `version-bump-step2.md` item (i) and `version-bump-intro.md` untouched;
  the SHOULD tiering and the post-1.0.0 escalation path are preserved verbatim.

This is a one-word edit (`MUST` → `SHOULD` on the headline sentence) with zero
blast radius beyond the single sentence; no downstream consumer of
`provider-error-mapping.md` keys on the MUST.

## Consequence

**Severity:** correctness

Two reasonable implementers — or two reasonable reviewers signing off a bump
commit — diverge on whether the gate has been satisfied: one reads MUST and
fails the bump for missing fixture evidence, the other reads SHOULD and lets
the bump land on contributor discipline alone. Worse, the spec offers no
mechanical adjudicator: the un-gated note is explicit that no CI check fires,
so the divergence is resolved per reviewer rather than per evidence.
