## Finding

The opening sentence of **Provider error mapping** in
`docs/spec_topics/pi-integration-contract/provider-error-mapping.md` states that
"The matching rules are version-coupled to `@earendil-works/pi-ai` and **MUST**
be re-validated on each upgrade." The very next sentence (the *Re-validation
gate (loom 1.0.0)* paragraph) names the only acceptance mechanism that backs
that obligation: "a contributor performing the bump **SHOULD** run [the
fixtures] against the candidate `@earendil-works/pi-ai` minor per item (i) of
the *Editorial-review checklist for unpinned host presuppositions*". The
paragraph then notes outright that no CI gate is wired ("loom 1.0.0 ships
without a CI-wired bump-procedure step that re-runs the provider-error
fixtures").

The defect is the modal pairing: a MUST-level obligation whose **only** stated
acceptance check is SHOULD-level (and acknowledged to be un-gated). A reader
cannot derive a definitive pass/fail signal — under MUST a bump that skipped
the fixtures has shipped non-conforming, but under the SHOULD acceptance
mechanism the same bump is conforming. The two modals contradict.

The same pairing recurs in `version-bump-step2.md` item (i), which is internally
consistent ("The contributor **SHOULD** re-run … This obligation is SHOULD-level
only at loom 1.0"); the contradiction lives entirely in the
provider-error-mapping page's choice of modal for the headline obligation.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**

- `82be125` (`spec: adopt P1 typed-query two-phase loop (Option A)`, 2026-05-04) — earliest commit reachable from `main` to introduce a "MUST be re-validated on each upgrade" sentence into the spec corpus.
- `aebec4e` (`pi-loom spec: resolve "Pi version-bump gate cannot detect three unstated host/provider presuppositions"`, 2026-06-04) — added the *Re-validation gate (loom 1.0.0)* sub-paragraph in `provider-error-mapping.md`, initially worded as "a contributor performing the bump **is expected to** run them" (non-modal, so no MUST/SHOULD contradiction yet).
- `c326bfa` (`pi-loom spec: resolve "Shard-13 clarity / cruft cluster on provider-error and session-shutdown pages"`, 2026-06-07) — rewrote that sub-paragraph's "is expected to run them" to "**SHOULD** run them … per item (i) of the *Editorial-review checklist*". This is the commit that introduced the literal MUST/SHOULD pairing on the same paragraph and made the modal contradiction explicit.

**History:** The MUST-level obligation on the matching rules predates the
*Re-validation gate* sub-paragraph by a month. When the sub-paragraph was first
inserted (`aebec4e`) it used the non-modal phrase "is expected to", which is
soft enough that it does not visibly contradict the MUST above it. Three days
later (`c326bfa`) the soft phrase was rewritten as SHOULD to align with the
editorial-review checklist's vocabulary, which is internally consistent — but
the rewrite did not revisit the still-MUST obligation in the immediately
preceding sentence on the same page, surfacing the defect. The durable defect
is therefore the interaction between a pre-existing MUST and a later SHOULD on
adjacent sentences, not a single regression.

## Solution Space

**Shape:** multiple

### Option A — Lower the headline obligation to SHOULD

**Approach.** In `provider-error-mapping.md`, change "The matching rules are
version-coupled to `@earendil-works/pi-ai` and **MUST** be re-validated on each
upgrade" to "**SHOULD** be re-validated on each upgrade". Leave the
*Re-validation gate (loom 1.0.0)* sub-paragraph as-is (SHOULD-run-fixtures, no
CI gate). No edit needed to `version-bump-step2.md` item (i) or
`version-bump-intro.md`.

**Spec edits.**

- One word in `provider-error-mapping.md`, opening sentence of **Provider error mapping**: `MUST` → `SHOULD`.
- Optionally append a half-clause cross-referencing item (i) for the acceptance mechanism, e.g. "… SHOULD be re-validated on each upgrade per item (i) of the *Editorial-review checklist* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below"; this duplicates wording already present in the next sentence, so it is optional.

**Pros.**

- Matches the explicitly-stated loom-1.0.0 reality (no CI gate, fixture re-run is contributor discipline).
- Aligns with the rest of the *Editorial-review checklist* paragraph in `version-bump-step2.md`, which is uniformly SHOULD-level at loom 1.0 by design.
- The escalation-to-MUST path is already documented elsewhere (item (i)'s closing sentence: "wiring the fixture re-run into the bump procedure as a mechanical CI gate is the post-loom 1.0.0 maintenance follow-up"), so no normative content is lost — only the modal is brought into line with the rest of the editorial-review tier.
- One-word edit; zero blast radius beyond the single sentence.

**Cons.**

- A SHOULD-level acceptance check on a version-pin re-validation reads weaker than a downstream reader of just this page would expect; the rationale (no CI gate at loom 1.0) is in the next paragraph but a casual reader scanning headlines may miss it.

**Risks.** Negligible. The change is purely a modal weakening that brings the
prose into alignment with the already-stated acceptance mechanism; no
downstream consumer of `provider-error-mapping.md` keys on the MUST.

### Option B — Raise the acceptance mechanism to MUST

**Approach.** In `provider-error-mapping.md`'s *Re-validation gate (loom
1.0.0)* sub-paragraph, change "SHOULD run them" to "**MUST** run them" and
add a normative phrase such as "a Pi minor bump MUST NOT land until the
provider-error fixtures pass against the candidate minor." Then propagate the
same MUST modal into `version-bump-step2.md` item (i) for consistency
("**MUST** re-run the provider-error fixtures") and into `version-bump-intro.md`
as a named MUST-level gate in the procedure listing. Delete or rewrite the
"loom 1.0.0 ships without a CI-wired bump-procedure step" sentence, since a
MUST-level gate without a mechanical check is itself self-contradictory: the
gate becomes a reviewer obligation.

**Spec edits.**

- `provider-error-mapping.md`: rewrite the *Re-validation gate (loom 1.0.0)* sub-paragraph to assert a MUST-level reviewer gate (with the un-gated-by-CI carve-out either deleted or recast as "the gate is editorial at loom 1.0").
- `version-bump-step2.md` item (i): change "SHOULD re-run" → "MUST re-run"; delete the closing sentence "This obligation is SHOULD-level only at loom 1.0; wiring … is the post-loom 1.0.0 maintenance follow-up" (the SHOULD-level loom-1.0 carve-out is exactly what is being reversed).
- `version-bump-intro.md`: add the fixture-rerun MUST to the enumerated steps so the procedure's "mechanical gates that fail loudly until the corresponding manual step has been completed" sentence does not over-promise.

**Pros.**

- Strengthens the cross-provider error-mapping correctness guarantee at the bump commit.

**Cons.**

- Multi-page edit that touches the bump procedure's overall MUST/SHOULD tiering, not just one sentence.
- Promotes a single editorial-review item out of the otherwise-uniform SHOULD tier in `version-bump-step2.md`, creating a new asymmetry across the eighteen+ items (a)–(ap) of the checklist that will invite follow-up findings asking why items (a)–(h), (j)–(ap) remain SHOULD.
- A MUST-level reviewer gate without a CI check still admits the original defect class (silent skip on a bump commit), just with a stronger paper obligation; the real fix is the post-1.0.0 mechanical gate already named, which is out of scope here.

**Risks.** Re-tiering one item triggers consistency pressure on the rest of
the checklist, expanding scope beyond a single-paragraph modal fix.

### Recommendation

Adopt **Option A**. The *Editorial-review checklist for unpinned host
presuppositions* is uniformly SHOULD-level at loom 1.0 by design, with
escalation to MUST documented as the post-1.0.0 maintenance follow-up; lowering
the headline obligation on `provider-error-mapping.md`'s opening sentence to
SHOULD brings the modal into line with that design without re-tiering the
checklist. The fix is a one-word edit (`MUST` → `SHOULD` on the headline
sentence), leaves the *Re-validation gate (loom 1.0.0)* sub-paragraph and
`version-bump-step2.md` item (i) untouched, and preserves the existing
post-1.0.0 escalation path verbatim. Implementer note: do not also delete the
*Re-validation gate (loom 1.0.0)* sub-paragraph — it carries the
acceptance-mechanism citation that the SHOULD now points at, and removing it
would re-open the same modal gap from the opposite direction.

## Consequence

**Severity:** correctness

Two reasonable implementers — or two reasonable reviewers signing off a bump
commit — diverge on whether the gate has been satisfied: one reads MUST and
fails the bump for missing fixture evidence, the other reads SHOULD and lets
the bump land on contributor discipline alone. Worse, the spec offers no
mechanical adjudicator: the un-gated note is explicit that no CI check fires,
so the divergence is resolved per reviewer rather than per evidence.
