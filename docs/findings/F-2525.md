## Finding

The `model` row of the field-contract table in `frontmatter/frontmatter-fields-a.md` describes the absent-`model:` default as "The loom inherits **the caller's model** at invocation time and pins it for the duration of that one invocation." Every other authoritative statement of the same default names a different referent: the row's own default column reads "Pi session's model at invocation time"; the `model` prose bullet below the table reads "the loom inherits **Pi's session default** at invocation time … re-inherits **`ctx.model`** afresh"; and the host contract (`host-interfaces-core.md`) pins `ctx.model` as "host's currently-selected model", with subagent mode forwarding **the parent's** `ctx.model` (the user-session view), never a loom's internally-pinned model. The lone table-cell phrase "caller's model" is therefore the outlier.

The phrasing is load-bearing because "caller" is an overloaded term in this corpus — the glossary documents two distinct referents (slash caller vs invoke parent), a split flagged by a separate finding. Under the invoke-parent reading, "inherits the caller's model" wrongly suggests an `invoke`-ed child whose frontmatter omits `model:` adopts the parent loom's pinned model. That contradicts the actual mechanism: an invoke child reads `ctx.model`, which `host-interfaces-core.md` forwards as the parent's session-view value (looms never write their pinned model back into `ctx.model`), and `invocation.md` independently states "the caller's settings are not inherited." A reader who stops at the table cell can wire the invoke-inheritance path to the wrong source.

The heading's secondary concern — that the per-invocation pin assumes `ctx.model` is synchronously readable and stable across the invocation — is already settled by the spec. The prose pins read-once-and-snapshot semantics ("fixed for the duration of that one invocation and does not re-resolve per query within that invocation"), so a mid-invocation mutation of `ctx.model` is immaterial, and `ctx.model` is declared as a plain synchronous field (`model: Model<any> | undefined`) on `ExtensionContext`. The live defect is the table-cell referent wording only.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `fb54586` (2026-05-04, "pi-loom spec: resolve \"Frontmatter field required/optional/default contract is unspecified\"")
**History:** `git log -S"inherits the caller's model"` over the spec corpus identifies `fb54586` as the commit that first authored the field-contract table, introducing the `model` row as `| ... | Pi session's model at invocation time | The loom inherits the caller's model and pins it for the loom's lifetime ... |`. The body/default-column term mismatch ("caller's model" body vs "Pi session's model" header) was present from that authoring commit. A later commit, `616f046` (2026-06-04, resolve "Frontmatter `model` — loom's lifetime vs per-invocation inheritance"), rewrote the row and prose to fix the per-invocation snapshot-stability ambiguity (loom's lifetime → "duration of that one invocation"; the heading's secondary concern) but carried the "caller's model" body wording through unchanged, so the referent mismatch persisted into the current text.

## Solution Space

**Shape:** single

### Recommendation

In the `model` row's body cell of the field-contract table, replace "The loom inherits the caller's model at invocation time" with "The loom inherits the Pi session's active model (`ctx.model`) at invocation time", matching the row's own default column ("Pi session's model at invocation time"), the `model` prose bullet, and the `ctx.model` pin in `host-interfaces-core.md`. Keep the existing "and pins it for the duration of that one invocation; a later invocation … re-inherits the session model afresh" tail unchanged. Do not introduce a new term — reuse "Pi session's active model" / `ctx.model` exactly as the prose bullet already does, so the row, prose, and host contract are word-for-word consistent. No behavioural change and no edit to the snapshot-stability prose is required.

## Consequence

**Severity:** advisory

A reader consulting the `model` row in isolation can read "caller's model" as the invoke parent's pinned model and wire absent-`model:` invoke inheritance to the wrong source. The risk is bounded because the same page's prose, the default column, the host contract, and `invocation.md` all converge on the Pi session model (`ctx.model`); a careful implementer reading the full page produces correct behaviour.
