## Finding

The ERR-17 forced-respond non-compliance block in `docs/spec_topics/errors-and-results/queryerror-variants.md` defines the terminal exhaustion envelope as a literal `Err(QueryError { ... })`, including the line `message: "model did not call the forced respond tool"`. The string is quoted verbatim, but the surrounding prose carries no MUST/MAY declaration over it — neither the bullet that introduces the literal ("Terminal exhaustion returns:") nor the `raw_response` disambiguation paragraph immediately below state whether the runtime is obliged to emit this exact byte sequence or whether implementations may substitute their own wording.

This contrasts directly with how `docs/spec_topics/errors-and-results/error-model.md` handles its panic-message literals one block over: a dedicated **Panic message string (normative)** paragraph declares that "a conformant runtime MUST emit the registered string … and conformance tests MAY assert on the exact string," and the six templates that follow inherit that declaration explicitly. The ERR-17 outer message receives no such cover. The two inner `ValidationIssue.message` literals (`"model returned plain text instead of calling the forced respond tool"` and `"model invoked tool '<provider-emitted-tool-name>' instead of the forced respond tool '__loom_respond_<slug>'"`) are pinned by ERR-17's "the `message` literal varies by branch" sentence and the surrounding two-arm contract, so an implementer can defend asserting on them; the outer `ValidationError.message` is the only string in the block with no such anchoring. Plan leaves `V4d` and `V13d` close ERR-17 against this page (per `docs/plan_topics/coverage-matrix.md`), but their *Tests* bullets specify only path/keyword/branch-message shape for the synthesised `ValidationIssue` — they say nothing about the outer envelope's `message` field, so the absence is structurally invisible to the closing gate.

The downstream consequence is small but real: a conformance-test author cannot tell whether `assertEqual(err.message, "model did not call the forced respond tool")` is a legitimate assertion or a brittle implementation-detail test. Two reviewers will reach opposite conclusions in good faith, and the spec offers no citation that resolves the disagreement.

## Solution Space

**Shape:** single

### Recommendation

Add a one-sentence normative declaration to the ERR-17 block that pins the outer `ValidationError.message` literal under the same MUST + MAY-assert framing the **Panic message string (normative)** paragraph already establishes in `error-model.md`. Place the sentence immediately after the terminal-exhaustion code fence (between the existing fence and the `raw_response` disambiguation paragraph) so it scopes unambiguously to the `message:` line in the preceding ```` ``` ```` block and inherits the same "two-arm contract" precedent the inner `ValidationIssue.message` literals already enjoy.

Suggested wording:

> The outer `message` literal is normative: a conformant runtime MUST emit the byte-exact string `"model did not call the forced respond tool"` on the terminal-exhaustion envelope, and conformance tests MAY assert on the exact value. This mirrors the [Panic message string (normative)](./error-model.md#runtime-panics) convention for the closed set of registered runtime-panic templates; like those, the literal carries no `<…>` placeholders and is emitted unchanged on every firing.

Implementer notes the editor must watch:

- Do **not** weaken the wording to "implementation-defined" or "for diagnostic purposes only" — the literal already appears verbatim in a normative code fence and the closer `V4d`/`V13d` plan leaves expect a single fixed shape; admitting a free-form `message` would also retrofit the existing wording into a non-normative example, which is a larger change than a one-sentence pin.
- The declaration covers *only* the outer `ValidationError.message`. The two inner `ValidationIssue.message` branches (plain-text and wrong-tool) are already pinned by the existing "the `message` literal varies by branch and is fixed by the two-arm contract there" sentence and by the explicit code fences; do not duplicate normative cover for them.
- Update the `V4d-T` and `V13d-T` *Tests* bullets in `docs/plan_topics/` to assert the outer `message` equality alongside the existing `ValidationIssue` shape assertions, so the new normative cover is exercised by the closing-gate citing-test scan.
- Pattern-match wording on the `error-model.md` precedent (MUST + "MAY assert on the exact string"), not on a custom phrasing — the symmetry is the whole point of the fix, and a divergent phrasing here would re-open the testability ambiguity at a different surface.

## Consequence

**Severity:** advisory

A conformance test asserting `err.message === "model did not call the forced respond tool"` could be admitted or rejected with equal plausibility. Implementations that round-trip the literal and implementations that substitute their own wording both pass the rest of the ERR-17 contract; the divergence is silent until a downstream consumer pattern-matches on the string. Authors are explicitly invited to pattern-match on `InvokeInfraError.message` for panic discrimination (per `error-model.md`); applying the same instinct to `ValidationError.message` here is a foreseeable foot-gun the spec does not address either way.
