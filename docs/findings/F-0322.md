## Finding

The manual real-host smoke gate scores live-run codes with pass criterion (e): the emitted `loom-system-note` codes must be a **subset** of the committed permitted-code list. That list is, by the plan's own statement, a "manually-maintained, best-effort union … completeness is not mechanically verified," and only the `golden ⊆ permitted` containment is mechanically gated (by `H7a`). Both `real-host-smoke-gate.md` (criterion (e)) and `H7a` document the resulting *stale-omit* error mode: a list missing a benign code a slice can emit produces a false fail — the safe direction, recoverable by extending the list and re-running.

The unaddressed error mode is the opposite one. Because the live smoke run does not assert against `H7a`'s exact-transcript / exact-code-set goldens (a non-deterministic live model is not assumed to reproduce them), criterion (e) is the *only* check the smoke applies to the emitted code set. An over-broad permitted list — one carrying a code the deterministic fixture path should never legitimately emit — therefore lets a genuine behavioural divergence that emits a permitted-but-unexpected code pass criterion (e) silently. The `golden ⊆ permitted` containment gate does not constrain this direction: it only guarantees the in-process gate and the smoke cannot disagree on a code the golden path actually emits; it says nothing about codes the permitted list admits beyond the golden set.

The plan deliberately chose a best-effort union with no emittable-code registry to verify against, and it names the stale-omit residual. It does not name the over-broad residual, which is the one with a soundness consequence for the smoke gate. The gap is a documentation/risk-annotation omission, not a build defect.

## Solution Space

**Shape:** single

### Recommendation

Annotate the over-broad residual at the two sites that own criterion (e) and the permitted-code list, consistent with the plan's deliberate best-effort design (do not introduce a mechanical completeness gate — the plan explicitly chose "no emittable-code registry to verify against," and `V7b`'s registry is a full code registry, not a per-slice emittable-code map that could close the gap):

- In `docs/plan_topics/real-host-smoke-gate.md`, extend the criterion (e) text so it states the over-broad residual alongside the existing stale-omit note: that because the live run scores only against (a)–(e) and not against `H7a`'s exact goldens, criterion (e) is the sole emitted-code check, and an over-broad permitted list therefore admits a divergence that emits a permitted-but-unexpected code without scoring a fail — an accepted limit of the manual best-effort gate. Add that the stale-omit direction yields a false fail recoverable by extending the list and re-running.

- In `docs/plan_topics/H7a-integration-acceptance.md`, in the Adds paragraph and the permitted-code-list Tests bullet that already say "a stale union can omit a benign code a slice emits," add the symmetric statement that an over-broad union can admit a code that masks a real divergence under the smoke's criterion (e), and that only `golden ⊆ permitted` is gated — neither completeness direction is.

Edge case for the author: keep `H6a`'s Ships-when restatement consistent — it currently inherits the "completeness is not mechanically verified" phrasing by reference to `H7a`; if it is left referencing `H7a` it needs no separate edit, but it must not be left asserting a stronger guarantee than `H7a` now documents.

## Consequence

**Severity:** advisory

If the plan ships unfixed, the manual smoke gate's emitted-code check (criterion (e)) has a known-but-undocumented blind spot: an over-broad permitted-code list lets a real divergence that emits a permitted-but-unexpected code pass the only emitted-code check the live run applies, so the gate can report green on a genuine fidelity regression. The plan reads as though the only completeness risk is the safe stale-omit (false-fail) direction; the implementer/auditor is given no recoverability guidance for stale-omit and no acknowledgement that the over-broad direction is an accepted limit of the manual best-effort design.
