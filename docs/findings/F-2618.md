## Finding

The inbound translation pass (`runtime-value-model.md`, *Wire-name translation* → *Inbound*) walks the AJV-validated JSON value and, at each string position, consults the lowering pass's *Named-enum positions* sidecar to decide whether to reattach a declaring-enum tag. That sidecar is **keyed by JSON Pointer into the lowered schema fragment** (`schema-subset.md` Lowering Algorithm step 5), but the inbound pass walks the **data value**. The two address spaces do not coincide, and the procedure that maps a data position to the corresponding schema pointer is never specified.

The mismatch is concrete at three points the walk must cross:

- **Arrays.** A data position is an index (`/items_list/0`, `/items_list/1`), but the schema position for every element is the single `items` subschema (`/properties/items_list/items`). The walk must collapse all data indices onto the one `items` pointer; nothing says so.
- **`anyOf` arm selection.** The lowered schema emits no `discriminator` keyword (Lowering Algorithm step 5 / step 6), so after AJV accepts a value the runtime is not told which `anyOf` arm matched. To know which arm's subschema pointer applies at a union position — and therefore whether that position is an enum — the walk must re-derive the selected arm (via the per-variant `const` discriminator for discriminated object unions, or by re-validating arms for mixed unions). The selection rule, and the tie-break when more than one arm validates, are undefined.
- **`$ref` / `$defs` boundaries.** Named and inline schemas lower to `{ "$ref": "#/$defs/<Name>" }`. The data walk continues straight through a `$ref` with no structural marker, but the sidecar key is rooted in the `$defs` graph. The walk must resolve each `$ref` and compose the canonical `$defs`-rooted pointer to match the sidecar's keys; this resolution is not stated.

Because the correlation is undefined, two conforming implementations can descend the same validated value and arrive at different schema pointers for the same string — so one tags a position as a declaring-enum variant while the other leaves it bare. The same parallel descent also underpins the wire-name rebuild (step (a), "using each schema's translation map"), so the gap is not confined to enum tagging.

## Solution Space

**Shape:** single

### Recommendation

Specify the inbound pass as a **schema-guided parallel descent**: the runtime walks the validated value and the lowered schema in lockstep, starting from the (root) response/`$defs` schema paired with the value root and maintaining the current schema position as a canonical `$defs`-rooted JSON Pointer that matches the sidecar's key form. At each value node:

- **Object** — for each present property (wire name), descend into the schema's `properties/<wire-name>` subschema.
- **Array** — for each element index, descend into the schema's single `items` subschema (all indices map to the one `items` pointer).
- **`$ref`** — resolve `#/$defs/<Name>` and continue at the referenced subschema, composing the pointer in canonical `$defs`-rooted form so it matches the sidecar keys.
- **`anyOf`** — select the one arm the value satisfies and descend into that arm's subschema only: for discriminated object unions use the per-variant `const` discriminator field (the field identified by Lowering Algorithm step 6); for other unions select the first arm (in lowered source order) whose subschema the value validates against under JSON Schema 2020-12 semantics. Pin first-match-in-source-order as the deterministic tie-break when more than one arm validates.
- **Scalar string** — look up the current schema JSON Pointer in the *Named-enum positions* sidecar; if present, reattach the named enum's declaring-enum tag. Otherwise leave the string bare.

State that this same descent drives the wire-name rebuild (step (a)) so the two consumers share one traversal, and note that `$ref` recursion is bounded by the existing depth-5 runtime cap. Edge cases the implementer must watch: arrays collapse every index onto `items`; `anyOf` arm re-selection must be deterministic (first-match-in-source-order); and the walk-derived pointer must be normalised to the same canonical `$defs`-rooted form the lowering pass uses to key the sidecar, or lookups silently miss.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on which validated string positions receive the declaring-enum tag — especially under `anyOf` arms and across `$ref` boundaries — so the same model output yields enum values that compare equal to a locally constructed variant in one runtime and merely string-equal (cross-form `false`) in another. Equality semantics, a core observable of the language, then differ between conforming implementations.
