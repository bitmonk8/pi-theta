## Finding

The **Failure** bullet of the closed terminal-outcomes trichotomy in `error-model.md` carves out, among other things, `Err`s that are *"not discarded via `let _ = …` (or the equivalent `void`-tail-expression form)"*. The phrase **`void`-tail-expression form** is load-bearing — it determines which of two reasonable readings of the trichotomy applies — but the error-model page neither defines it nor links to a definition: there is no `void-tail-expression` anchor anywhere in the corpus, no glossary entry, and no in-shard explanation.

The concept itself is real and *is* defined elsewhere. `query/query-escapes-stringification.md` introduces it inline above QRY-19 (*"A `void`-returning function whose **tail expression** is `@`...`` is also a discard…"*), QRY-20 then uses the spelling **`void`-tail form**, and `pi-integration-contract/runtime-event-channel.md` reuses the term again as **`void`-tail-expression discard / void-tail-function form**. The Failure paragraph in `error-model.md` is the *canonical* statement of which `Err`s count as a Failure terminal outcome, so a reader landing there cannot work out, without a cross-shard treasure hunt, whether the carve-out reaches:

- a `void`-annotated `fn` whose tail expression is `@`...`` (the QRY-19/20 case — yes),
- a `void`-annotated `fn` whose tail expression is any other `Result<T, QueryError>`-typed expression (also discarded by the FN-4 *"explicit `void` return type still discards any tail expression value silently"* rule — yes), or
- an empty-tail body that elaborates implicitly to `null` per FN-4 (no — there is no `Err` at the tail at all).

The wording is also inconsistent across the shards that already use it (`void`-tail form vs. `void`-tail-expression form vs. void-tail-expression discard vs. void-tail-function form), which compounds the lookup problem.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `8168ea8` (2026-05-10, *pi-loom spec: resolve "T30 ceiling #4 model-driven row trichotomy carve-out"*)
**History:** `git log -G 'tail.expression form' -- spec/ spec_topics/` shows `8168ea8` is the earliest commit to add the phrase **`void`-tail-expression form** to the Failure bullet of the terminal-outcomes trichotomy. Before that commit the bullet read *"or exhausted a runtime-class hard ceiling (per … the load-time binder cap is excluded …)"* with no discard clause at all; `8168ea8` rewrote the bullet to add the `match`/`let _ =` carve-out and, in doing so, coined the parenthetical *"(or the equivalent `void`-tail-expression form)"* without an anchor or cross-reference. The underlying concept was already defined in `query.md` since `b3bc4ce` (2026-05-04, *"spec: pin failure-observability surface (Cluster A, Option C)"*) under the spelling **`void`-tail form**, so the defect is precisely a missing cross-shard link plus a spelling drift introduced when the term migrated into a new home — not a fundamental gap in the language semantics.

## Solution Space

**Shape:** single

### Recommendation

In `error-model.md`, replace the bare parenthetical *"(or the equivalent `void`-tail-expression form)"* with a parenthetical that both names the concept canonically and links to the page that defines it. Use the QRY-20 spelling — **`void`-tail discard form** — as the single canonical term, and point at QRY-20 (the discard-observability rule), which is the section that already enumerates both shapes (`let _ = @`...`` and the `void`-tail form) as a single class.

Concretely, the edited Failure bullet should end:

> *"…not consumed by a caller `match` and not [discarded](../query/query-escapes-stringification.md#qry-20) (via `let _ = …`, or via the equivalent `void`-tail discard form — a tail expression whose enclosing `fn` or loom return type is `void`, per [Function Definitions — Empty-tail body](../functions.md#empty-tail-body))."*

Edge cases the fixer must keep straight:

- The carve-out covers only frames whose return type is *explicitly* `void`. An empty-tail body with no annotation infers `null` per FN-4 and never carries a tail `Err`, so it falls outside both arms.
- The `void`-tail discard form applies symmetrically to top-level looms and nested `fn`s. The cross-reference target in `functions.md#empty-tail-body` already covers both.
- While editing, align the existing call sites in `runtime-event-channel.md` (which currently spell the term **void-tail-expression discard** and **void-tail-function form**) and `query-escapes-stringification.md` QRY-20 (which says **`void`-tail form**) to the single chosen spelling `**void`-tail discard form**, so the term is searchable with a single grep.

Do **not** introduce a glossary entry for this — the concept is a one-sentence consequence of FN-4 and is more clearly read in situ at QRY-20 than as a glossary lemma. A glossary entry would be a third spelling to keep in sync.

## Consequence

**Severity:** correctness

Two reasonable implementers reading the Failure bullet in isolation will diverge on what counts as a Failure terminal outcome: one will treat any `Err` that reaches a `void` return-annotated frame as Success (loom body ran to completion, value discarded), the other will treat only the explicit `let _ = …` shape as the discard carve-out and route an `Err` that falls off a tail expression to Failure. Because Failure-vs-Success is the difference between "no final value flows" and "final value `null` flows" at the `invoke` boundary, the divergence is observable to callers.
