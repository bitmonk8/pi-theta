## Finding

In the `?`-operator section of `expressions.md`, the obligation to handle a `Result` is stated as: *"A function that uses neither `?` nor an explicit `Result` return type is required to handle every query failure with `match` (or to discard explicitly per [Query — Discarded query results])."* The phrase **"every query failure"** is the ambiguity: it reads naturally as scoped to the failure of a `@`-query specifically, yet the surrounding machinery is not query-scoped. The same section admits `?` on any operand of static type `Result<T, QueryError>` "for instance a `@`-query, an `invoke(...)`, or an explicit `Ok(...)` / `Err(...)`" (ERR-18), and a code-side tool call `<name>(args)` lowers to `Err(CodeToolError { kind: "code_tool", ... })`, which `host-interfaces-core.md` confirms is a `QueryError` variant. So tool calls and `invoke` produce the same `Result<T, QueryError>` type a `@`-query does.

Two reasonable readings diverge on whether a bare tool-call or `invoke` expression-statement whose `Result` is dropped is a defect:

- **Broad:** the mandatory-handling rule covers every `Result<T, QueryError>`-typed expression; a bare `foo()` (tool call) or bare `invoke(...)` statement that drops its `Result` must be rejected the same way a bare `@`-query is.
- **Narrow:** the rule covers only `@`-queries; tool-call and `invoke` Results may be silently dropped in statement position with no diagnostic.

The ambiguity is sharpened by the enforcing diagnostic. The only must-use parse error in the registry is `loom/parse/discarded-query-result`, and both its registry row (`code-registry-parse.md`) and QRY-19 (`query-escapes-stringification.md`) scope it explicitly to a *"Bare `@`...`` expression-statement"*. There is no `loom/parse/discarded-tool-result` or generic discarded-`Result` diagnostic. Under the broad reading the spec is missing the diagnostic that would enforce the rule for non-query Results; under the narrow reading the spec silently drops tool-call and `invoke` Results in statement position — a must-use hole that contradicts the corpus's "silent success on denial is forbidden" stance. The QRY-19 note that *"`let _ = expr` is a real binding form for any expression — not just queries — making the same escape hatch available to any future `#[must_use]`-style type"* hints the enforcement is intended to generalise, but never says so normatively.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 53f5831 ("pi-loom spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03)
**History:** `git log -S 'required to handle every query'` traces the sentence to 53f5831; its parent (the initial scaffold) contains no such text, so the construct entered the corpus at 53f5831 already carrying the "every query failure" framing while the discard parse error was scoped to bare `@`...`` statements in the same commit. The phrasing was carried verbatim through the F-2145 relocation (87dfa1d, 2026-06-28) that moved the `match`/`?` material from the error-model into the expressions cluster. The `CodeToolError` / `invoke` Result-producing forms were consolidated shortly afterward (around 2026-05-04/05) without reconciling this sentence's scope, so the scope gap has been latent in the text since the sentence was first authored.

## Solution Space

**Shape:** single

### Recommendation

Pin the broad reading: state that the mandatory-handling obligation applies to **every expression of static type `Result<T, QueryError>`** — `@`-queries, code-side tool calls (`CodeToolError`), and `invoke(...)` alike — not only `@`-queries. Replace "every query failure" with "every `Result<T, QueryError>`-typed expression's failure" (or equivalent) at the `expressions.md` site.

Close the enforcement gap this exposes: a bare expression-statement of type `Result<T, QueryError>` that is neither `?`-propagated, `match`-handled, nor `let _ =`-discarded is a parse error regardless of how the `Result` was produced. Either (a) widen `loom/parse/discarded-query-result` to cover any dropped `Result`-typed expression-statement and adjust its registry-row *Trigger* text and QRY-19 wording away from the `@`...``-only framing, or (b) register a sibling diagnostic for the tool-call / `invoke` statement-position case. Option (a) keeps the registry minimal and matches QRY-19's already-general `let _ = expr` escape hatch; prefer it unless a per-source-form code is wanted.

Implementer edge cases to keep consistent across the edit: the `void`-tail-function discard (QRY-19 / FN-4) and the `let _ =` escape hatch already apply to any expression, so they need no scope change — only the *enforcement* trigger and the `expressions.md` sentence do. Ensure the operator-facing always-log observability QRY-20 defines for discarded query `Err`s is stated to apply to discarded `CodeToolError` / `invoke` `Err`s on the same footing.

## Consequence

**Severity:** correctness

Two conforming front-ends diverge on whether a bare tool-call or `invoke` expression-statement that drops its `Result` is a `loom/parse/*` error or a silently-accepted discard. The same source loads on one implementation and fails to parse on another, and under the narrow reading authors lose the operator-facing always-log observability of discarded non-query `Err`s that QRY-20 guarantees for queries.
