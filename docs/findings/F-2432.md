## Finding

The binder `context.messages` bullet relies on every loom-1.0 provider accepting a `complete()` call whose message array is exactly one fixed `user` message plus the system prompt, but states this only as an inline rationale ("to satisfy providers that reject a system-only request with no non-system message"). It is pinned by no named presupposition and routed to no re-validation item, unlike the two sibling `complete()` presuppositions on the same call.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 61592d7 — pi-loom spec: resolve "Binder structured-output tool name and label undefined" (2026-06-05, Thomas Andersen); b884cda — pi-loom spec: resolve "Binder complete() context.messages content undefined" (2026-06-05, Thomas Andersen)
**History:** `binder-inference.md` entered the corpus at `f5e89f4` (the spec-set topic split) with both the `context.messages` and `context.tools` lines under-specified. Commit `61592d7` pinned the binder tool's `name`/`label`/`description` as the "three remaining required `ToolDefinition` fields", crystallising the omission of the fifth required `execute` member. The next-day commit `b884cda` replaced the vague "rendered binder input transcript" with the fixed single-`user`-message text and its "satisfy providers that reject a system-only request" rationale, introducing the unpinned provider presupposition. Neither commit added a presupposition pin or the `execute` member; the two defects were introduced independently during the same binder `complete()` specification buildout.

## Solution Space

**Shape:** single

### Recommendation

Add a named, anchored presupposition — alongside the existing `complete()` presuppositions in `conversation-drive.md` (e.g. a `#complete-single-non-system-message-presupposition` anchor) or pinned inline at the `context.messages` bullet — asserting that each loom-1.0-supported provider accepts a `complete()` request whose `context.messages` is a single fixed non-system (`user`) message accompanying the system prompt. Cite it from the `context.messages` bullet, and route re-validation to a new editorial-review checklist item in the Pi version-bump procedure (mirroring items (u)/(ab)), since no SDK surface pins this behaviour.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the required `execute` member of the forced binder tool (throwing stub vs no-op vs asserting function), and the unpinned single-message presupposition can silently break the binder on a future provider/SDK change with no detection path. Neither blocks implementation, but both produce divergent or fragile implementations of a load-bearing call.
