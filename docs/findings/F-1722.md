## Finding

The `looms.binderModel` hot-reload recovery note in `binder/binder-model-and-context.md` (the unnumbered paragraph at the `#binder-model-hot-reload` anchor, the BNDR-11 neighbour) pins `recovery.looms` to "**exactly** the looms whose original load failure was `loom/load/binder-model-unresolved` and whose binder model now re-resolves to a model under the changed setting, computed by re-running binder-model resolution alone with no other load-pass step re-run". The filter therefore has two independent conjuncts: (a) original load-failure code = `loom/load/binder-model-unresolved`, and (b) the binder-model two-step chain now resolves under the changed setting.

Only conjunct (b) is computable on-demand — the recovery pass explicitly re-runs binder-model resolution alone. Conjunct (a) requires the runtime to know, for every previously-discovered loom, which `loom/load/*` code its *original* load failed with — including looms whose failure was something other than `binder-model-unresolved` (a `.loom` parse error, a broken `.warp` import, a schema-lowering failure, etc.), which must be **excluded** from `recovery.looms` even when their binder-model two-step chain happens to resolve under the new setting. Nothing in the spec corpus introduces this per-loom load-failure classification state. `LoomRegistry` is a `Map<slashName, parsedLoom>` of *successfully-loaded* looms (drain-state contract in `pi-integration-contract/drain-state-contract.md`); the diagnostics channel emits failure codes but is described as an emission surface, not an addressable registry of past failures; and `BNDR-11` immediately above only pins the *successfully-loaded* path's behaviour ("each such loom continues using its previously-resolved binder-model handle until its next load"), leaving the previously-failed-loom set undefined.

Two reasonable implementers will diverge: one persists a per-loom `lastLoadFailureCode` (or a parallel `failed-looms` map keyed by slash name) and computes the conjunction correctly; another treats "now resolves under the new setting" as sufficient and over-emits `recovery.looms` for looms whose original failure was unrelated to binder-model resolution, producing a recovery note that promises a `/reload` will help when it will not. Both produce a passing `V11a` recovery-note fixture (which today only exercises the binder-model-unresolved → re-resolves path) but observably different operator-facing notes the moment a corpus contains a mixed-failure set.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `97430aa` (2026-06-06) — *pi-loom spec: resolve "Hot-reload recovery note over-promises /reload success"*
**History:** The paragraph was first authored in `f88698e` (2026-06-05) using membership prose "the looms whose re-resolution under the changed setting would now succeed" — a single conjunct (b) condition that the recovery pass's "re-run binder-model resolution alone" computes directly, with no need for any persisted per-loom failure classification. `97430aa` tightened the prose to "exactly the looms whose original load failure was `loom/load/binder-model-unresolved` and whose binder model now re-resolves to a model under the changed setting" while keeping the "by re-running binder-model resolution alone with no other load-pass step re-run" computation clause unchanged. That edit silently introduced conjunct (a) — a precondition on each loom's *prior* load-failure classification — without specifying where that classification is persisted or how the membership computation accesses it. The subsequent `BNDR-11` coining in `1b6d2f1` (2026-06-24) pinned the no-re-resolve-on-already-loaded-looms rule above the paragraph but did not touch the recovery-membership clause and so neither introduced nor cured the defect.

## Solution Space

**Shape:** multiple

### Option A — Persist a per-loom original load-failure classification

**Approach.** Extend the loom 1.0 load-pass state with a per-loom `lastLoadFailureCode: string | null` (or a sibling `failedLooms: Map<slashName, { code: string, … }>` parallel to `LoomRegistry`). Pin the write site (the load pass that emitted the failure diagnostic also records the code) and the read site (the `looms.binderModel` hot-reload recovery pass intersects "binder-model now re-resolves" against "`lastLoadFailureCode === 'loom/load/binder-model-unresolved'`"). Site the new structure under registration / `LoomRegistry` ownership in `pi-integration-contract/registration-steps.md` step 5 (`V9b`'s surface), with a forward link from the `#binder-model-hot-reload` paragraph.

**Spec edits.**
- In `pi-integration-contract/registration-steps.md` step 5 (or `drain-state-contract.md`'s `LoomRegistry` field list), introduce a sibling per-loom failed-load map keyed by slash name, populated by the load pass on each `loom/load/*` failure and cleared on a successful `/reload` of that loom.
- In `binder/binder-model-and-context.md` `#binder-model-hot-reload`, replace the unspecified "exactly the looms whose original load failure was …" prose with an explicit reference: "the looms whose entry in <named registry> carries `code === 'loom/load/binder-model-unresolved'` and whose binder-model two-step chain re-resolves under the changed setting".
- In `pi-integration-contract/runtime-event-channel.md`'s `details: { recovery: { looms: string[] } }` row, add a one-sentence pointer to the membership source.

**Pros.** Lands the spec's stated semantics exactly (preserves the `/reload`-honesty property `97430aa` was added to fix); makes the data dependency explicit; opens the door to other "previously-failed loom" hot-reload recoveries (parse errors, etc.) without further spec change.

**Cons.** Introduces a new long-lived per-extension-instance data structure and write site; requires deciding the lifetime / clear semantics (does a successful reload clear the entry? does an unloaded loom drop out?) and pinning it.

**Risks.** Drift between the failed-loom map's write site and the diagnostics-channel emission site; if the two diverge, the recovery note's membership computation desynchronises from what the operator actually saw fail.

### Option B — Re-run binder-model resolution alone, and drop the original-failure-code conjunct

**Approach.** Accept the existing recovery-pass computation ("re-run binder-model resolution alone") as the *full* membership predicate, and weaken the recovery-note prose to match. `recovery.looms` becomes "looms whose binder-model two-step chain previously did not resolve and now resolves under the changed setting" — computable from the current load state and the new setting value with no persisted classification. The honesty clause already present ("Membership therefore makes no claim that a listed loom's next `/reload` succeeds: a loom whose binder model now re-resolves can still fail its `/reload` for an independent reason …") covers the case where the listed loom's original failure was something else.

**Spec edits.**
- In `binder/binder-model-and-context.md` `#binder-model-hot-reload`, replace the "exactly the looms whose original load failure was `loom/load/binder-model-unresolved`" clause with the binder-model-only formulation, and slightly strengthen the existing "Membership therefore makes no claim" sentence to make it clear it covers both "stale `.loom` parse error" *and* "loom whose primary failure was unrelated to binder-model resolution".
- No other spec sites need touching.

**Pros.** Zero new runtime state; the recovery pass is exactly what the spec already says it computes ("re-running binder-model resolution alone with no other load-pass step re-run"); cheaper to implement and to test.

**Cons.** Reverts (semantically) part of `97430aa`'s tightening: an operator who sees a loom in `recovery.looms` whose original failure was, say, a `.loom` parse error will still see the note prompt `/reload`, and the `/reload` will still fail with the unchanged parse error — the load-bearing "honesty clause" carries the brunt of explaining why this is fine.

**Risks.** The note's value to operators degrades when the loom set is mixed; the existing honesty clause has to do more work explaining a possibly-confusing observation.

### Recommendation

Adopt **Option A**. The "honesty clause" `97430aa` added is informational; the *membership pin* (`exactly the looms whose original load failure was loom/load/binder-model-unresolved`) is what makes the note actionable, since it is the only thing that lets an operator treat `recovery.looms` as a high-signal "these will help" set rather than a low-signal "these may help" set. Dropping it (Option B) regresses the operator-facing contract `97430aa` was specifically introduced to establish. Option A also generalises cleanly: future hot-reload recoveries on other settings (`loomPaths`, `looms.scanPackages*`) can re-use the same per-loom failed-load registry without a second round of spec coining. Implementer edge cases to watch: (1) clear semantics on a successful `/reload` of a specific loom (the entry must drop) and on a watcher-driven content-edit reload (same); (2) interaction with the `LoomRegistry` drain-state contract — the failed-loom map is read by the recovery pass but the recovery note itself is suppressed entirely when `LoomRegistry.readDrainState()` indicates a degraded state has set in (no new slash work accepted); (3) the entry written on the very first load pass must use the slash name that survives the cross-format collision pass in `pi-integration-contract/registration-steps.md` step 3, so the recovery note's `<names>` join produces the canonical slash names rather than pre-collision intermediates.

## Consequence

**Severity:** correctness

Implementations will diverge on `recovery.looms` membership the moment a session carries a mixed previously-failed loom set: one runtime emits a recovery note that includes only the binder-model-unresolved looms (the spec's stated set), another includes every previously-failed loom whose binder-model chain now resolves regardless of original cause. The conformance test wired today (binder-model-unresolved → resolves) cannot distinguish these, so the divergence ships silently and produces operator-misleading notes that prompt `/reload` for looms whose `/reload` is guaranteed to fail again on the unchanged root cause.
