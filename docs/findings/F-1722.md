## Finding

The `looms.binderModel` hot-reload recovery note in `binder/binder-model-and-context.md` (the unnumbered paragraph at the `#binder-model-hot-reload` anchor, the BNDR-11 neighbour) pins `recovery.looms` to "**exactly** the looms whose original load failure was `loom/load/binder-model-unresolved` and whose binder model now re-resolves to a model under the changed setting, computed by re-running binder-model resolution alone with no other load-pass step re-run". The filter therefore has two independent conjuncts: (a) original load-failure code = `loom/load/binder-model-unresolved`, and (b) the binder-model two-step chain now resolves under the changed setting.

Only conjunct (b) is computable on-demand — the recovery pass explicitly re-runs binder-model resolution alone. Conjunct (a) requires the runtime to know, for every previously-discovered loom, which `loom/load/*` code its *original* load failed with — including looms whose failure was something other than `binder-model-unresolved` (a `.loom` parse error, a broken `.warp` import, a schema-lowering failure, etc.), which must be **excluded** from `recovery.looms` even when their binder-model two-step chain happens to resolve under the new setting. Nothing in the spec corpus introduces this per-loom load-failure classification state. `LoomRegistry` is a `Map<slashName, parsedLoom>` of *successfully-loaded* looms (drain-state contract in `pi-integration-contract/drain-state-contract.md`); the diagnostics channel emits failure codes but is described as an emission surface, not an addressable registry of past failures; and `BNDR-11` immediately above only pins the *successfully-loaded* path's behaviour ("each such loom continues using its previously-resolved binder-model handle until its next load"), leaving the previously-failed-loom set undefined.

Two reasonable implementers will diverge: one persists a per-loom `lastLoadFailureCode` (or a parallel `failed-looms` map keyed by slash name) and computes the conjunction correctly; another treats "now resolves under the new setting" as sufficient and over-emits `recovery.looms` for looms whose original failure was unrelated to binder-model resolution, producing a recovery note that promises a `/reload` will help when it will not. Both produce a passing `V11a` recovery-note fixture (which today only exercises the binder-model-unresolved → re-resolves path) but observably different operator-facing notes the moment a corpus contains a mixed-failure set.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `97430aa` (2026-06-06) — *pi-loom spec: resolve "Hot-reload recovery note over-promises /reload success"*
**History:** The paragraph was first authored in `f88698e` (2026-06-05) using membership prose "the looms whose re-resolution under the changed setting would now succeed" — a single conjunct (b) condition that the recovery pass's "re-run binder-model resolution alone" computes directly, with no need for any persisted per-loom failure classification. `97430aa` tightened the prose to "exactly the looms whose original load failure was `loom/load/binder-model-unresolved` and whose binder model now re-resolves to a model under the changed setting" while keeping the "by re-running binder-model resolution alone with no other load-pass step re-run" computation clause unchanged. That edit silently introduced conjunct (a) — a precondition on each loom's *prior* load-failure classification — without specifying where that classification is persisted or how the membership computation accesses it. The subsequent `BNDR-11` coining in `1b6d2f1` (2026-06-24) pinned the no-re-resolve-on-already-loaded-looms rule above the paragraph but did not touch the recovery-membership clause and so neither introduced nor cured the defect.

## Solution Space

**Shape:** single

### Recommendation

Introduce a persisted per-loom original load-failure classification and compute `recovery.looms` from it, so the recovery note's membership pin is honoured exactly.

In `pi-integration-contract/registration-steps.md` step 5 (or, equivalently, `drain-state-contract.md`'s `LoomRegistry` field list), add a sibling per-loom failed-load map keyed by slash name and parallel to `LoomRegistry` — for example `failedLooms: Map<slashName, { code: string, … }>`. The load pass writes an entry on each `loom/load/*` failure, recording the failure code, and clears the entry on a successful `/reload` of that loom (including a watcher-driven content-edit reload). Use the slash name that survives the cross-format collision pass in `pi-integration-contract/registration-steps.md` step 3, so downstream joins produce the canonical slash names rather than pre-collision intermediates.

In `binder/binder-model-and-context.md` at the `#binder-model-hot-reload` paragraph, replace the unspecified "exactly the looms whose original load failure was `loom/load/binder-model-unresolved` …" prose with an explicit reference to this map: `recovery.looms` is the set of looms whose entry in the failed-load map carries `code === 'loom/load/binder-model-unresolved'` **and** whose binder-model two-step chain re-resolves under the changed setting (computed by re-running binder-model resolution alone, with no other load-pass step re-run). Add a forward link from this paragraph to the named registry. The recovery note is suppressed entirely when `LoomRegistry.readDrainState()` indicates a degraded state (no new slash work accepted), so the failed-load map is read by the recovery pass only outside that degraded state.

In `pi-integration-contract/runtime-event-channel.md`'s `details: { recovery: { looms: string[] } }` row, add a one-sentence pointer to the failed-load map as the membership source.

Pin the lifetime semantics explicitly: a successful `/reload` of a specific loom (manual or watcher-driven) drops its entry, and an unloaded loom drops out of the map.

## Consequence

**Severity:** correctness

Implementations will diverge on `recovery.looms` membership the moment a session carries a mixed previously-failed loom set: one runtime emits a recovery note that includes only the binder-model-unresolved looms (the spec's stated set), another includes every previously-failed loom whose binder-model chain now resolves regardless of original cause. The conformance test wired today (binder-model-unresolved → resolves) cannot distinguish these, so the divergence ships silently and produces operator-misleading notes that prompt `/reload` for looms whose `/reload` is guaranteed to fail again on the unchanged root cause.
