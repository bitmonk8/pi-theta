## Finding

The *Specific exception types only* rule in `conventions.md` pins the `// allow-broad-catch:` closing-gate predicate: every allow-comment's cited token MUST resolve to one of four arms — a coverage-matrix REQ-ID, an enumerated *Code-keyed obligation areas* entry, **a `loom/...` diagnostics-registry code present in the [diagnostics registry]**, or the structural `pi-sdk-boundary` token. The worked example that follows then states "site (3) cites its `loom/host/session-shutdown-*` diagnostics-registry code." The cited token is a glob, not a registry code: no entry named `loom/host/session-shutdown-*` exists in the diagnostics registry — the registry holds the concrete codes `loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-runtime-degraded`, and `loom/host/session-shutdown-teardown-step-failed`. The rule never states whether the allow-comment carries a literal concrete code, or whether the gate's `loom/...`-arm resolver accepts a wildcard family.

Site (3) is the *Unknown-reason rule* `event.reason` property-access read (`unknown-reason-rule.md`). When that read throws it is treated as an unknown reason and the handler emits `loom/host/session-shutdown-reason-unknown` — the single concrete code that justifies the broad catch at that site. The glob is doubly imprecise: it both names no real registry entry and spans codes the `event.reason` read does not emit. The complication the glob papers over is that the spec permits the handler-entry `try`/`catch` to be authored as one widened block spanning both the snapshot read (which routes to `loom/host/session-shutdown-pinned-constant-unreadable`) and the `event.reason` read (which routes to `loom/host/session-shutdown-reason-unknown`); a single widened catch can therefore emit either of two concrete codes, while the `// allow-broad-catch:` grammar provides one `<registry-code>` slot.

## Issue introduction

**Verdict:** present-since-inception (of the broad-catch token-resolution worked example)
**Introducing commit:** `96563f9` — "pi-loom plan: resolve \"broad-catch allow-comment predicate unsatisfiable for exempt sites\"" (2026-06-10)
**History:** `git log -S "session-shutdown-*" -- docs/plan_topics/conventions.md` returns exactly one commit, `96563f9`, which authored the entire "Each site cites a token the predicate above admits…" clause — including the `loom/...` registry-code arm added so "site (3) needs" a satisfiable token, and the `loom/host/session-shutdown-*` glob in that same sentence. The glob has been present unchanged since; no later commit touched the token string. The defect is intrinsic to the worked example as first written: the commit added the registry-code arm to make site (3) satisfiable but illustrated it with a glob that the gate's "code present in the registry" resolver cannot satisfy.

## Solution Space

**Shape:** multiple

### Option A — Concrete code(s) in the worked example only

**Approach:** Edit the conventions.md worked-example clause to replace `loom/host/session-shutdown-*` with the concrete code(s) the site emits — `loom/host/session-shutdown-reason-unknown` for the `event.reason` read, and `loom/host/session-shutdown-pinned-constant-unreadable` where a single widened handler-entry catch also spans the snapshot read.
**Plan edits:** `conventions.md` only.
**Spec edits:** None.
**Pros:** Smallest diff; corrects the inconsistency at its source; the existing gate predicate ("code present in the registry") already excludes globs, so no gate change is needed for the single-concrete-code case.
**Cons:** If a leaf author writes one widened catch emitting both codes, the single `<registry-code>` slot in the `// allow-broad-catch:` grammar still cannot cite both; the example then implies a citation shape the grammar does not admit.
**Risks:** Leaves the multi-code/widened-catch interaction unresolved until a leaf author hits it.

### Option B — State the token-resolution rule, then fix the example

**Approach:** Add one clause to the *Specific exception types only* rule stating the `loom/...`-arm token MUST be one or more concrete diagnostics-registry codes present in the registry (never a glob or wildcard family), and that where a single broad-catch site can emit more than one concrete code the comment cites each; then update the site (3) worked example to the concrete code(s) per Option A.
**Plan edits:** `conventions.md` (rule clause + example); `H5c` and `H2a` token-resolution/grammar fixtures if the comment grammar is widened to admit a comma-separated concrete-code set.
**Spec edits:** None.
**Pros:** Closes the wildcard-acceptance question for every `loom/...` citation site, not just site (3); keeps the spec-permitted single-widened-`try`/`catch` shape viable by letting one comment cite the code set that catch emits.
**Cons:** Larger surface; touches the H2a/H5c gate fixtures.
**Risks:** Widening the grammar to a code set must keep the resolver's "each cited code present in the registry" check, or it reintroduces the silent-admission risk.

### Recommendation

Take Option B. The gate predicate already rejects globs ("a `loom/...` code present in the registry"), so the actionable defect is the worked example plus the unstated handling of a catch that emits more than one concrete code; the `unknown-reason-rule.md` spec deliberately leaves the implementer free to author the handler-entry catch as one widened block, so the citation form must be able to name the concrete code set that block emits rather than forcing a per-read catch shape. State in `conventions.md` that the `loom/...` token is one or more concrete registry codes (no wildcard), and set the site (3) example to `loom/host/session-shutdown-reason-unknown` (plus `loom/host/session-shutdown-pinned-constant-unreadable` for the widened-catch case). Edge case the implementer must watch: if the H5c/H2a allow-comment grammar stays single-token, the unknown-reason handler must instead use per-read catches so each cites exactly one concrete code — confirm the gate fixtures and the grammar agree before landing.

## Consequence

**Severity:** correctness

A V9h implementer who copies the worked example into a real `// allow-broad-catch:` comment writes the literal glob, which the H5c token-resolution arm rejects as "resolves to none of the four admitted arms," reddening the closing gate; a gate implementer who instead reads the example as licence to accept a wildcard family builds glob-matching into the resolver that the registry-code-present predicate does not warrant. The two diverge, and a wildcard resolver silently admits any future mistyped `loom/...` token under the same family prefix.
