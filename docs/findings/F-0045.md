## Finding

`H5a` (REQ-ID / diagnostic-code closing-gate automation) declares `Deps. H4a`, which transitively serialises it behind the whole horizontal infrastructure chain `H1a â†’ H2a â†’ H3a â†’ H4a`. But `H5a` consumes nothing `H4a` produces. `H4a`'s deliverables are the Pi extension factory shell, the in-process end-to-end harness, the session double, and the session-double fidelity contract. `H5a` is a CI gate that builds recogniser/reconciliation arms over **plan files and seeded test fixtures**: it reconciles the spec REQ-ID set, the diagnostics code registry, `coverage-matrix.md`, the asserting tests, and reads `H5b`'s `Deps.` line for the transitive-completeness arm. This is parallelizable-step false-sequencing: the `Deps. H4a` edge over-constrains the build order, blocking `H5a` from being picked up until two leaves it does not need (`H3a`, `H4a`) complete.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H5a-closing-gate-automation.md`, change the `Deps.` field from `H4a` to `H2a`. `H2a` is the correct target because `H5a`'s gate code lives under `src/**` and must satisfy `H2a`'s `no-broad-catch` / `no-restricted-syntax` / module-mutable-binding gates to keep `npm test` green, and `H2a` transitively carries the `H1a` toolchain prerequisite. This drops the spurious `H3a`/`H4a` dependency while preserving every prerequisite `H5a` genuinely needs.

Edge case for the implementer: if `H5a`'s closing-gate tooling is placed outside the lint-scoped `src/**` root (so `H2a`'s gates do not apply to it), `H1a` alone is the minimal sufficient dep â€” re-point to `H1a` in that case. Verify where the gate code lands before choosing between `H2a` and `H1a`.
