## Finding

The five-row *Per-boundary surfaces* table under `## Depth Enforcement` → **Error shape** in `docs/spec_topics/schema-subset.md` prescribes carrier shape, destination, and (for the slash-load `params` row) the cross-ceiling routing for surfaces whose normative owners live elsewhere — `query.md` / `errors-and-results.md` (row #1), `query.md` (row #2), `tool-calls.md` + `errors-and-results.md` (row #3), `invocation.md` + `binder/determinism-cancellation-failure.md` (row #4), `invocation.md` (row #5). The paragraph that closes the section says only that "this section's table mirrors it [the per-boundary table for ceiling #4] with the depth-walk-specific carrier details" and cross-links `overview-and-orientation.md#hard-runtime-ceilings`.

Two problems compound. First, the cross-link target is the informative four-ceiling orientation aggregator on `overview-and-orientation.md`, which under [Governance — GOV-30](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-30) carries no normative weight; the actual normative ceiling-#4 per-boundary table is `hard-ceilings/ceilings-3-and-4.md#ceiling-4-table`. So "mirrors it" already targets the wrong layer of the spec. Second, even with the target corrected, the schema-subset table does not declare whether it is itself normative (a second normative authority that must be kept in lock-step with the hard-ceilings table) or informative (a reader convenience whose authority is the hard-ceilings table alone). The word "mirrors" leaves both readings open: an implementer auditing rows #2–#5 cannot tell which carrier-shape claims to trust if the two tables drift, and a future editor cannot tell whether changing one row obliges a change in the other.

A drift surface already exists: `hard-ceilings/ceilings-3-and-4.md`'s row for `params` validation does **not** carry the `<ajv-summary>` placeholder clause that schema-subset.md's row #4 spells out (`the row's <ajv-summary> placeholder carries this enforcement point's canonical issue …`). If schema-subset's table is informative, that clause is undocumented prescription living in the wrong file; if it is normative, the two tables are already out of step. The same authority question applies row-by-row to #2's `ModelToolError` exclusion, #3's `CodeToolError { cause: "validation" }` shape, and #5's `cause: "return_validation"` literal — each of which is also restated, not delegated, here.

## Solution Space

**Shape:** single

### Recommendation

Make `hard-ceilings/ceilings-3-and-4.md#ceiling-4-table` the sole normative owner of ceiling #4's per-boundary destination/surface contract, and demote the schema-subset.md *Error shape* table to an informative depth-walk-specific restatement.

In `schema-subset.md` under `## Depth Enforcement` → **Error shape**, open the five-row *Per-boundary surfaces* table block with an explicit *Informative restatement* label sentence that reads as a depth-walk-specific restatement of [Per-boundary destination/surface table (ceiling #4)](./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table) and states that the hard-ceilings table is authoritative. Replace the trailing "this section's table mirrors it with the depth-walk-specific carrier details" sentence with that disposition, retargeting its cross-link from the GOV-30 informative orientation aggregator `overview-and-orientation.md#hard-runtime-ceilings` to `hard-ceilings/ceilings-3-and-4.md#ceiling-4-table`. Constrain the schema-subset table's rows to a strict subset of the hard-ceilings table's content — limited to depth-walk-specific carrier details the hard-ceilings table does not already state; any restated content beyond that subset is an editorial defect to remove.

Relocate the slash-load `<ajv-summary>` placeholder clause that currently lives only on schema-subset.md's row #4 into the corresponding `params` row of `hard-ceilings/ceilings-3-and-4.md#ceiling-4-table`, where it joins the existing slash-load arm prose; if that row already covers the clause, delete it from schema-subset.md instead. Confirm the `<ajv-summary>` placeholder clause is present on the hard-ceilings table's `params` row, adding it if absent.

On the plan side, update each affected leaf's **Spec** field (and any "delegated live-carrier witness" bullets) to cite `ceilings-3-and-4.md#ceiling-4-table` only for the carrier-shape claim, removing the schema-subset.md citation for carrier shapes and keeping the schema-subset reference scoped to the depth-walk algorithm. Specifically, `V5e`'s Adds prose and Tests bullets currently lean on schema-subset.md for the carrier contract — narrow that leaf's **Spec** citation set so the schema-subset reference covers the depth-walk algorithm only, not the carrier shapes.

This informative restatement still needs to be kept in rough textual sync with the hard-ceilings table; drift is a documentation defect subject to the reviewer SHOULD-flag obligation under GOV-30, not a correctness defect. The explicit *Informative restatement* label is the backstop against a future editor reading only the schema-subset table and inventing new carrier-shape claims there.

## Consequence

**Severity:** correctness

Two implementers reading rows #2–#5 cannot agree on which table is canonical and therefore cannot agree on which file to update when (for example) `cause: "return_validation"` changes shape: one will edit schema-subset.md alone and consider the work done, the other will edit hard-ceilings/ceilings-3-and-4.md and leave schema-subset.md behind. The slash-load `<ajv-summary>` clause already demonstrates that drift can land without notice. The depth-walk's wire contract (the `schema_keyword`, message, and carrier `cause` for each of the five enforcement points) is part of the on-wire contract the spec pins, so a silent inter-table divergence on these rows is a correctness defect for any cross-shard reader.
