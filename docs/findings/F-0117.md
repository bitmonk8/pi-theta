## Problem

`V18c`'s editorial-review checklist carries a load-bearing blocking distinction (per `version-bump-step2.md`, items `(a)`–`(e)` are MUST-level while `(f)`–`(ak)` are SHOULD-level-audit-but-MUST-record), but that distinction appears only in the non-binding **Adds** prose. The leaf's **Ships when** clause — the authoritative exit gate — states only that the commit message records a per-item outcome for every item, a recording-completeness check with no blocking semantics. A reviewer reading the exit gate alone would accept a bump whose commit message records `fail` on a MUST-level `(a)`–`(e)` item. Separately, the terminal **Adds** sentence ("Its done condition is not `npm test` but …") duplicates the done condition that **Ships when** also states, with slightly divergent wording, so the two normative copies can drift apart on a later edit.

## Issue introduction

**Verdict:** introduced by a single later edit
**Introducing commit:** `f1cda48` (2026-06-11) — "pi-loom plan: resolve \"V18c mislabels editorial-review checklist as a static build-time gate\""
**History:** `git log -S`/`-G` over `docs/plan_topics/V18c-version-bump-checklist.md` shows the MUST/SHOULD characterization (`SHOULD-level-audit-but-MUST-record`), the terminal Adds done-condition sentence (`Its done condition is not …`), and the extended Ships-when recording clause all entered in the same commit `f1cda48`. That commit re-classified the editorial checklist away from "static build-time gate"; in doing so it placed the blocking distinction in Adds and added a records-only acceptance to Ships-when, leaving the blocking semantics unanchored. The defect is an artefact of that single resolution edit, not present at the file's inception and not a multi-commit interaction.

## Solution approach

Move the MUST/SHOULD blocking distinction into `V18c`'s **Ships when** editorial-review acceptance clause so the exit gate states that a non-`pass` outcome on a MUST-level item `(a)`–`(e)` blocks the bump while a recorded `fail`/`not-performed` on a SHOULD-level item `(f)`–`(ak)` does not, citing `version-bump-step2.md`. Delete the terminal **Adds** sentence beginning "Its done condition is not `npm test` …" so the done condition lives in one authoritative home; the one-line MUST/SHOULD characterization may remain in Adds as descriptive scope.

## Solution constraints

- The editorial-review acceptance is a manual gate; do not route the blocking distinction through new **Tests** bullets or an `npm test` assertion — the leaf defines its done condition as editorial review, not `npm test`.
- The spec is read-only for this fix; `version-bump-step2.md` already carries the MUST/SHOULD distinction and the per-item outcome vocabulary.
