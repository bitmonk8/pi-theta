## Finding

The `DIAG-3` test bullet in both `V7b-code-registry.md` and `V7b-T-code-registry.md` reads:

> `DIAG-3`: codes are stable identifiers — a renamed code fails the gate (renames deferred to loom 2.0).

The loom-1.0 test obligation is the clause before the parenthetical: *a renamed code fails the gate*. That clause is complete and self-contained — it fully specifies what the gate must do and what the `V7b-T` red test must assert.

The trailing `(renames deferred to loom 2.0)` is not a test constraint. It is rationale duplicated from the authoritative spec rule `DIAG-3` (`diagnostics/diagnostic-shape.md`, anchor `#diag-3`), which states the deferral and its version target ("A rename is therefore deferred to loom 2.0 migration, outside the GOV-15 diagnostic-registry carve-out"). The plan leaf carries no mechanical guard tying its copy of the `loom 2.0` version constant back to the spec, so the parenthetical is a divergence surface: if the spec's deferral disposition or target version changes, the two leaf copies silently go stale. This is the same spec-owned-constant-snapshotted-into-plan-prose pattern flagged elsewhere in the review for version/floor snapshots.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** The plan corpus is git-tracked. `git log --follow` on both `docs/plan_topics/V7b-code-registry.md` and `docs/plan_topics/V7b-T-code-registry.md` shows c6a664e as the earliest (creation) commit for each. `git log -S"renames deferred to loom 2.0"` and `git log -G"DIAG-3.*renames deferred"` over `docs/plan_topics/` both return only c6a664e, confirming the parenthetical entered with the leaves' original authoring and was untouched by the later edit (f284534, a DIAG-2 resolution that did not modify the DIAG-3 bullet).

## Solution Space

**Shape:** single

### Recommendation

Strike `(renames deferred to loom 2.0)` from the `DIAG-3` Tests bullet in both `docs/plan_topics/V7b-code-registry.md` and `docs/plan_topics/V7b-T-code-registry.md`, leaving the bullet as:

> `DIAG-3`: codes are stable identifiers — a renamed code fails the gate.

The deferral rationale and the `loom 2.0` target remain owned by spec `DIAG-3` (`diagnostics/diagnostic-shape.md#diag-3`); the plan does not need to restate them to specify the test. The `loom 2.0` constant is read-only here — do not edit the spec. No other field on either leaf changes.

## Consequence

**Severity:** cosmetic

The `DIAG-3` test obligation is unchanged by the parenthetical, so no implementer diverges on what the gate or its red test must do. The only concrete defect is a maintenance one: the spec-owned `loom 2.0` version constant is mirrored verbatim into two plan leaves with no mechanical reconciliation, so a future change to the spec's deferral disposition leaves two stale plan copies.
