## Finding

The closing-gate machinery treats a `<new>` placeholder coverage-matrix row as "names no real closing leaf" and therefore as *not* satisfying the un-anchored-MUST closing-leaf condition. This rule is stated three times — in `coverage-matrix.md`'s *Code-keyed obligation areas* paragraph ("A `<new>` placeholder row names no real closing leaf"), in `conventions.md` *REQ-ID discipline* ("one whose closing-leaf column names no real leaf yet"), and in `H5a`'s un-anchored-MUST arm ("a `<new>` placeholder coverage-matrix row whose closing-leaf column names no real leaf … passes when that fixture MUST is enumerated with a real closing leaf"). In all three places the recognition criterion is **semantic** ("names no real leaf"); the mechanical recogniser the `H5a` implementer must build is never pinned.

Two reasonable implementers diverge. One reads the rule as a literal-token match — only the exact string `<new>` marks a placeholder, so any other token in the closing-leaf column is treated as "naming a closing leaf" and the un-anchored MUST is counted as enumerated (gate stays green). The other reads "a real closing leaf" as resolution against the actual plan leaf-ID universe — `<new>` does not resolve and *neither does a mistyped or non-existent ID* (e.g. a row whose closing-leaf cell reads `V99z`), so both redden the gate. Under the first reading, an un-anchored MUST mapped only to a typo'd / non-existent closing-leaf token silently satisfies the `H5a` un-anchored-MUST arm and the coverage gate passes vacuously — an uncovered obligation ships green with no asserting test ("a skipped test is a lie").

The ambiguity is compounded by an inconsistency with `H5d`, which already pins *its* recogniser mechanically: the transitive-completeness arm tokenises the closing-leaf cell to its backtick-delimited code spans and treats "no backtick leaf-ID token → empty tokenised set → excluded." `H5d` therefore keys placeholder recognition on *backtick presence* (implying `<new>` is written without backticks), while `H5a`/`conventions.md`/`coverage-matrix.md` key it on the semantic "names no real leaf." The two arms can classify the same cell differently, and the literal token form (whether `<new>` is backtick-delimited) is unpinned. `H5d` partially backstops the typo case — a backtick-delimited `` `V99z` `` is not in `H5b`'s `Deps.` and so reddens the transitive-completeness arm — but the `H5a` un-anchored-MUST arm answers a different question (is the MUST enumerated with a real closing leaf?) and its recogniser remains unpinned.

## Solution Space

**Shape:** single

### Recommendation

Pin the recognition rule mechanically and propagate it to the surfaces that restate it:

- In `coverage-matrix.md`'s *Code-keyed obligation areas* placeholder-convention paragraph, state that the placeholder is exactly the literal closing-leaf token `<new>`, and that any other closing-leaf token that does not resolve to an existing plan leaf ID is a gate defect (red), not a benign placeholder. To keep this literal-token rule and `H5d`'s tokeniser recognising the same cell, write the placeholder without backtick delimiters so `H5d`'s "no backtick leaf-ID token → empty tokenised set → excluded" path and the `H5a` literal-token path select the identical cell.
- In `H5a`'s un-anchored-MUST arm (the *Adds* clause and the `<new>`-placeholder *Tests* bullet), state that the arm classifies a row as "enumerated with a real closing leaf" only when the closing-leaf column resolves to an existing plan leaf ID; the literal `<new>` marks the obligation uncovered (red on the live footing), and any other non-resolving closing-leaf token also reddens the gate as a defect rather than being read as a placeholder.
- In `conventions.md` *REQ-ID discipline*, replace the bare semantic phrasing ("one whose closing-leaf column names no real leaf yet") with a reference to the pinned literal-`<new>`-token-plus-non-resolving-token-failure rule, pointing at `coverage-matrix.md` as the definition home.
- `H5d` already pins its recogniser ("no backtick leaf-ID token → empty tokenised set → excluded"); confirm the literal `<new>` token form chosen above keeps `H5d`'s exclusion and `H5a`'s recogniser selecting the same cell, so a backtick-delimited non-resolving token such as `` `V99z` `` reddens both arms.

Edge case the implementer must watch: a placeholder written *with* backticks (`` `<new>` ``) would be tokenised by `H5d` as a non-`Deps.` leaf-ID token and reddened, so the chosen literal form must be unambiguous about backtick delimiting; the `<new>` fixture under the seeded test-fixtures root should exercise the recogniser against both `<new>` and a non-resolving backtick token.

## Consequence

**Severity:** correctness

A literal-`<new>` recogniser counts an un-anchored MUST mapped to a mistyped or non-existent closing-leaf token (e.g. `V99z`) as enumerated, so the `H5a` un-anchored-MUST arm passes vacuously and an uncovered obligation ships green with no asserting test. Because `H5a` (semantic) and `H5d` (backtick-token) describe the recogniser differently, two implementers also produce arms that classify the same cell inconsistently.
