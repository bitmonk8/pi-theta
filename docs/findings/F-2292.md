## Finding

`H2a`'s **Adds** defines the `no-restricted-syntax` blocking-runtime check as a *general* rule: it flags "every call whose callee name ends in the literal suffix `Sync`, whether a member access such as `fs.readFileSync` or a bare identifier such as `execSync`" — and explicitly notes "those two cited as examples of the recognised form, not a finite set". The check's whole contract is the suffix-match generalisation.

`H2a`'s **Tests**, however, exercise only the two example callees: the blocking-runtime bullet asserts that "a blocking `fs.readFileSync` or `execSync` call in `src/**` without an `// allow-sync:` comment fails lint" and that the same call with the comment passes. No fixture exercises an arbitrary `…Sync`-suffixed callee that is neither of the two named examples, and no fixture asserts that a non-`Sync` callee is left unflagged.

Because the acceptance criteria name only the two examples, an implementation that hard-codes a two-entry allowlist (`fs.readFileSync`, `execSync`) rather than performing the suffix match passes the leaf's own tests while violating the Adds-stated behaviour. The leaf can be marked done with the generalisation unverified, and a later regression narrowing the predicate to that hardcoded pair would go undetected.

## Issue introduction

**Verdict:** present-since-inception (of the blocking-runtime check)
**Introducing commits:** `c33d309` (`pi-loom plan: resolve "Blocking-runtime ban concedes enforcement to manual review"`, 2026-06-17), aggravated by `8561a7c` (`pi-loom plan: resolve "H2a blocking-runtime *Sync enumeration is ambiguous"`, 2026-06-18).
**History:** `git log -S 'allow-sync'` / `-S 'blocking \`fs.readFileSync\` or \`execSync\`'` over `docs/plan_topics/H2a-cross-cutting-gates.md` shows the blocking-runtime Tests bullet first appeared in `c33d309`, which created the lint check and its acceptance bullet naming only `fs.readFileSync`/`execSync`; the Adds at that commit already described "`fs.readFileSync`, `execSync`, and the `*Sync` family`", so the Tests-vs-Adds divergence (general rule asserted only by two examples) was present from the bullet's inception. `8561a7c` then rewrote the Adds to make the suffix-match generalisation explicit ("every call whose callee name ends in the literal suffix `Sync` … those two cited as examples … not a finite set") without adding a corresponding general-case fixture, widening the gap. No commit ever narrowed an earlier, broader set of test fixtures — the coverage gap is original to the feature.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H2a-cross-cutting-gates.md`, extend the blocking-runtime Tests coverage (the `Convention:` (*Sequential by default* — blocking runtime) bullet) so the suffix-match generalisation is asserted, not just the two example callees. The added coverage must assert both:

- an arbitrary `…Sync`-suffixed callee that is neither `fs.readFileSync` nor `execSync` — e.g. a bare `fooSync()` call (or a member access `bar.bazSync()`) — in `src/**` without an `// allow-sync:` comment fails lint, and the same call with a same-line `// allow-sync: <reason>` comment passes;
- a non-`Sync`-suffixed callee — e.g. `foo()` — in `src/**` is not flagged.

This pins the gate to its Adds-stated behaviour (suffix match, not a fixed allowlist) so a hardcoded two-entry implementation or a later narrowing regression fails the leaf. The existing `**/*.test.ts`-is-unrestricted and `// allow-sync:` exemption assertions remain unchanged.

## Consequence

**Severity:** advisory

The blocking-runtime gate's defining behaviour — the general `Sync`-suffix match — has no asserting fixture, so a hardcoded two-callee implementation (or a later regression narrowing to one) satisfies the leaf yet silently lets every other `…Sync` blocking call through. The gate's purpose is to keep synchronous calls off Pi's shared event loop; an under-verified gate degrades to catching only the two demo cases.
