## Problem

`docs/spec_topics/pi-integration-contract/host-interfaces-core.md` carries four Pi-owned member-pinning surfaces â€” the three fenced `ts interface { â€¦ }` blocks under `#extensioncontext-interface`, `#extensioncommandcontext-interface`, and `#agentsession-interface`, plus the inline `CompactOptions`/`CompactionResult` literal paragraph â€” none of which marks how much of the pinned shape is normative. Each fuses two distinct properties without naming the split: (i) which members loom is permitted to touch (normative on the loom runtime, as the surrounding prose asserts via "their absence from this table is normative â€” the loom runtime MUST NOT call them"), and (ii) each member's TypeScript signature (a non-normative snapshot at the loom 1.0 Pi-SDK pin that may drift on any Pi minor and is re-validated at the bump). The sibling loom-owned DI seams on `host-interfaces-services.md` carry an explicit GOV-18-arm-(a) non-normative-reference marker, so the Pi-owned surfaces read as a silent third thing. An implementer cannot tell whether a signature is a normative pin (a drifting Pi minor is a violation) or a snapshot the runtime adapts to, and a reviewer copying the DI-seam wording would over-correct and authorise loom to call deliberately-excluded members.

## Solution approach

Cross-reference GOV-18 arm (a) and the version-bump procedure as the existing surfaces already do.
