## Finding

The `H5d` transitive-completeness gate arm iterates **every** row of `coverage-matrix.md`'s *Numbered REQ-IDs* and *Code-keyed obligation areas* tables, tokenises each row's closing-leaf cell into the set of its backtick-delimited spans, and fails CI on any cell none of whose listed leaves is a member of `H5b`'s expanded `Deps.`. Both `H5d-transitive-completeness-gate.md` and the `conventions.md` *Transitive-completeness plan-maintenance* obligation state exactly one exclusion from this check: the bare `<new>` placeholder rows, "whose closing-leaf column lists no real leaf yet."

The *Code-keyed obligation areas* table currently carries ten retired rows whose closing-leaf cell holds the literal `*(numbered above)*` rather than any backtick-delimited leaf-ID: `cka-7`, `cka-12`, `cka-18`, `cka-21`, `cka-22`, `cka-23`, `cka-24`, `cka-25`, `cka-28`, `cka-29`. Each marks an obligation that was migrated to numbered REQ-IDs and closed in the *Numbered REQ-IDs* table above; the row is retained only because the *Token* column is append-only. None of these cells contains a backtick span, so each tokenises to the empty set. An empty set has no leaf in `H5b.Deps`, which vacuously satisfies the arm's "none of whose listed leaves is in it" failure condition — so a faithful implementation reddens CI on all ten rows, permanently, against the live `coverage-matrix.md`.

The retired rows are not `<new>` rows, so the single stated exclusion does not cover them, and the implementer has no text resolving the two divergent readings (always-fail vs. silently-skip-every-empty-cell). The skip-every-empty reading is not a safe default either: the *Closing-leaf column authoring* note relies on an emptied single-leaf cell surfacing as a spurious failure to catch a mis-authored un-delimited leaf-ID, so silently skipping all empty-tokenised cells would swallow exactly the defect that note is designed to surface. The gate needs an explicit third case distinguishing a retired `*(numbered above)*` row from both a `<new>` placeholder and a mis-authored un-delimited cell.

## Consequence

**Severity:** blocking

The `H5d` arm is a standing gate that reads the live `coverage-matrix.md` at every footing (it has no live-corpus flip). Run against the real table it reddens on all ten retired `*(numbered above)*` rows, so the transitive-completeness arm — and therefore the unified closing-gate suite it runs inside — cannot pass against the current plan. An implementer building the arm to the stated tokenise-and-at-least-one rule produces a CI gate that is red on first run and stays red until the rows are re-shaped or the gate is patched, with no plan text telling them which.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `617a3ad` (gate definition with `<new>`-only exclusion) interacting with `716a13b`, `47915a2`, `ffe2cc9` (later cka-row retirements that introduced the `*(numbered above)*` cell shape)
**History:** `617a3ad` (2026-06-16, "H5a bundles gate arms spanning four data sources") created the `H5d` arm; its row-exclusion set was `<new>`-only and complete for the corpus as it then stood — no `*(numbered above)*` row existed. The first retired row carrying that literal entered in `716a13b` (2026-06-25, F-1286, query-subtree migration retiring `cka-12`), with further retirements in `47915a2` (2026-06-27, F-1890) and `ffe2cc9` (2026-06-28, F-2317, retiring `cka-7`). None of those retirement commits extended the gate's exclusion set, so the defect emerged from the interaction between the gate's fixed exclusion set and the later retirement commits rather than from any single commit in isolation.

## Solution Space

**Shape:** single

### Recommendation

Add a third recognised empty-tokenisation case to the `H5d` arm so a closing-leaf cell whose content is exactly the literal `*(numbered above)*` (the retired code-keyed-area marker) is excluded from the at-least-one check, alongside the existing bare `<new>` exclusion. Propagate the same recogniser to all three homes that describe the arm:

- `docs/plan_topics/H5d-transitive-completeness-gate.md` — in **Adds** (where the `<new>`-empty-set exclusion is stated), **Tests**, and **Ships when**, state that a cell holding exactly `*(numbered above)*` is excluded from the arm just as a `<new>` cell is, and add a seeded fixture exercising a retired `*(numbered above)*` row staying green.
- `docs/plan_topics/conventions.md` — in the *Transitive-completeness plan-maintenance* obligation, extend the parenthetical "excluding the `<new>` placeholder rows whose closing-leaf column lists no real leaf yet" to also exclude rows whose closing-leaf cell is the literal `*(numbered above)*` retired marker.
- `docs/plan_topics/coverage-matrix.md` — in §"Closing-leaf column authoring", define the `*(numbered above)*` retired-row marker as a third recognised empty-tokenising cell form, distinct from the `<new>` placeholder and from a mis-authored un-delimited single-leaf cell.

The recogniser keys on the exact literal cell content `*(numbered above)*`. Any other empty-tokenised cell — one with no backtick span that is neither the literal `<new>` nor the literal `*(numbered above)*` — must continue to redden the arm, preserving the mis-authored-un-delimited-cell detection the *Closing-leaf column authoring* note depends on. The spec is read-only for this fix, and the cka-token append-only / never-renamed rule is unaffected (the retired rows stay in place with their existing tokens).
