## Finding

`docs/spec_topics/schema-subset.md` Lowering Algorithm step 5 ("Per-schema sidecar") fixes a specific in-memory representation: the lowering pass captures, *alongside each `$defs` entry*, *a sidecar with two maps*, the second of which is *keyed by JSON Pointer into the lowered schema fragment*. The cross-reference in `docs/spec_topics/runtime-value-model.md` (Inbound translation) reinforces the same shape by naming the "*Named-enum positions* sidecar" as a JSON-Pointer-keyed map the inbound pass *reads*.

Neither the number of maps, the JSON-Pointer keying, nor the "captured alongside each `$defs` entry" attachment point is observable at any spec boundary. What is observable, and what the prose around step 5 actually pins, is the information the inbound translation pass needs to do its job: (a) for every renamed field, the wire↔loom name correspondence that lets it rebuild loom-side names from the validated JSON; and (b) for every value position in the lowered fragment, whether that position originated in a named `enum` declaration (in which case the declaring-enum tag is reattached so cross-enum equality and `JSON.stringify` behave per `runtime-value-model.md`) or in an anonymous string-literal union (in which case no tag is attached and equality falls back to plain string comparison). An implementation that annotates the lowered AST in place, walks the schema in lockstep with the value, materialises one map keyed by something other than JSON Pointer, or splits the information across more than two maps satisfies every black-box test the spec writes — but the current step-5 wording forbids each of them.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") — introduced the original step 5 as a single "sidecar map per schema" capturing wire-name translation; `7f217a8` (2026-06-07, "pi-loom spec: resolve \"Language-core hidden assumptions: enum-tag sidecar, AJV-config silence, Markdown claim\"") — expanded it to "a sidecar with two maps", added the JSON-Pointer-keyed *Named-enum positions* map, and propagated the same data-structure framing into `runtime-value-model.md`'s inbound bullet.
**History:** `git log -S 'sidecar' -- docs/spec_topics/schema-subset.md` and the matching `git show` diffs confirm that the data-structure framing ("sidecar map", later "sidecar with two maps") has been present since step 5 first landed. `fae85d3` introduced the pattern: a single map prescribed as the carrier of wire-name translation. `7f217a8` solved a real adjacent defect — the inbound pass had no spec'd way to know which positions were named enums versus anonymous string-literal unions — by adding a *second* map and pinning its key type (JSON Pointer) and attachment site (alongside each `$defs` entry); that fix is what made the over-prescription concrete enough to flag. The current defect is therefore the cumulative state, not a single regression: the data-structure framing arrived with the section's inception, and the second map plus JSON-Pointer keying were added as the structural answer to a behavioural question rather than as a statement of the information that must be available.

## Solution Space

**Shape:** single

### Recommendation

Rewrite Lowering Algorithm step 5 to specify the *information* the lowering pass must make available to the inbound translation pass, not the data structure it lives in. Concretely, the step should pin:

1. For every renamed field reachable from the lowered fragment, the wire↔loom name correspondence (the example `{ loom: "first_name", wire: "FirstName" }` stays as illustration, not as schema).
2. For every value position in the lowered fragment, whether that position originated in a named `enum` declaration — in which case the declaring-enum's loom-side name is recoverable — or in an anonymous string-literal union, in which case no enum name is associated. The asymmetry between named enums (tag reattached) and anonymous unions (no tag) remains normative.

Drop the "*a sidecar with two maps*", the "*keyed by JSON Pointer into the lowered schema fragment*", and the "*captured alongside each `$defs` entry*" attachment-point clauses. The cited illustration of the wire-name pair and the named-vs-anonymous asymmetry stay. Rename the construct from "sidecar with two maps" to something information-shaped (e.g. "per-schema translation metadata"); a single noun the inbound section can link to is fine — what matters is that the noun names the information, not its container.

Mirror the rewording in `runtime-value-model.md`'s inbound bullet: replace "the lowering pass's *Named-enum positions* sidecar … maps to a declaring-enum name" with prose that says the inbound pass consults the lowering pass's per-position record of declaring-enum names; preserve the named-vs-anonymous distinction verbatim. Edge case the rewrite must preserve: the inbound pass walks arrays, nested object fields, and `Result.Ok`/`Result.Err` payloads, and tags are attached at the same depth as the value the schema annotates — that is a behavioural obligation already pinned in `runtime-value-model.md` and must not be lost when the JSON-Pointer keying it implicitly leans on is removed.

## Consequence

**Severity:** advisory

Two reasonable implementers can still produce equivalent black-box behaviour, so the spec is not broken — but one of them is pointlessly forced to mirror the prescribed structure (or to argue with a future reviewer that their annotation-on-AST encoding is conformant). The plan leaves inherit the over-binding and would mark a conformant implementation non-conformant on a Tests bullet that only checks shape.
