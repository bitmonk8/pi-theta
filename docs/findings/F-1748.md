## Finding

`active-invocation-registry.md` opens by declaring the registry as `an extension-instance-scoped Set<{ loomAbort: AbortController; disposeBarrier: Promise<void>; shutdownReason: string | undefined; loom: string; invocationId: string }>`, and the *Registry contract* §"Iteration order" bullet then justifies its iteration-order MUST by appealing to "the V8 `Set` insertion-order invariant the rest of the spec already relies on." The same page also explicitly disclaims the registry's *name* as "internal … not part of the loom-facing or extension-facing contract; tests assert on observable side effects (entry counts via probe seams, ordered `loomAbort.abort()` calls, `disposeBarrier` settlement) rather than on the symbol itself."

The two framings are inconsistent at the type level the same way they are at the name level: every observable obligation in the contract (insertion at three sites, removal in the per-invocation `finally`, sub-step-2 ordered `loomAbort.abort()` over every entry, sub-step-3 `Promise.allSettled` over every entry's `disposeBarrier`, swallowing per-entry `try`/`catch` during sub-step 2) is satisfied by any insertion-ordered collection of distinct entries — an `Array`, a `Map` keyed on `invocationId`, a doubly-linked list. Nothing in the contract distinguishes `Set` from those (no `.has(entry)` membership check is exercised; entries are reached only by iteration; uniqueness is structural — each insertion site creates a fresh object — not enforced by the collection). Worse, justifying the iteration MUST by appeal to a runtime-specific invariant ("V8 `Set` insertion-order") couples a normative conformance point to a non-Pi-pinned implementation property; an implementation that switched to a runtime without that invariant would have to *re-derive* the MUST from the spec, which the spec should already do directly.

The fix is small and behaviour-preserving: present `Set<…>` as illustrative (the page already does so for the *name*) and state the requirement as "an insertion-ordered collection of per-invocation entries, iterated in insertion order at every iteration site." The fix is a NIT — the V8 invariant happens to hold and the reference implementation will use `Set` — but the spec is the contract surface, and overspecifying it here mirrors the same overreach the page already calls out one bullet below.

## Solution Space

**Shape:** single

### Recommendation

In the opening paragraph of `active-invocation-registry.md`, replace the `Set<{ … }>` type literal with prose: state that the registry is "an extension-instance-scoped, insertion-ordered collection of per-invocation entries, each carrying `{ loomAbort: AbortController; disposeBarrier: Promise<void>; shutdownReason: string | undefined; loom: string; invocationId: string }`." Keep the five-field entry shape pinned exactly as today (it *is* the wire/observable surface).

In the *Registry contract* §"Iteration order" bullet, restate the MUST directly — "Iteration order in sub-step 2 and sub-step 3 MUST be insertion order" — and drop the "matching the V8 `Set` insertion-order invariant the rest of the spec already relies on" justification. The MUST does not need V8 to back it; the collection chosen by the implementation has to honour it.

Add one sentence next to the existing "**The registry name is internal**" bullet (or fold into it) noting that the `Set` spelling used in worked examples / fixtures elsewhere in the corpus is **illustrative**: any insertion-ordered collection satisfying the iteration MUST conforms. Audit the file once for residual `Set`-implies-`Set.add`-semantics phrasing (the *Dispatch-site setup wrap* bullet currently says "the registry-`Set.add` insertion" twice — rephrase to "the insertion-site `add` step" or equivalent so the prose does not silently re-pin the type).

Edge cases the implementer must watch:

- The reference implementation may still use `Set` — that is fine and expected; no behavioural change is intended.
- Tests asserting on iteration order must assert against the order MUST, not against `instanceof Set` or `Set.prototype.values()`-shaped iteration; confirm `V9e-T` acceptance criteria read this way.
- The "five-field entry" shape MUST remain pinned; the relaxation is purely on the *collection wrapping* those entries, not on the entries themselves.

## Consequence

**Severity:** cosmetic

Two reasonable implementers using a `Map<invocationId, entry>` or an `Array` of entries would both satisfy every observable obligation on the page, yet a literal reading of the type pin would flag either as non-conforming. The defect is contained — no diagnostic, wire shape, or behavioural outcome shifts — but the iteration-order MUST currently rests on a V8-specific invariant rather than on a directly-stated spec rule.
