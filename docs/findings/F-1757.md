## Problem

`inventory-audit-intro.md` names the single step-2(a) SDK surface-inventory literal-read assertion two non-equivalent ways three paragraphs apart, with nothing on the page bridging them: the *Inventory-closure audit* paragraph (anchor `sdk-cap-inventory-closure-audit`) calls it "the positive-direction literal-read assertion that owns items 1–7 above", while the *Strict-capability absence-under-the-probed-name arm* paragraph (anchor `strict-capability-absence-under-probed-name`) calls it "the existing `strict-capability-probe` literal-read assertion". The second phrasing is structurally misleading because `strict-capability-probe` is the name of one `SDK_SURFACE_INVENTORY` entry kind, so a reader has every reason to read it as a per-kind sub-assertion rather than the umbrella step-2(a) gate whose branch table dispatches on every kind. A downstream agent can then land a separate per-kind "strict-capability-probe literal-read" check that duplicates or fragments step-2(a)'s branch table instead of extending the one assertion the spec pins.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e5cd514` (2026-05-09, "pi-loom spec: resolve T35 SDK capability inventory closed-set negative-direction audit") + `6829a79` (2026-06-06, "pi-loom spec: resolve Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)")
**History:** The two contested phrasings were not introduced together. `e5cd514` first coined "the positive-direction literal-read assertion that owns items 1–7" inside the new *Inventory-closure audit* paragraph (then on the monolithic `pi-integration-contract.md`), at the point the negative-direction closure audit was added as a sibling of the seven-capability inventory. A month later `6829a79` (the Pi-SDK hidden-assumption-cluster resolution that pinned the strictCapable-absence claim and added its absence-under-the-probed-name MUST) authored "the existing `strict-capability-probe` literal-read assertion (per [step 2(a)](…))" — placed in `audit-target-categories.md` (then `audit-recognised-shapes.md`), naming the assertion via the entry-kind name in order to point at the existing step-2(a) gate the new MUST extends, without reconciling against the *Inventory-closure audit* paragraph's "positive-direction" phrasing. `bf92fe6` (2026-06-25, F-1290) then relocated the MUST sentence from `audit-target-categories.md` to `inventory-audit-intro.md` so it sits in the same file as the *Inventory-closure audit* paragraph, which is the configuration the current finding observes; the relocation was scoped to placement and did not reword either phrasing. The defect is therefore the interaction between the two earlier coining commits, surfaced into one-page co-location by the later relocation rather than introduced by it.

## Solution approach

Rename both assertion-naming phrasings in `inventory-audit-intro.md` to the canonical "step-2(a) SDK surface-inventory literal-read assertion" already used at the definition site `version-bump-step2.md#bump-step-2-positive`, and link each occurrence to that anchor: the "positive-direction literal-read assertion that owns items 1–7 above" phrase in the *Inventory-closure audit* paragraph, and the "existing `strict-capability-probe` literal-read assertion" phrase in the *Strict-capability absence-under-the-probed-name arm* paragraph. Apply the same rename to the back-linking clause in `audit-target-categories.md`'s strict-capability-absence-pin paragraph so both back-link sites use the canonical name.

## Solution constraints

- Do not touch `strict-capability-probe` in its entry-kind role — the `SDK_SURFACE_INVENTORY` `kind` discriminator in `audit-target-categories.md`'s minimum-shape exemption list, the V18a/V18c plan-leaf prose, and the probe-constant `probedName`-bearing row; rename only the prose phrases that use it as a name for the assertion.
