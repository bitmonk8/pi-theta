## Finding

`docs/spec_topics/pi-integration-contract/inventory-audit-intro.md` refers to the step-2(a) build-time check by two different phrasings three paragraphs apart, with nothing on the page tying them together:

- The *Inventory-closure audit* paragraph calls it "the positive-direction literal-read assertion that owns items 1–7 above".
- The *Strict-capability absence-under-the-probed-name arm* paragraph calls it "the existing `strict-capability-probe` literal-read assertion (per [step 2(a)](./version-bump-step2.md#bump-step-2-positive) below)".

Both refer to the same assertion — the single step-2(a) SDK surface-inventory literal-read assertion defined in `version-bump-step2.md` — but the second phrasing is structurally misleading. `strict-capability-probe` is the name of *one entry kind* inside `SDK_SURFACE_INVENTORY` (alongside `node-floor`, `peer-dep-range`, `api-coverage`, `namespace-function`, `type-union-snapshot`, and the new audit-participating kinds), pinned at `audit-target-categories.md`'s minimum-shape exemption list. A reader who hits "`strict-capability-probe` literal-read assertion" with no equivalence bridge has every reason to read it as "the literal-read assertion *for the `strict-capability-probe` entry kind*" — a per-kind branch — rather than as the umbrella step-2(a) assertion whose branch table dispatches on every kind. The strictCapable absence-under-the-probed-name MUST is then routed *into that umbrella assertion as an additional failure arm*, but on the page the reader is told it extends "the `strict-capability-probe` literal-read assertion", which sounds like a per-kind sub-assertion that doesn't otherwise exist.

The hazard is concrete: a downstream agent (implementation, test author, or another spec edit) can land a separate, per-kind "strict-capability-probe literal-read" check that duplicates or fragments step-2(a)'s branch table rather than extending the single existing assertion the spec actually pins.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e5cd514` (2026-05-09, "pi-loom spec: resolve T35 SDK capability inventory closed-set negative-direction audit") + `6829a79` (2026-06-06, "pi-loom spec: resolve Pi-SDK hidden-assumption cluster (strictCapable absence, decode steps, pi.on origin)")
**History:** The two contested phrasings were not introduced together. `e5cd514` first coined "the positive-direction literal-read assertion that owns items 1–7" inside the new *Inventory-closure audit* paragraph (then on the monolithic `pi-integration-contract.md`), at the point the negative-direction closure audit was added as a sibling of the seven-capability inventory. A month later `6829a79` (the Pi-SDK hidden-assumption-cluster resolution that pinned the strictCapable-absence claim and added its absence-under-the-probed-name MUST) authored "the existing `strict-capability-probe` literal-read assertion (per [step 2(a)](…))" — placed in `audit-target-categories.md` (then `audit-recognised-shapes.md`), naming the assertion via the entry-kind name in order to point at the existing step-2(a) gate the new MUST extends, without reconciling against the *Inventory-closure audit* paragraph's "positive-direction" phrasing. `bf92fe6` (2026-06-25, F-1290) then relocated the MUST sentence from `audit-target-categories.md` to `inventory-audit-intro.md` so it sits in the same file as the *Inventory-closure audit* paragraph, which is the configuration the current finding observes; the relocation was scoped to placement and did not reword either phrasing. The defect is therefore the interaction between the two earlier coining commits, surfaced into one-page co-location by the later relocation rather than introduced by it.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/pi-integration-contract/inventory-audit-intro.md`, rename both phrasings to a single canonical reference — **"the step-2(a) SDK surface-inventory literal-read assertion"** — and link each occurrence to `version-bump-step2.md#bump-step-2-positive`:

- In the *Inventory-closure audit* paragraph, replace "the positive-direction literal-read assertion that owns items 1–7 above" with "the [step-2(a) SDK surface-inventory literal-read assertion](./version-bump-step2.md#bump-step-2-positive) (whose positive-direction branch enumerates items 1–7 above)". Keep the "items 1–7" qualifier as descriptive prose only — it loses load-bearing status.
- In the *Strict-capability absence-under-the-probed-name arm* paragraph, replace "The existing `strict-capability-probe` literal-read assertion (per [step 2(a)](./version-bump-step2.md#bump-step-2-positive) below)" with "The [step-2(a) SDK surface-inventory literal-read assertion](./version-bump-step2.md#bump-step-2-positive) below".

Apply the same rename to the carrying clause in `audit-target-categories.md`'s *Strict-capability absence under the pin* paragraph — "The `strict-capability-probe` literal-read assertion's *absence-under-the-probed-name* extension" becomes "The step-2(a) SDK surface-inventory literal-read assertion's *absence-under-the-probed-name* extension" — so the phrasing is consistent across both sites that back-link to the relocated MUST.

The closing sentence of step 2(a) at `version-bump-step2.md` already names this same canonical phrasing ("The literal-read assertion additionally carries the *strict-capability absence-under-the-probed-name* arm — pinned at …"), so the rename aligns the inbound references with the definition site rather than introducing a new term. The entry-kind name `strict-capability-probe` is left untouched in its proper role (a `SDK_SURFACE_INVENTORY` `kind` discriminator in the audit-target-categories minimum-shape exemption list, in `V18a` / `V18c` plan-leaf prose, and as the probe-constant `probedName`-bearing row), so this is purely a fix to the assertion-naming prose. Implementer edge case: the rename MUST NOT touch the `strict-capability-probe` entry-kind references in `audit-target-categories.md`'s minimum-shape exemption sentence or in the plan-leaf prose — only the prose phrases that use it as a name for the assertion.

## Consequence

**Severity:** advisory

A spec reader and a downstream implementer can plausibly read the two phrasings as two distinct assertions (an umbrella "owns items 1–7" check vs. a per-kind "`strict-capability-probe`" check), which invites a duplicate or split build-time gate when the spec in fact pins exactly one. The damage is bounded because the second phrasing carries the `version-bump-step2.md#bump-step-2-positive` anchor, which a careful reader can chase to resolve the ambiguity; the gap is in establishing the equivalence on the page itself.
