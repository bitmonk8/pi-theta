## Problem

In `docs/plan_topics/H4b-response-programming-surface.md`, the **Adds** enumeration of the response-programming surface lists injection categories (a)–(e). Category (b) uses the term "mixed-success" (in "a mixed-success parallel `tool_use` batch") with no inline gloss, and it is the only enumerated category that carries no direct spec citation at its point of use — sibling categories (c) and (d) cite their governing spec topics inline, while (b)'s only anchor is the bare "(the `V14b` tool-calls vector)" leaf parenthetical. An implementer reading `H4b` in isolation must open `V14b` or chase the cross-reference to recover both the definition and the `isError` / per-sibling `tool_use`-lowering semantics, degrading the self-containment of the leaf that owns the shared scripting contract.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `27e12be` (authored), `fd6169b` (relocated)
**History:** The response-programming-surface category-(b) clause "a mixed-success parallel `tool_use` batch" was first authored in `27e12be` (2026-06-10, "pi-loom plan: resolve \"Session double's model / tool / binder-response scripting surface is undefined\""); `git log -S "a mixed-success parallel" --all` reports it as the earliest plan-text commit containing the clause, at which point the surface lived in the then-combined `H4a` leaf and the clause cited the `V13c` tool-calls vector with no inline definition of "mixed-success". `fd6169b` (2026-06-12, "pi-loom plan: resolve \"H4a bundles three independent units into one leaf\"") extracted the standalone `docs/plan_topics/H4b-response-programming-surface.md` and carried the clause verbatim (the vector citation was later re-pointed to `V14b`). The term has been undefined inline since its first authoring; the defect predates the file extraction.

## Solution approach

In `H4b`'s **Adds** category (b), gloss "mixed-success" inline — a mixed-success parallel `tool_use` batch is one in which at least one sibling call succeeds and at least one fails (`execute()` throwing or resolving `{ content, isError: true }`). Add a direct spec cross-reference at the point of use to `tool-calls.md` §Concurrency (`../spec_topics/tool-calls.md#concurrency`), where the settle-all-before-next-turn and independent-per-sibling-lowering semantics are normatively defined, bringing category (b) into line with categories (c)/(d).

## Solution constraints

- Do not add a `Spec.` field — `H4b` is a horizontal leaf citing `Convention.`; the spec anchor belongs inline in the Adds prose, as for categories (c)/(d).
- Retain the existing "(the `V14b` tool-calls vector)" leaf parenthetical — `V14b` owns that vector cross-reference.
