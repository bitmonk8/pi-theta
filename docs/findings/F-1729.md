## Problem

In `docs/spec_topics/governance/anchor-scheme-and-retired.md` under the `id="retired-req-ids"` *Retired REQ-IDs* sub-table, the GOV-12 and GOV-14 rows carry the literal `` `pending` `` in their `Retired in` cell. The column-definition paragraph immediately below the table pins that column to "either the 7-character abbreviated commit SHA or a release tag", so `pending` is neither and the two rows contradict the column's own stated grammar. The sibling *Retired prefixes* sub-table on `req-id-prefix-table-retired.md` pins the same column to a strict SHA-or-tag value-space, so the contradiction is also cross-page. Any mechanical SHA/tag validator run against the column reddens those two cells.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** fe8aa5c — pi-loom spec: resolve "GOV-12 bundles ≥3 independently-testable sub-obligations" (2026-06-05, Thomas Andersen); 5a772e3 — pi-loom spec: resolve "GOV-14 two SHOULD-NOTs admit no mechanical witness" (2026-06-05, Thomas Andersen)
**History:** A prior review pass (commit `3f00256`, 2026-06-05) had explicitly closed off placeholder values in the *Retired in* column by replacing every `<retirement commit>` / `<demotion commit>` token with a concrete abbreviated SHA and by deleting the parenthetical that had previously authorised such placeholders. Later the same day, `fe8aa5c` appended the GOV-12 retirement row with `` `pending` `` in `Retired in`, and `5a772e3` appended the GOV-14 row using the same token; neither commit re-opened the column-definition paragraph (still pinning SHA-or-tag), so the contradiction was introduced by the interaction of the column-grammar tightening in `3f00256` and the two retirement-row appends in `fe8aa5c` / `5a772e3`.

## Solution approach

Rewrite the `` `pending` `` cell in each row's `Retired in` column with the 7-character abbreviated SHA of the retiring commit itself — `fe8aa5c` for GOV-12 and `5a772e3` for GOV-14 — so both cells match the column's stated SHA-or-tag grammar.

## Solution constraints

- Do not widen the *Retired REQ-IDs* column definition to admit a transient `pending` value; the SHA-or-tag value-space is shared with the *Retired prefixes* sub-table on `req-id-prefix-table-retired.md` and widening it here reintroduces cross-table inconsistency.
- Out of scope: amending the GOV-8 append-only edit procedure to mandate SHA-populated retirement rows.
