## Finding

`docs/spec_topics/governance/anchor-scheme-and-retired.md` §*Retired REQ-IDs* has two rows — GOV-12 and GOV-14 — whose `Retired in` cell carries the literal `` `pending` ``. The paragraph immediately under the same table states the value-space of that column: *"The `Retired in` column carries either the 7-character abbreviated commit SHA or a release tag."* `pending` is neither, so the two rows contradict the column's own grammar.

The grammar is also reinforced cross-page: the sibling *Retired prefixes* sub-table on `docs/spec_topics/governance/req-id-prefix-table-retired.md` pins its `Retired in` column to "the 7-character abbreviated commit SHA (e.g. `7851d7c`) or a release tag (e.g. `v0.42.0`), nothing else — no prose, no parentheticals, no qualifiers", and an earlier review pass (commit `3f00256`) deliberately removed an explicit `<demotion commit>` / `<retirement commit>` placeholder carve-out from both pages, replacing the placeholders with concrete SHAs (`64cdc60`, `1d773d6`, `8e3ccb4`, `877d57b`). The two retirements added afterwards (GOV-12 in `fe8aa5c`, GOV-14 in `5a772e3`) re-introduced the un-grammatical `pending` token without re-opening the column definition to permit it, leaving the retired-table grammar internally inconsistent and any mechanical SHA/tag validator that ran against the column poised to flag those two cells.

The retirements themselves are real and recorded; only the `Retired in` cells are wrong — they should carry the abbreviated SHA of the very commits that performed the retirement (`fe8aa5c` for GOV-12, `5a772e3` for GOV-14).

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** fe8aa5c — pi-loom spec: resolve "GOV-12 bundles ≥3 independently-testable sub-obligations" (2026-06-05, Thomas Andersen); 5a772e3 — pi-loom spec: resolve "GOV-14 two SHOULD-NOTs admit no mechanical witness" (2026-06-05, Thomas Andersen)
**History:** A prior review pass (commit `3f00256`, 2026-06-05) had explicitly closed off placeholder values in the *Retired in* column by replacing every `<retirement commit>` / `<demotion commit>` token with a concrete abbreviated SHA and by deleting the parenthetical that had previously authorised such placeholders. Later the same day, `fe8aa5c` appended the GOV-12 retirement row with `` `pending` `` in `Retired in`, and `5a772e3` appended the GOV-14 row using the same token; neither commit re-opened the column-definition paragraph (still pinning SHA-or-tag), so the contradiction was introduced by the interaction of the column-grammar tightening in `3f00256` and the two retirement-row appends in `fe8aa5c` / `5a772e3`.

## Solution Space

**Shape:** single

### Recommendation

Replace each `` `pending` `` cell in the *Retired REQ-IDs* sub-table with the 7-character abbreviated SHA of the retiring commit itself — `fe8aa5c` for the GOV-12 row and `5a772e3` for the GOV-14 row — so the column matches its stated SHA-or-tag grammar. Do **not** widen the column definition to admit a transient `pending` value: the sibling *Retired prefixes* table on `req-id-prefix-table-retired.md` and a prior review pass (commit `3f00256`) have already settled, for both retirement sub-tables, on a strict "SHA-or-tag, no prose, no parentheticals, no qualifiers" rule, and admitting a transient placeholder here would reintroduce the cross-table inconsistency that pass closed.

Implementer note: since the retiring commit's own SHA is what the cell must carry, the cell value is necessarily decided at retirement time. The convention that future retirement commits MUST land their row already populated with the retiring commit's abbreviated SHA — and never with a `pending` placeholder — is an obvious follow-on but is **out of scope** of this finding: it is GOV-8 *append-only* edit-procedure surface and should be raised as a separate finding against the governance procedure if it is not already covered there.

## Consequence

**Severity:** correctness

The table's own column-grammar paragraph and the GOV-12 / GOV-14 rows directly contradict each other, so any mechanical retired-ID validator (or human review pass) that takes the column definition at face value reddens those two cells, and any consumer that scrapes the `Retired in` field expecting a SHA/tag receives a non-resolvable token instead. The defect is bounded to two cells but it sits inside the governance corpus, which a downstream coverage / closing-gate workflow is expected to take as authoritative.
