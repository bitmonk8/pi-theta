## Problem

BNDR-4 (integer rendering), BNDR-5 (number rendering), and the numeric rows of the BNDR-6 reference-rendering table live in `docs/spec_topics/binder/defaulting-system-note-echo.md` under *Format rules*, a section whose declared purpose is rendering the per-call binding-echo display line. Those same rules are the authoritative numeric-serialisation algorithm for the canonical schema hash: `docs/spec_topics/schema-subset.md`'s Canonical schema hash § step 2 pins them (via `#bndr-4`, `#bndr-5`, `#bndr-6`) as the algorithm for serialising numeric literals in the lowered fragment, and that hash output names slug-keyed `$defs` entries and synthesised tool names. A central infrastructure contract therefore sources its normative numeric serialisation from a UX echo-formatter section, forcing a hash implementer to navigate out of the schema infrastructure into the binder display section to find the algorithm. The gap is one of discoverability and ownership clarity, not correctness.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** c5bb5e1 — pi-loom spec: resolve "Canonical schema hash numeric serialization underspecified" (2026-06-03, Thomas Andersen)
**History:** The numeric-rendering rules pre-existed as binder echo-formatter concerns. Commit c5bb5e1 created the ownership inversion by adding the Canonical schema hash step-2 clause that points the hash recipe at the binder *Echo policy › Format rules* `integer`/`number` rendering as its authoritative numeric-serialisation algorithm; before that commit no central contract depended on those rules and there was no inversion. A later edit (111e533, 2026-06-07) refined the cross-reference (per-anchor links, scope carve-out) but did not change the placement.

## Solution approach

Rewrite the *Format rules* heading and intro in `docs/spec_topics/binder/defaulting-system-note-echo.md` so the block reads as the canonical value-serialisation authority consumed both by the per-call binding-echo line and by the canonical schema-hash recipe in `schema-subset.md`, rather than a binder/UX-only concern; leave BNDR-4, BNDR-5, and the BNDR-6 reference table in place. In `schema-subset.md`, clarify the Canonical schema hash § step 2 numeric-literals bullet to state that the hash recipe consumes the numeric-serialisation rule from that canonical binder home.

## Solution constraints

- The borrowed-scope carve-out MUST remain intact: the 120-code-point line cap and the `(default)` suffix apply only to the echo line and MUST NOT apply to hashing.
- The `schema-subset.md` scope note and the relabelled binder heading MUST agree on which rules are shared (BNDR-4, BNDR-5, and the numeric rows of BNDR-6).
