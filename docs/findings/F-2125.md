## Finding

BNDR-4 (integer rendering) and BNDR-5 (number rendering), together with the numeric rows of the BNDR-6 reference-rendering table, are defined in `docs/spec_topics/binder/defaulting-system-note-echo.md` under *Echo policy › Format rules* — a user-facing section whose declared purpose is rendering the per-call binding-echo line.

However, those same rules are the authoritative numeric-serialisation algorithm for the **canonical schema hash**. `docs/spec_topics/schema-subset.md` (Canonical schema hash § step 2) pins: *"numeric literals embedded in the lowered fragment … are serialised by the `integer` / `number` rendering algorithm pinned in [Binder — Echo policy] *Format rules* (its [`integer`](#bndr-4) and [`number`](#bndr-5) value-rendering rules together with the [normative reference-renderings table](#bndr-6) …)."* The canonical schema hash is a pinned on-disk/on-wire contract: its 16-hex output (the **schema slug**) names `__inline_<slug>`, `__loom_respond_<slug>`, `__loom_callee_<slug>__<name>`, and `__loom_bind_<slug>`, and keys three slug-keyed caches.

The result is an ownership inversion: a central infrastructure contract sources its normative numeric serialisation from a section architecturally dedicated to UX echo display. An implementer or auditor reading `schema-subset.md` to build the hash must cross-navigate into the binder echo-formatter section to find the algorithm. The plan already recognises the rule as shared — leaf `V2d` ("Canonical integer/number renderer") factors it as "the canonical number-rendering pure function shared by the schema canonical-hash recipe (`V5f`) and the binder argument echo (`V11h`)" — so the spec-prose placement lags the design intent the plan already encodes.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** c5bb5e1 — pi-loom spec: resolve "Canonical schema hash numeric serialization underspecified" (2026-06-03, Thomas Andersen)
**History:** The numeric-rendering rules pre-existed as binder echo-formatter concerns. Commit c5bb5e1 created the ownership inversion by adding the Canonical schema hash step-2 clause that points the hash recipe at the binder *Echo policy › Format rules* `integer`/`number` rendering as its authoritative numeric-serialisation algorithm; before that commit no central contract depended on those rules and there was no inversion. A later edit (111e533, 2026-06-07) refined the cross-reference (per-anchor links, scope carve-out) but did not change the placement.

## Solution Space

### Recommendation

Keep BNDR-4, BNDR-5, and the numeric rows of the BNDR-6 reference-rendering table where they physically live in `docs/spec_topics/binder/defaulting-system-note-echo.md`, and reframe the *Format rules* block so it reads as the canonical value-serialisation authority shared by the binder argument echo and the canonical schema-hash recipe — not as a binder/UX-only concern.

Spec edits:

- In `defaulting-system-note-echo.md`, relabel the *Format rules* heading and rewrite its intro sentence to declare that this block is the canonical value-serialisation authority consumed both by the per-call binding-echo line and by the canonical schema-hash recipe in `schema-subset.md`. Leave BNDR-4, BNDR-5, and the BNDR-6 reference table (rows 6a–6x) in place and intact — do not split the table or migrate any BNDR REQ-IDs.
- In `schema-subset.md`, add a one-line scope statement to the Canonical schema hash § step 2 (numeric-literals bullet) noting that the hash recipe consumes the numeric-serialisation rule from the binder *Format rules* canonical home. The existing cross-reference already carries the exact per-anchor links and bounded scope; keep it.

Ensure the `schema-subset.md` scope note and the relabelled binder heading agree on exactly which rules are shared — BNDR-4, BNDR-5, and the numeric rows of BNDR-6 — and that the borrowed-scope carve-out remains intact: the 120-code-point line cap and the `(default)` suffix apply only to the echo line and do **not** apply to hashing.

## Consequence

**Severity:** advisory

A central infrastructure contract (the canonical schema hash) sources its normative numeric serialisation from a UX echo-formatter section, so every hash implementer must navigate out of the schema infrastructure into the binder display section to find the algorithm. No behavioural divergence results — the cross-reference carries exact anchors, worked examples (`1e21` → `1000000000000000000000`, `42.0` → `42`, `-0` → `0`), and an explicit carve-out of what is *not* borrowed — so the gap is one of discoverability and ownership clarity, not correctness.
