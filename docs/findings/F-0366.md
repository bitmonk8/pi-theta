## Finding

The per-type stringification Tests bullet in [`V6d`](../../../docs/plan_topics/V6d-system-interpolation.md) ends with a scope-attribution clause: "Row-level correctness stays owned by `V13a`; this leaf only routes the resolved value through the shared renderer." That clause is ownership/boundary prose — it states which leaf owns the stringification table and that `V6d` merely consumes it — not a condition the `system:` interpolation test asserts. A reader scanning `V6d`'s Tests for the REQ assertions the leaf claims to close has to mentally discard the trailing half-sentence, and a test author could misread it as an additional assertion to write.

Per the leaf template, `Adds.` is the descriptive field that records what the leaf introduces and the seams it consumes; scope-delegation belongs there, while `Tests.` bullets enumerate the assertions the leaf closes. The clause is correctly worded but mis-placed.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 3625ee0 — pi-loom plan: resolve "system: interpolation per-type stringification not witnessed" (2026-06-11, Thomas Andersen)
**History:** Before 3625ee0 the relevant Tests bullet read only "`\${` escapes interpolation; a Path-only `${…}` resolves and stringifies per the table." — there was no scope-delegation clause. That commit, resolving a separate finding about per-type stringification not being witnessed, rewrote the bullet into the long per-type-vector assertion and appended the "Row-level correctness stays owned by `V13a`; this leaf only routes the resolved value through the shared renderer" sentence at its tail. The misplaced clause is therefore exactly as old as that remediation edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V6d-system-interpolation.md`, strike the trailing sentence "Row-level correctness stays owned by `V13a`; this leaf only routes the resolved value through the shared renderer." from the per-type stringification Tests bullet, so that bullet ends at "…compact `JSON.stringify` with wire-name translation)."

Fold the same scope fact into the `Adds.` field, which is the leaf's descriptive home for consumed seams and ownership boundaries. Append to the existing `Adds.` sentence a clause stating that `V6d` routes resolved values through the `V13a`-owned shared stringification renderer and does not own row-level stringification correctness — e.g. extend `Adds.` to read "…the `\${` escape, and the stringification table; per-type stringification is delegated to the shared renderer owned by `V13a`, with `V6d` owning only the routing of resolved `system:` values into it."

The `V13a` dependency is already listed in `V6d` Deps, so this is a within-file move with no edits to other leaves or to the spec.

## Consequence

**Severity:** cosmetic

Pure organisation. No implementer diverges: the assertions in the bullet are intact and the V13a-ownership fact is true and visible. The only cost is that a test author reading `V6d`'s Tests must separate one assertion bullet from an embedded ownership note.
