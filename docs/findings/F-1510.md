## Finding

The *Lookup-failure-to-discriminator routing* paragraph in `unknown-reason-rule.md` correctly observes that the iteration primitive used to scan `SDK_SURFACE_INVENTORY` (`Array.prototype.find`, `Map.get` against a derived index, an explicit `for`/`switch`, "or any equivalent") is **leaf-owned** — but then mandates a specific code shape for one of the four arms: "to satisfy this routing the handler MUST guard the lookup with an explicit `SDK_SURFACE_INVENTORY === undefined` check **before** invoking the iteration primitive." This MUST elevates one syntactic guard form to normative status on a routing whose only observable property is the discriminator value.

The behavioural requirement the spec actually needs to pin — and already pins separately — is that an inventory that resolves to `undefined` at handler-entry time routes to `details.failure === "missing-entry"`, not to the `"throw:TypeError: …"` arm. Several equally-conformant code shapes deliver that routing: an explicit `=== undefined` check, a nullish guard (`SDK_SURFACE_INVENTORY == null`), a wider explicit `typeof SDK_SURFACE_INVENTORY === "undefined"` check, a `try`/`catch` whose catch arm inspects the thrown error's shape and re-routes a `TypeError: Cannot read properties of undefined` to `"missing-entry"`, or a fused predicate that combines the check with the lookup. Each of these satisfies the four-arm routing the paragraph already declares spec-owned. Mandating one of them by name forecloses the others without serving the operator-deduplication invariant the paragraph cites as the reason for spec ownership.

The companion `?.find` anti-pattern call-out (a naive `SDK_SURFACE_INVENTORY.find(…)?.literals` routes the case to the throw arm) is the *useful* rationale and should stay — it explains why some guard is mandatory. The defect is the form-specific MUST, not the requirement that a guard exist.

## Solution Space

**Shape:** single

### Recommendation

Replace arm (1)'s current sentence — "discriminator MUST be `"missing-entry"`, **not** any `"throw:TypeError: …"` value, and to satisfy this routing the handler MUST guard the lookup with an explicit `SDK_SURFACE_INVENTORY === undefined` check **before** invoking the iteration primitive (a naive `SDK_SURFACE_INVENTORY.find(…)?.literals` pattern routes this case to the throw arm instead and would split operator deduplication buckets across implementations whose iteration-primitive choice is otherwise immaterial)" — with a behavioural pin that keeps the `?.find` anti-pattern call-out as rationale:

> discriminator MUST be `"missing-entry"`, **not** any `"throw:TypeError: …"` value (a naive `SDK_SURFACE_INVENTORY.find(…)?.literals` pattern routes this case to the throw arm instead and would split operator deduplication buckets across implementations whose iteration-primitive choice is otherwise immaterial — so the handler must perform some guard sufficient to route the undefined-inventory case to `"missing-entry"`; an explicit `SDK_SURFACE_INVENTORY === undefined` check before invoking the iteration primitive is one such guard).

Edge cases for the implementer: the relaxation must not weaken the catch-arm discriminator's existing pin that a thrown `TypeError` originating from the undefined access routes to `"throw:…"` rather than `"missing-entry"` — under the looser spec text, an implementation that elects to route via the catch arm must still inspect the thrown error and re-classify the undefined-access `TypeError` as `"missing-entry"`, since the routing pin is fixed regardless of the guard form chosen. The paragraph's existing leaf-owned / spec-owned split already carries this invariant.

## Consequence

**Severity:** advisory

An implementer reading the current text is forced into a specific guard form even when an alternative (e.g. a nullish guard, or routing via the catch arm) would deliver the same operator-observable routing. No observer sees the difference; nothing breaks if implementers comply. The cost is a needless implementation-shape pin on a non-observable surface, on a page whose *Lookup-failure-to-discriminator routing* paragraph explicitly takes care to separate spec-owned routing from leaf-owned primitive choice — so the form pin reads as an internal inconsistency to the page's own discipline.
