## Problem

H4a's **Adds** declares the factory entry point as returning `void | Promise<void>` and "MUST NOT throw out of the factory body", and the first **Tests** bullet asserts the factory "never throws even when a host seam is absent". Both are worded only for the synchronous throw path. Because the declared return type admits `Promise<void>`, an `async` factory body can satisfy "MUST NOT throw" while still surfacing a failure as a rejected returned Promise — an observable host-boundary fault that passes H4a's stated gate yet is the exact Class-1 failure the leaf exists to prevent. Two reasonable implementers diverge: one writes a synchronous `void`-returning factory with no rejection path, the other an `async` factory whose internal `await` can reject.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** eafd4a0 — pi-loom plan: resolve "H4a factory framed as returning an extension object" (2026-06-10, Thomas Andersen)
**History:** Before eafd4a0, H4a's Adds framed the entry point as "the Pi extension factory entry point (returns an extension object without throwing)" and the Tests bullet as "the factory returns an extension object and never throws" — a synchronous, non-Promise return with no async-rejection path. Commit eafd4a0 reframed the factory to "the standard `default function (pi: ExtensionAPI)` factory returning `void | Promise<void>` … MUST NOT throw out of the factory body", introducing the `Promise<void>` return arm alongside a prohibition and a self-check worded only for synchronous throws, which is what opened the unaddressed returned-rejected-Promise gap.

## Solution approach

In `docs/plan_topics/H4a-factory-shell-and-harness.md` **Adds**, pin loom's factory to the synchronous arm of the host signature: state that the factory body runs synchronously and returns `void` (not declared `async`, awaits no work), so "MUST NOT throw out of the factory body" is the complete never-fault prohibition with no returned-`Promise` arm to reject. In the first **Tests** bullet, extend the "never throws" assertion to also witness that the factory returns synchronously (a non-`async`, non-thenable return), not merely that it does not throw synchronously.

## Solution constraints

- The factory's declared signature type stays `void | Promise<void>` for host-interface conformance; the fix pins which arm loom uses, it does not narrow the host signature.
- The spec is read-only for this fix; the spec already models synchronous registration (`extension-bootstrap-and-per-loom.md`).
