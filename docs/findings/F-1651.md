## Finding

`docs/spec_topics/runtime-value-model.md` contains a paragraph explicitly labelled "**Reference encoding (non-normative).**" whose last sentence carries a normative `MUST`:

> If a future host-interop surface ever exposes a live `Result` or enum value to JS (rather than a JSON-serialised projection), this section MUST be revisited before that surface ships.

A reader using the standard "scan for normative keywords" pass will either (a) skip the paragraph because the section header disclaims normativity and miss the revisit obligation, or (b) honour the `MUST` and be left uncertain about how much of the surrounding `__loomEnum` / `{ ok, value }` description is also binding. The two readings produce different conformance pictures, and the page provides no way to choose between them.

The fix is to separate the load-bearing obligation from the implementation-illustration paragraph it currently sits inside; the `__loomEnum` / `{ ok, value }` shapes are genuinely non-normative reference choices (witnessed by commit `0a7884a`, which deliberately demoted them out of the representation table), but the "revisit before that surface ships" rule is a real spec-maintenance obligation that belongs alongside the rest of the normative content.

**Verdict:** single-commit-introduction

## Solution Space

**Shape:** single

### Recommendation

Split the offending sentence out of the "Reference encoding (non-normative)." paragraph and promote it to a peer normative sentence on the page. Concretely, replace the trailing clause with two paragraphs:

> **Reference encoding (non-normative).** The reference interpreter implements the enum tag as a non-enumerable `__loomEnum` string property on the JS string wrapper, and represents `Result<T, E>` as `{ ok: true, value: T }` for `Ok(v)` and `{ ok: false, error: E }` for `Err(e)`. These shapes are implementation details — neither is reachable from loom code, neither appears in any wire schema, and either may change without a spec revision.
>
> **Revisit trigger.** Before any future host-interop surface exposes a live `Result` or enum value to JS (rather than a JSON-serialised projection), this page MUST be revised to state the wire-observable contract for that surface.

The "(non-normative)" label stays attached only to the `__loomEnum` / `{ ok, value }` description it was meant to cover; the maintenance obligation becomes a one-line normative paragraph a keyword scan will catch. Do not weaken the `MUST` to a `SHOULD` and do not delete it — the obligation is a real spec-revisit gate the host-interop work needs.

Edge case: keep the paragraph on `runtime-value-model.md` rather than relocating it to a governance or host-interop page; the trigger condition is phrased in terms of this page's own contract ("a live `Result` or enum value", "rather than a JSON-serialised projection") and the obligation is to revise this page.

## Consequence

**Severity:** cosmetic

Two reasonable readers reach opposite conclusions about whether the `MUST` binds them, but the obligation is a future-facing spec-maintenance trigger ("revisit before a not-yet-existing surface ships"), not an implementation rule any current leaf will act on. No behaviour observable today changes either way.
