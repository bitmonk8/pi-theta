## Finding

The schema-inference algorithm at `docs/spec_topics/query/query-forms.md` §Schema inference algorithm partitions every enclosing AST construct into two lists — **Crossed (transparent)** and **Stopped (opaque)** — and tells the implementer to walk outward until one of them is hit. The postfix error-propagation `?` operator (ERR-18) sits between the query and the sink in the dominant typed-query idiom that the spec uses as its very first worked example: `let x: ReviewScore = @\`...\`?`. That operator appears in neither list.

The classification matters and is unrecoverable from elsewhere on the page. The "Stopped (opaque)" list does name **unary** operators by enumerating only `!` and `-`, and lists postfix `?` nowhere; an implementer reading the bullet literally has at least two plausible readings: (a) `?` is an unenumerated unary/postfix operator and therefore opaque, in which case the walk stops before reaching the binding annotation and the query is untyped — making the spec's own first worked example silently violate QRY-2 and the typed-query idiom unwritable without an explicit `@<Schema>` ascription; or (b) `?` is implicitly transparent because the operand-type precondition (ERR-18) guarantees `Result<T, _>` on the inside and the unwrapped `T` is propagated outward, in which case the binding annotation supplies the schema and the worked example works. Reading (b) is the intended one — every typed-query example on the page relies on it — but the algorithm does not state it, and reading (a) is the literal one.

The fix is to add postfix `?` (the operand-of position) to the **Crossed (transparent)** list and to update one worked example to make the unwrap-then-flow path observable. The choice is forced: opaque would break QRY-2 in the spec's flagship idiom and contradict every typed-query worked example on the page; only transparent is consistent with the rest of the section.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/query/query-forms.md` §Schema inference algorithm, add postfix `?` (operand-of position) to the **Crossed (transparent)** bullet alongside parenthesisation and the RHS of `let x: T = …`, with one explanatory clause naming the unwrap: "the operand of postfix `?` (the walk continues from `?`'s enclosing context into the operand because `?` requires its operand to be `Result<T, QueryError>` per [ERR-18](../errors-and-results/error-model.md#err-18) and yields `T`; the sink seen from outside `?` supplies a schema for `T`, which is the response type of the inner `@`-query)."

Make the unwrap step observable in the worked-examples list: in the existing `let x: ReviewScore = @\`...\`? — sink at the binding annotation. ✅` bullet, add a half-clause "(the walk crosses `?` per the unwrap rule; the binding annotation `ReviewScore` is the schema)" so a reader cross-referencing the algorithm and the example sees the rule applied. The `@\`...\`? + 1` example need not change — `+` is opaque on either operand, so the walk stops at `+` before reaching `?` regardless.

Implementer-relevant edge cases: (i) the rule names only the postfix `?` operand position — a `?` nested inside any opaque operator (`f(@\`...\`?) + 1`, `match @\`...\`? { … }`) remains opaque at the outer operator per the existing Stopped bullet, and the inner `?` cannot rescue typing; (ii) the rule composes with the explicit-ascription override (QRY-3 / QRY-4) without change — an explicit `@<Schema>\`…\`?` continues to win regardless of where it appears, since the override is independent of the walk; (iii) the `match @\`...\` { … }` worked example does not gain `?` (a query inside a `match` scrutinee is opaque at `match` whether or not `?` is present), so its rationale ("explicit form required") is unchanged.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on the typed-query idiom `let x: T = @\`...\`?`. One treats postfix `?` as an unenumerated unary/postfix operator and classifies it as opaque (per the second bullet's "binary and unary operators … as do the unary `!` and `-`"), making every typed-query that uses `?` untyped and rejecting the spec's own first worked example as a QRY-2 violation. The other treats it as transparent by analogy with the RHS of `let`, matching the worked examples. Both readings are defensible from the page as written; the conformance test set inherited by `V13b` cannot adjudicate because the normative rule does not exist.
