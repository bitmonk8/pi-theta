## Problem

The *No globals, statics, singletons* rule in `conventions.md` and the matching `Convention.` test bullets in `H3a` enumerate the banned ambient/timing primitives and assert the timing set is sourced from the spec's `host-interfaces-services.md` `Clock`/`WallClock` ban "so the two stay in lockstep". The word "lockstep" promises a maintained equivalence, but the enumerated list is a hand-copied snapshot embedded in the `H3a` scan source with no reconciliation mechanism — neither a CI check nor a shared single definition — re-checking it against the spec ban after `H3a` lands. If a future spec edit extends the `Clock`/`WallClock` surface, the scan's hardcoded list does not move with it and nothing flags the divergence; the list already carries `Date.prototype.getTime` beyond the spec's literal enumeration, confirming the two sides are a manual derivation rather than a mechanically-equal pair.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `c6824f20a6d3b4539e10ad4b57ba53fe07bcd9a7` — "pi-loom plan: resolve \"ambient-primitive scan omits spec-banned clock primitives\"" (2026-06-12)

**History:** The ambient-primitive ban itself predates this commit (present in `conventions.md` since the docs reorg `31ff060`), but in its earlier form it enumerated only `process.env`, `process.cwd`, `crypto.randomUUID`, `Date.now`, `setTimeout` and carried no equivalence claim. Commit `c6824f2`, while closing the prior review finding "ambient-primitive scan omits spec-banned clock primitives", expanded the timing set to `Date.now`, `performance.now`, `Date.prototype.getTime`, `setTimeout`, `clearTimeout` in both `conventions.md` and `H3a-di-seam-skeleton.md` and simultaneously added the "sourced directly from the `host-interfaces-services.md` `Clock`/`WallClock` ban so the two stay in lockstep" phrasing (`git show c6824f2` shows both insertions). The defect is fix-induced over-promise: the same edit that broadened the list also introduced the unbacked "lockstep" guarantee, with no reconciliation mechanism added alongside it. No later commit closed the gap.

## Solution approach

Delete the "so the two stay in lockstep" clause from the `conventions.md` *No globals, statics, singletons* rule and from `H3a`'s `Convention.` bullets, restating the enumerated timing/ambient set as a manually-maintained derivation of the spec's `host-interfaces-services.md` `Clock`/`WallClock` ban. Add an item to the `loom 1.0 release-time residue inspection` checklist in `conventions.md` that re-reconciles the scan's primitive list against the current spec ban, following the existing GOV-22/GOV-15 residue-routing pattern already used by that checklist.

## Solution constraints

- The spec is read-only for this fix; the ban stays owned by `host-interfaces-services.md` (PIC-12) and the reconciliation is a plan-side review surface — do not add a machine-readable spec artifact.
- The scan set is an intentional superset of the spec's literal ban (it includes `Date.prototype.getTime` beyond the spec's `performance.now`/`Date.now`/`setTimeout`/`clearTimeout` enumeration); the reconciliation must not require strict equality with the spec ban.
