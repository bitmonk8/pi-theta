## Finding

SLSH-4 row **SNK-j** defines the per-`kind` template for `invoke_callee` as `"loom /<name> returned Err: invoked <callee_path> returned Err — <inner.kind>"` and SLSH-4's umbrella rule mandates that renderers MUST emit it verbatim. SLSH-5, however, mandates that on any `invoke_callee` cascade (i.e. every concrete `InvokeCalleeError`, since the variant's `inner: QueryError` field is required per [`errors-and-results/queryerror-variants.md`](docs/spec_topics/errors-and-results/queryerror-variants.md) — there is no inner-less `invoke_callee`) "the descriptive text in the per-`kind` row is computed from the **leaf** `kind` (the innermost non-`invoke_callee` variant), not from the wrapper". The two rules cover the same input and prescribe different output strings: SNK-j would emit `"… invoked /abs/child.loom returned Err — transport"`, while SLSH-5 forces the SNK-c row to be used with the leaf's fields, yielding `"… transport — connection reset from /abs/child.loom invoked at /abs/parent.loom:42"`.

Nothing in the page states which rule wins. A conformance test asserting on SNK-j's verbatim text (which the SLSH-4 umbrella explicitly permits, and which `V12b`'s `SNK-j` test bullet asserts as a discrete obligation) is satisfiable only by a renderer that violates SLSH-5; a renderer that satisfies SLSH-5 fails SNK-j. The SNK-j row is, in practice, dead code superseded by SLSH-5, but a reader of SLSH-4 alone cannot know that.

The same hazard does not affect the other ten rows: each of SNK-a … SNK-i and SNK-k describes the formatting of its own leaf `kind`, and SLSH-5 explicitly says the chain suffix is appended *to* those per-`kind` rows. SNK-j is the unique row whose `kind` is the one SLSH-5 strips out.

## Issue introduction

**Verdict:** single-commit-regression
**Introducing commit:** `2d217b1` (2026-05-06, *pi-loom spec: resolve "Call chain suffix is illustrative, not normative"*)
**History:**

- `907c029` (2026-05-04) added the `invoke_callee_error` row to the per-`kind` table with the template `"… invoked <callee_path> returned Err — <inner.kind>"`. At that point the table row was the renderer's sole output for an `invoke_callee_error`; an inline parenthetical hinted at chain-printing but was not normative ("the note identifies the leaf and prints the call chain"). No ordering conflict existed: the row was the rule.
- `2d217b1` (2026-05-06) elevated the chain-attribution paragraph to a normative MUST and added the leaf-kind clause: *"the descriptive text in the per-`kind` row is computed from the **leaf** `kind` (the innermost non-`invoke_callee_error` variant), not from the wrapper"*. This is the regression site: the new rule made the `invoke_callee_error` row unreachable for any concrete `InvokeCalleeError` (whose `inner` is non-optional), but the row was not removed and no supersession note was added.
- `27b2576` (2026-06-05) added the `SNK-a … SNK-k` inline labels for citation, including `SNK-j` for the already-dead row. This formalised the unreachable row as a separately-citable obligation (and propagated into `V12b`'s test list as a discrete bullet), amplifying the defect's surface but not introducing it.
- `1ccbb75` (2026-06-07) refined `<line>` definition and renamed `invoke_callee_error` → `invoke_callee` to match the wire-kind pattern; the leaf-kind / wrapper-row conflict was preserved verbatim through both renames.

## Solution Space

**Shape:** multiple

### Option A — Remove SNK-j; replace it with a prose note

**Approach.** Delete the SNK-j row from the SLSH-4 table. In its place, add a short prose paragraph (or footnote on SLSH-4) stating that `invoke_callee` has no standalone row because SLSH-5 always supersedes it: the renderer selects the per-`kind` row of the *leaf* `kind` and appends the chain suffix. Keep the existing "For `invoke_callee` the chain-attribution suffix described in the next paragraph handles the deeper `inner` recursion …" sentence and extend it to say the row is intentionally absent. Update [`V12b`](docs/plan_topics/V12b-top-level-err-chain.md) and [`V12b-T`](docs/plan_topics/V12b-T-top-level-err-chain.md) to drop the `SNK-j: the invoke_callee note renders verbatim` bullet (its content is now covered by SLSH-5 plus the leaf row). Update [`coverage-matrix.md`](docs/plan_topics/coverage-matrix.md)'s `SNK-a … SNK-k` closure paragraph to read `SNK-a … SNK-i, SNK-k` (ten labels) and note that SNK-j was retired by the SLSH-5 supersession edit. Update `GOV-16`'s `SNK` prefix registration if it pins the label count.

**Spec edits.**
- `docs/spec_topics/slash-invocation.md`: delete the SNK-j table row; add one or two normative sentences immediately after the table stating that `invoke_callee` has no dedicated row because every `InvokeCalleeError` triggers SLSH-5's leaf-kind selection.
- `docs/plan_topics/V12b-top-level-err-chain.md` and `docs/plan_topics/V12b-T-top-level-err-chain.md`: remove the `SNK-j` bullet from the Tests list.
- `docs/plan_topics/coverage-matrix.md`: adjust the `SNK-a … SNK-k` closure-rationale prose to the new ten-label range.

**Pros.**
- Removes the dead row instead of papering over it. After the edit, SLSH-4's verbatim-emit MUST has no row it cannot honour.
- Eliminates the `V12b` test-pair contradiction by construction — there is no second obligation to assert against.
- The per-`kind` table becomes a faithful enumeration of rows a renderer actually emits, matching SLSH-4's "exhaustive over the `QueryError` union" prose once the prose is updated to call out the SLSH-5-mediated coverage of `invoke_callee`.

**Cons.**
- Touches three plan-side files (`V12b`, `V12b-T`, `coverage-matrix.md`) and the `GOV-16` prefix registration; wider edit surface than Option B.
- Breaks the "one row per `QueryError.kind`" symmetry of the table; readers scanning the table for `invoke_callee` find the prose-only treatment instead. Mitigated by leaving the prose note in the row's former position.
- The `SNK-a … SNK-k` label range becomes non-contiguous if `SNK-j` is retired without renumbering. Renumbering would break stable-label citations elsewhere; leaving the gap is the lesser harm but is itself a small ongoing cost.

**Risks.**
- A test author who has already cached the eleven-label list may write `SNK-j` assertions; the `coverage-matrix.md` retirement note and the in-table prose are the only places that flag the absence.

### Option B — Keep SNK-j; add an explicit "SLSH-5 supersedes SNK-j" ordering rule

**Approach.** Leave the SNK-j row and its label in place. Add a normative sentence to the SLSH-4 umbrella (or, equivalently, to the existing "For `invoke_callee` the chain-attribution suffix …" sentence following the table) of the form: *"SNK-j applies only when SLSH-5 does not fire. Since `InvokeCalleeError.inner` is required by [`InvokeCalleeError`](./errors-and-results/queryerror-variants.md#invoke-variants), SLSH-5 fires on every `invoke_callee` and supersedes SNK-j; SNK-j is retained for forward-compatibility with any future `invoke_callee`-shaped variant exempt from chain attribution."* Reword the `V12b` `SNK-j` test bullet to assert the supersession (e.g. *"`SNK-j`: superseded by SLSH-5 on every concrete `InvokeCalleeError`; the renderer never emits the SNK-j row's verbatim text in loom 1.0.0"*) so the test agrees with the spec.

**Spec edits.**
- `docs/spec_topics/slash-invocation.md`: add one or two sentences of ordering text either to the SLSH-4 umbrella or to the post-table paragraph; no row edits.
- `docs/plan_topics/V12b-top-level-err-chain.md` and `docs/plan_topics/V12b-T-top-level-err-chain.md`: reword the `SNK-j` bullet to assert the supersession rather than verbatim emission.

**Pros.**
- Smallest spec edit; no row removal, no label-range disturbance, no `coverage-matrix.md` rewrite, no `GOV-16` change.
- Preserves SNK-j as a citation anchor in case the `QueryError` union gains a future `invoke_callee`-shaped variant that is intentionally exempt from chain attribution.

**Cons.**
- The `invoke_callee` row stays in the table even though no loom 1.0.0 renderer ever emits it. The dead-code smell is documented away rather than removed.
- Asks readers to hold a two-step rule — *"SLSH-4 says verbatim; SLSH-5 wins on `invoke_callee`"* — instead of presenting the rendering rule as a single coherent recipe.
- The forward-compatibility argument is speculative: the spec defines no such future variant and `NOCEIL-1 … NOCEIL-4`-style non-existence claims are the corpus's normal way to handle "may exist later" extensions.

**Risks.**
- A future editor who removes the supersession sentence (treating it as redundant prose) silently reintroduces the conflict.

### Recommendation

Take **Option A**. The defect is a dead row; the right fix is to delete it, not to add prose explaining why it stays dead. SNK-j's only function under SLSH-5 is as a coverage-table placeholder, and SLSH-4's "exhaustive over the `QueryError` union" property is preserved by an explicit prose note that `invoke_callee` is covered by SLSH-5 + the leaf row. The retired-label gap in the `SNK-a … SNK-k` range is a one-time cosmetic cost; the forward-compatibility argument in Option B does not justify keeping an unreachable normative MUST in the table.

Implementer notes:
- When deleting the row, also delete the `<a id="snk-j"></a>` anchor; grep the corpus for `#snk-j` / `SNK-j` and remove or reword each citation. The two `V12b` plan files and the `coverage-matrix.md` closure paragraph are the known sites; `GOV-16`'s prefix registration is the other one to check.
- Keep the post-table "For `invoke_callee` the chain-attribution suffix described in the next paragraph handles the deeper `inner` recursion …" sentence and extend it to explicitly state the row's intentional absence — that sentence is the natural home for the prose note.
- Do not renumber `SNK-k` down to `SNK-j`; the inline labels are governed identifiers under `GOV-16` and renumbering would break existing citations for no reader benefit.

## Consequence

**Severity:** correctness

Two conforming implementers diverge on every user-visible `invoke_callee` failure note. A test author who reads SLSH-4 in isolation will assert SNK-j's verbatim string and reject every SLSH-5-compliant renderer; a test author who reads SLSH-5 in isolation will assert the chain-attribution form and reject every SNK-j-compliant renderer. The current `V12b` test list bakes the conflict into the plan by enumerating both as independent obligations.
