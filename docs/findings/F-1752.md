## Finding

SLSH-4 row **SNK-j** defines the per-`kind` template for `invoke_callee` as `"loom /<name> returned Err: invoked <callee_path> returned Err — <inner.kind>"` and SLSH-4's umbrella rule mandates that renderers MUST emit it verbatim. SLSH-5, however, mandates that on any `invoke_callee` cascade (i.e. every concrete `InvokeCalleeError`, since the variant's `inner: QueryError` field is required per [`errors-and-results/queryerror-variants.md`](docs/spec_topics/errors-and-results/queryerror-variants.md) — there is no inner-less `invoke_callee`) "the descriptive text in the per-`kind` row is computed from the **leaf** `kind` (the innermost non-`invoke_callee` variant), not from the wrapper". The two rules cover the same input and prescribe different output strings: SNK-j would emit `"… invoked /abs/child.loom returned Err — transport"`, while SLSH-5 forces the SNK-c row to be used with the leaf's fields, yielding `"… transport — connection reset from /abs/child.loom invoked at /abs/parent.loom:42"`.

Nothing in the page states which rule wins. A conformance test asserting on SNK-j's verbatim text (which the SLSH-4 umbrella explicitly permits, and which `V12b`'s `SNK-j` test bullet asserts as a discrete obligation) is satisfiable only by a renderer that violates SLSH-5; a renderer that satisfies SLSH-5 fails SNK-j. The SNK-j row is, in practice, dead code superseded by SLSH-5, but a reader of SLSH-4 alone cannot know that.

The same hazard does not affect the other ten rows: each of SNK-a … SNK-i and SNK-k describes the formatting of its own leaf `kind`, and SLSH-5 explicitly says the chain suffix is appended *to* those per-`kind` rows. SNK-j is the unique row whose `kind` is the one SLSH-5 strips out.

## Issue introduction

**Verdict:** single-commit-regression
**Introducing commit:** `2d217b1` (2026-05-06, *pi-loom spec: resolve "Call chain suffix is illustrative, not normative"*)
**History:**

- `907c029` (2026-05-04) added the `invoke_callee_error` row to the per-`kind` table with the template `"… invoked <callee_path> returned Err — <inner.kind>"`. At that point the table row was the renderer's sole output for an `invoke_callee_error`; an inline parenthetical hinted at chain-printing but was not normative ("the note identifies the leaf and prints the call chain"). No ordering conflict existed: the row was the rule.
- `2d217b1` (2026-05-06) elevated the chain-attribution paragraph to a normative MUST and added the leaf-kind clause: *"the descriptive text in the per-`kind` row is computed from the **leaf** `kind` (the innermost non-`invoke_callee_error` variant), not from the wrapper"*. This is the regression site: the new rule made the `invoke_callee_error` row unreachable for any concrete `InvokeCalleeError` (whose `inner` is non-optional), but the row was not removed and no supersession note was added.
- `27b2576` (2026-06-05) added the `SNK-a … SNK-k` inline labels for citation, including `SNK-j` for the already-dead row. This formalised the unreachable row as a separately-citable obligation (and propagated into `V12b`'s test list as a discrete bullet), amplifying the defect's surface but not introducing it.
- `1ccbb75` (2026-06-07) refined `<line>` definition and renamed `invoke_callee_error` → `invoke_callee` to match the wire-kind pattern; the leaf-kind / wrapper-row conflict was preserved verbatim through both renames.

## Solution Space

### Recommendation

Delete the SNK-j row from the SLSH-4 per-`kind` table in `docs/spec_topics/slash-invocation.md`, including its `<a id="snk-j"></a>` anchor. In its place, immediately after the table, add one or two normative sentences stating that `invoke_callee` has no dedicated per-`kind` row because every concrete `InvokeCalleeError` (whose `inner: QueryError` field is required) triggers SLSH-5's leaf-kind selection: the renderer selects the per-`kind` row of the *leaf* `kind` (the innermost non-`invoke_callee` variant) and appends the chain-attribution suffix. Extend the existing post-table sentence — "For `invoke_callee` the chain-attribution suffix described in the next paragraph handles the deeper `inner` recursion …" — to explicitly state that the row is intentionally absent; that sentence is the natural home for the prose note. This keeps SLSH-4's "exhaustive over the `QueryError` union" property intact: `invoke_callee` is covered by SLSH-5 plus the leaf row.

Apply the matching edits across the plan side:

- `docs/spec_topics/slash-invocation.md`: delete the SNK-j table row and its anchor; add the post-table normative sentences described above.
- `docs/plan_topics/V12b-top-level-err-chain.md` and `docs/plan_topics/V12b-T-top-level-err-chain.md`: remove the `SNK-j: the invoke_callee note renders verbatim` bullet from the Tests list — its content is now covered by SLSH-5 plus the leaf row.
- `docs/plan_topics/coverage-matrix.md`: adjust the `SNK-a … SNK-k` closure-rationale prose to the ten-label range `SNK-a … SNK-i, SNK-k`, and note that SNK-j was retired by the SLSH-5 supersession edit.
- `GOV-16`'s `SNK` prefix registration: update it if it pins the label count.

Do not renumber `SNK-k` down to `SNK-j`; the inline labels are governed identifiers under `GOV-16` and renumbering would break existing citations. Leave the `SNK-a … SNK-k` range non-contiguous, with the gap at SNK-j flagged by the `coverage-matrix.md` retirement note and the in-table prose. Grep the corpus for `#snk-j` / `SNK-j` and remove or reword each remaining citation; the two `V12b` plan files, the `coverage-matrix.md` closure paragraph, and the `GOV-16` prefix registration are the known sites.

## Consequence

**Severity:** correctness

Two conforming implementers diverge on every user-visible `invoke_callee` failure note. A test author who reads SLSH-4 in isolation will assert SNK-j's verbatim string and reject every SLSH-5-compliant renderer; a test author who reads SLSH-5 in isolation will assert the chain-attribution form and reject every SNK-j-compliant renderer. The current `V12b` test list bakes the conflict into the plan by enumerating both as independent obligations.
