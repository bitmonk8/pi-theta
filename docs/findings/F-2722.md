## Finding

`real-host-smoke-gate.md` defines the manual real-host smoke run as the pre-merge gate that catches a double-vs-real divergence on the four fidelity-contract axes (streamed-token order relative to `ctx.waitForIdle()`, single-turn prompt-mode append, the `pi.on` cancel-forward subscription, and cancellation propagation). The trigger-(2) prose concludes that for "any merge whose diff touches the four fidelity-contract axes … a divergence introduced by a loom change rather than a Pi bump is **caught before it reaches `main`**." Two sentences later the same paragraph states the opposite: the step is a "**contributor obligation, not a CI-enforced gate**," loom 1.0 "has no mechanical hook that observes whether the smoke ran," trigger-(2) "has no automatic diff detector," and a skipped or unrecognised run is "**not blocked from merging** … detectable only **after the fact**" by the `H6a` auditor reading the triggering commit message.

These two claims cannot both be true. Nothing observes whether the smoke ran, nothing detects that a diff touched the four axes, and the only backstop is `H6a`'s post-hoc audit at the single release event. A contributor who does not recognise their diff as fidelity-axis-touching merges with no signal, and the divergence — on the production seam underlying every real-host invocation — reaches `main` unverified. The `ERR-8` live-surface confirmation (criterion (c)) is deferred here from `npm test` by design, so that behaviour has no automated assertion anywhere; the over-claiming clause masks that gap behind a guarantee the mechanism does not deliver.

The actionable defect is the internal contradiction: the "caught before it reaches `main`" clause asserts a pre-merge gate the doc itself elsewhere disclaims. A reader taking the over-claim at face value misjudges the residual risk loom 1.0 ships with and may assume a mechanical detector exists to build against.

## Consequence

**Severity:** correctness

A double-vs-real divergence on the four fidelity axes can reach `main` unverified, while the doc simultaneously promises it is "caught before it reaches `main`." Two reasonable implementers diverge: one reads the over-claim and assumes a mechanical pre-merge detector to build/rely on; the other reads the disclaimer and builds nothing — and the doc cannot adjudicate between them because it asserts both.

## Solution Space

**Shape:** multiple

### Option A — Soften the over-claim to match the manual-gate design

**Approach.** Reconcile the contradictory prose so the note no longer asserts a pre-merge guarantee the mechanism does not provide, accepting the manual gate as the deliberate loom-1.0 design (echoed in `H6a` and `H4a`, both of which state loom 1.0 has no mechanical real-host fidelity gate).

**Plan edits.** In `real-host-smoke-gate.md`, replace the trailing trigger-(2) clause "so a divergence introduced by a loom change rather than a Pi bump is caught before it reaches `main`" with prose stating that the contributor's obligation is to run the smoke before the merge lands, but that — absent any CI hook or automatic diff detector — an unrecognised fidelity-axis divergence can reach `main` and is detectable only after the fact via the `H6a` commit-message audit, consistent with the "contributor obligation, not a CI-enforced gate" sentence already in the same paragraph. Note inline that the `ERR-8` live-surface confirmation (criterion (c)) consequently has no automated assertion under `npm test` by design.

**Spec edits.** None.

**Pros.** Smallest diff; makes the note internally consistent; faithfully reflects the deliberate manual-gate design and accurately states the residual risk loom 1.0 ships with.

**Cons.** Leaves the residual risk unmitigated — divergence reaching `main` relies wholly on contributor discipline plus the after-the-fact `H6a` audit.

**Risks.** Low.

### Option B — Soften, and add best-effort detection

**Approach.** Everything in Option A, plus a best-effort, non-gating detector mirroring the `H5b` warn-only canary idiom: a CI step that flags diffs touching the fidelity-axis seam files / double-modelling modules and prompts "trigger (2) may apply — run the real-host smoke," and/or a CI grep asserting a smoke-evidence line is present in the triggering commit; plus a `H6a` release-gate auditor obligation to enumerate every fidelity-axis-touching commit since the last activation and reconcile each against a recorded smoke result.

**Plan edits.** Add a "best-effort detection" paragraph to `real-host-smoke-gate.md`; add the warn-only CI canary to `H2a` (or a new horizontal leaf `<new>`); extend `H6a`'s Ships-when auditor obligation to reconcile fidelity-axis-touching commits against recorded smoke results.

**Spec edits.** None.

**Pros.** Reduces the real risk by surfacing likely trigger-(2) diffs at merge time; consistent with the existing `H5b` warn-only-canary pattern.

**Cons.** Larger change spanning multiple leaves; a path/heuristic detector is necessarily best-effort and cannot prove a diff touches the four axes, so it adds CI maintenance and a possible false sense of completeness while remaining advisory, not gating.

**Risks.** Contributors may over-trust an advisory signal; the detector's seam-file list needs its own maintenance owner (compare the permitted-code-list staleness gap already noted on this doc).

### Recommendation

Take **Option A**. The plan deliberately designates this a manual gate — `real-host-smoke-gate.md`, `H6a`, and `H4a` all state loom 1.0 has no mechanical real-host fidelity gate — so the actionable defect is the internal contradiction, which a localized edit fully resolves. The spec host-prerequisites the note cites are read-only for this fix, and the actual gate criteria, merge-blocker, and revert path must not be weakened. Watch that the softened clause stays consistent with `H6a`'s parallel "mis-recorded run is detectable after the fact by reading the commit message" wording and with the existing deferral / precondition "un-runnable-gate" framing, and that it introduces no new claim that the `H6a` audit can catch a dishonestly-recorded result.
