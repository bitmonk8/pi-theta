## Finding

BNDR-2 states that `ambiguous.candidates` MUST remain in the envelope schema "(`array<string> | null`; the binder may emit it; AJV accepts `null`)". The clause "the binder may emit it" is ambiguous. Read one way it means the binder *may include the key or omit it* — i.e. `candidates` is optional. Read another way it means the binder *may set it to a non-null value* — i.e. the key is always present but its value may be `null` or an array. The two readings imply different envelope schemas.

The authoritative schema block in the same section resolves the question against the "optional" reading: the `ambiguous` arm declares `"required": ["kind", "message", "candidates"]` with `"candidates": { "type": ["array", "null"], "items": { "type": "string", "maxLength": 500 } }`. So the contract is: the `candidates` key is **always present** (it is in `required`); its value is either `null` or an array of strings (empty arrays are admitted — there is no `minItems`). The prose phrase "may emit it" pulls in the opposite direction from the schema it is annotating.

This matters because BNDR-2 itself declares that "Dropping it from the schema is breaking", and the relaxed-copy / `additionalProperties: false` discipline elsewhere in the section makes envelope conformance byte-sensitive. An implementer who weights the prose over the schema could omit `candidates` from the `required` list (treating it as optional), producing a non-conforming envelope on a surface BNDR-2 explicitly pins as a breaking-change contract.

## Solution Space

**Shape:** single

### Recommendation

Replace "the binder may emit it" with wording that matches the schema's `required` membership and `["array", "null"]` type: state that the `candidates` key is always present in the `ambiguous` arm and its value is either `null` (when the binder has no candidates to record) or an array of strings. For example: "(`array<string> | null`, always present per the envelope schema's `required` list; the binder sets it to `null` when it records no candidates, otherwise an array of strings; AJV accepts `null`)." Do not introduce any "optional"/omittable framing, and keep the existing "Dropping it from the schema is breaking" and non-surfacing sentences intact. Edge case to preserve: empty arrays remain valid (no `minItems`), so the wording must not imply a non-empty array is required.

## Consequence

**Severity:** advisory

The adjacent JSON schema block unambiguously lists `candidates` in `required`, so an implementer who consults it produces a conforming envelope. The risk is confined to a reader who acts on the loose "may emit it" prose alone and makes `candidates` optional in the schema — which BNDR-2 itself calls a breaking change. The defect is a wording gap, not a missing rule.
