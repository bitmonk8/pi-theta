## Finding

The capability-probe page declares a single source of truth for the four
probe constants (Node floor, `AbortSignal`/`AbortController` member-with-kind
list, factory-probable capability list, peer-dep range) plus
`CAPABILITY_OBLIGATIONS`, stating they "live in one source of truth inside the
extension module that the build-time literal-read assertions also consume."
Two distinct build-time consumers depend on reading that block: the step-2(a)
positive-direction surface-inventory literal-read assertion, and the
`CAPABILITY_OBLIGATIONS.length === 7` cardinality assertion. Both must import
the extension-module-resident constants at build time and evaluate them
mechanically.

The spec never states the precondition that makes this consumption sound: that
the pinned-constants block is reachable by a **side-effect-free import** —
i.e. importing it to read the constants MUST NOT invoke the
`default function (pi: ExtensionAPI)` factory, perform any `pi.register*` /
`pi.on` host-binding call, or trigger any other import-time side effect. The
extension entry point (`extensions/index.ts`) is precisely the module that
both holds the factory (whose body performs all registration) and is the
natural home for the constants; nothing pins where the block lives relative to
the factory, nor that the constants must sit in a position whose static
evaluation is independent of factory invocation. Two reasonable implementers
diverge: one places the block as top-level `const` declarations in
`index.ts` (sound today, but silently coupled to any future import-time side
effect added to that module), another in a dedicated side-effect-free
constants module. The build-time assertions' soundness rests on the former
remaining side-effect-free, which the spec leaves implicit.

## Solution Space

**Shape:** single

### Recommendation

In `capability-probe.md`, at the single-source-of-truth note that already
states the constants "live in one source of truth inside the extension module
that the build-time literal-read assertions also consume," add the
importability precondition: the pinned-constants block MUST be reachable by a
side-effect-free import — importing it to read the constants MUST NOT invoke
the `default function (pi: ExtensionAPI)` factory or perform any
`pi.register*` / `pi.on` host-binding or other import-time side effect — so
that both the step-2(a) surface-inventory literal-read assertion and the
`CAPABILITY_OBLIGATIONS.length === 7` assertion can consume the block at build
time without triggering registration. State it behaviourally (side-effect-free
static reachability) rather than prescribing a file layout, so an
implementation may satisfy it by top-level constants in the extension module
(whose import does not execute the factory) or by a dedicated constants
module.

## Consequence

**Severity:** advisory

A working system is still producible because the established factory pattern
keeps all registration inside the factory body, so a top-level constants
import is side-effect-free by convention. But the guarantee is implicit: an
implementation that co-locates the block with later-added import-time
initialisation, or behind a module whose top level has side effects, breaks
both build-time assertions (or makes them register host bindings as a side
effect of the test) without any spec text being violated.
