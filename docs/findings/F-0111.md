## Problem

`V18a` (SDK capability inventory) carries two `PIC-15` Tests bullets, but its paired tests leaf `V18a-T` contains only the first (the seven-capability cardinality and factory-probed/verified-otherwise partition reconciliation against `V9a`'s `FACTORY_PROBABLE_CAPABILITIES`). The second bullet — asserting that the non-capability `SDK_SURFACE_INVENTORY` rows this leaf populates (`pi.registerFlag`, `pi.getFlag`, `pi-engines-node`, `peer-dep-range`, `strict-capability-probe`, `api-coverage`) resolve and each carries its kind under the leaf-owned entry-kind taxonomy — is absent from `V18a-T`, and `V18a`'s Ships-when gates only the cardinality/partition assertion. The non-capability-row resolution and entry-kind taxonomy this leaf owns therefore have no red test and no exit gate, breaking the per-phase TDD ritual in `conventions.md`. The only witnesses are indirect and later (`V18b`'s closure audit, `V18c`'s row reads).

That commit — itself a plan-fix resolving a different finding about `V18b` — added the owned assertion to the impl leaf only; it did not add a matching failing-test bullet to `V18a-T-capability-inventory.md` nor a Ships-when clause to `V18a`. The asymmetry between the impl leaf's Tests and its `-T` pair / exit gate dates from that single commit. ## Solution approach

Add a `PIC-15` Tests bullet to `docs/plan_topics/V18a-T-capability-inventory.md` mirroring `V18a`'s second Tests bullet, asserting that each populated non-capability `SDK_SURFACE_INVENTORY` row resolves and carries its kind under the leaf-owned entry-kind taxonomy. Add a clause to `V18a`'s Ships-when so the exit gate observes the same property alongside the existing cardinality/partition assertion.

## Solution constraints

- The new `-T` bullet and Ships-when clause stay under the existing `PIC-15` REQ-ID — do not mint a new REQ-ID, and do not restate or alter the existing `CAPABILITY_OBLIGATIONS.length === 7` cardinality / `FACTORY_PROBABLE_CAPABILITIES` partition assertion the first bullet owns.
