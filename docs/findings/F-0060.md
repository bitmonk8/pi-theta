## Finding

The implementation-task loop in `conventions.md` (§MVP and vertical features — tests-task → implementation-task pairing, the **Implement** step) bans speculative APIs: "do not add speculative APIs (unused exports, public hooks no test in this leaf or its declared downstream consumers exercises)." The phrase "declared downstream consumers" is never defined anywhere in the plan corpus — it occurs exactly once, at this site — and the leaf format defines no field by that name.

Two readings are available and they select different leaf sets: (a) leaves that declare this leaf in their `Deps.` (reverse-dependency edges), (b) leaves this leaf imports or that import its module at runtime, or (c) the leaves named as consumers in the leaf's `Adds.` Class-2 seam prose. The plan already has a precise, mechanically-checkable relation — a leaf's reverse-`Deps.` set — but the prose does not commit to it. Because this clause is a cross-cutting rule every implementation task applies during its self-review, the ambiguity propagates to every leaf rather than to one.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** f9c5354 (2026-06-09) "pi-loom plan: add scope section + two-task TDD pairing convention"
**History:** `git log -L 30,30:docs/plan_topics/conventions.md` shows the speculative-API parenthetical entered with the two-task TDD pairing convention in f9c5354; the line predating it was unrelated ("A leaf is not complete until its exit gate is met"). The phrase "declared downstream consumers" has carried its undefined meaning since that clause was first authored — it has never been defined in any later edit (`git log -S "declared downstream consumers"` returns only the later `docs/` move). The defect is original to the clause, not introduced by a subsequent edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, in the Implementation-task **Implement** step, replace the speculative-API parenthetical so the consumer set is named in terms of the existing `Deps.` relation rather than the undefined "declared downstream consumers." Strike:

> unused exports, public hooks no test in this leaf or its declared downstream consumers exercises

and insert text equivalent to:

> unused exports, or public hooks that no test in this leaf — and no test in any leaf that lists this leaf in its `Deps.` — exercises

This pins the consumer set to the reverse-`Deps.` relation already defined in §Leaf format (`Deps.` = "Other leaf IDs that must be complete first"), which is the only consumer relation the plan declares. No new term is introduced and no other file needs to change.

## Consequence

**Severity:** advisory

Two implementers applying the speculative-API ban to the same leaf will scope "downstream consumers" differently — one allowing an export consumed only by a reverse-`Deps.` leaf, another flagging it — so the ban is applied inconsistently across the corpus. The resulting leaf still works either way; the cost is divergent self-review judgements, not a broken or spec-divergent artefact.
