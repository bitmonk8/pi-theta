## Finding

The dep-driven build order is governed by the **Pickability predicate** in `conventions.md` §Leaf format (**Deps**): a `Deps.` entry is satisfied — and a leaf becomes pickable — exactly when the `<id>-complete` / `<id>-T-complete` git tag exists and is visible to the evaluating clone. The predicate pins tag *visibility* to a remote: "completion tags are pushed to **the shared remote** at the §Per-phase TDD ritual phase exit gate … and pickability is evaluated against a clone that has fetched tags." The two §Per-phase TDD ritual phase-exit gates likewise instruct contributors to "push the tag to **the shared remote**."

All three uses are definite-article singletons: they presuppose exactly one canonical remote that every contributor pushes completion tags to and fetches tags from, yet the convention never names that remote (e.g. `origin`) nor states the rule that designates it. The soundness argument the predicate makes — tags are absent only when genuinely not yet produced, modulo the disclosed unpushed-tag / shallow-clone caveats — holds only if "the shared remote" resolves to a single agreed target. Where a clone has multiple remotes configured, or work happens across forks with per-fork tag namespaces, one evaluator can read tags from a different remote than another pushed to, and the predicate silently misjudges pickability.

## Consequence

**Severity:** advisory

The pickability predicate is the authoritative dep-driven build signal, but "the shared remote" it evaluates against is undefined. In the single-repo build model an evaluator falls back to whatever remote is configured (conventionally `origin`) and the ambiguity rarely bites; under any multi-remote or per-fork workflow, a completion tag pushed to one remote may be unobservable to a clone reading another, so a leaf is judged pickable before its dependency actually landed (or stays blocked after it did) with no diagnostic. The defect is a guidance gap, not a build break: an implementer can still produce working leaves, but the predicate's stated soundness is not guaranteed.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, define "the shared remote" once — at its first occurrence in the §Per-phase TDD ritual, or co-located with the tag-visibility precondition in the §Leaf format Deps pickability predicate — so the three existing references resolve to a single named target. State the canonical remote the build pushes completion tags to and fetches them from (the single repository's default push remote, conventionally `origin`, in the single-repo build model this plan assumes), and that pickability is evaluated against a clone tracking that remote. This completes the tag-visibility precondition already present in the same paragraph: it currently establishes *that* tags must be pushed/fetched but leaves *where* unbound.

If a single-repo / single-orchestrator build is the intended model (consistent with the rest of the plan, which describes one clone advancing leaves in dep order), it is sufficient to state that explicitly and name the one remote; no multi-fork tag-namespace handling need be specified. Keep the existing unpushed-tag / shallow-clone caveats — they remain correct and now attach to a named remote.
