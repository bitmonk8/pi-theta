## Finding

`V11b` couples two concerns that fail red for different reasons, exercise different fixtures, and can ship and be tested independently:

- A **parse-time frontmatter field contract** — the `bind_context` (`none`|`session`) field and its `loom/parse/bind-context-session-on-subagent` rejection on subagent-mode looms. This is a frontmatter-parse concern, verified with frontmatter fixtures, with no runtime-rendering dependency.
- A **runtime transcript-rendering surface** — the compact-transcript renderer (the nine byte-exact `BNDR-7a–7i` renderings plus `BNDR-8` array/key-ordering) and the `BNDR-9` `custom-type-unsafe` abort. `BNDR-9` is not an independent concern from the renderer: the forbidden characters (`\n`, `\r`, `]`, and the `: ` two-byte sequence) are exactly the transcript-format delimiters, so the abort is a render-time safety guard over the same rendering pass that `BNDR-7/8` define. The renderer and its safety guard belong together; what is genuinely separable is the parse-time field contract.

Holding the frontmatter-parse rule and the byte-exact runtime renderer in one TDD pass means a red on a frontmatter fixture and a red on a render vector implicate unrelated mechanisms in the same leaf, enlarging the verification surface and the per-leaf review diff. (A prior plan refactor already extracted the runtime session-context truncation walk into `V11i`; this remaining coupling was left in place.)

## Consequence

**Severity:** advisory

If `V11b` ships as one leaf, an implementer must verify a parse-time frontmatter field rule and a byte-exact runtime transcript renderer (plus its `custom-type-unsafe` render-time abort) in a single TDD pass, with the two concerns' red fixtures implicating unrelated mechanisms. The leaf is still implementable, so this degrades leaf granularity and inflates the review surface rather than blocking pickup or causing a spec mismatch.

**History:** `V11b`'s first committed form (c6a664e) bundled the `bind_context` field, the truncation walk, the transcript renderer, and the `BNDR-9` custom-type-unsafe path in one leaf. A later refactor (d0c127f, "resolve 'V11b bundles transcript renderer with truncation walk'") extracted the runtime truncation walk into `V11i` but left the parse-time `bind_context` field contract bundled with the runtime transcript-rendering surface. Subsequent commits (ef76896, fe694dd, 3763100, 31c8cf2, f41af5d) refined the `BNDR-9` wording, the diagnostic-code citation, and the truncation-boundary test, but none touched this coupling. The defect has been present since the plan corpus was first built, narrowed once but never fully separated.

## Solution Space

**Shape:** single

### Recommendation

Extract the parse-time `bind_context` field contract into a new leaf (`<new>` + its `-T` partner): the `bind_context` (`none`|`session`) field and the `loom/parse/bind-context-session-on-subagent` rejection on subagent-mode looms, with the matching `bind_context: session` subagent-rejection parse test. Retain `V11b` for the runtime transcript-rendering surface — the `BNDR-7a–7i` renderer, the `BNDR-8` ordering, and the `BNDR-9` `custom-type-unsafe` render-time abort (the unsafe-`customType` guard stays with the renderer it protects, since its forbidden characters are the transcript delimiters). Retitle `V11b` to drop "Bind context" once the field moves out.

Keeping the renderer on `V11b` holds the `BNDR-7, BNDR-8, BNDR-9 | V11b` coverage-matrix row, the `V11i.Deps`-on-`V11b` edge, and the `H7a.Deps`-on-`V11b` edge unchanged. Give the new field-contract leaf the Deps its parse-time rule needs (it consumes the frontmatter contract, not the renderer), link it into the §Vertical slices V11 listing in `docs/plan.md`, and — if the `bind-context-session-on-subagent` parse rule is itself coverage-matrix-mapped — point that row at the new leaf rather than `V11b`.
