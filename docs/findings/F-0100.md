## Finding

The V15 leaves render the same `InvokeInfraError` struct in two different notations. The `INV-1` bullet of `V15a-invocation-core.md` (and its `-T` twin) writes `InvokeInfraError{load_failure}`, and the `INV-4` bullet of `V15b-invoke-depth-cycle.md` (and its `-T` twin) writes `InvokeInfraError{panic}` — a brace-shorthand that places the discriminator value directly inside the braces with no field name. The same leaves elsewhere use the spec's explicit field form: `Err(InvokeInfraError { cause: "validation" })` and `{ cause: "return_validation" }` (V15a Ships-when and spec-cite bullets).

The spec uses the explicit field form uniformly — every occurrence in `errors-and-results/error-model.md`, `invocation.md`, `queryerror-variants.md`, `ceilings-3-and-4.md`, and the glossary writes `InvokeInfraError { cause: "<value>" }`. Critically, the spec partitions `InvokeInfraError` into a wire `kind` discriminant (`"invoke_infra"`) and a `cause` sub-discriminator field (`"load_failure"`, `"panic"`, `"validation"`, `"return_validation"`, `"internal_error"`, `"subagent_model_unresolved"`) — see `queryerror-variants.md` and `glossary.md` §`cause`. The brace-shorthand `{panic}` is ambiguous against that partition: a reader cannot tell whether `panic` is a `cause` value, the `kind`, or some other discriminant, because the spec's own `kind`/`cause` distinction is exactly the thing the shorthand collapses.

The same shorthand also appears in `V14a-tool-calls.md` / `V14a-T-tool-calls.md` as `InvokeInfraError{validation}`, so the inconsistency is corpus-wide, not confined to V15.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the four shorthand occurrences to the spec's explicit field form so each leaf names the `cause` field:

- `docs/plan_topics/V15a-invocation-core.md`, `INV-1` bullet: replace `InvokeInfraError{load_failure}` with `InvokeInfraError { cause: "load_failure" }`.
- `docs/plan_topics/V15a-T-invocation-core.md`, `INV-1` bullet: same replacement.
- `docs/plan_topics/V15b-invoke-depth-cycle.md`, `INV-4` bullet: replace `InvokeInfraError{panic}` with `InvokeInfraError { cause: "panic" }`.
- `docs/plan_topics/V15b-T-invoke-depth-cycle.md`, `INV-4` bullet: same replacement.

Sweep the same notation in `docs/plan_topics/V14a-tool-calls.md` and `docs/plan_topics/V14a-T-tool-calls.md`, replacing `InvokeInfraError{validation}` with `InvokeInfraError { cause: "validation" }`, so the corpus uses one notation throughout. The spec (`queryerror-variants.md`, `glossary.md`) is the read-only authority for the target form and the `kind` vs `cause` distinction; do not edit it. The `cause` value tokens (`load_failure`, `panic`, `validation`) are unchanged — only the surrounding `{ cause: "…" }` framing is added.

## Consequence

**Severity:** cosmetic

A reader of the `{panic}` / `{load_failure}` shorthand cannot tell from the notation alone whether the bracketed token is a `cause` value, the wire `kind`, or another discriminant — precisely the `kind`/`cause` partition the spec keeps distinct. The leaf bodies and cited spec disambiguate, so no implementer would build the wrong behaviour; the cost is reader friction and notation drift within the corpus.
