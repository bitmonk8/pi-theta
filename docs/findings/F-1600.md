## Finding

The Pi version-bump procedure is paginated across `version-bump-step2.md` (page title `# Version bump step2`) and `version-bump-step2b.md` (page title `# Version bump step2b`). Those two suffix-letter names are pagination artefacts, but they sit on top of a pre-existing, unrelated designation namespace: the procedure's prose names two co-numbered surface-inventory assertions **step 2(a)** (positive direction, literal-read) and **step 2(b)** (negative direction, inventory-closure audit), referenced over a hundred times across the corpus as the link text `step 2(a)` / `step 2(b)`.

The two namespaces do not line up with the files:

- Both the **step 2(a)** and the **step 2(b)** assertions are *defined* inside `version-bump-step2.md` — their anchors `bump-step-2-positive` and `bump-step-2-negative` are both declared on a single line of that file. `version-bump-step2.md` additionally hosts the entire editorial-review checklist `(a)`–`(ap)`.
- `version-bump-step2b.md` does **not** hold "step 2(b)"; it holds the five *resolution branches* of step 2(b)'s inventory-closure audit (anchors `bump-step-2b-delete` … `bump-step-2b-rewrite-shape`).

So a reader who follows a `step 2(a)` cross-reference lands on a file whose name (`step2`) omits the `(a)` designation, while the sibling file's name (`step2b`) reads as "step 2(b)" yet contains only step 2(b)'s branch list, not step 2(b)'s definition. A TOC entry reading "Version bump step2b" gives no clue what the page contains and invites the false inference that step 2(b) is defined there. All cross-references still resolve — the anchors are disjoint and correct — so no link is broken; the cost is navigation friction at the corpus's most heavily cross-referenced procedure.

(Distinct from the in-prose label overload between the editorial-review checklist's `(a)`/`(b)` rows and the step-2 sub-step's `(a)`/`(b)` arms, which is a separate same-page collision. This finding is about the *file/page-title* pagination scheme, not the within-page prose tokens.)

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `f5e89f4` (2026-06-04) — "docs: enforce 100KB hard / 30KB recommended size cap on spec set"
**History:** The `step 2(a)` / `step 2(b)` textual designations predate the split — `git show f5e89f4^:docs/spec.md` and the pre-split `pi-integration-contract.md` already carry the `step 2(a)` form, so the designation namespace was not introduced by this commit. `f5e89f4` is the commit that, to satisfy the size cap, split the consolidated version-bump prose out of `pi-integration-contract.md` into the child files `version-bump-intro.md`, `version-bump-step2.md`, `version-bump-step2b.md`, and `version-bump-triggers.md` (`git log --diff-filter=A --follow` dates the creation of both `version-bump-step2.md` and `version-bump-step2b.md` to `f5e89f4`, and `git show f5e89f4:…/version-bump-step2b.md` shows the `# Version bump step2b` title already present at creation). The defect is the choice, at that split, of a letter-suffix overflow naming scheme (`step2` + `step2b`) that collides with the older `(a)`/`(b)` designation namespace; it is attributable to that single commit rather than to a multi-commit interaction.

## Solution Space

**Shape:** multiple

### Option A — Descriptive, content-based page titles and filenames

- **Approach:** Drop the `step2`/`step2b` suffix scheme entirely. Retitle and rename both pages after their content — e.g. `version-bump-step2.md` → a name conveying "surface-inventory assertions + editorial-review checklist", and `version-bump-step2b.md` → a name conveying "inventory-closure resolution branches". Keep the in-body `bump-step-2-*` / `bump-step-2b-*` anchors stable so existing deep links survive; update the TOC, `split_spec.py` basename tuples, and the `./version-bump-step2*.md` link targets.
- **Spec edits:** Rewrite both `# ` page titles; update `pi-integration-contract.md` TOC entries; update the `split_spec.py` tuples; repoint every inbound `./version-bump-step2.md` / `./version-bump-step2b.md` file link; repoint `V18c` / `V18c-T` **Spec** links.
- **Pros:** Removes the collision at the root; titles become self-describing; aligns with the corpus pattern of preferring descriptive titles over arbitrary pagination markers.
- **Cons:** Broad cross-reference and tooling churn for a NIT; touches plan-leaf Spec fields.
- **Risks:** A missed inbound link target breaks navigation; the rename interacts with any concurrent relabel of the `2(a)`/`2(b)` designations and should be sequenced to avoid double-churn.

### Option B — In-place heading clarifier (keep filenames)

- **Approach:** Leave the filenames as-is and rewrite each page's `# ` title (and add a one-line lead) to state exactly what the page holds — e.g. `# Version bump — step 2 (surface-inventory assertions 2(a)/2(b)) and the editorial-review checklist` for `version-bump-step2.md`, and `# Version bump — step 2(b) inventory-closure resolution branches` for `version-bump-step2b.md`.
- **Spec edits:** Two `# ` title rewrites plus the matching TOC link-text update in `pi-integration-contract.md`. No file rename, no inbound-link or `split_spec.py` change, no plan-leaf change.
- **Pros:** Lowest churn; removes the "what does `step2b` contain?" ambiguity; touches no plan leaf and no tooling; does not pre-commit a filename scheme that an adjacent relabel could moot.
- **Cons:** The opaque `step2`/`step2b` *filenames* remain (visible only to authors browsing the tree, not to rendered-doc readers).
- **Risks:** None beyond keeping the new titles in sync with the page content on future edits.

### Recommendation

Take **Option B**. The defect is cosmetic navigation friction; an in-place title/lead clarifier removes the ambiguity at minimal churn, touches no plan leaf or tooling, and avoids a corpus-wide rename whose value is diminished if the `2(a)`/`2(b)` designation tokens are separately relabelled. Edge case the implementer must watch: the clarified titles must name what each page actually holds — `version-bump-step2.md` defines *both* step 2(a) and step 2(b) plus the `(a)`–`(ap)` checklist, and `version-bump-step2b.md` holds only step 2(b)'s *resolution branches* — so a naive "step2 = step 2(a), step2b = step 2(b)" retitle would be wrong.

## Consequence

**Severity:** cosmetic

Every reading of the version-bump procedure must rely on surrounding context to reconcile the `step2`/`step2b` filenames against the `step 2(a)`/`step 2(b)` designations, and a TOC reader cannot tell what `step2b` holds without opening it. No implementer is misrouted — anchors resolve correctly and no obligation changes — so the impact is editorial navigation friction at a high-traffic procedure, not behavioural divergence.
