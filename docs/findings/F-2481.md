## Finding

`diagnostics/diagnostic-shape.md` carries a normative ordering rule: *"The renderer MUST be registered synchronously inside the extension factory **before** the first discovery scan kicks off, so the first batch of scan diagnostics renders through the loom-specific renderer rather than as raw fallback text."* Three problems are bundled into this one sentence.

First, the rationale asserts a concrete Pi host behaviour — that an unregistered `customType` `CustomMessage` renders as *"raw fallback text"* — with no citation or verification. Every other Pi host behaviour the corpus relies on is pinned at the loom 1.0 Pi-SDK pin with known-fragile evidence and a version-bump-checklist audit item (e.g. the renderer-resolution behaviour is item (q); `pi.registerTool` post-startup acceptance is item (z)). This assertion has neither. The PIC renderer-registration section only pins what happens when *registration* fails (`pi.registerMessageRenderer` throwing) or when two extensions collide on a `customType` (first-loaded wins via `getMessageRenderer` first-hit iteration); it never establishes what Pi does for a `customType` with *no* registered renderer.

Second, the ordering MUST is mis-located. Its natural owner is the PIC renderer-registration block (`pi-integration-contract/extension-bootstrap-and-per-loom.md` §"Renderer registration" → *Registration timing*), which already pins the weaker rule that the factory MUST call `pi.registerMessageRenderer` *"synchronously inside the factory body, before the factory returns."* That weaker rule does not imply the stronger one: the discovery scan runs **at factory time** (`registration-steps.md` step 1 walks all five sources synchronously inside the factory), so an implementer can register the renderer after the scan walk yet still before the factory returns, satisfying PIC while violating the diagnostics-shard ordering. A PIC reader reasoning solely from the renderer-registration timing rule never sees the before-the-scan constraint.

Third, the MUST has no observable acceptance criterion. "Raw fallback text" is undefined — there is no spec-pinned observable distinguishing a loom-rendered note from an unrendered one, and no test anchor for the ordering itself.

## Solution Space

**Shape:** single

### Recommendation

Consolidate the rule at its PIC owner and substantiate it. In `pi-integration-contract/extension-bootstrap-and-per-loom.md` §"Renderer registration" → *Registration timing*, strengthen the existing "synchronously … before the factory returns" bullet to additionally require registration **before the factory-time discovery scan walk runs** (the `registration-steps.md` step 1 walk), so the first scan's diagnostic batch is delivered into a registered renderer. Replace the undefined "raw fallback text" phrasing with the observable it stands for — first-scan diagnostics that reach the transcript without passing through the `loom-system-note` renderer.

Substantiate the unregistered-`customType` behaviour the same way every other unpinned Pi behaviour is handled: cite it at the loom 1.0 Pi-SDK pin as known-fragile evidence and add a version-bump editorial-review checklist item (alongside the existing items (q)/(z)) so a Pi minor that changes unregistered-`customType` handling is caught. If the behaviour cannot be verified, reframe the rationale as a presupposition rather than a fact.

Pin the observable acceptance criterion at the PIC site: a test that spies `pi.registerMessageRenderer("loom-system-note", …)` and asserts it is called before any `pi.sendMessage` carrying that `customType` (i.e. before the first scan's diagnostic batch). Leave a cross-reference in `diagnostic-shape.md` pointing at the relocated rule rather than restating it.

Edge cases the implementer must watch: the `ctx.reload()` path re-invokes the factory against a Pi state with no prior `loom-system-note` registration (per the *Teardown* bullet), so the before-the-scan ordering must hold on every factory invocation, not just first load; and the rule is moot on the degraded `pi.registerMessageRenderer`-failed branch, where system notes already fall back to `ctx.ui.notify`.

## Consequence

**Severity:** correctness

Because the discovery scan runs synchronously inside the factory, two implementers reading only the PIC "before the factory returns" rule can legitimately order renderer registration after the scan walk; the first batch of scan diagnostics then bypasses the loom renderer and surfaces in whatever raw form Pi uses for an unregistered `customType` — a visible divergence in first-load diagnostic rendering. The uncited Pi-behaviour claim also has no audit hook, so a Pi minor that changes unregistered-`customType` handling would silently break the rationale.
