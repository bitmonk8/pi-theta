## Finding

The *Lookup-failure-to-discriminator routing* clause on `docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` defines four exhaustive arms for the snapshot-side failure space. Arm (1) names *"`SDK_SURFACE_INVENTORY` itself resolves to `undefined` (a circular-init or live-binding gap in which the source-of-truth pinned-constants block has not finished initialising by handler-entry time — the dominant loom 1.0 failure mode for the snapshot read)"* and mandates `"missing-entry"` as its discriminator, justifying this by also mandating that *"the handler MUST guard the lookup with an explicit `SDK_SURFACE_INVENTORY === undefined` check before invoking the iteration primitive"*.

The runtime is pinned ES-modules-only: `extension-bootstrap-and-per-loom.md` (`#extension-esm-type-module`) requires `package.json#type === "module"`, and `audit-recognised-shapes.md` plus the family-(4) CJS-reach prohibitions forbid any `createRequire`/`require()` reach. Under that module model the "circular-init or live-binding gap" scenario arm (1) describes does not produce `SDK_SURFACE_INVENTORY === undefined`. ES-module bindings are live and TDZ-protected: a read of a `const` (or `let`) export performed before its initialiser has run throws `ReferenceError: Cannot access 'SDK_SURFACE_INVENTORY' before initialization`, not the falsy value the guard tests for. A `var` export *would* read as `undefined` in TDZ, but a top-level `var SDK_SURFACE_INVENTORY = [...]` is not the spec-pinned shape (the cross-cited `V18a` plan leaf and `inventory-audit-intro.md` describe a `const`-shaped pinned-constants block, and a `var` rebinding would also defeat the source-of-truth invariant the audit relies on). Verified empirically: a three-module Node ≥22 ESM cycle in which an importer reaches the still-uninitialised constant through a transitive callback produces `ReferenceError: Cannot access 'SDK_SURFACE_INVENTORY' before initialization` at the guard site itself — never `=== undefined`.

The consequence is that the `=== undefined` guard the spec mandates is dead code under the runtime's own module model, and the very routing the spec rejects ("a naive `SDK_SURFACE_INVENTORY.find(…)?.literals` pattern routes this case to the throw arm instead") is the only thing that actually happens: the TDZ-`ReferenceError` lands in arm (3) `"throw:<String(error)>"`, populating `details.failure` with a `"throw:ReferenceError: Cannot access 'SDK_SURFACE_INVENTORY' before initialization"` payload — the mis-routing the guard exists to prevent. Arm (1) as written is reachable only for an off-spec module shape (a `var` export, a CommonJS-reach import that pre-binds the namespace to `undefined`, or an explicit `let X; export { X }` with deferred assignment) — none of which the pinned ESM-only model permits. The contract obligation, the four-arm exhaustiveness claim, and `V9h`'s fixture floor for the `"missing-entry"` discriminator are all built on a premise the runtime cannot exhibit.

## Issue introduction

**Verdict:** indeterminate
**Introducing commits:** none identified with confidence
**History:** The *Lookup-failure-to-discriminator routing* clause and its `SDK_SURFACE_INVENTORY === undefined` guard mandate were authored in `75a9626` (2026-05-09, *"pi-loom spec: resolve 'SessionShutdownEvent.reason closed set has no build-time pin against the SDK type'"*) into the then-monolithic `docs/spec_topics/pi-integration-contract.md`. Before that commit, the **Unknown-reason rule** paragraph existed but only covered the closed-set membership check and the `event.reason` read; the snapshot lookup, its four-arm discriminator routing, and the `=== undefined` guard did not exist. `75a9626` introduced both the four-arm enumeration and the guard sentence in the same diff hunk, so the defect (the mismatch between arm (1)'s premise and the runtime's pinned ESM module model) is present-since-inception of the routing clause itself rather than introduced by a later edit.

The defect is also `multi-commit-interaction`-shaped in a weaker sense: the ESM-only pin (`extension-esm-type-module`) was relocated and refined in commits prior to and after `75a9626` (e.g. `ce48e09` 2026-06-22 — *"fix F-0961 — relocate ESM type:module obligation to host-prerequisites"*), so the routing clause was authored against an ESM-only premise that has only hardened since. No single commit "introduces" the contradiction — `75a9626` introduces a routing whose arm (1) premise was already incompatible with the ESM pin at authoring time, and the ESM pin has only become more load-bearing since. Subsequent edits to `unknown-reason-rule.md` after the `f5e89f4` (2026-06-04) split (`a7d2a80`, `c326bfa`, `aa2f9d7`, `a37d4ef`, `a9fb344`, `8008b91`) refined the discriminator grammar, the `pinnedConstantReadOk` seam, sub-anchors, and the `"literals-shape-invalid"` four-sub-case fixture obligation, but none of them edited arm (1)'s premise text or the `=== undefined` guard mandate. Verdict left `indeterminate` because the "introducing commit" is the same commit that introduced the routing arm; there is no prior state in which the arm existed without the defect, so the defect is co-extensive with the clause rather than a regression.

## Solution Space

**Shape:** multiple

### Option A — Recategorise TDZ-`ReferenceError` into arm (1) `"missing-entry"`

**Approach.** Keep arm (1)'s `"missing-entry"` discriminator and its operator-deduplication-bucket semantics, but redefine its trigger from *"`SDK_SURFACE_INVENTORY` itself resolves to `undefined`"* to *"the bare-identifier read of `SDK_SURFACE_INVENTORY` either resolves to `undefined` (the off-spec `var`-export shape, retained as a defensive arm) or throws a `ReferenceError` whose stack frame is the bare-identifier read at handler-entry (the ESM TDZ / circular-init / live-binding-gap case)."* Drop the *"MUST guard the lookup with an explicit `SDK_SURFACE_INVENTORY === undefined` check"* sentence and replace it with: the handler MUST detect the TDZ case by wrapping the bare-identifier read in its own `try`/`catch` and routing a caught `ReferenceError` whose `.message` matches `/Cannot access .* before initialization/` (or whose name is `"ReferenceError"`, leaf-owned predicate choice) to the `"missing-entry"` discriminator rather than to the `"throw:<String(error)>"` arm.

**Spec edits.** Edit the *Lookup-failure-to-discriminator routing.* clause arm (1) per the redefinition above. Edit PIC-47 sub-anchor (iii) to reflect that the catch-arm discriminator distinguishes a snapshot-side TDZ `ReferenceError` from a generic snapshot-side throw. Edit the four-arm exhaustiveness sentence to make clear that arm (1) covers both `undefined` reads (for completeness) and the TDZ-`ReferenceError` case (the dominant ESM-only failure mode). Mandate a fixture under PIC-47 / `V9h-T` that constructs a three-module ESM cycle reaching the bare-identifier read pre-initialisation and asserts `details.failure === "missing-entry"` (no `"throw:"` prefix). Leave the discriminator-union TypeScript type and `diagnostics.md` row unchanged.

**Pros.** Preserves the operator-deduplication bucket the spec already pins for "the snapshot constant didn't materialise" — a `ReferenceError` from a TDZ access is operationally the same failure-class as a missing entry, so collapsing them into one bucket matches the operator's mental model. No change to the discriminator union, the `diagnostics.md` row, or the inbound substring-(d) contract. Keeps the four-arm enumeration coherent.

**Cons.** Requires a `.message`-substring or `.name`-equality predicate at the catch arm, which the spec elsewhere avoids (string-matching error messages is brittle across V8 versions, though the *"before initialization"* phrasing has been stable since Node 14). The leaf-owned predicate adds a footnote of fragility the rest of `unknown-reason-rule.md` keeps out of the discriminator path.

**Risks.** A future Node version that rephrases the TDZ-`ReferenceError` `.message` would silently demote the TDZ case back into arm (3) `"throw:<String(error)>"` without a build-time signal. Mitigation: assert the predicate from a dedicated fixture (the same one above) so a Node bump fails `V9h-T` red rather than silently regressing.

### Option B — Drop arm (1)'s `undefined`-read premise; route TDZ throws to arm (3) explicitly

**Approach.** Acknowledge the runtime's ESM-only pin and remove the `=== undefined` guard mandate entirely. Edit arm (1) to cover only the case the iteration primitive actually exposes — *"the iteration primitive returns no matching entry"* — and merge it with arm (2) (which already says the same thing). State explicitly that under the ESM-only pin (cross-link `#extension-esm-type-module`) the snapshot-binding-uninitialised case manifests as a `ReferenceError` from the bare-identifier read and routes to the `"throw:<String(error)>"` arm with a `details.failure` payload of the form `"throw:ReferenceError: Cannot access 'SDK_SURFACE_INVENTORY' before initialization"`. Add a non-normative rationale paragraph noting that the operator-deduplication bucket for that scenario is the throw-arm bucket keyed on the `startsWith("throw:")` predicate, not a separate `"missing-entry"` bucket — and that operators triaging `pinned-constant-unreadable` rows should group by `(code, startsWith(details.failure, "throw:ReferenceError"))` for the circular-init class.

**Spec edits.** Edit *Lookup-failure-to-discriminator routing.* to drop the four-arm split into three arms (`"missing-entry"` = lookup-returned-nothing only; `"throw:<String(error)>"`; `"literals-shape-invalid"`). Delete the `=== undefined` guard mandate sentence. Add the ESM-only rationale paragraph above. Update PIC-47 sub-anchor (iii) to drop the routing-of-`undefined`-binding obligation. Update PIC-48 substring (d) if and only if the three-literal vs four-literal set drift requires it — the literals themselves are unchanged, only the enumeration framing is. Drop the `V9h-T` fixture for arm (1) (the undefined-binding fixture has no production trigger and would have to mock a non-spec module shape). Leave the `diagnostics.md` row unchanged.

**Pros.** Aligns the spec with what the runtime can actually exhibit; eliminates a dead-code obligation and the corresponding mockable-only fixture. No error-message-string-matching predicate. Reduces the discriminator surface to what operators will actually see in production. The "guard against a naive `?.find(…)?.literals`" rationale survives as a non-normative implementation note (the naive pattern still produces the same `"throw:ReferenceError: …"` discriminator the explicit guard would, so the rationale collapses entirely).

**Cons.** Loses the operator-side observability distinction between "the entry was deleted" (genuine missing entry) and "the binding wasn't initialised yet" (TDZ); both end up keyed on `"throw:ReferenceError: …"` for the latter case, which buckets differently from the deleted-entry case. Operator triage for the circular-init scenario depends on a substring predicate over an error message, which is the leaf-owned-predicate fragility Option A also has — but here the fragility is implicit rather than spec-pinned.

**Risks.** A future loom build that introduces a non-ESM module-loader shape (e.g. a Vite-style HMR layer that exposes `SDK_SURFACE_INVENTORY` as an `undefined`-readable getter pre-init) would silently route into arm (3) instead of into a dedicated `"missing-entry"` bucket. The ESM-only pin makes this unlikely in loom 1.0; a future relax of that pin would need to revisit this clause.

### Recommendation

**Option A.** The spec already invested in the `"missing-entry"` bucket as a contract surface (anchor-stable substring (d) of PIC-48; a row in `diagnostics.md`; the *"the dominant loom 1.0 failure mode for the snapshot read"* framing on arm (1) itself), and operator triage genuinely benefits from collapsing "binding didn't materialise" — whether by being absent from the entry list or by being read pre-initialisation — into one dedup bucket. The cost of Option A's leaf-owned `.message`-substring predicate is bounded (one fixture exercises it; a Node bump that rephrases the message fails that fixture red) and is cheaper than Option B's loss of operator-observability for the dominant failure mode the spec itself named.

The implementer must watch the following edges: (i) the predicate that classifies a caught `ReferenceError` as TDZ-class versus generic must not match on the *identifier name* `SDK_SURFACE_INVENTORY` literally (a future rename of the constant would silently regress) — match on the `"Cannot access … before initialization"` phrase or on a `.name === "ReferenceError"` heuristic, and pin the choice as leaf-owned; (ii) the `try`/`catch` covering the bare-identifier read must run *before* the iteration primitive, because once the iteration primitive runs against the live binding the TDZ has already been resolved either way; (iii) the existing arm-(3) carve-out for *"a per-entry `.kind` or `.path` read itself throws during iteration"* must remain in `"throw:<String(error)>"` — those throws are not TDZ-class and must not collapse into `"missing-entry"`.

## Consequence

**Severity:** correctness

The four-arm routing the spec presents as exhaustive is incoherent under its own module pin: the "dominant loom 1.0 failure mode" the spec singles out for arm (1) cannot reach arm (1) at runtime, and the discriminator string operators dedup on for the circular-init / live-binding-gap scenario is `"throw:ReferenceError: …"` rather than `"missing-entry"`. Two conformant implementations that both honour the spec's own ESM pin will agree on the (wrong) discriminator, so operator triage is not split — but every reader of `unknown-reason-rule.md` is misled about what `details.failure === "missing-entry"` actually buckets, and the `V9h-T` fixture inventory the plan derives from the four arms encodes a state machine no production binding can drive into arm (1).
