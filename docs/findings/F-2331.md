## Problem

The `loom/load/model-unresolved` Tests bullet phrases the `provider/modelId` reject case with an object-less relative clause ŌĆö "a `provider/modelId` reference matching no available model whose `Model<Api>.provider` (not the api-shaped `.api`) and `Model<Api>.id` both match". The clause reads two ways: the intended no-match resolution outcome (no available model has the named fields equal to the reference tokens), or a structural precondition on the reference syntax itself. The malformed reading would lead an implementer to assert the wrong property for this vector. The identical bullet appears verbatim in both `V6a` and `V6a-T`.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** e084853 ŌĆö pi-loom spec-plan: fix F-1359 ŌĆö define Model<Api>.id/.provider matcher inputs + wire V11a/V6a/V9b (cross-boundary) (2026-06-24, Thomas Andersen)
**History:** Before e084853 the bullet's `provider/modelId` case read simply "a reference matching no available model, or a bare `modelId` ambiguous across providers". The F-1359 fix expanded it to name the `Model<Api>.provider` / `Model<Api>.id` matcher inputs and, in doing so, introduced the object-less "whose ŌĆ” both match" relative clause that produces the ambiguity. The same edit landed the identical phrasing in both `V6a` and `V6a-T`.

## Solution approach

Rewrite the `provider/modelId` reject case in the `loom/load/model-unresolved` Tests bullet so the relative clause names what is matched and describes a model that does *not* exist ŌĆö e.g. "a `provider/modelId` reference for which no available model has `Model<Api>.provider` (not the api-shaped `.api`) equal to the provider token and `Model<Api>.id` equal to the modelId token". Apply the identical edit in both `docs/plan_topics/V6a-frontmatter-contract.md` and `docs/plan_topics/V6a-T-frontmatter-contract.md` so the paired leaves stay verbatim-aligned.
