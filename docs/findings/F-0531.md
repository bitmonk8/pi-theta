## Finding

The `ToolDefinition.label` derivation pinned in `extension-bootstrap-and-per-loom.md` §Per-loom registration says the label is "the loom file's basename with hyphens preserved and the leading character capitalised: `summarise.loom` → `"Summarise"`, `code-review.loom` → `"Code-review"`." This fixes the casing of exactly one character (the leading one) and is silent on every other character. Two reasonable implementations satisfy the rule as written:

- **Preserve interior casing** — uppercase the first character and leave the remaining characters byte-for-byte: `s[0].toUpperCase() + s.slice(1)`.
- **Sentence-case the stem** — uppercase the first character and lowercase the remainder: `s[0].toUpperCase() + s.slice(1).toLowerCase()`.

Every worked example in the spec and in the V9f / V9f-T leaves uses an all-lowercase basename (`summarise`, `code-review`), so both implementations produce identical output for them; the example set cannot witness the divergence. The divergence is reachable, however: a `.loom` file referenced by another loom's `tools:` path (or by `invoke("./…")`) is **not** a discovered slash command and is therefore not subject to the discovery `^[a-z0-9][a-z0-9_-]*$` lowercase-stem rule (`discovery/discovery-sources.md` §Filename validity, `loom/load/invalid-slash-name`). Such a callee still materialises a `ToolDefinition` whose `label` is derived from its basename. Its derived tool name only has to satisfy loom's lowercase-*first* identifier rule (`lexical.md` §Identifiers — internal casing is explicitly free, `lowerCamelCase` accepted), so `./codeReview.loom` (tool name `codeReview`) and `./code-Review.loom` (tool name `code_Review`) are both legal callables with interior uppercase in the basename. For `codeReview.loom` the two readings yield `"CodeReview"` vs `"Codereview"` — observably different labels.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `cb43deb` (2026-05-04, "pi-loom spec: resolve \"ToolDefinition is mis-shaped: missing label, and parameters must be a TypeBox TSchema\"")
**History:** `git log -S "leading character capitalised"` / `git log -L` on the `label` bullet show the derivation rule entered in `cb43deb`, which introduced the "leading character capitalised" phrasing alongside an example (`code-review.loom` → `"Code-Review"`). That phrasing never specified the casing of non-leading characters, so the preserve-vs-lowercase ambiguity has existed since the rule's inception. A later commit `6d8b472` (2026-06-05, "label derivation rule and example capitalisation disagreement") touched the same sentence but resolved a *different* facet — the per-segment-vs-leading-only contradiction between the rule text and its `"Code-Review"` example — by rewriting the example to `"Code-review"`. That edit narrowed the rule but left this interior-casing question open (and, by replacing the only mixed-case example with another all-lowercase one, removed the sole example that could have witnessed the remaining ambiguity).

## Solution Space

**Shape:** multiple

### Option A — Preserve interior casing

**Approach.** State that the derivation uppercases only the first character and leaves all remaining characters unchanged.

**Plan edits.** In `V9f`, `V9f-T`, and the `cka-17` row of `coverage-matrix.md`, restate the derivation as "uppercase the first character of the basename and leave every remaining character unchanged."

**Spec edits.** In `extension-bootstrap-and-per-loom.md` §Per-loom registration, replace the `label` bullet's casing clause with an explicit preserve-the-rest statement and add a mixed-case worked example, e.g. `codeReview.loom` → `"CodeReview"`.

**Pros.** Matches the literal reading of "*only* the leading character capitalised" (the capitalisation operation touches one character; the rest is untouched). Honours an author's deliberate interior casing. Simplest operation (`s[0].toUpperCase() + s.slice(1)`).

**Cons.** Produces non-sentence-case labels (`"CodeReview"`) that some readers may find visually inconsistent.

**Risks.** None beyond label aesthetics.

### Option B — Sentence-case the stem

**Approach.** State that the derivation uppercases the first character and lowercases all remaining characters.

**Plan edits.** Same three sites, restated as "uppercase the first character of the basename and lowercase every remaining character."

**Spec edits.** Same bullet, stated as sentence-case, with a mixed-case worked example, e.g. `codeReview.loom` → `"Codereview"`.

**Pros.** Yields a uniform sentence-case appearance regardless of authoring casing.

**Cons.** Silently discards an author's deliberate interior casing; contradicts the natural reading of the existing "*only* the leading character" phrasing.

**Risks.** None beyond label aesthetics.

### Recommendation

Adopt **Option A**. The existing phrasing "only the leading character capitalised" already signals that the capitalisation operation is confined to the first character, so preserving the remainder is the lower-surprise reconciliation and keeps the spec edit additive rather than redefining the verb. Update `extension-bootstrap-and-per-loom.md`'s `label` bullet to state the preserve-the-rest rule explicitly and carry a mixed-case worked example (`codeReview.loom` → `"CodeReview"`), then mirror that example into the `V9f` / `V9f-T` Tests and Ships-when restatements and the `cka-17` coverage-matrix row so the disambiguation is observable in the leaf gate. Edge case the implementer must watch: a basename whose first character is a digit (`2fa.loom`) has no uppercase form — the derivation leaves it unchanged (`"2fa"`), independent of which option is chosen.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the `label` value for any callee whose basename carries interior uppercase (e.g. `codeReview.loom` → `"CodeReview"` vs `"Codereview"`), and the V9f / V9f-T acceptance check cannot distinguish them because every fixture uses an all-lowercase basename — so the closing gate passes vacuously for the ambiguous case. The blast radius is narrow (the label is a display string on the materialised `ToolDefinition`), but the spec admits two distinct observable outputs for the same input.
