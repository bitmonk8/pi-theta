## Finding

PIC-22 (*Parallel subagent spawn initiation*) on `subagent.md` opens with a tight behavioural requirement — when the model emits N ≥ 2 subagent-mode `.loom` callables as parallel tool calls in one assistant turn, the runtime MUST initiate `createAgentSession` for all N before any one returns — followed by its conformance witness and the upstream-serialisation caveat. From "Beyond the absence of an admission cap and scheduler pinned above…" onward, however, the same paragraph carries a multi-sentence architecture-layer NFR: that the loom adds no per-class threshold and no cross-sibling aggregation/throttling/storm-detection, and that the limits on runtime-value heap, OS file-descriptors/sockets/child-process slots, and provider rate-limit/quota are owned by the host JavaScript runtime, the host OS, and the LLM provider respectively (with the per-call surfacing routing for each class enumerated).

This resource-ownership boundary is a system-boundaries NFR, not part of the spawn-initiation behaviour PIC-22 names. Bundling it into the behavioural MUST forces an operator reasoning about resource limits to read it inside a parallel-spawn conformance rule, where they are unlikely to look, while inflating the PIC-22 requirement body with content that does not constrain the spawn-initiation behaviour at all. The natural home is the adjacent concurrency-disposition material: the `concurrent-invocation-isolation` / `no-invocation-cap` block immediately above already owns the "no admission cap, no scheduler" disposition that this resource-ownership boundary directly continues.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `a389af0` (2026-05-13), `7a3b907` (2026-06-10)
**History:** The resource-ownership prose was authored at `a389af0` ("rewrite T20 as positive ownership-boundary statement"), which resolved the T20 finding about undisclaimed non-memory resource exhaustion. At that point it lived in the `no-invocation-cap` concurrency-disposition paragraph in the then-`implementation-notes.md`, attached to the no-cap/no-scheduler disposition — not yet a distinct anchored MUST. The embedding defect crystallised at `7a3b907` ("subagent parallel-initiation MUST has no closing leaf"), which carved the `<a id="pic-22">` anchor out of the combined concurrency paragraph: it split the spawn-initiation MUST into the standalone PIC-22 requirement but carried the resource-ownership sentences into the new PIC-22 body rather than leaving them on the `concurrent-invocation-isolation` / `no-invocation-cap` paragraph that owns the no-cap disposition. The subsequent `f5e89f4` 100KB-cap split only relocated the page; it did not introduce the bundling.

## Solution Space

**Shape:** single

### Recommendation

Extract the resource-ownership NFR (the run beginning "Beyond the absence of an admission cap and scheduler pinned above, the loom adds no per-class threshold…" through "…on the same footing as any other engine fatal.") out of the PIC-22 requirement body and relocate it to the `concurrent-invocation-isolation` / `no-invocation-cap` block immediately above, as a distinct "Resource ownership" NFR note continuing that block's no-cap/no-scheduler disposition. Leave PIC-22 purely behavioural: the spawn-initiation MUST, its conformance witness, the upstream Pi-serialisation caveat, the "does not promise resource unboundedness" pointer (repointed to the relocated note), and the closing "Widening or narrowing this rule is a major-version concern."

Preserve every cross-reference verbatim during the move (`NOCEIL-3` at `ceiling-invariants-and-audit.md#no-additional-ceilings`; `loom/runtime/internal-error` at `error-model.md#runtime-panics`; `TransportError` at `queryerror-variants.md#queryerror-variants`) and keep the `no-invocation-cap` anchor stable so existing inbound links (e.g. the GOV-12 aggregator's forward-link inventory, which references "No invocation cap — loom 1.0 concurrency disposition") continue to resolve to the disposition that now also carries the resource-ownership boundary. No diagnostic codes, predicates, or behavioural obligations change.

## Consequence

**Severity:** cosmetic

No observer's behaviour diverges; the resource-ownership content is correct and stays in the corpus. The defect is purely organisational: the system-boundaries NFR is buried inside a behavioural parallel-spawn MUST, degrading discoverability for an operator scanning concurrency/NFR material and bloating the PIC-22 requirement body.
