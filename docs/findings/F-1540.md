## Finding

CTRL-1's closing clause reads "reassigning a `let mut` from inside the body does not change the already-snapshotted sequence." The phrase `a let mut` is indefinite — the sentence introduces no antecedent — so two readings are open:

- **(A, intended.)** If the iterand expression read from a `let mut` binding, mutating that *same* binding inside the body does not retroactively re-evaluate the iterand or swap out the snapshotted array. This is a real, non-trivial property: it pins the once-only-evaluation guarantee against the case where the binding the iterand named is itself swung mid-loop.
- **(B, trivial.)** Reassigning *any* `let mut` in the loop body cannot change the snapshot. Vacuously true, contributes nothing, and would not warrant a normative clause.

CTRL-1 is the only place the snapshot-stability property is pinned for the iterand-binding case, so an implementer or test author who falls into reading (B) will neither implement nor cover the case (A) targets. The fix is a single-sentence rewording that anchors the `let mut` to the iterand binding explicitly.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the final clause of CTRL-1 to bind the `let mut` to the iterand:

> "…and if the iterand expression read from a `let mut` binding, reassigning that binding from inside the body does not change the already-snapshotted sequence."

Notes for the editor:

- Keep the rule co-located with the once-only commitment sentence; do not split CTRL-1.
- "Read from" is preferred over "was" because the iterand can be a sub-expression of a larger expression (`for x in xs.concat(ys)`), and the property holds whenever the iterand evaluation touched the binding.
- The accompanying `V3c`/`V3c-T` plan acceptance bullet ("a mid-body `let mut` reassignment does not alter the snapshot") should track the same wording so the conformance vector unambiguously targets the iterand-binding case.

## Consequence

**Severity:** advisory

A tester reading (B) writes a vacuous reassign-an-unrelated-`let mut` vector and never exercises the only behaviour CTRL-1's clause is there to pin; an implementer who treats the clause as vacuous can ship a loop that re-reads the iterand binding on every iteration without failing any CTRL-1 test. The surrounding sentence makes (A) recoverable on a careful read, so this is a clarity defect rather than a divergence-driver.
