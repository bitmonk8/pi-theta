## Finding

`audit-resolution.md` (*Exemption mechanism*) pins, for a `// allow-pi-surface:` marker placed on a family-(4) source line: *"MUST NOT suppress the family (4) discriminator and is itself an audit failure under that discriminator."* The antecedent of *"that discriminator"* is grammatically ambiguous — the only discriminator named in the preceding clause is family (4), but the conceptual referent (a marker-shaped audit failure) belongs to family (5). The two readings produce different wire records, and the `*Failure-surface contract*` declares the five families exhaustive, so each conforming implementation will pick one and silently diverge from the other on every fixture that seeds this case.

The downstream machinery does not resolve the ambiguity. The `*Malformed-marker discriminator*` in `audit-failures.md` enumerates clauses (a)–(g); none of them names *"marker placed on a non-exemptible family-(4) line"*, so a well-formed marker on a family-(4) line is not malformed under that discriminator's own definition. The `*Malformed-marker dual-emission co-commit obligation*` in `version-bump-step2b.md` only co-emits family (1) / (2) / (3) records alongside a family-(5) red, not family (4). The result is that the only sentence that names this case lives one paragraph away from both the family enumeration and the malformed-marker clause list it must coordinate with, and it lands in the gap between them.

A reader cannot determine which `<family>` token an implementation must mint, which `<symptom>` token attaches, whether a family-(5) record co-emits with the family-(4) shape record, or which fixture under the per-`<symptom>` floor exercises the case.

## Solution Space

**Shape:** single

### Recommendation

Route a `// allow-pi-surface:` marker placed on a non-exemptible family-(4) source line to family (5) as a malformed-marker audit failure, leaving the family-(4) shape discriminator to fire independently for the shape. The audit emits the family-(4) record for the shape and a family-(5) record for the marker under a new clause (h) `<symptom>` token; no family-(4) marker-shaped record exists.

Apply these spec edits:

- `audit-resolution.md` *Exemption mechanism* — replace *"and is itself an audit failure under that discriminator"* with *"and is itself a malformed exemption marker under clause (h) of the Malformed-marker discriminator below (family (5))"*. Add an explicit forward link to clause (h) so a reader of this section is routed to family (5) at the point the family-(4) sentence appears.
- `audit-failures.md` *Malformed-marker discriminator* — add clause (h): *"presence on a source line whose shape matches a family-(4) non-exemptible shape"*, byte-for-byte parallel to the existing (e) *off-line placement* clause. The (h) `<symptom>` token must be distinct from clauses (a)–(g) so it earns its own fixture under the per-`<symptom>` floor. Update the *Failure-surface contract*'s nine-token-floor wording to ten tokens (or whatever the resulting count is) and the *Note on family (5) routing* in the same file accordingly.
- `audit-failures.md` *Per-family record-shape table* family (5) row — extend the `proposed-resolution` sub-case list to enumerate the (h) sub-case: delete the marker and resolve the underlying shape via branch (5) [rewrite-shape](#bump-step-2b-rewrite-shape) (branch (5), not branch (4), is the underlying-shape recovery arm).
- `version-bump-step2b.md` branch (4) — extend the *Malformed-marker dual-emission co-commit obligation* to cover the family-(4) case: clause (h) co-emits family (4) (resolved through branch (5) [rewrite-shape](#bump-step-2b-rewrite-shape)) rather than (1)/(2)/(3); add a sentence pinning that contributor recovery is branch (5) for the shape and that marker-deletion is implicit in the rewrite.
- `V18b` / `V18b-T` leaf Test bullets — re-align to cite clause (h) explicitly. The current "surfaces the family-(5) marker record but does not suppress the family-(4) shape record" wording already matches the outcome on the wire (both records still emit); the leaf bullet must name clause (h) once it lands.

## Consequence

**Severity:** correctness

Two conforming audit implementations seeded with the same `// allow-pi-surface:` marker on a family-(4) line will emit different `audit/<class>/<family>/<symptom>` discriminators (one routes through family (4), the other through family (5), a third may dual-emit both); a CI parser pinned to one wire shape silently rejects the other. The per-`<symptom>` fixture floor cannot be satisfied unambiguously because the case has no agreed `<family>` token to mint a fixture against.
