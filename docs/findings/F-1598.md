## Finding

`audit-resolution.md` (*Exemption mechanism*) pins, for a `// allow-pi-surface:` marker placed on a family-(4) source line: *"MUST NOT suppress the family (4) discriminator and is itself an audit failure under that discriminator."* The antecedent of *"that discriminator"* is grammatically ambiguous — the only discriminator named in the preceding clause is family (4), but the conceptual referent (a marker-shaped audit failure) belongs to family (5). The two readings produce different wire records, and the `*Failure-surface contract*` declares the five families exhaustive, so each conforming implementation will pick one and silently diverge from the other on every fixture that seeds this case.

The downstream machinery does not resolve the ambiguity. The `*Malformed-marker discriminator*` in `audit-failures.md` enumerates clauses (a)–(g); none of them names *"marker placed on a non-exemptible family-(4) line"*, so a well-formed marker on a family-(4) line is not malformed under that discriminator's own definition. The `*Malformed-marker dual-emission co-commit obligation*` in `version-bump-step2b.md` only co-emits family (1) / (2) / (3) records alongside a family-(5) red, not family (4). The result is that the only sentence that names this case lives one paragraph away from both the family enumeration and the malformed-marker clause list it must coordinate with, and it lands in the gap between them.

A reader cannot determine which `<family>` token an implementation must mint, which `<symptom>` token attaches, whether a family-(5) record co-emits with the family-(4) shape record, or which fixture under the per-`<symptom>` floor exercises the case.

## Solution Space

**Shape:** multiple

### Option A — Add malformed-marker clause (h); route to family (5) only

**Approach.** Extend the *Malformed-marker discriminator* clause list with clause (h): *"presence on a source line whose shape matches a family-(4) non-exemptible shape"*. The audit fires only the family-(4) record for the shape and the family-(5) record for the marker under the (h) `<symptom>` token; no family-(4) marker-shaped record exists. Rewrite the *Exemption mechanism* sentence to: *"is itself an audit failure under the malformed-marker discriminator clause (h) below (family (5))"*.

**Spec edits.**

- `audit-resolution.md` *Exemption mechanism* — replace *"and is itself an audit failure under that discriminator"* with *"and is itself a malformed exemption marker under clause (h) of the Malformed-marker discriminator below (family (5))"*.
- `audit-failures.md` *Malformed-marker discriminator* — add clause (h) to the (a)–(g) enumeration; update the *Failure-surface contract*'s nine-token-floor wording to "ten tokens" (or whatever the resulting count is) and the *Note on family (5) routing* in the same file accordingly.
- `version-bump-step2b.md` branch (4) — extend the *Malformed-marker dual-emission co-commit obligation* to cover the family-(4) case: clause (h) co-emits family (4) (resolved through branch (5) [rewrite-shape](#bump-step-2b-rewrite-shape)) rather than (1)/(2)/(3); add a sentence pinning that contributor recovery is branch (5) for the shape and marker-deletion is implicit in the rewrite.
- `audit-failures.md` *Per-family record-shape table* family (5) row — extend the `proposed-resolution` sub-case list to enumerate the (h) sub-case (delete the marker and resolve the underlying shape via branch (5)).

**Pros.** Preserves the conceptual cleanliness that markers are family-(5) artefacts; clause (h) is byte-for-byte parallel to the existing (e) *off-line placement* clause; family (4) records remain shape-only.

**Cons.** Widens the malformed-marker clause set across two files; the (h) `<symptom>` token needs its own fixture; touches the family-(5) `<symptom>` token count downstream.

**Risks.** A contributor reading only `audit-resolution.md` still sees the family-(4) sentence first and may miss the routing to family (5) without the explicit forward link.

### Option B — Dual-emit family (4) for the shape and family (5) for the marker

**Approach.** Keep the family-(4) discriminator firing on the shape, and additionally fire a family-(5) record for the marker under a dedicated `<symptom>` token (e.g. `marker-on-non-exemptible-shape`). Generalise the *Malformed-marker dual-emission co-commit obligation* to route the underlying-shape co-emission through family (4) as a peer of family (1)/(2)/(3). The original suggested fix in the source report.

**Spec edits.**

- `audit-resolution.md` *Exemption mechanism* — rewrite the sentence to: *"MUST NOT suppress the family (4) shape discriminator, AND MUST additionally fire a family (5) malformed-marker discriminator under clause (h) below — both records emit on the same `(path, line)` key per the dual-emission co-commit obligation in version-bump-step2b.md branch (4)."*
- `audit-failures.md` *Malformed-marker discriminator* — add clause (h) (same definition as Option A) and clarify in the dual-emission paragraph that the underlying record is family (1) / (2) / (3) **or** family (4) per the shape on the originating line.
- `version-bump-step2b.md` branch (4) — extend the *Malformed-marker dual-emission co-commit obligation* to enumerate family (4) as a co-emission target; pin contributor recovery as branch (5) [rewrite-shape](#bump-step-2b-rewrite-shape) for the shape + marker-deletion (per branch (4) sub-case (v)) for the marker, both in the same commit.
- `audit-failures.md` *Per-family record-shape table* — extend family (5)'s `proposed-resolution` enumeration with the (h) sub-case; extend family (4)'s row with a note that the same `(path, line)` may pair with a family (5) clause-(h) record.

**Pros.** Symmetric with the existing family (1)/(2)/(3) dual-emission rule — every marker-on-violation case dual-emits regardless of which family the underlying violation belongs to; the contributor sees both records on the same `(path, line)` key, removing any guess about whether the marker was processed.

**Cons.** Adds a third co-emission target to a rule that was carefully scoped to (1)/(2)/(3); the family-(4) `symbol` value (the verbatim shape literal) and a family-(5) record share `(path, line)` but carry different `symbol` values, which a `(path, line)`-keyed pairing must tolerate; widens the per-`<symptom>` fixture floor by one token on family (5).

**Risks.** A naive parser that pairs strictly by `(path, line)` and unifies records may misreport the symbol; the spec edit must clarify that pairing is `(path, line)`-keyed but the two records remain independent on the wire.

### Recommendation

**Option A** (clause (h), family (5) only). The conceptual asymmetry it preserves is correct: a marker is a marker-shaped audit artefact and belongs in the family-(5) bucket regardless of which family the underlying source line would have triggered. The dual-emission machinery's current scope to (1)/(2)/(3) is grounded in those families being inventory-resolvable surfaces a contributor can authorise — family (4) is by construction non-authorisable, so co-emitting it as a separate `(path, line)` peer adds no contributor-actionable signal beyond the family-(4) shape record itself. Edge cases the fixer must watch: the `<symptom>` token under clause (h) must be distinct from clauses (a)–(g) per the per-`<symptom>` fixture floor; the *Per-family record-shape table* family-(5) row's `proposed-resolution` enumeration must add the (h) sub-case naming branch (5) (not branch (4)) as the underlying-shape recovery arm; and the `V18b` / `V18b-T` leaf Test bullets must be re-aligned (the current "surfaces the family-(5) marker record but does not suppress the family-(4) shape record" wording already aligns with Option A's outcome on the wire — both records still emit — but the leaf bullet should cite clause (h) explicitly once it lands).

## Consequence

**Severity:** correctness

Two conforming audit implementations seeded with the same `// allow-pi-surface:` marker on a family-(4) line will emit different `audit/<class>/<family>/<symptom>` discriminators (one routes through family (4), the other through family (5), a third may dual-emit both); a CI parser pinned to one wire shape silently rejects the other. The per-`<symptom>` fixture floor cannot be satisfied unambiguously because the case has no agreed `<family>` token to mint a fixture against.
