## Finding

`H4a`'s **Adds** declares an **in-memory fixture-supply mechanism**: the harness supplies fixture `.loom` content to the extension in memory so a harness-driven leaf's discovery reads from the harness-provided source rather than the real filesystem (avoiding any ambient `src/**` filesystem read or `V8b` `FileSystem`-seam dependency). It is a first-class deliverable of this leaf — `M` and `M-T` are named as its first consumers for their single-source happy-path discovery.

None of `H4a`'s three **Tests** bullets exercises that mechanism. They cover (1) the factory completing its registrations, never throwing, and returning synchronously; (2) the harness loading the extension and dispatching a **no-op** command end-to-end against the session double; and (3) the session-double fidelity-contract self-check (streamed-token ordering, single-turn append, `pi.on` cancel-forward, cancellation propagation). The no-op dispatch does not feed `.loom` content through discovery, and the fidelity self-check is about the double's output behaviours, not the fixture-supply input path. The **Ships when** gate (`npm test` loads the extension, dispatches a no-op command, fidelity self-check passes) likewise never observes a discovery read consuming harness-provided fixture content.

The result: `H4a` can ship green with the fixture-supply mechanism broken or absent. The first place the path is actually exercised is `M`/`M-T`, well downstream of the owning leaf, so a defect surfaces far from where it was introduced.

## Consequence

**Severity:** advisory

The fixture-supply deliverable has no leaf-local falsifiability: a broken or incomplete supply path leaves `H4a` green and is first caught by `M`/`M-T` downstream, where the failure is harder to attribute to its origin. No implementer divergence results (the mechanism is eventually exercised), so the gap degrades early detection rather than blocking pickup.

## Issue introduction

**Verdict:** introduced-by-commit (single-commit)
**Introducing commits:** `2d52ad1` — "pi-loom plan: resolve 'M's happy-path discovery read has no defined source under the ambient-access ban'" (2026-06-11)
**History:** The in-memory fixture-supply mechanism did not exist in `H4a` before `2d52ad1`. `git log -S "in-memory fixture-supply"` on the leaf returns that single commit, and `git show 2d52ad1` confirms it added the mechanism to the **Adds** paragraph (to give `M`/`M-T` a defined discovery source under the ambient-access ban) without adding any corresponding **Tests** bullet or **Ships when** clause to gate it. The three Tests bullets and the Ships-when present at that commit (factory never-throw, no-op command dispatch, fidelity-contract self-check) have not since been extended to exercise the fixture-supply path; later edits to the leaf (`1e3a062`, `6a42c1a`, `4b4382f`) addressed unrelated concerns (fidelity-axis citations, the synchronous-void claim, manifest-path relocation). The leaf-local coverage gap entered with the mechanism at `2d52ad1` and persists.

## Solution Space

**Shape:** single

### Recommendation

Add a fourth `Convention:` **Tests** bullet to `docs/plan_topics/H4a-factory-shell-and-harness.md` asserting the in-memory fixture-supply mechanism: the harness is programmed with fixture `.loom` content, and a discovery-style read driven through the harness returns that programmed content (rather than reading the real filesystem). Extend the **Ships when** clause to include this assertion among the conditions `npm test` must satisfy, so the supply path is gated at its owning leaf rather than first exercised by `M`/`M-T`.

The assertion is internal to plan files — no spec edit is required, and the existing no-ambient-read / no-`V8b`-dependency property the mechanism already claims should be the observable the bullet checks (programmed content returned, no filesystem touched).
