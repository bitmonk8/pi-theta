## Finding

The `lexical.md` **Number literals** bullet states: "`integer` widens implicitly to `number` in arithmetic and assignment positions; the reverse is `loom/parse/integer-narrowing`." This is the type-compatibility subtyping relation `integer ⊑ number`, not a lexical fact. `type-system.md` already owns that relation canonically as **TYPE-2** ("`integer ⊑ number`. One-way only; the reverse is `loom/parse/integer-narrowing`") inside the closed `⊑` list, a section that declares itself normative and that "every site … cites … by anchor".

The relation is therefore stated twice, and ownership is inverted. TYPE-2 — the canonical compatibility rule — defers outward with "(see [Lexical — Number literals])", and the parse diagnostic registry (`diagnostics/code-registry-parse.md`) names Lexical as the owning page for `loom/parse/integer-narrowing`, while `frontmatter/frontmatter-fields-a.md` instead cites `type-system.md#type-compatibility` for the same rule. The closed `⊑` relation thus points at a lexical bullet for one of its own members.

`lexical.md` legitimately owns the literal-token type assignment ("a literal with no fractional or exponent part has type `integer`; otherwise `number`") and the per-token magnitude diagnostics (`integer-literal-out-of-range`, `number-literal-not-finite`); it should not also state the widening relation. The duplicate copy is additionally under-scoped: it limits widening to "arithmetic and assignment positions", whereas TYPE-2 governs every compatibility site (invoke return annotations, `match`-arm and ternary common types, `array<T>` element-against-sink, `params:` defaults, the `+` mixed-numeric case). A reader of `lexical.md` alone can come away believing `integer → number` widening is narrower than the spec actually intends.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `8eea523` ("spec: fill implementation-blocking gaps"), `0226b80` ("pi-loom plan: resolve \"V15c references an undefined compatibility relation\"")
**History:** `8eea523` (2026-05-04) added the new Lexical Structure section to the then-monolithic `spec.md`; its Number-literals bullet introduced "`integer` widens implicitly to `number` in arithmetic and assignment positions; the reverse is a parse error" at a time when no type-compatibility section existed, so Lexical was the only home. `fecb504` (same day) split the corpus per-topic, carrying the bullet verbatim into `lexical.md` (pure relocation). `0226b80` (2026-05-05) then introduced the formal `⊑` relation in `type-system.md`, with TYPE-2 stating `integer ⊑ number`, but did not consolidate the pre-existing lexical copy — instead wiring TYPE-2 to cite Lexical. The duplication and inverted ownership are the interaction of the original placement (`8eea523`) and the later canonical-relation addition (`0226b80`) that left the earlier copy in place.

## Solution Space

**Shape:** single

### Recommendation

Make TYPE-2 the single canonical statement of the `integer ⊑ number` relation and reduce `lexical.md` to a cross-reference:

- In `type-system.md` TYPE-2, state the relation and the reverse diagnostic directly; drop the "(see [Lexical — Number literals])" deferral for the relation. TYPE-2 may retain a note that integer-vs-number *literal-token typing* is defined in Lexical, but the compatibility semantics live here.
- In `lexical.md` **Number literals**, keep the literal-token type assignment ("a literal with no fractional or exponent part has type `integer`; otherwise `number`") and the per-token magnitude diagnostics, but replace the "`integer` widens implicitly to `number` in arithmetic and assignment positions; the reverse is `loom/parse/integer-narrowing`" sentence with a cross-reference to `type-system.md#type-2` for the relation. Do **not** re-state the "arithmetic and assignment positions" scope — that under-scoping is part of the defect.
- Reword the trailing magnitude-rule clause in the same bullet ("consistent with the `loom/parse/integer-narrowing` treatment of `integer`/`number` boundary violations") so it cites TYPE-2 for the compatibility semantics rather than asserting them in Lexical.
- Update the `loom/parse/integer-narrowing` row's "owning page" column in `diagnostics/code-registry-parse.md` to point at `type-system.md#type-2`, aligning the registry with the relocated relation and with the existing `frontmatter` citation. This is a consequence of consolidating the relation, not a separate change.

Edge case for the implementer: `loom/parse/integer-narrowing` is a parse-time diagnostic that also backs the lexical magnitude rules; after the move, `lexical.md` must still be able to reference the diagnostic by name (it remains a `loom/parse/*` code firing at lexed-token sites), so cross-reference the code rather than deleting every mention from Lexical.

## Consequence

**Severity:** advisory

TYPE-2 is declared normative and governs every compatibility site, so a careful implementer reaches the correct relation; nothing breaks at runtime. The defect is a redundant, mutually-cross-referenced statement whose lexical copy under-scopes the relation to "arithmetic and assignment positions" — a clarity and future-divergence risk (an editor updating one copy can leave the other stale) rather than a behavioural one.
