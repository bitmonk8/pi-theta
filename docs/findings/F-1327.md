## Problem

`V9i`'s **Adds** names the spawn-time check "the pre-spawn **binder-model** guard," and both `V9i` and `V9i-T` list `V11a` (binder-model resolution) in **Deps** ÔÇö pointing an implementer at binder-model resolution as the value the guard inspects. That is wrong: `subagent.md` ┬ºPre-spawn model guard and spawn-block rule 5 pin the guarded value to the subagent loom's own resolved `model:` (frontmatter `model:`, or inherited `ctx.model` when absent), and fire `loom/runtime/subagent-model-unresolved` only when that value is `undefined`. The binder model (`bind_model:`, resolved by `V11a`) is a distinct value not consumed at the subagent spawn site. The leaf is also internally inconsistent ÔÇö its own Tests bullet correctly says "pre-spawn **model** guard."

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (`pi-loom plan: build/update plan for spec.md + review`)
**History:** `docs/plan_topics/V9i-subagent-isolation.md` was created in `c6a664e`, and that creating revision already carries both the "pre-spawn binder-model guard" Adds wording and the `V11a` Deps edge (`git show c6a664e:ÔÇª` shows `**Deps.** \`V9i-T\`, \`V9a\`, \`V17a\`, \`V11a\``). No later commit altered the label or the dependency; the defect entered with the leaf at plan authoring.

## Solution approach

In `V9i`'s **Adds**, rename "the pre-spawn binder-model guard" to "the pre-spawn model guard" and state that the guard inspects the subagent loom's own resolved `model:` value (frontmatter `model:`, or inherited `ctx.model` when absent), failing the spawn with `loom/runtime/subagent-model-unresolved` when that value is `undefined`, per `subagent.md` rule 5 / ┬ºPre-spawn model guard. In both `V9i` and `V9i-T` **Deps**, replace the `V11a` edge with `V6a` (`V6a-frontmatter-contract.md`), the leaf that owns the loom `model:` resolution hook the guard's input flows through.
