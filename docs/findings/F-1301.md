## Problem

The *Count semantics — invocation-site framing* clause states emission-count bounds are measured at the handler invocation site "for the same `session_shutdown` event", yet its own enumeration includes the **idempotent** clause's at-most-once-per-extension-instance-lifetime bound on `loom/host/session-shutdown-runtime-degraded`. A per-extension-instance-lifetime bound necessarily ranges over multiple `session_shutdown` events delivered to the same instance (the multi-event delivery host-prerequisite (b) admits), so the "for the same `session_shutdown` event" qualifier cannot describe it, and the clause never states how invocation-site counts compose across events to deliver "at most once per extension-instance lifetime". An implementer cannot read this clause to derive both the per-event branch behaviours and the lifetime bound.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `52b9101` (2026-05-11) — "resolve 'T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state'"
**History:** `52b9101` introduced both the **idempotent** clause's at-most-once-per-extension-instance-lifetime bound and the *Count semantics — invocation-site framing* clause that names it, in one large multi-concern edit, without reconciling the latter's "same event" framing against the former. Subsequent commits preserved both phrasings verbatim.

## Solution approach

Clarify the *Count semantics — invocation-site framing* clause in `diagnostic-emission-isolation.md` so the count scopes it ranges over are distinguished: separate the per-event bounds from the **idempotent** clause's per-extension-instance-lifetime bound on `loom/host/session-shutdown-runtime-degraded`. State how invocation-site enterings compose across `session_shutdown` events to deliver the lifetime bound.
