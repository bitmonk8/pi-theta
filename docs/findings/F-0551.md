## Finding

Spec rule 2 of `binder/defaulting-system-note-echo.md` §System-note rendering states two things about the 120-code-point cap: the runtime truncates at a Unicode scalar boundary to exactly 120 code points (the `…` counts toward the cap), **and** an implementation MAY additionally back the truncation point off to the nearest preceding extended grapheme cluster boundary as a non-normative rendering courtesy, provided the result is still ≤120 code points. Because the cluster back-off is permitted but non-normative, the spec closes with an explicit conformance constraint: "this back-off is non-normative and **tests MUST NOT assert cluster-aware behaviour**."

V11e and V11e-T reproduce only the first half. Their rule-2 Tests bullet reads "truncates at a Unicode scalar boundary with a trailing `…` to exactly 120 code points … a note ≤120 code points gets no `…`," and never carries the prohibition against asserting cluster-aware behaviour. The omission leaves the test author free to choose a vector where the scalar boundary and the grapheme-cluster boundary differ and assert the exact truncated string. Such a test pins one of the two spec-conformant outcomes and rejects the other — it would fail a conforming implementation that exercises (or declines) the permitted cluster back-off, turning a non-normative courtesy into a test-enforced requirement.

## Solution Space

**Shape:** single

### Recommendation

Extend the rule-2 (120-code-point cap) Tests bullet in both `docs/plan_topics/V11e-system-note-determinism.md` and `docs/plan_topics/V11e-T-system-note-determinism.md` to reproduce the spec's cluster-aware prohibition. Append to the existing rule-2 bullet text that the grapheme-cluster back-off permitted by spec rule 2 is non-normative and tests MUST NOT assert cluster-aware truncation — the test verifies only that an over-cap note is reduced to ≤120 code points ending in `…` and that a ≤120-code-point note is unchanged, choosing vectors where the scalar and cluster boundaries coincide (or asserting only the ≤120 invariant) rather than asserting the exact truncation point.

The `cka-41` row in `coverage-matrix.md` already summarises rule 2 as "120-code-point truncation-with-`…`" and need not change; it tracks the rule, not its test-authoring constraint.

## Consequence

**Severity:** correctness

The V11e-T author can write a rule-2 test that asserts truncation at a specific code-point/cluster boundary; two reasonable authors diverge on which conformant outcome to pin, and a test that asserts cluster-aware truncation contradicts the spec's explicit "tests MUST NOT assert cluster-aware behaviour" constraint. The result is a test that rejects a spec-conformant implementation of the 120-cap rather than one that verifies the actual normative invariant (note is ≤120 code points; `…` appended only on overflow).
