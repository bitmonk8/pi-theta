## Finding

The `loom/runtime/watcher-terminated` row's *Trigger* column in `code-registry-runtime.md` reads "The chokidar file watcher over the discovered roots terminated — its `error` route fired or it threw such that one or more watched roots stop delivering events …", and the preamble paragraph immediately above the table introduces the same code as "emitted when the chokidar file watcher's `error` route fires or it throws such that one or more watched roots stop delivering events". Both surfaces are normative — they sit inside the closed diagnostic registry the DIAG-2 gate consumes — and both name a specific npm package (`chokidar`) and one of that package's internal event names (`error`).

Everywhere else in the spec corpus the watcher abstraction is the `FileWatcher` seam pinned by [PIC-14](../spec_topics/pi-integration-contract/host-interfaces-services.md#pic-14); `chokidar` itself is explicitly deferred to implementation-notes via the `Host prerequisites — Loom-package implementation dependencies` paragraph, which lists `chokidar` as a `FileWatcher`-side production choice the spec does not pin. The trigger condition the row records is observable as an `Unsubscribe`-equivalent post-condition over the seam (no further events delivered for one or more watched roots); naming the production adapter and its private event in the trigger conflates the seam contract with one of its conforming implementations, makes the row look like it would need a registry edit if the adapter were ever swapped, and is inconsistent with sibling rows (`registry-swap-failed`, `system-note-delivery-failed`, etc.) which name the abstract operation rather than its npm implementation.

The same wording also appears in the [PIC-55](../spec_topics/pi-integration-contract/registration-steps.md#pic-55) prose that the row back-references, but PIC-55 is the watcher-lifecycle surface that owns the chokidar adapter reference (PIC-14 explicitly names it there) and is a separate finding. The fix here is confined to the diagnostic registry text — the registry's Trigger and preamble should describe the observable post-condition over the seam, not the production adapter and its private event name.

## Solution Space

**Shape:** single

### Recommendation

Rewrite both occurrences to key on the `FileWatcher` seam's observable post-condition rather than naming the production adapter and its private event:

- In the runtime-registry **preamble** at `code-registry-runtime.md` (the long paragraph above the table), replace "emitted when the chokidar file watcher's `error` route fires or it throws such that one or more watched roots stop delivering events" with wording such as "emitted when the `FileWatcher` seam over the discovered roots terminates — one or more watched roots stop delivering events and the adapter is left torn down rather than re-armed". Keep the back-reference to [PIC-55](../spec_topics/pi-integration-contract/registration-steps.md#pic-55) unchanged.
- In the `watcher-terminated` row's **Trigger** cell, replace the opening "The chokidar file watcher over the discovered roots terminated — its `error` route fired or it threw such that one or more watched roots stop delivering events" with wording such as "The `FileWatcher` seam over the discovered roots terminated — one or more watched roots stop delivering events (whether via the seam's adapter-internal error path or a synchronous throw out of the watcher; see [PIC-55](../spec_topics/pi-integration-contract/registration-steps.md#pic-55))". The remainder of the cell (the `LoomRegistry`-stays-live clause, the no-drain-state-tag clause, the `loom-system-note` channel sentence) is unchanged.

Edge cases the implementer must watch:
- The `Message template` column (`loom watcher terminated; hot-reload halted until /reload`) is **not** edited — it is asserted character-for-character by `DIAG-4` tests under `V7b`.
- The Spec-rule pointer (`[Pi Integration Contract — PIC-55](../spec_topics/pi-integration-contract/registration-steps.md#pic-55)`) is also unchanged.
- PIC-55's prose in `registration-steps.md` still names `chokidar` and its `error` route; that wording is appropriate there because PIC-55 sits next to the PIC-14 production-adapter reference and is the lifecycle-surface owner, and a separate finding addresses any reshaping of PIC-55's observability story. Do **not** sweep PIC-55's wording as part of this fix.

## Consequence

**Severity:** advisory

Implementers can produce a working system from the current text — the post-condition is recoverable, and the surrounding PIC-55 prose pins behaviour. The cost is the inconsistency with how every other row keys its trigger on the abstract operation rather than the npm package, and a misleading impression that swapping the `FileWatcher` adapter would require a registry edit.
