## Problem

The paired `V13g` and `V13g-T` leaves each carry a single `Tests.` bullet that folds two independent obligations into one assertion: the parse-time `loom/parse/discarded-query-result` error on a bare `` @`…` `` expression-statement, and the runtime discard-observability event for a discarded query. The bullet's own wording ("…separately, a discarded query…") concedes the two halves are distinct behaviours sourced from different spec pages. A static parse diagnostic and a runtime channel event are divisible — they fail independently and are exercised by different fixtures — so folding them violates the per-rule red attribution that `conventions.md` §Per-phase TDD ritual relies on. A TDD red that exercises one arm while the other fails reds the single bullet with no signal as to which obligation broke.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `4a04570` (2026-06-17) — "pi-loom plan: resolve "V13a bundles five independent query surfaces into one leaf""
**History:** `git log --follow` over `V13g-query-discard-observability.md` and `V13g-T-query-discard-observability.md` shows the leaf was created by splitting it out of `V13a` in `4a04570`. That inaugural revision already stated both obligations in one bullet joined by "separately". The later commits `650adcf` and `4e395ef` (both 2026-06-18) only enriched the runtime half; neither split the bullet. The merged-bullet defect has existed since the leaf's inception.

## Solution approach

Split the single `Tests.` bullet in both `V13g` and `V13g-T` into two bullets so each reds in isolation and names the rule it asserts: one bullet for the parse error (`loom/parse/discarded-query-result` on a bare `` @`…` `` expression-statement) and one bullet for the runtime discard-observability event (the discarded-query event behaviour, channel routing, payload preservation, and `discard_site` stamping). Keep the two leaves' Tests sections identical, as they currently carry the same bullet verbatim.

## Solution constraints

- The spec is read-only for this fix — both obligations already exist in the cited spec pages; no spec edit is required.
