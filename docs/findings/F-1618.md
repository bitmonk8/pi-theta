## Finding

The "Why the transcript bytes are pinned" sub-section of `binder-model-and-context.md` asserts an *input-reproducibility* contract: the same session MUST construct the same binder-input bytes on every run and across processes (and BNDR-7a–BNDR-7j are pinned MUST-reproduce-exactly on that basis). Those bytes are a function of the included-turn set chosen by the session-context truncation walk, which selects turns by summing per-message counts returned by the `TokenEstimator` seam (PIC-16); production wiring delegates to the `estimateTokens` named import owned by Pi.

Nothing in the spec asserts that production `TokenEstimator.estimate(message)` (equivalently `estimateTokens(message)`) is a deterministic pure function of the `message` argument. PIC-16 acknowledges only that Pi's heuristic is "version-nondeterministic" (its motivation for offering a `FakeTokenEstimator` to conformance tests) and the `estimateTokens` reference entry says the algorithm "is owned by Pi and is not re-specified here." A production estimator that returns different counts across calls on identical inputs — or whose result depends on hidden mutable state, time, randomness, or thread-local context — silently shifts the included-turn boundary and therefore the transcript bytes across runs, defeating the input-reproducibility claim without violating any stated rule.

The consequence is that loom's strongest-stated determinism claim depends on a behavioural property of an external dependency that neither the spec nor the Pi SDK pin asserts. Conformance tests sidestep the question by injecting `FakeTokenEstimator`, so the gap is invisible inside the test boundary. The natural place to close it is on PIC-16's bullet list (as a behavioural precondition on the `PiTokenEstimator` adapter / the `estimateTokens` import it delegates to), with the input-reproducibility prose calling the precondition out by reference — paralleling the `.messages` chronological-ordering and leading-`user` presuppositions already pinned at `host-prerequisites.md`.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `8f7266c` (2026-06-03, "spec: resolve 'Binder Determinism reproducibility scope overreaches provider contract'"), `1f37b15` (2026-06-05, "spec: resolve 'Session-context truncation — no TokenEstimator DI seam for estimateTokens'")
**History:** `8f7266c` re-scoped the binder-determinism prose from a provider-side reproducibility claim to an *input-reproducibility* contract pinning the prompt bytes loom constructs, while explicitly disclaiming provider sampling. `1f37b15` then introduced the PIC-16 `TokenEstimator` seam and split production (`PiTokenEstimator` → `estimateTokens`) from tests (`FakeTokenEstimator`), motivating the seam as defending conformance tests against "Pi's version-nondeterministic estimation heuristic" — a phrasing that names the variability `loom` is exposed to but stops short of constraining it. The defect emerged at the interaction of these two commits: the input-reproducibility contract names the bytes it owns but does not name the per-message-determinism precondition the truncation walk silently relies on, and the seam definition names test-side determinism but does not assert the production adapter's input-determinism over the message argument. Subsequent commits (`05e84dd`, `f088698e`, `97ab51e`, `0ffc66e`) refined adjacent contracts (`.messages` ordering, BNDR-7 oracles, BNDR-8 canonical JSON, presupposition relocation) without revisiting this gap.

## Solution Space

### Recommendation

Add an explicit behavioural precondition to PIC-16's bullet list pinning the production `TokenEstimator` as deterministic in its `message` argument, cross-link it from the input-reproducibility prose, and add a re-audit item to the Pi version-bump checklist so the property is re-checked on each Pi minor bump.

Apply these spec edits:

- `host-interfaces-services.md` PIC-16: append a bullet — "Production `TokenEstimator.estimate(message)` is a deterministic pure function of `message` at any single [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin): repeated calls with the same `AgentMessage` return the same integer, and the result depends on no mutable runtime state, ambient time source, or randomness. This is a behavioural precondition on the `PiTokenEstimator` adapter / `estimateTokens` import; it is the property the [input-reproducibility contract](../binder/binder-model-and-context.md#why-the-transcript-bytes-are-pinned) relies on to keep the truncation walk's included-turn set stable across runs. A Pi minor bump may legitimately change per-message counts (Pi owns the algorithm); see [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) for the re-audit duty."
- `binder-model-and-context.md` "Why the transcript bytes are pinned": after the sentence ending "stays stable across runs.", add a clause naming the new PIC-16 precondition: "The truncation walk's included-turn set additionally relies on the `TokenEstimator` seam's [per-message determinism precondition](../pi-integration-contract/host-interfaces-services.md#pic-16); shifting per-message counts at a fixed Pi pin would shift the included-turn boundary and therefore the transcript bytes."
- `host-prerequisites.md`: add a row to the host-presuppositions register pointing at the new PIC-16 bullet, mirroring the form used for the `.messages` ordering and leading-`user` presuppositions.
- `version-bump-step2.md` editorial checklist: add an item — "(r) Re-audit that `estimateTokens` remains deterministic in its `AgentMessage` argument at the new pin (PIC-16 precondition); record any change to per-message counts in the bump notes so test fixtures and any prompt-cache keys depending on the included-turn set can be re-baselined."

Implementer edge cases: (a) the precondition is on the *message argument*, not on hidden state — a per-runtime memoisation cache inside `PiTokenEstimator` that only ever returns the result of `estimateTokens(message)` is permitted; (b) on a Pi minor bump that legitimately shifts per-message counts, conformance tests using `FakeTokenEstimator` remain green by construction (they bypass `estimateTokens`), so the audit item is the *only* mechanism that catches drift — do not rely on the test suite to surface it.

## Consequence

**Severity:** correctness

Two reasonable implementers can diverge on observable binder input: one assumes `estimateTokens` is pure and ships the BNDR-7 byte-identical contract on that basis; another, reading only what is written, treats the estimator as opaque and accepts a `PiTokenEstimator` adapter that, on a future Pi minor bump, returns different counts for the same message — silently shifting which turns the truncation walk includes and breaking the MUST-reproduce-exactly transcript pins without violating any stated rule. The defect also widens the Pi-version-bump checklist's auditing duty without naming the property being audited.
