## Finding

The "Why the transcript bytes are pinned" sub-section of `binder-model-and-context.md` asserts an *input-reproducibility* contract: the same session MUST construct the same binder-input bytes on every run and across processes (and BNDR-7aâ€“BNDR-7j are pinned MUST-reproduce-exactly on that basis). Those bytes are a function of the included-turn set chosen by the session-context truncation walk, which selects turns by summing per-message counts returned by the `TokenEstimator` seam (PIC-16); production wiring delegates to the `estimateTokens` named import owned by Pi.

Nothing in the spec asserts that production `TokenEstimator.estimate(message)` (equivalently `estimateTokens(message)`) is a deterministic pure function of the `message` argument. PIC-16 acknowledges only that Pi's heuristic is "version-nondeterministic" (its motivation for offering a `FakeTokenEstimator` to conformance tests) and the `estimateTokens` reference entry says the algorithm "is owned by Pi and is not re-specified here." A production estimator that returns different counts across calls on identical inputs â€” or whose result depends on hidden mutable state, time, randomness, or thread-local context â€” silently shifts the included-turn boundary and therefore the transcript bytes across runs, defeating the input-reproducibility claim without violating any stated rule.

The consequence is that loom's strongest-stated determinism claim depends on a behavioural property of an external dependency that neither the spec nor the Pi SDK pin asserts. Conformance tests sidestep the question by injecting `FakeTokenEstimator`, so the gap is invisible inside the test boundary. The natural place to close it is on PIC-16's bullet list (as a behavioural precondition on the `PiTokenEstimator` adapter / the `estimateTokens` import it delegates to), with the input-reproducibility prose calling the precondition out by reference â€” paralleling the `.messages` chronological-ordering and leading-`user` presuppositions already pinned at `host-prerequisites.md`.

**History:** `8f7266c` re-scoped the binder-determinism prose from a provider-side reproducibility claim to an *input-reproducibility* contract pinning the prompt bytes loom constructs, while explicitly disclaiming provider sampling. `1f37b15` then introduced the PIC-16 `TokenEstimator` seam and split production (`PiTokenEstimator` â†’ `estimateTokens`) from tests (`FakeTokenEstimator`), motivating the seam as defending conformance tests against "Pi's version-nondeterministic estimation heuristic" â€” a phrasing that names the variability `loom` is exposed to but stops short of constraining it. The defect emerged at the interaction of these two commits: the input-reproducibility contract names the bytes it owns but does not name the per-message-determinism precondition the truncation walk silently relies on, and the seam definition names test-side determinism but does not assert the production adapter's input-determinism over the message argument. Subsequent commits (`05e84dd`, `f088698e`, `97ab51e`, `0ffc66e`) refined adjacent contracts (`.messages` ordering, BNDR-7 oracles, BNDR-8 canonical JSON, presupposition relocation) without revisiting this gap.

## Solution Space

**Shape:** multiple

### Option A â€” Assert estimator input-determinism as a PIC-16 precondition

**Approach.** Add an explicit behavioural precondition to PIC-16's bullet list: production `TokenEstimator.estimate(message)` (and the `estimateTokens` it delegates to) MUST be a deterministic pure function of the `message` argument at any single Pi-SDK pin â€” calls with `===`-equal `AgentMessage` graphs return the same integer count, with no dependence on time, randomness, thread state, or hidden mutable state. Cross-link it from the `binder-model-and-context.md` "Why the transcript bytes are pinned" paragraph as a named consumption-precondition (paralleling the `.messages` chronological-ordering and leading-`user` presuppositions already pinned at `host-prerequisites.md`), and add a corresponding entry to the Pi version bump procedure's *Editorial-review checklist for unpinned host presuppositions* so the property is re-audited on each Pi minor bump.

**Spec edits.**
- `host-interfaces-services.md` PIC-16: append a bullet â€” "Production `TokenEstimator.estimate(message)` is a deterministic pure function of `message` at any single [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin): repeated calls with the same `AgentMessage` return the same integer, and the result depends on no mutable runtime state, ambient time source, or randomness. This is a behavioural precondition on the `PiTokenEstimator` adapter / `estimateTokens` import; it is the property the [input-reproducibility contract](../binder/binder-model-and-context.md#why-the-transcript-bytes-are-pinned) relies on to keep the truncation walk's included-turn set stable across runs. A Pi minor bump may legitimately change per-message counts (Pi owns the algorithm); see [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) for the re-audit duty."
- `binder-model-and-context.md` "Why the transcript bytes are pinned": after the sentence ending "stays stable across runs.", add a clause naming the new PIC-16 precondition: "The truncation walk's included-turn set additionally relies on the `TokenEstimator` seam's [per-message determinism precondition](../pi-integration-contract/host-interfaces-services.md#pic-16); shifting per-message counts at a fixed Pi pin would shift the included-turn boundary and therefore the transcript bytes."
- `host-prerequisites.md` (optional): add a row to the host-presuppositions register pointing at the new PIC-16 bullet, mirroring the form used for the `.messages` ordering and leading-`user` presuppositions.
- `version-bump-step2.md` editorial checklist: add an item â€” "(r) Re-audit that `estimateTokens` remains deterministic in its `AgentMessage` argument at the new pin (PIC-16 precondition); record any change to per-message counts in the bump notes so test fixtures and any prompt-cache keys depending on the included-turn set can be re-baselined."

**Pros.**
- Preserves the stronger, already-pinned input-reproducibility contract (BNDR-7aâ€“BNDR-7j stay MUST-reproduce-exactly).
- Surfaces a real Pi-side property that current implementations almost certainly already satisfy (`@earendil-works/pi-coding-agent`'s `compaction.ts` is a pure character-count function), so adoption cost is zero at the pin.
- Matches the seam-presupposition idiom (`.messages` chronological order, leading-`user` message) already used elsewhere in PIC, so it composes cleanly with existing audit machinery.

**Cons.**
- Adds an external-dependency obligation that loom cannot mechanically enforce; the audit duty has to ride the Pi-version-bump checklist forever.
- Reads as "loom requires Pi to be deterministic" â€” a slightly aggressive posture toward an upstream the spec elsewhere treats as a black box (`estimateTokens` is non-normative orientation; algorithm is owned by Pi).

**Risks.**
- If a future Pi version legitimately makes its estimator non-pure (e.g. memoised against a session-level cache that mutates), the precondition is violated and the input-reproducibility claim breaks at the bump rather than degrading silently. That is the *intended* outcome â€” failure is loud at the audit boundary â€” but it does require the audit actually runs.

### Option B â€” Narrow the input-reproducibility contract to "given an estimator that is pure in its argument"

**Approach.** Leave PIC-16 unchanged and instead narrow the prose in "Why the transcript bytes are pinned" so the input-reproducibility claim is conditioned on the consumed estimator's purity, rather than asserting it as a global property. State that the BNDR-7aâ€“BNDR-7j byte pins are MUST-reproduce-exactly *given* the truncation walk produced a particular included-turn set, and that the included-turn set is a function of whatever the injected `TokenEstimator` returns â€” so input-reproducibility holds *for any fixed estimator behaviour* but is not itself a claim about `estimateTokens`'s stability across calls or across Pi versions.

**Spec edits.**
- `binder-model-and-context.md` "Why the transcript bytes are pinned": rewrite the "the same session always constructs the same prompt bytes" clause to read "the same session, *given the same included-turn set chosen by the [truncation walk](#session-context-truncation-bind_context-session)*, always constructs the same prompt bytes â€¦". Add a follow-on sentence: "The included-turn set is a function of the injected `TokenEstimator`'s per-message counts; loom does not constrain how the production [`estimateTokens`](../pi-integration-contract/host-interfaces-core.md#estimatetokens-named-export) computes those counts, and a Pi minor bump that shifts counts may legitimately shift the included-turn set and therefore the transcript bytes."
- BNDR-7 reference renderings: add a one-line normative note that each rendering's MUST-reproduce-exactly status is scoped to "given the included-turn set the worked example fixes" â€” making the pre-walked included-turn set part of the test fixture rather than an emergent claim.

**Pros.**
- Honest about what loom owns: prompt-construction bytes are loom code; per-message token counts are Pi code.
- No external-dependency obligation; the Pi version-bump checklist gains no new item.

**Cons.**
- Weakens the contract authors plausibly want â€” "the same session always produces the same binder input" is the cleaner claim, and conditioning it on "given the same included-turn set" pushes a non-trivial latent dependency onto downstream readers (any prompt-cache or audit log keyed on transcript bytes now has to model truncation-walk drift across Pi pins).
- BNDR-7 oracles already assume a specific included-turn set; the rewrite is mostly editorial but invites a reader to ask "so when *does* the byte pin hold, in production?", which Option A answers and Option B does not.

**Risks.**
- Future readers may re-discover the gap and re-raise it, because the rewritten claim is provably weaker than what the surrounding BNDR-7 / determinism prose still implies.

### Recommendation

**Take Option A.** The input-reproducibility contract is the load-bearing claim the BNDR-7 oracles, prompt-cache stability, and the Pi-version-bump audit checklist already rest on; narrowing it (Option B) pushes the latent dependency onto every downstream reader and weakens audit value. Pi's current `estimateTokens` is pure-in-argument at the pin, so Option A documents a property that already holds and converts a silent assumption into a checklist item â€” matching how the spec handles `.messages` chronological ordering and the leading-`user` presupposition. Edge cases for the implementer: (a) the precondition is on the *message argument*, not on hidden state â€” a per-runtime memoisation cache inside `PiTokenEstimator` that only ever returns the result of `estimateTokens(message)` is permitted; (b) on a Pi minor bump that legitimately shifts per-message counts, conformance tests using `FakeTokenEstimator` remain green by construction (they bypass `estimateTokens`), so the audit item is the *only* mechanism that catches drift â€” do not rely on the test suite to surface it.

## Consequence

**Severity:** correctness

Two reasonable implementers can diverge on observable binder input: one assumes `estimateTokens` is pure and ships the BNDR-7 byte-identical contract on that basis; another, reading only what is written, treats the estimator as opaque and accepts a `PiTokenEstimator` adapter that, on a future Pi minor bump, returns different counts for the same message â€” silently shifting which turns the truncation walk includes and breaking the MUST-reproduce-exactly transcript pins without violating any stated rule. The defect also widens the Pi-version-bump checklist's auditing duty without naming the property being audited.
