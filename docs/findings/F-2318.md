## Finding

The `ToolDefinition.label`-derivation Tests bullet appears verbatim in both paired leaves `V9f` and `V9f-T`, but only the `V9f` copy carries the governance parenthetical `(GOV-22 un-anchored residue)`; the `V9f-T` copy omits it. The annotation marks the obligation as an un-anchored normative MUST tracked under `governance.md` GOV-22 and closed at this leaf (coverage-matrix row `cka-17` → `V9f`).

Across the rest of the plan corpus this annotation is a consistent paired-file convention: both members of every other pair that closes a GOV-22 residue obligation carry it (e.g. `V9l`/`V9l-T` and `V14a`/`V14a-T` each carry `(un-anchored; GOV-22 residue)` on the matching bullet in both files). `V9f-T` is the lone outlier whose mirror bullet drops the marker, leaving the two paired files inconsistent on a bullet that is otherwise identical.

The divergence is a metadata inconsistency, not a behavioural one: the assertion text the implementer acts on is identical in both files, and reconciling it only aligns the GOV-22 navigational annotation with the prevailing corpus convention.

The paired mirror bullet was authored later in `66e4a27` (2026-06-17) on `V9f-T` without the annotation (`git log -G "GOV-22"` returns no match on the `V9f-T` file), introducing the divergence at that commit. The corpus convention to carry the annotation in both paired members predates and surrounds this leaf.

## Solution Space

**Shape:** single

### Recommendation

Reconcile the two paired files so the `ToolDefinition.label`-derivation Tests bullet carries the GOV-22 residue annotation consistently. Add the annotation to the `V9f-T` bullet so it matches `V9f` and the prevailing corpus convention (both members of `V9l`/`V9l-T` and `V14a`/`V14a-T` carry it). Concretely, in `docs/plan_topics/V9f-T-tool-registration-lifetime.md` the bullet currently reads:

> `extension-bootstrap-and-per-loom.md` §Per-loom registration `ToolDefinition.label` derivation: materialising `code-review.loom` yields `label: "Code-review"` …

and should be brought into line with the `V9f` copy, which reads:

> `extension-bootstrap-and-per-loom.md` §Per-loom registration `ToolDefinition.label` derivation (GOV-22 un-anchored residue): materialising `code-review.loom` yields `label: "Code-review"` …

i.e. insert `(GOV-22 un-anchored residue)` after `derivation` in the `V9f-T` bullet so both paired bullets read identically. (Deleting the annotation from `V9f` instead would resolve the pair internally but break it against the corpus convention, so harmonise toward keeping the marker.)

## Consequence

**Severity:** cosmetic

A reader or maintainer scanning the paired files sees one carrying the GOV-22 residue annotation and the other not, which reads as an authoring slip rather than an intentional difference; nothing the implementer builds or tests changes. The mismatch is purely organisational.
