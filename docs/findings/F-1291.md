## Problem

`type-system.md` TYPE-9 routes ternary `cond ? a : b` static type-mismatches through the array-and-ternary common-type machinery in expressions.md, emitting `loom/parse/array-element-type-mismatch` or `loom/parse/array-no-common-type`. But the only place the spec enumerates a sink set *exhaustively* is grammar.md §"`array<T>` literal type-sink rule", whose title and scope cover `array<T>` literals only; expressions.md rule 1 turns on a non-exhaustive "etc.". Nothing normative declares that the grammar enumeration is the canonical sink set for ternary branches too. So for `let x: T = cond ? a : b` with a branch statically `⋢ T`, one implementer reads the enumeration strictly and never fires rule 1 for ternaries; another propagates the `let` annotation into the branches and fires `array-element-type-mismatch` — and a conformance test has no normative answer to assert.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f841b12` (2026-05-04) authored the array-scoped exhaustive sink section; `648365c` (2026-06-06) added TYPE-9 routing ternary into the same shared machinery without extending that enumeration.
**History:** `git log -L 196,205:docs/spec_topics/grammar.md` shows the "`array<T>` literal type-sink rule" section was added by `f841b12` ("spec: resolve Cluster 2 (literal sublanguage + grammar appendix)"), already titled/worded for `array<T>` literals only and internally consistent because no other construct claimed the enumeration. `648365c` ("spec: resolve 'Three subtype check sites name no diagnostic codes'") then introduced TYPE-9 routing ternary mismatches "through the array-and-ternary common-type machinery" and naming the two diagnostics, touching only `type-system.md` + `code-registry-parse.md`; the `grammar.md` sink enumeration was not revisited and remained array-scoped. The `expressions.md` "*(and ternary branches)*" subtitle has been present since the original split `fecb504` with a non-exhaustive "etc." in rule 1. The gap is the interaction between an exhaustive enumeration written for one consumer and a second consumer pointed at it without extending it.

## Solution approach

Generalise grammar.md's §"`array<T>` literal type-sink rule" so its exhaustive sink-set enumeration is the single source of truth governing both `array<T>` literals and ternary branches, not array literals alone. Replace the non-exhaustive "etc." in expressions.md's common-type rule 1 with a forward-link to that generalised grammar section. Tighten the glossary's informal ternary-transparency aside in the "type sink" entry to cite the generalised section.

## Solution constraints

- Out of scope: the match-arm common-type consumer in `error-model.md` — it is the same-cluster sibling finding's territory; do not extend the generalised enumeration to match arms here.
- The `for x in expr` carve-out must remain in force — `for` does not supply `T`, and `for x in (cond ? [] : [])` is still `loom/parse/array-no-common-type`.
