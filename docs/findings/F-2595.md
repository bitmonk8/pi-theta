## Finding

DISC-6 (`docs/spec_topics/discovery/package-and-settings.md`) specifies the per-read deadline race that bounds each candidate `package.json` read. On per-read timeout the in-flight read is abandoned, and the spec states: *"the read's eventual settlement MUST be silenced with `.catch(() => {})` and its result MUST NOT be re-routed back into the discovery pass"*. The `.catch(() => {})` clause pins an exact JavaScript token as a normative MUST.

The observable contract here is twofold and mechanism-independent: the abandoned read's eventual settlement (a) must not re-enter the discovery pass, and (b) must not surface as an `unhandledRejection`. Any of several constructs satisfy both — a no-op rejection handler, `void p.catch(() => {})`, a `try`/`catch` around an awaited race, or an `AbortController`-derived discard — and the spec has no reason to prefer one. Pinning the literal `.catch(() => {})` token elevates an implementation detail to a conformance requirement.

This is internally inconsistent with the same paragraph, which deliberately leaves the await construct to the implementer: *"the implementation construct used to await whichever resolves first is left to the implementer (e.g. an `AbortController` derived from `Clock.setTimeout`)"*. The suppression mechanism and the await mechanism are the same class of implementation choice; one is delegated, the other is pinned. The fix is to state the behavioural requirement (no re-entry into the pass; no `unhandledRejection`) and demote the specific token to a non-normative illustrative example, matching how the await construct is already handled.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `25e8a60` — "pi-loom spec: resolve \"T08 — Per-package.json read timeout: overrun is unbounded\"" (2026-05-07)
**History:** The corpus is git-tracked. `git log -S'.catch(() => {})' -- docs/` and `git show 25e8a60` show the per-read deadline mechanism — and with it the `.catch(() => {})` MUST — entered the normative spec in `25e8a60` (then in `discovery.md`, later relocated verbatim into the split topic file `discovery/package-and-settings.md` by `f5e89f4` without altering the token). That commit resolved the earlier "T08" finding; the fix's recommendation prose itself carried the line *"the runtime MUST attach a `.catch(() => {})` to silence unhandled-rejection warnings without re-routing the late result"*, and the mechanism token was copied verbatim into the normative body alongside the (correct) behavioural `MUST NOT be re-routed` clause. The over-prescription is therefore original to the per-read-deadline feature's introduction, not a later regression: there is no prior revision that stated the behavioural requirement mechanism-free.

## Solution Space

**Shape:** single

### Recommendation

In the DISC-6 abandoned-read clause, replace the normative `.catch(() => {})` mandate with the two observable requirements and demote the token to a non-normative example. Concretely, change *"the read's eventual settlement MUST be silenced with `.catch(() => {})` and its result MUST NOT be re-routed back into the discovery pass"* to a behavioural form such as: *"the read's eventual settlement MUST NOT be re-routed back into the discovery pass and MUST NOT surface as an `unhandledRejection` (e.g. by attaching a no-op rejection handler such as `.catch(() => {})`)"*. This mirrors the paragraph's existing treatment of the await construct, which already names `AbortController` only as an illustrative `e.g.`. Keep both MUSTs (no re-entry; no unhandled rejection) normative — they are the testable surface V10b-T asserts.

## Consequence

**Severity:** advisory

Implementers can still build a working bounded walk; the pinned token does not cause behavioural divergence. The risk is that a conformance check keyed on the literal `.catch(() => {})` string would wrongly reject an equally-correct suppression mechanism, and that the over-pinned token contradicts the paragraph's own delegation of the await construct, sending mixed signals about what is normative.
