## Finding

`docs/spec_topics/pi-integration-contract/audit-wire-and-canary.md` ends with a standalone paragraph beginning *"Item 5 (cancellation propagation) is verified jointly by Step 0 item (b)'s `AbortSignal` / `AbortController` shape probe … Item 7 (binder LLM model) is verified at per-loom load time …"*. The paragraph is the inventory's per-item routing of how the two non-factory-probable capabilities are verified, and it terminates with the *"name-links each item back to the obligation here so that downstream tests cite a single anchored source"* sentence — anchored language that refers to the seven-item inventory enumeration, not to the wire/canary/infrastructure-failure contract the rest of the page pins.

The owning surface for this content is the SDK capability inventory itself, which now lives in `inventory-audit-intro.md` (PIC-15 preamble) and `capability-inventory-items.md` (the seven enumerated items, including item 5 cancellation propagation and item 7 binder LLM model). On the page where the paragraph currently sits, every other section is about the audit's stdout wire format, its infrastructure-failure handling, and its non-empty-scan canary. The items-5/7 paragraph constrains none of those surfaces and is reachable from neither: it is a stranded fragment a reader scanning audit-wire-and-canary.md for routing or fail-closed obligations will not look for, and a reader of the capability inventory will not find at all because the cross-link goes the wrong way.

The orphaning originates in the size-cap split that produced this directory layout (`f5e89f4`, *"docs: enforce 100KB hard / 30KB recommended size cap on spec set"*): the paragraph used to sit immediately above the items 1–7 enumeration in the single-file `pi-integration-contract.md` and was correctly adjacent there. The split routed the enumeration to `capability-inventory-items.md` but left the paragraph attached to the preceding wire/canary block, where it now reads as a non-sequitur trailer.

## Issue introduction

**Verdict:** edit-introduced
**Introducing commits:** `f5e89f4` (`docs: enforce 100KB hard / 30KB recommended size cap on spec set`, 2026-06-04) for the orphaning; `5c8e630` (`pi-loom spec: resolve "Subagent cancellation wiring depends on a non-existent createAgentSession({ signal }) option"`, 2026-05-07) for the most recent in-place edit to the paragraph; `126ff8a` (`pi-loom spec: Step 0 capability probe rewrite + V1.0 deferrals + Choice B`, 2026-05-06) for the paragraph's first introduction in approximately its current shape.
**History:** Before the size-cap split (`f5e89f4`), the items-5/7 verification paragraph lived in the single-file `spec_topics/pi-integration-contract.md` immediately above the items 1–7 enumeration (`git show f5e89f4^:docs/spec_topics/pi-integration-contract.md` shows it at the start of the *SDK capability inventory* section). The split mechanically routed the enumeration to a new child `capability-inventory-items.md` (commit message: *"pi-integration-contract.md (576KB) -> index + 30 children"*) but the paragraph stayed with the audit-wire/canary text that immediately preceded it in source order, so the split inverted the previous adjacency: the paragraph is now a trailer to `audit-wire-and-canary.md` rather than a header to the enumeration. `git log` against the current file shows only `f5e89f4` and a subsequent rename commit (`81bf52a` `docs: collapse spec/ section pages into spec_topics/`); the paragraph itself has not been edited in the new file. The substantive last edit was in `5c8e630`, which extended the item-5 routing to mention `AgentSession.prototype.abort` jointly (rather than the original "covered by step (b) above" wording introduced by `126ff8a`). None of the three commits is a misplacement bug in its own diff — `126ff8a` and `5c8e630` placed the paragraph correctly given the file layout at their time; `f5e89f4`'s split is what cleaved paragraph from enumeration. The defect is therefore a split-induced placement regression, not a load-bearing decision that needs unwinding.

## Solution Space

**Shape:** single

### Recommendation

Move the trailing paragraph from `audit-wire-and-canary.md` into `capability-inventory-items.md`, placing it directly above the numbered enumeration so it acts as that section's per-item-verification preamble (restoring the pre-split adjacency). Concretely:

1. Cut the *"Item 5 (cancellation propagation) is verified jointly … cite a single anchored source"* paragraph from the end of `audit-wire-and-canary.md`.
2. Paste it at the top of `capability-inventory-items.md`, between the `# Capability inventory items` heading and the `1. <a id="sdk-cap-slash-command-registration">` line.
3. Update relative links in the moved paragraph for the new file location: `../binder/binder-model-and-context.md#strict-capability-requirement` and `../overview-and-orientation.md#orientation` are at the same relative depth and need no change; verify the rendered links in the new location still resolve.
4. Leave `inventory-audit-intro.md` untouched — its PIC-15 preamble already names items 1, 2, 3, 4, and 6 as factory-probable; the moved paragraph complements it on the items-5/7 axis from the enumeration page.
5. Re-run the corpus link/anchor validator (`docs/_tools/check_links.py`) to confirm no broken anchors; the paragraph carries no `<a id="…">` of its own so anchor inventory is unchanged.

Edge cases the implementer must watch: the trailing sentence *"name-links each item back to the obligation here so that downstream tests cite a single anchored source"* now refers to the very page it sits on, so its "here" reference is self-consistent at the new location. Do not split the paragraph across the two destination pages; the items-5/7 routing is a unit, and keeping it on the same page as the seven enumerated items preserves the single-source-of-truth property the trailing sentence asserts.

## Consequence

**Severity:** advisory

A reviewer auditing how each of the seven capabilities is verified will read the enumeration in `capability-inventory-items.md` and find no routing for items 5 and 7 there; the routing exists, but in a page whose title and surrounding content disavow any inventory-level concern. An implementer following the existing cross-references reaches the right answer (item 5 → Step 0 (b) plus the `AgentSession.prototype.abort` member in Step 0 (c); item 7 → per-loom load time, `loom/load/binder-model-unresolved`), so the bug is navigational, not semantic. Risk grows over time: a future edit to either capability's verification path that touches only the enumeration page will silently desynchronise from this trailer.
