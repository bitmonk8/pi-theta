## Finding

The *Post-cancel resolution* bullet in `tool-calls.md` (and the normative home it cites, the *swallowing-handler attachment* paragraph in `cancellation.md`) raises a specific JavaScript implementation mechanism to a `MUST`: the runtime "MUST attach [a swallowing handler] to the `execute()` Promise **at construction time**" and the handler "MUST be installed **before the first microtask boundary**", with the rationale that "a lazily-attached `.catch(() => {})` will not catch a rejection already queued for `unhandledRejection`". This is mechanism, not observable behaviour. What an observer (or a conformance test) can actually witness is: (1) the late settlement is discarded ‚Äî no rebind, no second `Err`, no second `RuntimeEvent` (CNCL-1/2/3), and (2) a late rejection never escapes as a Node `unhandledRejection` process event (nor as a diagnostic of any severity).

The "before the first microtask boundary" timing `MUST` has no observable correlate distinct from outcome (2). Any implementation that satisfies "no `unhandledRejection` ever fires" has necessarily attached its suppression early enough; conversely, the only way to detect a too-late attachment is to observe a leaked `unhandledRejection` ‚Äî which is exactly what outcome (2) already forbids. The timing clause therefore cannot be tested independently of the outcome it exists to produce, and it over-constrains: an implementation that suppresses the side channel by some other correct means (e.g. a different attachment point that is still race-free, or process-level filtering) would be non-conformant against the letter of the `MUST` while producing precisely the required observable behaviour.

The mechanism description is genuinely useful as implementer guidance ‚Äî the microtask-ordering trap is subtle and worth recording ‚Äî but it belongs in a non-normative note, not in a normative `MUST` whose conformance cannot be checked without inspecting the runtime's internal Promise wiring. Because `tool-calls.md` defers to `cancellation.md` "for the normative discard rule", demoting the mechanism in only one of the two pages would leave the `MUST` binding via the cross-reference; both sites must move in lockstep.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `2ebe288` ‚Äî "pi-loom spec: resolve \"Post-cancel late Promise settlement: unhandledRejection discard\"" (2026-05-07)

**History:** Before `2ebe288`, the `tool-calls.md` *Post-cancel resolution* bullet was observable-only ("‚Ä¶is discarded ‚Äî no rebinding, no second `Err`, no second `RuntimeEvent`. See Cancellation ‚Äî Race semantics‚Ä¶") and `cancellation.md` carried no swallowing-handler text. `2ebe288` ‚Äî the commit that fixed a real `unhandledRejection` leak ‚Äî introduced the mechanism prescription verbatim into both files: the "swallowing handler the runtime MUST attach ‚Ä¶ at construction time" / "MUST be installed before the first microtask boundary" wording in `tool-calls.md` and the "MUST be attached at the same site that constructs the Promise, before the first microtask boundary" wording in `cancellation.md` (`git log -S` on both phrases returns only this commit as the originating change). The defect is thus an over-correction: the unhandledRejection concern was resolved by mandating a specific JS mechanism rather than the observable suppression property. Later commits (`482098b` splitting the discard into CNCL-1/2/3 anchors, `8362824`/`8ad3d33` reworking adjacent testability MUSTs, the `docs/` relocation) reshaped the surrounding paragraph but preserved the construction-time / first-microtask mechanism `MUST`.

## Solution Space

**Shape:** single

### Recommendation

State the observable contract as the normative requirement and demote the JS mechanism to a non-normative implementation note, in both pages in lockstep.

In `cancellation.md` (the normative home), keep the observable `MUST`s: a late settlement arriving after the checkpoint has surfaced `cause: "cancelled"` is discarded (CNCL-1/2/3) and MUST NOT surface as a Node `unhandledRejection` process event, a second `RuntimeEvent` on the always-log channel, or a diagnostic of any severity ‚Äî regardless of the late rejection's `.message`. Move the implementation detail ‚Äî "attach a swallowing handler at the same site that constructs the Promise, before the first microtask boundary; a lazily-registered `.catch(() => {})` will not catch a rejection already queued for `unhandledRejection`" ‚Äî into a clearly marked non-normative note (or "Implementation guidance") immediately following, framed as the recommended way to achieve the observable requirement rather than as a conformance obligation.

In `tool-calls.md`, reduce the *Post-cancel resolution* bullet to the observable statement (discard per CNCL-1/2/3; never escapes as `unhandledRejection`) plus the existing cross-reference to `cancellation.md`; drop the inline "MUST attach ‚Ä¶ at construction time" / "MUST be installed before the first microtask boundary" mechanism so the bullet no longer re-asserts a normative mechanism that the cited page is also demoting.

Edge cases the editor must preserve: the totality of the discard (an `OOMError`-style late reject is still discarded; no promotion to `loom/runtime/internal-error`), the non-traversal of the discarded value (resource cleanup stays the tool's responsibility), and the symmetric application to the `@`-query provider Promise, the `invoke` callee top-level Promise, and the subagent `AgentSession.abort()` Promise ‚Äî these are observable/scope properties, not mechanism, and stay normative.

## Consequence

**Severity:** advisory

A conformance suite cannot test the "before the first microtask boundary" `MUST` except by white-box inspection of the runtime's Promise-attachment site, which contradicts the spec's elsewhere-stated reliance on the observable `Checkpoint`-seam substrate "without relying on JS microtask scheduling". The over-constraint also makes correct-but-differently-structured implementations nominally non-conformant. Implementers can still build a working runtime ‚Äî the observable requirement is clear ‚Äî so this is a guidance/normative-framing defect rather than a behavioural one.
