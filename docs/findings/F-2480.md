## Finding

The *Runtime construction obligation* clause in `diagnostic-shape.md` (`session_shutdown` `details` shape conventions paragraph) states: "The runtime MUST construct `details.event` as a fresh object containing exactly the pinned fields â€¦ (no `...event` spread, no `Object.assign(..., event, ...)`, no per-key copy loop over `Object.keys(event)`, and no other dynamic-field-set mechanism whose member set is sourced from the SDK-delivered `SessionShutdownEvent` â€¦), with each pinned field assigned by an explicit static property name." This pins *how* the runtime builds the object â€” a set of forbidden JavaScript idioms and a mandated "explicit static property name" assignment style â€” rather than the observable wire result.

The observable obligation that actually matters is already stated in the same paragraph and elsewhere on the page: `details.event` carries **exactly** `{reason}` for `loom/host/session-shutdown-runtime-degraded` and **exactly** `{reason, loom, invocation_id}` for `loom/runtime/cancelled-by-session-shutdown`, with no `SessionShutdownEvent` sibling beyond `reason` appearing on the wire. Under the loom 1.0 SDK pin, `SessionShutdownEvent` carries only `reason`, so every banned mechanism and the mandated explicit-assignment style produce byte-identical wire output. A conformance test built from type-valid V1 `SessionShutdownEvent` values therefore cannot distinguish a spread from an explicit construction; the prohibition only "bites" against a hypothetical future SDK that grows new siblings â€” a case the page already gates behind a required spec edit ("a future SDK addition â€¦ requires a spec edit to this paragraph and to the affected row(s) before it is permitted on the wire"). The mechanism enumeration is thus forward-compat insurance with no falsifiable correlate at the V1 pin, expressed as a normative MUST about internal construction style.

Secondarily, the construction style is a runtime-handler-internal concern that sits more naturally alongside the `session_shutdown` teardown handler and the *Construction-site wrap* / *Hoist obligation* material the page already cedes to Pi Integration Contract â€” Diagnostic-emission isolation, which owns the fallback construction of the same payload. `diagnostic-shape.md` self-describes its scope as owning "only the wire-format nesting convention," yet this clause reaches past the wire shape into how the object is assembled.

## Solution Space

**Shape:** single

### Recommendation

Retain the observable wire-shape MUST as the sole normative obligation in `diagnostic-shape.md`: `details.event` is exactly `{reason}` for `loom/host/session-shutdown-runtime-degraded` and exactly `{reason, loom, invocation_id}` for `loom/runtime/cancelled-by-session-shutdown`, and no `SessionShutdownEvent`-derived sibling beyond `reason` ever appears on the wire (the existing future-addition-requires-spec-edit pin already enforces the closed set).

Demote the construction-method enumeration ("no `...event` spread, no `Object.assign`, no `Object.keys` loop, â€¦ explicit static property name") to a non-normative implementation note, framed as forward-compat rationale: a spread/assign-style construction would silently leak a future SDK sibling onto the wire, so a fresh object with statically named fields is the recommended implementation. Co-locate this note with the runtime `session_shutdown` handler material â€” i.e. either keep it inline as an explicitly non-normative aside or cross-reference Pi Integration Contract â€” Diagnostic-emission isolation, which already owns the *Construction-site wrap* / *Hoist obligation* construction of the same payload â€” rather than asserting it as a wire-shape MUST on a page scoped to the nesting convention.

Edge cases the implementer must watch: the closed-set guarantee is the contract that earns the operator-tooling strict-validation entitlement; if a future reviewer wants a falsifiable test of the no-leak property, the correct correlate is an over-populated mock `SessionShutdownEvent` (one carrying an extra sibling) asserting the extra field is absent from `details.event` â€” that test belongs to the emission leaves, not to the idiom ban.

## Consequence

**Severity:** advisory

A normative MUST prescribes internal construction idioms whose distinction is unobservable at the loom 1.0 SDK pin, so the conformance surface carries an obligation no type-valid V1 test can falsify and an implementer cannot tell which part is the real contract (the wire shape) versus implementation guidance (the idiom ban). The current text is implementable and not wrong; the cost is a misleading normative surface and a small scope creep past the page's stated wire-format-only remit.
