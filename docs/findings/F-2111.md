## Problem

The diagnostic-registry column legend in `diagnostic-shape.md` (`#column-legend`) defines the `Phase` column as exactly one value drawn from a closed set, phrased with an exclusive "or" (`lex` ŌĆ” `parse` ŌĆ” `type` ŌĆ” `load` ŌĆ” or `runtime`); the grammar admits no multi-valued form. The `loom/load/invoke-path-escape` row in `code-registry-load.md` carries two values (`load, runtime`) ŌĆö the only multi-valued `Phase` cell across the four sharded registry tables ŌĆö because the code fires at both load time and at runtime. Because the legend is the normative definition of the column and is silent on multi-valued cells, an implementer (and any conformance assertion) cannot tell whether `load, runtime` is a sanctioned multi-phase form, shorthand for two distinct codes, or an authoring slip.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** c8b8790 ŌĆö pi-loom plan: resolve "invoke-path-escape parse-only TOCTOU" (2026-05-05, Thomas Andersen)
**History:** The `loom/load/invoke-path-escape` row originally carried the single `Phase` value `load`. Commit c8b8790 added the runtime open re-check to its `Trigger` and changed the cell to `load, runtime`. At that same commit the column legend already framed `Phase` as a single value drawn from the set with an exclusive "or", so the divergence was introduced by adding the second phase value to one row without reconciling the legend, and has persisted (across the later `81bf52a` page split and the `a37d4ef` legend relocation) ever since.

## Solution approach

Reconcile the `Phase` column definition in the legend (`#column-legend`) with the `loom/load/invoke-path-escape` row in `code-registry-load.md` so the column's value domain unambiguously covers a cell that names more than one pipeline stage. Either redefine `Phase` to admit one or more values from the closed set, or resolve the row to a single stage; whichever direction is chosen, the legend's normative value-domain statement and the row's cell MUST agree afterward.
