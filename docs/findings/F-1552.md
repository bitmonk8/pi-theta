## Finding

The *Runtime construction obligation* paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` (the closing paragraph of the `## session-shutdown-details-conventions` block, beginning "The runtime MUST construct `details.event` as a fresh object containing exactly the pinned fields…") states its requirement by enumerating forbidden JavaScript constructs — "no `...event` spread, no `Object.assign(..., event, ...)`, no per-key copy loop over `Object.keys(event)`, and no other dynamic-field-set mechanism whose member set is sourced from the SDK-delivered `SessionShutdownEvent`" — and then prescribes the positive construction as "each pinned field assigned by an explicit static property name". The observable property the spec actually needs is that the member set of `details.event` on the wire is *exactly* the set pinned by this paragraph for each code (`{reason}` for `loom/host/session-shutdown-runtime-degraded`; `{reason, loom, invocation_id}` for `loom/runtime/cancelled-by-session-shutdown`) regardless of what fields the SDK's `SessionShutdownEvent` carries today or adds tomorrow. Re-stated behaviourally — "the member set is statically fixed by this paragraph and is not derived from the SDK-delivered `SessionShutdownEvent`" — an implementer is free to choose any construction primitive that satisfies it, and the wire-equality test the closed-`{reason}` strictness pin already entails is the verification surface.

The page placement compounds the prescription issue. `diagnostic-shape.md`'s `## session-shutdown-details-conventions` block declares its own scope as "the wire-format nesting convention; the teardown-handler emission's `try`/`catch` envelope and the bare-`code` fallback on serialiser throw are owned by [Pi Integration Contract — Diagnostic-emission isolation]". The runtime-construction rule is exactly such an emission-side construction obligation: the PIC isolation page already owns the *Construction-site wrap* sub-clause (which catches construction-site throws from this very construction), the *Hoist obligation* clause (which constrains the construction order of the sibling `detailsEventReason` local), and PIC-25/26 (which govern the construction-site `try`/`catch` and self-wrap). The construction obligation is the third sibling of those clauses; homing it on the shape page splits one tightly-coupled MUST set across two files, and the PIC page only reaches it via a cross-reference (`per the [Runtime construction obligation](../diagnostics/diagnostic-shape.md#session-shutdown-details-conventions)`) — an implementer auditing PIC's emission-isolation contract end-to-end has to chase that link out to a different topic page to find a rule that constrains the construction the PIC clauses immediately above it wrap.

The forbidden-construct enumeration ("no `...event` spread, no `Object.assign(..., event, ...)`, no per-key copy loop over `Object.keys(event)`") and the "explicit static property name" positive prescription were present at first appearance, not introduced by a later refactor. The paragraph was carried over to `docs/spec_topics/diagnostics/diagnostic-shape.md` unchanged in content (only the path changed) by `f5e89f4` (2026-06-04) when the 100KB / 30KB size cap split `diagnostics.md` into an index plus child pages. The misplacement therefore predates the split: the rule was authored on the shape page in `52b9101` while the sibling *Construction-site wrap* / *Hoist obligation* clauses landed on the PIC `diagnostic-emission-isolation` page, and the over-prescriptive framing is the form the rule has had since its single authoring commit.

## Solution Space

**Shape:** single

### Recommendation

On `docs/spec_topics/diagnostics/diagnostic-shape.md`, replace the *Runtime construction obligation* paragraph with one behavioural sentence and a forward link, e.g.: "*Runtime construction obligation.* The per-code `details.event` member set is statically fixed by this paragraph — `{reason}` for `loom/host/session-shutdown-runtime-degraded` and `{reason, loom, invocation_id}` for `loom/runtime/cancelled-by-session-shutdown` — and is **not** derived from the SDK-delivered `SessionShutdownEvent`. A future SDK addition (e.g. a hypothetical `event.timestamp`) requires a spec edit to this paragraph and to the affected row(s) before it is permitted on the wire. The construction-site rule that defends this property on the runtime side, together with the construction-site `try`/`catch` and the per-code catch-arm fallback forms, lives at [Pi Integration Contract — Diagnostic-emission isolation, *Runtime construction*](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation)." Keep the runtime-constructed-vs-SDK-derived discrimination (it is what the closed-`{reason}` strictness pin and the *Runtime-constructed sibling carve-out* both key on); drop the forbidden-construct enumeration and the "explicit static property name" prescription.

On `docs/spec_topics/pi-integration-contract/diagnostic-emission-isolation.md`, add a sibling *Runtime construction* clause to the existing *Construction-site wrap* / *Hoist obligation* block — placed immediately before *Construction-site wrap* so the construction it describes precedes the wrap that defends it — stating the rule behaviourally: "*Runtime construction.* The runtime MUST construct `details.event` so the wire member set on each emission equals the per-code set pinned by the [`details` shape conventions for the `session_shutdown` event payload](../diagnostics/diagnostic-shape.md#session-shutdown-details-conventions) — `{reason}` for `loom/host/session-shutdown-runtime-degraded` and `{reason, loom, invocation_id}` for `loom/runtime/cancelled-by-session-shutdown` — independently of the member set the SDK-delivered `SessionShutdownEvent` carries at runtime. A construction primitive whose member set is sourced from the SDK event (e.g. a spread, an `Object.assign(..., event, ...)`, or a `for…in` / `Object.keys(event)` copy loop) MUST NOT be used in a position where it can introduce a `details.event` sibling that is neither named in the per-code set above nor admitted by the runtime-constructed-sibling carve-out for `loom`/`invocation_id`. Verification: a test that emits each row with an SDK fixture carrying an unpinned sibling (e.g. a hypothetical `event.timestamp`) MUST observe that sibling absent from the wire `details.event`."

The relocated clause sits naturally in PIC's emission-isolation block: *Construction-site wrap* already references "the runtime construction of the structured payload's `details.event` field" and lists the constructed shapes per code; the new *Runtime construction* clause makes that construction the explicit MUST on the same page rather than a forward reference. Implementer-relevant edge case: a future runtime-constructed sibling addition (the residual-gap clauses already pin `loom` and `invocation_id` as carve-outs) requires updating both the per-code set on the shape page and the per-code set in *Runtime construction* / *Construction-site wrap* in the same commit, so the wire-equality test the new clause specifies stays in lockstep with the construction-site fallback token shapes. The behavioural verification surface (wire member-set equality against a fixture carrying an unpinned SDK sibling) closes the original prescription's defensive intent without naming a JS construct.

## Consequence

**Severity:** advisory

An implementer who treats the forbidden-construct list as the contract may believe a refactor that uses a frozen helper or an immutable record builder is non-compliant even when it satisfies the behavioural property the rule is defending; the converse — that a hand-assigned-static-property construction that read `Object.keys(event)` once for unrelated logging satisfies the letter of the rule — is also a misread the spec invites. The split home additionally raises the audit cost of the construction-site emission contract: an implementer following PIC's *Construction-site wrap* sub-clause to its sibling construction obligation must traverse to a different topic page to find it, and the obligation's own audit-trail to PIC's failure modes (the catch arm's `"<unreadable>"` substitution, the per-code fallback token shapes) lives on the receiving page rather than co-located with the construction rule.
