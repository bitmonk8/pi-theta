## Finding

The `#session-shutdown-details-conventions` block in `diagnostic-shape.md` is a single, unbroken paragraph of roughly 1,400 words that bundles five distinct, independently-enforceable normative sub-topics for the `session_shutdown` event payload:

1. the `{reason}` field-set closure at loom 1.0.0 (no new `SessionShutdownEvent` siblings without a spec edit);
2. the per-code `details.event.reason` value spaces (the wide `"quit" | "reload" | "new" | "resume" | "fork" | string | "<unreadable>"` union for `loom/runtime/cancelled-by-session-shutdown` versus the closed `{"new","resume","fork"}` set for `loom/host/session-shutdown-runtime-degraded`, which consumers MUST validate per-code, not uniformly);
3. the runtime-constructed sibling carve-out for `details.event.loom` and `details.event.invocation_id` (exempt from the strict-validation entitlement; MUST be present on every structured emission);
4. the presence-entitlement scope that distinguishes structured emissions from the two fallback-path token shapes; and
5. the runtime construction obligation (MUST construct `details.event` as a fresh object with explicit static property names — no `...event` spread, no `Object.assign`, no key-copy loop sourced from the SDK event).

The block carries multiple MUST-bearing rules, yet they are demarcated only by inline italic run-in labels (`*Per-code …*`, `*Runtime-constructed sibling carve-out.*`, `*Presence-entitlement scope …*`, `*Runtime construction obligation.*`) embedded mid-paragraph. The surrounding file uses anchored bold run-in leads as its sub-section device (`**Located-site fields …**`, `**Transient toasts …**`, `**Serialised content format.**`), so a reader or auditor scanning the document's structure sees this entire API/wire-contract specification as a single opaque paragraph behind one lead. The construction obligation in particular — arguably the most consequential MUST in the block, since a spread-style refactor would silently leak future SDK fields onto the wire — is buried at the far end of the paragraph where structure-scanning will not surface it.

This is a wire-contract specification (how implementers construct `details.event.*` objects and how consumers validate them), not a description of a delivery channel. It should read as a headed, navigable section.

The misplacement therefore predates the split: the rule was authored on the shape page in `52b9101` while the sibling *Construction-site wrap* / *Hoist obligation* clauses landed on the PIC `diagnostic-emission-isolation` page, and the over-prescriptive framing is the form the rule has had since its single authoring commit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/diagnostics/diagnostic-shape.md`, split the single `#session-shutdown-details-conventions` paragraph into discrete sub-sections, one per the five normative sub-topics currently demarcated only by inline italic run-in labels. Use the file's existing structural device — anchored **bold run-in leads** (as in `**Located-site fields …**`, `**Transient toasts …**`, `**Serialised content format.**`) — for each sub-section:

1. `{reason}` field-set closure at loom 1.0.0 (no new `SessionShutdownEvent` siblings without a spec edit);
2. per-code `details.event.reason` value spaces (the wide `"quit" | "reload" | "new" | "resume" | "fork" | string | "<unreadable>"` union for `loom/runtime/cancelled-by-session-shutdown` versus the closed `{"new","resume","fork"}` set for `loom/host/session-shutdown-runtime-degraded`, validated per-code, not uniformly);
3. the runtime-constructed sibling carve-out for `details.event.loom` and `details.event.invocation_id`;
4. the presence-entitlement scope distinguishing structured emissions from the two fallback-path token shapes; and
5. the runtime construction obligation (construct `details.event` as a fresh object with explicit static property names — no `...event` spread, no `Object.assign`, no key-copy loop sourced from the SDK event).

Preserve every MUST-bearing rule verbatim; move no content out of the file. Retain the existing `<a id="session-shutdown-details-conventions"></a>` anchor on the lead (field-set-closure) sub-section so the six inbound cross-references in `code-registry-host.md`, `code-registry-runtime.md`, `active-invocation-registry.md`, `diagnostic-emission-isolation.md`, `session-only-degraded-state.md`, and the in-file `located-site-classification` reference all continue to resolve onto the field-set-closure material they cite. Place the construction-obligation sub-section so its MUST surfaces under structure-scanning rather than at the far end of a paragraph.

The file's residual ~37 KB size relative to the 30 KB recommended cap (well under the 100 KB hard cap) is out of scope for this edit and is handled as a separate size-cap pass.

## Consequence

**Severity:** advisory

The normative rules are all present and correct, so a careful full-paragraph read yields a working implementation. The risk is discoverability: an implementer or auditor scanning by document structure can miss one of the buried MUSTs — most critically the "fresh object, no spread/assign" construction obligation, whose violation would silently leak future SDK fields onto the wire under a later refactor — or apply one code's `details.event.reason` value space to the other and over- or under-validate.
