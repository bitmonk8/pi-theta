## Finding

Two teardown-side diagnostics are defined as *advisory only* in the subagent session lifecycle (`PIC-9`): `loom/runtime/subagent-dispose-failure` (raised when `AgentSession.dispose()` throws inside the per-invocation `finally`) and `loom/runtime/internal-error` (raised when the synchronous `session.abort()` call throws during teardown). Both pages are explicit that these diagnostics never alter the invocation result: on a normal-return teardown "the parent still observes the loom's `Ok` final value … the disposal throw never promotes that `Ok` to an `Err`." Nothing was aborted; the loom succeeded.

The runtime-event channel, however, routes these diagnostics through the operator-facing `loom-system-note` channel as **group B** (`details: { diagnostics: Diagnostic[] }`). Group B is enumerated as "Runtime panics — every row of [Diagnostics — `loom/runtime/*`] … **except** the five `console.error`-routed teardown-handler codes." Neither `subagent-dispose-failure` nor `internal-error` is in that five-code exclusion list, so by the literal membership rule both fall into group B. The panic-emission rule (under *Deduplication and lifetime rules*, and the *Per-variant `display`/`content` pairings* matrix) then prescribes, for every group-B diagnostic note, `display: true` with `content` fixed to `"loom /<name> aborted: <message>"` (or `"loom /<name> aborted with internal error: <message>"` for `internal-error`).

The result is a contradiction: a loom that returns `Ok` but hits a `dispose()` or teardown-`abort()` hiccup would surface a `display: true` operator note framed as `"loom /<name> aborted: subagent dispose failed: …"` even though the loom was never aborted. Because every `loom-system-note` (regardless of `display`) is converted to a `user`-role transcript entry by Pi's `convertToLlm` transform, this false "aborted" claim also enters the model context durably on a successful run. The spec pins `display` / `content` for the genuine-panic case but never carves out the surfacing contract for an advisory teardown diagnostic on a non-aborting teardown — the `display` flag, the companion `content`, and the target session for that case are all unpinned, leaving two reasonable implementers free to diverge (emit the misleading "aborted" note, suppress it, or reframe it).

A secondary internal tension corroborates the gap: the always-log set is described as comprising "`loom/runtime/*` **panic codes**" (a narrower phrasing), while the group-B bullet says "**every row** of `loom/runtime/*`." `subagent-dispose-failure` is not a panic code (loom 1.0 has exactly six closed panic sources, and it is not one of them), so the two phrasings disagree on whether it is even in the channel — and if it is, the panic-emission framing is the wrong template for it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `78a2f94` (broad group-B routing + panic-emission "aborted" content established on the relocated runtime-event-channel page), `7238428` (advisory-on-`Ok`-teardown semantics concretised on subagent.md / code-registry-runtime.md)
**History:** The corpus is git-tracked. The panic-emission `"loom /<name> aborted: <message>"` content for group-B diagnostic notes has been present since the spec was split into topic files (`f5e89f4`, 2026-06-04), and `78a2f94` (2026-06-04) relocated the runtime-event channel into its own page carrying the broad group-B membership "Runtime panics — every row of `loom/runtime/*` … except the five `console.error` codes." Independently, `7238428` (2026-06-06, "Teardown-side throw on the Ok path") sharpened the teardown diagnostics on subagent.md and code-registry-runtime.md to be explicitly *advisory only* — adding "on a normal-return teardown the parent still observes the loom's `Ok` final value … the disposal throw never promotes that `Ok` to an `Err`" for both `subagent-dispose-failure` and the teardown-side `internal-error`. That change reasoned about the *invocation result* (no masking of `Ok`) but not about the *operator-facing surfacing channel*, and never reconciled the advisory diagnostics against the broad group-B "aborted" framing on the runtime-event-channel page. Subsequent refinements of the group-A/B partition (`038dab3` "always-log set definition narrower than membership", `2e1888c` "Group B routing self-contradicts", `bf42043` qualifying the QueryError component, `d3b775e` correcting the `RuntimeEvent.kind` comment) repeatedly touched the partition without closing the advisory-teardown framing collision; the narrower "panic codes" phrasing in the always-log sentence versus the "every row" phrasing in the group-B bullet is a surviving residue of that incomplete reconciliation. The defect is the unresolved interaction between the two independently-reasonable rules rather than a single introducing edit.

## Solution Space

**Shape:** multiple

### Option A — Carve out the advisory-teardown surfacing contract within group B

**Approach:** Keep `subagent-dispose-failure` and the teardown-side `abort()`-throw `internal-error` on the `loom-system-note` channel, but add a normative carve-out to the panic-emission rule (and a matching row in the *Per-variant `display`/`content` pairings* matrix): these two advisory teardown diagnostics MUST NOT use the panic `"loom /<name> aborted: <message>"` content. Instead they emit with `display: false` and a non-"aborted" advisory framing (e.g. `content: ""` with the diagnostic carried structurally, or a distinct `"loom /<name> teardown diagnostic: <message>"` template), regardless of teardown trigger — because these diagnostics never abort the loom themselves on any trigger (on an `Err`/panic teardown the primary error already emits its own note; on a normal-return teardown nothing aborted). Pin the target session as the spawned subagent's transcript / the same `sendSystemNote` path the lifecycle already uses. Reconcile the always-log "panic codes" phrasing with the group-B "every row" phrasing so both agree these two codes are present-but-advisory.

**Spec edits:** in `runtime-event-channel.md`, add the carve-out to the panic-emission rule and a matrix row; in `code-registry-runtime.md`, state the `display`/`content` disposition on the two registry rows; in `subagent.md` `PIC-9`, cross-reference the carve-out where the advisory rule is stated.

**Pros:** lowest blast radius; preserves the operator/structured-payload visibility of teardown failures; keys the fix on the diagnostic identity (always advisory) rather than on per-trigger state.

**Cons:** adds a third disposition to the group-B routing rule, which already carries the five-code `console.error` exclusion; renderers/tests must handle the new non-panic group-B framing.

**Risks:** if the carve-out is keyed on teardown trigger instead of on diagnostic identity, the `Err`/panic-teardown case is left ambiguous again.

### Option B — Exclude advisory teardown diagnostics from the loom-system-note channel

**Approach:** Treat `subagent-dispose-failure` and the teardown-side `abort()`-throw `internal-error` like the existing five `console.error`-routed teardown codes: exclude them from the always-log set / group-B `loom-system-note` routing entirely and emit them only through the standard persistent-diagnostics channel. Extend the group-B exclusion list accordingly and align the always-log "panic codes" phrasing.

**Spec edits:** in `runtime-event-channel.md`, add the two codes to the group-B exclusion list and reconcile the always-log sentence; in `code-registry-runtime.md`, pin the persistent-diagnostics channel on the two rows; in `subagent.md` `PIC-9`, cross-reference.

**Pros:** removes the collision at the root — no group-B "aborted" framing can ever apply; keeps `loom-system-note` reserved for genuine top-level panics and always-log failures.

**Cons:** the teardown-side `internal-error` row also covers non-teardown internal errors that legitimately do abort and belong in group B, so the exclusion must be scoped to the teardown-`abort()` sub-case only, not the whole `internal-error` code — a finer split that is harder to express on a single registry row.

**Risks:** advisory teardown failures lose their operator-visible `loom-system-note` surface (and LLM-context entry), reducing operator visibility of a real (if non-fatal) infra hiccup.

### Recommendation

Adopt **Option A**. Carve the two advisory teardown diagnostics out of the panic-emission "aborted" framing and emit them with `display: false` and a non-"aborted" advisory `content`, keyed on the diagnostic identity (these codes are advisory on *every* teardown trigger, not only on normal return). This fixes the false "aborted" claim and its `convertToLlm` context pollution while preserving operator/structured visibility of the teardown failure, and avoids Option B's awkward sub-case split of the `internal-error` row. Implementer edge cases: (1) apply the carve-out uniformly across normal-return, `Err`, panic, and cancellation teardown triggers — the primary error (when one exists) emits its own note independently; (2) ensure the always-log "panic codes" wording and the group-B "every row" wording are reconciled in the same edit so the two codes are unambiguously present-but-advisory; (3) the renderer's existing `display === false` short-circuit must tolerate the carve-out's `content` value.

## Consequence

**Severity:** correctness

On a successful (`Ok`) subagent run with a `dispose()` / teardown-`abort()` hiccup, the literal rules produce a `display: true` operator note asserting the loom was "aborted," which is false, and — via `convertToLlm` — injects that false claim into the model context durably. Two reasonable implementers diverge: one emits the misleading "aborted" note, another suppresses or reframes it, because the surfacing contract for advisory teardown diagnostics on a non-aborting teardown is unpinned.
