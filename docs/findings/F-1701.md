## Problem

The version-bump editorial-review checklist in `version-bump-step2.md` frames its ~38 SHOULD-level host-presupposition items `(f)`–`(ap)` uniformly; the only structural distinction the list carries is the MUST `(a)`–`(e)` / SHOULD `(f)`–`(ap)` cut. The SHOULD set is heterogeneous in *failure visibility*: some items regress loudly (load-time or runtime `TypeError`), while others — for example `(h)` `AgentMessage[]` ordering, `(ag)` leading-`UserMessage`, `(ab)` `complete()` `options.signal`, and `(ad)` abort-flip — regress silently with no SDK signal, corrupting transcripts, cancellation, or turn-grouping. The checklist surface presents these flat, with no tier annotation or audit-priority subgroup, so a reviewer working sequentially under time pressure has no cue that the silent-failure items deserve disproportionate scrutiny. Because the preamble already permits a conformant `not-performed` outcome on any `(f)`–`(ap)` item, a contributor can skip the highest-cost items with no spec-level signal that doing so is more consequential than skipping a loud-fail item.

**History:** The checklist was built incrementally: 79fa3b9 first introduced the "SHOULD re-confirm … via editorial review" boilerplate when checklist item (k) was coined; 803106d added the authoritative-source / known-fragile-evidence framing across the existing presupposition paragraphs; subsequent commits (db69f2b is the most recent example, adding (ao)) extended the list to its current ~42 items, each new item reusing the same uniform-SHOULD shape. No commit ever introduced a within-SHOULD tier or audit-priority annotation; the MUST(a–e) / SHOULD(f–ap) cut is the only structural distinction the list has ever carried. The defect — uniform weighting across the SHOULD subset — is therefore not localised to any single commit but emerged as each addition preserved the homogeneous shape.

## Solution approach

In `version-bump-step2.md`, add a failure-visibility tier to each `(f)`–`(ap)` item that distinguishes silent-wrong-behaviour items from loud-fail items, reading each item's tier off the failure mode its body already names. Add a paragraph to the step-2 preamble defining the two tiers and the reviewer attention guidance they encode. Extend the existing future-presupposition catch-all clause so a newly-added checklist item also carries a tier.

## Solution constraints

- The tier is editorial-convention metadata only: it MUST NOT change the per-item `pass` / `fail` / `N/A` / `not-performed` recording shape, MUST NOT make any `(f)`–`(ap)` `fail` bump-blocking (only MUST items `(a)`–`(e)` block), and `not-performed` on any item — including a silent-wrong-behaviour item — MUST remain conformant.
