## Finding

The contributor version-bump checklist in `version-bump-step2.md` enumerates ~42 host-behaviour presupposition items `(a)`–`(ap)` against the loom 1.0 Pi-SDK pin. The list carries one coarse cut — items `(a)`–`(e)` are MUST-level (a `fail` blocks the bump) and items `(f)`–`(ap)` are "SHOULD-audit-but-MUST-record" (a `fail` or `not-performed` is conformant per the V18c exit gate) — but the ~38 SHOULD-level items are all framed identically: each carries a "SHOULD re-confirm … via editorial review" obligation and the same boilerplate escalation clause, with no further differentiation.

The SHOULD-level set is heterogeneous in *failure visibility*, and that heterogeneity is what a bump reviewer needs in order to allocate attention under realistic time pressure. A loud regression — for example, a renamed `register*` member, a dropped `AgentSession` method, or an `engines.node` floor move — generally surfaces at load time or as an immediate runtime `TypeError` whether or not the auditor noticed it. The dangerous SHOULD-level items are the *silent-wrong-behaviour* ones: an `AgentMessage[]` reordering ((h)) corrupts every transcript renderer and binder turn-grouping site with no SDK signal; a leading-`UserMessage` invariant break ((ag)) makes turn-grouping non-total; a slash-handler promise-lifecycle change ((g), (j), (v), (ad)) silently degrades cancellation and hang-handling; a `pi.on` per-session origin marker ((ah)) invalidates PIC-18's no-detach argument; a `complete()` `options.signal` regression ((ab)) leaves cancelled binder calls running to completion. The spec itself flags these silent-failure modes inline in each item's body, but the checklist surface presents them flat, with no tier annotation or "audit-these-first" subgroup, so a reviewer working through the list sequentially has no signal that (h), (ag), (ab), (ad) deserve more care than (n) or (p).

The current shape is also self-undermining at scale: the preamble already permits a `not-performed` outcome on any `(f)`–`(ap)` item as conformant, so a contributor who under-prioritises silently-wrong items has both the latitude to skip them and no spec-level cue that doing so would be more consequential than skipping a loud-fail item.

**History:** The checklist was built incrementally: 79fa3b9 first introduced the "SHOULD re-confirm … via editorial review" boilerplate when checklist item (k) was coined; 803106d added the authoritative-source / known-fragile-evidence framing across the existing presupposition paragraphs; subsequent commits (db69f2b is the most recent example, adding (ao)) extended the list to its current ~42 items, each new item reusing the same uniform-SHOULD shape. No commit ever introduced a within-SHOULD tier or audit-priority annotation; the MUST(a–e) / SHOULD(f–ap) cut is the only structural distinction the list has ever carried. The defect — uniform weighting across the SHOULD subset — is therefore not localised to any single commit but emerged as each addition preserved the homogeneous shape.

## Solution Space

**Shape:** single

### Recommendation

Add a one-line `**Audit priority:** silent-wrong-behaviour | loud-fail` tag at the head of each `(f)`–`(ap)` item body, and a paragraph in the step-2 preamble defining the two tiers and the reviewer guidance they encode (silent-wrong-behaviour: audit first, never record `not-performed` without a written rationale; loud-fail: load-time / runtime-`TypeError` failure surface, lower attention burden, `not-performed` remains conformant).

Tag assignments (read off each item's existing body, where the failure mode is already named):

- `silent-wrong-behaviour`: (h) `AgentMessage[]` chronological ordering, (ag) `AgentMessage[]` leading-`UserMessage` guarantee, (g) / (j) / (v) / (ad) slash-handler-promise lifecycle and turn-lifecycle event delivery, (ab) `complete()` `options.signal` honouring, (ah) `pi.on` process-global delivery + no per-session origin marker, (ak) user-visible streaming ordering, (al) `ctx.waitForIdle()` no-rejection, (ac) driven-turn session-commit ordering, (am) `tools`-allowlist suppression, (an) parallel-tool-batch delivery, (af) `complete()` `onResponse` invocation, (aj) `SessionManager.inMemory` transcript discard, (s) settings write-back key preservation, (r) `convertToLlm` LLM-context conversion, (ao) `pi.registerFlag('loom', …)` overlap + flag-value-slot behaviour, (ap) `pi.registerTool` registration persistence, (k) model-registry population, (l) `pi.getCommands()` completeness, (i) provider overflow-signature wording, (u) `complete()` forced-tool behaviour, (aa) `complete()` retry-timing, (x) teardown-vs-session-swap reason partition, (w) factory-time `cwd == project root`, (ai) interactive-session slash dispatch, (m) prompt-mode / subagent-mode transport-failure fields, (ae) per-extension privilege-scoping facet (absence), (q) pre-bind throw-closure / renderer-resolution.
- `loud-fail`: (n) `AgentSession` member surface (rename → runtime `TypeError` at subagent spawn), (o) subagent-spawn satellite types (rename → runtime error at spawn), (p) `estimateTokens` export (rename → runtime error at `PiTokenEstimator` adapter), (t) `ExtensionRuntime.invalidate(...)` (rename → runtime error mid-teardown), (y) `AgentSession.dispose()` mid-`abort()` safety (regression → runtime hang at teardown), (z) `pi.registerTool` post-startup acceptance (regression → `loom/runtime/registry-swap-failed` on first content edit), (f) session-binding contract (covered by other operator-visible regressions).

The tier is editorial-convention metadata only, on the same footing as the existing escalation clause: it does not change the per-item `pass` / `fail` / `N/A` / `not-performed` recording shape, and it does not change whether a `fail` is bump-blocking (only MUST items `(a)`–`(e)` block, unchanged). It only orders the reviewer's attention and discourages `not-performed` on the high-cost subgroup without a rationale. V18c's **Adds** and **Ships when** sentences gain one parenthetical clause referencing the new tier; no new build-time gate is introduced.

Edge cases the implementer must watch:
- The `Audit priority` tag is per-item metadata, not a normative obligation in its own right — keep its level distinct from the existing `SHOULD` framing so the tier is not misread as adding a second MUST surface above (a)–(e).
- The preamble paragraph defining the two tiers must explicitly state that `not-performed` on a `silent-wrong-behaviour` item remains *conformant* (otherwise the tier silently smuggles in a new bump-blocker, which is out of scope for this finding).
- The catch-all rule that any future presupposition added to the page MUST be added to this checklist in the same edit needs a co-edit clause: the new item also carries an `Audit priority` tag.

## Consequence

**Severity:** advisory

A bump reviewer working sequentially through ~38 uniformly-weighted SHOULD items has no signal that a small subset (the silent-wrong-behaviour items) deserves disproportionate scrutiny and is the worst place to record `not-performed`. Implementation can still proceed and the bump procedure still runs; the cost is realised as deferred regressions that escape into a shipped Pi minor and only surface downstream as misrendered transcripts, missed cancellations, or hung loops.
