## Finding

The `loom/host/session-shutdown-*` rows in `diagnostics/code-registry-host.md` pin emission timing relative to the `session_shutdown` teardown sub-steps — "*before* sub-step 1", "*between sub-steps 4 and 5*", the per-sub-step `details.step ∈ {1,3,4,5}` closed-set, and the per-call-site `details.call` labels. The five-sub-step sequence and the per-call-site sets these rows depend on are defined normatively in `pi-integration-contract/session-shutdown-semantics.md` (the fixed-order list of sub-steps 1–5 and the **Per-step isolation** paragraph, which enumerates the closed `details.call` set per sub-step). That normative content exists and is internally consistent, and the call-site labels are additionally restated in-shard, so an implementer is not behaviourally blocked.

The defect is that the cross-references from the diagnostics cluster that pin these seams do not resolve to those definitions. Several links point at the bare cluster index page `../pi-integration-contract.md` while their link text names a section — "**Per-step isolation** rule", "Extension entry point", "**idempotent** clause", "System notes" — that no longer exists on that page. `pi-integration-contract.md` is now a pure table-of-contents (a list of child-topic links with no prose); the named sections live in the child topic files (`session-shutdown-semantics.md`, `extension-bootstrap-and-per-loom.md`, `drain-state-contract.md`, etc.). An implementer following "[PIC's **Per-step isolation** rule](../pi-integration-contract.md)" or "[Pi Integration Contract — Extension entry point](../pi-integration-contract.md)" lands on a contents list and must hunt across thirty child files to find the normative paragraph the row claims to defer to.

The affected references include, at minimum: in `code-registry-host.md`, the `session-shutdown-runtime-degraded` row's "Per-step isolation rule" and "idempotent clause" links, and the "Extension entry point" links repeated across all four `session-shutdown-*` rows; in `diagnostic-shape.md`, the "System notes" / "Renderer registration" links; and in `placeholder-rendering-b.md`, the "Extension-bootstrap SDK failures" link. Each resolves to the index rather than to the owning topic-file anchor.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap on spec set")
**History:** `git log -S '**Per-step isolation** rule](../pi-integration-contract.md)'` over `docs/spec_topics/diagnostics/code-registry-host.md` returns `f5e89f4` as the sole introducing commit, and the link is unchanged since. `git show --stat f5e89f4` shows that commit split the 576KB monolithic `pi-integration-contract.md` into an index + 30 children and the 152KB `diagnostics.md` into an index + 7 children (including `code-registry-host.md`, created at +12 lines in that commit). Before the split, `pi-integration-contract.md` was the full page that contained the "Per-step isolation", "Extension entry point", "idempotent", and "System notes" sections, so the sibling links resolved. The split relocated that prose into child topic files but left the diagnostics-cluster links pointing at the now-emptied index. A later commit `c3de47a` ("relocate universal session_shutdown teardown to session-shutdown-semantics.md; sweep spec + plan citations") swept some citations but did not touch these anchorless index links.

## Solution Space

**Shape:** single

### Recommendation

Repoint each anchorless `](../pi-integration-contract.md)` reference in the diagnostics cluster to the child topic file (plus anchor) that now owns the named section, so every "sub-step N" / teardown-seam reference resolves to exactly one normative definition:

- "**Per-step isolation** rule" / "Per-step isolation paragraph" → `../pi-integration-contract/session-shutdown-semantics.md`. Add an `<a id="per-step-isolation"></a>` anchor on the **Per-step isolation** paragraph (none exists today) and link to it; the same file's fixed-order list is the canonical home of the five sub-steps and the closed per-`details.step` `details.call` set.
- "Extension entry point" → `../pi-integration-contract/extension-bootstrap-and-per-loom.md#extension-entry-point`.
- "**idempotent** clause" → `../pi-integration-contract/drain-state-contract.md#idempotent-handler-clause`.
- "System notes" / "Renderer registration" (in `diagnostic-shape.md`) → the topic file that owns the system-note renderer-registration prose; add a stable anchor there if absent and link to it.
- "Extension-bootstrap SDK failures" (in `placeholder-rendering-b.md`) → `../pi-integration-contract/extension-bootstrap-and-per-loom.md` at its bootstrap-failure anchor.

Leave the in-shard restatement of the `details.step` / `details.call` closed sets as the implementer-facing copy, but ensure each defers (via the corrected link) to the `session-shutdown-semantics.md` **Per-step isolation** paragraph as the single source of truth, matching the deferral already asserted in the row prose. Edge case: `pi-integration-contract.md` is a pure index — do not add section anchors to it; the fix is to retarget the links to the child files, not to re-add prose to the index.

## Consequence

**Severity:** advisory

A maintainer or implementer following the teardown-emission cross-references reaches a table-of-contents page on which the named section does not appear, forcing a manual search across the PIC child files for the normative sub-step sequence and per-call-site set. Behaviour does not diverge — the call-site labels are restated in-shard and the normative source exists — so this degrades navigability and single-source-of-truth traceability rather than implementability.
