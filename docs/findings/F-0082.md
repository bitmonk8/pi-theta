## Problem

`V11b` implements the 8000-token / 20-turn whole-turn truncation walk via `TokenEstimator`, and `V11b-T` drives that walk with a `FakeTokenEstimator` (PIC-16) stub. The `TokenEstimator` host seam is owned by `V8e`, but neither leaf declares `V8e` in Deps (`V11b`: `V11b-T, V11a, V9i`; `V11b-T`: `V11a, V9i`), and no transitive Deps path reaches it. A Deps-order scheduler can therefore schedule `V11b-T` / `V11b` before `V8e` introduces the seam, leaving the truncation tests referencing a `TokenEstimator` / `FakeTokenEstimator` surface that does not yet exist (compile-time red).

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `c6a664e` (pi-loom plan: build/update plan for spec.md + review)
**History:** At the initial plan build `c6a664e`, `V11b` already named `TokenEstimator` in its Adds and carried Deps `V11b-T, V11a, V9i`, while `V11b-T` carried Deps `V11a, V9i` ŌĆö neither declared the seam-owning leaf. The `V8e` seam file did not yet exist at `c6a664e`; the later seam-split commit `cb8737a` ("Systemic leaf over-bundling across the plan corpus") moved `TokenEstimator` ownership into the newly-created `V8e` but did not add the missing Deps edge. The missing-dependency defect therefore predates `V8e`'s creation and merely re-targeted onto `V8e` when ownership was split out.

## Solution approach

Add `V8e` to the **Deps** field of both `docs/plan_topics/V11b-bind-context-transcript.md` and `docs/plan_topics/V11b-T-bind-context-transcript.md` ŌĆö `V8e` is the leaf that adds the `TokenEstimator` host seam (PIC-16) the truncation walk and `FakeTokenEstimator` stub consume. Mirror the resolution applied to the sibling `Clock` (V8d) seam findings on `V10b` / `V10c`.
