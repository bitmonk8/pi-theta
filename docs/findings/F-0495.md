## Finding

`V2a` ("Type grammar and loom literal sublanguage") closes two distinct parsing surfaces in a single leaf. Its `Adds.` names (i) the type-expression parser plus `array<T>` type-sink resolution, witnessed by `loom/parse/generic-arity-mismatch`, `loom/parse/void-in-non-return-position`, `loom/parse/result-in-schema-position`, and `loom/parse/array-no-common-type`; and (ii) the loom literal sublanguage (primitive / named-value / array / object literals with the full-field requirement and is-literal check), witnessed by `loom/parse/default-not-literal`, `loom/parse/tool-arg-not-literal`, and `loom/parse/missing-object-field`. The paired `V2a-T` carries the same two code families.

These are separable: the literal sublanguage is parsed and validated against the type grammar (the is-literal and full-field checks reference declared/inline-object types), so it depends on the type grammar, but the type grammar ships and is tested without it. That asymmetry is exactly the "smallest feature that can ship and be tested independently" granularity `conventions.md` §"Vertical slices" sets for a leaf. The corpus has already split sibling bundled leaves on the same lens — `V15c` (import resolution/diagnostics vs export-visibility) and `V15h` (two orthogonal failure modes) — so the split is the corpus-consistent resolution rather than a novel restructuring.

This is a minor (NIT) organisation defect, not a correctness gap: every code in both families is tested and gated today. The cost of leaving it bundled is that a contributor tracing a literal-sublanguage failure (e.g. `default-not-literal`) opens a leaf that also owns unrelated type-grammar machinery.

## Solution Space

**Shape:** single

### Recommendation

Extract the loom literal sublanguage into its own vertical leaf, keeping `V2a` as the type-grammar leaf so existing consumer `Deps.` need not change:

- In `docs/plan_topics/V2a-type-grammar.md`, reduce `Adds.` to the type-expression parser and `array<T>` type-sink resolution, and keep in `Tests.` only `loom/parse/generic-arity-mismatch`, `loom/parse/void-in-non-return-position`, `loom/parse/result-in-schema-position`, and `loom/parse/array-no-common-type`. Retitle the leaf to drop "loom literal sublanguage". Mirror the same trimming in `docs/plan_topics/V2a-T-type-grammar.md`.
- Create a new implementation leaf (ID `<new>`) plus its `-T` partner for the loom literal sublanguage, owning `Adds.` of the primitive / named-value / array / bare- and named-object literals with the full-field requirement and is-literal check, and `Tests.` of `loom/parse/default-not-literal`, `loom/parse/tool-arg-not-literal`, and `loom/parse/missing-object-field`. Set the new leaf's `Deps.` to its `-T` partner plus `V2a` (the literal checks resolve against the type grammar). Link the new leaf under §"V2 — Type system and values" in `docs/plan.md`.
- Add the new leaf to `docs/plan_topics/H5b-warn-only-canary.md`'s `Deps.` per the *Transitive-completeness plan-maintenance* clause in `conventions.md` §REQ-ID discipline, because the extracted leaf carries `loom/...` diagnostic-registry codes with their own asserting tests.
- In `docs/plan_topics/coverage-matrix.md`, determine from `grammar.md` whether the three literal-sublanguage codes close a distinct facet of the `cka-1` (`grammar.md` GRAM) obligation area; if so, add the new leaf to the `cka-1` closing-leaf cell alongside `V2a` so the facet has its own closing leaf for the split-row joint-completeness sweep (residue-inspection item 7). The per-cell at-least-one gate (`H5d`) stays green either way; this addition is for facet completeness.

Watch: the new leaf and its `-T` partner must each list `<new>-T` / `V2a` (respectively) in `Deps.` so the pairing and type-grammar dependency are explicit; the literal sublanguage MUST NOT be made a dependency of `V2a` (the edge runs literal-sublanguage → type-grammar, not the reverse).

## Consequence

**Severity:** advisory

If shipped unfixed, nothing breaks mechanically — both code families are tested and gated. A contributor working a literal-sublanguage parse failure must read a leaf that also owns unrelated type-grammar parsing, a minor review-surface tax. This is the same atomicity lens already applied to split `V15c` and `V15h`.
