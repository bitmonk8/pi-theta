## Finding

`V18b` introduces a fail-closed build-time gate over the whole audited source tree (`src/**/*.ts` minus the enumerated exclusions) and, in the same commit, sweeps every existing off-inventory `src/` Pi-side reference — deleting, promoting, exempting, or rewriting each — so the audit lands green on `main`. A walker false positive, or a wrong rewrite during the sweep, reddens the entire `npm test` suite for every contributor on the shared CI substrate until the commit is corrected or reverted. The leaf documents no recovery path and no blast-radius statement for this event.

The fail-closed, land-green, single-commit-sweep shape is spec-mandated, not a plan choice: `inventory-audit-intro.md` §Inventory-closure audit requires the introducing commit to include the source-tree sweep "in the same commit" and explicitly forbids a "land red, fix incrementally" / staged-rollout gate. A warn-only canary stage mirroring `H5b` would therefore contradict the spec and is not the fix. The genuine gap is that two sibling gate leaves carrying comparable or smaller blast radius do annotate their recovery: `H6a` carries an explicit **Rollback** block ("single revert … returns to seeded-fixture footing … no data, schema, or registry migration to unwind"), and `V18d` carries an explicit **Blast radius** block plus a one-commit revert path. `V18b` — whose botched landing reddens CI for everyone — carries neither, leaving a contributor who hits a sweep false positive without the stated recovery the sibling leaves provide.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V18b-inventory-audit.md`, add a recovery annotation and a blast-radius annotation to the **Adds** prose, mirroring `H6a`'s **Rollback** block and `V18d`'s **Blast radius** block:

- A revert note stating that the gate and its same-commit `src/` sweep are reverted together by reverting the single introducing commit, and that there is no data, schema, or registry migration to unwind, so the commit revert is the complete recovery step.
- A blast-radius note stating that, because the audit is fail-closed over the whole audited source tree and the sweep edits `src/` in the same commit, a walker false positive or a wrong sweep rewrite reddens the entire `npm test` suite for every contributor on the shared CI substrate until the commit is corrected or reverted.

Do not add a warn-only canary stage and do not frame the fail-closed landing as a defect to soften: `inventory-audit-intro.md` §Inventory-closure audit mandates the same-commit land-green sweep and forbids a staged "land red, fix incrementally" rollout, so the annotation must record that V18b ships fail-closed by spec design with the gate-commit revert as recovery. The spec is read-only for this fix; the edit is confined to the V18b leaf.

## Consequence

**Severity:** advisory

A sweep false positive or wrong rewrite in the V18b introducing commit reddens the whole `npm test` suite for every contributor, and the leaf states no recovery path, so a contributor who hits it has no documented recourse and no blast-radius warning to set expectations — unlike the sibling `H6a` and `V18d` gate leaves. The leaf remains pickable and implementable; this is an operability and cross-leaf-consistency gap, not a behavioural ambiguity.
