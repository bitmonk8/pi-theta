## Finding

`docs/spec_topics/diagnostics/placeholder-rendering-b.md` §6 ("Underlying-error coercion") pins a two-step recipe for reducing a caught thrown value `v` to a string:

> 1. when `v` is an object whose `.message` is a string, the underlying string is that `.message`;
> 2. otherwise the underlying string is `String(v)`, or the literal text `<unreadable>` when the `String(v)` coercion itself throws.

Step 1 reads `v.message` to test whether it is a string. JavaScript permits arbitrary side-effects on that read: a thrown value whose own prototype defines `message` as a getter that throws (or a `Proxy` whose `get` trap throws on the `"message"` key) makes the expression `v.message` itself throw before step 2 is reached. The `<unreadable>` fallback is scoped textually to "when the `String(v) coercion` itself throws", so the recipe currently leaves the step-1 throw uncaught: the caller (a diagnostic-message renderer or a producer site populating `details.error`, `details.cause`, `hint`, `message`, or `CodeToolError.message`) ends up propagating a synchronous exception out of what was supposed to be a total coercion. Two reasonable implementers will diverge: one wraps the whole inspection and renders `<unreadable>`; another implements the recipe literally and crashes the renderer on a hostile thrown value.

The same spec already establishes elsewhere that property-access throws are a real, contemplated hazard for this exact shape of read — PIC's **Unknown-reason rule** in `docs/spec_topics/pi-integration-contract/unknown-reason-rule.md` requires the `event.reason` read to be wrapped in `try`/`catch` "so a property-access throw (e.g., a getter that throws) is treated as an unknown reason rather than escaping the handler". The §6 coercion is the only sibling host-value-inspection recipe that does not extend the same guard to its property read, so the gap is local to this paragraph rather than an unintentional cross-cutting omission.

The textual scoping of the `<unreadable>` fallback to "the `String(v)` coercion itself throws" — leaving the step-1 `v.message` read unguarded — is therefore present in the very revision that introduced the paragraph, not a later regression. The Unknown-reason rule sibling in `pi-integration-contract/unknown-reason-rule.md`, which already treats a getter-throw on `event.reason` as a fallback case, predates the §6 recipe and was not generalised when §6 was authored.

## Solution Space

**Shape:** single

### Recommendation

Restructure the §6 coercion so the entire inspection of `v` — both the `v.message` access in step 1 and the `String(v)` call in step 2 — falls under one guard, and any synchronous throw from either step yields the literal text `<unreadable>`. Concretely, replace the current two-bullet recipe with prose of the form:

> The caught value `v` is coerced to the underlying string by the following total recipe; any synchronous throw at any step (including a throwing `.message` getter, a `Proxy` `get` trap that throws, or a throw from `String(v)`) yields the literal text `<unreadable>`:
>
> 1. when `v` is an object whose `.message` is a string, the underlying string is that `.message`;
> 2. otherwise the underlying string is `String(v)`.

Edge cases the implementer must watch:

- The guard covers the *read* of `v.message`, the *typeof check* on the read result, and the `String(v)` call — all three are potential throw sites and all three resolve to `<unreadable>`. A reference implementation wraps the whole recipe in a single `try`/`catch` rather than per-step `try`/`catch`es; either is conformant as long as every synchronous throw produces `<unreadable>`.
- The §6 `<unreadable>` literal is already overloaded with §7's three producing paths for `<reason>`; adding a "step-1 throw" producer does not require a new sentinel or any change to that disambiguation discussion (the existing teardown-diagnostic and `<list>`-membership signals continue to discriminate `<reason>`'s paths; the §6 `<unreadable>` for `<error.message>` / `<dispose error first line>` / §8 caught-throw placeholders has only ever needed to mean "the host value could not be reduced to a string").
- The `<original content first line>` parenthetical ("binds a string by construction and skips this coercion") is unaffected and stays as-is.
- The §8 reference into §6's `[underlying-error coercion](#underlying-error-coercion)` anchor needs no change — the anchor and the recipe-by-name continue to mean the same thing; only the recipe's *body* widens.

## Consequence

**Severity:** correctness

A thrown value with a throwing `.message` getter (or `Proxy.get` trap) makes step 1 throw before step 2's `<unreadable>` fallback can catch it. Conforming implementers will diverge — some will render `<unreadable>`, others will propagate a `TypeError` (or arbitrary host throw) out of the diagnostic renderer — defeating the property §6 was added to enforce: that "a non-Error throw never yields … a synchronous `TypeError`". A single hostile thrown value can take down the diagnostic path that exists to report it.
