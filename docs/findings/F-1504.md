## Problem

The §6 `#underlying-error-coercion` recipe scopes its `<unreadable>` fallback to "when the `String(v)` coercion itself throws", but step 1 first reads `v.message` to test its type. That read can throw on its own — a thrown value whose `.message` is a throwing getter, or a `Proxy` whose `get` trap throws on `"message"` — before step 2's fallback can apply. The recipe therefore leaves the step-1 throw uncaught, propagating a synchronous exception out of a coercion that is supposed to be total, and defeating the §6 property that "a non-Error throw never yields … a synchronous `TypeError`". PIC's Unknown-reason rule (`docs/spec_topics/pi-integration-contract/unknown-reason-rule.md`) already wraps the analogous `event.reason` property read; §6 is the only sibling host-value-inspection recipe that does not extend the same guard to its property access.

## Issue introduction

**Verdict:** present-since-inception

**Introducing commits:** bf850929 (`pi-loom spec: resolve "non-Error-throw coercion + session_start pi.getCommands() failure contract"`, 2026-06-06)

**History:** `git log -S "underlying-error coercion" -- docs/spec_topics/diagnostics/placeholder-rendering-b.md` returns a single commit, bf850929, which both anchored `<a id="underlying-error-coercion"></a>` and authored the two-step recipe (`when v is an object whose .message is a string …; otherwise … String(v), or the literal text <unreadable> when the String(v) coercion itself throws`). The textual scoping of the `<unreadable>` fallback to "the `String(v)` coercion itself throws" — leaving the step-1 `v.message` read unguarded — is therefore present in the very revision that introduced the paragraph, not a later regression. The Unknown-reason rule sibling in `pi-integration-contract/unknown-reason-rule.md`, which already treats a getter-throw on `event.reason` as a fallback case, predates the §6 recipe and was not generalised when §6 was authored.

## Solution approach

Rewrite the §6 `#underlying-error-coercion` recipe so the `<unreadable>` guard covers the entire inspection of `v` — the step-1 `.message` read and type check as well as the step-2 `String(v)` call — and any synchronous throw at any step yields the literal text `<unreadable>`.
