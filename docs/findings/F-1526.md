## Finding

[BNDR-8](../../docs/spec_topics/binder/binder-model-and-context.md#bndr-8) names the "canonical no-whitespace JSON serialisation" as the sole normative source for every JSON the transcript renderer emits — both `[tool-call …]` `<args-json>` and the `toolResult` non-text content blocks of rule 4. The pin says three things: no whitespace; object keys in ascending Unicode code-point order at every nesting level; array element order preserved verbatim. It is silent on two further byte-shaped concerns the *MUST-reproduce-exactly* BNDR-7 oracles depend on:

- **String escaping.** Which Unicode scalars are emitted as `\uXXXX` escapes vs. literal UTF-8 bytes (control chars U+0000–U+001F, U+007F, the JSON-mandatory `\"` / `\\`, lone surrogates, characters above U+FFFF that must be encoded as a UTF-16 surrogate pair), and which `\uXXXX` form is canonical (lowercase vs. uppercase hex), is undefined. Two host implementations of `JSON.stringify` (or a polyfill written from RFC 8259 alone) can disagree on the output bytes for a `ToolCall.arguments` field containing, say, `"\u007f"`, `"\n"`, or `"𝐀"` (U+1D400, requires surrogate-pair output) — and a non-`TextContent` `toolResult` block whose serialised form is a worked oracle in BNDR-7g/7j hits the same gap.
- **In-JSON number formatting.** BNDR-5's no-exponent / shortest-round-tripping rules live in `defaulting-system-note-echo.md` and are explicitly framed as the **echo formatter's** per-`number` row; they govern the system-note line, not in-transcript JSON. So `1e21` in a `ToolCall.arguments` numeric field — which `JSON.stringify` renders as `1e+21` — and `1e-7`-class small numbers — which `JSON.stringify` renders as `1e-7` — produce different bytes in the binder transcript vs. the echo line for the same value.

BNDR-7's oracles (7a–7j) exercise only ASCII strings and integer or modest-decimal numbers, so neither concern is closed by an existing rendering. Because BNDR-7 is framed as MUST-reproduce-exactly and BNDR-8 is the renderer's "single normative source for emitting JSON anywhere in the transcript", any divergence on these axes is a conformance defect with no rule to cite.

## Issue introduction

**Verdict:** single-commit-introduced
**Introducing commit:** `1a41db2` — *pi-loom spec: resolve "Compact-transcript BNDR-7 oracle coverage and assistant byte-determinism (T034/T106)"* (Sat Jun 6 19:40:34 2026)
**History:** Before `1a41db2`, the assistant-body rule-4 text said `<args-json>` is `JSON.stringify(arguments)` with no whitespace — a direct reference to the host primitive that implicitly inherited its string-escape and number-format semantics. `1a41db2` replaced that wording with a forward reference to BNDR-8, and the newly-coined BNDR-8 pinned only no-whitespace and the key-sort overlay. That replacement is what stripped the implicit string-escape / number-format inheritance the previous text carried, and is therefore the single commit at which the gap entered. Commit `97ab51e` (Thu Jun 25 09:46:30 2026, the same day this review was generated) widened BNDR-8's scope from "assistant `<args-json>`" to "the renderer's single normative source for emitting JSON anywhere in the transcript" (covering `toolResult` non-text blocks too) but kept the no-whitespace + key-order pins as the only listed canonical-JSON properties, so it propagated the same gap to a new surface rather than introducing it.

## Solution Space

**Shape:** single

### Recommendation

In BNDR-8, after the existing no-whitespace + key-order + array-order sentence, add a single sentence pinning the remaining semantics by reference: *Apart from the key-ordering overlay above, the canonical serialisation MUST otherwise produce the byte-for-byte output of ECMA-262 `JSON.stringify` called on the same value — specifically the `\"` / `\\` / `\/` (optional) / `\b` / `\f` / `\n` / `\r` / `\t` short escapes and lowercase-hex `\u00XX` escapes for the remaining C0/U+007F controls; literal UTF-8 bytes for every other Unicode scalar (no `\uXXXX` escaping of non-ASCII); UTF-16 surrogate-pair `\uXXXX\uXXXX` form only for code points above U+FFFF when they appear unpaired in a host UTF-16 source string; and ECMA-262 `Number.prototype.toString(10)` for numbers (which means scientific notation at the `|n| ≥ 1e21` / `0 < |n| < 1e-6` thresholds is permitted here — BNDR-5's no-exponent / shortest-round-tripping rule is scoped to the [echo formatter's `number` row](./defaulting-system-note-echo.md#bndr-5) and does **not** apply to in-transcript JSON).* Then extend BNDR-7 with one additional reference rendering whose `ToolCall.arguments` exercises (a) a string containing a non-ASCII character (e.g. `"café"`) emitted as literal UTF-8 and (b) one of the exponential-notation magnitudes (e.g. `1e21`) emitted as `1e+21`, so the oracle is observable.

Edge cases the implementer must watch: when the host primitive is itself `JSON.stringify`, the canonical serialiser is `JSON.stringify` composed with the key-sort overlay (sort keys in-place at every object level before serialising, or post-process the output) — no further escape transform is required. The two-axis scope split (BNDR-5 echo numbers ≠ BNDR-8 transcript JSON numbers) is intentional and must be called out in the new sentence to prevent a future editor from "harmonising" them and breaking the implicit-inheritance fix.

## Consequence

**Severity:** correctness

Two conforming implementations can produce different transcript bytes for the same `ToolCall.arguments` (or `toolResult` non-text block) whenever it carries a non-ASCII string, a control character, or a number in `JSON.stringify`'s exponential-notation magnitudes (|x| ≥ 1e21 or 0 < |x| < 1e-6). That contradicts the *MUST-reproduce-exactly* contract BNDR-7 hangs on, and silently breaks the input-reproducibility rationale (prompt-cache stability, auditability) the surrounding "Why the transcript bytes are pinned" paragraph asserts.
