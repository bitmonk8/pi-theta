## Problem

The model-reference-matcher contract underpinning binder-model resolution (`V11a`), frontmatter `model:` resolution (`V6a`), and the load-pass production wiring (`V9b`) is specified as constructing/calling Pi's `findExactModelReferenceMatch` against `ctx.modelRegistry.getAvailable()`. At the pinned Pi minor (`@earendil-works/pi-coding-agent` `~0.75.5`, resolving to `0.75.5`) that symbol is not reachable through the package's public surface: it is declared only at `dist/core/model-resolver.d.ts`, is not re-exported from `dist/index.d.ts` (which exposes only `ModelRegistry` and its `getAvailable()`), and the `package.json#exports` map (only `"."` and `"./hooks"`) blocks a deep import. A named root import and a deep import both fail, so the load-pass construction `V9b` mandates does not compile against the supported pin. The symbol is also outside the factory-probed / surface-inventoried capability set, so loom's drift gate gives no signal â€” the failure surfaces only as an import/build error.

**History:** `git log -S findExactModelReferenceMatch` traces the token's spec origin to `54c4e39`, which authored the `#binder-model-parse-rule` clause and the `#model-registry-pin` sentence pinning the symbol by its `dist/core/model-resolver.d.ts` declaration site, without confirming it is re-exported on the package-root surface. The plan inherited the framing wholesale at `c6a664e` and at the `V6a`/`V9b` seam-producer edits (`5953b18`, `44a96f5`). The defect is original to the model-reference-matcher framing, not a later regression.

## Solution approach

Rewrite the matcher to be loom's own exact-match resolver run over the public `ctx.modelRegistry.getAvailable()` surface, dropping all dependence on importing `findExactModelReferenceMatch`. In `V9b`'s Adds and Tests, change the load-pass production wiring to construct loom's resolver over `getAvailable()`; in `V6a`'s Adds, strike the `findExactModelReferenceMatch` attribution from the injection-seam description (the seam and its `loom/load/model-unresolved` resolution are unchanged); in `V11a`'s Adds, replace the `findExactModelReferenceMatch` reference with loom's resolution over `getAvailable()`; and update `V9b-T`, `V6a-T`, `V11a-T` to assert against loom's resolver. The resolver must replicate the documented semantics â€” accept a canonical `provider/modelId` reference or a bare `modelId`, and treat a bare `modelId` matching models under more than one provider as ambiguous (resolving to no model, not pick-first). The plan inherits the `findExactModelReferenceMatch` framing from spec `#binder-model-parse-rule` / `#model-registry-pin`, so align those spec sites to the `getAvailable()` re-implementation as well.
