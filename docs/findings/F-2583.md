## Finding

The Identifiers rule in `lexical.md` requires PascalCase for "`schema` names, `enum` names, `enum` variant names, **and any user identifier introduced as a type-like binding**", and a violation fires the parse error `loom/parse/schema-case-mismatch`. The trailing clause is an open-ended catch-all: "type-like binding" is never defined and its member set is never enumerated, so an implementer cannot decide which identifier positions, beyond the three explicit declaration forms, the parser must reject for lowercase-first spelling.

The gap is not hypothetical — the corpus does introduce type-like bindings outside the three listed positions:

- **Import / re-export aliases of type symbols.** `imports.md` admits `import { Author as Reviewer }` and `export { Author as Reviewer } from "..."`. When the source symbol resolves to a `schema` or `enum`, the local alias (`Reviewer`) is a type-like binding. Whether it must be PascalCase is unstated, yet it is load-bearing: `expressions.md` pattern grammar disambiguates `match` arms purely by first-letter case (uppercase = type/constructor reference, lowercase = fresh binding). Allowing a type symbol to be aliased to a lowercase local name would silently break the ability to match against that type by its local name.
- **Type-alias schemas.** `schemas.md` defines `schema X = R`; its name `X` is covered by "`schema` names" (it uses the `schema` keyword), so it is *not* an extra position — but the diagnostics registry row for `loom/parse/schema-case-mismatch` (`code-registry-parse.md`) describes the trigger as "schema / enum / variant / **type-alias** position", a *third* enumeration that differs from both the explicit list and the catch-all, and still omits import/export aliases.

So the two normative descriptions of the same parse error disagree, and neither pins the import/export-alias case. Two reasonable implementers will diverge on whether `import { Author as reviewer }` is accepted or rejected.

## Issue introduction

**Verdict:** single introducing commit
**Introducing commits:** `8eea523` ("spec: fill implementation-blocking gaps", 2026-05-04)
**History:** The corpus is git-tracked. `git log -S 'type-like binding' --all --reverse` and `git show 8eea523` show the entire **Identifiers** block — the PascalCase bullet including the trailing "and any user identifier introduced as a type-like binding" clause — was added wholesale in `8eea523`, then carried unchanged through the `fecb504` spec split and the `31ff060` `docs/` move (verified with `git log --follow` over `docs/spec_topics/lexical.md`). The catch-all was present from the moment the casing rule entered the corpus; there was no earlier enumeration onto which it was later bolted, and the explicit three-position list and the catch-all entered together in the same hunk.

## Solution Space

**Shape:** single

### Recommendation

Replace the open-ended catch-all in `lexical.md` with the explicit closed set of PascalCase positions, and align the `loom/parse/schema-case-mismatch` registry row to the same set. The closed set is:

1. `schema` names — both object schemas (`schema X { ... }`) and type-alias / union schemas (`schema X = R`); both use the `schema` keyword.
2. `enum` names.
3. `enum` variant names.
4. The local binding introduced by an `import` or re-export `as`-alias **whose source symbol resolves to a `schema` or `enum`** — `import { Author as Reviewer }` and `export { Author as Reviewer } from "..."`. A `fn`-symbol alias keeps the lowercase-first rule; the required case of an `as`-alias is determined by the kind of the resolved source symbol.
5. Built-in `Ok`, `Err`, `Result`.

State explicitly that no other position requires PascalCase (the casing rule is otherwise the only enforced naming constraint). Update the `code-registry-parse.md` description from "schema / enum / variant / type-alias position" to the enumerated set above so the two normative statements agree; "type-alias" is redundant (alias schemas are `schema` names) and the row currently omits import/export aliases.

Edge case for the implementer: the casing requirement on an `as`-alias depends on the kind of the resolved source symbol, which is known only after the resolved `.warp` file's parse completes — the same ordering precondition that `loom/parse/import-unknown-symbol` already relies on. The alias-casing check therefore cannot be a pure tokeniser-position check; it is enforced after import resolution, not at lex time.

## Consequence

**Severity:** correctness

Reading the catch-all broadly versus reading the explicit declaration list (or the registry's third variant) yields different accept/reject decisions for `import { Author as reviewer }` / `export { Author as reviewer } from`. Divergent implementations accept or reject the same program, and a type aliased to a lowercase local name silently defeats the first-letter `match`-arm disambiguation that the casing rule exists to guarantee.
