## Finding

The plan's build order is dep-driven: `docs/plan.md` §How to use step 3 says "Pick the next leaf whose **Deps** are satisfied," and `conventions.md` §Leaf format defines **Deps** as "Other leaf IDs that must be complete first." Completion is recorded by a git tag — `conventions.md` §Per-phase TDD ritual tags `<id>-T-complete` on the tests task and `<id>-complete` on the implementation task (and a single `<id>-complete` for horizontal leaves).

What the plan never states is the *predicate* that joins these two facts: it does not say that "Deps satisfied" is evaluated by checking for the presence of each Deps leaf's `<id>-complete` (or `<id>-T-complete`) git tag, nor does it name the authoritative completion signal a picker must read. A human reader can infer the bridge, but the plan declares an automated orchestrator (`.pi/project-config.md` drives commits and `commitAddPaths`), and a deterministic picker needs the satisfaction test stated explicitly rather than inferred. As written, two orchestrator implementations could disagree on what counts as "complete" (tag presence vs. commit existence vs. a merged PR vs. reviewer judgement).

**History:** The original plan (288f191) introduced **Deps** as "Other leaf IDs that must be complete first" together with the how-to-use "pick the next leaf whose Deps are satisfied" instruction, and recorded completion only via the `<id>-complete` tag — never stating the predicate that binds Deps-satisfaction to tag presence. f9c5354 split the tag mechanism into the paired `<id>-T-complete` / `<id>-complete` form but likewise added no satisfaction predicate, so the gap has been present since the plan's first commit.

## Solution Space

**Shape:** single

### Recommendation

State the pickability predicate explicitly at the point where the picking instruction lives — `docs/plan.md` §How to use step 3 (the sentence beginning "Pick the next leaf whose **Deps** are satisfied"), or equivalently in `conventions.md` §Leaf format's **Deps** bullet. The added text names the authoritative completion signal: a leaf is pickable when, for every leaf ID listed in its **Deps**, the corresponding completion git tag is present — `<id>-complete` for an implementation/horizontal dependency and `<id>-T-complete` for a `-T` tests-task dependency. Because an implementation task already lists its paired `<id>-T` in **Deps**, that pairing's gate falls out of the same predicate.

Suggested literal insertion (adapt wording to the chosen host sentence): "**Deps** are *satisfied* iff every leaf ID listed in the leaf's **Deps** has its completion git tag present — `<id>-complete` for an implementation or horizontal leaf, `<id>-T-complete` for a `-T` tests-task dependency. Tag presence is the authoritative completion signal a picker (human or orchestrator) reads."

## Consequence

**Severity:** advisory

A deterministic automated orchestrator has no stated rule for evaluating Deps-satisfaction, so it must infer that completion means the `<id>-complete` tag is present. Two reasonable picker implementations could diverge on the completion signal (tag vs. commit vs. merge), and the ambiguity is invisible until an orchestrator is built. The convention is otherwise inferable, so leaf authoring is not blocked.
