## Finding

The V6a Tests bullet reads: "An unknown frontmatter key fires `loom/load/unknown-frontmatter-field` (severity `W`) and is tolerated." The word "tolerated" names a posture without an observable. A test author cannot tell what positive state must hold: does the loom still load? Does it register and become invocable? Is the unknown key kept out of `params` / the binder? A faithful test could assert only that the `W` diagnostic fires and never witness that the loom survived the unknown key — exactly the behaviour the field contract exists to guarantee.

The spec pins the success path unambiguously. `frontmatter-fields-a.md` §"Unknown-key policy" states that unknown top-level keys "are ignored at the value level (the runtime does not propagate them into `params`, the binder, or any other surface) and surface as a single warning per key," and that "The loom continues to load and register in both cases." The V6a (implementation) and V6a-T (tests) leaves both carry the identical under-specified bullet, so the gap is in the paired leaf as well as the implementation leaf.

## Consequence

**Severity:** advisory

A test author closing V6a could assert only the `W`-severity diagnostic and leave the load-and-register success path unwitnessed; the leaf would then ship green while a regression that drops a loom on an unknown forward-compat key goes undetected. The spec states the behaviour clearly, so a faithful implementer is likely to get it right, but the gate does not force the assertion.

## Solution Space

**Shape:** single

### Recommendation

In the `V6a` and `V6a-T` Tests bullet, replace the bare "and is tolerated" with the explicit observable from `frontmatter-fields-a.md` §"Unknown-key policy": the loom still loads and registers (becomes invocable) despite the unknown key, the unknown key's value is not propagated into `params` or any other surface, and the `loom/load/unknown-frontmatter-field` diagnostic is emitted at severity `W`. Keep the bullet text identical in `V6a` and `V6a-T` so the implementation/tests pair stays in lockstep.

The literal text to assert is the spec's: "The loom continues to load and register" with the unknown key value ignored at the value level. Cite `frontmatter-fields-a.md` §"Unknown-key policy" as the source.

`V6a`'s Ships-when ("`npm test` parses frontmatter, requires `mode:`, and tolerates unknown keys as warnings") carries the same loose "tolerates" wording; align it to name the load-and-register success state if the bullet is reworded, so the gate criterion matches the test observable.
