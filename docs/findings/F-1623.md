## Finding

The intro paragraph of `docs/spec_topics/descriptions.md` ends:

> "[descriptions] lower to JSON Schema `description:` fields and are passed to the model when the schema is used for structured output — **they materially improve output quality**."

The bolded clause is positioned as an in-line factual / normative claim in the same sentence as a deterministic lowering rule, but it is not a falsifiable obligation: it pins no threshold, no benchmark, no observable behaviour, and no measurement protocol. "Materially" and "output quality" are both undefined in the spec corpus. Two implementers cannot disagree about whether their implementation satisfies it, and no conformance test could ever pass or fail on it — there is no rule to test.

Everything implementers actually need from this section is already normatively specified in the surrounding bullets (placement, multi-line join, no interpolation, byte-for-byte emission, the `(Non-normative provenance.)` Markdown observation). The "materially improve" clause adds nothing observable; it is motivation prose that has drifted into the normative intro. Leaving it in invites two failure modes: (a) downstream readers read it as a behavioural obligation and look for nonexistent benchmarks, and (b) it sets a precedent for unfalsifiable quality claims elsewhere in the corpus.

The clause has therefore been in the corpus the entire life of the Descriptions section as a single-author oversight from the original rewrite, not a regression introduced by later work.

## Solution Space

**Shape:** single

### Recommendation

Remove the offending clause from the normative intro and, if any of the motivation is worth keeping, fold the kept fragment into a parenthetical `*(Non-normative.)*` aside parallel to the existing `*(Non-normative provenance.)*` aside in the "No transformation" bullet. Concretely, replace the current intro paragraph with two sentences that keep only the falsifiable half:

> "Loom uses Rust-style `///` doc comments to attach descriptions. They lower to JSON Schema `description:` fields and are passed to the model when the schema is used for structured output."

If the editorial intent of the original clause (that descriptions exist because providers' structured-output models attend to them) is worth preserving as motivation, append a separate sentence tagged as non-normative — e.g.:

> "*(Non-normative.)* Authors should write descriptions for every schema, enum, field, and variant intended for structured output; provider models at loom 1.0 authoring time attend to `description:` text when producing structured outputs, and omitting them leaves the model with only field/variant identifiers."

This matches the corpus's established discipline (already used in the "No transformation" bullet on the same page) of fencing empirical / motivational claims behind an explicit `*(Non-normative…)*` tag rather than letting them sit inside normative prose. The fixer should make sure nothing later in the spec corpus links to the original sentence as a normative anchor (a quick `grep` for "materially improve" should return zero hits after the edit).

## Consequence

**Severity:** cosmetic

A reader cannot mechanically verify the claim and the surrounding bullets already carry every implementer-binding obligation, so nothing in the implementation changes when this clause is fixed. The risk is purely editorial drift: the line normalises unfalsifiable quality assertions in normative intro prose.
