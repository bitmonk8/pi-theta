## Problem

`conventions.md` §REQ-ID discipline defines three closing-gate surfaces, but the third is asymmetric with the other two on assertion existence. The numbered-`PREFIX-N` surface fails on a coverage-matrix-mapped REQ-ID with no citing test, and the diagnostic-registry surface fails on a registry code with no asserting test — both reach a test-existence check. The third surface (un-anchored normative MUST/MUST-NOTs, the GOV-22 residue class enumerated under `coverage-matrix.md` *Code-keyed obligation areas (no numbered REQ-IDs)*) fails only when a matrix row names no closing leaf; nothing forces the named leaf to carry a test asserting the MUST. The release-time backstop does not close the gap — §Loom 1.0 release-time residue inspection item 5 is "Clean when" each obligation is enumerated with a named closing leaf (an enumeration condition, no asserting-test requirement), and item 7 (split-row joint-completeness, which does require per-facet citing tests) sweeps only *Numbered REQ-IDs* rows — so a pure-prose spec MUST carrying neither a numbered REQ-ID nor a `loom/...` code can ship untested with every gate green.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `0603eb4` (2026-06-10), `953e3fa` (2026-06-10)
**History:** The third closing-gate surface for un-anchored normative MUSTs entered in `0603eb4` ("un-anchored normative MUSTs invisible to closing gate") with only an enumeration gate — a CI failure for an un-anchored MUST with no enumerated closing leaf. Roughly ten hours later, `953e3fa` ("closing gate checks numbered-REQ-ID matrix mapping, not test assertion") added a citing-test scan to the numbered-REQ-ID surface, explicitly bringing it "to the registry-code↔asserting-test parity," while in the same edit it left the third (un-anchored) surface enumeration-only. The diagnostic-code surface already carried registry↔asserting-test parity at that point. The asymmetry — assertion-existence gating on the numbered and diagnostic surfaces but not on the un-anchored-MUST surface — was cemented by `953e3fa`, which raised the numbered surface to parity without propagating the same treatment to the un-anchored surface sitting beside it in the same paragraph.

## Solution approach

Extend §Loom 1.0 release-time residue inspection item 5 so its "Clean when" condition requires, for each *Code-keyed obligation areas* row whose obligation carries neither a numbered REQ-ID nor a `loom/...` code, that the reviewer confirm the named closing leaf carries a test asserting that MUST — not merely that the row names a closing leaf. Scope the addition to the pure-prose subset so code-keyed rows anchored by a `loom/...` diagnostic code, already covered by the diagnostic-code parity check, are not double-gated. A mechanical fast-fail surface in §REQ-ID discipline may layer on later but is not required to close this gap.

## Solution constraints

- Out of scope: re-anchoring any individual MUST as a `PREFIX-N` REQ-ID — that is a spec-side GOV-22 decision owned by the relevant spec-coverage finding, not this plan rule.
