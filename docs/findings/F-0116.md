## Finding

`V18c`'s **Ships when** confirms the editorial-review checklist's acceptance "by editorial review — not by `npm test` — that the bump commit message **records** a per-item outcome for every item `(a)`–`(ak)` (item `(e)` recording its two sub-outcomes) in the shape `version-bump-step2.md` mandates." `V18c`'s **Adds** mirrors this, listing the admissible recording shape as `pass / fail / N/A / not-performed` for the whole `(a)`–`(ak)` range. As written, the gate is satisfied by the *presence* of a recorded outcome string per item; it does not distinguish the outcome's *value*.

The spec draws a load-bearing distinction the gate erases. `version-bump-step2.md` makes items `(a)`–`(e)` MUST-level (the contributor MUST audit them; the recording shape for these is `pass / fail / N/A` — `not-performed` is reserved for the SHOULD-level `(f)`–`(aj)` skip case), and attaches merge consequences to a non-acceptable outcome: item `(e)` "MAY merge only when that second-pass outcome is `pass`" (after the recovery serialisation is added and re-run), and item `(a)` carries an open question that "MUST be resolved" with an authoritative determination recorded. By contrast `(f)`–`(ak)` are explicitly record-only — a recorded `fail` or `not-performed` there is conformant.

Because `V18c`'s gate checks only that *an* outcome is recorded, a bump whose MUST-level item `(a)`–`(e)` is recorded `fail` (or, per the leaf's own over-broad Adds shape, `not-performed`) passes the leaf's acceptance gate while violating what the spec permits to merge. The gate enforces the SHOULD-level record-only rule uniformly and never enforces the MUST-level pass/resolution semantics for `(a)`–`(e)`.

**Verdict:** single-commit-introduction
**Introducing commit:** `f1cda48` — "pi-loom plan: resolve \"V18c mislabels editorial-review checklist as a static build-time gate\"" (Thomas Andersen, 2026-06-11)

Before `f1cda48`, the editorial-review checklist was (incorrectly) lumped into the static build-time gates in **Adds** with no separate **Ships when** acceptance clause; that commit correctly peeled it out as a manual, non-`npm test` gate but framed acceptance solely as "records a per-item outcome for every item `(a)`–`(ak)` … in the shape `version-bump-step2.md` mandates" and added `not-performed` to the admissible recording shape for the whole range. The record-only framing — and the absence of any MUST-level outcome-acceptability constraint for `(a)`–`(e)` — entered with that single commit and has not changed since (`git log --follow` shows later V18c commits touched unrelated gates).

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V18c-version-bump-checklist.md`, tighten the editorial-review acceptance so it gates on outcome acceptability for MUST-level items while leaving SHOULD-level items record-only:

- In **Ships when**, change the editorial-review clause so the bump is acceptable only when, in addition to every item `(a)`–`(ak)` carrying a recorded outcome, each MUST-level item `(a)`–`(e)` records an *acceptable* outcome — `pass`, or a justified `N/A` per `version-bump-step2.md`'s `N/A` definition — with item `(e)` acceptable only on its second-pass `pass` after the recovery serialisation is added and re-run (per `version-bump-step2.md`'s "MAY merge only when that second-pass outcome is `pass`"), and item `(a)` acceptable only when its open clause-(a) contradiction is authoritatively resolved. A recorded `fail` or `not-performed` on any of `(a)`–`(e)` blocks the bump. Items `(f)`–`(ak)` remain record-only: any of `pass`/`fail`/`N/A`/`not-performed` with a one-line rationale satisfies the gate.

- In **Adds**, correct the done-condition's stated recording shape so `not-performed` is not listed as admissible for `(a)`–`(e)` (the spec restricts those to `pass` / `fail` / `N/A`), and state that a MUST-level `fail`/`not-performed` is non-acceptable for merge, distinct from the SHOULD-level record-only rule.

The authoritative wording for the acceptable/non-acceptable outcomes and item `(e)`'s second-pass and `N/A — superseded by <named Pi mechanism>` shapes is in `version-bump-step2.md`; cite it rather than restating the recovery-serialisation prose. The spec is read-only for this fix — the distinction already exists there; the leaf must enforce it.

## Consequence

**Severity:** correctness

The leaf's editorial-review acceptance gate passes vacuously for the MUST-level semantics: a contributor can record `fail` (or `not-performed`) on a MUST-level presupposition `(a)`–`(e)` and still satisfy `V18c`'s **Ships when**, even though `version-bump-step2.md` forbids merging item `(e)` other than on a second-pass `pass` and requires item `(a)`'s open contradiction to be authoritatively resolved. Two reasonable implementers would build the gate differently — one record-only, one outcome-gating — and the record-only reading lets a non-conformant bump through.
