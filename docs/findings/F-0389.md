## Finding

`V11h`'s **Adds** instructs the implementer to supply the number renderer's `integer`-vs-`number` kind discriminator "from the bound value's static type via the value model (`V2c`) **rather than from runtime integrality**." The phrase "runtime integrality" is a coined term that appears nowhere in the spec (`docs/spec_topics/` has no occurrence of "integrality" or any equivalent definition) and is left unexplained in the plan. It names the rejected alternative — deciding the kind by inspecting the value at runtime rather than from its declared static type — but does not say concretely what that runtime inspection would be (`Number.isInteger(v)`, `v % 1 === 0`, etc.).

The positive instruction is unambiguous: the discriminator comes from the static type via `V2c`, so an implementer will not actually derive it from the value at runtime. The defect is purely that the rejected-alternative clause is expressed in coined jargon rather than concrete terms. Replacing it with an explicit `Number.isInteger(v)`-style description makes the static-vs-runtime contrast crisp without changing the pinned behaviour. The illustrative case is a value such as `2.0`: runtime integrality would classify it `integer`, while the value model's static type keeps it `number` — exactly the divergence the static-type rule is there to prevent.

## Issue introduction

**Verdict:** introduced-by-later-edit (single commit)
**Introducing commit:** `4807958` — "pi-loom plan: resolve 'renderer input contract: integer-vs-number discriminator origin'" (2026-06-17)
**History:** The phrase "rather than from runtime integrality" entered `docs/plan_topics/V11h-argument-echo.md`'s **Adds** in commit `4807958`. Before that commit the **Adds** carried no discriminator clause at all; the same commit pinned the `integer`-vs-`number` kind discriminator to the bound value's static type via `V2c` (also adding `V2c` to **Deps**) and, in doing so, coined "runtime integrality" as the name for the rejected alternative. `git log -S 'runtime integrality'` confirms `4807958` is the only commit touching the phrase across the corpus; the sibling `V2d` leaf received the parallel "never from the value's runtime integrality" wording in the same resolution.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V11h-argument-echo.md`, in the **Adds** field, strike the clause `rather than from runtime integrality` and replace it with `rather than from a runtime `Number.isInteger(v)` test on the bound value`. The surrounding clause ("...from the bound value's static type via the value model (`V2c`)...") stays intact. For terminology consistency, the parallel "never from the value's runtime integrality" wording in `docs/plan_topics/V2d-number-rendering.md`'s **Adds** may be aligned to the same `Number.isInteger(v)` phrasing in the same pass. No spec edit and no behaviour change are required — the discriminator origin remains the static type via `V2c`.

## Consequence

**Severity:** cosmetic

The pinned instruction (discriminator from static type via `V2c`) is already unambiguous, so no implementer is misled into divergence; the only cost is the reader must infer the meaning of a coined, spec-unanchored phrase. Making the rejected alternative concrete removes that inference step.
