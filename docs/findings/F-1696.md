## Finding

PIC-55 in `docs/spec_topics/pi-integration-contract/registration-steps.md` (under step 5) splits the post-`error` / post-throw watcher posture into two arms keyed on whether "the watcher keeps delivering events on the watched roots after the error" (transient `ctx.ui.notify` toast, no further action) versus "the error or throw leaves one or more watched roots no longer delivering events" (persistent `loom/runtime/watcher-terminated` `loom-system-note`, watcher torn down, operator runs `/reload`). The diagnostic-table row for `loom/runtime/watcher-terminated` in `docs/spec_topics/diagnostics/code-registry-runtime.md` repeats the same predicate verbatim ("its `error` route fired or it threw such that one or more watched roots stop delivering events"), and the toast-channel paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` is now tied to the same two-arm split.

The classifying predicate is not observable. The `FileWatcher` seam (PIC-14, `docs/spec_topics/pi-integration-contract/host-interfaces-services.md`) carries only the three event kinds `add | change | unlink` plus the `Unsubscribe` return; the adapter sees chokidar's `error` event (whose payload is an `Error`, with no "is-terminal" / "I have stopped delivering" flag) and any synchronous throw out of the event callback. "No longer delivering events" is an absence-of-future-events condition that cannot be decided at the moment the `error` or throw is observed — chokidar publishes no contract that says "after this error I will / will not continue delivering". Two reasonable implementers will therefore diverge: one routes every `error` to the transient toast and never emits `watcher-terminated` (so a dead watcher silently strands hot-reload), the other routes every `error` to the persistent note and unnecessarily prompts `/reload` for benign transient errors.

PIC-55 must either name the observable input the adapter uses to pick an arm (e.g. a closed list of terminal `Error.code` values, or a "the adapter unconditionally calls `close()` on `error` and re-arms only by operator `/reload`" rule), or be collapsed to a single arm whose classifier is the act of observing `error` / a sync throw itself.

## Solution Space

**Shape:** multiple

### Option A — Collapse to a single terminal posture

**Approach.** Drop the two-arm bifurcation. Any chokidar `error` event or synchronous throw out of an adapter-installed callback is treated as terminal: the adapter unconditionally calls `close()` on the underlying chokidar `FSWatcher` (idempotently, per the existing `Unsubscribe`-double-call no-op rule) and the runtime emits the single persistent `loom/runtime/watcher-terminated` `loom-system-note`. The transient-toast arm for the chokidar-error case is removed.

**Spec edits.**
- PIC-55: replace the two-case body with a single posture — "On any `error` event or synchronous throw out of the adapter's event callback, the adapter closes the watcher and the runtime emits a single persistent `loom/runtime/watcher-terminated` `loom-system-note`." Keep the existing drain-state-tag-MUST-NOT-write paragraph; drop the `(Continues-delivering.)` paragraph.
- `code-registry-runtime.md` `loom/runtime/watcher-terminated` row: replace the trigger clause "its `error` route fired or it threw such that one or more watched roots stop delivering events" with "its `error` route fired or it threw, and the adapter consequently closed it" (observable; no future-events predicate).
- `diagnostic-shape.md` `transient-toasts` paragraph case (1): drop the "the terminal stopped-delivery case additionally emits…" qualifier and the "transient toast stays reserved for the informational chokidar-error case" carve-out — the chokidar-error case is no longer a toast case. Retain the toast surface for the other two listed origins (settings I/O exceptions outside `loom/load/settings-*`, internal invariant violations).
- `host-interfaces-services.md` PIC-14 bullet on `watch`: no edit required; the cross-reference back to PIC-55 still holds.

**Pros.**
- Removes the unobservable predicate entirely; the classifier becomes "we caught an `error` or a throw", which is directly observable at the adapter boundary.
- No dependency on chokidar `Error.code` stability or on cataloguing terminal codes.
- Matches the spec's existing "watcher torn down rather than re-armed" disposition for the throw case (`diagnostic-shape.md` case (1) already documents this for the bare-throw path), so the change is to extend that disposition to `error` rather than to invent new behaviour.

**Cons.**
- Operators see a `/reload` prompt for chokidar `error` events that would in practice have been transient (e.g. one missed event due to a brief EACCES on a temp file inside a watched root).
- The transient-toast arm for chokidar `error` (described as an "informational" case in `diagnostic-shape.md`) is lost; if loom 1.0 has any expectation of surfacing benign watcher errors without operator action, that surface is gone.

**Risks.** Over-prompting in noisy filesystems. Mitigated by the fact that chokidar `error` is rare in normal operation; the spec already accepts re-emission of the persistent diagnostic across reloads (`diagnostic-shape.md` "Re-scan deduplication").

### Option B — Pin an explicit terminal-classifier in the adapter

**Approach.** Keep the two-arm bifurcation but specify what the adapter inspects to pick an arm. The adapter classifies an `error` as terminal iff the underlying `Error.code` is in a closed loom-pinned set (initially `ENOSPC`, `EMFILE`, `ENFILE`, `EBADF`, plus a synchronous throw out of the adapter's event callback, which is unconditionally terminal because the watcher state after a thrown handler is undefined). All other `error` events are classified as continues-delivering and emit the transient toast; the watcher is left running. The closed set is re-audited on each chokidar minor bump via a new editorial-review checklist item routed through the existing `version-bump-step2.md` pattern (mirroring item (z) for `pi.registerTool` post-startup acceptance).

**Spec edits.**
- PIC-55: keep the two-case structure; replace "When the watcher keeps delivering events on the watched roots after the error" with "When the chokidar `Error.code` is outside the loom-pinned terminal set below" and "When the error or throw leaves one or more watched roots no longer delivering events" with "When the chokidar `Error.code` is in the loom-pinned terminal set, or a synchronous throw escaped the adapter's event callback". Append a closed list of terminal `Error.code` values with an editorial-review-checklist cross-reference.
- `host-interfaces-services.md` PIC-14 bullet on `watch`: add a sentence stating the adapter performs the terminal classification before routing — the seam still delivers only the three event kinds to the runtime, but the chokidar-adapter's own routing decision is now spec-defined.
- `code-registry-runtime.md` `loom/runtime/watcher-terminated` row: replace the absence-of-future-events predicate with "its `error` route fired with an `Error.code` in the [terminal set](../pi-integration-contract/registration-steps.md#pic-55) or a synchronous throw escaped the adapter's event callback".
- `diagnostic-shape.md` `transient-toasts` paragraph case (1): retain the bifurcation but cross-reference the terminal-set pin rather than the unobservable predicate.
- A new editorial-review-checklist row in `version-bump-step2.md` for the chokidar terminal-code set, on the same footing as item (z).

**Pros.**
- Preserves the existing two-arm posture and the informational-toast surface for non-terminal chokidar errors.
- The terminal predicate becomes observable: `Error.code` is a string the adapter can match against a closed set at the failure site.

**Cons.**
- Introduces a new known-fragile presupposition on the stability and completeness of chokidar's `Error.code` taxonomy (chokidar wraps Node `fs` errors plus its own; the code surface is undocumented as a stable contract).
- Adds a new editorial-review-checklist item and a new closed-list pin that must be maintained across chokidar minor bumps.
- Risks miscategorising a real terminal failure as transient if it surfaces with an `Error.code` outside the pinned set, silently stranding hot-reload — the same defect Option A eliminates, narrowed to the unpinned-code case.

**Risks.** Coverage gaps in the terminal-code set. Mitigated only by the editorial-review process, which is itself unverified for this taxonomy.

### Recommendation

Adopt **Option A**. The two-arm posture's only practical value is preserving a transient-toast surface for "benign" chokidar errors, but the spec has no documented benign-error population to preserve and Option B introduces a new external-contract dependency (chokidar `Error.code` stability) to support that surface. Option A's cost — operator `/reload` prompts on rare chokidar `error` events that might in practice have been transient — is bounded by how rare chokidar `error` is in normal operation, and the persistent-note channel already documents re-emission across reloads, so the failure mode is "the operator sees the note again on the next reload until the underlying condition resolves" rather than "the runtime mis-routes". Edge cases the implementer must watch: the adapter's `close()` call must be idempotent against the existing `Unsubscribe` (PIC-14 already requires this); the synchronous-throw path must catch out of every adapter-installed event handler (not just the three load-bearing kinds, so that a throw escaping the filtered-out `addDir` / `unlinkDir` / `ready` handlers also reaches the terminal path); the drain-state-tag MUST-NOT-write rule under PIC-55 carries over verbatim.

## Consequence

**Severity:** correctness

Two implementers building against PIC-55 today will produce materially different observable behaviour: one will silently lose hot-reload after the first chokidar `error` (no `watcher-terminated` note ever fires), the other will demand `/reload` after every transient `error` even when the watcher would have continued. Both can claim conformance, because the spec's bifurcation predicate has no defined observable input the adapter can evaluate at the failure site.
