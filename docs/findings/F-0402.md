## Finding

`V15h` adds two unrelated witnesses at the runtime `invoke` site in a single leaf: (1) the ceiling-#4 depth-surface rows that wrap a depth-6 `invoke(...)` `params` argument and a depth-6 `invoke<T>` return value into `InvokeInfraError` (the delegated live-carrier witness for `V5e`'s routing decision), and (2) the `invoke` child top-level execution Promise's swallowing-handler attachment plus three-side-channel late-settlement suppression at the `Checkpoint` seam (a cancellation race). These are orthogonal failure modes — a schema-depth validation surface and a Promise-lifetime cancellation race — that share only the `invoke` call site. They have disjoint dependency sets (the depth half rests on `V5e`; the swallowing half rests on `V8a` and the `H4b` late-settlement harness) and disjoint spec anchors (`hard-ceilings/ceilings-3-and-4.md` vs `cancellation.md`).

The parallel tool-call site already keeps these two concerns in separate leaves: `V14e` is the ceiling-#4 depth-6 code-driven-tool-args live carrier (deps `V14a`, `V5e`), and `V14f` is the code-side `execute()` swallowing-handler per-site routing leaf (deps `V8a`, `H4b`). `V15h` is the only one of the four swallowing-handler routing sites (`V14f`, `V13f`, `V15h`, `V9o`) that also carries an unrelated depth-surface obligation, so the invoke site is structurally inconsistent with its siblings.

## Solution Space

**Shape:** single

### Recommendation

Extract the ceiling-#4 depth-surface half into a new leaf, leaving `V15h` as the invoke-child swallowing-handler leaf only — mirroring the `V14e` (depth) / `V14f` (swallowing) split already in place at the tool-call site. Keep the residual `V15h` as the swallowing-handler owner so its membership in the four-site routing set (`V14f`, `V13f`, `V15h`, `V9o`) and the `cancellation.md` swallowing-handler row in `coverage-matrix.md` stay pointed at `V15h` unchanged; only the depth obligation moves. Concretely:

- In `docs/plan_topics/V15h-invoke-ceiling-swallowing.md`: strike the ceiling-#4 depth-surface clauses from `Adds.`, `Tests.`, and `Ships when.` (the `params` / `invoke<T>`-return → `InvokeInfraError` rows carrying `schema_keyword:"maxDepth"`), and remove `V5e` from `Deps.` (leaving `V15h-T`, `V15a`, `V8a`). The retained leaf covers only the `invoke` child swallowing-handler attachment + three-channel late-settlement suppression.
- Create `docs/plan_topics/<new>-...md` and its paired `<new>-T-...md` holding the struck depth-surface obligation: the delegated live-carrier witness for `V5e`'s `params` / `invoke<T>`-return ceiling-#4 routing rows (depth-6 `invoke(...)` `params` → `Err(InvokeInfraError { cause: "validation" })` and depth-6 `invoke<T>` return → `Err(InvokeInfraError { cause: "return_validation" })`, both `schema_keyword: "maxDepth"`, message `"JSON document depth exceeds 5"`; the `CIO-1` slash-load-`params` cross-route note stays with the depth witness). Its `Deps.` are the new test leaf, `V15a`, and `V5e` — matching how `V14e` deps `V14a`, `V5e`.
- In `docs/plan.md`, add a link to the new leaf in the `V15 — Invocation and imports` list.
- In `docs/plan_topics/H5b-warn-only-canary.md`, extend the `V15a`–`V15h` Deps range to include the new leaf's ID.

The implementer assigns the real ID for `<new>` per the plan's leaf-ID scheme (the next free `V15*` slot); do not retarget the four-site routing-set references in `V13f`, `V14f`, `V9o`, `V17a`, or `coverage-matrix.md` row for the swallowing-handler rule — those stay on `V15h`.

## Consequence

**Severity:** cosmetic

`V15h` remains fully implementable as written — both halves are well specified and a single implementer can ship them. The only cost is a larger atomic unit (two unrelated spec anchors and two disjoint dependency sets behind one `Ships when`) and an inconsistency with the tool-call site, where `V14e` and `V14f` already split exactly these two concerns. No implementer divergence, spec mismatch, or blocked pickup results from leaving it bundled.
