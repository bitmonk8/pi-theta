## Finding

The typed-query reference-implementation bullet in `implementation-notes.md` describes the forced-respond turn's tool-choice delivery channel as `complete()`'s `options.toolChoice` (`{ type: "tool", name }`), then states the concrete provider-native payloads that pi-ai's per-provider adapter maps it to: *"Anthropic `tool_choice: { type: "tool", name }`; OpenAI Completions `tool_choice: { type: "function", function: { name } }`; Mistral and Bedrock map analogously."* These payload shapes are owned by each provider's published HTTP API and by pi-ai's adapter mapping — neither of which is in-tree — yet they are written as flat fact, with no pointer to the provider API docs or to pi-ai's adapter as the authoritative mapping, and without any "provider-owned, may drift" hedge.

The corpus already has an established discipline for exactly this class of unpinned, provider-owned fact. `pi-integration-contract/provider-error-mapping.md` carries a *Provider-owned-wording presupposition* that names the wording as provider-owned, observes that a provider can reword or relocate it with no pi-ai version change (so neither the SDK surface-inventory test nor the pi-ai version pin can detect the drift), and routes detection to editorial review via the *Editorial-review checklist for unpinned host presuppositions* in the Pi version-bump procedure. The `tool_choice` payload assertion is the same kind of provider-owned, version-pin-invisible fact but receives none of that treatment, so a provider reshaping its `tool_choice` request payload would leave this bullet silently stale with no re-check obligation.

The normatively load-bearing fact here is only that the runtime forces tool choice through pi-ai's `options.toolChoice` abstraction; the per-provider wire shapes are illustrative and belong behind a pointer to their owners, not asserted as durable spec fact.

The defect has therefore been present since the assertion's inception in `82be125`, not introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

Demote the concrete provider-native payloads to a non-normative illustration and anchor the durable statement to the abstraction and its owners. Keep the load-bearing fact — the runtime forces tool choice through `complete()`'s `options.toolChoice` (`{ type: "tool", name }`) — as the normative claim, and reframe the per-provider shapes as illustrative, pointing to pi-ai's per-provider adapter as the authoritative `options.toolChoice` → provider-native mapping and to each provider's published API docs as the payload owner.

Tie the provider-native payloads into the same drift discipline the rest of the corpus already uses for provider-owned facts: name them as a provider-owned presupposition that the pi-ai version pin cannot detect drift on, and route re-check to the *Editorial-review checklist for unpinned host presuppositions* in the Pi version-bump procedure (mirroring `provider-error-mapping.md`'s *Provider-owned-wording presupposition*). Minimal acceptable form: state only that pi-ai maps `options.toolChoice` to the provider-native form and drop the verbatim shapes entirely; the fuller form keeps the shapes as a non-normative example behind the pointer plus the editorial-review re-check.

## Consequence

**Severity:** advisory

If a provider reshapes its `tool_choice` request payload, the concrete shapes listed here go stale with no re-validation obligation to catch it, because the change is invisible to the pi-ai version pin and the SDK surface-inventory test. The normative behaviour (force tool choice via `options.toolChoice`) is unaffected — implementers can still build correctly against the pi-ai abstraction — so the impact is a guidance/drift hazard rather than a behavioural divergence.
