## Finding

The *Closing-leaf column authoring* rule in `coverage-matrix.md` states that "A leaf-ID left un-delimited is dropped from the tokenised set with no signal", then immediately enumerates two downstream outcomes: a single-leaf cell emptied to the empty set produces a spurious gate failure, while a multi-leaf cell silently stays green with the dropped facet-closer unverified.

"With no signal" is left undefined, and as a blanket modifier of the drop it is in tension with the very next clause. The single-leaf branch does produce a CI signal (the spurious gate failure); only the multi-leaf branch is genuinely silent. So the phrase cannot mean "the gate never reacts". What is actually unspecified is whether the tokeniser emits any diagnostic that identifies the drop itself — a warning, a log line, or a drop-specific gate message naming the un-delimited token. An author debugging a miscounted closing-leaf cell has no stated expectation of what (if anything) the machinery will surface to point at the authoring slip.

The H5d gate leaf (`H5d-transitive-completeness-gate.md`) carries the mirror phrasing ("the gate misreads silently"); `coverage-matrix.md` is the authoritative home of the rule, which H5d explicitly defers to.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `8b69cd9` (2026-06-18) — "pi-loom plan: resolve \"H5d tokeniser depends on unstated backtick authoring convention\""
**History:** The string "with no signal" enters `coverage-matrix.md` only in `8b69cd9`; `git log -S"with no signal"` returns that single commit. The same commit added the mirror "misreads silently" prose to `H5d-transitive-completeness-gate.md`. The defect was introduced by that prior review fix: the backtick-authoring rule it added used the imprecise "no signal" wording from inception rather than scoping the term.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/coverage-matrix.md`, §Closing-leaf column authoring, revise the sentence beginning "A leaf-ID left un-delimited is dropped from the tokenised set with no signal" so the term denotes the absence of a drop-identifying diagnostic, not the absence of any CI reaction. State that the tokeniser silently excludes the un-delimited span and emits no warning, log, or gate message naming the dropped leaf-ID, and keep the two existing downstream-consequence clauses (single-leaf cell → spurious gate failure; multi-leaf cell → green with the facet-closer dropped and unverified) as the distinct effects of that silent drop. Do not adopt a "without any CI failure" framing — the single-leaf branch does redden CI (spuriously), so a blanket no-failure claim would be wrong.

`coverage-matrix.md` owns this rule; the spec is not involved. The mirror prose in `H5d-transitive-completeness-gate.md` ("misreads silently") may be aligned to the same scoping in the same edit, but H5d defers to the `coverage-matrix.md` rule and need not restate the diagnostic detail.

## Consequence

**Severity:** advisory

The H5d tokeniser behaviour is fully specified elsewhere, so no leaf produces a wrong gate; the gap is in the explanatory prose an author consults when a closing-leaf cell tokenises to an unexpected set. Without a defined drop-diagnostic expectation, the author cannot tell whether a missing leaf-ID was silently dropped versus genuinely absent, slowing diagnosis of a miscounted cell.
