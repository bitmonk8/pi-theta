## Finding

DISC-5 defines the candidate universe its `pi.looms` glob/override pipeline operates over: *"When no plain (un-prefixed) include pattern is present, the starting set is every path under the package root; otherwise the starting set is the union of the paths matched by the plain include patterns."* The phrase "every path under the package root" never states whether "under" means the full recursive subtree or only the package root's immediate children.

The depth of this starting set is load-bearing for the final discovered loom set in two ways. First, it governs the exclusion-only manifest case (`pi.looms: ["!test/**"]`, or any manifest carrying only `!`/`+`/`-` operands): the surviving set is exactly whatever "every path under the package root" enumerates, so a recursive reading discovers deep `.loom` files while an immediate-child reading does not. Second, it governs whether a plain `**/*.loom` include can match anything below the top level â€” `minimatch` will expand `**`, but only against paths that are actually present in the candidate universe, so an immediate-child starting set silently neutralises the `**` segment.

The surrounding text pulls in conflicting directions, which is what makes the gap a genuine ambiguity rather than an obvious omission. The page repeatedly emphasises *non*-recursion â€” the `looms/` convention fallback is "scanned non-recursively", a directory *match* expands only to its non-recursive `*.loom` children, and a "global non-recursion rule" is cited at the top â€” yet the matcher is `minimatch` with `**` support and the contract claims to mirror Pi's `pi.extensions`/`applyPatterns` behaviour, which enumerates package files recursively. An implementer can defensibly read the starting set either way, and the two readings produce different discovered loom sets for the same package.

## Solution Space

**Shape:** single

### Recommendation

State explicitly that the no-plain-include starting set is the **recursive** enumeration of every file under the package root, then disambiguate it from the two non-recursion rules it sits next to. Concretely, replace "the starting set is every path under the package root" with wording such as "the starting set is every file at any depth below the package root (a recursive walk)", and add a one-clause note that this recursive candidate universe is distinct from (a) the non-recursive `*.loom` expansion applied to a *directory match*, and (b) the non-recursive `looms/` convention fallback â€” the patterns select from the full tree, but a matched directory still contributes only its immediate `*.loom` children.

The recursive reading is the one consistent with the contract's stated mirror of Pi's `applyPatterns` (whose candidate file set is collected recursively) and the only reading under which the `**` segment of a `**/*.loom` include is meaningful. Implementer edge cases to keep in view: the recursive walk is still bounded by the DISC-5 `maxFiles`/`timeoutMs` limits (a deep tree must not bypass the bound), and the per-match contribution rule (`.loom` file registers / directory â†’ non-recursive children / other-filtered) applies unchanged to whatever survives the override steps.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the discovered loom set for any package whose `pi.looms` carries only override patterns, or any `**`-bearing include pattern: a recursive reading registers deep `.loom` files, an immediate-child reading does not. The divergence is silent (no diagnostic), so it surfaces only as packages whose looms inexplicably fail to load on one conforming implementation.
