## Finding

The DISC-3 Tests bullet in both `V10a-discovery-walk.md` and `V10a-T-discovery-walk.md` mixes a full-path diagnostic-code citation with bare ones in the same line: case collisions fire `loom/load/case-collision` (full path), but the same bullet says non-canonical extensions fire `non-canonical-extension` and an invalid stem fires `invalid-slash-name` — both written without the `loom/load/` prefix. The DISC-4 bullet likewise cites `cross-format-collision` bare.

The diagnostic registry (`docs/spec_topics/diagnostics/code-registry-load.md`) defines all three as fully-qualified codes: `loom/load/non-canonical-extension` (W), `loom/load/invalid-slash-name` (E), and `loom/load/cross-format-collision` (E). The bare forms are not registry codes. An implementer who emits or asserts the literal bare tokens produces code strings that do not exist in the registry, and the diagnostic-code arm of the H5a closing gate — which reconciles asserting tests against the registry's fully-qualified codes — would not find an asserting test for the three registry rows.

The two leaf files carry byte-identical DISC-3/DISC-4 Tests bullets, so the defect is present in both. (The Adds line's prose phrase "cross-source-shadow vs cross-format-collision resolution" is descriptive narrative, not a code-emission citation, and is out of scope.)

## Solution Space

**Shape:** single

### Recommendation

In both `V10a-discovery-walk.md` and `V10a-T-discovery-walk.md`, expand the three bare diagnostic-code citations in the DISC-3 and DISC-4 Tests bullets to their fully-qualified registry strings:

- DISC-3: `non-canonical-extension` → `loom/load/non-canonical-extension`
- DISC-3: `invalid-slash-name` → `loom/load/invalid-slash-name`
- DISC-4: `cross-format-collision` → `loom/load/cross-format-collision`

`loom/load/case-collision` in DISC-3 is already correct; confirm it against the registry but leave it unchanged. The two leaves carry identical DISC-3/DISC-4 bullets, so apply the same substitutions to both. Use the exact strings as they appear in `docs/spec_topics/diagnostics/code-registry-load.md`.

## Consequence

**Severity:** correctness

An implementer reading the bullets literally emits or asserts non-registry code strings (`non-canonical-extension`, `invalid-slash-name`, `cross-format-collision`) for DISC-3/DISC-4, diverging from the spec registry's `loom/load/...` strings. The H5a closing-gate diagnostic-code arm reconciles asserting tests against the fully-qualified registry codes, so the three rows would report no asserting test even when V10a-T exercises them.
