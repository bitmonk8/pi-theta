## Finding

The prompt-mode *Resolution* clause in `conversation-drive.md` states that `ctx.waitForIdle()` "resolves once Pi emits `agent_end` for the user session and every awaited `agent_end` listener settles". The phrase "every awaited `agent_end` listener settles" admits two materially different readings:

1. **Listeners are awaited** — Pi awaits each registered `agent_end` handler's returned promise and `waitForIdle()` resolves only after all of them have settled, so a slow async listener (loom's own persistent `agent_end` handler from PIC-18, or a concurrent subagent's) delays resolution.
2. **Listeners are not awaited** — Pi resolves after synchronous dispatch of `agent_end` regardless of any promise a listener returns, and "awaited" is incidental wording rather than a guarantee.

The clause neither states which reading holds nor backs the asserted behaviour with a citation. This is conspicuous because the same page is otherwise meticulous about exactly this class of unpinned Pi behaviour: the adjacent `waitforidle-no-rejection-presupposition`, `driven-turn-commit-ordering-presupposition`, and `user-visible-streaming-ordering-presupposition` blocks each record a load-bearing `waitForIdle()`/`agent_end` property as a named loom-side consumption posture routed to an editorial-review checklist item (ac/ak/al) for re-confirmation on each Pi minor bump. The listener-settling property the *Resolution* clause depends on is the one such property left as a bare inline assertion.

The two readings differ in the timing relationship between `waitForIdle()` resolution and loom's persistent prompt-mode `agent_end` handlers, and they govern what the runtime may assume about ordering for the post-resolution error-detection probe (PIC-51) and `Ok(string)` extraction (PIC-53) that run immediately after `await ctx.waitForIdle()`. The clause should commit to one reading and — consistent with its three sibling presuppositions — record the property as a consumption posture so an SDK bump re-confirms it.

## Solution Space

**Shape:** single

### Recommendation

Make the *Resolution* clause state a single reading explicitly and record the property as a loom-side consumption posture, mirroring the three sibling presuppositions already on the page:

- Rewrite the listener half of the clause so the verb is unambiguous — e.g. "`waitForIdle()` resolves once Pi has emitted `agent_end` for the user session **and awaited every registered `agent_end` listener's returned promise to settlement**" (reading 1, the reading the existing wording and the d54d798 rationale already imply). If verification against the pinned Pi SDK shows reading 2 holds, state that instead ("Pi does not await listener return values before resolving"); the requirement is a single stated reading, not a particular one.
- Add a `<a id="waitforidle-listener-settling-presupposition">`-style consumption-posture block adjacent to `waitforidle-no-rejection-presupposition`, recording that the listener-settling behaviour is not encoded in `@earendil-works/pi-coding-agent`'s `waitForIdle(): Promise<void>` typed surface and routing re-confirmation to a new lettered item in the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md` (or fold it into the existing `(al)` no-rejection item if the same fixture covers both). Keep `waitForIdle()` itself out of scope as a loom build artefact — loom consumes it; the posture documents only what loom relies on.

Edge case the implementer must watch: loom's own persistent `agent_end` handlers (PIC-18) forward cancellation synchronously, so they do not themselves stall reading 1; the posture's risk surface is a *third-party or concurrent-subagent* async `agent_end` listener under reading 1, and a non-settled loom-relevant handler under reading 2 — state which the runtime tolerates.

## Consequence

**Severity:** advisory

A reader cannot tell whether loom may assume all `agent_end` listeners (including its own persistent handlers) have settled by the time `waitForIdle()` resolves. Because the post-resolution probe's correctness already rests on the separately-pinned commit-ordering posture, an implementer can still produce a working system, but the unrecorded, unpinned assumption escapes the page's own editorial-review re-confirmation discipline, so a Pi minor that changed the listener-settling behaviour would slip through with no build-time or review-time signal.
