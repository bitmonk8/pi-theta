## Problem

The prompt-mode *Resolution* clause in `conversation-drive.md` states that `ctx.waitForIdle()` resolves "once Pi emits `agent_end` for the user session and every awaited `agent_end` listener settles". That phrase admits two materially different readings — Pi awaits each registered `agent_end` listener's returned promise before resolving (so a slow async listener delays resolution), or Pi resolves after synchronous dispatch regardless of any promise a listener returns — and the clause states neither which holds nor backs it with a citation. The three sibling presuppositions on the same page (`waitforidle-no-rejection-presupposition`, `driven-turn-commit-ordering-presupposition`, `user-visible-streaming-ordering-presupposition`) each record their load-bearing `waitForIdle()`/`agent_end` property as a named loom-side consumption posture routed to editorial review for re-confirmation on each Pi minor bump; this listener-settling property is the one such property left as a bare inline assertion. The ambiguity governs what the post-resolution error-detection probe (PIC-51) and `Ok(string)` extraction (PIC-53) may assume about ordering.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commit:** d54d798 — "pi-loom spec: resolve \"waitForIdle resolution, error, and hang semantics not contracted\"" (2026-05-06)
**History:** `git log -S "every awaited"` and `git log -S "every awaited \`agent_end\` listener settles"` over `docs/` both bottom out at d54d798; the pre-image of that commit had no resolution semantics for `waitForIdle` at all (the driver text stopped at "the prompt-mode driver's authoritative completion signal"). d54d798 added the *Resolution* clause verbatim with the phrase "every awaited `agent_end` listener settles". Notably, the finding text that commit was resolving carried an explanatory rationale for the behaviour ("the underlying `activeRun.promise` is constructed with a single `resolve`; provider/transport errors land on session state, not on the promise"), but that rationale was not carried into the normative clause and the property was never recorded as a citation or consumption posture. The phrase has been unchanged since; later commits touching the clause (31ff060 move into `docs/`, f5e89f4 size-cap split, 2d11ac9 PIC-2 relocation) only relocated the text. The defect therefore entered fully formed at d54d798 as a precision loss at introduction, not through a later edit or a multi-commit interaction.

## Solution approach

Rewrite the listener half of the *Resolution* clause so it states a single, unambiguous reading of the `agent_end` listener-settling behaviour, verified against the pinned Pi SDK. Add a consumption-posture block adjacent to `waitforidle-no-rejection-presupposition`, recording that the listener-settling behaviour is not encoded in `@earendil-works/pi-coding-agent`'s typed `waitForIdle(): Promise<void>` surface and routing re-confirmation to a lettered item in the *Editorial-review checklist for unpinned host presuppositions*, mirroring the (ac)/(ak)/(al) routing of the three sibling postures.

## Solution constraints

- Out of scope: `waitForIdle()` as a loom build artefact — loom consumes it, so the edit documents only the property loom relies on as a loom-side consumption posture, not Pi-side implementation behaviour expressed as a Pi normative obligation.
