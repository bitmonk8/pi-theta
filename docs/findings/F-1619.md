## Finding

DIAG-2 (`docs/spec_topics/diagnostics/diagnostic-shape.md`, "Code registry rules (normative)" §2) states that "Adding a new code, removing a code, or changing a code's namespace, severity, or trigger are all spec changes — not implementation changes." Read against the `loom/runtime/internal-error` row in `code-registry-runtime.md`, that rule appears to be violated: the row's *Trigger* cell is deliberately open-ended, enumerating examples ("a host-function `TypeError`, an internal invariant violation, an unanticipated SDK reject …") rather than a closed condition.

The reconciling interpretation — that DIAG-2 "constrains *codes*, not triggers", so registering one stable code with an open-ended trigger is compliant — is normative and load-bearing for both the design and any conformance audit. But it appears in exactly one place: the second sentence of the `code-registry-runtime.md` preamble paragraph that introduces `loom/runtime/internal-error` ("The code is registered and stable; only its trigger condition is intentionally open-ended, which does not weaken the closed-registry rule (rule 2 above) — that rule constrains *codes*, not triggers."). The DIAG-2 rule text itself in `diagnostic-shape.md` does not state this carve-out, and the `internal-error` *Trigger* column in `code-registry-runtime.md` does not state it either. A reader who reaches DIAG-2 directly — from a glossary cross-link, a `loom/typecheck/*` discussion, a parse/load registry, or simply by reading the rules block top-to-bottom — sees a closed-trigger rule with no carve-out, and a reader who lands on the `internal-error` row first sees an open-ended trigger with no rule-text justification on either side of the link. The carve-out is discoverable only from a third entry point (the runtime-preamble's `internal-error` introduction).

## Solution Space

**Shape:** multiple

### Option A — Relocate the carve-out into the DIAG-2 rule text

**Approach.** Move the open-trigger carve-out into DIAG-2 itself in `diagnostic-shape.md`. Replace the runtime preamble's reconciling sentence with a cross-reference to the rule.

**Spec edits.**
- In `docs/spec_topics/diagnostics/diagnostic-shape.md`, extend DIAG-2 with the carve-out, e.g. append: "DIAG-2 constrains *codes* (their identity, namespace, severity, and the trigger *category* a row belongs to); it does not require every row's *Trigger* cell to enumerate a closed set of conditions. A row whose trigger is intentionally open-ended — currently `loom/runtime/internal-error`, whose runtime-defect surface is owned by [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics) — is compliant with DIAG-2 provided the code itself is registered and stable."
- In `docs/spec_topics/diagnostics/code-registry-runtime.md`, in the preamble paragraph introducing `loom/runtime/internal-error`, replace the second sentence with: "The code is registered and stable; its trigger is intentionally open-ended per the DIAG-2 open-trigger carve-out."

**Pros.** Single canonical home for normative rules; readers approaching DIAG-2 cold see the carve-out; the runtime preamble shrinks; future open-trigger codes (if any) inherit the carve-out automatically.
**Cons.** Forward-references one named code (`internal-error`) from the rules block, which the rest of the rules block currently avoids.
**Risks.** None material.

### Option B — Pin the carve-out on the `internal-error` *Trigger* cell

**Approach.** Add the carve-out into the `internal-error` row's *Trigger* column itself, where the open trigger lives. Leave DIAG-2 unchanged and shrink the preamble to a cross-reference.

**Spec edits.**
- In `docs/spec_topics/diagnostics/code-registry-runtime.md`, prepend the *Trigger* cell of `loom/runtime/internal-error` with: "This row's *Trigger* is intentionally open-ended; DIAG-2 constrains the code's identity, namespace, and severity, not the openness of the trigger category — the runtime-defect surface owned by [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics) is the closed concept here, not the enumeration that follows."
- Strip the reconciling sentence from the preamble paragraph; leave only the count/closure framing and the underlying-error-coercion sentence.

**Pros.** The carve-out sits exactly where the apparent conflict is visible (the row itself); DIAG-2 stays terse; no forward reference from rules to a specific code.
**Cons.** A reader who lands on DIAG-2 first still doesn't see the carve-out; the *Trigger* cell — already the longest cell in the registry — grows further.
**Risks.** Future audit tooling that scans rule blocks for the closed-registry property will not surface the carve-out unless it also reads every row's *Trigger* cell.

### Recommendation

Adopt **Option A**. The carve-out is a normative interpretation of DIAG-2, so its canonical home is DIAG-2; readers reach it from every entry point (including the rules-block top-down read and any forward audit), and the runtime preamble — already a long paragraph flagged separately for cruft — gets shorter rather than longer. The one named-code forward reference from the rules block to `internal-error` is acceptable because `internal-error` is the only open-trigger row in the registry today and the rules block already names diagnostic categories (`loom/parse/*`, `loom/load/*`, `loom/runtime/*`, `loom/host/*`).

Edge cases the implementer must watch:
- Keep the carve-out scoped to *open-ended trigger enumeration*; do **not** weaken DIAG-2's MUST-be-spec-change discipline for code additions/removals or namespace/severity changes (those are governed by GOV-15 per the existing DIAG-2 second sentence).
- Preserve the cross-reference to the runtime-defect-surface definition in `errors-and-results/error-model.md#runtime-panics` so the carve-out remains grounded in the framing established by 9d7b1b0; do not duplicate the panic-vs-defect taxonomy into `diagnostic-shape.md`.
- The sibling finding "Runtime-registry preamble duplicates each row's Trigger/Spec-rule cells" recommends keeping "the `internal-error` open-trigger statement" in a trimmed preamble; sequence these two edits so that the trimmed preamble's `internal-error` sentence is the cross-reference Option A leaves behind (not a duplicate of the new DIAG-2 carve-out text).

## Consequence

**Severity:** advisory

An implementer or conformance-test author who reads DIAG-2 in isolation can reasonably conclude that the `internal-error` row's open-ended trigger violates DIAG-2, and the spec's own normative reconciliation is invisible at the point of the apparent conflict. The downstream effect is friction (mis-citation, redundant clarification questions, conformance tests that assert the wrong thing about open-vs-closed triggers) rather than divergent implementations — V18m/V18n and the `internal-error` row itself are unambiguous about behaviour — but the rules block is the natural primary reference for anyone auditing the registry surface, and the carve-out belongs there.
