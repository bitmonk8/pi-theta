## Finding

Version-bump step 5 (`version-bump-triggers.md`) carries a standing build-infrastructure feature requirement inside its prose: *"The implementation MUST implement a build-time audit pass over the same three grep targets (`src/`, `test/`, and `docs/`) and second-stage anchor filter so that a result-delta against the documented enumeration fails `npm test` red rather than relying on contributor-side discipline."* This is not a version-bump-time action — the audit runs on **every** `npm test` invocation (the step itself says the gate "re-runs that enumeration on each `npm test`"), so it is a permanent build-time gate the loom implementation must build and ship, on the same footing as the SDK surface-inventory assertion, the `engines.node` floor gate, the provider seed-field `Api`-coverage assertion, and the strict-capability probe.

Every one of those sibling build-time gates is either anchored as a named obligation or owned by a clearly-labelled section. The closest analogue, the `Inventory-closure audit`, is given its own `<a id="sdk-cap-inventory-closure-audit">` anchor and a "The implementation MUST publish a build-time *inventory-closure audit*…" framing with an explicit behavioural-contract-vs-tooling split (`inventory-audit-intro.md`). The triplet-sweep audit, by contrast, is a single sentence with no anchor, no label, and no REQ-ID, dropped into the middle of an already dense procedural paragraph about reconciling the `SessionShutdownEvent['reason']` snapshot. There is no referenceable handle for it, and a reader scanning the corpus for the loom's standing build-time gates will not find it presented as one.

The consequence is concrete and observable in the plan: the version-bump-static-gates leaf (`V18c`) enumerates the SDK surface-inventory, `engines.node`, peer-dep, snapshot, provider seed-field, and strict-capability gates — but does **not** enumerate the triplet-sweep build-time audit among them, precisely because the obligation is buried and unanchored rather than surfaced as a named gate. The fix is to give the audit-pass obligation a named, anchored home (mirroring the `Inventory-closure audit` treatment) so it is discoverable and referenceable as the standing build-time gate it is.

**History:** The inbound-reference sweep and its build-time audit pass entered the step-5 prose at `52b9101`, where the audit was an **optional** clause — *"The H1 leaf … MAY additionally implement a build-time audit pass over the same three grep targets … PIC does not require the audit"* — embedded in the procedural paragraph from the outset. `c0259ca` then promoted the clause from `MAY` to `MUST` (*"The implementation MUST implement a build-time audit pass …"*), converting an optional aside into a mandatory feature requirement while leaving it in the same unanchored mid-paragraph position. The placement was always somewhat off; the `MAY`→`MUST` promotion is what turned it into a feature-requirement-buried-in-a-procedure defect. Both commits resolved unrelated findings, so the obligation rode along without ever being given its own anchor or named-obligation framing. Later structural commits (`f5e89f4` page-split under the size cap, `81bf52a` collapse into `spec_topics/`) only relocated the file; they did not introduce the defect.

## Solution Space

**Shape:** multiple

### Option A — Anchor the audit-pass obligation in place

- **Approach:** Promote the existing build-time audit-pass sentence in step 5 to a named, anchored obligation — add an `<a id="...">` anchor and a bold label (e.g. **Triplet-sweep build-time audit gate**), and frame it with the same behavioural-contract-vs-tooling split the `Inventory-closure audit` uses (PIC pins the gate's behaviour; concrete tooling and discriminator strings are implementation-owned). Leave it physically in step 5, since the audit exists to mechanically back the step's inbound-reference sweep.
- **Spec edits:** Wrap the audit-pass sentence in an anchored, labelled mini-block; reference the new anchor from the surrounding sweep prose and from the "build-time audit gate (pinned later in this step)" forward-reference earlier in the step.
- **Pros:** Minimal disruption; keeps the audit adjacent to the sweep it enforces, preserving cohesion; gives the gate a stable referenceable handle for `V18c` and for the sibling complement-grep finding.
- **Cons:** The audit remains physically inside a long procedural paragraph, so it is still less prominent than a dedicated audit-page section.
- **Risks:** Low — purely additive anchoring/labelling, no behavioural change.

### Option B — Relocate the obligation to the audit-page family

- **Approach:** Move the audit-pass *requirement* to sit alongside the `Inventory-closure audit` (in `inventory-audit-intro.md` or the audit-target pages), where the corpus already homes build-time audit features, leaving a cross-reference in step 5's sweep prose pointing back to the canonical obligation.
- **Spec edits:** Add a labelled, anchored "session-only-reason sweep audit" obligation on the audit page; replace the inline step-5 MUST with a cross-reference to it.
- **Pros:** Co-locates all build-time audit features in one place, maximising discoverability for an implementer scoping the gate set.
- **Cons:** Splits a tightly-coupled obligation from the sweep it enforces; the audit's grep targets, anchor filter, and exclusion list are defined entirely within step 5, so relocating only the requirement scatters the contract across two pages.
- **Risks:** Cross-reference drift between the audit page and step 5's sweep mechanics.

### Recommendation

Take **Option A**. The audit's entire contract — grep targets, second-stage anchor filter, documented enumeration, exclusion list — lives in step 5 and is meaningless apart from the sweep it backs, so anchoring it in place gives it a referenceable handle without scattering the contract. Watch one edge case: the new anchor must be wired into both the earlier "build-time audit gate (pinned later in this step)" forward-reference and `V18c`'s gate enumeration, so the gate is no longer reachable only by reading the paragraph in full.

## Consequence

**Severity:** advisory

A standing build-time gate the loom must ship is presented as an inline clause in a version-bump procedure rather than as a named requirement, so it is easy to under-scope or miss when planning build infrastructure — as evidenced by its absence from `V18c`'s enumerated gate set. The audit's behaviour is fully specified in place, so an implementer who reads the whole step can still build it correctly; the gap is discoverability and referenceability, not buildability.
