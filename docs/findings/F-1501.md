## Finding

§2 Runtime-value placeholders (the rule governing `<scrutinee summary>` and `<value>`) delegates rendering to the canonical interpolation-stringification table in `query/query-escapes-stringification.md` (QRY-18), then layers exactly one extension and one supplementary case on top: a normative 80-Unicode-code-point cap (truncated to 77 code points + literal `...`) on the `string` row, and an `Ok(<inner>)`/`Err(<inner>)` form for the `Result<T, E>` row. No bound is layered on the `array<T>` or *Schema-typed object* rows, which delegate verbatim to compact `JSON.stringify` with recursive wire-name translation.

The §2 test vector itself makes this concrete: `MatchError: no arm matched {"name":"fluffy"}` is the byte-identical rendering for a schema-typed-object scrutinee, with no truncation step in the rule's path. A `loom/runtime/match-error` (or any future `<scrutinee summary>` / `<value>` emitter) on a deeply-nested or large-array runtime value emits a `MatchError: no arm matched <multi-megabyte JSON blob>` as its single normative message string. The same string flows unchanged through every routing surface in `errors-and-results.md` Panic-message rule — the user-facing system-note, the `InvokeFailure.message` field that `?` propagates, the operator-facing diagnostic channel — and is byte-asserted by DIAG-4 / `V7c` conformance tests, so the unbounded form is not just permitted but pinned.

The boundedness posture elsewhere is consistent: §6 truncates underlying-error strings to their first line; the §2 sibling `string` row caps at 80 code points; the transport-failure note caps `<message>` at 120 code points; the sibling finding on `details.observed` recommends a 256-UTF-16-code-unit cap. §2's object/array path is the lone unbounded normative emission site for a `<…>` placeholder governed by Category 2.

## Issue introduction

**Verdict:** single-introducing-commit
**Introducing commits:** `61ea1a5` (2026-05-06) — `pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"`
**History:** Commit `61ea1a5` introduced §2 Runtime-value placeholders into the then-monolithic `spec_topics/diagnostics.md`, delegating object/array rendering to the canonical interpolation-stringification table (which already specified compact `JSON.stringify` for arrays and schema-typed objects, added in `0c9ce54` on 2026-05-04) and layering the string-only 80-code-point truncation rule on top. Object and array scrutinees have been governed by an unbounded `JSON.stringify` path since that commit. Commit `055ab2f` (2026-06-01, *§2 Runtime-value placeholder rule contradicts its own Cat { name: "fluffy" } test vector*) made the gap visually evident by replacing the misleading `Cat { name: "fluffy" }` test vector with the schema-typed-object form `{"name":"fluffy"}` that actually exercises the unbounded JSON path, but did not address it. Commit `f5e89f4` (2026-06-04) split `diagnostics.md` into the subdirectory layout and carried §2 across into `placeholder-rendering-a.md` byte-identically.

## Solution Space

**Shape:** multiple

### Option A — Bound the rendered output

**Approach.** Extend §2's "String truncation" bullet so the 80-code-point cap applies to the result of the canonical stringification table for *every* row that produces a textual rendering — `string`, `array<T>`, *Schema-typed object*, and the recursive `<inner>` of `Result<T, E>` — measured on the post-render UTF-8 string in Unicode code points, with the same `77 + literal "..."` truncation form.

**Spec edits.**
- Replace the current `**String truncation.**` bullet in `placeholder-rendering-a.md` §2 with a `**Length bound.**` bullet that states the cap applies to the rendered output of the canonical stringification table (whatever row produced it), preserving the byte-identical 80-code-point cap, 77-code-point prefix, and literal `...` suffix and the code-point counting basis.
- Update the §2 test vectors: the schema-typed-object vector keeps the short example (`{"name":"fluffy"}`) and adds a normative boundary vector for a schema-typed-object whose compact JSON exceeds 80 code points (renders to 77 code points + `...`); the existing 100-character ASCII string vector is unchanged.
- In `placeholder-rendering-b.md` *Edge cases* → *Category 2 boundary*, generalise the boundary obligation from "string scrutinee" to "any §2-rendered scrutinee" so the 80/81-code-point boundary test obligation covers schema-typed-object and `array<T>` renderings too.

**Pros.** Behaviour is consistent with every other bounded `<…>` emission site (§6 underlying-error first-line, sibling `string` row, transport-failure `<message>` 120-CP cap, the proposed `details.observed` 256-code-unit cap). One rendering rule, one truncation form, byte-stable across all rows. `InvokeFailure.message` and system-note surfaces are protected from multi-MB payloads by construction.

**Cons.** Truncating at a code-point boundary in the middle of a compact JSON serialisation produces lexically-invalid JSON (e.g. `{"name":"fluf...`), so the rendered tail is no longer parseable as a value. The truncation is for human display, so this is acceptable, but it should be noted explicitly.

**Risks.** A naïve implementation that truncates by UTF-16 code unit (e.g. JavaScript `.slice`) silently violates the rule on supplementary-plane code points; this is the same risk §2's existing `string` row carries and the same boundary test catches it.

### Option B — Pin the rendering as unbounded, document at every emission site

**Approach.** Leave compact `JSON.stringify` rendering unbounded for `array<T>` and *Schema-typed object* `<scrutinee summary>` / `<value>` renderings, but state the choice explicitly in §2 and require every `<scrutinee summary>` / `<value>` consumer (`loom/runtime/match-error`, `loom/runtime/custom-type-unsafe`, plus the routing through `InvokeFailure.message`, the user-facing system-note, and the operator-facing channel) to document that the field may carry an arbitrarily large payload.

**Spec edits.**
- In `placeholder-rendering-a.md` §2, add a normative paragraph stating that the canonical stringification table's `array<T>` and *Schema-typed object* rows render without a length bound, and that consumer surfaces are obliged to handle arbitrarily large message strings without truncating mid-codepoint or dropping the field.
- Cross-reference the unbounded rendering from `errors-and-results/error-model.md`'s Panic-message-string table (so a reader of `loom/runtime/match-error` sees the field's size posture).

**Pros.** No change to the byte-identical fixtures or to the canonical stringification table; preserves the "panic message is the rendered scrutinee" property losslessly.

**Cons.** Diverges from every other `<…>` placeholder's boundedness posture, and pushes the size-handling obligation downstream onto every consumer surface (system-note rendering, `InvokeFailure.message` propagation, log scrapers) — most of which already cap or sanitise. The user-facing system-note `loom /<name> aborted: <message>` becomes the de-facto carrier of unbounded payloads.

**Risks.** A downstream surface that *does* cap (the existing transport-failure note caps at 120 code points) silently mis-renders panic messages, defeating the byte-identical conformance contract at the surface boundary even when §2 itself is byte-identical.

### Recommendation

Option A. The byte-identical rendering surface is normatively cap-bounded for every other `<…>` placeholder of comparable shape; extending the §2 cap to apply post-render (covering the `array<T>` and *Schema-typed object* rows that delegate to compact `JSON.stringify`, plus the recursive `<inner>` of `Result<T, E>`) is the change that makes §2 self-consistent and aligns it with §6, the sibling `string` row, and the consumer surfaces. Implementer must watch: (1) truncation operates on the post-stringification UTF-8 string measured in Unicode code points, not on the pre-stringification value or on UTF-16 code units — the same counting basis as the existing `string`-row rule; (2) the boundary test vector at 80/81 code points must be authored for at least one non-string row to lock the post-render measurement basis; (3) the resulting truncated form is not required to be valid JSON.

## Consequence

**Severity:** correctness

Two conformant implementations diverge: one applies an 80-code-point cap to the compact-JSON rendering of an object/array scrutinee, the other emits the value verbatim. The cap-applying implementation fails the existing `{"name":"fluffy"}` byte-identical test vector unless the test vector is also restated; the no-cap implementation can emit multi-megabyte `MatchError` strings that propagate through `InvokeFailure.message`, the user-facing system-note channel, and the operator-facing diagnostic channel. The behaviour also leaks into `loom/runtime/custom-type-unsafe`, which renders its offending `customType` per the same §2 `<value>` rule.
