## Finding

§2 Runtime-value placeholders (the rule governing `<scrutinee summary>` and `<value>`) delegates rendering to the canonical interpolation-stringification table in `query/query-escapes-stringification.md` (QRY-18), then layers exactly one extension and one supplementary case on top: a normative 80-Unicode-code-point cap (truncated to 77 code points + literal `...`) on the `string` row, and an `Ok(<inner>)`/`Err(<inner>)` form for the `Result<T, E>` row. No bound is layered on the `array<T>` or *Schema-typed object* rows, which delegate verbatim to compact `JSON.stringify` with recursive wire-name translation.

The §2 test vector itself makes this concrete: `MatchError: no arm matched {"name":"fluffy"}` is the byte-identical rendering for a schema-typed-object scrutinee, with no truncation step in the rule's path. A `loom/runtime/match-error` (or any future `<scrutinee summary>` / `<value>` emitter) on a deeply-nested or large-array runtime value emits a `MatchError: no arm matched <multi-megabyte JSON blob>` as its single normative message string. The same string flows unchanged through every routing surface in `errors-and-results.md` Panic-message rule — the user-facing system-note, the `InvokeFailure.message` field that `?` propagates, the operator-facing diagnostic channel — and is byte-asserted by DIAG-4 / `V7c` conformance tests, so the unbounded form is not just permitted but pinned.

The boundedness posture elsewhere is consistent: §6 truncates underlying-error strings to their first line; the §2 sibling `string` row caps at 80 code points; the transport-failure note caps `<message>` at 120 code points; the sibling finding on `details.observed` recommends a 256-UTF-16-code-unit cap. §2's object/array path is the lone unbounded normative emission site for a `<…>` placeholder governed by Category 2.

## Issue introduction

**Verdict:** single-introducing-commit
**Introducing commits:** `61ea1a5` (2026-05-06) — `pi-loom spec: resolve "Diagnostic message placeholders have no defined rendering"`
**History:** Commit `61ea1a5` introduced §2 Runtime-value placeholders into the then-monolithic `spec_topics/diagnostics.md`, delegating object/array rendering to the canonical interpolation-stringification table (which already specified compact `JSON.stringify` for arrays and schema-typed objects, added in `0c9ce54` on 2026-05-04) and layering the string-only 80-code-point truncation rule on top. Object and array scrutinees have been governed by an unbounded `JSON.stringify` path since that commit. Commit `055ab2f` (2026-06-01, *§2 Runtime-value placeholder rule contradicts its own Cat { name: "fluffy" } test vector*) made the gap visually evident by replacing the misleading `Cat { name: "fluffy" }` test vector with the schema-typed-object form `{"name":"fluffy"}` that actually exercises the unbounded JSON path, but did not address it. Commit `f5e89f4` (2026-06-04) split `diagnostics.md` into the subdirectory layout and carried §2 across into `placeholder-rendering-a.md` byte-identically.

## Solution Space

### Recommendation

Apply the 80-Unicode-code-point cap to the rendered output of the canonical interpolation-stringification table for *every* row that produces a textual rendering — `string`, `array<T>`, *Schema-typed object*, and the recursive `<inner>` of `Result<T, E>` — measured on the post-render UTF-8 string in Unicode code points, using the same `77 code points + literal "..."` truncation form already specified for the `string` row.

Spec edits:
- Replace the current `**String truncation.**` bullet in `placeholder-rendering-a.md` §2 with a `**Length bound.**` bullet stating that the cap applies to the rendered output of the canonical stringification table whatever row produced it, preserving the 80-code-point cap, the 77-code-point prefix, the literal `...` suffix, and the Unicode-code-point counting basis byte-identically.
- Update the §2 test vectors: keep the short schema-typed-object example (`{"name":"fluffy"}`) and add a normative boundary vector for a schema-typed-object whose compact JSON exceeds 80 code points (rendering to 77 code points + `...`); leave the existing 100-character ASCII string vector unchanged.
- In `placeholder-rendering-b.md` *Edge cases* → *Category 2 boundary*, generalise the boundary obligation from "string scrutinee" to "any §2-rendered scrutinee" so the 80/81-code-point boundary test covers schema-typed-object and `array<T>` renderings too.

Implementer constraints:
- Truncation operates on the post-stringification UTF-8 string measured in Unicode code points — not on the pre-stringification value, and not on UTF-16 code units (a `.slice`-style UTF-16 truncation silently mis-handles supplementary-plane code points). This is the same counting basis as the existing `string`-row rule.
- Author the 80/81-code-point boundary test vector for at least one non-string row to lock the post-render measurement basis.
- The resulting truncated form is not required to be valid JSON: truncating mid-serialisation (e.g. `{"name":"fluf...`) yields a lexically-invalid fragment, which is acceptable because the rendering is for human display. State this explicitly in the rule.

## Consequence

**Severity:** correctness

Two conformant implementations diverge: one applies an 80-code-point cap to the compact-JSON rendering of an object/array scrutinee, the other emits the value verbatim. The cap-applying implementation fails the existing `{"name":"fluffy"}` byte-identical test vector unless the test vector is also restated; the no-cap implementation can emit multi-megabyte `MatchError` strings that propagate through `InvokeFailure.message`, the user-facing system-note channel, and the operator-facing diagnostic channel. The behaviour also leaks into `loom/runtime/custom-type-unsafe`, which renders its offending `customType` per the same §2 `<value>` rule.
