## Finding

The tests-task gate in `conventions.md` §Per-phase TDD ritual ships a leaf's `<id>-T-complete` tag only once the suite "fails red for the intended reason" — i.e. each test reds out on its own primary assertion because the implementation under test is absent or incomplete, **not** on a compile error, a missing/unconfigured fixture, a harness/setup throw, or an unrelated exception. Distinguishing a behavioural red from a non-behavioural red is a manual judgment made by the contributor at tagging time.

Nothing observable records that the judgment was made or what its evidence was. The `<id>-T-complete` git tag — the authoritative pickability signal downstream leaves depend on — carries no trace of *why* the suite was red. Once the tag exists, a later reviewer (or the contributor on the paired implementation task) cannot confirm the red was behavioural rather than a compile-time or harness red that was mis-classified. The companion clarity finding sharpens *what* "intended reason" means; this finding is about the absence of any durable evidence that the gate was satisfied for the right reason, independent of how precisely the criterion is worded.

This is the same `No silent test skipping` failure mode the plan elsewhere works hard to foreclose: a tests task that red for a non-behavioural reason but was tagged `<id>-T-complete` reports success when nothing about the absent implementation was actually witnessed, and there is no after-the-fact way to catch it.

## Solution Space

**Shape:** multiple

### Option A — Recommended commit-message record (non-gating)

**Approach:** In `conventions.md` §Per-phase TDD ritual, under the tests-task description, add a recommendation that the `<id>-T` commit message records the observed failing assertion message for each test, so the intended-reason determination is auditable after the fact. Keep it a recommendation in the ritual body, not part of the **Ships when** clause.
**Plan edits:** Edit `docs/plan_topics/conventions.md` only; add the recommendation alongside the existing tests-task text. The **Ships when** wording stays "the tests exist, compile, and fail red for the intended reason as defined above".
**Spec edits:** None.
**Pros:** Near-zero cost; produces a durable audit trail that lets the paired implementation task and any reviewer confirm each red was behavioural; no `<id>-T` leaf page changes.
**Cons:** Still contributor-discipline (a contributor can record nothing or record inaccurately); no mechanical gate enforces the record.
**Risks:** Minimal — purely additive guidance.

### Option B — Acknowledge as accepted contributor-discipline residue

**Approach:** Add an explicit sentence to `conventions.md` §Per-phase TDD ritual stating that the intended-reason determination is a manual judgment with no observable record, accepted as contributor-discipline residue (parallel to the manual-review residues the cross-cutting rules already route to self-review and the release-time residue inspection), and add no new mechanism.
**Plan edits:** Edit `docs/plan_topics/conventions.md` only; one acknowledgement sentence.
**Spec edits:** None.
**Pros:** Smallest possible diff; makes the gap an intentional, documented decision rather than an unstated one.
**Cons:** Provides no audit trail; the verification weakness remains, only now it is named.

### Recommendation

Adopt Option A. Recording the observed failing assertion message in the `<id>-T` commit gives a cheap, durable audit trail and keeps the record non-gating, so it does not force a **Ships when** edit across every `<id>-T` leaf. Edge case for the implementer: the record is evidence, not a gate — do not wire it into the closing-gate automation or the pickability predicate, and do not promote it into the **Ships when** clause (doing so would ripple into every tests-task leaf for no enforcement benefit).

## Consequence

**Severity:** advisory

A contributor or reviewer cannot later verify that an `<id>-T-complete` tag was earned by a behavioural red rather than a mis-classified compile-time or harness red; the gate's correctness rests entirely on unrecorded judgment. A tests task tagged on a non-behavioural red ships undetected, weakening the TDD evidence chain the rest of the plan relies on. Implementers can still produce working leaves, so this degrades audit confidence rather than blocking work.
