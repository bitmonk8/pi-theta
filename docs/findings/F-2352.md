## Problem

The spec's `active-invocation-registry.md` §"Registry contract → Dispatch-site setup wrap" makes the setup wrap a hard `MUST NOT mask` contract at each of the three insertion sites, mandating three failure-path behaviours: a setup throw/rejection routes through the runtime-defect surface (`loom/runtime/internal-error` / `InvokeInfraError { cause: "internal_error" }`), a throw before the registry `Set.add` completes leaks no entry, and a cleanup-arm `loomAbort.abort()` throw is dropped without masking the original setup throw. Neither `V9e-T` nor `V9e` carries a Tests bullet asserting any of these; the three existing bullets cover only the happy path (insertion-order iteration, `disposeBarrier` settlement, `invocationId` allocation), and the `Ships when` condition speaks only to teardown order. With the DIAG-2 reverse-gate binding only at the H6a release footing, `V9e` can be marked done with the masking-prevention, no-leak, and internal-error-routing contract unimplemented or wrong.

The later edits to the pair (75b6a9b, 659aa21, cb8737a) reworded the happy-path bullets and Deps but never added setup-wrap failure-path coverage; the gap has existed since the leaf first appeared.

## Solution approach

Add a setup-wrap failure-path Tests bullet to `V9e-T`, mirrored into `V9e`'s Tests per the paired-leaf convention, citing `active-invocation-registry.md` §"Registry contract → Dispatch-site setup wrap". The coverage should drive a setup-sequence throw and rejection at an insertion site and assert the three contract behaviours: routing through the runtime-defect surface, no leaked registry entry when the throw precedes `Set.add`, and the original setup throw surviving a cleanup-path `loomAbort.abort()` throw. Assert on observable side effects (the emitted diagnostic/error surface and the entry-count probe), consistent with the leaf's existing "registry name is internal" testing posture.
