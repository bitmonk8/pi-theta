## Finding

`H7a` is the terminal integration-acceptance gate. Its **Adds**, its first **Tests** bullet, and its **Ships when** all enumerate the driven pipeline as `typed query → tool loop → code-tool invoke → schema lowering/validation → binder → cancellation`, naming **schema lowering/validation** as a composed stage the single fixture run exercises. Its **Deps** are `H4a, H4b, V5d, V8a, V11a, V11b, V11i, V11c, V11d, V11e, V11f, V13a, V13b, V13c, V13d, V13f, V14a, V14b, V16a, V17a`.

The *validation* half of that stage is unbacked. `V8c` (the `SchemaValidator` seam, PIC-11) is absent, and so is `V11g` (fill-if-absent defaulting and post-merge AJV validation), the binder leaf that actually re-validates merged `args` through `SchemaValidator.validate()` — `V11g`'s own **Deps** are `V11g-T, V11a, V5d, V5f, V8c`. A typed-query run that drives the binder to structured output inherently exercises this validation path; with neither `V8c` nor `V11g` declared, H7a either (a) does drive validation, in which case both are undeclared dependencies and H7a's pickability can fire before they are complete, or (b) does not drive validation, in which case "schema lowering/validation" overstates what the terminal gate composes. The leaf states neither.

The *lowering* half is already covered: `V5f` (schema lowering and canonical hash) is transitively pulled in through `V13b` (Query schema inference), whose **Deps** include `V5f`. So the real omission is the validation seam and its binder consumer, not lowering — though H7a's **Deps** already redundantly list transitive entries (`V5d`, a dependency of `V13b`, is named directly), so the absence of `V8c`/`V11g` is a substantive gap rather than a stylistic one.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `e7e51cc` — "pi-loom plan: resolve \"Plan has no terminal end-to-end integration-acceptance leaf\"" (2026-06-10)
**History:** The H7a leaf was created in `e7e51cc` with the `schema lowering/validation` pipeline prose already present and a Deps set (`H4a, V5d, V8a, V11f, V13c, V14a, V17a`) that already omitted `V8c`. `git log -S 'V8c' -- docs/plan_topics/H7a-integration-acceptance.md` and `git log --all -G 'Deps.*V8c' -- <same>` both return empty: the `V8c` token has never appeared in H7a's Deps in any revision. The Deps set was expanded across later commits (adding H4b, V8a, V11a–e, V13a/b/d/f, V14b, V16a) but the validation seam was never added. The mismatch between the named validation stage and the omitted seam is original to the leaf, not introduced by a later edit.

## Solution Space

**Shape:** multiple

### Option A — Declare the validation dependencies

**Approach:** Keep the `schema lowering/validation` pipeline prose; add the leaves that back the validation half to H7a's **Deps**.

**Plan edits:** Add `V8c` (the SchemaValidator seam) and `V11g` (post-merge AJV validation) to `docs/plan_topics/H7a-integration-acceptance.md`'s **Deps**, alongside the existing entries. `V5f` is already transitively pulled in via `V13b`; list it directly too for consistency with the leaf's existing practice of naming transitive entries (it already lists `V5d`). Because the permitted-code list is a per-Deps-slice union, fold any `loom/...` codes `V8c`/`V11g` can emit (e.g. `validator-cache-collision`) into that union per H7a's existing per-Deps-slice provenance obligation.

**Spec edits:** None.

**Pros:** Makes the stated pipeline honest; the terminal gate genuinely composes and verifies model-output validation; pickability correctly orders H7a after the validation seam is built.

**Cons:** Widens an already-large Deps set and the permitted-code union.

**Risks:** If the H7a fixture's typed query does not in fact route through `V11g`'s post-merge AJV path, the added `V11g` edge is spurious.

### Option B — Narrow the stated pipeline

**Approach:** Strike the validation claim so the prose matches the composed Deps.

**Plan edits:** Remove `validation` (or `schema lowering/validation`) from the pipeline enumeration in H7a's **Adds**, first **Tests** bullet, and **Ships when**, leaving the stages the Deps actually back.

**Spec edits:** None.

**Pros:** Smallest diff; no Deps growth; removes the overclaim with no new edges.

**Cons:** The terminal cross-slice gate then never witnesses the schema-validation composition — a real integration surface left uncovered end-to-end.

**Risks:** Under-narrowing — if the fixture does drive validation, the prose would understate what runs.

### Recommendation

Take **Option A**. A typed-query integration run drives the binder to produce structured output, which is re-validated through `V11g` → `V8c.validate()`; the validation stage is genuinely exercised, so the honest fix is to declare `V8c` and `V11g` (and explicitly `V5f`) in Deps rather than delete the claim. Edge case the implementer must resolve first: confirm the H7a fixture's typed query actually reaches `V11g`'s post-default-merge AJV path — if the fixture is deliberately constructed to avoid structured-output validation, switch to Option B instead. When adding the deps, reconcile the permitted-code list union (see the related permitted-code findings below).

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one authors an H7a fixture whose typed query drives binder validation, then finds H7a is marked pickable before `V8c`/`V11g` are complete and the integrated run cannot compose; another authors a run that skips validation, leaving the named `schema lowering/validation` stage unexercised so the terminal gate vacuously certifies a composition it does not actually drive — the exact cross-slice gap H7a exists to catch.
