## Problem

The enum-violations Tests bullet in both `V5a-schema-decls.md` and `V5a-T-schema-decls.md` qualifies five enum diagnostic codes with the bare parenthetical `(name-first ordering)`. That phrase admits two readings: the ordering of variant names in the rendered diagnostic message (a message-text assertion), or the spec's check-precedence rule that the name-duplication check runs before the value-duplication check. These produce materially different tests. The spec's intended meaning is the check-precedence rule (`schemas.md` §Enum declarations), but the two-word parenthetical does not recover it, leaving the discriminating case — which code a distinct-explicit-value name collision fires — to the implementer's guess.

The ambiguous parenthetical was present in the leaf pair's first and only revision — it entered with the leaves rather than being introduced by a later edit.

## Solution approach

In the enum-violations Tests bullet of both `V5a-schema-decls.md` and `V5a-T-schema-decls.md`, replace the `(name-first ordering)` parenthetical with text that names the check-precedence rule from `schemas.md` §Enum declarations: the name-duplication check fires before the value-duplication check. Pin the discriminating case the current wording misses — a distinct-explicit-value name collision (e.g. `enum X { Low = "a", Low = "b" }`) emits `duplicate-enum-variant-name`, while `duplicate-enum-value` is reserved for distinct names sharing one explicit value. Draw the literal example strings from the spec rather than inventing them.

## Solution constraints

- The two leaves carry byte-identical bullets and must be edited in lockstep.
- The spec is read-only for this fix; cite or paraphrase the existing `schemas.md` rule rather than restating it normatively in the leaf.
