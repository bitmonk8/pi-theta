## Finding

`V18c`'s **Ships when** condition splits the version-bump gate into two arms: a mechanical arm (`npm run typecheck` + `npm test` over the SDK surface-inventory, `engines.node`, peer-dep, provider seed-field, and strict-capability gates) and an editorial arm. The editorial arm's acceptance is "confirmed by editorial review — not by `npm test`" that the bump commit message records a per-item outcome for every checklist item and that no MUST-level item carries a `fail`. The semantic audit (judging whether each host presupposition still holds against the candidate Pi minor) genuinely requires human judgement and is correctly routed to editorial review.

The defect is narrower: the *completeness and shape* of the per-item record is a purely mechanical property, yet nothing checks it mechanically. `version-bump-step2.md` makes the record mandatory — "the contributor MUST record the per-item audit outcome … in the bump commit message" and "omitting the row is non-conformant" — and pins an exact recording shape (`pass` / `fail` / `N/A` / `not-performed` + one-line rationale, with item `(e)` recording two sub-outcomes `(e.i)`/`(e.ii)`). A reviewer eyeballing a long commit message can silently miss an omitted item row, a malformed sub-outcome, or a MUST-level `fail` slipping through as a non-blocking outcome. Because the only check is human reading, a non-conformant record can pass the gate even though the spec forbids it.

A commit-message lint over the bump commit can enforce the mechanical half (every required item ID present and well-formed; no MUST-level item `(a)`–`(e)` records `fail`) and fail closed on a missing or malformed ID, leaving the semantic verdict to editorial review. This is a complement to editorial review, not a replacement.

## Solution Space

**Shape:** single

### Recommendation

Add a mechanical completeness/shape gate over the bump commit message to `V18c`, with a paired failing test in `V18c-T`, alongside the existing `npm test` static gates. The gate asserts, over the recorded per-item audit outcomes in the bump commit message:

- Every required editorial-checklist item ID defined by `version-bump-step2.md` is present, including item `(e)`'s two sub-outcomes `(e.i)` and `(e.ii)`. Source the required item-ID set from `version-bump-step2.md` rather than a hand-copied list, so it tracks the spec's actual extent (the corrected range — see Related Findings).
- Each recorded outcome is one of the spec-defined tokens (`pass` / `fail` / `N/A` / `not-performed`, plus item `(a)`'s authoritative-determination shape and item `(e)`'s `N/A — superseded by <named Pi mechanism>` form) with its one-line rationale present.
- No MUST-level item `(a)`–`(e)` records a `fail` outcome (item `(e)` failing on either sub-outcome), since a MUST-level `fail` blocks the bump.

The gate fails closed: a missing item ID, a malformed/unrecognised outcome token, or a MUST-level `fail` reds the gate. State explicitly that this mechanical gate complements — does not replace — editorial review: the semantic verdict on each presupposition stays editorial, and the new gate only enforces the record's completeness and shape.

Reflect the new gate in `V18c` **Adds** (it becomes a mechanical build-time gate, distinct from the editorial audit it backstops), add the asserting bullet to `V18c` / `V18c-T` **Tests**, and add the gate's green condition to `V18c` **Ships when** while retaining the editorial-review confirmation clause for the semantic audit.

## Consequence

**Severity:** advisory

A non-conformant bump commit message — an omitted item row, a malformed `(e.i)`/`(e.ii)` sub-outcome, or a MUST-level item carrying `fail` — can pass the version-bump gate when the human reviewer overlooks it, even though `version-bump-step2.md` declares such records non-conformant. The mechanical gate arm is green, so the only thing standing between a defective record and a merged bump is reviewer attention over a long, dense commit message.
