## Finding

`cancellation.md` §Granularity states a normative obligation: the interpreter checks the cancellation signal immediately before each of the five checkpoint sites (each `for`/`while` iteration, each `@`-query dispatch, each tool call, each `invoke`, and the slash-command argument binder's LLM call) "and no others", together with the loop-iteration macrotask-yield property. `V17c` (Checkpoint granularity) is the closing leaf for this obligation — its `Adds`, `Tests`, and `Ships when` all target `cancellation.md` §Granularity, and it is reachable from `H5b`'s `Deps.` through the `V17a`–`V17c` range entry.

That obligation has no traceability row anywhere in `coverage-matrix.md`. The *Numbered REQ-IDs* table maps `CNCL-1`…`CNCL-6` to `V17a` (Race-semantics and abort-reason obligations), none of which is §Granularity. The *Code-keyed obligation areas* table carries two `cancellation.md` rows — `cka-32` (§Forwarding-listener throw → `V17b`) and `cka-33` (§Race semantics — swallowing-handler → `V17a` + per-site co-witnesses) — but neither covers §Granularity. The string "Granularity" has never appeared in `coverage-matrix.md`.

Because §Granularity is phrased without a literal `MUST`/`MUST-NOT` token, `H5e`'s un-anchored-MUST text scan does not surface it as a candidate needing a row, and `H5d`'s transitive-completeness arm only validates rows that already exist. The obligation therefore has no closing-gate traceability hook at all: an auditor following the coverage matrix finds no row mapping §Granularity → `V17c`, and the matrix-driven gates pass vacuously for it.

### Recommendation

Add one row to the *Code-keyed obligation areas (no numbered REQ-IDs)* table in `docs/plan_topics/coverage-matrix.md`, taking the next free append-only token (`cka-47` at the current table tail, which ends at `cka-46`; the *Token* column is append-only — never reuse or renumber an existing token). The row's spec-area cell names `cancellation.md` (CNCL) §Granularity — the fixed five-site checkpoint set (each `for`/`while` iteration, each `@`-query dispatch, each tool call, each `invoke`, and the slash-command argument binder's LLM call, and at no other node kinds) together with the loop-iteration macrotask-yield property — and its closing-leaf cell is `` `V17c` ``.

Honour the *Closing-leaf column authoring* rule: `V17c` is the only backtick-delimited span in the closing-leaf cell, and any code identifiers in the spec-area annotation prose are written bare.

No `H5b` `Deps.` change is required: `V17c` is already reachable through the `V17a`–`V17c` range entry in `H5b`'s `Deps.`, which `H5d` expands by letter suffix, so the transitive-completeness arm resolves the new row green once it exists.

Implementer/auditor note: §Granularity is phrased without a literal `MUST`/`MUST-NOT` token, so `H5e`'s un-anchored-MUST text scan will not independently enforce this row. The new `cka` row, validated by `H5d`, is the sole closing-gate traceability hook for §Granularity — do not rely on `H5e` to catch a future regression here.

## Consequence

**Severity:** correctness

A normative spec obligation (§Granularity) has no entry in the coverage matrix, the authoritative traceability artefact. The matrix-driven closing gates cannot verify §Granularity is closed — `H5d` has no row to check and `H5e` does not flag the obligation because it carries no literal `MUST` token — so the obligation's coverage is silently untracked, and a future regression dropping `V17c`'s granularity tests would not be caught by the matrix gate. Two auditors reconciling the matrix against the spec would diverge on whether §Granularity is covered.
