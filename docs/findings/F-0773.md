## Finding

`runtime-event-channel.md`'s `RuntimeEvent` payload shape declares `loom: string` with a type-comment that says only "slash name of the loom that owned the failure (`/code-review`)" ŌĆö it does not pin where the field is read from. The sibling `invocation_id` line, by contrast, was tightened in a later edit to pin its sourcing in the type comment itself ("per-invocation UUID sourced from the ActiveInvocationRegistry entry's `invocationId` field"), and that pin applies to *every* group-A `RuntimeEvent`. So the gap is asymmetric: `invocation_id` is pinned for all group-A events via the type comment, but `loom` is not.

The only place `loom` is pinned to the `ActiveInvocationRegistry` entry on the runtime-event channel is the **Binder-failure sourcing** paragraph immediately below the type definition. That paragraph is scoped to binder failures and motivated as a binder-specific workaround ("Binder-failure events ŌĆ” carry no `query_site` ŌĆö the binder runs before any loom code ŌĆö so for these events `invocation_id` and `loom` are read from the invocation's [`ActiveInvocationRegistry`](./active-invocation-registry.md#active-invocation-registry) entry"). Read against the silence on the other five group-A kinds (`transport`, `model_tool`, `code_tool`, `tool_loop_exhausted`, `invoke_infra`), the carve-out framing creates a negative implicature that those kinds source `loom` from somewhere else ŌĆö plausibly from a `query_site.file`-to-loom mapping or from the call-site closure, neither of which is pinned and both of which can diverge from the registry's canonical-key form (post-`pi.looms` mapping, post-`as` rename, no leading `/`) that `active-invocation-registry.md` pins as the single source of truth.

The downstream-consumer pin in `active-invocation-registry.md` enumerates `RuntimeEvent.invocation_id` and the `cancelled-by-session-shutdown` row's `details.event.loom` as the registry-sourced consumers, but does not name `RuntimeEvent.loom`. So even tracing through the registry contract, an implementer reading runtime-event-channel.md has no general statement that every group-A `RuntimeEvent.loom` is registry-sourced.

## Solution Space

**Shape:** single

### Recommendation

Add a single general-sourcing sentence to `runtime-event-channel.md` that applies to *every* group-A `RuntimeEvent`, and demote the existing **Binder-failure sourcing** paragraph to a clarifying note that no longer needs to carry the sourcing pin on its own.

Concretely:

- Replace the `loom: string` type-comment with one that matches the existing `invocation_id` pattern: e.g. `// slash name of the loom that owned the failure, sourced from the ActiveInvocationRegistry entry's loom field (canonical-key form per active-invocation-registry.md: post-pi.looms mapping, post-as rename, no leading "/"; e.g. "code-review")`. Fix the example to drop the leading `/`, which contradicts the registry's "no leading `/`" rule.
- Immediately under the `RuntimeEvent` block, add one sentence covering both fields for all group-A events: e.g. *"Every group-A `RuntimeEvent` reads `invocation_id` from the active `ActiveInvocationRegistry` entry's `invocationId` field and `loom` from that entry's `loom` field, regardless of whether the failure was reached from binder, from typed-query / tool-call sites, or from `invoke` infrastructure. The entry is present at every group-A construction site because the dispatch-site insertion (per [Active invocation registry ŌĆö Registry contract, *Insertion*](./active-invocation-registry.md#active-invocation-registry)) precedes any awaitable work, including the binder and any subsequent loom code."*
- Rewrite the **Binder-failure sourcing** paragraph to be the binder-specific *consequence* of the general rule (it still earns its keep by explaining why `query_site` is absent on binder rows), but stop carrying the sourcing pin on its own ŌĆö that now lives in the general statement above.
- In `active-invocation-registry.md`'s downstream-consumer enumeration on the *Registry contract* paragraph, add `RuntimeEvent.loom` alongside the already-named `RuntimeEvent.invocation_id` so the two pages agree.

Implementer edge case: on the still-rare path where a group-A `RuntimeEvent` is constructed inside the dispatch-site setup wrap's `catch` arm (before any registry entry has been inserted, per the `Set.add`-throws sub-case of the *Dispatch-site setup wrap* rule), the runtime-defect surface routes through `loom/runtime/internal-error` per that wrap's existing rule and does *not* emit a group-A `RuntimeEvent`; the new general statement is therefore satisfied on every reachable group-A construction site.

## Consequence

**Severity:** correctness

Two implementers reading the runtime-event-channel page can reasonably diverge on the `loom` field for the five non-binder group-A kinds: one reads from the `ActiveInvocationRegistry` entry (matching the binder carve-out and the registry's canonical-key form), the other constructs it from the call site (`query_site.file`-derived, or from the dispatching closure's authored name). The two forms split operator dedup buckets on `details.event.loom` ŌĆö exactly the failure mode the registry's canonical-name resolver was introduced to prevent ŌĆö and break the sibling-invocation distinction on operator-visible surfaces when the same loom runs under aliases.
