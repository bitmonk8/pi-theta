## Finding

Within `docs/spec_topics/pi-integration-contract/version-bump-step2.md`, two distinct enumerations on the same page share the bareword labels `(a)` and `(b)`:

1. The 41-item **editorial-review checklist for unpinned host presuppositions** uses `(a)`, `(b)`, `(c)` … `(ao)` for individual presupposition audits (e.g. `(a)` = instance survival, `(b)` = multi-`session_shutdown` delivery).
2. The **step-2 SDK surface-inventory tests** sub-step uses `(a)` and `(b)` for the *positive direction* (literal-read assertion) and *negative direction* (inventory-closure audit), pinned at the anchors `bump-step-2-positive` and `bump-step-2-negative`.

Hyperlinked cross-references resolve unambiguously through the anchors (`#bump-checklist-instance-survival` vs `#bump-step-2-positive`), but free-running prose on the same page repeatedly says "item (a)", "item (b)", "Step 2(a)", "Step 2(b)" — and the page already cites "step 2(a)" 30+ times from inside checklist items, including from checklist `(a)` itself. A reader following item-(b)'s body sentence "see Step 2(b) below" momentarily cannot tell whether `(b)` refers to the checklist's multi-`session_shutdown` row or the negative-direction inventory-closure audit, and the disambiguator is buried in the surrounding sentence rather than carried by the token.

The collision lives in a single file (other downstream consumers — `version-bump-step2b.md`, the V18c plan leaves, and a handful of finding pages — likewise use the bare `step 2(a)/2(b)` forms in running text) and is the kind of label-overload a one-shot rename can clear.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `e5cd514` (2026-05-09) — "pi-loom spec: resolve T35 SDK capability inventory closed-set negative-direction audit" — split the SDK surface-inventory step into two co-numbered arms and pinned them as `(a)` *Positive direction* / `(b)` *Negative direction* under anchors `bump-step-2-positive` / `bump-step-2-negative`.
- `52b9101` (2026-05-11) — "pi-loom spec: resolve T24 — Fork-reason watcher closure leaves the extension in an unspecified, silently degraded state" — added the four-item *Editorial-review checklist for unpinned host presuppositions* using `(a)`/`(b)`/`(c)`/`(d)` for the presupposition rows, inside the same numbered step in `docs/spec_topics/pi-integration-contract.md`.
- `f5e89f4` (2026-06-04) — "docs: enforce 100KB hard / 30KB recommended size cap on spec set" — split `pi-integration-contract.md` into the index-plus-children layout, materialising both enumerations inside the single child file `version-bump-step2.md` and putting the collision into one file's running text where the two sets touch.

**History:** Neither commit individually created a collision: e5cd514 introduced the step-2 `(a)`/`(b)` pair when no checklist existed, and 52b9101 introduced the checklist's `(a)`/`(b)` pair against a single-letter target — both arms were valid enumerations on their own. The defect is the interaction of the two label spaces in the same numbered step plus the f5e89f4 split that pulled the two enumerations into one rendered file. Subsequent commits (the checklist grew from `(d)` to `(g)` to `(am)` to `(ao)` between 2026-05-12 and 2026-06-19) widened the checklist arm but did not introduce the collision; the durable fix addresses the label-space overlap, not any single later edit.

## Solution Space

**Shape:** single

### Recommendation

Relabel the **step-2 SDK surface-inventory sub-step pair** from `(a)` / `(b)` to `(P)` / `(N)` (Positive / Negative), updating the two inline anchor IDs to `bump-step-2-P` and `bump-step-2-N`, and sweep every cross-reference to "step 2(a)" / "step 2(b)" across the corpus to the new tokens. Renaming the step-2 pair (rather than the checklist) is the smaller edit by surface count: the checklist has 41 individually-anchored rows and is heavily back-referenced from neighbouring spec pages, plan leaves, and downstream findings, while the step-2 pair has exactly two anchors and is referenced primarily from within the same file plus a closed set of adjacent pages.

Edge cases the implementer must watch:

- The new tokens must be distinct from every other parenthesised-letter convention already in the file. The step-5 "groupings (i)/(ii)/(iii)/(iv)/(v)" use lowercase Roman numerals; the item-(e) sub-outcomes use `(e.i)`/`(e.ii)`; neither overlaps `(P)`/`(N)`. The audit failure-family enumeration also uses bare `(1)`–`(5)` for both families and resolution branches — that is a separate, unresolved collision tracked by the sibling finding *Step-2(b) branch numbers (1)–(5) collide with the externally-owned failure-family numbers (1)–(5)*; do not collapse the two findings.
- Update the two inline `<a id="bump-step-2-positive">` and `<a id="bump-step-2-negative">` anchors to the new IDs in the same commit, and search the corpus for every `bump-step-2-positive` / `bump-step-2-negative` literal (link targets) plus every textual "step 2(a)" / "step 2(b)" / "Step 2(a)" / "Step 2(b)" / "step-2(a)" / "step-2(b)" / "2(a)" / "2(b)" form. The latter forms appear in `V18c-version-bump-checklist.md`'s **Adds** and **Ships when** prose and in `V18d-version-bump-acceptance.md`'s revert-gate description.
- Leave the legacy anchors in place only if a redirect convention exists in the corpus; otherwise the rename is a hard cutover and the inbound-reference sweep covers correctness.
- The checklist preamble's "items (a)–(e) below" / "items (f) through (ao) below" / "items (a)–(ao)" prose is unaffected by the step-2 rename and stays as-is.

## Consequence

**Severity:** cosmetic

A reader of `version-bump-step2.md`'s running text must rely on surrounding context ("Step", "item", "checklist", "sub-step") to disambiguate `(a)` and `(b)` between two enumerations whose subjects (presupposition audits vs. surface-inventory assertion arms) are unrelated. No implementer is misrouted because the anchors are disjoint, but every bump-procedure reading momentarily stalls at the overloaded token. The defect is editorial; it does not change any obligation.
