## Finding

`queryerror-variants.md` contains two same-file deictic references to the **Pattern grammar (loom 1.0)** table, but that table does not live on this page — it lives in the sibling page `error-model.md`:

- In the **Discriminator type-openness** paragraph: "rely on the catch-all `_ => …` arm pattern (see the **Pattern grammar (loom 1.0)** table at the top of this file)".
- In the **Notes** section: "Authors `match` on `QueryError { kind: "...", ... }` (pattern grammar above) and only the relevant variant's fields are in scope."

Both phrasings — "at the top of this file" and "pattern grammar above" — assert the table is co-located on `queryerror-variants.md`. It is not: the `**Pattern grammar (loom 1.0).**` table is the second block of `errors-and-results/error-model.md`. A reader following either pointer scans `queryerror-variants.md` (which opens with the `QueryError` union declaration, not a pattern grammar) and finds nothing matching the description.

The defect is a stale intra-file deixis left over from a page split: the two reference sites and the table were once in one file, where "at the top of this file" / "above" were accurate. After the split they reference a table that moved to a different file, and the deixis was never converted to a cross-page link.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `f5e89f4` — "docs: enforce 100KB hard / 30KB recommended size cap on spec set" (2026-06-04)
**History:** Before `f5e89f4`, the Pattern grammar table and both deictic references lived in one monolithic file (`docs/spec_topics/errors-and-results.md` at its parent `f5e89f4~1`: table at the file's top, references in the body), so "at the top of this file" and "pattern grammar above" were internally accurate. The size-cap split moved the Pattern grammar table into the new sibling `error-model.md` and the two reference sites into the new `queryerror-variants.md`. The commit's map-driven rewrite repointed markdown cross-reference *links* across the corpus but did not touch prose deixis ("at the top of this file", "above"), which carries no link target to rewrite — so the split silently broke both references. `git log -S` for each token ("at the top of this file", "pattern grammar above") on `queryerror-variants.md` returns only `f5e89f4` as the introducing commit on this path.

## Solution Space

**Shape:** single

### Recommendation

Replace both same-file deictic phrasings with an explicit cross-link to the table's real home on `error-model.md`, following the corpus convention `[Errors and Results — …](./error-model.md#…)` already used elsewhere in `queryerror-variants.md`'s sibling references.

- The `**Pattern grammar (loom 1.0).**` paragraph in `error-model.md` is a bold lead-in, not a heading, so it has no auto-generated anchor. Add an explicit anchor (e.g. `<a id="pattern-grammar"></a>` immediately before the bold lead-in), matching the `<a id>` convention used on the other bold rules in that file.
- In the Discriminator type-openness paragraph, change "(see the **Pattern grammar (loom 1.0)** table at the top of this file)" to "(see the [Pattern grammar (loom 1.0)](./error-model.md#pattern-grammar) table)".
- In the Notes section, change "(pattern grammar above)" to "(see the [Pattern grammar (loom 1.0)](./error-model.md#pattern-grammar) table)".

## Consequence

**Severity:** advisory

A reader following either in-text pointer to confirm the catch-all `_ => …` arm form or the `match` arm grammar lands on the wrong page and finds no table; the authoritative grammar is one page away with no working link. No behaviour changes — the grammar itself is reachable — but the navigational guidance is wrong.
