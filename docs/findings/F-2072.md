## Finding

`audit-recognised-shapes.md` (*Recognised import/access shapes*, the CJS-prohibition clause) mandates that the inventory-closure audit detect "**any call to a function whose source binding transitively resolves to the `createRequire` named export of the Node `"module"` package** when invoked with one of the in-scope package specifiers." It enumerates the shapes this must reach through: aliased imports (`createRequire as cr`), namespace imports (`M.createRequire`), default imports, default-namespace re-exports of `"module"`, captured locals (`const cr = createRequire`), "further `as`-rename re-export, intermediate barrel module," and "any other re-binding shape … transitively."

The same cluster simultaneously positions the audit as satisfiable by a pure static-AST walker. `inventory-audit-intro.md` (*SDK capability inventory* / *Inventory-closure audit*) states that describing only the behavioural contract "lets the implementation choose a static-AST walker, a TypeScript-program symbol-table walk, or any equivalent tooling," and the closing sentence of the recognised-shapes bullet asserts "the audit's behavioural contract is invariant under tooling choice; what changes is only the implementation."

These two positions conflict for the `createRequire` clause specifically. Tracing whether an arbitrary call expression's callee "transitively resolves" to `createRequire` through re-export chains, intermediate barrel modules, and cross-module captured bindings is a whole-program def-use / symbol-resolution problem. A pure static-AST walker that processes one file at a time without a program load or symbol table cannot follow a binding across module boundaries back to the `"module"`-package export. The obligation as written is therefore only satisfiable by the type-aware variant the spec frames as optional — contradicting the tooling-invariance claim and the static-AST floor.

Note the contrast with the adjacent `pi`/`ctx` captured-rebinding clause, which the spec deliberately made statically decidable: it observes that "the static-AST audit can resolve every such access … without chasing data-flow through an intermediate binding" *because each rebinding shape is itself independently prohibited and locally detectable*. The `createRequire` clause does not get that treatment — it reaches for transitive cross-module resolution rather than flagging the locally-visible binding shapes, which is what breaks the floor.

## Solution Space

**Shape:** single

### Recommendation

Bound the `createRequire` detection obligation to a statically-decidable, within-module scope. In `audit-recognised-shapes.md`, replace the transitive cross-module formulation — "any call to a function whose source binding transitively resolves to the `createRequire` named export … through … intermediate barrel module … transitively" — with a within-module formulation, e.g. "any call expression within a module whose callee binding resolves, by intra-module def-use, to that module's import of `createRequire` from `"module"`". Add a sentence noting that cross-module transport of a `createRequire` binding is reached via the separately-prohibited peer-package re-export and barrel-module shapes, not by transitive resolution.

This keeps the static-AST floor and the "invariant under tooling choice" claim in `inventory-audit-intro.md` honest, and loses no real coverage: the cross-module laundering primitives (`export *` and `export * as` of peer packages, aliased re-export specifiers, barrel modules) are already prohibited surface shapes that close every laundering path. When narrowing the clause, verify those primitives genuinely close every cross-module laundering path so the within-module bound leaves no gap.

The within-module bound must still catch locally-captured and namespace-member rebindings in the same module — `const cr = createRequire; cr("@earendil-works/<peer>")` and `M.createRequire(...)("@earendil-works/<peer>")` — and the `.resolve(<specifier>)` path-resolution carve-out (reserved for the Step 0(d) peer-dep read) must remain out of family (4).

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads the tooling-invariance claim and builds a static-AST walker that silently misses `createRequire` laundered through a cross-module re-export or barrel chain; another builds a type-aware symbol-table walker that catches it. The first produces an audit with a hole in the very surface-set-closure guarantee the audit exists to enforce — a CJS reach into a peer package can ship undetected.
