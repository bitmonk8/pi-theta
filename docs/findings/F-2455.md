## Finding

The extension factory runs a fixed five-step bootstrap sequence (`registerFlag` + `resources_discover` subscription, parse + pending list, `session_start` subscription, `session_shutdown` subscription, watcher/`LoomRegistry` construction), with the `loom-system-note` renderer registered before the first scan. The `loom/load/extension-bootstrap-failed` contract (owned by [Extension-bootstrap SDK failures](../) in `extension-bootstrap-and-per-loom.md`) classifies a `pi.registerFlag` failure (step 1) and any factory-time `pi.on(...)` subscription failure (steps 1/3/4) as **fatal to the whole extension**, and pins the forward behaviour: the factory "skips every subsequent `pi.register*` and `pi.on` call (steps 2–5 do not execute)" and emits one diagnostic. What it never states is the disposition of registrations that *already succeeded* before the fatal step — the renderer (and, for a `pi.on` failure at step 3 or 4, the `pi.registerFlag` call and the earlier `resources_discover`/`session_start` subscriptions). The contract specifies what does **not** run forward but is silent on what to do with what has already run.

Two facets are left open. First, the already-installed Pi-side registrations (renderer, flag, earlier subscriptions) have no stated post-fatal disposition: are they unwound, or left in Pi's loader for the extension-instance lifetime? Pi exposes no `unregisterMessageRenderer` / `pi.unregisterCommand` / unsubscribe surface, so loom-side rollback is impossible by construction — but the contract never says this, even though the analogous hot-reload registry-swap path *does* spell out the parallel posture ("any `pi.registerTool` call that already succeeded before the throw leaves an orphaned entry … the cosmetic accumulation that section already accepts"). Second, and more consequential: when the fatal step is the `session_shutdown` subscription at step 4, the `session_start` handler subscribed at step 3 is already live and will fire at session start, where it runs the cross-format collision pass and calls `pi.registerCommand` for each survivor — but step 5's `LoomRegistry` (which the dispatch closure and that handler depend on) was never built because the factory aborted. The spec gives no rule requiring the half-bootstrapped extension to self-disable (for example by setting the `LoomRegistry` drain-state tag so the live `session_start` handler and any dispatch route no-op), so an implementer is left to invent whether a fatally-failed extension still registers and dispatches per-loom commands.

The drain-state contract and the `/reload` recovery mechanism exist precisely to express "extension degraded," yet the fatal bootstrap path never references them, so the post-fatal steady state of the extension is undefined.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `2d2fc8e` (2026-05-06, "pi-loom spec: resolve \"Pi SDK capability calls have no failure contract\"")
**History:** The corpus is git-tracked. `git log -S` over the fatal-rule tokens (`"do not execute, mirroring the step-0 failure semantics"`, `"Extension-bootstrap SDK failures"`) traces the extension-bootstrap SDK-failure granularity contract to its first authoring in `2d2fc8e`, which introduced the `pi.registerFlag` fatal bullet ("steps 2–5 do not execute") without ever specifying the disposition of registrations that had already succeeded. The omission was carried forward unchanged when the `pi.on(...)` granularity bullet was added (`79ea4e8`, 2026-06-06) and when the section was split into its own file under the size-cap commit (`f5e89f4`, 2026-06-04); subsequent relocations (`6a42c1a`, `4b4382f`) did not touch it. The tokens `rollback` and `unwind` have never appeared in the file's history. The defect is an omission present since the contract's inception, not a regression introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

In `extension-bootstrap-and-per-loom.md`, extend the fatal `pi.registerFlag` and `pi.on(...)` bullets (or add a shared trailing clause to the **Extension-bootstrap SDK failures** section) that pins the post-fatal state on two axes:

1. **Already-succeeded Pi-side registrations are left in place; there is no loom-side rollback.** State that the renderer, the `--loom` flag, and any earlier `pi.on` subscriptions that succeeded before the fatal step remain installed for the extension-instance lifetime — Pi exposes no `unregisterMessageRenderer` / `pi.unregisterCommand` / unsubscribe surface, so unwinding is impossible by construction. Pi reclaims them when it tears down the extension instance (`/reload`, `/new`, quit). Frame this with the same "orphaned registration / accepted cosmetic accumulation" posture the hot-reload registry-swap path already uses, so the two failure surfaces read consistently.

2. **The half-bootstrapped extension is non-dispatchable.** Specify that on a fatal failure the extension declares itself degraded so that a `session_start` handler subscribed before the fatal step (step 3) does not register per-loom commands or dispatch into a never-built `LoomRegistry` (step 5). The concrete mechanism should reuse the existing `LoomRegistry` drain-state contract — e.g. the fatal path writes the drain-state tag (or the handler treats an absent/unbuilt registry as drained) so the step-3 handler and every dispatch route through the degraded arm and return the existing `"extension degraded; /reload to recover"` system note rather than registering survivors. `/reload` is the recovery path, consistent with the other bootstrap-failure cases.

Edge cases the implementer must watch: the drain-state-tag write must not collide with the `session_shutdown` handler-entry short-circuit predicate (which assumes the tag is written only by the teardown path — see PIC-55's equivalent caution); and a `registerFlag` failure at step 1 (before any subscription is installed) needs no handler-disable step, only the "orphaned renderer left in place" clause.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the steady state after a fatal mid-sequence bootstrap failure. The acute case is the step-4 (`session_shutdown` subscription) failure: the already-subscribed step-3 `session_start` handler still fires and reaches `pi.registerCommand` against a `LoomRegistry` (step 5) that was never built — one implementer guards the handler (drain-state / no-op), another lets it register per-loom commands into an extension declared non-functional, producing live but unbacked slash commands and divergent dispatch behaviour. The orphaned Pi-side registrations are otherwise harmless (no unregister API exists), but the contract never says so.
