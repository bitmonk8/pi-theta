## Finding

SLSH-3 in `slash-invocation.md` describes the slash-dispatch-boundary system note as follows: "The note never dumps the full `QueryError` JSON — it summarises the failure category and the most-relevant detail." This is discretionary, summary-level prose. The phrase "the most-relevant detail" gives no objective criterion for what to include, and "summarises the failure category" reads as an informal characterisation rather than a pointer to a concrete format.

The actual normative contract for the note's content lives in SLSH-4, immediately below: a per-`kind` table of verbatim templates (`SNK-a` … `SNK-k`) that renderers MUST emit exactly, with only `<…>` placeholders interpolated. SLSH-3 never forward-references SLSH-4, so a reader who treats SLSH-3 as the requirement — or stops reading at the end of the SLSH-3 paragraph — would synthesise their own free-form summary instead of emitting the mandated template strings. The two clauses called out in the heading ("most-relevant detail" weasel and the implicit forward-dependency on SLSH-4's category formatting) are the same defect viewed from two angles: SLSH-3's prose stands in for, but does not name, the normative formatting that SLSH-4 owns.

The fix is a single edit: replace SLSH-3's discretionary characterisation with an explicit reference to the SLSH-4 per-kind templates as the authoritative formatting rule, keeping only the non-normative "never dumps full JSON" intent as colour.

## Solution Space

**Shape:** single

### Recommendation

In SLSH-3, replace the clause "it summarises the failure category and the most-relevant detail" with an explicit reference to the normative SLSH-4 per-`kind` templates as the formatting authority. Concretely, state that the note is formatted per the SLSH-4 per-`kind` templates (the `SNK-a` … `SNK-k` rows), retaining the "never dumps the full `QueryError` JSON" sentence as non-normative intent. This removes the discretionary "most-relevant detail" wording and makes SLSH-3's dependency on SLSH-4 explicit rather than implicit.

Edge case for the implementer: keep SLSH-3 as the trigger/scope rule (when the note fires and which boundary it fires at) and let SLSH-4 own the rendered string; do not duplicate the template text into SLSH-3, which would create a second source of truth subject to the SLSH-4 "wording changes are spec-versioned breaking changes" rule.

## Consequence

**Severity:** advisory

An implementer who reads SLSH-3 as the content requirement, without continuing to the SLSH-4 templates, would emit a hand-rolled failure summary that fails the SLSH-4 verbatim-template conformance tests. Implementers who read the whole page converge on SLSH-4, so a working, conformant renderer is still producible; the gap is in guidance discoverability, not in the underlying contract.
