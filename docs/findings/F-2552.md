## Finding

grammar.md's *Generic-application constructors* paragraph declares `GenericType` a closed set — `array` (arity 1) and `Result` (arity 2) — and states flatly that "No other identifier is parameterisable." It then names a diagnostic for one of the two ways that closure can be violated: a wrong type-argument count on a valid head (`array<T, U>`, `Result<T>`) is `loom/parse/generic-arity-mismatch`. The other violation — applying `<…>` to a head that is *not* a constructor at all, e.g. `Foo<T>` where `Foo` is a schema/enum `NamedType`, or `string<T>` on a `PrimitiveType` — is named nowhere. The grammar simply has no production for it (`Foo` matches `NamedType`, after which `<` is unexpected), and the `loom/parse/*` registry contains no row for the case.

This breaks the diagnostic symmetry the spec otherwise maintains for every forbidden type-grammar form. `void` in a non-return position has `loom/parse/void-in-non-return-position`; `Result` in a schema-feeding position has `loom/parse/result-in-schema-position`; wrong arity has `loom/parse/generic-arity-mismatch`. Only the non-parameterisable-head case is left to a generic "unexpected token" with no specified code. A conformance-test author cannot write an assertion against `Foo<T>` because there is no stable code to assert, and the closed-registry gate (`DIAG-2`/`DIAG-3`/`DIAG-4`, which make the registry the closed, stable, normative source of every asserted code) has no row to anchor that test to.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `1c552ff` (pi-loom spec: resolve "Type grammar admits no generic application beyond array<T>", 2026-06-04)
**History:** The entire generic-application surface — the `GenericType` nonterminal, the closed `{array, Result}` set, the arity-2 `Result` constructor, the *Generic-application constructors* paragraph, and the `loom/parse/generic-arity-mismatch` diagnostic — was introduced in `1c552ff`. Before that commit the Type grammar inlined only `"array" "<" Type ">"` as a bare alternative with no `GenericType` nonterminal, so applying `<…>` to any other head had no production and the concept of a "non-parameterisable identifier" did not exist as a surface. The same commit that created the surface named a diagnostic for the wrong-arity violation but never named one for the non-parameterisable-head violation; the gap entered the corpus at that moment and has been present unchanged since. `git log -S` confirms no `generic-not-parameterisable` / `not-parameterisable` code has ever existed in the registry.

## Solution Space

**Shape:** single

### Recommendation

Register `loom/parse/generic-not-parameterisable` (severity `E`, phase `parse`) in `docs/spec_topics/diagnostics/code-registry-parse.md`, and name it in grammar.md's *Generic-application constructors* paragraph as the diagnostic for applying `<…>` to a head outside the closed `{array, Result}` set.

- Trigger: a generic application `H<…>` whose head `H` is not a member of the closed `GenericType` constructor set — i.e. any `NamedType` (PascalCase schema/enum name) or `PrimitiveType` followed by `<`. This is distinct from `loom/parse/generic-arity-mismatch`, which fires only when the head *is* a valid constructor (`array`/`Result`) but the type-argument count is wrong.
- Suggested message form, parallel to the existing arity message (`generic type '<ctor>' expects <expected> type argument(s); got <actual>`): `'<name>' is not a parameterisable type; only array<T> and Result<T, E> take type arguments in loom 1.0`.
- State the precedence ordering against `loom/parse/generic-arity-mismatch` explicitly: arity-mismatch presupposes a valid constructor head, so the head-membership check fires first and the two are mutually exclusive.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the observable diagnostic emitted for `Foo<T>`: one may fall back to a generic parse error, another to `loom/parse/unsupported-feature`, a third may mis-route it to `loom/parse/generic-arity-mismatch`. Because loom treats diagnostic codes as a closed, stable, normatively-tested surface, that divergence is an interop defect, and no conformance test can pin the case.
