## Problem

The DISC-6 paragraph (`#disc-6`) says the `loom/load/discovery-slow` warning names the root being scanned and the cap that fired, but neither that prose nor the `loom/load/discovery-slow` row in `code-registry-load.md` pins a structured `details` payload — only the rendered `Message` template is pinned. The sibling `loom/load/package-read-timeout` row pins `details.kind = "package-read-timeout"`; `discovery-slow` names no `details` field. Per `diagnostic-shape.md`, each row's *Trigger* prose is the source of truth for its `details` shape, so `discovery-slow`'s is empty. Operator tooling can only discriminate the two cap arms by parsing the rendered message text.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `475155c` (2026-05-05, "V14m discovery walk omits scoped packages and has no upper bound") and `25e8a60` ("T08 Per-package.json read timeout: overrun is unbounded") together established the DISC-6 cap+root prose and the `loom/load/discovery-slow` registry row; the row pinned the rendered Message template but never the `details` shape.
**History:** The fixing commits each closed a different prior gap but none paired the new prose with a structured `details` observable — a "fix-without-observable" pattern shared across several DISC findings.

## Solution approach

Pin a structured `details` payload for `loom/load/discovery-slow` so the cap arm is structurally discriminable rather than observable only by parsing the rendered message. Extend the DISC-6 paragraph (`#disc-6`) and mirror onto the `loom/load/discovery-slow` row's *Trigger* prose in `code-registry-load.md`, following the sibling `loom/load/package-read-timeout` row's `details.kind` discriminator convention. Lift the cap identity and root already constrained by the existing `<cap>` rendering and the dual-cap tie-break into the `details` contract.
