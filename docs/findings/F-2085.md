## Finding

`System-prompt structure (normative)` item 4 requires each Parameters-block per-field line to render `<wire-name> (<type>) <requirement>`, with `<type>` "rendered per *Type display* below." *Type display* states the general rule — render the field's declared Loom type "written in the surface syntax of [Type System], not the JSON Schema lowering" — and then pins a reference table whose renderings are "normative; conforming implementations MUST reproduce these exactly." The table enumerates only eight forms: `string`, `integer`, `boolean`, an enum name, a named-schema name, `array<integer>`, `string | null`, and `Cat | Dog`.

The Loom type grammar admits considerably more than these. `type-system.md` and `grammar.md#type-grammar` permit inline anonymous object types (`ObjectType ::= "{" Field ("," Field)* ","? "}"`), literal types (`"validation"`, `42`, `true`), nested generics (`array<{ a: string }>`, `Result<T, E>`), and arbitrary unions over all of the above — every one of which is legal in a `params:` annotation position. For these compound forms the spec gives no canonical byte-exact rendering. "Surface syntax" does not pin the whitespace the grammar leaves free: the brace-interior spacing of an inline object (`{a: string}` vs `{ a: string }`), the post-comma spacing in a multi-field object, whether a trailing comma is emitted, and the spacing inside nested generic brackets are all unconstrained. Note that the sibling token-order rule added later ("renderers MUST NOT insert additional whitespace between [the three tokens] beyond the single U+0020") pins whitespace *between* `<wire-name>`, `(<type>)`, and `<requirement>`, but says nothing about whitespace *inside* the `<type>` rendering itself.

This collides with the binder's input-reproducibility contract. The system-prompt bytes are not revisable illustration: `binder-model-and-context.md#why-the-transcript-bytes-are-pinned` establishes that loom controls "byte-identical binder input," and the Parameters block is part of those bytes. Two conforming implementations handed the same loom whose `params:` declare a compound-typed field would emit different `<type>` bytes, producing different binder input, defeating the byte-identical-input guarantee (and any prompt-cache keyed on it) for those looms. The "MUST reproduce these exactly" clause therefore makes a promise the spec does not deliver for any param type outside the eight enumerated rows.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 5a4ec51 — pi-loom spec: resolve "Binder system prompt information content lacks verifiable field set" (2026-05-06, Thomas Andersen)
**History:** Commit 5a4ec51 introduced both item 4's "rendered per *Type display* below" rule and the *Type display* table with its "conforming implementations MUST reproduce these exactly" clause into the predecessor `spec_topics/binder.md`. The table enumerated only the same closed reference set then in place (`string`, `int`, `boolean`, enum, named schema, `array<int>`, `string | null`, `Cat | Dog`) with no general canonical-whitespace rule for compound types, so the gap was present at the rule's introduction. The later token-order/indentation commit (4188412, 2026-05-07) pinned whitespace *between* the three tokens but not *within* `<type>`, and the size-cap split (f5e89f4, 2026-06-04) moved the rule verbatim into `binder/binder-bypass-and-envelope.md`; neither closed the gap.

## Solution Space

**Shape:** single

### Recommendation

In `binder/binder-bypass-and-envelope.md`, extend *Type display* from a closed reference table into a normative recursive canonical-rendering rule covering every type form the Loom grammar admits, so no compound param type is left with unpinned bytes. Replace the table's "Reference renderings" framing with a recursive rendering definition that fixes the whitespace the grammar leaves free, and keep the existing eight rows as illustrative examples of that rule.

State the whitespace rules against the grammar's free positions, not by example:

- Inline object types render as `{ field: T, ... }` — one U+0020 after `{`, one before `}`, and one after each field-separating comma; no trailing comma.
- Nested generics render with no spaces inside the angle brackets, e.g. `array<{ a: string }>`.
- Literal types render their source literal verbatim, e.g. `"validation"`, `42`, `true`.
- Unions use the ` | ` separator (one U+0020 either side of `|`).
- The rule recurses into element, field, and union-arm types, so it applies through unions, `array<T>`, `Result<T, E>`, and nested inline objects.

Add at least one inline-object and one nested-generic worked reference rendering to the table. Keep the rendering shape consistent with the sibling `default=<literal>` rendering rule (F-2089) so both rendering tables are pinned the same way.

This preserves the binder's byte-identical-input contract for every loom regardless of param type, consistent with how the rest of the binder prompt (separators, indentation, em-dash) is pinned byte-exact, and requires no carve-out against *Why the transcript bytes are pinned*.

## Consequence

**Severity:** correctness

A loom whose `params:` use an inline-object, literal, nested-generic, or `Result<T, E>` field type reaches a `<type>` rendering the spec does not pin byte-for-byte. Two reasonable implementers diverge on the interior whitespace, emitting different binder system-prompt bytes for the same loom — a direct violation of the input-reproducibility contract the binder prompt is built to honour, and a possible silent prompt-cache miss.
