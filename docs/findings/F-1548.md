## Finding

The Per-loom registration bullet in `extension-bootstrap-and-per-loom.md` enumerates the five required `ToolDefinition` fields (`name`, `label`, `description`, `parameters`, `execute`) as a normative field-derivation list. Wedged between the closing normative lead-in sentence ("All five fields … are required by Pi's `ToolDefinition` interface.") and the field-derivations sub-list is a self-labelled `*Non-normative implementation note.*` recommending construction via `defineTool(...)` from `@earendil-works/pi-coding-agent` and explicitly stating that "the construction-helper choice is a maintainability convention, not an observable conformance point."

The note's content is fine in isolation — it preserves TypeBox's generic inference on `parameters` and is useful guidance — but its placement breaks the reader's mode: the field-derivations bullets that immediately follow are normative MUSTs (e.g. `label`'s basename-with-leading-capital rule, the synthesised typed-query one-shot tool's literal `"Loom typed-query response"` label), and the inline non-normative aside makes a skim of the enumeration ambiguous about which sentences carry obligation. Implementation-hint sentences belong on `implementation-notes.md` (whose Runtime section is already cross-linked from this same page); that page currently narrates the `Type.Unsafe<unknown>(loweredJsonSchema)` bridging convention without mentioning the `defineTool(...)` helper, so the relocation has a natural home that today's text fails to populate.

## Issue introduction

**Verdict:** single-commit-regression
**Introducing commit:** `8362824` (2026-06-04, "pi-loom spec: resolve 'Demote over-prescriptive and non-testable MUSTs on unobservable implementation structure'")
**History:** The `defineTool(...)` sentence first appeared in `fae85d3` (2026-05-04, "Tighten spec for implementation-plan readiness") as a normative inline claim adjacent to the field-derivations list. Commit `8362824` demoted the same sentence to "*Non-normative implementation note.*" — the correct demotion in substance — but left it textually inline between the lead-in and the enumeration, introducing the inline-non-normative-aside-inside-normative-enumeration shape this finding identifies. The previous shape (normative-inline) had no such placement defect; the regression is a side effect of the demotion, not of the original authoring.

## Solution Space

**Shape:** single

### Recommendation

Move the `defineTool(...)` sentence out of the Per-loom registration bullet and into `implementation-notes.md`'s Runtime section, immediately adjacent to the existing `Type.Unsafe<unknown>(loweredJsonSchema)` bridging-convention narration in the typed-query bullet (or as a peer bullet under Runtime if cleaner). Phrase it there as: "the reference implementation constructs each materialised `ToolDefinition` through `defineTool(...)` (exported from `@earendil-works/pi-coding-agent`) rather than a bare object literal so TypeBox's generic inference on `parameters` is preserved when the value is spread into `customTools: ToolDefinition[]` (subagent path) or passed to `pi.registerTool` (prompt-mode path); the construction-helper choice is a maintainability convention, not an observable conformance point." In `extension-bootstrap-and-per-loom.md`, delete the inline note in full and let the normative lead-in flow directly into the "Field derivations:" sub-list. Do not leave a forward-link stub at the deletion site — the existing page-top cross-reference to `implementation-notes.md` already covers discoverability, and a stub would re-introduce reader-mode friction in the same enumeration this fix is clearing. Edge case for the implementer: preserve the substantive caveat about `Type.Unsafe<unknown>` (already on `implementation-notes.md`) — it is the load-bearing half of the helper's purpose and must not be silently dropped along with the placement fix.

## Consequence

**Severity:** cosmetic

A diligent reader still parses the note as non-normative because it is explicitly labelled so. The harm is reader friction at the start of an enumeration of MUSTs, plus a minor placement-discipline drift that, repeated, erodes the convention that non-normative implementation hints live on `implementation-notes.md`. No observable behaviour, no diagnostic, and no conformance gate is affected.
