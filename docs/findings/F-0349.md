## Problem

`V2b`'s **Adds** field claims the leaf adds "a runtime AJV safety-net for statically-unresolvable operands," and its **Ships when** field "leaves statically-unresolvable operands to the runtime AJV safety-net" — both phrasings present the runtime AJV check as something `V2b` owns or installs. Nothing in the leaf backs that up: all ten tests (`TYPE-1 … TYPE-10`) assert the static `⊑` relation on statically-resolvable operands, and `V2b`'s **Deps** (`V2b-T`, `V2a`, `V5d`) include neither the validator seam nor the schema-lowering pass an AJV call would require. The spec section [`type-system.md` §Type compatibility — "Unresolvable operands"](../spec_topics/type-system.md#type-compatibility) is unambiguous that the engine's job here is to do nothing and defer the safety-net to the consuming sites; the runtime AJV safety-net is owned downstream by `V15f`, `V13b`, and `V14a`, which carry that dependency closure. The wording therefore mis-attributes a downstream obligation to `V2b`.

**History:** The "runtime AJV safety-net for statically-unresolvable operands" wording entered `V2b` in its first commit (c6a664e), where Ships-when also read "defers unresolved operands to runtime AJV." 210ed9b reframed both fields to a "deferral marker … consumed by the downstream runtime-validator sites (`V6b`, `V11d`)" to escape a dependency-closure critique; a0f6c6b then reverted that reframing — because the deferral-marker claim falsely asserted `V6b`/`V11g` consume the seam — restoring the inception "runtime AJV safety-net" wording that this finding now flags. The defect thus originates at inception and survives to the current text after a brief reframe-and-revert.

## Solution approach

Reword `V2b`'s **Adds** and **Ships when** in `docs/plan_topics/V2b-type-compat-engine.md` so neither field attributes the runtime AJV safety-net to `V2b`. State that `V2b` skips the static `⊑` decision for operands past the parser's static view and decides nothing for them, per [`type-system.md` §Type compatibility — "Unresolvable operands"](../spec_topics/type-system.md#type-compatibility). Make clear in **Ships when** that no runtime AJV wiring lands in `V2b` and the runtime safety-net is owned downstream by `V15f`, `V13b`, and `V14a`.

## Solution constraints

- The runtime AJV safety-net is owned downstream by `V15f`, `V13b`, and `V14a`; do not wire an AJV call into `V2b` or add Deps onto the validator seam (`V8c`) or schema-lowering pass (`V5f`). `V2b`'s **Deps** stay as-is (`V2b-T`, `V2a`, `V5d`).
- The spec is read-only for this fix; `type-system.md` already specifies the skip-then-defer posture and needs no change.
