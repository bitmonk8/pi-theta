## Finding

FN-4 (**Empty-tail body**) enumerates the statement forms whose presence as a
body's last form makes the body empty-tail — inferring return type `null`. The
list reads: `let`, `for`, `while`, `if`-statement form, assignment, `break`,
`continue`, and **"expression-statement of `void` type"**. That last entry
names a category the type system explicitly denies: type-system.md states that
`void` "is **not** a primitive value type … never in a value-bearing type
position", that "`void` does not participate in `⊑`", and — categorically —
that "no value is ever statically typed `void`". If no expression is ever
statically `void`, then "expression-statement of `void` type" has no referent:
an implementer cannot decide which expression-statements qualify.

The intended referent is recoverable from surrounding prose: the functions.md
intro paragraph describes a `void`-returning function whose tail expression is
discarded "as in the expression-statement form", and query.md's discard rules
treat a call to a `void`-returning function as a discard site. The category FN-4
means is therefore *a call to a `void`-returning function used as a statement* —
the call expression's static type is a value type, but the callee's `void`
return annotation means its value is discarded, so the call sits in
statement position rather than as a value-bearing tail expression.

The classification is load-bearing, not merely cosmetic: it decides whether a
trailing `void`-returning call is treated as the body's tail expression (so the
body's value would be that call) or as a statement (so the empty-tail rule fires
and the body infers `null`). FN-4 intends the latter, but the undefined phrasing
leaves that turn on a term the type system says denotes nothing.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 52219c4 — Address all 41 spec-review findings (2026-05-06, Thomas Andersen); 4c76dfb — pi-loom spec: resolve "void admitted by grammar but absent from Type System enumeration" (2026-06-05, Thomas Andersen)
**History:** Commit 52219c4 added the **Empty-tail body** rule to functions.md, whose statement-form list already contained the phrase "expression-statement of `void` type". At that point type-system.md carried no categorical statement about `void` typing. Commit 4c76dfb later hardened type-system.md, adding both "`void` does not participate in `⊑`" and "no value is ever statically typed `void`". The contradiction is the product of the two: the FN-4 wording, benign when written, became inconsistent once 4c76dfb declared that no expression can statically be `void`.

## Solution Space

**Shape:** single

### Recommendation

In the FN-4 statement-form list, replace the entry "expression-statement of
`void` type" with "a call to a `void`-returning function used as a statement"
(equivalently: "an expression-statement whose expression is a call to a
`void`-returning function"). This names the actual construct — a call expression
in statement position whose callee carries the return-only `void` annotation —
without invoking a "`void`-typed expression", keeping FN-4 consistent with
type-system.md's "no value is ever statically typed `void`" and with the
return-only status of `void` in grammar.md's `ReturnType` production. The
behavioural rule is unchanged: such a trailing call is a statement, so the body
is empty-tail and infers `null`.

## Consequence

**Severity:** advisory

An implementer can recover the intended meaning from neighbouring prose, so a
working implementation is still reachable; but the term as written contradicts
the type system's categorical "no value is ever statically typed `void`", and a
reader who takes that contradiction at face value could mis-decide whether a
trailing `void`-returning call is a tail expression or a statement — the exact
distinction FN-4 exists to settle.
