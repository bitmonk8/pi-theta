## Finding

`session-only-degraded-state.md` opens with a clause-scoped implementation gate. It declares two clauses buildable now (the *Drain-state tag write (state-independent)* setter contract and the *Isolation* sub-clause) and gates **every other clause on the page** — the *Predicate split*, *Recovery path and prohibitions*, *Diagnostic emission*, *Idempotent re-entry and the all-three-throw corner case*, *Accepted residual gaps*, and the acceptance-criteria fixtures (a)–(m) — as "blocked pending resolution of the open contradiction" recorded at `host-prerequisites.md#degraded-state-host-prerequisites` clause (a), clearing only when version-bump item (a) resolves to the `provably-differs` verdict.

The sibling page `drain-state-contract.md` carries no gate banner and normatively pins the same degraded-branch behaviour as implement-now requirements: the `idempotent` clause (the canonical home of the degraded re-entry semantics that `session-only-degraded-state.md`'s gated *Idempotent re-entry* section merely links to), PIC-29's three-arm tuple-exhaustion including arm (c) `"degraded-needs-reload"`, PIC-30's prohibition on a fourth arm, PIC-31's read-failure fallback that short-circuits on arm (c), and the `readDrainState` arm-(c) routing that returns the `"degraded; /reload to recover"` note. It also adds its own acceptance-criteria section. None of this material is marked contingent.

The result is a direct cross-file contradiction about the same behaviour: the writer-side branch logic that produces the `"degraded-needs-reload"` tag is blocked on one page, while the reader-side routing, re-entry semantics, and corner-case handling that consume that tag are presented as fully specified and buildable on the other. The gate banner asserts that `drain-state-contract.md` only *consumes* the non-gated seam contracts, but that page in fact restates the gated branch behaviour as its own normative requirements. An implementer following the spec forward cannot tell whether the degraded transition and arm (c) are in-scope now or blocked, and the gate's intended purpose — to prevent building the degraded branch before the unresolved instance-survival presupposition is settled — is defeated for anyone working from the sibling page.

## Issue introduction
**Verdict:** single introducing commit (regression introduced by a one-sided gate)
**Introducing commit:** `b8f004f` — "pi-loom spec-plan: fix F-0719 — SM-4/5/6/3b degraded-state implementation-gate contingency markers" (2026-06-19)
**History:** The degraded-arm reader contract in `drain-state-contract.md` (the `"degraded-needs-reload"` arm (c) and the idempotent re-entry narrative) predates the gate — it was already present when the two pages were split under the size cap in `f5e89f4` (2026-06-04), and `git show f5e89f4:.../drain-state-contract.md` shows the `degraded-needs-reload` arm already established. The contradiction entered in `b8f004f`, which added the `> *Implementation gate.*` banner to `session-only-degraded-state.md` only ("The obligations on this page are blocked …"), gating that page's branch behaviour while the parallel ungated normative description of the same branch on `drain-state-contract.md` was left untouched. The subsequent re-scope `8bcd52a` (F-1163, 2026-06-23, "gate scopes by clause not page") refined the banner to carve out the seam contracts and explicitly named `drain-state-contract.md` as a *consumer* of the non-gated clauses, but did not extend the gate to the branch-behaviour material that `drain-state-contract.md` also pins, so the contradiction persisted. Later commits `3137321` (F-1349, coin PIC-29..32) and `8008b91` (F-1217, add acceptance criteria) deepened the ungated branch-behaviour content on `drain-state-contract.md`.

## Solution Space

### Recommendation

Extend the clause-scoped implementation gate into `drain-state-contract.md` so both degraded-state pages reference one shared gate state and agree on what is blocked. Treat the host-prerequisites clause-(a) gate as a single gate state that both pages point at, and mark the degraded-branch clauses on `drain-state-contract.md` as gated on the same condition while leaving the genuinely-buildable seam contracts ungated.

Spec edits:

- Add a clause-scoped gate note to `drain-state-contract.md` (top of file, or inline at the affected clauses) declaring **gated on host-prerequisites clause (a) `provably-differs`**: the degraded-branch portions of the `idempotent` clause (the `"degraded-needs-reload"`-arm re-entry narrative and the all-three-throw corner case), PIC-30's degraded-arm prohibition, and PIC-31's degraded read-failure-fallback re-entry path.
- Keep explicitly **non-gated (buildable now)** as seam contracts: the `LoomRegistry` field/method definitions including the `markRuntimeDegraded` setter (already declared non-gated on the sibling), PIC-32 (`drain`), the arm-(a)/(b) `"shutting-down"` routing, PIC-29's three-arm tuple-exhaustion structure, and the `(drained: true, tag: undefined) → arm (b)` acceptance criteria.
- Repoint `session-only-degraded-state.md`'s gate banner and `drain-state-contract.md`'s note at one shared gate-state anchor so the two pages cannot drift again.

Draw the gate boundary so that pure seam contracts (the `markRuntimeDegraded` setter, the arm-(c) tag *read* and routing, PIC-29's tuple structure) stay buildable while the branch *decision* that writes the degraded tag and the degraded re-entry/corner-case semantics are gated. Take care not to over-gate the `"shutting-down"` arm (b) — it fires on always-tear-down `quit`/`reload` reasons and is not degraded-branch-specific. The `"shutting-down"` arm (b), PIC-32, and the entry-table `Map` interface are likewise not degraded-branch-specific and must remain ungated.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reading `drain-state-contract.md` builds the post-sub-step-4 degraded transition, arm-(c) routing, and degraded re-entry semantics immediately; one reading `session-only-degraded-state.md` treats the same branch as blocked and skips it. The gate exists to hold the degraded branch until the instance-survival contradiction at clause (a) is settled; leaving the sibling page ungated lets that work proceed unguarded, risking exactly the rework the gate is meant to prevent.
