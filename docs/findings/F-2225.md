## Problem

`session-only-degraded-state.md` opens with a clause-scoped implementation gate that blocks every degraded-branch clause on the page — the predicate split, recovery path and prohibitions, diagnostic emission, idempotent re-entry / all-three-throw corner case, and the acceptance-criteria fixtures (a)–(m) — pending resolution of the open contradiction recorded at `host-prerequisites.md#degraded-state-host-prerequisites` clause (a), clearing only on the `provably-differs` verdict. The sibling `drain-state-contract.md` carries no gate banner yet normatively pins the same degraded-branch behaviour as buildable-now: the `idempotent-handler-clause` degraded re-entry semantics, PIC-29's arm (c) `"degraded-needs-reload"`, PIC-30's fourth-arm prohibition, PIC-31's read-failure fallback, and the `readDrainState` arm-(c) routing. The result is a direct cross-file contradiction — the writer-side branch logic that produces the degraded tag is blocked on one page while the reader-side routing, re-entry semantics, and corner-case handling that consume it are presented as fully specified on the other. An implementer cannot tell whether the degraded transition and arm (c) are in scope now or blocked, and the gate's purpose is defeated for anyone working from the sibling page.

## Issue introduction
**Verdict:** single introducing commit (regression introduced by a one-sided gate)
**Introducing commit:** `b8f004f` — "pi-loom spec-plan: fix F-0719 — SM-4/5/6/3b degraded-state implementation-gate contingency markers" (2026-06-19)
**History:** The degraded-arm reader contract in `drain-state-contract.md` (the `"degraded-needs-reload"` arm (c) and the idempotent re-entry narrative) predates the gate — it was already present when the two pages were split under the size cap in `f5e89f4` (2026-06-04), and `git show f5e89f4:.../drain-state-contract.md` shows the `degraded-needs-reload` arm already established. The contradiction entered in `b8f004f`, which added the `> *Implementation gate.*` banner to `session-only-degraded-state.md` only ("The obligations on this page are blocked …"), gating that page's branch behaviour while the parallel ungated normative description of the same branch on `drain-state-contract.md` was left untouched. The subsequent re-scope `8bcd52a` (F-1163, 2026-06-23, "gate scopes by clause not page") refined the banner to carve out the seam contracts and explicitly named `drain-state-contract.md` as a *consumer* of the non-gated clauses, but did not extend the gate to the branch-behaviour material that `drain-state-contract.md` also pins, so the contradiction persisted. Later commits `3137321` (F-1349, coin PIC-29..32) and `8008b91` (F-1217, add acceptance criteria) deepened the ungated branch-behaviour content on `drain-state-contract.md`.

## Solution approach

Extend the clause-scoped implementation gate into `drain-state-contract.md` so both degraded-state pages reference one shared gate state gated on `host-prerequisites.md#degraded-state-host-prerequisites` clause (a) `provably-differs`. Add a gate note to `drain-state-contract.md` marking the degraded-branch portions as gated — the `idempotent-handler-clause` degraded re-entry narrative and all-three-throw corner case, PIC-30's degraded-arm prohibition, and PIC-31's degraded read-failure-fallback re-entry path — while leaving the genuinely-buildable seam contracts ungated (the `LoomRegistry` field/method definitions including the `markRuntimeDegraded` setter, PIC-32, the arm-(a)/(b) `"shutting-down"` routing, PIC-29's three-arm tuple-exhaustion structure, and the `(drained: true, tag: undefined) → arm (b)` acceptance criteria). Repoint `session-only-degraded-state.md`'s gate banner and the new note at one shared gate-state anchor so the two pages cannot drift again.

## Solution constraints

- Do not gate the `"shutting-down"` arm (b), PIC-32, or the entry-table `Map` interface — they fire on always-tear-down `quit`/`reload` reasons and are not degraded-branch-specific.
- Out of scope: the gated-write site owned by F-2224 and the visual-separation surface owned by F-2226.
