## Finding

`.pi/project-config.md` §"Spec rules" → "Stable inline labels" enumerates the live inline-label prefixes as `HC3`, `NOCEIL` ("both bound to `hard-ceilings.md`"). That list is stale: `docs/spec_topics/governance.md` — the authoritative GOV-16 owner per the project's `idAllocationPolicy: governed-by-spec` — registers a third live inline-label prefix, `SNK-a`…`SNK-k`, bound to `slash-invocation.md`, and `slash-invocation.md` carries the eleven `SNK-*` rows with their per-page GOV-16 prefix registration. The plan corpus is consistent with the spec here: `V12b`/`V12b-T` correctly cite `SNK-a..k` under `SLSH-4`. Only the project-config snapshot lags.

The hazard is sharpened by an asymmetry inside project-config.md itself: the adjacent REQ-IDs bullet carries an explicit "this snapshot is not authoritative — `governance.md` owns the live GOV set" disclaimer, but the "Stable inline labels" bullet carries no such disclaimer, so its two-prefix list reads as complete. A fixer consulting it to police the GOV-16 "MAY NOT silently introduce a new prefix" allocation rule would conclude `SNK` is unregistered and could wrongly challenge `V12b`'s correct usage as an invented prefix.

This is a project-doc-vs-spec alignment defect, not a plan defect: the plan and spec already agree. Because `.pi/project-config.md` lies outside the orchestrator's `commitAddPaths: [docs/]` and is a project-configuration artefact rather than a plan or spec document, the durable plan-side fix is a doc-impact note in the plan flagging the snapshot for follow-up, with `governance.md` left as the authoritative source.

## Issue introduction

**Verdict:** introduced-by-a-later-spec-change
**Introducing commit:** `27b2576` (2026-06-05) "pi-loom spec: resolve \"SLSH-4 covers 11 independently normative template rows under one REQ-ID\""
**History:** The "Stable inline labels" bullet listing `HC3`, `NOCEIL` was authored on 2026-05-22 in `899b8fa` (`git blame -L 40,41 .pi/project-config.md`), when those were the only two live inline-label prefixes. `git log -S 'SNK-a'` shows `SNK-a`…`SNK-k` were registered in both `docs/spec_topics/governance.md` and `docs/spec_topics/slash-invocation.md` by a single later commit, `27b2576` (2026-06-05). That commit did not touch `.pi/project-config.md` (it is absent from `git log -- .pi/project-config.md`), so the spec gained a third live inline-label prefix while the project-config snapshot was left at two — the divergence was introduced by the SNK registration's omission of a corresponding snapshot update, not present since the snapshot's inception.

## Solution Space

**Shape:** single

### Recommendation

Add a doc-impact note to `docs/plan_topics/conventions.md` §REQ-ID discipline, adjacent to the GOV-16 inline-label back-reference, recording that the project-config snapshot is stale and pointing at the authoritative source. Suggested literal text:

> **Doc impact:** `.pi/project-config.md` §"Spec rules" → "Stable inline labels" currently lists `HC3` and `NOCEIL` as the live inline-label prefixes; the snapshot is stale — `SNK` (on `slash-invocation.md`, per GOV-16) is also registered and live. `governance.md` is the authoritative source per `idAllocationPolicy: governed-by-spec`; the `.pi/project-config.md` listing needs a follow-up update.

Constraints for the fixer:

- The spec is read-only for this fix — `governance.md` and `slash-invocation.md` already register `SNK` correctly; do not edit them.
- Do not edit `.pi/project-config.md` directly. It is a project-configuration artefact outside `commitAddPaths: [docs/]`, so the orchestrator will not stage it; the durable plan-side resolution is the in-plan doc-impact note, leaving the actual snapshot update as a flagged human follow-up.
- `docs/plan.md` §Authoring (or §Spec coverage) is an acceptable alternate home for the note if the conventions §REQ-ID discipline placement is unsuitable; the content is the same.

## Consequence

**Severity:** advisory

A fixer or implementer consulting `.pi/project-config.md`'s inline-label snapshot sees an incomplete prefix space (`HC3`, `NOCEIL`) presented as complete, and may wrongly treat `V12b`'s correct `SNK` citations as silently introducing an unregistered GOV-16 prefix. `governance.md` remains authoritative, so a careful actor recovers, but the stale snapshot is a latent misdirection at the exact surface that gates prefix allocation.
