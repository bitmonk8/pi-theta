## Finding

In `docs/spec_topics/pi-integration-contract/version-bump-triggers.md`, the `<a id="bump-step-5-section7-reason-row"></a>*§7 closed-enum `<reason>` row co-update (convention).*` paragraph at the tail of step 5 assigns two different modal weights to the same edit, in adjacent sentences. The first sentence says *"the contributor SHOULD update that table's `<reason>` enumeration in the same commit as the snapshot edit"* (RFC 2119 `SHOULD`: deviation requires reasoned justification and is reviewable). The very next sentence says *"This co-update is a contributor-discipline convention, not a mechanically-enforced co-edit"* — relegating it to convention, which is weaker than `SHOULD` (no documented expectation of justification on deviation; nothing for a reviewer to flag against). The paragraph then justifies the convention framing by observing that no build-time gate fires and "a §7 row left transiently stale carries no runtime-correctness impact".

The two framings are not equivalent: a `SHOULD` left unjustified is a reviewable defect, whereas a *convention* left unfollowed is not. A contributor reading the first sentence will treat the §7 row update as a same-commit obligation; a contributor reading the second sentence will treat it as omittable. Reviewers face the same fork: the `SHOULD` invites them to block the bump commit on a missing §7 update; the *convention* sentence tells them not to. The "next page-touching editorial sweep" reconciliation tail compounds the ambiguity by naming an out-of-commit reconciliation path that is consistent with the convention reading but not with the `SHOULD` reading.

The defect entered with commit `a4cc7ee` ("pi-loom spec: resolve '§7 reason-row co-edit obligation and its enforcement gap (T54/T55)'"), which folded two predecessor review findings into one paragraph: T54 asked for the §7 co-edit obligation to be stated at step 5 (motivating the `SHOULD` sentence) and T55 asked for either a mechanical gate or a demotion to a SHOULD-level convention (motivating the second sentence). The combined edit kept both halves verbatim and produced the modal conflict.

## Solution Space

**Shape:** multiple

### Option A — Keep the `SHOULD`, drop the convention disclaimer

**Approach.** The §7 row update is a same-commit obligation at `SHOULD` weight; deviation requires reviewer-visible justification on the bump PR.

**Spec edits.**
- Remove the sentence *"This co-update is a contributor-discipline convention, not a mechanically-enforced co-edit"* in its entirety.
- Rewrite the *"so a §7 row left transiently stale carries no runtime-correctness impact … reconciled by the next page-touching editorial sweep"* tail. Replace it with the no-mechanical-gate disclosure framed as a reviewer-flag note: e.g. *"no build-time gate fires when the §7 `<reason>` row is left unedited, so the obligation is reviewer-enforced; a bump PR that elides the §7 update MUST carry a justification, and a §7 row left transiently stale carries no runtime-correctness impact (the closed-arm `<reason>` renderer emits the runtime `event.reason` value, not the spec table)."*
- Drop the "reconciled by the next page-touching editorial sweep … five in-prose illustrative restatements" comparison (that precedent applies to no-modal-weight editorial restatements, not to `SHOULD`-weight obligations, and keeping it re-introduces the conflict).

**Pros.**
- Restores T54's original intent (same-commit co-edit at the step that enumerates the bump commit's edits).
- Aligns the §7 row's modal weight with the inline-triplet sweep above it in the same step, which is `MUST`-weight on the same surface (the `SessionShutdownEvent['reason']` snapshot).
- Reviewer-enforced `SHOULD` is the same enforcement model used elsewhere in the procedure (e.g. step 6's seed-field judgement obligation).

**Cons.**
- Imposes a per-bump reviewer-attention cost on a paragraph the original T55 fix wanted to discharge to background editorial sweeps.
- Reviewers must actually check the §7 row on every bump PR; reviewer drift can silently weaken the `SHOULD`.

**Risks.**
- Low. The closest analogue (seed-field judgement at step 6) has the same shape and works.

### Option B — Drop the `SHOULD`, keep the convention framing

**Approach.** The §7 row update is a non-load-bearing editorial convention reconciled by the next page-touching editorial sweep; no bump-commit obligation, no reviewer flag.

**Spec edits.**
- Replace *"the contributor SHOULD update that table's `<reason>` enumeration in the same commit as the snapshot edit"* with *"the contributor MAY update that table's `<reason>` enumeration in the same commit as the snapshot edit; otherwise the row is reconciled by the next page-touching editorial sweep."* (`MAY` is the RFC 2119 modal that matches "convention".)
- Keep the *"This co-update is a contributor-discipline convention, not a mechanically-enforced co-edit"* sentence and the editorial-sweep tail; they are now consistent with the `MAY`.

**Pros.**
- Honours T55's "demote to a SHOULD-level convention" alternative and the no-runtime-correctness-impact observation.
- Zero per-bump cost on contributors and reviewers; the editorial-sweep precedent applies cleanly.

**Cons.**
- Re-opens T54: step 5 (which enumerates the bump commit's edits) is no longer the place the §7 row update is stated as a bump-commit edit, because there is no bump-commit edit on this surface.
- Lengthens the window during which the §7 row drifts out of sync with the snapshot constant (bounded only by the cadence of unrelated page-touching edits to `placeholder-rendering-b.md`).

**Risks.**
- The §7 row is a publicly-documented GOV-7 / GOV-8 surface; observer-visible drift between the spec table and the runtime set may surprise consumers reading the spec to predict the closed-arm value set. The convention framing accepts that surprise as a known cost.

### Recommendation

Option A. The inline-triplet sweep one paragraph above (`MUST` on the same `SessionShutdownEvent['reason']` snapshot) sets the expectation that snapshot-surface co-edits land in the bump commit; the §7 row sits on the same snapshot surface and should land on the same footing one notch weaker (`SHOULD` rather than `MUST`, because there is no runtime-correctness fallout). Option B's editorial-sweep deferral is the right shape for in-prose illustrative restatements but not for a publicly-documented GOV-7 / GOV-8 closed-enum table. The implementer should additionally drop the "five in-prose illustrative restatements" comparison so the paragraph does not silently re-import the convention framing through the comparison's precedent.

## Consequence

**Severity:** correctness

Two reasonable contributors will land divergent behaviour: one will read the `SHOULD` and update the §7 `<reason>` table in the bump commit (and reviewers will block bumps without the update); the other will read the "convention, not a mechanically-enforced co-edit" disclaimer and defer the §7 edit to a later editorial sweep (and reviewers will not block). The §7 table will silently drift out of sync with the `SessionShutdownEvent['reason']` snapshot constant for an unbounded window on some bumps but not others, and review-time enforcement will be inconsistent across bump PRs. There is no runtime-correctness fallout (the closed-arm renderer reads the runtime `event.reason` value, not the spec table), but the spec table is a publicly-documented GOV-7 / GOV-8 surface, so the drift is observer-visible.
