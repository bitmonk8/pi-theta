## Finding

In `docs/spec_topics/pi-integration-contract/version-bump-triggers.md`, the `<a id="bump-step-5-section7-reason-row"></a>*§7 closed-enum `<reason>` row co-update (convention).*` paragraph at the tail of step 5 assigns two different modal weights to the same edit, in adjacent sentences. The first sentence says *"the contributor SHOULD update that table's `<reason>` enumeration in the same commit as the snapshot edit"* (RFC 2119 `SHOULD`: deviation requires reasoned justification and is reviewable). The very next sentence says *"This co-update is a contributor-discipline convention, not a mechanically-enforced co-edit"* — relegating it to convention, which is weaker than `SHOULD` (no documented expectation of justification on deviation; nothing for a reviewer to flag against). The paragraph then justifies the convention framing by observing that no build-time gate fires and "a §7 row left transiently stale carries no runtime-correctness impact".

The two framings are not equivalent: a `SHOULD` left unjustified is a reviewable defect, whereas a *convention* left unfollowed is not. A contributor reading the first sentence will treat the §7 row update as a same-commit obligation; a contributor reading the second sentence will treat it as omittable. Reviewers face the same fork: the `SHOULD` invites them to block the bump commit on a missing §7 update; the *convention* sentence tells them not to. The "next page-touching editorial sweep" reconciliation tail compounds the ambiguity by naming an out-of-commit reconciliation path that is consistent with the convention reading but not with the `SHOULD` reading.

The defect entered with commit `a4cc7ee` ("pi-loom spec: resolve '§7 reason-row co-edit obligation and its enforcement gap (T54/T55)'"), which folded two predecessor review findings into one paragraph: T54 asked for the §7 co-edit obligation to be stated at step 5 (motivating the `SHOULD` sentence) and T55 asked for either a mechanical gate or a demotion to a SHOULD-level convention (motivating the second sentence). The combined edit kept both halves verbatim and produced the modal conflict.

## Solution Space

### Recommendation

Make the §7 closed-enum `<reason>` row update a same-commit obligation at `SHOULD` weight, with reviewer-visible justification required on the bump PR when the contributor deviates. Edit the `*§7 closed-enum `<reason>` row co-update*` paragraph at the tail of step 5 in `docs/spec_topics/pi-integration-contract/version-bump-triggers.md` as follows:

- Remove the sentence *"This co-update is a contributor-discipline convention, not a mechanically-enforced co-edit"* in its entirety.
- Rewrite the *"so a §7 row left transiently stale carries no runtime-correctness impact … reconciled by the next page-touching editorial sweep"* tail as a reviewer-flag note: *"no build-time gate fires when the §7 `<reason>` row is left unedited, so the obligation is reviewer-enforced; a bump PR that elides the §7 update MUST carry a justification, and a §7 row left transiently stale carries no runtime-correctness impact (the closed-arm `<reason>` renderer emits the runtime `event.reason` value, not the spec table)."*
- Drop the "reconciled by the next page-touching editorial sweep … five in-prose illustrative restatements" comparison entirely. That editorial-sweep precedent applies to no-modal-weight in-prose illustrative restatements, not to a `SHOULD`-weight obligation on a publicly-documented GOV-7 / GOV-8 closed-enum table; retaining the comparison silently re-imports the convention framing through its precedent.

This keeps the §7 row's modal weight aligned with the inline-triplet sweep one paragraph above it in the same step (which is `MUST`-weight on the same `SessionShutdownEvent['reason']` snapshot surface), landing the §7 row on the same footing one notch weaker — `SHOULD` rather than `MUST`, because there is no runtime-correctness fallout. Reviewer-enforced `SHOULD` is the same enforcement model the procedure uses elsewhere (e.g. step 6's seed-field judgement obligation).

## Consequence

**Severity:** correctness

Two reasonable contributors will land divergent behaviour: one will read the `SHOULD` and update the §7 `<reason>` table in the bump commit (and reviewers will block bumps without the update); the other will read the "convention, not a mechanically-enforced co-edit" disclaimer and defer the §7 edit to a later editorial sweep (and reviewers will not block). The §7 table will silently drift out of sync with the `SessionShutdownEvent['reason']` snapshot constant for an unbounded window on some bumps but not others, and review-time enforcement will be inconsistent across bump PRs. There is no runtime-correctness fallout (the closed-arm renderer reads the runtime `event.reason` value, not the spec table), but the spec table is a publicly-documented GOV-7 / GOV-8 surface, so the drift is observer-visible.
