## Finding

The step-2 preamble of `docs/spec_topics/pi-integration-contract/version-bump-step2.md` (the single sentence opening the *Editorial-review checklist for unpinned host presuppositions* block) scopes the SHOULD-level audit and the MUST-level recording obligation to "items (f) through (aj)" three times (twice as `(f) through (aj)`, once as the dash form `(f)–(aj)`). Item `(ak)` (`bump-checklist-user-visible-streaming-ordering`, *User-visible streaming ordering before `waitForIdle()` resolution*) sits below `(aj)` in the same enumeration and is shaped as a SHOULD-level audit with an escalation clause identical to the surrounding `(f)`–`(aj)` items.

This contradicts itself in two directions. The preamble's recording rule — "omitting the row is non-conformant" — is bounded to `(f)–(aj)`, so an auditor who omits `(ak)` from the bump commit message is conformant on the preamble's plain text. The same auditor recording `(ak)` is also conformant (the item is real and SHOULD-audited). Both behaviours satisfy the spec, so the recording schema admits two non-equivalent conformant outputs at the bump commit. The downstream plan leaf `V18c` already mandates per-item outcomes for `(a)`–`(ak)` (its **Ships when** explicitly names that range), so the spec's narrower preamble actively under-specifies the obligation `V18c` reads off it.

The fix is mechanical: extend both `(f) through (aj)` occurrences and the `(f)–(aj)` dash form in the preamble sentence to `(ak)`. The `Baseline acquisition` paragraph's cross-reference list (items `(e)`, `(r)`, `(aj)`) names `(aj)` for a different reason — `(aj)` reads a `dist/*.js` JS body and the baseline serves that read — and is unaffected; `(ak)` does no `dist/*.js` body read.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commit:** `85fc906` — *pi-loom plan: resolve "streamed-token-before-waitForIdle() ordering has no version-bump checklist item"* (2026-06-10)
**History:** `git log -S "(f) through (aj)" -- docs/spec_topics/pi-integration-contract/version-bump-step2.md` and `git log -S "through (ai)" -- docs/spec_topics/pi-integration-contract/version-bump-step2.md` walk the preamble range across its rewrites. Commit `9ced9f1` (2026-06-08, *subagent transcript-discard not routed to bump checklist*) added item `(aj)` and correctly extended the preamble range from `(f) through (ai)` to `(f) through (aj)` in the same edit. Commit `85fc906` then added item `(ak)` two days later (the streaming-ordering presupposition) but did not extend the preamble range in the same edit — `git show 85fc906 -- docs/spec_topics/pi-integration-contract/version-bump-step2.md` shows only the `(ak)` row insertion and a one-line `(aj)` cross-reference update, with the preamble sentence at line 3 untouched. The defect is therefore a co-edit miss in commit `85fc906`: the `(aj)`-adding commit had set the co-edit pattern (extend the preamble range alongside any new tail item), and the `(ak)`-adding commit broke it. The closing-paragraph mandatory-expansion gate already names this category of obligation ("Any future presupposition added to this page ... MUST be added to this checklist in the same edit") but does not call out the preamble-range edit as part of the same-edit obligation, which is the seam the miss slipped through.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/pi-integration-contract/version-bump-step2.md`, edit the step-2 preamble sentence (the sole occurrence at line 3) to replace each of its three range references — the two `(f) through (aj)` occurrences and the single `(f)–(aj)` dash form — with `(f) through (ak)` and `(f)–(ak)` respectively. Leave the `Baseline acquisition` paragraph's `[(aj)](#bump-checklist-inmemory-transcript-discard)` cross-reference untouched: that list names items that prescribe `dist/*.js` JS body reads (`(e)`, `(r)`, `(aj)`) and `(ak)` is not such an item. No other file changes are required; `V18c` / `V18c-T` already enumerate the correct `(a)`–`(ak)` / `(f)`–`(ak)` ranges. Implementer-relevant edge case: the same-edit obligation in the closing-paragraph mandatory-expansion gate should be read as covering the preamble-range extension as well as the item insertion, so the next new tail item (`(al)`, etc.) does not regress the preamble in the same way; this edit does not need to restate that obligation, but a contributor adding the next item must extend the preamble range in the same commit.

## Consequence

**Severity:** correctness

Two reasonable bump auditors diverge on whether `(ak)` must appear in the bump commit message: the preamble's recording rule reads as scoping `(ak)` out, while item `(ak)`'s own SHOULD-audit shape and the `V18c` **Ships when** read it in. A bump commit that omits the `(ak)` row passes editorial review under one reading and fails it under the other, and the disagreement falls on the merge gate that `V18c` enforces.
