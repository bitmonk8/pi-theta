## Problem

QRY-18's stringification table pins the scalar `integer` / `number` rows to BNDR-4 / BNDR-5 (never scientific notation at any IEEE-754 magnitude, `-0 â†’ 0`), but the `array<T>` and `Schema-typed object` rows delegate to bare `JSON.stringify`. Node's `JSON.stringify` renders numbers via the engine's `String(n)` path, emitting exponential form at exactly the magnitudes BNDR-5 forbids â€” `${{x: 1e21}}` produces `{"x":1e+21}` while `${1e21}` produces `1000000000000000000000`. The same `number` value therefore renders to different model-visible bytes depending on whether it sits at the top of an interpolation or nested inside an object/array, contradicting the global never-scientific-notation pin BNDR-5 declares. The delegation also leaves object key order under-specified, since `JSON.stringify` walks keys in construction-history insertion order rather than as a function of the value.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `a9c5d7b` (2026-06-05) â€” *"pi-loom spec: resolve "number row omits scientific-notation and -0 pins""*
**History:** Commit `0c9ce54` (2026-05-04, *"Stringification rule for ${expr} interpolations is unspecified"*) introduced the entire stringification table in its original form, with `integer`/`number` rows stating only "shortest round-trip decimal" and the `array<T>`/`Schema-typed object` rows delegating to `JSON.stringify`. At that point the table was internally consistent: neither the scalar rows nor the JSON-delegating rows explicitly forbade scientific notation, so a uniform `String(n)`-based implementation satisfied both. Commit `a9c5d7b` (2026-06-05) tightened only the scalar rows, rewriting them to *"per [BNDR-4] (canonical decimal `42`, `-7`; never scientific notation; `-0` â†’ `0`)"* and *"finite values per [BNDR-5] (... never scientific notation; `-0` â†’ `0`)"* without touching the `array<T>` / `Schema-typed object` rows; this asymmetric tightening introduced the by-context divergence. Commit `716a13b` (2026-06-25) subsequently coined the `qry-18` anchor on the unchanged paragraph above the table but did not alter the table content. The key-ordering under-specification is a separate, milder symptom of the same delegate-to-bare-`JSON.stringify` pattern and has been present in the table since `0c9ce54`.

## Solution approach

Rewrite QRY-18's `array<T>` and `Schema-typed object` rows so `JSON.stringify` is named only for the compact-envelope shape and every primitive leaf is re-emitted by its own row of the table â€” `number` leaves per [BNDR-5](../binder/defaulting-system-note-echo.md#bndr-5), `integer` leaves per [BNDR-4](../binder/defaulting-system-note-echo.md#bndr-4) â€” rather than by the engine default. Add an object-key-ordering overlay that emits keys at every nesting level in ascending Unicode code-point order, citing [BNDR-8](../binder/binder-model-and-context.md#bndr-8) as the canonical-serialisation precedent. Add conformance vectors covering a forbidden-magnitude nested number and a key-order normalisation case. The fix lives entirely in `query-escapes-stringification.md`.
