## Finding

QRY-18's stringification table pins the scalar `integer` and `number` rows to BNDR-4/BNDR-5 — explicitly "never scientific notation" at every IEEE-754 magnitude, including the two forbidden `String(n)` switches at `±1e21` and `|value| < 1e-7`. The `array<T>` and `Schema-typed object` rows, by contrast, name `JSON.stringify` of the value as the rendering rule. Node's `JSON.stringify` formats numbers via the engine's `String(n)` path and therefore emits exponential form at exactly those forbidden magnitudes (e.g. `JSON.stringify({x: 1e21})` → `{"x":1e+21}`, `JSON.stringify([1e-8])` → `[1e-8]`).

The consequence is that the same `number` value renders to different model-visible bytes depending on whether it appears at the top of a `${expr}` interpolation or nested inside an interpolated object/array: `${1e21}` produces `1000000000000000000000` while `${{x: 1e21}}` produces `{"x":1e+21}`. This directly contradicts the "never scientific notation in loom 1.0" pin that BNDR-5 — cited by the very table that contains the defect — declares for the `number` type globally.

The same delegation also leaves object key order under-specified at this site. `JSON.stringify` walks own enumerable string keys in insertion order, which is determined by however the loom-side value was constructed (literal, object-builder, validator round-trip, etc.) and is not a property of the *value* but of its construction history. BNDR-8 has already had to introduce a key-sorting overlay on top of `JSON.stringify` to make the binder transcript byte-reproducible; the interpolation table needs the same overlay for the same reason (the rendered prompt is hashed into session bytes downstream of binder rendering).

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `a9c5d7b` (2026-06-05) — *"pi-loom spec: resolve "number row omits scientific-notation and -0 pins""*
**History:** Commit `0c9ce54` (2026-05-04, *"Stringification rule for ${expr} interpolations is unspecified"*) introduced the entire stringification table in its original form, with `integer`/`number` rows stating only "shortest round-trip decimal" and the `array<T>`/`Schema-typed object` rows delegating to `JSON.stringify`. At that point the table was internally consistent: neither the scalar rows nor the JSON-delegating rows explicitly forbade scientific notation, so a uniform `String(n)`-based implementation satisfied both. Commit `a9c5d7b` (2026-06-05) tightened only the scalar rows, rewriting them to *"per [BNDR-4] (canonical decimal `42`, `-7`; never scientific notation; `-0` → `0`)"* and *"finite values per [BNDR-5] (... never scientific notation; `-0` → `0`)"* without touching the `array<T>` / `Schema-typed object` rows; this asymmetric tightening introduced the by-context divergence. Commit `716a13b` (2026-06-25) subsequently coined the `qry-18` anchor on the unchanged paragraph above the table but did not alter the table content. The key-ordering under-specification is a separate, milder symptom of the same delegate-to-bare-`JSON.stringify` pattern and has been present in the table since `0c9ce54`.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the `array<T>` and `Schema-typed object` rows so the JSON envelope no longer carries number formatting or key order on its own. The structural envelope stays `JSON.stringify`-compatible (compact, no pretty-printing, wire-name translation applied recursively); the rendering of every *primitive* leaf inside that envelope inherits the same per-type rules the table already names for top-level interpolation. Concretely:

- `number` leaves render per [BNDR-5](../binder/defaulting-system-note-echo.md#bndr-5), including the no-scientific-notation pin and the `-0 → 0` normalisation; `NaN`, `Infinity`, `-Infinity` render as the literal unquoted tokens `NaN`, `Infinity`, `-Infinity` (matching the scalar `number` row, even though those tokens are not legal JSON — this is a model-facing string, not parseable JSON, and the scalar row already commits to it).
- `integer` leaves render per [BNDR-4](../binder/defaulting-system-note-echo.md#bndr-4).
- `string` leaves render quoted, with `JSON.stringify`'s string-escaping (the standard `\"`, `\\`, `\n`, `\r`, `\t`, `\b`, `\f`, and `\u00XX` for other C0/U+2028/U+2029, identical to the engine default) — this is the one place the JSON envelope must override the bare `string` row's "no quoting" rule, because the leaves sit inside a structural JSON container that requires quoted strings.
- `boolean` leaves render `true` / `false`; `null` renders `null`; enum-variant leaves render as the quoted wire string (the brand is dropped, matching the existing scalar enum row).
- Object key emission overlays a sort: keys at every nesting level are emitted in ascending Unicode code-point order, identical to BNDR-8's canonical-no-whitespace overlay, so the rendered bytes are a function of the value rather than its construction history. Cite BNDR-8 as the precedent rather than restating the algorithm.
- Keep "compact, no pretty-printing" and "wire-name translation applied recursively" as structural requirements unchanged.

State the inheritance explicitly so an implementer cannot fall back to bare `JSON.stringify`: e.g. *"`JSON.stringify` is not normative; it is named only for the compact-envelope shape. Every primitive leaf is re-emitted by the corresponding row of this table, and object keys are emitted in ascending Unicode order per [BNDR-8]'s canonical-no-whitespace overlay."* Add at least one conformance vector covering a forbidden-magnitude nested number (e.g. `${{x: 1e21}}` → `{"x":1000000000000000000000}`) and one covering key-order normalisation (a two-key object constructed insertion-order `b, a` rendering as `{"a":...,"b":...}`).

The fix lives entirely in `query-escapes-stringification.md`; no edit is needed in `defaulting-system-note-echo.md`, `binder-model-and-context.md`, or `runtime-value-model.md` (which are read-only references the rewritten rows cite).

## Consequence

**Severity:** correctness

Two implementers writing to the current spec will diverge on the rendered prompt bytes for any interpolated object/array containing a number with magnitude `≥ 1e21` or `0 < |n| < 1e-7`, and for any object whose keys were not constructed in ascending Unicode order. The defect propagates beyond `@`-query templates because the same table powers `system:` interpolation (per QRY-18's own cross-reference). Downstream consequences include divergent session bytes, divergent transcript hashing, and a behavioural contradiction with the explicit BNDR-5 pin that loom 1.0 must reproduce a value identically wherever it surfaces to the model.
