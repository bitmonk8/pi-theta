## Finding

The registered diagnostic `loom/load/cross-format-collision` fires for the **entire** same-priority slash-name collision rule — both the cross-format arm (`.loom` vs a Pi-owned `.md` prompt / `.md` skill / another extension's command) and the same-format arm (two `.loom` files at the same priority: two packages each shipping `code-review.loom`, two settings entries, or two `--loom` components). The `cross-format` token in the code name describes only the first arm. The code-registry-load.md Trigger cell itself documents the contradiction ("its scope is the full same-priority slash-name collision rule, not only the cross-format arm") and DISC-4 in discovery-sources.md enumerates the loom-vs-loom arm under this code, so the behaviour is fully pinned — but the public identifier that authors, log readers, tests, and LSP integrations key on still announces the narrower scope.

The mislabel propagates into cross-references. `active-invocation-registry.md` cites the rule twice as "[Discovery — Slash-name collisions across formats]" while using that link specifically for the same-priority `.loom`-vs-`.loom` arm (callee-only basename collisions, `pi.looms` collisions) — the same page's other two references to the identical target instead read "[Discovery — Slash-name collision rules]". The "across formats" label is therefore inaccurate for the arm it is attached to.

The code cannot be renamed in loom 1.0: DIAG-3 makes every registered code a stable public-contract identifier, and a rename is a breaking change deferred to loom 2.0 (outside the GOV-15 add/remove carve-out). The actionable 1.0 work is editorial — align the inaccurate cross-reference text, surface the dual-arm scope earlier in the Trigger cell, and record the 2.0 rename intent — not a code change.

## Issue introduction

**History:** `git log -S 'cross-format-collision'` shows the code entered as a cross-format-only diagnostic — through `c5c4e79` (2026-05-04, "resolve 'Cross-format slash collision rule cannot achieve neither is registered'") its trigger read "A `.loom` candidate's slash name collides with an already-registered Pi prompt template … the loom is dropped", and the name was accurate. `d124b8c` (Decision #10) expanded the same code to "Two or more candidates at the same priority derive the same slash name. Covers both the same-format case … and the cross-format case", folding the loom-vs-loom arm into the existing code while retaining the `cross-format` token. The same commit renamed the discovery.md section heading to "Slash-name collisions at the same priority" but left the `cross-format` code name and the older "across formats" cross-reference label in place — the moment the name/scope mismatch and the stale link text were introduced.

## Solution Space

**Shape:** single

### Recommendation

Keep the registered code `loom/load/cross-format-collision` (DIAG-3 forbids a 1.0 rename) and resolve the mismatch editorially:

1. In `active-invocation-registry.md`, change both "[Discovery — Slash-name collisions across formats](../discovery.md)" link texts to "[Discovery — Slash-name collision rules](../discovery.md)" — matching the DISC-4 bold lead-in ("**Slash-name collision rules.**") in discovery-sources.md and the page's other two references to the same target, so the same-format arm is no longer labelled "across formats".
2. In the code-registry-load.md `loom/load/cross-format-collision` Trigger cell, move the scope clarification ("its scope is the full same-priority slash-name collision rule, not only the cross-format arm") to the start of the cell, ahead of the cross-format detail, so a reader mapping the code to its scope sees the dual-arm coverage first.
3. Record the intended loom 2.0 rename to `loom/load/slash-name-collision` in `future-considerations.md`, so the deferral is tracked rather than only acknowledged inline.

Edge cases for the implementer: do **not** alter the code token, the severity (`E`), the DISC-4 behaviour, or the rendered message template `slash name '<name>' collides at the same priority: <paths>` — all are stable contract. The cross-reference target `discovery.md` is a Contents/aggregator page with no per-section anchor, so fix only the link text to match the DISC-4 heading; do not invent an anchor fragment that does not exist.

## Consequence

**Severity:** advisory

No observer's behaviour diverges: both collision arms drop all colliding looms, and the rendered message (`slash name '<name>' collides at the same priority: <paths>`) carries no "cross-format" wording. The cost is a clarity/diagnosis gap — an author or log reader who sees `loom/load/cross-format-collision` fire on two `.loom` files, or follows the "collisions across formats" cross-reference for the same-format arm, may conclude the code or the rule does not apply to loom-vs-loom collisions.
