## Finding

Version-bump checklist item (a) (`#bump-checklist-instance-survival`) records a one-shot *open contradiction*: whether loom's package-loaded `extensions/index.ts` survives a session-only `session_shutdown`, or is torn down and rebound per Pi's documented lifecycle. The spec handles the *machinery* of this open question thoroughly — item (a) carries a defined two-verdict recording shape (`governed-by-rebind` / `provably-differs`) under the `Outcome recording conventions` block, host-prerequisites clause (a) records the contradiction narrative, and `SM-3b`/`SM-4`/`SM-5`/`SM-6` and the `session-only-degraded-state.md` page each carry explicit implementation-gate markers that block on item (a) resolving to `provably-differs`.

What is missing is the **loom-1.0 release disposition for the unresolved-at-ship case**. The spec states what each *verdict* implies (`governed-by-rebind` → SM-4/5/6 reworked; `provably-differs` → degraded-state branch implementable as written), but it never states what loom 1.0 does if **neither** verdict has been authoritatively recorded by the release gate. Two readings are equally consistent with the current text: (1) recording a verdict is a loom-1.0 release-blocking gate — 1.0 cannot ship with item (a) open; or (2) loom 1.0 may ship with the degraded-state branch descoped to the safe full-teardown behaviour and SM-4/5/6 deferred, with the contradiction left open for a later minor. The spec corpus itself encodes no choice between these; only the plan's `§Blocked obligations` pre-designed fallback (V9l collapsing to full teardown + `/reload`) silently picks reading (2), leaving the normative spec and the plan out of step on a scope question.

The original heading's "recorded as normative with no verdict" framing overstates the defect: the open question is deliberately tracked with a defined recording contract, so the *absence of a verdict* is by-design (the verdict is recorded by the contributor at the resolving commit). The durable, real gap is the missing release disposition.

## Solution Space

**Shape:** single

### Recommendation

State the loom-1.0 release disposition for an unresolved item (a) at the version-bump-step2.md item (a) site (and mirror it at host-prerequisites clause (a)). Adopt the disposition the plan has already pre-designed in `plan.md §Blocked obligations`, so spec and plan agree: **loom 1.0 MAY ship with item (a) unresolved; in that case the degraded-state branch is descoped to safe full teardown + `/reload` (no degraded tag write), and SM-3b/SM-4/SM-5/SM-6 are deferred / reworked spec-side before any later minor enables the branch.** Add an explicit sentence to item (a) and to clause (a) stating this — i.e. that a recorded verdict is *not* a loom-1.0 release-blocking gate, that the unresolved state at ship resolves to the descoped (full-teardown) behaviour, and forward-link the SM-3b/4/5/6 gate markers to that disposition.

Edge cases for the implementer: keep the disposition statement consistent across all four gate-marker sites (SM-3b, SM-4, SM-5, SM-6) and the `session-only-degraded-state.md` gate blockquote so none implies a release-blocking reading; do not add a verdict to item (a) itself (the open question stays genuinely open — only its release disposition is being pinned).

## Consequence

**Severity:** correctness

Reading the spec alone, two reasonable implementers diverge on whether loom 1.0 is release-blocked until item (a) is authoritatively resolved or may ship with the degraded-state branch descoped. The degraded-state feature cluster (SM-4/5/6) thereby has an undefined ship status: one team blocks the release on a recorded verdict, another ships 1.0 with the branch deferred. The plan has silently chosen the latter, so the normative spec and the plan disagree on a scope/release question.
