## Finding

The closing sentence of the **Dispatch-site setup wrap** bullet in `active-invocation-registry.md` pins a specific control-flow shape for handling a throw from the dispatch-site `catch` arm's cleanup `loomAbort.abort()` call: "the `catch` arm wraps that cleanup `loomAbort.abort()` in its own nested `try`/`catch`, drops any throw from it, and then routes the originally captured setup throw to the runtime-defect surface (…) unchanged." The only externally observable property the rest of the contract actually depends on is that the cleanup throw must not mask the original setup throw being routed to `loom/runtime/internal-error` / `InvokeInfraError { cause: "internal_error" }`; the `try`/`catch`-and-drop shape is one valid implementation of that obligation, not the obligation itself.

Naming the mechanism this narrowly does not enable any additional test (a conformance test against `loom/runtime/internal-error` carrying the original `error.message` / `error.stack` already pins the observable). It does close off equally correct shapes — for example, routing the originally captured throw before invoking `loomAbort.abort()` so that any cleanup-side throw is raised after the diagnostic surface has already been emitted — and shifts the bullet's register from "what the surface must guarantee" to "what the source must literally look like". The rest of this bullet is unavoidably mechanism-shaped (it must name `AbortController`, `Set.add`, `Promise.withResolvers()`, the listener-attach calls, etc. because each of those is observable elsewhere in the contract). The cleanup-masking clause is the one place where the only externally observable consequence is the behavioural MUST.

## Solution Space

**Shape:** single

### Recommendation

Replace the closing sentence of the **Dispatch-site setup wrap** bullet — currently "the `catch` arm wraps that cleanup `loomAbort.abort()` in its own nested `try`/`catch`, drops any throw from it, and then routes the originally captured setup throw to the runtime-defect surface (`loom/runtime/internal-error` / `InvokeInfraError { cause: "internal_error" }`) unchanged" — with a single behavioural MUST that pins only the externally observable property:

> A throw from the `catch` arm's own cleanup `loomAbort.abort()` call, raised while tearing down a half-constructed entry, MUST NOT mask the originally captured setup throw: the originally captured setup throw MUST be the one routed to the runtime-defect surface (`loom/runtime/internal-error` / `InvokeInfraError { cause: "internal_error" }`), with its `error.message` / `error.stack` carried unchanged.

Implementer-facing notes the fixer must preserve when applying the edit:

- Keep the cross-reference to *Runtime panics* — that link is what wires "runtime-defect surface" to the concrete diagnostic code and `InvokeInfraError` variant.
- Do **not** delete or weaken the obligation itself — the masking risk is real (the F-1118 fix predicate that prompted today's wording remains correct); only the *how* (nested `try`/`catch`, drop throw) is being lifted out so the rule reads as a behavioural MUST.
- The fix is purely editorial: no other clause in the bullet — `loomAbort` / `disposeBarrier` setup ordering, `Set.add` placement, listener-attach steps, the routing target — changes. No anchors are added or removed.

## Consequence

**Severity:** cosmetic

The mechanism the bullet prescribes is the canonical idiom for the behaviour it requires, so two reasonable implementers will land on the same code regardless of the rewording. The defect is one of register, not divergence risk: the spec carries one extra control-flow sentence where a behavioural MUST suffices, slightly enlarging the surface that any "spec-prescription" lens or audit must triage.
