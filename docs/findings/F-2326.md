## Problem

`V6c` owns two load-time obligations that both require the set of valid Pi tool names and a way to materialise each one's `ToolDefinition`: the `loom/load/unknown-tool` rejection and the frozen resolution snapshot's strong-reference capture. `V6c`'s only **Spec.** citation is `frontmatter-fields-a.md`, whose `tools` prose defers the resolution mechanism ("resolve against Pi's tool registry … exactly as for Pi subagents") without pointing anywhere concrete. The defining sources — the closed `ToolName` / `allToolNames` set and per-tool `create<Name>ToolDefinition` factory family in `pi-integration-contract/subagent.md` §`customTools`, and the captured-reference dispatch rule in `frontmatter-fields-b-and-templates.md` §Resolution snapshot — are cited by neither `V6c` nor `V6c-T`, and none of `V6c`'s declared deps (`V8a`–`V8e`, `V15a`, `V9f`) leads there. A fresh implementer therefore has no in-leaf pointer to where the Pi-tool universe or the captured `ToolDefinition` objects come from.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` ("pi-loom plan: build/update plan for spec.md + review")
**History:** `V6c` was created in `c6a664e` with **Spec.** citing only `frontmatter-fields-a.md`, while its **Adds.** already described "the frozen resolution snapshot" and Pi-tool entries — so the resolution-source gap was present from the leaf's first authoring. A later commit (`60c9400`) added the `loom/load/unknown-tool` test bullet, sharpening the obligation to determine valid Pi tool names, but did not add a citation naming where that universe or the `ToolDefinition` capture comes from. The gap predates and survived that edit rather than being introduced by it.

## Solution approach

Add **Spec.** citations to `docs/plan_topics/V6c-tools-set.md` pointing at `../spec_topics/pi-integration-contract/subagent.md` (§`customTools`) and `../spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` (§Resolution snapshot), and mirror the same additions into `docs/plan_topics/V6c-T-tools-set.md`. In `V6c`'s **Adds.**, name the Pi-tool universe as the closed `allToolNames` / `ToolName` set resolved through the `@earendil-works/pi-coding-agent` package-root surface (not the LLM `modelRegistry`), with each entry's `ToolDefinition` captured into the frozen snapshot via its `create<Name>ToolDefinition` factory and dispatch thereafter going through the captured reference.

## Solution constraints

- The spec is read-only for this fix; close the gap by adding plan-leaf **Spec.** citations and **Adds.** text, not by editing `frontmatter-fields-a.md` or any cited spec section.
