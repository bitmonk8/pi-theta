## Finding

The "Discovery roots" section of `discovery-sources.md` defines the union of *active discovery roots* for a Pi session, and that union is normatively load-bearing: the path-restriction rule in `invocation.md` requires a resolved `invoke` / `tools:` callee path (post-`realpath`) to lie within at least one active root, otherwise `loom/load/invoke-path-escape` fires. Each source's root contribution is spelled out — the global/project roots contribute themselves, settings `loomPaths` entries resolve by an explicit file-vs-directory rule ("a directory entry contributes its own path; a file entry contributes its parent directory"), and `--loom` CLI components reuse that same file-vs-directory rule.

The package source is the exception. Its bullet contributes "the package's `looms/` directory, or each directory reached through a `pi.looms` glob" — wording that only accounts for *directory* matches. But `pi.looms` (DISC-5 in `package-and-settings.md`) explicitly supports glob patterns that match an individual `.loom` file directly: "a match that is a `.loom` file registers that file directly; a match that is a directory is scanned non-recursively". When a `pi.looms` glob resolves to a `.loom` file, the spec registers the loom but never says what active root that match contributes — whether the file's parent directory becomes an active root (mirroring the settings/CLI file rule) or whether no root is contributed at all.

The gap is observable: a loom registered via a direct `pi.looms` file match would, under the "no root" reading, be unable to `invoke` a sibling `.loom` under its own directory (the callee path escapes every active root), whereas under the "parent directory" reading the sibling resolves normally. Two implementers reading the same text will diverge on which behaviour is correct.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 833d8d5 — spec: resolve five invoke edge cases (Decision #4) (2026-05-05, author per git history)
**History:** The `pi.looms` capability to match an individual `.loom` file (vs. a directory) already existed by d6cbb37 (2026-05-04). The "Discovery roots" definition and the path-restriction rule that consumes it were both authored the next day in 833d8d5, and that bullet described the package contribution only as "each directory reached through a `pi.looms` glob" — omitting the already-existing file-match case even while the sibling settings/CLI bullets in the same list spelled out an explicit file-vs-directory rule. The defect has been present in the discovery-roots definition since its first commit; later commits (31ff060 doc move, f5e89f4 size-cap split into `discovery-sources.md`) only relocated the unchanged text.

## Solution Space

**Shape:** single

### Recommendation

In the "Discovery roots" package bullet of `discovery-sources.md`, state the file-vs-directory contribution rule for `pi.looms` matches symmetrically with the settings/CLI rule: a `pi.looms` match that is a directory contributes that directory as an active root; a `pi.looms` match that is a `.loom` file contributes its parent directory. Rephrase the bullet to cover both arms, e.g. "Each scanned package's contributing root(s): the package's `looms/` fallback directory, or — for each `pi.looms` match — the matched directory itself, or the parent directory of a directly-matched `.loom` file." Keep the file→parent-directory direction consistent with the settings and `--loom` rules so the active-root semantics are uniform across all five sources.

## Consequence

**Severity:** correctness

A `pi.looms` glob that matches an individual `.loom` file leaves the loom's active-root contribution undefined. Two implementers will diverge on whether that loom may `invoke`/`tools:`-reference siblings under its own directory — one admits the callee, the other rejects it with `loom/load/invoke-path-escape` — producing incompatible path-restriction behaviour for the same package layout.
