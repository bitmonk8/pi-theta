## Problem

`docs/spec_topics/functions.md` paragraph 2 asserts that a function whose body uses `?` "must declare" a `Result<_, QueryError>` return type — read naturally, that the annotation is required at the source. The same file's **Loom return type** rule (`#loom-return-type`) states the opposite: an annotation-less `fn` body *infers* its return type, wrapping in `Result<T, QueryError>` whenever a `?` appears. The `loom/parse/question-outside-result-fn` diagnostic fires on the inferred-or-declared return type, not on a missing annotation, so the "must declare" sentence is backed by no construct and is contradicted by the surrounding rules. An implementer reading top-down reaches opposite conclusions about whether `fn f() { @\`…\`? }` parses, a divergence that survives the rest of the spec.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `53f5831` ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03), `8eea523` ("spec: fill implementation-blocking gaps", 2026-05-04), `08886d9` ("pi-loom spec: resolve 'loom return-type inference does not reconcile early-return operand types'", 2026-06-07).
**History:** `git log -S 'must declare a \`Result' -- spec.md docs/spec_topics/functions.md` shows the "must declare" sentence first lands in `53f5831`, when `?` was the only path to a `Result` return and the spec carried no return-type inference; at that point the sentence was an accurate operational rule. The Loom-return-type inference rule arrives one day later in `8eea523` (`git log -S 'Loom return type'`) as a single sentence ("wrapped in `Result<T, QueryError>` if any `?` appears in the body") — this is the commit at which the two paragraphs first describe the same situation differently, and is the moment the defect actually exists. `08886d9` then hardens the inference rule, adding the explicit clause "An annotation-less `fn` body infers its return type by this same rule" — which sharpens the contradiction but does not introduce it. No single commit owns the defect; it is the interaction of a sentence inherited from the pre-inference spec with the inference rule layered on top of it.

## Solution approach

Rewrite the offending sentence in `docs/spec_topics/functions.md` paragraph 2 so it describes the consequence — the scope returns `Result<_, QueryError>` whether by explicit annotation or by inference — rather than mandating a declaration. Route the reader to the canonical rules at `#loom-return-type` and to the `?`-operator return-type precondition owned by `errors-and-results/error-model.md`. Keep the `Result<_, QueryError>` spelling.

## Solution constraints

- Out of scope: do not resolve or alter the `_`-in-type-argument-position question — keep the `Result<_, QueryError>` spelling unchanged (owned by the same-cluster `_`-wildcard finding).
- Do not introduce a new "missing return annotation" diagnostic; `loom/parse/question-outside-result-fn` already covers the only failure mode here on the inferred-or-annotated return type.
