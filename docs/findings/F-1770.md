## Finding

The `<a id="resolution-snapshot"></a>` paragraph in `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` (line 22) says, twice, that an entry in the per-loom callable table "holds a **strong reference** to the resolved `ToolDefinition` object" / "holds a **strong reference** to the parsed callee plus its lowered tool spec." "Strong reference" is a runtime-GC term of art (the inverse of `WeakRef`/`WeakMap`) and pins a specific memory-management mechanism that is not, on its own, observable: the runtime could as well hold the callable in a closure, an array, a frozen object, or any other container that keeps it alive — what the rest of the paragraph cares about is that *every subsequent call dispatches through that captured value and the runtime never re-queries Pi's tool registry by name during execution*, which is the actual conformance contract.

That observable rule is already pinned by the very next clause ("Subsequent calls … dispatch through the held reference; the runtime never re-queries Pi's tool registry by name during execution") and by every "Consequences" bullet beneath it, which talk about "captured `execute`," "captured callee parse," and "the held reference still has a contracted outcome." The "strong reference" phrasing adds nothing those clauses do not already say, and it imports a JavaScript-engine implementation vocabulary into a paragraph that otherwise reads purely at the behaviour level.

The same wording leaks one shard away in `docs/spec_topics/tool-calls.md` (line 25), which justifies the unreachability of `unknown_tool` mid-invocation by saying "the load-time `tools:` table holds **strong references** to resolved callables." Both occurrences were introduced together in commit `7ce0ffa` (the original Resolution-snapshot block) and need to move together so the spec speaks with one voice.

## Issue introduction

**Verdict:** single-commit

**Introducing commits:** `7ce0ffa` ("pi-loom spec: resolve \"Tool registry mutation during execution and model-driven parallel tool calls\"", 2026-05-04)

**History:** `git log --all -S 'strong reference to the resolved'` shows the phrase entering the corpus exactly once, in `7ce0ffa`, when the **Resolution snapshot** block was first added to the then-monolithic `spec_topics/frontmatter.md`. The same commit also added the "holds strong references to resolved callables" parenthetical to `spec_topics/tool-calls.md`. Subsequent commits — `f5e89f4` (2026-06-04, the 100KB-cap shard split that moved the paragraph into `frontmatter/frontmatter-fields-b-and-templates.md`) and `e279e0c` (2026-06-06, which only added the `<a id="resolution-snapshot"></a>` anchor) — preserved the wording verbatim. The defect is the original phrasing choice in `7ce0ffa`; no later commit re-introduced or reinforced it.

## Solution Space

**Shape:** single

### Recommendation

Reword the two `holds a strong reference` sentences in the Resolution-snapshot paragraph to speak in capture/dispatch terms only, and apply the matching edit to the `tool-calls.md` parenthetical so the two shards stay in sync.

Concretely, in `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` line 22, replace:

> Each Pi-tool entry holds a strong reference to the resolved `ToolDefinition` object (its `execute`, `parameters`, and metadata as returned by Pi's tool registry at the moment of load); each `.loom`-path entry holds a strong reference to the parsed callee plus its lowered tool spec …

with wording along the lines of:

> Each Pi-tool entry captures (and retains, for the lifetime of the loom) the resolved `ToolDefinition` object (its `execute`, `parameters`, and metadata as returned by Pi's tool registry at the moment of load); each `.loom`-path entry captures the parsed callee plus its lowered tool spec …

Leave the following clause ("Subsequent calls … dispatch through the held reference; the runtime never re-queries Pi's tool registry by name during execution") untouched — it is the observable contract — but change `held reference` to `captured callable` so the paragraph stops appealing to "reference" vocabulary entirely.

In `docs/spec_topics/tool-calls.md` line 25, replace `the load-time tools: table holds strong references to resolved callables` with `the load-time tools: table captures (retains) the resolved callables`, preserving the existing cross-link to `[Parameters and Frontmatter](./frontmatter.md)`.

Edge cases the implementer must watch:

- Do **not** drop the "for the lifetime of the loom" / "until it terminates" lifetime claim — the first Consequences bullet ("Unregistering a Pi tool from Pi's registry mid-run does not affect calls from a loom that already resolved it; the loom continues to use the captured `execute` function until it terminates") depends on the snapshot outliving any mid-run registry mutation, and that lifetime is the *observable* part of what "strong reference" was gesturing at.
- Keep the four Consequences bullets that already use "captured `execute`," "captured callee parse," and "the held reference" coherent: if you rename `held reference` to `captured callable` in the lead paragraph, sweep the bullets for the same noun phrase so the page reads as one voice.
- The `<a id="resolution-snapshot"></a>` anchor MUST stay on the same paragraph — other shards link into it (e.g. the Hot-reloading bullet says "per the **Resolution snapshot** rule above").

## Consequence

**Severity:** advisory

Two reasonable implementations cannot diverge here — any container that keeps the callable alive and is dispatched through satisfies both phrasings — so nothing breaks. The cost is editorial: GC vocabulary in a behaviour-only paragraph confuses readers about whether the runtime is required to reach for a particular reference-management primitive, and undermines the page's own observable-only stance that the surrounding "Consequences" bullets carefully maintain.
