## Finding

PIC-17 step 2 mandates that the runtime call `pi.setActiveTools([...loomCallableSetNames, respondToolName?])` and the *Acceptance criteria — PIC-17 active-set install vector* paragraph asserts that fixtures verify the step-2 install vector "exactly". `loomCallableSetNames` is, however, never defined positionally: nothing at the PIC-17 site (or at any of its inbound references) tells the implementer whether it iterates frontmatter `tools:` declaration order, post-rename `Resolution snapshot` table order, lexicographic order, or some other order.

For the empty-callable-set case the question is moot, so both currently-enumerated fixtures — (a) empty set + untyped turn, (b) forced-respond turn whose worked install vector is `[respondToolName]` — slide past it. The first fixture that exercises a non-empty callable set on a free turn (the nominal production path) will need to spell the vector out, and two conformant implementations will be entitled to install different vectors while both passing a "set-equality" reading of "exactly" and both failing a "sequence-equality" reading.

`Resolution snapshot` already pins a per-loom iteration order at the frontmatter site (a frozen table of `{ post-rename name → resolved callable }` derived from the source declaration), so the fix is a one-clause cross-reference rather than a new invariant — but until it lands, the "exactly" in step 2 is not a normative obligation an implementer can act on.

**History:** Commit `27ec53d` authored the PIC-17 protocol in a single hunk, introducing the `[...loomCallableSetNames, respondToolName?]` install vector with no ordering clause; a later commit (`3536c67`, 2026-05-21) tightened the union-vs-replace semantics of that line but did not pin order. Commit `8008b91` introduced the *Acceptance criteria — PIC-17 active-set install vector* paragraph and its two fixtures — promoting "exactly" to a fixture-level obligation while leaving the order gap intact (both enumerated fixtures use empty callable sets, so neither requires the order to be known to pass). The defect has therefore been present since `27ec53d` and became fixture-visible at `8008b91`.

## Solution Space

**Shape:** single

### Recommendation

At the end of PIC-17 step 2, add a clause pinning the order of `loomCallableSetNames` to the loom's [Resolution snapshot](../frontmatter/frontmatter-fields-b-and-templates.md#resolution-snapshot) iteration order — i.e. the order of `{ post-rename name → resolved callable }` entries in the frozen per-loom table, which is itself the loom's frontmatter `tools:` declaration order after each entry's default name rewrite (`as`-rename or hyphen→underscore) is applied. Suggested replacement for the parenthetical phrase in step 2: "where `loomCallableSetNames` is the loom's post-rename callable-set names in [Resolution snapshot](../frontmatter/frontmatter-fields-b-and-templates.md#resolution-snapshot) iteration order, and `respondToolName` (when present) is appended last".

Mirror the pin in the *Acceptance criteria* paragraph by adding a single normative sentence: "Fixtures MUST assert the step-2 install vector as an ordered sequence, not a set." No change to the existing (a) / (b) fixtures is needed (empty-set ordering is vacuous); the related finding *PIC-17 acceptance criteria miss the non-empty-callable-set untyped-turn fixture* adds the first fixture under which the ordering pin becomes observable.

Edge cases the implementer must watch: (i) the `as`-rename applies before the order is captured — the snapshot iterates *post-rename* names, not source spellings; (ii) the rename's default hyphen→underscore rewrite for `.loom`-path entries also applies before order capture; (iii) `respondToolName` is always appended last, never interleaved with `loomCallableSetNames`; (iv) the order is captured once at loom-load time and is stable across every query of that loom invocation — it does not re-derive from the snapshot per-call.

## Consequence

**Severity:** correctness

Two reasonable implementations are entitled to install `[a, b]` vs `[b, a]` (or any other permutation) for the same loom and both claim conformance with the "exactly" wording; the fixture suite cannot catch the divergence as long as every enumerated fixture uses an empty callable set. The model's observed `tool_use` ordering preferences in some providers are sensitive to active-set order, so two "exact"-conformant runtimes can produce observably different conversation traces on identical input.
