## Finding

The transport-failure row renders `(<provider>: <message>)` but only `<provider>` is pinned. `<message>` has no pinned source; the sole `message` derivation in `queryerror-variants.md` is the prompt-mode path, which does not apply to the binder's off-session `complete()` call, so the binder-path source is unstated and implementations diverge on the rendered text.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 83802db — spec: collapse to single typed argument style; add LLM-driven slash-command binder (2026-05-04, Thomas Andersen); be82734 — pi-loom spec: resolve "Ceiling #3 (binder retry budget): provider-failure classification rule undefined" (2026-05-07, Thomas Andersen); f41af5d — pi-loom spec-plan: fix F-1131 — add the custom-type-unsafe Failure-mode-templates row (2026-06-24, Thomas Andersen)
**History:** The failure-mode-templates table was introduced with the LLM-driven binder at 83802db, where the transport row's `<message>` placeholder was already present and unpinned. be82734 added the rule routing structural `ContextOverflowError` into that same transport row without defining how an error variant lacking a `provider` field populates `<provider>`/`<message>`. f41af5d later added the custom-type-unsafe row carrying `<value>` without pinning its source or escaping. (A subsequent clarity pass, 2fb5783, pinned `<provider>` and `<ajv-summary>` explicitly but left `<message>` and `<value>` unpinned.)

## Solution Space

**Shape:** single

### Recommendation

In the Failure-mode templates prose, pin the transport row's `<message>` to `TransportError.message`, rendered through System-note-rendering rule 1, and state the off-session `message` source for binder transport failures — either by extending the `queryerror-variants.md` `message` derivation to cover the binder's `complete()`-sourced failures, or by cross-referencing the derivation and confirming which arm applies off-session.

## Consequence

**Severity:** correctness

The rendered failure notes are a declared spec-versioned breaking-change surface, yet three placeholders have no pinned source or escaping. Two reasonable implementers diverge on the transport row's `<message>` content, on what `<provider>`/`<message>` resolve to when a structural `ContextOverflowError` is folded into the transport row (the variant has no `provider` field), and on the `<value>` content/escaping for a `customType` containing `'`/`]`/`: ` — and none of these rows is testable against a fixed byte oracle.
