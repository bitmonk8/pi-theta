## Problem

Seven sites across `docs/spec_topics/` spell the `?` operator's operand type and the `?`-using scope's return type as the literal `Result<_, QueryError>`, using `_` as an informal "any T" wildcard. The Type grammar admits no such form: `NamedType ::= Ident` excludes `_`, `GenericType` is a closed two-constructor set, and `lexical.md` reserves `_` exclusively as the discard binding — so the wildcard never reaches type position. The two `code-registry-parse.md` rows are the sharpest case: under DIAG-4 the *Message* column is byte-normative, so a conforming runtime would be obligated to emit a diagnostic whose own message text contains a token sequence the parser rejects. An implementer cannot tell whether `_` is wildcard type syntax (grammar incomplete) or informal shorthand (prose wrong).

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `53f5831` (2026-05-03, *spec: rewrite query primitive, system prompts, and schema syntax*)
**History:** `git log --reverse --all -G 'Result<\s*_\s*,\s*QueryError\s*>'` shows the literal `Result<_, QueryError>` was introduced into the then-monolithic `spec.md` by `53f5831` as part of the original `Result` / `QueryError` / `?`-operator design — both the `?`-operator gate paragraph and the function-return paragraph carried `_`-in-type-position from that first day. `8eea523` (2026-05-04, *spec: fill implementation-blocking gaps*) added a third occurrence in the same file. `fecb504` (2026-05-04, *Split spec.md and plan.md into per-topic / per-phase files*) moved the offending sentences into `spec_topics/functions.md` and `spec_topics/errors-and-results.md` without altering the wildcard. The grammar productions that make the violation textually visible were tightened later — `1c552ff` (2026-06-04, *resolve "Type grammar admits no generic application beyond array<T>"*) added the `GenericType ::= "Result" "<" Type "," Type ">"` line — but the May commits already excluded `_` from the `Type` production via the absence of `_` from `NamedType ::= Ident` and the lexical reservation of `_`. No commit ever introduced `_` as a type-position metavariable convention.

## Solution approach

The defect is the wording, so carry the value: rewrite every `Result<_, QueryError>` occurrence across the seven sites to `Result<T, QueryError>` with an explicit "for some `T`" binder, matching the success-payload binder convention already in force in `errors-and-results/error-model.md` and `tool-calls.md`. At the `?`-operator gate keep the success-payload binder and the enclosing-scope binder distinct. Rewrite the byte-normative *Message* column of the `loom/parse/question-outside-result-fn` row to `'?' used in a scope whose return type is not Result<T, QueryError>`.

## Solution constraints

- Do not introduce a type-position wildcard or metavariable meaning for `_`; it remains the discard binding only per the `lexical.md` reservation.
- Out of scope: the must-declare-vs-inference contradiction on the same `functions.md` paragraph, owned by the same-cluster sibling finding.
