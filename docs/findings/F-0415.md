## Finding

`V18c`'s **Ships when** field defines the editorial-review part of the completion gate by re-stating the normative levels and merge semantics that `version-bump-step2.md` already owns: clause (2) asserts "no MUST-level item `(a)`–`(e)` carries a `fail` outcome … : a `fail` on any MUST-level item `(a)`–`(e)` blocks the bump, whereas a recorded `fail` / `not-performed` on a SHOULD-level item `(f)`–`(ak)` is a conformant non-blocking outcome, per `version-bump-step2.md`."

The MUST/SHOULD partition over items `(a)`–`(ak)` and the per-item blocking-vs-conformant outcome rules are authoritatively defined in the spec — `version-bump-step2.md`'s step-2 preamble fixes the MUST-level audit of `(a)`–`(e)`, the SHOULD-level audit of the remaining items, and the mandatory per-item recording, and each `(f)`–`(ak)` item additionally carries its own escalation clause that promotes it to a build-time pin once Pi exposes a typed contract. Copying those levels and the blocking semantics into a leaf completion-criterion field creates a second normative surface that must be kept in lockstep by hand. The Ships-when copy and the spec are consistent today, so the leaf is buildable as written; the defect is the duplicated normative surface, not a present contradiction.

The completion criterion still needs to reference the merge-blocking distinction — clause (2) was added precisely to make the exit gate name it — but it can do so by deferring to `version-bump-step2.md` as the authority rather than restating which items are MUST vs SHOULD and which outcomes block the bump.

The defect was therefore introduced by that specific remediation edit, not present at the leaf's inception.

## Solution Space

**Shape:** single

### Recommendation

In `V18c`'s **Ships when** field, keep the observable build-time gate list unchanged (`npm run typecheck` brand-string assertion and the `npm test` surface-inventory / `engines.node` / peer-dep / snapshot / provider-seed-field / strict-capability gates), and keep the editorial-review acceptance referencing `version-bump-step2.md`. Strike the inline restatement of the normative partition and blocking semantics — the text "no MUST-level item `(a)`–`(e)` carries a `fail` outcome … a `fail` on any MUST-level item `(a)`–`(e)` blocks the bump, whereas a recorded `fail` / `not-performed` on a SHOULD-level item `(f)`–`(ak)` is a conformant non-blocking outcome" — and instead have the editorial-acceptance clause confirm that the bump meets `version-bump-step2.md`'s acceptance criteria, citing that page as the authority for which recorded outcomes block the bump.

Because clause (2) was added to make the exit gate name the merge-blocking criterion, the deferral must still point the gate at that criterion — confirming the bump carries no outcome that `version-bump-step2.md` defines as merge-blocking — rather than dropping the blocking distinction entirely; only the per-item MUST/SHOULD enumeration and the verbatim block/non-block wording move out, sourced from the spec by reference.

## Consequence

**Severity:** advisory

If `version-bump-step2.md` later changes the normative surface — e.g. an `(f)`–`(ak)` item escalates from SHOULD to a MUST-level build-time pin (each item carries that escalation clause), or the merge-blocking rule is reworded — `V18c`'s Ships-when copy silently goes stale and the leaf's stated completion gate would enforce the old MUST/SHOULD partition. The leaf still builds today because the copy currently matches the spec; the risk is a future spec edit that updates one surface and not the other.
