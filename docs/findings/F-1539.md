## Finding

`runtime-event-channel.md` states the `convertToLlm` context-entry fact twice. The canonical statement lives in the *Custom-message channel persistence and LLM-context entry* paragraph (anchor `#custom-message-context-entry-presupposition`): every `CustomMessage` (including `loom-system-note`) is converted to a `{ role: "user", content }` transcript entry on every subsequent provider call, `triggerTurn: false` suppresses only the immediate turn fire, `display` controls renderer behaviour not serialization, and there is no `excludeFromContext` opt-out at the loom 1.0 pin. That paragraph also draws the operator-facing implication explicitly ("Operators authoring looms should expect parse errors, binder failures, panic notes, and always-log runtime events emitted in a session to be visible to subsequent model turns").

The **Runtime event channel** paragraph then restates the same fact in a parenthetical — "(Pi's `convertToLlm` transform converts every `CustomMessage` to a `user`-role transcript entry on every subsequent provider call, with no opt-out under the loom 1.0 SDK pin)" — and appends a redundant **Note (non-normative)** re-deriving the same operator-visibility implication. The sentence already cross-references the upstream paragraph ("still enters subsequent provider calls per **Delivery surface** above"), so the parenthetical and the note add no new constraint over the cross-reference; they merely repeat the canonical statement at a second site.

The load-bearing content of the sentence is the `display: false` gating rule (`display: false` gates only renderer visibility while the message still enters provider calls). That rule is worth keeping at this site; the convertToLlm restatement and the trailing non-normative note are not.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 830341b ("pi-loom spec: resolve \"Operator-bound MUSTs mis-classified as runtime conformance requirements\"", 2026-06-04)
**History:** The corpus is git-tracked. The canonical *Custom-message channel persistence and LLM-context entry* paragraph (the authoritative `convertToLlm` statement) already existed in `binder-inference.md` before `830341b`. `830341b` added the `display: false` gating sentence whose parenthetical re-states the convertToLlm fact, together with the trailing `**Note (non-normative):**`, into the same file — creating the duplication against the already-present canonical paragraph. Commit `78a2f94` ("Misplaced sections within the pi-integration-contract pages", 2026-06-04) later relocated both the canonical paragraph and the duplicating sentence+note intact from `binder-inference.md` into the current `runtime-event-channel.md`, carrying the pre-existing duplication along rather than introducing it. No subsequent commit reconciled the two copies.

## Solution Space

**Shape:** single

### Recommendation

In the **Runtime event channel** paragraph, reduce the sentence to the `display: false` gating rule plus its cross-reference, deleting the parenthetical convertToLlm restatement and the trailing `**Note (non-normative):**`. Resulting text: "`display: false` gates only renderer visibility; the underlying `CustomMessage` still enters subsequent provider calls per the Custom-message channel persistence and LLM-context entry paragraph above."

Point the cross-reference at the *Custom-message channel persistence and LLM-context entry* paragraph (anchor `#custom-message-context-entry-presupposition`), which is where the convertToLlm fact and its no-opt-out qualification actually live — not the *Delivery surface* paragraph, which describes the single-`pi.sendMessage` delivery shape and does not state the convertToLlm transform. The operator-visibility implication is already carried by the canonical paragraph's "Operators authoring looms should expect … to be visible to subsequent model turns" sentence, so dropping the non-normative note loses no information.

## Consequence

**Severity:** cosmetic

The two copies are currently consistent, so no implementer diverges and no behaviour changes. The cost is reading burden at a dense normative site plus a drift hazard: a future edit to one copy (e.g. when the opt-out field lands in a later Pi minor) can leave the other stale, at which point a reader could trust the wrong statement.
