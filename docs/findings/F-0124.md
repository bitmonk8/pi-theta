## Problem

The `?`-operator Tests bullets in both `V4a-match-result.md` and `V4a-T-match-result.md` name two diagnostic codes as bare identifiers — `question-on-non-result` and `question-outside-result-fn` — while the canonical registry (`docs/spec_topics/diagnostics/code-registry-parse.md`) spells them `loom/parse/question-on-non-result` and `loom/parse/question-outside-result-fn`. The leaf's other code-keyed bullets already use full registry paths (`loom/parse/match-arm-type-mismatch`, `loom/runtime/match-error`), so these two are the lone bare-identifier outliers. The registry-code↔asserting-test parity arm (`conventions.md` §REQ-ID discipline) matches on the full `loom/parse/...` token, so a test asserting the bare string leaves both registry codes reading as un-asserted and reddens the closing gate. `question-outside-result-fn` has no numbered `PREFIX-N` REQ-ID; its registry code is itself the closing surface.

## Solution approach

In `V4a-match-result.md` and `V4a-T-match-result.md`, expand both bare identifiers to their canonical registry paths `loom/parse/question-on-non-result` and `loom/parse/question-outside-result-fn` everywhere they appear — the two `?`-operator Tests bullets in both leaves and the `question-on-non-result` reference in `V4a`'s Ships-when sentence. Keep the inline `ERR-18` anchor on its bullet.

## Solution constraints

- The spec is read-only for this fix; the canonical strings come from `docs/spec_topics/diagnostics/code-registry-parse.md` and are not edited here.
- No new coverage-matrix row is required and no numbered REQ-ID is to be invented for `question-outside-result-fn`.
