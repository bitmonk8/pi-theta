## Finding

`docs/spec_topics/pi-integration-contract/session-only-degraded-state.md` opens with an *Implementation gate* blockquote that scopes **by clause, not by file**. It declares exactly two clauses buildable now — the `## Drain-state tag write (state-independent)` paragraph (the `LoomRegistry.markRuntimeDegraded` setter contract) and the **Isolation** clause under `## Diagnostic emission` (the dedicated `try`/`catch` cross-reference target) — and gates everything else (the branch-behaviour clauses and acceptance-criteria fixtures (a)–(m)) pending the unresolved clause-(a) open contradiction recorded in `host-prerequisites.md`.

The two non-gated clauses are physically scattered through the page among the gated material rather than collected into a separable buildable-now region: the non-gated `## Drain-state tag write` section sits between the gated branch-behaviour prose and the gated recovery-path/predicate-split clauses, and the non-gated **Isolation** clause is embedded mid-way through the otherwise-gated `## Diagnostic emission` section. The page's actual physical layout therefore does not reflect the clause partition the gate asserts; the only thing separating buildable-now contracts from blocked content is an enumeration in the preamble prose. The sibling `V9m`/`V9g` slices that consume the non-gated contracts must reconstruct that partition by hand.

Compounding this, the gate itself is governance/release-status metadata (which clauses are blocked, on which open question, clearing on which verdict) carried inline at the top of a normative contract page. A reader must parse that status block — and continuously hold it in mind while reading — before they can tell which of the requirements they are reading actually apply to loom 1.0. The governance status belongs in a governance home with a forward-reference, not threaded through the contract surface.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `b8f004f` (2026-06-19, "fix F-0719 — SM-4/5/6/3b degraded-state implementation-gate contingency markers"); `8bcd52a` (2026-06-23, "fix F-1163 — session-only-degraded-state gate scopes by clause not page")
**History:** The corpus is git-tracked. `git log --follow` shows the page was first carved into its own topic file at `f5e89f4` (the spec-set size-cap split), with no gate preamble. `git log -S "Implementation gate"` localises the gate's arrival to `b8f004f`, whose diff adds the inline *Implementation gate* blockquote — but at that point the gate was **file-level** ("The obligations on this page are blocked"), so the "governance metadata inline" half of the defect dates here while no gated/non-gated interleaving yet existed. `git log -S "scopes by clause, not by file"` localises the second half to `8bcd52a`, whose diff (confirmed via `git show`) rewrites the blockquote to scope **by clause**, enumerating the two non-gated contracts in prose while leaving them physically interleaved among the gated clauses. The interleaving defect the finding describes is thus the product of the two commits interacting: `b8f004f` introduced the inline gate; `8bcd52a` split it by clause without a matching physical reorganization of the page.

## Solution Space

**Shape:** multiple

### Option A — Extract the gated clauses to a dedicated phase-2 / "blocked" file

- **Approach:** Move the gated branch-behaviour clauses and acceptance criteria (a)–(m) into a separate file (e.g. `session-only-degraded-state-branch.md`) carrying its own "Blocked — pending clause (a)" banner. Keep only the two buildable-now contracts (the `markRuntimeDegraded` setter contract and the **Isolation** clause) on `session-only-degraded-state.md`. Relocate the gate-status metadata to the governance/phase-status home and leave a one-line forward-reference from both files.
- **Spec edits:** Create the new file; move the gated sections verbatim with their anchors preserved; update inbound cross-references (including `V9l`/`V9l-T` **Spec** citations and any other pages linking the moved clauses); add the governance-page gate entry; replace the inline gate block with a forward-reference.
- **Pros:** Physical layout matches the clause partition exactly; over-building the gated branch becomes structurally hard; the buildable-now page is small and self-contained for `V9m`/`V9g`.
- **Cons:** This page is densely cross-referenced (anchors such as `#substep-2-stamp-throw-residual-gap`, the **Predicate split** and **Isolation** clauses referenced from sibling pages); moving content risks breaking the cross-reference web and forces a citation sweep across spec and plan.
- **Risks:** Anchor/link breakage if any inbound reference is missed; churn is wasted if clause (a) later resolves to *branch unreachable as written* and the gated content is reworked or descoped wholesale.

### Option B — Keep one page; hoist the buildable-now contracts and demarcate the blocked region in place

- **Approach:** Reorder the page so the two non-gated contracts sit together at the top (immediately after the title), then introduce a single explicit divider heading (e.g. `## Blocked — pending clause (a) resolution`) under which all gated clauses live unchanged. Move the gate-status metadata (which question, which verdict clears it) to the governance/phase-status home; replace the inline preamble with a short forward-reference plus the divider.
- **Spec edits:** Reorder the `## Drain-state tag write` section and the **Isolation** clause to the top; insert the divider heading before the first gated clause; preserve every existing anchor in place; add the governance-page gate entry; collapse the inline gate block to a forward-reference.
- **Pros:** Preserves all in-page anchors and the dense cross-reference web (no inbound-citation sweep); low churn; the physical divider gives the visual gated/non-gated boundary the prose-only partition lacks.
- **Cons:** Buildable-now and blocked content still share one file; a future editor can still append a clause on the wrong side of the divider (mitigated by the existing "later clauses default to gated" rule).
- **Risks:** Reordering the **Isolation** clause out of `## Diagnostic emission` must keep its `try`/`catch` cross-reference target intact, or the sibling pages pointing at it break.

### Recommendation

Take **Option B**. The by-clause gate is correct — the two non-gated contracts genuinely are buildable now — so the durable fix is to make the page's physical layout reflect that partition, not to revert the clause-level scoping. In-place hoisting plus a divider achieves the separation while preserving this page's dense anchor/cross-reference web, and avoids speculative file-restructuring churn while clause (a) is unresolved (a *branch unreachable as written* verdict would otherwise force re-moving the same content). In all cases, lift the release-status metadata to a governance home and leave only a forward-reference inline. Watch that the **Isolation** clause keeps its existing anchor when it moves, since sibling pages target it directly.

## Consequence

**Severity:** advisory

An implementer can still determine which clauses apply by reading the preamble enumeration, so a working system is reachable. But the mismatch between the prose partition and the physical layout makes over-building the gated branch easy (the buildable-now contracts give no visual signal where blocked content begins), and the inline governance status forces every reader of the contract to first parse release-management metadata.
