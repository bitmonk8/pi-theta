## Finding

`V7b` and `V7b-T` carry an identical `DIAG-4` Tests bullet: "the Message column is normative — every asserting test sources its expected string from the registry, not prose." Stated under `Tests.`, this reads as an automated acceptance criterion the closing gate enforces. It is not enforceable as written. *Provenance* — whether a test obtained its expected string by reading the registry's *Message* column versus copy-pasting identical prose from a spec page — is invisible to any mechanical reconciliation. A test that hardcodes a byte-identical string passes exactly as a test that reads the registry does; the gate cannot distinguish the two.

The spec's `DIAG-4` (`spec_topics/diagnostics/diagnostic-shape.md#diag-4`) correctly frames the provenance half as an authoring obligation: "Tests … MUST source the string from this column rather than copy-pasting prose." `conventions.md` *Diagnostic message anchors* echoes the same authoring MUST. What is mechanically checkable — and what `H5a`'s diagnostic arm actually reconciles — is the closed-set / asserting-test relationship (`DIAG-2`) and, at most, value-equality between a test's asserted string and the registry value. The plan bullet recasts the spec's authoring MUST as a gate-verified Tests assertion, claiming enforcement the gate does not provide.

`V7b-T`'s exit gate requires its Tests to "compile, and fail red for the intended reason." A provenance property has no observable assertion, so the "intended reason" for `DIAG-4` is undefined: an implementer cannot author a red-then-green test that observes "sourced from registry, not prose."

The provenance phrasing was carried verbatim from the spec's `DIAG-4` authoring MUST into the leaf `Tests.` field at inception, where it became a gate-assertion overclaim.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V7b-code-registry.md` and `docs/plan_topics/V7b-T-code-registry.md`, restate the `DIAG-4` Tests bullet so it asserts the mechanically observable property — value-equality between the test's asserted message and the registry — and relocate the provenance ("not prose") half to where it is already owned as an authoring obligation rather than a gate assertion.

Concretely, replace the bullet text "the Message column is normative — every asserting test sources its expected string from the registry, not prose" with a value-equality statement, e.g. "`DIAG-4`: the Message column is normative — each asserting test's expected message is byte-equal to the registry *Message* column value (placeholders interpolated)." Drop the unobservable "sources … from the registry, not prose" clause from the `Tests.` field; the authoring/self-review obligation it expressed is already carried normatively by `conventions.md` *Diagnostic message anchors* and spec `DIAG-4`, so no new normative surface is needed. Apply the same edit to both leaves so `V7b-T`'s failing test and `V7b`'s closing implementation stay 1:1.

Leave `V7c`'s separate `DIAG-4` bullet ("each of the eight categories renders its normative vector byte-identically") untouched — it already states an observable byte-identity property and is not the subject of this finding. Do not edit the spec `diagnostics/diagnostic-shape.md#diag-4` MUST or the `coverage-matrix.md` `DIAG-4 | V7b, V7c` row; the spec and matrix are read-only for this fix.

## Consequence

**Severity:** advisory

An implementer authoring the `DIAG-4` test has no observable to assert for "sourced from the registry, not prose," so `V7b-T`'s red-then-green ritual has no well-defined intended-reason and the bullet's mechanical promise cannot be met. In practice the implementer will write the value-equality test (the only observable form), leaving the "not prose" clause as misleading dead text that overstates what the closing gate verifies.
