## Finding

The **Baseline acquisition** paragraph in the version-bump step-2(a) checklist binds the contributor with a MUST to a specific command verb: "the contributor MUST materialise the loom 1.0 Pi-SDK pin's `dist/*.js` tree into a scratch directory by running `npm pack @earendil-works/pi-coding-agent@<pinned-version>` and extracting the resulting tarball." The genuinely binding contract here is behavioural — the diff's canonical "from" side must be the pinned minor's `dist/*.js` tree, materialised without pulling the lock-step siblings' transitive dependency tree, kept ephemeral, and never committed. The `npm pack … && extract` invocation is one recipe that satisfies that contract; an equivalent acquisition (e.g. fetching the registry tarball directly, or extracting an already-cached pinned tarball) would satisfy it equally. Elevating the command verb itself to a MUST conflates the recipe with the requirement.

This is inconsistent with how the same corpus handles parallel cases. The capability-probe steps demote concrete tool invocations under an explicit *"Recommended recipe (non-normative)"* heading (e.g. the `semver.satisfies(...)` recipe at Step 0(a)/0(d) and the `createRequire(...).resolve(...)` version-read mechanic), and step 5's session-only-reason sweep states the behavioural obligation (the line-level co-occurrence set) as the MUST while demoting the `rg -nP …` command to an explicit *"non-normative recipe"* that permits "a future swap to a different grep tool or a typed AST search." The baseline-acquisition paragraph should follow the same shape: state the materialisation outcome as the MUST and demote the `npm pack`/extract command (and its `npm install --prefix` avoidance rationale) to a non-normative recipe.

## Solution Space

**Shape:** single

### Recommendation

Restate the binding requirement behaviourally and demote the command to a non-normative recipe:

- Keep as the MUST: before running the `dist/*.js` diffs items (e), (r), (aj), (am) prescribe, the contributor MUST materialise the loom 1.0 Pi-SDK pin's `dist/*.js` tree into a scratch directory that serves as the canonical "from" side; the materialised tree MUST exclude the lock-step siblings' transitive dependency tree, MUST be ephemeral, MUST NOT be retained as an in-repo snapshot, and MUST NOT be committed. Preserve the existing ordering/specifier obligation (materialise before step 1's install, or pass the explicit `@<pinned-version>` specifier if ordered after) — that is a behavioural constraint, not a recipe.
- Demote to a non-normative recipe (mirroring the *"Recommended recipe (non-normative)"* blocks in `capability-probe.md` and the `rg` recipe in step 5): "`npm pack @earendil-works/pi-coding-agent@<pinned-version>` then extract the resulting tarball; the `package/dist/` tree is the from side. `npm pack` is preferred over `npm install --prefix` because it does not pull the siblings' transitive dependency tree. Any equivalent acquisition that yields the pinned `dist/*.js` tree without the transitive dependency closure satisfies the obligation."

The "no transitive dependency tree" property is part of the contract (it is what makes the choice of acquisition observable), so it stays on the MUST side; only the literal command and the `npm install --prefix` comparison move into the recipe.

## Consequence

**Severity:** advisory

A contributor reading the MUST may believe `npm pack` is the only conformant way to obtain the baseline and treat an equivalent acquisition (direct tarball fetch, cached extract) as non-compliant. No build or runtime behaviour breaks; the effect is an over-constrained, harder-to-evolve procedure inconsistent with the corpus's recipe/MUST separation elsewhere.
