## Problem

`V9j` builds the binder `complete()` inference call from the forced-tool envelope schema that `V11c` declares (and that `V11c`'s **Adds** names as consumed by the inference call), but neither `V9j` (`Deps. V9j-T, V4d, V11a`) nor `V9j-T` (`Deps. V4d, V11a`) lists `V11c`. Because build order is dep-driven (a leaf is pickable only when each `Deps.` entry's completion tag is present), nothing forces `V11c` to complete before `V9j`/`V9j-T`. If `V9j-T` is picked first, its `complete()`-envelope assertion references the not-yet-declared envelope-schema symbol and reds out as a compile-time error — a red the conventions reject, so the `V9j-T-complete` gate cannot fire honestly.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (`pi-loom plan: build/update plan for spec.md + review`)
**History:** `git log --follow` shows `V9j`, `V9j-T`, and `V11c` were all authored together in `c6a664e`; `V11c` was later edited by `cb8737a` (over-bundling resolution) without touching the dependency relationship. `git log -S 'V11c' -- V9j-binder-inference-provider-mapping.md` returns no commits — the `V11c` edge has never appeared in `V9j`'s `Deps.`. The missing dependency edge has been present since the leaf pair entered the corpus.

## Solution approach

Add `V11c` to the `Deps.` field of both `docs/plan_topics/V9j-binder-inference-provider-mapping.md` and `docs/plan_topics/V9j-T-binder-inference-provider-mapping.md`, so the inference call and its tests cannot be picked before the envelope schema they consume has landed. The edge on `V9j` also closes the transitive gap for `V11e` and `V11f`, which reach the envelope schema through their existing `V9j` dependency.

## Solution constraints

- Out of scope: `V13c` (its typed two-phase `complete()` uses the query-path forced-respond turn, not `V11c`'s binder envelope) and `V11e`/`V11f` (they reach the envelope transitively through their existing `V9j` edge) need no `V11c` edge.
