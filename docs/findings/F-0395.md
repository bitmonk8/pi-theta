## Problem

`V13g` and `V13g-T` carry an identical Tests bullet whose discard-observability clause reads "the runtime discard-observability event fires on an explicit `Err` discard (`let _ = @`...`` or the `void`-tail form)." The phrase fuses two distinct conditions: "explicit discard" is a syntactic property of the call site, while "`Err`" is a runtime property of the settled query result. One reading fires the event on every discard (including `Ok`); the other fires only when the result settles to `Err`. The owning spec page `query/query-escapes-stringification.md` §"Observability of discarded results" resolves it — the event fires exactly once per discarded `Err`, and `Ok` discards produce no event — so an implementer taking the syntactic reading would emit events on `Ok` discards in violation of the spec.

## Issue introduction

**Verdict:** introduced-by-prior-fix (single introducing commit, later carried forward by a split)
**Introducing commits:** 6698d91 — pi-loom plan: resolve "V13a Tests bullet conflates parse diagnostic with runtime discard event" (2026-06-16, Thomas Andersen); carried forward verbatim by 4a04570 — pi-loom plan: resolve "V13a bundles five independent query surfaces into one leaf" (2026-06-17, Thomas Andersen)
**History:** The discard concern originally lived in `V13a` / `V13a-T`. Commit 6698d91 — a prior plan-review fix that separated the parse-time `loom/parse/discarded-query-result` diagnostic from the runtime discard-observability event — introduced the exact clause "the runtime discard-observability event fires on an explicit `Err` discard (`let _ = @`...`` or the `void`-tail form)" into both `V13a` and `V13a-T`. That fix resolved the parse-vs-runtime conflation but coined the new syntactic-form-vs-runtime-type ambiguity. Commit 4a04570 then split `V13a` into multiple query leaves, removing the discard sentence from `V13a` and relocating the ambiguous clause verbatim into the new `V13g` / `V13g-T` files, where it stands today. The defect is exactly as old as commit 6698d91; the split neither introduced nor corrected it.

## Solution approach

Rewrite the discard-observability clause of the Tests bullet in both `V13g` and `V13g-T` so the event trigger is keyed on the settled-result type rather than the discard syntax: the event fires only when a discarded query (via `let _ = @`...`` or the `void`-tail form) settles to `Err`, and a discarded `Ok` via either form emits no event. Keep the `loom/parse/discarded-query-result` parse-error clause of the same bullet intact, and keep the implementation leaf and its test leaf in lockstep.

## Solution constraints

- The spec page `query/query-escapes-stringification.md` is read-only for this fix; tighten the plan wording to match it.
