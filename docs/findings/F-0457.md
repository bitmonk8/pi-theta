## Finding

`coverage-matrix.md` (§*Code-keyed obligation areas*, the `<new>`-placeholder-convention paragraph) and `H5e` (§Adds) both specify that the closing-gate recogniser "resolves each closing-leaf token against the **real plan leaf-ID universe**". This resolution is load-bearing: it is the three-way classifier that distinguishes (a) the literal bare `<new>` placeholder marker, (b) a token that resolves to an existing plan leaf and therefore satisfies the closing-leaf condition, and (c) any other token — a typo'd or non-existent leaf ID such as `V99z` — which must redden the gate as a defect rather than be read as a benign placeholder or as naming a real leaf.

Neither file defines **how that universe is enumerated**. The plan exposes several candidate enumeration sources that can drift relative to one another: the leaf link text/hrefs in `plan.md`, the `plan_topics/<id>-<short-name>.md` filenames, and the leaf-page H1 headings. A leaf file can exist without being linked into `plan.md`, or be linked before its file lands, so two implementers keying the recogniser off different sources will classify the same token differently — one reddens on a legitimate leaf, another stays green on a typo.

The membership boundary of the universe is also unstated in two specific places the finding calls out: whether `-T` tests-task IDs (e.g. `V9c-T`) are members, and whether editorial group tokens (e.g. `V9`, `V11`) are members. Closing-leaf cells only ever name implementation leaves, so a recogniser that admits `-T` IDs or bare group tokens into the universe would silently accept a malformed closing-leaf cell that should redden. Which tokens the gate reddens on therefore changes with the unstated membership decision.

## Issue introduction

**Verdict:** introduced-by-specific-edit
**Introducing commit:** `f3bb326` — "pi-loom plan: resolve \"<new> placeholder recognition rule pinned to a literal token\"" (2026-06-18)
**History:** `git log -S"real plan leaf-ID universe"` over `docs/plan_topics/` returns exactly one commit, `f3bb326`, which simultaneously introduced the term in both `coverage-matrix.md` (§Code-keyed obligation areas) and `H5e` (§Adds). That commit pinned the `<new>` placeholder to a literal token and added the typo'd-token (`V99z`) redden rule, but introduced the dependency on a "real plan leaf-ID universe" without ever defining its enumeration source. The defect is co-resident with the recogniser it serves — the recogniser concept and its undefined input entered the corpus in the same edit; there is no earlier well-defined state to revert to.

## Solution Space

**Shape:** single

### Recommendation

In `coverage-matrix.md` §*Code-keyed obligation areas*, in the paragraph that introduces the term "real plan leaf-ID universe", pin a single canonical enumeration source for the universe and state membership explicitly for the two boundary cases the finding names.

- **Pin the source.** Define the universe as the set of implementation-leaf IDs derivable from the `docs/plan_topics/<id>-<short-name>.md` leaf files (the `<id>` of each leaf file, excluding the paired `<id>-T-<short-name>.md` tests-task files). This source is always present, machine-readable, and does not drift between "the leaf exists" and "the leaf is linked into `plan.md`". Whichever source is pinned, it must be the single source both `H5d`'s tokeniser context and `H5e`'s recogniser read.
- **State `-T` membership.** State that `-T` tests-task IDs are **not** members of the universe — closing-leaf cells name only implementation leaves, so a `-T` token in a closing-leaf cell is a non-resolving token that reddens the gate.
- **State group-token membership.** State that editorial group tokens (`V9`, `V11`, `H5`, `M`, …) are **not** members — a bare group token in a closing-leaf cell is a non-resolving token that reddens the gate.

Carry a one-clause back-reference in `H5e` §Adds so its recogniser points at this pinned definition rather than restating it.

## Consequence

**Severity:** correctness

Two reasonable implementers will build the `H5e` recogniser against different enumeration sources (`plan.md` link text vs `plan_topics/` filenames vs leaf headings) and against different `-T`/group-token membership rules, producing recognisers that diverge: one reddens the gate on a legitimate closing leaf, another stays green on a typo'd or out-of-scope token. The gate that is supposed to catch an un-anchored MUST mapped to a non-existent leaf can therefore silently pass, or block a correct cell.
