## Problem

The `<new>`-placeholder paragraph in `coverage-matrix.md` §*Code-keyed obligation areas* and `H5e` §Adds both specify that the closing-gate recogniser resolves each closing-leaf token against the "real plan leaf-ID universe", but neither file defines how that universe is enumerated. The plan exposes several candidate sources that can drift relative to one another — leaf link text/hrefs in `plan.md`, the `plan_topics/<id>-<short-name>.md` filenames, and leaf-page H1 headings — so two implementers keying the recogniser off different sources will classify the same token differently. The membership boundary is also unstated in two specific places: whether `-T` tests-task IDs (e.g. `V9c-T`) are members, and whether editorial group tokens (e.g. `V9`, `V11`) are members. A recogniser that admits either would silently accept a malformed closing-leaf cell that should redden, so the un-anchored-MUST gate can pass on a token naming no real leaf or block a correct cell.

## Issue introduction

**Verdict:** introduced-by-specific-edit
**Introducing commit:** `f3bb326` — "pi-loom plan: resolve \"<new> placeholder recognition rule pinned to a literal token\"" (2026-06-18)
**History:** `git log -S"real plan leaf-ID universe"` over `docs/plan_topics/` returns exactly one commit, `f3bb326`, which simultaneously introduced the term in both `coverage-matrix.md` (§Code-keyed obligation areas) and `H5e` (§Adds). That commit pinned the `<new>` placeholder to a literal token and added the typo'd-token (`V99z`) redden rule, but introduced the dependency on a "real plan leaf-ID universe" without ever defining its enumeration source. The defect is co-resident with the recogniser it serves — the recogniser concept and its undefined input entered the corpus in the same edit; there is no earlier well-defined state to revert to.

## Solution approach

In the `<new>`-placeholder paragraph of `coverage-matrix.md` §*Code-keyed obligation areas*, pin a single canonical enumeration source for the leaf-ID universe so that `H5d`'s tokeniser context and `H5e`'s recogniser read the same source. State membership explicitly for the two boundary cases the finding names: whether `-T` tests-task IDs are members and whether editorial group tokens (`V9`, `V11`, `H5`, `M`, …) are members. Carry a one-clause back-reference from `H5e` §Adds pointing at the pinned definition rather than restating it.

## Solution constraints

- Pin only the enumeration source and the two membership boundaries; do not redefine the `<new>` placeholder semantics or the backtick-delimited tokeniser rule owned by *Closing-leaf column authoring* / `H5d`.
