## Finding

`conventions.md` §Loom 1.0 release-time residue inspection is the single manual-review backstop every cross-cutting rule routes its conceded mechanical-scan residue to. It enumerates nine items — closure/lazy-cache/DI-container singletons, indirect ambient-primitive reads, blocking-runtime calls, doc-update artifacts, un-anchored-MUST residue, broad-catch category membership, split-row joint-completeness, entry-shim purity, and ambient timing-primitive list reconciliation — and asserts that "a residue defect found at any item blocks the loom 1.0 release until fixed."

The inspection is described only as "a reviewer reading `src/**`" running "at the loom 1.0 release gate." It names no owning leaf, no executor, and no wiring into a leaf's exit gate. The terminal release-gate leaf `H6a` — which is the loom 1.0 release gate the inspection says it runs at — carries no reference to the residue inspection anywhere in its **Adds**, **Tests**, or **Ships when**. `H6a`'s **Ships when** obligates the live-corpus closing gate, the removed-mapping/removed-citation checks, and a manual real-host smoke run recorded in the gate-activation commit message, but never the residue inspection. No other leaf references it either.

The result is a backstop that the plan asserts is release-blocking but that no leaf's acceptance criterion obligates anyone to perform. The release can ship green with the entire conceded-residue surface never swept, because nothing observable ties the inspection to the release event.

## Solution Space

**Shape:** single

### Recommendation

Wire the residue inspection into `H6a` as a release-gate exit-gate obligation, parallel to the manual real-host smoke run `H6a` already carries:

- In `docs/plan_topics/H6a-live-corpus-activation.md`, add the residue inspection to the **Ships when** acceptance condition: the loom 1.0 release does not pass until the `conventions.md` §Loom 1.0 release-time residue inspection has been performed against the live corpus with every item clean (or each finding dispositioned), and that outcome is recorded in the same gate-activation commit message that already records the real-host smoke run. This reuses the existing falsifiable-commit-message evidence convention `H6a` defines, so the inspection becomes detectable-after-the-fact the same way the smoke run is.
- In `docs/plan_topics/conventions.md` §Loom 1.0 release-time residue inspection, name `H6a` as the owning leaf/executor: state that the inspection is an exit-gate obligation of the terminal `H6a` release-gate leaf, performed by the `H6a` gate-activation owner/auditor and recorded in the gate-activation commit message, replacing the ownerless "a reviewer reading `src/**`" framing with the named owner.

Keep `conventions.md` authoritative for the checklist content and `H6a` authoritative for the gating obligation; the conventions.md text should point at `H6a` as the owner rather than restate the gating mechanics. Edge case the implementer must watch: a residue defect found at inspection time blocks the release, so the gate-activation commit must record either an all-clean result or each dispositioned-residue entry — a bare "performed" line is insufficient evidence.

## Consequence

**Severity:** correctness

The residue inspection is the sole backstop for nine residue classes the mechanical gates concede they cannot witness. Because no leaf's exit gate obligates the sweep and `H6a` does not reference it, loom 1.0 can pass every gate green while the inspection is never performed — every conceded residue class (indirect ambient reads, blocking-runtime stalls, split-row untested facets, entry-shim impurity, etc.) ships unverified. Two implementers also diverge on who runs it and when, since the plan names no executor.
