## Finding

The SLSH-4 surface paragraph in `slash-invocation.md` states that its render table "is exhaustive over the loom 1.0.0 `QueryError` union (nine variants, listed in [Query ‚Äî Failure modes](./query.md))". Following that link lands on `query.md`, which is now a four-line page containing only an `## Contents` list of four sub-pages ‚Äî it has no "Failure modes" section and no variant enumeration. A reader who clicks through to confirm the "nine variants" claim or to look up a specific variant's fields finds nothing.

The canonical enumeration of all nine `loom 1.0.0` `QueryError` variants now lives in `errors-and-results/queryerror-variants.md` (under the `## QueryError variants` heading, anchor `#queryerror-variants`). The very same sentence in `slash-invocation.md` already links that file correctly for `ValidationError` (`./errors-and-results/queryerror-variants.md#query-time-variants`), so the page has two cross-references to the same destination written two different ways ‚Äî one live, one dangling.

The fix is to repoint the "listed in ‚Ä¶" clause at the real home of the enumeration: `[Errors and Results ‚Äî QueryError variants](./errors-and-results/queryerror-variants.md#queryerror-variants)` (or drop the parenthetical entirely, since the sibling `ValidationError` link already establishes the destination page).

## Issue introduction

**Verdict:** introduced-by-later-commit
**Introducing commit:** `f5e89f4` ‚Äî "docs: enforce 100KB hard / 30KB recommended size cap on spec set" (2026-06-04)
**History:** The link was authored correctly in `b3bc4ce` ("spec: pin failure-observability surface (Cluster A, Option C)", 2026-05-04): at that point `query.md` still carried a `**Failure modes.**` section that enumerated the `QueryError` variants, so `[Query ‚Äî Failure modes](./query.md)` resolved to real content (`git show b3bc4ce:spec_topics/query.md` shows the section). Commit `f5e89f4` split `query.md` into per-feature sub-pages to meet the size cap, reducing it to a four-item `## Contents` index and relocating the `QueryError` enumeration to `errors-and-results/queryerror-variants.md`. That same commit edited the SLSH-4 sentence to update the adjacent `ValidationError` link (`./errors-and-results.md#...` ‚Üí `./errors-and-results/queryerror-variants.md#query-time-variants`) but left the "[Query ‚Äî Failure modes](./query.md)" link untouched, leaving it dangling (`git show f5e89f4 -- docs/spec_topics/slash-invocation.md` shows the ValidationError half updated and the Failure-modes half unchanged). The defect is a referential-integrity break from a content relocation, not a fault in the original authoring.

## Solution Space

**Shape:** single

### Recommendation

Repoint the "listed in" clause in the SLSH-4 surface paragraph of `slash-invocation.md` from `[Query ‚Äî Failure modes](./query.md)` to `[Errors and Results ‚Äî QueryError variants](./errors-and-results/queryerror-variants.md#queryerror-variants)`. The `#queryerror-variants` anchor resolves to the `## QueryError variants` heading and is already used as a link target within that file. Use the section-level anchor (`#queryerror-variants`) rather than `#query-time-variants`, since the claim is about the full nine-variant union, not only the query-time subset that the adjacent `ValidationError` link targets.

## Consequence

**Severity:** advisory

A reader following the "nine variants, listed in ‚Ä¶" link to verify or look up `QueryError` variant fields reaches a bare contents index and must hunt for the real enumeration. No implementer divergence results ‚Äî the variant list exists and is even linked correctly elsewhere in the same sentence ‚Äî but the dangling cross-reference degrades navigability of the failure-observability surface.
