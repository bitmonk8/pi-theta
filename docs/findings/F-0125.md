## Problem

In the `V4b` and `V4b-T` (runtime-panics) leaves, the Tests list follows a pattern where a bullet leading with a `loom/runtime/...` diagnostic code asserts that the named code **fires**. The third Tests bullet breaks that pattern: it leads with `loom/runtime/internal-error` but asserts the opposite — that an uncatchable host-fatal condition (NOCEIL-3 carve-out) emits no diagnostic. Because its leading token is a diagnostic code in a list where every other leading code denotes a firing, a test writer following the established pattern can write an inverted test asserting `loom/runtime/internal-error` **fires** on an uncatchable host fatal — behaviour NOCEIL-3 forbids the runtime from delivering.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e — pi-loom plan: build/update plan for spec.md + review (2026-06-10, Thomas Andersen)
**History:** Both `V4b-runtime-panics.md` and `V4b-T-runtime-panics.md` were created in the single plan-build commit c6a664e, and `git log -S 'emits no diagnostic'` localises the defective bullet wording to that same commit. The inverted code-first phrasing has been present since the leaf files first entered the corpus; no later edit introduced it.

## Solution approach

Rewrite the third Tests bullet in both `V4b` and `V4b-T` so the uncatchable condition is the subject and the absence of any diagnostic is the explicit assertion, rather than leading with the `loom/runtime/internal-error` code. Retain the inline `NOCEIL-3` citation, and apply the identical replacement to both leaves so the paired tests bullets stay aligned.

## Solution constraints

- The spec is read-only for this fix; NOCEIL-3 already grounds the corrected wording.
