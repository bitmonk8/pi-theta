## Problem

`V3a`'s **Adds** names "identifier resolution (local > fn > import > callable)" as a load-bearing evaluator behaviour, and `expressions.md` §Identifier resolution defines it as a normative first-match-wins ordering with explicit shadowing semantics. None of `V3a`'s four **Tests** bullets exercises this rule — they cover operator precedence/associativity, short-circuit observability, equality/ordering/arithmetic widening, and the `loom/parse/non-boolean-condition` reject, but none asserts which binding wins when a name is visible at more than one tier. An evaluator that resolves the tiers in the wrong order satisfies every existing `V3a` assertion and ships green, leaving a behaviour the leaf claims to add unasserted.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (`pi-loom plan: build/update plan for spec.md + review`, 2026-06-10)
**History:** `V3a` was created in c6a664e with "identifier resolution (local > fn > import > callable)" already in its `Adds` and no corresponding `Tests` bullet. `git log -S 'local > fn > import > callable'` returns only c6a664e for this file, and `git log -p --follow` over the leaf shows the `Adds` clause present from creation while the `Tests` section never gained an identifier-resolution assertion across the later edits (stdlib split-out 2a4f797/20b4689, heading-scope edit 8704946, etc.). The gap has existed in every revision of the leaf.

## Solution approach

Add a `Tests` bullet to `docs/plan_topics/V3a-expression-evaluator.md` asserting identifier-resolution order, citing `expressions.md` §Identifier resolution (`expressions.md#identifier-resolution`). Scope the assertion to what is constructible at `V3a`'s dep scope: a local `let` binding / function parameter shadows a same-file top-level `fn` of the same name (tier 1 beats tier 2), and a bare identifier with no binding resolves to `loom/parse/unknown-identifier`. Extend `Ships when.` to state that `npm test` proves the local-shadows-`fn` resolution order and the `loom/parse/unknown-identifier` no-match disposition.

## Solution constraints

- Out of scope: tier-3 (import) and tier-4 (callable-set) precedence and the load-time collision rejection across tiers (2)–(4) depend on binding sources introduced by `V15c` (imports) and `V6c` (callable set), which are outside `V3a`'s `Deps` (`V3a-T`, `V2b`, `V2c`) — do not force a four-tier assertion into `V3a`.
