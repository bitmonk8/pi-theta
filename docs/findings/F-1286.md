## Problem

The `query/` subtree is registered under prefix `QRY` in the REQ-ID prefix table (`governance/req-id-prefix-table-active-a.md`) with no `(no IDs — narrative)` exclusion, yet no `QRY-N` anchor has ever been coined across its four pages (`query-forms.md`, `query-failure-and-repair.md`, `query-tool-loop.md`, `query-escapes-stringification.md`). GOV-9's `#prefix-n` cross-link contract is therefore permanently unsatisfiable for the entire subtree: its independently-testable obligations are reachable today only via section anchors or `(QRY code-keyed area)` shims, which the plan leaves `V13a/b/c/d/g/h` and the `cka-12` coverage-matrix row currently route around. This is the canonical "page registered, prefix assigned, IDs never coined" residue class GOV-22 was written to drain.

## Issue introduction

**Verdict:** present-since-inception (for the QRY subtree as it currently exists), with the un-coined obligations carried forward across two structural rewrites.

**Introducing commits:**

- `c1e5d64` (2026-05-05, "spec: REQ-ID infrastructure + Phase 12a heading promotion on five named pages (Decision #12, partial)") — registered `query.md` under prefix `QRY` in the new REQ-ID prefix table. The commit message itself records the deferral: "Phase 12b (per-rule REQ-ID assignment) and the per-leaf REQ-ID citation pass remain as the mechanical sweep across the rest of spec_topics/ and plan_topics/." Prefix registered, IDs deliberately not coined.
- `1d773d6` (2026-06-04, "pi-loom spec: resolve 'GOV-3/GOV-6 subdirectory scan scope and prefix-table hub-stub bindings'") — rewrote the row from `query.md → QRY` to `query/ → QRY` (with `GOV-3`'s scan glob widened to `spec_topics/**/*.md`), extending the un-coined registration over the four-file subtree the page had since been split into without coining any anchors at that point either.

**History:** `git log -S '| \`query.md\` | \`QRY\` |'` and `git log -S '| \`query/\` | \`QRY\` |'` give the two registration events above. `git log --all -S 'QRY-1'` against the spec corpus returns no commit that ever added a `QRY-1` (or any other `QRY-N`) anchor to `docs/spec_topics/query/`, and `grep -rn 'QRY-' docs/spec_topics/query/` returns nothing on `HEAD`. The drain mechanism, GOV-22 (commit `4f588e0`, 2026-06-03, "pi-loom spec: resolve 'Non-narrative pages with zero coined REQ-IDs non-referenceable'"), was added one day before the subtree-renaming commit `1d773d6` and is the rule under which the residue is supposed to drain progressively; no QRY-row coinage has yet ridden that mechanism. The plan-side workaround — the `(QRY code-keyed area)` shim in the V13* leaves — entered alongside the matrix's existing `(un-anchored; GOV-22 residue)` annotations and is the visible artefact of the un-drained residue.

## Solution approach

Coin `QRY-N` anchors across the four `docs/spec_topics/query/` pages in GOV-1 dual-form, one per independently-testable obligation, allocating numeric tails per GOV-3. Then repoint the `(QRY code-keyed area)` citations in the `V13a`, `V13b`, `V13c`, `V13d`, `V13g`, and `V13h` plan leaves to `#qry-n` cross-links per GOV-9 and drop the `cka-12` coverage-matrix row's reliance on shim citations. GOV-22 makes the coinage progressive — it may land page-by-page as each page is next edited, with no corpus-wide pass reserved.

## Solution constraints

- `QRY` is a single numbering pool across the four-page subtree — the prefix table registers the directory, not each page — so allocate per GOV-3 across the whole subtree and do not restart numbering per page.
