## Finding

The provider-error-mapping table and the **`TransportError.retryable` population** rule are keyed on an HTTP status class. The *Classifier input surface* note in `provider-error-mapping.md` pins the only described source of that status class: `ProviderResponse.status`, delivered through `@earendil-works/pi-ai`'s `StreamOptions.onResponse` callback, which the runtime "registers `onResponse` on every `complete()` call." `complete()` is the off-session free-function path used by the binder and by the typed-query *forced respond turn* only.

Two other authoritative sites route the **query paths** through that same status-keyed classification. `binder-inference.md` says the binder classifies provider failures "through the Provider error mapping table exactly as the typed and untyped query paths classify provider failures." `query-failure-and-repair.md` QRY-10 delegates query-time `ContextOverflowError`/`TransportError` classification wholesale to the table, including its HTTP-status arms ("a non-overflow 4xx/5xx response â€¦ the three-arm classification is owned by Provider error mapping").

But the typed/untyped query paths do not drive through `complete()`. Untyped queries (and a typed query's free phase) drive through `pi.sendUserMessage` / `session.sendUserMessage`, which return `void`, expose no `onResponse` callback, and surface provider failure only via the post-`waitForIdle()` probe of the trailing `assistant` message's `stopReason` / `errorMessage` (PIC-51 prompt mode; PIC-43 subagent). There is no `ProviderResponse.status` on these paths. PIC-51 consequently hard-codes every `stopReason: "error"` to a fixed `TransportError { http_status: null, retryable: false }` â€” it never consults the status-keyed table at all.

The corpus therefore specifies the query-path classification two incompatible ways: QRY-10 and the binder cross-reference say it runs the full HTTP-status-keyed table (which distinguishes 5xx/429 `retryable: true` from 4xx `retryable: false`, populates `http_status`, and can recognise an HTTP-400 overflow signature), while PIC-51/PIC-43 reduce every `sendUserMessage`-surfaced failure to a single status-less `TransportError`. No status-capture mechanism exists for the `sendUserMessage`-driven paths, so an implementer cannot build the table-conformant classifier QRY-10 promises there. The two readings produce observably different `QueryError` values for the same failure.

## Solution Space

**Shape:** single

### Recommendation

Reconcile the two classification stories by scoping the HTTP-status-keyed table to the `complete()`-driven paths and defining the reduced query-path classifier explicitly:

- In `provider-error-mapping.md` *Classifier input surface*, state that `ProviderResponse.status` (via `onResponse`) is available **only** on the `complete()` paths (binder inference + typed-query forced respond turn). State that the `sendUserMessage`-driven query paths (untyped queries; typed-query free phase; their subagent equivalents) have no HTTP-status input and classify from the trailing `assistant` message's `stopReason` / `errorMessage` alone.
- Define the reduced query-path mapping the table-less paths actually use: a `stopReason: "length"` output-boundary terminator â†’ `ContextOverflowError` (per the existing Stop-reason classification rule, which already reads only `stopReason`); an `errorMessage` matching a provider overflow signature â†’ `ContextOverflowError`; every other `stopReason: "error"` â†’ `TransportError { http_status: null, retryable: false }`. This makes PIC-51/PIC-43's fixed mapping the *specified* query-path behaviour rather than a silent contradiction of QRY-10.
- Correct QRY-10 in `query-failure-and-repair.md` and the `binder-inference.md` cross-reference so they no longer claim the query paths run the full HTTP-status-keyed table. QRY-10 should delegate the **overflow-signature and stop-reason** arms to the table (which key on `errorMessage` / `stopReason`, both available on the query paths) while pinning the HTTP-status arms (`http_status` population, the 5xx/429-vs-4xx `retryable` split) to the `complete()` paths only.

Edge cases the fixer must preserve: the binder and typed-query forced respond turn keep the full status-keyed table unchanged; the `retryable` default for the status-less query path stays `false`; and the existing `stopReason: "length"` â†’ `ContextOverflowError` rule already covers clean output-boundary overflow on the query paths, so only HTTP-status-only failures (which the query path genuinely cannot observe) are lost â€” that loss must be stated, not left implicit.

(`pi.sendUserMessage` is Pi's own user-session provider call; loom observes only the resulting `AssistantMessage`, so a status-capture mechanism for the query paths is not constructible â€” the scoping reconciliation is the available fix, not a new `onResponse` wiring.)

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one follows QRY-10 / the binder cross-reference and tries to run the HTTP-status-keyed table on the query paths (impossible â€” there is no status input on `sendUserMessage`), the other follows PIC-51/PIC-43 and emits a fixed `TransportError { http_status: null, retryable: false }`. The resulting `QueryError` differs in `http_status`, in `retryable` (a query-path 5xx/429 that should be `retryable: true` is forced to `false`), and in whether an HTTP-400 overflow on a query path is classified as `ContextOverflowError` or `TransportError` â€” divergent observable failure values across conforming implementations.
