## Finding

The opening sentence of the Depth-Enforcement *Enforcement point* paragraph in [`docs/spec_topics/schema-subset.md`](../../../docs/spec_topics/schema-subset.md) — "The depth check lives in the `SchemaValidator` service (the same component that owns the AJV cache; see [Implementation Notes — Runtime](./implementation-notes.md#runtime))" — pins an internal component layout to a single named DI seam. The behavioural obligation is independent of which component hosts the check: the depth walk must fire **before AJV** at the five enumerated enforcement sites, with the per-boundary surfaces and error shape fixed by the same section.

The named binding is also redundant with the surrounding contract. `SchemaValidator`'s normative behavioural contract lives in [`pi-integration-contract/host-interfaces-services.md` — PIC-11 `SchemaValidator` interface](../../../docs/spec_topics/pi-integration-contract/host-interfaces-services.md#schemavalidator-interface), and even that contract is declared a behavioural list with the TypeScript shape "non-normative" per GOV-18 arm (a). Hard-binding the depth check to that named service in `schema-subset.md` therefore over-constrains the implementation more tightly than even the seam's own owning page does, and conflicts with the validator-neutral framing the rest of the corpus uses for validation obligations (e.g. the AJV-naming finding in `defaulting-system-note-echo.md`).

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `216130d` (2026-05-04, "pi-loom spec: resolve \"Depth ≤5: counting algorithm and enforcement point unspecified\"")
**History:** Before `216130d` the `Depth ≤ 5` bullet in `schema-subset.md` carried no Depth-Enforcement section at all (only a one-line cap with a parenthetical about OpenAI's cap). `216130d` introduced the whole `## Depth Enforcement` section in one edit, including the *Enforcement point* paragraph whose opening sentence pins the check to "the `SchemaValidator` service (the same component that owns the AJV cache)". `git log -S 'lives in the \`SchemaValidator\`'` returns only that commit (plus the later `81bf52a` path-reshuffle that moved `spec_topics/` under `docs/` without altering the sentence and `f7a505e` which only touched an internal cross-reference target inside the same section). The named-component binding was therefore present from the moment the Depth-Enforcement section entered the corpus.

## Solution Space

**Shape:** single

### Recommendation

Restate the *Enforcement point* paragraph's opening sentence behaviourally and drop the `SchemaValidator` binding. Replace

> The depth check lives in the `SchemaValidator` service (the same component that owns the AJV cache; see [Implementation Notes — Runtime](./implementation-notes.md#runtime)). It runs at every site where a Loom-declared schema is validated against runtime JSON:

with a behavioural lead-in such as

> The depth check runs at every site where a Loom-declared schema is validated against runtime JSON, immediately before AJV at each site:

and keep the existing numbered list of five enforcement sites unchanged. If a pointer to the `SchemaValidator` seam is desirable for navigation, demote it to a parenthetical non-normative aside — e.g. "(The reference implementation co-locates the walk with the `SchemaValidator` service that owns the AJV cache; see [PIC-11](./pi-integration-contract/host-interfaces-services.md#schemavalidator-interface) and [Implementation Notes — Runtime](./implementation-notes.md#runtime).)" — placed after the numbered list rather than inside the normative obligation.

Edge cases for the implementer:

- Do not touch the rest of the *Enforcement point* paragraph, the per-boundary table, or the *Error shape* / *Edge cases* sub-paragraphs — those carry the boundary-dependent routing and are not at issue.
- The plan leaves already use "loom-owned depth walk" wording (`V5e`); the spec change brings the spec into agreement with that framing and requires no plan edit.
- The companion finding "Depth check prescribes 'recursive descent … with a depth counter'" edits the *next* sentence of the same paragraph; if both land together, retain the behavioural-only lead-in and remove the recursive-descent prescription in one coherent rewrite.

## Consequence

**Severity:** cosmetic

The behavioural contract (five sites, before AJV, per-boundary surfaces and error shape) is fully pinned, so no implementer is misled about *what* must happen. The over-prescription only forces the depth-walk code to physically reside in the `SchemaValidator` implementation when a behaviourally equivalent placement (e.g. a sibling helper invoked at each call site) would satisfy every observable obligation.
