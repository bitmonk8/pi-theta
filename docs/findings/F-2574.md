## Finding

The hot-reload invalidation set is specified as "the changed file and every transitive `.warp` importer" (implementation-notes.md, Static-resolution load pass bullet), and the parallel PIC text re-parses "just the changed file plus every transitive `.warp` importer reached through the import graph" (`pi-integration-contract/registration-steps.md`, Hot-reload subsystem; the watcher "maintains â€¦ a `.warp` import graph"). Taken literally, the invalidation set is restricted to importers that are themselves `.warp` files.

But the import relation is asymmetric: `.loom` files import `.warp` files, and `.loom` files are *not* importable from each other (`imports.md`). A `.loom` is therefore always an importer and never an importee â€” it sits at the root of every import chain it participates in. The static-resolution pass caches parsed/lowered results keyed per visited file, and a `.loom` entry's cached type information (cross-loom return-type inference at `invoke<T>` / `.loom`-callable sites, `invoke(...)` argument-type checking, lowered query schemas) is derived from the schemas and functions it imports out of `.warp` files.

Under the literal reading, when a `.warp` schema or `fn` signature changes, only `.warp` importers are re-parsed; the `.loom` files that transitively import that `.warp` are excluded from the invalidation set and keep their stale cached type information until a full reload. The phrasing names only `.warp` importers and the import graph is described as a "`.warp` import graph", so an implementer cannot tell whether `.loom`â†’`.warp` edges are tracked and whether `.loom` entries are re-staged on a transitively-imported `.warp` change. The two pages must say the same thing, and both currently say `.warp`.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** d38ffdb ("spec: hot-reload via in-process LoomRegistry swap; no ctx.reload() for content edits (Decision #8)", 2026-05-05)
**History:** The "transitive `.warp` importer" phrasing entered with the hot-reload mechanism itself in d38ffdb; `git log -S 'transitive \`.warp\` importer'` returns only that commit, so no later edit altered it. That commit's own message frames the design as tracking "`.warp` transitive importers via in-memory import graph", confirming the `.loom`-importer exclusion was present from the feature's introduction rather than a regression. The same commit rewrote the PIC registration step end-to-end, propagating the identical `.warp`-only phrasing to `registration-steps.md`.

## Solution Space

**Shape:** single

### Recommendation

Define the invalidation set as **every file (whether `.warp` or `.loom`) that transitively imports the changed file**, and align both occurrences:

- implementation-notes.md: replace "every transitive `.warp` importer" with "every file (whether `.warp` or `.loom`) that transitively imports it".
- registration-steps.md: replace both "every transitive `.warp` importer" occurrences and the "`.warp` import graph" description with an import graph that records `.loom`â†’`.warp` edges, so the staged re-parse / validator-recompile / `pi.registerTool` set includes the `.loom` entries in `LoomRegistry` that transitively import the changed `.warp`.

The changed file is always re-parsed, so a directly-edited `.loom` is already covered; the gap is purely the transitive-importer set when a `.warp` changes. Edge cases the implementer must handle: re-export chains (`export â€¦ from`) extend the transitive set; the import graph must be keyed so a `.warp` change resolves to the root `.loom` entries, since those are the entries that live in `LoomRegistry` and get dispatched.

## Consequence

**Severity:** correctness

Under the literal `.warp`-only reading, editing a `.warp` file's schema or function signature does not refresh the cached type checks, lowered query schemas, or AJV validators of the `.loom` files that import it, so those looms dispatch against stale type information after a save until a full reload. Two implementers â€” one tracking only `.warp`â†’`.warp` edges, one tracking `.loom`â†’`.warp` edges too â€” produce divergent hot-reload behaviour.
