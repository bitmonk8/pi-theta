## Finding

The *Failure-surface contract* in `audit-failures.md` pairs two adjacent sub-clauses. The first says: "Each negative-test fixture MUST assert that (a) the discriminator token it emits matches the `audit/<class>/<family>/<symptom>` shape above ŌĆ” and (b) the `<family>` token it emits is the one the leaf-side routing table maps to the resolution arm ŌĆ” ; PIC pins these two fixture assertions as required". The second ŌĆö the *Per-family record-shape table* ŌĆö pins normative per-family value ranges for both `symbol` and `proposed-resolution`, and declares: "a violation record ŌĆ” that carries a value outside the per-family range below for a given family, violates this rule."

The two clauses do not meet. The fixture-assertion list (a)/(b) is closed at exactly the two assertions and never names `proposed-resolution` (nor `symbol`). Per-family `proposed-resolution` ranges include "inventory-entry-suggestion: the `SDK_SURFACE_INVENTORY` entry the contributor would add" (families 1ŌĆō3), "recognised-shape suggestion" (family 4), and "deletion-or-rewrite suggestion ŌĆ” naming whichever of that branch's five sub-cases the marker falls under" (family 5). None of these is observable through the pinned fixture surface: a conformant audit can emit the literal `<n/a>` sentinel as the `proposed-resolution` value on every violation record and pass its own pinned negative-test fixtures, because the fixtures only assert the discriminator shape and the `<family>` token.

The defect is symmetric with respect to `symbol` ŌĆö the same omission lets an implementation emit `<n/a>` for `symbol` on family-1/2/3/4 records ŌĆö but `proposed-resolution` is the wider gap because its per-family ranges are more elaborate (inventory-entry-suggestion vs. exemption-marker form vs. rewrite-shape vs. step-2(b) sub-case naming) and only those ranges encode the routing the contributor uses to fix a violation.

## Solution Space

**Shape:** single

### Recommendation

Extend the mandatory fixture-assertion list from two assertions to three. Replace "PIC pins these two fixture assertions as required" with a third assertion clause and the matching pin: "(c) the `proposed-resolution` field value falls within the per-family range pinned by the *Per-family record-shape table* below for the family the fixture seeds ŌĆö for families (1)/(2)/(3) the value is either an `SDK_SURFACE_INVENTORY` inventory-entry suggestion or a canonical `// allow-pi-surface:` marker per step 2(b) branches (2)/(3); for family (4) a recognised-shape rewrite suggestion per step 2(b) branch (5); for family (5) a deletion-or-rewrite suggestion naming one of step 2(b) branch (4)'s five sub-cases; PIC pins these three fixture assertions as required". The literal `<n/a>` sentinel is permitted for `proposed-resolution` only on infrastructure-failure records (per *Infrastructure-failure handling*) and on green-path canary records (per *Non-empty-scan canary*) ŌĆö it MUST NOT appear on a violation record's `proposed-resolution`, and the new assertion (c) carries that prohibition into the fixture surface.

The leaf-side `<family>` token names and the exact wording of the per-family proposed-resolution paraphrases remain leaf-owned (the *Wire serialisation* clause already requires the field be a printable-ASCII paraphrase that names the step-2(b) arm by reference); assertion (c) binds only the per-family range membership, not the exact paraphrase text. Update `V18b-T` and `V18b`'s **Tests** bullets to require that the off-inventory-reference fixture and the family-(4)-shape fixture each assert the third predicate on the emitted `proposed-resolution`. Edge case for the implementer: a malformed-marker dual-emission run emits two records for the same `(path, line)` ŌĆö the family-(5) record carries a step-2(b) branch-(4) sub-case suggestion and the dual-emitted family-(1)/(2)/(3) record carries an inventory-entry suggestion; assertion (c) applies independently to each record under its own family's range.

## Consequence

**Severity:** correctness

Two implementers honouring only the pinned fixture surface diverge on the `proposed-resolution` field: one emits per-family inventory suggestions / step-2(b) sub-case references and one emits `<n/a>` uniformly. Both pass their own fixtures, both technically violate the normative per-family range rule, but the rule has no test surface that would catch the second implementation ŌĆö and CI parsers a contributor relies on for routing a violation to the right step-2(b) arm break silently against the `<n/a>` variant.
