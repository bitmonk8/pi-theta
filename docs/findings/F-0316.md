## Finding

`spec_topics/pi-integration-contract/host-interfaces-core.md` §*Tool execution from loom code* (anchor `#tool-execution-from-loom-code`) carries the normative procedure that lowers a Pi tool's `{ content, isError }` envelope to loom 1.0's `Result<string, QueryError>`. That procedure is a cluster of pure-prose MUSTs with no numbered REQ-ID and no `loom/…` registry code:

- filter `content` to entries with `type === "text"` and join their `.text` values with a single `"\n"` (no separator before the first or after the last block);
- a clean `isError: false` envelope lowers to `Ok(<joined text>)`, and an empty result (`content: []` or no surviving text blocks) is a legal `Ok("")` — non-text blocks are discarded with **no** `RuntimeEvent`, `loom-system-note`, or diagnostic;
- an `isError: true` envelope lowers to `Err(CodeToolError { cause: "execution", message: <m>, … })` where `<m>` is the filtered/joined text UTF-8-encoded and truncated to at most 4096 bytes on a Unicode code-point boundary (a code point that would straddle the limit is dropped entirely);
- when no text survives the filter under `isError: true`, `<m>` is the fixed string `"tool reported an error with no text content"`;
- an `execute()` throw lowers to `Err(CodeToolError { cause: "execution", … })` whose `<m>` is the thrown value coerced to the underlying-error string and truncated under the same 4096-byte rule — `execute()`-throw and `isError: true` share the `"execution"` cause and are distinguished only by the `message` source.

No plan leaf owns these MUSTs and no coverage-matrix row anchors them. `V14a`'s `Spec.` cites only `tool-calls.md`; its `Tests.` assert the closed `CodeToolError` enum, the two return-type *accepted-path* rows, and the `.loom`-callable failure surface — none of them exercise the text-filter, the `\n` join, the `Ok("")` empty case, the 4096-byte code-point-boundary truncation, the fixed no-text-content message, or the `execute()`-throw coercion. `V14c` covers off-surface outcome routing, not the message/value content of the accepted path. The coverage matrix's `tool-calls.md (TOOL)` row maps to `V14a`/`V14b`/`V14c`/`V14e` but says nothing about `host-interfaces-core.md`, and no separate row for that page's tool-execution surface exists.

These are exactly the third closing-gate surface that `conventions.md` *REQ-ID discipline* defines (a normative MUST on a non-narrative `spec_topics/**` page carrying no numbered `PREFIX-N` REQ-ID and no `loom/…` code — the GOV-22 un-anchored-obligation residue). The matrix already carries rule-driven rows of this exact kind for sibling `pi-integration-contract/**` pages (e.g. the `conversation-drive.md §untyped-query-ok-extraction` row → `V9c`); the tool-execution lowering surface is the one such page that has no row at all.

## Solution Space

### Recommendation

Anchor and test the tool-call envelope-lowering MUSTs entirely within the plan files; `host-interfaces-core.md` and `tool-calls.md` stay read-only, since the MUSTs already exist there and only need to be cited and tested. Apply the two edits in order — the coverage-matrix row first, then the `V14a` extension — so the matrix row fixes `V14a` as the owner and enumerates exactly which lowering MUSTs are in scope before the `V14a` `Spec.`/`Tests.` edit lands against that already-anchored baseline.

First, add one rule-driven row to the *Code-keyed obligation areas (no numbered REQ-IDs)* table in `coverage-matrix.md`, keyed on `host-interfaces-core.md §Tool execution from loom code` and naming `V14a` as the closing leaf, formatted to mirror the existing `pi-integration-contract/**` un-anchored-MUST rows (e.g. the `conversation-drive.md §untyped-query-ok-extraction → V9c` row) and carrying the `(un-anchored; GOV-22 residue)` marker those rows use. The row's spec-area cell enumerates the lowering MUSTs it closes: text-filter + `\n` join; the `Ok("")` empty case with no diagnostic; `isError: true` → `Err(CodeToolError{cause:"execution"})` with 4096-byte code-point-boundary truncation; the fixed `"tool reported an error with no text content"` message; and `execute()`-throw coercion under the same truncation rule. Keep the cell's wording traceable to the `#tool-execution-from-loom-code` prose so the enumeration does not drift from the spec's MUST list.

Then, add `host-interfaces-core.md` (anchor `#tool-execution-from-loom-code`) to `V14a-tool-calls.md`'s `Spec.` field alongside the existing `tool-calls.md` citation, add `Tests.` entries in `V14a-T-tool-calls.md` asserting the five lowering mechanics on the accepted path, and extend `V14a`'s `Ships when.` to require those assertions green. The new tests assert: text-filter + `\n` join produces the expected `Ok(<text>)`; a no-surviving-text envelope yields `Ok("")` with no diagnostic emitted; an `isError: true` envelope yields `Err(CodeToolError{cause:"execution", message})` with the message truncated on a code-point boundary at the 4096-byte limit; the no-text `isError` case yields the fixed `"tool reported an error with no text content"` message; and an `execute()` throw yields `cause:"execution"` with the underlying-error string coerced and truncated. Scope these tests to the accepted-path content MUSTs only — the non-conforming-shape disposition (`loom/runtime/internal-error`, `details.kind = "tool-return-shape"`) is code-keyed, routed off `CodeToolError`, and owned elsewhere, so do not pull that off-surface routing into this leaf.

## Consequence

**Severity:** correctness

No plan step verifies the envelope→`string` lowering, so two reasonable implementers diverge on observable behaviour: the `\n` join, the `Ok("")` empty-result case, the 4096-byte code-point-boundary truncation, the fixed `"tool reported an error with no text content"` string, and `execute()`-throw coercion can each ship in a non-conforming form that no test catches. Because the MUSTs are un-anchored residue with no closing-leaf row, the `H5a` closing gate reddens against them once the terminal `H6a` activation leaf flips the matrix to its live-corpus footing.
