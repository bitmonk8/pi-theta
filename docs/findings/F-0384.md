## Problem

`V9i`'s parallel-spawn step (PIC-22) dispatches `createAgentSession` for all `N â‰Ą 3` subagent-mode tool calls via a `Promise.all` over the per-call spawns. Under that reading, one rejecting spawn rejects the whole `Promise.all` while sibling spawns that already resolved hold live `AgentSession` handles, and `Promise.all` discards those fulfilled values, so the runtime never gets a reference to dispose them. Such a session is never entered into PIC-9's per-invocation disposal `finally` (its interpreter execution never begins) and is out of reach of the `session_shutdown` settle-all (it predates its registry entry), so it is disposed by no path â€” silently violating PIC-9's mandatory-disposal invariant. The only PIC-22 test witnesses the happy path, so an implementer who writes the orphan-prone spawn shape passes the test and ships a session/connection leak.

## Solution approach

Extend `V9i`'s **Adds.** parallel-spawn clause to pin the load-bearing disposal property: on any rejected spawn within the `N â‰Ą 3` batch, every sibling `createAgentSession` that already resolved to a live `AgentSession` is disposed, so PIC-9's mandatory-disposal guarantee holds even for sessions whose per-invocation interpreter `finally` never began â€” leaving the spawn mechanism that achieves this to the implementer. Add a partial-spawn-failure case to `V9i`'s **Tests.** (and mirror it in `V9i-T`): with `N = 3` where one `createAgentSession` rejects and the other two resolve to live sessions, assert both resolved sessions are disposed with no live session left undisposed while the batch failure still surfaces through the existing `loom/runtime/internal-error` routing.
