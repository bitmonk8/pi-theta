## Finding

The Loom literal sublanguage in `grammar.md` admits `BareObjectLit` in two positions — the RHS of a `params:` default and the single positional argument of a Pi-tool call — and then states one **Field rule** that "applies identically to `BareObjectLit` and `NamedObjectLit`": *"Every declared field of the LHS schema (or the named variant schema) MUST be present. Omissions are `loom/parse/missing-object-field`."* The section closes by asserting that "the is-literal check runs at parse time against the Loom AST; it does not require a separate parser." Both statements place the field-presence check in the pure parse phase.

For the Pi-tool argument position the "LHS schema" is, per the same page's position rule, *"the Pi tool's registered input schema."* That schema is not a parse-phase artefact: it is obtained only after the callee is resolved to a Pi tool against the per-loom callable table, which is materialised at loom-load time (the resolution snapshot in `frontmatter.md`), not at parse time. So the field-presence predicate as written presupposes (a) callee resolution to a Pi tool and (b) availability of that tool's registered input schema — neither of which is in scope for a "pure parse-time AST check."

`tool-calls.md` has already resolved the timing in the opposite direction for Pi tools: *"a Pi-tool argument that does not match the tool's input schema is never a parse error — it is always caught by the runtime AJV check and surfaces at runtime as `Err(CodeToolError { cause: "validation", ... })`. The `loom/parse/tool-arg-not-literal` and `loom/parse/tool-arg-arity` parse checks remain live for Pi-tool calls regardless; only the type-mismatch check is runtime-only on the Pi-tool arm."* `grammar.md` and `tool-calls.md` therefore contradict each other on whether a missing field in a Pi-tool argument is `loom/parse/missing-object-field` (parse-phase, per `grammar.md`'s blanket field rule) or a runtime AJV `CodeToolError { cause: "validation" }` (per `tool-calls.md`).

The rule also presupposes a flat, all-required object schema — the loom-schema model, where every declared field is required by default and optionality is written `T | null`. An externally-owned Pi-tool JSON Schema does not obey that model: it carries an explicit `required: [...]` subset (so some declared properties are optional), may set `additionalProperties`, and need not even have an object root. "Every declared field MUST be present" has no faithful mapping onto such a schema; applied literally at parse time it would reject a call that legitimately omits an optional field. The page never states which fields count as "declared" for an external schema, how optional fields are treated, or that the predicate is delegated to runtime AJV against the schema's own `required` set.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `f841b12` (2026-05-04, "spec: resolve Cluster 2 (literal sublanguage + grammar appendix)"); `4385d86` (2026-06-04, "pi-loom spec: resolve \"Pi-tool argument type-checking timing\"")
**History:** `git log -S` over the cited tokens traces the defect to the interaction of two commits, not a single inception. `f841b12` introduced the entire Loom literal sublanguage in `grammar.md`, including the `BareObjectLit` Pi-tool position rule, the blanket "Every declared field of the LHS schema … MUST be present → `loom/parse/missing-object-field`" field rule, and the "is-literal check runs at parse time … does not require a separate parser" assertion; at that point `grammar.md` and `tool-calls.md` were mutually consistent — both routed Pi-tool arguments through the parse-time literal sublanguage. `4385d86` then edited `tool-calls.md` (and `diagnostics.md`) to make a Pi-tool argument that does not match the tool's input schema "never a parse error … always caught by the runtime AJV check," keeping only `tool-arg-not-literal`/`tool-arg-arity` as parse checks — but did **not** touch `grammar.md` (confirmed: `git show 4385d86 --stat` lists no `grammar.md` change). That asymmetric edit severed the prior consistency, leaving `grammar.md`'s blanket parse-time field-presence rule applying to the Pi-tool `BareObjectLit` position that `tool-calls.md` had moved to runtime. The durable fix addresses the interaction (reconciling `grammar.md`'s field rule with the runtime-only Pi-tool arm) rather than reverting either commit.

## Solution Space

**Shape:** single

### Recommendation

In `grammar.md`'s **Field rules**, stop applying the parse-time field-presence rule uniformly across both `BareObjectLit` sub-positions and split the predicate by where the type comes from:

- **`params:` default and `NamedObjectLit` (loom-schema-backed positions).** The type is a loom-declared type or named variant schema, available at parse time. Retain the existing rule: every declared field of that loom schema MUST be present, omissions are `loom/parse/missing-object-field`, and the is-literal check is parse-time.
- **Pi-tool argument position (externally-owned schema).** State that the type is the Pi tool's registered input schema, which is owned by Pi and obtained only after callee resolution against the load-time callable table (resolution snapshot, `frontmatter.md`). The parse-time checks on this arm are confined to the is-literal check (`loom/parse/tool-arg-not-literal`) and arity (`loom/parse/tool-arg-arity`), neither of which needs a schema. Field presence, optionality, non-object roots, and `additionalProperties` are **not** re-imposed by loom's "every declared field required" model; they are validated at runtime by AJV against the Pi tool's own schema (honouring its `required` subset), surfacing as `Err(CodeToolError { cause: "validation", ... })`. Forward-link [Tool Calls — Argument shape](./tool-calls.md) as the owner of this runtime split.

Make the phase ordering explicit at the point of the field rule: the literal-shape checks (is-literal, arity) run at parse and require no schema; the field/type-matching step requires the resolved callable and its registered input schema and runs at runtime. Edge case the implementer must watch: do not let the `loom/parse/missing-object-field` and (if added elsewhere) surplus-field diagnostics fire against a Pi-tool argument — those diagnostics are scoped to loom-schema-backed positions only.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one follows `grammar.md`'s blanket field rule and emits `loom/parse/missing-object-field` at parse time when a Pi-tool argument omits any property the tool's schema declares — including optional ones — failing the loom to load; the other follows `tool-calls.md`, admits the call at parse time, and lets runtime AJV decide against the schema's actual `required` set. The first wrongly rejects valid calls that omit optional Pi-tool fields and reports the wrong diagnostic at the wrong phase.
