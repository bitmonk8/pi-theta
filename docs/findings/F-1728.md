## Finding

The `respond_repair` prose at `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md:57` uses two spellings of the same mechanism name in a single sentence: the canonical hyphenated **respond-repair** ("the block configures **respond-repair**") and the variant **response-repair** ("the response-repair mechanism specified in [Query — Schema-validation respond-repair]"). They differ by one letter and refer to the same loop. The glossary entry at `docs/spec_topics/glossary.md:21` — which is the entry that pins the canonical spelling — exhibits the identical typo in its own closing sentence ("the response-repair loop above is the only 'repair' mechanism the spec defines").

A `grep` across `docs/` shows these are the only two surviving `response-repair` occurrences; every other site (error model, queryerror variants, query failure-and-repair, PIC, implementation notes, runtime event channel, hard ceilings) is already on the canonical `respond-repair`. The defect is a residue of the `coercion` → `respond_repair` global rename that did not touch the "response-repair mechanism" / "response-repair loop" phrasings.

The 2026-06-04 size-cap split (`f5e89f4`) moved the affected paragraph from `spec_topics/frontmatter.md` into the current `frontmatter-fields-b-and-templates.md` shard verbatim; it did not introduce or fix the defect. `git log -L` on the surviving line confirms `f5e89f4` is the only commit that touches it at the current path.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md:57`, change "the response-repair mechanism specified in" to "the respond-repair mechanism specified in" (drop the `e`). In `docs/spec_topics/glossary.md:21`, change "the response-repair loop above" to "the respond-repair loop above". Both replacements are bare token substitutions — no surrounding wording or linked anchors need to change (the linked anchor `Query — Schema-validation respond-repair` is already on the canonical spelling). After the fix, `grep -rn 'response-repair' docs/` should return zero hits, leaving `respond-repair` as the sole prose spelling.

## Consequence

**Severity:** cosmetic

A naming-glossary that itself uses a variant spelling weakens the term's role as the single source of truth. An implementer searching for "response-repair" will land on this prose and the glossary entry and may keep using the variant in code identifiers / log messages / event field names; an implementer searching for "respond-repair" will see the same sentence and reasonably wonder whether the two are distinct. No behaviour changes either way; the impact is doc-internal cohesion.
