## Finding

`V9l` (session-only degraded-state branch) is the closing leaf for the `session-only-degraded-state.md` coverage-matrix row, and `V9l` is **blocked** until the version-bump editorial-review checklist item (a) resolves the host-prerequisites clause (a) open contradiction (the block is correctly declared on `V9l`'s `Deps.` edge to `V18c`, and documented in §Blocked obligations and the V9 interleave note). Because `V9l` falls inside `H5b`'s `V9a`–`V9p` coverage-producing `Deps.` range, the warn-only canary `H5b` — and therefore the terminal `H6a` release gate that depends on it — is transitively blocked until that open question closes. The DAG itself is correct: the `Deps.` machinery enforces the ordering, and a "branch unreachable" resolution would require re-mapping the `session-only-degraded-state.md` coverage row before `H5b` re-derives a complete coverage-producing set.

The gap is purely one of visibility. The §Release gate section of `docs/plan.md` describes the chain as `H5b → H6a` ("`H6a` sequences immediately after the canary (its **Deps** name `H5b`)") and says nothing about the entire release gate being gated behind a blocked-obligation resolution. A reader working from §Release gate would not learn that loom 1.0's release gate cannot complete until clause (a) is resolved, nor that a "branch unreachable" outcome demands a coverage-matrix edit first. The information is discoverable only by cross-walking §Blocked obligations, the V9 interleave note, `H5b`'s `Deps.` field, and the coverage matrix.

Because `V9l` already sits inside `H5b`'s `V9a`–`V9p` coverage-producing `Deps.` range, that same commit transitively gated the `H5b → H6a` release-gate chain on `V9l`'s resolution — but it did not add a corresponding cross-reference to §Release gate. §Release gate was last substantively edited by `ea6b1da` (2026-06-11), one day before this change, and no later commit to `docs/plan.md` has addressed its silence on the gating. The visibility gap has therefore been present since `55ed974`.

## Solution Space

**Shape:** single

### Recommendation

Add a note to `docs/plan.md` §Release gate stating that `H6a` (and the `H5b` canary it follows) is transitively gated on `V9l`'s host-prerequisites clause (a) resolution, because `V9l` is a member of `H5b`'s coverage-producing `Deps.` set, and that a "branch unreachable" resolution requires re-mapping the `session-only-degraded-state.md` coverage-matrix row before `H5b` re-derives a complete coverage-producing set. Cross-reference §Blocked obligations (`#blocked-obligations`) so the existing block record is the single source of the resolution detail rather than restating it. Keep the note navigation-only: do not alter any `Deps.` field, the coverage matrix, or the spec — the dependency edges are already correct and this finding adds only the missing cross-reference at §Release gate.

## Consequence

**Severity:** advisory

A reader of §Release gate cannot see that the entire loom 1.0 release gate is transitively gated on `V9l`'s clause-(a) resolution, nor that a "branch unreachable" outcome forces a coverage-matrix re-map before `H5b` can re-derive a complete coverage-producing set. The dependency machinery still enforces correct ordering, so no implementer goes off-script; the cost is a navigation gap that obscures a release-blocking precondition.
