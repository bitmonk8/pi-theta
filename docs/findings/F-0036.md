## Finding

The spec pins, at [`host-prerequisites.md` §"Extension entry point"](../../../docs/spec_topics/pi-integration-contract/host-prerequisites.md), a concrete discovery contract: *"A single Pi extension module (`extensions/index.ts`, the file Pi auto-discovers from `package.json#pi.extensions = ["./extensions"]`) exporting the standard `default function (pi: ExtensionAPI)` factory."* Two halves of that contract have no closing leaf in the plan:

- **The manifest discovery entry.** `H1a`'s `Adds.` exhaustively enumerates the `package.json` surface it owns — `peerDependencies` (the four `@earendil-works/pi-*` packages + `typebox`), `devDependencies`, runtime `dependencies`, `engines.node`, and the `npm` scripts — but never declares the `pi.extensions` key. Pi cannot discover the extension at all without `package.json#pi.extensions = ["./extensions"]`, yet no leaf populates it.
- **The factory file path.** `H4a`'s `Adds.` describes the factory shape in detail (the `default function (pi: ExtensionAPI)` factory, its never-throw boundary, the per-call `try`/`catch` registration discipline) but never pins the file it lives in. The spec requires that file to be `extensions/index.ts` (the path `package.json#pi.extensions` resolves to); `H4a` leaves the factory's on-disk location to implementer invention.

A grep across `docs/plan_topics/` finds no occurrence of `pi.extensions` or `extensions/index` in any leaf. The repository currently carries only `extensions/.gitkeep`, so neither half exists yet on disk. The shard gates and the end-to-end harness do not catch the gap because the harness loads the factory by importing it directly rather than through Pi's package-discovery path — every automated gate stays green while a build that a real Pi host cannot actually discover ships silently.

## Solution Space

**Shape:** multiple

The finding carries two coordinated obligations that land in two different leaves and cannot be resolved by a single leaf edit. Both are required; the options below decompose the fix by obligation rather than offering alternatives.

### Option A — H1a owns the `pi.extensions` manifest discovery entry

**Approach.** Add the `pi.extensions` key to the `package.json` surface `H1a`'s `Adds.` enumerates, with value `["./extensions"]`, citing `host-prerequisites.md` §"Extension entry point". Add a build-time literal-read Tests bullet asserting `package.json#pi.extensions` equals `["./extensions"]`, on the same footing as `H1a`'s existing `engines.node` / `peerDependencies` literal-read assertions.

**Plan edits.** `H1a`'s `Adds.` `package.json` field list gains `pi.extensions = ["./extensions"]` with the spec citation; `H1a`'s `Tests.` gains a `Convention:` literal-read assertion for the key and value.

**Spec edits.** None — the contract is already in `host-prerequisites.md` §"Extension entry point"; H1a cites it.

**Pros.** Pins the discovery directory at the scaffold leaf, where every other `package.json` field is already owned and gated; the literal-read assertion fails `H1a`'s own `npm test` if the key is dropped or mis-valued.

**Cons.** None material.

**Risks.** The manifest entry alone is insufficient — Pi resolves `["./extensions"]` to a directory and still needs `extensions/index.ts` present (Option B); landing A without B yields a manifest that points at an empty directory.

### Option B — H4a pins the factory file to `extensions/index.ts`

**Approach.** State in `H4a`'s `Adds.` that the factory file is `extensions/index.ts` (the file Pi auto-discovers from `package.json#pi.extensions = ["./extensions"]`), citing `host-prerequisites.md` §"Extension entry point".

**Plan edits.** `H4a`'s `Adds.` factory-entry-point sentence names the on-disk file `extensions/index.ts` and cross-references the `pi.extensions` directory entry that resolves to it.

**Spec edits.** None — same citation target as Option A.

**Pros.** Co-locates the factory file path with the factory shape `H4a` already owns; closes the half of the contract `H1a` cannot reach (the file's existence and location).

**Cons.** None material.

**Risks.** If the separate "H4a — too large" finding splits `H4a`, the factory-path pin must follow the reduced factory-shell leaf, not the governance/self-check leaf.

### Recommendation

Resolve **Option A first, then Option B.** The `pi.extensions = ["./extensions"]` manifest entry is the scope-bounding edit — it fixes the directory the factory file must live in — so pinning `extensions/index.ts` in `H4a` lands against a stable directory contract. Both halves cite the same `host-prerequisites.md` §"Extension entry point" anchor rather than restating the literal contract. Watch that the `H1a` value (`["./extensions"]`, a directory) and the `H4a` file path (`extensions/index.ts`, inside that directory) agree: Pi resolves the directory entry to the `index.ts` within it, so a mismatch (e.g. naming the manifest entry as a file path) breaks discovery.

## Consequence

**Severity:** correctness

A spec rule (the extension-discovery contract) has no closing leaf, so the implementer builds `H1a`'s `package.json` without the `pi.extensions` key and authors `H4a`'s factory at an unconstrained path. The resulting artifact type-checks, passes every shard gate, and passes the end-to-end harness — yet a real Pi host cannot auto-discover the extension, so loom never loads outside the test harness. The defect is invisible until someone runs the built extension against a live Pi install.
