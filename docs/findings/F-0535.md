## Finding

`V9h`'s **Adds** field lists "the four-arm routing" as one element of the unknown-reason partition, with no antecedent naming which four arms are meant. The term is defined in the spec the leaf cites: `unknown-reason-rule.md`'s *Lookup-failure-to-discriminator routing* clause pins exactly **four arms** that exhaust the snapshot-side failure space — (1) `SDK_SURFACE_INVENTORY` resolves to `undefined` → `"missing-entry"`; (2) the iteration primitive returns no matching entry → `"missing-entry"`; (3) a matching entry whose `.literals` access (or a per-entry `.kind`/`.path` read) throws → `"throw:<String(error)>"`; (4) the matching entry's `.literals` field is structurally invalid → `"literals-shape-invalid"`. This is a `details.failure`-discriminator routing over snapshot-read failure modes, and `V9h` owns all four of them (its second **Tests** bullet exercises the discriminator literals and the four `"literals-shape-invalid"` sub-cases).

The defect is that the unqualified term sits one sentence away from "The session-only degraded-state branch for `new`/`resume`/`fork` is split out to `V9l`". A reader scanning the **Adds** field naturally reconciles "four-arm routing" with the adjacent session-reason prose and concludes the four arms are a routing over the `{quit, reload, new, resume, fork}` closed set, of which the fourth (the new/resume/fork degraded branch) was moved to `V9l` — i.e. that `V9h` now owns only three arms. That reading is wrong: the four arms are the snapshot-failure discriminator arms, all owned by `V9h`, and have nothing to do with the session-reason partition or the `V9l` split.

The remedy is to qualify the term at its one occurrence so it cannot be read as a session-reason routing. The leaf's **Tests** bullets already pin the actual snapshot-failure behaviour concretely, so the ambiguity is confined to the summary parenthetical.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9h-unknown-reason-rule.md`, qualify the **Adds** term "the four-arm routing" so it unambiguously denotes the snapshot-lookup-failure-to-`details.failure`-discriminator routing rather than a routing over session reasons. Replace the bare phrase with one that names what the four arms route, e.g. "the four-arm snapshot-lookup-failure → `details.failure` discriminator routing (per `unknown-reason-rule.md` *Lookup-failure-to-discriminator routing*)". This keeps the term aligned with the four arms the spec clause pins and removes the collision with the adjacent sentence splitting the `new`/`resume`/`fork` degraded branch to `V9l`. No change to `V9h`'s **Tests**, **Deps**, or **Ships when** is required; the **Tests** bullets already pin the four arms' observable behaviour.

## Consequence

**Severity:** advisory

A reader scanning `V9h`'s **Adds** can misread "four-arm routing" as a routing over the session-shutdown reason set and conclude `V9h` owns only three arms, mis-attributing the fourth to `V9l` (the exact misreading the source finding made). Because `V9h`'s **Tests** bullets pin the snapshot-failure discriminator behaviour concretely, no implementation divergence follows — the cost is a misleading index/scan-level summary, not a divergent leaf.
