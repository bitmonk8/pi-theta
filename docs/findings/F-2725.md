## Finding

The *REQ-ID discipline* convention (`conventions.md`) introduces the third closing-gate surface once as "the un-anchored-obligation class [`governance.md` GOV-22] names," then refers to the identical class with the compound term "un-anchored normative MUST/MUST-NOT" everywhere else in the same paragraph ("the closing gate treats an un-anchored normative MUST/MUST-NOT with no enumerated closing leaf as a CI failure", "the un-anchored normative-MUST text-scan arm", etc.). The dominant term "un-anchored normative MUST/MUST-NOT" is also the spelling carried by the leaves that mechanise the surface — `H5e` (Adds: "the un-anchored normative MUST/MUST-NOT obligations"), `H5b`, and `H6a` all read this way.

The lone "un-anchored-obligation class" wording is the outlier. A reader scanning the section for the class definition meets two names for one concept with no statement that they are synonyms. (The compounds "un-anchored-obligation residue" in `coverage-matrix.md` and "un-anchored-obligation surface" in `H5e`'s Convention line are distinct noun phrases — *residue*, *surface* — and are not in scope here; the defect is the bare class label inside the *REQ-ID discipline* section.)

## Consequence

**Severity:** cosmetic

A contributor reads two labels for one closing-gate class within a single paragraph. The closing gate is mechanical (`H5e` text-scan + `coverage-matrix.md` enumeration), so no behaviour or gate outcome changes; the cost is reader friction only.

## Issue introduction

**Verdict:** present-since-feature-introduction
**Introducing commits:** `0603eb4` (2026-06-10, "pi-loom plan: resolve \"un-anchored normative MUSTs invisible to closing gate\"")
**History:** The plan corpus is git-tracked. `git log -S 'un-anchored-obligation class'` and `git log -S 'un-anchored normative MUST'` over `docs/plan_topics/conventions.md` both bottom out at `0603eb4`, the commit that added the un-anchored-MUST third closing-gate surface to the *REQ-ID discipline* section. `git show 0603eb4:docs/plan_topics/conventions.md` confirms both spellings ("un-anchored-obligation class" ×1 and "un-anchored normative MUST" ×1) were present in that same introducing edit, so the inconsistency is co-extensive with the feature — it was never a regression introduced by a later edit.

## Solution Space

**Shape:** single

### Recommendation

In `conventions.md`'s *REQ-ID discipline* convention, change the single occurrence of "the un-anchored-obligation class" so the class is named with the section's dominant term "un-anchored normative MUST/MUST-NOT". The phrase reads:

> … a normative MUST/MUST-NOT on a non-narrative `spec_topics/**` page that carries neither a numbered `PREFIX-N` REQ-ID nor a `loom/...` diagnostics-registry code is the un-anchored-obligation class [`governance.md` GOV-22](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-22) names.

Replace `the un-anchored-obligation class` with `the un-anchored normative MUST/MUST-NOT class` (the rest of the sentence, including the GOV-22 anchor, is unchanged). After the edit every reference to this class inside the section uses one term, matching `H5e`/`H5b`/`H6a`.

Leave `coverage-matrix.md`'s "un-anchored-obligation residue" and `H5e`'s "un-anchored-obligation surface" untouched — those are distinct noun phrases, not the class label this finding concerns.
