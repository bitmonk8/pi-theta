## Finding

`V15e` (hot-reload static-resolution cache eviction) requires the three path identities a cache drop matches on â€” the watcher-delivered changed-file path (`V9b`), the static-resolution per-pass parse-cache key (`V15a`), and the **`.warp` import-edge-graph node identity** (`V15c`) â€” to all be compared in "the single canonical `realpath`-then-forward-slash path form defined under invocation.md â€” Resolution". That `realpath`-then-forward-slash form is, per invocation.md, scoped specifically to `invoke(...)` `.loom` paths and `.loom` `tools:` entries: the resolved callee path is run through `FileSystem.realpath` (full transitive symlink resolution, byte-stable canonical output) before discovery-root containment.

`V15c`'s `.warp` import resolution does not produce that form. Per imports.md IMP-1 and the `Resolver` interface, `.warp` resolution is a relative `path.join`-style computation (`resolve(spec, fromFile)` joins `spec` against the directory of `fromFile`) followed by a **byte-for-byte** final-segment match taken against the bytes `FileSystem.readdir` returns â€” explicitly with no `realpath` / symlink-normalisation step, and explicitly not folded against any Unicode-normalised form. The spec deliberately keeps `.warp` resolution byte-exact-without-realpath ("The byte-for-byte rule holds on every host regardless of the host filesystem's case-equivalence model"). So the `.warp` import-edge-graph node identity `V15c` constructs is in the imports.md relative-join form, not the invoke-side `realpath`-then-forward-slash form `V15e` assumes.

On a host where the two forms diverge â€” a symlinked path, or a case-variant entry on a case-insensitive host â€” the watcher-delivered changed-file path (canonicalised by `V15e` to the `realpath` form) will not match the import-edge-graph node identity (`V15c`'s imports.md form), so the transitive-importer walk fails to identify importers of the changed `.warp` file and their stale parse-cache entries survive the `LoomRegistry` swap. That silent eviction miss is the exact stale-cache-serve failure `V15e` claims to prevent. No leaf currently owns reconciling the two path-identity forms, so an implementer following `V15e` as written would build an eviction that under-evicts on precisely the non-canonical hosts the leaf's own test vector targets.

The immediately prior revision (`a3dd2f0`, 2026-06-16) referenced "the `.warp` import-edge graph (`V15c`)" with no canonical-form claim, so the form mismatch did not exist before `b63e915`. `V15c`'s own history (`git log --follow docs/plan_topics/V15c-imports.md`: `c6a664e` â†’ `2316cac` â†’ `f930314` â†’ `6b7c7e8`) never introduced a `realpath` step; its `.warp` resolution has always tracked imports.md IMP-1's byte-for-byte-against-`readdir` rule. The introducing commit was a single-identity-canonicalisation fix that over-reached by binding `V15c`'s node identity to a form `V15c` does not (and per IMP-1 should not) produce.

## Solution Space

### Recommendation

Keep `V15c`'s `.warp` resolution exactly as imports.md IMP-1 specifies â€” relative `path.join`-style resolution plus a byte-for-byte final-segment match against `FileSystem.readdir` output, with no `realpath` step and no Unicode normalisation. Make `V15e` own an explicit re-canonicalisation that maps each import-edge-graph node's imports.md-form identity onto the form the changed-file path is compared in, before the transitive-importer match runs.

Plan edits:
- In `docs/plan_topics/V15e-hot-reload-cache-eviction.md` (Adds): replace the clause asserting the `.warp` import-edge-graph node identity (`V15c`) is in the canonical `realpath`-then-forward-slash form. State instead that `V15c`'s node identity is in imports.md's relative-resolution / byte-for-byte form (IMP-1), and that `V15e` re-canonicalises it to the eviction-comparison form before matching. Name `V15e` as the owner of that mapping. Also state what form `V15a`'s parse-cache key for reachable `.warp` files is in, so all three identities in the three-way match are well-defined.
- In `docs/plan_topics/V15e-hot-reload-cache-eviction.md` (Tests / Ships when) and `docs/plan_topics/V15e-T-hot-reload-cache-eviction.md` (Tests): extend the non-canonical-form vector so it exercises a changed `.warp` file whose importer's node identity, in `V15c`'s native form, differs from the changed-file path's `realpath` form, asserting the importer is still found and its stale parse-cache entry dropped.

Edge case the implementer must handle: a deletion or rename of a `.warp` file (the `unlink` arm) cannot be `realpath`-canonicalised after the fact, because the path no longer exists to resolve. The mapping must match against the stored node identity rather than re-resolving the vanished path.

Do not add a `realpath` step to `.warp` resolution in `V15c` or `V15a`: that would contradict IMP-1's byte-for-byte-against-`readdir` rule and regress the deliberate cross-host case-sensitivity guarantee. No spec edit is required.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge: one reads `V15e` literally and assumes `V15c` already yields a `realpath`-canonicalised node identity (it does not), and ships an eviction whose transitive-importer match silently misses on symlinked or case-variant hosts; another notices the mismatch and invents an ad-hoc reconciliation with no named owner. The first produces the exact stale-parse-cache serve `V15e` exists to prevent, surfacing only on hot reload on those hosts.
