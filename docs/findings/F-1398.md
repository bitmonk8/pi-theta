## Problem

The **Transient toasts (auxiliary)** paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` enumerates three bookkeeping failure modes the runtime surfaces via `ctx.ui.notify(message, "error")` — the chokidar watcher itself throwing, an unrecoverable settings I/O exception no `loom/load/settings-*` code covers, and an internal extension invariant violation — and pins the surfacing sink, the persistent-channel ban, and the inner-throw swallow rule. It does not pin the post-failure disposition of the subsystem each toast surfaces: for each of the three arms the spec is silent on the observable runtime state after the toast fires (watcher re-armed vs torn down; settings cache retained vs abandoned; registry live vs drained vs tagged degraded) and on whether any accompanying `LoomRegistry` drain-state tag write happens. Two implementers diverge on each arm, and the divergence is conformance-test-observable — e.g. whether the registry is dispatchable after a watcher-throw toast.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `ac8c3af` (2026-05-04, `pi-loom spec: resolve "LoadExtensionsResult.errors is the wrong delivery channel for loom diagnostics"`)
**History:** The **Transient toasts (auxiliary)** paragraph was introduced at `ac8c3af` in `spec_topics/diagnostics.md` as part of splitting loom's delivery channels into "persistent" and "auxiliary" surfaces. The paragraph as originally written named the three bookkeeping failure modes, pinned `ctx.ui.notify("error")` as the surfacing sink, and stated the swallow-on-primary-sink rule — but said nothing about the subsystem's post-failure state. A later move (`f5e89f4`, 2026-06-04, the 100 KB / 30 KB shard split) relocated the paragraph verbatim into `spec_topics/diagnostics/diagnostic-shape.md`, and one further edit (`32d2b76`, 2026-06-05, `pi-loom spec: resolve "No behaviour defined when ctx.ui.notify itself throws"`) added the inner-throw swallow rule for a throwing `ctx.ui.notify`. None of those three edits touched the post-failure disposition gap — the paragraph has carried it from the inception commit through the present.

## Solution approach

Clarify the **Transient toasts (auxiliary)** paragraph in `docs/spec_topics/diagnostics/diagnostic-shape.md` to pin, for each of the three named failure modes (chokidar-watcher throw, unrecoverable settings I/O exception, internal invariant violation), the runtime's post-failure subsystem state and whether the toast is accompanied by a `LoomRegistry` drain-state tag write. For any arm that leaves the runtime needing recovery-by-reload, route it through the existing `LoomRegistry.markRuntimeDegraded()` seam so subsequent slash dispatches hit arm (c) of `readDrainState`'s three-arm enumeration; cross-reference [PIC's drain-state contract](../pi-integration-contract/drain-state-contract.md#loom-registry-drain-state-contract) for the setter and the degraded-state slash-handler arm.

## Solution constraints

- Out of scope: the toast's user-visible message-content gap, owned by the same-cluster sibling finding — this clause pins post-failure runtime state, not the displayed message text.
