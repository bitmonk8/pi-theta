## Problem

The `tools:` frontmatter field page's `#resolution-snapshot` subsection (in `frontmatter-fields-b-and-templates.md`) carries runtime tool-call-dispatch and error-routing obligations — mid-run unregistration behaviour, `unknown_tool` reachability, file-watcher schema-cache rebuild, and hot-reload-orphan error routing into `CodeToolError` / `loom/runtime/internal-error` / runtime panics — that belong to the runtime-dispatch surface owned by `tool-calls.md`. `tool-calls.md` already defines the `CodeToolError` cause enum (including the `unknown_tool` reachability fact) and cross-references back to the field page, so that reachability statement is duplicated across both pages and can drift if either is edited. The load-time half of the subsection (resolution produces a frozen per-loom table of `{ post-rename name → resolved callable }`, dispatched through the held reference) is a genuine `tools:` field semantic and is correctly placed.

## Issue introduction
**Verdict:** single-commit
**Introducing commits:** 7ce0ffa — pi-loom spec: resolve "Tool registry mutation during execution and model-driven parallel tool calls" (2026-05-04, Thomas Andersen); 503221b — pi-loom spec: resolve "Mid-loom Pi-extension hot-reload held closure outcome" (2026-06-07, Thomas Andersen)
**History:** The "Resolution snapshot" subsection — including the runtime-dispatch consequence bullets (held-reference dispatch, mid-run unregistration, `unknown_tool` reachability, file-watcher rebuild) — was authored whole into the then-single `spec_topics/frontmatter.md` by 7ce0ffa, the same commit that rewrote `tool-calls.md` to dispatch through the held reference and to cross-reference back to the frontmatter page; the placement of the runtime obligations on the field page dates to that commit. 503221b later expanded the hot-reload-orphan bullet from a one-line "out of scope" note into the full CodeToolError / `loom/runtime/internal-error` / runtime-panics error-routing prose, deepening the overreach. The f5e89f4 size-cap split (2026-06-04) carried the subsection verbatim into `frontmatter-fields-b-and-templates.md`; it relocated the file but did not change the placement decision.

## Solution approach

Move the four runtime-dispatch consequence bullets — mid-run unregistration, `unknown_tool` reachability, file-watcher rebuild, and hot-reload-orphan error routing — out of `#resolution-snapshot` in `frontmatter-fields-b-and-templates.md` into the dispatch / off-surface outcome-routing area of `tool-calls.md`, and add a forward-link from the retained load-time snapshot definition to where those consequences now live. Fold the relocated `unknown_tool` reachability bullet into `tool-calls.md`'s existing `CodeToolError` `unknown_tool` description rather than restating it. Retain on the field page only the load-time definition of the resolution snapshot.

## Solution constraints

- Keep the `#resolution-snapshot` anchor in `frontmatter-fields-b-and-templates.md` unchanged — inbound cross-references in `tool-registration-lifetime.md`, `subagent.md`, `slash-invocation.md`, and `implementation-notes.md` consume it.
- After the move, the `unknown_tool` reachability fact MUST exist on only one page — no cross-page duplication between `frontmatter-fields-b-and-templates.md` and `tool-calls.md`.
