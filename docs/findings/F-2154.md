## Finding

The `tools:` frontmatter field page hosts a "Resolution snapshot" subsection (anchor `#resolution-snapshot`, in `frontmatter-fields-b-and-templates.md`) that, beyond defining what load-time resolution produces, carries a block of runtime tool-call-dispatch normative obligations: (1) every call dispatches through the captured `ToolDefinition` strong reference and the runtime never re-queries Pi's tool registry by name during execution; (2) unregistering a Pi tool mid-run does not affect in-flight invocations; (3) the `unknown_tool` `CodeToolError` cause is reachable only via the file-watcher reload path; (4) file-watcher rebuilds invalidate the schema cache and rebuild `tools:` tables on the next invocation; and (5) a hot-reloaded extension leaves the captured `execute` closure orphaned, with full error-routing of the orphaned dispatch into `Err(CodeToolError { cause: "execution" })`, `loom/runtime/internal-error`, and the runtime-panics surface.

These are runtime tool-call-dispatch and error-model rules, not field-contract rules. The page that owns that surface is `tool-calls.md` — which already defines the `CodeToolError` cause enum (including the `unknown_tool` reachability fact) and the off-surface outcome routing, and which already cross-references *back* to the frontmatter page for the held-reference behaviour. The result is split ownership: consequence bullet (3) duplicates the `unknown_tool` reachability statement that `tool-calls.md` also makes (a drift risk if either is edited), and bullet (5) plants deep error-routing prose (CodeToolError / `loom/runtime/internal-error` / runtime panics) on a frontmatter field page rather than on the error-routing page it references.

The load-time half of the subsection — "resolution produces a frozen per-loom table of `{ post-rename name → resolved callable }` entries" — is intrinsically a `tools:` field semantic and belongs with the field. The defect is confined to the runtime-dispatch consequences co-located with it.

## Solution Space

### Recommendation

Keep the load-time snapshot definition on the `tools:` field page and move the runtime-dispatch consequences to `tool-calls.md`.

In `frontmatter-fields-b-and-templates.md`, retain in the "Resolution snapshot" subsection only the load-time definition — "resolution produces a frozen per-loom table of `{ post-rename name → resolved callable }` entries; all dispatches during that loom's execution go through the held reference; the runtime never re-queries Pi's registry by name during execution" — and keep its `#resolution-snapshot` anchor unchanged so the inbound cross-references in `tool-registration-lifetime.md`, `subagent.md`, and `slash-invocation.md` stay valid. Cut the four runtime-dispatch consequence bullets from this subsection: mid-run unregistration not affecting in-flight invocations, `unknown_tool` reachability, file-watcher schema-cache invalidation / `tools:`-table rebuild, and the hot-reload-orphan error routing (`Err(CodeToolError { cause: "execution" })`, `loom/runtime/internal-error`, runtime-panics surface). In their place add a one-line pointer to the dispatch-consequences section of `tool-calls.md`, explicit enough that an implementer reading the field page knows dispatch semantics are specified there, not unspecified.

In `tool-calls.md`, paste the relocated consequence bullets into the dispatch / off-surface outcome-routing area, where the hot-reload-orphan error routing naturally joins the existing off-surface outcome routing. When relocating the `unknown_tool` reachability bullet, fold it into `tool-calls.md`'s existing `CodeToolError` `unknown_tool` description rather than restating it, so the duplication is removed rather than merely moved.

## Consequence

**Severity:** advisory

A reader consulting the frontmatter field page encounters runtime-dispatch and error-routing obligations that the runtime-dispatch page also owns, and an implementer or auditor reading `tool-calls.md` to find the dispatch model is bounced to a frontmatter page for it. The duplicated `unknown_tool` reachability statement can drift between the two pages. No runtime behaviour is wrong today (the two pages currently agree), so implementers can still build a working system; the cost is interpretive burden and future divergence risk.
