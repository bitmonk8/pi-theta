## Finding

`V9n` (prompt-mode transport-error mapping) and its paired `V9n-T` give two mutually contradictory sources for the synthesised `TransportError.provider` field, and **both contradict the spec**:

- `V9n`'s **Adds** states the `provider` field "is sourced from `V9j`'s provider-error-mapping surface, not re-derived here". `V9j` is the binder / forced-respond-turn classifier, which runs off-session through pi-ai's `complete()` and keys its `provider` off the loom's own resolved model ŌĆö a different surface from the prompt-mode driven turn `V9n` actually handles.
- The PIC-51 **Tests** bullet (verbatim in both `V9n` and `V9n-T`) instead attributes `provider` to "the loom's resolved `model:` `Model<Api>.api` value (the frontmatter `model:`, or the inherited `ctx.model` when `model:` is absent; resolution owned by V6a)".

The spec is unambiguous and matches neither. PIC-50 (`conversation-drive.md`) pins the prompt-mode `provider` field to the `Api`-shaped `Model<Api>.api` value of **the user session's currently-selected model ŌĆö read from `ctx.model`** ŌĆö and adds, in those exact terms, "**not** the loom's resolved `model:` (which is the loom's own off-session / subagent model and does not govern the user session's provider)". When `ctx.model` is `undefined`, `provider` MUST be the fixed sentinel string `"unknown"`. PIC-51 derives its `provider` "per PIC-50's user-session-model derivation". `queryerror-variants.md` ┬¦`provider` derivation states the same split explicitly: prompt-mode driven-turn failures take the user session's selected-model `.api` (`ctx.model`) with the `"unknown"` sentinel, whereas the loom-resolved-model `.api` applies only to the binder pass, the typed-query forced respond turn, the subagent driven turn, and load/dispatch-time unsupported-provider synthesis.

`V9n` owns PIC-50 and PIC-51 (per `coverage-matrix.md`), whose surfaces ŌĆö the `stopReason: "error"` probe and the synchronous `pi.sendUserMessage` throw ŌĆö are precisely the `ctx.model`-derived prompt-mode paths, not the off-session `complete()` paths `V9j` classifies. The Adds therefore mis-routes `provider` to `V9j`, and the Tests bullet asserts a value (the loom's resolved `model:`) the spec specifically forbids for this surface.

## Solution Space

**Shape:** single

### Recommendation

Reconcile both attributions to the spec's user-session derivation (PIC-50/PIC-51, `queryerror-variants.md` ┬¦`provider` derivation): the prompt-mode `TransportError.provider` field is the `api`-shaped `Model<Api>.api` value of the user session's currently-selected model, read from `ctx.model`, with the fixed sentinel `"unknown"` when `ctx.model` is `undefined`.

- In `V9n` Adds, strike "the synthesised `transport` error's `provider` field is sourced from `V9j`'s provider-error-mapping surface, not re-derived here" and replace it with text to the effect of: "the synthesised `transport` error's `provider` field is the `api`-shaped `Model<Api>.api` value of the user session's currently-selected model read from `ctx.model` (the model the driven user turn routes through, **not** the loom's resolved `model:`), substituting the fixed sentinel `\"unknown\"` when `ctx.model` is `undefined`, per PIC-50's user-session-model derivation."
- In the PIC-51 Tests bullet of **both** `V9n` and `V9n-T`, replace "the `provider` value sourced from the loom's resolved `model:` `Model<Api>.api` value (the frontmatter `model:`, or the inherited `ctx.model` when `model:` is absent; resolution owned by V6a), per conversation-drive.md's resolved-`model:` rule and queryerror-variants.md's `TransportError.provider` derivation" with text to the effect of: "the `provider` value is the user session's selected-model `Model<Api>.api` (`ctx.model`), substituting the fixed sentinel `\"unknown\"` when `ctx.model` is `undefined`, per PIC-50's user-session-model derivation and queryerror-variants.md's `TransportError.provider` derivation". Remove the "resolution owned by V6a" parenthetical and the "loom's resolved `model:`" phrasing entirely.
- Consider adding a `V9n`/`V9n-T` test vector exercising the `ctx.model === undefined` ŌåÆ `provider: "unknown"` sentinel path, since the corrected derivation introduces an observable branch the current bullets do not cover.

Once `provider` no longer flows from `V9j`, re-evaluate whether `V9j` remains a required `Deps` edge on `V9n` and `V9n-T`; if no other `V9n` obligation depends on `V9j`, strike it from both `Deps` lines. `V6a` is not ŌĆö and need not become ŌĆö a `V9n-T` dependency under this derivation, since the user-session provider comes from `ctx.model`, not from any loom-side `model:` resolution.

## Consequence

**Severity:** correctness

An implementer reading `V9n`'s Adds wires `provider` from `V9j`'s off-session classifier, while the PIC-51 test gates `provider` against the loom's resolved `model:` / V6a resolution ŌĆö two divergent, both spec-wrong derivations in one leaf pair. The test passes against an implementation that misattributes the user-session provider (and never exercises the `"unknown"` sentinel path), so the shipped prompt-mode `TransportError.provider` silently violates PIC-50 while CI stays green.
