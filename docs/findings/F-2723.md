## Finding

The `H5e` un-anchored-MUST gate's **un-rowed-page recogniser** reddens CI on any `spec_topics/**` page absent from the [REQ-ID prefix table](docs/spec_topics/governance/req-id-prefix-table-active-a.md) altogether, treating it as an un-rowed-page residue defect — *"unless it is a [GOV-24] table-of-contents hub stub — a subtree TOC file that deliberately carries no row and no REQ-ID anchors — which is excluded"* (`conventions.md` §REQ-ID discipline; restated in `H5e` Adds/Tests/Ships-when).

The two properties the rule supplies to identify an excluded hub stub — *"no row"* and *"no REQ-ID anchors"* — are exactly the properties every in-scope un-rowed page also has. An un-rowed page is in scope *because* it has no prefix-table row, and a page with no registered prefix carries no `PREFIX-N` REQ-ID anchors. So the excluded hub stub and the reddening residue page are indistinguishable by the only mechanical properties the rule names. The remaining discriminators — *"table-of-contents"*, *"subtree TOC file"*, *"deliberately"* — are semantic descriptions, not properties a token/path scan can read. The cited GOV-24 (`req-id-prefix-table-active-b.md`) does not close the gap either: it describes a hub stub as a subtree-TOC file written with a trailing-slash `Page` cell whose stub *"carries no REQ-ID anchors and is not itself a table row,"* but supplies no in-file marker, filename rule, or anchor pattern by which the gate can recognise one.

A faithful implementer of `H5e` therefore has to invent a discriminator. Two reasonable implementers will pick different ones, and at the live-corpus gate (activated by `H6a`) the divergence is observable: an over-broad rule false-fails a legitimate hub stub (blocking the release), while an over-narrow rule false-cleans a genuine un-rowed obligation page (silently letting its content escape the gate the recogniser exists to enforce).

## Consequence

**Severity:** correctness

Two good-faith implementations of the un-rowed-page recogniser diverge: one false-fails a legitimate hub stub at the `H6a` release gate (blocking release), the other false-cleans a real un-rowed obligation page (the un-anchored MUST that page may carry escapes the very gate the recogniser exists to enforce). Because the recogniser binds only against the live corpus at the `H6a` flip, the divergence surfaces at the single most expensive point to remediate.

## Solution Space

**Shape:** multiple

### Option A — In-file hub-stub sentinel marker

**Approach.** Define a canonical in-file sentinel (e.g. an HTML-comment marker such as `<!-- GOV-24 hub stub -->`) that marks a `spec_topics/**` file as a TOC hub stub. The un-rowed-page recogniser excludes an un-rowed page from the residue defect iff the file carries the sentinel; any other un-rowed page reddens.
**Plan edits.** In `conventions.md` §REQ-ID discipline, replace the semantic "subtree TOC file that deliberately carries no row and no REQ-ID anchors" exclusion clause with the sentinel test, naming the exact marker token the recogniser greps for; mirror the same token in `H5e` Adds/Tests/Ships-when and make the seeded GOV-24 hub-stub fixture page carry the sentinel. Update residue-inspection item 5's hub-stub parenthetical to the same sentinel test.
**Spec edits.** Pin the sentinel in GOV-24 as the canonical hub-stub marker so the spec owner writes it when creating a hub stub; add the sentinel to each existing live hub-stub page under `docs/spec_topics/**`.
**Pros.** Fully mechanical via the same token-scan machinery the recogniser already uses; robust to arbitrary hub-stub filenames and directory layouts; unambiguous (a page is a hub stub exactly when it carries the token).
**Cons.** Requires touching every live hub-stub page to add the marker.
**Risks.** A hub stub created without the sentinel false-fails at the `H6a` gate; mitigated by pinning the marker as a GOV-24 authoring obligation.

### Option B — Derive hub-stub paths from subtree-named prefix-table rows

**Approach.** GOV-24 already binds a hub stub to a subtree-named prefix-table row (trailing-slash `Page` cell, e.g. `binder/`, `governance/`). Pin a mechanical path relationship between such a row and its stub file, and have the recogniser exclude exactly the stub files derived from the set of trailing-slash rows; every other un-rowed page reddens.
**Plan edits.** In `conventions.md` §REQ-ID discipline and `H5e`, replace the semantic exclusion with the derived-path rule (the recogniser computes the excluded hub-stub path set from the trailing-slash rows of the prefix table); shape the seeded GOV-24 hub-stub fixture and its fixture prefix table so the fixture stub's path matches the derivation.
**Spec edits.** GOV-24 must pin the exact file-path convention tying a subtree-named row to its hub-stub file (e.g. the file naming the subtree), so the derivation is well-defined.
**Pros.** No per-page marker edits to live hub stubs; reuses the prefix table the recogniser already reads.
**Cons.** Requires GOV-24 to define a uniform hub-stub path convention; brittle if hub stubs do not follow one naming pattern.
**Risks.** A hub stub whose path does not match the pinned convention false-fails at the gate; a non-hub page that happens to match the derived path false-cleans.

### Recommendation

Take **Option A**: pin a canonical in-file hub-stub sentinel and make the recogniser's exclusion turn on the sentinel's presence. It is mechanical through the same `MUST`-token text scan the recogniser already performs, is robust to arbitrary hub-stub paths (Option B depends on a uniform path convention GOV-24 does not currently guarantee), and gives the recogniser a single unambiguous test. Watch two edges: the seeded GOV-24 hub-stub fixture at `H5e` must carry the sentinel so the green-path test exercises the real discriminator rather than an implicit one; and every live hub stub must carry the sentinel before the `H6a` live-corpus activation, or it false-fails at the release gate — pin that as the GOV-24 authoring obligation.
