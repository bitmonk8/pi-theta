## Finding

The diagnostic-code registry (`docs/spec_topics/diagnostics/code-registry-load.md`) defines the per-read package-walk timeout as the full-path code `loom/load/package-read-timeout`. That row also notes the diagnostic carries a separate field `details.kind = "package-read-timeout"` — so the bare string `package-read-timeout` is a legitimate *payload field value*, but it is **not** the emitted code.

Both `V10b` (`docs/plan_topics/V10b-package-discovery.md`) and its tests partner `V10b-T` cite the timeout in bare form, `package-read-timeout`, in every DISC-6 occurrence, while the sibling code `loom/load/discovery-slow` is consistently written full-path in the same bullets. The asymmetry reads as if the two codes have different naming conventions. The bare occurrences are: V10b Tests DISC-6 bullet, V10b Tests DISC-6 (per-read deadline) bullet, V10b Ships-when, V10b-T Tests DISC-6 bullet, and V10b-T Tests DISC-6 (per-read deadline) bullet — five in total across the two leaves.

The diagnostic-code closing gate reconciles each leaf's inline code citation against the registry by exact string match. A citation of bare `package-read-timeout` does not match the registry's `loom/load/package-read-timeout`, so the DISC-6 code is either reported as having no asserting test or an implementer following the bare form emits the wrong code string.

**Verdict:** present-since-inception

**History:** The plan corpus is git-tracked. `git log -G 'package-read-timeout'` over `V10b-package-discovery.md` and `V10b-T-package-discovery.md` shows the bare form was authored in the initial plan build c6a664e, in the DISC-6 Tests bullet and the V10b Ships-when line, both already pairing bare `package-read-timeout` against full-path `loom/load/discovery-slow`. The later commit 750ecbb ("resolve 'V10b per-read-deadline formula variable t is undefined'", 2026-06-12) added the DISC-6 per-read-deadline bullet to both leaves and carried the same bare form forward, reinforcing the inconsistency rather than introducing it. The defect has existed in every authored version of the two leaves.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V10b-package-discovery.md` and `docs/plan_topics/V10b-T-package-discovery.md`, replace each bare `package-read-timeout` that denotes the emitted diagnostic code with the registry-canonical full path `loom/load/package-read-timeout`. The occurrences are:

- V10b Tests, DISC-6 bullet: `… and \`package-read-timeout\` at the per-read deadline` → `… and \`loom/load/package-read-timeout\` at the per-read deadline`.
- V10b Tests, DISC-6 (per-read deadline scales) bullet: `fires the per-read \`package-read-timeout\` at` → `fires the per-read \`loom/load/package-read-timeout\` at`.
- V10b Ships when: `fires \`discovery-slow\`/\`package-read-timeout\` at the limits` → use `loom/load/package-read-timeout` (matching the full-path form for the paired code).
- V10b-T Tests, DISC-6 bullet: same replacement as V10b Tests DISC-6.
- V10b-T Tests, DISC-6 (per-read deadline scales) bullet: same replacement as V10b per-read bullet.

The registry is read-only for this fix — take the exact string from `code-registry-load.md`, do not edit the registry. Do not alter any reference to the payload field `details.kind = "package-read-timeout"`, which is correctly bare and is a distinct concept from the emitted code.

## Consequence

**Severity:** correctness

An implementer reading the bare `package-read-timeout` citations alongside the full-path `loom/load/discovery-slow` cannot tell whether the timeout code is genuinely bare or an abbreviation; two implementers diverge (one emits `package-read-timeout`, one emits `loom/load/package-read-timeout`). A test asserting the bare string fails against the registry-conformant emitted code, and the diagnostic-code closing gate reports DISC-6 as lacking an asserting test.
