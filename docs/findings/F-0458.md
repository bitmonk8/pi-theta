## Finding

`GOV-16` (in [`spec_topics/governance/stable-inline-labels.md`](../../../docs/spec_topics/governance/stable-inline-labels.md#gov-16)) registers three stable-inline-label prefixes: `HC3` (`hard-ceilings.md`), `NOCEIL` (`hard-ceilings.md`), and `SNK` (`slash-invocation.md`). `SNK-a … SNK-k` are the eleven per-`kind` system-note rendering rows under `SLSH-4` in [`spec_topics/slash-invocation.md`](../../../docs/spec_topics/slash-invocation.md) — each row pins the verbatim rendered template for one `QueryError.kind`.

`conventions.md` §REQ-ID discipline requires every GOV-16 inline-label family to close in one of two shapes — **(a)** per-label single-leaf closure (one *Tests* bullet per label, each label cited inline, as `HC3-a` … `HC3-e` do on `V11f`), or **(b)** distributed-seam closure for no-single-leaf families (each label witnessed by a distributed diagnostic-code *Tests* bullet carrying the label as a parenthetical seam annotation, as `NOCEIL-1` … `NOCEIL-4` do across `V6a`/`V4d`/`V4b`/`V15b`) — and pins that "the [`coverage-matrix.md`](../../../docs/plan_topics/coverage-matrix.md) *`CIO` / `HC3` / `NOCEIL` / `CEIL` closure rationale* … owns which family takes which shape." An inline-label family carries no coverage-matrix table row of its own.

`SNK` receives neither treatment. The coverage-matrix closure-rationale prose names `HC3` and `NOCEIL` (and `CIO`/`CEIL`) but never mentions `SNK`, so no closure shape is designated for it. `V12b`'s *Tests* (and its `V12b-T` partner) collapse all eleven per-`kind` rows into a single bundled bullet — `` `SLSH-4`: the per-kind note templates (SNK-a..k) render verbatim … `` — rather than the per-label bullets shape (a) requires. Because an inline-label family is exercised by none of the three closing-gate surfaces (`H5a`/`H5d`/`H5e`), there is no mechanical backstop either: nothing requires or verifies that each per-`kind` rendering carries its own citing test. The result is an asymmetric, undesignated treatment of an obligation that — per the spec's own catch-all (`SNK-k`) — is meant to be total over the `QueryError` union.

## Solution Space

**Shape:** single

### Recommendation

Bring `SNK` into conformance with the per-label single-leaf closure shape (shape (a)), mirroring how `HC3` is handled — `SNK-a … SNK-k` all close in the single slash-dispatch-boundary leaf `V12b`, so distributed-seam closure (shape (b)) does not fit.

Two coordinated edits:

1. In `V12b-top-level-err-chain.md` (and the matching `V12b-T-top-level-err-chain.md`) *Tests*, keep the `SLSH-4` REQ-ID bullet but replace its bundled `(SNK-a..k)` parenthetical with one *Tests* bullet per inline label, each label cited inline exactly as `V11f` cites `HC3-a` … `HC3-e`. Source each row's expected verbatim template from the `SNK-a … SNK-k` table in `slash-invocation.md` (e.g. `` SNK-f: a `cancelled` kind renders "loom `/<name>` cancelled" `` ), and include a bullet for the `SNK-k` catch-all asserting the renderer is total over any unlisted `kind`.

2. In `coverage-matrix.md`, extend the closure-rationale prose to designate `SNK`'s closure shape — per-label single-leaf closure on `V12b`, parallel to the existing `HC3-a … HC3-e close in V11f` sentence — without adding a coverage-matrix table row (inline-label families carry no row of their own per `conventions.md` §REQ-ID discipline).

Watch: do not add a `<new>`-style or `SNK-a … SNK-k` table row — that would contradict the no-row rule and the `H5d`/`H5a` tokenisers would mis-read it. The fix is entirely within plan files; the spec `SNK` table and GOV-16 registration are correct and unedited.

## Consequence

**Severity:** correctness

Each of the eleven per-`kind` system-note renderings (`SNK-a … SNK-k`) is a distinct verbatim-template obligation, yet the single bundled `SLSH-4` bullet means a leaf can ship with only some `kind` rows tested while no gate reddens (inline-label families have no mechanical closing gate). Two reasonable implementers would diverge on how many of the eleven rows need their own citing test, and the catch-all (`SNK-k`) totality property can silently go unverified — the *No silent test skipping* failure mode the plan forbids, applied per-`kind`.
