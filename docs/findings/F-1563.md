## Finding

In `docs/spec_topics/discovery/package-and-settings.md` the `looms.scanPackages` bullet (under "Keys read") closes its rationale with:

> The three `looms.scanPackages*` keys below are operator-tunable in loom 1.0 rather than hardcoded because package counts and walk time vary by install; the defaults are *upper bounds*, not target performance.

The "*upper bounds*, not target performance" framing is correct for the two integer caps (`looms.scanPackagesMaxFiles`, `looms.scanPackagesTimeoutMs`), but `looms.scanPackages` is a boolean on/off switch whose default `true` is not an upper bound on anything â€” it is the enabled state. Sweeping the boolean into "the three keys' defaults are upper bounds" makes the sentence trivially false for one of the three it names, and a literal reading invites confusion about whether `scanPackages` somehow caps something.

The same paragraph reads cleanly if the "upper bounds" half is restricted to the two cap values; the broader operator-tunability rationale ("rather than hardcoded because package counts and walk time vary by install") legitimately covers all three keys and should stay.

The matching prose in Â§"Package discovery" â†’ "Edge cases" already speaks only of "the default caps" / "the bounds are operator-tunable", so it is internally consistent; only the "Keys read" bullet conflates the boolean.

## Solution Space

**Shape:** single

### Recommendation

Restrict the "upper bounds" half of the rationale to the two cap values while keeping the operator-tunability half covering all three keys. Replace the trailing sentence of the `looms.scanPackages` bullet:

> The three `looms.scanPackages*` keys below are operator-tunable in loom 1.0 rather than hardcoded because package counts and walk time vary by install; the defaults are *upper bounds*, not target performance.

with:

> The three `looms.scanPackages*` keys are operator-tunable in loom 1.0 rather than hardcoded because package counts and walk time vary by install. For the two integer caps (`looms.scanPackagesMaxFiles`, `looms.scanPackagesTimeoutMs`) the defaults are *upper bounds*, not target performance: a healthy install completes the walk well under both caps and never trips `loom/load/discovery-slow`.
