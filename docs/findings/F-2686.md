## Finding

The step-2 editorial-review checklist in `version-bump-step2.md` requires the contributor to record, for each presupposition-audit item, an outcome of `pass` / `fail` / `N/A` with a one-line rationale (with `not-performed` for a skipped SHOULD-level audit). `N/A` is given a precise legitimacy condition for exactly one item — item (e), whose `N/A` is defined as "the candidate Pi minor has surfaced an explicit per-session dispatch-lock mechanism" (recorded as `N/A — superseded by <named Pi mechanism>`), and whose **conservative-posture rule** forbids recording `N/A` when the auditor merely cannot demonstrate the property (that must be `fail`). Item (a) carries a separate one-shot authoritative-determination shape.

For every remaining item — (f) through (aq) and item (as) — the only stated constraint on the outcome is "one of `pass` / `fail` / `N/A` with a one-line rationale … when the SHOULD-level audit was performed". Nothing states *when* `N/A` is a legitimate outcome versus when an audit that was performed but could not establish the presupposition must be recorded `fail`. A contributor who performs an audit, is uncertain, and writes `N/A — could not determine` is conformant under the letter of the rule. A real candidate-minor regression on a consequential presupposition — e.g. (aj) `SessionManager.inMemory` transcript non-retention, (s) settings write-back key preservation, (af) `complete()` `onResponse` delivery — can therefore be silently discharged as `N/A`, defeating the editorial gate that is loom 1.0's only enforcement for these unpinned host presuppositions.

Item (ar) is a distinct hole of the opposite kind. It is not a candidate-minor presupposition audit but a MUST-confirm-on-each-bump reconciliation (the step-5 inbound-reference enumeration-delta sweep), and the preamble records it "under the same pass / fail / N/A one-line-rationale shape". But `N/A` has no meaning for an obligation that must be performed on every bump — there is no presupposition that can "fail to apply", so no situation discharges it as not-applicable. Unlike the generic items, item (ar) cannot be repaired by a generic "presupposition does not apply" definition because it has no presupposition; its `N/A` must either be removed or given a bespoke condition.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `52b9101` (generic-`N/A` arm); `1a59ccf` (item-(ar) facet)
**History:** The editorial-review checklist was introduced in `52b9101` (2026-05-11, "T24 — Fork-reason watcher closure …"), whose added preamble already read `MUST record the per-item audit outcome (pass / fail / N/A with one-line rationale)` with four presuppositions and **no** generic `N/A` legitimacy condition. Subsequent commits expanded the SHOULD-level item range to (f)–(as) under that same shape (`aebec4e`, `29f4cba`, `fdd8e76`, and others), and item (e) alone later acquired a specific `N/A` definition plus the conservative-posture rule (`c5f97f6`, "Item (e) recovery over-prescribes mutex; N/A and recording conventions buried") — the generic items never did. The item-(ar) facet entered with `1a59ccf` (F-2264, "demote build-time inbound-reference audit gate to advisory/post-1.0, wire 1.0 enforcement via editorial-checklist item (ar)"), which added item (ar) as a MUST-each-bump reconciliation but recorded it under the inherited `pass / fail / N/A` shape, carrying the undefined `N/A` onto an obligation that has no presupposition to disapply.

## Solution Space

**Shape:** single

### Recommendation

Add a generic `N/A` legitimacy condition to the recording-outcome rule covering items (f)–(aq) and (as): `N/A` is permitted only when the audited presupposition demonstrably does not apply to the candidate minor for a named reason (mirroring item (e)'s `N/A — superseded by <named Pi mechanism>` form, where the rationale names the reason); an audit that was performed but could not establish the presupposition MUST be recorded `fail`, not `N/A` (generalising item (e)'s conservative-posture rule to all SHOULD-level items); a skipped SHOULD-level audit remains `not-performed`.

In the same edit, remove `N/A` from item (ar)'s recorded-outcome shape (record `pass` / `fail` only), since item (ar) is a mandatory per-bump reconciliation with no presupposition that can fail to apply; if an `N/A` outcome is nonetheless wanted, state its single explicit condition rather than inheriting the generic shape.

Edge cases for the implementer: item (e)'s existing bespoke `N/A` definition and conservative-posture rule, and item (a)'s one-shot authoritative-determination shape, must remain explicit carve-outs that the generic rule does not override. The natural home for the generic `N/A` definition is the `Outcome recording conventions` subsection the preamble already points "below" at; that subsection does not yet exist in the file, so placement depends on it being added.

## Consequence

**Severity:** correctness

The editorial checklist is loom 1.0's sole enforcement for roughly forty unpinned host/provider presuppositions. With `N/A` undefined for the SHOULD-level items, a performed-but-uncertain audit can be recorded `N/A` rather than `fail`, silently passing a real candidate-minor regression through the gate; two contributors confronted with the same uncertainty will diverge on whether to record `N/A` or `fail`. Item (ar)'s spurious `N/A` lets a mandatory per-bump reconciliation be marked not-applicable.
