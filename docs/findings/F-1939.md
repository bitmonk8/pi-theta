## Finding

`V15b` owns the hard-ceiling-#1 invoke-depth bound (INV-4). Its Adds declares the per-chain counter "counting direct `invoke`, `.loom`-via-`tools:`, and cross-file `.warp fn` frames", and both `V15b` and `V15b-T` author an INV-4 boundary-chain test that requires a single chain mixing all three countable frame classes — including "one cross-file `.warp fn` frame (caller and callee in different source files, per `invocation.md` §INV-4)" — to reach the 32-frame boundary and fire `loom/runtime/invoke-depth-exceeded` at 33 > 32.

For a cross-file `.warp fn` frame to be counted, two runtime capabilities must exist: (1) a `.warp fn` call must actually be *evaluated* — executing the imported function body — and (2) the call site must classify the call as cross-file vs intra-file by the callee's *residence* (its declaration `.warp` file, seeing through re-exports and `as`-aliased imports per `invocation.md` §INV-4), then push a frame that increments the shared per-chain counter only when the call crosses a file boundary. Neither capability is owned by any leaf reachable as built by `V15b`. `V15b`'s Deps are `V15a`, `V15c`, `V16a`: `V15a` is the `invoke(...)` core (it does not evaluate `fn` calls), `V15c` binds imported `.warp` symbols and raises import diagnostics but explicitly defers nothing to a call-evaluation path, and `V16a` is the cross-ceiling arbitrator. The leaves that touch `fn` semantics — `V3a` (expression evaluator, identifier resolution `local > fn > import > callable`) and `V3d` (functions, return, the function-result seam) — describe declaration and return-value semantics but neither names a call-frame hook, the per-chain counter, nor residence classification, and `V15b` depends on neither directly.

The net effect: INV-4's cross-file `.warp fn` arm cannot be constructed. The `V15b-T` test bullet that mixes all three frame classes has no runtime substrate to exercise for the `.warp fn` class, and `V15b`'s Adds claims to *count* a frame class that no leaf produces or instruments. (A prior fix added `V15c` to `V15b`'s Deps to cover import *binding*, but binding a symbol is not the same as evaluating a call, pushing a frame, or determining residence — the deeper producer gap remained.)

## Solution Space

### Recommendation

Keep the per-chain counter and all three countable-frame classes in `V15b`, which already owns INV-4 and the counter. Instrument the cross-file `.warp fn` call site so it increments the shared per-chain counter, classifying cross-file vs intra-file by the callee's residence — its declaration `.warp` file per `invocation.md` §INV-4.

In `docs/plan_topics/V15b-invoke-depth-cycle.md` Adds, state that `V15b` instruments the `.warp fn` call site with the per-chain counter and determines a fn's residence as its declaration `.warp` file per `invocation.md` §INV-4. Residence is unaffected by re-exports / `as`-aliased imports: a re-exported or `as`-aliased fn imported through an intermediary still counts by where it is declared, not where re-exported. An intra-file `fn` call (caller and callee in the same `.warp` file) is not a countable frame. The counter increments before the child frame begins executing.

Add to `V15b.Deps` and `V15b-T.Deps` the leaves that own `.warp fn` call evaluation — `V3d` (function-result seam) and `V3a` (call-expression evaluation) — so the call site is directly reachable, plus `V15i` (export visibility / re-exports) so residence can see through aliases. (`V15c` is already a Dep.) The instrumentation point must be identified within the call-evaluation surface of `V3a`/`V3d`, which do not currently name a call-frame hook.

No spec edits.

## Consequence

**Severity:** blocking

`V15b` cannot be completed as written: its INV-4 boundary chain requires a real cross-file `.warp fn` frame, and no leaf builds the `.warp fn` call-execution path that pushes a per-chain countable frame or classifies a fn's residence. An implementer either invents an out-of-scope call-execution mechanism inside `V15b` (diverging from the V3 fn-call surface) or quietly omits the `.warp fn` arm, leaving one of INV-4's three countable frame classes unverified at the depth ceiling; the INV-4 citing-test coverage gate then passes without exercising the `.warp fn` class.
