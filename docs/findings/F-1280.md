## Problem

`invocation.md` INV-4 (`id="inv-4"`) caps an `invoke` chain at depth 32 and counts cross-file `.warp` `fn` calls as one of three frame classes against that cap, but neither `functions.md` nor `imports.md` — the pages that introduce `.warp` `fn` declarations and `.warp` imports — signals that a cross-file `fn` call is a countable frame. `functions.md` describes `fn` declarations and cross-links Invocation for invoke-cycle semantics, yet never states that a `fn` call across a source-file boundary counts against the depth bound. `imports.md` notes that imported `fn` bodies may call `invoke(...)` and that cycle detection walks invoke paths, yet never notes that calling an imported `.warp` `fn` increments the invoke-depth counter while a same-file `fn` call does not. An implementer working from either page has no signal that cross-file `fn` dispatch must increment the counter, risking silent under-counting that lets warp-library recursion exceed 32 frames without the `loom/runtime/invoke-depth-exceeded` panic firing.

**History:** `git log -G "depth" -- docs/spec_topics/imports.md docs/spec_topics/functions.md` returns no commits — neither page has ever carried any reference to the invoke-depth bound. The defect against the two feature pages is present-since-inception of INV-4 (commit `833d8d5`); the two later refinements (`1b540e9`, `bf9d0a8`) widened the rule's surface on its owner page without back-propagating any forward-link or scope note to the feature pages that lexically introduce `.warp` `fn` declarations and `.warp` imports.

## Solution approach

Add a forward-reference to `functions.md`'s `**Placement.**` paragraph pointing at `invocation.md`'s `id="inv-4"`, signalling that a cross-file `fn` call is a countable frame against the invoke-depth bound while an intra-file call is not. Add a parallel forward-reference to `imports.md`'s `.warp` file-rules bullet (the one noting `fn` bodies may call `invoke(...)`), cross-linking the same `#inv-4` anchor. Leave INV-4's normative definition on `invocation.md` as the single source of truth — link the rule rather than restating it.

## Solution constraints

- Neither `functions.md` nor `imports.md` may duplicate INV-4's normative prose; the discoverability fix is a forward reference, with `invocation.md` remaining the single source of truth.
