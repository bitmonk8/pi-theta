## Problem

The V15 invocation leaves name the shared static-resolution parse cache "static-resolution per-pass parse cache" — in V15a (Adds, Ships when, the Spec-citation bullet), V15a-T, V15e (Adds, Deps, Ships when, its Spec-citation bullets), V15e-T, V15f (Adds), and twice in the `coverage-matrix.md` `cka-44` row. The normative spec source these leaves cite, `invocation.md` §Static resolution (`#static-resolution`), calls the same structure a "per-load-pass parse cache". The plan's spelling drops the load-bearing "load" qualifier and reorders the descriptor, so the plan and the normative spec term for one structure differ everywhere they meet. This is naming friction, not behavioural divergence — every leaf links the term to the defining spec anchor — but an implementer reconciling plan against spec meets multiple spellings for one cache.

## Issue introduction

**Verdict:** introduced by a specific later plan edit (single commit)
**Introducing commit:** `f541bc3` — "pi-loom plan: resolve \"static-resolution per-pass parse cache named inconsistently across V15a and V15e\"" (2026-06-17)
**History:** The plan corpus is git-tracked. `git log -S "static-resolution per-pass parse cache"` identifies `f541bc3` as the commit that first introduced this exact spelling across the V15 leaves and `coverage-matrix.md`. That commit resolved a prior review finding about the cache being named inconsistently *within the plan* (it previously appeared as "static-resolution parse cache", "per-pass cache", and "per-pass static-resolution cache") by standardising all sites on "static-resolution per-pass parse cache". The standardisation reconciled the plan internally but adopted the spelling of the supporting `implementation-notes.md` note rather than the normative `invocation.md` §Static resolution term, so it left the plan uniformly diverging from the normative spec term. Both spec spellings predate the plan standardisation (present since the `docs/` reorganisation `31ff060`); the plan-vs-spec divergence is what `f541bc3` introduced.

## Solution approach

Rename the cache to the spec's normative term "per-load-pass parse cache" at every plan site that currently carries "static-resolution per-pass parse cache" — V15a, V15a-T, V15e, V15e-T, V15f, and both occurrences in the `coverage-matrix.md` `cka-44` row. The noun phrase must contain "per-load-pass parse cache" verbatim so it greps against `invocation.md` §Static resolution; a "static-resolution" qualifier may be retained as a leading or parenthetical descriptor where a leaf names the pass.

## Solution constraints

- The spec is read-only for this fix: do not edit `invocation.md` or `implementation-notes.md`. The residual `implementation-notes.md`-vs-`invocation.md` spec-internal spelling mismatch ("per-pass parse cache" vs "per-load-pass parse cache") is spec-owned and out of scope here.
