## Finding

The **Discovery roots** block in `discovery-sources.md` opens by defining a discovery root as "the directory each source contributes to *the discovery walk*", then states that for a settings `loomPaths` entry "a directory entry contributes its own path; a file entry contributes its parent directory." Read against the opening definition, a file entry's parent directory is a directory "contributed to the discovery walk" — which an implementer can reasonably read as *scan that parent directory for `*.loom` children*. Under that reading, a `loomPaths` file entry would silently pull in sibling `.loom` files that the author never named.

That is not the intended behaviour. The `loomPaths` entry schema in `package-and-settings.md` (under **File entries** / **Directory entries**) is unambiguous: a directory entry expands to its non-recursive `*.loom` children, whereas a file entry registers exactly the one named `.loom` file and nothing else. The parent directory contributed by a file entry exists only to serve the path-restriction rule on `invoke` / `.loom` `tools:` resolution (a resolved callee path must lie within at least one active root) — it is never a discovery-walk input. The Discovery-roots block never makes this dual purpose explicit and carries no cross-reference to the `loomPaths` entry schema that settles it, so the disambiguating rule is reachable only by a reader who already knows to look on the other page.

The root cause is that a single term — *discovery root* — is overloaded for two distinct roles (an input to the file-discovery walk, and a path-restriction anchor for callee containment), and the definition sentence binds it to the first role while a file entry uses it only in the second.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 833d8d5 — spec: resolve five invoke edge cases (Decision #4) (2026-05-05, Thomas Andersen)
**History:** The Discovery-roots concept did not exist before Decision #4; commit 833d8d5 introduced it (in the then-monolithic `spec.md`) specifically to anchor the new `invoke` / `tools:` path-restriction rule, and the conflation was present at birth — the same block both defined a root as "the directory each source contributes to the discovery walk" and used a file entry's parent directory solely as a path-restriction anchor. Later commits only relocated and relabelled the text (4aef0e2 renamed `looms`→`loomPaths`; f5e89f4 split the spec into topic files, moving the block verbatim into `discovery-sources.md`) without resolving the overload.

## Solution Space

**Shape:** single

### Recommendation

In the **Discovery roots** block of `discovery-sources.md`, separate the term's two roles explicitly:

- State up front that a discovery root serves the path-restriction rule (callee-containment for `invoke` / `.loom` `tools:`), and that *which `.loom` files each source contributes* is governed separately by that source's own contribution rule — it is not a directory scan of every active root.
- On the `loomPaths` bullet, add a parenthetical making the file-entry case unambiguous: a file entry contributes its parent directory **as a path-restriction anchor only**; the discovered-loom set for that entry is exactly the single named `.loom` file (the parent directory is **not** scanned for additional looms). A directory entry, by contrast, expands to its non-recursive `*.loom` children.
- Cross-reference the authoritative `loomPaths` entry schema (`package-and-settings.md#loompaths-entry-schema`, the **File entries** / **Directory entries** rows) so the discovery-set semantics have one owner and one pointer.

This is a clarity-and-cross-reference edit confined to the Discovery-roots block; it changes no behaviour, only removes the directory-scan misreading. `V10a`'s settings-source discovery test should pin the distinction (a `loomPaths` file entry registers only the named file; a sibling `.loom` in the same directory is not discovered).

## Consequence

**Severity:** correctness

Two reasonable implementers reading only the Discovery-roots block diverge on the discovered-loom set for a `loomPaths` file entry: one registers exactly the named file, the other scans the entry's parent directory and registers every sibling `.loom`. The divergent set changes which slash commands exist and which sources participate in collision detection — an observable, silent behavioural difference.
