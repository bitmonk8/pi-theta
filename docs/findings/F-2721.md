## Finding

Real-host smoke pass criterion (e) (`real-host-smoke-gate.md`) scores a **fail** — a confirmed behavioural-divergence finding — whenever the live run emits a `loom-system-note` code outside the **committed permitted-code list** that `H7a` checks in alongside its fixture `.loom`. That list is, by `H7a`'s own description, a "manually-maintained, best-effort" union of the codes its **Deps** slices *can* emit, and its completeness "is not mechanically verified" (only the `golden ⊆ permitted` containment is gated, by `H7a`). The only mechanism keeping the list current is the prose phrase "kept current by a per-Deps-slice provenance obligation."

That phrase asserts an obligation without grounding it: no leaf, no named owner, and no concrete trigger is charged with appending a newly-introduced diagnostic code to the permitted-code list. The plan carries an analogous maintenance trigger for the `H5b` canary (How-to-use step 2: a new leaf whose tests close a coverage-matrix REQ-ID must be added to `H5b`'s **Deps**), but the permitted-code list has no equivalent declared trigger.

Consequently, when a slice in `H7a`'s **Deps** legitimately begins emitting a benign new `loom/...` code that the stale union omits, the next smoke run scores criterion (e) as a behavioural-divergence fail — a merge/bump blocker — even though no real double-vs-real divergence occurred. The smoke gate documents no recovery path for this case (edit the list, re-run), and its existing "benign live-model variance … is not a fail" carve-out covers *which permitted codes fire*, not *a code absent from a stale permitted list*.

## Consequence

**Severity:** advisory

If the plan ships unfixed, the permitted-code list silently drifts stale as code-emitting leaves land, because nothing obliges anyone to update it. A later smoke run then scores a spurious criterion-(e) fail and blocks an otherwise-clean merge or version bump, and the contributor has no documented recovery, so the false positive is indistinguishable from a genuine behavioural divergence at the point it fires.

**History:** `git log --follow` on `H7a-integration-acceptance.md` and `real-host-smoke-gate.md` shows the permitted-code list was added at `052b019` and refined across several later commits (`542395a` widened its Deps, `f340c3b` added the cannot-disagree guarantee). The maintenance-ownership gap was present from the mechanism's inception: `052b019` introduced the committed list without a concrete update trigger, and `f340c3b` hardened the disclosure that completeness is unverified and asserted the abstract "provenance obligation" without pinning who updates the list, when, or how a spurious (e) fail recovers. No commit ever closed the gap.

## Solution Space

**Shape:** single

### Recommendation

Make the abstract "per-Deps-slice provenance obligation" a concrete, locatable maintenance trigger and document the spurious-fail recovery:

- In `docs/plan.md` How-to-use, add a maintenance-trigger step parallel to the existing `H5b.Deps` trigger (step 2): when a new or modified leaf begins emitting a `loom/...` code that the `H7a` fixture path can surface, the authoring contributor appends that code to `H7a`'s committed permitted-code list as part of the same change. Name the contributor authoring/modifying the emitting leaf as the owner.
- In `docs/plan_topics/H7a-integration-acceptance.md`, replace the bare "kept current by a per-Deps-slice provenance obligation" wording with a pointer to that maintenance trigger, so the list's currency rests on a declared rule rather than an unanchored assertion.
- In `docs/plan_topics/real-host-smoke-gate.md` criterion (e), record the recovery path: a (e) fail whose offending code is a benign code the emitting slice legitimately produces but a stale permitted-code list omitted is an out-of-date-list condition, not a behavioural-divergence finding — the owner appends the code to the committed list and re-runs, rather than treating it as a merge-blocking divergence.

A mechanical re-derivation of the union before each smoke run is not available: `H7a` states there is "no emittable-code registry to verify against," so the union cannot be reconstructed automatically. The fix is therefore the declared maintenance trigger plus the recovery note, keeping the plan's stated "manually-maintained, best-effort" posture intact.

Edge case for the implementer: the recovery note must not let a genuine divergence be waved through as "stale list" — the owner appends only after confirming the emitting slice legitimately produces the code; an unexpected code with no owning slice in `H7a`'s Deps remains a fail.
