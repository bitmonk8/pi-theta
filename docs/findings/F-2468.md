## Finding

The prompt-mode *Error detection* probe (`#prompt-mode-error-detection`) asserts that "every transport- or provider-level failure of the user turn that is not a synchronous throw at the call site surfaces on the driven turn's trailing `assistant` message", then narrows its actual detection to a single field test: it "treats `stopReason` as opaque apart from the `"error"` member it keys on", and PIC-51 maps only `stopReason: "error"` to `Err(QueryError { kind: "transport", … })`. Everything that is neither a synchronous throw, nor the cancellation short-circuit, nor a trailing `assistant` message with `stopReason: "error"` falls through to PIC-53, which extracts the trailing turn as a successful `Ok(string)`.

This is a positive, load-bearing behavioural claim about Pi — *the failure universe of a driven user turn maps exhaustively onto a committed trailing `assistant` message carrying `stopReason: "error"`* — yet, unlike the three adjacent consumption-postures on the same page (`#driven-turn-commit-ordering-presupposition`, `#user-visible-streaming-ordering-presupposition`, `#waitforidle-no-rejection-presupposition`), it has no named presupposition block and no editorial-review hook in the Pi version-bump checklist. The commit-ordering posture only pins *when* the message is committed; the no-rejection posture only pins that `waitForIdle()` never rejects. Neither establishes that the failure manifests specifically as `stopReason: "error"` rather than as some other terminal stop reason or as an absent/empty trailing turn.

The gap is sharpened by a divergence with the sibling off-session path. `provider-error-mapping.md`'s *Stop-reason classification* explicitly handles non-`"error"` terminators on the `complete()` path — `stopReason: "length"` maps to `ContextOverflowError`, and content-filter / unrecognised terminators map to `TransportError` — but the driven-turn probe ignores all of them. A driven user turn that terminates with `stopReason: "length"` (context overflow), a content-filter stop reason, or no committed `assistant` message at all is therefore neither caught by PIC-51 nor classified the way the off-session path classifies the same terminator; it is silently extracted as a successful `Ok(string)` (an empty string when the turn produced no assistant text). PIC-53 would silently return `Ok` for a turn that actually failed.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `d37c563` (2026-06-05, `pi-loom spec: resolve "Prompt-mode errorMessage probe unreachable from specified surface"`)
**History:** The prompt-mode error-detection probe was reworked twice. At `8c6cfec` (2026-05-07, `pi-loom spec: resolve "pi.sendUserMessage returns void; prompt-mode transport-error detection path is mis-described"`) the probe moved from a (mistaken) throw/reject-of-`sendUserMessage` surface to reading `AgentSession.errorMessage`, mapping any non-empty value to `transport` — already an exhaustive single-field failure test, but keyed on `errorMessage` non-emptiness. `d37c563` then migrated the probe to read the trailing `assistant` message's `stopReason` / `errorMessage` from the `ReadonlySessionManager` surface and narrowed detection to `stopReason: "error"`, adding the "opaque apart from the `"error"` member it keys on" wording — this is the present concrete form of the defect and the point at which the `"length"` / content-filter blind spot was introduced. Two days later `ef4cfb8` (2026-06-07, `pi-loom spec: resolve "Stop-reason → QueryError variant mapping is undefined"`) added the *Stop-reason classification* (`length` → overflow, other → transport) to `provider-error-mapping.md` for the off-session `complete()` path, formalising the very non-`"error"` terminators the driven-turn probe ignores — but never propagated that classification to the driven turn. The sibling consumption-postures were authored separately (`3683345`, 2026-06-06, commit-ordering; `1ca13ec`, no-rejection item `(al)`) without ever adding a posture for the failure-universe→`stopReason: "error"` mapping. The gap has carried from `d37c563` through the present.

## Solution Space

**Shape:** multiple

### Option A — Presuppose-and-audit consumption posture

- **Approach:** Add a named consumption-posture block on `conversation-drive.md` (sibling to the commit-ordering and no-rejection postures) asserting that Pi normalises every non-synchronous-throw transport/provider-level failure of the driven user turn into a committed trailing `assistant` message with `stopReason: "error"` (and `errorMessage` populated where available), and that no driven-turn failure surfaces with a different terminal `stopReason` or as an absent/empty trailing turn. Add a matching editorial-review checklist item (e.g. `(am)`) under `version-bump-step2.md`, mirroring items `(ac)` / `(al)`. PIC-51, PIC-53, and the *Error detection* probe cite it.
- **Spec edits:** New `<a id="...">` posture paragraph after the `#waitforidle-no-rejection-presupposition` block; new checklist item in `version-bump-step2.md`; back-references from the *Error detection* probe and PIC-53's short-circuit list.
- **Pros:** Matches the spec's established treatment of unpinnable Pi behaviours (postures + editorial hooks); no runtime-classification change; the driven-turn path deliberately keeps `stopReason` opaque apart from `"error"`.
- **Cons:** The presupposition is a strong empirical claim. If Pi surfaces context overflow on the user turn as `stopReason: "length"` (which `provider-error-mapping.md` already documents as a real terminator on the sibling path), the posture is simply false, and the editorial audit cannot make it true — the runtime still silently returns `Ok` on a real overflow.
- **Risks:** Entrenches the divergence from `provider-error-mapping.md`; the "no committed message" case still needs an explicit disposition or it remains a silent `Ok`.

### Option B — Classify the driven-turn stop-reason universe in-runtime

- **Approach:** Extend the *Error detection* probe so that, after `waitForIdle()` resolves and the cancellation short-circuit is passed, it applies the stop-reason arm of `provider-error-mapping.md`'s *Stop-reason classification* to the trailing `assistant` message's `stopReason`: `"error"` → `TransportError` (current PIC-51), `"length"` → `ContextOverflowError`, any other terminator that is not a normal turn boundary (`stop` / `end_turn` / `tool_use`) → `TransportError(retryable: false)`. Define an explicit disposition for an absent trailing `assistant` message (synthesise a `TransportError`). PIC-53's `Ok` extraction runs only after this broadened classification passes. A residual posture is still needed only for commit-timing / the no-message case.
- **Spec edits:** Broaden PIC-51 (or coin a sibling REQ-ID) to the full stop-reason classification, citing `provider-error-mapping.md#stop-reason-classification` as the shared authority; restate PIC-53's short-circuit precondition to name the broadened classification; state the absent-message disposition.
- **Pros:** Robust to whatever terminal `stopReason` Pi writes; reuses an already-normative classification; closes the `"length"`/overflow silent-`Ok` hole without depending on an empirical claim that may be false.
- **Cons:** The stop-reason arm of `provider-error-mapping.md` is reproducible from the session-read surface (it needs only `stopReason` + `errorMessage`), but the HTTP-status arm is not available post-hoc, so the driven-turn path can only mirror the stop-reason arm — the spec must scope the reuse to that arm explicitly. Slightly more runtime surface.
- **Risks:** Couples the driven-turn path to `provider-error-mapping.md`'s classification; depends on that page's stop-reason enumeration being correct (see Related Findings).

### Recommendation

Option B. The non-`"error"` terminators it covers — especially `stopReason: "length"` for context overflow — are documented current Pi/pi-ai behaviour that the sibling off-session path already classifies, not a speculative future regression; treating them as an editorial-audit presupposition (Option A) would leave a real, present overflow failure silently extracted as `Ok`. Reuse the stop-reason arm of `provider-error-mapping.md`'s *Stop-reason classification* on the driven turn and scope the reuse to that arm (no HTTP status is reachable from the `ReadonlySessionManager` read surface). Implementers must still pin the disposition for an absent trailing `assistant` message and confirm the ordering: cancellation short-circuit → broadened stop-reason classification → PIC-53 `Ok` extraction.

## Consequence

**Severity:** correctness

A driven user turn that fails with any terminal `stopReason` other than `"error"` (notably `"length"` for context overflow), or with no committed trailing `assistant` message, is extracted by PIC-53 as a successful `Ok(string)` — frequently the empty string — instead of an `Err`. The loom observes a silent success for a failed query, and a value (or empty value) flows downstream as if the model had answered. Two implementers also diverge: one keys only on `stopReason: "error"` as written, another mirrors `provider-error-mapping.md`'s stop-reason classification onto the driven turn.
