## Finding

The active-set gating window — the `snapshot → setActiveTools → query → finally` restore bracket on the prompt path — is introduced and owned by `V9c` (it carries `PIC-17` and the coverage matrix maps the per-query snapshot/set/restore-in-finally facet to `V9c`). `V9f` does not redefine that window: its `Adds.` states the snapshot/restore is "the active-set gating window's snapshot/restore on the prompt path (defined in `V9c`)", and its failure-arm tests drive that same window — `PIC-8` makes the step-4 finally-restore throw, and `PIC-19` makes the step-1/step-2 snapshot/swap-install throw.

`V9f` declares `Deps. V9f-T, V9a, V5d` and `V9f-T` declares `Deps. V9a, V5d`. Neither dependency chain reaches `V9c`. Under the pickability predicate (a `Deps.` entry is satisfied only when the named leaf's completion tag exists), `V9f-T` becomes pickable as soon as `V9a` and `V5d` are complete — before `V9c` lands and therefore before the snapshot/restore window exists as a production symbol.

A `V9f-T` picked in that window writes `PIC-8`/`PIC-19` tests that reference a restore/snapshot step with no production target. The suite reds at compile time (or for a missing-symbol reason), which is disqualified by the project's "fail red for the intended reason" gate. `V9f-T-complete` cannot be tagged honestly, and `V9f` (which lists `V9f-T` in its `Deps.`) cannot turn those tests green.

## Consequence

**Severity:** blocking

`V9f-T` is pickable once `V9a` and `V5d` complete, before `V9c` introduces the snapshot/restore window. Its `PIC-8` (step-4 restore throw) and `PIC-19` (step-1/2 snapshot/swap-install throw) tests then reference a window step with no production symbol, so the suite reds for a compile/missing-symbol reason that fails the "fail red for the intended reason" gate — `V9f-T-complete` cannot be tagged correctly, and `V9f` cannot drive those tests green.

## Solution Space

**Shape:** single

### Recommendation

Add `V9c` to the `Deps.` of both leaves:

- In `docs/plan_topics/V9f-T-tool-registration-lifetime.md`, change `**Deps.** \`V9a\`, \`V5d\`` to include `V9c` (e.g. `\`V9a\`, \`V5d\`, \`V9c\``). `V9f-T` is the load-bearing edge: it is the leaf whose `PIC-8`/`PIC-19` tests reference the V9c-owned window step, so its pickability must wait on `V9c-complete`.
- In `docs/plan_topics/V9f-tool-registration-lifetime.md`, add `V9c` to `**Deps.** \`V9f-T\`, \`V9a\`, \`V5d\`` (e.g. `\`V9f-T\`, \`V9a\`, \`V5d\`, \`V9c\``). `V9f` already reaches `V9c` transitively once `V9f-T` lists it, but `V9f`'s own `Adds.` and tests cite the V9c window directly, so listing the edge explicitly keeps the Deps DAG self-describing.

Cite the leaf ID exactly as `V9c` (the existing `Prompt-mode conversation drive and active-set gating` leaf); do not edit `V9c` itself or the spec — the window definition already exists there.
