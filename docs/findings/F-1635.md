## Problem

In `docs/spec_topics/pi-integration-contract/audit-failures.md`, the *Malformed-marker discriminator* paragraph closes by requiring the audit to surface every malformed marker under "family (5)'s discriminator" (singular), irrespective of which clause `(a)`–`(g)` the marker failed. Read in isolation, that wording permits an audit that emits one shared `<symptom>` token for all malformed markers. The opposing obligation — one distinct stable `<symptom>` token per clause `(a)`–`(g)` — lives only in a counting aside inside the *Failure-surface contract* paragraph two bullets below, so an implementer can satisfy the singular-discriminator wording while leaving the per-clause routing collapsed.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09) "pi-loom spec: resolve \"T35 SDK capability inventory closed-set negative-direction audit\""

Commit `f5e89f4` (2026-06-04, "docs: enforce 100KB hard / 30KB recommended size cap on spec set") later relocated the two paragraphs verbatim into the sharded `pi-integration-contract/audit-failures.md` file without altering the wording. The singular-vs-plural mismatch was introduced in the same edit that introduced both halves and has been present since the paragraphs first existed.

## Solution approach

Rewrite the closing sentence of the *Malformed-marker discriminator* paragraph so that surfacing malformed markers under family (5) routes each violating clause `(a)`–`(g)` to its own stable `<symptom>` token, with no collapsing of clauses onto a shared token. Keep the existing additional obligation to surface the underlying out-of-inventory reference under its own family (1) / (2) / (3) discriminator on the same run. The implementer still owns the literal `<symptom>` strings; pin only that each clause gets a distinct one, on the same footing as the family-(5) stale sub-kinds `(s1)` / `(s2)`.
