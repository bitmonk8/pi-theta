## Finding

The *Binder model transport failure* row of the Failure-mode templates table
(`docs/spec_topics/binder/determinism-cancellation-failure.md`, **Failure-mode
templates (normative)**) renders the byte-pinned system note
`loom /<name>: argument binder unavailable (<provider>: <message>)`. The
surrounding paragraph classifies `<message>` as "runtime- or
classifier-/validator-derived content" — i.e. determined content under GOV-15
observable (c) — and pins its sibling placeholders explicitly: `<provider>`
is pinned to `TransportError.provider` (api-shaped `Model<Api>.api`),
`<ajv-summary>` is pinned to the validator-neutral concatenation of
`ValidationIssue` entries, and even `<name>` and `<value>` have unambiguous
sources. `<message>` is the only determined placeholder in that paragraph
without a stated field source, and the shard nowhere else binds it.

Two derivative behaviours are left unspecified as a consequence:

1. *Empty / absent source.* The system-note rendering rules in
   `defaulting-system-note-echo.md` define empty-after-stripping handling
   only for model-supplied content (rule 4, which reroutes an empty
   binder `message` to the malformed-envelope row). They do not cover an
   empty / missing runtime-derived `<message>`, which would render as
   `(<provider>: )` — a parenthetical with a dangling colon — with no
   reroute path.
2. *`ContextOverflowError`-as-transport.* The Failure-class taxonomy
   folds `ContextOverflowError` into the transport class for retry
   purposes and explicitly states there is no dedicated row: a
   structurally-overflowing binder surfaces through the transport row.
   `TransportError.message` does not exist on `ContextOverflowError`
   (the variants are separate schemas in
   `errors-and-results/queryerror-variants.md`), so the
   `ContextOverflowError`-as-transport path has no defined source for
   `<message>` even after the obvious `TransportError.message` pin is
   added.

Because the surrounding template is normative and the note is byte-pinned
under GOV-15, conforming renderers cannot reproduce the row without each
placeholder having a stated source.

## Solution Space

**Shape:** single

### Recommendation

In **Failure-mode templates (normative)** (alongside the existing
`<provider>` and `<ajv-summary>` pins), state that the transport row's
`<message>` placeholder is sourced from the classifier-produced
`TransportError.message` field (whose own derivation is pinned at
`errors-and-results/queryerror-variants.md` — `message` derivation
(prompt mode), with the fixed-string fallback `"provider transport
failure"` already specified there), rendered verbatim and then carried
through the shared **System-note rendering** discipline in
`defaulting-system-note-echo.md` (rules 1–3: whitespace
collapse / trim, 120-cp cap, prefix/suffix boundary). Specify the two
edge cases explicitly:

- *Empty after sanitisation.* When `TransportError.message` is the empty
  string, or collapses to empty under rendering rule 1 (whitespace
  only), the runtime substitutes the fixed string `provider transport
  failure` — the same string `queryerror-variants.md` names as the
  absent-`errorMessage` fallback. This keeps the suffix non-empty, the
  parenthetical well-formed (`(<provider>: provider transport
  failure)`), and the row's behaviour aligned with the upstream
  field-derivation rule.
- *`ContextOverflowError`-as-transport.* When the classifier output is
  `ContextOverflowError` (folded into the transport class for retry
  purposes per **Failure-class taxonomy**), `<message>` is sourced from
  `ContextOverflowError.message`, with the same empty / whitespace-only
  fallback above. This closes the "no dedicated row" gap without adding
  a new row.

Edge cases the implementer must watch: the cap-and-sanitise pipeline
runs against the full rendered note (loom prefix + interpolated
content), so the 120-cp cap applies to `loom /<name>: argument binder
unavailable (<provider>: <message>)` as a whole — a long provider value
or loom name eats into the budget available to `<message>`; do not
pre-truncate the field source. The fallback string is loom-controlled,
so it does not pass through rule 1's whitespace-collapse step (the
input is already a fixed literal), but it is still subject to the
120-cp cap.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge on the transport-row suffix.
Without the pin, one may render the underlying SDK / HTTP error message
verbatim while another renders a fixed string ("provider transport
failure") or omits the field entirely, producing different byte sequences
for the same failure; under GOV-15 observable (c) these are
non-equivalent. The `ContextOverflowError`-as-transport case is worse
again: there is no `TransportError`-shaped `message` field to read, so
even the obvious pin leaves the overflow sub-case unspecified.
