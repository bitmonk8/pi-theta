## Problem

The *Sequential by default* and *Specific exception types only* cross-cutting rules in `conventions.md` permit a production-code exemption only when its same-line `// allow:` / `// allow-broad-catch:` comment cites a token the loom 1.0 closing gate can resolve, and both admit a "code-keyed obligation area" as one such token kind. The other three admitted token kinds (a numbered `PREFIX-N` REQ-ID, a `loom/...` registry code, the `pi-sdk-boundary` literal) each have a canonical form a comment can write and the gate can match exactly; a code-keyed obligation area has none, because the `coverage-matrix.md` *Code-keyed obligation areas (no numbered REQ-IDs)* table keys every row by free-prose page+section text with no canonical short token. The two real exempt sites already diverge — `V9g` writes "(PIC code-keyed area)" and `V14b` writes "(TOOL code-keyed area)" — and a bare spec prefix cannot resolve to a single row since one prefix keys many distinct code-keyed rows. Consequently the `H5c` broad-catch resolution arm and the *Sequential by default* `// allow:` resolution arm cannot be built to fire deterministically, and `V9g`/`V14b` cannot author an exemption guaranteed to resolve.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** c6a664e — pi-loom plan: build/update plan for spec.md + review (2026-06-10, Thomas Andersen); 75b6a9b — resolve "Sequential by default carve-out admits only a numbered REQ-ID" (2026-06-10, Thomas Andersen); 96563f9 — resolve "broad-catch allow-comment predicate unsatisfiable for exempt sites" (2026-06-10, Thomas Andersen)
**History:** c6a664e built the `coverage-matrix.md` *Code-keyed obligation areas* table as free-prose page+section rows with no canonical per-row token. 75b6a9b then extended the *Sequential by default* carve-out — previously admitting only a numbered REQ-ID — to also admit a "code-keyed obligation area" as a citable `// allow:` token, and 96563f9 extended the *Specific exception types only* broad-catch predicate the same way. Each commit made a code-keyed area a citable token without ever defining its form or the gate's resolution rule against the prose-keyed table; the defect has been present since the code-keyed-area citable-token concept was introduced.

## Solution approach

Add a row-unique canonical citable token to every *Code-keyed obligation areas (no numbered REQ-IDs)* row in `coverage-matrix.md`, surfaced in the row itself so it identifies one individual obligation row (a bare spec prefix is insufficient because several rows share one prefix); the token-naming scheme is the fixer's choice. In `conventions.md`, in both *Sequential by default* and *Specific exception types only*, state the gate's resolution rule for the code-keyed-area arm: such a token resolves iff it matches exactly one code-keyed-area row's canonical token. Key the `H5c` broad-catch resolution arm and the *Sequential by default* `// allow:` resolution arm on that token, and retarget the `V9g` and `V14b` `// allow:` citations to cite it in place of their current ad-hoc parentheticals.
