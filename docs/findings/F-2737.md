## Finding

`V14b` models the concurrent / settle-all-before-next-turn / independent-per-sibling-lowering shape of a model-driven parallel tool-call batch. Its **Adds** correctly concedes this shape is a real-Pi *consumption posture*, not a loom-side guarantee: `@earendil-works/pi-coding-agent` encodes no typed contract pinning it, loom performs no batching or settle-barrier of its own on this path, and the in-process `H4a` session double "does not enumerate parallel-batch delivery in its fidelity contract." `V14b`'s **Tests** and **Ships when** drive the batch (one succeeding + one failing sibling) only through that double. The double's own modelling of the batch is therefore unconstrained by the fidelity contract, so the host-behaviour premise the lowering rules rest on is asserted by nothing observable.

The only routing for the real-Pi behaviour is version-bump editorial-review item **(an)** (`version-bump-step2.md`; `tool-calls.md#parallel-tool-batch-delivery-presupposition`). Item (an) is a SHOULD-level item in the `(f)`–`(aq)` band: per `V18c`'s Ships-when, a recorded `fail` / `not-performed` on a SHOULD-level item is a conformant non-blocking outcome. Item (an) also only asks the contributor to confirm the *candidate* Pi minor "still" delivers the assumed shape — it detects a future host-side *change*, never establishes that the *current* pinned Pi delivers the assumed shape at all. The real-host smoke gate does not close the gap either: its scored pass criteria (a)–(e) do not include parallel-batch delivery, and criterion (d)'s four fidelity-contract axes (streamed-token order, single-turn append, `pi.on` cancel-forward, cancellation propagation) do not cover it; a turn-ordering divergence would in any case fall under the smoke's "benign live-model variance in turn order/count is not a fail" rule.

The leaf frames its routing as "mirroring the way SLSH-2's real-Pi streaming-ordering behaviour routes to item (ak)." That parity is asymmetric: SLSH-2's streamed-token ordering *is* one of the real-host smoke's four scored fidelity axes (criterion (d) axis (i)), so SLSH-2 carries a real-host measurement in addition to its editorial item. Parallel-batch delivery carries no such axis, so the "mirroring" framing overstates `V14b`'s real-host coverage relative to its cited sibling.

## Consequence

**Severity:** advisory

The leaf is implementable and spec-faithful, so an implementer can still produce a working `V14b`. But the real-Pi parallel-batch delivery shape that `V14b`'s lowering rules entirely presuppose is measured by no gate at any point for loom 1.0 — not by the double (`H4a` does not enumerate it), not by the real-host smoke (not a scored criterion or fidelity axis), and not by item (an) (which only audits a future host change, not the current pin). If real Pi diverges — constructing the next user turn before every sibling settles, or collapsing a failing sibling — `V14b` ships green while prompt-mode parallel tool calls misbehave in live sessions, and the divergence surfaces, if ever, only at a non-blocking version-bump editorial audit.

## Solution Space

**Shape:** multiple

### Option A — Add a real-host measurement observable

**Approach.** Make the current pinned Pi's parallel-batch delivery shape an observable scored at a real-host gate: require `H7a`'s committed fixture `.loom` to exercise a model-driven parallel tool-call batch mixing a succeeding and a failing sibling, and add the settle-all-before-next-turn / independent-per-sibling-lowering shape as a scored real-host smoke observable (a new criterion or fidelity axis).
**Plan edits.** `H7a` fixture-content requirement (the fixture must contain a parallel batch); `real-host-smoke-gate.md` pass criteria (add the parallel-batch observable and define what holds for it on a real host).
**Spec edits.** `tool-calls.md#parallel-tool-batch-delivery-presupposition` and `version-bump-step2.md` item (an) would need to be reconciled, since both currently state detection routes to editorial review *because* there is no real-host signal; item (an)'s own text says the obligation "escalates to a build-time pin … once Pi exposes a typed parallel-tool-batch-delivery contract," i.e. it deliberately keeps this SHOULD-level until then.
**Pros.** Establishes the current pinned Pi actually delivers the assumed shape; closes the gap the finding names.
**Cons.** Directly contradicts the spec's deliberate SHOULD-level disposition; broad blast across spec + smoke gate + `H7a`; a turn-ordering divergence is the kind of variance the smoke explicitly tolerates as benign, so the new observable must define a falsifiable shape check rather than a turn-order match.
**Risks.** Re-opens a spec decision (the consumption-posture routing) rather than implementing it.

### Option B — Correct the disclosure, keep the spec's routing (recommended)

**Approach.** Keep the spec's SHOULD-level editorial routing and fix the `V14b` leaf so it no longer implies real-host parity with SLSH-2 and states plainly that the current pinned Pi's batch shape is an unmeasured presupposition.
**Plan edits.** In `V14b-tool-calls-parallel-batch.md` Adds, replace the "mirroring the way SLSH-2's real-Pi streaming-ordering behaviour routes to item (ak)" clause with an accurate statement: the parallel-batch delivery shape is, unlike SLSH-2's streamed-token ordering, not covered by any real-host smoke fidelity axis, so for loom 1.0 it is dispositioned but unmeasured against a real host, and item (an) audits only a future host-side change at a version bump rather than confirming the current pin.
**Spec edits.** None.
**Pros.** Spec-faithful; plan-internal; narrow blast; removes the misleading parity framing.
**Cons.** Leaves the real-host shape unmeasured — but by the spec's own design, not by oversight.
**Risks.** None material.

### Option C — Add a runtime divergence diagnostic

**Approach.** Emit a `loom/...` diagnostic at runtime when an arriving batch shape contradicts the presupposition (e.g. a sibling outcome observed collapsed rather than lowered independently).
**Plan edits.** A new `loom/...` code + registry entry and an owning leaf (`<new>`) implementing the runtime check.
**Spec edits.** `tool-calls.md` would need to define the diagnostic and the observable condition that fires it.
**Pros.** Production-time detection of a real divergence.
**Cons.** loom "performs no batching or settle-barrier of its own on this path," so it has limited vantage to detect that Pi built the next turn early; new code + registry + leaf scope for a posture the spec accepts as unpinned; heavier than the gap warrants.
**Risks.** A diagnostic that cannot reliably observe the divergence is worse than none.

### Recommendation

Take **Option B**. The spec deliberately dispositions parallel-batch delivery as an unpinned consumption posture routed to SHOULD-level editorial review until Pi exposes a typed contract; the defensible plan-internal fix is to make `V14b`'s Adds state that accurately rather than imply real-host parity with SLSH-2 (which carries a smoke fidelity axis `V14b` lacks). The corrected disclosure should make clear that item (an) audits a future host change, not the current pin, and that no real-host gate scores the batch shape for loom 1.0. Options A and C re-open the spec's routing decision and should only be taken if the smoke gate is intentionally being extended to cover this posture spec-side.
