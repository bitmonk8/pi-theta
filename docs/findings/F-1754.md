## Finding

Step 0(a) of the capability probe (`docs/spec_topics/pi-integration-contract/capability-probe.md`) states the normative rule as: *"a Node nightly or release-candidate build (e.g. `"23.0.0-nightly20250101abcdef"`) whose stable equivalent satisfies the floor MUST satisfy the check."* The term **"stable equivalent"** is undefined; the prose does not say whether it means *strip the prerelease tag and compare the resulting release version against the range* or *compare under SemVer 2.0.0 precedence*. The two readings give opposite results precisely for prereleases of the floor release itself.

Concrete divergence at `process.versions.node = "22.19.0-rc1"` against the pinned floor `>=22.19.0`:

- **Strip-tag reading.** Stable equivalent is `22.19.0`, which satisfies `>=22.19.0` → the probe MUST accept.
- **SemVer-precedence reading.** Under SemVer 2.0.0 §11, `22.19.0-rc1` has lower precedence than `22.19.0`, so it does not satisfy `>=22.19.0` → the probe rejects with `kind: "node-floor"`.

The "Recommended recipe (non-normative)" callout immediately below pins the recommended implementation as `semver.satisfies(process.versions.node, ">=22.19.0", { includePrerelease: true })`. `includePrerelease: true` only relaxes `semver`'s default rule that prereleases never match a range comparator that names a non-prerelease tuple; it does **not** override SemVer precedence, so the recommended recipe implements the SemVer-precedence reading and rejects `22.19.0-rc1`. The contract paragraph ("range membership with prerelease tags eligible") that the prose says any conformant SemVer comparator must preserve is the same shape as the recipe — but the normative prose above it, read literally, demands the opposite outcome on this one input class. A second-source implementer who reads only the normative paragraph and substitutes a hand-rolled strip-tag comparator passes the recipe's example (`23.0.0-nightly…`, which both readings accept), reads the contract as satisfied, and diverges from the recommended recipe at every prerelease of the floor release.

For all other prerelease inputs the two readings agree: a prerelease above the floor (e.g. `22.20.0-rc1`) is accepted under both, and a prerelease below the floor (e.g. `22.18.9-rc1`) is rejected under both. The defect is narrow but observably testable: a conformance test that exercises `22.19.0-rc1` against the production probe pins exactly one reading, and the spec does not state which.

## Solution Space

**Shape:** multiple

### Option A — Pin SemVer-precedence semantics (align prose with the recipe)

- **Approach.** Rewrite the Step 0(a) normative paragraph so the rule is stated as SemVer 2.0.0 range membership with `includePrerelease`-style relaxation: a build is accepted iff its SemVer-parsed version compares `>=` the floor under SemVer precedence (so prereleases of the floor release are rejected, prereleases of any version above the floor are accepted). Drop the "stable equivalent" gloss. Keep the recipe paragraph as-is — it already implements this reading.
- **Spec edits.** In `capability-probe.md` §(a), replace the second sentence (`"a Node nightly or release-candidate build (e.g. \`"23.0.0-nightly20250101abcdef"\`) whose stable equivalent satisfies the floor MUST satisfy the check"`) with prose along the lines of *"a build whose SemVer-parsed version is `>=` the floor under SemVer 2.0.0 precedence — including prerelease tags in the ordering — MUST satisfy the check; a prerelease of the floor release itself (e.g. `"22.19.0-rc1"`) has lower SemVer precedence than the floor and does NOT satisfy the check, while a prerelease of any version above the floor (e.g. `"23.0.0-nightly20250101abcdef"`) does."* Update the load-bearing-clause sentence to talk about "the SemVer-precedence membership rule" rather than "the prerelease-eligibility clause".
- **Pros.** Zero recipe change; the recommended `semver.satisfies(..., { includePrerelease: true })` already conforms. The conformance test surface stays small (one positive prerelease vector above the floor, one negative prerelease vector at the floor). Aligns the gate with the conservative spirit of the version-bump procedure — an RC of the floor is not the floor.
- **Cons.** Rejects `22.19.0-rc1` even though the floor release itself is `22.19.0`, which a developer who upgraded to the floor's first RC may find surprising; they have to fall back to a prior stable.
- **Risks.** None observable beyond the user-surprise above; the rejection routes to the existing `loom/load/host-incompatible (node-floor)` surface with no new diagnostic.

### Option B — Pin strip-tag semantics (define "stable equivalent" and change the recipe)

- **Approach.** Define "stable equivalent" as *the version with the prerelease tag stripped*, then state the rule as: a build is accepted iff its stable equivalent satisfies the floor range under release-only SemVer membership. Replace the recommended recipe with one that implements this reading.
- **Spec edits.** In `capability-probe.md` §(a), insert a single-sentence definition before the rule: *"The stable equivalent of a build is the version with any prerelease tag stripped (e.g. the stable equivalent of `"22.19.0-rc1"` is `"22.19.0"`, and of `"23.0.0-nightly20250101abcdef"` is `"23.0.0"`)."* Then rephrase the rule to use the defined term: *"a build whose stable equivalent satisfies the floor range under release-only SemVer membership MUST satisfy the check."* In the recipe callout, replace `semver.satisfies(process.versions.node, ">=22.19.0", { includePrerelease: true })` with a recipe that first strips the prerelease tag (e.g. `semver.satisfies(semver.coerce(process.versions.node).version, ">=22.19.0")`), and update the contract paragraph accordingly.
- **Pros.** Accepts `22.19.0-rc1`, matching the more permissive intuitive reading and unblocking developers running the floor's own RCs.
- **Cons.** Requires changing the recipe and the contract paragraph that downstream conformant alternate implementations must preserve. The conformance surface is wider: the test must pin both the strip-tag preprocessing (e.g. `22.19.0-rc1` accepted, `22.19.0-beta.1+build.7` accepted) and the discriminator route for unparseable inputs.
- **Risks.** A consumer running an unreleased build of the floor RC may exhibit Pi-SDK behaviour the runtime is not designed against; the conservative-gate rationale that justifies probing at all weakens slightly.

### Recommendation

Take **Option A**. The probe's purpose is a conservative host-qualification gate, the recommended recipe (which is the gravitational centre of the corpus and the implementation `V9a` is expected to ship) already implements the SemVer-precedence reading, and the divergence is at exactly one input class — prereleases of the floor release — where the conservative outcome is appropriate. The implementer-relevant edge cases the rewritten prose must cover are: (i) a prerelease of the floor itself (rejected with `kind: "node-floor"`); (ii) a prerelease of a version above the floor (accepted); (iii) a syntactically invalid `process.versions.node` (continues to route to `kind: "probe-failed"` via the existing `semver.valid` discriminator, unchanged). Pin `22.19.0-rc1` as the negative-test vector and `23.0.0-nightly20250101abcdef` as the positive-test vector in `V9a-T`'s Tests bullet so the conformance contract names the divergence point explicitly.

## Consequence

**Severity:** correctness

Two reasonable implementers reading only the normative paragraph diverge on `22.19.0-rc1` (and any other prerelease of the floor release itself): one rejects it as `kind: "node-floor"`, the other accepts it as a qualifying host. The probe is the single load-bearing host-binding gate and its discriminator routes to a `loom/load/host-incompatible` payload, so the divergence is operator-visible — a host that boots loom under one implementation refuses it under the other, with no other signal to triage the disagreement.
