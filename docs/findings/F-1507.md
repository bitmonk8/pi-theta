## Finding

In `docs/spec_topics/pi-integration-contract/capability-probe.md`, Step 0 (c)'s normative bullet list of factory-probable function members tags each entry with the SDK capability it contributes to. Every other bullet uses a single tag — `(capability 1)`, `(capability 2)`, `(capability 4)`, etc. — but the `AgentSession.prototype.abort` bullet uses `(capability 3 / capability 5; …)`. The `/` is conventionally read as "or" (one or the other), which contradicts the actual contract: this single function member is a load-bearing element of **both** the Subagent-mode-isolated-session capability (item 3 in `capability-inventory-items.md`) and the Cancellation-propagation capability (item 5), and both inventory entries explicitly cross-reference this same probe site. Item 5's prose calls out exactly that joint coverage: it states that `AgentSession.prototype.abort` is "enumerated alongside `createAgentSession` under Step 0 (c)'s factory-probable members". A reader scanning the probe list to reconstruct the capability-to-member coverage map will either undercount (treat the bullet as contributing to one of the two capabilities) or be unsure which side to count it on.

The defect is purely editorial — a slash where conjunction is meant. The fix is a one-word-class change inside the parenthetical (replace `/` with `and`, or rephrase as "jointly contributes to capabilities 3 and 5"); no normative behaviour, count, or downstream rule moves.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/pi-integration-contract/capability-probe.md`, Step 0 (c), change the opening of the `AgentSession.prototype.abort` bullet's parenthetical from

> `AgentSession.prototype.abort` (capability 3 / capability 5; the cancellation primitive …

to

> `AgentSession.prototype.abort` (capabilities 3 and 5 jointly; the cancellation primitive …

This is the smallest disambiguating edit that matches the conjunctive reading already pinned by `capability-inventory-items.md` items 3 and 5. Edge cases the implementer must watch:

- Do **not** change the surrounding per-capability count narrative in the same paragraph ("capability 1 contributes one, capability 2 contributes one, capabilities 3 and 6 each contribute two, and capability 4 contributes three") — that count already assigns this member solely to capability 3 for partition purposes, and the `FACTORY_PROBABLE_CAPABILITIES` constant (five items: 1/2/3/4/6) likewise does not include capability 5. The "jointly" phrasing describes inventory coverage of the member, not the probe's iteration partition; the two are deliberately distinct.
- Do **not** touch the bullet's load-bearing tail (import path, factory-time `typeof` recipe, `loom/load/host-incompatible` mapping) — only the leading capability tag changes.
- No corresponding edits needed in `capability-inventory-items.md`, plan leaves, or test fixtures.

## Consequence

**Severity:** cosmetic

A careful reader still arrives at the right understanding once they cross-reference inventory items 3 and 5, but the bullet as written invites a mis-tally of the capability-to-member coverage map (e.g. when reviewing PIC-5's "exactly five checks" claim or auditing the per-capability function-member count `1+1+2+3+2` in the surrounding paragraph). No implementation or test outcome changes.
