## Finding

The spec's `binder/binder-model-and-context.md` §`binder-model-hot-reload` recovery-note contract has two facets: (1) emission — a `looms.binderModel` change that lets a previously `binder-model-unresolved` loom re-resolve emits the consolidated `loom-system-note`; and (2) coalescing — "Multiple `looms.binderModel` changes observed within a single settings-file `250 ms` debounce window (the window defined in [Discovery — Caching and reload]) coalesce into one note; changes in distinct windows fire separately." The plan maps the whole contract to `V11a` (coverage-matrix `cka-38`).

`V11a` carries a citing test for facet (1) ("a hot reload that recovers a previously-unresolved model emits the recovery note") but nothing for facet (2). The 250 ms window the coalescing clause names is not owned by `V11a` — it is the watcher-event reload-debounce owned by `V10d` (`cka-36`, whose own test asserts a burst of watcher events produces exactly one reload). `V11a`'s `Deps` are `V11a-T, V9b, V10c`; `V10d` is absent, and no `V11a` text states that within-window coalescing of the recovery note is inherited from `V10d`'s coalesced reload pass or verified there.

The result is an unanchored assumption: the recovery note is emitted "on the next loom load", and one-note-per-window holds only if the recovery computation runs once per coalesced reload rather than once per raw watcher event. With the H5a citing-test scan satisfied by the facet-(1) test alone, `cka-38` stays green while the coalescing facet is neither tested at `V11a` nor attributed to `V10d`. An implementer who wires the recovery note off raw `looms.binderModel` change events rather than off the coalesced reload pass would ship per-change note firing, violating the consolidated-note contract, with no gate reddening.

## Solution Space

**Shape:** multiple

### Option A — Attribute coalescing to V10d (lightweight; documentation-only)

**Approach.** Record that the within-window coalescing of the recovery note is structurally supplied by `V10d`'s reload-debounce coalescing (`cka-36`): the recovery computation runs once per coalesced reload pass, so one-note-per-window follows from one-reload-per-window. `V11a` verifies only facet (1) (single-recovery emission).

**Plan edits.** In `V11a-binder-model-resolution.md`, extend the hot-reload recovery-note bullet (Adds and the Tests bullet) to state that the recovery note is emitted during the reload pass and that the spec's within-250 ms-window coalescing is supplied by `V10d`'s reload-debounce coalescing, verified at `V10d` (`cka-36`), not re-tested here. In `coverage-matrix.md`, annotate the `cka-38` row that the `#binder-model-hot-reload` coalescing facet is closed via `cka-36`/`V10d`.

**Spec edits.** None.

**Pros.** No redundant coalescing test (the reload-coalescing is already asserted at `V10d`); reflects the actual architecture (coalescing is a reload-pass property); no new `Deps` edge or reordering.

**Cons.** Relies on `V11a`'s implementation actually emitting off the coalesced reload pass; the attribution is prose, not a mechanical assertion at `V11a`.

**Risks.** If a later refactor moves recovery-note emission off the reload pass, the prose attribution silently goes stale.

### Option B — Declare the dependency and verify coalescing at V11a

**Approach.** Add `V10d` to `V11a`'s `Deps` and add a `V11a`/`V11a-T` test asserting that multiple `looms.binderModel` changes within one 250 ms window produce exactly one recovery note (and changes in distinct windows fire separately).

**Plan edits.** In `V11a-binder-model-resolution.md` and `V11a-T-binder-model-resolution.md`, add `V10d` to `Deps` and add a coalescing-facet citing test driven through the `FakeClock` seam. Reflect the new closing test against the `cka-38` row in `coverage-matrix.md`.

**Spec edits.** None.

**Pros.** Makes the coalescing facet mechanically verified at the leaf the contract maps to; the assumption becomes an asserted property rather than inherited prose.

**Cons.** Duplicates `V10d`'s reload-coalescing assertion against the same `FakeClock` mechanism; adds a `Deps` edge that constrains `V11a`'s pickability ordering; larger diff.

**Risks.** The new test couples `V11a` to the `FakeClock`/reload-debounce harness `V10d` owns, widening `V11a`'s test surface.

### Recommendation

Take Option A: the coalescing is a property of the coalesced reload pass (`V10d`/`cka-36`), so attributing the facet there and pinning the emission-during-reload-pass expectation in `V11a` closes the ownership gap without duplicating `V10d`'s coalescing assertion. The implementer must ensure `V11a`'s recovery computation is driven off the coalesced reload pass, not raw `looms.binderModel` watcher events — the exact point Option A's prose must make explicit.

## Consequence

**Severity:** advisory

The recovery-note contract's coalescing facet has no closing test and no declared ownership edge, so the closing gate certifies `cka-38` green while one-note-per-window is unverified. An implementer who reads only `V11a` could attach the note to raw watcher events instead of the coalesced reload pass and ship per-change firing that violates the consolidated-note contract; the spec clause itself remains correct, so a careful implementer can still produce a conforming leaf.
