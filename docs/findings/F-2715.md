## Finding

`H6a`'s **Ships when** field gates the release on a falsifiable evidence record committed in the gate-activation commit message, and twice claims that record makes a defective run catchable after the fact:

- for the manual real-host smoke run: "so a skipped, **mis-recorded**, or stale-pin run is detectable after the fact by reading the commit message";
- for the nine-item release-time residue inspection: "so a skipped or **mis-recorded** inspection is detectable after the fact by reading the commit message."

Reading the self-attested commit message reliably detects only two of the named classes: an **absent** record (the run/inspection was skipped — no evidence lines present, or per-item outcomes missing/incomplete) and a **stale-pin** run (the recorded Pi-SDK pin literal disagrees with the single-source-of-truth pin at `host-prerequisites.md#pi-sdk-pin`). The broad word "mis-recorded" also covers the class the read cannot catch: a **dishonest or incorrect result** — a run recorded "passed" that actually failed, or a residue item recorded "clean" that was not. The record is self-attested by the same contributor who ran the gate; a reader of the commit message has no independent surface to contradict an honest-looking but wrong "passed"/"clean". The claim therefore asserts a detection capability the mechanism does not provide for the dishonest-result class.

## Consequence

**Severity:** advisory

A reader or the release-gate auditor may believe that reading the commit message catches any defective run, when it catches only absence (skip), incomplete fields, and pin-literal disagreement — not a dishonestly-recorded "passed"/"clean" outcome. The leaf still ships and the gate still functions; the cost is an overstated assurance about an audit that is in fact self-attestation for the result classes.

The same overclaim wording was replicated to the residue-inspection sentence in ee9acd0 when `H6a` was pinned as the residue-inspection owner. Both instances bundle "mis-recorded" with "detectable" without distinguishing the catchable classes from a dishonest self-attested result.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H6a-live-corpus-activation.md` **Ships when**, narrow both detectability claims so they assert only what reading a self-attested commit message can establish, and disclose that the recorded result itself is self-attested.

For the smoke-run sentence, replace the "skipped, mis-recorded, or stale-pin run is detectable after the fact by reading the commit message" clause with wording that scopes detection to: a skipped run (no recorded evidence), a run with missing/incomplete evidence fields, and a stale-pin run (the recorded pin literal disagreeing with the single-source-of-truth pin at `host-prerequisites.md#pi-sdk-pin`); and state inline that the recorded pass/fail result is self-attested by the running contributor, so the gate cannot detect a dishonestly-recorded result (a failed run recorded as passed).

For the residue-inspection sentence, apply the parallel narrowing: a skipped inspection or one with missing/incomplete per-item outcomes is detectable by reading the commit message, but a dishonestly-recorded "clean" item is self-attested and not catchable by the read.

The fix is confined to plan prose in this one leaf; no spec edit and no change to the gate mechanism is required (the spec is read-only for this fix).
