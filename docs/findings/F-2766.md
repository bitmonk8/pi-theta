## Finding

`V6d` (and its paired `V6d-T`) asserts that a Path-only `${…}` "resolves against the validated `params` object," and its per-type stringification vectors drive reference params through the `system:` surface. The spec confirms the semantics: `frontmatter-fields-b-and-templates.md` states the path is resolved "against the validated params object" at conversation-creation time, after AJV validation. The producer of that validated `params` object — the leaf that owns the `params:` contract and its AJV validation — is `V6b`. Neither `V6d` nor `V6d-T` names where the object comes from, and `V6b` is absent from both leaves' `Deps` (`V6a`, `V3a`, `V13a`).

Because `V6b` is not in `Deps`, `V6d`/`V6d-T` become pickable once `V6a`/`V3a`/`V13a`/`V6d-T` complete — even if `V6b` has not landed. An implementer reaching the per-type-stringification vectors then has no established source for the validated `params` object: one implementer wires the leaf against `V6b`'s validated output, another invents an unvalidated stand-in or assumes a different source. The dependency graph does not record the real `V6d → V6b` edge that the test prose presupposes.

## Consequence

**Severity:** correctness

The `V6d → V6b` dependency the test prose relies on is unrecorded, so `V6d`/`V6d-T` can be picked before `V6b`'s validated-`params` machinery exists. Two reasonable implementers diverge on the source of the validated `params` object — one wiring against `V6b`, another inventing an alternate or unvalidated source — producing inconsistent dependency graphs and a leaf whose stringification vectors may run against an unvalidated object.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 3625ee0 — pi-loom plan: resolve "system: interpolation per-type stringification not witnessed" (2026-06-11, Thomas Andersen)
**History:** At the plan's first commit (c6a664e, 2026-06-10) `V6d`'s `Deps` were `V6d-T, V6a, V3a, V13a` and the leaf body carried no reference to a validated `params` object. Commit 3625ee0 added the "resolves against the validated `params` object" prose and the per-type stringification vectors to both `V6d` and `V6d-T` while leaving the `Deps` set unchanged, so the reference to a validated-`params` producer was introduced without the corresponding `V6b` dependency edge.

## Solution Space

**Shape:** single

### Recommendation

Establish `V6b` as the producer of the validated `params` object on both `V6d` and `V6d-T`. In the `Tests.` bullet that currently reads "`\${` escapes interpolation; a Path-only `${…}` resolves against the validated `params` object," name the producer — that the `params` object is the one validated by `V6b`'s AJV path — and add `V6b` to each leaf's `Deps.` field so the dependency graph records the `V6d → V6b` and `V6d-T → V6b` edges. Cite only the existing `V6b` ID; do not introduce a new leaf. The spec is read-only for this fix — the resolution semantics in `frontmatter-fields-b-and-templates.md` are already correct and need no edit.
