## Finding

`V9f`'s **Adds** enumerates "the `Type.Unsafe` schema bridge" as a behaviour the leaf owns. The bridge is the step that wraps the lowered JSON-Schema artefact (produced by [Schema Subset — Lowering Algorithm](../spec_topics/schema-subset.md#lowering-algorithm)) via `Type.Unsafe<unknown>(loweredJsonSchema)` to produce the TypeBox `TSchema` carried in the registered `ToolDefinition.parameters` — the parameters surface the model actually sees (`extension-bootstrap-and-per-loom.md` §Per-loom registration; `tool-registration-lifetime.md`).

Neither `V9f` nor its paired `V9f-T` carries a Tests bullet that names this bridge. The four Tests bullets assert the `PIC-8` restore-failure path, the `PIC-19` snapshot/swap-install-failure path, callable-set visibility plus `registration-cache-collision` disambiguation, and the `ToolDefinition.label` derivation. None of these exercises whether the lowered schema is faithfully bridged into the materialised `ToolDefinition.parameters`.

The bridge has behaviour beyond a trivial pass-through: it must preserve the lowered canonical schema shape into the registered `ToolDefinition.parameters`. The `registration-cache-collision` test does not close this gap — the cache slug and byte-equality check key on the lowered canonical-form bytes (the bridge *input*), so a bridge that drops or reshapes fields on the way into `ToolDefinition.parameters` (the bridge *output*) can still slug consistently and leave every cache-collision assertion green. A wrong-shaped bridge therefore ships through `V9f`'s exit gate undetected and surfaces only downstream at a consumer (`V6c`, `V14a`, `V15d`).

**Introducing commits:** c6a664e1194227df0702609667f3ee8be2e8c6b2 (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** `V9f`'s first authoring (c6a664e) already named "the `Type.Unsafe` schema bridge" in **Adds** while its **Tests** asserted only `PIC-8`, `PIC-19`, and visibility/cache-collision — so the bridge has never had an asserting bullet. The later commit cfd76f7 (2026-06-12, "Per-loom registration ToolDefinition field-derivation MUSTs unnamed by any leaf") added the `ToolDefinition.label`-derivation Tests bullet, and 66e4a27 / 8bd82be (2026-06-17) touched the label gate and the active-set window wording, but none of these added a Type.Unsafe-bridge assertion. The gap is original to the leaf, not introduced by a subsequent edit.

## Solution Space

**Shape:** single

### Recommendation

Add a Tests bullet to `V9f-T`'s **Tests** field asserting that a lowered tool `parameters` schema is bridged faithfully into the registered `ToolDefinition.parameters` via `Type.Unsafe` — i.e. the materialised `ToolDefinition.parameters` carries the same lowered canonical schema (the canonical form produced by the lowering algorithm that also feeds the registration-cache slug). Mirror the bullet in `V9f`'s **Tests** field, and extend `V9f`'s **Ships when** to name the new assertion alongside `PIC-8`, `PIC-19`, and the `ToolDefinition.label` derivation.

The test fixture should drive a known lowered JSON-Schema document through the bridge and assert the resulting `ToolDefinition.parameters` round-trips to that same lowered schema (every field preserved), targeting the runtime's own bridge seam rather than a real Pi registration surface. No spec edit is required — the bridge is already specified in `tool-registration-lifetime.md` and `extension-bootstrap-and-per-loom.md` §Per-loom registration, and `Type.Unsafe` callability is probed by `V9a`.

## Consequence

**Severity:** advisory

A `Type.Unsafe` bridge that materialises a wrong-shaped `ToolDefinition.parameters` (dropping or reshaping lowered-schema fields) would pass `V9f`'s exit gate, because no test witnesses the bridged output and the cache-collision test keys on the bridge input. The defect then surfaces only at a downstream consumer. The leaf remains implementable; the gap is missing regression coverage, not a contradiction or an unpickable leaf.
