## Finding

`patch-skew-degradation.md` opens with the claim that "the [loom 1.0 Pi-SDK pin range](./host-prerequisites.md#pi-sdk-pin) permits patch-only Pi updates without re-running [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure)," and then proceeds to specify a *single* surface — `SessionShutdownEvent['reason']` — that can transiently disagree during the resulting patch-skew window. For that one surface the spec is exhaustive: it pins widening/narrowing/mixed semantics, a bidirectional build-time type-equality assertion as the eventual detection gate, and three fixture-obligation categories that the runtime falls back through during the window.

The contract is silent on the rest of the unpinned-host-presupposition catalogue that `host-prerequisites.md` enumerates. At least nine separately-named presuppositions — `messages-chronological-order-presupposition`, `messages-leading-user-message-presupposition`, `getcommands-completeness-presupposition`, `register-tool-post-startup-presupposition`, `registry-persistence-presupposition`, `pi-slash-handler-promise-lifecycle-presupposition` (six sub-properties), `model-registry-population-presupposition`, `settings-write-back-preservation-presupposition`, `operator-always-present-presupposition`, `loom-flag-namespace-presupposition`, plus the four degraded-state sub-clauses and clause-(w) cwd presupposition — each route their detection to editorial review under the Pi version bump procedure (checklist items (h), (ag), (l), (z), (ap), (g)/(j)/(v)/(ad), (k), (s), (ai), (ao), (w), …). Every one of those items is gated on a *minor* bump only. Several of them are additionally flagged "known-fragile evidence" because they live in a bundler-output `dist/*.js` path that a re-bundle could silently rename — and a re-bundle is exactly the kind of change a Pi patch admits under the tilde range.

The patch-skew contract therefore tacitly extends to every presupposition in that catalogue the same allowance it explicitly authors for `SessionShutdownEvent['reason']` — but without the build-time gate, without the runtime fallback, without the fixtures, and without naming the assumption. A Pi patch that reorders `AgentMessage[]`, prepends a leading `assistant` message, retires a `pi.registerTool` registration mid-lifetime, ships a partial `ctx.modelRegistry.getAvailable()` view, drops loom keys on settings write-back, re-routes `pi.on`, or re-bundles any of the six `dist/*.js` evidence sites would land on a user's machine the moment it appears upstream of the tilde range, with nothing on the loom side to detect it until the next *minor* bump's editorial review runs. The contract should either name the assumption ("loom 1.0 assumes Pi patch releases preserve all unpinned presuppositions") so a Pi patch that violates it is at least a contract violation rather than a silent gap, or scope the "permitted without re-running the bump procedure" allowance to enumerate which presuppositions the patch route is actually not gated against.

**History:** the defect is the *interaction* between two corpora that evolved independently. The patch-skew contract was authored in May for one surface only — at the time it landed, the host-prerequisites file did not yet enumerate the ten-plus presupposition paragraphs the contract now implicitly covers. The June presupposition-adding commits then doubled the host-prerequisites inventory without anyone going back to widen the patch-skew window's scope; each one added a checklist item under "re-audit on each Pi minor bump" without touching the patch-skew page. No single commit authored the silent assumption — it was created by additions to one file leaving a peer file's scope claim stale. The durable fix addresses that scope interaction at the patch-skew opening paragraph rather than reverting any of the per-presupposition adds.

## Solution Space

### Recommendation

In the *Patch-skew degradation contract* opening paragraph of `docs/spec_topics/pi-integration-contract/patch-skew-degradation.md`, immediately after the existing "permits patch-only Pi updates without re-running [the bump procedure]" sentence, add a load-bearing assumption clause: state that loom 1.0 *assumes* a Pi patch release preserves every unpinned host presupposition catalogued in `host-prerequisites.md`'s presupposition inventory, including the bundler-output evidence sites those presuppositions cite as "known-fragile evidence." Name the failure mode explicitly: a Pi patch that violates any of those presuppositions reaches users with no loom-side build-time gate, and detection routes through the next minor bump's editorial-review checklist (items (a)–(ap)). Cite the host-prerequisites presupposition list by anchor so the assumption travels with the inventory rather than enumerating presupposition names inline.

This is one added paragraph or expanded sentence at the top of `patch-skew-degradation.md`; it requires no edits to `host-prerequisites.md` or `version-bump-*`. It keeps the patch-skew page as the single home of patch-window scope, makes the assumption a *named* one that future findings can cite, and leaves the existing bump-procedure detection routing unchanged.

Implementer edge cases to honour:

- Cite the host-prerequisites inventory by anchor so a presupposition added later is automatically covered by the assumption clause.
- Add a one-line reminder at the top of the host-prerequisites presupposition inventory pointing back to the patch-window assumption clause, so a contributor adding a new presupposition is reminded their addition inherits the unprotected-patch-window posture by default and must escalate (e.g. to a build-time gate) if that posture is unacceptable for the new presupposition.
- Explicitly include the "known-fragile evidence" bundler-output paths in the assumption's scope, so a Pi re-bundle that renames `dist/core/extensions/loader.js` is a named violation rather than an interpretive question.

## Consequence

**Severity:** correctness

A Pi patch that silently violates any of the ten-plus unpinned presuppositions reaches user machines under the existing tilde range with no loom-side detection until the next minor bump's editorial-review checklist runs — potentially months. Failure modes named in the cited host-prerequisites paragraphs include silent corruption of every binder context window (re-ordered or leading-non-`user` `AgentMessage[]`), dispatch-time `TypeError`s in the hot-reload registry-swap path (post-startup `pi.registerTool` retirement), erased loom configuration on Pi-initiated `/settings` rewrite, and degraded but undetected cancellation-forwarding behaviour. Two reasonable implementers will read the silence differently: one will treat the patch-skew contract as exhaustive ("only `SessionShutdownEvent['reason']` can drift on a patch — everything else is stable by Pi's semver discipline"), the other will treat every "re-audited on each Pi minor bump" sentence as binding only on minor bumps and therefore unprotected against patches.
