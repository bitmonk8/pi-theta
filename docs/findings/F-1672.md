## Finding

The **Unresolvable operands.** paragraph in [`type-system.md`](../../../docs/spec_topics/type-system.md) (around line 48, immediately before TYPE-9) phrases its trigger condition as *"When either side of a compatibility check is **past the parser's static view**…"*. The phrase has no definition anywhere in the corpus and never recurs; it is followed by an `e.g.` list of two examples ("an inferred binding whose RHS depends on a Pi-tool call whose registered schema is not visible at parse time", and "an `invoke` against a callee that produced `loom/load/callee-has-errors`"). An `e.g.` list does not close the predicate, so an implementer has no rule for cases the examples do not cover: a `.loom`-callable call (not just `invoke`) against a callee that produced `loom/load/callee-has-errors`; a callee that emitted some other load-time error and was not registered; a type derived through `match` whose arms depend on operands of the above kinds; an `invoke<Schema>(…)` whose `Schema` is itself an alias whose RHS is unresolvable. Two reasonable implementers will draw the line in two different places.

The corpus already carries the precise notion the paragraph wants: every sibling rule that talks about the same skip — [`invocation.md` *Static resolution*, *Typed return*, *Argument binding*, *Argument arity*](../../../docs/spec_topics/invocation.md), [`functions.md` *Loom return type*](../../../docs/spec_topics/functions.md), [`tool-calls.md` *Return type* / *Argument shape*](../../../docs/spec_topics/tool-calls.md) — does so in terms of the *statically resolvable* predicate defined under [`invocation.md` *Static resolution*](../../../docs/spec_topics/invocation.md), and the V2b plan leaves use "statically-unresolvable operands" in their `Adds` / `Ships when` clauses. Type-system.md's own ad-hoc "past the parser's static view" wording is the lone outlier, leaving the otherwise-coherent terminology unanchored at the rule that defines the relation.

The first `e.g.` example is also wrong on its own terms: per [`tool-calls.md` *Return type*](../../../docs/spec_topics/tool-calls.md), a Pi-tool call's static return type is always `Result<string, QueryError>` — the tool's *input* schema (TypeBox / JSON Schema exposed by Pi at registration) is not what drives compatibility on the return side, and there is no parse-time fact about a Pi tool's registered schema that makes the LHS of a `⊑` check unknown. The example a reader can substitute is the same one the rest of the corpus carries: a `.loom`-callable callee (whether reached through `invoke(...)` or through a `tools:` entry) that is not statically resolvable.

## Solution Space

**Shape:** single

### Recommendation

Replace the prose trigger with the corpus's existing closed predicate. Rewrite the **Unresolvable operands.** paragraph to:

1. Open with: *"When either side of a `T₁ ⊑ T₂` check has a static type that is **not statically resolvable** (per [Invocation — Static resolution](./invocation.md#static-resolution)), the parse-time check is skipped and the runtime AJV check is the only safety net."*
2. Enumerate the closed list of unresolvable-operand kinds, each citing its anchor:
   - A typed `invoke<Schema>(...)` site (or a `.loom`-callable call dispatched through a `tools:` entry) whose callee is not statically resolvable — i.e. the callee file is unreadable, fails to parse, or fails its own structural checks and therefore produced `loom/load/callee-has-errors` per [Invocation — Static resolution](./invocation.md#static-resolution). The RHS (callee inferred return type) is unknown.
   - The annotation `Schema` on `invoke<Schema>(...)` itself when `Schema` is a `NamedType` whose declaration is not loaded for the same reason. The LHS is unknown.
   - A `.loom`-callable-call argument slot whose callee is not statically resolvable per the same predicate — the param schema the argument must satisfy is unknown.
   - A binding or expression whose static type is computed through inference from one of the above (e.g. `let x = invoke<Schema>("./unresolvable.loom", ...)` where `Schema` itself is unresolvable, or a `match` whose arm bodies invoke an unresolvable callee) — the inferred type is propagated as "unresolvable" and any `⊑` check downstream of it is also skipped.

   Make the list closed: *"No site outside this enumeration treats an operand as unresolvable; in particular, Pi-tool calls (whose static return type is fixed at `Result<string, QueryError>` per [Tool Calls — Return type](./tool-calls.md)), schema-inferred bindings whose schema was resolved at parse time, and `match` arms whose bodies all type-resolve are **always** statically resolvable."*
3. Drop the misleading "Pi-tool call whose registered schema is not visible at parse time" example. A Pi tool's input schema does not enter `⊑` on its return side, and a Pi tool's *return* type is always the fixed `Result<string, QueryError>` per [Tool Calls — Return type](./tool-calls.md); the `e.g.` example invites confusion at the very site that should be closing it.
4. Keep the final cross-reference sentence intact ("cite this paragraph rather than restating it"), so the consolidation invariant the paragraph already serves continues to hold.

Edge cases the implementer must watch:

- The "alias whose RHS is unresolvable" case is reached through TYPE-11 alias-schema transparency: when transparency unfolds a `NamedType` whose declaration is missing, treat the unfolded operand as unresolvable rather than re-firing `loom/parse/type-alias-cycle` or any other diagnostic — the unresolvability is a load-time fact already surfaced as `loom/load/callee-has-errors` (or the equivalent), and TYPE-11 is not the surfacing site.
- `loom/parse/invoke-arity-too-many` remains parse-time even against an unresolvable callee (per [Invocation — Argument arity](./invocation.md#argument-binding) — extra positional arguments cannot be matched and there is no runtime safety net possible). The closed list above is for `⊑` checks only; arity is a separate axis.
- The `void` exclusion already in the paragraph above (`void` does not participate in `⊑`) is orthogonal and stays where it is — `loom/parse/void-in-non-return-position` is the earlier-firing rejection, so a `void` operand never reaches the unresolvable-operand question.

## Consequence

**Severity:** correctness

Two implementers compiling the spec will draw the parse-time / runtime-AJV boundary differently, producing different parse-error coverage for the same source: one treats only the two named `e.g.` cases as unresolvable, the other reads the prose as a permissive escape hatch and skips the parse check whenever any operand depends on a chain involving a tool call or a not-yet-loaded file. Conformance tests against any case outside the two examples are unspecified. The mismatch is observable in user-facing diagnostics (`loom/parse/invoke-return-type-mismatch` etc. fire or do not fire on otherwise identical source), so the divergence escapes the loom and breaks the per-site code coverage TYPE-9 and `code-registry-parse.md` rely on.
