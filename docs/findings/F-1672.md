## Problem

The **Unresolvable operands.** paragraph in `type-system.md` phrases its trigger as *"when either side of a compatibility check is past the parser's static view"* — a phrase defined nowhere in the corpus and used nowhere else. Its `e.g.` list of two examples does not close the predicate, so implementers have no rule for cases the examples omit (a `.loom`-callable call rather than `invoke` against a `loom/load/callee-has-errors` callee, an unresolvable alias annotation, an inferred type derived through `match`), and will draw the parse-time / runtime-AJV boundary differently. The first example is also wrong: a Pi-tool call's static return type is always `Result<string, QueryError>` per `tool-calls.md`, so its registered input schema never drives a `⊑` check. Every sibling rule expresses the same skip via the already-defined *statically resolvable* predicate; this paragraph is the lone outlier leaving the terminology unanchored.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `0226b80` (`pi-loom plan: resolve "V15c references an undefined compatibility relation"`, 2026-05-05)
**History:** `git log -S "Unresolvable operands"` and `git log -L /\*\*Unresolvable operands/,+1:docs/spec_topics/type-system.md` both return a single commit, `0226b80`, which added the entire **Type compatibility** section (operational definition, the eight-row structural-cases table, and the **Unresolvable operands.** paragraph) in one move while resolving the plan-review finding *"V15c references an undefined compatibility relation"*. `git blame -L 48,48 -- docs/spec_topics/type-system.md` confirms the paragraph has never been re-touched. The defect — using "past the parser's static view" as the trigger phrase rather than the already-defined *statically resolvable* predicate, plus the dubious Pi-tool `e.g.` — entered with the paragraph in `0226b80` and has been present unchanged for the paragraph's entire lifetime; no later commit narrowed or widened it.

## Solution approach

Rewrite the **Unresolvable operands.** paragraph to trigger on the existing *not statically resolvable* predicate, cross-linked to [Invocation — Static resolution](./invocation.md#static-resolution), in place of the undefined "past the parser's static view" wording. Close the operand list so it covers the unresolvable-callee, unresolvable-annotation, unresolvable-argument-slot, and inference-propagated cases rather than leaving an open `e.g.`. Drop the Pi-tool `e.g.` example, since a Pi tool's static return type is fixed at `Result<string, QueryError>` per [Tool Calls — Return type](./tool-calls.md) and never enters a return-side `⊑` check.

## Solution constraints

- Confine the edit to the **Unresolvable operands.** paragraph; the sibling rules that cite it (`invocation.md`, `functions.md`, `tool-calls.md`) consume it by reference and must not be made to restate the rule.
