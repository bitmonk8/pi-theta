## Problem

The `**Loom return type.**` paragraph at `functions.md#loom-return-type` says return-type inference reconciles the tail-expression type with the operand type of every "reachable" early `return expr`, but the spec defines no reachability predicate for the type system to consume. The cross-referenced `return.md` pins only the converse — code after a `return` is unreachable and emits the `loom/parse/unreachable-code` warning — while the parallel LUB sites (`match` arms, ternary branches, array literals) carry no reachability concept, so the qualifier resolves nowhere. Two readings — every syntactically-present `return` contributes, versus dead-code `return` operands are pruned first — yield different inferred types, divergent `loom/parse/return-no-common-type` firing, and different types observed at cross-loom `invoke<Schema>` static checks.

## Issue introduction

**Verdict:** introduced-by-fix
**Introducing commits:** `08886d9` (`pi-loom spec: resolve "loom return-type inference does not reconcile early-return operand types"`, 2026-06-07)
**History:** Before `08886d9` the `**Loom return type.**` paragraph was a single sentence ("inferred from its body using the same rule as a function: the type of its tail expression, wrapped in `Result<T, QueryError>` if any `?` appears in the body") with no `return`-operand reconciliation, so no reachability qualifier was needed. Commit `08886d9` rewrote the paragraph to introduce the LUB-over-tail-and-`return`-operands discipline that closed the prior "does not reconcile early-return operand types" finding; in the same edit it added the qualifier "every **reachable** early `return expr`" without defining `reachable` and without a companion edit in `return.md` (whose only reachability mention remains the pre-existing `loom/parse/unreachable-code` warning on code *after* a `return`). The reconciliation was the fix; the undefined qualifier was the dropped stitch. `git log --follow docs/spec_topics/functions.md` shows no later commit has touched the paragraph.

## Solution approach

Delete the bare qualifier "reachable" from the `**Loom return type.**` paragraph and state that inference includes the operand type of every early `return expr` syntactically present in the body, regardless of static reachability — the LUB-site default the parallel `match`-arm, ternary, and array-literal rules already follow. Keep the existing cross-reference to `return.md`. Clarify that loom 1.0 defines no reachability predicate the type system consumes and that `loom/parse/unreachable-code` is an editorial warning rather than an inference input, so a `return` in code the parser warns is unreachable still contributes to the LUB and can still trigger `loom/parse/return-no-common-type`.

- Out of scope: the same-paragraph `"must declare Result<_, QueryError>"`-vs-infer contradiction and the `_`-in-type-argument-position syntax issue, each owned by a separate same-cluster finding.
