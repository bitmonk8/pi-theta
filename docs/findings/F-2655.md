## Finding

SLSH-5 (`slash-invocation.md`, "Chain attribution") is a normative MUST: for an `invoke_callee` cascade, renderers MUST append, per hop, the literal text ` from <callee_path> invoked at <parent_path>:<line>`, walking `inner` leaf-first. Two of the three interpolated values per hop have no carrier in the spec's data model:

- `<callee_path>` **is** carried — `InvokeCalleeError` (`errors-and-results/queryerror-variants.md#queryerror-variants`) declares `callee_path: string` on the wrapper.
- `<parent_path>` and `<line>` are **not** carried anywhere. The `InvokeCalleeError` schema is `{ kind, message, callee_path, inner: QueryError }` — it has no parent-path field and no call-site-line field, and the `### Notes` block states each variant "carries only its meaningful fields."

SLSH-5 sources the two missing values from "the immediate parent's invocation record" / values "recorded at the invocation site (per [Invocation], 'Resolution')". But Invocation's *Resolution* section only defines `realpath` normalisation and discovery-root containment for the **callee** path; it never defines a per-`invoke`-hop record that captures the parent loom's post-`realpath` path together with the 1-indexed source line of the call-site token. A corpus-wide search finds no normative definition of any such "invocation record" — the phrase appears only inside SLSH-5 itself, referring forward to a structure that is never established.

The consequence is that SLSH-5's rendered suffix — a byte-exact, spec-versioned surface under SLSH-4 — cannot be produced from the data the spec actually defines. An implementer reading the error model has no signal that the runtime must capture and thread per-hop `(parent_path, line)` provenance through the invoke chain; the renderer's input simply does not exist in the documented model. Notably, the plan already carries a dedicated producer seam (`V15g`, "Invoke invocation-record provenance seam") and a consumer (`V12b`) for exactly this record, confirming the metadata is required at build time — yet the normative spec never gives that record a home.

## Issue introduction

**Verdict:** introduced by a single later commit
**Introducing commit:** `2d217b1` (2026-05-06) — "pi-loom spec: resolve \"Call chain suffix is illustrative, not normative\""
**History:** The chain suffix originated as illustrative prose — an ellipsis-prefixed, parenthesised example (`"... from child.loom invoked at parent.loom:42"`) carrying no normative obligation, so no data-model carrier was implied. `2d217b1` promoted it to the normative SLSH-5 MUST (` from <callee_path> invoked at <parent_path>:<line>`), introducing the per-hop `<parent_path>`/`<line>` rendering obligation while attributing the values only to paths "recorded at the invocation site (per Invocation, Resolution)" — a section that defines callee-path `realpath` normalisation but no per-hop record. The `InvokeCalleeError` schema has carried `{ kind, message, callee_path, inner }` and no source-position fields since inception (`git log -S "InvokeCalleeError"` shows only the later `8455d5e` wire-kind rename, which left the field set unchanged). A subsequent commit `1ccbb75` (2026-06-07) refined the `<line>` definition (call-site token vs receiving binding, `.loom`-callable bare-identifier form) but still sourced it from "the immediate parent's invocation record" without ever defining that record. The gap is therefore the normalisation step `2d217b1` performed on the renderer side without a matching extension of the data model.

## Solution Space

**Shape:** multiple

### Option A — Establish a per-hop invocation-record data model in the spec

- **Approach:** Add a normative definition of the per-`invoke`-hop invocation record — the parent loom's post-`realpath` absolute path plus the 1-indexed source line of the call-site token (the `invoke(` token of a literal call, or the callee-name identifier of a `.loom`-callable bare-identifier call) — owned by `invocation.md` (alongside *Resolution*, which already owns the `realpath` path semantics SLSH-5 cites). SLSH-5 then reads `<parent_path>` and `<line>` from that record by anchor, leaving `<callee_path>` on the `InvokeCalleeError` wrapper as today.
- **Spec edits:** In `invocation.md`, define the per-hop record and anchor it; in `slash-invocation.md`, repoint SLSH-5's "recorded at the invocation site" / "the immediate parent's invocation record" phrases to that anchor.
- **Pros:** Preserves the full operator-facing chain suffix and all worked examples; aligns the normative spec with the plan's existing `V15g` producer / `V12b` consumer split; keeps the record off the `QueryError` wire schema, honouring the "each variant carries only its meaningful fields" design.
- **Cons:** Introduces a new runtime-side data structure to the spec and must pin its ownership boundary against the `QueryError` model.
- **Risks:** If the record is under-specified (e.g. multi-line call line attribution, `.loom`-callable identifier line), implementers still diverge; the definition must be as precise as SLSH-5's existing `<line>` prose.

### Option B — Narrow SLSH-5 to the metadata the error model already carries

- **Approach:** Restrict the chain suffix to values present on `InvokeCalleeError` (`callee_path`, `message`, leaf `kind`), dropping `<parent_path>:<line>` from the rendered text.
- **Spec edits:** Rewrite SLSH-5's suffix template and worked examples to omit `invoked at <parent_path>:<line>`.
- **Pros:** No new data model; the renderer becomes producible from the documented `QueryError` alone.
- **Cons:** Removes call-site provenance from the multi-hop chain attribution, materially degrading the operator's ability to locate the failing `invoke` in a deep cascade; contradicts the plan's `V15g` provenance seam, which would then have no consumer.
- **Risks:** Loses debugging fidelity that the spec deliberately added in `2d217b1`/`1ccbb75`.

### Recommendation

Take Option A: define the per-hop invocation record normatively in `invocation.md` and have SLSH-5 read `<parent_path>` and `<line>` from it. This keeps the established chain-attribution surface intact and matches the plan's already-built producer/consumer split, which treats the record as a runtime-side seam distinct from the `QueryError` wire schema. Do not fold `parent_path`/`line` into `InvokeCalleeError` — that contradicts the variant-minimalism design and would put runtime-internal source positions on the wire-contract error type. Watch the precision of the line rule for `.loom`-callable bare-identifier calls and multi-line call sites, which SLSH-5 already pins and the record definition must not contradict.

## Consequence

**Severity:** correctness

The SLSH-5 chain suffix is a byte-exact normative surface (SLSH-4 declares wording changes spec-versioned breaking changes), yet the spec never establishes the `(parent_path, line)` data the renderer interpolates per hop. Two reasonable implementers diverge: one threads a per-hop provenance record and produces the full suffix, another — finding no such field in the `QueryError` model — omits the suffix or fabricates the values. The operator-facing chain attribution, the spec's primary multi-hop invoke debugging aid, is the observable that breaks.
