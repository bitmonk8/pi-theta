## Finding

The four cross-cutting exemption mechanisms in `conventions.md` each key their exemption on a comment placed on a single source line, using the phrases "its line" or "same-line comment":

- *No globals, statics, singletons* (ambient ban): "A direct reference is an **exempt ambient site** only if its line carries a `// allow-ambient: <primitive> — <seam>` comment".
- *Specific exception types only* (broad catch): "an **exempt broad-catch site** whose line carries a `// allow-broad-catch: … — <spec-page>` comment".
- *Sequential by default* (Promise combinators): "each entry MUST carry a same-line comment `// allow: <REQ-ID-or-code-keyed-area-token> — <spec-page>`".
- *Sequential by default* (blocking runtime): "permitting a call only where its line carries a `// allow-sync: <reason>` comment".

None of the four defines what "the line" is when the exempted construct spans multiple source lines — a `catch (e: unknown)` whose clause keyword and binding sit on different lines, a `Promise.allSettled([` call whose combinator token and argument list wrap, an ambient read inside a multi-line member-access chain, or a `*Sync` call split across a wrapped argument list. The exemption discovery is done in two independent places that must agree: the ESLint rules and the H3a identifier-keyed scan decide whether a site is exempt, and the H5a/H5c closing-gate reconciliation scans the same comments as the authoritative allow-list set. If "the line" means the line carrying the exempted token in one component and a different line in the other (e.g. the line carrying the comment), the two components classify the same multi-line site differently.

The fix the source proposes is correct and well-scoped: add a single definition of "the line" at the top of the Cross-cutting rules section, applying to all four families, so the comment-placement contract is unambiguous for multi-line expressions and every consumer (the lint rules, the ambient scan, and the closing-gate reconciliation) reads the same site classification.

**History:** `git log -S 'its line'` / `-S 'same-line comment'` / `-S 'allow-ambient'` / `-S 'allow-sync'` on `docs/plan_topics/conventions.md` shows the four allow-* comment families were each given their single-line comment representation in a separate allow-list-representation fix, on different dates. Each fix independently adopted the "its line" / "same-line comment" phrasing to anchor its own family's exemption, and none defined "the line" for a multi-line construct. The cross-family ambiguity is therefore an emergent property of these convergent representation fixes — no single commit introduced it; it surfaced once all four families standardised on the same under-specified phrase.

## Solution Space

**Shape:** single

### Recommendation

Add one definition near the top of the `## Cross-cutting rules (every leaf)` section of `docs/plan_topics/conventions.md`, before the four bullet rules, pinning what "the line" / "the same line" means for every `// allow-ambient:`, `// allow-broad-catch:`, `// allow:`, and `// allow-sync:` comment: the qualifying line is the source line containing the primary exempted token — the `catch` keyword for `allow-broad-catch`, the banned ambient-primitive identifier for `allow-ambient`, the Promise-combinator token (`Promise.all` / `race` / `any` / `allSettled`) for `allow:`, and the `*Sync` callee identifier for `allow-sync` — and a comment on a continuation line of the same expression does not satisfy the exemption.

State that this one definition governs all four families and is the contract both the enforcement scans (the H2a ESLint rules and the H3a identifier-keyed scan) and the H5a/H5c closing-gate allow-list reconciliation read, so the two sides classify a multi-line site identically. The existing per-rule "its line" / "same-line comment" phrasings can then point to this definition rather than each restating it.

## Consequence

**Severity:** correctness

For any multi-line exempted construct, the ESLint/H3a scan that decides whether the site is exempt and the H5a/H5c reconciliation that audits the allow-list comment can resolve "the line" differently, producing either a spurious CI failure on a legitimately-exempt multi-line site or a silently-unaudited exemption that grep-based allow-list audits miss. Two implementers building the H2a/H3a enforcement side and the H5a/H5c reconciliation side would reasonably pick incompatible definitions, so the gates would not agree on what counts as an exempt site.
