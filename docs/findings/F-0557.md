## Finding

The paired test-first leaf `V13c-T` and its implementation leaf `V13c` describe the same typed depth-6 co-fire vector with different mechanism specificity. `V13c`'s depth-6 bullet reads "a depth-6 typed-query response **trips the loom-owned depth walk (`V5e`) before AJV** and ceiling #4 surfaces as `Err(QueryError { kind: "validation", cause: "schema_validation" })` (`schema_keyword: "maxDepth"`)…". `V13c-T`'s corresponding bullet drops that mechanism clause and states only the outcome: "ceiling #4 surfaces on the typed-query response as `Err(QueryError { kind: "validation", cause: "schema_validation" })` (`schema_keyword: "maxDepth"`)…".

`V13c-T` is the authoritative authoring of the failing tests (its **Ships when** is "the tests above exist, compile, and fail red for the intended reason"); the impl leaf restates the same Tests block. Because the `-T` bullet specifies only the error surface and not the ordering, a test written from it alone is satisfied by a non-conforming implementation that relies on AJV to emit a coincidentally-identical `schema_keyword:"maxDepth"` error, never exercising the loom-owned depth walk that `V5e` exists to own. The loom-owned-walk-runs-before-AJV property is the load-bearing behaviour of `V5e` (per its **Adds.**: "run before AJV at the five enforcement sites"); the live `V13c` query surface is exactly where that ordering is observable, so the assertion is witnessable on the `V13c-T` carrier.
