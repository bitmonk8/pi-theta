## Finding

The *Infrastructure-failure handling* paragraph in `docs/spec_topics/pi-integration-contract/audit-wire-and-canary.md` reasons about the audit's parse-/syntax-failure obligation as "**not** duplicative with the project-wide `npm run typecheck` gate", using that gate as load-bearing established fact in the non-duplication argument (both gates must be green for `npm test` to be green; the audit gate is what survives a "mis-wired or temporarily skipped" typecheck script). However, the spec corpus never enumerates that script: the closest anchor, `host-prerequisites.md` §`tsc-toolchain-floor`, pins the `tsc` toolchain floor and the build-time brand-string assertion but does not name any `npm` script, and the same file's `npm test` paragraph explicitly states "PIC does not pin a separate `npm` script for the audit". The phrase "project-wide `npm run typecheck` gate" therefore points at a contract the spec corpus does not own — neither defined here nor cross-referenced to an owning location.

A reader who treats the spec corpus as self-contained (the May 2026 corpus-direction sweep that removed plan-side `h1-scaffold.md` references from the spec was the explicit commit to this property) cannot decide what the gate is, what its scope is ("project-wide" against which file set?), or which spec rule guarantees its existence, and so cannot evaluate the audit's non-duplication argument on its own terms.

## Issue introduction

**Verdict:** introduced-then-regressed
**Introducing commits:** `e5cd514` (Sat May 9 2026, *pi-loom spec: resolve "T35 SDK capability inventory closed-set negative-direction audit"*) for the original load-bearing reference; `9e53613` (Tue May 26 2026, *pi-loom spec: resolve T03/T24a/T25/T26 corpus-direction sweep — drop H1 / plan_topics / plan-leaf-ID references from spec corpus*) for the regression that severed the anchor.
**History:** The "project-wide `npm run typecheck` gate" phrase entered the spec at `e5cd514` as part of the same edit that introduced the inventory-closure audit's *Infrastructure-failure handling* paragraph. At that revision the phrase was qualified — "the project-wide `npm run typecheck` gate per [`h1-scaffold.md`](../plan_topics/h1-scaffold.md)" — so the load-bearing reference had a named owner, even though that owner lived in the plan. Commit `9e53613` then ran a corpus-direction sweep across the spec set that, per its log message, dropped H1 / `plan_topics/` / plan-leaf-ID references from spec prose; the audit-wire-and-canary diff in that commit deleted the trailing `per [`h1-scaffold.md`](...)` qualifier from this exact phrase while leaving the bare load-bearing reference behind, with no substitute in-corpus anchor added. The defect is therefore the residue of a corrective sweep that, on this one site, removed the only cross-reference that grounded the phrase without rerouting it to an in-corpus owner — the gate itself has always been real, but its in-corpus anchor has been absent since `9e53613`.

## Solution Space

**Shape:** multiple

### Option A — Enumerate `npm run typecheck` in `host-prerequisites.md` §`tsc-toolchain-floor`

**Approach.** Add one sentence to the `tsc-toolchain-floor` paragraph naming `npm run typecheck` as the `tsc`-invocation entry point that surfaces the build-time brand-string assertion at typecheck time, and replace the bare phrase in *Infrastructure-failure handling* with an anchored cross-reference to it (e.g. "the project-wide typecheck gate pinned at [`tsc-toolchain-floor`](./host-prerequisites.md#tsc-toolchain-floor)").

**Spec edits.**
- `host-prerequisites.md` §`tsc-toolchain-floor`: append a sentence pinning that the project-wide typecheck gate is exposed as the `npm run typecheck` entry point, which invokes the pinned `tsc` toolchain across the loom source tree.
- `audit-wire-and-canary.md` *Infrastructure-failure handling*: replace "the project-wide `npm run typecheck` gate" with the anchored cross-reference; keep the non-duplication argument otherwise unchanged.

**Pros.** Makes the spec self-contained on its own terms — the load-bearing reference now grounds at a named PIC anchor, and the non-duplication argument can be evaluated without leaving the corpus.

**Cons.** PIC currently keeps npm-script wiring out of scope ("PIC does not pin a separate `npm` script for the audit", same page). Enumerating one npm script in PIC but not the other re-opens that boundary asymmetrically.

**Risks.** A future contributor reads the new PIC pin as authorising further npm-script enumeration in PIC and the boundary erodes; mitigated by phrasing the addition as a single-script pin justified by the audit-side load-bearing reference rather than as a general permission.

### Option B — Rephrase the audit's non-duplication argument to cite `tsc`, not the npm script

**Approach.** Drop the `npm run typecheck` phrase from *Infrastructure-failure handling* and reword the non-duplication argument to talk about the `tsc` typecheck pass pinned at `tsc-toolchain-floor`. The argument's substance is unchanged — two gates catch a TypeScript syntax error in `src/` (the `tsc` pass at typecheck time; the audit's walker at audit time), and a green `npm test` requires both — but the bare-npm-script reference disappears, so nothing in PIC needs to enumerate it.

**Spec edits.**
- `audit-wire-and-canary.md` *Infrastructure-failure handling*: replace "the project-wide `npm run typecheck` gate" with "the project-wide `tsc` typecheck pass pinned at [`tsc-toolchain-floor`](./host-prerequisites.md#tsc-toolchain-floor)"; replace "the typecheck script is mis-wired or temporarily skipped" with "the typecheck pass is mis-wired or temporarily skipped"; keep every other clause of the paragraph unchanged.

**Pros.** Preserves the PIC boundary that holds npm-script wiring out of scope. The substantive obligation (two-gates-required-for-green) survives intact and now grounds at an existing in-corpus anchor.

**Cons.** Leaves the unrelated `npm run typecheck` mention in `version-bump-intro.md` step 1 (a contributor-procedure command rather than a load-bearing assumption) un-grounded against PIC. That mention is not on the same footing — it is an imperative the contributor follows, not a reference reasoning about an existing gate — so the asymmetry is principled, but a future reviewer may still object.

**Risks.** Minimal. The rewording is local and the PIC anchor it cites already exists.

### Recommendation

Take **Option B**. PIC explicitly holds npm-script wiring out of scope on the same page that hosts the defect ("PIC does not pin a separate `npm` script for the audit"), so re-introducing a single npm-script pin under Option A would create a localised contradiction the audit page itself just disclaimed. Option B preserves that boundary, grounds the non-duplication argument at an existing in-corpus anchor (`tsc-toolchain-floor`), and is a one-paragraph local rewording. The implementer should keep every other clause of *Infrastructure-failure handling* intact — the rewording targets only the two "gate" / "script" mentions inside the non-duplication argument, not the `npm test` sentences elsewhere in the paragraph (those name the test-side gate PIC genuinely pins, and are out of scope).

## Consequence

**Severity:** advisory

A reader cannot ground the non-duplication argument on its own terms. In practice an implementer infers the gate from convention (or from the plan, which is out of corpus), so no rule is mis-built; the cost is reviewer friction and a self-containment hole in a corpus that was explicitly swept for that property.
