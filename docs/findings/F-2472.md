## Finding

The **Multi-line** rule in `descriptions.md` says: "Consecutive `///` lines are joined with newlines into one description string. Common leading whitespace inside the description is stripped (same algorithm as query-template dedent)." The same page's **No transformation** rule then promotes the result to a byte-exact contract: "Loom emits description text byte-for-byte into the lowered schema; no escaping, dedenting, or wrapping is performed beyond the multi-line join and common-leading-whitespace strip defined above." A byte-exact `description:` output is therefore claimed, but the operation that produces it is defined only by an informal prose reference ("query-template dedent") with no link.

The referenced algorithm does exist and is fully normative — it lives in `query/query-forms.md` under QRY-7 ("Dedent and newline-trim — normative behaviour", anchor `#qry-7`), which pins whitespace-only-line handling, the longest-common-literal-prefix rule, the `{U+0020, U+0009}` dedent alphabet, and LF-only line splitting. An implementer reading `descriptions.md` alone cannot reach those rules: there is no anchor to follow, and the cited term is the informal phrase "query-template dedent" rather than a rule identifier.

A second, subtler gap is scope. QRY-7 bundles **two** normalisations — newline-trim **and** dedent — and newline-trim is specific to the backtick `@`-template form (it strips a newline immediately after the opening backtick / before the closing backtick). Descriptions have no backticks and are built by joining `///` lines, so it is undefined whether "the same algorithm" pulls in newline-trim at all, and whether the first joined line participates in the common-prefix computation. Both questions must be answered for the byte-exact claim to be reproducible.

## Solution Space

**Shape:** single

### Recommendation

In `descriptions.md`, replace the parenthetical "(same algorithm as query-template dedent)" with an explicit anchored link to the dedent rules and a one-clause scope note. Concretely, in the **Multi-line** bullet:

- Link the dedent step to [`query/query-forms.md` QRY-7](./query/query-forms.md#qry-7) by anchor (the existing corpus convention — `descriptions.md` already cross-references `errors-and-results/queryerror-variants.md#queryerror-variants` by anchor and explicitly avoids restating it).
- State that descriptions inherit **only** the common-leading-whitespace dedent portion of QRY-7 (the whitespace-only-line normalisation, longest-common-literal-prefix rule, and `{U+0020, U+0009}` dedent alphabet), and **not** the backtick newline-trim, which is specific to `@`-templates.
- State that every joined `///` line participates in the common-prefix computation (there is no leading/trailing line to trim, since the description is built by the multi-line join, not by backtick delimiters).

Keep the cross-reference rather than restating the full algorithm inline: `descriptions.md` already declares drift-avoidance as a page principle, and QRY-7 is the single normative home for the dedent rules. Implementer edge case to watch: an empty `///` line is already specified to "become a blank line", which must line up with QRY-7 rule 1 (whitespace-only lines are ignored for the common prefix and normalised to empty) — the scope note should make that alignment explicit so the byte-exact output is unambiguous.

## Consequence

**Severity:** advisory

Implementers who locate QRY-7 will converge on the dedent step, so a working implementation is reachable. The exposure is friction plus a narrow byte-exactness risk: because the cited "query-template dedent" bundles newline-trim, an implementer could disagree on whether newline-trim or first-line participation applies to a joined description, and any such disagreement breaks the page's own byte-for-byte stability guarantee.
