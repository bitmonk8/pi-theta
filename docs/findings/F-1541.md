## Finding

The prompt-mode driver chooses between two send forms from a single state observation: `pi.sendUserMessage(text)` when `ctx.isIdle()` is true, and `pi.sendUserMessage(text, { deliverAs: "steer" })` when the agent is mid-stream. The page itself pins that the wrong choice is not benign — "calling the no-options form mid-stream throws" — and that synchronous throw is then mapped by PIC-50 to `Err(QueryError { kind: "transport", ... })`. So if the session's idle/mid-stream state changes between the `isIdle()` read and the `sendUserMessage` call, the runtime either issues the no-options form into a now-mid-stream session (a spurious transport error) or issues the steer form into a now-idle session (delivery to a turn that no longer exists, with undefined behaviour).

The spec never states that the observation and the call execute as one indivisible step. Under Pi's single-threaded event loop the natural implementation — read `isIdle()`, branch, call `sendUserMessage`, all with no intervening `await` — is in fact race-free, because no turn-lifecycle event can be processed between two synchronous statements. But that invariant is load-bearing and is left implicit: an implementer who inserts any `await` (a log flush, a capability re-check, a yield) between the probe and the call opens a window in which the form can mismatch live state, and nothing in the spec forbids it or defines the resulting behaviour.

PIC-2 (prompt-mode sequential execution) does not close this gap. PIC-2 governs *cross-dispatch* overlap — that two slash-dispatch bodies against the same session cannot interleave their snapshot/restore windows — not the *within-call* window between an `isIdle()` read and its immediately-following `sendUserMessage`. The page's other host-behaviour dependencies (the `waitForIdle()` no-rejection posture, the driven-turn commit-ordering posture) are each recorded explicitly as consumption postures; the observe-then-act atomicity that the form selection rests on is the one such dependency left unwritten.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `fae85d3` ("Tighten spec for implementation-plan readiness", 2026-05-04) introduced the stream-state-driven form selection (`pi.sendUserMessage(text)` / `{ deliverAs: "steer" }` "if the agent is mid-stream"); `c993401` ("pi-loom spec: resolve 'Prompt-mode driver calls a non-existent ctx.sendUserMessage'", 2026-05-04) sharpened it into the explicit `ctx.isIdle()`-probe-then-call mechanism and added the "calling the no-options form mid-stream throws" clause.
**History:** `git log -S` on the form-selection tokens over `docs/spec_topics/pi-integration-contract/conversation-drive.md` and its monolithic predecessor (`spec_topics/pi-integration-contract.md`) shows the atomicity assumption was never stated in any revision. The defect is an omission latent since the form-selection design first appeared (`fae85d3`); the `isIdle()`/throw framing the finding cites was made explicit by `c993401` but introduced no new defect — it carried the same unstated invariant forward. This is a since-inception gap, not a regression a later commit broke.

## Solution Space

**Shape:** single

### Recommendation

Add one sentence to the untyped-query form-selection clause (adjacent to the existing "calling the no-options form mid-stream throws" parenthetical) pinning the atomicity invariant: the runtime MUST observe `ctx.isIdle()` and issue the corresponding `pi.sendUserMessage` form with no intervening `await`, so the observed idle/mid-stream state holds through the call. Tie the rationale to Pi's single-threaded event-loop model — because no turn-lifecycle event is processed between two synchronous statements, the observation cannot be invalidated before the call — and note that this is the within-call counterpart to PIC-2's cross-dispatch serialisation, so the two together leave no window in which the form mismatches live state.

Phrase it in the page's existing register: a normative `MUST` on the synchronous observe-and-act sequencing is sufficient; a separate "consumption posture" callout is not required because the guarantee is loom-side (loom controls whether it yields), not a Pi behavioural presupposition. Edge case for the implementer to watch: any future refactor that makes the driver's send path async between the probe and the call (e.g. an `await`ed pre-send hook) violates this `MUST` and must re-read `isIdle()` immediately before the call.

## Consequence

**Severity:** advisory

The natural synchronous implementation is correct, so a careful implementer is unaffected; the risk is that an implementer who interposes an `await` between the `isIdle()` probe and the `sendUserMessage` call gets an intermittent, non-deterministic wrong-form bug — a spurious `kind: "transport"` error or a steer message delivered to no live turn — with no spec rule flagging the mistake. Stating the invariant converts a latent footgun into a checkable obligation.
