## Problem

The `## Acceptance criteria — PIC-17 active-set install vector` block in `docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md` enumerates only two fixtures — (a) empty callable set on an untyped free turn (install vector `[]`) and (b) forced-respond on a typed-query response turn — but omits the production-nominal shape: a non-empty callable set on an untyped free turn (no respond tool) with a non-empty step-1 `pi.getActiveTools()` snapshot disjoint from the callable set. Under fixture (a) the install vector is `[]`, so the "no snapshot member appears in the install vector" check is vacuously satisfied by emptiness; under (b) the synthesised respond tool dominates the vector, so neither fixture witnesses non-leakage of step-1 snapshot members on the free-turn path. The block's preamble commits to fixture-completeness ("all of which the implementation MUST satisfy"), so the omission is an enumeration gap, not a deliberate scoping choice. As written, the `frontmatter-fields-a.md#tools` "ambient tools are deliberately *not* inherited" invariant is unfalsifiable on its nominal path.

The PIC-17 normative paragraph the block decomposes long predates the acceptance block and is not implicated; the defect is a fixture-enumeration omission at the moment of the acceptance block's authoring, not a regression on previously-present coverage.

## Solution approach

Add a third fixture for the nominal shape to the `## Acceptance criteria — PIC-17 active-set install vector` block: a non-empty callable set on an untyped free turn (no respond tool active), with the step-1 `pi.getActiveTools()` snapshot non-empty and disjoint from the callable set. The fixture asserts the step-2 install vector is exactly `[...loomCallableSetNames]` (order as pinned at the [PIC-17](#pic-17) install-vector site) and contains no member of the step-1 snapshot. The non-empty, disjoint snapshot is the witnessing condition the existing two fixtures lack — without it the non-leakage assertion collapses to a tautology.
