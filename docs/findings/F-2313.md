## Problem

The **Adds** field in `docs/plan_topics/V4a-match-result.md` describes the `match` expression as "exhaustive destructuring over the six pattern forms", reusing the spec section header from `errors-and-results/error-model.md` without the disambiguation the spec attaches: exhaustiveness is not statically checked in loom 1.0, and a `match` that fails to cover the scrutinee at runtime raises `MatchError`. Read in isolation, "exhaustive destructuring" invites an implementer to build a static exhaustiveness checker that rejects non-exhaustive `match` at parse time — the opposite of the spec's runtime-`MatchError` contract. The correct contract is already stated in the leaf's Tests bullet and Ships-when, leaving the Adds the sole carrier of the ambiguous label.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` (pi-loom plan: build/update plan for spec.md + review, 2026-06-10)
**History:** The plan corpus is git-tracked. `git log -L '5,5:docs/plan_topics/V4a-match-result.md'` and `git log -S 'exhaustive destructuring' -- docs/plan_topics/V4a-match-result.md` both resolve to a single content-introducing commit, `c6a664e`, the file's creation. The Adds clause "exhaustive destructuring over the six pattern forms" has carried no no-static-check disambiguation since that commit. Subsequent V4a edits left the gap in place: `b14edfa` (2026-06-11) corrected `match` statement → expression on the same line without adding the disambiguation, and the disambiguation was instead added only to the sibling fields — Ships-when in `516fd86` (2026-06-11) and `6135143` (2026-06-17), and the Tests bullet in `02654f5` (2026-06-17) — deepening the asymmetry between the Adds and the rest of the leaf.

## Solution approach

Clarify the **Adds** `match` clause in `docs/plan_topics/V4a-match-result.md` so it carries the same no-static-check signal already present in the Tests bullet and Ships-when: state that loom 1.0 performs no static exhaustiveness check and that a runtime scrutinee matching none of the six pattern forms raises `MatchError` (`loom/runtime/match-error`). The Adds must no longer present "exhaustive destructuring" as a standalone label.

## Solution constraints

- Out of scope: the spec is read-only for this fix — the disambiguation already exists in `errors-and-results/error-model.md`.
