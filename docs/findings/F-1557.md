## Finding

DISC-3 (`docs/spec_topics/discovery/discovery-sources.md`) states the case-collision tie-break as "the lexicographically-first path under case-sensitive byte comparison wins" without saying which form of the path the comparison is performed on (absolute vs. relative; OS-native vs. POSIX-normalised). The applicable form — "the normalised forward-slash form described under 'Path literals' in Lexical Structure" — is given two sentences later, after the cross-source-priority caveat, and is positioned as it were governing only general path comparison rather than the tie-break itself.

A Windows implementer following only the rule sentence would naturally compare the values returned by the host walker (`C:\\…\\Plan.loom` vs. `C:\\…\\plan.loom`), which is byte-identical to a POSIX comparison for ASCII-only stems and so passes casual testing, but diverges as soon as the colliding files sit in differently-cased parent directories or differ in path depth between sources. Two conforming implementations can therefore pick different winners for the same input on the same OS, and the divergence is silent — both still emit `loom/load/case-collision`.

The fix is purely local to the rule statement: pull the path-form clause into the same sentence as the tie-break verdict so the comparison key cannot be read independently of its normalisation.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the DISC-3 rule sentence to bind the comparison key inline:

> the lexicographically-first path — taken in the normalised forward-slash form described under "Path literals" in [Lexical Structure](../lexical.md), and compared as case-sensitive bytes — wins.

Then delete the trailing "Path comparison uses the normalised forward-slash form described under 'Path literals' in [Lexical Structure](../lexical.md)." sentence at the end of the DISC-3 paragraph (now redundant). Keep the cross-source-priority caveat in place. Edge cases the editor should preserve while making the change: the comparison must remain *within a single discovery source* (the existing "per source" / "intra-source only" wording), and the form chosen is the same form already used by the non-canonical-extension dedup later in the paragraph, so no new normalisation surface is introduced.

## Consequence

**Severity:** advisory

Two conforming implementations can pick different `case-collision` winners on the same input when the parent-directory casing or path depth differs across the colliding entries, and the divergence is undetectable from the diagnostic alone. Most realistic inputs (sibling files in one directory, ASCII-only paths) hide the divergence, which makes the latent disagreement worse than a hard contradiction — it survives casual cross-implementation testing.
