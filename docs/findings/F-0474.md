## Finding

`H4a`'s Adds defines an **in-memory fixture-supply mechanism**: the harness supplies fixture `.loom` content to the extension in memory so a harness-driven leaf's discovery reads from the harness-provided source "rather than the real filesystem — no ambient `src/**` filesystem read, no `FileSystem` seam (`V8b`) dependency, keeping the `H3a` ambient-access scan clean." The mechanism's *constraints* are pinned (in-memory, no ambient read, no `FileSystem` seam), but its *integration point* is not. The plan never states how the factory's discovery actually receives the harness-provided content, nor whether doing so requires a collaborator outside the `H3a` runtime seam set (`Checkpoint`, `SchemaValidator`, `Clock`, `FileSystem`, `FileWatcher`, `TokenEstimator`, `IdSource`).

This matters because the production discovery path is filesystem-backed: the discovery walk deepened by `V10a` reads through the `FileSystem` seam (`~/` expansion via `FileSystem.homedir()`, `readtext`/`readdir`, etc.). `H4a` simultaneously forbids the harness path from using that seam. An implementer is therefore forced to invent (a) a source-provider collaborator that discovery consumes in place of a direct `FileSystem` read, and (b) a harness-vs-production selection rule deciding which backing implementation is wired in. Two reasonable implementers will diverge on both: one may add a new injected seam to the `H3a` graph (changing the ambient-access scan surface), another may bolt an ad-hoc parameter onto the discovery entry point that is not reusable across `M`/`M-T`, and a third may reach for a primitive that trips the `H3a` ambient-access scan precisely because the "clean scan" requirement was stated as an outcome but not as a construction.

`M` and `M-T` are the immediate consumers ("the fixture `.loom` this single-source discovery reads is supplied **in-memory by the `H4a` harness**"), so the ambiguity propagates directly into the narrowest end-to-end vertical that must demonstrate the pipeline.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H4a-factory-shell-and-harness.md`, extend the **in-memory fixture-supply mechanism** sentence in the `Adds.` field to pin the injection contract. State, in substance:

- The factory's discovery obtains its `.loom` source through a **constructor-injected source-provider collaborator** that discovery consumes, rather than by reading a host primitive or the `FileSystem` seam directly.
- This source-provider is a **discovery-layer collaborator distinct from the `H3a` host-seam set** (`Checkpoint`/`SchemaValidator`/`Clock`/`FileSystem`/`FileWatcher`/`TokenEstimator`/`IdSource`); it is not a new `H3a` seam, so `H3a`'s seam graph and its ambient-access scan are unchanged. (This is what keeps the "scan stays clean" property a construction rather than an aspiration.)
- **Production discovery** (the walk deepened by `V10a`) binds this collaborator to a `FileSystem`-seam-backed implementation; the **harness** binds it to the in-memory fixture-supply implementation. Selection between the two is the choice of which implementation is injected at the factory's construction site — not a runtime branch inside the discovery code.
- The collaborator's exact member signatures remain the implementer's to finalise, consistent with the leaf's existing stance that the harness API's exact shape is the implementer's to finalise.

Then confirm the "harness-supplied source" wording in `M`'s and `M-T`'s `Adds.` references this injected collaborator, so the consumer leaves and the defining leaf name the same seam.

Edge case the implementer must watch: the production binding for this collaborator routes through `V8b` (`FileSystem`), while the harness binding must not — the selection point therefore has to exist before `V8b` lands so that `M`/`M-T` (which list no `FileSystem` dependency) can wire only the in-memory implementation.

## Consequence

**Severity:** correctness

`M`/`M-T` depend on `H4a` and cannot be implemented unambiguously while the injection contract is unstated: the implementer must invent a source-provider collaborator and a harness-vs-production selection rule, and a wrong choice either trips the `H3a` ambient-access scan (the exact failure the requirement exists to avoid) or produces a non-reusable per-leaf shape. Two reasonable implementers diverge, and the divergence is discovered only when `M`/`M-T` are built against an `H4a` harness whose discovery wiring does not match what those leaves assumed.
