## Finding

`loom/load/host-incompatible` carries a `details.member = "<path>"` sibling field on the `"sdk-capability-missing"` arm, naming the first member in the normative iteration order whose `typeof <path> === "function"` check returned false (pinned in [capability-probe.md Step 0(c)](../../docs/spec_topics/pi-integration-contract/capability-probe.md), `On an "sdk-capability-missing" failure ...` paragraph). The `"abortsignal-shape"` arm has no equivalent: Step 0(b) only specifies that `details.observed` / `details.required` depend on the failing member's `kind` (the `typeof <path>` string vs `"function"`, or the `"absent"` vs `"present"` pair for `prototype-property` members), and the rendered message template `host incompatible (abortsignal-shape): observed undefined, required function` resolves identically whether the missing member is `AbortSignal.any`, `AbortSignal.timeout`, `AbortSignal.prototype.throwIfAborted`, or any of the other six `typeof`-checked members in the kind table. Operators triaging an `abortsignal-shape` failure have no payload field to key on and no diagnostic-side way to distinguish the failing member.

The asymmetry is load-bearing for operator triage because the kind table in Step 0(b) enumerates eight named members (two constructors, two static methods, two prototype methods, two prototype properties) probed in a fixed iteration order analogous to step (c)'s nine-member loop; without a sibling field the eight failure modes collapse to the same wire payload (modulo the two-value `observed/required` split between `typeof`-checked and `prototype-property` members). The existing `sdk-capability-missing` precedent already establishes the contract (sibling field, deterministic first-failure path naming, no growth of the `kind` discriminator enumeration); applying the same precedent to `abortsignal-shape` is the symmetric extension.

`node-floor`'s `details.observed` (the host's reported `process.versions.node` string, host-derived per §8) and `details.required` (the pinned floor literal `>=22.19.0`) are already pinned in the [`loom/load/host-incompatible` per-`kind` `<observed>` / `<required>` enumeration](../../docs/spec_topics/diagnostics/placeholder-rendering-b.md) table; that arm carries no analogous member-discriminator question (the check is whole-value, not per-member). Only the `abortsignal-shape` member-discriminator gap remains open.

## Issue introduction

**Verdict:** single-commit

**Introducing commits:**
- `a33bd49` (2026-06-03, *pi-loom spec: resolve "host-incompatible sdk-capability-missing discriminator"*)

**History:** The asymmetry was created when `a33bd49` added `details.member = "<path>"` to the `sdk-capability-missing` arm of `loom/load/host-incompatible` (introducing both the normative listed-order short-circuit clause in Step 0(c) and the sibling-field paragraph immediately after the nine-bullet member list). The earlier `c5cc01b` (2026-05-06, *Runtime version / capability mismatch: no failure contract*) seeded `loom/load/host-incompatible` with four kinds (`node-floor`, `abortsignal-shape`, `sdk-capability-missing`, `peer-dep-out-of-range`) and no member discriminator on any arm; at that point the four kinds were symmetric. `fa9f2d5` (2026-06-02, *host-incompatible per-kind observed/required rendering*) added the per-kind `<observed>` / `<required>` enumeration table without touching member discriminators, preserving the symmetry. `a33bd49` then refined `sdk-capability-missing` alone — explicitly noting `details.member` "mirrors the `details.package` precedent the peer-dep kinds establish in step (d)" — without extending the same precedent to `abortsignal-shape`, whose Step 0(b) kind table already enumerated eight named members with the same first-failure short-circuit potential. Later commits (`9371309`, `949d1fc`, `d8d5ad0`, `62d49d6`) refined the peer-dep arms and the `probe-failed` `details.cause` route but left the `abortsignal-shape` member-discriminator gap untouched.

## Solution Space

**Shape:** single

### Recommendation

Extend `details.member` to the `abortsignal-shape` arm by adding a new paragraph to [capability-probe.md Step 0(b)](../../docs/spec_topics/pi-integration-contract/capability-probe.md), immediately after the existing `On an abortsignal-shape failure ...` paragraph, with the following obligations:

- The probe iterates the Step 0(b) kind table **in the row order in which it appears**, stopping at the first member whose check (`typeof <path> === "function"` for the typeof-checked rows, `"<name>" in AbortSignal.prototype` for the prototype-property rows) returns false — the same first-failure short-circuit discipline Step 0(c) already pins. This iteration order is normative and makes the failing member deterministic.
- On failure, the `loom/load/host-incompatible` payload carries `details.member = "<path>"` (e.g. `"AbortSignal.any"`, `"AbortSignal.prototype.aborted"`, `"AbortController.prototype.abort"`) naming the first member whose check returned false. `details.member` is a sibling field that refines the existing `"abortsignal-shape"` kind; the `kind` discriminator enumeration in *On failure: refusal and diagnostic* does **not** grow.
- The `details.observed` / `details.required` substrings remain as the existing paragraph pins them (typeof-string vs `"function"`, or `"absent"` vs `"present"`).

In parallel, mirror three things to the diagnostics-side surface:

- Add `member` to the *Trigger* column's `details: { kind, observed, required, ...optional }` enumeration for the `loom/load/host-incompatible` row in `code-registry-load.md`, alongside the existing `member` reference for `sdk-capability-missing`, explicitly noting it is carried for `abortsignal-shape` too.
- Extend the two existing `abortsignal-shape` test vectors in `placeholder-rendering-b.md` (the `AbortSignal.any` `typeof`-checked example and the `AbortSignal.prototype.aborted` `prototype-property` example) to assert `details.member` equals `"AbortSignal.any"` / `"AbortSignal.prototype.aborted"` respectively, mirroring the existing `sdk-capability-missing` test vector's `details.member === "createAgentSession"` assertion.
- Keep the *per-`kind` `<observed>` / `<required>` enumeration* table unchanged — it pins rendered substrings only, not the sibling discriminator fields, and the closing paragraph already excludes `details.member` from the table's scope.

The Description's framing that `details.observed` / `details.required` are unspecified for `abortsignal-shape` and `node-floor` is a misreading: both are already pinned (Step 0(b) prose for `abortsignal-shape`, the rendering table for `node-floor`). Only the `abortsignal-shape` member-discriminator gap is open, and only that gap needs an edit. No `node-floor` edit is owed.

## Consequence

**Severity:** advisory

Operators triaging an `abortsignal-shape` failure cannot tell from the wire payload which of the eight probed `AbortSignal` / `AbortController` members the host lacks, because the rendered message and the `details.observed` / `details.required` substrings collapse all eight members to at most two distinct payloads. Implementation can still proceed (the probe iteration order is normative and a developer can reproduce locally), but the spec ships an asymmetry against `sdk-capability-missing`'s `details.member` precedent that conformance tests targeting deterministic member-naming will trip on.
