## Finding

The `loom/parse/unsupported-feature` registry row in `code-registry-parse.md` describes its trigger as *"A loom 1.0-deferred or non-Loom syntactic construct (arrow function, spread, optional chaining, `===`, bitwise op, comma op, nested template, etc.)"* — an open-ended set, signalled by "etc." Its message template is fixed as `unsupported syntactic feature: <construct>`, and the **only** rendering rule for `<construct>` is category 3 of `placeholder-rendering-a.md`, which mandates rendering from a **closed** token-name table of 15 JS-syntax constructs (`arrow function`, `spread`, … `await`). That table is declared a closed, build-time-enforced, GOV-7/GOV-8-governed surface.

The two surfaces are inconsistent: the trigger admits constructs the rendering table does not, so a triggering construct outside the table has no defined `<construct>` rendering, breaking the byte-identical-message guarantee (GOV-15) that `placeholder-rendering-a.md` exists to provide.

This is not merely sloppy wording. The plan's own leaves fire this code from at least two emitting sites that are **not** in the closed JS-syntax token table:

- **`V5d` (schema-subset reject gate)** fires `loom/parse/unsupported-feature` for every JSON-Schema construct outside the permitted subset — explicitly including constructs *"absent from the enumerated unsupported-keyword list"*, i.e. a reject-by-default, inherently **open** universe of rejected keywords.
- **`V1b` (literals)** fires `loom/parse/unsupported-feature` for reserved hex/octal/binary/underscore numeric forms.

Neither rejected JSON-Schema keywords nor reserved numeric forms appear in the category-3 token-name table. Two conformant implementers therefore cannot agree on the rendered `<construct>` string for `arr[0o17]` or for a rejected `pattern` keyword: one might invent a token name, another might emit the source token, a third might fail to render — diverging on an observable, GOV-15-governed diagnostic message. The reject-by-default semantics of `V5d` make a pure "enumerate exactly" fix impossible, because the rejected-keyword universe cannot be closed.

## Solution Space

**Shape:** single

### Recommendation

Make `<construct>` rendering total across every emitting site, and stop the registry row from implying an enumeration that cannot exist:

1. In `code-registry-parse.md`, remove "etc." from the `loom/parse/unsupported-feature` trigger description. Replace the open parenthetical with a statement that the construct *universe* (which sources fire this code) is owned by the emitting sites — `expressions.md` "Not supported", the schema-subset reject gate, and the reserved-numeric-literal rules — and that the rendered `<construct>` token is governed by `placeholder-rendering-a.md` §3.

2. In `placeholder-rendering-a.md` §3, keep the closed token-name table as the rendering for the JS-syntax constructs it lists, and add a **deterministic fallback** for emitting sites outside that table so rendering is total: render the offending JSON-Schema keyword name verbatim (reject-gate path) and the offending lexical token verbatim (reserved-numeric-form path). Verbatim source rendering is already the established pattern for `<expr>` in the same category, so this preserves byte-identicality without attempting to enumerate the reject-by-default keyword universe.

A fallback *rule* is itself a single closed rule (it renders open input deterministically, exactly as the existing `<expr>` verbatim-span rule does), so it does not weaken the §3 closure or its GOV-7/GOV-8 posture. Do not attempt to resolve this by enumerating every rejected construct: `V5d`'s allowlist semantics make the rejected-keyword set inherently open, so enumeration would silently drop the reject-by-default cases.

Edge cases the implementer must watch: the reject-gate fallback must cover a keyword the spec has never heard of (reject-by-default), not only the enumerated unsupported keywords; and the numeric-form fallback must render the lexed token after newline-normalisation, consistent with the `<expr>` span rule.

## Consequence

**Severity:** correctness

Constructs that fire `loom/parse/unsupported-feature` from emitting sites outside the closed JS-syntax token table (rejected JSON-Schema keywords in `V5d`, reserved numeric forms in `V1b`) have no defined `<construct>` rendering, so two reasonable implementations produce different diagnostic strings for the same source defect — a divergence on the GOV-15 byte-identical-message contract. The `V7c-T` conformance author cannot write the byte-identical test vector for these cases without a spec answer.
