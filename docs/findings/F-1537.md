## Finding

The binder inference call's closing paragraph in [`docs/spec_topics/pi-integration-contract/binder-inference.md`](../../docs/spec_topics/pi-integration-contract/binder-inference.md) delegates "per-attempt backoff and any server-requested wait (e.g. an HTTP `Retry-After`)" to `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`, and the per-invocation retry-budget paragraph in [`docs/spec_topics/binder/determinism-cancellation-failure.md`](../../docs/spec_topics/binder/determinism-cancellation-failure.md) says the same — but the enumerated `options` bullets that immediately precede the closing paragraph populate only `temperature`, `signal`, the seed key, and `onResponse`. Two implementers writing to this spec will produce two different `complete()` call sites: one passing the SDK's default `maxRetries` (whatever `dist/stream.js` chooses, currently nonzero), the other passing `0` to bypass SDK-internal retries. The implementer cannot tell from the spec which is conforming.

That ambiguity then propagates into the hard-ceiling label. [`docs/spec_topics/hard-ceilings/ceilings-3-and-4.md`](../../docs/spec_topics/hard-ceilings/ceilings-3-and-4.md) HC3-d states the worst-case sum as "3 binder LLM calls per slash invocation". With nonzero SDK `maxRetries`, a single loom-level attempt can issue several provider HTTP calls before `complete()` resolves, so "3 binder LLM calls" can be read as (a) 3 loom-level `complete()` invocations (i.e. up to 3 × (1 + `maxRetries`) HTTP attempts) or (b) 3 provider HTTP attempts in total. The retry-budget paragraph and the `Per-invocation retry-budget` paragraph speak of "calls" without distinguishing the two layers, and the conformance test for HC3-d (`V11f` "at most three LLM calls per invocation") inherits the same ambiguity.

The two halves are inseparable: pinning the `options` values without resolving the accounting unit leaves HC3-d ambiguous, and resolving HC3-d without pinning the `options` values leaves the cap unenforceable. The spec already grants `complete()` retry-timing ownership (see the `complete-retry-and-cancellation-presupposition` paragraph in [`docs/spec_topics/pi-integration-contract/conversation-drive.md`](../../docs/spec_topics/pi-integration-contract/conversation-drive.md)); what is missing is the concrete value loom passes and the loom-level vs HTTP-level accounting boundary for HC3-d.

## Issue introduction

**Verdict:** introduced-by-fix
**Introducing commits:** `d0bcdb2` (2026-06-05) "pi-loom spec: resolve "binder per-invocation retry inter-attempt timing""
**History:** Before `d0bcdb2`, the binder-inference page closed its `complete()` paragraph with "inter-attempt timing is out of scope here" and made no reference to `maxRetries`/`maxRetryDelayMs`; `determinism-cancellation-failure.md` similarly made no claim about per-attempt backoff. The HC3-d wording "3 binder LLM calls per slash invocation (1 initial attempt + 1 transport-class retry + 1 malformed-envelope-class retry)" was introduced at the same time the per-class retry budget was first authored and remained unchanged through `1f4f536` (2026-06-07), which only re-anchored the HC3-a..HC3-e labels. `d0bcdb2` then resolved a different finding ("binder per-invocation retry inter-attempt timing") by delegating per-attempt backoff and `Retry-After` to `StreamOptions.maxRetries`/`StreamOptions.maxRetryDelayMs` in both pages — adding the dependency on those fields without adding them to the enumerated `options` bullet list and without revisiting whether HC3-d's "binder LLM calls" now counts loom invocations or HTTP attempts. The defect is the unfollowed-through edit: a fix that introduced a new SDK-layer behaviour into the contract without pinning loom's values for it or reconciling it with the pre-existing ceiling label.

## Solution Space

**Shape:** single

### Recommendation

Pin both halves in one coordinated edit, in this order so the binder-inference change is in place before the ceiling label is rewritten:

1. In the `runtime populates one complete(...) call per binder attempt` bullet list in `docs/spec_topics/pi-integration-contract/binder-inference.md`, add two new bullets immediately after `options.signal`:
   - `options.maxRetries` is `0`. The binder owns its own per-class retry budget; SDK-internal retries would shadow that budget and would make the HC3-d cap unenforceable at the loom layer (the runtime cannot observe individual HTTP attempts hidden behind a single `complete()` resolution).
   - `options.maxRetryDelayMs` is unspecified (no value passed). With `maxRetries: 0` the field has no effect; loom redefines neither field's default. (The `Retry-After` handling that the `complete-retry-and-cancellation-presupposition` paragraph in `conversation-drive.md` pins is thereby reduced to a no-op for the binder call; that paragraph stays as-is — it documents SDK behaviour, not loom's use of it.)
2. In the closing paragraph of `binder-inference.md`, replace the "delegating per-attempt backoff and any server-requested wait (e.g. an HTTP `Retry-After`) to this call's `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`" clause with: "with no per-attempt SDK retry (this call passes `maxRetries: 0` per the bullet list above), so each loom-budgeted retry corresponds to exactly one provider HTTP attempt."
3. In the `Per-invocation retry budget` paragraph in `docs/spec_topics/binder/determinism-cancellation-failure.md`, replace the "Client-side retry of a *single* underlying attempt … owned by `@earendil-works/pi-ai`'s `StreamOptions.maxRetries` and `StreamOptions.maxRetryDelayMs`" sentence with: "The binder call passes `maxRetries: 0` (see [Binder inference call](../pi-integration-contract/binder-inference.md#binder-inference-call)), so each loom-budgeted retry corresponds to exactly one provider HTTP attempt; no SDK-internal retry layer is interposed between the per-class budget and the underlying HTTP call."
4. In `docs/spec_topics/hard-ceilings/ceilings-3-and-4.md` HC3-d, replace "Worst-case sum: 3 binder LLM calls per slash invocation" with "Worst-case sum: at most 3 provider HTTP attempts per slash invocation, one per loom-issued `complete()` invocation, since the binder call passes `maxRetries: 0` (see [Binder inference call](../pi-integration-contract/binder-inference.md#binder-inference-call))". Leave the parenthetical "(1 initial attempt + 1 transport-class retry + 1 malformed-envelope-class retry)" unchanged — it now correctly refers to provider HTTP attempts.

Edge cases the implementer must watch:

- The `onResponse` callback fires once per provider HTTP attempt. With `maxRetries: 0` it fires at most once per `complete()` resolution, which keeps the "multi-`onResponse` join" rule in `binder-inference.md` consistent (the multi-fire branch becomes unreachable for the binder, but the join rule still applies to other `complete()` consumers).
- `Retry-After` and HTTP 429 are still classified through the provider-error mapping table; they consume the transport-class budget at the loom layer rather than being absorbed by the SDK. This is the intended trade: loom's HC3-d cap stays enforceable at the cost of not honouring per-HTTP-call `Retry-After` waits — `Retry-After` becomes a same-tick re-issue, which is consistent with the "immediate re-issue" rule already in the per-invocation-retry-budget paragraph and with NOCEIL-1.
- The V11f test for HC3-d should assert the cap by counting `complete()` invocations the test harness intercepts; with `maxRetries: 0` that count is also the provider-HTTP-attempt count.

## Consequence

**Severity:** correctness

Without an authored `options.maxRetries`/`maxRetryDelayMs` value, two conforming implementations diverge on how many provider HTTP attempts a single binder invocation issues, which directly perturbs observable behaviour under `Retry-After` and TLS/network flake. Without an HC3-d accounting unit, the "3 binder LLM calls" cap is testable two different ways: one implementation passes the V11f test by counting `complete()` resolutions, the other only by counting HTTP attempts, and the same flake-storm chain can pass one test and fail the other.
