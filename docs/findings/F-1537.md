## Problem

The binder `complete()` closing paragraph in `binder-inference.md` (and the `#per-invocation-retry-budget` paragraph in `determinism-cancellation-failure.md`) delegates per-attempt backoff and any server-requested wait to `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`, but the enumerated `options` bullets that populate the `complete(...)` call list only `temperature`, `signal`, the seed key, and `onResponse` — never `maxRetries` or `maxRetryDelayMs`. Two conforming implementers therefore produce different call sites: one passing the SDK's nonzero default, one passing `0`. That ambiguity propagates into HC3-d in `ceilings-3-and-4.md`, whose "3 binder LLM calls per slash invocation" does not say whether a "call" is a loom-level `complete()` invocation or a provider HTTP attempt, so the V11f cap is testable two incompatible ways. Pinning the `options` value without resolving the accounting unit leaves HC3-d ambiguous; resolving HC3-d without pinning the value leaves the cap unenforceable.

## Issue introduction

**Verdict:** introduced-by-fix
**Introducing commits:** `d0bcdb2` (2026-06-05) "pi-loom spec: resolve "binder per-invocation retry inter-attempt timing""
**History:** Before `d0bcdb2`, the binder-inference page closed its `complete()` paragraph with "inter-attempt timing is out of scope here" and made no reference to `maxRetries`/`maxRetryDelayMs`; `determinism-cancellation-failure.md` similarly made no claim about per-attempt backoff. The HC3-d wording "3 binder LLM calls per slash invocation (1 initial attempt + 1 transport-class retry + 1 malformed-envelope-class retry)" was introduced at the same time the per-class retry budget was first authored and remained unchanged through `1f4f536` (2026-06-07), which only re-anchored the HC3-a..HC3-e labels. `d0bcdb2` then resolved a different finding ("binder per-invocation retry inter-attempt timing") by delegating per-attempt backoff and `Retry-After` to `StreamOptions.maxRetries`/`StreamOptions.maxRetryDelayMs` in both pages — adding the dependency on those fields without adding them to the enumerated `options` bullet list and without revisiting whether HC3-d's "binder LLM calls" now counts loom invocations or HTTP attempts. The defect is the unfollowed-through edit: a fix that introduced a new SDK-layer behaviour into the contract without pinning loom's values for it or reconciling it with the pre-existing ceiling label.

## Solution approach

Add an `options.maxRetries` bullet to the `complete(...)` bullet list at `#binder-inference-call`, authoring the value loom passes — `0` — so SDK-internal retries do not shadow the binder's own per-class retry budget and each loom-budgeted retry corresponds to exactly one provider HTTP attempt. Rewrite HC3-d's "3 binder LLM calls per slash invocation" to name a single accounting layer consistent with that value. Reconcile the `binder-inference.md` closing paragraph and the `#per-invocation-retry-budget` paragraph's `StreamOptions.maxRetries` sentence so all three sites describe the same loom-level-to-HTTP-attempt mapping.

## Solution constraints

- Out of scope: the `complete-retry-and-cancellation-presupposition` paragraph in `conversation-drive.md` — it documents SDK behaviour, not loom's use of it, and must not be modified.
