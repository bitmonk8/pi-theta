## Finding

DISC-1 in `docs/spec_topics/discovery/discovery-sources.md` carves an exclusion against `~user` and ends "— only the bare `~` followed by `/` (or end-of-string)." The "(or end-of-string)" clause means a path consisting of nothing but `~` is, per the rule, an honoured form — but DISC-1 never says what the post-expansion value (the user's home directory) then *means* at any of the `~`-eligible sites the same paragraph enumerates.

Each enumerated site has a non-trivial reading once `~` alone is honoured. A `loomPaths` directory entry resolves to the home directory and, per the entry schema, expands to its non-recursive `*.loom` children — i.e. enumerating every top-level `*.loom` file in the user's home directory, almost certainly unintended. A `--loom` flag component is identical. The global discovery root expression is never authored as bare `~` in practice, but the rule admits it; same for the global settings-file path. No diagnostic, classification, or rejection is wired to this case.

The sibling clarification for `~user` (the negative half of the same sentence) was split out by F-1304 and pins a byte-verbatim passthrough that surfaces via DISC-2. The bare-`~` half — the positive half implied by "(or end-of-string)" — has no parallel pin, leaving two compliant implementations free to either (a) silently scan the home directory for loose `*.loom` files or (b) treat the segment as literal and fail through DISC-2. The author's intent is not recoverable from the text.

## Solution Space

**Shape:** single

### Recommendation

Strike "(or end-of-string)" from DISC-1 so the only honoured form is `~` followed by `/`. State explicitly that a path segment consisting of the single character `~` (no trailing `/`) is **not** expanded — it is passed byte-verbatim to the `FileSystem` seam, exactly as the `~user` form is, and surfaces through the standard DISC-2 failure modes at each site (a `loomPaths` directory entry whose post-resolution value resolves to a path the seam cannot enumerate as a directory of `*.loom` files surfaces `loom/load/wrong-type-source` or `loom/load/missing-source` per DISC-2's table; the same holds for a `--loom` flag component). Forward-link to `id="disc-2"` from the new sentence so the failure-mode wiring is explicit.

Add one bullet to `V10a-T`'s DISC-1 line asserting the bare-`~` vector: a `loomPaths: ["~"]` (or `--loom ~`) entry does **not** expand to the home directory and surfaces via the DISC-2 failure path, matching the parallel `~user` assertion already on that line.

Edge cases the implementer must watch: do not treat the post-strike rule as license to honour `~` at *any* path depth — the rule still applies only to a leading `~` followed immediately by `/`. A path of the form `~/foo` continues to expand exactly as today; the only behaviour change is at the bare-`~` boundary. Keep DISC-1's "implementations MUST NOT read `process.env.HOME` … MUST NOT use any platform-conditional branch" sentence intact — it covers the seam-routing obligation independently of the bare-`~` carve-out.

## Consequence

**Severity:** correctness

Two reasonable implementers will diverge: one will honour `~` as the home directory (per the literal reading of "or end-of-string") and surface the home directory's top-level `*.loom` files as discovered looms; the other will treat bare `~` as a non-honoured segment and surface DISC-2 wrong-type / unreadable failures. The first reading risks silently picking up unintended `*.loom` files placed (or seeded by an unrelated tool) in `$HOME`; the second is operator-visible. No conformance test in `V10a-T` distinguishes them today.
