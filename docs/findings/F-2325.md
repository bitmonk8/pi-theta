## Problem

The witness across `V11a`, `V11a-T`, `V9b`, `V9b-T`, and `V6a` Adds claims that `V6a`'s load-time `model:` resolution (`loom/load/model-unresolved`) and `V11a`'s binder-model resolution (`loom/load/binder-model-unresolved`) "cannot diverge" on the "reference matches no available model" condition, grounding that guarantee solely on instance identity — both paths bind the single matcher instance `V9b` constructs and injects at the load pass, "observed by single-source-of-construction (instance identity), not equivalence-of-outcome." Instance identity proves only that both call sites were handed the same matcher object; it does not prove that each site delegates its no-match verdict to that object's matching method. A faithful implementer can satisfy the instance-identity assertion while one path inlines its own negative-match check, diverging on exactly the condition the plan claims is inviolable. The asserted mechanism is strictly weaker than the stated conclusion, and the gap is invisible to the gate as written.

## Solution approach

Reconcile the witness so the claimed guarantee matches what the test establishes, taking one of two directions consistently. Either narrow the "cannot diverge on the 'reference matches no available model' condition" claim — in `V11a` Adds and Ships-when, the single-matcher cross-resolution reconciliation Tests bullet of `V11a` and `V11a-T`, the model-reference-matcher production-wiring Tests bullet of `V9b` and `V9b-T`, plus `V9b` Adds/Ships-when and `V6a` Adds — down to what instance identity actually establishes; or strengthen those tests to assert that each path delegates its no-match verdict to the injected matcher (equivalence-of-outcome), not merely binds the same instance. Apply the chosen treatment across all witness sites so the Adds prose, Tests bullets, and Ships-when assertions agree.
