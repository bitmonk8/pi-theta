## Finding

`V9f` ([`docs/plan_topics/V9f-tool-registration-lifetime.md`](docs/plan_topics/V9f-tool-registration-lifetime.md)) lists "the no-`unregisterTool` rule" among its **Adds** deliverables, but no **Tests** bullet, source scan, or named observable in either `V9f` or its paired `V9f-T` witnesses it, and the **Ships when** gate does not reference it. The leaf's Tests cover the restore-failure path (`PIC-8`), the snapshot/swap-install-failure path (`PIC-19`), the cache-hit byte-equality protocol (`PIC-44`), the `ToolDefinition.label` derivation, and a vaguely-stated visibility bullet â€” none of which assert that the runtime never relies on an `unregisterTool` call.

The rule is a real spec invariant: [`tool-registration-lifetime.md`](docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md) pins that Pi's registry has no `unregisterTool` API, that `pi.registerTool` entries persist for the process lifetime, and that loom-synthesised tool *visibility* is therefore controlled by the prompt-mode active-set snapshot/restore window and by subagent `dispose()` rather than by unregistering. The whole per-mode wiring design depends on that property (pinned as the registry-persistence presupposition in [`host-prerequisites.md`](docs/spec_topics/pi-integration-contract/host-prerequisites.md)).

Because the property is named as a deliverable but is mechanically un-gated, `V9f` can ship green while the design intent it claims to encode is unverified. The other paired-rule bans in the plan (ambient-primitive access, broad catch) carry a concrete `src/**` source scan as their mechanical witness; the no-`unregisterTool` rule carries none.

## Consequence

**Severity:** advisory

The leaf claims to deliver a design rule it does not gate. At the loom 1.0 Pi-SDK pin the property holds vacuously (Pi exposes no `unregisterTool` to call), so nothing breaks today; the risk is that a future implementer â€” for instance after the spec's noted "if Pi later adds `pi.unregisterTool`" path â€” could introduce an unregister-based teardown that regresses the persistence-vs-visibility design with no test reddening.

## Solution Space

**Shape:** single

### Recommendation

Add a mechanical witness for the no-`unregisterTool` rule to `V9f` and its paired `V9f-T`, and reference it in `V9f`'s Ships-when. The witness is a `src/**` source-scan assertion that no `unregisterTool` (nor `pi.unregisterTool`) call site exists in the production tree â€” the same negative-space style the *No globals, statics, singletons* rule uses for its `// allow-ambient` scan, except no allow-list is needed because the count is zero (Pi exposes no such API at the loom 1.0 pin).

- In `docs/plan_topics/V9f-T-tool-registration-lifetime.md` Tests, add a red-first bullet asserting `src/**` contains zero `unregisterTool` call sites, citing the no-`unregisterTool` rule in [`tool-registration-lifetime.md`](docs/spec_topics/pi-integration-contract/tool-registration-lifetime.md).
- In `docs/plan_topics/V9f-tool-registration-lifetime.md` Tests, add the matching implementation-side bullet, and extend **Ships when** so `npm test` asserts the no-`unregisterTool` scan alongside the `PIC-8` / `PIC-19` / `ToolDefinition.label` criteria it already lists.

The alternative of "assert the prompt-path teardown restores via snapshot" is not the missing piece: that positive mechanism is already asserted by `PIC-8` here and by `PIC-17` on `V9c`, and asserting it again would not witness the *absence* of an unregister path the rule actually claims.
