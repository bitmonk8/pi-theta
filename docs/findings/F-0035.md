## Problem

H1a's toolchain skeleton rests on two package-manager-specific behaviours without ever naming the manager loom's own build/test/CI runs under. The `eslint-plugin-loom-local` devDependency uses a `file:./tools/eslint-plugin-loom-local` specifier whose resolution semantics differ across managers, and H1a's Ships-when treats a green `npm install` as the witness that it resolves; separately, the four `@earendil-works/pi-*` packages are declared in loom's own `devDependencies` so a fresh install provisions them "under loom's own `node_modules`" for V18a/V18b/V18c reads, including V18c's live read of the installed `package.json`. Both behaviours presume npm-style `file:` resolution and a flat/hoisted `node_modules` tree. The `pnpm`/`yarn` language in `host-prerequisites.md` §`#pi-sdk-pin` is consumer-facing (install-time skew detection for hosts installing loom) and does not govern loom's own checkout, whose layout-dependent reads would differ under pnpm's isolated linker.

## Issue introduction

**Verdict:** present-by-omission, accreted across multiple commits
**Introducing commits:** `fe9d03a` (2026-06-11, "resolve Lint-toolchain package identities unspecified") introduced the `file:./tools/eslint-plugin-loom-local` specifier; `c8564bb` (2026-06-12, "resolve Pi SDK never provisioned into build-test environment") introduced the "fresh `npm install` provisions … under loom's own `node_modules`" devDependency reliance.
**History:** H1a was created in `c6a664e` (2026-06-10). The two package-manager-specific reliances were added later, in `fe9d03a` and `c8564bb` respectively, each deepening the dependence on npm-style `file:` resolution and a flat `node_modules` layout. `git log -G 'package manager|pnpm|yarn' -- docs/plan_topics/` returns no commits: no revision of any plan-topics file has ever stated which package manager loom's own dev/CI runs under. The defect is therefore an omission that became material as the reliance was introduced, never closed by a subsequent edit — not a regression a single commit introduced.

## Solution approach

State, in H1a's **Adds**, the package manager loom's own build/test/CI runs under (npm) and the `node_modules` layout (flat/hoisted) that the leaf's `file:` plugin resolution and V18a/V18b/V18c provisioning reads assume, scoping those two reliant guarantees to that manager+layout. Reference `host-prerequisites.md` §`#pi-sdk-pin` to record that its `pnpm`/`yarn` language is the consumer-facing install-time skew-detection contract and does not govern loom's own checkout.
