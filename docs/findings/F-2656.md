## Finding

The SLSH-4 *Per-`kind` formatting* table in `docs/spec_topics/slash-invocation.md` assigns the stable inline labels `SNK-a, SNK-b, … SNK-i, SNK-k` — it runs `a`→`i`, then jumps straight to `k`, omitting `SNK-j`. The introductory prose immediately above the table still reads "Each row below carries a stable inline label (`SNK-a` … `SNK-k`, in the `Label` column)", asserting a span whose endpoints imply a contiguous `a`–`k` set while the table itself is non-contiguous.

`SNK` is a GOV-16-governed inline-label prefix registered to this page. GOV-16's *Retirement section* clause (`governance/stable-inline-labels.md#gov-16`) is a MUST: "All inline-label retirements MUST be recorded in a trailing `## Retired inline labels` section on each page that owns inline-label prefixes, structurally parallel to the `## Retired REQ-IDs` section." `docs/spec_topics/slash-invocation.md` owns the `SNK` prefix but has no `## Retired inline labels` section anywhere in the file. The `SNK-j` gap therefore exists with no governance record accounting for it.

The gap is intentional, not an accident: `SNK-j` formerly carried the `invoke_callee` template row, which was removed because SLSH-5's chain-attribution rule computes the descriptive text from the **leaf** `kind`, and every concrete `InvokeCalleeError` carries a required `inner: QueryError` — there is no inner-less `invoke_callee` for a dedicated row to format (the table's narrative paragraph already states this rationale). The removal is correct, but the retirement was never recorded at the site GOV-16 mandates, and the SLSH-4 prose range was left implying contiguity. A conformance author or auditor reconciling the `SNK` label set against the page cannot tell whether `SNK-j` was retired, never existed, or is missing in error.

**History:** `git log -S 'SNK-j' -- docs/spec_topics/slash-invocation.md` returns exactly two commits. `27b2576` (2026-06-05) coined the `SNK-a … SNK-k` table with a live `SNK-j` row for `invoke_callee`; at that point the sequence was contiguous and the prose range was accurate, so there was no defect. `4c2121d` (2026-06-27) then deleted the `SNK-j` row (`git show 4c2121d` confirms the single-line removal) to resolve finding F-1752 — the `invoke_callee` template is unreachable under SLSH-5's leaf-`kind` rule — and updated the plan-side `V12b`/`V12b-T` bullets and the `coverage-matrix.md` note, but it left the SLSH-4 prose range reading "`SNK-a` … `SNK-k`" and did not add the GOV-16-mandated `## Retired inline labels` section. The defect is the bookkeeping omission in that single removal commit.

## Solution Space

**Shape:** single

### Recommendation

Complete the `SNK-j` retirement bookkeeping on `docs/spec_topics/slash-invocation.md` in two coupled edits:

1. Add a trailing `## Retired inline labels` section, structurally parallel to the `## Retired REQ-IDs` sub-table (columns `| Label | Retired in | Replaced by | Reason |`), with one row:
   - **Label:** `SNK-j`
   - **Retired in:** the 7-character abbreviated SHA of the commit that removed the `SNK-j` row (the value-space the `## Retired REQ-IDs` *Retired in* column uses — a 7-char SHA or a release tag).
   - **Replaced by:** SLSH-5 (the chain-attribution leaf-`kind` rule).
   - **Reason:** the `invoke_callee` template is unreachable — every concrete `InvokeCalleeError` carries a required `inner: QueryError`, so SLSH-5 always recurses to the leaf `kind` and renders the leaf's row; there is no inner-less `invoke_callee` for a dedicated row to format.

2. Reconcile the SLSH-4 prose so it no longer implies a contiguous set: keep the `SNK-a … SNK-k` span (its endpoints are correct) and add a clause noting `SNK-j` is retired, pointing at the new `## Retired inline labels` section, so the non-contiguity is explicit rather than apparently erroneous.

Do **not** renumber `SNK-k` (or any other `SNK` label) to close the gap: `SNK` labels are GOV-16-governed identifiers, renumbering breaks existing citations, and the retirement record is the prescribed way to account for a non-contiguous range. The `SNK-a … SNK-k` span descriptions in `governance/stable-inline-labels.md` and `governance.md` name only the endpoints and remain correct once the retirement is recorded.

## Consequence

**Severity:** correctness

The spec self-contradicts: SLSH-4's prose asserts an `SNK-a … SNK-k` span while the table omits `SNK-j`, and GOV-16's mandatory `## Retired inline labels` record is absent. Two maintainers reconciling the `SNK` label set against the page would diverge on whether `SNK-j` was retired, never existed, or is missing in error, and the GOV-15 release-time corpus review (the inline-label analogue of the per-prefix numbering-hole gate) would flag the unaccounted `j` position at the loom 1.0 release footing.
