## Problem

`diagnostic-shape.md` *Persistent diagnostics (default)* asserts a closed routing rule: every `loom/runtime/*` diagnostic is delivered via `pi.sendMessage` for `loom-system-note` with exactly five carved-out `console.error` exceptions, none of which is `loom/runtime/system-note-delivery-failed`. That code is the diagnostic the runtime emits *because* `pi.sendMessage` for a `loom-system-note` has just thrown, so the closed-list rule routes it back through the channel it observed throwing — a self-referential path forbidden by the re-entry guard in `runtime-event-channel.md`. Reading the two pages together yields three incompatible sinks (another `loom-system-note`, `console.error` per the "if both channels fail" prose, or a re-invoked `ctx.ui.notify`), so the diagnostic — the runtime's only observable signal of a system-note delivery failure — has no single consistent wire destination. The `code-registry-runtime.md` row pins the emission *site* (after `ctx.ui.notify`) but not the sink, so it does not break the tie.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `11b9199` (2026-05-04) — *pi-loom spec: resolve "pi.sendMessage failure leaves system notes silently dropped"* — introduced the `system-note-delivery-failed` code and the `ctx.ui.notify` → diagnostic → terminal `console.error` fallback chain in `pi-integration-contract.md`. At this point `diagnostic-shape.md` carried no carve-out enumeration, so the fallback's terminal `console.error` was the *only* exception and was not in tension with any closure claim.
- `5e8b499` (2026-05-06) — *pi-loom spec: resolve "Hot-reload teardown contract for the loom extension instance is unspecified"* — introduced the first carve-out clause ("with one carved-out exception") in `diagnostics.md` for `reload-teardown-timeout`, framing it as the sole `console.error`-routed exception to the persistent-diagnostic channel.
- `9559e19` / `6ce416d` / `75a9626` / `52b9101` (2026-05-07 — 2026-05-11) — successively widened the carve-out to "two" → "three" → "four" → "five carved-out exceptions" as further `loom/host/session-shutdown-*` codes were added; the enumeration was kept exhaustive each time.
- `b2f780b` (2026-06-24) — *pi-loom spec-plan: fix F-1152 — classify diagnostic emission sites* — split `diagnostics.md` into `diagnostic-shape.md` + per-namespace pages, carrying the "five carved-out exceptions" framing forward verbatim into `diagnostic-shape.md`.

**History:** The defect is the product of two independently-correct edits never reconciled: commit `11b9199` pinned `console.error` as the terminal sink of the system-note fallback chain (before any closure claim existed), and commits `5e8b499` → `52b9101` built up an exhaustive five-code `console.error` carve-out scoped entirely to the `session_shutdown` handler — without revisiting whether `system-note-delivery-failed`'s own terminal `console.error` was an implicit sixth carve-out. Neither edit on its own contradicts itself; the contradiction lives in their interaction at the `diagnostic-shape.md` ↔ `runtime-event-channel.md` boundary.

## Solution approach

Pin a single sink for `loom/runtime/system-note-delivery-failed` and reconcile all three pages on it. In `diagnostic-shape.md` *Persistent diagnostics (default)*, add this code to the carve-out enumeration so the closure claim no longer routes it through `loom-system-note`. In `runtime-event-channel.md` *System notes* fallback chain, rewrite step 2 to name the chosen single sink and resolve the now-vacuous "if both channels fail" branch. In the `code-registry-runtime.md` `loom/runtime/system-note-delivery-failed` row, pin the wire sink in the *Trigger* / *Spec rule* cells so the registry agrees with the other two pages.

## Solution constraints

- The diagnostic MUST NOT be routed through `loom-system-note`; the re-entry guard in `runtime-event-channel.md` forbids re-invoking `pi.sendMessage` from this fallback.
- A `console.error` sink for this code MUST be wrapped under the [Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation) swallow rule, so a throwing last-resort sink does not unwind the emitting site.
