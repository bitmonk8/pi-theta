## Finding

`V18b` owns a one-commit land-green sweep over the audited source tree: its **Adds** field requires that, in the same commit that introduces the closure audit, *every* existing `src/` Pi-side reference the new audit would surface is deleted, promoted to an `SDK_SURFACE_INVENTORY` entry, marked with an `// allow-pi-surface:` exemption, or rewritten into a recognised shape, so the audit lands green on `main`. This is a mechanical mass-rewrite of every Pi-touching reference in the tree, landing in a single commit.

The only stated gate on that sweep is the closure audit itself (**Ships when** — `npm test` runs the closure audit green on `main`). That audit is a *static shape/inventory* check: it asserts each reference resolves to an inventory entry, an allow-list, or an exemption, and that off-shape references fire the family-(4) discriminator. It does **not** verify that a given rewrite preserved the reference's runtime semantics. A reference rewritten into a *recognised-but-wrong* shape — promoted against the wrong inventory row, or wrongly exempted — passes the audit green while silently changing behaviour.

The leaf carries no statement that the sweep is behaviour-preserving, no expectation that a behaviour-breaking mis-rewrite is caught downstream of the static audit (by the full `npm test` runtime suite, which the same Ships-when gate already runs), and no stated recovery path for a bad sweep. An implementer following the leaf as written has no instruction distinguishing a shape-only rewrite from a semantics-altering one, and no recorded recovery should the mass-rewrite go wrong.

## Consequence

**Severity:** advisory

If the leaf ships as written, an implementer performing the mass-rewrite has no stated invariant that each rewrite must preserve runtime behaviour, and no recorded recovery for a bad sweep. A reference rewritten into a recognised-but-wrong shape passes the static audit green; whether it is then caught depends on incidental runtime test coverage of that reference rather than on a stated expectation. The closure audit alone cannot detect the regression, and the leaf leaves the detection and recovery story implicit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V18b-inventory-audit.md`, extend the `Adds` field's land-green-sweep sentence (or add an adjacent sentence) so the leaf records three things the sweep currently leaves implicit:

- The sweep is **behaviour-preserving**: each per-reference action (delete, promote to `SDK_SURFACE_INVENTORY`, `// allow-pi-surface:` exemption, or rewrite into a recognised shape) changes only the reference's surface classification, not its runtime semantics.
- A behaviour-breaking mis-rewrite (a reference rewritten into a recognised-but-wrong shape) is **not** caught by the static closure audit — it is caught by the full `npm test` runtime suite that the existing `Ships when` already invokes; the static audit verifies shape/inventory closure only.
- Recovery from a bad sweep is `git revert` of the single sweep commit (the sweep lands in one commit precisely so it reverts atomically).

Suggested literal addition to `Adds`, appended after the existing land-green-sweep sentence: "The sweep is behaviour-preserving — each rewrite changes only a reference's surface classification, not its runtime semantics; a behaviour-breaking mis-rewrite (a reference rewritten into a recognised-but-wrong shape) is caught by the full `npm test` runtime suite the Ships-when gate already runs, not by the static closure audit, and is recovered by `git revert` of the single sweep commit."
