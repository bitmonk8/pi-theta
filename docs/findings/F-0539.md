## Finding

`V9n` (prompt-mode transport-error mapping) names two different sources for the `provider` field of the synthesised `TransportError` within the same leaf. The **Adds** field states the field "is sourced from `V9j`'s provider-error-mapping surface, not re-derived here." The **Tests** field (in both `V9n` and `V9n-T`) states the value is "sourced from the loom's resolved `model:` `Model<Api>.api` value (the frontmatter `model:`, or the inherited `ctx.model` when `model:` is absent; resolution owned by V6a), per conversation-drive.md's resolved-`model:` rule and queryerror-variants.md's `TransportError.provider` derivation." An implementer reading Adds wires the field off V9j's classifier; an implementer reading Tests wires it off the resolved-model api value. The two chains are not the same surface.

The spec is unambiguous on which chain is correct. `errors-and-results/queryerror-variants.md` (`provider` derivation) states the field "is the resolved `Model<Api>.api` value of the model the failing query routed through — the same `api`-shaped field the Provider error mapping table is keyed on." So the provider *value* comes from the resolved model (resolution owned by `V6a`); `provider-error-mapping.md` (the surface `V9j` implements) is *keyed on* that same `api` value to classify the error and populate `retryable` — it does not own or supply the `provider` value. The Tests clause matches the spec; the Adds clause mis-attributes the value's source to V9j.

The same divergent claim is propagated into `coverage-matrix.md`'s `cka-24` row ("the synthesised error's `provider` field is sourced from V9j's provider-error-mapping surface, not re-derived here"), so the contradiction exists in two plan locations, not one.

## Issue introduction

**History:** The prompt-mode transport-failure detection entered `V9c` in `10cbf96` (2026-06-16) with **both** Adds and Tests citing "V9j's provider-error-mapping surface" — internally consistent, no divergence. `c04b0a6` then rewrote only the Tests clause (in `V9c` and `V9c-T`, two lines) to the spec-faithful resolved-`model:` `Model<Api>.api` / V6a chain, but left the Adds clause and the `coverage-matrix.md` `cka-24` cell still naming V9j's surface — a partial fix that introduced the contradiction it was nominally resolving. `0c3a2a2` (2026-06-17) subsequently split `V9c` into `V9n`, carrying the already-divergent Adds and the corrected Tests into the new leaf and re-targeting the `cka-24` closing cell to `V9n`.

## Solution Space

**Shape:** single

### Recommendation

Complete the correction `c04b0a6` started: bring the `provider`-source claim in `V9n`'s Adds (and the `cka-24` cell) into line with the Tests and the spec, which derive the value from the resolved model rather than V9j.

- In `docs/plan_topics/V9n-transport-error-mapping.md`, **Adds** field: strike "the synthesised `transport` error's `provider` field is sourced from `V9j`'s provider-error-mapping surface, not re-derived here." Replace it with a statement that the `provider` field is the resolved `model:` `Model<Api>.api` value (resolution owned by `V6a`; the inherited `ctx.model` when `model:` is absent), per `queryerror-variants.md`'s `TransportError.provider` derivation — matching the Tests bullet. If the Adds means to record that V9j's provider-error-mapping surface populates the *classification* (`retryable`/`kind`), scope that statement to those fields, not to `provider`.
- In `docs/plan_topics/coverage-matrix.md`, `cka-24` row: strike "the synthesised error's `provider` field is sourced from V9j's provider-error-mapping surface, not re-derived here" and replace with the same resolved-model derivation, preserving the existing `(un-anchored; GOV-22 residue)` annotation.

The resolved-model chain is the value the spec mandates and the value `V9n`'s Tests already assert; V9j's surface remains the classifier keyed on that same `api` value, not the source of it.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on where the `provider` value originates: the Adds points at V9j's error classifier (which does not own the resolved-model api value), the Tests point at the resolved model owned by V6a. The Tests' observable-value assertion would catch an outright-wrong value, but the contradictory Adds — mirrored in the un-anchored `cka-24` matrix cell — misdirects the production wiring and leaves the leaf's authoritative source for the field unresolved.
