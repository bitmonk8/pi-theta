## Problem

`V9n` (prompt-mode transport-error mapping) names two different sources for the `provider` field of the synthesised `TransportError` within the same leaf. The **Adds** field states the field is sourced from `V9j`'s provider-error-mapping surface; the **Tests** field states it is sourced from the resolved `model:` `Model<Api>.api` value (resolution owned by `V6a`). The spec (`errors-and-results/queryerror-variants.md`, `TransportError.provider` derivation) is unambiguous: the value comes from the resolved model, and `V9j`'s surface is merely *keyed on* that same `api` value to classify the error — it does not supply the `provider` value. The same divergent claim is mirrored in `coverage-matrix.md`'s `cka-24` row, so the contradiction exists in two plan locations.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commit:** `c04b0a6` — "pi-loom plan: resolve 'prompt-mode TransportError.provider cites V9j's classifier'" (2026-06-17)
**History:** The prompt-mode transport-failure detection entered `V9c` in `10cbf96` (2026-06-16) with **both** Adds and Tests citing "V9j's provider-error-mapping surface" — internally consistent, no divergence. `c04b0a6` then rewrote only the Tests clause (in `V9c` and `V9c-T`, two lines) to the spec-faithful resolved-`model:` `Model<Api>.api` / V6a chain, but left the Adds clause and the `coverage-matrix.md` `cka-24` cell still naming V9j's surface — a partial fix that introduced the contradiction it was nominally resolving. `0c3a2a2` (2026-06-17) subsequently split `V9c` into `V9n`, carrying the already-divergent Adds and the corrected Tests into the new leaf and re-targeting the `cka-24` closing cell to `V9n`.

## Solution approach

Rewrite the `provider`-source claim in `V9n`'s **Adds** field to match the Tests bullet and the spec: derive the value from the resolved `model:` `Model<Api>.api` value (resolution owned by `V6a`), per `queryerror-variants.md`'s `TransportError.provider` derivation, rather than from `V9j`'s surface. If the Adds intends to record that `V9j`'s provider-error-mapping surface populates the *classification* (`retryable`/`kind`), scope that claim to those fields, not to `provider`. Apply the same correction to `coverage-matrix.md`'s `cka-24` row, preserving its existing `(un-anchored; GOV-22 residue)` annotation.
