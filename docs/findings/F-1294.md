## Problem

IMP-1 decides whether an import is unresolvable by a byte-for-byte match of the import literal's final segment against the bytes `FileSystem.readdir` returns for the resolved parent directory, explicitly not folding NFC against NFD spellings. PIC-13's `readdir` bullet pins neither the encoding of the returned entry names nor any no-normalisation guarantee — it states only "entry names only (no full paths); rejects with the same Node-style `.code` shape as `readText`". Two conforming `FileSystem` implementations — one passing names through verbatim, one NFC-normalising on the way out — would diverge on whether an NFD-spelled import matches an NFC on-disk entry, with no normative seam clause to test that boundary against.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:**
- `8a6d15f` (2026-05-06) — introduced PIC-13's `readdir` member with the deliberately narrow rejection-only contract; no encoding or normalisation pin.
- `9aa73b1` (2026-06-07) — rewrote IMP-1 to decide unresolvability against the byte-for-byte `readdir` return value with explicit NFC-vs-NFD non-folding, lifting the consumption site onto a contract surface the seam never pinned.

**History:** The defect emerges from the interaction between the seam-introduction commit (narrow rejection-only contract) and the consumption-site sharpening in `9aa73b1` (byte-exact match), where the seam member's encoding/normalisation properties were never pinned.

## Solution approach

Clarify PIC-13's `readdir` bullet (anchor `id="pic-13"`, `host-interfaces-services.md`) to pin that returned entry names are the raw on-disk filename bytes interpreted as UTF-8 and are not subjected to Unicode normalisation, so an entry whose on-disk bytes spell `café` in NFD is returned in NFD. Mirror the encoding posture already pinned on `pic-13-readbytes` and `lexical.md` *Encoding*, and state the clause binds every conforming seam implementation, parallel to the `realpath` case-canonicalisation clause.
