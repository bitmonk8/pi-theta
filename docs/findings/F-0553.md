## Finding

`defaulting-system-note-echo.md` bold-names the hook this section installs as **post-default-merge AJV validation** — the `SchemaValidator.validate()` call that AJV-validates the merged `args` against the lowered `params` schema *after* the runtime fills declared defaults. The plan-side leaves that own and reference this hook drop the word "default" and call it "post-merge AJV validation":

- `V11g-defaulting-revalidation.md` — H1 heading and **Ships when** (`fill-if-absent + post-merge AJV path`).
- `V11g-T-defaulting-revalidation.md` — H1 heading.
- `V11d-binder-system-prompt.md` — **Adds** (`fill-if-absent defaulting with post-merge AJV validation is owned by V11g`).
- `plan.md` — the V11 slice list entry linking `V11g`.

`coverage-matrix.md` (`cka-40`) already uses the full spec term `post-default-merge AJV validation` and its `#post-default-merge-ajv-validation` anchor, so the plan corpus is internally inconsistent: the matrix row and the spec agree, the leaf headings/prose disagree. The bare "post-merge" form is ambiguous about *which* merge is meant (default-fill merge vs settings/loomPaths merges that also use "post-merge" elsewhere in the corpus, e.g. `V9b`).

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (2026-06-10, initial plan build) and cb8737a (2026-06-12, over-bundling split)
**History:** The corpus is git-tracked. `git log -S"post-merge AJV"` over the plan corpus shows the "post-merge AJV validation" string first entered at c6a664e, in the then-bundled `V11d-defaulting-echo.md` leaf's **Adds** and **Ships when** — authored without "default" at inception, even though the same commit's review prose correctly used the spec's "post-default-merge AJV" term. The over-bundling split cb8737a (earliest commit for `V11g-defaulting-revalidation.md` and `V11g-T-defaulting-revalidation.md` per `git log --follow`) carried the truncated name forward into the new `V11g`/`V11g-T` leaves and the trimmed `V11d`. The divergence is therefore an original authoring drift propagated by a later refactor, not a defect a specific edit introduced into previously-correct prose.

## Solution Space

**Shape:** single

### Recommendation

Replace the string `post-merge AJV validation` with `post-default-merge AJV validation` at each plan-side occurrence so the leaf prose matches the spec bold-name and `cka-40`:

- `docs/plan_topics/V11g-defaulting-revalidation.md` — H1 heading: `# \`V11g\` — Fill-if-absent defaulting and post-default-merge AJV validation`; and **Ships when**: `the fill-if-absent + post-default-merge AJV path`.
- `docs/plan_topics/V11g-T-defaulting-revalidation.md` — H1 heading: `# \`V11g-T\` — Fill-if-absent defaulting and post-default-merge AJV validation (tests)`.
- `docs/plan_topics/V11d-binder-system-prompt.md` — **Adds**: `The fill-if-absent defaulting with post-default-merge AJV validation is owned by \`V11g\``.
- `docs/plan.md` — the V11 list entry: `[\`V11g\` — Fill-if-absent defaulting and post-default-merge AJV validation](./plan_topics/V11g-defaulting-revalidation.md)`.

`coverage-matrix.md` `cka-40` and the spec are already correct and are read-only for this fix; leave them unchanged.

## Consequence

**Severity:** cosmetic

A reader comparing the V11g/V11d prose against the spec and `cka-40` sees three different-looking names for one hook and must resolve whether they denote the same obligation. Behaviour does not diverge — each leaf's Tests bullet and the `#post-default-merge-ajv-validation` anchor spell out the full hook — so no implementer is blocked or misled into a wrong implementation; the cost is purely the consistency/searchability of the corpus.
