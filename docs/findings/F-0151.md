## Problem

`docs/spec_topics/pi-integration-contract/registration-steps.md` is named as the primary `Spec.` target of `V9b` and `V9b-T`, and carries five normative obligations (four `MUST`, one `MUST NOT`) that define no `PIC-*` REQ-ID and are therefore GOV-22 un-anchored residue requiring a code-keyed coverage-matrix row. The "Code-keyed obligation areas" table in `docs/plan_topics/coverage-matrix.md` cites `registration-steps.md` in no row. In particular the step-5 structural-change watcher-note pair (anchor `#structural-changes-no-unregister` — `MUST NOT` emit on an empty window; `MUST` emit on a same-window rename) is closed by no row and asserted by no plan leaf, so it can ship unimplemented with the H5a closing gate green and `V9b` tagged complete.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commit:** `c6a664e` — "pi-loom plan: build/update plan for spec.md + review" (2026-06-10)
**History:** The coverage matrix was created at `fecb504` (2026-05-04, the spec/plan per-topic split) and `git log -S "registration-steps" -- docs/plan_topics/coverage-matrix.md` returns no commits — the token has never entered the matrix. `V9b`/`V9b-T` were introduced at `c6a664e` (2026-06-10) already citing `registration-steps.md` as their primary `Spec.` target. Every subsequent coverage-matrix edit that added `V9b` coverage (`adb521f` drain-state row, later re-pointed to `V9m`; `249cec5` `implementation-notes.md` IMPL row) closed adjacent obligations without ever adding a `registration-steps.md`-keyed row. The gap has therefore existed since the V9b leaves were authored and was never subsequently closed.

## Solution approach

Audit the `MUST` / `MUST NOT` obligations of `registration-steps.md` and add a code-keyed row to the "Code-keyed obligation areas" table of `coverage-matrix.md` citing `pi-integration-contract/registration-steps.md` (tagged `un-anchored; GOV-22 residue`) with `V9b` as closing leaf, covering the obligations not already closed under another row — at minimum the registry-swap atomicity / `loom/runtime/registry-swap-failed` obligation and the step-5 watcher-note pair at `#structural-changes-no-unregister`. Add a `V9b-T` Tests bullet and the paired `V9b` behaviour asserting the empty-window suppression and the same-window-rename emission of the structural-change note, so the new row's closing-leaf claim is backed by an asserting leaf. If the audit finds every obligation already closed elsewhere, add a one-line note to that effect in the matrix rather than leaving the file silently absent.

## Solution constraints

- Out of scope: the drain-state-gated-dispatch `MUST` (closed at `V9m` via the `drain-state-contract.md` row) and the build-time SDK surface-inventory `MUST` for `SlashCommandInfo` / `SlashCommandSource` (closed under the `V18a` / `V18b` capability-inventory rows) — do not add `registration-steps.md` rows duplicating these.
- The spec is read-only for this fix; close the obligations via matrix rows and plan leaves, not by adding `PIC-*` anchors to `registration-steps.md`.
