## Finding

The spec mandates a *bidirectional* cancellation linkage in prompt mode. Beyond forwarding Pi's `ctx.signal` *into* `loomAbort`, the reverse direction is normative: when `loomAbort.abort()` fires while a prompt-mode user turn is in flight, the runtime MUST invoke the unwrapped, Pi-supplied `abort()` on the raw `ExtensionCommandContext` Pi passed to the slash-command handler — **not** the synthesised tool-execution `ExtensionContext` whose overridden `ctx.abort()` wrapper itself calls `loomAbort.abort()` — to tear down the user run and unblock `await ctx.waitForIdle()`. This is pinned in `cancellation.md` §*Forwarding into `loomAbort`* ("Forwarding is also bidirectional in prompt mode…") and in `conversation-drive.md` §*Hang handling* ("`loomAbort.abort()` invokes the unwrapped, Pi-supplied `abort()` … which cancels the active user run and lets `waitForIdle()` resolve"). The propagation is guarded by a one-shot flag against re-entrant double-cancel.

No plan leaf asserts this teardown direction. `V17a` (cancellation core) asserts forwarding *into* `loomAbort` across the three entry points plus `CNCL-1..6`, but explicitly excludes the bidirectional prompt-mode propagation from its scope (`CNCL-4` notes it "goes the other direction across the linkage and is unaffected by this rule"). `V9c` — the prompt-mode conversation-drive leaf that the spec's *Hang handling* clause lives under (`conversation-drive.md` is `V9c`'s primary `Spec.`) — covers `sendUserMessage`, `waitForIdle`, trailing-turn extraction, the `stopReason:"error"` probe, the `pi.on` subscription (described as "cancel-forward only, forwarding Pi's `ctx.signal` *into* the `loomAbort` controller"), and the active-set gating window. Its Adds never mentions the reverse `loomAbort.abort()` → `ctx.abort()` teardown, and its Tests (`PIC-2`, `PIC-17`, `PIC-18`, `PIC-53`) do not assert it. `V9n` asserts the *post-resolution* cancellation short-circuit (when `loomAbort.signal.aborted` is true after `waitForIdle()` resolves, synthesise `kind:"cancelled"`) but presupposes the wait already resolved; it does not exercise the teardown act that *causes* that resolution during an in-flight turn.

The teardown is therefore a normative behaviour with no closing test in the plan. An implementer building `V9c` can ship a green leaf whose prompt-mode driver never wires (or mis-wires) the reverse teardown, and the gap surfaces only as a hung prompt-mode cancel.

## Consequence

**Severity:** correctness

If shipped unfixed, `V9c` passes its gate without ever exercising the `loomAbort.abort()` → unwrapped `ctx.abort()` linkage, so a faithful implementer may omit it or wire the abort to the synthesised tool-execution `ctx.abort()` wrapper (whose body re-calls `loomAbort.abort()`, looping instead of tearing down the user run). Either way a prompt-mode cancel — Esc during an in-flight user turn — leaves `await ctx.waitForIdle()` blocked indefinitely and the loom hangs, while the leaf reports green. A spec MUST silently ships unverified.

## Solution Space

**Shape:** single

### Recommendation

Close the teardown at its owning leaf, `V9c` (`docs/plan_topics/V9c-conversation-drive.md`), whose primary `Spec.` already carries the *Hang handling* clause.

- Extend `V9c`'s **Adds** to include the reverse teardown alongside the existing `pi.on` cancel-forward: when `loomAbort.abort()` fires during an in-flight prompt-mode user turn, the runtime invokes the unwrapped, Pi-supplied `abort()` on the raw `ExtensionCommandContext` Pi passes to the slash-command handler (captured for the loom's lifetime) — not the synthesised tool-execution `ExtensionContext` whose overridden `ctx.abort()` wrapper itself calls `loomAbort.abort()` — to tear down the user run and unblock `await ctx.waitForIdle()`, guarded by the one-shot flag against re-entrant double-cancel.
- Add a citing **Tests** assertion to `V9c` and `V9c-T` (`docs/plan_topics/V9c-T-conversation-drive.md`): with a prompt-mode user turn in flight, fire `loomAbort.abort()` and assert (i) the unwrapped raw-`ctx` `abort()` is invoked (and the synthesised tool-execution `ctx.abort()` wrapper is *not* the call site that drives teardown) and (ii) `await ctx.waitForIdle()` subsequently resolves. Cite `conversation-drive.md` §*Hang handling* (and `cancellation.md` §*Forwarding into `loomAbort`*) as the spec source.

Edge cases the implementer must watch: the test's determinism rests on the `H4a` in-process session-double / fidelity contract modelling `ctx.abort()` → `waitForIdle()` resolution, so the deterministic teardown observation depends on that harness behaviour (confirm `V9c`'s `H4a`/`V8a` reach suffices, or note the dependency); and the one-shot guard must make a re-entrant `loomAbort.abort()` (e.g. an `agent_end` observer that itself observes the abort) a no-op rather than a second teardown.
