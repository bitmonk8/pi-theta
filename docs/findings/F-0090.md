## Problem

V13a's second Tests bullet (mirrored verbatim in V13a-T) reads *"Each loom type stringifies per the table; `interpolated-result` and `discarded-query-result` (with its runtime event) fire."* The parenthetical "(with its runtime event)" attaches a runtime event to the parse-time `discarded-query-result` diagnostic, but these are two mutually-exclusive mechanisms: `loom/parse/discarded-query-result` is a parse-phase error on a **bare** `@`...`` expression-statement (the loom does not load, so no runtime event can accompany it), while the discard-observability event is emitted at runtime for an **explicit** discard (`let _ = @`...`` or the `void`-tail form) whose `Result` is `Err`. An implementer cannot author a faithful test from the bundled bullet: one reads the parenthetical as decoration and tests only the parse code; another tries to assert a runtime event co-fires with the parse error, which is impossible. The distinct runtime discard-observability event may ship untested under either reading.

## Solution approach

Rewrite the second Tests bullet in both `docs/plan_topics/V13a-query-render.md` and `docs/plan_topics/V13a-T-query-render.md` so it asserts two distinct observables and drops the "(with its runtime event)" parenthetical: the parse-time `discarded-query-result` diagnostic firing on a bare `@`...`` expression-statement, and — separately — the runtime discard-observability event emitted on an explicit `Err` discard. Ground the rewrite in `query-escapes-stringification.md` §"Discarded query results are a parse error" and §"Observability of discarded results".

## Solution constraints

- The spec is read-only for this fix; both behaviours are already correctly defined there.
- The runtime-event contract is owned by V9d (already in V13a's Deps); do not restate its shape here.
