## Problem

The barrel-module operational definition in `audit-recognised-shapes.md` defines clause (iv) membership as "an import statement whose binding is used by an (i) statement on the same module" — i.e. "used" means consumed as a re-export source. Three clauses later the exclusion disqualifies any module with "a top-level `import` whose binding is unused", without restating which sense of "unused" applies. Two readings result: strict negation of (iv) (not consumed by any `export … from` source) versus dead-import (referenced nowhere at all). The two readings classify an `import type` whose binding is referenced outside the re-export block differently, so two conformant audit implementations can return divergent barrel verdicts on identical input.

**Verdict:** present-since-inception
**Introducing commits:** `e5cd514` (2026-05-09, *pi-loom spec: resolve "T35 SDK capability inventory closed-set negative-direction audit"*)
**History:** `git log --follow -G 'barrel module'` over `docs/spec_topics/pi-integration-contract/audit-recognised-shapes.md` and its pre-split ancestor `docs/spec_topics/pi-integration-contract.md` returns `e5cd514` as the first commit to introduce the five-clause barrel-module operational definition; `git log -G 'whose binding is unused'` against the same files identifies the same commit as the one that introduced the conflicting "unused" exclusion. The defect entered in the same edit that created the surrounding paragraph: clauses (iv) and the "unused" exclusion were authored together, with the inconsistent vocabulary present from the start. The file later moved through `f5e89f4` (the 100KB-cap corpus split) which preserved the wording verbatim, so the ambiguity has never been touched since inception.

## Solution approach

Rewrite the "unused" exclusion clause in the barrel-module operational definition to restate clause (iv)'s predicate rather than introduce a new undefined word — disqualify a module whose top-level `import` binding is not consumed as a re-export source by any `export … from "@earendil-works/<peer>"` statement in the same module. This pins the strict-negation-of-(iv) reading, which is the only one consistent with the surrounding paragraph's clauses (iv) and (v) and its resolution rules.
