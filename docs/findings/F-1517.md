## Finding

The barrel-module operational definition in `audit-recognised-shapes.md` defines membership by enumerating five allowed top-level statement shapes, including (iv) "an import statement whose binding is *used by an (i) statement* on the same module" — i.e. "used" is defined locally as "consumed as a re-export source". Three sentences later the exclusion clause disqualifies any module that contains "a top-level `import` whose binding is **unused**", without restating which sense of "unused" is meant.

Two readings are possible and the file picks neither:
- **(A)** "unused" = the strict negation of (iv): not consumed by any `export … from "@earendil-works/<peer>"` statement in the same module. Under (A), an `import type { Foo } from "@earendil-works/<peer>"` whose `Foo` only appears in a type-annotation, JSDoc reference, or elsewhere outside an (i) statement — but nowhere as an `export … from` source — disqualifies the module from being a barrel.
- **(B)** "unused" = dead import, referenced nowhere at all in the module. Under (B), the same type-annotation-only import keeps the module classified as a barrel, because the binding *is* referenced — just not by an (i) statement.

The two readings classify the same module differently, which flips whether `export * from` / `export * as` (universally prohibited regardless of barrel status) is the only failure, or whether per-symbol named re-exports — which the same paragraph permits *only* outside barrel modules — also fail. Two conformant audit implementations can therefore produce divergent verdicts on identical input, and a contributor reading the definition cannot tell which shape is safe.

The conjugate inclusion clause (iv) already carries the operational answer the audit needs ("used by an (i) statement"); the exclusion clause should align with that wording rather than introduce a new, undefined "unused" predicate.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the exclusion clause to restate (iv)'s operational predicate rather than introduce a new word. Replace

> a top-level `import` whose binding is unused

with

> a top-level `import` statement whose binding is not referenced by any `export … from "@earendil-works/<peer>"` statement in the same module (i.e. not consumed as a re-export source per clause (iv) above)

This pins reading (A), which is the only reading consistent with the surrounding paragraph: clause (iv) already defines "used" exclusively in re-export-source terms, the typebox-equivalent clause (v) covers `export type … from`, and the audit's resolution rules (the `SDK_SURFACE_INVENTORY` join key, the typebox named-import allow-list, the canonical-carrier discipline) all operate by tracing import bindings through re-export statements rather than through arbitrary in-module references. Reading (B) would silently authorise modules whose type-only imports launder peer-package symbols through type annotations the audit's category-(2) recogniser does not walk, which the surrounding paragraph spends several clauses prohibiting elsewhere.

Implementer edge cases the rewrite must keep visible:
- An `import type { Foo } from "@earendil-works/<peer>"` whose `Foo` is re-exported only via the clause-(v) `export type … from` form (rather than as the *source* of an (i) statement) is covered: clause (v) is itself an allowed top-level shape, and the import-statement exclusion fires only when the import's binding is not consumed by any (i) `export … from` statement, leaving the type-only re-export path uncontested.
- A side-effect-only `import "@earendil-works/<peer>"` is already prohibited under the family-(4) side-effect-only-imports clause later in the same paragraph, so it never reaches the barrel-module classifier.

## Consequence

**Severity:** correctness

Two reasonable implementers of the family-(4) barrel-module recogniser will disagree on whether a module containing an `import type` whose binding is referenced outside the re-export block is a barrel. Under one reading, the same module passes the audit; under the other, its per-symbol named re-exports fire a family-(4) record. Contributors writing barrel modules cannot predict which shape is safe without picking a reading the spec has not endorsed.
