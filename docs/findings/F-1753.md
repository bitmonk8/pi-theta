## Finding

The *Stale-citation limitation (acknowledged soft surface)* sub-bullet in `docs/spec_topics/pi-integration-contract/audit-failures.md` is a single long sentence-paragraph that fuses three distinct kinds of content: (1) a descriptive enumeration of *what* the soft surface is (a syntactically well-formed citation whose REQ-ID / PIC anchor no longer resolves), (2) the *contributor-side mitigation* in step 2(b) branch (4), and (3) a trailing clause that sketches the *implementation* of the deferred capability — "widening the audit to validate citation resolution would require attaching a registry of valid REQ-IDs and PIC anchors to the audit input, which is a PIC edit deferred past loom 1.0."

The trailing clause is rationale-only design speculation about a non-loom-1.0 capability. It names a concrete input artefact ("a registry of valid REQ-IDs and PIC anchors"), a join key ("attaching … to the audit input"), and a deferral label — phrased close enough to a specification that a careful reader could mistake it for a forward-binding obligation, while the actual normative content (citation resolution is purely syntactic; no resolution sub-check is performed) is already stated earlier in the same sub-bullet and pinned authoritatively at the *Citation-token grammar* paragraph in `audit-resolution.md`. The deferral itself is real and useful to record; the implementation sketch is not.

The same shape recurs in the two sibling soft-surface bullets on this page (*Substitution limitation*, *Generated-output-in-src limitation*), each carrying its own rationale tail that bloats the normative file with content that belongs in a design note or in `future-considerations/`.

## Solution Space

**Shape:** single

### Recommendation

Replace the *Stale-citation limitation (acknowledged soft surface)* sub-bullet's trailing implementation clause with a bare deferral label, leaving the descriptive what-it-is sentence and the step 2(b) branch (4) mitigation sentence intact. Concretely, in `docs/spec_topics/pi-integration-contract/audit-failures.md`:

- Keep the opening sentence ("A citation token that parses as well-formed per the *Citation-token grammar* above but whose REQ-ID does not (or no longer) name a stable spec ID, …, is *not* mechanically detectable by the audit: the *Citation-token grammar* check above is purely syntactic and the audit performs no citation-resolution sub-check.") unchanged.
- Keep the mitigation sentence ("This soft surface is mitigated by the same contributor obligation in [step 2(b) branch (4)](./version-bump-step2b.md#bump-step-2b-stale-rewrite) below that mitigates the *Substitution limitation* above — every marker on a line touched by a refactor or bump commit MUST be re-read against the *Exemption mechanism* citation rule on the diff before the commit merges, and a citation that has gone stale MUST be rewritten under that branch's *citation has gone stale* case rather than carried over — and is **not** owed an additional audit discriminator in loom 1.0.") unchanged.
- Delete the trailing clause "; widening the audit to validate citation resolution would require attaching a registry of valid REQ-IDs and PIC anchors to the audit input, which is a PIC edit deferred past loom 1.0." Replace with a terminating full stop on the preceding sentence.
- Append a short replacement: "Citation-resolution against a REQ-ID / PIC-anchor registry is out of scope for loom 1.0." This preserves the deferral label that the *Citation-token grammar* paragraph in `audit-resolution.md` already cross-references via "the acknowledged soft surface this leaves … is mitigated per the *Stale-citation limitation* sub-bullet" without retaining the implementation sketch.

Do not move the deleted clause into `future-considerations/`: the spec has not committed to building the capability and the registry-as-audit-input shape is one design choice among several. If anyone later wants the design sketch preserved, it can be re-authored as a fresh note then; preserving the sentence verbatim now would re-introduce the same forward-binding hazard.

Edge cases for the implementer: the *Citation-token grammar* paragraph in `audit-resolution.md` contains its own parallel "widening the audit to validate citation resolution against a registry is a PIC edit deferred past loom 1.0" tail (visible in the same `git show e5cd514` diff). That cross-reference tail is short and confined to a single trailing clause, but the same scope-leakage concern applies — the fixer should apply the same trim there in the same edit (replace with the same "out of scope for loom 1.0" wording) so the two pages stay consistent and the cross-reference in `audit-failures.md` still resolves to a sub-bullet labelled the same way.

## Consequence

**Severity:** cosmetic

A future implementer reading the sub-bullet straight through could plausibly read "attaching a registry of valid REQ-IDs and PIC anchors to the audit input" as an in-scope design constraint rather than a deferred-capability sketch, because the deferral label arrives only at the very end of the sentence. The damage is bounded — the *Citation-token grammar* paragraph and the five-family closure together make the syntactic-only contract unambiguous — but the bullet adds reading load and forward-binds an implementation choice (registry-as-audit-input) for a capability the spec has not committed to building.
