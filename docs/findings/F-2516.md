## Finding

`TYPE-9` (type-system.md, `#type-9`) names the per-site parse diagnostic for a plain function-argument slot: "an argument to a **top-level `fn` call** that is neither an `invoke(...)` nor a `.loom`-callable call — reports `loom/parse/fn-arg-type-mismatch`." The qualifier "top-level" is misleading. Per `FN-1` (functions.md) every `fn` declaration is top-level — nested `fn` is forbidden (`loom/parse/nested-fn`) — so "a top-level `fn` call" reduces to a tautology (every `fn` call qualifies) and adds no constraint. A second, divergent reading is also available: "top-level" could be taken positionally, as "a `fn` call appearing at the top level of the loom body," which would wrongly exclude `fn` calls nested inside other expressions or blocks from `fn-arg-type-mismatch`.

The intended referent is a **user-defined `fn` call** — the case distinguished from `invoke(...)` and `.loom`-callable calls, which carry their own diagnostics. The trailing parenthetical ("neither an `invoke(...)` nor a `.loom`-callable call") already captures that distinction, so "top-level" is redundant at best and invites the positional misreading at worst. The rule applies to user-defined `fn` calls regardless of where the call appears (loom-body top level or nested) and regardless of intra-file vs cross-file `.warp` residence.

## Solution Space

**Shape:** single

### Recommendation

In `TYPE-9` (type-system.md), replace "an argument to a top-level `fn` call that is neither an `invoke(...)` nor a `.loom`-callable call" with "an argument to a user-defined `fn` call (i.e. one that is neither an `invoke(...)` nor a `.loom`-callable call)". This drops the redundant/misleading "top-level" qualifier and names the actual referent.

Edge cases for the implementer: the rule fires for both intra-file `fn` calls and cross-file `.warp` `fn` calls, and for `fn` calls wherever they appear (loom-body top level or nested inside an expression/block). The replacement wording must not reintroduce any positional reading that would scope the diagnostic to a single syntactic level.

## Consequence

**Severity:** advisory

An implementer who takes "top-level `fn` call" positionally could scope `loom/parse/fn-arg-type-mismatch` to only `fn` calls at the loom-body top level, leaving type-mismatched arguments to `fn` calls nested inside other expressions unchecked at parse time. Intent is recoverable from the parenthetical and the diagnostic name, so a working implementation is still reachable, but the wording is a needless ambiguity in a normative type rule.
