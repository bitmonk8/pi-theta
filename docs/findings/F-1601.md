## Finding

The `**Loom return type.**` paragraph at `docs/spec_topics/functions.md#loom-return-type` (FN-3) says:

> the `loom/parse/unreachable-code` warning pinned by [Return Statement](./return.md) is an editorial warning, not an inference input, so a `return expr` in code the parser warns is unreachable still contributes to the LUB and can still trigger `loom/parse/return-no-common-type`.

The qualifier *editorial warning* is not defined anywhere in the corpus. The diagnostic registry (`docs/spec_topics/diagnostics/code-registry-parse.md`) classifies `loom/parse/unreachable-code` only as severity `W` (warning) under the `parse` category, and the cross-referenced RET-3 in `docs/spec_topics/return.md` likewise says only "warning, not error" — neither anchor introduces an *editorial* sub-class of warning, and no other site in the spec uses the term. The implementer is left to infer the intended meaning ("does not feed type inference; emit-and-continue") from the *because*-clause that follows, which spells out the inference-input consequence but says nothing about whether the warning has any other normative effect (e.g. whether it must still be emitted, whether it counts toward any diagnostic budget, whether it suppresses downstream type errors on the dead code).

The bug is purely terminological. The accompanying clause already pins the only normative consequence FN-3 needs (the unreachable `return` still contributes to the LUB), so the load-bearing qualifier "editorial warning" is doing no work that the surrounding text does not already do — it is residue from the F-1171 rewrite where the LUB-over-all-syntactically-present-`return`s rule was introduced and a one-word hedge was attached to `unreachable-code` without being defined.

## Issue introduction

**Verdict:** introduced-by-fix
**Introducing commits:** `71bb4b7` (`pi-loom spec-plan: fix F-1171 — remove undefined 'reachable' qualifier from Loom return-type inference`, 2026-06-25)
**History:** `git log -S "editorial warning" -- docs/spec_topics/functions.md` shows a single touching commit: `71bb4b7`, which landed the F-1171 fix. Before that commit, FN-3 used the qualifier "every **reachable** early `return expr`" without defining *reachable* — the original F-1171 defect. The fix in `71bb4b7` replaced "reachable" with "syntactically present in the body, regardless of static reachability" and, to head off the obvious follow-up question (*then what is `loom/parse/unreachable-code` for?*), appended the sentence calling the warning "an editorial warning, not an inference input." The phrase "editorial warning" was coined in that same edit without a definition site and without a companion edit in `return.md` or the diagnostics registry to ground it; F-1171's recorded solution-approach prose used the same coinage verbatim, so it is a verbatim carry-through from the fix proposal. `git log --follow docs/spec_topics/functions.md` shows no subsequent commit has touched the paragraph. This is the same shape as the original F-1171 dropped stitch (an undefined qualifier added in a fix commit) at the adjacent token; the fix corrected one undefined qualifier and introduced another.

## Solution Space

**Shape:** single

### Recommendation

In the FN-3 paragraph, replace the clause

> the `loom/parse/unreachable-code` warning pinned by [Return Statement](./return.md) is an editorial warning, not an inference input, so a `return expr` in code the parser warns is unreachable still contributes to the LUB and can still trigger `loom/parse/return-no-common-type`.

with a statement that names the consequence directly and drops the undefined sub-class:

> the `loom/parse/unreachable-code` warning pinned by [Return Statement](./return.md) does not feed return-type inference, so a `return expr` in code the parser warns is unreachable still contributes to the LUB and can still trigger `loom/parse/return-no-common-type`.

This keeps the cross-reference to RET-3, keeps the explicit "still contributes to the LUB" guarantee that closed F-1171, and removes the undefined "editorial warning" coinage. Use the predicate *does not feed return-type inference* rather than the broader *is purely informative* the original suggested fix used: the broader formulation would implicitly resolve other open questions about the warning's role (e.g. its interaction with diagnostic budgets) that this paragraph is not the right home for. RET-3 and the diagnostics-registry row continue to be the sole authorities on the warning's severity class and emission obligation; no edit to either is needed.

## Consequence

**Severity:** cosmetic

Two implementers will reach the same observable behaviour because the *because*-clause spells out the only consequence (unreachable `return` operands still feed the LUB and can still trigger `loom/parse/return-no-common-type`). The defect is a load-bearing-sounding qualifier that carries no normative content, so it adds reader friction at a paragraph readers must already parse carefully, but does not cause divergence.
