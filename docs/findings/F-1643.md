## Finding

`docs/spec_topics/binder/binder-bypass-and-envelope.md` § *System-prompt structure (normative)*, item 8, reads:

> **No-invent-defaults instruction.** The prompt MUST contain a single line that includes both the literal substring `defaulted` (case-sensitive) and at least one of the directive substrings `Do not`, `omit`, or `skip` (case-sensitive). The rest of the wording is non-normative.

"A single line that includes …" has two defensible readings:

1. **Existence (≥1):** there exists at least one line in the prompt that satisfies the predicate. A prompt with two such lines still conforms.
2. **Uniqueness (exactly 1):** there is one and only one such line. A prompt with two such lines is non-conforming.

The surrounding items in the same numbered list use precise quantifier wording that makes the gap conspicuous: item 1 ends "Exactly one such line per prompt." and item 3 says "MUST appear exactly once" for the uniqueness reading, while items 2, 5, 6, 7 say only "MUST appear" / "MUST contain" for the existence reading. Item 8's "a single line" sits between the two established phrasings and matches neither verbatim, so a conformance-test author cannot derive the intended predicate from the page's own conventions. Under the existence reading the test is `∃ line. P(line)`; under the uniqueness reading it is `|{ line : P(line) }| == 1`. The two predicates disagree on any prompt that contains the no-invent guidance more than once.

The fenced example prompt earlier in the section (`Do not invent values for defaulted parameters that the user did not specify; omit them.`) emits exactly one such line, so the example is conforming under either reading and gives the wording no disambiguating signal.

Wording is non-normative; the instruction's presence is." `93439f0` rewrote the obligation to its present form ("MUST contain a single line that includes both the literal substring `defaulted` … and at least one of the directive substrings `Do not`, `omit`, or `skip`") to give conformance tests a deterministic predicate. The rewrite landed the "a single line" quantifier verbatim and has been carried unchanged through the subsequent topic split (`fecb504` → `31ff060`) and edits to surrounding items; the resolved-finding write-up in `93439f0` did not address whether the quantifier was existence or uniqueness, so the ambiguity is an artefact of that single resolution commit rather than an interaction across history.

## Solution Space

**Shape:** single

### Recommendation

Replace "a single line" with "at least one line", giving item 8:

> **No-invent-defaults instruction.** The prompt MUST contain at least one line that includes both the literal substring `defaulted` (case-sensitive) and at least one of the directive substrings `Do not`, `omit`, or `skip` (case-sensitive). The rest of the wording is non-normative.

The existence reading is the appropriate one: the underlying obligation is that the no-invent guidance be *present* in the prompt, not that it appear exactly once. Items 2, 5, 6, and 7 already use the existence pattern ("MUST appear" / "MUST contain") for analogous presence-only obligations; aligning item 8 with that pattern removes the ambiguity without changing intent. Items 1 and 3 reserve the explicit uniqueness phrasing ("Exactly one such line per prompt" / "MUST appear exactly once") for cases where multiple occurrences would actually be wrong (e.g. two `Loom:` lines would lie about which loom is being bound); no analogous failure mode exists for a duplicated no-invent instruction.

Edge cases for the implementer:

- The conformance predicate is `∃ rendered-line. line.includes("defaulted") && (line.includes("Do not") || line.includes("omit") || line.includes("skip"))`. The same-line co-occurrence requirement is preserved — `defaulted` and the directive token must appear on the *same* line, not anywhere in the prompt.
- `V11d`-T's Tests bullet asserting on item 8 stays a single positive assertion; no negative-half assertion is required (item 8 is unconditional).
- The reference fenced prompt continues to satisfy the rule unchanged, so no example update is needed.

## Consequence

**Severity:** advisory

Two implementers reading the obligation produce predicates that disagree only on prompts containing the no-invent instruction more than once — a corner case no implementer is likely to hit deliberately, since the natural builder emits the line once. The spec still produces a working system either way; the cost is a conformance-test gap (the predicate is under-specified) and the cross-implementer divergence on the duplicate-line edge case.
