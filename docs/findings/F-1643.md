## Problem

In `docs/spec_topics/binder/binder-bypass-and-envelope.md` § *System-prompt structure (normative)*, item 8 requires the prompt to contain "a single line" carrying the `defaulted` substring and a directive substring. "A single line" is ambiguous between an existence reading (≥1 such line) and a uniqueness reading (exactly one such line); the two predicates disagree on any prompt containing the no-invent guidance more than once. The surrounding items make the gap conspicuous — items 1 and 3 use explicit uniqueness phrasing ("Exactly one such line per prompt" / "MUST appear exactly once") while items 2, 5, 6, 7 use existence phrasing ("MUST appear" / "MUST contain") — and item 8 matches neither, so a conformance-test author cannot derive the intended predicate from the page's own conventions.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `93439f0` ("pi-loom spec: resolve \"T05 — Item 8 of binder system-prompt structure has no testable surface\"", 2026-05-07)
**History:** Item 8 originated in `5a4ec51` (2026-05-06) with non-testable wording: "The prompt MUST contain an instruction directing the model not to invent values for defaulted parameters the user did not specify. Wording is non-normative; the instruction's presence is." `93439f0` rewrote the obligation to its present form ("MUST contain a single line that includes both the literal substring `defaulted` … and at least one of the directive substrings `Do not`, `omit`, or `skip`") to give conformance tests a deterministic predicate. The rewrite landed the "a single line" quantifier verbatim and has been carried unchanged through the subsequent topic split (`fecb504` → `31ff060`) and edits to surrounding items; the resolved-finding write-up in `93439f0` did not address whether the quantifier was existence or uniqueness, so the ambiguity is an artefact of that single resolution commit rather than an interaction across history.

## Solution approach

Clarify item 8's quantifier to the existence reading, aligning it with the existence pattern items 2, 5, 6, and 7 already use for presence-only obligations — e.g. replace "a single line" with "at least one line". The underlying obligation is that the no-invent guidance be present, not that it appear exactly once; no failure mode analogous to items 1 and 3 (where duplicate occurrences would actually be wrong) exists for a duplicated no-invent instruction.
