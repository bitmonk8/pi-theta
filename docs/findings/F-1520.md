## Finding

The **Baseline acquisition** block in `version-bump-step2.md` requires the bump contributor to materialise the canonical diff "from" side by running `npm pack @earendil-works/pi-coding-agent@<pinned-version>` and extracting the tarball. This is the sole defined source for the extracted `package/dist/` tree that every `dist/*.js` diff on the page reads against — consumed by item (e)/(e.i)/(e.ii) (`dist/core/agent-session.js`), item (r) (`dist/core/messages.js`), item (aj) (`dist/core/session-manager.js`), and item (am) (`dist/core/sdk.js`).

The command silently presupposes that the pre-bump `<pinned-version>` is still retrievable from the registry at bump time — i.e. still published (not unpublished/deprecated to the point of being unfetchable) and reachable over the network. That precondition is never stated, and no fallback is specified for when it does not hold. The page in fact closes off the most obvious fallback by mandating the scratch tree be ephemeral and "MUST NOT be committed," so a retained in-repo snapshot is explicitly not the answer.

A fallback nonetheless already exists implicitly in the procedure and is left unsaid: before [step 1](version-bump-intro.md) overwrites the local install with the candidate minor, `node_modules/@earendil-works/pi-coding-agent/dist/` still holds the pinned tree, so a contributor who runs acquisition before that install can copy the locally-present pinned tree into the scratch directory without any registry round-trip. The spec mandates the network `npm pack` form without naming this already-present local copy as the fallback when the tarball cannot be fetched. Item (e) is a MUST-level audit (it sits in the (a)–(e) must-audit set), so the gap leaves a mandatory audit with an undefined recovery path in the unpublished-pin / offline case.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** ccc6e38 — pi-loom spec: resolve "Item (e)/(r) dist/*.js baseline source unspecified" (2026-06-07, Thomas Andersen)
**History:** Before ccc6e38 the baseline was described only abstractly as "the loom 1.0 Pi-SDK pin's `dist/*.js` snapshot" with no concrete acquisition mechanism. ccc6e38 introduced the **Baseline acquisition** block and its `npm pack @...@<pinned-version>` command to resolve a prior review finding about the unspecified baseline source; the same edit that pinned the concrete fetch mechanism also stated the ephemeral/MUST-NOT-commit rule but never stated the fetchability precondition or a fallback, so the gap entered with that commit.

## Solution Space

**Shape:** single

### Recommendation

In the Baseline acquisition block, state the fetchability precondition explicitly (the `npm pack` form requires the pinned `<pinned-version>` to be retrievable from the registry — still published and the network reachable) and add a fallback that does not violate the no-commit rule:

- Direct the contributor to run acquisition **before** step 1's install (the block already permits this ordering) and, when `npm pack` cannot retrieve the pinned tarball, copy the locally-installed pre-bump `node_modules/@earendil-works/pi-coding-agent/dist/` tree into the scratch directory as the "from" source — this tree holds the pinned version until step 1 overwrites it, and the copied tree remains ephemeral and uncommitted exactly like the `npm pack` output.
- State the terminal case: if neither the registry tarball nor a local pre-bump pinned tree is obtainable, the diff-based audits (e.i/e.ii, r, aj, am) cannot establish their "from" side, so item (e) is recorded `fail` under the existing conservative-posture rule (which already forces `fail` when the evidence cannot be established) and the bump cannot merge until the baseline is obtained.

Keep the scratch tree's ephemeral / MUST-NOT-commit property unchanged for every acquisition path.

## Consequence

**Severity:** advisory

If the pinned Pi-SDK version is unpublished/deprecated or the registry is unreachable at bump time, `npm pack @...@<pinned-version>` fails and the contributor has no defined "from" side for the diff-based audits (e.i/e.ii, r, aj, am), with no stated fallback. The MUST-level item (e) audit then cannot be completed and the bump procedure stalls without guidance, even though the pinned tree is in fact still present locally before step 1's install overwrites it.
