## Problem

The **Baseline acquisition** block (`#bump-baseline-acquisition`) requires running `npm pack @earendil-works/pi-coding-agent@<pinned-version>` to materialise the canonical "from" side for every `dist/*.js` diff on the page — the tree consumed by items (e), (r), (aj), and (am). The command silently presupposes the pinned version is still fetchable from the registry at bump time (still published and reachable over the network), but that precondition is never stated and no fallback source is specified. The block also mandates the scratch tree be ephemeral and "MUST NOT be committed," closing off a retained in-repo snapshot as the fallback. Item (e) is a MUST-level audit, so the gap leaves a mandatory audit with no defined recovery path in the unpublished-pin / offline case.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** ccc6e38 — pi-loom spec: resolve "Item (e)/(r) dist/*.js baseline source unspecified" (2026-06-07, Thomas Andersen)
**History:** Before ccc6e38 the baseline was described only abstractly as "the loom 1.0 Pi-SDK pin's `dist/*.js` snapshot" with no concrete acquisition mechanism. ccc6e38 introduced the **Baseline acquisition** block and its `npm pack @...@<pinned-version>` command to resolve a prior review finding about the unspecified baseline source; the same edit that pinned the concrete fetch mechanism also stated the ephemeral/MUST-NOT-commit rule but never stated the fetchability precondition or a fallback, so the gap entered with that commit.

## Solution approach

In the `#bump-baseline-acquisition` block, state the fetchability precondition (the `npm pack` form requires the pinned `<pinned-version>` to be registry-retrievable) and add a fallback "from" source for when it is not: the locally-installed pre-bump `node_modules/@earendil-works/pi-coding-agent/dist/` tree, which holds the pinned version until `#pi-version-bump-procedure` step 1 overwrites it. Name the terminal case where neither source is obtainable as a recorded `fail` on item (e) under the existing conservative-posture rule.

## Solution constraints

- The scratch tree's ephemeral / MUST-NOT-commit property must hold unchanged for every acquisition path, including the local-copy fallback.
