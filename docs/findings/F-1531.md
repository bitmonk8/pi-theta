## Finding

The *Category-(1)/(3) inventory join key (per-category specialisation)* paragraph in `audit-resolution.md` defines resolution as an exact match between the access's member name and "the **rightmost dot-separated segment** of an entry's `path` field" — without restricting the candidate entry set to the kind that matches the access category. A `pi.<member>` access (carrier `pi: ExtensionAPI`, category (1)) and a `ctx.<member>` access (carrier `ctx: ExtensionContext` / `ExtensionCommandContext`, category (3)) both consult the same global pool of audit-participating entries via this rule, so any cross-category name collision on the rightmost segment is ambiguous: a `pi.signal` access could resolve against a category-(3) `ctx.signal` entry (and vice versa), authorising surface the inventory never intended to expose under that carrier.

The same paragraph extends the cross-entry duplicate-rightmost-segment uniqueness rule symmetrically from the category-(2) leftmost-segment uniqueness rule one paragraph above. As written it is also global: a legitimate, distinct pair `path: "pi.signal"` (category-(1) kind) and `path: "ctx.signal"` / `path: "ExtensionContext.signal"` (category-(3) kind) — two unrelated surfaces that happen to share a leaf name across categories — would trip `literal-read/duplicate-rightmost-segment` even though neither overlap is observable to any single access carrier.

Both halves are facets of the same omission: the spec asserts a per-category specialisation in the paragraph heading and reasons explicitly about entry shapes that differ by category, but never bolts the `kind` filter onto either the resolution match or the duplicate-segment uniqueness check. The implementation can either honour the per-category framing and scope by `kind`, or honour the literal "an entry's `path` field" wording and scope globally; the two readings give different audit verdicts on the same source tree.

## Solution Space

**Shape:** single

### Recommendation

Add the `kind` filter explicitly to both the resolution rule and the duplicate-segment uniqueness rule in the *Category-(1)/(3) inventory join key (per-category specialisation)* paragraph and its predecessor *Category-(2) inventory join key* paragraph's closing uniqueness sentence:

- In the category-(1) resolution sentence, replace "the **rightmost dot-separated segment** of an entry's `path` field" with "the **rightmost dot-separated segment** of the `path` field of an `SDK_SURFACE_INVENTORY` entry whose `kind` is a category-(1) audit-participating kind (per *Target surface categories* above and the implementation-owned entry-kind enumeration that pins it)". Apply the symmetric edit to the category-(3) resolution sentence.
- In the duplicate-segment uniqueness rule that closes the *Category-(2) inventory join key* paragraph, replace "For category-(1) and category-(3) audit-participating kinds, the same uniqueness rule applies symmetrically on the rightmost dot-separated segment" with two scoped statements: "Within the set of category-(1) audit-participating kinds, no two entries may share their rightmost dot-separated `path` segment; the same rule applies independently within the set of category-(3) audit-participating kinds. The check is **not** taken across the two category sets — a legitimate `path: \"pi.signal\"` (category-(1) kind) and `path: \"ExtensionContext.signal\"` (category-(3) kind) pair MUST resolve cleanly and MUST NOT fire `literal-read/duplicate-rightmost-segment`."
- Keep the existing lex-min tie-break and the cross-reference to the literal-read assertion's `literal-read/duplicate-rightmost-segment` discriminator unchanged; both already key off the per-category-scoped duplicate-segment rule once the scope is pinned.

Edge cases the implementer should watch: (i) the `path` convention for category-(3) entries is implementation-owned (the spec illustrates both `ExtensionContext.modelRegistry` and `ctx.modelRegistry`); the rightmost-segment match is invariant under that choice, but the per-category `kind` filter MUST consume the entry's `kind` discriminator rather than the literal carrier prefix in the `path` value, since the latter is unstable. (ii) The category-(2) leftmost-segment uniqueness rule one paragraph above is already implicitly per-category because no category-(1)/(3) entry's leftmost segment is a peer-import symbol, but the parallel edit above leaves that rule untouched — make the per-category scoping explicit there too if a follow-up wants symmetric wording, otherwise it remains correct as-is. (iii) The category-(3) drift-smoke-check `SHOULD` in `version-bump-step2.md` already iterates entries by category; no edit is required there.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the same source tree. Under the unscoped reading, a `ctx.signal` entry silently authorises a `pi.signal` access (and vice versa) — the audit's whole point is that the inventory closes the SDK surface per carrier, so cross-carrier resolution is a soundness hole. Under the same unscoped reading, the duplicate-rightmost-segment uniqueness rule fires on legitimate cross-category name pairs, blocking otherwise-conformant inventories from loading. Either way the audit's verdict on `main` is implementation-defined, which defeats the gate.
