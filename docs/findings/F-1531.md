## Problem

The `#audit-category-1-3-inventory-join-key` resolution rule matches an access's member name against the rightmost dot-separated segment of "an entry's `path` field" without restricting the candidate entries to the `kind` that matches the access category. A category-(1) `pi.<member>` access and a category-(3) `ctx.<member>` access therefore consult the same global entry pool, so a cross-category leaf-name collision (e.g. a `pi.signal` access resolving against a `ctx.signal` entry) authorises surface the inventory never meant to expose under that carrier. The symmetric duplicate-rightmost-segment uniqueness rule that closes `#audit-category-2-inventory-join-key` is likewise global: a legitimate distinct pair `path: "pi.signal"` (category-(1) kind) and `path: "ctx.signal"` / `path: "ExtensionContext.signal"` (category-(3) kind) trips `literal-read/duplicate-rightmost-segment` even though neither overlap is observable to any single carrier. The paragraph asserts a per-category specialisation but never bolts the `kind` filter onto either the match rule or the uniqueness check, so the two readings give different audit verdicts on the same source tree.

## Solution approach

Narrow the `#audit-category-1-3-inventory-join-key` resolution rule so each category's rightmost-segment match consults only entries whose `kind` is an audit-participating kind for that category, keeping category (1) and category (3) on separate candidate sets. Narrow the duplicate-rightmost-segment uniqueness rule that closes `#audit-category-2-inventory-join-key` so uniqueness is enforced independently within the category-(1) kind set and within the category-(3) kind set, and not across the two sets.

## Solution constraints

- The per-category filter MUST key on the entry's `kind` discriminator, not on the carrier prefix in the entry's `path` value (the `path` convention for category-(3) entries is implementation-owned and unstable).
