## Finding

`V6c` asserts (Tests + Ships-when) that `loom/load/unknown-tool` rejects a `tools:` entry naming a Pi tool "absent from the registry … at load time." The spec rule this implements — [`frontmatter-fields-a.md` §`tools`](../../docs/spec_topics/frontmatter/frontmatter-fields-a.md) ("Pi tool names … resolve against Pi's tool registry at loom-load time, exactly as for Pi subagents") — is one-shot: the per-loom load pass reads Pi's tool registry once and rejects any name not present at that instant. Correctness of that rejection therefore rests on a host-ordering precondition: that Pi's tool registry is **fully populated** by the time the `session_start`-time per-loom load pass runs. Nothing in `V6c`, in the cited spec rule, or in the `tool-registration-lifetime.md` contract states that precondition — `tool-registration-lifetime.md` governs the lifetime/visibility of *loom-synthesised* tools, not the population timing of Pi's built-in registry.

The spec already pins the structurally identical concern for the **model** registry: [`host-prerequisites.md` §"Host prerequisite for binder-model / `model:` load-time resolution"](../../docs/spec_topics/pi-integration-contract/host-prerequisites.md#model-registry-population-presupposition) treats "the set `getAvailable()` returns to a `session_start`-time load pass is the fully-populated set, not a partial or still-warming view" as a named presupposition, records the one-shot/no-retry consequence (a provider arriving after the load pass leaves the loom load-failed until `/reload`), and routes re-confirmation to version-bump editorial-review checklist [item (k)](../../docs/spec_topics/pi-integration-contract/version-bump-step2.md#bump-checklist-model-registry-population). The Pi **tool** registry has no equivalent presupposition paragraph and no parallel checklist item, even though `loom/load/unknown-tool` is one-shot and no-retry in exactly the same way.

The gap is a missing host-prerequisite pin in the spec, surfaced through the plan leaf that depends on it. `V6c` faithfully implements the spec rule; it is the dependency on an unpinned host-ordering property — present for `model:`/`bind_model:` and explicitly absent for `tools:` — that is the defect.

## Issue introduction

**Verdict:** present-since-inception (partial-fix asymmetry)
**Introducing commits:** `79fa3b9` (asymmetry crystallised), `60c9400` (plan-side dependency made concrete)
**History:** The `tools:`-resolve-against-Pi's-registry-at-loom-load-time rule predates the spec topic-file split (it lived in the monolithic `spec.md`; `f5e89f4` only relocated it) and never carried a registry-population presupposition — so the gap is present since the rule's inception. The structurally identical model-registry concern was raised and resolved spec-side in `79fa3b9` (2026-06-04, "pi-loom spec: resolve 'Binder model / model: resolution — registry-population timing'"), which added the `#model-registry-population-presupposition` paragraph and version-bump checklist item (k) but did not add the parallel tool-registry presupposition; that commit is where the model-vs-tool asymmetry became visible. The plan-side dependency was made concrete in `60c9400` (2026-06-17, "pi-loom plan: resolve 'tools:-resolution load codes have no asserting plan leaf'"), which authored `V6c`'s `loom/load/unknown-tool` "at load time" rejection.

## Solution Space

**Shape:** single

### Recommendation

The durable fix is spec-side: pin the tool-registry-population precondition the same way the model-registry one is pinned.

- In `docs/spec_topics/pi-integration-contract/host-prerequisites.md`, add a host-prerequisite presupposition paragraph for `tools:` load-time resolution against Pi's tool registry, parallel to the existing `#model-registry-population-presupposition`: state that the `session_start`-time per-loom load pass resolves `tools:` Pi-tool names one-shot against Pi's tool registry, that loom 1.0 treats "the registry is fully populated by the time the load pass reads it" as a presupposition (same footing as the model-registry and *No concurrent user sessions* framings), and record the one-shot/no-retry consequence (a tool registered after the load pass leaves any loom whose `tools:` named it load-failed with `loom/load/unknown-tool` until `/reload`).
- In `docs/spec_topics/pi-integration-contract/version-bump-step2.md`, add an editorial-review checklist item mirroring item (k) that has the contributor confirm, against the candidate Pi minor, that Pi's tool registry is fully populated by the first `session_start`-time load pass; SHOULD-level, escalating to a build-time pin once Pi exposes a typed tool-registry-population-ordering contract the surface-inventory probe can verify.
- Cross-reference the new anchor from the §`tools` rule in `frontmatter-fields-a.md`, as the model rule cross-references its presupposition.

Once that anchor exists, `V6c` (and, if it wires a tool-registry snapshot through the load pass, `V9b`) may forward-reference it exactly as `V9b` forward-references `#model-registry-population-presupposition`.

**Ownership pins (Class 1 / Class 2):**

- The spec (`docs/spec.md` + `docs/spec_topics/**`) is **read-only for the inner plan-diff-fix loop**; the plan-diff-fixer MUST NOT edit it. The host-prerequisite presupposition and the version-bump checklist item above are an out-of-loop spec edit, owned per the **Spec drift** rule in `conventions.md` ("fix the spec first").
- No in-loop plan edit is required: `V6c`/`V6c-T` already implement the spec's `loom/load/unknown-tool` load-time rejection correctly. The plan-side forward-reference is contingent on the spec edit landing and honours the existing leaf format and `H<n><letter>`/`V<n><letter>` ID scheme (`no-invented-ids`).

## Consequence

**Severity:** advisory

If unfixed, the assumption that Pi's tool registry is fully populated at the loom load pass stays an unpinned host-ordering property — unlike its explicitly-pinned model-registry twin. A Pi minor (or host behaviour) that registers a built-in tool after the `session_start` load pass would cause a loom's otherwise-valid `tools:` entry to be wrongly rejected as `loom/load/unknown-tool` and the loom to not register, with no version-bump editorial-review item to catch the regression — the same one-shot/no-retry failure mode the spec already deemed worth pinning for the model registry.
