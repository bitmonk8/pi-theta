## Problem

`H5a` (`docs/plan_topics/H5a-closing-gate-automation.md`) builds a single CI gate whose `Adds.` bundles many reconciliation arms over heterogeneous inputs. Most arms reconcile the spec REQ-ID set, the diagnostics code registry, and the test corpus, but the standing transitive-completeness arm reads a different data source entirely — the `coverage-matrix.md` closing-leaf cells and `H5b`'s `Deps.` line — with its own range-token expansion grammar, named-singleton handling, IMPL-row back-reference exclusion, and fixture family. One leaf therefore builds and fixtures distinct recognisers across spec, registry, test-corpus, and plan-file inputs, making the single implementation step too large for one coherent review pass.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `c6a664e` (2026-06-10, initial plan build — leaf born with five arms: unmapped-REQ-ID, registry-code, asserted-code-absent, retired/live clash, numbering hole); `0603eb4` (2026-06-10, added un-anchored-MUST arm); `953e3fa` (2026-06-10, added numbered-REQ-ID citing-test arm); `08bd641` (2026-06-11, added the plan-structural transitive-completeness arm — the decisive cross-data-source addition, resolving "H5b coverage Deps completeness has no mechanical backstop"); `ea6b1da` (2026-06-11, added warn-only live-corpus mode).
**History:** `git log --follow` shows the leaf was multi-arm at inception (`c6a664e`) and accreted further arms across subsequent fix-loop commits. The specific defect the finding flags — multiple arms spanning *different data sources* — became true at `08bd641`, when the plan-structural arm that reads plan files (rather than the spec/registry/test surfaces the earlier arms read) was added. A later over-bundling pass (`cb8737a`, 2026-06-12) split the `no-broad-catch` arm out to `H5c` but left the transitive-completeness arm in `H5a`. The durable fix therefore targets the accreted cross-data-source bundling, not any single revert.

## Solution approach

Extract the plan-structural transitive-completeness arm out of `H5a` into a new horizontal leaf that depends on `H5a`'s gate framework, mirroring how `H5c` factored the `no-broad-catch` arm out. Strike the transitive-completeness arm from `H5a`'s `Adds.`, its Tests bullet, and its Ships-when clause, and carry that arm's `Adds.`/Tests/Ships-when into the new leaf with `Deps.` naming `H5a` and explicit reuse of `H5a`'s seeded test-fixtures root. Link the new leaf into the Horizontal phases list in `docs/plan.md`. Keep the warn-only live-corpus mode and the shared gate framework in `H5a`.

## Solution constraints

- The new leaf's ID must follow the horizontal `H<n><letter>` scheme (next free letter in the phase; never reuse a retired ID) per the project `## Plan rules`.
- Splitting the arm into a new leaf must keep `docs/plan_topics/conventions.md` and `docs/plan_topics/coverage-matrix.md` consistent per the project `## Plan rules`.
