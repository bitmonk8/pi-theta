## Finding

`H5a` (`docs/plan_topics/H5a-closing-gate-automation.md`) builds a single CI gate whose `Adds.` enumerates many reconciliation arms over heterogeneous inputs: unmapped-executable-REQ-ID, numbered-REQ-ID-citing-test, registry-code-with-no-asserting-test, asserted-code-absent-from-registry, un-anchored-MUST, retired/live ID clash, and per-prefix numbering hole (all reading the spec REQ-ID set, the diagnostics registry, and the test corpus), plus a **plan-structural transitive-completeness arm** that reads a different data source entirely — the `coverage-matrix.md` closing-leaf cells and `H5b`'s `Deps.` line — and a **warn-only live-corpus mode**. One leaf therefore builds and fixtures several distinct recognisers across spec, registry, test-corpus, and plan-file inputs, each with its own seeded fixtures.

The transitive-completeness arm is the outlier: it is the only arm that parses plan files (range-token expansion over `<group><letter>` suffixes, named-singleton handling, IMPL-row back-reference exclusion) rather than reconciling spec/registry/test surfaces. It exercises a separate recognition path and a separate fixture family from the REQ-ID and diagnostic-code arms it ships alongside.

This is borderline rather than clear-cut: all arms genuinely share one gate-build harness, one seeded-fixtures root, and one `H6a` live-corpus activation flip, which is real cohesion. The prior over-bundling review pass (commit `cb8737a`) already extracted the `no-broad-catch` allow-list arm into its own leaf `H5c` (which depends on `H5a` and reuses the shared fixtures root), establishing a precedent for peeling a data-source-distinct arm off `H5a` while keeping the shared machinery in place.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `c6a664e` (2026-06-10, initial plan build — leaf born with five arms: unmapped-REQ-ID, registry-code, asserted-code-absent, retired/live clash, numbering hole); `0603eb4` (2026-06-10, added un-anchored-MUST arm); `953e3fa` (2026-06-10, added numbered-REQ-ID citing-test arm); `08bd641` (2026-06-11, added the plan-structural transitive-completeness arm — the decisive cross-data-source addition, resolving "H5b coverage Deps completeness has no mechanical backstop"); `ea6b1da` (2026-06-11, added warn-only live-corpus mode).
**History:** `git log --follow` shows the leaf was multi-arm at inception (`c6a664e`) and accreted further arms across subsequent fix-loop commits. The specific defect the finding flags — multiple arms spanning *different data sources* — became true at `08bd641`, when the plan-structural arm that reads plan files (rather than the spec/registry/test surfaces the earlier arms read) was added. A later over-bundling pass (`cb8737a`, 2026-06-12) split the `no-broad-catch` arm out to `H5c` but left the transitive-completeness arm in `H5a`. The durable fix therefore targets the accreted cross-data-source bundling, not any single revert.

## Solution Space

**Shape:** multiple

### Option A — Keep H5a whole; make the data-source partition explicit

**Approach.** Leave all arms in `H5a`. In the `Adds.`, surface the shared gate-harness-plus-per-arm-recogniser structure and call out which arms read which input (spec REQ-ID set, diagnostics registry, test corpus, plan files) so a reviewer can verify arm-by-arm.

**Plan edits.** `docs/plan_topics/H5a-closing-gate-automation.md` `Adds.` only — group the arm enumeration by input source so the plan-file arm is visibly distinct from the spec/registry/test arms.

**Spec edits.** None.

**Pros.** No new leaf; preserves the single shared fixtures root and single `H6a` activation flip; lowest churn.

**Cons.** The leaf stays large; the reviewability concern is mitigated, not removed.

**Risks.** None material.

### Option B — Extract the plan-structural transitive-completeness arm to a new leaf

**Approach.** Move the transitive-completeness plan-structural arm (range-token expansion, named-singleton handling, IMPL-row back-reference exclusion, and its seeded `coverage-matrix.md`/`H5b`-`Deps.` fixtures) out of `H5a` into a new horizontal leaf that depends on `H5a`'s gate framework, mirroring how `H5c` already factors the `no-broad-catch` arm out of `H5a` and reuses the shared fixtures root. Keep the warn-only live-corpus mode with the framework in `H5a`.

**Plan edits.** In `docs/plan_topics/H5a-closing-gate-automation.md`, strike the transitive-completeness arm paragraph from `Adds.`, its Tests bullet, and the corresponding Ships-when clause. Create the new leaf (`<new>`) carrying that arm's `Adds.`/Tests/Ships-when, with `Deps.` naming `H5a`; reuse the dedicated test-fixtures root `H5a` establishes. Link the new leaf into the Horizontal phases list in `docs/plan.md` (and the Release-gate prose if the arm's relationship to `H5b`/`H6a` is described there). Because the arm reads `H5b`'s `Deps.` line, keep that input relationship intact in the new leaf's text.

**Spec edits.** None.

**Pros.** Each leaf is reviewable in one pass; the cut follows the data-source boundary (plan files vs spec/registry/test); consistent with the established `H5c` precedent.

**Cons.** One more leaf and one more `<id>-complete` sequencing edge; the new leaf must restate the shared-fixtures-root and live-corpus-activation relationships.

**Risks.** Splitting could fragment the seeded-fixtures story if the new leaf does not explicitly reuse `H5a`'s fixtures root (the `H5c` text shows how to state that reuse).

### Recommendation

Take **Option B**: extract the plan-structural transitive-completeness arm into a new horizontal leaf depending on `H5a`, keeping warn-only mode with the `H5a` framework. The cut follows the one real data-source boundary in the leaf and reuses the `H5c` pattern already in the corpus, so the shared-fixtures-root and `H6a`-activation relationships have a known phrasing to mirror. Implementer must watch: the new leaf's arm reads `H5b`'s `Deps.` line and the `coverage-matrix.md` closing-leaf cells, so its `Deps.` and fixtures must preserve those inputs; and the IMPL-row back-reference exclusion and range-token grammar travel with the arm to the new leaf. If minimising churn is the binding constraint instead, Option A is acceptable since the arms share one gate harness and one activation flip.

## Consequence

**Severity:** advisory

If shipped as-is, an implementer can still produce a working gate — the arms are well specified — but the leaf is large enough that a reviewer cannot verify the spec/registry/test recognisers, the plan-file transitive-completeness recogniser, the warn-only mode, and all their seeded fixtures in one coherent pass, and the single implementation step is correspondingly large. No two implementers would diverge on behaviour; the cost is reviewability and step size, not correctness.
