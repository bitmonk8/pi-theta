## Finding

The normative restrictions on `@`...`` query templates — that `@` "has only one lexical role" and "never appears inside `${...}`", and that "there is no bash-style argument-slice sugar (`${@:N}`, `$1`, `$@`, `$ARGUMENTS`)" — live exclusively in `docs/spec_topics/frontmatter/frontmatter-fields-b-and-templates.md` §"Template Interpolation". They constrain the lexical structure of the `@`-template interpolation slot, but the query topic (`query.md` + `query/query-forms.md` + `query/query-escapes-stringification.md`) contains no statement of either rule and no forward link to the frontmatter page that does.

A reviewer or implementer reading the Query subtree for the full template-interpolation contract finds QRY-17 (escapes) and QRY-18 (stringification by static type) and nothing about which lexical forms are admissible inside `${...}` — so they cannot tell, from the query pages alone, that `@${...something...}` is unreachable, or that an author can't write `${@:1:2}` inside a `@`-template. The frontmatter location is incidental history: the prohibition replaced bash-style sugar that previously lived under "Argument Styles" / "Template Interpolation Disambiguation" in the pre-split spec, and the rephrased version stayed where the deleted text used to be rather than moving to its new natural home in the Query topic.

## Solution Space

**Shape:** single

### Recommendation

Move the two lexical rules — `@` has no role inside `${...}`, and there is no bash-style argument-slice sugar (`${@:N}`, `$1`, `$@`, `$ARGUMENTS`) — into `docs/spec_topics/query/query-escapes-stringification.md`, adjacent to QRY-17 (which already owns the lexical rules for the `@`-template body: escapes, unterminated templates, the special meaning of `${`). Add a new short sub-section (e.g. "## Interpolation slot syntax" placed between "## Escapes" and "## Stringification of interpolated values") that states:

- A `${...}` interpolation inside a `@`...`` query template contains a Loom expression from the Expression Sublanguage, evaluated up to the matching `}`.
- The `@` character has only one lexical role — introducing a query template at top level — and never appears inside `${...}`.
- There is no bash-style argument-slice sugar (`${@:N}`, `$1`, `$@`, `$ARGUMENTS`); slash-command arguments are bound to typed `params` via the Slash-Command Argument Binding machinery and referenced by their declared parameter names like any other identifier.

In `frontmatter/frontmatter-fields-b-and-templates.md` §"Template Interpolation", replace the relocated text with a one-sentence forward link: "The lexical rules for `${...}` inside a `@`...`` template — including the prohibition on bash-style argument-slice sugar — are pinned in [Query — Interpolation slot syntax](../query/query-escapes-stringification.md#interpolation-slot-syntax)." Keep the cross-reference paragraph that ties stringification to the canonical Query table (already present) intact.

Anchor the new sub-section so the frontmatter forward link resolves stably. The existing reference to the binder machinery (`../binder.md`) moves with the relocated text and resolves correctly from the new location (the relative path becomes `../binder.md` from the query/ directory as well).

## Consequence

**Severity:** advisory

An implementer reading only the Query topic builds a `@`-template lexer without these two constraints and may produce divergent behaviour at the `${...}` boundary (e.g. treating `@` as significant inside the interpolation slot, or recognising bash-style argument-slice forms because nothing in the Query topic forbids them). The rule is not silently lost — it does exist in the frontmatter section — but the locality cost is real: the prohibition lives where lexer authors will not look for it.
