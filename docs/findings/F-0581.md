## Finding

In `cancellation.md`, the three-clause late-settlement discard at the tool-call boundary is anchored as CNCL-1/CNCL-2/CNCL-3. The clauses are deliberately split into per-REQ-ID sub-obligations so a test or cross-page citation can pin one without dragging in the others. CNCL-2 reads "The runtime MUST NOT emit a second `Err` **for the same invocation**", but CNCL-3 reads only "The runtime MUST NOT emit a second `RuntimeEvent`" — the scope qualifier is dropped.

Because the page itself invites per-clause citation in isolation, CNCL-3 cited alone reads as a *session-wide* prohibition on emitting a second `RuntimeEvent`, when the intended scope is per-invocation suppression of the late settlement for the cancelled call. Both a session-wide and a per-invocation implementation are literally compliant with the bare CNCL-3 text. The intended scope is unambiguous elsewhere in the same section: the swallowing-handler paragraph states "no second `RuntimeEvent` reaches the always-log channel **for the same invocation**", and the plan's `V17a` test restores the qualifier ("no second `RuntimeEvent` is produced **per invocation**"). The clause text is the only place the qualifier is missing.

## Issue introduction

**Verdict:** introduced-by-commit (single commit; defect present since the rule's first authoring)
**Introducing commit:** `11ef3b9c26ee5dcc9a1ba1e5970af9912f2e1c07` — "pi-loom spec: resolve \"Tool failure modes beyond throw / isError: true unspecified\"" (2026-05-07)
**History:** `git log -G 'second .Err. for the same invocation'` and `git show 11ef3b9 -- spec_topics/cancellation.md` show this commit first added the three-obligation discard sentence to `cancellation.md`. From that first appearance the text read "...MUST NOT emit a second `Err` **for the same invocation**, and MUST NOT emit a second `RuntimeEvent`." — clause (b) carried the qualifier, clause (c) did not. The phrase was absent from the parent tree (`git grep` against `11ef3b9^`), confirming this commit is the origin. The asymmetry was then carried through unchanged by `d9f42d0` (2026-05-08, "split four cancellation MUSTs into per-obligation paragraphs") and `482098b` (2026-06-04, "split into three CNCL-N sub-anchors"); neither refactor corrected it. The defect is therefore present since the rule was introduced, not injected by a later edit.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/cancellation.md`, append the scope qualifier to the CNCL-3 clause so it reads:

> **CNCL-3.** *(clause (c) — no-second-`RuntimeEvent`.)* The runtime MUST NOT emit a second `RuntimeEvent` for the same invocation.

This matches the qualifier CNCL-2 already carries and the swallowing-handler paragraph's existing "no second `RuntimeEvent` reaches the always-log channel for the same invocation" phrasing in the same section.

The two cross-reference glosses (`host-interfaces-core.md` and `tool-calls.md`) cite CNCL-3 by anchor with the short label "no-second-`RuntimeEvent`"; they need no change. The plan is read-only for this fix — `V17a`/`V17a-T` already assert the per-invocation scope and require no edit.

## Consequence

**Severity:** correctness

A consumer that cites CNCL-3 in isolation (as the per-clause anchoring invites) can implement session-wide `RuntimeEvent` suppression while another implements per-invocation suppression; both pass a literal reading of the clause, and two reasonable implementers diverge. The divergence is masked for `V17a` (its test already pins "per invocation") but the underspecified clause remains a trap for any other citation of CNCL-3.
