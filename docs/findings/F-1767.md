## Problem

The **Stop-reason classification** subsection (`#stop-reason-classification`) in `provider-error-mapping.md` keys `ContextOverflowError` off the inline assertion that pi-ai surfaces output truncated at the model's length boundary as `stopReason: "length"` for every supported provider. That is a behavioural property of pi-ai's per-provider adapters collapsing each native truncation terminator (OpenAI `finish_reason: "length"`, Anthropic `stop_reason: "max_tokens"`, Mistral `finish_reason: "length"`, Bedrock-native `stopReason: "max_tokens"`) to the literal `"length"` — not a typed surface gated by pi-ai's version. Unlike the sibling `#provider-overflow-wording-presupposition` and `#classifier-input-surface` paragraphs, this presupposition carries no name, no editorial-review routing, and no checklist item. A pi-ai minor that stopped collapsing (or a new adapter forwarding a native terminator) falls through to the catch-all arm and silently misclassifies a real overflow as `TransportError { retryable: false }`, caught by no existing gate.

## Issue introduction

**Verdict:** introduced-by-commit
**Introducing commits:** `ef4cfb8` ("pi-loom spec: resolve \"Stop-reason → QueryError variant mapping is undefined\"", 2026-06-07)
**History:** `git log -S 'stopReason: "length"' -- docs/spec_topics/pi-integration-contract/provider-error-mapping.md` reports a single commit, `ef4cfb8`, which added the entire **Stop-reason classification** subsection (the `stop-reason-classification` anchor and both bullets) and is the first introduction of the `stopReason: "length"` literal anywhere in `docs/spec_topics/`. The earlier `44782e9` ("TransportError catch-all / pi-ai provider-error surface") had already established the *Classifier input surface* and *Provider-owned-wording presupposition* sibling paragraphs with their editorial-review hooks (items (af), (i)), but those were body-envelope-error presuppositions and did not cover the HTTP-200 stop-reason terminator path. `ef4cfb8` introduced that path's classifier and the cross-provider `"length"` collapse in the same edit without adding a parallel editorial-review hook for the new presupposition — the defect is precisely the omission of the hook on the new subsection, not a regression of a pre-existing hook.

## Solution approach

Promote the inline length-terminator clause under `#stop-reason-classification` into a named, anchored presupposition paragraph on the same footing as the `#provider-overflow-wording-presupposition` sibling, naming the per-provider native terminators it collapses, its silent-misclassification failure mode, and its authoritative evidence trail. Add a forward-link from that paragraph to a matching SHOULD-level audit item in the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md`, and add that checklist item routing detection of a collapse regression to the per-provider length-terminator fixtures, worded after the existing item (i) / (af) templates.

## Solution constraints

- When adding the new checklist item, co-edit any count-bearing prose that names the checklist letter range to the new high-water letter in the same commit.
- Out of scope: the classifier-precedence wording owned by F-1521 and the **Stop-reason classification** bullet-pair wording owned by F-1649.
