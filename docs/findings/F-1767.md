## Finding

`provider-error-mapping.md` keys `ContextOverflowError` off a single literal: pi-ai surfaces output truncated at the model's length boundary as `stopReason: "length"` for every loom 1.0-supported provider. That sentence is doing real cross-provider work — OpenAI emits `finish_reason: "length"`, Anthropic emits `stop_reason: "max_tokens"`, Mistral emits `finish_reason: "length"`, and the Bedrock unified surface emits `stopReason: "max_tokens"` for the Anthropic family. The rule is therefore not a property of one provider's wire format but a behavioural property of pi-ai's per-provider adapters: they must collapse each native truncation terminator to the literal string `"length"` on `AssistantMessage.stopReason`.

That is exactly the shape of presupposition the page already treats with care. The neighbouring *Provider-owned-wording*, *Classifier input surface*, and *`onResponse` provider-response delivery* paragraphs each name the unpinned behaviour, state which loom rule depends on it, and route detection to a numbered item of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) (items (i), (af)). The length-terminator rule is asserted inline with no such hook: it carries no name, no editorial-review routing, and no checklist item to fail loud on a bump.

A pi-ai minor that stopped collapsing (e.g. surfaced Anthropic's `"max_tokens"` raw, or introduced a new provider whose adapter forwards a native terminator) is caught by neither the SDK surface-inventory test (pi-ai's `StopReason` typed union covers the literal but does not gate which adapters emit it) nor any other gate the page enumerates. The rule's failure mode is silent misclassification: a real context-overflow output-boundary terminator falls through to the catch-all "any other non-terminal stop reason" arm and maps to `TransportError` with `retryable: false`, exactly the wrong outcome for an overflow.

## Solution Space

**Shape:** single

### Recommendation

Lift the inline length-terminator clause into a named *pi-ai length-terminator normalisation presupposition* paragraph immediately under the **Stop-reason classification** subsection (on the same footing as the *Provider-owned-wording presupposition* sibling), and add a corresponding numbered item to the *Editorial-review checklist for unpinned host presuppositions* in `version-bump-step2.md`.

Concretely, the edit shape is:

1. In `provider-error-mapping.md`, after the two **Stop-reason classification** bullets, add a paragraph anchored e.g. `<a id="length-terminator-normalisation-presupposition"></a>` that:
   - States the presupposition explicitly: for each loom 1.0-supported provider (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`), the per-provider pi-ai adapter collapses the provider-native truncation terminator (OpenAI `finish_reason: "length"`, Anthropic `stop_reason: "max_tokens"`, Mistral `finish_reason: "length"`, Bedrock-native `stopReason: "max_tokens"`) to the literal `AssistantMessage.stopReason: "length"`.
   - Notes the failure mode: a silent regression — or a new provider whose adapter forwards a native terminator — misclassifies a real output-boundary overflow as `TransportError { retryable: false }` via the catch-all arm.
   - Routes detection to editorial review per a new item (let the bump-checklist edit choose the next free letter, e.g. `(aq)`) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure), and notes the per-provider length-terminator fixtures the editorial run exercises.
   - Adds an *Authoritative source* sentence on the same footing as the sibling presuppositions, pinning the per-provider adapter modules under `dist/providers/` in `@earendil-works/pi-ai` (e.g. `dist/providers/anthropic.js`, `dist/providers/openai-completions.js`, `dist/providers/mistral.js`, `dist/providers/amazon-bedrock.js`) as the **known-fragile evidence** trail, parallel to the *`complete()` forced-tool* and *retry-and-cancellation* presuppositions in `conversation-drive.md`.

2. In `version-bump-step2.md`, add the matching SHOULD-level audit item (worded after the (i) / (af) / (am) templates):
   - Anchor: `<a id="bump-checklist-length-terminator-normalisation"></a>`.
   - Body: SHOULD confirm against the candidate `@earendil-works/pi-ai` minor that each supported provider's adapter still collapses its native truncation terminator to `AssistantMessage.stopReason: "length"`; SHOULD re-run the per-provider length-terminator fixtures under `npm test`. Escalates to a build-time pin once pi-ai exposes a typed per-adapter terminator-mapping contract (same closing sentence the (al) / (am) items use).
   - Re-mention the item in the **Provider error mapping — *Re-validation gate*** paragraph's enumeration if the bump-procedure side already lists items by letter (the existing *Re-validation gate (loom 1.0.0)* prose references item (i) explicitly; touch only the same prose if it gathers a list).

Implementer edge cases: (i) Bedrock's Anthropic-family adapter and Bedrock's own native terminator surface MAY differ — the presupposition is per pi-ai adapter, not per upstream provider, so the fixture set keys on pi-ai adapter modules under `dist/providers/`. (ii) The new presupposition is independent of the *Provider-owned-wording presupposition* — the latter governs the `AssistantMessage.errorMessage` overflow-signature wording on HTTP-error paths, the former governs `AssistantMessage.stopReason` on the HTTP-200 truncation path; both editorial runs are needed on a Pi minor bump. (iii) The numbered-item letter (e.g. `(aq)`) is whatever is next free after `(ap)` at the time of the edit, and the count-bearing prose elsewhere on the page that names the checklist range (e.g. "items `(a)`–`(ap)`" or "items `(f)`–`(ap)`" in `V18c`'s **Ships when**) MUST be co-edited to the new high-water letter in the same commit.

## Consequence

**Severity:** correctness

A `@earendil-works/pi-ai` minor (or a new pi-ai provider adapter) that surfaces a provider's native length terminator unchanged — `"max_tokens"`, `"MAX_TOKENS"`, `"truncated"`, etc. — silently misroutes every output-boundary truncation through the catch-all into `TransportError { retryable: false }` instead of `ContextOverflowError`. Two consequences follow: (1) the truncated assistant response is reported to loom code as a transport failure rather than as overflow, defeating the [`Detection of ContextOverflowError`](../spec_topics/query/query-failure-and-repair.md#detection-of-contextoverflowerror) repair path, and (2) the failure is the spec's own "definite outcome of the request, not a transient condition" classification, so no retry covers for the misclassification. The defect is invisible to every existing gate (SDK surface inventory, peer-dep pin, capability probe).
