## Problem

`H4a`'s in-memory fixture-supply mechanism states its load-bearing properties — the harness supplies fixture `.loom` content in memory, with no `FileSystem` seam (`V8b`) dependency and no ambient `src/**` read — but never states the contract by which that content reaches a harness-driven leaf's discovery; no seam, parameter, or other injection point is named. `H3a`'s injected-seam list carries no fixture-source seam, and `M`/`M-T` consume the mechanism for their single-source happy-path discovery without naming the point at which discovery binds to the supplied source. Because the injection point is unspecified, the `H4a` implementer must invent an ad-hoc one and `M`'s discovery may bind to a different point than the harness supplies, so the supply and read halves cannot be wired coherently and may diverge silently.

## Issue introduction

**Verdict:** single-commit
**Introducing commit:** `2d52ad1` — `pi-loom plan: resolve "M's happy-path discovery read has no defined source under the ambient-access ban"`
**History:** The in-memory fixture-supply mechanism was introduced in `2d52ad1` (the fix for `M`'s undefined discovery-read source). That commit added the mechanism's load-bearing properties (in memory, no `FileSystem` seam, no ambient read) to `H4a` and routed `M`/`M-T` to consume it, but never defined the injection contract by which the content reaches discovery. `git log -G "in-memory fixture-supply"` shows no later commit added a seam/parameter definition; the gap has been present since the mechanism's inception, and `H3a`'s seam list (older than `2d52ad1`) never carried a fixture-source seam.

## Solution approach

In `H4a`'s in-memory fixture-supply sentence, name the single injection point through which a harness-driven leaf's discovery obtains the fixture `.loom` content, distinct from the `FileSystem` seam (`V8b`) and from any ambient read, and state that `M`/`M-T` discovery binds to that same injection point so supply and read agree by construction. Follow the delegation pattern `H4b` already uses for its response-programming surface — fix the content the contract must define while leaving the API's exact shape to the implementer.

## Solution constraints

- Out of scope: the production filesystem-backed discovery walk deepened by `V10a` — the injection point is the harness-mode source only.
- The injection point must not reintroduce a `FileSystem` seam (`V8b`) or an ambient `src/**` read dependency into `M`, keeping the `H3a` ambient-access scan clean.
