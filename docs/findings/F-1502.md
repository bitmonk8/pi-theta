## Finding

The §4 numeric-placeholder rule in `docs/spec_topics/diagnostics/placeholder-rendering-a.md` declares that `Infinity` and `NaN` "are unreachable for these placeholders by construction (every emitting site is bounded — array length is non-negative, invoke depth is bounded by 32, etc.)" and then, in the same sentence, says "a renderer that nonetheless encounters one MUST surface it through `loom/runtime/internal-error` rather than emitting `Infinity` or `NaN` into a panic message."

These two claims pull in opposite directions on the implementation. A strict reader treats "unreachable by construction" as a formal unreachability claim, drops the `Infinity`/`NaN` branch as dead code (and may omit a conformance test for it). A defensive reader treats the MUST as load-bearing, implements the branch, routes through `loom/runtime/internal-error`, and adds a test vector. Both readings are textually defensible; the resulting implementations differ in whether a future arithmetic regression that produces a non-finite integer surfaces as a clean `internal-error` diagnostic or as raw `Infinity`/`NaN` bytes inside a panic message.

The framing is also weaker than the spec needs. The §4 emitting sites enumerated today (array length, invoke depth, etc.) are bounded by their respective rules, not by JavaScript's type system — they are "not produced by any documented emission site in loom 1.0.0", which is exactly the precondition that justifies routing any future appearance through the runtime-defect surface. That is also the framing `loom/runtime/internal-error` itself uses in `code-registry-runtime.md` and `errors-and-results/error-model.md`: a code reserved for throws "outside the closed loom 1.0.0 panic-source list", with a deliberately open-ended trigger.

## Solution Space

**Shape:** single

### Recommendation

Replace the formal-unreachability framing with an emission-site framing that preserves the defensive MUST. In §4's **Rule** paragraph of `docs/spec_topics/diagnostics/placeholder-rendering-a.md`, rewrite the closing sentence to:

> `Infinity` and `NaN` are not produced by any documented emission site in loom 1.0.0 — every such site is bounded (array length is non-negative, invoke depth is bounded by 32, etc.) — so an `Infinity` or `NaN` value reaching one of these placeholders is, by definition, a runtime defect outside the closed loom 1.0.0 panic-source list; a renderer that observes one MUST surface it through `loom/runtime/internal-error` (the runtime-defect surface defined in [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics)) rather than emitting the literal `Infinity` or `NaN` token into a panic message.

This (a) drops the formal "by construction" claim that licenses the dead-code reading, (b) names the runtime-defect surface explicitly so the MUST's destination is unambiguous, and (c) keeps the existing emitting-site enumeration as descriptive justification rather than as a formal proof of unreachability.

Edge cases the implementer must cover:

- The new MUST applies symmetrically to every category-4 placeholder (`<i>`, `<length>`, `<depth>`, `<offset>`, `<count>`, `<index>`, `<required>`, `<provided>`, `<max>`) — not only the two named in the §4 enumeration. The §4 paragraph already mentions `<required>` is scope-carved per the §4 sub-clause; that carve-out is unaffected.
- The conformance vector under `V7c-T` asserts the `loom/runtime/internal-error` routing on observing a non-finite numeric value, not on emitting one — the test injects the non-finite value at the renderer boundary, since no production loom path produces it.
- Negative-zero normalisation (`-0` → `0`) is governed by the preceding clause of the same Rule sentence and is independent of this fix; do not bundle it.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on whether the `Infinity`/`NaN` branch exists at all. One ships a renderer with no defensive routing, so a future arithmetic-bound regression leaks `"Infinity"` or `"NaN"` into a user-facing panic message and bypasses the runtime-defect channel; the other ships the branch and a corresponding test vector. The divergence is narrow but observable end-to-end and silently weakens the `loom/runtime/internal-error` safety net §4 was meant to plug into.
