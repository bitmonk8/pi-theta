## Problem

V8b's Adds and the `PIC-13` Tests bullet (mirrored in `V8b-T`) attribute the `readBytes` member and its raw-pre-decode-bytes `.code` rejection contract (`ENOENT`/`EACCES`/`EPERM`) to `PIC-13`. But `PIC-13` (`../spec_topics/pi-integration-contract/host-interfaces-services.md`) enumerates a `FileSystem` interface of `readText`/`writeText`/`exists`/`homedir`/`cwd`/`readdir`/`lstat`/`realpath` — no `readBytes` or any raw-byte read member exists anywhere in `spec_topics/**`. The section that actually motivates a byte-read seam is `lexical.md §Encoding` (invalid-UTF-8 detection and byte-offset reporting pre-normalisation), which V1a cites but V8b's Spec citations omit. Under GOV-18 arm (a), `PIC-13` binds only its enumerated members' behaviour, so a verifier reconciling V8b against `PIC-13` finds no backing for the byte-read contract.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** fa18160 — pi-loom plan: resolve "V1a presupposes a pre-decode byte source that no seam supplies" (2026-06-10, Thomas Andersen)
**History:** The `readBytes` member and its byte-read `.code`-rejection assertion entered V8b and V8b-T together in fa18160, a targeted fix that supplied V1a's pre-decode byte source. `git log -S'readBytes'` / `git log -G'rejection mapping'` localise both additions to that single commit (V8b's only earlier touch is the unrelated cb8737a over-bundling split). The fix correctly added the raw-`Uint8Array` seam but placed the new member's `.code`-rejection contract under the existing `PIC-13` token — attributing to PIC-13 a member it does not enumerate — and did not add `lexical.md` (the raw-bytes basis) to V8b's Spec citations.

## Solution approach

Add `../spec_topics/lexical.md` to V8b's Spec citations, matching V1a's grounding of the byte-read consumer. Rescope the `PIC-13` token across V8b's Adds, the V8b/V8b-T `PIC-13` Tests bullet, and V8b's Ships-when so it covers only the members `PIC-13` enumerates, and re-express the `readBytes` raw-bytes return and its Node-`.code` rejection as a GOV-18 arm (a)-permitted loom decomposition grounded in `lexical.md §Encoding`.

## Solution constraints

- The spec is read-only for this fix; all edits are plan-internal to V8b and V8b-T.
- Preserve the existing `ENOENT`/`EACCES`/`EPERM` rejection behaviour and the `readBytes` raw-`Uint8Array` return — the change is to citation/grounding, not behaviour.
- Keep the no-direct-`process.env`/`process.cwd` ambient-access ban that shares the `PIC-13` Tests bullet intact.
