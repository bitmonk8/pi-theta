## Finding

The paired binder-bypass leaves `V11c-bypass-envelope.md` and `V11c-T-bypass-envelope.md` each carry the same final Tests bullet:

> The envelope is a relaxed copy (`maxLength:500` as a model budget, not a user cap); the bypass path skips the LLM call.

The three preceding Tests bullets each name a `BNDR-N` REQ-ID (`BNDR-1`/`BNDR-2`/`BNDR-3`, mapped to `V11c` in the coverage matrix), so each is a falsifiable, code-keyed assertion. The fourth bullet names no REQ-ID, no `loom/` diagnostic code, and no code-keyed obligation area. It instead reads as prose behaviour description, and it embeds a design-rationale parenthetical ("as a model budget, not a user cap") that constrains nothing a test could fail on.

Both facts the bullet describes are real spec obligations: the `maxLength: 500` cap on `message` / `candidates[i]` lives in `binder/binder-bypass-and-envelope.md` §"Binder envelope schema (constructed dynamically from `params:`)", and the binder-call skip lives in the same file's §"Binder bypass" (anchor `#bypass-cases`, the no-params and single-string bypass cases). Both are currently un-anchored (no `BNDR-N` token) — GOV-22 residue — so the only place either obligation is named for `V11c` is this non-falsifiable bullet. An implementer who satisfies the exit gate by writing the three `BNDR-*` tests and a vacuous or descriptive fourth test parks both the `maxLength:500` budget and the "bypass issues zero provider calls" behaviour with no red-capable coverage.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V11c-bypass-envelope.md` and `docs/plan_topics/V11c-T-bypass-envelope.md`, replace the final descriptive Tests bullet with falsifiable assertions, each pointing at its existing spec obligation area (the spec is read-only for this fix — do not add new anchors to `binder-bypass-and-envelope.md`):

- An assertion that the compiled envelope schema's `message` field (and each `candidates[i]`) carries `maxLength === 500`, citing `binder/binder-bypass-and-envelope.md` §"Binder envelope schema (constructed dynamically from `params:`)".
- An assertion that the bypass path (the no-params and single-string cases) constructs no envelope schema and issues zero binder/provider calls, citing `binder/binder-bypass-and-envelope.md#bypass-cases`.

Move the "`maxLength:500` as a model budget, not a user cap" rationale out of Tests and into `V11c`'s **Adds** (it is design context, not a testable obligation). `V11c-T`'s Adds need not carry the rationale.

Edge cases the implementer must watch:
- The bypass "zero provider calls" assertion is observable only against a binder seam that records call counts; assert against the same session/binder double the other `V11c-T` bullets use, not a live provider.
- Both leaf files currently carry byte-identical Tests blocks; keep them identical after the edit so `V11c-T`'s failing tests align 1:1 with `V11c`'s exit gate.
- If the implementer judges the `maxLength:500` cap and the bypass-skip to be GOV-22 normative residue that H5a must witness, a code-keyed coverage-matrix row naming `V11c` as closing leaf may be added; the existing `BNDR-1, BNDR-2, BNDR-3 | V11c` row already maps the leaf, so this is only needed if these sub-obligations require their own traceable cell.

## Consequence

**Severity:** advisory

If left, the leaf still ships working code (the three `BNDR-*` bullets carry the load-bearing assertions), but the `maxLength:500` envelope cap and the bypass-path "no binder/provider call" behaviour have no falsifiable test of their own; a regression in either (e.g. the cap drifting, or the bypass path accidentally issuing a provider call) would not redden the exit gate. Two implementers reading the descriptive bullet will also diverge on whether to author any assertion at all.
