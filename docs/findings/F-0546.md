## Finding

`V9a`'s **Adds** field names the consumers of the pinned-constants-block constant `SHUTDOWN_AWAIT_CAP_MS` as "`V9g`, `V9i`, and `V17a` import rather than redeclare" it. `V17a`'s **Adds** field frames itself as only "a conceptual consumer" of the constant's *semantics* and enumerates the actual block-constant consumers as "`V9g`'s session-shutdown await cap and `V9i`'s subagent-disposal bound … no leaf redeclares it" — pointedly omitting itself from the consumer list.

The two statements cannot both be true: `V9a` asserts `V17a` is a runtime importer of the literal, while `V17a` asserts it consumes only the semantics and is not among the leaves that read the block constant. The corpus resolves against `V17a` being a runtime importer — `V17a`'s own **Tests** (`CNCL-1`…`CNCL-6`, forwarding, downward-only propagation, swallowing-handler suppression) exercise no site that reads the numeric cap; the cap is awaited only at the `session_shutdown` teardown (`V9g`) and the subagent disposal bound (`V9i`). `V17a` owns the *semantics* of the cap, not a runtime read of its value.

This is a flat consistency defect between two leaf pages describing the same constant's consumer set.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 622d0d3 — pi-loom plan: resolve "SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block placement" (2026-06-11, Thomas Andersen)
**History:** Commit 622d0d3 authored both halves of the contradiction in the same change: it rewrote `V9a`'s Adds to read "`V9g`, `V9i`, and `V17a` consume it without redeclaring it" while simultaneously writing `V17a`'s Adds to enumerate only `V9g` and `V9i` as block-constant consumers and frame `V17a` itself as a "conceptual consumer." A later commit 2f4ecea (2026-06-17) sharpened `V9a`'s "consume it without redeclaring" to "import rather than redeclare," making the already-present contradiction starker but not introducing it.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V9a-capability-probe.md`, edit the **Adds** field so the consumer enumeration names only `V9g` and `V9i` as the leaves that import the block constant, matching `V17a`'s variant. Replace the clause "`V9g`, `V9i`, and `V17a` import rather than redeclare" so it reads to the effect of "`V9g` and `V9i` import rather than redeclare" — and, to preserve the semantic-ownership pointer already present, state that `V17a` owns the cancellation-runtime *semantics* of the constant without importing the literal. Leave `V17a`'s Adds unchanged; it is already the correct variant. The single source-of-truth physical home (the pinned-constants block established by `V9a`) and `V18c`'s build-time literal-read assertion are unaffected.

## Consequence

**Severity:** correctness

An implementer reading `V9a` believes `V17a` imports `SHUTDOWN_AWAIT_CAP_MS` at runtime and may add a spurious (unused) import to `V17a`, since no `V17a` site reads the value; an implementer reading `V17a` treats it as a semantics-only consumer and adds no import. The two reasonable readings diverge on the same fact, and the contradiction also undermines the single-source-of-truth narrative the two pages otherwise share.
