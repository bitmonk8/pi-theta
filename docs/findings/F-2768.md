## Finding

`V7b`'s closed-set requirement (`DIAG-2`) has two directions: (1) a code asserted by a test with no matching registry row fails the gate, and (2) a registry code with no asserting test fails the gate. The `DIAG-2` Tests bullet in `V7b-code-registry.md` states direction (1) as `V7b`'s own assertion, then appends direction (2) as a parenthetical — "(and, per the `H5a` gate's `Adds.`, a registry code with no asserting test also fails the gate)" — sharing the same bullet with the primary assertion.

Both directions are in fact enforced: `H5a`'s gate Adds lists both "a registry code with no asserting test" and "an asserted code absent from the registry," and `H5a`'s *Diagnostic message anchors* Tests bullet exercises both. The defect is purely placement: the second direction of `V7b`'s own closed-set contract is the least visible part of the bullet, so a reader scanning `V7b`'s Tests for what the registry's closed-set enforcement covers can overlook that the reverse direction exists and is delegated to `H5a`. The bullet conflates `V7b`'s primary assertion with a delegated obligation that has no standalone home.

## Consequence

**Severity:** cosmetic

The reverse closed-set direction is already enforced by `H5a`, so no implementer produces a divergent or under-tested leaf; the parenthetical also names the delegation target. The cost is readability: the reverse direction of `V7b`'s closed-set contract is easy to miss when scanning its Tests, blurring which direction `V7b` asserts directly versus delegates.

The placement defect entered with that parenthetical; no later commit touched the bullet (`git log -G "registry code with no asserting test also fails" -- docs/plan_topics/V7b-code-registry.md` returns only f284534).

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V7b-code-registry.md`, lift the reverse direction out of the `DIAG-2` parenthetical into its own Tests entry that delegates to `H5a`, leaving the existing `DIAG-2` bullet to state only direction (1). Concretely, the existing bullet becomes:

> `DIAG-2`: the registry is closed — a code asserted by a test with no registry row fails the gate.

and a sibling entry states the reverse direction with explicit delegation, e.g.:

> `DIAG-2` (reverse): a registry code with no asserting test fails the gate; this direction is gated by `H5a` (see `H5a` Adds and its *Diagnostic message anchors* test).

`H5a` is read-only for this fix — it already enforces both directions; do not add or restate the gate logic there. Do not alter `V7b`'s `Deps` (`H5a` is already declared) or its Ships-when. The alternative of relocating the reverse direction into `V7b`'s Adds as a scope note is acceptable if a Tests entry is judged redundant with `H5a`'s ownership, but keeping it in Tests preserves both closed-set directions where a reader scans for the registry's enforcement contract.
