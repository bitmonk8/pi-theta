## Finding

A structural `ContextOverflowError` is explicitly surfaced via the transport-failure row, but `ContextOverflowError` carries no `provider` field, while the row's `<provider>` is pinned to `TransportError.provider`. No mapping states how `<provider>`/`<message>` are populated when the underlying failure is a context overflow.

## Solution Space

**Shape:** single

### Recommendation

Add a normative mapping for the overflow-via-transport-row case: either state that a structural overflow is re-expressed as a `TransportError` before rendering (and source `<provider>` from the binder's resolved `Model<Api>.api`, consistent with the existing binder-resolved-failure `provider` rule), or define `<provider>`/`<message>` directly from the `ContextOverflowError` plus the resolved binder model.
