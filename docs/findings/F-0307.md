## Finding

The implementation-task **Implement** step of the per-phase TDD ritual in `conventions.md` bans speculative APIs: "do not add speculative APIs (unused exports, public hooks no test in this leaf or its **declared downstream consumers** exercises)." The phrase "declared downstream consumers" occurs exactly once in the entire plan corpus, at this site, and is never defined. The leaf format defines no field, relation, or token by that name.

Three readings are available and they select different leaf sets: (a) leaves that list this leaf in their `Deps.` (the reverse-dependency edge — the only consumer relation the plan actually declares, defined in §Leaf format as "Other leaf IDs that must be complete first"); (b) leaves that name a Class-2 cross-leaf seam this leaf provides (a strictly narrower set — a Class-2 consumer must both list this leaf in `Deps.` *and* name the seam in its own `Adds.`/`Tests.`, per §Leaf format); or (c) leaves that import this module at runtime. Because this clause is a cross-cutting rule every implementation task applies during its self-review, the ambiguity is exercised against every leaf rather than one.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, in the Implementation-task **Implement** step (the first numbered sub-step of §MVP and vertical features → tests-task → implementation-task pairing), re-point the consumer set from the undefined "declared downstream consumers" to the reverse-`Deps.` relation the plan already defines in §Leaf format. Strike:

> unused exports, public hooks no test in this leaf or its declared downstream consumers exercises

and insert text equivalent to:

> unused exports, or public hooks that no test in this leaf — and no test in any leaf that lists this leaf in its `Deps.` field — exercises

This binds the consumer set to the only consumer relation the plan declares (reverse-`Deps.`), introduces no new term, and needs no edit to any other file. Reading (a) is the correct target: it is mechanically determinable and matches the surface-then-runtime intent recorded in the introducing commit's own resolution note (a `V4` lowering may export shapes only a `V6` test exercises, as long as `V6` lists `V4` in its `Deps.`).

## Consequence

**Severity:** advisory

Two implementers applying the speculative-API ban to the same leaf scope "downstream consumers" differently — one keeps an export consumed only by a reverse-`Deps.` leaf's tests, another flags it as speculative — so the ban is applied inconsistently across the corpus. The leaf still ships a working artefact either way; the cost is divergent self-review judgements about which exports to retain, not a broken or spec-divergent leaf.
