## Finding

In `audit-target-categories.md` the *Target surface categories* paragraph pins the two fixed typebox allow-lists (the `{ Unsafe }` member-access allow-list and the `{ Type }` named-import allow-list) to a specific physical location: the member-access allow-list "lives in the **same source-of-truth pinned-constants block in the extension module that already holds the four Step 0 (Capability probe) constants** (Node floor, `AbortSignal` member list, capability list, peer-dep range)", and the named-import allow-list is "**co-located with the member-access allow-list in the same pinned-constants block**".

The only contract the spec actually needs from these two sentences is *no drift*: the build-time inventory-closure audit and the runtime Step 0 (e) host-shape probe must consume one shared declaration of each allow-list, so the surface the audit authorises is exactly the surface the probe gates. The spec already states that behavioural requirement — "the build-time audit consumes it from the same single site the runtime-side Step 0 (e) probe is wired against". Everything beyond that — that the declaration sit in the *same code block*, that the block be the one *holding the four Step-0 constants*, and that it live in *the extension module* — pins implementation layout the observable contract does not constrain. An implementer who declared the allow-lists in a sibling constants module, or in their own block, while still feeding both the audit and the Step 0 (e) probe from that one declaration, satisfies the no-drift contract but technically violates the prose.

## Solution Space

**Shape:** single

### Recommendation

Restate both allow-list sentences as a behavioural no-drift contract and drop the physical-layout pins. Keep:

- the classification that each allow-list is **not** an `SDK_SURFACE_INVENTORY` entry (a separate, correct contract — leave intact);
- the single-source / no-drift obligation, phrased observably: each typebox allow-list MUST be a single shared declaration that both the build-time inventory-closure audit and the runtime Step 0 (e) host-shape probe consume, so the surface the audit authorises is exactly the surface Step 0 (e) gates;
- the cross-reference to Step 0 (e) (`capability-probe.md#entry-capability-probe`) as the runtime consumer.

Remove the prescriptive clauses "in the same pinned-constants block in the extension module that already holds the four Step 0 constants" and "co-located with the member-access allow-list in the same pinned-constants block", leaving the concrete code-block and module placement to the implementer.

Edge case for the implementer: the no-drift contract is between the *audit* and the *Step 0 (e) probe* specifically — both typebox allow-lists must trace to one declaration each. Keep the symmetry note that the named-import allow-list (`{ Type }`) and member-access allow-list (`{ Unsafe }`) are two distinct allow-lists; loosening placement must not be read as permission to merge them or to fold either into `SDK_SURFACE_INVENTORY` or into the four-peer-package iteration.

## Consequence

**Severity:** advisory

Two reasonable implementers produce identical runtime behaviour either way, so nothing breaks at runtime. The cost is over-constraint: the spec reads as mandating a particular module and code-block layout, so a conformant implementation that places the allow-lists elsewhere (while still single-sourcing them to the audit and the Step 0 (e) probe) appears non-conformant against the literal prose.
