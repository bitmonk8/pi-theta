## Finding

The `V7b` Ships-when reads "`npm test` reconciles test-asserted codes against the registry and fails on a missing/renamed code," and both `V7b` and `V7b-T` carry a `DIAG-3` Tests bullet: "codes are stable identifiers — a renamed code fails the gate (renames deferred to loom 2.0)." The gate these bullets name is the `H5a` closing gate, whose diagnostic arm is a point-in-time **parity** reconciliation: per `H5a`'s Tests and `conventions.md` *Diagnostic message anchors*, it fails when a test asserts a code absent from the registry, and when a registry code has no asserting test. It holds no prior-version baseline.

Under a parity check, only a **one-sided** rename reds — the registry row is renamed while the asserting test still cites the old code (or vice versa), which is exactly the `DIAG-2` "asserted code absent from registry" / "registry code with no asserting test" failure already covered. A **consistent** rename — the code renamed in both the registry row and every asserting test in the same edit — leaves parity fully satisfied and passes silently, indistinguishable from a `DIAG-2`-permitted remove+add. The cross-version-stability property `DIAG-3` exists to protect (a registered code name is a frozen public-contract identifier, deferred to loom 2.0 per the spec) is therefore never witnessed by this mechanical gate.

The spec itself frames `DIAG-3` correctly: `diagnostic-shape.md#diag-3` states a rename is a breaking change to the public diagnostics contract, deferred to loom 2.0 and outside the GOV-15 carve-out — i.e. a contributor-discipline / release-review obligation, not something the loom 1.0 parity gate mechanically enforces. The plan prose overclaims by promising the gate "fails on a renamed code" when it cannot detect the consistent-rename case that `DIAG-3` is about.

## Consequence

**Severity:** correctness

`V7b-T`'s `DIAG-3` Tests bullet cannot be made to fail red for the intended reason: a consistent rename keeps parity satisfied, so no red test witnesses cross-version stability, and `V7b`'s "fail red for the intended reason" Ships-when is unsatisfiable for the property `DIAG-3` names. Two implementers reading the prose diverge — one builds only the parity check (which silently passes consistent renames), another over-builds a prior-version baseline mechanism the spec does not require — and the plan leaves the actual rename-stability obligation (contributor-discipline / release-review) unstated.

## Solution Space

**Shape:** multiple

### Option A — Reframe the prose to what the parity gate delivers

**Approach.** Align the `V7b`/`V7b-T` `DIAG-3` Tests bullet and the `V7b` Ships-when with what the `H5a` parity gate mechanically delivers: it reds on an *inconsistent* (one-sided) rename, while a consistent both-sides rename stays a contributor-discipline / release-review obligation, matching the spec's loom-2.0-deferral framing in `diagnostic-shape.md#diag-3`.

**Plan edits.**
- In `docs/plan_topics/V7b-code-registry.md`, rewrite the Ships-when so it no longer promises the gate fails on a *renamed* code per se. Express it in terms the parity gate satisfies — e.g. that `npm test` reconciles test-asserted codes against the registry and fails on a code asserted without a registry row or a registry code with no asserting test (the one-sided/inconsistent-rename case), and note inline that a consistent code rename is a loom-2.0-deferred contract change carried by contributor discipline / release review rather than this gate.
- In `docs/plan_topics/V7b-code-registry.md` and `docs/plan_topics/V7b-T-code-registry.md`, rewrite the `DIAG-3` Tests bullet so the asserting test it names is one the parity gate can actually red on — an inconsistent rename — and state that the consistent-rename stability property is the loom-2.0-deferred contract obligation, not a loom 1.0 mechanical gate check.

**Spec edits.** None. The spec already frames `DIAG-3` as a loom-2.0-deferred breaking change; the fix only brings the plan prose into line with it.

**Pros.** Matches the spec's own framing; no new mechanism or scope; smallest diff; keeps the `DIAG-3` red test buildable.

**Cons.** Loom 1.0 gains no mechanical defence against an accidental consistent rename — it remains a discipline obligation.

**Risks.** Low. The reframed bullet must not collapse `DIAG-3` into a verbatim restatement of `DIAG-2`; keep the cross-version-stability obligation named even though its enforcement is review-time.

### Option B — Add a prior-version baseline to the gate

**Approach.** Give the `H5a`/`V7b` gate a prior-version registry baseline (append-only registry-diff or git-baseline comparison of the registry table) so a renamed code — even a consistent both-sides rename — is detected as a removed-old-name + added-new-name pair and reds the gate.

**Plan edits.**
- Extend `docs/plan_topics/V7b-code-registry.md`'s Adds/Tests to introduce a baseline-comparison arm (snapshot or git-baseline of the registry's code-name set) and a `DIAG-3` test that reds when a previously-registered code name disappears without a retirement record.
- Mirror the baseline-aware `DIAG-3` test in `docs/plan_topics/V7b-T-code-registry.md`.

**Spec edits.** None strictly required, though a baseline mechanism implies a registry-history/retirement convention the spec does not currently mandate.

**Pros.** Delivers genuine mechanical cross-version rename detection; the prose would then be accurate as written.

**Cons.** Adds material scope (a registry baseline / diff infrastructure) the spec does not require; `diagnostic-shape.md#diag-3` explicitly defers renames to loom 2.0, so loom 1.0 enforcement is unjustified by the spec; larger blast radius.

**Risks.** A git-baseline comparison is environment-sensitive (shallow clones, fixture corpora) and could destabilise the gate; an append-only registry-diff needs a retirement-record convention that does not yet exist.

### Recommendation

Take Option A. The spec defers code renames to loom 2.0 and treats stability as a public-contract / release-review obligation, so the plan should not promise a loom 1.0 mechanical gate the spec neither requires nor describes. Reframe `V7b`'s Ships-when and the `V7b`/`V7b-T` `DIAG-3` bullets to the inconsistent-rename case the parity gate actually reds on, and state inline that the consistent-rename stability property is the loom-2.0-deferred contract obligation carried by contributor discipline / release review. Watch the edge case that the reframed `DIAG-3` bullet must still name the cross-version-stability obligation distinctly rather than reading as a duplicate of `DIAG-2`.
