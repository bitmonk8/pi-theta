## Problem

The `pi-integration-contract/registration-steps.md` row in the *Code-keyed obligation areas* table of `coverage-matrix.md` maps three independently-verifiable obligations to the single leaf `V9b` with no sub-identifiers distinguishing them: the step-5 registry-swap atomicity MUST (`loom/runtime/registry-swap-failed`), the step-5 structural-change watcher-note pair at `#structural-changes-no-unregister`, and the step-2 `pi.getCommands()` snapshot read-only-by-convention MUST. The watcher-note obligation is itself two behaviourally-opposite predicates — empty-window suppression and same-window-rename emission — that `V9b`'s and `V9b-T`'s structural-change-watcher-note Tests bullet folds into one bullet, so neither predicate is independently citable. Because the row names a single leaf, `H5d`'s per-cell at-least-one check goes green the moment any `V9b` test cites the row, and no mechanical per-facet check exists over code-keyed rows, so a facet can ship with no asserting test while every gate passes. Concretely, the `getCommands()` read-only MUST currently has no dedicated `V9b` Tests bullet yet the row witnesses it as closed at `V9b`.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `b722ce7a623e33d22746b7952892f4baf642114b` — "pi-loom plan: resolve \"registration-steps.md has normative MUSTs but no coverage-matrix row\"" (2026-06-17)

**History:** The `registration-steps.md` → `V9b` coverage-matrix row did not exist before `b722ce7`; that commit created it already bundling all three obligation areas under `V9b` with no sub-IDs (`git log -S 'registry-swap atomicity obligation' -- docs/plan_topics/coverage-matrix.md` returns only `b722ce7`). The same commit added the structural-change-watcher-note Tests bullet to `V9b`/`V9b-T` carrying both opposite predicates in one bullet (`git show b722ce7:docs/plan_topics/V9b-registration-reload-wiring.md` is the first revision containing `watcher note`; the prior revision `433b641` contains none). The defect was present at the row's inception and has not been touched by the subsequent `V9b` edits (`894baad`, `3424d54`), which adjusted naming and the matcher-witness wording without revisiting the sub-ID granularity.

## Solution approach

Give each independently-verifiable obligation area in the `registration-steps.md` left cell of `coverage-matrix.md` a stable, citable sub-identifier, and cite the matching label inline in the corresponding `V9b` / `V9b-T` Tests bullets. Split the structural-change-watcher-note Tests bullet in `V9b-registration-reload-wiring.md` and `V9b-T-registration-reload-wiring.md` so the empty-window-suppression predicate and the same-window-rename-emission predicate are each their own citable assertion. Back the step-2 `getCommands()` read-only sub-label with an actual asserting `V9b` test, since it has none today and a row label alone leaves the obligation unwitnessed.

## Solution constraints

- Out of scope: the step-3 drain-state-gated-dispatch MUST (owned by `V9m`) and the step-2 SDK surface-inventory MUST (owned by `V18a` / `V18b`) must not be folded into `V9b`'s sub-labels.
- The spec is read-only for this fix — edit the `coverage-matrix.md` row and the `V9b` / `V9b-T` Tests bullets, not `registration-steps.md`.
