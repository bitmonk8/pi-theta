## Finding

The *Overflow token-count extraction* rule on `provider-error-mapping.md` says that when the scan over `errorMessage` yields exactly two numeric runs, the larger integer populates `tokens_used` and the smaller populates `tokens_limit`, justifying the assignment with "a context-overflow response has `tokens_used ≥ tokens_limit` by construction". The `≥` invariant is genuinely tautological for an overflow (an overflow means the input exceeded the limit), but the load-bearing step is the *prior* one — that the two numeric runs the regex captures are the input-token count and the limit-token count rather than some other pair of integers the message happens to contain. That step is an unverified empirical claim about provider phrasing, not a construction.

Concrete failure modes the current text licences silently:

- A provider rewords the overflow message so that the only two numeric runs are an unrelated pair (e.g. a per-request quota and a daily quota; a retry-after value and a token count) — `tokens_used` and `tokens_limit` then carry arbitrary integers with no signal to the consumer.
- A provider rewords the message so that the two-number pair includes the limit and a *related-but-not-the-input-count* number (e.g. "200000 token context, 50000 tokens remaining"), inverting the magnitudes and producing transposed field values.
- A provider rewords the message so that the larger number is an identifier or version token rather than a token count (request ID, account ID, API version with no separator) — the scan still yields exactly two runs and the rule still fires.

The neighbouring *Provider-owned-wording presupposition* on the same page already declares the overflow *signature* regexes a non-typed provider surface that drifts silently and routes it to editorial review; the same hazard applies to the *order* of the two captured numbers, but the extraction rule does not inherit that hedge. An implementer reading the "by construction" parenthetical takes the magnitude-based assignment as an analytic truth and ships transposed values when phrasing drifts; the only currently-mentioned drift class ("changes the numeric-run count") moves the response to the safe `null` fallback, not this one.

## Solution Space

**Shape:** single

### Recommendation

Reframe the magnitude-based assignment as an explicit assumption about provider phrasing and route its drift through the same editorial-review path the *Provider-owned-wording presupposition* already uses for the overflow-signature regexes. Concretely, in the first bullet of *Overflow token-count extraction*:

- Replace the parenthetical "a context-overflow response has `tokens_used ≥ tokens_limit` by construction" with a statement that the assignment rests on a phrasing presupposition — currently every recognised overflow message names the input-token count and the context-window limit as its only two integers, and (because the response is an overflow) the input-token count is the larger of the two — so until that presupposition drifts the larger run is `tokens_used`. Keep the equal-runs clause (both fields take that value).
- Extend the closing paragraph (after "silently moves the affected response into the `null` fallback") so the editorial-review routing covers both drift classes the rule depends on: a rewording that changes the numeric-run count (existing wording, still routes to `null`), and a rewording that preserves two runs but changes their identity or relative magnitudes (new wording, routes to transposed/arbitrary values). Both are caught by the same item-(i) editorial review of provider error fixtures the *Provider-owned-wording presupposition* already invokes — say so explicitly rather than relying on the reader to infer the second class.

Edge cases for the implementer to watch: the rule must remain deterministic on a fixed payload (a behavioural test should still pin the larger-run-to-`tokens_used` mapping on the current fixture corpus), and the equal-runs tie-break (both fields take that value) is unaffected. The change is prose-only — no behaviour change at the loom 1.0 pin, only the presupposition is named and routed.

## Consequence

**Severity:** advisory

Under current provider phrasings the larger-run assignment is almost always correct, so the wire-level extraction will usually populate the fields sensibly; the gap is that the spec presents an empirical phrasing assumption as analytic ("by construction") and routes only the run-count drift through editorial review, leaving the run-order assumption unguarded. A future provider rewording that keeps two numeric runs but reorders or repurposes them produces silently transposed or arbitrary `tokens_used` / `tokens_limit` values with no diagnostic.
