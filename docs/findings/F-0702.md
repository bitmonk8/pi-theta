## Finding

The `No concurrent user sessions in the same host process.` bullet under **loom 1.0 non-goals** in `future-considerations/model-changes-and-non-goals.md` is a single paragraph of roughly 900 words. Only the opening clause ("The loom 1.0 runtime presupposes a single active user session per extension instance…") and the closing *Recorded at:* citations carry non-goal content; everything in between is procedural analysis of how a future Pi minor that surfaces concurrent sessions would or would not be detected:

- which mutations of pinned shapes (`kind`-enum additions, new `ExtensionContext` fields) surface at the `npm run typecheck` step of the Pi version-bump procedure, and the carve-out that this only fires "insofar as loom code performs exhaustive matching that TypeScript can flag";
- the explicit statement that the build-time SDK surface-inventory assertion is a presence check on seven named capabilities and *not* an enumeration of new top-level members, so a wholly new method (the hypothetical `pi.openParallelSession`) is invisible to it and routes to editorial review;
- a point-release runtime contingency: an unanticipated concurrent-session signal arriving inside the SDK-pin range, and how pre-existing closed-set validators (notably the `session_shutdown` unknown-reason rule) would react;
- a wrap-up that re-states the same carve-out for wholly new top-level members and the same editorial-review fallback;
- a final cross-reference to the mid-loom user-session-replacement seam and an enumeration of every site loom 1.0 scopes to a single active user session, before restating "the loom 1.0 runtime defines no new disposition for it".

None of that machinery is a loom 1.0 obligation. The first three constructs are properties owned by `pi-integration-contract/version-bump-step2.md` (surface-inventory probe limits, version-bump checklist), `pi-integration-contract/capability-probe.md` (Step 0 capability probe scope), and `pi-integration-contract/unknown-reason-rule.md` (closed-set validator behaviour). The non-goal page is supposed to declare a scope boundary and forward to its anchors; instead it has absorbed a contingency essay that defends the boundary against possible future Pi releases. A reader trying to learn what loom 1.0 chose not to do has to mine the bullet for the single load-bearing sentence, and a contributor maintaining the version-bump procedure has a second, drifting copy of that procedure's coverage limits to keep in lockstep.

## Solution Space

**Shape:** single

### Recommendation

Shrink the bullet to a non-goal statement plus the existing *Recorded at:* citation list, and relocate the detection-mechanism prose to the pages that own each mechanism.

The shrunk bullet keeps the load-bearing two sentences already present:

> **No concurrent user sessions in the same host process.** The loom 1.0 runtime presupposes a single active user session per extension instance, per [`spec.md` — Session model — SM-1 single-active-session binding](../session-model-and-appendix.md#sm-1-single-active-session-binding); concurrent user sessions in the same host process are out of loom 1.0 scope and the runtime exposes no workaround. A future Pi minor that surfaces such a facet is not anticipated by loom 1.0 and would require migration; for what the runtime *does* do if an unanticipated concurrent-session signal nonetheless reaches the extension within the loom 1.0 Pi-SDK pin range, the runtime applies its pre-existing closed-set validators on whatever pinned surface site the signal happens to reach (no new disposition is defined: no refuse-to-load, no bind-to-first-session, no per-session registry keying, no host-incompatibility diagnostic), per [Pi Integration Contract — Session-binding contract](../pi-integration-contract/host-prerequisites.md#session-binding-contract) and [Pi Integration Contract — loom 1.0 seam: mid-loom user-session replacement](../pi-integration-contract/conversation-drive.md#v1-seam-mid-loom-user-session-replacement).
> *Recorded at:* [Pi Integration Contract — Session-binding contract](../pi-integration-contract/host-prerequisites.md#session-binding-contract); [Pi Integration Contract — loom 1.0 seam: mid-loom user-session replacement](../pi-integration-contract/conversation-drive.md#v1-seam-mid-loom-user-session-replacement); [Pi version bump procedure — surface-inventory presence-check](../pi-integration-contract/version-bump-step2.md); [`spec.md` — Session model — SM-1](../session-model-and-appendix.md#sm-1-single-active-session-binding).

Three relocations carry the displaced material:

1. **`pi-integration-contract/version-bump-step2.md`** — under the description of the **step 2(a) literal-read surface-inventory assertion**, add a paragraph (with a stable `<a id="surface-inventory-presence-check-scope">` anchor) that pins the *scope* of the assertion in one place: "presence check on the seven named SDK capabilities — does not enumerate new top-level members and does not walk closed-set enum membership; mutations to a pinned shape (additions to a previously-closed enum, new field on a closed object) instead surface at the `npm run typecheck` step (step 1), and only insofar as loom code performs exhaustive matching against the previously-closed shape that TypeScript can flag; a wholly new top-level member outside the pinned sites is detected only by the editorial review the bump procedure presupposes." Cite this anchor from both the concurrent-sessions and the stdio-capture non-goal bullets in place of repeating the carve-outs.
2. **`pi-integration-contract/capability-probe.md`** — add one sentence to the Step 0 scope paragraph that the probe is the five-check load-bearing list pinned there and does *not* include a concurrent-session detector (concrete example so a contributor maintaining the probe knows the question has been considered and disposed of).
3. **`pi-integration-contract/unknown-reason-rule.md`** (or whichever page owns the `session_shutdown` closed-`{"quit","reload","new","resume","fork"}` set and its `loom/host/session-shutdown-reason-unknown` emission — already the rule's authoritative home) — no change needed; the relocated bullet just stops paraphrasing this rule and lets the cross-reference do the work.

Edge cases the implementer must watch:

- The companion `No reliance on a Pi extension-host stdio-capture facet.` bullet immediately below restates the same surface-inventory presence-check carve-out. After step 1 of the relocation it should also be reduced to its non-goal statement plus a citation to the new `surface-inventory-presence-check-scope` anchor — otherwise the duplication just migrates from one bullet to its neighbour. (This overlaps the sibling "No parallel fan-out" finding's same-cluster concern about inline restatement.)
- The "Disposition for an unanticipated second session…" enumeration of every loom-1.0 single-session site (factory-captured `pi`, `ActiveInvocationRegistry`, per-session slash-dispatch serialisation) is the resolution of an earlier review finding (T22b) — it is load-bearing and must not be dropped. Either fold its content into SM-1's body as a "Disposition for an unanticipated second session" sub-paragraph, or keep the *no new disposition* sentence in the shrunk bullet (as the recommended bullet above does) and let the cited surfaces speak for what stays in scope.
- The aliased `<a id="loom-1-0-non-goals">` / `<a id="v1-non-goals">` anchors at the top of the section MUST be preserved; cross-references from `language-and-architecture.md`, `session-model-and-appendix.md`, and `plan.md` resolve through them.

## Consequence

**Severity:** advisory

A reader cannot quickly identify the loom 1.0 disposition because four paragraphs of procedural defence have buried it. Maintenance is the bigger cost: the bullet duplicates the version-bump procedure's surface-inventory presence-check semantics and the capability-probe's scope (no concurrent-session probe), so any future edit to either authoritative page must hunt this paragraph down to avoid divergence. Nothing in implementation is blocked; the harm is to scope clarity and to the single-source-of-truth discipline the non-goals page is supposed to honour.
