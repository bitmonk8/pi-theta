## Finding

PIC-52 in `pi-integration-contract/trust-boundary.md` reads: "The runtime MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy) in any loom 1.x release." This is a prohibition on a *class of implementation structures*, not on an observable outcome. Nothing a black-box test can examine — return values, diagnostic codes, channel traffic, `loom-system-note` output — distinguishes a conforming implementation from one that has interposed an extra capability filter or mediated proxy between loom code and the Pi extension host.

The paragraph's companion sentence ("silent success on denial is forbidden"; denials map to `Err(QueryError{kind:"code_tool", cause:"execution"})`) is observable, but it only narrows a single side-effect of mediation. A hypothetical capability-filter layer that adds extra denials and routes them through the same `code_tool`/`execution` envelope passes every denial-surface check the spec defines while violating the MUST NOT. `cka-18` and the closing leaf `V14d` confirm this: they close the denial-surface obligation and nothing else from PIC-52. The architectural half of the rule sits on the normative surface with no closing test, no closing review gate, and no mechanism in the corpus that distinguishes conformance from violation.

The result is a normative-grade prohibition whose effective enforcement is "a reader notices during PR review" — a posture the spec uses elsewhere (e.g. GOV-15's source-language stability) but always under an explicit "release-process goal, not a normative obligation" framing. PIC-52 does not carry that framing.

The rule shipped anyway, so the defect coincides with the introduction of the rule.

## Solution Space

**Shape:** multiple

### Option A — Narrow PIC-52 to its observable denial-surface obligation

**Approach.** Rewrite PIC-52 so the MUST is bound to what black-box tests can witness: "Host-side denials of filesystem, network, or Pi-API access reach loom code through the tool that issued the request as `Err(QueryError{kind:"code_tool", cause:"execution"})`; silent success on denial is forbidden." Demote the architectural prohibition ("interposes no privilege layer", "MUST NOT introduce … sandbox, capability filter, mediated proxy") to a non-normative orientation note immediately above the MUST, or move it into the existing `future-considerations/model-changes-and-non-goals.md` "No per-loom sandbox or capability model" bullet (which already restates the same claim and carries the *Recorded at:* trail).

**Spec edits.**
- `pi-integration-contract/trust-boundary.md`: split the existing paragraph. Keep `<a id="pic-52"></a>` on the denial-surface MUST. Place the no-privilege-layer / no-extra-channels content in a preceding plain paragraph (non-normative) or fold it into the future-considerations bullet, whichever the editor prefers. Drop the "widening or narrowing this rule is a major-version concern" clause (it pinned the now-demoted half).
- `overview-and-orientation.md`: trim the Trust-boundary NFR bullet's "interposes no additional access channels" forward-link so it points at the demoted non-normative paragraph, not at PIC-52.
- `future-considerations/model-changes-and-non-goals.md`: if the demoted text moves here, fold the prose in; otherwise leave the bullet untouched.
- `coverage-matrix.md`, `V14d-*.md`: anchors stay valid because `#pic-52` still names the denial-surface MUST.

**Pros.**
- PIC-52 becomes fully test-observable — `V14d` already closes it, no new fixture needed.
- Eliminates the dual-authority hazard between PIC-52 and the future-considerations bullet that already states the same architectural claim.
- Honest about what the project can actually enforce on a PR.

**Cons.**
- Loses the explicit normative-grade promise that the runtime will not wrap host access channels. Consumers reading only PIC-52 no longer see an architectural guarantee.
- Slightly increases the importance of the future-considerations bullet, which becomes the only home for the architectural claim.

**Risks.**
- A future PR could legitimately add a capability-filter mediator because the architectural rule is now non-normative; the reviewer's only handle is the future-considerations bullet, which is weaker by design.

### Option B — Keep the architectural MUST but mark it review-enforced, on the GOV-15 model

**Approach.** Leave PIC-52's text largely intact, but add explicit framing identifying the architectural half as a *release-process / review-enforced* obligation rather than a test-observable one — matching the precedent already established for source-language stability in `governance/source-language-stability.md` ("This is a release-process goal, not a normative obligation … enforced by review, not by a fixture suite"). The denial-surface clause retains its existing test-backed MUST status.

**Spec edits.**
- `pi-integration-contract/trust-boundary.md`: prepend a short qualifier to the architectural sentence, e.g. "The architectural posture below is enforced by review on any PR that touches `src/**` access plumbing; no mechanical gate detects an added mediation layer, and none is planned." Leave the MUST NOT and the major-version clause in place. Keep the denial-surface sentence as the test-backed half.
- Optionally cross-reference the GOV-15 precedent so the review-enforced framing is grounded.
- `coverage-matrix.md` `cka-18` row: unchanged (still closes the denial-surface half via `V14d`).
- No change to overview-and-orientation.md or future-considerations.md needed.

**Pros.**
- Preserves the architectural promise that loom 1.x will not wrap host access channels.
- Aligns PIC-52 with an existing, accepted corpus pattern (GOV-15) for review-only obligations.
- Smallest edit; no relocation, no anchor churn.

**Cons.**
- Mixes a test-observable MUST and a review-enforced MUST in the same paragraph, which readers must parse carefully to keep separated.
- Adds another review-only obligation to the project's load, with no automated reminder when the relevant `src/**` plumbing changes.

**Risks.**
- Without a mechanical gate, drift between the architectural promise and the code is detected only when a reviewer notices; the GOV-15 precedent has the same risk but is bounded by the source-language equivalence framing, which PIC-52 lacks.

### Recommendation

Option A. The corpus already restates the no-extra-mediation architectural claim in `future-considerations/model-changes-and-non-goals.md` (the "No per-loom sandbox or capability model" bullet, with its full *Recorded at:* trail back to the trust-boundary page and the version-bump editorial-review item `(ae)`), so demoting PIC-52's architectural half loses no information — it just stops calling the unenforceable half normative. The denial-surface MUST is the part that earns its `PIC-N` anchor: it is closed by `V14d` and is what `cka-18` actually exercises. Edge cases for the implementer: (i) keep `<a id="pic-52"></a>` bound to the denial-surface sentence so `V14d-T`/`V14d` and the `coverage-matrix.md` row do not need re-pointing; (ii) drop the "widening or narrowing this rule is a major-version concern" clause as part of the same edit, since it pinned the demoted half (and is itself flagged as ambiguous by the sibling finding on PIC-52); (iii) when trimming the `overview-and-orientation.md` Trust-boundary NFR bullet, preserve the forward-link to the demoted paragraph so the architectural claim remains discoverable from the entry-point page.

## Consequence

**Severity:** advisory

Two reasonable implementers reading PIC-52 will agree on the testable denial-surface obligation and diverge on the architectural prohibition: one will treat the MUST NOT as forbidding any wrapping layer; the other will treat any wrapping layer whose denials surface as `code_tool`/`execution` as compliant. Neither divergence is detectable by tests or by any review gate the spec defines, so the corpus ships a normative-grade promise that the project has no mechanism to enforce. Downstream loom authors auditing the runtime's security posture cannot rely on the MUST NOT as a hard guarantee.
