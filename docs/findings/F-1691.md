## Finding

PIC-52 in `pi-integration-contract/trust-boundary.md` reads: "The runtime MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy) in any loom 1.x release." This is a prohibition on a *class of implementation structures*, not on an observable outcome. Nothing a black-box test can examine — return values, diagnostic codes, channel traffic, `loom-system-note` output — distinguishes a conforming implementation from one that has interposed an extra capability filter or mediated proxy between loom code and the Pi extension host.

The paragraph's companion sentence ("silent success on denial is forbidden"; denials map to `Err(QueryError{kind:"code_tool", cause:"execution"})`) is observable, but it only narrows a single side-effect of mediation. A hypothetical capability-filter layer that adds extra denials and routes them through the same `code_tool`/`execution` envelope passes every denial-surface check the spec defines while violating the MUST NOT. `cka-18` and the closing leaf `V14d` confirm this: they close the denial-surface obligation and nothing else from PIC-52. The architectural half of the rule sits on the normative surface with no closing test, no closing review gate, and no mechanism in the corpus that distinguishes conformance from violation.

The result is a normative-grade prohibition whose effective enforcement is "a reader notices during PR review" — a posture the spec uses elsewhere (e.g. GOV-15's source-language stability) but always under an explicit "release-process goal, not a normative obligation" framing. PIC-52 does not carry that framing.

## Solution Space

**Shape:** single

### Recommendation

Rewrite PIC-52 in `pi-integration-contract/trust-boundary.md` so its MUST is bound to what black-box tests can witness. Split the existing paragraph:

- Keep `<a id="pic-52"></a>` on the denial-surface MUST and state it as: "Host-side denials of filesystem, network, or Pi-API access reach loom code through the tool that issued the request as `Err(QueryError{kind:"code_tool", cause:"execution"})`; silent success on denial is forbidden." Binding the anchor to this sentence keeps `V14d-T`/`V14d` and the `coverage-matrix.md` `cka-18` row valid with no re-pointing.
- Demote the architectural prohibition ("interposes no privilege layer", "MUST NOT introduce … sandbox, capability filter, mediated proxy") out of normative status. Place this no-privilege-layer / no-extra-channels content either in a preceding plain (non-normative) paragraph or fold it into the existing `future-considerations/model-changes-and-non-goals.md` "No per-loom sandbox or capability model" bullet (which already restates the same claim and carries the *Recorded at:* trail), at the editor's discretion.
- Drop the "widening or narrowing this rule is a major-version concern" clause as part of the same edit, since it pinned the now-demoted half.

In `overview-and-orientation.md`, trim the Trust-boundary NFR bullet's "interposes no additional access channels" forward-link so it points at the demoted non-normative paragraph rather than at PIC-52, preserving the forward-link so the architectural claim stays discoverable from the entry-point page.

In `future-considerations/model-changes-and-non-goals.md`, fold the demoted prose in if it moves here; otherwise leave the bullet untouched. `coverage-matrix.md` and `V14d-*.md` anchors stay valid because `#pic-52` still names the denial-surface MUST.

This makes PIC-52 fully test-observable (`V14d` already closes it, no new fixture needed) and removes the dual-authority hazard between PIC-52 and the future-considerations bullet that already states the same architectural claim.

## Consequence

**Severity:** advisory

Two reasonable implementers reading PIC-52 will agree on the testable denial-surface obligation and diverge on the architectural prohibition: one will treat the MUST NOT as forbidding any wrapping layer; the other will treat any wrapping layer whose denials surface as `code_tool`/`execution` as compliant. Neither divergence is detectable by tests or by any review gate the spec defines, so the corpus ships a normative-grade promise that the project has no mechanism to enforce. Downstream loom authors auditing the runtime's security posture cannot rely on the MUST NOT as a hard guarantee.
