## Finding

The `mode: prompt` paragraph in `overview.md` tells authors that, because the loom's final `Ok` return value is not surfaced to the user as a distinct artefact, "authors who want the user to see a final value should issue a final query whose text contains it." The "whose text contains it" clause is ambiguous between two readings that have opposite implications for authoring:

- **(a) The query *template* must contain the value** — the author embeds the value through `${value}` interpolation in the trailing `` @`…` `` template, so the rendered turn the model receives (and the user sees) literally includes the value. This is the only author-controlled, deterministic reading.
- **(b) The model's *response* must contain it** — the author phrases a final query that prompts the model to repeat the value back. This relies on model behaviour the author cannot guarantee and contradicts the rest of the paragraph, which treats the loom's value-producing logic as already complete by this point.

`overview.md` is the file an implementer or loom author hits first, and the surrounding examples in `query-forms.md` (`@`Critique this code:\n${code}`?`, `@`Rate the critique 1-5: ${critique}`?`) make `${…}` the natural way to inject a known value into a turn — but the prose never names that mechanism, so a reader who picks reading (b) gets a working-but-flaky pattern that the spec appears to endorse.

## Issue introduction

**Verdict:** present-since-inception (at the current `mode: prompt` framing)
**Introducing commits:** `53f5831` ("spec: rewrite query primitive, system prompts, and schema syntax", 2026-05-03)
**History:** `git log -S 'whose text contains it'` shows the clause first entered the corpus in `53f5831`, in a single hunk that also introduced the `@`...`` query primitive and the prompt-mode / subagent-mode split — i.e. the same commit that introduced the `${…}` interpolation mechanism reading (a) refers to. The two later touches to the line (`fecb504` splitting `spec.md` into `spec_topics/overview.md`, `31ff060` moving the tree under `docs/`) are pure relocations with no edits to the wording. The ambiguity is therefore as old as the current execution-mode framing itself, not a regression a subsequent edit introduced; it has simply never been re-examined since the rewrite that coined both the `mode: prompt` paragraph and the `${…}` mechanism that resolves it.

## Solution Space

**Shape:** single

### Recommendation

Replace the trailing clause "should issue a final query whose text contains it" with wording that names the `${…}` interpolation mechanism directly, e.g.:

> …authors who want the user to see a final value should embed it in a final `` @`…` `` query template via `${value}` interpolation, so the rendered turn the user sees contains the value directly.

Keep the surrounding sentence unchanged (the return value still exists for programmatic consumers). No glossary entry, no new normative ID, and no cross-reference is required — `${…}` interpolation is already pinned in `query/query-escapes-stringification.md` (QRY-18) and demonstrated in `query/query-forms.md`. The edit is purely a one-clause prose substitution in `overview.md`.

## Consequence

**Severity:** cosmetic

A loom author following reading (b) writes a final query that asks the model to echo a computed value, producing non-deterministic user-visible output for the case the sentence exists to address. No diagnostic, no runtime impact, and the workable pattern (reading (a)) is one example away in `query-forms.md`; the cost is wasted author iteration and surface guidance that points at the model-prompting reading rather than at `${…}`.
