## Finding

`V6e`'s `FRNT-1` Tests bullet describes the two `tool_loop.max_rounds` exhaustion paths asymmetrically. The untyped arm carries an explicit delegation marker — "on an untyped query, exhaustion produces `QueryError{tool_loop_exhausted}` (runtime behaviour owned by `V13c`/`V13d`)" — telling the reader the runtime behaviour is asserted elsewhere. The companion typed arm states a normative CIO-4 ceiling-interaction rule with no such marker: "On a typed query the CIO-4 `max_rounds`-final branch dispatches the forced respond turn as the exempt-routed terminator, so `tool_loop_exhausted` is unreachable on that path."

That typed-arm behaviour is not owned by `V6e`. The forced-respond / CIO-4-final mechanics are owned and tested downstream: `V13c` asserts the CIO-4 free-phase round exemption and the `max_rounds:0` typed forced-respond branch (`CIO-4`, `QRY-14`), and `V16a` encodes the CIO-4 arbitration (`CIO-4`: "`max_rounds:0` typed takes the final branch at start"). `V6e`'s own **Ships when** gate scopes to "`FRNT-1` bounds, the disable-at-`0` boundary, and the range code" — it never asserts the typed-arm unreachability claim, consistent with that behaviour being delegated.

The defect is the missing delegation marker on the typed arm. A `V6e` implementer reading the bullet sees one arm marked as delegated and one stated as a flat assertion, and may read the typed clause as a `V6e` obligation; an implementer of `V13c`/`V16a` skimming `V6e` gets no signal that the typed forced-respond rule is theirs to close. There is no coverage gap — `V13c`/`V16a` already assert the rule — only a wording asymmetry that misattributes ownership.

## Consequence

**Severity:** advisory

A `V6e` implementer may misread the unmarked typed-arm clause as a `V6e` test obligation rather than downstream-delegated behaviour, and a `V13c`/`V16a` implementer gets no cross-reference signal that the typed forced-respond rule closes in their leaf. No behaviour ships unverified — `V13c`/`V16a` already assert the rule — so this is a guidance asymmetry, not a coverage break.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V6e-respond-repair-tool-loop.md`, attach a delegation marker to the typed-arm clause of the `FRNT-1` Tests bullet so it mirrors the untyped arm's `(runtime behaviour owned by V13c/V13d)`. The typed forced-respond / CIO-4-final behaviour is owned by `V13c` (`CIO-4`, `QRY-14`) and `V16a` (`CIO-4`), so the marker text is `(runtime behaviour owned by V13c/V16a)` appended to the existing typed-arm sentence ("On a typed query the CIO-4 `max_rounds`-final branch dispatches the forced respond turn as the exempt-routed terminator, so `tool_loop_exhausted` is unreachable on that path").

`V13c` and `V16a` already carry the asserting tests, so no test moves and neither downstream leaf needs editing — the `FRNT-1` typed clause stays as context for the frontmatter field and only gains the ownership pointer. Leave the untyped arm, the range code bullet, the **Deps**, and the **Ships when** gate unchanged.
