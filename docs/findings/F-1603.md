## Finding

The *Generated-output-in-src limitation (acknowledged soft surface)* paragraph under [`docs/spec_topics/pi-integration-contract/audit-resolution.md` — *Audit scope*](../../docs/spec_topics/pi-integration-contract/audit-resolution.md#audit-scope) does two things. The first half pins the contributor-discipline rule — generated build outputs MUST NOT be emitted into `src/` and the inventory-closure audit has no mechanical surface for detecting them — which is a coherent normative obligation, on a par with the *Substitution limitation* and *Stale-citation limitation* framings it parallels. The second half then sketches an explicitly optional, not-required scope check: it names a specific discriminator token (`scope-audit/generated-source-in-tree`), a specific detection heuristic (a `// @generated` leading-comment scan on every walked file), and a namespace constraint (a separate top-level prefix outside the `audit/` namespace, with `scope-audit/<symptom>` named as the illustrative shape and any other non-`audit/` prefix permitted).

That second half is rationale-only content for a capability the same paragraph immediately disclaims — "PIC does not require it; widening the inventory-closure audit's family-set to subsume this check is a PIC edit deferred past loom 1.0." The paragraph thus has to do double duty: it pins the prohibition as in-scope, and in the same breath sketches part of an out-of-scope sibling check. Soft-surface paragraphs elsewhere on the page (e.g. *Substitution limitation*) confine themselves to the in-scope acknowledgement and route the deferred work to a one-line "out of scope for loom 1.0" framing; this one does not, and a careful reader has to extract the prohibition from the optional-feature design sketch around it.

The half about the optional check also has a subtler effect: by naming a concrete discriminator token and a `<class>`-namespace constraint for a check the spec disclaims requiring, it constrains future implementations of a check that PIC explicitly punts to a post-loom 1.0 edit — pinning shape for a feature the spec has not yet committed to. That is the kind of forward-binding the *future-considerations* page exists for, not the audit-scope paragraph.

## Solution Space

**Shape:** single

### Recommendation

In the *Audit scope* paragraph, trim the *Generated-output-in-src limitation (acknowledged soft surface)* sub-clause so it states only:

- the contributor-discipline prohibition (generated `.ts` files MUST NOT be emitted into `src/`);
- the acknowledgement that the inventory-closure audit walks `src/**/*.ts` blindly and so has no mechanical surface for detecting the prohibition's violation, mitigated by the same contributor obligation as the parallel *Substitution limitation* and *Stale-citation limitation* sub-bullets in `audit-failures.md`;
- a one-line "Detecting generated source in `src/` is out of scope for loom 1.0" sentence framing the deferral, on the model of the *Stale-citation limitation*'s closing sentence ("widening the audit … is a PIC edit deferred past loom 1.0").

Delete the three pieces of optional-feature detail from the paragraph: (i) the named discriminator token `scope-audit/generated-source-in-tree`, (ii) the `// @generated` leading-comment heuristic, and (iii) the namespace constraint sentence specifying that a sibling scope check "MUST use a separate top-level prefix outside the `audit/` namespace" with `scope-audit/<symptom>` as the illustrative shape. None of the three is required at loom 1.0, and pinning them in the normative body forward-binds shape on a check the spec disclaims requiring.

Keep the closure-protection sentence already in the paragraph that prevents this check (if ever built) from being subsumed under the inventory-closure audit's three-class `<class>` enumeration — but re-anchor it to the abstract "any sibling scope-violation check" rather than to the deleted `scope-audit/generated-source-in-tree` example. The closed enumeration `violation` / `infra` / `canary` pinned by *Failure-surface contract* binds the inventory-closure audit's own records, and that protection holds independently of which concrete sibling check is named.

The "Future change … MUST extend either the include glob or the exclusion list" sentence at the end of the *Audit scope* paragraph already covers the codegen / `*.gen.ts` case for in-tree convention changes; it is unaffected and should not be touched.

## Consequence

**Severity:** advisory

Implementers do not act on the deferred detail (the paragraph itself says PIC does not require it), so no in-scope behaviour diverges. The cost is that a normative audit-scope paragraph carries a design sketch for an optional sibling check, which forward-binds the token name (`scope-audit/generated-source-in-tree`), the heuristic (`// @generated` leading comment), and a `<class>`-namespace constraint for a feature the spec has not committed to and which would have to be re-litigated whenever the deferred PIC edit lands.
