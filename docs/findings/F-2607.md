## Finding

QRY-18 ("Stringification of interpolated values") drives `${expr}` rendering off a closed table keyed by the **Loom static type** of the interpolated expression: `string`, `integer`, `number`, `boolean`, `null`, `Enum variant`, `array<T>`, `Schema-typed object`, and `Result<T, E>`. A value whose static type is a *discriminated union* (`schema X = A | B | C`) has no matching row. Such a value is reachable at both interpolation surfaces the table governs: a `${pet}` in an `@`-query template whose param is union-typed, and the bare-path `${param}` form in the frontmatter `system:` field, where a top-level union-typed param is admitted (only descent *into* an un-narrowed union via `.field` is rejected, by `loom/parse/system-interp-bad-field`).

An implementer building the renderer as a flat switch on static type therefore has no defined case for the union arm. The natural answer is the `Schema-typed object` row — every discriminated-union variant is itself an object schema, so the runtime value is always a JSON object and `JSON.stringify` (compact, with recursive wire-name translation) applies — but the table never says so, and "Schema-typed object" reads as the single-`schema X { ... }` case rather than the union. The asymmetry is concrete: the parallel binder-echo object rule (BNDR section in `binder/defaulting-system-note-echo.md`) *does* explicitly fold the discriminated-union case into its object rule ("For a value whose static type is a discriminated union, the variant's declared fields are used in the variant's own source order"), so the two formatting surfaces disagree on whether the union case is spelled out.

The original finding paired this with a request to confirm that `integer`/`number` rendering delegates to deterministic, scientific-notation-free algorithms. That half is already satisfied — BNDR-4 pins canonical decimal for `integer` and BNDR-5 forbids scientific notation across all IEEE-754 magnitudes for `number` (with the BNDR-6r/6s reference renderings), and QRY-18 links both — so no defect remains there. The single live defect is the missing discriminated-union row.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 0c9ce54
**History:** The QRY-18 "Stringification of interpolated values" table was first authored in `0c9ce54` (2026-05-04, "pi-loom spec: resolve 'Stringification rule for `${expr}` interpolations is unspecified'"). Its rows at introduction — `string`, `integer`, `number`, `boolean`, `null`, `Enum variant`, `array<T>`, `Schema-typed object`, `Result<T, E>` — never included a discriminated-union row. The later `f5e89f4` size-cap split that relocated the table from `query.md` into `query/query-escapes-stringification.md` preserved the rows verbatim. The gap has existed since the table was first created; no later commit introduced or widened it.

## Solution Space

**Shape:** single

### Recommendation

Add a discriminated-union row to the QRY-18 table that delegates to the object rule: a value whose static type is a discriminated union renders as `JSON.stringify` of the value, **compact**, with wire-name translation applied recursively — identical to the `Schema-typed object` row, justified by every variant being an object schema so the runtime value is always a JSON object. Equivalently, broaden the existing `Schema-typed object` row label to "Schema-typed object or discriminated-union value" with a one-clause note that the union's runtime value is one variant object. Edge case for the implementer: the discriminator field is part of the variant's wire object and is emitted by `JSON.stringify` like any other field; no enum-style brand stripping applies here because the union's variants are object schemas, not enum variants.

## Consequence

**Severity:** advisory

Two implementers reading QRY-18 strictly find no row for a union-typed interpoland and must guess: the likely choice is the `Schema-typed object` row (compact `JSON.stringify`), which is correct, but a literal reading could instead reject the value or fall through to JavaScript `String(...)` — the very `[object Object]` corruption QRY-18 exists to prevent. The most natural reading lands on the right answer, so a working system is still reachable, but the rendered prompt text is not guaranteed identical across conforming implementations.
