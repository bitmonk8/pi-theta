## Finding

`V16a-T` is the failing-tests leaf for the `V16a` cross-ceiling arbitration seam — a stateless pure function `arbitrate(candidate) → { surfaced, masked? }`. Both leaves are explicit that the test strategy drives *synthesised ceiling-candidates* through the seam in isolation and observes its `{ surfaced, masked }` output; the live `invoke`-entry, AJV-boundary, and round-boundary enforcement sites (`V5e`/`V11f`/`V13c`/`V15b`) do not exist when `V16a-T` is picked up (its `Deps` is `V9d` only). Each `CIO-*` test bullet must therefore read as an assertion about the seam's input (a tagged candidate) and output (the `surfaced` identifier plus optional `masked` set).

`V16a-T`'s `CIO-1` and `CIO-3` are framed this way ("a synthesised candidate tagged as an AJV-boundary event resolves ceiling #4 … as the first sub-check"). `CIO-2` and `CIO-4` are not: they are written as claims about live evaluation machinery — `CIO-2` says "ceiling #1 (`invoke` depth) **is evaluated at** `invoke` entry before the callee body" and `CIO-4` says "ceiling #2 (`tool_loop.max_rounds`) **is evaluated at** the round boundary, post-slot-increment, pre-next-turn." Those phrasings describe the full invocation machinery rather than the stateless seam's candidate→`{surfaced, masked}` mapping.

The paired implementation leaf `V16a` already states the seam-framed form of the same two bullets — `CIO-2`: "a synthesised candidate tagged as an `invoke`-entry event resolves ceiling #1 (`invoke` depth) before the callee body"; `CIO-4`: "a synthesised candidate tagged as a round-boundary event resolves ceiling #2 (`tool_loop.max_rounds`) post-slot-increment, pre-next-turn; `max_rounds:0` typed takes the final branch at start." So `V16a-T`'s `CIO-2`/`CIO-4` are inconsistent with their own implementation leaf and with the seam-isolation strategy both leaves otherwise share.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V16a-T-ceiling-order-masked.md`, rewrite the `CIO-2` and `CIO-4` Tests bullets to the seam-input/output framing already used by `CIO-1`/`CIO-3` in the same leaf and by `V16a`'s own `CIO-2`/`CIO-4`:

- `CIO-2`: `a synthesised candidate tagged as an \`invoke\`-entry event resolves ceiling #1 (\`invoke\` depth) before the callee body.`
- `CIO-4`: `a synthesised candidate tagged as a round-boundary event resolves ceiling #2 (\`tool_loop.max_rounds\`) post-slot-increment, pre-next-turn; \`max_rounds:0\` typed takes the final branch at start.`

This makes every `CIO-*` bullet in `V16a-T` an assertion over the stateless seam's tagged-candidate input and `{ surfaced, masked }` output, consistent with the leaf's seam-isolation strategy and `Deps: V9d`. The implementation leaf `V16a` already carries this wording, so no edit to `V16a` is required.

## Consequence

**Severity:** advisory

A fresh agent reading `V16a-T`'s `CIO-2`/`CIO-4` could try to assert live `invoke`-entry / round-boundary evaluation behavior rather than the stateless seam's synthesised-candidate input → `{ surfaced, masked }` output. Those enforcement sites are not built when `V16a-T` is picked up (`Deps: V9d`), so such a test cannot be written as worded and drifts toward verifying machinery owned by downstream leaves. The intent stays recoverable from `V16a`'s parallel bullets and the surrounding seam-isolation prose, so this degrades clarity rather than blocking the leaf.
