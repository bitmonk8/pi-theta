## Finding

`bindings.md` §Reassignment makes a normative compatibility obligation: "The plain form and the compound forms `+=`, `-=`, `*=`, `/=`, `%=` are all legal on `let mut` bindings; the RHS must be compatible with the binding's declared or inferred type per [Type System — Type compatibility](./type-system.md#type-compatibility)." No diagnostic code is named at that site, and the cross-link target does not pick one up either.

In `type-system.md`, the check-site list at the top of §Type compatibility enumerates "the RHS of a typed `let`, a function-argument slot, an `invoke<T>` return annotation, the common type of `match` arms or ternary branches, an `array<T>` element against its sink, the `+` operator's mixed-numeric case, a frontmatter `params:` default" — reassignment is not in that list. TYPE-9 then closes the loop for exactly **three** sites named under *Structural cases* — typed-`let` RHS (`loom/parse/let-rhs-type-mismatch`), plain function-argument slot (`loom/parse/fn-arg-type-mismatch`), and ternary (routed through the array-common-type codes) — and reassignment is again absent. The `loom/parse/let-rhs-type-mismatch` row in `code-registry-parse.md` is explicitly scoped: "The RHS initialiser of a **typed binding** `let x: T = expr`…" — it does not cover `count = expr` or compound-assignment forms on a `let mut`.

The net effect: a static `T₁ ⋢ T₂` failure at a `let mut` reassignment site (plain `=` or any of the five compound forms) has a defined obligation to fail compatibility but no defined parse-time surfacing. Two reasonable implementers will diverge — one will fire a closest-match parse code (likely the misnamed `let-rhs-type-mismatch`), another will silently defer to the runtime AJV safety net, and a third may emit nothing at all.

## Solution Space

**Shape:** multiple

### Option A — Dedicated `loom/parse/reassignment-type-mismatch` code

**Approach.** Add the reassignment site to the check-site list at the top of `type-system.md` §Type compatibility, and add a fourth clause to TYPE-9 naming a new code:

> A reassignment to a `let mut` binding (plain `=` and the compound forms `+=`, `-=`, `*=`, `/=`, `%=`) reports `loom/parse/reassignment-type-mismatch` when the RHS type is statically resolvable and `T_rhs ⋢ T_binding`. For compound forms, the implicit operator-result type is the one checked.

Register a single new row in `code-registry-parse.md` (E / type; Spec: bindings.md §Reassignment + type-system.md TYPE-9; Message: `cannot reassign '<name>': type mismatch: expected <expected>, got <actual>`) and add a one-line back-pointer at the end of `bindings.md` §Reassignment ("A static mismatch at this site is `loom/parse/reassignment-type-mismatch`; see [TYPE-9](./type-system.md#type-9)").

**Spec edits.**

- `type-system.md` §Type compatibility check-site list — append "a reassignment to a `let mut` binding (plain `=` and the compound forms `+=`, `-=`, `*=`, `/=`, `%=`)".
- `type-system.md` TYPE-9 — extend from "Three of the compatibility check sites" to "Four of the compatibility check sites" and add the reassignment clause and code, with the registry pointer updated to "The `let`-, `fn`-, and reassignment-site rows are registered in [Diagnostics — code registry](./diagnostics/code-registry-parse.md#code-registry)".
- `code-registry-parse.md` — insert the new row after `loom/parse/let-rhs-type-mismatch`.
- `bindings.md` §Reassignment — append a sentence linking to the new code via TYPE-9.

**Pros.**

- Preserves the per-site code granularity TYPE-9 established (each enumerated site has its own code); a fixer reading the registry sees reassignment as a first-class site.
- Code name describes the site accurately; message template can name the binding and both types directly.
- Single code uniformly covers plain `=` and all five compound forms.

**Cons.**

- One additional registry code to maintain, register-test, and stabilise under DIAG-3.

**Risks.** None beyond the standard cost of adding a new diagnostic code.

### Option B — Widen `loom/parse/let-rhs-type-mismatch` scope to cover reassignment

**Approach.** Re-scope the existing `let-rhs-type-mismatch` code in `code-registry-parse.md` to fire on every binding-RHS compatibility failure — typed-`let` initialiser, plain reassignment, and compound-assignment forms — and update TYPE-9 and the check-site list to say so. Rename the row's Trigger column accordingly, generalise the message template, and add the reassignment site to the check-site list.

**Spec edits.**

- `code-registry-parse.md` — rewrite the `let-rhs-type-mismatch` Trigger to cover the RHS of a typed `let x: T = expr` **and** a reassignment to a `let mut` binding (plain and compound). Generalise the Message to a form that reads naturally for both kinds, or split it into two templates dispatched by site.
- `type-system.md` TYPE-9 — replace the first clause to say `loom/parse/let-rhs-type-mismatch` covers both the typed-`let` initialiser **and** every `let mut` reassignment (plain and compound).
- `type-system.md` §Type compatibility check-site list — append the reassignment site.
- `bindings.md` §Reassignment — append a sentence pointing readers at TYPE-9 / `let-rhs-type-mismatch`.

**Pros.**

- No new code to register / gate / stabilise.

**Cons.**

- Code name `let-rhs-type-mismatch` literally says "let RHS"; firing it for `count = expr` where `count` was declared on a previous line reads as a misnomer in implementer-facing diagnostics.
- Breaks the per-site granularity TYPE-9 set up (the other two sites — fn-arg, ternary — kept their own codes); the existing two-site precedent is "one code per syntactic site".
- The single Message template has to be either generic enough to fit both sites (less informative) or branched at render time (extra logic).
- A rename of the code would be a cleaner long-term fix but DIAG-3 forbids it without a loom 2.0 boundary.

**Risks.** Future readers and tooling that bucket diagnostics by code prefix or that key UX on the code name will read "let RHS" and look for a `let` form, mislocating the offending source span.

### Recommendation

Take **Option A.** Adding `loom/parse/reassignment-type-mismatch` matches the per-site code granularity TYPE-9 already established for the other enumerated sites and avoids cementing a misnomer that DIAG-3 would freeze under loom 1.x. The implementer must keep two edges in mind: (i) compound-assignment forms (`+=`, `-=`, `*=`, `/=`, `%=`) all route through the single new code (the check is against the implicit operator-result type vs the binding's type), and (ii) the runtime AJV safety net for statically-unresolvable operands continues to apply under the same posture as the other TYPE-9 sites — the parse-time check is skipped when the RHS type is past the parser's static view, exactly as the §Type compatibility *Unresolvable operands* paragraph specifies.

## Consequence

**Severity:** correctness

A static-type failure on a `let mut` reassignment has no specified surface, so independent implementations will diverge in observable behaviour — some surfacing the failure at parse time under whatever code seems closest, others letting it slip to the runtime AJV safety net, others passing it silently. The omission also breaks the registry-closure / GOV-22 discipline (every code-keyed normative obligation has a registered code) at exactly the sites `bindings.md` already promises to type-check.
