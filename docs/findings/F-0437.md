## Problem

In `conventions.md` §Per-phase TDD ritual, the "Fail red for the intended reason" definition lists "unrelated exception" as a disqualifying non-behavioural red in both clause 2 and the summary paragraph, but never defines the term. It sits next to "harness/setup throw," which already covers exceptions from the harness, setup, or fixtures, leaving two open readings: an exception from infrastructure/collaborator code around the code under test, or any exception not logically connected to the cited spec rule — including one thrown *by* the incomplete implementation under test before its primary assertion is reached. These readings yield different `<id>-T-complete` go/no-go decisions for the same suite. Because every MVP/vertical `-T` leaf defers to this one definition via its "Ships when," the ambiguity is corpus-wide.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** e7815d0 — pi-loom plan: resolve "fail red for the intended reason undefined" (2026-06-10, Thomas Andersen)
**History:** The `**"Fail red for the intended reason" (defined).**` block — including clause 2's "unrelated exception" wording and the matching summary paragraph — was added by e7815d0, which resolved a prior finding that the term was wholly undefined. Before that commit the §Per-phase TDD ritual section (added by f9c5354) used "fail red for the intended reason" without any definition. The resolution defined the term but left "unrelated exception" itself unanchored, introducing this narrower residual ambiguity.

## Solution approach

Disambiguate "unrelated exception" in clause 2 of the "Fail red for the intended reason" definition and in the summary paragraph that repeats the list, in `conventions.md` §Per-phase TDD ritual. Pin the term to exceptions originating outside the implementation under test, distinct from the adjacent "harness/setup throw." Resolve the implementation-throw edge case explicitly: state whether a suite that reds because the absent/incomplete implementation under test throws — rather than because its cited primary assertion observed a wrong value — satisfies the gate, keeping the answer consistent with clause 2's existing "reds out on its own primary assertion" requirement.

## Solution constraints

- The clarification must not exclude tests whose cited primary assertion is itself a throw assertion (`expect(...).toThrow(...)`) — an exception that *is* the asserted behaviour remains a valid intended-reason red.
- The spec is read-only for this fix; no REQ-ID is involved.
