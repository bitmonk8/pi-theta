## Finding

The `## Per-phase TDD ritual` definition of *fail red for the intended reason* in `conventions.md` lists the disqualifying non-behavioural reds in two places. Clause 2 of the definition says each test must red "on its own primary assertion because the implementation under test is absent or incomplete — **not on a missing or unconfigured fixture, a harness/setup throw, or an unrelated exception**," and the summary paragraph repeats "(compile error, missing/unconfigured fixture, harness/setup throw, unrelated exception)."

"Unrelated exception" carries no definition and sits next to "harness/setup throw," which already covers exceptions thrown by the test harness, setup, or fixtures. Two readings remain open: (a) an exception thrown by infrastructure/collaborator code *around* the code under test (a near-synonym of the adjacent "harness/setup throw"), or (b) any exception not logically connected to the spec rule the test cites — which would include an exception thrown *by* the incomplete implementation under test before the primary assertion is reached.

These readings produce different `<id>-T-complete` go/no-go decisions. Consider a tests-task whose suite reds because the not-yet-implemented production symbol throws (e.g. a stub method throws, or a missing branch raises a `TypeError`) rather than because its cited primary assertion observed a wrong value. Under reading (a) that throw originates from the implementation under test, so it is not an "unrelated exception" and the suite passes the gate; under reading (b) it is a non-behavioural red that disqualifies the suite. The definition never states which, so two reasonable tests-task authors tag the same suite differently. The gate is inherited by every MVP/vertical `-T` leaf (each defers to this definition via its "Ships when"), so the ambiguity is corpus-wide.

## Solution Space

**Shape:** single

### Recommendation

Disambiguate the term in `conventions.md` §Per-phase TDD ritual, in clause 2 of the "Fail red for the intended reason" definition and the summary paragraph that repeats the list. The edit must accomplish two things:

1. Define "unrelated exception" so it is distinct from the adjacent "harness/setup throw" — i.e. pin it to exceptions originating outside the implementation under test (infrastructure, setup, harness, fixture, or collaborator code not being exercised by this tests-task), rather than leaving it as an undefined near-synonym.
2. Resolve the implementation-throw edge case explicitly: state whether a suite that reds because the absent/incomplete implementation under test *throws* (rather than because its cited primary assertion observed a wrong value) satisfies the gate. Make this consistent with clause 2's existing "reds out on its own primary assertion" requirement — i.e. state that a red driven by an implementation-thrown exception that pre-empts the cited primary assertion does not satisfy the gate unless that throw is itself the behaviour the cited REQ-ID asserts; the tests-task author drives the red through the cited assertion.

Edge case the implementer must watch: an exception that *is* the asserted behaviour (a test whose cited REQ-ID requires a specific throw) must still count as a valid intended-reason red — the clarification must not exclude tests whose primary assertion is `expect(...).toThrow(...)`.

The fix is read-only with respect to the spec (no REQ-ID is involved) and confined to `conventions.md`; the inheriting `-T` leaves and `leaf-template.md` need no edit because they reference this definition rather than restating it.

## Consequence

**Severity:** correctness

Two reasonable tests-task authors reach different `<id>-T-complete` go/no-go decisions for the same suite: one tags complete when the suite reds via an exception thrown by the incomplete implementation, the other rejects it and rewrites the tests to red on the cited primary assertion. Because every MVP/vertical tests-task defers to this one definition, the divergence is systematic across the vertical phases rather than confined to a single leaf.
