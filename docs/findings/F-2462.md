## Finding

The `loom/runtime/internal-error` row in `code-registry-runtime.md` describes the
loom-side string for the same diagnostic in three places that do not reconcile:

- The **Message template** column gives `internal error: <error.message>`. Per
  DIAG-4 this column is the single normative source and renderers MUST emit it
  character-for-character.
- The **serialised persistent-channel content** is, per the *Serialised content
  format* rule in `diagnostic-shape.md`, the location-less `"<code>: <message>"`
  form, i.e. `loom/runtime/internal-error: internal error: <error.message>`.
- The **Trigger** prose states that "a slash-command surface emits
  `"loom /<name> aborted with internal error: <message>"` as a system note."

The first two compose cleanly (the serialised form wraps the Message-column
value with the `<code>: ` prefix), so the real defect is the third. The Trigger
prose invents a bespoke slash-note string that diverges from the systematic
panic-routing framing this code explicitly claims to share ("Routed alongside
panics"). `error-model.md` fixes that framing normatively: there is *exactly one
message string per panic*, and the slash surface wraps it as
`"loom /<name> aborted: <message>"`, where the prefix `"loom /<name> aborted: "`
*wraps* the message rather than replacing it. Applying that rule to this code
yields `loom /<name> aborted: internal error: <error.message>`. The Trigger prose
instead writes `aborted with internal error: <message>`, which (a) replaces the
systematic `aborted: ` prefix with `aborted with internal error: `, and (b) is
ambiguous about its `<message>` token — read as the Message-column value it
doubles the literal ("…aborted with internal error: internal error: …"), so it
can only mean the bare `error.message`, which silently drops the normative
`internal error: ` segment from the Message column.

A conformance test that asserts the slash-command system-note content therefore
has no single string to target: the systematic rule produces
`loom /<name> aborted: internal error: <error.message>` while this row's prose
produces `loom /<name> aborted with internal error: <error.message>`. Two
implementers reading the two governing passages emit observably different bytes
for the same delivery.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** f8f4813 — pi-loom plan: resolve "Panic routing has no defined path for unexpected interpreter exceptions" (2026-05-05, Thomas Andersen)
**History:** The `loom/runtime/internal-error` row was added in f8f4813, born with both its Message template `internal error: <error.message>` and the divergent Trigger-prose slash-note string `"loom /<name> aborted with internal error: <message>"` in the same diff. The systematic panic-message convention it contradicts (`"exactly one message string per panic"`, with the `aborted: ` prefix wrapping the message) already existed one day earlier (0c230a1, 2026-05-04) and was reinforced in error-model.md the following day (5c8d782, 2026-05-06), so the row diverged from an established convention at the moment it was written.

## Solution Space

**Shape:** single

### Recommendation

Make the `internal-error` row follow the single systematic panic surface framing
rather than restate a competing literal:

- Keep the **Message template** column as the one normative message string for
  this code: `internal error: <error.message>` (DIAG-4 source of truth).
- In the **Trigger** column, replace the inline string
  `"loom /<name> aborted with internal error: <message>"` with the systematic
  framing from `error-model.md` — the slash-command surface emits the panic
  routing form `"loom /<name> aborted: <message>"`, where `<message>` is this
  row's Message-template value — or, preferably, drop the literal entirely and
  cross-reference *Errors and Results — Runtime panics* "Panic message string
  (normative)" as the owner of the slash-surface framing, so the string is
  stated once.
- Leave the `invoke`-parent envelope sentence as-is; it is consistent with the
  panic model.

Edge cases the implementer must watch: the slash-note `content` is the
`aborted: `-framed string (per the panic routing rule), which is distinct from
the `"<code>: <message>"` persistent-channel serialised form — state which form
a DIAG-4 test asserts for this code so the two layers are not conflated; and the
`<error.message>` placeholder is the deterministically-coerced underlying-error
string (per the `placeholder-rendering-b.md#underlying-error-coercion` note at
the top of the table), not a raw `error.message` that may be `undefined`.

## Consequence

**Severity:** correctness

Byte-exact diagnostic message strings are a versioned conformance contract
(DIAG-4). Two reasonable implementers diverge on the `internal-error`
slash-command system-note content — one emits `loom /<name> aborted: internal
error: <m>`, the other `loom /<name> aborted with internal error: <m>` — so
their DIAG-4 tests cannot both pass, and operator-visible output differs across
conformant builds.
