## Finding

`ProviderResponse = { status, headers }` (`provider-error-mapping.md`, `binder-inference.md`), `CompactOptions = { … }` (`host-interfaces-core.md`), and `AgentToolResult` is `{ content, details, terminate? }` (`host-interfaces-core.md`) state the external type's member set as bald equalities, missing the "non-normative snapshot of the Pi-owned shape … re-validated on each Pi minor bump" framing that the larger interface blocks (`ExtensionContext`, the `Model<Api>` matcher fields) carry.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** 78a2f94 — resolve "Misplaced sections within the pi-integration-contract pages" (2026-06-04); f5e89f4 — docs: enforce 100KB hard / 30KB recommended size cap on spec set (2026-06-04); 44782e9 — resolve "TransportError catch-all / pi-ai provider-error surface" (2026-06-06); 15c0f17 — fix F-1284, add uniform Pi-owned-snapshot markers to the four host-interfaces-core.md member-pinning surfaces (2026-06-24); 31debf1 — fix F-1578, re-anchor execute() outcome routing on AgentToolResult (2026-06-27)
**History:** When the renderer-registration section was placed (78a2f94) it pinned `Component` to pi-tui's `dist/tui.d.ts` but never pinned the sibling `Theme`. The bald `CompactOptions` equality entered host-interfaces-core.md during the size-cap split (f5e89f4) and `ProviderResponse` entered the provider-error pages soon after (44782e9), both without a snapshot marker. The marker convention was later standardized across four host-interfaces-core.md surfaces (15c0f17, F-1284) but not extended to these inline equalities, and a still-later edit re-authored the `AgentToolResult` bald equality (31debf1, F-1578) after the convention existed without applying it — so the defect is the convention's uneven roll-out across these sites rather than any single commit.

## Solution Space

**Shape:** single

### Recommendation

Annotate each of the three inline equalities with the non-normative-snapshot marker already used on the four host-interfaces-core.md member-pinning surfaces — i.e. mark the member set as a non-normative snapshot of the Pi-owned shape at the loom 1.0 Pi-SDK pin, re-validated on each Pi minor bump. (`ProviderResponse` already states "loom redefines neither"; add the snapshot/re-validation framing to match.)

## Consequence

**Severity:** advisory

A reader cannot resolve where `Theme` is declared (a navigability gap that the sibling `Component` does not have), and the three bald equalities invite an implementer to read them as loom owning/redefining the external type's exact member set rather than as re-validated snapshots. Implementers can still build a working system — the surrounding prose already says loom redefines none of these — so the gap is one of clarity and convention consistency, not divergence.
