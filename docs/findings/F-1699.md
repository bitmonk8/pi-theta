## Finding

`host-prerequisites.md` §`#pi-sdk-pin` justifies loom's deliberate deviation from Pi's `docs/packages.md` `"*"` convention — the four `@earendil-works/*` peers are tilde-pinned at the loom 1.0 Pi-SDK pin range instead of `"*"` — on the strength of a single behavioural claim about third-party package managers:

> `pnpm` (in its default `node-linker=isolated` mode) and `yarn` do not auto-deduplicate transitive peer-dep ranges, so a tilde-range pin in loom's `peerDependencies` causes the install to fail loudly when the host has resolved any of the four `@earendil-works/*` packages outside the pinned minor line, whereas a `"*"` declaration would silently accept the skew and defer the failure to a runtime `TypeError` at the first SDK call site.

The same claim recurs in the preceding "belt-and-braces" sentence of the lock-step paragraph and is the entire load-bearing benefit cited for accepting the deviation's documented cost ("gives up Pi-prescribed alignment"). It is the *only* reason given for not collapsing the four entries back to the `"*"` Pi convention.

The claim is stated as fact about the current behaviour of two named third-party tools, yet it carries no citation, no pnpm or yarn version qualifier (pnpm has shipped multiple `node-linker` modes; yarn `1.x` "classic" and yarn `2.x+` "Berry"/PnP resolve peer dependencies via materially different algorithms), and no fixture or build-time test pinning the behaviour to a regression check. If either tool deduplicates more aggressively than the spec asserts — or did at some point in the supported range — the install-time "fail loudly on skew" property the rationale promises never materialises and the deviation buys nothing in exchange for the alignment cost it documents itself paying.

The deviation does have other defensive gates (Step 0 (d)'s runtime peer-dep probe in `capability-probe.md` and the build-time `peerDependencies` literal-read assertion gated by [Pi version bump procedure — step 2](./version-bump-intro.md#pi-version-bump-procedure)), but the spec text positions those as *consequences* of the four-anchor invariant the install-time PM gate is supposed to underwrite, not as substitutes for it. The rationale paragraph therefore cannot be deleted without leaving the deviation un-justified, and cannot stand as written without being verified.

## Solution Space

**Shape:** multiple

### Option A — Verify and cite (keep the claim load-bearing)

**Approach.** Confirm the claim against the supported pnpm and yarn versions / linker modes (at minimum: pnpm with `node-linker=isolated` and with `node-linker=hoisted`; yarn 1.x classic and yarn 2.x+/Berry with `nodeLinker: node-modules` and with PnP). Record the verification result inline.

**Spec edits.** In the *Deliberate deviation* sub-paragraph of `host-prerequisites.md`§`#pi-sdk-pin`, narrow the existing claim to the verified scope and cite each PM's resolution documentation by URL or doc-section name (e.g. pnpm's "Peer dependencies" doc, yarn's "Peer dependencies" / "Workspaces" doc). Use the form "Verified at <date> against pnpm <X> (`node-linker=<mode>`) and yarn <Y> (`<linker>`); each refuses to install when …". Echo the same scoping in the belt-and-braces sentence of the lock-step paragraph (or replace it with a back-reference to the deviation paragraph so the citation lives in one place).

**Pros.**

- Keeps the deviation's strongest benefit on the record and makes it auditable, so a future re-alignment to `"*"` is correctly weighed against a concrete loss.
- The runtime Step 0 (d) probe and the build-time literal-read assertion remain unchanged supplementary gates.

**Cons.**

- Adds maintenance load: PM behaviour can drift between minors, and the cited claim then has to be re-verified on each pnpm / yarn major bump.
- The verification has no mechanical gate; only editorial review under the version-bump procedure catches drift.

**Risks.**

- The verification might surface that one of the named PMs (or one of its modes) does in fact dedupe, narrowing the rationale to a subset of the matrix and forcing a partial re-framing of the deviation's value.

### Option B — Demote to non-load-bearing colour, rest skew-detection on the build-time and runtime gates

**Approach.** Treat the install-time PM behaviour as a non-load-bearing observation and explicitly name the build-time `peerDependencies` literal-read assertion (gated by `version-bump-intro.md` step 2) plus Step 0 (d)'s runtime peer-dep probe as the two load-bearing skew-detection gates. The PM-side behaviour, if any, becomes a third-line bonus rather than the deviation's justification.

**Spec edits.** In the *Deliberate deviation* sub-paragraph, replace the load-bearing PM-behaviour sentence with: "Install-time enforcement is package-manager-dependent and non-load-bearing; the build-time `peerDependencies` literal-read assertion ([`version-bump-intro.md` step 2](./version-bump-intro.md#pi-version-bump-procedure)) and the runtime Step 0 (d) peer-dep probe ([`capability-probe.md`](./capability-probe.md#entry-capability-probe)) are the two load-bearing gates that detect skew across the four `@earendil-works/*` peers; under package managers that do not auto-deduplicate transitive peer-dep ranges (observed under pnpm and yarn in practice; not pinned by this spec) the tilde range additionally causes the install itself to fail loudly, which is a useful bonus but not relied on." Re-state the deviation's *primary* value as the four-anchor invariant the two pinned gates need — not the PM-side fail-loudly behaviour. Delete or downgrade the parallel claim in the belt-and-braces sentence so the demoted framing is consistent across the paragraph. The same framing already exists at `spec.md`'s orientation block (`peerDependencies` declares the supported range, but install-time enforcement is package-manager-dependent (e.g. `npm install --engine-strict` is opt-in) and is non-load-bearing); harmonising with that phrasing avoids re-introducing dual authority.

**Pros.**

- Eliminates the unverified third-party-behaviour claim from the load-bearing rationale and lines the deviation's justification up with the gates the spec already mandates and tests.
- Aligns with `spec.md`'s orientation block, which already treats `peerDependencies` install-time enforcement as PM-dependent and non-load-bearing.
- No editorial drift surface — the two named gates have build-time / runtime witnesses that fail loudly if a future bump weakens them.

**Cons.**

- Concedes one of the deviation's two original benefits, which weakens (without erasing) the case against re-aligning to `"*"` in a future audit.
- Requires a tighter restatement of why the deviation is still worth its documented cost (the build-time literal-read assertion needs a literal range to assert against; `"*"` would defeat that assertion, which is the real four-anchor argument).

**Risks.**

- A future maintainer reading only the demoted paragraph might conclude the deviation has no benefit beyond what `"*"` already gives, miss the four-anchor argument, and re-align — exactly the failure mode the current paragraph was trying to prevent. The rewrite must therefore foreground the build-time literal-read assertion's dependence on a literal range as the deviation's *primary* benefit, not buried in the same sentence as the PM-side note.

### Recommendation

Take Option B. The corpus already owns two mechanical, regression-tested gates (the build-time `peerDependencies` literal-read assertion and the runtime Step 0 (d) peer-dep probe) that detect the same skew the PM claim was reaching for; resting the deviation on the gates it has mechanical witnesses for is more durable than resting it on an unverified, version-fragile behavioural claim about two third-party tools. Option A is viable but adds an editorial-review obligation that would itself have to be routed through the version-bump procedure to stay honest, and only ever produces what Option B already gives mechanically.

Edge cases the implementer must watch:

- Foreground the four-anchor argument (a literal range is needed for the build-time literal-read assertion to fire; `"*"` would defeat it) as the deviation's primary value — do not let the rewrite collapse to "install-time enforcement is non-load-bearing", which would mislead a future reader into thinking the deviation has no value and the `"*"` convention can be safely adopted.
- Sweep the lock-step paragraph's belt-and-braces sentence at the same time; leaving the old wording there while the deviation sub-paragraph demotes the claim re-creates the same dual-authority drift the corpus is trying to avoid elsewhere.
- The `H1a` *Adds* paragraph's "consumer-facing install-time skew-detection contract" phrasing in `plan_topics/H1a-scaffold-and-toolchain.md` is downstream of the rewritten rationale; update its one-line characterisation in the same commit so the plan leaf does not assert a property the spec no longer claims.

## Consequence

**Severity:** advisory

The current text reads as a verified behavioural statement about third-party tools and is the sole stated benefit for accepting a documented deviation from Pi's bundled-package convention. If the claim is wrong (or is right only under unstated PM versions / modes), a future maintainer auditing the deviation against `packages.md` finds a documented cost paid for a benefit that does not exist, and the natural fix — re-aligning the four entries to `"*"` — silently removes whatever install-time signal does happen to exist today. The build still ships because the runtime Step 0 (d) probe and the build-time literal-read assertion are independent gates; the cost is to maintainer reasoning, not to loom 1.0 runtime behaviour.
