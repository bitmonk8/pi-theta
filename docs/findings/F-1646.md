## Finding

`NOCEIL-1` in `hard-ceilings/ceiling-invariants-and-audit.md` asserts that the absence of a per-call wall-clock timeout is "enforced at parse time by rejecting any `timeout:` field at every site where a future per-call timeout could land", and the four-axis closure list's `time` row repeats the same claim under the label `NOCEIL-1`'s `timeout:`-field rejection. Both readings are load-bearing for the four-axis audit closure: they are the only mechanism that makes the time axis verifiable today rather than deferred.

The two outbound links on the bullet — `[Cancellation]` and `[Future Considerations]` — point at where per-call timeouts are *deferred*, not at where the parse-time rejection rule is defined nor at the closed set of sites it covers. The rule itself is the row `loom/parse/timeout-field-rejected` in `diagnostics/code-registry-parse.md`, whose trigger column reads "`timeout:` field declared on a query, tool call, or invoke", and the same three-element list is restated prose-only in `cancellation.md` ("declaring a `timeout:` field on a query, tool call, or invoke is `loom/parse/timeout-field-rejected`"). The three site-owner pages — `query/query-failure-and-repair.md`, `tool-calls.md`, and `invocation.md` — each carry a "loom 1.0 seam — per-call timeout" blockquote describing the open-struct seam that *would* admit the future field, but none of them states that parse-time rejection of a present-day `timeout:` is enforced at this site, and none cross-links to `NOCEIL-1` to acknowledge the audit dependency.

The result is that `NOCEIL-1`'s "every site" universal sits over a site set that has to be reconstructed by a reader: the diagnostic-row trigger string, the cancellation-page prose, and the three seam blockquotes have to be cross-walked to convince oneself the closure holds. The audit methodology paragraph immediately below leans on that closure, so the gap propagates: a future spec-editor running the four-axis check on a candidate ceiling has no canonical enumeration to verify against, and a future leaf that adds a fourth call-site for `timeout:` (e.g. a respond-repair invocation envelope, a `.warp`-level `timeout:`, a tools-set entry) has no normative checklist to keep in lock-step.

## Solution Space

**Shape:** single

### Recommendation

Bind `NOCEIL-1`'s enforcement claim to its rule owner and pin the closed site set on the page where the audit closure depends on it. Concretely, on the `NOCEIL-1` bullet in `hard-ceilings/ceiling-invariants-and-audit.md`, replace the bare "rejecting any `timeout:` field at every site where a future per-call timeout could land" tail with: "rejecting any `timeout:` field at any of the three call-expression sites a future per-call timeout could land on — a query expression, a tool-call expression, or an `invoke(...)` expression — under `loom/parse/timeout-field-rejected` per [Diagnostics — `loom/parse/timeout-field-rejected`](../diagnostics/code-registry-parse.md) (`NOCEIL-1` is a requirement on that registry row)." Update the four-axis `time` row to reference the same enumerated three-site list rather than restating "every site" in shorter form, so the audit closure cannot drift from the bullet it summarises. The three sites are exactly the three "loom 1.0 seam — per-call timeout" blockquote sites already enumerated by the corpus (`query/query-failure-and-repair.md`, `tool-calls.md`, `invocation.md`), so no new site discovery is required — the fix is to name them as a closed set in the closure-bearing paragraph.

Implementer edge cases:

- The diagnostic row's trigger string ("declared on a query, tool call, or invoke") must keep this same three-element ordering and naming after the edit, so a reader following the cross-reference lands on a row with byte-matching site vocabulary; if the spec ever extends the site set (a fourth parse site for `timeout:`), the `NOCEIL-1` bullet, the `time` row, the registry row trigger text, the cancellation-page prose, and the three seam blockquotes all need to update in the same commit. Add a one-sentence lock-step note to the `NOCEIL-1` bullet ("a future leaf that adds a fourth parse site for `timeout:` updates this enumeration, the `time` row of the closure list, the registry row, the closing paragraph in [Cancellation](../cancellation.md), and the three seam blockquotes in the same commit per GOV-30") so the lock-step obligation is local to where the closure is asserted.
- Do not import the V6a leaf's frontmatter-level wording: the diagnostic row's normative trigger is "query, tool call, or invoke" (call-expression parse sites), and V6a's test happens to exercise the parser; the spec's site enumeration must read the call-expression list, not "the frontmatter parser".
- Do not delete the `[Cancellation]` and `[Future Considerations]` links — they carry the deferral rationale and surface-extension inventory that the bullet still needs; add the registry-row cross-reference alongside them, do not replace.

## Consequence
