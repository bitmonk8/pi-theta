## Finding

QRY-6 describes a suppression hatch for the `loom/parse/empty-template` warning: an author who genuinely intends a whitespace-only prompt may "suppress the warning by writing an explicit literal escape (`\n`)". The reasoning hinges on the empty-template predicate being evaluated *before* escape rewriting, so the static body is the **two non-whitespace source characters** backslash-then-`n` rather than a single U+000A. The text spells this out ("the body is the non-whitespace two-character sequence `` \n ``").

The problem is purely notational: the bare `\n` glyph in the parenthetical `(`\n`)` is the same glyph the spec uses elsewhere for the line-feed *character* U+000A — most directly in the sibling section QRY-7 rule 4 ("split … on the line-feed `\n` (U+000A) only"), and in the escapes table in `query-escapes-stringification.md` where `\n` is one of the recognised escapes whose rendered value *is* U+000A. In rendered Markdown the two readings are indistinguishable. A reader who carries the U+000A reading into the parenthetical reaches the opposite conclusion: a single whitespace newline would *not* suppress the warning (it is whitespace-only), so the suppression hatch appears not to work. The behaviour the implementer derives is inverted relative to the intended pre-escape reading.

The surrounding prose ("two-character sequence", "post-escape it would be a single whitespace newline") does disambiguate for a reader who consumes the whole sentence, so the defect is a clarity gap at the glyph level rather than an outright contradiction. The fix is to render the source-sequence reading unambiguously at the point of first use.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** 82be125 (2026-05-04, "spec: adopt P1 typed-query two-phase loop (Option A)")
**History:** The corpus is git-tracked. `git log -S 'suppress the warning by writing an explicit literal escape'` and `git log -S 'loom/parse/empty-template'` trace the empty-template warning and its suppression hatch to 82be125, the commit that first authored the two-layer degenerate-template defence. That commit already wrote the prose hatch as `(`\n`)` while the diagnostics-table hint in the same commit wrote `\\n`, so the glyph overload existed at the feature's inception. A later commit (2637aec, 2026-06-20, "fix F-0755 — empty-template predicate pinned to pre-escape source segments") expanded the explanation to add "the non-whitespace two-character sequence `` \n ``" but retained the ambiguous bare `\n` in the parenthetical, so the defect was never the product of a single regression — it has been carried forward since 82be125.

## Solution Space

**Shape:** single

### Recommendation

Render the source-sequence reading unambiguously at first use in the suppression-hatch clause. Replace the bare parenthetical `(`\n`)` with an escaped-backslash form plus an explicit disambiguator, e.g. "by writing an explicit two-character literal-escape source sequence `` `\\n` `` (backslash followed by `n`, **not** the U+000A character)". Keep the existing "non-whitespace two-character sequence" follow-on clause; the goal is only to make the first occurrence self-disambiguating so a skimming reader cannot import the QRY-7 / escapes-table `\n` = U+000A reading. No behavioural change: the pre-escape predicate semantics pinned by F-0755 stay exactly as written.

## Consequence

**Severity:** advisory

An implementer who reads the parenthetical `\n` as U+000A concludes the suppression hatch does not work and may either omit the pre-escape suppression behaviour or calibrate the `empty-template` warning/suppression test against the wrong input. The full sentence disambiguates, so a careful reader recovers the intended behaviour; the risk is a skimming reader implementing or testing the inverted reading.
