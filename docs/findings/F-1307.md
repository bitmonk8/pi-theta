## Problem

The settings-file read path can raise an exception class that no `loom/load/settings-*` code covers (the file existed at first-stat, then disappeared or EIO'd before parse). The *Transient toasts (auxiliary)* paragraph in `diagnostic-shape.md` routes that uncovered exception to a toast but does not state what the in-progress discovery scan does in response — abort, continue against built-in defaults, or continue against the last successfully merged settings snapshot. The choice is observable: it determines whether discovery completes with a partial loom registry, an empty registry, or not at all.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `ac8c3af` — "resolve 'LoadExtensionsResult.errors is the wrong delivery channel for loom diagnostics'" (2026-05-04)
**History:** `ac8c3af` introduced the *Transient toasts (auxiliary)* paragraph carrying the three trigger sites and only the surfacing rule; the settings-IO scan-continuation rule was absent from the first emission and no later edit (`32d2b76`, `f5e89f4`) added it.

## Solution approach

Decide and state the in-progress-scan continuation rule for an uncovered settings-IO exception, anchored in the *Failure modes* section of "Settings file reads" in `package-and-settings.md` (adjacent to the `#watcher-time-reload-failures` paragraph that owns the related reload-failure modes). Add a back-reference from the *Transient toasts (auxiliary)* paragraph in `diagnostic-shape.md` so the rule is reachable from the toast surface.
