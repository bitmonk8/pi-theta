## Finding

The `params` row of the frontmatter Field contract (`frontmatter-fields-a.md`) pins three input shapes — absent, `params: {}`, and the rejected `params: null` (`loom/load/params-null`) — but says nothing about a `params:` mapping that declares the same parameter name twice (e.g. `params: { language: string, language: integer }`). Because `params:` is a YAML block mapping, the outcome is currently whatever the (unpinned) YAML parser does with a duplicate key: silent last-wins, first-wins, or a parse error, depending on the parser and the YAML version. No loom-side diagnostic or precedence rule is stated.

This is the only duplicate-name site in the spec left unspecified. The body-level loom grammar already rejects duplicates uniformly — `loom/parse/duplicate-enum-variant-name`, `loom/parse/duplicate-enum-value`, and `loom/parse/duplicate-discriminator-value` are all load/parse errors (see `schemas.md` and the parse registry). The `params:` mapping has no equivalent, so the loom layer silently inherits the YAML parser's behaviour instead of asserting its own.

A silent last-wins resolution is a correctness hazard, not a cosmetic one: the surviving declaration determines the binder parameter table, the AJV input schema shape, and the positional arity used by `invoke(...)` and `.loom`-callable argument binding. Two conforming implementations on different YAML parsers (or YAML 1.1 vs 1.2 semantics) can therefore disagree on what a duplicate-key loom validates against and how many positional arguments it accepts.

**History:** fb54586 introduced the frontmatter Field-contract table and its `params` row, enumerating only the absent / `params: {}` (and later `params: null`) cases; the duplicate-key case was never addressed at that authoring nor in any later edit (subsequent `params`-row commits touched the type-side grammar and no-params-overflow text only). The prose form the table replaced likewise covered only the absent-vs-`{}` equivalence, so the omission has been present since the `params` disposition was first written.

## Solution Space

**Shape:** single

### Recommendation

Add a duplicate-parameter-name disposition to the `params` row: a `params:` mapping that declares the same loom-side parameter name more than once is the load-time error `loom/load/duplicate-param-name`, and the loom is not registered. Register the code in `code-registry-load.md` (Sev `E`, Phase `load`) alongside the existing `params-null` entry, with a message naming the offending parameter (e.g. `duplicate parameter name '<name>' in 'params:'`).

State that this check is performed by the loom loader against the parsed frontmatter mapping and does not delegate to the underlying YAML parser's duplicate-key behaviour — so the disposition is identical regardless of which YAML parser/version the runtime uses. This keeps `params:` consistent with the body-level reject-duplicates posture (`loom/parse/duplicate-enum-variant-name` et al.) rather than admitting a silent last-wins. Detection precedes any wire-name / `as`-rename processing, since a duplicate loom-side name is ill-formed before renames are considered.

## Consequence

**Severity:** correctness

A `params:` mapping with a repeated parameter name has no defined disposition, so the resolved parameter table, the lowered AJV input schema, and the positional arity all depend on the host YAML parser's duplicate-key behaviour. Two reasonable implementers — one whose parser errors, one whose parser keeps last-wins — diverge on whether the loom loads at all and on what it validates against.
