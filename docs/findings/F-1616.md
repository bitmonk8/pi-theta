## Finding

IMP-1 in `docs/spec_topics/imports.md` opens with "A `Resolver` MUST signal an unresolvable spec by throwing; loom 1.0's load pipeline treats a throw from `resolve` as a resolution failure, emits the load-time diagnostic `loom/load/unresolvable-warp-path` … and does not register that file." The same page declares the `Resolver` seam itself non-observable — the immediately preceding paragraph and the boxed "loom 1.0 seam — `Resolver` interface" note both end with "the single-seam factoring is a maintainability convention, not an observable conformance point" — and the `Resolver` type signature is `resolve(spec: string, fromFile: string): string`, i.e. it has no other channel for signalling failure. `throw` is therefore the only return-shape that can mean "unresolvable" against that signature; pinning that mechanism to a MUST is what the page elsewhere explicitly forbids.

The observable contract is fully determined by the rest of the paragraph: the importing file emits `loom/load/unresolvable-warp-path` against the closed set of unresolvable cases (non-relative spec, no byte-exact final-segment entry, unreadable entry) and is not registered. That set is what `V15c-T-imports.md` and `V15c-imports.md` test against (`IMP-1`: "the `Resolver` signals an unresolvable `.warp` path by throwing → `loom/load/unresolvable-warp-path`, and the file is not registered"). A conformant non-`Resolver`-shaped import pipeline that produces the same diagnostic + non-registration with the same trigger set satisfies every observer; the `throw` discipline is purely an internal-mechanism choice for the one shipped `Resolver` implementation.

The defect is on a load-bearing rule (every `.warp` import flows through this), but its reach is narrow — it adds one extra constraint on internal seam shape that the same page already disclaims, rather than misstating any observable. The IMP-1 sibling clause that is wanted as MUST (the diagnostic + non-registration outcome and the closed unresolvable-case set) is already MUST-shaped; only the leading mechanism clause needs to move into the existing non-normative implementation note.

## Solution Space

**Shape:** single

### Recommendation

Rewrite IMP-1's first sentence so the MUST is on the observable, and demote the `throw` mechanism to a non-normative implementation note alongside the existing one on the same paragraph. Concretely:

- Replace "A `Resolver` MUST signal an unresolvable spec by throwing; loom 1.0's load pipeline treats a throw from `resolve` as a resolution failure, emits the load-time diagnostic `loom/load/unresolvable-warp-path` … against the importing file, and does not register that file." with a MUST whose subject is the load pipeline and whose obligation is the observable, e.g.: "When the `Resolver` cannot resolve a spec, loom 1.0's load pipeline MUST emit the load-time diagnostic `loom/load/unresolvable-warp-path` (see [Diagnostics — `loom/load/*` registry](./diagnostics/code-registry-load.md)) against the importing file and MUST NOT register that file."
- Preserve verbatim the rest of the paragraph that closes the unresolvable-case set (non-relative specs, no byte-exact final-segment entry, unreadable entry, NFC-vs-NFD non-folding, `<path>`-placeholder rendering) — that is the observable rule and stays MUST-bound.
- Append to (or merge into) the existing trailing "*Non-normative implementation note.*" on the same paragraph: "The shipped relative-path `Resolver` signals an unresolvable spec by throwing from `resolve(spec, fromFile)`, which the load pipeline catches and routes through the rule above; the `throw`-as-signal shape is an internal mechanism choice for this implementation, not a conformance point — alternative `Resolver` shapes (e.g. a `Result`-returning seam) that produce the same diagnostic and non-registration are conformant."

Preserve the `<a id="imp-1"></a>` anchor at its current position so the `loom/load/unresolvable-warp-path` row in `docs/spec_topics/diagnostics/code-registry-load.md` and the plan-leaf references continue to resolve. After the spec edit, update the `IMP-1` Tests bullet in both `V15c-imports.md` and `V15c-T-imports.md` to restate the test obligation as the observable ("an unresolvable `.warp` import emits `loom/load/unresolvable-warp-path` against the importing file and the importing file is not registered") so the plan-leaf acceptance criterion mirrors the spec MUST rather than the demoted mechanism; the existing companion success-path bullet stays as-is. Edge case: do not weaken the closed unresolvable-case set — the byte-for-byte / NFC-vs-NFD / unreadable rules are observable and stay MUST-bound; only the leading mechanism clause demotes.

## Consequence

**Severity:** advisory

A reader treating IMP-1 literally locks the `Resolver` to a `throw`-shaped failure channel forever, foreclosing equivalent compliant shapes (a `Result`-returning `Resolver`, an out-parameter, a sentinel return) the same page's non-observable disclaimer otherwise permits. No observer is affected today because the one shipped implementation already throws, but the rule contradicts the page's stated observable-only stance on the `Resolver` seam and would force a spec-change rather than a refactor if a future implementation wanted a different internal failure shape.
