## Finding

`V9i` carries two behaviourally distinct subagent concerns in one leaf. The first is the basic single-invocation lifecycle (`PIC-9`): a private `AgentSession`, the spawn sequence, the pre-spawn binder-model guard, and the mandatory idempotent `dispose()` in `finally`. The second is the `PIC-22` parallel-spawn step: for N≥3 subagent-mode `.loom` callables emitted as parallel tool calls in one assistant turn, dispatch `createAgentSession` for all N before any returns via `Promise.all` over the per-call spawns.

The two concerns exercise different test infrastructure. The `PIC-9` criteria assert dispose-in-`finally` idempotency and isolation against an ordinary session double. The `PIC-22` criterion requires a fake `AgentSession` whose `sendUserMessage` blocks until explicitly released, plus a timing assertion that all N `createAgentSession` calls completed and each session entered `sendUserMessage` before any blocked call is released. They share the same `createAgentSession` spawn code site, which is the reason they were co-located, but they are independent acceptance obligations on a single `Ships when`.

This is a leaf-granularity observation, not a defect in the leaf's content: both obligations are present and well-specified, and an implementer can produce a working leaf as written.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 7a3b907 — pi-loom plan: resolve "subagent parallel-initiation MUST has no closing leaf" (2026-06-10, Thomas Andersen)
**History:** `V9i` was created as the subagent lifecycle leaf in the initial plan build (c6a664e, 2026-06-10). About an hour later, 7a3b907 added the `PIC-22` parallel-spawn step into `V9i` (and `V9i-T`) to give the previously-unclosed subagent parallel-initiation MUST a closing leaf, which created the current bundle. A later de-bundling pass (6d7661d, 2026-06-17, resolving "V9i bundles three separable concerns into one leaf") split the swallowing-handler suppression concern out of `V9i` but left the `PIC-9` lifecycle and `PIC-22` parallel-spawn obligations co-located.

## Solution Space

**Shape:** multiple

### Option A — Keep PIC-9 and PIC-22 in V9i

**Approach.** Leave `V9i`/`V9i-T` as-is; the parallel-spawn step remains a documented extension of the same spawn sequence.

**Plan edits.** None.

**Spec edits.** None.

**Pros.** Both obligations live where the `createAgentSession` code they exercise lives; no new leaf, no new dependency edge, no coverage-matrix churn.

**Cons.** One `Ships when` aggregates two distinct test surfaces (ordinary session double vs. blocking-`sendUserMessage` fake with timing assertions).

**Risks.** None of substance.

### Option B — Extract a parallel-spawn leaf

**Approach.** Move the `PIC-22` parallel-spawn step and its `PIC-22` test criterion out of `V9i`/`V9i-T` into a new leaf (`<new>` + its `-T` twin) that depends on `V9i`, leaving `V9i` to own only `PIC-9` and the isolation/return/model-guard criteria.

**Plan edits.** Strike the `PIC-22` clause from `V9i`'s Adds and the `PIC-22` bullet from both `V9i` and `V9i-T` Tests; add the `<new>` leaf and `<new>-T` files with the `PIC-22` Adds clause, the `PIC-22` test bullet, and `Deps` including `V9i`; split the `PIC-9, PIC-22 | V9i` coverage-matrix row so `PIC-22` maps to `<new>`; add the `<new>` leaf to `docs/plan.md` §V9 listing.

**Spec edits.** None.

**Pros.** Each leaf owns one behavioural concern with its own acceptance gate.

**Cons.** The new leaf re-touches the same `createAgentSession` spawn code `V9i` establishes, creating an artificial split across one code site plus a new dependency edge and coverage-matrix churn.

**Risks.** Sequencing: the parallel-spawn leaf cannot be picked up until `V9i`'s spawn sequence lands, so the split adds ordering without isolating implementation effort.

### Recommendation

Adopt Option A — keep `PIC-9` and `PIC-22` in `V9i`. The parallel-spawn step is the single-invocation spawn sequence dispatched over N rather than an independent unit of work, so splitting it out fragments one `createAgentSession` code site across two leaves for marginal organisational benefit. Edge case for the implementer: if the related orphan-session-cleanup finding is resolved by expanding `PIC-22` (e.g. `Promise.allSettled` + dispose-every-fulfilled), that obligation lands in `V9i` under this option rather than in a separate leaf.

## Consequence

**Severity:** cosmetic

Nothing functional breaks if shipped as-is: both `PIC-9` and `PIC-22` are fully specified obligations on `V9i`, and either organisation produces the same implemented behaviour. The only cost of bundling is a marginally larger single leaf whose green `Ships when` aggregates two unrelated test surfaces, so a future failure in one does not localise to a single obligation.
