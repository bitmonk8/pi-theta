## Problem

The child-invoke arm of the **Surfacing** bullet (under `id="surfacing"` in `docs/spec_topics/cancellation.md`) keys its `kind` choice on "when the parent's own signal fired first", which admits two distinct readings. The spawn-time reading restricts bare `kind: "cancelled"` to the synchronous no-session short-circuit; the causal-ordering reading makes it apply to *every* parent-originated abort, because the child's derived signal can never fire before its source. The two readings produce different observable `QueryError.kind` values for the same in-flight, parent-initiated `invoke` cancellation, so two conforming implementations diverge and the `slash-invocation.md` per-`kind` system-note table emits different notes for the same event.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `fae85d3` ("Tighten spec for implementation-plan readiness", 2026-05-04) and `8395c03` ("pi-loom spec: resolve 'Cancellation source ctx.signal is undefined when a slash-handler starts'", 2026-05-04)
**History:** The exact surfacing-bullet text — "directly as `kind: "cancelled"` when the parent's own signal fired first" — was present in the cancellation section's first appearance in `spec.md` at `fae85d3`. At that point the *invoke(...) entry* rule was the simpler "child loom inherits a derived signal from its caller — the child aborts whenever the caller does", under which reading (A) was the only available interpretation (no spawn-time short-circuit was described, and "fired first" naturally meant "before the child started"). Commit `8395c03` replaced that paragraph with the current derived-controller wording and added the "If the parent's signal is already aborted at child-spawn time, the child surfaces `Err(QueryError { kind: "cancelled", ... })` synchronously without spawning a session" rule. That edit activated reading (B): under derived-controller semantics, "parent's signal fired first" is a vacuous tautology, and the spawn-time short-circuit becomes a more precise (but textually unconnected) home for the bare-`cancelled` surface. The surfacing bullet was carried through `fecb504` (split into `spec_topics/`) and `8455d5e` (wire-kind rename `invoke_callee_error` → `invoke_callee`) unchanged; neither touched the ambiguous phrasing. The defect is therefore the interaction between the original surfacing wording and the later forwarding-rule rewrite — not a single bad edit.

## Solution approach

Rewrite the child-invoke arm of the **Surfacing** bullet (under `id="surfacing"`) so each `kind` arm is keyed to a concrete observation trigger already named in the file instead of the ambiguous "parent's own signal fired first" clause: bind `invoke_callee` / `inner: cancelled` to the child's checkpoint observing the abort (covering both child-originated aborts and parent-propagated aborts the child observes while in flight), and bind bare `cancelled` to the parent's signal being already aborted at child-spawn time with no session spawned. Forward-cite the spawn-time short-circuit in the `invoke(...) entry` forwarding rule so the two paragraphs cannot drift apart again.
