## Finding

The **Surfacing** bullet for child-invoke cancellation in `docs/spec_topics/cancellation.md` reads:

> A child invoke whose signal aborts surfaces to the parent as `Err(QueryError { kind: "invoke_callee", inner: { kind: "cancelled", ... } })` when the abort originated inside the child, or directly as `kind: "cancelled"` when the parent's own signal fired first.

The clause "when the parent's own signal fired first" admits two distinct readings, which produce different observable `kind` values for the same in-flight cancellation:

- (A) **Spawn-time reading.** The parent's pre-`invoke` checkpoint (or its already-aborted state at child-spawn time) observed the abort before the child session was spawned. This matches the *Forwarding into `loomAbort` — invoke(...) entry* paragraph earlier in the same file, which says "If the parent's signal is already aborted at child-spawn time, the child surfaces `Err(QueryError { kind: "cancelled", ... })` synchronously without spawning a session." Under this reading the bullet identifies a meaningful sub-case: synchronous, no session, bare `cancelled`.
- (B) **Causal-ordering reading.** The parent's signal preceded the child's derived signal in time. Under the *invoke(...) entry* paragraph's derived-controller design — "the child constructs its own `loomAbort` as a derived controller that aborts when the parent's signal aborts but not vice versa" — *every* parent-originated abort satisfies this condition tautologically: the derived signal cannot fire before its source. Under this reading bare `cancelled` would apply to all parent-originated cancellations including in-flight ones, contradicting the "abort originated inside the child" arm of the same bullet (a parent-driven derived-signal abort *also* fires the child's in-flight checkpoints).

Two reasonable implementers reading this in isolation will diverge on the in-flight case: one will wrap a parent-initiated in-flight cancellation as `InvokeCalleeError { inner: cancelled }`, the other will surface it as bare `kind: "cancelled"`. Downstream `match` arms and the `slash-invocation.md` per-`kind` system-note table behave differently for the two shapes.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `fae85d3` ("Tighten spec for implementation-plan readiness", 2026-05-04) and `8395c03` ("pi-loom spec: resolve 'Cancellation source ctx.signal is undefined when a slash-handler starts'", 2026-05-04)
**History:** The exact surfacing-bullet text — "directly as `kind: "cancelled"` when the parent's own signal fired first" — was present in the cancellation section's first appearance in `spec.md` at `fae85d3`. At that point the *invoke(...) entry* rule was the simpler "child loom inherits a derived signal from its caller — the child aborts whenever the caller does", under which reading (A) was the only available interpretation (no spawn-time short-circuit was described, and "fired first" naturally meant "before the child started"). Commit `8395c03` replaced that paragraph with the current derived-controller wording and added the "If the parent's signal is already aborted at child-spawn time, the child surfaces `Err(QueryError { kind: "cancelled", ... })` synchronously without spawning a session" rule. That edit activated reading (B): under derived-controller semantics, "parent's signal fired first" is a vacuous tautology, and the spawn-time short-circuit becomes a more precise (but textually unconnected) home for the bare-`cancelled` surface. The surfacing bullet was carried through `fecb504` (split into `spec_topics/`) and `8455d5e` (wire-kind rename `invoke_callee_error` → `invoke_callee`) unchanged; neither touched the ambiguous phrasing. The defect is therefore the interaction between the original surfacing wording and the later forwarding-rule rewrite — not a single bad edit.

## Solution Space

**Shape:** single

### Recommendation

Rewrite the child-invoke surfacing bullet to anchor each `kind` arm to a concrete trigger already named elsewhere in the file, eliminating the ambiguous "fired first" phrasing:

> A child invoke whose signal aborts surfaces to the parent as `Err(QueryError { kind: "invoke_callee", inner: { kind: "cancelled", ... } })` when the child's checkpoint observed the abort (whether the abort originated inside the child or propagated down from the parent's derived signal while the child was in flight), or directly as `Err(QueryError { kind: "cancelled", ... })` when the parent's signal was already aborted at child-spawn time and no session was spawned (per *Forwarding into `loomAbort` — `invoke(...)` entry* above).

The wording explicitly partitions the cancellation space by *which side observed the abort first* (child checkpoint vs. parent's pre-spawn check), not by causal ordering of the signals. It cites the existing spawn-time short-circuit rule by reference so the two paragraphs cannot drift again. Edge case the implementer must watch: the in-flight arm covers both child-originated aborts (a tool inside the child threw) and parent-propagated-then-child-observed aborts — both surface as `InvokeCalleeError { inner: cancelled }`, because in both cases the child's session ran far enough to settle through its own checkpoint.

## Consequence

**Severity:** correctness

Two conforming implementations will produce different `QueryError.kind` values for in-flight parent-initiated `invoke` cancellation. Loom code using `match QueryError { kind: "invoke_callee", ... } | { kind: "cancelled", ... }` arms will branch differently across implementations, and the slash-invocation per-`kind` system-note table will emit different notes for the same user-visible event.
