## Finding

The statement-form control-flow block rule in `grammar.md` (the `if` / `for` / `while` paragraph after the `BlockExpr` grammar) states that a present tail `Expr`'s value is discarded "except that a tail `Expr` ending in the postfix `?` operator … still triggers `?` early-return on failure." The qualifier "ending in the postfix `?`" is imprecise: it does not say whether it means the tail's **outermost (top-level) operation** is `?` (i.e. the tail is exactly `innerExpr?`), the tail's **last lexical token** is `?` (so `a + b?` qualifies but `b? + a` does not), or that a `?` **appears anywhere** in the tail.

This matters because the carve-out describes a control-flow side effect, not a value. Per `expressions.md` the postfix `?` is a tightest-binding (precedence level 1), left-associative operator whose semantics — `on Err, early-return the Err from the enclosing function/loom` (ERR-18, `#question-operator`) — fire wherever the `?` is **evaluated**, independent of its position in the surrounding expression. In `bar(foo()?)` the inner `foo()?` early-returns on `Err` before `bar` is ever called; the discard of the block's tail value applies only to a value that was successfully produced, never to the `?` early-return.

The "ending in" phrasing invites an implementer to special-case the carve-out to the outermost-`?` tail and treat a discarded statement-block tail whose outermost operation is **not** `?` (e.g. `bar(foo()?)`, or `foo()? && cond`) as fully value-discarded — silently dropping the nested `?`'s early-return. That contradicts the global `?` semantics and produces divergent runtime control flow between conforming implementations. The fix is to drop "ending in" and tie the carve-out to the existing position-independent `?` semantics: the tail `Expr` is fully evaluated, only its successfully-produced value is discarded, and any postfix `?` evaluated within it still early-returns.

## Solution Space

**Shape:** single

### Recommendation

Rephrase the statement-form control-flow block sentence so the carve-out is stated in terms of evaluation, not a positional "ending in" predicate:

> A present tail `Expr` is evaluated and its successfully-produced value is discarded (the block produces no value); evaluation itself is unaffected. In particular, any postfix `?` operator evaluated within the tail `Expr` still early-returns its `Err` from the enclosing function or loom on failure — per [Expression Sublanguage — `?` operand-type precondition](./expressions.md#err-18) and [Return Statement](./return.md) — regardless of whether `?` is the tail's outermost operation.

This removes the ambiguous "ending in the postfix `?`" qualifier and binds the rule to the position-independent `?` early-return semantics already defined in `expressions.md`. Do not restrict the carve-out to the outermost-`?` (`innerExpr?`) case: that would contradict `?`'s defined behaviour, under which a nested `?` such as `foo()?` in `bar(foo()?)` early-returns when evaluated. Implementer note: the discard applies only on the success path (the unwrapped/whole value is dropped); on the `Err` path control never reaches the discard because `?` has already returned.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on whether a postfix `?` that is not the outermost operation of a discarded statement-block tail (e.g. `bar(foo()?)` as the tail of a `for`/`while`/`if` body) still early-returns. One applies the general position-independent `?` semantics; the other reads "ending in the postfix `?`" narrowly and suppresses the early-return, producing different terminal outcomes for the same loom.
