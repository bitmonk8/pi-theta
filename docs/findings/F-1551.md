## Finding

`schema-subset.md` § Depth Enforcement, paragraph following the five-site enforcement list, reads: "Implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits and produces the failure." The clause-final phrase carries the observable contract (short-circuit at the first too-deep node, then surface it). The clause-initial phrase prescribes a specific traversal strategy — recursive descent driven by an explicit depth counter — on a surface the spec elsewhere treats as non-observable: the only externally-visible artefacts are (a) ordering relative to AJV (already pinned, "runs **before** AJV at each site"), (b) the counting algorithm (already pinned in § "Counting algorithm" as scalar/empty = 1, non-empty = 1 + max child, `anyOf` arms not levels), and (c) the failure payload (`schema_keyword:"maxDepth"`, path-to-first-too-deep-node, message; already pinned in § "Error shape").

An equally-conforming iterative walk — e.g. an explicit-stack DFS or a worklist tracking `(node, depth)` pairs — would produce byte-identical observable behaviour. Tail-recursive transformations and trampolining are also closed off by the literal reading. The implementer is denied an implementation choice that has no observable consequence, on a hot path where avoiding native call-stack growth on deep payloads is a legitimate engineering concern (the cap is 5, but the same routine runs at five sites and a malformed-but-parseable payload could otherwise be much deeper).

## Issue introduction

**Verdict:** introduced-in-resolution
**Introducing commit:** `216130d` — "pi-loom spec: resolve \"Depth ≤5: counting algorithm and enforcement point unspecified\"" (2026-05-04)
**History:** The Depth Enforcement section did not exist before `216130d`; the depth cap was a single bullet ("Depth ≤ 5 levels of nesting at runtime …") in the subset list. The resolving commit added the full § Depth Enforcement block — counting algorithm, worked examples, enforcement points, error shape, edge cases — and in the paragraph that establishes the AJV-ordering contract appended "Implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits and produces the failure." `git log -L` and `git log -S "recursive descent"` confirm the phrase was introduced in that single commit and has not been edited since (later touches to the surrounding paragraph — `7f217a8`, `f7a505e`, `ba68984` — extended the error-shape and per-boundary table without altering this sentence). The defect is a classic over-fix: while pinning the observable contract the resolution also reached for an implementation strategy, mixing two registers in one sentence.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/schema-subset.md` § Depth Enforcement, edit the paragraph after the five-site enforcement list so the prescription drops out while the observable contract is preserved. Replace:

> The walk runs **before** AJV at each site: it is a cheap fast-fail and avoids feeding pathologically deep payloads into the validator. Implementation is a recursive descent over the parsed JSON value with a depth counter; the first node whose depth would exceed `5` short-circuits and produces the failure.

with:

> The walk runs **before** AJV at each site: it is a cheap fast-fail and avoids feeding pathologically deep payloads into the validator. The walk short-circuits at the first node whose depth would exceed `5`, and that node's location populates the `path` of the resulting failure (see Error shape below).

Edge cases for the implementer: the "first node" tiebreak is fully determined by the counting algorithm in § Counting algorithm together with document order (parent before children; siblings in source order) — no new rule is needed, but reviewers will read "first" as document-order-first, so an iterative walk must preserve that.

## Consequence

**Severity:** advisory

A literal reader gates conformance on the traversal strategy: an iterative walk with identical observable behaviour reads as non-conforming. The practical risk is small (the routine is straightforward and the cap is shallow), but the clause constrains implementation freedom on a non-observable surface without earning anything, and a future reviewer policing prescription leakage will keep flagging it.
