## Problem

`V2e`'s **Adds** field claims the full inbound/outbound wire-name translation boundary, including "outbound wire-name JSON", matching `runtime-value-model.md` §Wire-name translation, which declares translation in two directions. But the sole wire-translation **Tests** bullet in both `V2e` and `V2e-T` asserts only the inbound direction; nothing asserts the outbound path — that a `field as "Wire"` rename serialises to the wire name on the way out, or that a non-renamed field is emitted unchanged. The outbound half of the boundary can therefore ship with a broken rename — leaking loom-internal field names to tool input, query-response payloads, and `invoke` arguments — while the red/green gate stays green.

**History:** The asymmetry exists in the corpus's first plan build (c6a664e): the original `V2c` leaf already named "outbound wire-name JSON" in **Adds** while its only wire-name **Tests** bullet asserted the inbound direction. The later `V2c`→`V2e` split (d4f6e5c) moved the wire-name boundary into the new `V2e` leaf and carried the inbound-only Tests bullet across verbatim, so the gap was inherited rather than introduced by the split.

## Solution approach

Add an outbound-direction **Tests** bullet to `V2e-T`, citing `runtime-value-model.md` §Wire-name translation, that asserts the outbound walk renders `field as "Wire"` to its wire name and emits a non-renamed field unchanged. Mirror that bullet into `V2e` so the implementation leaf's green gate matches the failing test, and extend `V2e`'s **Ships when** so `npm test` covers the outbound vector alongside the inbound one.

## Solution constraints

- The spec is read-only for this fix — `runtime-value-model.md` already declares the two-direction rule.
