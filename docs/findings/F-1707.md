## Finding

QRY-19 — *"The author must pick one of `?` / `let _ =` / a binding"*, surfaced as the parse error `loom/parse/discarded-query-result` on a bare `@`...`` expression-statement — is a normative rule about the **form of a query call site** and the obligations imposed by its `Result` return type. It is a peer of QRY-1 (untyped form, `Result<string, QueryError>`) and QRY-2 (typed form, `Result<Schema, QueryError>`): all three pin what an author may write at a query call site and what the static contract obliges them to do with the returned `Result`.

The rule currently lives in `query-escapes-stringification.md`, under the H2 `## Discarded query results are a parse error (\`loom/parse/discarded-query-result\`)`. That page covers backtick/`$`/`\\` escapes (QRY-17) and per-type stringification of `${expr}` interpolations (QRY-18) — both of which concern the **inside** of a template body, not the call-site shape of the surrounding expression. An author reading `query-forms.md` for the full call-site contract — the natural entry point, both topically and via the V13a leaf's Spec field — finishes QRY-5/QRY-6/QRY-7 (multi-line, degenerate, dedent) without ever learning that a bare `@`...`` expression-statement is a parse error, and the only way to land on QRY-19 is to follow the unrelated "Escapes" page heading.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `f5e89f4` ("docs: enforce 100KB hard / 30KB recommended size cap on spec set", 2026-06-04)
**History:** The "Discarded query results are a parse error" rule first entered the corpus in `53f5831` (2026-05-03, "spec: rewrite query primitive, system prompts, and schema syntax") as a `**Discarded query results are a parse error.**` paragraph inside the monolithic `spec.md`. The split-into-topics commit `fecb504` carried it across into `spec_topics/query.md` unchanged. The misplacement was introduced at `f5e89f4` when `query.md` was split into per-feature subpages — the discard rule was placed into the newly-created `query-escapes-stringification.md` alongside escapes (QRY-17) and stringification (QRY-18) rather than into the sibling `query-forms.md` that received QRY-1/QRY-2. The later REQ-ID assignment in `716a13b` (2026-06-25, "fix F-1286 — coin QRY-1..21 across query subtree") named the block QRY-19 but did not relocate it.

## Solution Space

**Shape:** single

### Recommendation

Move the QRY-19 block from `query-escapes-stringification.md` to `query-forms.md`, inserting it as a new H2 `## Discarded query results are a parse error (\`loom/parse/discarded-query-result\`)` immediately after the `## Typed form` section (after QRY-2) and before `## Multi-line templates` (QRY-5). The moved block is the three paragraphs currently at lines 38–50 of `query-escapes-stringification.md`:

1. The H2 heading itself (carrying the `loom/parse/discarded-query-result` diagnostic-code suffix).
2. The QRY-19 `<a id="qry-19"></a>` block with its three-form code example and the "The author must pick one of:" framing.
3. The diagnostic-message paragraph ("The diagnostic on a bare `@\`...\`` expression-statement reads...").
4. The `let _ = expr` / `void`-tail-discard paragraph that establishes the discard hatch is a generic binding form and that `void`-tail is the equivalent at the function-return surface — this paragraph belongs with QRY-19 because it defines the universe of forms QRY-19 covers; QRY-20 (which sits in the next section on the source page) only refers to that universe.

Implementer-relevant edge cases:

- The `<a id="qry-19"></a>` anchor must travel with the block so that the `code-registry-parse.md` row's "Defined at" reference (currently the file-level `../query.md` pointer) can be repointed to `../query/query-forms.md#qry-19`. Run a corpus-wide grep for `qry-19`, `QRY-19`, and `discarded-query-result` after the move to catch any other anchor consumers (`coverage-matrix.md` uses the REQ-ID, not the anchor, and is unaffected).
- The source-page `## Observability of discarded results` (QRY-20) and its `### Panics during interpolation are not caught by \`let _ =\`` subsection (QRY-21) remain on `query-escapes-stringification.md` for now; QRY-20's first sentence ("`let _ = @\`...\`` … is a true discard at the user-facing surface") still parses without the QRY-19 framing on the same page because it names the form directly. A separate finding ("QRY-20/QRY-21 on the wrong page") covers their own relocation independently — see Related Findings.
- The V13g leaf and its tests-leaf `Spec` field add `query-forms.md` alongside the existing `query-escapes-stringification.md` (or replace it if QRY-20/QRY-21 also move out, per the sibling finding). The leaf's `Adds` / `Tests` / `Ships when` text is unchanged.

## Consequence

**Severity:** advisory

A reader consulting `query-forms.md` for the complete query call-site contract finishes the page (QRY-1 through QRY-7) without ever learning that `@\`...\`` as a bare expression-statement is a parse error, and there is no in-page forward pointer from `query-forms.md` to the rule. Implementations can still find QRY-19 — the diagnostic is registered in `code-registry-parse.md`, the rule has a REQ-ID, and `V13g` cites the owning page — so the system can still be built correctly, but every reader pays a discovery tax and adjacent findings ("page title understates scope", "QRY-20/QRY-21 on the wrong page") confirm the same accumulating mis-scoping of this file.
