## Problem

In `V12a-T` and `V12a`, the single REQ-ID `SLSH-2` is exercised by three distinct `Tests` bullets — streaming-ordering, `Err`-after-partial-text, and mid-stream-cancellation — yet all three open with the same bare `SLSH-2:` token. A `SLSH-2` test-failure report therefore names no facet, and an implementer cannot derive three distinct test-function names from the identifier alone. The same shared-identifier pattern appears in the MVP pair (`M-T` carries two `SLSH-2:` bullets, `M` one), so disambiguating only the V12a leaves leaves the MVP instance inconsistent.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** `c6a664e` ("pi-loom plan: build/update plan for spec.md + review", 2026-06-10)
**History:** `V12a-T-slash-dispatch.md` and `V12a-slash-dispatch.md` were both created in `c6a664e`; `git show c6a664e:docs/plan_topics/V12a-T-slash-dispatch.md` shows all three `SLSH-2:` bullets present at that first authoring. Two later commits touched these leaves (`85fc906` added the version-bump-checklist link to the streaming bullet; `1abb15a` added the failure-path assertions to `V12a` `Ships when`) but neither introduced or addressed the shared identifier — the collision dates to inception.

## Solution approach

Disambiguate the three `SLSH-2` `Tests` bullets in `V12a-T` and `V12a` so each streaming facet — ordering, `Err`-after-partial-text, mid-stream-cancellation — carries a distinct per-bullet handle. Apply the same disambiguation to the two `SLSH-2` bullets in `M-T` and the one in `M` in the same pass so the MVP and V12 instances stay consistent. The facet wording is the implementer's to pick, provided each scenario's descriptor is distinct within its leaf.

## Solution constraints

- The spec is read-only for this fix.
- Do not mint new spec REQ-IDs (e.g. `SLSH-2a/2b/2c`); REQ-ID allocation is governed by `governance.md` and would require a spec-side change.
