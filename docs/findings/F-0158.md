## Finding

`SHUTDOWN_AWAIT_CAP_MS = 2000` lives in V9a's single source-of-truth pinned-constants block (`V9a-capability-probe.md` Adds), which explicitly names V9g, V9i, and V17a as the leaves that consume it without redeclaring. V9g's Adds and Tests both consume the constant (sub-step-3 `Promise.allSettled` await cap; the `reload-teardown-timeout` cap assertion), yet V9g's `Deps.` field (`V9g-T, V9e, V9h, V17a`) carries no direct `V9a` edge.

The two sibling consumers of the same constant declare it directly: V9i's Deps lists `V9a` (and consumes the constant in `PIC-9`), and V17a's Deps lists `V9a`. V9g is the only one of the three named consumers whose Deps omit the constant's physical owner.

This is a declaration-completeness / parity gap, not an execution defect. V9a is reachable from V9g transitively along two declared, acyclic paths — V9g→V9e→V9b→V9a and V9g→V17a→V9a — so the DAG is satisfied and the leaf is pickable in correct order. The convention's `Deps.` rule (`conventions.md` §Leaf format) requires only that leaf IDs which must be complete first be listed; it does not mandate a direct edge for every directly-consumed seam, so the gap violates no hard rule. The cost is purely traceability: a maintainer scanning V9g's Deps to locate the physical home of the constant it imports does not see V9a, and the asymmetry against V9i/V17a reads as an oversight.

## Solution Space

**Shape:** single

### Recommendation

Add `V9a` to V9g's `Deps.` field in `docs/plan_topics/V9g-session-shutdown.md`, making the line read `**Deps.** \`V9g-T\`, \`V9e\`, \`V9h\`, \`V17a\`, \`V9a\``. This mirrors the sibling constant-consumers V9i and V17a, both of which already list `V9a` directly, and makes the physical home of `SHUTDOWN_AWAIT_CAP_MS` traceable from V9g's Deps. The edge is already transitively present (V9e→V9b→V9a, V17a→V9a), so adding it changes no sequencing — it only records the direct seam consumption. No spec edit and no change to V9a, V9i, or V17a is required.

## Consequence

**Severity:** advisory

Nothing breaks at execution time — V9a is transitively satisfied via both V9e→V9b→V9a and V17a→V9a, so V9g is pickable in correct order. The residue is traceability and parity: V9g is the lone named consumer of `SHUTDOWN_AWAIT_CAP_MS` whose Deps omit the constant's physical owner, so a maintainer cannot read the import home off V9g's Deps the way V9i and V17a allow.
