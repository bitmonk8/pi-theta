## Finding

`docs/spec_topics/implementation-notes.md` §Runtime — the typed-query *reference implementation* "Runs the two-phase loop" sub-bullet — justifies the no-`before_provider_request`-hook posture with the clause "*a global hook that rewrites every provider call in every session is the mechanism **Option B of the resolved typed-query design** rejected*". The italicised proper noun references an Option A / Option B deliberation that lived in the now-pruned `docs/spec-review.md` decision log. No live spec page defines an "Option A" or "Option B" of any "typed-query design"; the only surviving "Option A" mention in `docs/spec_topics/` is the dated string `(Option A)` carried inside commit-message-style strings in finding files. An implementer who tries to chase "Option B of the resolved typed-query design" finds no anchor, no normative target, and no rejected-option register.

The substantive content of the sentence — that loom 1.0 takes the per-call `options.toolChoice` route on `complete()` scoped to the forced respond turn rather than a global `before_provider_request` payload-rewrite hook — is implementer-relevant and is already pinned by the surrounding sentences (the `complete()` channel and the per-provider `toolChoice` mapping). The "Option B" reference is the decision-log residue that adds no obligation.

## Solution Space

**Shape:** single

### Recommendation

In the typed-query reference-implementation "Runs the two-phase loop" sub-bullet of `docs/spec_topics/implementation-notes.md` §Runtime, replace the existing rationale clause

> The runtime does **not** install a `before_provider_request` payload-rewrite hook in loom 1.0 — a global hook that rewrites every provider call in every session is the mechanism Option B of the resolved typed-query design rejected; the per-call `options.toolChoice` route on `complete()` is scoped to exactly the forced respond turn and avoids the cross-session-fingerprinting machinery a global hook would require.

with the implementer-relevant remainder, dropping the "Option B…rejected" sub-clause:

> The runtime does **not** install a `before_provider_request` payload-rewrite hook in loom 1.0; the per-call `options.toolChoice` route on `complete()` is scoped to exactly the forced respond turn and avoids the cross-session-fingerprinting machinery a global payload-rewrite hook would require.

This preserves both load-bearing facts (no `before_provider_request` hook in loom 1.0; per-call `options.toolChoice` is the chosen route, specifically because a global hook would impose cross-session fingerprinting) while deleting the dangling reference to a sibling option no live page defines. Make the same edit to the closing sentence of the `conversation-drive.md` typed-query bullet, which forwards to this Implementation Notes paragraph via "The full mechanism — including the `before_provider_request` payload-rewrite hook loom 1.0 rejects — is described in [Implementation Notes — Runtime]", only if that sentence's "loom 1.0 rejects" phrasing reads as a similar dangling-option pointer after this site's edit; otherwise leave it (it forwards by mechanism, not by Option label).
