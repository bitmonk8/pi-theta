## Finding

V15e's hot-reload cache-eviction step drops the static-resolution per-pass parse-cache entry for the changed file and for every file that transitively imports it **across the `.warp` import-edge graph** (`V15c`). That is the only edge type the eviction walk follows. But the static-resolution per-pass parse cache (`V15a`) is populated by a transitive walk over a different and larger graph: `invocation.md` §Static resolution states the walk "is transitive: callees referenced by a callee's own literal `invoke` paths and `.loom` entries in `tools:` are loaded into the same cache", and the same cache backs `invoke` argument-type checking, cross-loom `invoke<T>` return-type inference, `.loom`-callable return inference, and the invocation-cycle walker.

`invocation.md` then pins the hot-reload contract over that graph explicitly: "Hot-reload of a callee re-runs the static-resolution pass for **every parent that has the callee in its reachable graph**." The reachable graph is the `invoke` + `.loom tools:` graph, not the `.warp` import-edge graph. A `.loom` parent `P` that reaches a changed `.loom` callee `X` only through an `invoke("./X.loom", …)` literal or a `tools:` `.loom` entry is **not** a `.warp` importer of `X`, so V15e's walk never visits `P`. If `P`'s cached parse/lower retains its cross-callee static checks against the pre-change `X` (argument-type, `invoke<T>` return-type, cycle), the `LoomRegistry` swap leaves those entries stale and `P` is served from cache on the next pass.

The plan never states the load-bearing premise that makes `.warp`-only eviction sufficient — i.e. that `invoke` / `tools:`-edge parents need no eviction because their cross-callee checks are recomputed each pass rather than cached. As written, V15e under-evicts relative to `invocation.md`'s "every parent in the reachable graph" mandate, or the sufficiency invariant is unstated. The recently-added negative-direction test (commit `c0a8a04`) closes a *different* stale-serve gap — a newly-established `.warp` import edge across the swap — and does not touch the `invoke` / `tools:`-edge case.

## Solution Space

**Shape:** multiple

### Option A — Re-walk the full static-resolution reachable graph

**Approach.** On the `LoomRegistry` swap, evict (or re-run the static-resolution pass for) not only the changed file and its transitive `.warp` importers, but also every parent that reaches the changed file through `invoke(...)` paths and `.loom tools:` entries — the same transitive graph `V15a`'s static-resolution walk builds — matching `invocation.md`'s "re-runs the static-resolution pass for every parent that has the callee in its reachable graph."

**Plan edits.** In `V15e`'s Adds/Tests/Ships-when, extend the eviction set from "transitive `.warp` importers" to also include `invoke`/`.loom tools:`-edge parents reachable to the changed file. Add a `V15e-T` case where a changed `.loom` callee's signature change forces a re-check at an `invoke` / `tools:`-edge parent (stale `invoke<T>` return-type or arg-type check is dropped and recomputed). Update the `coverage-matrix.md` cka-44 parenthetical so its scope description matches the broadened walk.

**Spec edits.** Reconcile `implementation-notes.md` §Static-resolution load pass so its eviction sentence covers the changed file + transitive `.warp` importers + `invoke`/`.loom tools:`-edge reachable parents, removing its divergence from `invocation.md`.

**Pros.** Matches the authoritative `invocation.md` contract; no stale cross-callee check is ever served.

**Cons.** Larger re-parse set per hot reload; more test surface.

**Risks.** Mis-scoping the reachable graph; a callee reachable through more than one edge type must be evicted once, not multiply.

### Option B — Document a recompute-each-pass sufficiency invariant

**Approach.** Keep `.warp`-only eviction and instead state the invariant that makes it sufficient: an `invoke`/`.loom tools:`-edge parent's cross-callee static checks are recomputed from the (re-parsed) callee on the parent's next load pass and are never reused from a cached parent result, so only the changed file and its `.warp` importers need explicit eviction.

**Plan edits.** State the recompute-each-pass invariant in `V15e`'s Adds; have `V15a` confirm cross-callee static-resolution results are not persisted across passes; add a `V15e-T` assertion that an `invoke`/`tools:`-edge parent observes a changed callee's new signature on its next pass without explicit parent eviction.

**Spec edits.** Add a reconciliation note in `invocation.md` / `implementation-notes.md` stating that `.warp` eviction plus per-pass recompute together satisfy "re-runs the static-resolution pass for every parent in the reachable graph."

**Pros.** Smaller eviction set.

**Cons.** Sound only if parent cross-callee results are genuinely not cached; since V15e's whole premise is that the per-pass parse cache persists entries across passes (otherwise nothing would need eviction), the same persistence likely applies to parent results, which would make B unsound.

**Risks.** Silently relies on an unverified non-caching property; the parent-observes-new-signature test could pass vacuously if the cache is in fact rebuilt for unrelated reasons.

### Recommendation

Take Option A. `invocation.md` §Static resolution already mandates re-running the static-resolution pass for every parent that has the changed callee in its reachable graph, and that graph is the `invoke` + `.loom tools:` graph — so the eviction/re-walk scope must follow those edges, not only `.warp` import edges. Reconcile `implementation-notes.md` §Static-resolution load pass and the `coverage-matrix.md` cka-44 scope parenthetical in the same change so all three artefacts state one scope. Edge cases the implementer must watch: a callee reachable through both a `tools:` entry and an `invoke` literal is evicted once; and the reachable-graph walk runs over the post-swap rebuilt graph, consistent with the rebuild-before-walk ordering the existing negative-direction test already pins.

## Consequence

**Severity:** correctness

On an in-process hot reload of a `.loom` callee whose signature changed (params, arity, or return type), a parent that reaches it only through an `invoke` literal or a `.loom tools:` entry keeps serving its stale cached static-resolution checks — the parent's `invoke`-arg / `invoke<T>`-return / cycle diagnostics reflect the pre-change callee, diverging from `invocation.md`'s mandate to re-run the static-resolution pass for every reachable-graph parent. Two implementers reading V15e versus `invocation.md` would build different eviction scopes.
