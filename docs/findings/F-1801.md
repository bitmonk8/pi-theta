## Finding

The load-bearing captured-value protection (discriminator uses `capturedEventReason`, not a fresh `event.reason` access) has no acceptance fixture for a getter that yields a valid session-only reason on first read then throws on re-access. Fixture (d) covers only the unknown/`"<unreadable>"` path (zero rows), so a non-conformant fresh-read implementation passes every current happy-path fixture.

## Issue introduction

## Solution Space

**Shape:** single

### Recommendation

Add an acceptance fixture (e.g. `(n)`) to the *Acceptance criteria* list: an `event.reason` whose getter returns `"new"` on the handler-entry capture read and **throws on any subsequent access**. Assert that the handler emits exactly one `runtime-degraded` row with `details.event.reason = "new"` (the captured value), that no throw escapes the handler, and that the next slash invocation returns `"degraded; /reload to recover"`. State that the fixture witnesses the captured-string-triplet protection against a non-conformant fresh-read discriminator; optionally extend across `{"resume","fork"}`.
