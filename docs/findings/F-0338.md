## Finding

`H5a` (closing-gate automation) is a static-reconciliation CI gate: it reconciles the spec REQ-ID set, the diagnostics-code registry, the `coverage-matrix.md` mappings, and the seeded gate fixtures against the asserting test corpus, all run under `npm test`. Its only real prerequisite is the test-runner scaffold that `H1a` establishes (a green `npm test` on a fresh checkout with the `eslint-plugin-loom-local` package resolving).

`H5a` declares `**Deps.** H4a`. Following the horizontal chain `H1a → H2a → H3a → H4a`, this transitively serialises `H5a` behind all three of `H2a`, `H3a`, and `H4a`. But `H5a` consumes none of their outputs: it references no `H2a` cross-cutting lint/architectural rule (the `no-broad-catch` allow-list reconciliation arm is explicitly owned by `H5c`, not `H5a`), no `H3a` DI-seam / runtime-root construction, and no `H4a` extension-factory shell or end-to-end harness. `H5a`'s gate operates over docs and a dedicated seeded test-fixtures root, never loading the extension or driving a session double.

The dep edge is therefore a false sequencing constraint. It does not break `H5a`'s executability — the leaf still implements correctly when picked up — but it removes the build-order parallelism that would let `H5a` proceed as soon as `H1a` is complete, independent of the `H2a`/`H3a`/`H4a` production-scaffold track.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H5a-closing-gate-automation.md`, change the **Deps.** field from `` `H4a` `` to `` `H1a` ``. `H1a` is `H5a`'s true floor: it provides the `npm test` runner the gate hooks into, and `H5a` consumes nothing from `H2a`, `H3a`, or `H4a`.

Edge cases the implementer must confirm: the downstream consumers `H5b` (Deps already lists `H1a, H5a, …` explicitly), `H5c` (Deps `H5a`), and `H5d` (Deps `H5a`) all depend on `H5a`'s completion rather than on the removed `H4a` edge, so re-pointing does not break their pickability.

## Consequence

**Severity:** advisory

If shipped unfixed, the dep DAG carries an edge with no underlying consumption: `H5a` is unnecessarily blocked from being picked up until `H4a` (and transitively `H2a`/`H3a`) completes, even though it only needs `H1a`. A parallel or reordered build loses the opportunity to land `H5a` alongside the production-scaffold track. No implementation diverges and no gate misfires; the cost is lost scheduling freedom and a misleading dependency signal.
