## Finding

The canonical schema hash recipe in `schema-subset.md` § *Canonical schema hash* — part of the loom 1.0 on-disk and on-wire contract — does not define how numeric literals embedded in the lowered fragment are serialised. Instead, step 2 ("Canonical form") forwards to `binder/defaulting-system-note-echo.md` § *Echo policy* and pulls in **BNDR-4** (integer rendering), **BNDR-5** (number rendering), and the **BNDR-6** reference-renderings table as the normative algorithm. The hash thereby treats a sibling page whose primary topic is *binder system-note echo formatting* as the authoritative source of a byte-level cryptographic contract.

This makes two unrelated concerns co-own one rule set. The binder echo page evolves under echo-formatter pressure (line-cap wording, `(default)` tagging, ambiguity-message phrasing, BNDR-6 table back-fills); the schema hash evolves under cache-stability pressure where *any* byte change is a breaking change for `__inline_<slug>`, `__loom_respond_<slug>`, `__loom_callee_<slug>__*`, `__loom_bind_<slug>`, the AJV compiled-validator cache, and every replayable provider payload or fixture snapshot. A future editor tightening, say, BNDR-5's `1e21` rendering for echo-readability reasons would silently invalidate every cached lowered-schema artefact across the system, with no signal from the hash recipe that it has changed. The recipe's own "the loom 1.0 spec pins it exactly" sentence is undermined by the pin living on a page whose stability budget is different.

A second-order effect: an implementer reading `schema-subset.md` to build the hash must navigate to the binder page and then carefully extract the number-rendering bullets while *not* applying the 120-code-point cap or the `(default)` suffix (the recipe text explicitly warns against this). The "only that rendering algorithm is borrowed" fence works in prose but is fragile against the obvious co-located rules a reader on that page will see.

## Solution Space

**Shape:** multiple

### Option A — Inline the number-rendering rule in `schema-subset.md`

**Approach.** Promote BNDR-4 / BNDR-5 / BNDR-6 to live in `schema-subset.md` § *Canonical schema hash* (or a dedicated sub-section adjacent to it), retaining their REQ-IDs. Have `binder/defaulting-system-note-echo.md` § *Echo policy* cite the schema-subset page for the integer/number value-rendering, keeping only the echo-specific concerns (whitespace handling, unquoted-string predicate, 120-code-point cap, `(default)` suffix, array `…+N more` rule) at the binder page.

**Spec edits.**
- Move the BNDR-4 integer-rendering paragraph, BNDR-5 number-rendering paragraph + non-normative derivation recipe, and the BNDR-6 reference-renderings table into `schema-subset.md`.
- Replace step 2's "are serialised by the … rendering algorithm pinned in [Binder — Echo policy] …" sentence with a direct in-place statement.
- Rewrite the four integer/number rows of the echo's Format-rules list as a forward-link to the new home; keep all other rows untouched.
- Update coverage-matrix `BNDR-4, BNDR-5 | V2d` and `BNDR-6 | V11h` rows if the REQ-IDs move pages (the IDs themselves can stay regardless).

**Pros.** Co-locates the rule with its primary stability contract (the cryptographic hash). A reader of `schema-subset.md` gets the algorithm without a cross-page hop into echo-irrelevant surroundings. Eliminates the "only that rendering algorithm is borrowed" fence and the warning against the 120-code-point cap.

**Cons.** Moves three REQ-IDs to a page whose section anchors are not currently the BNDR ones; inbound link surgery across binder leaves and the coverage matrix.

**Risks.** Existing fixtures or external citations referencing `#bndr-4` / `#bndr-5` / `#bndr-6` on the binder page would need redirects (alias anchors per GOV-26) for stability.

### Option B — Move the rule to a shared `number-rendering.md` topic page

**Approach.** Create `docs/spec_topics/number-rendering.md` (or fold into a renamed neutral page) owning BNDR-4 / BNDR-5 / BNDR-6. Both `schema-subset.md` § *Canonical schema hash* and `binder/defaulting-system-note-echo.md` § *Echo policy* become consumers that cite the shared page.

**Spec edits.**
- New topic page carrying BNDR-4 / BNDR-5 / BNDR-6 verbatim, plus a short scope statement ("this rule is the integer/number value-rendering primitive consumed by the canonical schema hash and by the binder argument echo").
- `schema-subset.md` step 2: replace the inline citation with a forward-link to the new page.
- `binder/defaulting-system-note-echo.md` Format rules: replace the integer/number bullets with a forward-link.
- Add the new page to `spec.md` and to the appropriate aggregator under GOV-30.
- Glossary: add a "number rendering" entry pointing at the new page; touch the schema-slug entry's cross-reference.

**Pros.** Mirrors the plan structure exactly — `V2d` is already framed as a shared renderer consumed by both `V5f` and `V11h`; the spec topology gains the same factoring. Neither consumer-page owns rules the other depends on; future tightening of the rule has a single, clearly-scoped home.

**Cons.** Adds a topic page (one more file in the corpus); requires touching `spec.md`, the aggregator, and the glossary. Heavier edit than Option A.

**Risks.** Same anchor-stability concern as Option A (BNDR-6 sub-letter anchors `bndr-6a`–`bndr-6x` migrate); GOV-26 alias anchors mitigate.

### Recommendation

**Option B**, because the plan already treats the integer/number renderer as a shared seam (`V2d`, consumed by both `V5f` and `V11h`) and a shared topic page makes the spec topology match. Option A is acceptable if minimising the edit surface outweighs the symmetry argument, but it leaves the renderer's normative home on a page whose primary topic is unrelated to the canonical hash. Under either option, edge cases the implementer must watch: (i) preserve the BNDR-4 / BNDR-5 / BNDR-6 REQ-IDs verbatim (REQ-ID retirement requires GOV machinery); (ii) keep the BNDR-6 sub-letter anchors (`bndr-6a` … `bndr-6x`) reachable via alias per GOV-26 since `V2d-T` and BNDR-coverage rows cite them individually; (iii) retain the "the 120-code-point cap and `(default)` suffix are echo-formatter concerns and do not apply to the hash" scope-fence wording on whichever page the rule ends up *not* on, so a reader landing from the other consumer does not over-apply; (iv) keep `V5f`'s phrase "binder-number-rendered" updated to "canonical-number-rendered" (or equivalent) to match the new ownership.

## Consequence

**Severity:** correctness

Two distinct stability contracts share one rule set. An edit driven by binder-echo concerns can silently change canonical-hash bytes, invalidating cached lowered schemas, fixture snapshots, and replayable provider payloads across the system; the recipe's own "pinned exactly" claim is undermined by the pin sitting on a different page whose evolution pressure is different. An implementer reading `schema-subset.md` in isolation cannot derive the integer/number rendering without traversing to a page that also carries non-applicable echo rules.
