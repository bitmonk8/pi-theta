## Finding

The canonical interpolation-stringification table (QRY-18 in `query/query-escapes-stringification.md`) renders an interpolated value "by the **Loom static type** of the expression" and enumerates exactly nine rows: `string`, `integer`, `number`, `boolean`, `null`, `Enum variant`, `array<T>`, `Schema-typed object`, and `Result<T, E>`. It has **no row for a union type**. A union-typed interpoland is reachable through every surface that feeds the table:

- In `system:`, the path grammar explicitly states that "Arrays, discriminated unions, and primitive scalars terminate the path: `${param}` (no further `.`) is always allowed and is rendered by the stringification rule below" — so a discriminated-union-typed `params` field is admitted as `${param}` and deferred to the table.
- In `@`-query templates, `${obj[k]}` has static type "the union of the receiver's declared field types" (per `expressions.md`), and any binding typed `T | null`, `A | B`, or a primitive union (`string | integer`) is a legal interpoland.
- The runtime-value placeholder rule (`diagnostics/placeholder-rendering-a.md` §2) routes `match`-scrutinee rendering through the same canonical table, so a union-typed scrutinee in a `MatchError` hits the same gap.

Because the table is keyed on static type and a union is none of the nine enumerated types, the rendered output is undefined: one implementer falls back to `JSON.stringify`, another to JavaScript `String(...)`, another panics. The QRY-18 rule itself warns that an unspecified rendering "would silently corrupt prompts without any diagnostic" — which is exactly the failure mode a union value triggers today.

(The original framing also flagged that the `system:` text's claim "`params:` types do not include `Result`" rests on an unenforced premise; in fact `params:` field types are a lowered-schema position, and `loom/parse/result-in-schema-position` — whose registry entry explicitly names "a `params:` field type" — rejects a `Result`-typed param at parse time, so that premise *is* enforced. Likewise the nested-enum rendering concern is already covered: `array<T>` and `Schema-typed object` rows apply `JSON.stringify` with recursive wire-name translation, and `JSON.stringify` of an enum value yields the bare wire string per the Runtime Value Model. The only real defect is the missing union row.)

The later split into `query/query-escapes-stringification.md` (`git log --follow`) carried the table verbatim. ## Solution Space

**Shape:** single

### Recommendation

Add a union row to the QRY-18 canonical table (and mirror it through `frontmatter-fields-b-and-templates.md`'s `system:` Stringification reference, which defers to the same table). Specify union rendering by dispatch on the resolved runtime value's concrete arm type:

- A discriminated-union or object-union value (e.g. `Cat | Dog`) renders per the `Schema-typed object` row — compact `JSON.stringify` with recursive wire-name translation.
- A mixed or primitive union (`string | integer`, `T | null`) renders per the table row matching the resolved runtime value's concrete type — e.g. the integer `42` in a `string | integer` slot renders per the `integer` row; `null` in a `T | null` slot renders per the `null` row (the literal text `null`).

State the rule as "render by the row matching the resolved runtime value's concrete static-arm type," so the union itself never needs its own rendering algorithm — it reuses the existing nine rows. Carry the existing `Result` exception through: a union that includes a `Result<T, E>` arm in interpolation position still fires `loom/parse/interpolated-result` (Result is non-interpolable on every surface). Add a union-typed test vector to `V6d`'s per-type enumeration and keep row-level correctness owned by `V13a`.

## Consequence

**Severity:** correctness

A union-typed `${...}` interpoland — a discriminated-union `params` field via `${param}`, an `${obj[k]}` whose static type is a field-type union, or any `T | null` / primitive-union binding — has no defined rendering, so two reasonable implementers diverge (compact `JSON.stringify` vs `String(...)` vs panic). Because the rendered text is fed verbatim to the model in both `@`-query templates and `system:` prompts, divergent rendering changes model behaviour and breaks the table's stated invariant that the model sees the same rendering of a given value regardless of surface.
