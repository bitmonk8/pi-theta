## Finding

`V6e`'s **Adds** field promises the `respond_repair` and `tool_loop` frontmatter fields "with range validation and defaults (`{3, validator_error}` / `{25}`)", and the spec pins those defaults normatively: `frontmatter-fields-b-and-templates.md` states an absent `methodology:` defaults to `validator_error` and the FRNT-1 field-contract table fixes the `respond_repair.attempts = 3` and `tool_loop.max_rounds = 25` defaults.

The acceptance criteria do not exercise that resolution. Both `V6e-T` (the failing-test leaf) and `V6e` carry only two Tests bullets — `FRNT-1` (the `max_rounds` bounds, the disable-at-`0` boundary, the exhaustion outcome) and the out-of-range `loom/load/frontmatter-value-out-of-range` code. `V6e`'s **Ships when** likewise enumerates only "`FRNT-1` bounds, the disable-at-`0` boundary, and the range code." No bullet asserts that a loom omitting `respond_repair`, `tool_loop`, and `methodology:` resolves to the pinned default values.

The default-resolution path is therefore untested. An implementation that omits default application, or pins a wrong default (e.g. `max_rounds: 20`, or `methodology: schema_repeat`), satisfies every existing assertion and ships green.

## Issue introduction

**Verdict:** present-since-inception
**Introducing commits:** c6a664e (2026-06-10, "pi-loom plan: build/update plan for spec.md + review")
**History:** `V6e` was authored at c6a664e already carrying the `defaults (`{3, validator_error}` / `{25}`)` promise in **Adds** while its Tests block listed only the `FRNT-1` bounds bullet and the out-of-range code bullet — the absent-field default-resolution assertion was never present. The only later edit to the leaf (4c76a7c) merely renamed the out-of-range diagnostic from `loom/parse/frontmatter-value-out-of-range` to `loom/load/frontmatter-value-out-of-range`; it did not touch the default-coverage gap. The gap has existed since the leaf was created.

## Solution Space

**Shape:** single

### Recommendation

Add a Tests bullet asserting absent-field default resolution to both `V6e-T` (first, so the failing test lands before the implementation) and `V6e`, and extend `V6e`'s **Ships when** to require it.

- In `docs/plan_topics/V6e-T-respond-repair-tool-loop.md` and `docs/plan_topics/V6e-respond-repair-tool-loop.md`, add a Tests bullet asserting that a loom which omits `respond_repair`, `tool_loop`, and `methodology:` resolves to the pinned defaults: `respond_repair = {attempts: 3, methodology: validator_error}` and `tool_loop = {max_rounds: 25}`. The bullet should also cover the partial case where `respond_repair` is present but `methodology:` is absent, resolving `methodology` to `validator_error`.
- In `docs/plan_topics/V6e-respond-repair-tool-loop.md`, extend the **Ships when** sentence so the asserted set includes the resolved absent-field defaults alongside the existing `FRNT-1` bounds, disable-at-`0` boundary, and range code.

Keep the assertion at the frontmatter-resolution layer that `V6e` owns; the downstream behavioural consumption of these values is exercised by `V13c`/`V13d` and is out of scope here.

## Consequence

**Severity:** correctness

The leaf's acceptance gate accepts a build whose default-resolution is absent or wrong without firing red, because no test resolves a loom with the fields omitted. A loom relying on the implicit defaults would then run with whatever value the implementer happened to apply, diverging silently from the spec's normative `{3, validator_error}` / `{25}`.
