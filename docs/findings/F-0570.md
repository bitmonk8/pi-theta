## Finding

The swallowing-handler suppression tests in `V17a` and its four per-site witness leaves (`V14f`, `V13f`, `V15h`, `V9o`) each assert that a late settlement landed via the `Checkpoint` seam after the checkpoint has surfaced `cause: "cancelled"` is suppressed along all three side channels, the first of which is "no Node `unhandledRejection`". The `Checkpoint` seam (`V8a`) makes the *landing* of the late settlement deterministic â€” `cancellation.md` Â§*Race semantics* says it provides "the deterministic-test substrate ... without relying on JS microtask scheduling". But asserting the *absence* of a process-level `unhandledRejection` is a different problem: Node dispatches `unhandledRejection` on a later turn of the event loop after a rejection remains unhandled, so a test concluding "it did not fire" must (a) install some observation mechanism (a `process.on('unhandledRejection', ...)` listener, or a harness-provided capture) and (b) drain a defined quiescence boundary â€” a fixed number of microtask/macrotask turns â€” before reading the negative result.

Neither the five leaves, nor `H4a` (the end-to-end harness), nor `conventions.md` states either part. The bare phrase "no Node `unhandledRejection`" is restated verbatim across all five leaves and their `-T` partners with no shared definition of how that absence is observed or when the observation window closes.

This is the classic false-green shape of a negative-event test: a build that *omits* the swallowing handler can still pass the assertion if the test reads its negative result before the `unhandledRejection` macrotask has had a chance to fire. The assertion would then report success without verifying the property it exists to verify.

## Solution Space

**Shape:** single

### Recommendation

Establish, in one authoritative location, both halves of the negative assertion's contract and have the per-site leaves reference it rather than each restating the bare "no Node `unhandledRejection`" phrase:

1. **Observation mechanism.** State how the test observes the absence of a Node `unhandledRejection` process event â€” e.g. a `process.on('unhandledRejection', ...)` listener installed for the duration of the assertion, or a harness-provided capture surface. If the capture is provided as harness infrastructure, that capability is owned by `H4a` and the assertion text references it.

2. **Quiescence boundary.** State the deterministic point at which the assertion reads its negative result â€” e.g. drain a fixed number of macrotask turns after landing the late settlement through the `Checkpoint` seam, or await a harness-provided quiescence signal â€” so the read happens only after any pending `unhandledRejection` macrotask would have fired.

The natural single home is `V17a`'s swallowing-handler side-channel suppression Tests bullet (and its `Ships when`), since `V17a` already owns the substrate and delegates the per-site routing to `V14f` / `V13f` / `V15h` / `V9o`. Add the mechanism + boundary statement there once; reduce the four per-site leaves' (and their `-T` partners') restatements to a reference to `V17a`'s definition so the contract has a single source of truth. The specific drain count / quiescence-signal value is an implementer decision; the plan obligation is that a deterministic boundary is named rather than left implicit.

## Consequence

**Severity:** advisory

Without a stated observation mechanism and quiescence boundary, each of the five leaves' implementers picks their own, and a leaf can ship with a negative `unhandledRejection` assertion that reads its result too early â€” passing green against a build that has *no* swallowing handler at all. That is precisely the silent-test-skipping failure mode the plan forbids: the suppression guarantee these leaves exist to verify would not actually be exercised. A competent implementer following standard negative-event practice will avoid this, so it degrades test fidelity rather than blocking pickup.
