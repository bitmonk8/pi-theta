## Finding

`host-interfaces-services.md` carries a named Pi behavioural
presupposition — `<a id="cwd-project-root-presupposition">` *Factory-time
`cwd == project root` presupposition* — inline inside the PIC-13
`cwd()` bullet of the `FileSystem` DI seam (lines ~110), instead of on
`host-prerequisites.md` alongside the other named host-behaviour
presuppositions. The block is framed identically to those siblings:
a "loom 1.0 treats it as a **presupposition** — in the same sense as
the [Host prerequisites for the degraded-state branch]…" clause that
explicitly cites two of the host-prerequisites entries as peers, an
*Authoritative source* paragraph with the same `known-fragile evidence`
boilerplate the siblings use, and a version-bump re-audit hook to
`version-bump-step2.md` checklist item `(w)`
(`#bump-checklist-cwd-project-root`).

A presupposition-inventory reviewer or version-bump contributor reading
only `host-prerequisites.md` does not see this entry — `host-prerequisites.md`
already enumerates ten named Pi behavioural presuppositions
(`degraded-state-host-prerequisites`,
`model-registry-population-presupposition`,
`settings-write-back-preservation-presupposition`,
`operator-always-present-presupposition`,
`loom-flag-namespace-presupposition`,
`getcommands-completeness-presupposition`,
`register-tool-post-startup-presupposition`,
`registry-persistence-presupposition`,
`messages-chronological-order-presupposition`,
`messages-leading-user-message-presupposition`,
`pi-slash-handler-promise-lifecycle-presupposition`), each of the same
shape as the misplaced `cwd-project-root-presupposition` block. The
defect is purely placement: content, anchor, cross-cites, and
bump-checklist coverage are all sound, but the inventory page they
should be findable from does not list this entry.

## Solution Space

**Shape:** single

### Recommendation

Move the `<a id="cwd-project-root-presupposition">` paragraph — its
heading line, body, and *Authoritative source* sentence — from the
PIC-13 `cwd()` bullet on `host-interfaces-services.md` into the
named-presupposition stack on `host-prerequisites.md`, placing it
alongside the existing sibling presuppositions it already cites as
peers. Preserve the `id="cwd-project-root-presupposition"` anchor
verbatim at the relocated paragraph so existing inbound links resolve
unchanged at the anchor portion. Add a sibling cross-cite to the
relocated paragraph naming the same peers the other entries do (the
`degraded-state-host-prerequisites` entry and the *No concurrent user
sessions* framing already cited in the paragraph's opening clause).

Leave a short forward-link under the PIC-13 `cwd()` bullet pointing to
the relocated paragraph (matching the stub pattern the sibling
"`Six slash-handler-promise-lifecycle presuppositions belong in
host-prerequisites.md`" finding plans for the `Cancellation source`
site), so a reader walking the `cwd()` seam definition still reaches
the launch-time presupposition that governs how the captured value is
used.

Repoint the file portion of the inbound link in
`version-bump-step2.md` checklist item `(w)`
(`#bump-checklist-cwd-project-root`'s link-target paragraph) from
`./host-interfaces-services.md#cwd-project-root-presupposition` to
`./host-prerequisites.md#cwd-project-root-presupposition`; item text is
unaffected. Grep the spec corpus for any other reference to the same
anchor and repoint the file portion of each (only the
`version-bump-step2.md` checklist site and the in-page anchor itself
are currently known consumers).

Implementer-relevant edge cases: the *Authoritative source* sentence
mentions `dist/core/extensions/loader.js` per the known-fragile
evidence pattern — this stays inside the moved paragraph and needs no
edit. The paragraph's opening "in the same sense as the [Host
prerequisites for the degraded-state branch]…" clause currently lives
on a different file from its referent; after the move both sit on the
same page, so leave that cite as-is (a same-page anchor link) rather
than restating it as a self-reference.

## Consequence

**Severity:** advisory

A bump auditor enumerating Pi behavioural presuppositions from
`host-prerequisites.md` misses this one entirely. The bump-checklist
walk (which goes through items `(a)`–`(ao)` in `version-bump-step2.md`)
still hits item `(w)`, so the audit obligation itself is not lost — but
the inventory page is the canonical roster a reviewer consults to ask
"have I named every load-bearing assumption loom makes about Pi?", and
that question gets the wrong answer here. A silent Pi change that broke
the launch-time `cwd == project root` contract would still be caught by
the `(w)` re-confirmation step if the auditor performed every checklist
item, but not by an inventory-driven audit.
