## Finding

The *Sequential by default* rule (`conventions.md` §Cross-cutting rules) permits a `Promise.all` / `race` / `allSettled` / `any` call in `src/**` only when **(a)** the calling leaf carries a `// allow: <token> — <page>` comment whose cited token resolves to a coverage-matrix REQ-ID or a `cka-<n>` code-keyed-area token, **and (b)** the cited obligation's *normative text actually mandates concurrency at the specific operation the combinator implements*. The two mechanical surfaces the rule names — the `H2a` `no-restricted-syntax` allow-list and the `H5a` closing-gate token-resolution arm — verify only requirement (a): that the comment is present and its token resolves to exactly one matrix REQ-ID / `cka-<n>` row. Neither surface inspects the cited obligation's prose, so a leaf may cite any resolvable matrix token — including one wholly unrelated to concurrency — and slip an unjustified combinator past CI green.

This is the same requirement-(a)-only / requirement-(b)-residue split the sibling *Specific exception types only* (broad-catch) rule has, but the two rules are treated asymmetrically. The broad-catch rule's **Enforcement posture** explicitly states that its mechanical surfaces "verify only requirement (a) … neither checks requirement (b)," declares the requirement-(b) residue carries "no mechanical gate," and routes it to the *Per-phase TDD ritual* self-review step and to release-time residue-inspection **item 6**. The *Sequential by default* rule's own **Enforcement posture** paragraph addresses only the *blocking-runtime* residue (busy-wait loops, dynamically-constructed sync calls → residue-inspection item 3); it concedes nothing about the Promise-combinator requirement-(b) gap and routes that residue to no review item. Meanwhile the *Doc updates* rule lists *Sequential by default* (`no-restricted-syntax` allow-list) flatly among the cross-cutting rules "backed by a mechanical surface," with the same confidence it grants broad-catch — overstating what the mechanical surface actually verifies for the combinator exemption.

The two production combinator-exemption sites the plan already declares — `V14b` (`// allow: cka-13 — tool-calls.md`) and `V9g` (`// allow: cka-31 — session-shutdown-semantics.md`) — both happen to cite obligations whose text genuinely mandates concurrency, so requirement (b) holds for them today. The defect is that nothing in the plan is charged with *checking* requirement (b): no gate, no self-review hook scoped to the combinator exemption, and no residue-inspection item.

## Consequence

**Severity:** correctness

An unjustified concurrency exemption can ship through a green gate: a leaf citing any resolvable matrix REQ-ID / `cka-<n>` token — even one unrelated to the operation the combinator performs — passes both mechanical surfaces, and no self-review hook or release-time residue item is charged with verifying the cited obligation actually mandates concurrency there. A reviewer or implementer reading the *Doc updates* grouping is told *Sequential by default* is "backed by a mechanical surface" and reasonably concludes the exemption is fully gated, while its sibling broad-catch rule discloses exactly the gap this rule hides.

## Solution Space

**Shape:** single

### Recommendation

Mirror the broad-catch (*Specific exception types only*) enforcement-posture treatment, which already resolves the identical requirement-(a)-only / requirement-(b)-residue split in `conventions.md`:

1. In `conventions.md` §Cross-cutting rules, *Sequential by default* **Enforcement posture**, add a statement (parallel to the broad-catch paragraph's "verify only requirement (a) … neither checks requirement (b)") that the `H2a` `no-restricted-syntax` allow-list and the `H5a` closing-gate token-resolution arm verify only that each `// allow:` site carries a comment whose cited token resolves to a coverage-matrix REQ-ID or a `cka-<n>` code-keyed-area token, and do **not** verify that the cited obligation's normative text mandates concurrency at the specific operation the combinator implements. State that this requirement-(b) residue carries no mechanical gate and is verified by the *Per-phase TDD ritual* self-review step and, at release time, by the residue-inspection item added in step 2.

2. In `conventions.md` §Loom 1.0 release-time residue inspection, add an item (alongside the existing item 6 for broad-catch category membership) tagged *(Sequential by default)*: the reviewer reads each `// allow:` Promise-combinator exempt site in `src/**` and confirms the cited obligation's normative text mandates concurrency at the operation the combinator implements. Give it a **Clean when** condition stating every combinator exempt site cites an obligation whose normative text mandates concurrency at that operation.

3. In `docs/plan_topics/H6a-live-corpus-activation.md` Adds., update the "nine-item manual sweep" count to match the new item total.

The *Doc updates* grouping may keep listing *Sequential by default* among the mechanically-backed rules — it is partly mechanically backed (requirement a) — because steps 1–2 remove the overclaim by disclosing and routing the requirement-(b) residue, exactly as the grouping already coexists with the broad-catch concession.

Implementer notes: the two existing exempt sites (`V14b` / `cka-13`, `V9g` / `cka-31`) satisfy requirement (b) under their current citations, so this fix surfaces no immediate residue defect; it closes the asymmetry and gives the requirement-(b) check an owner. Keep the spec read-only — requirement (b) is evaluated against existing spec obligation text, and no spec edit is needed.
