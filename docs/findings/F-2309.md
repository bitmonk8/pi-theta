## Finding

The spec deliberately delegates the `SDK_SURFACE_INVENTORY` entry-kind taxonomy to the implementation: `inventory-audit-intro.md` ¬ßSDK capability inventory states the taxonomy ‚Äî "the `namespace-function` kind ‚Ä¶ plus the additional non-`namespace-function` kinds" ‚Äî "is implementation-owned and not re-enumerated here", and the strict-capability arm repeats that "the literal kind name and per-kind payload shape remain implementation-owned". So there is no spec-pinned expected kind value for the six populated non-capability rows (`pi.registerFlag`, `pi.getFlag`, `pi-engines-node`, `peer-dep-range`, `strict-capability-probe`, `api-coverage`), and `V18a`'s implementation-side bullet correctly asserts only presence ‚Äî each row "resolves and carries its kind".

`V18a-T`'s paired second `PIC-15` Tests bullet, however, claims that "a missing row or an untagged/mis-tagged entry-kind reddens at build time." The "mis-tagged" half of that claim is unsubstantiated: with the kind set and per-row mapping implementation-owned and unenumerated anywhere (spec or leaf), there is no authoritative source the test can compare a row's tag against. As written the assertion is self-referential ‚Äî it can detect an *absent* row or an *untagged* row, but not a row tagged with the wrong kind, because "wrong" has no defined referent. A fresh implementer reading the test cannot tell what mechanism is supposed to catch a mis-tag.

The downstream consumers reinforce that the kind values are not externally pinned: `V18b` resolves Pi-surface references "against the `SDK_SURFACE_INVENTORY` entries and entry-kind taxonomy `V18a` establishes", and `V18c` consumes the `strict-capability-probe` entry's `probedName` payload ‚Äî both rely on whatever taxonomy `V18a` defines, not on an independent enumeration. The defect is local to the `V18a`/`V18a-T` Tests wording: the acceptance criterion overclaims a mis-tag detection that the implementation-owned taxonomy cannot ground.

The defect is the interaction: a test claiming mis-tag detection layered on top of a deliberately-unenumerated, implementation-owned taxonomy.

## Solution Space

**Shape:** single

### Recommendation

Reconcile the second `PIC-15` Tests bullet in both `docs/plan_topics/V18a-T-capability-inventory.md` and its mirror in `docs/plan_topics/V18a-capability-inventory.md` so the acceptance criterion no longer claims mis-tag detection it cannot ground. Do **not** add or reference a spec enumeration of the non-`namespace-function` kinds ‚Äî `inventory-audit-intro.md` ¬ßSDK capability inventory deliberately leaves the kind set and per-row mapping implementation-owned and "not re-enumerated here", so an enumerated expected-value source would contradict the spec.

State instead that the kind set and per-row kind mapping are implementer-defined (per `inventory-audit-intro.md` ¬ßSDK capability inventory), and that the test asserts presence and internal consistency against that implementer-defined taxonomy: each populated non-capability row resolves and carries a kind drawn from the implementation's own defined kind set (e.g. a `namespace-function` row carries the function-member payload that kind implies). Strike the "mis-tagged entry-kind reddens" clause, or rephrase it to the internal-consistency property the implementation can actually enforce (a row whose tag is not a member of the defined kind set, or whose payload does not match its tag's shape, reddens). Keep the "missing row" and "untagged row" halves ‚Äî those are substantiated. Align `V18a`'s Ships-when "carries its kind under the leaf-owned entry-kind taxonomy" clause with the same framing so the implementation leaf and its tests describe the same assertable property.

## Consequence

**Severity:** advisory

The acceptance criterion claims to catch a mis-tagged entry-kind, but with the taxonomy implementation-owned and unenumerated there is no referent for "mis-tagged", so the assertion can only verify presence; two implementers diverge on whether to write a presence-only check or an internal-consistency check, and neither violates the spec. The test reads as guaranteeing a guard it cannot actually substantiate, weakening the leaf's self-documenting acceptance gate without blocking a working implementation.
