## Finding

The capability-probe spec defines `PIC-5` as a single invariant: *"Probes MUST NOT be added beyond the five enumerated checks"* (`capability-probe.md`, anchor `pic-5`), where the "five" are the probe sub-steps `(a)`–`(e)`. Both `V9a` and its paired tests leaf `V9a-T` carry **two** Tests bullets prefixed `PIC-5:` with materially different assertions:

1. `PIC-5: there are exactly five checks; no probes beyond them.` — the faithful restatement of the spec invariant (count of probe sub-steps).
2. `PIC-5: FACTORY_PROBABLE_CAPABILITIES enumerates exactly the five factory-probable capability identifiers (inventory items 1/2/3/4/6) and nothing else, pinning the symbol V18a/V18c import.` — a different obligation about the cardinality of an exported constant, tracking the capability-inventory partition rather than the probe-step count.

The two bullets even refer to two different "fives": the five probe sub-steps `(a)`–`(e)` versus the five factory-probable capability identifiers (which themselves contribute nine probed function members per spec step `(c)`). A test implementer reading the leaf cannot tell which bullet is the normative `PIC-5` obligation and which is a derived consistency check, and the shared anchor implies a single spec source where there are in fact two different obligations (one anchored at `pic-5`, the other derived from the capability-inventory partition that `V18a`/`V18c` reconcile).

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** `b9de3eec5baa05af6d092602abbcafbb0fc0318a` (2026-06-11)
**History:** The first `PIC-5` Tests bullet (`there are exactly five checks; no probes beyond them`) is present at plan inception in `c6a664e` (2026-06-10) and faithfully restates the spec invariant. The second `PIC-5` bullet — the `FACTORY_PROBABLE_CAPABILITIES` enumeration assertion — was added in `b9de3ee` (2026-06-11), the resolution of "V18a partition assertion imports undeclared V9a probe constant," which introduced the exported constant and reused the existing `PIC-5` label for the new derived assertion in both `V9a-capability-probe.md` and `V9a-T-capability-probe.md`, creating the duplicate-label collision.

## Solution Space

**Shape:** single

### Recommendation

In both `docs/plan_topics/V9a-capability-probe.md` and `docs/plan_topics/V9a-T-capability-probe.md`, retain the first Tests bullet (`PIC-5: there are exactly five checks; no probes beyond them.`) as the sole bullet carrying the `PIC-5` anchor. Re-label the second bullet so it no longer claims the `PIC-5` anchor — annotate it as derived from `PIC-5` together with the capability-inventory partition, e.g. prefix it `PIC-5 (derived from capability inventory):` or `FACTORY_PROBABLE_CAPABILITIES (derived from PIC-5 + V18a inventory partition):`. Keep the bullet's assertion text intact — the `FACTORY_PROBABLE_CAPABILITIES`-cardinality test is a real obligation pinning the symbol `V18a`/`V18c` import; only its label changes. Apply the identical relabel to both leaves so the implementation leaf and its `-T` partner stay in sync.

The spec (`capability-probe.md`) is read-only for this fix: `PIC-5` is correctly defined there as the single five-checks invariant, and the derived constant-cardinality obligation has no separate spec anchor — the relabel is internal to the two plan leaves and does not require a spec edit.

## Consequence

**Severity:** advisory

A test implementer authoring the `V9a-T` suite sees two bullets under the same `PIC-5` anchor and must guess which is the canonical spec obligation; both assertions are legitimate tests, so the leaf still yields working code, but the mislabel invites a future maintainer to treat the derived constant-cardinality check as the spec invariant and edit one when the spec change touches the other.
