## Finding

The `bind_context: session` truncation walk sources its message list from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages` (binder-model-and-context.md, Session-context truncation), and the *User-arguments line* in `binder-bypass-and-envelope.md` item 5 separately carries the raw slash text into the binder's system prompt. The spec never pins whether the user-message carrying the triggering slash command is present in `getEntries()` at the moment the slash-command handler enters and runs the walk. This silence is doubly observable: it changes the binder's prompt bytes (the slash text appears once via the User-arguments line, or twice ‚Äî once there and once as the trailing `[user]: /<name> ‚Ä¶` turn of the rendered transcript), and it changes which turns count toward the 8000-token / 20-turn caps.

Two consequences follow. First, the [BNDR-7](../spec_topics/binder/binder-model-and-context.md#bndr-7) reproduce-exactly contract and the input-reproducibility contract one paragraph above it (*Why the transcript bytes are pinned*) become under-specified: conformance fixtures that construct a session ending in `/<name> ‚Ä¶` and assert byte-identical binder input cannot be written without first choosing an answer, and two reasonable implementations driven against the production Pi runtime can produce divergent binder inputs (and thus divergent binder outputs and downstream argument bindings) for the same observable session. Second, this is a host-behavioural question ‚Äî whether Pi has committed the inbound slash-trigger user message to `sessionManager` before invoking the registered `handler` is a property of `@earendil-works/pi-coding-agent`'s slash dispatch path, not of loom ‚Äî and the spec already pins a sibling property in the same shape (*Driven-turn session-commit ordering (consumption posture)*, `conversation-drive.md`); the symmetric slash-handler-entry case is missing.

The User-arguments line already carries the slash text verbatim, which strongly suggests the intended semantics is exclusion (the truncation walk grounds anaphora against *prior* context, not the trigger), but that intent is not pinned and the Pi-side ordering it depends on is not inventoried under Host prerequisites.

## Solution Space

**Shape:** single

### Recommendation

Pin exclusion of the triggering slash turn in the *Session-context truncation* paragraph, and inventory the supporting Pi behaviour as a named host-behaviour presupposition in the same edit. Concretely:

1. **Spec rule (binder-model-and-context.md, Session-context truncation).** Add a sentence to the existing paragraph immediately after the `buildSessionContext(...).messages` source clause, stating: *"At slash-invocation entry the triggering slash-command user message ‚Äî the message whose text drives the User-arguments line of [`binder-bypass-and-envelope.md` item 5](../binder/binder-bypass-and-envelope.md#system-prompt-structure-normative) ‚Äî is NOT included in the walked snapshot: `ctx.sessionManager.getEntries()` read at handler entry MUST NOT yet contain it, so the newest turn the walk evaluates is the one preceding the slash invocation. The slash text reaches the binder exactly once, via the User-arguments line. A session whose only entries are the just-arrived slash trigger yields zero included turns and no `Recent session context` block (per [BNDR-7i](#bndr-7i))."*

2. **Host prerequisite (host-prerequisites.md).** Add a new presupposition paragraph keyed off an anchor like `#slash-trigger-turn-commit-ordering-presupposition`, modelled on the *Driven-turn session-commit ordering (consumption posture)* paragraph in `conversation-drive.md`, stating that for a registered slash-command handler the inbound slash-trigger `UserMessage` is committed to the `ReadonlySessionManager` surface only **after** the handler returns (i.e. is absent from `ctx.sessionManager.getEntries()` throughout the handler's synchronous prefix), and noting that `@earendil-works/pi-coding-agent` encodes no typed contract pinning the ordering, so a Pi minor that committed the slash turn before invoking the handler would silently double-render the slash text in binder input with no SDK surface-inventory signal.

3. **Bump checklist (version-bump-step2.md).** Append a new lettered item (next free letter in the *Editorial-review checklist for unpinned host presuppositions* list) routing detection to editorial review on each Pi minor bump, with the same structure as items (ac) and (ak).

4. **Plan (V11i, V11i-T).** Extend the *Tests* / *Ships when* clauses with one fixture: a `FakeSessionManager` whose `getEntries()` is configured to mirror the presupposition (the slash-trigger user message absent), asserting the walk's newest evaluated turn is the one preceding the slash invocation and that a session containing only the slash trigger yields zero included turns.

Edge cases the implementer must watch: (a) the [`/lookup foo` ‚Üí `loom-system-note` ‚Üí `try again`](../spec_topics/binder/binder-model-and-context.md#bndr-7c) fixture (BNDR-7c) already shows a turn whose user line is `/lookup foo`, which is a *past* slash invocation visible in the session's historical record ‚Äî this is correctly included (the new rule excludes only the *currently-triggering* slash turn, not past ones); (b) the [BNDR-9](../spec_topics/binder/binder-model-and-context.md#bndr-9) `customType` unsafe-rejection path runs before transcript construction, unchanged; (c) BNDR-10's `mode: subagent` skip is unchanged (the walk is bypassed wholesale, so the inclusion question never arises).

## Consequence

**Severity:** correctness

Two reasonable implementations of the truncation walk diverge on a Pi-rendered session whose newest entry is the triggering slash-command user message: one renders the slash text twice in binder input (User-arguments line + transcript tail), the other once. The byte-identical binder-input contract that BNDR-7 and the *Why the transcript bytes are pinned* paragraph rest on cannot be exercised against the production Pi runtime until the answer is chosen, and authors of `bind_context: session` looms cannot predict which turns occupy the 20-turn / 8000-token budget.
