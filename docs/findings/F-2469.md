## Finding

PIC-50 (Prompt-mode synchronous-throw transport mapping) in `conversation-drive.md` maps a synchronous throw from `pi.sendUserMessage` to `Err(QueryError { kind: "transport", message: <error.message>, http_status: null, provider: <provider>, retryable: false })`. The `message` field is sourced by a bare `.message` read on the caught thrown value. JavaScript lets any value be thrown — a string, `null`, `undefined`, or a plain object with no `message` property — and for every such non-Error throw `<error.message>` evaluates to `undefined` (or, for a `null`/`undefined` throw, a `TypeError` on the property access). PIC-50 names no coercion and no fallback for that case, so two conforming implementations diverge on what `message` carries for a non-Error throw, and a conformance author has no expected value to assert against.

Every other site in the corpus that surfaces a caught throw's message routes it through the single normative **Underlying-error coercion** rule (`placeholder-rendering-b.md` §6, anchor `#underlying-error-coercion`): the `.message` when the throw is an object with a string `.message`, otherwise `String(error)`, falling back to the literal `"<unreadable>"` if the coercion itself throws — guaranteeing a deterministic non-null string. That coercion is cited by `extension-bootstrap-and-per-loom.md` (`details.error`), `code-registry-load.md` (`loom/load/extension-bootstrap-failed` and `loom/load/host-incompatible` `details.cause`), `capability-probe.md` (self-failure `details.cause`), `code-registry-runtime.md` (the `<error.message>`/`<error>` placeholders), `code-registry-host.md` (`session-shutdown-teardown-step-failed` `details.error`), and `host-interfaces-core.md` (`CodeToolError.message`), each explicitly noting it exists "so a non-Error throw yields a deterministic payload rather than `undefined`". PIC-50 is the lone transport-failure site that still reads `.message` raw.

The gap is load-bearing rather than cosmetic because `transport` is a member of the always-log set (group A) in `runtime-event-channel.md`. A group-A failure constructs a `RuntimeEvent` whose `message` field is typed as a non-optional `string`, and the `(kind, query_site, message, occurred_at)` deduplication tuple keys on that `message`. A synthesised `TransportError.message` of `undefined` therefore propagates an `undefined` into a field the channel requires to be a string and into the dedup key, corrupting the exactly-once emission contract for the non-Error-throw case.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `bf85092` (2026-06-06, "resolve non-Error-throw coercion + session_start pi.getCommands() failure contract") in interaction with the pre-existing synchronous-throw transport mapping (present at least since `31ff060`, 2026-05-08).
**History:** The synchronous-throw transport mapping has carried the raw `message: <error.message>` read since before the spec was relocated into `docs/` (`31ff060`, 2026-05-08) — predating the coercion convention by roughly a month. Commit `bf85092` (2026-06-06) introduced the `#underlying-error-coercion` rule in `placeholder-rendering-b.md` §6 and the deterministic-payload convention, and subsequent edits propagated explicit citations of it to the sibling diagnostic sites (extension-bootstrap, host-incompatible/`details.cause`, capability-probe self-failure, code-registry-runtime, `CodeToolError`). That convention-introduction did not retrofit the pre-existing prompt-mode transport synchronous-throw read. Later commits that touched the mapping — `dd09e20` (2026-06-25, RFC-2119 force on PIC-50/51) and `6f38d17` (2026-06-25, provider derivation) — formalised it as the named PIC-50 rule while preserving the raw `.message` read. The defect is therefore the interaction between the long-standing raw read and the later corpus-wide coercion convention that left this one site behind, not a single introducing edit.

## Solution Space

**Shape:** single

### Recommendation

In PIC-50, replace the raw `message: <error.message>` with the throw coerced through the [underlying-error coercion](../diagnostics/placeholder-rendering-b.md#underlying-error-coercion), worded identically to the sibling sites (e.g. `host-interfaces-core.md`'s `CodeToolError` and `extension-bootstrap-and-per-loom.md`'s `details.error`): the synthesised `message` is the caught throw's underlying-error string — the `.message` when the throw is an object with a string `.message`, otherwise `String(error)`, falling back to `"<unreadable>"` — so a non-Error throw (including a `null`/`undefined` throw) yields a deterministic non-null string rather than `undefined` or a synchronous `TypeError`. Add a non-Error-throw boundary acceptance criterion to PIC-50's mapping (and mirror it in V9n / V9n-T): driving a synchronous non-Error throw from `pi.sendUserMessage` produces a `kind: "transport"` `QueryError` whose `message` equals the coerced string, and the resulting group-A `RuntimeEvent.message` is that same non-null string. This is the same shape PIC-51 already uses for its absent-`errorMessage` case (the fixed `"provider transport failure"` fallback), so the two prompt-mode transport sub-rules become symmetric on their respective degenerate inputs.

## Consequence

**Severity:** correctness

For a non-Error synchronous throw from `pi.sendUserMessage`, two conforming implementations produce different `TransportError.message` values (one `undefined`, another whatever ad-hoc coercion it chooses), and the `undefined` flows into the always-log channel's non-optional `RuntimeEvent.message` field and its dedup key. The Error-throw happy path is fully specified, so implementation can proceed; only the non-Error edge diverges and is untestable.
