## Finding

`version-bump-step2.md` is the file for step 2 of the Pi-version-bump procedure, and it contains two independent letter-keyed enumerations:

1. An *editorial-review checklist* of host presuppositions with items `(a)`–`(ak)`, where item `(a)` is the extension-instance-survival open question.
2. A numbered sub-section "2. Re-run the SDK surface-inventory tests" with its own internal sub-items **(a)** "Positive direction — literal-read of the imported namespace" and **(b)** "Negative direction — inventory-closure audit over `src/`".

The phrase **"step 2(a)"** is used 32 times on the page as the escalation anchor in every SHOULD-level item — e.g. *"escalates to a build-time pin (on the same footing as the seven-capability literal-read assertion under step 2(a) above)"* — and is also the citation form used from `version-bump-step2b.md`, `version-bump-triggers.md`, `audit-wire-and-canary.md`, the `V18c` plan leaves, and `V18d-version-bump-acceptance.md`. Because the file *is* step 2, "step 2(a)" parses just as naturally as "step 2's checklist item (a)" — pointing at the extension-instance-survival audit — as it does as "step 2, sub-section 2, sub-item (a)" — pointing at the literal-read assertion. Surrounding prose (`"the seven-capability literal-read assertion under …"`) usually disambiguates, but the bare token "step 2(a)" appears in plan-leaf prose, in commit-message templates, and in cross-page citations where the disambiguating context does not travel with it.

The collision compounds the existing ambiguity that item `(a)` of the editorial-review checklist itself is non-trivial (the unresolved Pi-lifecycle open question on which `V9l` is gated), so a contributor reading `V9l`'s blocked-leaf note and following `step2.md#bump-checklist-instance-survival` may reasonably read a downstream "step 2(a)" reference as pointing back to that anchor rather than at the surface-inventory assertion.

## Solution Space

**Shape:** single
**State:** shaped

### Recommendation

Retire the structurally ambiguous "step 2(a)" / "step 2(b)" notation rather than re-spelling it. Keep the internal `(a)` / `(b)` sub-item labels (they read naturally inside their numbered sub-step), and at the ~32 escalation sites replace the bare token "step 2(a)" with the self-contained descriptive name "the SDK surface-inventory literal-read assertion" and "step 2(b)" with "the inventory-closure audit". This addresses the *interaction* of the two letter-keyed lists at its root — the descriptive phrase is self-describing in any context (plan-leaf prose, commit-message guidance, cross-page citation) without loading the step-2 file.

Spec edits:
- Replace the recurring phrase across the ~32 escalation sites in `version-bump-step2.md`, keeping an explicit link (`[the SDK surface-inventory literal-read assertion](#bump-step-2-positive)`) for navigability; keep the `#bump-step-2-positive` / `#bump-step-2-negative` anchors so existing inbound links survive.
- Apply the same convention to the cross-citing pages (`version-bump-step2b.md`, `version-bump-triggers.md`, `audit-recognised-shapes.md`, `audit-wire-and-canary.md`) and the `V18c` / `V18d` plan leaves.
- Pick the canonical replacement phrases up front (literally "the SDK surface-inventory literal-read assertion" and "the inventory-closure audit") so no third synonym creeps in.
- Run a final `rg -P 'step[- ]?2\([ab]\)'` sweep across `docs/spec_topics/` and `docs/plan_topics/` to confirm no stragglers remain.


## Consequence

**Severity:** correctness

A contributor reading an isolated "step 2(a)" reference (in a downstream page, in a plan leaf, or in commit-message guidance) can plausibly resolve it to either the editorial-review extension-instance-survival item or the surface-inventory literal-read assertion. The two have completely different audit shapes — a free-form editorial determination recorded in the bump commit message vs. an `npm test`-gated assertion — and routing a fix to the wrong one leaves the actual gate unfixed while looking conformant on the diff.
