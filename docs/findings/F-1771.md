## Finding

`docs/spec_topics/expressions.md` §"Grammar disambiguation" states that "these positions require parentheses around any constructor" and gives the example `if (Author { name: "x", role: "r", experience_years: 0 } == author) { ... }`. The prose and the example describe two different rules and a parser implementer cannot tell which is normative:

- **Rule A (wrap the whole condition).** Whenever a scrutinee contains a constructor, the entire scrutinee expression must be parenthesised. The example satisfies this — the outer parens enclose the whole `Author { ... } == author` comparison. Under A, the bare form `if Author { ... } == author { ... }` is rejected and the directly-parenthesised form `if (Author { ... }) == author { ... }` is also rejected (the constructor sits in the scrutinee but the scrutinee as a whole is not parenthesised).
- **Rule B (wrap each constructor).** The constructor itself must be directly enclosed by parens, so that its `{` cannot collide with the body `{`. Under B, the minimal legal form is `if (Author { ... }) == author { ... }`; the form `if a == (Other { ... }) { ... }` is admitted; the example's wider parens are over-parenthesisation, not the rule the prose names.

The grammar production `IfStmt ::= "if" Expr StmtBlock` in `docs/spec_topics/grammar.md` §"Block expressions" admits a bare `Expr` for the condition (the lexical-structure prose only outlaws single-line bodies via `loom/parse/single-line-if`, not bare conditions), and `docs/spec_topics/control-flow.md`'s own example writes `if author.experience_years < 2 { ... }` with no outer parens. That demonstrates a bare condition is legal in general — so the scrutinee rule is an *added* requirement that fires only when a constructor is present, and the spec must say whether it fires at the whole-condition level (A) or at each constructor (B).

Two implementers will diverge: one writes the parser to demand an outer paren-expression directly under `if` / `while` / `match` / `for` whenever any descendant is a constructor (A); the other writes it to demand a paren-expression directly enclosing each constructor token in scrutinee position (B). Both are defensible readings of the current text; neither matches the example without leftover oddities.

## Solution Space

**Shape:** multiple

### Option A — Wrap the whole scrutinee

**Approach.** State that whenever any constructor appears anywhere in the condition of `if` / `while`, the scrutinee of `match`, or the iterated expression of `for`, the entire scrutinee expression must be a directly parenthesised expression: `(<scrutinee>)`. Bare conditions remain admissible when no constructor is present (`if x < 2 { ... }` is fine; `if Author { ... } == author { ... }` and `if (Author { ... }) == author { ... }` are both `loom/parse/...`).

**Spec edits.** In `expressions.md` §"Grammar disambiguation", replace the second sentence with: "These positions therefore require the entire scrutinee expression to be directly parenthesised whenever it contains a constructor: `if (Author { name: "x", role: "r", experience_years: 0 } == author) { ... }`." Keep the existing example as-is; it already satisfies this rule.

**Pros.** Matches the existing example verbatim. Simplest parser rule — the parser, on encountering `Schema {` while parsing a scrutinee `Expr`, can demand the *enclosing* scrutinee start with `(`. Easiest for authors to remember ("if your condition mentions a constructor, wrap the whole condition").

**Cons.** More verbose than necessary in the common case `if x == (Other { ... })` — the constructor is already syntactically isolated by the inner parens but the rule still demands outer ones. Mildly redundant when the scrutinee is `(Other { ... })` (the constructor wrap and the scrutinee wrap collapse into one set of parens, but the rule conceptually requires two).

**Risks.** Authors who write `if a == (Other { ... }) { ... }` reasonably and find it rejected may file a defect; the diagnostic message must spell out the whole-scrutinee-wrap rule, not just say "wrap the constructor".

### Option B — Wrap each constructor directly

**Approach.** State that a constructor that appears anywhere inside a scrutinee position must be directly enclosed by parens: `(Schema { ... })`. The scrutinee as a whole need not be parenthesised; bare conditions remain admissible (`if x < 2 { ... }`, `if a == (Other { ... }) { ... }`).

**Spec edits.** In `expressions.md` §"Grammar disambiguation", change the rule sentence to: "Every constructor that appears in such a position must be directly enclosed by parens: `(Schema { ... })`." Change the example to a minimally-parenthesised form: `if (Author { name: "x", role: "r", experience_years: 0 }) == author { ... }`. Add a worded clarification that an outer paren around the whole condition is not required and does not by itself satisfy the rule for a constructor it merely contains.

**Pros.** Matches the prose ("parentheses around any constructor") literally. More permissive — admits `if a == (Other { ... }) { ... }` without forcing redundant outer parens. The scrutinee-rule trigger is local to each constructor, so the parser only needs to look at the constructor token, not at the enclosing scrutinee.

**Cons.** The current example is over-parenthesised under this rule and would need to be rewritten. Slightly more work for authors to learn ("each constructor wrapped" vs "whole condition wrapped"). The current prose is technically already this rule, but readers consistently interpret it as A because of the example.

**Risks.** Some authors will reflexively wrap the whole condition (the way they would in Rust or C); the diagnostic for an unparenthesised constructor inside an otherwise-parenthesised condition should be specific enough that the fix is obvious.

### Recommendation

Adopt **Option B**. The current prose already names this rule ("parentheses around any constructor"); the defect is solely that the example demonstrates a different, wider parenthesisation that the rule does not require. Rewrite the example to `if (Author { name: "x", role: "r", experience_years: 0 }) == author { ... }` and add one sentence stating that an outer condition-wrap does not by itself satisfy the rule for a constructor it merely contains.

Edge cases the implementer must watch: (i) nested constructors (`Outer { inner: (Inner { ... }) }` in scrutinee position — only the *outermost* constructor of the scrutinee subterm needs the parens, since the inner constructor's braces are already inside the outer constructor's body and not in body-brace-collision position); (ii) constructors inside `${...}` template interpolations within a scrutinee `@`…`` query (the interpolation's own `{` boundary already disambiguates and no extra parens are required); (iii) constructors inside a `()` call-argument list within a scrutinee (the call's parens already disambiguate; the rule does not double-fire). The new diagnostic for a bare constructor in scrutinee position should reuse or sit beside the existing `loom/parse/single-line-if` neighbourhood in `code-registry-parse.md`, naming the construct ("constructor in `if` / `while` / `match` / `for` scrutinee must be parenthesised").

## Consequence

**Severity:** correctness

A parser written to rule A will reject inputs a parser written to rule B accepts (and vice-versa, for the directly-parenthesised-only form). The disagreement is mechanical and silent — the same source file produces different parse results on different conforming implementations, and the spec offers no diagnostic code to disambiguate.
