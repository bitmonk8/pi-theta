## Finding

PIC-43 (subagent mode) resolves an untyped query from the terminal (`willRetry: false`) `agent_end` event by applying two short-circuits before extraction: a cancellation short-circuit, then a transport-failure short-circuit "that reads the trailing `assistant` message's `stopReason` from the same terminal `agent_end` `messages` array ŌĆ” and maps a `stopReason` of `"error"` to `Err(QueryError { kind: "transport", ŌĆ” })`". The downstream assistant-text extraction explicitly handles the case where that array has no usable assistant content ŌĆö "a final turn with no assistant text yields the empty string" ŌĆö but the transport short-circuit immediately above it has no matching disposition for the same void. When the terminal `agent_end` `messages` array ends in a tool or user message, or is empty, there is no "trailing `assistant` message" to read a `stopReason` from, and the short-circuit's behaviour is left undefined.

This is not a hypothetical input. The extraction rule the same paragraph relies on contemplates "a pure tool-use turn" with no final assistant message, so a terminal `agent_end` lacking a trailing assistant message is a case the spec itself admits is reachable. With no stated disposition, two reasonable implementers diverge: one reads `messages[last].stopReason` literally and, finding `undefined ŌēĀ "error"`, falls through to extraction (empty string); another filters to the trailing *assistant* message, finds none, and dereferences `null`/`undefined`, throwing ŌĆö which routes to `loom/runtime/internal-error` rather than a clean `Ok("")`. The same gap exists symmetrically on the prompt-mode side: PIC-51's error-detection probe "reads that turn's final `assistant` message ŌĆ” and inspects its `stopReason`", while PIC-53's paired extraction yields the empty string for a turn that produced no assistant text. Both transport short-circuits are silent about the no-trailing-assistant case while their paired extraction branches define it.

## Solution Space

**Shape:** single

### Recommendation

In PIC-43 (`subagent.md`), state that when the terminal `agent_end` `messages` array has no trailing `assistant` message ŌĆö it ends in a tool/user message, or is empty ŌĆö the transport-failure short-circuit does not fire (no `kind: "transport"` error is synthesised) and the runtime proceeds to the assistant-text extraction rule, which already yields the empty string. Mirror the identical disposition in PIC-51 (`conversation-drive.md`) so the two paired transport short-circuits stay symmetric with their paired extraction branches.

Phrase it as the absence of a readable `stopReason`: the short-circuit keys only on a trailing `assistant` message whose `stopReason === "error"`; any terminal turn lacking a trailing `assistant` message is, by definition, not a transport failure and falls through to extraction. Edge cases the implementer must cover: a terminal `agent_end` whose `messages` is empty, and one whose last element is a `toolResult`/`user`/`custom` message ŌĆö both must reach `Ok("")`, not an internal-error, and the disposition must read the *trailing assistant* message (not the literal last array element) when one exists.

## Consequence

**Severity:** correctness

A subagent (or prompt-mode) query whose terminal turn ends in a tool/user message or is empty has no defined transport-short-circuit behaviour. One implementer falls through cleanly to the empty-string extraction; another dereferences a missing assistant message and surfaces `loom/runtime/internal-error` instead of `Ok("")` ŌĆö divergent observable results on a reachable error path the paired extraction branch already handles.
