## Finding

`tool-registration-lifetime.md` establishes that Pi's tool registry has no `unregisterTool` API: `pi.registerTool` is the only mutation, registrations persist for the process lifetime, and tool *visibility* is gated solely through the active-set snapshot/restore window rather than through a register/unregister cycle. The whole snapshot/restore design (PIC-17, PIC-8, PIC-19) is load-bearing precisely because unregistration is unavailable; "the cached `pi.registerTool` entries themselves are intentionally permanent."

`V9f`'s `Adds` names "the no-`unregisterTool` rule" as one of the leaf's responsibilities, but its `Tests.` and `Ships when.` cover only the PIC-8 restore-failure path, the PIC-19 snapshot/swap-install path, callable-set visibility plus `registration-cache-collision`, and the `ToolDefinition.label` derivation. The paired `V9f-T` mirrors exactly that set. No test or check asserts that the runtime never depends on or invokes tool-unregistration. The permanence-based visibility design therefore rests on a MUST-NOT invariant that nothing observably enforces; a future change that reached for an unregistration semantic (or, if a later Pi minor adds `pi.unregisterTool`, wired it in prematurely) would red no V9f gate.

## Solution Space

**Shape:** single

### Recommendation

Add a failing test to `V9f-T`'s `Tests.` that asserts the runtime never invokes tool-unregistration across the per-loom registration lifecycle, and add a matching arm to `V9f`'s `Tests.` and a clause to its `Ships when.` so the gate fires. Because the loom 1.0 Pi-SDK pin exposes no `unregisterTool` member, the natural realisation is a source-level assertion that no `unregisterTool` reference appears in the loom runtime registration path (rather than a behavioural spy on a non-existent method); the invariant being asserted is "tool registrations are permanent â€” visibility is achieved only through the active-set snapshot/restore window, never through unregistration."

Edge cases for the implementer: the assertion must scope to the loom runtime's own registration/visibility code, not flag the spec-prose mention of a hypothetical future `pi.unregisterTool` in comments or docs; and it complements (does not replace) the existing visibility test ("the callable set's visibility tracks the invocation"), which exercises the positive active-set mechanism.

## Consequence

**Severity:** advisory

The permanence-of-registration / no-unregistration invariant that PIC-17/PIC-8/PIC-19 visibility gating depends on has no regression guard, so a later edit that introduced an unregistration call or relied on unregistration semantics would ship without reddening any V9f gate. At the loom 1.0 Pi-SDK pin the immediate exposure is bounded (the SDK exposes no `unregisterTool` member to call), so this is a guard against future drift rather than a live divergence.
