## Finding

In subagent mode, PIC-43's transport-failure short-circuit (`subagent.md`) maps `stopReason: "error"` to `Err(QueryError { kind: "transport", …, provider: <resolved-model-provider>, … })`. The `<resolved-model-provider>` placeholder is never expanded locally, and PIC-43 backs it only with "the same shape as the prompt-mode mapping at [prompt-mode error detection]". Both crutches fail an implementer reading PIC-43:

- **The placeholder name invites the wrong field.** `host-interfaces-core.md` (model-registry-pin) draws an explicit distinction between `Model<Api>.provider` (typed `Provider = KnownProvider | string`, the short id form such as `anthropic` / `openai`) and `Model<Api>.api` (the api-shaped value such as `anthropic-messages` / `openai-completions`). The canonical contract in `queryerror-variants.md` (the `provider` derivation paragraph) requires the `TransportError.provider` field to carry the **`.api`-shaped** value — the same field the [Provider error mapping](docs/spec_topics/pi-integration-contract/provider-error-mapping.md) table is keyed on — explicitly *not* the short `.provider` form. A reader who takes `<resolved-model-provider>` at face value extracts `Model.provider` (`anthropic`), producing a value outside the documented value space that any author `match`-ing on `provider` will fail to match.

- **The "same as prompt-mode" link points at the wrong model source.** PIC-50/PIC-51 derive `provider` from `ctx.model.api` — the *user session's* currently-selected model — and substitute the sentinel `"unknown"` when `ctx.model` is `undefined`. That is the correct source for prompt mode (the driven turn runs on the user session) but the *wrong* source for subagent mode, whose spawned session runs the loom-resolved `model`, not `ctx.model`. "Same shape as prompt-mode" is true only of the `QueryError` struct shape; it does not (and must not) carry the model-source derivation.

The derivation itself is not missing from the corpus: `queryerror-variants.md`'s `provider`-derivation paragraph already states that surfaces whose model loom resolved — including "the subagent-mode driven turn, whose spawned session runs the loom-resolved model" — take **that resolved model's `.api`**, while prompt-mode driven-turn failures take the user session's `ctx.model.api`. The defect is purely that PIC-43 does not cite that owner and uses a placeholder/cross-reference pair that actively steers an implementer toward the wrong field (`.provider` vs `.api`) and the wrong model (`ctx.model` vs the resolved model).

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `222aacd` (2026-06-23, "fix F-1227 — subagent untyped-query transport-failure short-circuit (+ version-bump item m)"); `6f38d17` (2026-06-25, "fix F-1145 — derive prompt-mode TransportError.provider from ctx.model + sentinel")
**History:** The subagent transport-failure short-circuit and its bare `provider: <resolved-model-provider>` placeholder entered `subagent.md` in `222aacd` (the token's only introducing commit per `git log -S`). At that time `queryerror-variants.md` carried only a generic `provider` value-space (added `43bef30`, 2026-06-05) with no per-surface source clause, so the placeholder had no owner to cite. `6f38d17` then authored the per-surface derivation in `queryerror-variants.md` — distinguishing the loom-resolved-model surfaces (including the subagent driven turn → resolved model `.api`) from the prompt-mode user-session source (`ctx.model.api`) — but did not back-link `subagent.md` PIC-43 to it or expand the placeholder. The defect is the interaction: the placeholder authored before the owning text existed was never rewired once `6f38d17` created that owner, leaving PIC-43's local crutches ("`<resolved-model-provider>`" + "same as prompt-mode") pointing away from the now-authoritative derivation.

## Solution Space

**Shape:** single

### Recommendation

In `subagent.md` PIC-43, replace the bare `provider: <resolved-model-provider>` placeholder and the standalone "same shape as the prompt-mode mapping" framing with an explicit derivation plus a cross-reference to the owner:

- State that the `provider` field carries the **`.api`-shaped value of the loom-resolved model** that the spawned session runs (the `model` resolved per PIC-40), i.e. the resolved `Model<Api>.api`, citing the `provider`-derivation paragraph in `queryerror-variants.md` as the single owner of the value space and the api-shaped-vs-short-`.provider` distinction.
- Scope the "same shape as prompt-mode" comparison to the `QueryError` *struct shape only*, and note that the *model source* differs: prompt mode attributes from the user session's `ctx.model.api` (PIC-50/PIC-51), whereas subagent mode attributes from the resolved model the spawned session runs — there is no `ctx.model`/`"unknown"` sentinel path here because the [pre-spawn model guard](docs/spec_topics/pi-integration-contract/subagent.md) (PIC-40, `loom/runtime/subagent-model-unresolved`) already guarantees a resolved model before the spawn.

Edge case the implementer must watch: when frontmatter `model:` is absent the resolved model is the inherited `ctx.model` (a `Model<any>`), but its `.api` field is still the attribution source — the inheritance affects which model handle is used, not the field read off it.

## Consequence

**Severity:** correctness

`TransportError.provider` is an observable error field authors `match` against and the runtime-event channel always-logs. Two implementers reading PIC-43 alone diverge: one emits the short `Model.provider` form (`anthropic`) per the placeholder name, another follows "same as prompt-mode" and attributes via the user-session `ctx.model` rather than the spawned session's resolved model. Either deviation puts the field outside the api-shaped value space the Provider error mapping table and author `match` arms expect, silently mis-routing subagent transport failures.
