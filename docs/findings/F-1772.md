## Problem

PIC-43's subagent transport-failure short-circuit (`subagent.md`, anchor `pic-43`) maps `stopReason: "error"` to `Err(QueryError { kind: "transport", …, provider: <resolved-model-provider>, … })`. The `<resolved-model-provider>` placeholder is never expanded and is backed only by "the same shape as the prompt-mode mapping", and both crutches steer an implementer wrong: the placeholder name invites the short `.provider`-id form (e.g. `anthropic`) when the `provider`-derivation owner requires the `.api`-shaped value (e.g. `anthropic-messages`), and "same as prompt-mode" implies the user-session `ctx.model` source when subagent mode must attribute from the spawned session's loom-resolved model. The authoritative per-surface derivation already exists in `queryerror-variants.md`'s `provider`-derivation paragraph, but PIC-43 neither cites it nor expands the placeholder.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `222aacd` (2026-06-23, "fix F-1227 — subagent untyped-query transport-failure short-circuit (+ version-bump item m)"); `6f38d17` (2026-06-25, "fix F-1145 — derive prompt-mode TransportError.provider from ctx.model + sentinel")
**History:** The subagent transport-failure short-circuit and its bare `provider: <resolved-model-provider>` placeholder entered `subagent.md` in `222aacd` (the token's only introducing commit per `git log -S`). At that time `queryerror-variants.md` carried only a generic `provider` value-space (added `43bef30`, 2026-06-05) with no per-surface source clause, so the placeholder had no owner to cite. `6f38d17` then authored the per-surface derivation in `queryerror-variants.md` — distinguishing the loom-resolved-model surfaces (including the subagent driven turn → resolved model `.api`) from the prompt-mode user-session source (`ctx.model.api`) — but did not back-link `subagent.md` PIC-43 to it or expand the placeholder. The defect is the interaction: the placeholder authored before the owning text existed was never rewired once `6f38d17` created that owner, leaving PIC-43's local crutches ("`<resolved-model-provider>`" + "same as prompt-mode") pointing away from the now-authoritative derivation.

## Solution approach

Rewrite PIC-43's transport-failure short-circuit to expand the `provider` placeholder as the `.api`-shaped value of the loom-resolved model the spawned session runs (the `model` resolved per PIC-40), citing the `provider`-derivation paragraph in `errors-and-results/queryerror-variants.md` as the owner of that value space. Narrow the "same shape as prompt-mode" comparison to the `QueryError` struct shape only, and note that the model source differs from prompt mode's user-session `ctx.model.api` (PIC-50/PIC-51).

## Solution constraints

- Out of scope: the `provider` value-space definition (`.api`-shaped vs short `.provider`-id) owned by the `provider`-derivation paragraph in `errors-and-results/queryerror-variants.md` — cite it, do not restate or redefine it.
- Do not introduce a prompt-mode-style `ctx.model`/`"unknown"` sentinel path into the subagent mapping; PIC-40's pre-spawn model guard already guarantees a resolved model before the spawn.
