## Problem

Item 7 (*Split-row joint-completeness*) of the loom 1.0 release-time residue inspection in `conventions.md` sweeps only the *Numbered REQ-IDs (runtime obligations)* table of `coverage-matrix.md`; the *Code-keyed obligation areas (no numbered REQ-IDs)* table is never swept. That table contains genuine multi-facet split rows whose closing leaves each close a distinct facet — swallowing-handler attachment (`V17a`, `V14f`, `V13f`, `V15h`, `V9o`), extension-bootstrap SDK failures (`V9k`, `V9p`), and the IMPL static-resolution load pass (`V15a`, `V15e`, `V7a`, `V7d`). For these rows no mechanical gate enforces per-facet citing tests: H5d's transitive-completeness arm is a per-cell at-least-one check, H5a only checks the obligation is enumerated, and the code↔test parity gate either does not apply (swallowing-handler carries no diagnostic code) or accepts one test per shared code. A facet can therefore ship with no citing test while every gate stays green — e.g. `V9o`'s subagent-`AgentSession.abort()` suppression — which is exactly the *No silent test skipping* failure mode the plan forbids.

## Issue introduction

**Verdict:** single-commit

**Introducing commit:** `cda161b415a1dd0829e31ee74517a8ba85ab682c` — "pi-loom plan: resolve \"split REQ-IDs pass closing gate on one facet citing test\"" (2026-06-17)

**History:** Item 7 did not exist before `cda161b`; that commit created the *Split-row joint-completeness* checklist item, added the item-7 reader-input clause ("for item 7, `coverage-matrix.md` and the test corpus"), and added the REQ-ID-discipline sentence routing the split-row residue to it. `git show cda161b -- docs/plan_topics/conventions.md` shows the new item 7 opening with "Sweep every *Numbered REQ-IDs* row in `coverage-matrix.md`…" — the *Code-keyed obligation areas* table was outside its scope from inception. The defect is therefore present since the item's introduction, not a later regression: the item was authored to mirror the per-REQ-ID citing-test scan's numbered-row footing and never extended to the code-keyed surface, even though the swallowing-handler / bootstrap / IMPL split rows already lived in `coverage-matrix.md` at that commit.

## Solution approach

Broaden item 7's sweep clause in `conventions.md` so the *Code-keyed obligation areas (no numbered REQ-IDs)* table of `coverage-matrix.md` is swept alongside the *Numbered REQ-IDs* table. Add per-facet annotations to the three multi-leaf code-keyed rows (swallowing-handler attachment, extension-bootstrap SDK failures, IMPL static-resolution load pass) so each row's facet partition is well-defined for the sweep. Note that the swallowing-handler row carries no diagnostic code, so its five sites — including `V9o`'s subagent-abort suppression — are reachable by no other gate and depend entirely on this sweep.

## Solution constraints

- The facet-vs-co-witness partition-derivation rule item 7 reuses is owned by F-0319; the annotations and sweep authored here must agree with how F-0319 marks co-witness leaves.
