## Problem

The `parameters` bullet under **Per-loom registration** imposes a normative MUST on the `.loom`-callable adapter — AJV-validate `params` against the lowered JSON Schema before forwarding — without stating what the adapter does on AJV rejection, leaving the obligation with no defined failure surface. The only post-validation cross-reference in the paragraph points at *Schema-validation respond-repair*, which governs the typed-query **response** boundary, not the model-driven `tool_use` **args** boundary this MUST creates. Because the correct destination for this site (the model-driven row of `#ceiling-4-table`) is never connected, implementers will disagree on the failure shape — throw out of `execute()`, return `{ isError: true, ... }` directly, surface `InvokeInfraError`, or leak an unhandled rejection.

## Issue introduction

**Verdict:** single-commit-introduction
**Introducing commits:** `cb43deb` (2026-05-04, *pi-loom spec: resolve "ToolDefinition is mis-shaped: missing label, and parameters must be a TypeBox TSchema"*)
**History:** `git log -G "AJV-validate" --reverse` over `spec_topics/pi-integration-contract.md` and the later split file `docs/spec_topics/pi-integration-contract/extension-bootstrap-and-per-loom.md` shows `cb43deb` as the first commit that introduces the `parameters` field-derivation bullet and, inside it, the *"adapter MUST AJV-validate `params` before forwarding"* clause. The same commit added the misdirected cross-reference (then citing `prepareArguments?` and the V13k–m coercion flow; later edits redirected the pointer to *Schema-validation respond-repair* but never closed the failure-scope gap). Subsequent edits to the bullet (`13cf76d`, `708a42f`, `52219c4`, `f072903`) revised the dependency framing and the `prepareArguments?` sentence; none added a failure clause for the AJV-rejection path. The defect is co-resident with the obligation: the MUST was authored without an outcome and the gap has been preserved across every later revision.

## Solution approach

Clarify the adapter's AJV-rejection failure routing at the `parameters` bullet under **Per-loom registration**, naming the model-driven `tool_use` args row of `#ceiling-4-table` as the destination. Replace the misdirected *Schema-validation respond-repair* cross-reference with a forward-link to that row, and cross-reference `#cio-3` so the depth-walk-before-AJV precedence is reachable from this site.

## Solution constraints

- Out of scope: the code-driven `<name>(args)` / `invoke(...)` path that surfaces `InvokeInfraError { cause: "validation" }` per `docs/spec_topics/tool-calls.md` — Failures. The adapter-AJV failure being declared is exclusively the model-driven `tool_use` path.
- For any internal or non-`Error` throw inside the adapter's AJV step, reuse the existing `loom/runtime/internal-error` route per `docs/spec_topics/tool-calls.md` — Failures (*Pre-evaluation setup throw inside the `.loom`-callable adapter*); do not mint a new route.
