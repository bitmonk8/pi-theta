## Problem

In the System-notes fallback chain, the closing paragraph handling a throw from `ctx.ui.notify` says to "wrap it in the same try/catch and proceed to the diagnostic step." The phrase "the same try/catch" has no antecedent ‚Äî the fallback is an ordered prose list (step 1 `ctx.ui.notify`, step 2 the `system-note-delivery-failed` diagnostic, terminal `console.error`) and none of those steps is introduced as a wrapped `try`/`catch`. The only explicitly-named `try`/`catch` in the section is PIC-54's, which wraps the *terminal* `console.error` step, so a reader resolves the dangling reference to the wrong (terminal) step rather than to step 1. The intended outcome (swallow the `ctx.ui.notify` throw, advance to step 2) is recoverable only from the trailing clause, leaving the structural mechanism ambiguous.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `11b9199` (2026-05-04) ‚Üí `cfb3943` (2026-06-25)
**History:** the sentence "`ctx.ui.notify` itself can throw ‚Ä¶ wrap it in the same try/catch and proceed to the diagnostic step" entered the normative spec body in `11b9199` ("pi-loom spec: resolve \"pi.sendMessage failure leaves system notes silently dropped\""), which first specified the `loom-system-note` `sendMessage` fallback chain (then on the pre-split `pi-integration-contract.md`; the prior appearance in `b87d4a6`'s `docs/spec-review.md` was a proposed-fix draft, not normative text). At that point the fallback was a flat ordered list with no `try`/`catch` named anywhere, so "the same try/catch" was already a dangling reference with no antecedent ‚Äî but no competing referent existed. `cfb3943` ("fix F-1395 ‚Äî coin PIC-54 wrapping terminal `console.error`") then introduced **PIC-54**, the section's only explicitly-named `try`/`catch`, wrapping the terminal `console.error` step. This retroactively supplied a plausible-but-wrong antecedent for "the same try/catch," converting a latent dangling reference into an active mis-resolution hazard. The split commit `f5e89f4` carried the unchanged sentence into `runtime-event-channel.md`. The durable fix targets the phrase itself (give `ctx.ui.notify` its own guard), not either commit in isolation.

## Solution approach

Rewrite the closing `ctx.ui.notify` throw-handling sentence in the fallback chain so the step-1 `ctx.ui.notify` call gets its own explicit guard rather than referencing "the same try/catch". Name step 1 directly so the guard cannot be confused with `#pic-54`'s terminal-`console.error` wrap, and state the outcome: a throw from `ctx.ui.notify` is swallowed and the fallback proceeds to the step-2 `loom/runtime/system-note-delivery-failed` diagnostic.

## Solution constraints

- Leave step 1's `display: false` / `content: ""` skip rules unchanged ‚Äî they govern whether step 1 runs, independently of the throw-handling guard.
