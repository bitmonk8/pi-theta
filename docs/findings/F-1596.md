## Finding

In the System-notes fallback chain (`docs/spec_topics/pi-integration-contract/runtime-event-channel.md`), the closing paragraph states: "`ctx.ui.notify` itself can throw (e.g. in print mode where Pi's UI is not attached); wrap it in the same try/catch and proceed to the diagnostic step." The phrase **"the same try/catch"** is a dangling reference — nothing earlier in the section names a `try`/`catch` for it to corefer with. The fallback is described as an ordered prose list (step 1 `ctx.ui.notify`, step 2 the `system-note-delivery-failed` diagnostic, terminal `console.error`); none of those steps is introduced as a wrapped `try`/`catch` block.

The only explicitly-named `try`/`catch` in the entire section belongs to **PIC-54**, which wraps the *terminal* `console.error` step. A reader resolving "the same try/catch" to PIC-54's catch would attach the `ctx.ui.notify` throw-handling to the wrong (terminal) step rather than to step 1. The intended behaviour is recoverable only from the trailing clause "and proceed to the diagnostic step," which pins the outcome (swallow the `ctx.ui.notify` throw, advance to step 2) but leaves the structural mechanism ambiguous.

The fix is to give the `ctx.ui.notify` call its own guard with an explicit outcome, e.g. "wrap the `ctx.ui.notify` call in its own `try`/`catch`; on a throw, silently swallow it and proceed to the step-2 diagnostic." This removes the cross-reference to a non-existent antecedent and stops PIC-54's terminal-`console.error` catch from being mistaken for it.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `11b9199` (2026-05-04) → `cfb3943` (2026-06-25)
**History:** the sentence "`ctx.ui.notify` itself can throw … wrap it in the same try/catch and proceed to the diagnostic step" entered the normative spec body in `11b9199` ("pi-loom spec: resolve \"pi.sendMessage failure leaves system notes silently dropped\""), which first specified the `loom-system-note` `sendMessage` fallback chain (then on the pre-split `pi-integration-contract.md`; the prior appearance in `b87d4a6`'s `docs/spec-review.md` was a proposed-fix draft, not normative text). At that point the fallback was a flat ordered list with no `try`/`catch` named anywhere, so "the same try/catch" was already a dangling reference with no antecedent — but no competing referent existed. `cfb3943` ("fix F-1395 — coin PIC-54 wrapping terminal `console.error`") then introduced **PIC-54**, the section's only explicitly-named `try`/`catch`, wrapping the terminal `console.error` step. This retroactively supplied a plausible-but-wrong antecedent for "the same try/catch," converting a latent dangling reference into an active mis-resolution hazard. The split commit `f5e89f4` carried the unchanged sentence into `runtime-event-channel.md`. The durable fix targets the phrase itself (give `ctx.ui.notify` its own guard), not either commit in isolation.

## Solution Space

**Shape:** single

### Recommendation

Replace "wrap it in the same try/catch and proceed to the diagnostic step" with an explicit self-contained guard for step 1: state that the runtime wraps the `ctx.ui.notify` call in its **own** `try`/`catch`, and that a throw from it MUST be silently swallowed and the fallback MUST proceed to the step-2 `loom/runtime/system-note-delivery-failed` diagnostic. Do not reuse the words "the same try/catch"; name step 1 directly so the guard cannot be confused with PIC-54's terminal-`console.error` wrap. Keep the existing `display: false` / `content: ""` skip rules for step 1 unchanged — they govern whether step 1 runs at all, independently of the throw-handling that governs what happens when it does run and throws.

## Consequence

**Severity:** advisory

The behavioural outcome (swallow the `ctx.ui.notify` throw, advance to the diagnostic step) is pinned by the trailing clause, so two implementers are unlikely to diverge on runtime behaviour. The risk is a structural misreading — attaching the guard to PIC-54's terminal `console.error` catch instead of step 1 — that produces a correct-looking implementation whose `try`/`catch` placement does not match the spec's intent, and a fixture author with no clear referent for which step is guarded.
