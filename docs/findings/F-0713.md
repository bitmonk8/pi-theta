## Finding

PIC-10 (`Checkpoint` seam) requires that for the `loop-iter` checkpoint kind, `before(...)` "releases the event loop for one macrotask turn before resolving," and the parallel prose in `cancellation.md` §Granularity says "the runtime therefore yields one macrotask turn at each loop-iteration boundary." Neither location names the primitive that achieves the yield. In Node, the candidates are `setTimeout(fn, 0)`, `setImmediate(fn)`, and `await new Promise(r => Clock.setTimeout(r, 0))` — they are not interchangeable for either correctness or test ergonomics.

PIC-12 (`Clock` / `FakeClock` interface) then states that the runtime "MUST NOT call `Date.now`, `performance.now`, `Date.prototype.getTime`, or the global `setTimeout` / `clearTimeout` outside the `WallClock` adapter," with a non-normative grep-test enforcing exactly that surface. The most obvious primitive for the yield — a bare global `setTimeout(fn, 0)` at the `loop-iter` site — directly trips that ban. Routing the same call through `Clock.setTimeout(fn, 0)` complies with the ban but couples `loop-iter` semantics to the `FakeClock` advance-on-`advance(ms)` model rather than to a true macrotask boundary, which subtly changes what a `V8a` / `V17c` conformance test is asserting. `setImmediate(fn)`, the semantically-cleanest Node primitive for "one macrotask turn," is neither named nor enumerated on the grep-ban list, so it would escape the ban only by silence; the `Clock` seam does not expose it either.

The seam therefore mandates a behaviour ("one macrotask turn") that has no spec-sanctioned implementation. Two reasonable implementers will pick differently, with material consequences for both the production cancellation path and the FakeClock-driven `V8a` / `V17c` tests.

## Solution Space

**Shape:** single

### Recommendation

Pin the `loop-iter` macrotask-yield primitive to `Clock.setTimeout(fn, 0)` (i.e., the existing PIC-12 timer-seam surface) and add one sentence to PIC-10 stating that explicitly.

Edit PIC-10's production-wiring rule (the bullet that begins "Production wiring constructs one `Checkpoint` per `loomAbort`…") to replace "`before(...)` releases the event loop for one macrotask turn before resolving" with a concrete realisation:

> For the `loop-iter` kind, `before(...)` resolves on a macrotask turn by scheduling its resolver through the runtime's injected [`Clock.setTimeout(resolve, 0)`](#clock--fakeclock-interface) seam and awaiting the resulting promise — keeping the macrotask yield on the same DI surface as every other timer the runtime uses, so the PIC-12 timer-surface ban remains the runtime's only path to a host timer.

Add one sentence to PIC-12 (in the `FakeClock` paragraph) clarifying that a 0-ms timer registered through the seam is fired by `advance(0)` in registration order, so the `V8a` / `V17c` conformance tests assert the macrotask-yield property by issuing `clock.advance(0)` between the loom-side signal flip and the next loop-iteration signal-check rather than by manipulating any real-time queue. The existing FakeClock prose ("fires every timer whose deadline has elapsed in deadline order; equal-deadline timers fire in registration order") already permits this; the sentence is a confirmation, not a new obligation. Cross-reference from `cancellation.md` §Granularity to the same PIC-10 anchor in passing.

Implementer edge cases to watch:

- The `Clock.setTimeout`-driven yield in production runs `fn` on the platform's timer-callback macrotask, which is one timer-phase turn — sufficient to let the Pi-dispatched abort macrotask flip `loomAbort.signal.aborted` before the next iteration under presupposition (vi) of [Pi-side slash-handler promise lifecycle (consumption posture)](../spec_topics/pi-integration-contract/host-interfaces-core.md#pi-slash-handler-promise-lifecycle-presupposition). The "single macrotask turn" wording in `cancellation.md` should stay aligned with that presupposition; do not silently expand to "≥1 turns."
- The `V8a` / `V17c` tests must drive cancellation through `FakeClock.advance(0)` between iterations — not by relying on a real event-loop turn — because the FakeClock `advance` is synchronous by spec. The fact that this collapses the production macrotask to a synchronous step under test is the intended test ergonomics, not a divergence.
- Do **not** introduce `setImmediate(fn)` either as a bare primitive or as a new `Clock` member; the existing `Clock.setTimeout(fn, 0)` surface is sufficient, and widening the ambient-primitive surface would force a parallel update to the `H3a` identifier-keyed scan and the `V8d` Tests-field allow-list contract.

## Consequence

**Severity:** correctness

Two implementers will pick different primitives — one routing through `Clock.setTimeout(fn, 0)`, one using bare `setImmediate(fn)`, one using bare `setTimeout(fn, 0)` and tripping the grep-ban at build time. The first two produce different test ergonomics under `FakeClock`; the third produces a divergent build outcome. The cancellation property under test (a Pi-dispatched abort landing before the next loop-iteration signal-check) is the load-bearing Esc-during-compute-bound-loop guarantee, so the divergence reaches an observable user-visible behaviour rather than only an internal naming choice.
