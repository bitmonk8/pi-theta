## Finding

The DISC-6 paragraph in `docs/spec_topics/discovery/package-and-settings.md` describes three observable trip paths for `loom/load/discovery-slow`: the `looms.scanPackagesMaxFiles` cap firing alone, the `looms.scanPackagesTimeoutMs` cap firing alone, and the simultaneous-cap tie-break. The tie-break sentence pins the `cap` field value verbatim ("the warning's `cap` field is `looms.scanPackagesMaxFiles`"). The two single-cap sentences only say the warning "names … the cap that fired" — without inlining either of the two settings-key strings the field is supposed to carry.

The value set is in fact already closed in one place: `docs/spec_topics/diagnostics/placeholder-rendering-b.md` §7 defines `<cap> ∈ {looms.scanPackagesMaxFiles, looms.scanPackagesTimeoutMs}`, and `docs/spec_topics/diagnostics/code-registry-load.md` rows the `discovery-slow` *Message* template as `package-discovery walk aborted at <root>: <cap> cap reached`. So the value a conformance test must assert is derivable — but only by chasing two cross-references from the DISC-6 site and pairing the closed enum with the "names the cap that fired" verb to infer the predicate-to-token mapping.

The asymmetry is the actual defect: a reader of DISC-6 in isolation gets a verbatim asserted value for one of three trip paths and a behavioural description for the other two. Either all three trip paths state the `cap` value at this site, or none do (and DISC-6 defers the rendering to the closed enum for all three uniformly). Today neither is true.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `475155c` (pi-loom plan: resolve "V14m discovery walk omits scoped packages and has no upper bound", 2026-05-05) and `0378193` (pi-loom spec: resolve "T12 dual-cap simultaneous-breach tie-break", 2026-05-18)
**History:** Commit `475155c` introduced the `loom/load/discovery-slow` registry row, the bounded-walk paragraph, and the *Message* template `package-discovery walk aborted at <root>: <cap> cap reached`, with the prose "names the root being scanned and the cap that fired". At that point all three trip paths were equally implicit — the `<cap>` value had to be inferred from context for every case. Commit `0378193` (resolving T12) then added the tie-break sentence pinning `cap` = `looms.scanPackagesMaxFiles` for the simultaneous case, but did not back-fill verbatim values for the two single-cap cases. The asymmetry this finding flags is the interaction of those two edits: the original paragraph treated all paths symmetrically (none stated, all inferable), and the tie-break patch broke the symmetry by stating one. The closed-enum entry at `placeholder-rendering-b.md` §7 and the registry-row test vector were added on later commits and would have made a full pinning straightforward at the time of the tie-break edit.

## Solution Space

**Shape:** single

### Recommendation

Tighten DISC-6 so the three trip paths render symmetrically. Replace the current "names the root being scanned and the cap that fired" clause with: "names the root being scanned and the cap that fired — `looms.scanPackagesMaxFiles` when the file-count predicate trips alone, `looms.scanPackagesTimeoutMs` when the wall-clock predicate trips alone, and `looms.scanPackagesMaxFiles` for the simultaneous tie-break per the rule below." Then keep the existing tie-break sentence as a redundant local explainer for *why* maxFiles wins on ties (without restating the value).

Cross-reference the §7 `<cap>` closed enum in `placeholder-rendering-b.md` parenthetically so a reader spotting the inlined values knows the rendering surface is closed and spec-versioned per GOV-7/GOV-8. Do not delete or reword the closed-enum entry itself — the inlined list at the DISC-6 site is the binding from predicate-to-token; the enum entry is the rendering-surface contract.

Update `V10b-T`'s `DISC-6` test bullet to make all three `cap` token assertions explicit: maxFiles-only trip → `cap = "looms.scanPackagesMaxFiles"`; timeoutMs-only trip → `cap = "looms.scanPackagesTimeoutMs"`; simultaneous trip → `cap = "looms.scanPackagesMaxFiles"`. The existing "file-count is checked before time on a tie" bullet covers the tie-break ordering but is silent on the two single-cap token bindings.

## Consequence

**Severity:** cosmetic

A conformance test can already derive the `cap` token by composing the §7 closed enum with the "names the cap that fired" verb, so no observable behaviour is at risk. The cost is reader-side: DISC-6 reads as if the tie-break case is uniquely worth pinning, inviting the question "why is the single-cap value not stated?" — the very question this finding records. Tightening the local statement removes the false signal.
