## Finding

`H1a`'s **Adds** declares `minimatch` in loom's own runtime `dependencies` block, on the same footing as `semver`, `chokidar`, and `yaml`, as the glob-matcher engine that realises the `pi.looms` resolution contract — the `!`/`+`/`-` override ordering and exact-path operand treatment fixed at [`discovery/package-and-settings.md` §DISC-5](docs/spec_topics/discovery/package-and-settings.md) and consumed at `V10b`. The spec's runtime-dependency enumeration, [`implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)"](docs/spec_topics/implementation-notes.md), names `semver`, `chokidar`, and `yaml` explicitly (with AJV carried separately in a non-normative implementation hint), but never names a glob engine. `minimatch` is therefore a fourth runtime dependency present in the plan but absent from the spec page that enumerates that exact category.

DISC-5 genuinely requires a glob engine, so `H1a` is correct to provision one; the gap is purely that the spec's enumeration is incomplete relative to the plan's actual dependency set. The plan even has to cite DISC-5 rather than the implementation-notes enumeration for `minimatch`, precisely because the enumeration omits it. The spec does not assert closure over all runtime dependencies, so this is a documentation-parity asymmetry rather than scope creep — and it is the same class of gap the spec previously closed for `yaml` by adding it to this enumeration.

**History:** `minimatch` first entered the plan corpus at `c6a664e`, named in `V10b` as the DISC-5 glob engine but not yet provisioned in `H1a` and with no matching entry in the spec's `implementation-notes.md` enumeration. The provisioning gap was closed at `8279cb1`, which added `minimatch` to `H1a`'s runtime `dependencies` block (alongside `semver`/`chokidar`/`yaml`) but touched only `docs/plan-review.md` and `docs/plan_topics/H1a-scaffold-and-toolchain.md` — it did not extend the spec enumeration. That left `minimatch` declared as a peer of the three enumerated runtime dependencies in the plan while remaining absent from the spec page that enumerates them. The same parity pattern was previously resolved for `yaml` (commit `1064946`, "resolve \"YAML frontmatter parsing mechanism is never declared as a dependency\"") by adding it to this enumeration; `minimatch` is the next un-resolved instance.

## Solution Space

**Shape:** single

### Recommendation

In `docs/spec_topics/implementation-notes.md` §"Loom-package implementation dependencies (loom 1.0)", add a clause naming `minimatch` as a runtime dependency in the loom package's `dependencies` block, parallel to the existing `chokidar` and `yaml` clauses: state that the loom 1.0 runtime depends on `minimatch` as the glob-matcher engine realising the `pi.looms` glob resolution — the `!`/`+`/`-` override ordering and exact-path operand treatment — pinned at DISC-5, that the clause names the dependency and not its version, and (mirroring the `chokidar`/`yaml` clauses) that `minimatch` is neither one of the four `@earendil-works/*` peer packages nor `typebox` and so is out of scope for the inventory-closure audit's *Target surface categories* and needs no `// allow-pi-surface:` exemption marker at its import sites. The spec is the only file edited; `H1a` and `V10b` already declare and consume `minimatch` correctly and stay unchanged.

Edge case for the implementer to confirm when wiring the engine: that `minimatch`'s `!`/`+`/`-` precedence and exact-path operand semantics reproduce the DISC-5-exact ordering (plain → `!` → `+` → `-`); if a chosen `minimatch` configuration diverges, that is a `V10b` implementation concern, not a reason to change this enumeration edit.

## Consequence

**Severity:** cosmetic

The plan is complete and an implementer following `H1a` provisions `minimatch` correctly; nothing is blocked or ambiguous. The only effect is that a reader cross-checking the spec's dependency enumeration against the plan finds an unexplained fourth runtime dependency, weakening the enumeration's value as a parity reference.
