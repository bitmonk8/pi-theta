## Finding

`H4a`'s harness defines an **in-memory fixture-supply mechanism** whose Adds describes its payload as content only: "the harness supplies the fixture `.loom` content to the extension in memory so a harness-driven leaf's discovery reads from the harness-provided source rather than the real filesystem". `M` and `M-T` are the named consumers of that mechanism — `M`'s "fixed single-source discovery" reads exclusively from the harness-supplied source, with no ambient `src/**` filesystem read and no `FileSystem` (`V8b`) fallback.

But `M` also performs a `registerCommand` call and dispatches `pi /<name>`. Per the spec, the slash `<name>` is **the loom's filename stem taken verbatim** (`discovery/discovery-sources.md` §Filename validity, stem matching `^[a-z0-9][a-z0-9_-]*$`), with the hyphen→underscore default-name remap (`frontmatter/frontmatter-fields-a.md`). The only input `M`'s discovery sees is the harness-supplied source, and that source is specified as carrying content alone — a filename / stem / path is not part of the stated input. A fresh implementer who builds the mechanism to the letter (content only) leaves `M` with no in-contract way to obtain the stem its `registerCommand` and `pi /<name>` dispatch require.

## Issue introduction

**Verdict:** single-commit
**Introducing commits:** 2d52ad1 — pi-loom plan: resolve "M's happy-path discovery read has no defined source under the ambient-access ban" (2026-06-11, Thomas Andersen)
**History:** The in-memory fixture-supply mechanism was added to `H4a` (and wired into `M` / `M-T`) in this single commit, which resolved an earlier finding that `M`'s happy-path discovery had no defined source under the ambient-access ban. The mechanism was authored from the outset as supplying the fixture `.loom` *content* in memory; the filename / stem that `M` needs to derive its slash `<name>` was never part of the stated payload, so the content-only narrowing is present from the mechanism's introduction.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H4a-factory-shell-and-harness.md`, in the Adds sentence describing the in-memory fixture-supply mechanism, state that the harness supplies the fixture's **name/path** (the filename stem) alongside its content, so a harness-driven leaf's discovery can derive the slash name. The sentence currently reading "the harness supplies the fixture `.loom` content to the extension in memory" should convey that it supplies both the fixture's name/path and its content; leave the API shape to the implementer.

Edge case the implementer must respect: the slash name derives from the filename stem per `discovery/discovery-sources.md` §Filename validity (stem matching `^[a-z0-9][a-z0-9_-]*$`) and the hyphen→underscore default-name remap in `frontmatter/frontmatter-fields-a.md`, so the harness-supplied name must be a stem that satisfies that rule for `M`'s `registerCommand` and `pi /<name>` dispatch to resolve. No change to `M` / `M-T` is required once the `H4a` mechanism states it provides the name.

## Consequence

**Severity:** advisory

If the harness's in-memory fixture-supply is implemented strictly as written (content only), `M`'s discovery has no filename stem from which to derive the slash `<name>` for its `registerCommand` call and `pi /<name>` dispatch, so the `H4a` implementer and the `M` implementer would diverge on how the name reaches discovery (extend the mechanism ad hoc, or hardcode a name out-of-band). The contract gap is in `H4a`'s Adds wording rather than in `M`'s behaviour.
