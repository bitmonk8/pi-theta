## Problem

`query/query-tool-loop.md` states a normative MUST that the runtime right-trims any trailing U+000A bytes from the rendered prompt text before inserting the single-line-feed separator that precedes the forced respond turn's instruction wording, keeping the prompt-to-instruction gap exactly one line feed in every case. No test vector pins this trim: `V13c-T`'s asserted vectors cover the free-phase tool loop, `max_rounds: 0`, exhaustion, and the depth-6 co-fire case, but not the right-trim. A template ending in multiple trailing newlines versus one is observably different under a non-trimming implementation and byte-identical under a trimming one, yet no fixture proves the trim happened.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `bbe2602` (2026-06-05, "Initial forced respond turn — instruction wording status undeclared") — landed the *Initial forced respond turn template* section and the right-trim MUST in the paragraph that references it. The template body became byte-exact in the same commit, but the right-trim itself never got a paired vector.
**History:** A normative MUST landed without a paired conformance vector — a "fix-without-observable" pattern.

## Solution approach

Add a normative right-trim test vector to the `id="initial-forced-respond-turn-template-normative"` section of `query/query-tool-loop.md`, asserting that a typed query whose rendered prompt body carries trailing U+000A bytes produces a forced-respond opening turn with exactly one line-feed separator before the template body — byte-equal to the same query with no trailing newline. Add a corresponding `V13c-T` assertion bullet citing the new vector's anchor.

## Solution constraints

- The vector MUST distinguish trailing U+000A trimming from general trailing-whitespace trimming: the contract right-trims newline bytes only, so the vector MUST NOT assert that non-newline trailing whitespace is stripped.
