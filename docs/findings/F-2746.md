## Finding

`V18c`'s build-time literal-read assertion consumes two constants that `V9a`
owns and exports: `FACTORY_PROBABLE_CAPABILITIES` (the closed five-element
factory-probable capability list, the symbol `V18a`'s partition assertion also
consumes) and `SHUTDOWN_AWAIT_CAP_MS = 2000` (the cancellation-runtime cap).
Both are read from `V9a`'s single source-of-truth pinned-constants block — the
`V9a` leaf itself states `SHUTDOWN_AWAIT_CAP_MS` is the constant "which `V18c`'s
build-time literal-read assertion reads," and names `V18a`/`V18c` as the
importers of `FACTORY_PROBABLE_CAPABILITIES`.

`V18c.Deps` is `V18c-T`, `V18a`, `V18b` — it reaches `V9a` only transitively,
through `V18a` (whose `Deps` declares `V9a`). The build is executable as written
because the transitive edge keeps `V9a` ordered ahead of `V18c`, but the
declared dependency contract understates the coupling: `V18c` has a direct
source-level read of `V9a`'s constants with no `Deps` entry recording it. A
future edit that re-scoped `V18a` to stop importing `V9a` would sever the only
declared path to `V9a` and silently invalidate `V18c`'s assertion, with nothing
in `V18c`'s own `Deps` signalling that `V18c` needs `V9a`. The paired test leaf
`V18c-T` (`Deps`: `V18a`, `V18b`) authors the failing tests for the same
literal-read assertion and inherits the same gap.

## Consequence

**Severity:** advisory

The build orders correctly today (`V9a` is reachable transitively via `V18a`),
so no implementer is blocked. The risk is latent: `V18c`'s `Deps` does not
record its direct read of `V9a`'s `FACTORY_PROBABLE_CAPABILITIES` /
`SHUTDOWN_AWAIT_CAP_MS`, so a later narrowing of `V18a` that drops its `V9a`
import would break `V18c`'s assertion with no dependency-graph warning.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `b9de3ee`, `622d0d3`
**History:** The plan corpus is git-tracked. `V18c.Deps` has read
`V18c-T, V18a, V18b` since well before the V9a coupling existed (present
unchanged at `8af3204` and its predecessors; `git log -G '\*\*Deps\.\*\*'
--follow` on the leaf shows no later edit to that line touching `V9a`). The
direct V9a reads were each added to `V18c` afterward without updating the
`Deps` line: `git log -S 'FACTORY_PROBABLE_CAPABILITIES' --follow` traces the
`FACTORY_PROBABLE_CAPABILITIES` consumption into `b9de3ee` (2026-06-11, "resolve
\"V18a partition assertion imports undeclared V9a probe constant\"") — a sweep
that added `V9a` to `V18a.Deps` and mentioned the symbol in `V18c`'s Adds while
asserting "these gates need only the inventory leaves," leaving `V18c.Deps`
untouched; `git log -S 'SHUTDOWN_AWAIT_CAP_MS' --follow` traces the
`SHUTDOWN_AWAIT_CAP_MS` read into `622d0d3` (2026-06-11, "resolve
\"SHUTDOWN_AWAIT_CAP_MS physical home contradicts pinned-constants-block
placement\""), also without a `Deps` edit. The under-declaration is therefore an
interaction: a pre-existing `Deps` line that two later constant-introducing
commits each added a direct `V9a` read against, neither reconciling the edge.

## Solution Space

**Shape:** single

### Recommendation

Add `V9a` to `V18c.Deps` in
`docs/plan_topics/V18c-version-bump-checklist.md`, so the leaf's `Deps` reads
`V18c-T`, `V18a`, `V18b`, `V9a` and records the direct constant read rather than
relying on the transitive edge through `V18a`. Add `V9a` to `V18c-T.Deps` in
`docs/plan_topics/V18c-T-version-bump-checklist.md` as well, since the test leaf
authors the failing literal-read assertions over the same
`FACTORY_PROBABLE_CAPABILITIES` / `SHUTDOWN_AWAIT_CAP_MS` constants. No leaf
prose needs to change — `V18a` already correctly declares `V9a`, so the existing
transitive ordering is preserved; the added edges only make `V18c`/`V18c-T`'s
own dependence on `V9a` explicit.
