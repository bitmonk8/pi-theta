## Finding

The acceptance-criteria fixture (b) in `drain-state-contract.md` reads: "Assert `LoomRegistry.readDrainState` returning `(drained: true, tag: undefined)` routes through arm (b) — the `"loom /<name>: extension shutting down"` system note." The grammatical subject of "routes" is `readDrainState`, but `readDrainState`'s entire contract is to *return a snapshot of the two drain-state fields* — it performs no routing and has no arms of its own. Arm selection is defined by PIC-29 as a behaviour of the **slash-command handler**, which consults the snapshot at entry and chooses one of three branches. Attributing the routing to `readDrainState` conflates the snapshot-return obligation with the arm-selection obligation.

The concrete divergence this enables is a collapse between fixtures (a) and (b). Fixture (a) already asserts that the post-handler snapshot tuple is `(drained: true, tag: undefined)`. If a test author reads fixture (b)'s subject literally — "`readDrainState` returning the tuple" — they may write a fixture that simply re-asserts the returned tuple and labels it "arm-(b) routing," never exercising the tuple→arm mapping that PIC-29 marks normative. Two reasonable authors would then produce fixtures of different strength: one asserting the `(true, undefined)` → arm (b) mapping (the intended obligation), the other duplicating fixture (a)'s tuple assertion.

The trailing sentence ("The routing is witnessed at the `readDrainState` return mapping rather than via a live slash dispatch…") partially mitigates this by pinning the witness mechanism, but it does not repair the subject: it reinforces the framing that the *return* routes, leaving the arm-as-handler-branch identity (PIC-29) implicit. The fix is to name the slash handler as the routing subject and state the obligation as the tuple→arm mapping, while keeping the existing witness caveat intact.

## Solution Space

**Shape:** single

### Recommendation

Reword fixture (b) so the routing subject is the slash-command handler and the asserted obligation is PIC-29's tuple→arm mapping, not `readDrainState`'s return value alone. For example: "Assert that the snapshot tuple `(drained: true, tag: undefined)` selects slash-handler arm (b) — the `"loom /<name>: extension shutting down"` system note — per [PIC-29](#pic-29)'s closed three-arm tuple-to-arm map (arm-(b) tuple set `(false, "shutting-down")`, `(true, "shutting-down")`, `(true, undefined)`)." Keep the existing "witnessed at the `readDrainState` return mapping rather than via a live slash dispatch" sentence verbatim — do **not** replace it with a live-handler-dispatch assertion, since on `reason ∈ {"quit","reload"}` Pi tears the runtime down before any subsequent dispatch arrives.

Make explicit that this obligation is distinct from fixture (a): fixture (a) asserts the post-handler snapshot *is* `(true, undefined)`; fixture (b) asserts that tuple *maps to* handler arm (b). The two assertions must not collapse into one.

## Consequence

**Severity:** advisory

A fixture author following the literal subject may assert only the returned tuple, duplicating fixture (a) and never exercising PIC-29's normative tuple→arm mapping. The slash-handler arm-selection branch then ships with weaker coverage than the spec intends, though the surrounding witness sentence keeps a correct reading available.
