## Finding

`H6a`'s **Deps** note carries an inline recap of the coverage-producing set it inherits transitively through `H5b`: "`H5b` owns the complete coverage-producing dependency set (`H1a`, `H5a`, `M`, `V1a`–`V18d`)". That enumeration is stale. `H5b`'s actual **Deps** list two further bare singletons — `H5e` (the un-anchored normative-MUST recogniser arm) and `H5f` (the per-facet citing-test arm for multi-leaf rows) — that the recap omits. `H5b`'s own Deps note names all four singletons (`H1a`, `H5a`, `H5e`, `H5f`, plus `M`) and explains each one's coverage-producing role.

`H6a`'s real dependency edge is `H5b, H5c, H7a`, and the note is explicit that `H6a` "inherits that completeness transitively through `H5b` rather than restating the set". Pickability and build order are therefore unaffected — `H5b` is the authoritative carrier and the recap closes no edge. The recap's stated purpose, however, is to assert that "the warn-only and hard-fail footings cover an identical set of leaves"; an inline enumeration that lists only four of the six singletons defeats that purpose and invites a reader auditing `H6a`'s inherited prerequisites to under-count the set by the two `H5*` gate-arms.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `5f33832` (2026-06-18, "resolve \"H5a bundles seven gate arms behind a single Ships-when\""), `26a8794` (2026-06-18, "resolve \"split coverage-matrix rows can ship a facet untested\"")
**History:** The recap was last synchronised to `H5b`'s set at `1d22b5c` (2026-06-11, "resolve \"H6a Deps-note parenthetical omits H1a\""), which added `H1a` to make it `(H1a, H5a, M, V1a–V18d)` — correct for `H5b.Deps` as it stood that day. `H5e` was then added to `H5b.Deps` at `5f33832`, and `H5f` at `26a8794`; neither commit propagated the new singleton into the `H6a` recap — and `26a8794` edited `H6a`'s **Adds** body to reference the new `H5f` arm while leaving the recap untouched. A later edit at `9d63d2a` (2026-06-21, "pin H5c edge on H6a Deps") revised the same parenthetical to pin the `H5c` edge but still did not add `H5e`/`H5f`. The recap thus drifted out of sync as the `H5e`/`H5f` arms landed in `H5b` without a matching recap update.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/H6a-live-corpus-activation.md`'s Deps note, replace the inline coverage-producing-set enumeration with a pointer to `H5b`'s Deps as the single authoritative list — e.g. change "`H5b` owns the complete coverage-producing dependency set (`H1a`, `H5a`, `M`, `V1a`–`V18d`)" so the parenthetical no longer restates the member list but instead refers the reader to `H5b`'s **Deps** for the authoritative set. The history shows the inline copy drifts every time a singleton is added to `H5b`; a pointer removes that drift class entirely and is consistent with the note's own statement that `H6a` inherits the set "rather than restating it".

If the inline enumeration is retained instead, its literal content must add the two missing singletons so it reads `(\`H1a\`, \`H5a\`, \`H5e\`, \`H5f\`, \`M\`, \`V1a\`–\`V18d\`)`, matching `H5b.Deps`. The spec is read-only for this fix; do not edit `H5b`, whose set is already correct.

## Consequence

**Severity:** advisory

No implementer goes off-script: `H6a`'s authoritative edge is `H5b`, which carries the complete and correct set, so build order and pickability are unchanged. The defect is a documentation-accuracy gap — a reader consulting `H6a`'s recap to understand the inherited coverage-producing set under-counts it by `H5e` and `H5f`, and the recap's "identical set of leaves" claim is no longer self-evidently true.
