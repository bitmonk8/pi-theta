## Problem

At anchor `#snapshot-restore-pi-behavioural-preconditions`, precondition (a) describes `pi.setActiveTools(string[])` as "synchronous-and-atomic on the JS event loop (a single swap or restore call cannot be interleaved with anything else)". The parenthetical gloss restates a property every synchronous JS call satisfies trivially — the single-threaded event loop cannot interleave any other task while a synchronous call runs — so the conjunction is either tautological or self-contradictory and pins no property a future Pi minor could measurably weaken. The property downstream consumers actually rely on is that the call returns synchronously without yielding and the active-set change is committed and observable by the time it returns. PIC-19's "step-2 throw leaves the install uncommitted" claim and PIC-17's snapshot-target reasoning silently rest on that useful reading, which the current wording fails to pin.

## Issue introduction

**Verdict:** present-since-the-feature-was-introduced
**Introducing commits:** `e7ba4ba` ("pi-loom spec: resolve \"pi.setActiveTools snapshot/restore relies on an unstated Pi serialisation guarantee\"", 2026-05-06) introduced the two-precondition framing with the wording "`pi.setActiveTools(string[])` is synchronous and atomic on the JS event loop, so a single swap or restore call cannot be interleaved with anything else"; commit `68ff2d1` ("pi-loom spec: resolve \"Pi API surfaces asserted without .d.ts citations\"", 2026-05-07) refactored the same paragraph to add the `.d.ts` citation and SDK-pin framing and tightened the wording to "synchronous-and-atomic on the JS event loop (a single swap or restore call cannot be interleaved with anything else)" — the form the present finding cites.
**History:** The defect entered with the feature itself — the precondition was vacuous on day one of the snapshot/restore protocol's behavioural-precondition framing. Subsequent edits (`19b193b` adding PIC-19, `823ce54` adding the recovery-mutex contract, `4d8dc02` adjusting the hot-reload note) preserved the wording unchanged and built dependents on top of it (PIC-19's "step-2 throw leaves the install uncommitted" cites this precondition by anchor), so the wording is now load-bearing for at least one downstream normative claim, even though that claim quietly relies on the *useful* reading the wording fails to pin.

## Solution approach

Rewrite precondition (a) at `#snapshot-restore-pi-behavioural-preconditions` in `tool-registration-lifetime.md` to pin a synchronous-completion-and-commit property in place of the vacuous "synchronous-and-atomic" gloss: that `pi.setActiveTools(string[])` returns synchronously without yielding, the active-set change is committed by the time the call returns (observable to a subsequent `pi.getActiveTools()` in the same task), and a throwing swap-install leaves the active set unchanged. Verify PIC-19's "step-2 throw leaves the install uncommitted" cross-reference at `#pic-19` still reads correctly against the rewritten wording.

## Solution constraints

- Precondition (a) MUST retain the commit-on-return half (the change observable by the time the call returns and a throwing swap-install leaving the set unchanged), not just synchronous return — PIC-19's swap-install-throw rule (`#pic-19`) and PIC-17's snapshot-target reasoning depend on it.
- Out of scope: precondition (b) and the recovery-mutex runtime contract (`#recovery-mutex-runtime-contract`) — same-paragraph neighbours owned by sibling findings F-1213 / F-1228; edit only precondition (a).
