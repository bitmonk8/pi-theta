## Finding

`V9a` declares and exports `FACTORY_PROBABLE_CAPABILITIES`, described in its **Adds** as "the closed five-element list of factory-probable capability identifiers (inventory items 1/2/3/4/6)" and in its `PIC-5` Tests bullet as enumerating "exactly the five factory-probable capability identifiers ŌĆ” and nothing else". Neither statement pins the **element form** of the list: an element could be an ordinal (`1`, `2`, `3`, `4`, `6`), the spec's per-capability anchor-ID string (`sdk-cap-slash-command-registration`, etc.), or a Pi member name. The phrase "inventory items 1/2/3/4/6" names *which* capabilities, not *how each one is represented* as a value in the array.

`FACTORY_PROBABLE_CAPABILITIES` is consumed as a cross-leaf reconciliation target. `V18a`'s `PIC-15` partition assertion verifies "the set of factory-probed-flagged entries [of `CAPABILITY_OBLIGATIONS`] equals the Step-0 factory-probable capability set imported from `V9a`'s exported `FACTORY_PROBABLE_CAPABILITIES` constant (not a literal re-listed here)", and `V18c`'s build-time literal-read assertion reads the same symbol. A set-equality reconciliation is only meaningful if both sides agree on the element representation. Because `V9a` (the owner) leaves the form unstated, the `V9a` implementer and the `V18a`/`V18c` implementer can independently pick divergent representations; the partition assertion then compares non-comparable values and is wrong-by-construction (it either fails to type-check or compares two sets that can never be equal).

The fix belongs at the declaration site: `V9a` must state the literal element form once, so every importer keys on the same representation. This is a Class-2 reference (pinning a stable identifier representation), not a new shape mandate.

## Solution Space

**Shape:** single

### Recommendation

In `V9a`'s **Adds**, pin the literal element form of `FACTORY_PROBABLE_CAPABILITIES` where it is declared. State that each element is the per-capability anchor-ID string from `capability-inventory-items.md` for items 1/2/3/4/6 ŌĆö `sdk-cap-slash-command-registration`, `sdk-cap-prompt-conversation-drive`, `sdk-cap-subagent-isolated-session`, `sdk-cap-tool-registration-gating`, and `sdk-cap-custom-message-renderer` ŌĆö so the constant's element type is a fixed, spec-sourced identifier rather than an unstated value. This is a Class-2 reference to `capability-inventory-items.md`'s existing anchors; it adds no new shape obligation beyond naming the representation the existing reconciliation already presupposes.

Ensure the representation named in `V9a` is the same identifier `V18a`'s `CAPABILITY_OBLIGATIONS` entries carry for their partition flags, since `PIC-15`'s set-equality assertion reconciles the two: whichever representation `V9a` declares is the one `V18a` must key on. The `V9a-T` `PIC-5` cardinality bullet may be aligned to assert the pinned elements, but the load-bearing edit is the `V9a` declaration-site statement. Leave `capability-inventory-items.md` unedited ŌĆö the spec is read-only for this fix; it already supplies the anchor IDs.

## Consequence

**Severity:** correctness

Two reasonable implementers diverge on the element representation of `FACTORY_PROBABLE_CAPABILITIES` (ordinal vs anchor-ID string vs member name), so `V18a`'s `PIC-15` partition reconciliation ŌĆö which asserts set-equality between `CAPABILITY_OBLIGATIONS`'s factory-probed-flagged entries and the imported constant ŌĆö compares non-comparable values and is wrong-by-construction; the same ambiguity reaches `V18c`'s build-time literal-read. The defect surfaces as an unbuildable or vacuous cross-leaf gate rather than the intended factory-probe / inventory partition check.
