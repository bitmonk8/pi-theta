## Finding

`V5f`'s `Adds` states that the canonical-hash recipe composes the `V2d` number renderer and supplies its `integer`-vs-`number` kind discriminator "from the JSON-Schema `type` of the enclosing `const`/`enum` position rather than from runtime integrality." `V2d`'s `Adds` repeats the same derivation source verbatim ("`V5f` from the JSON-Schema `type` of the enclosing `const`/`enum` position"). That source is unreadable for the only values it has to render.

The lowering algorithm in `docs/spec_topics/schema-subset.md` (Lowering Algorithm step 3) emits a numeric literal as a bare `{ "const": <value> }` — no sibling `type` keyword. It also emits every enum (and every string-literal union) as `{ "type": "string", "enum": [...] }`, and `schemas.md` pins "loom 1.0 enums carry string values only." So numeric values reach a lowered fragment **only** in `const` positions, and those positions carry no `type` keyword to read; enum positions carry a `type`, but it is always `"string"` and never holds a number. The canonical-hash recipe itself (schema-subset.md *Canonical form* step 2) confirms the only numeric literals it must render are "the values appearing in `const` and `enum` positions" — i.e. numeric `const` only — yet the discriminator source `V5f` names does not exist at that position.

An implementer following `V5f` literally has no `type` keyword to inspect for a numeric `const` and is left to either read a non-existent keyword (and fail or guess) or fall back to the value's runtime integrality — the exact fallback `V2d` forbids ("never from the value's runtime integrality"). The correct, available source is the loom-side static type of the numeric literal threaded through the lowering pass: lowering "is purely a function of the parsed source" (schema-subset.md), so the loom AST already knows whether a literal's declared type is `integer` or `number`, independent of the value's magnitude. `V11h`, the other `V2d` caller, already derives its discriminator from the bound value's static type via the value model (`V2c`); `V5f` needs the analogous loom-side-type source, not a lowered-fragment `type` keyword.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `4807958`
**History:** The defect phrase entered in `4807958` (2026-06-17, "pi-loom plan: resolve \"renderer input contract: integer-vs-number discriminator origin\""). That commit resolved a prior review finding by making `V2d` receive an explicit caller-supplied kind discriminator instead of inferring from runtime integrality — a correct change — but in the same edit it assigned `V5f`'s discriminator source as "the JSON-Schema `type` of the enclosing `const`/`enum` position" in both `docs/plan_topics/V2d-number-rendering.md` and `docs/plan_topics/V5f-subset-lowering-hash.md`. `git log -G` over the phrase reports `4807958` as the only commit touching it; before that commit `V2d`'s `Adds` read "with no dependency on the runtime value model or schema declarations" and named no `V5f` derivation source, so the unreadable-position defect did not exist prior to the fix that introduced it. The defect is therefore a regression in the discriminator-origin fix, not present since inception.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/V5f-subset-lowering-hash.md`, in `Adds`, replace the clause `supplying that renderer's integer-vs-number kind discriminator from the JSON-Schema `type` of the enclosing `const`/`enum` position rather than from runtime integrality` with a clause that sources the discriminator from the loom-side static type of the numeric literal as threaded through the lowering pass (lowering being a pure function of the parsed source per `schema-subset.md`), still rather than from runtime integrality. In `docs/plan_topics/V2d-number-rendering.md`, in `Adds`, apply the identical correction to the parenthetical describing `V5f`'s obligation (`V5f` from the JSON-Schema `type` of the enclosing `const`/`enum` position) so the two leaves keep one consistent derivation source; leave the `V11h` parenthetical (already "from the bound value's static type via the value model (`V2c`)") unchanged.

Edge cases the implementer must watch:
- Enums never carry numeric values (lowered as `{ "type": "string", "enum": [...] }`, string-only per `schemas.md`), so the discriminator applies only to numeric `const` positions; the corrected derivation source need not cover an enum position at all, and the `const`/`enum` framing can collapse to `const`.
- The corrected source must remain magnitude-independent: an `integer`-typed literal whose value happens to be representable as a whole `number` must still render via BNDR-4, not BNDR-5.
- If a numeric-`const` asserting test is added to close this in `V5f-T`, it should assert the canonical form of an `integer`-typed `const` and a `number`-typed `const` that share a value but render differently.

## Consequence

**Severity:** correctness

`V5f`'s canonical-hash recipe is unimplementable as written for any schema fragment containing a numeric `const`: the named kind-discriminator source does not exist at that position. Two reasonable implementers diverge — one threads the loom-side static type, another falls back to runtime integrality — and produce different canonical bytes for the same literal (e.g. an `integer`-typed `42` as `42` versus a `number`-typed value rendered shortest-fixed-point), hence different 16-hex slugs. Because the slug is part of the on-disk and on-wire contract (cache keys, `__inline_<slug>` `$defs` names, dedup), the divergence corrupts dedup and replayable payloads.
