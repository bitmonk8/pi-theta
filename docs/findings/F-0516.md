## Problem

`V5f`'s `Adds` (and the matching parenthetical in `V2d`'s `Adds`) sources the `V2d` renderer's `integer`-vs-`number` kind discriminator "from the JSON-Schema `type` of the enclosing `const`/`enum` position." That source does not exist for the only values it must render: per `schema-subset.md`'s Lowering Algorithm, a numeric literal is lowered to a bare `{ "const": <value> }` with no sibling `type` keyword, and enum positions always carry `type: "string"` and never a number. An implementer following `V5f` literally has no `type` to inspect for a numeric `const` and must either read a non-existent keyword or fall back to the value's runtime integrality — the exact fallback `V2d` forbids — causing two implementers to produce divergent canonical bytes and slugs.

## Issue introduction

**Verdict:** single-commit introduction
**Introducing commits:** `4807958`
**History:** The defect phrase entered in `4807958` (2026-06-17, "pi-loom plan: resolve \"renderer input contract: integer-vs-number discriminator origin\""). That commit resolved a prior review finding by making `V2d` receive an explicit caller-supplied kind discriminator instead of inferring from runtime integrality — a correct change — but in the same edit it assigned `V5f`'s discriminator source as "the JSON-Schema `type` of the enclosing `const`/`enum` position" in both `docs/plan_topics/V2d-number-rendering.md` and `docs/plan_topics/V5f-subset-lowering-hash.md`. `git log -G` over the phrase reports `4807958` as the only commit touching it; before that commit `V2d`'s `Adds` read "with no dependency on the runtime value model or schema declarations" and named no `V5f` derivation source, so the unreadable-position defect did not exist prior to the fix that introduced it. The defect is therefore a regression in the discriminator-origin fix, not present since inception.

## Solution approach

Rewrite the discriminator-source clause in `V5f`'s `Adds` so the kind discriminator derives from the loom-side static type of the numeric literal threaded through the lowering pass (lowering being a pure function of the parsed source), still not from runtime integrality. Apply the identical correction to the `V5f` parenthetical in `V2d`'s `Adds` so the two leaves keep one consistent derivation source.

## Solution constraints

- Leave the `V11h` parenthetical in `V2d`'s `Adds` ("from the bound value's static type via the value model (`V2c`)") unchanged.
- The spec (`schema-subset.md`, `schemas.md`) is read-only for this fix; the correction is confined to the two plan leaves.
