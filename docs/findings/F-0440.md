## Finding

The *Sequential by default* cross-cutting rule (`docs/plan_topics/conventions.md`, §Cross-cutting rules) permits a banned `Promise` combinator in `src/**` only when the calling leaf's `Spec.` field cites a concurrency-mandating obligation **and** "the leaf's `Adds.` field names the construct together with that REQ-ID or code-keyed obligation area." The phrase "names the construct" is unqualified: it does not state whether `Adds.` must contain the literal combinator token (e.g. `Promise.allSettled`) or merely describe the concurrent behaviour in prose. A leaf author and a reviewer applying the rule can reasonably read it either way.

The ambiguity is sharpened by the §Leaf format **Adds.** definition, which classifies "the concurrency construct the *Sequential by default* rule requires `Adds.` to name" as a binding Class-1 mechanism — but the same definition explicitly relaxes the *Class-2 seam* naming test to "an unambiguous prose reference … [that] does not require the consumer to repeat an exact identifier token." With one neighbouring naming obligation explicitly token-free, the absence of a parallel statement for the Class-1 concurrency construct leaves the reader unsure whether the combinator token is required here.

Every existing combinator-using leaf already names the literal token in `Adds.` — `V14b` (`Promise.allSettled`, `cka-13`), `V9g` (`Promise.allSettled`, `cka-31`), and `V9i` (`Promise.all`, `PIC-22`) — so the de facto convention is token-naming. The defect is that the rule text does not say so, leaving new combinator-using leaves authored under How-to-use step 2 without an unambiguous instruction.

**History:** The unqualified word "construct" has been the operative wording of this requirement since it was first authored at `861e935`. The wording survived the plan reset/re-pivot (`657ee76` → `15f69aa`) intact and the carve-out rework at `75b6a9b` (2026-06-10), which broadened the carve-out to code-keyed obligation areas but kept "names the construct" unqualified. No later commit introduced the ambiguity; it is original to the rule.

## Solution Space

**Shape:** single

### Recommendation

In `docs/plan_topics/conventions.md`, in the *Sequential by default* cross-cutting rule, the clause "and the leaf's `Adds.` field names the construct together with that REQ-ID or code-keyed obligation area" pins the combinator token. Rewrite "the construct" to name the specific banned `Promise` combinator used at the site (one of `Promise.all` / `Promise.race` / `Promise.allSettled` / `Promise.any`) together with the mandating REQ-ID or `cka-<n>` token — matching the existing practice in `V14b`, `V9g`, and `V9i`, whose `Adds.` already cite the literal combinator alongside the mandating token.

Align the §Leaf format **Adds.** definition's Class-1 parenthetical — "(e.g. the concurrency construct the *Sequential by default* rule requires `Adds.` to name alongside its mandating REQ-ID or code-keyed obligation area)" — to the same token-naming wording, so the cross-reference does not reintroduce the unqualified "construct". This is the Class-1 cross-reference whose authoritative home is the *Sequential by default* rule; the **Adds.** definition only points at it, so both must agree.

The fix is internal to `conventions.md`; the spec is read-only for this change.

## Consequence

**Severity:** advisory

A new leaf authored under How-to-use step 2 that uses a `Promise` combinator could write an `Adds.` describing concurrent behaviour without naming the specific combinator token, and reviewers would inconsistently accept or reject it against this rule. No shipped leaf is at risk — the mechanical `// allow: <token> — <spec-page>` comment and the closing-gate token resolution enforce the combinator independently of `Adds.`, and the existing combinator-using leaves already name the token — so the gap is authoring guidance only.
