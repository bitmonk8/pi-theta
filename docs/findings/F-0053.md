## Finding

H7a owns and produces a single multi-feature fixture `.loom` that the terminal integration-acceptance gate drives through the H4a harness. In H7a's own `Adds.` and its first `Tests.` bullet the artifact is named the **"representative multi-feature fixture `.loom`"**. The two leaves that back-reference the same artifact name it differently: H4a's real-host-smoke `Tests` bullet calls it the **"committed multi-feature fixture `.loom`"**, and H6a's `Release-gate acceptance` item and its dependency note also call it the **"committed multi-feature fixture `.loom`"**.

The same object therefore carries two adjectives across the three leaves that reference it, with no inline equivalence statement linking the two. A reader following the H4a/H6a back-references to the producing leaf must infer that "committed multi-feature fixture `.loom`" and "representative multi-feature fixture `.loom`" denote one and the same artifact rather than two distinct fixtures.

## Issue introduction

**Verdict:** multi-commit-interaction
**Introducing commits:** `e7e51cc9aa1867795564212a74bd221972318f52` (2026-06-10) and `e7f14dda7b9701ce9eb8823b842a1fc2cd5a5d0d` (2026-06-11)
**History:** H7a was created in `e7e51cc` ("resolve 'Plan has no terminal end-to-end integration-acceptance leaf'"), which introduced the term "representative multi-feature fixture `.loom`" in H7a's Adds. The divergent term "committed multi-feature fixture `.loom`" was introduced later in `e7f14dd` ("resolve 'Real-host fidelity of the session double has no reproducible detection point'"), which added H4a's back-reference using "committed" without reconciling it against the already-present "representative" in H7a. `1cab2ae` ("resolve 'H6a consumes H7a's golden artifacts but no dependency edge'") then propagated "committed" into H6a. The inconsistency is the product of the second commit's term choice colliding with the first commit's pre-existing term, not present at H7a's inception alone.

## Solution Space

**Shape:** single

### Recommendation

Normalise H7a's two occurrences onto the term the back-referencing leaves (H4a, H6a) already use, "committed multi-feature fixture `.loom`":

- In `docs/plan_topics/H7a-integration-acceptance.md` `Adds.`, replace "a single representative multi-feature fixture `.loom`" with "a single committed multi-feature fixture `.loom`".
- In the same file's first `Tests.` bullet, replace "running the representative multi-feature fixture `.loom`" with "running the committed multi-feature fixture `.loom`".

This leaves all four references (H7a Adds, H7a Tests, H4a smoke bullet, H6a release-gate item) on one adjective. H4a and H6a need no edit â€” they already carry the canonical term. If the descriptive sense of "representative" (a best-effort representative live run) must be retained anywhere, keep it only in H4a's existing "best-effort representative run" smoke-narrative sentence, which describes the run, not the fixture artifact.

## Consequence

**Severity:** cosmetic

A reader resolving H4a's or H6a's "committed multi-feature fixture `.loom`" back-reference to its producing leaf finds it named "representative multi-feature fixture `.loom`" and must infer the two refer to one artifact. No implementer is blocked; the cost is the inference and a slight risk of treating them as separate fixtures.
