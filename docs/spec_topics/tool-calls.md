# Tool Calls

Theta code calls a Pi tool or a `.theta` callable directly via the bare-identifier form `<name>(args)`, where `<name>` is an entry in the theta's *callable set* (the resolved `tools:` field after any `as` rename and the default hyphen→underscore theta-basename rewrite; see [Parameters and Frontmatter](./frontmatter.md) and [Glossary — `callable set`](./glossary.md)). The same callable set is what the model sees during a `@`...`` query — the declaration is shared between the model-driven and code-driven call paths.

```theta
let contents = read({ path: "src/main.ts" })?
let matches  = grep({ pattern: "TODO", path: "src" })?
let summary  = summarise(contents)?
let label    = triage(summary)?
```

**No conversation turn.** A tool call is a direct call against Pi's tool runtime (or, for a `.theta` callable, a fresh subagent invocation; see below). It does **not** add a turn to the theta's conversation, does **not** consume model tokens, and does **not** appear in the conversation transcript. This is the deliberate distinction from `@`...`` queries: queries cross code → model in the current conversation; tool calls cross code → side-effect (or code → child conversation, for a `.theta` callable) without disturbing the current one.

**Argument shape.** Pi tools take a single object argument matching the tool's input schema (TypeBox / JSON Schema, exposed by Pi at registration). The argument's **shape** is fixed: a single bare object literal written inline at the call site (`read({ path: base + "/main.ts" })`) — the second of the two documented exceptions to the bare-object-literal prohibition in [Expression Sublanguage — Object construction](./expressions.md) — with the field names supplied by the Pi tool's registered input schema. Its **field values are full Theta expressions**: identifier references, operators, function and tool calls, `?`, `${...}` string interpolation, query results already bound via `let`, and nested arrays and objects whose leaves are themselves expressions are all admitted (see [Grammar Appendix — Pi-tool argument grammar](./grammar.md#pi-tool-argument-grammar) for the `ToolArg` / `ToolField` productions). This is the one respect in which a Pi-tool argument differs from a `params:` default, whose RHS stays restricted to the [Theta literal sublanguage](./grammar.md#theta-literal-sublanguage) (see [Frontmatter — Defaults](./frontmatter.md)). Field-value expressions **evaluate left-to-right in source order, at call time, before dispatch**; a panic or an early-returning `?` inside a field expression follows ordinary expression semantics and aborts the call — the tool is not dispatched. The shape rule is otherwise unchanged: the argument must be written inline as a bare object literal, so passing a whole `let`-bound object (`read(args)`) is not admitted — the inline literal is what lets the tool's registered input schema supply the field shape. A multi-argument form (`read({...}, {...})`) is `theta/parse/tool-arg-arity` regardless of the argument shapes. `.theta` callables take their callee `params:` as already-typed values, positionally in declaration order — the same argument-binding rules `invoke(...)` uses; bare object literals do not apply (the callee's typed `params:` supply the names, not an external schema). Type-mismatch handling is asymmetric across the two callee kinds, because "statically resolvable" is defined only over `.theta` callees (a callee referenced by a literal `invoke(...)` or by a `.theta` entry in `tools:`; see [Invocation — Static resolution](./invocation.md#static-resolution)) and a Pi tool is not a `.theta` file. For a `.theta` callable, an argument that does not type-check against the callee's `params:` surfaces as `theta/parse/tool-arg-type-mismatch` when the callee is statically resolvable; otherwise the runtime AJV check is the safety net and surfaces at runtime as `Err(InvokeInfraError { cause: "validation", ... })`, matching the non-statically-resolvable `invoke(...)` input-validation arm in [Invocation — Failures](./invocation.md) (see **Failures** below). For a Pi tool, "statically resolvable" has no application, so a Pi-tool argument that does not match the tool's input schema is, in general, never a parse error — it is caught by the runtime AJV check and surfaces at runtime as `Err(CodeToolError { cause: "validation", ... })` (see **Failures** below), with one narrow, sound parse-time exception described in the **Provable-disjointness check** paragraph below. The `theta/parse/tool-arg-arity` shape check remains live for Pi-tool calls regardless; the type-mismatch check is runtime-only on the Pi-tool arm except where that parse-time provable-disjointness check fires. Argument arity is checked under the same rules as `invoke(...)` — see [Invocation — Argument arity](./invocation.md#argument-binding) for the parse-time vs runtime split (`theta/parse/invoke-arity-too-few` / `theta/parse/invoke-arity-too-many` apply equally to a `.theta` callable call). The path-restriction rule from [Invocation — Resolution](./invocation.md) also applies to `.theta` paths used as `tools:` entries: a path that escapes the active discovery roots is rejected with `theta/load/invoke-path-escape` and the callable is not created. Slash-command argument binding (LLM-driven; see [Slash-Command Argument Binding](./binder.md)) does not apply here — code-side callers pass typed values directly.

**Provable-disjointness check (parse time).** The parser emits an *error*-severity `theta/parse/tool-arg-schema-conflict` diagnostic when, and only when, a field-value expression's static type is **provably disjoint** from the tool's registered input-schema type for that field, mapped through the [schema subset](./schema-subset.md). *Provably disjoint* means the accepted-value sets of the mapped static type and the schema type have empty intersection under the subset mapping. This is sound: a provable disjointness guarantees the runtime AJV check would reject the same value, so the parse error only front-runs a certain `Err(CodeToolError { cause: "validation", ... })`. The check is scoped to provable disjointness only — anything the schema subset cannot represent (a `format`, a `pattern`, a numeric refinement such as `minimum`/`maximum`, or a widened union with at least one satisfiable arm) is **not** provable and MUST fall through to the runtime AJV check rather than erroring. The check therefore never rejects a program the runtime AJV check would accept; a merely-suspicious or narrowing mismatch is not a parse error.

**Return type.** The result type depends on the callee kind:

| Callee kind | Return type | Notes |
|---|---|---|
| Pi tool | `Result<string, QueryError>` | theta 1.0 returns the tool's final output as a single `string` (mirroring untyped queries). Pi tool definitions ship an input schema but no output schema; provider tool-use conventions treat outputs as freeform text the model interprets. |
| Registered theta (subagent-mode) | `Result<T, QueryError>` where `T` is the callee's inferred return type | Same inference rule as `invoke<T>(...)`: when the callee `.theta` is statically resolvable per [Invocation — Static resolution](./invocation.md#static-resolution), its inferred return type (per [Function Definitions — Theta return type](./functions.md#theta-return-type)) flows into the call site. Otherwise the runtime AJV check enforces it. |

As with queries and `invoke`, the call returns a `Result`; use `?` to propagate failure or `match` to handle.

**Failures.** Failure surfacing splits by callee kind. A **Pi-tool** call's failures surface as the `CodeToolError` variant of `QueryError`; a **`.theta`-callable** call's failures cascade instead through `InvokeCalleeError` / `InvokeInfraError`, with `CodeToolError` arising only in the `"unknown_tool"` safety-net case (the `.theta`-callable cascade paragraph below states this split in full). The `CodeToolError` schema lives in [Errors and Results — QueryError variants](./errors-and-results/queryerror-variants.md#queryerror-variants); its cause enum — the contract surface for the Pi-tool arm and the `.theta`-callable `"unknown_tool"` arm — is `validation` (arguments failed input-schema validation), `execution` (tool's `execute()` threw or returned `isError: true`), `cancelled` (AbortSignal fired), or `unknown_tool` (callable was lost across a file-watcher reload — cannot occur during the lifetime of a single invocation, since the load-time `tools:` table holds strong references to resolved callables; see [Parameters and Frontmatter](./frontmatter.md)).

*Outcome enumeration (normative).* The `CodeToolError.cause` enum is the contract surface for theta authors and is closed at the four members above; it is **not** widened to cover every observable disposition of `execute()`. Three further tool-side outcomes are routed off the `CodeToolError` surface and pinned here so the lowering rule in [Pi Integration Contract — Tool execution from theta code](./pi-integration-contract/host-interfaces-core.md#tool-execution-from-theta-code) is total over what `execute()` can do:

- *Non-settling Promise.* If `execute()` returns a Promise that never settles, the runtime makes no theta 1.0 attempt at internal timeout enforcement (consistent with [`spec.md` — Hard ceilings, NOCEIL-1](./overview-and-orientation.md#hard-runtime-ceilings)). The tool-call expression blocks at its `await` until `thetaAbort.signal` fires, at which point cancellation surfaces through the existing `cause: "cancelled"` path. The deferred per-call timeout extension is reserved on the *theta 1.0 seam — per-call timeout* below; see [Future Considerations](./future-considerations.md).
- *Pre-evaluation setup throw inside the `.theta`-callable adapter.* When the model emits a parallel tool call (per **Concurrency** below) targeting a `.theta` callable and the adapter's pre-evaluation dispatch-setup sequence throws — i.e. one of the steps pinned in [Pi Integration Contract — `ActiveInvocationRegistry` Dispatch-site setup wrap](./pi-integration-contract/active-invocation-registry.md#active-invocation-registry) (`new AbortController()`, `ActiveInvocationRegistry.add`, the inbound `signal.addEventListener('abort', …)` attach that registers the forwarding listener on the tool-exposed adapter's `signal` parameter, the subagent child-process spawn failure in subagent mode, or the post-spawn `thetaAbort.signal.addEventListener('abort', …)` attach that wires the one-shot RPC-`abort`-command listener) raises before the theta body runs — the adapter MUST translate the captured error into a clean `{ isError: true, content: [{ type: "text", text: "theta <name> aborted with internal error: <error.message>" }] }` value (where `<name>` is the *callable set* identifier from this page's opening sentence — the caller's post-`as`, post-hyphen→underscore-rewrite entry in `tools:`; deliberately the bare entry name, **not** the slash-prefixed `/<name>` form used by the user-facing framings in [Errors and Results — Runtime panics](./errors-and-results/error-model.md#runtime-panics) and [Pi Integration Contract — Runtime event channel](./pi-integration-contract/runtime-event-channel.md), because this adapter is entered from the model's parallel-tool batch and carries no slash context) and return it to Pi normally; in parallel, the runtime emits one `theta/runtime/internal-error` diagnostic (carrying `error.message` in `message` and `error.stack` — or `"<no stack available>"` when falsy — in `hint`) and one `theta-system-note` so an operator observes the failure even though the model's surface is the tool result. The adapter MUST NOT let the throw propagate as an unhandled rejection out of `tool.execute(...)`: silent loss of the call into Pi's parallel-tool-call dispatch is forbidden by the same "silent success on denial is forbidden" stance pinned in [Pi Integration Contract — No additional access channels](./pi-integration-contract/trust-boundary.md#no-extra-mediation). The runtime-defect surface is the routing class — see [Errors and Results — Runtime panics](./errors-and-results/error-model.md#runtime-panics) — and the `.theta`-adapter `{ isError: true, ... }` translation is the per-surface form.
- *Non-conforming return shape.* If `execute()` resolves with a value whose shape violates the `{ content, isError }` envelope (the resolved value is not an object, `content` is not iterable, an entry is missing `type` / `text`, etc.), the lowering wrapper in [Pi Integration Contract — Tool execution from theta code](./pi-integration-contract/host-interfaces-core.md#tool-execution-from-theta-code) catches the resulting throw and routes it through the `theta/runtime/internal-error` runtime-defect surface with `details.kind = "tool-return-shape"`, **not** through `CodeToolError`. A non-conforming Pi tool is a defect in that tool, not a value theta authors can usefully `match` on; the routing class is the same one [Errors and Results — Runtime panics](./errors-and-results.md) uses for unanticipated SDK throws.
- *Post-cancel resolution.* If `thetaAbort` fires and the tool-call checkpoint surfaces `cause: "cancelled"`, any later settlement of the underlying Promise (resolve or reject, `Ok` or `isError: true` or throw) is discarded — no rebinding ([CNCL-1](./cancellation.md#cncl-1)), no second `Err` ([CNCL-2](./cancellation.md#cncl-2)), no second `RuntimeEvent` ([CNCL-3](./cancellation.md#cncl-3)). The discard is mechanically realised by a swallowing handler the runtime MUST attach to the `execute()` Promise at construction time so that a late rejection arriving after the checkpoint surfaced `cause: "cancelled"` does **not** surface as a Node `unhandledRejection` process event; the handler MUST be installed before the first microtask boundary (a lazily-attached `.catch(() => {})` will not catch a rejection already queued for `unhandledRejection`). The discard is total and applies regardless of the late rejection's `.message` (an `OOMError`-style late reject is still discarded; promotion to `theta/runtime/internal-error` would re-introduce the second-event surface this rule forbids), and the discarded value is not traversed (cleanup of any handles it carries is the tool implementation's responsibility, not the runtime's). See [Cancellation — Race semantics](./cancellation.md) for the normative discard rule and its symmetric application to `@`-query provider Promises and `invoke` callee top-level Promises.

`CodeToolError` is distinct from `ModelToolError`, and the two surfaces dispose of the *same* `execute()` outcomes differently because the contexts differ: a code-side `<name>(args)` call lowers an `execute()` throw or an `{ content, isError: true }` return to `Err(CodeToolError { cause: "execution", ... })`, whereas the model-driven `@`...`` tool-call loop feeds that same outcome back to the model as the failing `tool_use` block's tool-result and continues looping (see [Query — Tool calls during a query](./query.md)). `ModelToolError` is reserved for the non-recoverable adapter-layer conditions enumerated in [Errors and Results — `ModelToolError`](./errors-and-results/queryerror-variants.md#queryerror-variants). The variants also carry different fields — a code-side tool call has no `tool_call_id` issued by Pi's tool-loop machinery and no `raw_response` from the model; a model-loop adapter failure has both. Authors who want to handle every tool failure uniformly write two `match` arms or a final `_ => ...` catch-all.

For a `.theta` callable, failures the callee returned cascade through the standard `InvokeCalleeError` variant (the call is, semantically, an `invoke`); failures from the theta infrastructure itself (callee unloadable, validation mismatch on the return value) cascade through `InvokeInfraError`. The only situation where `CodeToolError` arises for a `.theta` callable is theta 1.0's `"unknown_tool"` safety net. An input-side argument validation failure on a `.theta`-callable call, when the callee is *not* statically resolvable (so the parse-time `theta/parse/tool-arg-type-mismatch` check did not fire), surfaces as `Err(InvokeInfraError { cause: "validation", ... })` — the same `invoke`-shaped arm `invoke(...)` uses for that case, and distinct from the return-side `cause: "return_validation"`. The consequence is that an author handling every input-validation failure across both callee kinds in one place must `match` both `CodeToolError { cause: "validation", ... }` (the Pi-tool arm) and `InvokeInfraError { cause: "validation", ... }` (the `.theta`-callable arm).

<a id="concurrency"></a>**Concurrency.** theta 1.0 tool calls from theta code are sequential and synchronous-looking: the runtime awaits each call's underlying Promise before evaluating the next expression, so the theta interpreter yields to Pi's event loop during the wait (the TUI render loop, keypress handlers, signals, and other Pi machinery continue to run — the call is non-blocking at the runtime level even though it appears synchronous to the author). Streaming partial results (Pi's `onUpdate` callback) are not surfaced to theta code. Concurrent tool execution exists in Pi's model-driven "parallel tool mode" inside `@`...`` queries — when the model issues multiple tool-use blocks in one assistant message, Pi runs them concurrently. When such a batch mixes successful and failing calls, the runtime awaits every call in the batch to settle before constructing the next user turn; each call's outcome is lowered independently — a failing sibling (`execute()` throwing or resolving `{ content, isError: true }`) becomes that `tool_use` block's `isError: true` tool-result and is fed back to the model alongside the successful siblings' results, per [Query — Tool calls during a query](./query.md). The batch consumes one `tool_loop.max_rounds` slot regardless of the success/failure mix (see [Query — Tool-call loop bound](./query/query-tool-loop.md#tool-call-loop-bound)). theta 1.0 permits this for both Pi tools and `.theta` callables: when the model emits parallel tool calls during a query whose callable set includes a `.theta` callable, each invocation spawns its own child `pi` process with its own captured callable set, its own derived `AbortSignal`, and its own private (ephemeral `--no-session`) transcript (see [Pi Integration Contract](./pi-integration-contract.md)). The wrapping `defineTool` adapter for a `.theta` callable is therefore re-entrant: two concurrent calls into the same callee execute as two independent invocations on the event loop. The schema cache (see [Implementation Notes — Runtime](./implementation-notes.md)) is read-mostly and safe to share across concurrent invocations; compiled validators are immutable post-compile and cache writes only occur on initial compile or file-watcher invalidation. No theta-level concurrency primitive (e.g. a `parallel { ... }` block) is exposed in theta 1.0; see [Future Considerations](./future-considerations.md).

<a id="parallel-tool-batch-delivery-presupposition"></a>*Parallel-tool-mode batch delivery (consumption posture).* The batch behaviour described above — that when the model issues multiple `tool_use` blocks in one assistant message Pi runs them concurrently, awaits every call in the batch to settle before constructing the next user turn, and lowers each call's outcome independently (a failing sibling becoming that `tool_use` block's `isError: true` tool-result fed back to the model alongside the successful siblings' results) — presupposes that real Pi's model-driven parallel-tool mode delivers the batch with exactly that shape. theta performs no batching or settle-barrier of its own on this path, so the concurrent-execution, settle-all-before-next-turn, and independent-per-sibling-lowering properties are entirely Pi-side: were a Pi minor to construct the next user turn before every sibling settled, or to collapse a failing sibling rather than feeding its `isError: true` tool-result back alongside the successful siblings', the lowering rules above would be falsified. theta records this as a theta-side consumption posture because `@earendil-works/pi-coding-agent` encodes no typed contract pinning the parallel-tool-batch delivery shape, and the property carries no build-time SDK surface-inventory signal. Detection therefore routes to editorial review per item (an) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure).

> **theta 1.0 seam — per-call timeout.** The runtime-internal options record passed into the tool-call primitive (carrying per-call configuration not surfaced to authors in theta 1.0 — cancellation hookup, the captured `ToolDefinition` reference, the synthesised `theta-direct:` `toolCallId`) is an **open struct**, not a closed positional record. theta 1.0 reserves the right to add fields in a minor revision (e.g. a per-call timeout) without breaking call sites or test fixtures. *Non-normative implementation note.* Consumers tolerate unknown fields rather than assuming a closed positional record; this is a maintainability convention enabling the additive extension, not an observable conformance point. The seam is what allows the deferred per-call timeout extension in [Future Considerations](./future-considerations.md) to land additively.

**Relationship with `invoke`.** `invoke("./path.theta", ...)` and a `.theta` callable call (`my_summariser(...)` after listing `./summariser.theta` in `tools:`) are operationally equivalent for subagent-mode callees — both spawn a fresh isolated conversation, both validate the return value against the callee's inferred or annotated schema, and both surface failures through the same `QueryError` variants, and both apply the arity, return-type-compatibility, and path-restriction rules from [Invocation](./invocation.md). Subagent-mode callees see only their own callable set inside their spawned child `pi` process (via its `--tools` allowlist), never the caller's; prompt-mode callees (reachable only via `invoke(...)`) follow the prompt → prompt tools-lifetime swap defined in [Invocation — Tools and model](./invocation.md). Prompt-mode callees are rejected by `tools:` for the reason given in [Parameters and Frontmatter](./frontmatter.md). The two surfaces share a single error model and a single schema-lowering pipeline.
