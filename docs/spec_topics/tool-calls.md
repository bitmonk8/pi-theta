# Tool Calls

Loom code calls a Pi tool or a `.loom` callable directly via the bare-identifier form `<name>(args)`, where `<name>` is an entry in the loom's *callable set* (the resolved `tools:` field after any `as` rename and the default hyphen‚Üíunderscore loom-basename rewrite; see [Parameters and Frontmatter](./frontmatter.md) and [Glossary ‚Äî `callable set`](./glossary.md)). The same callable set is what the model sees during a `@`...`` query ‚Äî the declaration is shared between the model-driven and code-driven call paths.

```loom
let contents = read({ path: "src/main.ts" })?
let matches  = grep({ pattern: "TODO", path: "src" })?
let summary  = summarise(contents)?
let label    = triage(summary)?
```

**No conversation turn.** A tool call is a direct call against Pi's tool runtime (or, for a `.loom` callable, a fresh subagent invocation; see below). It does **not** add a turn to the loom's conversation, does **not** consume model tokens, and does **not** appear in the conversation transcript. This is the deliberate distinction from `@`...`` queries: queries cross code ‚Üí model in the current conversation; tool calls cross code ‚Üí side-effect (or code ‚Üí child conversation, for a `.loom` callable) without disturbing the current one.

**Argument shape.** Pi tools take a single object argument matching the tool's input schema (TypeBox / JSON Schema, exposed by Pi at registration). The argument is written as a bare object literal (`read({ path: "src/main.ts" })`) ‚Äî the second of the two documented exceptions to the bare-object-literal prohibition in [Expression Sublanguage ‚Äî Object construction](./expressions.md). The literal is parsed by the [Loom literal sublanguage](./grammar.md#loom-literal-sublanguage), the same notation used for `params:` defaults: primitive literals, `null`, array literals, bare-key object literals, `Enum.Variant` access, and variant-schema construction (`Cat { ... }`) are admitted. Operators, `let`-bound identifier references, function calls inside the argument, and `${...}` interpolation are not ‚Äî violations are `loom/parse/tool-arg-not-literal`. To pass a computed value, bind it via `let` first and reference it inside a typed `params:` callee or use `invoke(...)`. A multi-argument form (`read({...}, {...})`) is `loom/parse/tool-arg-arity` regardless of the argument shapes. `.loom` callables take their callee `params:` as already-typed values, positionally in declaration order ‚Äî the same argument-binding rules `invoke(...)` uses; bare object literals do not apply (the callee's typed `params:` supply the names, not an external schema). Type-mismatch handling is asymmetric across the two callee kinds, because "statically resolvable" is defined only over `.loom` callees (a callee referenced by a literal `invoke(...)` or by a `.loom` entry in `tools:`; see [Invocation ‚Äî Static resolution](./invocation.md#static-resolution)) and a Pi tool is not a `.loom` file. For a `.loom` callable, an argument that does not type-check against the callee's `params:` surfaces as `loom/parse/tool-arg-type-mismatch` when the callee is statically resolvable; otherwise the runtime AJV check is the safety net and surfaces at runtime as `Err(InvokeInfraError { cause: "validation", ... })`, matching the non-statically-resolvable `invoke(...)` input-validation arm in [Invocation ‚Äî Failures](./invocation.md) (see **Failures** below). For a Pi tool, "statically resolvable" has no application, so a Pi-tool argument that does not match the tool's input schema is never a parse error ‚Äî it is always caught by the runtime AJV check and surfaces at runtime as `Err(CodeToolError { cause: "validation", ... })` (see **Failures** below). The `loom/parse/tool-arg-not-literal` and `loom/parse/tool-arg-arity` parse checks remain live for Pi-tool calls regardless; only the type-mismatch check is runtime-only on the Pi-tool arm. Argument arity is checked under the same rules as `invoke(...)` ‚Äî see [Invocation ‚Äî Argument arity](./invocation.md#argument-binding) for the parse-time vs runtime split (`loom/parse/invoke-arity-too-few` / `loom/parse/invoke-arity-too-many` apply equally to a `.loom` callable call). The path-restriction rule from [Invocation ‚Äî Resolution](./invocation.md) also applies to `.loom` paths used as `tools:` entries: a path that escapes the active discovery roots is rejected with `loom/load/invoke-path-escape` and the callable is not created. Slash-command argument binding (LLM-driven; see [Slash-Command Argument Binding](./binder.md)) does not apply here ‚Äî code-side callers pass typed values directly.

**Return type.** The result type depends on the callee kind:

| Callee kind | Return type | Notes |
|---|---|---|
| Pi tool | `Result<string, QueryError>` | loom 1.0 returns the tool's final output as a single `string` (mirroring untyped queries). Pi tool definitions ship an input schema but no output schema; provider tool-use conventions treat outputs as freeform text the model interprets. |
| Registered loom (subagent-mode) | `Result<T, QueryError>` where `T` is the callee's inferred return type | Same inference rule as `invoke<T>(...)`: when the callee `.loom` is statically resolvable per [Invocation ‚Äî Static resolution](./invocation.md#static-resolution), its inferred return type (per [Function Definitions ‚Äî Loom return type](./functions.md#loom-return-type)) flows into the call site. Otherwise the runtime AJV check enforces it. |

As with queries and `invoke`, the call returns a `Result`; use `?` to propagate failure or `match` to handle.

**Failures.** Tool-call failures surface as the `CodeToolError` variant of `QueryError`. The schema lives in [Errors and Results ‚Äî QueryError variants](./errors-and-results/queryerror-variants.md#queryerror-variants); the cause enum is `validation` (arguments failed input-schema validation), `execution` (tool's `execute()` threw or returned `isError: true`), `cancelled` (AbortSignal fired), or `unknown_tool` (callable was lost across a file-watcher reload ‚Äî cannot occur during the lifetime of a single invocation, since the load-time `tools:` table holds strong references to resolved callables; see [Parameters and Frontmatter](./frontmatter.md)).

*Outcome enumeration (normative).* The `CodeToolError.cause` enum is the contract surface for loom authors and is closed at the four members above; it is **not** widened to cover every observable disposition of `execute()`. Three further tool-side outcomes are routed off the `CodeToolError` surface and pinned here so the lowering rule in [Pi Integration Contract ‚Äî Tool execution from loom code](./pi-integration-contract/host-interfaces-core.md#tool-execution-from-loom-code) is total over what `execute()` can do:

- *Non-settling Promise.* If `execute()` returns a Promise that never settles, the runtime makes no loom 1.0 attempt at internal timeout enforcement (consistent with [`spec.md` ‚Äî Hard ceilings, NOCEIL-1](./overview-and-orientation.md#hard-runtime-ceilings)). The tool-call expression blocks at its `await` until `loomAbort.signal` fires, at which point cancellation surfaces through the existing `cause: "cancelled"` path. The deferred per-call timeout extension is reserved on the *loom 1.0 seam ‚Äî per-call timeout* below; see [Future Considerations](./future-considerations.md).
- *Pre-evaluation setup throw inside the `.loom`-callable adapter.* When the model emits a parallel tool call (per **Concurrency** below) targeting a `.loom` callable and the adapter's pre-evaluation dispatch-setup sequence throws ‚Äî i.e. one of the steps pinned in [Pi Integration Contract ‚Äî `ActiveInvocationRegistry` Dispatch-site setup wrap](./pi-integration-contract/active-invocation-registry.md#active-invocation-registry) (`new AbortController()`, `ActiveInvocationRegistry.add`, the inbound `signal.addEventListener('abort', ‚Ä¶)` attach that registers the forwarding listener on the tool-exposed adapter's `signal` parameter, `createAgentSession(...)` rejection in subagent mode, or the post-`createAgentSession` `loomAbort.signal.addEventListener('abort', ‚Ä¶)` attach that wires the one-shot `AgentSession.abort()` listener) raises before the loom body runs ‚Äî the adapter MUST translate the captured error into a clean `{ isError: true, content: [{ type: "text", text: "loom <name> aborted with internal error: <error.message>" }] }` value (where `<name>` is the *callable set* identifier from this page's opening sentence ‚Äî the caller's post-`as`, post-hyphen‚Üíunderscore-rewrite entry in `tools:`; deliberately the bare entry name, **not** the slash-prefixed `/<name>` form used by the user-facing framings in [Errors and Results ‚Äî Runtime panics](./errors-and-results/error-model.md#runtime-panics) and [Pi Integration Contract ‚Äî Runtime event channel](./pi-integration-contract/runtime-event-channel.md), because this adapter is entered from the model's parallel-tool batch and carries no slash context) and return it to Pi normally; in parallel, the runtime emits one `loom/runtime/internal-error` diagnostic (carrying `error.message` in `message` and `error.stack` ‚Äî or `"<no stack available>"` when falsy ‚Äî in `hint`) and one `loom-system-note` so an operator observes the failure even though the model's surface is the tool result. The adapter MUST NOT let the throw propagate as an unhandled rejection out of `tool.execute(...)`: silent loss of the call into Pi's parallel-tool-call dispatch is forbidden by the same "silent success on denial is forbidden" stance pinned in [Pi Integration Contract ‚Äî No additional access channels](./pi-integration-contract/trust-boundary.md#no-extra-mediation). The runtime-defect surface is the routing class ‚Äî see [Errors and Results ‚Äî Runtime panics](./errors-and-results/error-model.md#runtime-panics) ‚Äî and the `.loom`-adapter `{ isError: true, ... }` translation is the per-surface form.
- *Non-conforming return shape.* If `execute()` resolves with a value whose shape violates the `{ content, isError }` envelope (the resolved value is not an object, `content` is not iterable, an entry is missing `type` / `text`, etc.), the lowering wrapper in [Pi Integration Contract ‚Äî Tool execution from loom code](./pi-integration-contract/host-interfaces-core.md#tool-execution-from-loom-code) catches the resulting throw and routes it through the `loom/runtime/internal-error` runtime-defect surface with `details.kind = "tool-return-shape"`, **not** through `CodeToolError`. A non-conforming Pi tool is a defect in that tool, not a value loom authors can usefully `match` on; the routing class is the same one [Errors and Results ‚Äî Runtime panics](./errors-and-results.md) uses for unanticipated SDK throws.
- *Post-cancel resolution.* If `loomAbort` fires and the tool-call checkpoint surfaces `cause: "cancelled"`, any later settlement of the underlying Promise (resolve or reject, `Ok` or `isError: true` or throw) is discarded ‚Äî no rebinding ([CNCL-1](./cancellation.md#cncl-1)), no second `Err` ([CNCL-2](./cancellation.md#cncl-2)), no second `RuntimeEvent` ([CNCL-3](./cancellation.md#cncl-3)). The discard is mechanically realised by a swallowing handler the runtime MUST attach to the `execute()` Promise at construction time so that a late rejection arriving after the checkpoint surfaced `cause: "cancelled"` does **not** surface as a Node `unhandledRejection` process event; the handler MUST be installed before the first microtask boundary (a lazily-attached `.catch(() => {})` will not catch a rejection already queued for `unhandledRejection`). The discard is total and applies regardless of the late rejection's `.message` (an `OOMError`-style late reject is still discarded; promotion to `loom/runtime/internal-error` would re-introduce the second-event surface this rule forbids), and the discarded value is not traversed (cleanup of any handles it carries is the tool implementation's responsibility, not the runtime's). See [Cancellation ‚Äî Race semantics](./cancellation.md) for the normative discard rule and its symmetric application to `@`-query provider Promises and `invoke` callee top-level Promises.

`CodeToolError` is distinct from `ModelToolError`, and the two surfaces dispose of the *same* `execute()` outcomes differently because the contexts differ: a code-side `<name>(args)` call lowers an `execute()` throw or an `{ content, isError: true }` return to `Err(CodeToolError { cause: "execution", ... })`, whereas the model-driven `@`...`` tool-call loop feeds that same outcome back to the model as the failing `tool_use` block's tool-result and continues looping (see [Query ‚Äî Tool calls during a query](./query.md)). `ModelToolError` is reserved for the non-recoverable adapter-layer conditions enumerated in [Errors and Results ‚Äî `ModelToolError`](./errors-and-results/queryerror-variants.md#queryerror-variants). The variants also carry different fields ‚Äî a code-side tool call has no `tool_call_id` issued by Pi's tool-loop machinery and no `raw_response` from the model; a model-loop adapter failure has both. Authors who want to handle every tool failure uniformly write two `match` arms or a final `_ => ...` catch-all.

For a `.loom` callable, failures the callee returned cascade through the standard `InvokeCalleeError` variant (the call is, semantically, an `invoke`); failures from the loom infrastructure itself (callee unloadable, validation mismatch on the return value) cascade through `InvokeInfraError`. The only situation where `CodeToolError` arises for a `.loom` callable is loom 1.0's `"unknown_tool"` safety net. An input-side argument validation failure on a `.loom`-callable call, when the callee is *not* statically resolvable (so the parse-time `loom/parse/tool-arg-type-mismatch` check did not fire), surfaces as `Err(InvokeInfraError { cause: "validation", ... })` ‚Äî the same `invoke`-shaped arm `invoke(...)` uses for that case, and distinct from the return-side `cause: "return_validation"`. The consequence is that an author handling every input-validation failure across both callee kinds in one place must `match` both `CodeToolError { cause: "validation", ... }` (the Pi-tool arm) and `InvokeInfraError { cause: "validation", ... }` (the `.loom`-callable arm).

<a id="concurrency"></a>**Concurrency.** loom 1.0 tool calls from loom code are sequential and synchronous-looking: the runtime awaits each call's underlying Promise before evaluating the next expression, so the loom interpreter yields to Pi's event loop during the wait (the TUI render loop, keypress handlers, signals, and other Pi machinery continue to run ‚Äî the call is non-blocking at the runtime level even though it appears synchronous to the author). Streaming partial results (Pi's `onUpdate` callback) are not surfaced to loom code. Concurrent tool execution exists in Pi's model-driven "parallel tool mode" inside `@`...`` queries ‚Äî when the model issues multiple tool-use blocks in one assistant message, Pi runs them concurrently. When such a batch mixes successful and failing calls, the runtime awaits every call in the batch to settle before constructing the next user turn; each call's outcome is lowered independently ‚Äî a failing sibling (`execute()` throwing or resolving `{ content, isError: true }`) becomes that `tool_use` block's `isError: true` tool-result and is fed back to the model alongside the successful siblings' results, per [Query ‚Äî Tool calls during a query](./query.md). The batch consumes one `tool_loop.max_rounds` slot regardless of the success/failure mix (see [Query ‚Äî Tool-call loop bound](./query/query-tool-loop.md#tool-call-loop-bound)). loom 1.0 permits this for both Pi tools and `.loom` callables: when the model emits parallel tool calls during a query whose callable set includes a `.loom` callable, each invocation spawns its own `AgentSession` with its own captured callable set, its own derived `AbortSignal`, and its own private transcript (see [Pi Integration Contract](./pi-integration-contract.md)). The wrapping `defineTool` adapter for a `.loom` callable is therefore re-entrant: two concurrent calls into the same callee execute as two independent invocations on the event loop. The schema cache (see [Implementation Notes ‚Äî Runtime](./implementation-notes.md)) is read-mostly and safe to share across concurrent invocations; compiled validators are immutable post-compile and cache writes only occur on initial compile or file-watcher invalidation. No loom-level concurrency primitive (e.g. a `parallel { ... }` block) is exposed in loom 1.0; see [Future Considerations](./future-considerations.md).

<a id="parallel-tool-batch-delivery-presupposition"></a>*Parallel-tool-mode batch delivery (consumption posture).* The batch behaviour described above ‚Äî that when the model issues multiple `tool_use` blocks in one assistant message Pi runs them concurrently, awaits every call in the batch to settle before constructing the next user turn, and lowers each call's outcome independently (a failing sibling becoming that `tool_use` block's `isError: true` tool-result fed back to the model alongside the successful siblings' results) ‚Äî presupposes that real Pi's model-driven parallel-tool mode delivers the batch with exactly that shape. loom performs no batching or settle-barrier of its own on this path, so the concurrent-execution, settle-all-before-next-turn, and independent-per-sibling-lowering properties are entirely Pi-side: were a Pi minor to construct the next user turn before every sibling settled, or to collapse a failing sibling rather than feeding its `isError: true` tool-result back alongside the successful siblings', the lowering rules above would be falsified. loom records this as a loom-side consumption posture because `@earendil-works/pi-coding-agent` encodes no typed contract pinning the parallel-tool-batch delivery shape, and the property carries no build-time SDK surface-inventory signal. Detection therefore routes to editorial review per item (an) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure).

> **loom 1.0 seam ‚Äî per-call timeout.** The runtime-internal options record passed into the tool-call primitive (carrying per-call configuration not surfaced to authors in loom 1.0 ‚Äî cancellation hookup, the captured `ToolDefinition` reference, the synthesised `loom-direct:` `toolCallId`) is an **open struct**, not a closed positional record. loom 1.0 reserves the right to add fields in a minor revision (e.g. a per-call timeout) without breaking call sites or test fixtures. *Non-normative implementation note.* Consumers tolerate unknown fields rather than assuming a closed positional record; this is a maintainability convention enabling the additive extension, not an observable conformance point. The seam is what allows the deferred per-call timeout extension in [Future Considerations](./future-considerations.md) to land additively.

**Relationship with `invoke`.** `invoke("./path.loom", ...)` and a `.loom` callable call (`my_summariser(...)` after listing `./summariser.loom` in `tools:`) are operationally equivalent for subagent-mode callees ‚Äî both spawn a fresh isolated conversation, both validate the return value against the callee's inferred or annotated schema, and both surface failures through the same `QueryError` variants, and both apply the arity, return-type-compatibility, and path-restriction rules from [Invocation](./invocation.md). Subagent-mode callees see only their own callable set inside the spawned `AgentSession`, never the caller's; prompt-mode callees (reachable only via `invoke(...)`) follow the prompt ‚Üí prompt tools-lifetime swap defined in [Invocation ‚Äî Tools and model](./invocation.md). Prompt-mode callees are rejected by `tools:` for the reason given in [Parameters and Frontmatter](./frontmatter.md). The two surfaces share a single error model and a single schema-lowering pipeline.
