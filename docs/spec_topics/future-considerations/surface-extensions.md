# Surface extensions

## Tooling deferrals (no loom 1.0 impact)

- **LSP support** for `.loom` and `.warp` files (syntax highlighting, type checking, autocomplete).
- **`loom test` command** for dry-run execution that runs a loom against a recorded transcript or a stub model without hitting a live provider.
- **Richer runtime-event telemetry** — per-loom token aggregation, latency histograms, cost reporting, retention policies, and a consumer-facing read API for the `details: { event }` stream defined in [Pi Integration Contract — Runtime event channel](../pi-integration-contract.md). loom 1.0 ships only the per-occurrence emission and the `RuntimeEvent` payload shape; downstream consumers in loom 1.0 read from Pi's session transcript via existing surfaces.
- **Automated cadence for re-importing Pi patches** — loom 1.0 has no Renovate, no scheduled CI re-install, and no nightly cron that proactively re-imports candidate `@earendil-works/pi-coding-agent` patches into the loom contributor's typecheck cycle. The patch-skew window between a Pi patch landing on a user's machine and the loom-side snapshot edit shipping is therefore bounded only by contributor cadence; see [Pi Integration Contract — Patch-skew degradation contract](../pi-integration-contract/patch-skew-degradation.md#patch-skew-degradation-contract) for the loom 1.0 detection contract this defers.

## Surface extensions (loom 1.0 leaves a seam)

Each bullet below describes a deferred loom 1.0 extension and points at the normative paragraph that pins the loom 1.0 seam the extension consumes. The seams themselves live on the topic pages that own each surface; this page enumerates only what loom 1.0 chose not to do, and where to read the seam contract.

- **Per-call timeouts** on queries, tool calls, and invokes (loom 1.0 has only AbortSignal-driven cancellation; cf. [Cancellation](../cancellation.md)).
  *Anchored at:* [Query — Options surface](../query.md), [Tool Calls — Options surface](../tool-calls.md), [Invocation — Options surface](../invocation.md).
- **Pre-flight token-count check** before the runtime issues a user turn, so oversized rendered templates surface as `ContextOverflowError` without a wasted provider round-trip.
  *Anchored at:* [Query — loom 1.0 seam: pre-flight token-count nullability](../query/query-failure-and-repair.md#v1-seam-pre-flight-token-nullability) (the `tokens_used` / `tokens_limit` nullability is the seam; a pre-flight path may populate them from a local estimate without changing the variant shape).
- **Typed-query support for providers without named-tool forcing** (Gemini, OpenAI Responses, weak local models) via a JSON-mode / system-prompt fallback. loom 1.0 narrows typed queries to providers where pi-ai exposes named-tool forcing (per [Pi Integration Contract — Provider compatibility for typed queries](../pi-integration-contract.md)) and rejects unsupported providers with a load-time warning.
  *Anchored at:* [Pi Integration Contract — loom 1.0 seam: typed-query supported provider set](../pi-integration-contract/conversation-drive.md#v1-seam-typed-query-supported-provider-set).
- **Per-query overrides for `model`, `tools`, and `system`** (project → loom → query cascade).
  *Anchored at:* same query-options surface as per-call timeouts — [Query — Options surface](../query.md).
- **User-defined error types beyond `QueryError`.**
  *Anchored at:* [Errors and Results — loom 1.0 seam: discriminator type-openness](../errors-and-results/queryerror-variants.md#v1-seam-discriminator-type-openness) (one shared callout pins the seam for both this item and the `BinderError`-as-variant item below).
- **`BinderError` as a Loom-visible `QueryError` variant**, once looms become first-class values invocable from non-loom programmatic harnesses that need to observe binder failures structurally.
  *Anchored at:* same discriminator-type-openness seam as user-defined error types — [Errors and Results — loom 1.0 seam: discriminator type-openness](../errors-and-results/queryerror-variants.md#v1-seam-discriminator-type-openness).
  *Depends on:* First-class loom values (for non-loom harnesses needing structured observation).
- <a id="binder-refinement-loop"></a>**Binder refinement loop:** multi-turn `needs_info` negotiation (binder asks the user a clarifying question, gets a reply, retries) instead of loom 1.0's single-shot "system note then stop" behaviour; the same future revision also begins surfacing `ambiguous.candidates` as a user-facing pick-one disambiguation rather than a terminating system note.
  *Anchored at:* [Binder — loom 1.0 seam: binder refinement loop](../binder/binder-bypass-and-envelope.md#v1-seam-binder-refinement-loop) (the three loom 1.0 carriers — envelope-schema three-arm discriminator, `ambiguous.candidates` field, distinct failure-arm template prefixes — are pinned with MUST there).
- **Named-argument / key=value invocation syntax.**
  *Anchored at:* [Invocation — Argument-binding style](../invocation.md) (the AST node carries a `style: "positional" | "named"` discriminator; loom 1.0 only emits `"positional"`, but consumers match exhaustively).
- **Richer expression sublanguage inside frontmatter `system:`** (full `${expr}` interpolation rather than just `${param}` paths).
  *Anchored at:* [Parameters and Frontmatter — loom 1.0 seam: `system:` expression sublanguage](../frontmatter/frontmatter-fields-b-and-templates.md#v1-seam-system-expression-sublanguage) (the `${…}` parser is the same entry point used by `@`...`` query templates, restricted at parse time to dotted paths by a parser-level filter rather than a hand-rolled regex; widening the filter is additive).
- **Package-style (`@scope/pkg`) and project-rooted (`/looms/...`) `import` paths** — loom 1.0 supports relative paths only.
  *Anchored at:* [Imports — loom 1.0 seam: `Resolver` interface](../imports.md#v1-seam-resolver-interface) (every `.warp` import flows through a single named `Resolver` seam; additional implementations may be registered without rewriting import-site code).
- <a id="mid-loom-user-session-replacement"></a>**Mid-loom user-session replacement.** A future feature that calls `ctx.newSession()`, `ctx.fork()`, or `ctx.switchSession()` from inside a running loom invalidates the factory-captured `pi` reference used by the prompt-mode driver. The future implementation will need to re-acquire `pi` via `withSession` before the next `sendUserMessage`.
  *Anchored at:* [Pi Integration Contract — loom 1.0 seam: mid-loom user-session replacement](../pi-integration-contract/conversation-drive.md#v1-seam-mid-loom-user-session-replacement) (the single-captured-reference MUST is pinned there; the loom 1.0 invariant "loom 1.0 looms never trigger replacement" is the lifetime guarantee the seam preserves).
- <a id="symlink-resolution-hardening"></a>**Symlink-resolution hardening for invoke-path containment** — loom 1.0 implements the discovery-root containment check by `realpath`-then-`open`, accepting an irreducible kernel-level TOCTOU window between the two syscalls (see [Invocation — Resolution](../invocation.md)). A future pass replaces the two-step sequence with a single atomic resolve-and-open primitive — `openat2(..., RESOLVE_NO_SYMLINKS | RESOLVE_BENEATH)` on Linux, and the platform equivalents on macOS / Windows / other targets — closing the residual race without changing the loom 1.0-visible diagnostic codes (`loom/load/invoke-path-escape`) or `InvokeInfraError { cause: "load_failure" }` envelope.
  *Anchored at:* [Invocation — loom 1.0 seam: symlink-resolution hardening](../invocation.md#v1-seam-symlink-resolution-hardening) (the single-named-function MUST is pinned there; replacing its body is additive and does not perturb caller code).

### Surface extensions without a dedicated topic-page seam

<a id="surface-extensions-no-dedicated-seam"></a>

The five items below are **not** counted in the "Forward-compatibility seams" tally on [`spec.md` — Scope](../../spec/overview-and-orientation.md#scope): that tally's source set is the seams inventoried under [Surface extensions](#surface-extensions-v1-leaves-a-seam), and none of these five is inventoried there. Three ride [Parameters and Frontmatter — Unknown-key policy](../frontmatter.md) (or the closely related `tool_loop` field) as their forward-compatibility carrier: that policy provides forward tolerance for any unrecognised key but names no specific deferred field, and there is no per-item topic-page MUST that an implementer can fail to honour. Two are upstream Pi-API extensions whose loom 1.0 carrier is contingent on Pi extending its surface; if Pi never extends the surface, no loom 1.0 obligation activates. One of those two — the Pi-owned subagents item — nonetheless carries a defensive `> **loom 1.0 seam — <name>.**` blockquote (cited in its *Carrier* line) pinning the loom 1.0 collision-source-set invariant; because that blockquote pins an item in this sub-bucket rather than one inventoried under [Surface extensions](#surface-extensions-v1-leaves-a-seam), it is excluded from the 12-seam source set per [Governance — GOV-12](../governance/req-id-prefix-table-active-b.md#gov-12).

- **Per-loom `bind_temperature` knob**, if real usage shows authors need to tune the binder's nondeterminism budget.
  *Carrier:* [Parameters and Frontmatter — Unknown-key policy](../frontmatter.md).
- **Operator-level `looms.toolLoopMaxRounds` settings override** — loom 1.0 provides no operator-level fallback for `tool_loop.max_rounds`; the value is per-loom only. A future minor may add a `looms.toolLoopMaxRounds` settings key that supplies the default when `tool_loop:` is absent, mirroring the `bind_model` → `looms.binderModel` chain.
  *Carrier:* [Parameters and Frontmatter — `tool_loop`](../frontmatter.md) (the `tool_loop` field has no dedicated seam callout; the future settings key would be a new top-level configuration key, not a frontmatter widening).
- **User-overridable binder system prompt** — loom 1.0 fixes the binder prompt for predictability.
  *Carrier:* same unknown-key policy as `bind_temperature` — [Parameters and Frontmatter — Unknown-key policy](../frontmatter.md). The deferred extension also needs an injection point in the binder for an author-supplied prompt template; that injection point does not exist in loom 1.0.
- **Pi extension API: Pi-owned subagents exposed as enumerable slash commands.** Pi's `SlashCommandSource` union (`@earendil-works/pi-coding-agent`'s `core/slash-commands.d.ts`) is `"extension" | "prompt" | "skill"`; Pi-owned subagents are spawned via `createAgentSession(...)` and are not surfaced through `pi.getCommands()`, so the cross-format slash-name collision check in [Pi Integration Contract — Extension entry point](../pi-integration-contract.md) and [Discovery — Slash-name collisions at the same priority](../discovery.md) covers exactly three arms in loom 1.0.0. If a future Pi minor extends `SlashCommandSource` to include `"subagent"` (i.e. Pi-owned subagents become enumerable slash commands), the cross-format collision set widens to four arms and the cross-format collision test matrix gains a parallel `.md` subagent fixture.
  *Carrier:* [Pi Integration Contract — loom 1.0 seam: Pi-owned subagents collision source set](../pi-integration-contract/host-interfaces-services.md#v1-seam-pi-owned-subagents-collision-source-set) pins the loom 1.0 invariant (collision source set as a single named set) defensively; the future extension itself activates only if Pi extends `SlashCommandSource`, so no loom 1.0 forward-compatibility obligation rides this item beyond the invariant already captured at the cited callout.
- **Pi extension API: `argumentHint` field on `RegisteredCommand`** — Pi's built-in `.md` prompt templates carry an `argumentHint` that drives a distinct slot in the slash-command autocomplete dropdown, but extension-registered commands have no such slot. Loom's `argument-hint` frontmatter field is therefore consumed by the binder only in loom 1.0 (see [Slash-Command Argument Binding](../binder.md)). A future contributor either upstreams `argumentHint` to Pi's `RegisteredCommand` or revisits the loom 1.0 decision once Pi exposes the field.
  *Carrier:* [Parameters and Frontmatter](../frontmatter.md) (the binder simply has no hint to use beyond the frontmatter field) and [Slash-Command Invocation](../slash-invocation.md) (the `argument-hint` value is preserved on the loom's parsed AST and would only need to be threaded into the registration options).

### Surface extensions gated on an open design decision

<a id="surface-extensions-open-design"></a>

The item below is **not** counted in the "Forward-compatibility seams" tally on [`spec.md` — Scope](../../spec/overview-and-orientation.md#scope): unlike the committed deferrals inventoried under [Surface extensions](#surface-extensions-v1-leaves-a-seam), its committed loom 1.0 seam set is not yet settled, because the extension's eventual shape — and therefore which loom 1.0 carriers it composes with — depends on an open design decision (see the *Decision required* line on the item). Until that decision is made, the item reserves no committed forward-compatibility seam and so is excluded from the 12-seam source set per [Governance — GOV-12](../governance/req-id-prefix-table-active-b.md#gov-12). It nonetheless carries a defensive `> **loom 1.0 seam — <name>.**` blockquote (cited in its *Carrier* line) pinning a genuine loom 1.0 runtime invariant; because that blockquote pins an item in this sub-bucket rather than one inventoried under [Surface extensions](#surface-extensions-v1-leaves-a-seam), it is likewise excluded from the 12-seam source set, mirroring the Pi-owned-subagents blockquote in the sibling [Surface extensions without a dedicated topic-page seam](#surface-extensions-no-dedicated-seam) sub-bucket.

- **Automatic context escalation:** when binding fails without context, automatically retry with `bind_context: session` attached — trades a second binder call for a smoother success rate on context-sensitive looms that forgot to opt in.
  *Carrier:* [Slash-Command Argument Binding — loom 1.0 seam: automatic context escalation](../binder/binder-model-and-context.md#v1-seam-automatic-context-escalation) pins the binder-invocation re-entrancy invariant defensively; the committed seam set for the extension itself is not settled until the *Decision required* line below resolves.
  *Depends on:* [Binder refinement loop](#binder-refinement-loop), only if escalation surfaces user-visible turns; otherwise independent.
  *Decision required before this item can be scoped:* whether automatic escalation surfaces a user-visible turn (composing with the [Binder refinement loop](#binder-refinement-loop)) or stays operator-only.
