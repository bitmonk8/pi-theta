# Model changes and non goals

## Model-level changes (no theta 1.0 seam expected)

- **First-class theta values** (`Theta<T>` type, passing thetas as arguments, higher-order composition) — theta 1.0 only supports literal-path `invoke` and frontmatter-registered callables.
- **Theta-level concurrency primitives** (e.g. `parallel { ... }` blocks or a parallel-`for` form) building on Pi tools' Promise-returning shape — theta 1.0 keeps every tool call sequential and synchronous-looking.
- **Streaming partial tool results** from Pi's `onUpdate` callback into theta code (e.g. an iterator-style consumption form) — theta 1.0 returns only the final result.
- **Structured tool output schemas**, when Pi (or upstream providers) introduce a strict output-schema contract for tools — theta 1.0 returns `string` from every Pi tool call.
- **Richer untyped-query return shape** (e.g. `Result<string | AssistantMessage, QueryError>` exposing tool-use traces, multiple content blocks, citations) — theta 1.0 returns plain `Result<string, QueryError>`. A future widening would change the value model even though existing call sites would keep working under the union form.
- **Per-parameter `mut` on function parameters** (Rust-style `fn f(mut x: T)`) — theta 1.0 keeps all function parameters immutable.
- **Value-carrying `break expr`** inside `for` / `while` loops — theta 1.0's `break` and `continue` carry no value.
- **`match` guards** (`Ok(x) if x.value > 3 => ...`) and **rest patterns** (`[first, ...rest]`, `{ kind, ...other }`) — neither is in theta 1.0.
- **Non-decimal number literals** (hex `0x...`, octal `0o...`, binary `0b...`) **and underscore digit separators** (`1_000_000`) — theta 1.0 accepts decimal literals only with no separators.
- **Integer-division operator** — theta 1.0's `/` always produces `number`; there is no dedicated truncating-division operator.
- **Additional string and array stdlib methods** beyond the theta 1.0 set — the theta 1.0 method surface is deliberately small; extending it is non-breaking by design (anything not on the list is a parse-time "unknown method" error, leaving the namespace open).

## theta 1.0 non-goals

The theta 1.0 non-goals are cross-cutting theta 1.0 scope decisions where theta 1.0 ships without a forward-compatibility seam: a future revision adding the disposition is not anticipated by theta 1.0 and will require a migration. Two of the items below — no parallel-`invoke` surface and no parallel fan-out across prompt-mode `.theta` callees — were subsequently delivered in theta 1.1 by [RFC 0003 — `par for`](../../rfcs/0003-parallel-fanout.md) as a backward-compatible minor requiring no migration; they are retained below as historical theta-1.0 scope notes, not as standing requires-migration non-goals. The non-goals here are distinct from the model-level changes in category 3 of the intro list above — category 3 covers post-theta 1.0 changes to theta 1.0's runtime value, evaluation, or tool-result contracts, whereas the non-goals below are cross-cutting "we chose not to deliver this" scope dispositions that do not change those value-model contracts. (The seam-vs-non-goal distinction is stated in category 4 of the intro list at the top of this page; see also [`spec.md` — theta 1.0 non-goals](../language-and-architecture.md#theta-1-0-non-goals-aggregator) and [Governance — GOV-30](../governance.md).)

- **No per-theta sandbox or capability model.** Pi exposes no per-extension privilege scoping in theta 1.0; theta code runs at full host-process privilege. The runtime's no-extra-mediation rule is owned by [Pi Integration Contract — No additional access channels](../pi-integration-contract/trust-boundary.md#no-extra-mediation); the per-theta `tools:` allowlist is a configuration knob over the *model's* reachable callable set, not a host-process sandbox. A future per-theta capability model is not anticipated by theta 1.0 and will require a migration. Detection of a future Pi per-extension privilege-scoping facet is routed to editorial review under [Pi version bump procedure — Per-extension privilege-scoping facet](../pi-integration-contract/version-bump-step2.md#bump-checklist-privilege-scoping-facet); the build-time SDK surface-inventory assertion is a presence check on the seven named SDK capabilities and does not enumerate new top-level members.
  *Recorded at:* [Pi Integration Contract — No additional access channels](../pi-integration-contract/trust-boundary.md#no-extra-mediation); [`spec.md` — Orientation — Non-functional requirements — Trust boundary](../overview-and-orientation.md#non-functional-requirements); [Pi version bump procedure — Per-extension privilege-scoping facet](../pi-integration-contract/version-bump-step2.md#bump-checklist-privilege-scoping-facet).
- **No formal source-language migration mechanism for major-version transitions.** theta 1.0 declares the release-process goal that a `.theta` or `.thetalib` file loading cleanly under V1.0 loads and behaves identically under every theta 1.x release (per [Governance — GOV-15](../governance/source-language-stability.md#gov-15); intent statement, not a normative obligation, with no automated equivalence gate planned for theta 1.0); migration across major versions is out of theta 1.0 scope and will require a separate mechanism when theta 2.0 ships.
  *Recorded at:* [Governance — GOV-15](../governance/source-language-stability.md#gov-15); [`spec.md` — Orientation — Non-functional requirements — Source-language stability](../overview-and-orientation.md#source-language-stability).
- **No non-Node JavaScript host support.** theta 1.0 binds to Node exclusively (via `process.versions.node` in the Step 0 (a) probe and via Pi's `engines.node` floor). Bun, Deno, browser-embed hosts, and other JavaScript engines are not anticipated by theta 1.0 and would require a probe re-design before being added.
  *Recorded at:* [`spec.md` — Orientation — Prerequisites — Host runtime](../overview-and-orientation.md#orientation); [Runtime Value Model — JavaScript engine assumptions](../runtime-value-model.md#javascript-engine-assumptions).
- **No concurrent user sessions in the same host process.** The theta 1.0 runtime presupposes a single active user session per extension instance, per the Session model paragraph cited below; this is a presupposition rather than a claim that the runtime has audited Pi's full extension API surface and confirmed the absence of any concurrent-sessions facet, and concurrent-session detection is not part of the Step 0 capability probe. A future Pi minor that surfaces concurrent user sessions by mutating an existing pinned shape (e.g. an addition to a previously-closed `kind` enum or a new field on `ExtensionContext`) would surface at the `npm run typecheck` step (step 1) of [Pi version bump procedure](../pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) — but only insofar as theta code performs exhaustive matching against the previously-closed shape that TypeScript can flag — before the spec contract drifts; if it instead surfaces as a wholly new method whose name was not previously pinned (e.g. a hypothetical `pi.openParallelSession`), neither the typecheck step nor the build-time SDK surface-inventory assertion (which is a presence check confirming each of the seven named SDK capabilities enumerated in [SDK capability inventory](../pi-integration-contract/inventory-audit-intro.md#sdk-capability-inventory) is present on the imported namespace, not an enumeration of new top-level members or a walk of closed-set enum membership) detects it, and the disposition relies instead on the editorial review that [Pi version bump procedure](../pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) presupposes — i.e. the surface descriptions on [Pi Integration Contract](../pi-integration-contract.md) being kept current as the contributor walks the candidate package against them — to catch the API addition. If a Pi point release nonetheless surfaces an unanticipated concurrent-session signal at runtime before the next SDK-pin bump trips the build-time assertion, the theta 1.0 runtime defines no *new* rule for the signal *as such*; its pre-existing closed-set validators still fire on whatever previously-closed surface site the signal happens to reach (notably the `session_shutdown` unknown-reason rule at [Pi Integration Contract — Extension entry point](../pi-integration-contract.md), which validates `event.reason` against the closed `{"quit", "reload", "new", "resume", "fork"}` set and emits `theta/host/session-shutdown-reason-unknown` before proceeding with the fixed teardown sequence), and any value appearing *inside* one of those previously-closed surface sites that is not in the closed set is itself an SDK-pin violation that the `npm run typecheck` step cited above is designed to catch before ship — again only insofar as theta code performs exhaustive matching against the closed set that TypeScript can flag. A wholly new top-level member *outside* the pinned surface sites — including a wholly new event channel — is the case already carved out earlier in this bullet: neither the typecheck step nor the surface-inventory assertion (the latter being a presence check on the seven named capabilities, not an enumeration of new top-level members) detects it, and detection relies on the editorial review the bump procedure presupposes, cited earlier in this bullet. The runtime-side binding to Pi's one-active-user-session model is the lifetime invariant that theta 1.0 thetas never trigger replacement (anchored on a single factory-captured `pi` reference per theta invocation), stated next to the related seam at [Pi Integration Contract — theta 1.0 seam: mid-theta user-session replacement](../pi-integration-contract/conversation-drive.md#theta-1-0-seam-mid-theta-user-session-replacement); that seam itself is for an adjacent concept — a future feature that calls `ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` from inside a running theta and re-acquires `pi` via `withSession` before the next `sendUserMessage` — and is therefore a category-2 surface extension on its own terms, but the lifetime guarantee that rides on it (theta 1.0 thetas never trigger replacement) is what binds theta 1.0 to Pi's one-active-user-session model and reserves no forward-compatibility seam of its own. A future Pi minor that surfaces concurrent user sessions in the same host process is not anticipated by theta 1.0, and the theta 1.0 runtime exposes no workaround. Disposition for an unanticipated second session that nonetheless reaches the extension within the [theta 1.0 Pi-SDK pin range](../pi-integration-contract/host-prerequisites.md#pi-sdk-pin) or a subsequent pin: every site theta 1.0 scopes to a single active user session — the single factory-captured `pi` reference per theta invocation pinned at [Pi Integration Contract — theta 1.0 seam: mid-theta user-session replacement](../pi-integration-contract/conversation-drive.md#theta-1-0-seam-mid-theta-user-session-replacement), the extension-scoped [`ActiveInvocationRegistry`](../pi-integration-contract/active-invocation-registry.md#active-invocation-registry), and the per-session slash-dispatch serialisation underpinning prompt-mode sequencing per [Pi Integration Contract — Session-binding contract](../pi-integration-contract/host-prerequisites.md#session-binding-contract) — remains so scoped under theta 1.0, and the second session is itself out of theta 1.0 scope; the theta 1.0 runtime defines no new disposition for it (no refuse-to-load, no bind-to-first-session, no per-session registry keying, no host-incompatibility diagnostic).
  *Recorded at:* [Pi Integration Contract — Session-binding contract](../pi-integration-contract/host-prerequisites.md#session-binding-contract); [Pi Integration Contract — theta 1.0 seam: mid-theta user-session replacement](../pi-integration-contract/conversation-drive.md#theta-1-0-seam-mid-theta-user-session-replacement); [Pi Integration Contract — Extension entry point](../pi-integration-contract.md); [`spec.md` — Session model — SM-1 single-active-session binding](../session-model-and-appendix.md#sm-1-single-active-session-binding).
- <a id="pi-stdio-capture-facet"></a>**No reliance on a Pi extension-host stdio-capture facet.** The teardown-handler diagnostic emissions enumerated under [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation) write to `console.error` as their last-resort sink. This presupposes that Pi's extension host does not capture, redirect, or suppress `console.*` writes during teardown in a way that hides them from the operator; the theta 1.0 Pi-SDK pin's `dist/core/extensions/types.d.ts` and the `ExtensionAPI` / `ExtensionContext` surfaces enumerated under **Host prerequisites** on [Pi Integration Contract](../pi-integration-contract.md) expose no stdio-capture facet the runtime can rely on, and Pi's documented extension-host behaviour does not pin a stdio-capture policy. theta 1.0 therefore treats teardown-time `console.error` visibility as best-effort. A future Pi minor adding an explicit stdio-capture facet — a typed surface the runtime could query or subscribe to in order to route teardown diagnostics to an operator-visible sink with delivery guarantees — is not anticipated by theta 1.0; adopting it would require revisiting the last-resort sink choice in the **Diagnostic-emission isolation** rule. Detection is by editorial review under [Pi version bump procedure](../pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) — the build-time SDK surface-inventory assertion is a presence check confirming each of the seven pinned SDK capabilities is still present on the imported namespace and does not enumerate new top-level members, so the addition of an unrelated stdio-capture facet does not surface at that gate.
  *Recorded at:* [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation).
- **No parallel-`invoke` surface.** theta 1.0 exposes no API by which a parent body can spawn sibling `invoke(...)` children concurrently: every `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns, and no language form opens a parallel-children scope, per [Invocation — Cross-mode semantics](../invocation.md). Adding a parallel-invoke form would require a new evaluation primitive and was not anticipated by theta 1.0. **Superseded in theta 1.1 by [RFC 0003 — `par for`](../../rfcs/0003-parallel-fanout.md):** the statement above records theta-1.0 truth only — theta 1.0 had no parallel-children scope. theta 1.1 adds one: a `par for` body opens a parallel-children scope whose iterations may each run `invoke(...)`, including to prompt-mode callees (CTRL-4, [Control flow — `par for`](../control-flow.md#ctrl-4)), delivered as a backward-compatible minor with no migration. This is therefore a historical scope note, not a requires-migration non-goal.
  *Recorded at:* [`spec.md` — Session model — SM-7c prompt-mode sequential execution](../session-model-and-appendix.md#sm-7c-prompt-mode-sequential-execution); [Invocation — Cross-mode semantics](../invocation.md); [RFC 0003 — `par for`](../../rfcs/0003-parallel-fanout.md).
- **No parallel fan-out across prompt-mode `.theta` callees.** "Fan-out" here means the runtime dispatching, within a single tool-call round, multiple model-emitted `tool_use` blocks concurrently rather than sequentially (the parallel-dispatch behaviour pinned at [Parameters and Frontmatter — `tool_loop`](../frontmatter.md), "the runtime executing them all (in parallel where the provider supports parallel tool calls)"). Of the two callable kinds actually reachable from any theta's `tools:` table — Pi-host tools and subagent-mode `.theta` callees — theta 1.0 permits fan-out for both; the prompt-mode `.theta` axis is excluded by a load-time mechanism: prompt-mode `.theta` paths listed in another theta's `tools:` are rejected per [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools), so the model never sees a prompt-mode `.theta` callable in its tool table to fan out across in the first place (the runtime-level absence of parallel prompt-mode tool calls is a downstream consequence of that load-time rejection, not an independent runtime check). Pi-host tools and subagent-mode `.theta` callees remain present in the tool table and may be fanned out concurrently within a prompt-mode body — Pi-host-tool fan-out rides the same provider-driven parallel-dispatch behaviour cited above at [Parameters and Frontmatter — `tool_loop`](../frontmatter.md) ("the runtime executing them all (in parallel where the provider supports parallel tool calls)"), and the carve-out for subagent-mode `.theta` callees is anchored at [Tool Calls — Concurrency](../tool-calls.md#concurrency) (the normative owner of parallel tool-call concurrency, which permits parallel fan-out for both Pi tools and `.theta` callables and pins the per-sibling spawn shape — own `AgentSession`, captured callable set, derived `AbortSignal`, private transcript — that makes subagent-mode `.theta` callees the surviving `.theta`-callee concurrency surface, with [`spec.md` — Session model — SM-7c prompt-mode sequential execution](../session-model-and-appendix.md#sm-7c-prompt-mode-sequential-execution) cross-linked as informative orientation that pins prompt-mode body serialisation against the three sources of in-session overlap). Lifting the tool-table restriction itself would require revisiting prompt-mode body serialisation and was not anticipated by theta 1.0. **Superseded in theta 1.1 by [RFC 0003 — `par for`](../../rfcs/0003-parallel-fanout.md):** the load-time restriction above still holds — a prompt-mode `.theta` callable still cannot appear in another theta's `tools:` table — but the broader "no parallel fan-out across prompt-mode `.theta` callees" scope claim is superseded. theta 1.1 delivers that fan-out by a different route: an `invoke(...)` to a prompt-mode callee inside a `par for` body (CTRL-4 admits `invoke` in the body, [Control flow — `par for`](../control-flow.md#ctrl-4)), run concurrently across iterations, as a backward-compatible minor with no migration. This is therefore a historical scope note, not a requires-migration non-goal.
  *Recorded at:* [Tool Calls — Concurrency](../tool-calls.md#concurrency); [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools); [`spec.md` — Session model — SM-7c prompt-mode sequential execution](../session-model-and-appendix.md#sm-7c-prompt-mode-sequential-execution); [RFC 0003 — `par for`](../../rfcs/0003-parallel-fanout.md).
- **No admission cap on in-flight theta invocations.** theta 1.0 imposes no cap on in-flight theta invocations (across both modes) within an extension instance and interposes no scheduler above Pi's event loop; per-invocation budgets (binder retry budget, `tool_loop.max_rounds`, `invoke`-chain depth) are scoped to a single invocation and not pooled across siblings, per [Pi Integration Contract — No invocation cap](../pi-integration-contract/subagent.md#no-invocation-cap). The rule constrains the runtime only — Pi's own session-level serialisation (e.g. prompt-mode slash-command turns within one user session, owned by [Pi Integration Contract — Tool-registration lifetime and visibility (Pi behavioural preconditions on snapshot/restore)](../pi-integration-contract/tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions), where it is recorded as a Pi behavioural precondition the type signature does not encode and that MUST be re-validated as part of the [Pi version bump procedure](../pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) if it weakens) is upstream and remains in force, and the rule does not promise resource unboundedness: catchable allocation failures (the `RangeError` family — `Invalid string length`, `Invalid array length`, `Maximum call stack size exceeded`) still route through `theta/runtime/internal-error` per [Errors and Results](../errors-and-results.md), while uncatchable host fatals (V8 heap-limit OOM via the `OOMErrorCallback` / `abort()` path, and any other engine fatal that bypasses the JavaScript exception machinery) terminate the host process before any wrap can observe them and emit no `theta/runtime/internal-error` diagnostic, per the ownership-boundary statement at the same site (which assigns runtime-value heap to the host JavaScript runtime, OS-level file descriptors / sockets / child-process slots to the host OS, and provider-side rate-limit / quota to the LLM provider, and pins the absence of any theta-level per-class threshold or cross-sibling aggregation, throttling, or storm-detection surface over those classes). A future admission-control surface — and a future cross-sibling aggregation, throttling, or storm-detection surface over any of those resource classes — is not anticipated by theta 1.0.
  *Recorded at:* [`spec.md` — Session model — SM-7d theta 1.0 no-cap / no-scheduler disposition](../session-model-and-appendix.md#sm-7d-no-cap-no-scheduler); [Pi Integration Contract — No invocation cap](../pi-integration-contract/subagent.md#no-invocation-cap).
