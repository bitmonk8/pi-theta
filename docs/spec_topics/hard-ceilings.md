# Hard Runtime Ceilings

This page is the normative owner of the cross-ceiling content that no single per-ceiling topic page is responsible for: the [ceiling-set invariants](#ceiling-set-invariants) (audience-coverage invariant and ceiling-#1 panic-uniqueness invariant), the binder per-class retry budget's sub-obligations (HC3-a … HC3-e), the JSON-document-depth ceiling's per-boundary destination/surface table, the fixed evaluation order between ceilings (CIO-1 … CIO-6), the `masked` field's closed identifier set and wire-location split (the per-site reachable mask domain, V1 reachable predicate, pure-read MUST, verbatim-copy obligation on cascade-twin re-emission, and dedup-key non-inclusion rule are owned by [Pi Integration Contract — PIC-1](./pi-integration-contract.md#pic-1)), the four enumerated non-existence claims (NOCEIL-1 … NOCEIL-4), the worked consequences of the interaction order, and the four-axis [Audit methodology](#audit-methodology) that closes the NOCEIL-1 … NOCEIL-4 set. The four-item ceiling list itself, with its per-ceiling routing-class one-liner, is owned by [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings) as an aggregator paragraph per [Governance — GOV-12](./governance.md); this page carries the load-bearing definitions that aggregator forward-links into.

*Identifier classes used on this page.* The `HC3-a` … `HC3-e` and `NOCEIL-1` … `NOCEIL-4` markers below are *stable inline labels* in the second governed identifier class introduced at [Governance — GOV-16](./governance.md#gov-16); the `HC3` and `NOCEIL` prefixes are registered on this page in GOV-16's per-page inline-label prefix table. The `CIO-1` … `CIO-6` markers are full REQ-IDs under the `CIO` prefix — the second of two prefixes co-owned by this page (the other being `CEIL`) per the [REQ-ID prefix table](./governance.md#req-id-prefix-table) and GOV-4's many-to-one prefix → page binding.

Each per-ceiling rule's *first* enforcement point still lives on the topic page that owns the breach surface — `loom/runtime/invoke-depth-exceeded` on [Invocation](./invocation.md), `Err(QueryError { kind: "tool_loop_exhausted", … })` on [Errors and Results](./errors-and-results.md) and [Query](./query.md), the binder retry classifier on [Slash-Command Argument Binding](./binder.md), the depth-walk on [Schema Subset](./schema-subset.md). This page pins the *cross-ceiling* obligations that no single per-ceiling page can own.

## Per-class retry budget (ceiling #3)

<a id="hc3"></a>

The binder per-class retry budget is ceiling #3 in the four-item list at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings). Routing class: **load-time system note**. The loom does not start; the operator-facing note is rendered from the failure-mode template matching the *most recent* failure's class — see [Slash-Command Argument Binding — Failure-mode templates](./binder.md#failure-mode-templates-normative). Not an evaluation outcome — nothing reaches loom code, no `Result` value is observable. Governed by the load-time-failure carve-out under [Errors and Results — Terminal outcomes](./errors-and-results.md#terminal-outcomes). The classification rule that maps an observed binder-call outcome onto a retry class is the provider-error classifier in [Slash-Command Argument Binding — Failure-class taxonomy](./binder.md#failure-class-taxonomy). The five sub-obligations — each a stable inline label citable from tests and downstream revisions — are:

- **HC3-a.** At most one transport-class retry per slash invocation.
- **HC3-b.** At most one malformed-envelope-class retry per slash invocation.
- **HC3-c.** AJV-on-`args` failures are not retried (no per-class retry budget).
- **HC3-d.** Worst-case sum: 3 binder LLM calls per slash invocation (1 initial attempt + 1 transport-class retry + 1 malformed-envelope-class retry).
- **HC3-e.** When the chain ends with both budgets exhausted, the surfaced system note is the row matching the *most recent* failure observed.

## Per-boundary destination/surface table (ceiling #4)

<a id="ceiling-4-table"></a>

Ceiling #4 is the JSON-document depth-5 ceiling against the five enforcement points listed in the per-boundary table below — typed-query response, model-driven tool-call args, code-driven tool-call args, `params` validation, and `invoke<T>` return value (ceiling #4's entry at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings)). Routing class: **boundary-dependent** per the table below. Every breach surfaces with `schema_keyword: "maxDepth"` and the canonical message `"JSON document depth exceeds 5"` per [Schema Subset — Depth Enforcement](./schema-subset.md); the carrier shape and destination depend on which of the five enforcement points fires:

| Check point | Destination | Surface |
|---|---|---|
| Typed-query response | loom code | `Err(QueryError { kind: "validation", cause: "schema_validation", validation_errors: [{ schema_keyword: "maxDepth", … }], … })` per [Errors and Results — `ValidationError`](./errors-and-results.md#queryerror-variants) |
| Tool-call args, model-driven (`@`...`` loop) | the model (loop continues; round counts against `tool_loop.max_rounds`) | tool-error result fed back as the next user turn per [Query — Tool calls during a query](./query.md); no `QueryError` surfaces unless the loop later hits ceiling #2 |
| Tool-call args, code-driven (`<name>(args)`) | loom code | `Err(CodeToolError { cause: "validation", validation_errors: [{ schema_keyword: "maxDepth", … }], … })` per [Tool Calls — Failures](./tool-calls.md) and [Errors and Results — `CodeToolError`](./errors-and-results.md#queryerror-variants) |
| `params` validation | depends on call site | `invoke(...)`: `Err(InvokeInfraError { cause: "validation", … })` per [Invocation — Failures](./invocation.md). Slash-load: routes through ceiling #3's no-retry classification (the binder's AJV-on-`args` arm at the [post-default-merge AJV validation](./binder.md#post-default-merge-ajv-validation) hook) and renders through the AJV-on-`args` row of [Slash-Command Argument Binding — Failure-mode templates](./binder.md#failure-mode-templates-normative); the row's `<ajv-summary>` placeholder carries the depth-walk's canonical issue (`<JSON-Pointer> JSON document depth exceeds 5`) per that table's depth-walk clause. The binder LLM call that produced the depth-violating envelope still counts toward HC3-d's worst-case 3-call budget exactly as a regular AJV-on-`args` failure does — no separate depth-violation accounting rule applies and HC3-c's no-retry disposition settles the chain. Not an evaluation outcome, no `Result` value observable |
| `invoke<T>` return value | calling loom | `Err(InvokeInfraError { cause: "validation", … })` per [Invocation — Failures](./invocation.md) |

<a id="ceiling-4-table-reconciliation"></a>The model-driven row is the only one whose surface is silent at the loom-code level — the depth violation reaches the model as a tool-result and reaches the operator only via the `loom-system-note` channel if and when the loop later exhausts under ceiling #2 (the `validation` runtime event defined in [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md) is keyed off the `Err` variant and therefore does not fire on this row). The depth violation is materialised in the model's tool-result and the round still counts against `tool_loop.max_rounds`. The slash-load arm of `params` validation is the only row that crosses ceilings — the depth-walk fires at the `params` AJV boundary per CIO-3 below, but the failure routes through ceiling #3's load-time system-note classification rather than ceiling #4's recoverable-`Err` path; this cross-ceiling handoff is the only *load-time* case where ceiling #4's enforcement point produces a non-`Err` surface (the runtime non-`Err` surface is the model-driven tool-call args row in the per-boundary table, whose surface is a tool-result fed to the model rather than a `Result::Err` returned to loom code).

## Interaction between ceilings

<a id="ceiling-interaction-order"></a>

Each ceiling is checked at a distinct point in single-threaded interpreter execution. The first ceiling reached along an event's control-flow path that finds its precondition satisfied fires and ends the event; ceilings situated at later points on the same path are not evaluated for that event. The fixed evaluation order — each item below is a REQ-ID citable from tests and downstream revisions — is:

- **CIO-1.** Ceiling #3 (binder per-class retry budget) is evaluated at slash-load time, before any *runtime-class* ceiling fires; the slash-load `params` arm of ceiling #4 also occurs at slash-load time and is routed by ceiling #3's failure-mode templates per the per-boundary table at [Per-boundary destination/surface table (ceiling #4)](#ceiling-4-table).
- **CIO-2.** Ceiling #1 (`invoke`-chain depth) is evaluated at `invoke` entry, before the callee body runs.
- **CIO-3.** Ceiling #4 (JSON-document depth) is the first sub-check at every AJV validation boundary (typed-query response, model-driven `tool_use` args, code-driven `<name>(args)` args, `params` validation, `invoke<T>` return value — five sites, in the same order as the per-boundary table at [Per-boundary destination/surface table (ceiling #4)](#ceiling-4-table)); the depth-walk runs *before* AJV at the same site per [Schema Subset — Depth Enforcement](./schema-subset.md), so #4 always precedes the boundary's other validation. The Anthropic-API term `tool_use` refers only to model-emitted tool-call blocks; the code-driven `<name>(args)` site is a peer AJV boundary in loom code that is NOT a `tool_use` block and is enumerated separately above and in the per-boundary table.
- **CIO-4.** Ceiling #2 (`tool_loop.max_rounds`) is evaluated at the tool-call-round boundary — *after* the round's tool calls have completed and the slot count has been incremented for the just-completed free-phase round (the *tool-call round slot accounting*: the bookkeeping ceiling #2 consults to decide whether the next free-phase model turn would exceed `max_rounds`; see [Glossary — tool-call round slot accounting](./glossary.md) and [Query — Tool-call loop bound](./query.md#tool-call-loop-bound)), and *before* the next free-phase model turn is requested. The check has two outcomes, keyed on the post-increment slot count: *free-phase continuation* (`slot_count < max_rounds`: the next free-phase model turn is requested) and the *`max_rounds`-final branch* (`slot_count == max_rounds`: for untyped queries the runtime surfaces `Err(QueryError { kind: "tool_loop_exhausted", … })` before any further turn is issued; for typed queries the runtime dispatches the forced respond turn as the exempt-routed terminator that follows this gating check — the forced respond turn is not itself counted against `max_rounds` and CIO-4 is not re-evaluated against it). The `max_rounds: 0` typed-query boundary is the same `max_rounds`-final branch fired at typed-query start (`slot_count == max_rounds` holds at initialisation, 0 == 0) with no free-phase round preceding it; see [Query — Typed queries are tool-loop-shaped](./query.md#typed-queries-are-tool-loop-shaped).
- **CIO-5.** Ceiling #3 never interleaves with #1, #2, or #4: the binder runs only at slash-invocation load time, and `invoke(...)` calls do not invoke the binder per [Slash-Command Argument Binding — Binder bypass](./binder.md#bypass-cases).
- **CIO-6.** At most one ceiling surfaces per event. The surfaced diagnostic / runtime event carries an optional `masked` field enumerating any other ceiling whose precondition was *also* satisfied at the same check site. The closed identifier set and wire-location split are owned by [`masked` field](#masked-field) below; the per-site reachable mask domain, V1 reachable predicate, and probe-purity (pure-read) MUST are owned by [Pi Integration Contract — PIC-1](./pi-integration-contract.md#pic-1).

## Ceiling-set invariants

<a id="ceiling-set-invariants"></a>

Two set-level claims hold over the loom 1.0.0 four-ceiling set as a whole. They are derivations from the per-ceiling material — the per-ceiling routing-class one-liners on [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings), the [per-boundary destination/surface table for ceiling #4](#ceiling-4-table), and the CIO-1 … CIO-6 evaluation order in [Interaction between ceilings](#ceiling-interaction-order); restating them here keeps the panic-path claim available as the panic-uniqueness claim that the *runtime panic* routing-class label is held by exactly one ceiling — ceiling #1 — in the loom 1.0.0 set.

- **Audience-coverage invariant.** Each ceiling has an observable failure surface — addressed to at least one of *loom code*, *the model*, or *the operator* — and no ceiling is unobservable to all three. (Ceiling #4's model-driven tool-call args row is silent at the loom-code level; it is observable to the model directly and, if the loop later exhausts, to the operator via ceiling #2 — see the model-driven row of the [per-boundary destination/surface table](#ceiling-4-table) and the reconciliation paragraph that follows it.) The membership predicate — which audience(s) a given ceiling is observable to — is not redefined here; it is read off the per-ceiling owner pages (the per-ceiling routing-class one-liners on [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings), the per-ceiling owner pages those one-liners forward-link into, and for ceiling #4 the per-arm classification owned by the [per-boundary destination/surface table](#ceiling-4-table)). A future ceiling-set edit governed by [GOV-15 — Ceiling-set carve-out](./governance.md#ceiling-set-carve-out) MUST verify, by inspection of the per-ceiling owner pages just named, that the post-edit set preserves this invariant for every ceiling in the post-edit set. For inspection purposes, *observable to loom code* means the owner page names a panic surface or a typed value (`Result`, `Err` variant, or per-boundary-table-classified surface) that loom code can `match` or propagate; *observable to the model* means the owner page names a tool-call response, follow-up prompt, or in-loop transcript surface that reaches the model without further loom-code mediation; *observable to the operator* means the owner page names a `loom-system-note`, load-time system note, terminal-print, or operator-facing log surface. The per-boundary table's `calling loom` Destination counts as *loom code* for this invariant; the table's `depends on call site` Destination is read by following the table's per-call-site disambiguation and classifying each disambiguated arm under the three audience definitions just given.

- **Ceiling-#1 panic-uniqueness invariant.** Of the four ceilings, ceiling #1 (`invoke`-chain nesting depth 32) is the *only* one whose breach reaches the success / fail / cancelled trichotomy's *fail* arm via the panic path. Ceiling #2 (`tool_loop.max_rounds`) reaches *fail* only via unhandled [query-terminating](./glossary.md#query-terminating) `Err`; ceiling #3 is *load-time* and off the trichotomy per [Errors and Results — Terminal outcomes](./errors-and-results.md#terminal-outcomes); ceiling #4's query-terminating runtime arms (per the [per-boundary destination/surface table](#ceiling-4-table)) likewise reach *fail* only via unhandled [query-terminating](./glossary.md#query-terminating) `Err` — none panic. (Per CIO-3 / *Five-site list co-edit obligation*, `params` validation is one of CIO-3's five sites and contributes two arms here — the query-terminating `invoke(...)` arm and the slash-load arm that cross-routes through ceiling #3 — so the six ceiling-#4 arms touched by this invariant map onto CIO-3's five sites.) The remaining arms of ceiling #4 — the [in-loop](./glossary.md#in-loop) model-driven tool-call args row and the slash-load `params` arm that cross-routes through ceiling #3 — are not themselves trichotomy events: the in-loop model-driven row's downstream consequences, if the loop later exhausts, surface via ceiling #2's `Err` arm, and the slash-load `params` arm cross-routes through ceiling #3 and so resolves at load-time off the trichotomy. A future ceiling-set edit governed by [GOV-15 — Ceiling-set carve-out](./governance.md#ceiling-set-carve-out) MUST preserve or update this invariant in lock-step (whether by introducing a second panic-class ceiling, removing or relaxing ceiling #1, renumbering, or changing the four-count; routing-class demotions are covered by *Relax* per [GOV-15 — Ceiling-set carve-out](./governance.md#ceiling-set-carve-out)).

## `masked` field

<a id="masked-field"></a>

Each `masked` entry is one of the string identifiers `"ceiling#1"`, `"ceiling#2"`, `"ceiling#3"`, `"ceiling#4"`; the field is **omitted** when no co-fire occurred (canonical absence — implementations MUST NOT emit `masked: []`).

The field's location on the wire is `details.masked` for diagnostic-shape surfaces (the `loom/runtime/invoke-depth-exceeded` panic per [Diagnostics](./diagnostics.md)) and `details.event.masked` for runtime-event surfaces (the `validation` and `tool_loop_exhausted` cases per [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md)).

The per-site reachable mask domain (which sibling ceilings can populate `masked` at each surfacing site), the V1 reachable predicate (the boolean read that determines whether `["ceiling#2"]` fires on the typed-query response branch), the pure-read MUST on the masking probe, the verbatim-copy obligation on cascade-twin re-emission, and the dedup-key non-inclusion rule are owned by [Pi Integration Contract — PIC-1 *Hard-ceiling co-fire (`masked`)*](./pi-integration-contract.md#pic-1). The closed identifier set above and the wire-location split are this page's contribution; everything that constrains *when* a non-empty `masked` is permitted to fire — in particular the V1 fact that `["ceiling#2"]` is the only non-empty `masked` reachable in V1, and the predicate that gates it — lives at PIC-1.

## Worked consequences

Worked consequences of the interaction order — each illustrates one or more `CIO-N` rules above and is named so a conformance test can cite it directly:

- *Depth-6 tool-call args mid-loop* (CIO-3, CIO-4): a `tool_use` round whose tool-call args payload is depth-6 produces ceiling #4 at the AJV boundary per CIO-3 (the model-driven row of the per-boundary table above — a tool-error result fed back to the model as the next user turn, NOT a `QueryError` returned to loom code), and the round is then permitted to continue the loop — that is one round of `tool_loop`, not exhaustion (CIO-4's slot accounting has not yet fired for this round, and the model gets to react to the validation error).
- *Depth-6 forced respond at `max_rounds`* (CIO-3, CIO-4): on a typed query at the final round permitted by `max_rounds`, CIO-4's slot-increment for the last free-phase round has already occurred and the forced respond turn has been dispatched. When that turn produces a depth-6 payload, CIO-3's depth-walk fires at the typed-query response AJV boundary inside the synthesised respond tool's `execute` and surfaces `Err(QueryError { kind: "validation", cause: "schema_validation", validation_errors: [{ schema_keyword: "maxDepth", … }], … })` per the typed-query response row of the ceiling #4 per-boundary table above. The `tool_loop_exhausted` variant is not reachable on this path — the forced respond turn is precisely the typed-query terminating mechanism CIO-4's `max_rounds`-final branch routes to. Respond-repair (when configured) may follow; per [Query — Tool-call loop bound](./query.md), each respond-repair follow-up gets a fresh `tool_loop` budget, so retry is gated by `respond_repair.attempts`, not by `max_rounds`. (An untyped query at `max_rounds` has no forced respond turn and is governed by the plain-text exhaustion case, not this bullet.)
- *`invoke` panic mid-loop* (CIO-2, CIO-4): an `invoke(...)` issued from inside a tool-loop round counts against the parent chain's depth budget per CIO-2 and, if it panics with `loom/runtime/invoke-depth-exceeded`, the panic propagates per [Errors and Results — Runtime panics](./errors-and-results.md) and the parent's `tool_loop` counter is irrelevant (the loop never resumes — CIO-4's slot accounting is not reached for the aborted round).

## No additional V1 runtime ceiling applies

<a id="no-additional-ceilings"></a>

The four enumerated non-existence claims below — each a stable inline label citable from tests and downstream revisions — are:

- **NOCEIL-1.** No wall-clock timeout per query / tool-call / invoke. Per-call timeouts are deferred per [Cancellation](./cancellation.md) and [Future Considerations](./future-considerations.md), and the absence is enforced at parse time by rejecting any `timeout:` field at every site where a future per-call timeout could land.
- **NOCEIL-2.** No per-query response-token cap and no cumulative-token budget. The only token-domain failure surface in V1 is provider-detected `ContextOverflowError` per [Errors and Results — `ContextOverflowError`](./errors-and-results.md#queryerror-variants); a pre-flight token-count check is deferred per [Future Considerations](./future-considerations.md).
- **NOCEIL-3.** No runtime-value memory ceiling. String length, array length, and total heap are bounded only by the host process. *Catchable* host allocation failures — `RangeError: Invalid string length`, `RangeError: Invalid array length`, `RangeError: Maximum call stack size exceeded`, and any other JavaScript-exception-shaped allocation failure the host throws — are part of the runtime-defect surface and route through `loom/runtime/internal-error` per [Errors and Results — Runtime panics](./errors-and-results.md). *Uncatchable* host fatals — V8's `FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory` and any other engine fatal that bypasses the JavaScript exception machinery (the V8 `OOMErrorCallback` / `abort()` path) — terminate the host process; the runtime cannot observe them and emits no diagnostic. Operators relying on a finite heap ceiling MUST use a host-side mechanism (e.g. `--max-old-space-size`, container memory caps, `--heapsnapshot-near-heap-limit` for triage); V1 makes no claim that the loom runtime can intercept the fatal path.
- **NOCEIL-4.** No loom-level host-language stack-depth ceiling distinct from the 32-level `invoke`-chain bound above. The 32-level `invoke`-chain bound is the only loom-level frame-depth ceiling. The host language's native call-stack bound (e.g. V8's recursion limit) is not separately ceilinged by the runtime; an exhaustion of that bound surfaces through the catchable-`RangeError` arm of NOCEIL-3 (`RangeError: Maximum call stack size exceeded`).

<a id="audit-methodology"></a>

*Audit methodology.* The closure of the `NOCEIL-1` … `NOCEIL-4` set above rests on a four-axis enumeration of where a runtime resource bound could fire.

<a id="four-axis-closure-list"></a>

***Four-axis closure list.*** The four axes are:

- *(i)* **time** — *(ii)* `NOCEIL-1` — *(iii)* wall-clock latency on a query, tool call, or invoke is uncapped in V1; the absence is enforced at parse time by `NOCEIL-1`'s `timeout:`-field rejection.
- *(i)* **token economy** — *(ii)* `NOCEIL-2` — *(iii)* per-call response tokens and cumulative provider-billed tokens are unceilinged, with provider-detected `ContextOverflowError` the sole token-domain failure surface in V1 per `NOCEIL-2`.
- *(i)* **runtime-value memory** — *(ii)* `NOCEIL-3` — *(iii)* string length, array length, total runtime-value heap, and any host allocation failure that surfaces as a catchable JavaScript exception or uncatchable host fatal are bounded only by the host process per `NOCEIL-3`.
- *(i)* **frame depth** — *(ii)* `NOCEIL-4` (with the 32-level `invoke`-chain bound as the only loom-level ceiling on this axis) — *(iii)* the host-language native call-stack bound is not separately ceilinged by the runtime and falls through `NOCEIL-3`'s catchable-`RangeError` arm.

Four further candidate shapes route outside or through the four axes above without earning a separate ceiling. Provider-side rate-limit / quota exhaustion is *outside the runtime ceiling closure entirely* and surfaces on the provider turn (no `execute()` call, no `tool_call_id`, and no `cause: "execution"`) as `Err(QueryError { kind: "transport", http_status: 429, … })` — or other 4xx-not-overflow `http_status` per the binder's transport-class classifier at [Slash-Command Argument Binding — Failure-class taxonomy](./binder.md#failure-class-taxonomy) — per [Errors and Results — `TransportError`](./errors-and-results.md#queryerror-variants), and is governed by the provider contract, not the loom 1.0.0 ceiling set. Host-tool resource exhaustion (file-descriptor exhaustion such as `EMFILE` on `read`, child-process-handle exhaustion such as `ECHILD` on subprocess spawn, network-socket exhaustion, I/O bandwidth, and any other host-tool-side resource shape) is likewise *outside the runtime ceiling closure entirely* and surfaces as `Err(QueryError { kind: "code_tool", … })` from the tool's `execute()` outcome (cause `execution` for a host-side throw or `isError: true` return) per [Tool Calls — Failures](./tool-calls.md) and [Errors and Results — `CodeToolError`](./errors-and-results.md#queryerror-variants), and is governed by the host-tool contract, not the loom 1.0.0 ceiling set. Schedulable concurrency / event-loop saturation falls through `NOCEIL-1` (wall-clock) for any visible delay. AJV validator compilation cost falls through `NOCEIL-1` (wall-clock) at parse / load time and through `NOCEIL-3`'s catchable-`RangeError` arm at runtime.

***Routing obligation for future ceilings.*** Any future runtime resource the spec might consider ceilinging MUST surface either as a new entry in the four-item ceiling list at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings) or as a documented fall-through to one of the four `NOCEIL-1` … `NOCEIL-4` claims above (each `NOCEIL-N` claim names the documented fall-through for its axis); a runtime resource that does neither is a closure defect under this paragraph. A future spec-editor introducing, retiring, or relaxing a ceiling under [Governance — GOV-12](./governance.md) MUST re-run this four-axis check and update the four-axis closure list above in lock-step with the change: an *introduction* adds a one-line entry under one of three cases: (1) *default case* — the new ceiling cleanly fits one of the four existing axes, and the entry names that axis; (2) *split-axis case* — the new ceiling carves a sub-axis from within one of the four existing axes, and the entry names the parent axis being split, the new sub-axis, and the residual sub-axis, with rationale for treating the two sub-axes as separate axes; (3) *new-axis case* — the new ceiling occupies a fundamentally new resource axis not covered by the four (e.g. file-descriptor count, child-process count, network-connection count, on-disk artefact size, where the new resource is ceilinged at the runtime layer rather than passed through to the `code_tool` `Err` outside-the-closure surface above), and the entry names the axis and the rationale and, in the same commit, adds the new resource type to the four-axis closure list at the head of this paragraph so the closure is widened in lock-step; a *retirement* does NOT delete the corresponding per-axis entry while the resource the retired ceiling bounded still exists at runtime — deleting the entry would leave the resource unrouted in violation of the routing-obligation MUST above; instead, retirement rewrites the per-axis entry into one of four forms — one per post-retirement runtime trajectory enumerated at the *Retire* clause of [Governance — Operational definitions](./governance.md#operational-definitions): (i) **succeed** — the resource is no longer bounded at the loom layer; the per-axis entry is rewritten to point at the verbal claim "no loom-layer bound; bounded only by the host process or by upstream ceilings" and the routing destination is recorded as "no V1 ceiling on this axis"; (ii) **fall through to a documented `NOCEIL-N`** — the per-axis entry is rewritten to point at the existing `NOCEIL-N` claim's documented fall-through (demoting the routing destination from the retired ceiling itself to that `NOCEIL-N`); where no existing `NOCEIL-N` applies, this form coins a new `NOCEIL-N` entry in the `NOCEIL-1` … `NOCEIL-4` closure list above (extending the list per GOV-16's inline-label allocation discipline) and points the per-axis entry at it as the special case of (ii); (iii) **fall through to another ceiling** — the per-axis entry is rewritten to point at the absorbing ceiling's per-axis entry (the routing destination becomes the other ceiling, not a `NOCEIL-N`); or (iv) **non-terminating absent cancellation** — the per-axis entry is rewritten to a documented `NOCEIL-N` or to a verbal claim ("operator MUST rely on cancellation; no V1 termination guarantee"), at the spec-editor's choice; this case is recorded explicitly because it is an operator-visible behaviour change. The "annotates the entry in place as retired with a pointer to the retirement-release diff" annotation is preserved as the bookkeeping form of any of (i)–(iv), and the pointer takes the SHA-or-tag form pinned at the *Retire* clause of [Governance — Operational definitions](./governance.md#operational-definitions) (either the 7-character abbreviated commit SHA of the retirement commit or the release tag of the V1.x release that performed the retirement, nothing else); a *relaxation* rewrites the corresponding per-axis entry to reflect the new bound while leaving the routing destination unchanged. ***Release gate.*** A closure defect under this paragraph surfaces during the GOV-15 reviewer-inspection step at release time as a GOV-12 lock-step violation and blocks the release until the per-operation acceptance criterion below is satisfied: an *introduction* — the new ceiling fits one of the four axes above (or one of the four `NOCEIL-1` … `NOCEIL-4` claims' documented fall-throughs) with a documented routing, or a new axis is introduced with the rationale required by the *new-axis case* above; a *retirement* — the per-axis entry for the retired ceiling has been updated to one of the four forms (i)–(iv) enumerated in the *retirement* clause of *Routing obligation* above, and the four-axis closure list still covers the resource (either via a NOCEIL-N pointer, an absorbing ceiling, or the explicit no-loom-bound annotation); a *relaxation* — the per-axis entry has been rewritten to reflect the relaxed bound and the routing destination still exists. Deferral of the four-axis closure list update past the release that introduces, retires, or relaxes the ceiling is not permitted. The GOV-15 reviewer-inspection step under [GOV-15 — Ceiling-set carve-out](./governance.md#ceiling-set-carve-out) MUST confirm that, for each [Ceiling-set invariant](#ceiling-set-invariants), the per-invariant MUST on that bullet is discharged for the post-edit set (audience-coverage: the post-edit set preserves the invariant for every ceiling per that bullet; panic-uniqueness: the invariant is preserved or updated in lock-step per the cases enumerated on that bullet).
