# Code registry host

### `loom/host/*` — host-observed anomalies

The column legend, the closed *Phase* taxonomy, and the *E/W* severity carve-out that govern this table are defined in [Diagnostic shape — Code registry rules (normative)](./diagnostic-shape.md#column-legend).

Diagnostics in this namespace are emitted from loom's interactions with the Pi host. The namespace covers both anomalies the loom runtime observes coming *from* the Pi host that the spec's host-prerequisites contract does not enumerate, and loom-internal failures of host-related machinery — pinned-constant blocks that snapshot host SDK types, host-event handlers, and similar — whose immediate cause is loom-internal corruption but whose observation site is a host-driven code path. They route via `console.error` because the typical observation site is the `session_shutdown` teardown handler, where Pi's `ExtensionRuntime.invalidate(...)` may already have invalidated the persistent-diagnostic channel — the same delivery-sink carve-out as `loom/runtime/reload-teardown-timeout`. Every `console.error` call this namespace makes from the teardown handler is wrapped per [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation): an inner throw from `console.error` MUST be swallowed and MUST NOT unwind the handler.

| `loom/host/session-shutdown-reason-unknown` | W | runtime | The `session_shutdown` handler observed an `event.reason` outside the closed set `{"quit", "reload", "new", "resume", "fork"}`, or the read of `event.reason` itself threw. The handler routes the unknown reason through the full five-sub-step teardown sequence per PIC's **Unknown-reason rule**; this diagnostic surfaces the drift so an operator (or a Pi-version-bump regression test) can observe it. | `loom/host/session-shutdown-pinned-constant-unreadable` | W | runtime | The `session_shutdown` handler's entry-time read of the pinned-constant source-of-truth for `event.reason`'s closed set failed: the snapshot-entry lookup or its `literals` field read failed (per [PIC's **Unknown-reason rule**](../pi-integration-contract/unknown-reason-rule.md#unknown-reason-rule), which owns the lookup contract — including the snapshot-entry composite predicate, the defensive kind-mismatch handling, and the four `literals`-field shape failures: not an array, an array containing a non-string element, the empty array, or an array of strings containing the empty string `""`). The handler emits this diagnostic exactly once per `session_shutdown` event, *before* sub-step 1 runs (so it survives a later handler throw, symmetric with `session-shutdown-reason-unknown`), then routes the shutdown through the unknown-reason path for the teardown sequence only (set-membership check skipped; full five-sub-step teardown runs) so the constant-read failure never escapes the handler. This code and `loom/host/session-shutdown-reason-unknown` are **mutually exclusive** — at most one of the two fires per `session_shutdown` event: on snapshot-lookup failure this code fires *instead of* `session-shutdown-reason-unknown`, so a corrupted snapshot paired with a closed-set-member `event.reason` (e.g. `"quit"`) does not surface a misleading `session_shutdown event.reason outside closed set: quit` message (per PIC's **Unknown-reason rule**). `details: { failure: <discriminator string> }` carries one of the two literal forms `"missing-entry"` or `"literals-shape-invalid"`, or a `"throw:<String(error)>"` template-family value, describing the failure mode — `"missing-entry"` covers both the lookup-returned-nothing case and the defensive kind-mismatch case (per PIC's **Unknown-reason rule**), and `"literals-shape-invalid"` covers all four `literals`-field shape failures (not an array; array with a non-string element; empty array; array of strings containing `""`); the field is deliberately named `failure` rather than `observed` because it carries an internal failure-mode discriminator from a closed discriminator set (two literal forms — `"missing-entry"`, `"literals-shape-invalid"` — plus the `"throw:<…>"` template family whose payload is constructed at the emission site per the truncation-and-escape rule pinned by PIC's **Unknown-reason rule**), not an externally-observed SDK value (operator triage tooling that keys on `details.observed` for `session-shutdown-reason-unknown` MUST NOT silently extend that key's semantics to this code, per PIC's **Unknown-reason rule**). The `"throw:<String(error)>"` discriminator's grammar is pinned by PIC's **Unknown-reason rule**: the colon is a fixed prefix separator and is not escaped; the `String(error)` payload is truncated to at most 256 UTF-16 code units, then has its control characters escaped (`\n`/`\r`/`\t` → the two-character literal escapes `\\n`/`\\r`/`\\t`; every other code point in U+0000–U+001F and U+007F–U+009F → the six-character escape `\\uXXXX`, where `XXXX` is exactly four lowercase hexadecimal digits (`0`–`9`, `a`–`f`) zero-padded to the full four-digit width, e.g. U+007F → `\\u007f`, per PIC's **Unknown-reason rule**) before concatenation with the prefix; a literal colon or a `"throw:"` substring inside the truncated payload is passed through unchanged, and operator triage tooling MUST key on `startsWith("throw:")` against the discriminator. On a coercion failure (the `String(error)` call itself throws), the discriminator falls back to the literal string `"throw:<unreadable>"` rather than the bare `"<unreadable>"` — the `"throw:"` prefix is preserved so operator deduplication on `(code, details.failure)` continues to bucket coercion-failure rows under the throw discriminator instead of collapsing them into a separate `"<unreadable>"` bucket. The bare `"<unreadable>"` fallback convention used by `session-shutdown-reason-unknown`'s `details.observed` and `session-shutdown-teardown-step-failed`'s `details.error` payloads is unchanged — those payloads carry no `"throw:"` prefix to preserve. | `loom/host/session-swap-instance-survived` | E | runtime | The loom fail-fast tripwire fired: a loom-registered slash `handler` (at entry) or the `session_start` handler observed the private per-extension-instance `sessionSwapTornDown` flag armed — the flag the `session_shutdown` handler sets after a session-only teardown (`event.reason ∈ {"new","resume","fork"}`) — proving the extension instance survived a session-only `session_shutdown` and is dispatching again, contradicting the `governed-by-rebind` determination recorded at [host-prerequisites clause (a)](../pi-integration-contract/host-prerequisites.md#degraded-state-host-prerequisites) (at the loom 1.0 Pi-SDK pin Pi tears the runtime down and rebinds a fresh instance on every session-only swap, so this never occurs on a conformant minor). | `loom/host/session-shutdown-teardown-step-failed` | W | runtime | A teardown sub-step inside the `session_shutdown` handler threw, was caught by the per-step `try`/`catch` mandated by PIC's **Per-step isolation** paragraph, and did not prevent the remaining sub-steps from running. Covers sub-step 1 (`LoomRegistry.drain` and `LoomRegistry.initDrainStateTag` — two independent call sites, one potential emission each), sub-step 3 (the three bounded-await `Clock` calls — the deadline capture `Clock.now()`, the cap-arming `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, ...)`, and the success-path `Clock.clearTimeout(...)` — three independent call sites, one potential emission each), sub-step 4 (`discoveryWatcher.close()`, `settingsWatcher.close()`, the debounce-handle `Clock.clearTimeout(...)` call — three independent call sites, one potential emission each), and sub-step 5 (each enumerated listener-detach call — one potential emission per failing call). Sub-step 2 retains its own per-entry isolation rule and does **not** emit this code; sub-step 3's `Promise.allSettled` over `disposeBarrier`s likewise does **not** emit this code (only its three bounded-await `Clock` calls do). `details: { step: 1 \| 3 \| 4 \| 5, call: <static call-site label>, error: <error.message> }` — the `call` label is a fixed string per call site, drawn from a **closed normative set per `details.step`** (PIC's **Per-step isolation** paragraph is the source of truth): sub-step 1: `"loomRegistry.drain"`, `"loomRegistry.initDrainStateTag"`; sub-step 3: `"Clock.now()"`, `"Clock.setTimeout(awaitCap)"`, `"Clock.clearTimeout(awaitCap)"`; sub-step 4: `"discoveryWatcher.close"`, `"settingsWatcher.close"`, `"Clock.clearTimeout(debounce)"`; sub-step 5: `"ctx.signal.removeEventListener"`, `"toolSignal.removeEventListener"`, `"parentInvokeSignal.removeEventListener"`. Adding a new call site requires a spec edit to extend this set in the same commit; the labels are wire contract rather than implementation-chosen, so deduplication on `(code, details.step, details.call)` is meaningful both across runs and across independent implementations. `details.error` carries the captured throw's [underlying-error string](placeholder-rendering-b.md#underlying-error-coercion) — the `.message` when the throw is an object with a string `.message`, otherwise `String(error)`, falling back to the literal string `"<unreadable>"` if that coercion itself throws (the same `"<unreadable>"` convention `session-shutdown-reason-unknown`'s `details.observed` applies to its captured value). Routed via `console.error` rather than the persistent-diagnostic channel because Pi's `ExtensionRuntime.invalidate(...)` may already have invalidated the `pi.*` surface (see [Pi Integration Contract — Extension entry point](../pi-integration-contract.md)). | [Pi Integration Contract — Extension entry point](../pi-integration-contract.md) | — | `session_shutdown teardown step <step> failed at <call>: <error>`. |
