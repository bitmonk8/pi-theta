# Code registry host

### `loom/host/*` — host-observed anomalies

The column legend, the closed *Phase* taxonomy, and the *E/W* severity carve-out that govern this table are defined in [Diagnostic shape — Code registry rules (normative)](./diagnostic-shape.md#column-legend).

Diagnostics in this namespace are emitted from loom's interactions with the Pi host. The namespace covers both anomalies the loom runtime observes coming *from* the Pi host that the spec's host-prerequisites contract does not enumerate, and loom-internal failures of host-related machinery — pinned-constant blocks that snapshot host SDK types, host-event handlers, and similar — whose immediate cause is loom-internal corruption but whose observation site is a host-driven code path. They route via `console.error` because the typical observation site is the `session_shutdown` teardown handler, where Pi's `ExtensionRuntime.invalidate(...)` may already have invalidated the persistent-diagnostic channel — the same delivery-sink carve-out as `loom/runtime/reload-teardown-timeout`. Every `console.error` call this namespace makes from the teardown handler is wrapped per [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation): an inner throw from `console.error` MUST be swallowed and MUST NOT unwind the handler.

On a coercion failure (the `String(error)` call itself throws), the discriminator falls back to the literal string `"throw:<unreadable>"` rather than the bare `"<unreadable>"` — the `"throw:"` prefix is preserved so operator deduplication on `(code, details.failure)` continues to bucket coercion-failure rows under the throw discriminator instead of collapsing them into a separate `"<unreadable>"` bucket. The bare `"<unreadable>"` fallback convention used by `session-shutdown-reason-unknown`'s `details.observed` and `session-shutdown-teardown-step-failed`'s `details.error` payloads is unchanged — those payloads carry no `"throw:"` prefix to preserve.
