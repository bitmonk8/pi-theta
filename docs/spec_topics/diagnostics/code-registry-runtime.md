# Code registry runtime

### `loom/runtime/*` ‚Äî runtime panics, runtime-defect surface, and delivery failures

loom 1.0.0 has exactly six **panic sources** (the rows below tagged as panics). Each surfaces through the channels defined in [Errors and Results ‚Äî Runtime panics](../errors-and-results.md) (slash-command system note for top-level looms; `Err(QueryError { kind: "invoke_infra_error", cause: "panic", ... })` to an `invoke` parent). One further code ‚Äî `loom/runtime/internal-error` ‚Äî covers the **runtime-defect surface** also defined in [Errors and Results ‚Äî Runtime panics](../errors-and-results.md): an unanticipated interpreter or adapter throw outside the closed panic-source list, routed through the same channels as panics but not itself a panic source. The code is registered and stable; only its trigger condition is intentionally open-ended, which does not weaken the closed-registry rule (rule 2 above) ‚Äî that rule constrains *codes*, not triggers. Six further codes ‚Äî `system-note-delivery-failed`, `subagent-dispose-failure`, `registry-swap-failed`, `registration-cache-collision`, `reload-teardown-timeout`, and `active-set-restore-failed` ‚Äî are delivery-, rebuild-, registration-, teardown-, or active-set-restore-failure diagnostics emitted, respectively, by the system-note fallback path, the subagent session lifecycle teardown, the watcher's build-aside-then-publish swap, the prompt-mode tool-registration cache's collision check, the `session_shutdown` handler's bounded await on in-flight subagent disposal, and the prompt-mode (and prompt ‚Üí prompt cross-mode `invoke`) tool-active-set restore-failure protocol, all defined in [Pi Integration Contract](../pi-integration-contract.md). `reload-teardown-timeout` differs from the other five in delivery channel: because Pi's `ExtensionRuntime.invalidate(...)` may already have invalidated the `pi.*` surface by the time the teardown handler runs, the timeout diagnostic is emitted via `console.error` rather than through the standard persistent-diagnostic channel. One further code ‚Äî `loom/runtime/subagent-model-unresolved` ‚Äî is a subagent-spawn precondition failure emitted by the pre-spawn model guard in [Pi Integration Contract ‚Äî Conversation drive ‚Äî subagent mode](../pi-integration-contract/subagent.md#subagent-pre-spawn-model-guard) when the resolved `model` is `undefined`; it routes through the standard `loom-system-note` channel.

The presence entitlement for `details.event.invocation_id` is likewise scoped to the structured-emission path: neither the serialiser-throw two-token fallback nor the construction-site-throw three-token fallback (per the *Construction-site wrap* sub-clause of [Pi Integration Contract ‚Äî Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation), preserved textually unchanged at the three-token shape) carries a separable `details.event.invocation_id` slot; consumers MUST first discriminate the path before asserting presence and MUST treat fallback-path emissions as carrying no observable `details.event.invocation_id` discriminator. `display: false` on the companion `loom-system-note` (operator-visible only via the structured payload, no inline transcript content). Division by zero, modulo by zero, integer overflow, and explicit author-driven panics are deliberately not in loom 1.0.0's panic catalogue ‚Äî division and modulo by zero yield IEEE-754 `Infinity` / `NaN` per the `number` rules in [Expressions](../expressions.md), and there is no `panic(...)` builtin. Adding any of those is a registry-level change per rule 2 above.
