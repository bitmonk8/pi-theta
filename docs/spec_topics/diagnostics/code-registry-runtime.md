# Code registry runtime

### `theta/runtime/*` — runtime panics, runtime-defect surface, and delivery failures

The column legend, the closed *Phase* taxonomy, and the *E/W* severity carve-out that govern this table are defined in [Diagnostic shape — Code registry rules (normative)](./diagnostic-shape.md#column-legend).

theta 1.0.0 has exactly six **panic sources** — the closed panic-source list enumerated in [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics); the table below carries one row per code in that list, identifiable by matching each row's *Code* cell against that closed list. Each surfaces through the channels defined in [Errors and Results — Runtime panics](../errors-and-results.md) (slash-command system note for top-level thetas; `Err(QueryError { kind: "invoke_infra", cause: "panic", ... })` to an `invoke` parent). One further code — `theta/runtime/internal-error` — covers the **runtime-defect surface** also defined in [Errors and Results — Runtime panics](../errors-and-results.md): an unanticipated interpreter or adapter throw outside the closed panic-source list, routed through the same channels as panics but not itself a panic source. The code is registered and stable; only its trigger condition is intentionally open-ended, which does not weaken the closed-registry rule (rule 2 above) — that rule constrains *codes*, not triggers. Six further codes — `system-note-delivery-failed`, `subagent-dispose-failure`, `registry-swap-failed`, `registration-cache-collision`, `reload-teardown-timeout`, and `active-set-restore-failed` — are delivery-, rebuild-, registration-, teardown-, or active-set-restore-failure diagnostics emitted, respectively, by the system-note fallback path, the subagent child-process teardown (`subagent-dispose-failure` re-scoped to any subagent-mode teardown-step throw — stdin close or bounded kill), the watcher's build-aside-then-publish swap, the prompt-mode tool-registration cache's collision check, the `session_shutdown` handler's bounded await on in-flight subagent teardown, and the prompt-mode (and prompt → prompt cross-mode `invoke`) tool-active-set restore-failure protocol, all defined in [Pi Integration Contract](../pi-integration-contract.md). Ten further codes cover the subagent child-process failure classes of the child-process theta design (see [Pi Integration Contract — Conversation drive — subagent mode](../pi-integration-contract/subagent.md#pic-9)): `subagent-spawn-failed`, `subagent-child-crashed`, `subagent-wire-parse-failed`, `subagent-teardown-timeout`, `subagent-callable-hash-mismatch`, `subagent-model-preflight-mismatch`, and the four RFC 0006 marshalling codes `subagent-envelope-parse-failed`, `subagent-envelope-schema-skew`, `subagent-exit-without-envelope`, and `subagent-params-validation-failed`. `reload-teardown-timeout` differs from the other five in delivery channel: because Pi's `ExtensionRuntime.invalidate(...)` may already have invalidated the `pi.*` surface by the time the teardown handler runs, the timeout diagnostic is emitted via `console.error` rather than through the standard persistent-diagnostic channel. One further code — `theta/runtime/subagent-model-unresolved` — is a subagent-spawn precondition failure emitted by the pre-spawn model guard in [Pi Integration Contract — Conversation drive — subagent mode](../pi-integration-contract/subagent.md#subagent-pre-spawn-model-guard) when the resolved `model` is `undefined`; it routes through the standard `theta-system-note` channel. One further code — `theta/runtime/validator-cache-collision` — is a content-addressing collision diagnostic emitted by the per-query AJV compiled-validator cache when two distinct lowered schema documents hash to the same schema slug; it is the runtime-cache application of the [schema-slug collision posture](../schema-subset.md#schema-slug-collision-posture) defined in Schema Subset (and the [PIC-11](../pi-integration-contract/host-interfaces-services.md#pic-11) validator contract), is kept distinct from the load-time `theta/load/schema-slug-collision` (`$defs` hoist) and the `theta/runtime/registration-cache-collision` (tool-registration) diagnostics so a collision's originating cache is identifiable from the code alone, and routes through the standard persistent-diagnostic channel. One further code — `theta/runtime/watcher-terminated` — is a watcher-lifecycle event emitted when the chokidar file watcher's `error` route fires or it throws such that one or more watched roots stop delivering events (per [Pi Integration Contract — PIC-55](../pi-integration-contract/registration-steps.md#pic-55)); it routes through the standard `theta-system-note` channel as its primary sink. One further code — `theta/runtime/custom-type-unsafe` — is a binder-input precondition failure emitted by the compact-transcript renderer when an included `CustomMessage`'s `customType` is not transcript-safe (per [BNDR-9](../binder/binder-model-and-context.md#bndr-9)); the binder rejects the message and aborts transcript construction before rendering rather than emit bytes that break [BNDR-7](../binder/binder-model-and-context.md#bndr-7)'s reproduce-exactly contract, and it routes through the standard `theta-system-note` channel.

Wherever a row below carries a caught throw's error message — in its Message template (the `<error.message>`, `<error>`, and `<dispose error first line>` placeholders) or in a `message` / `hint` field — that string is the throw's [underlying-error string](placeholder-rendering-b.md#underlying-error-coercion): a non-Error throw is coerced deterministically rather than yielding `undefined` or a synchronous `TypeError`.
