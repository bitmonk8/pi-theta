# Diagnostic shape

**Persistent diagnostics (default).** All `loom/parse/*`, `loom/load/*`, and `loom/runtime/*` diagnostics are delivered via the channel below, with five carved-out exceptions — all emitted via `console.error` from the `session_shutdown` handler because Pi's `ExtensionRuntime.invalidate(...)` (the canonical `ExtensionRuntime` pin lives at [Pi Integration Contract — Registration steps step 4](../pi-integration-contract/registration-steps.md#extension-runtime-invalidate-teardown)) may already have invalidated `pi.sendMessage` by the time the handler runs (see [Pi Integration Contract — Extension entry point](../pi-integration-contract.md)): `loom/runtime/reload-teardown-timeout` (the bounded-await timeout in sub-step 3), `loom/host/session-shutdown-reason-unknown` (the unknown-reason validation defined in PIC's **Unknown-reason rule**), `loom/host/session-shutdown-pinned-constant-unreadable` (the handler-entry pinned-constant lookup failure defined in PIC's **Unknown-reason rule**), `loom/host/session-shutdown-teardown-step-failed` (the per-step isolation rule defined in PIC's **Per-step isolation** paragraph), and `loom/host/session-shutdown-runtime-degraded` (the post-sub-step-4 degraded-state notification defined in [PIC's `reason: "new" | "resume" | "fork"` edge-case bullet](../pi-integration-contract/session-only-degraded-state.md#session-only-reason-degraded-state)). The structured `Diagnostic` shape below still applies; only the delivery sink differs. Each of these five teardown-handler `console.error` calls is itself wrapped per [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation): a throw from `console.error` (closed stdio, file-descriptor exhaustion, a host that proxies `console` and rejects writes during teardown) MUST be swallowed so a failing last-resort sink does not unwind the teardown handler. The "no ceiling fails silently" closure relies on this defended sink rather than on an implicit assumption that `console.error` always succeeds.

```
pi.sendMessage(
  {
    customType: "loom-system-note",
    content:    <serialised batch>,            // see "Serialised content format" below
    display:    true,
    details:    { diagnostics: <Diagnostic[]> } // single-element array for runtime/single-error cases
  },
  { triggerTurn: false }
)
```


The `pi.registerMessageRenderer("loom-system-note", …)` registered by the loom extension formats these into transcript-persistent, dim-styled notes; see [Pi Integration Contract — System notes](../pi-integration-contract.md) for the renderer registration rules and the best-effort fallback (`ctx.ui.notify` then `loom/runtime/system-note-delivery-failed` then `console.error`) that applies when `pi.sendMessage` itself throws (`pi.sendMessage` returns `void`; only synchronous throws are observable). The renderer MUST be registered synchronously inside the extension factory **before** the first discovery scan kicks off, so the first batch of scan diagnostics renders through the loom-specific renderer rather than as raw fallback text. This is the only diagnostic sink for loom-author-facing parse, type, load, and runtime errors.

The `loom-system-note` channel also carries operator-facing runtime failure events (`details: { event: RuntimeEvent }`) for group-A members of the always-log set; the two `details` payload shapes are disjoint by key and are specified under "System notes" and "Runtime event channel" in [Pi Integration Contract](../pi-integration-contract.md). Renderers MUST switch on which key is present and MUST NOT assume both.

**Transient toasts (auxiliary).** Failures internal to the loom extension's own bookkeeping that the user must see immediately but that do **not** belong in the transcript — the chokidar watcher itself throwing, an unrecoverable settings I/O exception that no `loom/load/settings-*` code covers, an internal extension invariant violation — use `ctx.ui.notify(message, "error")` directly. This is a narrow secondary surface; loom-author-facing diagnostics (anything with a `loom/parse/*`, `loom/load/*`, or `loom/runtime/*` code) MUST go through the persistent channel above and MUST NOT be routed through `ctx.ui.notify` as their primary sink. A synchronous throw from `ctx.ui.notify` on this last-resort surface (`ui.notify` is documented synchronous and able to throw — see [Pi Integration Contract — Host interfaces](../pi-integration-contract/host-interfaces-core.md)) MUST be swallowed, mirroring the `console.error`-throw swallow rule at [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#pic-emission-swallow): the toast surface is itself the bottom of the stack for these failures, so a failing `ctx.ui.notify` does not unwind the watcher callback or invariant-check site that invoked it and no further diagnostic is emitted on the inner throw.

**Re-scan deduplication.** A watcher-triggered reload re-emits the persistent diagnostic for any file whose contents are still broken. The runtime does **not** attempt to clear or supersede prior `loom-system-note` messages from the transcript — Pi's `pi.sendMessage` API has no documented "remove" or "replace by metadata" counterpart, and `display: true` system notes are part of the immutable session log. Authors will therefore see the same diagnostic line recur after each reload until the underlying file is fixed; this is the loom 1.0 contract and the renderer MUST NOT attempt to suppress duplicates.

Internal diagnostic shape:

```
{
  severity: "error" | "warning",
  code:     string,                          // e.g., "loom/parse/binding-case-mismatch"
  file?:    string,                          // absolute path; omitted for the location-less codes enumerated below
  range?:   { start: { line, column }, end: { line, column } },  // 1-indexed; end exclusive; omitted together with file
  message:  string,                          // single-line summary
  hint?:    string,                          // optional suggested fix
  related?: array<{ file, range, message }>, // related sites (e.g., the colliding declaration)
  masked?:  array<string>,                   // hard-ceiling co-fire enumeration; see below
  details?: object,                          // optional per-row structured payload; shape pinned by each row's Trigger prose — see note below
}
```

The optional `details?` field carries a diagnostic's per-row structured wire payload; its shape is pinned normatively by that row's *Trigger* prose in the [Code registry](./code-registry-parse.md#code-registry) below, and the `details` shape conventions paragraph immediately below governs only the `session_shutdown` event-payload subset (the `details.event.{...}` nesting on `loom/runtime/cancelled-by-session-shutdown` and `loom/host/session-shutdown-runtime-degraded`). Every row-documented `details.{...}` payload lives on the `Diagnostic` object — i.e. inside `CustomMessage.details.diagnostics[i].details` — not on the outer `CustomMessage.details`, which stays closed to `{ diagnostics: Diagnostic[] }` plus the runtime-event-channel carve-out (`details: { event: RuntimeEvent }`) noted above. A rule-4-compliant test asserting on a row's payload therefore inspects `diagnostics[i].details`.

**Location-less diagnostics (`file` / `range` omitted).** A closed set of diagnostics has no source location at emission time and carries neither `file` nor `range`: the four `loom/load/*` codes `loom/load/host-incompatible`, `loom/load/extension-bootstrap-failed`, `loom/load/discovery-slow`, and `loom/load/package-read-timeout`, plus the four `loom/host/session-shutdown-*` teardown codes `loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, and `loom/host/session-shutdown-runtime-degraded`. For these codes the wire shape omits `file` and `range` entirely rather than carrying a sentinel `file` or a zero-valued `range`, so a consumer that validates the envelope strictly sees a `Diagnostic` whose two located-site fields are absent (the optional-marked `file?` / `range?` above) rather than one that fails validation against a fabricated location. This set is closed in loom 1.0.0: adding or removing a location-less code requires a spec edit to this paragraph (and, for an addition, the new code's registry row) before the omitted-fields form is permitted on the wire — mirroring the closed-set future-addition obligation the [`details` shape conventions](#session-shutdown-details-conventions) paragraph below pins for `details.event.*`. The `related?` element shape is unaffected: `related` entries always describe located sites and retain their required `{ file, range, message }` shape.

<a id="session-shutdown-details-conventions"></a> **`details` shape conventions for the `session_shutdown` event payload.** Diagnostics that surface the Pi `session_shutdown` event payload — currently the per-invocation `loom/runtime/cancelled-by-session-shutdown` note and the teardown-handler `loom/host/session-shutdown-runtime-degraded` row, both registered below — nest the payload under `details.event.{...}` so a single nested shape covers both codes for log-pipeline and operator-tooling consumers. **Closed at `{reason}` in loom 1.0.0.** The only field pinned by this convention in loom 1.0.0 is `details.event.reason`; the convention does **not** authorise additional `SessionShutdownEvent` sibling fields under the `details.event.` prefix between SDK-pin bumps, and a future Pi SDK addition that the runtime decides to surface (e.g. a hypothetical `details.event.timestamp`) requires a spec edit to this paragraph and to the affected row(s) below before it is permitted on the wire. Operator-tooling consumers parsing across `loom/runtime/cancelled-by-session-shutdown` and `loom/host/session-shutdown-runtime-degraded` are therefore entitled to validate `details.event` strictly against the closed `{reason}` shape between SDK-pin bumps. *Per-code `details.event.reason` value spaces.* The `{reason}` shape closure above pins the *set of fields* but does **not** pin a uniform value space for `details.event.reason` itself: the two codes' value spaces diverge and consumers MUST validate the field against the per-code value space, not against a uniform value-space pin. `loom/runtime/cancelled-by-session-shutdown` admits the wider union `"quit" | "reload" | "new" | "resume" | "fork" | string` — the closed five-arm union plus a `string` arm reserved for the unknown-reason path that carries the verbatim observed value coerced via `String(event.reason)`, plus the literal `"<unreadable>"` sentinel for the throwing-property-access and snapshot-failure paths — per the captured-value enumeration owned by [PIC's **Unknown-reason rule**](../pi-integration-contract/unknown-reason-rule.md#unknown-reason-rule) and restated in that row's description below. `loom/host/session-shutdown-runtime-degraded` is closed to `{"new", "resume", "fork"}` per the diagnostic-emission predicate `pinnedConstantReadOk && capturedEventReason ∈ {"new", "resume", "fork"}` owned by [PIC's `reason: "new" | "resume" | "fork"` edge-case bullet](../pi-integration-contract/session-only-degraded-state.md#session-only-reason-degraded-state) and restated in that row's description below — the unknown-reason and snapshot-failure paths are excluded by that predicate, and the `"quit"` and `"reload"` arms are excluded because the runtime is being torn down on those reasons. The closed-set entitlement above applies to **structured emissions only**: on the construction-site fallback path the per-row wire token shape is pinned by the *Construction-site wrap* sub-clause of [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation) (which the *Runtime construction obligation* paragraph below cross-references for the `"<unreadable>"` sentinel substitution and is the source of truth for the per-row token shapes), and the per-row shape differs across the two nested-shape codes: for `loom/host/session-shutdown-runtime-degraded` the runtime emits the unstructured two-token bare-`` `${code} <unreadable>` `` form carrying the literal `"<unreadable>"` sentinel for `details.event.reason` (the only payload field this row carries under `details.event`); for the per-invocation `loom/runtime/cancelled-by-session-shutdown` note the runtime emits the unstructured three-token form `` `${code} ${entry.loom} <unreadable>` `` (the catch arm reads `entry.loom` separately from the [`ActiveInvocationRegistry`](../pi-integration-contract/active-invocation-registry.md#active-invocation-registry) entry the per-invocation `finally` already holds, preserving per-invocation `loom` discrimination via the interpolated `entry.loom` token while substituting the `"<unreadable>"` literal for `details.event.reason` only). Consumers MUST treat the construction-site fallback `details.event.reason` value on either row as a separate value-space outside the closed `{"new", "resume", "fork"}` set, mirroring the `"<unreadable>"` enumeration carved out for `loom/runtime/cancelled-by-session-shutdown` above. A consumer that learns the value space from one code and applies it to the other will under- or over-validate; the per-code value space is the normative contract. *Runtime-constructed sibling carve-out.* The `details.event.loom` field on `loom/runtime/cancelled-by-session-shutdown` (the slash-name of the cancelled invocation, registered under that row below) is a **runtime-constructed sibling** under the `details.event.` prefix — not an SDK-derived `SessionShutdownEvent` field — and is exempt from the strictness entitlement above; the `{reason}` closed-shape pin governs only SDK-derived siblings (i.e., fields whose source is the `SessionShutdownEvent` payload Pi delivers), and operator-tooling consumers MUST whitelist `details.event.loom` on the `loom/runtime/cancelled-by-session-shutdown` code when applying the strict-validation entitlement. The runtime MUST include `details.event.loom` on every emission of `loom/runtime/cancelled-by-session-shutdown`, sourced from the `loom` field of the cancelled invocation's [`ActiveInvocationRegistry`](../pi-integration-contract/active-invocation-registry.md#active-invocation-registry) entry — registered at invocation start per that registry's contract, in its canonical form (post-`pi.looms` mapping, post-`as` rename, basename-hyphen-normalised) with no leading `/`; the field is therefore a normative part of that row's wire contract and tests are entitled to assert its presence. The `details.event.invocation_id` field on the same row is similarly a **runtime-constructed sibling** under the `details.event.` prefix — not an SDK-derived `SessionShutdownEvent` field — and is exempt from the strictness entitlement above on the same footing as `details.event.loom`. The runtime MUST include `details.event.invocation_id` on every emission of `loom/runtime/cancelled-by-session-shutdown`, sourced from the `invocationId` field of the cancelled invocation's [`ActiveInvocationRegistry`](../pi-integration-contract/active-invocation-registry.md#active-invocation-registry) entry (registered at invocation start per that registry's contract, in canonical lowercase 8-4-4-4-12 hex form per the [§7 `<invocation-id>` placeholder convention](./placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders)); the field is therefore a normative part of that row's wire contract on the structured-emission path and tests are entitled to assert its presence. *Presence-entitlement scope — structured emissions only.* The presence entitlement above applies to **structured emissions only**: on the serialiser-throw fallback path the wire form is the unstructured two-token sequence `` `${code} ${detailsEventReason}` `` per the *Two-token fallback for nested-shape codes* sub-clause of [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation) (which carries no `loom` slot at all), and on the construction-site-throw fallback path the wire form is the unstructured three-token sequence `` `${code} ${entry.loom} <unreadable>` `` per that paragraph's *Construction-site wrap* sub-clause (which is the source of truth for the per-row token shapes and which, for the per-invocation note, preserves per-invocation `loom` discrimination on the wire by interpolating the `entry.loom` token read separately from the [`ActiveInvocationRegistry`](../pi-integration-contract/active-invocation-registry.md#active-invocation-registry) entry alongside the `"<unreadable>"` sentinel substituted for `details.event.reason` only; the `runtime-degraded` row's *Construction-site wrap* shape is unchanged at the two-token form `` `${code} <unreadable>` `` since that row carries no `loom` field under `details.event` to preserve). Consumers MUST treat fallback-path emissions as a separate wire shape from the structured emission and MUST NOT validate them against the structured presence-entitlement above; a test that asserts `details.event.loom` presence without first discriminating the structured-emission path from the two fallback paths over-fails on every multi-invocation shutdown whose per-invocation notes route through either fallback. The same scope applies to `details.event.invocation_id` on the per-invocation note's structured emission per the *Runtime-constructed sibling carve-out* clause above: neither fallback path carries a separable `details.event.invocation_id` slot (the *Construction-site wrap* sub-clause's three-token shape preserves only `entry.loom`, on the same footing as the existing carve-out), and a test asserting its presence MUST first discriminate the structured-emission path on the same footing as the `details.event.loom` discrimination rule above. Future runtime-constructed sibling fields under `details.event.*` similarly require a spec edit to this paragraph and to the affected row(s) below before they are permitted on the wire (on the same footing as the SDK-derived future-addition obligation above), but are not blocked by this convention's strictness pin once enumerated. *Runtime construction obligation.* The runtime MUST construct `details.event` as a fresh object containing exactly the pinned fields per the per-code field set enumerated above (no `...event` spread, no `Object.assign(..., event, ...)`, no per-key copy loop over `Object.keys(event)`, and no other dynamic-field-set mechanism whose member set is sourced from the SDK-delivered `SessionShutdownEvent` rather than from this paragraph) — for `loom/host/session-shutdown-runtime-degraded` the constructed object is `{ reason }` and for `loom/runtime/cancelled-by-session-shutdown` it is `{ reason, loom, invocation_id }`, with each pinned field assigned by an explicit static property name. This makes the operator-tooling consumer's strict-validation entitlement above deserved by construction rather than by implicit coincidence with the V1 `SessionShutdownEvent` member set, so a future SDK addition (e.g. a hypothetical `event.timestamp`) does not silently leak onto the wire under a future contributor's spread-style refactor; future SDK-derived or runtime-constructed fields require a spec edit to this paragraph and to the affected row(s) below before they may be added to the constructed object. Per-code runtime-constructed failure-mode payloads — `details.observed` for `loom/host/session-shutdown-reason-unknown`, `details.failure` for `loom/host/session-shutdown-pinned-constant-unreadable`, `details.step` / `.call` / `.error` for `loom/host/session-shutdown-teardown-step-failed` — remain flat under `details` because they are not the SDK event payload. The teardown-handler emission's `try`/`catch` envelope and the bare-`code` fallback on serialiser throw are owned by [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation); this paragraph owns only the wire-format nesting convention. The two-token bare-`` `${code} ${details.event.reason}` `` fallback for these two codes on serialiser throw is owned by [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation). The construction-site `try`/`catch` that defends *this* paragraph's runtime construction of `details.event` (and the PIC *Hoist obligation* clause's pre-wrap construction of the `detailsEventReason` local) against a construction-time throw — routing the catch arm through the bare-`code` or two-token fallback above with the literal `"<unreadable>"` sentinel substituted for both `details.event.reason` and (on the per-invocation note) `details.event.loom` — is owned by the *Construction-site wrap* sub-clause of [Pi Integration Contract — Diagnostic-emission isolation](../pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation), which also pins the `"<unreadable>"` literal as the single source of truth for catch-arm payload defaults so the construction-site failure path's value space stays inside the captured-value enumeration the PIC **Unknown-reason rule** pins for `details.event.reason`.

**Code namespaces:**

- `loom/parse/*` — diagnostics produced by the parse pipeline: lexical errors, syntactic errors, and static type-system checks. The `Phase` column in the registry table distinguishes `lex` / `parse` / `type` within this namespace.
- `loom/load/*` — file-load and registration errors (unreadable file, missing or wrong-type discovery source, name collision, invalid frontmatter, unresolvable `tools:` entry).
- `loom/runtime/*` — runtime-phase events: execution panics (`MatchError`, index out of bounds, etc.), the runtime-defect surface (`loom/runtime/internal-error`), and delivery / lifecycle / teardown failures. Routing varies per code — see the [`loom/runtime/*` registry section](./code-registry-runtime.md#loomruntime-runtime-panics-runtime-defect-surface-and-delivery-failures) and [Errors and Results — Runtime panics](../errors-and-results.md) for per-code channels (most route through `loom-system-note`; `loom/runtime/reload-teardown-timeout` routes through `console.error`).
- `loom/host/*` — anomalies that surface during loom's interaction with the Pi host. The namespace covers both anomalies the loom runtime observes coming *from* the Pi host that the spec's host-prerequisites contract does not enumerate (e.g., a `session_shutdown` `event.reason` outside the closed five-arm union, or a throw from a `session_shutdown` teardown sub-step) and loom-internal failures of host-related machinery — pinned-constant blocks that snapshot host SDK types, host-event handlers, and similar — whose immediate cause is loom-internal corruption but whose observation site is a host-driven code path (e.g., a corrupted snapshot of `SessionShutdownEvent['reason']` failing the `session_shutdown` handler's entry-time pinned-constant lookup; the row's *Trigger* describes the loom-internal corruption while its emission site is the host event handler). Emitted via `console.error` since the typical observation site is mid-teardown when the persistent-diagnostic channel may already be invalidated.

<a id="loom-typecheck-namespace"></a> **Out of scope: `loom/typecheck/*`.** The `loom/typecheck/*` namespace is a parallel surface reserved for build-time `tsc` brand strings (e.g. `loom/typecheck/session-shutdown-reason-snapshot`, embedded in the build-time bidirectional type-equality assertion per [Pi Integration Contract — Pi version bump procedure](../pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) step 5 so a typecheck failure surfaces the brand verbatim in the failing `tsc` diagnostic). Brands in this namespace share the `loom/...` prefix but are **not** runtime diagnostics — they carry no registry row, do not flow through any of the delivery channels above, and are out of scope for the Code registry rules below. A reader auditing the `loom/...` surface should treat `loom/typecheck/*` matches as build-time brands owned by PIC step 5 rather than as missing registry entries. *Naming convention.* Each brand has the shape `loom/typecheck/<kebab-case-name>` — the rule-name component is kebab-case (lowercase ASCII letters, digits, and `-`) and MUST NOT contain `<...>`-style placeholders, since brand strings are TypeScript string-literal types compared verbatim by `tsc` and a placeholder would break the verbatim-surfacing property step 5 keys on. *Definition site.* Each `loom/typecheck/*` brand MUST be defined inline at the spec-prose site that introduces its build-time assertion (the brand is named where the assertion is pinned); the namespace is intentionally distributed and there is **no** central registry of `loom/typecheck/*` brands analogous to the Code registry below. *Audit-gate carve-out.* Audit gates that scan source or spec for `loom/...` literals and reconcile them against the Code registry below MUST carve out `loom/typecheck/` as a prefix-disjoint namespace — a `loom/typecheck/*` literal MUST NOT be reported as a missing registry entry.

**Serialised content format.** This rule fixes the **input** to the renderer (the `content` field on the `CustomMessage` argument), not the renderer's return value. The renderer itself returns a `pi-tui` `Component | undefined`, per the **Renderer registration** block in [Pi Integration Contract — System notes](../pi-integration-contract.md). The `content` string of each `loom-system-note` follows the line format `"<file>:<line>:<col>: <code>: <message>"`, optionally followed by `"\n  hint: <hint>"` when a hint is present. When a diagnostic omits `file` and `range` (the location-less codes enumerated under **Location-less diagnostics** above), its line drops the leading `<file>:<line>:<col>:` segment and renders as `"<code>: <message>"` (the optional `"\n  hint: <hint>"` continuation is unchanged); the renderer reads the omitted fields as absent rather than substituting a sentinel `file` or zero-valued `range` into the line. Each related site (one per element of the `Diagnostic.related` array, whose entry shape is `{ file, range, message }`) is appended on its own line, indented by two spaces (matching the hint line) and following the format `"  <file>:<line>:<col>: <message>"` — the same `<file>:<line>:<col>` triple as the main line and **no** `<code>` prefix, because related entries carry no code. The `<line>:<col>` are the 1-indexed `range.start.line` and `range.start.column` of the related entry's `range` (the `start` endpoint, consistent with the main line; the exclusive `end` endpoint is not rendered). When a single message carries a multi-error batch (the scan-time case), each `Diagnostic` becomes one such line block and successive blocks are separated by a single blank line; the corresponding structured `Diagnostic[]` is carried in `details.diagnostics` for the renderer and for downstream consumers (LSP integrations, test harnesses) that want the typed shape rather than the rendered string.

**Multi-error reporting.** Every parse / type pass collects all errors from the full file (and from transitive `.warp` imports) before failing. The loom is rejected with the complete list in **one `pi.sendMessage` call per `.loom` file** — `content` carries the full batch in the format above; `details.diagnostics` carries the same as a structured array — rather than fast-failing on the first error or fanning out one message per error. Authors get every problem in the file at once, in a single transcript entry.

## Code registry rules (normative)

Four rules govern the diagnostic-code surface:

1. <a id="diag-1"></a> **DIAG-1.** **Every author-visible diagnostic emitted by the runtime MUST carry a code from the registry below.** Emitting an unregistered code is a defect; tests are entitled to assert on the specific code at every documented diagnostic site.
2. <a id="diag-2"></a> **DIAG-2.** **The registry is closed.** Adding a new code, removing a code, or changing a code's namespace, severity, or trigger are all spec changes — not implementation changes. New diagnostic sites added by future spec work MUST land their codes in this table at the same time.
3. <a id="diag-3"></a> **DIAG-3.** **Codes are stable identifiers.** Renaming a registered code is a breaking change to the public diagnostics contract (tests, LSP integrations, system-note formatters, doc cross-links). Treat each code as part of the language surface.
4. <a id="diag-4"></a> **DIAG-4.** **The Message column is normative.** Every row in the registry below carries a *Message* (the rendered author-facing string with `<…>`-style placeholders for interpolated content); renderers MUST emit it character-for-character with placeholders interpolated. Tests asserting a diagnostic's rendered message MUST source the string from this column rather than copy-pasting prose from the spec rule's home page. Wording changes are spec-versioned breaking changes.

Naming convention for codes is `<namespace>/<kebab-case-rule-name>`. The rule-name component derives from the spec rule's short name; it is not generated, so the registry below is the source of truth.

<a id="masked-field-hard-ceiling-co-fire"></a>

**`masked` field (hard-ceiling co-fire).** The optional `masked` field on the `Diagnostic` shape above carries the hard-ceiling co-fire enumeration defined in [Hard Runtime Ceilings — `masked` field](../hard-ceilings/ceilings-3-and-4.md#masked-field) and [Hard Runtime Ceilings — Interaction between ceilings](../hard-ceilings/ceilings-3-and-4.md#ceiling-interaction-order). It is populated only on diagnostics emitted from a hard-ceiling check site whose precondition fired while at least one *sibling* hard ceiling's precondition was simultaneously satisfied at the same site — in V1 this is the `loom/runtime/invoke-depth-exceeded` row below (the only diagnostic-shape hard-ceiling surface; the other co-fire-eligible surfaces are runtime events, not diagnostics, and carry the same enumeration as `event.masked` per [Pi Integration Contract — Runtime event channel](../pi-integration-contract.md)). Each entry is one of `"ceiling#1"`, `"ceiling#2"`, `"ceiling#3"`, `"ceiling#4"`. The field is **omitted** when no co-fire occurred; implementations MUST NOT emit `masked: []`. The per-site reachable mask domain (in V1: empty for `loom/runtime/invoke-depth-exceeded`, so this diagnostic always omits `masked`), the V1 reachable predicate, and the pure-read MUST on the masking probe are owned by [Pi Integration Contract — PIC-1 *Hard-ceiling co-fire (`masked`)*](../pi-integration-contract/runtime-event-channel.md#pic-1).
