# Implementation Notes

This page collects the parser, runtime, and orientation notes. Two more focused implementation pages are split out:

- [Runtime Value Model](./runtime-value-model.md) — how Theta values are represented at runtime, equality semantics, wire-name translation.
- [Pi Integration Contract](./pi-integration-contract.md) — the named SDK surface from `@earendil-works/pi-coding-agent` the runtime depends on.

## Parser

- **Contract.** The parser MUST emit diagnostics matching [Diagnostics](./diagnostics.md), including multi-error reporting across the whole `.theta` or `.thetalib` file and its transitively imported `.thetalib` modules before failing. Parse errors surface as structured diagnostics compatible with Pi's error reporting.
- **Implementation hint (non-normative).** The reference implementation uses Chevrotain (TypeScript-native, no separate lexer generator required), with the grammar derived from the Chevrotain TinyC example and extended with `@`...`` query templates, `schema`, and the type-context schema-inference pass. The choice of toolkit and grammar lineage have no observable effect on a theta author and may be replaced by any implementation that satisfies the contract above.

## Runtime

- Implemented in TypeScript as a Pi extension module
- **Static-resolution load pass.** Each top-level theta load runs a single per-pass parse cache shared by every parse-time check that needs a callee's source: `invoke(...)` argument-type checking, cross-theta return-type inference at `invoke<T>(...)` and `.theta` callable call sites, and the parse-time invocation-cycle walker (per [Invocation — Static resolution](./invocation.md#static-resolution)). The pass walks transitively from the entry theta across literal `invoke` paths and `.theta` entries in `tools:`, parses and lowers each visited file once, and aggregates each visited file's diagnostics into the entry theta's drain (sorted by `(file, line, col)` per [Diagnostics — Multi-error reporting](./diagnostics.md)). When a transitively-reached callee fails to parse or lower, the parent emits `theta/load/callee-has-errors` at the referencing site — severity `error` for `tools:` entries (callable cannot be created; parent does not register), severity `warning` for `invoke(...)` literals (parent registers; static checks against the callee are skipped; runtime AJV is the safety net). The in-process re-parse path (per [Pi Integration Contract — Extension entry point](./pi-integration-contract.md)) drops the per-pass cache entry for the changed file and every transitive `.thetalib` importer; the validator-cache key (see the **Schema validation** bullet below) remains the schema slug, so a callee re-parse that produces an identical lowered schema reuses cached validators across reload. The drop happens as part of the `ThetaRegistry` swap, not as a separate watcher-driven path
- Holds a reference to the target conversation (the caller's session for prompt mode; a freshly spawned isolated session for subagent mode) and drives it turn-by-turn
- Each `@`...`` query issues a user turn against that conversation, services any tool-call loop the model returns, and resolves to the final assistant response
- Typed queries (schema inferred from binding type, function parameter, return type, or explicit `@<Schema>`...``) lower the schema to a provider-shaped strict tool-input schema; the returned tool-call payload is validated by the injected `SchemaValidator` service (see the **Schema validation** seam in [SchemaValidator interface](./pi-integration-contract/host-interfaces-services.md#schemavalidator-interface)) before being handed back to theta code. The behavioural contract for typed queries — the two-phase loop, the round cap, the provider compatibility surface — is normative and lives in [Pi Integration Contract — Conversation drive (prompt mode)](./pi-integration-contract.md) and [Query — Typed queries are tool-loop-shaped](./query.md). The theta 1.0 reference implementation:
    - Synthesises a one-shot tool named `__theta_respond_<slug>` (where `<slug>` is the schema slug of the lowered query response schema, produced by the recipe in [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash)). The tool's `parameters` is the lowered query schema wrapped via `Type.Unsafe<unknown>(loweredJsonSchema)` (per the bridging convention in [Pi Integration Contract — Per-theta registration](./pi-integration-contract.md)); its `label` is the literal string `"Theta typed-query response"`; its `execute` AJV-checks the `unknown`-typed `params` against the original lowered JSON Schema and resolves the query's promise with the validated value.
    - Registers the respond tool alongside the theta's callable set for the duration of the typed query. Registration lifetime and visibility are governed by [Pi Integration Contract — Tool-registration lifetime and visibility](./pi-integration-contract.md): in **subagent mode** the respond tool's name is included in the spawned child `pi` process's `--tools` allowlist alongside the frontmatter tools and lives only inside that child process; in **prompt mode** the respond tool is registered once per unique `<slug>` via the registration cache (subsequent typed queries with the same lowered schema reuse the same registration) and made visible for the typed query via the `pi.setActiveTools` snapshot/restore protocol, with `[...frontmatter callable set, respondToolName]` as the active set for the duration of the query. Pi has no `unregisterTool` API; the spec deliberately does not call one.
    - Runs the two-phase loop. In the **free phase** the runtime drives the conversation through `pi.sendUserMessage` (prompt mode) or `session.sendUserMessage` (subagent mode) with **no** `tool_choice` constraint; on each turn, if the model emits `tool_use` blocks the runtime services them (frontmatter tool or respond tool, dispatched by name) and loops. When the model emits a plain text turn, the runtime dispatches the **forced respond turn** off-session through pi-ai's `complete()` free function — the same channel the binder uses, per [Pi Integration Contract — Binder inference call](./pi-integration-contract/binder-inference.md#binder-inference-call) — passing the synthesised respond tool as the single `context.tools` entry and a follow-up user message that inlines the lowered response schema as the trailing `context.messages` entry, and forcing tool choice to the respond tool *for that turn only*. `complete()` is the channel that carries the forced tool choice: it is set via `complete()`'s `options.toolChoice` (`{ type: "tool", name }`), which the per-provider pi-ai adapter maps to the provider-native payload (Anthropic `tool_choice: { type: "tool", name }`; OpenAI Completions `tool_choice: { type: "function", function: { name } }`; Mistral and Bedrock map analogously). The session drivers `pi.sendUserMessage` / `session.sendUserMessage` expose no `options.toolChoice` and are used only for the free phase; the forced respond turn runs off-session through `complete()` and attaches no turn to the driven session. The runtime does **not** install a `before_provider_request` payload-rewrite hook in theta 1.0 — a global hook that rewrites every provider call in every session is the mechanism Option B of the resolved typed-query design rejected; the per-call `options.toolChoice` route on `complete()` is scoped to exactly the forced respond turn and avoids the cross-session-fingerprinting machinery a global hook would require.
    - Enforces the `tool_loop.max_rounds` cap (per [Parameters and Frontmatter — `tool_loop`](./frontmatter.md)) by counting free-phase tool-call rounds only; the forced respond turn that terminates a typed query is the exempt-routed terminator dispatched by CIO-4's `max_rounds`-final branch (see [Hard Runtime Ceilings — CIO-4](./hard-ceilings/ceilings-3-and-4.md#cio-4)) and is not counted in the cap. On the `slot_count == max_rounds` outcome, that branch dispatches the forced respond turn; `tool_loop_exhausted` is unreachable on the typed-query path (the variant remains live for untyped queries per [Errors and Results — `ToolLoopExhaustedError`](./errors-and-results/queryerror-variants.md#queryerror-variants)). Respond-repair follow-ups (per [Query — Schema-validation respond-repair](./query.md)) restart the whole two-phase loop with a fresh `tool_loop` budget.
    - At the user-facing layer the user sees the free-phase tool-call cards (when the model uses frontmatter tools) and the final free-phase plain-text turn rendered normally in the driven session; the forced respond turn runs off-session through `complete()` (above) and is therefore not rendered as a transcript turn, matching the behavioural contract's "no user-visible turns beyond the natural free-phase turns" clause. The validated payload is delivered to theta code; prompt-mode user-visibility of any value the author wants the user to see follows the [success-side null-policy](./pi-integration-contract/runtime-event-channel.md#success-side-null-policy)
- Tool calls from theta code (`<name>(args)`) dispatch through the per-theta callable table that was materialised at load time; resolution is by held reference, not by re-querying Pi's tool registry on every call (see [Parameters and Frontmatter — Resolution snapshot](./frontmatter.md)). For Pi tools the runtime invokes the captured `ToolDefinition`'s `execute(toolCallId, params, signal, onUpdate, ctx)` directly with a synthesized `toolCallId` and a no-op `onUpdate`, and awaits the resulting Promise before returning to theta code (non-blocking at the runtime level, sequential at the language level); for `.theta` callables the runtime spawns a subagent invocation equivalent to `invoke<T>(path, ...)`. Pi tools registered in `tools:` are also wired into every `@`...`` query as model-callable tools; `.theta` callables are lowered to a tool spec (params → input schema, inferred return → output schema, frontmatter `description` → tool description) and exposed alongside Pi tools to the model
- For subagent-mode thetas, the spawned conversation's system prompt is taken from frontmatter `system:` (with `${param}` interpolation resolved at conversation-creation time) and applied to every query against that conversation
- Parameter schemas (frontmatter `params`) are likewise validated by the injected `SchemaValidator` service (see the **Schema validation** seam in [SchemaValidator interface](./pi-integration-contract/host-interfaces-services.md#schemavalidator-interface)) at invocation time
- For slash-command invocation, the runtime first runs the binder (per [Slash-Command Argument Binding](./binder.md)) unless the bypass condition holds. The binder is a one-shot ephemeral call to `bind_model` (resolved at theta-load time from a two-step chain — frontmatter `bind_model:`, then the theta-extension setting `theta.binderModel` per [Settings file reads](./discovery/package-and-settings.md#settings-file-reads); no further fallback). The resolved model must support strict structured-output / strict tool-input; both the resolution and the capability check are load-time gates that surface `theta/load/binder-model-unresolved` and `theta/load/binder-model-not-strict-capable` respectively. The binder call itself runs with `temperature: 0` and a fixed seed where supported. The runtime constructs the binder's response schema dynamically from the theta's `params:` schema (the three-arm envelope `ok | needs_info | ambiguous`), passes it as a strict structured-output contract, and validates the returned `args` (on `kind: "ok"`) with AJV before merging in defaults and starting the theta. Binder failures and non-`ok` envelopes are surfaced as one-line system notes in the user's session; the theta is not started
- **Schema validation.** Schema validation is provided by an injected `SchemaValidator` service whose normative behavioural contract and `interface SchemaValidator` shape are owned by [Pi Integration Contract — `SchemaValidator` interface](./pi-integration-contract/host-interfaces-services.md#schemavalidator-interface). In summary: the validator reports every error in a single pass (no fast-fail), performs no value conversion and no default-filling, resolves `$ref` only within the lowered per-query schema document, silently accepts unknown JSON-Schema `format` keywords, and is deterministic for a given (schema, value) pair; one instance is constructed per runtime instance (never a module-level global) and the file watcher drives its cache-invalidation path. Load-bearing call sites: typed-query response validation and frontmatter `params` validation at invocation time (both above), and the canonical error-ordering map in [Errors and Results — `ValidationIssue` ordering](./errors-and-results/queryerror-variants.md#err-14).
- **Clock.** Wall-clock time and deferred-work scheduling are provided by an injected `Clock` service (see [Pi Integration Contract — `Clock` / `FakeClock` interface](./pi-integration-contract/host-interfaces-services.md#clock--fakeclock-interface) and the runtime's DI seams). The seam carries `now()` (monotonic milliseconds), `wallNow()` (Unix epoch milliseconds), `setTimeout`, and `clearTimeout`; production wires a `WallClock` adapter (with `wallNow()` delegating to `Date.now()`) and tests use a `FakeClock` whose `advance(ms)` deterministically fires due timers in deadline + registration order. One `Clock` instance is constructed per runtime instance — never as a module-level global — and lives for the lifetime of that runtime; parallel runtimes get independent clocks. The runtime MUST NOT call `Date.now`, `performance.now`, `Date.prototype.getTime`, or the global `setTimeout` / `clearTimeout` outside the `WallClock` adapter (a conforming implementation MUST carry a build-time check rejecting direct lexical references to those primitives outside the adapter, per the conformance criterion in [Pi Integration Contract — PIC-12](./pi-integration-contract/host-interfaces-services.md#pic-12); how the check is realised is an implementation choice). Load-bearing call sites: `RuntimeEvent.occurred_at` stamping at the originating emission site (via `wallNow()`; see [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md)), the watcher's 250 ms debounce window (see [Pi Integration Contract — Extension entry point](./pi-integration-contract.md)), the settings-watcher debounce (see [Discovery — Settings file reads](./discovery/package-and-settings.md#settings-file-reads)), and the `theta.scanPackagesTimeoutMs` cap on the package-discovery walk (see [Discovery — Package discovery](./discovery/package-and-settings.md#package-discovery)).
- **Implementation hint (non-normative).** The reference implementation uses AJV v8 (`ajv ^8`, `ajv-formats ^3`), declared in theta's own `dependencies` rather than provided by Pi, with `strict: false`, `allErrors: true`, and `ajv-formats` registered as an instance-level side effect performed by each `SchemaValidator` on its own `Ajv`. These flag values are how AJV happens to satisfy the contract above: `allErrors: true` for one-pass multi-error reporting, the absence of AJV's `coerceTypes` / `useDefaults` options for no-type-conversion / no-defaults, AJV's default in-document `$ref` resolver for the ref-scope rule, and `ajv-formats` for silent format-keyword acceptance. The reference implementation also caches compiled validators per lowered-schema document, keyed by the schema slug of that document plus the AJV major version; the in-process re-parse path defined in [Pi Integration Contract — Extension entry point](./pi-integration-contract.md) drops cache entries belonging to changed files as part of the `ThetaRegistry` swap; the slug-collision verification on a cache hit (byte-comparing the candidate document's canonical-form bytes before serving a cached validator) is normative and owned by [PIC-11](./pi-integration-contract/host-interfaces-services.md#pic-11) per the [schema-slug collision posture](./schema-subset.md#schema-slug-collision-posture), but cache hit/miss policy beyond that verification and the changed-file drop is an optimisation and does not affect the deterministic-validation contract above

This is an implementation-side recipe rather than a normative contract: the comparator-swap escape hatch already promised by those two paragraphs remains genuine, and a future swap to a different SemVer implementation is permitted on the conditions they enumerate. The pinned version range that satisfies theta 1.0 is an implementation choice carried in the theta package's `package.json`; this paragraph names the dependency, not its version. The theta 1.0 runtime additionally depends on the [`chokidar`](https://www.npmjs.com/package/chokidar) npm package as the production adapter behind the `FileWatcher` seam (the **`FakeFileWatcher` / `FileWatcher` interface** block in [Pi Integration Contract](./pi-integration-contract/host-interfaces-services.md#filewatcher-interface), which stays library-agnostic); `chokidar` is declared in the theta package's `dependencies` block on the same footing as `semver`, and this clause names the dependency, not its version. `chokidar` is out of scope for the [inventory-closure audit's *Target surface categories*](./pi-integration-contract/audit-target-categories.md#audit-target-surface-categories) — those categories cover only the canonical `pi` and `ctx` carriers and named imports from the four `@earendil-works/*` peer packages and `typebox`, none of which `chokidar` is — so no `// allow-pi-surface:` exemption marker is required at any `chokidar` import site. The theta 1.0 runtime additionally depends on the [`yaml`](https://www.npmjs.com/package/yaml) npm package as the YAML-parsing mechanism behind the frontmatter parser (see [Parameters and Frontmatter](./frontmatter.md)); `yaml` is declared in the theta package's `dependencies` block on the same footing as `semver` and `chokidar`, ships its own TypeScript bindings (so it needs no `@types/*` `devDependencies` entry), and this clause names the dependency, not its version. Like `chokidar`, `yaml` is neither one of the four `@earendil-works/*` peer packages nor `typebox`, so it too is out of scope for the inventory-closure audit's *Target surface categories* and requires no `// allow-pi-surface:` exemption marker at any `yaml` import site.
- **Per-invocation single-threaded execution.** The theta interpreter runs as plain async TypeScript on Pi's event loop. There are no worker threads and no theta-level concurrency primitive in theta 1.0. Within a single invocation the interpreter is strictly sequential: every `await` (query, tool call, invoke) yields the event loop and the next theta expression cannot run until the awaited Promise resolves, so an invocation can never re-enter itself. The schema-validator cache and the file-watcher are shared across invocations but are read-mostly: compiled validators are immutable post-compile and cache writes only occur on initial compile or watcher-driven invalidation, so concurrent reads from sibling invocations are safe without locking. Sibling invocations can interleave on the event loop — this happens whenever the model emits parallel tool calls into the same `.theta` callable (see [Tool Calls — Concurrency](./tool-calls.md)). The normative cross-invocation isolation contract (subagent-mode private child `pi` process — no shared transcript or `tools:` table) and the theta 1.0 no-cap / no-scheduler concurrency disposition (the `#no-invocation-cap` anchor) are owned by [Pi Integration Contract — Concurrent-invocation isolation and concurrency disposition](./pi-integration-contract/subagent.md#concurrent-invocation-isolation)
- The theta's overall return value is the value of the last expression of its top-level block
