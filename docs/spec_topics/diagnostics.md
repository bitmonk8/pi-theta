# Diagnostics

Loom emits structured diagnostics through two delivery channels, both owned by the loom extension. Pi's own `LoadExtensionsResult.errors` field is **not** used: that field belongs to Pi's extension loader and is only populated while Pi is `import()`-ing the loom extension's entry point. A failure there is a bootstrap failure the user sees on Pi startup, orthogonal to the diagnostics defined here, which all fire after the extension is already live (during scan, watcher reload, or slash-command execution).

**Persistent diagnostics (default).** All `loom/parse/*`, `loom/load/*`, and `loom/runtime/*` diagnostics are delivered via the channel below, with five carved-out exceptions — all emitted via `console.error` from the `session_shutdown` handler because Pi's `ExtensionRuntime.invalidate(...)` may already have invalidated `pi.sendMessage` by the time the handler runs (see [Pi Integration Contract — Extension entry point](./pi-integration-contract.md)): `loom/runtime/reload-teardown-timeout` (the bounded-await timeout in sub-step 3), `loom/host/session-shutdown-reason-unknown` (the unknown-reason validation defined in PIC's **Unknown-reason rule**), `loom/host/session-shutdown-pinned-constant-unreadable` (the handler-entry pinned-constant lookup failure defined in PIC's **Unknown-reason rule**), `loom/host/session-shutdown-teardown-step-failed` (the per-step isolation rule defined in PIC's **Per-step isolation** paragraph), and `loom/host/session-shutdown-runtime-degraded` (the post-sub-step-4 degraded-state notification defined in [PIC's `reason: "new" | "resume" | "fork"` edge-case bullet](./pi-integration-contract.md#session-only-reason-degraded-state)). The structured `Diagnostic` shape below still applies; only the delivery sink differs. Each of these five teardown-handler `console.error` calls is itself wrapped per [Pi Integration Contract — Diagnostic-emission isolation](./pi-integration-contract.md#diagnostic-emission-isolation): a throw from `console.error` (closed stdio, file-descriptor exhaustion, a host that proxies `console` and rejects writes during teardown) MUST be swallowed so a failing last-resort sink does not unwind the teardown handler. The "no ceiling fails silently" closure relies on this defended sink rather than on an implicit assumption that `console.error` always succeeds.

```
pi.sendMessage(
  {
    customType: "loom-system-note",
    content:    <serialised batch>,            // see "Serialised content format" below
    display:    true,
    details:    { diagnostics: <Diagnostic[]> } // single-element array for runtime/single-error cases
  },
  { triggerTurn: false }
)
```


The `pi.registerMessageRenderer("loom-system-note", …)` registered by the loom extension formats these into transcript-persistent, dim-styled notes; see [Pi Integration Contract — System notes](./pi-integration-contract.md) for the renderer registration rules and the best-effort fallback (`ctx.ui.notify` then `loom/runtime/system-note-delivery-failed` then `console.error`) that applies when `pi.sendMessage` itself throws (`pi.sendMessage` returns `void`; only synchronous throws are observable). The renderer MUST be registered synchronously inside the extension factory **before** the first discovery scan kicks off, so the first batch of scan diagnostics renders through the loom-specific renderer rather than as raw fallback text. This is the only diagnostic sink for loom-author-facing parse, type, load, and runtime errors.

The `loom-system-note` channel also carries operator-facing runtime failure events (`details: { event: RuntimeEvent }`) for non-panic `QueryError` failures in the always-log set; the two `details` payload shapes are disjoint by key and are specified under "System notes" and "Runtime event channel" in [Pi Integration Contract](./pi-integration-contract.md). Renderers MUST switch on which key is present and MUST NOT assume both.

**Transient toasts (auxiliary).** Failures internal to the loom extension's own bookkeeping that the user must see immediately but that do **not** belong in the transcript — the chokidar watcher itself throwing, an unrecoverable settings I/O exception that no `loom/load/settings-*` code covers, an internal extension invariant violation — use `ctx.ui.notify(message, "error")` directly. This is a narrow secondary surface; loom-author-facing diagnostics (anything with a `loom/parse/*`, `loom/load/*`, or `loom/runtime/*` code) MUST go through the persistent channel above and MUST NOT be routed through `ctx.ui.notify` as their primary sink.

**Re-scan deduplication.** A watcher-triggered reload re-emits the persistent diagnostic for any file whose contents are still broken. The runtime does **not** attempt to clear or supersede prior `loom-system-note` messages from the transcript — Pi's `pi.sendMessage` API has no documented "remove" or "replace by metadata" counterpart, and `display: true` system notes are part of the immutable session log. Authors will therefore see the same diagnostic line recur after each reload until the underlying file is fixed; this is the loom 1.0 contract and the renderer MUST NOT attempt to suppress duplicates.

Internal diagnostic shape:

```
{
  severity: "error" | "warning",
  code:     string,                          // e.g., "loom/parse/binding-case-mismatch"
  file:     string,                          // absolute path
  range:    { start: { line, column }, end: { line, column } },  // 1-indexed; end exclusive
  message:  string,                          // single-line summary
  hint?:    string,                          // optional suggested fix
  related?: array<{ file, range, message }>, // related sites (e.g., the colliding declaration)
  masked?:  array<string>,                   // hard-ceiling co-fire enumeration; see below
  details?: object,                          // optional per-row structured payload; shape pinned by each row's Trigger prose — see note below
}
```

The optional `details?` field carries a diagnostic's per-row structured wire payload; its shape is pinned normatively by that row's *Trigger* prose in the [Code registry](#code-registry) below, and the `details` shape conventions paragraph immediately below governs only the `session_shutdown` event-payload subset (the `details.event.{...}` nesting on `loom/runtime/cancelled-by-session-shutdown` and `loom/host/session-shutdown-runtime-degraded`). Every row-documented `details.{...}` payload lives on the `Diagnostic` object — i.e. inside `CustomMessage.details.diagnostics[i].details` — not on the outer `CustomMessage.details`, which stays closed to `{ diagnostics: Diagnostic[] }` plus the runtime-event-channel carve-out (`details: { event: RuntimeEvent }`) noted above. A rule-4-compliant test asserting on a row's payload therefore inspects `diagnostics[i].details`.

<a id="session-shutdown-details-conventions"></a> **`details` shape conventions for the `session_shutdown` event payload.** Diagnostics that surface the Pi `session_shutdown` event payload — currently the per-invocation `loom/runtime/cancelled-by-session-shutdown` note and the teardown-handler `loom/host/session-shutdown-runtime-degraded` row, both registered below — nest the payload under `details.event.{...}` so a single nested shape covers both codes for log-pipeline and operator-tooling consumers. **Closed at `{reason}` in loom 1.0.0.** The only field pinned by this convention in loom 1.0.0 is `details.event.reason`; the convention does **not** authorise additional `SessionShutdownEvent` sibling fields under the `details.event.` prefix between SDK-pin bumps, and a future Pi SDK addition that the runtime decides to surface (e.g. a hypothetical `details.event.timestamp`) requires a spec edit to this paragraph and to the affected row(s) below before it is permitted on the wire. Operator-tooling consumers parsing across `loom/runtime/cancelled-by-session-shutdown` and `loom/host/session-shutdown-runtime-degraded` are therefore entitled to validate `details.event` strictly against the closed `{reason}` shape between SDK-pin bumps. *Per-code `details.event.reason` value spaces.* The `{reason}` shape closure above pins the *set of fields* but does **not** pin a uniform value space for `details.event.reason` itself: the two codes' value spaces diverge and consumers MUST validate the field against the per-code value space, not against a uniform value-space pin. `loom/runtime/cancelled-by-session-shutdown` admits the wider union `"quit" | "reload" | "new" | "resume" | "fork" | string` — the closed five-arm union plus a `string` arm reserved for the unknown-reason path that carries the verbatim observed value coerced via `String(event.reason)`, plus the literal `"<unreadable>"` sentinel for the throwing-property-access and snapshot-failure paths — per the captured-value enumeration owned by [PIC's **Unknown-reason rule**](./pi-integration-contract.md#unknown-reason-rule) and restated in that row's description below. `loom/host/session-shutdown-runtime-degraded` is closed to `{"new", "resume", "fork"}` per the diagnostic-emission predicate `pinnedConstantReadOk && capturedEventReason ∈ {"new", "resume", "fork"}` owned by [PIC's `reason: "new" | "resume" | "fork"` edge-case bullet](./pi-integration-contract.md#session-only-reason-degraded-state) and restated in that row's description below — the unknown-reason and snapshot-failure paths are excluded by that predicate, and the `"quit"` and `"reload"` arms are excluded because the runtime is being torn down on those reasons. The closed-set entitlement above applies to **structured emissions only**: on the construction-site fallback path the per-row wire token shape is pinned by the *Construction-site wrap* sub-clause of [Pi Integration Contract — Diagnostic-emission isolation](./pi-integration-contract.md#diagnostic-emission-isolation) (which the *Runtime construction obligation* paragraph below cross-references for the `"<unreadable>"` sentinel substitution and is the source of truth for the per-row token shapes), and the per-row shape differs across the two nested-shape codes: for `loom/host/session-shutdown-runtime-degraded` the runtime emits the unstructured two-token bare-`` `${code} <unreadable>` `` form carrying the literal `"<unreadable>"` sentinel for `details.event.reason` (the only payload field this row carries under `details.event`); for the per-invocation `loom/runtime/cancelled-by-session-shutdown` note the runtime emits the unstructured three-token form `` `${code} ${entry.loom} <unreadable>` `` (the catch arm reads `entry.loom` separately from the [`ActiveInvocationRegistry`](./pi-integration-contract.md#active-invocation-registry) entry the per-invocation `finally` already holds, preserving per-invocation `loom` discrimination via the interpolated `entry.loom` token while substituting the `"<unreadable>"` literal for `details.event.reason` only). Consumers MUST treat the construction-site fallback `details.event.reason` value on either row as a separate value-space outside the closed `{"new", "resume", "fork"}` set, mirroring the `"<unreadable>"` enumeration carved out for `loom/runtime/cancelled-by-session-shutdown` above. A consumer that learns the value space from one code and applies it to the other will under- or over-validate; the per-code value space is the normative contract. *Runtime-constructed sibling carve-out.* The `details.event.loom` field on `loom/runtime/cancelled-by-session-shutdown` (the slash-name of the cancelled invocation, registered under that row below) is a **runtime-constructed sibling** under the `details.event.` prefix — not an SDK-derived `SessionShutdownEvent` field — and is exempt from the strictness entitlement above; the `{reason}` closed-shape pin governs only SDK-derived siblings (i.e., fields whose source is the `SessionShutdownEvent` payload Pi delivers), and operator-tooling consumers MUST whitelist `details.event.loom` on the `loom/runtime/cancelled-by-session-shutdown` code when applying the strict-validation entitlement. The runtime MUST include `details.event.loom` on every emission of `loom/runtime/cancelled-by-session-shutdown`, sourced from the `loom` field of the cancelled invocation's [`ActiveInvocationRegistry`](./pi-integration-contract.md#active-invocation-registry) entry — registered at invocation start per that registry's contract, in its canonical form (post-`pi.looms` mapping, post-`as` rename, basename-hyphen-normalised) with no leading `/`; the field is therefore a normative part of that row's wire contract and tests are entitled to assert its presence. The `details.event.invocation_id` field on the same row is similarly a **runtime-constructed sibling** under the `details.event.` prefix — not an SDK-derived `SessionShutdownEvent` field — and is exempt from the strictness entitlement above on the same footing as `details.event.loom`. The runtime MUST include `details.event.invocation_id` on every emission of `loom/runtime/cancelled-by-session-shutdown`, sourced from the `invocationId` field of the cancelled invocation's [`ActiveInvocationRegistry`](./pi-integration-contract.md#active-invocation-registry) entry (registered at invocation start per that registry's contract, in canonical lowercase 8-4-4-4-12 hex form per the [§7 `<invocation-id>` placeholder convention](#7-identifier--descriptor--and-closed-enum-placeholders)); the field is therefore a normative part of that row's wire contract on the structured-emission path and tests are entitled to assert its presence. *Presence-entitlement scope — structured emissions only.* The presence entitlement above applies to **structured emissions only**: on the serialiser-throw fallback path the wire form is the unstructured two-token sequence `` `${code} ${detailsEventReason}` `` per the *Two-token fallback for nested-shape codes* sub-clause of [Pi Integration Contract — Diagnostic-emission isolation](./pi-integration-contract.md#diagnostic-emission-isolation) (which carries no `loom` slot at all), and on the construction-site-throw fallback path the wire form is the unstructured three-token sequence `` `${code} ${entry.loom} <unreadable>` `` per that paragraph's *Construction-site wrap* sub-clause (which is the source of truth for the per-row token shapes and which, for the per-invocation note, preserves per-invocation `loom` discrimination on the wire by interpolating the `entry.loom` token read separately from the [`ActiveInvocationRegistry`](./pi-integration-contract.md#active-invocation-registry) entry alongside the `"<unreadable>"` sentinel substituted for `details.event.reason` only; the `runtime-degraded` row's *Construction-site wrap* shape is unchanged at the two-token form `` `${code} <unreadable>` `` since that row carries no `loom` field under `details.event` to preserve). Consumers MUST treat fallback-path emissions as a separate wire shape from the structured emission and MUST NOT validate them against the structured presence-entitlement above; a test that asserts `details.event.loom` presence without first discriminating the structured-emission path from the two fallback paths over-fails on every multi-invocation shutdown whose per-invocation notes route through either fallback. The same scope applies to `details.event.invocation_id` on the per-invocation note's structured emission per the *Runtime-constructed sibling carve-out* clause above: neither fallback path carries a separable `details.event.invocation_id` slot (the *Construction-site wrap* sub-clause's three-token shape preserves only `entry.loom`, on the same footing as the existing carve-out), and a test asserting its presence MUST first discriminate the structured-emission path on the same footing as the `details.event.loom` discrimination rule above. Future runtime-constructed sibling fields under `details.event.*` similarly require a spec edit to this paragraph and to the affected row(s) below before they are permitted on the wire (on the same footing as the SDK-derived future-addition obligation above), but are not blocked by this convention's strictness pin once enumerated. *Runtime construction obligation.* The runtime MUST construct `details.event` as a fresh object containing exactly the pinned fields per the per-code field set enumerated above (no `...event` spread, no `Object.assign(..., event, ...)`, no per-key copy loop over `Object.keys(event)`, and no other dynamic-field-set mechanism whose member set is sourced from the SDK-delivered `SessionShutdownEvent` rather than from this paragraph) — for `loom/host/session-shutdown-runtime-degraded` the constructed object is `{ reason }` and for `loom/runtime/cancelled-by-session-shutdown` it is `{ reason, loom, invocation_id }`, with each pinned field assigned by an explicit static property name. This makes the operator-tooling consumer's strict-validation entitlement above deserved by construction rather than by implicit coincidence with the V1 `SessionShutdownEvent` member set, so a future SDK addition (e.g. a hypothetical `event.timestamp`) does not silently leak onto the wire under a future contributor's spread-style refactor; future SDK-derived or runtime-constructed fields require a spec edit to this paragraph and to the affected row(s) below before they may be added to the constructed object. Per-code runtime-constructed failure-mode payloads — `details.observed` for `loom/host/session-shutdown-reason-unknown`, `details.failure` for `loom/host/session-shutdown-pinned-constant-unreadable`, `details.step` / `.call` / `.error` for `loom/host/session-shutdown-teardown-step-failed` — remain flat under `details` because they are not the SDK event payload. The teardown-handler emission's `try`/`catch` envelope and the bare-`code` fallback on serialiser throw are owned by [Pi Integration Contract — Diagnostic-emission isolation](./pi-integration-contract.md#diagnostic-emission-isolation); this paragraph owns only the wire-format nesting convention. The two-token bare-`` `${code} ${details.event.reason}` `` fallback for these two codes on serialiser throw is owned by [Pi Integration Contract — Diagnostic-emission isolation](./pi-integration-contract.md#diagnostic-emission-isolation). The construction-site `try`/`catch` that defends *this* paragraph's runtime construction of `details.event` (and the PIC *Hoist obligation* clause's pre-wrap construction of the `detailsEventReason` local) against a construction-time throw — routing the catch arm through the bare-`code` or two-token fallback above with the literal `"<unreadable>"` sentinel substituted for both `details.event.reason` and (on the per-invocation note) `details.event.loom` — is owned by the *Construction-site wrap* sub-clause of [Pi Integration Contract — Diagnostic-emission isolation](./pi-integration-contract.md#diagnostic-emission-isolation), which also pins the `"<unreadable>"` literal as the single source of truth for catch-arm payload defaults so the construction-site failure path's value space stays inside the captured-value enumeration the PIC **Unknown-reason rule** pins for `details.event.reason`.

**Code namespaces:**

- `loom/parse/*` — diagnostics produced by the parse pipeline: lexical errors, syntactic errors, and static type-system checks. The `Phase` column in the registry table distinguishes `lex` / `parse` / `type` within this namespace.
- `loom/load/*` — file-load and registration errors (unreadable file, missing or wrong-type discovery source, name collision, invalid frontmatter, unresolvable `tools:` entry).
- `loom/runtime/*` — runtime-phase events: execution panics (`MatchError`, index out of bounds, etc.), the runtime-defect surface (`loom/runtime/internal-error`), and delivery / lifecycle / teardown failures. Routing varies per code — see the [`loom/runtime/*` registry section](#loom-runtime-namespace) and [Errors and Results — Runtime panics](./errors-and-results.md) for per-code channels (most route through `loom-system-note`; `loom/runtime/reload-teardown-timeout` routes through `console.error`).
- `loom/host/*` — anomalies that surface during loom's interaction with the Pi host. The namespace covers both anomalies the loom runtime observes coming *from* the Pi host that the spec's host-prerequisites contract does not enumerate (e.g., a `session_shutdown` `event.reason` outside the closed five-arm union, or a throw from a `session_shutdown` teardown sub-step) and loom-internal failures of host-related machinery — pinned-constant blocks that snapshot host SDK types, host-event handlers, and similar — whose immediate cause is loom-internal corruption but whose observation site is a host-driven code path (e.g., a corrupted snapshot of `SessionShutdownEvent['reason']` failing the `session_shutdown` handler's entry-time pinned-constant lookup; the row's *Trigger* describes the loom-internal corruption while its emission site is the host event handler). Emitted via `console.error` since the typical observation site is mid-teardown when the persistent-diagnostic channel may already be invalidated.

<a id="loom-typecheck-namespace"></a> **Out of scope: `loom/typecheck/*`.** The `loom/typecheck/*` namespace is a parallel surface reserved for build-time `tsc` brand strings (e.g. `loom/typecheck/session-shutdown-reason-snapshot`, embedded in the build-time bidirectional type-equality assertion per [Pi Integration Contract — Pi version bump procedure](./pi-integration-contract.md#pi-version-bump-procedure) step 5 so a typecheck failure surfaces the brand verbatim in the failing `tsc` diagnostic). Brands in this namespace share the `loom/...` prefix but are **not** runtime diagnostics — they carry no registry row, do not flow through any of the delivery channels above, and are out of scope for the Code registry rules below. A reader auditing the `loom/...` surface should treat `loom/typecheck/*` matches as build-time brands owned by PIC step 5 rather than as missing registry entries. *Naming convention.* Each brand has the shape `loom/typecheck/<kebab-case-name>` — the rule-name component is kebab-case (lowercase ASCII letters, digits, and `-`) and MUST NOT contain `<...>`-style placeholders, since brand strings are TypeScript string-literal types compared verbatim by `tsc` and a placeholder would break the verbatim-surfacing property step 5 keys on. *Definition site.* Each `loom/typecheck/*` brand MUST be defined inline at the spec-prose site that introduces its build-time assertion (the brand is named where the assertion is pinned); the namespace is intentionally distributed and there is **no** central registry of `loom/typecheck/*` brands analogous to the Code registry below. *Audit-gate carve-out.* Audit gates that scan source or spec for `loom/...` literals and reconcile them against the Code registry below MUST carve out `loom/typecheck/` as a prefix-disjoint namespace — a `loom/typecheck/*` literal MUST NOT be reported as a missing registry entry.

**Serialised content format.** This rule fixes the **input** to the renderer (the `content` field on the `CustomMessage` argument), not the renderer's return value. The renderer itself returns a `pi-tui` `Component | undefined`, per the **Renderer registration** block in [Pi Integration Contract — System notes](./pi-integration-contract.md). The `content` string of each `loom-system-note` follows the line format `"<file>:<line>:<col>: <code>: <message>"`, optionally followed by `"\n  hint: <hint>"` when a hint is present. Related sites are appended as additional indented lines. When a single message carries a multi-error batch (the scan-time case), each `Diagnostic` becomes one such line block and successive blocks are separated by a single blank line; the corresponding structured `Diagnostic[]` is carried in `details.diagnostics` for the renderer and for downstream consumers (LSP integrations, test harnesses) that want the typed shape rather than the rendered string.

**Multi-error reporting.** Every parse / type pass collects all errors from the full file (and from transitive `.warp` imports) before failing. The loom is rejected with the complete list in **one `pi.sendMessage` call per `.loom` file** — `content` carries the full batch in the format above; `details.diagnostics` carries the same as a structured array — rather than fast-failing on the first error or fanning out one message per error. Authors get every problem in the file at once, in a single transcript entry.

## Code registry rules (normative)

Three rules govern the diagnostic-code surface:

1. **Every author-visible diagnostic emitted by the runtime MUST carry a code from the registry below.** Emitting an unregistered code is a defect; tests are entitled to assert on the specific code at every documented diagnostic site.
2. **The registry is closed.** Adding a new code, removing a code, or changing a code's namespace, severity, or trigger are all spec changes — not implementation changes. New diagnostic sites added by future spec work MUST land their codes in this table at the same time.
3. **Codes are stable identifiers.** Renaming a registered code is a breaking change to the public diagnostics contract (tests, LSP integrations, system-note formatters, doc cross-links). Treat each code as part of the language surface.
4. **The Message column is normative.** Every row in the registry below carries a *Message* (the rendered author-facing string with `<…>`-style placeholders for interpolated content); renderers MUST emit it character-for-character with placeholders interpolated. Tests asserting a diagnostic's rendered message MUST source the string from this column rather than copy-pasting prose from the spec rule's home page. Wording changes are spec-versioned breaking changes.

Naming convention for codes is `<namespace>/<kebab-case-rule-name>`. The rule-name component derives from the spec rule's short name; it is not generated, so the registry below is the source of truth.

<a id="masked-field-hard-ceiling-co-fire"></a>

**`masked` field (hard-ceiling co-fire).** The optional `masked` field on the `Diagnostic` shape above carries the hard-ceiling co-fire enumeration defined in [Hard Runtime Ceilings — `masked` field](./hard-ceilings.md#masked-field) and [Hard Runtime Ceilings — Interaction between ceilings](./hard-ceilings.md#ceiling-interaction-order). It is populated only on diagnostics emitted from a hard-ceiling check site whose precondition fired while at least one *sibling* hard ceiling's precondition was simultaneously satisfied at the same site — in V1 this is the `loom/runtime/invoke-depth-exceeded` row below (the only diagnostic-shape hard-ceiling surface; the other co-fire-eligible surfaces are runtime events, not diagnostics, and carry the same enumeration as `event.masked` per [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md)). Each entry is one of `"ceiling#1"`, `"ceiling#2"`, `"ceiling#3"`, `"ceiling#4"`. The field is **omitted** when no co-fire occurred; implementations MUST NOT emit `masked: []`. The per-site reachable mask domain (in V1: empty for `loom/runtime/invoke-depth-exceeded`, so this diagnostic always omits `masked`), the V1 reachable predicate, and the pure-read MUST on the masking probe are owned by [Pi Integration Contract — PIC-1 *Hard-ceiling co-fire (`masked`)*](./pi-integration-contract.md#pic-1).

## Placeholder rendering (normative)

The registry's *Message* column carries `<…>` placeholders that the renderer interpolates at the diagnostic site (rule 4 above). This subsection groups every placeholder used by V1 messages into eight categories and fixes one rendering rule per category, so two conformant implementations produce byte-identical strings (or byte-identical surround around an implementation-defined-tail interpolation, for category 8) for the same source defect. Each rule below carries normative test vectors; conformance tests asserting on a rendered message MUST match these vectors and SHOULD include the boundary cases called out in *Edge cases* at the end of this subsection. The closed token-name table in category 3 and the closed value tables in category 7 carry the same GOV-7 / GOV-8 governance posture as the category-to-placeholder map itself.

**Closure.** The eight categories below form the closed loom 1.0.0 placeholder-rendering surface. Every `<…>` token appearing in the *Message* column of any row in [Code registry](#code-registry) is either (a) a placeholder enumerated in categories 1–7 (rendered byte-identically per its category's rule), (b) a §8 placeholder (rendered as a byte-identical prefix and suffix around an implementation-defined-tail interpolation per §8's rule), (c) the bespoke `<list>` placeholder in `loom/runtime/reload-teardown-timeout` (whose row carries an inline rendering rule that decomposes into the §7 sub-rules for `<slash-name>`, `<invocation-id>`, and the comma-space join), or (d) the `<failure>` carve-out documented in §8 below (a loom-internally-constructed discriminator whose grammar is pinned at the diagnostic-construction site, not by §8's host-derived rule). No other placeholders are admitted; this closure is enforced at build time. Introducing a new placeholder, retiring one, or moving a placeholder between categories is a spec-versioned breaking change governed by **GOV-7** and **GOV-8** in [`spec.md` — Governance](../spec.md), exactly as the registry rows themselves are.

### 1. Static-type placeholders

**Placeholders.** `<type>`, `<expected>`, `<actual>`, `<left>`, `<right>`, `<element>`.

**Rule.** Render the Loom static type by re-serialising it in the source-grammar form defined in [Type System](./type-system.md):

- Primitive type names lowercase: `string`, `integer`, `number`, `boolean`, `null`.
- Literal types as their literal source: `"foo"`, `42`, `true`.
- Unions joined by ` | ` (space-pipe-space) with no surrounding parentheses.
- Arrays as `array<T>` (the angle-bracket form; never `T[]`, never `[T]`).
- Named schemas, enums, and type aliases by their loom-side identifier (no wire-name translation; the identifier shape is fixed by [Lexical — Identifiers](./lexical.md)).
- `Result<T, E>` rendered as written, with the inner types recursing this rule.
- Inline anonymous object types as `{ f₁: T₁, f₂: T₂ }`, fields in declaration order, single space after each `:` and after each `,`.

**Test vectors.**

- A binding typed `array<integer | string>` renders as `array<integer | string>`.
- A binding typed `Foo | null` renders as `Foo | null`.

### 2. Runtime-value placeholders

**Placeholders.** `<scrutinee summary>`, `<value>`.

**Rule.** Render via the canonical interpolation-stringification table in [Query — Stringification of interpolated values](./query.md), with one extension and one supplementary case:

- **String truncation.** A string whose length exceeds 80 Unicode code points is truncated to the first 77 code points followed by the literal three-character ellipsis `...`. The 80-code-point cap, the 77-code-point prefix, and the literal ellipsis are normative; counting is by Unicode code point, not by UTF-16 code unit and not by grapheme cluster.
- **`Result<T, E>` values.** A scrutinee whose static type is `Result<T, E>` (the case the query-stringification table reserves for a static parse error) renders as `Ok(<inner>)` or `Err(<inner>)`, with `<inner>` recursing this rule. Panics may legitimately fire on `Result` values (e.g. a `match` whose arms collectively miss an `Err` variant), so this case is reachable here even though it is unreachable in interpolated query templates.

**Test vectors.**

- A `match` panic on a schema-typed-object scrutinee whose runtime value is `{ name: "fluffy" }` renders `MatchError: no arm matched {"name":"fluffy"}` — the *Schema-typed object* row of the stringification table renders it as compact `JSON.stringify` with wire-name translation, and the schema name does not surface in the rendered string.
- A `match` panic on the integer `42` renders `MatchError: no arm matched 42`.
- A `match` panic on a 100-character ASCII string `s` renders `MatchError: no arm matched ` followed by the first 77 code points of `s` followed by the literal `...` (a single trailing three-character ellipsis, no surrounding quotes — the `string` row of the stringification table renders strings verbatim without quoting).

### 3. Syntactic-construct placeholder

**Placeholders.** `<construct>` in `loom/parse/unsupported-feature`; `<expr>` in `loom/parse/default-not-literal` and `loom/parse/tool-arg-not-literal`.

**Rule.**

- For `<expr>` in the two literal-sublanguage codes, render the offending source span verbatim, copied byte-for-byte from the source file between the offending sub-expression's start and end token positions (post-newline-normalisation per [Lexical — Encoding](./lexical.md)), with internal whitespace preserved.
- For `<construct>` in `loom/parse/unsupported-feature`, the offending site is a whole node category with no single source-span anchor (e.g. `=>` lambdas span the entire arrow form, including the body). Use the closed token-name table below.

| Construct | Token name |
|---|---|
| arrow function (`=>`) | `arrow function` |
| spread / rest (`...`) | `spread` |
| optional chaining (`?.`) | `optional chaining` |
| nullish coalescing (`??`) | `nullish coalescing` |
| strict equality (`===` / `!==`) | `strict equality` |
| bitwise op (`&`, `\|`, `^`, `~`, `<<`, `>>`, `>>>`) | `bitwise <op>` (where `<op>` is the source token verbatim) |
| comma operator (expression-position `,`) | `comma operator` |
| nested template literal | `nested template` |
| `new` expression | `new` |
| `typeof` operator | `typeof` |
| `instanceof` operator | `instanceof` |
| `delete` operator | `delete` |
| `void` operator | `void` |
| `yield` expression | `yield` |
| `await` expression | `await` |

**Test vectors.**

- A loom containing `let f = (x) => x + 1` renders `unsupported syntactic feature: arrow function`.
- A `params:` default whose RHS is `a + b` renders `params default RHS must be a literal-sublanguage form; offending sub-expression: a + b`.

### 4. Numeric placeholders

**Placeholders.** `<i>`, `<length>`, `<depth>`, `<offset>`, `<count>`, `<index>`, `<required>`, `<provided>`, `<max>`.

**Rule.** Render as the shortest decimal representation per the `integer` row of the canonical stringification table in [Query — Stringification of interpolated values](./query.md): no scientific notation, no leading zeros, leading `-` for negatives, `0` for the value `-0` (signed zero is normalised at the rendering boundary). `Infinity` and `NaN` are unreachable for these placeholders by construction (every emitting site is bounded — array length is non-negative, invoke depth is bounded by 32, etc.); a renderer that nonetheless encounters one MUST surface it through `loom/runtime/internal-error` rather than emitting `Infinity` or `NaN` into a panic message.

**Test vectors.**

- A negative-index OOB on a 3-element array renders `index out of bounds: -1 not in 0..3`.
- A 33-deep `invoke` chain renders `invoke chain depth exceeded: 33 > 32`.

### 5. Source-derived placeholders

**Placeholders.** `<path>`, `<file>`, `<descriptor>`, `<name>`, `<field>`, `<param>`, `<variant>`, `<keyword>`, `<key>`, `<char>`.

**Rule.** Render verbatim from the source as the offending text would appear there:

- **`<path>`, `<file>`** — the literal text inside the path-literal quotes, with no realpath normalisation, no symlink resolution, no scheme prefixing. This generalises the convention already pinned at `loom/parse/invoke-non-loom-extension`.
- **`<name>`, `<field>`, `<param>`, `<variant>`, `<keyword>`** — identifier-shaped per [Lexical — Identifiers](./lexical.md); rendered unquoted. Any quoting in the rendered message comes from the surrounding registry template (e.g. the literal `'<name>'` in `cannot reassign immutable binding '<name>'`), not from the placeholder.
- **`<key>`** — quoted with double quotes only when the key string is *not* identifier-shaped per [Lexical — Identifiers](./lexical.md) (i.e. would not match `[A-Za-z_][A-Za-z0-9_]*`); otherwise rendered bare. The identifier-shape predicate is a runtime check on the key string, not a parse-time grammar production.
- **`<char>`** — the raw character itself when printable (any code point outside the Unicode general categories `Cc` and `Cn`); otherwise the standard `\xNN` (for code points ≤ U+00FF) or `\u{NNNN}` escape per the escape forms accepted in [Lexical — String literals](./lexical.md).
- **`<descriptor>`** — the discovery-source descriptor as a `kind:value` pair, rendered as `<kind>:"<value>"` (the kind unquoted, the value double-quoted). Kinds are drawn from the closed set defined in [Discovery](./discovery.md) (`settings`, `cli-flag`, `package`); the value is the descriptor's source text verbatim (the settings entry, the CLI flag string, the npm package name). The descriptor format is normative.

**Test vectors.**

- An unknown frontmatter field `wibble` renders `unknown field 'wibble'` (the registry template supplies the surrounding single quotes; the placeholder itself contributes the bare identifier).
- A non-existent settings-sourced loom directory `~/work/looms` renders `discovery source path does not exist: settings:"~/work/looms"`.
- A `match` on `obj["my-key"]` against a missing key renders `missing object key: "my-key"` (key is not identifier-shaped, so quoted).
- A member access `obj["kind"]` on a missing key renders `missing object key: kind` (key is identifier-shaped, so bare).

### 6. Underlying-error placeholders

**Placeholders.** `<error.message>`, `<original content first line>`, `<dispose error first line>`.

**Rule.** Render as the underlying string truncated to its first line:

1. Newline-normalise the underlying string per [Lexical — Encoding](./lexical.md) (`\r\n` and bare `\r` collapse to `\n`).
2. Take the prefix up to (but not including) the first `\n`.
3. Preserve trailing whitespace on the resulting line — no rstrip.
4. When the underlying string is empty, or contains nothing before its first `\n`, render the literal text `<no message>`.

Only `\n` (post-normalisation) is a line break for this rule. The Unicode line separators `\u2028` and `\u2029` are *not* recognised by the lexical newline rule and are *not* recognised here either: a host throwing an `Error` whose message embeds `\u2028` produces a single first-line cut at the first `\n` (or at end-of-string if there is none), with `\u2028` rendered as an ordinary character.

**Test vectors.**

- A `pi.sendMessage` synchronous throw whose underlying error message is `file:5:3: loom/parse/binding-case-mismatch: ...` renders `system-note delivery failed: file:5:3: loom/parse/binding-case-mismatch: ...`.
- A `pi.sendMessage` synchronous throw whose underlying error was constructed without a message (`new Error()`) renders `system-note delivery failed: <no message>`.
- An `AgentSession.dispose()` rejection whose error message is `connection closed\nstack trace ...\n` renders the prefix up to the first `\n` — the literal text `connection closed` — preserving any trailing space on that line.

### 7. Identifier-, descriptor-, and closed-enum placeholders

**Placeholders.** `<callee>`, `<enum>`, `<schema>`, `<method>`, `<model>`, `<provider>`, `<slash-name>`, `<slug>`, `<invocation-id>`, `<uuid>`, `<name1>`, `<name2>`, `<X>`, `<A>`, `<B>`, `<root>`, `<paths>`, `<path-a>`, `<path-b>`, `<source>`, `<higher>`, `<lower>`, `<fields>`, `<ms>`, `<deadline>`, `<cap>`, `<capability>`, `<dotted-key>`, `<reason>`, `<kind>`, `<op>`, `<value>` (in the parse-time literal-value position; the runtime-value usage of `<value>` belongs to category 2 and is distinguished by emitting site).

**Rule.** Each placeholder renders by the closed sub-rule below; sub-rules compose with categories 1–6.

- **Identifier-shaped** (`<callee>`, `<enum>`, `<schema>`, `<method>`, `<model>`, `<provider>`, `<slash-name>`, `<slug>`, `<invocation-id>`, `<uuid>`, `<name1>`, `<name2>`, `<X>`) — rendered unquoted in identifier form per [Lexical — Identifiers](./lexical.md). Any surrounding quoting comes from the registry template, not from the placeholder. The leading `/` of `<slash-name>` is contributed by the surrounding template (e.g. `/<slash-name>`), not by the placeholder. `<uuid>` and `<invocation-id>` render in canonical lowercase 8-4-4-4-12 hex form. `<callee>` renders as the registered slash name without leading `/` when one exists; otherwise it falls through to category 5's `<path>` rule (verbatim path-literal source text).
- **Path-shaped** (`<root>`, `<paths>`, `<path-a>`, `<path-b>`, `<source>`, `<A>`, `<B>` — the latter two as the file-path stems used in `loom/load/import-cycle` and `loom/load/invocation-cycle`) — rendered via category 5's `<path>` rule (the literal text inside the path-literal quotes, with no realpath normalisation, no symlink resolution, no scheme prefixing). List-valued forms (`<paths>`) are joined with `, ` (comma-space) in declaration order; for `loom/load/cross-format-collision` the order is: registered candidates first, then the slash-name-deriving `.md` siblings, both internally ordered by discovery-source priority then by absolute path.
- **Descriptor-shaped** (`<higher>`, `<lower>`) — rendered via category 5's `<descriptor>` rule (`<kind>:"<value>"`).
- **Joined-identifier list** (`<fields>`) — identifier-shaped names joined with `, ` (comma-space) in declaration order.
- **Numeric** (`<ms>`, `<deadline>`) — rendered via category 4's integer rule.
- **Closed-enum** — each placeholder draws from a closed value table:
  - `<cap>` ∈ {`looms.scanPackagesMaxFiles`, `looms.scanPackagesTimeoutMs`} (the two scan-package caps in [Discovery — Package discovery](./discovery.md)).
  - `<capability>` ∈ {`pi.registerMessageRenderer`, `pi.registerCommand`, `pi.registerFlag`} (per [Pi Integration Contract — Extension-bootstrap SDK failures](./pi-integration-contract.md)).
  - `<dotted-key>` ∈ {`tool_loop.max_rounds`, `respond_repair.attempts`} (the two non-negative-integer frontmatter scalars in [Frontmatter — Field contract](./frontmatter.md); other dotted keys join the closed set only as new frontmatter scalars are added under GOV-7).
  - `<reason>` ∈ {`quit`, `reload`, `new`, `resume`, `fork`} for the closed-arm path; for the unknown-reason path (per PIC's **Unknown-reason rule**), the verbatim observed value coerced to string via `String(event.reason)`, or the literal `<unreadable>` when the property access threw. The literal `<unreadable>` rendering is overloaded across three distinct producing paths — the throwing-property-access path above, the snapshot-failure path where the handler-entry pinned-constant lookup failed before `event.reason` was read, and the sub-step-2 stamp-throw path per PIC's [*Accepted loom 1.0 residual gap — sub-step-2 stamp-throw case*](./pi-integration-contract.md#substep-2-stamp-throw-residual-gap), where payload-construction substitutes the `<unreadable>` sentinel because the affected invocation's `entry.shutdownReason` field is unset (sub-step 2's stamp threw before populating it) and the per-invocation `finally` later reads that unset field. PIC's [**Unknown-reason rule**](./pi-integration-contract.md#unknown-reason-rule)'s captured-value enumeration covers the first two paths via its four captured `event.reason` forms (closed-set member; `String(event.reason)`; `<unreadable>` on throwing-access; `<unreadable>` on snapshot-failure); the third path's `<unreadable>` is sourced from the per-entry `entry.shutdownReason` field rather than from the captured value and is owned by the residual-gap paragraph just cited. The three `<unreadable>` causes are disambiguated by orthogonal signals rather than by the rendered `<reason>` value alone: the co-emitted teardown diagnostic (`loom/host/session-shutdown-reason-unknown` versus `loom/host/session-shutdown-pinned-constant-unreadable`) discriminates the throwing-access versus snapshot-failure paths, and the affected invocation's dual appearance in the `loom/runtime/reload-teardown-timeout` `<list>` of the same `session_shutdown` event (per the residual-gap paragraph) discriminates the sub-step-2 stamp-throw path from the other two.
  - `<kind>` is a closed token-name table per emitting site:
    - In `loom/parse/non-string-enum-value` and `loom/parse/non-string-discriminator`: one of `string`, `integer`, `number`, `boolean`, `null` (the type-kind of the offending literal).
    - In `loom/load/settings-invalid-entry` and `loom/load/manifest-invalid`: the JSON kind, one of `string`, `number`, `boolean`, `null`, `array`, `object`.
    - In `loom/load/host-incompatible`: one of `node-floor`, `abortsignal-shape`, `sdk-capability-missing`, `peer-dep-out-of-range`, `peer-dep-malformed-version`, `typebox-shape`, `probe-failed` (the closed Step 0 capability-probe kinds).
  - `<op>` is the source token verbatim, drawn from the closed table in category 3, extended to `++` and `--` for `loom/parse/increment-decrement`.
- **Parse-time literal-value** (`<value>`) — when used in a parse-time row (`loom/load/unknown-mode-value`, `loom/parse/duplicate-enum-value`, `loom/parse/duplicate-discriminator-value`), rendered as the literal source text verbatim per category 5's `<key>` (bare-when-identifier-shaped, double-quoted otherwise) — except that YAML scalars with no enclosing source quoting (e.g. `mode:` values) render unquoted regardless of identifier shape.

**Test vectors.**

- A `loom/runtime/reload-teardown-timeout` with two in-flight invocations against slash names `plan` and `review`, UUIDs `aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa` and `bbbbbbbb-bbbb-bbbb-bbbb-bbbbbbbbbbbb`, and elapsed wall time 2003 ms renders `reload teardown timed out after 2003ms; 2 invocation(s) still in flight: /plan:aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa, /review:bbbbbbbb-bbbb-bbbb-bbbb-bbbbbbbbbbbb`.
- A `loom/load/discovery-slow` aborted at root `~/work/looms` because the `looms.scanPackagesMaxFiles` cap fired renders `package-discovery walk aborted at ~/work/looms: looms.scanPackagesMaxFiles cap reached`.
- A `loom/load/cross-source-shadow` where the same slash name `plan` resolves from a higher-priority `--loom` flag pointing at `/proj/a/plan.loom` and a lower-priority settings entry pointing at `~/work/plan.loom` renders `slash name 'plan' shadowed across discovery sources: 'cli-flag:"--loom /proj/a/plan.loom"' wins over 'settings:"~/work/plan.loom"'`.

### 8. Host-derived freeform-tail placeholders

**Placeholders.** `<error>`, `<message>`, `<observed>`.

**Rule.** Render the underlying host-supplied string by category 6's first-line truncation rule (newline-normalise per [Lexical — Encoding](./lexical.md), take the prefix up to but not including the first `\n`, preserve trailing whitespace, render the literal `<no message>` when empty). Byte-identicality of the surrounding template still holds: every byte of the *Message* template before the §8 placeholder, and every byte after, is byte-identical across implementations. Only the §8 placeholder's interpolation is implementation-defined-tail at the byte level (the first-line truncation rule still applies to bound the rendering). Tests asserting on a *Message* template containing a §8 placeholder MUST construct partial-match patterns anchored on the byte-identical prefix and suffix; conformance to category 8 is therefore both decidable and testable.

**Carve-out: `<failure>` is not a §8 placeholder.** The `<failure>` placeholder used by the `loom/host/session-shutdown-pinned-constant-unreadable` row is **not** host-derived freeform tail and is **not** rendered per the §8 rule above. It is a loom-internally-constructed discriminator drawn from a closed set of two byte-determined literal forms (`"missing-entry"`, `"literals-shape-invalid"`) plus a `"throw:<…>"` template family whose `<…>` payload has already been truncated to at most 256 UTF-16 code units and control-character-escaped at the diagnostic-construction site (per [Pi Integration Contract — Unknown-reason rule](./pi-integration-contract.md#unknown-reason-rule), which owns the grammar). The renderer MUST emit the discriminator string verbatim: it MUST NOT re-truncate or re-escape the `"throw:<…>"` payload, MUST NOT substitute `<no message>` for an empty string (the empty case is unreachable by construction — every discriminator value is non-empty), and MUST NOT apply §8's first-line truncation. Tests asserting on the `pinned-constant-unreadable` row's rendered message MAY use byte-identical strict equality for the two literal forms, and MAY use a `startsWith("throw:")` partial-match for the throw-template family (matching the operator-side parse pinned by PIC's **Unknown-reason rule**). The discriminator set is **not** a `Set<string>`-membership-checkable enumeration: the third arm is a template family with an open-ended payload, not a third literal, so a `Set<string>.has(d)` check against the two literals would reject every legitimate `"throw:..."` value. The canonical membership predicate operator tooling MUST use against a `details.failure` value `d` is `d === "missing-entry" || d === "literals-shape-invalid" || d.startsWith("throw:")` (or an equivalent expression that admits the same three discriminator forms — the two literals by `===` and the throw-template family by `startsWith("throw:")`); the `"throw:<unreadable>"` coercion-failure fallback pinned in the row description below is admitted by the `startsWith("throw:")` arm and needs no separate predicate clause. The §8 prefix/suffix-anchoring rule below does **not** apply to this row, and the build-time prohibition on strict-equality assertions enumerates only rows whose Message templates contain an actual §8 placeholder. A future tightening of §8's rendering rules MUST NOT change `<failure>`'s wire format — that wire format is pinned at the construction site, and operator tooling that parses `details.failure` with `startsWith("throw:")` keys on it directly.

**Test vectors.**

- A `loom/load/host-incompatible` whose `<observed>` is `v18.19.0` and `<required>` is `>=22.19.0` renders the byte-identical surround `host incompatible (node-floor): observed ` followed by the host's reported version (truncated to the first line per the rule above) followed by the byte-identical suffix `, required >=22.19.0`. A test asserting this row matches `^host incompatible \(node-floor\): observed (.*), required >=22\.19\.0$`.
- A `loom/runtime/active-set-restore-failed` for a slash command `/plan` whose underlying restore error message is `connection closed\nstack ...` renders `failed to restore tool active-set after /plan: connection closed`.

### Edge cases

- **Category 2 boundary.** A test that constructs an exactly-80-code-point string scrutinee MUST observe the full string (no truncation); a test at 81 code points MUST observe the truncated form (77-code-point prefix + literal `...`). Boundary-condition test vectors are required at the 80/81 code-point boundary.
- **Category 5 identifier predicate.** The identifier-shape check for `<key>` is a *runtime* string predicate against the regular expression `^[A-Za-z_][A-Za-z0-9_]*$`; a key like `kind` renders bare, `my-key` renders quoted. Keys whose string value happens to match a reserved keyword from [Lexical — Reserved keywords](./lexical.md) are still identifier-shaped for this rule (the reserved-keyword rule fires at parse time on identifier *positions* in source, not at runtime on string values).
- **Category 6 line-separator scope.** Restating for emphasis: `\u2028` and `\u2029` are ordinary characters for this rule. Implementations MUST NOT split on them, MUST NOT strip them, and MUST NOT promote them into `\n` — the runtime is byte-stable in their presence even though authors cannot introduce them through a regular string literal (see [Lexical — String literals](./lexical.md)).
- **Category 7 `<callee>` fallback.** When `<callee>` falls through to the path form (the parent has no slash-name binding for the callee — e.g. a sibling `.warp` `fn`), it renders verbatim per the path-literal rule rather than as an absolute path. This matters for the `loom/parse/invoke-arity-too-few` test vectors: a `.warp` `fn` callee renders as the source-text path the `import` declared, not a realpath.
- **Category 7 closed-enum closure.** The closed value tables for `<cap>`, `<capability>`, `<dotted-key>`, `<reason>` (closed-arm path), and `<kind>` are themselves spec-versioned per **GOV-7** and **GOV-8**: extending any table is a breaking change to the rendering surface even when the underlying registry row is unchanged.
- **Category 8 prefix/suffix anchoring.** A test that asserts equality against the full *Message* string of a category-8 row is non-conformant under §8's rule: the placeholder is implementation-defined-tail at the byte level and only the surround is byte-identical. This non-conformance is detected at build time by enumerating registry rows whose Message templates contain a §8 placeholder and asserting that no test in the suite asserts strict equality against those rows' full Message strings.

## Code registry

The table enumerates every diagnostic the V1 spec defines. *Severity* is `error` (`E`) or `warning` (`W`); a few discovery-source codes carry severity `E/W`, meaning the severity is decided per-source by the table in [Discovery — Failure modes](./discovery.md) rather than fixed at the code level. *Phase* identifies which pipeline stage emits the diagnostic: `lex` (lexing / encoding), `parse` (parsing / static checks performed by the parser), `type` (type-system checks), `load` (file-load, registration, discovery), or `runtime` (panic during execution). *Trigger* is the canonical condition; *Spec rule* points to the topic page where the rule is stated; *Hint* gives the normative author-facing hint when the spec mandates one; *Message* is the rendered author-facing string template (one line; `<…>` placeholders are interpolated by the renderer).

### `loom/runtime/*` — runtime panics, runtime-defect surface, and delivery failures

loom 1.0.0 has exactly six **panic sources** (the rows below tagged as panics). Each surfaces through the channels defined in [Errors and Results — Runtime panics](./errors-and-results.md) (slash-command system note for top-level looms; `Err(QueryError { kind: "invoke_failure", cause: "panic", ... })` to an `invoke` parent). One further code — `loom/runtime/internal-error` — covers the **runtime-defect surface** also defined in [Errors and Results — Runtime panics](./errors-and-results.md): an unanticipated interpreter or adapter throw outside the closed panic-source list, routed through the same channels as panics but not itself a panic source. The code is registered and stable; only its trigger condition is intentionally open-ended, which does not weaken the closed-registry rule (rule 2 above) — that rule constrains *codes*, not triggers. Six further codes — `system-note-delivery-failed`, `subagent-dispose-failure`, `registry-swap-failed`, `registration-cache-collision`, `reload-teardown-timeout`, and `active-set-restore-failed` — are delivery-, rebuild-, registration-, teardown-, or active-set-restore-failure diagnostics emitted, respectively, by the system-note fallback path, the subagent session lifecycle teardown, the watcher's build-aside-then-publish swap, the prompt-mode tool-registration cache's collision check, the `session_shutdown` handler's bounded await on in-flight subagent disposal, and the prompt-mode (and prompt → prompt cross-mode `invoke`) tool-active-set restore-failure protocol, all defined in [Pi Integration Contract](./pi-integration-contract.md). `reload-teardown-timeout` differs from the other five in delivery channel: because Pi's `ExtensionRuntime.invalidate(...)` may already have invalidated the `pi.*` surface by the time the teardown handler runs, the timeout diagnostic is emitted via `console.error` rather than through the standard persistent-diagnostic channel.

### `loom/host/*` — host-observed anomalies

Diagnostics in this namespace are emitted from loom's interactions with the Pi host. The namespace covers both anomalies the loom runtime observes coming *from* the Pi host that the spec's host-prerequisites contract does not enumerate, and loom-internal failures of host-related machinery — pinned-constant blocks that snapshot host SDK types, host-event handlers, and similar — whose immediate cause is loom-internal corruption but whose observation site is a host-driven code path.
