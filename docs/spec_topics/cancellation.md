# Cancellation

Every theta invocation runs under an `AbortSignal` provided by Pi. theta 1.0 cancellation rules:

**Signal source.** Each theta invocation owns a fresh `AbortController` (`thetaAbort`) that the runtime constructs at invocation start. Its `thetaAbort.signal` — never `ctx.signal` directly — is the single source of truth that every downstream component sees: the interpreter's loop, `@`-query, tool-call, and `invoke` checkpoints; the `signal` argument forwarded to `tool.execute(...)` for code-side tool calls; the synthesised `ExtensionContext.signal` for the same; the parent signal handed to a child invoke; and (in subagent mode) the trigger for the one-shot listener that initiates the child cancellation grace signal — closing the child's parent-held stdin pipe, then a bounded grace and process-tree kill — since under the RFC 0006 child-process theta design the parent drives no in-process handle and the `-p` child exposes no RPC abort command (see [Pi Integration Contract — Cancellation (PIC-63)](./pi-integration-contract/subagent.md#pic-63) and [Subagent child-process lifecycle (PIC-9)](./pi-integration-contract/subagent.md#pic-9)). `thetaAbort.signal` is always defined; tool adapters and Pi APIs that accept an `AbortSignal` receive it directly without optional-chaining.

**Forwarding into `thetaAbort`** depends on the entry point:

- *Slash-command entry.* The handler subscribes for the duration of the theta run. Whenever a Pi turn is active and the per-handler `ctx.signal` is observed inside one of the runtime's own event handlers (`tool_call`, `tool_result`, `message_update`, `turn_end`, `agent_end`), an aborted `ctx.signal` triggers `thetaAbort.abort()`; equivalently, an `agent_end` event reporting a user-cancelled turn aborts `thetaAbort`. This is the path that makes Esc-during-`@`-query work end-to-end. The runtime MUST tolerate `ctx.signal` being `undefined` at slash-command entry — Pi documents `ctx.signal` as `undefined` in idle, non-turn contexts (which is exactly when the slash-command handler fires) — and MUST NOT depend on its truthiness for any pre-turn checkpoint. Forwarding is also bidirectional in prompt mode: when `thetaAbort.abort()` fires while a prompt-mode user turn is in flight, the runtime calls the unwrapped, Pi-supplied `abort()` on the captured `ExtensionCommandContext` — not the synthesised tool-execution `ExtensionContext` whose overridden `ctx.abort()` wrapper itself calls `thetaAbort.abort()` (see the [`ExtensionContext` *Per-mode override semantics* table](./pi-integration-contract/host-interfaces-core.md#extensioncontext-interface)) — to tear down the user run and unblock `await ctx.waitForIdle()` (see [Pi Integration Contract — Conversation drive — prompt mode](./pi-integration-contract.md)). The propagation is guarded by a one-shot flag so a re-entrant `thetaAbort.abort()` does not double-cancel. The subagent-mode counterpart is its own forwarding wiring: immediately after the child spawn is initiated and its stdio is wired, the runtime registers a one-shot `thetaAbort.signal` listener that initiates the child's cancellation grace signal (closing the child's parent-held stdin pipe) and then proceeds through the bounded await → process-tree kill teardown (and detaches the listener in the same per-invocation `finally` that tears the child down, removing every other forwarding listener — see [Pi Integration Contract — Cancellation (PIC-63)](./pi-integration-contract/subagent.md#pic-63) and [Subagent child-process lifecycle (PIC-9)](./pi-integration-contract/subagent.md#pic-9)). If `thetaAbort.signal.aborted` is already true at the moment the child stdio is ready, the runtime initiates the grace signal synchronously *before* the listener is registered, so the spawn-then-immediate-cancel path remains correct without depending on microtask ordering. The one-shot listener and the existing one-shot guard on `thetaAbort.abort()` together make a double-abort (e.g. operator presses Esc twice) a no-op.
- *Tool-exposed entry (a theta registered into another theta's `tools:`).* The `signal` parameter passed to `execute(toolCallId, params, signal, onUpdate, ctx)` is wired so that `signal.aborted` triggers `thetaAbort.abort()` via a one-shot listener registered at entry. The runtime may also call `thetaAbort.abort()` itself on any internal failure that should cascade.
- *`invoke(...)` entry.* The parent passes its own `thetaAbort.signal` into the child's constructor; the child constructs its own `thetaAbort` as a derived controller that aborts when the parent's signal aborts but not vice versa, matching the downward-only **Propagation** rule below. If the parent's signal is already aborted at child-spawn time, the child surfaces `Err(QueryError { kind: "cancelled", ... })` synchronously without spawning a session.

<a id="cncl-4"></a> **CNCL-4.** **Abort-reason propagation.** When any of the three forwarding paths above fires `thetaAbort.abort(...)`, the runtime MUST propagate the source's `reason` so that `thetaAbort.signal.reason === source.reason` is observable at every downstream checkpoint. Concretely: the slash-command `ctx.signal`-aborted trigger calls `thetaAbort.abort(ctx.signal.reason)`; the tool-exposed `signal`-aborted trigger calls `thetaAbort.abort(signal.reason)`; and the `invoke`-parent derived-controller trigger calls `thetaAbort.abort(parentSignal.reason)`. Implementations MAY use `AbortSignal.any([source])` (whose derived signal's `reason` mirrors the source's `reason` per WHATWG) or a manual `addEventListener` callback that forwards the source's `reason` argument; the two are observably equivalent under this rule. The slash-command path's `agent_end`-driven trigger has no source `AbortSignal` — there is no `reason` to forward — so the runtime synthesises a reason, a JavaScript `Error` whose `message` is exactly `"theta cancelled by agent_end"`, and calls `thetaAbort.abort(reason)` with it. The one-shot guard on `thetaAbort.abort()` already documented above means a second forwarder firing after the first does not re-stamp the reason; the first source's reason wins. The bidirectional prompt-mode propagation (`thetaAbort.abort()` → `ctx.abort()`) goes the other direction across the linkage and is unaffected by this rule, since Pi's `ExtensionCommandContext.abort()` accepts no reason argument.

**Forwarding-listener throw.** A throw from a steady-state forwarding listener's `thetaAbort.abort(source.reason)` call — the slash-command `ctx.signal`-aborted trigger, the tool-exposed `signal`-aborted trigger, or the `invoke`-parent derived-controller trigger — is trapped at the listener boundary and routed through the **runtime-defect surface** defined in [Errors and Results — Runtime panics](./errors-and-results/error-model.md#runtime-panics) on the `theta/runtime/internal-error` channel, with the `cause: "internal_error"` arm of [`InvokeInfraError`](./errors-and-results/queryerror-variants.md#invoke-variants) at an `invoke` parent. The trap MUST NOT swallow the cancellation itself — only the defect throw: `source.signal.aborted` is unchanged, and the next [`Checkpoint` seam](./pi-integration-contract/host-interfaces-services.md#checkpoint-seam) await still observes cancellation through the source path and surfaces the `Err(QueryError { kind: "cancelled" })` variant per the **Surfacing** rules below. The clause mirrors the grace-signal-send-throws rule in [Pi Integration Contract — Cancellation (PIC-63)](./pi-integration-contract/subagent.md#pic-63) and the [Dispatch-site setup wrap](./pi-integration-contract/active-invocation-registry.md#active-invocation-registry) rule that covers the listener-*attach* throw on the same surface; the `session_shutdown` sub-step 2 swallow rule remains the governing rule for the teardown-iteration path and is not subsumed by this clause, which covers steady-state forwarding only. The `ctx.abort()` override row of the [`ExtensionContext` member-surface table](./pi-integration-contract/host-interfaces-core.md#extensioncontext-interface) cites this clause so a throw from the underlying `thetaAbort.abort()` invoked by the `ctx.abort()` wrapper from theta-driven tool code is covered by reference rather than re-derivation.

All forwarding listeners (Pi-side or invoke-parent-side) are removed when the theta returns or panics, in the same `finally` block that tears down any subagent child process (see [Pi Integration Contract — Subagent child-process lifecycle](./pi-integration-contract/subagent.md#pic-9)); listener cleanup is mandatory to prevent `AbortController` leakage across long-running Pi sessions.

A second trigger fires `thetaAbort.abort()` independently of the per-invocation `finally`: the extension's `session_shutdown` handler (see [Pi Integration Contract — Extension entry point](./pi-integration-contract.md), step 4) iterates the [`ActiveInvocationRegistry`](./pi-integration-contract/active-invocation-registry.md#active-invocation-registry) on `/reload`, `/new`, fork, or quit and aborts every entry's `thetaAbort` with a synthesised reason — a JavaScript `Error` whose `message` is exactly `"theta cancelled by session shutdown"` — i.e., calls `thetaAbort.abort(reason)` per the **Abort-reason propagation** rule above, so that subagent provider connections, in-flight queries, and child invokes drain before Pi's `ExtensionRuntime.invalidate(...)` runs. The teardown handler also detaches the same forwarding listeners listed above, duplicating the per-invocation `finally`'s listener-cleanup work for the case where Pi tears the runtime down before that `finally` reaches its cleanup. The two paths are designed to overlap safely: each listener-removal call is idempotent at the call site, and the abort itself is a no-op on an already-aborted controller.

**Propagation.** Cancellation propagates *down* (parent → child invokes, parent → in-flight queries, parent → in-flight tool calls). It does *not* propagate *up*: a child theta cancelling internally surfaces as `Err(QueryError { kind: "cancelled" })` (or the appropriate sub-variant) to the parent, which may handle it (`match`) or propagate it (`?`). Under the `session_shutdown` carve-out (see [Pi Integration Contract — Session-swap behaviour for in-flight invocations](./pi-integration-contract/session-shutdown-semantics.md#session-shutdown-semantics)), parent and child invocations are co-aborted in the same iteration of the teardown handler by the same external trigger; the parent's `?` / `match` arms covering the child's `Err` do not run inside the bounded-await window because the parent's own `thetaAbort` has already fired and its next [`Checkpoint` seam](./pi-integration-contract/host-interfaces-services.md#checkpoint-seam) observes the abort. The downward-only propagation rule above describes steady state, not co-abort by the teardown handler.

**Granularity.** The interpreter checks the cancellation signal at exactly these points and no others:

- immediately before each iteration of a `for` or `while` body,
- immediately before dispatching each `@`...`` query,
- immediately before each tool call,
- immediately before each `invoke` call,
- immediately before issuing the slash-command argument binder's LLM call (and the signal is forwarded to the binder model's provider invocation, so an abort observed *during* the binder call also surfaces).

The loop-iteration checkpoint releases the event loop before its signal-check: a compute-bound `for`/`while` body with no genuine `await` between iterations would otherwise drain only the microtask queue and never return a turn to the event loop, so the Pi-dispatched event that flips `thetaAbort.signal.aborted` (a macrotask) could not run and an Esc during such a loop would never land. The runtime therefore yields one macrotask turn at each loop-iteration boundary before reading the signal, so a pending abort is observed before the next iteration; this is the `loop-iter` case of the [`Checkpoint` seam](./pi-integration-contract/host-interfaces-services.md#checkpoint-seam). This single macrotask turn is sufficient only under the theta-side presupposition that Pi flips `thetaAbort.signal.aborted` within a single macrotask — recorded as presupposition (vi) of [Pi Integration Contract — Pi slash-handler promise lifecycle (theta consumption posture)](./pi-integration-contract/host-prerequisites.md#pi-slash-handler-promise-lifecycle-presupposition). The other checkpoints incidentally yield because they precede real async I/O.

Synchronous in-process work — schema lowering at file-load time, AJV validation of already-received bytes, default-merging — is not a checkpoint; it runs to completion. No checkpoint fires inside a primitive operation (arithmetic, comparison, field/index access). A compound expression that contains multiple checkpointed sub-expressions can therefore be cancelled between them: in `let x = f() + g()` where `f` and `g` are tool calls, an abort observed between `f()`'s return and `g()`'s pre-call checkpoint aborts the statement after `f`'s side effects have committed but before `g` is dispatched, and `x` is never bound. Partially evaluated expressions are discarded; cancellation does not unwind side effects of sub-expressions that have already completed (see [Errors and Results](./errors-and-results.md)).

<a id="cncl-5"></a> **CNCL-5.** **Race semantics — no retroactive rewrite of a completed `Ok`.** Cancellation is observed only at checkpoints. An operation that has already returned `Ok(v)` retains that value even if the signal fires before the next checkpoint executes; the interpreter must not retroactively rewrite a completed `Ok` into `Err({kind:"cancelled"})`. The cancellation surfaces at the next checkpoint the interpreter reaches — typically the pre-evaluation check of the next statement's first cancellable sub-operation (loop iteration, `@`-query, tool call, or `invoke`).

<a id="cncl-6"></a> **CNCL-6.** **Race semantics — no top-level synthesis on tail abort.** If no further checkpoint executes before the theta returns (the abort fired after the final cancellable operation), the theta's top-level result is the value it would otherwise have produced; the runtime does **not** synthesize a top-level `cancelled` in that case.

Symmetrically, an in-flight operation whose underlying provider observes the abort surfaces as `Err` per the **Surfacing** rules below; this is the only path by which an operation's own result becomes `cancelled`.

**Race semantics — late-settlement discard at the tool-call checkpoint.** The no-retroactive-rewrite rule extends to the tool-call boundary specifically: once a `tool-call` checkpoint has surfaced `Err(QueryError { kind: "code_tool", cause: "cancelled", ... })` for a given tool invocation, any later settlement of the underlying `execute()` Promise (resolve with `Ok` or `isError: true`, reject with a throw, or any other outcome enumerated in [Pi Integration Contract — Tool execution from theta code](./pi-integration-contract/host-interfaces-core.md#tool-execution-from-theta-code)) is discarded across three coupled sub-obligations, each carrying its own REQ-ID anchor so a test, conformance fixture, or cross-page citation can pin one sub-obligation without dragging in the other two:

- <a id="cncl-1"></a> **CNCL-1.** *(clause (a) — no-rebind.)* The runtime MUST NOT rebind the call site to the late value.
- <a id="cncl-2"></a> **CNCL-2.** *(clause (b) — no-second-`Err`.)* The runtime MUST NOT emit a second `Err` for the same invocation.
- <a id="cncl-3"></a> **CNCL-3.** *(clause (c) — no-second-`RuntimeEvent`.)* The runtime MUST NOT emit a second `RuntimeEvent` for the same invocation.

The discriminator is whether cancellation has already been surfaced at the checkpoint, not the late-settle kind. The three-clause discard above is scoped to the tool-call boundary; the other cancellation-surfaced Promise sites the runtime may abandon — the underlying provider Promise of an `@`-query, the `invoke` callee's top-level execution Promise, and the subagent-mode cancellation grace-signal send Promise documented as "discarded" by [Pi Integration Contract — Cancellation (PIC-63)](./pi-integration-contract/subagent.md#pic-63) — are governed only by the swallowing-handler obligation in the next paragraph (suppression of all three side channels — no Node `unhandledRejection`, no second `RuntimeEvent`, and no diagnostic of any severity — without extending clauses (a) or (b) above to those sites), and are not covered by clauses (a) or (b) above (the `@`-query and `invoke` sites have their own per-checkpoint `Err` surfaces governed by the **Surfacing** rules below, and the grace-signal send Promise has no theta-language call site to rebind and no per-invocation `Err` channel). The `Checkpoint` seam (per [Pi Integration Contract — `Checkpoint` seam](./pi-integration-contract/host-interfaces-services.md#checkpoint-seam)) is the deterministic-test substrate for this discard.

**Race semantics — swallowing-handler attachment on every abandonable Promise.** The side-channel suppression that complements the previous paragraph's discard is implemented through a swallowing-handler rule that closes the `unhandledRejection` side channel: the runtime MUST keep a swallowing handler attached to the underlying `execute()` Promise (and, symmetrically, to the underlying `@`-query provider Promise and the `invoke` callee's top-level execution Promise) for the lifetime of that Promise, so that a late rejection arriving after the corresponding checkpoint has already surfaced `cause: "cancelled"` is silently absorbed and does not surface as a Node `unhandledRejection` process event. The swallowing-handler suppression is total along its three side channels: no Node `unhandledRejection` process event fires, no second `RuntimeEvent` reaches the always-log channel for the same invocation, and no diagnostic of any severity is emitted (a late rejection whose `.message` would otherwise be diagnostic-worthy — e.g. an `OOMError`-style host failure — is still discarded; promotion to `theta/runtime/internal-error` would re-introduce the second-event surface this rule forbids). The two `Err`-channel obligations — "no `Err` rebinds the call site" ([clause (a)](#cncl-1)) and "no second `Err` for the same invocation" ([clause (b)](#cncl-2)) — are owned by the previous paragraph and remain tool-call-only; this paragraph does not extend them to the other three sites. The handler MUST be attached at the same site that constructs the Promise, before the first microtask boundary; a `.catch(() => {})` registered lazily after cancellation fires will not catch a rejection that has already been queued for `unhandledRejection`. The discarded value is not traversed; cleanup of any resources it carries (file descriptors, native handles) is the tool implementation's responsibility, not the runtime's. The swallowing-handler attachment / no-side-channel obligation in this paragraph applies uniformly to every site that owns a Pi-returned Promise the runtime might abandon under cancellation: the code-side `execute()` Promise, the underlying provider Promise of an `@`-query, the `invoke` child's top-level execution Promise, and the subagent-mode cancellation grace-signal send Promise documented as "discarded" by [Pi Integration Contract — Cancellation (PIC-63)](./pi-integration-contract/subagent.md#pic-63). The `Checkpoint` seam provides the deterministic-test substrate for landing the late settlement at a chosen point so the rule is verifiable without relying on JS microtask scheduling.

Edge cases:

- Statement boundaries are *not* themselves checkpoints; the next checkpoint is the next loop-iter boundary, `@`-query, tool call, or `invoke`. A straight-line statement sequence with no such operations runs to completion regardless of when the abort fired.
- For `invoke`, the child's own checkpoints honour the derived signal independently — the parent does not need to re-check between child completion and binding the child's result.
- The top-level "no further checkpoint" rule means a theta that ends in a pure-arithmetic tail can complete `Ok` even if the user pressed Esc during that tail. This is intentional: there is nothing left to cancel.
- The Granularity rule fixes *which* operations carry a checkpoint; the Race rule disambiguates *which* checkpoint observes a given abort. The two together imply that the smallest unit of work the runtime guarantees to either complete or skip is one checkpointed sub-expression (one loop iteration, one `@`-query, one tool call, or one `invoke`), not one statement.
- A query or tool call's pre-call checkpoint fires *before* the call is dispatched. An abort observed *during* an in-flight query, tool call, or `invoke` surfaces through the underlying provider's abort path as the corresponding `Err` variant per the **Surfacing** rules below, not through a pre-call checkpoint.
- The two race rules — no retroactive `Ok(v)` → `Err({kind:"cancelled"})` rewrite, and no top-level synthesis when no further checkpoint executes — are only deterministically testable through a runtime-internal hook that fires synchronously *before* each checkpoint's signal-check. That hook is the `Checkpoint` seam declared in [Pi Integration Contract — `Checkpoint` seam](./pi-integration-contract/host-interfaces-services.md#checkpoint-seam). Production wiring is a no-op (an already-resolved promise per checkpoint); tests use it to land an abort at a chosen checkpoint boundary without depending on JS microtask scheduling. The seam is purely a test surface and imposes no observable behaviour on production code beyond the always-`await`ed no-op.

<a id="surfacing"></a>

**Surfacing.**

- An in-flight query whose signal aborts returns `Err(QueryError { kind: "cancelled", ... })`. The `...` is the same omitted-fields convention the sibling arms below use; `message` is unconstrained per [Errors and Results — `CancelledError`](./errors-and-results/queryerror-variants.md#queryerror-variants), so no byte-exact constraint applies to it.
- A tool call whose signal aborts returns `Err(QueryError { kind: "code_tool", cause: "cancelled", ... })`.
- A child invoke whose signal aborts surfaces to the parent as `Err(QueryError { kind: "invoke_callee", inner: { kind: "cancelled", ... } })` when the abort originated inside the child, or directly as `kind: "cancelled"` when the parent's own signal fired first.
- A cancelled binder call (abort observed before or during the binder's LLM call) is runtime-internal: it never surfaces as a `Result` to theta code (the theta never starts). Instead it produces the cancelled-binder system note defined in the failure-modes table in [Slash-Command Argument Binding](./binder.md). The theta does not run.
- The theta's *top-level* cancellation surfaces to Pi as the `cancelled` row in the per-`kind` system-note table in [Slash-Command Invocation](./slash-invocation.md#slsh-4).
- For any of the surfacing arms above whose underlying turn streamed user-visible tokens before the abort fired, the runtime's non-mutation obligation toward the Pi-committed conversation (no truncation, re-write, replacement, removal, or compensating injection of assistant tokens, tool-call cards, or system notes) is owned by [Errors and Results — Mid-stream cancellation, conversation state](./errors-and-results/error-model.md#mid-stream-cancellation-conversation-state). The surfacing rules above describe the `Result`-typed envelope returned to theta code; the linked rule describes what the runtime is forbidden from doing to the conversation Pi observes between that envelope and the next driver send.

Per-call timeouts (a separate cancellation source independent of the user) are deferred to a later release; declaring a `timeout:` field on a query, tool call, or invoke is `theta/parse/timeout-field-rejected`. See [Future Considerations](./future-considerations.md) and [Diagnostics](./diagnostics.md).
