# Invocation

A theta may invoke another theta via the built-in `invoke` expression. This is the only way for a `.theta` to spawn or attach to another `.theta`'s execution by an inline path literal; for repeated or model-exposed callees, register the path in frontmatter `tools:` and call by name (see [Tool Calls](./tool-calls.md)). `import` is reserved for `.thetalib` library code.

```theta
let plan: Plan = invoke<Plan>("./plan.theta", topic, depth)?
let _ = invoke("./logger.theta", note)?
```

**Resolution.** `path` is a string literal, resolved at parse time relative to the calling theta's directory. It must end in `.theta` — the extension match is byte-exact lowercase per [Lexical — Extension matching](./lexical.md#extension-matching); an `invoke(...)` path whose literal does not end in `.theta` (including a `.thetalib` path or any non-lowercase variant such as `.THETA`) is a parse error `theta/parse/invoke-non-theta-extension`. The same code applies to `tools:` `.theta` entries whose path string does not end in `.theta`. The path uses forward-slash separators only — a backslash inside the path literal is a parse error per the "Path literals" rule in [Lexical Structure](./lexical.md). Dynamic dispatch (a runtime-computed path) is not supported in theta 1.0.

The resolved path — after `realpath` symlink normalisation — must lie within the union of **discovery roots** active for the current Pi session (see [Directory Convention — Discovery roots](./discovery/discovery-sources.md#discovery-roots)). The containment comparison is decided on the byte-exact output of [`FileSystem.realpath`](./pi-integration-contract/host-interfaces-services.md#pic-13) — whose normative full-transitive-symlink-resolution and byte-stable canonical-output guarantees this check relies on — applied to *both* the resolved callee path and each active discovery root: the two canonical paths are compared byte-for-byte after forward-slash normalisation (per the "Path literals" rule in [Lexical Structure](./lexical.md)), with no independent case-folding — the canonical form is whatever `realpath` returns on the host. This constrains how the comparison is performed; the within-root predicate it composes with is **segment-boundary containment**: after normalisation strips any trailing host path separator from a discovery root, a resolved callee path is within that root when it either equals the root byte-for-byte or begins with the root followed by a single host path separator. A bare prefix test is insufficient — it admits sibling-prefix escape, where a resolved path such as `/proj/foo-bar/evil.theta` would pass a `startsWith("/proj/foo")` test while sitting outside root `/proj/foo` — so the trailing-separator requirement is load-bearing, and the resolved path is within the union exactly when it satisfies this segment-boundary predicate against at least one active root. The same byte-exact-on-`realpath`-output comparison governs both the load-time check and the runtime-open re-check (INV-1 below pins identical containment semantics across both call sites; this is the case-sensitivity instantiation of that pin). A resolved path that escapes every active root is a load-time error `theta/load/invoke-path-escape`; the parent theta does not register the call site, and a `tools:` `.theta` entry that escapes likewise fails to register the callable. The realpath step is mandatory: a symlink farm inside a discovery root that resolves outside it is still rejected. The same restriction applies to `.theta` paths used as `tools:` entries (see [Tool Calls](./tool-calls.md)). The realpath + discovery-root containment check is re-run at the moment the runtime opens the callee for invocation. A symlink swapped between load and invocation that lands outside every active root surfaces in two places: a `theta/load/invoke-path-escape` diagnostic on the diagnostics drain, and `Err(InvokeInfraError { cause: "load_failure", callee_path, ... })` to the parent. The two-channel report is deliberate — operators reading the drain can detect escape attempts; the parent's `Err` cannot distinguish escape from deletion, both of which are legitimate causes of `load_failure`. The runtime re-check uses the *currently* active discovery-root set rather than a snapshot taken at load, so a hot-reload that removed the root the load-time check relied on causes the invocation to fail closed. An irreducible kernel-level race remains between the runtime `realpath` and the subsequent `open(2)`; this re-check narrows the window from minutes to microseconds but does not close it. theta 1.0 accepts the residual (see [Future Considerations — Symlink-resolution hardening for invoke-path containment](./future-considerations/surface-extensions.md#symlink-resolution-hardening)). Cross-root composition (a project theta invoking a global utility) stays legal because the check uses the *union* of active roots, not the calling theta's own root. *Non-normative implementation note.* On case-insensitive hosts Node's `fs.promises.realpath` returns the on-disk casing of the resolved tail, so filesystem-equivalent paths canonicalise to the same byte sequence and the byte-exact comparison is correct; on case-sensitive hosts byte-exactness already coincides with filesystem equivalence. A symlink-farm callee whose `realpath` crosses a case-variant directory entry stays well-defined because both sides are compared on `realpath`'s output. Stating the rule as byte-exact on `realpath` output rather than as any platform's case behaviour is the guard a future port to a host whose `realpath` does not preserve on-disk casing must respect — the test `FakeFileSystem` mimics per-host `realpath` casing accordingly.

> <a id="inv-1"></a> **INV-1.** **Symlink-resolution-hardening seam.**
>
> **theta 1.0 seam — symlink-resolution hardening.** <a id="theta-1-0-seam-symlink-resolution-hardening"></a> The load-time check (parent theta registration / `tools:` `.theta` entry registration) and the invocation-time re-check (the moment the runtime opens the callee for invocation) MUST apply the identical `realpath`-then-discovery-root-containment semantics — including the segment-boundary within-root predicate — defined under *Resolution* above, and a resolved path that escapes every active discovery root MUST surface on both channels: a `theta/load/invoke-path-escape` diagnostic on the diagnostics drain, and `Err(InvokeInfraError { cause: "load_failure", callee_path, ... })` to the parent. *Non-normative implementation note.* theta 1.0 expects to factor this check behind a single named function in the invoke runtime, shared by both call sites, so the deferred *Symlink-resolution hardening* extension in [Future Considerations](./future-considerations/surface-extensions.md#symlink-resolution-hardening) can replace its body in place with a single atomic resolve-and-open primitive (`openat2(..., RESOLVE_NO_SYMLINKS | RESOLVE_BENEATH)` on Linux, platform equivalents on macOS / Windows / other targets) while preserving that diagnostic surface; the single-function factoring is a maintainability convention, not an observable conformance point.

<a id="static-resolution"></a>

**Static resolution.** A callee referenced by a literal `invoke("./path.theta", ...)` or by a `.theta` entry in `tools:` is *statically resolvable* if the runtime can open, parse, and lower the callee file during the calling theta's load pass. Static resolution is performed by a single per-load-pass parse cache shared by every parse-time check that needs the callee's source: argument-type checking ([Argument binding](#argument-binding) below; [Tool Calls — Argument shape](./tool-calls.md)), cross-theta return-type inference ([Function Definitions — Theta return type](./functions.md), [Tool Calls — Return type](./tool-calls.md)), and cycle detection ([Cycle detection](#cycle-detection) below). The walk is transitive: callees referenced by a callee's own literal `invoke` paths and `.theta` entries in `tools:` are loaded into the same cache. Each visited file is parsed once per pass and its diagnostics are aggregated into the entry theta's drain (sorted by `(file, line, col)` per [Diagnostics — Multi-error reporting](./diagnostics.md)).

A callee whose file is unreadable, fails to parse, or fails its own structural checks is *not statically resolvable* and the parent emits `theta/load/callee-has-errors` at the referencing site, naming the callee and listing the underlying error sites via `related`. The severity is per surface: a `tools:` `.theta` entry pointing at an unparseable callee is **error** — the callable cannot be created, and the parent theta does not register; a literal `invoke("./path.theta", ...)` whose callee is unparseable is **warning** — the parent registers, static checks against that callee are skipped, and the runtime AJV check is the safety net for the skipped checks. The asymmetry is deliberate: `tools:` exposes the callee as a callable to both code and model and therefore requires a working callable at registration; `invoke(...)` is a code-side reference that already has a documented runtime-AJV fallback. The callee, when later loaded as its own slash command, fails to register on its own merits — the warning-vs-error distinction matters only for the parent's cross-reference. Hot-reload of a callee re-runs the static-resolution pass for every parent that has the callee in its reachable graph (see [Implementation Notes — Runtime](./implementation-notes.md)).

<a id="typed-return"></a>

**Typed return.** `invoke<Schema>(...)` annotates the expected return type; the runtime AJV-validates the child's return value against the schema. Untyped `invoke(...)` returns `Result<null, QueryError>` — the runtime discards the child's return value entirely. Use `invoke<Schema>` whenever the caller needs the value back; the untyped form exists only for fire-and-forget orchestration (loggers, side-effect-only children).

When both the annotated `Schema` and the callee are statically resolvable (per [Static resolution](#static-resolution) above), the parser checks `T_calleeReturn ⊑ Schema` per [Type System — Type compatibility](./type-system.md#type-compatibility) — compatibility, not equality, so a callee returning a narrower type is legal under a wider annotation (e.g. `Cat ⊑ Animal`). Mismatch is a parse error `theta/parse/invoke-return-type-mismatch`. When either side is not statically resolvable, the runtime AJV check on the child's return value remains the safety net (no parse error fires).

*Empty-tail callee compatibility.* A callee whose body has no tail expression has inferred return type `null` per [Function Definitions — Empty-tail body](./functions.md#empty-tail-body). At a typed-invoke site against such a callee: `invoke<null>("./empty-tail.theta", ...)` is accepted (the static check `null ⊑ null` holds and AJV admits the literal `null`); `invoke<Schema>("./empty-tail.theta", ...)` for any non-`null`-compatible `Schema` is `theta/parse/invoke-return-type-mismatch` when statically resolvable, and `Err(InvokeInfraError { cause: "return_validation", ... })` when the runtime AJV check rejects the literal `null` against a non-nullable `Schema`.

<a id="final-value-propagation"></a>

**Final-value propagation across callees.** A callee's *final value* (the language-level definition is owned by [Function Definitions — Final value](./functions.md#final-value-language-definition)) flows to an `invoke` parent only on the callee's success outcome. A `prompt`-mode child attaches to the caller's current conversation, but the final value still propagates through the same return surface. A `subagent`-mode child runs the whole callee — interpreter included — in a spawned child `pi` process per the [Subagent state-isolation matrix](./pi-integration-contract/subagent.md#subagent-state-isolation-matrix); the final value crosses the subagent boundary as the `ok` arm of the single-JSONL-line `{"theta_result": …}` return envelope the child emits on stdout (the envelope schema, its versioning, and its skew-detection rule are pinned at [Subagent — Return-value envelope (PIC-59)](./pi-integration-contract/subagent.md#pic-59)), while the spawned conversation's transcript is private and not retained after the callee returns. <a id="inv-5"></a> **INV-5.** **Subagent return-value propagation over the envelope.** For a subagent-mode callee the parent MUST derive the `invoke` result solely from the child's `theta_result` envelope: the envelope `ok` arm maps to `Ok(v)` (AJV-validated against a typed `invoke<Schema>` annotation as below), and the envelope `err` arm maps to the corresponding `QueryError` — every `Err` an in-process subagent could surface (`QueryError` variants, `CodeToolError`, the `InvokeInfraError` causes, and panics routed as `internal_error`) is representable in the pinned schema, so `InvokeCalleeError` wrapping is applied to the reconstructed leaf exactly as for an in-process callee. A subagent child that exits **without** emitting an envelope (crash, kill, timeout) MUST map fail-closed to `Err(InvokeInfraError { cause: "internal_error", ... })` with the exit detail — never a fabricated `Ok` — per [PIC-59](./pi-integration-contract/subagent.md#pic-59). A `subagent fn` call (theta 1.2, [Functions — FN-6](./functions.md#fn-6)) crosses this same subagent boundary inline: on success its final value reaches the caller as the `Ok` payload and its spawned transcript is discarded on return; on failure or cancellation no final value flows, exactly as for an `invoke`d subagent-mode callee. On callee failure (`Err` propagation, panic, exhausted runtime ceiling) or cancellation, no final value flows in either mode — the caller observes only the `InvokeCalleeError` (callee returned `Err`) or `InvokeInfraError` (panic / load failure / validation / internal error) envelope per **Failures** below.

**Argument binding.** Arguments bind positionally to the callee's `params:` in declaration order, with each argument type-checked against the param's declared schema. Type mismatches surface as `theta/parse/invoke-arg-type-mismatch` when the callee is statically resolvable per [Static resolution](#static-resolution) above; otherwise the runtime AJV check is the safety net. The LLM-driven binder used at the slash-command boundary (see [Slash-Command Argument Binding](./binder.md)) does not run here — `invoke(...)` callers pass already-typed values. Positional binding is unambiguous because the callee's `params:` are required to place every defaulted field after every non-defaulted one; that structural precondition is enforced upstream at `params:` parse time (see [Parameters and Frontmatter — `params:`](./frontmatter.md)), so every defaulted slot is at or after every required slot at the call site.

> <a id="inv-2"></a> **INV-2.** **Named-argument-invocation seam.**
>
> **theta 1.0 seam — named-argument invocation.** The invocation AST node carries an explicit `style: "positional" | "named"` discriminator on each argument list. theta 1.0's only invocation surface syntax is positional (`invoke("./path.theta", a, b, c)`); the deferred named-argument / `key=value` invocation surface in [Future Considerations](./future-considerations.md) is not part of the theta 1.0 grammar, so the `"positional"` arm is the only one with defined behaviour. *Non-normative implementation note.* The seam lets that deferred surface land without rewriting AST consumers; theta 1.0 expects every AST consumer (argument-binding, arity checks, the runtime trampoline, the lowered-tool-spec emitter) to switch on the discriminator exhaustively rather than assuming positional, but — mirroring [INV-1](#inv-1) — this exhaustive-switch factoring is a maintainability convention, not an observable conformance point.

> <a id="inv-3"></a> **INV-3.** **Per-call-timeout seam.**
>
> **theta 1.0 seam — per-call timeout.** The runtime-internal options record passed into the invoke primitive (carrying per-call configuration not surfaced to authors in theta 1.0 — cancellation hookup, the resolved callee handle, the typed-return schema slot) is an **open struct**, not a closed positional record. theta 1.0 reserves the right to add fields in a minor revision (e.g. a per-call timeout) without breaking call sites or test fixtures. *Non-normative implementation note.* Consumers tolerate unknown fields rather than assuming a closed positional record; this is a maintainability convention enabling the additive extension, not an observable conformance point. The seam is what allows the deferred per-call timeout extension in [Future Considerations](./future-considerations.md) to land additively.

**Argument arity.** Arity is checked **before** per-argument type checking, so an arity error is reported as such rather than as a confusing per-argument type error on the first extra slot.

- *Too few arguments* — fewer than the count of non-defaulted `params:` (defaults are defined in [Parameters and Frontmatter — `params:`](./frontmatter.md)). When the callee is statically resolvable, this is a parse error `theta/parse/invoke-arity-too-few`. Otherwise it surfaces at runtime as `Err(InvokeInfraError { cause: "validation", ... })` from the AJV check on the missing required field(s).
- *Too many arguments* — more than the total `params:` count. Always a parse error `theta/parse/invoke-arity-too-many`, even when the callee is not statically resolvable: extra positional arguments cannot be matched to any param, and there is no runtime safety net possible.

The same arity rules apply to `.theta` callable calls through `tools:` entries (see [Tool Calls — Argument shape](./tool-calls.md)). Pi tool calls take a single object argument and are unaffected.

**Cross-mode semantics.** The callee's mode controls whether it gets a fresh conversation or attaches to its caller's current conversation. The caller's mode is irrelevant to that decision — a subagent's "current conversation" is already its own private one, so a prompt-mode child writing into it stays inside that private context.

| Caller mode | Callee mode | Effect |
|---|---|---|
| prompt | prompt | Child attaches to caller's current conversation (the user's session). Child's queries are user-visible turns. |
| prompt | subagent | Child spawns a fresh isolated conversation; only the return value reaches the caller. |
| subagent | prompt | Child attaches to the caller's current conversation — which is the caller subagent's own private one. Nothing leaks to the grandparent. |
| subagent | subagent | Child spawns a fresh isolated conversation, sibling to (not nested under) the caller's. |

A `subagent fn` call (theta 1.2, [Functions — FN-8](./functions.md#fn-8)) is the inline realisation of the **prompt → subagent** and **subagent → subagent** cells: the callee body is always a fresh isolated conversation and only its return value reaches the caller. Because that body is always a subagent session (never the interleaved prompt-into-caller direction), a `subagent fn` is callable from a `mode: prompt` theta and the load-time rejection of prompt-mode *callees* (`theta/load/prompt-mode-callable`, which rejects a prompt-mode `.theta` path in a `tools:` table) does not apply to it.

**Tools and model.** The child uses *its own* frontmatter `model`, `tools`, and `system`. The caller's settings are not inherited. Same justification as for queries: tool/model/system inheritance produces surprise.

For the **prompt → prompt** cell of the cross-mode matrix above (child attaches to the caller's existing user session), the child's callable set replaces the parent's *for the duration of the child's body*: on entry the runtime snapshots the user session's ambient tool set and calls `pi.setActiveTools(childCallableSet)`; on return (or any `Err` / panic / cancellation) it restores the snapshot in a `finally` block. `invoke(...)` to a prompt-mode callee **suspends the parent's body at the call site until the child returns** — nested prompt → prompt invokes therefore execute strictly sequentially, and at most one prompt-mode body holds an open snapshot/restore window at a time. theta 1.0 exposes no parallel-`invoke` surface, so a single parent cannot spawn sibling prompt-mode children that overlap; combined with the load-time rejection of prompt-mode `.theta` callees in another theta's `tools:` (per [Parameters and Frontmatter — `tools`](./frontmatter/frontmatter-fields-a.md#tools)), the only way two prompt-mode bodies can be in flight against the same user session at all is the nested case, and the snapshot/restore stack discipline is LIFO-safe by construction. This is the same snapshot/restore protocol used per-query in [Pi Integration Contract — Tool-registration lifetime and visibility](./pi-integration-contract.md), generalised to the child's whole body, and rests on the same two Pi guarantees (synchronous-atomic `pi.setActiveTools`; per-session serial slash-command dispatch) pinned there. Nested prompt → prompt invokes stack: each level snapshots the immediately-prior set and restores it on return, so peeling back the call stack restores the user session's original ambient tool set in reverse order. The cross-extension overwrite rule from that section applies at every nesting level: if another extension calls `pi.setActiveTools` between any level's snapshot and its `finally` restore, the theta's restore overwrites that change with no diagnostic. The child's queries see only the child's callable set; on child return, the parent's queries again see the parent's callable set — the interleaving is invisible to the user other than via tool-call cards in the transcript.

For the other three cells (any callee in subagent mode, or a subagent caller invoking a prompt-mode child into the subagent's own private session), the whole callee runs inside a spawned child `pi` process (per the [Subagent state-isolation matrix](./pi-integration-contract/subagent.md#subagent-state-isolation-matrix)); the callable set is enforced by the child theta itself (optionally mirrored to the child launch's `--tools` allowlist as defence-in-depth) and dies when the child process exits; no parent-side active-set mutation is involved.

**Failures.** `invoke` returns `Result<T, QueryError>`. Invoke-specific failures surface via two `QueryError` variants in addition to the query-time variants:

- `InvokeInfraError` (wire `kind: "invoke_infra"`) covers infra-side failure *around* the callee body — load failure, parse failure, argument (input args/params) validation failure, return-value validation failure, callee panic, a subagent pre-spawn model-resolution failure, or an unexpected interpreter exception (the *runtime-defect surface* of [Errors and Results — Runtime panics](./errors-and-results/error-model.md#runtime-panics), which includes throws from the pre-evaluation dispatch-setup frame at the `invoke(...)` spawn site per [Pi Integration Contract — `ActiveInvocationRegistry`](./pi-integration-contract/active-invocation-registry.md#active-invocation-registry)). Its `cause` enum carries the sub-discriminator (per [Glossary — `cause`](./glossary.md), the uniform field name used by every `QueryError` variant whose `kind` partitions into multiple sub-arms); the full enum (including the `internal_error` arm for the runtime-defect surface) is fixed by the schema declaration in [Errors and Results — Invoke variants](./errors-and-results/queryerror-variants.md#invoke-variants).
- `InvokeCalleeError` wraps an `Err` the callee itself returned; `inner: QueryError` is the callee's original failure.

The full schema for both variants lives in [Errors and Results — QueryError variants](./errors-and-results/queryerror-variants.md#queryerror-variants).

The naming carries the partition: `InvokeInfraError` is everything the theta infrastructure raised before the callee could produce a value; `InvokeCalleeError` is the callee's own `Err` propagated through. Folding both into `QueryError` keeps the theta's error type uniform: a function or theta that mixes `?` on queries and `?` on invokes still has a single `Result<T, QueryError>` (for some `T`) return type and a single `match` shape to handle. `InvokeCalleeError.inner` is recursive — `QueryError` referencing itself is exactly the discriminated-union pattern from [Schema Declarations](./schemas.md), applied to Theta's own runtime type, and the recursion is now self-contained within the consolidated section. theta 1.0 has no configurable per-invoke timeout; cancellation is the only externally driven termination.

<a id="cycle-detection"></a>

**Cycle detection.** Invocation cycles are detected at parse time by walking the per-load-pass static-resolution graph defined under [Static resolution](#static-resolution) above. If `A.theta` invokes `B.theta` invokes `A.theta`, the second discovery is `theta/load/invocation-cycle` ("invocation cycle: A → B → A"). Unresolvable callees (those that produced a `theta/load/callee-has-errors` diagnostic) terminate their own walk arm — the walker treats the unresolvable node as a leaf and the existing diagnostic at every parent invoke or `tools:` site is the visible trace; cycles routed through such a node are not detected until the underlying file is fixed and the watcher re-walks the graph (per [Implementation Notes — Runtime](./implementation-notes.md)). Recursion through subagent-mode thetas is allowed where each invocation spawns a fresh sibling conversation; recursion through prompt-mode thetas is allowed but must terminate via control flow, just like ordinary function recursion. A `subagent fn` (theta 1.2, [Functions — FN-6](./functions.md#fn-6)) is a countable invoke frame, so a `subagent fn` that references itself — directly or transitively — forms an invocation cycle and is `theta/load/invocation-cycle`; the self-reference ban owned by [Functions — FN-6](./functions.md#fn-6) is enforced through this same cycle-detection channel.

<a id="inv-4"></a> **INV-4.** **Invocation depth bound.** The interpreter caps the nesting depth of an `invoke` chain at **32**, counting direct `invoke(...)`, `.theta` callable calls through `tools:`, cross-file `.thetalib` `fn` calls, and `subagent fn` calls (the last added in theta 1.2 — a `subagent fn` call crosses the same subagent boundary and is therefore a countable frame, see [Functions — FN-6](./functions.md#fn-6); the count is per-chain, not per-process — sibling invokes do not share budget). Exceeding the cap raises a runtime panic with code `theta/runtime/invoke-depth-exceeded`; per the panic-routing rules in [Errors and Results](./errors-and-results.md), a top-level overflow surfaces as a Pi system note and an overflow inside an invoke chain surfaces to the parent as `Err(InvokeInfraError { cause: "panic", ... })`. Recursion through cycle-detected paths is unreachable (the cycle fires earlier at parse time); the depth bound exists for legitimate-but-runaway recursive divide-and-conquer.

Depth is the count of *countable frames* on the active call chain, where a countable frame is any direct `invoke(...)` call, any `.theta` callable call dispatched through a `tools:` entry, any cross-file `.thetalib` `fn` call, or any `subagent fn` call (theta 1.2; a `subagent fn` call is a countable frame irrespective of whether caller and callee share a source file, because it always spawns a fresh subagent session) — a `.thetalib` `fn` call is *cross-file* whenever the caller resides in a different source file from the callee — whether the caller is another `.thetalib` file or a `.theta` file — and an intra-file `fn` call (caller and callee in the same `.thetalib` source file) is not a countable frame. A `fn`'s residence is its declaration site — the `.thetalib` file in which the `fn` is declared — so re-exports (`export { foo } from "./other.thetalib"`) and `as`-aliased imports per [Imports](./imports.md) do not change a `fn`'s residence for this cross-file test. The slash-invoked top-level theta is depth 0; the first such frame nested inside it is depth 1. The cap is breached when the runtime is about to push a frame that would bring the count to 33, so the legal range is 1 ≤ depth ≤ 32 and the diagnostic renders `invoke chain depth exceeded: 33 > 32` (matching [Diagnostics — code registry](./diagnostics/code-registry-parse.md#code-registry)). The counter is incremented before the child frame begins executing, so a child whose body itself overflows from depth 32 surfaces the panic at the child's very first nested `invoke`, not after some work has run.

The counter is per-chain and crosses subagent-mode boundaries unchanged. A `subagent → subagent` or `prompt → subagent` invocation does **not** reset the count: from the perspective of the cap, the subagent's spawned child `pi` process is a continuation of the same call chain even though it owns a fresh conversation. The `subagent` carve-outs in the cross-mode matrix concern *conversation isolation*, not *call-chain accounting*. Two concurrent invokes spawned from the same parent (the "sibling invokes do not share budget" rule above) — only reachable when both children resolve to subagent mode, since sibling prompt-mode children are forbidden by construction (theta 1.0 exposes no parallel-`invoke` surface, and `invoke(...)` to a prompt-mode callee suspends the parent until the child returns per *Cross-mode semantics* above) — remain independent regardless of mode; the existing per-chain definition already covers that case. For subagent-mode children the counter crosses the child-process boundary as **wire-level data**, and this carriage is **normative**: it is carried in an environment variable on the child `pi` process launch (alongside the `PI_THETA_SUBAGENT_ROOT=<slug>` regime marker, per [Subagent — Subagent-root regime (PIC-58)](./pi-integration-contract/subagent.md#pic-58) and the explicitly-marshalled enumeration in the [state-isolation matrix](./pi-integration-contract/subagent.md#subagent-state-isolation-matrix)), read by the theta extension loaded inside the child, and thereby continues the per-chain count across the process boundary. This **reverses** the former in-process invariant — the counter is now part of the wire-level data passed to the spawned child (carried in the child's environment), rather than a runtime-side invariant held only in the parent's invoke trampoline and absent from any wire surface. Because RFC 0006 makes every nesting level its own process, the existing per-chain depth cap is thereby also the **process-tree depth bound** — no second cap is introduced. A `theta/runtime/invoke-depth-exceeded` raised inside a subagent-mode child trips in whichever process hits the cap and still surfaces to its `invoke` parent as `Err(InvokeInfraError { cause: "panic", ... })` through the child's return envelope (per [INV-5](#inv-5) / [PIC-59](./pi-integration-contract/subagent.md#pic-59)) — the contract changes only which frame trips the cap, not the routing of the resulting panic.
