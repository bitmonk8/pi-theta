**GOV-4 (per-row invariant).** The prefix → page binding is append-only and immutable: once a prefix is allocated to a page (per GOV-7 *Add* or GOV-7 *Narrative-to-normative promotion*), the binding never changes silently and the prefix is never reassigned to a different page (per GOV-7 *Delete* and GOV-7 *Normative-to-narrative demotion*, retired prefixes move to the *Retired prefixes* sub-table below and are never reused). The *table rows themselves* are NOT immutable — GOV-7 *Rename* updates the Page column without changing the prefix, and GOV-7 *Delete* / *Merge* / *Normative-to-narrative demotion* remove or relocate rows. Introducing a new non-narrative page requires appending a new row whose prefix is *previously-unused* — meaning absent from both this table and the *Retired prefixes* sub-table below. The prefix → page binding is many-to-one: a single page MAY own more than one prefix (e.g. `hard-ceilings.md` owns both `CEIL` and `CIO`), in which case each prefix sits on its own row and the per-row invariants apply to each row independently. Adding a second (or further) prefix to a page that already appears in the table follows the *Add prefix to existing page* sub-clause of GOV-7 below. The live table's `Prefix` column is a key: duplicate live `Prefix` values, and any live prefix that also appears in the *Retired prefixes* sub-table, are spec defects.

**GOV-5 (disjoint-prefix invariant).** Each row's `Prefix` value is a complete identifier token, not a search prefix. REQ-ID prefixes are word-boundary-distinct: every REQ-ID anchor and back-reference in the spec corpus matches the token regex `\b<PREFIX>-[1-9][0-9]*\b`, and two prefixes that share a common substring (e.g. `BNDS` / `BNDR`) are distinct tokens — neither is a sub-prefix of the other.

**GOV-6 (table-completeness invariant).** At every commit on `main`, the set of prefixes appearing in REQ-IDs across `spec_topics/*.md` is a subset of the union of (live prefix table, *Retired prefixes* sub-table).

**GOV-7 (mutation procedures).**

- **Add.** New page → append a row with a previously-unused prefix.
- **Add prefix to existing page.** Existing page acquires a second (or further) prefix → append an additional row, whose Page cell repeats the existing page's filename and whose Prefix cell is *previously-unused* on the same terms as *Add*. The pre-existing row(s) for that page stay as they are (per GOV-4's prefix → page binding stability — appending a new prefix row for an existing page is not one of the GOV-7 mutations *Rename* / *Delete* / *Merge* / *Normative-to-narrative demotion* that GOV-4 enumerates as the exhaustive set of operations permitted to modify or relocate existing rows, so the existing rows' Prefix and Page columns are untouched).
- **Rename.** Prefix follows the page; the row's Page column updates, the Prefix column does not. Existing in-page anchors are not rewritten.
- **Delete.** The row is moved from the live table to the Retired prefixes sub-table. The prefix MUST NOT be reused.
- **Merge.** When `<absorbed-page>` is merged into `<surviving-page>`:
    - Every live REQ-ID on the absorbed page undergoes a GOV-8 *Merge* into a freshly allocated ID under the surviving prefix, appended at the surviving page's tail. The absorbed-page IDs retire on the absorbed page (its `## Retired REQ-IDs` section gains one row per merged ID) before the file is deleted.
    - The absorbed page's pre-existing `## Retired REQ-IDs` rows transplant verbatim into the surviving page's `## Retired REQ-IDs` section.
    - The absorbed prefix moves to the *Retired prefixes* sub-table; the *Formerly* cell records `<absorbed-page> (merged into <surviving-page> at <sha>)`.
    - The absorbed page file is deleted in the same commit (otherwise the prefix-table-completeness gate sees both a retired prefix and a live page carrying it).
- **Narrative-to-normative promotion.** Replace the `(no IDs — narrative)` cell with a freshly allocated prefix in the same edit that introduces the first obligation.
- **Normative-to-narrative demotion.** When a page that previously owned a prefix becomes pure narrative (every claim it carried is canonically owned elsewhere): (a) move its prefix from the live table to the *Retired prefixes* sub-table per GOV-7 *Delete*; (b) append a new live table row for the page carrying `(no IDs — narrative)`. Re-promotion later (per *Narrative-to-normative promotion* above) requires a *fresh* prefix — the original is retired and immutable per GOV-4 / GOV-5, and the *Retired prefixes* sub-table is itself append-only.

**GOV-8 (REQ-ID lifecycle).**

*Scope note (non-normative).* GOV-8 is a *bookkeeping* discipline: it pins how a substantive edit is recorded, not whether two loom 1.x releases produce the same output for the same `.loom` / `.warp` source file. The behavioural loom 1.x stability statement (per **GOV-15** at the page tail) is intentionally *not* derived from this rule.

<a id="gov-14"></a> **GOV-14 (review posture on the loom 1.x-equivalence release-process goal).** Reviews of `spec.md` and `spec_topics/*.md` SHOULD NOT cite GOV-8 as a substitute for the loom 1.x source-language equivalence release-process goal recorded at [GOV-15](./req-id-prefix-table-retired.md#gov-15) and SHOULD NOT re-raise the absence of an automated equivalence gate as a loom 1.0 correctness finding. The loom 1.0 release decision treats the absence of an automated equivalence gate as a recorded scope choice, not a defect.

- **Split.** When one rule splits into N rules, the original ID retires and N fresh IDs are appended at the page's tail.
- **Merge.** When N rules merge into one, all N source IDs retire and one fresh ID is appended at the page's tail.
- **Deletion.** Rule removed without replacement → ID retires; the prefix-position number MUST NOT be reused.
- **Pure rewording.** Typo fixes, sentence restructuring, link updates leave the ID unchanged. A change that alters which inputs are accepted, which outputs are produced, which diagnostics fire, or which invariants hold is substantive and MUST be modelled as a split, merge, or deletion-plus-add — never as an in-place edit. This boundary is enforced by review on the PR that touches a `**PREFIX-N.**` paragraph; no CI gate detects substantive in-place edits, and none is planned, because semantic equivalence between two prose paragraphs is not mechanically decidable. Worked examples:
  - Modal strengthening or weakening (`should` → `MUST`, `MUST` → `should`, `MAY` → `MUST`, etc.) inside a `**PREFIX-N.**` paragraph is **substantive** — the obligation's force has shifted even when the inputs and outputs at the moment of the edit appear unchanged. Treat as a split-or-deletion-plus-add. (This is the most commonly missed case in review; flag it first.)
  - Adding or removing a diagnostic code on a `**PREFIX-N.**` paragraph is **substantive** — it alters which diagnostics fire, per the test above. Treat as a split-or-deletion-plus-add. The new code's diagnostic-registry entry is a separate, additive obligation and does NOT discharge the GOV-8 retirement requirement.
  - Editing a non-normative example (a fenced code block, parenthetical illustration, or worked sample) attached to a `**PREFIX-N.**` paragraph is **pure rewording**, provided the rule's normative sentences are unchanged. Examples do not constrain implementations. When in doubt about whether a sentence is normative or illustrative, retire-and-re-add.

All retirements (per GOV-7 *Delete* / *Merge* and per GOV-8 *Split* / *Merge* / *Deletion*) MUST be recorded:

- **Per-prefix retirements** appear in the *Retired prefixes* sub-table immediately below.
- **Per-ID retirements** appear in a trailing `## Retired REQ-IDs` section on each non-narrative page. After a GOV-7 *Merge*, the surviving page's `## Retired REQ-IDs` section accumulates rows under both the surviving and the absorbed prefixes (per the *Merge* sub-bullet above); the absorbed page's section is deleted with its file.

**GOV-9 (cross-link form).** Each spec page that depends on a normative rule from another topic page MUST either state the rule locally or reference it by a markdown link whose target is the specific REQ-ID anchor (`#prefix-n`) of the depended-upon rule. Where the depended-upon page is pure-narrative (its row in the [REQ-ID prefix table](./req-id-prefix-table-active-a.md#req-id-prefix-table) carries the `(no IDs — narrative)` cell per GOV-3), a section-level link to the relevant heading on that page suffices. Every REQ-ID anchor on a non-narrative spec page MUST resolve as a `#prefix-n` URL fragment, so that any cross-page link of the form `…#prefix-n` formed per the preceding sentence resolves to the depended-upon rule's anchor site; this universal-resolvability obligation is the contract that GOV-1 *Required HTML-anchor contexts* and *Dual-form layout* deliver against. Pre-existing non-conformant anchor sites — sites whose dual-form HTML anchor is absent and therefore do not yet expose a `#prefix-n` fragment — are standing spec defects on the same per-page terms as GOV-1 *Per-page progressive normalisation* (which owns this rule's delivery mechanism): the defect set drains progressively as pages are edited, and no corpus-wide normalisation commit is reserved. An obligation site that carries no REQ-ID anchor at all — distinct from a coined anchor that is merely missing its dual-form HTML token — is likewise a standing spec defect under this contract, drained progressively by [GOV-22](#gov-22) on the same per-commit terms.

<a id="gov-22"></a> **GOV-22 (progressive coinage of un-anchored obligations).** GOV-1 *Per-page progressive normalisation* upgrades existing REQ-ID anchor sites to dual-form, but its trigger fires only where the commit touches an existing REQ-ID anchor token; a non-narrative page that carries no REQ-ID anchor, and any individual obligation that has never been coined as a REQ-ID, are therefore unreachable by that trigger, leaving GOV-9's `#prefix-n` cross-link contract permanently unsatisfiable for them. This rule drains that residue on the same per-commit, progressive terms — no corpus-wide coinage commit is reserved. At every commit on `main` — taking the first-parent diff for merge or squash commits and the diff against the sole parent for single-parent commits, on the same terms as GOV-1 *Per-page progressive normalisation* — where the commit's diff adds a *defining obligation site*, or edits the normative-modal content of an existing one, on a non-narrative spec page and that site does not carry a co-located REQ-ID anchor in the post-diff text, the committer MUST coin a `PREFIX-N` REQ-ID anchor at that site in the same commit, in the dual-form layout GOV-1 mandates and under the page's prefix already registered in the [REQ-ID prefix table](./req-id-prefix-table-active-a.md#req-id-prefix-table); the numeric tail is allocated per GOV-3 (the next free integer under that prefix, numbering never collapsing to fill holes). A *defining obligation site* is a source line that states an obligation the page owns and carries an RFC-2119 normative-modal token (`MUST`, `MUST NOT`, `SHALL`, `SHALL NOT`, `SHOULD`, `SHOULD NOT`), excluding tokens inside fenced code blocks, inline code spans, and HTML comments (the GOV-3 exclusion list) and excluding back-references to obligations another page owns. Editing a defining obligation site that already carries its own co-located REQ-ID anchor does not trigger this rule — such an edit is governed by GOV-8's lifecycle and by GOV-1 *Per-page progressive normalisation*. Pre-existing un-anchored obligation sites on a page whose diff does not touch them under the trigger predicate above are standing spec defects not bound by this MUST on the current commit; the defect set drains progressively as pages are edited, exactly as GOV-1 *Per-page progressive normalisation* states for non-dual-form anchor sites. This rule binds the spec corpus itself (arm (b) per [GOV-18 arm (b)](./corpus-direction-and-scope.md#gov-18-arm-b)); the committer and the reviewer of the triggering PR are its satisfaction mechanism, and no CI gate is reserved.

The *single-source-page* shape enumerates a closed set of items all sourced (for aggregator lock-step purposes) from a single sub-bucket on a single topic page — the *aggregator-source page*, which is distinct from the per-item *normative owners* of the underlying dispositions and may differ from them (the per-item normative owners may live on other topic pages, cited in the sub-bucket's per-item `Recorded at:` lines or equivalent) (currently: the loom 1.0 non-goals aggregator paragraph on `spec.md`, whose aggregator-source page is [`future-considerations.md`](../future-considerations/model-changes-and-non-goals.md#v1-non-goals) — it enumerates the no-seam dispositions by name and forward-links the bullet list there, while the per-bullet normative owners cited in each `Recorded at:` line live on their respective topic pages; only that one sub-bucket on `future-considerations.md` is in lock-step scope — sibling sub-buckets (Tooling deferrals, Surface extensions, Model-level changes) on the same page are not; a *navigation-pointer cross-link* — defined as a cross-link whose role is wayfinding (pointing the reader at related material) rather than item enumeration; it does not name an item that is under this rule's lock-step scope — from the aggregator into one of those out-of-scope sibling sub-buckets (or to any other anchor outside the aggregator-source sub-bucket) carries no anchor-stability obligation under this rule, so a future PR that renames such a sibling sub-bucket's heading silently breaks the cross-link without a normative trigger — this is a documentation defect, not a normative defect, and the rename-PR author's responsibility (i.e. the PR that renames the heading SHOULD repoint each affected aggregator-side cross-link in the same commit, caught by reviewer / grep)). The lock-step parity criterion for the *single-source-page* shape is *bidirectional referent identity*: the aggregator's enumeration MUST be a 1:1 referent-identity mapping with the aggregator-source sub-bucket — every bullet in the source sub-bucket has exactly one item in the aggregator that refers to the same disposition, and every item in the aggregator has exactly one bullet in the source sub-bucket that refers to the same disposition. Aggregator-side stylistic rewording (prose phrasing, bullet-vs-inline rendering, ordering) is permitted if it preserves that 1:1 referent mapping — bullet-name string identity, item-count rendering parity, and content parity beyond the referent are not required (this mirrors the GOV-8 *Pure rewording* limit cited in the undecidability disclaimer below, and shares its consequence: whether a given paraphrase preserves the referent is a reviewer judgement, not a mechanical check). The bidirectional shape makes this parity criterion mutually reinforcing with the add/remove/rename lock-step trigger below: a PR that adds, removes, or renames a source-side bullet without the matching aggregator-side update breaks the 1:1 mapping and so violates this criterion in addition to the lock-step trigger. Aggregator paragraphs are maintained in lock-step with their source pages by editorial convention: a PR that adds, removes, or renames an item on a source page (e.g. introduces a fifth hard ceiling on a topic page, retires an SDK capability, splits a Host runtime precondition, adds a sixth `.warp` permitted form, adds, removes, or renames a topic-page obligation enumerated in the Session Model section's forward-link inventory above, or adds, removes, or renames a bullet under the loom 1.0 non-goals list on `future-considerations.md`) MUST update the corresponding `spec.md` aggregator in the same commit. Drift between an aggregator and its sources is a documentation defect, not a correctness defect; reviewers SHOULD flag it on the PR that introduced the drift, and the spec corpus reserves no mechanical witness for such drift (semantic equivalence between an aggregator paragraph and a set of topic-page paragraphs is not mechanically decidable, mirroring the GOV-8 *Pure rewording* limit). Aggregators of the *integer-count* shape (paragraphs and, in the single-bullet case noted within the enumeration below, a single Scope bullet — taking the shape of an integer count of source items; currently: the 12 typed/structural seams (the lone *single-bullet* aggregator in this enumeration — its host is the *Forward-compatibility seams* Scope bullet under [`spec.md` — Scope](../../spec/overview-and-orientation.md#scope), not an aggregator paragraph; the host bullet is simultaneously one of the source items counted by the *two Scope bullets* aggregator listed below in this enumeration — the dual role poses no integer-count ambiguity because each aggregator is evaluated independently against its own source set; this entry is therefore declared as a GOV-12 aggregator by its membership in this integer-count `currently:` enumeration alone and is intentionally not registered under either of the *multi-source-page* or *single-source-page* shapes enumerated earlier in this paragraph), the four hard runtime ceilings, the five `.warp` permitted top-level forms, the eight loom 1.0 non-goals, the two Scope bullets, and the three Host runtime preconditions) carry one additional spec-side invariant beyond the lock-step MUST above: each such aggregator MUST preserve a parseable integer literal — in some parseable form (e.g. decimal-digit `13` or English-number-word `thirteen`) — anywhere within the same aggregator paragraph (or, for single-bullet aggregators, the same bullet); the literal's sentence position, list rendering, and ordering relative to other prose remain reworkable under the aggregator-side rewording permission stated for the *single-source-page* shape above, but the literal MUST survive in parseable form somewhere in that aggregator's text. For each integer-count aggregator, the *source set* is the closed enumeration that the aggregator's host paragraph on `spec.md` either contains directly or forward-links to; concretely, for the integer-count aggregators currently enumerated above: the *two Scope bullets* are the bullet list under [`spec.md` — Scope](../../spec/overview-and-orientation.md#scope); the *three Host runtime preconditions* are the three numbered host preconditions enumerated under the *Host runtime* paragraph at [`spec.md` — Prerequisites](../../spec/overview-and-orientation.md#prerequisites); the *four hard runtime ceilings* are the four numbered items in the list under [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings); the *five `.warp` permitted top-level forms* are the forms enumerated at [Imports — Permitted top-level forms](../imports.md#permitted-top-level-forms); the *eight loom 1.0 non-goals* are the bullets at [Future Considerations — loom 1.0 non-goals](../future-considerations/model-changes-and-non-goals.md#v1-non-goals); the *12 typed/structural seams* are the seams inventoried at [Future Considerations — Surface extensions](../future-considerations/surface-extensions.md#surface-extensions-v1-leaves-a-seam), each pinned by a `> **loom 1.0 seam — <name>.**` blockquote on its owning topic page per the *Forward-compatibility seams* Scope bullet on `spec.md`; The literal's numeric value MUST equal the count of items in the aggregator's source set at every commit on `main`. Paraphrasing the literal away entirely (e.g. rewording "the two Scope bullets" to "the Scope bullets" with no surviving parseable integer literal anywhere in the aggregator), or letting the literal's value drift from the source-set count, is a spec defect on the same terms as any other arm (b) invariant violation. The literal-vs-source-count parity property is mechanically decidable (counting items in a fixed source set against a literal integer); per [GOV-18 arm (b)](./corpus-direction-and-scope.md#gov-18-arm-b) the spec corpus pins the invariant but does not specify the auditing machinery, which lives outside the spec corpus. Pure renames do not change the source-item count and so do not interact with the literal-preservation MUST; the *rename* arm of the lock-step MUST remains in force on the same editorial-convention basis as for non-integer-count aggregators, with no mechanical backstop reserved within the spec corpus. Aggregators whose count is already mechanically enforced by a leaf-owned in-code constant (currently the SDK-capabilities aggregator alone, enforced by [`CAPABILITY_OBLIGATIONS.length === 7`](../pi-integration-contract/version-bump-step2.md#bump-step-2-positive) against the leaf-owned constant per [PIC — SDK capability inventory](../pi-integration-contract/inventory-audit-intro.md#sdk-capability-inventory) — the markdown enumeration of items 1–7 there and the [`spec.md` Prerequisites](../../spec/overview-and-orientation.md#prerequisites) cross-reference list are maintained by editorial convention per that anchor; additional aggregators may be added to this in-code-constant carve-out class only via a substantive GOV-12 amendment that names the aggregator and its in-code-constant witness, on the same `currently:` lock-step convention used by the shape enumerations above — a leaf that introduces a new `length === N`-style check elsewhere does NOT thereby remove its aggregator from the integer-count invariant's scope until such an amendment lands) are exempt from the literal-preservation MUST above; the in-code-constant assertion takes its place as the witness of the count's truth on the implementation side, while the spec corpus retains no further spec-side invariant on the count beyond editorial convention. The orientation prose literal that names the count on `spec.md` (e.g. "seven SDK capabilities") is likewise out of scope of the literal-preservation MUST above (which binds only aggregators of the *integer-count* shape, not in-code-constant carve-out aggregators) and remains under editorial convention on the same terms as the lock-step MUST and reviewer-SHOULD-flag obligation under the undecidability disclaimer above — a PR that paraphrases the literal away (e.g. rewording "seven SDK capabilities" to "the SDK capabilities") is a documentation defect that reviewers SHOULD flag. This orientation-vs-constant drift gap is intentional under this rule. Separately, the Session Model section's forward-link inventory (registered above under the *multi-source-page* shape) is also out of scope of the literal-preservation MUST, on different grounds: it is an open-ended forward-link inventory of topic-page obligations rather than a counted set, and so does not surface a single-integer cardinality in any lexical slot to preserve; its lock-step MUST remains reviewer-enforced under the undecidability disclaimer above. This rule is the single canonical home for the lock-step convention; previous load-bearing MUSTs scattered across `spec.md` aggregator paragraphs (most notably the hard-runtime-ceilings migration MUST) are removed in favour of citing GOV-12.
