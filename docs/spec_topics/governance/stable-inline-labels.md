# Stable inline labels

## Stable inline labels

<a id="gov-16"></a> **GOV-16 (stable inline labels — second governed identifier class).** Some normative obligations carry a *stable inline label* — a cross-page-citable identifier that is not a REQ-ID. Inline labels are a second governed identifier class with their own per-page registry, anchor form, lifecycle, and retirement table; the rule is structured to mirror GOV-1 / GOV-3 / GOV-8 / GOV-9 so that the two classes are gated by parallel machinery. Three inline-label prefixes are currently registered: `HC3` (the per-class binder retry sub-obligations `HC3-a` … `HC3-e`) and `NOCEIL` (the four enumerated non-existence claims `NOCEIL-1` … `NOCEIL-4`) on [`hard-ceilings.md`](../hard-ceilings.md), and `SNK` (the per-`kind` loom-system-note rendering rows `SNK-a` … `SNK-k` under [SLSH-4](../slash-invocation.md#slsh-4)) on [`slash-invocation.md`](../slash-invocation.md). The `CIO-1` … `CIO-6` labels on the same page are *not* inline labels under this rule — they are full REQ-IDs under the `CIO` row of the [REQ-ID prefix table](./req-id-prefix-table-active-a.md#req-id-prefix-table) (co-owned with `CEIL` per GOV-24) and are governed by GOV-1 / GOV-3 / GOV-8 / GOV-9 directly.

- **Canonical form.** Each inline label's defining anchor is the inline `**PREFIX-<tail>.**` marker, where `<tail>` is either `[1-9][0-9]*` (numeric tail, e.g. `NOCEIL-3`) or a single lowercase ASCII letter `[a-z]` (alphabetic tail, e.g. `HC3-a`). The bold-with-period decoration is normative on the same terms as for REQ-IDs (per GOV-1 *Canonical form*) — it distinguishes anchor sites from back-references and is part of the anchor form.
- **GOV-1 dual-form layout applies.** Inline-label anchor sites participate in GOV-1's *Required HTML-anchor contexts* and *Dual-form layout* obligations on identical terms — the witness regex `<a id="prefix-n"></a> \*\*PREFIX-N\.\*\*` is read as `<a id="prefix-tail"></a> \*\*PREFIX-<tail>\.\*\*` with `<tail>` per the *Canonical form* above and rendered lowercase in the `id` attribute to match GitHub's slug convention. GOV-1's *Per-page progressive normalisation* clause also applies to existing inline-label anchor sites on the same terms. The two identifier classes are gated independently per page: a commit's diff that touches an existing inline-label anchor's tokens (the inline `**PREFIX-<tail>.**` marker bytes or the dual-form `<a id="prefix-tail"></a>` HTML token co-located with it) triggers the *Per-page progressive normalisation* MUST only across inline-label anchor sites on that page, and a commit's diff that touches an existing REQ-ID anchor's tokens triggers it only across REQ-ID anchor sites — neither class's edit pulls the other class's pre-existing anchor sites into the per-page sweep.
- **Per-page inline-label prefix table.** Inline-label prefixes are registered per page below; the table is append-only and follows GOV-24 / GOV-5 / GOV-7 on the same terms as the REQ-ID prefix table — live inline-label prefixes are disjoint from each other and from the live + retired union of REQ-ID prefixes, and a retired inline-label prefix is never reused. Inline-label prefixes are uppercase ASCII letters of length 2–6, optionally with ASCII digits in non-leading positions (e.g. `HC3`, `XY7`, `NOCEIL`) — matched by `[A-Z][A-Z0-9]{1,5}`; the upper bound widens beyond REQ-IDs' 4 to admit `NOCEIL`, and admitting digits in non-leading positions is a deliberate divergence from GOV-3's letters-only REQ-ID prefix grammar (`[A-Z]{2,4}`) so that digit-bearing prefixes such as `HC3` can be registered. Matching is case-sensitive per GOV-3.

  | Page | Inline-label prefix | Tail form |
  |---|---|---|
  | `hard-ceilings.md` | `HC3` | `alphabetic` |
  | `hard-ceilings.md` | `NOCEIL` | `numeric` |
  | `slash-invocation.md` | `SNK` | `alphabetic` |

  The `Tail form` column pins each inline-label prefix to exactly one tail shape — `numeric` (matching `[1-9][0-9]*`) or `alphabetic` (matching `[a-z]`) — drawn from the two tail forms enumerated in *Canonical form* above. The column is normative: it is consulted by the *Third extractor* and the *Inline-label grammar and closure invariant* below to bind each prefix to one tail form, and is part of the GOV-24 / GOV-7 append-only table contract on the same terms as the `Page` and `Inline-label prefix` columns. A row's `Tail form` cell is fixed at the prefix's allocation moment and is not silently re-pointed; switching a live inline-label prefix from one tail form to the other is a GOV-7 *Delete* of the original prefix plus a GOV-7 *Add* of a fresh prefix under the new tail form.

  *Retired inline-label prefixes.* Empty in loom 1.0; the sub-table is append-only on the same terms as the *Retired prefixes* sub-table for REQ-IDs and is materialised here when the first inline-label prefix retires.

- **Third extractor.** Inline labels in the spec corpus match the regex `\b(<alphabetic-tail-inline-label-prefix-alternation>)-[a-z]\b|\b(<numeric-tail-inline-label-prefix-alternation>)-[1-9][0-9]*\b`, a two-armed alternation whose prefix groups are partitioned by the `Tail form` column of the per-page inline-label prefix table above. `<alphabetic-tail-inline-label-prefix-alternation>` is built at evaluation time from exactly those live inline-label prefixes whose row carries `Tail form` = `alphabetic` (currently `HC3` and `SNK`); `<numeric-tail-inline-label-prefix-alternation>` is built at evaluation time from exactly those live inline-label prefixes whose row carries `Tail form` = `numeric` (currently `NOCEIL`). Each arm consumes only its registered prefixes paired with its registered tail form, so cross-class malformed shapes (e.g. `HC3-7`, `NOCEIL-a`) do not match — discharging the `Tail form` column's "consulted by the *Third extractor*" claim above. Neither alternation is hard-coded. Leading zeros in the numeric tail are forbidden on the same terms as GOV-3. Exclusion stripping (fenced code blocks, HTML comments, inline code spans) applies on the same terms as GOV-3.
- **Inline-label grammar and closure invariant.** A spec-corpus token of shape `<prefix>-<tail>` is a well-formed inline label iff `<prefix>` matches `\b[A-Z][A-Z0-9]{1,5}\b`, `<prefix>` appears in the per-page inline-label prefix table above (live or retired — retired inline-label prefixes carry the same pinned tail form they held when live), AND `<tail>` matches the prefix's pinned tail form per the `Tail form` column. The prefix grammar widens beyond GOV-3's REQ-ID prefix grammar (`[A-Z]{2,4}`) by lengthening to 2–6 to admit `NOCEIL` and admitting ASCII digits in non-leading positions to admit `HC3`; the tail admits the single-lowercase-letter form alongside the numeric form to admit `HC3-a` … `HC3-e`. GOV-3's *Unknown-prefix closure invariant* extends to the inline-label class on the same terms: every token in the exclusion-stripped spec corpus that matches the widened detector regex `\b[A-Z][A-Z0-9]{1,5}-(?:[1-9][0-9]*|[a-z])\b` MUST be either (a) a well-formed REQ-ID per GOV-3 (prefix in the REQ-ID prefix-table union, numeric tail), or (b) a well-formed inline label per the preceding sentences (prefix in the inline-label prefix-table union, tail matching the prefix's pinned tail form). Cross-class malformed tokens (e.g. `HC3-7` with `HC3`'s pinned tail form `alphabetic`, `NOCEIL-a` with `NOCEIL`'s pinned `numeric`, or a REQ-ID-table prefix paired with a single-lowercase-letter tail such as `BNDR-a`) violate the invariant on the same terms as an unenumerated prefix. The spec corpus pins this grammar and closure invariant as arm (b) content; tooling that needs to distinguish well-formed identifiers from neither-arm tokens reads this grammar and the per-page tables, per [GOV-18 arm (b)](./corpus-direction-and-scope.md#gov-18-arm-b).
- **Lifecycle.** Inline labels follow the GOV-8 lifecycle (*Split*, *Merge*, *Deletion*, *Pure rewording*) on identical terms — substantive edits MUST be modelled as a split, merge, or deletion-plus-add; pure rewording leaves the label unchanged. The GOV-8 *Pure rewording* worked examples (modal strengthening / weakening, diagnostic-code addition / removal, edits to attached non-normative examples) apply verbatim to inline labels.
- **Retirement section.** All inline-label retirements MUST be recorded in a trailing `## Retired inline labels` section on each page that owns inline-label prefixes, structurally parallel to the `## Retired REQ-IDs` section.
- **Cross-link form.** GOV-9's cross-link contract applies to inline labels on identical terms — every cross-page reference to an inline label MUST resolve as a `#prefix-tail` URL fragment (rendered lowercase per *GOV-1 dual-form layout applies* above), so any cross-page link of the form `…#hc3-a` or `…#noceil-3` resolves to the depended-upon label's anchor site.
- **GOV-30 lock-step coverage.** GOV-30's lock-step convention names `HC3-a`…`HC3-e`, `CIO-1`…`CIO-6`, and `NOCEIL-1`…`NOCEIL-4` explicitly. That enumeration is correct under this rule and stays as-written; a PR that adds, removes, or renames any of the identifiers enumerated above on `hard-ceilings.md` (whether an inline label under this rule or a `CIO-N` REQ-ID under GOV-1 / GOV-3 / GOV-8 / GOV-9) MUST update the corresponding `spec.md` aggregator in the same commit per GOV-30.
