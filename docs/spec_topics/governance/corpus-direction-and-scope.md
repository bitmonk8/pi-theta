# Corpus direction and scope

## Corpus direction and binding scope

<a id="gov-17"></a> **GOV-17 (corpus direction).** The spec corpus comprises `docs/spec.md` and `docs/spec_topics/*.md`. A *cross-reference* from one document to another is a CommonMark link (per the CommonMark Spec §6.3 Links; subsequent uses of "Markdown link" in this rule denote a CommonMark link) in the first document's body — either the inline form `[text](path)` or the reference-style form `[text]: path` — whose `path` (parsed per the URL/URI grammar of RFC 3986 §3.3 (path), §3.4 (query), §3.5 (fragment), with any `#fragment` and `?query` components stripped before path resolution), resolved relative to the linking document's directory, is the filesystem path of the second document. Back-ticked path mentions inside `code spans`, prose filename mentions in running text, and any byte-level substring match outside Markdown-link syntax are NOT cross-references under this rule; this includes the back-ticked path mentions in GOV-17's own body and the back-ticked path mentions cited as per-rule reasons in the *Retired REQ-IDs* sub-table. A document or resource outside the spec corpus is a *dependency* of the spec corpus iff its body contains no cross-reference into the spec corpus, and a *dependent* iff it does. The spec corpus MAY cross-reference its dependencies and MUST NOT cross-reference its dependents — equivalent to "no circular Markdown-link cross-references between the spec corpus and any other document". The dependent check is operational on the candidate target's side: enumerate the Markdown links in the target's body, resolve each `path` against the target's directory, and test whether the resolved path is `docs/spec.md` or matches `docs/spec_topics/*.md`; any such link makes the target a dependent. (Non-normative aid: `grep -nE '\]\([^)]*spec(\.md|_topics/)' <target>` flags candidate Markdown-link callsites for inspection; a precise check still requires resolving each link's `path` against the target's directory, since the regex over-matches incidental substrings and the relative-path forms `spec.md` / `./spec.md` / `../spec.md` / `../../docs/spec.md` all resolve to the same spec-corpus files.) The currently-known dependents, all forbidden as cross-reference targets from the spec corpus, are: the plan documents (`docs/plan.md` and `docs/plan_topics/*.md`), the repo-level `README.md`, and `CHANGELOG.md`. External non-Markdown resources (third-party documentation rendered as HTML, package documentation under `node_modules/`, third-party websites, PDFs, plain-text files) carry no Markdown-link `[text](path)` / `[text]: path` syntax in their bodies and are therefore vacuously dependencies under this rule — the operational dependent check above (which enumerates Markdown links in the target's body) returns empty on them by construction, so the spec MAY cite them without running the check. External Markdown resources (e.g. a third-party `.md` file reachable through the citing link's resolved `path`) ARE subject to the operational dependent check on the same terms as in-repo Markdown. The rule has no carve-out for forward-reference, future-tense, or hypothetical citations: a dependent is forbidden as a cross-reference target regardless of the citing prose's modality.

*Triggering and scope (non-normative).* GOV-17's MUST-NOT is gated per-commit on the same per-page terms as GOV-1 *Per-page progressive normalisation*: it is evaluated against cross-references the commit's first-parent diff adds, edits, or moves in the spec corpus, not against the spec corpus as a whole on every commit. The rule does NOT impose a corpus-wide audit obligation on every commit; pre-existing in-spec cross-references the commit does not touch are not gated by this rule on the current commit, and a pre-existing in-spec cross-reference that becomes non-conformant solely by a change outside the spec corpus (e.g. a commit on `docs/plan.md` or `docs/plan_topics/*.md` that adds a link into the spec, newly classifying its source document as a dependent under the operational dependent check above) is a standing spec defect actionable when a future commit touches the in-spec cross-reference's site. The rule pins the cross-reference-direction invariant but does NOT specify auditing machinery; per [GOV-18 arm (b)](#gov-18-arm-b), the audit lives outside the spec corpus.

<a id="gov-18"></a> **GOV-18 (binding scope).** The spec corpus's normative obligations bind exactly two parties:

- <a id="gov-18-arm-a"></a> **Arm (a) — the implementation target.** The loom runtime, the binder, the type system, the Pi integration, and any other software whose behaviour the spec exists to constrain. *Operational test for the "any other software" tail.* A program is arm (a) iff its externally-observable behaviour on `.loom` or `.warp` inputs is described by the spec; programs whose behaviour on `.loom` / `.warp` inputs the spec does not describe fall outside arms (a) and (b), regardless of where they ship. Borderline cases the test resolves include software shipped inside the loom repo alongside the runtime (e.g. a `loom doctor` sub-command, a build-time validator embedded in the loom binary, a developer-tooling library invoked by the loom runtime) — arm (a) when the test holds, third-party tooling otherwise. A normative obligation whose subject is one of these is spec content.
- <a id="gov-18-arm-b"></a> **Arm (b) — the spec corpus itself.** Invariants about the form, structure, and content of `docs/spec.md` and `docs/spec_topics/*.md` — including REQ-ID grammars, anchor placement, prefix-table closure, retirement procedures, and cross-link form. A normative obligation whose subject is the spec corpus is self-binding and is spec content.

<a id="gov-18-binding-prohibition"></a> The spec corpus MUST NOT bind any third party — including but not limited to documentation tooling, CI gates, coverage matrices, REQ-ID extractors, doc generators, the project `README.md`, the `CHANGELOG.md`, or any other document or program outside arms (a) and (b). Where third-party tooling needs to consume a published spec invariant, the spec publishes the invariant as a property of its own content (arm (b)); this rule takes no position on whether or how the third party records any obligations of its own. A spec rule whose pre-diff wording cited a verification venue (e.g. "the coverage-matrix gate enforces this", "the anchor-insertion CI gate") loses that citation under this rule; the bare invariant survives as arm (b) content, and auditing the invariant is third-party tooling's responsibility carried outside the spec corpus. No verifier-replacement obligation is created by this rule.

<a id="gov-18-internals-prohibition"></a> The spec corpus MAY name a downstream consumer in implementation-neutral terms (e.g. "automated tooling that maps REQ-IDs to implementation work") to anchor an obligation's *purpose*; it MUST NOT specify the consumer's internals, workflow, configuration, or failure surface.

Worked examples — each retiring or surviving rule classified per the arm-membership test:

- **GOV-1 (REQ-ID anchor placement) — arm (b).** Constrains the spec corpus's own anchor sites; survives.
- **GOV-5 (disjoint-prefix invariant) — arm (b).** Constrains the spec corpus's own identifier-token grammar; survives as reshaped (the consumer-side MUST is rewritten as a spec-side invariant in the version of GOV-5 below).
- **Retired GOV-2.** Bound a downstream coverage-matrix tracker to map every REQ-ID to a closing leaf and exit non-zero on unmapped IDs; the tracker is neither the implementation target (arm a) nor the spec corpus itself (arm b).
- **Retired GOV-10.** Bound an implementer's reading-scope to a downstream document's per-leaf `**Spec**` field; the implementer's reading process is neither the implementation target (arm a) nor the spec corpus itself (arm b).
- **Retired GOV-11.** Bound a downstream document's `**Spec**` field to a closure-under-cross-link obligation; the downstream document's data-structure shape is neither the implementation target (arm a) nor the spec corpus itself (arm b).

The implied-consumer edge case is permitted: a rule whose surface wording presupposes a consumer (e.g. GOV-1 presupposes a consumer that tracks REQ-IDs) but whose actual subject is the spec corpus's own content (the anchor-site form) is arm (b) and passes.
