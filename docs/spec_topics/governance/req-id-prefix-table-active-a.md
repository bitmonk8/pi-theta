# Req id prefix table active a

## REQ-ID prefix table

**GOV-1 (anchor placement).** Each spec page that carries normative obligations is assigned a stable per-page REQ-ID prefix (table below). Each non-narrative page in that table MUST carry a `PREFIX-N` anchor at the defining site of every live REQ-ID under its prefix.

- **Canonical form.** Each REQ-ID's defining anchor is the inline `**PREFIX-N.**` marker. The bold-with-period decoration is normative: it distinguishes anchor sites (`**BNDR-7.**`) from back-references (`per BNDR-7`) and is part of the anchor form. The bold-with-period bytes are the witness pattern for any third-party REQ-ID extractor; per [GOV-18 arm (b)](./corpus-direction-and-scope.md#gov-18-arm-b), this rule pins only the form of the anchor on the spec page, not the extractor's behaviour.
- **Required HTML-anchor contexts.** The `<a id="prefix-n"></a>` HTML form MUST accompany the inline marker at every REQ-ID anchor site, in each of the four enumerated contexts below. These are the cases where the inline marker is not by itself a sufficient anchor (Markdown bold-with-period either does not render, breaks the surrounding construct, or — being rendered as bold text — does not by itself produce a stable URL fragment in any common Markdown renderer):
    - inside a Markdown table cell;
    - inside an ATX heading (`#` … `######`);
    - on the line immediately preceding a fenced code block whose content is the rule's normative example;
    - in body-paragraph context, where the inline marker renders fine as bold text but does not by itself produce a stable URL fragment in any common Markdown renderer, so the HTML form is the canonical fragment source for body-paragraph REQ-IDs (the three rendering-driven contexts above each carry their own per-context placement mechanics in *Edge cases* below — heading-slug repointing for ATX headings, the table-cell leading-token rule, and the line-preceding-a-fenced-code-block placement). *Body-paragraph context* is defined operationally as the negation of the three rendering-driven contexts above: every source-line position that is neither inside a Markdown table cell, nor inside an ATX heading, nor on the line immediately preceding a fenced code block whose content is the rule's normative example. This explicitly includes prose paragraphs, top-level and nested unordered list items, ordered list items, blockquote bodies, definition-list bodies, footnote bodies, and the line *following* a fenced code block.

  The HTML form is permitted only as part of a dual-form pair (per *Dual-form layout* below — i.e. co-located on the same source line as an inline `**PREFIX-N.**` marker) in one of the four enumerated contexts above; no other source-line position permits the HTML form. A bare `<a id="prefix-n"></a>` anchor with no co-located inline marker on the same line is NOT a GOV-1 anchor. A REQ-ID inside a fenced code block, an inline code span, or an HTML comment is neither an anchor nor a back-reference and is invisible to GOV-3 extraction (per the exclusion list in GOV-3 below). *Multi-context overlap.* The dual-form anchor is required in every one of the four contexts above, so the applicability question never depends on which context a site is classified under; only the per-context *mechanics* in *Edge cases* below differ. When a REQ-ID anchor site syntactically matches more than one enumerated context (e.g. a table-cell marker whose row is followed by a fenced code block, or an ATX-heading marker whose line precedes a fenced code block), the more-specific (rendering-driven) context determines which *Edge cases* mechanics apply, in this fixed precedence order: (1) table-cell, (2) ATX-heading, (3) line-preceding-a-fenced-code-block, (4) body-paragraph (which by construction cannot overlap with any of the three rendering-driven contexts, per the operational definition above). The first context whose syntactic conditions the site satisfies wins; lower-ranked matches are suppressed for the purposes of selecting *Edge cases* mechanics.
- **Dual-form layout.** When the HTML form is used, it MUST appear on the same source line as the inline marker, in the order `<a id="prefix-n"></a> **PREFIX-N.**` — the HTML anchor first, a single ASCII space, then the inline marker. This ordering keeps the inline marker adjacent to the rule text it introduces and pins a single witness regex for the dual-form pair (`<a id="prefix-n"></a> \*\*PREFIX-N\.\*\*`). The dual-form pair MUST be the first non-prefix tokens on the line; the regex is matched line-by-line against each source line after stripping (a) any leading ASCII whitespace, and (b) a Markdown block-prefix sequence built from the closed grammar below. Tabs anywhere in the line-prefix region (i.e. before the dual-form pair) are normalised to spaces per CommonMark before stripping. The permitted block-prefix sequence has three ordered slots, each independently optional, in this fixed order; each slot's prefix bytes MAY be preceded by ASCII whitespace per CommonMark indentation rules:

  1. **Blockquote slot.** Zero or more `> ` blockquote prefixes (so `> `, `> > `, `> > > `, etc. all match).
  2. **List-item slot.** Zero or one list-item marker, drawn from:
     - an unordered-list marker `- `, `* `, or `+ `; or
     - an ordered-list marker `<digits>.` or `<digits>)`, in either case followed by a single ASCII space.
  3. **Construct-marker slot.** Zero or one construct-marker prefix, drawn from:
     - a table-cell leading `|` with optional surrounding ASCII whitespace (per CommonMark/GFM table syntax — i.e. an optional leading-pipe at column 0 plus the inter-cell pipes that separate cell content);
     - an ATX-heading `#{1,6} ` (one to six `#` characters followed by a single ASCII space);
     - a definition-list body marker `: ` (colon followed by a single ASCII space); or
     - a footnote body marker `[^<identifier>]: ` (a footnote label `[^<identifier>]` followed by colon and a single ASCII space).

  Slots are stripped in the order 1 → 2 → 3; no other ordering is recognised. A prefix from a later slot appearing before a prefix from an earlier slot on the same line, or two prefixes drawn from the same slot 2 or slot 3 on the same line (the slot-1 `> ` is the only prefix that may repeat, per its zero-or-more cardinality), is not stripped, so the dual-form pair is not the first non-prefix tokens on the line and the witness regex fails. The legal compositions therefore include `> - `, `> > - `, `> > > 1. `, and a blockquoted ATX heading `> ## `, while `- > ` and `## - ` are not recognised.

  This strip-list MUST stay co-extensive with the union of (a) the table-cell and ATX-heading rendering-driven contexts enumerated in *Required HTML-anchor contexts* above and (b) every construct enumerated in the body-paragraph operational definition there (which is what brings definition-list bodies, footnote bodies, list items, and blockquote bodies into scope). Adding a new context to either side of *Required HTML-anchor contexts* without extending this strip-list (or vice versa) is a spec defect: the witness regex would either reject every legal anchor in the new context or silently extend by implementer guess. In the three rendering-driven contexts (table cells, ATX headings, and the line preceding a fenced-code-block normative example), the mechanics of placing the dual-form anchor are governed by *Edge cases* below.

  *Per-page progressive normalisation.* At every commit on `main` — taking the first-parent diff (in git's sense, per `git log --first-parent` / `git diff-tree --first-parent`) for merge or squash commits, and the diff against the commit's sole parent for single-parent commits — on each non-narrative spec page where the commit's diff touches the anchor token of an existing REQ-ID anchor site (i.e. modifies, in any byte, either the inline `**PREFIX-N.**` marker bytes or the dual-form `<a id="prefix-n"></a>` HTML token co-located with it, at a site that was present at the commit's first parent), every REQ-ID anchor site on that page MUST be in dual-form in the post-diff text — both the touched site and every other site on the same page. Adding a new REQ-ID anchor site (a source line newly carrying an inline `**PREFIX-N.**` marker that was not present at the commit's first parent) does not by itself trigger this clause; new sites are bound directly by *Required HTML-anchor contexts* above. Retiring an existing site (deleting its marker line, or replacing it with an `<!-- … retired … -->` comment) likewise does not trigger; the site is gone and there is nothing to leave in dual-form. Edits to a `**PREFIX-N.**` paragraph's rule body, edits to surrounding prose on the marker's source line that leave both the inline marker bytes and the dual-form HTML token byte-unchanged, and line-renumbering shifts caused by insertions or deletions on unrelated source lines, do not by themselves trigger this MUST. Pre-existing non-conformant anchor sites on pages whose existing anchor tokens the commit does not touch under the trigger predicate above are standing spec defects — they are not bound by this MUST on the current commit, and remain so until a future commit performs a triggering operation on the same page. The defect set drains progressively as pages are edited; no corpus-wide normalisation commit is reserved.
- **Edge cases.** Inserting an anchor inside an ATX heading shifts the heading's auto-generated GitHub fragment ID; where every affected in-corpus cross-link can be repointed atomically with the anchor insertion, every such cross-link MUST be repointed in the same diff. When repointing the heading-slug fragment is impractical in the same edit (e.g. the heading slug is referenced from outside the spec corpus or the cross-link cannot be repointed atomically with the anchor insertion), the entire dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`) MUST instead be placed on the source line immediately preceding the heading, and the heading text MUST NOT carry the inline `**PREFIX-N.**` marker. In that case the anchor site classifies as body-paragraph context per the operational definition in *Required HTML-anchor contexts* above (the line preceding an ATX heading is body-paragraph by negation, since it is not itself inside the heading), so the *Dual-form layout* same-source-line MUST and witness regex apply normally with no ATX-heading-specific carve-out. Inside a table cell, the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`, in the order specified by *Dual-form layout* above) MUST be the cell's leading token sequence; where carrying the rule body in the cell would distort column widths, lift the entire rule out of the table — the cell then carries only a back-reference (e.g. `per PREFIX-N`) and the dual-form pair sits at the start of the rule's body-paragraph source line outside the table, per *Dual-form layout*. In the line-preceding-a-fenced-code-block context, the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`, in the order specified by *Dual-form layout* above) MUST appear on the source line immediately above the opening fence, with no intervening blank line, and that line MUST NOT carry any other content. The line-preceding-a-fenced-code-block *rendering-driven context* (and its per-context placement mechanics) is reserved for code blocks whose content is the rule's normative example; the line preceding a decorative or illustrative fence does not classify as that context and so does not invoke its per-context mechanics. Such a line is body-paragraph context per the operational definition in *Required HTML-anchor contexts* above, so a dual-form REQ-ID anchor on that line is still required if the line carries an inline `**PREFIX-N.**` marker — it is governed by the body-paragraph mechanics, not by the line-preceding-a-fenced-code-block context. Body-paragraph context carries no per-context edge case beyond the *Dual-form layout* same-source-line MUST: the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`) sits at the start of the body-paragraph source line that carries the inline marker, and any list-item / blockquote / nested-block indentation that precedes the pair is handled by the witness-regex anchoring rule rather than by a body-paragraph-specific carve-out here.

**GOV-3.** REQ-ID prefixes are byte-exact uppercase ASCII tokens of length 2–4 (`[A-Z]{2,4}`). Prefix matching is case-sensitive; `lex-1` does not match `LEX-1`. REQ-ID extraction operates on raw Markdown source bytes, not on rendered HTML. Before regex application, the following are stripped, in order: (i) fenced code blocks (` ```…``` ` and `~~~…~~~`, inclusive of fence lines), (ii) HTML comments (`<!--…-->`), (iii) inline code spans (`` `…` `` and the multi-backtick variants). Markdown link text is in scope; link targets are out of scope.

Two regexes define the in-corpus form of an extractable REQ-ID, against the exclusion-stripped corpus described above:

1. **Primary extractor.** `\b(<live-prefix-alternation>)-[1-9][0-9]*\b`, where `<live-prefix-alternation>` is built from the live REQ-ID prefix table below at evaluation time (never hard-coded). Leading zeros in the numeric tail are forbidden.
2. **Unknown-prefix closure invariant.** Every token in the exclusion-stripped spec corpus that matches `\b[A-Z]{2,4}-[1-9][0-9]*\b` MUST have a prefix that appears in the union of (live REQ-ID prefix table, *Retired prefixes* sub-table). [GOV-16](./stable-inline-labels.md#gov-16) extends this closure invariant to the inline-label identifier class (widening the prefix character class to `\b[A-Z][A-Z0-9]{1,5}\b` and the tail to admit the single-lowercase-letter form, against the widened membership union that includes the per-page inline-label prefix table).

Pages whose row in the prefix table below carries the literal cell `(no IDs — narrative)` are excluded from extraction; all other rows in `spec_topics/*.md` are in scope. The exclusion cell's canonical byte sequence is `(no IDs — narrative)` — open paren, `no IDs`, ASCII space, U+2014 EM DASH, ASCII space, `narrative`, close paren. Cosmetic variants (smart quotes, trailing whitespace, en-dash, ASCII double-hyphen) are not recognised as the narrative cell, and the cell-value comparison that classifies a row as narrative MUST be byte-exact.

IDs are immutable: when a rule is split, the original ID retires and two new IDs appear; numbering never collapses to fill holes.
