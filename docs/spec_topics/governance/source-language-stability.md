# Source-language stability

This page owns GOV-15 — the loom 1.x source-language equivalence release-process goal — together with its loads-cleanly predicate, ceiling-set carve-out, attribution test, and the operational definitions of the four ceiling-set verbs. The retired REQ-ID prefix registry lives on [Req id prefix table retired](./req-id-prefix-table-retired.md).

<a id="gov-15"></a> **GOV-15 (loom 1.x source-language equivalence — release-process goal).** A `.loom` or `.warp` file that loads cleanly under loom 1.0.0 is expected to load under every loom 1.x release and to produce, for any given input, identical (a) return values, (b) ordered diagnostic-code sequences, and (c) equivalent `loom-system-note` content strings. Two content strings are equivalent under observable (c) when they are byte-identical — i.e. equal as UTF-8 byte sequences, the encoding substrate pinned by [Lexical — Encoding](../lexical.md) — after normalising the placeholder sub-fields whose rendered value is permitted to vary per invocation or per run — the per-invocation identifiers (e.g. `<invocation-id>` / `<uuid>`), the wall-clock-derived values (e.g. `<ms>`), and the category-8 host-derived freeform tails — as classified by the placeholder-rendering categories in [Diagnostics — Placeholder rendering](../diagnostics/placeholder-rendering-a.md#placeholder-rendering-normative); the fixed (non-variable) placeholder renderings and the surrounding literal template bytes are themselves byte-identical. Observable (c)'s equivalence relation, not the exclusion list, governs variation that is embedded as a placeholder sub-field inside a content string. Wall-clock timing, token counts, and log-line volume remain excluded from this expectation as independent observables in their own right — distinct from the wall-clock-derived sub-fields that observable (c) normalises within a content string. This rule states a release-process goal, not a verifiable obligation: loom 1.0.0 ships without an automated equivalence gate; equivalence between two loom 1.x releases is a release-process responsibility verified by reviewer inspection of the diff against the prior loom 1.x release. <a id="gov-15-fixture-suite"></a> A conformance fixture suite that mechanically diffs the three observables above against a frozen loom 1.0.0 baseline is a recognised post-loom 1.0.0 follow-up; see [Future Considerations](../future-considerations.md). The companion review-posture guidance on not re-raising the missing gate as a loom 1.0.0 correctness finding was formerly GOV-14, now retired and demoted to a non-normative editorial note (see [Retired REQ-IDs](./anchor-scheme-and-retired.md#retired-req-ids)). The (a)–(c) observable list and the wall-clock / token-count / log-line-volume exclusion list are both closed at loom 1.0.0: any other observable not named in either list is, by default, neither in scope of the equivalence promise nor of the explicit-exclusion list, and adding a further observable to either list is a substantive edit to GOV-15 governed by the present rule's edit conventions, not by GOV-8's *Pure rewording* arm. Such a third-bucket observable diverging between releases is, by default, a documentation defect against GOV-15 — to be resolved by adding the observable to one of the two lists in a follow-up release — and is not itself a departure from the current release's equivalence promise. References elsewhere in the corpus to "four ceilings", "four-item list", "four NOCEIL-1 … NOCEIL-4 claims", "four-axis closure list", and any other count-bearing phrase naming the loom 1.0.0 starting set are bound by GOV-30 lock-step: any operation that changes the count those phrases name (e.g. *Introduce*, append-trailing *Retire*, retirement form (ii) coining a new `NOCEIL-N`, or *new-axis case* introduction extending the closure axis count) also updates the affected count-bearing phrases in the same commit.

<a id="gov-15-loads-cleanly"></a>

*Loads-cleanly predicate.* "Loads cleanly under loom 1.0.0" — the *Given* that selects the input set GOV-15's equivalence promise ranges over — means: loading the `.loom` or `.warp` file under loom 1.0.0 emits no diagnostic of effective severity `error` (`E`). Diagnostics of effective severity `warning` (`W`) do not disqualify a clean load. The no-`E` predicate alone selects GOV-15's input set; it is not co-extensive with whether the runtime registers the file as a loom (a `W`-severity load can leave the file unregistered — e.g. `loom/load/unreadable`). *Effective severity* is read from the *Severity* column of the diagnostics [Code registry](../diagnostics/code-registry-parse.md#code-registry); for the discovery-source codes whose *Severity* cell carries `E/W`, the effective severity is the per-source value resolved by the [Discovery — Failure modes](../discovery.md) tables rather than a code-level fixed value. This predicate coins no new severity classification — it names the existing *Severity*-column landmark as the selector. A file that does not load cleanly under loom 1.0.0 is outside the GOV-15 equivalence promise's input set.

<a id="ceiling-set-carve-out"></a>

*Ceiling-set carve-out.* Introducing a new hard runtime ceiling within loom 1.x — or retiring or relaxing an existing one — is a recognised carve-out to this equivalence. The carve-out's in-scope input set is defined *post-hoc* over the diff between the two releases under comparison and is a property of that inter-release diff, not a property the runtime can statically derive at single-release evaluation time: an input is *in scope* of the equivalence promise iff, when run under both releases, it either (i) trips no hard runtime ceiling under either release, or (ii) trips exactly the same hard runtime ceiling (identified by its bullet ordinal in the four-item list at [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings), persistent across the inter-release diff) under both releases. An input that trips a ceiling absent from one of the releases, or that trips a ceiling under one release but succeeds under the other (whether via the ceiling being introduced, retired, relaxed, or never having been ceilinged on the other side), is *outside* the in-scope set and the carve-out applies to it. "Trips" here means *surfaces* — the single ceiling whose firing reaches loom code under the CIO-1 … CIO-6 evaluation order; co-fires recorded under `masked` are not separately tested. A release that reorders surfaced ceilings (via a CIO-ordering or `masked`-domain change) without performing a ceiling-set change does NOT trigger the carve-out — the carve-out is keyed to ceiling-set changes only. Such surfacing-reordering edits remain bound by the equivalence promise on observables (a)/(b)/(c) and any divergence on inputs that previously surfaced ceiling X but now surface ceiling Y is a GOV-15 release-process violation; CIO-ordering and `masked`-domain edits that change observable surfacing therefore require loom 2.0 migration unless they preserve every input's surfaced ceiling identity. For inputs outside the in-scope set defined above, observable (a) is by construction permitted to differ between releases — the success/fail outcome is the very change the ceiling-set change produces — and the carve-out also covers this (a) drift; only on the in-scope set is the equivalence promise unaltered for (a), subject to the routing-class-demotion exception below. For inputs that remain in-scope but whose routing-class is demoted under *Relax* (e.g. panic→recoverable-`Err`), observable (a) is permitted to differ between releases — the routing-class change's own (a) impact (panic-shape → `Err`-value-shape, or analogous demotion) is carve-out-covered for those inputs; only the changed-ceiling's own discriminator and message text are bound by the *Attribution test*. The [deferred conformance-fixture suite](#gov-15-fixture-suite) is the mechanism that verifies the carve-out post-hoc against the inter-release diff. The carve-out covers only the changed ceiling's own diagnostic emissions — i.e. observables (b) and (c) of GOV-15's first paragraph as they pertain to the changed ceiling — enumerated parallel to the three carve-out-eligible ceiling-set changes: for an *introduced* ceiling, the new discriminator and the canonical message it adds; for a *retired* ceiling, the removed discriminator and the canonical message it removes; for a *relaxed* ceiling, the existing discriminator (which continues to apply but fires under different conditions) and any canonical-message text change that re-states the new bound (e.g. `exceeded 50` replacing `exceeded 25`) or otherwise reflects the relaxed predicate; plus, in all three cases, any `loom-system-note` content directly attributable to the ceiling-set change (per the *Attribution test* below). On in-scope inputs, an unrelated discriminator whose multiplicity (number of emissions per query) changes solely as a behavioural consequence of the ceiling-set change — e.g. a per-round diagnostic that fires more often after a loop bound is relaxed — is treated as carve-out-covered drift, not as a GOV-15(b) violation; the canonical-message content of the *unrelated* discriminator must remain unchanged on each emission; the changed ceiling's own diagnostic content is governed by the *Attribution test* below. Collateral diagnostic-code edits or `loom-system-note` content edits landed in the same release on unrelated surfaces (per the same *Attribution test*) are forbidden in a ceiling-set release per the *Attribution test* below; they must be split into a separate loom 1.x release that lands no ceiling-set change. When a single loom 1.x release performs more than one ceiling-set change (e.g. introduces one ceiling and retires another, or introduces two ceilings at once), the carve-out applies as the union of the per-change carve-outs — each ceiling-set change is individually scoped per the *Attribution test* below, and an input that falls outside the in-scope set under any one of the constituent changes is outside the in-scope set for the release as a whole. An edit jointly attributable to two or more constituent ceiling-set changes is in-scope for the release; an edit attributable to at least one constituent change is in-scope, even if it is collateral with respect to another constituent change. The *Collateral* set is the residue. The orientation aggregator at [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings) names this carve-out in lock-step with this rule per [GOV-30](./req-id-prefix-table-active-b.md#gov-30). The GOV-15 reviewer-inspection step verifies the [ceiling-set invariants](../hard-ceilings/ceilings-3-and-4.md#ceiling-set-invariants) per the canonical clause at [Hard Runtime Ceilings — Release gate](../hard-ceilings/ceiling-invariants-and-audit.md#release-gate).

<a id="attribution-test"></a>

*Attribution test.* The carve-out's two phrases — *directly attributable to the ceiling-set change* and *collateral … on unrelated surfaces* — are pinned by the following syntactic predicate, mechanically applicable on the release diff. A wording change to a diagnostic code or to `loom-system-note` content is *directly attributable to the ceiling-set change* iff at least one of the following holds: (i) its discriminator (`details.kind` for diagnostic-channel ceilings, the `Err`-variant `kind` for `QueryError` ceilings, or the discriminator named in ceiling #4's per-boundary table at [`hard-ceilings.md` — Per-boundary destination/surface table (ceiling #4)](../hard-ceilings/ceilings-3-and-4.md#ceiling-4-table)) equals one of the changed ceiling's own discriminators (plural where the changed ceiling cross-routes — e.g. ceiling #4's `loom/runtime/*` and `loom/load/*` arms both count); (ii) its canonical message text contains either the ceiling's ordinal identifier (e.g. `ceiling #N`) or the literal numeric bound being introduced, retired, or relaxed; (iii) the change is the minimal edit required to keep a message-template row, header line, or `details.kind`/discriminator enumeration shared with another ceiling well-formed under the changed ceiling (the *shared-template* case — including a discriminator rename that subsumes both the changed ceiling's old discriminator and an unrelated runtime discriminator, where the rename's normative widening is the minimal edit required and any further wording change to the unrelated discriminator's surface is not the minimal edit required and falls into the *Collateral* set below); or (iv) for a routing-class demotion or enforcement-point narrowing with no numeric bound, the canonical message text contains an explicit reference to the routing-class or enforcement-point surface change (e.g. `no longer fatal`, `no longer enforced at <site>`) minimal to reflect the relaxed predicate. Each ceiling's owner page is the source of truth for that ceiling's discriminator inventory. *Collateral* is any wording change for which none of the four sufficient conditions (i)–(iv) above holds (i.e. for which *Directly attributable* is false). A loom 1.x release that performs a ceiling-set change is, under GOV-15's release-process goal, expected to carry no *Collateral* diagnostic-code or `loom-system-note` content edits in that release: the carve-out covers only *Directly attributable* edits, so a *Collateral* edit landed alongside a ceiling-set change is not carve-out-covered and departs from the equivalence promise on the unrelated surfaces it touches. Such collateral edits belong in a separate loom 1.x release that lands no ceiling-set change. This clause states a property of the release content; per [GOV-18](./corpus-direction-and-scope.md#gov-18-binding-prohibition) the spec corpus takes no position on whether or how any release process audits it.

<a id="operational-definitions"></a>

*Operational definitions.* The four ceiling-set verbs name distinct operations. *Introduce* adds the new ceiling's ordinal-bearing entry to the four-item list at [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings) (extending it to a five-item list), creates the per-ceiling owner-page section on the appropriate topic page (defaulting to a new sub-section on [`hard-ceilings.md`](../hard-ceilings.md) if no other topic page is the natural owner), and adds a per-axis entry under [Hard Runtime Ceilings — Audit methodology — Four-axis closure list](../hard-ceilings/ceiling-invariants-and-audit.md#four-axis-closure-list), all in the same commit per GOV-30; *Introduce* is a normative addition not modelled by any GOV-8 lifecycle case (which addresses only edits to existing rules), and is governed directly by the present rule and by the audit-methodology paragraph at [Hard Runtime Ceilings — Audit methodology](../hard-ceilings/ceiling-invariants-and-audit.md#audit-methodology). *Retire* — remove a ceiling; the retired ceiling's owner-page entry is annotated in place with a pointer to the retirement-release diff. The pointer takes the same SHA-or-tag form as the `Retired in` column of the [*Retired prefixes*](./req-id-prefix-table-retired.md#retired-prefixes) sub-table — either the 7-character abbreviated commit SHA of the retirement commit or the release tag of the loom 1.x release that performed the retirement, nothing else. For inputs that previously tripped the retired ceiling, the retired ceiling's owner-page annotation MUST document the post-retirement runtime trajectory (succeed; fall through to a documented `NOCEIL-N` surface; fall through to another ceiling; or non-terminating absent cancellation); silent unbounded behaviour is forbidden. Cardinality after retirement: the retired ceiling's bullet is left as an in-place retirement-annotated stub (preserving the persistent ordinals the carve-out's in-scope predicate relies on); the bullet MUST NOT be removed outright while any later bullet exists, since outright removal would shift subsequent ordinals and break the in-scope predicate's identity test. Bullets MAY be removed outright only when retired in append-trailing position (no later bullet exists), in which case the editor MUST also update every count-bearing reference (per GOV-30 lock-step) in the same commit. *Relax* — widen the input set the ceiling permits (raise a numeric bound, narrow the enforcement-point surface, or demote routing class — e.g. `panic` to recoverable `Err`). *Tighten* — the inverse of relax (narrow the input set the ceiling permits — lower a numeric bound, broaden the enforcement-point surface, or promote routing class) — is explicitly NOT a recognised carve-out under this rule, because it can break the equivalence promise on inputs that previously succeeded; it is permitted within loom 1.x only when no input in the loom 1.x conformance fixture suite (once it ships) is made to fail by the change. Until that suite ships, *Tighten* is forbidden under loom 1.x and any required tightening is deferred to loom 2.0 migration per [Future Considerations — loom 1.0 non-goals](../future-considerations/model-changes-and-non-goals.md#v1-non-goals); a tightening that would fail any input in the eventually-shipped suite remains likewise deferred to loom 2.0 and MUST NOT land within loom 1.x. Once *Tighten* becomes permitted (per the conformance-fixture-suite condition above), it falls under the same GOV-30 lock-step co-edit and *Audit methodology* release-gate obligations as *Relax*: once *Tighten* is unlocked, any spec-side enumeration that lists "introduce, retire, or relax" is read as covering *Tighten* in the same shape as *Relax* — but the inheritance is procedural only (lock-step + per-axis-entry update + release gate), not equivalence: *Tighten*-affected inputs remain bound by the GOV-15 main-paragraph equivalence promise without any *Tighten*-specific carve-out (numeric-bound, enforcement-point surface, or routing-class edit to an existing ceiling without retiring its identity). The Relax/Tighten sub-case enumeration above (numeric-bound, enforcement-point surface, routing-class) is closed at loom 1.0.0; a fourth widening or narrowing shape is forbidden in-place and requires loom 2.0 migration. Both *Relax* and *Tighten* are substantive ceiling-page edits in GOV-8 terms; they are licensed for in-place edit (rather than the GOV-8 retire-and-readd path) only by the present rule's *Ceiling-set carve-out* (for *Relax*) or by the present rule's *Tighten* permission clause (for *Tighten*). Either operation is recorded by amending the ceiling owner-page entry in place; GOV-8's split-or-deletion-plus-add path is explicitly not required for either operation. loom 1.0 specifies no upgrade-time warning surface for retirement-induced dead `match` arms in loom code (e.g. a `match` arm on `Err(QueryError { kind: "tool_loop_exhausted", … })` that becomes statically unreachable if `tool_loop.max_rounds` were ever retired); such drift is a known consequence of *Retire* and is deferred to a future tooling surface (no loom 1.0 tooling specification exists; whether such a warning surface is added in a future loom 1.x release is out of scope here). Symmetrically, a routing-class demotion under *Relax* (e.g. `panic` → recoverable `Err`) makes a previously-unreachable `match` arm reachable on previously-panicking inputs; loom 1.0 specifies no upgrade-time warning surface for the new-arm reachability case either.
