# Slash-Command Invocation

A theta is invoked as a slash command using its filename, exactly like a Pi prompt template:

```
/code-review TypeScript focusing on error handling and async, by Ada Lovelace, senior engineer 12y
```

The runtime extracts typed `params:` values from the user's free-form slash arguments via an LLM-driven binder. The full mechanism is described in [Slash-Command Argument Binding](./binder.md); the short version is that the binder model (resolved from `bind_model:` or the `theta.binderModel` setting at load time) is given the theta's `params:` schema and the raw slash text and asked to return a structured envelope (`ok`, `needs_info`, or `ambiguous`). Successful binding feeds AJV-validated params into the theta; unsuccessful binding surfaces a one-line system note in the user's session and the theta does not run.

On successful binding the runtime appends a one-line echo system note to the session before the theta starts, summarising the bound arguments. The echo is on by default, suppressed by `bind_echo: false`, and auto-suppressed for the single-string-param bypass case (where `bind_echo: true` is a parse warning). See [Slash-Command Argument Binding](./binder.md) for the formatting rules and the bypass condition.

The `argument-hint` frontmatter field is passed to the binder as additional grounding for argument extraction — it appears in the binder's system prompt under `Argument hint:` and helps the model interpret the user's free-form text. It is **not** displayed in Pi's slash-command autocomplete UI; the field's autocomplete contract — why extension-registered commands surface no hint and what authors should do instead — is owned by [Parameters and Frontmatter — `argument-hint`](./frontmatter/frontmatter-fields-a.md#argument-hint-binder-only). Key=value or named-argument syntax (e.g. `/code-review language=TypeScript`) is *not* part of the theta 1.0 surface; users type free-form text and the binder does the work.

<a id="slsh-1"></a> **SLSH-1.** **No-params overflow.** When the theta takes no parameters (`params:` absent or `params: {}`; see [Parameters and Frontmatter](./frontmatter.md)) the binder is bypassed (see [Slash-Command Argument Binding — Binder bypass](./binder.md)). The runtime trims leading and trailing [slash-argument whitespace](./binder/binder-bypass-and-envelope.md#slash-argument-whitespace) from the slash text after the command name; if the trimmed remainder is non-empty, the runtime emits a single `customType: "theta-system-note"` message before the theta starts, formatted as `theta /<name>: ignoring extra arguments — this theta takes no parameters`, then runs the theta. Whitespace-only remainders trim to empty and emit no note. The note is informational and never blocks execution. The note fires only for the slash-invocation path; `invoke(...)` and registered-tool callers skip the slash parser entirely and have no notion of "extra text."

Once a theta is invoked:

- In **prompt mode**, the theta drives the *current* conversation — every query is a turn the user sees in their session. The theta's final `Ok` return value is **not** surfaced to the user (and no `theta-system-note` keyed on the `Ok(v)` outcome is emitted — see [Pi Integration Contract — Runtime event channel — Success-side null-policy](./pi-integration-contract/runtime-event-channel.md#success-side-null-policy)); the conversation is the user-facing surface, and any value the author wants the user to see should be issued as a final query whose text contains it. The return value exists only for programmatic consumers (an `invoke` caller, a future theta harness).
- In **subagent mode**, a fresh isolated conversation is spawned for the theta — with the system prompt set from frontmatter `system:` if present. Every query is a turn in that private conversation. When the theta finishes, only its return value reaches the caller (and no `theta-system-note` keyed on the `Ok(v)` outcome is emitted — see [Pi Integration Contract — Runtime event channel — Success-side null-policy](./pi-integration-contract/runtime-event-channel.md#success-side-null-policy)); the intermediate transcript stays inside the subagent. When such a theta is the slash-dispatch boundary (dispatched directly by a slash command, with no invoke parent) and terminates with a top-level `Err(QueryError)`, that failure is surfaced to the user's session per [SLSH-3](#slsh-3) even though the transcript stays private.

<a id="slsh-2"></a> **SLSH-2.** **User-visible streaming.** In prompt mode, assistant tokens for both untyped and typed queries stream into the user's transcript in real time as Pi's TUI receives them; the theta interpreter resumes execution only after `ctx.waitForIdle()` resolves, but the stream's appearance in the transcript is independent of the theta's resumption — the user sees the response unfold while the theta is still mid-query. For typed queries, the free-phase tool calls render normally in the prompt-mode transcript as ordinary Pi tool-call cards. The forced respond turn that obtains the schema-conformant response via the synthesised one-shot tool is dispatched off-session through pi-ai's `complete()` free function (see [Pi Integration Contract — Conversation drive](./pi-integration-contract/conversation-drive.md) and [Binder inference call](./pi-integration-contract/binder-inference.md#binder-inference-call)); it attaches no turn to the user session and therefore does not render a transcript card. The validated payload is delivered to theta code, and prompt-mode user-visibility of any value the author wants the user to see follows the [success-side null-policy](./pi-integration-contract/runtime-event-channel.md#success-side-null-policy). Edge cases:

- An `Err` propagated by `?` after the user has already seen partial assistant text: the streamed prefix stays in the transcript (Pi's behaviour) and the `theta-system-note` describing the failure is appended after, not interleaved.
- Cancellation mid-stream (see [Cancellation](./cancellation.md)): whatever partial text Pi has already rendered remains visible; partial output is not rolled back. The cancellation system note is appended after the partial prefix. The runtime's non-mutation obligation toward Pi-committed conversation surfaces (assistant tokens, tool-call cards, system notes) on this path is normative — see [Errors and Results — Mid-stream cancellation, conversation state](./errors-and-results/error-model.md#mid-stream-cancellation-conversation-state).

In subagent mode, no assistant tokens, tool-call cards, or system notes from a subagent's queries surface to any ancestor transcript. The subagent's queries are unobservable to the human operator by construction, and the enforcing mechanism is twofold: the subagent runs the whole callee inside a spawned child `pi` process launched with **`--no-session`** (no session file is written), and that child session's stdout is **process-private** to the child — it is read only by the parent runtime for the `{"theta_result": …}` return envelope and never routed into Pi's user-facing UI (see [Pi Integration Contract — Query mechanics run in the child](./pi-integration-contract/subagent.md#subagent-untyped-query-ok-extraction) and [Return-value envelope (PIC-59)](./pi-integration-contract/subagent.md#pic-59)). The substance is unchanged — the child session never reaches the user's UI — only the enforcing mechanism (`--no-session` + process-private stdout in the RFC 0006 child) is restated. The only artefact that crosses back into the parent context is the theta's return value (or an `InvokeCalleeError` / `InvokeInfraError`, per [Invocation](./invocation.md)).

Underlying SDK delivery mechanics live in [Pi Integration Contract — Conversation drive](./pi-integration-contract.md).

<a id="slsh-3"></a> **SLSH-3.** **Top-level `Err` at the slash-dispatch boundary.** When a theta dispatched directly by a slash command — one with a slash caller and no invoke parent (see [Glossary — caller](./glossary.md#caller)) — terminates by returning `Err(QueryError)` to that boundary, Pi appends a one-line system note to the user's session formatted from the error, regardless of whether the theta ran in prompt or subagent mode. The note never dumps the full `QueryError` JSON — it summarises the failure category and the most-relevant detail. For a directly-slash-invoked subagent-mode theta this note is the only user-facing surface for the failure: the subagent's intermediate transcript stays private (see [User-visible streaming](#slsh-2)), so the note is emitted at the slash-dispatch boundary in the user's session, not in the subagent's private transcript. A subagent-mode theta reached from inside another theta via `invoke(...)` or a `.theta`-callable call has an invoke parent, is therefore not a slash-dispatch boundary, and does not trigger SLSH-3; its top-level `Err` cascades to that parent per [Invocation](./invocation.md) and the operator-facing runtime event follows the subagent-mode `display: false` rule (see [Pi Integration Contract — Runtime event channel](./pi-integration-contract/runtime-event-channel.md#runtime-event-channel)).

<a id="slsh-4"></a> **SLSH-4.** **The shapes below are normative templates.** Renderers MUST emit the surrounding template text verbatim; only the `<…>` placeholders are interpolated. Wording changes are spec-versioned breaking changes. Conformance tests MAY assert on the exact rendered string. Where a placeholder carries free-form content sourced from a model (e.g. `<message>` on rows whose underlying error message originated outside the runtime), only the surrounding template is normative — the interpolated content itself is non-deterministic. The inline backticks in the `System note shape` cells below are Markdown code-span formatting for readability and are not part of the emitted string: renderers emit the cell text with the surrounding backtick characters removed (the SLSH-5 worked examples below show the intended backtick-free output). The stripping applies only to the template text — backtick characters that arrive inside an interpolated placeholder (e.g. a `<message>` whose model-sourced content contains them) are part of that free-form content and pass through unchanged.

Per-`kind` formatting. Each row below carries a stable inline label (`SNK-a` … `SNK-k`, in the `Label` column) so a conformance test or coverage leaf can cite a single row's rendering rule independently of the SLSH-4 parent. These labels are the second governed identifier class per [Governance — GOV-16](./governance/stable-inline-labels.md#gov-16) — they are *not* REQ-IDs; the `SNK` prefix is registered for this page in GOV-16's per-page inline-label prefix table. SLSH-4 remains the parent REQ-ID owning the umbrella MUST above.

| Label | `QueryError.kind` | System note shape |
|---|---|---|
| <a id="snk-a"></a> **SNK-a.** | `validation` (`cause: "schema_validation"`) | "theta `/<name>` returned `Err`: model failed schema after `<n>` respond-repair attempts" |
| <a id="snk-b"></a> **SNK-b.** | `validation` (`cause: "empty_template"`) | "theta `/<name>` returned `Err`: rendered query template was empty — no provider turn was issued" |
| <a id="snk-c"></a> **SNK-c.** | `transport` | "theta `/<name>` returned `Err`: transport — `<message>`" |
| <a id="snk-d"></a> **SNK-d.** | `model_tool` | "theta `/<name>` returned `Err`: tool `<tool_name>` failed — `<message>`" |
| <a id="snk-e"></a> **SNK-e.** | `context_overflow` | "theta `/<name>` returned `Err`: context overflow" |
| <a id="snk-f"></a> **SNK-f.** | `cancelled` | "theta `/<name>` cancelled" |
| <a id="snk-g"></a> **SNK-g.** | `code_tool` | "theta `/<name>` returned `Err`: tool `<tool_name>` call failed (`<cause>`) — `<message>`" |
| <a id="snk-h"></a> **SNK-h.** | `tool_loop_exhausted` | "theta `/<name>` returned `Err`: tool-call loop exhausted after `<rounds>` rounds (last tool: `<last_tool_name>`)" |
| <a id="snk-i"></a> **SNK-i.** | `invoke_infra` | "theta `/<name>` returned `Err`: invoke of `<callee_path>` failed (`<cause>`)" |
| <a id="snk-k"></a> **SNK-k.** | _any unlisted `kind`_ (catch-all) | "theta `/<name>` returned `Err`: `<kind>` — `<message>`" |

The table is exhaustive over the theta 1.0.0 `QueryError` union (nine variants, listed in [Query — Failure modes](./query.md)); the catch-all row makes the renderer's contract total against any future variant added to the union, so a renderer never has "no defined output" for a well-formed `QueryError`. The `validation` variant occupies two rows keyed on the `cause` field of [`ValidationError`](./errors-and-results/queryerror-variants.md#query-time-variants): `schema_validation` (a post-provider schema rejection) and `empty_template` (the pre-provider empty-rendered-template short-circuit per [Query — Degenerate rendered templates](./query.md), which fires with `attempts: 0` before any model turn). New variants SHOULD ship with a dedicated row in the same edit; the catch-all is a normative fallback, not an excuse to skip the per-kind row. For `tool_loop_exhausted`, `<last_tool_name>` is rendered as the literal string `respond` when `last_tool_name` is `null` (a defensive rendering retained for forward compatibility — `null` has no theta 1.0-reachable case since the typed-query forced respond turn is the exempt-routed terminator dispatched by CIO-4's `max_rounds`-final branch and does not reach this exhaustion path; see [Hard Runtime Ceilings — CIO-4](./hard-ceilings/ceilings-3-and-4.md#cio-4)). `invoke_callee` has no dedicated per-`kind` row: every concrete `InvokeCalleeError` carries a required `inner: QueryError`, so there is no inner-less `invoke_callee` for a row to format, and SLSH-5's chain attribution (described in the next paragraph) always recurses through `inner` to the leaf `kind` and renders that leaf's row. The chain suffix recurses into `inner` so the leaf `kind` (not the wrapper) drives the descriptive text. The chain suffix applies to every row, including the catch-all, whenever the failure cascaded from an invoked child.

The session is not aborted; the user can type a follow-up turn.

<a id="slsh-5"></a> **SLSH-5.** **Chain attribution.** When the leaf failure originated inside an `invoke`d child theta that cascaded out via `?` (i.e. the rendered `QueryError` is `invoke_callee`, possibly nested), renderers MUST append a chain suffix to the per-`kind` row above. The suffix is built by walking `inner` recursively to the leaf and emitting, for each `invoke_callee` hop encountered, the literal text ` from <callee_path> invoked at <parent_path>:<line>`, in leaf-first order (innermost hop first), each hop separated from the next by a single space. `<callee_path>` and `<parent_path>` are the post-`realpath` absolute paths recorded at the invocation site (per [Invocation](./invocation.md), "Resolution" — the same `realpath`-normalised absolute paths used for discovery-root containment; for a `.theta`-callable bare-identifier call the `<callee_path>` is the resolved path held in the parent's [Resolution snapshot](./frontmatter/frontmatter-fields-b-and-templates.md#resolution-snapshot)). `<line>` is the 1-indexed source line (no column) of the call-site token in `<parent_path>` that produced the `invoke_callee` hop — the `invoke(` token of a literal `invoke(...)` expression, or the callee-name identifier of a `.theta`-callable bare-identifier call (e.g. the `summarise` token of `summarise(doc)` when `./summariser.theta` is listed in `tools:`, operationally equivalent to `invoke(...)` per [Tool Calls — Relationship with `invoke`](./tool-calls.md)). The line is that of the call-site token, not of any receiving binding, so multi-line calls are unambiguous. A `.theta` callable invoked by the model during a `@`-query loop feeds its failure back as a tool-error result rather than an `invoke_callee` cascade, so SLSH-5 emits no chain suffix for that surface. The descriptive text in the per-`kind` row is computed from the **leaf** `kind` (the innermost non-`invoke_callee` variant), not from the wrapper, with `<message>`, `<tool_name>`, etc. drawn from the leaf `QueryError` and `<callee_path>` on the wrapper drawn from the immediate parent's invocation record. The suffix applies to every row including the catch-all.

Worked examples:

- **Single-hop.** A `transport` failure inside `child.theta` cascaded out of `parent.theta` at the `invoke(` token on line 42 renders as `theta /entry returned Err: transport — connection reset from /abs/path/to/child.theta invoked at /abs/path/to/parent.theta:42`.
- **Single-hop (`.theta`-callable parent).** A `transport` failure inside a `.theta` callable registered as `summarise` (resolving to `./summariser.theta`) and called by the bare-identifier form `summarise(doc)` on line 18 of `parent.theta` renders as `theta /entry returned Err: transport — connection reset from /abs/path/to/summariser.theta invoked at /abs/path/to/parent.theta:18` — `<line>` is the line of the `summarise` callee-name identifier; the suffix is identical to the literal-`invoke(...)` form because the two surfaces share one error model.
- **Multi-hop (three-level).** A cascade `grandchild → child → parent` whose leaf is `model_tool` renders as `theta /entry returned Err: tool foo failed — bad arg from /abs/grandchild.theta invoked at /abs/child.theta:7 from /abs/child.theta invoked at /abs/parent.theta:42` — leaf-first ordering, single space between hops.
- **Catch-all interaction.** When the leaf `kind` is unlisted in the per-`kind` table, the catch-all row renders first (`<kind> — <message>` with leaf values), then the chain suffix appends in the same form.

Every row above emits as a `theta-system-note` carrying `details: { event: RuntimeEvent }`, where the `RuntimeEvent` payload is the same value emitted at the originating failure site (consumers deduplicate on `(kind, query_site, message, occurrence-timestamp)`). See [Pi Integration Contract — System notes](./pi-integration-contract.md) for the `pi.sendMessage` call shape, the disjoint `details` payload variants, the renderer registration, and the best-effort fallback chain, and [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md) for the `RuntimeEvent` shape and the `display: true` vs `display: false` rule.
