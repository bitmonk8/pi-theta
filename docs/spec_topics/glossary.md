# Glossary

This page collects the spec's coined vocabulary in one alphabetised list. Each entry is descriptive — a one-paragraph reminder of what a term means and where it is used. The canonical, normative definition lives on the feature page named in the entry's `See:` reference; if a glossary entry and its canonical page disagree, the canonical page wins. Add new entries here when the spec coins a new term that is reused on more than one page.

---

- **always-log set** — The closed subset of `QueryError` `kind` values whose runtime occurrence emits exactly one `loom-system-note` event regardless of whether the author matched the `Err`, propagated it via `?`, or discarded it via `let _ =`. The set partitions into group A (routed via `details: { event: RuntimeEvent }`) and group B (routed via `details: { diagnostics: Diagnostic[] }`). The members are enumerated, and the four exclusions (`validation`, `context_overflow`, `cancelled`, `invoke_callee_error`) justified, on the canonical page. See: [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md).

- **binder** — The LLM-driven argument-binding mechanism that maps a slash-command's free-form argument string onto a loom's typed `params:` envelope. Suppressed entirely in either of the two static *binder bypass* conditions (see *no-params bypass* / *single-string bypass*, below). See: [Slash-Command Argument Binding](./binder.md).

- **binder model** — The LLM the binder calls to map a slash-command's argument string onto a loom's typed `params:` envelope; resolved at loom-load time from `bind_model:` frontmatter or the `looms.binderModel` setting. The per-surface root-word convention for this concept across frontmatter, settings, diagnostic, and prose surfaces is pinned in the *Binder-model root-word convention* sub-paragraph in [Parameters and Frontmatter](./frontmatter/frontmatter-fields-a.md#binder-model-root-word-convention). See: [Slash-Command Argument Binding — Binder model](./binder/binder-model-and-context.md#binder-model).

- **caller** — Spec prose uses *caller* in two distinct senses, distinguished by context: (1) **slash caller** — the user whose slash-command turn dispatched a `prompt`-mode loom into their session (the user-visible column in [Errors and Results — Per-cause caller surfaces](./errors-and-results.md)); (2) **invoke parent** — the loom (or `.warp` `fn` reached from a loom) whose `invoke(...)` expression or `.loom`-callable call spawned the current loom. The two senses cannot co-occur for a single invocation: a slash dispatch always has a user as caller and no invoke parent; an invoked child always has an invoke parent and no direct slash caller (its root chain may have one). Authors and implementers SHOULD use the disambiguated terms `slash caller` and `invoke parent` whenever the bare word `caller` is ambiguous in context. See: [Slash-Command Invocation](./slash-invocation.md), [Invocation](./invocation.md).

- **callable set** — The unified set of Pi tools and `.loom` callables declared by a loom's frontmatter `tools:` field. Both kinds of entry are callable from the model (during a query's tool-call loop) and from loom code (via the bare `<name>(...)` form). The Pi session's *ambient tool set* (the host's currently-active tools) is deliberately *not* inherited; if `tools:` is omitted the loom runs with an empty callable set. Authors and implementers MUST use the term `callable set` (or, with an explicit scoping modifier, `frontmatter callable set` / `loom's declared callable set`) for the per-loom resolved tool list; the bare phrases `tools: set`, `tool set`, `loom's tools`, and `available tools` are spec-prose synonyms to be avoided because they erode the no-inheritance invariant. The single distinct concept that does use the bare phrase `ambient tool set` is the host Pi session's tools, which a callable set deliberately excludes. Cf. *Pi tool* vs *`.loom` callable* below — those are the two kinds of entry that populate a callable set. See: [Parameters and Frontmatter](./frontmatter.md), [Tool Calls](./tool-calls.md), [Pi Integration Contract — Tool-registration lifetime and visibility](./pi-integration-contract.md).

- **`cause`** — The closed sub-discriminator field that refines a `QueryError.kind` into design-level sub-arms; every variant whose `kind` partitions into multiple causes carries this field, with the same name across variants (`ValidationError.cause`, `CodeToolError.cause`, `InvokeInfraError.cause`). Authors `match` on `cause` to write arm-specific recovery; authors who omit it from the destructure get arm-uniform handling. Distinct from the `reason` field on the `session_shutdown` host event (see [Pi Integration Contract — Session-shutdown semantics](./pi-integration-contract/session-shutdown-semantics.md#session-shutdown-semantics)), which is an event payload, not a `QueryError` sub-discriminator. See: [Errors and Results — QueryError variants](./errors-and-results/queryerror-variants.md#queryerror-variants).

- **canonical schema hash** — The recipe defined in [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash) for content-addressing a lowered JSON Schema fragment: serialise the fragment to a deterministic UTF-8 JSON byte sequence (canonical form), take the SHA-256 digest, and truncate to 16 lowercase hex characters. The 16-hex output is the *schema slug* (below) — "canonical schema hash" names the algorithm, "schema slug" names its output. Spec prose uses "canonical schema hash" only when referring to the recipe or the section that defines it. See: [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash).

- **respond-repair** — The typed-query repair loop that appends a follow-up *user* turn to the same conversation when AJV rejects the model's final response, asking the model to fix its output. Configured by the `respond_repair:` frontmatter block; bounded by `respond_repair.attempts` (default 3); exhaustion surfaces as `Err(QueryError { kind: "validation", cause: "schema_validation", ... })`. Hyphenated as `respond-repair` in prose; underscored as a YAML key. Loom performs no implicit type conversion in expressions; the response-repair loop above is the only "repair" mechanism the spec defines. See: [Query](./query.md).

- <a id="in-loop"></a>**in-loop** — A descriptor for a hard-ceiling enforcement point that surfaces *inside* the tool-call loop as a tool-error result fed back to the model as the next *tool-result* in the conversation, rather than as a `QueryError` returned to loom code or as a `loom-system-note` rendered to the operator. In V1 the only in-loop enforcement point is ceiling #4's model-driven tool-call args row (a `maxDepth` schema-check enforcement point). This enforcement-point firing is not re-surfaced to loom code (no `Err` flows at the firing) or to the operator (no `loom-system-note` is rendered at the firing — the `validation` runtime event does not fire on this row). Both surfaces become reachable only if the loop later exhausts under ceiling #2: at exhaustion, `Err(QueryError { kind: "tool_loop_exhausted", … })` flows to loom code and the corresponding `loom-system-note` is rendered to the operator. Contrast with *query-terminating* and the slash-load `params` cross-routing arm — the three together partition ceiling #4's enforcement-point arms per the per-boundary table cited below. See: [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table).

- **GOV-N (governance rule)** — The numbered governance rules owned by [Governance](./governance.md), which gate how the spec corpus's governed identifiers and corpus-level conventions are coined, anchored, retired, and applied. The live range is the set registered on `governance.md` (currently running through `GOV-23`, with some IDs retired — see [Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids)); cite `governance.md` for the authoritative range rather than a fixed upper bound. The rules cover spec-corpus conventions beyond REQ-ID lifecycle, including stable inline labels (a second governed identifier class), `spec.md` aggregator lock-step, corpus direction and binding scope, release-version naming and legacy-token aliases, the loom 1.x source-language equivalence release-process goal, and `spec.md` Session Model anchor stability. The `GOV` prefix is reserved for these rules in the per-page prefix table; the page that owns the rules is itself a row in that table. See: [Governance](./governance.md).

- **`InvokeInfraError`** / **`kind: "invoke_infra_error"`** — The infra-side invoke failure variant of `QueryError`. The schema name carries an `Infra` qualifier to partition it from `InvokeCalleeError` (the callee's own `Err` propagated through), and the wire `kind` discriminant `"invoke_infra_error"` mirrors that partition. See: [Errors and Results — Invoke variants](./errors-and-results.md), [Invocation — Failures](./invocation.md).

- **loom (file unit)** — A single `.loom` file viewed as a named, invocable unit: the artefact a slash command resolves to, the thing `invoke(...)` targets, the thing whose frontmatter declares `mode:`. Used with an article ("a loom", "the loom"). Lowercase. See: [Overview — Scope of a loom file](./overview.md), [Invocation](./invocation.md).

- <a id="loom-1-0-design-scope"></a>**loom 1.0** / **loom 1.x** / **loom 2.0** (release-version literal, design-scope sense) — Names a major design scope: the entire `<major>.<minor>` line (`loom 1.0`), the entire `<major>` line (`loom 1.x`), or the corresponding scope for a future major (`loom 2.0`). A design-scope literal makes claims about design direction, scope decisions, scope carve-outs, and forward-compatibility seams shared by every release in the named scope; it does not by itself pin closed-enumeration membership at a publishing moment (that is the frozen-baseline sense below). The companion sense, the closure heuristic that selects between the two senses for legacy `V1` callsites, and the legacy-token alias mapping are pinned canonically at [Governance — GOV-19 / GOV-20](./governance/release-version-naming.md#gov-19). See: [Governance](./governance/release-version-naming.md#gov-19).

- <a id="loom-1-0-0-frozen-baseline"></a>**loom 1.0.0** (release-version literal, frozen-baseline sense) — Names a frozen-baseline release: a closed inventory of the release's enumerated invariants at the publishing moment. A frozen-baseline literal makes claims about closed enumerations ("exactly N panic sources", "the closed `details.kind` set", "loads cleanly under"), discriminator inventories, and any other closure-shaped invariant whose membership is fixed at the publishing moment. The companion sense (`loom 1.0` design-scope above), the closure heuristic, and the legacy-token alias mapping are pinned canonically at [Governance — GOV-19 / GOV-20](./governance/release-version-naming.md#gov-19). Inter-release stability between adjacent patch releases (e.g. `loom 1.0.0` vs `loom 1.0.1`) is governed by [GOV-15](./governance/req-id-prefix-table-retired.md#gov-15), not by this entry's referent definition. See: [Governance](./governance/release-version-naming.md#gov-19).

- <a id="operator"></a>**operator** — The human running the Pi TUI session that hosts the loom extension. Distinct from the *author* (who wrote the `.loom`), the *caller* (which may be another loom via `invoke(...)` or a Pi prompt-mode user turn), and any downstream log consumer. Spec prose calling a surface "operator-facing" means the surface is rendered into the active TUI session via the `loom-system-note` channel under `display: true`, where the operator can read it without leaving the session. Surfaces emitted under `display: false` are still on the operator's channel but not rendered into their transcript by default. <a id="operator-v1-invariant"></a>loom 1.0 invariant: every loom invocation runs inside an active Pi TUI session, so an operator is always present; non-interactive invocation paths — namely the deferred `loom test` command and the deferred non-loom programmatic harness named in [Future Considerations](./future-considerations.md) — are out of loom 1.0 scope, and the operator-facing channel's behaviour outside a TUI session is undefined. See: [Pi Integration Contract — System notes](./pi-integration-contract.md), [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md).

- **Loom (language)** — The scripting language whose grammar and semantics this spec defines, shared by `.loom` and `.warp` files. Capitalised when used as a proper noun ("Loom expressions", "Loom code", "Loom follows Rust's…"); the lowercase form is reserved for sense 1 above. See: [Lexical Structure](./lexical.md), [Expression Sublanguage](./expressions.md), [Influences](./influences.md).

- **loom-side name** vs. **wire name** — A pair. The *loom-side name* is the identifier a schema field is declared and referenced by in loom code; it is the only name visible from the language. The *wire name* is the JSON property name used in lowered JSON Schema, in tool input, and in AJV validation, optionally renamed via `field as "WireName": T`. The runtime translates between the two at the validation boundary in both directions; loom code never references the wire name directly. See: [Schema Declarations](./schemas.md); operational detail in [Runtime Value Model](./runtime-value-model.md) and [Schema Subset](./schema-subset.md).

- **no-params bypass** vs. **single-string bypass** — A pair. Both are static binder-bypass conditions decided at loom-load time from the declared `params:` shape; both suppress the binder call entirely and construct no envelope schema. The *no-params bypass* triggers when `params:` is absent or `{}` — the loom takes no parameters and the binder does not run; `bind_echo: true` on such a loom is the load warning `loom/load/bind-echo-without-params` and produces no echo. The *single-string bypass* triggers when `params:` declares exactly one field, that field's type is `string`, and the field has no default — the runtime sets the param's value to the entire slash-argument string (whitespace-trimmed) and skips the binder; AJV validation still runs as a safety net. The no-params check runs before the single-string check, so `params: {}` does not accidentally match the single-string branch. Declaring `bind_echo: true` on a single-string-bypass loom is the parse warning `loom/parse/bind-echo-on-bypass` (the no-params case is the load warning `loom/load/bind-echo-without-params`, above). See: [Slash-Command Argument Binding — Binder bypass](./binder/binder-bypass-and-envelope.md#binder-bypass).

- **PIC** — Abbreviation of *Pi Integration Contract*, the boundary page describing how pi-loom binds to the Pi host SDK. The token is load-bearing rather than cosmetic: the `// allow-pi-surface:` exemption grammar admits a `PIC#<anchor>` citation form, so contributors type the abbreviation directly. See: [Pi Integration Contract](./pi-integration-contract.md).

- **pi-loom (extension)** — The Pi extension that registers `.loom` files as slash commands, owns the discovery walk, and runs loom code. Referred to in prose as `pi-loom` or "the loom extension"; never capitalised "Loom Extension". See: [Pi Extension Integration](./pi-integration.md), [Pi Integration Contract](./pi-integration-contract.md).

- <a id="query-terminating"></a>**query-terminating** — A descriptor for a runtime-class hard-ceiling enforcement point whose breach surfaces as a panic or recoverable `Err` (rather than being absorbed in-loop and fed back to the model). Per-ceiling routing-class resolution: ceiling #1 panics (the panic-class breach unconditionally lands on the *fail* arm); ceiling #2 returns a recoverable `Err`, as do all of ceiling #4's query-terminating points (none panic). For `Err`-class query-terminating points, the breach reaches the *fail* trichotomy arm only if the resulting `Err` value goes unhandled (i.e. propagated via `?` or returned, not consumed by a caller `match` and not discarded via `let _ = …` or the equivalent `void`-tail-expression form); a recovered or discarded `Err` keeps evaluation on the *success* arm (the discard's operator-facing observability is governed by the always-log set per [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md), and applies per-carrier under that set's documented exclusions: of the four `Err`-class query-terminating-arm carriers — `ValidationError` (wire `kind: "validation"`), `CodeToolError` (wire `kind: "code_tool"`), `InvokeInfraError` (wire `kind: "invoke_infra_error"`), and ceiling #2's `ToolLoopExhaustedError` (wire `kind: "tool_loop_exhausted"`) — `ValidationError` is one of the four kinds the always-log set excludes (alongside `cancelled`, `context_overflow`, and `invoke_callee_error`) and so a discarded `ValidationError` is silent on the operator channel even at a discard site, while `CodeToolError`, `InvokeInfraError`, and `ToolLoopExhaustedError` are in the always-log set and DO emit on this channel at a discard site; see also [Query — Observability of discarded results](./query/query-escapes-stringification.md#observability-of-discarded-results)). Panic-class query-terminating points (ceiling #1) always reach the *fail* arm. Used to distinguish those points from *in-loop* enforcement points (above) — in particular ceiling #4's model-driven tool-call args row, which feeds a tool-error result back to the model and does not by itself terminate the query. Ceilings #1 and #2 are uniformly query-terminating; the per-arm classification for ceiling #4 (which enforcement-point arms are query-terminating, which is in-loop, and which cross-routes to ceiling #3's load-time classification) is owned by the per-boundary destination/surface table cited below. Contrast with *in-loop* and the slash-load `params` cross-routing arm — the three together partition ceiling #4's enforcement-point arms per the per-boundary table. See: [`spec.md` — Hard ceilings](../spec/overview-and-orientation.md#hard-runtime-ceilings), [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table).

- **prompt mode** / **subagent mode** — The two execution modes a `.loom` runs under, selected by frontmatter `mode:`. In *prompt mode* the loom drives the user's existing Pi session — its queries become user turns in that session and its emissions are visible to the user. In *subagent mode* the loom spawns a fresh, isolated `AgentSession` whose transcript is private to the loom and discarded when the loom returns. The two modes differ in driver mechanics, in what frontmatter fields are accepted (e.g. `system:` is subagent-only), and in which Pi APIs the runtime calls. See: [Overview and Conceptual Model](./overview.md), [Parameters and Frontmatter](./frontmatter.md), [Pi Integration Contract](./pi-integration-contract.md).

- **schema slug** — The 16-lowercase-hex output of the canonical schema hash recipe (above); a content-addressed identifier for one lowered JSON Schema fragment. Used as the placeholder token `<slug>` in the synthesised `$defs` key `__inline_<slug>` produced by the lowering pass (per [Schema Subset — Lowering Algorithm](./schema-subset.md#lowering-algorithm)) and in the synthesised tool names `__loom_respond_<slug>` and `__loom_callee_<slug>__<post-rename-name>`, as the key of the prompt-mode `pi.registerTool` registration cache (`Map<schemaSlug, registeredToolName>`), and as the AJV compiled-validator cache key. Authors and implementers MUST use `schema slug` (or the bare form `slug` when context is unambiguous). The synonyms `schema hash`, `schema-hash`, `sha12`, `lowered-schema hash`, and `lowered-schema content hash` are spec-prose drift to be avoided; a future grep gate uses this entry as its source of truth. The rename is purely terminology — the on-the-wire tool names (`__loom_respond_<actual-16-hex-chars>` and `__loom_callee_<actual-16-hex-chars>__<name>`) and the `<slug>` placeholder token in diagnostic registry rows and template bodies are unaffected. See: [Schema Subset — Canonical schema hash](./schema-subset.md#canonical-schema-hash).

- **Pi tool** vs. **`.loom` callable** — The two kinds of entry that make up a *callable set* (above). A *Pi tool* is a tool registered against the Pi runtime (built-in or extension-supplied), referenced in `tools:` by name. A *`.loom` callable* is a path entry in `tools:` pointing at a subagent-mode loom file, which the runtime wraps as a tool. The two share the bare-name call form `<name>(args)` and differ only in resolution path and argument shape (Pi tools take a single object matching their input schema; `.loom` callables take their `params:` positionally). Authors and implementers MUST use the canonical term `.loom callable` (or, in running prose, `` `.loom` callable ``) for a `.loom` entry in `tools:`; the synonyms `registered loom`, `registered subagent loom`, `registered-loom call`, and `loom callee` are spec-prose drift to be avoided. The rename does NOT affect any code identifiers, file extensions, diagnostic codes, frontmatter field names, or wire formats — it is purely terminology. See: [Parameters and Frontmatter](./frontmatter.md), [Tool Calls](./tool-calls.md).

- **tool-call round slot accounting** — The per-query bookkeeping that increments the `tool_loop` slot count by one each time a free-phase tool-call round completes (one round = one model turn that emits one or more `tool_use` blocks plus the runtime's parallel execution of those calls and their result feedback). The slot count is the input to ceiling #2's check at the tool-call-round boundary (see [`spec.md` — Hard ceilings](../spec/overview-and-orientation.md#hard-runtime-ceilings) and CIO-4 in [Hard Runtime Ceilings — Interaction between ceilings](./hard-ceilings/ceilings-3-and-4.md#ceiling-interaction-order)): once the slot for the just-completed free-phase round is recorded, ceiling #2 decides whether the next free-phase model turn would exceed `max_rounds`. On untyped queries an over-cap result returns `Err(QueryError { kind: "tool_loop_exhausted", ... })` before any further turn is issued; on typed queries reaching the `max_rounds`-final branch the runtime instead dispatches the forced respond turn as the exempt-routed terminator. The forced respond turn for typed queries is the exempt-routed terminator per [Query — Typed queries are tool-loop-shaped](./query/query-tool-loop.md#typed-queries-are-tool-loop-shaped) and is not itself counted in the slot accounting. See: [Query — Tool-call loop bound](./query/query-tool-loop.md#tool-call-loop-bound), [Parameters and Frontmatter — `tool_loop`](./frontmatter.md).

- **top-level loom** — The outermost block of a `.loom` file. Acts as the implicit function body for the file: its tail expression is the loom's return value, `return expr` exits it, and `?` early-returns from it. Distinct from any nested function bodies declared inside the file. See: [Functions](./functions.md), [Return Statement](./return.md).

- **type sink** — An enclosing AST position whose declared type can supply the schema for a query expression during the outward type-inference walk. Examples include the RHS of a `let x: T = ...`, an argument matched to a typed parameter, and the operand of a `return` from a function with a declared return type. The walk crosses transparent constructs (parenthesisation, ternary branches when the ternary itself has a sink, etc.) and stops at any expression that consumes its operand without preserving its type; if no sink is found, the query is untyped and returns `string`. See: [Query](./query.md).

- **V1** / **V1.0** / **V1.x** / **V2** (legacy release-version aliases) — The legacy spelling for loom release versions, retained as governed aliases under [GOV-20](./governance/release-version-naming.md#gov-20). `V1.x` aliases `loom 1.x` (design-scope only); `V2` aliases `loom 2.0` (design-scope only); `V1` and `V1.0` are sense-overloaded and map to either `loom 1.0` or `loom 1.0.0` per GOV-20's closure heuristic at the callsite. Conversion to the new-scheme spelling is incremental; the spec corpus carries mixed spellings during the transitional period with every untouched callsite governed by the alias mapping. The bare plan-phase identifiers `V1` … `Vn` reserved by `docs/plan_topics/conventions.md:9` are NOT aliases under GOV-20 — those are plan-document identifiers, out of the spec corpus's binding scope per [GOV-17](./governance/corpus-direction-and-scope.md#gov-17) / [GOV-18](./governance/corpus-direction-and-scope.md#gov-18). See: [Governance — GOV-20](./governance/release-version-naming.md#gov-20).

- **`.warp` file (library module)** — A library file containing only top-level `schema`, `enum`, and `fn` declarations, importable from `.loom` and `.warp` files via `import`, never discovered as a slash command, and never invocable via `invoke(...)` or a `tools:` entry. Pairs with `.loom` (the invocable file unit, above): the file-extension split — `.loom` for invocables, `.warp` for libraries — is the spec's only structural separation between callable surface and reusable building blocks. Both extensions are matched byte-exact lowercase per [Lexical — Extension matching](./lexical.md#extension-matching). See: [Imports](./imports.md), [Lexical Structure](./lexical.md), [Directory Convention](./discovery.md).
