# Session model and appendix

## Session Model

<a id="session-model"></a>The runtime's session-level contract decomposes into the eight sub-obligations below; each carries a stable page-local anchor (`sm-1-…` … `sm-8-…`) so inbound citations can pin a single sub-obligation without the others. The umbrella `session-model` anchor is preserved for orientation-level inbound links. This section is **informative orientation** per [Governance — GOV-30](./governance.md): every SM-N obligation's normative contract is owned by a topic page that the corresponding entry forward-links — including [SM-7c](#sm-7c-prompt-mode-sequential-execution)'s prompt-mode sequential-execution conclusion, owned by [Pi Integration Contract — PIC-2](./pi-integration-contract/subagent.md#pic-2), and [SM-8](#sm-8-per-invocation-budget-non-sharing)'s per-invocation budget non-sharing, owned per budget by the topic pages SM-8 forward-links.

<!-- Anchor-scheme stability and orientation-vs-obligation citation rules for the `sm-N-…` anchors relocated to [Governance — GOV-23](./governance/anchor-scheme-and-retired.md#gov-23). -->

- <a id="sm-1-single-active-session-binding"></a> **SM-1 — Single-active-session binding.** A Pi extension instance is bound to exactly one active user session at a time per [Pi Integration Contract — Session-binding contract](./pi-integration-contract/host-prerequisites.md#session-binding-contract). Concurrent *user sessions* in the same host process are out of scope for loom 1.0 — the runtime *presupposes* a single active user session per extension instance rather than having audited Pi's full extension API surface for a concurrent-sessions facet, and the full presupposition framing and the build-time / editorial review surfaces that would catch a future Pi minor introducing such a facet are owned by [Future Considerations — loom 1.0 non-goals](./future-considerations/model-changes-and-non-goals.md#loom-1-0-non-goals), "No concurrent user sessions in the same host process".

- <a id="sm-2-closed-shutdown-reason-set"></a> **SM-2 — Closed `session_shutdown` reason set.** Pi fires `session_shutdown` (`SessionShutdownEvent` in `@earendil-works/pi-coding-agent`'s `dist/core/extensions/types.d.ts`) with `event.reason: "quit" | "reload" | "new" | "resume" | "fork"` (reproduced here for illustration; the SDK `SessionShutdownEvent['reason']` type is the source of truth for both the payload path and the enumeration of the closed reason set, so a future SDK pin bump diff-audits against that interface). SM-2's spec-owned obligation is the routing partition over that reason set — closed-set members vs. unknown reason — whose contract is owned by [Pi Integration Contract — Unknown-reason rule](./pi-integration-contract/unknown-reason-rule.md#unknown-reason-rule). An `event.reason` outside that closed set (or one whose property access throws) routes through the full teardown sequence and emits one `loom/host/session-shutdown-reason-unknown` (W, runtime — see the [Code registry — column legend](./diagnostics/code-registry-parse.md#code-registry)) diagnostic via the teardown-handler last-resort sink defined in [Pi Integration Contract — Extension entry point — Diagnostic-emission isolation](./pi-integration-contract/diagnostic-emission-isolation.md#diagnostic-emission-isolation) — the sink's two sub-obligations, both owned canonically by that linked Diagnostic-emission isolation section, are: **(i) Pi-side stdio visibility** — a deliberate non-claim, not an independently-verifiable obligation, defined at [Pi Integration Contract — Diagnostic-emission isolation — Pi-side stdio visibility](./pi-integration-contract/diagnostic-emission-isolation.md#pic-emission-visibility); and **(ii) handler isolation** — the independently-verifiable swallow obligation, defined at [Pi Integration Contract — Diagnostic-emission isolation — emission swallow](./pi-integration-contract/diagnostic-emission-isolation.md#pic-emission-swallow); the validation, fail-safe routing, and `details.event.reason` widening on `loom/runtime/cancelled-by-session-shutdown` are owned by [Pi Integration Contract — Extension entry point, step 4](./pi-integration-contract.md).

- <a id="sm-3-session-shutdown-handler"></a> **SM-3 — Session-shutdown handler.** SM-3 groups the two handler-internal obligations that run inside the `session_shutdown` handler — the per-reason fixed teardown sequence (SM-3a) and the PIC-owned session-swap semantics for in-flight invocations (SM-3b, whose contract is owned by PIC rather than by the loom runtime's handler) — each carrying a stable page-local anchor (`sm-3a-…` / `sm-3b-…`) so inbound citations can pin a single sub-obligation without the other. The umbrella `sm-3-session-shutdown-handler` anchor is preserved for orientation-level inbound links.

  - <a id="sm-3a-teardown-sequence"></a> **SM-3a — Fixed teardown sequence.** The runtime's handler runs the full fixed teardown sequence — including watcher closure in sub-step 4 — for every reason in the closed set. The fixed teardown sequence itself is owned by [Pi Integration Contract — Extension entry point, step 4](./pi-integration-contract.md).

  - <a id="sm-3b-session-swap-semantics"></a> **SM-3b — Session-swap semantics for in-flight invocations.** Session-swap semantics for in-flight invocations — the abort-and-await sequence, the per-invocation operator-visibility surface, partial-append fate, and the `invoke`-parent observation rule — are owned by [Pi Integration Contract — Session-swap behaviour for in-flight invocations](./pi-integration-contract/session-shutdown-semantics.md#session-shutdown-semantics). *Implementation gate:* SM-3b is gated on the same open contradiction recorded at [Pi Integration Contract — Host prerequisites for the degraded-state branch, clause (a)](./pi-integration-contract/host-prerequisites.md#degraded-state-host-prerequisites).

- <a id="sm-4-session-only-degraded-state"></a> **SM-4 — Post-teardown degraded state for the session-only reasons.** On the *session-only reasons* (`event.reason ∈ {"new", "resume", "fork"}`) the runtime remains live afterwards in a deterministically degraded state until operator-issued `/reload`. The degraded-state contract is owned by [Pi Integration Contract — `reason: "new" | "resume" | "fork"` edge-case bullet](./pi-integration-contract/session-only-degraded-state.md#session-only-reason-degraded-state). *Implementation gate:* implementing SM-4 is blocked pending resolution of the open contradiction recorded at [Pi Integration Contract — Host prerequisites for the degraded-state branch, clause (a)](./pi-integration-contract/host-prerequisites.md#degraded-state-host-prerequisites).

- <a id="sm-5-tag-transition-predicate"></a> **SM-5 — Tag-transition predicate scope.** PIC's **tag-transition predicate** covers the [session-only reasons](#sm-4-session-only-degraded-state) directly *plus* the snapshot-failure and unknown-reason-with-successful-snapshot paths. The post-sub-step-4 `LoomRegistry` drain-state transition and the `loom /<name>: extension degraded; /reload to recover` system note that fronts subsequent slash invocations (with editor-driven `.loom` and settings edits silently no-opping until recovery) follow the tag-transition predicate. The predicate definition and the per-path scoping for tag-transition and slash-note coverage are owned by [Pi Integration Contract — `reason: "new" | "resume" | "fork"` edge-case bullet](./pi-integration-contract/session-only-degraded-state.md#session-only-reason-degraded-state). *Implementation gate:* SM-5 is gated on the same open contradiction recorded at [Pi Integration Contract — Host prerequisites for the degraded-state branch, clause (a)](./pi-integration-contract/host-prerequisites.md#degraded-state-host-prerequisites).

- <a id="sm-6-diagnostic-emission-predicate"></a> **SM-6 — Diagnostic-emission predicate scope.** PIC's **diagnostic-emission predicate** — narrower than [SM-5's tag-transition predicate](#sm-5-tag-transition-predicate) — is scoped to the [session-only reasons](#sm-4-session-only-degraded-state) AND a successful snapshot read of the SDK pinned-constant; the snapshot-failure path is excluded by the snapshot-success conjunct. The new `loom/host/session-shutdown-runtime-degraded` (W, runtime) diagnostic follows the diagnostic-emission predicate (the snapshot-failure and unknown-reason paths cover their own diagnostic-channel observability via `loom/host/session-shutdown-pinned-constant-unreadable` and `loom/host/session-shutdown-reason-unknown` rather than a co-emitted `runtime-degraded` row). The predicate definition and the per-path scoping for emission coverage are owned by [Pi Integration Contract — `reason: "new" | "resume" | "fork"` edge-case bullet](./pi-integration-contract/session-only-degraded-state.md#session-only-reason-degraded-state). *Implementation gate:* SM-6 is gated on the same open contradiction recorded at [Pi Integration Contract — Host prerequisites for the degraded-state branch, clause (a)](./pi-integration-contract/host-prerequisites.md#degraded-state-host-prerequisites).

- <a id="sm-7-mode-qualified-concurrency"></a> **SM-7 — Mode-qualified isolation contract.** Concurrent loom invocations within a session carry **mode-qualified** isolation; the contract decomposes into the five sub-obligations below, each with a stable page-local anchor (`sm-7a-…` … `sm-7e-…`) so inbound citations can pin a single sub-obligation without the others. The umbrella `sm-7-mode-qualified-concurrency` anchor is preserved for orientation-level inbound links.

  - <a id="sm-7a-cancellation-independence"></a> **SM-7a — Cancellation independence across modes.** *Cancellation* is always independent: every invocation — prompt mode and subagent mode alike — carries its own `AbortController` (`loomAbort`) per [Cancellation — Signal source](./cancellation.md) and is tracked as a distinct entry in an extension-scoped registry of in-flight invocations whose normative shape, lifecycle, and teardown semantics live at [Pi Integration Contract — `ActiveInvocationRegistry`](./pi-integration-contract/active-invocation-registry.md#active-invocation-registry).

  - <a id="sm-7b-subagent-mode-isolation"></a> **SM-7b — Subagent-mode transcript and tool-table isolation.** *Transcript and tool-table isolation* are **subagent-mode-only** — a subagent-mode invocation runs its own private `AgentSession` (no shared transcript or `tools:` table) per [Pi Integration Contract — Concurrent-invocation isolation and concurrency disposition](./pi-integration-contract/subagent.md#concurrent-invocation-isolation), while a prompt-mode invocation drives the user session's transcript by definition.

  - <a id="sm-7c-prompt-mode-sequential-execution"></a> **SM-7c — Prompt-mode sequential execution.** *Prompt-mode bodies execute strictly sequentially* within a single user session — at most one prompt-mode body holds an open `pi.setActiveTools` snapshot/restore window at a time — is owned by [Pi Integration Contract — PIC-2 *Prompt-mode sequential execution*](./pi-integration-contract/subagent.md#pic-2), which derives the property from (i) <a id="sm-7c-slash-handler-serialisation"></a>Pi's per-session slash-handler serialisation, (ii) <a id="sm-7c-load-time-prompt-callee-rejection"></a>the load-time rejection of prompt-mode `.loom` callees in another loom's `tools:` per [Parameters and Frontmatter — `tools`](./frontmatter/frontmatter-fields-a.md#tools), (iii) <a id="sm-7c-invoke-suspension"></a>the parent-suspending `invoke(...)` rule and the absence of a parallel-`invoke` surface owned by [Invocation — Cross-mode semantics](./invocation.md), and (iv) <a id="sm-7c-subagent-to-prompt-fanout-closed"></a>the closure of the `subagent → prompt` fan-out path into the user session owned by the same [Invocation — Cross-mode semantics](./invocation.md) rule.

  - <a id="sm-7d-no-cap-no-scheduler"></a> **SM-7d — loom 1.0 no-cap / no-scheduler disposition.** loom 1.0 concurrency disposition for in-flight invocations — no admission cap, no scheduler interposed above Pi's event loop — is owned by [Pi Integration Contract — No invocation cap](./pi-integration-contract/subagent.md#no-invocation-cap).

  - <a id="sm-7e-downward-only-cancellation"></a> **SM-7e — Downward-only cancellation propagation.** Cancellation propagates downward only (parent → children) per [Cancellation — Propagation](./cancellation.md); a sibling invocation's cancellation does not affect its siblings.

- <a id="sm-8-per-invocation-budget-non-sharing"></a> **SM-8 — Per-invocation budget non-sharing.** Each per-invocation budget — the binder retry budget, the `tool_loop.max_rounds` budget, and the `invoke`-chain depth budget — is scoped to a single invocation and is not shared, pooled, or replenished across sibling invocations; this non-sharing rule is owned, per budget, by its topic page: the binder retry budget by [Slash-Command Argument Binding — per-invocation retry budget](./binder/determinism-cancellation-failure.md#per-invocation-retry-budget), the `tool_loop.max_rounds` budget by [Query — Tool-call loop bound](./query/query-tool-loop.md#tool-call-loop-bound), and the `invoke`-chain depth budget by [Invocation — Invocation depth bound](./invocation.md).
