# Frontmatter fields b and templates

    YAML-shape: `tools:` accepts two interchangeable spellings ‚Äî a comma-separated short form and a YAML list form ‚Äî and both are parsed by the same per-entry grammar defined in [Parameters and Frontmatter ‚Äî `tools`](frontmatter-fields-a.md#tools). The comma form is the YAML plain scalar split on commas, each resulting entry trimmed of surrounding whitespace and then parsed by that grammar; the list form takes one entry per sequence item. A `.theta` path or an `as` rename is therefore legal in either form ‚Äî entry boundaries stay unambiguous because neither a `.theta` path nor an `as` target contains a comma ‚Äî and a malformed entry surfaces the same load diagnostic (`theta/load/invalid-tool-rename`, `theta/load/unresolvable-theta-path`, etc.) regardless of which spelling produced it. The short form reads best for plain Pi-tool names; the list form reads best once entries carry paths or `as` renames:

    ```yaml
    tools: read, grep, bash
    ```

    ```yaml
    tools:
      - read
      - bash
      - ./summarise.theta              # callable as `summarise`
      - ./code-review.theta            # callable as `code_review`
      - ./classify.theta as triage     # callable as `triage`
    ```

    Unknown Pi tool names (`theta/load/unknown-tool`), unresolvable `.theta` paths (`theta/load/unresolvable-theta-path`), `.theta` callees that fail to parse during the load pass (`theta/load/callee-has-errors`, severity `error` for `tools:` entries per [Invocation ‚Äî Static resolution](../invocation.md#static-resolution)), prompt-mode theta paths (`theta/load/prompt-mode-callable`), name collisions (`theta/load/tool-name-collision`), and invalid `as` targets (`theta/load/invalid-tool-rename`) all surface through the theta diagnostics channel defined in [Diagnostics](../diagnostics.md) and prevent the theta from being registered.

    The parsed callee captured at load time ‚Äî used both for return-type inference at code-side call sites and for the lowered tool spec exposed to the model ‚Äî comes from the same per-load-pass parse cache `invoke(...)` static resolution uses (see [Invocation ‚Äî Static resolution](../invocation.md#static-resolution)). One walk of the reachable graph supplies both surfaces; a callee parsed for `tools:` is not re-parsed when an `invoke(...)` literal in the same parent references it.

    <a id="resolution-snapshot"></a>**Resolution snapshot.** Theta-load time resolution produces a frozen per-theta table of `{ post-rename name ‚Üí resolved callable }` entries. Each Pi-tool entry holds a strong reference to the resolved `ToolDefinition` object (its `execute`, `parameters`, and metadata as returned by Pi's tool registry at the moment of load); each `.theta`-path entry holds a strong reference to the parsed callee plus its lowered tool spec (params ‚Üí input schema, inferred return ‚Üí output schema, frontmatter `description` ‚Üí tool description). Subsequent calls ‚Äî both model-driven (during a `@`...`` query's tool-call loop) and code-driven (`<name>(args)`) ‚Äî dispatch through the held reference; the runtime never re-queries Pi's tool registry by name during execution. Consequences:

    - Unregistering a Pi tool from Pi's registry mid-run does not affect calls from a theta that already resolved it; the theta continues to use the captured `execute` function until it terminates.
    - The `unknown_tool` cause on `CodeToolError` (see [Tool Calls](../tool-calls.md)) is reachable only via the file-watcher reload path: when the watcher rebuilds a theta's table and a previously-resolved Pi tool is no longer in Pi's registry, the *next* invocation of the rebuilt theta records `theta/load/unknown-tool` and refuses to register; in-flight invocations against the previously-built table run to completion.
    - For `.theta`-callee entries, the held reference points at the parsed callee version captured at load. File-watcher reloads invalidate the schema cache and rebuild `tools:` tables on the next invocation; an in-flight invocation completes against its captured callee parse.
    - **Subagent leg (child-load snapshot).** Because the RFC 0006 child-process theta design runs the **whole callee** ‚Äî interpreter included ‚Äî inside the child, the resolution snapshot for a subagent-mode theta is taken by the **child at child load**, not held as a marshalled strong `ToolDefinition` reference in the parent: no executable definition crosses the process boundary. The parent's role reduces to a **name/schema-only admission**: at parent load the parent verifies each callable-set entry resolves by name against `pi.getAllTools()` and, for the RFC-0002 disjointness check, reads its schema ‚Äî it does **not** materialise or hold an `execute`. The child re-runs the full resolution at child startup against its own on-disk discovery: built-ins and extension tools resolve by name in the child's registry, and each `.theta` callable's discovered source is verified against a **content hash of the whole callee file plus its transitive `.thetalib` imports**, marshalled by the parent at spawn. The parent-load/child-load skew window RFC 0005 covered for the model-facing `.theta` callables therefore **widens to the whole callee file** (see [Subagent ‚Äî `.theta` callable content-hash verification](../pi-integration-contract/subagent.md#subagent-theta-callable-hash)); a `.theta` callable whose child-side hash differs from the parent-marshalled hash **refuses the invocation** with `theta/runtime/subagent-callable-hash-mismatch` (child-verified, fail-closed), and a marshalled model reference the child re-resolves to the wrong model fails with `theta/runtime/subagent-model-preflight-mismatch` (per [PIC-62](../pi-integration-contract/subagent.md#pic-62)). Under RFC 0006 the parent interpreter no longer runs the subagent body, so the parent holds no code-side `execute` reference for a subagent callee ‚Äî code-side dispatch (including extension-tool host-loop dispatch) runs inside the child per [PIC-61](../pi-integration-contract/subagent.md#pic-61); the parent-held strong references in this snapshot back code-side `<name>(args)` dispatch only for **prompt-mode** thetas, whose interpreter runs in the parent.
    - Hot-reloading a Pi extension whose tools are held by a running theta (`ctx.reload()` on the source extension) leaves the captured `execute` closure referencing disposed module state, but the held reference still has a contracted outcome. The runtime has no mechanism to detect the orphaned reference, so it dispatches through the captured `execute` like any other captured Pi-tool callable (per the **Resolution snapshot** rule above). Whatever the stale closure does routes through the standard `execute()` outcome surfaces ‚Äî a catchable throw lowers to `Err(CodeToolError { cause: "execution" })`, and a return value that violates the `{ content, isError }` envelope routes to `theta/runtime/internal-error`, both defined in [Tool Calls](../tool-calls.md); an uncatchable host-fatal error surfaces through the [Errors and Results ‚Äî Runtime panics](../errors-and-results/error-model.md#runtime-panics) surface. Authors who need clean swap-out of tool implementations should rely on the full re-discovery semantics of Pi's `/reload` (`ctx.reload()`, which re-executes the entire extension factory; see [Pi Integration Contract](../pi-integration-contract.md)) rather than reloading individual extensions mid-theta.
  - Per-query overrides and a project ‚Üí theta ‚Üí query cascade are deferred (see [Future Considerations](../future-considerations.md)).
- `system` declares the conversation's system prompt. **Subagent-mode only** ‚Äî `system:` in a `mode: prompt` theta is `theta/parse/system-on-prompt-mode`, since prompt-mode thetas attach to the user's existing Pi session whose system prompt belongs to Pi, not to the theta. In subagent mode, the field is fixed once when the spawned conversation is created and applies to every query the theta issues against it. If omitted, the spawned conversation has no system prompt (the model behaves under its training defaults).
  - **`subagent fn` `with { system: ‚Ä¶ }` (theta 1.2).** The five session-config keys of a `subagent fn`'s `with { ‚Ä¶ }` clause ‚Äî `system`, `model`, `tools`, `tool_loop`, `respond_repair` ‚Äî are validated against the same rules as the like-named frontmatter fields and **reuse those fields' diagnostics** (see [Functions ‚Äî FN-7](../functions.md#fn-7)). The one carve-out is `theta/parse/system-on-prompt-mode`: it does **not** apply to a `subagent fn`'s `with`-clause `system`, because that `system` prompt configures the function's own private spawned session rather than the enclosing conversation. A `subagent fn` declared inside a `mode: prompt` theta may therefore set `with { system: ‚Ä¶ }` even though the enclosing theta cannot carry `system:` in its own frontmatter.
  - **Interpolation.** The `system:` field supports `${param}` and `${param.field}` interpolation against the theta's typed `params`. The full Theta expression sublanguage is **not** available in this slot ‚Äî only bare identifier paths ‚Äî because the system prompt is evaluated once at conversation-creation time, before any theta code runs, and the simpler rule is unambiguous and easy to debug. For richer logic, omit `system:` and accept the reduced control-flow surface. The three surfaces of the slot ‚Äî what parses, what string the model sees, and which diagnostics fire ‚Äî are pinned down below.

    *Path grammar.*

    ```
    SystemInterp := '${' Path '}'
    Path         := Ident ('.' Ident)*
    ```

    `Ident` is the lexical identifier from [Lexical Structure ‚Äî Identifiers](../lexical.md) (case-sensitive, exact byte match against the declared `params` field name). There is no depth bound ‚Äî arbitrary chains of `.Ident` are accepted. The head `Ident` must name a declared `params` entry; each subsequent `.Ident` must name a reachable field of an *object* schema in the theta-side `params:` declaration. `${param.field}` paths use theta-side `params` names throughout ‚Äî never an `as "WireName"` rename target ‚Äî consistent with the [Runtime Value Model](../runtime-value-model.md) invariant that theta code never sees wire names and the object-indexing rules in [Expression Sublanguage](../expressions.md). Arrays, discriminated unions, and primitive scalars terminate the path: `${param}` (no further `.`) is always allowed and is rendered by the stringification rule below; `${param.field}` is only allowed when the resolved type one step in is an object. A path that descends into an arm of a discriminated union without a discriminator narrowing is rejected at parse time ‚Äî theta 1.0 has no narrowing in this slot.

    Indexed access (`${arr[0]}`), call syntax (`${f()}`), optional chaining (`${a?.b}`), arithmetic, and any other expression form are not part of the grammar and produce a parse error (see below). For richer logic, omit `system:` (see [Future Considerations](../future-considerations.md), "Richer expression sublanguage inside frontmatter `system:`").

    *Stringification.* Resolve the path against the validated params object, then render the resolved value by its static type. The rule is the canonical interpolation-stringification table defined once in [Query ‚Äî Stringification of interpolated values](../query.md): `@`...`` query templates and `system:` paths use the same table so that the model sees the same rendering of a given value regardless of which surface introduced it. Resolution happens once, at conversation-creation time; a param whose resolved value is `null` renders as the literal text `null` (per the canonical table), not the empty string. The `Result<T, E>` row of that table cannot fire here ‚Äî `params:` types do not include `Result`. The `number` row's `NaN` / `¬±Infinity` cases *are* reachable from this slot: a `number`-typed param supplied through the `invoke(...)` or `.theta`-callable argument path can carry a non-finite IEEE-754 double (the slash-command path enters through JSON, which has no `NaN` / `¬±Infinity` literal, so only the non-slash arms introduce them), and such a value renders per the `number` row of the canonical table (e.g. `NaN` renders as the literal text `NaN`).

    *Escapes.* A literal `${` is written `\${` inside the YAML block scalar ‚Äî the same escape used in `@`-template bodies (see [Template Interpolation](#template-interpolation)). The backslash survives YAML processing and suppresses interpolation only when the next character is `{`; in any other position `\` is passed through verbatim.

    *Parser entry point.* The `${‚Ä¶}` syntax in `system:` is parsed by the **same expression entry point** the [Expression Sublanguage](../expressions.md) uses for `${‚Ä¶}` interpolation in `@`...`` query templates, restricted at parse time to the `Path` production above by a parser-level filter (anything else fires `theta/parse/system-interp-not-path`). The restriction is applied to the parsed expression, not by a separate hand-rolled `${param}` regex; the seam is what allows the deferred richer-expression extension in [Future Considerations](../future-considerations.md) to land by widening the filter rather than rewriting the parser.

    > **theta 1.0 seam ‚Äî `system:` expression sublanguage.** <a id="theta-1-0-seam-system-expression-sublanguage"></a> A `${‚Ä¶}` body in `system:` MUST be a `Path` (the `Path` production above); any other body fires one of the four `theta/parse/system-interp-*` diagnostics listed under *Parse errors* below. *Non-normative implementation note.* theta 1.0 realises this restriction by reusing the shared `${‚Ä¶}` expression entry point (see *Parser entry point* above, which routes `system:` through the same [Expression Sublanguage](../expressions.md) parser as `@`...`` query templates) and applying a parser-level filter, so the deferred richer-`system:`-expression extension in [Future Considerations ‚Äî Surface extensions](../future-considerations/surface-extensions.md#surface-extensions-v1-leaves-a-seam) widens the filter rather than rewriting the parser; the implementation shape is not an observable conformance point.

    *Parse errors.* All four codes fire at frontmatter-parse time, before AJV validates the params payload, and are listed in [Diagnostics](../diagnostics.md):

    - `theta/parse/system-interp-not-path` ‚Äî the body of `${...}` is not a `Path` (e.g. `${arr[0]}`, `${a + b}`, `${f(x)}`, `${a?.b}`, `${"x"}`).
    - `theta/parse/system-interp-unknown-param` ‚Äî the head `Ident` is not a declared `params` field.
    - `theta/parse/system-interp-bad-field` ‚Äî a `.Ident` step does not name a reachable theta-side `params` object field (or attempts to descend into an array or un-narrowed discriminated union).
    - `theta/parse/system-interp-unterminated` ‚Äî `${` is not closed by a matching `}` before the YAML scalar ends.
- `respond_repair` controls how typed queries recover from schema-validation failures (see the [Query](../query.md) section). `attempts` bounds the number of follow-up respond-repair turns; `methodology` selects the phrasing strategy. The block configures **respond-repair** ‚Äî the response-repair mechanism specified in [Query ‚Äî Schema-validation respond-repair](../query.md). The verb "retry" is reserved for genuinely retry-shaped behaviour elsewhere in the spec (`TransportError.retryable`, the binder's single-shot transport retry); they are different mechanisms. Recognised methodologies (theta 1.0) ‚Äî the literal user-turn text each non-`none` methodology emits is fixed by [Query ‚Äî Follow-up turn templates (normative)](../query/query-failure-and-repair.md#follow-up-turn-templates-normative); the bullets below name the methodology and its summary intent, the templates carry the wording:
  - `validator_error` (default) ‚Äî the follow-up turn includes the AJV validation error from the previous attempt. Wording per [Query ‚Äî Follow-up turn templates (normative)](../query/query-failure-and-repair.md#follow-up-turn-templates-normative).
  - `schema_repeat` ‚Äî the follow-up turn re-states the expected schema without quoting a specific error. Wording per [Query ‚Äî Follow-up turn templates (normative)](../query/query-failure-and-repair.md#follow-up-turn-templates-normative).
  - `none` ‚Äî no follow-up; the first failure is returned as `Err` immediately. Equivalent to `attempts: 0`. No template applies (no turn is emitted). A declared non-zero `respond_repair.attempts` is silently ignored under `methodology: none` ‚Äî the theta loads with no diagnostic (`theta/load/unknown-methodology-value` fires only on values outside the recognised set, and `none` is in the set) and behaves exactly as `attempts: 0`. Accordingly `attempts` on the returned [`ValidationError`](../errors-and-results/queryerror-variants.md#query-time-variants) is `0` on every path that surfaces a `ValidationError` under `methodology: none`: initial AJV failure, depth-walk failure, and the [ERR-17](../errors-and-results/queryerror-variants.md#err-17) forced-respond branch.

  An absent `methodology:` defaults to `validator_error` (the default in the field-contract table above). A present `methodology:` value outside the recognised set ‚Äî including a non-string scalar ‚Äî is the load-time error `theta/load/unknown-methodology-value` and the theta is not registered, the same recognised-key / unrecognised-value split as `mode:`.
- <a id="frnt-1"></a> **FRNT-1.** `tool_loop` bounds the model's tool-call loop (see [Query ‚Äî Tool-call loop bound](../query.md)). `max_rounds` is a non-negative integer (theta 1.0 imposes no upper bound); one round is the model emitting one or more `tool_use` blocks, the runtime executing them all (in parallel where the provider supports parallel tool calls), and feeding the results back. `max_rounds` counts free-phase rounds only; the forced respond turn that terminates a typed query is the exempt-routed terminator dispatched by CIO-4's `max_rounds`-final branch (see [Hard Runtime Ceilings ‚Äî CIO-4](../hard-ceilings/ceilings-3-and-4.md#cio-4)) and is not counted against `max_rounds` ‚Äî the runtime issues it on every typed query reaching that branch, including `max_rounds: 0` where it is the only turn. The cap applies independently to each query and, separately, to each respond-repair follow-up (a follow-up does **not** inherit a depleted budget from the original turn) ‚Äî and to every query inside an `invoke`d callee, which uses the callee's own `tool_loop` frontmatter; the parent's budget is not debited by `invoke`. When the cap is reached without the model producing a terminating turn, the query returns `Err(QueryError { kind: "tool_loop_exhausted", ... })`. `max_rounds: 0` disables model-driven tool calls entirely ‚Äî the model cannot use any frontmatter `tools:` entry during the query, while theta code's bare `<name>(...)` calls remain unaffected. Cancellation via `AbortSignal` preempts the loop at any round boundary; the cap is a ceiling, not a floor. Out-of-range values (negative integers, non-integer numbers such as `25.5`, non-number scalars such as `"25"`, or `null`) are rejected at frontmatter-parse time as `theta/load/frontmatter-value-out-of-range`; YAML integer-ness is judged on the parsed numeric value, not the lexical form (`25` and `25.0` are both accepted). The same out-of-range rule applies to `respond_repair.attempts`. theta 1.0 provides no operator-level override for `max_rounds`; the value is per-theta only. Adding a `theta.toolLoopMaxRounds` settings key is deferred per [Future Considerations](../future-considerations.md).

## Template Interpolation

A `${...}` interpolation inside a `@`...`` query template contains a Theta expression from the [Expression Sublanguage](../expressions.md), evaluated up to the matching `}`. The `@` character has only one lexical role ‚Äî introducing a query template at top level ‚Äî and never appears inside `${...}`. There is no bash-style argument-slice sugar (`${@:N}`, `$1`, `$@`, `$ARGUMENTS`); slash-command arguments are bound to typed `params` via the [Slash-Command Argument Binding](../binder.md) machinery and referenced by their declared parameter names like any other identifier.

The interpolated value's rendering into the prompt text ‚Äî by the expression's Theta static type, with parse-time rejection of `Result<T, E>` interpolands ‚Äî is specified once in [Query ‚Äî Stringification of interpolated values](../query.md). The same canonical table covers the bare-path `${param}` / `${param.field}` form in the `system:` field above.
