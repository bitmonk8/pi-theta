# Governance

*Audience.* This page is directed at spec maintainers and at automated tooling that maps REQ-IDs to implementation work, not at implementers of the loom runtime. It governs the spec corpus itself — how REQ-IDs are coined, anchored, and retired — and sources no software requirement on the runtime, the binder, the type system, or the Pi integration.

This page owns the spec corpus's REQ-ID governance: the per-page prefix table, the per-prefix retirement registry, the corpus-direction and binding-scope rules (GOV-17 / GOV-18), and the rules (GOV-1, GOV-3 through GOV-9, GOV-12, GOV-14 through GOV-16; GOV-2, GOV-10, GOV-11, GOV-13 retired — see [Retired REQ-IDs](#retired-req-ids)) that govern how REQ-IDs are coined, anchored, and retired. GOV-16 additionally governs a second cross-page-citable identifier class — *stable inline labels* (currently `HC3-a`…`HC3-e` and `NOCEIL-1`…`NOCEIL-4`, both bound to [`hard-ceilings.md`](./hard-ceilings.md)) — on terms parallel to GOV-1 / GOV-3 / GOV-8 / GOV-9. The rules apply to every non-narrative page under `spec_topics/`.

## Corpus direction and binding scope

<a id="gov-17"></a> **GOV-17 (corpus direction).** The spec corpus comprises `docs/spec.md` and `docs/spec_topics/*.md`. A *cross-reference* from one document to another is a CommonMark link (per the CommonMark Spec §6.3 Links; subsequent uses of "Markdown link" in this rule denote a CommonMark link) in the first document's body — either the inline form `[text](path)` or the reference-style form `[text]: path` — whose `path` (parsed per the URL/URI grammar of RFC 3986 §3.3 (path), §3.4 (query), §3.5 (fragment), with any `#fragment` and `?query` components stripped before path resolution), resolved relative to the linking document's directory, is the filesystem path of the second document. Back-ticked path mentions inside `code spans`, prose filename mentions in running text, and any byte-level substring match outside Markdown-link syntax are NOT cross-references under this rule; this includes the back-ticked path mentions in GOV-17's own body and the back-ticked path mentions cited as per-rule reasons in the *Retired REQ-IDs* sub-table. A document or resource outside the spec corpus is a *dependency* of the spec corpus iff its body contains no cross-reference into the spec corpus, and a *dependent* iff it does. The spec corpus MAY cross-reference its dependencies and MUST NOT cross-reference its dependents — equivalent to "no circular Markdown-link cross-references between the spec corpus and any other document". The dependent check is operational on the candidate target's side: enumerate the Markdown links in the target's body, resolve each `path` against the target's directory, and test whether the resolved path is `docs/spec.md` or matches `docs/spec_topics/*.md`; any such link makes the target a dependent. (Non-normative aid: `grep -nE '\]\([^)]*spec(\.md|_topics/)' <target>` flags candidate Markdown-link callsites for inspection; a precise check still requires resolving each link's `path` against the target's directory, since the regex over-matches incidental substrings and the relative-path forms `spec.md` / `./spec.md` / `../spec.md` / `../../docs/spec.md` all resolve to the same spec-corpus files.) The currently-known dependents, all forbidden as cross-reference targets from the spec corpus, are: the plan documents (`docs/plan.md` and `docs/plan_topics/*.md`), the repo-level `README.md`, and `CHANGELOG.md`. External non-Markdown resources (third-party documentation rendered as HTML, package documentation under `node_modules/`, third-party websites, PDFs, plain-text files) carry no Markdown-link `[text](path)` / `[text]: path` syntax in their bodies and are therefore vacuously dependencies under this rule — the operational dependent check above (which enumerates Markdown links in the target's body) returns empty on them by construction, so the spec MAY cite them without running the check. External Markdown resources (e.g. a third-party `.md` file reachable through the citing link's resolved `path`) ARE subject to the operational dependent check on the same terms as in-repo Markdown. The rule has no carve-out for forward-reference, future-tense, or hypothetical citations: a dependent is forbidden as a cross-reference target regardless of the citing prose's modality.

*Triggering and scope (non-normative).* GOV-17's MUST-NOT is gated per-commit on the same per-page terms as GOV-1 *Per-page progressive normalisation*: it is evaluated against cross-references the commit's first-parent diff adds, edits, or moves in the spec corpus, not against the spec corpus as a whole on every commit. The rule does NOT impose a corpus-wide audit obligation on every commit; pre-existing in-spec cross-references the commit does not touch are not gated by this rule on the current commit, and a pre-existing in-spec cross-reference that becomes non-conformant solely by a change outside the spec corpus (e.g. a commit on `docs/plan.md` or `docs/plan_topics/*.md` that adds a link into the spec, newly classifying its source document as a dependent under the operational dependent check above) is a standing spec defect actionable when a future commit touches the in-spec cross-reference's site. The rule pins the cross-reference-direction invariant but does NOT specify auditing machinery; per [GOV-18 arm (b)](#gov-18-arm-b), the audit lives outside the spec corpus.

<a id="gov-18"></a> **GOV-18 (binding scope).** The spec corpus's normative obligations bind exactly two parties:

- <a id="gov-18-arm-a"></a> **Arm (a) — the implementation target.** The loom runtime, the binder, the type system, the Pi integration, and any other software whose behaviour the spec exists to constrain. *Operational test for the "any other software" tail.* A program is arm (a) iff its externally-observable behaviour on `.loom` or `.warp` inputs is described by the spec; programs whose behaviour on `.loom` / `.warp` inputs the spec does not describe fall outside arms (a) and (b), regardless of where they ship. Borderline cases the test resolves include software shipped inside the loom repo alongside the runtime (e.g. a `loom doctor` sub-command, a build-time validator embedded in the loom binary, a developer-tooling library invoked by the loom runtime) — arm (a) when the test holds, third-party tooling otherwise. A normative obligation whose subject is one of these is spec content.
- <a id="gov-18-arm-b"></a> **Arm (b) — the spec corpus itself.** Invariants about the form, structure, and content of `docs/spec.md` and `docs/spec_topics/*.md` — including REQ-ID grammars, anchor placement, prefix-table closure, retirement procedures, and cross-link form. A normative obligation whose subject is the spec corpus is self-binding and is spec content.

<a id="gov-18-binding-prohibition"></a> The spec corpus MUST NOT bind any third party — including but not limited to documentation tooling, CI gates, coverage matrices, REQ-ID extractors, doc generators, the project `README.md`, the `CHANGELOG.md`, or any other document or program outside arms (a) and (b). Where third-party tooling needs to consume a published spec invariant, the spec publishes the invariant as a property of its own content (arm (b)); this rule takes no position on whether or how the third party records any obligations of its own. A spec rule whose pre-diff wording cited a verification venue (e.g. "the coverage-matrix gate enforces this", "the anchor-insertion CI gate") loses that citation under this rule; the bare invariant survives as arm (b) content, and auditing the invariant is third-party tooling's responsibility carried outside the spec corpus. No verifier-replacement obligation is created by this rule.

<a id="gov-18-internals-prohibition"></a> The spec corpus MAY name a downstream consumer in implementation-neutral terms (e.g. "automated tooling that maps REQ-IDs to implementation work") to anchor an obligation's *purpose*; it MUST NOT specify the consumer's internals, workflow, configuration, or failure surface.

Worked examples — each retiring or surviving rule classified per the arm-membership test:

- **GOV-1 (REQ-ID anchor placement) — arm (b).** Constrains the spec corpus's own anchor sites; survives.
- **GOV-5 (disjoint-prefix invariant) — arm (b).** Constrains the spec corpus's own identifier-token grammar; survives as reshaped (the consumer-side MUST is rewritten as a spec-side invariant in the version of GOV-5 below).
- **Retired GOV-2.** Bound a downstream coverage-matrix tracker to map every REQ-ID to a closing leaf and exit non-zero on unmapped IDs; the tracker is neither the implementation target (arm a) nor the spec corpus itself (arm b).
- **Retired GOV-10.** Bound an implementer's reading-scope to a downstream document's per-leaf `**Spec**` field; the implementer's reading process is neither the implementation target (arm a) nor the spec corpus itself (arm b).
- **Retired GOV-11.** Bound a downstream document's `**Spec**` field to a closure-under-cross-link obligation; the downstream document's data-structure shape is neither the implementation target (arm a) nor the spec corpus itself (arm b).

The implied-consumer edge case is permitted: a rule whose surface wording presupposes a consumer (e.g. GOV-1 presupposes a consumer that tracks REQ-IDs) but whose actual subject is the spec corpus's own content (the anchor-site form) is arm (b) and passes.

## REQ-ID prefix table

**GOV-1 (anchor placement).** Each spec page that carries normative obligations is assigned a stable per-page REQ-ID prefix (table below). Each non-narrative page in that table MUST carry a `PREFIX-N` anchor at the defining site of every live REQ-ID under its prefix.

- **Canonical form.** Each REQ-ID's defining anchor is the inline `**PREFIX-N.**` marker. The bold-with-period decoration is normative: it distinguishes anchor sites (`**BNDR-7.**`) from back-references (`per BNDR-7`) and is part of the anchor form. The bold-with-period bytes are the witness pattern for any third-party REQ-ID extractor; per [GOV-18 arm (b)](#gov-18-arm-b), this rule pins only the form of the anchor on the spec page, not the extractor's behaviour.
- **Required HTML-anchor contexts.** The `<a id="prefix-n"></a>` HTML form MUST accompany the inline marker at every REQ-ID anchor site, in each of the four enumerated contexts below. These are the cases where the inline marker is not by itself a sufficient anchor (Markdown bold-with-period either does not render, breaks the surrounding construct, or — being rendered as bold text — does not by itself produce a stable URL fragment in any common Markdown renderer):
    - inside a Markdown table cell;
    - inside an ATX heading (`#` … `######`);
    - on the line immediately preceding a fenced code block whose content is the rule's normative example;
    - in body-paragraph context, where the inline marker renders fine as bold text but does not by itself produce a stable URL fragment in any common Markdown renderer, so the HTML form is the canonical fragment source for body-paragraph REQ-IDs (the three rendering-driven contexts above each carry their own per-context placement mechanics in *Edge cases* below — heading-slug repointing for ATX headings, the table-cell leading-token rule, and the line-preceding-a-fenced-code-block placement). *Body-paragraph context* is defined operationally as the negation of the three rendering-driven contexts above: every source-line position that is neither inside a Markdown table cell, nor inside an ATX heading, nor on the line immediately preceding a fenced code block whose content is the rule's normative example. This explicitly includes prose paragraphs, top-level and nested unordered list items, ordered list items, blockquote bodies, definition-list bodies, footnote bodies, and the line *following* a fenced code block.

  The HTML form is permitted only as part of a dual-form pair (per *Dual-form layout* below — i.e. co-located on the same source line as an inline `**PREFIX-N.**` marker) in one of the four enumerated contexts above; no other source-line position permits the HTML form. A bare `<a id="prefix-n"></a>` anchor with no co-located inline marker on the same line is NOT a GOV-1 anchor. A REQ-ID inside a fenced code block, an inline code span, or an HTML comment is neither an anchor nor a back-reference and is invisible to GOV-3 extraction (per the exclusion list in GOV-3 below). *Multi-context overlap.* The dual-form anchor is required in every one of the four contexts above, so the applicability question never depends on which context a site is classified under; only the per-context *mechanics* in *Edge cases* below differ. When a REQ-ID anchor site syntactically matches more than one enumerated context (e.g. a table-cell marker whose row is followed by a fenced code block, or an ATX-heading marker whose line precedes a fenced code block), the more-specific (rendering-driven) context determines which *Edge cases* mechanics apply, in this fixed precedence order: (1) table-cell, (2) ATX-heading, (3) line-preceding-a-fenced-code-block, (4) body-paragraph (which by construction cannot overlap with any of the three rendering-driven contexts, per the operational definition above). The first context whose syntactic conditions the site satisfies wins; lower-ranked matches are suppressed for the purposes of selecting *Edge cases* mechanics.
- **Dual-form layout.** When the HTML form is used, it MUST appear on the same source line as the inline marker, in the order `<a id="prefix-n"></a> **PREFIX-N.**` — the HTML anchor first, a single ASCII space, then the inline marker. This ordering keeps the inline marker adjacent to the rule text it introduces and pins a single witness regex for the dual-form pair (`<a id="prefix-n"></a> \*\*PREFIX-N\.\*\*`). The dual-form pair MUST be the first non-prefix tokens on the line; the regex is matched line-by-line against each source line after stripping (a) any leading ASCII whitespace, and (b) a Markdown block-prefix sequence built from the closed grammar below. Tabs anywhere in the line-prefix region (i.e. before the dual-form pair) are normalised to spaces per CommonMark before stripping. The permitted block-prefix sequence has three ordered slots, each independently optional, in this fixed order; each slot's prefix bytes MAY be preceded by ASCII whitespace per CommonMark indentation rules:

  1. **Blockquote slot.** Zero or more `> ` blockquote prefixes (so `> `, `> > `, `> > > `, etc. all match).
  2. **List-item slot.** Zero or one list-item marker, drawn from:
     - an unordered-list marker `- `, `* `, or `+ `; or
     - an ordered-list marker `<digits>.` or `<digits>)`, in either case followed by a single ASCII space.
  3. **Construct-marker slot.** Zero or one construct-marker prefix, drawn from:
     - a table-cell leading `|` with optional surrounding ASCII whitespace (per CommonMark/GFM table syntax — i.e. an optional leading-pipe at column 0 plus the inter-cell pipes that separate cell content);
     - an ATX-heading `#{1,6} ` (one to six `#` characters followed by a single ASCII space);
     - a definition-list body marker `: ` (colon followed by a single ASCII space); or
     - a footnote body marker `[^<identifier>]: ` (a footnote label `[^<identifier>]` followed by colon and a single ASCII space).

  Slots are stripped in the order 1 → 2 → 3; no other ordering is recognised. A prefix from a later slot appearing before a prefix from an earlier slot on the same line, or two prefixes drawn from the same slot 2 or slot 3 on the same line (the slot-1 `> ` is the only prefix that may repeat, per its zero-or-more cardinality), is not stripped, so the dual-form pair is not the first non-prefix tokens on the line and the witness regex fails. The legal compositions therefore include `> - `, `> > - `, `> > > 1. `, and a blockquoted ATX heading `> ## `, while `- > ` and `## - ` are not recognised.

  This strip-list MUST stay co-extensive with the union of (a) the table-cell and ATX-heading rendering-driven contexts enumerated in *Required HTML-anchor contexts* above and (b) every construct enumerated in the body-paragraph operational definition there (which is what brings definition-list bodies, footnote bodies, list items, and blockquote bodies into scope). Adding a new context to either side of *Required HTML-anchor contexts* without extending this strip-list (or vice versa) is a spec defect: the witness regex would either reject every legal anchor in the new context or silently extend by implementer guess. In the three rendering-driven contexts (table cells, ATX headings, and the line preceding a fenced-code-block normative example), the mechanics of placing the dual-form anchor are governed by *Edge cases* below.

  *Per-page progressive normalisation.* At every commit on `main` — taking the first-parent diff (in git's sense, per `git log --first-parent` / `git diff-tree --first-parent`) for merge or squash commits, and the diff against the commit's sole parent for single-parent commits — on each non-narrative spec page where the commit's diff touches the anchor token of an existing REQ-ID anchor site (i.e. modifies, in any byte, either the inline `**PREFIX-N.**` marker bytes or the dual-form `<a id="prefix-n"></a>` HTML token co-located with it, at a site that was present at the commit's first parent), every REQ-ID anchor site on that page MUST be in dual-form in the post-diff text — both the touched site and every other site on the same page. Adding a new REQ-ID anchor site (a source line newly carrying an inline `**PREFIX-N.**` marker that was not present at the commit's first parent) does not by itself trigger this clause; new sites are bound directly by *Required HTML-anchor contexts* above. Retiring an existing site (deleting its marker line, or replacing it with an `<!-- … retired … -->` comment) likewise does not trigger; the site is gone and there is nothing to leave in dual-form. Edits to a `**PREFIX-N.**` paragraph's rule body, edits to surrounding prose on the marker's source line that leave both the inline marker bytes and the dual-form HTML token byte-unchanged, and line-renumbering shifts caused by insertions or deletions on unrelated source lines, do not by themselves trigger this MUST. Pre-existing non-conformant anchor sites on pages whose existing anchor tokens the commit does not touch under the trigger predicate above are standing spec defects — they are not bound by this MUST on the current commit, and remain so until a future commit performs a triggering operation on the same page. The defect set drains progressively as pages are edited; no corpus-wide normalisation commit is reserved.
- **Edge cases.** Inserting an anchor inside an ATX heading shifts the heading's auto-generated GitHub fragment ID; where every affected in-corpus cross-link can be repointed atomically with the anchor insertion, every such cross-link MUST be repointed in the same diff. When repointing the heading-slug fragment is impractical in the same edit (e.g. the heading slug is referenced from outside the spec corpus or the cross-link cannot be repointed atomically with the anchor insertion), the entire dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`) MUST instead be placed on the source line immediately preceding the heading, and the heading text MUST NOT carry the inline `**PREFIX-N.**` marker. In that case the anchor site classifies as body-paragraph context per the operational definition in *Required HTML-anchor contexts* above (the line preceding an ATX heading is body-paragraph by negation, since it is not itself inside the heading), so the *Dual-form layout* same-source-line MUST and witness regex apply normally with no ATX-heading-specific carve-out. Inside a table cell, the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`, in the order specified by *Dual-form layout* above) MUST be the cell's leading token sequence; where carrying the rule body in the cell would distort column widths, lift the entire rule out of the table — the cell then carries only a back-reference (e.g. `per PREFIX-N`) and the dual-form pair sits at the start of the rule's body-paragraph source line outside the table, per *Dual-form layout*. In the line-preceding-a-fenced-code-block context, the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`, in the order specified by *Dual-form layout* above) MUST appear on the source line immediately above the opening fence, with no intervening blank line, and that line MUST NOT carry any other content. The line-preceding-a-fenced-code-block *rendering-driven context* (and its per-context placement mechanics) is reserved for code blocks whose content is the rule's normative example; the line preceding a decorative or illustrative fence does not classify as that context and so does not invoke its per-context mechanics. Such a line is body-paragraph context per the operational definition in *Required HTML-anchor contexts* above, so a dual-form REQ-ID anchor on that line is still required if the line carries an inline `**PREFIX-N.**` marker — it is governed by the body-paragraph mechanics, not by the line-preceding-a-fenced-code-block context. Body-paragraph context carries no per-context edge case beyond the *Dual-form layout* same-source-line MUST: the dual-form pair (`<a id="prefix-n"></a> **PREFIX-N.**`) sits at the start of the body-paragraph source line that carries the inline marker, and any list-item / blockquote / nested-block indentation that precedes the pair is handled by the witness-regex anchoring rule rather than by a body-paragraph-specific carve-out here.

**GOV-3.** REQ-ID prefixes are byte-exact uppercase ASCII tokens of length 2–4 (`[A-Z]{2,4}`). Prefix matching is case-sensitive; `lex-1` does not match `LEX-1`. REQ-ID extraction operates on raw Markdown source bytes, not on rendered HTML. Before regex application, the following are stripped, in order: (i) fenced code blocks (` ```…``` ` and `~~~…~~~`, inclusive of fence lines), (ii) HTML comments (`<!--…-->`), (iii) inline code spans (`` `…` `` and the multi-backtick variants). Markdown link text is in scope; link targets are out of scope.

Two regexes define the in-corpus form of an extractable REQ-ID, against the exclusion-stripped corpus described above:

1. **Primary extractor.** `\b(<live-prefix-alternation>)-[1-9][0-9]*\b`, where `<live-prefix-alternation>` is built from the live REQ-ID prefix table below at evaluation time (never hard-coded). Leading zeros in the numeric tail are forbidden.
2. **Unknown-prefix closure invariant.** Every token in the exclusion-stripped spec corpus that matches `\b[A-Z]{2,4}-[1-9][0-9]*\b` MUST have a prefix that appears in the union of (live REQ-ID prefix table, *Retired prefixes* sub-table). [GOV-16](#gov-16) extends this closure invariant to the inline-label identifier class (widening the prefix character class to `\b[A-Z][A-Z0-9]{1,5}\b` and the tail to admit the single-lowercase-letter form, against the widened membership union that includes the per-page inline-label prefix table).

Pages whose row in the prefix table below carries the literal cell `(no IDs — narrative)` are excluded from extraction; all other rows in `spec_topics/*.md` are in scope. The exclusion cell's canonical byte sequence is `(no IDs — narrative)` — open paren, `no IDs`, ASCII space, U+2014 EM DASH, ASCII space, `narrative`, close paren. Cosmetic variants (smart quotes, trailing whitespace, en-dash, ASCII double-hyphen) are not recognised as the narrative cell, and the cell-value comparison that classifies a row as narrative MUST be byte-exact.

IDs are immutable: when a rule is split, the original ID retires and two new IDs appear; numbering never collapses to fill holes.

**GOV-4 (per-row invariant).** The prefix → page binding is append-only and immutable: once a prefix is allocated to a page (per GOV-7 *Add* or GOV-7 *Narrative-to-normative promotion*), the binding never changes silently and the prefix is never reassigned to a different page (per GOV-7 *Delete* and GOV-7 *Normative-to-narrative demotion*, retired prefixes move to the *Retired prefixes* sub-table below and are never reused). The *table rows themselves* are NOT immutable — GOV-7 *Rename* updates the Page column without changing the prefix, and GOV-7 *Delete* / *Merge* / *Normative-to-narrative demotion* remove or relocate rows. Introducing a new non-narrative page requires appending a new row whose prefix is *previously-unused* — meaning absent from both this table and the *Retired prefixes* sub-table below. The prefix → page binding is many-to-one: a single page MAY own more than one prefix (e.g. `hard-ceilings.md` owns both `CEIL` and `CIO`), in which case each prefix sits on its own row and the per-row invariants apply to each row independently. Adding a second (or further) prefix to a page that already appears in the table follows the *Add prefix to existing page* sub-clause of GOV-7 below. The live table's `Prefix` column is a key: duplicate live `Prefix` values, and any live prefix that also appears in the *Retired prefixes* sub-table, are spec defects.

**GOV-5 (disjoint-prefix invariant).** Each row's `Prefix` value is a complete identifier token, not a search prefix. REQ-ID prefixes are word-boundary-distinct: every REQ-ID anchor and back-reference in the spec corpus matches the token regex `\b<PREFIX>-[1-9][0-9]*\b`, and two prefixes that share a common substring (e.g. `BNDS` / `BNDR`) are distinct tokens — neither is a sub-prefix of the other.

**GOV-6 (table-completeness invariant).** At every commit on `main`, the set of prefixes appearing in REQ-IDs across `spec_topics/*.md` is a subset of the union of (live prefix table, *Retired prefixes* sub-table).

**GOV-7 (mutation procedures).**

- **Add.** New page → append a row with a previously-unused prefix.
- **Add prefix to existing page.** Existing page acquires a second (or further) prefix → append an additional row, whose Page cell repeats the existing page's filename and whose Prefix cell is *previously-unused* on the same terms as *Add*. The pre-existing row(s) for that page stay as they are (per GOV-4's prefix → page binding stability — appending a new prefix row for an existing page is not one of the GOV-7 mutations *Rename* / *Delete* / *Merge* / *Normative-to-narrative demotion* that GOV-4 enumerates as the exhaustive set of operations permitted to modify or relocate existing rows, so the existing rows' Prefix and Page columns are untouched).
- **Rename.** Prefix follows the page; the row's Page column updates, the Prefix column does not. Existing in-page anchors are not rewritten.
- **Delete.** The row is moved from the live table to the Retired prefixes sub-table. The prefix MUST NOT be reused.
- **Merge.** When `<absorbed-page>` is merged into `<surviving-page>`:
    - Every live REQ-ID on the absorbed page undergoes a GOV-8 *Merge* into a freshly allocated ID under the surviving prefix, appended at the surviving page's tail. The absorbed-page IDs retire on the absorbed page (its `## Retired REQ-IDs` section gains one row per merged ID) before the file is deleted.
    - The absorbed page's pre-existing `## Retired REQ-IDs` rows transplant verbatim into the surviving page's `## Retired REQ-IDs` section.
    - The absorbed prefix moves to the *Retired prefixes* sub-table; the *Formerly* cell records `<absorbed-page> (merged into <surviving-page> at <sha>)`.
    - The absorbed page file is deleted in the same commit (otherwise the prefix-table-completeness gate sees both a retired prefix and a live page carrying it).
- **Narrative-to-normative promotion.** Replace the `(no IDs — narrative)` cell with a freshly allocated prefix in the same edit that introduces the first obligation.
- **Normative-to-narrative demotion.** When a page that previously owned a prefix becomes pure narrative (every claim it carried is canonically owned elsewhere): (a) move its prefix from the live table to the *Retired prefixes* sub-table per GOV-7 *Delete*; (b) append a new live table row for the page carrying `(no IDs — narrative)`. Re-promotion later (per *Narrative-to-normative promotion* above) requires a *fresh* prefix — the original is retired and immutable per GOV-4 / GOV-5, and the *Retired prefixes* sub-table is itself append-only.

**GOV-8 (REQ-ID lifecycle).**

*Scope note (non-normative).* GOV-8 is a *bookkeeping* discipline: it pins how a substantive edit is recorded, not whether two loom 1.x releases produce the same output for the same `.loom` / `.warp` source file. The behavioural loom 1.x stability statement (per **GOV-15** at the page tail) is intentionally *not* derived from this rule.

<a id="gov-14"></a> **GOV-14 (review posture on the loom 1.x-equivalence release-process goal).** Reviews of `spec.md` and `spec_topics/*.md` SHOULD NOT cite GOV-8 as a substitute for the loom 1.x source-language equivalence release-process goal recorded at [GOV-15](#gov-15) and SHOULD NOT re-raise the absence of an automated equivalence gate as a loom 1.0 correctness finding. The loom 1.0 release decision treats the absence of an automated equivalence gate as a recorded scope choice, not a defect.

- **Split.** When one rule splits into N rules, the original ID retires and N fresh IDs are appended at the page's tail.
- **Merge.** When N rules merge into one, all N source IDs retire and one fresh ID is appended at the page's tail.
- **Deletion.** Rule removed without replacement → ID retires; the prefix-position number MUST NOT be reused.
- **Pure rewording.** Typo fixes, sentence restructuring, link updates leave the ID unchanged. A change that alters which inputs are accepted, which outputs are produced, which diagnostics fire, or which invariants hold is substantive and MUST be modelled as a split, merge, or deletion-plus-add — never as an in-place edit. This boundary is enforced by review on the PR that touches a `**PREFIX-N.**` paragraph; no CI gate detects substantive in-place edits, and none is planned, because semantic equivalence between two prose paragraphs is not mechanically decidable. Worked examples:
  - Modal strengthening or weakening (`should` → `MUST`, `MUST` → `should`, `MAY` → `MUST`, etc.) inside a `**PREFIX-N.**` paragraph is **substantive** — the obligation's force has shifted even when the inputs and outputs at the moment of the edit appear unchanged. Treat as a split-or-deletion-plus-add. (This is the most commonly missed case in review; flag it first.)
  - Adding or removing a diagnostic code on a `**PREFIX-N.**` paragraph is **substantive** — it alters which diagnostics fire, per the test above. Treat as a split-or-deletion-plus-add. The new code's diagnostic-registry entry is a separate, additive obligation and does NOT discharge the GOV-8 retirement requirement.
  - Editing a non-normative example (a fenced code block, parenthetical illustration, or worked sample) attached to a `**PREFIX-N.**` paragraph is **pure rewording**, provided the rule's normative sentences are unchanged. Examples do not constrain implementations. When in doubt about whether a sentence is normative or illustrative, retire-and-re-add.

All retirements (per GOV-7 *Delete* / *Merge* and per GOV-8 *Split* / *Merge* / *Deletion*) MUST be recorded:

- **Per-prefix retirements** appear in the *Retired prefixes* sub-table immediately below.
- **Per-ID retirements** appear in a trailing `## Retired REQ-IDs` section on each non-narrative page. After a GOV-7 *Merge*, the surviving page's `## Retired REQ-IDs` section accumulates rows under both the surviving and the absorbed prefixes (per the *Merge* sub-bullet above); the absorbed page's section is deleted with its file.

**GOV-9 (cross-link form).** Each spec page that depends on a normative rule from another topic page MUST either state the rule locally or reference it by a markdown link whose target is the specific REQ-ID anchor (`#prefix-n`) of the depended-upon rule. Where the depended-upon page is pure-narrative (its row in the [REQ-ID prefix table](#req-id-prefix-table) carries the `(no IDs — narrative)` cell per GOV-3), a section-level link to the relevant heading on that page suffices. Every REQ-ID anchor on a non-narrative spec page MUST resolve as a `#prefix-n` URL fragment, so that any cross-page link of the form `…#prefix-n` formed per the preceding sentence resolves to the depended-upon rule's anchor site; this universal-resolvability obligation is the contract that GOV-1 *Required HTML-anchor contexts* and *Dual-form layout* deliver against. Pre-existing non-conformant anchor sites — sites whose dual-form HTML anchor is absent and therefore do not yet expose a `#prefix-n` fragment — are standing spec defects on the same per-page terms as GOV-1 *Per-page progressive normalisation* (which owns this rule's delivery mechanism): the defect set drains progressively as pages are edited, and no corpus-wide normalisation commit is reserved. An obligation site that carries no REQ-ID anchor at all — distinct from a coined anchor that is merely missing its dual-form HTML token — is likewise a standing spec defect under this contract, drained progressively by [GOV-22](#gov-22) on the same per-commit terms.

<a id="gov-22"></a> **GOV-22 (progressive coinage of un-anchored obligations).** GOV-1 *Per-page progressive normalisation* upgrades existing REQ-ID anchor sites to dual-form, but its trigger fires only where the commit touches an existing REQ-ID anchor token; a non-narrative page that carries no REQ-ID anchor, and any individual obligation that has never been coined as a REQ-ID, are therefore unreachable by that trigger, leaving GOV-9's `#prefix-n` cross-link contract permanently unsatisfiable for them. This rule drains that residue on the same per-commit, progressive terms — no corpus-wide coinage commit is reserved. At every commit on `main` — taking the first-parent diff for merge or squash commits and the diff against the sole parent for single-parent commits, on the same terms as GOV-1 *Per-page progressive normalisation* — where the commit's diff adds a *defining obligation site*, or edits the normative-modal content of an existing one, on a non-narrative spec page and that site does not carry a co-located REQ-ID anchor in the post-diff text, the committer MUST coin a `PREFIX-N` REQ-ID anchor at that site in the same commit, in the dual-form layout GOV-1 mandates and under the page's prefix already registered in the [REQ-ID prefix table](#req-id-prefix-table); the numeric tail is allocated per GOV-3 (the next free integer under that prefix, numbering never collapsing to fill holes). A *defining obligation site* is a source line that states an obligation the page owns and carries an RFC-2119 normative-modal token (`MUST`, `MUST NOT`, `SHALL`, `SHALL NOT`, `SHOULD`, `SHOULD NOT`), excluding tokens inside fenced code blocks, inline code spans, and HTML comments (the GOV-3 exclusion list) and excluding back-references to obligations another page owns. Editing a defining obligation site that already carries its own co-located REQ-ID anchor does not trigger this rule — such an edit is governed by GOV-8's lifecycle and by GOV-1 *Per-page progressive normalisation*. Pre-existing un-anchored obligation sites on a page whose diff does not touch them under the trigger predicate above are standing spec defects not bound by this MUST on the current commit; the defect set drains progressively as pages are edited, exactly as GOV-1 *Per-page progressive normalisation* states for non-dual-form anchor sites. This rule binds the spec corpus itself (arm (b) per [GOV-18 arm (b)](#gov-18-arm-b)); the committer and the reviewer of the triggering PR are its satisfaction mechanism, and no CI gate is reserved.

<a id="gov-15"></a> **GOV-15 (loom 1.x source-language equivalence — release-process goal).** A `.loom` or `.warp` file that loads cleanly under loom 1.0.0 is expected to load under every loom 1.x release and to produce, for any given input, identical (a) return values, (b) ordered diagnostic-code sequences, and (c) equivalent `loom-system-note` content strings. Two content strings are equivalent under observable (c) when they are byte-identical — i.e. equal as UTF-8 byte sequences, the encoding substrate pinned by [Lexical — Encoding](./lexical.md) — after normalising the placeholder sub-fields whose rendered value is permitted to vary per invocation or per run — the per-invocation identifiers (e.g. `<invocation-id>` / `<uuid>`), the wall-clock-derived values (e.g. `<ms>`), and the category-8 host-derived freeform tails — as classified by the placeholder-rendering categories in [Diagnostics — Placeholder rendering](./diagnostics.md#placeholder-rendering-normative); the fixed (non-variable) placeholder renderings and the surrounding literal template bytes are themselves byte-identical. Observable (c)'s equivalence relation, not the exclusion list, governs variation that is embedded as a placeholder sub-field inside a content string. Wall-clock timing, token counts, and log-line volume remain excluded from this expectation as independent observables in their own right — distinct from the wall-clock-derived sub-fields that observable (c) normalises within a content string. This rule states a release-process goal, not a verifiable obligation: loom 1.0.0 ships without an automated equivalence gate; equivalence between two loom 1.x releases is a release-process responsibility verified by reviewer inspection of the diff against the prior loom 1.x release. <a id="gov-15-fixture-suite"></a> A conformance fixture suite that mechanically diffs the three observables above against a frozen loom 1.0.0 baseline is a recognised post-loom 1.0.0 follow-up; see [Future Considerations](./future-considerations.md). The companion review-posture rule that bars reviewers from re-raising the missing gate as a loom 1.0.0 correctness finding is [GOV-14](#gov-14). The (a)–(c) observable list and the wall-clock / token-count / log-line-volume exclusion list are both closed at loom 1.0.0: any other observable not named in either list is, by default, neither in scope of the equivalence promise nor of the explicit-exclusion list, and adding a further observable to either list is a substantive edit to GOV-15 governed by the present rule's edit conventions, not by GOV-8's *Pure rewording* arm. When the reviewer-inspection step encounters such a third-bucket observable diverging between releases, it raises the divergence as a documentation defect against GOV-15 (to add the observable to one of the two lists in a follow-up release) but does NOT block the current release. References elsewhere in the corpus to "four ceilings", "four-item list", "four NOCEIL-1 … NOCEIL-4 claims", "four-axis closure list", and any other count-bearing phrase naming the loom 1.0.0 starting set are bound by GOV-12 lock-step: any operation that changes the count those phrases name (e.g. *Introduce*, append-trailing *Retire*, retirement form (ii) coining a new `NOCEIL-N`, or *new-axis case* introduction extending the closure axis count) also updates the affected count-bearing phrases in the same commit.

<a id="gov-15-loads-cleanly"></a>

*Loads-cleanly predicate.* "Loads cleanly under loom 1.0.0" — the *Given* that selects the input set GOV-15's equivalence promise ranges over — means: loading the `.loom` or `.warp` file under loom 1.0.0 emits no diagnostic of effective severity `error` (`E`). Diagnostics of effective severity `warning` (`W`) do not disqualify a clean load. The no-`E` predicate alone selects GOV-15's input set; it is not co-extensive with whether the runtime registers the file as a loom (a `W`-severity load can leave the file unregistered — e.g. `loom/load/unreadable`). *Effective severity* is read from the *Severity* column of the diagnostics [Code registry](./diagnostics.md#code-registry); for the discovery-source codes whose *Severity* cell carries `E/W`, the effective severity is the per-source value resolved by the [Discovery — Failure modes](./discovery.md) tables rather than a code-level fixed value. This predicate coins no new severity classification — it names the existing *Severity*-column landmark as the selector. A file that does not load cleanly under loom 1.0.0 is outside the GOV-15 equivalence promise's input set.

<a id="ceiling-set-carve-out"></a>

*Ceiling-set carve-out.* Introducing a new hard runtime ceiling within loom 1.x — or retiring or relaxing an existing one — is a recognised carve-out to this equivalence. The carve-out's in-scope input set is defined *post-hoc* over the diff between the two releases under comparison and is a property of that inter-release diff, not a property the runtime can statically derive at single-release evaluation time: an input is *in scope* of the equivalence promise iff, when run under both releases, it either (i) trips no hard runtime ceiling under either release, or (ii) trips exactly the same hard runtime ceiling (identified by its bullet ordinal in the four-item list at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings), persistent across the inter-release diff) under both releases. An input that trips a ceiling absent from one of the releases, or that trips a ceiling under one release but succeeds under the other (whether via the ceiling being introduced, retired, relaxed, or never having been ceilinged on the other side), is *outside* the in-scope set and the carve-out applies to it. "Trips" here means *surfaces* — the single ceiling whose firing reaches loom code under the CIO-1 … CIO-6 evaluation order; co-fires recorded under `masked` are not separately tested. A release that reorders surfaced ceilings (via a CIO-ordering or `masked`-domain change) without performing a ceiling-set change does NOT trigger the carve-out — the carve-out is keyed to ceiling-set changes only. Such surfacing-reordering edits remain bound by the equivalence promise on observables (a)/(b)/(c) and any divergence on inputs that previously surfaced ceiling X but now surface ceiling Y is a GOV-15 release-process violation; CIO-ordering and `masked`-domain edits that change observable surfacing therefore require loom 2.0 migration unless they preserve every input's surfaced ceiling identity. For inputs outside the in-scope set defined above, observable (a) is by construction permitted to differ between releases — the success/fail outcome is the very change the ceiling-set change produces — and the carve-out also covers this (a) drift; only on the in-scope set is the equivalence promise unaltered for (a), subject to the routing-class-demotion exception below. For inputs that remain in-scope but whose routing-class is demoted under *Relax* (e.g. panic→recoverable-`Err`), observable (a) is permitted to differ between releases — the routing-class change's own (a) impact (panic-shape → `Err`-value-shape, or analogous demotion) is carve-out-covered for those inputs; only the changed-ceiling's own discriminator and message text are bound by the *Attribution test*. The [deferred conformance-fixture suite](#gov-15-fixture-suite) is the mechanism that verifies the carve-out post-hoc against the inter-release diff. The carve-out covers only the changed ceiling's own diagnostic emissions — i.e. observables (b) and (c) of GOV-15's first paragraph as they pertain to the changed ceiling — enumerated parallel to the three carve-out-eligible ceiling-set changes: for an *introduced* ceiling, the new discriminator and the canonical message it adds; for a *retired* ceiling, the removed discriminator and the canonical message it removes; for a *relaxed* ceiling, the existing discriminator (which continues to apply but fires under different conditions) and any canonical-message text change that re-states the new bound (e.g. `exceeded 50` replacing `exceeded 25`) or otherwise reflects the relaxed predicate; plus, in all three cases, any `loom-system-note` content directly attributable to the ceiling-set change (per the *Attribution test* below). On in-scope inputs, an unrelated discriminator whose multiplicity (number of emissions per query) changes solely as a behavioural consequence of the ceiling-set change — e.g. a per-round diagnostic that fires more often after a loop bound is relaxed — is treated as carve-out-covered drift, not as a GOV-15(b) violation; the canonical-message content of the *unrelated* discriminator must remain unchanged on each emission; the changed ceiling's own diagnostic content is governed by the *Attribution test* below. Collateral diagnostic-code edits or `loom-system-note` content edits landed in the same release on unrelated surfaces (per the same *Attribution test*) are forbidden in a ceiling-set release per the *Attribution test* below; they must be split into a separate loom 1.x release that lands no ceiling-set change. When a single loom 1.x release performs more than one ceiling-set change (e.g. introduces one ceiling and retires another, or introduces two ceilings at once), the carve-out applies as the union of the per-change carve-outs — each ceiling-set change is individually scoped per the *Attribution test* below, and an input that falls outside the in-scope set under any one of the constituent changes is outside the in-scope set for the release as a whole. An edit jointly attributable to two or more constituent ceiling-set changes is in-scope for the release; an edit attributable to at least one constituent change is in-scope, even if it is collateral with respect to another constituent change. The *Collateral* set is the residue. The orientation aggregator at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings) names this carve-out in lock-step with this rule per [GOV-12](#gov-12). The GOV-15 reviewer-inspection step verifies the [ceiling-set invariants](./hard-ceilings.md#ceiling-set-invariants) per the canonical clause at [Hard Runtime Ceilings — Release gate](./hard-ceilings.md#release-gate).

<a id="attribution-test"></a>

*Attribution test.* The carve-out's two phrases — *directly attributable to the ceiling-set change* and *collateral … on unrelated surfaces* — are pinned by the following syntactic predicate, mechanically applicable on the release diff. A wording change to a diagnostic code or to `loom-system-note` content is *directly attributable to the ceiling-set change* iff at least one of the following holds: (i) its discriminator (`details.kind` for diagnostic-channel ceilings, the `Err`-variant `kind` for `QueryError` ceilings, or the discriminator named in ceiling #4's per-boundary table at [`hard-ceilings.md` — Per-boundary destination/surface table (ceiling #4)](./hard-ceilings.md#ceiling-4-table)) equals one of the changed ceiling's own discriminators (plural where the changed ceiling cross-routes — e.g. ceiling #4's `loom/runtime/*` and `loom/load/*` arms both count); (ii) its canonical message text contains either the ceiling's ordinal identifier (e.g. `ceiling #N`) or the literal numeric bound being introduced, retired, or relaxed; (iii) the change is the minimal edit required to keep a message-template row, header line, or `details.kind`/discriminator enumeration shared with another ceiling well-formed under the changed ceiling (the *shared-template* case — including a discriminator rename that subsumes both the changed ceiling's old discriminator and an unrelated runtime discriminator, where the rename's normative widening is the minimal edit required and any further wording change to the unrelated discriminator's surface is not the minimal edit required and falls into the *Collateral* set below); or (iv) for a routing-class demotion or enforcement-point narrowing with no numeric bound, the canonical message text contains an explicit reference to the routing-class or enforcement-point surface change (e.g. `no longer fatal`, `no longer enforced at <site>`) minimal to reflect the relaxed predicate. Each ceiling's owner page is the source of truth for that ceiling's discriminator inventory. *Collateral* is any wording change for which none of the four sufficient conditions (i)–(iv) above holds (i.e. for which *Directly attributable* is false). A release that lands collateral diagnostic-code or `loom-system-note` content edits inside the same release as a ceiling-set change is flagged by this rule's reviewer-inspection step (per the reviewer-inspection step in GOV-15's main paragraph above) as a GOV-15 release-process violation, blocking the release until the collateral edits are split into a separate loom 1.x release that lands no ceiling-set change.

<a id="operational-definitions"></a>

*Operational definitions.* The four ceiling-set verbs name distinct operations. *Introduce* adds the new ceiling's ordinal-bearing entry to the four-item list at [`spec.md` — Hard ceilings](../spec.md#hard-runtime-ceilings) (extending it to a five-item list), creates the per-ceiling owner-page section on the appropriate topic page (defaulting to a new sub-section on [`hard-ceilings.md`](./hard-ceilings.md) if no other topic page is the natural owner), and adds a per-axis entry under [Hard Runtime Ceilings — Audit methodology — Four-axis closure list](./hard-ceilings.md#four-axis-closure-list), all in the same commit per GOV-12; *Introduce* is a normative addition not modelled by any GOV-8 lifecycle case (which addresses only edits to existing rules), and is governed directly by the present rule and by the audit-methodology paragraph at [Hard Runtime Ceilings — Audit methodology](./hard-ceilings.md#audit-methodology). *Retire* — remove a ceiling; the retired ceiling's owner-page entry is annotated in place with a pointer to the retirement-release diff. The pointer takes the same SHA-or-tag form as the `Retired in` column of the [*Retired prefixes*](#retired-prefixes) sub-table above — either the 7-character abbreviated commit SHA of the retirement commit or the release tag of the loom 1.x release that performed the retirement, nothing else. For inputs that previously tripped the retired ceiling, the retired ceiling's owner-page annotation MUST document the post-retirement runtime trajectory (succeed; fall through to a documented `NOCEIL-N` surface; fall through to another ceiling; or non-terminating absent cancellation); silent unbounded behaviour is forbidden. Cardinality after retirement: the retired ceiling's bullet is left as an in-place retirement-annotated stub (preserving the persistent ordinals the carve-out's in-scope predicate relies on); the bullet MUST NOT be removed outright while any later bullet exists, since outright removal would shift subsequent ordinals and break the in-scope predicate's identity test. Bullets MAY be removed outright only when retired in append-trailing position (no later bullet exists), in which case the editor MUST also update every count-bearing reference (per GOV-12 lock-step) in the same commit. *Relax* — widen the input set the ceiling permits (raise a numeric bound, narrow the enforcement-point surface, or demote routing class — e.g. `panic` to recoverable `Err`). *Tighten* — the inverse of relax (narrow the input set the ceiling permits — lower a numeric bound, broaden the enforcement-point surface, or promote routing class) — is explicitly NOT a recognised carve-out under this rule, because it can break the equivalence promise on inputs that previously succeeded; it is permitted within loom 1.x only when no input in the loom 1.x conformance fixture suite (once it ships) is made to fail by the change. Until that suite ships, *Tighten* is forbidden under loom 1.x and any required tightening is deferred to loom 2.0 migration per [Future Considerations — loom 1.0 non-goals](./future-considerations.md#v1-non-goals); a tightening that would fail any input in the eventually-shipped suite remains likewise deferred to loom 2.0 and MUST NOT land within loom 1.x. Once *Tighten* becomes permitted (per the conformance-fixture-suite condition above), it falls under the same GOV-12 lock-step co-edit and *Audit methodology* release-gate obligations as *Relax*: once *Tighten* is unlocked, any spec-side enumeration that lists "introduce, retire, or relax" is read as covering *Tighten* in the same shape as *Relax* — but the inheritance is procedural only (lock-step + per-axis-entry update + release gate), not equivalence: *Tighten*-affected inputs remain bound by the GOV-15 main-paragraph equivalence promise without any *Tighten*-specific carve-out (numeric-bound, enforcement-point surface, or routing-class edit to an existing ceiling without retiring its identity). The Relax/Tighten sub-case enumeration above (numeric-bound, enforcement-point surface, routing-class) is closed at loom 1.0.0; a fourth widening or narrowing shape is forbidden in-place and requires loom 2.0 migration. Both *Relax* and *Tighten* are substantive ceiling-page edits in GOV-8 terms; they are licensed for in-place edit (rather than the GOV-8 retire-and-readd path) only by the present rule's *Ceiling-set carve-out* (for *Relax*) or by the present rule's *Tighten* permission clause (for *Tighten*). Either operation is recorded by amending the ceiling owner-page entry in place; GOV-8's split-or-deletion-plus-add path is explicitly not required for either operation. loom 1.0 specifies no upgrade-time warning surface for retirement-induced dead `match` arms in loom code (e.g. a `match` arm on `Err(QueryError { kind: "tool_loop_exhausted", … })` that becomes statically unreachable if `tool_loop.max_rounds` were ever retired); such drift is a known consequence of *Retire* and is deferred to a future tooling surface (no loom 1.0 tooling specification exists; whether such a warning surface is added in a future loom 1.x release is out of scope here). Symmetrically, a routing-class demotion under *Relax* (e.g. `panic` → recoverable `Err`) makes a previously-unreachable `match` arm reachable on previously-panicking inputs; loom 1.0 specifies no upgrade-time warning surface for the new-arm reachability case either.

## Stable inline labels

<a id="gov-16"></a> **GOV-16 (stable inline labels — second governed identifier class).** Some normative obligations carry a *stable inline label* — a cross-page-citable identifier that is not a REQ-ID. Inline labels are a second governed identifier class with their own per-page registry, anchor form, lifecycle, and retirement table; the rule is structured to mirror GOV-1 / GOV-3 / GOV-8 / GOV-9 so that the two classes are gated by parallel machinery. Two inline-label prefixes are currently registered, both on [`hard-ceilings.md`](./hard-ceilings.md): `HC3` (the per-class binder retry sub-obligations `HC3-a` … `HC3-e`) and `NOCEIL` (the four enumerated non-existence claims `NOCEIL-1` … `NOCEIL-4`). The `CIO-1` … `CIO-6` labels on the same page are *not* inline labels under this rule — they are full REQ-IDs under the `CIO` row of the [REQ-ID prefix table](#req-id-prefix-table) (co-owned with `CEIL` per GOV-4) and are governed by GOV-1 / GOV-3 / GOV-8 / GOV-9 directly.

- **Canonical form.** Each inline label's defining anchor is the inline `**PREFIX-<tail>.**` marker, where `<tail>` is either `[1-9][0-9]*` (numeric tail, e.g. `NOCEIL-3`) or a single lowercase ASCII letter `[a-z]` (alphabetic tail, e.g. `HC3-a`). The bold-with-period decoration is normative on the same terms as for REQ-IDs (per GOV-1 *Canonical form*) — it distinguishes anchor sites from back-references and is part of the anchor form.
- **GOV-1 dual-form layout applies.** Inline-label anchor sites participate in GOV-1's *Required HTML-anchor contexts* and *Dual-form layout* obligations on identical terms — the witness regex `<a id="prefix-n"></a> \*\*PREFIX-N\.\*\*` is read as `<a id="prefix-tail"></a> \*\*PREFIX-<tail>\.\*\*` with `<tail>` per the *Canonical form* above and rendered lowercase in the `id` attribute to match GitHub's slug convention. GOV-1's *Per-page progressive normalisation* clause also applies to existing inline-label anchor sites on the same terms. The two identifier classes are gated independently per page: a commit's diff that touches an existing inline-label anchor's tokens (the inline `**PREFIX-<tail>.**` marker bytes or the dual-form `<a id="prefix-tail"></a>` HTML token co-located with it) triggers the *Per-page progressive normalisation* MUST only across inline-label anchor sites on that page, and a commit's diff that touches an existing REQ-ID anchor's tokens triggers it only across REQ-ID anchor sites — neither class's edit pulls the other class's pre-existing anchor sites into the per-page sweep.
- **Per-page inline-label prefix table.** Inline-label prefixes are registered per page below; the table is append-only and follows GOV-4 / GOV-5 / GOV-7 on the same terms as the REQ-ID prefix table — live inline-label prefixes are disjoint from each other and from the live + retired union of REQ-ID prefixes, and a retired inline-label prefix is never reused. Inline-label prefixes are uppercase ASCII letters of length 2–6, optionally with ASCII digits in non-leading positions (e.g. `HC3`, `XY7`, `NOCEIL`) — matched by `[A-Z][A-Z0-9]{1,5}`; the upper bound widens beyond REQ-IDs' 4 to admit `NOCEIL`, and admitting digits in non-leading positions is a deliberate divergence from GOV-3's letters-only REQ-ID prefix grammar (`[A-Z]{2,4}`) so that digit-bearing prefixes such as `HC3` can be registered. Matching is case-sensitive per GOV-3.

  | Page | Inline-label prefix | Tail form |
  |---|---|---|
  | `hard-ceilings.md` | `HC3` | `alphabetic` |
  | `hard-ceilings.md` | `NOCEIL` | `numeric` |

  The `Tail form` column pins each inline-label prefix to exactly one tail shape — `numeric` (matching `[1-9][0-9]*`) or `alphabetic` (matching `[a-z]`) — drawn from the two tail forms enumerated in *Canonical form* above. The column is normative: it is consulted by the *Third extractor* and the *Inline-label grammar and closure invariant* below to bind each prefix to one tail form, and is part of the GOV-4 / GOV-7 append-only table contract on the same terms as the `Page` and `Inline-label prefix` columns. A row's `Tail form` cell is fixed at the prefix's allocation moment and is not silently re-pointed; switching a live inline-label prefix from one tail form to the other is a GOV-7 *Delete* of the original prefix plus a GOV-7 *Add* of a fresh prefix under the new tail form.

  *Retired inline-label prefixes.* Empty in loom 1.0; the sub-table is append-only on the same terms as the *Retired prefixes* sub-table for REQ-IDs and is materialised here when the first inline-label prefix retires.

- **Third extractor.** Inline labels in the spec corpus match the regex `\b(<alphabetic-tail-inline-label-prefix-alternation>)-[a-z]\b|\b(<numeric-tail-inline-label-prefix-alternation>)-[1-9][0-9]*\b`, a two-armed alternation whose prefix groups are partitioned by the `Tail form` column of the per-page inline-label prefix table above. `<alphabetic-tail-inline-label-prefix-alternation>` is built at evaluation time from exactly those live inline-label prefixes whose row carries `Tail form` = `alphabetic` (currently `HC3`); `<numeric-tail-inline-label-prefix-alternation>` is built at evaluation time from exactly those live inline-label prefixes whose row carries `Tail form` = `numeric` (currently `NOCEIL`). Each arm consumes only its registered prefixes paired with its registered tail form, so cross-class malformed shapes (e.g. `HC3-7`, `NOCEIL-a`) do not match — discharging the `Tail form` column's "consulted by the *Third extractor*" claim above. Neither alternation is hard-coded. Leading zeros in the numeric tail are forbidden on the same terms as GOV-3. Exclusion stripping (fenced code blocks, HTML comments, inline code spans) applies on the same terms as GOV-3.
- **Inline-label grammar and closure invariant.** A spec-corpus token of shape `<prefix>-<tail>` is a well-formed inline label iff `<prefix>` matches `\b[A-Z][A-Z0-9]{1,5}\b`, `<prefix>` appears in the per-page inline-label prefix table above (live or retired — retired inline-label prefixes carry the same pinned tail form they held when live), AND `<tail>` matches the prefix's pinned tail form per the `Tail form` column. The prefix grammar widens beyond GOV-3's REQ-ID prefix grammar (`[A-Z]{2,4}`) by lengthening to 2–6 to admit `NOCEIL` and admitting ASCII digits in non-leading positions to admit `HC3`; the tail admits the single-lowercase-letter form alongside the numeric form to admit `HC3-a` … `HC3-e`. GOV-3's *Unknown-prefix closure invariant* extends to the inline-label class on the same terms: every token in the exclusion-stripped spec corpus that matches the widened detector regex `\b[A-Z][A-Z0-9]{1,5}-(?:[1-9][0-9]*|[a-z])\b` MUST be either (a) a well-formed REQ-ID per GOV-3 (prefix in the REQ-ID prefix-table union, numeric tail), or (b) a well-formed inline label per the preceding sentences (prefix in the inline-label prefix-table union, tail matching the prefix's pinned tail form). Cross-class malformed tokens (e.g. `HC3-7` with `HC3`'s pinned tail form `alphabetic`, `NOCEIL-a` with `NOCEIL`'s pinned `numeric`, or a REQ-ID-table prefix paired with a single-lowercase-letter tail such as `BNDR-a`) violate the invariant on the same terms as an unenumerated prefix. The spec corpus pins this grammar and closure invariant as arm (b) content; tooling that needs to distinguish well-formed identifiers from neither-arm tokens reads this grammar and the per-page tables, per [GOV-18 arm (b)](#gov-18-arm-b).
- **Lifecycle.** Inline labels follow the GOV-8 lifecycle (*Split*, *Merge*, *Deletion*, *Pure rewording*) on identical terms — substantive edits MUST be modelled as a split, merge, or deletion-plus-add; pure rewording leaves the label unchanged. The GOV-8 *Pure rewording* worked examples (modal strengthening / weakening, diagnostic-code addition / removal, edits to attached non-normative examples) apply verbatim to inline labels.
- **Retirement section.** All inline-label retirements MUST be recorded in a trailing `## Retired inline labels` section on each page that owns inline-label prefixes, structurally parallel to the `## Retired REQ-IDs` section.
- **Cross-link form.** GOV-9's cross-link contract applies to inline labels on identical terms — every cross-page reference to an inline label MUST resolve as a `#prefix-tail` URL fragment (rendered lowercase per *GOV-1 dual-form layout applies* above), so any cross-page link of the form `…#hc3-a` or `…#noceil-3` resolves to the depended-upon label's anchor site.
- **GOV-12 lock-step coverage.** GOV-12's lock-step convention names `HC3-a`…`HC3-e`, `CIO-1`…`CIO-6`, and `NOCEIL-1`…`NOCEIL-4` explicitly. That enumeration is correct under this rule and stays as-written; a PR that adds, removes, or renames any of the identifiers enumerated above on `hard-ceilings.md` (whether an inline label under this rule or a `CIO-N` REQ-ID under GOV-1 / GOV-3 / GOV-8 / GOV-9) MUST update the corresponding `spec.md` aggregator in the same commit per GOV-12.

## Release-version naming and legacy aliases

<a id="gov-19"></a> **GOV-19 (release-version naming scheme).** Loom releases are named under two parallel literal forms:

- **Design-scope literal.** `loom <major>.<minor>` (e.g. `loom 1.0`, `loom 2.0`) and `loom <major>.x` (e.g. `loom 1.x`) name a major design scope. A design-scope literal makes claims about the design direction, scope decisions, scope carve-outs, and forward-compatibility seams shared by every release in the named scope.
- **Frozen-baseline literal.** `loom <major>.<minor>.<patch>` (e.g. `loom 1.0.0`, `loom 1.0.1`) names a frozen-baseline release — a closed inventory of the release's enumerated invariants at the publishing moment. A frozen-baseline literal makes claims about closed enumerations ("exactly N panic sources", "the closed `details.kind` set", "loads cleanly under"), discriminator inventories, and any other closure-shaped invariant whose membership is fixed at the publishing moment.

The two literal forms are lexically distinguishable: `loom <major>.<minor>` matches `loom \d+\.(?:\d+|x)`; `loom <major>.<minor>.<patch>` matches `loom \d+\.\d+\.\d+`. A callsite's spelling chooses its sense. Versioning follows semver semantics: `loom <major>.<minor>` is the design scope of every `loom <major>.<minor>.<patch>` release sharing that `<major>.<minor>` prefix; the relation `loom 1.0 ⊇ loom 1.0.0` holds, so a claim that binds `loom 1.0` binds every concrete release the design scope admits including `loom 1.0.0`. The frozen-baseline literal does not by itself promise stability between adjacent patch releases under that scope — inter-release stability across `loom 1.0.x` is governed by [GOV-15](#gov-15)'s source-language-equivalence release-process goal, not by this rule.

A *closed enumeration*, as used in the frozen-baseline literal definition above, is any spec-corpus enumeration whose intensional or extensional membership is asserted to be exhaustive at the publishing moment — detected by surface phrases including "exactly N", "the closed set", "the N-element set", a count-bearing list ("the four hard ceilings", "the eight non-goals"), "loads cleanly under", "closed at", "frozen at", or a closed `details.kind` / `Err`-variant / discriminator inventory. The phrase set is the same closure heuristic GOV-20's sense resolver below consumes.

<a id="gov-20"></a> **GOV-20 (legacy version-token aliases).** The legacy version tokens `V1`, `V1.0`, `V1.x`, and `V2` are governed aliases under the present rule. Each legacy token maps to the new naming scheme per the table below; the alias is in force at every callsite in the spec corpus until that callsite is converted to its alias-mapped spelling. Conversion is incremental — a future spec edit MAY convert any single legacy-token callsite without converting siblings, and the spec MAY carry mixed spellings during a transitional period with every untouched callsite governed by this rule.

| Legacy token | Maps to | Sense |
|---|---|---|
| `V1` | `loom 1.0` or `loom 1.0.0` | sense-overloaded — resolved per the *closure heuristic* below |
| `V1.0` | `loom 1.0` or `loom 1.0.0` | sense-overloaded — resolved per the *closure heuristic* below |
| `V1.x` | `loom 1.x` | design-scope only |
| `V2` | `loom 2.0` | design-scope only |

*Closure heuristic.* The legacy tokens `V1` and `V1.0` are sense-overloaded between design-scope and frozen-baseline. At a callsite where the surrounding sentence carries any closure phrase enumerated under GOV-19's *closed enumeration* definition above, the token aliases to `loom 1.0.0` (frozen-baseline). At every other callsite the token aliases to `loom 1.0` (design-scope). Default on ambiguity: `loom 1.0.0` — frozen-baseline is the stronger commitment, and design-scope claims subsume frozen-baseline claims.

*Persistence of the alias.* For prose tokens, the alias is *transitional* — once every prose callsite of a legacy token has been converted to its alias-mapped new-scheme spelling, that token's row above MAY be retired in a future edit that updates this table per GOV-7 / GOV-8 lifecycle. For HTML `<a id>` anchors, the alias is *permanent* — see GOV-21 below.

*Out-of-scope tokens.* The following tokens are NOT governed by this rule and MUST NOT be classified as legacy version-token aliases:

- Pi SDK version literals (e.g. `~0.74.1`, `0.75.5`) — Pi-side `peerDependencies` versions, not loom versions.
- Node version literals (e.g. `>= 20.6.0`, `>= 22.19.0`) — runtime engine versions.
- Diagnostic codes (e.g. `loom/parse/non-string-enum-value`) — opaque tokens.
- Inline labels `HC3-a` … `HC3-e`, `NOCEIL-1` … `NOCEIL-4` (governed by GOV-16) and REQ-IDs `CIO-1` … `CIO-6` (governed by GOV-1 / GOV-3 / GOV-8 / GOV-9).
- `V8` (JavaScript engine).
- Plan-phase identifiers `V1` … `Vn` reserved by `docs/plan_topics/conventions.md:9` — these are plan-document identifiers governed by the plan's own conventions, and are out of the spec corpus's binding scope per [GOV-17](#gov-17) / [GOV-18](#gov-18). The plan-side `V1` … `Vn` reservation is unaffected by this rule.

<a id="gov-21"></a> **GOV-21 (dual-anchor convention for legacy-token HTML anchors).** GOV-20's prose-token alias mechanism does not directly govern HTML `<a id>` anchors, whose identity is byte-literal: an inbound `#v1-foo` cross-link resolves only to a literal `<a id="v1-foo">` anchor at the target. To preserve back-compat for inbound `#v1-…` cross-links while converting anchor sites to the new-scheme spelling, this rule pins a dual-anchor convention.

A heading or paragraph is *dual-anchored* iff it carries both a `<a id="v1-…">` anchor and a `<a id="loom-1-0-…">` (or `<a id="loom-1-0-0-…">`) anchor that resolve to the same target. The convention is forward-looking — it governs how future conversions are performed and how dual-anchored sites, once they exist, behave under retirement. Pre-existing single-anchor `<a id="v1-…">` sites that have not yet been converted are governed by GOV-20's prose-token alias mechanism alone; they do not violate this rule's per-axis obligations until they are converted to dual-anchored form.

1. <a id="gov-21-canonical-arm"></a> **Canonical arm.** When a heading or paragraph is dual-anchored, new cross-references in the spec corpus MUST cite the canonical arm (the slug containing the `loom-1-0` or `loom-1-0-0` version segment) rather than the `v1`-segment alias arm. The alias arm is retained as a back-compat fragment per the *Alias permanence* sub-clause below.

   <a id="gov-21-incidental-auto-id"></a> *Incidental auto-id prohibition.* Where the dual-anchored target is a heading whose pre-rename and post-rename heading texts produce GitHub-rendered auto-id slugs distinct from both authored anchor arms (e.g. the heading `### loom 1.0 non-goals` whose rendered auto-id is a third fragment distinct from the authored `loom-1-0-non-goals` canonical arm and the authored `v1-non-goals` alias arm), that incidental rendered auto-id is NOT a citable stable anchor under this rule. New cross-references MUST cite one of the two authored arms; the incidental auto-id MUST NOT be authored as, or cited as, a stable anchor. The exact byte sequence of the incidental auto-id is determined by the renderer's slug algorithm (GFM per [GOV-18 arm (a)](#gov-18-arm-a)) and is out of spec-corpus scope per [GOV-18 binding scope](#gov-18); what GOV-21 binds is the prohibition against citing it.

2. <a id="gov-21-alias-permanence"></a> **Alias permanence.** The `v1-*` arm of a dual-anchored heading or paragraph is a permanent back-compat alias, not a deprecated arm. It is NOT eligible for retirement (i.e. removal of its `<a id="v1-…">` tag while keeping the heading and its `loom-1-0-*` arm) under ordinary GOV-7 *Rename* / GOV-8 *Retire* operations except as the *Retirement discharge* sub-clause below permits. Ordinary full-section retirement (GOV-7 *Delete* / *Merge*, GOV-8 *Deletion*) of the entire heading remains permitted on its own terms and is NOT gated on the *Retirement discharge* witness; only alias-only retirement is gated.

3. <a id="gov-21-intensional-definition"></a> **Intensional definition.** "Dual-anchored heading or paragraph" is defined intensionally above. The definition admits every dual-anchor placement class consistent with it, including: at-heading explicit pair (two `<a id>` tags on consecutive lines immediately before an ATX heading); inline-blockquote pair (two `<a id>` tags inline within a `> **... seam — ...**` blockquote label on a single line); at-heading explicit-pair-replacing-auto-id (two `<a id>` tags immediately before a heading whose GitHub auto-id slug shifted under the conversion); inline-paragraph pair (two `<a id>` tags inline within a paragraph or list item body). The enumeration above is open by construction: any further placement class consistent with the intensional definition is admitted without an enumeration amendment. Both ids of a dual-anchored pair resolve to the same target on any GFM-compatible renderer; the spec corpus is rendered under GFM per [GOV-18 arm (a)](#gov-18-arm-a).

   <a id="gov-21-version-segment-position"></a> *Version-segment position.* The `v1` version segment on the alias arm and the `loom-1-0` / `loom-1-0-0` version segment on the canonical arm MAY occupy any position in the slug: leading (`v1-X` paired with `loom-1-0-X`), mid-slug (`Y-v1-X` paired with `Y-loom-1-0-X`), or trailing (`Y-v1` paired with `Y-loom-1-0`). The token-position match is scoped to the version segment delimited by slug start, slug end, or hyphens; an arbitrary `v1` substring inside another word (e.g. `device1-foo`) is NOT a version segment. The canonical-arm derivation under this clause is mechanical for each position: leading `v1-` substitutes to `loom-1-0-`; mid-slug `-v1-` substitutes to `-loom-1-0-`; trailing `-v1` substitutes to `-loom-1-0`. The frozen-baseline variant (`v1-0-0-*` or trailing/mid-slug equivalents under closure-callsite contexts per GOV-20's closure heuristic) substitutes analogously with `loom-1-0-0` in place of `loom-1-0`.

4. <a id="gov-21-retirement-discharge"></a> **Retirement discharge.** The `v1-*` alias on a given dual-anchored heading or paragraph MAY be retired (its `<a id="v1-…">` tag removed) only when both (a) and (b) hold:

   - (a) **Spec-corpus discharge predicate.** No inbound `#fragment` cross-reference to that specific alias remains in the spec corpus (`docs/spec.md` + `docs/spec_topics/*.md`). "That specific alias" means a fragment whose slug contains the `v1` version segment at the same position as on the retired alias arm, per the *Version-segment position* clause above. The position-agnostic non-normative detection recipe `grep -rnE '#([a-z0-9]+-)*v1(-[a-z0-9]+|$)' docs/spec.md docs/spec_topics/` matches leading, mid-slug, and trailing positions of the `v1` version segment (the parenthesized prefix `([a-z0-9]+-)*` admits zero-or-more leading segments before the `v1`; the trailing `(-[a-z0-9]+|$)` admits either a hyphenated continuation or end-of-slug). The recipe is illustrative; the normative gate is the absence of inbound references regardless of detection method.
   - (b) **Audit record.** A retirement-record row is appended to the *Retired anchor aliases* sub-table on this page (immediately after *Retired REQ-IDs* below).

5. <a id="gov-21-canonical-arm-uniqueness"></a> **Cross-page canonical-arm uniqueness.** Every canonical-arm slug (the `loom-1-0` or `loom-1-0-0` version-segment arm authored under this rule) MUST resolve uniquely across `docs/spec.md` + `docs/spec_topics/*.md`: no two distinct sites may carry byte-identical canonical-arm slugs. Where two distinct sites carry byte-identical `v1`-version-segment alias slugs (so the mechanical derivation under *Intensional definition* / *Version-segment position* would yield the same canonical-arm slug at both), the GOV-12 lock-step **normative-owner** site retains the bare mechanically-derived canonical arm and every other (aggregator / secondary) site authors a site-distinguishing canonical arm with a tail marking it as the aggregator copy. The disambiguating tail is at the author's discretion subject to lens-acceptance of clarity; the tail SHOULD read as an aggregator label of the owner rather than an independent target (e.g. `loom-1-0-non-goals-aggregator` for the orientation copy that aggregates the authoritative `loom-1-0-non-goals` on `future-considerations.md`). The `v1`-version-segment alias arms at all colliding sites are unaffected by this disambiguation and remain byte-identical across sites per *Alias permanence*; only the canonical arms diverge to satisfy uniqueness. The known instance at the time of this rule's authoring is `v1-non-goals` (present on both `docs/spec.md` and `docs/spec_topics/future-considerations.md`); future collisions are handled by the same rule without amendment.

*Cross-corpus scope.* This rule narrows to the spec corpus alone. Plan-side citations of `#v1-…` (none currently exist) are ungoverned by GOV-21; GOV-17 is NOT amended to extend cross-corpus jurisdiction. External documents that link into the spec corpus by `#v1-…` fragment are GOV-17 dependents and carry no spec-side governance; the *Spec-corpus discharge predicate* above is correspondingly scoped to in-corpus links only.

*Sibling-rule reconciliation.* GOV-21 interacts with GOV-7 (lifecycle operations), GOV-8 (anchor-stability), GOV-17 (corpus direction), and GOV-18 (binding scope). The GOV-7 / GOV-8 interaction is the *alias-only retirement* vs *full section retirement* distinction in the *Alias permanence* sub-clause above; alias-only retirement is gated by *Retirement discharge*, while full-section retirement uses the GOV-7 / GOV-8 paths without GOV-21 interaction. The GOV-17 interaction is the *Cross-corpus scope* paragraph above. The GOV-18 interaction binds GOV-21 to arm (b) (the spec corpus itself); the detection recipe under *Retirement discharge* is non-normative per arm (b)'s separation of invariant from auditing machinery.

## Retired anchor aliases

Records the retirement of `<a id="v1-…">` back-compat aliases under [GOV-21 *Retirement discharge*](#gov-21-retirement-discharge). Retirement is per-alias: each row retires one `v1-*` alias from one owner heading or paragraph, leaving the heading's surviving `loom-1-0-*` (or `loom-1-0-0-*`) arm intact. The sub-table is append-only on the same terms as the *Retired prefixes* sub-table earlier on this page and the *Retired REQ-IDs* sub-table above; retired alias slugs MUST NOT be re-coined.

| Retired alias slug | Owner heading or paragraph | Retired in | Reason |
|---|---|---|---|

- **Retired alias slug** — the kebab-case slug (e.g. `v1-seam-binder-refinement-loop`), without the leading `#`.
- **Owner heading or paragraph** — the heading text or paragraph reference the alias was attached to, with a markdown cross-link to the heading's surviving `loom-1-0-*` (or `loom-1-0-0-*`) arm.
- **Retired in** — the 7-character abbreviated commit SHA of the retirement commit, or the release tag of the loom 1.x release that performed the retirement, on the same value-space as the *Retired in* column of the *Retired prefixes* sub-table.
- **Reason** — one short prose line.

Applicability: alias-only retirements on `docs/spec.md` and `docs/spec_topics/*.md` only. Plan-corpus aliases (none currently exist) and out-of-corpus pages (`README.md`, `CHANGELOG.md` — [GOV-17](#gov-17) dependents) are out of scope of this sub-table.
