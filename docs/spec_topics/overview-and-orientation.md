# Overview and orientation

## Overview

`pi-theta` is a [Pi Coding Agent](https://github.com/earendil-works/pi-mono) extension that adds a domain-specific scripting language for prompts and agentic operations.

A `.theta` file interleaves code with literal text destined for the model. Theta evaluation appends turns to a conversation: the *caller's* current conversation in `prompt` mode, or a separate conversation in `subagent` mode. The state-isolation contract for subagent invocation is enumerated in [Pi Integration Contract — Subagent state-isolation matrix](./pi-integration-contract/subagent.md#subagent-state-isolation-matrix); the *callable set* concept that matrix references is defined in [Glossary — `callable set`](./glossary.md). Mode is selected per-theta by the required `mode:` frontmatter field — see [Parameters and Frontmatter](./frontmatter.md). On the success outcome, evaluation also produces a *final value* — the value of the theta's tail expression, or — if an explicit `return expr` executes first — the operand of that `return`, per [Function Definitions — Final value](./functions.md#final-value-language-definition) — consumed by `invoke` callers and propagated across the subagent boundary per [Invocation — Typed return](./invocation.md#typed-return). The theta language itself has no file-writing, network, or process-spawning primitive; effects of those kinds occur only through allowlisted tools, and the language's effect surface is owned by [Runtime Value Model — Effects](./runtime-value-model.md#effects).

<a id="terminal-outcomes-aggregator"></a>Theta evaluation produces one of three terminal outcomes: it succeeds (turns appended; final value available to programmatic callers), it fails (by returning `Err`, by panicking, or via a [query-terminating](./glossary.md#query-terminating) enforcement point of one of the runtime-class [hard ceilings](#hard-runtime-ceilings) below — query-terminating points route via panic or `Err`, with the per-arm classification *for ceiling #4* owned by [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table); the following hard-ceiling cases are excluded from the *fail* arm of the success/fail/cancelled trichotomy (they are not evaluation Failures): (a) binder argument-binding failure — including ceiling #4's slash-load `params` arm cross-routed through ceiling #3 per [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./hard-ceilings/ceilings-3-and-4.md#ceiling-4-table) (slash-load only; the `invoke(...)` `params` arm of ceiling #4 IS an evaluation Failure that surfaces as `Err(InvokeInfraError { cause: "validation", ... })` per [Invocation — Failures](./invocation.md)) — per the binder argument-binding failure item in [Errors and Results — Terminal outcomes](./errors-and-results/error-model.md#terminal-outcomes); see ceiling #3 in [Hard ceilings](#hard-runtime-ceilings), (b) ceiling #4's [in-loop](./glossary.md#in-loop) model-driven tool-call args row (which feeds a tool-error result back to the model rather than terminating the query)), or it is cancelled per [Cancellation](./cancellation.md). The cancellation signal is carried by the runtime-owned `thetaAbort.signal` defined in [Pi Integration Contract — Cancellation source](./pi-integration-contract/host-interfaces-core.md#cancellation-source); the runtime forwards Pi's per-handler `ctx.signal` (and the tool-exposed `signal` parameter, and parent-`invoke` signals) into `thetaAbort` per the per-entry-point rules in [Cancellation — Forwarding into `thetaAbort`](./cancellation.md) and [Pi Integration Contract — Cancellation source](./pi-integration-contract/host-interfaces-core.md#cancellation-source). The closed enumeration of these three terminal outcomes — and the per-cause caller surface map (what an `invoke` parent sees, what a slash caller sees, what the operator (per [Glossary — operator](./glossary.md#operator)) observes per channel) — are owned by [Errors and Results — Terminal outcomes](./errors-and-results/error-model.md#terminal-outcomes); the *partial-append contract* (turns appended before the terminal event remain in the driven conversation; the runtime performs no implicit rollback; mid-stream user-visible streaming fragments are governed separately) is owned by [Errors and Results — Partial-append contract](./errors-and-results/error-model.md#partial-append-contract). Failures that occur before evaluation begins are owned, with their closed enumeration and per-cause routing rule, by [Errors and Results — Terminal outcomes](./errors-and-results/error-model.md#terminal-outcomes); they never become evaluation outcomes. The success / fail / cancelled trichotomy applies only once evaluation has begun.

<a id="file-extension-grammar"></a>A theta is stored in one of two file extensions that share a single grammar and type system. `.theta` files are invocable as slash commands (see [Slash-Command Invocation](./slash-invocation.md)); `.thetalib` files are library modules whose top level is restricted to the declaration forms enumerated in [Imports — Permitted top-level forms](./imports.md#permitted-top-level-forms) (currently five: `import`, `export`, `fn`, `schema`, `enum` — *orientation aggregator per [Governance — GOV-30](./governance.md)*). `.thetalib` files are never directly invoked. Path literals and discovery enforce these extensions per the rules in [Imports](./imports.md), [Invocation](./invocation.md), and [Discovery](./discovery.md); see those pages for the diagnostic codes that fire on each violation. See [Discovery — File-extension namespace](./discovery/discovery-sources.md#file-extension-namespace) for the namespace-clearance note.

### Non-functional requirements

These cross-cutting theta 1.0 non-functional dispositions — the trust/security posture, the source-language-equivalence release goal, and the operator-facing observability contract — are not enumerated as a unit by any single topic page. The bullets are *informative orientation*: each one forward-links the topic page that owns the normative contract, and the two no-seam theta 1.0 dispositions among them — Trust boundary and Source-language stability — are also indexed under [Future Considerations — theta 1.0 non-goals](./future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals); the consolidated theta 1.0 non-goals list (cross-cutting dispositions theta 1.0 ships without a forward-compatibility seam) is owned by [Future Considerations — theta 1.0 non-goals](./future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals) and forward-linked from [theta 1.0 non-goals](./language-and-architecture.md#theta-1-0-non-goals-aggregator) below. The aggregator-vs-source lock-step convention for keeping these bullets honest as topic pages evolve is owned by [Governance — GOV-30](./governance.md).

- **Trust boundary.** theta 1.0 thetas execute inside the Pi extension-host process at full Node host-process privilege. theta 1.0 imposes no theta-level sandbox: filesystem, network, and Pi-API access available to theta code and to its declared callables are exactly those available to any Pi extension running in the same process — Pi's `ExtensionAPI` and `ExtensionContext` surfaces (declared in `@earendil-works/pi-coding-agent`'s `dist/core/extensions/types.d.ts` across the release range pinned by [Pi Integration Contract — Host prerequisites — Pi SDK pin](./pi-integration-contract/host-prerequisites.md#pi-sdk-pin)) and the resource-loader and tool-host surfaces grant capabilities at the host-process level; Pi exposes no per-extension privilege scoping the runtime can rely on as a security boundary. That absence is a presupposition on the pinned Pi minor rather than a claim that the runtime has audited Pi's full `ExtensionAPI` / `ExtensionContext` surface and confirmed no per-extension privilege facet exists: the build-time SDK surface-inventory assertion run by [Pi version bump procedure](./pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) is a presence check on the seven named SDK capabilities and does not enumerate new top-level members, so a future Pi minor adding a privilege-scoping facet does not surface at that gate; detection of any such facet is routed to the per-bump editorial-review item [Per-extension privilege-scoping facet](./pi-integration-contract/version-bump-step2.md#bump-checklist-privilege-scoping-facet). The runtime itself interposes no additional access channels (sandbox, capability filter, mediated proxy) per [Pi Integration Contract — No additional access channels](./pi-integration-contract/trust-boundary.md#no-extra-mediation). The theta's declared *callable set* (the entries listed under the `tools:` frontmatter field; default rule for absent or empty `tools:` is owned by [Parameters and Frontmatter — `tools`](./frontmatter/frontmatter-fields-a.md#tools)) is a configuration knob over the *model's* reachable callable set, NOT a host-process sandbox: it is enforced mechanically by the per-mode wiring rule owned by [Pi Integration Contract — Tool-registration lifetime and visibility](./pi-integration-contract/tool-registration-lifetime.md#tool-registration-lifetime-and-visibility), which owns both the per-mode composition of that set and the SDK-call mechanism. Any callable a theta declares exposes the full underlying capability of that callable to the theta's model; the runtime applies no privilege classification or filtering. `bash` and `read` are illustrative — the rule does not depend on a callable being deemed "privileged." The host-side denial-surface rule — how denials of filesystem, network, or Pi-API access reach theta code through the tool that issued the request, and that silent success on denial is forbidden — is owned by [Pi Integration Contract — No additional access channels](./pi-integration-contract/trust-boundary.md#no-extra-mediation); the complete enumeration of observable `execute()` outcomes is owned by [Tool Calls — Failures](./tool-calls.md) and [Pi Integration Contract — Tool execution from theta code](./pi-integration-contract/host-interfaces-core.md#tool-execution-from-theta-code). A per-theta capability model is **out of scope for theta 1.0**; the disposition is owned by the **No per-theta sandbox or capability model** bullet under [Future Considerations — theta 1.0 non-goals](./future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals).

- <a id="source-language-stability"></a> **Source-language stability.** A `.theta` or `.thetalib` file that [loads cleanly](./governance/source-language-stability.md#gov-15-loads-cleanly) under theta 1.0.0 is expected to load under every theta 1.x release and to produce, for any given input, identical (a) return values, (b) ordered diagnostic-code sequences, and (c) equivalent `theta-system-note` content strings. Two content strings are equivalent under observable (c) when they are byte-identical — i.e. equal as UTF-8 byte sequences, the encoding substrate pinned by [Lexical — Encoding](./lexical.md) — after normalising the placeholder sub-fields whose rendered value is permitted to vary per invocation or per run — the per-invocation identifiers (e.g. `<invocation-id>` / `<uuid>`), the wall-clock-derived values (e.g. `<ms>`), and the category-8 host-derived freeform tails — as classified by the placeholder-rendering categories in [Diagnostics — Placeholder rendering](./diagnostics/placeholder-rendering-a.md#placeholder-rendering-normative); the fixed (non-variable) placeholder renderings and the surrounding literal template bytes are themselves byte-identical. Observable (c)'s equivalence relation, not the exclusion list, governs variation that is embedded as a placeholder sub-field inside a content string. Wall-clock timing, token counts, and log-line volume remain excluded from this expectation as independent observables in their own right — distinct from the wall-clock-derived sub-fields that observable (c) normalises within a content string. This is a release-process goal, not a normative obligation; the equivalence statement, the theta 1.0 review-vs-mechanical-gate posture, and the deferred-conformance-suite framing are owned by [Governance — GOV-15](./governance/source-language-stability.md#gov-15); the companion review-posture guidance (formerly GOV-14) is now retired and demoted to a non-normative editorial note (see [Retired REQ-IDs](./governance/anchor-scheme-and-retired.md#retired-req-ids)). Introducing a new hard runtime ceiling within theta 1.x — or retiring or relaxing an existing one — is a recognised carve-out to that equivalence per [GOV-15](./governance/source-language-stability.md#gov-15); the in-scope input set (defined post-hoc over the inter-release diff), the carve-out's diagnostic-emissions scope (per [GOV-15 — Ceiling-set carve-out](./governance/source-language-stability.md#ceiling-set-carve-out)), and the attribution test that distinguishes carve-out-eligible edits from collateral edits are all owned by [GOV-15](./governance/source-language-stability.md#gov-15). Editing the diagnostics code registry under DIAG-2/3/4 is a second recognised carve-out to that equivalence — code additions and removals are admissible within a theta 1.x minor for the inputs they touch, while renames and *Message* rewords are breaking changes deferred to theta 2.0 — owned by [GOV-15 — Diagnostic-registry carve-out](./governance/source-language-stability.md#diagnostic-registry-carve-out). The long-term migration framing (no major-version migration mechanism) is owned by [Future Considerations — theta 1.0 non-goals](./future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals). Migration across major versions is out of theta 1.0 scope.
