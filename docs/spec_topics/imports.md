# Imports

`.theta` files import schemas and functions from **`.thetalib`** files ‚Äî a separate extension dedicated to shared theta library code. `.theta` files are *not* importable from each other. This split keeps invocable thetas (slash commands) and reusable building blocks (libraries) cleanly separated.

```theta
import { Author, persona_block } from "./shared/personas.thetalib"
```

<a id="permitted-top-level-forms"></a>

**`.thetalib` file rules:**

- Top-level may contain only `import`, `export`, `schema`, `enum`, and `fn` declarations. No top-level statements, `let` bindings, or queries (`theta/parse/thetalib-top-level-statement`).
- Inside `fn` bodies, the full Theta language is available, including `@`...`` queries. A query inside an imported function executes against the *calling* `.theta`'s conversation when the function is invoked.
- Never slash-command-discovered. A `.thetalib` file is invisible to the `/<name>` autocomplete; it is only ever reached via `import`.
- May call `invoke(...)`. The path resolves relative to the `.thetalib` file's location; the invocation executes against the *calling* `.theta`'s conversation (or spawns a fresh isolated one if the callee is subagent-mode), exactly like a `@`...`` query inside a thetalib function. Cycle detection from [Invocation](./invocation.md) walks invoke paths originating from thetalib functions too. A cross-file `.thetalib` `fn` call is itself a countable frame against the invoke-depth bound owned normatively by [Invocation ‚Äî Invocation depth bound](./invocation.md#inv-4).
- May declare a `subagent fn` (theta 1.2; [Functions ‚Äî FN-9](./functions.md#fn-9)). The `subagent` modifier is admissible on a `.thetalib` `fn`, giving a shared, properly isolated in-library helper ‚Äî the `.thetalib` counterpart of the "spawns a fresh isolated one" rule for a subagent-mode `invoke` callee above. Its `@`...`` queries target a **fresh spawned session** regardless of which theta imported the helper, unlike an ordinary `.thetalib` `fn` whose queries run against the calling `.theta`'s conversation and therefore provide no isolation. With no `with { ‚Ä¶ }` clause the spawned session inherits the **calling** theta's session configuration ‚Äî the same "calling theta" anchor as the conversation rule above ‚Äî and a `with { tools: [ ‚Ä¶ ] }` clause resolves its tool names against the **calling** theta's callable set (a `.thetalib` has no frontmatter `tools:` of its own), with a name absent there failing through the existing callable-resolution channel. A `subagent fn` call is a countable frame against the invoke-depth bound ([Invocation ‚Äî INV-4](./invocation.md#inv-4)) whether or not caller and callee share a source file.

**Path resolution.** theta 1.0 supports relative paths only: `"./shared/personas.thetalib"`, `"../lib/schemas.thetalib"`. Paths must end in `.thetalib` ‚Äî the extension match is byte-exact lowercase per [Lexical ‚Äî Extension matching](./lexical.md#extension-matching); an `import` path whose literal does not end in `.thetalib` (including a `.theta` path or any non-lowercase variant such as `.THETALIB`) is a parse error `theta/parse/import-non-thetalib-extension`. Paths resolve relative to the importing file's directory. Path literals use forward-slash separators only ‚Äî a backslash is a parse error per the "Path literals" rule in [Lexical Structure](./lexical.md). Project-rooted (`/theta/...`) and package-style (`@scope/pkg`) imports are out of scope for theta 1.0. See [Future Considerations](./future-considerations.md).

**Resolver interface.** Import-path resolution goes through a single named seam ‚Äî a `Resolver` interface with the shape `resolve(spec: string, fromFile: string): string` ‚Äî rather than a hard-coded relative-path computation inlined at the import call site. theta 1.0.0 ships exactly one implementation: a relative-path resolver that joins `spec` against the directory of `fromFile` and requires the `.thetalib` extension; non-relative specs (`@scope/pkg`, `/theta/...`) fail this resolver as unresolvable paths and surface through the same load-time diagnostic channel as any other unresolvable `.thetalib` import. *Non-normative implementation note.* theta 1.0 expects every `.thetalib` import to flow through this single named seam rather than inline `path.resolve` calls at import sites, so that the deferred package-style and project-rooted import extensions in [Future Considerations](./future-considerations.md) can be added by registering additional `Resolver` implementations rather than rewriting import-site code; the single-seam factoring is a maintainability convention, not an observable conformance point.

<a id="imp-1"></a> **IMP-1 (resolver failure contract).** A `Resolver` MUST signal an unresolvable spec by throwing; theta 1.0's load pipeline treats a throw from `resolve` as a resolution failure, emits the load-time diagnostic `theta/load/unresolvable-thetalib-path` (see [Diagnostics ‚Äî `theta/load/*` registry](./diagnostics/code-registry-load.md)) against the importing file, and does not register that file. A spec is unresolvable when it is non-relative (`@scope/pkg`, `/theta/...`, or any other spec the relative-path resolver rejects) or when, after relative resolution against the importing file's directory, no directory entry exists whose name matches the path literal's final segment **byte-for-byte** (UTF-8, case-sensitive). A byte-for-byte-matching entry that exists but is not readable ‚Äî `EACCES` / `EPERM`, or a broken symlink on that entry ‚Äî is likewise unresolvable. The byte-for-byte rule holds on every host regardless of the host filesystem's case-equivalence model, mirroring [Directory Convention ‚Äî Non-canonical extension case](./discovery/discovery-sources.md); it composes with the byte-exact `.thetalib` extension check in [Lexical ‚Äî Extension matching](./lexical.md#extension-matching), which is one component of the byte-exact final-segment match rather than a separate stage. Byte-exactness is taken against the bytes [`FileSystem.readdir`](./pi-integration-contract/host-interfaces-services.md#pic-13) returns for the resolved parent directory, not against any Unicode-normalised form of the path literal (NFC vs NFD spellings are not folded; see [Lexical ‚Äî Encoding](./lexical.md)). `theta/load/unresolvable-thetalib-path` covers both the no-entry case and the case where a case-variant (or otherwise non-byte-exact) entry exists; the diagnostic still renders the path literal as written per the `<path>` placeholder rule ([Diagnostics ‚Äî Placeholder rendering](./diagnostics/placeholder-rendering-b.md)). *Non-normative implementation note.* A conformant `Resolver` cannot rely on a single `FileSystem.exists` / `readText` on a case-insensitive host ‚Äî that would succeed on `Personas.thetalib` when the literal said `personas.thetalib`; it enumerates the resolved parent directory once via `FileSystem.readdir` and compares the final segment byte-for-byte, paying one extra `readdir` only on the failure path.

> **theta 1.0 seam ‚Äî `Resolver` interface.** <a id="theta-1-0-seam-resolver-interface"></a> Import-path resolution flows through the named `Resolver` seam (`resolve(spec: string, fromFile: string): string`). theta 1.0.0 ships exactly one implementation ‚Äî a relative-path resolver that requires the `.thetalib` extension ‚Äî and non-relative specs (`@scope/pkg`, `/theta/...`) fail this resolver as unresolvable paths. The seam is what lets the deferred package-style and project-rooted import extensions in [Future Considerations ‚Äî Surface extensions](./future-considerations/surface-extensions.md#surface-extensions-v1-leaves-a-seam) land by registering additional `Resolver` implementations rather than rewriting import-site code. *Non-normative implementation note.* Routing through the single named seam rather than open-coding `path.resolve` at import sites is a maintainability convention, not an observable conformance point.

**Visibility.** Every top-level `schema`, `enum`, and `fn` in a `.thetalib` file is implicitly exported. There is no `export` keyword on declarations and no privacy modifier; `.thetalib` files have no internal-only symbols in theta 1.0.

**Re-exports.** A `.thetalib` may re-export a symbol from another `.thetalib` using a dedicated form that creates no local binding:

```theta
export { Author } from "./personas.thetalib"
export { Author as Reviewer } from "./personas.thetalib"
```

A plain `import { Author } from "./personas.thetalib"` does **not** re-export `Author` from the importing file ‚Äî only declarations and explicit `export ... from` forms are visible to downstream importers.

**Unknown imported symbol.** An `import { Foo }` or `export { Foo } from` specifier ‚Äî including the `as`-aliased forms `import { Foo as Bar }` and `export { Foo as Bar } from` ‚Äî that names a symbol `Foo` which is neither a top-level declaration nor a transitive re-export (`export ‚Ä¶ from`) of the resolved `.thetalib` file is a static error `theta/parse/import-unknown-symbol`. The error names the source symbol (`Foo`), not the alias (`Bar`). The check fires after the resolved `.thetalib` file's own parse completes: the resolved file's set of top-level declarations and `export ‚Ä¶ from` re-exports must be known before an importing specifier can be matched against it. It participates in the [Diagnostics ‚Äî Multi-error reporting](./diagnostics.md) batching rule rather than fast-failing ‚Äî an unknown-symbol error is collected alongside every other parse / type error from the importing file and its transitive `.thetalib` imports, and all are reported in one batch. This error is distinct from `theta/parse/unknown-identifier`, which is scoped to bare identifiers in expression position and is never raised for `import` or `export ‚Ä¶ from` specifiers.

**Name collisions.** Two imports bringing in the same symbol name is `theta/parse/import-name-collision`. Resolve with `as`-aliasing:

```theta
import { Author as AuthorA } from "./team-a.thetalib"
import { Author as AuthorB } from "./team-b.thetalib"
```

The same `as` form is also available for self-clarity (`import { ReviewScore as Score } from "./scoring.thetalib"`). An imported symbol whose name collides with a top-level declaration in the same file is also `theta/parse/import-name-collision` ‚Äî no implicit shadowing.

**Cycles.** Import cycles between `.thetalib` files are detected at parse time by walking the static import graph and reported as `theta/load/import-cycle` with the cycle path printed (`"import cycle: a.thetalib ‚Üí b.thetalib ‚Üí a.thetalib"`). `.thetalib` files contain only declarations ‚Äî no top-level statements, no initialisation order ‚Äî so cycles serve no purpose and only happen by accident.
