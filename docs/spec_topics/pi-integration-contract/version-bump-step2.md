2. **Re-run the SDK surface-inventory tests.** The build-time test surface bundles two co-numbered assertions whose inputs and resolution recipes are distinct; PIC pins them as <a id="bump-step-2-positive"></a>**(a)** and <a id="bump-step-2-negative"></a>**(b)** below so a failing red can be unambiguously routed.

    **(a) Positive direction — literal-read of the imported namespace.** The literal-read assertion imports `@earendil-works/pi-coding-agent` at the candidate version and asserts each of the seven named SDK capabilities enumerated in [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) above is present on the imported namespace. A red here means a capability has been removed or renamed; resolve by updating both the `SDK_SURFACE_INVENTORY` entry and the corresponding capability obligation in [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) (and the matching probe-side description and the `spec.md` cross-reference list) in the same edit before the bump. Step 2(a) MUST additionally assert that the `CAPABILITY_OBLIGATIONS` constant has exactly seven entries (`CAPABILITY_OBLIGATIONS.length === 7`). `CAPABILITY_OBLIGATIONS` is a separate constant in the pinned-constants block, distinct from the [Step 0 (c)](./capability-probe.md#entry-capability-probe) factory-probable capability list (which is a five-element array of probe-time capability names — capabilities 1, 2, 3, 4, and 6 — whose entries enumerate the ten `typeof <path> === "function"` namespace-function paths that step (c) probes). The two constants have different cardinalities (7 vs 5), different runtime consumers (the literal-read assertion of step 2(a) above vs the [Step 0 (c)](./capability-probe.md#entry-capability-probe) capability probe), and different bump-procedure groupings (step 5 grouping (iv) below vs step 5 grouping (i) below). The left-hand side of that count-equality comparison is sourced from the `CAPABILITY_OBLIGATIONS` constant — a closed array of the seven anchor IDs [`sdk-cap-slash-command-registration`](./capability-inventory-items.md#sdk-cap-slash-command-registration) … [`sdk-cap-binder-llm-model`](./capability-inventory-items.md#sdk-cap-binder-llm-model), pinned in the same source-of-truth pinned-constants block in the extension module that holds the four Step-0 probe constants — and the assertion compares `CAPABILITY_OBLIGATIONS.length === 7`; adding an 8th capability obligation therefore requires extending `CAPABILITY_OBLIGATIONS` and bumping the integer literal in the same edit per [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote)'s co-edit checklist below. This assertion is the mechanical gate against drift between the `CAPABILITY_OBLIGATIONS` constant and the integer literal `7`. Drift between the markdown items-1–7 enumeration and the natural-language `"seven"` restatements elsewhere (this section's preamble sentence "The runtime depends on the seven named SDK capabilities enumerated below", the [`spec.md` — Orientation — Prerequisites](../../spec/overview-and-orientation.md#orientation) sentence "The seven SDK capabilities the runtime depends on …", and any other cardinality-bearing prose on the seven count) is contributor-discipline; [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote) below's catch-all co-edit obligation is the contributor-side mitigation. A red on this assertion means `CAPABILITY_OBLIGATIONS.length` and the integer literal `7` have diverged, and is resolved by [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote) below (which co-edits the integer literal alongside the inventory in the same commit). An unrecognised-`kind` literal-read failure — emitted by the routable-discriminator obligation pinned under [Target surface categories](./audit-recognised-shapes.md#audit-target-surface-categories) above when an `SDK_SURFACE_INVENTORY` entry's `kind` field is not recognised by the literal-read assertion's branch table — is resolved by **adding a recognising branch to the literal-read assertion's branch table for the new `kind`, in the same edit that adds the new entry** (on the same footing as [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote) below's promote-and-co-edit obligation, and consuming the new entry's per-kind payload field shape from the implementation's inventory the same edit lands per *Target surface categories* above). This sub-step remains the single owner of import-specifier re-pinning for both the seven function capabilities and that existing `type-union-snapshot` entry: a TS230x-family module-import-resolution failure (`TS2305`, `TS2724`, `TS2307`, `TS2614`, or any other TS code in the same family) routed here from step 5 below — including one originating from step 5's `SessionShutdownEvent['reason']` brand-string assertion when a Pi patch drops or renames `SessionShutdownEvent` on the package-root re-export surface — is resolved by re-pinning that type's import specifier (to whatever sub-export, or deep `dist/...` path as a last resort, the candidate Pi minor exposes the type on) in the same edit as the rest of the bump. *Category-(3) drift contributor obligation.* Step 2(a) additionally pins a contributor obligation against category-(3) staleness on a Pi minor bump (the bump-time analogue of the *Target surface categories* (3) framing under [Inventory-closure audit](./inventory-audit-intro.md#sdk-cap-inventory-closure-audit) above, which makes the SDK type declarations the single source of truth for category (3)'s closed in-scope set): on each bump, the contributor MUST cross-reference the candidate `@earendil-works/pi-coding-agent` minor's `.d.ts` declarations of `ExtensionContext` and `ExtensionCommandContext` against the `SDK_SURFACE_INVENTORY`'s category-(3) entries. A new SDK-side member the runtime does not yet consume is **not** required to be added to the inventory at the bump (the inventory tracks consumed surface, not the entire SDK type's member set); a new member the runtime starts to consume in the same bump MUST be added to `SDK_SURFACE_INVENTORY` as a category-(3) entry under [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote)'s promote arm in the same edit, on the same footing as any other newly consumed surface that branch (2) catches. This obligation is contributor-discipline rather than a type-aware mechanical gate (a mechanical gate would commit step 2 to a TypeScript-program load PIC declines to require), and is the bump-time counterpart to the [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote) promote arm that the inventory-closure audit catches at non-bump times. The literal-read assertion's branch table arm for category-(3) entries SHOULD additionally include a smoke check that asserts every category-(3) entry's `path` member-name segment (per the [*Category-(1)/(3) inventory join key (per-category specialisation)*](./audit-resolution.md#audit-category-1-3-inventory-join-key) sub-paragraph above, which pins the category-(3) lookup as a rightmost-dot-separated-segment match between the SDK type's member name and the entry's `path` field) still resolves on the imported `ExtensionContext` and `ExtensionCommandContext` types — for example, a per-entry `"<member>" in <importedExtensionContextLikePrototype>`-equivalent check, or its type-side analogue — so a Pi minor that drops or renames a category-(3) member the inventory enumerates surfaces as a literal-read failure rather than as a silent stale entry the contributor obligation above did not catch.
