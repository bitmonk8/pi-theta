      A **fail** outcome is recorded when **any** of the following conditions holds. Each condition is self-contained; the **AND** in the final textual-locatability condition binds only within that condition and does not reach the other conditions:

      - The auditor cannot demonstrate that the candidate minor's `prompt` body still establishes the [precondition (b)](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions) behavioural property above (i.e. that two loom-invocation slash dispatches against the same session cannot overlap their snapshot/restore windows — whether via the loom 1.0-pinned cooperative-`await` chain `prompt` → `_tryExecuteExtensionCommand` → `await command.handler(args, ctx)` end-to-end (which structurally witnesses the strictly stronger property of handler-body non-overlap and trivially establishes the narrower snapshot/restore-window non-overlap precondition (b) actually depends on, since the snapshot/restore window is bounded inside the handler body), or via any equivalent Pi-side mechanism that establishes the same non-overlap invariant; non-exhaustive examples of pattern shapes that trip the fail predicate include the `_tryExecuteExtensionCommand` step becoming fire-and-forget, the awaited dispatch being replaced by a queue whose per-session ordering is weaker than a serialised awaited dequeue (the comparator baseline — a per-session FIFO whose dequeue body is itself `await`ed end-to-end satisfies the non-overlap property and is a pass; a queue that dispatches the next handler before the previous handler's body resolves does not), the `await` being short-circuited around, handlers dispatched via `Promise.all` over a per-session batch, or microtask-deferred handlers whose bodies do not serialise; non-exhaustive examples of mechanism-preserving variants that satisfy the non-overlap property and are a pass include an explicit per-session lock wrapping the handler-body await chain, a single-consumer channel that dequeues each handler with `await` before pulling the next, and a microtask-deferred handler chain whose bodies still serialise via the same await semantics).
      - Any enumerated `session.prompt(...)` call site (in the loom 1.0.0 driver set files or in any newly-discovered glob match) dispatches without awaiting it before initiating the next.
      - Any newly-discovered `dist/modes/*/*.js` file with a reachable `session.prompt(...)` call site (per [(e.ii)](#bump-checklist-slash-dispatch-serialisation-ii)'s call-graph reachability check) appears in the candidate minor.
      - *Textual-locatability with no equivalent site:* the loom 1.0-cited evidence trail is no longer textually locatable in the candidate minor (the `AgentSession.prototype.prompt` body, the `_tryExecuteExtensionCommand` step inside it, the `Editor.onSubmit` site, or any of the cited `dist/*.js` paths cannot be identified by their loom 1.0 names or paths — e.g. because Pi has minified, re-bundled, renamed the `_`-prefixed symbol, or restructured the `dist/` tree, per the *known-fragile evidence* flagging in precondition (b) of [snapshot/restore Pi behavioural preconditions](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions) above — so the recipe cannot reach a pass verdict on textual diffability alone) **AND** the auditor cannot identify an equivalent reachable `session.prompt(...)` call site under [(e.ii)](#bump-checklist-slash-dispatch-serialisation-ii)'s call-graph reachability check from a per-mode entry point that supplies the same cooperative-await contribution as the un-locatable loom 1.0 site (e.g. for the `Editor.onSubmit` TUI-layer detail flagged as orthogonal under precondition (b) above, a renamed or replaced submit-pump callback whose body still awaits each `session.prompt(...)` call before initiating the next is the equivalent reachable site, and locating it satisfies [(e.ii)](#bump-checklist-slash-dispatch-serialisation-ii)'s reachability check and exempts the un-locatable loom 1.0 name from the textual-locatability fail trigger; the conservative-posture rule applies if the auditor cannot positively identify such an equivalent site).

      **conservative-posture rule:** if the auditor cannot positively demonstrate the non-overlap property holds against the candidate minor — whether because the evidence trail has drifted, because the new shape's serialisation argument is unclear, or for any other reason — the audit outcome MUST be recorded as `fail`, never `N/A`.

      A fail outcome requires adding a defensive per-extension-instance runtime serialisation of the snapshot → swap → body → restore sequence (established by any synchronisation construct that establishes the precondition (b) non-overlap property — the loom 1.0 reference construct is a per-extension-instance mutex, but an `async`-aware lock, a single-consumer awaited queue, a channel, or any equivalent of the mechanism-preserving variants the pass arm enumerates above is equally conformant — and keyed on the factory-captured `pi: ExtensionAPI` reference — the only identity the loom holds at the snapshot/restore site, since `ExtensionCommandContext` is per-slash-handler-invocation and the user `AgentSession` is not exposed to prompt-mode looms; the per-extension-instance keying granularity is well-defined only while **both** the page-pinned one-`pi`-per-extension-instance lifetime (see the [loom 1.0 seam — mid-loom user-session replacement](./conversation-drive.md#v1-seam-mid-loom-user-session-replacement) paragraph above) AND the loom 1.0 one-active-user-session-per-extension-instance presupposition (pinned at *No concurrent user sessions in the same host process* in [Future Considerations](../future-considerations/model-changes-and-non-goals.md#v1-non-goals)) hold — together, on the recovery path, the keyed recovery serialisation is the unique serialisation point per user session (it exists only when this recovery path is engaged on a fail outcome of this audit recipe — loom 1.0 ships without it on a pass outcome, per precondition (b)'s cooperative-`await` framing) — and a future Pi minor that weakens **either** keying-granularity invariant — the one-`pi`-per-extension-instance lifetime or the one-active-user-session-per-extension-instance presupposition — requires revisiting the keying granularity as part of this checklist item at the same time the precondition is re-validated (the closing "If either ... weakens" sentence under [snapshot/restore Pi behavioural preconditions](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions) above scopes only the two behavioural preconditions enumerated there and does not absorb the keying-granularity revisitation obligation)) in the same edit as the bump.

The literal-read assertion's branch table arm for category-(3) entries SHOULD additionally include a smoke check that asserts every category-(3) entry's `path` member-name segment (per the [*Category-(1)/(3) inventory join key (per-category specialisation)*](./audit-resolution.md#audit-category-1-3-inventory-join-key) sub-paragraph above, which pins the category-(3) lookup as a rightmost-dot-separated-segment match between the SDK type's member name and the entry's `path` field) still resolves on the imported `ExtensionContext` and `ExtensionCommandContext` types — for example, a per-entry `"<member>" in <importedExtensionContextLikePrototype>`-equivalent check, or its type-side analogue — so a Pi minor that drops or renames a category-(3) member the inventory enumerates surfaces as a literal-read failure rather than as a silent stale entry the contributor obligation above did not catch. The smoke check is `SHOULD` rather than `MUST` because the contributor obligation above is the primary gate; the smoke check is a belt-and-braces backstop and the implementation MAY route its concrete shape (per-entry membership probe vs. type-side assertion) without re-litigation here. The literal-read assertion additionally carries the *strict-capability absence-under-the-probed-name* arm — pinned at [Inventory-closure audit — strict-capability absence-under-the-probed-name arm](./inventory-audit-intro.md#strict-capability-absence-under-probed-name) above — which fails the assertion if a reachable `Model<Api>` declaration exposes a `strictCapable` member under the probed name, so both the rename-detection arm ([bump-procedure step 7](./version-bump-triggers.md)) and the absence arm are enumerated at the assertion's own definition.
