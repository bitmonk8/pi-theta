# Audit failures

  - *Stale sub-kinds.* The audit MUST surface every stale marker under family (5) with a stable `<symptom>` token per sub-kind (per the *Failure-surface contract* below): (s1) *no-surface-on-line* — the originating line per *Marker scope* / *Multi-line surface placement* above contains zero in-scope surface references at audit time, whether because the marker was placed on a line that never held a recognised in-scope reference (a placement-error sub-case) or because one or more recognised in-scope references previously originated on that line and have since been removed (an all-references-removed sub-case left over from a refactor that deleted the references it authorised); the two sub-cases collapse into the single `<symptom>` token (s1) because they are not in general statically discriminable from a source-tree snapshot without git history, and the contributor recovery action is the same for both — delete the marker line per [step 2(b) branch (4)](./version-bump-step2b.md#bump-step-2b-stale-rewrite) below sub-case 1 ("no-surface-on-line"); the placement-error vs all-references-removed distinction is preserved here as commentary for the contributor's diff-reading benefit, not as separate `<symptom>` tokens. (s2) *all-in-inventory* (short name; the predicate covers both `SDK_SURFACE_INVENTORY` resolution and typebox sibling allow-list resolution per *Target surface categories* typebox carve-out above, so the token is more precisely "all-in-inventory-or-allow-list" — PIC retains the shorter `<symptom>` token to keep the wire-discriminator stable, and the parenthetical here is the normative scope) — every in-scope reference on the originating line now resolves to an `SDK_SURFACE_INVENTORY` entry, or (for references whose carrier is the `typebox` peer package per *Target surface categories* typebox carve-out above) to a typebox sibling allow-list entry — `{ Type }` for named-import references and `{ Unsafe }` for `Type.<member>` member-access references — per the *Inventory-first resolution* rule above (the marker authorises nothing the inventory or the typebox allow-lists do not already), recovered by deleting the marker per [step 2(b) branch (4)](./version-bump-step2b.md#bump-step-2b-stale-rewrite) below sub-case 2 ("surface-now-in-inventory"; the same sub-case covers a previously-unauthorised typebox surface — for example a `Type.Object` access — that has since been added to the `{ Unsafe }`-style member-access allow-list under [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote)'s typebox-promotion arm). For both sub-kinds only the family (5) discriminator fires on the marked line — the line carries no out-of-inventory reference for a family (1) / (2) / (3) discriminator to attach to. The two `<symptom>` tokens MUST be distinct so a contributor routing to [step 2(b) branch (4)](./version-bump-step2b.md#bump-step-2b-stale-rewrite) below can tell stray-or-stale-marker removal (sub-case 1) from promotion-to-inventory removal (sub-case 2) without re-reading the diff.

  - *Substitution limitation (acknowledged soft surface).* Substituting one out-of-inventory surface for a *different* out-of-inventory surface on a marked line (e.g., an `// allow-pi-surface:` marker authorising `pi.alpha` rewritten to `pi.beta` with the marker carried over — illustrative placeholder names; neither symbol is in `SDK_SURFACE_INVENTORY` at loom 1.0, and the rule applies regardless of which non-inventory symbols are involved) is *not* mechanically detectable by the audit: per-line scope means the marker continues to authorise the substituted reference, and the citation's free-text justification cannot be machine-validated against the new symbol. This soft surface is mitigated by the contributor obligation in [step 2(b) branch (4)](./version-bump-step2b.md#bump-step-2b-stale-rewrite) below — every marker on a line touched by a refactor or bump commit MUST be re-read against the *Exemption mechanism* citation rule on the diff before the commit merges — and is **not** owed an additional audit discriminator.

  - *Stale-citation limitation (acknowledged soft surface).* A citation token that parses as well-formed per the *Citation-token grammar* above but whose REQ-ID does not (or no longer) name a stable spec ID, or whose PIC anchor does not (or no longer) name a stable anchor on this page (e.g. a typo'd `PIC#nonexistent-anchor`, a citation to a PIC anchor that has since been renamed or deleted, or a citation to a REQ-ID that was retired or never allocated), is *not* mechanically detectable by the audit: the *Citation-token grammar* check above is purely syntactic and the audit performs no citation-resolution sub-check. This soft surface is mitigated by the same contributor obligation in [step 2(b) branch (4)](./version-bump-step2b.md#bump-step-2b-stale-rewrite) below that mitigates the *Substitution limitation* above — every marker on a line touched by a refactor or bump commit MUST be re-read against the *Exemption mechanism* citation rule on the diff before the commit merges, and a citation that has gone stale MUST be rewritten under that branch's *citation has gone stale* case rather than carried over — and is **not** owed an additional audit discriminator in loom 1.0; widening the audit to validate citation resolution would require attaching a registry of valid REQ-IDs and PIC anchors to the audit input, which is a PIC edit deferred past loom 1.0.

  *Malformed-marker discriminator.* A *malformed exemption marker* is any same-line trailing comment whose token-prefix — defined as the longest leading substring consisting of either the literal `//` line-comment opener or the literal `/*` block-comment opener, followed by any leading intra-comment whitespace, followed by an ASCII run of letters and `-` whose case-folded form (each ASCII letter compared without regard to case) equals the literal `allow-pi-surface` — matches the well-formed lowercase-ASCII literal `// allow-pi-surface` either exactly or up to one or both of the case-folding and block-comment-opener relaxations above (so `// allow-pi-surface`, `// Allow-Pi-Surface`, `// ALLOW-PI-SURFACE`, `/* allow-pi-surface`, and `/* Allow-Pi-Surface` all match this prefix as candidates, while `// allow_pi_surface`, `// allow-pi-surfaces`, `//allowpisurface`, and other near-miss spellings whose ASCII letter-and-`-` run does not case-fold to `allow-pi-surface` do **not** match this prefix and are not lexical exemption candidates) but which fails any of (a) presence of the trailing `:` immediately after the `allow-pi-surface` run (a marker like `// allow-pi-surface PIC#x — ...` without the colon is malformed); (b) presence of a citation token (REQ-ID or PIC anchor) before the separator; (c) the separator-tolerance rule above (unrecognised separator character, or recognised separator without the required surrounding spaces); (d) the minimum-justification rule above; (e) placement as a trailing comment on the offending surface line itself (a marker on the line above, on the line below, or as a standalone comment line is *not* on the offending source line and is therefore malformed); (f) all-lowercase-ASCII spelling of the `allow-pi-surface` run (a candidate whose run case-folds to `allow-pi-surface` but is not byte-for-byte the lowercase form — e.g. `// Allow-Pi-Surface: PIC#x — ...`, `// ALLOW-PI-SURFACE: PIC#x — ...` — is malformed; the well-formed prefix MUST be lowercase ASCII); or (g) shape as a single-line `//` comment (a candidate that opens with `/*` — including the single-line block form `/* allow-pi-surface: PIC#x — ... */` and any multi-line block form — is malformed; the well-formed marker MUST be a single-line `//` trailing comment, since the audit's per-line scope semantics in *Marker scope is per source line* above are defined against single-line trailing comments only). The audit MUST surface every malformed marker under family (5)'s discriminator, AND MUST additionally surface the underlying out-of-inventory reference (if any) under its own family (1) / (2) / (3) discriminator on the same audit run, so the contributor sees both that the marker did not parse and which underlying surface remains unauthorised; routing the malformed marker only to the underlying-surface discriminator would silently demote the contributor's intended exemption and is prohibited. Comments whose token-prefix does not match `// allow-pi-surface` at all per the definition above are not lexical exemption candidates and surface only under the underlying-surface discriminator (no malformed-marker report is owed for them); the implementation MAY additionally emit a non-blocking advisory for near-miss spellings, but PIC does not require it.
- <a id="audit-failure-surface-contract"></a>*Failure-surface contract.* Each violation the audit emits MUST be unambiguously routable by a reader to exactly one stable failure-mode discriminator naming the failure family. The family set is **exhaustive at exactly the five families enumerated here** — (1) unauthorised `pi.<member>` access; (2) unauthorised peer-package named import from one of the four `@earendil-works/*` peers, or an unauthorised typebox surface (a named import other than `Type` from `typebox`, or a `Type.<member>` member-access where `<member>` is not in the typebox member-access allow-list `{ Unsafe }`) per the carve-out under *Target surface categories* above; (3) unauthorised `ctx.<member>` member access on the canonical `ctx` identifier (per *Target surface categories* (3) and *Recognised import/access shapes* above); (4) out-of-scope import/access shape (per *Recognised import/access shapes* above); (5) stale or malformed exemption marker (per *Exemption mechanism* above — both stale markers whose underlying surface has been resolved or removed and malformed markers per the malformed-marker discriminator there route to this single family). An audit implementation MUST NOT emit a violation outside this set. Each violation record MUST identify the offending source path, line, symbol, and either the inventory entry or exemption marker the contributor would add to satisfy the audit, mirroring step 5 below's *PIC owns the routing semantics* obligation. The discriminator string carrying the family name MUST be a single grep-able token of the structural shape `audit/<class>/<family>/<symptom>` — `<class>` is fixed at exactly one of the three literal ASCII tokens `violation`, `infra`, or `canary` (the three-class partition PIC pins below); for the five violation families above the `<class>` segment is the literal `violation` and for every emitted record the `<class>` segment is one of these three literals (no other `<class>` tokens are permitted). `<family>` is the implementation-chosen short name for one of the five families above (one stable `<family>` token per family, fixed at first implementation and pinned as a permanent test invariant) and `<symptom>` is an implementation-chosen kebab-case symptom identifier (lowercase ASCII letters, digits, and `-` only; no whitespace, no `/`) — and MUST appear verbatim in every emitted violation record so a CI parser can route violations to families by token-prefix match without an out-of-band translation table, and additionally to one of the three classes by the `<class>` segment alone (the `audit/<class>/...` two-segment prefix) without consulting an out-of-band manifest. The implementation owns the literal `<family>` and `<symptom>` strings, the failure-message wording, and the negative-test fixtures, and pins them into permanent test invariants. The implementation MUST publish at least one negative-test fixture per stable `<symptom>` token enumerated under each family (so families with multiple sub-kinds — notably family (5) with stale sub-kinds (s1)/(s2) per *Stale sub-kinds* above and malformed-marker clauses (a)–(g) per *Malformed-marker discriminator* above — get one fixture per sub-kind/clause). A fixture floor on `<family>` count alone is too coarse to exercise the per-`<symptom>` routability obligation this paragraph pins, since a single family-(5) fixture would leave eight of the nine family-(5) `<symptom>` tokens unexercised (the family has nine tokens total: both stale sub-kinds (s1) and (s2) plus all seven malformed-marker clauses (a)–(g); one fixture exercises one token); PIC pins the per-`<symptom>` floor here and the implementation MUST NOT collapse multiple `<symptom>` tokens onto a single fixture. The implementation MUST additionally publish at least one negative-test fixture per multi-line surface-placement shape enumerated under *Multi-line surface placement* above (one fixture each for category-(1) `pi.<member>` split, category-(2) named-import split, category-(3) `ctx`-rooted chain split, and per-symbol marker placement on multi-line named imports), exercising the originating-line rule for each shape — without these the asymmetric originating-line rule between (i) and (iii) is unexercised by the per-`<symptom>` fixture floor above and an audit implementation that always uses line 1 of any multi-line span would satisfy the floor while violating the placement rule. PIC pins the family-set closure, the structural `audit/<class>/<family>/<symptom>` shape, the three-class `<class>` enumeration above, the three-class partition pinned in the next sentence, the per-`<symptom>` fixture floor pinned in the previous sentence, and the per-discriminator routability obligation here. *Three-class partition.* PIC pins the partition of the audit's leaf-chosen `<family>` tokens across the three `<class>` prefixes as: `violation` covers the five inventory-closure-audit families enumerated above; `infra` covers the infrastructure-failure-handling families per [Infrastructure-failure handling](./audit-wire-and-canary.md#audit-infrastructure-failure-handling) below; `canary` covers the non-empty-scan canary's families per [Non-empty-scan canary](./audit-wire-and-canary.md#audit-non-empty-scan-canary) below. The leaf's existing `<family>` tokens MUST be re-namespaced under one of the three `<class>` prefixes per this partition (each `<family>` token belongs to exactly one class, which class is fixed by the obligation that introduces it), and the per-class disjoint-token obligations the *Infrastructure-failure handling* and *Non-empty-scan canary* sub-paragraphs below pin reduce to within-class `<family>` disjointness for each class — cross-class disjointness is structurally guaranteed by the `<class>` prefix and need not be re-asserted on the `<family>` tokens themselves.
