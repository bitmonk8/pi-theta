# Audit wire and canary

    *Wire serialisation.* The audit's violation records (and the infrastructure-failure records the *Infrastructure-failure handling* sub-paragraph below pins onto the same shape) MUST be emitted as line-delimited UTF-8 text on **stdout** — one record per line, terminated by a single `\n`, with no JSON, no record grouping (e.g. by family or by file), and no trailing summary lines that share the leading-token shape below. The `audit/<class>/<family>/<symptom>` discriminator token the *Failure-surface contract* above pins MUST appear as the **leading whitespace-separated field** on each emitted line so a CI parser can route by leading-token prefix match without parsing the rest of the line, and partition records into the three classes by the `audit/<class>/...` two-segment prefix alone; the four record fields (`path`, `line`, `symbol`, `proposed-resolution` — in this fixed order) are packed into three tab-delimited record segments that follow the discriminator on the same line: segment (1) is the `path:line` pair (the `path` and `line` field values joined by a single ASCII colon `:`; a parser splits this segment on the rightmost `:` to recover the two field values, which accommodates Windows hosts whose `path` field may itself contain a `:` in the drive prefix), segment (2) is the `symbol` field value, and segment (3) is the `proposed-resolution` field value. The between-segment separator is the ASCII tab character (U+0009): each emitted line is the `audit/<class>/<family>/<symptom>` discriminator token, then a single ASCII tab, then segment (1) (`path:line`), then a single ASCII tab, then segment (2) (`symbol`), then a single ASCII tab, then segment (3) (`proposed-resolution`) — four tab-delimited tokens per line in total (the discriminator plus the three record segments). The four field values (`path`, `line`, `symbol`, `proposed-resolution`) MUST NOT contain the ASCII tab character on any record class (violation or infrastructure-failure) — bare identifiers, package-qualified imported names, the literal `<n/a>` sentinel, integer `line` values, and file paths are tab-free by their pinned shapes, and the family (4) `symbol` value is tab-free because the *Per-family record-shape table* above already collapses every run of one or more whitespace characters (including tabs and newlines) to a single ASCII space. The `proposed-resolution` field MUST be plain printable ASCII text (no tabs, no newlines, no non-ASCII characters); when the resolution arm a contributor would take cannot be expressed in plain ASCII, the leaf MUST collapse it to a printable-ASCII paraphrase that names the arm by reference (e.g. by the relevant `#bump-step-2b-*` anchor on this page) rather than by quoted source text. The four-field tab-separated shape (discriminator, `path:line`, `symbol`, `proposed-resolution`) is the wire contract a CI parser may rely on: a parser splitting each line on the ASCII tab obtains four fields, and a parser splitting the second field on the right-most `:` obtains the `path` and `line` sub-fields (the right-most-`:` rule accommodates Windows hosts whose `path` field may itself contain a `:` in the drive prefix). PIC pins the separator and the escaping rule above; the implementation MUST NOT substitute a different separator character or relax the no-tab obligation on any field. Records MUST be emitted in source-walk order — the audit's file-iteration order (which the implementation MUST pin to a deterministic ordering, e.g. lexicographic on POSIX-form path, so two runs against the same source tree emit byte-identical output) followed by within-file appearance order of references — and MUST NOT be re-ordered for visual grouping. The literal `<n/a>` sentinel above (lowercase ASCII, five characters, including the angle brackets) is the single permitted encoding for an absent named symbol on family (5) records and for an absent value on any infrastructure-failure-record field per the *Infrastructure-failure handling* sub-paragraph below; alternative encodings of an absent value (`""`, `null`, an empty field, `n/a` without brackets, `<N/A>`) MUST NOT be emitted. Stderr MAY carry a non-machine-parseable human-readable summary or warning text the implementation chooses (the *Non-empty-scan canary* sub-paragraph below already permits implementation-chosen failure-message wording), but the line-delimited stdout stream is the **sole** machine-parseable surface a CI parser is permitted to consume in loom 1.0; emitting violation or infrastructure-failure records on stderr, or in a JSON or grouped form on either stream, violates this rule.
- <a id="audit-infrastructure-failure-handling"></a>*Infrastructure-failure handling.* The audit MUST be fail-closed: any infrastructure failure that prevents it from reaching its assertions over the *audited source tree* (per *Audit scope* above) MUST surface as a non-zero exit from the `npm test` invocation step 2(b) names rather than as a silent green. The infrastructure-failure surface comprises (i) **implementation-agnostic failures** — a parse / syntax failure raised by the audit's own walker on an audited file, an audit-side file-system read failure on an audited path, a peer-package import failure preventing the audit module itself from loading, and a tool-side crash inside the audit (OOM, walker panic, parser/checker-host panic, write failure on the diagnostic sink); and (ii) **tooling-conditional failures** that arise only in a type-aware implementation (per the introductory paragraph's permission for a TypeScript-program symbol-table walk as an alternative to a static-AST walker) — a checker-side symbol-resolution failure on the audited surface (defined here by *category of failure* — a type-checker symbol-resolution failure on a Pi-imported symbol the audit relies on for type-directed resolution — and routed by that category description rather than by TS error code, since TypeScript publishes no closed taxonomy of error codes the spec could pin; illustrative examples at the loom 1.0 TypeScript pin are TS2304 (`Cannot find name`), TS2339 (`Property does not exist on type`), and TS2551 (the symbol-suggestion variant of TS2339), and any other diagnostic that fits the category description is on this rule. Assignability diagnostics — notably TS2322 (`Type X is not assignable to type Y`), and any other diagnostic whose category is type-equality / assignability rather than symbol resolution — are explicitly **not** in this category and remain owned by [step 1](./version-bump-intro.md#pi-version-bump-procedure) below's spec-prose-update obligation; routing them through this infrastructure-failure rule would silently swallow a real SDK-side type-equality regression on a load-bearing Pi-imported type as an infrastructure failure, which step 1 specifically prohibits by requiring the spec-prose update on every type error not carved out by the brand-string discriminator), and a module-import-resolution failure (the TS230x category step 1 above already names — TS2305 / TS2307 / TS2614 / TS2724 and any other diagnostic of the same category) originating at the bare-package import the audit performs to load the audited surface against the SDK. The two-category list is **non-exhaustive**: any failure raised inside the audit process before successful record emission that prevents the audit from reaching its assertions is on this rule, routed through an implementation-chosen `audit/infra/<family>/<symptom>` discriminator on the same footing as the named items. The behavioural predicate (raised inside the audit before successful record emission, prevents reaching assertions) is the sole gate on this expansion — implementer-side judgements about whether a given failure was "foreseen" are not part of the criterion, so uncaught exceptions thrown anywhere in the audit, host-process OOM the audit catches, file-handle exhaustion, and any other process-internal failure with the same behavioural shape are uniformly on this rule whether or not the implementer enumerated them in advance. A static-AST implementation has no checker-side resolve failure and no run-time bare-package import the audit relies on for SDK-side type resolution, so category (ii) is vacuously satisfied by such an implementation (the implementation-agnostic peer-package import failure of (i) — covering, e.g., a static-AST walker that nevertheless imports a peer package to read its `package.json` — is *not* in (ii) and remains live for both implementation choices); a type-aware implementation carries both. The TS230x routing carve-out — a module-import-resolution failure of the TS230x category above originating at the bare-package import the audit relies on routes through [step 2(a)](./version-bump-step2.md#bump-step-2-positive) below's surface-inventory update per the same rule step 5 names for the brand-string assertion's import-resolution failure — is meaningful only for type-aware implementations; in a static-AST implementation it has no surface to bind to and is a vacuous obligation rather than a missing one. The parse-/syntax-failure obligation in (i) is **not** duplicative with the project-wide `npm run typecheck` gate: both surfaces catch a TypeScript syntax error in `src/` (typecheck at typecheck time; the audit's walker at audit time), and a green `npm test` therefore requires both gates to be green; the audit gate is what guarantees the surface-set-closure property is checked against a parseable source tree even on a configuration where the typecheck script is mis-wired or temporarily skipped, and is the audit's own fail-closed wrapper rather than a delegation to the typecheck script. Each infrastructure failure MUST carry a stable discriminator distinct from the violation discriminators above (the same `audit/<class>/<family>/<symptom>` structural shape the *Failure-surface contract* pins, with the `<class>` segment fixed at the literal `infra` per the three-class partition there; cross-class disjointness against the five violation families above and the *Non-empty-scan canary* discriminators is structurally guaranteed by the `<class>` prefix, and the implementation MUST keep its `infra`-class `<family>` tokens within-class disjoint), and MUST emit one infrastructure-failure record per failure carrying the same `path`, `line`, `symbol` triple the *Failure-surface contract*'s violation records carry; `path`, `line`, and `symbol` MAY each independently be the literal `<n/a>` sentinel pinned by the *Wire serialisation* clause of the *Failure-surface contract* above when the failure precedes the resolution that field would otherwise carry — `path` is `<n/a>` when the failure precedes file-level resolution (e.g. a peer-package import failure or an audit-module load failure that never reaches a specific source file), `line` is `<n/a>` when the failure precedes line-level resolution within an otherwise-resolved file, and `symbol` is `<n/a>` when no named symbol applies to the failure (e.g. a parse failure on the file header). The four-field record shape and per-line wire serialisation pinned by the *Failure-surface contract* above apply unchanged to infrastructure-failure records; `proposed-resolution` on an infrastructure-failure record carries the implementation-chosen resolution arm a contributor would take to clear that infrastructure failure (e.g. a re-pin recipe per [step 2(a)](./version-bump-step2.md#bump-step-2-positive) for a TS230x module-import-resolution failure routed through the carve-out above), or the literal `<n/a>` sentinel when no contributor-actionable resolution is owed at the level the failure surfaces. *Partial-evaluation semantics.* An audit run that emits both real violations and infrastructure-failure records in the same invocation is permitted and is the expected behaviour when an infrastructure failure terminates the audit after some files have already been resolved: the run MUST emit every violation it has resolved before the infrastructure failure terminated it, then the infrastructure-failure record(s), then exit non-zero. The exit code is the single non-zero `npm test` exit and is not further subdivided — a contributor discriminates violation-only from violation-plus-infrastructure outcomes by reading the per-record `audit/<class>/<family>/<symptom>` discriminators on the audit's output (the `audit/<class>/...` two-segment prefix alone partitions records into the three classes), not by reading the exit code. *Wall-clock budget.* The audit MUST enforce a wall-clock budget on its own execution (the implementation pins the budget value, e.g. 60 s); on budget expiry the audit MUST emit one record under a stable `<symptom>` token in the infrastructure-failure family and exit non-zero. A hung audit that does not emit and does not exit falls under the *Pre-emission termination carve-out* below (the `npm test` red exit alone satisfies fail-closed, with the implementation pinning a `npm test`-side timeout to bound the overall run). *Pre-emission termination carve-out.* If the audit process is terminated before any record can be emitted — e.g. `SIGKILL` from the host OOM-killer, host shutdown, a walker panic that crashes the process mid-write, or a diagnostic-sink write failure on a torn pipe before the first complete record has been flushed — the non-zero `npm test` exit alone satisfies the fail-closed property; no record-format contract (the *Failure-surface contract*'s per-family record shape, the *Wire serialisation* per-line stdout obligation, or the per-discriminator routability obligation above) applies on this path, since by construction no record was emitted to which those obligations could bind. The contributor's recovery path on a `npm test` red with no `audit/<class>/<family>/<symptom>` line on the audit's stdout is to re-run with diagnostic-sink redirection (the implementation-chosen mechanism, e.g. capturing the audit's stderr or attaching to the host's process-exit logs) to obtain whatever signal the underlying termination left behind.
- <a id="audit-non-empty-scan-canary"></a>*Non-empty-scan canary.* The audit MUST emit, on every invocation, two counters — the number of audited source files actually walked, and the number of in-scope surface references (the union of categories (1), (2), and (3) under *Target surface categories* above, plus any reference recognised by the audit's prohibited-shape detection per *Recognised import/access shapes* above — including the family-(4) shapes that are by construction not category (1)/(2)/(3) references) actually *recognised* by the audit, where a reference is *recognised* iff it either (a) resolves against `SDK_SURFACE_INVENTORY`, (b) resolves against a typebox sibling allow-list entry (per *Target surface categories* typebox carve-out), (c) is authorised by a declared `// allow-pi-surface:` exemption, or (d) surfaces as a violation against any of the five families of the *Failure-surface contract* above (so an all-violation run on a fresh codebase recognises every reference and does **not** trip the canary spuriously alongside the violations) — and MUST assert both counters strictly greater than zero. A misconfigured walker that scans zero files in the audited source tree, or that walks files but recognises zero in-scope surface references (e.g. a glob root, `tsconfig` `include`, or AST-visitor selector that silently matches nothing — i.e. the second counter guards only against under-walking, not against a population of unauthorised references), MUST surface as `npm test` red rather than as a silent green; without this canary the *Failure-surface contract*'s violation-emission obligation, the *Infrastructure-failure handling* obligation above, and the surface-set-closure property [`spec.md` — Orientation — Prerequisites](../../spec/overview-and-orientation.md#orientation) leans on are all vacuously satisfied by a no-op audit. Each non-zero-floor failure MUST carry its own stable discriminator (the same `audit/<class>/<family>/<symptom>` structural shape the *Failure-surface contract* pins, with the `<class>` segment fixed at the literal `canary` per the three-class partition there; cross-class disjointness against the five violation families above and the *Infrastructure-failure handling* discriminators is structurally guaranteed by the `<class>` prefix, and the implementation MUST keep its `canary`-class `<family>` tokens within-class disjoint) so a contributor can route a no-op-walker breakage separately from both a real inventory miss and an infrastructure-side breakage. *Green-path emission contract.* On every invocation — green or canary-red — the audit MUST emit exactly one canary record on stdout in the line-delimited four-field wire format the *Wire serialisation* clause of the *Failure-surface contract* above pins, leading with the canary's stable `audit/<class>/<family>/<symptom>` discriminator token (with `<class>` the literal `canary`; a green-path run MAY use a leaf-chosen `<symptom>` token disjoint from the canary-red `<symptom>` token under the same canary `<family>`, so a CI parser can distinguish green-path from canary-red records by the leading-token prefix without needing to parse the `proposed-resolution` field). The `proposed-resolution` field MUST carry the two counter values as the plain-ASCII key=value pair set `walked=<N> recognised=<M>` (or an equivalent implementation-chosen plain-ASCII key=value pair set whose pair names identify the same two counters this paragraph defines — the audited-files-walked counter and the in-scope-references-recognised counter, in that order), so a CI parser may inspect the audit's stdout to confirm both counters were observed on a green build. On a green run (both counters above their floors and no canary-red), `path`, `line`, and `symbol` MUST each be the literal `<n/a>` sentinel pinned by the *Wire serialisation* clause above. On a canary-red run, `path` / `line` / `symbol` identify the misconfiguration site if the audit can resolve it (e.g. the offending glob-root or `tsconfig` `include` source line); when the audit cannot resolve the misconfiguration site at the level the field would otherwise carry, the field carries the literal `<n/a>` sentinel per the same per-field rule the *Infrastructure-failure handling* sub-paragraph above pins. CI parsers MAY assert that exactly one canary record appears per invocation. The implementation MUST NOT emit zero canary records on any invocation (including a green run), MUST NOT emit more than one canary record on any invocation, and MUST NOT route the canary record through any sink other than stdout (mirroring the *Wire serialisation* sole-machine-parseable-surface rule above). *Infra-aborted-run carve-out.* If the audit terminates due to an infrastructure failure (per *Infrastructure-failure handling* above) before reaching the canary's two-counter computation, the "exactly one canary record per invocation" obligation (and the "zero canary records" / "more than one canary record" prohibitions immediately above) do NOT apply: the run is permitted to emit zero or one canary records depending on how far the audit progressed before terminating, and the non-zero `npm test` exit (per the *Pre-emission termination carve-out* and the *Wall-clock budget* clauses of *Infrastructure-failure handling* above) is the fail-closed surface. CI parsers asserting on the canary obligation MUST exclude infra-aborted runs (identifiable by the presence of one or more infrastructure-failure records — records whose leading-token's `<class>` segment is the literal `infra` per the three-class partition pinned in the *Three-class partition* clause of the *Failure-surface contract* above — in the same stdout stream). *Canary-red contributor-facing resolution arm.* A canary-red record names a no-op-walker breakage rather than a member of the five violation families, so the family→step-2(b) routing table the *Failure-surface contract* above pins is **not** the routing surface for it; instead, a contributor seeing a canary-red record MUST resolve it by widening the audit's scope configuration (the audit's glob root, `tsconfig` `include`, AST-visitor selector, or equivalent walker-input pin under the audit implementation) until both counters clear their floors on the next `npm test` invocation, and a reviewer MUST check that fit on the canary-red-resolution diff. This arm is owned by PIC here rather than by step 2(b); the family→step-2(b) routing table remains restricted to the five violation families. *Order of evaluation against violation records.* The canary's counter-emission and assertion MAY co-emit with the audit's violation records on the same invocation: a run that emits both real violations and a canary-red record is permitted (and is the expected behaviour when the second counter trips on a glob-root miss whose narrowed scope nevertheless captured one or more in-scope but out-of-inventory references on its way to the floor breach), and a canary-red record MUST NOT suppress, defer, or otherwise demote any violation record the same run would otherwise emit. The exit code is the single non-zero `npm test` exit and is not further subdivided in loom 1.0 (mirroring the *Partial-evaluation semantics* clause of the *Infrastructure-failure handling* rule above) — a contributor discriminates violation-only from violation-plus-canary-red outcomes by reading the per-record `audit/<class>/<family>/<symptom>` discriminators on the audit's output (the `audit/<class>/...` two-segment prefix alone partitions records into the three classes), not by reading the exit code. The implementation owns the literal `<family>` and `<symptom>` strings and the failure-message wording, and pins them into permanent test invariants. The per-counter floor MUST be `> 0` for both counters; the implementation MAY pin a higher floor as a permanent test invariant if it has a baseline-snapshot mechanism (so the higher floor is mechanically tied to a recorded baseline rather than to a justification a reviewer would have to weigh), but PIC pins only the `> 0` floor here. PIC pins the two-counter requirement, the strictly-greater-than-zero floor, the disjoint-discriminator obligation, the green-path emission contract (the once-per-invocation canary record, its four-field wire shape, the `walked=<N> recognised=<M>` `proposed-resolution` pair set or its implementation-chosen equivalent, and the `<n/a>`-sentinel rule for `path` / `line` / `symbol` on a green run), the canary-red contributor-facing resolution arm, and the order-of-evaluation rule against violation records here.

The audit runs as part of the SDK surface-inventory tests step 2 below names (specifically, it is step 2(b)); `npm test` is the entry harness PIC pins as the test-side gate (the `npm test` script's existence, its concrete command, and its harness wiring — including any Vitest-style or equivalent test-runner wrapping — are implementation choices), and PIC does not pin a separate `npm` script for the audit. The implementation MUST wire the audit so that **both** an audit violation record (per the *Failure-surface contract* above) **and** an infrastructure-failure record (per the *Infrastructure-failure handling* sub-paragraph above) cause the `npm test` invocation to exit non-zero — irrespective of whether the audit runs in-process to the harness (e.g. as a Vitest test case whose `expect(...).toBe(0)` over the audit's exit code becomes the test's pass/fail) or as a child process the harness spawns and whose own non-zero exit code becomes the test's pass/fail; either wiring is permitted as long as the fail-closed property holds end-to-end. The implementation MUST wire the audit so that an audit child process whose exit code is non-zero AND that emits zero records (including process-killed and uncaught-exception terminations, per the *Pre-emission termination carve-out* above) causes `npm test` to exit non-zero. For the in-process variant, the wrapping `expect(...)` MUST be unconditionally registered before the audit invocation so a process kill aborts the harness and propagates non-zero. An implementation wiring that interprets empty stdout as 'no failures' violates this obligation. The audit's record emission MUST reach a sink whose contents survive the harness wrapping (stdout / stderr per the *Wire serialisation* rule above — the audit MUST NOT route records through a harness-internal API, an in-memory test reporter slot, or any other surface a CI parser cannot consume by reading process stdout / stderr), so that a CI parser can still consume the `audit/<class>/<family>/<symptom>` discriminator tokens on either wiring choice. The implementation MUST adopt one concrete audit implementation and one concrete `npm test` wiring so this contract has a single concrete adoption; an implementation whose `npm test` invocation does not exit non-zero on either record class above, or whose failure output does not satisfy the per-discriminator routability obligation above, or whose record emission does not survive the harness wrapping per the *Wire serialisation* rule, violates this section.

Item 5 (cancellation propagation) is verified jointly by Step 0 item (b)'s `AbortSignal` / `AbortController` shape probe (the two Pi-side entry-point signals) and Step 0 item (c)'s `AgentSession.prototype.abort` member check (the subagent-mode primitive). Item 7 (binder LLM model) is verified at per-loom load time and surfaces as `loom/load/binder-model-unresolved` per [Slash-Command Argument Binding — Strict-capability requirement](../binder/binder-model-and-context.md#strict-capability-requirement); it is **not** factory-probable and the two diagnostic surfaces (`loom/load/host-incompatible` and `loom/load/binder-model-unresolved`) are disjoint.
