# Version bump steps 5–7

5. **Update the capability-probe pinned constants.** *Bundled-concerns orientation:* this step covers (1) the runtime constants update, (2) the build-time type-equality assertion shape, and (3) the per-sub-trigger routing semantics; concerns (2) and (3) are implementation-owned for their literal strings, discriminator names, and fixture inventory (PIC pins only the assertion's required structural shape under (2) and the routability obligation under (3) — see the *PIC owns the routing semantics* sentence later in this step), so this step's contributor-facing obligation is concern (1) plus verifying that concerns (2) and (3) still hold via the SDK surface-inventory tests passing on the bumped pin. Four logical groupings live in the single source-of-truth pinned-constants block in the extension module: (i) the four constants the **Step 0 (Capability probe)** above consumes at runtime (Node floor, `AbortSignal` member list, capability list, peer-dep range), (ii) the `SessionShutdownEvent['reason']` closed-set snapshot entry in the pinned-constants block, whose literal-array field is consumed at runtime by the [Unknown-reason rule](./unknown-reason-rule.md#unknown-reason-rule) under **Extension entry point** sub-step 4 above (the runtime's set-membership check reads that field) and whose element type is compared at build time by the bidirectional `SessionShutdownEvent['reason']` type-equality assertion this step gates (which runs under `npm run typecheck` per step 1, not under step 2(a)'s `npm test` literal-read; the two assertions live in the same surface-inventory test file but have different gates and different failure-routing per the brand-string carve-out in step 1), and (iii) the *two typebox allow-lists* (the *named-import allow-list* `{ Type }` and the *member-access allow-list* `{ Unsafe }`, per *Target surface categories* under [Inventory-closure audit](./inventory-audit-intro.md#sdk-cap-inventory-closure-audit) above), whose entries are consumed at build time by the inventory-closure audit — the named-import allow-list by the audit's category (2) resolver for `typebox` named imports (per the *typebox-only sibling* paragraph above), and the member-access allow-list by the audit's category (2) check on `Type.<member>` accesses — and whose member-access entry `Unsafe`'s presence on the imported `Type` namespace is additionally consumed at runtime by the [Step 0 (e)](./capability-probe.md#entry-capability-probe) host-shape probe; a Pi/`typebox` bump that renames `Type`, renames `Unsafe`, or otherwise drifts the permitted symbol names on either the `typebox` package's named-export surface or the `Type` namespace requires updating the affected allow-list in the same commit as the rest of the bump, on the same footing as the four Step-0-probe constants under (i), and (iv) the `CAPABILITY_OBLIGATIONS` constant per [step 2(a)](./version-bump-step2.md#bump-step-2-positive) above (the seven-element array of capability-anchor IDs the literal-read count-equality assertion consumes; this constant is co-edited on every capability-add or capability-remove bump per [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote) above's catch-all). Step 5 is in scope for groupings (i)–(iii); grouping (iv) is enumerated here for completeness but is gated by [step 2(a)](./version-bump-step2.md#bump-step-2-positive)'s `CAPABILITY_OBLIGATIONS.length === 7` count-equality assertion (and co-edited via [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote)'s catch-all on capability-add/remove bumps), not by any step-5-side trigger or fixture. The same block also holds non-probe entries — for example `SHUTDOWN_AWAIT_CAP_MS` per **Extension entry point** sub-step 4 above, and the additional snapshot entry kinds the implementation owns — which are out of scope for this step and have no enumerated count on this page (the implementation owns the entry-level inventory). The build-time `SessionShutdownEvent['reason']` type-equality assertion this step gates is bidirectional (so a Pi patch that either widens or narrows the union fails `npm run typecheck` rather than reaching user runtime), and MUST surface the literal brand string `loom/typecheck/session-shutdown-reason-snapshot` verbatim in the failing `tsc` diagnostic. The verbatim, grep-able brand-string surfacing this assertion relies on is pinned to a confirmed TypeScript floor at [Host prerequisites — Build-toolchain floor](./host-prerequisites.md#tsc-toolchain-floor); a TypeScript bump past that floor is re-confirmed by editorial review under this procedure. The exact source-code form of the assertion is an implementation choice; PIC pins only the verbatim-brand property above and does not prescribe a particular TypeScript construct, and any equivalent form is permitted as long as the verbatim-brand property holds. *Non-normative example:* a string-literal-type-annotated `const` binding whose type annotation is the brand string and whose initialiser resolves to the brand-string literal when the bidirectional comparison succeeds and to `never` otherwise satisfies the verbatim-brand property by surfacing on widen-or-narrow as a `tsc` diagnostic that quotes the brand string in the type-mismatch message; this is illustrative and any other construction with the same verbatim-brand surfacing behaviour is equally acceptable. The implementation MUST adopt one concrete form so this contract has a single concrete adoption; an implementation whose failing `tsc` diagnostic does not carry the brand string verbatim violates this step. Named-alias mechanisms (TypeScript identifiers cannot contain the `/` character the brand string requires) cannot embed the brand and MUST NOT be used as the surfacing mechanism — a `type` alias, an `interface` name, a declared `function` name, a `const` binding *name*, or any other TypeScript identifier would prevent `tsc` from surfacing the brand verbatim, and the step-1 carve-out, the step-5 trigger (ii) discriminator, and the [Patch-skew degradation contract](./patch-skew-degradation.md#patch-skew-degradation-contract)'s build-time gate all key on the brand appearing grep-ably in `tsc` output and would silently fail to fire if the embedding were demoted to a named alias. The assertion MUST be reached by the `npm run typecheck` invocation step 1 above names — whether the implementation places its source file under `test/` (alongside the surface-inventory test) or under `src/extension/` (a separate assertion-only module co-located with the pinned-constants block, which when placed under `src/extension/` MUST be a dedicated file matching the suffix convention `src/extension/**/*.assert.ts` pinned by *Audit scope* above — so the inventory-closure audit can identify and exclude it syntactically; inlining the assertion into a non-suffix-matching `src/extension/**/*.ts` file is **not** an *equivalent form* under this step's permission, since the assertion belongs in a dedicated module so its compile-time-only role (typecheck-time bidirectional `SessionShutdownEvent['reason']` brand-string assertion) is structurally separable from any runtime code under `src/extension/`. Inlining would couple the typecheck-only assertion to runtime code paths that share the file, complicating both `tsconfig` configuration and the separation-of-concerns invariant. The audit's authorisation of `Type.Unsafe` (per typebox carve-out) is unrelated to this prohibition.), the loom `tsconfig` MUST include it, and a split-tsconfig layout that excludes it is a violation of this step rather than a silent demotion to the runtime-only `loom/host/session-shutdown-reason-unknown` fallback. The assertion imports `SessionShutdownEvent` from the bare-package specifier `@earendil-works/pi-coding-agent` (the package's `package.json#exports`'s `"."` entry, resolving to `dist/index.d.ts`, re-exports `SessionShutdownEvent` from its declaration site `dist/core/extensions/types.d.ts` at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin); the deep-path citations elsewhere on this page and in [`spec.md` — Session model — SM-2 closed shutdown reason set](../../spec/session-model-and-appendix.md#sm-2-closed-shutdown-reason-set) name that declaration site, not an import specifier). A Pi patch that drops or renames `SessionShutdownEvent` on the package-root re-export surface (so the import resolves to a module-import-resolution failure — `TS2305` (`"Module '<path>' has no exported member '<name>'."`), `TS2724` (the "did you mean Y?" rename variant), `TS2307` (module not found), `TS2614` (default-vs-named-import mismatch), or any other TS code in the same module-import-resolution family — rather than to a value/type comparison) is **not** covered by this step's brand-string carve-out — every TS code in that family is routed instead through [step 2(a)](./version-bump-step2.md#bump-step-2-positive)'s surface-inventory update, which re-pins the import specifier (per the surface-inventory description) to whatever sub-export (or deep `dist/...` path, as a last resort) the candidate Pi minor exposes the type on, in the same edit as the rest of the bump. Routing keys on the TS error code, not on the message-text quote — the message-text quote above is illustrative of the TS2305 wording at the loom 1.0 Pi-SDK pin and may drift across TypeScript versions without affecting the routing contract. Update the constant in the same commit if either <a id="bump-step-5-trigger-i"></a>**(i)** steps 1–4 have changed any of the four Step-0-probe constants (Node floor, `AbortSignal` member list, capability list, peer-dep range) or either of the two typebox allow-lists (the named-import allow-list `{ Type }` or the member-access allow-list `{ Unsafe }`, per grouping (iii) above), or <a id="bump-step-5-trigger-ii"></a>**(ii)** either the build-time `SessionShutdownEvent['reason']` type-equality assertion above fails `npm run typecheck` against the candidate Pi version (identified mechanically by the literal brand string `loom/typecheck/session-shutdown-reason-snapshot` appearing in the failing `tsc` diagnostic; a same-invocation typecheck red that does not carry the brand string is owned by step 1 above, not by this step; the carve-out applies per-assertion, not per-invocation), or step 2(a)'s (the SDK surface-inventory literal-read assertion's) *literal-array consistency check* on the `SessionShutdownEvent['reason']` closed-set snapshot entry fails — the consistency check's predicate is that the snapshot entry's `literals` field is a non-empty array of distinct strings, every member of which appears in the SDK union per the type-equality assertion's element-type comparison, and step 2(a) (the SDK surface-inventory literal-read assertion) is the runner that evaluates this predicate against the imported pinned-constants block on each `npm test` invocation. Both sub-triggers under (ii) concern the same snapshot entry's correctness and route to the same fix (a snapshot-edit-only sweep on the `SessionShutdownEvent['reason']` snapshot entry's `literals` field). The two sub-triggers above MUST surface as orthogonal, individually-discriminable failures so a contributor cannot silently misroute a fix to the wrong sub-trigger by editing a compensating constant in the other half. PIC owns the routing semantics: each failure-output entry the implementation emits MUST be unambiguously routable by a reader to exactly one of trigger [(i)](#bump-step-5-trigger-i) or trigger [(ii)](#bump-step-5-trigger-ii) without an out-of-band translation table — and the implementation MUST expose a stable test-output discriminator on each such failure that satisfies that routability obligation. The implementation owns the literal strings — the exact discriminator names it emits in test-failure output, the failure-message wording, and the negative-test fixtures (one per Step-0-probe-constant failing entry under grouping (i) of trigger (i), one per typebox-allow-list entry sub-trigger under grouping (iii) of trigger (i) (across both the named-import allow-list and the member-access allow-list), and one per `SessionShutdownEvent['reason']` snapshot-entry sub-trigger under trigger (ii)) — and pins them into permanent test invariants; PIC does not pin the literal strings. The brand string above remains the identifier for the type-equality-assertion sub-trigger of trigger (ii) because it is emitted by `tsc` rather than by the implementation; every other discriminator on either trigger is implementation-owned. Step 2(a) (the SDK surface-inventory literal-read assertion) imports the pinned-constants block and will fail until the change is consistent with the rest of the bump. A failure of the bidirectional type-equality assertion against `SessionShutdownEvent['reason']` — caused by a Pi patch that widens or narrows that union, so the literal-array field of the `SessionShutdownEvent['reason']` closed-set snapshot entry in the pinned-constants block no longer matches the SDK type — surfaces at `npm run typecheck` via the bidirectional type-equality assertion above; resolve by updating that entry's literal-array field in the pinned-constants block to match the new SDK union in the same edit. *Inbound-reference sweep for session-only-reason subsets.* The same edit MUST also sweep every inbound runtime reference to a session-only subset of `SessionShutdownEvent['reason']` — currently the inline `{"new", "resume", "fork"}` triplet at the `loom/host/session-shutdown-runtime-degraded` emission site per the [*Edge cases* `reason: "new" | "resume" | "fork"` bullet](./session-only-degraded-state.md#session-only-reason-degraded-state) under **Extension entry point** sub-step 4 above (whose source-of-truth pin that bullet owns) — so a session-only literal rename (e.g. `"fork" → "branch"`) updates the inline triplet in lock-step with the snapshot edit and the renamed literal does not silently no-op its `session-shutdown-runtime-degraded` emission. *Enumeration mechanism — grep target.* The single-site triplet enumeration above is a **snapshot at last-bump-time**, not a standing guarantee that no other inbound runtime reference exists; the sweep MUST be backed by a concrete enumeration pass the contributor runs on each bump rather than by contributor recall of the documented site alone. Concretely, with the candidate-minor-edit branch checked out, the contributor MUST run a grep across the layout-invariant superset `src/`, `test/`, and `docs/` (chosen to subsume any future inbound runtime reference added under top-level `docs/` or under any `src/` or `test/` subdirectory not yet present at last-bump-time, so a sibling under `docs/` that is not `spec_topics/` — for example `docs/spec.md`'s Session model paragraph, which carries the same triplet and complement literals — is not silently missed; the second-stage `event.reason|SessionShutdownEvent|capturedEventReason|entry.shutdownReason` filter below controls the false-positive rate this widening admits) for string-literal occurrences of any of `"new"`, `"resume"`, or `"fork"` co-located on the same source line as either `event.reason` or `SessionShutdownEvent` (non-normative recipe: `rg -nP '"(new|resume|fork)"' src/ test/ docs/ | rg -P 'event\.reason|SessionShutdownEvent|capturedEventReason|entry\.shutdownReason'`; any equivalent two-stage grep that surfaces line-level co-occurrences of one triplet member with one of the listed anchors satisfies the obligation, and a future swap to a different grep tool or a typed AST search is permitted as long as the same line-level co-occurrence set is surfaced. The second-stage anchor set above is the closed enumeration of every runtime-side reference shape the loom 1.0 spec uses to read or stamp a `SessionShutdownEvent['reason']` value at the inbound sites this sweep is designed to catch — the direct property read `event.reason`, the type identifier `SessionShutdownEvent`, the handler-scoped capture `capturedEventReason` (the name pinned by the *Predicate split* clause's broader tag-transition predicate `capturedEventReason ∉ {"quit", "reload"}` at the [`LoomRegistry.markRuntimeDegraded`](./drain-state-contract.md#loom-registry-drain-state-contract) call site, which is the canonical site this sweep is designed to catch on the `runtime-degraded` emission row), and the per-entry stamped field `entry.shutdownReason` (the field the `ActiveInvocationRegistry` entry carries per sub-step 2 above)), and reconcile every hit against the documented enumeration in the same edit: a hit that corresponds to the existing inline-triplet site above is a no-op; a hit at an illustrative restatement that [step 1](./version-bump-intro.md#pi-version-bump-procedure)'s carve-out classifies as transiently stale with no runtime-correctness impact — for instance the `docs/spec.md` SM-2 listing at [`spec.md` — Session model — SM-2 closed shutdown reason set](../../spec/session-model-and-appendix.md#sm-2-closed-shutdown-reason-set) — is likewise a no-op and is excluded from the new-inbound-runtime-reference arm below; a hit that does not corresponds to a new inbound runtime reference whose enumeration MUST be added to this paragraph and whose triplet update MUST be applied at the newly-discovered site on the same footing as the existing inline-triplet site, in the same edit. A positive result-delta against the documented enumeration that is not reconciled in the same edit is non-conformant. Any future runtime-side reference shape the spec adds — a new handler-scoped capture name, a new per-entry stamped field, or any other inbound site that reads or stamps a `SessionShutdownEvent['reason']` value without re-reading `event.reason` — MUST extend this second-stage anchor set in the same edit that introduces the new shape, on the same footing as the documented enumeration update obligation above; an inbound runtime reference whose anchor name is absent from the filter would be silently elided by the sweep on the next bump and is non-conformant. The closed `{"quit", "reload"}` complement is the partition owned by Pi's `session_shutdown` type-doc teardown-vs-session-swap distinction (loom 1.0 carries no other inbound runtime reference to a `{"quit", "reload"}` literal subset; should one be added in a future edit it MUST be enumerated here on the same footing as the session-only triplet); the symmetric grep with `"(quit|reload)"` in place of `"(new|resume|fork)"` MUST also be run on each bump to audit that complement-set claim, and any hit MUST be enumerated here on the same footing as the future-edit obligation in the parenthesis above. A rename of `"quit"` or `"reload"` itself is caught by the same brand-string-carrying type-equality red and is handled by re-checking that the renamed literal still belongs to the always-tear-down half of the partition at the inline-triplet site in the same edit. The implementation MAY additionally implement a build-time audit pass over the same three grep targets so a result-delta against the documented enumeration fails `npm test` rather than relying on contributor-side discipline; PIC does not require the audit, and an implementation that adopts it owns the failure-discriminator names and message wording on the same footing as the other implementation-owned literal strings under groupings (i)–(iii) above. The user-facing behaviour during the window between a Pi patch landing on a user's machine and this snapshot edit shipping — for both widening and narrowing patches, and the (lack of) automated cadence that closes the window — is owned by the [Patch-skew degradation contract](./patch-skew-degradation.md#patch-skew-degradation-contract) above; this step does not restate it. <a id="bump-step-5-section7-reason-row"></a>*§7 closed-enum `<reason>` row co-update (convention).* A trigger [(ii)](#bump-step-5-trigger-ii) snapshot edit that widens or narrows the `SessionShutdownEvent['reason']` union also changes the GOV-7 / GOV-8 rendering surface for the §7 `<reason>` closed-enum value table declared at [Diagnostics — §7 Identifier-, descriptor-, and closed-enum placeholders](../diagnostics/placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders); the contributor SHOULD update that table's `<reason>` enumeration in the same commit as the snapshot edit. This co-update is a contributor-discipline convention, not a mechanically-enforced co-edit: the snapshot entry's `literals` field is gated by trigger (ii)'s `loom/typecheck/session-shutdown-reason-snapshot` type-equality assertion and step 2(a)'s literal-array consistency check, but no build-time gate fires when the §7 `<reason>` row is left unedited, so a §7 row left transiently stale carries no runtime-correctness impact (the closed-arm `<reason>` renderer emits the runtime `event.reason` value, not the spec table) and is reconciled by the next page-touching editorial sweep on the same footing as the five in-prose illustrative restatements named in [step 1](./version-bump-intro.md#pi-version-bump-procedure).
6. **Re-validate the provider seed-field table.** Run the provider seed-field fixtures against the candidate `@earendil-works/pi-ai` minor. If any supported `Api` value's seed field has been renamed, retyped, or moved between the supporting and non-supporting sets, update the table in [Provider seed-field mapping](./provider-error-mapping.md#provider-seed-field-mapping) and the corresponding fixture inputs in the same commit. If the candidate minor introduces a new `Api` value not yet listed, add a row (with `omitted` as the conservative default unless pi-ai documents a seed field for that provider) and a corresponding fixture in the same commit. The build-time `Api`-coverage assertion is the mechanical gate that fails until this step has been completed. When adding a row, the contributor MUST consult pi-ai release notes and the new provider's request-adapter source to decide between a named field and `omitted` — do not silently regress a provider for which pi-ai has added seed support.
7. **Update the binder strict-capability probe.** The binder strict-capability check (per [Slash-Command Argument Binding — Strict-capability requirement](../binder/binder-model-and-context.md#strict-capability-requirement)) reads `Model<Api>.strictCapable` by name (`Model` and its `Api` type parameter are pinned at [model-registry surface](./host-interfaces-core.md#model-registry-pin) above). If the candidate `@earendil-works/pi-coding-agent` minor introduces the indicator under a different name, rename the probe constant in the same edit; the SDK surface-inventory assertion that catches "field present on `Model<Api>` namespace but not under the probed name" is the mechanical gate that fails until this step has been completed. If pi-ai exposes the field under the probed name `strictCapable`, this step is a no-op.

The procedure produces no separate artefact file — its outputs are (a) a green build-time test run that proves the SDK surface, the `peerDependencies` literals, and the `engines.node` floor are consistent at the new pin, and (b) a single bump commit that touches the loom `package.json`, the constant block in the extension module, the spec sentences a bump touches (the single-source-of-truth Pi-SDK pin literal at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) site, edited on a version bump, plus any surface-description sentence that [step 1](./version-bump-intro.md#pi-version-bump-procedure) above requires updating when its underlying SDK shape changed), and any inventory bullet whose underlying SDK shape changed. A bump commit that updates `peerDependencies` without a matching update to the constant block, the spec sentences, or the inventory bullets fails the build-time tests and MUST be amended before merge.
