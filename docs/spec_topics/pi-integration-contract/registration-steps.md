# Registration steps

1. Registers the `--loom` CLI flag via `pi.registerFlag('loom', { type: 'string', description: … })` **before** subscribing to `resources_discover`, so the value is readable from inside the discovery handler via `pi.getFlag('loom')`. The flag name `'loom'`, `type: 'string'`, the presence of a `description`, and the registration ordering relative to `resources_discover` are the observable pins; the exact `description` string is non-normative — Pi renders it only in `--help`, where it keys no loom surface — and `'Loom file or directory paths, joined with the OS path-list separator'` is an illustrative example. Walks the five discovery sources defined in [Directory Convention](../discovery.md) directly — the global directory `~/.pi/agent/looms/`, the project directory `.pi/looms/`, every installed package's `pi.looms` entry or conventional `looms/` directory (per [Package discovery](../discovery/package-and-settings.md#package-discovery)), the `loomPaths` settings array (per [Settings file reads](../discovery/package-and-settings.md#settings-file-reads)), and the `--loom` CLI flag (whose `pi.getFlag('loom')` value is split on `path.delimiter` per [Directory Convention](../discovery.md)) — at factory time and again on every `resources_discover` event. The event payload (`{ cwd, reason }`) supplies the project root used for the project-local source (the factory-time scan resolves the project root from the [`FileSystem.cwd()` seam member](./host-interfaces-services.md#pic-13); on every `resources_discover` event `event.cwd` is preferred over that seam value, so a future per-session cwd change is honoured on reload) and distinguishes initial scan (`reason: "startup"`) from reload (`reason: "reload"`); Pi delivers no paths to the handler — `ResourcesDiscoverResult` has no `loomPaths` slot — and the `pi` manifest namespace does not enumerate `pi.looms`. The loom extension owns the walk for all five sources; the event is purely a re-trigger, and the handler returns the empty result `{}` to satisfy the typed return.
2. Parses each discovered `.loom` file and builds a *pending registration list* keyed by slash name. The factory does **not** call `pi.registerCommand` synchronously — Pi's prompt templates and other extensions' commands are not yet enumerable at factory time, so cross-format collision detection cannot run here. Concretely, on the pre-bind runtime returned to the extension factory by `createExtensionRuntime()`, every action method that reads or mutates session-bound state — including `getCommands`, `getActiveTools`, `getAllTools`, `setActiveTools`, `sendMessage`, `sendUserMessage`, `setModel`, `getThinkingLevel`, `setThinkingLevel`, and the rest of the `notInitialized`-tagged slots in `@earendil-works/pi-coding-agent`'s `dist/core/extensions/loader.js` — is bound to a throwing closure (`Error("Extension runtime not initialized. Action methods cannot be called during extension loading.")`) that `Runner.bindCore()` later overwrites with the real implementation alongside `session_start`. The token `notInitialized` is the internal variable name of that throwing closure inside Pi's loader source; it is **not** an observable sentinel value an extension can compare against or switch on. Calling any of these methods from inside the factory raises an unhandled exception that aborts factory load. <a id="pre-bind-throw-closure-evidence"></a>The `dist/core/extensions/loader.js` path, the `notInitialized` internal variable name, the private `Runner.bindCore()` overwrite step, and the verbatim throw-string text above are **known-fragile evidence** at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) — diffable artefacts witnessing that the pre-bind action-method slots throw at the pin, not load-bearing claims about their stable shape across patches (a Pi patch may rename the `_`-prefixed loader internals, re-bundle the `dist/` tree, or reword the throw string). The observable rules in this step — that calling a session-state action method pre-bind aborts factory load, while the registration sinks and `refreshTools` are safe before bind — hold independent of that evidence trail, which is re-audited on each Pi minor bump via [item (q) of the *Editorial-review checklist for unpinned host presuppositions*](./version-bump-step2.md#bump-checklist-pre-bind-and-renderer-resolution) under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. The methods Pi documents as safe before bind are `pi.registerCommand` / `pi.registerTool` / `pi.registerFlag` / `pi.registerMessageRenderer` (registration sinks) and `refreshTools` (a no-op on the pre-bind runtime). Every other reach for session state is deferred to a `session_start` (or later) handler. The signature pinned by loom 1.0 is `pi.getCommands(): SlashCommandInfo[]` (synchronous, returns the current registered-commands snapshot, where `SlashCommandInfo` is the interface exported from `@earendil-works/pi-coding-agent`'s `core/slash-commands.d.ts` with fields `{ name, description?, source: SlashCommandSource, sourceInfo }`); the returned array is mutable in the type system but MUST be treated as read-only by convention (the loom 1.0 collision check is a single forward pass and never mutates it); the snapshot is only meaningful on or after `session_start`. The build-time SDK surface inventory (per *Host prerequisites*) MUST list `SlashCommandInfo` and the three-arm `SlashCommandSource = "extension" | "prompt" | "skill"` explicitly so a Pi minor that renames either fails the build.
3. Subscribes a `session_start` handler that consults `pi.getCommands()`, drops any pending loom whose slash name collides with an existing entry whose `source` is in the [loom 1.0 cross-format collision source set](./host-interfaces-services.md#v1-seam-pi-owned-subagents-collision-source-set) defined below — the three arms of Pi's `SlashCommandSource` union (`"prompt"`, `"extension"`, `"skill"`), per `@earendil-works/pi-coding-agent`'s `core/slash-commands.d.ts` and per [Discovery — Slash-name collisions across formats](../discovery.md) — <a id="slash-handler-registration"></a>registers each survivor via `pi.registerCommand(name, { description, handler })` — one slash command per loom — and emits a `loom/load/cross-format-collision` diagnostic for each drop through the channel defined in [Diagnostics](../diagnostics.md). The `description` field passed to `pi.registerCommand` carries `frontmatter.description` verbatim and only that — `argument-hint` is **not** concatenated, since Pi's extension-command autocomplete UI has no `argumentHint` slot and the binder is the only loom 1.0 consumer of `argument-hint` (see [Slash-Command Argument Binding](../binder.md)). loom 1.0 passes only the two keys `description` and `handler`; the optional `getArgumentCompletions` key is omitted, since Pi's `RegisteredCommand` interface types it as optional and loom 1.0 exposes no autocomplete-completion surface (a deferred `argumentHint`-style upstream is tracked in [Future Considerations](../future-considerations/surface-extensions.md#surface-extensions-without-a-dedicated-topic-page-seam)). The same handler also de-registers any previously-registered loom whose name now collides with a newly-appeared `.md` template or extension command (e.g. after a settings reload or another extension's activation), emitting the same diagnostic. *Drain-state-gated dispatch.* The `handler` value passed as the `handler` field of every `pi.registerCommand(name, { description, handler })` call above MUST, on every slash dispatch, invoke `LoomRegistry.readDrainState()` as its first action and route through the closed three-arm enumeration (a)/(b)/(c) pinned by the **`LoomRegistry` drain-state contract**'s *Methods* paragraph below — including that paragraph's *Read-failure fallback* clause, which pins arm (c) (the `"loom /<name>: extension degraded; /reload to recover"` system note) as the conservative fail-safe when the `readDrainState` call itself throws. This obligation is normative at this registration site; the arm enumeration and the read-failure routing are owned by the *Methods* paragraph below and are not restated here.

   <a id="getcommands-completeness-presupposition"></a>Step 3's first `session_start` collision pass additionally relies on a Pi *behavioural* precondition the `SlashCommandInfo[]` snapshot does not encode: that by the time Pi emits the first `session_start`, `pi.getCommands()` already enumerates every command Pi registers from prompt templates and skills, and every command a sibling extension registers either at factory time or from a `session_start` handler Pi load-orders before loom's. Pi controls extension load order and emits each extension's `session_start` in that order, so a sibling loaded after loom that first registers a colliding command inside its own `session_start` handler is absent from loom's first snapshot; the first-pass check passes and Pi disambiguates the duplicate with its own numeric-suffix form (`<name>:1` / `<name>:2`) — the observable surface of a first-pass miss — rather than loom dropping the colliding loom and emitting `loom/load/cross-format-collision`. The next-cycle re-evaluation of such later-appearing siblings — the de-registration of a previously-registered loom whose name now collides with a newly-appeared `.md` template or extension command — is owned by [Discovery — DISC-4](../discovery/discovery-sources.md#disc-4) and the de-registration clause of this step above; this presupposition does not restate that recovery. The precondition is taken on a best-effort basis against the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) above and is re-audited on each Pi minor bump via item (l) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect a load-order or population-timing change from the type surface alone.
4. Subscribes a `session_shutdown` handler (`pi.on("session_shutdown", handler)`) that owns disposing every resource the extension instance holds. The Pi SDK documents `session_shutdown` as fired before an extension runtime is torn down due to quit, reload, or session replacement, with `event.reason: "quit" | "reload" | "new" | "resume" | "fork"`; loom uses it as the single teardown contract for the outgoing extension instance so that Pi's subsequent `ExtensionRuntime.invalidate(...)` call does not strand subagent provider connections, chokidar file handles, debounce timer handles, or `AbortController` listeners across `/reload`, `/new`, fork, or quit.
