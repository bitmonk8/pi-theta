# Registration steps

1. Registers the `--loom` CLI flag via `pi.registerFlag('loom', { type: 'string', description: … })` **before** subscribing to `resources_discover`, so the value is readable from inside the discovery handler via `pi.getFlag('loom')`. `pi.getFlag` is declared on `ExtensionAPI` as `getFlag(name): boolean | string | undefined` at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (non-capability status pinned at [Non-capability `pi.<member>` surfaces](./inventory-audit-intro.md#non-capability-pi-members)); because loom registers `'loom'` with `type: 'string'` and no `default`, loom's own read narrows to `string | undefined`, yielding `undefined` when no `--loom` flag was passed on the CLI. An `undefined` (absent-flag) read contributes no `--loom` paths to the walk; the `path.delimiter` split below applies only to a present string value. The `'loom'` flag name's cross-extension namespace clearance — and the first-load-wins behaviour of `pi.registerFlag` on a name overlap — is recorded in [Discovery — File-extension namespace](../discovery/discovery-sources.md#loom-flag-namespace). The flag name `'loom'`, `type: 'string'`, the presence of a `description`, and the registration ordering relative to `resources_discover` are the observable pins; the exact `description` string is non-normative — Pi renders it only in `--help`, where it keys no loom surface — and `'Loom file or directory paths, joined with the OS path-list separator'` is an illustrative example. Walks the five discovery sources defined in [Directory Convention](../discovery.md) directly — the global directory `~/.pi/agent/looms/`, the project directory `.pi/looms/`, every installed package's `pi.looms` entry or conventional `looms/` directory (per [Package discovery](../discovery/package-and-settings.md#package-discovery)), the `loomPaths` settings array (per [Settings file reads](../discovery/package-and-settings.md#settings-file-reads)), and the `--loom` CLI flag (whose `pi.getFlag('loom')` value is split on `path.delimiter` per [Directory Convention](../discovery.md)) — at factory time and again on every `resources_discover` event. The event payload (`{ cwd, reason }`) supplies the project root used for the project-local source (the factory-time scan resolves the project root from the [`FileSystem.cwd()` seam member](./host-interfaces-services.md#pic-13); on every `resources_discover` event `event.cwd` is preferred over that seam value, so a future per-session cwd change is honoured on reload) and distinguishes initial scan (`reason: "startup"`) from reload (`reason: "reload"`); Pi delivers no paths to the handler — `ResourcesDiscoverResult` has no `loomPaths` slot — and the `pi` manifest namespace does not enumerate `pi.looms`. The loom extension owns the walk for all five sources; the event is purely a re-trigger, and the handler returns the empty result `{}` to satisfy the typed return.
2. Parses each discovered `.loom` file and builds a *pending registration list* keyed by slash name. The factory does **not** call `pi.registerCommand` synchronously — Pi's prompt templates and other extensions' commands are not yet enumerable at factory time, so cross-format collision detection cannot run here. Concretely, on the pre-bind runtime returned to the extension factory by `createExtensionRuntime()`, every action method that reads or mutates session-bound state — including `getCommands`, `getActiveTools`, `getAllTools`, `setActiveTools`, `sendMessage`, `sendUserMessage`, `setModel`, `getThinkingLevel`, `setThinkingLevel`, and the rest of the `notInitialized`-tagged slots in `@earendil-works/pi-coding-agent`'s `dist/core/extensions/loader.js` — is bound to a throwing closure (`Error("Extension runtime not initialized. Action methods cannot be called during extension loading.")`) that `Runner.bindCore()` later overwrites with the real implementation alongside `session_start`. The token `notInitialized` is the internal variable name of that throwing closure inside Pi's loader source; it is **not** an observable sentinel value an extension can compare against or switch on. Calling any of these methods from inside the factory raises an unhandled exception that aborts factory load. <a id="pre-bind-throw-closure-evidence"></a>The `dist/core/extensions/loader.js` path, the `notInitialized` internal variable name, the private `Runner.bindCore()` overwrite step, and the verbatim throw-string text above are **known-fragile evidence** at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) — diffable artefacts witnessing that the pre-bind action-method slots throw at the pin, not load-bearing claims about their stable shape across patches (a Pi patch may rename the `_`-prefixed loader internals, re-bundle the `dist/` tree, or reword the throw string). The observable rules in this step — that calling a session-state action method pre-bind aborts factory load, while the registration sinks and `refreshTools` are safe before bind — hold independent of that evidence trail, which is re-audited on each Pi minor bump via [item (q) of the *Editorial-review checklist for unpinned host presuppositions*](./version-bump-step2.md#bump-checklist-pre-bind-and-renderer-resolution) under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. The methods Pi documents as safe before bind are `pi.registerCommand` / `pi.registerTool` / `pi.registerFlag` / `pi.registerMessageRenderer` (registration sinks) and `refreshTools` (a no-op on the pre-bind runtime). Every other reach for session state is deferred to a `session_start` (or later) handler. The signature pinned by loom 1.0 is `pi.getCommands(): SlashCommandInfo[]` (synchronous, returns the current registered-commands snapshot, where `SlashCommandInfo` is the interface exported from `@earendil-works/pi-coding-agent`'s `core/slash-commands.d.ts` with fields `{ name, description?, source: SlashCommandSource, sourceInfo }`); the returned array is mutable in the type system but MUST be treated as read-only by convention (the loom 1.0 collision check is a single forward pass and never mutates it); the snapshot is only meaningful on or after `session_start`. The build-time SDK surface inventory (per *Host prerequisites*) MUST list `SlashCommandInfo` and the three-arm `SlashCommandSource = "extension" | "prompt" | "skill"` explicitly so a Pi minor that renames either fails the build.
3. Subscribes a `session_start` handler that consults `pi.getCommands()`, drops any pending loom whose slash name collides with an existing entry whose `source` is in the [loom 1.0 cross-format collision source set](./host-interfaces-services.md#v1-seam-pi-owned-subagents-collision-source-set) defined below — the three arms of Pi's `SlashCommandSource` union (`"prompt"`, `"extension"`, `"skill"`), per `@earendil-works/pi-coding-agent`'s `core/slash-commands.d.ts` and per [Discovery — Slash-name collisions across formats](../discovery.md) — <a id="slash-handler-registration"></a>registers each survivor via `pi.registerCommand(name, { description, handler })` — one slash command per loom — and emits a `loom/load/cross-format-collision` diagnostic for each drop through the channel defined in [Diagnostics](../diagnostics.md). The `description` field passed to `pi.registerCommand` carries `frontmatter.description` verbatim and only that — `argument-hint` is **not** concatenated, since Pi's extension-command autocomplete UI has no `argumentHint` slot and the binder is the only loom 1.0 consumer of `argument-hint` (see [Slash-Command Argument Binding](../binder.md)). loom 1.0 passes only the two keys `description` and `handler`; the optional `getArgumentCompletions` key is omitted, since Pi's `RegisteredCommand` interface types it as optional and loom 1.0 exposes no autocomplete-completion surface (a deferred `argumentHint`-style upstream is tracked in [Future Considerations](../future-considerations/surface-extensions.md#surface-extensions-without-a-dedicated-topic-page-seam)). The same handler also de-registers any previously-registered loom whose name now collides with a newly-appeared `.md` template or extension command (e.g. after a settings reload or another extension's activation), emitting the same diagnostic. *Drain-state-gated dispatch.* The `handler` value passed as the `handler` field of every `pi.registerCommand(name, { description, handler })` call above MUST, on every slash dispatch, invoke `LoomRegistry.readDrainState()` as its first action and route through the closed three-arm enumeration (a)/(b)/(c) pinned by the **`LoomRegistry` drain-state contract**'s *Methods* paragraph below — including that paragraph's *Read-failure fallback* clause, which pins arm (c) (the `"loom /<name>: extension degraded; /reload to recover"` system note) as the conservative fail-safe when the `readDrainState` call itself throws. This obligation is normative at this registration site; the arm enumeration and the read-failure routing are owned by the *Methods* paragraph below and are not restated here.

   <a id="getcommands-completeness-presupposition"></a>Step 3's first `session_start` collision pass additionally relies on a Pi *behavioural* precondition the `SlashCommandInfo[]` snapshot does not encode: that by the time Pi emits the first `session_start`, `pi.getCommands()` already enumerates every command Pi registers from prompt templates and skills, and every command a sibling extension registers either at factory time or from a `session_start` handler Pi load-orders before loom's. Pi controls extension load order and emits each extension's `session_start` in that order, so a sibling loaded after loom that first registers a colliding command inside its own `session_start` handler is absent from loom's first snapshot; the first-pass check passes and Pi disambiguates the duplicate with its own numeric-suffix form (`<name>:1` / `<name>:2`) — the observable surface of a first-pass miss — rather than loom dropping the colliding loom and emitting `loom/load/cross-format-collision`. The next-cycle re-evaluation of such later-appearing siblings — the de-registration of a previously-registered loom whose name now collides with a newly-appeared `.md` template or extension command — is owned by [Discovery — DISC-4](../discovery/discovery-sources.md#disc-4) and the de-registration clause of this step above; this presupposition does not restate that recovery. The precondition is taken on a best-effort basis against the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) above and is re-audited on each Pi minor bump via item (l) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect a load-order or population-timing change from the type surface alone.
4. Subscribes a `session_shutdown` handler (`pi.on("session_shutdown", handler)`) that owns disposing every resource the extension instance holds. The Pi SDK documents `session_shutdown` as fired before an extension runtime is torn down due to quit, reload, or session replacement, with `event.reason: "quit" | "reload" | "new" | "resume" | "fork"`; loom uses it as the single teardown contract for the outgoing extension instance so that Pi's subsequent `ExtensionRuntime.invalidate(...)` call does not strand subagent provider connections, chokidar file handles, debounce timer handles, or `AbortController` listeners across `/reload`, `/new`, fork, or quit. <a id="extension-runtime-invalidate-teardown"></a>This step is the canonical owner of the `ExtensionRuntime` pin: the `ExtensionRuntime` interface — and the `invalidate(...)` member the teardown contract names — is declared at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above); loom redefines neither and consumes neither (it never calls `invalidate(...)` — Pi does). The teardown contract presupposes that Pi fires `session_shutdown` on the outgoing instance before it calls `ExtensionRuntime.invalidate(...)`, and that a `pi.*` call reaching an already-invalidated runtime throws. Neither the member nor that post-invalidation throw behaviour is factory-probed or enumerated by step 2(a)'s seven-capability literal-read assertion, so a silent rename or behavioural change surfaces as a runtime error mid-teardown rather than at load time, and is re-audited on each Pi minor bump via [item (t) of the *Editorial-review checklist for unpinned host presuppositions*](./version-bump-step2.md#bump-checklist-extension-runtime-invalidate) under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below.
5. <a id="watcher-hot-reload-registration"></a>Registers a chokidar file watcher over the discovered roots and maintains an internal mutable `LoomRegistry` (a `Map<slashName, parsedLoom>`) plus a `.warp` import graph. The slash-command `handler` registered in step 3 closes over the registry and dispatches by slash name on every invocation, so a registry swap takes effect on the next call without a `pi.registerCommand` re-registration. On a chokidar event for an **existing** loom or `.warp` file the watcher debounces (250 ms window, to coalesce editor `change` + `rename` bursts; the 250 ms window is measured against the runtime's injected `Clock` seam (per the runtime's DI skeleton — see [`Clock` / `FakeClock` interface](./host-interfaces-services.md#clock--fakeclock-interface) below), the seam exists to make burst-coalescing assertions deterministic and is not a tunable runtime knob), re-parses just the changed file plus every transitive `.warp` importer reached through the import graph, and atomically swaps the affected entries in `LoomRegistry` via build-aside-then-publish: the runtime re-parses the changed file plus every transitive `.warp` importer into a staging set, recompiles their AJV validators into a staging cache, and re-registers the tools synthesised from prompt-mode `.loom` callables through `pi.registerTool` for every newly-distinct schema slug the staged rebuild produces (<a id="register-tool-post-startup-presupposition"></a>that `pi.registerTool` accepts registration calls after startup is an unpinned Pi *behavioural* presupposition the SDK type surface does not encode — taken on a best-effort basis against the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) above and re-audited on each Pi minor bump via [item (z) of the *Editorial-review checklist for unpinned host presuppositions*](./version-bump-step2.md#bump-checklist-register-tool-post-startup) under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; existing cached registrations under unchanged slugs are reused, so no `pi.registerTool` call is issued for them) — and only after every step, including the `pi.registerTool` re-registration, succeeds installs the staged registry, the staged validator cache, and the staged prompt-mode registration-cache entries into `LoomRegistry`, the validator cache, and the registration cache in a single synchronous publish step. If any rebuild step throws — including a `pi.registerTool` throw for a newly-distinct schema slug, which is sequenced before publish — the staging set is discarded and the prior `LoomRegistry` snapshot, AJV validator cache, and prompt-mode registration cache all remain live; the failure surfaces as a single `loom/runtime/registry-swap-failed` diagnostic (see [Diagnostics](../diagnostics.md)). Because Pi exposes no `pi.unregisterTool` (per [Tool-registration lifetime and visibility](./tool-registration-lifetime.md#tool-registration-lifetime-and-visibility)), any `pi.registerTool` call that already succeeded before the throw leaves an orphaned entry in Pi's global tool registry — the cosmetic accumulation that section already accepts — while loom-side rollback is limited to discarding the never-published staged registry snapshot and staged registration-cache entries. Compiled AJV validators for swapped entries are dropped from the per-loom cache as part of the publish step (not before). The runtime does **not** call `ctx.reload()` for content edits — doing so would tear down every Pi extension, skill, prompt, and theme in the session for a single keystroke and would also race the post-`await` runtime-deadness rule that `ctx.reload()` documents. The ordering constraint requiring step 4's `session_shutdown` subscription to follow this registry/watcher construction is the [**Factory-ordering pin**](./session-shutdown-semantics.md#factory-ordering-pin) under **Session shutdown semantics** below.

   **In-flight invocation rule.** The slash-handler closure registered in step 3 (which calls `LoomRegistry.readDrainState()` per the [*Drain-state-gated dispatch*](#slash-handler-registration) clause and routes through the entry table for the dispatch arm) reads the entry exactly once at handler entry; a swap mid-execution does not affect a running invocation. Slash-command dispatch is gated by the three-arm `LoomRegistry.readDrainState()` enumeration pinned in the [*Drain-state-gated dispatch*](#slash-handler-registration) clause appended to step 3 above and the [`LoomRegistry` drain-state contract](./drain-state-contract.md#loom-registry-drain-state-contract) below (including that contract's *Methods* paragraph's [*Read-failure fallback*](./drain-state-contract.md#read-failure-fallback) clause); that contract supersedes the prior single-arm `drained`-keyed short-circuit and is the sole authority on which non-dispatch system note (the `"shutting-down"` arm's legacy `"loom /<name>: extension shutting down"` template, the `"degraded-needs-reload"` arm's `"loom /<name>: extension degraded; /reload to recover"` template, or arm (a) dispatch) the handler returns. The next invocation sees the post-swap version. In-flight invocations therefore complete against the pre-swap snapshot and any tool calls or queries they have already committed remain final, consistent with [Errors and Results — No rollback](../errors-and-results.md).

   **Structural changes.** Adding a brand-new `.loom` file or removing an existing one cannot register / unregister a slash command after extension load (Pi exposes no symmetric `pi.unregisterCommand` for the removal case, and adding a new command after the extension factory has run is not a documented post-startup mutation). When the watcher observes such an event, it emits a single `loom-system-note` with `content` formatted as `loom watcher: <N> file(s) added or removed; run /reload to refresh the slash command list`, `display: true`, and the `details` payload defined under System notes. The substitution rule is fixed: only `<N>` is substituted; every other character of the template — including the literal token `file(s)` (always rendered with the parenthesised `s`, never pluralised), the literal slash-command name `/reload` (not substituted from any registered command), and the trailing `; run /reload to refresh the slash command list` clause — ships verbatim regardless of `N`. `<N>` is rendered in base 10 with no thousands separator, no leading zero, and no sign, and equals `details.structural.added.length + details.structural.removed.length` (a path that appears in both arrays — e.g., a rename observed as removal-then-addition within the same debounce window — counts twice, since the two arrays are disjoint by role and not deduplicated against each other). Worked examples: with `added.length = 1, removed.length = 0`, the rendered content is `loom watcher: 1 file(s) added or removed; run /reload to refresh the slash command list`; with `added.length = 3, removed.length = 2`, it is `loom watcher: 5 file(s) added or removed; run /reload to refresh the slash command list`. The note MUST NOT be emitted when `added.length + removed.length === 0`. A debounce window that closes with no resolved add or remove paths produces no `loom-system-note` — including a settings-file edit whose post-merge `loomPaths` array is byte-identical to the previous resolved set, an edit that touches only `looms.binderModel` or unrelated keys, and a chokidar burst over discovery roots that does not net any added or removed `.loom` / `.warp` files. The watcher's other obligations for the window (validator-cache invalidation per the **In-flight invocation rule** above, settings-file cache invalidation per the settings-file watcher path) still run; only the note is suppressed. A rename observed within the same debounce window as `removed` of path P followed by `added` of path P still has `added.length + removed.length === 2` and MUST emit the note (the two arrays are disjoint by role and not deduplicated against each other, per the rule above); the suppression applies only to the strictly-empty case. The note prompts the user to run `/reload` (which invokes Pi's `ctx.reload()` and re-executes the entire extension factory). Multiple structural events observed within a single 250 ms debounce window coalesce into one note; events in distinct windows fire separately. The note is informational, not a diagnostic; no `loom/load/*` code is registered for it. Settings-file edits affecting the `loomPaths` array are treated as structural changes and routed through the same prompt; per-`.loom`-file content edits are not.
