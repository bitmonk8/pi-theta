# Registration steps

1. Registers the `--loom` CLI flag via `pi.registerFlag('loom', { type: 'string', description: 'Loom file or directory paths, joined with the OS path-list separator' })` **before** subscribing to `resources_discover`, so the value is readable from inside the discovery handler via `pi.getFlag('loom')`. Walks the five discovery sources defined in [Directory Convention](../discovery.md) directly ‚Äî the global directory `~/.pi/agent/looms/`, the project directory `.pi/looms/`, every installed package's `pi.looms` entry or conventional `looms/` directory (per [Package discovery](../discovery/package-and-settings.md#package-discovery)), the `loomPaths` settings array (per [Settings file reads](../discovery/package-and-settings.md#settings-file-reads)), and the `--loom` CLI flag (whose `pi.getFlag('loom')` value is split on `path.delimiter` per [Directory Convention](../discovery.md)) ‚Äî at factory time and again on every `resources_discover` event. The event payload (`{ cwd, reason }`) supplies the project root used for the project-local source (`event.cwd` is preferred over any factory-captured `cwd`, so a future per-session cwd change is honoured on reload) and distinguishes initial scan (`reason: "startup"`) from reload (`reason: "reload"`); Pi delivers no paths to the handler ‚Äî `ResourcesDiscoverResult` has no `loomPaths` slot ‚Äî and the `pi` manifest namespace does not enumerate `pi.looms`. The loom extension owns the walk for all five sources; the event is purely a re-trigger, and the handler returns the empty result `{}` to satisfy the typed return.
2. Parses each discovered `.loom` file and builds a *pending registration list* keyed by slash name. The factory does **not** call `pi.registerCommand` synchronously ‚Äî Pi's prompt templates and other extensions' commands are not yet enumerable at factory time, so cross-format collision detection cannot run here. Concretely, on the pre-bind runtime returned to the extension factory by `createExtensionRuntime()`, every action method that reads or mutates session-bound state ‚Äî including `getCommands`, `getActiveTools`, `getAllTools`, `setActiveTools`, `sendMessage`, `sendUserMessage`, `setModel`, `getThinkingLevel`, `setThinkingLevel`, and the rest of the `notInitialized`-tagged slots in `@earendil-works/pi-coding-agent`'s `dist/core/extensions/loader.js` ‚Äî is bound to a throwing closure (`Error("Extension runtime not initialized. Action methods cannot be called during extension loading.")`) that `Runner.bindCore()` later overwrites with the real implementation alongside `session_start`. The token `notInitialized` is the internal variable name of that throwing closure inside Pi's loader source; it is **not** an observable sentinel value an extension can compare against or switch on. Calling any of these methods from inside the factory raises an unhandled exception that aborts factory load. The methods Pi documents as safe before bind are `pi.registerCommand` / `pi.registerTool` / `pi.registerFlag` / `pi.registerMessageRenderer` (registration sinks) and `refreshTools` (a no-op on the pre-bind runtime). Every other reach for session state is deferred to a `session_start` (or later) handler. The signature pinned by loom 1.0 is `pi.getCommands(): SlashCommandInfo[]` (synchronous, returns the current registered-commands snapshot, where `SlashCommandInfo` is the interface exported from `@earendil-works/pi-coding-agent`'s `core/slash-commands.d.ts` with fields `{ name, description?, source: SlashCommandSource, sourceInfo }`); the returned array is mutable in the type system but MUST be treated as read-only by convention (the loom 1.0 collision check is a single forward pass and never mutates it); the snapshot is only meaningful on or after `session_start`. The build-time SDK surface inventory (per *Host prerequisites*) MUST list `SlashCommandInfo` and the three-arm `SlashCommandSource = "extension" | "prompt" | "skill"` explicitly so a Pi minor that renames either fails the build.
3. Subscribes a `session_start` handler that consults `pi.getCommands()`, drops any pending loom whose slash name collides with an existing entry whose `source` is in the [loom 1.0 cross-format collision source set](./host-interfaces-services.md#v1-seam-pi-owned-subagents-collision-source-set) defined below ‚Äî the three arms of Pi's `SlashCommandSource` union (`"prompt"`, `"extension"`, `"skill"`), per `@earendil-works/pi-coding-agent`'s `core/slash-commands.d.ts` and per [Discovery ‚Äî Slash-name collisions across formats](../discovery.md) ‚Äî <a id="slash-handler-registration"></a>registers each survivor via `pi.registerCommand(name, { description, handler })` ‚Äî one slash command per loom ‚Äî and emits a `loom/load/cross-format-collision` diagnostic for each drop through the channel defined in [Diagnostics](../diagnostics.md). The `description` field passed to `pi.registerCommand` carries `frontmatter.description` verbatim and only that ‚Äî `argument-hint` is **not** concatenated, since Pi's extension-command autocomplete UI has no `argumentHint` slot and the binder is the only loom 1.0 consumer of `argument-hint` (see [Slash-Command Argument Binding](../binder.md)). loom 1.0 passes only the two keys `description` and `handler`; the optional `getArgumentCompletions` key is omitted, since Pi's `RegisteredCommand` interface types it as optional and loom 1.0 exposes no autocomplete-completion surface (a deferred `argumentHint`-style upstream is tracked in [Future Considerations](../future-considerations/surface-extensions.md#surface-extensions-without-a-dedicated-topic-page-seam)). The same handler also de-registers any previously-registered loom whose name now collides with a newly-appeared `.md` template or extension command (e.g. after a settings reload or another extension's activation), emitting the same diagnostic. *Drain-state-gated dispatch.* The `handler` value passed as the `handler` field of every `pi.registerCommand(name, { description, handler })` call above MUST, on every slash dispatch, invoke `LoomRegistry.readDrainState()` as its first action and route through the closed three-arm enumeration (a)/(b)/(c) pinned by the **`LoomRegistry` drain-state contract**'s *Methods* paragraph below ‚Äî including that paragraph's *Read-failure fallback* clause, which pins arm (c) (the `"loom /<name>: extension degraded; /reload to recover"` system note) as the conservative fail-safe when the `readDrainState` call itself throws. This obligation is normative at this registration site; the arm enumeration and the read-failure routing are owned by the *Methods* paragraph below and are not restated here.
4. Subscribes a `session_shutdown` handler (`pi.on("session_shutdown", handler)`) that owns disposing every resource the extension instance holds. The Pi SDK documents `session_shutdown` as fired before an extension runtime is torn down due to quit, reload, or session replacement, with `event.reason: "quit" | "reload" | "new" | "resume" | "fork"`; loom uses it as the single teardown contract for the outgoing extension instance so that Pi's subsequent `ExtensionRuntime.invalidate(...)` call does not strand subagent provider connections, chokidar file handles, debounce timer handles, or `AbortController` listeners across `/reload`, `/new`, fork, or quit.
