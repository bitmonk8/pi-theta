## Pi version bump procedure

This subsection enumerates the contributor checklist that any change widening, narrowing, or otherwise moving the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) on `@earendil-works/pi-coding-agent` (and the lock-step `pi-agent-core` / `pi-ai` / `pi-tui` minors) MUST follow. The three “requires re-validating” sentences elsewhere in the spec ([SDK capability inventory — Re-validation on `peerDependencies` widening](./capability-inventory-items.md#sdk-cap-peer-dep-revalidation), the **Host prerequisites — Pi SDK pin** paragraph above on this page, and the strict-capability sentence in [Slash-Command Argument Binding — Strict-capability requirement](../binder/binder-model-and-context.md#strict-capability-requirement)) all resolve to the steps below; the build-time tests cited at each step are the mechanical gates that fail loudly until the corresponding manual step has been completed.

1. **Re-typecheck against the new package.** Install the candidate `@earendil-works/pi-coding-agent` minor (and its lock-step siblings), run `npm run typecheck`, and confirm no Pi-imported types under the loom-load-bearing surface have changed shape in a way that breaks the runtime. Type errors here mean the candidate minor is not a drop-in replacement and the surface descriptions on this page MUST be updated before the bump can proceed. A failure of the bidirectional `SessionShutdownEvent['reason']` type-equality assertion (per step 5 below; the runtime invariant the assertion guards is the [Unknown-reason rule](./unknown-reason-rule.md#unknown-reason-rule) under **Extension entry point** sub-step 4 above) is the first expected exception to the spec-prose-update obligation in the immediately preceding sentence (the second is the module-import-resolution failure mode — `TS2305` / `TS2307` / `TS2614` / `TS2724` and any other TS code in the same module-import-resolution family that arises from a Pi-side rename/drop on the package-root re-export surface — discussed later in this step): it is handled by step 5 below (an edit limited to the `SessionShutdownEvent['reason']` snapshot entry's `literals` field in the extension module's pinned-constants block, per step 5; no spec-prose change). The mechanical discriminator the carve-out keys on is the literal brand string `loom/typecheck/session-shutdown-reason-snapshot` appearing in the failing `tsc` diagnostic, per step 5 below's brand-string requirement; a typecheck red whose diagnostic does not carry that brand string is not in the carve-out (this is what disambiguates the in-scope union-widening/narrowing failure from look-alike failures such as a TS2314 generic-arguments-required diagnostic produced by a Pi-side re-shape of `SessionShutdownEvent` into `SessionShutdownEvent<TReason>`, which mentions `SessionShutdownEvent` but does not originate at the brand-emitting assertion). The carve-out applies per-assertion, not per-invocation: an `npm run typecheck` invocation that reports both the brand-string-carrying red and any other typecheck red has only the brand-string-carrying red carved out, and every non-carved-out red in the same invocation still requires the spec-prose update this step prescribes. A module-import-resolution failure originating from the same assertion — surfacing as `TS2305` (`"Module '<path>' has no exported member '<name>'."`), `TS2724` (the "did you mean Y?" rename variant), `TS2307` (module not found), `TS2614` (default-vs-named-import mismatch), or any other TS code in the same module-import-resolution family produced by a Pi-side rename/drop of `SessionShutdownEvent` on the package-root re-export surface (so the assertion never reaches a value/type comparison) — is a second expected exception, and is **not** covered by step 5: route every TS code in that family through [step 2(a)](./version-bump-step2.md#bump-step-2-positive)'s surface-inventory update, re-pinning the import specifier in the same edit as the rest of the bump (per step 5 below's import-resolution routing). Routing keys on the TS error code, not on the message-text quote — the message-text quote above is illustrative of the TS2305 wording at the loom 1.0 Pi-SDK pin and may drift across TypeScript versions without affecting the routing contract. The "no spec-prose change" carve-out applies to the snapshot mechanics only: the source-of-truth for `event.reason`'s closed set is the extension module's pinned-constants block (per the [Unknown-reason rule](./unknown-reason-rule.md#unknown-reason-rule)), and the in-prose closed-set listings elsewhere — the sub-step 4 introductory paragraph, the [Unknown-reason rule](./unknown-reason-rule.md#unknown-reason-rule) itself (including its parenthesised re-listing), the [*Edge cases* `reason: "new" | "resume" | "fork"` bullet](./session-only-degraded-state.md#session-only-reason-degraded-state) (scoped to the closed-set listings within it — i.e. the prose enumerations of the closed `{"quit", "reload", "new", "resume", "fork"}` set — and explicitly excluding the inline `{"new", "resume", "fork"}` triplet at the `loom/host/session-shutdown-runtime-degraded` emission site, which is a normative runtime literal subject to the lock-step inbound-reference-sweep update obligation in step 5 below), the **Per-invocation operator visibility (clean-cancel path)** bullet's `"quit" | "reload" | "new" | "resume" | "fork" | string` widening, and the corresponding listing in [`spec.md` — Session model — SM-2 closed shutdown reason set](../../spec/session-model-and-appendix.md#sm-2-closed-shutdown-reason-set) — are illustrative restatements of that constant pinned at write time, not independent normative copies. A Pi widening/narrowing handled under this carve-out therefore leaves those listings transiently stale with no runtime-correctness impact (the runtime consumes the snapshot constant, not the prose; the type-equality assertion of step 5 is the gate); bringing the prose back into byte-for-byte alignment with the new union is an editorial sweep folded into the next page-touching edit rather than a blocker on this step. The diagnostics §7 `<reason>` closed-enum row — declared under [Diagnostics — §7 Identifier-, descriptor-, and closed-enum placeholders](../diagnostics/placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders) and spec-versioned per GOV-7 / GOV-8 (extending the table is a breaking change to the rendering surface) — is the sole exception to that "illustrative restatement" status and is therefore the **sixth** co-edit site this carve-out enumerates, alongside the five illustrative restatements above. A Pi-side widening of `SessionShutdownEvent['reason']` *is* a GOV-7 / GOV-8 rendering-surface change for §7's `<reason>` row, and the §7 row MUST be updated in the same commit as the snapshot constant so the GOV-7 / GOV-8 invariant on the closed-enum rendering surface is preserved synchronously with the snapshot edit (no transiently-stale window for the §7 table; the snapshot-edit-only permission of this carve-out covers the five in-prose illustrative restatements only and does **not** authorise a snapshot edit to ship without its matching §7 row update). A Pi-side narrowing is symmetric: the §7 row's enumeration MUST be narrowed in the same commit as the snapshot constant. Treating the §7 row as part of the snapshot edit's commit (rather than as a separately re-litigated spec-prose change) is what keeps the carve-out's "no spec-prose re-litigation" property intact for the GOV-7 / GOV-8 surface — the rendering-surface change rides the snapshot edit synchronously rather than reopening the spec-review for the §7 table. Other typecheck failures still require the spec-prose update this step prescribes.
