# Audit resolution

- <a id="audit-scope"></a>*Audit scope.* The *audited source tree* — the canonical scope name every other paragraph on this page (and the [`spec.md` — Orientation — Prerequisites](../../spec/overview-and-orientation.md#orientation) cross-reference) MUST use when naming this audit's scope — is defined as the closed file set matching the positive include glob `src/**/*.ts` (hand-authored TypeScript runtime source files under `src/`, recursive), minus the excluded files enumerated here. The include glob is closed: files outside `src/` (notably anything under `test/`, which is wholly outside the included root) and files inside `src/` whose extension is not bare `.ts` (e.g. `.tsx`, `.cts`, `.mts`, `.d.cts`, `.d.mts`, `.js`, `.mjs`, `.cjs`, `.json`, `.md`, fixture data files in any other extension) are out of scope by not matching the include glob and need no separate exclusion. *Cross-platform glob normalisation.* The glob is case-sensitive on the literal segment text — both the directory segments (`src/`) and the filename leaf — and the implementation MUST normalise filename comparisons to case-sensitive matching across host OSes, so a Linux contributor (case-sensitive FS) and a Windows / macOS contributor (case-insensitive FS by default) audit the same closed file set for a given source tree. A file whose on-disk name is, e.g., `src/extension/Index.TS` does **not** match the include glob; the audit MUST surface no Pi-side reference inside it and any in-scope reference there is a contributor-side scope violation rather than a silently-walked input. *Symlink handling.* Symlinks under `src/` MUST NOT be followed: the implementation MUST treat each symlink as opaque (the link entry itself is not walked as a file, and its target is excluded from the walk regardless of whether the target lies inside or outside `src/`). This rule holds uniformly for file-symlinks and directory-symlinks under `src/`, so an audit run on a tree that contains a symlink into `node_modules/`, into the build target, or into another worktree produces the same closed file set as a tree without the symlink. *Extension-suffix matching.* The `.ts` extension is matched case-sensitively against the literal `.ts` suffix on the leaf segment: `.tsx`, `.cts`, `.mts`, `.d.cts`, `.d.mts`, and any case-variant of `.ts` (e.g. `.TS`, `.Ts`) are out of scope by failing the include glob and need no separate exclusion. The `.d.ts` declaration / type-stub case is the one nominally-`.ts` suffix that *does* match the include glob and is therefore handled by the explicit `*.d.ts` entry in the exclusions list below, not by this paragraph. *Source-file encoding.* Source files in scope MUST be UTF-8-encoded; UTF-8 with BOM is permitted (the BOM is stripped before parsing). The implementation MUST decode each in-scope file with a strict UTF-8 decoder that raises on any invalid byte sequence (lone surrogates, overlong forms, truncated multi-byte sequences, or any byte sequence outside the UTF-8 grammar) before invoking the audit's walker, and on raising MUST report the zero-based byte offset of the first invalid byte sequence in the original (pre-normalisation) file content (for a non-UTF-8 BOM the reported offset is `0`, the BOM's leading byte), mirroring the offset semantics pinned for the runtime `loom/load/invalid-encoding` diagnostic; that decode error is the mechanism that produces the parse-failure outcome below, so a tolerant decoder that substitutes replacement characters and admits mojibake into the walker is non-conformant. Files in any other encoding (e.g. UTF-16 LE / BE, UTF-32, or any non-Unicode legacy encoding) therefore fail the audit under the *Infrastructure-failure handling* (i) implementation-agnostic category below as a parse failure, on the same footing as a syntax error raised by the audit's own walker on an audited file. Excluded files matching the include glob are: any `*.test.ts` co-located under `src/`; any `*.spec.ts` co-located under `src/`; any `*.d.ts` declaration / type-stub file (which carries no executable Pi-side surface reference at runtime); any file whose path contains a `__tests__/` directory segment at any depth under `src/`; and the `SessionShutdownEvent['reason']` brand-string assertion module if the implementation places it under `src/extension/` *and* the file path matches the suffix convention `src/extension/**/*.assert.ts` (the suffix is pinned exclusively to `*.assert.ts` here; the implementation MUST NOT substitute another suffix, since a permissive substitution — e.g. a leaf picking `*.ts` — would defeat the audit's scope altogether and a per-leaf alternative offers no audit-side value over the pinned suffix) — the suffix is the audit's only syntactic discriminator for the assertion module's role under `src/extension/`, since a static-AST walker has no other signal to identify it without coupling to the brand-string text. An inlined or otherwise embedded `SessionShutdownEvent['reason']` brand-string assertion in any other `src/extension/**/*.ts` file (i.e. one that does not match the pinned suffix) is **not** excluded by this item; the audit MUST resolve the brand-string assertion's `Type.Unsafe` (and any other in-scope) references in such a file through the normal inventory rules above on the same footing as any other `src/extension/` source file (its other permitted location, `test/`, is already outside the included root, per step 5 below). Generated build outputs MUST NOT be emitted into `src/` — the include glob describes hand-authored runtime source, and the build target (`dist/` or equivalent) is outside the included root by construction; an implementation that violates this rule by writing generated `.ts` files under `src/` violates this section rather than silently expanding audit coverage. *Generated-output-in-src limitation (acknowledged soft surface).* The prohibition is **contributor-discipline-only**: the inventory-closure audit walks `src/**/*.ts` blindly per the include glob above, so a generated `.ts` file emitted under `src/` would be walked on the same footing as a hand-authored file and any Pi-side surface references in it would either surface as family (1) / (2) / (3) violations against `SDK_SURFACE_INVENTORY` or be silently authorised by inventory entries (or `// allow-pi-surface:` markers) the generator emits — the prohibition itself has no mechanical surface inside the inventory-closure audit's five-family set above (its closure rule above forbids widening to a sixth family in this PIC edit), and is mitigated by the contributor obligation to keep the build target outside `src/` rather than by an audit discriminator. This soft surface is parallel to the *Substitution limitation* and *Stale-citation limitation* framings under *Stale-marker discriminator* below: PIC pins the prohibition and acknowledges that detection is contributor-side. The implementation MAY publish a separate scope-audit check (e.g. a `// @generated` leading-comment heuristic on every `src/**/*.ts` file walked by the inventory-closure audit) under its own `scope-audit/generated-source-in-tree` discriminator — sibling to but disjoint from the inventory-closure audit's five-family set, since this is a scope-violation discriminator rather than an inventory-resolution discriminator — but PIC does not require it; widening the inventory-closure audit's family-set to subsume this check is a PIC edit deferred past loom 1.0. The three-class `<class>` closure pinned by the *Failure-surface contract* below (the closed enumeration `violation` / `infra` / `canary`) binds the inventory-closure audit's own records only; a sibling scope check of this kind, which is not part of the inventory-closure audit, MUST use a separate top-level prefix outside the `audit/` namespace (the `scope-audit/<symptom>` shape above is illustrative, and the implementation MAY pick another non-`audit/` prefix at its discretion), so its discriminator is not subject to the three-class closure and does not perturb that closure's exhaustiveness. A future change adopting any in-tree test, fixture, generated-output, or other source-tree convention not enumerated above (e.g. a `*.fixture.ts` convention, a `**/fixtures/**` directory under `src/`, a `*.tsx` runtime extension, a `*.gen.ts` codegen output) MUST extend either the include glob or the exclusion list in this paragraph in the same edit that introduces the convention; silently inheriting an unenumerated convention as either in-scope or out-of-scope is a violation of this section.
- <a id="audit-category-2-inventory-join-key"></a>*Category-(2) inventory join key.* The audit resolves a category-(2) named import **from one of the four `@earendil-works/*` peer packages** (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`) (per *Target surface categories* (2) above) to an `SDK_SURFACE_INVENTORY` entry by **exact match** between the imported symbol identifier and the **leftmost dot-separated segment** of the entry's `path` field. This rule is scoped to those four peer packages exclusively: `typebox` imports are out of scope of this rule; see the *Target surface categories* `typebox` carve-out above for the `typebox` named-import allow-list resolution path (and the parallel `typebox` member-access allow-list for `Type.<member>` access). Concretely, an `import { <Name> } from "@earendil-works/<peer>"` declaration resolves iff some inventory entry's `path` field equals `<Name>` exactly or starts with the literal substring `<Name>.` (i.e. `<Name>` followed by a single ASCII `.` followed by any non-empty suffix); a `path` field whose leftmost segment is a strict-prefix-but-not-equal of `<Name>` (e.g. inventory `path: "Foo.bar"` against import `<Name> = "FooBar"`) MUST NOT resolve, and a `path` field whose leftmost segment is `<Name>` followed by any character other than `.` (no segment separator) MUST NOT resolve. The package specifier is **not** part of the join key — multiple `import { <Name> } from "@earendil-works/<peer>"` declarations of the same `<Name>` from the same peer package resolve to the same inventory entry, and the audit MUST NOT require one inventory entry per `(package, name)` pair. `<Name>` in this rule is the **canonical (source) name** of the named-import specifier — i.e. the identifier that appears before any `as`-rename in an `import { <Name> } from "@earendil-works/<peer>"` declaration; the join key (and, symmetrically, the `typebox` named-import allow-list whose single permitted entry is `Type`, which resolves `typebox` imports per the *Target surface categories* carve-out above rather than through this rule) consumes the canonical name only and never the renamed local binding. The `as`-rename form `import { <Name> as <Alias> } from "@earendil-works/<peer>"` (and the `typebox` and re-export analogues) does not reach this resolution path — it is forbidden under family (4) by *Recognised import/access shapes* above as an aliased rebinding — so the question of whether the join key would consume `<Name>` or `<Alias>` is moot by construction; an audit implementation MUST NOT permit the renamed form to silently resolve through either authorisation path. The existing `type-union-snapshot` entry at `path: "SessionShutdownEvent.reason"` accordingly resolves an `import { SessionShutdownEvent } from "@earendil-works/pi-coding-agent"` declaration via the leftmost-segment rule above (the leftmost segment of `"SessionShutdownEvent.reason"` is the literal `SessionShutdownEvent`, which equals the imported identifier exactly); the audit does NOT detect member-access references on category-(2) named imports (per *Target surface categories* above), so `SessionShutdownEvent.reason` member-access drift is gated by the brand-string assertion of [step 5](./version-bump-intro.md#pi-version-bump-procedure) (typecheck-time) rather than by category-(2) audit-time resolution. New category-(2) entry kinds coined per *Target surface categories* above MUST carry a `path` field whose leftmost dot-separated segment is the imported symbol identifier so they participate in this join key without a per-kind carve-out. Two `SDK_SURFACE_INVENTORY` entries MUST NOT share the same whole `path` value, and (for category-(2) audit-participating kinds) no two entries may both be bare leftmost segments — e.g. two entries with `path: "Foo"` collide and fire `literal-read/duplicate-path-segment` (or a category-specific variant the implementation coins, e.g. `literal-read/duplicate-leftmost-segment`), but two entries with `path: "Foo.bar"` and `path: "Foo.baz"` are legal multi-segment matches resolved per the lex-min tie-break below. For category-(1) and category-(3) audit-participating kinds, the same uniqueness rule applies symmetrically on the rightmost dot-separated segment (per *Category-(1)/(3) inventory join key* below; the parallel category-specific variant the implementation may coin is `literal-read/duplicate-rightmost-segment`). For legal multi-segment matches (e.g. `Foo.bar` and `Foo.baz` both resolving against `import { Foo }`, where neither entry is a bare `Foo`), the audit's `proposed-resolution` field reports the entry with the lexicographically-smallest `path` for determinism.

  <a id="audit-category-1-3-inventory-join-key"></a>*Category-(1)/(3) inventory join key (per-category specialisation).* The leftmost-segment rule above is the category-(2) named-import resolution rule and is preserved unchanged for that category. Categories (1) and (3) resolve by a per-category specialisation against the same `SDK_SURFACE_INVENTORY` source of truth, because the existing category-(1)/(3) entry shapes carry the carrier name as the leftmost dot-separated segment of the `path` field rather than the accessed member name — so a leftmost-segment match against the carrier-named segment cannot identify the accessed member, and the audit MUST instead match on the rightmost segment. Concretely: for category (1) `pi.<member>` accesses (carrier `pi: ExtensionAPI`), the audit resolves against `SDK_SURFACE_INVENTORY` by **exact match** between `<member>` and the **rightmost dot-separated segment** of an entry's `path` field — so an `SDK_SURFACE_INVENTORY` entry with `path: "pi.registerCommand"` (e.g. the existing `namespace-function` entries enumerated for the nine factory-probable function members) resolves against a `pi.registerCommand(...)` access by the rightmost segment `registerCommand` matching `<member>`. For category (3) `ctx.<member>` accesses (carrier `ctx: ExtensionContext` / `ctx: ExtensionCommandContext`), the audit resolves by **exact match** between `<member>` and the **rightmost dot-separated segment** of an entry's `path` field — so an entry with `path: "ExtensionContext.modelRegistry"` or `path: "ctx.modelRegistry"` resolves against a `ctx.modelRegistry(...)` access by the rightmost segment `modelRegistry` matching `<member>`. The category-(2) leftmost-segment rule above governs category-(2) named-import resolution exclusively and is unaffected by this specialisation; the rightmost-segment rule here governs category-(1) and category-(3) member-access resolution exclusively and does not perturb the category-(2) join key.
- <a id="audit-exemption-mechanism"></a>*Exemption mechanism.* A surface reference outside the inventory MAY be permitted only via a same-line comment marker on the offending source line of the form `// allow-pi-surface: <REQ-ID-or-PIC-anchor> <separator> <justification>`. The same-line-trailing-comment design and the leading `// allow-…:` token shape parallel the `// allow-broad-catch: <REQ-ID> — <spec-page>` and `// allow: <REQ-ID> — <spec-page>` grammars the parallel ESLint rules already pin, but the post-`:` payload deliberately differs in both positions: position 1 admits a PIC anchor as a peer of a REQ-ID (rather than as a deprecated transitional spec-page-anchor form), because PIC owns the inventory surface this audit guards and before REQ-ID anchors land there are no REQ-IDs to cite for PIC-owned surface; position 2 takes a free-form justification (rather than a spec-page reference), because the audit's per-discriminator routability obligation below requires the marker to explain *why* a specific surface is intentionally unenumerated, which a bare spec-page anchor cannot carry. Contributors expecting the cited two-position `<REQ-ID> — <spec-page>` shape MUST NOT carry it over verbatim. The marker MUST cite either a REQ-ID (once those anchors land) or, in the interim, a PIC anchor explaining why this specific surface is intentionally outside the inventory; an exemption marker without a citation or without a justification clause fails the audit. There is no separate `kind: "exemption"` entry on `SDK_SURFACE_INVENTORY` — the marker is the sole declaration site, mirroring the parallel ESLint rules' same-line-comment design. The exemption mechanism is scoped to families (1), (2), and (3) of the *Failure-surface contract* below — i.e., it MAY silence an unauthorised `pi.<member>` access, an unauthorised peer-package named import or an unauthorised typebox surface (a named import other than `Type` from `typebox`, or a `Type.<member>` member-access where `<member>` is not in the typebox member-access allow-list `{ Unsafe }`), or an unauthorised canonical-`ctx` member access, each of which names a specific out-of-inventory surface a marker can authorise. Family (4) (out-of-scope import/access shape) is **not** exemptible: the absolute prohibition in *Recognised import/access shapes* above forbids `import * as pi`, default/dynamic/side-effect-only peer imports, computed access, namespace destructuring, aliased or captured rebindings (including the `import { Foo as Bar } from "@earendil-works/<peer>"` / `import { Foo as Bar } from "typebox"` aliased-import specifier and the symmetric `export { Foo as Bar } from` re-export specifier from those same packages, on the same footing as value-binding aliased rebindings; the canonical name MUST be used unrenamed), off-canonical-name `ExtensionContext` / `ExtensionCommandContext` / `ExtensionAPI` parameters, off-canonical-annotation `ctx` parameters (a parameter named `ctx` whose explicit type annotation is not `ExtensionContext` or `ExtensionCommandContext`, including the unannotated form, the `ctx: any` form, the `ctx: unknown` form, and any other annotation), off-canonical-annotation `pi` parameters (a parameter named `pi` whose explicit type annotation is not `ExtensionAPI`, including the unannotated form, the `pi: any` form, the `pi: unknown` form, and any other annotation), `Object.assign({}, pi)`, `keyof typeof pi`, and barrel-module re-exports regardless of any same-line marker; a contributor confronted with such a shape MUST rewrite the source line into a recognised shape (per-symbol named import, `pi.<member>` static access, or canonical-`ctx` member access) rather than annotate it, per [step 2(b) branch (5)](./version-bump-step2b.md#bump-step-2b-rewrite-shape) below. An `// allow-pi-surface:` marker on a family (4) line MUST NOT suppress the family (4) discriminator and is itself an audit failure under that discriminator. Family (5) (stale or malformed exemption marker) is resolved by *Delete or rewrite the stale or malformed exemption marker* per [step 2(b) branch (4)](./version-bump-step2b.md#bump-step-2b-stale-rewrite) below, not by adding another marker. The audit MUST detect stale exemption markers (a marker on a line whose surface reference would now resolve to an inventory entry, or whose surface reference has been removed altogether) and surface them under their own failure-mode discriminator (per the *Stale-marker discriminator* sub-paragraph below, which decomposes "surface reference would now resolve to an inventory entry, or has been removed altogether" into the two stale sub-kinds (s1)/(s2) and pins marker scope, multi-line placement, the precedence with the underlying-surface discriminator, and the substitution soft-surface), so an exemption cannot silently outlive the surface it justified.

  *Separator tolerance.* The `<separator>` token between the citation and the justification MUST be either the em-dash `—` (U+2014) or the ASCII hyphen-minus `-` (U+002D), with at least one ASCII space character (U+0020) on each side; the audit MUST accept both separator characters identically (so a Windows / mixed-encoding edit that substitutes hyphen-minus for the em-dash is a recognised marker, not a malformed one). The audit MUST NOT accept any other character in the separator slot — exhaustively, the rejected set is every Unicode code point assigned to general category `Pd` (Dash punctuation) other than U+002D and U+2014, plus U+2212 MINUS SIGN (which is general category `Sm`, not `Pd`, and so MUST be enumerated explicitly); illustrative members of the rejected set include U+2010 HYPHEN, U+2012 FIGURE DASH, U+2013 EN DASH, U+2015 HORIZONTAL BAR, U+2212 MINUS SIGN, U+FE58 SMALL EM DASH, U+FE63 SMALL HYPHEN-MINUS, and U+FF0D FULLWIDTH HYPHEN-MINUS, but the closed boundary is the `Pd \ {U+002D, U+2014}` ∪ `{U+2212}` set above and not this illustrative listing. The audit MUST NOT accept the separator without at least one space on each side; markers using an unrecognised separator character or omitting the surrounding spaces are *malformed* per the malformed-marker discriminator below.

  *Minimum justification.* The `<justification>` slot MUST be at least four characters long after trimming leading and trailing ASCII whitespace, AND MUST contain at least one character that is neither ASCII whitespace nor an ASCII punctuation character (where the ASCII punctuation set is the 32 characters ``!"#$%&'()*+,-./:;<=>?@[\]^_`{|}~``). A justification that is empty after trimming, shorter than the four-character floor, or composed exclusively of whitespace-and-punctuation characters fails the audit as *malformed* per the discriminator below. The four-character floor and the alphanumeric-content requirement together rule out the trivially-empty (`—`-then-whitespace), pure-punctuation (`— ...`, `— ???`), and one-or-two-letter-tautology (`— x`, `— ok`) cases the routability obligation cannot meaningfully consume.

  *Citation-token grammar.* The `<REQ-ID-or-PIC-anchor>` slot MUST match exactly one of two syntactic shapes: (i) a REQ-ID of the form `REQ-<digits>` where `<digits>` is one or more ASCII decimal digits (regex `REQ-[0-9]+`); or (ii) a PIC anchor of the form `PIC#<kebab-case-anchor>` where `<kebab-case-anchor>` matches the regex `[a-z0-9]+(-[a-z0-9]+)*` (lowercase ASCII letters and digits joined by single ASCII hyphen-minus characters, with no leading or trailing `-` and no consecutive `--`, no whitespace, no other characters). A citation token whose literal form does not match either shape fails the *Malformed-marker discriminator* clause (b) below as *malformed*. The check is purely lexical: the audit MUST NOT additionally resolve the cited REQ-ID against any spec-side rule registry or the cited PIC anchor against any anchor registry, so a marker citing a syntactically valid but non-existent `REQ-9999` or `PIC#nonexistent-anchor` parses as well-formed and authorises the surface. The acknowledged soft surface this leaves — a syntactically valid but stale or fabricated citation that nevertheless authorises the surface — is mitigated per the *Stale-citation limitation* sub-bullet under the *Stale-marker discriminator* below, on the same footing as the *Substitution limitation* there; widening the audit to validate citation resolution against a registry is a PIC edit deferred past loom 1.0.

  *Stale-marker discriminator.* A *stale exemption marker* is a same-line `// allow-pi-surface:` marker that is well-formed (passes every clause of the *Malformed-marker discriminator* below) but does not authorise any currently out-of-inventory in-scope surface reference. The rule decomposes as follows.

  - <a id="audit-inventory-first-resolution"></a>*Inventory-first resolution.* The audit MUST resolve every recognised in-scope surface reference (per *Target surface categories* and *Recognised import/access shapes* above) against `SDK_SURFACE_INVENTORY` first; references that fail inventory resolution that originate from the `typebox` peer package then consult the typebox sibling allow-lists (per *Target surface categories* typebox carve-out); references that fail both then consult any same-line `// allow-pi-surface:` marker for authorisation. The three resolution paths are strictly ordered: (1) `SDK_SURFACE_INVENTORY` first; (2) the typebox sibling allow-lists (per *Target surface categories* typebox carve-out) for typebox-package references only; (3) `// allow-pi-surface:` markers last. A reference resolving at any earlier step short-circuits before consulting later steps — equivalently, marker authorisation MUST NOT be consulted for a reference that already resolves through the inventory or through a typebox sibling allow-list, the typebox sibling allow-lists MUST NOT be consulted for a reference that already resolves through the inventory, and neither the inventory nor the typebox sibling allow-lists MUST be widened to mask a marker. This precedence is what makes the (s2) sub-kind below mechanically detectable — a marker on a line whose every recognised in-scope reference resolves through the inventory or through a typebox sibling allow-list is observed by the audit as authorising nothing, since the earlier resolution paths short-circuit before the marker is consulted; a marker-first ordering would silently re-authorise the same references and (s2) detection would be impossible.

  - *Marker scope is per source line.* A marker on line N authorises every in-scope surface reference (per *Target surface categories* above) whose recognised import/access shape originates on line N — see *Multi-line surface placement* below for which physical line "originates on" picks out when a reference spans multiple lines. A line MAY carry multiple in-scope references and a single marker authorises all of them; the audit MUST NOT require one marker per reference, since the same-line trailing-comment grammar admits only one marker per line. Contributors MAY split a marked line into per-reference lines for clarity, but the audit MUST accept the unsplit form.

  - *Multi-line surface placement.* For an in-scope reference whose source-text spans multiple physical lines, the *originating line* — the line a marker MUST trail to authorise that reference, and equivalently the "offending surface line" the *Malformed-marker discriminator* below clause (e) names — is fixed by shape: (i) for a category-(1) `pi.<member>` member-access expression split across lines (e.g., `pi\n  .registerCommand(...)`), the originating line is the line of the property identifier (the `.registerCommand(...)` line), not the line of the `pi` namespace identifier; (ii) for a named-import declaration split across lines whose marker trails the `import`-keyword line, the originating line is the line of the `import` keyword and the marker authorises every imported symbol on the declaration; (iii) for a category-(3) member-access expression on the canonical `ctx` identifier split across lines (e.g., `ctx\n  .modelRegistry\n  .find(...)`), the originating line is the line of the `ctx` identifier itself, not the line of any property identifier in the chain — the asymmetry with (i) is deliberate and is the audit's recognised marker placement for category-(3) access (a contributor authorising a runtime-passed `ctx`-rooted chain MUST place the marker on the `ctx` line); (iv) for a named-import declaration split across lines whose marker trails an individual imported-symbol line rather than the `import`-keyword line, the originating line is the line of that imported symbol's identifier and the marker authorises only that symbol — per-symbol marker placement on multi-line named imports is an alternative the audit MUST accept alongside the (ii) import-keyword placement, so a contributor can authorise individual symbols selectively without authorising the whole declaration. A marker on any other line of the multi-line span — including a line of the dotted chain in (i) or (iii) that is neither the relevant property/namespace line, an intermediate property identifier line in a (iii) `ctx`-rooted chain, an interior comment line inside the dotted chain, or a non-symbol line of a multi-line named import (e.g. an opening-brace-only line, a closing-brace-only line, a comma-only line, or an interior comment line) — fails malformed-marker clause (e) and routes to family (5) under that clause's `<symptom>` token; the underlying out-of-inventory reference additionally fires its own family (1) / (2) / (3) discriminator on the same audit run per the malformed-marker dual-emission rule below.
