# Audit recognised shapes

- <a id="audit-recognised-shapes"></a>*Recognised import/access shapes.* The audit MUST detect at minimum: named-import declarations from each of the four `@earendil-works/*` peer packages and from `typebox` (including type-only `import type { ... }` declarations, and the inline-`type`-modifier form `import { type Foo, bar }` in which a single value-import declaration mixes type-only and value-only specifiers — the audit MUST treat each named specifier independently per its inline-`type` modifier, so `import { type Foo, bar }` is two specifiers (`Foo` as a type-only specifier, `bar` as a value-only specifier) and each is detected and resolved per category (2) per its modifier rather than by a uniform per-declaration verdict; the same per-specifier rule applies symmetrically to the inverse form `import type { Foo, type Bar }` where the declaration is type-only but an individual specifier carries a redundant inline `type` modifier). Package specifier matching MUST cover both the bare-package form (e.g. `@earendil-works/pi-coding-agent`) and any sub-path form (e.g. `@earendil-works/pi-coding-agent/dist/types`, `@earendil-works/pi-coding-agent/sub-export`); a named import is in scope of category (2) iff its specifier matches `@earendil-works/<peer>` exactly OR `@earendil-works/<peer>/<non-empty-sub-path>` for one of the four peer packages (and similarly for `typebox` / `typebox/<sub-path>`). Step 2(a) / step 5 deep-`dist/...`-path re-pins per the [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below therefore surface to category-(2) detection on the same footing as the bare-package form, and the same prohibitions later in this paragraph (`import * as`, default imports, dynamic `import()`, side-effect-only imports, aliased rebindings, barrel-module re-exports, etc.) apply to the sub-path form on the same footing as the bare-package form. The package-specifier-not-part-of-the-join-key rule pinned by [Category-(2) inventory join key](./audit-resolution.md#audit-category-2-inventory-join-key) below applies under this broader specifier set: a named import resolved under either the bare-package form or any sub-path form of the same peer package routes to the same `SDK_SURFACE_INVENTORY` entry by canonical name, with no per-(specifier-shape, name) entry duplication required. `pi.<member>` static member-access expressions; member-access expressions on the canonical `ctx` identifier as defined under *Target surface categories* (3) above; and `Type.<member>` member-access expressions on the canonical `Type` import from `typebox` as defined under the typebox carve-out of *Target surface categories* above. Re-exports of in-scope surface count as references at the re-exporting site — on the same footing as a direct named import — only for category (2), i.e. per-symbol `export { <name>[, ...] } from "@earendil-works/<peer>"` declarations naming one or more enumerated symbols; categories (1) (`pi.<member>`) and (3) (canonical-`ctx` member access) are not standalone bindings and so cannot be re-exported as such. The "re-exports through barrel modules" prohibition below applies under this operational definition: a module is a *barrel module* iff every top-level statement is one of: (i) an `export … from "@earendil-works/<peer>"` re-export of a peer-package surface (per-symbol named, never `export *` — see prohibition below); (ii) a comment; (iii) a directive prologue; (iv) an import statement whose binding is used by an (i) statement on the same module; (v) an `export type … from "@earendil-works/<peer>"` type-only re-export from a peer package. Any module with a top-level `export const`/`export function`/`export class`/locally-defined-symbol export, or a plain `export {}`, or a top-level `import` whose binding is unused, is NOT a barrel module under this definition (and may use per-symbol named re-exports per the rule above). Independently of the barrel-module shape, `export * from "@earendil-works/<peer>"` and `export * as ns from "@earendil-works/<peer>"` of any of the four peer packages are prohibited everywhere in the audit's scope (they launder the entire namespace through the re-exporting site without per-symbol audit; these shapes are forbidden because the audit's resolution rules — the canonical-`pi` and canonical-`ctx` carriers, the `SDK_SURFACE_INVENTORY` category-(2) join key on the canonical name per *Category-(2) inventory join key* below, and the `typebox` named-import allow-list keyed on `Type` and `Unsafe` per [*Target surface categories*](./audit-recognised-shapes.md#audit-target-surface-categories) above (typebox carve-out) — all consume the canonical name and would silently fail to resolve a renamed, aliased, or laundered reference even under the type-aware tooling variant the implementation MAY adopt below, so the prohibition is a behavioural rule on the source-text shape rather than a tooling-implementation constraint, and applies on the same footing to every other shape enumerated below); per-symbol `export { <name>[, ...] } from "@earendil-works/<peer>"` re-exports are permitted iff the re-exporting module is not a barrel module under the operational definition above. The same-line `// allow-pi-surface:` exemption marker on a re-export line covers only the surface re-exported by that single statement, not the surrounding module. Other syntactic shapes that fall under the same behavioural rule — `import * as pi`, default imports of any of the four peer packages, dynamic `import()` of any of the four peer packages, side-effect-only imports of any of the four peer packages, any CommonJS access path to one of the four `@earendil-works/*` peer packages or `typebox` — the runtime MUST be ES-modules-only at loom 1.0 — `package.json` MUST set `"type": "module"` as part of the extension's adoption commit, and the implementation's land-green precondition (per the introductory paragraph above) MUST verify both this `package.json#type` value and the absence of CJS-reach shapes in `src/` before the audit's first run on `main` — and CJS reach is unsupported, so this prohibition covers (i) `require("@earendil-works/<peer>")` and `require("@earendil-works/<peer>/<sub-path>")` (and `require("typebox")` / `require("typebox/<sub-path>")`), (ii) `createRequire(...)` plus `req("@earendil-works/<peer>")` (or `req("typebox")`) style indirection — defined as any function call whose call site uses `createRequire` imported from the Node `"module"` package and whose result is invoked with one of the in-scope package specifiers (bare-package or sub-path form, on the same footing as the named-import specifier-matching rule above); the prohibition holds regardless of whether `createRequire` is imported under its canonical name, under an aliased rebinding (e.g. `import { createRequire as cr } from "module"; cr(import.meta.url)("@earendil-works/<peer>")`), through a namespace import (e.g. `import * as M from "module"; M.createRequire(import.meta.url)("@earendil-works/<peer>")`), through a default import or default-namespace re-export of the `"module"` package, or through any other re-binding shape (further `as`-rename re-export, intermediate barrel module, captured local `const cr = createRequire`, etc.), and applies on the same footing to any call to a function whose source binding transitively resolves to the `createRequire` named export of the Node `"module"` package when invoked with one of the in-scope package specifiers — no such call MUST exist anywhere in the audited source tree, and the aliased-import / aliased-rebinding prohibitions enumerated later in this paragraph (which are scoped to the four `@earendil-works/*` peer packages and `typebox`) MUST NOT be read as authorising the `"module"`-package shapes above by omission — and (iii) any other CJS-style indirection reaching the four peer packages or `typebox`; the prohibition is in family (4) under its own discriminator on the same footing as the dynamic-`import()` and side-effect-only-import clauses immediately preceding, computed access (`pi[name]` or `ctx[name]`), namespace destructuring (`const { registerCommand } = pi`, `const { ui } = ctx`), aliased rebindings — both value-binding aliases (`const reg = pi.registerCommand`, `const c = ctx`, `const api = pi`) and aliased-import / aliased-re-export specifiers from any of the four `@earendil-works/*` peer packages or from `typebox` (`import { Foo as Bar } from "@earendil-works/<peer>"`, `import { Foo as Bar } from "typebox"`, and the symmetric re-export forms `export { Foo as Bar } from "@earendil-works/<peer>"` / `export { Foo as Bar } from "typebox"`; the `as`-rename specifier launders the canonical name on which `SDK_SURFACE_INVENTORY` category-(2) join-key resolution and the `typebox` named-import allow-list both depend, on the same footing as the value-binding aliases above — the canonical (source) name MUST be used unrenamed at every import and re-export site, and the inline-`type` modifier MAY of course still combine with the unrenamed specifier), captured rebindings (`this.ctx = ctx`, `self.ctx = ctx`, `this.pi = pi`, `self.pi = pi`, and — read broadly, not narrowly to inner-closure-captured bindings — any local variable, field, or closure-captured binding anywhere in the audited source tree whose initialiser is a reference to `ctx`, to `pi`, or to any descendant member-access expression rooted on either of those canonical carriers, e.g. `const cwd = ctx.cwd`, `const reg = pi.registerCommand`, `let savedCtx = ctx`, `this.foo = ctx.foo`; the broader reading is the normative one and the behavioural intent is that every `pi.<member>` and every `ctx.<member>` access reaches its read site through the canonical carrier — the typed `pi` parameter or the typed `ctx` parameter — so the static-AST audit can resolve every such access against `SDK_SURFACE_INVENTORY` without chasing data-flow through an intermediate binding), any function/method parameter declaration whose explicit type annotation is `ExtensionContext` or `ExtensionCommandContext` but whose name is not the literal identifier `ctx`, any function/method parameter declaration whose name is the literal identifier `ctx` but whose explicit type annotation is not the literal `ExtensionContext` or `ExtensionCommandContext` (including the absence of any explicit type annotation, an `any` annotation, an `unknown` annotation, or any other annotation than the two literals above — the literal `ctx` parameter name is reserved across the audited source tree for category-(3) carriers, so a parameter typed otherwise (e.g. `ctx: HttpRequestContext` for an HTTP handler that never touches Pi-side surface) MUST be named something other than `ctx` (e.g. `httpCtx`, `request`); the reservation is enforced unconditionally on the parameter name regardless of whether the parameter ever transports Pi-side surface, and the prohibition is in family (4) under its own discriminator on the same footing as the off-canonical-name clause immediately preceding), any function/method parameter declaration whose explicit type annotation is `ExtensionAPI` but whose name is not the literal identifier `pi`, any function/method parameter declaration whose name is the literal identifier `pi` but whose explicit type annotation is not the literal `ExtensionAPI` (including the absence of any explicit type annotation, an `any` annotation, an `unknown` annotation, or any other annotation than the literal above — the literal `pi` parameter name is reserved across the audited source tree for category-(1) carriers, so a parameter typed otherwise MUST be named something other than `pi`; the reservation is enforced unconditionally on the parameter name regardless of whether the parameter ever transports Pi-side surface, on the same footing as the canonical-name `ctx` reservation immediately preceding the off-canonical-name `pi` clause, and the prohibition is in family (4) under its own discriminator on the same footing as the off-canonical-name `pi` clause immediately preceding), any function/method parameter declaration whose binding shape in parameter position is a destructuring pattern (an object binding pattern such as `{ ui }` or an array binding pattern such as `[first]`, in either case in parameter position rather than a single named identifier) and whose explicit type annotation on that destructured parameter is the literal `ExtensionAPI`, `ExtensionContext`, or `ExtensionCommandContext` — the canonical carriers under *Target surface categories* (1) and (3) above are required to be a single named identifier (`pi` for `ExtensionAPI`, `ctx` for `ExtensionContext` / `ExtensionCommandContext`), so a destructured form such as `function handler({ ui }: ExtensionContext)` or `function factory({ registerCommand }: ExtensionAPI)` is prohibited even though no off-canonical identifier name appears (the destructured binding has no parameter-name identifier the off-canonical-name and off-canonical-annotation clauses above could key on); the prohibition is in family (4) under its own discriminator on the same footing as the off-canonical-name and off-canonical-annotation clauses immediately preceding, any function/method parameter declaration whose explicit type annotation is a wrapped, intersected, union-typed, generic-applied, or aliased form of `ExtensionAPI`, `ExtensionContext`, or `ExtensionCommandContext` rather than the bare textual literal — including but not limited to type-operator wrappings (`Readonly<ExtensionAPI>`, `Partial<ExtensionContext>`, `Required<ExtensionCommandContext>`, and any other built-in or user-defined type-operator wrapping), intersection forms (`ExtensionAPI & SomeMixin` and the symmetric `SomeMixin & ExtensionAPI`), union forms (`ExtensionAPI | SomeOther`), generic-applied forms (`ExtensionAPI<T>`, whether or not the SDK declares `ExtensionAPI` as generic at the loom 1.0 Pi-SDK pin), and aliased forms (`pi: API` where `API` is introduced by an `import type { ExtensionAPI as API } from "@earendil-works/pi-coding-agent"` rename specifier — itself separately prohibited under the aliased-import-specifier clause above — or by any local `type API = ExtensionAPI` alias declaration anywhere in the audited source tree) — only the three bare textual literals `ExtensionAPI`, `ExtensionContext`, and `ExtensionCommandContext` are permitted as the canonical carrier annotation, since the carrier rule under *Target surface categories* (1) and (3) above resolves the annotation textually (per the *resolved against the textual annotation rather than against an inferred type* clauses there) and a wrapped, intersected, union-typed, generic-applied, or aliased annotation would silently bypass that textual resolution; the prohibition is in family (4) under its own discriminator on the same footing as the off-canonical-annotation clauses immediately preceding, any function/method parameter declaration whose explicit type annotation is a type-parameter identifier whose declared constraint is `ExtensionAPI`, `ExtensionContext`, or `ExtensionCommandContext` (e.g. `function makeHandler<C extends ExtensionContext>(c: C)`, `function init<P extends ExtensionAPI>(p: P)`, `function wrap<T extends ExtensionCommandContext>(t: T)`, including the symmetric class- and method-level forms `class Foo<C extends ExtensionContext> { m(c: C) {} }` and the equivalent arrow-function and method-shorthand declarations) — only the three bare textual literals `ExtensionAPI`, `ExtensionContext`, and `ExtensionCommandContext` are permitted as the canonical carrier annotation, since the parameter's textual annotation in this generic-laundering shape is the type-parameter identifier (e.g. `C`, `P`, `T`) rather than one of the three carrier literals and neither the carrier-detection rules under *Target surface categories* (1) and (3) above (which key on the bare textual annotation being the literal `ExtensionAPI` for category (1) or the literal `ExtensionContext` / `ExtensionCommandContext` for category (3)) nor the off-canonical-annotation clauses immediately preceding (which key on a parameter named `pi` or `ctx` whose textual annotation is some non-canonical-literal annotation, and which therefore do not fire when the parameter name is anything other than `pi` / `ctx` — e.g. `c`, `t`, `arg`) would fire on it, so the binding would silently transport the full Pi-side surface and category-(1)/(3) audit coverage of every member access reached through that binding would be silently lost; the prohibition is in family (4) under its own discriminator on the same footing as the off-canonical-annotation clauses immediately preceding, any TypeScript declaration anywhere in the audited source tree that creates a subtype of `ExtensionAPI`, `ExtensionContext`, or `ExtensionCommandContext` by naming one of those three carrier literals in `extends`/`implements`/`&` position — `interface MySubAPI extends ExtensionAPI { … }`, `interface MyCtx extends ExtensionContext { … }`, `interface MyCmdCtx extends ExtensionCommandContext { … }`, `class MyHandler implements ExtensionCommandContext { … }`, `class MyApi implements ExtensionAPI { … }`, `type MyApi = ExtensionAPI & SomeMixin` (and the symmetric `type MyApi = SomeMixin & ExtensionAPI` and the equivalent `interface MyApi extends ExtensionAPI, SomeMixin {}` form), and any other subtype-creation form that names one of the three carrier literals in an `extends`/`implements`/`&` position — and any subsequent function/method parameter declaration that consumes that subtype as its explicit type annotation (e.g. `function handler(myCtx: MyCtx) { myCtx.modelRegistry.… }`, `function init(api: MySubAPI) { api.registerCommand(…) }`) is prohibited even though no off-canonical identifier name appears on a parameter typed with one of the three carrier literals; the canonical-carrier rule under *Target surface categories* (1) and (3) above resolves the parameter's annotation textually against the literal `ExtensionAPI` / `ExtensionContext` / `ExtensionCommandContext`, so a subtype-named annotation (e.g. `MyCtx`) silently bypasses the carrier-detection branch (the off-canonical-name clauses fire only when the parameter's textual annotation is one of the three carrier literals; the off-canonical-annotation clauses fire only when the parameter's name is `pi` or `ctx`; a parameter named `myCtx` typed `MyCtx` matches neither), so the binding would silently transport Pi-side surface past the canonical-carrier rule; the prohibition is on the subtype-creation declarations themselves rather than on the downstream parameter annotations (since the declaration is the unique site the audit can syntactically detect by `extends`/`implements`/`&` position over one of the three carrier literals, and removing the declaration removes every downstream consumer in one edit), and the existing canonical declarations of the three carrier types in `@earendil-works/pi-coding-agent` itself are out of scope (they live outside `src/` per *Audit scope* below); the prohibition is in family (4) under its own discriminator on the same footing as the off-canonical-annotation clauses immediately preceding, any class field declaration, class property declaration, top-level `const` / `let` / `var` binding, or class-property initialiser anywhere in the audited source tree whose explicit type annotation is the literal `ExtensionAPI`, `ExtensionContext`, or `ExtensionCommandContext` (e.g. `class Loom { private api: ExtensionAPI; … }`, `class Loom { private ctx: ExtensionContext; … }`, `const sharedCtx: ExtensionContext = makeCtx();`, `let cachedApi: ExtensionAPI = …;`) — the canonical carriers under *Target surface categories* (1) and (3) above are pinned syntactically as a function or method parameter declaration whose name is the literal identifier `pi` (for `ExtensionAPI`) or `ctx` (for `ExtensionContext` / `ExtensionCommandContext`), so a non-parameter binding (a class field, a top-level binding, or a class-property initialiser) carrying one of the three carrier literals as its explicit type annotation transports Pi-side surface past the parameter-position carrier-detection branch and past the off-canonical-name and off-canonical-annotation clauses immediately preceding (which key on parameter position and the literal `pi` / `ctx` parameter-name reservation), so any `pi.<member>` or `ctx.<member>` access reached through the non-parameter binding would be silently lost from inventory-closure coverage; Pi-side surface MUST flow through the canonical `pi` / `ctx` parameter carriers per *Target surface categories* (1) / (3), and constructor or method parameters that initialise such a field by the captured-rebinding rule (`this.api = pi`, `this.ctx = ctx`) are already prohibited under the captured-rebindings clause above — this clause closes the gap for non-parameter bindings whose annotation sources the carrier type from a non-canonical path (e.g. a factory return value, a module-scope constant, an injected handle); the prohibition is on the binding-declaration site itself rather than on the downstream member-access expressions (since the declaration is the unique site the audit can syntactically detect by the textual `ExtensionAPI` / `ExtensionContext` / `ExtensionCommandContext` annotation outside parameter position, and removing the declaration removes every downstream consumer in one edit); the prohibition is in family (4) under its own discriminator on the same footing as the off-canonical-annotation clauses immediately preceding, `Object.assign({}, pi)`, `keyof typeof pi`, and re-exports through barrel modules — are out of scope of inventory-resolution and MUST NOT be used on the `pi` namespace, on the canonical `ctx` identifier, or against any of the four peer packages anywhere in the audit's scope. The audit itself MUST enforce that prohibition under its own discriminator (see *Failure-surface contract* below) so the `pi.<member>` / `ctx.<member>` surface remains exhaustively coverable by syntactic recognition without committing PIC to a TypeScript-program-load tooling choice. The implementation MAY use a type-aware variant (e.g. a TypeScript-program symbol-table walk) provided the canonical-`pi`-carrier and canonical-`ctx`-carrier rules and the prohibition above continue to be enforced with the same per-discriminator routing — the audit's behavioural contract is invariant under tooling choice; what changes is only the implementation.
