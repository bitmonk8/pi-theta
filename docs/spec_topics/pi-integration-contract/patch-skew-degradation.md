# Patch skew degradation

   <a id="patch-skew-degradation-contract"></a>
   **Patch-skew degradation contract.** The *patch-skew window* is the interval during which a user's installed `@earendil-works/pi-coding-agent` patch is newer than the loom patch on the same machine — the [loom 1.0 Pi-SDK pin range](./host-prerequisites.md#pi-sdk-pin) permits patch-only Pi updates without re-running [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below, so the Pi-side `SessionShutdownEvent['reason']` union and the loom-side `literals` snapshot can transiently disagree. During the window, the runtime fallback above (the `loom/host/session-shutdown-reason-unknown` diagnostic and the full five-sub-step teardown for any reason not in the snapshot's `literals`) is the **sole** user-facing contract for any new reason; no other surface promises observability or graceful degradation. The skew is detected by the bidirectional type-equality assertion below firing red on the next loom typecheck cycle that imports the candidate Pi patch — loom does **not** run an automated cadence (no Renovate, no scheduled CI re-install, no nightly cron) that proactively re-imports patches in loom 1.0, so the build-time gate is best-effort: the window's duration depends on contributor cadence and is not bounded by spec, and it detects the skew only as fast as a loom contributor next runs `npm run typecheck` against the candidate patch (the window itself closes only when a release shipping the resulting snapshot edit reaches the user; an automated cadence for re-importing Pi patches is deferred to [Future Considerations — Tooling deferrals](../future-considerations.md#tooling-deferrals-no-v1-impact)). The window is silent on the user side until a non-snapshot reason actually arrives at runtime (routed through the unknown-reason path above) or the contributor's next typecheck cycle catches the drift; this is intentional loom 1.0 behaviour. The contract is asymmetric across widening and narrowing of `SessionShutdownEvent['reason']`, with the mixed widening+narrowing case handled by the **Mixed** sub-rule below. **Widening** (a Pi patch adds a new literal): every shutdown carrying the new literal routes through the unknown-reason path above, with `details.observed` carrying the `String(event.reason)` coercion of the new literal so operator triage identifies the cause without a code change; the full five-sub-step teardown still runs and the snapshot edit step 5 below prescribes is the loom-side resolution. **Narrowing** (a Pi patch removes a literal): the snapshot's `literals` array is transiently superset to the SDK union, but Pi never delivers the removed literal at runtime, so the stale snapshot entry is harmless — no runtime regression, no diagnostic, no observable behaviour change. The build-time bidirectional assertion fires red on the next typecheck (the SDK-union side becomes the strict subset and the right-to-left direction breaks); the only required follow-up is the snapshot-edit-only sweep under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below (which step 1's carve-out keeps prose-update-free). **Mixed** (a Pi patch both adds and removes literals in the same release — rename, replace, or swap): the patch decomposes into its widening half (the added literal is treated per the **Widening** rule above — runtime routes the new literal through the unknown-reason path with `details.observed` carrying the coerced value) and its narrowing half (the removed literal is treated per the **Narrowing** rule above — the snapshot's `literals` array is transiently superset and the stale entry is harmless until the snapshot edit ships); both halves are observed and handled independently, and no composite handling is owed. Three behaviours pin this contract and MUST be covered by build-time fixtures; the (P-a)/(P-b)/(P-c) labelling here uses a `"P-"` prefix to be deliberately distinct from both the (i)/(ii)/(iii) sub-anchor labelling of the *Sub-anchors and pinned substrings* paragraph immediately above **and** the (a)–(d) anchor-stable substring labelling of that same paragraph (the substring labels are anchor-stable surface and cannot themselves be relabelled), so the three adjacent enumerations cannot collide on cross-reference: (P-a) an *unknown-reason runtime* fixture that synthesises a `session_shutdown` event whose `reason` is not a snapshot `literals` member and asserts the handler emits exactly one `loom/host/session-shutdown-reason-unknown` with `details.observed` carrying the coerced literal, then runs the full five-sub-step teardown; (P-b) a *snapshot-only widening* fixture pair that constructs a synthetic value with the same shape as the production `SDK_SURFACE_INVENTORY`'s `SessionShutdownEvent['reason']` snapshot entry — same composite-predicate keys (`kind: "type-union-snapshot"`, `path: "SessionShutdownEvent.reason"`) and same `literals: [...] as const` shape — whose `literals` array contains one extra string `"synthetic-new-reason"` not present in the production constant (no mutation of the production `SDK_SURFACE_INVENTORY` import — the `as const` declaration on the production constant remains intact and is not written through), and exercises the runtime's set-membership check against that synthetic value through a *test-only* substitution of the production `SDK_SURFACE_INVENTORY` module export — the substitution mechanic MUST preserve the handler's static import of `SDK_SURFACE_INVENTORY` from the source-of-truth module (sub-anchor (ii) of the **Unknown-reason rule** above); the choice of primitive that achieves that preservation is an implementation choice. PIC pins only the static-import-preservation behavioural property above and the behavioural assertions below; the synthetic constant's exact source is an implementation choice. The fixture pair asserts both directions: **(P-b-1, positive)** with the substitution applied so the handler's static import resolves to the synthetic constant, a `session_shutdown` event with `reason: "synthetic-new-reason"` is accepted as a closed-set member — **no** `loom/host/session-shutdown-reason-unknown` is emitted, the full five-sub-step teardown still runs, and no other failure-mode diagnostic from the **Unknown-reason rule** above fires; **(P-b-2, negative)** a parallel test in which the substitution is **not** applied (so the handler's static import resolves to the unmodified production `SDK_SURFACE_INVENTORY`, whose `literals` array does not contain `"synthetic-new-reason"`) MUST emit exactly one `loom/host/session-shutdown-reason-unknown` for the same `reason: "synthetic-new-reason"` event. The differential between (P-b-1) and (P-b-2) — same input event, opposite outcomes, with the only varying factor being whether the substitution is applied — proves the handler reads through the live module export rather than a hardcoded copy of the snapshot's `literals` field (a hardcoded copy would not swing on the substitution state and the two halves would observe the same outcome). The pair together — and only together — proves the single-source-of-truth wiring above; (P-c) a *narrowing no-regression* fixture pair that proves the asymmetric-skew property — runtime tolerance plus build-time gating — across two separate test phases that cannot share a single process invocation (the runtime-acceptance assertion requires compiled code to run, while the build-time-red assertion requires `tsc` to fail, and a `tsc` failure aborts `npm run typecheck` before any vitest harness starts; no single-process harness can assert both, so the obligation is split into two sub-fixtures with the same logical pairing pattern as (P-b-1)/(P-b-2)). **(P-c-1, runtime)** constructs a synthetic snapshot value with the same shape as the production `SDK_SURFACE_INVENTORY`'s `SessionShutdownEvent['reason']` snapshot entry whose `literals` array is a strict superset of the SDK union (e.g., the SDK has narrowed to drop `"fork"` but the synthetic snapshot still pins it alongside the rest of the narrowed union), substitutes it through the same static-import-preserving substitution mechanic (P-b) uses, dispatches a `session_shutdown` event with `reason: "fork"` (the previously-snapshotted literal that the synthetic SDK union no longer contains), and asserts the runtime accepts it as a closed-set member — **no** `loom/host/session-shutdown-reason-unknown` is emitted, the full five-sub-step teardown still runs, and no other failure-mode diagnostic from the **Unknown-reason rule** above fires. **(P-c-2, build-time)** is a separate test (one whose harness is the `tsc` invocation itself, not a vitest case) that exercises the bidirectional type-equality assertion of [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below against a synthetic SDK type one literal narrower than the synthetic snapshot, and asserts the assertion produces a `tsc` diagnostic carrying the literal brand string `loom/typecheck/session-shutdown-reason-snapshot` verbatim. The two sub-fixtures together — and only together — prove the asymmetric-skew property the **Narrowing** paragraph above pins. The fixtures are normative-by-reference here; the implementation owns their `<file>` placement and exact assertion shape.

   *Fixture-obligation categories.* This contract and the [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below together pin four distinct *categories* of fixture obligation the implementation MUST cover — the headline count "four" refers to categories, not individual fixtures, since each category may comprise multiple sub-fixtures. The categories are: the three *patch-skew* fixture-obligation categories named immediately above (the *unknown-reason runtime* category, the *snapshot-only widening* category whose (P-b-1, positive) and (P-b-2, negative) sub-fixtures together comprise it, and the *narrowing no-regression* category whose (P-c-1, runtime) and (P-c-2, build-time) sub-fixtures likewise together comprise it), and the *per-sub-trigger negative-test fixtures* category named in [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below (one sub-fixture per Step-0-probe-constant failing entry under [trigger (i)](./version-bump-triggers.md#bump-step-5-trigger-i), and one sub-fixture per `SessionShutdownEvent['reason']` snapshot-entry sub-trigger under [trigger (ii)](./version-bump-triggers.md#bump-step-5-trigger-ii)). PIC does not enumerate concrete `<file>` placements or assertion shapes; those are implementation choices.

   The handler runs in fixed order regardless of `event.reason`:

   1. **Stop accepting new work.** <a id="substep-1-drain-flag"></a>Mark the extension-scoped `LoomRegistry` as drained <a id="substep-1-init-drain-state-tag"></a>and initialise its drain-state tag to `"shutting-down"`. <a id="substep-1-act-order"></a>The two acts run in fixed normative order — `LoomRegistry.drain()` first, then `LoomRegistry.initDrainStateTag()` — so downstream sites that key on which act threw and which succeeded (the **Per-step isolation** paragraph below, which routes the two throws to distinct `(code, details.step, details.call)` dedup buckets; the [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause's enumeration of the prior-state combinations a follow-up `session_shutdown` may observe; the `LoomRegistry.initDrainStateTag` method-contract clause below that pins it as called *immediately after* `LoomRegistry.drain`; and any acceptance criterion that distinguishes between the first and second sub-step-1 act) can refer to "the second sub-step-1 act" unambiguously; the per-call **Per-step isolation** rule continues to apply independently to each of the two calls regardless of the order pin. On entry into a drained registry the slash-command `handler` registered in step 3 returns a `loom-system-note` (`display: true` — the `display` setting is fixed by this umbrella opener and applies uniformly to **both** arms below, so the `"degraded; /reload to recover"` note is operator-visible on the same footing as the `"shutting down"` note; only the `content` differs across arms) — without dispatching the loom — whose `content` is selected from the closed two-arm drain-state tag set: <a id="substep-1-shutting-down-arm"></a>the `"shutting-down"` arm (the default set here, used for every shutdown whose runtime the host will tear down regardless — `"quit"`, `"reload"`, and the in-flight teardown window of any branch up to the post-sub-step-4 transition below — including the snapshot-failure path's own in-flight window before that interstitial transition runs; the *in-flight teardown window* is the interval from `session_shutdown` handler entry up to (but not including) the post-sub-step-4 `"shutting-down" → "degraded-needs-reload"` tag transition pinned in the `reason: "new" | "resume" | "fork"` edge-case bullet below, and is enumerated here only for closed-set completeness — so every transient state of the drain-state tag during the handler is accounted for by exactly one arm of this two-arm set — and (scoped to that *in-flight teardown window only*, **not** to the `"shutting-down"` arm as a whole) is in practice unreachable by a slash invocation under the loom 1.0 Pi SDK pin, because Pi's per-session slash-handler serialisation pinned under **Tool-registration lifetime and visibility** below means no slash invocation can run concurrently with the `session_shutdown` handler on the same session; the runtime MAY therefore assume the in-flight window is unreachable and need not implement additional reader-side defence beyond the reader behaviour pinned in this arm. Post-handler slash invocations on the `"shutting-down"` arm — including the seam-tag-write-throw recovery path of branch (k) under *Acceptance criteria* below, where the post-sub-step-4 `LoomRegistry.markRuntimeDegraded` write was swallowed and the registry retains its `"shutting-down"` tag value — are reachable and MUST be serviced normally per the slash-handler short-circuit rule (returning the `"loom /<name>: extension shutting down"` system note); the unreachability clause above does **not** extend to them) emits the cancelled-binder-style content `"loom /<name>: extension shutting down"` shared with binder cancellation; <a id="substep-1-unset-tag-fallback"></a>the slash-handler reader MUST also treat an *unset* drain-state tag (a runtime *state* of the field, distinct from the closed two-arm *value* set above — the value set admits only the two literals `"shutting-down"` and `"degraded-needs-reload"` in loom 1.0.0, but the runtime state of the field may additionally be unset, reachable when sub-step 1's `LoomRegistry.initDrainStateTag` per-call site threw and was caught by the **Per-step isolation** paragraph below — leaving the registry drained but tag-less) as the `"shutting-down"` arm and emit the same legacy template, so a tag-init throw degrades safely to the cancelled-binder-style content rather than producing undefined slash-handler short-circuit behaviour. <a id="substep-1-degraded-needs-reload-arm"></a>The `"degraded-needs-reload"` arm (written **only** by the post-sub-step-4 transition pinned in the `reason: "new" | "resume" | "fork"` edge-case bullet's *Predicate split* clause below, on the branches that clause's broader tag-transition predicate `capturedEventReason ∉ {"quit", "reload"}` covers — i.e. the session-only `{"new", "resume", "fork"}` triplet, **and** the snapshot-failure path, **and** the unknown-reason path with a successful snapshot read (the latter two paths absorbed by the complement check via the `"<unreadable>"` sentinel and the off-snapshot `String(event.reason)` coercion respectively, even though the `loom/host/session-shutdown-runtime-degraded` *diagnostic emission* is suppressed on both per the same clause's narrower diagnostic-emission predicate) — where the runtime remains live but permanently degraded until operator-issued `/reload`) emits the distinct content `"loom /<name>: extension degraded; /reload to recover"`. The tag *value* set is closed at these two literals in loom 1.0.0 (no third literal value beyond `"shutting-down"` and `"degraded-needs-reload"` may be written to the field without a spec edit); the unset runtime state called out above is an implementation-side artefact of the **Per-step isolation** swallow rule, not a third literal value, and the slash-handler reader's unset → `"shutting-down"` mapping above ensures the closed value set is preserved in the observable behaviour of the slash note. The two `content` templates are deliberately disjoint at the substrings `"shutting down"` and `"degraded"` so operator tooling that grepped the legacy single-template surface MUST extend its match to both substrings, and so an operator on a stdio-capturing host (where the best-effort `console.error` visibility of `loom/host/session-shutdown-runtime-degraded` may be lost) still has an in-band signal — the `"degraded; /reload to recover"` substring — telling them to issue `/reload` rather than treating repeated `"shutting down"` notes as a transient teardown.
   2. **Cancel in-flight invocations.** For every entry in the **`ActiveInvocationRegistry`** (introduced below), first stamp the handler-scoped captured `event.reason` string (per the **Unknown-reason rule** above — the closed-set member, the `String(event.reason)` unknown-reason coercion, the throwing-access `"<unreadable>"`, or the snapshot-failure `"<unreadable>"`) onto the entry's `shutdownReason` field, then call `loomAbort.abort(reason)` where `reason` is the synthesised teardown value pinned by [Cancellation — Abort-reason propagation](../cancellation.md). The stamp MUST happen before `loomAbort.abort()` on the same entry so the per-invocation `finally` (which is what reads `entry.shutdownReason` per the **Per-invocation operator visibility (clean-cancel path)** rule below) observes a populated field by the time the abort propagates and the `finally` runs; the per-entry `try`/`catch` (per the registry contract below) wraps both the stamp and the `abort()` call — on a stamp throw (frozen entry, proxied entry, write-rejecting setter, OOM during property assignment) the per-entry `catch` arm proceeds to the next entry (the `abort()` for this entry is skipped, since the stamp failed) so a per-entry write failure cannot escape sub-step 2's loop and defeat the no-throw-escapes-the-handler invariant; on an `abort()` throw, behaviour is unchanged from the registry-contract rule below. No diagnostic is emitted on a stamp-throw catch, symmetric to the existing `abort()`-throw rule. The `abort()` call propagates through the [`Checkpoint` seam](./host-interfaces-core.md#checkpoint-seam) and, in subagent mode, through the one-shot `loomAbort.signal` listener that the runtime registered at spawn time and that calls `AgentSession.abort()` on the spawned session — per [Cancellation — Forwarding into `loomAbort`](../cancellation.md). The listener is detached in the same per-invocation `finally` that calls `AgentSession.dispose()` (see **Subagent session lifecycle** below).
   3. **Await subagent disposal.** `await Promise.allSettled(activeInvocations.map(inv => inv.disposeBarrier))`, where `disposeBarrier` is the promise that the **Subagent session lifecycle** `finally` block below settles after `AgentSession.dispose()` returns (subagent mode) or settles immediately (prompt mode). The await is bounded by the constant `SHUTDOWN_AWAIT_CAP_MS = 2000` measured against the runtime's injected [`Clock`](./host-interfaces-services.md#clock--fakeclock-interface) seam: at handler entry the runtime captures `deadline = Clock.now() + SHUTDOWN_AWAIT_CAP_MS`, and the await resolves on the earlier of `Promise.allSettled(...)` settling or `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, ...)` firing. The deadline is absolute, not a refreshing slide — a slow `loomAbort.abort()` propagation in sub-step 2 does not extend the await. The pending `Clock.setTimeout` handle is cleared on the success path so a reload that completes early does not leak a timer onto the about-to-be-invalidated runtime. On timeout the runtime emits one `loom/runtime/reload-teardown-timeout` diagnostic naming the still-in-flight invocations and proceeds to step 4. The `SHUTDOWN_AWAIT_CAP_MS` constant lives in the same source-of-truth pinned-constants block as the Step 0 capability-probe constants so the build-time literal-read assertions can assert it.
   4. **Close watchers.** Call `discoveryWatcher.close()` (the chokidar watcher established in step 5 below) and `settingsWatcher.close()` (the settings-file watcher per [Discovery — Settings file reads](../discovery/package-and-settings.md#settings-file-reads)); cancel any pending debounce timer handles via `Clock.clearTimeout`.
   5. **Detach forwarding listeners.** Remove every listener the runtime attached to Pi-side `ctx.signal`, tool `execute(signal)`, and parent-`invoke` signals — the same listeners [Cancellation](../cancellation.md) requires the per-invocation `finally` to remove, duplicated here for the reload-during-invocation path where Pi tears the runtime down before that `finally` ever reaches its forwarding-listener cleanup.

   **Per-step isolation.** Sub-steps 1, 4, and 5 each run inside their own `try`/`catch`; a throw from any of them MUST be caught, MUST emit exactly one `loom/host/session-shutdown-teardown-step-failed` (W, runtime) diagnostic via `console.error` with `details: { step: 1 | 4 | 5, call: <static call-site label>, error: <error.message> }`, and MUST NOT prevent the remaining sub-steps from running. Within sub-step 1 the rule applies independently to each of `LoomRegistry.drain` (the drain-flag set) and `LoomRegistry.initDrainStateTag` (the `"shutting-down"` tag-init write — the two acts sub-step 1 now performs per the prose above, two potential emissions, one per failing call); a drain-flag throw and a tag-init throw therefore occupy distinct `(code, details.step, details.call)` dedup buckets in operator logs and a future addition of a third sub-step-1 act MUST be added here on the same footing rather than collapsed into one of the existing labels. Within sub-step 4 the rule applies independently to each of `discoveryWatcher.close()`, `settingsWatcher.close()`, and the debounce-handle `Clock.clearTimeout(...)` call (three potential emissions, one per failing call). Within sub-step 5 the rule applies independently to each enumerated listener-detach call. Sub-steps 2 and 3 retain their existing per-entry isolation rules (per-entry `try`/`catch` around `loomAbort.abort()`; `Promise.allSettled` over `disposeBarrier`s) and do not emit `teardown-step-failed`. <a id="interstitial-seam-tag-write-swallow"></a>The **interstitial seam between sub-steps 4 and 5** — the `"shutting-down" → "degraded-needs-reload"` drain-state tag write pinned in the `reason: "new" | "resume" | "fork"` edge-case bullet below — is similarly wrapped in its own dedicated `try`/`catch`: a throw from the tag write (frozen `LoomRegistry`, proxied setter, OOM during property assignment) MUST be caught and silently swallowed, MUST NOT, by itself, suppress the subsequent `loom/host/session-shutdown-runtime-degraded` emission whose own diagnostic-emission predicate licenses it on this branch, nor prevent sub-step 5 from running, and MUST NOT emit `teardown-step-failed` or any other diagnostic. The trade-off this swallow accepts is that the degraded-arm `"loom /<name>: extension degraded; /reload to recover"` slash note will not fire on subsequent invocations of this extension instance (the registry tag retains its `"shutting-down"` value, so subsequent slash invocations short-circuit on the `"shutting-down"` arm instead); the rest of the handler — the emission and sub-step 5 — completes regardless. This handler-internal teardown-bookkeeping swallow is **not** a relaxation of the [Diagnostics — *Persistent diagnostics*](../diagnostics.md) "no ceiling fails silently" closure: that closure scopes to the runtime diagnostic-delivery ceiling (the defended `console.error` sink the **Diagnostic-emission isolation** rule below wraps), not to handler-internal state-bookkeeping writes — the tag write has no diagnostic of its own, and its failure mode's downstream effect (the degraded-arm slash-note suppression noted in the trade-off sentence above) is documented prose rather than a silent backstop failure of a delivery sink. The emission's own [**Diagnostic-emission isolation**](./diagnostic-emission-isolation.md#diagnostic-emission-isolation) wrap continues to cover the `console.error` that immediately follows. The static call-site label in `details.call` is a fixed string per call site, drawn from a **closed normative set per `(code, details.step)`** so operator dedup on `(code, details.step, details.call)` is meaningful both across runs and across independent implementations of this contract — the labels are wire contract, not implementation-chosen stable-per-impl strings. The closed set is: sub-step 1: `"loomRegistry.drain"`, `"loomRegistry.initDrainStateTag"`; sub-step 4: `"discoveryWatcher.close"`, `"settingsWatcher.close"`, `"Clock.clearTimeout(debounce)"`; sub-step 5: `"ctx.signal.removeEventListener"`, `"toolSignal.removeEventListener"`, `"parentInvokeSignal.removeEventListener"` (one label per inbound Pi-side signal source sub-step 5 enumerates by name — Pi-side `ctx.signal`, the tool-exposed adapter's `signal` parameter, and the `invoke`-parent signal respectively). Adding a new call site under any of these sub-steps requires a spec edit to extend this closed set in the same commit; an implementation MUST NOT introduce a `details.call` value outside this enumeration. The interstitial seam's tag write per the **`LoomRegistry` drain-state contract** below carries the fixed label `"loomRegistry.markRuntimeDegraded"` for prose cross-reference only and is not part of the closed set above, because no `details.call` is emitted for it: the seam's own dedicated `try`/`catch` swallows silently per the rule above. When the captured throw is not an `Error` instance, `details.error` is `String(error)`, falling back to the literal string `"<unreadable>"` if that coercion itself throws — symmetric with the `details.observed` convention pinned by the **Unknown-reason rule** above. The `console.error` call this paragraph mandates is itself wrapped per the **Diagnostic-emission isolation** rule defined under *Edge cases* below; a throw from `console.error` MUST be swallowed and MUST NOT prevent the remaining sub-steps from running. The new code is **not** added to the always-log set defined under **Runtime event channel** below, matching the existing `reload-teardown-timeout` and `session-shutdown-reason-unknown` exclusions.
