# Subagent

**Conversation drive — subagent mode.** The theta interpreter stays in the parent process; a subagent-mode invocation drives its conversation through a **spawned child `pi` process** rather than an in-process `AgentSession`. One child process is launched per subagent-mode invocation, at the point the pre-spawn model guard ([PIC-40](#pic-40) below) admits the invocation. Queries, completion events, cancellation, and teardown travel over Pi's documented RPC JSONL protocol (`docs/rpc.md` in `@earendil-works/pi-coding-agent`, the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above), spoken on the child's stdin/stdout, instead of in-process method calls. The child is launched as:

```
pi --mode rpc --no-session
   --system-prompt <resolved system:>
   --tools <name1,name2,…> | --no-tools
   --provider <p> --model <id>
   --no-skills --no-prompt-templates --no-themes --no-context-files
   [--approve | --no-approve]
```

The child runs Pi's full startup extension discovery, so extension-registered tools (`finding_store`, `projection`, …) are natively registered in the child and are reachable by the subagent's **model** through the `--tools` allowlist. No executable tool definition crosses any process boundary. The Pi surface consumed here — the CLI flags above, the RPC JSONL framing, and the `agent_end` event shape — is verified against the installed Pi (`pi --help` and `docs/rpc.md`) at v0.80.10 and re-audited on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below.

<a id="subagent-executable-resolution"></a>
**Executable resolution.** The runtime resolves the child executable by re-launching the running process from its own invocation, via a two-rung ladder — the mechanism proven by pi-config's `subagent` extension (`getPiInvocation()`):

1. **Entry-script rung.** When `process.argv[1]` names an existing file, spawn `process.execPath` (the Node/Bun binary) with that script as its first argument. The child is the exact same binary + entry script as the parent.
2. **Compiled-binary rung.** When `process.argv[1]` is unusable and `process.execPath` is not a generic runtime (`node` / `bun`), Pi itself is the executable — spawn `process.execPath` directly.

There is **no `PATH` fallback** — a deliberate divergence from the prior art. A `PATH`-resolved `pi` may be a different version or install than the parent, silently violating the pinned wire contract; a spec-governed language construct cannot tolerate that risk. If both rungs fail, resolution fails **closed at load time**: the theta does not register and the diagnostic (`theta/load/subagent-executable-unresolved`, see [Diagnostics — `theta/load/*`](../diagnostics/code-registry-load.md)) names the reason. The rung-1 existence check runs at **probe time** (see [Step 0 — Capability probe](./capability-probe.md#entry-capability-probe) above), not at first spawn. Parent `execArgv` (Node flags) is **not** forwarded to the child; the contract records this as accepted — a child that needs a Node flag its parent carried is out of scope for theta 1.0.

<a id="subagent-launch-contract"></a>
**Launch contract.** The child inherits the parent's **full environment** and runs with the forwarded `ctx.cwd` as its working directory, so parent and child read the identical settings, profiles, and credential sources. Argv is assembled from the same inputs the subagent drive consumes:

| Subagent-drive input | Child-process carrier |
|---|---|
| resolved-and-interpolated frontmatter `system:` (per Parameters and Frontmatter) | `--system-prompt <text>` |
| the theta's callable set (`tools:` allowlist) | `--tools <name1,name2,…>` (built-ins and extension tools alike, by name) |
| empty callable set | `--no-tools` (the CLI's explicit empty active set) |
| `.theta` callables in `tools:` | resolved child-side by name, plus content-hash verification (see [*`.theta` callable content-hash verification*](#subagent-theta-callable-hash) below) |
| the theta's resolved `model` | `--provider <p> --model <id>` (reference only, re-resolved by the child; see [*Model marshalling*](#subagent-model-marshalling) below) |
| in-memory / ephemeral transcript | `--no-session` (no session file written) |
| empty skills / prompts / themes / context files | `--no-skills --no-prompt-templates --no-themes --no-context-files` |
| full extension discovery is the point | (no `--no-extensions` flag — extensions load in the child) |
| project-local trust inference | `--approve` / `--no-approve` per the trust rule (see [*Isolation and trust*](#subagent-isolation-and-trust) below) |
| forwarded `ctx.cwd` | child working directory |

An environment marker `PI_THETA_SUBAGENT_CHILD=1` identifies the child so the theta extension loading inside it suppresses its own file watcher and any behaviour that must not recurse; the marker also carries the parent PID (see the parent-PID-watchdog fallback under [PIC-9](#pic-9) below).

<a id="subagent-tools-allowlist-suppression"></a>
**`--tools` / `--no-tools` allowlist suppression.** The `--tools` allowlist (or `--no-tools` for the empty callable set) is what enforces the "ambient Pi tools NOT inherited" invariant, matching `tools: []` ≡ absent `tools:` from [Parameters and Frontmatter](../frontmatter.md). Three behavioural sub-properties the CLI-flag surface does not encode in its argument type carry the invariant: (1) an explicit non-empty `--tools` list restricts the child's active tool set to exactly those names; (2) `--no-tools` yields an empty active set distinct from omission; and (3) omitting both `--tools` and `--no-tools` re-enables Pi's default `read` / `bash` / `edit` / `write` built-ins — which the launch contract therefore never does. These are re-validated on each Pi minor bump per item (am) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. Built-in Pi tools are selected by name through the same allowlist; the child registers them natively at startup, so no per-tool built-in materialisation crosses the boundary.

<a id="subagent-theta-callable-hash"></a>
**`.theta` callable content-hash verification.** A `.theta` callable name in `--tools` resolves against the **child's own** registry: the theta extension loads in the child and discovers the same `.theta` / `.thetalib` files from disk. A file edit between parent load and child spawn would make the child silently run a callee the parent never validated — a silent-divergence failure mode, and racy under parallel siblings. The parent therefore records, at load time, a **content hash of each `.theta` callable's transitive closure** (the file plus its `.thetalib` imports — an import edit changes behaviour as much as a root-file edit) and marshals the hashes to the child. The child verifies each hash after its own parse and **refuses the invocation on mismatch** with a precise diagnostic (`theta/runtime/subagent-callable-hash-mismatch`, fail-closed — see [Diagnostics — `theta/runtime/*`](../diagnostics/code-registry-runtime.md)). Full content marshalling (running the child on copies of the parent-loaded sources) is the recorded escalation only if hashing proves insufficient; it is rejected as the default for its disproportionate machinery (transitive-closure copying, diagnostics pointing at copies).

<a id="subagent-model-marshalling"></a>
**Model marshalling and pre-flight.** The model crosses the boundary as a **reference** (`--provider <p> --model <id>`), never as a live object, and **credentials are never marshalled**: with full env inheritance and a shared agent dir, parent and child read the identical config universe, so credential skew is structurally impossible. Explicit credential marshalling (`--api-key`, env injection) is rejected: the parent cannot extract auth from Pi's extension API, and argv/env secrets are a security regression. One case has no prior-art cover: a theta with `model:` absent inherits the caller's *live session model*, state that exists only in the parent session. For that path the runtime adds a **child-side pre-flight** unique to the RPC child: before the first query, it confirms via the child's RPC state surface (`get_state` / `get_available_models` per `docs/rpc.md`) that the marshalled reference resolved to the intended model, and **fails the invocation with a precise diagnostic** (`theta/runtime/subagent-model-preflight-mismatch`) naming expected vs. resolved when it did not. Credential failures are not pre-flightable without a provider call and surface through the transport-error mapping below.

<a id="subagent-pre-spawn-model-guard"></a><a id="pic-40"></a> **PIC-40. Pre-spawn model guard and model pre-flight.** The `model` reference marshalled to the child is the theta's resolved model — the theta's frontmatter `model:`, or the model inherited at invocation time when `model:` is absent (resolution owned by [Parameters and Frontmatter — `model`](../frontmatter.md)). The guard has two obligations:

1. *Resolved value not `undefined`.* When resolution yields `undefined` — frontmatter `model:` absent and the inherited `ctx.model` is `undefined` — the runtime MUST NOT spawn the child; it fails the invocation with `theta/runtime/subagent-model-unresolved` (see [Diagnostics — `theta/runtime/*`](../diagnostics/code-registry-runtime.md#thetaruntime-runtime-panics-runtime-defect-surface-and-delivery-failures)). The check runs per invocation, after `ctx.model` is captured and before the spawn, and behaves identically at the slash-command, `tools:` `tool.execute`, and `invoke(...)` entry points.
2. *Child-side pre-flight confirmation.* After spawn and before the first query, the runtime confirms via the child's RPC state surface that the marshalled reference resolved to the intended model, per [*Model marshalling and pre-flight*](#subagent-model-marshalling) above, and fails with `theta/runtime/subagent-model-preflight-mismatch` on mismatch.

Both failures are terminal for that invocation — the runtime does not retry — and surface through the same per-surface channels as the runtime-defect surface in [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics): a `theta-system-note` on the slash/prompt surface, and `Err(InvokeInfraError { kind: "invoke_infra", cause: "subagent_model_unresolved", ... })` (or `cause: "subagent_model_preflight_mismatch"`) to an `invoke` parent.

<a id="subagent-isolation-and-trust"></a>
**Isolation and trust.** The in-process clean room (empty resource loader, zero extensions) is rebuilt from the CLI flag set in the launch contract (`--no-skills --no-prompt-templates --no-themes --no-context-files --no-session`), audited at the Pi pin to reproduce the empty-member behaviour. Two deliberate dispositions:

- **Extension ambience is accepted and stated.** Full extension discovery in the child is the point of this design, and loaded extensions contribute more than tools (system-prompt appends, handlers, providers). The isolation guarantee is stated spec-side as: *no user/project context (files, skills, templates) is inherited; installed extensions' contributions are present, as in any Pi session.* This is a spec-visible weakening, stated rather than hidden. Curated loading (`--no-extensions` + explicit `-e` per the `sourceInfo` of each declared tool) is the recorded escalation if extension ambience proves contaminating.
- **Project-local trust is inherited by necessity-inference.** The launcher passes `--approve` **iff** the callable set contains a tool whose `pi.getAllTools()` `sourceInfo` is **project-local** — the parent could only have admitted that tool because the operator already trusted its extension in the parent session, so the child inherits a decision already made — and `--no-approve` otherwise (least privilege).

<a id="subagent-drive-mapping"></a>
**Drive mapping (RPC).** The RPC surface carries every member the former in-process drive consumed:

| Former `AgentSession` member | RPC equivalent (`docs/rpc.md`) |
|---|---|
| `sendUserMessage(text)` | `{"type":"prompt","message":…}` command on stdin |
| `subscribe` → `agent_end` event | `agent_end` event on stdout (`messages: AgentMessage[]`, `willRetry: boolean`) |
| terminal-event selection (`willRetry === false`) | identical, on the streamed events |
| `abort()` | `{"type":"abort"}` command |
| `dispose()` | graceful shutdown (stdin close) with a bounded kill fallback (see [PIC-9](#pic-9)) |

The framing rules in `docs/rpc.md` — **strict JSONL, LF-only splitting** — become a pinned consumed surface. A line that does not parse as a JSON RPC event is a wire-protocol parse failure (see [*Error fidelity*](#subagent-error-fidelity) below).

<a id="subagent-untyped-query-ok-extraction"></a><a id="pic-43"></a> **PIC-43. `agent_end` query-result extraction.** The query's `Ok(string)` value is extracted from the `agent_end` event itself — the event-delivered `messages: AgentMessage[]` array on the `agent_end` RPC event — and never from any handle-side `messages` getter. An `agent_end` event whose `willRetry` field is `true` precedes an automatic SDK retry and does not resolve the query: the runtime ignores such an event, keeps reading the stream, and resolves only from the terminal `agent_end` event whose `willRetry` is `false`. Once that terminal event is in hand, the runtime applies the same two short-circuits the prompt-mode driver pins, in the same order: first the cancellation short-circuit (`thetaAbort.signal.aborted`, which synthesises `Err(QueryError { kind: "cancelled" })` per [Cancellation](../cancellation.md)); then a transport-failure short-circuit that reads the trailing `assistant` message's `stopReason` from the terminal event's `messages` array and maps a `stopReason` of `"error"` to `Err(QueryError { kind: "transport", message: <errorMessage>, http_status: null, provider: <resolved-model-provider>, retryable: false })` — the same shape as the prompt-mode mapping at [prompt-mode error detection](./conversation-drive.md#prompt-mode-error-detection) (`errorMessage` is optional on `AssistantMessage`, so when it is absent `message` is the fixed string `"provider transport failure"`; `stopReason` and `errorMessage` are members of the externally-owned `AssistantMessage` type declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`, which theta redefines neither and treats `stopReason` as opaque apart from the `"error"` member it keys on). The string is then assembled — only once both short-circuits have been passed — from that terminal event's `messages` array by the same assistant-text rule the prompt-mode driver pins (the `text` content of the final turn's `assistant` messages, concatenated in chronological order with a single `\n` separator, with the provider-internal `thinking` array and all `toolCalls` entries omitted per the `[assistant]` body selection in [Compact-transcript format](../binder/binder-model-and-context.md#compact-transcript-format-normative) item 4; a final turn with no assistant text yields the empty string). This event-payload extraction is a named consumer of the [`AgentMessage[]` chronological-ordering presupposition](./host-prerequisites.md#messages-chronological-order-presupposition) (surface (2) there), re-audited on each Pi minor bump per item (h) of the editorial-review checklist under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) — the presupposition now covers the terminal `agent_end` RPC event's `messages` array rather than any in-process handle. The extraction rule itself transfers unchanged from the former in-process contract onto the RPC event payload.

<a id="pic-42"></a> **PIC-42. Terminal-event selection over the RPC event stream.** The runtime resolves each query from the child's RPC event stream (the strict-JSONL stdout), selecting the terminal `agent_end` event for the query in flight, and never from a session-global side channel. Because one child process serves one subagent-mode invocation, the event stream is per-invocation by construction; a fresh terminal-event read is performed per query, and the reader for a resolved query does not observe a subsequent query's events. Typed queries against the child honour the same behavioural contract as prompt-mode typed queries and use the same mechanism described in [Implementation Notes — Runtime](../implementation-notes.md#runtime). Each free-phase turn of a subagent typed query drives through the RPC `prompt` command and resolves from its own terminal `agent_end` event, so the transport-failure short-circuit above applies to those turns identically; the **forced respond turn continues to run off-session in the parent** through pi-ai's `complete()` free function (see [Conversation drive](./conversation-drive.md) and [Binder inference call](./binder-inference.md#binder-inference-call)) — exactly as before — so its provider failures never cross the wire and are classified through the [Provider error mapping](./provider-error-mapping.md#provider-error-mapping) table instead. The child transcript is ephemeral (`--no-session`) — the spec mandates the transcript stays private to the theta and is discarded when the theta returns; the `--no-session` flag is the mechanism that prevents subagent output from reaching Pi's user-facing UI, and user-visible behaviour is specified in [Slash-Command Invocation — User-visible streaming](../slash-invocation.md#slsh-2). Each subagent-mode invocation gets its own child process, its own captured `tools:` table (the load-time resolution snapshot from [Parameters and Frontmatter](../frontmatter.md)), and its own derived `AbortSignal`; sibling invocations triggered by parallel tool calls do not share mutable state.

<a id="subagent-error-fidelity"></a>
**Error fidelity.** Typed error values (`QueryError`) are reconstructed from child stdout rather than from live in-process objects, with expected fidelity loss of **zero** on the query path: the subagent error contract is already payload-derived (PIC-43 reads the trailing assistant message's `stopReason` / `errorMessage` and fixes `http_status: null`, `retryable: false`), and the RPC `agent_end` event carries the same `messages` array. The [Provider error mapping — `QueryError` audit table](./provider-error-mapping.md#subagent-queryerror-audit) tables every `QueryError` variant reachable on the subagent path and verifies each against the pinned event schema. Two genuinely new failure classes are pinned there, both mapping to the `transport`-kind error: **child crash / nonzero exit mid-query** (the pending query resolves to `Err(QueryError { kind: "transport", … })` with the exit detail in `message`, plus a diagnostic per [PIC-9](#pic-9)) and **unparseable wire output** (a stdout line that violates the strict-JSONL framing). Extending the wire (richer RPC state queries) is held in reserve only if the audit exposes a gap; degraded fidelity without the audit is rejected.

- **Consumed CLI flags:** `--mode rpc`, `--no-session`, `--system-prompt`, `--tools`, `--no-tools`, `--provider`, `--model`, `--no-skills`, `--no-prompt-templates`, `--no-themes`, `--no-context-files`, `--approve`, `--no-approve` (verified via `pi --help` at v0.80.10). Their semantics — chiefly the `--tools` / `--no-tools` suppression sub-properties ([PIC-40](#subagent-tools-allowlist-suppression) rule above) and the `--no-session` ephemeral-transcript behaviour — are re-audited per item (am) and item (aj) of the editorial-review checklist under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below.
- **Consumed RPC wire protocol (`docs/rpc.md`):** the strict-JSONL, LF-only framing; the `prompt` and `abort` stdin commands; the `get_state` / `get_available_models` state queries the model pre-flight reads; the `agent_end { messages: AgentMessage[], willRetry: boolean }` event; and the **stdin-EOF exit behaviour** the orphan-prevention layer under [PIC-9](#pic-9) depends on. These are re-audited per the CLI/RPC-protocol editorial-review items enumerated under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below (flag semantics, JSONL framing, `agent_end` payload shape, and stdin-EOF exit behaviour).

<a id="subagent-state-isolation-matrix"></a>

**Subagent state-isolation matrix.** A subagent-mode invocation runs in an isolated child process whose state composition is fixed by the table below. The matrix is the canonical enumeration of what the child inherits from the theta's frontmatter, what is forwarded from the caller's `ExtensionCommandContext`, and what is *not* inherited. The three columns are populated by distinct mechanisms (frontmatter resolution, the launch-contract forwarding rule above, and the explicit CLI-flag suppression rules above), and a future widening of any cell is a spec-versioned change.

| Inherited from the theta's frontmatter | Forwarded from the caller's `ExtensionCommandContext` | Not inherited from the caller |
|---|---|---|
| `system:` prompt (with `${param}` interpolation resolved at conversation-creation time) → `--system-prompt` | `ctx.cwd` (child working directory) | caller's transcript (the child runs `--no-session`; transcript is ephemeral, private, and discarded when the theta returns) |
| `model` — marshalled as a **reference** (`--provider <p> --model <id>`), re-resolved child-side and pre-flight-confirmed ([PIC-40](#pic-40)); never a live object, never credentials | (model is no longer forwarded from `ctx` as a live handle — `ctx.modelRegistry` does **not** cross the process boundary) | caller's system prompt (the child's `--system-prompt` is the *complete* prompt; no user/project context, files, skills, templates, or context files are inherited — `--no-skills --no-prompt-templates --no-themes --no-context-files`) |
| the theta's callable set → `--tools` / `--no-tools` (the theta's *callable set*; see [Glossary — `callable set`](../glossary.md)); `.theta` callables content-hash-verified child-side | (`ctx.getContextUsage` does **not** cross the process boundary) | ambient Pi tool set (suppressed by the explicit `--tools` allowlist / `--no-tools`; default rule when `tools:` is absent or empty is owned by [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools)). Installed extensions' contributions (tools, system-prompt appends, handlers, providers) **are** present in the child, as in any Pi session — a stated isolation weakening, per [*Isolation and trust*](#subagent-isolation-and-trust) above |
| captured `params` (load-time per-invocation values) | other forwarded `ExtensionContext` members per **`ExtensionContext` (member surface theta touches)** below (narrowed: the process boundary does not carry `ctx.modelRegistry` or `ctx.getContextUsage`) | caller's `thetaAbort` controller (each invocation gets a fresh `AbortController`; cancellation forwards via the RPC `abort` command, not shared controller identity — see **Cancellation source** below and [Cancellation](../cancellation.md)) |
| | | caller's `params` and variable bindings (the subagent runs a different theta file with its own `params`) |
| | | caller's `withActiveTools` snapshot scope (subagent-mode tool selection is the child's `--tools` allowlist, not `pi.setActiveTools`; see **Tool-registration lifetime and visibility** above) |

The `thetaAbort` row is the structural reason a subagent-mode child cannot observe its parent's cancellation directly: cancellation forwards via the parent's `thetaAbort.abort()` call propagating through a one-shot `thetaAbort.signal` listener that sends the RPC `abort` command to the child, not via shared controller identity. Nested-subagent cancellation behaviour is owned by [Cancellation — Signal source](../cancellation.md).

The matrix is consumed by `spec.md`'s preamble forward-link and by [Overview — Scope of a Theta File](../overview.md#scope-of-a-theta-file)'s isolation prose. The restated isolation guarantee — *no user/project context is inherited; installed extensions' contributions are present, as in any Pi session* — is the canonical wording those consumers restate.

<a id="pic-9"></a> **PIC-9. Subagent child-process lifecycle and orphan prevention.** The runtime owns the spawned child process for exactly the duration of one subagent-mode theta invocation. Teardown is mandatory and runs in a `finally` block that wraps the entire interpreter execution against that child, so it fires on:

- normal return (the theta's tail expression resolves);
- any `Err` returned to the parent (query failure, tool failure, child invoke failure, validation failure, cancellation);
- any panic raised inside the subagent (`MatchError`, index out of bounds, null access — see [Errors and Results](../errors-and-results.md));
- any unexpected exception thrown by the interpreter or the Pi SDK.

**Teardown (disposal successor).** The per-invocation `finally` closes the child's stdin and awaits child exit. If the child does not exit within the existing `SHUTDOWN_AWAIT_CAP_MS = 2000` budget, the runtime **kills** it (process-tree kill on Windows; no POSIX signal dependence). `disposeBarrier` settles on **observed child exit**. Teardown is invoked at most once per child and treated as idempotent at the call site: a second call (e.g. defensive cleanup in an outer scope) is a no-op. The same `finally` also detaches the one-shot `thetaAbort.signal` listener registered at spawn (per **Cancellation** below) — this is the listener-cleanup obligation [Cancellation — Forwarding into `thetaAbort`](../cancellation.md) cites for the subagent-mode entry — and it runs alongside the detach of every other forwarding listener the runtime attached during the invocation. If a teardown step itself throws (e.g. the stdin close or the kill call), the runtime logs the error via `theta/runtime/subagent-dispose-failure` (re-scoped from its former `AgentSession.dispose()`-throw trigger to any subagent-mode teardown-step throw — stdin close or bounded kill — keeping the stable diagnostics-contract code name; see [Diagnostics](../diagnostics.md)) but does not alter the invocation result on any teardown trigger — the diagnostic is advisory only. On an `Err`/panic teardown the original error is not masked; on a normal-return teardown the parent still observes the theta's `Ok` final value, and a teardown throw never promotes that `Ok` to an `Err`. The `session_shutdown` handler's `SHUTDOWN_AWAIT_CAP_MS` budget covers the entire teardown phase, including the bounded await of child exit and the kill fallback — there is no separate budget for the kill.

**Cancellation teardown.** The one-shot `thetaAbort.signal` listener sends the RPC `abort` command (per [PIC-41](#pic-41)); teardown then proceeds as above (stdin close → bounded await → kill). A *late failure* of the discarded RPC command send — one arriving after the synchronous send has returned — is **not** routed to `theta/runtime/internal-error`: it is one of the abandonable Promises governed by Cancellation's swallowing-handler rule, which keeps a swallowing handler attached so the late failure is silently absorbed along all three side channels (see [Cancellation — swallowing-handler attachment on every abandonable Promise](../cancellation.md)). The runtime is re-entrancy-safe by construction: the one-shot listener and the existing one-shot guard on `thetaAbort.abort()` make a double-abort (e.g. operator presses Esc twice) a no-op.

<a id="subagent-orphan-prevention"></a>**Orphan prevention** — a layered guarantee for a hazard with no in-process counterpart (a child outliving a dead parent is an unattended agent with tool access):

1. *Controlled parent paths* (normal return, errors, `/reload`, session shutdown, catchable process exit): the runtime kills the child per the teardown rule above. **Hard obligation.**
2. *Uncontrolled parent death* (hard kill, OOM): the child's stdin is a pipe from the parent; the OS closes it on parent death regardless of how the parent died. Pi's RPC mode **exiting on stdin EOF** is the class-2 mechanism — a behavioural presupposition verified at the Pi pin (`docs/rpc.md`, v0.80.10) and re-audited per version bump (it is not `typeof`-probable). If verification at the pin fails, the recorded fallback is a **child-side parent-PID watchdog**: the theta extension already loads inside the child and is already env-marked (`PI_THETA_SUBAGENT_CHILD=1`), so the marker carries the parent PID and the extension exits the child when the parent vanishes.
3. *Residual exposure*, documented: between parent death and child exit, an in-flight provider call may run for seconds; that window is irreducible without OS-level tethering (Job Objects / `PR_SET_PDEATHSIG`), which is **rejected** — platform-specific native dependencies contradict the codebase's dependency posture.

**Spawn failure** (ENOENT, EPERM, immediate exit) routes as an unanticipated SDK reject does today: `theta/runtime/internal-error`, no child to tear down. A dedicated spawn-failure diagnostic (`theta/runtime/subagent-spawn-failed`) additionally records the failure for operator triage. No child was launched, so the **finally** has nothing to tear down; the registry entry from **`ActiveInvocationRegistry`** is removed by the same `finally` in the normal way. **Child crash / nonzero exit mid-query** is a new failure class: the pending query resolves to `Err(QueryError { kind: "transport", … })` with the exit detail in `message`, and `theta/runtime/subagent-child-crashed` is emitted.

Subagent-mode tool registration has no separate teardown step: the child registers its own tools at startup and they die with the child process; the parent never registers anything for the subagent. No `pi.unregisterTool` call is required (or possible — Pi exposes none); see **Tool-registration lifetime and visibility** above for the full rule and the prompt-mode counterpart.

The per-invocation `finally` also settles the `disposeBarrier` promise the runtime registered with the **`ActiveInvocationRegistry`** (see **Extension entry point** step 4 above) — on observed child exit — and removes the registry entry. This is what lets the `session_shutdown` handler's `Promise.allSettled(...)` await observe completion of an in-flight subagent invocation that was cancelled by step 4's `thetaAbort.abort()` call. If Pi tears the runtime down before this `finally` runs (a `/reload` issued mid-turn while the subagent is still talking to its provider), the teardown handler's bounded await is what guarantees the child is killed and its provider connections released; the per-invocation `finally` and the teardown handler are two paths to the same child kill, and teardown's call-site idempotence (above) is what makes the overlap safe.

Edge cases:

- The `finally` block must run even when the parent's `AbortSignal` (see [Cancellation](../cancellation.md)) fires before the subagent's first turn — i.e. spawn-then-immediate-cancel still requires the child to be killed.
- For nested subagent invocations (a subagent that itself invokes a subagent-mode child), each level owns its own child process and runs its own `finally`. Teardown order is deepest-first, naturally produced by the call stack's `finally` unwinding.
- Teardown must precede the parent observing the result — the parent's `match`/`?` arm runs after the child has exited, so no child listener can leak into work the parent does next.

<a id="pic-41"></a> **PIC-41. Abort command and bounded kill.** Immediately after the child spawn is initiated and its RPC stdio is wired, the runtime registers a one-shot listener on `thetaAbort.signal` that sends the RPC `{"type":"abort"}` command on the child's stdin. The listener is the sole cancellation-forwarding mechanism in subagent mode. If `thetaAbort.signal.aborted` is already true at the moment the child stdio is ready, the runtime sends the `abort` command synchronously *before* the listener is registered and *before* issuing the first `prompt` command, so the spawn-then-immediate-cancel path remains correct without depending on microtask ordering. The listener is detached in the same per-invocation `finally` that tears the child down (per [PIC-9](#pic-9)). If sending the `abort` command throws, the runtime traps the error and routes it through `theta/runtime/internal-error` (see [Diagnostics](../diagnostics.md)) but does not alter the invocation result — mirroring the teardown-throws rule: on an `Err`/panic teardown the original error is not masked, and on a normal-return teardown the parent still observes the theta's `Ok` value. After the `abort` command (or if it could not be delivered), teardown proceeds through the stdin close → bounded await → kill sequence of [PIC-9](#pic-9), so cancellation of a wedged child always terminates within the `SHUTDOWN_AWAIT_CAP_MS` budget via the process-tree kill fallback. A thrown or failed child spawn is treated as an unanticipated SDK reject and routed through `theta/runtime/internal-error` per [PIC-9](#pic-9)'s spawn-failure rule; the parent invocation observes the `internal-error` routing (slash-command system note `"theta /<name> aborted with internal error: <message>"`; `Err(InvokeInfraError { kind: "invoke_infra", cause: "internal_error", ... })` to an `invoke` parent).

<a id="concurrent-invocation-isolation"></a>**Concurrent-invocation isolation and concurrency disposition.** Concurrent theta invocations within an extension instance — tracked as entries in the **`ActiveInvocationRegistry`** above — carry mode-qualified isolation and a fixed theta 1.0 concurrency disposition. *Transcript and tool-table isolation are subagent-mode-only:* a subagent-mode invocation runs its own child process (per **Conversation drive — subagent mode** above), so transcripts, captured `tools:` tables, and derived `AbortSignal`s are per-invocation and do not leak across siblings, while a prompt-mode invocation drives the user session's transcript by definition. <a id="no-invocation-cap"></a> theta 1.0 imposes no maximum on the number of in-flight theta invocations within an extension instance and no fairness or scheduling rule above Pi's event-loop ordering, interposing no admission cap, priority queue, or scheduler between sibling invocations and the event loop.

<a id="pic-22"></a> **PIC-22. Parallel subagent spawn initiation.** This disposition is observable at the parallel-tool-call surface (see [Tool Calls — Concurrency](../tool-calls.md#concurrency)): when the model emits N (for any fixed N ≥ 2) subagent-mode `.theta` callables as parallel tool calls in a single assistant turn, the runtime MUST, before any one of the N invocations returns, both **initiate the child-process spawn** for all N invocations and enter each child's first `prompt` command (reach the per-child drive point). A conformance test witnesses the obligation with a **fake process launcher** whose spawned children block their first `prompt` acknowledgement until explicitly released: a conformant runtime has spawned all N children and entered each child's first `prompt` before any blocked call is released, whereas a runtime that admitted invocations behind a cap, queue, or scheduler would leave at least one child unspawned. The conformance test's re-basing onto a fake process/RPC launcher (in place of the former fake in-process `AgentSession`) is the sole change to this rule; the obligation carries over unchanged. The test MUST tolerate real spawn times. The rule constrains the runtime only — Pi's own session-level serialisation is upstream and remains in force, and the rule does not promise resource unboundedness. Beyond the absence of an admission cap and scheduler pinned above, the theta adds no per-class threshold and no cross-sibling aggregation, throttling, or storm-detection over the resources concurrent theta invocations consume. Those resources — runtime-value heap, OS-level file descriptors / sockets / **child-process slots**, and provider-side rate-limit / quota — are owned by the host JavaScript runtime, the host OS, and the LLM provider respectively; their limits and any aggregation over those limits live in those layers, not in the theta contract. Simultaneous cold-boot resource contention among N freshly-spawned children falls under this same "resources are owned by the OS layer" disposition. The theta observes only the per-call surfacing of those limits through the existing routing it already pins: catchable host throws — the V8 `RangeError` family per `NOCEIL-3` at [Hard ceilings](../hard-ceilings/ceiling-invariants-and-audit.md#no-additional-ceilings), plus catchable OS-level descriptor / port / child-process-slot exhaustion surfaced as throws (a spawn that fails on child-process-slot exhaustion routes through the spawn-failure path of [PIC-9](#pic-9)) — route through `theta/runtime/internal-error` per [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics); per-query provider throttles (HTTP 429 and equivalents) surface as [`TransportError`](../errors-and-results/queryerror-variants.md#queryerror-variants) on the same page; uncatchable host fatals terminate the host process before any wrap can observe them and emit no `theta/runtime/internal-error` diagnostic. Widening or narrowing this rule is a major-version concern.

The prompt-mode counterpart to this subagent-mode concurrency — that prompt-mode bodies execute strictly sequentially within a single user session — is owned by [PIC-2 *Prompt-mode sequential execution*](./conversation-drive.md#pic-2) on the prompt-mode driver page.

<a id="subagent-performance-disposition"></a>
**Performance disposition.** Per-invocation process start (Node boot + extension discovery) replaces a cheap in-process construction, multiplied under `par for` fan-out and nested subagents. Bounding context: every subagent invocation also makes at least one LLM call, so the overhead is a fraction of typical wall time — but pure overhead. Disposition: ship as-is; benchmark spawn-to-first-prompt at the Pi pin and record the figure in the contract as an **expected-overhead note, not a guarantee**. **Process pooling is rejected**, and the rejection is recorded deliberately: a reused child carries state across invocations (extension module state, temp files, per-theta `--system-prompt`), trading the per-invocation isolation invariant — the property this design exists to guarantee — for latency. Startup-cost mitigation (e.g. trimming child startup work) is out of scope for this RFC; if measurement ever shows the overhead to be a binding constraint, a mitigation is designed then, on its own record.
