# Subagent

**Conversation drive — subagent mode.** The loom interpreter spawns a fresh in-process `AgentSession` via `createAgentSession`. The `createAgentSession(options: CreateAgentSessionOptions)` function and the `CreateAgentSessionOptions` interface are declared at `dist/core/sdk.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above); the inline call shape and the no-`signal`-field claim below are pinned to that file and MUST be re-validated against it on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. The two tool-related fields on `CreateAgentSessionOptions` carry distinct payloads and must both be populated from the same lowered set:

```ts
const customTools: ToolDefinition[] = lowerLoomToolsToPi(loom.tools); // built-in Pi tools resolved by name → their ToolDefinition; loom callees wrapped via defineTool
const loomSystemPrompt: string = renderLoomSystemPrompt(loom.frontmatter.system, params); // resolved-and-interpolated `system:` per Parameters and Frontmatter
const resourceLoader: ResourceLoader = {
  // Loom-owned adapter: getSystemPrompt is the only loom-load-bearing member; the rest return empty/defaults.
  getSystemPrompt: () => loomSystemPrompt,
  getAppendSystemPrompt: () => [],
  getExtensions: () => ({ extensions: [], errors: [] }),
  getSkills: () => ({ skills: [], diagnostics: [] }),
  getPrompts: () => ({ prompts: [], diagnostics: [] }),
  getThemes: () => ({ themes: [], diagnostics: [] }),
  getAgentsFiles: () => ({ agentsFiles: [] }),
  extendResources: () => {},
  reload: async () => {},
};
const { session } = await createAgentSession({
  customTools,
  tools: customTools.map((t) => t.name), // explicit allowlist suppresses Pi's default built-ins
  model,
  sessionManager: SessionManager.inMemory(cwd),
  resourceLoader,
  // ...
});
```

Four rules govern the spawn call:

1. `customTools` carries every `ToolDefinition` the subagent may use — both Pi built-ins (resolved by name from the model registry / extension API to their `ToolDefinition`) and `defineTool`-wrapped `.loom` callables. Each entry uses the same `name` / `label` / `description` / `parameters` / `execute` shape and the same `Type.Unsafe<unknown>(loweredJsonSchema)` `parameters` wrap defined in the **Per-loom registration** section above. `ToolDefinition.label` is required; the lowering step must supply it (basename-derived per the **Per-loom registration** rule).
2. `tools` is **always** passed as an explicit allowlist of those same names, even when the loom's callable set is empty (in which case `tools: []` is passed, matching `tools: []` ≡ absent `tools:` from [Parameters and Frontmatter](../frontmatter.md)). The explicit allowlist is what enforces the "ambient Pi tools NOT inherited" invariant; omitting `tools` would re-enable Pi's default built-in `read` / `bash` / `edit` / `write` tools regardless of what the loom declared.
3. The `tools` allowlist and the `customTools` array are derived from the same lowered set in a single step; the runtime does not let them drift. A `.loom` callable wrapped via `defineTool` has a `name` chosen by the loom runtime (post-`as` rename per the basename rules in [Parameters and Frontmatter](../frontmatter.md)); that same name must appear in the `tools` allowlist.
4. The `resourceLoader` passed to `createAgentSession` is a loom-constructed `ResourceLoader` adapter whose `getSystemPrompt()` returns the resolved-and-interpolated frontmatter `system:` string verbatim and whose `getAppendSystemPrompt()` returns `[]` (the loom's `system:` is the *complete* system prompt, not an append on top of a host base). The remaining `ResourceLoader` members return empty values / defaults (`getExtensions`, `getSkills`, `getPrompts`, `getThemes`, `getAgentsFiles`) or are no-ops (`extendResources`, `reload`); the loom MUST NOT pass the parent session's `DefaultResourceLoader` (pinned at [*Spawn-block satellite type pins*](#subagent-spawn-satellite-types) below) through unchanged. `CreateAgentSessionOptions` exposes no `systemPrompt` / `system` field in the loom 1.0 Pi SDK pin, so the loader's `getSystemPrompt()` is the only available delivery channel for the loom's `system:` value into the spawned `AgentSession.systemPrompt`. The adapter MAY be expressed as the object literal above or as a `class implements ResourceLoader` declaration; both are conformant. `DefaultResourceLoader`'s `systemPromptOverride: (base) => string | undefined` constructor option is an alternative channel but is **not** the recommended construction: it still runs all of `DefaultResourceLoader`'s discovery side-effects against the parent `cwd` to compute `base`, only to ignore the result, so the custom-adapter form is preferred.

<a id="subagent-spawn-satellite-types"></a>
*Spawn-block satellite type pins.* The two Pi-owned satellite types the spawn call consumes by name are pinned to their declarations at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) on `@earendil-works/pi-coding-agent`, on the same footing as the other consumed Pi types on this page. The `ResourceLoader` interface the rule-4 adapter implements is declared at `dist/core/resource-loader.d.ts` (re-exported from the package root); its loom-load-bearing member surface is exactly the set rule 4 enumerates — `getSystemPrompt` as the sole `system:` delivery channel, the remaining members populated with the empty / default / no-op values rule 4 fixes — and loom redefines none of them and treats every other property of `ResourceLoader` as opaque. `SessionManager.inMemory` — the static factory whose return value populates the `sessionManager` field and enforces the capability-item-3 transcript-privacy guarantee — is declared as `static inMemory(cwd?: string): SessionManager` on the `SessionManager` class at `dist/core/session-manager.d.ts` (the same declaration the `ReadonlySessionManager` `Pick<>` cited under [`ExtensionContext` (member surface loom touches)](./host-interfaces-core.md#extensioncontext-interface) tracks). The `ResourceLoader` interface and `SessionManager.inMemory` (the two types the spawn call consumes by name) MUST be re-validated against the cited declarations on each Pi minor bump per item (o) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; neither is factory-probed or enumerated by step 2(a)'s seven-capability literal-read assertion, so a silent rename or member reshape surfaces as a runtime error at the spawn site rather than at load time — the same posture pinned for the non-probed `AgentSession` members under item (n). The `DefaultResourceLoader` class and its `DefaultResourceLoaderOptions` constructor-options type are named in rule 4 above only to be rejected — the loom MUST NOT pass the parent session's `DefaultResourceLoader` through unchanged, and `DefaultResourceLoaderOptions.systemPromptOverride` is the not-recommended construction channel — and are pinned here on the same footing: both are declared at `dist/core/resource-loader.d.ts` on `@earendil-works/pi-coding-agent` at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin), and loom redefines and consumes neither. The `DefaultResourceLoader` / `DefaultResourceLoaderOptions` pin is folded into the same item (o) re-validation obligation even though loom consumes neither, so a Pi-side reshape that invalidates rule 4's rejection rationale is caught at the bump rather than going silently stale.

<a id="subagent-pre-spawn-model-guard"></a>**Pre-spawn model guard.** The `model` value passed to `createAgentSession` above is the loom's resolved model — the loom's frontmatter `model:`, or the model inherited at invocation time when `model:` is absent (resolution owned by [Parameters and Frontmatter — `model`](../frontmatter.md)). When that resolution yields `undefined` — frontmatter `model:` absent and the inherited `ctx.model` is `undefined` — the runtime MUST NOT call `createAgentSession`; it instead fails the invocation with `loom/runtime/subagent-model-unresolved` (see [Diagnostics — `loom/runtime/*`](../diagnostics/code-registry-runtime.md#loomruntime-runtime-panics-runtime-defect-surface-and-delivery-failures)). The check runs per invocation, after `ctx.model` is captured and before the spawn call, and behaves identically at the slash-command, `tools:` `tool.execute`, and `invoke(...)` entry points. The failure is terminal for that invocation — the runtime does not retry — and surfaces through the same per-surface channels as the runtime-defect surface in [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics): a `loom-system-note` on the slash/prompt surface, and `Err(InvokeInfraError { kind: "invoke_infra", cause: "subagent_model_unresolved", ... })` to an `invoke` parent.

Immediately after the awaited `createAgentSession(...)` resolves, the runtime registers a one-shot listener on `loomAbort.signal` that calls `AgentSession.abort()` on the returned handle. The listener is the sole cancellation-forwarding mechanism in subagent mode — `CreateAgentSessionOptions` (declared at `dist/core/sdk.d.ts` in the loom 1.0 Pi-SDK pin per the introductory paragraph above) has no `signal` field, so passing one would be silently ignored; the spawn call MUST NOT include a `signal` field. If `loomAbort.signal.aborted` is already true at the moment `createAgentSession(...)` resolves, the runtime invokes `session.abort()` synchronously *before* the listener is registered and *before* issuing the first user turn, so the spawn-then-immediate-cancel path remains correct without depending on microtask ordering. The listener is detached in the same per-invocation `finally` that calls `AgentSession.dispose()` (see **Subagent session lifecycle** below). The spawned session inherits the loom's `system:` prompt (from frontmatter, with `${param}` interpolation resolved at conversation-creation time), the lowered `customTools` set described above, and the loom's `model`. Queries against the spawned session use `session.sendUserMessage(text)` on the returned `AgentSession` and await completion by `session.subscribe(event => { if (event.type === "agent_end") … })` (both members, and `dispose()` / `abort()`, are pinned at [`AgentSession` (member surface loom touches)](./host-interfaces-core.md#agentsession-interface) above), which is scoped to that `AgentSession`'s own `ExtensionRunner` and cannot fire for any other session. <a id="subagent-untyped-query-ok-extraction"></a>The query's `Ok(string)` value is extracted from the `agent_end` event itself — the event-delivered `messages: AgentMessage[]` array on the `agent_end` variant of `AgentSessionEvent` (declared at `dist/core/agent-session.d.ts` in `@earendil-works/pi-coding-agent` at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin); re-validated on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below) — and never from the `AgentSession.messages` getter on the session handle. An `agent_end` event whose `willRetry` field is `true` precedes an automatic SDK retry and does not resolve the query: the runtime ignores such an event, keeps the subscription live, and resolves only from the terminal `agent_end` event whose `willRetry` is `false`. The string is then assembled from that terminal event's `messages` array by the same assistant-text rule the prompt-mode driver pins above (the `text` content of the final turn's `assistant` messages, concatenated in chronological order with a single `\n` separator, with the provider-internal `thinking` array and all `toolCalls` entries omitted per the `[assistant]` body selection in [Compact-transcript format](../binder/binder-model-and-context.md#compact-transcript-format-normative) item 4; a final turn with no assistant text yields the empty string), so the "same final-assistant-text rule on both sides" invariant holds. This event-payload extraction is a named consumer of the [`AgentMessage[]` chronological-ordering presupposition](./host-interfaces-core.md#messages-chronological-order-presupposition) (surface (2) there): the "chronological order" concatenation above presupposes the terminal `agent_end` event's `messages` array is ordered oldest-to-newest, which the `AgentSessionEvent` type does not encode, and that presupposition is re-audited on each Pi minor bump per item (h) of the editorial-review checklist under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure). The runtime MUST use the session-local `subscribe` API rather than the global `pi.on("agent_end", …)` event for the same per-session-scoping reason given for prompt mode. The runtime unsubscribes (via the function returned by `subscribe`) before resolving each query's promise so that the next query on the same long-lived subagent session does not observe the previous query's listener; a fresh subscription is attached per query. Typed queries against the spawned session honour the same behavioural contract as prompt-mode typed queries above and use the same mechanism described in [Implementation Notes — Runtime](../implementation-notes.md#runtime). The session is in-memory only — the spec mandates the transcript stays private to the loom and is discarded when the loom returns. The in-memory `SessionManager` is the mechanism that prevents subagent output from reaching Pi's user-facing UI; user-visible behaviour is specified in [Slash-Command Invocation — User-visible streaming](../slash-invocation.md#slsh-2). Each subagent-mode invocation gets its own `AgentSession`, its own captured `tools:` table (the load-time resolution snapshot from [Parameters and Frontmatter](../frontmatter.md)), and its own derived `AbortSignal`; sibling invocations triggered by parallel tool calls do not share mutable session state.

<a id="subagent-state-isolation-matrix"></a>

**Subagent state-isolation matrix.** A subagent-mode invocation runs in an isolated session whose state composition is fixed by the table below. The matrix is the canonical enumeration of what the spawned session inherits from the loom's frontmatter, what is forwarded from the caller's `ExtensionCommandContext`, and what is *not* inherited from the caller. The three columns are populated by distinct mechanisms (frontmatter resolution, the override-table forwarding rule under **`ExtensionContext` (member surface loom touches)** below, and the explicit suppression rules in **Conversation drive — subagent mode** above) and a future widening of any cell is a spec-versioned change.

| Inherited from the loom's frontmatter | Forwarded from the caller's `ExtensionCommandContext` | Not inherited from the caller |
|---|---|---|
| `system:` prompt (with `${param}` interpolation resolved at conversation-creation time) | `ctx.cwd` | caller's transcript (the spawned session uses `SessionManager.inMemory(cwd)`; transcript is private and discarded when the loom returns) |
| `model` (resolved from frontmatter) | `ctx.modelRegistry` | caller's system prompt (suppressed by the loom-constructed `ResourceLoader` adapter passed to `createAgentSession` whose `getSystemPrompt()` returns the resolved-and-interpolated loom frontmatter `system:` and whose `getAppendSystemPrompt()` returns `[]`; see rule 4 under **Conversation drive — subagent mode** above. `CreateAgentSessionOptions` exposes no `systemPrompt` field — delivery via the loader is the only available channel, and passing the parent session's `DefaultResourceLoader` unchanged would silently inherit the host project's `AGENTS.md`-derived prompt) |
| lowered `customTools` set (the loom's *callable set*; see [Glossary — `callable set`](../glossary.md)) | `ctx.getContextUsage` | ambient Pi tool set (suppressed by the explicit `tools` allowlist passed to `createAgentSession`; default rule when `tools:` is absent or empty is owned by [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools)) |
| captured `params` (load-time per-invocation values) | other non-overridden `ExtensionContext` members per **`ExtensionContext` (member surface loom touches)** below | caller's `loomAbort` controller (each invocation gets a fresh `AbortController`; lineage and forwarding are owned by **Cancellation source** below and [Cancellation](../cancellation.md)) |
| | | caller's `params` and variable bindings (the subagent runs a different loom file with its own `params`) |
| | | caller's `withActiveTools` snapshot scope (subagent-mode tool registration uses `customTools`, not `pi.setActiveTools`; see **Tool-registration lifetime and visibility** above) |

The `loomAbort` row is the structural reason a subagent-mode child cannot observe its parent's cancellation directly: cancellation forwards via the parent's `loomAbort.abort()` call propagating through a one-shot `loomAbort.signal` listener that calls `AgentSession.abort()` on the spawned `AgentSession`, not via shared controller identity and not via any `createAgentSession` option (no `signal` field exists on `CreateAgentSessionOptions` in the loom 1.0 Pi SDK pin). Nested-subagent cancellation behaviour is owned by [Cancellation — Signal source](../cancellation.md).

The matrix is consumed by `spec.md`'s preamble forward-link and by [Overview — Scope of a Loom File](../overview.md#scope-of-a-loom-file)'s isolation prose.

<a id="pic-9"></a> **PIC-9. Subagent session lifecycle.** The runtime owns the spawned `AgentSession` for exactly the duration of one subagent-mode loom invocation. Disposal is mandatory and runs in a `finally` block that wraps the entire interpreter execution against that session, so it fires on:

- normal return (the loom's tail expression resolves);
- any `Err` returned to the parent (query failure, tool failure, child invoke failure, validation failure, cancellation);
- any panic raised inside the subagent (`MatchError`, index out of bounds, null access — see [Errors and Results](../errors-and-results.md));
- any unexpected exception thrown by the interpreter or the Pi SDK.

Disposal calls `AgentSession.dispose()`, which removes all event listeners (including any in-flight `agent_end` subscription the runtime attached via `session.subscribe(...)` for the current query) and disconnects from the underlying agent. The same `finally` also detaches the one-shot `loomAbort.signal` listener registered at spawn (per **Conversation drive — subagent mode** above) — this is the listener-cleanup obligation [Cancellation — Forwarding into `loomAbort`](../cancellation.md) cites for the subagent-mode entry, and it runs alongside the existing detach of every other forwarding listener the runtime attached during the invocation. `dispose()` is invoked at most once per session and treated as idempotent at the call site: a second call (e.g. defensive cleanup in an outer scope) is a no-op. If `dispose()` itself throws, the runtime logs the disposal error via the `loom/runtime/subagent-dispose-failure` diagnostic (see [Diagnostics](../diagnostics.md)) but does not mask the original error that triggered teardown — the parent still observes the original `Err` or panic.

`AgentSession.abort()` (invoked by the cancellation listener above) returns `Promise<void>` and resolves once the agent has become idle. The runtime does **not** `await` the returned promise from inside the listener — the listener fires synchronously, calls `session.abort()`, and discards the returned promise; teardown progress is observed instead through the `disposeBarrier` and the subsequent `dispose()` call in the per-invocation `finally`. If the synchronous `session.abort()` call throws, the runtime traps the error and routes it through `loom/runtime/internal-error` (see [Diagnostics — `loom/runtime/*`](../diagnostics.md)) rather than masking any original error the surrounding `finally` was protecting; this mirrors the `dispose()`-throws rule above. A *late rejection* of the discarded promise — one arriving after the synchronous `abort()` call has already returned — is **not** routed to `loom/runtime/internal-error`: the `AgentSession.abort()` Promise is one of the abandonable Promises governed by Cancellation's swallowing-handler rule, which keeps a swallowing handler attached so the late rejection is silently absorbed along all three side channels and explicitly forbids promotion to `loom/runtime/internal-error` (the late-rejection disposition is owned in one place; see [Cancellation — swallowing-handler attachment on every abandonable Promise](../cancellation.md)). The `session_shutdown` handler's `SHUTDOWN_AWAIT_CAP_MS = 2000` budget covers the entire disposal phase, including any time `AgentSession.abort()`'s underlying idle-wait contributes via the subsequent `dispose()` call — there is no separate budget for `abort()`. The runtime is re-entrancy-safe by construction: the one-shot `loomAbort.signal` listener and the existing one-shot guard on `loomAbort.abort()` together make a double-abort (e.g. operator presses Esc twice) a no-op.

A thrown or rejected `createAgentSession`, or a returned handle whose `dispose` member is not a function, is treated as an unanticipated SDK reject and routed through `loom/runtime/internal-error` (see [Diagnostics — `loom/runtime/*`](../diagnostics.md), whose trigger description explicitly enumerates `createAgentSession` among the unanticipated SDK rejects). The parent invocation observes the routing already specified for `internal-error`: a slash-command surface emits `"loom /<name> aborted with internal error: <message>"` as a system note; an `invoke` parent observes `Err(InvokeInfraError { kind: "invoke_infra", cause: "internal_error", ... })`. No `AgentSession` was constructed, so the **Subagent session lifecycle** `finally` below has nothing to dispose; the registry entry from **`ActiveInvocationRegistry`** is removed by the same `finally` in the normal way.

Subagent-mode tool registration has no separate teardown step: the `.loom` callables and the typed-query one-shot tool reach the model via `customTools` on `createAgentSession`, so they exist only for the lifetime of the spawned `AgentSession` and are released by `dispose()` itself. No `pi.unregisterTool` call is required (or possible — Pi exposes none); see **Tool-registration lifetime and visibility** above for the full rule and the prompt-mode counterpart.

The per-invocation `finally` also settles the `disposeBarrier` promise the runtime registered with the **`ActiveInvocationRegistry`** (see **Extension entry point** step 4 above) and removes the registry entry. This is what lets the `session_shutdown` handler's `Promise.allSettled(...)` await observe completion of an in-flight subagent invocation that was cancelled by step 4's `loomAbort.abort()` call. If Pi tears the runtime down before this `finally` runs (a `/reload` issued mid-turn while the subagent is still talking to its provider), the teardown handler's bounded await is what guarantees `dispose()` is called and provider connections are released; the per-invocation `finally` and the teardown handler are two paths to the same disposal, and `dispose()`'s call-site idempotence (above) is what makes the overlap safe.

Edge cases:

- The `finally` block must run even when the parent's `AbortSignal` (see [Cancellation](../cancellation.md)) fires before the subagent's first turn — i.e. spawn-then-immediate-cancel still requires disposal.
- For nested subagent invocations (a subagent that itself invokes a subagent-mode child), each level owns its own session and runs its own `finally`. Disposal order is deepest-first, naturally produced by the call stack's `finally` unwinding.
- Disposal must precede the parent observing the result — the parent's `match`/`?` arm runs after the child's session is gone, so the child's listeners cannot leak into work the parent does next.

<a id="concurrent-invocation-isolation"></a>**Concurrent-invocation isolation and concurrency disposition.** Concurrent loom invocations within an extension instance — tracked as entries in the **`ActiveInvocationRegistry`** above — carry mode-qualified isolation and a fixed loom 1.0 concurrency disposition. *Transcript and tool-table isolation are subagent-mode-only:* a subagent-mode invocation runs its own private `AgentSession` (per **Conversation drive — subagent mode** above), so transcripts, captured `tools:` tables, and derived `AbortSignal`s are per-invocation and do not leak across siblings, while a prompt-mode invocation drives the user session's transcript by definition. <a id="no-invocation-cap"></a> loom 1.0 imposes no maximum on the number of in-flight loom invocations within an extension instance and no fairness or scheduling rule above Pi's event-loop ordering, interposing no admission cap, priority queue, or scheduler between sibling invocations and the event loop. This disposition is observable at the parallel-tool-call surface (see [Tool Calls — Concurrency](../tool-calls.md#concurrency)): when the model emits N (for any fixed N ≥ 3) subagent-mode `.loom` callables as parallel tool calls in a single assistant turn, the runtime MUST initiate `createAgentSession` for all N invocations before any one of the N returns. A conformance test witnesses the obligation with a fake `AgentSession` whose `sendUserMessage` blocks until explicitly released: a conformant runtime has created all N sessions and entered each session's `sendUserMessage` before any blocked call is released, whereas a runtime that admitted invocations behind a cap, queue, or scheduler would leave at least one session uncreated. The rule constrains the runtime only — Pi's own session-level serialisation (e.g. prompt-mode slash-command turns within one user session, per the [snapshot/restore Pi behavioural preconditions](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions) above) is upstream and remains in force, and the rule does not promise resource unboundedness. Beyond the absence of an admission cap and scheduler pinned above, the loom adds no per-class threshold and no cross-sibling aggregation, throttling, or storm-detection over the resources concurrent loom invocations consume. Those resources — runtime-value heap (V8 string / array / total-heap allocation), OS-level file descriptors / sockets / child-process slots, and provider-side rate-limit / quota — are owned by the host JavaScript runtime, the host OS, and the LLM provider respectively; their limits and any aggregation over those limits live in those layers, not in the loom contract. The loom observes only the per-call surfacing of those limits through the existing routing it already pins: catchable host throws — the V8 `RangeError` family (`Invalid string length`, `Invalid array length`, `Maximum call stack size exceeded`) per `NOCEIL-3` at [Hard ceilings — No additional loom 1.0 runtime ceiling applies](../hard-ceilings/ceiling-invariants-and-audit.md#no-additional-ceilings), plus catchable OS-level descriptor / port / child-process-slot exhaustion surfaced as throws by the host JavaScript runtime — route through `loom/runtime/internal-error` per [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics); per-query provider throttles (HTTP 429 and equivalents) surface as [`TransportError`](../errors-and-results/queryerror-variants.md#queryerror-variants) on the same page; uncatchable host fatals — V8 heap-limit OOM via the `OOMErrorCallback` / `abort()` path, OS process-kill, and any other engine fatal that bypasses the JavaScript exception machinery — terminate the host process before any wrap can observe them and emit no `loom/runtime/internal-error` diagnostic, on the same footing as any other engine fatal. Widening or narrowing this rule is a major-version concern.

<a id="pic-2"></a> **PIC-2. Prompt-mode sequential execution.** Prompt-mode bodies execute strictly sequentially within a single user session: at most one prompt-mode body — top-level slash dispatches and nested prompt → prompt `invoke(...)` calls alike — holds an open `pi.setActiveTools` snapshot/restore window at a time. This is a *derived* property rather than an independent runtime obligation: it follows from Pi's per-session slash-handler serialisation guarantee (snapshot/restore [behavioural precondition (b)](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions) above, under which two loom-invocation slash dispatches against the same session cannot overlap their snapshot/restore windows), composed with the loom-runtime-owned guarantees that (i) prompt-mode `.loom` callees in another loom's `tools:` are rejected at load time per [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools), (ii) `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns and loom 1.0 exposes no parallel-`invoke` surface per [Invocation — Cross-mode semantics](../invocation.md), and (iii) the `subagent → prompt` fan-out path into the user session is closed per the same [Invocation — Cross-mode semantics](../invocation.md) rule. The three potential sources of in-session overlap therefore produce genuine concurrency only between subagent-mode invocations, per **Concurrent-invocation isolation and concurrency disposition** above.
