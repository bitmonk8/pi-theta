# Subagent

**Conversation drive ŌĆö subagent mode.** A subagent-mode invocation runs the **whole callee** ŌĆö interpreter included ŌĆö inside a **spawned child `pi` process**, not only its conversation. Where the predecessor design (RFC 0005) kept the interpreter in the parent and remoted only the `AgentSession` over RPC, this design moves the callee's interpreter into the child: the child runs Pi's full startup extension discovery, the callee executes against the child's own host session, and the parent no longer drives a remote session for any subagent invocation. One child process is launched per subagent-mode invocation, at the point the pre-spawn model guard ([PIC-62](#pic-62) below) admits the invocation. The child is launched as:

```
pi --theta <dirs> --mode json -p "/<slug>" --no-session [--tools <name1,name2,ŌĆ”>]
```

The child runs the callee under the **subagent-root regime** ([PIC-58](#pic-58) below) ŌĆö the invocation regime of a `mode: subagent` theta that is itself the root theta of its own process. Params are marshalled structurally ([PIC-60](#pic-60)), the callee's final value returns to the parent as a single machine-readable envelope on the child's stdout ([PIC-59](#pic-59)), and the child's own host agent loop backs both model-facing and code-side extension-tool reach ([PIC-61](#pic-61)). No executable tool definition crosses any process boundary. The Pi surface consumed here ŌĆö the CLI flags above, the `--mode json` event stream, and the reserved-key stdout envelope line ŌĆö is verified against the installed Pi (`pi --help` and `docs/rpc.md`) at v0.80.10 and re-audited on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below.

<a id="subagent-launch-contract-differences"></a>
**Launch-contract differences from the RFC 0005 remote-session driver.** This design reuses RFC 0005's launcher ŌĆö executable resolution, ambient-suppression flags, `--no-session`, the trust rule, env inheritance, teardown, kill semantics, orphan handling, and spawn-failure diagnostics ŌĆö and retires only the RPC *drive contract*. The differences are:

| Concern | RFC 0005 (remote session) | This design (remote theta) |
|---|---|---|
| Child mode | `--mode rpc`, long-lived, multi-turn | `--mode json -p "/<slug>"`, one invocation per process |
| Interpreter | parent | child |
| `--tools` | callable-set allowlist for the session | **not required for correctness** ŌĆö the child theta enforces its own callable set; MAY be passed as defence-in-depth |
| `--system-prompt` | resolved `system:` | same, consumed by the subagent-root regime ([PIC-58](#pic-58)) |
| Params | parent interpolates into per-query text | marshalled structurally ([PIC-60](#pic-60)) |
| Result | per-query `agent_end` payloads extracted parent-side | one final-value envelope on stdout ([PIC-59](#pic-59)) |

<a id="subagent-executable-resolution"></a>
**Executable resolution.** The runtime resolves the child executable by re-launching the running process from its own invocation, via a two-rung ladder ŌĆö the mechanism proven by pi-config's `subagent` extension (`getPiInvocation()`):

1. **Entry-script rung.** When `process.argv[1]` names an existing file, spawn `process.execPath` (the Node/Bun binary) with that script as its first argument. The child is the exact same binary + entry script as the parent.
2. **Compiled-binary rung.** When `process.argv[1]` is unusable and `process.execPath` is not a generic runtime (`node` / `bun`), Pi itself is the executable ŌĆö spawn `process.execPath` directly.

There is **no `PATH` fallback** ŌĆö a deliberate divergence from the prior art. A `PATH`-resolved `pi` may be a different version or install than the parent, silently violating the pinned wire contract; a spec-governed language construct cannot tolerate that risk. If both rungs fail, resolution fails **closed at load time**: the theta does not register and the diagnostic (`theta/load/subagent-executable-unresolved`, see [Diagnostics ŌĆö `theta/load/*`](../diagnostics/code-registry-load.md)) names the reason. The rung-1 existence check runs at **probe time** (see [Step 0 ŌĆö Capability probe](./capability-probe.md#entry-capability-probe) above), not at first spawn. Parent `execArgv` (Node flags) is **not** forwarded to the child; the contract records this as accepted ŌĆö a child that needs a Node flag its parent carried is out of scope for theta 1.0.

<a id="subagent-root-regime"></a><a id="pic-58"></a> **PIC-58. Subagent-root regime.** The **subagent-root regime** is the invocation regime of a `mode: subagent` theta that is the root theta of its own child `pi` process. A naive child launch of a `mode: subagent` callee would be circular ŌĆö the theta runtime inside the child would see `mode: subagent` and spawn *another* child ŌĆö so the child MUST instead drive the root theta against the process's own host session (prompt-mode driver mechanics) while applying the subagent-mode frontmatter contract:

- **Driver selection.** The regime, not the frontmatter `mode:`, selects the driver: the child runs the root theta's queries against its own host session with the prompt-mode conversation drive ([Conversation drive ŌĆö prompt mode](./conversation-drive.md)), even though the theta is `mode: subagent`.
- **`system:` delivery.** The resolved-and-interpolated frontmatter `system:` is installed as the child session's system prompt, delivered via `--system-prompt <text>` at launch.
- **Active-set protocol (degenerate).** The callable set governs the child session's active tools for each query window through the prompt-mode `pi.setActiveTools` snapshot/restore protocol ([Tool-registration lifetime and visibility](./tool-registration-lifetime.md)), degenerate here: there is no pre-existing user state to restore for, so the snapshot is the child session's empty/default ambient set and the restore is a no-op of consequence.
- **Env-marker selection rule.** The regime is selected by the environment marker `PI_THETA_SUBAGENT_ROOT=<slug>`, set **only by the parent launcher** and **never authorable from a `.theta` file**. A theta author cannot request the regime; it is invisible to the language surface. This marker subsumes RFC 0005's `PI_THETA_SUBAGENT_CHILD` marker and carries its duties (watcher suppression, no-recursion guard, parent-PID carriage for the [orphan-prevention](#subagent-orphan-prevention) watchdog) alongside regime selection.
- **`theta/parse/system-on-prompt-mode` does not fire.** The callee is still `mode: subagent`, so its `system:` frontmatter is valid on its own terms; the regime drives it against a host session but does not reclassify it as `mode: prompt`. The parse diagnostic that rejects `system:` on a prompt-mode theta ([Parameters and Frontmatter](../frontmatter.md)) MUST NOT fire under this regime ŌĆö this is a regime-scoped carve-out mirroring the `subagent fn` `with { system: ŌĆ” }` precedent.
- **No-recursion guarantee.** The regime applies **only to the process root**. A nested `mode: subagent` callee invoked *by* this theta (via `invoke(...)` or a `.theta`-callable call) spawns its own child `pi` process in the normal way; the marker is set per child launch and governs only that child's own root theta, so no recursion of the regime occurs within a single process.

<a id="subagent-launch-contract"></a>
**Launch contract.** The child inherits the parent's **full environment** and runs with the forwarded `ctx.cwd` as its working directory, so parent and child read the identical settings, profiles, and credential sources. Argv and the marshalled channels are assembled from the same inputs the subagent invocation consumes:

| Subagent-invocation input | Child-process carrier |
|---|---|
| resolved-and-interpolated frontmatter `system:` | `--system-prompt <text>` (consumed by the subagent-root regime, [PIC-58](#pic-58)) |
| the callee slug | `-p "/<slug>"` (one invocation per process) |
| the theta's callable set (`tools:` allowlist) | `--tools <name1,name2,ŌĆ”>` ŌĆö **optional** defence-in-depth; the child theta enforces its own callable set regardless |
| empty callable set | `--no-tools` when `--tools` is passed at all (the CLI's explicit empty active set) |
| `.theta` callables in `tools:` | resolved child-side by name, plus content-hash verification (see [*`.theta` callable content-hash verification*](#subagent-theta-callable-hash) below) |
| the theta's resolved `model` | `--provider <p> --model <id>` (reference only, re-resolved by the child; see [*Model marshalling*](#subagent-model-marshalling) below) |
| already-typed `params` values | marshalled structurally on `PI_THETA_PARAMS` / `PI_THETA_PARAMS_FILE` (see [PIC-60](#pic-60)) |
| the per-chain invoke-depth counter | marshalled as wire-level data on an environment variable (see [Invocation ŌĆö Invocation depth bound](../invocation.md#inv-4)) |
| in-memory / ephemeral transcript | `--no-session` (no session file written) |
| empty skills / prompts / themes / context files | `--no-skills --no-prompt-templates --no-themes --no-context-files` |
| full extension discovery is the point | (no `--no-extensions` flag ŌĆö extensions load in the child) |
| project-local trust inference | `--approve` / `--no-approve` per the trust rule (see [*Isolation and trust*](#subagent-isolation-and-trust) below) |
| forwarded `ctx.cwd` | child working directory |
| regime selection | `PI_THETA_SUBAGENT_ROOT=<slug>` (parent-launcher only; [PIC-58](#pic-58)) |

<a id="subagent-tools-allowlist-suppression"></a>
**`--tools` / `--no-tools` allowlist suppression (defence-in-depth).** Because the child runs the whole callee, the child theta's own callable set is authoritative for both model-facing and code-side reach; the parent-passed `--tools` allowlist is **not required for correctness** and serves only as defence-in-depth against the child's session inheriting Pi's default built-ins. When the allowlist is passed, three behavioural sub-properties the CLI-flag surface does not encode in its argument type carry the isolation intent: (1) an explicit non-empty `--tools` list restricts the child session's active tool set to exactly those names; (2) `--no-tools` yields an empty active set distinct from omission; and (3) omitting both `--tools` and `--no-tools` re-enables Pi's default `read` / `bash` / `edit` / `write` built-ins ŌĆö which is why the launch contract passes `--no-tools` for an empty callable set rather than omitting the flag. These are re-validated on each Pi minor bump per item (am) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. Built-in Pi tools are selected by name; the child registers them natively at startup, so no per-tool built-in materialisation crosses the boundary.

<a id="subagent-theta-callable-hash"></a>
**`.theta` callable content-hash verification.** A `.theta` callable name in the callee's `tools:` resolves against the **child's own** registry: the theta extension loads in the child and discovers the same `.theta` / `.thetalib` files from disk. A file edit between parent load and child spawn would make the child silently run a callee the parent never validated ŌĆö a silent-divergence failure mode, and racy under parallel siblings. Because this design moves the whole callee into the child, the parent-load/child-load skew window widens from the model-facing `.theta` callables to the **whole callee file**: the parent records, at load time, a **content hash of the transitive closure of the root callee `.theta` plus every `.thetalib` import it (and its callable-set `.theta` entries) transitively reach**, and marshals those hashes to the child. The child verifies each hash after its own parse and **refuses the invocation on mismatch** with a precise diagnostic (`theta/runtime/subagent-callable-hash-mismatch`, fail-closed ŌĆö see [Diagnostics ŌĆö `theta/runtime/*`](../diagnostics/code-registry-runtime.md)). Full content marshalling (running the child on copies of the parent-loaded sources) is the recorded escalation only if hashing proves insufficient; it is rejected as the default for its disproportionate machinery.

<a id="pic-60"></a> **PIC-60. Marshalled-params channel (binder bypass).** The parent already holds the callee's **already-typed** param values (from the slash binder, an `invoke(...)` caller's typed arguments, or a `tools:` `tool.execute` object). Routing them through the slash string `-p "/<slug> ŌĆ”"` would re-enter the binder ŌĆö a model turn ŌĆö and lose typing, so the runtime MUST marshal them structurally as canonical JSON per the theta's `params:` schema and the child MUST validate the received JSON against **the same schema** before running the callee. The channel is **dual**, keyed on a pinned size threshold chosen conservatively under the tightest platform environment-block cap with headroom for the inherited environment (order of **8 KB** payload; Windows caps the entire environment block at ~32 KB):

- **Below the threshold** the JSON travels as the environment variable `PI_THETA_PARAMS=<canonical JSON>`.
- **At or above the threshold** the JSON is written to a **0600** temp file whose path travels as `PI_THETA_PARAMS_FILE`; the child reads and **deletes** it, and the parent's per-invocation `finally` deletes it as a backstop. This is the same temp-file mechanism the return-value envelope names as its escalation (see [PIC-59](#pic-59)), built once for both directions.

Env-only with a size guard is rejected ŌĆö it would silently narrow the language ("params that fit in an env var") for an implementation reason no author should have to know about. Stdin delivery is rejected ŌĆö Pi's `-p` mode has its own stdin conventions, and an extension consuming stdin inside a `pi` child is an unprobeable host-behaviour dependency (the child's stdin is spawned as a parent-held pipe used only for orphan-prevention EOF signalling, per [PIC-9](#pic-9)). The **binder is skipped entirely** on the marshalled path: binder inference remains exclusive to human slash invocation ([Slash-Command Argument Binding](../binder.md)), unchanged. A received-params payload that fails to parse as JSON, or that fails schema validation against the callee's `params:` schema, refuses the invocation with `theta/runtime/subagent-params-validation-failed` (fail-closed ŌĆö see [Diagnostics ŌĆö `theta/runtime/*`](../diagnostics/code-registry-runtime.md)); the failure surfaces to an `invoke` parent through the return envelope as `Err(InvokeInfraError { cause: "validation", ... })`.

<a id="subagent-return-envelope"></a><a id="pic-59"></a> **PIC-59. Return-value envelope.** The child theta's final value MUST reach the parent with `Result` fidelity. The child runtime emits a **single machine-readable envelope** as one JSONL line with the reserved top-level key `theta_result` on stdout, alongside the `--mode json` event stream; the parent matches the reserved key and ignores every other line:

```json
{"theta_result": {"v": 1, "ok": ŌĆ”}}
{"theta_result": {"v": 1, "err": {"kind": "query", "variant": {ŌĆ”}}}}
```

Requirements:

- **`Ok` values** serialise per the runtime value model (JSON-representable by construction).
- **Every `Err` variant** an in-process subagent invocation could surface is representable in the pinned envelope schema: every `QueryError` variant, `CodeToolError`, the `InvokeInfraError` causes, and panics routed as internal-error (`kind: "invoke_infra", cause: "internal_error"`). The envelope schema is **pinned in this contract** and its `err` arm mirrors the `QueryError` union declared in [Errors and Results ŌĆö QueryError variants](../errors-and-results/queryerror-variants.md#queryerror-variants).
- **Versioning + skew detection.** The envelope schema is **versioned**; parent and child assert compatibility on the reserved version field **`v`** carried inside the `theta_result` payload (`{"theta_result": {"v": <version>, ŌĆ”}}`, as shown above). Because the same installed theta extension serves both sides, skew arises only from a concurrent upgrade ŌĆö it is **detected, not tolerated**: a version the parent does not recognise refuses with `theta/runtime/subagent-envelope-schema-skew`. A line carrying the reserved key that does not parse against the pinned schema refuses with `theta/runtime/subagent-envelope-parse-failed`.
- **Fail-closed child-exit-without-envelope.** A child that exits without emitting an envelope (crash, kill, timeout) MUST map to `Err(InvokeInfraError { cause: "internal_error", ... })` with the exit detail ŌĆö **never a fabricated value** ŌĆö and emit `theta/runtime/subagent-exit-without-envelope` (see [Diagnostics ŌĆö `theta/runtime/*`](../diagnostics/code-registry-runtime.md)). The companion `theta/runtime/subagent-child-crashed` records the crash exit detail for operator triage on this path.

Sharing stdout with the event stream is safe by construction: the envelope writer (the theta runtime in the child) and Pi's event writer are the same process, and Node serialises same-process stream writes, so the reserved-key line cannot be split mid-write. **Implementation pin (Pi v0.80.10):** in `--mode json` / `-p` Pi's `takeOverStdout()` reassigns the extension-visible `process.stdout.write` to route to **stderr** (so stray extension stdout cannot corrupt the event channel), while Pi's own event writer keeps a captured raw handle to fd 1. The envelope writer must therefore emit to **fd 1 directly** (`fs.writeSync(1, line)`, one atomic newline-terminated syscall ŌĆö Windows-safe), NOT through `process.stdout.write`; emitting through the extension API would land the envelope on stderr where the parent's stdout scan never sees it. Stray lines from other extensions can still appear in the stream but cannot corrupt the envelope, and the parent ignores every non-`theta_result` line. A dedicated extra file descriptor is rejected (nonstandard, platform-sensitive on Windows spawn contexts, no atomicity gained). A temp-file handoff (parent-supplied path, atomic write + rename, 0600, read after exit) is the recorded escalation on two named triggers ŌĆö envelope payloads too large for single-line handling, or observed stream corruption in practice ŌĆö and if engaged serves the params channel symmetrically ([PIC-60](#pic-60)), one mechanism for both directions. The parent-side subagent contract reduces to **envelope consumption**: the parent launches, awaits the envelope, and maps `ok`/`err` to `Ok`/`Err`.

<a id="subagent-host-loop-dispatch"></a><a id="pic-61"></a> **PIC-61. Code-side extension-tool dispatch and permission surface.** Within the child the callee runs under the subagent-root regime ŌĆö a real host session with an agent loop ŌĆö so code-side calls (`<name>(args)`) resolve per the callable set inside the child:

- **built-ins** ŌĆö direct `execute` dispatch, unchanged;
- **`.theta` callables** ŌĆö unchanged (a nested subagent-mode callee spawns its own child per [PIC-58](#pic-58)'s no-recursion rule);
- **extension tools** ŌĆö **host-loop dispatch**: the child runtime registers a theta-controlled provider whose stream function authors the `tool_use` itself, carrying the code-supplied arguments verbatim; the child's host agent loop (which holds every registered tool's `execute`) runs the call and appends the tool result; the runtime reads the result back and returns it to code. Arguments are deterministic, zero model tokens are spent, and **no executable definition is ever obtained by theta code** ŌĆö documented Pi APIs only. Host-loop dispatch's costs ŌĆö a fabricated turn in the transcript and a temporary session-model switch ŌĆö are confined to the child's **private, discarded session**, never the user's.

**Code-side dispatch ladder (fail-closed).** Extension-tool code-side dispatch resolves against a probe-asserted, fail-closed ladder:

1. **Preferred rung (if ever available):** an upstream `pi.getToolDefinition` on the extension API ŌĆö a clean registry read that would replace host-loop dispatch without changing this design's process architecture. Requested upstream, so far refused; when it lands it slots in as the top rung child-side.
2. **Current rung:** host-loop dispatch (above).
3. **No rung available:** a theta whose code calls an extension tool **refuses to register** with `theta/load/extension-tool-unreachable` (fail-closed ŌĆö see [Diagnostics ŌĆö `theta/load/*`](../diagnostics/code-registry-load.md)); the runtime never silently falls through. The load-time refusal walks the theta's own body (its direct and local-`fn` code-side call sites); a code-side extension-tool call inside an imported `.thetalib` `fn` cannot arise, since a `.thetalib` fn naming a caller-scoped extension tool fails its standalone parse with `theta/parse/unknown-identifier` and un-registers the importer first, and the runtime host-loop rung remains the fail-closed floor for any path that bypasses the load check.

**Host-loop wiring (pinned by the shipped implementation, Pi v0.80.10).** The rung-2 mechanism resolves these otherwise-open contract details as follows (verified end-to-end by the RFC-0006 prototype and wired in `src/extension/production-host-loop-dispatch.ts`): (a) the theta-controlled provider is registered via `pi.registerProvider` under a **per-dispatch unique name** (`theta-host-loop-bridge-<nonce>`) with a dummy literal `apiKey` (so the bridge model is selectable with no network call) and a required dummy `baseUrl`, and torn down via `pi.unregisterProvider` after the turn; (b) the code-supplied request is encoded as a marker-prefixed user message the provider's two-state `streamSimple` decodes and authors as the `tool_use` with the arguments **verbatim**, keyed on the dispatch nonce so unrelated turns are tolerated; (c) the fabricated turn is awaited on the **`agent_settled` event, armed before `sendUserMessage`** ŌĆö `waitForIdle()` alone returns before a `-p`/print-mode turn even starts, so it is insufficient; (d) the tool result is read back from `ctx.sessionManager.getEntries()` by matching the newest `toolResult` message whose `toolName` equals the dispatched name (its `isError` is preserved to code); (e) the dispatched tool is installed as the sole active tool for the turn via the PIC-17 `pi.setActiveTools` snapshot/restore protocol, and the session model is switched to the bridge and **restored in a `finally`** (never left installed, incl. the abort path); (f) concurrent code-side dispatches are serialised, since the session-model switch is a shared session resource. The rung is established only inside the subagent-root child (a real host session + agent loop) with the required Pi surfaces present; the parent / prompt context keeps the rung unavailable (fail-closed).

**Permission surface.** Code-side dispatch of extension tools introduces **no new permission gate**, deliberately. The model-turn checkpoint it removes was already removed by the language itself: since RFC 0002, theta code calls `bash` with fully computed arguments, and `bash` is the maximal capability ŌĆö an extension tool cannot do to the machine anything `bash` cannot. The effective gates are the existing ones: (1) the author's `tools:` declaration ŌĆö a theta calls only what its frontmatter declares, readable by any operator; (2) the operator's extension-install and trust decisions, inherited by the child via the necessity-inference trust rule (`--approve`, see [*Isolation and trust*](#subagent-isolation-and-trust)); (3) fail-closed registration (rung 3 above). Per-tool capability annotations are rejected (author burden encoding a distinction with no security rationale); runtime approval prompts are rejected (they destroy the channel's purpose ŌĆö deterministic, unattended orchestration ŌĆö and a headless child has nobody to ask). Because code-side dispatch travels through the host agent loop, any tool-execution permission machinery Pi itself ever adds sits in the path automatically ŌĆö **host-loop routing is the seat for any future Pi-level tool-permission gate**, and theta needs no gate of its own for a future Pi gate to apply.

<a id="subagent-model-marshalling"></a>
**Model marshalling.** The model crosses the boundary as a **reference** (`--provider <p> --model <id>`), never as a live object, and **credentials are never marshalled**: with full env inheritance and a shared agent dir, parent and child read the identical config universe, so credential skew is structurally impossible. Explicit credential marshalling (`--api-key`, env injection) is rejected: the parent cannot extract auth from Pi's extension API, and argv/env secrets are a security regression. One case has no prior-art cover: a theta with `model:` absent inherits the caller's *live session model*, state that exists only in the parent session. For that path the parent marshals its live session model as the `--provider`/`--model` reference, and the child **re-resolves** it against its own (identical) config universe. The child confirms ŌĆö from its own model registry, without any RPC round-trip ŌĆö that the reference resolved to the intended model and **fails the invocation** with a precise diagnostic (`theta/runtime/subagent-model-preflight-mismatch`) naming expected vs. resolved when it did not; the failure surfaces to the parent through the return envelope. Credential failures are not pre-flightable without a provider call and surface through the child's own [provider-error mapping](./provider-error-mapping.md#provider-error-mapping) into the envelope.

<a id="subagent-pre-spawn-model-guard"></a><a id="pic-62"></a> **PIC-62. Pre-spawn model guard and child-side model resolution.** The `model` reference marshalled to the child is the theta's resolved model ŌĆö the theta's frontmatter `model:`, or the model inherited at invocation time when `model:` is absent (resolution owned by [Parameters and Frontmatter ŌĆö `model`](../frontmatter.md)). The guard has two obligations:

1. *Resolved value not `undefined` (parent-side).* When resolution yields `undefined` ŌĆö frontmatter `model:` absent and the inherited `ctx.model` is `undefined` ŌĆö the runtime MUST NOT spawn the child; it fails the invocation with `theta/runtime/subagent-model-unresolved` (see [Diagnostics ŌĆö `theta/runtime/*`](../diagnostics/code-registry-runtime.md#thetaruntime-runtime-panics-runtime-defect-surface-and-delivery-failures)). The check runs per invocation, after `ctx.model` is captured and before the spawn, and behaves identically at the slash-command, `tools:` `tool.execute`, and `invoke(...)` entry points.
2. *Child-side resolution confirmation.* The child re-resolves the marshalled reference against its own model registry and, per [*Model marshalling*](#subagent-model-marshalling) above, fails with `theta/runtime/subagent-model-preflight-mismatch` on mismatch ŌĆö reported to the parent through the return envelope rather than over any RPC state surface.

Both failures are terminal for that invocation ŌĆö the runtime does not retry ŌĆö and surface through the same per-surface channels as the runtime-defect surface in [Errors and Results ŌĆö Runtime panics](../errors-and-results/error-model.md#runtime-panics): a `theta-system-note` on the slash/prompt surface, and `Err(InvokeInfraError { kind: "invoke_infra", cause: "subagent_model_unresolved", ... })` (or `cause: "subagent_model_preflight_mismatch"`) to an `invoke` parent.

<a id="subagent-isolation-and-trust"></a>
**Isolation and trust.** The in-process clean room (empty resource loader, zero extensions) is rebuilt from the CLI flag set in the launch contract (`--no-skills --no-prompt-templates --no-themes --no-context-files --no-session`), audited at the Pi pin to reproduce the empty-member behaviour. Two deliberate dispositions:

- **Extension ambience is accepted and stated.** Full extension discovery in the child is the point of this design, and loaded extensions contribute more than tools (system-prompt appends, handlers, providers). The isolation guarantee is stated spec-side as: *no user/project context (files, skills, templates) is inherited; installed extensions' contributions are present, as in any Pi session.* This is a spec-visible weakening, stated rather than hidden. Curated loading (`--no-extensions` + explicit `-e` per the `sourceInfo` of each declared tool) is the recorded escalation if extension ambience proves contaminating.
- **Project-local trust is inherited by necessity-inference.** The launcher passes `--approve` **iff** the callable set contains a tool whose `pi.getAllTools()` `sourceInfo` is **project-local** ŌĆö the parent could only have admitted that tool because the operator already trusted its extension in the parent session, so the child inherits a decision already made ŌĆö and `--no-approve` otherwise (least privilege).

<a id="subagent-untyped-query-ok-extraction"></a>
**Query mechanics run in the child.** Every query the subagent callee issues ŌĆö untyped and typed alike ŌĆö executes **inside the child** against its own host session, using the same prompt-mode conversation-drive mechanics the [prompt-mode driver](./conversation-drive.md) pins (untyped-query trailing-turn extraction, the `stopReason: "error"` transport short-circuit, the typed-query two-phase free-phase / forced-respond loop, and respond-repair). The parent runs **no** per-query extraction and holds no session handle; it only consumes the final-value envelope ([PIC-59](#pic-59)). The typed-query forced-respond turn runs off-session in the **child** through pi-ai's `complete()` free function, so its provider errors are classified by the child's own [Provider error mapping](./provider-error-mapping.md#provider-error-mapping) ŌĆö the mapping table follows the interpreter into the child. The child transcript is ephemeral (`--no-session`): the spec mandates the transcript stays private to the theta and is discarded when the theta returns; `--no-session` + the child session being **process-private** (never Pi's user-facing UI) is the mechanism, and user-visible behaviour is specified in [Slash-Command Invocation ŌĆö User-visible streaming](../slash-invocation.md#slsh-2). Each subagent-mode invocation gets its own child process, its own captured `tools:` table, and its own derived `AbortSignal`; sibling invocations triggered by parallel tool calls do not share mutable state.

<a id="subagent-error-fidelity"></a>
**Error fidelity.** Typed error values (`QueryError`) are reconstructed in the parent from the child's `theta_result` envelope rather than from live in-process objects, with expected fidelity loss of **zero**: the child runs the identical query machinery the in-process path ran, and every `Err` variant it can produce is representable in the pinned envelope schema ([PIC-59](#pic-59)). The [Provider error mapping ŌĆö subagent `QueryError` audit](./provider-error-mapping.md#subagent-queryerror-audit) tables every variant the child surfaces and its envelope encoding. Two child-lifecycle failure classes are pinned there, both mapping to the fail-closed internal-error path rather than a fabricated value: **child crash / nonzero exit without an envelope** (`theta/runtime/subagent-exit-without-envelope`, with `theta/runtime/subagent-child-crashed` recording the crash detail) and **envelope corruption** (`theta/runtime/subagent-envelope-parse-failed` / `theta/runtime/subagent-envelope-schema-skew`). Extending the envelope (richer childŌåÆparent state) is held in reserve only if the audit exposes a gap; degraded fidelity without the audit is rejected.

<a id="pic-9"></a> **PIC-9. Subagent child-process lifecycle and orphan prevention.** The runtime owns the spawned child process for exactly the duration of one subagent-mode theta invocation. Teardown is mandatory and runs in a `finally` block that wraps the entire parent-side invocation against that child, so it fires on:

- normal return (the child emitted its `Ok` envelope);
- any `Err` returned to the parent (envelope `err`, child-exit-without-envelope, validation failure, cancellation);
- any panic raised inside the parent-side invocation path;
- any unexpected exception thrown by the parent runtime or the Pi SDK.

**Teardown.** The per-invocation `finally` closes the child's stdin and awaits child exit. If the child does not exit within the existing `SHUTDOWN_AWAIT_CAP_MS = 2000` budget, the runtime **kills** it (process-tree kill on Windows; no POSIX signal dependence). `disposeBarrier` settles on **observed child exit**. Teardown is invoked at most once per child and treated as idempotent at the call site. The same `finally` also detaches the one-shot `thetaAbort.signal` listener registered at spawn (per [PIC-63](#pic-63)) and deletes any `PI_THETA_PARAMS_FILE` temp file as the [PIC-60](#pic-60) backstop. If a teardown step itself throws (the stdin close or the kill call), the runtime logs it via `theta/runtime/subagent-dispose-failure` (re-scoped from its former `AgentSession.dispose()`-throw trigger to any subagent-mode teardown-step throw, keeping the stable diagnostics-contract code name; see [Diagnostics](../diagnostics.md)) but does not alter the invocation result ŌĆö the diagnostic is advisory only. On an `Err`/panic teardown the original error is not masked; on a normal-return teardown the parent still observes the theta's `Ok` final value. The `session_shutdown` handler's `SHUTDOWN_AWAIT_CAP_MS` budget covers the entire teardown phase, including the bounded await of child exit and the kill fallback.

<a id="subagent-orphan-prevention"></a>**Orphan prevention** ŌĆö a layered guarantee for a hazard with no in-process counterpart (a child outliving a dead parent is an unattended agent with tool access):

1. *Controlled parent paths* (normal return, errors, `/reload`, session shutdown, catchable process exit): the runtime kills the child per the teardown rule above. **Hard obligation.**
2. *Uncontrolled parent death* (hard kill, OOM): the child's stdin is a pipe from the parent ŌĆö spawned and held open by the parent, unused for data ŌĆö so the OS closes it on parent death regardless of how the parent died. Pi's `-p` mode **exiting on stdin EOF** is the class-2 mechanism ŌĆö a behavioural presupposition verified at the Pi pin and re-audited per version bump (it is not `typeof`-probable). If verification at the pin fails, the recorded fallback is a **child-side parent-PID watchdog**, guaranteed available here because the theta extension always loads in the child: the extension is env-marked (`PI_THETA_SUBAGENT_ROOT=<slug>`) and the marker carries the parent PID, so the extension exits the child when the parent vanishes.
3. *Residual exposure*, documented: between parent death and child exit, an in-flight provider call may run for seconds; that window is irreducible without OS-level tethering (Job Objects / `PR_SET_PDEATHSIG`), which is **rejected** ŌĆö platform-specific native dependencies contradict the codebase's dependency posture.

**Spawn failure** (ENOENT, EPERM, immediate exit) routes as an unanticipated SDK reject does: `theta/runtime/internal-error`, no child to tear down, plus a dedicated `theta/runtime/subagent-spawn-failed` for operator triage. The registry entry from **`ActiveInvocationRegistry`** is removed by the same `finally`.

Subagent-mode tool registration has no separate teardown step: the child registers its own tools at startup and they die with the child process; the parent never registers anything for the subagent. The per-invocation `finally` also settles the `disposeBarrier` promise the runtime registered with the **`ActiveInvocationRegistry`** ŌĆö on observed child exit ŌĆö and removes the registry entry. This is what lets the `session_shutdown` handler's `Promise.allSettled(...)` await observe completion of an in-flight subagent invocation that was cancelled by the teardown handler's `thetaAbort.abort()` call.

Edge cases:

- The `finally` block must run even when the parent's `AbortSignal` fires before the child's first turn ŌĆö spawn-then-immediate-cancel still requires the child to be killed.
- For nested subagent invocations, each level owns its own child process and runs its own `finally`; teardown order is deepest-first, naturally produced by the call stack's `finally` unwinding.
- Teardown must precede the parent observing the result ŌĆö the parent's `match`/`?` arm runs after the child has exited, so no child listener can leak into work the parent does next.

<a id="pic-63"></a> **PIC-63. Cancellation ŌĆö stdin close, bounded grace, process-tree kill.** A `-p` child exposes **no RPC abort command**; cancellation of a subagent child is therefore effected by closing its parent-held stdin pipe (the grace signal), a bounded grace period, and then a process-tree kill. Immediately after the child spawn is initiated and its stdio is wired, the runtime registers a one-shot listener on `thetaAbort.signal` that initiates the grace signal (stdin close / platform signal) to the child. The listener is the sole cancellation-forwarding mechanism in subagent mode. If `thetaAbort.signal.aborted` is already true at the moment the child stdio is ready, the runtime initiates the grace signal synchronously *before* the listener is registered, so the spawn-then-immediate-cancel path remains correct without depending on microtask ordering. The listener is detached in the same per-invocation `finally` that tears the child down (per [PIC-9](#pic-9)). After the grace signal (or if it could not be delivered), teardown proceeds through the bounded-await ŌåÆ process-tree-kill sequence of [PIC-9](#pic-9), so cancellation of a wedged child always terminates within the `SHUTDOWN_AWAIT_CAP_MS` budget via the kill fallback. If initiating the grace signal throws, the runtime traps the error and routes it through `theta/runtime/internal-error` (see [Diagnostics](../diagnostics.md)) but does not alter the invocation result ŌĆö mirroring the teardown-throws rule. The runtime is re-entrancy-safe by construction: the one-shot listener and the one-shot guard on `thetaAbort.abort()` make a double-abort (e.g. operator presses Esc twice) a no-op. A *late failure* of the discarded grace-signal send is one of the abandonable Promises governed by Cancellation's swallowing-handler rule and is silently absorbed (see [Cancellation](../cancellation.md)).

<a id="concurrent-invocation-isolation"></a>**Concurrent-invocation isolation and concurrency disposition.** Concurrent theta invocations within an extension instance ŌĆö tracked as entries in the **`ActiveInvocationRegistry`** above ŌĆö carry mode-qualified isolation and a fixed theta 1.0 concurrency disposition. *Transcript and tool-table isolation are subagent-mode-only:* a subagent-mode invocation runs its own child process, so transcripts, captured `tools:` tables, and derived `AbortSignal`s are per-invocation and do not leak across siblings, while a prompt-mode invocation drives the user session's transcript by definition. <a id="no-invocation-cap"></a> theta 1.0 imposes no maximum on the number of in-flight theta invocations within an extension instance and no fairness or scheduling rule above Pi's event-loop ordering, interposing no admission cap, priority queue, or scheduler between sibling invocations and the event loop.

<a id="pic-22"></a> **PIC-22. Parallel subagent spawn initiation.** This disposition is observable at the parallel-tool-call surface (see [Tool Calls ŌĆö Concurrency](../tool-calls.md#concurrency)): when the model emits N (for any fixed N Ōēź 2) subagent-mode `.theta` callables as parallel tool calls in a single assistant turn, the runtime MUST, before any one of the N invocations returns, **initiate the child-process spawn** for all N invocations (reach the per-child spawn-initiated point). A conformance test witnesses the obligation with a **fake process launcher** whose spawned children block their exit-envelope emission until explicitly released: a conformant runtime has spawn-initiated all N children before any blocked child is released, whereas a runtime that admitted invocations behind a cap, queue, or scheduler would leave at least one child unspawned. The obligation carries over from the RFC 0005 driver unchanged except that "enter each child's first `prompt` command" is replaced by "spawn initiated". The test MUST tolerate real spawn times. The rule constrains the runtime only ŌĆö Pi's own session-level serialisation is upstream and remains in force, and the rule does not promise resource unboundedness. Beyond the absence of an admission cap and scheduler, the theta adds no per-class threshold and no cross-sibling aggregation, throttling, or storm-detection over the resources concurrent theta invocations consume. Those resources ŌĆö runtime-value heap, OS-level file descriptors / sockets / **child-process slots**, and provider-side rate-limit / quota ŌĆö are owned by the host JavaScript runtime, the host OS, and the LLM provider respectively; a process tree's resource cost (each child a full Node runtime with extension discovery loaded) is an OS-owned quantity, not a governed one, and per-process memory footprint is a documented operator consideration, not a governed quantity. A process-count cap or throttle is **rejected** ŌĆö it would contradict both the no-additional-ceilings invariant and the parallel-initiation conformance obligation, and the right value is machine-dependent, which is precisely why the spec assigns it to the OS layer. Simultaneous cold-boot resource contention among N freshly-spawned children falls under this same "resources are owned by the OS layer" disposition. The theta observes only the per-call surfacing of those limits through the existing routing it already pins: catchable host throws ŌĆö the V8 `RangeError` family per `NOCEIL-3` at [Hard ceilings](../hard-ceilings/ceiling-invariants-and-audit.md#no-additional-ceilings), plus catchable OS-level descriptor / port / child-process-slot exhaustion surfaced as throws (a spawn that fails on child-process-slot exhaustion ŌĆö `EAGAIN` / `EMFILE` ŌĆö routes through the spawn-failure path of [PIC-9](#pic-9)) ŌĆö route through `theta/runtime/internal-error` per [Errors and Results ŌĆö Runtime panics](../errors-and-results/error-model.md#runtime-panics); per-query provider throttles (HTTP 429 and equivalents) surface as [`TransportError`](../errors-and-results/queryerror-variants.md#queryerror-variants) on the same page; uncatchable host fatals terminate the host process before any wrap can observe them and emit no `theta/runtime/internal-error` diagnostic. Widening or narrowing this rule is a major-version concern.

The prompt-mode counterpart to this subagent-mode concurrency ŌĆö that prompt-mode bodies execute strictly sequentially within a single user session ŌĆö is owned by [PIC-2 *Prompt-mode sequential execution*](./conversation-drive.md#pic-2) on the prompt-mode driver page.

<a id="subagent-performance-disposition"></a>
**Performance disposition.** Per-invocation process start (Node boot + extension discovery) replaces a cheap in-process construction, multiplied under `par for` fan-out and nested subagents. Bounding context: every subagent invocation also makes at least one LLM call, so the overhead is a fraction of typical wall time ŌĆö but pure overhead. Disposition: ship as-is; benchmark spawn-to-first-envelope at the Pi pin and record the figure in the contract as an **expected-overhead note, not a guarantee**. **Process pooling is rejected**, and the rejection is recorded deliberately: a reused child carries state across invocations (extension module state, temp files, per-theta `--system-prompt`), trading the per-invocation isolation invariant ŌĆö the property this design exists to guarantee ŌĆö for latency. Startup-cost mitigation (e.g. trimming child startup work) is out of scope; if measurement ever shows the overhead to be a binding constraint, a mitigation is designed then, on its own record.
