# Runtime event channel

**System notes.** Echoes (binder result), `Err`-in-prompt-mode notes, binder failure messages, watcher structural-change notes, and operator-facing runtime events all emit through a single call shape:

```ts
pi.sendMessage(
  { customType: "loom-system-note", content, display, details },
  { triggerTurn: false },
);
```

The `display` and `content` arguments vary per the `details` variant documented below ‚Äî they are not fixed to `display: true` or to a non-empty string. A `pi.registerMessageRenderer("loom-system-note", renderer)` registered by the loom extension factory (see [Renderer registration](./extension-bootstrap-and-per-loom.md#renderer-registration)) formats these as one-line, dim-styled notes in the transcript. The custom-type approach (rather than `ctx.ui.notify(...)`) is chosen because notes need to persist in the session transcript for replay and `/tree` navigation, not just appear transiently.

**Delivery surface.** All four `details` payload variants below emit through the single `pi.sendMessage` call above. The runtime has no second channel for `display: false` notes; they land in the same session transcript as `display: true` notes and are filtered out of visible rendering by the renderer (or by Pi's own `display` handling), but remain available to transcript-replay and `/tree` consumers. *Subagent-mode `display: false` cascades* (the subagent top-level `Err` case noted under **Runtime event channel** below) use the same `pi.sendMessage` call against the spawned `AgentSession` ‚Äî i.e., they reach `pi.sendMessage` after the per-loom `sessionManager` swap from [Tool execution from loom code](./host-interfaces-core.md#tool-execution-from-loom-code) has routed messaging at the spawned session ‚Äî so the note lands in the subagent's private transcript, not the parent user session's.

<a id="custom-message-context-entry-presupposition"></a> *Custom-message channel persistence and LLM-context entry.* Per `@earendil-works/pi-coding-agent`'s `dist/core/messages.js` `convertToLlm` transformer, every `CustomMessage` (including `loom-system-note`) is converted to `{ role: "user", content }` on every subsequent provider call. `triggerTurn: false` suppresses the immediate turn fire only; it does not exclude the message from the LLM context window. The `display` flag controls renderer behaviour, not serialization. Loom diagnostics therefore enter the user-session model context durably and contribute to `ctx.getContextUsage()` and compaction decisions. Operators authoring looms should expect parse errors, binder failures, panic notes, and always-log runtime events emitted in a session to be visible to subsequent model turns. The [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) exposes no `CustomMessage.excludeFromContext` opt-out analogous to `BashExecutionMessage.excludeFromContext`; a future Pi minor that adds such a field would be picked up by the `pi-version-bump-procedure` re-validation gate, at which point the runtime SHOULD set the opt-out on the canonical `sendSystemNote` call. This `convertToLlm` conversion behaviour is unobservable from the loom-side SDK surface and is therefore re-audited by editorial review on each Pi minor bump per [version-bump editorial-review checklist item (r)](./version-bump-step2.md#bump-checklist-convert-to-llm-context-entry).

**Empty `content` is legal.** When the variant prescribes `content: ""` (the `display: false` runtime-event case below), the runtime passes the empty string verbatim. `pi.sendMessage`'s `content` parameter accepts any string per Pi's `CustomMessage<T>` interface; no placeholder substitution is required or permitted. The renderer registered for `loom-system-note` MUST tolerate `content === ""` ‚Äî in loom 1.0 this only ever co-occurs with `display: false` and `details: { event: RuntimeEvent }`, so the renderer MAY short-circuit on `display === false` without inspecting `content`.

<a id="system-note-details-shapes"></a> The `details` field carries one of four normative payload shapes, distinguished by which key is present:

- `details: { diagnostics: Diagnostic[] }` ‚Äî a parse / load / type / runtime-panic diagnostic batch, exactly the shape defined in [Diagnostics](../diagnostics.md). The companion `content` is the serialised `<file>:<line>:<col>: <code>: <message>` line(s) for parse / load / type batches; for the runtime-panic case (a single-element batch carrying a `loom/runtime/*` diagnostic) the companion `content` is the user-facing framing `"loom /<name> aborted: <message>"` (or `"loom /<name> aborted with internal error: <message>"` for `loom/runtime/internal-error`) per [Errors and Results ‚Äî Runtime panics](../errors-and-results.md). See **Runtime event channel** below for the partition rule that routes panics through this shape rather than `details: { event }`.
- `details: { event: RuntimeEvent }` ‚Äî a runtime failure event from one of the always-log kinds (see **Runtime event channel** below). The companion `content` is the normative user-facing template from [Slash-Command Invocation ‚Äî Top-level `Err` in prompt mode](../slash-invocation.md#slsh-3) when `display: true`, or omitted (empty string) when `display: false`.
- `details: { structural: { added: string[]; removed: string[] } }` ‚Äî informational note for watcher-observed structural changes (added or removed `.loom` files, settings-array changes that add or remove sources); `added` and `removed` carry absolute file paths from the debounce-window batch. Settings-array edits surface as the resolved `.loom` file paths the source change brought in or removed, not the settings-file path itself, so a single settings edit that adds N sources contributes N entries to `added`. The shape is disjoint from the `diagnostics` and `event` shapes by key, per the additive-only convention below. The companion `content` is the verbatim template defined under **Structural changes** above; `display: true` always.
- `details: { recovery: { looms: string[] } }` ‚Äî informational note for a `looms.binderModel` hot-reload change that would let a previously-failed load succeed; `looms` carries the affected looms' final derived slash names. The shape is disjoint from the `diagnostics`, `event`, and `structural` shapes by key, per the additive-only convention below. The companion `content` is the verbatim template defined under [Binder model and context ‚Äî binder-model hot-reload](../binder/binder-model-and-context.md#binder-model-hot-reload); `display: true` always.

Per-variant `display` / `content` pairings (normative):

| Variant | `display` | `content` |
|---|---|---|
| `details: { diagnostics: Diagnostic[] }`, parse / load / type batch | `true` | serialised `<file>:<line>:<col>: <code>: <message>` line(s) (multi-line for batches) ‚Äî non-empty |
| `details: { diagnostics: [Diagnostic] }`, runtime panic (single-element batch, `loom/runtime/*` code) | `true` | `"loom /<name> aborted: <message>"` (or `"loom /<name> aborted with internal error: <message>"` for `loom/runtime/internal-error`) per [Errors and Results ‚Äî Runtime panics](../errors-and-results.md) ‚Äî non-empty |
| `details: { event: RuntimeEvent }`, top-level cascade in prompt mode | `true` | normative user-facing template per [Slash-Command Invocation ‚Äî Top-level `Err` in prompt mode](../slash-invocation.md#slsh-3) ‚Äî non-empty |
| `details: { event: RuntimeEvent }`, author-handled or subagent-mode cascade | `false` | `""` (empty string, verbatim) |
| `details: { structural: { added; removed } }` | `true` | verbatim template from **Structural changes** above ‚Äî non-empty |
| `details: { recovery: { looms } }`, `looms.binderModel` hot-reload recovery | `true` | verbatim template from [Binder model and context ‚Äî binder-model hot-reload](../binder/binder-model-and-context.md#binder-model-hot-reload) ‚Äî non-empty |

The four `details` shapes are disjoint by key; renderers MUST NOT assume more than one is present. Future payload shapes added to this channel are additive and disjoint by the same convention. Future variants MAY widen the `display: false` + `content: ""` combination to other `details` shapes additively; the renderer's `display === false` short-circuit above remains safe under that widening because it does not inspect `content`.

**Runtime event channel.** The **always-log set** ‚Äî `QueryError` `kind` values, binder failure causes, and `loom/runtime/*` panic codes ‚Äî emit a structured note through the `loom-system-note` channel exactly once per occurrence, regardless of whether the author matched the `Err`, propagated it via `?`, or discarded it via `let _ =`. The always-log set partitions by routing channel: members in **group A** emit `details: { event: RuntimeEvent }`; members in **group B** emit `details: { diagnostics: Diagnostic[] }`. A given failure emits through exactly one shape ‚Äî there is no fan-out across the two `details` variants for a single failure.

`display: false` gates only renderer visibility; the underlying `CustomMessage` still enters subsequent provider calls per **Delivery surface** above (Pi's `convertToLlm` transform converts every `CustomMessage` to a `user`-role transcript entry on every subsequent provider call, with no opt-out under the loom 1.0 SDK pin). **Note (non-normative):** because of this transform, operators consuming the channel see all `loom-system-note` content (regardless of `display`) as durable session-context input rather than transient renderer output.

*Engine-assumption carve-out.* The exactly-once emission guarantee assumes the [Runtime Value Model ‚Äî JavaScript engine assumptions](../runtime-value-model.md#javascript-engine-assumptions) hold (IEEE-754 numbers, native `Map`/`Set`, native `JSON.stringify`, and the primitive-equality relation defined there ‚Äî `NaN == NaN`, `+0 == -0` ‚Äî a non-checked invariant). Under a violation of those assumptions, emission may be silently dropped (e.g. a corrupted `Map` loses the dedup key), duplicated, or skipped entirely (e.g. a `pi.sendMessage` host throw bypasses the helper); some manifestations land on `loom/runtime/internal-error` via the runtime-defect surface in [Errors and Results ‚Äî Runtime panics](../errors-and-results.md), but silent value corruption has no observable signal. **Note (non-normative):** for operators, a missing terminal event is therefore ambiguous ‚Äî it indicates one of (a) the loom did not fail; (b) the loom failed with a kind not in the always-log set; or (c) an engine-assumption violation ‚Äî and there is no in-band signal that distinguishes (a) from (c).

Group A ‚Äî `details: { event: RuntimeEvent }`:

- `transport`
- `model_tool`
- `code_tool`
- `tool_loop_exhausted`
- `invoke_infra`
- Binder failures (every row of [Binder ‚Äî Failure modes](../binder.md), including `needs_info`, `ambiguous`, malformed envelope, AJV validation, transport, cancellation)

Group B ‚Äî `details: { diagnostics: Diagnostic[] }`:

- Runtime panics ‚Äî every row of [Diagnostics ‚Äî `loom/runtime/*`](../diagnostics.md) plus the four console.error-routed teardown-handler `loom/host/session-shutdown-*` codes enumerated in the exclusion list below, routed through the `details: { diagnostics: [...] }` shape per the panic-emission rule under **Deduplication and lifetime rules** below, with five console.error-routed teardown-handler exclusions: `loom/runtime/reload-teardown-timeout`, `loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, and `loom/host/session-shutdown-runtime-degraded` (per the **Edge cases** bullet under **Extension entry point** sub-step 4 above and the **Persistent diagnostics** paragraph in [Diagnostics](../diagnostics.md)). A new diagnostic added to either of those sinks MUST be added to this exclusion list in the same edit. The `RuntimeEvent` payload shape and the `(kind, query_site, message, occurred_at)` dedup key defined below apply to group A only and have no analogue for group B.

`validation`, `context_overflow`, `cancelled`, and `invoke_callee` are deliberately **not** in the always-log set: `validation` is a query-internal repair signal whose final outcome is what matters (the `attempts` field on the eventual terminal event captures the count); `context_overflow` and `cancelled` are user/operator-initiated and self-explanatory in context; `invoke_callee` wraps an inner `Err` whose origin already emitted its own runtime event at the originating site.

*Discard-site disposition.* The four excluded kinds emit nothing on this channel even at a discard site (`let _ = @"‚Ä¶"`, `void`-tail-expression discard, or any other disposition that drops the `Err`). The exclusion rationale survives the discard case for each kind: for `validation`, a discarded validation failure is by construction a programming error ‚Äî the author wrote `let _ = @<Schema>\`‚Ä¶\`` and ignored the schema mismatch; the diagnostic at the originating site (when one fires ‚Äî depth-5 violations and respond-repair failures both have `loom/runtime/*` codes available) plus the absence of any user-visible outcome is the intended signal. For `cancelled`, the author's choice to discard a user/operator-initiated outcome is itself the disposition. For `context_overflow`, the author's choice to discard a model-state outcome is itself the disposition. For `invoke_callee`, the inner `Err`'s origin-site emission is unchanged by the wrapper's disposition. The symmetric statement on the consumer side is owned by [Query ‚Äî Observability of discarded results](../query.md).

<a id="success-side-null-policy"></a>*Success-side null-policy.* A loom terminating with `Ok(v)` ‚Äî including the case where a child loom's `Ok(v)` flows to its `invoke` parent per [Invocation ‚Äî Final-value propagation across callees](../invocation.md#final-value-propagation) ‚Äî emits no `loom-system-note` keyed on the `Ok(v)` outcome. This follows by construction from the always-log set above, whose members are all `QueryError` kinds (group A) or runtime panics (group B); an `Ok(v)` termination is neither, so no always-log entry has a satisfied emission predicate for it. Earlier mid-run emissions on the same channel ‚Äî always-log members raised inside the loom body before its terminating expression ‚Äî are governed by the **Runtime event channel** paragraph above and are unaffected; this paragraph adds no new emission predicate and removes none. The operator-observable surfaces on the success side are mode-dependent: in prompt mode the driven conversation per [Slash-Command Invocation](../slash-invocation.md) is the user-facing surface (any value the author wants the user to see should be issued by the author as a final query), and the `Ok(v)` return value reaches a programmatic caller only when one exists (none on a user-typed slash dispatch; an `invoke` parent otherwise); in subagent mode there is no user-facing surface for the terminating value (the intermediate subagent transcript stays private per [Slash-Command Invocation ‚Äî User-visible streaming](../slash-invocation.md#slsh-2)), and the `Ok(v)` return value reaches a programmatic caller only when one exists (none on a user-typed slash dispatch; an `invoke` parent otherwise, per [Invocation ‚Äî Final-value propagation across callees](../invocation.md#final-value-propagation)). The two pre-evaluation surfaces ‚Äî the binder echo on `bind_echo: true` per [Slash-Command Argument Binding](../binder.md) and the no-params overflow note per [Slash-Command Invocation ‚Äî No-params overflow](../slash-invocation.md#slsh-1) ‚Äî emit before the loom's terminating expression evaluates and are unaffected by this paragraph regardless of the run's outcome.

The `RuntimeEvent` payload shape is normative and additive-only:

```ts
type RuntimeEvent = {
  kind: string;                       // QueryError.kind, the binder failure cause, or the loom/runtime/* code for panics
  code?: string;                      // diagnostic code when one applies (loom/runtime/*, loom/load/*, etc.)
  loom: string;                       // slash name of the loom that owned the failure ("/code-review")
  invocation_id: string;              // per-invocation UUID sourced from the ActiveInvocationRegistry entry's `invocationId` field (canonical lowercase 8-4-4-4-12 hex form per ¬ß7 of Diagnostics); distinguishes concurrent sibling invocations of the same loom on operator-visible surfaces
  query_site?: { file: string; line: number; column: number }; // source location of the @-template, tool call, or invoke; absent for binder failures (the binder runs before any loom code)
  discard_site?: { file: string; line: number; column: number }; // source location of the discarding `let _ =` / `void`-tail expression; present only on discarded-query events, omitted (not `null` or empty) on all other events; not part of the dedup key (each discard is a distinct occurrence stamped with its own `occurred_at`, so two discards of the same originating template stay distinct)
  message: string;                    // the same message string surfaced through the user-facing template (or the panic message template, for panics that take the diagnostic shape)
  attempts?: number;                  // populated for `validation` events on respond-repair exhaustion; absent otherwise
  tokens_used?: number;               // populated for `context_overflow` events when the provider supplies the count; absent otherwise
  masked?: string[];                  // hard-ceiling co-fire enumeration; see "Hard-ceiling co-fire" below
  occurred_at: number;                // Unix epoch ms, stamped at the originating emission site via Clock.wallNow()
};
```

The channel is operator-facing: events emit with `display: false` when the author handled the `Err` (matched, discarded with `let _ =`, or propagated via `?` to a non-top-level frame that itself handled it), and with `display: true` when the failure cascades out of the slash-command boundary as a top-level `Err` in prompt mode (in which case the `content` carries the normative user-facing template per [Slash-Command Invocation ‚Äî Top-level `Err` in prompt mode](../slash-invocation.md#slsh-3)). Subagent-mode top-level `Err` cascades likewise emit with `display: false` (the subagent transcript is private; see [Slash-Command Invocation ‚Äî User-visible streaming](../slash-invocation.md#slsh-2)).

Deduplication and lifetime rules:

- A `?`-propagation chain emits exactly once at the originating site, not at each rethrow. A failure that is created at site A, propagated through frames B and C via `?`, and finally cascades out as a top-level `Err` at frame D produces one runtime event (origin: A) plus the user-facing top-level note at D when `display: true` applies; both share the same `RuntimeEvent` payload. The runtime emits the same `RuntimeEvent` payload at most twice per origin: once at the originating site (always), and once again at the boundary as the `details: { event }` payload of the user-facing top-level note when a cascade applies. Both emissions are byte-identical including `occurred_at`. Consumers that aggregate the event stream may therefore deduplicate on `(kind, query_site, message, occurred_at)` to collapse the cascade twin; this is a non-normative consumer concern and not part of loom 1.0 conformance. Re-emissions for symmetry MUST copy the originating `RuntimeEvent` instance verbatim ‚Äî including `occurred_at` ‚Äî rather than re-stamping. Two emissions from the same `query_site` with the same `kind` and `message` but distinct `occurred_at` values represent two distinct occurrences.
- Respond-repair attempts on a typed query (see [Query ‚Äî Schema-validation respond-repair](../query.md)) do not emit per-failed-attempt events. Only the terminal outcome of the query is observable as a runtime event ‚Äî either a `validation` event with `attempts` set to the exhausted count (when respond-repair runs out), or a non-validation event from the always-log set (when a respond-repair follow-up itself fails for a transport / tool / cancellation / overflow reason and propagates per [Query ‚Äî Non-validation failures during a respond-repair follow-up](../query.md)).
- A panic emits exactly one `loom-system-note` per top-level panic with `details: { diagnostics: [Diagnostic] }` carrying a single `loom/runtime/*` diagnostic (message per the registered template in [Diagnostics](../diagnostics.md)). The companion `content` field on that same note carries the user-facing framing `"loom /<name> aborted: <message>"` (or `"loom /<name> aborted with internal error: <message>"` for `loom/runtime/internal-error`) per [Errors and Results ‚Äî Runtime panics](../errors-and-results.md), where `<message>` is the registered diagnostic message template. The diagnostic-shape `details` and the `aborted:` framing in `content` are the **same** emission, not two notes; there is no separate `details: { event }` emission for panics, and the cascade-twin dedup key above does not apply to group B. The fallback chain to `ctx.ui.notify` and `loom/runtime/system-note-delivery-failed` (defined below) still applies if `pi.sendMessage` throws on the panic note; the fallback's diagnostic emission is independent of the panic note and not subject to any dedup key.
- Tool-host emission ownership: when a Pi tool that the loom invoked (either code-side via `<name>(args)` or model-side inside a query's tool-call loop) emits its own diagnostics, those are independent of the loom-runtime event. The loom runtime owns `code_tool` and `model_tool` events; the tool host owns whatever it emits internally. Renderers MUST NOT cross-correlate the two.

Aggregation, latency histograms, per-loom token reports, retention policies, and a consumer-facing read API for the event stream are deferred to [Future Considerations](../future-considerations.md). loom 1.0 ships the always-log set and the payload shape only; the channel is intentionally write-only at the spec level, with downstream consumers free to read from Pi's session transcript via existing surfaces.

<a id="pic-1"></a> **PIC-1. Hard-ceiling co-fire (`masked`).** The optional `masked` field on `RuntimeEvent` (and on the `Diagnostic` shape per [Diagnostics ‚Äî `masked` field](../diagnostics/diagnostic-shape.md#masked-field-hard-ceiling-co-fire)) carries the hard-ceiling co-fire enumeration. This rule is the normative owner of the per-site reachable mask domain, the V1 reachable predicate, the pure-read MUST, the verbatim-copy obligation, and the dedup-key non-inclusion rule; the four-ceiling list, the ordering rules CIO-1‚Ä¶CIO-6, and the wire-location split are owned by [Hard Runtime Ceilings ‚Äî `masked` field](../hard-ceilings/ceilings-3-and-4.md#masked-field) and [Hard Runtime Ceilings ‚Äî Interaction between ceilings](../hard-ceilings/ceilings-3-and-4.md#ceiling-interaction-order).

*(a) Closed identifier set.* Each `masked` entry is one of the four string identifiers `"ceiling#1"`, `"ceiling#2"`, `"ceiling#3"`, `"ceiling#4"`, naming a sibling hard ceiling whose precondition was *also* satisfied at the same check site at the moment the surfaced ceiling fired.

*(b) Canonical-absence rule.* The field is **omitted** when no co-fire occurred; implementations MUST NOT emit `masked: []`. A consumer reading `RuntimeEvent` (or a `Diagnostic`) and finding the property absent MUST treat it as identical to absence ‚Äî there is no tri-state.

*(c) Co-fire-eligible surfaces and per-site reachable mask domain.* `masked` is populated only on runtime events whose `kind` is one of the runtime-class hard-ceiling surfaces ‚Äî in V1 this is (i) the `validation` event when its `cause` is `"schema_validation"` (ceiling #4 surface at the typed-query response / model-driven `tool_use` args / `params` validation / `invoke<T>` return value boundary, per CIO-3 in [Hard Runtime Ceilings](../hard-ceilings/ceilings-3-and-4.md#ceiling-interaction-order)), (ii) the `code_tool` event when its underlying `CodeToolError.cause` is `"validation"` and the `validation_errors[].schema_keyword` is `"maxDepth"` (ceiling #4 surface at the code-driven `<name>(args)` args boundary per CIO-3 ‚Äî the fifth row of ceiling #4's per-boundary table), and (iii) the `tool_loop_exhausted` event (ceiling #2 surface at the tool-call-round boundary). The diagnostic-shape co-fire-eligible surface is the `loom/runtime/invoke-depth-exceeded` panic per [Diagnostics](../diagnostics.md). The per-site reachable mask domain in V1 is:

| Surfacing site | Surfaced ceiling | Reachable mask domain |
|---|---|---|
| Typed-query response / model-driven `tool_use` args / `params` validation / `invoke<T>` return value AJV boundary (CIO-3) | ceiling #4 | `{ceiling#2}` (only on the typed-query response branch and only when the *V1 reachable predicate* below holds; empty on the other three sub-rows of this AJV boundary because no `tool_loop` slot has been incremented yet for the surfacing event's frame) |
| Code-driven `<name>(args)` AJV boundary (CIO-3, fifth row of ceiling #4's per-boundary table) | ceiling #4 | empty (the call site is not inside a tool-call-round counter context, so ceiling #2's slot accounting is not reachable; ceilings #1 and #3 are unreachable per CIO-1 / CIO-2 / CIO-5) |
| `invoke` entry (CIO-2) | ceiling #1 | empty (the callee body has not yet run, so neither #4's depth-walk nor #2's tool-call round slot accounting has been reached for this frame) |
| Tool-call-round boundary (CIO-4) | ceiling #2 | empty (ceiling #4's depth-walk runs at every AJV boundary earlier in the round per CIO-3, so any depth-6 payload would have already surfaced #4 and short-circuited the round before #2's slot accounting) |
| Slash-load `params` AJV (ceiling #3, load-time) | ceiling #3 | empty (ceiling #3's load-time surfaces always omit `masked` by construction. The slash-load `params` cross-ceiling sub-case ‚Äî where ceiling #4's depth-walk *did* fire at the [post-default-merge AJV validation](../binder/defaulting-system-note-echo.md#post-default-merge-ajv-validation) hook before being routed to ceiling #3's AJV-on-`args` arm per CIO-1 and the per-boundary table at [Hard Runtime Ceilings ‚Äî Per-boundary destination/surface table](../hard-ceilings/ceilings-3-and-4.md#ceiling-4-table) ‚Äî still omits `masked`: the originating ceiling is recoverable from the rendered note's `<ajv-summary>` placeholder (which names `maxDepth` and the canonical depth-violation message) per [Slash-Command Argument Binding ‚Äî Failure-mode templates](../binder/determinism-cancellation-failure.md#failure-mode-templates-normative)) |

In loom 1.0 the only non-empty reachable mask is therefore `["ceiling#2"]`, reachable only on the `validation` event at the typed-query response sub-row of CIO-3's AJV boundary. Every other co-fire-eligible surface MUST omit `masked`.

*(d) V1 reachable predicate (the only co-fire that fires in V1).* On a `validation` event with `cause: "schema_validation"` raised at the typed-query response boundary, `masked` carries `["ceiling#2"]` *if and only if* the surfaced event was raised on a forced respond turn whose origin round, after CIO-4's slot increment for the just-completed free-phase round, leaves the `tool_loop` slot count equal to `max_rounds` (i.e. the forced respond turn is the final permitted slot). In every other event ‚Äî including a respond-repair follow-up's depth-6 forced respond turn (each follow-up gets a fresh `tool_loop` budget per [Query ‚Äî Tool-call loop bound](../query/query-tool-loop.md#tool-call-loop-bound), so the predicate is evaluated against the follow-up's own slot count, not the parent query's), a typed query with `max_rounds: 0` (the model gets an empty tool-set during the free phase, so the forced respond turn is the *only* turn and the predicate is unreachable; `masked` is always omitted), and any non-typed-query-response site ‚Äî `masked` is omitted. The predicate collapses to two scalar reads: the post-increment `tool_loop` slot count and the surfacing turn kind (forced-respond vs. free-phase). Both scalars are state the two-phase tool-loop driver already maintains.

*(e) Pure-read MUST.* Detection at the surfaced site MUST be a pure read of the sibling preconditions: evaluating the predicate above MUST NOT advance the `tool_loop` counter, mutate round bookkeeping, re-invoke ceiling #4's depth-walk, or trip a second emission. Implementations MAY cache the two scalars at the moment the surfacing event is constructed; they MUST NOT re-derive them by re-running ceiling-checking code paths.

*(f) Verbatim-copy obligation on cascade-twin re-emission.* The cascade-twin re-emission rule under **Deduplication and lifetime rules** above (a `?`-propagation chain that cascades to a top-level boundary emits the originating `RuntimeEvent` payload twice ‚Äî once at the origin, once at the boundary) MUST copy the originating instance verbatim including `masked`. Re-deriving `masked` at the boundary site is forbidden ‚Äî the originating site is the only authoritative source, and the boundary site has no access to the per-frame scalars the predicate above reads.

*(g) Dedup-key non-inclusion rule.* The dedup tuple under **Deduplication and lifetime rules** above is `(kind, query_site, message, occurred_at)`; consumers MUST NOT include `masked` in the dedup tuple. Two emissions that differ only in `masked` represent the same origin and a malformed re-emission, not two distinct occurrences.

The wire location of `masked` (`details.event.masked` for runtime-event surfaces, `details.masked` for diagnostic-shape surfaces) is owned by [Hard Runtime Ceilings ‚Äî `masked` field](../hard-ceilings/ceilings-3-and-4.md#masked-field). A worked example exhibiting a non-empty `masked` payload is at [Query ‚Äî Tool-call loop bound, *Worked example: depth-6 forced respond at `max_rounds`*](../query/query-tool-loop.md#tool-call-loop-bound).

The `pi.sendMessage` call for `loom-system-note` is treated as best-effort. `pi.sendMessage` returns `void` (synchronous); the runtime MUST NOT `await` it and MUST NOT attach a `.catch` handler. The best-effort fallback below covers synchronous throws only. If it throws, the runtime falls back in this order:

1. `ctx.ui.notify(content, "error")` ‚Äî a transient surface so the user still sees the message in the current session, even if it does not persist in the transcript. **Skipped when `display: false`**: notifying the user transiently about an event whose author handled the underlying `Err` (or a subagent-private cascade) defeats the purpose of `display: false`; the fallback proceeds straight to step 2. This step is also skipped when the original note's `content` is `""`, since `ctx.ui.notify("", "error")` carries no signal ‚Äî but in loom 1.0 this case is fully covered by the `display: false` skip rule above (the only `content: ""` variant is the `display: false` runtime-event case).
2. A `loom/runtime/system-note-delivery-failed` diagnostic emitted through the standard diagnostics channel (see [Diagnostics](../diagnostics.md)), with `message` set to the original note's `content` (which MAY be `""`) and `hint` set to the underlying `sendMessage` error's message. This diagnostic is itself best-effort: if both channels fail, the failure is logged to `console.error` and execution continues.

The fallback path is taken on any thrown value from `sendMessage`; it does not retry the original call. The fallback never aborts the slash-command handler or the spawned subagent session. Implementers must guard against re-entry: if a future `loom/runtime/*` handler ever routes diagnostics back through `loom-system-note`, the diagnostic step in this fallback MUST NOT re-invoke `pi.sendMessage`. `ctx.ui.notify` itself can throw (e.g. in print mode where Pi's UI is not attached); wrap it in the same try/catch and proceed to the diagnostic step. For panic-routed notes, the original panic message MUST be included in the final-resort `console.error` log so post-mortem debugging retains the stack.
