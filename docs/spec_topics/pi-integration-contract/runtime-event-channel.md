# Runtime event channel

The `RuntimeEvent` payload shape is normative and additive-only:

```ts
type RuntimeEvent = {
  kind: string;                       // QueryError.kind, the binder failure cause, or the loom/runtime/* code for panics
  code?: string;                      // diagnostic code when one applies (loom/runtime/*, loom/load/*, etc.)
  loom: string;                       // slash name of the loom that owned the failure ("/code-review")
  invocation_id: string;              // per-invocation UUID sourced from the ActiveInvocationRegistry entry's `invocationId` field (canonical lowercase 8-4-4-4-12 hex form per §7 of Diagnostics); distinguishes concurrent sibling invocations of the same loom on operator-visible surfaces
  query_site?: { file: string; line: number; column: number }; // source location of the @-template, tool call, or invoke; absent for binder failures (the binder runs before any loom code)
  discard_site?: { file: string; line: number; column: number }; // source location of the discarding `let _ =` / `void`-tail expression; present only on discarded-query events, omitted (not `null` or empty) on all other events; not part of the dedup key (each discard is a distinct occurrence stamped with its own `occurred_at`, so two discards of the same originating template stay distinct)
  message: string;                    // the same message string surfaced through the user-facing template (or the panic message template, for panics that take the diagnostic shape)
  attempts?: number;                  // populated for `validation` events on respond-repair exhaustion; absent otherwise
  tokens_used?: number;               // populated for `context_overflow` events when the provider supplies the count; absent otherwise
  masked?: string[];                  // hard-ceiling co-fire enumeration; see "Hard-ceiling co-fire" below
  occurred_at: number;                // Unix epoch ms, stamped at the originating emission site via Clock.wallNow()
};
```

The channel is operator-facing: events emit with `display: false` when the author handled the `Err` (matched, discarded with `let _ =`, or propagated via `?` to a non-top-level frame that itself handled it), and with `display: true` when the failure cascades out of the slash-command boundary as a top-level `Err` in prompt mode (in which case the `content` carries the normative user-facing template per [Slash-Command Invocation — Top-level `Err` in prompt mode](../slash-invocation.md#slsh-3)). Subagent-mode top-level `Err` cascades likewise emit with `display: false` (the subagent transcript is private; see [Slash-Command Invocation — User-visible streaming](../slash-invocation.md#slsh-2)).

Deduplication and lifetime rules:

- A `?`-propagation chain emits exactly once at the originating site, not at each rethrow. A failure that is created at site A, propagated through frames B and C via `?`, and finally cascades out as a top-level `Err` at frame D produces one runtime event (origin: A) plus the user-facing top-level note at D when `display: true` applies; both share the same `RuntimeEvent` payload. The runtime emits the same `RuntimeEvent` payload at most twice per origin: once at the originating site (always), and once again at the boundary as the `details: { event }` payload of the user-facing top-level note when a cascade applies. Both emissions are byte-identical including `occurred_at`. Consumers that aggregate the event stream may therefore deduplicate on `(kind, query_site, message, occurred_at)` to collapse the cascade twin; this is a non-normative consumer concern and not part of loom 1.0 conformance. Re-emissions for symmetry MUST copy the originating `RuntimeEvent` instance verbatim — including `occurred_at` — rather than re-stamping. Two emissions from the same `query_site` with the same `kind` and `message` but distinct `occurred_at` values represent two distinct occurrences.
- Respond-repair attempts on a typed query (see [Query — Schema-validation respond-repair](../query.md)) do not emit per-failed-attempt events. Only the terminal outcome of the query is observable as a runtime event — either a `validation` event with `attempts` set to the exhausted count (when respond-repair runs out), or a non-validation event from the always-log set (when a respond-repair follow-up itself fails for a transport / tool / cancellation / overflow reason and propagates per [Query — Non-validation failures during a respond-repair follow-up](../query.md)).
- A panic emits exactly one `loom-system-note` per top-level panic with `details: { diagnostics: [Diagnostic] }` carrying a single `loom/runtime/*` diagnostic (message per the registered template in [Diagnostics](../diagnostics.md)). The companion `content` field on that same note carries the user-facing framing `"loom /<name> aborted: <message>"` (or `"loom /<name> aborted with internal error: <message>"` for `loom/runtime/internal-error`) per [Errors and Results — Runtime panics](../errors-and-results.md), where `<message>` is the registered diagnostic message template. The diagnostic-shape `details` and the `aborted:` framing in `content` are the **same** emission, not two notes; there is no separate `details: { event }` emission for panics, and the cascade-twin dedup key above does not apply to group B. The fallback chain to `ctx.ui.notify` and `loom/runtime/system-note-delivery-failed` (defined below) still applies if `pi.sendMessage` throws on the panic note; the fallback's diagnostic emission is independent of the panic note and not subject to any dedup key.
- Tool-host emission ownership: when a Pi tool that the loom invoked (either code-side via `<name>(args)` or model-side inside a query's tool-call loop) emits its own diagnostics, those are independent of the loom-runtime event. The loom runtime owns `code_tool` and `model_tool` events; the tool host owns whatever it emits internally. Renderers MUST NOT cross-correlate the two.

Aggregation, latency histograms, per-loom token reports, retention policies, and a consumer-facing read API for the event stream are deferred to [Future Considerations](../future-considerations.md). loom 1.0 ships the always-log set and the payload shape only; the channel is intentionally write-only at the spec level, with downstream consumers free to read from Pi's session transcript via existing surfaces.

<a id="pic-1"></a> **PIC-1. Hard-ceiling co-fire (`masked`).** The optional `masked` field on `RuntimeEvent` (and on the `Diagnostic` shape per [Diagnostics — `masked` field](../diagnostics/diagnostic-shape.md#masked-field-hard-ceiling-co-fire)) carries the hard-ceiling co-fire enumeration. This rule is the normative owner of the per-site reachable mask domain, the V1 reachable predicate, the pure-read MUST, the verbatim-copy obligation, and the dedup-key non-inclusion rule; the four-ceiling list, the ordering rules CIO-1…CIO-6, and the wire-location split are owned by [Hard Runtime Ceilings — `masked` field](../hard-ceilings/ceilings-3-and-4.md#masked-field) and [Hard Runtime Ceilings — Interaction between ceilings](../hard-ceilings/ceilings-3-and-4.md#ceiling-interaction-order).

*(a) Closed identifier set.* Each `masked` entry is one of the four string identifiers `"ceiling#1"`, `"ceiling#2"`, `"ceiling#3"`, `"ceiling#4"`, naming a sibling hard ceiling whose precondition was *also* satisfied at the same check site at the moment the surfaced ceiling fired.

*(b) Canonical-absence rule.* The field is **omitted** when no co-fire occurred; implementations MUST NOT emit `masked: []`. A consumer reading `RuntimeEvent` (or a `Diagnostic`) and finding the property absent MUST treat it as identical to absence — there is no tri-state.

*(c) Co-fire-eligible surfaces and per-site reachable mask domain.* `masked` is populated only on runtime events whose `kind` is one of the runtime-class hard-ceiling surfaces — in V1 this is (i) the `validation` event when its `cause` is `"schema_validation"` (ceiling #4 surface at the typed-query response / model-driven `tool_use` args / `params` validation / `invoke<T>` return value boundary, per CIO-3 in [Hard Runtime Ceilings](../hard-ceilings/ceilings-3-and-4.md#ceiling-interaction-order)), (ii) the `code_tool` event when its underlying `CodeToolError.cause` is `"validation"` and the `validation_errors[].schema_keyword` is `"maxDepth"` (ceiling #4 surface at the code-driven `<name>(args)` args boundary per CIO-3 — the fifth row of ceiling #4's per-boundary table), and (iii) the `tool_loop_exhausted` event (ceiling #2 surface at the tool-call-round boundary). The diagnostic-shape co-fire-eligible surface is the `loom/runtime/invoke-depth-exceeded` panic per [Diagnostics](../diagnostics.md). The per-site reachable mask domain in V1 is:

| Surfacing site | Surfaced ceiling | Reachable mask domain |
|---|---|---|
| Typed-query response / model-driven `tool_use` args / `params` validation / `invoke<T>` return value AJV boundary (CIO-3) | ceiling #4 | `{ceiling#2}` (only on the typed-query response branch and only when the *V1 reachable predicate* below holds; empty on the other three sub-rows of this AJV boundary because no `tool_loop` slot has been incremented yet for the surfacing event's frame) |
| Code-driven `<name>(args)` AJV boundary (CIO-3, fifth row of ceiling #4's per-boundary table) | ceiling #4 | empty (the call site is not inside a tool-call-round counter context, so ceiling #2's slot accounting is not reachable; ceilings #1 and #3 are unreachable per CIO-1 / CIO-2 / CIO-5) |
| `invoke` entry (CIO-2) | ceiling #1 | empty (the callee body has not yet run, so neither #4's depth-walk nor #2's tool-call round slot accounting has been reached for this frame) |
| Tool-call-round boundary (CIO-4) | ceiling #2 | empty (ceiling #4's depth-walk runs at every AJV boundary earlier in the round per CIO-3, so any depth-6 payload would have already surfaced #4 and short-circuited the round before #2's slot accounting) |
| Slash-load `params` AJV (ceiling #3, load-time) | ceiling #3 | empty (ceiling #3's load-time surfaces always omit `masked` by construction. The slash-load `params` cross-ceiling sub-case — where ceiling #4's depth-walk *did* fire at the [post-default-merge AJV validation](../binder/defaulting-system-note-echo.md#post-default-merge-ajv-validation) hook before being routed to ceiling #3's AJV-on-`args` arm per CIO-1 and the per-boundary table at [Hard Runtime Ceilings — Per-boundary destination/surface table](../hard-ceilings/ceilings-3-and-4.md#ceiling-4-table) — still omits `masked`: the originating ceiling is recoverable from the rendered note's `<ajv-summary>` placeholder (which names `maxDepth` and the canonical depth-violation message) per [Slash-Command Argument Binding — Failure-mode templates](../binder/determinism-cancellation-failure.md#failure-mode-templates-normative)) |

In loom 1.0 the only non-empty reachable mask is therefore `["ceiling#2"]`, reachable only on the `validation` event at the typed-query response sub-row of CIO-3's AJV boundary. Every other co-fire-eligible surface MUST omit `masked`.

*(d) V1 reachable predicate (the only co-fire that fires in V1).* On a `validation` event with `cause: "schema_validation"` raised at the typed-query response boundary, `masked` carries `["ceiling#2"]` *if and only if* the surfaced event was raised on a forced respond turn whose origin round, after CIO-4's slot increment for the just-completed free-phase round, leaves the `tool_loop` slot count equal to `max_rounds` (i.e. the forced respond turn is the final permitted slot). In every other event — including a respond-repair follow-up's depth-6 forced respond turn (each follow-up gets a fresh `tool_loop` budget per [Query — Tool-call loop bound](../query/query-tool-loop.md#tool-call-loop-bound), so the predicate is evaluated against the follow-up's own slot count, not the parent query's), a typed query with `max_rounds: 0` (the model gets an empty tool-set during the free phase, so the forced respond turn is the *only* turn and the predicate is unreachable; `masked` is always omitted), and any non-typed-query-response site — `masked` is omitted. The predicate collapses to two scalar reads: the post-increment `tool_loop` slot count and the surfacing turn kind (forced-respond vs. free-phase). Both scalars are state the two-phase tool-loop driver already maintains.

*(e) Pure-read MUST.* Detection at the surfaced site MUST be a pure read of the sibling preconditions: evaluating the predicate above MUST NOT advance the `tool_loop` counter, mutate round bookkeeping, re-invoke ceiling #4's depth-walk, or trip a second emission. Implementations MAY cache the two scalars at the moment the surfacing event is constructed; they MUST NOT re-derive them by re-running ceiling-checking code paths.

*(f) Verbatim-copy obligation on cascade-twin re-emission.* The cascade-twin re-emission rule under **Deduplication and lifetime rules** above (a `?`-propagation chain that cascades to a top-level boundary emits the originating `RuntimeEvent` payload twice — once at the origin, once at the boundary) MUST copy the originating instance verbatim including `masked`. Re-deriving `masked` at the boundary site is forbidden — the originating site is the only authoritative source, and the boundary site has no access to the per-frame scalars the predicate above reads.

*(g) Dedup-key non-inclusion rule.* The dedup tuple under **Deduplication and lifetime rules** above is `(kind, query_site, message, occurred_at)`; consumers MUST NOT include `masked` in the dedup tuple. Two emissions that differ only in `masked` represent the same origin and a malformed re-emission, not two distinct occurrences.

The wire location of `masked` (`details.event.masked` for runtime-event surfaces, `details.masked` for diagnostic-shape surfaces) is owned by [Hard Runtime Ceilings — `masked` field](../hard-ceilings/ceilings-3-and-4.md#masked-field). A worked example exhibiting a non-empty `masked` payload is at [Query — Tool-call loop bound, *Worked example: depth-6 forced respond at `max_rounds`*](../query/query-tool-loop.md#tool-call-loop-bound).

The `pi.sendMessage` call for `loom-system-note` is treated as best-effort. `pi.sendMessage` returns `void` (synchronous); the runtime MUST NOT `await` it and MUST NOT attach a `.catch` handler. The best-effort fallback below covers synchronous throws only. If it throws, the runtime falls back in this order:

1. `ctx.ui.notify(content, "error")` — a transient surface so the user still sees the message in the current session, even if it does not persist in the transcript. **Skipped when `display: false`**: notifying the user transiently about an event whose author handled the underlying `Err` (or a subagent-private cascade) defeats the purpose of `display: false`; the fallback proceeds straight to step 2. This step is also skipped when the original note's `content` is `""`, since `ctx.ui.notify("", "error")` carries no signal — but in loom 1.0 this case is fully covered by the `display: false` skip rule above (the only `content: ""` variant is the `display: false` runtime-event case).
2. A `loom/runtime/system-note-delivery-failed` diagnostic emitted through the standard diagnostics channel (see [Diagnostics](../diagnostics.md)), with `message` set to the original note's `content` (which MAY be `""`) and `hint` set to the underlying `sendMessage` error's message. This diagnostic is itself best-effort: if both channels fail, the failure is logged to `console.error` and execution continues.

The fallback path is taken on any thrown value from `sendMessage`; it does not retry the original call. The fallback never aborts the slash-command handler or the spawned subagent session. Implementers must guard against re-entry: if a future `loom/runtime/*` handler ever routes diagnostics back through `loom-system-note`, the diagnostic step in this fallback MUST NOT re-invoke `pi.sendMessage`. `ctx.ui.notify` itself can throw (e.g. in print mode where Pi's UI is not attached); wrap it in the same try/catch and proceed to the diagnostic step. For panic-routed notes, the original panic message MUST be included in the final-resort `console.error` log so post-mortem debugging retains the stack.

**`estimateTokens` (named export).** Imported from `@earendil-works/pi-coding-agent`. Signature: `estimateTokens(message: AgentMessage): number`. The export is declared at `dist/core/compaction/compaction.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above), where its parameter type `AgentMessage` is imported from `@earendil-works/pi-agent-core` (declared at that package's `dist/types.d.ts` as the agent-state superset of the LLM-side `Message` wire type — `AgentMessage` additionally carries `thinking` blocks, `toolCall` blocks, and host-defined custom messages). The estimation algorithm is owned by Pi and is not re-specified here: loom calls the function as-imported and does not redefine how it walks the message. loom's observable contract: the returned `number` is the per-message token estimate consumed by [Slash-Command Argument Binding — Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) for the per-message and per-turn token counts the truncation walk sums. **Non-normative.** At the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin), Pi's estimator (in `@earendil-works/pi-coding-agent`'s `core/compaction/compaction.ts`) derives a conservative character-quarter count (`Math.ceil(chars / 4)`-style, counting UTF-16 code units over the message's contributing strings); for orientation, a user message whose only content is `"😀😀😀"` (six UTF-16 code units) estimates to roughly 2 tokens. This description is orientation only, is owned by Pi, and may drift on any Pi revision; the authoritative behaviour is Pi's implementation at the pin.

**`buildSessionContext` (named export).** Imported from `@earendil-works/pi-coding-agent`; this is a free function, **not** a method on `ctx.sessionManager` — `ExtensionContext.sessionManager` is typed as `ReadonlySessionManager` (a `Pick<>` of the underlying `SessionManager`; see the inline shape below) and deliberately excludes `buildSessionContext` and every mutating member. Signature: `buildSessionContext(entries: SessionEntry[], leafId?: string | null, byId?: Map<string, SessionEntry>): SessionContext`. The runtime calls it as `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId())` — both `getEntries()` and `getLeafId()` are inside the `ReadonlySessionManager` `Pick<>`, so the call composes against the read-only surface in both prompt mode and subagent mode. The returned `SessionContext`'s `.messages` field is the ordered message list the runtime consumes for session-aware features; its element type is `AgentMessage[]`, pinned in [`SessionContext` and the `.messages` element shape](./host-interfaces-core.md#sessioncontext-shape) below. loom 1.0 callsite: [Slash-Command Argument Binding — Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) sources the binder's caller-session message list from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`. The runtime MUST NOT substitute `ctx.getContextUsage()` for per-turn token accounting; that API reports aggregate session usage, not per-turn counts, and is reserved for compaction triggers and footer rendering.
