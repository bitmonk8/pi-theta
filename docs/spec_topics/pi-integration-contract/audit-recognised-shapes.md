# Audit target categories

- <a id="audit-target-surface-categories"></a>*Target surface categories.* The audit covers exactly three categories: (1) `pi.<member>` static member-access expressions on the canonical `pi` identifier — defined syntactically as a function or method parameter declaration whose name is the literal identifier `pi` and whose explicit type annotation is the literal `ExtensionAPI` (qualified or unqualified, including `import type` re-imports, but resolved against the textual annotation rather than against an inferred type), in practice the loom's extension-factory entry-point parameter (per **Extension entry point** above; the same `default function (pi: ExtensionAPI)` signature) — including but not limited to the four `pi.register*` calls, `pi.on`, `pi.registerFlag`, and `pi.getFlag` named in this page's other obligations (the four `pi.register*` calls and `pi.on` are factory-time host-binding calls per **Extension entry point** above; `pi.registerFlag` and `pi.getFlag` are not themselves entries in the seven-capability [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) below but are `pi.<member>` surfaces the runtime touches and are in scope of category (1) here on the same footing as the capability members, with any inventory entry the audit would require for them resolved per [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote)). Pinning the carrier syntactically (canonical parameter name plus textual type annotation) on the same footing as category (3) below is what lets a static-AST walker satisfy this category without a TypeScript-program load; aliased, destructured, captured (`this.pi = pi`), or otherwise rebound forms — including any function/method parameter declaration whose explicit type annotation is `ExtensionAPI` but whose name is **not** `pi` — MUST NOT exist in the audited source tree (the audit enforces that prohibition under *Recognised import/access shapes* below). Local variables named `pi` whose explicit type annotation is not `ExtensionAPI` (or which carry no explicit type annotation) are out of scope of category (1) — the carrier rule keys on the textual `ExtensionAPI` annotation, not on the bare identifier text `pi`, so a static-AST walker neither over-fires on every local variable named `pi` nor under-fires on a renamed `ExtensionAPI`-typed parameter (which is instead caught by the off-canonical-name prohibition above on the same footing as the off-canonical-name `ctx` rule); (2) named imports from the four `@earendil-works/*` peer packages — `pi-coding-agent`, `pi-agent-core`, `pi-ai`, and `pi-tui`; (3) member-access expressions on the canonical `ctx` identifier — defined syntactically as a function or method parameter declaration whose name is the literal identifier `ctx` and whose explicit type annotation is the literal `ExtensionContext` or `ExtensionCommandContext` (qualified or unqualified, including `import type` re-imports, but resolved against the textual annotation rather than against an inferred type) — covering every member of `ExtensionContext` and `ExtensionCommandContext` as declared by the `@earendil-works/pi-coding-agent` peer dependency's `.d.ts` declarations at the loom 1.0 SDK pin — the SDK type declarations are the single source of truth for category (3)'s closed in-scope set, so a Pi minor that adds, removes, or renames a member of either type extends or contracts category (3) automatically and the implementation's `SDK_SURFACE_INVENTORY` category-(3) entries (per the new entry kind enumerated below) MUST be derived from those declarations rather than from any prose enumeration on this page. Illustrative members at the loom 1.0 pin include `waitForIdle`, `cwd`, `ui`, `model`, `modelRegistry`, `isIdle`, `getContextUsage`, `hasPendingMessages`, `compact`, `shutdown`, `getSystemPrompt`, `sessionManager`, and `signal`; this listing is illustrative only and the audit's category-(3) closed set MUST NOT be seeded from it. At the loom 1.0 Pi-SDK pin both `ExtensionContext` and `ExtensionCommandContext` are exported as named types from `@earendil-works/pi-coding-agent` on the package's `package.json#exports`'s `"."` entry (the SDK declaration site `dist/core/extensions/types.d.ts`, re-exported from the package root via `dist/index.d.ts`), with `ExtensionCommandContext extends ExtensionContext` per [`ExtensionCommandContext` (per-handler subtype loom touches)](./host-interfaces-core.md#extensioncommandcontext-interface) below — so the audit MUST detect category-(3) member access on a carrier parameter whose textual type annotation is either of the two literals (`ExtensionContext` or `ExtensionCommandContext`) on the same footing, and a member declared on one of the two types but not the other resolves against any `SDK_SURFACE_INVENTORY` category-(3) entry whose `path` field matches the access per the [Category-(2) inventory join key](./audit-resolution.md#audit-category-2-inventory-join-key) below (generalised to category (3) per this paragraph), regardless of which of the two annotations the carrier parameter declared. Pinning the carrier syntactically (canonical parameter name plus textual type annotation) is what lets a static-AST walker satisfy this category without a TypeScript-program load; aliased, destructured, captured (`this.ctx = ctx`), or otherwise rebound forms — including any function/method parameter declaration whose explicit type annotation is `ExtensionContext` or `ExtensionCommandContext` but whose name is **not** `ctx` — MUST NOT exist in the audited source tree (the audit enforces that prohibition under *Recognised import/access shapes* below). Category (2) for the four `@earendil-works/*` peer packages covers only the named-import declarations themselves; member-access references on those imports are out of scope of category (2) and would surface only via the canonical-`pi` carrier in category (1) for the `pi: ExtensionAPI` case or via the canonical-`ctx` carrier in category (3) for the `ctx: ExtensionContext` / `ExtensionCommandContext` case — the four peer packages currently expose no other in-runtime member-access carriers, and the typebox `Type.<member>` member-access carve-out pinned in the next sub-paragraph applies exclusively to the typebox `Type` identifier and MUST NOT be generalised to the four `@earendil-works/*` peers. The fifth bundled package `typebox` (per **Host prerequisites — Pi SDK pin** above) is in scope for category (2) with a single permitted *named import* — `Type` (consumed as `import { Type } from "typebox"`, the same shape [Step 0 (e)](./capability-probe.md#entry-capability-probe)'s host-shape check probes against, since `.Unsafe` and every other typebox surface the runtime might touch is reached as a member access on the `Type` namespace rather than as its own named import) — and the audit additionally enforces a `typebox`-specific *member-access allow-list* on `Type.<member>` references whose single permitted entry is `Unsafe` (i.e. `Type.Unsafe`), whose runtime presence is gated by that same [Step 0 (e)](./capability-probe.md#entry-capability-probe) check. The audit's detection rule for these `Type.<member>` references is pinned syntactically here as a typebox-only extension to category (2) (the four `@earendil-works/*` peers do not get an analogous member-access detection rule per the prior sentence prohibiting member-access detection on the four `@earendil-works/*` peers (above)): a `Type.<member>` access is detected as a category-(2) member-access reference whose carrier is the imported `Type` identifier (per the `import { Type } from "typebox"` named-import shape immediately above), and is resolved against the typebox member-access allow-list `{ Unsafe }` rather than against `SDK_SURFACE_INVENTORY` (parallel to how the named-import `Type` itself resolves through the typebox named-import allow-list per the *typebox-only sibling* paragraph below). Note the carrier-rule asymmetry with categories (1) and (3): the typebox `Type` carrier is an imported binding rather than a parameter typed `ExtensionAPI` (category (1)) or `ExtensionContext` / `ExtensionCommandContext` (category (3)), and a static-AST walker recognises it by the named-import declaration above rather than by a typed-parameter declaration; this carve-out applies exclusively to the typebox `Type` identifier and MUST NOT be generalised to member-access detection on any other imported binding. Every other named import from `typebox` (e.g., `Static`, `TSchema`) MUST surface under family (2) of the *Failure-surface contract* below, on the same footing as an unauthorised named import from any of the four `@earendil-works/*` peer packages; every other `Type.<member>` access (e.g., `Type.Object`, `Type.Static`) MUST surface under that same family under the same per-discriminator routability; both MAY be exempted only via the same-line `// allow-pi-surface:` marker per *Exemption mechanism* below. The four-peer-package prohibitions on `import * as`, default imports, dynamic `import()`, side-effect-only imports, computed access, aliased rebindings, captured rebindings, `Object.assign`, `keyof typeof`, and barrel-module / `export * from` re-exports per *Recognised import/access shapes* below apply identically to `typebox`. The single-entry member-access allow-list above (the *typebox member-access allow-list*) is **not** an `SDK_SURFACE_INVENTORY` entry — it is a fixed allow-list, sibling to but separate from `SDK_SURFACE_INVENTORY`, that lives in the same source-of-truth pinned-constants block in the extension module that already holds the four [Step 0 (Capability probe)](./capability-probe.md#entry-capability-probe) constants (Node floor, `AbortSignal` member list, capability list, peer-dep range), so the build-time audit consumes it from the same single site the runtime-side [Step 0 (e)](./capability-probe.md#entry-capability-probe) probe is wired against; the audit consumes it alongside `SDK_SURFACE_INVENTORY` (mirroring the same-line marker's status under *Exemption mechanism* below as a non-`SDK_SURFACE_INVENTORY` declaration site). Symmetrically, the `typebox` *named-import allow-list* whose single permitted entry is `Type` is **not** an `SDK_SURFACE_INVENTORY` entry either — it is a second fixed allow-list, sibling to but separate from `SDK_SURFACE_INVENTORY`, co-located with the member-access allow-list in the same pinned-constants block, and consulted by the audit's category-(2) resolver as a *typebox-only sibling* to `SDK_SURFACE_INVENTORY` for `typebox`-package imports specifically, so an `import { Type } from "typebox"` declaration resolves through this allow-list rather than through the [Category-(2) inventory join key](./audit-resolution.md#audit-category-2-inventory-join-key) rule below (which would otherwise refuse to resolve `Type`, since no `SDK_SURFACE_INVENTORY` entry's leftmost dot-separated `path` segment equals `Type`); this typebox-only resolution path is parallel to the already-pinned authorisation of `Type.Unsafe` member access through the member-access allow-list above and applies exclusively to `typebox`-package imports — the four `@earendil-works/*` peer packages still resolve solely against `SDK_SURFACE_INVENTORY` per the join-key rule. The single-entry allow-list MUST NOT be folded into the four-peer-packages iteration the [Step 0 (d)](./capability-probe.md#entry-capability-probe) probe pins, and neither the `Type` named import nor the `Unsafe` member-access entry extends the implementation's entry-kind taxonomy. These allow-list contents (`{ Type }` for named imports, `{ Unsafe }` for member access) are pinned at the loom 1.0 Pi-SDK pin against the runtime's current typebox usage; the implementation's source-tree sweep (per the *Inventory-closure audit* introductory paragraph above) verifies that no other typebox surface is reached anywhere in the audited source tree, and any future widening (e.g. promoting `Type.Object` into the member-access allow-list, or `Static` into the named-import allow-list) follows [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote) of the [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below applied to the typebox allow-lists rather than to `SDK_SURFACE_INVENTORY`. The contributor-facing maintenance obligation on a Pi/`typebox` bump that renames `Unsafe` (or otherwise drifts the permitted member name on the `Type` namespace) is owned by [step 5](./version-bump-intro.md#pi-version-bump-procedure) of the [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below, where this allow-list is grouping (iii) of the in-scope pinned-constants groupings. Satisfying the three in-scope categories MUST extend `SDK_SURFACE_INVENTORY`'s entry-kind taxonomy beyond the existing kinds, with at minimum one new entry kind per category not already covered: a category (3) kind for canonical-`ctx` member access against `ExtensionContext` / `ExtensionCommandContext`; a category (1) kind for `pi.<member>` references that are not function-member capabilities (`pi.registerFlag`, `pi.getFlag`, and any other such `pi.<member>` the runtime touches that is not one of the seven capabilities); and a category (2) kind for named imports from the four `@earendil-works/*` peer packages where not already covered by an existing entry kind (the existing `type-union-snapshot` entry for `SessionShutdownEvent.reason` and the existing `namespace-function` entries for the nine factory-probable function-member entries are the two existing-coverage cases). The new kinds MUST be coined in the implementation's entry-kind enumeration as a precondition to the audit's first run on `main`; the implementation MUST land the new entry kinds (with their `kind` discriminator strings and per-kind payload field shapes), the literal-read assertion's branch-table arms for each, and the audit's category-(1)/(2)/(3) detection wiring before the first `npm test` invocation that runs the audit. The spec edit pins the audit's behavioural contract; the implementation closes the loop. PIC delegates the literal kind names and per-kind payload field shapes to the implementation, which holds the single source of truth (mirroring the entry-level inventory carve-out in step 5 below). PIC pins the minimum entry shape every kind MUST satisfy: every `SDK_SURFACE_INVENTORY` entry that participates in inventory-closure-audit resolution (i.e., the existing `namespace-function` and `type-union-snapshot` kinds, and any new category-(1) / category-(2) / category-(3) entry kinds defined for the audit per the *Target surface categories* paragraph above) MUST carry at least two fields — `kind: string` (the discriminator the SDK surface-inventory literal-read assertion's branch table dispatches on, and the discriminator the routability obligation in the next sentence keys on) and `path: string` (a stable identifier the [Category-(2) inventory join key](./audit-resolution.md#audit-category-2-inventory-join-key) below resolves against per the per-category join-key rules pinned in *Category-(2) inventory join key* and *Category-(1)/(3) inventory join key (per-category specialisation)* below; category (2) uses leftmost-segment matching, categories (1) and (3) use rightmost-segment matching); per-kind payload fields beyond these two are implementation-owned per the delegation above. The minimum-shape obligation above is **scoped to audit-participating kinds**: the existing `node-floor`, `strict-capability-probe`, `peer-dep-range`, and `api-coverage` entry kinds are positive-direction-only literal-read entries that the inventory-closure audit does not consult for surface resolution, and are therefore exempt from the `path` requirement above (they retain their implementation-pinned payload shapes — `literal`, `probedName`, `packages`, `apiNamespace` — without a `path` field). The exemption is closed to the four kinds named in this sentence; any future entry kind, audit-participating or not, MUST carry the `kind` field, and any future audit-participating kind MUST additionally carry `path` per the rule above. The SDK surface-inventory literal-read assertion (per [step 2(a)](./version-bump-step2.md#bump-step-2-positive)) MUST additionally enforce the minimum-entry-shape obligation by surfacing each shape failure under its own `literal-read/<symptom>` discriminator, including at minimum: a missing `path` field on an audit-participating entry; a non-string `path` value; an empty-string `path`; an empty leftmost-dot-separated segment of `path` (e.g. `".foo"`); a missing `kind` field; a non-string `kind` value. Illustrative implementation-owned `<symptom>` tokens: `literal-read/missing-path-on-audit-kind`, `literal-read/empty-path`, `literal-read/empty-leftmost-segment`, `literal-read/missing-kind-field`, `literal-read/non-string-kind`. The implementation publishes the literal `<symptom>` strings and one negative-test fixture per token under the existing per-`<symptom>` fixture floor. The new entries MUST NOT be folded into the existing `namespace-function` kind — its definition is exclusively the nine function members of the factory-probable subset of the seven capability obligations enumerated below (capabilities 1, 2, 3, 4, and 6 — "five capabilities, nine function members"), and folding non-factory-probable surface into this kind would silently corrupt the factory-probable subset enumeration that the implementation and the [Step 0 (c)](./capability-probe.md#entry-capability-probe) probe both consume. ([Step 2(a)](./version-bump-step2.md#bump-step-2-positive)'s count-equality assertion is over the count of capability obligations enumerated in [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) below (items 1–7), not over `namespace-function` entries, so widening this kind does not directly perturb that assertion's `7`; the corruption surface is the factory-probable subset enumeration above.) The SDK surface-inventory literal-read assertion's branch table MUST gain an arm for each new kind under the same precondition obligation above (i.e. before the first `npm test` invocation that runs the audit on `main`, not necessarily in the same git commit as this spec edit); an inventory entry whose `kind` discriminator the branch table does not recognise MUST fail the assertion with a routable discriminator rather than silently passing through, ignoring the entry, or surfacing as an unhandled throw. The literal-read assertion's failure-discriminator shape is `literal-read/<symptom>` (a peer to the inventory-closure audit's `audit/<class>/<family>/<symptom>` shape pinned under [Failure-surface contract](./audit-failures.md#audit-failure-surface-contract) below — the `literal-read/...` discriminator is **not** under the `audit/` namespace and carries no `<class>` segment, since it never participates in the three-class partition the audit pins), with `<symptom>` strings owned by the implementation on the same footing as the implementation-owned literal strings under [step 5](./version-bump-intro.md#pi-version-bump-procedure) below; the unrecognised-`kind` failure MUST surface under a stable `literal-read/<symptom>` discriminator (illustrative implementation-owned token: `literal-read/unknown-entry-kind`), and the contributor recovery arm is [step 2(a)](./version-bump-step2.md#bump-step-2-positive)'s 'add a recognising branch to the literal-read assertion's branch table for the new `kind`' obligation. The audit's *Failure-surface contract* obligation under *Failure-surface contract* below to identify "the inventory entry the contributor would add" presupposes these per-kind field shapes are pinned by the implementation; an implementation that lands the audit without the corresponding kind-shape pins violates this paragraph.
- <a id="strict-capability-absence-pin"></a>*Strict-capability absence under the pin (`strict-capability-probe` extension).* Under the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin), the `strictCapable` member of the externally-owned `Model<Api>` type (declared in `@earendil-works/pi-ai` and pinned at [model-registry surface](./host-interfaces-core.md#model-registry-pin) above) is absent on every Pi-supplied `Model<Api>` instance, so the binder strict-capability probe degrades in production to the universal-`W` branch surfaced as `loom/load/binder-model-strict-capability-unknown`. This is the single citation site every "absent / universal under the pin" claim about `strictCapable` references — the [`bind_model` row](../frontmatter/frontmatter-fields-a.md), the [strict-capability requirement](../binder/binder-model-and-context.md#strict-capability-requirement), and the [`binder-model-strict-capability-unknown` diagnostic row](../diagnostics/code-registry-load.md) each cite this anchor rather than restating the absence claim. The existing `strict-capability-probe` literal-read assertion (per [step 2(a)](./version-bump-step2.md#bump-step-2-positive)) MUST additionally fail if any reachable `Model<Api>` declaration at the cited site exposes a `strictCapable` member under the probed name — the *absence-under-the-probed-name* gate that completes the rename-detection arm owned by [bump-procedure step 7](./version-bump-triggers.md) (which catches the field present under a *different* name). PIC pins this behavioural contract; the literal kind name and per-kind payload shape remain implementation-owned per the delegation above, and the `strict-capability-probe` kind's `path`-exemption is unchanged.
