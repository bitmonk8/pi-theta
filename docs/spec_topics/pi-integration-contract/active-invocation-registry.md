# Active invocation registry

   <a id="active-invocation-registry"></a>

   The **`ActiveInvocationRegistry`** is an extension-instance-scoped `Set<{ loomAbort: AbortController; disposeBarrier: Promise<void>; shutdownReason: string | undefined; loom: string; invocationId: string }>` that the runtime maintains in lock-step with each loom invocation: the `loom` field carries the invocation's canonical-key name (the *final derived name* per [Discovery — Slash-name collisions across formats](../discovery.md) — post-`pi.looms` mapping, post-`as` rename, basename-hyphen-normalised — with **no leading `/`**) and is populated at insertion time when the entry is created at each of the three insertion sites enumerated below (slash-command handler entry, `tool.execute(...)` adapter entry, and `invoke` spawn-site entry). All three insertion sites consume the **same canonical-name resolver**: the `loom` field is read from the canonical key under which the callee is registered in the runtime's loom catalog (the `LoomRegistry` `Map<slashName, parsedLoom>` entry-table whose framing the [`LoomRegistry` drain-state contract](./drain-state-contract.md#loom-registry-drain-state-contract) above pins, accessed via that contract's existing entry-table accessor — by name in loom 1.0 the slash-command handler's dispatch lookup against that same table — for slash-registered looms; an analogous canonical-key lookup over the loaded-`.loom`-file set for callee-only `.loom` files reached via literal `invoke("./path.loom", ...)` or via `tool.execute(...)` for subagent-mode tool-only callees that are never slash-registered, sourced from the per-load-pass static-resolution cache per [Invocation — Static resolution](../invocation.md#static-resolution) for the `invoke("./path.loom", ...)` arm and from the parent loom's load-time `tools:` resolution snapshot per [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools) for the subagent-mode `tool.execute(...)` arm, both of which carry the same canonical-key field on which the `LoomRegistry` entry-table keys), keyed on each arm by the slash-name lookup result the arm's underlying registration / cache / snapshot already carries: the `LoomRegistry` `Map<slashName, parsedLoom>` key for slash-registered looms, and — for callee-only `.loom` files reached via literal `invoke("./path.loom", ...)` or via `tool.execute(...)` for subagent-mode tool-only callees — the slash-name field the [Invocation — Static resolution](../invocation.md#static-resolution) cache entry and the [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools) snapshot entry already carry on those entries (the same final-derived-name form per [Discovery — Slash-name collisions across formats](../discovery.md): post-`pi.looms` mapping, post-`as` rename, basename-hyphen-normalised, no leading `/`). loom 1.0 carries no separate canonical-key minting obligation on the [`invocation.md#static-resolution`](../invocation.md#static-resolution) cache entries or the [`frontmatter.md#tools`](../frontmatter/frontmatter-fields-a.md#tools) snapshot entries beyond that already-present slash-name field; the resolver consumes the slash-name lookup result on those entries only, rather than reading a separate canonical-key field that loom 1.0 does not require those entries to carry. Two callee-only `.loom` files whose basenames would otherwise collide on the same final-derived form are dropped at load time via `loom/load/cross-format-collision` per [Discovery — Slash-name collision rules](../discovery.md) (the same-priority `.loom`-vs-`.loom` arm), so the resolver never observes a collided callee-only basename at insertion time and no path-namespacing step is introduced into the canonical form to disambiguate one, so no `<callee-only>` sentinel value is introduced and the wire shape on `details.event.loom` admits this slash-name verbatim on every arm; the canonical name is **not** re-derived at the insertion site from the authored identifier the call site carries (the slash-command handler's registered command name, the `tool.execute(...)` adapter's source loom, or the `invoke(...)` literal's authored path — the third of which would otherwise leave the `pi.looms` mapping → `as` rename → basename-hyphen-normalisation order ambiguous at the spawn site and split operator dedup buckets on `details.event.loom` between two implementations that converged on different canonical forms). The resolver **fails the load-time validation that precedes any insertion-site code** when its callee would otherwise be unresolvable: callee-only basename collisions and `pi.looms` collisions are caught at discovery / load time and trigger `loom/load/cross-format-collision` per [Discovery — Slash-name collisions](../discovery.md) before reaching the insertion site, so the resolver never sees an unresolvable callee at the insertion site. By the time an entry is inserted into `ActiveInvocationRegistry` the callee has already been resolved against the loom catalog above (slash entry points dispatch off the `LoomRegistry` arm; `tool.execute(...)` entry points dispatch off the `LoomRegistry` arm for slash-registered tool-exposed looms (the prompt-mode case, where the tool reaches the model via `pi.registerTool` / `pi.setActiveTools` against a loom that is also slash-registered) and off the parent loom's load-time `tools:` resolution snapshot per [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools) for subagent-mode tool-only callees that are never slash-registered (the `customTools`-only case); `invoke(...)` spawn sites consume the parent loom's load-time `tools:` resolution snapshot for `tools:`-routed callables and the per-load-pass static-resolution cache per [Invocation — Static resolution](../invocation.md#static-resolution) for literal `invoke("./path.loom", ...)` callees, all of which fail their own load-time validation before any insertion-site code is reachable), so the `loom` field is **read** from the resolved catalog entry's slash-name field rather than constructed at the insertion site. The field is set on entry creation and never mutated thereafter, so it is observable in its registered form for the entire lifetime of the entry — including the [**Per-invocation operator visibility (clean-cancel path)**](./session-shutdown-semantics.md#session-shutdown-semantics) rule's `details.event.loom` read in the per-invocation `finally`, on every arm including the [*Accepted loom 1.0 residual gap — sub-step-2 stamp-throw case*](./session-only-degraded-state.md#substep-2-stamp-throw-residual-gap) below where the `shutdownReason` stamp threw before the `abort()` ran (the `loom` field's pre-stamp registration means it is unaffected by the stamp throw). The `shutdownReason` field is `undefined` at insertion and is populated only by sub-step 2 of the `session_shutdown` handler above (the stamp happens before that entry's `loomAbort.abort()` call) so the per-invocation `finally`'s `loom/runtime/cancelled-by-session-shutdown` emission per the [**Per-invocation operator visibility (clean-cancel path)**](./session-shutdown-semantics.md#session-shutdown-semantics) rule has a handler-captured-string channel that does not depend on a runtime singleton (forbidden by the no-globals rule) or on `AbortSignal.reason` payload shape (Pi-side shape not pinned). The `invocationId` field is a per-invocation UUID in canonical lowercase 8-4-4-4-12 hex form per the [§7 `<invocation-id>` placeholder convention](../diagnostics/placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders), minted through the injected [`IdSource` seam (PIC-20)](./host-interfaces-services.md#idsource-interface) at registry-insertion time inside the **Dispatch-site setup wrap** `try`/`catch` (see *Registry contract* below) before any awaitable work, and set on entry creation and never mutated thereafter; downstream consumers — the per-invocation `finally`'s `loom/runtime/cancelled-by-session-shutdown` `details.event.invocation_id` per the [**Per-invocation operator visibility (clean-cancel path)**](./session-shutdown-semantics.md#session-shutdown-semantics) rule, and the `RuntimeEvent.invocation_id` wire field defined under [**Runtime event channel**](./runtime-event-channel.md) — read this single registry-sourced value rather than re-deriving an id at the emission site, and `loom/runtime/reload-teardown-timeout`'s `<list>` rendering reads the same field per its row in [Diagnostics](../diagnostics.md). The registry-side identifier is spelled `invocationId` in lowerCamelCase (matching this section's other runtime-state members `loomAbort`, `disposeBarrier`, `shutdownReason`), while the corresponding wire-side names — `details.event.invocation_id`, `RuntimeEvent.invocation_id`, and the `<invocation-id>` placeholder in `loom/runtime/reload-teardown-timeout`'s `<list>` — are rendered in snake_case (matching the rest of this corpus's wire-payload field surface); downstream consumers therefore read the registry entry's field as `entry.invocationId` and surface its value under the snake_case wire name at each emission site (e.g. the *Runtime construction obligation* in [Diagnostics — `details` shape conventions for the `session_shutdown` event payload](../diagnostics/diagnostic-shape.md#session-shutdown-details-conventions) constructs `details.event` with the static property name `invocation_id` assigned from `entry.invocationId`). The sibling `loom` field happens to spell identically on both sides because its token is already in snake-shape, not because the registry-side and wire-side names are uniformly identity-mapped. The runtime maintains the registry in lock-step with each loom invocation: the slash-command `handler`, the `tool.execute(...)` adapter for tool-exposed looms, and the parent-`invoke` spawn site each insert their own entry on entry and remove it from the same `finally` block that already disposes any subagent session (per [**Subagent session lifecycle**](./subagent.md#pic-9)). The `disposeBarrier` is the promise that the per-invocation `finally` settles after `AgentSession.dispose()` returns (subagent mode) or immediately (prompt mode). Concurrent loom invocations therefore each carry one entry; the registry is the structure the teardown handler iterates.

   *Registry contract.*

   - **Insertion** happens at slash-command handler entry, `tool.execute(...)` adapter entry, and `invoke` spawn-site entry, before any awaitable work.
   - **Removal** happens in the same `finally` block that disposes the subagent `AgentSession`, after `disposeBarrier` settles.
   - **Dispatch-site setup wrap.** Each of the three insertion sites MUST wrap the pre-evaluation setup sequence — `new AbortController()` for `loomAbort`, the `Promise.withResolvers()` construction (or equivalent) that yields the entry's `disposeBarrier` promise and its closure-held resolve handle, the registry-`Set.add` insertion, the per-entry-point `source.addEventListener('abort', …)` attach that registers the forwarding listener on the inbound Pi-side `AbortSignal` (the slash-command handler's `ctx.signal`, the tool-exposed adapter's `signal` parameter, or the `invoke`-parent's parent-signal — see [Cancellation — Forwarding into `loomAbort`](../cancellation.md)), and (subagent mode only) the awaited `createAgentSession(...)` call followed by the post-resolution `loomAbort.signal.addEventListener('abort', …)` attach that wires the one-shot `AgentSession.abort()` listener — in a `try`/`catch` whose `catch` arm routes the captured throw or rejection through the **runtime-defect surface** defined in [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics) (single rule, three call sites; the `createAgentSession`-rejects paragraph in [**Subagent session lifecycle**](./subagent.md#pic-9) is a worked example of this rule rather than a one-off). No new `details.kind` discriminator is introduced for the listener-attach throws — the existing `loom/runtime/internal-error` surface carries `error.message` and `error.stack` and that is sufficient for triage. On a throw before the `Set.add` insertion completes no registry entry leaks (nothing to remove); on a throw after insertion the same per-invocation `finally` removes the entry and settles `disposeBarrier`. The `cause: "internal_error"` arm of [`InvokeInfraError`](../errors-and-results/queryerror-variants.md#invoke-variants) and the slash-command system-note template `"loom /<name> aborted with internal error: <error.message>"` defined in [Errors and Results — Runtime panics](../errors-and-results/error-model.md#runtime-panics) apply unchanged. A throw from `loomAbort.abort()` invoked by the `catch` arm during cleanup of a half-constructed entry is itself swallowed by the per-entry `try`/`catch` of the **`session_shutdown`** sub-step 2 rule above and MUST NOT mask the original setup throw.
   - **`disposeBarrier` resolver storage.** The resolver for each entry's `disposeBarrier` is held in the dispatch-site insertion-site closure (alongside the `loomAbort` controller), captured from the `Promise.withResolvers()` step of the setup sequence above before the registry-`Set.add` insertion so it is in scope when the entry is constructed. The per-invocation `finally` invokes that closure-held resolver to settle `disposeBarrier` (timing per the intro paragraph above); the `session_shutdown` sub-step 3 `Promise.allSettled` iteration reads `entry.disposeBarrier` (the `Promise<void>`), never the resolver. The pinned five-field entry shape is unchanged — the resolver is closure-scoped, not a sixth registry-entry field — so observers of the entry (tests, audit fixtures) see exactly the five members the entry literal pins.
   - **Iteration order** in the `session_shutdown` handler's sub-step 2 (`loomAbort.abort()` over every entry) and sub-step 3 (`Promise.allSettled` over every entry's `disposeBarrier`) is insertion order, matching the V8 `Set` insertion-order invariant the rest of the spec already relies on. The order is observable to tests asserting on the order of `loom/runtime/reload-teardown-timeout`'s `<list>` rendering.
   - **`loomAbort.abort()` throwing inside the sub-step 2 iteration** is swallowed by a per-entry `try`/`catch`; the handler continues to the next entry. No diagnostic is emitted (the abort is best-effort and the entry's own `finally` will still settle `disposeBarrier`, which sub-step 3 already awaits with `Promise.allSettled`).
   - **The registry name is internal.** It is not part of the loom-facing or extension-facing contract; tests assert on observable side effects (entry counts via probe seams, ordered `loomAbort.abort()` calls, `disposeBarrier` settlement) rather than on the symbol itself.
   - **Per-mode concurrency invariants.** The registry tracks both prompt-mode and subagent-mode entries (so the `session_shutdown` teardown above reaches every in-flight invocation); the *insertion shape* differs by mode, but the entry shape and the iteration rule do not. *Subagent-mode* entries may be genuine concurrent siblings: parallel tool calls into the same `.loom` callable, sibling `invoke(...)` sites whose children resolve to subagent mode, and parent→child stacks of subagent-mode invocations all coexist as distinct registry entries that progress on the event loop independently. *Prompt-mode* entries are constrained to **linear nesting** within a single user session: at most one prompt-mode body holds an open `pi.setActiveTools` snapshot/restore window at a time, but a prompt-mode parent suspended on an `invoke(...)` to a prompt-mode child legitimately keeps its own registry entry while the child's entry is also live (the parent is suspended at the call site, not dispatched, and only the child's snapshot/restore window is open). The teardown `loomAbort.abort()` iteration in sub-step 2 above must reach the suspended parent's entry as well as the child's. The supporting mechanisms that pin prompt-mode linear nesting — Pi's per-session slash-handler serialisation pinned under [**Tool-registration lifetime and visibility**](./tool-registration-lifetime.md#tool-registration-lifetime-and-visibility), the load-time rejection of prompt-mode `.loom` callees in another loom's `tools:` per [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools), and the suspend-on-invoke rule from [Invocation — Cross-mode semantics](../invocation.md) — are owned by those pages; the registry contract above relies on them but does not restate them. Subagent-mode children spawned from a prompt-mode parent are genuinely concurrent and add their own registry entries on the same insertion-site rules above.

   *Edge cases the handler must observe:*

   - `ExtensionRuntime.invalidate()` (the canonical `ExtensionRuntime` pin lives at [Registration steps — step 4 `session_shutdown` teardown contract](./registration-steps.md#extension-runtime-invalidate-teardown)) may already have been called by Pi when the handler runs; any `pi.*` call that reaches the stale runtime throws. The teardown sequence above touches only loom-internal state and the SDK objects (`AbortController`, chokidar `FSWatcher`, `AgentSession`) directly — it does not call back into `pi.*`.
   - The `sendSystemNote` fallback chain (the best-effort chain defined under [**System notes**](./runtime-event-channel.md)) MUST NOT be invoked from the teardown handler — `pi.sendMessage` against a stale runtime throws, and the fallback to `ctx.ui.notify` would re-enter the same stale surface. Diagnostics emitted from the teardown handler use `console.error` directly, bypassing the persistent-diagnostic channel defined in [Diagnostics](../diagnostics.md). Five diagnostics may be emitted from this handler, all via `console.error`: `loom/runtime/reload-teardown-timeout` (referenced in sub-step 3), `loom/host/session-shutdown-reason-unknown` (referenced in the **Unknown-reason rule** above), `loom/host/session-shutdown-pinned-constant-unreadable` (referenced in the **Unknown-reason rule** above, on a failure of the handler-entry pinned-constant lookup), `loom/host/session-shutdown-teardown-step-failed` (referenced in the **Per-step isolation** paragraph above), and `loom/host/session-shutdown-runtime-degraded` (referenced in the `reason: "new" | "resume" | "fork"` bullet below, after sub-step 4 returns on the three session-only reasons; emission is independent of sub-step 4's per-call outcomes per that bullet). No other diagnostic code may be emitted from this handler.
