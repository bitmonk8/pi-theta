# Unknown reason rule

   <a id="unknown-reason-rule"></a>
   **Unknown-reason rule.** Before sub-step 1 runs, the handler validates `event.reason` against the closed set `{"quit", "reload", "new", "resume", "fork"}`. The validation read of `event.reason` MUST be wrapped in a `try`/`catch` so a property-access throw (e.g., a getter that throws) is treated as an unknown reason rather than escaping the handler. The three pre-sequence operations the handler performs are exactly (1) the snapshot lookup-and-`literals` read described later in this paragraph (a throw-prone read whose failure modes route to `loom/host/session-shutdown-pinned-constant-unreadable` per the catch-arm discriminator below), (2) the `event.reason` read (a throw-prone read whose failure mode routes to `loom/host/session-shutdown-reason-unknown` per the catch-arm discriminator below), and (3) the set-membership comparison of `event.reason` against the snapshot's `literals` field (the only branch the handler MAY take, whose non-member result also routes to `loom/host/session-shutdown-reason-unknown` per this rule's unknown-reason fallback above); operations (1) and (2) are the throw-prone reads covered by the widened `try`/`catch` with catch-arm discrimination later in this paragraph; the only pre-sequence work permitted in addition to operations (1)–(3) is the closed two-item carve-out (a) the `LoomRegistry.readDrainState()` call and short-circuit predicate evaluation that constitute step (I) of the *Handler control-flow ordering* clause in the [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause below (which precedes operations (1)–(3) on every invocation per that clause's fixed (I)→(VI) ordering, and short-circuits the entire handler before operations (1)–(3) run when the predicate fires per step (II) of that ordering), and (b) the handler-entry pre-initialisations of the two handler-scoped variables — `capturedEventReason` to the literal string `"<unreadable>"` and `pinnedConstantReadOk` to `false` — pinned later in this paragraph as the pre-init sites that operations (1) and (2) may subsequently overwrite on their success arms; no other pre-sequence operation is permitted beyond operations (1)–(3) and carve-outs (a) and (b), and all three of operations (1)–(3) feed the same downstream behaviour of running the full five-sub-step teardown sequence as if the reason were `"quit"` or `"reload"`. On an unknown reason — or on a throwing access — the handler emits exactly one `loom/host/session-shutdown-reason-unknown` (W, runtime — see the [Diagnostic shape — column legend](../diagnostics/diagnostic-shape.md#column-legend)) diagnostic via `console.error` with `details: { observed: <String(event.reason)> }` *before* sub-step 1 runs (so the diagnostic survives even if the handler later throws), then runs the full five-sub-step sequence as if the reason were `"quit"` or `"reload"`. The no-active-invocations fast-path described in the *Edge cases* paragraph below MUST NOT be taken on the unknown-reason path: the registry drain, the bounded `Promise.allSettled` await, the watcher closure, and the listener detachment all run unconditionally. The `String(event.reason)` coercion in the `details.observed` payload tolerates symbols, `undefined`, numbers, booleans, and ordinary objects without itself throwing; on a throwing property access (where the value never materialised) `details.observed` is the literal string `"<unreadable>"`. The coercion itself MUST be wrapped in `try`/`catch` — symmetric with the sibling `String(error)` defence in the `"throw:<String(error)>"` discriminator below — so that an `event.reason` whose `toString`/`Symbol.toPrimitive` throws (making `String(...)` itself throw, the one object case the "without itself throwing" tolerance above does not cover) does not escape the handler; on such a coercion throw `details.observed` falls back to the same literal string `"<unreadable>"` as the throwing-property-access path (the `details.observed` payload carries no `"throw:"` prefix to preserve, so the bare `"<unreadable>"` sentinel is reused directly, unlike the `details.failure` discriminator's `"throw:<unreadable>"` fallback). The handler captures this single coerced string into a handler-scoped variable at handler entry — the same site this rule's `try`/`catch` runs, before sub-step 1 — and the same string is reused as `details.event.reason` by every `loom/runtime/cancelled-by-session-shutdown` note emitted under the **Per-invocation operator visibility (clean-cancel path)** rule below; the per-invocation `finally` does not re-read `event.reason`. The captured value is a closed-set member when Pi delivered one of the five literals (`"quit" | "reload" | "new" | "resume" | "fork"`), `String(event.reason)` on the unknown-reason path (or `"<unreadable>"` when that coercion itself throws, per the coercion-throw clause above), `"<unreadable>"` on the throwing-access path, or `"<unreadable>"` on the snapshot-failure path (where the fixed read order above means sub-step 1's `event.reason` read was never reached because the snapshot read failed first; to make this case well-defined, the handler-scoped variable is pre-initialised to `"<unreadable>"` at handler entry before either of the two reads runs, and the snapshot-failure path simply leaves that pre-initialisation unchanged). A second handler-scoped variable, the boolean `pinnedConstantReadOk`, is pre-initialised to `false` at the same handler-entry site before either of the two reads runs, and is set to `true` on the success arm of the snapshot lookup-and-`literals` read — i.e., when the catch arm that emits `loom/host/session-shutdown-pinned-constant-unreadable` (pinned later in this paragraph) does not fire on this shutdown; on the snapshot-failure path the pre-initialised `false` is left unchanged. This parallels the captured-string pre-init above and is the same `pinnedConstantReadOk` seam the *Predicate split* clause of the [*Edge cases* `reason: "new" | "resume" | "fork"` bullet](./session-only-degraded-state.md#session-only-reason-degraded-state) below consumes as the narrower diagnostic-emission predicate's first conjunct. The captured value reaches the per-invocation `finally` of the **Per-invocation operator visibility (clean-cancel path)** rule below via sub-step 2's stamp onto each `ActiveInvocationRegistry` entry's `shutdownReason` field — that field is the spec-pinned channel for all four captured-value cases, including the snapshot-failure `"<unreadable>"`; the `finally` MUST NOT use a runtime singleton (forbidden by the no-globals rule) or the `AbortSignal.reason` payload (Pi-side shape not pinned). The snapshot-failure path deliberately overloads the `"<unreadable>"` sentinel with a fourth meaning beyond the throwing-property-access case it already covers; operators distinguish the two by the co-emitted teardown-handler diagnostic (`loom/host/session-shutdown-pinned-constant-unreadable` for the snapshot-failure path versus `loom/host/session-shutdown-reason-unknown` with `details.observed = "<unreadable>"` for the throwing-access path), not by the per-invocation `details.event.reason` value alone. The same-string reuse between the handler-scoped capture and `details.observed` of `loom/host/session-shutdown-reason-unknown` is therefore preserved on the closed-set and unknown-reason paths only — on the snapshot-failure path no `details.observed` is emitted at all (the `loom/host/session-shutdown-pinned-constant-unreadable` diagnostic carries `details.failure` instead per the catch-arm discriminator above), and the per-invocation `details.event.reason` defaults to the sentinel with no `details.observed` counterpart to compare it against. Pin the coercion to `String(event.reason)` explicitly — a template-literal coercion (`` `${event.reason}` ``) re-introduces the symbol-throw bug `String()` is chosen to avoid. The closed set `{"quit", "reload", "new", "resume", "fork"}` literal in this paragraph is sourced from the `SessionShutdownEvent['reason']` closed-set snapshot entry inside the extension module's source-of-truth pinned-constants block — the static `SDK_SURFACE_INVENTORY` constant under `src/extension/` that the SDK surface-inventory test also consumes, and the same block that already holds the [Step 0 (Capability probe)](./capability-probe.md#entry-capability-probe) constants. The `session_shutdown` handler MUST import `SDK_SURFACE_INVENTORY` directly from that module rather than re-declare a runtime-only copy of the snapshot entry — single source of truth across build-time test and runtime handler is the load-bearing invariant; the runtime's set-membership check above consumes that snapshot entry's `literals` field directly (single-site edit, no separate copy in the handler). The runtime locates that snapshot entry by looking it up in the same block and reading its `literals` field; the lookup MUST be keyed on the composite predicate `(entry.kind === "type-union-snapshot") && (entry.path === "SessionShutdownEvent.reason")` against the entry shape `{ kind: "type-union-snapshot", path: "SessionShutdownEvent.reason", literals: [...] as const }` (there is no separate stable identifier field on the entry, and one MUST NOT be invented for this lookup) rather than on the entry's position in the block (which the implementation owns and may reorder without semantic breakage). Both clauses of the composite are required: matching on `kind` alone would silently shadow on a future second `type-union-snapshot` entry covering a different SDK union, and matching on `path` alone — acceptable today, since `path` is unique across the current pinned-constants block — is a fragile invariant once additional non-snapshot entry kinds may carry a `path` field of their own (e.g. an `abortsignal-member` entry whose `path` happens to collide), so the `kind` clause is retained as a cheap structural defence whose cost is one additional `===` per handler entry. The composite predicate above is the same predicate `diagnostics.md`'s `loom/host/session-shutdown-pinned-constant-unreadable` row pins as the lookup site whose failure modes that diagnostic surfaces; the two pages MUST stay aligned on the predicate text in any future edit. *Lookup-failure-to-discriminator routing.* The iteration primitive used to scan `SDK_SURFACE_INVENTORY` for the matching snapshot entry — `Array.prototype.find`, a `Map.get` against a derived index, an explicit `for`/`switch`, or any equivalent — is **leaf-owned** and not pinned here; the routing of the failure modes that primitive can produce to the closed `details.failure` discriminator set is **spec-owned**, and is pinned as four arms that together exhaust the snapshot-side failure space the catch-arm discriminator below disambiguates: (1) **`SDK_SURFACE_INVENTORY` itself resolves to `undefined`** (a circular-init or live-binding gap in which the source-of-truth pinned-constants block has not finished initialising by handler-entry time — the dominant loom 1.0 failure mode for the snapshot read) — discriminator MUST be `"missing-entry"`, **not** any `"throw:TypeError: …"` value, and to satisfy this routing the handler MUST guard the lookup with an explicit `SDK_SURFACE_INVENTORY === undefined` check **before** invoking the iteration primitive (a naive `SDK_SURFACE_INVENTORY.find(…)?.literals` pattern routes this case to the throw arm instead and would split operator deduplication buckets across implementations whose iteration-primitive choice is otherwise immaterial); (2) **the iteration primitive returns no matching entry** (the composite predicate above matches nothing) — discriminator is `"missing-entry"`; (3) **the iteration primitive returns a matching entry but the subsequent `.literals` access throws**, or **a per-entry `.kind` or `.path` read itself throws during iteration** (a getter-style trap on a non-matching entry, or a frozen-but-mutated import whose property reads now throw) — discriminator is `"throw:<String(error)>"` per the discriminator-grammar rule below; (4) **the matching entry's `.literals` field is structurally invalid** per the four sub-cases of the catch-arm-discriminator paragraph below — discriminator is `"literals-shape-invalid"`. An implementation that routes any of (1)–(4) to a different discriminator is non-conformant regardless of which iteration primitive it chose. The two reads MUST run in the fixed order **(1) snapshot lookup-and-`literals` read, then (2) `event.reason` read**, and both MUST be inside the same handler-entry `try`/`catch` discipline as the `event.reason` read above. The implementation choice between (a) widening the existing `try`/`catch` to cover both reads with a state variable tracking which read was in progress, and (b) using two adjacent `try`/`catch` blocks (snapshot first, then `event.reason`), is left to the implementation — the load-bearing pin is behavioural: no throw escapes the handler, and the catch arm selects the diagnostic that corresponds to the read that actually threw. The catch-arm discriminator is fully determined by the fixed read order: if the **snapshot read** (lookup or literal-array read) threw, was missing, or returned an entry whose `literals` field has a structurally invalid shape (circular module init, a frozen-but-mutated import, an import resolved to the wrong shape after a buggy patch), the handler emits exactly one `loom/host/session-shutdown-pinned-constant-unreadable` (W, runtime) diagnostic via `console.error` with `details: { failure: <discriminator string describing the failure mode — one of the literals `"missing-entry"`, `"literals-shape-invalid"`, or `"throw:<String(error)>"`> }`. The TypeScript type of the `details.failure` field is the closed template-literal union ``"missing-entry" | "literals-shape-invalid" | `throw:${string}` `` — operator-tooling consumers MAY type-narrow on this union (e.g. against the literal arm `"missing-entry"`, the literal arm `"literals-shape-invalid"`, or the template-literal arm via `startsWith("throw:")` per the operator-key rule pinned later in this paragraph) and the runtime MUST emit only values that inhabit it. The `"literals-shape-invalid"` discriminator covers four structurally-distinct sub-cases that all leave the runtime with no usable closed set and warrant the same operator response (the snapshot itself is broken; resolve under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below): (1) the `literals` field is not an array at all; (2) the `literals` field is an array but at least one element is not a string; (3) the `literals` field is the empty array; (4) the `literals` field is an array of strings but at least one element is the empty string `""`. A conformance test suite MUST cover each of these four sub-cases with its own fixture: each fixture constructs a synthetic snapshot value — via the same test-only static-import-preserving substitution mechanic the (P-b) and (P-c-1) fixtures in [Patch skew degradation](./patch-skew-degradation.md#patch-skew-degradation-contract) establish — whose `literals` field exhibits exactly that one sub-case's structural defect, dispatches a `session_shutdown` event, and asserts the handler emits exactly one `loom/host/session-shutdown-pinned-constant-unreadable` with `details.failure === "literals-shape-invalid"` and that no `loom/host/session-shutdown-reason-unknown` fires; a suite that exercises only a subset of the four sub-cases does not witness the closed `"literals-shape-invalid"` discriminator's coverage. The `"missing-entry"` discriminator covers both the literal lookup-returned-nothing case and the defensive kind-mismatch case (an entry exists at the matching `path` but its `kind` is somehow not `"type-union-snapshot"`, e.g. after future entry-shape drift); both reach the catch arm via the same `(entry.kind === "type-union-snapshot") && (entry.path === "SessionShutdownEvent.reason")` composite predicate above failing to match, so the handler cannot — and need not — distinguish them, regardless of whether `event.reason` would itself be readable (the snapshot-side failure is dominant precisely because the snapshot read ran first and the `event.reason` read was never reached); if the snapshot read **succeeded** and the subsequent `event.reason` read threw, the handler emits the `loom/host/session-shutdown-reason-unknown` diagnostic per the **Unknown-reason rule** above, with `details.observed = "<unreadable>"` per that rule's throwing-property-access clause. The pinned-constant-unreadable diagnostic, when emitted, fires *before* sub-step 1 runs (so the diagnostic survives a later handler throw, symmetric with the **Unknown-reason** emission), then proceeds as if `event.reason` were unknown — routing through the unknown-reason path above for the **teardown sequence only** (the set-membership check is skipped for this shutdown, the full five-sub-step sequence runs, and a faulty constant import never escapes the handler). The fixed snapshot-first ordering is load-bearing: it makes the disambiguation deterministic across implementations (the fixture for a throwing-getter `event.reason` paired with a successful snapshot read always observes `loom/host/session-shutdown-reason-unknown`; the fixture for a throwing snapshot lookup always observes `loom/host/session-shutdown-pinned-constant-unreadable` regardless of whether `event.reason` is itself readable), and it routes the dominant failure mode (a throwing `event.reason` getter on an otherwise-healthy snapshot) to the diagnostic whose message most directly identifies the cause. The unknown-reason path's own diagnostic emission is **not** also performed on this path: `loom/host/session-shutdown-pinned-constant-unreadable` and `loom/host/session-shutdown-reason-unknown` are mutually exclusive — at most one of the two fires per `session_shutdown` event, with the snapshot-lookup-failure path firing `pinned-constant-unreadable` instead of `reason-unknown`. This avoids the misleading-message failure mode in which a corrupted snapshot lookup, paired with an `event.reason` that is in fact a closed-set member (e.g. `"quit"`), would otherwise surface a `session_shutdown event.reason outside closed set: quit` `loom/host/session-shutdown-reason-unknown` message whose claim is wrong by inspection — the snapshot was broken, not `event.reason`. Operator deduplication is well-defined under either code — `(code, details.observed)` for `session-shutdown-reason-unknown` and `(code, details.failure)` for `session-shutdown-pinned-constant-unreadable` — and a given `session_shutdown` event contributes at most one such pair across the two codes (the codes are mutually exclusive per the rule above). The `"throw:<String(error)>"` discriminator's grammar is pinned: the literal colon between the `"throw:"` prefix and the coerced payload is a fixed prefix separator and is **not** escaped; the `String(error)` payload is truncated to at most 256 UTF-16 code units (`String.prototype.length`) before concatenation with the prefix, so a thrown `Buffer`, a multi-MB stack trace, or any other `error` whose `String()` coercion expands to a large blob does not blow up the discriminator string or the teardown `console.error` write cost it carries; control characters in the truncated payload are escaped before concatenation — `\n`, `\r`, and `\t` as the two-character literal escapes `\\n`, `\\r`, `\\t`, and every other Unicode code point in U+0000–U+001F (excluding the three named above) and U+007F–U+009F as the six-character escape `\\uXXXX`; all other characters are passed through unchanged. A literal colon inside the truncated payload is not escaped (the prefix separator is positional, not delimiter-based — operators key on `startsWith("throw:")` against the discriminator, not on splitting at the first colon), and a `"throw:"` substring inside the truncated payload receives no special handling either (the same `startsWith("throw:")` predicate is the only operator-side parse the rule pins; an inner `"throw:"` substring is contract-compatible noise). The `String(error)` coercion MUST itself be wrapped in `try`/`catch`; on inner throw, the discriminator is the literal `"throw:<unreadable>"`. On a coercion failure (the `String(error)` call itself throws — e.g. an `error` whose `Symbol.toPrimitive` or `toString` throws), the discriminator falls back to the literal string `"throw:<unreadable>"` — the `"throw:"` prefix is preserved so operator deduplication on `(code, details.failure)` continues to bucket coercion-failure rows under the throw discriminator rather than collapsing them into a separate `"<unreadable>"` bucket that an operator filter on `startsWith("throw:")` would silently miss. The bare `"<unreadable>"` fallback convention is shared with the `details.observed` and `details.error` payloads pinned elsewhere in this paragraph and under **Per-step isolation** below — those payloads carry no `"throw:"` prefix to preserve and are unchanged by this rule; the `details.failure` field name itself is deliberately **not** reused from `details.observed`, since `details.failure` carries an internal failure-mode discriminator from a closed discriminator set (two literal forms — `"missing-entry"`, `"literals-shape-invalid"` — plus the `"throw:<…>"` template family whose payload is constructed at this emission site per the truncation-and-escape rule above) whereas `details.observed` carries an externally-observed SDK value, and operator triage tooling that keys on `details.observed` MUST NOT silently extend that key's semantics to the constant-unreadable code. The `loom/host/session-shutdown-pinned-constant-unreadable` emission is itself wrapped per the [**Diagnostic-emission isolation**](./diagnostic-emission-isolation.md#diagnostic-emission-isolation) rule below; it is **not** added to the always-log set, matching the `reload-teardown-timeout`, `session-shutdown-reason-unknown`, and `session-shutdown-teardown-step-failed` exclusions. (See [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 for the build-time gate that catches snapshot/SDK union drift.) A snapshot whose `literals` field is a structurally-valid non-empty array of non-empty strings but whose element values do not match the Pi-side `SessionShutdownEvent['reason']` union (e.g. a typo like `["quitt", "reload", "new", "resume", "fork"]` in place of the intended `"quit"`) is **not** runtime-detectable: the four `"literals-shape-invalid"` sub-cases above are the only structural failures the handler can recognise without an oracle for the SDK union, and a value-level mismatch passes all of them. Such a typo is caught exclusively by the build-time `loom/typecheck/session-shutdown-reason-snapshot` assertion owned by [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below, which fails `npm run typecheck` before the snapshot ever reaches a user runtime.

   *Sub-anchors and pinned substrings.* The three runtime-side obligations this rule pins are independently verifiable and each carry their own stable sub-anchor so a reviewer marking one "satisfied" cannot silently elide the other two: <a id="unknown-reason-rule-membership-check"></a>**(i) Closed-set membership check** — the runtime's set-membership branch above against the snapshot's `literals` field, including the unknown-reason fallback (`loom/host/session-shutdown-reason-unknown` plus the full five-sub-step teardown), pinned at [`#unknown-reason-rule-membership-check`](#unknown-reason-rule-membership-check); <a id="unknown-reason-rule-constant-source"></a>**(ii) Constant-source pinning (single-site edit)** — the rule that the closed set above is sourced from the `SessionShutdownEvent['reason']` snapshot entry in the extension module's source-of-truth pinned-constants block and that the runtime's set-membership check consumes that snapshot entry's `literals` field with no separate copy in the handler, pinned at [`#unknown-reason-rule-constant-source`](#unknown-reason-rule-constant-source); <a id="unknown-reason-rule-handler-trycatch"></a>**(iii) Handler-internal `try`/`catch` discipline (including the unreadable-constant diagnostic)** — the handler-entry `try`/`catch` discipline covering both the snapshot lookup-and-literal-array read and the `event.reason` read in the fixed order (snapshot first, then `event.reason`), the implementation freedom between a single widened `try`/`catch` with an in-progress state variable and two adjacent `try`/`catch` blocks, the catch-arm-per-failing-read discriminator (snapshot-side failure → `pinned-constant-unreadable`; snapshot-OK and `event.reason`-side throw → `reason-unknown` with `details.observed = "<unreadable>"`), the routing through the unknown-reason path on a throwing lookup / missing entry / non-array `literals`, and the `loom/host/session-shutdown-pinned-constant-unreadable` (W, runtime) diagnostic with its `"missing-entry"` / `"literals-shape-invalid"` / `"throw:<String(error)>"` discriminator literals, pinned at [`#unknown-reason-rule-handler-trycatch`](#unknown-reason-rule-handler-trycatch). The build-time obligations that pair with this rule (the bidirectional `SessionShutdownEvent['reason']` type-equality assertion, its `loom/typecheck/session-shutdown-reason-snapshot` brand string, and the tsconfig-include / import-specifier obligations) are owned by [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step 5 below — not by this rule — and are not in scope for sub-anchor (i), (ii), or (iii). The following substrings are **anchor-stable contract surface** wherever they appear as inbound references from other sections, topic files, or downstream documents that key on the literal substring: a future edit MUST NOT drift them without updating every such inbound reference in the same commit — <a id="unknown-reason-rule-substring-a"></a>(a) the diagnostic code `loom/host/session-shutdown-reason-unknown`; <a id="unknown-reason-rule-substring-b"></a>(b) the diagnostic code `loom/host/session-shutdown-pinned-constant-unreadable`; <a id="unknown-reason-rule-substring-c"></a>(c) the brace-form closed-set literal `{"quit", "reload", "new", "resume", "fork"}`; and <a id="unknown-reason-rule-substring-d"></a>(d) the three pinned-constant-unreadable discriminator literals `"missing-entry"`, `"literals-shape-invalid"`, and the `"throw:<String(error)>"` template. Editorial drift in any of (a)–(d) at an inbound-reference site is a contract change, not a wording change, and is subject to the same inbound-reference sweep as a sub-anchor rename — where an *inbound-reference sweep* means searching the spec corpus (`docs/spec.md` and `docs/spec_topics/`) for the literal substring or anchor fragment, and updating each citation in the same commit as the substring change. The in-prose occurrences of the (c) literal inside the **Unknown-reason rule** paragraph immediately above — namely the introductory listing in the `event.reason` validation sentence (`the closed set {"quit", "reload", "new", "resume", "fork"}`) and the `"quit" | "reload" | "new" | "resume" | "fork"` parenthesised re-listing later in that same paragraph — are explicitly **out of scope** for the anchor-stable contract surface above: they are illustrative restatements of the `SessionShutdownEvent['reason']` snapshot constant per [Pi version bump procedure step 1](./version-bump-intro.md#pi-version-bump-procedure)'s carve-out and may transiently lag the snapshot during a patch-skew window without triggering an inbound-reference sweep. The carve-out is symmetric for the (a), (b), and (d) literals when they appear inside the **Unknown-reason rule** paragraph as restatements of the diagnostic codes and discriminator literals defined there; only inbound references from outside that paragraph are anchor-stable.
