# Drain state contract

   <a id="idempotent-handler-clause"></a>The handler is **idempotent**: any `session_shutdown` invocation that observes the extension-scoped `LoomRegistry` either already drained or already carrying a drain-state tag in either arm of sub-step 1's closed two-arm tag set at handler entry — whether a re-entrant invocation fired *before* the first returns, a fresh invocation fired *after* the first returned on a `reason: "new" | "resume" | "fork"` branch and left the runtime in the `"degraded-needs-reload"` arm of sub-step 1's drain-state tag set per the edge-case bullet below, or a fresh invocation fired *after* a prior invocation on which sub-step 1's `LoomRegistry.drain` call threw (and was caught by the **Per-step isolation** paragraph above, leaving `drained = false`) but on which a subsequent state-setting call (sub-step 1's `LoomRegistry.initDrainStateTag` setting the tag to `"shutting-down"`, or the post-sub-step-4 `LoomRegistry.markRuntimeDegraded` setting it to `"degraded-needs-reload"`) succeeded and recorded a tag value — is a no-op: the handler returns immediately without re-executing any of sub-steps 1–5 (so sub-step 4 never double-closes the already-closed `discoveryWatcher`/`settingsWatcher` or re-`Clock.clearTimeout`s the already-cleared debounce handle, and chokidar's unpinned double-close behaviour is consequently off the load path — no second `loom/host/session-shutdown-teardown-step-failed` row can be emitted from a re-entry sub-step 4 because sub-step 4 does not re-run) and without re-emitting any of the five teardown-handler diagnostics enumerated in the **Edge cases** bullet on diagnostic emission below; in particular, `loom/host/session-shutdown-runtime-degraded` fires **at most once per extension-instance lifetime, except on the all-three-throw corner case enumerated later in this paragraph and the `readDrainState`-throw re-entry case enumerated under the [*Read-failure fallback*](#read-failure-fallback) clause of `LoomRegistry.readDrainState`'s bullet below, on each of which one additional row is emitted per follow-up `session_shutdown` event whose handler proceeds through the diagnostic-emission predicate** — on the steady path the row is emitted by the first invocation that takes the post-sub-step-4 transition; on the *all-three-throw corner case* enumerated later in this paragraph (sub-step 1's `LoomRegistry.drain`, sub-step 1's `LoomRegistry.initDrainStateTag`, and the post-sub-step-4 `LoomRegistry.markRuntimeDegraded` all throw on a session-only `session_shutdown`, leaving `drained = false` and `tag = undefined` so the broadened short-circuit predicate `snapshot.drained === true || snapshot.tag !== undefined` does not fire on re-entry), additional `loom/host/session-shutdown-runtime-degraded` rows MAY be emitted on follow-up session-only `session_shutdown` events against the same extension instance — the per-instance bound is breached for this corner case only, and operator deduplication on `(code, details.event.reason)` collapses the resulting duplicate rows; and the diagnostics-row phrase "exactly once per `session_shutdown` event in this branch" is to be read as scoped to invocations that actually reach the branch, which a short-circuited re-entry does not. This applies symmetrically across `event.reason`: a follow-up `"quit"` (or any other closed-set reason, or an unknown reason, or a snapshot-failure path) fired after an initial `"new"`/`"resume"`/`"fork"` left the runtime degraded is short-circuited at handler entry just like a same-reason re-entry, and the "runtime is being torn down anyway" rationale that suppresses `session-shutdown-runtime-degraded` on the primary `"quit"`/`"reload"` paths is preserved on the re-entry path because the entire handler (including the post-sub-step-4 transition that would otherwise write the `"degraded-needs-reload"` tag and emit the diagnostic) is short-circuited before any sub-step runs. The short-circuit predicate is the disjunction `snapshot.drained === true || snapshot.tag !== undefined` observed at handler entry — i.e., either the registry-drained flag is set **or** the drain-state tag carries either arm of sub-step 1's closed two-arm tag set — so a prior invocation on which `LoomRegistry.drain` threw (leaving `drained = false`) but on which `LoomRegistry.initDrainStateTag` or the post-sub-step-4 `LoomRegistry.markRuntimeDegraded` subsequently succeeded still establishes the short-circuit on the next handler invocation, and the rule is uniform across the `"shutting-down"` and `"degraded-needs-reload"` arms; the **Unknown-reason rule**'s pre-sub-step-1 emissions (`loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`) are likewise skipped on a short-circuited re-entry because the predicate check happens before those emissions on the re-entry path, preserving the at-most-one-per-extension-instance-lifetime bound on every teardown-handler diagnostic. *Handler control-flow ordering.* To make the relative ordering above unambiguous on both first-entry and re-entry, the handler's fixed control-flow at entry is exactly: **(I)** `LoomRegistry.readDrainState()` and evaluation of the short-circuit predicate `snapshot.drained === true || snapshot.tag !== undefined` per the broadened predicate above — this is the first observable side-effect of the handler on every invocation; carve-out (b) of the **Unknown-reason rule**'s pre-sequence-operations enumeration permits the variable pre-initialisations of `pinnedConstantReadOk` (to `false`) and `capturedEventReason` (to the literal string `"<unreadable>"`) to precede step (I) as handler-entry preamble (those pre-initialisations write only handler-scoped local variables and produce no observable side-effect outside the handler frame); **(II)** if the predicate fires, return immediately, with **no** snapshot lookup-and-`literals` read, **no** `event.reason` read, **no** diagnostic emission, and **no** sub-step run; **(III)** only on a false predicate, the snapshot lookup-and-`literals` read of the **Unknown-reason rule** above (which sets `pinnedConstantReadOk`); **(IV)** the `event.reason` read of the **Unknown-reason rule** above (which populates the handler-scoped `capturedEventReason`); **(V)** the **Unknown-reason rule**'s pre-sub-step-1 `loom/host/session-shutdown-pinned-constant-unreadable` / `loom/host/session-shutdown-reason-unknown` emissions per the catch-arm-discriminator rules above; **(VI)** sub-step 1 onward. The pre-sub-step-1 actions enumerated by steps (I) and (III)–(V) of this ordering, together with the handler-entry pre-initialisations of `capturedEventReason` and `pinnedConstantReadOk` that step (III)/(IV) and the *Predicate split* clause consume, are the same set the **Unknown-reason rule** above's *exactly three pre-sequence operations* enumeration permits — operations (1)–(3) of that enumeration cover the snapshot lookup-and-`literals` read (step (III)), the `event.reason` read (step (IV)), and the set-membership comparison whose non-member arm drives step (V), and the carve-out (a)/(b) of that enumeration covers step (I)'s `readDrainState` short-circuit and the two variable pre-initialisations respectively; the two enumerations are aligned by reference and a future edit to either MUST sweep both in the same commit. The Unknown-reason rule's *emits before sub-step 1 runs* obligation is satisfied on first-entry by step (V) preceding step (VI) of this ordering, not by being the absolute first action of the handler; an implementation that places the predicate check after the snapshot lookup, after the `event.reason` read, or after the unknown-reason / pinned-constant-unreadable emissions leaks a `loom/host/session-shutdown-pinned-constant-unreadable` (or `loom/host/session-shutdown-reason-unknown`) row on the re-entry path and is non-conformant, and an implementation that places the predicate check before step (VI) but skips steps (III)–(V) on a *false* predicate (i.e. on first-entry) suppresses the Unknown-reason rule's emissions entirely and is also non-conformant. The all-three-throw corner case (drain throws, `initDrainStateTag` throws, `markRuntimeDegraded` throws) leaves both `drained = false` and `tag = undefined` so the predicate evaluates to `false` and a follow-up `session_shutdown` re-runs the full handler — including sub-step 4 against the already-closed `discoveryWatcher`/`settingsWatcher` and the already-cleared debounce handle, on chokidar's acknowledged-unpinned double-close behaviour; this is the loom 1.0 acceptance, the per-call `try`/`catch` of the **Per-step isolation** paragraph already routes any double-close throw through one additional `loom/host/session-shutdown-teardown-step-failed` row per failing call site, and no further compensation is pinned because all three swallow rules above already individually fired and the runtime carries no further state seam to recover from. *Slash-dispatch interaction.* The same `(drained: false, tag: undefined)` post-handler state means slash dispatches arriving between the failed handler returning and the next `session_shutdown` event observe the steady-state tuple at the slash-handler `LoomRegistry.readDrainState` site and route through arm (a) of that read's three-arm enumeration below — i.e., dispatch the loom — into a runtime where sub-step 4 has already closed both `discoveryWatcher`/`settingsWatcher`, sub-step 5 has already detached the forwarding listeners on `ctx.signal`, the tool-exposed adapter's `signal`, and the `invoke`-parent signal, and sub-step 2 has already aborted any then-in-flight `loomAbort`s. loom 1.0 accepts this silent dispatch into the half-torn-down runtime as an additional residual gap on the same loom 1.0-acceptance footing as the duplicate-emission and double-close behaviour pinned in the rest of this paragraph (the corner case is rare — it requires three independent swallowed throws on a single session-only `session_shutdown` — and operator recovery is one `/reload` away regardless, on the same recovery footing as every other arm of the degraded-state branch). <a id="pic-30"></a> **PIC-30.** The runtime MUST NOT introduce a third boolean drain-state field, an arm-specific "post-failed-handler" gate on `LoomRegistry.readDrainState`, or any other suppression mechanism that would conditionally short-circuit slash dispatch on this post-handler tuple — the `readDrainState` arm enumeration in the **`LoomRegistry` drain-state contract**'s *Methods* paragraph below is closed at three arms ((a) dispatch, (b) shutting-down note, (c) degraded-needs-reload note) and a fourth arm or an alternative gating field is forbidden by that contract. The full re-run also re-evaluates the *Predicate split* clause's diagnostic-emission predicate `pinnedConstantReadOk && capturedEventReason ∈ {"new", "resume", "fork"}` from scratch — so on a follow-up session-only `event.reason` (`"new"`/`"resume"`/`"fork"`) with a successful snapshot read on that follow-up, the runtime emits an additional `loom/host/session-shutdown-runtime-degraded` row, breaching the at-most-once-per-extension-instance-lifetime bound stated earlier in this paragraph for this corner case only (cross-referenced from the bound itself above and from the *Predicate split* clause's *all-three-throw corner-case interaction* sub-paragraph below); operator deduplication on `(code, details.event.reason)` collapses the resulting duplicate rows. It does **not** call `ctx.reload()` (Pi is already executing it) and does **not** call `pi.unregisterTool` (Pi exposes none; the registration-cache leak is the loom 1.0 cosmetic acceptance documented under **Tool-registration lifetime and visibility** below).

   <a id="loom-registry-drain-state-contract"></a>**`LoomRegistry` drain-state contract.** The `LoomRegistry` defined in step 5 above — whose entry table is a `Map<slashName, parsedLoom>` per that step — additionally carries two drain-related fields and exposes a closed four-method call surface for them; method-call framing is normative throughout the **`session_shutdown`** handler and step 5 above, and the property-write framing elsewhere in this section (frozen `LoomRegistry`, proxied setter, OOM during property assignment) names the implementation-side *throws* those methods may surface to the per-call `try`/`catch` discipline pinned by the **Per-step isolation** paragraph and by the [interstitial-seam swallow rule](./patch-skew-degradation.md#interstitial-seam-tag-write-swallow), not an alternative property-access API the spec sanctions.

   *Fields.* (1) A boolean `drained` flag, initialised to `false` when the factory constructs the registry in step 5 above (the only pre-factory state — there is no constructor argument and no other writer establishes this field) and flipped to `true` exactly once by sub-step 1's `LoomRegistry.drain` call; never reset for the lifetime of the extension instance, so an operator-issued `/reload` re-establishes a fresh `LoomRegistry` with `drained = false` rather than reusing the existing one. (2) A `drainStateTag` field of TypeScript type `"shutting-down" | "degraded-needs-reload" | undefined` (interpreted normatively per [sub-step 1's *unset tag fallback*](./patch-skew-degradation.md#substep-1-unset-tag-fallback) above as the closed two-arm value set `"shutting-down" | "degraded-needs-reload"` plus the unset runtime state `undefined` — the latter being an implementation-side artefact of the **Per-step isolation** swallow rule rather than a third literal value), initialised to `undefined` at registry construction (the third state outside the closed two-arm tag set sub-step 1 pins, reachable also after a sub-step-1 `LoomRegistry.initDrainStateTag` throw the **Per-step isolation** paragraph caught — both initial-`undefined` and tag-init-throw-`undefined` cases share the same field value); set to `"shutting-down"` by sub-step 1's `LoomRegistry.initDrainStateTag` call when that call returns successfully; transitioned to `"degraded-needs-reload"` by the interstitial-seam `LoomRegistry.markRuntimeDegraded` call between sub-steps 4 and 5 on the branch the *Predicate split* clause of the `reason: "new" | "resume" | "fork"` edge-case bullet below pins; never otherwise mutated. The two fields are independent: `drained` does not constrain `drainStateTag` and vice versa, and the slash-handler short-circuit predicates below consume them as a tuple rather than via any derived single-value collapse.

   *Methods.* The runtime mediates the two drain-state fields above through the method-call surface enumerated below; each method's behavioural contract, and the closed `drainStateTag` value set it maintains, is normative. *Non-normative editorial convention.* loom 1.0 keeps this surface closed to the four methods below — as an internal-shape convention, no other writer or reader of either field is permitted, and the spec is not to be read as authorising direct property writes (`registry.drained = true`, `registry.drainStateTag = "shutting-down"`) as an alternative to the calls below. This convention exists for loom's own maintainability and is not an observable conformance point: a caller cannot distinguish a conformant runtime from one that adds an internal helper. The internal field is named `drainStateTag`; the public snapshot key on `readDrainState`'s return value is the shorter `tag`. The rename is intentional — the snapshot's tuple keys are kept short to keep slash-handler dispatch sites concise (`snapshot.tag === "shutting-down"` rather than `snapshot.drainStateTag === "shutting-down"`) — and the two names refer to the same field; an editorial unification of either name to the other would have to sweep the sibling method names that already embed the longer field name (in full or as its `DrainState` prefix) — `initDrainStateTag` (embedding `DrainStateTag`) and `readDrainState` (embedding the `DrainState` prefix), the only two of the four `LoomRegistry` methods that do so — plus the `"loomRegistry.initDrainStateTag"` `details.call` label under the **Per-step isolation** paragraph above (the only label in that closed call-site set that embeds `DrainStateTag`), and every prose site that references the longer field name (including the **Host prerequisites for the degraded-state branch** bullet (a) above and sub-step 1's *unset tag fallback* clause), and is out of scope for loom 1.0.

   - <a id="pic-32"></a> **PIC-32.** `LoomRegistry.drain(): void` — sets `drained = true`. Called from sub-step 1 of the `session_shutdown` handler above; the per-call `try`/`catch` of the **Per-step isolation** paragraph above wraps this call under `details.call = "loomRegistry.drain"`. Self-idempotent on re-call (a re-entry would re-write the same `true`), but re-entry is short-circuited at handler entry by the [**idempotent**](#idempotent-handler-clause) clause above before this call is reached.
   - `LoomRegistry.initDrainStateTag(): void` — sets `drainStateTag = "shutting-down"` iff `drainStateTag === undefined`; a no-op when `drainStateTag` is already a member of the closed two-arm tag set (so a re-call after a successful first call cannot down-grade `"degraded-needs-reload"` back to `"shutting-down"`). Called from sub-step 1 immediately after `LoomRegistry.drain` and wrapped under `details.call = "loomRegistry.initDrainStateTag"` per the **Per-step isolation** paragraph above.
   - `LoomRegistry.markRuntimeDegraded(): void` — sets `drainStateTag = "degraded-needs-reload"` unconditionally (a tag-init throw the **Per-step isolation** paragraph caught leaves the field `undefined`, and this call's contract is to overwrite that, not to assert on the prior state). Called from the interstitial seam between sub-steps 4 and 5 whenever the broader **tag-transition predicate** `capturedEventReason ∉ {"quit", "reload"}` of the *Predicate split* clause below fires (the broader predicate, NOT the narrower diagnostic-emission predicate); the seam's dedicated `try`/`catch` per the **Per-step isolation** paragraph above wraps this call and silently swallows on throw — no `loom/host/session-shutdown-teardown-step-failed` row is emitted, the call-site label `"loomRegistry.markRuntimeDegraded"` exists for prose cross-reference only; emits no `details.call` due to the swallow rule.
   - `LoomRegistry.readDrainState(): { drained: boolean; tag: "shutting-down" | "degraded-needs-reload" | undefined }` — returns a snapshot of the two fields above. This is the **single read API** the slash-command `handler` registered at the [slash-handler registration site](./registration-steps.md#slash-handler-registration) in step 3 above consults at handler entry to choose between three behaviours: (a) **dispatch** the loom when `snapshot.drained === false && snapshot.tag === undefined` (the steady-state path — both fields at their factory-construction values); (b) short-circuit on the `"shutting-down"` arm, returning the cancelled-binder-style `"loom /<name>: extension shutting down"` system note, when `snapshot.tag === "shutting-down"` or `(snapshot.drained === true && snapshot.tag === undefined)` — the former covering both the steady drained-and-tag-init-succeeded case and the `LoomRegistry.drain`-threw-but-`LoomRegistry.initDrainStateTag`-succeeded recovery case (where `drained` is `false` but the tag carries the in-progress arm), and the latter covering the tag-init-throw degraded-safely fallback sub-step 1 above pins; (c) short-circuit on the `"degraded-needs-reload"` arm, returning the distinct `"loom /<name>: extension degraded; /reload to recover"` system note, when `snapshot.tag === "degraded-needs-reload"` — regardless of `snapshot.drained`, which covers both the steady post-sub-step-4 case (`drained === true`) and the `LoomRegistry.drain`-threw-but-`LoomRegistry.markRuntimeDegraded`-succeeded recovery case (`drained === false`). <a id="pic-29"></a> **PIC-29.** The three arms are mutually exclusive and exhaust the field-tuple state space (six tuples: `(false, undefined)` → arm (a); `(false, "shutting-down")` and `(true, "shutting-down")` and `(true, undefined)` → arm (b); `(false, "degraded-needs-reload")` and `(true, "degraded-needs-reload")` → arm (c)); the slash handler MUST NOT introduce a fourth arm or read the two fields through any other accessor. The **idempotent** clause above's short-circuit predicate is the disjunction `snapshot.drained === true || snapshot.tag !== undefined` over this same accessor's return value, read once at `session_shutdown` handler entry; the predicate keys on the union of the drained flag and tag-presence and is uniform across the `"shutting-down"` and `"degraded-needs-reload"` arms per that clause — equivalently, the short-circuit fires on every tuple except the steady-state `(drained: false, tag: undefined)` that arm (a) above dispatches. <a id="read-failure-fallback"></a><a id="pic-31"></a> **PIC-31.** *Read-failure fallback.* The `readDrainState` call itself MUST be wrapped in a per-call `try`/`catch` at both of its two call sites (the slash-command `handler` registered in step 3 above on every dispatch, and the `session_shutdown` handler at step (I) of the *Handler control-flow ordering* clause in the **idempotent** clause above), to absorb a throw out of the read (a frozen-but-mutated-import edge case, a future Proxy-backed field with a getter trap, OOM during the snapshot construction, or any other read-side throw at all the four sibling methods' per-call swallow rules already protect against on the write side). On a `readDrainState` throw the catch arm is the conservative fail-safe: at the **slash-command** call site the dispatch MUST short-circuit on arm (c) (returning the `"loom /<name>: extension degraded; /reload to recover"` system note) — the runtime cannot determine from a thrown read whether it is in the steady-state, shutting-down, or degraded-needs-reload arm and the degraded-arm note is the only one whose operator action (`/reload`) is correct on every non-dispatch arm; at the **`session_shutdown` handler-entry step (I)** call site the catch arm MUST treat the read as if it had returned the steady-state tuple `(drained: false, tag: undefined)` — equivalently, as if the predicate had **not** fired — so the handler proceeds into steps (III)–(VI) and runs the full five-sub-step teardown sequence as a fail-safe, rather than short-circuiting and silently stranding every resource the handler is responsible for releasing (chokidar `discoveryWatcher`/`settingsWatcher` handles, the debounce-handle `Clock.clearTimeout(...)` call, the forwarding listeners sub-step 5 detaches, the in-flight `loomAbort.abort()`s sub-step 2 fires, and the `disposeBarrier` awaits sub-step 3 owns) on a first-entry `session_shutdown` (the only event the handler will receive in a `/quit` or `/reload` window). This fail-safe is **asymmetric** with the slash-command call site's arm-(c) short-circuit above: at the slash-command site the degraded-arm note is the conservative operator action on every non-dispatch arm, but at the handler-entry site short-circuiting on a first-entry read failure would leak every teardown resource above silently (the rule explicitly forbids any diagnostic from this catch arm), so the spec routes the handler-entry catch arm in the opposite direction — towards executing the teardown rather than skipping it. The trade-off this fail-safe accepts is that on a re-entry the **idempotent** clause's predicate would otherwise have caught, the read-failure conceals the prior-shutdown signal from the predicate and the full sub-step sequence re-runs against an already-drained registry — additional `loom/host/session-shutdown-runtime-degraded` and other teardown-handler diagnostic rows are then emitted, breaching the at-most-once-per-extension-instance bound for this read-failure case on the same loom 1.0-acceptance footing as the *all-three-throw corner case* enumerated by the **idempotent** clause above (whose *all-three-throw corner-case interaction* sub-paragraph in the [`reason: "new" | "resume" | "fork"` edge-case bullet](./session-only-degraded-state.md#session-only-reason-degraded-state) below explicitly enumerates the same duplicate-emission shape on its own re-entry path); the duplicate emission on this read-failure branch is the loom 1.0-accepted path on the same footing as that corner case, and the runtime MUST NOT add a suppression mechanism — a sentinel flag, an in-handler dedup cache, or any other gate — that would conditionally swallow the second-and-later emissions on the read-failure re-entry; operator deduplication on `(code, details.event.reason)` collapses the resulting duplicate rows. The fail-safe also re-traverses sub-step 4 against the already-closed `discoveryWatcher`/`settingsWatcher` and the already-cleared debounce handle on chokidar's acknowledged-unpinned double-close behaviour on the re-entry path, on the same loom 1.0-acceptance grounds the *all-three-throw corner case* paragraph above pins (the per-call `try`/`catch` of the **Per-step isolation** paragraph already routes any double-close throw through one additional `loom/host/session-shutdown-teardown-step-failed` row per failing call site). No dedicated `readDrainState`-throw diagnostic is emitted from either catch arm in loom 1.0 — the slash-handler symptom (every subsequent dispatch returning the degraded-arm note rather than dispatching) is operator-observable in-band, and the handler-entry fail-safe surfaces any sub-step-internal throws through the existing per-step-isolation `loom/host/session-shutdown-teardown-step-failed` channel and the existing `(code, details.step, details.call)` deduplication keys; this is the loom 1.0 acceptance, on the same footing as the interstitial-seam swallow of `LoomRegistry.markRuntimeDegraded`'s own throw under the **Per-step isolation** paragraph above. A future edit MAY add a dedicated diagnostic for this read-failure path (e.g., `loom/host/loom-registry-read-failed`), but doing so requires a parallel edit to [Diagnostics](../diagnostics.md) and is out of scope for loom 1.0.

   The four-method call surface above is the source of truth for the method names referenced from sub-step 1, the **Per-step isolation** paragraph, the [**idempotent**](#idempotent-handler-clause) clause, and the [`reason: "new" | "resume" | "fork"` edge-case bullet](./session-only-degraded-state.md#session-only-reason-degraded-state)'s interstitial-seam transition reference; an editorial rename of any method MUST sweep all four sites in the same edit. *Non-normative editorial convention (cont.).* Consistent with the convention above, loom 1.0 does not add a fifth **drain-state** method to this surface or expose either of the two drain-state fields through a public accessor beyond `readDrainState`. *Scope of the closed surface and the property-write prohibition.* The closed four-method surface above governs the **drain-state** contract only — the two drain-state fields (`drained`, `drainStateTag`) and their reader/writer methods. The underlying entry-table `Map<slashName, parsedLoom>` interface that the slash-command [`handler`](./registration-steps.md#slash-handler-registration) consults to dispatch the loom (per the *Drain-state-gated dispatch* clause's arm (a) above, which routes the dispatch through this entry-table lookup after `readDrainState` returns the steady-state tuple) is unchanged from step 5 below's original `LoomRegistry` description; direct read access via that map's standard read interface (`Map.prototype.get`, `Map.prototype.has`, `Map.prototype.keys`, `Map.prototype.values`, `Map.prototype.entries`, the `for...of` iteration the watcher's rebuild path consumes per step 5 below, and the `Map.prototype.size` accessor the [Diagnostics](../diagnostics.md) row counts read) is permitted on the same footing as before this contract was added. The opening sentence of this paragraph's `no other writer or reader of either field is permitted` clause and the `direct property writes` prohibition that follows it are likewise **scoped to the two drain-state fields only** (`drained` and `drainStateTag`); they do not apply to the entry table or to any other `LoomRegistry` member step 5 below establishes. Methods unrelated to drain state (e.g., the entry-table lookup described earlier in this paragraph) are out of scope of the closed-surface prohibition — additions are spec edits to this paragraph rather than implementation freedom.
