# Drain state contract

> **Partial supersession (governed-by-rebind, loom 1.0).** Host-prerequisites [clause (a)](./host-prerequisites.md#degraded-state-host-prerequisites) resolved `governed-by-rebind`, retiring the session-only degraded-state branch. The former `LoomRegistry.markRuntimeDegraded` writer, the `"degraded-needs-reload"` value of `drainStateTag`, and `readDrainState` arm (c) have been **excised**: the session-only reasons now run full teardown and arm the private [session-swap fail-fast tripwire](./session-only-degraded-state.md#session-swap-fail-fast-tripwire) (a flag outside this drain-state surface). The **live** contract is the two-field/three-method surface with `readDrainState` reduced to the two reachable arms **(a) dispatch** (`(drained: false, tag: undefined)`) and **(b) shutting-down**. `PIC-29`…`PIC-32` remain as shipped by `V9m` in their reworded two-arm form.

   <a id="idempotent-handler-clause"></a>The handler is **idempotent**: any `session_shutdown` invocation that observes the extension-scoped `LoomRegistry` either already drained or already carrying a `"shutting-down"` drain-state tag at handler entry — whether a re-entrant invocation fired *before* the first returns, or a fresh invocation fired *after* a prior invocation on which sub-step 1's `LoomRegistry.drain` threw (caught by the **Per-step isolation** paragraph above, leaving `drained = false`) but on which sub-step 1's `LoomRegistry.initDrainStateTag` subsequently succeeded — is a no-op: the handler returns immediately without re-executing any of sub-steps 1–5 (so sub-step 4 never double-closes the already-closed `discoveryWatcher`/`settingsWatcher` or re-`Clock.clearTimeout`s the already-cleared debounce handle) and without re-emitting any teardown-handler diagnostic. The short-circuit predicate is the disjunction `snapshot.drained === true || snapshot.tag !== undefined` observed at handler entry — either the registry-drained flag is set **or** the drain-state tag carries `"shutting-down"` — so the short-circuit fires on every tuple except the steady-state `(drained: false, tag: undefined)` that arm (a) dispatches. The **Unknown-reason rule**'s pre-sub-step-1 emissions (`loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`) are likewise skipped on a short-circuited re-entry because the predicate check happens before those emissions, preserving the at-most-one-per-extension-instance-lifetime bound on every teardown-handler diagnostic. *Handler control-flow ordering.* The handler's fixed control-flow at entry is exactly: **(I)** `LoomRegistry.readDrainState()` and evaluation of the short-circuit predicate `snapshot.drained === true || snapshot.tag !== undefined` — the first observable side-effect of the handler on every invocation; carve-out (b) of the **Unknown-reason rule**'s pre-sequence-operations enumeration permits the variable pre-initialisations of `pinnedConstantReadOk` (to `false`) and `capturedEventReason` (to the literal string `"<unreadable>"`) to precede step (I) as handler-entry preamble; **(II)** if the predicate fires, return immediately, with **no** snapshot lookup-and-`literals` read, **no** `event.reason` read, **no** diagnostic emission, and **no** sub-step run; **(III)** only on a false predicate, the snapshot lookup-and-`literals` read of the **Unknown-reason rule** (which sets `pinnedConstantReadOk`); **(IV)** the `event.reason` read of the **Unknown-reason rule** (which populates `capturedEventReason`); **(V)** the **Unknown-reason rule**'s pre-sub-step-1 `loom/host/session-shutdown-pinned-constant-unreadable` / `loom/host/session-shutdown-reason-unknown` emissions; **(VI)** sub-step 1 onward. The pre-sub-step-1 actions of steps (I) and (III)–(V), together with the handler-entry pre-initialisations of `capturedEventReason` and `pinnedConstantReadOk`, are the same set the **Unknown-reason rule** above's *exactly three pre-sequence operations* enumeration permits; the two enumerations are aligned by reference and a future edit to either MUST sweep both in the same commit. The Unknown-reason rule's *emits before sub-step 1 runs* obligation is satisfied on first-entry by step (V) preceding step (VI). *Both-sub-step-1-acts-throw corner case.* When sub-step 1's `LoomRegistry.drain` **and** `LoomRegistry.initDrainStateTag` both throw on a `session_shutdown` (each caught by **Per-step isolation**, leaving `drained = false` and `tag = undefined`), the short-circuit predicate evaluates to `false` and a follow-up `session_shutdown` re-runs the full handler — including sub-step 4 against the already-closed watchers, on chokidar's acknowledged-unpinned double-close behaviour; this is the loom 1.0 acceptance, the per-call `try`/`catch` of **Per-step isolation** already routes any double-close throw through one additional `loom/host/session-shutdown-teardown-step-failed` row per failing call site, and no further compensation is pinned. *Slash-dispatch interaction.* The same `(drained: false, tag: undefined)` post-handler state means slash dispatches arriving between the failed handler returning and the next `session_shutdown` observe the steady-state tuple and route through arm (a) — i.e., dispatch the loom — into a runtime whose watchers are already closed, forwarding listeners already detached, and in-flight `loomAbort`s already aborted; loom 1.0 accepts this silent dispatch into the half-torn-down runtime as a residual gap (it requires two independent swallowed throws on a single `session_shutdown`, and operator recovery is one `/reload` away). <a id="pic-30"></a> **PIC-30.** The runtime MUST NOT introduce a third boolean drain-state field, an arm-specific "post-failed-handler" gate on `LoomRegistry.readDrainState`, or any other suppression mechanism that would conditionally short-circuit slash dispatch on this post-handler tuple — the `readDrainState` arm enumeration in the *Methods* paragraph below is closed at two arms ((a) dispatch, (b) shutting-down note) and a third arm or an alternative gating field is forbidden. It does **not** call `ctx.reload()` (Pi is already executing it) and does **not** call `pi.unregisterTool` (Pi exposes none; the registration-cache leak is the loom 1.0 cosmetic acceptance documented under **Tool-registration lifetime and visibility** below).

   <a id="loom-registry-drain-state-contract"></a>**`LoomRegistry` drain-state contract.** The `LoomRegistry` defined in step 5 above — whose entry table is a `Map<slashName, parsedLoom>` per that step — additionally carries two drain-related fields and exposes a closed three-method call surface for them; method-call framing is normative throughout the **`session_shutdown`** handler and step 5 above, and the property-write framing elsewhere in this section (frozen `LoomRegistry`, proxied setter, OOM during property assignment) names the implementation-side *throws* those methods may surface to the per-call `try`/`catch` discipline pinned by the **Per-step isolation** paragraph, not an alternative property-access API the spec sanctions.

   *Fields.* (1) A boolean `drained` flag, initialised to `false` when the factory constructs the registry in step 5 above (the only pre-factory state — there is no constructor argument and no other writer establishes this field) and flipped to `true` exactly once by sub-step 1's `LoomRegistry.drain` call; never reset for the lifetime of the extension instance, so an operator-issued `/reload` re-establishes a fresh `LoomRegistry` with `drained = false` rather than reusing the existing one. (2) A `drainStateTag` field of TypeScript type `"shutting-down" | undefined` (interpreted normatively per [sub-step 1's *unset tag fallback*](./session-shutdown-semantics.md#substep-1-unset-tag-fallback) above as the single value `"shutting-down"` plus the unset runtime state `undefined` — the latter being both the factory-construction value and an implementation-side artefact of the **Per-step isolation** swallow rule after a sub-step-1 `LoomRegistry.initDrainStateTag` throw), initialised to `undefined` at registry construction; set to `"shutting-down"` by sub-step 1's `LoomRegistry.initDrainStateTag` call when that call returns successfully; never otherwise mutated. The two fields are independent: `drained` does not constrain `drainStateTag` and vice versa, and the slash-handler short-circuit predicates below consume them as a tuple rather than via any derived single-value collapse.

   *Methods.* The runtime mediates the two drain-state fields above through the method-call surface enumerated below; each method's behavioural contract is normative. *Non-normative editorial convention.* loom 1.0 keeps this surface closed to the three methods below — as an internal-shape convention, no other writer or reader of either field is permitted, and the spec is not to be read as authorising direct property writes (`registry.drained = true`, `registry.drainStateTag = "shutting-down"`) as an alternative to the calls below. This convention exists for loom's own maintainability and is not an observable conformance point. The internal field is named `drainStateTag`; the public snapshot key on `readDrainState`'s return value is the shorter `tag`. The rename is intentional — the snapshot's tuple keys are kept short to keep slash-handler dispatch sites concise (`snapshot.tag === "shutting-down"` rather than `snapshot.drainStateTag === "shutting-down"`) — and the two names refer to the same field; an editorial unification of either name would have to sweep the sibling method names that embed the longer field name (`initDrainStateTag`, `readDrainState`), the `"loomRegistry.initDrainStateTag"` `details.call` label under the **Per-step isolation** paragraph above, and every prose site that references the longer field name, and is out of scope for loom 1.0.

   - <a id="pic-32"></a> **PIC-32.** `LoomRegistry.drain(): void` — sets `drained = true`. Called from sub-step 1 of the `session_shutdown` handler above; the per-call `try`/`catch` of the **Per-step isolation** paragraph above wraps this call under `details.call = "loomRegistry.drain"`. Self-idempotent on re-call (a re-entry would re-write the same `true`), but re-entry is short-circuited at handler entry by the [**idempotent**](#idempotent-handler-clause) clause above before this call is reached.
   - `LoomRegistry.initDrainStateTag(): void` — sets `drainStateTag = "shutting-down"` iff `drainStateTag === undefined`; a no-op when `drainStateTag` is already `"shutting-down"`. Called from sub-step 1 immediately after `LoomRegistry.drain` and wrapped under `details.call = "loomRegistry.initDrainStateTag"` per the **Per-step isolation** paragraph above.
   - `LoomRegistry.readDrainState(): { drained: boolean; tag: "shutting-down" | undefined }` — returns a snapshot of the two fields above. This is the **single read API** the slash-command `handler` registered at the [slash-handler registration site](./registration-steps.md#slash-handler-registration) in step 3 above consults at handler entry to choose between two behaviours: (a) **dispatch** the loom when `snapshot.drained === false && snapshot.tag === undefined` (the steady-state path — both fields at their factory-construction values); (b) short-circuit on the `"shutting-down"` arm, returning the cancelled-binder-style `"loom /<name>: extension shutting down"` system note, otherwise — i.e. when `snapshot.tag === "shutting-down"` or `snapshot.drained === true` — the former covering both the steady drained-and-tag-init-succeeded case and the `LoomRegistry.drain`-threw-but-`LoomRegistry.initDrainStateTag`-succeeded recovery case (where `drained` is `false` but the tag carries the in-progress arm), and the latter covering the tag-init-throw degraded-safely fallback (`drained === true`, `tag === undefined`) sub-step 1 above pins. <a id="pic-29"></a> **PIC-29.** The two arms are mutually exclusive and exhaust the field-tuple state space (four tuples: `(false, undefined)` → arm (a); `(false, "shutting-down")`, `(true, "shutting-down")`, and `(true, undefined)` → arm (b)); the slash handler MUST NOT introduce a third arm or read the two fields through any other accessor. The **idempotent** clause above's short-circuit predicate is the disjunction `snapshot.drained === true || snapshot.tag !== undefined` over this same accessor's return value, read once at `session_shutdown` handler entry — equivalently, the short-circuit fires on every tuple except the steady-state `(drained: false, tag: undefined)` that arm (a) above dispatches. <a id="read-failure-fallback"></a><a id="pic-31"></a> **PIC-31.** *Read-failure fallback.* The `readDrainState` call itself MUST be wrapped in a per-call `try`/`catch` at both of its two call sites (the slash-command `handler` registered in step 3 above on every dispatch, and the `session_shutdown` handler at step (I) of the *Handler control-flow ordering* clause above), to absorb a throw out of the read (a frozen-but-mutated-import edge case, a future Proxy-backed field with a getter trap, OOM during the snapshot construction, or any other read-side throw the sibling methods' per-call swallow rules already protect against on the write side). On a `readDrainState` throw the catch arm is the conservative fail-safe: at the **slash-command** call site the dispatch MUST short-circuit on arm (b) (returning the `"loom /<name>: extension shutting down"` system note) rather than dispatch the loom — the runtime cannot determine from a thrown read whether it is in the steady-state or shutting-down arm, and the conservative non-dispatch behaviour is arm (b); at the **`session_shutdown` handler-entry step (I)** call site the catch arm MUST treat the read as if it had returned the steady-state tuple `(drained: false, tag: undefined)` — equivalently, as if the predicate had **not** fired — so the handler proceeds into steps (III)–(VI) and runs the full five-sub-step teardown sequence as a fail-safe, rather than short-circuiting and silently stranding every resource the handler is responsible for releasing (chokidar `discoveryWatcher`/`settingsWatcher` handles, the debounce-handle `Clock.clearTimeout(...)` call, the forwarding listeners sub-step 5 detaches, the in-flight `loomAbort.abort()`s sub-step 2 fires, and the `disposeBarrier` awaits sub-step 3 owns) on a first-entry `session_shutdown`. This fail-safe is **asymmetric** with the slash-command call site's arm-(b) short-circuit: at the slash-command site the non-dispatch note is the conservative behaviour on a thrown read, but at the handler-entry site short-circuiting on a first-entry read failure would leak every teardown resource above silently (the rule explicitly forbids any diagnostic from this catch arm), so the spec routes the handler-entry catch arm towards executing the teardown rather than skipping it. The trade-off is that on a re-entry the **idempotent** clause's predicate would otherwise have caught, the read-failure conceals the prior-shutdown signal and the full sub-step sequence re-runs against an already-drained registry — additional teardown-handler diagnostic rows are emitted, breaching the at-most-once-per-extension-instance bound for this read-failure case on the same loom 1.0-acceptance footing as the *both-sub-step-1-acts-throw corner case* above; the runtime MUST NOT add a suppression mechanism (a sentinel flag, an in-handler dedup cache, or any other gate) that would conditionally swallow the second-and-later emissions on the read-failure re-entry. No dedicated `readDrainState`-throw diagnostic is emitted from either catch arm in loom 1.0 — the slash-handler symptom (every subsequent dispatch returning the shutting-down note rather than dispatching) is operator-observable in-band, and the handler-entry fail-safe surfaces sub-step-internal throws through the existing per-step-isolation `loom/host/session-shutdown-teardown-step-failed` channel; a future edit MAY add a dedicated diagnostic for this read-failure path but requires a parallel edit to [Diagnostics](../diagnostics.md) and is out of scope for loom 1.0.

   The three-method call surface above is the source of truth for the method names referenced from sub-step 1, the **Per-step isolation** paragraph, and the [**idempotent**](#idempotent-handler-clause) clause; an editorial rename of any method MUST sweep all sites in the same edit. *Non-normative editorial convention (cont.).* Consistent with the convention above, loom 1.0 does not add a fourth **drain-state** method to this surface or expose either of the two drain-state fields through a public accessor beyond `readDrainState`. *Scope of the closed surface and the property-write prohibition.* The closed three-method surface above governs the **drain-state** contract only — the two drain-state fields (`drained`, `drainStateTag`) and their reader/writer methods. The underlying entry-table `Map<slashName, parsedLoom>` interface that the slash-command [`handler`](./registration-steps.md#slash-handler-registration) consults to dispatch the loom (per the *Drain-state-gated dispatch* clause's arm (a) above, which routes the dispatch through this entry-table lookup after `readDrainState` returns the steady-state tuple) is unchanged from step 5 below's original `LoomRegistry` description; direct read access via that map's standard read interface (`Map.prototype.get`, `Map.prototype.has`, `Map.prototype.keys`, `Map.prototype.values`, `Map.prototype.entries`, the `for...of` iteration the watcher's rebuild path consumes per step 5 below, and the `Map.prototype.size` accessor the [Diagnostics](../diagnostics.md) row counts read) is permitted on the same footing as before this contract was added. The `no other writer or reader of either field is permitted` clause and the `direct property writes` prohibition are **scoped to the two drain-state fields only** (`drained` and `drainStateTag`); they do not apply to the entry table or to any other `LoomRegistry` member step 5 below establishes.

## Acceptance criteria — `(drained: true, tag: undefined)` → arm (b)

The `readDrainState` arm enumeration ([PIC-29](#pic-29)) decomposes into the fixture obligations below, all of which the implementation MUST satisfy (the leaf owns fixture-shape; branch coverage and the tuple-to-arm mapping are normative). Fixtures key on the closed `details.call` label set the **Per-step isolation** paragraph already pins and introduce no new diagnostic code or wire field.

- **(a) Reachability of `(drained: true, tag: undefined)`.** For each `reason ∈ {"quit","reload"}` (2 fixtures) where sub-step-1 `LoomRegistry.drain` succeeds (`drained = true`) but sub-step-1 `LoomRegistry.initDrainStateTag` throws (caught by **Per-step isolation**; emits exactly one `loom/host/session-shutdown-teardown-step-failed` under `details.call = "loomRegistry.initDrainStateTag"`; leaves `drainStateTag = undefined`): assert the post-handler snapshot tuple is `(drained: true, tag: undefined)`.
- **(b) Arm-(b) routing.** Assert `LoomRegistry.readDrainState` returning `(drained: true, tag: undefined)` routes through arm (b) — the `"loom /<name>: extension shutting down"` system note — per [PIC-29](#pic-29)'s closed two-arm tuple-to-arm map (the arm-(b) tuple set `(false, "shutting-down")`, `(true, "shutting-down")`, `(true, undefined)`). The routing is witnessed at the `readDrainState` return mapping rather than via a live slash dispatch: on `reason ∈ {"quit","reload"}` Pi tears the runtime down before any subsequent slash dispatch arrives.
