# Capability probe

<a id="entry-capability-probe"></a>
**Step 0 — Capability probe (precedes step 1 below).** Before any `pi.registerFlag`, `pi.registerCommand`, `pi.registerTool`, `pi.registerMessageRenderer`, or `pi.on` call — collectively, the **factory-time host-binding calls** — the factory runs a synchronous probe of six host preconditions — enumerated as sub-steps `(a)`–`(f)` below — in the **fixed short-circuit order `(a)` → `(b)` → `(c)+(d)` → `(e)` → `(f)`** (reconciled in full under the *six sub-steps* note after `(f)` below, where `(d)` runs in `(c)`'s short-circuit slot only if `(c)` passes — a `(c)` failure stops the probe before `(d)` — `(e)` is the `typebox` host-shape check, and `(f)` is the trailing subagent-executable-resolution check) and stops at the first failure (short-circuit). This count and order match [PIC-5](#pic-5)'s "six enumerated checks" prohibition below; the opening MUST NOT under-count the probe as ending at `(c)`, which would let a mis-versioned peer `(d)`, a missing callable `Type.Unsafe` `(e)`, or an unresolvable subagent child executable `(f)` pass Step 0 and fail later as an uncaught runtime `TypeError` (or a first-spawn failure) instead of refusing at load time. This paragraph is the **canonical source of truth** for every probe rule, the floor literal, the member-with-kind enumeration, and the factory-probable capability list; [`spec.md` — Host runtime preconditions 1–3](../overview-and-orientation.md#orientation) is informative orientation that forward-links here under [Governance — GOV-30](../governance.md). The JavaScript engine value model is a non-checked invariant by design and is **not** probed; it is owned by [Runtime Value Model — JavaScript engine assumptions](../runtime-value-model.md#javascript-engine-assumptions).

*The four pinned constants the probe consumes — Node floor, `AbortSignal` / `AbortController` member-with-kind list, factory-probable capability list, peer-dep range — live in **one source of truth** inside the extension module that the build-time literal-read assertions also consume.*

**(a) Node floor.** Compare `process.versions.node` against the pinned floor `>=22.19.0` under SemVer ordering, treating prerelease tags as eligible — a Node nightly or release-candidate build (e.g. `"23.0.0-nightly20250101abcdef"`) whose stable equivalent satisfies the floor MUST satisfy the check. The prerelease-eligibility clause is load-bearing and is part of the normative contract; an implementation that rejects qualifying prerelease builds violates this rule. `process.versions.node` is the bare version string (no leading `"v"`); do not pass `process.version`. A `process.versions.node` value that is not a syntactically valid SemVer string routes to `kind: "probe-failed"` (see *Self-failure* below), not `kind: "node-floor"`.

<a id="step-0-a-recommended-recipe"></a>
*Recommended recipe (non-normative).* `semver.satisfies(process.versions.node, ">=22.19.0", { includePrerelease: true })` from the [`semver`](https://www.npmjs.com/package/semver) npm package satisfies this contract; `semver.valid(process.versions.node)` discriminates the `"probe-failed"` route. A future swap to a different SemVer implementation (or a hand-rolled comparator) is permitted as long as the contract above — range membership with prerelease tags eligible, and a syntactic-validity check that routes failures to `"probe-failed"` — is preserved end-to-end.

**(b) `AbortSignal` / `AbortController` shape.** This sub-step is the **canonical source of truth** for the WHATWG `AbortSignal` / `AbortController` member surface the theta runtime depends on; spec.md Host runtime precondition 2 is informative orientation that forward-links here. The probe applies a per-member rule keyed on the member's `kind` (the table below is the source of truth, co-located with the member name in the extension module's pinned-constants block that the build-time surface-inventory assertion also consumes):

| Member                                       | Kind                  | Check                                                          |
|----------------------------------------------|-----------------------|----------------------------------------------------------------|
| `AbortController` (global)                   | `constructor`         | `typeof AbortController === "function"`                        |
| `AbortSignal` (global)                       | `constructor`         | `typeof AbortSignal === "function"`                            |
| `AbortController.prototype.abort`            | `prototype-method`    | `typeof AbortController.prototype.abort === "function"`        |
| `AbortSignal.any`                            | `static-method`       | `typeof AbortSignal.any === "function"`                        |
| `AbortSignal.timeout`                        | `static-method`       | `typeof AbortSignal.timeout === "function"`                    |
| `AbortSignal.prototype.throwIfAborted`       | `prototype-method`    | `typeof AbortSignal.prototype.throwIfAborted === "function"`   |
| `AbortSignal.prototype.addEventListener`     | `prototype-method`    | `typeof AbortSignal.prototype.addEventListener === "function"` |
| `AbortSignal.prototype.aborted`              | `prototype-property`  | `"aborted" in AbortSignal.prototype`                           |
| `AbortSignal.prototype.reason`               | `prototype-property`  | `"reason" in AbortSignal.prototype`                            |

The `prototype-property` rule is `in`-based, not `typeof`-based: `aborted` and `reason` are accessor properties whose getters throw `TypeError: Illegal invocation` when read off the prototype, so a `typeof === "function"` rule would refuse every conformant host. The probe MUST NOT instantiate `new AbortController()` to read instance members — the `in` operator against the prototype answers the question without depending on the constructors the probe is itself checking.

On an `abortsignal-shape` failure the `theta/load/host-incompatible` payload's `details.observed` / `details.required` depend on the failing member's `kind`: for a `constructor`, `static-method`, or `prototype-method` member (the `typeof`-checked members), `details.observed` is the `typeof <path>` string (e.g. `"undefined"`) and `details.required` is the literal `"function"`, mirroring `typebox-shape` in step (e); for a `prototype-property` member (`aborted`, `reason`, checked by `"<name>" in AbortSignal.prototype`), `details.observed` is the literal `"absent"` and `details.required` is the literal `"present"`, naming the absent-property condition rather than the `in`-operator boolean (a bare `false` is not a self-describing payload value). The [`theta/load/host-incompatible` per-`kind` enumeration](../diagnostics/placeholder-rendering-b.md#host-incompatible-observed-required) restates these as the diagnostics-side test contract.

**(c) Factory-probable SDK capabilities.** Check the seven named function members listed below by `typeof <path> === "function"`, **in the listed order — including the within-bullet order of the one bullet that names two members (`pi.setActiveTools` then `pi.getActiveTools`) — stopping at the first member whose check returns false** (the same first-failure short-circuit discipline the other probe steps follow). This fixed iteration order is normative: it makes the failing member deterministic, so a conformance test that removes exactly one of the seven members can predict which member the runtime names in the emitted payload. These are the runtime-detectable members of the **factory-probable subset** of the seven SDK capabilities owned by [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory): items 1, 2, 4, and 6 (four capabilities, seven function members — capability 1 contributes one, capability 2 contributes one, capability 4 contributes three, and capability 6 contributes two). Capability 3 (subagent-mode isolated session) contributes **no** `typeof` member: the former in-process `createAgentSession` pin left the probe under the child-process subagent design, and capability 3 is verified instead by the subagent-executable-resolution check in sub-step `(f)` below. Capability 2's awaiting-completion member `ExtensionCommandContext.waitForIdle` is **not** in this loop and contributes nothing to the count: it is an `ExtensionCommandContext` instance method with no runtime value at extension-factory entry (the interface carries no factory-time namespace binding for `typeof` to read), so a factory-time `typeof` probe of it is not well-defined; it is exercised at slash-handler invocation, audited as a canonical-`ctx` member-access surface per [audit category (3)](./audit-target-categories.md#audit-target-surface-categories), and contracted by [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) item 2. Item 5 (cancellation propagation) is covered by step (b) above (the WHATWG `AbortSignal` / `AbortController` shape probe) for the theta-side controller; its subagent-mode leg — the child cancellation grace signal (closing the child's parent-held stdin pipe, then a bounded grace and process-tree kill; the RFC 0006 `-p` child exposes no RPC abort command) — is not `typeof`-probable and is re-audited editorially at bump time as part of the CLI/stdout wire pins (its child entry point is verified by sub-step (f) below); see [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) item 5 for the inventory-side framing. Item 7 (binder LLM model) is **not** factory-probable because `ctx.modelRegistry` does not exist at extension-factory entry; it is detected at per-theta load time and surfaces as `theta/load/binder-model-unresolved` per [Slash-Command Argument Binding — Strict-capability requirement](../binder/binder-model-and-context.md#strict-capability-requirement), with no `details.kind` discriminator on this diagnostic. The two diagnostic surfaces (`theta/load/host-incompatible` and `theta/load/binder-model-unresolved`) MUST stay disjoint.

  - `pi.registerCommand` (capability 1)
  - `pi.sendUserMessage` (capability 2)
  - `pi.registerTool` (capability 4)
  - `pi.setActiveTools` and `pi.getActiveTools` (capability 4)
  - `pi.registerMessageRenderer` (capability 6)
  - `pi.sendMessage` (capability 6)

On an `sdk-capability-missing` failure the `theta/load/host-incompatible` payload carries `details.observed = <typeof <path>>` (e.g. the literal string `"undefined"`) and `details.required = "function"` (mirroring `typebox-shape` in step (e)), plus the sibling field `details.member = "<path>"` naming the failing member — the first member in the normative listed order above whose `typeof <path> === "function"` check returned false (e.g. `"pi.registerCommand"`, `"pi.registerTool"`, `"pi.setActiveTools"`). `details.member` is a sibling field that refines the existing `"sdk-capability-missing"` kind, mirroring the `details.package` precedent the peer-dep kinds establish in step (d) below; the `kind` discriminator enumeration in **On failure: refusal and diagnostic** below does not grow. The [`theta/load/host-incompatible` per-`kind` enumeration](../diagnostics/placeholder-rendering-b.md#host-incompatible-observed-required) restates the `details.observed` / `details.required` substrings as the diagnostics-side test contract.

**(d) Peer-dep version (lock-step, four packages).** This check is folded into step (c)'s same source-of-truth pinned-constants block but runs after the factory-probable capability checks complete (i.e. it is the last of the four pinned-constants checks). The probe iterates over the fixed array of four lock-step `@earendil-works/*` peer-dep packages in this normative order (the same source-of-truth list order shared with the build-time per-package coverage assertion):

  1. `@earendil-works/pi-coding-agent`
  2. `@earendil-works/pi-agent-core`
  3. `@earendil-works/pi-ai`
  4. `@earendil-works/pi-tui`

  **Deployment-shape precondition.** Step 0 (d) presupposes each of the four `@earendil-works/*` peers is installed as a discrete, on-disk package carrying its own synchronously-readable `package.json` reachable by Node's CommonJS resolver from the theta factory's module — the packaging model the [manifest lock-step](./host-prerequisites.md#pi-sdk-pin-manifest-lock-step) installs. A bundled or vendored build that collapses a peer into the host's own `package.json` (so the peer has no discrete on-disk `package.json` of its own) does not satisfy this precondition and is outside the version-read contract; such a build observably presents as condition (1) or (2) below rather than as a distinct route.

  For each package in turn, read the `version` field from the named peer's own `package.json` — the `package.json` whose `name` field equals the iteration's `<pkg>` — by any `exports`-independent mechanic. A second-source implementation is conformant by reproducing the observable contract below, not the specific resolution algorithm; the recommended mechanic (a `createRequire(import.meta.url).resolve("<pkg>")` + parent-directory walk, with its `ERR_PACKAGE_PATH_NOT_EXPORTED` rationale) is given in the [`#step-0-d-recommended-recipe`](#step-0-d-recommended-recipe) callout below. Compare against the [theta 1.0 Pi-SDK pin range](./host-prerequisites.md#pi-sdk-pin) under tilde-range SemVer membership — the conventional npm tilde semantic, which pins the minor uniformly across all major versions (so `~X.Y.Z` admits `X.Y.Z` through `<X.(Y+1).0`, applied to the pinned range sourced from [`#pi-sdk-pin`](./host-prerequisites.md#pi-sdk-pin)). The check terminates on the first out-of-range or malformed package in this fixed iteration order — the four-package iteration shares step (c)'s short-circuit slot and does not aggregate failures across packages. The read routes to `kind: "peer-dep-malformed-version"` on exactly these four installation-observable conditions: (1) the named peer's `package.json` is unresolvable from the theta factory's module; (2) no resolved `package.json` carries a `name` field equal to the iteration's `<pkg>`; (3) the located, parsed `package.json` object has no own `version` field; or (4) a `version` field that is present but is not a syntactically valid SemVer string. On these conditions the payload carries `details.observed` set per condition: the raw `version` string for condition (4) (a present-but-invalid-SemVer value, rendered host-derived), and the literal `"<unresolvable>"` for conditions (1), (2), and (3) (no readable `version` string was obtained — the package was unresolvable, no matching `package.json` was found, or the located `package.json` carried no own `version` field). It also carries `details.required = <theta 1.0 Pi-SDK pin range>` (sourced from [`#pi-sdk-pin`](./host-prerequisites.md#pi-sdk-pin)) and `details.package = "<scoped-name>"` naming the offending package. An installed `version` that parses but falls outside the pinned range routes to `kind: "peer-dep-out-of-range"` with the same `details.package` field set. The `kind` discriminator enumeration in **On failure: refusal and diagnostic** below does not grow — `details.package` is a sibling field that refines the existing two `kind`s. The four conditions above are *not* `kind: "probe-failed"`; every other throw inside step (d) — for instance a permission error (`EACCES`) on the `package.json` read, a `JSON.parse` failure against an empty or binary file, a parsed top-level value that is not a JSON object, or a hostile `Proxy` / throwing getter on the resolved object — is none of the four conditions above and routes to `kind: "probe-failed"` (see *Self-failure* below; `details.package` carries the offending package name on that route too).

  The fifth Pi-bundled package `typebox` is **not** included in this iteration — it is pinned to the literal `"*"` per the bundled-package convention (see **Host prerequisites — Pi SDK pin**, the `typebox` sub-paragraph) and therefore has no lock-step minor obligation. Its single load-bearing host-shape check (the presence of `Type.Unsafe` as a callable function) is owned by **Step 0 (e)** below, not by this iteration; the lock-step iteration MUST NOT be extended to cover `typebox`.

<a id="step-0-d-recommended-recipe"></a>
*Recommended recipe (non-normative).* `semver.satisfies(installedVersion, <pinned-range>)` and `semver.valid(installedVersion)` from the [`semver`](https://www.npmjs.com/package/semver) npm package satisfy this contract for each iteration step, where `<pinned-range>` is the literal sourced from the pinned-constants block (in turn sourced from the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)). A future swap to a different SemVer implementation is permitted as long as the tilde-range semantic above and the syntactic-validity discriminator that routes failures to `"peer-dep-malformed-version"` are preserved end-to-end.

*Recommended version-read mechanic (non-normative).* The recommended `exports`-independent read resolves the package's main entry via `createRequire(import.meta.url).resolve("<pkg>")` (resolution context is the theta factory's module, since the extension may be installed globally), then walks parent directories from the resolved entry path up to the first `package.json` whose `name` field equals the iteration's `<pkg>` and reads its `version` field. Resolving the package's main entry rather than a `./package.json` subpath avoids `ERR_PACKAGE_PATH_NOT_EXPORTED`, which Node's ESM resolver throws when an `exports` map does not enumerate the requested subpath. Under this mechanic the installation-observable conditions surface as: condition (1) when the main-entry resolution throws `MODULE_NOT_FOUND`; condition (2) when the parent-directory walk reaches the filesystem root without a matching `package.json`. This `createRequire(import.meta.url).resolve("<pkg>")` path-resolution shape is the one CJS-rooted form the inventory-closure audit's family-(4) prohibition does **not** cover — see [*Recognised import/access shapes*](./audit-recognised-shapes.md#audit-recognised-shapes) for the carve-out. A future swap to a different `exports`-independent mechanic is permitted as long as the four installation-observable conditions above are preserved end-to-end.

**(e) `typebox` host-shape.** Check `typeof Type.Unsafe === "function"` against the `Type` namespace imported from the `typebox` peer dependency declared in `package.json`. Position in the short-circuit order: after `(d)`, since `(e)` depends on the `typebox` peer-dep import resolving and is therefore meaningful only once the lock-step `@earendil-works/*` peer-dep iteration in `(d)` has succeeded. Failure routes to `kind: "typebox-shape"` with `details.observed = <typeof Type.Unsafe>` (e.g. the literal string `"undefined"` when the namespace lacks the member) and `details.required = "function"`. The probe MUST NOT itself construct a TypeBox schema (no `Type.Unsafe<unknown>({})` call); the `typeof` check alone is sufficient and preserves the probe-wide invariant that the probe never exercises the member it is checking. `details.observed` carries the `typeof` string rather than the namespace contents to keep the diagnostic payload small. Adding this check surfaces a missing single-symbol Pi-bundled import as `theta/load/host-incompatible` rather than as a runtime-time `TypeError` at the first tool-registration call.

**(f) Subagent-executable resolution.** Run the [executable-resolution ladder](./subagent.md#subagent-executable-resolution) and assert a **runnable child `pi` entry point** — the capability-3 replacement for the retired `createAgentSession` / `ResourceLoader` / `SessionManager.inMemory` `typeof` pins, which are not `typeof`-probable. Position in the short-circuit order: last, after `(e)`. The check is the ladder's rung-1 existence check (`process.argv[1]` names an existing file), falling through to the rung-2 compiled-binary condition (`process.execPath` is not a generic `node` / `bun` runtime); it does **not** spawn a child. If neither rung yields a runnable entry point, the check fails. This is the design's uniform fail-closed posture: a capability that cannot be verified refuses registration loudly rather than degrading silently. Unlike `(a)`–`(e)`, sub-step `(f)`'s failure emits its **own** precise diagnostic — `theta/load/subagent-executable-unresolved` (see [Diagnostics — `theta/load/*`](../diagnostics/code-registry-load.md)) — rather than `theta/load/host-incompatible`, because the failing surface is not an SDK-shape mismatch but an absent runnable executable; the refusal-of-every-subsequent-factory-time-host-binding-call posture is identical (the factory skips all `pi.register*` and `pi.on` calls). The `(f)` check is `filesystem-existence`-only — no spawn, no version handshake with the child (a version handshake would require a spawn, which the probe forbids); a version skew that a `PATH` fallback could introduce is structurally excluded by the ladder's no-`PATH`-fallback rule, not by this probe.

For brevity the probe is described above as six sub-steps `(a)`–`(f)`; `(d)` is part of step `(c)`'s pinned-constants execution group and shares its short-circuit slot, `(e)` is the `typebox` host-shape check, and `(f)` is the trailing subagent-executable-resolution check. The fixed normative order for short-circuit is: `(a)` Node floor, then `(b)` `AbortSignal` shape, then `(c)+(d)` capability list and peer-dep version, then `(e)` `typebox` host-shape, then `(f)` subagent-executable resolution. A simultaneous-failure host (e.g. Node 18 *and* a polyfilled `AbortSignal` missing `AbortSignal.any`) emits the `(a)` failure; the implementation MUST NOT race or aggregate.

**Probe-wide invariants.**

- <a id="pic-3"></a> **PIC-3.** The probe MUST NOT use any member it is itself checking. The kind table above is constructed so this rule holds without instance construction.
- <a id="pic-4"></a> **PIC-4.** The probe is `typeof`- and `in`-only — no arity sniffing, no return-shape sniffing, no construction.
- <a id="pic-5"></a> **PIC-5.** Probes MUST NOT be added beyond the six enumerated checks. The `(f)` subagent-executable-resolution check is a filesystem-existence check, not a `typeof` / `in` member check, and is the sole permitted exception to [PIC-4](#pic-4)'s `typeof`/`in`-only rule; it MUST NOT spawn a child or exercise the resolved executable.
- <a id="pic-6"></a> **PIC-6.** The factory MUST NOT throw. Each check is wrapped in a try/catch (see *Self-failure* below).

**On failure: refusal and diagnostic.**

- (i) The factory skips **every** subsequent factory-time host-binding call — the four `pi.register*` calls **and** every `pi.on` call (the pinned `ExtensionAPI` surface exposes no `pi.subscribe` member — `pi.on` is the sole event-subscription surface, per [Conversation drive — Prompt-mode turn-lifecycle event subscription (PIC-18)](./conversation-drive.md#pic-18)). Steps 1–5 below MUST NOT execute. The refusal is enforced once at the `PiExtensionAPI` adapter layer, not by per-call defensive checks.
- (ii) The factory emits exactly one diagnostic. For sub-steps `(a)`–`(e)` this is a `theta/load/host-incompatible` diagnostic (per [Diagnostics — `theta/load/*`](../diagnostics.md)) with `details: { kind, observed, required, ...optional }` where `kind ∈ { "node-floor", "abortsignal-shape", "sdk-capability-missing", "peer-dep-out-of-range", "peer-dep-malformed-version", "typebox-shape", "probe-failed" }`. For sub-step `(f)` the single diagnostic is `theta/load/subagent-executable-unresolved` instead (the one probe check whose failing surface is an absent runnable executable rather than an SDK-shape mismatch). Both routes go through the **System notes** fallback chain below (`sendSystemNote` → `ctx.ui.notify` → `console.error`), since the renderer itself may be the missing capability.
- (iii) The factory returns normally.

**Self-failure.** If any check throws (including `process.versions` evaluation, `typeof` evaluation against a hostile getter or `Proxy`, `"in"`-operator evaluation against a `null` prototype, `package.json` resolution / read / parse outside the four enumerated step (d) conditions that route to `"peer-dep-malformed-version"`, or a hostile getter / `Proxy` on the imported `typebox` namespace in step (e)), the probe traps the throw, terminates immediately, and emits `theta/load/host-incompatible` with `details.kind = "probe-failed"`, `details.step ∈ { "node-floor", "abortsignal-shape", "sdk-capability-missing", "peer-dep-version", "typebox-shape", "subagent-executable" }` naming which check threw (a throw inside sub-step `(f)` — e.g. a filesystem `stat` that throws `EACCES` while running the rung-1 existence check — routes to `"probe-failed"` with `details.step = "subagent-executable"`, not to `theta/load/subagent-executable-unresolved`, which is reserved for the clean "neither rung yields a runnable entry point" verdict) (the step-(d) label is the neutral `"peer-dep-version"`, not a `kind` value, because a throw during step (d) cannot determine whether the `peer-dep-out-of-range` or `peer-dep-malformed-version` arm would have fired), and `details.cause` carrying the caught throw's [underlying-error string](../diagnostics/placeholder-rendering-b.md#underlying-error-coercion) — coerced from the caught value, so a non-`Error` throw (or one whose `.message` access itself throws) yields a deterministic string rather than `undefined` or a synchronous `TypeError` that would escape the probe and violate PIC-6. When `details.step = "peer-dep-version"`, `details.package = "<scoped-name>"` additionally names which of the four lock-step packages was being checked at the moment of the throw; the deterministic iteration order pinned in (d) above guarantees this assignment is unambiguous. `details.cause` is the coerced underlying string, not the `Error` object — diagnostics are serialisable per the existing diagnostics contract. A `"probe-failed"` event supersedes any in-order `kind` for that check; the `(a)→(b)→(c)+(d)→(e)` short-circuit ordering applies only to comparison verdicts.

**Idempotency.** A `/reload` re-runs the probe with no carried state. Subagent mode is unaffected because it runs only after a session has been created, by which point step 0 has already gated the factory.

<a id="post-probe-sdk-shape-drift"></a>
**Post-probe SDK-shape drift.** The capability probe above runs **exactly once** per factory invocation (a `/reload` is a fresh factory invocation per *Idempotency* above) and is **not** re-run on subsequent SDK calls. Any post-probe interaction with a previously-probed Pi member — including the snapshot-side `pi.setActiveTools` of the swap, the per-invocation `pi.sendUserMessage` call, the per-call `ctx.modelRegistry.getAvailable()` re-resolution of the binder model handle (which [Slash-Command Argument Binding](../binder.md) constructs afresh on every binder call), the subagent child-process spawn and its stdout envelope-consumption interaction past launch, `pi.sendMessage` for system notes, and any `pi.register*` or `pi.on` member touched after factory entry — that **throws synchronously, returns a Promise that rejects, or returns a value of the wrong shape** routes to `theta/runtime/internal-error` per the catch-all row in [Diagnostics](../diagnostics.md), *except* where one of the four dedicated codes already applies: `theta/runtime/active-set-restore-failed` (the **restore-side** `pi.setActiveTools` of the swap, after a successful snapshot-side call; the snapshot-side throw remains on the `internal-error` route and aborts the swap before the wrapped `fn` runs), `theta/runtime/system-note-delivery-failed` (synchronous throw from `pi.sendMessage` for a `theta-system-note`, after the `ctx.ui.notify` fallback has been attempted), `theta/runtime/subagent-dispose-failure` (re-scoped to any subagent-mode teardown-step throw — stdin close or bounded kill — inside the subagent-mode teardown `finally`), `theta/runtime/registry-swap-failed` (the watcher's mid-rebuild throw on a changed `.theta`/`.thetalib` file), and the subagent-mode child-process failure codes (`theta/runtime/subagent-spawn-failed`, `theta/runtime/subagent-child-crashed`, `theta/runtime/subagent-wire-parse-failed`, `theta/runtime/subagent-teardown-timeout`, `theta/runtime/subagent-callable-hash-mismatch`, `theta/runtime/subagent-model-preflight-mismatch`, `theta/runtime/subagent-envelope-parse-failed`, `theta/runtime/subagent-envelope-schema-skew`, `theta/runtime/subagent-exit-without-envelope`, `theta/runtime/subagent-params-validation-failed`) per [Conversation drive — subagent mode](./subagent.md#pic-9). `pi.sendUserMessage` returns `void` per the `ExtensionAPI` surface (the `Promise<void>` shape is exclusive to `AgentSession.sendUserMessage` and `ReplacedSessionContext.sendUserMessage`); the runtime MUST NOT `await` it, so its observable post-probe failure mode is the synchronous-throw arm above. **Silent shape drift** — a previously-probed member that no longer matches its probed shape but does not throw, reject, or return a wrong-shape sentinel the catch-all wrap can observe (e.g. `pi.setActiveTools` becoming a `Promise<void>`-returning function whose rejection the runtime never sees because it does not `await`) — is **undefined behaviour**: the runtime is not required to detect it, and no diagnostic is owed for it. Operator triage of `theta/runtime/internal-error` therefore cannot, in theta 1.0, distinguish a host-drift failure from a theta-runtime defect on the `error.message` / `error.stack` payload alone; both surface through the same `"theta /<name> aborted with internal error: <message>"` system-note template (and the same `Err(InvokeInfraError { kind: "invoke_infra", cause: "internal_error", ... })` envelope on the `invoke` parent surface). A `details.kind` discriminator on `theta/runtime/internal-error` distinguishing host-drift from theta-defect is an additive registry-minor change deferred past theta 1.0 per [Diagnostics](../diagnostics.md)'s registry-change rules; nothing in the theta 1.0 routing contract precludes adding it later without breaking emitted shapes.

If step 0 succeeds, the factory then:
