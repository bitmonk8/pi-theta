# Capability probe

<a id="entry-capability-probe"></a>
**Step 0 — Capability probe (precedes step 1 below).** Before any `pi.registerFlag`, `pi.registerCommand`, `pi.registerTool`, `pi.registerMessageRenderer`, or `pi.on` call — collectively, the **factory-time host-binding calls** — the factory runs a synchronous probe of three host preconditions in the **fixed order (a) → (b) → (c)** below and stops at the first failure (short-circuit). This paragraph is the **canonical source of truth** for every probe rule, the floor literal, the member-with-kind enumeration, and the factory-probable capability list; [`spec.md` — Host runtime preconditions 1–3](../../spec/overview-and-orientation.md#orientation) is informative orientation that forward-links here under [Governance — GOV-12](../governance.md). The JavaScript engine value model is a non-checked invariant by design and is **not** probed; it is owned by [Runtime Value Model — JavaScript engine assumptions](../runtime-value-model.md#javascript-engine-assumptions).

*The four pinned constants the probe consumes — Node floor, `AbortSignal` / `AbortController` member-with-kind list, factory-probable capability list, peer-dep range — live in **one source of truth** inside the extension module that the build-time literal-read assertions also consume.*

**(a) Node floor.** Compare `process.versions.node` against the pinned floor `>=22.19.0` under SemVer ordering, treating prerelease tags as eligible — a Node nightly or release-candidate build (e.g. `"23.0.0-nightly20250101abcdef"`) whose stable equivalent satisfies the floor MUST satisfy the check. The prerelease-eligibility clause is load-bearing and is part of the normative contract; an implementation that rejects qualifying prerelease builds violates this rule. `process.versions.node` is the bare version string (no leading `"v"`); do not pass `process.version`. A `process.versions.node` value that is not a syntactically valid SemVer string routes to `kind: "probe-failed"` (see *Self-failure* below), not `kind: "node-floor"`.

<a id="step-0-a-recommended-recipe"></a>
*Recommended recipe (non-normative).* `semver.satisfies(process.versions.node, ">=22.19.0", { includePrerelease: true })` from the [`semver`](https://www.npmjs.com/package/semver) npm package satisfies this contract; `semver.valid(process.versions.node)` discriminates the `"probe-failed"` route. A future swap to a different SemVer implementation (or a hand-rolled comparator) is permitted as long as the contract above — range membership with prerelease tags eligible, and a syntactic-validity check that routes failures to `"probe-failed"` — is preserved end-to-end.

**(b) `AbortSignal` / `AbortController` shape.** This sub-step is the **canonical source of truth** for the WHATWG `AbortSignal` / `AbortController` member surface the loom runtime depends on; spec.md Host runtime precondition 2 is informative orientation that forward-links here. The probe applies a per-member rule keyed on the member's `kind` (the table below is the source of truth, co-located with the member name in the extension module's pinned-constants block that the build-time surface-inventory assertion also consumes):

| Member                                       | Kind                  | Check                                                          |
|----------------------------------------------|-----------------------|----------------------------------------------------------------|
| `AbortController` (global)                   | `constructor`         | `typeof AbortController === "function"`                        |
| `AbortSignal` (global)                       | `constructor`         | `typeof AbortSignal === "function"`                            |
| `AbortController.prototype.abort`            | `prototype-method`    | `typeof AbortController.prototype.abort === "function"`        |
| `AbortSignal.any`                            | `static-method`       | `typeof AbortSignal.any === "function"`                        |
| `AbortSignal.timeout`                        | `static-method`       | `typeof AbortSignal.timeout === "function"`                    |
| `AbortSignal.prototype.throwIfAborted`       | `prototype-method`    | `typeof AbortSignal.prototype.throwIfAborted === "function"`   |
| `AbortSignal.prototype.addEventListener`     | `prototype-method`    | `typeof AbortSignal.prototype.addEventListener === "function"` |
| `AbortSignal.prototype.aborted`              | `prototype-property`  | `"aborted" in AbortSignal.prototype`                           |
| `AbortSignal.prototype.reason`               | `prototype-property`  | `"reason" in AbortSignal.prototype`                            |

The `prototype-property` rule is `in`-based, not `typeof`-based: `aborted` and `reason` are accessor properties whose getters throw `TypeError: Illegal invocation` when read off the prototype, so a `typeof === "function"` rule would refuse every conformant host. The probe MUST NOT instantiate `new AbortController()` to read instance members — the `in` operator against the prototype answers the question without depending on the constructors the probe is itself checking.

On an `abortsignal-shape` failure the `loom/load/host-incompatible` payload's `details.observed` / `details.required` depend on the failing member's `kind`: for a `constructor`, `static-method`, or `prototype-method` member (the `typeof`-checked members), `details.observed` is the `typeof <path>` string (e.g. `"undefined"`) and `details.required` is the literal `"function"`, mirroring `typebox-shape` in step (e); for a `prototype-property` member (`aborted`, `reason`, checked by `"<name>" in AbortSignal.prototype`), `details.observed` is the literal `"absent"` and `details.required` is the literal `"present"`, naming the absent-property condition rather than the `in`-operator boolean (a bare `false` is not a self-describing payload value). The [`loom/load/host-incompatible` per-`kind` enumeration](../diagnostics/placeholder-rendering-b.md#host-incompatible-observed-required) restates these as the diagnostics-side test contract.

**(c) Factory-probable SDK capabilities.** Check the ten named function members listed below by `typeof <path> === "function"`, **in the listed order — including the within-bullet order of the one bullet that names two members (`pi.setActiveTools` then `pi.getActiveTools`) — stopping at the first member whose check returns false** (the same first-failure short-circuit discipline the other probe steps follow). This fixed iteration order is normative: it makes the failing member deterministic, so a conformance test that removes exactly one of the ten members can predict which member the runtime names in the emitted payload. These are the runtime-detectable members of the **factory-probable subset** of the seven SDK capabilities owned by [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory): items 1, 2, 3, 4, and 6 (five capabilities, ten function members — capability 1 contributes one, capabilities 2, 3, and 6 each contribute two, and capability 4 contributes three). Item 5 (cancellation propagation) is covered jointly by step (b) above (the WHATWG `AbortSignal` / `AbortController` shape probe) and step (c) below (the `AgentSession.prototype.abort` member listed alongside `createAgentSession`); see [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) item 5 for the inventory-side framing. Item 7 (binder LLM model) is **not** factory-probable because `ctx.modelRegistry` does not exist at extension-factory entry; it is detected at per-loom load time and surfaces as `loom/load/binder-model-unresolved` per [Slash-Command Argument Binding — Strict-capability requirement](../binder/binder-model-and-context.md#strict-capability-requirement), with no `details.kind` discriminator on this diagnostic. The two diagnostic surfaces (`loom/load/host-incompatible` and `loom/load/binder-model-unresolved`) MUST stay disjoint.

  - `pi.registerCommand` (capability 1)
  - `pi.sendUserMessage` (capability 2)
  - `ExtensionCommandContext.waitForIdle` (capability 2; checked at slash-handler invocation, not at factory time — the probe verifies only the static export shape on the `ExtensionAPI`-imported namespace)
  - `createAgentSession` (capability 3)
  - `AgentSession.prototype.abort` (capability 3 / capability 5; the cancellation primitive the runtime invokes from the one-shot `loomAbort.signal` listener registered at spawn time — see **Subagent session lifecycle** and **Cancellation source** below. Probed at factory time as `typeof AgentSession.prototype.abort === "function"` against the imported `AgentSession` class on the `@earendil-works/pi-coding-agent` namespace; the import path is load-bearing — `AgentSession` is re-exported by `dist/core/index.d.ts` of `@earendil-works/pi-coding-agent` (at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)) from `./agent-session.js`, and the loom runtime depends on the re-export specifically, not on direct consumption from `@earendil-works/pi-agent-core` (where the underlying class lives), so a Pi minor that drops the re-export surfaces at this probe as `loom/load/host-incompatible` rather than as a silent fallback to a different import path. A missing or renamed `abort` member surfaces through the same diagnostic rather than as a runtime-time `TypeError` at the spawn site)
  - `pi.registerTool` (capability 4)
  - `pi.setActiveTools` and `pi.getActiveTools` (capability 4)
  - `pi.registerMessageRenderer` (capability 6)
  - `pi.sendMessage` (capability 6)

On an `sdk-capability-missing` failure the `loom/load/host-incompatible` payload carries `details.observed = <typeof <path>>` (e.g. the literal string `"undefined"`) and `details.required = "function"` (mirroring `typebox-shape` in step (e)), plus the sibling field `details.member = "<path>"` naming the failing member — the first member in the normative listed order above whose `typeof <path> === "function"` check returned false (e.g. `"createAgentSession"`, `"AgentSession.prototype.abort"`, `"pi.setActiveTools"`). `details.member` is a sibling field that refines the existing `"sdk-capability-missing"` kind, mirroring the `details.package` precedent the peer-dep kinds establish in step (d) below; the `kind` discriminator enumeration in **On failure: refusal and diagnostic** below does not grow. The [`loom/load/host-incompatible` per-`kind` enumeration](../diagnostics/placeholder-rendering-b.md#host-incompatible-observed-required) restates the `details.observed` / `details.required` substrings as the diagnostics-side test contract.

**(d) Peer-dep version (lock-step, four packages).** This check is folded into step (c)'s same source-of-truth pinned-constants block but runs after the factory-probable capability checks complete (i.e. it is the last of the four pinned-constants checks). The probe iterates over the fixed array of four lock-step `@earendil-works/*` peer-dep packages in this normative order (the same source-of-truth list order shared with the build-time per-package coverage assertion):

  1. `@earendil-works/pi-coding-agent`
  2. `@earendil-works/pi-agent-core`
  3. `@earendil-works/pi-ai`
  4. `@earendil-works/pi-tui`

  For each package in turn, read the installed version from its `package.json` by an `exports`-independent mechanic: resolve the package's main entry via `createRequire(import.meta.url).resolve("<pkg>")` (resolution context is the loom factory's module, since the extension may be installed globally), then walk parent directories from the resolved entry path up to the first `package.json` whose `name` field equals the iteration's `<pkg>` and read its `version` field. Resolving the package's main entry rather than a `./package.json` subpath avoids `ERR_PACKAGE_PATH_NOT_EXPORTED`, which Node's ESM resolver throws when an `exports` map does not enumerate the requested subpath. Compare against the [loom 1.0 Pi-SDK pin range](./host-prerequisites.md#pi-sdk-pin) under tilde-range SemVer membership — the conventional npm tilde semantic, which pins the minor uniformly across all major versions (so `~X.Y.Z` admits `X.Y.Z` through `<X.(Y+1).0`, applied to the pinned range sourced from [`#pi-sdk-pin`](./host-prerequisites.md#pi-sdk-pin)). The check terminates on the first out-of-range or malformed package in this fixed iteration order — the four-package iteration shares step (c)'s short-circuit slot and does not aggregate failures across packages. The read routes to `kind: "peer-dep-malformed-version"` on exactly these four observable conditions, evaluated at the read site: (1) a `MODULE_NOT_FOUND` thrown by the main-entry resolution call; (2) a parent-directory walk that reaches the filesystem root without finding a `package.json` whose `name` field equals the iteration's `<pkg>`; (3) the located, parsed `package.json` object has no own `version` field; or (4) a `version` field that is present but is not a syntactically valid SemVer string. On any of the four, the payload carries `details.observed = <raw-string-or-"<unresolvable>">`, `details.required = <loom 1.0 Pi-SDK pin range>` (sourced from [`#pi-sdk-pin`](./host-prerequisites.md#pi-sdk-pin)), and `details.package = "<scoped-name>"` naming the offending package. An installed `version` that parses but falls outside the pinned range routes to `kind: "peer-dep-out-of-range"` with the same `details.package` field set. The `kind` discriminator enumeration in **On failure: refusal and diagnostic** below does not grow — `details.package` is a sibling field that refines the existing two `kind`s. The four conditions above are *not* `kind: "probe-failed"`; every other throw inside step (d) — for instance a permission error (`EACCES`) on the `package.json` read, a `JSON.parse` failure against an empty or binary file, a parsed top-level value that is not a JSON object, or a hostile `Proxy` / throwing getter on the resolved object — is none of the four conditions above and routes to `kind: "probe-failed"` (see *Self-failure* below; `details.package` carries the offending package name on that route too).

  The fifth Pi-bundled package `typebox` is **not** included in this iteration — it is pinned to the literal `"*"` per the bundled-package convention (see **Host prerequisites — Pi SDK pin**, the `typebox` sub-paragraph) and therefore has no lock-step minor obligation. Its single load-bearing host-shape check (the presence of `Type.Unsafe` as a callable function) is owned by **Step 0 (e)** below, not by this iteration; the lock-step iteration MUST NOT be extended to cover `typebox`.

<a id="step-0-d-recommended-recipe"></a>
*Recommended recipe (non-normative).* `semver.satisfies(installedVersion, <pinned-range>)` and `semver.valid(installedVersion)` from the [`semver`](https://www.npmjs.com/package/semver) npm package satisfy this contract for each iteration step, where `<pinned-range>` is the literal sourced from the pinned-constants block (in turn sourced from the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)). A future swap to a different SemVer implementation is permitted as long as the tilde-range semantic above and the syntactic-validity discriminator that routes failures to `"peer-dep-malformed-version"` are preserved end-to-end.

**(e) `typebox` host-shape.** Check `typeof Type.Unsafe === "function"` against the `Type` namespace imported from the `typebox` peer dependency declared in `package.json`. Position in the short-circuit order: after `(d)`, since `(e)` depends on the `typebox` peer-dep import resolving and is therefore meaningful only once the lock-step `@earendil-works/*` peer-dep iteration in `(d)` has succeeded. Failure routes to `kind: "typebox-shape"` with `details.observed = <typeof Type.Unsafe>` (e.g. the literal string `"undefined"` when the namespace lacks the member) and `details.required = "function"`. The probe MUST NOT itself construct a TypeBox schema (no `Type.Unsafe<unknown>({})` call); the `typeof` check alone is sufficient and preserves the probe-wide invariant that the probe never exercises the member it is checking. `details.observed` carries the `typeof` string rather than the namespace contents to keep the diagnostic payload small. The asymmetry with `AgentSession.prototype.abort` in `(c)` is removed by adding this check: both single-symbol Pi-bundled imports now surface absence as `loom/load/host-incompatible` rather than as a runtime-time `TypeError` at the first tool-registration call.

For brevity the probe is described above as five sub-steps `(a)`–`(e)`; `(d)` is part of step `(c)`'s pinned-constants execution group and shares its short-circuit slot, and `(e)` is the trailing `typebox` host-shape check. The fixed normative order for short-circuit is: `(a)` Node floor, then `(b)` `AbortSignal` shape, then `(c)+(d)` capability list and peer-dep version, then `(e)` `typebox` host-shape. A simultaneous-failure host (e.g. Node 18 *and* a polyfilled `AbortSignal` missing `AbortSignal.any`) emits the `(a)` failure; the implementation MUST NOT race or aggregate.

**Probe-wide invariants.**

- <a id="pic-3"></a> **PIC-3.** The probe MUST NOT use any member it is itself checking. The kind table above is constructed so this rule holds without instance construction.
- <a id="pic-4"></a> **PIC-4.** The probe is `typeof`- and `in`-only — no arity sniffing, no return-shape sniffing, no construction.
- <a id="pic-5"></a> **PIC-5.** Probes MUST NOT be added beyond the five enumerated checks.
- <a id="pic-6"></a> **PIC-6.** The factory MUST NOT throw. Each check is wrapped in a try/catch (see *Self-failure* below).

**On failure: refusal and diagnostic.**

- (i) The factory skips **every** subsequent factory-time host-binding call — the four `pi.register*` calls **and** every `pi.on` call (and every `pi.subscribe` call, if the SDK exposes one). Steps 1–5 below MUST NOT execute. The refusal is enforced once at the `PiExtensionAPI` adapter layer, not by per-call defensive checks.
- (ii) The factory emits exactly one `loom/load/host-incompatible` diagnostic (per [Diagnostics — `loom/load/*`](../diagnostics.md)) with `details: { kind, observed, required, ...optional }` where `kind ∈ { "node-floor", "abortsignal-shape", "sdk-capability-missing", "peer-dep-out-of-range", "peer-dep-malformed-version", "typebox-shape", "probe-failed" }`. Routed through the **System notes** fallback chain below (`sendSystemNote` → `ctx.ui.notify` → `console.error`), since the renderer itself may be the missing capability.
- (iii) The factory returns normally.

**Self-failure.** If any check throws (including `process.versions` evaluation, `typeof` evaluation against a hostile getter or `Proxy`, `"in"`-operator evaluation against a `null` prototype, `package.json` resolution / read / parse outside the four enumerated step (d) conditions that route to `"peer-dep-malformed-version"`, or a hostile getter / `Proxy` on the imported `typebox` namespace in step (e)), the probe traps the throw, terminates immediately, and emits `loom/load/host-incompatible` with `details.kind = "probe-failed"`, `details.step ∈ { "node-floor", "abortsignal-shape", "sdk-capability-missing", "peer-dep-out-of-range", "typebox-shape" }` naming which check threw, and `details.cause = <error.message>` carrying the underlying error text. When `details.step = "peer-dep-out-of-range"`, `details.package = "<scoped-name>"` additionally names which of the four lock-step packages was being checked at the moment of the throw; the deterministic iteration order pinned in (d) above guarantees this assignment is unambiguous. `details.cause` is the JS `Error.message` string, not the `Error` object — diagnostics are serialisable per the existing diagnostics contract. A `"probe-failed"` event supersedes any in-order `kind` for that check; the `(a)→(b)→(c)+(d)→(e)` short-circuit ordering applies only to comparison verdicts.

**Idempotency.** A `/reload` re-runs the probe with no carried state. Subagent mode is unaffected because it runs only after a session has been created, by which point step 0 has already gated the factory.

<a id="post-probe-sdk-shape-drift"></a>
**Post-probe SDK-shape drift.** The capability probe above runs **exactly once** per factory invocation (a `/reload` is a fresh factory invocation per *Idempotency* above) and is **not** re-run on subsequent SDK calls. Any post-probe interaction with a previously-probed Pi member — including the snapshot-side `pi.setActiveTools` of the swap, the per-invocation `pi.sendUserMessage` call, the per-call `ctx.modelRegistry.getAvailable()` re-resolution of the binder model handle (which [Slash-Command Argument Binding](../binder.md) constructs afresh on every binder call), `createAgentSession` and `AgentSession`-handle members past creation, `pi.sendMessage` for system notes, and any `pi.register*` or `pi.on` member touched after factory entry — that **throws synchronously, returns a Promise that rejects, or returns a value of the wrong shape** routes to `loom/runtime/internal-error` per the catch-all row in [Diagnostics](../diagnostics.md), *except* where one of the four dedicated codes already applies: `loom/runtime/active-set-restore-failed` (the **restore-side** `pi.setActiveTools` of the swap, after a successful snapshot-side call; the snapshot-side throw remains on the `internal-error` route and aborts the swap before the wrapped `fn` runs), `loom/runtime/system-note-delivery-failed` (synchronous throw from `pi.sendMessage` for a `loom-system-note`, after the `ctx.ui.notify` fallback has been attempted), `loom/runtime/subagent-dispose-failure` (`AgentSession.dispose()` throw inside the subagent-mode teardown `finally`), and `loom/runtime/registry-swap-failed` (the watcher's mid-rebuild throw on a changed `.loom`/`.warp` file). `pi.sendUserMessage` returns `void` per the `ExtensionAPI` surface (the `Promise<void>` shape is exclusive to `AgentSession.sendUserMessage` and `ReplacedSessionContext.sendUserMessage`); the runtime MUST NOT `await` it, so its observable post-probe failure mode is the synchronous-throw arm above. **Silent shape drift** — a previously-probed member that no longer matches its probed shape but does not throw, reject, or return a wrong-shape sentinel the catch-all wrap can observe (e.g. `pi.setActiveTools` becoming a `Promise<void>`-returning function whose rejection the runtime never sees because it does not `await`) — is **undefined behaviour**: the runtime is not required to detect it, and no diagnostic is owed for it. Operator triage of `loom/runtime/internal-error` therefore cannot, in loom 1.0, distinguish a host-drift failure from a loom-runtime defect on the `error.message` / `error.stack` payload alone; both surface through the same `"loom /<name> aborted with internal error: <message>"` system-note template (and the same `Err(InvokeInfraError { kind: "invoke_infra", cause: "internal_error", ... })` envelope on the `invoke` parent surface). A `details.kind` discriminator on `loom/runtime/internal-error` distinguishing host-drift from loom-defect is an additive registry-minor change deferred past loom 1.0 per [Diagnostics](../diagnostics.md)'s registry-change rules; nothing in the loom 1.0 routing contract precludes adding it later without breaking emitted shapes.
