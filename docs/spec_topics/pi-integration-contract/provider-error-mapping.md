# Provider error and seed-field mapping

<a id="provider-error-mapping"></a>

**Provider error mapping.** The runtime maps recognised provider error responses to `QueryError` variants per the overflow-signature list below. Every classifier-reaching response that does not match an overflow signature maps to `TransportError` regardless of HTTP status — every 4xx/5xx response, every HTTP-200 response carrying a non-overflow body-envelope error (an HTTP-200 response resolving with `AssistantMessage.stopReason: "error"` whose `errorMessage` does not match the overflow signature in the openai-completions entry below), every network-level failure whose `errorMessage` does not match an overflow signature, and every other captured status including a non-200 2xx (201/202/204) or a surfaced 3xx (the overflow-signature precedence is pinned in the [classifier input surface](#classifier-input-surface) note below) — except an HTTP-200 response the [stop-reason classification](#stop-reason-classification) rule below routes to `ContextOverflowError`. The matching rules are version-coupled to `@earendil-works/pi-ai` and MUST be re-validated on each upgrade. *Re-validation gate (theta 1.0.0).* theta 1.0.0 ships without a CI-wired bump-procedure step that re-runs the provider-error fixtures against a candidate `@earendil-works/pi-ai` minor; the fixtures exist in `npm test` and a contributor performing the bump SHOULD run them against the candidate `@earendil-works/pi-ai` minor per item (i) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below, which does not yet enumerate the run as a mechanical step of that procedure. Wiring this re-validation into the bump procedure as a mechanical gate is a recognised post-theta 1.0.0 maintenance follow-up.

<a id="classifier-input-surface"></a>*Classifier input surface.* The overflow-signature list below and the **`TransportError.retryable` population** rule below are written in terms of an HTTP status class, the provider error-body wording, and a separate no-HTTP-response class; theta obtains each from a fixed `@earendil-works/pi-ai` surface rather than from any single typed error object the SDK does not expose. The HTTP-status class (the list's `HTTP 400` / `HTTP 200` references and the retryable rule's `5xx` / `429` / other-`4xx` split) is read from `ProviderResponse.status`, delivered through `@earendil-works/pi-ai`'s `StreamOptions.onResponse` callback (`ProviderResponse = { status: number; headers: Record<string, string> }`), declared at `dist/types.d.ts` in `@earendil-works/pi-ai` (the same declaration [Binder inference call](./binder-inference.md#binder-inference-call) cites for `StreamOptions` and `AssistantMessage`); theta redefines neither `StreamOptions.onResponse` nor `ProviderResponse` and reads `status` as the only HTTP-class signal. The provider error-body wording the overflow signatures match (the `error.message` text, the OpenAI `context_length_exceeded` code where the formatter surfaces it as a substring, and the `ValidationException` body, named in the *Provider-owned-wording presupposition* below) reaches theta only as the `AssistantMessage.errorMessage` string produced by pi-ai's per-provider error formatter — pi-ai surfaces no parsed JSON error body — so every body-wording match in the list is a match against that string. The no-HTTP-response (network-level) class is the case where `onResponse` did not fire before `complete()` resolved with `AssistantMessage.stopReason: "error"`; it routes through `TransportError { retryable: true, http_status: null }` per the retryable rule below. To drive the classifier the runtime registers `onResponse` on every `complete()` call (see the binder's options enumeration in [Binder inference call](./binder-inference.md#binder-inference-call)) and joins the captured `{ status, headers }` with the resolved `AssistantMessage`, keyed per `complete()` invocation; when `onResponse` fires more than once before `complete()` resolves (an SDK-internal retry), the classifier joins the `ProviderResponse` captured by the last firing before resolution. Whether a given provider's pi-ai adapter invokes `onResponse` before resolving is a behavioural property of `@earendil-works/pi-ai` outside its typed surface: an SDK-only provider (`amazon-bedrock` in particular) that resolves with `stopReason: "error"` without an `onResponse` invocation is classified as network-level by the rule above **unless** its `AssistantMessage.errorMessage` matches that provider's overflow signature in the list below, in which case the overflow-signature match takes precedence and the response classifies as `ContextOverflowError` rather than `TransportError`; this presupposition routes to editorial review per item (af) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below.

<a id="provider-overflow-wording-presupposition"></a>*Provider-owned-wording presupposition.* The overflow signatures in the list below match against **provider-owned** HTTP error-body wording, surfaced to theta as the pi-ai-formatted `AssistantMessage.errorMessage` string (built from the provider's `error.message` text, the OpenAI `context_length_exceeded` code where the formatter surfaces it as a substring, and the `ValidationException` body). That wording is not part of `@earendil-works/pi-ai`'s typed surface and is not gated by pi-ai's package version: a provider can reword an overflow message — or move it to a different field — with no `@earendil-works/pi-ai` version change, so neither the SDK surface-inventory test nor the pi-ai version pin can detect the drift, and a silent rewording downgrades a real context-overflow from `ContextOverflowError` to `TransportError` with `tokens_used` / `tokens_limit` null. Detection therefore routes to editorial review: the provider-error fixtures (run under `npm test` per the *Re-validation gate* above) SHOULD be re-run on each Pi minor bump and whenever a provider publishes an error-format or API-version change, per item (i) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. See *Re-validation gate (theta 1.0.0)* above for the corresponding post-theta 1.0.0 maintenance follow-up. *Authoritative source.* The canonical overflow wording is owned by each provider's published HTTP API error-format documentation — it has no in-tree Pi or `@earendil-works/pi-ai` source — so the per-bump diff-target is the maintained corpus of captured provider error bodies the `npm test` overflow fixtures match against, re-captured per item (i) whenever a provider publishes an error-format or API-version change.

<a id="transport-error-retryable"></a>

**`TransportError.retryable` population.** The runtime populates `TransportError.retryable` by transport-error class at the point it constructs the variant: `true` for network-level failures (no HTTP response — TCP/TLS errors, provider-SDK timeouts, end-of-stream truncation, and any non-cancellation `complete()` rejection that carries no `AssistantMessage` — a DNS failure, a mid-stream truncation surfaced as a rejection, or a pi-ai-adapter throw on a malformed response — each routed through `TransportError { retryable: true, http_status: null }`; the `signal`-driven cancellation route at [`complete()` retry and cancellation presupposition](./conversation-drive.md#complete-retry-and-cancellation-presupposition) is unchanged), HTTP 5xx, and HTTP 429; `false` for every other captured status not assigned `true` above — every non-429 4xx, an HTTP-200 response carrying a non-overflow body-envelope error, and any other status reaching the classifier including a non-200 2xx (201/202/204) or a surfaced 3xx (the application-level provider error is a definite refusal returned despite a success status, not a transient condition). The unsupported-provider typed-query case described under **Provider compatibility for typed queries** above is the one path that pins `retryable: false` independent of HTTP class (the failure is a load-time capability gap, not a provider response); that pinned value is unchanged by this rule.

**Overflow signatures (per provider).** Each entry below maps a matching provider response to `ContextOverflowError`. The patterns are JS regular-expression literals in which every `|` is regex alternation; they are stated as a list rather than a GFM table so no Markdown table-cell pipe-escaping intervenes between the source text and its JS-regex reading — a backslash-escaped `\|` MUST NOT appear in any signature.

- `anthropic-messages` — HTTP 400 with `errorMessage` matching `/(prompt is too long|exceeds .* context window|maximum context length)/i`; `tokens_used` and `tokens_limit` extracted from `errorMessage` per *Overflow token-count extraction* below.
- `openai-completions` — HTTP 400 with `errorMessage` matching `/maximum context length|context_length_exceeded/i` (or HTTP 200 resolving with `stopReason: "error"` and `errorMessage` matching the same regex); `tokens_used` and `tokens_limit` extracted from `errorMessage` per *Overflow token-count extraction* below.
- `mistral` — HTTP 400 with body matching `/context.*length/i`; token counts not surfaced — both fields `null`.
- `amazon-bedrock` — `ValidationException` with body matching `/(input is too long|context window)/i`; token counts not surfaced — both fields `null`.

<a id="overflow-token-extraction"></a>

*Overflow token-count extraction.* For the `anthropic-messages` and `openai-completions` entries above, `tokens_used` and `tokens_limit` are derived from `errorMessage` by a single deterministic rule, so two conforming implementations produce identical values on the same payload. Scan the message for *numeric runs*. A *numeric run* is a maximal match of the regular expression `[0-9]+(?:[,_][0-9]+)*` against the message string; the scan takes all non-overlapping leftmost-longest matches in source order, so two matches separated by any non-matching character (including a stray separator) are distinct runs. Equivalently, a numeric run begins and ends with a decimal digit and may contain `,` or `_` separators, each flanked by a decimal digit on both sides; a separator that is not flanked by digits on both sides — a leading separator, a trailing separator, or either separator of an adjacent-separator pair — does not extend the run and is not consumed by it. After extraction, every `,` and `_` in the matched run is stripped and the remainder is parsed as a base-10 integer; for example `"requested 1,234,567 tokens, limit 200,000"` yields the two runs `1234567` and `200000`, and `"1,,234"` yields the two runs `1` and `234`. The scan result then determines the field values:

- When the scan yields **exactly two** numeric runs, the larger integer populates `tokens_used` and the smaller populates `tokens_limit` (a context-overflow response has `tokens_used ≥ tokens_limit` by construction; when the two runs are equal, both fields take that value).
- For **any other count** — zero, one, or three or more numeric runs — both `tokens_used` and `tokens_limit` are `null`.

The `null` fallback is the same value the `mistral` and `amazon-bedrock` entries fix unconditionally. This rule constrains theta's extraction only, not the provider's message text; a provider rewording that changes the numeric-run count silently moves the affected response into the `null` fallback, which the *Provider-owned-wording presupposition* above already routes to editorial review.

<a id="stop-reason-classification"></a>

**Stop-reason classification.** A successful HTTP-200 provider response can terminate on a stop reason other than the `end_turn` / `stop` / `tool_use` reasons the tool loop treats as a normal turn boundary (see [Typed queries are tool-loop-shaped](../query/query-tool-loop.md#typed-queries-are-tool-loop-shaped)). Such a terminator rides a normal 200 with no error envelope, so it matches none of the overflow signatures above; the runtime classifies it by the resolved `AssistantMessage.stopReason` value (the same pi-ai surface the *Classifier input surface* note above reads), not by HTTP status or body wording:

- An output-boundary terminator — pi-ai surfaces output truncated at the model's length boundary as `stopReason: "length"` — maps to `ContextOverflowError`, with `tokens_used` and `tokens_limit` both `null` (no error message is present to extract counts from; this is the same documented `null` condition the `mistral` / `amazon-bedrock` entries fix unconditionally). This is the clean counterpart of the mid-stream output-boundary rule in [Detection of `ContextOverflowError`](../query/query-failure-and-repair.md#detection-of-contextoverflowerror).
- Any other stop reason — that is, any reason other than the turn-boundary reasons (`end_turn` / `stop` / `tool_use`) and the output-boundary `length` handled above — including a content-filter terminator, or any stop reason the runtime does not otherwise recognise — maps to `TransportError` with `retryable: false` (the terminator is a definite outcome of the request, not a transient condition — the same rationale the **`TransportError.retryable` population** rule above applies to an HTTP-200 body-envelope error).

<a id="subagent-queryerror-audit"></a>

**Subagent-path `QueryError` audit.** Under the RFC 0006 child-process theta design the whole subagent callee — interpreter and all its queries — runs inside the child, so **the provider-error mapping on this page runs in the child**, exactly as it does for a prompt-mode theta: the child classifies its own provider errors, encodes the resulting `QueryError` into the `err` arm of the `theta_result` return envelope ([PIC-59](./subagent.md#pic-59)), and the parent reconstructs it from the envelope. The parent-side contract reduces to **envelope consumption**; it performs no per-query classification. The [error-fidelity](./subagent.md#subagent-error-fidelity) obligation requires this pinned audit: every `QueryError` variant the child can surface is representable in the pinned envelope schema, and expected fidelity loss is **zero** — the child runs the identical query and classification machinery the in-process path ran.

| Reachable `QueryError` / envelope outcome | Where classified | Envelope encoding / owner |
|---|---|---|
| `kind: "cancelled"` | in the child (the child's own driver observes its `thetaAbort`); the parent's cancellation closes the child's stdin + kills ([PIC-63](./subagent.md#pic-63)) | envelope `err` arm, or the fail-closed no-envelope path when the kill precedes emission; synthesised per [Cancellation](../cancellation.md) |
| `kind: "transport"` (provider transport failure) | in the child — the child's prompt-mode driver reads the trailing `assistant` `stopReason: "error"` and applies this page's mapping | envelope `err` arm carrying `{ message: <errorMessage \| "provider transport failure">, http_status: null, retryable: false }` |
| `kind: "context_overflow"` | in the child — [Stop-reason classification](#stop-reason-classification) / [Overflow signatures](#provider-error-mapping) applied in the child | envelope `err` arm |
| `kind: "validation"` (typed-query schema validation) | in the child — the forced-respond turn runs off-session **in the child** via pi-ai `complete()`, and its provider errors classify through this table in the child | envelope `err` arm; owned by [Query](../query.md) |
| `kind: "tool_loop_exhausted"` | in the child — untyped-query round cap reached across the child's free-phase turns | envelope `err` arm; owned by [Query — Tool-call loop bound](../query/query-tool-loop.md#tool-call-loop-bound) |
| **child exit without an envelope (crash / kill / timeout)** | parent-side, fail-closed | maps to `Err(InvokeInfraError { cause: "internal_error", ... })` with the exit detail — never a fabricated value; `theta/runtime/subagent-exit-without-envelope` (and `theta/runtime/subagent-child-crashed` for crash detail) emitted per [PIC-59](./subagent.md#pic-59) |
| **envelope corruption / schema skew** | parent-side, fail-closed | maps to `Err(InvokeInfraError { cause: "internal_error", ... })`; `theta/runtime/subagent-envelope-parse-failed` / `theta/runtime/subagent-envelope-schema-skew` emitted per [PIC-59](./subagent.md#pic-59) |

The two parent-side fail-closed rows are the only genuinely new failure classes; both map to `InvokeInfraError { cause: "internal_error" }` rather than a fabricated value. Extending the envelope (richer child→parent state) is held in reserve only if this audit exposes a gap; degraded fidelity without the audit is rejected. The per-bump re-validation of the `AssistantMessage` `stopReason` / `errorMessage` fields the child's driver reads is item (m) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure).

<a id="provider-seed-field-mapping"></a>

**Provider seed-field mapping.** Whether a binder request payload carries a fixed seed, and under which JSON field name, is governed by the static table below. The mapping is keyed on the resolved binder model's `api` field as reported by `@earendil-works/pi-ai`'s model registry; it is not derived from any pi-ai capability flag. The seed *value* (its derivation from the theta's qualified name) is specified in [Slash-Command Argument Binding — Determinism](../binder/determinism-cancellation-failure.md#determinism). Widening the seed-supporting set is a spec-versioned change. The mapping is version-coupled to `@earendil-works/pi-ai` and MUST be re-validated on each upgrade; the mechanical gate is the build-time `Api`-coverage assertion which enumerates pi-ai's exposed `Api` literal-union values and asserts every value appears as a row key in the seed-field table constant, plus the provider seed-field fixtures rerun as step 6 of [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. A new pi-ai `Api` value lights up the assertion red at the bump commit, exactly parallel to a new SDK capability.

| Provider | Seed field in request payload |
|---|---|
| `openai-completions` | `seed` |
| `mistral` | `random_seed` |
| `anthropic-messages` | omitted |
| `amazon-bedrock` | omitted |
