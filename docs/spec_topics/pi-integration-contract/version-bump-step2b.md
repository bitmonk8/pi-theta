# Version bump step2b

    **(b) Negative direction — inventory-closure audit over `src/`.** The [inventory-closure audit](./inventory-audit-intro.md#sdk-cap-inventory-closure-audit) above runs against the source tree and emits one failure record per audit violation (per the audit's per-discriminator routability obligation pinned there). A red here means either (ia) an `src/` reference resolves to neither an `SDK_SURFACE_INVENTORY` entry nor a declared `// allow-pi-surface:` exemption (families (1)–(3) of the *Failure-surface contract* above), or (ib) an `src/` reference uses an out-of-scope import/access shape (family (4) — non-exemptible per *Exemption mechanism*; routed through branch (5) [rewrite-shape](#bump-step-2b-rewrite-shape) rather than branch (3) [exempt](#bump-step-2b-exempt)), or (ii) a declared `// allow-pi-surface:` exemption marker is stale (per *Stale-marker discriminator*) or malformed (per *Malformed-marker discriminator*) — family (5). Resolve by exactly one of the following five branches, selected by the per-branch criterion below — the branches are peer alternatives and no normative preference order applies between them, but the criterion is normative (a contributor MUST pick the branch whose criterion the offending violation satisfies, and reviewers MUST check that fit on the bump-commit diff). The criteria for branches (1), (2), and (3) overlap on their faces, so the following tie-breaks are normative and resolve every plausible (1)∧(3) and (2)∧(3) overlap case among them: **if branches (2) and (3) both plausibly apply, pick branch (2) (promote)** — branch (3) (exempt) is reserved for the strict subset of cases where the offending reference resolves to exactly one callsite in the audited source tree (a single `<path>:<line>:<symbol>` triple, mechanically checkable on the audited source tree at bump time) *and* the exemption citation identifies the specific spec section that authorises this single use; **if branches (1) and (3) both plausibly apply, pick branch (1) (delete)** — branch (3) is for references the spec authorises (and the citation MUST resolve to such an authorising section), and branch (1) is the default for every unauthorised addition. A reference that plausibly satisfies both branch (1) (an inadvertent addition the runtime does not depend on) and branch (2) (a legitimate persistent surface dependency the inventory had not enumerated) — the (1)∧(2) overlap — is not resolved by a fixed tie-break here; it is a reviewer-discretion call on the bump-commit diff, because the choice between deleting and promoting turns on the substantive question of whether the runtime depends on the surface: the contributor records the call and a reviewer confirms it or pushes back on the diff. A reference whose authorising spec section a reviewer cannot identify on the bump-commit diff is by construction not a branch (3) candidate; this is the mechanical reviewer-disagreement failure mode the surrounding step pins (a post-merge dispute over branch (2) vs (3) or branch (1) vs (3) is resolved by re-applying the tie-break against the offending bump-commit diff and routing any necessary correction through the next bump's step 2(b) pass). The family→branch routing table the implementation publishes per the *Failure-surface contract* above is the byte-comparable expression of which subset of these five branches each `<family>` token routes to:
    1. <a id="bump-step-2b-delete"></a>**Delete the unauthorised reference** — when the reference was added inadvertently (e.g. accidental `pi.<member>` access, copy-paste from an unrelated path) and the runtime does not depend on it.
    2. <a id="bump-step-2b-promote"></a>**Promote the surface to an inventory entry** — when the reference is a legitimate, persistent SDK surface dependency the runtime depends on (i.e. the audit caught a real surface-set-closure miss the inventory had not yet enumerated); co-edit `SDK_SURFACE_INVENTORY` and, where the new entry is a capability, the [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) section above. In the same edit the contributor MUST update the integer literal `7` in step 2(a)'s count-equality assertion together with the `CAPABILITY_OBLIGATIONS` constant and the corresponding markdown capability-obligation enumeration item (items 1–7) in [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) above. Step 2(a)'s count-equality assertion is the mechanical gate over the `CAPABILITY_OBLIGATIONS` constant against the integer literal `7`, so updating the constant without updating the integer literal in the same edit fails that gate; drift between the markdown items-1–7 enumeration and `CAPABILITY_OBLIGATIONS` is contributor-discipline backed by that same assertion against the constant. The natural-language cardinality restatements that name the capability count elsewhere in the corpus (the `spec.md` orientation prose, the PIC-side preamble, and the other prose phrasings of the count) are NOT a mechanically-verifiable co-edit obligation, and branch (2) does not assert one: per [Governance — GOV-31](../governance/req-id-prefix-table-active-b.md#gov-31), the SDK-capabilities count is an in-code-constant carve-out aggregator whose value is witnessed by `CAPABILITY_OBLIGATIONS.length === 7`, and its prose restatements are maintained by editorial convention — a reviewer SHOULD flag a stale cardinality phrasing on the bump-commit diff. GOV-31 makes this constant-vs-prose drift gap intentional; branch (2) keeps the bump-time prose update under that editorial-convention treatment rather than reopening it as an open-ended sweep over an uncommitted set of sites. If the surface to be promoted is a typebox named import or `Type.<member>` member-access (i.e. a family (2) violation whose carrier is the `typebox` package per the carve-out under [Target surface categories](./audit-recognised-shapes.md#audit-target-surface-categories) above — for example a non-`Type` named import such as `Static` or `TSchema`, or a non-allow-listed `Type.<member>` access such as `Type.Object`), the "inventory entry" added is an entry in the corresponding typebox allow-list (named-import allow-list `{ Type, ... }` or member-access allow-list `{ Unsafe, ... }`) rather than in `SDK_SURFACE_INVENTORY`; the same co-edit obligations apply (re-pin [step 5](./version-bump-intro.md#pi-version-bump-procedure) grouping (iii); the `seven` capability cardinality is unaffected since typebox allow-lists are not capabilities). This co-edit obligation applies symmetrically to the capability-removal case (a Pi-side capability removed at the candidate version, e.g. 7 → 6): in the same edit the contributor MUST shrink `CAPABILITY_OBLIGATIONS` to match, update the integer literal in step 2(a)'s assertion downward, and remove the corresponding capability-obligation enumeration item; the natural-language cardinality restatements follow the GOV-31 editorial-convention treatment above on the same terms as the promote direction. Branch (2)'s 'promote' framing covers both directions.
    3. <a id="bump-step-2b-exempt"></a>**Add a declared exemption** — when the reference is an intentional, single-callsite use the spec authorises but does not warrant a permanent inventory entry (e.g. a one-off bridge or a deliberately scoped escape hatch); append a same-line `// allow-pi-surface: <REQ-ID-or-PIC-anchor> — <justification>` marker (here `PIC` abbreviates *Pi Integration Contract*) per the audit's *Exemption mechanism* grammar above, with the citation naming the spec section that authorises the unenumerated reference.
    4. <a id="bump-step-2b-stale-rewrite"></a>**Delete or rewrite the stale or malformed exemption marker** — when the audit reports a stale or malformed exemption marker (family (5)): if the marker's originating line carries no in-scope surface reference at audit time (whether because the underlying surface reference was removed or because the marker was placed on a line that never held a recognised in-scope reference — these are the two sub-cases the *Stale sub-kinds* paragraph above collapses under the single (s1) `<symptom>` token), delete the marker line; if the surface reference now resolves to an `SDK_SURFACE_INVENTORY` entry, delete the marker (the inventory entry now satisfies the audit on its own); if the marker still refers to an out-of-inventory surface but its citation has gone stale (e.g. the cited PIC anchor moved or the cited REQ-ID was retired), rewrite the marker's citation to a current REQ-ID or PIC anchor per the *Exemption mechanism* grammar above; if the marker is malformed per the *Exemption mechanism* malformed-marker discriminator (any clause (a)–(g) of the *Malformed-marker discriminator* above — currently: unrecognised separator, missing colon, under-length or punctuation-only justification, off-line placement, missing citation, case-folded `Allow-Pi-Surface`/`ALLOW-PI-SURFACE` spelling, or block-comment `/* … */` shape) AND a genuine exemption is intended, rewrite it to comply with the grammar; if the marker is malformed per the same discriminator AND no exemption was actually intended, delete the marker and resolve the underlying surface via branch 1, 2, or 3 above (the no-exemption-intended carve-out under malformed markers). *Malformed-marker dual-emission co-commit obligation.* When a malformed marker fires under the *Malformed-marker discriminator* above, the audit additionally fires the underlying out-of-inventory reference's own family (1) / (2) / (3) discriminator on the same audit run (per the dual-emission rule pinned there); both reds MUST be resolved in the same commit, resolving only the family (5) red (e.g., by deleting or rewriting the marker) leaves the family (1) / (2) / (3) red live against `main`, and resolving only the family (1) / (2) / (3) red (e.g., by promoting the underlying surface to an `SDK_SURFACE_INVENTORY` entry per branch (2) above without also deleting the now-stale marker) leaves the family (5) red live against `main`. The two records share the same `path` and `line` field values per the *Per-family record-shape table* above and can be paired by that `(path, line)` key for review on the bump-commit diff. The audit reports each red independently on each invocation regardless of whether the other has been resolved — there is no carve-out by which fixing one suppresses the other in a subsequent run.
    5. <a id="bump-step-2b-rewrite-shape"></a>**Rewrite the source line into a recognised shape** — when the audit reports an out-of-scope import/access shape (family (4) of the *Failure-surface contract* above, which the *Exemption mechanism* explicitly excludes from branch 3): rewrite the offending source line into one of the recognised shapes — per-symbol named import, `pi.<member>` static member-access, or canonical-`ctx` member access (per the *Exemption mechanism* family (4) prescription above and *Recognised import/access shapes* above) — and resolve the rewritten reference via branch 1, 2, or 3 above as that branch's criterion fits. An `// allow-pi-surface:` marker on a family (4) line MUST NOT be added (per the *Exemption mechanism* prohibition); branch 3 does not apply to family (4). The contributor MUST re-run the audit on the rewritten line until no family fires; a rewrite that introduces a new family (1)/(2)/(3)/(4)/(5) red MUST be resolved in the same commit before the bump merges (parallel to branch (4)'s dual-emission and stale-marker co-commit obligations).
3. **Re-confirm the `engines.node` floor.** The build-time `package.json` `engines.node` literal-read test asserts a three-way equality across (i) the loom `package.json#engines.node` literal, (ii) the in-repo pinned floor recorded in the `pi-engines-node` row of `SDK_SURFACE_INVENTORY`, and (iii) the `engines.node` floor read live at build time from the installed `@earendil-works/pi-coding-agent` `package.json` under `node_modules` at the candidate version. Operand (iii) is the only live read; because it reflects the installed dependency rather than an in-repo literal, a Pi minor bump that moves the upstream floor changes (iii) while operands (i) and (ii) stay pinned, so the test fails red at the `npm test` invocation rather than at user runtime and no contributor-side manual compare is required here. If the test fails red, the upstream floor has moved; update the loom `package.json#engines.node` literal (operand (i)), [Step 0 (a)](./capability-probe.md#entry-capability-probe) above (the comparator and floor reference), the informative-orientation sentence in [`spec.md` — Host runtime precondition 1](../../spec/overview-and-orientation.md#orientation), and the `pi-engines-node` row's `literal` field in `SDK_SURFACE_INVENTORY` (operand (ii), the in-repo pinned copy of the upstream floor that the assertion compares against the live read (iii)) in the same edit; if the test stays green, the floor has not moved and no edit is needed.
4. **Update the version pin in `peerDependencies` and the equivalent literal here.** Move all four `peerDependencies` entries together (the `peerDependencies` literal-read test, whose assertion shape is stated below, enforces the joint move) and update the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) literal at the top of this page in the same commit (this is the single in-corpus restatement-source per the [Manifest lock-step](./host-prerequisites.md#pi-sdk-pin-manifest-lock-step) sub-paragraph above: every other in-corpus reference to the pin cites the anchor by location). Splitting these into separate commits leaves a window in which that test is red against `main`; do not split. The `peerDependencies` literal-read test compares each of the four `@earendil-works/*` entries in `package.json#peerDependencies` (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`) against the canonical Pi-SDK-pin literal at [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin), using string-equal (byte-equal) comparison, per the [Manifest lock-step](./host-prerequisites.md#pi-sdk-pin-manifest-lock-step) sub-paragraph above; the test passes iff all four entries are byte-equal to that single source-of-truth literal. The `typebox` entry is excluded from this iteration — it is pinned to `"*"` and asserted separately by its own one-line literal-read assertion, per the [Manifest lock-step](./host-prerequisites.md#pi-sdk-pin-manifest-lock-step) carve-out above. The "joint move" property follows from this assertion shape — any one of the four entries diverging from the pin literal fails the test red — rather than being the assertion itself. The deliberate deviation from Pi's bundled-package convention recorded in **Host prerequisites — Pi SDK pin** above (the *Deliberate deviation from Pi's bundled-package convention* sub-paragraph) MUST be re-written at this step whenever the candidate Pi minor changes the lock-step expectation `packages.md` prescribes — namely if `packages.md` at the candidate minor stops naming the `@earendil-works/*` four as `"*"`-range packages, or if Pi changes the `pi-mono` release model so that the four packages are no longer guaranteed to ship together at a single minor. A satisfying re-write MUST (a) name which of those two triggering changes fired and cite its candidate-minor source (the `packages.md` revision, or the `pi-mono` release-model statement), and (b) state whether the deviation rationale stands as written or is invalidated and replaced; a re-write that names neither triggering change, or that only cosmetically rewords the existing rationale without making the (a)/(b) determination, does not satisfy this obligation. In either case, update the *Deliberate deviation* sub-paragraph together with the `peerDependencies` literal-read test, the [Step 0 (d) Peer-dep version](./capability-probe.md#entry-capability-probe) probe, and the literal-anchor list above — all four are updated together whenever this step's `packages.md` / `pi-mono` trigger fires, not conditionally per artefact — in the same commit; if `packages.md` and the `pi-mono` release model are unchanged at the candidate minor, the deviation rationale stands as written and no edit to that sub-paragraph is required.
