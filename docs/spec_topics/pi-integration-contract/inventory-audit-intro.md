## SDK capability inventory

<a id="pic-15"></a> **PIC-15. SDK capability inventory.** The runtime depends on the seven named SDK capabilities enumerated below. Items 1, 2, 3, 4, and 6 are factory-probable and verified by [Step 0 (Capability probe)](./capability-probe.md#entry-capability-probe) above (which on any failure skips every factory-time host-binding call — the four `pi.register*` calls and `pi.on` — and emits `loom/load/host-incompatible`).

The *surface-set closure* property cross-referenced from [`spec.md` — Orientation — Prerequisites](../../spec/overview-and-orientation.md#orientation) — every Pi-side surface reference in the *audited source tree* (the canonical scope name pinned under *Audit scope* below; equal to the closed set `src/**/*.ts` minus the excluded files enumerated there) resolves to an `SDK_SURFACE_INVENTORY` entry, the typebox sibling allow-lists (`{ Type }` named-import allow-list and `{ Unsafe }` member-access allow-list, per [Target surface categories (typebox carve-out)](./audit-recognised-shapes.md#audit-target-surface-categories) below), or a declared `// allow-pi-surface:` exemption — is an obligation the implementation MUST satisfy via the *inventory-closure audit* pinned immediately below. `SDK_SURFACE_INVENTORY` is strictly broader than the seven capabilities; its entry-kind taxonomy (the `namespace-function` kind that stores the nine function members of the factory-probable subset of capabilities 1, 2, 3, 4, and 6, plus the additional non-`namespace-function` kinds) is implementation-owned and not re-enumerated here.

<a id="non-capability-pi-members"></a>
**Non-capability `pi.<member>` surfaces (`pi.registerFlag` / `pi.getFlag`).** `pi.registerFlag` and `pi.getFlag` are declared on the `ExtensionAPI` interface at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)). They are **not** among the seven capabilities enumerated above and do not extend the `CAPABILITY_OBLIGATIONS` count: `pi.registerFlag` is one of the factory-time host-binding calls the [Step 0 capability probe](./capability-probe.md#entry-capability-probe) above gates (the probe skips it on any failure) but is not itself one of the nine `typeof`-probed members in [Step 0 (c)](./capability-probe.md#entry-capability-probe), and `pi.getFlag` is read inside the discovery walk after the probe has gated the factory. Both are `pi.<member>` surfaces the runtime touches and resolve through `SDK_SURFACE_INVENTORY` as audit category-(1) entries per [Target surface categories](./audit-recognised-shapes.md#audit-target-surface-categories) below, not as a declared `// allow-pi-surface:` exemption.

The seven-capability cardinality claim above is a separate assertion anchored by the `CAPABILITY_OBLIGATIONS` constant pinned in the extension module's pinned-constants block; it is enforced by [`CAPABILITY_OBLIGATIONS.length === 7`](./version-bump-step2.md#bump-step-2-positive)'s mechanical assertion against that constant (not against the markdown enumeration of items 1–7 below or the [`spec.md` — Orientation — Prerequisites](../../spec/overview-and-orientation.md#orientation) cross-reference list, both of which are contributor-discipline restatements), and not by the inventory-closure audit (which targets surface-set closure rather than capability count). Contributor-discipline (per [step 2(b) branch (2)](./version-bump-step2b.md#bump-step-2b-promote)'s co-edit obligation, the natural-language-prose arm of which is editorial convention per [Governance — GOV-31](../governance/req-id-prefix-table-active-b.md#gov-31)) carries the markdown-enumeration↔`CAPABILITY_OBLIGATIONS`-constant↔natural-language-prose chain. Describing the audit's behavioural contract here, rather than its concrete tooling, lets the implementation choose a static-AST walker, a TypeScript-program symbol-table walk, or any equivalent tooling that detects every recognised import/access shape per *Recognised import/access shapes* below and surfaces every violation under the family→discriminator obligations per *Failure-surface contract* below, and prevents the surface-set-closure guarantee [`spec.md` — Orientation — Prerequisites](../../spec/overview-and-orientation.md#orientation) leans on from depending on a tooling claim.

<a id="sdk-cap-inventory-closure-audit"></a>
**Inventory-closure audit.** The implementation MUST publish a build-time *inventory-closure audit* — a negative-direction check, sharing the same `SDK_SURFACE_INVENTORY` source of truth as the positive-direction literal-read assertion that owns items 1–7 above, that asserts every Pi-side surface reference in the audited source tree resolves to an `SDK_SURFACE_INVENTORY` entry, the typebox sibling allow-lists (`{ Type }` named-import allow-list and `{ Unsafe }` member-access allow-list, per [Target surface categories (typebox carve-out)](./audit-recognised-shapes.md#audit-target-surface-categories) below), or a declared `// allow-pi-surface:` exemption. The audit's name *inventory-closure audit* refers to the closure mechanism (every reference resolved through the inventory or an exemption); the property it enforces is termed *surface-set closure* in [`spec.md`](../../spec.md) and on this page interchangeably (per the cross-reference at the top of this section). PIC pins the audit's behavioural contract; concrete tooling and literal discriminator strings are implementation-owned, mirroring the ownership carve-out in step 5 of the [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. The commit that introduces this audit MUST also include whatever source-tree sweep is needed to keep `main` green: every existing `src/` reference that the new audit would surface as a violation MUST be either deleted, promoted to an `SDK_SURFACE_INVENTORY` entry, marked with an `// allow-pi-surface:` exemption, or rewritten into a recognised shape (per [step 2(b)](./version-bump-step2.md#bump-step-2-negative)'s five branches below), in the same commit. The audit is intended to land green; it is not introduced as a "land red, fix incrementally" gate, and an implementation whose first `npm test` invocation against `main` is red on this audit violates this paragraph on the same footing as the implementation-adoption obligations above.
