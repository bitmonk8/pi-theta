# Session-only degraded state

> *Implementation gate.* The obligations on this page are blocked pending resolution of the open contradiction recorded at [Pi Integration Contract â€” Host prerequisites for the degraded-state branch, clause (a)](./host-prerequisites.md#degraded-state-host-prerequisites); the open question is tracked on the version-bump editorial-review checklist at [item (a)](./version-bump-step2.md#bump-checklist-instance-survival). The block clears only when item (a)'s [authoritative-determination record](./version-bump-step2.md#bump-step-2-outcome-recording-conventions) resolves to the `provably-differs` verdict.

<a id="session-only-reason-degraded-state"></a> `reason: "new" | "resume" | "fork"` (the three *session-only reasons*: Pi swaps the user session without tearing the extension runtime down) does not always tear down the *extension* runtime â€” only the user session â€” but Pi's `session_shutdown` type doc states the event "is fired before an extension runtime is torn down", and the handler runs the full fixed five-sub-step sequence â€” including watcher closure in sub-step 4 â€” for **every** reason in the closed set. loom 1.0 accepts that watchers may be closed on a reason (e.g. `"fork"`) that did not in fact tear the extension down.

The four Pi SDK behaviours this branch silently relies on â€” extension-instance survival across session-only `session_shutdown`; multi-`session_shutdown` delivery to a single instance; `/reload` / `ctx.reload()` recovery semantics; and the teardown-vs-session-swap `event.reason` partition assignment â€” are enumerated under [**Host prerequisites for the degraded-state branch**](./host-prerequisites.md#degraded-state-host-prerequisites); the editorial-review obligation pinned there catches a Pi minor that violates any of them on a version bump.

After such a closure the extension runtime remains live but is **degraded for the remainder of its lifetime**: sub-step 1 marked the `LoomRegistry` drained and no rule ever un-drains it. The post-sub-step-4 closed `discoveryWatcher` and `settingsWatcher` mean editor-driven `.loom` and settings edits silently no-op.

## Drain-state tag write (state-independent)

Immediately **after sub-step 4 returns and before sub-step 5 runs**, the handler writes the registry's drain-state tag to `"degraded-needs-reload"` **unconditionally**. This is a *state-independent write* (`LoomRegistry.markRuntimeDegraded`, the *Methods* paragraph): it overwrites whatever the tag held â€” `"shutting-down"`, `undefined` from a prior `LoomRegistry.initDrainStateTag` throw, or any unexpected residual â€” **without reading the prior state**, and is the only writer of the second arm of sub-step 1's closed two-arm tag set. Therefore an unset source state (reachable when sub-step 1's `LoomRegistry.initDrainStateTag` threw and was caught by **Per-step isolation**, leaving `drainStateTag = undefined`) is overwritten on the same footing as `"shutting-down"`, and a tag-init throw cannot silently defeat the degraded-arm signal.

- **Non-conformant:** gating this write on observing a `"shutting-down"` source state (so an unset tag would no-op and strand the runtime on the `"shutting-down"` arm forever).

The effect: every subsequent slash invocation short-circuits with the distinct `"loom /<name>: extension degraded; /reload to recover"` system note rather than the misleading `"loom /<name>: extension shutting down"` template (which would falsely suggest a transient teardown and leave an operator on a stdio-capturing host â€” where the `console.error` visibility of `loom/host/session-shutdown-runtime-degraded` may be lost â€” with no in-band signal that `/reload` is required).

## Recovery path and prohibitions

Operator-issued `/reload` (which invokes `ctx.reload()` and re-executes the extension factory) is the **only** loom 1.0 recovery path. On the degraded branch the runtime:

- MUST NOT un-drain the registry;
- MUST NOT re-subscribe or re-establish the closed `discoveryWatcher` / `settingsWatcher`;
- MUST NOT poll for `.loom` or settings changes after watcher closure;
- MUST NOT introduce any other discovery, re-scan, or self-recovery path.

The "edits silently no-op" guarantee is delivered by this exhaustive prohibition, not by the implicit absence of other paths; a future relaxation (e.g. a polling fallback) MUST be routed through a spec edit to this page. Post-shutdown re-emission is already bounded: subsequent `session_shutdown` events against the same instance emit no further `loom/host/session-shutdown-runtime-degraded` row (the row is emitted only between sub-steps 4 and 5, so handler re-entry is the only path that could re-emit it; slash invocations have no `runtime-degraded` emission path), per the at-most-once-per-extension-instance-lifetime bound of the [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause.

## Diagnostic emission â€” `loom/host/session-shutdown-runtime-degraded`

The handler emits exactly one `loom/host/session-shutdown-runtime-degraded` (W, runtime â€” see the [Diagnostic shape â€” column legend](../diagnostics/diagnostic-shape.md#column-legend)) diagnostic via `console.error` **between sub-steps 4 and 5**, on the same handler-scoped seam as the tag write (tag write first; emission immediately after when its predicate holds). The diagnostic carries `details: { event: { reason: "new" | "resume" | "fork" } }`, where `details.event.reason` is the handler-scoped captured variable `capturedEventReason` (the same variable the **Unknown-reason rule** pins and sub-step 2 stamps onto each entry's `shutdownReason`). The row's source is the handler-scoped capture itself, **not** the per-entry `shutdownReason` stamp, so it is independent of any sub-step-2 stamp throw caught by **Per-step isolation**; on this branch it is necessarily a closed-set member.

**Isolation.** Both the tag write and the emission sit outside any sub-step's per-call `try`/`catch`, each wrapped by its own dedicated `try`/`catch`: the tag write by the [interstitial-seam swallow rule](./patch-skew-degradation.md#interstitial-seam-tag-write-swallow), the emission by [**Diagnostic-emission isolation**](./diagnostic-emission-isolation.md#diagnostic-emission-isolation). A throw out of `console.error` MUST be swallowed and MUST NOT unwind the handler; neither a tag-write throw nor a swallowed emission throw unwinds the handler before sub-step 5.

- **Non-conformant:** placing the emission *after* sub-step 5 returns â€” a sub-step-5 listener-detach throw would route through `loom/host/session-shutdown-teardown-step-failed` and the degraded notification would then co-emit out of order, or be skipped if a future edit grew an early return on that sub-step.

**Independence from sub-step 4.** The emission fires whether or not any of sub-step 4's three call sites (`discoveryWatcher.close()`, `settingsWatcher.close()`, the debounce-handle `Clock.clearTimeout(...)`) entered its per-call `try`/`catch`. The discriminator is the **branch** (the closed-set membership check), not any intra-sub-step-4 success flag â€” **Per-step isolation** already guarantees sub-step 4 returns, so a "sub-step 4 completed without a throw" predicate would be vacuous and the runtime MUST NOT introduce one.

**Inline triplet is normative.** The set `{"new", "resume", "fork"}` is **inlined at the emission site**, not read off the `SessionShutdownEvent['reason']` snapshot's `literals` field: the partition `{"quit","reload"}` vs `{"new","resume","fork"}` reflects Pi's documented teardown-vs-session-swap distinction, which no single SDK identifier pins (the partition assignment is grounded as [Host prerequisites for the degraded-state branch â€” presupposition (d)](./host-prerequisites.md#degraded-state-host-prerequisites), audited by editorial review on each Pi bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) checklist item (x)), and a parallel snapshot entry would double-pin the same union in two independently-drifting places. The inline triplet is therefore a **normative runtime literal** whose source of truth is this page (not an illustrative restatement under the [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) step-1 carve-out, which covers in-prose closed-set listings only). A Pi bump renaming a session-only literal (e.g. `"fork" â†’ "branch"`) surfaces as a bidirectional `SessionShutdownEvent['reason']` type-equality red carrying the brand string `loom/typecheck/session-shutdown-reason-snapshot` per [version bump](./version-bump-intro.md#pi-version-bump-procedure) step 5, whose inbound-reference sweep MUST update this inline triplet in the same edit so the diagnostic does not silently no-op for the renamed reason.

## Predicate split

Two distinct predicates, both backed **only** by the two handler-scoped seams `pinnedConstantReadOk` (captured at handler entry; `true` iff the snapshot lookup-and-`literals` read of the **Unknown-reason rule** succeeded, i.e. the `loom/host/session-shutdown-pinned-constant-unreadable` catch arm did not fire) and `capturedEventReason`:

| Predicate | Expression | Governs |
|---|---|---|
| **Diagnostic-emission** (narrow) | `pinnedConstantReadOk && capturedEventReason âˆˆ {"new","resume","fork"}` | the `loom/host/session-shutdown-runtime-degraded` emission only |
| **Tag-transition** (broad) | `capturedEventReason âˆ‰ {"quit","reload"}` | the unconditional `"degraded-needs-reload"` tag write |

The broad tag-transition predicate fires whenever the handler cannot determine the runtime is being torn down anyway, across three arms â€” each factually degraded (sub-step 4 closed both watchers; no un-drain rule):

1. **session-only triplet** `{"new","resume","fork"}` â€” `capturedEventReason` carries the delivered member;
2. **snapshot-failure path** â€” `capturedEventReason` retains its `"<unreadable>"` pre-initialisation (per the **Unknown-reason rule**'s fixed read order, the `event.reason` read was never reached), not in `{"quit","reload"}`;
3. **unknown-reason path with successful snapshot read** â€” `capturedEventReason` is the `String(event.reason)` coercion of the off-snapshot literal or the `"<unreadable>"` sentinel from a throwing-property-access read, neither in `{"quit","reload"}`.

The complement form `âˆ‰ {"quit","reload"}` is the value-independent seam pinning the conservative tag write across all three arms, so an operator on a stdio-capturing host (where the `console.error`-routed `loom/host/session-shutdown-pinned-constant-unreadable` and `loom/host/session-shutdown-reason-unknown` writes may be lost) still gets the in-band `"degraded; /reload to recover"` signal on the next slash invocation. The two `"<unreadable>"`-carrying arms (2, 3) are absorbed by this same complement check.

**Emission stays narrow on paths (2) and (3)** because each path's own diagnostic (`loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-reason-unknown`) already pins diagnostic-channel observability; a doubled-up `runtime-degraded` emission would describe the same event from a second angle and split operator dedup buckets across two codes without an operator-actionable distinction. The residual case where **both** the tag write and the path-specific `console.error` are lost is the *compound-failure residual gap* below.

**Harmlessness on `quit`/`reload` sub-cases:** on snapshot-failure-with-`quit`/`reload`, `capturedEventReason` is the `"<unreadable>"` sentinel (so in the fire set), but the host is tearing down anyway, no subsequent slash invocation reaches the tag, and observability stays on `pinned-constant-unreadable`. The same argument covers any unknown literal whose semantics coincide with `"quit"`/`"reload"`: the [version bump](./version-bump-intro.md#pi-version-bump-procedure) step-5 snapshot edit would admit it to the closed set, and the predicate would then correctly suppress the tag write.

**Seam minimality (normative).** The runtime MUST NOT introduce any handler-scoped state seam beyond `pinnedConstantReadOk` and `capturedEventReason` â€” no `branchTakenWasClosedSet` flag, no per-sub-step success roll-up, and no separate `!pinnedConstantReadOk` disjunct or `"<unreadable>"` sentinel-string equality check to back arms (2)/(3). The two seams exhaust both predicates' information requirements. Two load-bearing choices follow:

- `pinnedConstantReadOk` pins the snapshot-failure-with-closed-set-`event.reason` exclusion against an implementation that grew a fresh `event.reason` re-read at the discriminator site or mistakenly stamped the raw delivered value rather than the captured one.
- The **captured-string triplet check** (not a fresh `event.reason` property access at the discriminator site) is load-bearing against the unknown-reason path: a fresh access would either re-throw or read a value already classified as unknown â€” both bypass the catch-arm classification and are **non-conformant**.

## Co-emission rules

Co-emission for the same `session_shutdown` event is **expected and never a deduplication target** (distinct codes describe distinct facts; operator-tooling dedup keys bucket them separately by construction):

- **with `loom/host/session-shutdown-teardown-step-failed`** â€” when any sub-step-1 site (`LoomRegistry.drain`, `LoomRegistry.initDrainStateTag`), sub-step-4 site (`discoveryWatcher.close()`, `settingsWatcher.close()`, the debounce-handle `Clock.clearTimeout(...)`), or sub-step-5 site (`ctx.signal.removeEventListener`, `toolSignal.removeEventListener`, `parentInvokeSignal.removeEventListener`) threw. Dedup keys: `(code, details.step, details.call)` vs `(code, details.event.reason)`. The watcher-close-failed sub-case carries the most operator value and MUST NOT be silenced.
- **with `loom/runtime/cancelled-by-session-shutdown`** â€” one row per cleanly-cancelled in-flight `ActiveInvocationRegistry` entry, emitted from each invocation's own per-invocation `finally` (the **Per-invocation operator visibility (clean-cancel path)** rule), not from this handler: sub-step 2 stamps each entry's `shutdownReason` with `capturedEventReason` and calls `loomAbort.abort()`; sub-step 3 awaits each `disposeBarrier` within the `SHUTDOWN_AWAIT_CAP_MS` window. Row count = entries observed at sub-step-2 entry whose `disposeBarrier` settled inside that window: **zero** when the handler entered with an empty registry (e.g. `/new` from an idle prompt, where the [snapshot/restore preconditions](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions) precondition (b) â€” Pi serialises slash dispatch per session â€” plus the symmetric in-process LIFO-nesting guarantee for nested prompt â†’ prompt `invoke(...)` calls means none is in flight), **one per cleanly-cancelled invocation** otherwise (e.g. `/new` mid-tool-loop). Dedup keys: `(code, details.event.reason)` vs a per-invocation key whose `details.event.loom` slash-name discriminator is pinned by the [*Runtime-constructed sibling carve-out*](../diagnostics/diagnostic-shape.md#session-shutdown-details-conventions). The sub-step-1 cases are pinned in fuller detail by acceptance branches (i)/(j).

## Idempotent re-entry and the all-three-throw corner case

The diagnostic-emission predicate is re-evaluated from scratch on every handler invocation the [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause's broadened short-circuit predicate (`snapshot.drained === true || snapshot.tag !== undefined`) does not catch â€” including the loom 1.0-accepted **all-three-throw corner case** (sub-step 1's `LoomRegistry.drain`, sub-step 1's `LoomRegistry.initDrainStateTag`, and the post-sub-step-4 `LoomRegistry.markRuntimeDegraded` all throw on a session-only `session_shutdown`, leaving `drained = false`, `tag = undefined`). On a follow-up session-only `session_shutdown` against the same instance with a successful follow-up snapshot read, the predicate fires again and an additional `loom/host/session-shutdown-runtime-degraded` row is emitted, breaching the at-most-once bound **for this corner case only**; operator dedup on `(code, details.event.reason)` collapses the duplicates. No additional seam is introduced to suppress it, on the same acceptance grounds the idempotent clause pins (all three swallow rules already fired; the runtime carries no further state seam to recover from).

## Accepted loom 1.0 residual gaps

Each gap below is **documented so a future edit that reduces any contributing cost can be assessed against this acknowledged baseline rather than treated as a new defect.**

1. **Compound-failure case.** On the conjunction of (1) the snapshot-failure path (narrow predicate suppresses `runtime-degraded`), (2) the post-sub-step-4 `LoomRegistry.markRuntimeDegraded` write throwing (swallowed by the [interstitial-seam `try`/`catch`](./patch-skew-degradation.md#interstitial-seam-tag-write-swallow), so the registry stays on `"shutting-down"`), and (3) a stdio-capturing host that silently drops the `console.error`-routed `loom/host/session-shutdown-pinned-constant-unreadable` write, the operator's next slash invocation returns the misleading `"shutting down"` note with no in-band `/reload` signal; recovery depends on out-of-band discovery. Accepted (rather than widening the emission predicate to fire on `!pinnedConstantReadOk`, or demoting the seam swallow to emit a fallback diagnostic) because the assessed probability of the three-way conjunction is low enough that the diagnostics-catalogue ripple is not warranted in loom 1.0. A future reducer of any one probability (e.g. an explicit [Pi stdio-capture facet](../future-considerations/model-changes-and-non-goals.md#pi-stdio-capture-facet)) is assessed against this baseline.

2. **Edit-without-slash-invocation case.** An operator who edits a `.loom` file or settings between the handler returning and their next slash invocation gets no in-band evidence the edit was ignored: the watchers are closed, the runtime MUST NOT poll/re-subscribe/re-establish per the prohibition above, and the only in-band signal â€” the distinct `"degraded; /reload to recover"` slash note â€” fires only on the next slash command (the `console.error` `runtime-degraded` row is the out-of-band signal on stdio-surfacing hosts). Accepted (rather than a polling fallback, re-subscribe path, or separate edit-rejection diagnostic â€” each would contradict the prohibition or require a new persistent degraded-branch channel) because a forthcoming slash invocation is the natural recovery signal and the slash note already names watcher closure as the cause.

3. <a id="substep-2-stamp-throw-residual-gap"></a>**Sub-step-2 stamp-throw case.** On a sub-step-2 per-entry stamp throw (frozen/proxied entry, write-rejecting setter, OOM during property assignment â€” per sub-step 2's per-entry `catch` arm), that arm proceeds to the next entry and the `loomAbort.abort()` call for the affected entry is **skipped**, leaving that invocation un-cancelled by the handler (sub-step 2's "cancel in-flight invocations" framing is silently relaxed for the one entry). The operator-visible signal is the entry's eventual appearance in the single `loom/runtime/reload-teardown-timeout` `<list>` of sub-step 3 (nothing aborted it, so its `disposeBarrier` does not settle within `SHUTDOWN_AWAIT_CAP_MS`); that diagnostic carries no discriminator distinguishing "never called `abort()` because the stamp threw" from "called `abort()` and `disposeBarrier` did not settle". 
   - If the affected invocation's `finally` later reaches the **Per-invocation operator visibility (clean-cancel path)** rule (e.g. a downstream cancellation eventually settles its `disposeBarrier`), the `details.event.reason` it emits â€” when its `shutdownReason` is unset because the stamp threw â€” is the sentinel `"<unreadable>"`, on the same footing as the **Unknown-reason rule**'s convention, so the pinned type `"quit" | "reload" | "new" | "resume" | "fork" | string` is preserved. The `"<unreadable>"` substitution applies to `details.event.reason` **only**; the companion `details.event.loom` field is **unaffected** (populated at invocation-start insertion, before the sub-step-2 stamp, and never mutated), so the slash-name discriminator required by the [*Runtime-constructed sibling carve-out*](../diagnostics/diagnostic-shape.md#session-shutdown-details-conventions) on every `loom/runtime/cancelled-by-session-shutdown` emission survives in canonical form.
   - This co-emission (the entry in sub-step 3's `reload-teardown-timeout` `<list>` **and** a later `cancelled-by-session-shutdown` from its own `finally`) is the sole residual-gap exception the [`cancelled-by-session-shutdown` mutual-exclusion clause](../diagnostics/code-registry-runtime.md#cancelled-by-session-shutdown-mutual-exclusion) calls out to its "never both" invariant; consumers may treat the dual appearance as a residual-gap signal, not a contract violation.
   - Accepted (rather than routing the per-entry `catch` arm to a new diagnostic with an extended `details.step` enumeration) because a stamp throw on a `Set` entry the runtime itself allocates as a plain mutable object (its `shutdownReason` is `undefined` at insertion and is never frozen/proxied/write-rejecting by the runtime) is low-probability.

## Acceptance criteria

The rule above decomposes into the per-branch fixture obligations below, all of which the implementation MUST satisfy (the leaf owns fixture-shape; branch coverage and observable outcomes are normative). **Count semantics:** emission counts are measured at the **handler invocation site** (the number of times the handler enters the leaf-owned serialise-and-emit sequence for one event), per [Diagnostic-emission isolation](./diagnostic-emission-isolation.md#diagnostic-emission-isolation)'s *Count semantics* pin â€” **not** the number of rows reaching an operator sink (which may be zero on a host that drops `console.error`). The isolation rule's silent swallow on inner throw satisfies "emits exactly one" (the invocation occurred); a fixture observing zero invocations after the wrap entered would surface a non-conformant retry-suppression/short-circuit, not a dropped-write artefact.

- **(a) Positive â€” clean session-only reason.** For each `reason âˆˆ {"new","resume","fork"}` (3 fixtures) where the snapshot read succeeded and all three sub-step-4 sites returned without throwing: exactly one `runtime-degraded` row with `details.event.reason` = the reason; no `teardown-step-failed` row.
- **(b) Positive â€” session-only reason with one sub-step throw (co-emission).** For each `reason âˆˆ {"new","resume","fork"}` Ă— each single call site throwing alone (the two sub-step-1, three sub-step-4, and three sub-step-5 sites = **3 Ă— 8 = 24 fixtures**): exactly one `runtime-degraded` row **alongside** exactly one `teardown-step-failed` row whose `details.call` = the failing site. Confirms the independence invariant and its sub-step-1/sub-step-5 analogues (neither predicate carries a sub-step-1 success conjunct, so a caught sub-step-1 throw still reaches the post-sub-step-4 seam; sub-step-5 detaches run **after** that seam, so a caught sub-step-5 throw cannot retroactively silence the already-emitted row); the degraded row MUST NOT be silenced or deduplicated against the per-call failure row. Multiple simultaneously-failing sites are **out of scope** for (b) (covered only by the all-three-throw corner case). Sub-step-1 sub-cases are additionally covered, with post-handler drain-state assertions, by (i)/(j).
- **(c) Negative â€” always-tear-down reasons.** For each `reason âˆˆ {"quit","reload"}`: zero `runtime-degraded` rows.
- **(d) Negative â€” unknown-reason path (successful snapshot read).** For an `event.reason` outside the closed set (and the throwing-`event.reason`-property-access fixture), with `pinnedConstantReadOk = true` (captured value = the `String(event.reason)` coercion or the `"<unreadable>"` sentinel): zero `runtime-degraded` rows; the path's own `loom/host/session-shutdown-reason-unknown` covers observability. Additionally asserts the next slash invocation returns `"degraded; /reload to recover"` (**not** `"shutting down"`) â€” confirming the broad tag-transition predicate writes the degraded tag despite the suppressed diagnostic (arm 3).
- **(e) Negative â€” snapshot-failure path.** For a fixture forcing the handler-entry pinned-constant lookup to fail (`pinnedConstantReadOk = false`): zero `runtime-degraded` rows **even when** Pi delivers a session-only `event.reason` like `"new"`; the path's own `loom/host/session-shutdown-pinned-constant-unreadable` covers observability. Additionally asserts the next slash invocation returns `"degraded; /reload to recover"` (**not** `"shutting down"`) â€” confirming the broad predicate (arm 2; the `"<unreadable>"` sentinel is not in `{"quit","reload"}`). Covers both the session-only sub-case (e.g. `"new"`) and the always-tear-down sub-case (e.g. `"quit"`); the slash-invocation arm-check is meaningful only on the former (on the latter Pi tears down before any subsequent slash), but the zero-rows assertion holds on both.
- **(f) Drain-state observability (pre-recovery).** After any (a)/(b) shutdown whose interstitial-seam tag write did not throw (the throwing arm is (k)), the next slash invocation returns `"degraded; /reload to recover"`, **not** `"shutting down"`.
- **(g) Recovery.** After `/reload` re-executes the factory, the next slash invocation short-circuits on neither drain-state arm and dispatches its loom normally (neither note returned), confirming `/reload` is the recovery path and the fresh runtime starts with a non-drained `LoomRegistry`.
- **(h) Idempotent re-entry on the degraded branch.** A follow-up `session_shutdown` (any reason, including a fresh `"quit"`) after an (a)/(b) shutdown returned MUST be short-circuited at handler entry: zero additional `runtime-degraded` rows (preserving the at-most-once bound) and zero additional rows of any other teardown-handler code.
- **(i) State-independent write â€” tag-init throw still routes to the degraded arm.** For each `reason âˆˆ {"new","resume","fork"}` (3 fixtures) where the snapshot read succeeded but sub-step-1 `LoomRegistry.initDrainStateTag` throws (caught; emits one `teardown-step-failed` under `details.call = "loomRegistry.initDrainStateTag"`; leaves `drainStateTag = undefined`): the handler still reaches the post-sub-step-4 `markRuntimeDegraded` write, which completes the unconditional `"degraded-needs-reload"` write, and emits exactly one `runtime-degraded` row; the next slash invocation returns `"degraded; /reload to recover"`, not `"shutting down"`. Confirms the state-independent-write pin against an implementation that conditioned the transition on a `"shutting-down"` source state (which would strand the runtime on `"shutting-down"` whenever `initDrainStateTag` threw).
- **(j) Drain-throw recovery â€” short-circuit established by the tag setter.** For each `reason âˆˆ {"new","resume","fork"}` (3 fixtures) where the snapshot read succeeded but sub-step-1 `LoomRegistry.drain` throws (caught; emits one `teardown-step-failed` under `details.call = "loomRegistry.drain"`; leaves `drained = false`): the handler continues into the second sub-step-1 act (`initDrainStateTag` succeeds, `tag = "shutting-down"`), runs sub-steps 2â€“4, reaches the post-sub-step-4 `markRuntimeDegraded` write (`tag â†’ "degraded-needs-reload"`), and emits exactly one `runtime-degraded` row. The `reason âˆˆ {"quit","reload"}` variant (2 additional fixtures) MUST emit zero `runtime-degraded` rows (per (c)) and the post-handler tag via `readDrainState` is `"shutting-down"` (`markRuntimeDegraded` does not run); the variant MUST NOT include a slash-arm assertion (Pi tears the runtime down before any subsequent slash dispatch). ## Emergent fast-path (sub-steps 2â€“3 only)

The fast-path framing applies to **sub-steps 2 and 3 only** (the abort-and-await over `ActiveInvocationRegistry`): a session-only reason often leaves them with an empty registry (e.g. `/new` from an idle prompt, per the [snapshot/restore preconditions](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions) precondition (b) and the symmetric in-process LIFO-nesting guarantee), but the registry MAY be non-empty when the operator issues a session-only reason mid-invocation (e.g. `/new` mid-tool-loop) â€” sub-step 2's per-entry stamp + abort and sub-step 3's bounded await handle that on the same footing as the always-tear-down branch. A session-only reason does **not** skip sub-step 4 or any other sub-step. This fast-path is available only for the five enumerated reasons; an unknown `event.reason` is routed by the **Unknown-reason rule** and MUST run the full five-sub-step sequence even when the registry is empty.
