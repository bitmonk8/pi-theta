# Session-only session-swap disposition

> **RESOLVED (loom 1.0): `governed-by-rebind` — the session-only degraded-state branch is retired and replaced by the fail-fast tripwire below.** Host-prerequisites [clause (a)](./host-prerequisites.md#degraded-state-host-prerequisites) has been authoritatively resolved as `governed-by-rebind`: at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) Pi tears the extension runtime down and rebinds a fresh instance on every session-only `session_shutdown` (`"new"`/`"resume"`/`"fork"`), so the drained `LoomRegistry` the degraded branch depended on never survives. The *Session-swap fail-fast tripwire* section below is the **authoritative, governing** normative content of this page; the former degraded-state branch — the drain-state tag write, recovery path and prohibitions, the `loom/host/session-shutdown-runtime-degraded` emission, the predicate split, co-emission rules, idempotent re-entry / all-three-throw corner case, accepted residual gaps, and the acceptance-criteria fixtures (a)–(m) — has been **excised**. `SM-4/5/6/3b`, `drain-state-contract.md` (`LoomRegistry.markRuntimeDegraded` and `readDrainState` arm (c)), and the `loom/host/session-shutdown-runtime-degraded` diagnostic row are reworked/retired accordingly.

## Session-swap fail-fast tripwire (governed-by-rebind resolution)

Under the recorded `governed-by-rebind` determination the runtime does **not** implement a degraded-state branch. On the session-only reasons (`event.reason ∈ {"new","resume","fork"}`) the `session_shutdown` handler runs the **same full five-sub-step teardown as the always-tear-down reasons** `{"quit","reload"}` — it writes no `"degraded-needs-reload"` drain-state tag, performs no `LoomRegistry.markRuntimeDegraded` transition, emits no `loom/host/session-shutdown-runtime-degraded` row, fronts no `"degraded; /reload to recover"` slash note, and provides no degraded-recovery path (Pi's own teardown-and-rebind is the recovery: the swapped-in session runs on a fresh instance with a fresh, non-drained `LoomRegistry`).

**Fail-fast tripwire invariant (normative).** Because `governed-by-rebind` is the load-bearing premise, the runtime enforces it at runtime rather than trusting it silently:

- **Arm.** After completing teardown on a session-only `session_shutdown` (`capturedEventReason ∈ {"new","resume","fork"}`, read via the same handler-scoped seam the **Unknown-reason rule** pins), the handler sets a private per-extension-instance boolean `sessionSwapTornDown = true` on the closed-over `LoomRegistry`. This flag is **not** part of the public `readDrainState` snapshot surface (it is neither of the two drain-state fields; it adds no `readDrainState` arm) and is written idempotently, so a permitted multi-`session_shutdown` delivery to one instance re-arms harmlessly.
- **Trip.** Every loom-registered slash `handler` (at entry, before any dispatch or `readDrainState` branch) and the `session_start` handler MUST read `sessionSwapTornDown`. If it is set, the extension instance has survived a session-only `session_shutdown` and is dispatching again — the exact behaviour `governed-by-rebind` proves does not occur — so the runtime emits exactly one [`loom/host/session-swap-instance-survived`](../diagnostics/code-registry-host.md) (E, runtime) diagnostic via `console.error` (carrying `details: { event: { reason } }` with the armed session-only reason) and then terminates the process on the loom fail-fast path (the [NFR-2.1 unhandled-exception `Environment.FailFast`-equivalent](../../plan_topics/conventions.md) "let crash" posture). The runtime MUST NOT attempt any degraded-mode dispatch, recovery, or continued operation past the trip.
- **Dormancy.** Under the proven lifecycle the armed instance is disposed by `this.session.dispose()` and never dispatches again; a fresh instance carries a fresh, unarmed `LoomRegistry`. The tripwire is therefore dormant on every conformant Pi minor — it fires only if a future Pi minor violates the teardown-and-rebind lifecycle, surfacing the violation loudly at the contradiction point instead of running unspecified logic.
- **Non-guarded surfaces.** The Pi-host-owned `/reload` command is not a loom-registered `pi.registerCommand` handler and is not guarded (consistent with clause (c)); the tripwire guards only loom-registered slash handlers and `session_start`.

**Acceptance (normative; the leaf owns fixture-shape).** (1) For each `reason ∈ {"new","resume","fork"}`: the handler runs full teardown, arms `sessionSwapTornDown`, and emits none of the degraded-branch diagnostics/notes. (2) A slash `handler` (or `session_start`) invoked against an armed tripwire emits exactly one `loom/host/session-swap-instance-survived` row and fail-fast-terminates. (3) Under the normal rebind (fresh unarmed registry per swap) the guard is dormant — no dispatch or `session_start` observes an armed flag and the process is not terminated.

### Still-true operational notes (tripwire path)

These notes describe teardown-sequence behaviour that the tripwire path still relies on; they carry no degraded-tag, `runtime-degraded`, or predicate-split machinery.

- **Sub-step 2–3 fast-path emptiness.** A session-only reason often leaves the abort-and-await over the `ActiveInvocationRegistry` (sub-steps 2 and 3) with an empty registry (e.g. `/new` from an idle prompt, per the [snapshot/restore preconditions](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions) precondition (b) and the symmetric in-process LIFO-nesting guarantee), but the registry MAY be non-empty when the operator issues a session-only reason mid-invocation (e.g. `/new` mid-tool-loop) — sub-step 2's per-entry stamp + abort and sub-step 3's bounded await handle that on the same footing as the always-tear-down branch. A session-only reason does **not** skip sub-step 4 or any other sub-step, and the full five-sub-step sequence runs even when the registry is empty.
- **Co-emission of `loom/runtime/cancelled-by-session-shutdown`.** One `loom/runtime/cancelled-by-session-shutdown` row is emitted per cleanly-cancelled in-flight `ActiveInvocationRegistry` entry whose `disposeBarrier` settled inside the `SHUTDOWN_AWAIT_CAP_MS` window, from each invocation's own per-invocation `finally` (the **Per-invocation operator visibility (clean-cancel path)** rule), not from the teardown handler. The row count is **zero** when the handler entered with an empty registry and **one per cleanly-cancelled invocation** otherwise.
