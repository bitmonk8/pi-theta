**`estimateTokens` (named export).** Imported from `@earendil-works/pi-coding-agent`. Signature: `estimateTokens(message: AgentMessage): number`. The export is declared at `dist/core/compaction/compaction.d.ts` in `@earendil-works/pi-coding-agent` (the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above), where its parameter type `AgentMessage` is imported from `@earendil-works/pi-agent-core` (declared at that package's `dist/types.d.ts` as the agent-state superset of the LLM-side `Message` wire type ŌĆö `AgentMessage` additionally carries `thinking` blocks, `toolCall` blocks, and host-defined custom messages). The estimation algorithm is owned by Pi and is not re-specified here: theta calls the function as-imported and does not redefine how it walks the message. theta does not consume this import directly at the truncation callsite; it is the delegation target of the production `PiTokenEstimator` adapter behind the [`TokenEstimator` seam (PIC-16)](./host-interfaces-services.md#tokenestimator-interface), and [Slash-Command Argument Binding ŌĆö Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) reads its per-message and per-turn token counts through that seam. theta's observable contract is unchanged: the returned `number` is the per-message token estimate the truncation walk sums. The export is not factory-probed and not enumerated by the SDK surface-inventory literal-read assertion, so its shape is re-audited on each Pi minor bump via item (p) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. Theta additionally relies on this export being a deterministic pure function of its `message` argument at a fixed pin ŌĆö the [per-message-determinism presupposition](./host-prerequisites.md#token-estimator-determinism-presupposition) the binder's input-reproducibility contract depends on, re-audited via item (aq) below ŌĆö distinct from this paragraph's signature-shape re-audit. **Non-normative.** At the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin), Pi's estimator (in `@earendil-works/pi-coding-agent`'s `core/compaction/compaction.ts`) derives a conservative character-quarter count (`Math.ceil(chars / 4)`-style, counting UTF-16 code units over the message's contributing strings); for orientation, a user message whose only content is `"­¤śĆ­¤śĆ­¤śĆ"` (six UTF-16 code units) estimates to roughly 2 tokens. This description is orientation only, is owned by Pi, and may drift on any Pi revision; the authoritative behaviour is Pi's implementation at the pin.

**`buildSessionContext` (named export).** Imported from `@earendil-works/pi-coding-agent`; this is a free function, **not** a method on `ctx.sessionManager` ŌĆö `ExtensionContext.sessionManager` is typed as `ReadonlySessionManager` (a `Pick<>` of the underlying `SessionManager`; see the inline shape below) and deliberately excludes `buildSessionContext` and every mutating member. Signature: `buildSessionContext(entries: SessionEntry[], leafId?: string | null, byId?: Map<string, SessionEntry>): SessionContext`. The runtime calls it as `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId())` ŌĆö both `getEntries()` and `getLeafId()` are inside the `ReadonlySessionManager` `Pick<>`, so the call composes against the read-only surface in both prompt mode and subagent mode. The returned `SessionContext`'s `.messages` field is the ordered message list the runtime consumes for session-aware features; its element type is `AgentMessage[]`, pinned in [`SessionContext` and the `.messages` element shape](#sessioncontext-shape) below. theta 1.0 callsite: [Slash-Command Argument Binding ŌĆö Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) sources the binder's caller-session message list from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`. The runtime MUST NOT substitute `ctx.getContextUsage()` for per-turn token accounting; that API reports aggregate session usage, not per-turn counts, and is reserved for compaction triggers and footer rendering.

**`SessionContext` and the `.messages` element shape (theta-load-bearing subset).** `SessionContext` is declared at `dist/core/session-manager.d.ts` in `@earendil-works/pi-coding-agent` (the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)); its `.messages` field is typed `AgentMessage[]`, where `AgentMessage` is the agent-state superset cited in the **`estimateTokens` (named export)** paragraph above (declared at `@earendil-works/pi-agent-core`'s `dist/types.d.ts`). Two Pi *behavioural* preconditions govern the theta-consumed `AgentMessage[]` surfaces ŌĆö their [chronological ordering](./host-prerequisites.md#messages-chronological-order-presupposition) (oldest-to-newest) and their [leading-`user`-message guarantee](./host-prerequisites.md#messages-leading-user-message-presupposition) (non-empty iff the first element is a `UserMessage`) ŌĆö neither encoded by the `AgentMessage[]` type; both are inventoried with the other host-behaviour presuppositions on [Host prerequisites](./host-prerequisites.md) and apply to the `.messages` field pinned above as well as to the terminal `agent_end` `messages` field consumed in subagent mode.

*Per-mode override semantics.* The runtime synthesises the `ctx` it hands to `tool.execute(...)` by starting from the live host's `ExtensionContext` and overriding exactly three members. The overrides apply uniformly in both modes ŌĆö there is no member that is overridden in one mode and forwarded unchanged in the other:

| Member | Prompt mode | Subagent mode |
|---|---|---|
| `signal` | replaced with `thetaAbort.signal` (always defined) | replaced with `thetaAbort.signal` (always defined) |
| `sessionManager` | replaced with Pi's user-session manager (the host's, observed via the captured `ExtensionCommandContext`) | the parent handler's `ExtensionCommandContext.sessionManager` is forwarded for code-side tool calls; there is no in-process subagent session manager under the child-process theta design ŌĆö the subagent's private conversation lives in the spawned child `pi` process, which runs the whole callee and returns only its final-value envelope, never a substituted in-process `SessionManager` (see **Conversation drive ŌĆö subagent mode** [return-value envelope (PIC-59)](./subagent.md#pic-59) above) |
| `abort` | wrapped to call `thetaAbort.abort()` rather than the host's `abort()` ŌĆö so `ctx.abort()` tears down the theta's invocation, not the user's whole Pi turn; a throw from the underlying `thetaAbort.abort()` invoked by the wrapper is trapped per the [Cancellation ŌĆö Forwarding-listener throw](../cancellation.md) clause and routed through `theta/runtime/internal-error` | wrapped identically ŌĆö `ctx.abort()` tears down the subagent invocation, not the parent's turn; the same forwarding-listener-throw trap applies |

Every other member from the signature block above forwards unchanged to the live host:

- *Prompt mode.* The live host is the `ExtensionCommandContext` passed to the slash-command handler, retained for the theta's lifetime. `cwd`, `ui`, `modelRegistry`, `model`, `isIdle`, `getContextUsage`, `hasPendingMessages`, `compact`, `shutdown`, and `getSystemPrompt` all reflect the user-session view at the moment of the tool call.
- *Subagent mode.* Under the child-process subagent design the subagent conversation runs in a spawned child `pi` process and carries no in-process `ExtensionCommandContext`. For **code-side** tool calls the parent interpreter still synthesises `ctx` from the parent slash-command handler's `ExtensionCommandContext` for the non-overridden members (`cwd`, `ui`, `model`, `isIdle`, etc. reflect the parent's view). Two members that were previously described as forwarded do **not** cross into the child conversation and are not part of the subagent's isolated state: `ctx.modelRegistry` and `ctx.getContextUsage` are parent-process objects and do not cross the process boundary (per the [subagent state-isolation matrix](./subagent.md#subagent-state-isolation-matrix)); the subagent's model is instead marshalled to the child as a reference (`--provider`/`--model`), re-resolved and confirmed child-side per [PIC-62](./subagent.md#pic-62). The `cwd` value forwarded to the child (as its working directory) and to code-side `ctx` is the parent's `ctx.cwd` verbatim ŌĆö theta 1.0 has no per-subagent frontmatter `cwd` override; widening is reserved for [Future Considerations](../future-considerations.md).

The `abort()` override applies in **both** modes because no spec-supported caller of `ctx.abort()` from inside theta code wants to tear down the user's turn ŌĆö the in-theta code paths that call `abort()` are tools acting on behalf of the theta, whose authoritative cancellation signal is `thetaAbort`. The contract pinned by this paragraph ("`ctx.abort()` aborts the theta's invocation and not the parent's turn") under the spec's "Tool execution from theta code" surface holds in both prompt and subagent modes.

<a id="tool-execution-from-theta-code"></a>

**Tool execution from theta code.** Code-side `<name>(args)` calls invoke the Pi tool's `execute(toolCallId, params, signal, onUpdate, ctx)` directly:

- `toolCallId` is a synthesised UUID prefixed `theta-direct:`, whose UUID body renders in canonical lowercase 8-4-4-4-12 hex form (the [┬¦7 `<uuid>` placeholder convention](../diagnostics/placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders)) and is minted fresh per code-side tool call through the [`IdSource` seam (PIC-20)](./host-interfaces-services.md#pic-20).
- `params` is the theta value lowered to JSON (wire names applied).
- `signal` is the theta's current `thetaAbort.signal` (always defined; see **Cancellation source** below).
- `onUpdate` is a no-op (theta 1.0 does not surface streaming partial results to theta code).
- `ctx` is the synthesised `ExtensionContext` defined under [`ExtensionContext` (member surface theta touches)](#extensioncontext-interface) above: the live host's `ExtensionContext` with `signal`, `sessionManager`, and `abort()` overridden per the per-mode-override table, and every other member forwarded to the live host.

`execute(...)` returns an `AgentToolResult` and signals failure by **throwing**, not by setting a result flag. At the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) (from **Host prerequisites** above), `AgentToolResult` ŌĆö the declared return type of the tool `execute(...)` member of `ToolDefinition` ŌĆö is `{ content, details, terminate? }`; theta reads only its `content` field (typed `({ type: "text"; text: string } | ŌĆ”)[]`), and the type carries **no** `isError` field. The `isError` flag lives only on dispatcher-side types (`ToolResultMessage`, hook contexts, the `tool_execution_end` event) that theta never observes for an out-of-band code-side call. The `AgentToolResult` return shape (declared in `dist/core/extensions/types.d.ts` of `@earendil-works/pi-coding-agent`) is the contract every code-driven tool call's outcome routing depends on; the SDK surface-inventory test asserts the `AgentToolResult` field set so a Pi minor that adds an `isError`-style flag to the return type, drops the `text`-block discriminator, or widens `content` to a non-array shape surfaces at the build before the spec contract drifts. The returned `AgentToolResult` is lowered to the theta 1.0 string return value as follows. The lowering procedure inspects the resolved envelope and any throw raised during that inspection (the resolved value is not an object, `content` is not iterable, an entry is missing `type` / `text`, or any other shape violation that causes a property access or `.filter` / `.map` call to throw) is caught by the runtime and routed to `theta/runtime/internal-error` with `details.kind = "tool-return-shape"` (carrying `tool_name` and the `details.shape_check` discriminator token, whose closed value space is pinned normatively by that row's *Trigger* prose in [Code registry runtime](../diagnostics/code-registry-runtime.md)); the call site observes the `internal-error` routing per [Errors and Results ŌĆö Runtime panics](../errors-and-results.md), **not** an `Err(QueryError { kind: "code_tool", ... })`. The runtime filters `content` to entries with `type === "text"`, joins their `.text` values with `"\n"` (single line-feed; no separator before the first or after the last block), and returns the joined string. Non-text content blocks (images, resource references) are discarded during lowering; the theta-level observable is exhausted by the resulting `Ok("")` value. The runtime MUST NOT emit a `RuntimeEvent`, `theta-system-note`, or any other normative diagnostic on the discard path ŌĆö non-text discard is not a `QueryError` and is not a member of the always-log set. Implementations MAY emit non-normative debug-channel logs for operator visibility. Widening to a richer return shape (preserving non-text blocks) is reserved for [Future Considerations](../future-considerations.md). An empty result string ŌĆö either `content: []` or a content array with no surviving text blocks ŌĆö is a legal `Ok("")` value. When `execute()` throws, the runtime returns `Err(QueryError { kind: "code_tool", cause: "execution", message: <m>, tool_name, ... })` (this value is also referred to as `CodeToolError`; the two names denote the same `QueryError` variant ŌĆö the one whose `kind` discriminator is `"code_tool"` ŌĆö see [Errors and Results ŌĆö `CodeToolError`](../errors-and-results/queryerror-variants.md#code-side-tool-call-variant)), where `<m>` is the thrown value coerced to the [underlying-error string](../diagnostics/placeholder-rendering-b.md#underlying-error-coercion) (so a non-Error throw does not yield `undefined` or a synchronous `TypeError`), encoded as UTF-8 and truncated so that the resulting byte length is at most 4096 bytes. Truncation MUST cut on a Unicode code-point boundary: every code point in the output is represented by all of its UTF-8 bytes, and no bytes of a partial code point appear. When a code point would straddle the 4096-byte limit, that code point is dropped entirely; the resulting message MAY therefore be shorter than 4096 bytes by up to three bytes. Worked example: a coerced thrown-value string whose first 4095 bytes are ASCII followed by a 3-byte UTF-8 code point (e.g. U+2026 `ŌĆ”`) MUST truncate to 4095 bytes, because including the 3-byte code point would yield 4098 bytes.

*Outcome routing summary.* The complete theta 1.0 routing of an `execute()` outcome at this boundary is fixed: a cleanly-resolving `AgentToolResult` lowers to `Ok(<filtered/joined text>)` (possibly `Ok("")`); an `execute()` throw lowers to `Err(CodeToolError { cause: "execution", message: <underlying-error string truncated>, ... })`; a `thetaAbort.signal`-aborted call lowers to `Err(CodeToolError { cause: "cancelled", ... })` per the cancellation surfacing rule in [Cancellation ŌĆö Surfacing](../cancellation.md); a non-conforming envelope (anything that throws during the lowering above) routes to `theta/runtime/internal-error` with `details.kind = "tool-return-shape"` and is **not** observable as a `CodeToolError`; a non-settling Promise blocks the call's `await` indefinitely until cancellation fires, at which point the cancelled path applies (theta 1.0 makes no internal timeout attempt, consistent with [`spec.md` ŌĆö Hard ceilings, NOCEIL-1](../overview-and-orientation.md#hard-runtime-ceilings); the deferred per-call timeout extension lands additively on the open-struct seam in [Tool Calls](../tool-calls.md)); a settlement that arrives after the tool-call checkpoint has already surfaced `cause: "cancelled"` is discarded per [Cancellation ŌĆö late-settlement discard](../cancellation.md#cncl-1) ([CNCL-1](../cancellation.md#cncl-1) no-rebind, [CNCL-2](../cancellation.md#cncl-2) no-second-`Err`, [CNCL-3](../cancellation.md#cncl-3) no-second-`RuntimeEvent`) ŌĆö the runtime MUST keep a swallowing handler attached to the underlying `execute()` Promise (and, symmetrically, to the underlying `@`-query provider Promise and the `invoke` callee's top-level execution Promise) for the lifetime of that Promise, installed at construction time before the first microtask boundary, so that a late rejection on the discarded path does **not** surface as a Node `unhandledRejection` process event; the discard is total (no `Err` rebinds the call site, no `RuntimeEvent` reaches the always-log channel, no diagnostic of any severity is emitted, and the discarded value is not traversed). The `CodeToolError.cause` enum (closed at `validation` / `execution` / `cancelled` / `unknown_tool` per [Errors and Results ŌĆö `CodeToolError`](../errors-and-results/queryerror-variants.md#queryerror-variants)) is the contract surface for theta authors; the non-conforming-shape and non-settling-Promise dispositions are deliberately routed off it.

**Subagent conversation drive surface (formerly the `AgentSession` member surface).** Under the RFC 0006 child-process theta design the runtime holds **no** in-process `AgentSession` handle and drives **no** remote session: the former `createAgentSession(...)` handle and its `abort` / `dispose` / `sendUserMessage` / `subscribe` member surface, and RFC 0005's RPC drive contract that replaced them, both leave the contract entirely. The whole subagent callee runs inside a spawned child `pi --mode json -p "/<slug>"` process (per [Subagent ŌĆö Conversation drive](./subagent.md#pic-58)); the child drives its own queries against its own host session, and the theta-load-bearing surface the parent consumes is now the child's stdout **return envelope** ŌĆö the reserved-key `{"theta_result": ŌĆ”}` line pinned at [Return-value envelope (PIC-59)](./subagent.md#pic-59) ŌĆö alongside the `--mode json` event stream. That surface is a **consumed CLI/stdout-wire pin**, not a `typeof`-probable `.d.ts` member surface, so it is re-audited by editorial review at bump time (item (o) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure)) rather than by the SDK surface-inventory literal-read assertion.

<a id="cancellation-source"></a>

**Cancellation source.** Pi's delivery of an `AbortSignal` at the two extension entry points theta uses (`ctx.signal` and `tool.execute`'s `signal` parameter) is the SDK prerequisite this section consumes; see **Host prerequisites** above. The two entry-point declarations live in `dist/core/extensions/types.d.ts` of `@earendil-works/pi-coding-agent` at the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin): `ExtensionContext.signal: AbortSignal | undefined` (the slash-command handler's turn signal, with the JSDoc verbatim *ŌĆ£current abort signal, or undefined when the agent is not streamingŌĆØ*) and the `signal: AbortSignal | undefined` parameter of the tool `execute(...)` member of `ToolDefinition`. Subagent mode has no `AbortSignal` entry point ŌĆö the spawned child `pi` process carries no in-process signal parameter for the parent to pass ŌĆö and instead propagates cancellation by closing the child's parent-held stdin pipe (the grace signal), then a bounded grace and process-tree kill; the `-p` child exposes no RPC abort command, per [PIC-63 ŌĆö Cancellation](./subagent.md#pic-63) above. As described in [Cancellation](../cancellation.md), the theta's `AbortSignal` is `thetaAbort.signal` from a runtime-owned `AbortController` constructed fresh per invocation ŌĆö **not** `ctx.signal` directly. Per the JSDoc above, `ctx.signal` is `undefined` in idle / non-turn contexts, and a slash-command handler fires from idle, so the runtime MUST NOT use `ctx.signal` as the theta's authoritative signal: doing so would silently pass `undefined` into every adapter (the synthesised tool-execution `ExtensionContext.signal`, the child-invoke derived signal) and degrade every cancellation path to a no-op. Instead, the runtime forwards Pi's turn-side `ctx.signal` (when defined and observed inside one of the runtime's `tool_call` / `tool_result` / `message_update` / `turn_end` / `agent_end` handlers) into `thetaAbort.abort()`, and likewise forwards the `signal` parameter of a tool-exposed theta's `execute(...)` into `thetaAbort.abort()` via a one-shot listener registered at entry. All downstream queries, tool calls, and child invokes derive from `thetaAbort.signal`. Because `thetaAbort.signal` is always defined, the optional-chaining defence (`signal?.aborted`) used by tool adapters is no longer load-bearing inside the runtime; the runtime guarantees a defined signal to every adapter it calls. Subagent mode propagates cancellation through the one-shot `thetaAbort.signal` listener defined in [PIC-63 ŌĆö Cancellation](./subagent.md#pic-63) above: when `thetaAbort` fires, the listener initiates the grace signal (closing the child's parent-held stdin pipe), and teardown then follows the bounded stdin-close ŌåÆ await-exit ŌåÆ process-tree-kill lifecycle of [PIC-9 ŌĆö Subagent child-process lifecycle and orphan prevention](./subagent.md#pic-9) above. Forwarding listeners ŌĆö including the subagent abort listener ŌĆö are detached in the same `finally` block that tears the child process down (see [PIC-9](./subagent.md#pic-9) above). The `reason` value forwarded into `thetaAbort.abort(reason)` for each forwarding path ŌĆö including the runtime-synthesised values for the `agent_end`-driven and `session_shutdown`-driven triggers, which have no source `AbortSignal` ŌĆö is pinned by [Cancellation ŌĆö Abort-reason propagation](../cancellation.md); the rule applies uniformly across this paragraph's three forwarding shapes.
