# Host interfaces core

<a id="estimatetokens-named-export"></a>

**`estimateTokens` (named export).** Imported from `@earendil-works/pi-coding-agent`. Signature: `estimateTokens(message: AgentMessage): number`. The export is declared at `dist/core/compaction/compaction.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above), where its parameter type `AgentMessage` is imported from `@earendil-works/pi-agent-core` (declared at that package's `dist/types.d.ts` as the agent-state superset of the LLM-side `Message` wire type ŌĆö `AgentMessage` additionally carries `thinking` blocks, `toolCall` blocks, and host-defined custom messages). The estimation algorithm is owned by Pi and is not re-specified here: loom calls the function as-imported and does not redefine how it walks the message. loom does not consume this import directly at the truncation callsite; it is the delegation target of the production `PiTokenEstimator` adapter behind the [`TokenEstimator` seam (PIC-16)](./host-interfaces-services.md#tokenestimator-interface), and [Slash-Command Argument Binding ŌĆö Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) reads its per-message and per-turn token counts through that seam. loom's observable contract is unchanged: the returned `number` is the per-message token estimate the truncation walk sums. The export is not factory-probed and not enumerated by the SDK surface-inventory literal-read assertion, so its shape is re-audited on each Pi minor bump via item (p) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below. **Non-normative.** At the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin), Pi's estimator (in `@earendil-works/pi-coding-agent`'s `core/compaction/compaction.ts`) derives a conservative character-quarter count (`Math.ceil(chars / 4)`-style, counting UTF-16 code units over the message's contributing strings); for orientation, a user message whose only content is `"­¤śĆ­¤śĆ­¤śĆ"` (six UTF-16 code units) estimates to roughly 2 tokens. This description is orientation only, is owned by Pi, and may drift on any Pi revision; the authoritative behaviour is Pi's implementation at the pin.

**`buildSessionContext` (named export).** Imported from `@earendil-works/pi-coding-agent`; this is a free function, **not** a method on `ctx.sessionManager` ŌĆö `ExtensionContext.sessionManager` is typed as `ReadonlySessionManager` (a `Pick<>` of the underlying `SessionManager`; see the inline shape below) and deliberately excludes `buildSessionContext` and every mutating member. Signature: `buildSessionContext(entries: SessionEntry[], leafId?: string | null, byId?: Map<string, SessionEntry>): SessionContext`. The runtime calls it as `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId())` ŌĆö both `getEntries()` and `getLeafId()` are inside the `ReadonlySessionManager` `Pick<>`, so the call composes against the read-only surface in both prompt mode and subagent mode. The returned `SessionContext`'s `.messages` field is the ordered message list the runtime consumes for session-aware features; its element type is `AgentMessage[]`, pinned in [`SessionContext` and the `.messages` element shape](#sessioncontext-shape) below. loom 1.0 callsite: [Slash-Command Argument Binding ŌĆö Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) sources the binder's caller-session message list from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`. The runtime MUST NOT substitute `ctx.getContextUsage()` for per-turn token accounting; that API reports aggregate session usage, not per-turn counts, and is reserved for compaction triggers and footer rendering.

<a id="sessioncontext-shape"></a>

**`SessionContext` and the `.messages` element shape (loom-load-bearing subset).** `SessionContext` is declared at `dist/core/session-manager.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)); its `.messages` field is typed `AgentMessage[]`, where `AgentMessage` is the agent-state superset cited in the **`estimateTokens` (named export)** paragraph above (declared at `@earendil-works/pi-agent-core`'s `dist/types.d.ts`). The compact-transcript renderer at [Slash-Command Argument Binding ŌĆö Compact-transcript format](../binder/binder-model-and-context.md#compact-transcript-format-normative) is total over the `AgentMessage` arms listed below. The arms and their fields are owned by Pi; the list below is the loom-load-bearing subset the renderer reads (loom redefines none of these types and treats every field not listed here as opaque), and MUST be re-validated against the cited declarations on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure):

- `UserMessage` (declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`) ŌĆö its text content is read for the `[user]` body.
- `AssistantMessage` (declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`) ŌĆö its `content` is a `(TextContent | ThinkingContent | ToolCall)[]` array. The renderer concatenates each `TextContent.text` in array order, omits every `ThinkingContent`, and emits one `[tool-call ŌĆ”]` line per `ToolCall` (reading `ToolCall.name` and `ToolCall.arguments`). `TextContent`, `ThinkingContent`, and `ToolCall` are declared in the same `@earendil-works/pi-ai` `dist/types.d.ts`.
- `ToolResultMessage` (declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`) ŌĆö its `content` is a typed-block array; the renderer reads the text-block content for the `[tool]` body.
- `CustomMessage` (declared at `@earendil-works/pi-coding-agent`'s `dist/core/messages.d.ts`) ŌĆö its `customType` (a string) supplies the `[custom:<type>]` tag and its `content` (a `string | (TextContent | ImageContent)[]` union) supplies the body.

<a id="messages-chronological-order-presupposition"></a>Loom's `AgentMessage[]` consumers additionally rely on a Pi *behavioural* precondition the `AgentMessage[]` type does not encode: the array is **chronologically ordered, oldest-to-newest**. Loom reads an `AgentMessage[]` array on two distinct delivering surfaces, and this presupposition governs both: (1) `SessionContext.messages`, the array `buildSessionContext(...)` returns; and (2) the `messages` field on the terminal `agent_end` variant of `AgentSessionEvent` (declared at `dist/core/agent-session.d.ts` in `@earendil-works/pi-coding-agent` at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)), consumed only in subagent mode. Four loom consumers read an array in that order ŌĆö the [compact-transcript renderer](../binder/binder-model-and-context.md#compact-transcript-format-normative), the binder's [session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session), and prompt-mode untyped-query [trailing-turn extraction](./conversation-drive.md#untyped-query-ok-extraction) read surface (1), while subagent-mode untyped-query [trailing-turn extraction](./subagent.md#subagent-untyped-query-ok-extraction) reads surface (2) ŌĆö so a Pi minor that reordered either array (e.g. newest-first) would silently corrupt every one of them with no type or SDK surface-inventory signal. The ordering is taken on a best-effort basis against the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) and is re-audited on each Pi minor bump via item (h) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect a reordering from the type surface alone.

<a id="messages-leading-user-message-presupposition"></a>Loom's turn-grouping relies on a second Pi *behavioural* precondition the `AgentMessage[]` type does not encode and that governs both delivering surfaces above: each array is **non-empty iff its first element is a `UserMessage`** ŌĆö a non-empty array always begins with a `user` message, and the empty array is the only array that does not begin with one. The binder's [compact-transcript renderer](../binder/binder-model-and-context.md#compact-transcript-format-normative) and [session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) define a turn as a `user` message plus all subsequent assistant / toolResult / custom messages up to (but not including) the next `user` message; this precondition is what makes that definition total over the array, since a leading run of non-`user` messages (an `assistant` or `toolResult` left from an interrupted prior turn, or a `custom:loom-system-note` emitted before the first user input) would otherwise belong to no turn. The empty-array case is admitted by the biconditional and renders no `Recent session context` block, per [BNDR-7i](../binder/binder-model-and-context.md#bndr-7i). Like the ordering precondition above, this guarantee is taken on a best-effort basis against the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) and is re-audited on each Pi minor bump via item (ag) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect a leading non-`user` prefix from the type surface alone.

<a id="extensioncontext-interface"></a>

**`ExtensionContext` (member surface loom touches).** The runtime touches exactly the members enumerated in the signature block below; every other member of Pi's `ExtensionContext` is non-load-bearing for loom 1.0 and may be added or revised by Pi without spec changes. The canonical Pi-side declaration lives at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above); the inline shape below is the loom-load-bearing subset and MUST be re-validated against that file on each Pi minor bump:

```ts
interface ExtensionContext {
  cwd: string;                                          // working directory of the host Pi session
  signal: AbortSignal | undefined;                       // Pi-supplied turn signal; `undefined` in idle / non-turn contexts (see **Cancellation source** below)
  sessionManager: ReadonlySessionManager;                // host's session manager exposed as a read-only `Pick<SessionManager, ŌĆ”>` (members `getCwd | getSessionDir | getSessionId | getSessionFile | getLeafId | getLeafEntry | getEntry | getLabel | getBranch | getHeader | getEntries | getTree | getSessionName`); kept in lock-step with `dist/core/session-manager.d.ts`'s `ReadonlySessionManager` `Pick<>` and re-validated on each Pi minor bump. Subagent mode swaps the underlying object to the loom's spawned in-memory `SessionManager` (see per-mode override table below) but the static type narrows to the same `ReadonlySessionManager` surface in both modes ŌĆö loom code MUST NOT capture the wider `SessionManager` instance in a parallel closure to bypass this contract. The free `buildSessionContext` named export (see above) is the supported way to obtain a `SessionContext` off the read-only surface.
  modelRegistry: ModelRegistry;                          // resolves a model to a `Model<Api>`; consumed by the binder pass and by binder-model resolution (see [model-registry surface](#model-registry-pin) below)
  model: Model<any> | undefined;                         // host's currently-selected model; `undefined` arm reachable when no model is yet selected ŌĆö loom never relies on the value being defined
  ui: ExtensionUIContext;                                // transient toast surface; the only member loom calls is `ui.notify(message: string, type?: "info" | "warning" | "error"): void` (synchronous, may throw ŌĆö wrap in try/catch when called from the **System notes** fallback chain)
  isIdle(): boolean;                                     // true iff the host session has no in-flight turn; loom consults this only to choose between `pi.sendUserMessage` shapes (see **Conversation drive ŌĆö prompt mode** above)
  abort(): void;                                         // documented by Pi as "abort the current agent operation"; **the loom runtime wraps this** ŌĆö see override table below
  getContextUsage(): ContextUsage | undefined;           // aggregate session-level usage; `undefined` arm is the "not yet computable" sentinel (no throw); loom MUST NOT substitute this for per-turn token accounting (see `buildSessionContext` above)
  hasPendingMessages(): boolean;                          // observed by the slash-command handler's pre-turn idle probe; not consumed elsewhere in the loom runtime
  compact(options?: CompactOptions): void;               // host-driven compaction trigger; not invoked by loom in loom 1.0 (`await ctx.compact()` is a type-safe no-op ŌĆö `await void` resolves immediately and the completion signal is silently lost); completion / failure observed via `options.onComplete(result: CompactionResult)` / `options.onError(err: Error)`; forwarded unchanged in both modes, listed here so the touched-surface inventory is exhaustive
  shutdown(): void;                                      // documented by Pi for in-process shutdown; not invoked by loom in loom 1.0
  getSystemPrompt(): string;                             // host's effective system prompt; not consumed by loom in loom 1.0
}
```

`CompactOptions` and `CompactionResult` (referenced by the `compact` row above) are declared in `dist/core/extensions/types.d.ts` and `dist/core/compaction/compaction.d.ts` respectively in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above) as `CompactOptions = { customInstructions?: string; onComplete?: (result: CompactionResult) => void; onError?: (error: Error) => void }`; `CompactionResult` is the host-defined result payload threaded through `onComplete`. Loom does not construct either value in loom 1.0 ŌĆö both are listed only so the touched-surface inventory is closed against the cited declarations and re-validated on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below.

<a id="extensioncommandcontext-interface"></a>

**`ExtensionCommandContext` (per-handler subtype loom touches).** `ExtensionCommandContext` is the per-handler subtype Pi passes to a slash-command `handler`; it is declared at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above) as `interface ExtensionCommandContext extends ExtensionContext { ŌĆ” }`, so it inherits every member of the `ExtensionContext` signature block above. The runtime touches exactly the additional members enumerated below; every other `ExtensionCommandContext`-only member is non-load-bearing for loom 1.0 and the loom runtime MUST NOT call it. The inline shape below is the loom-load-bearing subset and MUST be re-validated against the cited file on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; a Pi minor that renames or moves the type, or removes any member loom touches, surfaces at the SDK surface-inventory test before the spec contract drifts.

```ts
interface ExtensionCommandContext extends ExtensionContext {
  waitForIdle(): Promise<void>;                          // resolves once Pi emits `agent_end` for the user session and every awaited `agent_end` listener settles; never rejects (see **Conversation drive ŌĆö prompt mode** above)
  // Members loom MUST NOT touch in loom 1.0: newSession, fork, navigateTree, switchSession, ReplacedSessionContext, ŌĆ”
}
```

Members loom MUST NOT touch in loom 1.0 (e.g. `newSession()`, `fork()`, `navigateTree()`, `switchSession()`, the `ReplacedSessionContext` surface) are deliberately omitted from the inline shape; their absence from this table is normative ŌĆö the loom runtime MUST NOT call them in loom 1.0, and a future widening (the deferred *Mid-loom user-session replacement* extension in [Future Considerations](../future-considerations/surface-extensions.md#mid-loom-user-session-replacement) is the only loom 1.0 surface that anticipates such use) is a spec-versioned change. The `waitForIdle()` member is the only `ExtensionCommandContext`-only member loom touches in loom 1.0; it is consulted on the captured slash-command handler context only (the `ExtensionCommandContext` Pi passes to the loom's slash-command `handler`), never on the synthesised tool-execution `ctx` from the **`ExtensionContext`** section above (which is typed as the wider `ExtensionContext` and does not expose `waitForIdle`).

*Per-mode override semantics.* The runtime synthesises the `ctx` it hands to `tool.execute(...)` by starting from the live host's `ExtensionContext` and overriding exactly three members. The overrides apply uniformly in both modes ŌĆö there is no member that is overridden in one mode and forwarded unchanged in the other:

| Member | Prompt mode | Subagent mode |
|---|---|---|
| `signal` | replaced with `loomAbort.signal` (always defined) | replaced with `loomAbort.signal` (always defined) |
| `sessionManager` | replaced with Pi's user-session manager (the host's, observed via the captured `ExtensionCommandContext`) | replaced with the loom's spawned in-memory `SessionManager` (the one passed to `createAgentSession({ sessionManager })` per **Conversation drive ŌĆö subagent mode** above) |
| `abort` | wrapped to call `loomAbort.abort()` rather than the host's `abort()` ŌĆö so `ctx.abort()` tears down the loom's invocation, not the user's whole Pi turn; a throw from the underlying `loomAbort.abort()` invoked by the wrapper is trapped per the [Cancellation ŌĆö Forwarding-listener throw](../cancellation.md) clause and routed through `loom/runtime/internal-error` | wrapped identically ŌĆö `ctx.abort()` tears down the subagent invocation, not the parent's turn; the same forwarding-listener-throw trap applies |

Every other member from the signature block above forwards unchanged to the live host:

- *Prompt mode.* The live host is the `ExtensionCommandContext` passed to the slash-command handler, retained for the loom's lifetime. `cwd`, `ui`, `modelRegistry`, `model`, `isIdle`, `getContextUsage`, `hasPendingMessages`, `compact`, `shutdown`, and `getSystemPrompt` all reflect the user-session view at the moment of the tool call.
- *Subagent mode.* The spawned `AgentSession` does not carry an `ExtensionCommandContext`; the runtime forwards the parent slash-command handler's `ExtensionCommandContext` for the non-overridden members (so `model`, `modelRegistry`, `cwd`, `getContextUsage`, etc. reflect the parent's view, not a per-subagent one). The `cwd` value is the parent's `ctx.cwd` verbatim ŌĆö loom 1.0 has no per-subagent frontmatter `cwd` override; widening is reserved for [Future Considerations](../future-considerations.md).

The `abort()` override applies in **both** modes because no spec-supported caller of `ctx.abort()` from inside loom code wants to tear down the user's turn ŌĆö the in-loom code paths that call `abort()` are tools acting on behalf of the loom, whose authoritative cancellation signal is `loomAbort`. The contract pinned by this paragraph ("`ctx.abort()` aborts the loom's invocation and not the parent's turn") under the spec's "Tool execution from loom code" surface holds in both prompt and subagent modes.

<a id="tool-execution-from-loom-code"></a>

**Tool execution from loom code.** Code-side `<name>(args)` calls invoke the Pi tool's `execute(toolCallId, params, signal, onUpdate, ctx)` directly:

- `toolCallId` is a synthesised UUID prefixed `loom-direct:`.
- `params` is the loom value lowered to JSON (wire names applied).
- `signal` is the loom's current `loomAbort.signal` (always defined; see **Cancellation source** below).
- `onUpdate` is a no-op (loom 1.0 does not surface streaming partial results to loom code).
- `ctx` is the synthesised `ExtensionContext` defined under [`ExtensionContext` (member surface loom touches)](#extensioncontext-interface) above: the live host's `ExtensionContext` with `signal`, `sessionManager`, and `abort()` overridden per the per-mode-override table, and every other member forwarded to the live host.

The tool's returned `{ content, isError }` envelope shape ŌĆö `content: ({ type: "text"; text: string } | ŌĆ”)[]` and `isError: boolean` ŌĆö is declared on the tool-result type family in `dist/core/extensions/types.d.ts` of `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above; the `isError` field appears on the relevant tool-result type declarations in that file). The envelope shape is the contract every code-driven tool call's outcome routing depends on; the SDK surface-inventory test asserts the field set on the cited type so a Pi minor that renames `isError`, drops the `text`-block discriminator, or widens `content` to a non-array shape surfaces at the build before the spec contract drifts. The envelope is lowered to the loom 1.0 string return value as follows. The lowering procedure inspects the resolved envelope and any throw raised during that inspection (the resolved value is not an object, `content` is not iterable, an entry is missing `type` / `text`, or any other shape violation that causes a property access or `.filter` / `.map` call to throw) is caught by the runtime and routed to `loom/runtime/internal-error` with `details.kind = "tool-return-shape"` (carrying `tool_name` and the `details.shape_check` discriminator token, whose closed value space is pinned normatively by that row's *Trigger* prose in [Code registry runtime](../diagnostics/code-registry-runtime.md)); the call site observes the `internal-error` routing per [Errors and Results ŌĆö Runtime panics](../errors-and-results.md), **not** an `Err(QueryError { kind: "code_tool", ... })`. The same rule applies if the lowering throws while assembling the `cause: "execution"` message described below (e.g. an `isError: true` return whose `content` array contains a non-iterable entry). The runtime filters `content` to entries with `type === "text"`, joins their `.text` values with `"\n"` (single line-feed; no separator before the first or after the last block), and returns the joined string. Non-text content blocks (images, resource references) are discarded during lowering; the loom-level observable is exhausted by the resulting `Ok("")` value. The runtime MUST NOT emit a `RuntimeEvent`, `loom-system-note`, or any other normative diagnostic on the discard path ŌĆö non-text discard is not a `QueryError` and is not a member of the always-log set. Implementations MAY emit non-normative debug-channel logs for operator visibility. Widening to a richer return shape (preserving non-text blocks) is reserved for [Future Considerations](../future-considerations.md). An empty result string ŌĆö either `content: []` or a content array with no surviving text blocks ŌĆö is a legal `Ok("")` value when `!isError`. When `isError`, the runtime returns `Err(QueryError { kind: "code_tool", cause: "execution", message: <m>, tool_name, ... })` (this value is also referred to as `CodeToolError`; the two names denote the same `QueryError` variant ŌĆö the one whose `kind` discriminator is `"code_tool"` ŌĆö see [Errors and Results ŌĆö `CodeToolError`](../errors-and-results/queryerror-variants.md#code-side-tool-call-variant)) where `<m>` is the same filtered/joined text encoded as UTF-8 and truncated so that the resulting byte length is at most 4096 bytes. Truncation MUST cut on a Unicode code-point boundary: every code point in the output is represented by all of its UTF-8 bytes, and no bytes of a partial code point appear. When a code point would straddle the 4096-byte limit, that code point is dropped entirely; the resulting message MAY therefore be shorter than 4096 bytes by up to three bytes. Worked example: a filtered/joined text whose first 4095 bytes are ASCII followed by a 3-byte UTF-8 code point (e.g. U+2026 `ŌĆ”`) MUST truncate to 4095 bytes, because including the 3-byte code point would yield 4098 bytes. When no text survives the filter, `<m>` is the fixed string `"tool reported an error with no text content"`. When `cause: "execution"` originates from an `execute()` throw rather than `isError: true`, `<m>` is the thrown value coerced to the [underlying-error string](../diagnostics/placeholder-rendering-b.md#underlying-error-coercion) (so a non-Error throw does not yield `undefined` or a synchronous `TypeError`) and then truncated under the same 4096-byte rule. Both forms share the `"execution"` cause; `execute()`-throw vs `isError: true` are distinguished by the `message` source, not by a new `cause` (a future revision MAY widen `cause` additively if the distinction needs to surface).

*Outcome routing summary.* The complete loom 1.0 routing of an `execute()` outcome at this boundary is fixed: a clean `{ content, isError: false }` lowers to `Ok(<filtered/joined text>)` (possibly `Ok("")`); a clean `{ content, isError: true }` lowers to `Err(CodeToolError { cause: "execution", ... })`; an `execute()` throw lowers to `Err(CodeToolError { cause: "execution", message: <underlying-error string truncated>, ... })`; a `loomAbort.signal`-aborted call lowers to `Err(CodeToolError { cause: "cancelled", ... })` per the cancellation surfacing rule in [Cancellation ŌĆö Surfacing](../cancellation.md); a non-conforming envelope (anything that throws during the lowering above) routes to `loom/runtime/internal-error` with `details.kind = "tool-return-shape"` and is **not** observable as a `CodeToolError`; a non-settling Promise blocks the call's `await` indefinitely until cancellation fires, at which point the cancelled path applies (loom 1.0 makes no internal timeout attempt, consistent with [`spec.md` ŌĆö Hard ceilings, NOCEIL-1](../../spec/overview-and-orientation.md#hard-runtime-ceilings); the deferred per-call timeout extension lands additively on the open-struct seam in [Tool Calls](../tool-calls.md)); a settlement that arrives after the tool-call checkpoint has already surfaced `cause: "cancelled"` is discarded per [Cancellation ŌĆö late-settlement discard](../cancellation.md#cncl-1) ([CNCL-1](../cancellation.md#cncl-1) no-rebind, [CNCL-2](../cancellation.md#cncl-2) no-second-`Err`, [CNCL-3](../cancellation.md#cncl-3) no-second-`RuntimeEvent`) ŌĆö the runtime MUST keep a swallowing handler attached to the underlying `execute()` Promise (and, symmetrically, to the underlying `@`-query provider Promise and the `invoke` callee's top-level execution Promise) for the lifetime of that Promise, installed at construction time before the first microtask boundary, so that a late rejection on the discarded path does **not** surface as a Node `unhandledRejection` process event; the discard is total (no `Err` rebinds the call site, no `RuntimeEvent` reaches the always-log channel, no diagnostic of any severity is emitted, and the discarded value is not traversed). The `CodeToolError.cause` enum (closed at `validation` / `execution` / `cancelled` / `unknown_tool` per [Errors and Results ŌĆö `CodeToolError`](../errors-and-results/queryerror-variants.md#queryerror-variants)) is the contract surface for loom authors; the non-conforming-shape and non-settling-Promise dispositions are deliberately routed off it.

<a id="agentsession-interface"></a>

**`AgentSession` (member surface loom touches).** Subagent mode touches exactly the members enumerated in the shape below on the `AgentSession` handle returned by `createAgentSession(...)` (see **Conversation drive ŌĆö subagent mode** above); every other member of Pi's `AgentSession` is non-load-bearing for loom 1.0 and may be added or revised by Pi without spec changes. The canonical Pi-side declaration lives at `dist/core/agent-session.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above); the inline shape below is the loom-load-bearing subset and MUST be re-validated against that file on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below:

```ts
interface AgentSession {
  abort(): Promise<void>;                                // cancellation primitive invoked from the one-shot `loomAbort.signal` listener; resolves once the agent is idle (see **Subagent session lifecycle** above) ŌĆö additionally factory-probed (see below)
  dispose(): void;                                       // mandatory teardown run in the per-invocation `finally`; removes all listeners and disconnects from the underlying agent (see **Subagent session lifecycle** above)
  sendUserMessage(text: string): Promise<void>;          // subagent-mode query driver; returns `Promise<void>` ŌĆö unlike `pi.sendUserMessage`'s `void` shape (see **Conversation drive ŌĆö prompt mode** above)
  subscribe(listener: (event: AgentSessionEvent) => void): () => void;  // session-local event subscription scoped to this `AgentSession`; returns an unsubscribe function the runtime calls before resolving each query
}
```

`abort()` is the only member of this subset additionally probed at factory time (`typeof AgentSession.prototype.abort === "function"`, per [Step 0 (c)](./capability-probe.md#entry-capability-probe) above and the [cancellation-propagation capability](./capability-inventory-items.md#sdk-cap-cancellation-propagation)), so a Pi minor that renames or drops it surfaces as `loom/load/host-incompatible` rather than as a runtime-time `TypeError`. `dispose`, `sendUserMessage`, and `subscribe` are **not** factory-probed and are not among the seven named SDK capabilities the [literal-read assertion](./version-bump-step2.md#bump-step-2-positive) enumerates; their re-validation against the cited declaration is carried by item (n) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below.

`AgentSessionEvent` (the `subscribe` listener's event type) is declared in the same `dist/core/agent-session.d.ts`; its terminal `agent_end` variant and that variant's `messages: AgentMessage[]` field are pinned under [`SessionContext` and the `.messages` element shape](#messages-chronological-order-presupposition) above (surface (2) there) and consumed per [subagent-mode trailing-turn extraction](./subagent.md#subagent-untyped-query-ok-extraction). The `AgentSession.messages` getter is **not** loom-load-bearing: the subagent-mode `Ok(string)` extraction reads the terminal `agent_end` event's `messages` array and never the handle's `messages` getter (see the cited extraction), so a Pi change to that getter is out of scope for loom 1.0 and its absence from the subset above is normative. The handle's own satellite types ŌĆö `CreateAgentSessionOptions`, `ResourceLoader`, and `SessionManager` passed at spawn ŌĆö are pinned at their own owning sites under **Conversation drive ŌĆö subagent mode** above (the `ResourceLoader` member surface and the `SessionManager.inMemory` factory at [spawn-block satellite type pins](./subagent.md#subagent-spawn-satellite-types)), not here.

<a id="cancellation-source"></a>

**Cancellation source.** Pi's delivery of an `AbortSignal` at the two extension entry points loom uses (`ctx.signal` and `tool.execute`'s `signal` parameter) is the SDK prerequisite this section consumes; see **Host prerequisites** above. The two entry-point declarations live in `dist/core/extensions/types.d.ts` of `@earendil-works/pi-coding-agent` at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin): `ExtensionContext.signal: AbortSignal | undefined` (the slash-command handler's turn signal, with the JSDoc verbatim *ŌĆ£current abort signal, or undefined when the agent is not streamingŌĆØ*) and the `signal: AbortSignal | undefined` parameter of the tool `execute(...)` member of `ToolDefinition`. Subagent mode has no `AbortSignal` entry point ŌĆö `CreateAgentSessionOptions` (declared at `dist/core/sdk.d.ts` in the loom 1.0 Pi-SDK pin per **Conversation drive ŌĆö subagent mode** above) has no `signal` field ŌĆö and instead consumes the `AgentSession.abort()` capability on the spawned handle, per **Subagent session lifecycle** above. As described in [Cancellation](../cancellation.md), the loom's `AbortSignal` is `loomAbort.signal` from a runtime-owned `AbortController` constructed fresh per invocation ŌĆö **not** `ctx.signal` directly. Per the JSDoc above, `ctx.signal` is `undefined` in idle / non-turn contexts, and a slash-command handler fires from idle, so the runtime MUST NOT use `ctx.signal` as the loom's authoritative signal: doing so would silently pass `undefined` into every adapter (the synthesised tool-execution `ExtensionContext.signal`, the child-invoke derived signal) and degrade every cancellation path to a no-op. Instead, the runtime forwards Pi's turn-side `ctx.signal` (when defined and observed inside one of the runtime's `tool_call` / `tool_result` / `message_update` / `turn_end` / `agent_end` handlers) into `loomAbort.abort()`, and likewise forwards the `signal` parameter of a tool-exposed loom's `execute(...)` into `loomAbort.abort()` via a one-shot listener registered at entry. All downstream queries, tool calls, and child invokes derive from `loomAbort.signal`. Because `loomAbort.signal` is always defined, the optional-chaining defence (`signal?.aborted`) used by tool adapters is no longer load-bearing inside the runtime; the runtime guarantees a defined signal to every adapter it calls. Subagent mode propagates cancellation through the one-shot `loomAbort.signal` listener defined in **Subagent session lifecycle** above: when `loomAbort` fires, the listener calls `AgentSession.abort()` on the spawned handle, which tears down the in-flight provider call. Forwarding listeners ŌĆö including the subagent `abort()` listener ŌĆö are torn down in the same `finally` block that disposes the subagent session (see **Subagent session lifecycle** above). The `reason` value forwarded into `loomAbort.abort(reason)` for each forwarding path ŌĆö including the runtime-synthesised values for the `agent_end`-driven and `session_shutdown`-driven triggers, which have no source `AbortSignal` ŌĆö is pinned by [Cancellation ŌĆö Abort-reason propagation](../cancellation.md); the rule applies uniformly across this paragraph's three forwarding shapes.

<a id="pi-slash-handler-promise-lifecycle-presupposition"></a>

**Pi-side slash-handler promise lifecycle (consumption posture).** The runtime side of the cancellation chain pinned above and in [`spec.md` ŌĆö Session model ŌĆö SM-3b session-swap semantics](../../spec/session-model-and-appendix.md#sm-3b-session-swap-semantics) ŌĆö `ctx.signal` triggers `loomAbort.abort(reason)`, the symmetric direction unblocks `await ctx.waitForIdle()`, and the `session_shutdown` handler awaits `Promise.allSettled(activeInvocations.map(inv => inv.disposeBarrier))` ŌĆö is load-bearing only under six loom-side presuppositions about how Pi schedules and drives to completion the slash-command handler the runtime registers via [`sdk-cap-slash-command-registration`](./capability-inventory-items.md#sdk-cap-slash-command-registration), and about how Pi delivers the turn-lifecycle events the runtime's cancellation-forwarding handlers consume. The runtime records them here as a loom-side consumption posture rather than as Pi-side guarantees, so a future Pi change in this area surfaces against a named presupposition rather than passing silently: (i) the loom presupposes Pi awaits the slash-handler's returned `Promise` for the full invocation, including any time after `ctx.signal` fires its abort (i.e. the handler's return is not treated as fire-and-forget once the signal aborts); (ii) the loom presupposes Pi imposes no internal deadline on the handler's `Promise` settlement (no force-resolve, abandon, or detach of the in-flight handler after `ctx.signal` aborts or otherwise); (iii) the loom presupposes `ctx.signal` is Pi's only out-of-band interaction with the in-flight handler (no parallel rejection, kill switch, or other channel by which Pi unilaterally terminates the handler outside the `AbortSignal` surface); and (iv) the loom presupposes Pi *eventually* delivers a terminal `agent_end` (`willRetry: false`) for the driven turn ŌĆö settling `await ctx.waitForIdle()` in prompt mode (and resolving the session-local terminal-`agent_end` subscription the driver awaits in subagent mode per **Conversation drive ŌĆö subagent mode** above) ŌĆö so a never-terminating turn, or a future Pi retry-cap that abandons the turn without a terminal `agent_end`, would hang the loom invocation until the cancellation path fires (`ctx.waitForIdle()` has no internal deadline per **Conversation drive ŌĆö prompt mode** above); (v) the loom presupposes Pi delivers each of the five turn-lifecycle events the runtime's cancellation-forwarding handlers subscribe to ŌĆö `tool_call`, `tool_result`, `message_update`, `turn_end`, and `agent_end` ŌĆö to the subscribed extension during the active user turn for which `ctx.signal` was minted, and that `ctx.signal.aborted` / `ctx.signal.reason` read from inside each of those handlers reflects the latest abort state for that turn (per the forwarding paragraph under **Cancellation source** above, the runtime forwards `ctx.signal` into `loomAbort.abort(reason)` from inside these handlers, so a Pi minor that renames, removes, gates, stops delivering, or changes the per-handler `ctx.signal` semantics of any of them would silently degrade the Esc-during-`@`-query forwarding path to a no-op); and (vi) the loom presupposes Pi flips `loomAbort.signal.aborted` within a single macrotask ŌĆö i.e. the abort runs synchronously inside one Pi-dispatched event handler rather than being deferred across multiple macrotasks ŌĆö so the single macrotask turn the loop-iteration checkpoint yields before its signal-check (per [Cancellation ŌĆö Granularity](../cancellation.md) and the [`Checkpoint` seam](./host-interfaces-services.md#checkpoint-seam) `loop-iter` production wiring) is sufficient for a pending abort to land before a compute-bound loop's next iteration, so a future Pi that deferred the flip across multiple macrotasks could let an Esc during a compute-bound loop be missed at the next iteration without surfacing against any named presupposition. The six presuppositions are taken on a best-effort basis against the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) on `@earendil-works/pi-coding-agent` from **Host prerequisites** above and are not pinned by any typed contract on `ExtensionAPI`, `ExtensionContext`, or `ExtensionCommandContext`; the SDK capability inventory's [cancellation-propagation entry](./capability-inventory-items.md#sdk-cap-cancellation-propagation) covers only that Pi *supplies* the `AbortSignal` at the two extension entry points and does not extend to the handler-promise's settle-time, deadline, out-of-band-interaction, turn-liveness, turn-lifecycle-event-delivery, or abort-flip-timing surface. The [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below carries the audit steps that re-check these presuppositions against the candidate Pi minor on every bump: item (g) under step 1 covers (i)ŌĆō(iii), item (j) covers the (iv) turn-liveness presupposition, item (v) covers the (v) turn-lifecycle-event-delivery presupposition, and item (ad) covers the (vi) single-macrotask abort-flip presupposition.
