# Session shutdown semantics

   <a id="session-shutdown-semantics"></a>

   *Session-swap behaviour for in-flight invocations.* The five-sub-step sequence below defines the abort-and-await sequence; this paragraph closes the three obligations the sequence does not by itself state — per-invocation operator visibility on the cancellation path, partial-append fate during teardown, and the `invoke`-parent observation rule.

   The handler runs in fixed order regardless of `event.reason`:

   1. **Stop accepting new work.** <a id="substep-1-drain-flag"></a>Mark the extension-scoped `LoomRegistry` as drained <a id="substep-1-init-drain-state-tag"></a>and initialise its drain-state tag to `"shutting-down"`. <a id="substep-1-act-order"></a>The two acts run in fixed normative order — `LoomRegistry.drain()` first, then `LoomRegistry.initDrainStateTag()` — so downstream sites that key on which act threw and which succeeded (the **Per-step isolation** paragraph below, which routes the two throws to distinct `(code, details.step, details.call)` dedup buckets; the [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause's enumeration of the prior-state combinations a follow-up `session_shutdown` may observe; the `LoomRegistry.initDrainStateTag` method-contract clause below that pins it as called *immediately after* `LoomRegistry.drain`; and any acceptance criterion that distinguishes between the first and second sub-step-1 act) can refer to "the second sub-step-1 act" unambiguously; the per-call **Per-step isolation** rule continues to apply independently to each of the two calls regardless of the order pin. <a id="substep-1-shutting-down-arm"></a>On entry into a drained registry the slash-command `handler` registered in step 3 returns a `loom-system-note` (`display: true`) — without dispatching the loom — carrying the cancelled-binder-style content `"loom /<name>: extension shutting down"` shared with binder cancellation; this is the sole non-dispatch drain-state note in loom 1.0 (the session-only degraded-state branch and its `"degraded; /reload to recover"` note are retired under the `governed-by-rebind` resolution — see the [Session-swap fail-fast tripwire](./session-only-degraded-state.md#session-swap-fail-fast-tripwire)). <a id="substep-1-unset-tag-fallback"></a>The slash-handler reader MUST also treat an *unset* drain-state tag (a runtime *state* of the field, distinct from the single `"shutting-down"` *value* the tag may hold in loom 1.0.0 — the runtime state of the field may additionally be unset, reachable when sub-step 1's `LoomRegistry.initDrainStateTag` per-call site threw and was caught by the **Per-step isolation** paragraph below, leaving the registry drained but tag-less) as the `"shutting-down"` arm and emit the same template, so a tag-init throw degrades safely to the cancelled-binder-style content rather than producing undefined slash-handler short-circuit behaviour. The tag *value* set is closed at the single literal `"shutting-down"` in loom 1.0.0 (no other literal value may be written to the field without a spec edit); the unset runtime state called out above is an implementation-side artefact of the **Per-step isolation** swallow rule, not a second literal value, and the slash-handler reader's unset → `"shutting-down"` mapping above ensures the closed value set is preserved in the observable behaviour of the slash note.
   2. **Cancel in-flight invocations.** For every entry in the **`ActiveInvocationRegistry`** (introduced below), first stamp the handler-scoped captured `event.reason` string (per the **Unknown-reason rule** above — the closed-set member, the `String(event.reason)` unknown-reason coercion, the throwing-access `"<unreadable>"`, or the snapshot-failure `"<unreadable>"`) onto the entry's `shutdownReason` field, then call `loomAbort.abort(reason)` where `reason` is the synthesised teardown value pinned by [Cancellation — Abort-reason propagation](../cancellation.md). The stamp MUST happen before `loomAbort.abort()` on the same entry so the per-invocation `finally` (which is what reads `entry.shutdownReason` per the **Per-invocation operator visibility (clean-cancel path)** rule below) observes a populated field by the time the abort propagates and the `finally` runs; the per-entry `try`/`catch` (per the registry contract below) wraps both the stamp and the `abort()` call — on a stamp throw (frozen entry, proxied entry, write-rejecting setter, OOM during property assignment) the per-entry `catch` arm proceeds to the next entry (the `abort()` for this entry is skipped, since the stamp failed) so a per-entry write failure cannot escape sub-step 2's loop and defeat the no-throw-escapes-the-handler invariant; on an `abort()` throw, behaviour is unchanged from the registry-contract rule below. No diagnostic is emitted on a stamp-throw catch, symmetric to the existing `abort()`-throw rule. The `abort()` call propagates through the [`Checkpoint` seam](./host-interfaces-services.md#checkpoint-seam) and, in subagent mode, through the one-shot `loomAbort.signal` listener that the runtime registered at spawn time and that calls `AgentSession.abort()` on the spawned session — per [Cancellation — Forwarding into `loomAbort`](../cancellation.md). The listener is detached in the same per-invocation `finally` that calls `AgentSession.dispose()` (see **Subagent session lifecycle** below).
   3. **Await subagent disposal.** `await Promise.allSettled(activeInvocations.map(inv => inv.disposeBarrier))`, where `disposeBarrier` is the promise that the **Subagent session lifecycle** `finally` block below settles after `AgentSession.dispose()` returns (subagent mode) or settles immediately (prompt mode). The await is bounded by the constant `SHUTDOWN_AWAIT_CAP_MS = 2000` measured against the runtime's injected [`Clock`](./host-interfaces-services.md#clock--fakeclock-interface) seam: at handler entry the runtime captures `deadline = Clock.now() + SHUTDOWN_AWAIT_CAP_MS`, and the await resolves on the earlier of `Promise.allSettled(...)` settling or `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, ...)` firing. The deadline is absolute, not a refreshing slide — a slow `loomAbort.abort()` propagation in sub-step 2 does not extend the await. The pending `Clock.setTimeout` handle is cleared on the success path so a reload that completes early does not leak a timer onto the about-to-be-invalidated runtime. On timeout the runtime emits one `loom/runtime/reload-teardown-timeout` diagnostic naming the still-in-flight invocations and proceeds to step 4. The `SHUTDOWN_AWAIT_CAP_MS` constant lives in the same source-of-truth pinned-constants block as the Step 0 capability-probe constants so the build-time literal-read assertions can assert it.
   4. **Close watchers.** Call `discoveryWatcher.close()` (the chokidar watcher established in step 5 above) and `settingsWatcher.close()` (the settings-file watcher per [Discovery — Settings file reads](../discovery/package-and-settings.md#settings-file-reads)); cancel any pending debounce timer handles via `Clock.clearTimeout`.
   5. **Detach forwarding listeners.** Remove every listener the runtime attached to Pi-side `ctx.signal`, tool `execute(signal)`, and parent-`invoke` signals — the same listeners [Cancellation](../cancellation.md) requires the per-invocation `finally` to remove, duplicated here for the reload-during-invocation path where Pi tears the runtime down before that `finally` ever reaches its forwarding-listener cleanup.

   **Per-step isolation.** Sub-steps 1, 3, 4, and 5 each run inside their own `try`/`catch`; a throw from any of them MUST be caught, MUST emit exactly one `loom/host/session-shutdown-teardown-step-failed` (severity `warning`, phase `runtime` — the `(W, runtime)` shorthand used for the teardown-handler diagnostics throughout these PIC pages is defined by the column legend in [Diagnostic shape — column legend](../diagnostics/diagnostic-shape.md#column-legend)) diagnostic via `console.error` with `details: { step: 1 | 3 | 4 | 5, call: <static call-site label>, error: <error.message> }`, and MUST NOT prevent the remaining sub-steps from running. Within sub-step 1 the rule applies independently to each of `LoomRegistry.drain` (the drain-flag set) and `LoomRegistry.initDrainStateTag` (the `"shutting-down"` tag-init write — the two acts sub-step 1 now performs per the prose above, two potential emissions, one per failing call); a drain-flag throw and a tag-init throw therefore occupy distinct `(code, details.step, details.call)` dedup buckets in operator logs and a future addition of a third sub-step-1 act MUST be added here on the same footing rather than collapsed into one of the existing labels. Within sub-step 3 the rule applies independently to each of the three bounded-await `Clock` calls — the deadline capture `Clock.now()`, the cap-arming `Clock.setTimeout(SHUTDOWN_AWAIT_CAP_MS, ...)`, and the success-path `Clock.clearTimeout(...)` (three potential emissions, one per failing call); a throw from any of the three MUST be caught, MUST emit exactly one `loom/host/session-shutdown-teardown-step-failed` with `details.step: 3`, and MUST NOT prevent sub-steps 4 and 5 from running. When the cap cannot be armed — the deadline capture `Clock.now()` or the cap-arming `Clock.setTimeout(...)` throws — the handler MUST NOT enter the bounded await unbounded: it skips the `Promise.allSettled(...)` await for this teardown and proceeds directly to sub-step 4, so a `Clock`-seam throw degrades to a skipped await rather than an unbounded one (a success-path `Clock.clearTimeout(...)` throw fires after the await has already resolved, so no degradation is needed beyond emitting and continuing). Within sub-step 4 the rule applies independently to each of `discoveryWatcher.close()`, `settingsWatcher.close()`, and the debounce-handle `Clock.clearTimeout(...)` call (three potential emissions, one per failing call). Within sub-step 5 the rule applies independently to each enumerated listener-detach call. Sub-step 2 retains its existing per-entry isolation rule (per-entry `try`/`catch` around `loomAbort.abort()`) and does not emit `teardown-step-failed`; sub-step 3's `Promise.allSettled` over `disposeBarrier`s likewise retains its per-entry settle semantics without emitting `teardown-step-failed` — only sub-step 3's three bounded-await `Clock` calls above emit it. The static call-site label in `details.call` is a fixed string per call site, drawn from a **closed normative set per `(code, details.step)`** so operator dedup on `(code, details.step, details.call)` is meaningful both across runs and across independent implementations of this contract — the labels are wire contract, not implementation-chosen stable-per-impl strings. The closed set is: sub-step 1: `"loomRegistry.drain"`, `"loomRegistry.initDrainStateTag"`; sub-step 3: `"Clock.now()"`, `"Clock.setTimeout(awaitCap)"`, `"Clock.clearTimeout(awaitCap)"`; sub-step 4: `"discoveryWatcher.close"`, `"settingsWatcher.close"`, `"Clock.clearTimeout(debounce)"`; sub-step 5: `"ctx.signal.removeEventListener"`, `"toolSignal.removeEventListener"`, `"parentInvokeSignal.removeEventListener"` (one label per inbound Pi-side signal source sub-step 5 enumerates by name — Pi-side `ctx.signal`, the tool-exposed adapter's `signal` parameter, and the `invoke`-parent signal respectively). Adding a new call site under any of these sub-steps requires a spec edit to extend this closed set in the same commit; an implementation MUST NOT introduce a `details.call` value outside this enumeration. When the captured throw is not an `Error` instance, `details.error` is `String(error)`, falling back to the literal string `"<unreadable>"` if that coercion itself throws — symmetric with the `details.observed` convention pinned by the **Unknown-reason rule** above. The `console.error` call this paragraph mandates is itself wrapped per the **Diagnostic-emission isolation** rule defined under *Edge cases* below; a throw from `console.error` MUST be swallowed and MUST NOT prevent the remaining sub-steps from running. The new code is **not** added to the always-log set defined under **Runtime event channel** below, matching the existing `reload-teardown-timeout` and `session-shutdown-reason-unknown` exclusions.

   *Reason-set pin (co-located with the consuming handler).* The closed set of reason values Pi delivers to the runtime's teardown handler on session-end matches the `SessionShutdownEvent['reason']` union declared at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)); the closed reason set itself is owned by sub-step 4's [Unknown-reason rule](./unknown-reason-rule.md#unknown-reason-rule) and is pinned at build time by the bidirectional `SessionShutdownEvent['reason']` type-equality assertion described at [Pi version bump procedure — step 5](./version-bump-intro.md#pi-version-bump-procedure). The single-active-session binding under which this reason set is delivered is [PIC-7 *Session-binding contract*](./host-prerequisites.md#pic-7).

   - **Per-invocation operator visibility (clean-cancel path).** For each `ActiveInvocationRegistry` entry whose `disposeBarrier` settles (whether inside the `SHUTDOWN_AWAIT_CAP_MS` window of sub-step 3 — i.e., the invocation cancelled cleanly within the bounded await — OR, on the [*Accepted loom 1.0 residual gap — sub-step-2 stamp-throw case*](./session-only-degraded-state.md#substep-2-stamp-throw-residual-gap) above, after the deadline via a downstream cancellation per the EXCEPT clause below) — the cancellation path inside that invocation's own `finally` block emits exactly one `loom/runtime/cancelled-by-session-shutdown` (E, runtime) note on `loom-system-note` with `display: false`, `details.event.reason` carrying the handler-captured `event.reason` string typed as `"quit" | "reload" | "new" | "resume" | "fork" | string`: a closed-set member when Pi delivered one of the five literals, otherwise the `String(event.reason)` coercion captured by the **Unknown-reason rule** above (or the literal string `"<unreadable>"` when the property-access read threw, or the same `"<unreadable>"` sentinel — overloaded per the **Unknown-reason rule**'s captured-value enumeration above — when the snapshot-side read failed before sub-step 1's `event.reason` read was reached and the handler-scoped variable retained its `"<unreadable>"` pre-initialisation; on this snapshot-failure path the same-string reuse with `details.observed` does not apply because the co-emitted teardown diagnostic is `loom/host/session-shutdown-pinned-constant-unreadable` carrying `details.failure` rather than `loom/host/session-shutdown-reason-unknown` carrying `details.observed`). The per-invocation `finally` reads this string from the entry's `shutdownReason` field (stamped by sub-step 2 of the `session_shutdown` handler above immediately before that entry's `loomAbort.abort()` call, per all four captured-value cases enumerated by the **Unknown-reason rule**) rather than re-reading `event.reason` — the `event.reason` read is handler-scoped and the `finally` runs in a different async context, so `entry.shutdownReason` is the spec-pinned channel by which all four captured-value cases reach this emission site. On the post-deadline arm of the widened opening precondition above (the [*Accepted loom 1.0 residual gap — sub-step-2 stamp-throw case*](./session-only-degraded-state.md#substep-2-stamp-throw-residual-gap)), the per-invocation `finally` STILL reads `entry.shutdownReason` by the same channel and emits the row; on that arm the field carries the literal `"<unreadable>"` sentinel per the residual-gap paragraph (sub-step 2's stamp threw before populating it, and the residual-gap paragraph pins payload-construction substitution of the sentinel for the otherwise-unset field), so the type pinned for `details.event.reason` (`"quit" | "reload" | "new" | "resume" | "fork" | string`) is preserved on the post-deadline arm on the same footing as the four `Unknown-reason rule` captured-value cases above. The wider `string` arm is the unknown-reason path; the value is always a string. The handler-captured string is the same single coerced value the **Unknown-reason rule** populates `details.observed` with — both diagnostics surface byte-identical reason strings for the same shutdown event, and `details.event.loom` carrying the invocation's canonical-key name (post-`pi.looms` mapping, post-`as` rename, no leading `/`) per the [`ActiveInvocationRegistry`](./active-invocation-registry.md#active-invocation-registry) `loom`-field source-of-truth above. The same `finally` also populates `details.event.invocation_id` by reading the registry entry's `invocationId` field (the same channel by which `details.event.loom` is read, per the [`ActiveInvocationRegistry`](./active-invocation-registry.md#active-invocation-registry) registry-insertion-site source-of-truth above), rendered as the per-invocation UUID in canonical lowercase 8-4-4-4-12 hex form per the [§7 `<invocation-id>` placeholder convention](../diagnostics/placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders); the field is not re-derived at the emission site, so concurrent siblings of the same loom carry distinct `details.event.invocation_id` values on the operator-visible row. The note is emitted from the invocation's own teardown — not from the `session_shutdown` handler itself — so the "`sendSystemNote` MUST NOT be invoked from the teardown handler" edge-case bullet above does not apply to it; the note routes through the standard `sendSystemNote` fallback chain (`pi.sendMessage` → `ctx.ui.notify` → `loom/runtime/system-note-delivery-failed` → `console.error`) defined under **System notes** below, which degrades through `ctx.ui.notify` and ultimately `console.error` if `ExtensionRuntime.invalidate(...)` has already invalidated `pi.sendMessage` by emission time. The chain's terminal `console.error` is wrapped per the [**Diagnostic-emission isolation**](./diagnostic-emission-isolation.md#diagnostic-emission-isolation) rule above: a throw from that terminal `console.error` MUST be swallowed and MUST NOT propagate out of the per-invocation `finally`. An invocation either emits `cancelled-by-session-shutdown` (cleanly cancelled within the window) or contributes to the single `loom/runtime/reload-teardown-timeout` `<list>` of sub-step 3 (still in flight at the deadline), never both — the per-invocation note is the operator-visibility counterpart to the timeout diagnostic — EXCEPT on the [*Accepted loom 1.0 residual gap — sub-step-2 stamp-throw case*](./session-only-degraded-state.md#substep-2-stamp-throw-residual-gap) above, where an affected invocation MAY both contribute to the `reload-teardown-timeout` `<list>` (because sub-step 2's `abort()` was skipped) AND later emit `cancelled-by-session-shutdown` from this rule (if its per-invocation `finally` settles via a downstream cancellation after the deadline); the [`loom/runtime/cancelled-by-session-shutdown` row's mutual-exclusion clause](../diagnostics/code-registry-runtime.md#cancelled-by-session-shutdown-mutual-exclusion) restates this carve-out and pins the operator-tooling consumer guidance for the dual appearance. The `cancelled-by-session-shutdown` code is **not** in the always-log set defined under **Runtime event channel** below; the exclusion mirrors the existing `cancelled` exclusion (operator-initiated, self-explanatory in context) and so the per-invocation note does not produce a `details: { event: RuntimeEvent }` group-A emission in addition to itself.

   - **Partial-append fate during teardown.** Conversation turns the runtime had already appended into the user session (prompt mode) or the spawned subagent session (subagent mode) before sub-step 2's `loomAbort.abort()` fired remain in whatever conversation Pi assigns them to under the swap's `event.reason`; the runtime makes no rollback, migration, or compensation attempt. This is the `event.reason`-scoped restatement of the [Errors and Results — Partial-append contract](../errors-and-results/error-model.md#partial-append-contract): `?`-propagation, panic, and steady-state cancellation already preserve appended turns, and a `session_shutdown`-driven cancellation preserves them on the same rule. The outgoing user session (when Pi tears it down on `reason: "new" | "resume" | "fork"`) carries the partial appends Pi assigns to it under the swap; the loom runtime makes no attempt to discard them, migrate them to the incoming session, or surface a manifest of what was appended. Authors who need teardown-safe append semantics must use [Errors and Results — No rollback](../errors-and-results/error-model.md#no-rollback)'s steady-state author obligations (idempotency, compensation in `match` arms before re-`Err`-ing).

   - **`invoke`-parent observation rule.** Parent and child invocations are aborted in the same iteration of sub-step 2: the registry is iterated once and `loomAbort.abort()` is called on every entry, parents and children alike. The parent's `?` / `match` arms covering the child's `Err(QueryError { kind: "cancelled" })` (or `kind: "invoke_callee"` wrapping it, per [Cancellation — Surfacing](../cancellation.md)) do **not** run inside the bounded-await window of sub-step 3: the parent's own `loomAbort` has already fired in the same iteration, its next [`Checkpoint` seam](./host-interfaces-services.md#checkpoint-seam) await observes the abort, and the parent unwinds without re-entering interpreter dispatch. The parent's `disposeBarrier` settles independently of whether its child surfaced an `Err` to it — the registry treats every entry symmetrically. The downward-only propagation rule of [Cancellation — Propagation](../cancellation.md) is not contradicted: parent and child are not propagating into each other, they are both aborted by the same external `session_shutdown` trigger, and the parent's `Err`-handling arms remain inert during teardown for the same checkpoint reason that any cancelled invocation's post-abort code is inert.

   <a id="factory-ordering-pin"></a>**Factory-ordering pin.** The `session_shutdown` handler MUST be subscribed (step 4 above) only after the `LoomRegistry` and watcher handles the handler reads are constructed (the [watcher / hot-reload registration step](./registration-steps.md#watcher-hot-reload-registration) — step 5 of the **Extension entry point** sequence, above). The implementation MAY reorder steps 4 and 5 internally provided that subscription happens after registry/watcher construction; a `session_shutdown` MUST NOT be reachable against a partially-constructed extension state, so the closure-captured `LoomRegistry` handle the handler reads at handler entry (per step (I) of the *Handler control-flow ordering* clause in the [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause above) is never `undefined` and `LoomRegistry.readDrainState()` cannot throw a `TypeError` against a missing dispatch in the gap between step 4 returning and registry/watcher construction completing.

   The watcher / hot-reload lifecycle this ordering pin protects — the chokidar file watcher, the 250 ms debounce, the build-aside-then-publish registry swap, the in-flight-invocation rule, and the structural-change note — is specified at the [watcher / hot-reload registration step](./registration-steps.md#watcher-hot-reload-registration) (step 5 of the **Extension entry point** sequence, above), not under this heading.
