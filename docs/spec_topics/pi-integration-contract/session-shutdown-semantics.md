# Session shutdown semantics

   <a id="session-shutdown-semantics"></a>

   *Session-swap behaviour for in-flight invocations.* The five-sub-step sequence above defines the abort-and-await sequence; this paragraph closes the three obligations the sequence does not by itself state — per-invocation operator visibility on the cancellation path, partial-append fate during teardown, and the `invoke`-parent observation rule.

   - **Per-invocation operator visibility (clean-cancel path).** For each `ActiveInvocationRegistry` entry whose `disposeBarrier` settles (whether inside the `SHUTDOWN_AWAIT_CAP_MS` window of sub-step 3 — i.e., the invocation cancelled cleanly within the bounded await — OR, on the [*Accepted loom 1.0 residual gap — sub-step-2 stamp-throw case*](./session-only-degraded-state.md#substep-2-stamp-throw-residual-gap) above, after the deadline via a downstream cancellation per the EXCEPT clause below) — the cancellation path inside that invocation's own `finally` block emits exactly one `loom/runtime/cancelled-by-session-shutdown` (E, runtime) note on `loom-system-note` with `display: false`, `details.event.reason` carrying the handler-captured `event.reason` string typed as `"quit" | "reload" | "new" | "resume" | "fork" | string`: a closed-set member when Pi delivered one of the five literals, otherwise the `String(event.reason)` coercion captured by the **Unknown-reason rule** above (or the literal string `"<unreadable>"` when the property-access read threw, or the same `"<unreadable>"` sentinel — overloaded per the **Unknown-reason rule**'s captured-value enumeration above — when the snapshot-side read failed before sub-step 1's `event.reason` read was reached and the handler-scoped variable retained its `"<unreadable>"` pre-initialisation; on this snapshot-failure path the same-string reuse with `details.observed` does not apply because the co-emitted teardown diagnostic is `loom/host/session-shutdown-pinned-constant-unreadable` carrying `details.failure` rather than `loom/host/session-shutdown-reason-unknown` carrying `details.observed`). The per-invocation `finally` reads this string from the entry's `shutdownReason` field (stamped by sub-step 2 of the `session_shutdown` handler above immediately before that entry's `loomAbort.abort()` call, per all four captured-value cases enumerated by the **Unknown-reason rule**) rather than re-reading `event.reason` — the `event.reason` read is handler-scoped and the `finally` runs in a different async context, so `entry.shutdownReason` is the spec-pinned channel by which all four captured-value cases reach this emission site. On the post-deadline arm of the widened opening precondition above (the [*Accepted loom 1.0 residual gap — sub-step-2 stamp-throw case*](./session-only-degraded-state.md#substep-2-stamp-throw-residual-gap)), the per-invocation `finally` STILL reads `entry.shutdownReason` by the same channel and emits the row; on that arm the field carries the literal `"<unreadable>"` sentinel per the residual-gap paragraph (sub-step 2's stamp threw before populating it, and the residual-gap paragraph pins payload-construction substitution of the sentinel for the otherwise-unset field), so the type pinned for `details.event.reason` (`"quit" | "reload" | "new" | "resume" | "fork" | string`) is preserved on the post-deadline arm on the same footing as the four `Unknown-reason rule` captured-value cases above. The wider `string` arm is the unknown-reason path; the value is always a string. The handler-captured string is the same single coerced value the **Unknown-reason rule** populates `details.observed` with — both diagnostics surface byte-identical reason strings for the same shutdown event, and `details.event.loom` carrying the invocation's canonical-key name (post-`pi.looms` mapping, post-`as` rename, basename-hyphen-normalised, no leading `/`) per the [`ActiveInvocationRegistry`](./active-invocation-registry.md#active-invocation-registry) `loom`-field source-of-truth above. The same `finally` also populates `details.event.invocation_id` by reading the registry entry's `invocationId` field (the same channel by which `details.event.loom` is read, per the [`ActiveInvocationRegistry`](./active-invocation-registry.md#active-invocation-registry) registry-insertion-site source-of-truth above), rendered as the per-invocation UUID in canonical lowercase 8-4-4-4-12 hex form per the [§7 `<invocation-id>` placeholder convention](../diagnostics/placeholder-rendering-b.md#7-identifier--descriptor--and-closed-enum-placeholders); the field is not re-derived at the emission site, so concurrent siblings of the same loom carry distinct `details.event.invocation_id` values on the operator-visible row. The note is emitted from the invocation's own teardown — not from the `session_shutdown` handler itself — so the "`sendSystemNote` MUST NOT be invoked from the teardown handler" edge-case bullet above does not apply to it; the note routes through the standard `sendSystemNote` fallback chain (`pi.sendMessage` → `ctx.ui.notify` → `loom/runtime/system-note-delivery-failed` → `console.error`) defined under **System notes** below, which degrades through `ctx.ui.notify` and ultimately `console.error` if `ExtensionRuntime.invalidate(...)` has already invalidated `pi.sendMessage` by emission time. The chain's terminal `console.error` is wrapped per the [**Diagnostic-emission isolation**](./diagnostic-emission-isolation.md#diagnostic-emission-isolation) rule above: a throw from that terminal `console.error` MUST be swallowed and MUST NOT propagate out of the per-invocation `finally`. An invocation either emits `cancelled-by-session-shutdown` (cleanly cancelled within the window) or contributes to the single `loom/runtime/reload-teardown-timeout` `<list>` of sub-step 3 (still in flight at the deadline), never both — the per-invocation note is the operator-visibility counterpart to the timeout diagnostic — EXCEPT on the [*Accepted loom 1.0 residual gap — sub-step-2 stamp-throw case*](./session-only-degraded-state.md#substep-2-stamp-throw-residual-gap) above, where an affected invocation MAY both contribute to the `reload-teardown-timeout` `<list>` (because sub-step 2's `abort()` was skipped) AND later emit `cancelled-by-session-shutdown` from this rule (if its per-invocation `finally` settles via a downstream cancellation after the deadline); the [`loom/runtime/cancelled-by-session-shutdown` row's mutual-exclusion clause](../diagnostics/code-registry-runtime.md#cancelled-by-session-shutdown-mutual-exclusion) restates this carve-out and pins the operator-tooling consumer guidance for the dual appearance. The `cancelled-by-session-shutdown` code is **not** in the always-log set defined under **Runtime event channel** below; the exclusion mirrors the existing `cancelled` exclusion (operator-initiated, self-explanatory in context) and so the per-invocation note does not produce a `details: { event: RuntimeEvent }` group-A emission in addition to itself.

   - **Partial-append fate during teardown.** Conversation turns the runtime had already appended into the user session (prompt mode) or the spawned subagent session (subagent mode) before sub-step 2's `loomAbort.abort()` fired remain in whatever conversation Pi assigns them to under the swap's `event.reason`; the runtime makes no rollback, migration, or compensation attempt. This is the `event.reason`-scoped restatement of the [Errors and Results — Partial-append contract](../errors-and-results/error-model.md#partial-append-contract): `?`-propagation, panic, and steady-state cancellation already preserve appended turns, and a `session_shutdown`-driven cancellation preserves them on the same rule. The outgoing user session (when Pi tears it down on `reason: "new" | "resume" | "fork"`) carries the partial appends Pi assigns to it under the swap; the loom runtime makes no attempt to discard them, migrate them to the incoming session, or surface a manifest of what was appended. Authors who need teardown-safe append semantics must use [Errors and Results — No rollback](../errors-and-results/error-model.md#no-rollback)'s steady-state author obligations (idempotency, compensation in `match` arms before re-`Err`-ing).

   - **`invoke`-parent observation rule.** Parent and child invocations are aborted in the same iteration of sub-step 2: the registry is iterated once and `loomAbort.abort()` is called on every entry, parents and children alike. The parent's `?` / `match` arms covering the child's `Err(QueryError { kind: "cancelled" })` (or `kind: "invoke_callee_error"` wrapping it, per [Cancellation — Surfacing](../cancellation.md)) do **not** run inside the bounded-await window of sub-step 3: the parent's own `loomAbort` has already fired in the same iteration, its next [`Checkpoint` seam](./host-interfaces-core.md#checkpoint-seam) await observes the abort, and the parent unwinds without re-entering interpreter dispatch. The parent's `disposeBarrier` settles independently of whether its child surfaced an `Err` to it — the registry treats every entry symmetrically. The downward-only propagation rule of [Cancellation — Propagation](../cancellation.md) is not contradicted: parent and child are not propagating into each other, they are both aborted by the same external `session_shutdown` trigger, and the parent's `Err`-handling arms remain inert during teardown for the same checkpoint reason that any cancelled invocation's post-abort code is inert.

5. *Factory-ordering pin.* The `session_shutdown` handler MUST be subscribed (step 4 above) only after the `LoomRegistry` and watcher handles the handler reads are constructed (this step). The implementation MAY reorder steps 4 and 5 internally provided that subscription happens after registry/watcher construction; a `session_shutdown` MUST NOT be reachable against a partially-constructed extension state, so the closure-captured `LoomRegistry` handle the handler reads at handler entry (per step (I) of the *Handler control-flow ordering* clause in the [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause above) is never `undefined` and `LoomRegistry.readDrainState()` cannot throw a `TypeError` against a missing dispatch in the gap between step 4 returning and step 5 completing. Registers a chokidar file watcher over the discovered roots and maintains an internal mutable `LoomRegistry` (a `Map<slashName, parsedLoom>`) plus a `.warp` import graph. The slash-command `handler` registered in step 3 closes over the registry and dispatches by slash name on every invocation, so a registry swap takes effect on the next call without a `pi.registerCommand` re-registration. On a chokidar event for an **existing** loom or `.warp` file the watcher debounces (250 ms window, to coalesce editor `change` + `rename` bursts; the 250 ms window is measured against the runtime's injected `Clock` seam (per the runtime's DI skeleton — see [`Clock` / `FakeClock` interface](./host-interfaces-services.md#clock--fakeclock-interface) below), the seam exists to make burst-coalescing assertions deterministic and is not a tunable runtime knob), re-parses just the changed file plus every transitive `.warp` importer reached through the import graph, and atomically swaps the affected entries in `LoomRegistry` via build-aside-then-publish: the runtime re-parses the changed file plus every transitive `.warp` importer into a staging set, recompiles their AJV validators into a staging cache, and only after every step succeeds installs both into `LoomRegistry` and the validator cache in a single synchronous step. If any rebuild step throws, the staging set is discarded and the prior `LoomRegistry` snapshot and AJV validator cache remain live; the failure surfaces as a single `loom/runtime/registry-swap-failed` diagnostic (see [Diagnostics](../diagnostics.md)). Compiled AJV validators for swapped entries are dropped from the per-loom cache as part of the publish step (not before). Tools synthesised from prompt-mode `.loom` callables are re-registered through `pi.registerTool` when the swap produces a new schema slug (`pi.registerTool` is documented as supported after startup); existing cached registrations under unchanged slugs are reused. The runtime does **not** call `ctx.reload()` for content edits — doing so would tear down every Pi extension, skill, prompt, and theme in the session for a single keystroke and would also race the post-`await` runtime-deadness rule that `ctx.reload()` documents.

   **In-flight invocation rule.** The slash-handler closure registered in step 3 (which calls `LoomRegistry.readDrainState()` per the [*Drain-state-gated dispatch*](./registration-steps.md#slash-handler-registration) clause and routes through the entry table for the dispatch arm) reads the entry exactly once at handler entry; a swap mid-execution does not affect a running invocation. Slash-command dispatch is gated by the three-arm `LoomRegistry.readDrainState()` enumeration pinned in the [*Drain-state-gated dispatch*](./registration-steps.md#slash-handler-registration) clause appended to step 3 above and the [`LoomRegistry` drain-state contract](./drain-state-contract.md#loom-registry-drain-state-contract) below (including that contract's *Methods* paragraph's [*Read-failure fallback*](./drain-state-contract.md#read-failure-fallback) clause); that contract supersedes the prior single-arm `drained`-keyed short-circuit and is the sole authority on which non-dispatch system note (the `"shutting-down"` arm's legacy `"loom /<name>: extension shutting down"` template, the `"degraded-needs-reload"` arm's `"loom /<name>: extension degraded; /reload to recover"` template, or arm (a) dispatch) the handler returns. The next invocation sees the post-swap version. In-flight invocations therefore complete against the pre-swap snapshot and any tool calls or queries they have already committed remain final, consistent with [Errors and Results — No rollback](../errors-and-results.md).

   **Structural changes.** Adding a brand-new `.loom` file or removing an existing one cannot register / unregister a slash command after extension load (Pi exposes no symmetric `pi.unregisterCommand` for the removal case, and adding a new command after the extension factory has run is not a documented post-startup mutation). When the watcher observes such an event, it emits a single `loom-system-note` with `content` formatted as `loom watcher: <N> file(s) added or removed; run /reload to refresh the slash command list`, `display: true`, and the `details` payload defined under System notes. The substitution rule is fixed: only `<N>` is substituted; every other character of the template — including the literal token `file(s)` (always rendered with the parenthesised `s`, never pluralised), the literal slash-command name `/reload` (not substituted from any registered command), and the trailing `; run /reload to refresh the slash command list` clause — ships verbatim regardless of `N`. `<N>` is rendered in base 10 with no thousands separator, no leading zero, and no sign, and equals `details.structural.added.length + details.structural.removed.length` (a path that appears in both arrays — e.g., a rename observed as removal-then-addition within the same debounce window — counts twice, since the two arrays are disjoint by role and not deduplicated against each other). Worked examples: with `added.length = 1, removed.length = 0`, the rendered content is `loom watcher: 1 file(s) added or removed; run /reload to refresh the slash command list`; with `added.length = 3, removed.length = 2`, it is `loom watcher: 5 file(s) added or removed; run /reload to refresh the slash command list`. The note MUST NOT be emitted when `added.length + removed.length === 0`. A debounce window that closes with no resolved add or remove paths produces no `loom-system-note` — including a settings-file edit whose post-merge `loomPaths` array is byte-identical to the previous resolved set, an edit that touches only `looms.binderModel` or unrelated keys, and a chokidar burst over discovery roots that does not net any added or removed `.loom` / `.warp` files. The watcher's other obligations for the window (validator-cache invalidation per the **In-flight invocation rule** above, settings-file cache invalidation per the settings-file watcher path) still run; only the note is suppressed. A rename observed within the same debounce window as `removed` of path P followed by `added` of path P still has `added.length + removed.length === 2` and MUST emit the note (the two arrays are disjoint by role and not deduplicated against each other, per the rule above); the suppression applies only to the strictly-empty case. The note prompts the user to run `/reload` (which invokes Pi's `ctx.reload()` and re-executes the entire extension factory). Multiple structural events observed within a single 250 ms debounce window coalesce into one note; events in distinct windows fire separately. The note is informational, not a diagnostic; no `loom/load/*` code is registered for it. Settings-file edits affecting the `loomPaths` array are treated as structural changes and routed through the same prompt; per-`.loom`-file content edits are not.
