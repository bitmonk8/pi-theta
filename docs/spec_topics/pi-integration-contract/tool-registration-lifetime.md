# Tool registration lifetime

<a id="tool-registration-lifetime-and-visibility"></a>**Tool-registration lifetime and visibility.** Pi's tool registry has no `unregisterTool` API — `pi.registerTool` is the only mutation, and entries persist for the process lifetime. The runtime therefore wires loom-synthesised tools (`.loom` callables from another loom's callable set, and the typed-query `__loom_respond_<slug>` one-shot tool described in [Implementation Notes — Runtime](../implementation-notes.md#runtime)) into the model's reach by two distinct mechanisms, one per **calling-loom** mode, designed so that tool *visibility* tracks the loom's invocation rather than the process lifetime, and so the [Parameters and Frontmatter](../frontmatter/frontmatter-fields-a.md#tools) "ambient tools are deliberately *not* inherited" invariant holds.

*Subagent mode (caller is `mode: subagent`).* The runtime never calls `pi.registerTool` for the synthesised tool. The `ToolDefinition` is passed through `createAgentSession({ customTools, tools, ... })` (see **Conversation drive — subagent mode** below). The tool exists only inside the spawned `AgentSession`, dies with `dispose()`, and is invisible to the user's bare Pi session and to every other extension or concurrent loom.

*Prompt mode (caller is `mode: prompt`).* The runtime maintains an extension-scoped registration cache: `Map<schemaSlug, registeredToolName>`, where the slug is the schema slug of the lowered tool `parameters` (produced by the recipe in [Schema Subset — Canonical schema hash](../schema-subset.md#canonical-schema-hash)). On first encounter of a unique slug the runtime calls `pi.registerTool` once with a content-addressed name (`__loom_callee_<slug>__<post-rename-name>` for loom callees; `__loom_respond_<slug>` for the typed-query one-shot tool). Subsequent uses of the same lowered shape reuse the existing registration.

On a subsequent cache hit (a new lowered schema hashing to a slug already present in the cache), the runtime MUST verify byte-equality of the cached canonical-form schema bytes against the new entry's canonical-form bytes before reusing the registration. The slug is a 64-bit truncation of SHA-256 (per [Schema Subset — Canonical schema hash](../schema-subset.md#canonical-schema-hash)), so silent collision is statistically negligible but not impossible; treating slug equality as schema equality without verification would silently alias two distinct tool schemas to one registered name. On byte-mismatch the runtime emits `loom/runtime/registration-cache-collision`, refuses to dedup, and registers the new schema under a disambiguated name with a monotonically increasing per-slug counter (`__loom_callee_<slug>_<n>__<post-rename-name>` for loom callees; `__loom_respond_<slug>_<n>` for typed-query one-shot tools, starting at `n = 2`). The cache stores the canonical-form bytes alongside the registered name so the equality check is a byte comparison, not a re-serialisation.

<a id="pic-17"></a> **PIC-17. Active-set allowlist gating.** Around each query — both untyped and typed — the runtime gates visibility via the active-set:

1. Snapshot `pi.getActiveTools()` into a local list.
2. Call `pi.setActiveTools([...loomCallableSetNames, respondToolName?])` — the set the model sees for this turn is exactly the loom's declared callable set, plus the respond tool when the turn is a typed-query response turn. The user-session snapshot taken in step 1 is held only for the step-4 restore; it is deliberately not unioned into the active set installed here, so the [Parameters and Frontmatter](../frontmatter/frontmatter-fields-a.md#tools) "ambient tools are deliberately *not* inherited" invariant holds during the query.
3. Issue the query.
4. In a `finally` block (so cancellation, panic, and provider exceptions all preserve the invariant), call `pi.setActiveTools(snapshot)` to restore the user session's prior active set.

<a id="pic-8"></a> **PIC-8. Restore-failure protocol.** If the restoring `pi.setActiveTools(snapshot)` call inside the `finally` block of step 4 throws or rejects, the runtime MUST: (a) re-attempt the restore exactly once with the same snapshot (covering transient failures); (b) on a second failure, emit `loom/runtime/active-set-restore-failed` (E, runtime) with `message` carrying the underlying error's message and `hint` listing the snapshot tool names so an operator can manually restore via Pi's `/tools` interface; (c) emit a `display: true` `loom-system-note` whose `content` is the verbatim template `loom: failed to restore tool active-set after /<name>; the user session may have unexpected tools active. Run /reload to reset.` (only `<name>` is substituted — every other character ships verbatim, including the literal `/reload` and the trailing clause); and (d) propagate the original exception (or terminal `Err`) that the `finally` was protecting — restore failure does not mask the inner error, and the cascade-twin dedup rule under **Runtime event channel** below treats the active-set-restore note as independent of any inner-failure note. The retry MUST NOT chain back into `pi.setActiveTools` beyond the single re-attempt above (no exponential backoff, no third try). Subagent-mode invocations are unaffected — the subagent's tools are scoped to the spawned `AgentSession` and released by `dispose()`, so this rule applies only to the prompt-mode and prompt → prompt cross-mode `invoke` snapshot/restore paths in this section.

The snapshot/restore is the only control surface for visibility; the cached `pi.registerTool` entries themselves are intentionally permanent. Consequences:

- `pi.getAllTools()` and `/tools` listings accumulate one entry per distinct lowered schema seen in the extension's lifetime; this is bounded (looms author a small finite number of typed-query and `.loom`-callable shapes) and is documented as cosmetic loom 1.0 behaviour.
- The user's bare Pi session — and every other extension — never sees a loom-synthesised tool in its active set, because no code path other than the loom runtime's own swap ever activates them.
- Hot reload (the `_loom-reload` path described in **Extension entry point** step 5) drops the registration cache so a fresh extension instance starts empty; the orphaned registry entries persist in Pi's global registry but are unreachable, matching the loom 1.0 acceptance of cosmetic accumulation.
- `pi.setActiveTools` snapshots are name lists, not snapshots of the global registry. The `pi.setActiveTools(names: string[]): void` and `pi.getActiveTools(): string[]` signatures are pinned by `ExtensionAPI` in `dist/core/extensions/types.d.ts` of `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above) and are covered by the [Step 0 (c)](./capability-probe.md#entry-capability-probe) factory-probable capability inventory; a Pi minor that renames either member, widens the return type of `pi.setActiveTools` past `void`, or splits the snapshot/restore pair surfaces at the SDK surface-inventory test before the spec contract drifts. <a id="snapshot-restore-pi-behavioural-preconditions"></a> The snapshot/restore protocol additionally relies on two Pi *behavioural* preconditions that the type signature does not encode: (a) the loom runtime treats `pi.setActiveTools(string[])` as synchronous-and-atomic on the JS event loop (a single swap or restore call cannot be interleaved with anything else); and (b) Pi serialises slash-command dispatch per session — two loom-invocation slash dispatches against the same session cannot overlap their snapshot/restore windows. This property is a *behavioural* precondition, **not** a build-time-asserted SDK capability of [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) above — the source-line citations that follow are loom 1.0-pin **evidence** that the property holds at the pin, not a binding contract on Pi internals (which is precisely why [item (e)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below re-audits it on each Pi minor bump rather than the SDK surface-inventory test catching a regression — the inventory test cannot detect a weakening here from the type surface alone). At the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) on `@earendil-works/pi-coding-agent` the evidence trail is `AgentSession.prototype.prompt` in `dist/core/agent-session.js`, whose body awaits the registered slash-command handler end-to-end (`await command.handler(args, ctx)`, dispatched via the private `_tryExecuteExtensionCommand` step) before returning, together with the per-mode user-input drivers — `dist/modes/rpc/rpc-mode.js` lexically awaits each `session.prompt(...)` call site before initiating the next, and `dist/modes/interactive/interactive-mode.js` dispatches user input via the `Editor.onSubmit` callback registered during mode setup whose submit pump establishes the equivalent non-overlap invariant (the per-driver audit recipe and its known-fragile-evidence caveats are owned by [item (e)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) below). The leading-underscore name `_tryExecuteExtensionCommand` (semver does not protect a `_`-prefixed symbol — a Pi *patch* may rename or inline it), the `Editor.onSubmit` TUI-layer detail (orthogonal to the snapshot/restore protocol — a TUI refactor that preserves the cooperative `await` may rename or replace the `onSubmit` pump), and the deep `dist/*.js` bundler-output paths above (which neither **Host prerequisites** above nor the SDK-surface inventory asserts remain un-bundled, un-minified, or stably named across the loom 1.0-pinned minor) are flagged as **known-fragile evidence** — diffable artefacts at the loom 1.0 pin sufficient to satisfy item (e)'s audit recipe on a pass outcome, but not load-bearing claims about their stable shape across patches. [Item (e)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) below absorbs that fragility by routing a textual-diffability failure of the audit recipe (the loom 1.0-cited symbols or paths no longer locatable under their pinned names in a candidate minor — e.g. because Pi has minified, re-bundled, or renamed the `_`-prefixed symbol) to the same defensive-mutex recovery path as a substantive cooperative-await regression, so the contributor is never left with an undefined verdict on a Pi minor that mangles the evidence trail without weakening the underlying behavioural property. The loom 1.0.0 driver set is asserted closed at exactly these two files; the bump-procedure (e) checklist below applies the glob `dist/modes/*/*.js` against the candidate Pi minor and treats any additional matching file with a reachable `session.prompt(...)` call site (per [(e.ii)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation-ii)'s call-graph reachability rule below — literal substring grep on the file's text is not the predicate, since the call may be wrapped in a callback registration during mode setup rather than appearing at the lexical top level) as audit-positive evidence of a driver mode beyond the loom 1.0.0 driver set's two files whose cooperative-await contribution to this precondition has not been load-tested and which triggers the recovery path. The property is therefore established cooperatively across the `AgentSession.prototype.prompt` body and the per-mode user-input driver(s) rather than by a single per-session dispatch lock the SDK exposes. The recovery path on a fail outcome — a defensive per-extension-instance runtime mutex wrapping the snapshot → swap → body → restore sequence, keyed on the factory-captured `pi: ExtensionAPI` reference (the only identity the loom holds at the snapshot/restore site, since `ExtensionCommandContext` is per-slash-handler-invocation and the user `AgentSession` is not exposed to prompt-mode looms) — is enumerated under [item (e) of the editorial-review checklist below](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation). The per-extension-instance keying granularity is well-defined only while **both** the page-pinned one-`pi`-per-extension-instance lifetime (see the [loom 1.0 seam — mid-loom user-session replacement](./conversation-drive.md#v1-seam-mid-loom-user-session-replacement) paragraph below) AND the loom 1.0 one-active-user-session-per-extension-instance presupposition (pinned at *No concurrent user sessions in the same host process* in [Future Considerations](../future-considerations/model-changes-and-non-goals.md#v1-non-goals)) hold; together, on the recovery path, the keyed mutex is the unique serialisation point per user session (the mutex exists only when the recovery path is engaged on a fail outcome of the [item (e)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) audit recipe — loom 1.0 ships without it on a pass outcome, since precondition (b)'s cooperative-`await` chain establishes the non-overlap property in that case), and a future Pi minor that weakens **either** invariant requires revisiting the keying granularity at the same time the precondition is re-validated. Precondition (b) covers only the slash entry point; the symmetric in-process guarantee that nested prompt → prompt `invoke(...)` calls cannot overlap their snapshot/restore windows is owned by the loom runtime itself — the `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns per [Invocation — Cross-mode semantics](../invocation.md), loom 1.0 exposes no parallel-`invoke` surface, and prompt-mode `.loom` callees in another loom's `tools:` are rejected at load time per [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools), so two prompt-mode snapshot/restore windows against the same user session can only ever be **strictly LIFO-nested**, never concurrently overlapping. If either Pi behavioural precondition above weakens in a future Pi minor, the snapshot/restore protocol above MUST be re-validated as part of [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect either weakening from the type surface alone.
- If another extension calls `pi.setActiveTools` between a loom's snapshot and its `finally` restore, the loom's restore overwrites that change. Extensions wishing to coexist with loom should mutate the active set outside of any in-flight loom invocation, or accept the overwrite. The loom runtime emits no diagnostic for this case. This overwrite rule covers prompt-mode untyped queries, prompt-mode typed-query free and forced phases, and prompt → prompt invocation bodies — same protocol, same rule. It does NOT apply to subagent mode, which never touches `pi.setActiveTools` on the user session.
- The `unknown_tool` cause on `CodeToolError` (see [Tool Calls](../tool-calls.md)) and the captured-reference dispatch model from [Parameters and Frontmatter — Resolution snapshot](../frontmatter.md#resolution-snapshot) are unaffected: code-side `<name>(...)` calls never touch the active-set machinery — they dispatch directly through the loom's load-time callable table.

No Pi-side change is required for loom 1.0; if Pi later adds `pi.unregisterTool`, the registration cache becomes the natural site to call it from on extension teardown without altering the per-mode wiring rule above.
