# Tool registration lifetime

<a id="tool-registration-lifetime-and-visibility"></a>**Tool-registration lifetime and visibility.** Pi's tool registry has no `unregisterTool` API — `pi.registerTool` is the only mutation, and entries persist for the process lifetime (that registrations stay present, valid, and dispatchable for the extension-instance lifetime absent a `pi.unregisterTool` call is pinned as the [registry-persistence presupposition](./host-prerequisites.md#registry-persistence-presupposition) in **Host prerequisites**). The runtime therefore wires theta-synthesised tools (`.theta` callables from another theta's callable set, and the typed-query `__theta_respond_<slug>` one-shot tool described in [Implementation Notes — Runtime](../implementation-notes.md#runtime)) into the model's reach by two distinct mechanisms, one per **calling-theta** mode, designed so that tool *visibility* tracks the theta's invocation rather than the process lifetime, and so the [Parameters and Frontmatter](../frontmatter/frontmatter-fields-a.md#tools) "ambient tools are deliberately *not* inherited" invariant holds.

*Subagent mode (caller is `mode: subagent`).* The runtime never calls `pi.registerTool` for the subagent's tools, and no tool crosses the process boundary. The child `pi` process runs its own startup extension discovery and holds its own tool registry; the callable set reaches the child's model through the `--tools <name1,name2,…>` allowlist (or `--no-tools` for an empty callable set) on the child launch, and `.theta` callables are resolved child-side by name and content-hash-verified (see **Conversation drive — subagent mode** below). A tool selected for the subagent therefore exists only inside the child process, dies when the child exits, and is invisible to the user's bare Pi session and to every other extension or concurrent theta. The parent holds no `execute` for any subagent-mode tool and issues no active-set mutation on the user session for it.

*Prompt mode (caller is `mode: prompt`).* The runtime maintains an extension-scoped registration cache: `Map<schemaSlug, registeredToolName>`, where the slug is the schema slug of the lowered tool `parameters` (produced by the recipe in [Schema Subset — Canonical schema hash](../schema-subset.md#canonical-schema-hash)). On first encounter of a unique slug the runtime calls `pi.registerTool` once with a content-addressed name (`__theta_callee_<slug>__<post-rename-name>` for theta callees; `__theta_respond_<slug>` for the typed-query one-shot tool). Subsequent uses of the same lowered shape reuse the existing registration.

<a id="pic-44"></a> **PIC-44. Cache-hit schema byte-equality verification.** On a subsequent cache hit (a new lowered schema hashing to a slug already present in the cache), the runtime MUST verify byte-equality of the cached canonical-form schema bytes against the new entry's canonical-form bytes before reusing the registration. The slug is a 64-bit truncation of SHA-256 (per [Schema Subset — Canonical schema hash](../schema-subset.md#canonical-schema-hash)), so silent collision is statistically negligible but not impossible; treating slug equality as schema equality without verification would silently alias two distinct tool schemas to one registered name. On byte-mismatch the runtime emits `theta/runtime/registration-cache-collision`, refuses to dedup, and registers the new schema under a disambiguated name with a monotonically increasing per-slug counter (`__theta_callee_<slug>_<n>__<post-rename-name>` for theta callees; `__theta_respond_<slug>_<n>` for typed-query one-shot tools, starting at `n = 2`). The cache stores the canonical-form bytes alongside the registered name so the equality check is a byte comparison, not a re-serialisation.

<a id="pic-17"></a> **PIC-17. Active-set allowlist gating.** Around each query — both untyped and typed — the runtime gates visibility via the active-set:

1. Snapshot `pi.getActiveTools()` into a local list.
2. Call `pi.setActiveTools([...thetaCallableSetNames, respondToolName?])` — the set the model sees for this turn is exactly the theta's declared callable set, plus the respond tool when the turn is a typed-query response turn. The user-session snapshot taken in step 1 is held only for the step-4 restore; it is deliberately not unioned into the active set installed here, so the [Parameters and Frontmatter](../frontmatter/frontmatter-fields-a.md#tools) "ambient tools are deliberately *not* inherited" invariant holds during the query.
3. Issue the query.
4. In a `finally` block (so cancellation, panic, and provider exceptions all preserve the invariant), call `pi.setActiveTools(snapshot)` to restore the user session's prior active set.

<a id="pic-8"></a> **PIC-8. Restore-failure protocol.** If the restoring `pi.setActiveTools(snapshot)` call inside the `finally` block of step 4 throws or rejects, the runtime MUST: (a) re-attempt the restore exactly once with the same snapshot (covering transient failures); (b) on a second failure, emit `theta/runtime/active-set-restore-failed` (E, runtime) with `message` carrying the underlying error's message and `hint` listing the snapshot tool names; (c) emit a `display: true` `theta-system-note` whose `content` is the verbatim template `theta: failed to restore tool active-set after /<name>; the user session may have unexpected tools active. Run /reload to reset.` (only `<name>` is substituted — every other character ships verbatim, including the literal `/reload` and the trailing clause); and (d) propagate the original exception (or terminal `Err`) that the `finally` was protecting — restore failure does not mask the inner error, and the cascade-twin dedup rule under **Runtime event channel** below treats the active-set-restore note as independent of any inner-failure note. The retry MUST NOT chain back into `pi.setActiveTools` beyond the single re-attempt above (no exponential backoff, no third try). Subagent-mode invocations are unaffected — the subagent's tools are scoped to the child `pi` process and released when the child exits, so this rule applies only to the prompt-mode and prompt → prompt cross-mode `invoke` snapshot/restore paths in this section.

<a id="pic-19"></a> **PIC-19. Snapshot/swap-install-failure protocol.** PIC-8 above governs only the step-4 restore. A throw or rejection from the step-1 `pi.getActiveTools()` snapshot or the step-2 `pi.setActiveTools([...])` swap-install is a *setup-side* failure that occurs before the query is issued: the query does not proceed, no restore is owed (a step-1 throw has committed no active-set change, and a step-2 throw leaves the install uncommitted under the synchronous-completion-and-commit precondition recorded at [snapshot/restore Pi behavioural preconditions](#snapshot-restore-pi-behavioural-preconditions) above), and the failure surfaces on the `theta/runtime/internal-error` runtime-defect channel per [Post-probe SDK-shape drift](./capability-probe.md#post-probe-sdk-shape-drift), which is the authoritative routing owner for both calls. PIC-8's single-re-attempt retry and `theta/runtime/active-set-restore-failed` emission are restore-side only and do not apply to the snapshot or swap-install calls.

The snapshot/restore is the only control surface for visibility; the cached `pi.registerTool` entries themselves are intentionally permanent. Consequences:

- `pi.getAllTools()` and `/tools` listings accumulate one entry per distinct lowered schema seen in the extension's lifetime; this is bounded (thetas author a small finite number of typed-query and `.theta`-callable shapes) and is documented as cosmetic theta 1.0 behaviour.
- The user's bare Pi session — and every other extension — never sees a theta-synthesised tool in its active set, because no code path other than the theta runtime's own swap ever activates them.
- Hot reload via Pi's `/reload` (the `ctx.reload()` factory re-execution prompted in **Extension entry point** step 5) drops the registration cache so a fresh extension instance starts empty; the orphaned registry entries persist in Pi's global registry but are unreachable, matching the theta 1.0 acceptance of cosmetic accumulation.
- `pi.setActiveTools` snapshots are name lists, not snapshots of the global registry. The `pi.setActiveTools(names: string[]): void` and `pi.getActiveTools(): string[]` signatures are pinned by `ExtensionAPI` in `dist/core/extensions/types.d.ts` of `@earendil-works/pi-coding-agent` (the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) from **Host prerequisites** above) and are covered by the [Step 0 (c)](./capability-probe.md#entry-capability-probe) factory-probable capability inventory; a Pi minor that renames either member, widens the return type of `pi.setActiveTools` past `void`, or splits the snapshot/restore pair surfaces at the SDK surface-inventory test before the spec contract drifts. <a id="snapshot-restore-pi-behavioural-preconditions"></a> The snapshot/restore protocol additionally relies on two Pi *behavioural* preconditions that the type signature does not encode: (a) the theta runtime treats `pi.setActiveTools(string[])` as completing synchronously without yielding the JS event loop — the active-set change is committed by the time the call returns (observable to a subsequent `pi.getActiveTools()` in the same task), and a swap-install that throws leaves the active set unchanged; and (b) Pi serialises slash-command dispatch per session — two theta-invocation slash dispatches against the same session cannot overlap their snapshot/restore windows. This property is a *behavioural* precondition, **not** a build-time-asserted SDK capability of [SDK capability inventory](./inventory-audit-intro.md#sdk-capability-inventory) above — the source-line citations that follow are theta 1.0-pin **evidence** that the property holds at the pin, not a binding contract on Pi internals (which is precisely why [item (e)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below re-audits it on each Pi minor bump rather than the SDK surface-inventory test catching a regression — the inventory test cannot detect a weakening here from the type surface alone). At the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) on `@earendil-works/pi-coding-agent` the evidence trail is `AgentSession.prototype.prompt` in `dist/core/agent-session.js`, whose body awaits the registered slash-command handler end-to-end (`await command.handler(args, ctx)`, dispatched via the private `_tryExecuteExtensionCommand` step) before returning, together with the per-mode user-input drivers — `dist/modes/rpc/rpc-mode.js` lexically awaits each `session.prompt(...)` call site before initiating the next, and `dist/modes/interactive/interactive-mode.js` dispatches user input via the `Editor.onSubmit` callback registered during mode setup whose submit pump establishes the equivalent non-overlap invariant (the per-driver audit recipe and its known-fragile-evidence caveats are owned by [item (e)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) below). The leading-underscore name `_tryExecuteExtensionCommand` (semver does not protect a `_`-prefixed symbol — a Pi *patch* may rename or inline it), the `Editor.onSubmit` TUI-layer detail (orthogonal to the snapshot/restore protocol — a TUI refactor that preserves the cooperative `await` may rename or replace the `onSubmit` pump), and the deep `dist/*.js` bundler-output paths above (which neither **Host prerequisites** above nor the SDK-surface inventory asserts remain un-bundled, un-minified, or stably named across the theta 1.0-pinned minor) are flagged as **known-fragile evidence** — diffable artefacts at the theta 1.0 pin sufficient to satisfy item (e)'s audit recipe on a pass outcome, but not load-bearing claims about their stable shape across patches. [Item (e)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) below absorbs that fragility by routing a textual-diffability failure of the audit recipe (the theta 1.0-cited symbols or paths no longer locatable under their pinned names in a candidate minor — e.g. because Pi has minified, re-bundled, or renamed the `_`-prefixed symbol) to the same defensive-mutex recovery path as a substantive cooperative-await regression, so the contributor is never left with an undefined verdict on a Pi minor that mangles the evidence trail without weakening the underlying behavioural property. The theta 1.0.0 driver set is asserted closed at exactly these two files; the bump-procedure (e) checklist below applies the glob `dist/modes/*/*.js` against the candidate Pi minor and treats any additional matching file with a reachable `session.prompt(...)` call site (per [(e.ii)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation-ii)'s call-graph reachability rule below — literal substring grep on the file's text is not the predicate, since the call may be wrapped in a callback registration during mode setup rather than appearing at the lexical top level) as audit-positive evidence of a driver mode beyond the theta 1.0.0 driver set's two files whose cooperative-await contribution to this precondition has not been load-tested and which triggers the recovery path. The property is therefore established cooperatively across the `AgentSession.prototype.prompt` body and the per-mode user-input driver(s) rather than by a single per-session dispatch lock the SDK exposes. The recovery path on a fail outcome — a defensive per-extension-instance runtime mutex wrapping the snapshot → swap → body → restore sequence, keyed on the factory-captured `pi: ExtensionAPI` reference (the only identity the theta holds at the snapshot/restore site, since `ExtensionCommandContext` is per-slash-handler-invocation and the user `AgentSession` is not exposed to prompt-mode thetas) — is enumerated under [item (e) of the editorial-review checklist below](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation). <a id="recovery-mutex-runtime-contract"></a>On that recovery path the mutex's runtime contract is a strictly serialising FIFO `await`: a second slash dispatch that arrives while the mutex is held waits in arrival order and acquires when the holder releases, and acquisition carries no timeout on the steady-state path (the cooperative-`await` baseline it emulates imposes none). The teardown interaction is *observed* rather than newly bounded: on `session_shutdown` teardown, sub-step 2's `thetaAbort.abort()` (per the [patch-skew degradation contract](./patch-skew-degradation.md#patch-skew-degradation-contract) above) cancels the holder, whose `finally` runs the step-4 restore and releases the mutex; a queued waiter then acquires, observes the already-aborted `thetaAbort.signal` at its first [`Checkpoint`](./host-interfaces-services.md#checkpoint-seam), and falls through its own `finally` without running a body — so that contract's sub-step 3 `SHUTDOWN_AWAIT_CAP_MS` await over `disposeBarrier` settlement already bounds total teardown wall-time across all waiters and the mutex adds no second budget. The per-extension-instance keying granularity is well-defined only while **both** the page-pinned one-`pi`-per-extension-instance lifetime (see the [theta 1.0 seam — mid-theta user-session replacement](./conversation-drive.md#theta-1-0-seam-mid-theta-user-session-replacement) paragraph below) AND the theta 1.0 one-active-user-session-per-extension-instance presupposition (pinned at *No concurrent user sessions in the same host process* in [Future Considerations](../future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals)) hold; together, on the recovery path, the keyed mutex is the unique serialisation point per user session (the mutex exists only when the recovery path is engaged on a fail outcome of the [item (e)](./version-bump-step2.md#bump-checklist-slash-dispatch-serialisation) audit recipe — theta 1.0 ships without it on a pass outcome, since precondition (b)'s cooperative-`await` chain establishes the non-overlap property in that case), and a future Pi minor that weakens **either** invariant requires revisiting the keying granularity at the same time the precondition is re-validated. Precondition (b) covers only the slash entry point; the symmetric in-process guarantee that nested prompt → prompt `invoke(...)` calls cannot overlap their snapshot/restore windows is owned by the theta runtime itself — the `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns per [Invocation — Cross-mode semantics](../invocation.md), theta 1.0 exposes no parallel-`invoke` surface, and prompt-mode `.theta` callees in another theta's `tools:` are rejected at load time per [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools), so two prompt-mode snapshot/restore windows against the same user session can only ever be **strictly LIFO-nested**, never concurrently overlapping. If either Pi behavioural precondition above weakens in a future Pi minor, the snapshot/restore protocol above MUST be re-validated as part of [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect either weakening from the type surface alone.
- If another extension calls `pi.setActiveTools` between a theta's snapshot and its `finally` restore, the theta's restore overwrites that change. Extensions wishing to coexist with theta should mutate the active set outside of any in-flight theta invocation, or accept the overwrite. The theta runtime emits no diagnostic for this case. This overwrite rule covers prompt-mode untyped queries, prompt-mode typed-query free and forced phases, and prompt → prompt invocation bodies — same protocol, same rule. It does NOT apply to subagent mode, which never touches `pi.setActiveTools` on the user session.
- The `unknown_tool` cause on `CodeToolError` (see [Tool Calls](../tool-calls.md)) and the captured-reference dispatch model from [Parameters and Frontmatter — Resolution snapshot](../frontmatter.md#resolution-snapshot) are unaffected: code-side `<name>(...)` calls never touch the active-set machinery — they dispatch directly through the theta's load-time callable table.

No Pi-side change is required for theta 1.0; if Pi later adds `pi.unregisterTool`, the registration cache becomes the natural site to call it from on extension teardown without altering the per-mode wiring rule above.

## Acceptance criteria — PIC-17 active-set install vector

The active-set allowlist-gating obligation ([PIC-17](#pic-17)) decomposes into the fixture obligations below, all of which the implementation MUST satisfy (the leaf owns fixture-shape; branch coverage and the exact install vector are normative). Fixtures assert the step-2 `pi.setActiveTools([...])` install vector exactly and that the step-1 `pi.getActiveTools()` snapshot is held only for the step-4 restore and is **not** unioned into the step-2 install, witnessing the [Parameters and Frontmatter](../frontmatter/frontmatter-fields-a.md#tools) "ambient tools are deliberately *not* inherited" invariant. No new active-set API or wire field is introduced.

- **(a) Empty callable set, untyped/free turn.** For a theta whose declared callable set is empty on an untyped query (no respond tool), with the step-1 `pi.getActiveTools()` snapshot non-empty: assert the step-2 install vector is exactly `[]` and contains no member of the step-1 snapshot.
- **(b) Forced-respond turn.** On a typed-query response turn (the respond tool active), assert the step-2 install vector is exactly `[...thetaCallableSetNames, respondToolName]` — i.e. `[respondToolName]` (`__theta_respond_<slug>`) when the callable set is empty — and contains no member of the step-1 snapshot.
