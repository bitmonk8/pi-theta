# Capability inventory items

1. <a id="sdk-cap-slash-command-registration"></a> **Slash-command registration.** Pi MUST expose `pi.registerCommand(name, { description, handler })` per **Extension entry point** above. The runtime calls this once per surviving discovered theta on `session_start`.

2. <a id="sdk-cap-prompt-conversation-drive"></a> **Prompt-mode conversation drive.** Pi MUST expose `pi.sendUserMessage(text [, { deliverAs }])` for issuing turns and `ExtensionCommandContext.waitForIdle()` for awaiting turn completion. Both are consumed by the prompt-mode driver per **Conversation drive — prompt mode** above.

3. <a id="sdk-cap-subagent-isolated-session"></a> **Subagent-mode isolated session.** Pi MUST provide a **runnable child `pi` entry point** — resolvable via the [executable-resolution ladder](./subagent.md#subagent-executable-resolution) — that runs a slash invocation under `--mode json -p "/<slug>"` and emits its `--mode json` event stream on stdout, so the runtime can spawn a fresh, isolated child that runs the whole callee and returns its final-value envelope, with a transcript ephemeral under the `--no-session` CLI flag. Consumed by **Conversation drive — subagent mode** and **Subagent child-process lifecycle** ([PIC-9](./subagent.md#pic-9)) above. Unlike the other capabilities, this one is **not** a factory-time `typeof` member: it is verified at probe time by the [executable-resolution probe](./capability-probe.md#entry-capability-probe) (Step 0), which runs the ladder's rung-1 existence check and refuses theta registration with a precise diagnostic when no runnable entry point is found. The former in-process satellites (`createAgentSession`, the theta-constructed `ResourceLoader` adapter, and `SessionManager.inMemory`) leave the inventory entirely; the capability's consumed Pi-owned surfaces are now the **CLI flags** and the **stdout wire surface** (`--mode json` event stream + `theta_result` return envelope) pinned at [CLI-flag and wire pins](./subagent.md#subagent-cli-wire-pins) above. The transcript-privacy property this item attributes to the child — the spawned subagent's transcript staying private to the theta and being discarded when the theta returns — is a presupposition on Pi's documented `--no-session` ephemeral-transcript semantics; the theta asserts no independent behavioural check that the child discards rather than persists the transcript. The on-bump re-validation splits across editorial-review checklist items under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below: [item (o)](./version-bump-step2.md#bump-checklist-subagent-cli-wire-pins) audits the consumed CLI flags and stdout wire surfaces (flag semantics, `--mode json` line framing, `theta_result` envelope line integrity, stdin-EOF exit behaviour), and [item (aj)](./version-bump-step2.md#bump-checklist-no-session-ephemeral) audits the `--no-session` *behaviour* — that the child discards rather than persists the transcript. Step 2(a)'s seven-capability surface-inventory assertion does not enumerate any of these CLI/stdout surfaces (they are not `typeof`-probable), so a silent flag-semantics or framing change is caught at editorial review at bump time rather than at the surface-inventory assertion. Neither the CLI-flag set nor the stdout wire surface mints a new SDK capability — both attach to this capability 3.

4. <a id="sdk-cap-tool-registration-gating"></a> **Tool registration and gating.** Pi MUST expose `pi.registerTool` (registration sink) and the `pi.getActiveTools` / `pi.setActiveTools` snapshot/restore pair for prompt-mode visibility control. Consumed by **Tool-registration lifetime and visibility** above. The lack of `pi.unregisterTool` and the resulting cosmetic registration-cache accumulation are documented in the same section. That registrations stay present, valid, and dispatchable for the extension-instance lifetime absent a `pi.unregisterTool` call — the positive property this inventory item records only as the *absence* of `pi.unregisterTool` — is pinned as the [registry-persistence presupposition](./host-prerequisites.md#registry-persistence-presupposition) under **Host prerequisites**.

5. <a id="sdk-cap-cancellation-propagation"></a> **Cancellation propagation.** Pi MUST supply an `AbortSignal` at the two extension entry points theta uses — `ctx.signal` on the slash-command `ExtensionCommandContext` and the `signal` parameter of `tool.execute(...)`. For subagent mode (which has no `AbortSignal` entry point), under the RFC 0006 child-process theta design the runtime's one-shot `thetaAbort.signal` listener tears the child down by closing its parent-held stdin pipe (the grace signal; the `-p` child exposes no RPC abort command); when the child does not exit within the teardown budget the runtime kills it (bounded process-tree kill, per [PIC-9](./subagent.md#pic-9) and [PIC-63](./subagent.md#pic-63)). The fixed-shape WHATWG `AbortSignal` / `AbortController` member surface is enumerated in [Step 0 (b)](./capability-probe.md#entry-capability-probe) above; the child cancellation grace signal is not `typeof`-probable and is re-audited at bump time as part of the CLI/stdout wire pins (see [item (o)](./version-bump-step2.md#bump-checklist-subagent-cli-wire-pins)), not through the factory-probable member loop. The theta-side `AbortController` wrapping rule and the listener-and-grace-signal (stdin-close) mechanism are in **Cancellation source** and **Subagent child-process lifecycle** above and in [Cancellation](../cancellation.md). This rule and **Host prerequisites #4 — Pi-supplied `AbortSignal`** above are the same obligation viewed from the inventory and the host-prerequisite axes; the inventory entry is the cross-reference target for `spec.md` and consumers, the **Host prerequisites #4** paragraph is the in-page neighbour of the other host preconditions.

6. <a id="sdk-cap-custom-message-renderer"></a> **Custom-message channel and renderer.** Pi MUST expose `pi.sendMessage({ customType, content, display, details }, { triggerTurn })` for routing system notes through the custom-message channel and `pi.registerMessageRenderer(customType, renderer)` for rendering them in the transcript. The signature, lifecycle, and ownership rules are in **System notes** and **Renderer registration (`pi.registerMessageRenderer`)** above.

7. <a id="sdk-cap-binder-llm-model"></a> **Binder LLM model.** Non-bypass thetas MUST be able to resolve a structured-output-capable LLM via `ctx.modelRegistry` (the `ModelRegistry` type pinned at [model-registry surface](./host-interfaces-core.md#model-registry-pin) above). The runtime applies the **bypass criterion** at load time (no `params:`, or single string-typed parameter with no default) per [Binder — Bypass cases](../binder/binder-bypass-and-envelope.md#bypass-cases); bypassable thetas skip the binder call entirely, and non-bypass thetas fail to load with `theta/load/binder-model-unresolved` if no structured-output-capable model is available. Strict-capability degradation, the same `theta/load/binder-model-unresolved` code on theta 1.0 strict-capability mismatch, and the `theta/load/binder-model-strict-capability-unknown` warning under the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) are owned by [Binder — Strict-capability requirement](../binder/binder-model-and-context.md#strict-capability-requirement); this rule defers the diagnostic-code semantics to that page.

<a id="sdk-cap-peer-dep-revalidation"></a>
**Re-validation on `peerDependencies` widening.** Widening `peerDependencies` requires re-validating the seven obligations above against the new Pi minor before the range moves; the re-validation artefact is the build-time SDK surface-inventory assertion passing against the new package, and the procedure that gates the bump is [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below.
