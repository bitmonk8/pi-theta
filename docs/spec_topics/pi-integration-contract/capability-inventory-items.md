# Capability inventory items

1. <a id="sdk-cap-slash-command-registration"></a> **Slash-command registration.** Pi MUST expose `pi.registerCommand(name, { description, handler })` per **Extension entry point** above. The runtime calls this once per surviving discovered loom on `session_start`.

2. <a id="sdk-cap-prompt-conversation-drive"></a> **Prompt-mode conversation drive.** Pi MUST expose `pi.sendUserMessage(text [, { deliverAs }])` for issuing turns and `ExtensionCommandContext.waitForIdle()` for awaiting turn completion. Both are consumed by the prompt-mode driver per **Conversation drive â€” prompt mode** above.

3. <a id="sdk-cap-subagent-isolated-session"></a> **Subagent-mode isolated session.** Pi MUST expose `createAgentSession(...)` returning a disposable `AgentSession` whose transcript is private when `SessionManager.inMemory(cwd)` is supplied. Consumed by **Conversation drive â€” subagent mode** and **Subagent session lifecycle** above; the loom-touched member surface of the returned handle (`abort`, `dispose`, `sendUserMessage`, `subscribe`) is pinned at [`AgentSession` (member surface loom touches)](./host-interfaces-core.md#agentsession-interface) above. The capability's two Pi-owned spawn-call satellites are pinned alongside the spawn block: `SessionManager.inMemory` â€” the in-memory-transcript factory this capability's privacy guarantee depends on â€” is declared as `static inMemory(cwd?: string): SessionManager` on the `SessionManager` class at `dist/core/session-manager.d.ts`, and the loom-constructed `ResourceLoader` adapter's member surface is declared at `dist/core/resource-loader.d.ts`; both (with their on-bump re-validation pointer) are pinned at [spawn-block satellite type pins](./subagent.md#subagent-spawn-satellite-types) above. The transcript-privacy property this item attributes to the in-memory session â€” the spawned subagent's transcript staying private to the loom and being discarded when the loom returns â€” is a presupposition on Pi's documented `SessionManager.inMemory` in-memory-transcript semantics; the loom asserts no independent behavioural check that the in-memory session discards rather than persists the transcript. The on-bump re-validation for this satellite splits across two editorial-review checklist items under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below: [item (o)](./version-bump-step2.md#bump-checklist-subagent-spawn-satellite-types) confirms the factory *signature* `static inMemory(cwd?: string): SessionManager`, and [item (aj)](./version-bump-step2.md#bump-checklist-inmemory-transcript-discard) audits the no-persistence *behaviour* â€” that the in-memory session discards rather than persists the transcript â€” by reading the in-memory-session branch's JS body against the bump baseline. Step 2(a)'s seven-capability surface-inventory assertion does not enumerate `SessionManager.inMemory` at all. A Pi-side change that preserved the signature but persisted the transcript would pass item (o) yet be caught by item (aj)'s behavioural audit, so this presupposition's detection is routed to editorial review at bump time on the same footing as the others audited under item (o). Neither satellite mints a new SDK capability â€” both attach to this capability 3.

4. <a id="sdk-cap-tool-registration-gating"></a> **Tool registration and gating.** Pi MUST expose `pi.registerTool` (registration sink) and the `pi.getActiveTools` / `pi.setActiveTools` snapshot/restore pair for prompt-mode visibility control. Consumed by **Tool-registration lifetime and visibility** above. The lack of `pi.unregisterTool` and the resulting cosmetic registration-cache accumulation are documented in the same section.

5. <a id="sdk-cap-cancellation-propagation"></a> **Cancellation propagation.** Pi MUST supply an `AbortSignal` at the two extension entry points loom uses â€” `ctx.signal` on the slash-command `ExtensionCommandContext` and the `signal` parameter of `tool.execute(...)`. For subagent mode (which has no `AbortSignal` entry point â€” `CreateAgentSessionOptions` has no `signal` field in the loom 1.0 Pi SDK pin), Pi MUST expose `AgentSession.abort(): Promise<void>` on the handle returned by `createAgentSession(...)` so the runtime's one-shot `loomAbort.signal` listener can tear down the spawned session's in-flight provider call. The fixed-shape WHATWG `AbortSignal` / `AbortController` member surface is enumerated in [Step 0 (b)](./capability-probe.md#entry-capability-probe) above; `AgentSession.prototype.abort` is enumerated alongside `createAgentSession` under [Step 0 (c)](./capability-probe.md#entry-capability-probe)'s factory-probable members so a Pi version that drops or renames it surfaces as `loom/load/host-incompatible` rather than as a runtime-time `TypeError` at the spawn site; the non-probed handle members the subagent driver also touches (`dispose`, `sendUserMessage`, `subscribe`) are pinned at [`AgentSession` (member surface loom touches)](./host-interfaces-core.md#agentsession-interface) above. The loom-side `AbortController` wrapping rule and the listener-and-`abort()` mechanism are in **Cancellation source** and **Subagent session lifecycle** above and in [Cancellation](../cancellation.md). This rule and **Host prerequisites #4 â€” Pi-supplied `AbortSignal`** above are the same obligation viewed from the inventory and the host-prerequisite axes; the inventory entry is the cross-reference target for `spec.md` and consumers, the **Host prerequisites #4** paragraph is the in-page neighbour of the other host preconditions.

6. <a id="sdk-cap-custom-message-renderer"></a> **Custom-message channel and renderer.** Pi MUST expose `pi.sendMessage({ customType, content, display, details }, { triggerTurn })` for routing system notes through the custom-message channel and `pi.registerMessageRenderer(customType, renderer)` for rendering them in the transcript. The signature, lifecycle, and ownership rules are in **System notes** and **Renderer registration (`pi.registerMessageRenderer`)** above.

7. <a id="sdk-cap-binder-llm-model"></a> **Binder LLM model.** Non-bypass looms MUST be able to resolve a structured-output-capable LLM via `ctx.modelRegistry` (the `ModelRegistry` type pinned at [model-registry surface](./host-interfaces-core.md#model-registry-pin) above). The runtime applies the **bypass criterion** at load time (no `params:`, or single string-typed parameter with no default) per [Binder â€” Bypass cases](../binder/binder-bypass-and-envelope.md#bypass-cases); bypassable looms skip the binder call entirely, and non-bypass looms fail to load with `loom/load/binder-model-unresolved` if no structured-output-capable model is available. Strict-capability degradation, the same `loom/load/binder-model-unresolved` code on loom 1.0 strict-capability mismatch, and the `loom/load/binder-model-strict-capability-unknown` warning under the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) are owned by [Binder â€” Strict-capability requirement](../binder/binder-model-and-context.md#strict-capability-requirement); this rule defers the diagnostic-code semantics to that page.
