# Diagnostic emission isolation

   - <a id="diagnostic-emission-isolation"></a> **Diagnostic-emission isolation.** Each `console.error` call inside the `session_shutdown` handler â€” the five teardown-handler emissions enumerated in the bullet immediately above (`loom/runtime/reload-teardown-timeout`, `loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, `loom/host/session-shutdown-runtime-degraded`) â€” MUST be wrapped in `try`/`catch`. *Wire format.* Each such `console.error` call carries a single argument: a serialised structured object containing the four required fields `code`, `severity`, `message`, and `details` (matching the row's columns in [Diagnostics](../diagnostics.md) and the `details` shape pinned per code in this paragraph and the **Unknown-reason rule** above), plus the optional `hint?` field from the [`Internal diagnostic shape`](../diagnostics.md) declaration when set; of the five teardown-handler codes only `loom/runtime/reload-teardown-timeout` pins `hint` (its row in [Diagnostics](../diagnostics.md) pins `hint` to the elapsed wall time and that pin stands; implementations MUST set `hint` accordingly on that code), and implementations MAY omit `hint` on the other four codes (`loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`, `loom/host/session-shutdown-runtime-degraded`), none of which pins `hint`; the other optional fields of the `Internal diagnostic shape` â€” `file`, `range`, `related?`, `masked?` â€” are likewise not pinned by any of the five codes and MAY be omitted, on the same footing as `hint` on those four codes; the exact serialisation primitive (e.g. `JSON.stringify`) is leaf-owned and not pinned here, but the field names and the per-code `details` shapes MUST be preserved verbatim so log-pipeline and operator-tooling consumers can parse a single contract across all five codes. *`details` nesting convention.* The wire-format nesting convention for `details` payloads that surface the `session_shutdown` event payload â€” the `details.event.{reason}` shape `loom/host/session-shutdown-runtime-degraded` carries (mirroring `loom/runtime/cancelled-by-session-shutdown`), the closed-at-`{reason}`-in-loom-1.0.0 pin governing additions, and the carve-out keeping the four sibling teardown-handler codes' runtime-constructed failure-mode payloads (`details.observed`, `details.failure`, `details.step` / `.call` / `.error`) flat under `details` â€” is owned by [Diagnostics â€” `details` shape conventions for the `session_shutdown` event payload](../diagnostics/diagnostic-shape.md#session-shutdown-details-conventions); this paragraph imports that contract by reference and adds no per-code shape rules of its own. The same wire-format pin applies to the terminal `console.error` of the `sendSystemNote` fallback chain referenced at the end of this paragraph. The entire serialisation-and-emission sequence â€” the leaf-owned serialisation primitive call followed by the `console.error` call it feeds â€” MUST be wrapped in a single `try`/`catch`; on a serialiser throw (e.g. a circular `cause`, a `BigInt` field, a host-shimmed `JSON.stringify` getter, or any other foreseeable serialiser failure) the catch arm MUST fall back to emitting `console.error(code)` â€” the bare diagnostic code string for the diagnostic that was being serialised â€” so the code remains grep-able in operator logs even when structured emission failed, and MUST NOT re-throw. *Two-token fallback for nested-shape codes.* For the two codes whose `details` payload nests the `session_shutdown` event under `details.event.*` per the [`details` shape conventions for the `session_shutdown` event payload](../diagnostics/diagnostic-shape.md#session-shutdown-details-conventions) â€” `loom/host/session-shutdown-runtime-degraded` and `loom/runtime/cancelled-by-session-shutdown` â€” the catch arm MUST instead emit the two-token form `` console.error(`${code} ${detailsEventReason}`) ``, where `detailsEventReason` is the `details.event.reason` string the structured payload would have carried (already a string per the captured-value cases the **Unknown-reason rule** above pins for `loom/host/session-shutdown-runtime-degraded`, and per the **Per-invocation operator visibility (clean-cancel path)** bullet below's `entry.shutdownReason` channel for `loom/runtime/cancelled-by-session-shutdown`; no further coercion is needed and the interpolation MUST NOT itself re-throw). *Hoist obligation.* The runtime MUST hoist that `details.event.reason` value into a local string binding (e.g. `const detailsEventReason: string = ...`) **before** entering the wrapped serialisation-and-emission sequence â€” i.e. as part of the same payload-construction code that, on the per-invocation clean-cancel path for `loom/runtime/cancelled-by-session-shutdown`, already performs the unset-`entry.shutdownReason` `"<unreadable>"` substitution pinned by the [*Accepted loom 1.0 residual gap â€” sub-step-2 stamp-throw case*](./session-only-degraded-state.md#session-only-reason-degraded-state) paragraph below (the substitution lives at payload-construction time so the structured `details.event.reason` field carries a string, never `undefined`) â€” and the catch arm MUST interpolate the pre-hoisted local rather than re-reading from the partially-constructed structured object or from `entry.shutdownReason` directly. This decouples the catch arm from the structured-object-construction order (so a serialiser throw mid-construction cannot leave the catch arm reading from an unbuilt or absent field) and from any defensive coercion the catch arm itself would otherwise need. On a sub-step-2 stamp-throw entry that later reaches the per-invocation `finally`'s clean-cancel emission, the hoisted local therefore carries the literal string `"<unreadable>"` per the same residual-gap paragraph, and the two-token catch arm interpolates that sentinel verbatim on the same footing as the closed-set, `String(event.reason)`-coerced, and throwing-property-access captured-value cases. This preserves the `details.event.reason` dedup discriminator that operator deduplication keys on for those two rows in [Diagnostics](../diagnostics.md) â€” without it, every session-only reason (and every cleanly-cancelled invocation across `event.reason` values on the per-invocation note) would collapse into a single bare-`code` dedup bucket on serialiser throw, splitting nothing rather than preserving the per-reason buckets the structured emission would have produced. The two-token form is otherwise governed by the same MUST-NOT-re-throw and silent-swallow obligations as the bare-`code` fallback above, and remains part of the *same* wrapped sequence per the *Count semantics* pin below (it does not count as a second invocation against per-event "exactly once" bounds). *Construction-site wrap.* The runtime construction of the structured payload's `details.event` field per the [Runtime construction obligation](../diagnostics/diagnostic-shape.md#session-shutdown-details-conventions) â€” the explicit-static-property construction of `{ reason }` for `loom/host/session-shutdown-runtime-degraded` and `{ reason, loom, invocation_id }` for `loom/runtime/cancelled-by-session-shutdown` â€” runs *before* the wrapped serialisation-and-emission sequence above, and the *Hoist obligation* clause's construction of the `detailsEventReason` local likewise lives at payload-construction time outside that wrap. A throw from either construction site (a frozen target object, a throwing setter on a host-shimmed `Object` prototype, an out-of-memory failure during property assignment, or any other foreseeable construction-site failure) is therefore not defended by the serialisation-and-emission wrap. The runtime MUST wrap the entire payload-construction code â€” both the `details.event` construction and the `detailsEventReason` hoist â€” in its own `try`/`catch` so a construction-site throw cannot escape the handler. *Scope.* The construction-site wrap rule applies **only to the two nested-shape codes** (`loom/host/session-shutdown-runtime-degraded`, `loom/runtime/cancelled-by-session-shutdown`), since by the wrap definition above (`details.event` construction + `detailsEventReason` hoist) those are the only payload-construction acts under this rule's coverage. The three flat-`details` teardown-handler codes (`loom/host/session-shutdown-reason-unknown`, `loom/host/session-shutdown-pinned-constant-unreadable`, `loom/host/session-shutdown-teardown-step-failed`) build their `details` payloads outside this wrap â€” via the `String(event.reason)` coercion (Unknown-reason), the closed-discriminator-literal selection on `details.observed` / `details.failure` (pinned-constant-unreadable), and the `String(error)` / `"<unreadable>"` coercion (teardown-step-failed) â€” all of which already tolerate non-string values per the **Unknown-reason rule** above and the *Per-step isolation* paragraph and are by construction infallible at the payload-build site. They therefore cannot enter this catch arm, and this rule pins no construction-site fallback for them; their bare-`code` fallback under the serialisation-and-emission wrap above remains unchanged. On a construction-site throw the catch arm MUST skip the structured serialisation-and-emission sequence and emit directly through a per-code fallback form whose token count differs by row. For `loom/host/session-shutdown-runtime-degraded`, the catch arm emits the two-token form `` console.error(`${code} <unreadable>`) `` â€” interpolating the literal `"<unreadable>"` sentinel rather than the (potentially-unbuilt) `detailsEventReason` local â€” and substitutes that literal for `details.event.reason` (the only payload field this row carries under `details.event`). For `loom/runtime/cancelled-by-session-shutdown` (the per-invocation note), the catch arm MUST instead read `entry.loom` separately from the [`ActiveInvocationRegistry`](./active-invocation-registry.md#active-invocation-registry) entry the per-invocation `finally` already holds and emit the three-token form `` console.error(`${code} ${entry.loom} <unreadable>`) `` â€” substituting the `"<unreadable>"` literal for `details.event.reason` while preserving `details.event.loom`'s per-invocation discrimination via the interpolated `entry.loom` token. The separate `entry.loom` read on the catch arm is no riskier than the per-invocation `finally`'s structured-emission read of the same field, because the [`ActiveInvocationRegistry`](./active-invocation-registry.md#active-invocation-registry) lifetime guarantee above pins `loom` as set on entry creation and never mutated thereafter, observable in its registered form for the entire lifetime of the entry â€” including on a construction-site throw that interrupts payload-construction code reading or stamping other fields. The `"<unreadable>"` literal remains the **single source of truth for catch-arm payload defaults** for `details.event.reason` under this clause on both rows, so the construction-site failure path's `details.event.reason` value space remains inside the captured-value enumeration the **Unknown-reason rule** above pins for that field, and the per-reason dedup buckets the two-token fallback preserves on serialiser throw also cover this path on the same footing as the throwing-property-access and snapshot-failure paths. The per-invocation note's three-token form additionally preserves the per-loom dedup bucket the `details.event.loom` field carries on the structured-emission path, which the runtime-degraded row's two-token form has no field to preserve (that row carries no `loom` under `details.event`). The catch-arm `console.error` invocation is itself subject to the inner-throw swallow obligation below: a throw from `console.error` MUST be swallowed on the same footing as the wrapped sequence's swallowed write, and the construction-site catch arm MUST NOT re-throw or attempt a second emission against the same event. *Construction-site catch-arm self-wrap.* Because the construction-site `console.error` invocation is structurally outside the serialisation-and-emission `try`/`catch` (whose catch arm is the inner-throw swallow obligation below's syntactic locus), the construction-site catch arm cannot inherit that catch arm's swallow by structural containment alone. The construction-site catch arm MUST therefore wrap its own fallback `console.error` call in a dedicated `try`/`catch` whose catch arm silently swallows the throw and returns control to the handler so it can continue to the next sub-step (or, for the **Unknown-reason rule** emission that runs *before* sub-step 1, into sub-step 1) on the same footing as the serialisation-and-emission wrap's swallowed write. No further fallback is available; the handler is the bottom of the stack with no further sink, and the construction-site path's *Count semantics* framing (the construction-site emission counts as the single emission against the per-event bound, per the previous sentence) is preserved on inner-throw exactly as the serialisation-and-emission wrap's bare-`code` fallback is. The construction-site path and the wrapped-sequence path are mutually exclusive â€” at most one of the two emits per event/invocation â€” and the construction-site emission counts as the single emission against the per-event "exactly once" / "at most once" bound the *Count semantics* clause below pins (neither path counts as a second invocation against the other). <a id="pic-emission-swallow"></a> *Handler-isolation swallow.* A throw out of `console.error` (closed stdio, file-descriptor exhaustion, a host that proxies `console` and rejects writes during teardown) MUST be swallowed; the handler MUST continue to the next sub-step (or, for the **Unknown-reason rule** emission that runs *before* sub-step 1, MUST continue into sub-step 1). No further diagnostic is emitted on the inner throw â€” the only sink available to the teardown handler has already failed and the handler is the bottom of the stack with no further fallback. *Count semantics â€” invocation-site framing.* All "exactly once" / "at most once" / "emits exactly one" / "Emitted exactly once" / "Emitted at most once" emission-count bounds that this paragraph's five wrapped teardown-handler diagnostic codes (and the terminal `console.error` of the per-invocation `sendSystemNote` fallback chain referenced at the end of this paragraph) cite â€” wherever they appear, including the **Unknown-reason rule** above's per-event bounds on `loom/host/session-shutdown-reason-unknown` and `loom/host/session-shutdown-pinned-constant-unreadable`, the **Per-step isolation** paragraph's per-failing-call-site bound on `loom/host/session-shutdown-teardown-step-failed`, the [`reason: "new" | "resume" | "fork"` edge-case bullet's](./session-only-degraded-state.md#session-only-reason-degraded-state) per-event-in-this-branch bound on `loom/host/session-shutdown-runtime-degraded` (and that bullet's *Acceptance criteria* per-branch "the handler emits exactly one" obligations, which inherit this framing), the **idempotent** clause above's at-most-once-per-extension-instance-lifetime bound on the same code, the **Per-invocation operator visibility (clean-cancel path)** bullet below's per-cleanly-cancelled-invocation bound on `loom/runtime/cancelled-by-session-shutdown`, sub-step 3's at-most-once-per-event bound on `loom/runtime/reload-teardown-timeout`, and the corresponding row wording in [Diagnostics](../diagnostics.md) for each code â€” are measured at the **handler invocation site**: the bound counts the number of times the handler enters the leaf-owned serialisation-and-emission sequence this rule wraps (the serialisation primitive call followed by the `console.error` call it feeds, including the bare-`code` fallback `console.error(code)` on serialiser throw) for the same `session_shutdown` event (or the same cleanly-cancelled invocation, for the per-invocation note), not the number of rows that reach an operator sink. A host that silently drops the resulting `console.error` write therefore still satisfies the bound â€” the invocation occurred even if the row never reached the sink â€” and a retry-on-inner-throw is forbidden, because issuing a second serialisation-and-emission sequence for the same event after the first inner-threw would breach the at-most-once half of the bound. The bare-`code` fallback on serialiser throw is part of the *same* wrapped sequence as the original structured `console.error` (the catch arm of the serialisation-and-emission `try`/`catch`, per the wrap clause above), not a second invocation, and therefore does not count against the bound. The wrap is around the serialisation-and-emission sequence (serialisation primitive call plus `console.error` call, including the bare-`code` fallback `console.error(code)` above) only; the `String(event.reason)` coercion that builds the **Unknown-reason** diagnostic's `details.observed` payload, and the `String(error)` / `"<unreadable>"` coercion that builds the **Per-step isolation** diagnostic's `details.error` payload, already tolerate non-string values per the rules above and are outside the wrap. Emission ordering is preserved: a swallowed `console.error` throw leaves the handler in the same position it would be after a successful emit (in particular, the **Unknown-reason rule**'s "emits *before* sub-step 1 runs" obligation is satisfied even when the emission inner-throws). The rule is symmetric for all five teardown-handler diagnostic codes; no per-code carve-outs. <a id="pic-emission-visibility"></a> *Pi-side stdio visibility.* Teardown-time `console.error` visibility is **best-effort**: the runtime emits through the channel and wraps each emission per this rule so a swallowed write never unwinds the handler, but the spec makes no normative claim that the write reaches an operator surface (see [Future Considerations â€” loom 1.0 non-goals â€” No reliance on a Pi extension-host stdio-capture facet](../future-considerations/model-changes-and-non-goals.md#pi-stdio-capture-facet) for the host presupposition this rests on, and for the future-work disposition if Pi adds an explicit stdio-capture facet). The same isolation MUST be applied to the terminal `console.error` of the `sendSystemNote` fallback chain when that chain is reached from the per-invocation `finally` of the **Per-invocation operator visibility (clean-cancel path)** paragraph below: a throw from that terminal `console.error` MUST be swallowed and MUST NOT propagate out of the per-invocation `finally`.
