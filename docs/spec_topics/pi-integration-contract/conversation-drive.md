**Conversation drive — prompt mode.** The theta interpreter drives the user's session by:

- Issuing untyped queries via `pi.sendUserMessage(text)` when the agent is idle, or `pi.sendUserMessage(text, { deliverAs: "steer" })` when the agent is mid-stream. `pi.sendUserMessage` returns `void` — the `ExtensionAPI` surface declares both `sendUserMessage(...)` and `sendMessage(...)` as `void`-returning; the `Promise<void>` shape exists only on Pi's `AgentSession.sendUserMessage` (a pure Pi-SDK contrast — a property of Pi's own `AgentSession` class, not a surface theta's subagent drive consumes, since subagent conversations now run as out-of-process child `pi` processes over RPC) and on `ReplacedSessionContext.sendUserMessage` (post-replacement contexts, out of scope here per the *captured `pi` reference* clause below). Only synchronous throws — e.g. argument-validation rejections inside Pi — are observable directly from the call. The runtime MUST NOT `await pi.sendUserMessage(...)` and MUST NOT attach a `.catch` handler — matching the rule pinned for `pi.sendMessage` under **System notes** below. The **primary** prompt-mode transport-failure surface is therefore the post-`waitForIdle()` probe of the driven turn's trailing `assistant` message (its `stopReason` / `errorMessage` fields) pinned under *Error detection* below, not the `pi.sendUserMessage` call site itself. As a narrow secondary mapping, <a id="pic-50"></a> **PIC-50. Prompt-mode synchronous-throw transport mapping.** The runtime MUST map a synchronous throw from `pi.sendUserMessage` (the only failure mode the call surface itself can signal) to `Err(QueryError { kind: "transport", message: <coerced caught-throw string>, http_status: null, provider: <provider>, retryable: false })`. The `message` field MUST be derived from the caught thrown value through the [underlying-error coercion](../diagnostics/placeholder-rendering-b.md#underlying-error-coercion) rule — **not** a raw `.message` read — so a non-Error throw (a thrown non-object, an object whose `.message` is absent or non-string, or a thrown `null`/`undefined`) yields a deterministic non-null `string` (`String(v)`, or the literal `<unreadable>` when that coercion itself throws) rather than `undefined` or a synchronous `TypeError`. It MUST NOT wrap the throw as `theta/runtime/internal-error`, MUST NOT let the throw propagate as an uncaught exception, and MUST NOT swallow it as a successful `Ok("")`. *Non-Error-throw boundary.* A conformance fixture MUST cover the synchronous non-Error throw (a thrown string and a thrown `null`/`undefined`) and assert the synthesised `message` is the coerced non-null string, because that value propagates into the always-log group-A `transport` emission on the [Runtime event channel](./runtime-event-channel.md) and into its `(kind, query_site, message, occurred_at)` dedup key, where an `undefined` would corrupt the exactly-once contract. This is the prompt-mode synchronous-throw counterpart to PIC-51's absent-`errorMessage` fallback below. The `provider` field is the `Api`-shaped [`Model<Api>.api`](./host-interfaces-core.md#model-registry-pin) value of the user session's currently-selected model — read from [`ctx.model`](./host-interfaces-core.md#extensioncontext-interface), the model the driven user turn actually routes through, **not** the theta's resolved `model:` (which is the theta's own off-session / subagent model and does not govern the user session's provider) — attributed identically whether the failure surfaced as a synchronous throw at the call site or via the `errorMessage` probe below, since the `errorMessage` string carries no structured provider field and the **Runtime event channel** always-log emission for `kind: "transport"` requires the field. When `ctx.model` is `undefined` (no user-session model selected) `provider` MUST be the fixed sentinel string `"unknown"` rather than an undefined value. Prompt mode adds no model-unresolved gate counterpart to subagent mode's [pre-spawn model guard](./subagent.md#subagent-pre-spawn-model-guard) (`theta/runtime/subagent-model-unresolved`): prompt mode spawns no subagent child process; the driven turn runs on the user session regardless of the theta's `model:` resolution, so the `undefined` case is handled by the `"unknown"` sentinel at attribution time rather than by blocking dispatch. The same dual-surface mapping applies when `pi.sendUserMessage` is the transport call inside a typed query's free phase; the forced respond turn does **not** use `pi.sendUserMessage` — it runs off-session through pi-ai's `complete()` free function (see the typed-query bullet below), and its provider failures are classified through the [Provider error mapping](./provider-error-mapping.md#provider-error-mapping) table exactly as the binder's `complete()` call is. The `pi` reference here is the `ExtensionAPI` captured by the extension factory (the `pi` parameter of `default function (pi: ExtensionAPI)` from the **Extension entry point** above), which the runtime holds for the lifetime of each theta invocation; `sendUserMessage` is **not** a method on `ExtensionCommandContext`, so the per-handler `ctx` is consulted for `ctx.isIdle()` to choose between the two forms (calling the no-options form mid-stream throws — see Pi's `examples/extensions/send-user-message.ts`) and for `ctx.model` to derive the synthesised `TransportError.provider` (per the `provider` derivation above). theta 1.0 prompt-mode thetas always use `deliverAs: "steer"` when not idle (the theta is the active driver and wants its message delivered to *this* turn); <a id="deliveras-followup-not-emitted"></a>`deliverAs: "followUp"` is reserved for future considerations and is not emitted by theta 1.0. The captured `pi` reference assumes the user session is **not** replaced mid-theta (`ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` invalidate it); theta 1.0 thetas never trigger replacement (see [Future Considerations — Mid-theta user-session replacement](../future-considerations/surface-extensions.md#mid-theta-user-session-replacement) for the post-theta 1.0 re-acquisition seam). Completion is awaited by `await ctx.waitForIdle()` on the `ExtensionCommandContext` passed to the slash-command handler. `waitForIdle` is session-scoped — it observes only the user session bound to this command — and is the prompt-mode driver's authoritative completion signal. *Resolution.* `waitForIdle()` returns `Promise<void>` and resolves once Pi emits `agent_end` for the user session and every awaited `agent_end` listener settles; it [never rejects](#waitforidle-no-rejection-presupposition), so the runtime MUST NOT wrap the `await` in a `try`/`catch` expecting rejection as a failure signal. <a id="prompt-mode-error-detection"></a>*Error detection.* Because `pi.sendUserMessage` returns `void` (no rejection path) and `waitForIdle()` never rejects, every transport- or provider-level failure of the user turn that is not a synchronous throw at the call site surfaces on the driven turn's trailing `assistant` message rather than on any rejection the call surface could raise. This post-`waitForIdle()` probe is the **primary** transport-failure surface for prompt mode; the synchronous-throw branch above is the narrow secondary. Immediately after `waitForIdle()` resolves the runtime reads that turn's final `assistant` message from the same read-only surface the `Ok(string)` extraction uses — `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`, applying the trailing-turn boundary defined at [the untyped-query trailing-turn extraction](#untyped-query-ok-extraction) below — and inspects its `stopReason` and `errorMessage` fields. (`stopReason` and `errorMessage` are members of the externally-owned `AssistantMessage` type declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`, the same declaration the [compact-transcript renderer's `AssistantMessage` arm](./host-interfaces-core.md#sessioncontext-shape) reads and the [binder inference path](./binder-inference.md#binder-inference-call) already probes; theta redefines neither field and treats `stopReason` as opaque apart from the members the classification below keys on — the `"error"` member (PIC-51) and the non-normal terminators classified by [PIC-51b](#pic-51b).) <a id="pic-51"></a> **PIC-51. Prompt-mode `stopReason: "error"` transport mapping.** A `stopReason` of `"error"` MUST be mapped to `Err(QueryError { kind: "transport", message: <errorMessage>, http_status: null, provider: <provider>, retryable: false })` per [Query](../query.md) — the `provider` field is derived per [PIC-50's user-session-model derivation](#pic-50) above; the runtime MUST NOT classify it as `theta/runtime/internal-error` and MUST NOT extract the driven turn as a successful `Ok(string)`. `errorMessage` is optional on `AssistantMessage`, so when it is absent `message` MUST be the fixed string `"provider transport failure"`. The probe reads the trailing turn's `assistant` message before the runtime issues any subsequent `pi.sendUserMessage` on the same context, matching the trailing-turn boundary the `Ok(string)` extraction below relies on. <a id="pic-51b"></a> **PIC-51b. Prompt-mode non-`"error"` terminator classification.** The driven turn can also terminate on a stop reason that is neither `"error"` nor the normal `end_turn` / `stop` / `tool_use` boundary — `"length"` (output/context boundary), a content-filter terminator, or any stop reason the runtime does not otherwise recognise — and it can resolve with no trailing `assistant` message at all. Each of these is a failed turn and MUST NOT be extracted as a successful `Ok(string)`. After the `thetaAbort.signal.aborted` cancellation short-circuit and the PIC-51 `"error"` short-circuit have both been passed, the runtime MUST classify the trailing turn's `stopReason` through the **stop-reason arm only** of [Stop-reason classification](./provider-error-mapping.md#stop-reason-classification): `"length"` maps to `Err(QueryError { kind: "context_overflow", tokens_used: null, tokens_limit: null })`, and every other non-normal terminator maps to `Err(QueryError { kind: "transport", message: <errorMessage, or the fixed string "provider transport failure" when absent>, http_status: null, provider: <provider per PIC-50>, retryable: false })`. The HTTP-status arm of that rule MUST NOT be consulted on this path — it is unreachable from the `ReadonlySessionManager` read surface the probe uses. These two dispositions are applied in a fixed order and are mutually exclusive on whether a trailing `assistant` message exists: **(i)** when a trailing `assistant` message exists, its `stopReason` is classified per the stop-reason arm above (PIC-51 having already short-circuited the `"error"` member) — so a `"length"` terminator maps to `context_overflow`, any other non-normal terminator maps to `transport`, and a normal `end_turn` / `stop` / `tool_use` boundary falls through to PIC-53; **(ii)** only when **no** trailing `assistant` message exists at all after a non-cancelled, non-`"error"` resolution — so there is no `stopReason` to classify — the turn is itself a failure and MUST map to `transport` `Err` with `message: "provider transport failure"` rather than to `Ok("")`. A trailing `assistant` message that is present but carries no text is **not** case (ii): it is classified by its `stopReason` under (i), so an empty-text turn on a normal boundary still reaches PIC-53's `Ok("")` (the pure tool-use turn). Only a trailing turn that terminated on a normal `end_turn` / `stop` / `tool_use` boundary reaches PIC-53's `Ok(string)` extraction below. This is the prompt-mode driven-turn counterpart of the off-session [Stop-reason classification](./provider-error-mapping.md#stop-reason-classification) the binder's `complete()` path applies. *Hang handling.* `waitForIdle()` has no internal deadline; hangs are bounded only by the cancellation path. `thetaAbort.abort()` invokes the unwrapped, Pi-supplied `abort()` on the raw `ExtensionCommandContext` Pi passes to the slash-command handler and the runtime captures for the theta's lifetime — **not** the synthesised tool-execution `ctx.abort()` wrapper, which the [*Per-mode override semantics* table](./host-interfaces-core.md#extensioncontext-interface) below lists as one of the three overridden members and whose body itself calls `thetaAbort.abort()` (re-entering this same path), which cancels the active user run and lets `waitForIdle()` resolve; the runtime guards the propagation with a one-shot flag so a re-entrant `thetaAbort.abort()` (e.g. from an `agent_end` observer that itself observes the abort) does not double-cancel. The post-`waitForIdle` error-state probe runs unconditionally on resolution, but if `thetaAbort.signal.aborted` is true the runtime synthesises `Err(QueryError { kind: "cancelled" })` per [Cancellation](../cancellation.md) instead of reading session error state — even when Pi tore down cleanly and `waitForIdle()` resolved without any error written to the session. The symmetric statement on the abort side lives at [Cancellation — Forwarding into `thetaAbort`](../cancellation.md). The runtime MUST NOT subscribe to the global `pi.on("agent_end", …)` event for query completion: that event fires for every `AgentSession` in the parent process (the user shell session and any concurrent prompt-mode thetas) with no per-session origin marker, so a global handler cross-fires across concurrent prompt-mode thetas and resolves the wait on the wrong turn. Subagent conversations no longer contribute to this cross-fire from the parent side: each runs as an out-of-process child `pi` process whose `agent_end` events arrive on the child's stdout stream, not the parent's process-global `pi.on` — but the ban still holds via the user shell session and any concurrent prompt-mode thetas. <a id="untyped-query-ok-extraction"></a><a id="pic-53"></a> **PIC-53. Untyped-query `Ok(string)` trailing-turn extraction.** The `Ok(string)` value of an untyped query is the accumulated assistant text of the final turn, extracted from the user session after `waitForIdle()` resolves and only once the cancellation short-circuit (`thetaAbort.signal.aborted`, which synthesises `Err(QueryError { kind: "cancelled" })`) the `stopReason: "error"` transport short-circuit (PIC-51), and the non-`"error"` terminator classification ([PIC-51b](#pic-51b)) pinned above have all been passed — the extraction runs downstream of those branches and never reorders or bypasses them, and runs only for a trailing turn that terminated on a normal `end_turn` / `stop` / `tool_use` boundary. The runtime reads the source from the same `ReadonlySessionManager` `Pick<>` members the binder uses: `ctx.sessionManager.getEntries()` and `ctx.sessionManager.getLeafId()` (both enumerated in the `sessionManager` row of **`ExtensionContext` (member surface theta touches)** below), fed to `buildSessionContext(...)` exactly as [Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) does, yielding the same chronological `Message` list. The *final turn* is the trailing turn of that list — the last `user` message (the theta-issued `pi.sendUserMessage` turn) plus every subsequent `assistant` / `toolResult` / `custom` message through the end of the list, using the turn boundary defined in that section; turns from earlier slash-command invocations on the long-lived user session precede that trailing `user` message and are excluded. The string is the `text` content of every `assistant` message in the final turn, concatenated in chronological order with a single `\n` separator between successive assistant messages; the provider-internal `thinking` array and all `toolCalls` entries are omitted, identical to the `[assistant]` body selection in [Compact-transcript format](../binder/binder-model-and-context.md#compact-transcript-format-normative) item 4. A final turn that produced no assistant text (a pure tool-use turn) yields the empty string. `pi-theta` performs no buffering, suppression, or re-styling of Pi's stream; user-visible behaviour is specified in [Slash-Command Invocation — User-visible streaming](../slash-invocation.md#slsh-2).
- Issuing typed queries that enforce the inferred or ascribed schema against the model's response. A typed query is a **two-phase tool-loop conversation**, exactly as specified in [Query — Typed queries are tool-loop-shaped](../query.md): (1) a *free phase* in which the model may call any frontmatter `tools:` entry the same way it does in an untyped query, with `tool_choice` left unconstrained; followed by (2) one *forced respond turn*, dispatched at whichever of two entry points the free phase reaches first — once the model emits a free-phase plain text turn (provider stop reason `end_turn` / `stop`), or on CIO-4's `max_rounds`-final branch (which degenerates to the only turn at `max_rounds: 0`, where no free-phase turn is ever issued) — in which the runtime issues a follow-up user message inlining the lowered response schema and forces the provider's tool choice to the synthesised one-shot respond tool. The synthesised tool's `execute` AJV-validates its input against the lowered schema and resolves the query's promise with the validated value. Only the forced respond turn uses provider tool-choice forcing; every preceding free-phase turn leaves the model free to think aloud and to use the frontmatter callable set, so the `frontmatter.md` "available to the model during query-time tool loops" guarantee, the `ModelToolError` surface, and the "same callable set serves both code and model" invariant all stay live for typed queries. The forced respond turn is the exempt-routed terminator (see [Hard Runtime Ceilings — CIO-4](../hard-ceilings/ceilings-3-and-4.md#cio-4) and [Query — Tool-call loop bound](../query.md)); `tool_loop.max_rounds` is not debited by it. respond-repair follow-ups (see [Query — Schema-validation respond-repair](../query.md)) restart the whole two-phase loop with a fresh `tool_loop` budget, so the model can re-tool before answering a respond-repair turn. The runtime MUST surface the validated payload exactly once and MUST NOT add user-visible turns to the conversation beyond the natural free-phase turns; the forced respond turn is dispatched off-session through pi-ai's `complete()` free function (see below), so it attaches no turn to the driven session and is not rendered in the transcript — the validated payload is delivered to theta code, and prompt-mode user-visibility of any value the author wants the user to see follows the [success-side null-policy](./runtime-event-channel.md#success-side-null-policy). The synthesised one-shot tool follows the same `name` / `label` / `description` / `parameters` / `execute` shape and the same `Type.Unsafe<unknown>(loweredJsonSchema)` `parameters` wrap defined in **Per-theta registration** above, and follows the registration cache + active-set snapshot/restore protocol defined in **Tool-registration lifetime and visibility** above, so the synthesised typed-query tool and the prompt-mode `.theta`-callable Pi tool use a single bridge from theta's lowered JSON Schema to Pi's TypeBox-typed `ToolDefinition` and a single visibility-control mechanism (in subagent mode the child `pi` process builds its own tool definitions from its startup discovery and the `--tools` allowlist, so no `ToolDefinition` is materialised parent-side for the subagent's model-facing tools). The forced respond turn is the only typed-query provider call routed through pi-ai's `complete()` free function, following the same `#binder-inference-call` pattern in [Binder inference call](./binder-inference.md#binder-inference-call); `complete()` is the channel that carries the forced tool choice via its `options.toolChoice` option (the per-provider pi-ai adapter maps `{ type: "tool", name }` to the provider-native payload; the per-provider behavioural presuppositions this relies on are pinned at [Provider compatibility — `complete()` forced-tool presupposition](#complete-forced-tool-presupposition) below), which the session drivers `pi.sendUserMessage` / `session.sendUserMessage` do not expose. The free phase continues to drive through `pi.sendUserMessage` / `session.sendUserMessage`. The full mechanism — including the `before_provider_request` payload-rewrite hook theta 1.0 rejects — is described in [Implementation Notes — Runtime](../implementation-notes.md#runtime).

<a id="driven-turn-commit-ordering-presupposition"></a>*Driven-turn session-commit ordering (consumption posture).* The prompt-mode [error-detection probe](#prompt-mode-error-detection) and the [`Ok(string)` extraction](#untyped-query-ok-extraction) above both read the driven turn's trailing `assistant` message from the `ReadonlySessionManager` surface (`ctx.sessionManager.getEntries()` / `ctx.sessionManager.getLeafId()`) immediately after `await ctx.waitForIdle()` resolves. Both presuppose that by the time Pi settles `waitForIdle()` — i.e. emits the terminal `agent_end` for the user session — the driven turn's `assistant` message, including its `stopReason` and `errorMessage` fields, has already been committed to the session state that surface exposes, so the post-resolution read observes the completed turn rather than a still-pending or partially-written one. theta records this as a theta-side consumption posture because `@earendil-works/pi-coding-agent` encodes no typed contract ordering session-state commit before `waitForIdle()` resolution; a Pi minor that resolved `waitForIdle()` before flushing the driven turn into the `ReadonlySessionManager` surface would make the probe read a stale or empty trailing turn — missing a real `stopReason: "error"` transport failure, or extracting an incomplete `Ok(string)` — with no build-time SDK surface-inventory signal. Detection therefore routes to editorial review per item (ac) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure).

<a id="user-visible-streaming-ordering-presupposition"></a>*User-visible streaming ordering (consumption posture).* [`slash-invocation.md` SLSH-2](../slash-invocation.md#slsh-2)'s user-visible-streaming guarantee — that in prompt mode the user sees assistant tokens unfold in the transcript while the theta is still mid-query, before the theta interpreter resumes on `ctx.waitForIdle()` resolution — presupposes that Pi's TUI streams assistant tokens into the user transcript as they arrive, before Pi fires the terminal `agent_end` that settles `ctx.waitForIdle()`, rather than buffering the whole assistant response and committing it to the transcript only at `agent_end`. theta performs no buffering of its own on this path, so the before-resume visibility is entirely Pi-side: were a Pi minor to buffer the streamed response until `agent_end`, the tokens would appear in the transcript only at the instant `waitForIdle()` resolves and the theta resumes, falsifying SLSH-2's property that the user sees the response unfold while the theta is still mid-query. theta records this as a theta-side consumption posture because `@earendil-works/pi-coding-agent` encodes no typed contract ordering transcript-streaming before `waitForIdle()` resolution, and the property carries no build-time SDK surface-inventory signal. Detection therefore routes to editorial review per item (ak) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure).

<a id="waitforidle-no-rejection-presupposition"></a>*`ctx.waitForIdle()` no-rejection (consumption posture).* The prompt-mode *Resolution* clause's `MUST NOT` against wrapping `await ctx.waitForIdle()` in a `try`/`catch`, the [error-detection probe](#prompt-mode-error-detection)'s reliance on the driven turn's trailing-`assistant` `stopReason: "error"` surface as the post-resolution transport-failure signal, and the *Hang handling* cancellation path that lets `waitForIdle()` resolve on abort all presuppose that `ctx.waitForIdle()` settles only by resolving — it never rejects. theta records this as a theta-side consumption posture because `@earendil-works/pi-coding-agent` types `waitForIdle()` as `Promise<void>`, a surface that does not encode the no-rejection behaviour; a Pi minor that made `waitForIdle()` reject — e.g. surfacing a provider transport failure as a rejection rather than on the driven turn's trailing `assistant` message — would silently falsify the `MUST NOT` and bypass the trailing-turn error probe, with no build-time SDK surface-inventory signal. Detection therefore routes to editorial review per item (al) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure).

<a id="driven-turn-failure-universe-presupposition"></a>*Driven-turn failure-universe → terminal-stop-reason mapping (consumption posture).* [PIC-51](#pic-51) and [PIC-51b](#pic-51b) presuppose that every non-synchronous-throw failure of the driven user turn surfaces on the trailing `assistant` message's `stopReason` (and `errorMessage`) — i.e. the failure universe is exhausted by the `"error"` / `"length"` / content-filter / unrecognised-terminator set plus the absent-trailing-message case, all readable from the post-`waitForIdle()` `ReadonlySessionManager` surface. theta records this as a theta-side consumption posture because `@earendil-works/pi-coding-agent` encodes no typed contract that the driven turn's terminal `stopReason` faithfully reflects a provider failure: a Pi minor that resolved a failed turn under a normal `end_turn` / `stop` reason, or surfaced a new terminal stop reason theta does not classify, would route a real failure into [PIC-53](#untyped-query-ok-extraction)'s `Ok(string)` extraction with no build-time SDK surface-inventory signal. Detection therefore routes to editorial review against the stop-reason arm of [Stop-reason classification](./provider-error-mapping.md#stop-reason-classification), whose provider-error fixtures are re-run per item (i) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure).

<a id="pic-18"></a> **PIC-18. Prompt-mode turn-lifecycle event subscription.** The prompt-mode driver observes Pi's turn-lifecycle events — `tool_call`, `tool_result`, `message_update`, `turn_end`, and `agent_end` — by registering handlers through `pi.on(event, handler)` on the factory-captured `ExtensionAPI` reference (the `pi` parameter of `default function (pi: ExtensionAPI)` from the [**Extension entry point**](./extension-bootstrap-and-per-theta.md#extension-entry-point) above), the same `pi.on` surface the `session_shutdown` subscription uses (see [Registration steps](./registration-steps.md), step 4). `pi.on` is the only event-subscription surface available in prompt mode; subagent mode does not use `pi.on` for query completion — it reads the child `pi` process's RPC `agent_end` events off the child's stdout stream (see [Conversation drive — subagent mode — Drive mapping (RPC)](./subagent.md#subagent-drive-mapping)) — and neither `ExtensionContext` nor `ExtensionCommandContext` exposes an event-subscription member. `pi.on` is declared on `ExtensionAPI` at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)); it is overloaded on a closed event-name union, of which theta 1.0 consumes exactly the five members above, and each member's handler payload type (for instance `AgentEndEvent`, `ToolCallEvent`) is declared in the same file. theta redefines none of these externally-owned types and treats each event payload as opaque apart from the cancellation-relevant fields the forwarding rules read. The pinned `ExtensionAPI` surface exposes no `pi.subscribe` member and no `off` / unsubscribe member, and each `pi.on(...)` overload returns `void`; the prompt-mode turn-lifecycle handlers are therefore process-global, registered once and persisting for the extension instance's lifetime. The per-invocation "subscribes for the duration of the theta run" behaviour pinned at [Cancellation — Forwarding into `thetaAbort`](../cancellation.md) is realised by these persistent handlers forwarding the active invocation's captured `ctx.signal` into its `thetaAbort`, not by per-run attach / detach of the `pi.on` registration. Because `pi.on` events are process-global and carry no per-session origin marker (the same property that bars `pi.on("agent_end", …)` as a query-completion signal under *Error detection* above), the prompt-mode driver MUST NOT use these events to resolve query completion; their sole prompt-mode role is cancellation forwarding, so cross-fire from an unrelated session's turn event is harmless — it triggers only a re-check of a non-aborted captured signal. The process-global / no-per-session-origin-marker property this paragraph depends on is re-audited against each candidate Pi minor per [bump-checklist item (ah)](./version-bump-step2.md#bump-checklist-pi-on-process-global).

<a id="pic-2"></a> **PIC-2. Prompt-mode sequential execution.** Prompt-mode bodies execute strictly sequentially within a single user session: at most one prompt-mode body — top-level slash dispatches and nested prompt → prompt `invoke(...)` calls alike — holds an open `pi.setActiveTools` snapshot/restore window at a time. This is a *derived* property rather than an independent runtime obligation: it follows from Pi's per-session slash-handler serialisation guarantee (snapshot/restore [behavioural precondition (b)](./tool-registration-lifetime.md#snapshot-restore-pi-behavioural-preconditions), under which two theta-invocation slash dispatches against the same session cannot overlap their snapshot/restore windows), composed with the theta-runtime-owned guarantees that (i) prompt-mode `.theta` callees in another theta's `tools:` are rejected at load time per [Parameters and Frontmatter — `tools`](../frontmatter/frontmatter-fields-a.md#tools), (ii) `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns and theta 1.0 exposes no parallel-`invoke` surface per [Invocation — Cross-mode semantics](../invocation.md), and (iii) the `subagent → prompt` fan-out path into the user session is closed per the same [Invocation — Cross-mode semantics](../invocation.md) rule. The three potential sources of in-session overlap therefore produce genuine concurrency only between subagent-mode invocations, per [Concurrent-invocation isolation and concurrency disposition](./subagent.md#concurrent-invocation-isolation).

> **theta 1.0 seam — mid-theta user-session replacement.** <a id="theta-1-0-seam-mid-theta-user-session-replacement"></a> The prompt-mode driver MUST read `pi` from a single factory-captured reference (the `pi: ExtensionAPI` parameter of `default function (pi: ExtensionAPI)` from the [**Extension entry point**](./extension-bootstrap-and-per-theta.md#extension-entry-point) above) held for the lifetime of each theta invocation; consumers MUST NOT re-acquire `pi` from any other surface (e.g. by reaching into `ExtensionCommandContext` or by re-entering the factory) during a theta invocation. The deferred *Mid-theta user-session replacement* extension in [Future Considerations](../future-considerations/surface-extensions.md#mid-theta-user-session-replacement) introduces a `withSession`-fronted re-acquisition of the captured reference before the next `sendUserMessage` (so a theta-driven `ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` call no longer invalidates the captured `pi`) at exactly that single capture site; duplicating the capture across call sites in theta 1.0 is breaking, because the deferred re-acquisition hook would then have to be threaded into every duplicate.

**Provider compatibility for typed queries.** The forced respond turn requires the provider to honour named-tool forcing. theta 1.0 supports the providers for which pi-ai exposes a named-tool `toolChoice` mapping: `anthropic-messages`, `openai-completions`, `mistral`, and `amazon-bedrock`. At theta-load time, when a theta contains any typed-query expression and its resolved `model:` routes through a provider outside this set, the runtime emits `theta/load/typed-query-unsupported-provider` (warning) naming the model; <!-- the warning's payload is exactly the model + provider carried by its diagnostics/code-registry-load.md message template; the former "listing the typed-query expression positions" clause was dropped as no positions format was ever pinned --> the theta still loads, and any typed query at runtime returns `Err(QueryError { kind: "transport", message: "<provider> does not support forced tool-use; typed queries unavailable", http_status: null, provider: "<provider>", retryable: false })`. Here `<provider>` is the resolved [`Model<Api>.api`](./host-interfaces-core.md#model-registry-pin) value of the theta's resolved `model:` (the `api`-shaped form, e.g. `mistral`), so the synthesised `provider` field carries the same `api`-shaped value as a classifier-produced [`TransportError.provider`](../errors-and-results/queryerror-variants.md#queryerror-variants). Untyped queries against the same theta continue to work. Future widening (Gemini, OpenAI Responses, weak local models via JSON-mode fallback) is listed in [Future Considerations](../future-considerations.md).

<a id="complete-forced-tool-presupposition"></a>*`complete()` forced-tool behavioural presupposition.* The forced respond turn above and the binder's [structured-output call](./binder-inference.md#binder-inference-call) both depend on two behavioural properties of `@earendil-works/pi-ai`'s `complete()` free function for each theta 1.0-supported provider (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`): (1) calling with `options.toolChoice = { type: "tool", name }` forces the named tool — the per-provider adapter maps that option to the provider-native forced-tool payload rather than leaving the choice unconstrained — and (2) the call attaches no turn to the concurrently-driven user session (nor to any subagent child conversation), since it runs off-session in the parent directly off the returned `Promise<AssistantMessage>` (per [Binder inference call](./binder-inference.md#binder-inference-call)). Neither property is part of `@earendil-works/pi-ai`'s typed surface, so a pi-ai minor that silently regresses either one is caught by neither the SDK surface-inventory test nor the pi-ai version pin; the regression would surface only as the [diagnostic limitation](#pic-typed-query-noncompliance) above misclassifying the model as non-compliant. Detection therefore routes to editorial review: the per-provider forced-tool fixtures and the off-session-turn inspection SHOULD be re-run on each Pi minor bump, per item (u) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure). *Authoritative source.* Neither property is documented on `@earendil-works/pi-ai`'s typed surface; at the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) `complete()` is observed at `dist/stream.js` in `@earendil-works/pi-ai` and its per-provider forced-tool mapping (`options.toolChoice` → provider-native payload) at that package's per-provider adapter modules under `dist/providers/` (e.g. `dist/providers/amazon-bedrock.js`, `dist/providers/anthropic.js`, `dist/providers/openai-completions.js`, `dist/providers/mistral.js`), with the per-provider behaviour captured in the `npm test` forced-tool fixtures re-run per item (u) (**known-fragile evidence**: these bundler-output paths are diffable artefacts at the pin rather than load-bearing claims about their stable shape across patches; a rename or re-bundle routes the audit's textual-diffability failure to the established version-bump recovery path).

<a id="complete-retry-and-cancellation-presupposition"></a>*`complete()` retry-timing and cancellation behavioural presupposition.* Beyond the forced-tool and off-session-turn properties pinned above, the binder's [structured-output call](./binder-inference.md#binder-inference-call) — and any other theta call routed through `@earendil-works/pi-ai`'s `complete()` free function — depends on two further behavioural properties of `complete()` for each theta 1.0-supported provider: (1) `complete()` drives inter-attempt timing internally from its `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs` options, including honouring any server-requested wait such as an HTTP `Retry-After`, so that theta re-issuing each budgeted retry immediately — per the binder's [per-invocation retry budget](../binder/determinism-cancellation-failure.md#per-invocation-retry-budget) — delegates per-attempt backoff to `complete()` rather than implementing its own delay; and (2) `complete()` honours `options.signal` for cancellation, aborting the in-flight provider call and rejecting the returned `Promise<AssistantMessage>` when the supplied `AbortSignal` (the binder call passes `thetaAbort.signal` via the `options.signal` bullet of [Binder inference call](./binder-inference.md#binder-inference-call)) aborts, so theta's [cancellation](../cancellation.md) chain tears the off-session provider call down rather than leaving it running to completion. Neither property is part of `@earendil-works/pi-ai`'s typed surface, so a pi-ai minor that silently regresses either one is caught by neither the SDK surface-inventory test nor the pi-ai version pin: a retry-timing regression would change the binder's observed inter-attempt latency without changing any theta-visible type, and a `signal`-honouring regression would leave a cancelled binder call running until it resolves on its own. Detection therefore routes to editorial review: the per-provider retry-timing and cancellation fixtures SHOULD be re-run on each Pi minor bump, per items (aa) and (ab) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure). *Authoritative source.* Neither property is documented on `@earendil-works/pi-ai`'s typed surface; at the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) `complete()`'s retry/backoff handling and its `options.signal` cancellation handling are observed at `dist/stream.js` in `@earendil-works/pi-ai`, with the per-provider behaviour captured in the `npm test` retry-timing and cancellation fixtures re-run per items (aa) and (ab) (**known-fragile evidence**: this bundler-output path is a diffable artefact at the pin rather than a load-bearing claim about its stable shape across patches; a rename or re-bundle routes the audit's textual-diffability failure to the established version-bump recovery path).

> **theta 1.0 seam — typed-query supported provider set.** <a id="theta-1-0-seam-typed-query-supported-provider-set"></a> The theta 1.0 typed-query-supported provider set (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`) is referenced by the load-time `theta/load/typed-query-unsupported-provider` warning emitter and the runtime guard that synthesises the unsupported-provider `TransportError`. The deferred *Typed-query support for providers without named-tool forcing* extension in [Future Considerations](../future-considerations.md) widens this set (and adds a JSON-mode / system-prompt fallback for the new entries) without touching the typed-query behavioural contract above; widening the set is the only behavioural surface the extension perturbs. *Non-normative implementation note.* theta 1.0 expects the provider set to be exposed as a single named runtime constant referenced by every consumer (rather than inlining the four-element list at each site) so there is one source of truth to widen; this single-constant factoring is a maintainability convention, not an observable conformance point.

<a id="pic-typed-query-noncompliance"></a> **theta 1.0 diagnostic limitation.** When a typed query routes through a provider in the theta 1.0 supported set (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`) — most commonly an OpenAI-compatible local backend — but the specific model ignores forced-tool selection on the forced respond turn (by emitting plain text, or by invoking a tool whose name is not `__theta_respond_<slug>`), the runtime cannot distinguish this from any other case where the model fails to call the respond tool. The runtime classifies the case as a schema-validation failure and routes it through the respond-repair pipeline; it does NOT surface as `tool_loop_exhausted`, which is unreachable on the forced respond turn because that turn is the exempt-routed terminator dispatched by CIO-4's `max_rounds`-final branch (see [Hard Runtime Ceilings — CIO-4](../hard-ceilings/ceilings-3-and-4.md#cio-4)). For each forced respond turn that does not invoke `__theta_respond_<slug>`, the runtime synthesises a single `ValidationIssue` and feeds it through that pipeline. The non-compliance branch is observable in the synthesised issue's `message` literal; `path` and `schema_keyword` are uniform across both branches. The normative synthesised shapes — the plain-text and wrong-tool `ValidationIssue` literals, the terminal `Err(QueryError { kind: "validation", cause: "schema_validation", ... })` shape, the per-attempt accounting, and the `raw_response` null-vs-body disambiguation — are defined at [Errors and Results — Forced-respond non-compliance synthesised shapes (ERR-17)](../errors-and-results/queryerror-variants.md#err-17). Separating provider-level from model-level non-compliance is out of scope for theta 1.0.
