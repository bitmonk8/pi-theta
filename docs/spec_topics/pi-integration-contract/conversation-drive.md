# Conversation drive

<a id="no-extra-mediation"></a>

**No additional access channels.** The runtime interposes no privilege layer between loom code and the Pi extension host. Loom-side filesystem, network, and Pi-API access are bounded only by Pi's extension-host grants and by the per-loom `tools:` allowlist, whose active-set enforcement is specified by [Tool registration lifetime — Active-set allowlist gating (PIC-17)](./tool-registration-lifetime.md#pic-17). The runtime MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy) in any loom 1.x release; widening or narrowing this rule is a major-version concern. The `tools:` allowlist constrains the *model's* reachable callable set, not the host process: a loom that declares a high-privilege callable (e.g. `bash`) exposes the full underlying capability of that callable to its model. Host-side denials of filesystem, network, or Pi-API access reach loom code through the tool that issued the request: a thrown or `isError: true` return is mapped to `Err(QueryError { kind: "code_tool", cause: "execution", ... })` per [Tool Calls — Failures](../tool-calls.md) and [Tool execution from loom code](./host-interfaces-core.md#tool-execution-from-loom-code) below; silent success on denial is forbidden.

**Conversation drive — prompt mode.** The loom interpreter drives the user's session by:

- Issuing untyped queries via `pi.sendUserMessage(text)` when the agent is idle, or `pi.sendUserMessage(text, { deliverAs: "steer" })` when the agent is mid-stream. `pi.sendUserMessage` returns `void` — the `ExtensionAPI` surface declares both `sendUserMessage(...)` and `sendMessage(...)` as `void`-returning; the `Promise<void>` shape exists only on `AgentSession.sendUserMessage` (subagent mode, see **Conversation drive — subagent mode** below and the [`AgentSession` member surface](./host-interfaces-core.md#agentsession-interface)) and on `ReplacedSessionContext.sendUserMessage` (post-replacement contexts, out of scope here per the *captured `pi` reference* clause below). Only synchronous throws — e.g. argument-validation rejections inside Pi — are observable directly from the call. The runtime MUST NOT `await pi.sendUserMessage(...)` and MUST NOT attach a `.catch` handler — matching the rule pinned for `pi.sendMessage` under **System notes** below. The **primary** prompt-mode transport-failure surface is therefore the post-`waitForIdle()` probe of the driven turn's trailing `assistant` message (its `stopReason` / `errorMessage` fields) pinned under *Error detection* below, not the `pi.sendUserMessage` call site itself. As a narrow secondary mapping, a synchronous throw from `pi.sendUserMessage` (the only failure mode the call surface itself can signal) is mapped to `Err(QueryError { kind: "transport", message: <error.message>, http_status: null, provider: <resolved-model-provider>, retryable: false })` — not to `loom/runtime/internal-error`. The `provider` field is populated from the loom's resolved `model:` regardless of which surface (synchronous throw at the call site vs. the `errorMessage` probe) detected the failure, since the `errorMessage` string carries no structured provider field and the **Runtime event channel** always-log emission for `kind: "transport"` requires the field. The same dual-surface mapping applies when `pi.sendUserMessage` is the transport call inside a typed query's free phase; the forced respond turn does **not** use `pi.sendUserMessage` — it runs off-session through pi-ai's `complete()` free function (see the typed-query bullet below), and its provider failures are classified through the [Provider error mapping](./provider-error-mapping.md#provider-error-mapping) table exactly as the binder's `complete()` call is. The `pi` reference here is the `ExtensionAPI` captured by the extension factory (the `pi` parameter of `default function (pi: ExtensionAPI)` from the **Extension entry point** above), which the runtime holds for the lifetime of each loom invocation; `sendUserMessage` is **not** a method on `ExtensionCommandContext`, so the per-handler `ctx` is consulted only for `ctx.isIdle()` to choose between the two forms (calling the no-options form mid-stream throws — see Pi's `examples/extensions/send-user-message.ts`). loom 1.0 prompt-mode looms always use `deliverAs: "steer"` when not idle (the loom is the active driver and wants its message delivered to *this* turn); `deliverAs: "followUp"` is reserved for future considerations and is not emitted by loom 1.0. The captured `pi` reference assumes the user session is **not** replaced mid-loom (`ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` invalidate it); loom 1.0 looms never trigger replacement (see [Future Considerations — Mid-loom user-session replacement](../future-considerations/surface-extensions.md#mid-loom-user-session-replacement) for the post-loom 1.0 re-acquisition seam). Completion is awaited by `await ctx.waitForIdle()` on the `ExtensionCommandContext` passed to the slash-command handler. `waitForIdle` is session-scoped — it observes only the user session bound to this command — and is the prompt-mode driver's authoritative completion signal. *Resolution.* `waitForIdle()` returns `Promise<void>` and resolves once Pi emits `agent_end` for the user session and every awaited `agent_end` listener settles; it [never rejects](#waitforidle-no-rejection-presupposition), so the runtime MUST NOT wrap the `await` in a `try`/`catch` expecting rejection as a failure signal. <a id="prompt-mode-error-detection"></a>*Error detection.* Because `pi.sendUserMessage` returns `void` (no rejection path) and `waitForIdle()` never rejects, every transport- or provider-level failure of the user turn that is not a synchronous throw at the call site surfaces on the driven turn's trailing `assistant` message rather than on any rejection the call surface could raise. This post-`waitForIdle()` probe is the **primary** transport-failure surface for prompt mode; the synchronous-throw branch above is the narrow secondary. Immediately after `waitForIdle()` resolves the runtime reads that turn's final `assistant` message from the same read-only surface the `Ok(string)` extraction uses — `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`, applying the trailing-turn boundary defined at [the untyped-query trailing-turn extraction](#untyped-query-ok-extraction) below — and inspects its `stopReason` and `errorMessage` fields. (`stopReason` and `errorMessage` are members of the externally-owned `AssistantMessage` type declared at `@earendil-works/pi-ai`'s `dist/types.d.ts`, the same declaration the [compact-transcript renderer's `AssistantMessage` arm](./host-interfaces-core.md#sessioncontext-shape) reads and the [binder inference path](./binder-inference.md#binder-inference-call) already probes; loom redefines neither field and treats `stopReason` as opaque apart from the `"error"` member it keys on.) A `stopReason` of `"error"` maps to `Err(QueryError { kind: "transport", message: <errorMessage>, http_status: null, provider: <resolved-model-provider>, retryable: false })` per [Query](../query.md); `errorMessage` is optional on `AssistantMessage`, so when it is absent `message` is the fixed string `"provider transport failure"`. The probe reads the trailing turn's `assistant` message before the runtime issues any subsequent `pi.sendUserMessage` on the same context, matching the trailing-turn boundary the `Ok(string)` extraction below relies on. *Hang handling.* `waitForIdle()` has no internal deadline; hangs are bounded only by the cancellation path. `loomAbort.abort()` invokes `ctx.abort()` on the captured slash-command handler context (the `ExtensionCommandContext.abort` member already in the forwarded set per **`ExtensionContext` (member surface loom touches)** below), which cancels the active user run and lets `waitForIdle()` resolve; the runtime guards the propagation with a one-shot flag so a re-entrant `loomAbort.abort()` (e.g. from an `agent_end` observer that itself observes the abort) does not double-cancel. The post-`waitForIdle` error-state probe runs unconditionally on resolution, but if `loomAbort.signal.aborted` is true the runtime synthesises `Err(QueryError { kind: "cancelled" })` per [Cancellation](../cancellation.md) instead of reading session error state — even when Pi tore down cleanly and `waitForIdle()` resolved without any error written to the session. The symmetric statement on the abort side lives at [Cancellation — Forwarding into `loomAbort`](../cancellation.md). The runtime MUST NOT subscribe to the global `pi.on("agent_end", …)` event for query completion: that event fires for every `AgentSession` in the process (the user shell session and any concurrent subagent-mode sessions) with no per-session origin marker, so a global handler cross-fires across concurrent looms or sibling subagents and resolves the wait on the wrong turn. <a id="untyped-query-ok-extraction"></a>The `Ok(string)` value of an untyped query is the accumulated assistant text of the final turn, extracted from the user session after `waitForIdle()` resolves and only once the cancellation short-circuit (`loomAbort.signal.aborted`, which synthesises `Err(QueryError { kind: "cancelled" })`) and the `stopReason: "error"` transport short-circuit pinned above have both been passed — the extraction runs downstream of those branches and never reorders or bypasses them. The runtime reads the source from the same `ReadonlySessionManager` `Pick<>` members the binder uses: `ctx.sessionManager.getEntries()` and `ctx.sessionManager.getLeafId()` (both enumerated in the `sessionManager` row of **`ExtensionContext` (member surface loom touches)** below), fed to `buildSessionContext(...)` exactly as [Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) does, yielding the same chronological `Message` list. The *final turn* is the trailing turn of that list — the last `user` message (the loom-issued `pi.sendUserMessage` turn) plus every subsequent `assistant` / `toolResult` / `custom` message through the end of the list, using the turn boundary defined in that section; turns from earlier slash-command invocations on the long-lived user session precede that trailing `user` message and are excluded. The string is the `text` content of every `assistant` message in the final turn, concatenated in chronological order with a single `\n` separator between successive assistant messages; the provider-internal `thinking` array and all `toolCalls` entries are omitted, identical to the `[assistant]` body selection in [Compact-transcript format](../binder/binder-model-and-context.md#compact-transcript-format-normative) item 4. A final turn that produced no assistant text (a pure tool-use turn) yields the empty string. `pi-loom` performs no buffering, suppression, or re-styling of Pi's stream; user-visible behaviour is specified in [Slash-Command Invocation — User-visible streaming](../slash-invocation.md#slsh-2).
- Issuing typed queries that enforce the inferred or ascribed schema against the model's response. A typed query is a **two-phase tool-loop conversation**, exactly as specified in [Query — Typed queries are tool-loop-shaped](../query.md): (1) a *free phase* in which the model may call any frontmatter `tools:` entry the same way it does in an untyped query, with `tool_choice` left unconstrained; followed by (2) one *forced respond turn* once the model emits a plain text turn, in which the runtime issues a follow-up user message inlining the lowered response schema and forces the provider's tool choice to the synthesised one-shot respond tool. The synthesised tool's `execute` AJV-validates its input against the lowered schema and resolves the query's promise with the validated value. Only the forced respond turn uses provider tool-choice forcing; every preceding free-phase turn leaves the model free to think aloud and to use the frontmatter callable set, so the `frontmatter.md` "available to the model during query-time tool loops" guarantee, the `ModelToolError` surface, and the "same callable set serves both code and model" invariant all stay live for typed queries. The forced respond turn is the exempt-routed terminator dispatched by CIO-4's `max_rounds`-final branch (see [Hard Runtime Ceilings — CIO-4](../hard-ceilings/ceilings-3-and-4.md#cio-4) and [Query — Tool-call loop bound](../query.md)); the runtime issues it on every typed query reaching that branch — including `max_rounds: 0` where it is the only turn — and `tool_loop.max_rounds` is not debited by it. respond-repair follow-ups (see [Query — Schema-validation respond-repair](../query.md)) restart the whole two-phase loop with a fresh `tool_loop` budget, so the model can re-tool before answering a respond-repair turn. The runtime MUST surface the validated payload exactly once and MUST NOT add user-visible turns to the conversation beyond the natural free-phase turns; the forced respond turn is dispatched off-session through pi-ai's `complete()` free function (see below), so it attaches no turn to the driven session and is not rendered in the transcript — the validated payload is delivered to loom code, and prompt-mode user-visibility of any value the author wants the user to see follows the [success-side null-policy](./runtime-event-channel.md#success-side-null-policy). The synthesised one-shot tool follows the same `name` / `label` / `description` / `parameters` / `execute` shape and the same `Type.Unsafe<unknown>(loweredJsonSchema)` `parameters` wrap defined in **Per-loom registration** above, and follows the registration cache + active-set snapshot/restore protocol defined in **Tool-registration lifetime and visibility** above, so the synthesised typed-query tool, the `.loom`-callable Pi tool, and the subagent-mode `customTools` entries all use a single bridge from loom's lowered JSON Schema to Pi's TypeBox-typed `ToolDefinition` and a single visibility-control mechanism. The forced respond turn is the only typed-query provider call routed through pi-ai's `complete()` free function, following the same `#binder-inference-call` pattern in [Binder inference call](./binder-inference.md#binder-inference-call); `complete()` is the channel that carries the forced tool choice via its `options.toolChoice` option (the per-provider pi-ai adapter maps `{ type: "tool", name }` to the provider-native payload; the per-provider behavioural presuppositions this relies on are pinned at [Provider compatibility — `complete()` forced-tool presupposition](#complete-forced-tool-presupposition) below), which the session drivers `pi.sendUserMessage` / `session.sendUserMessage` do not expose. The free phase continues to drive through `pi.sendUserMessage` / `session.sendUserMessage`. The full mechanism — including the `before_provider_request` payload-rewrite hook loom 1.0 rejects — is described in [Implementation Notes — Runtime](../implementation-notes.md#runtime).

<a id="driven-turn-commit-ordering-presupposition"></a>*Driven-turn session-commit ordering (consumption posture).* The prompt-mode [error-detection probe](#prompt-mode-error-detection) and the [`Ok(string)` extraction](#untyped-query-ok-extraction) above both read the driven turn's trailing `assistant` message from the `ReadonlySessionManager` surface (`ctx.sessionManager.getEntries()` / `ctx.sessionManager.getLeafId()`) immediately after `await ctx.waitForIdle()` resolves. Both presuppose that by the time Pi settles `waitForIdle()` — i.e. emits the terminal `agent_end` for the user session — the driven turn's `assistant` message, including its `stopReason` and `errorMessage` fields, has already been committed to the session state that surface exposes, so the post-resolution read observes the completed turn rather than a still-pending or partially-written one. loom records this as a loom-side consumption posture because `@earendil-works/pi-coding-agent` encodes no typed contract ordering session-state commit before `waitForIdle()` resolution; a Pi minor that resolved `waitForIdle()` before flushing the driven turn into the `ReadonlySessionManager` surface would make the probe read a stale or empty trailing turn — missing a real `stopReason: "error"` transport failure, or extracting an incomplete `Ok(string)` — with no build-time SDK surface-inventory signal. Detection therefore routes to editorial review per item (ac) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure).

<a id="user-visible-streaming-ordering-presupposition"></a>*User-visible streaming ordering (consumption posture).* [`slash-invocation.md` SLSH-2](../slash-invocation.md#slsh-2)'s user-visible-streaming guarantee — that in prompt mode the user sees assistant tokens unfold in the transcript while the loom is still mid-query, before the loom interpreter resumes on `ctx.waitForIdle()` resolution — presupposes that Pi's TUI streams assistant tokens into the user transcript as they arrive, before Pi fires the terminal `agent_end` that settles `ctx.waitForIdle()`, rather than buffering the whole assistant response and committing it to the transcript only at `agent_end`. loom performs no buffering of its own on this path, so the before-resume visibility is entirely Pi-side: were a Pi minor to buffer the streamed response until `agent_end`, the tokens would appear in the transcript only at the instant `waitForIdle()` resolves and the loom resumes, falsifying SLSH-2's property that the user sees the response unfold while the loom is still mid-query. loom records this as a loom-side consumption posture because `@earendil-works/pi-coding-agent` encodes no typed contract ordering transcript-streaming before `waitForIdle()` resolution, and the property carries no build-time SDK surface-inventory signal. Detection therefore routes to editorial review per item (ak) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure).

<a id="waitforidle-no-rejection-presupposition"></a>*`ctx.waitForIdle()` no-rejection (consumption posture).* The prompt-mode *Resolution* clause's `MUST NOT` against wrapping `await ctx.waitForIdle()` in a `try`/`catch`, the [error-detection probe](#prompt-mode-error-detection)'s reliance on the driven turn's trailing-`assistant` `stopReason: "error"` surface as the post-resolution transport-failure signal, and the *Hang handling* cancellation path that lets `waitForIdle()` resolve on abort all presuppose that `ctx.waitForIdle()` settles only by resolving — it never rejects. loom records this as a loom-side consumption posture because `@earendil-works/pi-coding-agent` types `waitForIdle()` as `Promise<void>`, a surface that does not encode the no-rejection behaviour; a Pi minor that made `waitForIdle()` reject — e.g. surfacing a provider transport failure as a rejection rather than on the driven turn's trailing `assistant` message — would silently falsify the `MUST NOT` and bypass the trailing-turn error probe, with no build-time SDK surface-inventory signal. Detection therefore routes to editorial review per item (al) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure).

<a id="pic-18"></a> **PIC-18. Prompt-mode turn-lifecycle event subscription.** The prompt-mode driver observes Pi's turn-lifecycle events — `tool_call`, `tool_result`, `message_update`, `turn_end`, and `agent_end` — by registering handlers through `pi.on(event, handler)` on the factory-captured `ExtensionAPI` reference (the `pi` parameter of `default function (pi: ExtensionAPI)` from the **Extension entry point** above), the same `pi.on` surface the `session_shutdown` subscription uses (see [Registration steps](./registration-steps.md), step 4). `pi.on` is the only event-subscription surface available in prompt mode: `AgentSession.subscribe` is scoped to a spawned `AgentSession` and is subagent-only (see [`AgentSession` member surface](./host-interfaces-core.md#agentsession-interface)), and neither `ExtensionContext` nor `ExtensionCommandContext` exposes an event-subscription member. `pi.on` is declared on `ExtensionAPI` at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)); it is overloaded on a closed event-name union, of which loom 1.0 consumes exactly the five members above, and each member's handler payload type (for instance `AgentEndEvent`, `ToolCallEvent`) is declared in the same file. loom redefines none of these externally-owned types and treats each event payload as opaque apart from the cancellation-relevant fields the forwarding rules read. The pinned `ExtensionAPI` surface exposes no `pi.subscribe` member and no `off` / unsubscribe member, and each `pi.on(...)` overload returns `void`; the prompt-mode turn-lifecycle handlers are therefore process-global, registered once and persisting for the extension instance's lifetime. The per-invocation "subscribes for the duration of the loom run" behaviour pinned at [Cancellation — Forwarding into `loomAbort`](../cancellation.md) is realised by these persistent handlers forwarding the active invocation's captured `ctx.signal` into its `loomAbort`, not by per-run attach / detach of the `pi.on` registration. Because `pi.on` events are process-global and carry no per-session origin marker (the same property that bars `pi.on("agent_end", …)` as a query-completion signal under *Error detection* above), the prompt-mode driver MUST NOT use these events to resolve query completion; their sole prompt-mode role is cancellation forwarding, so cross-fire from an unrelated session's turn event is harmless — it triggers only a re-check of a non-aborted captured signal. The process-global / no-per-session-origin-marker property this paragraph depends on is re-audited against each candidate Pi minor per [bump-checklist item (ah)](./version-bump-step2.md#bump-checklist-pi-on-process-global).

> **loom 1.0 seam — mid-loom user-session replacement.** <a id="loom-1-0-seam-mid-loom-user-session-replacement"></a><a id="v1-seam-mid-loom-user-session-replacement"></a> The prompt-mode driver MUST read `pi` from a single factory-captured reference (the `pi: ExtensionAPI` parameter of `default function (pi: ExtensionAPI)` from the **Extension entry point** above) held for the lifetime of each loom invocation; consumers MUST NOT re-acquire `pi` from any other surface (e.g. by reaching into `ExtensionCommandContext` or by re-entering the factory) during a loom invocation. The deferred *Mid-loom user-session replacement* extension in [Future Considerations](../future-considerations/surface-extensions.md#mid-loom-user-session-replacement) introduces a `withSession`-fronted re-acquisition of the captured reference before the next `sendUserMessage` (so a loom-driven `ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` call no longer invalidates the captured `pi`) at exactly that single capture site; duplicating the capture across call sites in loom 1.0 is breaking, because the deferred re-acquisition hook would then have to be threaded into every duplicate.

**Provider compatibility for typed queries.** The forced respond turn requires the provider to honour named-tool forcing. loom 1.0 supports the providers for which pi-ai exposes a named-tool `toolChoice` mapping: `anthropic-messages`, `openai-completions`, `mistral`, and `amazon-bedrock`. At loom-load time, when a loom contains any typed-query expression and its resolved `model:` routes through a provider outside this set, the runtime emits `loom/load/typed-query-unsupported-provider` (warning) naming the model; <!-- the warning's payload is exactly the model + provider carried by its diagnostics/code-registry-load.md message template; the former "listing the typed-query expression positions" clause was dropped as no positions format was ever pinned --> the loom still loads, and any typed query at runtime returns `Err(QueryError { kind: "transport", message: "<provider> does not support forced tool-use; typed queries unavailable", http_status: null, provider: "<provider>", retryable: false })`. Here `<provider>` is the resolved [`Model<Api>.api`](./host-interfaces-core.md#model-registry-pin) value of the loom's resolved `model:` (the `api`-shaped form, e.g. `mistral`), so the synthesised `provider` field carries the same `api`-shaped value as a classifier-produced [`TransportError.provider`](../errors-and-results/queryerror-variants.md#queryerror-variants). Untyped queries against the same loom continue to work. Future widening (Gemini, OpenAI Responses, weak local models via JSON-mode fallback) is listed in [Future Considerations](../future-considerations.md).

<a id="complete-forced-tool-presupposition"></a>*`complete()` forced-tool behavioural presupposition.* The forced respond turn above and the binder's [structured-output call](./binder-inference.md#binder-inference-call) both depend on two behavioural properties of `@earendil-works/pi-ai`'s `complete()` free function for each loom 1.0-supported provider (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`): (1) calling with `options.toolChoice = { type: "tool", name }` forces the named tool — the per-provider adapter maps that option to the provider-native forced-tool payload rather than leaving the choice unconstrained — and (2) the call attaches no turn to a concurrently-driven user or subagent `AgentSession`, since it runs off-session directly off the returned `Promise<AssistantMessage>` (per [Binder inference call](./binder-inference.md#binder-inference-call)). Neither property is part of `@earendil-works/pi-ai`'s typed surface, so a pi-ai minor that silently regresses either one is caught by neither the SDK surface-inventory test nor the pi-ai version pin; the regression would surface only as the [diagnostic limitation](#pic-typed-query-noncompliance) above misclassifying the model as non-compliant. Detection therefore routes to editorial review: the per-provider forced-tool fixtures and the off-session-turn inspection SHOULD be re-run on each Pi minor bump, per item (u) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure). *Authoritative source.* Neither property is documented on `@earendil-works/pi-ai`'s typed surface; at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) `complete()` is observed at `dist/stream.js` in `@earendil-works/pi-ai` and its per-provider forced-tool mapping (`options.toolChoice` → provider-native payload) at that package's per-provider adapter modules under `dist/providers/` (e.g. `dist/providers/amazon-bedrock.js`, `dist/providers/anthropic.js`, `dist/providers/openai-completions.js`, `dist/providers/mistral.js`), with the per-provider behaviour captured in the `npm test` forced-tool fixtures re-run per item (u) (**known-fragile evidence**: these bundler-output paths are diffable artefacts at the pin rather than load-bearing claims about their stable shape across patches; a rename or re-bundle routes the audit's textual-diffability failure to the established version-bump recovery path).

<a id="complete-retry-and-cancellation-presupposition"></a>*`complete()` retry-timing and cancellation behavioural presupposition.* Beyond the forced-tool and off-session-turn properties pinned above, the binder's [structured-output call](./binder-inference.md#binder-inference-call) — and any other loom call routed through `@earendil-works/pi-ai`'s `complete()` free function — depends on two further behavioural properties of `complete()` for each loom 1.0-supported provider: (1) `complete()` drives inter-attempt timing internally from its `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs` options, including honouring any server-requested wait such as an HTTP `Retry-After`, so that loom re-issuing each budgeted retry immediately — per the binder's [per-invocation retry budget](../binder/determinism-cancellation-failure.md#per-invocation-retry-budget) — delegates per-attempt backoff to `complete()` rather than implementing its own delay; and (2) `complete()` honours `options.signal` for cancellation, aborting the in-flight provider call and rejecting the returned `Promise<AssistantMessage>` when the supplied `AbortSignal` (the binder call passes `loomAbort.signal` via the `options.signal` bullet of [Binder inference call](./binder-inference.md#binder-inference-call)) aborts, so loom's [cancellation](../cancellation.md) chain tears the off-session provider call down rather than leaving it running to completion. Neither property is part of `@earendil-works/pi-ai`'s typed surface, so a pi-ai minor that silently regresses either one is caught by neither the SDK surface-inventory test nor the pi-ai version pin: a retry-timing regression would change the binder's observed inter-attempt latency without changing any loom-visible type, and a `signal`-honouring regression would leave a cancelled binder call running until it resolves on its own. Detection therefore routes to editorial review: the per-provider retry-timing and cancellation fixtures SHOULD be re-run on each Pi minor bump, per items (aa) and (ab) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure). *Authoritative source.* Neither property is documented on `@earendil-works/pi-ai`'s typed surface; at the [loom 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin) `complete()`'s retry/backoff handling and its `options.signal` cancellation handling are observed at `dist/stream.js` in `@earendil-works/pi-ai`, with the per-provider behaviour captured in the `npm test` retry-timing and cancellation fixtures re-run per items (aa) and (ab) (**known-fragile evidence**: this bundler-output path is a diffable artefact at the pin rather than a load-bearing claim about its stable shape across patches; a rename or re-bundle routes the audit's textual-diffability failure to the established version-bump recovery path).

> **loom 1.0 seam — typed-query supported provider set.** <a id="loom-1-0-seam-typed-query-supported-provider-set"></a><a id="v1-seam-typed-query-supported-provider-set"></a> The loom 1.0 typed-query-supported provider set (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`) is referenced by the load-time `loom/load/typed-query-unsupported-provider` warning emitter and the runtime guard that synthesises the unsupported-provider `TransportError`. The deferred *Typed-query support for providers without named-tool forcing* extension in [Future Considerations](../future-considerations.md) widens this set (and adds a JSON-mode / system-prompt fallback for the new entries) without touching the typed-query behavioural contract above; widening the set is the only behavioural surface the extension perturbs. *Non-normative implementation note.* loom 1.0 expects the provider set to be exposed as a single named runtime constant referenced by every consumer (rather than inlining the four-element list at each site) so there is one source of truth to widen; this single-constant factoring is a maintainability convention, not an observable conformance point.

<a id="pic-typed-query-noncompliance"></a> **loom 1.0 diagnostic limitation.** When a typed query routes through a provider in the loom 1.0 supported set (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`) — most commonly an OpenAI-compatible local backend — but the specific model ignores forced-tool selection on the forced respond turn (by emitting plain text, or by invoking a tool whose name is not `__loom_respond_<slug>`), the runtime cannot distinguish this from any other case where the model fails to call the respond tool. The runtime classifies the case as a schema-validation failure and routes it through the respond-repair pipeline; it does NOT surface as `tool_loop_exhausted`, which is unreachable on the forced respond turn because that turn is the exempt-routed terminator dispatched by CIO-4's `max_rounds`-final branch (see [Hard Runtime Ceilings — CIO-4](../hard-ceilings/ceilings-3-and-4.md#cio-4)). For each forced respond turn that does not invoke `__loom_respond_<slug>`, the runtime synthesises a single `ValidationIssue` and feeds it through that pipeline. The non-compliance branch is observable in the synthesised issue's `message` literal; `path` and `schema_keyword` are uniform across both branches. The normative synthesised shapes — the plain-text and wrong-tool `ValidationIssue` literals, the terminal `Err(QueryError { kind: "validation", cause: "schema_validation", ... })` shape, the per-attempt accounting, and the `raw_response` null-vs-body disambiguation — are defined at [Errors and Results — Forced-respond non-compliance synthesised shapes (ERR-17)](../errors-and-results/queryerror-variants.md#err-17). Separating provider-level from model-level non-compliance is out of scope for loom 1.0.
