# Conversation drive

<a id="no-extra-mediation"></a>

**No additional access channels.** The runtime interposes no privilege layer between loom code and the Pi extension host. Loom-side filesystem, network, and Pi-API access are bounded only by Pi's extension-host grants and by the per-loom `tools:` allowlist whose enforcement is specified above. The runtime MUST NOT introduce additional access channels (sandbox, capability filter, mediated proxy) in any loom 1.x release; widening or narrowing this rule is a major-version concern. The `tools:` allowlist constrains the *model's* reachable callable set, not the host process: a loom that declares a high-privilege callable (e.g. `bash`) exposes the full underlying capability of that callable to its model. Host-side denials of filesystem, network, or Pi-API access reach loom code through the tool that issued the request: a thrown or `isError: true` return is mapped to `Err(QueryError { kind: "code_tool", cause: "execution", ... })` per [Tool Calls — Failures](../tool-calls.md) and [Tool execution from loom code](./host-interfaces-core.md#tool-execution-from-loom-code) below; silent success on denial is forbidden.

**Conversation drive — prompt mode.** The loom interpreter drives the user's session by:

- Issuing untyped queries via `pi.sendUserMessage(text)` when the agent is idle, or `pi.sendUserMessage(text, { deliverAs: "steer" })` when the agent is mid-stream. `pi.sendUserMessage` returns `void` — the `ExtensionAPI` surface declares both `sendUserMessage(...)` and `sendMessage(...)` as `void`-returning; the `Promise<void>` shape exists only on `AgentSession.sendUserMessage` (subagent mode, see **Conversation drive — subagent mode** below) and on `ReplacedSessionContext.sendUserMessage` (post-replacement contexts, out of scope here per the *captured `pi` reference* clause below). Only synchronous throws — e.g. argument-validation rejections inside Pi — are observable directly from the call. The runtime MUST NOT `await pi.sendUserMessage(...)` and MUST NOT attach a `.catch` handler — matching the rule pinned for `pi.sendMessage` under **System notes** below. The **primary** prompt-mode transport-failure surface is therefore the post-`waitForIdle()` `AgentSession.errorMessage` probe pinned under *Error detection* below, not the `pi.sendUserMessage` call site itself. As a narrow secondary mapping, a synchronous throw from `pi.sendUserMessage` (the only failure mode the call surface itself can signal) is mapped to `Err(QueryError { kind: "transport", message: <error.message>, http_status: null, provider: <resolved-model-provider>, retryable: false })` — not to `loom/runtime/internal-error`. The `provider` field is populated from the loom's resolved `model:` regardless of which surface (synchronous throw at the call site vs. the `errorMessage` probe) detected the failure, since the `errorMessage` string carries no structured provider field and the **Runtime event channel** always-log emission for `kind: "transport"` requires the field. The same dual-surface mapping applies when `pi.sendUserMessage` is the transport call inside a typed query's free phase or its forced respond turn. The `pi` reference here is the `ExtensionAPI` captured by the extension factory (the `pi` parameter of `default function (pi: ExtensionAPI)` from the **Extension entry point** above), which the runtime holds for the lifetime of each loom invocation; `sendUserMessage` is **not** a method on `ExtensionCommandContext`, so the per-handler `ctx` is consulted only for `ctx.isIdle()` to choose between the two forms (calling the no-options form mid-stream throws — see Pi's `examples/extensions/send-user-message.ts`). loom 1.0 prompt-mode looms always use `deliverAs: "steer"` when not idle (the loom is the active driver and wants its message delivered to *this* turn); `deliverAs: "followUp"` is reserved for future considerations and is not emitted by loom 1.0. The captured `pi` reference assumes the user session is **not** replaced mid-loom (`ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` invalidate it); loom 1.0 looms never trigger replacement (see [Future Considerations — Mid-loom user-session replacement](../future-considerations/surface-extensions.md#mid-loom-user-session-replacement) for the post-loom 1.0 re-acquisition seam). Completion is awaited by `await ctx.waitForIdle()` on the `ExtensionCommandContext` passed to the slash-command handler. `waitForIdle` is session-scoped — it observes only the user session bound to this command — and is the prompt-mode driver's authoritative completion signal. *Resolution.* `waitForIdle()` returns `Promise<void>` and resolves once Pi emits `agent_end` for the user session and every awaited `agent_end` listener settles; it never rejects, so the runtime MUST NOT wrap the `await` in a `try`/`catch` expecting rejection as a failure signal. *Error detection.* Because `pi.sendUserMessage` returns `void` (no rejection path) and `waitForIdle()` never rejects, every transport- or provider-level failure of the user turn that is not a synchronous throw at the call site lands on the user `AgentSession`'s error state. This post-`waitForIdle()` probe is the **primary** transport-failure surface for prompt mode; the synchronous-throw branch above is the narrow secondary. Immediately after `waitForIdle()` resolves the runtime probes that state (the `errorMessage` field on the `AgentSession`, named per the pinned Pi version from **Host prerequisites** above), and a non-empty value maps to `Err(QueryError { kind: "transport", message: <errorMessage>, http_status: null, provider: <resolved-model-provider>, retryable: false })` per [Query](../query.md). The probe MUST read `errorMessage` synchronously on `waitForIdle()` resolution, before the runtime issues any subsequent `pi.sendUserMessage` on the same context — Pi may clear or overwrite the field on the next turn. The runtime MUST NOT install a global `pi.on("agent_error", …)` listener for this purpose, by the same per-session cross-fire reasoning given for the `agent_end` ban below. *Hang handling.* `waitForIdle()` has no internal deadline; hangs are bounded only by the cancellation path. `loomAbort.abort()` invokes `ctx.abort()` on the captured slash-command handler context (the `ExtensionCommandContext.abort` member already in the forwarded set per **`ExtensionContext` (member surface loom touches)** below), which cancels the active user run and lets `waitForIdle()` resolve; the runtime guards the propagation with a one-shot flag so a re-entrant `loomAbort.abort()` (e.g. from an `agent_end` observer that itself observes the abort) does not double-cancel. The post-`waitForIdle` error-state probe runs unconditionally on resolution, but if `loomAbort.signal.aborted` is true the runtime synthesises `Err(QueryError { kind: "cancelled" })` per [Cancellation](../cancellation.md) instead of reading session error state — even when Pi tore down cleanly and `waitForIdle()` resolved without any error written to the session. The symmetric statement on the abort side lives at [Cancellation — Forwarding into `loomAbort`](../cancellation.md). The runtime MUST NOT subscribe to the global `pi.on("agent_end", …)` event for query completion: that event fires for every `AgentSession` in the process (the user shell session and any concurrent subagent-mode sessions) with no per-session origin marker, so a global handler cross-fires across concurrent looms or sibling subagents and resolves the wait on the wrong turn. <a id="untyped-query-ok-extraction"></a>The `Ok(string)` value of an untyped query is the accumulated assistant text of the final turn, extracted from the user session after `waitForIdle()` resolves and only once the cancellation short-circuit (`loomAbort.signal.aborted`, which synthesises `Err(QueryError { kind: "cancelled" })`) and the non-empty-`errorMessage` transport short-circuit pinned above have both been passed — the extraction runs downstream of those branches and never reorders or bypasses them. The runtime reads the source from the same `ReadonlySessionManager` `Pick<>` members the binder uses: `ctx.sessionManager.getEntries()` and `ctx.sessionManager.getLeafId()` (both enumerated in the `sessionManager` row of **`ExtensionContext` (member surface loom touches)** below), fed to `buildSessionContext(...)` exactly as [Session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) does, yielding the same chronological `Message` list. The *final turn* is the trailing turn of that list — the last `user` message (the loom-issued `pi.sendUserMessage` turn) plus every subsequent `assistant` / `toolResult` / `custom` message through the end of the list, using the turn boundary defined in that section; turns from earlier slash-command invocations on the long-lived user session precede that trailing `user` message and are excluded. The string is the `text` content of every `assistant` message in the final turn, concatenated in chronological order with a single `\n` separator between successive assistant messages; the provider-internal `thinking` array and all `toolCalls` entries are omitted, identical to the `[assistant]` body selection in [Compact-transcript format](../binder/binder-model-and-context.md#compact-transcript-format-normative) item 4. A final turn that produced no assistant text (a pure tool-use turn) yields the empty string. `pi-loom` performs no buffering, suppression, or re-styling of Pi's stream; user-visible behaviour is specified in [Slash-Command Invocation — User-visible streaming](../slash-invocation.md#slsh-2).
- Issuing typed queries that enforce the inferred or ascribed schema against the model's response. A typed query is a **two-phase tool-loop conversation**, exactly as specified in [Query — Typed queries are tool-loop-shaped](../query.md): (1) a *free phase* in which the model may call any frontmatter `tools:` entry the same way it does in an untyped query, with `tool_choice` left unconstrained; followed by (2) one *forced respond turn* once the model emits a plain text turn, in which the runtime issues a follow-up user message inlining the lowered response schema and forces the provider's tool choice to the synthesised one-shot respond tool. The synthesised tool's `execute` AJV-validates its input against the lowered schema and resolves the query's promise with the validated value. Only the forced respond turn uses provider tool-choice forcing; every preceding free-phase turn leaves the model free to think aloud and to use the frontmatter callable set, so the `frontmatter.md` "available to the model during query-time tool loops" guarantee, the `ModelToolError` surface, and the "same callable set serves both code and model" invariant all stay live for typed queries. The forced respond turn is the exempt-routed terminator dispatched by CIO-4's `max_rounds`-final branch (see [Hard Runtime Ceilings — CIO-4](../hard-ceilings/ceilings-3-and-4.md#ceiling-interaction-order) and [Query — Tool-call loop bound](../query.md)); the runtime issues it on every typed query reaching that branch — including `max_rounds: 0` where it is the only turn — and `tool_loop.max_rounds` is not debited by it. respond-repair follow-ups (see [Query — Schema-validation respond-repair](../query.md)) restart the whole two-phase loop with a fresh `tool_loop` budget, so the model can re-tool before answering a respond-repair turn. The runtime MUST surface the validated payload exactly once and MUST NOT add additional user-visible turns to the conversation beyond the natural free-phase turns and the single forced respond turn (the forced respond turn is rendered as a tool-call card by Pi's standard transcript renderer, not as a redundant assistant turn). The synthesised one-shot tool follows the same `name` / `label` / `description` / `parameters` / `execute` shape and the same `Type.Unsafe<unknown>(loweredJsonSchema)` `parameters` wrap defined in **Per-loom registration** above, and follows the registration cache + active-set snapshot/restore protocol defined in **Tool-registration lifetime and visibility** above, so the synthesised typed-query tool, the `.loom`-callable Pi tool, and the subagent-mode `customTools` entries all use a single bridge from loom's lowered JSON Schema to Pi's TypeBox-typed `ToolDefinition` and a single visibility-control mechanism. The mechanism used to force tool choice on the respond turn (`options.toolChoice` on the per-provider pi-ai adapter) is described in [Implementation Notes — Runtime](../implementation-notes.md#runtime).

> **loom 1.0 seam — mid-loom user-session replacement.** <a id="loom-1-0-seam-mid-loom-user-session-replacement"></a><a id="v1-seam-mid-loom-user-session-replacement"></a> The prompt-mode driver MUST read `pi` from a single factory-captured reference (the `pi: ExtensionAPI` parameter of `default function (pi: ExtensionAPI)` from the **Extension entry point** above) held for the lifetime of each loom invocation; consumers MUST NOT re-acquire `pi` from any other surface (e.g. by reaching into `ExtensionCommandContext` or by re-entering the factory) during a loom invocation. The deferred *Mid-loom user-session replacement* extension in [Future Considerations](../future-considerations/surface-extensions.md#mid-loom-user-session-replacement) introduces a `withSession`-fronted re-acquisition of the captured reference before the next `sendUserMessage` (so a loom-driven `ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` call no longer invalidates the captured `pi`) at exactly that single capture site; duplicating the capture across call sites in loom 1.0 is breaking, because the deferred re-acquisition hook would then have to be threaded into every duplicate.

**Provider compatibility for typed queries.** The forced respond turn requires the provider to honour named-tool forcing. loom 1.0 supports the providers for which pi-ai exposes a named-tool `toolChoice` mapping: `anthropic-messages`, `openai-completions`, `mistral`, and `amazon-bedrock`. At loom-load time, when a loom contains any typed-query expression and its resolved `model:` routes through a provider outside this set, the runtime emits `loom/load/typed-query-unsupported-provider` (warning) naming the model and listing the typed-query expression positions; the loom still loads, and any typed query at runtime returns `Err(QueryError { kind: "transport", message: "<provider> does not support forced tool-use; typed queries unavailable", http_status: null, provider: "<provider>", retryable: false })`. Untyped queries against the same loom continue to work. Future widening (Gemini, OpenAI Responses, weak local models via JSON-mode fallback) is listed in [Future Considerations](../future-considerations.md).

> **loom 1.0 seam — typed-query supported provider set.** <a id="loom-1-0-seam-typed-query-supported-provider-set"></a><a id="v1-seam-typed-query-supported-provider-set"></a> The loom 1.0 typed-query-supported provider set (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`) MUST be exposed as a single named runtime constant referenced by every consumer — the load-time `loom/load/typed-query-unsupported-provider` warning emitter, and the runtime guard that synthesises the unsupported-provider `TransportError` — so that there is one source of truth to widen. Inlining the four-element provider list at any consumer is a breaking simplification: the deferred *Typed-query support for providers without named-tool forcing* extension in [Future Considerations](../future-considerations.md) widens this constant (and adds a JSON-mode / system-prompt fallback for the new entries) without touching the typed-query behavioural contract above; widening the constant is the only behavioural surface the extension perturbs.

<a id="pic-typed-query-noncompliance"></a> **loom 1.0 diagnostic limitation.** When a typed query routes through a provider in the loom 1.0 supported set (`anthropic-messages`, `openai-completions`, `mistral`, `amazon-bedrock`) — most commonly an OpenAI-compatible local backend — but the specific model ignores forced-tool selection on the forced respond turn (by emitting plain text, or by invoking a tool whose name is not `__loom_respond_<slug>`), the runtime cannot distinguish this from any other case where the model fails to call the respond tool. The runtime classifies the case as a schema-validation failure and routes it through the respond-repair pipeline below; it does NOT surface as `tool_loop_exhausted`, which is unreachable on the forced respond turn because that turn is the exempt-routed terminator dispatched by CIO-4's `max_rounds`-final branch (see [Hard Runtime Ceilings — CIO-4](../hard-ceilings/ceilings-3-and-4.md#ceiling-interaction-order)). For each forced respond turn that does not invoke `__loom_respond_<slug>`, the runtime synthesises a single `ValidationIssue` (per [Errors and Results — `ValidationError`](../errors-and-results/queryerror-variants.md#queryerror-variants)). The non-compliance branch is observable in the synthesised issue's `message` literal; `path` and `schema_keyword` are uniform across both branches.

*Plain-text branch* — the forced respond turn carried plain text and no `tool_use` block:

```
{ path: "",
  schema_keyword: "required",
  message: "model returned plain text instead of calling the forced respond tool" }
```

*Wrong-tool branch* — the forced respond turn carried only `tool_use` blocks whose `name` is not `__loom_respond_<slug>` (independent of whether plain text accompanies them):

```
{ path: "",
  schema_keyword: "required",
  message: "model invoked tool '<provider-emitted-tool-name>' instead of the forced respond tool '__loom_respond_<slug>'" }
```

The `<provider-emitted-tool-name>` placeholder is the `name` field of the provider's `tool_use` block, verbatim; when multiple wrong-tool calls appear in the same turn, the first block's `name` is used (consistent with the "single `ValidationIssue`" rule above).

The synthesised issue feeds the same respond-repair pipeline as an AJV failure: it consumes one `respond_repair.attempts` slot, and the follow-up template is the methodology-appropriate one (the `validator_error` template's `<ajv-summary>` placeholder is rendered from this synthesised issue exactly as if AJV had produced it). Terminal exhaustion returns:

```
Err(QueryError {
  kind: "validation",
  cause: "schema_validation",
  attempts: respond_repair.attempts,                  // the configured budget
  validation_errors: [<synthesised issue from final attempt>],
  raw_response: <plain-text body of the final forced respond turn, or null when that turn carried no plain-text body>,
  message: "model did not call the forced respond tool"
})
```

The `raw_response: null` case is the wrong-tool branch when the model emitted a `tool_use` block with no accompanying plain text; this matches the `ToolLoopExhaustedError` precedent under [Errors and Results — `raw_response`](../errors-and-results/queryerror-variants.md#queryerror-variants) ("`null` when exhaustion fired on a pure tool-use turn"). When the wrong-tool turn does carry plain text alongside the `tool_use` block, `raw_response` is that plain-text body.

With `respond_repair.methodology: none`, the *first* forced respond turn that does not invoke `__loom_respond_<slug>` returns the same `Err` shape with `attempts: 0` (no follow-ups issued, consistent with the existing `none` contract). When a stub provider alternates plain-text and wrong-tool payloads across respond-repair attempts, only the *last attempt's* issue appears in `validation_errors` (consistent with the existing "final malformed assistant text" wording in `errors-and-results.md`); `raw_response` is the plain-text body of the *last* forced respond turn, or `null` when that turn carried no plain-text body — not a concatenation across attempts. The synthesised `ValidationIssue.path` is `""` (the JSON Pointer to the document root) on both branches, matching AJV's empty-path convention for root-level failures. Separating provider-level from model-level non-compliance is out of scope for loom 1.0.
