# Binder inference

<a id="binder-inference-call"></a>

**Binder inference call.** The binder pass described in [Slash-Command Argument Binding](../binder.md) issues its one-shot structured-output completion through `@earendil-works/pi-ai`'s `complete` free function — `complete<TApi extends Api>(model, context, options?): Promise<AssistantMessage>`, declared at `dist/stream.d.ts` in `@earendil-works/pi-ai` (the sibling package re-exported through the [model-registry surface](./host-interfaces-core.md#model-registry-pin) above, resolved at the [theta 1.0 Pi-SDK pin](./host-prerequisites.md#pi-sdk-pin)). The `Context`, `Tool`, `StreamOptions` (the base of `ProviderStreamOptions = StreamOptions & Record<string, unknown>`), `AssistantMessage`, and `ToolCall` types the call consumes and returns are declared at `dist/types.d.ts` in the same package. The symbol, its signature, and the structured-output surface are theta-load-bearing and MUST be re-validated against those files on each Pi minor bump per [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; this paragraph pins theta's *use* of that surface, not the surface itself, which `@earendil-works/pi-ai` owns.

Unlike the prompt-mode and subagent-mode drivers above, the binder call uses neither the user `AgentSession` nor a spawned subagent child process: it runs off-session in the parent via pi-ai's `complete()` free function, attaches no turns, and resolves directly from the `Promise<AssistantMessage>` the entry point returns. The binder has two distinct events that occur at different phases: binder-*model* resolution runs at theta-load time (per [Binder model](../binder/binder-model-and-context.md#binder-model)), whereas each `complete(...)` call is issued per slash dispatch — at the [dispatch-site setup wrap](./active-invocation-registry.md#active-invocation-registry) that creates `thetaAbort`, where the user-supplied slash-argument string is live and the [per-invocation retry budget](../binder/determinism-cancellation-failure.md#per-invocation-retry-budget) applies. Its terminal signal is therefore the resolution (or rejection) of that promise, not an `agent_end` event — the `pi.on("agent_end", …)` prompt-mode channel and the subagent callee's in-child `agent_end` handling (which under the RFC 0006 child-process theta design happens inside the spawned child, not on any parent-consumed wire) play no part in the binder call, and the global-`agent_end` bans those drivers impose do not apply to it.

The runtime populates one `complete(model, context, options)` call per binder attempt as follows:

- `model` is the resolved binder `Model<Api>` handle (per [binder-model parse rule](../binder/binder-model-and-context.md#binder-model-parse-rule)).
- `context.systemPrompt` is the rendered binder system prompt (per [System-prompt structure](../binder/binder-bypass-and-envelope.md#system-prompt-structure-normative)).
- `context.messages` is a fixed single-element array carrying one `user`-role message whose `content` is the canonical literal string `Bind the slash-command arguments now.`. The variable binding context (theta identity, the parameter block, the user-argument line, and the optional session-context block) is carried by `context.systemPrompt` per [System-prompt structure](../binder/binder-bypass-and-envelope.md#system-prompt-structure-normative), so the message neither restates that context nor varies per invocation; it exists only to satisfy providers that reject a system-only request with no non-system message. The literal is a fixed constant of the binder call, recorded as such in [Determinism](../binder/determinism-cancellation-failure.md#determinism).
- `context.tools` carries exactly one entry — the **binder's structured-output tool** — whose `parameters` is the [binder envelope schema](../binder/binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params) wrapped as `Type.Unsafe<unknown>(envelopeSchema)` on the same lowered-JSON-Schema → TypeBox bridge the **Per-theta registration** section above uses. Its three remaining required `ToolDefinition` fields are fixed literals, mirroring the synthesised typed-query respond tool's pin in the **Per-theta registration** field enumeration above (where the respond tool's `label` `"Theta typed-query response"` is pinned): `name` is `__theta_bind_<slug>`, where `<slug>` is the schema slug of the lowered binder envelope schema produced by the [canonical schema hash](../schema-subset.md#canonical-schema-hash) recipe — the same recipe the typed-query `__theta_respond_<slug>` tool name uses; `label` is the literal string `"Theta binder envelope"`; and `description` is the fixed literal `"Return the binder result envelope for the slash-command argument binding."`. The provider's tool choice is forced to that single tool through the same per-provider `options.toolChoice` mechanism the typed-query forced respond turn uses (see [Implementation Notes — Runtime](../implementation-notes.md#runtime); the per-provider behavioural presuppositions this mechanism relies on are pinned at [Provider compatibility — `complete()` forced-tool presupposition](./conversation-drive.md#complete-forced-tool-presupposition)); theta 1.0 attaches the envelope as a single forced tool, not as a provider-native structured-output / JSON-schema response field.
- `options.temperature` is `0` (per [Determinism](../binder/determinism-cancellation-failure.md#determinism)).
- `options.signal` is `thetaAbort.signal` (always defined; see **Cancellation source** below). This is the cancellation wiring [Slash-Command Argument Binding — Cancellation](../binder/determinism-cancellation-failure.md#cancellation) forwards into the call. That `complete()` actually aborts the in-flight provider call when this signal fires is the host presupposition pinned upstream at [Conversation drive — `complete()` retry-timing and cancellation behavioural presupposition](./conversation-drive.md#complete-retry-and-cancellation-presupposition) (property 2), re-validated on each Pi minor bump through version-bump checklist item [(ab)](./version-bump-step2.md#bump-checklist-complete-signal-honouring).
- the fixed seed, when the resolved provider's `Api` carries a seed field, is placed under that field name as a `Record<string, unknown>` key on `options` per [Provider seed-field mapping](./provider-error-mapping.md#provider-seed-field-mapping); providers whose row omits the seed field receive no seed key.
- `options.onResponse` is the runtime's provider-response capture callback: the runtime registers it on every binder `complete()` call so the `ProviderResponse { status, headers }` `@earendil-works/pi-ai` delivers (after an HTTP response is received and before its body stream is consumed) can be joined with the resolved `AssistantMessage`, keyed to this call's invocation, to supply the HTTP-status input the [Provider error mapping](./provider-error-mapping.md#provider-error-mapping) classifier reads per [classifier input surface](./provider-error-mapping.md#classifier-input-surface); which captured `ProviderResponse` the classifier joins when `onResponse` fires more than once before `complete()` resolves is pinned there. `StreamOptions.onResponse` and `ProviderResponse` are declared at `dist/types.d.ts` in `@earendil-works/pi-ai` alongside the `StreamOptions` and `AssistantMessage` types this call already consumes; theta redefines neither.

The structured-output result is extracted from the returned `AssistantMessage.content`: the first `ToolCall` whose `name` matches the binder's structured-output tool supplies the envelope JSON in its `arguments`. Success extraction takes precedence over `stopReason`-based failure classification on the same `AssistantMessage`: when a matching `ToolCall` is present, theta extracts its `arguments` as the envelope and hands off to the post-extraction envelope/AJV pipeline regardless of `stopReason`, `errorMessage`, or HTTP status — so the forced-tool success path (`stopReason: "tool_use"`) resolves as success, and a degenerate response carrying both a matching `ToolCall` and a non-empty `errorMessage` also extracts as success. A matching `ToolCall` whose `arguments` then fail parse or schema validation is the malformed-envelope class pinned by [Failure-class taxonomy](../binder/determinism-cancellation-failure.md#failure-class-taxonomy), not a transport failure. The failure routing applies only when no matching `ToolCall` is present: an `AssistantMessage` carrying no such `ToolCall` — plain text only, or a `ToolCall` with a different `name` — is likewise the malformed-envelope condition pinned by [Failure-class taxonomy](../binder/determinism-cancellation-failure.md#failure-class-taxonomy), while a `stopReason` other than `"stop"` or `"tool_use"`, a non-empty `errorMessage`, a rejected promise, or any 4xx/5xx/network failure is classified through the [Provider error mapping](./provider-error-mapping.md#provider-error-mapping) table exactly as the typed and untyped query paths classify provider failures. How many times this entry point is invoked per slash invocation is governed by the binder's [per-invocation retry budget](../binder/determinism-cancellation-failure.md#per-invocation-retry-budget), which also pins inter-attempt timing: theta re-issues each budgeted retry immediately, delegating per-attempt backoff and any server-requested wait (e.g. an HTTP `Retry-After`) to this call's `StreamOptions.maxRetries` / `StreamOptions.maxRetryDelayMs`.
