# Host prerequisites

**Host prerequisites.** Four host-side prerequisites are load-bearing for the contract below:

1. <a id="pi-sdk-pin"></a>*Pi SDK pin.* The supported Pi minor is pinned at `@earendil-works/pi-coding-agent ~0.75.5`. This anchor (`#pi-sdk-pin`) is the **single source of truth** for that literal range in the spec corpus: every other reference to the supported Pi minor in the spec corpus MUST cite this anchor rather than restating the literal range, so each Pi minor bump is a one-character edit at this site (mirrored mechanically by the manifest lock-step sub-paragraph below) rather than an N-site sweep. The spec orientation block (see [`spec.md` — Orientation — Prerequisites](../../spec/overview-and-orientation.md#orientation)) references this pin by location rather than restating the literal value, as do every other Pi-version-mentioning paragraph elsewhere in PIC and on the sibling topic pages ([`binder.md`](../binder/binder-model-and-context.md#strict-capability-requirement), [`diagnostics.md`](../diagnostics.md), [`future-considerations.md`](../future-considerations/model-changes-and-non-goals.md#v1-non-goals)). The lock-step minor-version line is inherited from `@earendil-works/pi-coding-agent`'s own `dependencies` block, which pins `@earendil-works/pi-agent-core`, `@earendil-works/pi-ai`, and `@earendil-works/pi-tui` at the same `~X.Y.Z` minor-version line. All four packages are released together from the `pi-mono` monorepo; no skew across the four is supported, and loom does not attempt to detect or accommodate skew. A Pi minor bump moves all four together; before the loom `peerDependencies` range is widened, the contributor MUST follow [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below — the build-time `peerDependencies` literal-read assertion and the build-time SDK surface-inventory assertion are the mechanical gates that fail the build until the procedure has been completed. Loom does not at runtime read `pi-coding-agent`'s `package.json` to verify the upstream pin — doing so would re-introduce the very skew-tolerance disclaimed above; the build-time literal-read assertion is the gate that keeps the four `peerDependencies` minors aligned. Loom's `peerDependencies` block declares all four packages as belt-and-braces; the redundancy is intentional and surfaces a clear install-time error under package managers (pnpm/yarn) that do not auto-deduplicate transitive peer-dep ranges.

   <a id="pi-sdk-pin-manifest-lock-step"></a>*Manifest lock-step (one legitimate restatement site).* The four `@earendil-works/*` entries in `package.json#peerDependencies` (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`) MUST literally equal the canonical `~0.75.5` range stated above, because npm consumes the manifest mechanically and the manifest cannot anchor-cite. The manifest is therefore the one legitimate non-spec restatement of the pin literal, and is lock-stepped to this anchor on the same editorial-convention basis as [GOV-30](../governance/req-id-prefix-table-active-b.md#gov-30)'s aggregator lock-step convention (a PR that bumps the literal at this anchor co-edits the four `package.json` entries in the same commit; the build-time `peerDependencies` literal-read assertion gated by [Pi version bump procedure — step 2](./version-bump-intro.md#pi-version-bump-procedure) below is the mechanical witness on the manifest side, and a partial landing where the spec literal and the manifest literal disagree fails that assertion). The `typebox` peer-dep entry is **not** part of this lock-step — it is pinned to `"*"` per the *typebox* sub-paragraph below.

   *Deliberate deviation from Pi's bundled-package convention.* Pi's `@earendil-works/pi-coding-agent` `docs/packages.md` — *Dependencies* prescribes `"*"` as the `peerDependencies` range for all five bundled packages it names (`@earendil-works/pi-ai`, `@earendil-works/pi-agent-core`, `@earendil-works/pi-coding-agent`, `@earendil-works/pi-tui`, `typebox`). Loom deliberately deviates from that convention for the four `@earendil-works/*` entries by pinning them to the [loom 1.0 Pi-SDK pin range](#pi-sdk-pin) rather than `"*"`. The deviation gives up Pi-prescribed alignment (a future reader auditing loom's packaging against `packages.md` will find four entries that do not match the convention) and buys install-time skew detection on the package managers most production hosts use: `pnpm` (in its default `node-linker=isolated` mode) and `yarn` do not auto-deduplicate transitive peer-dep ranges, so a tilde-range pin in loom's `peerDependencies` causes the install to fail loudly when the host has resolved any of the four `@earendil-works/*` packages outside the pinned minor line, whereas a `"*"` declaration would silently accept the skew and defer the failure to a runtime `TypeError` at the first SDK call site. The four-anchor invariant is what the build-time `peerDependencies` literal-read test, the [Step 0 (d) Peer-dep version](./capability-probe.md#entry-capability-probe) probe, and the literal-anchor list in [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below are built around. The fifth package, `typebox`, follows the convention (`"typebox": "*"`) — see the sub-paragraph immediately below — because the runtime's only `typebox` dependency is `Type.Unsafe`, whose call shape is stable across the TypeBox `0.x` → `1.x` line and therefore does not need install-time minor-skew detection. The `typebox` `"*"` declaration MUST NOT be collapsed into the four-entry tilde-pinned group: the two ranges are deliberately distinct and reflect different stability arguments (lock-step minor for the `@earendil-works/*` four; convention-following floor for `typebox`).

   *`typebox` (the fifth Pi-bundled package).* The four `@earendil-works/*` packages above are pinned at the [loom 1.0 Pi-SDK pin range](#pi-sdk-pin) per the lock-step rule. `typebox` is the fifth Pi-bundled package the loom runtime imports (only `Type.Unsafe`, per [*Target surface categories*](./audit-recognised-shapes.md#audit-target-surface-categories) below (typebox carve-out)); per Pi's bundled-package convention (`@earendil-works/pi-coding-agent` `docs/packages.md` — *Dependencies*) it MUST be declared as `"typebox": "*"` so the host's bundled version wins. <a id="typebox-single-instance-precondition"></a>That bundled-version-wins outcome rests on two host-side preconditions the `"*"` declaration cannot itself guarantee: that the host bundles `typebox` at all, and that the package manager dedupes the `"*"` range to the host's single bundled instance. The `"*"` range carries no version-pinning guarantee of its own — it expresses only "accept whatever the host resolves" — so neither precondition is enforced at install time. No runtime probe verifies `typebox` provenance or single-instance resolution beyond the `Type.Unsafe`-callability check owned by **Step 0 (e)** below: a deduping failure that resolves a second, separately-resolved `typebox` instance alongside the host's passes that callability check and is treated with the same undefined-behaviour posture as the **Silent shape drift** carve-out at [Post-probe SDK-shape drift](./capability-probe.md#post-probe-sdk-shape-drift) — the runtime is not required to detect it, and no diagnostic is owed for it. In that case the mis-resolution surfaces downstream of `pi.registerTool` as a wrong-schema-shape failure rather than as a clean `loom/load/host-incompatible` refusal at factory entry. The `"*"` literal MUST NOT be folded into the four-entry tilde-pinned group asserted by the build-time lock-step assertion; it is asserted by its own one-line build-time literal-read assertion. The runtime's single load-bearing dependency on the bundled `typebox` namespace — the presence of `Type.Unsafe` as a callable function — is checked at extension-factory entry by **Step 0 (e)** below, which routes a missing or renamed member to `loom/load/host-incompatible` with `details.kind = "typebox-shape"` rather than to a runtime-time `TypeError` at the first tool-registration call (factory time for prompt mode, spawn time for subagent mode). This mirrors the framing already used for `AgentSession.prototype.abort` in **Step 0 (c)**: both are single-symbol Pi-bundled imports whose absence would otherwise crash the runtime at a registration call site rather than refuse cleanly at factory entry.
2. *Binder model.* Non-bypass looms require a structured-output-capable LLM resolved via `ctx.modelRegistry` for the binder pass described in [Slash-Command Argument Binding](../binder.md). When the registry cannot resolve a binder model, the loom fails to load with `loom/load/binder-model-unresolved`. Under the [loom 1.0 Pi-SDK pin](#pi-sdk-pin) the strict-capability probe is degraded and surfaces as `loom/load/binder-model-strict-capability-unknown` (W). Bypass cases (no-params, single-string with no default) skip the binder call entirely. loom 1.0 has no global binder opt-out; bypass is structural and per-loom.
3. *Binder credentials.* The binder reaches its model through the same `ctx.modelRegistry` / `Model<Api>` path (pinned at [model-registry surface](./host-interfaces-core.md#model-registry-pin) below) Pi uses for its own queries; credentials are not stored, fetched, or cached by loom. Loom relies on whatever credential resolution Pi already performs for the resolved model.
4. *Pi-supplied `AbortSignal`.* Pi delivers an `AbortSignal` to every extension entry point loom uses — `ctx.signal` on the slash-command `ExtensionCommandContext` and the `signal` parameter of `tool.execute(...)`. The Web-standard `AbortSignal` / `AbortController` shape (the Node-bundled WHATWG implementation) is a fixed-shape SDK precondition; the canonical member-with-kind enumeration the runtime depends on is pinned by [Step 0 (b)](./capability-probe.md#entry-capability-probe) below. Subagent mode does not consume a `createAgentSession({ signal })` option — `CreateAgentSessionOptions` has no `signal` field in the loom 1.0 Pi SDK pin, so passing one would be silently ignored — and instead consumes the `AgentSession.abort()` capability on the handle returned by `createAgentSession(...)`, invoked from a one-shot `loomAbort.signal` listener registered at spawn time; see **Subagent session lifecycle** and **Cancellation source** below. The loom runtime's own `AbortController` wrapping rule is in **Cancellation source** below; this prerequisite is what guarantees the inputs that rule consumes.

**Loom-package implementation dependencies.** Loom 1.0's own npm-package dependencies (e.g. `semver` behind the Step 0 capability-probe recipes, `chokidar` behind the `FileWatcher` seam) are implementation-side choices rather than part of the Pi-supplied contract this page owns; they are documented in [Implementation Notes — Runtime](../implementation-notes.md#loom-package-implementation-dependencies-loom-1-0).

<a id="session-binding-contract"></a>
<a id="pic-7"></a> **PIC-7. Session-binding contract.** A Pi extension instance is bound to exactly one active user session at any given time. The closed set of reason values Pi delivers to the runtime's teardown handler on session-end matches the `SessionShutdownEvent['reason']` union declared at `dist/core/extensions/types.d.ts` in `@earendil-works/pi-coding-agent` (the [loom 1.0 Pi-SDK pin](#pi-sdk-pin) from item 1 above); the closed reason set itself is owned by sub-step 4's [Unknown-reason rule](./unknown-reason-rule.md#unknown-reason-rule) below and is pinned at build time by the bidirectional `SessionShutdownEvent['reason']` type-equality assertion described at [Pi version bump procedure — step 5](./version-bump-intro.md#pi-version-bump-procedure) below. The premise that a session-only swap keeps the same extension instance loaded against the swapped-in session is pinned as [Host prerequisites for the degraded-state branch — presupposition (a)](#degraded-state-host-prerequisites) below.

<a id="degraded-state-host-prerequisites"></a>
**Host prerequisites for the degraded-state branch.** The [`reason: "new" | "resume" | "fork"` edge-case bullet](./session-only-degraded-state.md#session-only-reason-degraded-state) under **Extension entry point** sub-step 4 below depends on three Pi SDK behaviours that neither the loom 1.0 Pi-SDK pin's `dist/core/extensions/types.d.ts` nor the `ExtensionAPI` / `ExtensionContext` surfaces enumerated above pin as a typed contract. The loom 1.0 runtime treats each as a **presupposition** in the same sense as [Future Considerations — No concurrent user sessions in the same host process](../future-considerations/model-changes-and-non-goals.md#v1-non-goals) (which the same paragraph cites as the framing precedent for an unpinned host-behaviour assumption); detection of a Pi minor that violates any of them is by editorial review under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below — the build-time SDK surface-inventory assertion is a presence check on the seven named SDK capabilities and does not enumerate behavioural invariants of this kind, so contributor-side editorial review against this enumerated list is the gate.

  (a) *Extension-instance survival across session-only `session_shutdown`.* When `event.reason ∈ {"new", "resume", "fork"}`, Pi keeps the same extension instance loaded against the new user session after the `session_shutdown` handler returns, and continues to route slash invocations on the swapped-in session to the same instance (so the closed-over `LoomRegistry` handle the slash-command `handler` registered in step 3 below consults — including its drained flag and `drainStateTag` — is the very registry the prior handler invocation drained and tag-transitioned). This is what makes the post-handler `"degraded-needs-reload"` slash note pinned by the bullet observable to the operator on the swapped-in session. A Pi minor that instantiates a fresh extension on `"new"` (or that re-routes post-swap slash invocations to a fresh closed-over registry) would silently invalidate the bullet's recovery-by-`/reload` framing — the fresh instance/registry would start with a non-drained `LoomRegistry` and dispatch normally, hiding the degradation rather than surfacing it. No SDK surface pins this behaviour.

  (b) *Multi-`session_shutdown` delivery permitted to a single instance.* Pi may deliver more than one `session_shutdown` event to the same extension instance — both re-entrant deliveries during an in-flight teardown and fresh follow-up deliveries after a prior session-only handler has returned. The handler's [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause and the at-most-once-per-extension-instance-lifetime bound on `loom/host/session-shutdown-runtime-degraded` are written against this multi-delivery shape; a Pi minor that fires only one `session_shutdown` per instance leaves the **idempotent** clause vacuous but does not break the contract (the at-most-once bound holds trivially under at-most-one delivery). No SDK surface pins multi-delivery either way.

  (c) *`/reload` and `ctx.reload()` recovery.* Operator-issued `/reload` invokes Pi's `ctx.reload()` against the live (degraded) extension instance, and `ctx.reload()` tears down the extension runtime and re-invokes the registered extension factory, yielding a fresh runtime with a non-drained `LoomRegistry`. loom 1.0 has no fallback recovery path, so a Pi minor that changes either the slash dispatch routing of `/reload` or the runtime-invalidation/re-invocation semantics of `ctx.reload()` MUST be caught by editorial review under the bump procedure. The `reason: "new" | "resume" | "fork"` edge-case bullet's *Operator-issued `/reload`* sentence below and acceptance criterion (g) (the post-`/reload` fresh-runtime fixture) both presuppose this behaviour, and the at-most-once-per-extension-instance-lifetime bound on `loom/host/session-shutdown-runtime-degraded` is calibrated against the assumption that a fresh runtime starts a new lifetime; a Pi minor that no-ops `ctx.reload()` on an already-degraded extension, or routes `/reload` through a different facet, silently breaks recovery without tripping any other gate. No SDK surface pins this behaviour. Two corollaries follow from the *Operator-issued `/reload`* sentence above and govern how loom and Pi cooperatively keep that recovery path reachable on the degraded branch: **(c-i)** `/reload` is a Pi-host-owned slash name that Pi's slash dispatcher routes to `ctx.reload()` independently of any loom-registered `pi.registerCommand` handler, so the [drain-state-gated dispatch](./registration-steps.md#slash-handler-registration) short-circuit pinned at sub-step 3 below — which scopes only to loom-registered handlers (the `handler` value passed to every `pi.registerCommand(name, { description, handler })` call sub-step 3 makes) and routes through the `LoomRegistry.readDrainState()` three-arm enumeration — does **not** intercept `/reload` on the degraded `"degraded-needs-reload"` arm; the operator's `/reload` therefore reaches `ctx.reload()` even when every loom-registered slash handler in the same extension instance is short-circuiting on arm (c) of that enumeration. A Pi minor that demoted `/reload` to a `pi.registerCommand`-shaped handler (so loom's drain-state short-circuit would intercept it) is the same editorial-review failure mode the *slash dispatch routing of `/reload`* clause above already names. **(c-ii)** Loom MUST NOT register a `.loom` whose effective slash name (per [Discovery — Filename validity](../discovery.md), after `pi.looms` mapping, `as` rename, and basename hyphen-normalisation) is `reload`. A pending loom whose derived slash name resolves to `reload` is dropped at sub-step 3 below's session-start cross-format collision check on the same footing as any other collision against the [loom 1.0 cross-format collision source set](./host-interfaces-services.md#v1-seam-pi-owned-subagents-collision-source-set) — assuming Pi's `/reload` surfaces in `pi.getCommands()` under a `SlashCommandSource` arm in that set, which is itself part of this prerequisite (c) and a Pi minor that hides `/reload` from `pi.getCommands()` MUST be caught by editorial review under the bump procedure on the same footing as the other clauses of (c) above — and emits the same `loom/load/cross-format-collision` diagnostic. The two corollaries together guarantee that on the degraded branch the operator's `/reload` reaches `ctx.reload()` without being shadowed by either the drain-state-gated short-circuit (covered by (c-i)) or a colliding loom registration (covered by (c-ii)).


<a id="model-registry-population-presupposition"></a>
**Host prerequisite for binder-model / `model:` load-time resolution.** The per-loom load pass resolves `bind_model:`, `looms.binderModel`, and `model:` references one-shot against the model set `ctx.modelRegistry.getAvailable()` returns at that moment (the resolution path is pinned at [Slash-Command Argument Binding — binder-model parse rule](../binder/binder-model-and-context.md#binder-model-parse-rule), run against the [model-registry surface](./host-interfaces-core.md#model-registry-pin) above). loom 1.0 treats it as a **presupposition** — in the same sense as the degraded-state prerequisites above and the *No concurrent user sessions in the same host process* framing in [Future Considerations](../future-considerations/model-changes-and-non-goals.md#v1-non-goals) — that the set `getAvailable()` returns to a `session_start`-time load pass is the fully-populated model set the host will expose for the session, not a partial or still-warming view. Resolution is one-shot with no retry on a miss, so a model the host registers after the load pass has run leaves any loom whose `bind_model:` / `looms.binderModel` / `model:` referenced it load-failed (`loom/load/binder-model-unresolved` or `loom/load/model-unresolved`) until the operator runs `/reload` — the same hot-reload recovery the [Binder model](../binder/binder-model-and-context.md#binder-model) section describes for a settings change. No SDK surface pins the registry-population sequence; the contributor SHOULD re-confirm it by editorial review under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below (checklist item (k)), since a Pi minor that registers `ModelRegistry` synchronously but populates providers asynchronously would surface an empty or partial set to a load-time `getAvailable()` call.

**Extension entry point.** A single Pi extension module (`extensions/index.ts`, the file Pi auto-discovers from `package.json#pi.extensions = ["./extensions"]`) exporting the standard `default function (pi: ExtensionAPI)` factory.
