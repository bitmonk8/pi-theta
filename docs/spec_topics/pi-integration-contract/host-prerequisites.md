# Host prerequisites

**Host prerequisites.** Four host-side prerequisites are load-bearing for the contract below:

1. <a id="pi-sdk-pin"></a>*Pi SDK pin.* The supported Pi SDK is pinned by TWO literals declared here: the `peerDependencies` **floor** `@earendil-works/pi-coding-agent >=0.80.8` (open — no upper bound) and the `devDependencies` / runtime-probe **build pin** `@earendil-works/pi-coding-agent ~0.80.10` (the version the package is built and tested against). This anchor (`#pi-sdk-pin`) is the **single source of truth** for both literals in the spec corpus: every other reference to the supported Pi SDK in the spec corpus MUST cite this anchor rather than restating either literal, so a Pi bump is an edit at this site (mirrored mechanically by the manifest lock-step sub-paragraph below) rather than an N-site sweep. A Pi minor bump moves the **floor** — it raises the minimum Pi-SDK version whose API shape the runtime requires — and, when the build/test target advances, the build pin; the two literals move independently and are not required to share a minor line. The spec orientation block (see [`spec.md` — Orientation — Prerequisites](../overview-and-orientation.md#orientation)) references this pin by location rather than restating the literal values, as do every other Pi-version-mentioning paragraph elsewhere in PIC and on the sibling topic pages ([`binder.md`](../binder/binder-model-and-context.md#strict-capability-requirement), [`diagnostics.md`](../diagnostics.md), [`future-considerations.md`](../future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals)). Both the floor and the build pin cover the same four packages — `@earendil-works/pi-coding-agent`, `@earendil-works/pi-agent-core`, `@earendil-works/pi-ai`, and `@earendil-works/pi-tui` — which are released together from the `pi-mono` monorepo (the four-package co-move and widen-on-bump obligations are stated as [PIC-33](#pic-33) and [PIC-34](#pic-34) below). Theta does not at runtime read `pi-coding-agent`'s `package.json` to verify the upstream pin; the build-time literal-read assertion is the gate that keeps the four `peerDependencies` entries aligned to the floor and the four `devDependencies` entries aligned to the build pin. Theta's `peerDependencies` block declares all four packages as belt-and-braces; the redundancy is intentional and surfaces a clear install-time error under package managers (pnpm/yarn) that do not auto-deduplicate transitive peer-dep ranges when the host resolves any of the four below the floor.

   <a id="pic-33"></a> **PIC-33. Four-package lock-step co-move.** The four `@earendil-works/*` `peerDependencies` entries (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`) MUST co-move together to the **same floor literal** on a bump — they stay lock-stepped to each other, but the floor is open (`>=`), not a single minor line, so a newer host minor is not rejected. The four `devDependencies` entries for the same packages likewise MUST co-move together to the **same build-pin literal**.

   <a id="pic-34"></a> **PIC-34. Widen-on-bump procedure.** Before the theta `peerDependencies` **floor** is moved on a Pi bump — widen-on-bump here means raising the minimum, i.e. moving the floor to the new required-API version — the contributor MUST follow [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below — the build-time `peerDependencies` literal-read assertion and the build-time SDK surface-inventory assertion are the mechanical gates that fail the build until the procedure has been completed.

   <a id="pi-sdk-pin-manifest-lock-step"></a>*Manifest lock-step (one legitimate restatement site).* The four `@earendil-works/*` entries in `package.json#peerDependencies` (`pi-coding-agent`, `pi-agent-core`, `pi-ai`, `pi-tui`) MUST literally equal the canonical **floor** `>=0.80.8` stated above, and the four matching `package.json#devDependencies` entries MUST literally equal the canonical **build pin** `~0.80.10` stated above, because npm consumes the manifest mechanically and the manifest cannot anchor-cite. The manifest is therefore the one legitimate non-spec restatement of the two pin literals, and is lock-stepped to this anchor on the same editorial-convention basis as [GOV-30](../governance/req-id-prefix-table-active-b.md#gov-30)'s aggregator lock-step convention (a PR that bumps either literal at this anchor co-edits the corresponding four `package.json` entries in the same commit; the build-time `peerDependencies` and `devDependencies` literal-read assertions gated by [Pi version bump procedure — step 2](./version-bump-intro.md#pi-version-bump-procedure) below are the mechanical witnesses on the manifest side, and a partial landing where a spec literal and its matching manifest entries disagree fails that assertion). The `typebox` peer-dep entry is **not** part of this lock-step — it is pinned to `"*"` per the *typebox* sub-paragraph below.

   *Deliberate deviation from Pi's bundled-package convention.* Pi's `@earendil-works/pi-coding-agent` `docs/packages.md` — *Dependencies* prescribes `"*"` as the `peerDependencies` range for all five bundled packages it names (`@earendil-works/pi-ai`, `@earendil-works/pi-agent-core`, `@earendil-works/pi-coding-agent`, `@earendil-works/pi-tui`, `typebox`). Theta deliberately deviates from that convention for the four `@earendil-works/*` entries by pinning `peerDependencies` to the [Pi-SDK floor](#pi-sdk-pin) `>=0.80.8` rather than `"*"`. The floor is **not** a tilde range, so the earlier install-time minor-skew-detection rationale no longer applies to the peers: an open `>=0.80.8` floor expresses the **minimum** Pi-SDK version whose API shape the runtime requires — the earliest release at which all five ported API shapes coexist (`ModelRuntime`, the reshaped `ModelRegistry`, `CreateAgentSessionOptions.modelRuntime`, the `AuthStorage` root de-export, and the pi-ai `./compat` shape; evidence in `.pi/tmp/upgrade/min-version.md`) — and trades tilde skew-detection for forward-compatibility with future Pi minors. Minor-skew across the four peers is therefore **accepted**: the open floor deliberately does not reject a newer minor (0.81, 0.82, …), so `pnpm` (in its default `node-linker=isolated` mode) and `yarn` no longer fail the install when the host has resolved the four `@earendil-works/*` packages above the floor. The tilde is retained **only** for the `devDependencies` **build pin** `~0.80.10` — the version the package is built and tested against — which is not an install-time contract imposed on the host. The residual failure mode a future breaking Pi minor would surface as is a per-call `TypeError` at the first affected SDK call site rather than a loud install-time refusal; catching that regression is now owed to the build-time SDK surface-inventory assertion and the editorial-review gate of the [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below rather than to install-time skew detection. The floor/build-pin invariant is what the build-time `peerDependencies` and `devDependencies` literal-read tests, the [Step 0 (d) Peer-dep version](./capability-probe.md#entry-capability-probe) probe, and the literal-anchor list in [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below are built around. The fifth package, `typebox`, follows the convention (`"typebox": "*"`) — see the sub-paragraph immediately below — because the runtime's only `typebox` dependency is `Type.Unsafe`, whose call shape is stable across the TypeBox `0.x` → `1.x` line and therefore does not need install-time minor-skew detection.

   <a id="pic-35"></a> **PIC-35. `typebox` `"*"` no-collapse.** The `typebox` `"*"` declaration MUST NOT be collapsed into the four-entry tilde-pinned group: the two ranges are deliberately distinct and reflect different stability arguments (lock-step minor for the `@earendil-works/*` four; convention-following floor for `typebox`).

   *`typebox` (the fifth Pi-bundled package).* The four `@earendil-works/*` packages above are pinned at the [theta 1.0 Pi-SDK pin range](#pi-sdk-pin) per the lock-step rule. `typebox` is the fifth Pi-bundled package the theta runtime imports (only `Type.Unsafe`, per [*Target surface categories*](./audit-target-categories.md#audit-target-surface-categories) below (typebox carve-out)); per Pi's bundled-package convention (`@earendil-works/pi-coding-agent` `docs/packages.md` — *Dependencies*) it MUST be declared as `"typebox": "*"` so the host's bundled version wins. <a id="typebox-single-instance-precondition"></a>That bundled-version-wins outcome rests on two host-side preconditions the `"*"` declaration cannot itself guarantee: that the host bundles `typebox` at all, and that the package manager dedupes the `"*"` range to the host's single bundled instance. The `"*"` range carries no version-pinning guarantee of its own — it expresses only "accept whatever the host resolves" — so neither precondition is enforced at install time. No runtime probe verifies `typebox` provenance or single-instance resolution beyond the `Type.Unsafe`-callability check owned by **Step 0 (e)** below: a deduping failure that resolves a second, separately-resolved `typebox` instance alongside the host's passes that callability check and is treated with the same undefined-behaviour posture as the **Silent shape drift** carve-out at [Post-probe SDK-shape drift](./capability-probe.md#post-probe-sdk-shape-drift) — the runtime is not required to detect it, and no diagnostic is owed for it. In that case the mis-resolution surfaces downstream of `pi.registerTool` as a wrong-schema-shape failure rather than as a clean `theta/load/host-incompatible` refusal at factory entry. The `"*"` literal MUST NOT be folded into the four-entry tilde-pinned group asserted by the build-time lock-step assertion; it is asserted by its own one-line build-time literal-read assertion. The runtime's single load-bearing dependency on the bundled `typebox` namespace — the presence of `Type.Unsafe` as a callable function — is checked at extension-factory entry by **Step 0 (e)** below, which routes a missing or renamed member to `theta/load/host-incompatible` with `details.kind = "typebox-shape"` rather than to a runtime-time `TypeError` at the first tool-registration call (factory time; the parent-side `Type.Unsafe` use backs prompt-mode tool definitions, while a subagent child builds its own tool definitions in-process). This mirrors the fail-closed-at-factory-entry framing the probe applies throughout: a single-symbol Pi-bundled import whose absence would otherwise crash the runtime at a registration call site is instead refused cleanly at factory entry.
2. *Binder model.* Non-bypass thetas require a structured-output-capable LLM resolved via `ctx.modelRegistry` for the binder pass described in [Slash-Command Argument Binding](../binder.md). When the registry cannot resolve a binder model, the theta fails to load with `theta/load/binder-model-unresolved`. Under the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) the strict-capability probe is degraded and surfaces as `theta/load/binder-model-strict-capability-unknown` (W). Bypass cases (no-params, single-string with no default) skip the binder call entirely. theta 1.0 has no global binder opt-out; bypass is structural and per-theta.
3. *Binder credentials.* The binder reaches its model through the same `ctx.modelRegistry` / `Model<Api>` path (pinned at [model-registry surface](./host-interfaces-core.md#model-registry-pin) below) Pi uses for its own queries; credentials are not stored, fetched, or cached by theta. Theta relies on whatever credential resolution Pi already performs for the resolved model.
4. *Pi-supplied `AbortSignal`.* Pi delivers an `AbortSignal` to every extension entry point theta uses — `ctx.signal` on the slash-command `ExtensionCommandContext` and the `signal` parameter of `tool.execute(...)`. The Web-standard `AbortSignal` / `AbortController` shape (the Node-bundled WHATWG implementation) is a fixed-shape SDK precondition; the canonical member-with-kind enumeration the runtime depends on is pinned by [Step 0 (b)](./capability-probe.md#entry-capability-probe) below. Subagent mode has no in-process cancellation handle — it drives a spawned child `pi` process over the RPC wire — and instead consumes the RPC `{"type":"abort"}` command, sent from a one-shot `thetaAbort.signal` listener registered at spawn time, with a bounded process-tree kill fallback if the child does not exit within the teardown budget; see **Subagent child-process lifecycle** ([PIC-9](./subagent.md#pic-9)) and **Cancellation source** below. The theta runtime's own `AbortController` wrapping rule is in **Cancellation source** below; this prerequisite is what guarantees the inputs that rule consumes.

**Theta-package implementation dependencies.** Theta 1.0's own npm-package dependencies (e.g. `semver` behind the Step 0 capability-probe recipes, `chokidar` behind the `FileWatcher` seam) are implementation-side choices rather than part of the Pi-supplied contract this page owns; they are documented in [Implementation Notes — Runtime](../implementation-notes.md#theta-package-implementation-dependencies-theta-1-0).

<a id="tsc-toolchain-floor"></a>**Build-toolchain floor (TypeScript brand-string surfacing).** Unlike the version-unpinned `semver` / `chokidar` choices above, theta's TypeScript build toolchain carries a load-bearing version floor, because the build-time `SessionShutdownEvent['reason']` type-equality assertion gated by [Pi version bump procedure — step 5](./version-bump-intro.md#pi-version-bump-procedure) below relies on `tsc` surfacing the literal brand string `theta/typecheck/session-shutdown-reason-snapshot` verbatim and grep-ably in its type-mismatch diagnostic — the property the step-1 brand-string carve-out, the step-5 trigger (ii) discriminator, and the [Patch-skew degradation contract](./patch-skew-degradation.md#patch-skew-degradation-contract)'s build-time gate all key on. theta pins a TypeScript floor of `>=5.4.0`; the verbatim, grep-able brand-string surfacing was confirmed against that floor, where TypeScript's literal-type assignability diagnostics quote the offending string-literal type verbatim — including the `/` separators the brand string carries, which no TypeScript identifier could embed. A TypeScript bump past this floor MUST be re-confirmed by editorial review under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below — the same editorial-review gate the host-behaviour presuppositions on this page rely on — because a future TypeScript release that reshaped literal-type mismatch diagnostics so the brand no longer appeared grep-ably would otherwise silently disable the gate.

<a id="session-binding-contract"></a>
<a id="pic-7"></a> **PIC-7. Session-binding contract.** A Pi extension instance is bound to exactly one active user session at any given time. The premise that a session-only swap keeps the same extension instance loaded against the swapped-in session is pinned as [Host prerequisites for the degraded-state branch — presupposition (a)](#degraded-state-host-prerequisites) below. The closed-reason-set pin for the `SessionShutdownEvent['reason']` union Pi delivers to the teardown handler is stated beside the consuming handler at [Session-swap behaviour for in-flight invocations](./session-shutdown-semantics.md#session-shutdown-semantics) below.

<a id="degraded-state-host-prerequisites"></a>
**Host prerequisites for the degraded-state branch.** The [`reason: "new" | "resume" | "fork"` edge-case bullet](./session-only-degraded-state.md#session-only-reason-degraded-state) under **Extension entry point** sub-step 4 below depends on four Pi SDK behaviours that neither the theta 1.0 Pi-SDK pin's `dist/core/extensions/types.d.ts` nor the `ExtensionAPI` / `ExtensionContext` surfaces enumerated above pin as a typed contract. The theta 1.0 runtime treats clauses **(b)–(d)** as **presuppositions** in the same sense as [Future Considerations — No concurrent user sessions in the same host process](../future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals) (which the same paragraph cites as the framing precedent for an unpinned host-behaviour assumption); detection of a Pi minor that violates any of clauses (b)–(d) is by editorial review under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below — the build-time SDK surface-inventory assertion is a presence check on the seven named SDK capabilities and does not enumerate behavioural invariants of this kind, so contributor-side editorial review against this enumerated list is the gate. **Clause (a) is the exception:** at the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) it was **not** an unpinned presupposition awaiting a future-minor editorial check but an *open contradiction* against Pi's documented teardown-and-rebind extension lifecycle (see (a) immediately below); it has since been authoritatively **RESOLVED as `governed-by-rebind`** (the degraded-state branch is retired and replaced by the fail-fast tripwire — see the RESOLVED blockquote below).

> **RESOLVED (theta 1.0): `governed-by-rebind`.** Clause (a) below was an open contradiction in the theta 1.0 contract at the [theta 1.0 Pi-SDK pin](#pi-sdk-pin); it has been authoritatively resolved as **`governed-by-rebind`** — theta's package-loaded `extensions/index.ts` **is** governed by Pi's documented teardown-and-rebind lifecycle, so the session-only degraded-state branch is **unreachable as written** and the [SM-3b / SM-4 / SM-5 / SM-6](../session-model-and-appendix.md#sm-4-session-only-degraded-state) degraded-state obligations are reworked spec-side to the fail-fast tripwire disposition recorded on [session-only-degraded-state.md — *Session-swap fail-fast tripwire*](./session-only-degraded-state.md#session-swap-fail-fast-tripwire). *Evidence (cited code site, per [item (a)'s recording shape](./version-bump-step2.md#bump-step-2-outcome-recording-conventions)).* At the theta 1.0 Pi-SDK pin `~0.75.5`, every session-only path in `dist/core/agent-session-runtime.js` (`newSession` / `switchSession` / `fork`) runs `await this.teardownCurrent(reason, …)` — which fires `session_shutdown` and then calls `this.session.dispose()` — immediately followed by `this.apply(await this.createRuntime({…}))`; the `createRuntime` factory (`dist/main.js`) calls `createAgentSessionServices` (`dist/core/agent-session-services.js`), which constructs a fresh `DefaultResourceLoader` and `await resourceLoader.reload()`, which re-runs `loadExtensions` → `loadExtension` (`dist/core/extensions/loader.js`) re-invoking each extension `factory(api)` against a fresh `pi`. The swapped-in session therefore runs on a brand-new extension instance with a fresh, non-drained `ThetaRegistry`; the drained/tagged registry the degraded branch depended on does not survive. **Not release-blocking; now resolved.** The verdict gates *implementing* the degraded branch, not the release; it is recorded here (the theta 1.0 commit resolving the contradiction stands in for the bump-commit record item (a) prescribes) rather than left to the default descoped disposition. It was tracked on the version-bump editorial-review checklist at [item (a)](./version-bump-step2.md#bump-checklist-instance-survival) below.

  (a) *Extension-instance survival across session-only `session_shutdown` — RESOLVED (`governed-by-rebind`).* The session-only degraded-state branch presupposed that when `event.reason ∈ {"new", "resume", "fork"}`, Pi keeps the same extension instance loaded against the swapped-in session after the `session_shutdown` handler returns. At the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) that premise was an open contradiction against Pi's documented teardown-and-rebind extension lifecycle; it has been authoritatively resolved as **`governed-by-rebind`** (theta's package-loaded `extensions/index.ts` **is** governed by that lifecycle — the investigation is recorded at [Clause (a) contradiction narrative](#clause-a-contradiction-narrative) below, and the determination record is the RESOLVED blockquote above). The degraded-state branch is therefore **unreachable as written and has been retired**: the [SM-4 / SM-5 / SM-6](../session-model-and-appendix.md#sm-4-session-only-degraded-state) (and [SM-3b](../session-model-and-appendix.md#sm-3b-session-swap-semantics)) obligations are reworked spec-side to the fail-fast tripwire disposition — on the session-only reasons the runtime runs the same full teardown as `{"quit","reload"}` and arms the [session-swap fail-fast tripwire](./session-only-degraded-state.md#session-swap-fail-fast-tripwire) rather than entering a degraded state. The one-shot determination was tracked on the version-bump editorial-review checklist at [item (a)](./version-bump-step2.md#bump-checklist-instance-survival) below; its recording shape is [item (a)'s authoritative-determination recording shape](./version-bump-step2.md#bump-step-2-outcome-recording-conventions).

  (b) *Multi-`session_shutdown` delivery permitted to a single instance.* Pi may deliver more than one `session_shutdown` event to the same extension instance — both re-entrant deliveries during an in-flight teardown and fresh follow-up deliveries after a prior session-only handler has returned. The handler's [**idempotent**](./drain-state-contract.md#idempotent-handler-clause) clause and the at-most-once-per-extension-instance-lifetime bound on the teardown-handler diagnostics are written against this multi-delivery shape, as is the idempotent re-arm of the session-swap tripwire (`sessionSwapTornDown` written idempotently); a Pi minor that fires only one `session_shutdown` per instance leaves the **idempotent** clause vacuous but does not break the contract (the at-most-once bound holds trivially under at-most-one delivery). No SDK surface pins multi-delivery either way.

  (c) *`/reload` and `ctx.reload()` recovery.* Operator-issued `/reload` invokes Pi's `ctx.reload()` against the live extension instance, and `ctx.reload()` tears down the extension runtime and re-invokes the registered extension factory, yielding a fresh runtime with a non-drained `ThetaRegistry`. theta 1.0 has no fallback recovery path, so a Pi minor that changes either the slash dispatch routing of `/reload` or the runtime-invalidation/re-invocation semantics of `ctx.reload()` MUST be caught by editorial review under the bump procedure. The at-most-once-per-extension-instance-lifetime bound on the teardown-handler diagnostics is calibrated against the assumption that a fresh runtime starts a new lifetime; a Pi minor that no-ops `ctx.reload()`, or routes `/reload` through a different facet, silently breaks recovery without tripping any other gate. No SDK surface pins this behaviour. Two corollaries follow from the *Operator-issued `/reload`* sentence above and govern how theta and Pi cooperatively keep the `/reload` recovery path reachable: **(c-i)** `/reload` is a Pi-host-owned slash name that Pi's slash dispatcher routes to `ctx.reload()` independently of any theta-registered `pi.registerCommand` handler, so the [drain-state-gated dispatch](./registration-steps.md#slash-handler-registration) short-circuit pinned at sub-step 3 below — which scopes only to theta-registered handlers (the `handler` value passed to every `pi.registerCommand(name, { description, handler })` call sub-step 3 makes) and routes through the `ThetaRegistry.readDrainState()` two-arm enumeration — does **not** intercept `/reload`; the operator's `/reload` therefore reaches `ctx.reload()` even when every theta-registered slash handler in the same extension instance is short-circuiting on arm (b) of that enumeration. A Pi minor that demoted `/reload` to a `pi.registerCommand`-shaped handler (so theta's drain-state short-circuit would intercept it) is the same editorial-review failure mode the *slash dispatch routing of `/reload`* clause above already names. **(c-ii)** Theta MUST NOT register a `.theta` whose effective slash name (per [Discovery — Filename validity](../discovery.md), after `pi.theta` mapping and `as` rename) is `reload`. A pending theta whose derived slash name resolves to `reload` is dropped at sub-step 3 below's session-start cross-format collision check on the same footing as any other collision against the [theta 1.0 cross-format collision source set](./registration-steps.md#theta-1-0-seam-pi-owned-subagents-collision-source-set) — assuming Pi's `/reload` surfaces in `pi.getCommands()` under a `SlashCommandSource` arm in that set, which is itself part of this prerequisite (c) and a Pi minor that hides `/reload` from `pi.getCommands()` MUST be caught by editorial review under the bump procedure on the same footing as the other clauses of (c) above — and emits the same `theta/load/cross-format-collision` diagnostic. The two corollaries together guarantee that the operator's `/reload` reaches `ctx.reload()` without being shadowed by either the drain-state-gated short-circuit (covered by (c-i)) or a colliding theta registration (covered by (c-ii)).

  (d) *Teardown-vs-session-swap reason partition.* Pi's documented `session_shutdown` semantics split the closed `event.reason` set into a teardown half `{"quit", "reload"}` (Pi tears the extension runtime down after the handler returns) and a session-only-swap half `{"new", "resume", "fork"}` (Pi swaps the user session and leaves the extension runtime live — the premise presupposition (a) above builds on). This two-way partition — not merely the closed *set* of five reasons — is the discriminator the session-only teardown path keys on to decide it must arm the [session-swap fail-fast tripwire](./session-only-degraded-state.md#session-swap-fail-fast-tripwire). The `SessionShutdownEvent['reason']` union (item 1 above) pins set membership of the five literals and is gated at build time by the bidirectional type-equality assertion of [Pi version bump procedure — step 5](./version-bump-intro.md#pi-version-bump-procedure) below, but no SDK identifier pins the partition *assignment* of each literal to one half or the other; Pi's `session_shutdown` type doc carries the teardown-vs-session-swap distinction in prose only. theta 1.0 treats the partition assignment as a **presupposition** on the same footing as the *No concurrent user sessions in the same host process* framing in [Future Considerations](../future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals); the contributor SHOULD confirm it by editorial review under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below (checklist item (x)), since a Pi minor that moved a literal across the partition (e.g. making `"reload"` session-only, or `"fork"` always-tear-down) would silently invert the partition without tripping the set-membership type-equality assertion. No SDK surface pins this behaviour.

<a id="clause-a-contradiction-narrative"></a>
**Clause (a) contradiction narrative (skippable decision-log detail).** This sub-section records the investigation behind [presupposition (a)](#degraded-state-host-prerequisites)'s open-contradiction classification above; it is decision-log narrative, not load-bearing normative text — the gate obligation lives in clause (a) and the recording contract in item (a)'s [authoritative-determination recording shape](./version-bump-step2.md#bump-step-2-outcome-recording-conventions), so a reader tracking only the contract may skip it.

Clause (a)'s premise is that the closed-over `ThetaRegistry` handle the slash-command `handler` registered in step 3 below consults (including its drained flag and `drainStateTag`) is the very registry the prior handler invocation drained and tag-transitioned, which is what would have made the retired degraded branch's post-handler degraded slash note observable to the operator on the swapped-in session. Pi's published extension lifecycle documents the opposite of that premise at the [theta 1.0 Pi-SDK pin](#pi-sdk-pin), not at some hypothetical future minor. `@earendil-works/pi-coding-agent`'s `docs/extensions.md` ("Session replacement lifecycle and footguns") states that on `/new`, `/resume`, and `/fork` Pi fires `session_shutdown` for the old instance, then **tears the old runtime down and reloads-and-rebinds extensions for the new session**, then fires `session_start { reason: … }` (followed by `resources_discover`), and directs extensions to do cleanup in `session_shutdown` and reestablish in-memory state in `session_start`; it is explicit that "the old runtime has been torn down, the replacement session has been rebound, and the new extension instance has already received `session_start`," and that captured old `pi` / old command `ctx` session-bound objects are stale after replacement and throw if used. Under that documented behaviour the closed-over `ThetaRegistry` does not survive a session-only swap: the new instance starts with a fresh, non-drained registry and dispatches normally — hiding the degradation rather than surfacing it — which is exactly the failure mode the recovery-by-`/reload` framing depends on not occurring. The CLAUDE.md no-globals/statics/singletons rule and this contract's DI/closure routing rule out a process-global `ThetaRegistry` that could survive the rebind, so there is no escape hatch.

<a id="model-registry-population-presupposition"></a>
**Host prerequisite for binder-model / `model:` load-time resolution.** The per-theta load pass resolves `bind_model:`, `theta.binderModel`, and `model:` references one-shot against the model set `ctx.modelRegistry.getAvailable()` returns at that moment (the resolution path is pinned at [Slash-Command Argument Binding — binder-model parse rule](../binder/binder-model-and-context.md#binder-model-parse-rule), run against the [model-registry surface](./host-interfaces-core.md#model-registry-pin) above). theta 1.0 treats it as a **presupposition** — in the same sense as the degraded-state prerequisites above and the *No concurrent user sessions in the same host process* framing in [Future Considerations](../future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals) — that the set `getAvailable()` returns to a `session_start`-time load pass is the fully-populated model set the host will expose for the session, not a partial or still-warming view. <a id="theta-flag-namespace-presupposition"></a>
**`pi.registerFlag('theta', …)` flag-name overlap and shared flag-value-slot behaviour.** The [`--theta` flag-namespace paragraph](../discovery/discovery-sources.md#theta-flag-namespace) under **Discovery sources** depends on two host behaviours of `ExtensionAPI.registerFlag` / `ExtensionAPI.getFlag` (declared at `dist/core/extensions/types.d.ts`, per [Non-capability `pi.<member>` surfaces](./inventory-audit-intro.md#non-capability-pi-members) above) that neither member's typed signature encodes, so theta 1.0 treats each as a **presupposition** — in the same sense as the degraded-state prerequisites, the binder-model / `model:` load-time resolution prerequisite, the settings write-back key preservation prerequisite, and the interactive-session slash dispatch prerequisite above, and the *No concurrent user sessions in the same host process* framing in [Future Considerations](../future-considerations/model-changes-and-non-goals.md#theta-1-0-non-goals). The two behaviours are separable mechanisms and are independently observable:

  (1) *Non-throwing `registerFlag` overlap write.* `pi.registerFlag('theta', …)` does not throw when another extension has already registered a `'theta'` flag — the registration write records the flag against the calling extension and silently shadows rather than erroring — so theta's step-1 `pi.registerFlag` call cannot fail on a flag-name overlap and no `theta/load/...` code is owed for that case.

  (2) *Shared per-name flag-value slot read by `getFlag`, first-registration-wins.* `pi.getFlag('theta')` reads a per-flag-name value slot shared across the registering extensions rather than a per-extension slot, on a first-registration-wins basis, so a later sibling extension that also registers a `'theta'` flag is silently shadowed (its registration does not displace theta's shared-slot value) while theta's own `pi.getFlag('theta')` read still observes theta's registration when theta registered the `'theta'` flag first.

  No SDK surface pins either behaviour; the contributor SHOULD re-confirm both by editorial review under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below (checklist item (ao)), since a Pi minor that made `registerFlag` throw on overlap, or that moved the flag-value slot to per-extension scope, would change theta's flag-namespace behaviour with no SDK surface-inventory signal. *Authoritative source.* Pi publishes no flag-overlap or flag-value-slot documentation; at the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) both behaviours are observed at the flag-registry that backs `ExtensionAPI.registerFlag` / `getFlag` in `dist/core/extensions/loader.js` (**known-fragile evidence**: this bundler-output path is a diffable artefact at the pin rather than a load-bearing claim about its stable shape across patches; a rename or re-bundle routes item (ao)'s textual-diffability failure to the established version-bump recovery path).

<a id="messages-chronological-order-presupposition"></a>
**Theta-consumed `AgentMessage[]` chronological ordering (both delivering surfaces).** Theta's `AgentMessage[]` consumers rely on a Pi *behavioural* precondition the `AgentMessage[]` type does not encode: the array is **chronologically ordered, oldest-to-newest**. Theta reads an `AgentMessage[]` array on two distinct delivering surfaces, and this presupposition governs both: (1) `SessionContext.messages`, the array `buildSessionContext(...)` returns; and (2) the `messages` field on the terminal `agent_end` RPC event the subagent child emits on stdout (`docs/rpc.md` in `@earendil-works/pi-coding-agent` at the [theta 1.0 Pi-SDK pin](#pi-sdk-pin)), consumed only in subagent mode. Four theta consumers read an array in that order — the [compact-transcript renderer](../binder/binder-model-and-context.md#compact-transcript-format-normative), the binder's [session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session), and prompt-mode untyped-query [trailing-turn extraction](./conversation-drive.md#untyped-query-ok-extraction) read surface (1), while subagent-mode untyped-query [trailing-turn extraction](./subagent.md#subagent-untyped-query-ok-extraction) reads surface (2) — so a Pi minor that reordered either array (e.g. newest-first) would silently corrupt every one of them with no type or SDK surface-inventory signal. The ordering is taken on a best-effort basis against the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) and is re-audited on each Pi minor bump via item (h) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect a reordering from the type surface alone. *Authoritative source.* Pi publishes no array-ordering contract; at the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) the oldest-to-newest order is observed in the append-ordered session entry store both surfaces read from — `dist/core/session-manager.js` — reached for surface (1) through `buildSessionContext` in `dist/core/agent-session.js` and for surface (2) through the terminal `agent_end` RPC event the child serialises from that same store (**known-fragile evidence**: these bundler-output paths are diffable artefacts at the pin rather than load-bearing claims about their stable shape across patches; a rename or re-bundle routes item (h)'s textual-diffability failure to the established version-bump recovery path).

<a id="messages-leading-user-message-presupposition"></a>
**Theta-consumed `AgentMessage[]` leading-`user`-message guarantee (both delivering surfaces).** Theta's turn-grouping relies on a second Pi *behavioural* precondition the `AgentMessage[]` type does not encode and that governs both delivering surfaces above: each array is **non-empty iff its first element is a `UserMessage`** — a non-empty array always begins with a `user` message, and the empty array is the only array that does not begin with one. The binder's [compact-transcript renderer](../binder/binder-model-and-context.md#compact-transcript-format-normative) and [session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) define a turn as a `user` message plus all subsequent assistant / toolResult / custom messages up to (but not including) the next `user` message; this precondition is what makes that definition total over the array, since a leading run of non-`user` messages (an `assistant` or `toolResult` left from an interrupted prior turn, or a `custom:theta-system-note` emitted before the first user input) would otherwise belong to no turn. The empty-array case is admitted by the biconditional and renders no `Recent session context` block, per [BNDR-7i](../binder/binder-model-and-context.md#bndr-7i). Like the ordering precondition above, this guarantee is taken on a best-effort basis against the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) and is re-audited on each Pi minor bump via item (ag) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect a leading non-`user` prefix from the type surface alone.

<a id="token-estimator-determinism-presupposition"></a>
**Production `TokenEstimator` per-message determinism (single-pin).** Theta's [session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) walk relies on a Pi *behavioural* precondition the [`estimateTokens` named export](./host-interfaces-core.md#estimatetokens-named-export) and the [`TokenEstimator` seam (PIC-16)](./host-interfaces-services.md#pic-16) it backs do not encode in their signature: at a fixed [theta 1.0 Pi-SDK pin](#pi-sdk-pin), the production estimator is a **deterministic pure function of its `message` argument** — two calls with an identical `message` return the same integer count, depending on no state outside that argument. The walk sums per-message counts to choose the included-turn set, and the binder's [input-reproducibility contract](../binder/binder-model-and-context.md#why-the-transcript-bytes-are-pinned) (the basis for [BNDR-7](../binder/binder-model-and-context.md#bndr-7)) requires the same session to construct byte-identical binder input across runs and processes; a production estimator whose count varied on identical input would silently shift the included-turn boundary and the transcript bytes without violating any other stated rule. The precondition binds per-message determinism **only within a fixed pin** and makes no claim about per-message-count stability across Pi minor bumps — Pi owns the estimation algorithm, which is version-nondeterministic across minors (the property [PIC-16](./host-interfaces-services.md#pic-16) already acknowledges). Like the ordering and leading-`user`-message preconditions above, this guarantee is taken on a best-effort basis against the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) and is re-audited on each Pi minor bump via item (aq) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; the SDK surface-inventory test cannot detect a non-deterministic or input-impure estimator from the type surface alone.

<a id="triggering-slash-turn-commit-posture-presupposition"></a>
**Triggering slash-command user-turn commit posture (`bind_context: session` snapshot).** Theta's [session-context truncation](../binder/binder-model-and-context.md#session-context-truncation-bind_context-session) walk seeds its newest-to-oldest cap walk from the array tail of the snapshot `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages` returns at slash-invocation time, so the included-turn set — and the transcript bytes [BNDR-7](../binder/binder-model-and-context.md#bndr-7) pins as MUST-reproduce-exactly — depend on which turn sits at that tail. Theta relies on a Pi *behavioural* precondition neither the `ReadonlySessionManager` surface nor the `AgentMessage[]` type encodes: at the moment of that invocation-time read Pi has **not** committed the triggering slash-command user turn (the `/<name> <args>` message that fired the invocation) to the session-entry store the read observes, so the snapshot's tail is the most recent *completed* turn and the triggering turn is absent from the included-turn set. The raw slash text reaches the binder only through the [*User arguments:* line](../binder/binder-bypass-and-envelope.md#system-prompt-structure-normative) (System-prompt structure item 5), not through the session-context block, so the same session always seeds the walk from the same tail and constructs byte-identical binder input. Like the ordering, leading-`user`-message, and per-message-determinism preconditions above, this guarantee is taken on a best-effort basis against the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) and is re-audited on each Pi minor bump via item (as) of the *Editorial-review checklist for unpinned host presuppositions* under [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below; a Pi minor that committed the triggering slash turn before the handler's binder-invocation read would seed the walk from a different tail and silently shift the included-turn set and the transcript bytes with no `AgentMessage[]` type or SDK surface-inventory signal.

<a id="pi-slash-handler-promise-lifecycle-presupposition"></a>
**Pi slash-handler promise lifecycle (theta consumption posture).** The runtime side of the cancellation chain pinned below and in [`spec.md` — Session model — SM-3b session-swap semantics](../session-model-and-appendix.md#sm-3b-session-swap-semantics) — `ctx.signal` triggers `thetaAbort.abort(reason)`, the symmetric direction unblocks `await ctx.waitForIdle()`, and the `session_shutdown` handler awaits `Promise.allSettled(activeInvocations.map(inv => inv.disposeBarrier))` — is load-bearing only under six theta-side presuppositions about how Pi schedules and drives to completion the slash-command handler the runtime registers via [`sdk-cap-slash-command-registration`](./capability-inventory-items.md#sdk-cap-slash-command-registration), and about how Pi delivers the turn-lifecycle events the runtime's cancellation-forwarding handlers consume. The runtime records them here as a theta-side consumption posture rather than as Pi-side guarantees, so a future Pi change in this area surfaces against a named presupposition rather than passing silently: (i) the theta presupposes Pi awaits the slash-handler's returned `Promise` for the full invocation, including any time after `ctx.signal` fires its abort (i.e. the handler's return is not treated as fire-and-forget once the signal aborts); (ii) the theta presupposes Pi imposes no internal deadline on the handler's `Promise` settlement (no force-resolve, abandon, or detach of the in-flight handler after `ctx.signal` aborts or otherwise); (iii) the theta presupposes `ctx.signal` is Pi's only out-of-band interaction with the in-flight handler (no parallel rejection, kill switch, or other channel by which Pi unilaterally terminates the handler outside the `AbortSignal` surface); and (iv) the theta presupposes Pi *eventually* delivers a terminal `agent_end` (`willRetry: false`) for the driven turn — settling `await ctx.waitForIdle()` in prompt mode (and resolving the session-local terminal-`agent_end` subscription the driver awaits in subagent mode per **Conversation drive — subagent mode** below) — so a never-terminating turn, or a future Pi retry-cap that abandons the turn without a terminal `agent_end`, would hang the theta invocation until the cancellation path fires (`ctx.waitForIdle()` has no internal deadline per **Conversation drive — prompt mode** below); (v) the theta presupposes Pi delivers each of the five turn-lifecycle events the runtime's cancellation-forwarding handlers subscribe to — `tool_call`, `tool_result`, `message_update`, `turn_end`, and `agent_end` — to the subscribed extension during the active user turn for which `ctx.signal` was minted, and that `ctx.signal.aborted` / `ctx.signal.reason` read from inside each of those handlers reflects the latest abort state for that turn (per the forwarding paragraph under **Cancellation source** below, the runtime forwards `ctx.signal` into `thetaAbort.abort(reason)` from inside these handlers, so a Pi minor that renames, removes, gates, stops delivering, or changes the per-handler `ctx.signal` semantics of any of them would silently degrade the Esc-during-`@`-query forwarding path to a no-op); and (vi) the theta presupposes Pi flips `thetaAbort.signal.aborted` within a single macrotask — i.e. the abort runs synchronously inside one Pi-dispatched event handler rather than being deferred across multiple macrotasks — so the single macrotask turn the loop-iteration checkpoint yields before its signal-check (per [Cancellation — Granularity](../cancellation.md) and the [`Checkpoint` seam](./host-interfaces-services.md#checkpoint-seam) `loop-iter` production wiring) is sufficient for a pending abort to land before a compute-bound loop's next iteration, so a future Pi that deferred the flip across multiple macrotasks could let an Esc during a compute-bound loop be missed at the next iteration without surfacing against any named presupposition. The six presuppositions are taken on a best-effort basis against the [theta 1.0 Pi-SDK pin](#pi-sdk-pin) on `@earendil-works/pi-coding-agent` from **Host prerequisites** above and are not pinned by any typed contract on `ExtensionAPI`, `ExtensionContext`, or `ExtensionCommandContext`; the SDK capability inventory's [cancellation-propagation entry](./capability-inventory-items.md#sdk-cap-cancellation-propagation) covers only that Pi *supplies* the `AbortSignal` at the two extension entry points and does not extend to the handler-promise's settle-time, deadline, out-of-band-interaction, turn-liveness, turn-lifecycle-event-delivery, or abort-flip-timing surface. The [Pi version bump procedure](./version-bump-intro.md#pi-version-bump-procedure) below carries the audit steps that re-check these presuppositions against the candidate Pi minor on every bump: item (g) under step 1 covers (i)–(iii), item (j) covers the (iv) turn-liveness presupposition, item (v) covers the (v) turn-lifecycle-event-delivery presupposition, and item (ad) covers the (vi) single-macrotask abort-flip presupposition.

*Theta's extension-manifest obligations (the ES-module declaration and the entry-point module) are theta-owned manifest content rather than Pi-side host prerequisites; they are specified under [Extension bootstrap and per theta — Extension entry point](./extension-bootstrap-and-per-theta.md#extension-entry-point), not on this page.*
