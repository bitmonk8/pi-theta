# Future Considerations

Features deliberately deferred from loom 1.0, organised by the kind of decision loom 1.0 must make about each. The buckets answer two questions: which loom 1.0 surfaces must leave a forward-compatible seam, and which items are post-loom 1.0 work that loom 1.0 is *not* expected to anticipate.

The categories are:

1. **Tooling deferrals (no loom 1.0 impact).** Items that ship as new tools or commands and require no loom 1.0 runtime decision.
2. **Surface extensions (loom 1.0 leaves a seam).** Items that extend a loom 1.0 type, struct, or call shape in a backward-compatible way. Each item names the loom 1.0 seam it needs.
3. **Model-level changes (no loom 1.0 seam expected).** Items that change the runtime value model, evaluation model, or tool-result contract enough that loom 1.0 is not expected to anticipate them; adding them post-loom 1.0 will require a migration.
4. **loom 1.0 non-goals.** Cross-cutting loom 1.0 scope decisions where loom 1.0 deliberately leaves no seam because the disposition is "ship without one" rather than "defer the seam". Each item names the [normative owner](./governance.md#gov-12) — always a topic page per GOV-12 — that records the loom 1.0 disposition, alongside any `spec.md` orientation cross-link. (loom 1.0 dispositions that *do* reserve a forward-compatibility seam are deferred loom 1.0 surface extensions, not non-goals — see category 2 above. The contrast with category 3 is along a different axis: category 3 covers model-level changes that would alter what loom 1.0 means by its runtime value, evaluation, or tool-result contracts, whereas category 4 items are cross-cutting "we chose not to deliver this" scope dispositions that do not change those value-model contracts.)

Items occasionally carry a `Depends on:` annotation where they presuppose another item in the list.

loom 1.0 design choices that ship with a known behavioural or diagnostic gap — i.e. constraints on the loom 1.0 runtime itself rather than features deferred from it — are recorded on the topic page that owns the surrounding contract, not here. See for example [Pi Integration Contract — Provider compatibility for typed queries](./pi-integration-contract.md) for the typed-query diagnostic limitation on supported providers.

## Tooling deferrals (no V1 impact)

- **LSP support** for `.loom` and `.warp` files (syntax highlighting, type checking, autocomplete).
- **`loom test` command** for dry-run execution that runs a loom against a recorded transcript or a stub model without hitting a live provider.
- **Richer runtime-event telemetry** — per-loom token aggregation, latency histograms, cost reporting, retention policies, and a consumer-facing read API for the `details: { event }` stream defined in [Pi Integration Contract — Runtime event channel](./pi-integration-contract.md). loom 1.0 ships only the per-occurrence emission and the `RuntimeEvent` payload shape; downstream consumers in loom 1.0 read from Pi's session transcript via existing surfaces.
- **Automated cadence for re-importing Pi patches** — loom 1.0 has no Renovate, no scheduled CI re-install, and no nightly cron that proactively re-imports candidate `@earendil-works/pi-coding-agent` patches into the loom contributor's typecheck cycle. The patch-skew window between a Pi patch landing on a user's machine and the loom-side snapshot edit shipping is therefore bounded only by contributor cadence; see [Pi Integration Contract — Patch-skew degradation contract](./pi-integration-contract.md#patch-skew-degradation-contract) for the loom 1.0 detection contract this defers.

## Surface extensions (V1 leaves a seam)

Each bullet below describes a deferred loom 1.0 extension and points at the normative paragraph that pins the loom 1.0 seam the extension consumes. The seams themselves live on the topic pages that own each surface; this page enumerates only what loom 1.0 chose not to do, and where to read the seam contract.

- **Per-call timeouts** on queries, tool calls, and invokes (loom 1.0 has only AbortSignal-driven cancellation; cf. [Cancellation](./cancellation.md)).
  *Anchored at:* [Query — Options surface](./query.md), [Tool Calls — Options surface](./tool-calls.md), [Invocation — Options surface](./invocation.md).
- **Pre-flight token-count check** before the runtime issues a user turn, so oversized rendered templates surface as `ContextOverflowError` without a wasted provider round-trip.
  *Anchored at:* [Query — loom 1.0 seam: pre-flight token-count nullability](./query.md#v1-seam-pre-flight-token-nullability) (the `tokens_used` / `tokens_limit` nullability is the seam; a pre-flight path may populate them from a local estimate without changing the variant shape).
- **Typed-query support for providers without named-tool forcing** (Gemini, OpenAI Responses, weak local models) via a JSON-mode / system-prompt fallback. loom 1.0 narrows typed queries to providers where pi-ai exposes named-tool forcing (per [Pi Integration Contract — Provider compatibility for typed queries](./pi-integration-contract.md)) and rejects unsupported providers with a load-time warning.
  *Anchored at:* [Pi Integration Contract — loom 1.0 seam: typed-query supported provider set](./pi-integration-contract.md#v1-seam-typed-query-supported-provider-set).
- **Per-query overrides for `model`, `tools`, and `system`** (project → loom → query cascade).
  *Anchored at:* same query-options surface as per-call timeouts — [Query — Options surface](./query.md).
- **User-defined error types beyond `QueryError`.**
  *Anchored at:* [Errors and Results — loom 1.0 seam: discriminator type-openness](./errors-and-results.md#v1-seam-discriminator-type-openness) (one shared callout pins the seam for both this item and the `BinderError`-as-variant item below).
- **`BinderError` as a Loom-visible `QueryError` variant**, once looms become first-class values invocable from non-loom programmatic harnesses that need to observe binder failures structurally.
  *Anchored at:* same discriminator-type-openness seam as user-defined error types — [Errors and Results — loom 1.0 seam: discriminator type-openness](./errors-and-results.md#v1-seam-discriminator-type-openness).
  *Depends on:* First-class loom values (for non-loom harnesses needing structured observation).
- **Automatic context escalation:** when binding fails without context, automatically retry with `bind_context: session` attached — trades a second binder call for a smoother success rate on context-sensitive looms that forgot to opt in.
  *Anchored at:* [Slash-Command Argument Binding — Binder-invocation re-entrancy](./binder.md).
  *Depends on:* [Binder refinement loop](#binder-refinement-loop), only if escalation surfaces user-visible turns; otherwise independent.
  *Decision required before this item can be scoped:* whether automatic escalation surfaces a user-visible turn (composing with the [Binder refinement loop](#binder-refinement-loop)) or stays operator-only.
- <a id="binder-refinement-loop"></a>**Binder refinement loop:** multi-turn `needs_info` negotiation (binder asks the user a clarifying question, gets a reply, retries) instead of loom 1.0's single-shot "system note then stop" behaviour; the same future revision also begins surfacing `ambiguous.candidates` as a user-facing pick-one disambiguation rather than a terminating system note.
  *Anchored at:* [Binder — loom 1.0 seam: binder refinement loop](./binder.md#v1-seam-binder-refinement-loop) (the three loom 1.0 carriers — envelope-schema three-arm discriminator, `ambiguous.candidates` field, distinct failure-arm template prefixes — are pinned with MUST there).
- **Named-argument / key=value invocation syntax.**
  *Anchored at:* [Invocation — Argument-binding style](./invocation.md) (the AST node carries a `style: "positional" | "named"` discriminator; loom 1.0 only emits `"positional"`, but consumers match exhaustively).
- **Richer expression sublanguage inside frontmatter `system:`** (full `${expr}` interpolation rather than just `${param}` paths).
  *Anchored at:* [Parameters and Frontmatter — loom 1.0 seam: `system:` expression sublanguage](./frontmatter.md#v1-seam-system-expression-sublanguage) (the `${…}` parser is the same entry point used by `@`...`` query templates, restricted at parse time to dotted paths by a parser-level filter rather than a hand-rolled regex; widening the filter is additive).
- **Package-style (`@scope/pkg`) and project-rooted (`/looms/...`) `import` paths** — loom 1.0 supports relative paths only.
  *Anchored at:* [Imports — loom 1.0 seam: `Resolver` interface](./imports.md#v1-seam-resolver-interface) (every `.warp` import flows through a single named `Resolver` seam; additional implementations may be registered without rewriting import-site code).
- <a id="mid-loom-user-session-replacement"></a>**Mid-loom user-session replacement.** A future feature that calls `ctx.newSession()`, `ctx.fork()`, or `ctx.switchSession()` from inside a running loom invalidates the factory-captured `pi` reference used by the prompt-mode driver. The future implementation will need to re-acquire `pi` via `withSession` before the next `sendUserMessage`.
  *Anchored at:* [Pi Integration Contract — loom 1.0 seam: mid-loom user-session replacement](./pi-integration-contract.md#v1-seam-mid-loom-user-session-replacement) (the single-captured-reference MUST is pinned there; the loom 1.0 invariant "loom 1.0 looms never trigger replacement" is the lifetime guarantee the seam preserves).
- <a id="symlink-resolution-hardening"></a>**Symlink-resolution hardening for invoke-path containment** — loom 1.0 implements the discovery-root containment check by `realpath`-then-`open`, accepting an irreducible kernel-level TOCTOU window between the two syscalls (see [Invocation — Resolution](./invocation.md)). A future pass replaces the two-step sequence with a single atomic resolve-and-open primitive — `openat2(..., RESOLVE_NO_SYMLINKS | RESOLVE_BENEATH)` on Linux, and the platform equivalents on macOS / Windows / other targets — closing the residual race without changing the loom 1.0-visible diagnostic codes (`loom/load/invoke-path-escape`) or `InvokeInfraError { cause: "load_failure" }` envelope.
  *Anchored at:* [Invocation — loom 1.0 seam: symlink-resolution hardening](./invocation.md#v1-seam-symlink-resolution-hardening) (the single-named-function MUST is pinned there; replacing its body is additive and does not perturb caller code).

### Surface extensions without a dedicated topic-page seam

<a id="surface-extensions-no-dedicated-seam"></a>

The five items below have no dedicated `> **loom 1.0 seam — <name>.**` blockquote on a topic page. Three ride [Parameters and Frontmatter — Unknown-key policy](./frontmatter.md) (or the closely related `tool_loop` field) as their forward-compatibility carrier: that policy provides forward tolerance for any unrecognised key but names no specific deferred field, and there is no per-item topic-page MUST that an implementer can fail to honour. Two are upstream Pi-API extensions whose loom 1.0 carrier is contingent on Pi extending its surface; if Pi never extends the surface, no loom 1.0 obligation activates. Items in this sub-bucket are **not** counted in the "Forward-compatibility seams" tally on [`spec.md` — Scope](../spec.md#scope), which counts only items with a topic-page seam contract.

- **Per-loom `binder_temperature` knob**, if real usage shows authors need to tune the binder's nondeterminism budget.
  *Carrier:* [Parameters and Frontmatter — Unknown-key policy](./frontmatter.md).
- **Operator-level `looms.toolLoopMaxRounds` settings override** — loom 1.0 provides no operator-level fallback for `tool_loop.max_rounds`; the value is per-loom only. A future minor may add a `looms.toolLoopMaxRounds` settings key that supplies the default when `tool_loop:` is absent, mirroring the `bind_model` → `looms.binderModel` chain.
  *Carrier:* [Parameters and Frontmatter — `tool_loop`](./frontmatter.md) (the `tool_loop` field has no dedicated seam callout; the future settings key would be a new top-level configuration key, not a frontmatter widening).
- **User-overridable binder system prompt** — loom 1.0 fixes the binder prompt for predictability.
  *Carrier:* same unknown-key policy as `binder_temperature` — [Parameters and Frontmatter — Unknown-key policy](./frontmatter.md). The deferred extension also needs an injection point in the binder for an author-supplied prompt template; that injection point does not exist in loom 1.0.
- **Pi extension API: Pi-owned subagents exposed as enumerable slash commands.** Pi's `SlashCommandSource` union (`@earendil-works/pi-coding-agent`'s `core/slash-commands.d.ts`) is `"extension" | "prompt" | "skill"`; Pi-owned subagents are spawned via `createAgentSession(...)` and are not surfaced through `pi.getCommands()`, so the cross-format slash-name collision check in [Pi Integration Contract — Extension entry point](./pi-integration-contract.md) and [Discovery — Slash-name collisions at the same priority](./discovery.md) covers exactly three arms in loom 1.0.0. If a future Pi minor extends `SlashCommandSource` to include `"subagent"` (i.e. Pi-owned subagents become enumerable slash commands), the cross-format collision set widens to four arms and the cross-format collision test matrix gains a parallel `.md` subagent fixture.
  *Carrier:* [Pi Integration Contract — loom 1.0 seam: Pi-owned subagents collision source set](./pi-integration-contract.md#v1-seam-pi-owned-subagents-collision-source-set) pins the loom 1.0 invariant (collision source set as a single named set) defensively; the future extension itself activates only if Pi extends `SlashCommandSource`, so no loom 1.0 forward-compatibility obligation rides this item beyond the invariant already captured at the cited callout.
- **Pi extension API: `argumentHint` field on `RegisteredCommand`** — Pi's built-in `.md` prompt templates carry an `argumentHint` that drives a distinct slot in the slash-command autocomplete dropdown, but extension-registered commands have no such slot. Loom's `argument-hint` frontmatter field is therefore consumed by the binder only in loom 1.0 (see [Slash-Command Argument Binding](./binder.md)). A future contributor either upstreams `argumentHint` to Pi's `RegisteredCommand` or revisits the loom 1.0 decision once Pi exposes the field.
  *Carrier:* [Parameters and Frontmatter](./frontmatter.md) (the binder simply has no hint to use beyond the frontmatter field) and [Slash-Command Invocation](./slash-invocation.md) (the `argument-hint` value is preserved on the loom's parsed AST and would only need to be threaded into the registration options).

## Model-level changes (no V1 seam expected)

- **First-class loom values** (`Loom<T>` type, passing looms as arguments, higher-order composition) — loom 1.0 only supports literal-path `invoke` and frontmatter-registered callables.
- **Loom-level concurrency primitives** (e.g. `parallel { ... }` blocks or a parallel-`for` form) building on Pi tools' Promise-returning shape — loom 1.0 keeps every tool call sequential and synchronous-looking.
- **Streaming partial tool results** from Pi's `onUpdate` callback into loom code (e.g. an iterator-style consumption form) — loom 1.0 returns only the final result.
- **Structured tool output schemas**, when Pi (or upstream providers) introduce a strict output-schema contract for tools — loom 1.0 returns `string` from every Pi tool call.
- **Richer untyped-query return shape** (e.g. `Result<string | AssistantMessage, QueryError>` exposing tool-use traces, multiple content blocks, citations) — loom 1.0 returns plain `Result<string, QueryError>`. A future widening would change the value model even though existing call sites would keep working under the union form.
- **Per-parameter `mut` on function parameters** (Rust-style `fn f(mut x: T)`) — loom 1.0 keeps all function parameters immutable.
- **Value-carrying `break expr`** inside `for` / `while` loops — loom 1.0's `break` and `continue` carry no value.
- **`match` guards** (`Ok(x) if x.value > 3 => ...`) and **rest patterns** (`[first, ...rest]`, `{ kind, ...other }`) — neither is in loom 1.0.
- **Non-decimal number literals** (hex `0x...`, octal `0o...`, binary `0b...`) **and underscore digit separators** (`1_000_000`) — loom 1.0 accepts decimal literals only with no separators.
- **Integer-division operator** — loom 1.0's `/` always produces `number`; there is no dedicated truncating-division operator.
- **Additional string and array stdlib methods** beyond the loom 1.0 set — the loom 1.0 method surface is deliberately small; extending it is non-breaking by design (anything not on the list is a parse-time "unknown method" error, leaving the namespace open).

## V1 non-goals

The loom 1.0 non-goals are cross-cutting loom 1.0 scope decisions where loom 1.0 ships without a forward-compatibility seam: a future revision adding the disposition is not anticipated by loom 1.0 and will require a migration. The non-goals here are distinct from the model-level changes in category 3 of the intro list above — category 3 covers post-loom 1.0 changes to loom 1.0's runtime value, evaluation, or tool-result contracts, whereas the non-goals below are cross-cutting "we chose not to deliver this" scope dispositions that do not change those value-model contracts. (The seam-vs-non-goal distinction is stated in category 4 of the intro list at the top of this page; see also [`spec.md` — loom 1.0 non-goals](../spec.md#v1-non-goals) and [Governance — GOV-12](./governance.md).)

- **No per-loom sandbox or capability model.** Pi exposes no per-extension privilege scoping in loom 1.0; loom code runs at full host-process privilege. The runtime's no-extra-mediation rule is owned by [Pi Integration Contract — No additional access channels](./pi-integration-contract.md#no-extra-mediation); the per-loom `tools:` allowlist is a configuration knob over the *model's* reachable callable set, not a host-process sandbox. A future per-loom capability model is not anticipated by loom 1.0 and will require a migration.
  *Recorded at:* [Pi Integration Contract — No additional access channels](./pi-integration-contract.md#no-extra-mediation); [`spec.md` — Orientation — Scope — Trust boundary](../spec.md#scope).
- **No formal source-language migration mechanism for major-version transitions.** loom 1.0 declares the release-process goal that a `.loom` or `.warp` file loading cleanly under V1.0 loads and behaves identically under every loom 1.x release (per [Governance — GOV-15](./governance.md#gov-15); intent statement, not a normative obligation, with no automated equivalence gate planned for loom 1.0); migration across major versions is out of loom 1.0 scope and will require a separate mechanism when loom 2.0 ships.
  *Recorded at:* [Governance — GOV-15](./governance.md#gov-15); [`spec.md` — Orientation — Scope — Source-language stability](../spec.md#scope).
- **No non-Node JavaScript host support.** loom 1.0 binds to Node exclusively (via `process.versions.node` in the Step 0 (a) probe and via Pi's `engines.node` floor). Bun, Deno, browser-embed hosts, and other JavaScript engines are not anticipated by loom 1.0 and would require a probe re-design before being added.
  *Recorded at:* [`spec.md` — Orientation — Prerequisites — Host runtime](../spec.md#orientation); [Runtime Value Model — JavaScript engine assumptions](./runtime-value-model.md#javascript-engine-assumptions).
- **No concurrent user sessions in the same host process.** The loom 1.0 runtime presupposes a single active user session per extension instance, per the Session model paragraph cited below; this is a presupposition rather than a claim that the runtime has audited Pi's full extension API surface and confirmed the absence of any concurrent-sessions facet, and concurrent-session detection is not part of the Step 0 capability probe. A future Pi minor that surfaces concurrent user sessions by mutating an existing pinned shape (e.g. an addition to a previously-closed `kind` enum or a new field on `ExtensionContext`) would surface at the `npm run typecheck` step (step 1) of [Pi version bump procedure](./pi-integration-contract.md#pi-version-bump-procedure) — but only insofar as loom code performs exhaustive matching against the previously-closed shape that TypeScript can flag — before the spec contract drifts; if it instead surfaces as a wholly new method whose name was not previously pinned (e.g. a hypothetical `pi.openParallelSession`), neither the typecheck step nor the build-time SDK surface-inventory assertion (which is a presence check confirming each of the seven named SDK capabilities enumerated in [SDK capability inventory](./pi-integration-contract.md#sdk-capability-inventory) is present on the imported namespace, not an enumeration of new top-level members or a walk of closed-set enum membership) detects it, and the disposition relies instead on the editorial review that [Pi version bump procedure](./pi-integration-contract.md#pi-version-bump-procedure) presupposes — i.e. the surface descriptions on [Pi Integration Contract](./pi-integration-contract.md) being kept current as the contributor walks the candidate package against them — to catch the API addition. If a Pi point release nonetheless surfaces an unanticipated concurrent-session signal at runtime before the next SDK-pin bump trips the build-time assertion, the loom 1.0 runtime defines no *new* rule for the signal *as such*; its pre-existing closed-set validators still fire on whatever previously-closed surface site the signal happens to reach (notably the `session_shutdown` unknown-reason rule at [Pi Integration Contract — Extension entry point](./pi-integration-contract.md), which validates `event.reason` against the closed `{"quit", "reload", "new", "resume", "fork"}` set and emits `loom/host/session-shutdown-reason-unknown` before proceeding with the fixed teardown sequence), and any value appearing *inside* one of those previously-closed surface sites that is not in the closed set is itself an SDK-pin violation that the `npm run typecheck` step cited above is designed to catch before ship — again only insofar as loom code performs exhaustive matching against the closed set that TypeScript can flag. A wholly new top-level member *outside* the pinned surface sites — including a wholly new event channel — is the case already carved out earlier in this bullet: neither the typecheck step nor the surface-inventory assertion (the latter being a presence check on the seven named capabilities, not an enumeration of new top-level members) detects it, and detection relies on the editorial review the bump procedure presupposes, cited earlier in this bullet. The runtime-side binding to Pi's one-active-user-session model is the lifetime invariant that loom 1.0 looms never trigger replacement (anchored on a single factory-captured `pi` reference per loom invocation), stated next to the related seam at [Pi Integration Contract — loom 1.0 seam: mid-loom user-session replacement](./pi-integration-contract.md#v1-seam-mid-loom-user-session-replacement); that seam itself is for an adjacent concept — a future feature that calls `ctx.newSession()` / `ctx.fork()` / `ctx.switchSession()` from inside a running loom and re-acquires `pi` via `withSession` before the next `sendUserMessage` — and is therefore a category-2 surface extension on its own terms, but the lifetime guarantee that rides on it (loom 1.0 looms never trigger replacement) is what binds loom 1.0 to Pi's one-active-user-session model and reserves no forward-compatibility seam of its own. A future Pi minor that surfaces concurrent user sessions in the same host process is not anticipated by loom 1.0, and the loom 1.0 runtime exposes no workaround. Disposition for an unanticipated second session that nonetheless reaches the extension within the [loom 1.0 Pi-SDK pin range](./pi-integration-contract.md#pi-sdk-pin) or a subsequent pin: every site loom 1.0 scopes to a single active user session — the single factory-captured `pi` reference per loom invocation pinned at [Pi Integration Contract — loom 1.0 seam: mid-loom user-session replacement](./pi-integration-contract.md#v1-seam-mid-loom-user-session-replacement), the extension-scoped [`ActiveInvocationRegistry`](./pi-integration-contract.md#active-invocation-registry), and the per-session slash-dispatch serialisation underpinning prompt-mode sequencing per [Pi Integration Contract — Session-binding contract](./pi-integration-contract.md#session-binding-contract) — remains so scoped under loom 1.0, and the second session is itself out of loom 1.0 scope; the loom 1.0 runtime defines no new disposition for it (no refuse-to-load, no bind-to-first-session, no per-session registry keying, no host-incompatibility diagnostic).
  *Recorded at:* [Pi Integration Contract — Session-binding contract](./pi-integration-contract.md#session-binding-contract); [Pi Integration Contract — loom 1.0 seam: mid-loom user-session replacement](./pi-integration-contract.md#v1-seam-mid-loom-user-session-replacement); [Pi Integration Contract — Extension entry point](./pi-integration-contract.md); [`spec.md` — Session model — SM-1 single-active-session binding](../spec.md#sm-1-single-active-session-binding).
- <a id="pi-stdio-capture-facet"></a>**No reliance on a Pi extension-host stdio-capture facet.** The teardown-handler diagnostic emissions enumerated under [Pi Integration Contract — Diagnostic-emission isolation](./pi-integration-contract.md#diagnostic-emission-isolation) write to `console.error` as their last-resort sink. This presupposes that Pi's extension host does not capture, redirect, or suppress `console.*` writes during teardown in a way that hides them from the operator; the loom 1.0 Pi-SDK pin's `dist/core/extensions/types.d.ts` and the `ExtensionAPI` / `ExtensionContext` surfaces enumerated under **Host prerequisites** on [Pi Integration Contract](./pi-integration-contract.md) expose no stdio-capture facet the runtime can rely on, and Pi's documented extension-host behaviour does not pin a stdio-capture policy. loom 1.0 therefore treats teardown-time `console.error` visibility as best-effort. A future Pi minor adding an explicit stdio-capture facet — a typed surface the runtime could query or subscribe to in order to route teardown diagnostics to an operator-visible sink with delivery guarantees — is not anticipated by loom 1.0; adopting it would require revisiting the last-resort sink choice in the **Diagnostic-emission isolation** rule. Detection is by editorial review under [Pi version bump procedure](./pi-integration-contract.md#pi-version-bump-procedure) — the build-time SDK surface-inventory assertion is a presence check confirming each of the seven pinned SDK capabilities is still present on the imported namespace and does not enumerate new top-level members, so the addition of an unrelated stdio-capture facet does not surface at that gate.
  *Recorded at:* [Pi Integration Contract — Diagnostic-emission isolation](./pi-integration-contract.md#diagnostic-emission-isolation).
- **No parallel-`invoke` surface.** loom 1.0 exposes no API by which a parent body can spawn sibling `invoke(...)` children concurrently: every `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns, and no language form opens a parallel-children scope, per [Invocation — Cross-mode semantics](./invocation.md). Adding a parallel-invoke form would require a new evaluation primitive and is not anticipated by loom 1.0.
  *Recorded at:* [`spec.md` — Session model — SM-7c prompt-mode sequential execution](../spec.md#sm-7c-prompt-mode-sequential-execution); [Invocation — Cross-mode semantics](./invocation.md).
- **No parallel fan-out across prompt-mode `.loom` callees.** "Fan-out" here means the runtime dispatching, within a single tool-call round, multiple model-emitted `tool_use` blocks concurrently rather than sequentially (the parallel-dispatch behaviour pinned at [Parameters and Frontmatter — `tool_loop`](./frontmatter.md), "the runtime executing them all (in parallel where the provider supports parallel tool calls)"). Of the two callable kinds actually reachable from any loom's `tools:` table — Pi-host tools and subagent-mode `.loom` callees — loom 1.0 permits fan-out for both; the prompt-mode `.loom` axis is excluded by a load-time mechanism: prompt-mode `.loom` paths listed in another loom's `tools:` are rejected per [Parameters and Frontmatter — `tools`](./frontmatter.md#tools), so the model never sees a prompt-mode `.loom` callable in its tool table to fan out across in the first place (the runtime-level absence of parallel prompt-mode tool calls is a downstream consequence of that load-time rejection, not an independent runtime check). Pi-host tools and subagent-mode `.loom` callees remain present in the tool table and may be fanned out concurrently within a prompt-mode body — Pi-host-tool fan-out rides the same provider-driven parallel-dispatch behaviour cited above at [Parameters and Frontmatter — `tool_loop`](./frontmatter.md) ("the runtime executing them all (in parallel where the provider supports parallel tool calls)"), and the carve-out for subagent-mode `.loom` callees is anchored at [Tool Calls — Concurrency](./tool-calls.md#concurrency) (the normative owner of parallel tool-call concurrency, which permits parallel fan-out for both Pi tools and `.loom` callables and pins the per-sibling spawn shape — own `AgentSession`, captured callable set, derived `AbortSignal`, private transcript — that makes subagent-mode `.loom` callees the surviving `.loom`-callee concurrency surface, with [`spec.md` — Session model — SM-7c prompt-mode sequential execution](../spec.md#sm-7c-prompt-mode-sequential-execution) cross-linked as informative orientation that pins prompt-mode body serialisation against the three sources of in-session overlap). Lifting this restriction would require revisiting prompt-mode body serialisation and is not anticipated by loom 1.0.
  *Recorded at:* [Tool Calls — Concurrency](./tool-calls.md#concurrency); [Parameters and Frontmatter — `tools`](./frontmatter.md#tools); [`spec.md` — Session model — SM-7c prompt-mode sequential execution](../spec.md#sm-7c-prompt-mode-sequential-execution).
- **No admission cap on in-flight loom invocations.** loom 1.0 imposes no cap on in-flight loom invocations (across both modes) within an extension instance and interposes no scheduler above Pi's event loop; per-invocation budgets (binder retry budget, `tool_loop.max_rounds`, `invoke`-chain depth) are scoped to a single invocation and not pooled across siblings, per [Implementation Notes — No invocation cap](./implementation-notes.md#no-invocation-cap). The rule constrains the runtime only — Pi's own session-level serialisation (e.g. prompt-mode slash-command turns within one user session, owned by [Pi Integration Contract — Tool-registration lifetime and visibility (Pi behavioural preconditions on snapshot/restore)](./pi-integration-contract.md#snapshot-restore-pi-behavioural-preconditions), where it is recorded as a Pi behavioural precondition the type signature does not encode and that MUST be re-validated as part of the [Pi version bump procedure](./pi-integration-contract.md#pi-version-bump-procedure) if it weakens) is upstream and remains in force, and the rule does not promise resource unboundedness: catchable allocation failures (the `RangeError` family — `Invalid string length`, `Invalid array length`, `Maximum call stack size exceeded`) still route through `loom/runtime/internal-error` per [Errors and Results](./errors-and-results.md), while uncatchable host fatals (V8 heap-limit OOM via the `OOMErrorCallback` / `abort()` path, and any other engine fatal that bypasses the JavaScript exception machinery) terminate the host process before any wrap can observe them and emit no `loom/runtime/internal-error` diagnostic, per the ownership-boundary statement at the same site (which assigns runtime-value heap to the host JavaScript runtime, OS-level file descriptors / sockets / child-process slots to the host OS, and provider-side rate-limit / quota to the LLM provider, and pins the absence of any loom-level per-class threshold or cross-sibling aggregation, throttling, or storm-detection surface over those classes). A future admission-control surface — and a future cross-sibling aggregation, throttling, or storm-detection surface over any of those resource classes — is not anticipated by loom 1.0.
