# Ceilings 3 and 4

## Per-class retry budget (ceiling #3)

<a id="hc3"></a>

The binder per-class retry budget is ceiling #3 in the four-item list at [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings). Routing class: **load-time system note**. The loom does not start; the operator-facing note is rendered from the failure-mode template matching the *most recent* failure's class — see [Slash-Command Argument Binding — Failure-mode templates](../binder/determinism-cancellation-failure.md#failure-mode-templates-normative). Not an evaluation outcome — nothing reaches loom code, no `Result` value is observable. Governed by the load-time-failure carve-out under [Errors and Results — Terminal outcomes](../errors-and-results/error-model.md#terminal-outcomes). The classification rule that maps an observed binder-call outcome onto a retry class is the provider-error classifier in [Slash-Command Argument Binding — Failure-class taxonomy](../binder/determinism-cancellation-failure.md#failure-class-taxonomy). The five sub-obligations — each a stable inline label citable from tests and downstream revisions — are:

- **HC3-a.** At most one transport-class retry per slash invocation.
- **HC3-b.** At most one malformed-envelope-class retry per slash invocation.
- **HC3-c.** AJV-on-`args` failures are not retried (no per-class retry budget).
- **HC3-d.** Worst-case sum: 3 binder LLM calls per slash invocation (1 initial attempt + 1 transport-class retry + 1 malformed-envelope-class retry).
- **HC3-e.** When the chain ends with both budgets exhausted, the surfaced system note is the row matching the *most recent* failure observed.

## Per-boundary destination/surface table (ceiling #4)

<a id="ceiling-4-table"></a>

Ceiling #4 is the JSON-document depth-5 ceiling against the five enforcement points listed in the per-boundary table below — typed-query response, model-driven tool-call args, code-driven tool-call args, `params` validation, and `invoke<T>` return value (ceiling #4's entry at [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings)). Routing class: **boundary-dependent** per the table below. Every breach surfaces with `schema_keyword: "maxDepth"` and the canonical message `"JSON document depth exceeds 5"` per [Schema Subset — Depth Enforcement](../schema-subset.md); the carrier shape and destination depend on which of the five enforcement points fires:

| Check point | Destination | Surface |
|---|---|---|
| Typed-query response | loom code | `Err(QueryError { kind: "validation", cause: "schema_validation", validation_errors: [{ schema_keyword: "maxDepth", … }], … })` per [Errors and Results — `ValidationError`](../errors-and-results/queryerror-variants.md#queryerror-variants) |
| Tool-call args, model-driven (`@`...`` loop) | the model (loop continues; round counts against `tool_loop.max_rounds`) | tool-error result fed back as the next user turn per [Query — Tool calls during a query](../query.md); no `QueryError` surfaces unless the loop later hits ceiling #2 |
| Tool-call args, code-driven (`<name>(args)`) | loom code | `Err(CodeToolError { cause: "validation", validation_errors: [{ schema_keyword: "maxDepth", … }], … })` per [Tool Calls — Failures](../tool-calls.md) and [Errors and Results — `CodeToolError`](../errors-and-results/queryerror-variants.md#queryerror-variants) |
| `params` validation | depends on call site | `invoke(...)`: `Err(InvokeInfraError { cause: "validation", … })` per [Invocation — Failures](../invocation.md). Slash-load: routes through ceiling #3's no-retry classification (the binder's AJV-on-`args` arm at the [post-default-merge AJV validation](../binder/defaulting-system-note-echo.md#post-default-merge-ajv-validation) hook) and renders through the AJV-on-`args` row of [Slash-Command Argument Binding — Failure-mode templates](../binder/determinism-cancellation-failure.md#failure-mode-templates-normative); the row's `<ajv-summary>` placeholder carries the depth-walk's canonical issue (`<JSON-Pointer> JSON document depth exceeds 5`) per that table's depth-walk clause. The binder LLM call that produced the depth-violating envelope still counts toward HC3-d's worst-case 3-call budget exactly as a regular AJV-on-`args` failure does — no separate depth-violation accounting rule applies and HC3-c's no-retry disposition settles the chain. Not an evaluation outcome, no `Result` value observable |
| `invoke<T>` return value | invoke parent | `Err(InvokeInfraError { cause: "validation", … })` per [Invocation — Failures](../invocation.md) |

<a id="ceiling-4-table-reconciliation"></a>The model-driven row is the only one whose surface is silent at the loom-code level — the depth violation reaches the model as a tool-result and reaches the operator only via the `loom-system-note` channel if and when the loop later exhausts under ceiling #2 (the `validation` runtime event defined in [Pi Integration Contract — Runtime event channel](../pi-integration-contract.md) is keyed off the `Err` variant and therefore does not fire on this row). The depth violation is materialised in the model's tool-result and the round still counts against `tool_loop.max_rounds`. The slash-load arm of `params` validation is the only row that crosses ceilings — the depth-walk fires at the `params` AJV boundary per CIO-3 below, but the failure routes through ceiling #3's load-time system-note classification rather than ceiling #4's recoverable-`Err` path; this cross-ceiling handoff is the only *load-time* case where ceiling #4's enforcement point produces a non-`Err` surface (the runtime non-`Err` surface is the model-driven tool-call args row in the per-boundary table, whose surface is a tool-result fed to the model rather than a `Result::Err` returned to loom code).

## Interaction between ceilings

<a id="ceiling-interaction-order"></a>

Each ceiling is checked at a distinct point in single-threaded interpreter execution. The first ceiling reached along an event's control-flow path that finds its precondition satisfied fires and ends the event; ceilings situated at later points on the same path are not evaluated for that event. The fixed evaluation order — each item below is a REQ-ID citable from tests and downstream revisions — is:

- **CIO-1.** Ceiling #3 (binder per-class retry budget) is evaluated at slash-load time, before any *runtime-class* ceiling fires; the slash-load `params` arm of ceiling #4 also occurs at slash-load time and is routed by ceiling #3's failure-mode templates per the per-boundary table at [Per-boundary destination/surface table (ceiling #4)](#ceiling-4-table).
- **CIO-2.** Ceiling #1 (`invoke`-chain depth) is evaluated at `invoke` entry, before the callee body runs.
- **CIO-3.** Ceiling #4 (JSON-document depth) is the first sub-check at every AJV validation boundary (typed-query response, model-driven `tool_use` args, code-driven `<name>(args)` args, `params` validation, `invoke<T>` return value — five sites, in the same order as the per-boundary table at [Per-boundary destination/surface table (ceiling #4)](#ceiling-4-table)); the depth-walk runs *before* AJV at the same site per [Schema Subset — Depth Enforcement](../schema-subset.md), so #4 always precedes the boundary's other validation. The Anthropic-API term `tool_use` refers only to model-emitted tool-call blocks; the code-driven `<name>(args)` site is a peer AJV boundary in loom code that is NOT a `tool_use` block and is enumerated separately above and in the per-boundary table.
- **CIO-4.** Ceiling #2 (`tool_loop.max_rounds`) is evaluated at the tool-call-round boundary — *after* the round's tool calls have completed and the slot count has been incremented for the just-completed free-phase round (the *tool-call round slot accounting*: the bookkeeping ceiling #2 consults to decide whether the next free-phase model turn would exceed `max_rounds`; see [Glossary — tool-call round slot accounting](../glossary.md) and [Query — Tool-call loop bound](../query/query-tool-loop.md#tool-call-loop-bound)), and *before* the next free-phase model turn is requested. The check has two outcomes, keyed on the post-increment slot count: *free-phase continuation* (`slot_count < max_rounds`: the next free-phase model turn is requested) and the *`max_rounds`-final branch* (`slot_count == max_rounds`: for untyped queries the runtime surfaces `Err(QueryError { kind: "tool_loop_exhausted", … })` before any further turn is issued; for typed queries the runtime dispatches the forced respond turn as the exempt-routed terminator that follows this gating check — the forced respond turn is not itself counted against `max_rounds` and CIO-4 is not re-evaluated against it). The `max_rounds: 0` typed-query boundary is the same `max_rounds`-final branch fired at typed-query start (`slot_count == max_rounds` holds at initialisation, 0 == 0) with no free-phase round preceding it; see [Query — Typed queries are tool-loop-shaped](../query/query-tool-loop.md#typed-queries-are-tool-loop-shaped).
- **CIO-5.** Ceiling #3 never interleaves with #1, #2, or #4: the binder runs only at slash-invocation load time, and `invoke(...)` calls do not invoke the binder per [Slash-Command Argument Binding — Binder bypass](../binder/binder-bypass-and-envelope.md#bypass-cases).
- **CIO-6.** At most one ceiling surfaces per event. The surfaced diagnostic / runtime event carries an optional `masked` field enumerating any other ceiling whose precondition was *also* satisfied at the same check site. The closed identifier set and wire-location split are owned by [`masked` field](#masked-field) below; the per-site reachable mask domain, V1 reachable predicate, and probe-purity (pure-read) MUST are owned by [Pi Integration Contract — PIC-1](../pi-integration-contract/runtime-event-channel.md#pic-1).

## Ceiling-set invariants

<a id="ceiling-set-invariants"></a>

Two set-level claims hold over the loom 1.0.0 four-ceiling set as a whole. They are derivations from the per-ceiling material — the per-ceiling routing-class one-liners on [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings), the [per-boundary destination/surface table for ceiling #4](#ceiling-4-table), and the CIO-1 … CIO-6 evaluation order in [Interaction between ceilings](#ceiling-interaction-order); restating them here keeps the panic-path claim available as the panic-uniqueness claim that the *runtime panic* routing-class label is held by exactly one ceiling — ceiling #1 — in the loom 1.0.0 set.

- **Audience-coverage invariant.** Each ceiling has an observable failure surface — addressed to at least one of *loom code*, *the model*, or *the operator* — and no ceiling is unobservable to all three. (Ceiling #4's model-driven tool-call args row is silent at the loom-code level; it is observable to the model directly and, if the loop later exhausts, to the operator via ceiling #2 — see the model-driven row of the [per-boundary destination/surface table](#ceiling-4-table) and the reconciliation paragraph that follows it.) The membership predicate — which audience(s) a given ceiling is observable to — is not redefined here; it is read off the per-ceiling owner pages (the per-ceiling routing-class one-liners on [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings), the per-ceiling owner pages those one-liners forward-link into, and for ceiling #4 the per-arm classification owned by the [per-boundary destination/surface table](#ceiling-4-table)). A future ceiling-set edit governed by [GOV-15 — Ceiling-set carve-out](../governance/req-id-prefix-table-retired.md#ceiling-set-carve-out) MUST verify, by inspection of the per-ceiling owner pages just named, that the post-edit set preserves this invariant for every ceiling in the post-edit set. For inspection purposes, *observable to loom code* means the owner page names a panic surface or a typed value (`Result`, `Err` variant, or per-boundary-table-classified surface) that loom code can `match` or propagate; *observable to the model* means the owner page names a tool-call response, follow-up prompt, or in-loop transcript surface that reaches the model without further loom-code mediation; *observable to the operator* means the owner page names a `loom-system-note`, load-time system note, terminal-print, or operator-facing log surface. The per-boundary table's `invoke parent` Destination counts as *loom code* for this invariant; the table's `depends on call site` Destination is read by following the table's per-call-site disambiguation and classifying each disambiguated arm under the three audience definitions just given.

- **Ceiling-#1 panic-uniqueness invariant.** Of the four ceilings, ceiling #1 (`invoke`-chain nesting depth 32) is the *only* one whose breach reaches the success / fail / cancelled trichotomy's *fail* arm via the panic path. Ceiling #2 (`tool_loop.max_rounds`) reaches *fail* only via unhandled [query-terminating](../glossary.md#query-terminating) `Err`; ceiling #3 is *load-time* and off the trichotomy per [Errors and Results — Terminal outcomes](../errors-and-results/error-model.md#terminal-outcomes); ceiling #4's query-terminating runtime arms (per the [per-boundary destination/surface table](#ceiling-4-table)) likewise reach *fail* only via unhandled [query-terminating](../glossary.md#query-terminating) `Err` — none panic. (Per CIO-3 / *Five-site list co-edit obligation*, `params` validation is one of CIO-3's five sites and contributes two arms here — the query-terminating `invoke(...)` arm and the slash-load arm that cross-routes through ceiling #3 — so the six ceiling-#4 arms touched by this invariant map onto CIO-3's five sites.) The remaining arms of ceiling #4 — the [in-loop](../glossary.md#in-loop) model-driven tool-call args row and the slash-load `params` arm that cross-routes through ceiling #3 — are not themselves trichotomy events: the in-loop model-driven row's downstream consequences, if the loop later exhausts, surface via ceiling #2's `Err` arm, and the slash-load `params` arm cross-routes through ceiling #3 and so resolves at load-time off the trichotomy. A future ceiling-set edit governed by [GOV-15 — Ceiling-set carve-out](../governance/req-id-prefix-table-retired.md#ceiling-set-carve-out) MUST preserve or update this invariant in lock-step (whether by introducing a second panic-class ceiling, removing or relaxing ceiling #1, renumbering, or changing the four-count; routing-class demotions are covered by *Relax* per [GOV-15 — Ceiling-set carve-out](../governance/req-id-prefix-table-retired.md#ceiling-set-carve-out)).

## `masked` field

<a id="masked-field"></a>

Each `masked` entry is one of the string identifiers `"ceiling#1"`, `"ceiling#2"`, `"ceiling#3"`, `"ceiling#4"`; the field is **omitted** when no co-fire occurred (canonical absence — implementations MUST NOT emit `masked: []`).

The field's location on the wire is `details.masked` for diagnostic-shape surfaces (the `loom/runtime/invoke-depth-exceeded` panic per [Diagnostics](../diagnostics.md)) and `details.event.masked` for runtime-event surfaces (the `validation` and `tool_loop_exhausted` cases per [Pi Integration Contract — Runtime event channel](../pi-integration-contract.md)).

The per-site reachable mask domain (which sibling ceilings can populate `masked` at each surfacing site), the V1 reachable predicate (the boolean read that determines whether `["ceiling#2"]` fires on the typed-query response branch), the pure-read MUST on the masking probe, the verbatim-copy obligation on cascade-twin re-emission, and the dedup-key non-inclusion rule are owned by [Pi Integration Contract — PIC-1 *Hard-ceiling co-fire (`masked`)*](../pi-integration-contract/runtime-event-channel.md#pic-1). The closed identifier set above and the wire-location split are this page's contribution; everything that constrains *when* a non-empty `masked` is permitted to fire — in particular the V1 fact that `["ceiling#2"]` is the only non-empty `masked` reachable in V1, and the predicate that gates it — lives at PIC-1.
