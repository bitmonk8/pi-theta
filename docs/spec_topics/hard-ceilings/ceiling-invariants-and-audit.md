# Ceiling invariants and audit

## Worked consequences

Worked consequences of the interaction order — each illustrates one or more `CIO-N` rules above and is named so a conformance test can cite it directly:

- *Depth-6 tool-call args mid-loop* (CIO-3, CIO-4): a `tool_use` round whose tool-call args payload is depth-6 produces ceiling #4 at the AJV boundary per CIO-3 (the model-driven row of the per-boundary table above — a tool-error result fed back to the model as the next user turn, NOT a `QueryError` returned to loom code), and the round is then permitted to continue the loop — that is one round of `tool_loop`, not exhaustion (CIO-4's slot accounting has not yet fired for this round, and the model gets to react to the validation error).
- *Depth-6 forced respond at `max_rounds`* (CIO-3, CIO-4): on a typed query at the final round permitted by `max_rounds`, CIO-4's slot-increment for the last free-phase round has already occurred and the forced respond turn has been dispatched. When that turn produces a depth-6 payload, CIO-3's depth-walk fires at the typed-query response AJV boundary inside the synthesised respond tool's `execute` and surfaces `Err(QueryError { kind: "validation", cause: "schema_validation", validation_errors: [{ schema_keyword: "maxDepth", … }], … })` per the typed-query response row of the ceiling #4 per-boundary table above. The `tool_loop_exhausted` variant is not reachable on this path — the forced respond turn is precisely the typed-query terminating mechanism CIO-4's `max_rounds`-final branch routes to. Respond-repair (when configured) may follow; per [Query — Tool-call loop bound](../query.md), each respond-repair follow-up gets a fresh `tool_loop` budget, so retry is gated by `respond_repair.attempts`, not by `max_rounds`. (An untyped query at `max_rounds` has no forced respond turn and is governed by the plain-text exhaustion case, not this bullet.)
- *`invoke` panic mid-loop* (CIO-2, CIO-4): an `invoke(...)` issued from inside a tool-loop round counts against the parent chain's depth budget per CIO-2 and, if it panics with `loom/runtime/invoke-depth-exceeded`, the panic propagates per [Errors and Results — Runtime panics](../errors-and-results.md) and the parent's `tool_loop` counter is irrelevant (the loop never resumes — CIO-4's slot accounting is not reached for the aborted round).

## No additional loom 1.0 runtime ceiling applies

<a id="no-additional-ceilings"></a>

The four enumerated non-existence claims below — each a stable inline label citable from tests and downstream revisions — are:

- **NOCEIL-1.** No wall-clock timeout per query / tool-call / invoke. Per-call timeouts are deferred per [Cancellation](../cancellation.md) and [Future Considerations](../future-considerations.md), and the absence is enforced at parse time by rejecting any `timeout:` field at every site where a future per-call timeout could land.
- **NOCEIL-2.** No per-query response-token cap and no cumulative-token budget. The only token-domain failure surface in loom 1.0 is provider-detected `ContextOverflowError` per [Errors and Results — `ContextOverflowError`](../errors-and-results/queryerror-variants.md#queryerror-variants); a pre-flight token-count check is deferred per [Future Considerations](../future-considerations.md).
- **NOCEIL-3.** No runtime-value memory ceiling. String length, array length, and total heap are bounded only by the host process. *Catchable* host allocation failures — `RangeError: Invalid string length`, `RangeError: Invalid array length`, `RangeError: Maximum call stack size exceeded`, and any other JavaScript-exception-shaped allocation failure the host throws — are part of the runtime-defect surface and route through `loom/runtime/internal-error` per [Errors and Results — Runtime panics](../errors-and-results.md). *Uncatchable* host fatals — V8's `FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory` and any other engine fatal that bypasses the JavaScript exception machinery (the V8 `OOMErrorCallback` / `abort()` path) — terminate the host process; the runtime cannot observe them and emits no diagnostic. Operators relying on a finite heap ceiling MUST use a host-side mechanism (e.g. `--max-old-space-size`, container memory caps, `--heapsnapshot-near-heap-limit` for triage); loom 1.0 makes no claim that the loom runtime can intercept the fatal path.
- **NOCEIL-4.** No loom-level host-language stack-depth ceiling distinct from the 32-level `invoke`-chain bound above. The 32-level `invoke`-chain bound is the only loom-level frame-depth ceiling. The host language's native call-stack bound (e.g. V8's recursion limit) is not separately ceilinged by the runtime; an exhaustion of that bound surfaces through the catchable-`RangeError` arm of NOCEIL-3 (`RangeError: Maximum call stack size exceeded`).

<a id="audit-methodology"></a>

*Audit methodology.* The closure of the `NOCEIL-1` … `NOCEIL-4` set above rests on a four-axis enumeration of where a runtime resource bound could fire.

<a id="four-axis-closure-list"></a>

***Four-axis closure list.*** The four axes are:

- *(i)* **time** — *(ii)* `NOCEIL-1` — *(iii)* wall-clock latency on a query, tool call, or invoke is uncapped in loom 1.0; the absence is enforced at parse time by `NOCEIL-1`'s `timeout:`-field rejection.
- *(i)* **token economy** — *(ii)* `NOCEIL-2` — *(iii)* per-call response tokens and cumulative provider-billed tokens are unceilinged, with provider-detected `ContextOverflowError` the sole token-domain failure surface in loom 1.0 per `NOCEIL-2`.
- *(i)* **runtime-value memory** — *(ii)* `NOCEIL-3` — *(iii)* string length, array length, total runtime-value heap, and any host allocation failure that surfaces as a catchable JavaScript exception or uncatchable host fatal are bounded only by the host process per `NOCEIL-3`.
- *(i)* **frame depth** — *(ii)* `NOCEIL-4` (with the 32-level `invoke`-chain bound as the only loom-level ceiling on this axis) — *(iii)* the host-language native call-stack bound is not separately ceilinged by the runtime and falls through `NOCEIL-3`'s catchable-`RangeError` arm.

Four further candidate shapes route outside or through the four axes above without earning a separate ceiling. Provider-side rate-limit / quota exhaustion is *outside the runtime ceiling closure entirely* and surfaces on the provider turn (no `execute()` call, no `tool_call_id`, and no `cause: "execution"`) as `Err(QueryError { kind: "transport", http_status: 429, … })` — or other 4xx-not-overflow `http_status` per the binder's transport-class classifier at [Slash-Command Argument Binding — Failure-class taxonomy](../binder/determinism-cancellation-failure.md#failure-class-taxonomy) — per [Errors and Results — `TransportError`](../errors-and-results/queryerror-variants.md#queryerror-variants), and is governed by the provider contract, not the loom 1.0.0 ceiling set. Host-tool resource exhaustion (file-descriptor exhaustion such as `EMFILE` on `read`, child-process-handle exhaustion such as `ECHILD` on subprocess spawn, network-socket exhaustion, I/O bandwidth, and any other host-tool-side resource shape) is likewise *outside the runtime ceiling closure entirely* and surfaces as `Err(QueryError { kind: "code_tool", … })` from the tool's `execute()` outcome (cause `execution` for a host-side throw or `isError: true` return) per [Tool Calls — Failures](../tool-calls.md) and [Errors and Results — `CodeToolError`](../errors-and-results/queryerror-variants.md#queryerror-variants), and is governed by the host-tool contract, not the loom 1.0.0 ceiling set. Schedulable concurrency / event-loop saturation falls through `NOCEIL-1` (wall-clock) for any visible delay. AJV validator compilation cost falls through `NOCEIL-1` (wall-clock) at parse / load time and through `NOCEIL-3`'s catchable-`RangeError` arm at runtime.

***Routing obligation for future ceilings.*** Any future runtime resource the spec might consider ceilinging MUST surface either as a new entry in the four-item ceiling list at [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings) or as a documented fall-through to one of the four `NOCEIL-1` … `NOCEIL-4` claims above (each `NOCEIL-N` claim names the documented fall-through for its axis); a runtime resource that does neither is a closure defect under this paragraph. A future spec-editor introducing, retiring, or relaxing a ceiling under [Governance — GOV-12](../governance.md) MUST re-run this four-axis check and update the four-axis closure list above in lock-step with the change: an *introduction* adds a one-line entry under one of three cases: (1) *default case* — the new ceiling cleanly fits one of the four existing axes, and the entry names that axis; (2) *split-axis case* — the new ceiling carves a sub-axis from within one of the four existing axes, and the entry names the parent axis being split, the new sub-axis, and the residual sub-axis, with rationale for treating the two sub-axes as separate axes; (3) *new-axis case* — the new ceiling occupies a fundamentally new resource axis not covered by the four (e.g. file-descriptor count, child-process count, network-connection count, on-disk artefact size, where the new resource is ceilinged at the runtime layer rather than passed through to the `code_tool` `Err` outside-the-closure surface above), and the entry names the axis and the rationale and, in the same commit, adds the new resource type to the four-axis closure list at the head of this paragraph so the closure is widened in lock-step; a *retirement* does NOT delete the corresponding per-axis entry while the resource the retired ceiling bounded still exists at runtime — deleting the entry would leave the resource unrouted in violation of the routing-obligation MUST above; instead, retirement rewrites the per-axis entry into one of four forms — one per post-retirement runtime trajectory enumerated at the *Retire* clause of [Governance — Operational definitions](../governance/req-id-prefix-table-retired.md#operational-definitions): (i) **succeed** — the resource is no longer bounded at the loom layer; the per-axis entry is rewritten to point at the verbal claim "no loom-layer bound; bounded only by the host process or by upstream ceilings" and the routing destination is recorded as "no loom 1.0 ceiling on this axis"; (ii) **fall through to a documented `NOCEIL-N`** — the per-axis entry is rewritten to point at the existing `NOCEIL-N` claim's documented fall-through (demoting the routing destination from the retired ceiling itself to that `NOCEIL-N`); where no existing `NOCEIL-N` applies, this form coins a new `NOCEIL-N` entry in the `NOCEIL-1` … `NOCEIL-4` closure list above (extending the list per GOV-16's inline-label allocation discipline) and points the per-axis entry at it as the special case of (ii); (iii) **fall through to another ceiling** — the per-axis entry is rewritten to point at the absorbing ceiling's per-axis entry (the routing destination becomes the other ceiling, not a `NOCEIL-N`); or (iv) **non-terminating absent cancellation** — the per-axis entry is rewritten to a documented `NOCEIL-N` or to a verbal claim ("operator MUST rely on cancellation; no loom 1.0 termination guarantee"), at the spec-editor's choice; this case is recorded explicitly because it is an operator-visible behaviour change. The "annotates the entry in place as retired with a pointer to the retirement-release diff" annotation is preserved as the bookkeeping form of any of (i)–(iv), and the pointer takes the SHA-or-tag form pinned at the *Retire* clause of [Governance — Operational definitions](../governance/req-id-prefix-table-retired.md#operational-definitions) (either the 7-character abbreviated commit SHA of the retirement commit or the release tag of the loom 1.x release that performed the retirement, nothing else); a *relaxation* rewrites the corresponding per-axis entry to reflect the new bound while leaving the routing destination unchanged. Each per-axis entry added or rewritten under the MUST above must name (i) the axis (one of the four above, or a newly introduced axis), (ii) the routing destination (the new ceiling itself, or the specific `NOCEIL-N` claim whose documented fall-through carries the resource), and (iii) a one-clause justification tying the axis to the routing destination; a per-axis entry lacking any of these three components does not discharge the MUST and is itself a closure defect under this paragraph. The methodology paragraph and the four-axis closure list are themselves under the GOV-12 lock-step co-edit obligation in the next paragraph.

***loom 1.0.0 rejected-candidate record (closed).*** Three candidate ceiling cases considered and rejected as separate ceilings during loom 1.0.0 — regular-expression backtracking on the runtime's regex-bearing built-ins (forbidden at parse time as of loom 1.0.0 by the no-regex stance on `split` / `replace` in [Expressions — Built-in methods and properties](../expressions.md#built-in-methods-and-properties)), JSON Schema width pathologies via the JSON-Schema-width keywords listed under *Explicitly not supported* in the [Schema Subset](../schema-subset.md) intro (rejected at parse time as of loom 1.0.0 by that enumeration), and pattern-match recursion depth (any pattern-match evaluation strategy the runtime adopts that exhibits unbounded recursion or memory growth falls through `NOCEIL-3` / `NOCEIL-4` via the catchable-`RangeError` arm, as of loom 1.0.0) — are a closed record of loom 1.0.0 worked examples; future spec-editors performing the four-axis check supply their own per-edit rationale rather than appending to this loom 1.0.0 worked-examples sub-list (the four-axis closure list above remains extensible per the routing-obligation MUST).

<a id="release-gate"></a>

***Release gate.*** A closure defect under this paragraph surfaces during the GOV-15 reviewer-inspection step at release time as a GOV-12 lock-step violation and blocks the release until the per-operation acceptance criterion below is satisfied: an *introduction* — the new ceiling fits one of the four axes above (or one of the four `NOCEIL-1` … `NOCEIL-4` claims' documented fall-throughs) with a documented routing, or a new axis is introduced with the rationale required by the *new-axis case* above; a *retirement* — the per-axis entry for the retired ceiling has been updated to one of the four forms (i)–(iv) enumerated in the *retirement* clause of *Routing obligation* above, and the four-axis closure list still covers the resource (either via a NOCEIL-N pointer, an absorbing ceiling, or the explicit no-loom-bound annotation); a *relaxation* — the per-axis entry has been rewritten to reflect the relaxed bound and the routing destination still exists. Deferral of the four-axis closure list update past the release that introduces, retires, or relaxes the ceiling is not permitted. The GOV-15 reviewer-inspection step under [GOV-15 — Ceiling-set carve-out](../governance/req-id-prefix-table-retired.md#ceiling-set-carve-out) MUST confirm that, for each [Ceiling-set invariant](./ceilings-3-and-4.md#ceiling-set-invariants), the per-invariant MUST on that bullet is discharged for the post-edit set (audience-coverage: the post-edit set preserves the invariant for every ceiling per that bullet; panic-uniqueness: the invariant is preserved or updated in lock-step per the cases enumerated on that bullet).

The four-item ceiling list at [`spec.md` — Hard ceilings](../../spec/overview-and-orientation.md#hard-runtime-ceilings) is an aggregator under [Governance — GOV-12](../governance.md); a future loom 1.0 leaf that introduces a new ceiling — or retires or relaxes one — updates that aggregator, the changed ceiling's owner page (this one for cross-ceiling content, the per-ceiling owner pages for surface and emission), and the *Audit methodology* paragraph above (per-axis entry edit per the rule stated there — addition for an introduction, per-axis entry rewrite into one of forms (i)–(iv) per the *Routing obligation* paragraph above — for a retirement, rewrite for a relaxation) in the same commit per GOV-12. The carve-out at [Governance — GOV-15 — Ceiling-set carve-out](../governance/req-id-prefix-table-retired.md#ceiling-set-carve-out) names ceiling-set changes as a recognised carve-out to the loom 1.x source-language equivalence promise (scope per [Governance — GOV-15 — Ceiling-set carve-out](../governance/req-id-prefix-table-retired.md#ceiling-set-carve-out)).

*Five-site list co-edit obligation.* Ceiling #4's per-boundary table at [Per-boundary destination/surface table (ceiling #4)](./ceilings-3-and-4.md#ceiling-4-table) above is the canonical enumeration of the AJV enforcement points; a future loom 1.0 leaf that adds, removes, or splits a row of that table MUST update the five-site enumeration in CIO-3 above (in this page) and the per-site reachable-mask-domain table in [Pi Integration Contract — PIC-1](../pi-integration-contract/runtime-event-channel.md#pic-1) in the same commit, in addition to the GOV-12 aggregator update covered above.
