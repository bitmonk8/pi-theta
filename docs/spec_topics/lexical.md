# Lexical Structure

**Source files.** `.loom` and `.warp` files are decoded and normalised before lexing; every other rule on this page (and every other spec rule mentioning "newline") operates on the post-normalisation stream.

- **Encoding.** Source files are UTF-8. A leading UTF-8 BOM (`EF BB BF`) is consumed and ignored. Any other BOM, or any byte sequence that is not valid UTF-8 (including lone surrogates), is a load-time error `loom/load/invalid-encoding` reporting the file path and the zero-based byte offset of the first invalid byte in the original (pre-normalisation) file content; for a non-UTF-8 BOM the reported offset is `0` (the BOM's leading byte, not the byte at which the UTF-8 state machine first faults). The runtime performs no transcoding from other encodings (Latin-1, UTF-16, etc.); a Notepad-style UTF-16 save fails fast rather than producing mojibake.
- **Newline normalisation.** Before lexing, line endings are normalised: `\r\n` → `\n` and bare `\r` → `\n`. Every spec rule that mentions "newline" — statement separation below, the "literal newline in regular string is a parse error" rule below, `@`...`` query newline-trim and dedent (see [Query](./query.md)), and `///` doc-comment joining (see [Descriptions](./descriptions.md)) — operates on the normalised stream. A `.loom` checked in with CRLF line endings therefore tokenises and dedents byte-identically to the same file with LF endings; this is observable behaviour, not an implementation detail.
- **Path literals.** Every path literal in source — `import "..."`, `invoke("...", ...)`, and `.loom` entries inside `tools:` (see [Imports](./imports.md), [Invocation](./invocation.md), [Parameters and Frontmatter](./frontmatter.md)) — uses forward-slash separators only. A backslash inside a path literal is a parse error `loom/parse/invalid-path-separator`. (Path literals are otherwise ordinary string literals; `\\` already lowers to a single backslash via the standard escape table below, so the parse error is unambiguous — there is no "did the author mean an escape" interpretation.) Discovery roots and the `--loom <path>` CLI flag accept OS-native paths with no syntactic restriction; their resolved targets are compared in normalised forward-slash form for the discovery collision rule (see [Directory Convention](./discovery.md)).
- <a id="extension-matching"></a>**Extension matching.** The `.loom` and `.warp` file extensions are matched **byte-exact in lowercase ASCII** wherever they appear: in the discovery glob (`*.loom` per [Directory Convention](./discovery.md)), in path literals consumed by `import`, `invoke`, and `.loom` entries inside `tools:` (per the parse-time checks `loom/parse/import-non-warp-extension`, `loom/parse/invoke-non-loom-extension`, and the `tools:` `.loom`-path rule in [Parameters and Frontmatter](./frontmatter.md)), and in the settings/CLI extension check (`loom/load/invalid-extension` per [Directory Convention — `loomPaths` entry schema](./discovery/package-and-settings.md#loompaths-entry-schema)). No case-folding is performed at any of these sites: a file saved as `Plan.LOOM`, an `invoke("./mod.LOOM", ...)` literal, an `import { X } from "./lib.WARP"`, a `tools: [./helper.LOOM]` entry, and a `loomPaths` settings entry pointing at `foo.LOOM` all behave identically across operating systems — the lowercase-extension check fails, regardless of whether the host filesystem is case-sensitive. The rule composes with the per-source case-collision rule on stems (see [Directory Convention — Case-insensitive filesystem collisions](./discovery.md)) without contradiction: `Plan.LOOM` is not a `.loom` file at all, so the question of whether it collides with `plan.loom` does not arise.
- **Diagnostic spans.** BOM consumption and newline normalisation both happen *before* span recording, so diagnostic line/column numbers refer to logical lines on the normalised stream — column 1 of line 1 starts after the BOM, and a CRLF source line counts as one newline, not two.

**Identifiers.** `[A-Za-z_][A-Za-z0-9_]*`, case-sensitive. The **first letter's case is enforced** by the parser — it is what makes case-based pattern disambiguation in `match` work without additional grammar:

- **PascalCase** (uppercase first letter) is required for: `schema` names, `enum` names, `enum` variant names, and any user identifier introduced as a type-like binding. The built-in `Ok`, `Err`, and `Result` follow the same rule.
- **lowercase-first** (a lowercase letter, or `_`) is required for: `let` and `let mut` bindings, function parameters, function names, and schema field names. Both `snake_case` (`experience_years`) and `lowerCamelCase` (`experienceYears`) are accepted; the parser only cares about the first letter. The lowercase-first rule applies to the **loom-side** field identifier; the field's *wire* name (what appears in JSON sent to and received from the model) may be any string via the `as "WireName"` rename clause described in [Schema Declarations](./schemas.md).

Violating either rule is a parse error: `loom/parse/schema-case-mismatch` ("schema name must start with an uppercase letter") or `loom/parse/binding-case-mismatch` ("binding name must start with a lowercase letter or `_`"); see [Diagnostics](./diagnostics.md). Inside `match` patterns the same first-letter rule then disambiguates without ambiguity: a lowercase identifier introduces a fresh binding, an uppercase identifier refers to an existing schema, enum, or constructor in scope (see [Errors and Results](./errors-and-results.md)). The casing rule is the *only* enforced naming constraint; identifier length, internal casing, and underscore use are otherwise free.

**Reserved keywords.** Cannot be used as identifiers: `let`, `mut`, `fn`, `if`, `else`, `for`, `in`, `while`, `break`, `continue`, `return`, `match`, `schema`, `enum`, `import`, `export`, `from`, `as`, `by`, `invoke`, `true`, `false`, `null`, `Ok`, `Err`, `Result`, `string`, `number`, `integer`, `boolean`, `array`, `void`. Using one of these in identifier position is `loom/parse/reserved-keyword-as-identifier`. The discard binding `_` is also reserved (it is not a regular identifier and cannot be referenced after binding). `array` and `Result` remain reserved as identifiers, but double as the closed set of generic-type constructor keywords in type position: keeping them reserved is what stops them matching `NamedType ::= Ident`, while the generic-application production in [Grammar Appendix — Type grammar](./grammar.md#type-grammar) makes each reachable as a parameterised type.

**Statement terminators.** Statements are separated by newlines; semicolons are not part of the grammar. A statement implicitly continues across newlines only when the parser cannot otherwise close it — open `(` / `{` / `[`, a trailing binary or ternary operator, a trailing comma, or a leading binary or ternary operator on the next non-blank line. The continuation-trigger set is closed and is normatively tabulated in [Grammar Appendix — Newline continuation](./grammar.md#newline-continuation). **Blank lines do not break a continuation:** when a trigger holds, the parser continues across one or more newlines regardless of how many of them are blank (`let x =\n\n  foo` is one statement equivalent to `let x = foo`). The same rule means single-line `if (x) stmt` does not exist; bodies of `if` / `for` / `while` / `fn` are always braced blocks. Violations are `loom/parse/single-line-if`.

**Comments.** Line comments only. `//` is a regular comment; `///` is a doc comment that lowers to a JSON Schema `description:` (see [Descriptions](./descriptions.md)). Block comments (`/* ... */`) are not supported and surface as `loom/parse/block-comment`. Comments inside the *text* of a `@`...`` query template are not comments — they are part of the rendered prompt. Comments inside a `${...}` interpolation behave exactly as in any other expression position.

**String literals.** Single- (`'...'`) and double-quoted (`"..."`) forms are equivalent. Escape sequences: `\"`, `\'`, `\\`, `\n`, `\t`, `\r`, `\u{XXXX}` (Unicode scalar value, 1–6 hex digits). The escaped value `v` is well-formed iff `v ≤ 0x10FFFF` and `v` is outside the UTF-16 surrogate range `0xD800`–`0xDFFF`; a recognised `\u{...}` form whose value is out of range or names a surrogate is `loom/parse/invalid-unicode-escape`. A backslash followed by any other character is `loom/parse/illegal-escape`. EOF inside an unterminated string literal is `loom/parse/unterminated-string`. **Single-line only** — a literal newline inside a regular string is `loom/parse/literal-newline-in-string`. **No interpolation** — the sequence `${` inside a regular string is plain text. Multi-line text and interpolation belong inside `@`...`` query templates; for non-query multi-line text, build via `+` concatenation with `\n` escapes, or factor the text into a query.

**Number literals.** Decimal only: `42`, `3.14`, `1e10`, `1.5e-3`, `0`, `0.5`. The literal carries no sign — negation is the unary `-` operator applied at parse time. No hex / octal / binary forms and no underscore separators in loom 1.0 (those forms surface as `loom/parse/unsupported-feature`). See [Future Considerations](./future-considerations.md). A literal with no fractional or exponent part has type `integer`; otherwise `number`. `integer` widens implicitly to `number` in arithmetic and assignment positions; the reverse is `loom/parse/integer-narrowing`. The magnitude of a number literal is judged on each lexed token individually, before the parse-time unary-`-` fold (per the no-sign rule above): an `integer`-typed literal token whose value exceeds the safe-integer range (`|value| > 2^53 - 1`) is `loom/parse/integer-literal-out-of-range`, and a `number`-typed literal token whose parsed value is not a finite IEEE-754 double (e.g. `1e400`, which would otherwise parse to `Infinity`) is `loom/parse/number-literal-not-finite`. Both reject at parse time rather than silently rounding to the nearest representable double or yielding `Infinity`, consistent with the `loom/parse/integer-narrowing` treatment of `integer`/`number` boundary violations. Because the bound is per-token, `-12345678901234567890` is rejected — the unary `-` does not bring the magnitude back into range — whereas an out-of-range value reached only through arithmetic on in-range literals is governed by the runtime `number` rules in [Expressions](./expressions.md), not by this parse-time rule.
