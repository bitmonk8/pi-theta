# Determinism cancellation failure

## Determinism

Binder calls use `temperature: 0`. The binder's `context.messages` array is likewise a fixed constant: a single `user`-role message carrying the canonical literal `Bind the slash-command arguments now.` (pinned at [Pi Integration Contract — Binder inference call](../pi-integration-contract/binder-inference.md#binder-inference-call)), identical across looms and across attempts. It is therefore part of the binder call's fixed footprint, not among its variable inputs — the loom-dependent seed below and the rendered system prompt. The request payload layout that carries this `temperature: 0` pin and the seed below into the provider call — the entry-point symbol, the `options` field placement, and the envelope-as-forced-tool attachment mode — is pinned by [Pi Integration Contract — Binder inference call](../pi-integration-contract/binder-inference.md#binder-inference-call). The seed value for a binder call is the 32-bit FNV-1a hash (offset basis `0x811c9dc5`, prime `0x01000193`) of the loom's qualified name as it appears in the slash registry (the bare command name, without the leading `/`), masked to 32-bit unsigned. The byte sequence hashed is the UTF-8 encoding of the bare command name, with no BOM and no NUL terminator. Equivalently, on a slash name `s` matching `^[a-z0-9][a-z0-9_-]*$` the input bytes are the ASCII code points of `s` in order. Reference vectors (loom name → 32-bit unsigned seed, hex):

| Loom name | Seed |
| --- | --- |
| `code-review` | `0x7ba86b63` |
| `hello` | `0x4f9f2cab` |
| `a` | `0xe40c292c` |

Conforming implementations MUST reproduce these values exactly. The mask to 32-bit unsigned applies to the *output*; the multiply step's working width is an implementation choice (e.g. `Math.imul` chains in JS) and per-byte intermediate state is not separately masked beyond what the canonical algorithm specifies. JSON serialisation of the seed is a plain number, not a hex string — the hex notation above is for human cross-checking only. The same loom therefore derives the same seed *value* on every binder call across processes and runs; this fixes loom's deterministic input, not the provider's sampled output (whether the seed reaches the provider at all is governed by the [Provider seed-field mapping](../pi-integration-contract/provider-error-mapping.md#provider-seed-field-mapping), and byte-identical output across runs is a provider property, not a loom guarantee).

> **Note (non-normative):** FNV-1a 32-bit is non-cryptographic and offers no collision guarantees; the hash is used only to make the per-loom binder seed deterministic across processes, not as a registration key (registration-cache collisions are governed by `loom/runtime/registration-cache-collision`, which hashes lowered tool schemas under SHA-256, not binder names).

Whether the seed is included in the request payload, and under which field name, is governed by the per-provider mapping in [Pi Integration Contract — Provider seed-field mapping](../pi-integration-contract/provider-error-mapping.md#provider-seed-field-mapping).

## Cancellation

The binder participates in cancellation per [Cancellation](../cancellation.md). The runtime checks `ctx.signal` immediately before issuing the binder call and forwards the signal into the [binder inference call](../pi-integration-contract/binder-inference.md#binder-inference-call) as its `options.signal`; the initial attempt and every retry permitted by the per-invocation budget below honour the signal. A cancelled binder produces the cancelled-binder system note in the failure-modes table below and the loom does not run. The bypass path (single no-default `string` param, no LLM call) is naturally cancellable at the next regular checkpoint inside the loom body; the cancelled-binder system note does not apply to bypass-eligible looms.

## Failure modes

Binder failures are runtime-handled and surface as system notes in the user's session, never as `Result` values to loom code. loom 1.0.0 has no `BinderError` variant in the `QueryError` union (it would have nowhere to flow — a failed binder means the loom never starts). Every shape below is rendered through the shared discipline in [System-note rendering](./defaulting-system-note-echo.md#system-note-rendering); the table gives the pre-discipline templates.

<a id="failure-class-taxonomy"></a>

### Failure-class taxonomy (normative)

The binder reuses the provider-error classifier specified for typed and untyped queries: the [Provider error mapping](../pi-integration-contract/provider-error-mapping.md#provider-error-mapping) table in [Pi Integration Contract](../pi-integration-contract.md) (recognised overflow envelopes → `ContextOverflowError`; every other 4xx / 5xx response and every network-level failure → `TransportError`). The classifier is consulted independently on every binder attempt — the initial call and each retry permitted by the per-invocation budget below. Its output, combined with the post-classifier envelope/AJV checks below, determines which of the four classes the attempt's outcome belongs to and therefore which retry budget (if any) the failure consumes:

- *Transport-class.* Any classifier output of `TransportError`. Covers all 4xx-not-overflow responses (including HTTP 429 / rate-limit), all 5xx responses, network / TCP / TLS failures, provider-SDK timeouts, and mid-stream truncation classified at end-of-stream per the provider-error mapping table. Eligible for the single transport-class retry (ceiling [HC3-a](../../spec/overview-and-orientation.md#hard-ceiling-3)).
- *Malformed-envelope class.* The classifier returned a successful provider response, the assistant turn called the [binder's structured-output tool](../pi-integration-contract/binder-inference.md#binder-inference-call), but the resulting JSON either fails JSON parsing, fails the [envelope `anyOf` discriminator](./binder-bypass-and-envelope.md#binder-envelope-schema-constructed-dynamically-from-params), has a `kind` outside `ok | needs_info | ambiguous`, or violates the envelope schema (including the `maxLength: 500` cap on `message` and on each `candidates[i]`). The case where the assistant turn returned plain text instead of calling the structured-output tool (provider content-policy refusal, model failure to follow the structured-output contract) is also classified as malformed-envelope, not as transport. Eligible for the single malformed-envelope-class retry (ceiling [HC3-b](../../spec/overview-and-orientation.md#hard-ceiling-3)).
- *AJV-on-`args` class.* A `kind: "ok"` envelope whose `args` fail AJV against the lowered `params` schema after default-merge (per [Defaulting](./defaulting-system-note-echo.md#defaulting)). Not retried (ceiling [HC3-c](../../spec/overview-and-orientation.md#hard-ceiling-3)) — a binder that returned a structurally valid envelope whose `args` violate the params schema is hallucinating field shapes, and a re-prompt with the same system prompt would not change the outcome. Depth-walk fast-fails at the `params` boundary (the cross-ceiling sub-case routed by CIO-1 from ceiling #4 to ceiling #3 per [Hard Runtime Ceilings — Per-boundary destination/surface table](../hard-ceilings/ceilings-3-and-4.md#ceiling-4-table)) are also classified into this class and rendered through the AJV-on-`args` row of [Failure-mode templates](#failure-mode-templates-normative); for that sub-case the row's `<ajv-summary>` placeholder is synthesised from the depth-walk's canonical `ValidationIssue` (`schema_keyword: "maxDepth"`, message `"JSON document depth exceeds 5"`) per the placeholder's depth-walk clause in that table, not from `errorsText` over an AJV `errors` array (AJV did not run at this site).
- *Context-overflow handling.* A classifier output of `ContextOverflowError` is treated as transport-class for retry purposes. The binder system prompt is fixed across attempts, so a context-overflow failure is either flake-equivalent (the retry succeeds) or structural (the second attempt fails identically and surfaces as the *Binder model transport failure* row of the [failure-mode templates](#failure-mode-templates-normative) table). There is no `ContextOverflowError`-specific row in that table; that absence is intentional under this rule.

The interleaving rules in the per-invocation budget paragraph below (a transport failure observed on the retry of a malformed envelope consumes the transport budget, and symmetrically) follow from “the classifier is consulted independently on every attempt”.

### Failure-mode templates (normative)

Renderers MUST emit the surrounding template text verbatim; only the `<…>` placeholders are interpolated. The `<message>` and `<candidates>` placeholders carry model-supplied content and are non-deterministic, but the surrounding template (the `loom /<name>:` prefix, the `—` separator, the trailing parenthetical or `candidates:` clause) is fixed. Wording changes are spec-versioned breaking changes. The `<provider>` placeholder in the transport-failure row is the `provider` field of the classifier-produced [`TransportError`](../errors-and-results/queryerror-variants.md#queryerror-variants), rendered verbatim — the resolved [`Model<Api>.api`](../pi-integration-contract/host-interfaces-core.md#model-registry-pin) value of the binder model (the `api`-shaped form, e.g. `anthropic-messages`, `openai-completions`; source pinned at [`TransportError` — `provider` derivation](../errors-and-results/queryerror-variants.md#queryerror-variants)), not a normalised or display form. `<ajv-summary>` is the validator-neutral in-order concatenation of the failed validation's `ValidationIssue` entries (the `validation_errors` array element schema, per [`ValidationIssue`](../errors-and-results.md)): each entry is rendered as `<path> <message>` from the issue's `path` and `message` fields, and the entries are joined by the two-character separator `; ` in the canonical `validation_errors` order (see [Errors and Results — `ValidationIssue` ordering](../errors-and-results/queryerror-variants.md#err-14)). The summary content is therefore stable across runs against the same envelope and independent of which validator produced the issues; the surrounding template is normative regardless of how that validator evolves its internal formatting. (Non-normative: the loom 1.0 reference produces this string via the AJV helper `errorsText(errors, { separator: '; ' })` with the data-path prefix retained.) **Depth-walk fast-fail clause:** when the underlying failure is the depth-walk fast-fail at the `params` boundary (the cross-ceiling sub-case classified into AJV-on-`args` per [Failure-class taxonomy](#failure-class-taxonomy) above), AJV did not run and the `errors` array is empty; in that case `<ajv-summary>` is the depth-walk's single canonical `ValidationIssue` rendered as `<JSON-Pointer> JSON document depth exceeds 5` (single-issue form, no `; ` separator, with `<JSON-Pointer>` the path to the first too-deep node per [Schema Subset — Depth Enforcement](../schema-subset.md#depth-enforcement)). The literal token `maxDepth` (the `schema_keyword` value of that issue) and the canonical message string are the operator-visible origin marker for this sub-case; diagnostics consumers parsing the rendered note can recover the originating ceiling from either.

The six user-facing shapes:

| Cause | System note |
|---|---|
| `needs_info` | `loom /<name>: argument binding needs more info — <model's message>` |
| `ambiguous` | `loom /<name>: ambiguous arguments — <model's message>` (loom 1.0 does not render `candidates`; see [System-note rendering](./defaulting-system-note-echo.md#system-note-rendering) rule 5) |
| Binder model transport failure (after 1 retry) | `loom /<name>: argument binder unavailable (<provider>: <message>)` |
| Binder returned malformed envelope (after 1 retry) | `loom /<name>: argument binding failed — could not parse arguments` |
| AJV validation of the binder's `args` failed (no retry) | `loom /<name>: argument binding produced invalid args — <ajv-summary>` |
| `ctx.signal` aborted before or during the binder call | `loom /<name>: argument binding cancelled` |

Transport failures and malformed-envelope failures each get exactly one retry; the second failure of that class surfaces as the system note above. AJV validation failures on `args` are not retried — a binder that returned a structurally valid envelope whose `args` violate the params schema is hallucinating field shapes, and a re-prompt with the same system prompt would not change the outcome. Respond-repair follow-ups (the mechanism typed queries use for response-schema repair) do not apply to any binder failure; the frontmatter `respond_repair.attempts` / `respond_repair.methodology` knobs apply only to typed queries from loom code, and the binder uses fixed retry counts and does not consult them.

<a id="per-invocation-retry-budget"></a>Each retry-eligible failure class has a single retry budget **per slash invocation**: at most one transport-failure retry and at most one malformed-envelope retry, regardless of how the two interleave. A transport failure observed on the retry of a malformed envelope consumes the transport budget (not a second malformed attempt); a malformed envelope observed on the retry of a transport failure symmetrically consumes the malformed budget. Once a class's budget is consumed it is not replenished, even if the failure first appears as the consequence of another class's retry. Therefore the runtime MUST issue at most **3** binder LLM calls per slash invocation (1 initial attempt + at most 1 transport-class retry + at most 1 malformed-envelope-class retry). When the chain ends with both budgets exhausted, the surfaced system note is the row matching the **most recent** failure observed (e.g. a chain ending in a malformed envelope renders the malformed-envelope row, even if a transport failure occurred earlier in the chain). AJV validation of `args` is unaffected by the cap — it carries no retry budget of its own and may fire on whichever of the (up to 3) calls returns the first structurally valid envelope. This binder retry budget is scoped to a single slash invocation; it is not shared, pooled, or replenished across sibling invocations, and is not a per-session or per-loom resource.

The malformed-envelope retry must re-issue the binder call against the *same* envelope schema (no schema mutation between attempts) so the failure stays observable as a flake rather than a moving target. An abort observed during any retry permitted by the budget above suppresses that retry and surfaces the cancelled-binder note immediately, irrespective of which class's budget remains. An abort observed *after* the binder returned `ok` but *before* AJV validation runs lets validation complete (AJV is fast and uncancellable per [Cancellation](../cancellation.md)) and surfaces at the next checkpoint inside the loom body, consistent with the no-retroactive-`Ok`-to-`Err` rule.
