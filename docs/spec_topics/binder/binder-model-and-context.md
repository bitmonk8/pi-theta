# Binder model and context

## Binder model

Resolved at loom-load time from a two-step chain: `bind_model:` frontmatter field, then the loom-extension setting `looms.binderModel` in `settings.json` (read per [Settings file reads](../discovery/package-and-settings.md#settings-file-reads); not a Pi-recognised setting). There is **no further fallback** — no "tier-2" default, and the loom's own `model:` is not consulted (using it silently negates the cost premise that motivates a separate binder model). When neither source resolves and the loom is not bypass-eligible (per [Binder bypass](./binder-bypass-and-envelope.md#binder-bypass) below), the loom fails to load with `loom/load/binder-model-unresolved`; the loom is reported through the diagnostics channel ([Diagnostics](../diagnostics.md)) and its slash command is **not** registered.

> **Note (non-normative):** Binder calls are structurally function-calling tasks (schema in, JSON out), so cheaper structured-output-capable models (e.g. Claude Haiku, GPT-4o-mini, Gemini Flash) are usually adequate; authors with unusually subtle schemas can override per-loom via `bind_model:`. The model-selection guidance is advisory only — the only normative requirement is the strict-capability gate above.

<a id="strict-capability-requirement"></a>
The resolved model must support strict structured-output / strict tool-input. <a id="binder-model-parse-rule"></a>The binder-model value resolved from the two-step chain above is a single free-form string, not a pre-split `(provider, modelId)` pair; the runtime resolves it to a model at the same load-time pass by running loom's own exact-match resolution over the available models from `ctx.modelRegistry.getAvailable()` (the public model-registry surface pinned at [Pi Integration Contract — model-registry surface](../pi-integration-contract/host-interfaces-core.md#model-registry-pin); its load-time population is presupposed per [Host prerequisite for binder-model / `model:` load-time resolution](../pi-integration-contract/host-prerequisites.md#model-registry-population-presupposition)). The resolver accepts both a canonical `provider/modelId` reference and a bare `modelId`. A bare `modelId` is matched against each available model's [`Model<Api>.id`](../pi-integration-contract/host-interfaces-core.md#binder-model-matcher-inputs) (the model-identity field); a `provider/modelId` reference matches a model whose [`Model<Api>.provider`](../pi-integration-contract/host-interfaces-core.md#binder-model-matcher-inputs) equals the reference's `provider` component (the short provider-id form, e.g. `anthropic` — not the api-shaped `Model<Api>.api`) and whose `Model<Api>.id` equals the `modelId` component. A bare `modelId` that matches models under more than one provider is ambiguous, resolving to no model (not pick-first); loom coins no binder-specific parsing convention beyond this exact-match rule (so the frontmatter example `bind_model: claude-haiku` resolves iff `claude-haiku` matches the `Model<Api>.id` of exactly one available model, while `bind_model: anthropic/claude-haiku` resolves iff exactly one available model has `Model<Api>.provider` `anthropic` and `Model<Api>.id` `claude-haiku`). A configured string that matches no available model — or that is ambiguous across providers — resolves to no model, and the loom fails to load with `loom/load/binder-model-unresolved` (the same code the empty-chain case in [Binder model](#binder-model) above produces). The runtime then probes the resolved `Model<Api>` by name for a `strictCapable` field. The probe is duck-typed: `(model as { strictCapable?: boolean }).strictCapable` is read as a three-valued check — `true` admits the model with no diagnostic; `false` emits `loom/load/binder-model-not-strict-capable` (E) and refuses to register the loom; `undefined` (the field is absent) emits `loom/load/binder-model-strict-capability-unknown` (W) and registers the loom. Under the [loom 1.0 Pi-SDK pin](../pi-integration-contract/host-prerequisites.md#pi-sdk-pin)'s `Model<Api>` (`@earendil-works/pi-coding-agent` at that pinned range), the `strictCapable` field is absent (pinned and gated per [PIC — Strict-capability absence under the pin](../pi-integration-contract/audit-target-categories.md#strict-capability-absence-pin)), so production behaviour is the universal-W branch and runtime envelope-malformed failures surface via the failure-mode template (no diagnostic code; the user-facing system note is the operator surface). The runtime MUST short-circuit the `strictCapable` probe when binder-model resolution yields no model (governed by `loom/load/binder-model-unresolved` instead). The probe field name `strictCapable` is the loom-side constant; if a future `pi-coding-agent` minor exposes the indicator under a different name, the rename is enforced by the bump-procedure step (specifically the strict-capability probe step) and by the build-time SDK surface-inventory assertion that catches "a `Model<Api>` member resembling a strict-capability concept under a name other than the pinned one". Bypass-eligible looms (no-params bypass and single-string bypass; see [Binder bypass](./binder-bypass-and-envelope.md#binder-bypass)) skip both checks — they never call the binder.

<a id="binder-model-hot-reload"></a>Hot-reload of Pi settings (`looms.binderModel` changed at runtime) re-resolves on the next loom load; it does not retroactively fix already-failed loads.

<a id="bndr-11"></a> **BNDR-11.** A `looms.binderModel`-only settings edit MUST NOT re-run binder-model resolution or the [`strictCapable` probe](#strict-capability-requirement) for any already-loaded loom; each such loom continues using its previously-resolved binder-model handle until its next load (a `/reload`, or a content-edit reload per [watcher / hot-reload registration](../pi-integration-contract/registration-steps.md#watcher-hot-reload-registration)), at which load the new value is consulted. No symmetric loss-side `loom-system-note` is emitted: a previously-resolved binder model that would now fail to resolve or fail the strict-capability probe surfaces only at that next load, through the load's own `loom/load/binder-model-unresolved` or `loom/load/binder-model-not-strict-capable` diagnostic. This mirrors the structural-change suppression rule at [Registration steps — *Structural changes*](../pi-integration-contract/registration-steps.md#structural-changes-no-unregister).

When the change would let a previously binder-model-unresolved loom's binder model re-resolve, the runtime emits a single consolidated `loom-system-note` with `content` formatted as `loom watcher: looms.binderModel changed; <N> previously-failed loom(s) now resolve a binder model; run /reload to retry: <names>`, `display: true`, and the `details` payload `{ recovery: { looms: string[] } }` — the fourth shape on the disjoint-by-key `loom-system-note` channel defined under [Pi Integration Contract — System notes](../pi-integration-contract/runtime-event-channel.md#system-note-details-shapes). `recovery.looms` carries the affected looms' final derived slash names (no leading `/`, per [Discovery — Slash-name collisions](../discovery/discovery-sources.md#disc-4)) — exactly the looms whose original load failure was `loom/load/binder-model-unresolved` and whose binder model now re-resolves to a model under the changed setting, computed by re-running binder-model resolution alone with no other load-pass step re-run — in ascending Unicode code-point order. Membership therefore makes no claim that a listed loom's next `/reload` succeeds: a loom whose binder model now re-resolves can still fail its `/reload` for an independent reason (a stale `.loom` parse error, a broken `.warp` import, a schema-lowering failure, or any other `loom/load/*` cause), in which case that failure surfaces through its own `loom/load/*` diagnostic on the [Diagnostics](../diagnostics.md) channel and the recovery note carries no further obligation. The substitution rule is fixed: only `<N>` and `<names>` are substituted; every other character of the template — including the literal token `loom(s)` (always rendered with the parenthesised `s`, never pluralised), the literal settings key `looms.binderModel`, and the literal slash-command name `/reload` — ships verbatim. `<N>` equals `recovery.looms.length`, rendered in base 10 with no thousands separator, no leading zero, and no sign; `<names>` is the `recovery.looms` entries each prefixed with `/` and comma-and-space-joined (`, `) in the same order. The note MUST NOT be emitted when `recovery.looms.length === 0` (a `looms.binderModel` change under which no previously binder-model-unresolved loom re-resolves produces no note). Multiple `looms.binderModel` changes observed within a single settings-file `250 ms` debounce window (the window defined in [Discovery — Caching and reload](../discovery/package-and-settings.md#caching-and-reload)) coalesce into one note; changes in distinct windows fire separately. The note is informational, not a diagnostic; no `loom/load/*` code is registered for it.

## Binder context

Configured via `bind_context:` (`none` | `session`; default `none`).

- `none` — the binder sees only the slash text and the loom's frontmatter. Predictable, cheap, deterministic.
- `session` — prompt-mode-only; the binder additionally receives a bounded most-recent slice of the caller's session as grounding context (see [Session-context truncation](#session-context-truncation-bind_context-session) for the exact bounds).

> **Note (non-normative):** Use `bind_context: none` when the slash arguments are self-contained (e.g. `/code-review TypeScript focusing on error handling, by Ada Lovelace`). Use `bind_context: session` when the loom relies on conversational anaphora (e.g. `/review the spec` resolving "the spec" against the surrounding session). The choice is an authoring decision; runtime behaviour for each value is fully specified above.

Declaring `bind_context: session` on a subagent-mode loom is `loom/parse/bind-context-session-on-subagent` (warning, not error) — subagent-mode looms invoked from a slash command have no caller-session context to attach.

<a id="bndr-10"></a> **BNDR-10.** At slash-invocation time the runtime MUST treat `bind_context: session` on a `mode: subagent` loom as `bind_context: none` for binder-input construction: the [session-context truncation walk](#session-context-truncation-bind_context-session) is skipped and no *Recent session context* block is emitted.

> **loom 1.0 seam — automatic context escalation.** <a id="loom-1-0-seam-automatic-context-escalation"></a><a id="v1-seam-automatic-context-escalation"></a> The binder-invocation path is **re-entrant per loom turn**: the loom 1.0.0 runtime issues exactly one binder call per slash invocation (and `bind_context` is therefore observed at most once per invocation), but the path makes no assumption that `bind_context` is set at most once per loom over the loom's lifetime. The binder's input record (parameter table, raw slash text, optional session-context block) and the resolved binder-model handle are constructed afresh on every binder call, with no cached state that would prevent a second call from observing a different `bind_context` snapshot. The seam is what allows the deferred *automatic context escalation* extension in [Future Considerations — Surface extensions gated on an open design decision](../future-considerations/surface-extensions.md#surface-extensions-open-design) to land additively: a future revision in which a binder call returning `needs_info` triggers an automatic retry with `bind_context: session` attached needs no rework of the binder-invocation path. This re-entrancy invariant is a settled loom 1.0 binder-path property in its own right and is not contingent on the extension's unsettled shape: it holds identically whether the open design decision (escalation as a user-visible turn versus operator-only) resolves either way, since both outcomes reuse the same freshly-constructed second binder call. The normative pin above therefore commits loom 1.0 only to the binder-path re-entrancy property, not to the deferred extension's eventual surface.

<a id="session-context-truncation-bind_context-session"></a>
### Session-context truncation (`bind_context: session`)

Token counts are computed per message through the injected [`TokenEstimator` seam (PIC-16)](../pi-integration-contract/host-interfaces-services.md#tokenestimator-interface) — production wiring delegates to `estimateTokens`, and conformance tests inject a `FakeTokenEstimator` — and the message list is sourced from `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages`; both contracts are catalogued in [Pi Integration Contract](../pi-integration-contract.md). A turn's token count is the sum of the seam's per-message estimate over its constituent messages (user / assistant / toolResult / custom); a turn is a user message plus all subsequent assistant / toolResult / custom messages up to (but not including) the next user message. This turn definition is total over the array only because every non-empty `buildSessionContext(...).messages` begins with a `user` message — the behavioural precondition pinned at [Pi Integration Contract — leading-`user`-message guarantee](../pi-integration-contract/host-prerequisites.md#messages-leading-user-message-presupposition) (surface (1) there) — so no leading run of non-`user` messages can fall outside a turn; the same precondition governs the compact-transcript renderer below, which reuses this turn boundary. The walk relies on `buildSessionContext(...).messages` being chronologically ordered oldest-to-newest, so the newest turn sits at the array tail; this ordering is the behavioural precondition pinned at [Pi Integration Contract — chronological-ordering presupposition](../pi-integration-contract/host-prerequisites.md#messages-chronological-order-presupposition) (surface (1) there), not a property the `AgentMessage[]` type encodes. The runtime walks turns newest-to-oldest and stops including a turn the moment the running token sum would exceed 8000 *or* the running turn count would exceed 20 — whichever bound is reached first. The over-budget turn is excluded entirely (whole-turn truncation; partial messages are not split), as is everything older. Equivalently: a candidate turn is included iff, after inclusion, the running token total is ≤ 8000 *and* the running turn count is ≤ 20; the first candidate that would violate either inequality is excluded entirely and terminates the walk. The included context is rendered as a compact transcript and embedded in the binder's system prompt below the parameter table.

Conformance tests verify the 8000-token budget, the 20-turn cap, and the whole-turn-exclusion rule by injecting a `FakeTokenEstimator` configured with chosen per-message integer counts, so the running token total and turn count cross each cap at the boundaries the worked examples below enumerate without coupling to Pi's estimation heuristic.

*Worked example.* With per-turn token counts (newest first) `[1200, 900, 1500, 2000, 2800, …]` and the 8000-token budget, the walk includes the first four turns (running total 5600), then evaluates the fifth: 5600 + 2800 = 8400 > 8000, so the fifth turn and everything older is dropped. Final included context: 4 turns, 5600 tokens. *Single oversized turn at the front.* If the newest turn alone exceeds 8000 tokens, the walk includes nothing and the binder runs with no session-context block (no special-case; the same exclusive rule applies on the first turn evaluated). *Token-cap equality.* With per-turn counts (newest first) `[3000, 2500, 2500, 100, …]`, the walk includes the first three turns (running total exactly 8000, count 3) and evaluates the fourth: `8000 + 100 = 8100 > 8000`, so the fourth turn and everything older is dropped. Final included context: 3 turns, 8000 tokens. The cap-equality boundary is inclusive. *Turn-cap equality.* With 21 turns whose running token total never exceeds 8000, the walk includes the 20 newest turns (count exactly 20) and evaluates the 21st: count would become 21 > 20, so it is excluded regardless of its token weight. Final included context: 20 turns. The 20-turn boundary is inclusive.

<a id="compact-transcript-format-normative"></a>
#### Compact-transcript format (normative)

The truncation walk above selects which turns are included; this sub-section pins the rendering of those included turns into the body of the `Recent session context` block referenced by *System-prompt structure (normative)* item 6 below. The rendering function MUST be total over the `AgentMessage` union returned by `buildSessionContext(ctx.sessionManager.getEntries(), ctx.sessionManager.getLeafId()).messages` (`user` / `assistant` / `toolResult` / `custom`, whose per-variant field shapes are pinned at [Pi Integration Contract — `SessionContext` and the `.messages` element shape](../pi-integration-contract/host-interfaces-core.md#sessioncontext-shape)).

<a id="why-the-transcript-bytes-are-pinned"></a>
**Why the transcript bytes are pinned.** The enclosing [System-prompt structure (normative)](./binder-bypass-and-envelope.md#system-prompt-structure-normative) sub-section opens by allowing the prompt wording to vary, but the transcript rendering is stricter on purpose: it pins loom's *input-reproducibility* contract, not revisable illustration. What loom controls is byte-identical binder input and a deterministic seed value: the [Determinism](./determinism-cancellation-failure.md#determinism) section fixes `temperature: 0` and a per-loom FNV-1a seed so the same session always constructs the same prompt bytes and derives the same seed on every run and across processes. Whether the provider then maps that `(prompt-bytes, temperature: 0, seed)` input to byte-identical output is provider-dependent and outside loom's control: the seed is sent only for the transports that carry a seed field per [Pi Integration Contract — Provider seed-field mapping](../pi-integration-contract/provider-error-mapping.md#provider-seed-field-mapping) — it is omitted for `anthropic-messages` and `amazon-bedrock`, the transports the [binder-model guidance](#binder-model) steers authors toward — and even for a seed-accepting transport the upstream API may not guarantee bit-exact output across runs. Pinning the input bytes still matters because input wobble in the role tags, the `: ` separator, the turn delimiter, the `[tool-call …]` line form, or the trailing-space contract on the empty-content user message changes the prompt bytes and can change the sampled output even at `temperature: 0`, and because byte-stable input is what keeps the renderings auditable and any prompt-cache keyed on them stable. The included-turn set these bytes encode is chosen by the [session-context truncation](#session-context-truncation-bind_context-session) walk, which sums per-message counts from the [`TokenEstimator` seam](../pi-integration-contract/host-interfaces-services.md#pic-16); byte-identical input across runs and processes therefore additionally rests on the production estimator returning the same per-message count for an identical message — the [per-message-determinism precondition](../pi-integration-contract/host-interfaces-services.md#pic-16) pinned at that seam — since a count that varied on identical input would silently shift the included-turn boundary and the transcript bytes. The BNDR-7 reference renderings (BNDR-7a–BNDR-7j) below are therefore pinned as MUST-reproduce-exactly — the exact bytes loom's input-reproducibility contract depends on — rather than left as examples revisable for clarity. Alias-resolved binder models (per [binder-model parse rule](#binder-model-parse-rule)) add a further dependency: input-reproducibility yields a stable binder model only when the resolved concrete model stays stable across runs.

1. **Turn order is chronological.** The truncation walk runs newest-to-oldest, but the rendered transcript MUST be emitted in chronological (oldest-to-newest) order so the binder reads the conversation in the order it happened. An implementation that mirrors the walk direction directly produces a reversed transcript and is non-conforming.
2. **Turn delimiter.** Each included turn (the same turn boundary used by the walk above: a user message plus all subsequent assistant / toolResult / custom messages up to but not including the next user message) renders as a contiguous block. Successive turn blocks MUST be separated by exactly one blank line (a single `\n` between the trailing `\n` of one turn block and the leading byte of the next).
3. **Per-message line prefix.** Each message renders as one or more lines whose first line begins with a role tag drawn from the closed set `[user]`, `[assistant]`, `[tool]`, `[custom:<type>]` followed by `: ` (one U+003A then one U+0020). For the `[custom:<type>]` form, `<type>` is the `CustomMessage.customType` string verbatim, constrained to the transcript-safe class of [BNDR-9](#bndr-9) (the binder rejects any out-of-class `customType` before rendering). Continuation lines of the same message (additional content lines, the `[tool-call …]` siblings of an assistant message) carry no role-tag prefix.
4. **Per-variant body.**
   - **`user`** → the message's text content verbatim. The empty-content user message renders as `[user]: ` (the trailing U+0020 from rule 3 is preserved with no body), mirroring the *User-arguments line* convention in *System-prompt structure (normative)* item 5.
   - **`assistant`** → the concatenation of every `TextContent.text` in `AssistantMessage.content` array order, verbatim. `ThinkingContent` blocks MUST be omitted (they are not part of the conversation the binder is grounding against). Each `ToolCall` block in `content` renders as a sibling line of the form `[tool-call <name>(<args-json>)]` where `<name>` is `ToolCall.name` and `<args-json>` is the canonical no-whitespace JSON serialisation of `ToolCall.arguments` (the SDK field is `arguments`, not `args`) defined in [BNDR-8](#bndr-8); multiple `ToolCall` blocks render as multiple sibling lines in `content` array order, and the order of the merged text line relative to those sibling lines is pinned by [BNDR-8](#bndr-8). An assistant message with no `TextContent` but at least one `ToolCall` MUST still emit the `[assistant]: ` prefix line (with empty body) followed by the `[tool-call …]` lines, so the role boundary stays detectable; an implementation MUST NOT collapse to a bare `[tool-call …]` line with no owning role.
   - **`toolResult`** → renders under the `[tool]` role tag (chosen to keep the role vocabulary at three plus the `custom:` family — `[toolResult]` is not used). `ToolResultMessage.content` is a typed-block array; the body is the concatenation of its text-block content in array order, with any non-text block serialised by the canonical no-whitespace JSON serialisation defined in [BNDR-8](#bndr-8) and concatenated in the same array order.
   - **`custom`** → the `CustomMessage.content` rendered verbatim under the `[custom:<type>]` prefix: when `content` is a string it is emitted as-is; when it is a `(TextContent | ImageContent)[]` array the body is the concatenation of its `TextContent.text` in array order (`ImageContent` blocks contribute no transcript bytes). Custom messages with `display: false` are still surfaced in the transcript — consistent with the LLM-context behaviour pinned at [Pi Integration Contract](../pi-integration-contract.md) under *Custom-message channel persistence and LLM-context entry* (`convertToLlm` converts every `CustomMessage` to a `user`-role transcript entry on every subsequent provider call regardless of the `display` flag). The loom's own `loom-system-note` messages appearing in the caller transcript therefore render as `[custom:loom-system-note]: …`, which is correct: they are part of the conversational record the binder is grounding against.
5. **No sanitisation.** The newline-collapse / 120-code-point cap discipline from [System-note rendering](./defaulting-system-note-echo.md#system-note-rendering) does **not** apply to transcript rendering — that discipline is for one-line user-facing notes; the binder's transcript is a multi-line model input where preserving message structure is the point. Implementations MUST NOT over-apply the shared discipline here.

<a id="bndr-7"></a> **BNDR-7.** Reference renderings (normative; conforming implementations MUST reproduce these exactly). Each rendering below carries a stable page-local sub-letter anchor (`bndr-7a` … `bndr-7j`, assigned in block order) so a test, coverage-matrix leaf, or failure report can cite a single rendering without naming the others. These sub-letter anchors are page-local sub-obligation identifiers, not new REQ-IDs — the same page-local sub-letter anchor scheme that [GOV-23](../governance/anchor-scheme-and-retired.md#gov-23) describes for the `spec.md` Session Model sub-obligations. The `bndr-7` umbrella anchor and the BNDR-7 REQ-ID remain the load-bearing identifiers for inbound links that cite the renderings as a unit; a later rendering append takes the next free sub-letter and existing sub-letters are not renumbered. Each block below shows a scenario and its complete rendered transcript body — the bytes that follow the `Recent session context …:` opening line of the *Session-context block* up to (but not including) the terminating blank line that closes the block per [System-prompt structure (normative) item 6](./binder-bypass-and-envelope.md#system-prompt-structure-normative). One block ([BNDR-7i](#bndr-7i)) is the void-truncation case: it pins the *absence* of the entire Session-context block rather than a transcript body, and MUST NOT be read as an empty-body cell.

<a id="bndr-7a"></a> **BNDR-7a. Single-message user turn.** Session: one `user` message with text `hello`.

```text
[user]: hello
```

<a id="bndr-7b"></a> **BNDR-7b. User + assistant + tool-call + tool-result + assistant turn.** Session, in order: one `user` (text `What's the weather?`); one `assistant` (`content: [TextContent { text: "Let me check." }, ToolCall { name: "get_weather", arguments: { city: "Paris" } }]`); one `toolResult` (text `Sunny, 20°C`); one `assistant` (text `Sunny in Paris, 20°C.`).

```text
[user]: What's the weather?
[assistant]: Let me check.
[tool-call get_weather({"city":"Paris"})]
[tool]: Sunny, 20°C
[assistant]: Sunny in Paris, 20°C.
```

<a id="bndr-7c"></a> **BNDR-7c. Turn containing a `loom-system-note` custom message.** Session, in order: one `user` (text `/lookup foo`); one `custom` (`customType: "loom-system-note"`, `content: "loom /lookup: argument binding cancelled"`, `display: false`); one `user` (text `try again`). The first two messages form one turn; the third opens a second turn.

```text
[user]: /lookup foo
[custom:loom-system-note]: loom /lookup: argument binding cancelled

[user]: try again
```

<a id="bndr-7d"></a> **BNDR-7d. Empty-content user message edge case.** Session: one `user` whose text is the empty string. The line ends with the rule-3 trailing U+0020 (shown here as a literal trailing space) and a `\n`; the trailing space is part of the contract.

```text
[user]: 
```

<a id="bndr-7e"></a> **BNDR-7e. Assistant body interleaving with multi-key tool-call arguments.** Session, in order: one `user` (text `Plan my trip.`); one `assistant` whose `content` is, in array order, `ToolCall { name: "get_weather", arguments: { unit: "celsius", city: "Paris" } }`, `TextContent { text: "Checking now." }`, `ToolCall { name: "get_forecast", arguments: { region: { zone: "eu", lat: 48 }, days: [3, 1, 2] } }`. The merged text line is emitted first and the two tool-call lines follow in `content` array order per [BNDR-8](#bndr-8); each `arguments` object is serialised with keys in ascending Unicode code-point order at every nesting level (so `city` precedes `unit`, `days` precedes `region`, and `lat` precedes `zone`) while the `days` array order is preserved verbatim.

```text
[user]: Plan my trip.
[assistant]: Checking now.
[tool-call get_weather({"city":"Paris","unit":"celsius"})]
[tool-call get_forecast({"days":[3,1,2],"region":{"lat":48,"zone":"eu"}})]
```

<a id="bndr-7f"></a> **BNDR-7f. Assistant message with a `ToolCall` but no `TextContent`.** Session, in order: one `user` (text `Check weather.`); one `assistant` whose `content` is a single `ToolCall { name: "get_weather", arguments: { city: "Paris" } }` and no `TextContent`. The `[assistant]: ` line is emitted with an empty body and the rule-3 trailing U+0020 (shown here as a literal trailing space), followed by the `[tool-call …]` line.

```text
[user]: Check weather.
[assistant]: 
[tool-call get_weather({"city":"Paris"})]
```

<a id="bndr-7g"></a> **BNDR-7g. `toolResult` with mixed text and non-text content blocks.** Session, in order: one `user` (text `Run the report.`); one `toolResult` whose `content` typed-block array is, in array order, a text block (`Rows: `), then one non-text block whose `JSON.stringify` form (no whitespace) is `{"chartId":7}`, then a text block (` (rendered)`). The non-text block's internal shape is owned by the Pi-SDK tool-result type family (see [Pi Integration Contract — `SessionContext` and the `.messages` element shape](../pi-integration-contract/host-interfaces-core.md#sessioncontext-shape)); only its serialised form is reproduced here. The body concatenates the text content and the serialised non-text block in array order under the single `[tool]` line.

```text
[user]: Run the report.
[tool]: Rows: {"chartId":7} (rendered)
```

<a id="bndr-7h"></a> **BNDR-7h. `custom` message with a `(TextContent | ImageContent)[]` array body.** Session, in order: one `user` (text `Look.`); one `custom` (`customType: "extension-card"`) whose `content` is the array `[TextContent { text: "See: " }, ImageContent { … }, TextContent { text: "above." }]`. The body is the concatenation of the `TextContent.text` in array order; the `ImageContent` block contributes no transcript bytes.

```text
[user]: Look.
[custom:extension-card]: See: above.
```

<a id="bndr-7i"></a> **BNDR-7i. Void truncation result (zero included turns).** The [Session-context truncation](#session-context-truncation-bind_context-session) walk produced zero included turns (for example, a single oversized newest turn, or an empty session). No transcript is rendered: per [System-prompt structure (normative) item 6](./binder-bypass-and-envelope.md#system-prompt-structure-normative), when the walk produces zero included turns the entire Session-context block — opening line, body, and terminating blank line — is omitted. This oracle pins the *absence* of the block; an implementation that emits a `Recent session context` header with an empty body is non-conforming. There is no transcript-body fence for this case because there are no bytes to reproduce.

<a id="bndr-7j"></a> **BNDR-7j. `toolResult` with a multi-key non-text content block.** Session, in order: one `user` (text `Run the report.`); one `toolResult` whose `content` typed-block array is, in array order, a text block (`Rows: `), then one non-text block whose source insertion order is `chartId` then `caption` (so a bare `JSON.stringify` would emit `{"chartId":7,"caption":"Q3"}`), then a text block (` (rendered)`). Per the canonical no-whitespace JSON serialisation of [BNDR-8](#bndr-8) the non-text block's keys are emitted in ascending Unicode code-point order, so `caption` precedes `chartId`. As in [BNDR-7g](#bndr-7g), the non-text block's internal shape is owned by the Pi-SDK tool-result type family (see [Pi Integration Contract — `SessionContext` and the `.messages` element shape](../pi-integration-contract/host-interfaces-core.md#sessioncontext-shape)); only its serialised form is reproduced here.

```text
[user]: Run the report.
[tool]: Rows: {"caption":"Q3","chartId":7} (rendered)
```

<a id="bndr-8"></a> **BNDR-8.** Assistant-body byte determinism and canonical JSON serialisation. Within a single `assistant` message, the merged `[assistant]: <text>` line (the rule-4 text concatenation) MUST be emitted first, followed by every `[tool-call …]` sibling line in `content` array order, regardless of where the `ToolCall` and `TextContent` blocks appear relative to each other in `content`. The **canonical no-whitespace JSON serialisation** named by this rule is the renderer's single normative source for emitting JSON anywhere in the transcript — both the assistant `<args-json>` here and the `toolResult` non-text blocks of rule 4: the value is serialised with no whitespace, object keys MUST be emitted in ascending Unicode code-point order at every object nesting level, and array element order is preserved verbatim. `JSON.stringify` alone is not key-order-stable across SDK property-insertion orders, so a key-sorting canonicalisation step is required. On each `[tool-call …]` line, `<args-json>` is the canonical serialisation of `ToolCall.arguments` (the SDK field is `arguments`, not `args`). [BNDR-7e](#bndr-7e) reproduces the assistant-body pins and [BNDR-7j](#bndr-7j) reproduces the `toolResult` multi-key non-text block pin.

<a id="bndr-9"></a> **BNDR-9.** Transcript-safe `customType` precondition. A `CustomMessage.customType` is **transcript-safe** when it is non-empty and contains none of U+000A (`\n`), U+000D (`\r`), `]` (U+005D), or the two-byte sequence `: ` (U+003A U+0020) — the bytes whose appearance in the `[custom:<type>]` role tag (rule 3 above) would break the line, the closing `]`, or the `: ` role-tag separator and so violate [BNDR-7](#bndr-7)'s reproduce-exactly contract. Pi types `CustomMessage.customType` only as `string` (see [Pi Integration Contract — `CustomMessage`](../pi-integration-contract/host-interfaces-core.md#sessioncontext-shape)) and no upstream rule constrains its characters, so the binder MUST reject any included `CustomMessage` whose `customType` is not transcript-safe before rendering — aborting transcript construction for that invocation and emitting [`loom/runtime/custom-type-unsafe`](../diagnostics/code-registry-runtime.md#loomruntime-runtime-panics-runtime-defect-surface-and-delivery-failures) — rather than interpolate the unsafe value. The invocation-level outcome is failure: the affected slash/prompt invocation's argument binding does not proceed (there is no proceed-and-drop branch), consistent with the [`loom/runtime/custom-type-unsafe`](../diagnostics/code-registry-runtime.md#loomruntime-runtime-panics-runtime-defect-surface-and-delivery-failures) registry row, and the user-facing system note resolves through the [Failure-mode templates (normative)](./determinism-cancellation-failure.md#failure-mode-templates-normative) row for the custom-type-unsafe cause.
