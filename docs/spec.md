# pi-loom — Extension Specification

<a id="overview"></a>
## Overview

`pi-loom` is a [Pi Coding Agent](https://github.com/badlogic/pi-mono) extension that adds a domain-specific scripting language for prompts and agentic operations.

A `.loom` file interleaves code with literal text destined for the model. Loom evaluation appends turns to a conversation: the *caller's* current conversation in `prompt` mode, or a separate conversation in `subagent` mode. The state-isolation contract for subagent invocation — what the spawned session inherits from the loom's frontmatter, what is forwarded from the caller's `ExtensionCommandContext`, and what is *not* inherited from the caller (transcript, system prompt, ambient tool set, cancellation controller, caller's `params` and bindings) — is enumerated in [Pi Integration Contract — Subagent state-isolation matrix](./spec_topics/pi-integration-contract.md#subagent-state-isolation-matrix); the *callable set* concept that matrix references is defined in [Glossary — `callable set`](./spec_topics/glossary.md). Mode is selected per-loom by the required `mode:` frontmatter field — see [Parameters and Frontmatter](./spec_topics/frontmatter.md). On the success outcome, evaluation also produces a *final value* — the value of the loom's tail expression, or — if an explicit `return expr` executes first — the operand of that `return`, per [Function Definitions — Final value](./spec_topics/functions.md#final-value-language-definition) — consumed by `invoke` callers and propagated across the subagent boundary per [Invocation — Typed return](./spec_topics/invocation.md#typed-return). The loom language itself has no file-writing, network, or process-spawning primitive; effects of those kinds occur only through allowlisted tools, and the language's effect surface is owned by [Runtime Value Model — Effects](./spec_topics/runtime-value-model.md#effects).

<a id="terminal-outcomes-aggregator"></a>Loom evaluation produces one of three terminal outcomes: it succeeds (turns appended; final value available to programmatic callers), it fails (by returning `Err`, by panicking, or via a [query-terminating](./spec_topics/glossary.md#query-terminating) enforcement point of one of the runtime-class [hard ceilings](#hard-runtime-ceilings) below — query-terminating points route via panic or `Err`, with the per-arm classification *for ceiling #4* owned by [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./spec_topics/hard-ceilings.md#ceiling-4-table); the following hard-ceiling cases are excluded from the Failure-cause enumeration: (a) binder argument-binding failure — including ceiling #4's slash-load `params` arm cross-routed through ceiling #3 per [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./spec_topics/hard-ceilings.md#ceiling-4-table) (slash-load only; the `invoke(...)` `params` arm of ceiling #4 IS an evaluation Failure that surfaces as `Err(InvokeInfraError { cause: "validation", ... })` per [Invocation — Failures](./spec_topics/invocation.md)) — per the binder argument-binding failure item in [Errors and Results — Terminal outcomes](./spec_topics/errors-and-results.md#terminal-outcomes); see ceiling #3 in [Hard ceilings](#hard-runtime-ceilings), (b) ceiling #4's [in-loop](./spec_topics/glossary.md#in-loop) model-driven tool-call args row (which feeds a tool-error result back to the model rather than terminating the query)), or it is cancelled per [Cancellation](./spec_topics/cancellation.md). The cancellation signal is carried by the runtime-owned `loomAbort.signal` defined in [Pi Integration Contract — Cancellation source](./spec_topics/pi-integration-contract.md#cancellation-source); the runtime forwards Pi's per-handler `ctx.signal` (and the tool-exposed `signal` parameter, and parent-`invoke` signals) into `loomAbort` per the per-entry-point rules in [Cancellation — Forwarding into `loomAbort`](./spec_topics/cancellation.md) and [Pi Integration Contract — Cancellation source](./spec_topics/pi-integration-contract.md#cancellation-source). The closed enumeration of these three terminal outcomes — and the per-cause caller surface map (what an `invoke` parent sees, what a slash caller sees, what the operator observes per channel) — are owned by [Errors and Results — Terminal outcomes](./spec_topics/errors-and-results.md#terminal-outcomes); the *partial-append contract* (turns appended before the terminal event remain in the driven conversation; the runtime performs no implicit rollback; mid-stream user-visible streaming fragments are governed separately) is owned by [Errors and Results — Partial-append contract](./spec_topics/errors-and-results.md#partial-append-contract). Failures that occur before evaluation begins are owned, with their closed enumeration and per-cause routing rule, by [Errors and Results — Terminal outcomes](./spec_topics/errors-and-results.md#terminal-outcomes); they never become evaluation outcomes. The success / fail / cancelled trichotomy applies only once evaluation has begun.

<a id="file-extension-grammar"></a>A loom is stored in one of two file extensions that share a single grammar and type system. `.loom` files are invocable as slash commands (see [Slash-Command Invocation](./spec_topics/slash-invocation.md)); `.warp` files are library modules whose top level is restricted to the declaration forms enumerated in [Imports — Permitted top-level forms](./spec_topics/imports.md#permitted-top-level-forms) (currently five: `import`, `export`, `fn`, `schema`, `enum` — *orientation aggregator per [Governance — GOV-12](./spec_topics/governance.md)*). `.warp` files are never directly invoked. Path literals and discovery enforce these extensions per the rules in [Imports](./spec_topics/imports.md), [Invocation](./spec_topics/invocation.md), and [Discovery](./spec_topics/discovery.md); see those pages for the diagnostic codes that fire on each violation. See [Discovery — File-extension namespace](./spec_topics/discovery.md#file-extension-namespace) for the namespace-clearance note.

<a id="session-model"></a>*Session model.* A Pi extension instance is bound to exactly one active user session at a time. Pi fires `session_shutdown` (`SessionShutdownEvent` in `@mariozechner/pi-coding-agent`'s `dist/core/extensions/types.d.ts`) with `event.reason: "quit" | "reload" | "new" | "resume" | "fork"` (the closed normative set; the SDK type is the source of truth for both the payload path and the closed reason set, so a future SDK pin bump diff-audits against that interface). The runtime's handler runs the full fixed teardown sequence — including watcher closure — for every reason in the closed set, and the V1 position accepts that watchers may be closed on a reason (e.g. `"fork"`) that did not in fact tear the extension down (recovery is one `/reload` away). The fixed teardown sequence itself is owned by [Pi Integration Contract — Extension entry point, step 4](./spec_topics/pi-integration-contract.md). An `event.reason` outside that closed set (or one whose property access throws) routes through the full teardown sequence and emits one `loom/host/session-shutdown-reason-unknown` (W, runtime) diagnostic via the teardown-handler last-resort sink defined in [Pi Integration Contract — Extension entry point, step 4](./spec_topics/pi-integration-contract.md#diagnostic-emission-isolation) (the rule that pins `console.error` as the channel and wraps every teardown-handler emission in `try`/`catch` so a throw from the sink does not unwind the handler); the validation, fail-safe routing, and `details.event.reason` widening on `loom/runtime/cancelled-by-session-shutdown` are owned by [Pi Integration Contract — Extension entry point, step 4](./spec_topics/pi-integration-contract.md). Session-swap semantics for in-flight invocations — the abort-and-await sequence, the per-invocation operator-visibility surface, partial-append fate, and the `invoke`-parent observation rule — are owned by [Pi Integration Contract — Session-swap behaviour for in-flight invocations](./spec_topics/pi-integration-contract.md#session-shutdown-semantics). Concurrent loom invocations within a session carry **mode-qualified** isolation. *Cancellation* is always independent: every invocation — prompt mode and subagent mode alike — carries its own `AbortController` (`loomAbort`) per [Cancellation — Signal source](./spec_topics/cancellation.md) and is tracked as a distinct entry in an extension-scoped registry of in-flight invocations whose normative shape, lifecycle, and teardown semantics live at [Pi Integration Contract — `ActiveInvocationRegistry`](./spec_topics/pi-integration-contract.md#active-invocation-registry). *Transcript and tool-table isolation* are **subagent-mode-only** — a subagent-mode invocation runs its own private `AgentSession` (no shared transcript or `tools:` table) per [Implementation Notes — Per-invocation single-threaded execution](./spec_topics/implementation-notes.md), while a prompt-mode invocation drives the user session's transcript by definition. *Prompt-mode bodies execute strictly sequentially* within a single user session: at most one prompt-mode body — top-level slash dispatches and nested prompt → prompt `invoke(...)` calls alike — holds an open `pi.setActiveTools` snapshot/restore window at a time, supported by (i) Pi's per-session slash-handler serialisation pinned at [Pi Integration Contract — Tool-registration lifetime and visibility](./spec_topics/pi-integration-contract.md), (ii) the load-time rejection of prompt-mode `.loom` callees in another loom's `tools:` per [Parameters and Frontmatter — `tools`](./spec_topics/frontmatter.md#tools) (so the model cannot fan out parallel prompt-mode tool calls), and (iii) the rule, owned by [Invocation — Cross-mode semantics](./spec_topics/invocation.md), that `invoke(...)` to a prompt-mode callee suspends the parent's body until the child returns and that V1 exposes no parallel-`invoke` surface by which a parent could spawn sibling prompt-mode children concurrently. The three potential sources of in-session overlap therefore produce genuine concurrency **only between subagent-mode invocations**: parallel tool calls into the same `.loom` callable can target only subagent-mode callees (prompt-mode callees in `tools:` are rejected at load time per [Parameters and Frontmatter — `tools`](./spec_topics/frontmatter.md#tools)); sibling `invoke(...)` sites can overlap only when the children resolve to subagent mode (per (iii) above); and independent slash dispatches against the same session cannot overlap at all per (i). V1 concurrency disposition for in-flight invocations — no admission cap, no scheduler interposed above Pi's event loop — is owned by [Implementation Notes — No invocation cap](./spec_topics/implementation-notes.md#no-invocation-cap). Per-invocation budgets (binder retry budget per [Slash-Command Argument Binding — Failure modes](./spec_topics/binder.md), `tool_loop.max_rounds` per [Hard ceilings](#hard-runtime-ceilings), `invoke`-chain depth) are **scoped to a single invocation** and are not shared, pooled, or replenished across sibling invocations. Cancellation propagates downward only (parent → children) per [Cancellation — Propagation](./spec_topics/cancellation.md); a sibling invocation's cancellation does not affect its siblings. Concurrent *user sessions* in the same host process are out of scope for V1: the runtime presupposes a single active user session per extension instance and has not audited Pi's full extension API surface for a concurrent-sessions facet (see [Future Considerations — V1 non-goals](./spec_topics/future-considerations.md#v1-non-goals), "No concurrent user sessions in the same host process", for the presupposition framing and the build-time / editorial review surfaces that would catch a future Pi minor introducing one).

### Scope

This subsection pins five cross-cutting V1 dispositions that no single topic page enumerates as a unit. The bullets are *informative orientation*: each one forward-links the topic page that owns the normative contract, and the two no-seam V1 dispositions recorded here as scope bullets — Trust boundary and Source-language stability — are also indexed under [Future Considerations — V1 non-goals](./spec_topics/future-considerations.md#v1-non-goals); the editorial criterion applied when promoting a no-seam V1 disposition to a standalone Scope bullet (applied as orientation guidance, not a normative rule, and not in the GOV-12 lock-step scope) is whether the disposition is *load-bearing for understanding the V1 design at the orientation level* — i.e. whether it constrains how a reader interprets every other Scope bullet (Trust boundary defines V1's privilege model; Source-language stability defines V1's release-process invariant); other no-seam dispositions are recorded only under the [V1 non-goals](#v1-non-goals) aggregator below; the consolidated V1 non-goals list (cross-cutting dispositions V1 ships without a forward-compatibility seam) is owned by [Future Considerations — V1 non-goals](./spec_topics/future-considerations.md#v1-non-goals) and forward-linked from [V1 non-goals](#v1-non-goals) below. The aggregator-vs-source lock-step convention for keeping these bullets honest as topic pages evolve is owned by [Governance — GOV-12](./spec_topics/governance.md).

- **Trust boundary.** V1 looms execute inside the Pi extension-host process at full Node host-process privilege. V1 imposes no loom-level sandbox: filesystem, network, and Pi-API access available to loom code and to its declared callables are exactly those available to any Pi extension running in the same process — Pi's `ExtensionAPI` and `ExtensionContext` surfaces in `dist/core/extensions/types.d.ts` of `@mariozechner/pi-coding-agent ~0.72.1` (the V1 Pi-SDK pin owned by [Pi Integration Contract — Host prerequisites](./spec_topics/pi-integration-contract.md)) expose no per-extension privilege facet, and the resource-loader and tool-host surfaces grant capabilities at the host-process level; Pi exposes no per-extension privilege scoping the runtime can rely on as a security boundary. A future Pi minor that adds such a facet would surface at the build-time SDK surface-inventory assertion run by [Pi version bump procedure](./spec_topics/pi-integration-contract.md#pi-version-bump-procedure) before the spec contract drifts. The runtime itself interposes no additional access channels (sandbox, capability filter, mediated proxy) per [Pi Integration Contract — No additional access channels](./spec_topics/pi-integration-contract.md#no-extra-mediation). The loom's declared *callable set* (the entries listed under the `tools:` frontmatter field; default rule for absent or empty `tools:` is owned by [Parameters and Frontmatter — `tools`](./spec_topics/frontmatter.md#tools)) is a configuration knob over the *model's* reachable callable set, NOT a host-process sandbox: it is enforced mechanically by the per-mode wiring rule in [Pi Integration Contract — Tool-registration lifetime and visibility](./spec_topics/pi-integration-contract.md) (subagent mode: explicit `customTools` array (with the matching tool names listed in a `tools` allowlist) on `createAgentSession`; prompt mode: `pi.setActiveTools` snapshot/restore around each query). Any callable a loom declares exposes the full underlying capability of that callable to the loom's model; the runtime applies no privilege classification or filtering. `bash` and `read` are illustrative — the rule does not depend on a callable being deemed "privileged." Host-side denials of filesystem, network, or Pi-API access reach loom code through the tool that issued the request: clean denials (an `execute()` throw or an `isError: true` return) map to `Err(QueryError { kind: "code_tool", cause: "execution", ... })`, and the complete enumeration of observable `execute()` outcomes — including non-conforming return envelopes (routed off the `CodeToolError` surface to `loom/runtime/internal-error` with `details.kind = "tool-return-shape"`), non-settling Promises (cancellation is the V1 termination path), and post-cancel late settlements (discarded) — is owned by [Tool Calls — Failures](./spec_topics/tool-calls.md) and [Pi Integration Contract — Tool execution from loom code](./spec_topics/pi-integration-contract.md#tool-execution-from-loom-code); silent success on denial is forbidden. A per-loom capability model is **out of scope for V1** and is not anticipated by V1; introducing one would require a migration. See [Future Considerations — V1 non-goals](./spec_topics/future-considerations.md#v1-non-goals).

- <a id="source-language-stability"></a> **Source-language stability.** A `.loom` or `.warp` file that loads cleanly under V1.0 is expected to load under every V1.x release and to produce, for any given input, identical (a) return values, (b) ordered diagnostic-code sequences, and (c) `loom-system-note` content strings. Wall-clock timing, token counts, and log-line volume are explicitly excluded from this expectation. This is a release-process goal, not a normative obligation; the equivalence statement, the V1.0 review-vs-mechanical-gate posture, and the deferred-conformance-suite framing are owned by [Governance — GOV-15](./spec_topics/governance.md#gov-15) with the companion review-posture rule at [GOV-14](./spec_topics/governance.md#gov-14). Introducing a new hard runtime ceiling within V1.x — or retiring or relaxing an existing one — is a recognised carve-out to that equivalence per [GOV-15](./spec_topics/governance.md#gov-15); the in-scope input set (defined post-hoc over the inter-release diff), the carve-out's diagnostic-emissions scope (per [GOV-15 — Ceiling-set carve-out](./spec_topics/governance.md#ceiling-set-carve-out)), and the attribution test that distinguishes carve-out-eligible edits from collateral edits are all owned by [GOV-15](./spec_topics/governance.md#gov-15). The long-term migration framing (no major-version migration mechanism) is owned by [Future Considerations — V1 non-goals](./spec_topics/future-considerations.md#v1-non-goals). Migration across major versions is out of V1 scope.

- **Runtime observability.** *Operator*-facing runtime failure events are emitted on the Pi `loom-system-note` channel via the *always-log set* (see [Glossary](./spec_topics/glossary.md)) defined in [Pi Integration Contract — Runtime event channel](./spec_topics/pi-integration-contract.md). Diagnostics for parse / load / type / runtime-panic batches share the same channel under a disjoint `details` shape (see [Diagnostics](./spec_topics/diagnostics.md)). Aggregation, latency histograms, per-loom token reports, and a consumer-facing read API are deferred (see [Future Considerations — Richer runtime-event telemetry](./spec_topics/future-considerations.md)).

- **Forward-compatibility seams.** V1 reserves 13 typed/structural seams in the runtime to enable a fixed inventory of deferred extensions backward-compatibly; each seam is pinned with a `> **V1 seam — <name>.**` blockquote on its owning topic page, and the inventory and per-seam contracts are owned by [Future Considerations — Surface extensions](./spec_topics/future-considerations.md#surface-extensions-v1-leaves-a-seam). Items with no dedicated topic-page seam (those riding the frontmatter unknown-key policy as a generic carrier, plus the upstream Pi-API-contingent items) are tracked separately under [Future Considerations — Surface extensions without a dedicated topic-page seam](./spec_topics/future-considerations.md#surface-extensions-no-dedicated-seam) and are not counted in the 13. Per [Governance — GOV-12](./spec_topics/governance.md), the count and the source list move in lock-step.

- <a id="hard-runtime-ceilings"></a> **Hard ceilings.** *Orientation aggregator (per [Governance — GOV-12](./spec_topics/governance.md)).* The four ceilings below are the V1 set; the closure rests on the four non-existence claims `NOCEIL-1` … `NOCEIL-4` (no wall-clock, token, runtime-value memory, or additional frame-depth ceiling beyond the 32-level `invoke`-chain bound) at [Hard Runtime Ceilings — No additional V1 runtime ceiling applies](./spec_topics/hard-ceilings.md#no-additional-ceilings), which name each axis's `NOCEIL-N` claim and its documented fall-through (see [Hard Runtime Ceilings — Audit methodology](./spec_topics/hard-ceilings.md#audit-methodology) for the audit-methodology paragraph governing future ceiling-set edits). Each has a distinct, observable failure surface — addressed to at least one of *loom code*, *the model*, or *the operator* — and no ceiling is unobservable to all three. (Ceiling #4's model-driven tool-call args row is observable to the model and, if the loop later exhausts, to the operator via ceiling #2; see the per-boundary table and reconciliation paragraph at [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./spec_topics/hard-ceilings.md#ceiling-4-table).) Ceilings #1 and #2 are uniformly *runtime-class* — they fire during evaluation and surface as a panic or recoverable `Err` per the `query-terminating` glossary entry's terminal-outcome resolution (panic-class breaches unconditionally reach the success / fail / cancelled trichotomy's *fail* arm; `Err`-class breaches reach *fail* only when the `Err` is unhandled — propagated via `?` or returned, not consumed by a caller `match` and not discarded via `let _ = …` (or the equivalent `void`-tail-expression form)). Ceiling #4 has five enforcement points (per CIO-3 / *Five-site list co-edit obligation* on `hard-ceilings.md`); its `params` enforcement point's slash-load arm cross-routes through ceiling #3's load-time classification while its `invoke(...)` arm is [query-terminating](./spec_topics/glossary.md#query-terminating), so the per-arm classification owned by [the per-boundary table](./spec_topics/hard-ceilings.md#ceiling-4-table) partitions ceiling #4's enforcement-point arms across three routing classes — query-terminating runtime arms (surfacing as recoverable `Err` only — no ceiling-#4 query-terminating arm panics; per-ceiling routing-class and terminal-outcome resolution live in the `query-terminating` glossary entry), the [in-loop](./spec_topics/glossary.md#in-loop) model-driven tool-call args row, and the slash-load `params` arm that cross-routes through ceiling #3. Ceiling #3 (the binder per-class retry budget) is *load-time* — it fires at slash-invocation load time before evaluation begins, is governed by the pre-evaluation failure list in the opening paragraph above, and is NOT an evaluation outcome. The cross-ceiling content that no single per-ceiling page owns — the binder per-class retry budget's HC3-a … HC3-e sub-obligations, ceiling #4's per-boundary destination/surface table, the fixed evaluation order between ceilings (CIO-1 … CIO-6), the `masked` field's per-site reachable domain and wire-location split, the four NOCEIL-1 … NOCEIL-4 non-existence claims, the worked consequences of the interaction order, and the four-axis Audit methodology that closes the NOCEIL-1 … NOCEIL-4 set — is owned by [Hard Runtime Ceilings](./spec_topics/hard-ceilings.md). The four ceilings, with the per-ceiling routing-class one-liner each, are:

  1. **`invoke`-chain nesting depth 32** *(runtime)* — routing class: **runtime panic**. Surfaces as `loom/runtime/invoke-depth-exceeded` per [Invocation — Invocation depth bound](./spec_topics/invocation.md), routed per [Errors and Results — Runtime panics](./spec_topics/errors-and-results.md). This is the only one of the four whose breach lands on the success / fail / cancelled trichotomy's `fail` arm via the panic path.
  2. **`tool_loop.max_rounds`** per query, default 25 *(runtime)* — routing class: **recoverable `Err`**. Surfaces as `Err(QueryError { kind: "tool_loop_exhausted", … })` per [Parameters and Frontmatter — `tool_loop`](./spec_topics/frontmatter.md) and [Errors and Results — `ToolLoopExhaustedError`](./spec_topics/errors-and-results.md#queryerror-variants); reaches loom code as a value and is `match`-handleable.
  3. <a id="hard-ceiling-3"></a>**Binder per-class retry budget** *(load-time; not on the trichotomy)* — routing class: **load-time system note**. The loom does not start; the operator-facing note is rendered from the failure-mode template matching the *most recent* failure's class — see [Slash-Command Argument Binding — Failure-mode templates](./spec_topics/binder.md#failure-mode-templates-normative). Sub-obligations HC3-a … HC3-e (per-class retry caps, the worst-case-three-call sum, and the most-recent-failure template-selection rule) are owned by [Hard Runtime Ceilings — Per-class retry budget (ceiling #3)](./spec_topics/hard-ceilings.md#hc3); the classification rule that maps an observed binder-call outcome onto a retry class is the provider-error classifier in [Slash-Command Argument Binding — Failure-class taxonomy](./spec_topics/binder.md#failure-class-taxonomy). Not an evaluation outcome — nothing reaches loom code, no `Result` value is observable. Governed by the load-time-failure carve-out under [Errors and Results — Terminal outcomes](./spec_topics/errors-and-results.md#terminal-outcomes).
  4. **JSON-document depth 5** against typed-query response / model-driven tool-call args / code-driven tool-call args / `params` validation / `invoke<T>` return value *(runtime)* — routing class: **boundary-dependent**. Every breach surfaces with `schema_keyword: "maxDepth"` and the canonical message `"JSON document depth exceeds 5"` per [Schema Subset — Depth Enforcement](./spec_topics/schema-subset.md); the per-boundary destination/surface table (five enforcement points) and the cross-ceiling slash-load handoff are owned by [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./spec_topics/hard-ceilings.md#ceiling-4-table).

### V1 non-goals

*Orientation aggregator (per [Governance — GOV-12](./spec_topics/governance.md)).* The consolidated V1 non-goals list — cross-cutting V1 scope decisions where V1 ships without a forward-compatibility seam (a future revision adding the disposition is not anticipated by V1 and will require a migration) — is owned by [Future Considerations — V1 non-goals](./spec_topics/future-considerations.md#v1-non-goals); the seven items, separated by `;` so that the fan-out item's inner punctuation (commas and em-dashes) is visually distinguishable from the inter-item boundary, are: no per-loom sandbox or capability model; no formal source-language migration mechanism for major-version transitions; no non-Node JavaScript host support; no concurrent user sessions in the same host process; no parallel-`invoke` surface; no parallel fan-out *to* prompt-mode `.loom` callees (a prompt-mode `.loom` callable cannot appear in another loom's `tools:` table at all — rejected at load time per [Parameters and Frontmatter — `tools`](./spec_topics/frontmatter.md#tools) — so the model never sees one to fan out across, while Pi-host tools and subagent-mode `.loom` callees remain present in tool tables and may still be fanned out per [Tool Calls — Concurrency](./spec_topics/tool-calls.md#concurrency)); no admission cap on in-flight loom invocations.

The seam-vs-non-goal distinction is stated in category 4 of the intro list on [Future Considerations](./spec_topics/future-considerations.md); the canonical surface-extension inventory is owned by [Future Considerations — Surface extensions (V1 leaves a seam)](./spec_topics/future-considerations.md#surface-extensions-v1-leaves-a-seam) (e.g. per-call timeouts on queries / tool calls / invokes, the per-query overrides cascade). The aggregator-source page for this list (in GOV-12's *single-source-page* sense) is [Future Considerations — V1 non-goals](./spec_topics/future-considerations.md#v1-non-goals); the normative owners of each underlying disposition are a separate concept and remain on their topic pages; the [Prerequisites](#prerequisites) and [Scope](#scope) subsections and the [Session model](#session-model) paragraph above are themselves informative orientation aggregators, each forward-linking the topic-page sections it covers (per [Governance — GOV-12](./spec_topics/governance.md)), and each bullet's `Recorded at:` line in the [Future Considerations — V1 non-goals](./spec_topics/future-considerations.md#v1-non-goals) list cites the topic-page owner directly alongside any `spec.md` orientation cross-link.
