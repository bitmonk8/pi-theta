# pi-loom — Extension Specification

<a id="overview"></a>
## Overview

`pi-loom` is a [Pi Coding Agent](https://github.com/earendil-works/pi-mono) extension that adds a domain-specific scripting language for prompts and agentic operations.

A `.loom` file interleaves code with literal text destined for the model. Loom evaluation appends turns to a conversation: the *caller's* current conversation in `prompt` mode, or a separate conversation in `subagent` mode. The state-isolation contract for subagent invocation is enumerated in [Pi Integration Contract — Subagent state-isolation matrix](./spec_topics/pi-integration-contract.md#subagent-state-isolation-matrix); the *callable set* concept that matrix references is defined in [Glossary — `callable set`](./spec_topics/glossary.md). Mode is selected per-loom by the required `mode:` frontmatter field — see [Parameters and Frontmatter](./spec_topics/frontmatter.md). On the success outcome, evaluation also produces a *final value* — the value of the loom's tail expression, or — if an explicit `return expr` executes first — the operand of that `return`, per [Function Definitions — Final value](./spec_topics/functions.md#final-value-language-definition) — consumed by `invoke` callers and propagated across the subagent boundary per [Invocation — Typed return](./spec_topics/invocation.md#typed-return). The loom language itself has no file-writing, network, or process-spawning primitive; effects of those kinds occur only through allowlisted tools, and the language's effect surface is owned by [Runtime Value Model — Effects](./spec_topics/runtime-value-model.md#effects).

<a id="terminal-outcomes-aggregator"></a>Loom evaluation produces one of three terminal outcomes: it succeeds (turns appended; final value available to programmatic callers), it fails (by returning `Err`, by panicking, or via a [query-terminating](./spec_topics/glossary.md#query-terminating) enforcement point of one of the runtime-class [hard ceilings](#hard-runtime-ceilings) below — query-terminating points route via panic or `Err`, with the per-arm classification *for ceiling #4* owned by [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./spec_topics/hard-ceilings.md#ceiling-4-table); the following hard-ceiling cases are excluded from the Failure-cause enumeration: (a) binder argument-binding failure — including ceiling #4's slash-load `params` arm cross-routed through ceiling #3 per [Hard Runtime Ceilings — Per-boundary destination/surface table (ceiling #4)](./spec_topics/hard-ceilings.md#ceiling-4-table) (slash-load only; the `invoke(...)` `params` arm of ceiling #4 IS an evaluation Failure that surfaces as `Err(InvokeInfraError { cause: "validation", ... })` per [Invocation — Failures](./spec_topics/invocation.md)) — per the binder argument-binding failure item in [Errors and Results — Terminal outcomes](./spec_topics/errors-and-results.md#terminal-outcomes); see ceiling #3 in [Hard ceilings](#hard-runtime-ceilings), (b) ceiling #4's [in-loop](./spec_topics/glossary.md#in-loop) model-driven tool-call args row (which feeds a tool-error result back to the model rather than terminating the query)), or it is cancelled per [Cancellation](./spec_topics/cancellation.md). The cancellation signal is carried by the runtime-owned `loomAbort.signal` defined in [Pi Integration Contract — Cancellation source](./spec_topics/pi-integration-contract.md#cancellation-source); the runtime forwards Pi's per-handler `ctx.signal` (and the tool-exposed `signal` parameter, and parent-`invoke` signals) into `loomAbort` per the per-entry-point rules in [Cancellation — Forwarding into `loomAbort`](./spec_topics/cancellation.md) and [Pi Integration Contract — Cancellation source](./spec_topics/pi-integration-contract.md#cancellation-source). The closed enumeration of these three terminal outcomes — and the per-cause caller surface map (what an `invoke` parent sees, what a slash caller sees, what the operator (per [Glossary — operator](./spec_topics/glossary.md#operator)) observes per channel) — are owned by [Errors and Results — Terminal outcomes](./spec_topics/errors-and-results.md#terminal-outcomes); the *partial-append contract* (turns appended before the terminal event remain in the driven conversation; the runtime performs no implicit rollback; mid-stream user-visible streaming fragments are governed separately) is owned by [Errors and Results — Partial-append contract](./spec_topics/errors-and-results.md#partial-append-contract). Failures that occur before evaluation begins are owned, with their closed enumeration and per-cause routing rule, by [Errors and Results — Terminal outcomes](./spec_topics/errors-and-results.md#terminal-outcomes); they never become evaluation outcomes. The success / fail / cancelled trichotomy applies only once evaluation has begun.

<a id="file-extension-grammar"></a>A loom is stored in one of two file extensions that share a single grammar and type system. `.loom` files are invocable as slash commands (see [Slash-Command Invocation](./spec_topics/slash-invocation.md)); `.warp` files are library modules whose top level is restricted to the declaration forms enumerated in [Imports — Permitted top-level forms](./spec_topics/imports.md#permitted-top-level-forms) (currently five: `import`, `export`, `fn`, `schema`, `enum` — *orientation aggregator per [Governance — GOV-12](./spec_topics/governance.md)*). `.warp` files are never directly invoked. Path literals and discovery enforce these extensions per the rules in [Imports](./spec_topics/imports.md), [Invocation](./spec_topics/invocation.md), and [Discovery](./spec_topics/discovery.md); see those pages for the diagnostic codes that fire on each violation. See [Discovery — File-extension namespace](./spec_topics/discovery.md#file-extension-namespace) for the namespace-clearance note.

### Scope

This subsection pins five cross-cutting loom 1.0 dispositions that no single topic page enumerates as a unit. The bullets are *informative orientation*: each one forward-links the topic page that owns the normative contract, and the two no-seam loom 1.0 dispositions recorded here as scope bullets — Trust boundary and Source-language stability — are also indexed under [Future Considerations — loom 1.0 non-goals](./spec_topics/future-considerations.md#loom-1-0-non-goals); the consolidated loom 1.0 non-goals list (cross-cutting dispositions loom 1.0 ships without a forward-compatibility seam) is owned by [Future Considerations — loom 1.0 non-goals](./spec_topics/future-considerations.md#loom-1-0-non-goals) and forward-linked from [loom 1.0 non-goals](#loom-1-0-non-goals-aggregator) below. The aggregator-vs-source lock-step convention for keeping these bullets honest as topic pages evolve is owned by [Governance — GOV-12](./spec_topics/governance.md).

- **Trust boundary.** loom 1.0 looms execute inside the Pi extension-host process at full Node host-process privilege. loom 1.0 imposes no loom-level sandbox: filesystem, network, and Pi-API access available to loom code and to its declared callables are exactly those available to any Pi extension running in the same process — Pi's `ExtensionAPI` and `ExtensionContext` surfaces (declared in `@earendil-works/pi-coding-agent`'s `dist/core/extensions/types.d.ts` across the release range pinned by [Pi Integration Contract — Host prerequisites — Pi SDK pin](./spec_topics/pi-integration-contract.md#pi-sdk-pin)) expose no per-extension privilege facet, and the resource-loader and tool-host surfaces grant capabilities at the host-process level; Pi exposes no per-extension privilege scoping the runtime can rely on as a security boundary. A future Pi minor that adds such a facet would surface at the build-time SDK surface-inventory assertion run by [Pi version bump procedure](./spec_topics/pi-integration-contract.md#pi-version-bump-procedure) before the spec contract drifts. The runtime itself interposes no additional access channels (sandbox, capability filter, mediated proxy) per [Pi Integration Contract — No additional access channels](./spec_topics/pi-integration-contract.md#no-extra-mediation). The loom's declared *callable set* (the entries listed under the `tools:` frontmatter field; default rule for absent or empty `tools:` is owned by [Parameters and Frontmatter — `tools`](./spec_topics/frontmatter.md#tools)) is a configuration knob over the *model's* reachable callable set, NOT a host-process sandbox: it is enforced mechanically by the per-mode wiring rule owned by [Pi Integration Contract — Tool-registration lifetime and visibility](./spec_topics/pi-integration-contract.md#tool-registration-lifetime-and-visibility), which owns both the per-mode composition of that set and the SDK-call mechanism. Any callable a loom declares exposes the full underlying capability of that callable to the loom's model; the runtime applies no privilege classification or filtering. `bash` and `read` are illustrative — the rule does not depend on a callable being deemed "privileged." The host-side denial-surface rule — how denials of filesystem, network, or Pi-API access reach loom code through the tool that issued the request, and that silent success on denial is forbidden — is owned by [Pi Integration Contract — No additional access channels](./spec_topics/pi-integration-contract.md#no-extra-mediation); the complete enumeration of observable `execute()` outcomes is owned by [Tool Calls — Failures](./spec_topics/tool-calls.md) and [Pi Integration Contract — Tool execution from loom code](./spec_topics/pi-integration-contract.md#tool-execution-from-loom-code). A per-loom capability model is **out of scope for loom 1.0**; the disposition is owned by the **No per-loom sandbox or capability model** bullet under [Future Considerations — loom 1.0 non-goals](./spec_topics/future-considerations.md#loom-1-0-non-goals).

- <a id="source-language-stability"></a> **Source-language stability.** A `.loom` or `.warp` file that [loads cleanly](./spec_topics/governance.md#gov-15-loads-cleanly) under loom 1.0.0 is expected to load under every loom 1.x release and to produce, for any given input, identical (a) return values, (b) ordered diagnostic-code sequences, and (c) equivalent `loom-system-note` content strings. Two content strings are equivalent under observable (c) when they are byte-identical — i.e. equal as UTF-8 byte sequences, the encoding substrate pinned by [Lexical — Encoding](./spec_topics/lexical.md) — after normalising the placeholder sub-fields whose rendered value is permitted to vary per invocation or per run — the per-invocation identifiers (e.g. `<invocation-id>` / `<uuid>`), the wall-clock-derived values (e.g. `<ms>`), and the category-8 host-derived freeform tails — as classified by the placeholder-rendering categories in [Diagnostics — Placeholder rendering](./spec_topics/diagnostics.md#placeholder-rendering-normative); the fixed (non-variable) placeholder renderings and the surrounding literal template bytes are themselves byte-identical. Observable (c)'s equivalence relation, not the exclusion list, governs variation that is embedded as a placeholder sub-field inside a content string. Wall-clock timing, token counts, and log-line volume remain excluded from this expectation as independent observables in their own right — distinct from the wall-clock-derived sub-fields that observable (c) normalises within a content string. This is a release-process goal, not a normative obligation; the equivalence statement, the loom 1.0 review-vs-mechanical-gate posture, and the deferred-conformance-suite framing are owned by [Governance — GOV-15](./spec_topics/governance.md#gov-15) with the companion review-posture rule at [GOV-14](./spec_topics/governance.md#gov-14). Introducing a new hard runtime ceiling within loom 1.x — or retiring or relaxing an existing one — is a recognised carve-out to that equivalence per [GOV-15](./spec_topics/governance.md#gov-15); the in-scope input set (defined post-hoc over the inter-release diff), the carve-out's diagnostic-emissions scope (per [GOV-15 — Ceiling-set carve-out](./spec_topics/governance.md#ceiling-set-carve-out)), and the attribution test that distinguishes carve-out-eligible edits from collateral edits are all owned by [GOV-15](./spec_topics/governance.md#gov-15). The long-term migration framing (no major-version migration mechanism) is owned by [Future Considerations — loom 1.0 non-goals](./spec_topics/future-considerations.md#loom-1-0-non-goals). Migration across major versions is out of loom 1.0 scope.

- **Runtime observability.** *Operator*-facing runtime failure events surface on the Pi `loom-system-note` channel. The emission contract — the *always-log set* (see [Glossary](./spec_topics/glossary.md)), the success-side null-policy, and the per-mode operator surfaces — is owned by [Pi Integration Contract — Runtime event channel](./spec_topics/pi-integration-contract.md), with the success-side rule at [Success-side null-policy](./spec_topics/pi-integration-contract.md#success-side-null-policy) and the per-mode operator surface at [Slash-Command Invocation — Once a loom is invoked](./spec_topics/slash-invocation.md); the parse / load / type / runtime-panic diagnostic batches that share the channel are owned by [Diagnostics](./spec_topics/diagnostics.md). Deferred telemetry — aggregation, latency histograms, per-loom token reports, and a consumer-facing read API — is recorded under [Future Considerations — Richer runtime-event telemetry](./spec_topics/future-considerations.md).

- **Forward-compatibility seams.** loom 1.0 reserves 12 typed/structural seams in the runtime to enable a fixed inventory of deferred extensions backward-compatibly; each seam is pinned with a `> **loom 1.0 seam — <name>.**` blockquote on its owning topic page, and the inventory and per-seam contracts are owned by [Future Considerations — Surface extensions](./spec_topics/future-considerations.md#surface-extensions-v1-leaves-a-seam). Items not inventoried under that *Surface extensions* sub-bucket are tracked separately and not counted in the 12: those riding the frontmatter unknown-key policy as a generic carrier plus the upstream Pi-API-contingent items live under [Future Considerations — Surface extensions without a dedicated topic-page seam](./spec_topics/future-considerations.md#surface-extensions-no-dedicated-seam), and the item gated on an open design decision lives under [Future Considerations — Surface extensions gated on an open design decision](./spec_topics/future-considerations.md#surface-extensions-open-design); in each sub-bucket a defensive seam blockquote on an item (Pi-owned subagents, automatic context escalation) is excluded from the count on the same terms. Per [Governance — GOV-12](./spec_topics/governance.md), the count and the source list move in lock-step.

### V1 non-goals

*Orientation aggregator (per [Governance — GOV-12](./spec_topics/governance.md)).* The consolidated loom 1.0 non-goals list — cross-cutting loom 1.0 scope decisions where loom 1.0 ships without a forward-compatibility seam (a future revision adding the disposition is not anticipated by loom 1.0 and will require a migration) — is owned by [Future Considerations — loom 1.0 non-goals](./spec_topics/future-considerations.md#loom-1-0-non-goals); the eight items, separated by `;` so that items whose inner punctuation includes commas and em-dashes remain visually distinguishable from the inter-item boundary, are: no per-loom sandbox or capability model; no formal source-language migration mechanism for major-version transitions; no non-Node JavaScript host support; no concurrent user sessions in the same host process; no reliance on a Pi extension-host stdio-capture facet — teardown-time `console.error` diagnostics are treated as best-effort, per [Future Considerations — No reliance on a Pi extension-host stdio-capture facet](./spec_topics/future-considerations.md#pi-stdio-capture-facet); no parallel-`invoke` surface; no parallel fan-out *to* prompt-mode `.loom` callees (a prompt-mode `.loom` callable cannot appear in another loom's `tools:` table at all — rejected at load time per [Parameters and Frontmatter — `tools`](./spec_topics/frontmatter.md#tools) — so the model never sees one to fan out across, while Pi-host tools and subagent-mode `.loom` callees remain present in tool tables and may still be fanned out per [Tool Calls — Concurrency](./spec_topics/tool-calls.md#concurrency)); no admission cap on in-flight loom invocations.

The seam-vs-non-goal distinction is stated in category 4 of the intro list on [Future Considerations](./spec_topics/future-considerations.md); the canonical surface-extension inventory is owned by [Future Considerations — Surface extensions (loom 1.0 leaves a seam)](./spec_topics/future-considerations.md#surface-extensions-v1-leaves-a-seam) (e.g. per-call timeouts on queries / tool calls / invokes, the per-query overrides cascade).

## Session Model

<a id="session-model"></a>The runtime's session-level contract decomposes into the eight sub-obligations below; each carries a stable page-local anchor (`sm-1-…` … `sm-8-…`) so inbound citations can pin a single sub-obligation without the others. The umbrella `session-model` anchor is preserved for orientation-level inbound links. This section is **informative orientation** per [Governance — GOV-12](./spec_topics/governance.md): every SM-N obligation's normative contract is owned by a topic page that the corresponding entry forward-links — including [SM-7c](#sm-7c-prompt-mode-sequential-execution)'s prompt-mode sequential-execution conclusion, owned by [Pi Integration Contract — PIC-2](./spec_topics/pi-integration-contract.md#pic-2), and [SM-8](#sm-8-per-invocation-budget-non-sharing)'s per-invocation budget non-sharing, owned per budget by the topic pages SM-8 forward-links.

<!-- Anchor-scheme stability and orientation-vs-obligation citation rules for the `sm-N-…` anchors relocated to [Governance — GOV-23](./spec_topics/governance.md#gov-23). -->

- <a id="sm-1-single-active-session-binding"></a> **SM-1 — Single-active-session binding.** A Pi extension instance is bound to exactly one active user session at a time per [Pi Integration Contract — Session-binding contract](./spec_topics/pi-integration-contract.md#session-binding-contract). Concurrent *user sessions* in the same host process are out of scope for loom 1.0 — the runtime *presupposes* a single active user session per extension instance rather than having audited Pi's full extension API surface for a concurrent-sessions facet, and the full presupposition framing and the build-time / editorial review surfaces that would catch a future Pi minor introducing such a facet are owned by [Future Considerations — loom 1.0 non-goals](./spec_topics/future-considerations.md#loom-1-0-non-goals), "No concurrent user sessions in the same host process".

- <a id="sm-2-closed-shutdown-reason-set"></a> **SM-2 — Closed `session_shutdown` reason set.** Pi fires `session_shutdown` (`SessionShutdownEvent` in `@earendil-works/pi-coding-agent`'s `dist/core/extensions/types.d.ts`) with `event.reason: "quit" | "reload" | "new" | "resume" | "fork"` (the closed normative set; the SDK type is the source of truth for both the payload path and the closed reason set, so a future SDK pin bump diff-audits against that interface). An `event.reason` outside that closed set (or one whose property access throws) routes through the full teardown sequence and emits one `loom/host/session-shutdown-reason-unknown` (W, runtime) diagnostic via the teardown-handler last-resort sink defined in [Pi Integration Contract — Extension entry point — Diagnostic-emission isolation](./spec_topics/pi-integration-contract.md#diagnostic-emission-isolation) — the sink's two sub-obligations, both owned canonically by that linked Diagnostic-emission isolation section, are: **(i) Pi-side stdio visibility** — a deliberate non-claim, not an independently-verifiable obligation, defined at [Pi Integration Contract — Diagnostic-emission isolation — Pi-side stdio visibility](./spec_topics/pi-integration-contract.md#pic-emission-visibility); and **(ii) handler isolation** — the independently-verifiable swallow obligation, defined at [Pi Integration Contract — Diagnostic-emission isolation — emission swallow](./spec_topics/pi-integration-contract.md#pic-emission-swallow); the validation, fail-safe routing, and `details.event.reason` widening on `loom/runtime/cancelled-by-session-shutdown` are owned by [Pi Integration Contract — Extension entry point, step 4](./spec_topics/pi-integration-contract.md).

- <a id="sm-3-session-shutdown-handler"></a> **SM-3 — Session-shutdown handler.** SM-3 groups the two handler-internal obligations that run inside the `session_shutdown` handler — the per-reason fixed teardown sequence (SM-3a) and the PIC-owned session-swap semantics for in-flight invocations (SM-3b, whose contract is owned by PIC rather than by the loom runtime's handler) — each carrying a stable page-local anchor (`sm-3a-…` / `sm-3b-…`) so inbound citations can pin a single sub-obligation without the other. The umbrella `sm-3-session-shutdown-handler` anchor is preserved for orientation-level inbound links.

  - <a id="sm-3a-teardown-sequence"></a> **SM-3a — Fixed teardown sequence.** The runtime's handler runs the full fixed teardown sequence — including watcher closure in sub-step 4 — for every reason in the closed set. The fixed teardown sequence itself is owned by [Pi Integration Contract — Extension entry point, step 4](./spec_topics/pi-integration-contract.md).

  - <a id="sm-3b-session-swap-semantics"></a> **SM-3b — Session-swap semantics for in-flight invocations.** Session-swap semantics for in-flight invocations — the abort-and-await sequence, the per-invocation operator-visibility surface, partial-append fate, and the `invoke`-parent observation rule — are owned by [Pi Integration Contract — Session-swap behaviour for in-flight invocations](./spec_topics/pi-integration-contract.md#session-shutdown-semantics).

- <a id="sm-4-session-only-degraded-state"></a> **SM-4 — Post-teardown degraded state for the session-only reasons.** On the *session-only reasons* (`event.reason ∈ {"new", "resume", "fork"}`) the runtime remains live afterwards in a deterministically degraded state until operator-issued `/reload`. The degraded-state contract is owned by [Pi Integration Contract — `reason: "new" | "resume" | "fork"` edge-case bullet](./spec_topics/pi-integration-contract.md#session-only-reason-degraded-state).

- <a id="sm-5-tag-transition-predicate"></a> **SM-5 — Tag-transition predicate scope.** PIC's **tag-transition predicate** covers the [session-only reasons](#sm-4-session-only-degraded-state) directly *plus* the snapshot-failure and unknown-reason-with-successful-snapshot paths. The post-sub-step-4 `LoomRegistry` drain-state transition and the `loom /<name>: extension degraded; /reload to recover` system note that fronts subsequent slash invocations (with editor-driven `.loom` and settings edits silently no-opping until recovery) follow the tag-transition predicate. The predicate definition and the per-path scoping for tag-transition and slash-note coverage are owned by [Pi Integration Contract — `reason: "new" | "resume" | "fork"` edge-case bullet](./spec_topics/pi-integration-contract.md#session-only-reason-degraded-state).

- <a id="sm-6-diagnostic-emission-predicate"></a> **SM-6 — Diagnostic-emission predicate scope.** PIC's **diagnostic-emission predicate** — narrower than [SM-5's tag-transition predicate](#sm-5-tag-transition-predicate) — is scoped to the [session-only reasons](#sm-4-session-only-degraded-state) AND a successful snapshot read of the SDK pinned-constant; the snapshot-failure path is excluded by the snapshot-success conjunct. The new `loom/host/session-shutdown-runtime-degraded` (W, runtime) diagnostic follows the diagnostic-emission predicate (the snapshot-failure and unknown-reason paths cover their own diagnostic-channel observability via `loom/host/session-shutdown-pinned-constant-unreadable` and `loom/host/session-shutdown-reason-unknown` rather than a co-emitted `runtime-degraded` row). The predicate definition and the per-path scoping for emission coverage are owned by [Pi Integration Contract — `reason: "new" | "resume" | "fork"` edge-case bullet](./spec_topics/pi-integration-contract.md#session-only-reason-degraded-state).

- <a id="sm-7-mode-qualified-concurrency"></a> **SM-7 — Mode-qualified isolation contract.** Concurrent loom invocations within a session carry **mode-qualified** isolation; the contract decomposes into the five sub-obligations below, each with a stable page-local anchor (`sm-7a-…` … `sm-7e-…`) so inbound citations can pin a single sub-obligation without the others. The umbrella `sm-7-mode-qualified-concurrency` anchor is preserved for orientation-level inbound links.

  - <a id="sm-7a-cancellation-independence"></a> **SM-7a — Cancellation independence across modes.** *Cancellation* is always independent: every invocation — prompt mode and subagent mode alike — carries its own `AbortController` (`loomAbort`) per [Cancellation — Signal source](./spec_topics/cancellation.md) and is tracked as a distinct entry in an extension-scoped registry of in-flight invocations whose normative shape, lifecycle, and teardown semantics live at [Pi Integration Contract — `ActiveInvocationRegistry`](./spec_topics/pi-integration-contract.md#active-invocation-registry).

  - <a id="sm-7b-subagent-mode-isolation"></a> **SM-7b — Subagent-mode transcript and tool-table isolation.** *Transcript and tool-table isolation* are **subagent-mode-only** — a subagent-mode invocation runs its own private `AgentSession` (no shared transcript or `tools:` table) per [Pi Integration Contract — Concurrent-invocation isolation and concurrency disposition](./spec_topics/pi-integration-contract.md#concurrent-invocation-isolation), while a prompt-mode invocation drives the user session's transcript by definition.

  - <a id="sm-7c-prompt-mode-sequential-execution"></a> **SM-7c — Prompt-mode sequential execution.** *Prompt-mode bodies execute strictly sequentially* within a single user session — at most one prompt-mode body holds an open `pi.setActiveTools` snapshot/restore window at a time — is owned by [Pi Integration Contract — PIC-2 *Prompt-mode sequential execution*](./spec_topics/pi-integration-contract.md#pic-2), which derives the property from (i) <a id="sm-7c-slash-handler-serialisation"></a>Pi's per-session slash-handler serialisation, (ii) <a id="sm-7c-load-time-prompt-callee-rejection"></a>the load-time rejection of prompt-mode `.loom` callees in another loom's `tools:` per [Parameters and Frontmatter — `tools`](./spec_topics/frontmatter.md#tools), (iii) <a id="sm-7c-invoke-suspension"></a>the parent-suspending `invoke(...)` rule and the absence of a parallel-`invoke` surface owned by [Invocation — Cross-mode semantics](./spec_topics/invocation.md), and (iv) <a id="sm-7c-subagent-to-prompt-fanout-closed"></a>the closure of the `subagent → prompt` fan-out path into the user session owned by the same [Invocation — Cross-mode semantics](./spec_topics/invocation.md) rule.

  - <a id="sm-7d-no-cap-no-scheduler"></a> **SM-7d — loom 1.0 no-cap / no-scheduler disposition.** loom 1.0 concurrency disposition for in-flight invocations — no admission cap, no scheduler interposed above Pi's event loop — is owned by [Pi Integration Contract — No invocation cap](./spec_topics/pi-integration-contract.md#no-invocation-cap).

  - <a id="sm-7e-downward-only-cancellation"></a> **SM-7e — Downward-only cancellation propagation.** Cancellation propagates downward only (parent → children) per [Cancellation — Propagation](./spec_topics/cancellation.md); a sibling invocation's cancellation does not affect its siblings.

- <a id="sm-8-per-invocation-budget-non-sharing"></a> **SM-8 — Per-invocation budget non-sharing.** Each per-invocation budget — the binder retry budget, the `tool_loop.max_rounds` budget, and the `invoke`-chain depth budget — is scoped to a single invocation and is not shared, pooled, or replenished across sibling invocations; this non-sharing rule is owned, per budget, by its topic page: the binder retry budget by [Slash-Command Argument Binding — per-invocation retry budget](./spec_topics/binder.md#per-invocation-retry-budget), the `tool_loop.max_rounds` budget by [Query — Tool-call loop bound](./spec_topics/query.md#tool-call-loop-bound), and the `invoke`-chain depth budget by [Invocation — Invocation depth bound](./spec_topics/invocation.md).
