# Overview and orientation

## Overview

`pi-loom` is a [Pi Coding Agent](https://github.com/earendil-works/pi-mono) extension that adds a domain-specific scripting language for prompts and agentic operations.

A `.loom` file interleaves code with literal text destined for the model. Loom evaluation appends turns to a conversation: the *caller's* current conversation in `prompt` mode, or a separate conversation in `subagent` mode. The state-isolation contract for subagent invocation is enumerated in [Pi Integration Contract ‚Äî Subagent state-isolation matrix](../spec_topics/pi-integration-contract/subagent.md#subagent-state-isolation-matrix); the *callable set* concept that matrix references is defined in [Glossary ‚Äî `callable set`](../spec_topics/glossary.md). Mode is selected per-loom by the required `mode:` frontmatter field ‚Äî see [Parameters and Frontmatter](../spec_topics/frontmatter.md). On the success outcome, evaluation also produces a *final value* ‚Äî the value of the loom's tail expression, or ‚Äî if an explicit `return expr` executes first ‚Äî the operand of that `return`, per [Function Definitions ‚Äî Final value](../spec_topics/functions.md#final-value-language-definition) ‚Äî consumed by `invoke` callers and propagated across the subagent boundary per [Invocation ‚Äî Typed return](../spec_topics/invocation.md#typed-return). The loom language itself has no file-writing, network, or process-spawning primitive; effects of those kinds occur only through allowlisted tools, and the language's effect surface is owned by [Runtime Value Model ‚Äî Effects](../spec_topics/runtime-value-model.md#effects).

<a id="terminal-outcomes-aggregator"></a>Loom evaluation produces one of three terminal outcomes: it succeeds (turns appended; final value available to programmatic callers), it fails (by returning `Err`, by panicking, or via a [query-terminating](../spec_topics/glossary.md#query-terminating) enforcement point of one of the runtime-class [hard ceilings](#hard-runtime-ceilings) below ‚Äî query-terminating points route via panic or `Err`, with the per-arm classification *for ceiling #4* owned by [Hard Runtime Ceilings ‚Äî Per-boundary destination/surface table (ceiling #4)](../spec_topics/hard-ceilings/ceilings-3-and-4.md#ceiling-4-table); the following hard-ceiling cases are excluded from the Failure-cause enumeration: (a) binder argument-binding failure ‚Äî including ceiling #4's slash-load `params` arm cross-routed through ceiling #3 per [Hard Runtime Ceilings ‚Äî Per-boundary destination/surface table (ceiling #4)](../spec_topics/hard-ceilings/ceilings-3-and-4.md#ceiling-4-table) (slash-load only; the `invoke(...)` `params` arm of ceiling #4 IS an evaluation Failure that surfaces as `Err(InvokeInfraError { cause: "validation", ... })` per [Invocation ‚Äî Failures](../spec_topics/invocation.md)) ‚Äî per the binder argument-binding failure item in [Errors and Results ‚Äî Terminal outcomes](../spec_topics/errors-and-results/error-model.md#terminal-outcomes); see ceiling #3 in [Hard ceilings](#hard-runtime-ceilings), (b) ceiling #4's [in-loop](../spec_topics/glossary.md#in-loop) model-driven tool-call args row (which feeds a tool-error result back to the model rather than terminating the query)), or it is cancelled per [Cancellation](../spec_topics/cancellation.md). The cancellation signal is carried by the runtime-owned `loomAbort.signal` defined in [Pi Integration Contract ‚Äî Cancellation source](../spec_topics/pi-integration-contract/host-interfaces-core.md#cancellation-source); the runtime forwards Pi's per-handler `ctx.signal` (and the tool-exposed `signal` parameter, and parent-`invoke` signals) into `loomAbort` per the per-entry-point rules in [Cancellation ‚Äî Forwarding into `loomAbort`](../spec_topics/cancellation.md) and [Pi Integration Contract ‚Äî Cancellation source](../spec_topics/pi-integration-contract/host-interfaces-core.md#cancellation-source). The closed enumeration of these three terminal outcomes ‚Äî and the per-cause caller surface map (what an `invoke` parent sees, what a slash caller sees, what the operator (per [Glossary ‚Äî operator](../spec_topics/glossary.md#operator)) observes per channel) ‚Äî are owned by [Errors and Results ‚Äî Terminal outcomes](../spec_topics/errors-and-results/error-model.md#terminal-outcomes); the *partial-append contract* (turns appended before the terminal event remain in the driven conversation; the runtime performs no implicit rollback; mid-stream user-visible streaming fragments are governed separately) is owned by [Errors and Results ‚Äî Partial-append contract](../spec_topics/errors-and-results/error-model.md#partial-append-contract). Failures that occur before evaluation begins are owned, with their closed enumeration and per-cause routing rule, by [Errors and Results ‚Äî Terminal outcomes](../spec_topics/errors-and-results/error-model.md#terminal-outcomes); they never become evaluation outcomes. The success / fail / cancelled trichotomy applies only once evaluation has begun.

<a id="file-extension-grammar"></a>A loom is stored in one of two file extensions that share a single grammar and type system. `.loom` files are invocable as slash commands (see [Slash-Command Invocation](../spec_topics/slash-invocation.md)); `.warp` files are library modules whose top level is restricted to the declaration forms enumerated in [Imports ‚Äî Permitted top-level forms](../spec_topics/imports.md#permitted-top-level-forms) (currently five: `import`, `export`, `fn`, `schema`, `enum` ‚Äî *orientation aggregator per [Governance ‚Äî GOV-30](../spec_topics/governance.md)*). `.warp` files are never directly invoked. Path literals and discovery enforce these extensions per the rules in [Imports](../spec_topics/imports.md), [Invocation](../spec_topics/invocation.md), and [Discovery](../spec_topics/discovery.md); see those pages for the diagnostic codes that fire on each violation. See [Discovery ‚Äî File-extension namespace](../spec_topics/discovery/discovery-sources.md#file-extension-namespace) for the namespace-clearance note.

### Non-functional requirements

These cross-cutting loom 1.0 non-functional dispositions ‚Äî the trust/security posture, the source-language-equivalence release goal, and the operator-facing observability contract ‚Äî are not enumerated as a unit by any single topic page. The bullets are *informative orientation*: each one forward-links the topic page that owns the normative contract, and the two no-seam loom 1.0 dispositions among them ‚Äî Trust boundary and Source-language stability ‚Äî are also indexed under [Future Considerations ‚Äî loom 1.0 non-goals](../spec_topics/future-considerations/model-changes-and-non-goals.md#loom-1-0-non-goals); the consolidated loom 1.0 non-goals list (cross-cutting dispositions loom 1.0 ships without a forward-compatibility seam) is owned by [Future Considerations ‚Äî loom 1.0 non-goals](../spec_topics/future-considerations/model-changes-and-non-goals.md#loom-1-0-non-goals) and forward-linked from [loom 1.0 non-goals](./language-and-architecture.md#loom-1-0-non-goals-aggregator) below. The aggregator-vs-source lock-step convention for keeping these bullets honest as topic pages evolve is owned by [Governance ‚Äî GOV-30](../spec_topics/governance.md).

- **Trust boundary.** loom 1.0 looms execute inside the Pi extension-host process at full Node host-process privilege. loom 1.0 imposes no loom-level sandbox: filesystem, network, and Pi-API access available to loom code and to its declared callables are exactly those available to any Pi extension running in the same process ‚Äî Pi's `ExtensionAPI` and `ExtensionContext` surfaces (declared in `@earendil-works/pi-coding-agent`'s `dist/core/extensions/types.d.ts` across the release range pinned by [Pi Integration Contract ‚Äî Host prerequisites ‚Äî Pi SDK pin](../spec_topics/pi-integration-contract/host-prerequisites.md#pi-sdk-pin)) expose no per-extension privilege facet, and the resource-loader and tool-host surfaces grant capabilities at the host-process level; Pi exposes no per-extension privilege scoping the runtime can rely on as a security boundary. A future Pi minor that adds such a facet would surface at the build-time SDK surface-inventory assertion run by [Pi version bump procedure](../spec_topics/pi-integration-contract/version-bump-intro.md#pi-version-bump-procedure) before the spec contract drifts. The runtime itself interposes no additional access channels (sandbox, capability filter, mediated proxy) per [Pi Integration Contract ‚Äî No additional access channels](../spec_topics/pi-integration-contract/conversation-drive.md#no-extra-mediation). The loom's declared *callable set* (the entries listed under the `tools:` frontmatter field; default rule for absent or empty `tools:` is owned by [Parameters and Frontmatter ‚Äî `tools`](../spec_topics/frontmatter/frontmatter-fields-a.md#tools)) is a configuration knob over the *model's* reachable callable set, NOT a host-process sandbox: it is enforced mechanically by the per-mode wiring rule owned by [Pi Integration Contract ‚Äî Tool-registration lifetime and visibility](../spec_topics/pi-integration-contract/tool-registration-lifetime.md#tool-registration-lifetime-and-visibility), which owns both the per-mode composition of that set and the SDK-call mechanism. Any callable a loom declares exposes the full underlying capability of that callable to the loom's model; the runtime applies no privilege classification or filtering. `bash` and `read` are illustrative ‚Äî the rule does not depend on a callable being deemed "privileged." The host-side denial-surface rule ‚Äî how denials of filesystem, network, or Pi-API access reach loom code through the tool that issued the request, and that silent success on denial is forbidden ‚Äî is owned by [Pi Integration Contract ‚Äî No additional access channels](../spec_topics/pi-integration-contract/conversation-drive.md#no-extra-mediation); the complete enumeration of observable `execute()` outcomes is owned by [Tool Calls ‚Äî Failures](../spec_topics/tool-calls.md) and [Pi Integration Contract ‚Äî Tool execution from loom code](../spec_topics/pi-integration-contract/host-interfaces-core.md#tool-execution-from-loom-code). A per-loom capability model is **out of scope for loom 1.0**; the disposition is owned by the **No per-loom sandbox or capability model** bullet under [Future Considerations ‚Äî loom 1.0 non-goals](../spec_topics/future-considerations/model-changes-and-non-goals.md#loom-1-0-non-goals).

- <a id="source-language-stability"></a> **Source-language stability.** A `.loom` or `.warp` file that [loads cleanly](../spec_topics/governance/source-language-stability.md#gov-15-loads-cleanly) under loom 1.0.0 is expected to load under every loom 1.x release and to produce, for any given input, identical (a) return values, (b) ordered diagnostic-code sequences, and (c) equivalent `loom-system-note` content strings. Two content strings are equivalent under observable (c) when they are byte-identical ‚Äî i.e. equal as UTF-8 byte sequences, the encoding substrate pinned by [Lexical ‚Äî Encoding](../spec_topics/lexical.md) ‚Äî after normalising the placeholder sub-fields whose rendered value is permitted to vary per invocation or per run ‚Äî the per-invocation identifiers (e.g. `<invocation-id>` / `<uuid>`), the wall-clock-derived values (e.g. `<ms>`), and the category-8 host-derived freeform tails ‚Äî as classified by the placeholder-rendering categories in [Diagnostics ‚Äî Placeholder rendering](../spec_topics/diagnostics/placeholder-rendering-a.md#placeholder-rendering-normative); the fixed (non-variable) placeholder renderings and the surrounding literal template bytes are themselves byte-identical. Observable (c)'s equivalence relation, not the exclusion list, governs variation that is embedded as a placeholder sub-field inside a content string. Wall-clock timing, token counts, and log-line volume remain excluded from this expectation as independent observables in their own right ‚Äî distinct from the wall-clock-derived sub-fields that observable (c) normalises within a content string. This is a release-process goal, not a normative obligation; the equivalence statement, the loom 1.0 review-vs-mechanical-gate posture, and the deferred-conformance-suite framing are owned by [Governance ‚Äî GOV-15](../spec_topics/governance/source-language-stability.md#gov-15) with the companion review-posture rule at [GOV-14](../spec_topics/governance/req-id-prefix-table-active-b.md#gov-14). Introducing a new hard runtime ceiling within loom 1.x ‚Äî or retiring or relaxing an existing one ‚Äî is a recognised carve-out to that equivalence per [GOV-15](../spec_topics/governance/source-language-stability.md#gov-15); the in-scope input set (defined post-hoc over the inter-release diff), the carve-out's diagnostic-emissions scope (per [GOV-15 ‚Äî Ceiling-set carve-out](../spec_topics/governance/source-language-stability.md#ceiling-set-carve-out)), and the attribution test that distinguishes carve-out-eligible edits from collateral edits are all owned by [GOV-15](../spec_topics/governance/source-language-stability.md#gov-15). The long-term migration framing (no major-version migration mechanism) is owned by [Future Considerations ‚Äî loom 1.0 non-goals](../spec_topics/future-considerations/model-changes-and-non-goals.md#loom-1-0-non-goals). Migration across major versions is out of loom 1.0 scope.
